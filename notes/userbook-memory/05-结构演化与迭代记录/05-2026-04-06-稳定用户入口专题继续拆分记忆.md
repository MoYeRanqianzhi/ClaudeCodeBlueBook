# 稳定用户入口专题继续拆分记忆

## 本轮问题

`04-专题深潜` 已经把运行时主线、高频交互和恢复发现拆得更细，但仍存在一个结构偏差：

- 对运行时核心与交互细节写得较深
- 对“用户真正从哪里进入系统”的稳定入口仍然覆盖不均

缺的不是更多底层机制，而是三类高频入口对象：

- 非交互 / 后台 / 自动化
- 初始化 / 安装 / setup
- 账户 / 隐私 / 资格 / 升级

## 苏格拉底式自审

### 问：如果读者的目标是“把 Claude Code 接到脚本或宿主里”，他为什么要先读状态、恢复或终端效率专题？

答：不该先读。自动化入口首先关心的是 I/O 契约、会话持久化、后台附着关系，而不是交互式 REPL 体验。

### 问：`/init`、`claude install`、`/terminal-setup`、`/install-github-app` 能放进同一页吗？

答：能，但必须按“仓库对象 / 宿主对象 / 外部平台对象”分层，否则会重新把 setup 写成一团。

### 问：`/privacy-settings`、`/passes`、`/upgrade` 属于状态运营专题吗？

答：不属于。它们不是“观察与调参”，而是产品面、资格面和身份面的入口；混进去会让读者误把消费者订阅能力写成通用能力。

### 问：哪些地方最容易把灰度写成稳定？

答：

- `NEW_INIT` richer flow 容易被误写成 `/init` 的稳定合同
- `--bg` / `ps` / `logs` / `attach` / `kill` 的 handler 细节容易因为缺失 `src/cli/bg.js` 被过度推断
- `rate-limit-options` 容易被写成公开主菜单
- `/init-verifiers` 容易因为在源码中可见就被写成公开能力

### 问：哪些地方反而容易被误判成灰度，但正文应承认其稳定主线地位？

答：

- `claude -p/--print`
- `/login`
- `/logout`
- `/terminal-setup`
- `claude install`

## 结构决策

本轮在 `04-专题深潜/` 新增：

- `13-非交互、后台会话与自动化专题.md`
- `14-初始化、安装与开工环境搭建专题.md`
- `15-账户、隐私、资格与升级专题.md`

这样做的目的不是扩目录数量，而是让“入口对象”与“控制面对象”继续解耦：

- 入口专题回答“我从哪里开始做事”
- 控制面深挖回答“为什么系统把边界划在这里”

## 正文约束

继续坚持两条：

- 正文只保留读者面事实、运行时合同、误用点、用户策略
- 作者侧判断、拆分原因、边界保守策略、苏格拉底式反问，全部留在 `notes/userbook-memory/`

## 后续规则

### 非交互专题

只把 `-p/--print` 写成稳定主线；`--bg` / `ps` / `logs` / `attach` / `kill` 写成条件公开；`--sdk-url` 写成隐藏集成面；内部 runner/daemon 不写进普通主线。

### setup 专题

必须持续区分：

- 斜杠命令
- CLI 根命令
- 浏览器跳转型外部入口

### 账户专题

必须持续区分：

- `claude.ai` 消费者订阅
- Anthropic API / usage-billed 身份
- 3P provider 身份

只要这一层不分开，正文就会再次把产品边界写坏。
