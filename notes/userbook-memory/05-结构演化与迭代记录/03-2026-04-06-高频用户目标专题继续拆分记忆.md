# 2026-04-06 高频用户目标专题继续拆分记忆

## 本轮结构判断

上一轮把运行时主线拆细之后，正文里剩下的拥挤点不再是“底层机制”，而是“高频用户目标被旧专题挤在一起”。

具体表现：

- `01-代码工作流专题` 同时承担探索、修改、评审、提交、PR、导出、反馈。
- `01-主线使用/02-提问、补上下文...` 已指出入口，但还不足以承接“上下文如何真正进入系统”的长文解释。
- `05-控制面深挖/06-Status、Doctor、Usage...` 已解释控制面，但还没有用户节奏视角的专题页。

因此本轮应新增 3 类专题，而不是继续往旧页加段落：

1. 上下文接入、附件与提示编译
2. 评审、提交、导出与反馈
3. 状态、额度、模型与节奏运营

## 新的拆页纪律

### 1. 代码工作流专题不再继续吞掉“交付与外部化”

以后以下内容默认应外提：

- `/review`
- `/ultrareview`
- `/commit`
- `/commit-push-pr`
- `/export`
- `/feedback`
- `/release-notes`

原因：

- 它们处理的是“工作对象升级与外部化”，不是“代码探索与修改”本身。

### 2. 主线使用页只保留前门式说明，不承担长文机制解释

`01-主线使用/02-提问、补上下文与让模型继续工作.md` 可以保留：

- 用户应先做什么
- 有哪些入口
- 什么时候换入口

但不应继续堆：

- 附件链、选区链、目录扩张链、hook 回写链的深机制

这些应进入专题页。

### 3. 控制面页与用户运营页必须分开

例如：

- `Status / Doctor / Usage` 的控制面解释属于 `05-控制面深挖`
- “我该怎样运营一场长任务的状态、额度、模型和节奏”属于 `04-专题深潜`

不要再把控制面页写成用户操作长文，也不要把用户长文写成控制面 taxonomy。

## 本轮新增的去记忆化约束

本轮若写“为什么要把这些主题单独拆出来”，这段只能留在记忆层，不能回流正文。

正文里只保留：

- 用户目标
- 第一性原理
- 正确入口
- 隐藏运行时合同
- 最易误用点
- 稳定/条件边界

## 下一轮继续深入时优先怀疑的问题

1. 是否又把“控制面长文”和“用户目标长文”写重复了。
2. 是否又把“会话外部化”和“会话连续性”混成了一类动作。
3. 是否又把“输入编译”误缩写成“写好 prompt 就行”。
