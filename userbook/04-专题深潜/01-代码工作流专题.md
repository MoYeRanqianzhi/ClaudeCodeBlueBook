# 代码工作流专题

## 用户目标

完成最常见的开发闭环：

- 探索代码。
- 修改实现。
- 检查 diff。
- 审查变更。
- 生成 commit。

## 正确入口

### 探索与修改

默认应走普通提示词主线，让模型自行调度 `Read`、`Edit`、`Write`、`Grep`、`Glob`、`Bash`。

原因不是“更智能”，而是主循环会根据任务动态决定：

- 先读哪些文件。
- 是否需要 shell。
- 何时停止继续读、转入修改。

### 看变更

应优先用 `/diff`，因为它直接打开本地 diff UI，看的是会话与工作树关联后的变更面，而不是要求模型再口述一次 diff。

### 评审

应区分 `/review` 和 `ultrareview`：

- `/review` 是本地 prompt command，核心依赖 `gh pr list/view/diff`。
- `ultrareview` 是单独的 web 侧远程路径，且只在启用时出现。

所以“我要审查 PR”并不只有一种实现面。

### 提交

应优先用 `/commit`，而不是直接让模型自由发挥 git 命令。

原因在于 `src/commands/commit.ts` 已把这条工作流收紧成：

- 固定上下文：`git status`、`git diff HEAD`、当前分支、最近提交。
- 固定安全协议：禁止改 git config、禁止 `--amend`、禁止交互式 git。
- 固定工具白名单：只允许有限的 `Bash(git ...)` 调用。

这是一条非常典型的 Claude Code 设计哲学：高风险但高频的工作流，优先被压成受控 prompt command。

### 提交并开 PR

`/commit-push-pr` 比 `/commit` 更进一步，它不是简单把三步串起来，而是把完整 PR 工作流协议化：

- 如果当前在默认分支上，会先要求创建新分支。
- 分支名前缀优先使用 `SAFEUSER`，其次才回退到系统用户名。
- 先看 `git diff <defaultBranch>...HEAD`，不是只看最后一个 commit。
- 如果当前分支已经有 PR，走 `gh pr edit`；否则走 `gh pr create`。
- PR 标题被要求保持简短，细节放到 body。
- 在非 undercover 情况下，还会把 reviewer、changelog、Slack 通知检查写进协议。

这说明 `/commit-push-pr` 的真实角色不是“快捷按钮”，而是“把变更收口成团队协作对象”的工作流编译器。

## 隐藏的运行时合同

### `/commit`

它的关键不是“会提交”，而是“只允许以一种很窄的方式提交”：

- 允许的工具只有极小的 `git add/status/commit` 集合。
- 提交前总是先把仓库事实内联进 prompt。
- 安全协议比一般聊天 prompt 更硬。

### `/commit-push-pr`

它的关键不是“会开 PR”，而是“把 branch + commit + PR + 协作元信息编成一个对象升级流程”。

### `/review`

它的关键不是“让模型评审”，而是“围绕 PR 对象去拉取列表、详情和 diff，再落成本地审查流程”。

## 为什么 Claude Code 不用更简单方案

### 为什么不直接让模型自由使用所有 git / gh 命令

因为这会让高频高风险动作退化成无边界 shell 行为。

### 为什么不把 `/review` 和 `ultrareview` 合成一个按钮

因为本地审查和远程深审的宿主、成本、稳定性前提都不同。

## 使用设计事实

### 1. `/diff` 的价值是“直接站在本地事实面”

它是 `local-jsx` 命令，直接把 `context.messages` 交给本地 `DiffDialog`。这意味着 `/diff` 的输出不是模型重述，而是宿主侧真实差异面。

### 2. `/review` 与 `ultrareview` 不是深浅两个按钮

`/review` 的默认实现是本地 prompt command，本质依赖 `gh`；`ultrareview` 则是远程 web 路径。它们的宿主、时延、前提和稳定性都不同。

### 3. `/commit` 是“受控提交协议”

它不是把 git 全权交给模型，而是先通过 `executeShellCommandsInPrompt()` 拉取仓库事实，再把允许调用的 Bash 规则压到极小集合。

### 4. `/commit-push-pr` 解决的是“协作对象升级”

它真正把本地变更升级成 branch + PR 这个团队可消费对象，而不只是顺手多做两条 git/gh 命令。

## 不该误用的邻近能力

### 不要把 `/review` 当成自动远程深审

它的默认实现非常本地，而且依赖 `gh` CLI。

更具体地说：

- `/review` 只是一个本地 prompt command，核心动作仍是 `gh pr list/view/diff`。
- `ultrareview` 才是远程 web 路径，而且源码明确写明它是通往远程 bughunter 路径的唯一入口。

### 不要把 `/commit` 当成“自动全仓库提交”

它会基于当前变化生成提交，但其安全协议明确假定应谨慎分辨要不要 stage 某些文件。

### 不要默认一切都走 shell

Claude Code 同时保留文件工具和 shell，不是冗余，而是想让你把“可结构化的文件操作”和“不可避免的命令生态”区分开。

### 不要把 `/commit-push-pr` 当成无前提的一键发布

它隐含要求：

- 仓库有可用的远程。
- `gh` CLI 可用并已认证。
- 当前 diff 足以形成单个 PR 工作对象。
- 用户没有要求更复杂的分批提交或多 PR 拆分。

## 稳定面与灰度面

### 稳定面

- 普通提示词 + 文件工具。
- `/diff`
- `/review`
- `/commit`
- `/commit-push-pr`

### 灰度或条件面

- `ultrareview`
- 更远程化的 web 审查路径。
- 可能与 KAIROS/后台代理联动的 PR 工作流。

### 内部/敏感面

- `undercover` 提交伪装逻辑。
- ant-only attribution / prompt patch。

这些存在于源码中，但不应写进普通用户的稳定工作流。

## 对用户的策略结论

最稳的代码工作流通常是：

1. 先用普通提示词推进探索和修改。
2. 用 `/diff` 而不是让模型复述 diff。
3. 用 `/review` 或 `ultrareview` 处理审查，而不是混在主聊天里。
4. 用 `/commit` 收口提交，而不是手写一套临时 git 指令。
5. 需要完整协作闭环时，再用 `/commit-push-pr` 把变更升级成 branch + PR 对象。
