# 扩展与集成专题

## 用户目标

让 Claude Code 获得项目特定工作流、外部系统能力和运行时自动化，而不把所有要求都硬编码进一次性 prompt。

## 正确入口

### 技能

当你只是想固定某套工作流时，优先技能。

技能能声明：

- 说明文本。
- 工具白名单。
- 模型偏好。
- `inline` 或 `fork`。
- hooks。
- 附带参考文件。

这比“写个长 prompt 模板”更接近正式运行时扩展。

### 插件

当你需要把多条技能、hooks、MCP 服务器打成可启停扩展包时，优先插件。

`/plugin` 本质上是扩展包管理面，而不仅是安装器。

更具体地说，built-in plugins 和 marketplace plugins 在产品上被统一进 `/plugin`，这说明插件层真正承担的是“可治理的扩展包”，而不是单纯分发渠道。

### MCP

当你要连接外部系统，优先 MCP。

`services/mcp/client.ts` 的体量本身就说明 MCP 不是附带功能，而是大型子系统，负责：

- 连接 transport。
- OAuth / auth 缓存。
- 工具暴露。
- resource 暴露。
- 输出截断与持久化。
- 权限与状态管理。

`/mcp` 命令本身也说明了一个边界事实：

- 它支持 `enable` / `disable` / `reconnect` 这类直接管理动作。
- 某些构建里基础 `/mcp` 视图还会重定向到插件管理页。

这意味着产品层已经在把“插件 + MCP”视作相邻控制面，而不是完全割裂两套入口。

### Hooks

当你想在运行时关键事件上插入外部脚本时，用 hooks。

Hooks 适合：

- prompt 前后加工。
- 工具调用前后扩展。
- task / session 生命周期事件。

## 使用设计事实

### 1. 技能优先解决“方法复用”

如果目标只是固定一套做法，技能是最低摩擦的扩展层，因为它天然进入命令空间。

### 2. 插件优先解决“打包与治理”

插件层的真实价值在 marketplace、作用域、启停、错误引导和多组件打包，而不是代替技能或 MCP 自身。

### 3. MCP 优先解决“跨系统边界”

一旦问题涉及 transport、auth、resource、外部 API 和工具合并，就已经进入 MCP 语义，而不是普通技能语义。

### 4. Hooks 优先解决“运行时插针”

它们是在既有主循环上插入逻辑，而不是另起一套能力系统。

## 不该误用的邻近能力

### 不要用插件代替技能

如果只是固化一个工作流，插件层级太重。

### 不要用 MCP 代替本地工具

如果能力原生已经有，优先原生工具，因为它们更稳定、更可缓存、更好治理。

### 不要把 hooks 当“隐式魔法”

Claude Code 已把 hooks 做成正式运行时接口，应明确记录用途和触发点。

### 不要把 `/plugin` 和 `/mcp` 理解成纯安装页面

它们本质上都是运行时能力治理面，影响的是后续命令空间、工具池和行为边界。

## 稳定面与灰度面

### 稳定面

- `/skills`
- `/plugin`
- `/mcp`
- `/hooks`
- bundled skills 基础集合

### 灰度或条件面

- MCP skills
- 某些内置插件
- Claude API / Chrome / 远程调度类 skills
- 更深的 marketplace / official registry 行为

## 对用户的策略结论

扩展决策树很简单：

1. 固化流程，用技能。
2. 打包多能力，用插件。
3. 接外部系统，用 MCP。
4. 拦截运行时事件，用 hooks。
