# 扩展与集成专题

## 用户目标

让 Claude Code 获得项目特定工作流、外部系统能力和运行时自动化，而不把所有要求都硬编码进一次性 prompt。

更稳的理解不是“五选一”，而是两根轴同时工作：

- `plugin` 负责打包、分发与治理
- `skills / hooks / MCP` 分别负责提示型工作流、强制自动化、外部能力接入

## 正确入口

### 技能

当你只是想固定某套工作流时，优先技能。

技能能声明：

- 说明文本。
- 工具白名单。
- 模型偏好。
- `inline` 或 `fork`。
- hooks。
- 附带参考文件。

这比“写个长 prompt 模板”更接近正式运行时扩展。

### 插件

当你需要把多条技能、hooks、MCP 服务器打成可启停扩展包时，优先插件。

`/plugin` 本质上是扩展包管理面，而不仅是安装器。

更具体地说，built-in plugins 和 marketplace plugins 在产品上被统一进 `/plugin`，这说明插件层真正承担的是“可治理的扩展包”，而不是单纯分发渠道。

更进一步：

- plugin 是组合层
- 它可以携带 skills、agents、hooks 与 MCP
- 但它不等于更强的 settings 面

### MCP

当你要连接外部系统，优先 MCP。

`services/mcp/client.ts` 的体量本身就说明 MCP 不是附带功能，而是大型子系统，负责：

- 连接 transport。
- OAuth / auth 缓存。
- 工具暴露。
- resource 暴露。
- 输出截断与持久化。
- 权限与状态管理。

`/mcp` 命令本身也说明了一个边界事实：

- 它支持 `enable` / `disable` / `reconnect` 这类直接管理动作。
- 某些构建里基础 `/mcp` 视图还会重定向到插件管理页。

这意味着产品层已经在把“插件 + MCP”视作相邻控制面，而不是完全割裂两套入口。

### Hooks

当你想在运行时关键事件上插入外部脚本时，用 hooks。

Hooks 适合：

- prompt 前后加工。
- 工具调用前后扩展。
- task / session 生命周期事件。

但它们更适合快、机械、必须执行的动作，而不是深度验证、部署编排或长流程自治。

## 使用设计事实

### 1. 技能优先解决“方法复用”

如果目标只是固定一套做法，技能是最低摩擦的扩展层，因为它天然进入命令空间。

### 2. 插件优先解决“打包与治理”

插件层的真实价值在 marketplace、作用域、启停、错误引导和多组件打包，而不是代替技能或 MCP 自身。

### 2.5 普通用户更稳的选择顺序是“先能力，再分发”

更稳的次序通常是：

1. 先用 `skills` 固化 repo 工作流和知识
2. 需要强制时再升到 `hooks`
3. 需要外部系统能力时再上 `MCP`
4. 需要共享、版本化、安装体验时再把这些能力打进 `plugin`

### 3. MCP 优先解决“跨系统边界”

一旦问题涉及 transport、auth、resource、外部 API 和工具合并，就已经进入 MCP 语义，而不是普通技能语义。

### 4. Hooks 优先解决“运行时插针”

它们是在既有主循环上插入逻辑，而不是另起一套能力系统。

## 不该误用的邻近能力

### 不要用插件代替技能

如果只是固化一个工作流，插件层级太重。

### 不要用 MCP 代替本地工具

如果能力原生已经有，优先原生工具，因为它们更稳定、更可缓存、更好治理。

### 不要把 hooks 当“隐式魔法”

Claude Code 已把 hooks 做成正式运行时接口，应明确记录用途和触发点。

### 不要把 hooks 当工作流编排器

它们是事件触发，不适合承载重、慢、深的长流程。

### 不要把 `/plugin` 和 `/mcp` 理解成纯安装页面

它们本质上都是运行时能力治理面，影响的是后续命令空间、工具池和行为边界。

### 不要把 repo 约定塞进 MCP

repo 知识和方法约束通常更该放进 `skills`；MCP 更适合“接能力”而不是“存规则”。

## 稳定面与灰度面

### 稳定面

- `/skills`
- `/plugin`
- `/mcp`
- `/hooks`
- bundled skills 基础集合

### 灰度或条件面

- MCP skills
- 某些内置插件
- Claude API / Chrome / 远程调度类 skills
- 更深的 marketplace / official registry 行为
- managed settings、plugin-only policy 对来源层级的组织级覆盖

## 对用户的策略结论

扩展决策树更稳的版本是：

1. 固化 repo 工作流和知识，用技能。
2. 必须执行的快动作，用 hooks。
3. 接外部系统、浏览器或第三方工具，用 MCP。
4. 需要共享、版本化、安装体验和多组件组合时，再用插件。
