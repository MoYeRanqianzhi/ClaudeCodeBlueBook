# 终端交互、状态栏与输入效率专题

## 用户目标

不是只会“发一条 prompt”，而是让 Claude Code 在终端里更顺手：

- 输入更快
- 状态更容易盯
- 视觉和按键更贴合当前习惯
- 不必为低级交互摩擦反复打断主任务

## 第一性原理

Claude Code 的终端体验不是附属皮肤，而是正式运行面。

从源码看，这条线至少分成三层：

- 外观层：`/theme`、`/color`
- 输入层：`/vim`、`/keybindings`、提示输入框、history recall
- 持续观测层：`/statusline`

这意味着 Claude Code 并不把“输入框”当成被动文本框，而是把终端交互本身当作可治理子系统。

## 正确入口

### `/theme`

适合整体调整终端主题。

它走的是本地 JSX picker，改变的是当前终端体验，而不是提示词行为。

### `/color`

适合快速调整会话提示条或颜色标识，让不同会话和不同工作状态更容易被肉眼分辨。

### `/vim`

适合把输入行为切换到 Vim 模式。

这不是小彩蛋，而是正式交互模式切换。

### `/keybindings`

适合在终端内重新理解和管理快捷键，而不是靠记忆或外部文档硬背。

### `/statusline`

适合把持续需要盯的信息外显到状态栏。

它不是普通查询命令，而是一个 prompt 命令：会调用专门的 `statusline-setup` 子代理，读取你的 shell 现状并写入配置。

更稳的理解是：

- `/statusline` 不是“显示状态”
- 而是“配置如何持续显示状态”

### 输入框历史回调

适合快速找回最近 prompt，而不是重新手打。

这条能力不是 `/resume`，也不是普通会话搜索，而是当前输入层的 prompt 历史检索。

## 隐藏的运行时合同

### `/theme` 与 `/keybindings` 属于本地交互层

它们更接近本地 JSX 面板，而不是模型工作流。

### `/statusline` 不是本地面板，而是受控 prompt 工作流

它允许的工具非常窄：

- 创建指定子代理
- 读取用户环境
- 编辑 `~/.claude/settings.json`

所以它的目标不是“给你看一眼状态”，而是把状态观测正式写进终端配置。

### 历史检索不是 `/resume`

`HistorySearchDialog` 读的是 prompt 历史，支持：

- 初始查询
- 模糊匹配
- 右侧或底部预览
- 直接把选中条目重新送回输入框

它解决的是“我以前怎么提过这个问题”，不是“我以前在哪个会话里做过这项工作”。

## 最容易误用的点

### 把 `/statusline` 当成只读命令

它本质上会触发一次配置工作流，而不是只返回一段文本。

### 把主题、颜色和按键绑定当纯皮肤

这些设置会直接改变你在长任务里的摩擦成本，因此它们是效率面，不只是审美面。

### 把 prompt 历史检索和 `/resume` 混成一件事

前者找回的是输入，后者接回的是工作对象。

### 以为 Vim 模式只是快捷键集合

它改变的是输入行为本身，不只是多几个热键。

## 稳定面与条件面

### 稳定主线

- `/theme`
- `/color`
- `/vim`
- `/keybindings`
- prompt 历史检索

### 条件或依赖环境

- `/statusline` 的具体效果取决于配置、hooks、trust 和 shell 环境

## 用户策略

更稳的终端交互路径通常是：

1. 先用 `/theme`、`/color`、`/vim`、`/keybindings` 把输入环境调顺。
2. 再用 `/statusline` 把你会反复盯的状态外显出来。
3. 需要找回旧提问时，优先用输入层历史检索，而不是翻完整会话。
