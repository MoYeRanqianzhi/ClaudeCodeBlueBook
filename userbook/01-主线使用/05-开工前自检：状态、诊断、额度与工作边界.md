# 开工前自检：状态、诊断、额度与工作边界

## 第一性问题

真正会让 Claude Code 工作流失控的，往往不是“提示词写得不够好”，而是：

- 你以为自己在正确的会话里，其实状态已经漂移。
- 你以为环境可信，其实安装、配置、MCP 或 sandbox 已经失真。
- 你以为任务能继续跑，其实额度和速率已经接近边界。
- 你以为模型已经能安全处理另一个目录，其实工作边界还没正式扩张。

所以在真正投入工作前，最稳的顺序不是立刻提需求，而是先完成一次开工前自检。

## 先看 `/status`

`/status` 解决的不是“看点信息”，而是确认当前会话到底站在什么运行态上。

从源码看，它不是普通文本命令，而是直接打开 `Settings` 的 `Status` 页签。这意味着 Claude Code 把版本、模型、账户、连接、MCP、工具状态和部分诊断信息，统一当成运行时现场的一部分。

对用户的实际意义：

- 当你刚进入一个旧会话时，先确认当前模型、账户和连接状态。
- 当你怀疑某些工具为什么消失时，先看状态面，而不是直接猜 gate 或权限。
- 当你要接着前一次工作继续时，先确认自己看到的是不是同一个宿主现场。

最常见误用：

- 把 `/status` 当“关于页”。
- 连接、工具、MCP、模型异常时完全不看状态面，直接开始排查业务问题。

## 再看 `/doctor`

`/doctor` 解决的是“我现在还能不能信任这套工具环境”。

它打开的是独立的诊断屏，不只是简单文本输出。源码显示它会汇总 settings 错误、环境变量上限、sandbox 诊断、agent 目录、context warning、version lock 等多类环境事实。

对用户的实际意义：

- 当 Claude Code 本身表现异常时，用它先判断问题在安装、配置、权限还是扩展层。
- 当 `/status` 只能告诉你“有问题”，但还不知道问题落在哪一层时，用 `/doctor` 收敛根因。

最常见误用：

- 只在“命令装不上”时才想到 `/doctor`。
- 把它当项目代码调试器，而不是工具环境可信度检查器。

## 把 `/usage` 当运行时预算，而不是账单

`/usage` 解决的是“这次任务还能不能继续稳定推进”。

它在实现上也是 `Settings` 里的独立 `Usage` 页签，说明 Claude Code 把额度和使用量当成配置与治理面的一部分，而不是会话外的财务尾巴。

对用户的实际意义：

- 长任务前先看额度，比撞上限后再改策略更稳。
- 当你要决定是继续深挖、压缩、切会话，还是降低节奏时，预算本身就是运行时输入。

最常见误用：

- 直到 hit rate limit 才第一次去看 `/usage`。
- 以为它只是套餐账单页，和当前任务规划无关。

需要补一句边界：

- 它依赖 Claude AI 订阅与对应产品面，不应被写成所有 provider 都一致存在的命令。

## 用 `/add-dir` 正式扩张工作边界

`/add-dir` 解决的不是“切到另一个路径”，而是“让 Claude Code 正式承认另一片目录也属于当前工作面”。

源码里它会先验证目录是否适合加入当前 workspace，然后更新 session permission context，并刷新 sandbox 配置；如果用户选择记住，还会把变更持久化到 local settings。

对用户的实际意义：

- `cd` 到别处不等于模型已经正式拥有那片目录的工作权限。
- 当任务跨多个仓库、文档目录或辅助脚本目录时，应通过 `/add-dir` 扩张正式边界。

最常见误用：

- 把 `/add-dir` 当 shell 技巧。
- 忽略它同时在改“上下文边界”和“权限边界”。

## 一个更稳的开工顺序

从源码设计倒推，进入一项新工作或恢复一项旧工作时，更稳的顺序通常是：

1. 先用 `/status` 看清当前宿主现场。
2. 如果环境有异常征兆，再用 `/doctor` 判断工具本身是否可信。
3. 对长任务或高成本任务，先看 `/usage`，把预算当作运行时约束。
4. 若要把新目录纳入工作面，再用 `/add-dir` 正式扩张边界。
5. 完成这一步后，再开始给任务、附件和上下文。
