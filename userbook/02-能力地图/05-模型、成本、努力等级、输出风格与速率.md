# 模型、成本、努力等级、输出风格与速率

## Claude Code 把“怎么想”也当成配置面

`/model`、`/fast`、`/effort`、`/output-style`、`/usage`、`/cost`、`/stats`、`/rate-limit-options` 这些命令表明，Claude Code 管的不只是任务内容，还管：

- 用哪个模型。
- 以多高努力等级工作。
- 什么时候偏快，什么时候偏稳。
- 成本与额度如何被观察。

## 模型不是孤立开关

从 `src/commands/model/index.ts`、`src/utils/model/*`、`src/utils/effort.ts` 可以看到，模型选择会影响：

- 系统提示词装配。
- 特定功能是否可用。
- effort 行为。
- 限额与速率面。

所以正确理解不是“换个更强模型”，而是“整套运行时预算和能力面随之变化”。

## `fast` 和 `effort` 不是重复功能

它们控制的是不同维度：

- `fast` 更偏吞吐与响应速度策略。
- `effort` 更偏思考深度与任务投入度。

这两者一起出现，说明 Claude Code 已经把“推理强度”和“系统吞吐策略”拆开治理。

## 为什么成本与使用统计是一等公民

有独立的 `/cost`、`/usage`、`/stats`、`/extra-usage`，说明 Claude Code 不是假设预算无限，而是把预算当运行时输入。

对用户而言：

- 长会话要看 `/cost` 和 `/usage`，不是事后看账单。
- 当限额成为系统性阻塞时，要用 `/rate-limit-options` 或 `/extra-usage` 处理，而不是盲目重试。

## 输出风格也被产品化了

虽然 `/output-style` 已被标注为过时并引导到 `/config`，但这恰恰说明输出风格不是 prompt 小技巧，而是产品层设置。

## 用户实践

### 探索期

可以先用较快、较轻的配置摸清问题面。

### 实施期

当任务进入重构、审查、验证等阶段，再提高 effort 或换更强模型。

### 长时间工作

要把成本与配额观察纳入日常流程，而不是只盯结果。
