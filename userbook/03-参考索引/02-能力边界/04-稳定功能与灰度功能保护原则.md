# 稳定功能与灰度功能保护原则

这章不是在描述 Anthropic 的官方政策，而是在为这本 userbook 建立写作与使用规则，避免把灰度能力误写成稳定承诺。

## 原则 1：稳定面和灰度面不能用同一语气书写

像 `Read`、`Edit`、`Write`、`Bash`、`/resume`、`/compact` 这类能力，和 `VOICE_MODE`、`ULTRAPLAN`、`WORKFLOW_SCRIPTS`、`KAIROS` 这类能力，不应在文档中使用同样确定的表述。

## 原则 2：源码可见不等于用户可稳定依赖

影响真实能力面的因素至少还有：

- 账户类型。
- 组织策略。
- 平台环境。
- feature gate。
- 发布构建是否保留完整实现。

## 原则 3：保护稳定面的办法，是优先给用户稳定入口

文档优先推荐：

- 稳定命令入口。
- 稳定工具面。
- 稳定治理手段，如 `/permissions`、`/plan`、`EnterWorktree`、`/compact`。

而不是鼓励用户默认绑定高门控路径。

## 原则 4：灰度能力应被写成“可尝试路径”，不是“默认主线”

例如：

- 语音。
- 浏览器自动化。
- 某些 cron/trigger/notify 能力。
- assistant / proactive / KAIROS 路径。

它们可以被解释，但不应被写成所有用户都能直接复现的标准教程。

## 原则 5：内部能力主要用来理解方向，不用来承诺效果

内部或 ant-only 能力最有价值的地方，是帮助我们理解：

- 架构为未来预留了什么。
- 产品正在往哪里长。

它们不应用于对普通读者做稳定使用承诺。

## 面向读者的简单规则

### 可以纳入日常工作流

稳定公开能力。

### 可以做条件实验

灰度和条件公开能力。

### 可以帮助理解未来，但别绑定生产习惯

内部能力与缺失模块影子。

## 苏格拉底反诘

### 问：如果某能力从源码上看很强，为什么不直接写成“Claude Code 支持 X”？

答：因为 userbook 的任务不是放大可能性，而是保护判断精度。

### 问：如果灰度能力不稳定，为什么还要写？

答：因为不写就看不见产品方向；但写法必须把“方向洞察”和“稳定承诺”分开。
