# 并行执行与隔离专题

## 用户目标

把 Claude Code 从“单线程助手”提升到：

- 有后台任务。
- 有子代理。
- 有团队协作。
- 有调度。
- 有隔离执行面。

同时避免并行执行互相踩文件、踩上下文、踩责任边界。

## 正确入口

### 子代理

通过 `Agent` 工具进入。

它并不只是一个 prompt 转发器，而能声明：

- agent 类型。
- 模型。
- 背景运行。
- team 名称。
- permission mode。
- `worktree` 或某些构建里的 `remote` 隔离。

### 任务系统

通过 `/tasks` 和 `Task*` 工具进入。

`utils/tasks.ts` 说明任务系统的目标是多进程安全：

- 任务落盘。
- 文件锁防并发竞争。
- task list id 能和团队、session、teammate 上下文对齐。
- reset / high-water mark 防任务 ID 复用混乱。

还有一个容易误读的细节：

- `TaskOutputTool` 已被实现为偏兼容层，描述里明确写了“更推荐直接 `Read` 任务输出文件路径”。

这说明 Claude Code 在逐步把“后台结果查看”重新并回通用文件读取面，而不是维持一套完全独立的输出 API。

### 团队与消息

通过 `TeamCreate`、`TeamDelete`、`SendMessage` 进入。

它们的存在说明 Claude Code 的并行不是“开多个独立 agent”，而是已经进入了可通信协作阶段。

### 调度

通过 cron 工具和某些 bundled skills 进入：

- `CronCreate`
- `CronDelete`
- `CronList`
- `RemoteTrigger`
- `loop`
- `schedule`

这部分代表系统开始支持“未来时刻再行动”。

`CronCreateTool` 里的设计也很值得用户注意：

- 区分 recurring 和 one-shot。
- 区分 durable 和 session-only。
- durable teammate cron 会被直接拒绝，避免重启后形成孤儿任务。
- 任务数量有硬上限。

## 不该误用的邻近能力

### 不要把多代理当默认选项

只有当任务边界真的能拆清时，并行才带来净收益。

### 不要让多个执行体共用同一改动面

一旦任务会互相改同一片代码，应优先引入 worktree 或更严格分工。

### 不要把后台任务当黑箱

Claude Code 已经提供 `/tasks`、`TaskOutput`、`TaskStop`，应把它们纳入工作流。

### 不要把 durable 调度当免费自动化

它意味着行为会跨会话存活，应只在用户明确要求时使用。

## 稳定面与灰度面

### 稳定或已明显成形

- AgentTool
- Task tools
- `/tasks`
- worktree 隔离

### 灰度或条件面

- team/swarm
- cron / remote trigger
- coordinator mode
- 更深的 KAIROS / proactive 自治循环

## 对用户的策略结论

并行工作的稳妥顺序通常是：

1. 先在主线程把任务定义清楚。
2. 再决定是否拆给子代理。
3. 真并行时优先给 worktree 或其他隔离面。
4. 用任务系统看住后台执行，而不是靠记忆。
