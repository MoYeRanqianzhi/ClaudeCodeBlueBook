# 扩展与集成专题

## 用户目标

让 Claude Code 获得项目特定工作流、外部系统能力和运行时自动化，而不把所有要求都硬编码进一次性 prompt。

## 正确入口

### 技能

当你只是想固定某套工作流时，优先技能。

技能能声明：

- 说明文本。
- 工具白名单。
- 模型偏好。
- `inline` 或 `fork`。
- hooks。
- 附带参考文件。

这比“写个长 prompt 模板”更接近正式运行时扩展。

### 插件

当你需要把多条技能、hooks、MCP 服务器打成可启停扩展包时，优先插件。

`/plugin` 本质上是扩展包管理面，而不仅是安装器。

### MCP

当你要连接外部系统，优先 MCP。

`services/mcp/client.ts` 的体量本身就说明 MCP 不是附带功能，而是大型子系统，负责：

- 连接 transport。
- OAuth / auth 缓存。
- 工具暴露。
- resource 暴露。
- 输出截断与持久化。
- 权限与状态管理。

### Hooks

当你想在运行时关键事件上插入外部脚本时，用 hooks。

Hooks 适合：

- prompt 前后加工。
- 工具调用前后扩展。
- task / session 生命周期事件。

## 不该误用的邻近能力

### 不要用插件代替技能

如果只是固化一个工作流，插件层级太重。

### 不要用 MCP 代替本地工具

如果能力原生已经有，优先原生工具，因为它们更稳定、更可缓存、更好治理。

### 不要把 hooks 当“隐式魔法”

Claude Code 已把 hooks 做成正式运行时接口，应明确记录用途和触发点。

## 稳定面与灰度面

### 稳定面

- `/skills`
- `/plugin`
- `/mcp`
- `/hooks`
- bundled skills 基础集合

### 灰度或条件面

- MCP skills
- 某些内置插件
- Claude API / Chrome / 远程调度类 skills
- 更深的 marketplace / official registry 行为

## 对用户的策略结论

扩展决策树很简单：

1. 固化流程，用技能。
2. 打包多能力，用插件。
3. 接外部系统，用 MCP。
4. 拦截运行时事件，用 hooks。
