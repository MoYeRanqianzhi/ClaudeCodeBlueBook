# 连续性与记忆专题

## 用户目标

让一次开发工作不是“聊完即散”，而是能够：

- 在长任务中不中断地继续。
- 在离开后恢复现场。
- 在上下文变长后压缩继续。
- 把稳定偏好沉到记忆层。

## 正确入口

### 上下文观察

先看 `/context` 和 `/files`。

这是为了确认“当前共同工作现场”到底包含什么，而不是靠感觉判断是否该继续喂信息。

### 记忆管理

用 `/memory` 处理长期有效的偏好、规则和项目事实。

源码里的 `claudemd.ts` 说明记忆层不是单文件，而是分层加载：

- managed memory
- user memory
- project memory
- local memory

而且支持 `@include`。

### 压缩

用 `/compact`。

重要的不是“减少字数”，而是 Claude Code 会先做：

- compact boundary 处理。
- session memory compaction 尝试。
- microcompact。
- 再进入传统或 reactive compact 路径。

也就是说，它在尽量保留继续行动所需的结构，而不是随便生成一段摘要。

### 恢复

用 `/resume`。

`resume.tsx` 说明恢复逻辑考虑了：

- 同仓库工作树。
- 跨项目限制。
- 自定义标题匹配。
- session id 直达。
- lite log 与 full log。

所以 `/resume` 本质上是一个“恢复工作对象”的 UI，不只是聊天历史搜索器。

## 不该误用的邻近能力

### 不要把记忆当聊天备忘录

记忆适合长期稳定约束，不适合每回合临时任务状态。

### 不要把 `/compact` 当作清屏

它是连续性工具，不是 UI 整理工具。

### 不要把 `/resume` 当跨项目万能跳板

源码明确区分同目录、同仓库 worktree、不同项目三类恢复语义。

## 稳定面与灰度面

### 稳定面

- `/context`
- `/files`
- `/memory`
- `/compact`
- `/resume`
- `/rename`
- `/tag`
- `/export`

### 灰度或条件面

- 自动 session memory。
- 某些 reactive compact 或 cached microcompact 分支。
- 远程环境中的恢复补写逻辑。

## 对用户的策略结论

最稳的连续性工作流通常是：

1. 用 `/context` 看现场。
2. 用 `/memory` 沉淀长期规则。
3. 用 `/compact` 处理长任务。
4. 用 `/resume` 返回任务，而不是新开会话重讲。
