# 叶子模块、扼流点与循环依赖切断法

这一章回答五个问题：

1. 为什么 Claude Code 同时存在热点大文件和大量小而纯的 leaf module。
2. 哪些 chokepoint 最体现“复杂性收口”。
3. 哪些 leaf module 最体现“依赖图自觉”。
4. 为什么这两种力量并不矛盾，反而是同一套工程方法。
5. 对平台构建者来说，最值得迁移的结构习惯是什么。

## 0. 代表性源码锚点

- `claude-code-source-code/src/state/onChangeAppState.ts:43-92`
- `claude-code-source-code/src/tools.ts:185-367`
- `claude-code-source-code/src/utils/toolPool.ts:43-79`
- `claude-code-source-code/src/utils/plugins/pluginPolicy.ts:1-20`
- `claude-code-source-code/src/services/mcp/channelAllowlist.ts:1-76`
- `claude-code-source-code/src/services/mcp/normalization.ts:1-23`
- `claude-code-source-code/src/state/teammateViewHelpers.ts:5-21`
- `claude-code-source-code/src/services/api/promptCacheBreakDetection.ts:28-158`

## 1. 先说结论

Claude Code 的结构设计里一直有两种看似相反、实际互补的力量：

1. 用少数 chokepoint 集中维护全局不变量。
2. 用大量 leaf module 主动切断依赖图和循环引用。

前者解释了为什么会出现：

- `query.ts`
- `main.tsx`
- `attachments.ts`
- `tools.ts`

这类热点文件。

后者解释了为什么公开镜像里又反复出现：

- `pluginPolicy.ts`
- `normalization.ts`
- `toolPool.ts`
- `teammateViewHelpers.ts`

这类刻意缩小、刻意纯化的模块。

所以 Claude Code 的工程方法不是：

- 所有东西都拆小

也不是：

- 所有复杂性都堆到一个超大文件

而是：

- 让不变量收口，让依赖图变浅

## 2. chokepoint：把全局不变量集中到少数入口

### 2.1 `onChangeAppState(...)`

源码把它直接称作：

- single choke point for CCR/SDK mode sync

作者解释得很清楚：

- 原先 8+ 条 mutation path 里，只有 2 条会通知 CCR
- 其余路径会让 `external_metadata.permission_mode` stale

最后的做法不是给 8+ 个调用点分别补通知，而是：

- 把模式变化统一收口到 AppState diff 处

这就是典型的 chokepoint 思维：

- 找到全局不变量真正该被维护的位置

### 2.2 `tools.ts`

`getAllBaseTools()`、`getTools()`、`assembleToolPool()` 明确承担了：

- base tool truth
- mode filtering
- deny-rule filtering
- built-in / MCP 合并
- prompt-cache-stable 排序

这意味着工具可见性的真相没有散落到 REPL、SDK、AgentTool、host adapter 各处，而是被收敛到一个统一入口。

### 2.3 `promptCacheBreakDetection.ts`

这也是一种更隐蔽的 chokepoint：

- 它把 system hash、tools hash、cache control、betas、effort、extra body 等变化统一收口成 cache break 诊断对象

这说明作者不是等“用户感觉 prompt 变笨了”再猜原因，而是：

- 让 prompt 稳定性本身也有一个统一观测入口

## 3. leaf module：主动切断依赖图

### 3.1 `pluginPolicy.ts`

源码第一行就直说：

- kept as a leaf module
- only imports settings
- to avoid circular dependencies

而且它还被定义成 policy blocking 的 single source of truth。

这很值得研究，因为它说明作者不是为了“文件好看”才拆小，而是为了：

- 在高耦合子系统里保住一条可复用、可被多处依赖的纯判断路径

### 3.2 `normalization.ts`

这里更干脆：

- no dependencies to avoid circular imports

像 server name normalization 这种看似很小的逻辑，被明确抽成无依赖纯模块，是因为：

- 一旦它被许多 MCP 相关模块共用，就必须尽量不回咬整个依赖图

### 3.3 `toolPool.ts`

作者刻意把它放进 React-free 文件，原因也很直接：

- `print.ts` 需要导入
- 但不能被迫把 `react/ink` 整条模块图拉进 SDK path

所以它不是普通 util，而是：

- 为了模块图隔离而存在的 leaf module

### 3.4 `teammateViewHelpers.ts`

这里的注释也很典型：

- inlined from framework.ts
- importing creates a cycle
- inline type check instead of importing

这说明作者对循环依赖的处理不是“出了再说”，而是：

- 连一个类型守卫都愿意内联，只为切断 runtime edge

## 4. 为什么 chokepoint 和 leaf module 必须一起出现

如果只有 chokepoint，没有 leaf module，结果常常是：

- 中心入口越来越胖
- 任何修改都把更多依赖拖进来

如果只有 leaf module，没有 chokepoint，结果常常是：

- 每个 feature 都局部正确
- 全局不变量没有统一维护点

Claude Code 更成熟的地方在于同时接受：

1. 有些复杂性必须集中。
2. 集中之后又要尽量把外围依赖图削薄。

这就是为什么它会呈现出一种很典型的结构：

- 少数厚底盘
- 周围大量纯判断或纯变换小模块

## 5. 热点文件与叶子模块化并存，恰恰说明作者在偿债

如果只看：

- `query.ts`
- `main.tsx`
- `attachments.ts`

会很容易得出“文件太大”的结论。

但结合这些 leaf module 再看，判断就更准确：

- 作者知道复杂性需要集中
- 也知道哪些边缘逻辑应该被及时抽离

所以真正值得写进蓝皮书的，不是：

- 这套代码已经没有债

而是：

- 它在用正确方向偿还复杂性债务

## 6. 苏格拉底式追问

### 6.1 为什么不把所有逻辑都尽量贴近 feature 放置

因为有些逻辑一旦分散到 feature 层，就会很快失去：

- 全局一致性
- cache 稳定性
- 状态同步真相

### 6.2 为什么不把所有 chokepoint 都继续拆碎

因为有些复杂性天然就是全局性的。

比如：

- mode sync
- tool pool truth
- cache break diagnosis

这些东西如果完全拆散，反而会让不变量无人负责。

### 6.3 为什么 leaf module 不是“洁癖”，而是产品能力

因为 leaf module 真正保护的是：

- 模块图稳定性
- 可复用判断入口
- 重构摩擦

这些最后都会影响产品能否继续稳定演化。

## 7. 一句话总结

Claude Code 的工程先进性，不只在于有少数维护全局不变量的 chokepoint，还在于它有意识地用 leaf module 和依赖切断法保护这些 chokepoint 不被模块图反向拖垮。
