# 安全分层、策略收口与沙箱边界

这一章回答五个问题：

1. Claude Code 的安全设计为什么不等于 permission prompt。
2. 启动、权限、文件系统、sandbox、hooks、policy、plugin、MCP auth 分别承担什么边界。
3. 为什么作者反复做“收口”而不是“相信模型会自觉”。
4. 这些边界如何共同构成 fail-closed 的系统。
5. 这套设计透露了怎样的安全观。

## 1. 先说结论

Claude Code 的安全面至少有七层：

1. trust / init 边界
2. permission mode / rule / classifier 边界
3. 文件系统与危险路径边界
4. sandbox 与网络 allowlist 边界
5. hook / HTTP SSRF 边界
6. policy / plugin trust 边界
7. MCP auth / remote transport trust 边界

这说明它真正做的是“安全分层 + 策略收口”，而不是“弹窗问一次用户”。
permission modal 只是 `ask` 结果的一种 renderer，不是安全本体。

## 2. 第一层：trust 先于完整能力

`init.ts` 明确先做：

- safe env vars
- extra CA certs

而完整环境变量与 telemetry 初始化放在 trust 之后。

这说明 Claude Code 的第一层安全边界不是工具调用，而是：

- 在用户尚未信任前，进程能加载什么配置、拿到什么环境

证据：

- `claude-code-source-code/src/entrypoints/init.ts:71-79`
- `claude-code-source-code/src/entrypoints/init.ts:241-245`

继续沿着这条线往下看，`TrustDialog`、hooks 和 MCP `headersHelper` 共同表明：

- 项目目录里的可执行配置
- 会影响联网行为的头部辅助器
- 会拦截 prompt / tool 的 hooks

都不是“只要文件存在就启用”，而是先受 workspace trust 约束。

证据：

- `claude-code-source-code/src/components/TrustDialog/TrustDialog.tsx:208-208`
- `claude-code-source-code/src/utils/hooks.ts:268-268`
- `claude-code-source-code/src/utils/hooks.ts:1992-1992`
- `claude-code-source-code/src/services/mcp/headersHelper.ts:40-40`

## 3. 第二层：permission mode 的本质是策略收口

`permissionSetup.ts` 最重要的部分不是 UI，而是“危险 allow rule 检测与剥离”：

- Bash / PowerShell 的宽泛执行规则
- Agent 的过宽 delegation 规则
- 进入 auto mode 时 strip dangerous rules
- 离开 auto 时再恢复

这说明 auto mode 的安全性不来自 classifier 单点，而来自：

1. 先把绕过分类器的路径删掉
2. 再让 classifier 决策

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:84-245`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:469-579`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:1457-1532`

再往下看 `permissions.ts` 与 `types/permissions.ts`，会发现权限并不是布尔量，而是一套 typed decision engine：

- mode
- rule source
- destination
- decision reason

这也是为什么同一份安全语义能跨 CLI、bridge、SDK、channel relay 复用。

证据：

- `claude-code-source-code/src/types/permissions.ts:16-271`
- `claude-code-source-code/src/utils/permissions/permissions.ts:1158-1262`

## 4. 第三层：文件系统边界不是只看 cwd

`filesystem.ts` 明确做了大量防御：

- path normalization，防 `..` traversal
- dangerous directory / dangerous file 检测
- symlink / original path 双路径检查
- sandbox write allowlist 对照

这说明 Claude Code 不把“当前工作目录”当成天然安全区，而是持续检查：

- 这个路径是不是配置面
- 是不是危险目录
- 是不是用户允许的写入面

证据：

- `claude-code-source-code/src/utils/permissions/filesystem.ts:250-286`
- `claude-code-source-code/src/utils/permissions/filesystem.ts:427-480`

- `claude-code-source-code/src/utils/permissions/pathValidation.ts:226-242`

## 5. 第四层：managed policy 是独立治理平面

`policySettings` 不只是普通 settings source 的一员，而是更高一级的治理面：

- first source wins
- managed-only hooks
- managed-only permission rules
- managed-only MCP servers

而远程下发的 managed settings 若引入危险可执行项，还要再过一个单独的安全批准层。

这说明 Claude Code 把“组织治理”单独建模了，而不是让它混在本地设置里。

## 6. 第五层：sandbox 不是可有可无的附属

`sandbox-adapter.ts` 展现的是一整套边界工程：

- policy 可要求 `allowManagedDomainsOnly`
- 会把危险写路径显式 deny
- 会阻断 settings.json 等逃逸面
- 支持 failIfUnavailable / enabledPlatforms / excludedCommands
- 动态刷新配置

这说明 sandbox 在 Claude Code 里并不是“能开就开”，而是：

- 被纳入 settings / policy / command permission 的正式边界层

证据：

- `claude-code-source-code/src/utils/sandbox/sandbox-adapter.ts:149-266`
- `claude-code-source-code/src/utils/sandbox/sandbox-adapter.ts:449-607`
- `claude-code-source-code/src/utils/sandbox/sandbox-adapter.ts:645-694`

## 7. 第六层：hook 也被当成攻击面治理

`ssrfGuard.ts` 明确阻止 HTTP hooks 访问：

- 私网
- link-local
- cloud metadata endpoint

而且注释还明确写了：

- 有 proxy 时 guard 可能被绕开
- sandbox proxy 再承担自己的 allowlist

这说明作者没有把 hooks 当成“只是本地自动化”，而是按潜在攻击面来治理。

证据：

- `claude-code-source-code/src/utils/hooks/ssrfGuard.ts:5-17`
- `claude-code-source-code/src/utils/hooks/ssrfGuard.ts:40-111`

## 8. 第七层：policy / plugin trust 明确分域

`pluginOnlyPolicy.ts` 直接把 trust domain 写进代码：

- plugin / policySettings / built-in / bundled = admin-trusted
- userSettings / projectSettings / localSettings / flagSettings = user-controlled

而 `policyLimits` 服务又在组织层做 feature restriction，并且：

- API fail open
- ETag caching
- background polling

这说明 Claude Code 的安全设计不是一味收死，而是在：

- 谁可信
- 哪层配置可信
- 限制从哪一层下发

这三个问题上做清晰分域。

证据：

- `claude-code-source-code/src/utils/settings/pluginOnlyPolicy.ts:1-58`
- `claude-code-source-code/src/services/policyLimits/index.ts:1-19`

## 9. 第八层：MCP auth 也被视为独立 trust domain

`mcp/auth.ts` 的注释已明确指出：

- MCP auth server 与 org IdP 不一定是同一个 trust domain
- 某些 server 需要用户批准后连接
- token / metadata discovery / refresh 都要单独处理

这说明作者没有把“连上外部系统”写成一条通用 happy path，而是承认：

- 每个外部系统都可能带来独立的认证与信任问题

## 10. 为什么说它一直在“收口”

Claude Code 的安全逻辑一个共同特征是：

- 不把强能力完全删掉
- 但不断收紧它们的可达路径

具体表现为：

1. trust 前只放 safe env
2. auto mode 前先 strip dangerous permissions
3. sandbox 再收文件系统和网络
4. hooks 再收 SSRF
5. policy / plugin trust 再收配置来源
6. MCP auth 再收外部连接

这就是“策略收口”的真正含义。

## 11. 一句话总结

Claude Code 的安全强，不在于它权限弹窗多，而在于它把 trust、permission、filesystem、sandbox、hooks、policy、external auth 做成了一层层彼此补位的收口系统。
