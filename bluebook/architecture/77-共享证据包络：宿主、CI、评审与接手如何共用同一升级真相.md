# 共享证据包络：宿主、CI、评审与接手如何共用同一升级真相

这一章回答五个问题：

1. 为什么统一 rollout ABI 和证据真相面还不够，系统还必须长出 shared evidence envelope。
2. 为什么宿主、CI、评审与接手者虽然消费方式不同，却应该共用同一份升级真相。
3. 这份 evidence envelope 在 Claude Code 源码里究竟由哪些运行时平面共同提供。
4. 为什么没有这层包络，Prompt 魔力、安全/省 token 与源码先进性都会重新退回各说各话。
5. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 更成熟的一点，不只是它有：

1. 统一 rollout ABI
2. 升级证据真相面
3. 宿主可消费状态面

还在于这些东西其实已经能被继续收口成同一份：

- shared evidence envelope

这份包络至少包含五类真相：

1. 当前对象真相
2. 当前状态真相
3. 当前 diff 真相
4. 当前决策窗口真相
5. 当前回退边界真相

宿主、CI、评审与接手者消费的方式不同，但真正该共享的，就是这五类真相，而不是四套彼此平行的材料。

## 2. 宿主、CI、评审与接手为什么会分裂

如果系统没有 shared evidence envelope，最常见的分裂会长成这样：

1. 宿主看：
   - `worker_status`
   - `external_metadata`
2. CI 看：
   - 某些预算阈值
   - 某些 diff 报警
3. 评审看：
   - 文字总结
   - rollout 卡片
4. 接手者看：
   - transcript
   - 旧提交

四者都在看“同一次升级”，却并不消费同一份真相。

这样做的结果不是“信息更多”，而是：

- 判断主权再次分叉

所以 shared evidence envelope 的第一性原理不是：

- 报表合并

而是：

- 把不同消费者重新拉回同一份 authoritative upgrade truth

## 3. 共享包络的第一层：当前对象与当前状态

`sessionState.ts`、`onChangeAppState.ts`、`coreSchemas.ts` 和 `remoteIO.ts` 共同构成 shared envelope 的第一层：

1. 对象当前是什么。
2. 当前卡在哪。
3. 当前模式是什么。
4. 当前有没有 pending action。
5. 当前长任务摘要是什么。

宿主直接消费这些 signals。

CI、评审和接手者虽然不一定直接看运行时流，但他们应该消费这些 signals 的持久化结果，而不是重新猜“当时大概在什么状态”。

也就是说：

- shared envelope 的第一层，是可恢复的当前真相

## 4. 第二层：编译后请求真相，而不是原始文案

`promptCacheBreakDetection.ts` 与 `normalizeMessagesForAPI(...)` 说明：

- Prompt 相关证据真正该共享的，不是某段原文，而是编译后的 request truth

这包括：

1. system / tools / model 的关键字节
2. normalized messages
3. tool pairing 修复与裁剪
4. betas / effort / extra body params

对宿主来说，这些信息可能是 cache break summary 或 diff 引用。

对 CI 来说，这些信息可能是门禁或阈值检查。

对评审来说，这些信息可能是最小 diff 与解释。

对接手者来说，这些信息决定：

- 为什么这次 prompt 升级是成立的

所以 shared envelope 的第二层，是：

- compiled request truth

## 5. 第三层：决策窗口真相，而不是结果统计

`tokenBudget.ts` 与 `QueryGuard.ts` 说明：

- 继续、停止、升级对象、撤回自动化，都不是事后感想，而是正式决策窗口

这层真相至少回答：

1. 当前继续是否仍有增益。
2. 当前判断窗口是否已经结束。
3. 旧 finally / 旧 reservation 是否还有资格回写。

对宿主来说，这是：

- 当前 UI/bridge 应该继续显示什么、等待什么

对 CI 来说，这是：

- 当前门禁是否应该失败或要求对象升级

对评审来说，这是：

- 当前判断是基于哪一段 observed window 做出的

对接手者来说，这是：

- 现在是该继续，还是该换对象、回退或停止

所以 shared envelope 的第三层，是：

- decision-window truth

## 6. 第四层：控制仲裁真相

Claude Code 的治理升级不能只看结果。

还必须看：

1. `request_id`
2. `tool_use_id`
3. 当前 `pending_action`
4. 谁赢下仲裁
5. 是否发生 cancel / response race
6. 等待时间

因为安全与省 token 的成熟，本质上是在治理：

- 谁先判断
- 何时停止
- 何时交还主权

所以 shared envelope 的第四层不是：

- 有无审批

而是：

- control-arbitration truth

## 7. 第五层：回退对象真相

`sessionIngress.ts`、`task/framework.ts` 和相关恢复治理代码说明：

- 失败时真正要共享的，不是“改过哪些文件”，而是“该回退哪个对象”

这层真相至少包括：

1. rollback object type
2. rollback object id
3. authority surface
4. retained assets
5. dropped stale writers

这对不同消费者同样重要：

1. 宿主需要知道恢复目标。
2. CI 需要知道该阻断哪类推广。
3. 评审需要知道这次回退是否过宽或过窄。
4. 接手者需要知道从哪里继续，而不是从哪里重读。

所以 shared envelope 的第五层，是：

- rollback-boundary truth

## 8. 为什么这会统一 Prompt、安全与源码先进性

Prompt 魔力最终要求：

- 编译后请求真相可解释

安全与省 token 最终要求：

- 决策窗口与仲裁真相可复查

源码先进性最终要求：

- 回退对象与恢复资产可指认

看起来是三条线，实则都在要求同一件事：

- 不同消费者必须共享同一份升级判断真相

所以 shared evidence envelope 不是附属层，而是：

- 这三条线的共同落点

## 9. 第一性原理

一个成熟 runtime 不应只回答：

- 这次跑出来了什么

还必须回答：

- 不同消费者为什么会对这次升级得出同一判断

而让不同消费者得出同一判断的最小前提，就是：

- 他们共享同一套 evidence envelope

否则系统每增长一层，就会多长出一套平行解释。

## 10. 一句话总结

Claude Code 的 shared evidence envelope，本质上是在让宿主、CI、评审与接手者围绕同一份“对象、状态、diff、决策窗口与回退边界”继续判断，而不是围绕四套不同材料各自做局部最优。
