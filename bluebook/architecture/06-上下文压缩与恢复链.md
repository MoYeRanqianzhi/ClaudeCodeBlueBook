# 上下文压缩与恢复链

## 1. 先说结论

Claude Code 对“长上下文”采取的不是单一方案，而是一条串联链：

1. tool result budget
2. snip
3. microcompact
4. context collapse
5. autocompact
6. blocking limit / retry / fallback
7. reactive compact
8. max_output_tokens recovery

关键证据：

- `claude-code-source-code/src/query.ts:365-468`
- `claude-code-source-code/src/query.ts:592-648`
- `claude-code-source-code/src/query.ts:1065-1175`
- `claude-code-source-code/src/query.ts:1185-1305`

这说明 Claude Code 把“上下文管理”视为主循环的一部分，而不是超长会话崩了以后再补救。

## 2. 第一层：tool result budget

进入压缩链之前，Claude Code 先控制 aggregate tool result size。  
这一步运行在 microcompact 之前，并且支持把 content replacement 持久化到 transcript 相关路径。

证据：

- `claude-code-source-code/src/query.ts:369-394`

这很关键，因为 Claude Code 先削减“工具输出噪音”，再考虑“消息级压缩”。

## 3. 第二层：snip

`snip` 发生在 microcompact 之前，并且会显式记录 `tokensFreed`，让后面的 autocompact 阈值判断知道它已经帮忙释放了多少空间。

证据：

- `claude-code-source-code/src/query.ts:396-410`

这说明 snip 不是视觉裁剪，而是上下文预算链条中的正式信号。

## 4. 第三层：microcompact

microcompact 在 autocompact 之前执行，且支持缓存编辑与延迟 boundary message。

证据：

- `claude-code-source-code/src/query.ts:412-426`

它的角色不是“生成一个大摘要”，而更像细粒度、小步长的上下文整理器。

## 5. 第四层：context collapse

如果启用了 context collapse，它会在 autocompact 之前先投影 collapsed view，并尝试 commit 更多 collapse。

证据：

- `claude-code-source-code/src/query.ts:428-447`

注释明确写了一个重要意图：  
如果 collapse 已经足够把上下文压到阈值以下，就没必要再 autocompact 成一大块 summary。

这说明 Claude Code 更偏好“尽量保留颗粒度”，而不是盲目大摘要。

## 6. 第五层：autocompact

`deps.autocompact(...)` 才是最重的自动压缩路径。

成功时它会：

- 产出 compactionResult
- 记录 pre/post token count
- 更新 autoCompactTracking
- 构造 postCompactMessages

证据：

- `claude-code-source-code/src/query.ts:453-543`

失败时则会带着 consecutiveFailures 往后走，交给 circuit-breaker 或下一轮处理。

## 7. 第六层：blocking limit 与 proactive 拦截

在真正发模型请求前，还会检查 blocking limit。  
如果已经碰到硬阻塞阈值，就直接报 `PROMPT_TOO_LONG_ERROR_MESSAGE` 并返回。

但这里有很多跳过条件：

- 刚 compact 完
- querySource 是 compact/session_memory
- reactive compact enabled
- context collapse 拥有恢复权

证据：

- `claude-code-source-code/src/query.ts:592-648`

这说明 Claude Code 不是“到阈值就一刀切”，而是先判断当前是否还有更合理的恢复路径。

## 8. 运行中的 fallback 与 streaming 修复

进入模型流式阶段后，Claude Code 还要处理：

- streaming fallback
- orphaned messages tombstone
- tool_use / tool_result 对齐
- withheld recoverable errors

证据：

- `claude-code-source-code/src/query.ts:657-950`

尤其是 fallback 这一段很能说明设计风格：  
不是简单“失败重试”，而是显式清理旧 assistant messages、tool results、executor 状态，再换 fallback model 重新来。

## 9. withheld error 是恢复链的核心手法

对于一些可恢复错误，Claude Code 不会立刻把错误吐给用户，而是先“暂扣”：

- prompt too long
- media size error
- max_output_tokens

证据：

- `claude-code-source-code/src/query.ts:788-825`

这么做的目的只有一个：  
先尝试恢复，再决定是否把错误真正暴露出来。

## 10. prompt-too-long 的恢复链

如果最后一条消息是 withheld 413：

1. 先尝试 `contextCollapse.recoverFromOverflow(...)`
2. 如果还不够，再尝试 `reactiveCompact.tryReactiveCompact(...)`
3. 都不行，才真正 surface 错误

证据：

- `claude-code-source-code/src/query.ts:1065-1118`
- `claude-code-source-code/src/query.ts:1119-1175`

这条链很值得注意：  
Claude Code 先尝试保留更多细节的恢复方式，再退到更重的 summary/strip 方案。

## 11. max_output_tokens 的恢复链

`max_output_tokens` 不是简单“报错结束”。

Claude Code 至少有两段恢复：

### 11.1 升级 token cap

如果用了 capped default，并且 gate 开启，就先把同一请求从较小 cap 提升到更大 cap 重试。

证据：

- `claude-code-source-code/src/query.ts:1188-1220`

### 11.2 注入 meta recovery message

如果还没超过 recovery limit，就追加一条 meta user message，让模型直接续写，不道歉、不回顾，继续拆剩余工作。

证据：

- `claude-code-source-code/src/query.ts:1223-1252`

只有恢复次数耗尽，才真正 surface 错误。

## 12. stop hook 与恢复链的相互约束

这一段非常细。  
Claude Code 明确避免让 stop hooks 和恢复链互相打架：

- API error 时不跑正常 stop hooks，避免死循环
- stop hook blocking error 重试时保留 `hasAttemptedReactiveCompact`

证据：

- `claude-code-source-code/src/query.ts:1258-1265`
- `claude-code-source-code/src/query.ts:1282-1305`

这说明作者已经踩过“压缩 -> 错误 -> hook -> 重试 -> 再压缩”的循环坑，并在状态上明确防止它。

## 13. 从第一性原理看，这条链解决了什么

它解决的不是“怎么塞更多 token”，而是：

1. 如何在尽量保留细节的前提下继续工作。
2. 如何把恢复变成默认路径，而不是人工救火。
3. 如何避免恢复链和 hooks/fallback/streaming executor 互相踩踏。

所以 Claude Code 的上下文观不是“更大窗口”，而是“上下文工作集的动态管理”。

## 14. 一句话总结

Claude Code 的压缩恢复链像一个多级缓存回收与故障恢复系统：先细粒度削减，再重压缩，再模型 fallback，再 meta recovery；真正的强点不是上下文大，而是上下文快崩时还能继续工作。
