# 源码即治理界面：注释、命名与显式边界如何降低误改成本

这一章回答五个问题：

1. 为什么 Claude Code 的源码不仅是执行对象，也是治理界面。
2. 为什么注释、命名、dependency-free leaf module、显式 snapshot 都应该被看成治理工具。
3. 为什么这套代码同时在服务机器执行和人类维护。
4. 为什么“更好读”在这里不是审美，而是防误改、防误扩展、防失忆。
5. 这对 agent runtime 设计者意味着什么。

## 0. 代表性源码锚点

- `claude-code-source-code/src/utils/systemPromptType.ts:1-12`
- `claude-code-source-code/src/services/mcp/normalization.ts:1-20`
- `claude-code-source-code/src/utils/queryContext.ts:1-40`
- `claude-code-source-code/src/utils/toolSchemaCache.ts:1-20`
- `claude-code-source-code/src/utils/hooks/hooksConfigSnapshot.ts:1-120`
- `claude-code-source-code/src/query/config.ts:8-20`

## 1. 先说结论

Claude Code 的源码有一个很容易被低估的先进点：

- 它的代码本身就是治理界面

也就是说，作者不满足于：

- 系统能正确运行

还要求：

- 维护者更难误读
- 改动者更难误改
- 后来者更难把边界偷偷扩散

所以在这套仓库里：

- 注释
- 命名
- 显式边界
- dependency-free leaf modules

都不是附属物，而是治理工具。

## 2. 命名在这里承担风险标注

像：

- `DANGEROUS_uncachedSystemPromptSection`
- `authoritative`
- `single source of truth`
- `coalescePatches`

这类命名最重要的作用，不是优雅，而是：

- 让误用风险在阅读阶段就暴露出来

`systemPromptType.ts` 甚至专门用 branded type 和 dependency-free module 来保证：

- system prompt 的形状和 import 风险都被同时治理

这说明命名与类型在这里是治理界面的一部分。

## 3. 注释在这里不是教学，而是制度记忆

`query/config.ts` 直接写：

- future step() extraction tractable

`toolSchemaCache.ts` 直接写：

- 为什么放 leaf module
- 为什么会引发 cycle

`hooksConfigSnapshot.ts` 直接写：

- 哪些来源应被允许
- 为什么要 reset cache 拿一致快照

这些注释的价值不是教新人“这段代码在干嘛”。

它们更像在记录：

- 为什么以后不要随便改成别的样子

也就是说，注释在这里承担的是：

- 制度记忆

## 4. dependency-free leaf module 是“可审计的最小规则单元”

`systemPromptType.ts`、`normalization.ts` 这类小文件共同体现一条纪律：

- 高扇入规则应尽量放在 dependency-free 的极薄单元里

这样做的价值不仅是切循环。

还在于：

- 维护者可以一眼审计一条规则
- 不必把重模块图一起拉进来
- 规则边界不容易被无意扩大

这是一种非常成熟的治理取向：

- 让共享规则尽可能短、薄、可读、可审

## 5. snapshot 与显式边界说明时间一致性也被做成了可读制度

`hooksConfigSnapshot.ts` 和 `queryContext.ts` 都很能说明这一点。

它们不只在做快照或 helper。

它们还在显式告诉维护者：

- 什么时刻的真相算准
- 哪些调用者可以读这份真相
- 为什么某个 helper 只能由入口层导入

这说明源码的可读性不是静态结构问题，而是：

- 时间一致性与依赖一致性被写成了可见制度

## 6. 为什么这套源码同时服务机器执行和人类维护

机器需要：

- 类型
- 状态机
- 合并规则
- 明确协议

人类需要：

- 风险标注
- 边界解释
- 依赖动机
- 改动警告

Claude Code 的高级感在于，它没有把这两套表达分裂开。

同一段源码往往同时做到：

- 对机器可执行
- 对人类可治理

这也是为什么它在增长中仍能保住可维护性。

## 7. 一句话总结

Claude Code 的源码之所以显得成熟，不只因为它结构对，而是因为它把注释、命名和显式边界也做成了治理界面，让代码本身持续降低误改和误扩展成本。
