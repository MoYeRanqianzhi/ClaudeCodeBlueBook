# 公开源码镜像的先进性、热点与技术债

这一章回答五个问题：

1. 为什么这份公开源码镜像仍然会让人觉得“工程上很先进”。
2. 这种先进性在当前可见代码里主要体现在哪里。
3. 这份公开镜像自身又有哪些明显局限和技术债。
4. 为什么“先进性”和“热点大文件”可以同时成立。
5. 对平台构建者来说，最值得迁移的到底是什么。

## 1. 先说结论

这份公开源码镜像的先进性，主要不在“功能很多”，而在三件事：

1. 关键约束被显式写成结构对象。
2. 失败、恢复、cache、权限都被工程化，而不是散落在调用点。
3. 即使公开包不完整，仍能看出作者在主动维护 runtime invariants。

但同时也必须承认三类限制：

1. 公开源码不是完整 monorepo，缺失 `108` 个 gate 相关模块。
2. 少数热点文件依然极大，局部阅读成本高。
3. 当前公开树缺少可见测试面，因此对工程质量的判断必须聚焦“结构质量”，不能夸大为“测试闭环已证实”。

## 2. 先进性一：关键约束被对象化

在可见源码里，很多团队只会写进注释或约定的东西，被 Claude Code 做成了正式结构：

- prompt dynamic boundary
- permission decision union
- tool pool merge + cache-stable ordering
- budget tracker
- worker status sync

这类对象化的价值是：

- 新能力只能沿着既有约束接入
- 不容易在快速扩展时把 invariants 写散

## 3. 先进性二：复杂性被集中在基础设施，而不是复制到产品层

`tools.ts`、`types/permissions.ts`、`query.ts` 展示出一种很清晰的工程偏好：

- 把通用控制逻辑收进统一层
- 让产品层沿着这些统一层扩展

例如：

- 工具池排序和 deny 过滤在统一入口完成。
- 权限决策使用显式 union，而不是到处返回布尔值。
- token budget continuation 被做成独立逻辑，而不是在各个调用点临时拼接。

这就是为什么它看起来不像“到处有聪明技巧”，而像“有底盘的系统”。

## 4. 先进性三：大文件不等于缺乏架构

`query.ts`、`main.tsx`、`attachments.ts` 都很大，这是事实。

但它们“大”的原因并不完全一样：

- `query.ts` 是 turn runtime kernel，天然承担恢复、自循环、预算与错误分支。
- `main.tsx` 是装配总控层，天然会承接初始化、策略、远程与前台模式分流。
- `attachments.ts` 是动态状态晚绑定的总入口，天然会吸收多种上下文注入逻辑。

换句话说，热点文件的存在说明：

- 局部还需要继续拆薄

但不自动说明：

- 整体缺乏结构

## 5. 公开镜像的真实技术债

### 5.1 提取树不完整

`README.md` 已明确声明这是不完整的公开源码镜像。

因此所有关于：

- workflow engine 深处
- 内部门控模块
- 某些 worker / assistant / compact 细节

的判断都必须保持边界感。

### 5.2 热点文件阅读成本高

即使结构先进，热点文件依然意味着：

- 新读者上手慢
- 局部改动风险集中
- 细粒度职责还可继续拆薄

### 5.3 可见测试面缺失

当前公开提取树里没有显式可见的测试配置与测试文件。

因此更稳妥的说法应是：

- 当前可验证的是结构质量、注释质量、状态与边界设计质量

而不是：

- 完整测试工程质量

## 6. 为什么它仍然值得研究

因为即使在不完整镜像里，依然能清楚看出三种非常有迁移价值的能力：

1. contract-first：
   - 先写边界，再写功能。
2. runtime-first：
   - 先写循环、恢复、状态，再写交互表面。
3. budget-first：
   - 安全预算和上下文预算都在主路径，而不是补丁层。

这三点比“具体有哪些内部功能没公开”更值得平台构建者学习。

## 7. 苏格拉底式追问

### 7.1 如果没有完整源码，为什么还敢谈先进性

因为我们谈的是：

- 当前可见代码里的结构判断

而不是：

- 对内部全貌的想象

### 7.2 如果热点文件这么大，为什么不直接下结论说架构不好

因为大文件和结构好坏不是同一维度。

更关键的问题是：

- 关键 invariants 是否被集中管理
- 失败与恢复是否有统一语义
- 新能力是否沿统一 contract 接入

### 7.3 如果没有可见测试，为什么还能说它是高质量源码

更准确的说法应是：

- 当前可见的结构质量高，但测试闭环在公开镜像里不可见

这比直接说“高质量且已被充分测试”更严谨。

## 8. 一句话总结

这份公开源码镜像的先进性，来自它把复杂性收进了明确 contract 与 runtime 基础设施；它的局限，则来自热点大文件和公开镜像不完整。两者必须一起写，判断才稳。
