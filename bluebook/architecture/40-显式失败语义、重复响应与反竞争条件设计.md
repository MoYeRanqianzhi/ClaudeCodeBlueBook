# 显式失败语义、重复响应与反竞争条件设计

这一章回答五个问题：

1. 为什么 Claude Code 的成熟度不只体现在成功路径，而更体现在失败路径。
2. 为什么 duplicate response、orphan message、stale snapshot 在 agent runtime 里属于一等问题。
3. Claude Code 怎样用显式错误、去重与 fresh-state patch 防止系统进入假成功。
4. 为什么“局部能跑通”不等于“长会话、远程与多 Agent 下仍然正确”。
5. 这些设计对 agent 平台构建者有什么可迁移价值。

## 1. 先说结论

Claude Code 的可靠性，很大一部分来自三条纪律：

1. 不支持的事情显式报错，不给假成功。
2. 会重复到来的事情显式去重，不给双重处理。
3. 会被并发写坏的状态显式走 fresh-state patch，不给 stale snapshot 回写。

这三条看起来像实现细节，实际上决定了：

- 宿主会不会一直等一个永远不会来的回复
- 多 Agent / remote / bridge 会不会把同一条消息吃两次
- 长任务状态会不会在异步读取后被旧快照“写活”

所以 Claude Code 的工程先进性，不只在“能跑通”，更在：

- 它把全局错误语义和竞态语义都写进了运行时

## 2. 显式失败优先于假成功

`DirectConnectManager` 与 `RemoteSessionManager` 都有同一条原则：

- 遇到不支持的 `control_request` subtype，返回 `control_response.error`

而不是：

- 静默忽略
- 本地不处理但表面看似成功

这是很关键的运行时伦理。

如果不这么做，服务端会出现两种错觉：

1. 以为本地还在处理，继续等待。
2. 以为请求已经生效，其实本地根本没执行。

`bridgeMessaging.ts` 更进一步：

- outbound-only bridge 对可变控制请求直接回 error

作者显式写下原因：

- 要避免 claude.ai 看到“成功了”，但本地其实什么也没发生

这已经不是普通异常处理，而是：

- 把“拒绝假成功”当成产品正确性的一部分

## 3. Duplicate response 与 orphan message 不是边角问题

`StructuredIO` 里两个对象特别关键：

- `pendingRequests`
- `resolvedToolUseIds`

前者维护 request/response 闭环。
后者专门防 late duplicate control_response 再次进入 orphan handler。

源码注释点明了真实风险：

- 重复处理会把 duplicate assistant messages 推进 `mutableMessages`
- 最终触发 API 层 “tool_use ids must be unique” 之类的硬错误

`handleIngressMessage(...)` 也用两层 UUID 集合做防线：

- `recentPostedUUIDs` 防 echo
- `recentInboundUUIDs` 防 replay / redelivery

这说明 Claude Code 在桥接和远程链路上假设的世界不是：

- 网络稳定、消息只来一次

而是：

- echo、重放、时序错位都会发生

## 4. 反竞争条件设计：不要把 stale snapshot 写回 fresh state

`updateTaskState(...)`、`generateTaskAttachments(...)` 与 `applyTaskOffsetsAndEvictions(...)` 这一组代码，是公开镜像里最值得研究的 race-aware case 之一。

核心约束有三条：

1. updater 如果返回同一引用，就不 spread，不制造无意义重渲染。
2. `generateTaskAttachments(...)` 只返回 `outputOffset` patch，不返回完整 task。
3. patch 应用时重新读取 fresh state，重查 running/terminal/notified 条件。

源码把风险写得很直白：

- 如果异步读磁盘后再 spread 整个旧 task，可能会把已经 terminal 的 task 又写回去，直接 zombify

这类代码最能说明 Claude Code 的开发思路：

- 正确性不是“这一行逻辑看起来对”
- 而是“这段逻辑在 await、并发和状态转换之间还能不能保持不变量”

## 5. mailbox 也按竞态系统来设计

`getTeammateMailboxAttachments(...)` 明确处理了三类协作竞争：

1. structured protocol message 不能被当普通上下文附件吞掉
2. leader inbox 不能泄漏到 teammate 视角
3. file mailbox 与 `AppState.inbox` 同一消息可能会撞车，必须去重

源码甚至把 race 场景逐步写出来：

1. 先从文件邮箱读到消息 M
2. `InboxPoller` 又把同一条 M 放进 `AppState.inbox`
3. attachments 侧再次读到 M

如果不去重，前台与 LLM 上下文都会同时出现双份消息。

所以 Claude Code 处理的并不是：

- “让消息能显示出来”

而是：

- 让协作消息即使从两条路径到达，也不会污染真相

## 6. 权限审批也按“first resolver wins”处理

`StructuredIO.createCanUseTool(...)` 还有一个很强的细节：

- PermissionRequest hooks 与 SDK host prompt 并行 race

谁先返回，谁赢。
另一路被 abort 或忽略。

作者这么做，是为了防止：

- hook 慢就把 UI 卡住
- host prompt 已经答复了，后台 hook 又回来改写结果

这再次说明 Claude Code 的设计单位不是一个弹窗，而是：

- 一场多来源审批竞争里的正确仲裁

## 7. 苏格拉底式追问

### 7.1 为什么“能跑通 happy path”远远不够

因为在 agent runtime 里，更常见的不是 happy path，而是：

- 中断
- 重放
- 重复响应
- stale state
- 双来源消息

如果这些路径没有被一等建模，系统越复杂，错误就越隐蔽。

### 7.2 为什么 unsupported subtype 要回 error，而不是 silently ignore

因为 silently ignore 只会制造：

- 服务端挂等
- 宿主误判
- 用户错觉

Claude Code 明显更在意：

- 失败也必须可解释

### 7.3 为什么只回 offset patch，比直接回完整 task 更成熟

因为成熟系统优先考虑的是：

- await 期间状态是否已变化

而不是：

- 当前这次函数调用拿到的快照看起来完整不完整

## 8. 一句话总结

Claude Code 的显式失败、重复响应防护与反竞争条件设计说明：对 agent runtime 来说，真正的成熟度不是“成功路径很顺”，而是“错误、重放和并发下仍不制造假成功与状态污染”。
