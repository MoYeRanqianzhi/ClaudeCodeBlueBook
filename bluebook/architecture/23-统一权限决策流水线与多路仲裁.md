# 统一权限决策流水线与多路仲裁

这一章回答六个问题：

1. Claude Code 的权限系统到底先做什么，后做什么。
2. 为什么 permission dialog 只是最外层表面，而不是内核。
3. `rules`、`mode`、classifier、hooks、SDK prompt、sandbox 各位于哪一层。
4. 同一次权限请求为什么可能经过多路仲裁。
5. fail-open / fail-closed 在哪里发生。
6. 这套设计为什么比“弹窗确认”更像决策引擎。

## 1. 先说结论

Claude Code 的权限系统更适合被理解成一条 typed decision pipeline：

1. typed model：
   - mode
   - rule
   - update
   - decision
   - decision reason
   - `ToolPermissionContext`
2. 初始上下文装配：
   - mode 解析
   - dangerous rule strip / restore
   - policy/trust 限制
3. 规则层：
   - deny rule
   - ask rule
   - tool-specific rule
   - safety check
4. mode 层：
   - bypass
   - acceptEdits / sandbox auto allow
   - always-allowed rule
5. 自动化判定层：
   - auto-mode classifier
   - headless PermissionRequest hooks
6. relay / host 层：
   - bridge
   - channel relay
   - teammate leader approval
   - SDK host
7. 交互层：
   - TUI permission dialog
   - SDK `can_use_tool`
   - custom permission prompt tool
   - teammate mailbox / leader bridge
8. 外围安全壳：
   - sandbox adapter
   - managed settings security check
   - plugin-only policy
   - MCP auth / remote managed policy

所以权限真相不是一个 modal，而是：

- 先算出 typed decision
- 再选择由哪条 relay / 交互路径承接 ask
- 最后再落到 OS / network / policy 的硬边界

## 2. 第一层：先统一 typed decision 语言，而不是先写 UI

这一章最应该先看的文件其实不是 React 组件，而是权限类型空间。

Claude Code 里被传播和持久化的不是 `true/false`，而是：

- `PermissionMode`
- `PermissionRule`
- `PermissionUpdate`
- `PermissionDecision`
- `PermissionDecisionReason`
- `ToolPermissionContext`

这一步非常关键，因为只有先把权限语义类型化，后面这些消费端才能共享同一判断：

- 本地 TUI
- SDK host
- custom permission tool
- bridge / channel
- swarm leader

换句话说，多 transport 共用一套权限系统的前提，不是“都能弹窗”，而是“都说同一种判决语言”。

## 3. 第二层：先决定 permission context，而不是先弹窗

很多读者会从 `PermissionRequest.tsx` 开始看权限系统，但真正更靠前的是 `permissionSetup.ts`。

这里至少决定了：

- 初始 permission mode
- auto mode 是否允许进入
- 进入 auto mode 时是否 strip dangerous allow rules
- 离开 auto mode 时是否 restore
- plan/auto 之间的 mode transition side-effects

这里最有代表性的设计是：

### 2.1 auto mode 不是简单模式切换

进入 auto mode 时，代码会主动剥离那些会绕过 classifier 的危险 allow 规则；
退出时再恢复。

这说明 auto mode 的重点不是“少弹窗”，而是：

- 先把能破坏判定前提的规则拿掉
- 再让 classifier 生效

### 2.2 mode transition 被做成集中收口

`transitionPermissionMode(...)` 把 plan/auto 的进入退出副作用都集中到一个 choke point。
这使得 CLI、SDK、Shift+Tab、slash command 这些入口不需要各自重新实现一遍安全边界。

## 4. 第三层：规则判定永远先于 UI

`hasPermissionsToUseToolInner(...)` 与 `checkRuleBasedPermissions(...)` 很清楚地表明：

1. 先查 deny rule
2. 再查 ask rule
3. 再跑 `tool.checkPermissions(...)`
4. 再处理 content-specific ask 与 safety check
5. 之后才进入 mode 层和交互层

这条顺序非常重要，因为它解释了三个现象：

### 3.1 bypass 也不能绕开所有 ask

content-specific ask rule 与 safety check 明确写成 bypass-immune。
也就是说：

- 用户配置的特定 ask
- `.git` / `.claude` / shell config 等敏感路径

不会因为 mode 放宽就失效。

### 3.2 ask 不是 UI 结果，而是 decision type

代码先得出 `{ behavior: 'ask' }`，再由不同前端承接。
因此 ask 的本体是 typed decision，不是某个 React 组件。

### 3.3 safety check 与 rules 不是一回事

safety check 通常来自文件系统或工具实现层，它和普通 allow/deny rule 并列，但拥有更高的强制性。
这也是为什么很多路径会被写成“即使 bypass 也必须 ask”。

## 5. 第四层：mode 负责改写默认决策，不负责取代所有判定

当规则层没有提前终止时，mode 才开始介入。

这里至少有三种常见改写：

- `bypassPermissions`
- `plan` + bypass 可用
- Bash 在 sandbox 中的 auto allow

但 mode 只负责“放宽默认需要询问的情况”，并不推翻前面已经得出的：

- deny
- content-specific ask
- safety check

这正是 Claude Code 权限模型最成熟的地方之一：

- 放权有边界
- 模式切换不是总开关

## 6. 第五层：classifier 与 hooks 是自动化仲裁器

当决策仍然是 `ask`，Claude Code 并不会立刻弹窗，而是可能先走自动化仲裁。

### 5.1 auto-mode classifier

在 auto mode 下，classifier 会先判：

- allowed
- blocked
- unavailable
- transcript too long

而且不同失败分支不是同一处理：

- transcript too long：回退到普通权限流程
- classifier unavailable：按 gate 决定 fail closed 还是 fail open
- classifier blocked：记录 denial state，并可能在超过阈值后回到人工 review

这说明 classifier 不是简单 allow/deny，而是一个带自我失败语义的判定子系统。

### 5.2 headless hooks

对不能弹窗的 async/headless agent，代码会先跑 `PermissionRequest` hooks。
只有 hook 没给出决定，才自动 deny。

这说明 headless 并不等于“没有权限系统”，而是：

- 先把 ask 转给 hook 自动化
- 不能决定再 fail safe

## 7. 第六层：relay / host 先承接 ask，renderer 最后才出现

权限链在这里最容易被误写。

同一个 `ask`，在真正进入 renderer 之前，已经可能被这些 relay 处理：

- SDK `StructuredIO`
- bridge / channel
- teammate leader permission bridge
- headless hook

这意味着 ask 的“传输层”和 ask 的“展示层”必须分开写。

换句话说：

- relay 负责把 `ask` 送到某个能回应的人/宿主
- renderer 才负责把它画出来

## 8. 第七层：交互层有多条 ask 承接路径

同一个 `{ behavior: 'ask' }`，Claude Code 至少支持四条承接路径。

### 6.1 本地 TUI permission dialog

`PermissionRequest.tsx` 根据 tool 类型切不同组件：

- Bash
- FileEdit / FileWrite
- WebFetch
- Skill
- Workflow
- AskUserQuestion

这条路径负责的是“如何把 ask 呈现给人”，不是“如何得出 ask”。

### 8.2 SDK / StructuredIO 的 `can_use_tool`

`StructuredIO.createCanUseTool(...)` 会把 ask 发成 control request：

- `subtype: can_use_tool`
- 同时并行跑 PermissionRequest hooks
- hook 与 SDK host 先返回者获胜

这意味着 SDK host 不是旁路，而是权限系统的正式消费端。

### 8.3 custom permission prompt tool

`createCanUseToolWithPermissionPrompt(...)` 则允许把 ask 转成一个 MCP tool。
工具返回结果后，再由 `PermissionPromptToolResultSchema` 规约回 typed decision，并可顺手落盘 permission updates。

这说明 Claude Code 的 permission layer 支持“把 ask 外包给另一个工具”，但外包的只是交互，不是决策模型本身。

### 8.4 swarm / teammate permission bridge

多 Agent 场景下，teammate 还可以把 ask 通过 mailbox / leader bridge 交给 leader 处理。
这条链进一步说明权限系统不是“UI 附件”，而是可以跨 agent/transport 投射的控制协议。

## 9. renderer 是下游消费端，不是安全本体

`PermissionRequest.tsx` 与 `PermissionPrompt.tsx` 的职责应该被写得更保守一些：

- 它们消费 `ToolUseConfirm`
- 展示 tool-specific 信息
- 收集 `updatedInput`、`permissionUpdates`、feedback
- 把用户决定重新规约回 typed decision

但它们不负责：

- 判定 deny/ask/allow 的顺序
- 计算安全路径
- 维护 sandbox/policy 边界

这也是为什么“permission modal 不是安全本体”可以成为稳定结论。

## 10. sandbox、policy 与 managed security 是外围硬边界

即使前面所有判定都允许，外围安全壳仍然存在。

### 7.1 sandbox adapter

`sandbox-adapter.ts` 会把 Claude Code 自己的 settings / permission rule 语义转成 sandbox runtime 能执行的：

- 文件读写限制
- network host 限制
- deny-write 保护 `.claude` / settings / skills

也就是说，permission pipeline 负责“该不该允许”，sandbox 负责“即使允许了，最多也只能做到哪里”。

### 7.2 plugin-only policy

`strictPluginOnlyCustomization` 会直接锁死某些 customization surface，只接受：

- plugin
- policySettings
- built-in / bundled

这表明信任边界也被做成了装配前的结构性约束，而不是运行时补丁。

### 7.3 remote managed settings security

当远程 managed settings 引入危险设置变化时，还会触发独立安全确认。
这又是一层比 tool permission 更外面的政策壳。

## 11. fail-open 与 fail-closed 不是一个全局开关

Claude Code 在不同点位采用不同失败哲学：

- classifier unavailable：可配置 fail open / fail closed
- transcript too long：回落到人工 permission flow
- headless 无法交互：先 hooks，后 deny
- sandbox ask callback 失败：默认 deny network
- managed settings security 非交互：直接 skip dialog

这说明作者不是迷信某一种教条，而是在按风险类型做局部最优：

- 运行不可用时有时回退
- 安全边界不明确时更偏保守
- 交互丢失时宁可 deny

## 12. 对蓝皮书的启发

后续写权限系统时，必须先回答四个问题：

1. 我写的是 context 装配、规则层、mode 层、classifier 层，还是交互层。
2. 这个 ask 是 typed decision，还是某个 UI renderer。
3. 这里的最终边界是逻辑判定边界，还是 sandbox / policy 的硬边界。
4. 这是主线程权限，还是 SDK host / teammate / remote 的承接路径。

## 13. 一句话总结

Claude Code 的权限系统不是“遇到危险就弹窗”，而是“先用统一决策流水线算出 typed permission，再由多种前端与外围硬边界共同收口”。
