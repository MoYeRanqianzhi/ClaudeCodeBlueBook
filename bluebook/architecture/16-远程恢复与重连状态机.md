# 远程恢复与重连状态机

这一章回答五个问题：

1. Claude Code 远程链路在连接失败时如何恢复。
2. `SessionsWebSocket`、`remoteBridgeCore`、`RemoteIO` 分别处理哪一层失败。
3. 为什么 `401`、`4001`、`4003`、epoch mismatch、keepalive 都应该放进同一状态机理解。
4. 哪些失败是“可恢复”，哪些是“必须失败并退出”。
5. 这些设计透露了怎样的架构观。

## 1. 先说结论

Claude Code 的远程恢复至少分成三层：

1. 连接层：`SessionsWebSocket` 处理 close code、ping、reconnect。
2. transport rebuild 层：`remoteBridgeCore` 处理 JWT 过期、epoch 更新、401 恢复。
3. host 投射层：`RemoteIO` 处理 keepalive、token refresh、CCR state reporting。

所以远程恢复不是“断了就重连”的单一动作，而是一个带分级失败语义的状态机。

关键证据：

- `claude-code-source-code/src/remote/SessionsWebSocket.ts:234-403`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:456-588`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:780-878`
- `claude-code-source-code/src/cli/remoteIO.ts:54-103`
- `claude-code-source-code/src/cli/remoteIO.ts:111-196`

## 2. 第一层：`SessionsWebSocket` 的 close-code 状态机

### 2.1 永久失败

`PERMANENT_CLOSE_CODES` 当前至少包括：

- `4003` unauthorized

命中后：

- 立即 `onClose`
- 不再尝试 reconnect

证据：

- `claude-code-source-code/src/remote/SessionsWebSocket.ts:28-36`
- `claude-code-source-code/src/remote/SessionsWebSocket.ts:246-253`

### 2.2 条件恢复

`4001` session not found 被当作特殊情况：

- 可能是 compaction 窗口里的暂时性失联
- 允许有限次重试
- 超预算后才真正 `onClose`

证据：

- `claude-code-source-code/src/remote/SessionsWebSocket.ts:21-27`
- `claude-code-source-code/src/remote/SessionsWebSocket.ts:255-272`

### 2.3 一般 reconnect

若此前处于 connected，且预算未耗尽：

- `reconnectAttempts++`
- `scheduleReconnect(...)`

否则：

- `Not reconnecting`
- `onClose`

证据：

- `claude-code-source-code/src/remote/SessionsWebSocket.ts:274-299`

### 2.4 ping 保活

连接层还会用固定 ping interval 保活，但 ping failure 不直接抛复杂逻辑，而是让 close handler 统一兜底。

证据：

- `claude-code-source-code/src/remote/SessionsWebSocket.ts:301-323`

## 3. 第二层：`remoteBridgeCore` 的 401 恢复状态机

### 3.1 `401` 不是普通 close

`remoteBridgeCore` 对 transport close code 有额外分流：

- `401` 且当前不在恢复中 -> 走 `recoverFromAuthFailure()`
- 其他 code -> `onStateChange('failed', ...)`

因此 401 被明确定义为“可尝试恢复的鉴权失效”，而不是和一般 close 同级。

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:456-464`

### 3.2 恢复步骤

`recoverFromAuthFailure()` 的顺序是：

1. 原子性抢占 `authRecoveryInFlight`
2. 状态切到 `reconnecting`
3. OAuth refresh
4. `fetchRemoteCredentials(...)`
5. reset `initialFlushDone`
6. `rebuildTransport(...)`
7. finally 清掉 `authRecoveryInFlight`

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:529-588`

### 3.3 为什么还要 rebuild transport

源码注释写得很明确：

- 每次 `/bridge` 调用都会 bump epoch
- 只换 JWT 不够
- 必须 rebuild transport，否则旧 CCRClient 会心跳到 stale epoch，最终 409

这说明作者不是把 auth 看成“换个 token”这么简单，而是看成“连接语义也跟着变了”。

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:468-528`

## 4. 第三层：`RemoteIO` 的 remote worker 保活层

### 4.1 token refresh callback

`RemoteIO` 提供 `refreshHeaders()`，让 transport 在重连时重新读取：

- session ingress token
- environment runner version

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:71-85`

### 4.2 keepalive

bridge topology 下，`RemoteIO` 会定时发送：

- `keep_alive`

目的是：

- 防 upstream proxy / session-ingress 回收空闲 session

但语义层又会把它过滤掉，不进 UI。

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:174-196`

### 4.3 state reporting

开启 CCR v2 后，`RemoteIO` 还会把：

- command delivery
- session state
- session metadata

通过 CCRClient 上报出去。

所以远程恢复后的“重新连上”并不够，还要把状态重新纳回远端真相。

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:155-167`

## 5. 401 恢复期间为什么要主动丢弃消息

`authRecoveryInFlight` 期间，`remoteBridgeCore` 会主动 drop：

- `control_request`
- `control_response`
- `control_cancel_request`
- `result`

这不是粗暴，而是在保护状态机。

因为这段窗口里：

- 旧 transport 已经陈旧
- 新 transport 还没重建完成
- 如果继续写消息，就可能出现 silent loss 或 stale epoch 409

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:824-878`

## 6. 状态回写也是恢复的一部分

`remoteBridgeCore` 会在不同动作下主动 `reportState(...)`：

- user write batch -> `running`
- `can_use_tool` request -> `requires_action`
- `control_response` / `control_cancel_request` -> `running`
- result -> `idle`

这说明恢复不是只修链路，还要修远端看到的 worker_status。

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:780-878`

## 7. 当前可读的远程失败分级

按源码可读程度，当前至少能归成四类：

1. 可静默恢复：普通 transient disconnect，由 transport 自己重连
2. 可编排恢复：`401` / stale JWT，需要 refresh + rebuild
3. 有预算恢复：`4001` session not found，有限次重试
4. 不可恢复：`4003` unauthorized、预算耗尽、rebuild 失败

## 8. 设计内涵

这套状态机透露出三个重要判断：

1. 远程系统的失败首先是分类问题，不是统一重试问题。
2. 恢复不只修连接，还要修 epoch、flush state、worker_status。
3. 在恢复窗口中宁可 drop message，也不要制造 silent fake-success。

## 9. 一句话总结

Claude Code 的远程恢复本质上不是“自动重连”，而是“按失败类型分层处理连接、鉴权、epoch、flush 和状态回写”的状态机。
