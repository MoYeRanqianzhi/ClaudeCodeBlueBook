# 双通道状态同步与外部元数据回写

这一章回答五个问题：

1. Claude Code 为什么不把宿主真相只放在事件流里。
2. `SDKMessage` 与 `worker_status` / `external_metadata` 分别承担什么职责。
3. 这两条通道在代码里如何被统一接线。
4. 为什么 `WorkerStateUploader`、恢复初始化、state writeback 都应放在同一张架构图里看。
5. 这套设计透露了怎样的 runtime 观。

## 1. 先说结论

Claude Code 对宿主状态真相的设计，不是单通道，而是双通道：

1. 事件通道：`SDKMessage` 按时间顺序发出执行轨迹。
2. 快照通道：`worker_status`、`requires_action_details`、`external_metadata` 持续回写最新状态。

这两条通道不是各写各的，而是通过统一 listener 接线：

- `notifySessionStateChanged(...)`
- `notifySessionMetadataChanged(...)`
- `notifyPermissionModeChanged(...)`
- `RemoteIO` / `CCRClient`

所以它不是“再补几个字段”，而是一个明确的状态同步架构。

关键证据：

- `claude-code-source-code/src/utils/sessionState.ts:1-149`
- `claude-code-source-code/src/state/onChangeAppState.ts:19-91`
- `claude-code-source-code/src/cli/remoteIO.ts:111-168`
- `claude-code-source-code/src/cli/transports/ccrClient.ts:476-545`
- `claude-code-source-code/src/cli/transports/ccrClient.ts:645-662`
- `claude-code-source-code/src/cli/transports/WorkerStateUploader.ts:1-118`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:780-878`

## 2. 通道一：事件流负责时间顺序

`print.ts` 在 headless / SDK 路径里持续向 outbound queue 写入：

- `system:status`
- `auth_status`
- `rate_limit_event`
- `prompt_suggestion`
- query / task / tool / result 消息

而 `sdkEventQueue` 负责补充：

- `task_started`
- `task_progress`
- `task_notification`
- `session_state_changed`

这条通道的职责是给宿主“发生过什么”的时间线。

源码里一个很能说明问题的细节是：`print.ts` 在计算 `lastMessage` 时，会主动把 `session_state_changed`、`task_*`、`post_turn_summary` 等 SDK-only 状态信号排除出去，避免把它们误当“最终结果”。

证据：

- `claude-code-source-code/src/cli/print.ts:890-908`
- `claude-code-source-code/src/cli/print.ts:1051-1135`
- `claude-code-source-code/src/cli/print.ts:2238-2373`
- `claude-code-source-code/src/utils/sdkEventQueue.ts:1-121`

## 3. 通道二：快照回写负责当前真相

### 3.1 `sessionState.ts` 是核心 choke point

`notifySessionStateChanged(...)` 做了三件事：

1. 调用状态 listener，同步 `idle | running | requires_action`
2. 把 `requires_action` 镜像到 `external_metadata.pending_action`
3. 在 `idle` 时清掉 `task_summary`

这说明作者不是把状态真相留给下游自己推理，而是在 emit 点就决定哪些真相必须外化。

证据：

- `claude-code-source-code/src/utils/sessionState.ts:92-121`

### 3.2 `onChangeAppState.ts` 兜住跨路径 mode 变化

`onChangeAppState(...)` 又把：

- `permission_mode`
- `is_ultraplan_mode`

统一镜像到 `external_metadata`，并把 permission mode 变化通过单一 choke point 通知 SDK status stream。

这一步很关键，因为 mode 变化路径非常分散：

- Shift+Tab
- slash command
- rewind
- bridge `set_permission_mode`
- 其他 `setAppState` 调用

如果没有这一层，快照通道就会长期漂移。

证据：

- `claude-code-source-code/src/state/onChangeAppState.ts:43-91`

### 3.3 model 变化也走 metadata 回写

`set_model` 与 `apply_flag_settings` 都会调用：

- `notifySessionMetadataChanged({ model })`

这表明作者把 model 视为“需要恢复后继续保持一致”的状态，而不是只影响当前回合的临时参数。

证据：

- `claude-code-source-code/src/cli/print.ts:2918-2941`
- `claude-code-source-code/src/cli/print.ts:3746-3752`

## 4. `RemoteIO` 负责把两条通道接上远端

`RemoteIO` 初始化 CCR v2 后，会集中注册三类 listener：

- command lifecycle -> `reportDelivery(...)`
- session state -> `reportState(...)`
- session metadata -> `reportMetadata(...)`

换句话说，`RemoteIO` 的角色不是再发明一个远程协议，而是把已有的本地状态与事件面投射到远端 transport。

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:111-168`

## 5. `CCRClient` 与 `WorkerStateUploader` 负责把快照变成可恢复事实

### 5.1 `CCRClient.reportState(...)`

`CCRClient.reportState(...)` 把当前状态写成：

- `worker_status`
- `requires_action_details`

并在“状态没变且无 details”时直接去重。

这说明快照通道不是事件回放副本，而是“最后值语义”。

### 5.2 `CCRClient.reportMetadata(...)`

metadata 则走：

- `external_metadata`

### 5.3 `WorkerStateUploader`

真正把这些 patch 写到远端的，是 `WorkerStateUploader`：

- 只有 1 个 inflight PUT + 1 个 pending patch
- top-level last value wins
- `external_metadata` / `internal_metadata` 走 RFC 7396 merge
- 失败后指数退避，直到成功或 close

这已经不是普通“发个状态”辅助逻辑，而是专门设计过的快照同步器。

证据：

- `claude-code-source-code/src/cli/transports/ccrClient.ts:645-662`
- `claude-code-source-code/src/cli/transports/WorkerStateUploader.ts:4-118`

## 6. 为什么必须是双通道，而不是单通道

如果只保留事件流，会遇到四类问题：

1. consumer 可能晚加入，错过关键状态转移。
2. 某些 adapter 会过滤部分消息。
3. 重连 / 恢复后，单靠回放不一定能立刻重建“当前阻塞点”。
4. 远端 UI 往往更需要“现在是什么状态”，而不是“过去发过哪些消息”。

如果只保留 metadata，也不够：

1. `task_progress`
2. `auth_status`
3. `rate_limit_event`
4. `prompt_suggestion`

这些高频、时间性强、易失的信号更适合事件流。

因此 Claude Code 的设计不是“二选一”，而是：

- 时间线走事件流
- 最新真相走快照回写

## 7. 恢复时，状态同步本身也是恢复的一部分

`remoteBridgeCore` 在发送不同消息时会主动：

- 用户消息批次 -> `reportState('running')`
- `can_use_tool` -> `reportState('requires_action')`
- `control_response` / `control_cancel_request` -> `reportState('running')`
- `result` -> `reportState('idle')`

而在 `401` 恢复窗口里，又会主动 drop：

- `control_request`
- `control_response`
- `control_cancel_request`
- `result`

这说明作者把“状态同步正确”视为恢复的一部分，而不是恢复完成后的附属修饰。

证据：

- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:802-878`

## 8. 初始化时为什么还要清旧 metadata

`CCRClient.initialize()` 在 worker 注册时主动清：

- `pending_action`
- `task_summary`

原因很直接：

- 进程重启不会自动抹掉上次 crash 前的外化状态

如果不清，远端 UI 会以为新的 worker 还卡在旧请求上。

这进一步证明 metadata 不是“调试信息”，而是真正会影响产品行为的状态面。

证据：

- `claude-code-source-code/src/cli/transports/ccrClient.ts:476-489`

## 9. 设计内涵

这套结构透露出三个稳定判断：

1. 运行时真相不应只靠日志推断，关键状态必须外化。
2. 快照同步必须有明确 merge / retry / dedupe 语义，否则恢复后会失真。
3. 事件流与状态回写分工不同，但必须在同一状态机里维护一致性。

## 10. 一句话总结

Claude Code 的宿主状态设计不是“发一串消息给 UI 看”，而是“同时维护一条事件时间线和一份可恢复状态快照”，让远端宿主在实时、重连和恢复场景下都能看见同一份真相。
