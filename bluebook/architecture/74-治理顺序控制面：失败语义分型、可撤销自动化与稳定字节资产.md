# 治理顺序控制面：失败语义分型、可撤销自动化与稳定字节资产

这一章回答五个问题：

1. 为什么仅仅说“统一预算器”还不足以解释 Claude Code 的安全与省 token 底盘。
2. 为什么治理顺序、失败语义分型和自动化撤销路径才是更深层的运行时制度。
3. 为什么系统只该在仍有决策增益的地方继续花 token。
4. 为什么 prompt / permission / replacement 等关键字节应被当成稳定制度资产。
5. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 的安全与省 token 底盘，不只是：

- 一套统一预算器

更深一层，它由四块控制面组成：

1. 治理顺序：
   - 哪些检查必须先发生，哪些后发生。
2. 失败语义分型：
   - 不同资产失败时该 fail-open 还是 fail-closed。
3. 可撤销自动化：
   - 自动化何时继续，何时退回人工，何时直接终止。
4. 稳定字节资产：
   - 哪些字节一旦确立，就必须被稳定重放与持续解释。

这四块放在一起，才构成 Claude Code 更成熟的：

- governance-order control plane

## 2. 治理顺序：先决定谁必须先检查

`permissionSetup.ts` 和 `filesystem.ts` 展现了一个非常明确的底盘原则：

- 治理顺序优先于检查数量

系统不是“把所有检查全开”，而是：

1. 危险 Agent/Bash/PowerShell allow rule 会先被识别，因为它们会绕过后续 classifier；
2. `.claude/**` 的窄豁免只在极窄场景中早于 safety check；
3. 更一般的 allow rule 仍必须经过后续安全路径；
4. source gating 和 adapter enforcement 被分层安放，而不是揉成一个黑盒。

这说明 Claude Code 先解决的是：

- 检查顺序对不对

因为一旦顺序错了，系统会同时损失：

1. 安全性：
   - 风险动作提前绕过；
2. 经济性：
   - token 花在本不该继续评估的路径上。

从运行时角度看，Claude Code 的高明之处不是检查更多，而是：

- 把检查安排到最有治理价值的位置

## 3. 失败语义分型：不同资产不该共享同一种失败哲学

`remoteManagedSettings`、`policyLimits` 和 auto-mode classifier 共同说明：

- Claude Code 并不追求一刀切的 fail-open 或 fail-closed

系统会按资产分型：

1. 远程托管设置在没有缓存时可以 fail-open，避免把非核心配置问题升级成整体阻断；
2. 某些 policy limit 在 essential-traffic-only 下必须 fail-closed；
3. classifier 遇到 transcript too long 时：
   - 交互场景退回人工；
   - headless 场景直接终止；
4. autocompact 连续失败后触发 circuit breaker，而不是永远重试。

这说明 Claude Code 在做的不是“更保守”或“更流畅”，而是：

- 给不同资产设计不同失败语义

从底盘视角看，这一点尤其关键，因为运行时真正面对的是不同类别的资产：

1. 组织边界；
2. 高危配置；
3. 自动化辅助路径；
4. 低价值压缩尝试；
5. 远程配置获取。

它们当然不该拥有同一种失败处理法。

## 4. 可撤销自动化：自动化是租约，不是主权转移

`permissions.ts` 里最成熟的一层，不在 classifier 能否放行，而在：

- 自动化何时必须被收回

具体机制包括：

1. denial-limit fallback：
   - classifier 连续阻断后退回人工 review；
2. transcriptTooLong：
   - 已无判断增益时，不再假装“再试一次可能更好”；
3. bypass / auto 仍受 gate 与 killswitch 管理；
4. headless 场景缺乏人工承接时，会直接终止而不是无穷循环。

这说明 Claude Code 对自动化的理解不是：

- 自动化越多越先进

而是：

- 自动化必须始终可撤销

官方 engineering 公开叙述把这一点又钉死了一层：

1. auto mode 不是把治理权整体转给 classifier，而是先用 fast probe 压低低价值 ask，再只把高疑点请求升级给 reasoning。
2. 一旦 classifier 已经失去决策增益，交互场景退回人工，headless 场景直接终止，这说明“自动化租约”有明确到期条件。
3. classifier prompt 故意剥离 assistant prose 与 tool outputs，也说明它拿到的不是“更完整世界”，而是“仍足以改判的最小裁决对象”。

这同时是一条安全规则，也是一条省 token 规则，因为：

1. 当自动化再无决策增益时，继续跑只会更贵；
2. 当自动化缺乏人工兜底时，继续跑只会更危险。

## 5. 决策增益定价：不是所有检查都值得继续花 token

Claude Code 的一个隐藏但很重要的底盘判断是：

- 只有仍能改变决策的检查，才值得继续花 token

这在多处同时出现：

1. 危险 allow rule 会先被剔除，因为它们会让 classifier 丧失意义；
2. 某些动作在 allowlist 或更低价路径上已足够确定，就不再进入高价分类；
3. transcript 已长到 classifier 无法有效工作时，直接转人工或终止；
4. autocompact 连续失败后停止重试。

这说明 Claude Code 真正优化的不是“更少 token”，而是：

- 不把 token 花在已无制度收益的判断上

从运行时角度看，这是一条非常硬的设计纪律：

- decision-gain-aware token spending

## 6. 稳定字节资产：关键治理字节必须被持续重放与解释

Claude Code 的很多高级机制都在强调：

- 关键字节必须稳定

这并不只限于 prompt 本体。

至少包括：

1. system prompt section cache；
2. sticky beta / latched header；
3. tool result replacement preview；
4. prompt cache break diff 与 root-cause 解释。

`toolResultStorage.ts` 明确要求一旦为某个 `tool_use_id` 生成 preview，后续必须稳定重放；`claude.ts` 会 sticky-latch 某些 header；`promptCacheBreakDetection.ts` 又持续跟踪哪些字节真正改变了。

这说明 Claude Code 要保护的不是“缓存命中率”这个表面结果，而是：

- 会影响治理、成本与解释一致性的那些字节

因为这些字节一旦不稳定，系统就会同时丢失：

1. 成本可预测性；
2. 安全边界一致性；
3. prompt 行为解释力。

所以这类字节更准确的名字不是 cache trick，而是：

- stable-bytes assets

官方 `Manage costs effectively`、`Explore the context window` 与 Agent SDK `file checkpointing` 文档又把这件事从外表补齐：

1. prompt caching 与 auto-compaction 被明确写成默认的成本优化主路径，而不是附属技巧。
2. `/compact` 之后多数 startup content 会自动 reload，说明 stable bytes 的职责不是“永远不重放”，而是“总能按同一 reload law 重放”。
3. checkpointing 只覆盖 `Write / Edit / NotebookEdit`，不回滚 Bash 副作用，也不回滚 conversation 本身，这说明 rollback ceiling 本身也是治理顺序的一部分。

如果继续用苏格拉底式自校把这一节写得更硬，还应再问一句：

- 我们说“可撤销自动化”和“稳定字节资产”时，是否已经把 `rewind files != rewind authority != rewind conversation` 这条上限公开写出来；如果没写，later maintainer 仍会把 rollback 想得过强。

## 7. 为什么这是一块正式控制面，而不是预算器附属层

如果只把 Claude Code 理解成：

- 统一预算器

你只能看到“都在定价”这一层。

但把源码再往下读，会发现它还在维护：

1. 顺序：
   - 谁先检查；
2. 分型：
   - 谁该 fail-open，谁该 fail-closed；
3. 撤销：
   - 自动化何时退回人工；
4. 稳定：
   - 哪些字节必须被持续重放与解释。

这四层一起，才构成 Claude Code 更成熟的：

- governance-order control plane

## 8. 一句话总结

Claude Code 的安全与省 token 底盘之所以成熟，不只是因为它们共享预算器，更因为它把治理顺序、失败语义分型、可撤销自动化和稳定字节资产写成了同一套正式控制面。
