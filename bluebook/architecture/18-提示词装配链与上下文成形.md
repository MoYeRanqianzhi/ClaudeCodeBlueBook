# 提示词装配链与上下文成形

这一章回答五个问题：

1. Claude Code 的 prompt 到底是怎样被拼出来的。
2. 为什么作者要把 prompt 切成静态 prefix、动态 section、attachment delta 三层。
3. prompt cache 稳定性为什么会上升成架构约束。
4. fork / coordinator / custom agent / proactive 为什么会拿到不同 prompt。
5. 这条装配链为何能解释 Claude Code 的“prompt 魔力”。

## 1. 先说结论

Claude Code 的 prompt 不是一段“写得很巧的 system prompt”，而是一条装配链：

1. 取系统 prompt 部件
2. 取 user/system context
3. 应用角色优先级
4. 将动态信息尽量外移到 attachment
5. 保持 cache-key prefix 稳定

所以它真正优化的对象不是“某一段文案”，而是：

- 提示词 bytes
- 角色分层
- cache 命中
- 动态信息迁移
- 上下文污染控制

关键证据：

- `claude-code-source-code/src/utils/queryContext.ts:1-41`
- `claude-code-source-code/src/constants/prompts.ts:114-115`
- `claude-code-source-code/src/constants/prompts.ts:444-579`
- `claude-code-source-code/src/utils/systemPrompt.ts:27-121`
- `claude-code-source-code/src/constants/systemPromptSections.ts:1-58`
- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-95`
- `claude-code-source-code/src/utils/forkedAgent.ts:1-110`
- `claude-code-source-code/src/services/api/promptCacheBreakDetection.ts:1-170`

## 2. 第一层：先取 cache-key prefix 的三大块

`queryContext.ts` 明确把 API cache-key prefix 拆成三块：

1. `systemPrompt`
2. `userContext`
3. `systemContext`

并单独做了 `fetchSystemPromptParts(...)`，目的就是：

- 减少依赖环
- 让 entrypoint 层统一组装 cache-safe parts

这已经不是普通“拼 prompt helper”，而是在保护调用前缀的一致性。
同时也说明：

- `systemContext` 不等于 system prompt 主体
- `userContext` 也不一定以普通用户文本出现

它们很多时候会通过 `<system-reminder>` user message 这条隐藏传输层进入 API 请求。

证据：

- `claude-code-source-code/src/utils/queryContext.ts:1-41`

## 3. 第二层：system prompt 本身也分静态与动态

`constants/prompts.ts` 用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 把 prompt 明确切开：

1. 静态内容：
   - intro
   - system
   - doing tasks
   - actions
   - tool guidance
   - tone/style
   - output efficiency
2. 动态内容：
   - session guidance
   - memory
   - language
   - output style
   - scratchpad
   - token budget
   - brief / proactive
   - MCP instructions

这条边界甚至有显式注释提醒：

- 不能随便挪
- 否则要同步改 cache 逻辑

这说明 prompt 不是自由文本，而是带缓存语义的结构对象。

证据：

- `claude-code-source-code/src/constants/prompts.ts:104-115`
- `claude-code-source-code/src/constants/prompts.ts:444-579`

## 4. 第三层：dynamic section 有缓存策略

`systemPromptSections.ts` 又把动态段分成两类：

1. `systemPromptSection(...)`
   - 可缓存
2. `DANGEROUS_uncachedSystemPromptSection(...)`
   - 会 cache break
   - 需要明确 reason

这是一种非常罕见、但非常成熟的 prompt 工程写法：

- 不只问“内容对不对”
- 还问“这一段值会不会破坏 cache”

证据：

- `claude-code-source-code/src/constants/systemPromptSections.ts:1-58`

## 5. 第四层：角色优先级决定最终 prompt

`buildEffectiveSystemPrompt(...)` 把角色优先级写成正式规则：

1. override
2. coordinator
3. main-thread agent
4. custom system prompt
5. default system prompt
6. append system prompt

其中最关键的不是优先级本身，而是：

1. coordinator 明确换成另一套 prompt
2. proactive 模式下 agent prompt 追加而不是替换
3. built-in agent 可读 `toolUseContext`

这说明 Claude Code 的 prompt 不是“全局唯一人格”，而是按运行角色切换的控制说明书。

证据：

- `claude-code-source-code/src/utils/systemPrompt.ts:27-121`
- `claude-code-source-code/src/coordinator/coordinatorMode.ts:111-260`

## 6. 第五层：把高波动信息迁到 attachment

Claude Code 一直在把“容易波动、但又重要”的上下文从 static prompt 里拆出去。

典型例子有三个：

### 6.1 agent list 不再内联

`AgentTool/prompt.ts` 直接解释了原因：

- agent list 动态变化
- tool description 一变就 cache bust

所以改成：

- attachment / reminder 注入 agent list

### 6.2 deferred tools 不再默认全塞

`ToolSearchTool` 只在需要时拉完整 schema，这意味着：

- 未必把所有 tool schema 预先塞进上下文
- 工具装配也服从 token economics

### 6.3 MCP instructions 改走 delta

`constants/prompts.ts` 对 `mcp_instructions` 单独标成危险 uncached section，还注明：

- late connect 会 bust cache
- 所以改走 delta attachment

证据：

- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-64`
- `claude-code-source-code/src/tools/ToolSearchTool/prompt.ts:24-45`
- `claude-code-source-code/src/constants/prompts.ts:345-358`
- `claude-code-source-code/src/utils/attachments.ts:1-260`

这里最关键的理解是：

- `<system-reminder>` 不是装饰性标签
- 而是 Claude Code 的动态上下文隐藏传输层

agent listing、skill discovery、MCP delta、user/system context 的一部分，都可能沿着这条通道进入模型。

## 7. 第六层：fork 专门为 cache sharing 设计

`forkedAgent.ts` 明确写着：

1. cache-critical params 必须与 parent 相同
2. fork children 要共享 prompt cache
3. mutable state 要隔离

`AgentTool/prompt.ts` 甚至把这条产品化成 prompt 文字：

- fork 很便宜，因为共享 prompt cache
- 不要随意换 model
- 不要偷看 output_file，把噪音拉回主上下文

这说明 fork 的本质不是“再开一个 agent”，而是：

- 复用缓存
- 隔离噪音
- 保持 prefix 稳定

证据：

- `claude-code-source-code/src/utils/forkedAgent.ts:1-110`
- `claude-code-source-code/src/tools/AgentTool/prompt.ts:76-95`

## 8. 第七层：连调试都围绕 prompt cache 建模

`promptCacheBreakDetection.ts` 说明作者甚至专门为 prompt cache 建了诊断系统，跟踪：

- system hash
- tool hash
- cache_control hash
- per-tool schema hash
- global cache strategy
- beta headers
- effort / extra body params

这已经不是“顺带记点 telemetry”，而是在把 prompt cache break 当作一等架构事件。

证据：

- `claude-code-source-code/src/services/api/promptCacheBreakDetection.ts:1-170`

## 9. 为什么这条装配链能产生“prompt 魔力”

所谓“prompt 魔力”，至少有五个工程来源：

1. 角色分工清楚：
   - main thread、coordinator、worker、fork、verification 各有自己的说明书
2. 风险顺序清楚：
   - 先边界、再行动、再输出
3. 动态信息控制得住：
   - 高波动信息不轻易污染静态 prefix
4. token 经济被写进 prompt：
   - deferred tools、attachment、fork cache sharing
5. 错误恢复被写进 prompt：
   - compact prompt 的 no-tools preamble、coordinator 的 worker continuation 规则

它的“魔力”不是因为句子写得神，而是因为 prompt 与 runtime 结构咬合得非常紧。

## 10. 一句话总结

Claude Code 的 prompt 强，不在于某段 system prompt 本身，而在于作者把 prompt 做成了一条受角色、缓存、动态注入和上下文经济学共同约束的装配链。
