# 会话存储、记忆与回溯状态面

这一章不再把 Claude Code 看成“消息流”，而是把它当作一个有 durable state 的 runtime 来看。

它回答四个问题：

1. session transcript 到底如何落盘与恢复。
2. memory surfaces 为什么不只是一份 `CLAUDE.md`。
3. `/resume`、`/memory`、`/rewind` 背后各自依赖什么状态层。
4. 为什么 Claude Code 的可恢复性不是附加功能，而是架构中心。

## 1. 先说结论

Claude Code 的状态面至少分成五层：

1. transcript 层：主会话与子代理的 append-only JSONL。
2. metadata 层：title、tag、PR link、agent 标识、lastPrompt 等轻元信息。
3. memory 层：managed/user/project/local `CLAUDE.md` 体系。
4. session memory 层：按会话动态维护的 `session-memory/summary.md`。
5. rewind 层：file-history snapshots 与 `rewind_files` 控制面。

## 2. Transcript 不是聊天记录，而是主状态日志

### 2.1 路径模型已经暴露出“会话是文件系统对象”

`sessionStorage.ts` 里最关键的两个路径函数是：

- `getTranscriptPath()`
- `getTranscriptPathForSession(sessionId)`

而子代理还有：

- `getAgentTranscriptPath(agentId)`

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:198-258`

这意味着 Claude Code 的 session 不是数据库里的一条 row，也不是内存中的临时对象，而是可定位、可切换、可派生的日志文件树。

### 2.2 `recordTranscript(...)` 不是盲 append，而是带 parent chain 语义的增量写入

`recordTranscript(...)` 会：

1. 先清洗消息
2. 读取当前 session 已记录消息 UUID 集
3. 只写真正的新消息
4. 用 prefix-tracked 的方式维持正确 parent chain
5. 对 compact 场景特判，使新 compact boundary 截断旧链

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:1408-1449`

这说明 transcript 的作用不是“方便以后回看”，而是“维持可恢复的会话拓扑”。

### 2.3 子代理 transcript 与 sidechain 也是正式状态面

同一文件还支持：

- `recordSidechainTranscript(...)`
- `removeTranscriptMessage(...)`
- `recordContextCollapseCommit(...)`

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:1451-1474`
- `claude-code-source-code/src/utils/sessionStorage.ts:1536-1545`

所以 Claude Code 的 transcript 不是单线性对话，而是一个带 sidechain、subagent、collapse commit 的状态图。

## 3. `/resume` 之所以快，是因为它先读“轻元信息”

### 3.1 `readLiteMetadata(...)` 只读头尾，不扫全文

为了避免每次 `/resume` 都全量解析 JSONL，`readLiteMetadata(...)` 只读：

- 头部约 64KB：`isSidechain`、`cwd`、`teamName`、初始 prompt
- 尾部约 64KB：`customTitle`、`tag`、PR link、最新 gitBranch

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:4732-4813`

这一步的本质不是“省点 IO”，而是把“快速恢复可见 session 列表”当成一等需求。

### 3.2 `/resume` 依赖的是 lite log + enrich，而不是全量 transcript

`getSessionFilesLite(...)` 先基于 stat 拿候选集；`enrichLog(...)` 再做头尾元信息补全；`listSessionsImpl(...)` 则按 limit/offset 决定是否先 stat 再 read。

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:4970-5105`
- `claude-code-source-code/src/utils/listSessionsImpl.ts:425-454`

这说明 Claude Code 对 session 恢复的思路不是“把所有日志都全量读回来”，而是“先恢复索引，再按需深入”。

### 3.3 `/resume` 命令本身就是一个状态检索器

`/resume` 的 local-jsx 实现会：

- 加载 same-repo worktrees 的会话
- 支持 UUID 直达
- 支持 custom title 搜索
- 区分跨项目恢复与同项目恢复
- 必要时只给出恢复命令，而不直接切过去

证据：

- `claude-code-source-code/src/commands/resume/index.ts:3-10`
- `claude-code-source-code/src/commands/resume/resume.tsx:89-260`

因此 `/resume` 不是“打开历史聊天”，而是“跨会话状态空间导航”。

## 4. metadata 不是装饰，而是恢复体验的一部分

Claude Code 会把这些元信息单独写进 transcript：

- AI title
- task summary
- tag
- PR link
- agent name / color

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:2667-2729`
- `claude-code-source-code/src/utils/sessionStorage.ts:2819-2854`

更关键的是，compaction 后还会 `reAppendSessionMetadata()`，确保 title/tag 等信息仍在 tail window 内，供 `readLiteMetadata()` 快速扫描。

证据：

- `claude-code-source-code/src/utils/sessionStorage.ts:2807-2817`

这说明 metadata 的目标不是归档，而是恢复路径中的可见性。

## 5. Memory surface 不是一份 `CLAUDE.md`

### 5.1 `claudemd.ts` 明确把 memory 分成四层

加载顺序是：

1. managed memory
2. user memory
3. project memory
4. local memory

同时支持：

- 向上目录遍历
- `.claude/rules/*.md`
- `@include`
- frontmatter `paths`

证据：

- `claude-code-source-code/src/utils/claudemd.ts:1-26`
- `claude-code-source-code/src/utils/claudemd.ts:229-279`

所以 Claude Code 的“记忆”从来不是单文件设计，而是一套分层 instruction hierarchy。

### 5.2 `/memory` 暴露的是整组 memory surfaces

`/memory` 本身只是 local-jsx 命令壳；真正的 selector 会暴露：

- `~/.claude/CLAUDE.md`
- 项目根 `CLAUDE.md`
- 嵌套导入文件
- auto-memory folder
- team memory folder
- agent-specific memory folder

证据：

- `claude-code-source-code/src/commands/memory/index.ts:3-8`
- `claude-code-source-code/src/commands/memory/memory.tsx:21-89`
- `claude-code-source-code/src/components/memory/MemoryFileSelector.tsx:50-151`

这很关键，因为它表明“memory”在产品上也是一个目录化状态面，而不是单按钮。

## 6. Session memory 是动态生成的会话级记忆

### 6.1 路径和普通 `CLAUDE.md` 完全不同

session memory 的路径是：

- `{projectDir}/{sessionId}/session-memory/summary.md`

证据：

- `claude-code-source-code/src/utils/permissions/filesystem.ts:258-270`

这说明 session memory 不是全局 instruction，也不是项目规则；它是“当前这段会话”的局部长期记忆。

### 6.2 它由 post-sampling hook 在后台维护

`initSessionMemory()` 会注册 post-sampling hook；`extractSessionMemory(...)` 决定何时触发提取；真正提取过程则用 forked subagent 在后台更新 memory file。

证据：

- `claude-code-source-code/src/services/SessionMemory/sessionMemory.ts:267-375`

### 6.3 它有自己的阈值状态机

session memory 的触发条件至少包含：

- `minimumMessageTokensToInit`
- `minimumTokensBetweenUpdate`
- `toolCallsBetweenUpdates`
- 上一轮 assistant turn 是否仍有 tool calls

证据：

- `claude-code-source-code/src/services/SessionMemory/sessionMemoryUtils.ts:15-206`
- `claude-code-source-code/src/services/SessionMemory/sessionMemory.ts:134-181`

所以 session memory 不是“每轮都写摘要”，而是按上下文增长和工具活动做节流。

### 6.4 它对权限边界也很严格

session memory 抽取时会：

- 创建隔离 subagent context
- 先读取当前 `summary.md`
- 只允许精确编辑该 memory file

证据：

- `claude-code-source-code/src/services/SessionMemory/sessionMemory.ts:183-233`
- `claude-code-source-code/src/services/SessionMemory/sessionMemory.ts:315-325`
- `claude-code-source-code/src/services/SessionMemory/sessionMemory.ts:383-433`

这说明“系统自己写记忆”并不意味着它可以任意写盘。

## 7. Rewind 依赖 file-history snapshots，而不是消息删除

### 7.1 `/rewind` 命令只是打开选择器

`/rewind` 的实现本身很薄：

- 触发 message selector
- 返回 `skip`

证据：

- `claude-code-source-code/src/commands/rewind/index.ts:3-11`
- `claude-code-source-code/src/commands/rewind/rewind.ts:4-13`

这说明真正的 rewind 语义不在 slash command 层，而在底下的 file-history / transcript state。

### 7.2 file-history 是单独的状态层

`FileHistoryState` 自己维护：

- `snapshots`
- `trackedFiles`
- `snapshotSequence`

并受 `fileCheckpointingEnabled` 与 SDK/headless 独立 env flag 控制。

证据：

- `claude-code-source-code/src/utils/fileHistory.ts:45-78`

### 7.3 写前跟踪与按 turn 快照是两阶段

`fileHistoryTrackEdit(...)` 负责在写前备份旧内容；
`fileHistoryMakeSnapshot(...)` 负责在 turn 结束时形成新快照。

证据：

- `claude-code-source-code/src/utils/fileHistory.ts:80-193`
- `claude-code-source-code/src/utils/fileHistory.ts:195-320`

这意味着 rewind 的状态真相不是当前文件，而是“每个 turn 的版本链”。

### 7.4 resume 还要把 file-history 一起复制过来

`copyFileHistoryForResume(...)` 会在恢复到新 session 时：

- 为旧 session 的 backup files 建硬链接或 copy
- 只在迁移成功后记录对应 snapshot

证据：

- `claude-code-source-code/src/utils/fileHistory.ts:920-1046`

这进一步证明：Claude Code 的 resume 不是只切消息链，连代码回滚能力也要一起带过去。

### 7.5 `rewind_files` 还是正式 control API

headless / SDK 侧通过 `handleRewindFiles(...)` 与控制协议暴露：

- `user_message_id`
- `dry_run`
- `canRewind`
- `filesChanged`
- `insertions`
- `deletions`

证据：

- `claude-code-source-code/src/cli/print.ts:4520-4565`
- `claude-code-source-code/src/entrypoints/sdk/controlSchemas.ts:308-328`
- `claude-code-source-code/src/entrypoints/sdk/coreSchemas.ts:1213-1227`

所以 rewind 不是 REPL 私有特性，而是正式状态控制接口。

`agentSdkTypes.ts` 已显式暴露 session 相关函数：

- `getSessionMessages(...)`
- `listSessions(...)`
- `getSessionInfo(...)`
- `renameSession(...)`
- `tagSession(...)`
- `forkSession(...)`

证据：

- `claude-code-source-code/src/entrypoints/agentSdkTypes.ts:167-272`

## 9. 从第一性原理看，为什么状态面比聊天面更基础

如果一个 coding agent 真的要长期工作，它最先会遇到的不是“下一句怎么回复”，而是：

1. 当前会话的真实状态存在哪里。
2. 哪些记忆是全局的，哪些是项目级的，哪些是会话级的。
3. 一次错误修改如何回退。
4. 压缩、恢复、resume 之后哪些身份与元信息还保得住。

Claude Code 对这些问题的回答并不依赖“模型更强”，而依赖：

- append-only transcript
- head/tail metadata 索引
- layered memory hierarchy
- session memory extraction
- file-history checkpoints
- rewind / resume / fork 接口

所以它的状态观本质上是：

不是“对话历史越多越好”，而是“工程现场必须可恢复、可索引、可分层”。

## 10. 一句话总结

Claude Code 真正强的地方之一，是它把 session、memory、metadata、rewind 做成了一整张状态面：
对话只是表层表现，底层真正被维护的是一个可索引、可恢复、可分层迁移的工程状态系统。
