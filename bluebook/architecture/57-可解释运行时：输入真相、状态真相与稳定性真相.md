# 可解释运行时：输入真相、状态真相与稳定性真相

这一章回答五个问题：

1. 为什么 Claude Code 的 observability 不只是调试输出，而是正式运行时设计。
2. 为什么 `context usage`、`worker_status/external_metadata`、`prompt cache break detection` 应被放在同一张架构图里。
3. Claude Code 如何同时回答“模型这轮看到了什么”“worker 现在处于什么状态”“为什么前缀稳定或失稳”这三类问题。
4. 为什么这些解释面共同构成了可恢复、可分叉、多宿主 runtime 的基础。
5. 这条线为什么比“多打一堆日志”更接近工程先进性。

## 1. 先说结论

Claude Code 的 observability 之所以值得单独写，不是因为它“调试做得细”，而是因为它在维护一套：

- 可解释运行时

这个运行时至少要持续回答三件事：

1. 输入真相：
   - 模型这轮到底看到了什么
2. 状态真相：
   - worker 当前到底在什么状态
3. 稳定性真相：
   - 当前前缀为什么稳定，为什么失稳

Claude Code 分别用三组正式机制回答：

1. `get_context_usage` / `systemPromptSections` / `messageBreakdown`
2. `worker_status` / `external_metadata` / `session_state_changed`
3. `promptCacheBreakDetection`

所以更准确的说法不是：

- 它有不少观测工具

而是：

- 它把“运行时为什么这样”做成了正式控制面

## 2. 输入真相：系统会告诉你模型到底看到了什么

`SDKControlGetContextUsageResponseSchema` 说明 Claude Code 不满足于只暴露：

- total tokens

它还正式暴露：

- `systemPromptSections`
- `mcpTools`
- `deferredBuiltinTools`
- `agents`
- `skills`
- `messageBreakdown`

这意味着 Claude Code 在问的不是：

- 当前上下文大不大

而是：

- 当前上下文是由哪些对象组成的

`analyzeContext.ts` 进一步把这些对象整理成：

- categories
- grid
- messageBreakdown
- attachmentsByType
- toolCallsByType

这是一种非常成熟的观察方式，因为它直接把：

- “模型看到的世界”

做成了可查询对象。

## 3. 状态真相：事件流之外还要有权威快照

Claude Code 同时承认：

- 时间线事件很重要
- 但“当前状态”也必须被单独保存

`notifySessionStateChanged(...)` 会显式处理：

- `requires_action`
- `pending_action`
- `task_summary`
- `session_state_changed`

`onChangeAppState(...)` 又把：

- permission mode
- ultraplan mode

统一写回 metadata。

这意味着 Claude Code 不接受这种设计：

- 宿主自己去事件流里猜当前状态

相反，它要求：

- 当前状态必须有权威快照面

所以它的状态系统并不是“有事件就够了”，而是：

- 事件流 + 快照回写

## 4. 稳定性真相：cache break 必须能被归因

`promptCacheBreakDetection.ts` 更能说明 Claude Code 的成熟。

它没有把 cache miss 处理成：

- 一个模糊的性能下降

而是做成两阶段机制：

1. pre-call 记录 prompt/tool/request surface 快照
2. post-call 用真实 cache token 变化验证是否真的发生 break

它还能继续区分：

- model changed
- system prompt changed
- tool schemas changed
- betas changed
- expected drop
- compaction reset
- TTL / server-side miss 等解释路径

这说明 Claude Code 不是只在“监控命中率”，而是在做：

- stability causality tracking

## 5. 三类真相为什么必须合在一起

如果只有输入真相，没有状态真相，宿主会知道：

- 模型看见了什么

却不知道：

- 当前是不是 blocked
- 是不是在等批准

如果只有状态真相，没有稳定性真相，宿主会知道：

- 现在在跑还是在等

却不知道：

- 为什么这个会话突然变贵
- 为什么 side-loop 复用突然失效

如果只有稳定性真相，没有输入真相，又会知道：

- cache break 了

却不知道：

- 是哪一类对象把前缀打碎了

所以 Claude Code 的可解释运行时不是单点接口，而是三层同时成立：

1. 输入解释
2. 状态解释
3. 因果解释

## 6. 为什么这不是 debug dump，而是正式控制面

有两个信号最重要。

### 6.1 它们进入了正式 schema

`get_context_usage` 不是某个临时 flag，而是正式 control schema。

### 6.2 它们能继续驱动动作

`contextSuggestions.ts` 不只是读取这些观测对象，而是把它们翻译成：

- near capacity
- tool result bloat
- read result bloat
- memory bloat
- autoCompact disabled

以及下一步建议。

这说明 Claude Code 的观测不是“看看就算了”，而是：

- 观测 -> 解释 -> 建议 -> 调整

的正式闭环。

## 7. 一句话总结

Claude Code 的 observability 之所以高级，不在于它暴露了更多统计，而在于它把“模型看到了什么”“系统现在处于什么状态”“前缀为什么稳定或失稳”三类真相一起做成了正式运行时合同。
