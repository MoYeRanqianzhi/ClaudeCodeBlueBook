# 宿主路径时序与竞速

这一章回答五个问题：

1. 本地 SDK host、bridge、direct connect、remote session 三条路径在时间上分别如何运作。
2. 为什么 `can_use_tool`、`control_cancel_request`、error response、keepalive 都必须放进时序里理解。
3. bridge 为什么比 direct connect / remote session manager 更像“控制面适配器”。
4. 哪些 race 是源码显式防守的。
5. 这些时序透露出怎样的架构哲学。

## 1. 先说结论

Claude Code 宿主接入至少有四条典型时序：

1. 本地 SDK host 权限竞速链
2. bridge 权限往返链
3. direct connect 最薄权限链
4. remote session manager 的 WS + HTTP 分离链

这四条链共用同一协议封套，但：

- 时序复杂度不同
- 宿主参与深度不同
- race 防守点不同

关键证据：

- `claude-code-source-code/src/cli/structuredIO.ts:469-658`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:132-391`
- `claude-code-source-code/src/hooks/useReplBridge.tsx:367-586`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:422-456`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:824-878`
- `claude-code-source-code/src/server/directConnectManager.ts:81-185`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:146-297`

## 2. 本地 SDK host：`StructuredIO` 权限竞速链

### 2.1 时序图

```text
CLI/query/tool -> createCanUseTool()
               -> 先跑 local permission check
               -> 若需宿主裁决:
                    hookPromise 后台启动
                    sdkPromise(sendRequest can_use_tool) 立即发出
                    Promise.race(hookPromise, sdkPromise)

hook 先赢:
  -> abort sdk request
  -> enqueue control_cancel_request
  -> 返回 hook decision

host 先赢:
  -> 接收 control_response
  -> parse schema
  -> 返回 permission decision
```

### 2.2 关键时序点

- `sendRequest(...)` 先把 request 放进统一 FIFO，再注册 pending promise。
- abort 不等待对端确认，而是本地立即 reject。
- late response 会被 `resolvedToolUseIds` 护栏拦住。

证据：

- `claude-code-source-code/src/cli/structuredIO.ts:469-531`
- `claude-code-source-code/src/cli/structuredIO.ts:561-639`
- `claude-code-source-code/src/cli/structuredIO.ts:362-399`

## 3. bridge：权限竞速 + 状态回写链

### 3.1 权限往返时序

```text
本地交互层 -> useReplBridge permissionCallbacks.sendRequest(...)
          -> handle.sendControlRequest(can_use_tool)
          -> transport.write(control_request)
          -> server/UI 侧弹权限 prompt

server/UI -> control_response
         -> handlePermissionResponse(request_id)
         -> pendingPermissionHandlers.get(request_id)
         -> 本地 permission handler 收到结果

本地若先决策:
  -> cancelRequest(request_id)
  -> sendControlCancelRequest(request_id)
```

证据：

- `claude-code-source-code/src/hooks/useReplBridge.tsx:367-385`
- `claude-code-source-code/src/hooks/useReplBridge.tsx:538-586`
- `claude-code-source-code/src/bridge/replBridge.ts:1779-1819`

### 3.2 bridge 的额外复杂度

bridge 不是只转发权限，它还要：

- 在 ingress 侧区分 `control_response`、`control_request`、`user`
- 对 echo 和 re-delivery 去重
- 对 outbound-only 模式回 explicit error
- 在 remote bridge core 中回写 `requires_action -> running -> idle`

证据：

- `claude-code-source-code/src/bridge/bridgeMessaging.ts:132-208`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:265-283`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:824-878`

### 3.3 bridge 服务端控制请求时序

```text
server -> control_request(set_model / interrupt / ...)
      -> handleIngressMessage()
      -> handleServerControlRequest()

支持的 subtype:
  initialize / set_model / set_max_thinking_tokens / set_permission_mode / interrupt

未知 subtype:
  -> 立即回 error control_response
```

证据：

- `claude-code-source-code/src/bridge/bridgeMessaging.ts:150-157`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:243-391`

## 4. direct connect：最薄权限链

### 4.1 时序图

```text
server -> WebSocket -> DirectConnectSessionManager
       -> 若 control_request.can_use_tool:
            onPermissionRequest(request, request_id)
       -> 宿主回 success control_response

宿主主动动作:
  sendMessage(user)
  sendInterrupt()
```

### 4.2 特点

- 只认 `can_use_tool`
- 未知 control_request 直接 error
- 过滤 `control_response` / `control_cancel_request` / `keep_alive`

这条链没有 bridge 那样的本地状态映射与中间粘合层。

证据：

- `claude-code-source-code/src/server/directConnectManager.ts:81-112`
- `claude-code-source-code/src/server/directConnectManager.ts:144-185`

## 5. `RemoteSessionManager`：WS 接收 + HTTP 发送分离链

### 5.1 时序图

```text
宿主 -> sendMessage()
     -> HTTP POST /events

remote session -> WebSocket subscribe -> handleMessage()
               -> 若 control_request.can_use_tool:
                    pendingPermissionRequests.set(request_id)
                    onPermissionRequest(...)
               -> 若 control_cancel_request:
                    删除 pending
                    onPermissionCancelled(...)
               -> 若 control_response:
                    仅记录日志

宿主主动中断:
  cancelSession()
  -> websocket.sendControlRequest({subtype:'interrupt'})
```

### 5.2 特点

- 发送 user message 和接收 session message 分离在两条 transport 上
- 对 permission request 有 pending map
- 对控制协议的支持面依旧很窄

证据：

- `claude-code-source-code/src/remote/RemoteSessionManager.ts:146-214`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:217-297`

## 6. keepalive 与显式 error 为什么必须进时序

### 6.1 keepalive

bridge / `RemoteIO` 都会主动发 `keep_alive`，原因不是业务需求，而是：

- upstream proxy / session ingress 会回收空闲连接

如果不放进时序理解，读者会误以为这是产品消息。

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:174-196`
- `claude-code-source-code/src/bridge/replBridge.ts:1528-1548`

### 6.2 unsupported subtype 明确回 error

bridge、direct connect、`RemoteSessionManager` 都有同一个设计：

- 不支持的 control request 必须回 error
- 否则 server 会一直等，最终把连接判死

这说明显式失败本身就是时序稳定性的一部分。

证据：

- `claude-code-source-code/src/bridge/bridgeMessaging.ts:373-390`
- `claude-code-source-code/src/server/directConnectManager.ts:88-98`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:199-213`

## 7. 当前最重要的 race

当前源码显式防守的 race，至少包括：

1. hook vs host permission race
2. late duplicate control_response
3. orphan control_response
4. bridge echo / re-delivery dedup
5. outbound-only false-success risk
6. auth recovery 期间 dropping control messages

证据：

- `claude-code-source-code/src/cli/structuredIO.ts:362-399`
- `claude-code-source-code/src/cli/structuredIO.ts:490-504`
- `claude-code-source-code/src/cli/structuredIO.ts:561-639`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:168-185`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:265-283`
- `claude-code-source-code/src/bridge/remoteBridgeCore.ts:824-870`

## 8. 设计内涵

这些时序共同说明 Claude Code 团队有三个很清楚的判断：

1. 宿主接入首先是时序问题，不只是字段问题。
2. 显式 cancel / explicit error / state writeback 比“尽量看起来成功”更重要。
3. 不同宿主可以只支持子集，但必须在时序上诚实。

## 9. 一句话总结

Claude Code 的宿主接入不是“谁能发几个 JSON 消息”这么简单，而是“谁能在正确时序下维持权限、取消、错误和状态机的一致性”。
