# 扩展能力与远程架构

## 1. 结论先行

Claude Code 的扩展方式不是单一插件机制，而是至少并行存在四条线：

- 技能
- 插件
- MCP
- 远程 / bridge / coordinator

这四条线共同解决的是同一个问题：

如何让主循环在不失控的前提下，持续获得新能力。

## 2. 技能系统：本质上是带 frontmatter 的可加载工作流

`src/skills/loadSkillsDir.ts` 显示，技能不是随便一段 markdown。

技能可以声明：

- 名称、描述、何时使用
- `allowed-tools`
- `model`
- `effort`
- `hooks`
- `context`
- `agent`

这意味着技能本质上是“可声明、可约束、可装配”的工作流单元。

源码还专门处理了：

- frontmatter 解析
- 描述提取
- 参数替换
- 路径约束
- gitignore / filesystem 可见性

所以技能系统不是 prompt 片段仓库，而是轻量工作流 DSL。

## 3. bundled skill：把能力打包进 CLI

`src/skills/bundled/index.ts` 与 `src/skills/bundledSkills.ts` 说明：

- 一部分技能在启动时直接注册
- 技能可以带额外参考文件
- 首次调用时，这些文件会被安全地提取到本地目录
- 模型会收到一个 “Base directory for this skill” 前缀

这很关键，因为它把“文档资产”变成了模型可按需读取的能力边界，而不是每次都把全文塞进上下文。

## 4. 插件系统：与 bundled skill 有意分离

`src/plugins/builtinPlugins.ts` 的注释写得很清楚：

- built-in plugin 与 bundled skill 不是一回事
- plugin 强调用户可启停
- plugin 可以组合多个组件：skills、hooks、MCP servers

而 `src/plugins/bundled/index.ts` 当前几乎还是脚手架。

这说明插件系统在这份源码里更像“正在成形的平台层”，而不是已经完全展开的主能力层。

## 5. 远程会话：`RemoteSessionManager`

`src/remote/RemoteSessionManager.ts` 表明远程模式不是简单 websocket 聊天：

- SDK 消息和 control 消息被明确区分
- 权限请求可以跨远程会话往返
- 连接、断开、重连、错误都有专门回调
- 用户消息发送走 HTTP，消息接收走 WebSocket

这说明远程会话被当作“分布式执行环境”，而不是“终端镜像”。

## 6. bridge：把本机变成可被调度的执行节点

`src/bridge/bridgeMain.ts` 体量很大，说明 bridge 是独立子系统。

从当前可见代码看，它至少负责：

- 环境注册
- 会话拉起
- 轮询与 backoff
- heartbeat
- token refresh
- session worktree 管理
- 容量唤醒

这意味着 bridge 的角色不是“远程控制 UI”，而是“主循环外的运行时编排基础设施”。

## 7. coordinator mode：显式的编排者人格

`src/coordinator/coordinatorMode.ts` 非常值得重视，因为它把多代理哲学直接写进了系统提示词。

可见事实：

- coordinator mode 通过环境变量和 gate 决定
- 会给 worker 注入独立工具上下文
- 会把 scratchpad 明确定位为跨 worker 的耐久知识区
- 系统提示词直接要求 coordinator 做研究、综合、实现、验证的分工
- 并发被定义为“superpower”

这里已经不是“代码支持多代理”，而是“产品默认相信编排式工作流比单体 Agent 更强”。

## 8. 对使用方式的直接影响

从扩展与远程层可反推出几条实战规则：

1. 能写成技能的流程，就不要每次手工重复提示。
2. 当能力来自外部系统时，优先走 MCP，而不是把所有逻辑塞进主 prompt。
3. 长流程、隔离任务、高风险任务适合走 subagent/worktree/remote。
4. 多代理不是为了炫技，而是为了把研究、实现、验证解耦。

## 9. 一句话总结

Claude Code 的扩展能力之所以强，不是因为它只会“接插件”，而是因为它把本地、远程、技能、插件、MCP 全都放进了统一的调度视角里。
