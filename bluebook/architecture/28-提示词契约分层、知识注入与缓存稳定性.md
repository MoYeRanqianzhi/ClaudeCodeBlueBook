# 提示词契约分层、知识注入与缓存稳定性

这一章回答五个问题：

1. Claude Code 的 prompt 到底是怎样分层装出来的。
2. 为什么它必须把提示词、上下文和附件拆成多层。
3. 缓存稳定性为什么会反过来塑造 prompt 结构。
4. 知识注入为什么不是“把更多文字塞进 system prompt”。
5. 这套结构对外部宿主和使用者意味着什么。

## 1. 先说结论

Claude Code 的 prompt 不是单个字符串，而是四层合同：

1. 静态前缀：
   - 通用系统规则、做任务原则、工具使用原则、语气与输出约束。
2. 动态 section：
   - `memory`、`env_info`、`language`、`output_style`、`mcp_instructions`、`scratchpad`、`token_budget` 等按 section 缓存和重算。
3. 角色覆盖层：
   - coordinator prompt、agent prompt、`--system-prompt`、`--append-system-prompt`、override prompt。
4. 晚绑定附件层：
   - nested memory、relevant memories、plan/auto mode、team context、mailbox、skill delta、MCP instructions delta。

关键不是“层数多”，而是每一层各自承担不同稳定性：

- 静态前缀追求 cache-friendly
- 动态 section 追求可控变化
- 角色层追求职责清晰
- 附件层追求晚绑定与按需注入

代表性证据：

- `claude-code-source-code/src/constants/prompts.ts:444-576`
- `claude-code-source-code/src/utils/systemPrompt.ts:28-122`
- `claude-code-source-code/src/constants/systemPromptSections.ts:10-67`
- `claude-code-source-code/src/context.ts:113-188`
- `claude-code-source-code/src/utils/attachments.ts:817-913`

## 2. 第一层：静态前缀不是“默认 prompt”，而是 cache 基座

`getSystemPrompt()` 返回的不是一段全文，而是一组 section 数组。  
在非 simple 模式下，它先拼：

- intro
- system
- doing tasks
- actions
- using your tools
- tone and style
- output efficiency

然后在边界标记后再拼动态 section。

这里最关键的不是文案内容，而是 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`：

- 它把跨会话更稳定的前缀和更动态的尾部分开
- 让 cache scope 可以只命中稳定前缀
- 使“后面发生变化”不必重算整个 prompt

证据：

- `claude-code-source-code/src/constants/prompts.ts:105-115`
- `claude-code-source-code/src/constants/prompts.ts:560-576`

这说明 Claude Code 的 prompt 设计，从一开始就同时在做两件事：

1. 指导模型
2. 保护缓存

## 3. 第二层：动态 section 不是补丁，而是注册表

`systemPromptSection()` 与 `DANGEROUS_uncachedSystemPromptSection()` 把动态内容做成了正式对象：

- section 名称
- compute 函数
- 是否会 cache break

`resolveSystemPromptSections()` 再统一解析和缓存这些 section。

这意味着 Claude Code 没把“语言偏好”“scratchpad”“memory prompt”“MCP instructions”散落在各个调用点，而是集中到一个 section 注册表里。

证据：

- `claude-code-source-code/src/constants/systemPromptSections.ts:10-57`
- `claude-code-source-code/src/constants/prompts.ts:491-558`

更值得注意的是：

- `mcp_instructions` 被明确放进 `DANGEROUS_uncachedSystemPromptSection`

因为它可能随着 MCP server 的连接状态变化而变化。作者没有掩盖这一点，而是直接承认：

- 某些 section 的变化就是会 bust cache

这正是工程成熟度，而不是妥协。

## 4. 第三层：角色覆盖层决定“你是谁”

`buildEffectiveSystemPrompt()` 说明了角色层的优先级：

1. override prompt
2. coordinator prompt
3. agent prompt
4. custom system prompt
5. default system prompt
6. append system prompt 始终尾加

而且 coordinator 与 agent 的处理不是对称的：

- coordinator 会直接替换默认 prompt
- proactive 模式下，agent prompt 会追加在默认 prompt 后，而不是完全替换

这说明 prompt 不是“谁最后写谁赢”，而是：

- 不同角色有不同的替换策略

证据：

- `claude-code-source-code/src/utils/systemPrompt.ts:28-40`
- `claude-code-source-code/src/utils/systemPrompt.ts:56-75`
- `claude-code-source-code/src/utils/systemPrompt.ts:77-122`

## 5. 第四层：系统上下文和用户上下文不是附件，而是预装配知识

`context.ts` 明确把上下文分成：

1. system context：
   - git status
   - current branch
   - default branch
   - recent commits
   - optional cache breaker
2. user context：
   - `CLAUDE.md`
   - `CLAUDE.local.md`
   - current date

而且它们是 memoized 的，会在会话期内缓存。

证据：

- `claude-code-source-code/src/context.ts:36-111`
- `claude-code-source-code/src/context.ts:113-149`
- `claude-code-source-code/src/context.ts:155-188`

这意味着 Claude Code 的 prompt 并不是只有“系统规则 + 用户消息”，还包含一层：

- 会话开始时预装配的稳定上下文

它比附件更稳定，比硬编码 prompt 更贴近现场。

更重要的是，作者刻意把两类环境信息拆开了：

1. `env_info`：
   - `cwd`
   - shell
   - platform
   - worktree
   - model / knowledge cutoff
2. `git status` snapshot：
   - 当前分支
   - default branch
   - recent commits
   - 一次性的 repo 现场

这种拆法的含义是：

- 基础执行环境应相对稳定
- repo 工作快照则属于更高波动的系统上下文

证据：

- `claude-code-source-code/src/constants/prompts.ts:456-460`
- `claude-code-source-code/src/constants/prompts.ts:499-500`
- `claude-code-source-code/src/context.ts:36-111`

## 6. 第五层：附件层负责晚绑定，不污染稳定前缀

`attachments.ts` 里真正重要的，不是附件类型多，而是：

- 它把许多高波动信息从主 prompt 前缀中移了出去

统一附件流水线会按需注入：

- `nested_memory`
- `dynamic_skill`
- `agent_listing_delta`
- `mcp_instructions_delta`
- `plan_mode`
- `auto_mode`
- `teammate_mailbox`
- `team_context`

证据：

- `claude-code-source-code/src/utils/attachments.ts:817-913`

其中最有代表性的有三类：

### 6.1 nested memory

它把打开文件相关的 `CLAUDE.md` / rules / conditional rules 晚绑定注入，而不是把整个项目所有规则永久塞进前缀。

证据：

- `claude-code-source-code/src/utils/attachments.ts:1710-1774`
- `claude-code-source-code/src/utils/attachments.ts:1792-1861`

### 6.2 relevant memories

它把相关长期记忆做成异步 prefetch，在 turn 期间后台搜索，只有命中且未重复时才注入。

证据：

- `claude-code-source-code/src/utils/attachments.ts:2245-2424`
- `claude-code-source-code/src/utils/attachments.ts:2520-2540`

### 6.3 team context / teammate mailbox

它把多 Agent 协作协议写成 attachment，而不是嵌死在每个 worker prompt 里。

证据：

- `claude-code-source-code/src/utils/attachments.ts:3521-3768`
- `claude-code-source-code/src/utils/messages.ts:3453-3495`

而 appenders 又承担另一种完全不同的职责：

- teammate addendum
- proactive addendum
- assistant addendum
- custom agent instructions

它们更像“角色差异注入口”，而不是系统合同本体。

证据：

- `claude-code-source-code/src/main.tsx:1384-1387`
- `claude-code-source-code/src/main.tsx:2168-2168`
- `claude-code-source-code/src/main.tsx:2194-2208`

## 7. 为什么缓存稳定性会反过来塑造 prompt 结构

如果只从“模型理解”出发，会自然想把所有东西都放进 system prompt。  
Claude Code 没这么做，因为它还同时在优化：

1. prefix cache 命中率
2. late connect / late skill discover 的抖动
3. turn 内高波动信息的注入时机

这可以从三处看得很清楚：

1. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`
2. `mcp_instructions_delta` 与 deferred tool delta
3. tool pool 的稳定顺序

其中 `toolPool.ts` 明确说明：

- built-ins 必须保持连续前缀
- MCP tools 排到后面

证据：

- `claude-code-source-code/src/utils/toolPool.ts:43-79`

这意味着 Claude Code 不是先写出功能，再祈祷缓存好用；而是反过来：

- 缓存稳定性已经进入系统设计约束

## 8. 对宿主和使用者的启发

### 8.1 对宿主

如果你要接 Claude Code：

- 不要把所有业务上下文都塞到 `systemPrompt`
- 把稳定规则和高波动现场分层
- 尽量把高波动状态做成后置 attachment 或 control surface

### 8.2 对使用者

如果你想“把 Claude Code 用得更像 Claude Code”：

- 长期规则写进 `CLAUDE.md`
- 会话级补充放 `append-system-prompt`
- 文件相关细则交给 nested memory 与 skill frontmatter
- 不要频繁用一次性大段 prompt 去替代分层知识结构

另一个同样重要的反面原则是：

- 不是所有对协作有帮助的信息都该进主 prompt

例如 todos 和 status line，源码都更倾向把它们放在：

- output style
- runtime reminder attachment
- UI / hook 平面

这正是 Claude Code prompt 设计足够克制的原因之一。

## 9. 一句话总结

Claude Code 的 prompt 强，不是因为它更会写 prompt，而是因为它把提示词、知识注入和缓存稳定性一起做成了可编排的 runtime contract。
