# 运行时契约、知识层与生态边界

这一章回答四个问题：

1. 为什么 Claude Code 越往深处看，越不像“一个强提示词 + 一堆工具”。
2. 提示词、知识层、权限、协作、生态边界为什么必须被一起理解。
3. 为什么 `governance key / externalized truth chain / typed ask / decision window / continuation pricing / durable-transient cleanup` 必须一起理解。
4. 从第一性原理看，什么才是最难被抄走的部分。

## 1. 先说结论

更稳的压缩顺序不是先给三句判断和六个平面并列，而是先把 Claude Code 压回一条链：

- `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability`

在这条链已经成立后，才继续把 Claude Code 压回三句更高阶判断：

1. Prompt 线
   - request compiler、`message lineage` 与 `continuation asset` 必须持续对同一个世界保持诚实
2. 治理线
   - 一切扩张必须按同一价格秩序合法进入当前世界
3. 源码质量线
   - `current-truth writeback`、`consumer subset` 与 `hotspot kernel` 必须一起保护“现在不能被过去写坏”

这意味着 Claude Code 最深的一层不再只是：

- 提示词平面
- 治理平面
- 演化平面

而是：

- 世界如何进入模型
- 扩张如何进入当前世界
- 过去与失控复杂度怎样不得偷走当前世界

在这个前提下，Claude Code 的本体才可以继续被展开成六个运行时平面：

1. 提示词平面：
   - 不是单段 system prompt，而是静态前缀、动态 section、上下文注入、附件提醒共同组成的合同。
2. 知识平面：
   - 不是“历史消息越多越好”，而是 `CLAUDE.md`、typed memory、session memory、relevant memories、plan/tasks 的分层持久化。
3. 协作平面：
   - 不是“多开几个 agent”，而是 coordinator 规则、task 对象、mailbox、team context、background runtime。
4. 生态平面：
   - 不是“支持插件”四个字，而是 marketplace、builtin plugin、bundled skill、MCPB、LSP、channels 各自不同的信任与治理模型。
5. 当前真相平面：
   - 不是 transcript replay、UI snapshot 或 adapter cache，而是 `state writeback / worker_status / external_metadata` 共同外化的 host-facing current truth。
6. 演化平面：
   - 不是“feature 多”，而是 build gate、runtime gate、compat shim、consumer subset 的多层发布制度。

这六个平面一起解释了：

- 为什么 prompt 会显得聪明
- 为什么长任务能持续
- 为什么多 Agent 不容易立刻失控
- 为什么生态面看起来丰富但又被明显分层
- 为什么它能快速演化却不必每次重写整套系统

所以这六个平面不是新的顶层宪法，而是 `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability` 在提示词、知识、协作、生态、当前真相与演化上的分布式执行面。

如果把这层再压成工程对象，至少要继续对应六类治理对象：

1. `request compiler`
   - 世界怎样被正式装配进当前回合
2. `continuation asset`
   - compact、memory、summary 之后什么还算同一个世界
3. `governance key`
   - 谁先有资格改 provider、policy、permission mode 与 trust 前环境
4. `externalized truth chain / typed ask / decision window`
   - 宿主承认哪条当前真相、哪些扩张这次真的可执行
5. `current-truth writeback`
   - 哪条状态写回面有资格宣布当前快照，哪些投影只能消费不能篡位
6. `continuation pricing / durable-transient cleanup`
   - 继续、恢复、结算与重新绑定执行连续性时，哪些资产还能算当前世界

如果不把这六类对象继续对应到两类不同入口，同一世界编译链就很容易重新被读成理念总结，而不是可验证的系统结构：

- `authority / witness seams`
  - `api / settings-policy / mcp / state writeback`
- `continuation consumers`
  - `compact / memory / observability`

前者负责签 contract、witness 与 writeback seam；后者只负责消费、压缩、恢复与投影，不得越位改判 runtime truth。

公开证据也应在这里先分成两栏：

- `runtime-core evidence` 只签 contract、witness 与 writeback seam。
- `operator-governance evidence` 只签 `CLAUDE.md / hooks / skills / policy surface` 这类治理工件。
- 两者都重要，但不能混写成同一层 certainty；否则 guidance 和 governance artifact 会重新冒充 runtime truth。

再往下压一层，Prompt 线还应先被明确成：

- 不是单段 `system prompt`
- 而是一个 multi-surface 的 world-entry object

最小 surface 至少包括：

1. `system sections`
2. `tool descriptions`
3. `agent prompts`
4. `attachment deltas`

它们之所以仍算同一个 world-entry object，不是因为最后会拼成一段更长文本，而是因为不同 consumer 仍沿同一条 `message lineage` 消费同一份 request compile chain。

如果继续要求这条 Prompt 线可被 later maintainer 复证，更稳的写法也不是在这里再重发 6-link witness chain，而是明确：

- Prompt ABI 的唯一 owner 统一回 `philosophy/84`
- 本页只负责综合说明“不同 consumer 是否仍在消费同一 world-entry object”，不再第二次签发 witness order

如果继续把这章往“later maintainer 可直接拿来排查”的方向压，它还应再导出三组最小顺序：

1. Prompt same-world owner route
   - Prompt ABI、`world-defining winner` 与 continue qualification 统一回 `philosophy/84`
   - 本页只要求 later maintainer 能看出：这里仍是同一个 world-entry object，而不是多份 prompt surface 各自说话
2. governance failure order
   - `pricing-right mismatch -> truth-surface demotion -> asset-rollback ABI sealing -> shared reject verdict -> reopen qualification / human fallback`
   - 审批框只是 `typed ask` 的一个 consumer，不是治理本体；真正先决定 verdict 的是 `pricing-right` 是否成立、`truth-surface` 是否先被 attestation。
3. current-truth reject order
   - `shim exit condition -> recovery asset non-sovereignty -> stale authority eviction -> retreat layer`

只有这三组顺序在 `09 / 83 / 84 / guides` 里持续共用，这章才不只是理念总结，而是真正的同一世界编译链根页。

## 2. 为什么提示词、知识层、权限和生态必须一起看

如果只看 prompt，会得到一个错误印象：

- 似乎强点只在文案

如果只看 tools，又会得到另一个错误印象：

- 似乎强点只在能力数量

如果只看 UI，则又会低估一件事：

- 长任务真正稳定运行，需要认知控制面和可恢复状态面一起成立

Claude Code 的源码说明，真正的闭环是：

1. `prompts.ts` 和 `systemPrompt.ts` 决定角色合同怎么装。
2. `context.ts`、`claudemd.ts`、`memdir.ts`、`SessionMemory` 决定哪些知识被稳定地带进上下文。
3. `attachments.ts` 决定哪些动态状态在 turn 间晚绑定注入。
4. `coordinatorMode.ts`、task framework、mailbox 决定多 Agent 如何分工与汇报。
5. `onChangeAppState / sessionState / WorkerStateUploader` 决定哪些状态能被外化成 host-facing current truth。
6. `marketplaceManager.ts`、channels gate、LSP/MCPB 相关模块决定生态能力如何被治理。

所以“Claude Code 很强”的更准确说法不是：

- 它有一个很厉害的 agent

而是：

- 它把 agent 的提示词、知识、权限、协作、当前真相与生态边界都做成了同一个 runtime 的不同平面

## 2.1 最难被抄走的不是功能，而是四个跨平面函数

如果继续把这章再压硬一层，Claude Code 最难被抄走的不是某个平面本身，而是四个跨平面函数：

1. `compile`
   - request compiler 先把世界装进模型，而不是让模型从聊天记录里自己猜世界。
2. `price`
   - 治理链先给扩边界定价，再决定 ask、continue 与 cleanup。
3. `externalize`
   - runtime 先把当前真相、窗口条件与状态投影外化给 host，而不是让 host 自己猜。
4. `evict`
   - stale writer、ghost capability 与旧 authority 一旦过期，就必须被驱逐，而不是继续借壳留在当前世界。

只抄字符串、弹窗、目录树或 transport shell，抄不到这四个函数共同成立时的系统高级感。

## 3. 对使用者的直接启发

### 3.1 不要把所有长期知识都塞进一层

更好的分法是：

- 项目长期规则：`CLAUDE.md`
- 私有项目习惯：`CLAUDE.local.md`
- 可复用长期记忆：typed memory / `MEMORY.md`
- 当前会话连续性：session memory
- 当前任务执行状态：plan / tasks

只要把这些层混成一层，Claude Code 的长处就会被你自己抵消掉。

### 3.2 多 Agent 时，先写自包含 prompt，再谈并行

源码已经把规则写死得很清楚：

- worker 看不到你的对话
- coordinator 不能把理解外包
- background task 和 task notification 是正式对象

因此，多 Agent 的最小正确用法不是：

- “你们几个自己看看”

而是：

- 明确任务范围
- 明确输出格式
- 明确 ownership
- 明确何时汇报

### 3.3 把生态支持和产品成熟度分开看

源码里出现：

- plugin
- marketplace
- MCPB / DXT
- LSP
- channels

不等于这些东西处在同一成熟度，也不等于都属于同一种信任层。

真正稳妥的做法是：

- 先分清“代码支持了什么 schema”
- 再分清“产品真正承诺了什么能力”

## 4. 对系统设计者的直接启发

### 4.1 Prompt 不是文案对象，而是契约对象

如果你想复制 Claude Code 的强项，优先应该复制的不是某个 system prompt，而是：

- 静态前缀与动态 section 的分离
- 角色 prompt 的分层
- 动态知识的附件化
- 缓存稳定性的结构约束

### 4.2 安全与 token 经济其实在做同一件事

两者都在限制无序扩张：

- 安全限制动作空间
- token 经济限制上下文空间

它们的共同目标不是：

- 更会 deny
- 更会省字

而是：

- 让动作、能力、上下文与时间按同一价格秩序合法进入当前世界

更硬一点说，这条秩序不是先问“要不要拦”，而是先问“谁有资格改边界”。

因此更稳的顺序应是：

1. `governance key`
   - 先决定哪类 settings / policy / trusted inputs 有资格扩边界
2. `externalized truth chain`
   - 再决定宿主此刻承认哪条当前真相
3. `typed ask / decision window`
   - 再决定哪些动作这次真的可执行、哪些动作必须 ask 或 deny
4. `continuation pricing / durable-transient cleanup`
   - 最后决定这轮是否值得继续耗时和耗 token，以及恢复后哪些 authority 必须重审

更短的判断句也只该剩两句：

- 治理先外化当前真相，再批准当前扩张。
- 连治理检查自己都必须被定价；没有决策增益的检查，本身就是免费扩张。

如果继续按服务平面把这件事写硬一点，还应同时补上：

- `settings / policy / managed env` 产出 `governance key`
- `api / host bridge` 外化并消费 `externalized truth chain`
- `permissions / channels / hooks` 执行 `typed ask` 与 `decision window`
- `compact / token budget / recovery` 执行 `continuation pricing`
- `memory / cleanup / result storage` 区分 durable assets 与 transient authority

因此 `Context Usage` 在这里首先也不是成本面板，而是 `decision window` 是否仍值得继续支付的诚实投影。

### 4.3 生态的真正难点不是格式支持，而是信任与治理

支持 `.mcpb`、支持 `LSPTool`、支持 plugin marketplace，都不难写成“能跑”。

更难的是：

- 谁能默认启用
- 谁要走 allowlist
- 谁受 org policy 约束
- 哪些能力只允许某些 consumer subset 看见

Claude Code 的先进之处，正是在这里明显比普通 agent 产品更克制。

### 4.4 prompt 真正强在“行动语义更密”

如果沿 session memory、prompt suggestion、stop hooks、tool result replacement 与 message lineage 往下追，会发现 Claude Code 不只是在做 prompt contract。

它还在做：

- 语义压缩

也就是说，它真正想保住的不是原始世界更多，而是：

- 未来回合仍然可继续工作的行动语义

更稳的读法还要再补一句：

- `display truth / protocol truth / handoff truth` 不是三份平行文本，而是同一条 message lineage 的三种消费投影

### 4.5 runtime 其实在分配资源主权

如果沿 permissions、settings merge、managed hooks、token budget 与 context suggestions 往下追，会发现 Claude Code 也不只是“有安全层”和“有成本层”。

它更像在持续决定：

- 谁能改边界
- 宿主此刻承认哪条当前真相
- 哪些扩张必须 ask
- 继续与恢复是否还站得住

这也是为什么模型在这里从来不是资源的主人，runtime 才是。

而 host 也不是第二个资源主人。

更准确地说：

- runtime 决定 `mode / visible-set / pending_action / current truth`
- host 只消费 runtime 外化出来的 authority/status
- host 不该自己回放事件去拼 mode、tool pool、pending action 或 context truth

### 4.6 好源码的更深层价值是保留下一次重构可能性

如果继续沿 `query/config.ts`、`query/deps.ts`、leaf module 注释与 snapshot 逻辑往下追，会发现 Claude Code 的源码先进性也不只是“不变量治理”。

它还在持续保留：

- 下一次重排整套系统的可能性

这意味着它的工程高级感不只在当前分层，而在增长时仍不把未来重构的门焊死。

### 4.7 公开镜像下更稳的源码质量读法

如果继续沿 `README`、`Task.ts`、`tasks.ts`、`controlSchemas.ts`、`QueryGuard`、`sessionIngress` 与 `guides/102` 往下追，会发现公开镜像下真正稳的源码质量读法也已经很清楚：

1. 先看 `contract truth`
2. 再看 `registry truth`
3. 再看 `current-truth surface`
4. 再看 `consumer subset`
5. 最后才看 `hotspot kernel` 与 `mirror gap discipline`

这会把“目录很整齐”“大文件很多”“代码里提到了某能力”这些低阶体感压回去，让读者优先看到：

- 哪些复杂度有合法席位
- 哪些边界已经可被证明
- 哪些地方其实还不能被脑补补齐

### 4.8 prompt 还在组织“人类如何继续”

如果沿 sticky prompt、prompt suggestion、session memory、teammate navigation 与 UI transcript 往下追，会发现 Claude Code 的 prompt 契约不只服务模型。

它也在服务：

- 人类怎样理解当前现场
- 人类怎样接手下一步
- 人类怎样在多执行链之间切换

所以它是模型契约，也是协作接口。

更硬一点说，它同时在安排：

- `display truth`
- `protocol truth`
- `handoff truth`

### 4.9 runtime 还在分配“谁拥有什么自由”

如果继续沿 permission mode、acceptEdits、bypass、deferred、externalization 往下追，会发现 Claude Code 的 runtime 也不只是治理边界。

它还在持续决定：

- 模型当前拥有什么自由
- 哪些自由要先被延后或旁路
- 哪些自由必须带着约束才能成立

### 4.10 三张控制面图是这一章的稳定回链

如果把这章对 Claude Code 的设计内涵继续压缩，可以直接落成三张更硬的 runtime 地图：

1. [architecture/82](architecture/82-请求装配流水线：authority chain、section registry、protocol transcript、lawful forgetting与cache-safe forks.md)
   - 解释 `world entry / request assembly / six-stage assembly chain` 为什么首先来自请求装配，而不是文案技巧
2. [architecture/83](architecture/83-反扩张治理流水线：trusted inputs、distributed ask arbitration、deferred visibility与continuation pricing.md)
   - 解释 `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup` 为什么是同一条反扩张治理链
3. [architecture/84](architecture/84-权威面与反僵尸图谱：single-writer surfaces、409 adoption、bridge pointer freshness与release shaping.md)
   - 解释源码先进性为什么首先服务未来维护者对当前实现的正式反驳能力

这三张图把前面提到的五个平面进一步压成三个更不可约的问题：

1. 世界如何进入模型
2. 扩张如何被定价
3. 当前如何不被过去写坏

如果一个团队能先把这三个问题答清，再去谈功能模仿，通常会更接近 Claude Code 真正难被抄走的部分。

所以它管的不只是风险，也管有效行动力；也正因此，`07` 这种综合页真正值钱的地方不在再造术语，而在反复把六个平面压回这三个问题。

如果继续把这层关系写得更硬，这三张图至少还分别承担：

1. `82`
   - 暴露 Prompt same-world 的 next-hop 与 route pressure，不代签 witness ABI
2. `83`
   - 导出 governance failure order
3. `84`
   - 导出 current-truth reject order

而 `07` 的职责不是重复这些细节，而是保证它们始终被当成同一条 same-world compiler 的三种 consumers-facing 读法。

### 4.11 源码同时在写制度而不只是实现

如果继续沿命名、注释、dependency-free helpers 与 snapshot 逻辑往下追，会发现 Claude Code 的源码本身也在承载制度。

它不仅告诉机器：

- 怎么执行

还告诉人类：

- 为什么这样写
- 为什么这里不能轻易改
- 为什么未来要沿这个方向演化

## 5. 这一章怎样反过来约束蓝皮书前门

蓝皮书主线若要持续保持收束，应长期坚持四条约束：

1. `bluebook/` 根目录只写一跳能懂的主线结论，不把字段级细节塞回主线。
2. 真正复杂的机制继续下沉到 `architecture/`、`api/`、`philosophy/`、`guides/`。
3. 每新增一个“看起来很强的能力”，都要追问三次：
   - 它的运行时契约是什么。
   - 它的知识与状态从哪里来。
   - 它的产品边界和治理边界在哪里。
4. `00`、`02`、`navigation/01`、`navigation/02` 这类第一次进入会看到的文件，应先把三句高阶判断和公开镜像研究纪律前置给读者，而不是先把读者送回功能表和目录体感。

## 6. 一句话总结

Claude Code 最难被抄走的，不是某一层，而是它把提示词、知识层、协作、权限和生态边界全部纳入了同一条 same-world compiler。
