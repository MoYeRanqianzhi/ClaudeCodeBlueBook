# 设计哲学

## 0. 先把“设计哲学”读成制度设计

这一章如果只被读成“Claude Code 有很多高级设计原则”，仍然不够稳。

更准确的读法是：

1. 它首先在回答世界如何进入模型。
2. 它首先在回答扩张如何被定价。
3. 它首先在回答当前如何不被过去写坏。

后文只解释这三件事为什么成立，不再单独发明另一套主线术语。

所以后面这些“设计原则”真正要回答的，不是：

- Claude Code 又比别人多做了什么功能

而是：

- 它怎样把软件工程现场先制度化，再让模型进入这个现场

如果不先带着这层读法，这一章很容易重新退回：

- 世界定义需要被反复重谈
- 扩张只能靠事后拦截补洞
- 源码质量只剩目录观感

如果把 `03` 的制度设计再压成最短判断，也只剩一句：

- 本章只解释根入口已定的三条母线为什么成立，不在这里重发 owner 页的 `first test / counterfeit / minimum repair`。

这里还要先压住一个常见误读：`continuity` 不是第四条设计优点；它只是 Prompt `Continuation`、治理 `continuation pricing` 与源码质量 `recovery non-sovereignty / anti-zombie` 的共同时间轴。

如果后文任何原则解释还压不回这三条，它就还停在对象层或结果词层。

## 0.0 世界准入哲学只保留第一性原理与苏格拉底判断

如果只记一句：更值钱的不是 `same-world compiler` 这个词，而是 `verify / delegate / resume` 之后，later consumer 仍不必重答谁在定义世界、边界内哪些动作和工具仍合法；`same-world compiler` 只是这件事为什么成立的解释词。

若一段 Prompt 解释删掉 `next-hop / owner page` 之后就站不住，说明它还在管路由，不在讲 why；更硬的 acid test 只问 `compact / resume / handoff` 之后，同一 `compiled world verdict` 是否仍能沿同一 `lineage` 继续成立，并让 later consumer 不必再回答“谁在定义世界、边界在哪、工具还能不能用”。Prompt 线真正值钱的地方，也只是在 `verify / delegate / resume / handoff` 里把正确下一步压回已裁定边界，而不是再开一轮冷启动协商。
更硬一点说，这里保护的不只是 later consumer 的时间轴连续性，而是不同 consumer class 都只能继承同一份 `world-entry object`，而不能在接手时重新夺回 `world-definition source`。

治理线在这里只保留一个 acid test：

- 没有决策增益的检查，本身就是免费扩张。
- 如果这句还需要额外的 stage names、owner 顺序或排错矩阵才能成立，说明治理问题还没被压回 why，而是在重做导航层。

## 0.1 设计内涵不是三条优点，而是三种不可外包的制度分配

如果继续追问“Claude Code 的设计到底深在哪里”，更稳的回答不是：

- Prompt 看起来更能控制模型
- 安全看起来更会阻止坏事
- 源码看起来更整齐

而是它先把三种不可外包的制度分配提前制度化：

1. 谁配把当前现场编译成模型正在工作的那个世界。
2. 谁配批准动作、能力、上下文与时间继续扩张。
3. 谁配宣布现在仍是什么，哪些对象只能当恢复资产、消费者投影或 danger signal。

所以设计内涵真正深的地方，不在“功能比别人多”，而在：

- 它先把 `world-definition / pricing / stale-writer veto` 这三种权力写成 runtime 里的正式席位，再让模型、工具、人类和宿主在这些席位下行动。
- Prompt 线真正值钱的地方，因此不再是文案效果，而是不同 consumer、不同轮次与不同接手者能否继续继承同一份已编译世界判决，并把下一步动作选择压到已裁定边界内。
- 安全与省 token 因此不再是两套优化，而是扩张定价与 `authority lease` 是否仍由同一秩序结算。
- 源码先进性因此不再是目录美学，而是 present-truth veto、局部可反对性与下一次重构入口是否仍然单源。

如果 later maintainer 还需要第二组概括词才能复述这三条分配，说明同一个制度问题还在被重新协商；这时更该先补齐同一条制度判断的缺口，而不是再发第二套摘要词：Prompt 缺 same-world verdict，治理缺 repricing loop，源码质量缺 present-truth veto。
目录结构真正值钱的地方，也只是在降低 later consumer 重谈同一主语的次数，让后来的 verify、delegate、resume 与接手直接继承这些 artifact；删掉 route note 后如果结论仍成立，才算真正进入设计内涵。

如果 later maintainer 还回答不出“谁在定义世界、谁在批准扩张、谁在否决陈旧写入”，这页就仍停在高阶赞美，还没有进入设计内涵。

## 0.2 设计内涵的三次去魅

如果继续追问“怎么确认自己写的是设计内涵，而不是设计外观”，最稳的方式不是再换一组赞美词，而是先做三次去魅：

1. Prompt 去魅
   - 删掉“写得像专家”“更会下指令”这些修辞后，是否仍只剩一条硬判断：later consumer 不必重答世界定义、边界与继续资格。
   - 更硬一点说，`lawful forgetting / continuation qualification` 都只是 same-world test 落到时间轴后的故障标签；凡是接手者还要先重答世界、再决定动作的路径，Prompt 就已失败。
2. 治理去魅
   - 删掉 mode、modal、token 条、usage 面板后，是否仍只剩四问：最早 `unpaid expansion` 是什么、`repricing proof` 在哪、`lease checkpoint` 是否通过、旧 authority 是否已 cleanup。
3. 源码去魅
   - 删掉目录树观感、文件大小与作者说明后，是否仍能先说清谁在保护 `one writable present`、later maintainer 的第一条 veto 还在不在，而不是在 why 页里自己补第二套质量 verdict。

如果删掉这些投影后论证就站不住，这一章写的就还不是设计内涵，而只是设计外观。

如果三次去魅之后，页面仍需要靠“连续性系统”这类第四对象来补叙事，通常也说明这里写的还不是制度内核，而只是把时间轴误写成了独立平面。

## 0.3 源码组织首先是主权分配，不是功能分层

Claude Code 的目录之所以值得分析，不是因为它“分层得很漂亮”，而是因为它先把三种角色分开落进源码：

1. 谁定义现在
2. 谁只消费现在
3. 谁只能帮助恢复现在

所以源码结构真正暴露的不是功能库存，而是主权分配；目录只是这套主权分配在公开表面上的投影。

这里还要再多记一句：

- repo 里的 `CLAUDE.md`、slash commands、hooks、subagents、plugin marketplace 也不只是附属文档或使用素材，而是版本化的 operator-governance 工件；它们把“正确操作者行为”写进可审读、可 diff、可继承的公开 artifact，而不是继续留给作者口授。
- 这些公开 artifact 之所以算源码质量，而不只是文档卫生，是因为它们把正确操作与第一条反对路径版本化，降低的不是阅读负担，而是制度失忆、误改与过度主张成本。
- 但这些工件并不共享同一签字权：`CLAUDE.md` 这类 foundation 只配塑形上下文与偏好，permissions / sandboxing / hook veto 这类执行侧机制才配正式收紧与阻断；成熟 runtime 真正先进的地方，也正在于它把这组差异写成 later maintainer 不会混淆的公开层级，而不是把所有 operator artifact 都写成同级治理真相。

更硬一点看，这种主权分配要求“事件线真相”和“宿主当前真相”不能混写；若讨论已经进入 truth planes 或 writeback seam，统一回 `architecture/README`。

## 1. Claude Code 的强，不是单一模型能力，而是厚 runtime

源码最值得注意的地方是: 核心 loop 本身并不复杂，但外围 runtime 非常厚。

最小 loop 其实仍然是:

```text
messages -> model -> tool_use? -> execute -> append tool_result -> loop
```

真正让它强的，是这层 loop 外面包了:

- fast-path startup
- permission context
- transcript persistence
- compaction
- skill / command / tool 装配
- subagent / task system
- MCP / remote / bridge
- retry / fallback / recovery

证据:

- `claude-code-source-code/src/QueryEngine.ts:209-333`
- `claude-code-source-code/src/query.ts:365-467`
- `claude-code-source-code/src/query.ts:659-865`

这些 runtime 部件只有在共同守住同一份 `compiled world verdict` 时，才会被体验成 Prompt 很强；离开这点，它们只是 later consumer continuity 的支撑设施。

## 2. 设计原则一: 把“快”做成架构约束

Claude Code 到处都在为启动和首轮交互抢时间:

- `cli.tsx` 先 fast-path，再加载主程序
- `main.tsx` 里大量并行准备
- `init.ts` 里把非关键初始化延后
- skills/plugins 注册特地提前，以免并行加载时拿到空列表

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:28-33`
- `claude-code-source-code/src/entrypoints/cli.tsx:287-297`
- `claude-code-source-code/src/entrypoints/init.ts:44-47`
- `claude-code-source-code/src/main.tsx:1913-1929`

结论:

Claude Code 把“time-to-first-useful-action”视为一等指标，而不是后期优化项。

## 3. 设计原则二: 扩张先收费，再按诊断环决定失败语义

`buildTool()` 给工具的默认行为是保守的:

- 非并发安全
- 非只读
- 非破坏性
- 权限检查默认走统一 permission system

而 trust、permission rule、classifier、policy、managed settings 又在不同层重复加固。

证据:

- `claude-code-source-code/src/Tool.ts:743-792`
- `claude-code-source-code/src/entrypoints/init.ts:71-79`
- `claude-code-source-code/src/utils/permissions/permissions.ts:122-301`

这条原则更稳的写法只剩三句：

- 扩张必须先被单源定价
- runtime 决定收费顺序，模型不能自己免费扩张
- 若讨论已经退回投影替身或结果词，统一回治理入口与对应 owner README

结论:

Claude Code 不是“先放开再补洞”，而是“先收紧，再按上下文释放”。

更准确地说，这里不是“安全更严”这么简单，而是：

- 治理先外化当前真相，再批准当前扩张
- 没有决策增益的检查，本身就是免费扩张
- 失败语义不先按资产类型起步，而先看这次扩张有没有被正式定价与结算

## 4. 设计原则三: 一切能力都应该可装配

在源码里，command、skill、plugin、MCP、agent 都不是彼此隔离的产品功能，而是统一 runtime 里的可装配对象。

表现为:

- `getCommands()` 汇总多个来源
- `getTools()` / `assembleToolPool()` 动态装配工具
- skills 可以声明 hooks / allowedTools / model / effort / paths / agent
- plugin 可以同时提供 skills、hooks、MCP servers
- MCP 工具被转成统一 Tool 结构

证据:

- `claude-code-source-code/src/commands.ts:445-517`
- `claude-code-source-code/src/tools.ts:193-249`
- `claude-code-source-code/src/skills/loadSkillsDir.ts:185-261`
- `claude-code-source-code/src/plugins/builtinPlugins.ts:1-14`
- `claude-code-source-code/src/services/mcp/client.ts:1765-1833`

结论:

Claude Code 的可扩展性，不靠“插件中心”单点，而靠统一抽象层。

## 5. 设计原则四: 事件线真相不等于宿主当前真相

在 `QueryEngine.submitMessage()` 中，用户消息会在进入 query loop 前就先写 transcript，避免“请求中断后无法 resume”的问题。

证据:

- `claude-code-source-code/src/QueryEngine.ts:436-463`

再加上:

- compact boundary
- session info
- fork / resume / remote session
- `worker_status / requires_action_details / pending_action`

可以看出 transcript 在 Claude Code 中更像 event log，而不是 UI 历史。

这很关键，因为 agent 系统一旦跨 turn、跨 subagent、跨 remote，就必须让时间线和当前真相分层，而不是让宿主靠重放消息自己猜状态。若讨论已经进入 truth planes、current-truth writeback 或 host-facing surface，统一回 `architecture/README`。

## 6. 设计原则五: 大上下文不是解法，已定价工作集与合法外置才是解法

`query.ts` 不是单纯依赖模型上下文窗口，也不是在堆“压缩技巧”；它真正处理的是已定价工作集准入、合法外置 durable truth 与 continuation 续租检查。`snip / microcompact / context collapse / autocompact / reactive compact` 只是这些动作的手段：

- snip
- microcompact
- context collapse
- autocompact
- reactive compact

证据:

- `claude-code-source-code/src/query.ts:396-467`
- `claude-code-source-code/src/query.ts:1065-1085`

结论:

Claude Code 的思路不是“塞更多 token”，而是“把上下文当作有生命周期的工作集”。

## 7. 设计原则六: 并行是内建能力，不是高级玩法

多 Agent 在源码里不是隐藏彩蛋，而是明确的一等机制:

- `AgentTool` 有 isolation、background、team 等参数
- `Task` 系统统一本地 shell、local agent、remote agent
- `coordinatorMode` 直接把并行研究、串行写入写进调度原则

证据:

- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:90-101`
- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:282-316`
- `claude-code-source-code/src/Task.ts:6-29`
- `claude-code-source-code/src/coordinator/coordinatorMode.ts:198-237`

结论:

Claude Code 把“并行研究 + 汇总决策 + 定点执行 + 独立验证”当成默认工程流程，而不是偶尔使用的 trick。

## 8. 设计原则七: 远程能力必须是一等公民

很多工具把 remote 当附加模式处理，但 Claude Code 并不是。

源码里 remote 相关路径覆盖了:

- remote session config
- WebSocket event stream
- permission bridge
- remote-safe commands
- bridge environment
- heartbeat / reconnect / policy gating

证据:

- `claude-code-source-code/src/commands.ts:610-686`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:87-141`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-87`
- `claude-code-source-code/src/bridge/bridgeMain.ts:196-270`

结论:

Claude Code 的目标不是“本地 CLI”，而是“可以跨本地、云端、桥接环境流动的 agent runtime”。

## 9. 设计原则八: 产品实验与能力边界显式化

这个项目大量使用两类开关:

- 编译期开关: `feature('...')`
- 运行期开关: GrowthBook / env / policy

这意味着 Claude Code 的产品形态不是一个固定平面，而是一个按 build、账号、组织策略、环境上下文不断裁切的能力空间。

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:21-26`
- `claude-code-source-code/src/skills/bundled/index.ts:35-78`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:28-55`
- `claude-code-source-code/src/plugins/bundled/index.ts:17-23`

结论:

阅读 Claude Code 源码，不能只问“代码里有没有”，还必须问:

- 这是 build-time 还是 runtime gate
- public build 里是否保留
- 当前账号 / 组织策略是否允许

## 10. 设计原则九: 接手者若先重答世界，Prompt 就已经失败

本节只保留一句 why：Prompt 的价值不是文案强度，而是让 later consumer 不必再判一次“现在到底是什么世界”，可以直接沿同一份已授权边界继续动作；若接手者先重答世界而不是先继续已授权动作，Prompt 就已失败。

证据:

- `claude-code-source-code/src/utils/queryContext.ts:1-41`
- `claude-code-source-code/src/constants/prompts.ts:444-579`
- `claude-code-source-code/src/utils/systemPrompt.ts:27-121`
- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-95`

## 11. 设计原则十: 工程质量本身就是产品能力

很多系统把源码质量理解成：

- 代码卫生
- 风格统一
- 文件更短

Claude Code 更像把源码质量当作用户最终能力的一部分。

因为在 agent runtime 里，下面这些东西一旦写不好，就会直接外溢到产品体验：

- cache 抖动
- 权限失真
- 宿主状态不同步
- 长会话恢复失败
- 多 Agent 协作失控

所以这里的“工程质量”不是附属美德，而是产品能力本体。

更稳的写法不是在本章继续判 shared runtime correctness、证据梯度或 choke point 对象清单；本节只保留一句：工程质量会直接外溢成产品能力，本章不再另立源码质量入口卡。

## 12. 设计原则十一: 治理前门先问最早 unpaid expansion，安全/成本/体验只是投影

这一节只保留一句自校：不要把安全、成本、体验这些投影重新写成治理主语。

如果一个“安全 / 成本 / 体验”的解释还必须靠问题梯子、UI verdict 或 first reject path 才能站住，说明它已经离开 why 页，重新滑回 owner chain。

## 13. 设计原则十二: 运行时平面只是二级分析透镜，不是第一设计单位

把 prompt、安全、token、状态、协作与宿主放在一起看，确实会看见一组长期存在的 runtime plane；但它们更稳的读法不是第一设计单位，而是二级分析透镜。

至少包括：

- prompt/context plane
- permission/safety plane
- token economics plane
- state/recovery plane
- collaboration plane
- host/transport plane

这也解释了为什么它会同时呈现出：

- 世界准入不必被反复协商
- 扩张先被定价而不是先被暴露
- token 只继续花在仍有决策增益的地方

因为这些能力本来就在同一组运行时平面上协同工作。

但更进一步说，真正被设计的仍不是这些平面各自的“功能量”，而是三条母线先怎样分配主权：

- 谁负责世界进入模型
- 谁负责给扩张定价
- 谁负责维持当前世界不被过去与复杂度篡位

平面分析只是在对象层帮 later maintainer 看见这些主权怎样协同，不是新的顶层 first-answer。若继续把 `plane taxonomy` 当主语，三条母线就会再次退回 feature map。

## 14. 设计原则十三: 前台交互不是聊天壳，而是认知控制面

继续看 REPL 前台也会发现：

- sticky prompt
- transcript mode
- message actions
- teammate view
- queued commands
- search / selection / scroll semantics

这些都不是“终端 UI 小优化”，而是在解决：

1. 用户如何保持任务锚点
2. 用户如何在长历史里导航
3. 用户如何在多条执行链之间切换主语

所以 Claude Code 的前台不能只按“消息列表 + 输入框”理解，而应理解成：

- 人类对 runtime 的认知控制面

更具体地说，这个控制面至少由五个控制部件组成：

- `sticky prompt`
- `transcript mode`
- `message actions`
- `search-selection-scroll semantics`
- `UI transcript`

## 15. 设计原则十四: workflow 不是脚本，而是编排承载体

继续往下看 task、workflow progress、独立 transcript / worktree 边界之后，会发现 Claude Code 对 workflow 的理解并不是：

- 命令变体或多步 prompt 皮肤

而是：

- 一类正式运行承载体

这意味着它更在意：

- 长流程的生命周期
- 长流程的状态可观测性
- 长流程的归档与清理

而不只是“有没有多步自动化能力”。

把它继续压成承载语言，最小集合是：

- `local_workflow`
- `workflow_progress`
- `independent transcript`
- `worktree`
- `cleanup semantics`

命令只负责实例化这套承载体，生命周期、观测、归档与恢复都应由 workflow 自己负责。

## 16. 设计原则十五: 用户可见真相优于底层原始文本

从 transcript search、selection 与 sticky prompt 可以看出，Claude Code 还有一条很容易被忽略的前台原则：

- 当前台语义和底层原始文本冲突时，前台服从用户真正看见的真相

这就是为什么它会：

- 跳过某些 raw tool result 序列化文本
- 清理 system reminder
- 把特殊来源消息重新包装后再进入前台语义

这条原则让搜索、复制、导航与来源判断共享同一套认知真相。

更硬一点说：

- `UI transcript` 服务人类可见真相
- `protocol transcript` 服务模型侧协议合法性

两者相关，但不能互相冒充；否则系统就会重新滑回“看见什么”和“模型拿到什么”混成同一层真相的旧错误。

更稳的读法也不是要求三份文本逐字一致；若讨论已经进入 `message lineage` 或 handoff projection 的 owner 术语，统一回 `philosophy/84`。

## 17. 设计原则十六: 治理开关不是部署尾巴，而是输入边界

从 channels、managed settings、allowlist 与组织策略可以看出：

- 治理层并不是产品外壳，而是在改写 runtime 的输入边界

它决定的不是“功能按钮亮不亮”，而是：

- 哪些外部消息能进会话
- 哪些策略能覆盖本地配置
- 哪些对象被视为可信来源

所以治理不是外围限制，而是治理收费链本身的一部分。

## 18. 一句话收束

Claude Code 的设计哲学，最终更像：

- 先把世界准入、扩张定价与当前真相保护写成制度，再让运行时平面协同承接这些制度

而不是：

- 给模型不断外挂更多功能点

如果一定要把这种制度效力拆成工程语言，后面的运行时平面协同主要来自六个叠加项：

1. 一个足够小、足够稳的核心 loop。
2. 一套非常认真对待权限、恢复、持久化的外层 runtime。
3. 把 tool / skill / MCP / plugin / agent 放进统一抽象。
4. 把并行 delegation 视为默认路径，而不是补丁。
5. 把 remote、resume、compaction 当作常态，而不是异常流程。
6. 把复杂性写成正式结构，而不是团队默契。

## 19. 设计原则十七: lawful forgetting 不是信息堆叠

本节不新增 Prompt 首答（见 `0.0 / 10`），只保留一句压缩审读：压缩后若接手者不能直接继续同一已授权动作，就不是 lawful forgetting，而是把 `continue` 伪装成摘要；具体 continuation 资产载体统一回 `philosophy/84`。

## 20. 设计原则十八: runtime 本身是一部资源宪法

从权限、settings、budget continuation 与 context suggestions 可以看出，Claude Code 不先做“安全和成本优化”，而是在分配稀缺资源。更稳的 why 句只剩一句：模型不是资源的主人，runtime 才是。

## 21. 设计原则十九: 好架构不是更会重构，而是始终保留重构可能性

从 `query/config.ts`、`query/deps.ts`、leaf modules 与 snapshot 可以看出，Claude Code 的工程先进性还有一条更值钱的判断：真正成熟的系统，不是某次重构做得多漂亮，而是每次增长后仍保留下一次重构整套系统的可能性。本节不再展开重构资本的对象清单。

## 22. 设计原则二十: 人类接手路径也必须沿同一现场

本节只补一句：Claude Code 值钱的不只是让模型继续同一现场，也让人类接手仍沿同一现场；若人类接手还要先重谈主语，它压低的就只是表面 token，不是人机协调成本。

## 23. 设计原则二十一: 真正的自由是高行动力区间，而不是最大裸露面

从 permission mode、permission prompt、channel relay、deferred tools 与 externalization 可以看出，Claude Code 的自由观非常成熟：

- 不是最大原始自由
- 而是最大有效自由

也就是说，最好的约束不是消灭行动，而是：

- 让约束不再破坏行动

## 24. 设计原则二十二: 可读性本身也是治理界面

从命名、注释、dependency-free leaf modules 与 snapshot 解释可以看出，Claude Code 的源码还在做一件很硬的事：

- 让制度理由写进代码

这意味着可读性在这里不只是审美，而是：

- 防误改
- 防误扩展
- 防制度失忆

## 25. 设计原则二十三: continue verdict 不能脱离 same-world contract

本节不再新增 Prompt why；这里只保留一句自校：若 `continue` 被写成摘要接续，而不是已授权动作的继续，它就已经离开 same-world contract。

## 26. 设计原则二十四: 安全首先是资源定价，而不是事后拦截

本节只保留一句自校：若解释已经开始比较 mode、弹窗、门数量或 default continue，说明它已从 why 页退回投影层。

## 27. 设计原则二十五: 源码首先服务未来维护者，而不是只服务当前执行

本节不再重发源码质量 why；这里只补一句：命名、single choke point 注释与 seam 的价值，在于把反对权提前写进结构。

## 28. 设计原则二十六: same-world 边界判断不能退回句子强弱

本节不再重述 Prompt why；这里只保留一条自校：若判断退回“句子强弱”，`same-world compiler` 就又被误读成措辞优化。

## 29. 设计原则二十七: 安全与省 token 首先是治理顺序与失败语义

这一节不再列失败语义检查表，只保留一句自校：若安全与省 token 被写成两套优化，它就已退回治理投影层。

## 30. 设计原则二十八: 构建系统本身也在塑造源码秩序

从 external stubs 与恢复边界可以看出，Claude Code 的工程先进性还有一条更深判断：

- 构建系统本身也是架构工具

它不只决定“怎么打包”，还在反向塑造入口壳、依赖边界与恢复约束。源码质量的正式分流统一见 `0` 节第 3 条，不在这里重发 why / ladder / object 三层 next hop。

## 31. 设计原则二十九: 真正强的制度必须能自我审读，而不是只能自我证明

从 Prompt Constitution、治理顺序、构建系统塑形与多入口反查层可以看出，Claude Code 的设计内涵还能再收束一步：真正强的制度不只告诉你今天为什么成立，还把“它何时开始失真”一起写进制度本身。

更具体的审读协议统一见 `06`；本节不再另立固定顺序。

## 32. 设计原则三十: 真正强的制度必须能说清同题坏解为什么更差

从苏格拉底审读层可以看出，还有一条更苛刻的判断：真正强的制度，不只会给正解，还会给反例对照。若一个设计说不清同题坏解为什么更贵、更脆、更难恢复，它依然更像作者经验，而不是可迁移制度。

## 33. 设计原则三十一: 真正成熟的制度升级必须是保序迁移，而不是一次性重写

从迁移工单层可以看出，还有另一条更严的判断：真正成熟的制度升级，不先夸终态，而先证明增长不会牺牲当前真相、当前能力与恢复路径。更稳的自校，不是继续在这里列迁移步骤，而是反问：这次升级是否保住了可逆、可校正、可退回的改写秩序。

## 34. 设计原则三十二: 目录优化先自校是否又让同题多页首答

真正成熟的蓝皮书目录优化，先问的不是导航是否更长，而是：同一类第一问题，是否只在一个入口第一次被回答；同一条对象链，是否只在一个证据层第一次被展开；并行深挖之后，是否仍能回到同一问题主语。

只要 later maintainer 第一次进入时，仍可能在两个入口同时学到同一问题的第一答案，那种目录优化就仍只是路线整理，不是第一问题单源化。

更硬一点说，目录优化若要成立，至少要保住一组固定发言权：`README / userbook` 只定题，`philosophy` 只给 why-proof，`guides` 只给证据梯度，`architecture` 只给对象摘要，`navigation` 只补 artifact gap，`playbooks` 只发 execution verdict。

凡是某页同时偷拿两种以上发言权，都不是优化，而是在把 later consumer 再次送回“同题重学”。

如果争议已经从“这一页的 why 是否成立”变成“总控页之间的目录契约是否失真”，统一回 `meta/README`；那已经不是 why 页该自己解决的问题。

所以目录优化真正保护的，不是阅读舒适度，而是 later maintainer 第一次进入时不会学错第一问题。
