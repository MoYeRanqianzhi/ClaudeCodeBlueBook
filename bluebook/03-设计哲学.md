# 设计哲学

## 1. Claude Code 的强，不是单一模型能力，而是厚 runtime

源码最值得注意的地方是: 核心 loop 本身并不复杂，但外围 runtime 非常厚。

最小 loop 其实仍然是:

```text
messages -> model -> tool_use? -> execute -> append tool_result -> loop
```

真正让它强的，是这层 loop 外面包了:

- fast-path startup
- permission context
- transcript persistence
- compaction
- skill / command / tool 装配
- subagent / task system
- MCP / remote / bridge
- retry / fallback / recovery

证据:

- `claude-code-source-code/src/QueryEngine.ts:209-333`
- `claude-code-source-code/src/query.ts:365-467`
- `claude-code-source-code/src/query.ts:659-865`

## 2. 设计原则一: 把“快”做成架构约束

Claude Code 到处都在为启动和首轮交互抢时间:

- `cli.tsx` 先 fast-path，再加载主程序
- `main.tsx` 里大量并行准备
- `init.ts` 里把非关键初始化延后
- skills/plugins 注册特地提前，以免并行加载时拿到空列表

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:28-33`
- `claude-code-source-code/src/entrypoints/cli.tsx:287-297`
- `claude-code-source-code/src/entrypoints/init.ts:44-47`
- `claude-code-source-code/src/main.tsx:1913-1929`

结论:

Claude Code 把“time-to-first-useful-action”视为一等指标，而不是后期优化项。

## 3. 设计原则二: 安全默认值必须 fail-closed

`buildTool()` 给工具的默认行为是保守的:

- 非并发安全
- 非只读
- 非破坏性
- 权限检查默认走统一 permission system

而 trust、permission rule、classifier、policy、managed settings 又在不同层重复加固。

证据:

- `claude-code-source-code/src/Tool.ts:743-792`
- `claude-code-source-code/src/entrypoints/init.ts:71-79`
- `claude-code-source-code/src/utils/permissions/permissions.ts:122-301`

结论:

Claude Code 不是“先放开再补洞”，而是“先收紧，再按上下文释放”。

## 4. 设计原则三: 一切能力都应该可装配

在源码里，command、skill、plugin、MCP、agent 都不是彼此隔离的产品功能，而是统一 runtime 里的可装配对象。

表现为:

- `getCommands()` 汇总多个来源
- `getTools()` / `assembleToolPool()` 动态装配工具
- skills 可以声明 hooks / allowedTools / model / effort / paths / agent
- plugin 可以同时提供 skills、hooks、MCP servers
- MCP 工具被转成统一 Tool 结构

证据:

- `claude-code-source-code/src/commands.ts:445-517`
- `claude-code-source-code/src/tools.ts:193-249`
- `claude-code-source-code/src/skills/loadSkillsDir.ts:185-261`
- `claude-code-source-code/src/plugins/builtinPlugins.ts:1-14`
- `claude-code-source-code/src/services/mcp/client.ts:1765-1833`

结论:

Claude Code 的可扩展性，不靠“插件中心”单点，而靠统一抽象层。

## 5. 设计原则四: transcript 是系统真相，不只是聊天记录

在 `QueryEngine.submitMessage()` 中，用户消息会在进入 query loop 前就先写 transcript，避免“请求中断后无法 resume”的问题。

证据:

- `claude-code-source-code/src/QueryEngine.ts:436-463`

再加上:

- compact boundary
- session info
- fork / resume / remote session

可以看出 transcript 在 Claude Code 中更像 event log，而不是 UI 历史。

这很关键，因为 agent 系统一旦跨 turn、跨 subagent、跨 remote，就必须有稳定真相源。

## 6. 设计原则五: 大上下文不是解法，可恢复压缩才是解法

`query.ts` 不是单纯依赖模型上下文窗口，而是组合了多层压缩与恢复:

- snip
- microcompact
- context collapse
- autocompact
- reactive compact

证据:

- `claude-code-source-code/src/query.ts:396-467`
- `claude-code-source-code/src/query.ts:1065-1085`

结论:

Claude Code 的思路不是“塞更多 token”，而是“把上下文当作有生命周期的工作集”。

## 7. 设计原则六: 并行是内建能力，不是高级玩法

多 Agent 在源码里不是隐藏彩蛋，而是明确的一等机制:

- `AgentTool` 有 isolation、background、team 等参数
- `Task` 系统统一本地 shell、local agent、remote agent
- `coordinatorMode` 直接把并行研究、串行写入写进调度原则

证据:

- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:90-101`
- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:282-316`
- `claude-code-source-code/src/Task.ts:6-29`
- `claude-code-source-code/src/coordinator/coordinatorMode.ts:198-237`

结论:

Claude Code 把“并行研究 + 汇总决策 + 定点执行 + 独立验证”当成默认工程流程，而不是偶尔使用的 trick。

## 8. 设计原则七: 远程能力必须是一等公民

很多工具把 remote 当附加模式处理，但 Claude Code 并不是。

源码里 remote 相关路径覆盖了:

- remote session config
- WebSocket event stream
- permission bridge
- remote-safe commands
- bridge environment
- heartbeat / reconnect / policy gating

证据:

- `claude-code-source-code/src/commands.ts:610-686`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:87-141`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-87`
- `claude-code-source-code/src/bridge/bridgeMain.ts:196-270`

结论:

Claude Code 的目标不是“本地 CLI”，而是“可以跨本地、云端、桥接环境流动的 agent runtime”。

## 9. 设计原则八: 产品实验与能力边界显式化

这个项目大量使用两类开关:

- 编译期开关: `feature('...')`
- 运行期开关: GrowthBook / env / policy

这意味着 Claude Code 的产品形态不是一个固定平面，而是一个按 build、账号、组织策略、环境上下文不断裁切的能力空间。

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:21-26`
- `claude-code-source-code/src/skills/bundled/index.ts:35-78`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:28-55`
- `claude-code-source-code/src/plugins/bundled/index.ts:17-23`

结论:

阅读 Claude Code 源码，不能只问“代码里有没有”，还必须问:

- 这是 build-time 还是 runtime gate
- public build 里是否保留
- 当前账号 / 组织策略是否允许

## 10. 设计原则九: prompt 不是咒语，而是 runtime contract

继续深挖后会发现，Claude Code 的 prompt 强，不在于某段 system prompt 单独很强，而在于：

- system prompt 有静态 / 动态分层
- role prompt 有 coordinator / worker / fork / proactive 分化
- attachment 在替 prompt 承担动态注入
- cache 稳定性反过来约束 prompt 怎么写

证据:

- `claude-code-source-code/src/utils/queryContext.ts:1-41`
- `claude-code-source-code/src/constants/prompts.ts:444-579`
- `claude-code-source-code/src/utils/systemPrompt.ts:27-121`
- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-95`

结论:

Claude Code 的 prompt 魔力，本质上是“角色、工具、边界、缓存、上下文经济学一起写进 prompt contract”。

## 11. 设计原则十: 工程质量本身就是产品能力

很多系统把源码质量理解成：

- 代码卫生
- 风格统一
- 文件更短

Claude Code 更像把源码质量当作用户最终能力的一部分。

因为在 agent runtime 里，下面这些东西一旦写不好，就会直接外溢到产品体验：

- cache 抖动
- 权限失真
- 宿主状态不同步
- 长会话恢复失败
- 多 Agent 协作失控

所以这里的“工程质量”不是附属美德，而是产品能力本体。

证据:

- `claude-code-source-code/src/utils/lazySchema.ts:1-6`
- `claude-code-source-code/src/constants/systemPromptSections.ts:1-58`
- `claude-code-source-code/src/cli/transports/WorkerStateUploader.ts:1-118`
- `claude-code-source-code/src/services/api/promptCacheBreakDetection.ts:1-170`

## 12. 设计原则十一: 安全、成本与体验必须共用预算器

继续深挖安全与 token 经济之后，会发现 Claude Code 真正统一在做的，是预算管理。

- 权限系统约束动作空间
- token 经济约束上下文空间
- 前台状态与宿主回写约束认知噪音

这三者如果分开优化，结果通常会变成：

- 安全更硬，但不可用
- 成本更低，但上下文断裂
- 体验更顺，但边界失控

Claude Code 更成熟的地方在于：

- 把三者一起放进 runtime correction system

这也是为什么它会同时重视：

- deny rule
- policy gate
- tool result budget
- compact / reactive compact
- `permission_mode` / `worker_status` 外化

## 13. 设计原则十二: 真正的设计单位不是功能，而是运行时平面

继续深挖 prompt、安全、token、状态、协作与宿主之后，会发现 Claude Code 反复在设计的不是单个 feature，而是长期存在的 runtime plane。

至少包括：

- prompt/context plane
- permission/safety plane
- token economics plane
- state/recovery plane
- collaboration plane
- host/transport plane

这也解释了为什么它会同时呈现出：

- prompt 很强
- 安全边界很硬
- token 很节制
- 源码质量很稳定

因为这些能力本来就在同一组运行时平面上协同工作。

## 14. 设计原则十三: 前台交互不是聊天壳，而是认知控制面

继续看 REPL 前台也会发现：

- sticky prompt
- transcript mode
- message actions
- teammate view
- queued commands

这些都不是“终端 UI 小优化”，而是在解决：

1. 用户如何保持任务锚点
2. 用户如何在长历史里导航
3. 用户如何在多条执行链之间切换主语

所以 Claude Code 的前台不能只按“消息列表 + 输入框”理解，而应理解成：

- 人类对 runtime 的认知控制面

## 15. 设计原则十四: workflow 不是脚本，而是编排对象

继续往下看 task、workflow progress、独立 transcript / worktree 边界之后，会发现 Claude Code 对 workflow 的理解并不是：

- 一组可复用 prompt

而是：

- 一类正式运行对象

这意味着它更在意：

- 长流程的生命周期
- 长流程的状态可观测性
- 长流程的归档与清理

而不只是“有没有多步自动化能力”。

## 16. 设计原则十五: 用户可见真相优于底层原始文本

继续深挖 transcript search、selection、sticky prompt 之后，会发现 Claude Code 还有一条很容易被忽略的前台原则：

- 当前台语义和底层原始文本冲突时，前台服从用户真正看见的真相

这就是为什么它会：

- 跳过某些 raw tool result 序列化文本
- 清理 system reminder
- 把特殊来源消息重新包装后再进入前台语义

这条原则让搜索、复制、导航与来源判断共享同一套认知真相。

## 17. 设计原则十六: 治理开关不是部署尾巴，而是输入边界

继续深挖 channels、managed settings、allowlist 与组织策略之后，会发现：

- 治理层并不是产品外壳，而是在改写 runtime 的输入边界

它决定的不是“功能按钮亮不亮”，而是：

- 哪些外部消息能进会话
- 哪些策略能覆盖本地配置
- 哪些对象被视为可信来源

所以治理不是外围限制，而是 runtime contract 的一部分。

## 18. 一句话收束

Claude Code 的设计哲学，最终更像：

- 用一组可恢复、可组合、可迁移的运行时平面，去组织本来会失控的软件工程现场

而不是：

- 给模型不断外挂更多功能点

如果一定要把这种“魔力”拆成工程语言，它主要来自六个叠加项：

1. 一个足够小、足够稳的核心 loop。
2. 一套非常认真对待权限、恢复、持久化的外层 runtime。
3. 把 tool / skill / MCP / plugin / agent 放进统一抽象。
4. 把并行 delegation 视为默认路径，而不是补丁。
5. 把 remote、resume、compaction 当作常态，而不是异常流程。
6. 把复杂性写成正式结构，而不是团队默契。

## 19. 设计原则十七: prompt 首先是语义压缩器，不是信息堆叠器

继续往下看 session memory、prompt suggestion、stop hooks 与 tool result replacement 之后，会发现 Claude Code 的 prompt 还有一条更深的原则：

- 真正强的 prompt，不是带更多信息，而是带更高密度的行动语义

它更在意：

- 当前意图是否被保住
- 当前约束是否被保住
- 下一步可行动作是否被保住

而不是：

- 原始世界是否被原样拖进每一轮

## 20. 设计原则十八: runtime 本身是一部资源宪法

继续深挖权限、settings、budget continuation 与 context suggestions 之后，会发现 Claude Code 不只是在做安全和成本优化。

它更像在分配四类稀缺资源：

- 能力
- 时间
- 注意力
- 权威

这意味着：

- 模型不是资源的主人
- runtime 才是

模型只能在被授予的资源内推理和行动，而不能自己定义边界。

## 21. 设计原则十九: 好架构不是更会重构，而是始终保留重构可能性

继续深挖 `query/config.ts`、`query/deps.ts`、leaf modules 与 snapshot 之后，会发现 Claude Code 的工程先进性还有一条更值钱的判断：

- 真正成熟的系统，不是某次重构做得多漂亮，而是每次增长后仍保留下一次重构整套系统的可能性

它通过：

- config 冻结
- deps 抽离
- 状态机权威面
- leaf module 切边
- snapshot 一致性

持续给未来的系统重排留出口。

## 22. 设计原则二十: prompt 还在组织人类接手路径

继续深挖 sticky prompt、suggestion、session memory 与 teammate navigation 之后，会发现 Claude Code 的 prompt 架构不只在组织模型思考。

它还在持续组织：

- 人类下一步怎样接手
- 人类怎样给出低成本反馈
- 人类怎样在多执行链之间切回正确现场

所以它真正压低的不只是 token 成本，还有：

- 人类-模型协调成本

## 23. 设计原则二十一: 真正的自由是高行动力区间，而不是最大裸露面

继续深挖 permission mode、permission prompt、channel relay、deferred tools 与 externalization 之后，会发现 Claude Code 的自由观非常成熟：

- 不是最大原始自由
- 而是最大有效自由

也就是说，最好的约束不是消灭行动，而是：

- 让约束不再破坏行动

## 24. 设计原则二十二: 可读性本身也是治理界面

继续深挖命名、注释、dependency-free leaf modules 与 snapshot 解释之后，会发现 Claude Code 的源码还在做一件很硬的事：

- 让制度理由写进代码

这意味着可读性在这里不只是审美，而是：

- 防误改
- 防误扩展
- 防制度失忆

## 25. 设计原则二十三: prompt 首先是协作语法，而不是提示词文案

继续深挖 system prompt 权威顺序、protocol transcript 改写、sticky prompt、suggestion 与 session memory 之后，会发现 Claude Code 的 prompt 还有一层终局判断：

- prompt 首先是协作语法

它真正组织的是：

- 当前谁在发言
- 当前主语是否稳定
- 下一步如何继续
- 压缩与恢复后如何接手

所以 prompt 魔力不在措辞本身，而在：

- runtime 先把意图编译成一套跨当前、下一步与接手后的协作语法

## 26. 设计原则二十四: 安全首先是资源定价，而不是事后拦截

继续深挖 permission mode、审批竞速、deferred tools、tool result externalization 与 token continuation 之后，会发现 Claude Code 的安全观也可以再收束一步：

- 安全首先是资源定价

它在统一安排的不是若干“禁止规则”，而是：

- 哪些动作以什么价格被允许
- 哪些能力以什么价格进入当前可见面
- 哪些结果值得继续占用主上下文
- 哪些长回合值得继续支付时间与 token

所以真正成熟的安全，不是更会拦，而是：

- 让行动、能力、上下文与时间被纳入同一资源秩序

## 27. 设计原则二十五: 源码首先服务未来维护者，而不是只服务当前执行

继续深挖 `DANGEROUS` 命名、single choke point 注释、leaf module、config / deps seam 与 snapshot 语义之后，会发现 Claude Code 的源码先进性还有一条更根的判断：

- 未来维护者也是正式消费者

这就是为什么它会：

- 把风险写进命名
- 把制度理由写进注释
- 把共享规则压成小而硬的单一真相文件
- 把未来抽离点提早切成 seam

因此它的先进性不在静态分层本身，而在：

- 它让未来维护、未来重构与当前执行共享同一套边界设计

## 28. 设计原则二十六: prompt 首先是一部宪法，而不是一段文案

继续深挖 section boundary、角色优先级链、compact / recovery 与 prompt diff 之后，会发现 Claude Code 的 prompt 还有一条更高阶的判断：

- prompt 首先是一部受治理的宪法

它在规定的不是“怎么说更强”，而是：

- 哪些 section 构成正式 prompt 真相
- 谁有权覆写当前角色身份
- 删掉什么后系统仍然合法
- prompt 为什么稳定、为什么失稳

## 29. 设计原则二十七: 安全与省 token 首先是治理顺序与失败语义

继续深挖 permission order、selective fail-open/fail-closed、auto fallback 与 circuit breaker 之后，会发现 Claude Code 的安全与省 token 还有一条更高阶的判断：

- 成熟系统首先设计的是治理顺序与失败语义

它要回答的不是“多拦一点”或“再省一点”，而是：

- 哪些检查必须先发生
- 哪些失败应立即阻断
- 哪些自动化应退回人工
- 哪些判断已经没有继续花 token 的增益

## 30. 设计原则二十八: 构建系统本身也在塑造源码秩序

继续深挖 external stubs、portable shadow entry、transport shell、薄 registry 与 stale-state 治理之后，会发现 Claude Code 的工程先进性还有一条更深判断：

- 构建系统本身也是架构工具

它不只决定“怎么打包”，还在决定：

- 哪些依赖边能进入发布面
- 哪些入口能安全导入
- 哪些 transport 分叉必须被收进统一外壳
- 哪些恢复约束会反向塑造平时的状态写法

## 31. 设计原则二十九: 真正强的制度必须能自我审读，而不是只能自我证明

继续深挖 Prompt Constitution、治理顺序、构建系统塑形与多入口反查层之后，会发现 Claude Code 的设计内涵还能再收束一步：

- 真正强的制度必须能自我审读

它不只告诉你：

- 为什么这套设计今天成立

还提前安排了：

- prompt 失灵时该先反问主语、共享前缀、边界还是合法遗忘
- 安全与省 token 失真时该先反问输入边界、决策增益还是可撤销自动化
- 源码开始说谎时该先反问权威面、恢复资产、transport shell 还是 zombie 风险

所以 Claude Code 值得学的，不只是：

- 把设计写成运行时制度

还包括：

- 把质疑这套制度是否仍成立的问题，也一并写进制度本身

## 32. 设计原则三十: 真正强的制度必须能说清同题坏解为什么更差

继续深挖苏格拉底审读层之后，还会发现一条更苛刻的判断：

- 真正强的制度，不只会给正解，还会给反例对照

因为如果一个设计只能描述：

- 自己为什么成立

却不能同时说明：

- 同一个问题上，最容易被误写成什么
- 那些写法为什么会更贵、更脆、更难恢复
- 正确改写路径从哪里开始

那么它依然更像作者经验，而不是可迁移制度。

这也是为什么 Claude Code 的设计值得继续往“坏解法 / 正解 / 改写路径”方向写下去：

- 只有能说清坏解法为什么坏，制度边界才真正被刻出来
