# 会前体检与最短恢复路径：把误伤前移成可解释的预检系统

## 1. 为什么要单独补这一章

前面的章节已经说明了很多限制并不等于处罚。  
但在真实体验里，用户依然很容易在“已经开始干活之后”才意识到：

- 自己缺的不是权限，而是 full-scope login
- 自己不是没订阅，而是权益没激活到当前身份路径
- 自己不是彻底失败，而是高安全会话缺少刚登录后的可信设备窗口

这类问题有一个共同点：

不是系统完全看不见，而是系统把太多恢复信息留在事后暴露。

因此，这一章研究的重点不是“系统如何拒绝”，而是：

- 哪些事实其实可以在会前就体检出来
- 哪些恢复动作本该有更短路径
- 为什么这件事对高波动环境用户尤其重要

## 2. 先说结论

从源码看，Claude Code 已经具备不少“会前体检”和“最短恢复路径”的零件，只是这些零件现在分散在不同模块里：

1. `getBridgeDisabledReason()` 已能把 Remote Control 的资格缺口拆成可执行诊断。
2. `useCanSwitchToExistingSubscription()` 已能识别“你有权益，但当前没激活到这个登录路径”。
3. `errors.ts` 已把 token revoked、组织不允许、通用 auth 错误、模型不可用、refusal 等拆成不同恢复文案。
4. `trustedDevice` enrollment 明确依赖“fresh login 后的短窗口”，说明某些高安全能力其实天然需要会前体检。

这说明一个更深的判断：

成熟平台不应只在失败后解释，更应在高价值操作前做 preflight，  
把“误伤体感”前移成“可解释的预检结果”。

## 3. 源码里已经出现的 preflight 零件

### 3.1 Remote Control 资格其实已经能被提前拆开

`bridgeEnabled.ts` 的 `getBridgeDisabledReason()` 不是一个简单的布尔判断。

它已经能区分：

- 没有 claude.ai 订阅
- 不是 full-scope login token
- 拿不到组织画像
- gate 没开放
- build 本身不支持

这实际上就是一个准 preflight 系统。  
它说明平台已经知道：对高安全能力来说，“为什么不能用”必须比“不能用”更重要。

### 3.2 订阅未激活也已经有最短恢复提示

`useCanSwitchToExistingSubscription.tsx` 很关键：

- 它会识别你在 Console 侧其实已经有 Pro/Max 订阅
- 但当前并没有真正用订阅路径登录
- 然后直接提示 `/login to activate`

这说明平台在某些问题上已经不是“让用户自己猜”了，  
而是明确给出最短恢复动作。

它也进一步证明：

- 很多“权限没了”的体感，不是处罚
- 而是权益路径没有正确对齐

### 3.3 高安全可信设备其实天然要求会前检查

`trustedDevice.ts` 暴露了一个非常重要的工程事实：

- trusted device enrollment 依赖 fresh login 后的短时间窗口
- 太晚才做 lazy enrollment，会因为 `stale_session` 失败

这意味着：

- 高安全能力的某些前提并不是“随时都能补”
- 它们更适合在刚登录、环境稳定、网络连续的时候完成

如果用户是在长时间会话后、跨设备后、跨网络后才首次触发这类能力，  
误伤体感就会显著增强。

### 3.4 API 错误层已经具备最短恢复路径的雏形

`services/api/errors.ts` 不只是把 401/403/404 打印出来。  
它已经区分：

- `token_revoked`
- `oauth_org_not_allowed`
- 通用 `auth_error`
- model not available
- refusal

这说明恢复路径其实已经部分被系统掌握：

- 有些问题该去 `/login`
- 有些问题该换模型或找管理员
- 有些问题是策略拒绝，不是认证失效

所以真正缺的不是“有没有恢复线索”，而是“这些线索是否足够前置、足够可组合”。

## 4. 为什么中国和高波动环境用户更需要 preflight

更稳妥的技术表达仍然不是“平台一定针对某地区”，而是：

高波动环境更容易让这些 preflight 前提在真正使用前被破坏。

例如：

1. 登录后隔了很久、又切过网络，才首次进入高安全链路。
2. 权益本来存在，但当前身份路径没有正确激活。
3. full-scope login 与组织画像在跨环境切换后不再连续。
4. 原本该在稳定环境里一次做完的可信设备建立，被拖到失败之后才发现缺失。

这类问题都不是“系统更会封你”，而是“系统的前提条件更容易在你的环境里失稳”。

因此，对这类用户最有价值的保护，并不是寻找绕过方式，  
而是让高价值能力的关键前提在真正工作前就被确认。

## 5. 面向用户的最短恢复路径框架

下面只讨论合规恢复，不讨论规避平台策略。

### 5.1 看到“已有权益但没激活”

典型线索：

- 系统提示已有订阅
- 提示 `/login to activate`

更短动作：

- 先用正规登录路径激活当前身份
- 不要先切别的 token 和别的 provider

### 5.2 看到 Remote Control / bridge 不可用

典型线索：

- requires subscription
- full-scope login token required
- unable to determine organization
- not yet enabled

更短动作：

1. 先看自己到底缺的是哪一项
2. 如果是 full-scope 或组织画像问题，优先正规重新登录
3. 如果是资格未开放，就不要继续把它当处罚问题试错

### 5.3 看到高安全会话链路不稳定

典型线索：

- `auth_failed`
- 重连
- trusted device 相关问题

更短动作：

1. 不要边失败边换设备和网络
2. 优先在主力设备、稳定网络上重新建立连续性
3. 如果刚登录后更容易成功，那不是“系统在放水”，而是高安全链路本来就依赖 fresh session 前提

### 5.4 看到 `needs-auth`

更短动作：

- 只在该连接域内恢复
- 不要把整套账号都一起重置

### 5.5 看到 rate limit / extra usage / model not available

更短动作：

- 分别按计费、额度、模型可用性处理
- 不要误判成风控处罚

## 6. 面向平台的 preflight 设计蓝图

如果要把误伤再降一截，平台最值得补的不是更多 gate，而是一个显式 preflight 面。

至少应覆盖：

1. 当前权益是否已激活到当前身份路径
2. 当前是否为 full-scope login
3. 当前组织画像是否完整
4. trusted device / 高安全会话前提是否就绪
5. 当前网络和会话连续性是否适合进入高价值能力

理想状态下，用户在真正进入 Remote Control / bridge / 高安全操作之前，  
就应该先看到一张体检结果，而不是在中途撞墙。

## 7. 用苏格拉底式追问再压一层

### 问：如果系统已经能在事后解释，为什么还要做会前体检？

答：因为事后解释的成本更高。

一旦用户已经开始工作、已经进入故障窗口，再解释就会混入：

- 情绪
- 变量切换
- 证据污染
- 高噪声重试

而会前体检可以在这些噪声出现前就把问题暴露出来。

### 问：如果体检本身也可能误判，会不会又增加一个误伤层？

答：会，但代价仍然可能更低。

因为 preflight 的目标不是替代真实裁决，  
而是尽早暴露“你最可能缺的前提”。

哪怕它只是把问题缩小到：

- 权益路径
- 组织画像
- full-scope login
- trusted device

也比让用户在失败后自己猜要更公平。

### 问：为什么这件事在哲学上重要？

答：因为真正成熟的平台，不只是在“拒绝时保持正确”，还应在“拒绝前降低不必要的伤害”。

## 8. 给构建者的技术启示

1. 高价值能力不要只做 post-failure explanation，也要做 preflight。
2. 恢复路径应尽量短，不要让用户自己在多种可能原因间猜测。
3. 对 fresh session、trusted device 这类时效性前提，应显式暴露，而不是埋在失败语义后面。
4. 权益未激活、组织不允许、认证失效、模型不可用、额度耗尽，必须持续分开建模。

## 9. 一句话总结

把误伤前移成可解释的预检结果，比把所有解释都留到失败之后，更能保护用户，也更能体现平台治理的成熟度。
