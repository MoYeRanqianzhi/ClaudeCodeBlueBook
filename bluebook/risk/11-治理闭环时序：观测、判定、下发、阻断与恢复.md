# 治理闭环时序：观测、判定、下发、阻断与恢复

## 1. 这章回答什么问题

前面的章节大多是在拆模块。  
这一章回答的是一个更动态的问题：

Claude Code 的风控和能力治理，究竟是如何在时间上运转起来的？

## 2. 先说结论

从当前源码看，这套系统至少是一个六段式闭环：

1. 识别主体
2. 观测主体
3. 判定资格
4. 下发边界
5. 阻断或放行动作
6. 处理失败与恢复

如果只看到第 5 步，就会把它误读成单纯的本地权限系统。  
如果只看到第 3 步，就会把它误读成单纯的账号封控。  
真正的本质，是这六步的连续回路。

如果再往下压一层，这条回路也不是单纯流程图，而是统一定价控制面在时间轴上的结算顺序：

1. 识别 / 观测在给主体资格定价
2. 判定 / 下发在给权威与能力边界定价
3. 阻断 / 放行在给动作执行定价
4. 失败 / 恢复在给连续性与支持成本定价

## 3. 第一段：识别主体

时序起点不是“用户发出命令”，而是系统先确定：

- 当前会话走哪种身份源
- 该身份属于哪个账号、组织和订阅面
- 当前设备和运行环境是否处在正确上下文里

这是整条回路的锚点。  
如果主体识别错了，后面所有资格判断、策略下发和错误解释都会偏掉。

## 4. 第二段：观测主体

主体被识别后，系统会继续把它变成可计算对象：

- session
- device
- account
- organization
- subscription
- rate-limit tier

这些信息随后进入 telemetry 和 GrowthBook 语境。  
也就是说，平台不是面对一个模糊用户，而是在面对一个被结构化的运行实体。

## 5. 第三段：判定资格

有了主体画像之后，系统才开始回答：

- 这个主体是否具备某能力资格
- 当前组织是否允许这条能力链路
- 当前远程会话是否属于更高安全层级
- 当前连接是否需要 step-up auth 或重新认证

这一层决定的是“你能不能进入某个能力域”，还不是“这一步动作能不能执行”。

## 6. 第四段：下发边界

资格判定之后，边界会通过几种机制向客户端沉降：

- GrowthBook gates
- policy limits
- remote managed settings
- managed env

这一步的关键不在配置本身，而在于：

- 策略是运行时数据，不是硬编码常量
- 边界可以热更新
- 高风险变更会要求更强确认

## 7. 第五段：阻断或放行动作

边界下发后，真正的动作执行才开始面临本地仲裁：

- permission rules
- safety checks
- auto mode classifier
- sandbox
- trust
- trusted device

这一步做的是动作治理，不是主体治理。  
它处理的是“这次能不能执行”，而不是“你这个账号是否永久失格”。

## 8. 第六段：失败与恢复

高级系统不会把所有失败都当成“结束”。  
源码里能看到多种恢复与收口路径：

- OAuth 401 refresh
- cross-process refresh lock
- bridge reconnect
- stale cache fallback
- needs-auth cache
- step-up auth path
- fatal exit / auth_failed / failed 的区分

这说明风控系统的成熟度不只体现在判定准不准，也体现在：

失败发生后，系统是否能阻止噪声扩散，并把错误重新压回到正确的治理层。

也就是说，恢复不是“额外善后”，而是：

- 继续决定这条执行链还值不值得被续租

## 9. 这条闭环为什么先进

因为它同时具备三种常见系统很难兼得的特性：

1. 可分层
2. 可恢复
3. 可热更新

很多系统只能做到其中一个：

- 要么很严格，但一出错就把用户全部打死
- 要么很灵活，但边界模糊、治理不稳
- 要么能远程下发，但本地没有动作级防火墙

Claude Code 的设计更像是在追求三者兼得。

## 10. 从这条闭环倒推，什么最容易造成误伤

最容易误伤的，不是某一步单点错误，而是多段连续性同时变差：

- 身份源混乱
- 组织归属不清
- 设备和网络频繁切换
- 远程高安全会话叠加 auto mode 或高权限执行
- 401/403/needs-auth/policy denied 被混为一谈后反复试错

所以对用户来说，保护自己最有效的办法依然不是“躲系统”，而是减少闭环中的连续性破坏。

## 11. 一句话总结

Claude Code 的风控不是某个点，而是一条回路。  
只有把观测、判定、下发、阻断和恢复放到同一条时序里，才能真正理解它的设计本质。
