# 官方路径、云厂商路径与兼容入口的能力语义差清单：哪些只是能跑，哪些更接近等价

## 1. 为什么还要单独写这个收尾章

前面几章已经分别回答了：

- 该如何判断中国用户与中转站相关叙事的证据边界
- 如何把中转站依赖收敛成可控退出权
- 为什么中国用户现实里经常是在买入口，而不是直接买模型

但还缺最后一层：

`既然官方路径、Bedrock / Vertex / Foundry、第三方 relay、国内 Claude-compatible 入口都“能让 Claude Code 跑起来”，它们在语义上到底是不是同一回事？`

这一章就是这组板块的细致收尾。  
它回答的不是“能不能接”，而是：

1. 哪些只是传输层可用
2. 哪些更接近身份与资格层等价
3. 哪些只是工作流层替身
4. 用户应该按什么语义维度选路，而不是只按价格选路

## 2. 先说边界

这一章不提供：

- 绕过地区限制的方法
- 规避风控或隐藏主体的方法
- 把非第一方入口伪装成第一方入口的办法

这一章只讨论：

- 部署形态差异
- 身份语义差异
- 功能语义差异
- 支持、责任与退出语义差异

## 3. 先说结论

最重要的结论只有五条：

1. `能跑` 不等于 `身份等价`
2. `API 兼容` 不等于 `订阅资格等价`
3. `Claude-compatible` 不等于 `claude.ai entitlement 等价`
4. 官方云厂商路径通常比任意第三方 relay 更接近正式契约
5. 对中国用户而言，最佳策略不是寻找“最便宜且号称最像官方”的入口，而是先分清自己真正缺的是哪一层语义

换句话说：

`终端里都叫 Claude Code，不代表你买到的是同一种能力语义。`

## 4. 从第一性原理看，“等价”至少分四层

用户常把“这个入口也能用 Claude Code”理解成“它和官方差不多”。  
但从第一性原理上看，至少要把等价拆成四层。

### 4.1 传输层等价

也就是：

- 请求能不能发出去
- 模型能不能回结果
- 基本 API 形状能不能对上

这一层最容易被兼容入口替代。

### 4.2 认证层等价

也就是：

- 你到底是用什么 token 在认证
- 这个 token 属于谁
- 是 API key、gateway token，还是 claude.ai OAuth 身份

这一层就已经不是所有兼容入口都能替代。

### 4.3 资格层等价

也就是：

- 谁在承认你有权使用某项能力
- 这项资格来自订阅、云账户、组织策略还是 relay 运营方
- `/login`、`/status`、Remote Control 一类高阶能力到底认谁

这一层更不可能被“只是改个 URL”自动抹平。

### 4.4 治理层等价

也就是：

- 出问题时谁负责解释
- 谁保留成本、审计和支持证据
- 谁能真正恢复、关闭或批准某项能力

这一层决定了你在事故当天面对的到底是“产品”，还是“一串互相踢皮球的中间层”。

## 5. 现实里应区分的四类路径

### 5.1 Anthropic 第一方直接路径

典型特征：

- 第一方 Anthropic host
- 第一方登录与订阅语义
- 第一方支持与资格解释

这是最接近完整第一方契约的一类。

### 5.2 官方支持的云厂商路径

典型特征：

- Bedrock / Vertex / Foundry 这类被官方文档明确讨论的 provider 路径
- 提供商身份、计费、审计与合规语义更正式
- 不是 claude.ai 订阅语义，但也不是任意 relay 语义

它们通常比随意的 `ANTHROPIC_BASE_URL` relay 更接近“正式部署形态”。

### 5.3 第三方 LLM gateway / relay 路径

典型特征：

- 通过 `ANTHROPIC_BASE_URL` 或 provider-specific base URL 改写上游
- 由 gateway 处理部分认证、预算、审计或路由
- Anthropic 官方文档明确承认这种形态存在，但不会为任意具体第三方背书

这类路径的本质是：

`中心化接入层。`

### 5.4 国内 Claude-compatible 兼容入口路径

典型特征：

- 明确宣传 Claude Code 兼容
- 提供模型映射、token helper 和更低摩擦接入
- 工作流上尽量兼容 Claude Code，但身份与资格语义并不自动等于 Anthropic 第一方

这类路径更适合被理解成：

`Claude Code 工作流替身。`

## 6. 能力语义差清单

| 维度 | 第一方 Anthropic 直连 | 官方云厂商路径 | 第三方 gateway / relay | 国内 Claude-compatible 入口 |
| --- | --- | --- | --- | --- |
| 上游入口 | 第一方 Anthropic host | 官方支持 provider 模式 | 通过 gateway / proxy 改写上游 | 通过兼容端点改写上游 |
| 认证语义 | 最接近 claude.ai / Anthropic 原生语义 | provider 身份与云凭证语义 | gateway token / helper / 代理认证常见 | 兼容 token 与自定义入口常见 |
| 资格来源 | Anthropic 账户、订阅、组织 | 云账户、组织、provider 权限 | relay 或组织自定义策略 | 兼容提供方自身套餐或映射策略 |
| 高阶能力语义 | 最完整 | 部分正式 provider 语义 | 常出现部分等价 | 更像工作流兼容，不是完整第一方 entitlement |
| Beta / `tool_reference` | 最容易保持一致 | 取决于 provider 路径，但语义更正式 | 很多代理默认不完整支持 | 取决于兼容实现，不能默认当作完整等价 |
| 支持路径 | Anthropic / 管理员路径更清晰 | provider + 管理员路径较清晰 | 责任边界开始碎片化 | 提供方、上游、用户三方边界更复杂 |
| 成本与审计 | 第一方产品面与支持面 | provider 原生账单与审计 | 依赖 gateway 面板质量 | 依赖兼容提供方公开能力 |
| 退出成本 | 最低 | 中等 | 中高，常伴随本地配置扩散 | 中高，且容易形成工作流依赖 |

这张表真正想表达的是：

`从左往右，通常是接入摩擦下降，但语义偏差、责任偏差和退出成本会上升。`

## 7. 源码和官方文档为什么支持这个结论

### 7.1 客户端确实把“上游可配置”当成正式现实

从：

- `src/utils/apiPreconnect.ts`
- `src/services/api/filesApi.ts`
- `src/utils/model/providers.ts`

可以清楚看出：

- Claude Code 会读取 `ANTHROPIC_BASE_URL`
- 也支持 Bedrock / Vertex / Foundry 这样的 provider 路径

这说明 gateway、relay、兼容入口不是凭空幻想，  
而是客户端正式承认的现实部署形态。

### 7.2 但源码也明确区分第一方 host 与第三方 host

`src/utils/model/providers.ts` 不只是“支持改 URL”，  
还专门判断某个 `ANTHROPIC_BASE_URL` 是否仍属于第一方 Anthropic host。

这意味着设计者并没有把：

- 可配置上游
- 第一方语义

当作同一回事。

### 7.3 认证语义在源码里就是被明确分层的

`src/utils/auth.ts` 明确表现出：

- 第三方 provider 场景
- 外部 `ANTHROPIC_AUTH_TOKEN`
- `apiKeyHelper`
- claude.ai OAuth

是不同的认证来源。

它还直接指出，一旦进入外部 token / helper / 第三方 provider 场景，  
第一方 Anthropic auth 逻辑会被关闭或绕开。

这已经足够说明：

`兼容 token 可以让你请求成功，但不自动让你获得第一方身份语义。`

### 7.4 高阶能力最难被兼容的是“资格面”，不是“HTTP 面”

`src/bridge/bridgeEnabled.ts` 写得非常直白：

- Remote Control 需要 claude.ai subscription
- 它依赖 claude.ai OAuth token
- Bedrock / Vertex / Foundry、gateway deployment、env-var API key、Console API login 都不属于同一 entitlement 语义

这意味着很多用户最容易误判的一点恰恰是：

`把“我也能请求模型”误认成“我也拥有同样的高阶资格”。`

### 7.5 第三方 relay 常见的是“部分兼容”，不是“全部等价”

`src/utils/toolSearch.ts` 直接说明：

- `tool_reference` 这类 beta 形态很多 `ANTHROPIC_BASE_URL` 代理通常不支持
- 当 base URL 不是第一方 host 时，工具搜索会更保守

这说明 relay / proxy 的真实工程定义更准确地说是：

`部分工作流兼容层。`

而不是：

`完整官方语义镜像。`

### 7.6 本地配置传播会让入口选择沉淀成长期依赖

`src/utils/worktree.ts` 会把 `.claude/settings.local.json` 复制进 worktree。  
这意味着一旦某个 gateway / 兼容入口被写进本地配置，它会沿 worktree 继续扩散。

所以入口选择不是一次性按钮，  
它会逐渐沉淀成：

- 本地默认入口
- 团队习惯
- 会话恢复路径
- 未来退出成本

## 8. 用户最容易混淆的五件事

### 8.1 把“Claude-compatible”误写成“Anthropic 等价”

更准确的说法应当是：

- API 形状兼容
- 宿主工作流兼容
- 不代表身份与资格完全等价

### 8.2 把“能配置 `ANTHROPIC_BASE_URL`”误解成“官方默认支持所有 relay 等价”

官方文档承认 gateway 是架构形态，  
但官方也明确说第三方代理可能过时、也不会被 Anthropic 认可、维护或审计。

### 8.3 把“云厂商路径”与“任意 relay”混成同一层

Bedrock / Vertex / Foundry 这类路径虽然不等于 claude.ai 订阅语义，  
但它们仍属于被官方明确支持的正式 provider 形态。

这和任意第三方 relay 不是同一层。

### 8.4 把“我能请求模型”误判成“我能拥有同样的高阶控制面”

真正最难替代的通常不是普通推理请求，  
而是：

- OAuth 身份面
- 订阅 entitlement
- Remote Control 一类高阶资格面

### 8.5 把“低价”误判成“总成本更低”

如果一个入口：

- 支持面更碎
- 成本面更不透明
- 本地配置更容易扩散
- 退出更麻烦

那它的总成本可能并不比贵一点但契约清晰的路径更低。

## 9. 中国用户应该怎样按语义选路

### 9.1 如果你需要第一方身份与最高确定性

优先看第一方 Anthropic 直连路径。  
你真正买的是：

- 第一方身份面
- 第一方资格解释
- 第一方支持路径

### 9.2 如果你需要企业合规、采购与正式 provider 契约

优先看 Bedrock / Vertex / Foundry 这类官方支持路径。  
你真正买的是：

- 正式 provider 语义
- 云侧合规与审计
- 更可解释的企业部署契约

### 9.3 如果你需要中心化预算、审计和统一接入

gateway 可以成立，  
但你应明确接受一个事实：

`你买到的是中心化接入层，不是自动完整的第一方身份语义。`

### 9.4 如果你需要更低摩擦、更低试错成本和更低价格

国内 Claude-compatible 路径也可以成立，  
但它更准确的产品定义是：

`低摩擦工作流替身。`

它更适合被当作：

- 工作流学习入口
- 成本优化入口
- 迁移中间层

而不是被无差别写成：

`Claude 官方等价入口。`

## 10. 作为这一板块的收尾，61-64 四章到底在说什么

可以把这一组章节压成四句最短的话：

1. `61`：先把事实、观察和传闻分开
2. `62`：再把入口依赖变成可退出依赖
3. `63`：看清中国用户买的往往是入口，而不是抽象模型
4. `64`：最后把各种路径的能力语义差清楚，避免把“能跑”误写成“等价”

如果把这四章合起来，  
这一板块最成熟的最终结论就是：

`中国用户真正需要的不是神化某一个入口，而是能够看清不同入口到底替代了哪一层、没有替代哪一层，并始终把退出权和资产主权留在自己手里。`

## 11. 苏格拉底式追问

### 问：为什么“能跑”不是最重要的问题？

答：因为真正昂贵的差异常常不在第一次请求是否成功，  
而在身份、资格、支持和退出上。

### 问：为什么官方云厂商路径要和任意 relay 区分开？

答：因为前者属于被官方明确支持的 provider 语义，  
后者通常只是部分兼容的接入层。

### 问：为什么兼容入口最容易被高估？

答：因为用户看到了宿主没变、命令没变、模型似乎也能回，  
于是就误以为治理、资格和高阶能力也没有变。

### 问：为什么中国用户尤其需要这一章？

答：因为中国用户最容易被迫在多种非第一方路径之间做现实选择，  
如果不把语义差分清，就会在事故当天才发现自己买到的其实不是同一种东西。

## 12. 一句话总结

官方路径、云厂商路径、第三方 gateway 和国内 Claude-compatible 入口都可能让 Claude Code “跑起来”，但它们替代的层次并不相同：有的更接近第一方身份契约，有的更接近企业 provider 契约，有的只是工作流层的部分替身；对中国用户来说，真正成熟的选择标准不是“哪个最像官方宣传”，而是“它到底替代了哪层语义、又把哪层责任和退出成本转回给了我”。
