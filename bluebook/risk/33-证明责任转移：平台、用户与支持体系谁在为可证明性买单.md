# 证明责任转移：平台、用户与支持体系谁在为可证明性买单

## 1. 为什么要单独讨论“证明责任”

研究走到这里，已经能清楚看到一个事实：

- Claude Code 并不是一个单点封禁系统
- 它更像一个多层治理和可证明性系统

但随之而来的一个更难问题是：

当系统要求“你必须被持续证明为同一个主体、同一个组织、同一个高安全会话”时，  
到底是谁在为这个证明过程承担成本？

答案并不总是平台自己。  
很多时候，成本会在平台、用户和支持体系之间转移。

这一章讨论的就是这种转移。

## 2. 先说结论

从当前源码看，Claude Code 风控体系在很多地方已经努力降低误伤。  
但它仍然没有完全消除一个现实：

当系统内部知道的原因，比用户能直接看见的原因更多时，  
额外的证明责任就会向外转移。

这种外移主要体现在四个方面：

1. 平台承担判定成本。
2. 用户承担解释和变量收敛成本。
3. 支持体系承担语义翻译和证据拼接成本。
4. 高波动环境用户承担更高的连续性维护成本。

因此，真正要讨论的不是“平台有没有风控”，而是：

在一个需要持续证明主体连续性的系统里，证明责任是否被公平分配。

## 3. 源码里已经能看到哪些责任分配方式

### 3.1 平台已经承担了一部分解释责任

一些模块其实已经在主动替用户做解释：

- `getBridgeDisabledReason()` 会把 Remote Control 的缺口拆成订阅、full-scope、组织画像、gate、build
- `useCanSwitchToExistingSubscription()` 会直接提示你已有订阅但没激活到当前路径
- `errors.ts` 会把 token revoked、组织不允许、通用 auth、模型不可用、refusal 分成不同族

这说明平台并非完全把解释义务丢给用户。  
它已经开始把恢复路径前置成更具体的 reason family。

### 3.2 但关键边界上，用户仍需自己补完证明链

例如 `validateForceLoginOrg()`：

- 机器要求某个组织
- 本地缓存不被完全信任
- profile 拉不到就 fail-closed

这里的平台立场很明确：

- 组织边界必须被权威证明
- 证明不了，就宁可先挡住

从边界安全看，这是合理的。  
但从责任分配看，用户仍然需要自己补完这些事实：

- 当前 token 到底走哪条路径
- 当前 scope 是否完整
- 当前组织是否与机器要求一致

### 3.3 支持体系被迫承担语义翻译责任

只要体验层仍然存在语义压缩，支持体系就必须接手做二次翻译：

- 这不是处罚，是权益没激活
- 这不是全局账号失效，是 connector `needs-auth`
- 这不是永久失败，是远程高安全链路在重建

源码里 `diagLogs.ts`、`useMcpConnectivityStatus.tsx`、`sessionIngress.ts` 已经能看出这种倾向：

- 平台在为支持准备低敏感度证据
- 但还没有把所有 reason family 都直接显式暴露给用户

于是支持体系就成了“内部细语义”和“外部有限提示”之间的翻译层。

### 3.4 高波动环境用户被征收了更高的连续性维护税

这不是说平台有明确地域惩罚。  
更准确的说法是：

高波动环境用户更容易同时面对这些额外任务：

1. 维持稳定网络
2. 维持稳定设备
3. 维持稳定身份路径
4. 维持稳定组织边界
5. 维持更完整的证据包

这些任务本身就是证明责任的一部分。  
而它们在某些用户群体上更难完成。

## 4. 为什么这是平台正义问题，而不只是支持问题

如果一个系统需要大量用户自己解释：

- 我到底走的是哪条登录路径
- 我到底属于哪个组织
- 现在到底是 entitlement 缺失还是会话过期
- 我为什么在这个设备和网络上更容易失败

那平台实际上就在向用户转移内部复杂度。

这种转移未必是恶意的，  
也常常是工程现实的结果。  
但它依然会带来一个正义问题：

谁更有能力承担这部分复杂度，谁就更容易“看起来没事”。  
谁更难承担这部分复杂度，谁就更容易“看起来总像被封了”。

## 5. 从第一性原理看，为什么证明责任无法完全消失

如果平台要保护：

1. 主体真实性
2. 组织边界
3. 高安全会话可信性
4. 成本与额度边界

那它就不可能完全不要求证明。

所以真正的问题不是：

- 要不要证明

而是：

- 哪部分证明由系统自动完成
- 哪部分证明必须由用户配合
- 哪部分证明应被支持体系接住

这才是成熟平台的第一性原理问题。

## 6. 用苏格拉底式追问再压一层

### 问：只要平台给了错误提示，责任就已经尽到了吗？

答：不一定。

如果提示还不足以让用户选出最短恢复路径，  
那说明解释责任仍然没有真正完成。

### 问：是不是应该把所有内部 reason code 都直接暴露给用户？

答：也不一定。

原因有两个：

1. 某些内部细节不适合完全公开
2. 过度暴露也可能增加噪声和误解

真正成熟的做法不是“全部公开”，而是“公开足够支撑恢复路径的那部分”。

### 问：为什么中国和高波动环境用户更需要这层讨论？

答：因为他们更可能不是“犯错更多”，而是“证明更难”。

如果一个用户群体更难维持：

- 稳定网络
- 稳定设备
- 稳定身份路径
- 稳定支持与申诉环境

那他承担的就不是更高风险，而是更高证明成本。

### 问：平台如果真想更公平，最该减少什么？

答：最该减少的不是必要边界本身，而是“无法区分时把复杂度外包给用户”的部分。

这意味着平台要继续强化：

1. 结构化 reason families
2. 更短恢复路径
3. 会前体检
4. 可导出证据包
5. 面向高波动环境用户的低摩擦解释层

## 7. 对用户的合规启示

如果你想保护自己的利益，最重要的不是和系统对抗，  
而是减少自己被迫承担的无谓证明成本。

更具体地说：

1. 让身份路径尽量单一
2. 让组织边界尽量清晰
3. 在故障窗口里冻结变量
4. 把“最近一次成功时间”和错误原文都保留下来
5. 先判断问题属于哪一层，再决定支持路径

这些动作不是让你更容易“绕过”，  
而是让你更不容易在解释链里吃亏。

## 8. 对平台构建者的启示

这一章最值得迁移的经验有五条：

1. 不要只优化判定精度，也要优化证明责任分配。
2. reason family 的价值不只是 UX，而是减少用户承担的解释成本。
3. 证据包不是附属工具，而是平台正义基础设施。
4. 高波动环境用户最需要的未必是更松，而是更短、更清楚的证明链。
5. 支持体系不应只是兜底，而应被设计成控制平面的正式组成部分。

## 9. 一句话总结

在 Claude Code 这类多层治理系统里，真正关键的不是“有没有风控”，而是“可证明性的成本最后落到谁身上”；很多用户感受到的被限制，其实也是证明责任外移后的体感。
