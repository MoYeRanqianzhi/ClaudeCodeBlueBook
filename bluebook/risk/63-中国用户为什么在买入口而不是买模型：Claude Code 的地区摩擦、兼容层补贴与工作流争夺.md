# 中国用户为什么在买入口而不是买模型：Claude Code 的地区摩擦、兼容层补贴与工作流争夺

## 1. 为什么还要单独写这一章

前一章已经讨论了：

- 官方路径与中转站叙事该如何判断
- 传闻、观察与事实怎样分层
- 中国用户如何把中转站依赖收敛成退出权

但还缺一层更现实的问题：

`为什么中国用户围绕 Claude Code 的竞争，常常不是“谁的模型更强”，而是“谁能先卖给你一个可用入口”？`

这一章专门讨论：

1. 中国用户为什么会长期感觉 Claude Code 难用
2. 中国用户通常到底是怎么用上的
3. AnyRouter 这类中转站、Claude-compatible 兼容层和国内替代厂商到底在卖什么
4. 如果某家国内模型公司愿意为这种入口长期补贴，它最可能在买什么

## 2. 先说边界

这一章不做两件事：

- 不提供任何绕过地区限制、规避检测或伪装主体的方法
- 不把 AnyRouter 与智谱的关系写成已证事实

这一章只做三件事：

- 用截至 2026 年 4 月 2 日可公开核查的网页材料建立事实层
- 结合客户端源码与公开生态做高置信推断
- 把“补贴型兼容入口”的战略目的压成第一性原理结论

因此这一章里的判断会被分成三层：

1. 可证事实
2. 高置信推断
3. 不可证传闻之上的一般战略解释

## 3. 先说结论

对中国用户来说，Claude Code 的困难从来不只是网络问题。

更准确地说，它是五层摩擦叠加：

1. 官方可用性边界
2. 支付与订阅资格边界
3. 持续在线运行时与辅助依赖边界
4. 连续性与自证成本边界
5. 支持与责任归属边界

因此中国用户在现实里常常买到的不是“模型能力本身”，而是：

`更低摩擦的接入权、兼容权、试错权与未来迁移权。`

这也解释了为什么 AnyRouter 这类中转站会流行，为什么智谱这类国内厂商会公开提供 Claude-compatible 路径，以及为什么低价、免费额度和“一键兼容 Claude Code”在战略上远比一句“我们模型也很强”更有杀伤力。

## 4. 截至 2026 年 4 月 2 日，可以明确说的事实

### 4.1 Anthropic 官方确实设定了地区与资格边界

根据 Anthropic 官方 supported countries 页面与 Claude Code setup 文档，截至 2026 年 4 月 2 日，我核对到：

- supported countries 页面列出了日本、新加坡、韩国、台湾等，但没有列出中国大陆或中国香港
- Claude Code setup 文档明确要求互联网连接、处于支持地区，并且需要 Pro、Max、Teams、Enterprise 或 Console 账户
- 免费 Claude.ai 计划不包含 Claude Code 访问资格

这意味着很多中国用户从一开始面对的就不是“如何正常购买一个全球通用 SaaS”，而是：

`先解决能不能进入、能不能付费、能不能持续保持资格。`

### 4.2 Anthropic 官方也承认 gateway / proxy 是正式架构形态之一

Anthropic 官方 Claude Code LLM gateway 文档明确讨论了：

- 通过集中式 LLM gateway 统一接入 Anthropic、Bedrock、Vertex 等上游
- 在 gateway 层做模型路由、审计与统一凭证管理

这说明：

- gateway / proxy 不是纯社区黑科技
- 兼容入口是一种被官方文档承认的部署形态

但这不等于官方为任何具体第三方 relay 背书。

### 4.3 AnyRouter 的公开文案非常明确地在卖“低摩擦接入”

截至 2026 年 4 月 2 日，AnyRouter 公开文档和 FAQ 至少公开宣称了几件事：

- 自己是 Claude Code 共享平台
- 支持“国内直连”
- 注册赠送免费额度
- 无需信用卡或境外信用卡
- 通过 `ANTHROPIC_BASE_URL` 和兼容 token 接入 Claude Code
- FAQ 还明确写到：`offline` 往往只是 Claude Code 的 Google 连通性检查失败，而网页 `Fetch` 相关能力仍需要国际互联网连接

这些公开文案卖点几乎一一对应中国用户的现实摩擦：

1. 网络摩擦
2. 支付摩擦
3. 试错摩擦
4. 上手摩擦

### 4.4 AnyRouter 的公开域名信息仍然不足以证明幕后主体

WHOIS 公开页显示：

- `anyrouter.top` 注册于 2025 年 6 月
- 公开联系人国家/地区显示为 HK
- 使用阿里云 DNS

但这些信息并不能直接推出：

- 背后一定是智谱
- 一定存在股权或运营关系
- 免费额度一定由智谱承担

因此：

`AnyRouter 和智谱的关系，目前仍不能从公开资料直接坐实。`

### 4.5 智谱官方确实在公开争夺 Claude Code 这条工作流入口

截至 2026 年 4 月 2 日，我核对到智谱官方公开材料中至少有四类高度相关信息：

1. 官方 Claude Code 接入文档直接教用户在本地写入 `ANTHROPIC_BASE_URL`、`ANTHROPIC_AUTH_TOKEN`，并把 Claude Code 指向 `https://open.bigmodel.cn/api/anthropic` 这类兼容入口
2. 官方文档直接提供 Claude 模型名到 GLM 模型名的映射与配置 helper，说明目标不是抽象 API 调用，而是现成 Claude Code 工作流迁移
3. 智谱官网关于页公开把 2025 年旗舰模型描述为把推理、编码与智能体能力原生融合，并强调其开发者规模、OpenRouter 调用量与 API 收入
4. 智谱 `GLM-4.7` 页面明确写到可用于 Claude Code、Kilo Code、Roo Code、Cline 等编程智能体，并把 `GLM Coding Plan` 直接包装成以更低价格获得 Claude 级别编程体验的方案

这四点足以说明一个非常重要的事实：

`智谱不是在抽象地争夺“开发者 AI 市场”，而是在非常具体地争夺 Claude Code 用户的默认入口。`

## 5. 中国用户为什么会长期感觉 Claude Code 难用

### 5.1 这不是“模型好不好”的问题，而是“运行条件齐不齐”的问题

Claude Code 不是一个纯本地编辑器插件。  
它是一个持续依赖：

- 登录状态
- 模型调用
- 远程能力
- 在线配置
- 辅助网络可达性

的在线运行时。

所以中国用户面对的第一道门槛不是“模型能力是否先进”，  
而是：

`这套运行时前提条件在自己所处环境里是否同时成立。`

### 5.2 官方地区边界把“正常购买”变成了“先找入口”

一旦官方 supported countries 页面没有把你纳入自然支持范围，  
用户决策顺序就会反过来：

1. 先问哪里能接入
2. 再问哪里能付款
3. 再问哪里能稳定保持资格
4. 最后才问底层模型表现

从用户体验角度看，这会把原本应是产品比较的问题，  
提前扭曲成基础设施比较的问题。

### 5.3 资格和支付门槛进一步提高了入口价值

Claude Code 官方 setup 文档要求的不是任意免费账户，  
而是带付费或正式控制台资格的账户层级。

这意味着中国用户常见的痛点并不只是“网络慢”，  
而是：

- 资格本身不天然可得
- 支付路径不天然顺滑
- 一旦走第三方，还要承担额外责任边界

因此，谁能把：

- 资格获取
- 付款转换
- 免费试错
- 初始配置

这四件事打包得更轻，谁就更容易先占住用户。

### 5.4 真正难的是连续性与证明负担

本蓝皮书前文已经论证过：

- 中国用户更容易处在高波动环境
- 高波动环境更容易制造主体链、资格链、会话链噪声
- 噪声叠加后，很多问题的主观体感都会变成“像被封了”

因此中国用户最大的额外成本，常常不是单次连通性成本，  
而是：

`解释成本、支持分流成本和自证成本。`

### 5.5 责任边界越多，难度越高

官方直接路径里，  
至少还能大致分成：

- Anthropic 平台
- 组织管理员
- 本地环境

一旦加入中转站、兼容层和国内代理入口，  
责任边界会被改写成：

- 上游平台
- relay / gateway 运营方
- 自己本地环境
- 组织或团队管理员
- 可能的国内替代提供方

边界越碎，支持越慢，用户越容易觉得“全是黑盒”。

## 6. 中国用户通常到底是怎么用 Claude Code 的

结合官方文档、公开兼容文档与生态观察，可以把现实中的常见使用方式压成四类。

### 6.1 官方直接路径

- Anthropic 官方账户
- Anthropic API / Console
- Bedrock / Vertex / Foundry 等官方支持云路径

这是契约最清晰的一类，  
但对中国用户通常也是门槛最高的一类。

### 6.2 第三方 relay / 中转站路径

典型做法是：

- 保留 Claude Code 宿主与工作流
- 通过 `ANTHROPIC_BASE_URL` 和兼容 token 改写上游入口
- 由 AnyRouter 这类平台承接支付、额度与兼容接入

这一类解决的是：

`接入权和支付权。`

但它并不天然解决：

- 官方 OAuth 等价性
- 高阶能力等价性
- 责任边界清晰性
- 长期退出权

### 6.3 国内 Claude-compatible 兼容提供方路径

这是 2025 年后特别值得注意的一类。

以智谱官方公开材料为例，它并不只是说“我们的模型能写代码”，  
而是直接提供：

- Claude Code 兼容配置说明
- `ANTHROPIC_BASE_URL` / token 配置 helper
- Claude 到 GLM 的模型映射
- 面向 Claude Code 用户的套餐与定价叙事

这意味着竞争已经从“模型能不能替代”进一步前移成：

`工作流能不能无痛迁移。`

### 6.4 混合路径

还有一批成熟用户会采用混合策略：

1. 宿主工作流继续沿用 Claude Code
2. 高价值资产保持本地和可迁移
3. 入口层按成本、稳定性和资格动态切换

这类用户买的已经不是单一模型，  
而是一组可逆的入口组合。

## 7. 为什么这场竞争在技术上真的可行

### 7.1 客户端本身就允许改写上游入口

源码显示：

- `src/utils/apiPreconnect.ts`
- `src/services/api/filesApi.ts`
- `src/utils/model/providers.ts`

都明确读取或判断 `ANTHROPIC_BASE_URL`。

这说明第三方兼容入口之所以能成立，  
不是因为社区“黑进了客户端”，  
而是因为：

`客户端本来就把“上游入口可配置”当成正式现实。`

### 7.2 客户端还明确区分第一方与第三方 host

`src/utils/model/providers.ts` 里专门判断某个 `ANTHROPIC_BASE_URL` 是否仍是第一方 Anthropic host。  
`src/utils/auth.ts` 还明确写出，账户信息只对第一方 Anthropic API 提供。

这说明客户端设计者非常清楚：

- 自定义 base URL 是现实需求
- 但第三方入口和第一方入口不是同一语义层

### 7.3 第三方入口可以工作，但不等于官方能力全集

`src/utils/toolSearch.ts` 明确提醒：

- `tool_reference` 这类 beta 形态，很多 `ANTHROPIC_BASE_URL` 代理通常不支持
- 当 base URL 不是第一方 host 时，工具搜索会更保守地关闭默认 optimistic 路径

`src/bridge/bridgeEnabled.ts` 还进一步把：

- Bedrock / Vertex / Foundry
- gateway deployment
- env-var API key
- Console API login

与 claude.ai OAuth 订阅路径明确区分开来，说明 Remote Control 一类高阶能力天然依赖第一方 OAuth 身份面。

这意味着 relay / proxy / compatibility layer 的真实产品形态更接近：

`大体兼容，但带有语义差、能力差和支持差。`

### 7.4 本地配置与 worktree 传播会放大入口依赖

`src/utils/worktree.ts` 明确会把 `.claude/settings.local.json` 复制到 worktree。  
`src/utils/cleanup.ts` 还说明 `cleanupPeriodDays` 默认 30 天，影响 transcript 清理与 `/resume` 相关连续性。

这意味着入口层竞争不只发生在“今天能否连接”，  
也会沉淀到：

- 你的本地配置
- 你的 worktree 扩散面
- 你的会话与恢复资产

这也是为什么成熟用户必须重视退出权，而不是只重视免费入口。

## 8. 从产品学看，中转站和兼容层到底在卖什么

表面上看，它们卖的是模型接入。  
但更准确地说，它们在卖五种更底层的东西。

### 8.1 更低摩擦的接入权

不是“模型变强了”，  
而是：

- 更容易连上
- 更容易开始
- 更容易先试起来

### 8.2 更低摩擦的支付转换

对很多中国用户来说，真正难的不是愿不愿意付费，  
而是能不能顺滑完成支付、订阅和持续资格维持。

### 8.3 更低成本的试错权

免费额度、注册送金、低价月包的真正作用，  
往往不是直接盈利，  
而是降低首次尝试的心理和经济门槛。

### 8.4 默认配置槽位

谁先把自己的 `ANTHROPIC_BASE_URL`、token helper、模型映射写进用户环境，  
谁就更接近拥有：

- 默认调用权
- 默认对比权
- 默认迁移起点

### 8.5 未来迁移权

一旦用户已经把：

- IDE 习惯
- Prompt 习惯
- Claude Code 使用姿势
- 计费与充值关系

都建立在你的入口上，  
你未来要把底层模型逐步替换成别的提供方，就容易得多。

## 9. 如果有国内公司愿意长期补贴这种入口，它最可能在买什么

这一节是一般战略推演，  
不是对 AnyRouter 与任何具体公司关系的事实断言。

### 9.1 在买开发者入口，而不是只买调用量

入口比单次调用更值钱。  
因为入口决定：

- 谁先被试用
- 谁先被配置
- 谁先形成默认习惯

### 9.2 在买工作流教育，而不是只买品牌曝光

很多开发者并不知道前沿 agentic coding 到底意味着什么。  
最有效的教育方式不是宣讲，  
而是让他直接在熟悉的宿主里跑起来。

所以补贴顶级工作流入口，  
本质上是在做：

`先教育市场，再争取替代。`

### 9.3 在买“默认 base URL”位置

在 Claude Code 这种宿主里，  
谁进入了 `ANTHROPIC_BASE_URL`、模型映射和 token helper，  
谁就进入了用户的日常工作流骨架。

从战略上看，这比一次营销曝光更接近真正的控制点。

### 9.4 在买未来的迁移权

如果用户已经习惯：

- 在 Claude Code 里工作
- 通过兼容层调用
- 按兼容模型映射理解能力

那么未来把底层模型从外部顶级模型，  
逐步迁到本土模型，  
阻力会小得多。

因此补贴入口的深层意义更像是：

`先掌握迁移起点，再等待底座追平。`

### 9.5 在买时间

如果本土模型还在追赶，  
最稀缺的资源不是宣传，  
而是时间。

补贴一个足够好用的兼容入口，  
可以为自己换来：

- 用户不流失
- 工作流不旁落
- 真实工作流需求观察
- 底层能力追赶的缓冲期

## 10. 从第一性原理看，这背后的哲学内涵是什么

### 10.1 在可用性稀缺地区，入口先于底座

当官方可用性不是天然存在时，  
谁控制入口，谁就更接近控制“现实中的可用性”。

因此在中国用户场景里，  
真正稀缺的常常不是“最强模型”，  
而是：

`足够可用、足够稳定、足够好付费、足够容易上手的入口。`

### 10.2 开发者真正忠于的是工作流，而不是口号

开发者一般不会长期忠于抽象品牌，  
他们更可能忠于：

- 更低摩擦
- 更强产出
- 更可预测的日常习惯

谁承包了工作流，  
谁就更有机会在后面替换底层能力。

### 10.3 补贴不是慈善，而是在购买未来的可替代性

如果一个入口持续以低价甚至免费方式提供前沿工作流体验，  
它通常不是在做无条件公益，  
而是在购买：

- 用户未来迁移到自己底层能力的概率
- 用户对当前入口的默认信任
- 用户对成本与便利的依赖关系

### 10.4 幕后传闻的流行，本身也是入口不透明的副产品

为什么社区总会追问：

- 幕后是不是某家大模型公司
- 谁在出钱
- 为什么肯补贴

因为当入口比主体信息更显眼时，  
用户天然会想知道：

`谁才是真正拥有这条入口的人。`

这不是阴谋论冲动，  
而是对责任、信任和可持续性的自然追问。

## 11. 对用户最现实的启示

无论 AnyRouter 背后最终是谁，  
对中国用户都应坚持同一套判断框架：

1. 低价和免费不等于低总风险
2. relay / compatibility layer 卖的是入口，不是官方等价身份
3. 先问谁负责，再问多便宜
4. 先问怎样退出，再问怎样充值
5. 先保护本地资产与证据，再沉迷入口便利

用户真正该保护的不是某个入口本身，  
而是：

- 本地代码
- 会话与 transcript
- 配置与工作流文档
- 充值与支付证据
- 可替代工作流

## 12. 苏格拉底式追问

### 问：为什么中国用户常常像是在买“入口”而不是买“模型”？

答：因为在官方可用性、支付和连续性都不天然成立时，  
真正先被满足的需求是“如何进入”，而不是“底座究竟是谁”。

### 问：为什么兼容层补贴的战略价值这么高？

答：因为它可以先占住工作流，再等待底层模型追平。

### 问：为什么 AnyRouter 与智谱的传闻不能直接写成结论？

答：因为公开网页目前还不足以证明控制关系、出资关系或运营关系。

### 问：既然不能坐实，为什么还要讨论背后目的？

答：因为即使不绑定某一家公司，补贴兼容入口本身仍有清晰的战略结构。

### 问：这一章最重要的哲学结论是什么？

答：在地区可用性不对称的市场里，  
真正被争夺的往往不是一次模型调用，  
而是：

- 开发者的默认入口
- 开发者的默认工作流
- 开发者未来的迁移方向

## 13. 一句话总结

中国用户使用 Claude Code 的艰难，本质上是地区、支付、连续性和支持摩擦的叠加；AnyRouter 这类中转站与智谱这类 Claude-compatible 兼容提供方之所以重要，不是因为它们重新发明了模型，而是因为它们在争夺更稀缺的东西：开发者的入口、工作流和未来迁移权；至于具体幕后关系，公开证据不足以把传闻写成事实，但这并不妨碍我们看清入口补贴背后的真实战略结构，以及路由层、工作流层和未来迁移层的控制权究竟落在谁手里。
