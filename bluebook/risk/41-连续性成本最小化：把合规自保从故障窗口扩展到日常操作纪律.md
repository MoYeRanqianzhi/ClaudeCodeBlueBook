# 连续性成本最小化：把合规自保从故障窗口扩展到日常操作纪律

## 1. 为什么还要单独写这一章

`28` 已经讨论了故障窗口纪律。  
但真实世界里的误伤成本，往往不是在一分钟内决定的，  
而是在几天、几周的使用方式里慢慢累积出来的。

Claude Code 当前源码里到处都在暴露一个事实：

- 认证有缓存
- 刷新有锁
- 会话有 TTL
- 子系统有自己的 `needs-auth`
- 组织策略与 remote managed settings 有自己的轮询周期

这意味着用户面对的不是一次性判断，  
而是一套会随时间不断重新证明、重新失配、重新收敛的控制面。

如果只在出问题时才临时补救，  
往往已经太晚。  
更有效的合规自保，是把“减少连续性成本”变成日常操作纪律。

## 2. 先说结论

从源码设计看，最安全、最有效、也最不容易把自己推向误伤窗口的做法，不是“遇到异常再猛试”，而是长期降低三类成本：

1. 主体重证明成本
2. 资格重确认成本
3. 会话重建成本

把这三类成本压低，  
就能显著降低“像被封了”的体感频率。

对中国和其他高波动环境用户来说，  
真正要管理的不是“如何躲过风控”，  
而是“如何让系统更省成本地持续确认你还是同一个、同一组织、同一路径、同一会话族里的合法主体”。

## 3. 源码为什么支持这个结论

### 3.1 系统明确在对抗“多实例、多缓存、多时钟”造成的连续性破坏

`utils/auth.ts` 里有一段很关键的处理：

- 发现 `.credentials.json` 的修改时间变化后，会主动清 OAuth 缓存
- 401 恢复路径会强制从安全存储重读，而不是盲信内存
- token 刷新还做了跨进程锁和重试

这说明作者已经明确在对抗一种现实问题：

- 不是你没登录
- 而是不同终端、不同进程、不同缓存对“你当前是谁”产生了分叉

从用户角度看，这意味着长期混用多条身份路径、多处并发登录、多个环境反复抢刷新权，会明显提高连续性成本。

### 3.2 系统大量使用“提前刷新”而不是“出错后再刷新”

`services/mcp/auth.ts` 在 token 距离过期 5 分钟时就尝试预刷新。  
`bridge/envLessBridgeConfig.ts` 也把 `token_refresh_buffer_ms` 默认设为 300_000。  
这不是偶然重复，而是一种时间哲学：

- 真正昂贵的不是刷新本身
- 而是带着快过期凭证继续进入高价值路径，再在半路失败

对用户的启示也很直接：

- 高价值工作应尽量放在身份和会话都新鲜的窗口里开始
- 不要在已出现异常的身份状态上继续叠加长链路任务

### 3.3 系统把局部认证失败短期记忆下来，避免无意义抖动

`services/mcp/client.ts` 会把 `needs-auth` 缓存 15 分钟。  
`toolExecution.ts` 也会把特定 MCP 连接从 `connected` 改写为 `needs-auth`。

这说明 Claude Code 并不把所有连接域都看成“实时重新判定”的对象。  
相反，它允许子系统在一段时间内保持“需要重新认证”的局部真相。

这对长期纪律的启示是：

1. 不要把子系统局部失效误判成全局封禁
2. 不要因为某个连接域失效，就同时重置所有别的身份面
3. 先在局部域内恢复，再决定是否需要更大范围动作

### 3.4 系统把高安全能力和日常能力区分成不同连续性等级

`trustedDevice.ts` 显示：

- trusted device enrollment 必须发生在 fresh login 后 10 分钟内
- token 是持久化的，但有 90 天滚动有效期
- 登录时还会主动清旧 token，避免跨账号串用

这说明高安全能力并不是“普通登录成功的自然延伸”，  
而是额外要求会话新鲜度、设备连续性和账户一致性的另一层证明。

因此，长期纪律里必须明确区分：

- 普通 CLI 访问
- 高安全远程访问

两者不能共用同一套乐观预期。

### 3.5 组织治理是持续轮询、渐进收敛的，不是一次性静态配置

`remoteManagedSettings/index.ts` 与 `policyLimits/index.ts` 都表现出类似结构：

- 基于缓存和校验和工作
- 初始加载可等待
- 后台按小时轮询
- 失败时更倾向于 graceful degradation，而不是直接把所有功能打死

但 `securityCheck.tsx` 又说明，  
一旦下发的是危险设置变更，系统会要求交互确认，拒绝时直接停机。

这表明平台的真实治理逻辑是：

- 普通远程治理尽量降低运行摩擦
- 危险治理变更则必须重新获取明确同意

用户如果频繁切组织、切机器、切管理环境，  
就更容易把这些渐进收敛过程打断成噪声。

## 4. 长期连续性纪律，不是临时故障纪律

### 4.1 把“身份路径”收敛成主路径和备用路径

最稳妥的做法，不是保留很多可切换入口，  
而是明确区分：

1. 一个主身份路径
2. 一个明确知道何时才使用的备用路径

如果主路径和备用路径在日常工作里频繁混用，  
缓存、组织画像、token 刷新和高安全链路都更容易出现不必要分叉。

### 4.2 把“组织边界”收敛成工作边界

长期最容易制造解释成本的行为之一，是把个人用途、组织用途、测试用途混在同一环境里。

更低成本的做法是：

1. 组织环境只做组织工作
2. 个人环境只做个人工作
3. 测试环境单独隔离

这样做不是为了“更像正常用户”，  
而是为了让资格链和主体链少打架。

### 4.3 把“设备边界”收敛成主力设备

源码已经说明 trusted device、bridge heartbeat、session ingress 都是有时间和设备连续性假设的。  
因此，对高价值操作来说：

1. 固定主力设备优先于多设备轮换
2. 固定主力宿主优先于频繁切容器、切远端、切中转
3. 先确认高安全能力健康，再开始长时任务

### 4.4 把“网络边界”收敛成低波动窗口

不能根据客户端源码断言平台按地域针对某类用户。  
但可以明确说，高波动网络环境会更频繁地制造：

- token 过期与刷新竞态
- profile 拉取失败
- 高安全会话掉心跳
- 连接器进入局部 `needs-auth`

因此，中国用户及其他高波动环境用户，更应把“低波动窗口使用高价值能力”当成纪律，而不是侥幸。

## 5. 把自保从分钟级动作升级为周级例行检查

### 5.1 每周做一次低成本状态校验

周级校验的目标不是折腾配置，  
而是提前发现连续性债务。

建议关注：

1. 当前主身份路径是否仍然明确
2. 当前组织归属是否仍然与工作环境一致
3. 最近一次成功使用高价值能力是什么时候
4. 最近是否发生过设备或网络大切换

如果平台可用 `auth status` 之类诊断面，就把它当成证据入口，  
不要把它当成“问题来了以后才第一次看的命令”。

### 5.2 在重要任务前做一次会前体检

`30` 讨论的是预检系统应该怎样设计。  
从用户侧看，现实做法可以更朴素：

1. 登录状态确认
2. 当前组织确认
3. 当前设备确认
4. 当前网络稳定性确认
5. 当前是否涉及 MCP / bridge / Remote Control 确认

这套动作的价值，在于避免把脆弱状态带进长链路任务。

### 5.3 为自己维护一份最小连续性台账

不需要复杂系统。  
只要长期记录下面几类事实，支持效率就会高很多：

1. 最近一次成功使用时间
2. 当前主身份路径
3. 当前主设备
4. 当前主网络
5. 当前组织归属
6. 当前问题是否涉及局部连接域

这份台账的意义不是自证清白，  
而是把“我感觉不对劲”变成“我能指出哪条证明链最近发生了变化”。

## 6. 面向中国和高波动环境用户的合规结论

这一节只讨论合规自保，不讨论规避或对抗平台治理。

如果你处在高波动环境里，最值得坚持的长期纪律是：

1. 固定主力设备、主力网络、主力身份路径
2. 将个人、组织、测试环境分开
3. 在高价值能力使用前做一次轻量体检
4. 遇到局部失败时先局部恢复，不做全局重置
5. 持续保留最近成功时间、错误原文和环境变化记录

这些做法不会保证零误伤，  
但会显著降低连续性破坏和解释成本。  
从保护自身利益角度看，这比任何“反复试到成功”为高。

## 7. 给平台构建者的启示

这章最值得迁移的，不是某个具体 gate，  
而是“把用户连续性成本当成一等指标”。

一个更成熟的 agent 平台，应该主动帮助用户减少：

1. 身份分叉成本
2. 资格解释成本
3. 会话重建成本
4. 支持举证成本

当这些成本被持续压低时，  
误伤率未必立刻归零，  
但误伤总损害会明显下降。

## 8. 一句话总结

真正决定“像不像被封了”的，往往不是某次单独拒绝，而是几天几周里不断累积的连续性成本；合规自保的核心也不是躲避检测，而是长期降低主体、资格与会话三条证明链的重证明成本。
