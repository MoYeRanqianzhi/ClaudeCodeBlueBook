# Selective Fail-Open / Fail-Closed 的设计哲学

## 1. 为什么这章很关键

很多人以为安全系统只能二选一：

- 要么足够严格，凡异常都拒绝
- 要么足够好用，凡异常都放过

Claude Code 的源码说明，这种二分法太粗。  
它采用的是更成熟的做法：

不同层、不同风险、不同对象，采用不同的 fail-open / fail-closed 策略。

## 2. 先说结论

Claude Code 风控设计里一个非常核心、但很容易被忽略的哲学是：

不要追求“全局 fail-open”或“全局 fail-closed”，而是按风险层级做 selective failure semantics。

从当前源码可以抽象出四条原则：

1. 对会误伤大量正常用户的资格门槛，优先减少陈旧 `false`。
2. 对危险变更、组织越界、高权限远程能力，优先 fail-closed。
3. 对本地动作执行，优先把风险收口在动作级，而不是一上来就升级到账号级。
4. 对失败风暴本身，也要治理，不能让错误扩散取代了治理本身。

## 3. 哪些地方明显偏向 fail-open

### 3.1 远程设置和组织限制的缓存退化

`remoteManagedSettings` 和 `policyLimits` 都明显采用了：

- 有缓存则尽量继续
- 后台轮询继续拉新
- 出错时优先 graceful degradation

这背后的思路是：

如果一个组织限制接口暂时失败，系统不应无差别地把所有正常用户全部打成不可用。

### 3.2 某些 entitlement gate 对“陈旧 false”特别敏感

源码明确写出：  
对于按订阅/组织分桶的用户触发能力，如果陈旧缓存里的 `false` 会不公平地拦住用户，就要走 blocking 检查。

这其实是一种“减少误伤”的 fail-open 变体：

- 不轻信陈旧拒绝
- 让服务端新鲜裁决纠正本地旧判断

### 3.3 classifier unavailable 并不总是立刻 deny

Auto mode classifier 不可用时，行为还受专门 gate 控制：

- 有时 fail-closed
- 有时退回正常 permission handling

这说明作者没有把“模型不可用”简单硬编码成单一结局。

## 4. 哪些地方明显偏向 fail-closed

### 4.1 组织绑定

`forceLoginOrg` 是典型硬边界：

- 组织不匹配直接拒绝
- profile 拉不到也拒绝
- 本地缓存不可信

这属于“宁可拒绝，也不能跨组织放行”。

### 4.2 危险 managed settings 变更

当远程下发的 settings 包含危险 shell settings、危险 env vars 或 hooks 变化时：

- 需要显式确认
- 拒绝就退出

这说明“危险变更”不属于可以悄悄平滑降级的范畴。

### 4.3 高安全远程会话

trusted device、bridge、高安全远程环境、组织 policy 的组合，显然更靠近 fail-closed：

- 不满足条件就不给进
- 401/403/404/410 有明确 fatal 语义

### 4.4 某些解析和安全检查路径

源码里还有很多注释直接写明：

- unknown flag -> fail closed
- too complex -> fail closed
- official registry 缺失 -> fail closed

这说明作者很明确地区分了哪些错误一旦放过就可能扩大风险。

## 5. 为什么这种“有选择的失败语义”更高级

因为真实系统要同时优化三件事：

1. 安全性
2. 可用性
3. 可解释性

如果全局 fail-closed：

- 正常用户误伤会很严重
- 网络波动和局部故障会被放大

如果全局 fail-open：

- 组织边界和高风险动作会失守
- 远程高权限链路会失去可信度

所以更成熟的做法只能是：

按层级、按对象、按后果做选择性失败语义。

## 6. 这种哲学背后的第一性原理

可以把它还原成一句话：

不是所有错误都应该用同一种方式对待，因为不是所有错误的代价都一样。

例如：

- 一个陈旧的 entitlement `false`，代价主要是误伤用户。
- 一个跨组织放行错误，代价可能是组织边界破裂。
- 一个危险 managed settings 变更被静默接受，代价可能是本地执行边界被重写。
- 一个失效 token 被无限重试，代价是失败风暴扩散。

一旦这样看，selective fail-open / fail-closed 就不再是技术细节，而是治理哲学。

## 7. 给系统设计者的启示

### 7.1 先问“错放”和“错杀”哪个代价更大

### 7.2 不同层要有不同失败语义

### 7.3 高风险变更要有人类可感知确认

### 7.4 失败风暴要被视为独立风险对象

### 7.5 错误语义要足够清晰，便于支持和申诉

## 8. 一句话总结

Claude Code 这套风控设计最成熟的地方之一，不是“更严”，而是知道哪里该严、哪里该缓、哪里该恢复、哪里必须立刻收口。
