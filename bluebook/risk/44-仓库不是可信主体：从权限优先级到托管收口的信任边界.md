# 仓库不是可信主体：从权限优先级到托管收口的信任边界

## 1. 为什么还要写这一章

前面已经讨论了很多 gate、缓存、证明链和恢复语义。  
但还有一条更底层、也更容易被忽视的线索：

`Claude Code 并不把你当前所在的项目仓库，当成天然可信主体。`

这条线索一旦看清，  
很多设计就会突然变得统一：

- 为什么有些高风险 opt-in 明确排除 `projectSettings`
- 为什么 deny / ask 会先于 bypass / allow
- 为什么托管策略可以直接收缩规则面
- 为什么某些 allow 规则会被显式判定为“根本到不了”

这章专门讨论这条信任边界。

## 2. 先说结论

从权限引擎、设置合并和托管策略源码来看，Claude Code 的一个核心安全前提是：

`项目内容、项目配置、共享命令和共享规则，都不能自动代表用户本人或管理员本人。`

因此系统采用了三种非常明确的策略：

1. 共享来源不默认拥有高风险放权能力
2. 拒绝和强制询问优先于放行和 bypass
3. 托管策略在必要时可以直接收窄规则面，而不只是“在上面再叠一层规则”

如果把这三点放在一起，  
就能看清 Claude Code 的真实哲学：

`治理对象不能同时成为治理者。`

## 3. 源码里有哪些直接证据

### 3.1 拒绝和询问先于放行

`permissions.ts` 把权限求值顺序写得非常清楚：

1. 先看 whole-tool deny
2. 再看 whole-tool ask
3. 再看 tool-specific deny / ask / safety checks
4. 最后才轮到 bypass mode 和 always-allow

而且有些 ask 还是 bypass-immune 的，  
例如高风险路径安全检查不会因为 bypass mode 就消失。

这不是普通的“规则优先级”而已，  
而是在表达一条更强的安全原则：

`放行可以晚一点，阻断必须先确定。`

### 3.2 宽规则把窄 allow 遮蔽掉，会被当成显式问题

`shadowedRuleDetection.ts` 并没有把“宽 deny/ask 把窄 allow 盖掉”视作正常副作用，  
而是把它建模成 unreachable rule：

- 会给出原因
- 会指出是被 ask 还是 deny 遮蔽
- 会给出修复建议

这说明作者不把“规则写了但其实不会生效”当成可接受的模糊状态。  
相反，规则面本身必须可解释。

从风控视角看，这等于在说：

`安全治理不仅要做对，还要能说明为什么某条表面上的放权其实永远到不了。`

### 3.3 `projectSettings` 被明确排除在多个高风险 opt-in 之外

`settings.ts` 有几处注释非常直白：

- dangerous mode opt-in 不信任 `projectSettings`
- auto mode opt-in 不信任 `projectSettings`
- plan mode 是否沿用 auto mode 语义，也不让 `projectSettings` 决定
- auto mode config 本身也排除了 `projectSettings`

原因也写得很清楚：

- malicious project
- RCE risk

这意味着源码作者明确承认：

1. 仓库可能是敌对输入
2. 共享项目配置不能自动代表用户真实同意
3. 只要某项配置足以改变风险边界，它就不能由项目自己替用户回答

### 3.4 托管策略可以直接“收口规则面”

`permissionsLoader.ts` 的 `allowManagedPermissionRulesOnly` 很关键：

- 会隐藏 always-allow 相关选项
- 读取权限规则时只保留 policy source
- 持久化新规则时也会拒绝普通来源继续扩张规则面

这说明企业托管并不是简单的“高优先级规则叠加”。  
它在必要时会直接改写可编辑面的形状。

换句话说：

`托管策略可以决定“哪些规则有资格存在”，而不只是“已有规则里哪条赢”。`

### 3.5 组织策略在某些面上是“第一来源胜出”，不是全量合并

`settings.ts` 对 `policySettings` 采用 first source wins：

- remote managed settings
- MDM
- managed file
- HKCU

一旦更高优先级来源已有内容，  
更低层的 policy source 就不再继续参与这一面的裁决。

这再次说明：

`治理链条里，来源本身就是事实；不是所有配置都应该被民主合并。`

### 3.6 MCP 的 allow 和 deny 也被故意做成非对称

`mcp/config.ts` 显示：

- `allowManagedMcpServersOnly` 可以把 allowlist 收敛到 managed settings
- 但 denylist 仍然会从所有来源合并

这是一种很有意思的非对称设计：

- 管理员可以垄断“允许什么”
- 但用户仍保留“我自己不要连这个”的自我保护权

这说明平台并没有把组织治理理解成对用户的绝对剥夺，  
而是保留了用户侧的最小防御权。

## 4. 从第一性原理看，这章真正讲的是什么

这一章真正讨论的，不是某个字段或某条规则。  
而是下面三条更本质的原则。

### 4.1 被治理对象不能自动成为治理者

如果仓库本身可以替用户打开 dangerous mode，  
那就等于：

- 被执行对象
- 同时决定了自己的执行边界

这在安全上几乎总是错误的。

### 4.2 来源可信度高于内容便利性

同一句配置，  
来自用户本地、管理员托管、项目仓库、运行时 flag，  
它的含义不应该一样。

Claude Code 的实现恰恰承认了这件事。

### 4.3 安全系统必须允许“禁止放权”，而不只允许“增加放权”

很多系统的问题在于：

- allow 很多
- deny 很弱
- ask 很容易被跳过

Claude Code 恰恰相反。  
它让 deny、ask、safety check 在顺序上先站住。

这就是为什么它看起来会更保守，  
但也更像真正可用的治理系统。

## 5. 对用户、管理员和平台构建者分别意味着什么

### 5.1 对用户

不要把“项目里已经写了配置”理解成“平台就应该信这个项目”。  
如果某项能力足够高风险，  
平台不信任仓库替你做决定，本身是合理的。

### 5.2 对管理员

如果你启用了 managed-only 的权限或 MCP allowlist，  
你不是在“多加一层规则”，  
而是在改变整个规则面的形状。  
因此发布前必须准备：

1. 更清楚的策略说明
2. 更明确的支持路径
3. 对误伤的回滚与解释预案

### 5.3 对平台构建者

最值得迁移的不是 Claude Code 的具体字段名，  
而是这三条边界设计：

1. 共享来源不自动拥有高风险放权权
2. deny / ask / safety 优先于 allow / bypass
3. 托管策略必要时可以收缩规则面

## 6. 用苏格拉底式追问再压一层

### 问：为什么不让项目仓库自己声明“这里是安全的”？

答：因为一旦仓库可以替用户声明高风险安全边界，  
攻击面就被反转了。  
被执行对象开始替执行者做决定。

### 问：为什么不把 allow 和 deny 做成完全对称？

答：因为安全后果不对称。  
错误放权和错误阻断都不好，  
但它们的损害结构不同。

### 问：为什么托管策略要强到能改写规则面？

答：因为有些治理目标不是“比较规则谁赢”，  
而是“某些来源根本不应该再有资格发言”。

### 问：这会不会太保守？

答：会更保守。  
但对于共享项目、组织环境和高能力 agent 来说，  
这种保守比“仓库自己给自己开权限”更接近可持续治理。

## 7. 一句话总结

Claude Code 在权限和风控上的一个根本判断是：项目仓库不是可信主体，因此高风险放权不能由共享来源自动决定，而必须服从“来源可信度、拒绝优先级和托管收口权”这三条更高边界。
