# 错误语义、能力撤回与治理层矩阵

## 1. 为什么要单独做这一章

用户最容易把所有失败都理解成“被封了”。  
但从 Claude Code 源码看，同样是“不能用”，其治理含义可能完全不同。

这章的目标只有一个：

把常见错误语义、能力撤回表现和它们真正对应的治理层分开。

## 2. 核心矩阵

### 2.1 `401`

更常见的真实含义：

- token 过期
- token 刷新失败
- 会话连续性断裂
- 远程链路凭证失效

更接近的治理层：

- 身份层
- 会话层

不应直接等同于：

- 账号被封禁

### 2.2 `403`

更常见的真实含义：

- access denied
- insufficient scope
- 组织权限不足
- 高安全链路被拒绝

更接近的治理层：

- 资格层
- 组织层
- step-up auth 层

需要特别区分：

- 普通 `403`
- `403 insufficient_scope`
- bridge 中可抑制的“装饰性 403”

### 2.3 `not enabled for your account`

更常见的真实含义：

- rollout 未命中
- entitlement 不具备
- 某个能力桶未对当前账号开放

更接近的治理层：

- 资格层
- 产品分桶层

这通常不等于惩罚。

### 2.4 `disabled by your organization's policy`

更常见的真实含义：

- 组织明确关闭了某项能力
- 这条能力链路被组织治理层收口

更接近的治理层：

- 组织层

这通常也不等于账号失效。

### 2.5 `needs-auth`

更常见的真实含义：

- 某个远程连接器或 MCP 连接需要重新认证
- 当前连接状态被单独降级

更接近的治理层：

- 连接层
- 子系统认证层

它通常不是全局账号问题，而是某个连接域需要恢复。

### 2.6 risky action blocked / permission denied

更常见的真实含义：

- 动作被本地执行安全阻断
- classifier / safety check / sandbox 不放行

更接近的治理层：

- 本地执行层

这不是封号，而是动作级风控。

### 2.7 rate limit / extra usage exhausted

更常见的真实含义：

- 配额耗尽
- 套餐额度触顶
- overage 不可再继续

更接近的治理层：

- 计费层
- 消耗层

这也不应与风控处罚混写。

## 3. 能力撤回的几种典型形式

从源码看，能力撤回至少有五种典型形式：

1. 身份撤回：token 不再有效。
2. 资格撤回：某账号不再拥有某个 feature bucket。
3. 组织撤回：policy limits 把能力设为不允许。
4. 会话撤回：远程环境、高安全会话或 trusted device 失效。
5. 动作撤回：本地执行面阻断了某种行为。

其中只有第一种和极少数平台级事件，才比较接近用户心中的“封号”。

## 4. 为什么这个矩阵对用户很重要

因为如果把错误类型理解错了，处理顺序也会完全错误。

例如：

- 把 `not enabled for your account` 当成 token 问题，就会无意义地反复重登。
- 把 `disabled by your organization's policy` 当成本地设置问题，就会和组织治理对着干。
- 把本地 risky action block 当成账号处罚，就会误以为自己必须换号。

## 5. 为什么这个矩阵对研究也很重要

它帮助我们把研究对象拆成三类：

1. 处罚或硬终止
2. 能力分桶与能力撤回
3. 本地和远程执行边界

如果不这么拆，研究会不断把 rollout、权限、配额和真正的风控混在一起。

## 6. 本章的主要证据

主要证据来自以下模块群：

- `utils/auth`
- `bridge/*`
- `bridgeEnabled`
- `services/policyLimits`
- `services/mcp/client`
- `services/mcp/auth`
- `services/rateLimitMessages`

## 7. 一句话总结

Claude Code 的“不能用”不是单一语义。  
只有先分清错误语义，再谈风控、封禁、误伤和自保，结论才会稳定。
