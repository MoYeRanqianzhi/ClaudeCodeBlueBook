# 风控最小公理与反公理：从第一性原理重写控制面哲学

## 1. 为什么还要写“公理与反公理”

到 `41` 为止，我们已经有很多结论：

- 三条证明链
- 治理时钟
- 证明责任转移
- 共享词汇
- 连续性成本

但如果这些结论不能再压缩成更少的底层原则，  
读者仍然容易把它们看成很多分散专题。

这一章的目标，就是把 Claude Code 风控研究继续压到最小：

1. 哪些是最小公理
2. 哪些是最危险的反公理
3. 为什么这些公理比“封号”“误伤”“检测”这些表层词更本质

## 2. 先说结论

从源码整体看，Claude Code 的风控本质不是“识别坏用户并封掉”，  
而是：

`在不确定、分布式、多时钟、多角色协作环境里，用最低总伤害维持高能力行为的持续可证明性。`

如果把这句话再压短一点，  
就是：

`风控 = 持续证明 + 分层放权 + 受控恢复。`

围绕这个中心，我认为最小公理至少有七条。

## 3. 公理一：权限越高，证明越重

这是最基础的一条。

`bridgeEnabled.ts` 已经说明：

- 远程高能力并不只要求“你登录了”
- 还要求 claude.ai 订阅
- 还要求 profile scope
- 还要求组织画像可判定
- 还要求 gate 命中

`trustedDevice.ts` 进一步说明：

- 高安全能力还要求 fresh login
- 还要求设备 enrollment

因此第一公理是：

`权限不是统一放行的；能力越高，所需证明链越长、越新鲜、越难伪造。`

## 4. 公理二：对用户伤害大的否定，必须比肯定更谨慎

`checkGate_CACHED_OR_BLOCKING()` 的注释几乎已经把这条公理写明白了：

- stale `true` 可以暂时接受，因为真正的硬 gate 在服务端
- stale `false` 容易不公平地挡住用户，所以要阻塞拉一次新值

这说明成熟控制面并不追求对称。  
它追求的是：

- 低成本地接受可逆的乐观放行
- 高成本地避免不可解释的错误拒绝

而且这种非对称并不只存在于远程 entitlement。  
从 `permissions.ts` 与 `shadowedRuleDetection.ts` 看，

- whole-tool deny / ask
- tool-specific deny / ask
- safety check

都会先于 bypass mode 和 always-allow 规则被评估。  
甚至源码还会把“宽 deny/ask 把窄 allow 遮蔽掉”当成显式问题。

所以第二公理是：

`正向放行和负向拒绝不应采用同样的证据标准；用户伤害更大的否定，应当要求更强的新鲜证据。`

## 5. 公理三：本地真相不是权威真相

`validateForceLoginOrg()` 明确拒绝相信本地可写配置里的组织 UUID，  
而是强制去 profile endpoint 拉取权威组织信息。  
`auth.ts` 对 401 的处理也会清理缓存并回到安全存储重读。

这说明系统默认承认：

1. 本地缓存会陈旧
2. 本地配置可能被污染
3. 同一主体在不同进程里的真相可能分叉

因此第三公理是：

`本地状态只能作为提示或加速层，不能单独承担最终裁决。`

## 6. 公理四：治理必须分层，不能把所有失败压成“封”

源码里存在非常清楚的分层：

1. 主体认证失败
2. 资格未命中
3. 组织 policy 限制
4. 子系统 `needs-auth`
5. 高安全会话 `auth_failed` / `fatal`
6. 计费和速率限制

`errors.ts` 的 `token_revoked`、`oauth_org_not_allowed`、`auth_error`，  
`mcp/client.ts` 的 `needs-auth`，  
`rateLimitMessages.ts` 的额度语义，  
共同说明平台内部并不是在用单一“被封”语义工作。

所以第四公理是：

`成熟风控必须允许局部失效、局部撤权、局部恢复；不能把所有失败都压成单一处罚。`

## 7. 公理五：时间本身就是治理变量

这个系统里最显眼但最容易被忽略的，不是某一个 if/else，  
而是大量时间边界：

- 5 分钟预刷新
- 10 分钟 fresh login enrollment 窗口
- 15 分钟 `needs-auth` 缓存
- 20 秒 heartbeat 对 60 秒 TTL
- 小时级轮询 remote settings / policy limits

这说明风控不是静态判卷，  
而是一个随时间不断重新验证、重新降权、重新恢复的动态控制面。

所以第五公理是：

`风控判断不是时刻无关的；时间新鲜度本身就是证据的一部分。`

## 8. 公理六：低风险治理可以降级，高风险治理必须强确认

`remoteManagedSettings/index.ts` 明确表现出 fail-open 倾向：

- 拉取失败时优先使用 stale cache
- 没有 cache 时继续运行
- 背景轮询失败时不把系统打死

但治理不仅关心“值是什么”，还关心“值来自哪里”。  
`settings.ts` 对 dangerous mode、auto mode、plan mode 某些自动行为明确排除了 `projectSettings`，  
因为仓库本身被视作可能不可信。  
`permissionsLoader.ts` 的 `allowManagedPermissionRulesOnly` 甚至会直接收缩规则面，隐藏或清除非策略来源的权限规则。

但 `securityCheck.tsx` 同时又说明：

- 危险设置变更需要阻塞确认
- 用户拒绝则直接停止继续执行

因此第六公理是：

`治理不是一律 fail-open 或一律 fail-closed；它应按风险等级切换降级策略。`

## 9. 公理七：可观测性必须可用，但不能无限采集

`diagLogs.ts` 非常关键。  
它强调诊断日志必须避免 PII，包括：

- 文件路径
- 项目名
- 仓库名
- prompt

这说明平台并不是用“看得越多越好”的逻辑构建诊断面。  
它选择的是另一种平衡：

- 记录足够的事件和时长
- 但避免把支持调查建立在高敏感原始内容之上

所以第七公理是：

`可解释性不是无限采集；成熟风控需要低敏感度、可复用、可分流的证据面。`

## 10. 反公理：哪些直觉最危险

### 10.1 反公理一：风控就是一个封号开关

如果接受这条反公理，  
你会把 entitlement、组织策略、MCP 失效、计费限制、高安全会话失效全部误写成同一个问题。

这会同时伤害：

- 用户判断
- 支持分流
- 平台改进

### 10.2 反公理二：只要重新登录，一切都会恢复

主体链恢复，  
不等于资格链恢复，  
也不等于会话链恢复。

这条反公理会诱导用户做出大量错误的全局重置动作。

### 10.3 反公理三：重试越多，公平性越高

`withRetry.ts`、`mcp/client.ts`、`bridgeMain.ts` 都说明：

- 系统有恢复预算
- 有局部缓存
- 有熔断和时间边界

无限重试不是更公平，  
而常常只是把局部问题升级成系统性噪声。

### 10.4 反公理四：解释越模糊越安全

从 `errors.ts` 的 reason family、`diagLogs.ts` 的事件面、`needs-auth` 的局部状态看，  
平台其实已经在努力形成更细语义。

真正危险的不是“解释太多”，  
而是把所有失败继续压缩成一句“不能用”。

## 11. 从第一性原理再压一层

如果你自己要设计一个 agent 平台，  
真正要回答的不是：

- 如何找出坏人

而是：

1. 何时必须重新证明主体
2. 何时必须重新证明资格
3. 何时必须重新证明会话
4. 何时可以局部撤权而不是全局终止
5. 何时可以乐观降级，何时必须阻塞确认
6. 如何在低敏感证据面上完成诊断与恢复

这些问题才是控制面哲学的核心。

## 12. 用苏格拉底式追问自我反思

### 问：为什么不把所有 gate 都设计成 fail-closed？

答：因为这会把大量普通网络波动和缓存陈旧直接转化成用户伤害。  
对低风险能力，这种保守并不划算。

### 问：为什么不把所有 gate 都设计成 fail-open？

答：因为高能力路径、高安全路径、危险设置变更都不是可随意乐观放行的。  
那会把平台风险转嫁给用户、组织和环境。

### 问：为什么不直接给用户一个总状态灯？

答：因为总状态灯很容易再次制造语义压缩税。  
真正有价值的是分层状态，而不是一个伪统一答案。

### 问：这一章还可以继续压缩吗？

答：还可以。  
如果再压一层，七条公理最终会收束成三句：

1. 高能力必须由高质量证明支撑。
2. 高伤害拒绝必须比低伤害放行更谨慎。
3. 恢复必须分层、分时钟、分角色进行。

## 13. 一句话总结

Claude Code 风控最深层的哲学，不是“多识别一些可疑用户”，而是“在高能力、多时钟、分层治理环境里，让放权、拒绝、恢复和解释都服从持续证明与总伤害最小化”。
