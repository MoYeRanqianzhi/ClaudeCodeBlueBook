# 损失函数视角：平台究竟在最小化什么，而用户又在失去什么

## 1. 为什么还要写“损失函数”

到现在为止，已经可以看出 Claude Code 的风控不是单点封禁系统。  
但如果再往深一层问：

- 平台到底在优化什么目标
- 用户到底在哪些地方承受损失

答案还没有被压得足够清楚。

这一章的目标就是把前面分散的现象，  
统一翻译成一组损失函数视角。

因为只有把损失函数看清，  
很多看似矛盾的设计才会变得统一：

- 为什么有时 fail-open
- 为什么有时 fail-closed
- 为什么有时宁可短时 stale
- 为什么有时必须硬收口
- 为什么有时强调 reset time 而不是继续试

## 2. 先说结论

从源码综合判断，Claude Code 并不是在最小化一个单一指标。  
它至少同时在最小化四类损失：

1. 平台安全损失
2. 不可恢复状态损失
3. 用户误伤损失
4. 支持解释损失

而系统的很多“怪异感”，  
恰恰来自这四类损失并不总是同向。

因此更准确的说法是：

`Claude Code 风控 = 多目标损失平衡系统，而不是单目标封禁器。`

## 3. 平台最怕的第一类损失：错误放权

这类损失的典型来源是：

- 高风险能力被错误放行
- 错误组织或错误身份路径获得能力
- 共享项目替用户自动打开危险边界

所以源码里会出现：

- deny / ask / safety check 先于 allow / bypass
- `projectSettings` 不能替用户打开高风险 opt-in
- `forceLoginOrgUUID` 不匹配时直接 fail-closed

从平台角度看，  
这类损失一旦发生，  
往往比一次短期误伤更难逆转。

## 4. 平台最怕的第二类损失：不可恢复状态

这一类在源码里非常明显：

- `ackWork` 一定要在 commit-to-handle 之后
- session ingress 采用单链 append，409 时采纳链头
- heartbeat fatal 时主动 re-dispatch 或 teardown 旧状态
- tool result 已见后被冻结，避免后续重放破坏前缀稳定

这些都说明平台在极力避免一种情况：

- 表面上好像还能跑
- 实际上已经进入不可解释、不可恢复、不可回溯的坏状态

所以很多设计不是为了“继续撑住”，  
而是为了避免掉进不可恢复深坑。

## 5. 用户最怕的第一类损失：误伤与误判

从用户视角看，最直观的损失不是平台安全，  
而是：

- 明明应该能用却不能用
- 明明问题不大却像被封了
- 明明可恢复却被拖成长期问题

这就是为什么会有：

- `checkGate_CACHED_OR_BLOCKING()` 对 stale `false` 更谨慎
- remote managed settings 的 fail-open / stale-cache fallback
- policy limits 的 graceful degradation
- 额度和 reset time 的显式提示

这些设计都在试图降低“把普通问题错误升级成用户灾难”的概率。

## 6. 用户最怕的第二类损失：解释成本和自证成本

这一类在高波动环境用户身上尤其重。

如果系统只告诉用户：

- 不能用

而不告诉他：

- 是额度
- 是局部 needs-auth
- 是组织不允许
- 还是高安全链路失效

那么用户就会立刻承受额外损失：

1. 误分流
2. 错误重试
3. 错误求助
4. 错误自我归因

所以 reason family、shared vocabulary、diagnostics 这些东西，  
其实都在降低解释损失。

## 7. 为什么有时 fail-open，有时 fail-closed

如果不用损失函数解释，  
这两者看起来像矛盾。

但一旦放回损失视角，它们其实很一致。

### 7.1 fail-open 出现在“错误阻断损失更大”的地方

例如：

- remote managed settings 抓取失败但已有 stale cache
- 没有 cache 时继续运行基础能力
- policy limits 普通抓取失败先降级

因为这里更大的风险往往不是短时多放行一点，  
而是让用户和系统一起跌入解释灾难。

### 7.2 fail-closed 出现在“错误放权损失更大”的地方

例如：

- 错误组织 token
- 危险设置变更
- 高风险 opt-in 来源不可信
- 高安全能力路径

因为这里一旦放错，  
损失比一次短时阻断更大。

## 8. 为什么短时 stale 有时比立即清空更好

`auth.ts`、keychain 缓存、remote settings stale cache 都说明：  
系统并不总追求“瞬时最新”。

原因很简单：

- 瞬时空真相
- 有时比短时旧真相更伤用户，也更伤恢复

如果一次 transient failure 就把状态清成：

- 未登录
- 无组织
- 无策略

那系统会在多个面同时制造假故障。

所以短时 stale 的存在，  
本质上是在最小化“误清空损失”。

## 9. 为什么 reset time 这么重要

`rateLimitMessages.ts`、`withRetry.ts` 都把 reset time 当正式语义。  
这不是用户体验装饰，而是损失函数的一部分。

因为当系统明确知道：

- 此时继续重试大概率无益
- 但等待到某个时间点后可恢复

最小损失路径就不再是：

- 多试几次

而是：

- 少做错误动作，等到正确时间点

这也是为什么把额度问题误写成封禁，  
会立刻拉长用户损失。

## 10. 对中国和高波动环境用户，这个损失函数意味着什么

最重要的结论不是“更容易被针对”，  
而是：

`你更容易因为连续性噪声而同时承受多类损失。`

例如：

1. 主体链抖动带来误伤损失
2. 会话链抖动带来恢复损失
3. 错误归因带来解释损失
4. 故障窗口乱试又进一步扩大这些损失

所以高波动环境用户真正该做的，不是规避系统，  
而是优先减少会放大这四类损失的动作：

- 乱换路径
- 并发重登
- 全局重置
- 情绪化描述问题

## 11. 从第一性原理再压一层

如果把这一章继续压缩，  
Claude Code 的风控其实在持续回答两个问题：

1. 现在最不能承受的损失是什么？
2. 为了避免它，当前该接受哪种次优损失？

这就是为什么系统有时宁可：

- 慢一点
- 保留 stale
- 进入 cooldown
- 暂时局部撤权
- 或要求明确重认证

它不是任性，  
而是在做损失排序。

## 12. 用苏格拉底式追问再压一层

### 问：为什么用户体验有时会觉得系统“前后不一致”？

答：因为不同时间点、不同层级正在优化不同损失。  
而用户只看到统一表面。

### 问：为什么“误伤少一点”和“风控严一点”不能简单同时最大化？

答：因为它们常常对应不同损失函数。  
你压低一种损失，往往会抬高另一种。

### 问：用户最该学会的是什么？

答：不是猜平台内部规则，  
而是识别当前自己正在承受哪类损失，并避免把它放大。

## 13. 一句话总结

Claude Code 的风控设计，本质上是在平台安全、不可恢复状态、用户误伤和支持解释成本之间做动态损失平衡；理解这一点，比把一切问题都叫“封号”更接近系统的真实本质。
