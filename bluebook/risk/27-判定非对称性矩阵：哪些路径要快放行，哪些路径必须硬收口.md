# 判定非对称性矩阵：哪些路径要快放行，哪些路径必须硬收口

## 1. 为什么要单独做这张矩阵

`12` 章已经解释了 selective fail-open / fail-closed 的哲学。  
但研究再往前走一步，会遇到一个更实际的问题：

到底哪些路径在优先保护连续性，哪些路径在优先保护边界？

如果不把这些路径放到一张矩阵里，读者很容易只记住一句抽象口号，  
却仍然不知道：

- 为什么有些地方愿意先让你继续
- 为什么有些地方即使只是网络抖动也会先把你拦下
- 为什么有些失败应该先恢复，有些失败却必须先确认

## 2. 先说结论

Claude Code 的非对称判定，本质上是在比较三种成本：

1. 错放风险成本
2. 误伤正常用户成本
3. 失败扩散与支持成本

如果一条路径更怕误伤，它就更可能偏向 fail-open、stale-cache 退化或先恢复。  
如果一条路径更怕越界、升权、危险变更或高权限远程滥用，它就更可能偏向 fail-closed。

## 3. 主体与策略路径矩阵

| 路径 | 主要真相源 | 默认倾向 | 为什么这样设计 | 用户表面体感 | 合规动作 |
| --- | --- | --- | --- | --- | --- |
| `checkSecurityRestrictionGate(...)` | GrowthBook / 安全侧缓存 | 偏保守 | 安全限制类 gate 更怕漏放 | 某些限制在 auth 变化后仍很快生效 | 不要把安全限制误解成普通 entitlement 缺失 |
| `checkGate_CACHED_OR_BLOCKING(...)` | GrowthBook / 新鲜服务端值 | 偏减少误伤 | stale `false` 会挡住本来有资格的用户 | 某项功能重登后恢复，或短时“忽然能用了” | 先核对 entitlement 与画像，不要先下封号结论 |
| `isRemoteManagedSettingsEligible()` | 本地资格预检 + 服务端空响应兜底 | 接受假阳性 | 多一次查询代价小于企业治理完全不可见 | 某些 token 虽画像不全，仍会继续查设置 | 不要把“继续请求服务端”误解成已确认有资格 |
| `policyLimits` 常规路径 | 服务端 restrictions + 本地缓存 | 大体 fail-open | 普通组织策略接口短时不可达，不应把全部用户一刀切打死 | 有时策略生效滞后一拍，但不会因单次超时全局崩 | 对普通策略问题优先核对缓存、轮询和组织状态 |
| `ESSENTIAL_TRAFFIC_DENY_ON_MISS` 例外 | 受限场景下的本地策略 | 局部 fail-closed | 高敏感环境不能因 miss 而重新放开敏感能力 | 个别反馈或附加能力在 miss 时直接不可用 | 先承认这是高敏感治理，不要当成随机故障 |
| `validateForceLoginOrg()` | 服务端 profile | 强 fail-closed | 组织边界不能信本地可写缓存 | 网络抖动或 scope 不足也会表现成“进不去” | 先核对组织与 full-scope login，再做支持举证 |
| 危险 managed settings 变更 | 远程设置 + 本地交互确认 | 强 fail-closed | 危险设置变更不能静默接受 | 出现阻塞确认框，拒绝后退出 | 把它看成治理确认，不要机械点通过 |

## 4. 会话与解释路径矩阵

| 路径 | 主要真相源 | 默认倾向 | 为什么这样设计 | 用户表面体感 | 合规动作 |
| --- | --- | --- | --- | --- | --- |
| `bridgeEnabled` 资格诊断 | 订阅、profile scope、组织画像、gate | 分层诊断，不是一刀切 | 高安全远程能力要求主体和画像都完整 | `requires subscription` / `full-scope login` / `not enabled` | 先把问题归到资格层，不要扩大成处罚层 |
| `heartbeatActiveWorkItems()` 中的 `401/403` | 会话 ingress JWT | 先恢复，不直接判死 | token 过期属于可恢复失败 | `auth_failed`、反复重连 | 冻结环境，减少试错，优先恢复身份连续性 |
| `heartbeatActiveWorkItems()` 中的 `404/410` | 环境实例真相 | 更接近终局失败 | 环境失效或删除再重试意义很低 | `fatal`、remote env unavailable | 把问题收敛到远程环境生命周期，不要只会刷登录 |
| `needs-auth` / `insufficient_scope` | MCP 子系统 auth | 域内恢复 | scope 不足与 token 失效不是一回事 | 某个连接器坏了，但别的功能还活着 | 只修该子系统，不要全局重置 |
| `rateLimitMessages` | 计费与额度状态 | 解释优先 | 额度耗尽不是处罚 | `usage limit` / `out of extra usage` | 按计费问题处理，不要误报封号 |
| `diagLogs` | 无 PII 诊断事件 | 支持恢复，不参与处罚 | 平台需要低敏感度证据来解释失败 | 用户无感，但支持能力增强 | 主动保留时间线和错误原文，配合支持路径 |

## 5. 这张矩阵真正揭示了什么

它揭示的不是“系统哪里严格、哪里宽松”，而是更本质的一点：

不同路径保护的对象不同。

- entitlement 路径保护的是公平可用性
- force-login-org 路径保护的是组织边界
- dangerous settings 路径保护的是本地执行边界
- bridge heartbeat 路径保护的是高安全会话连续性
- rate-limit 路径保护的是成本与计费真相
- diagLogs 路径保护的是解释与支持能力

因此，把所有失败都写成“风控封号”，在技术上并不准确。

## 6. 为什么这对中国和高波动网络环境用户尤其重要

高波动环境的问题不只是“更容易失败”，而是更容易让多条路径同时失败：

1. profile scope / 组织画像不完整
2. refresh 与 heartbeat 更脆弱
3. 设备与网络切换让主体连续性噪声更大
4. 错误语义被压缩后，更像统一处罚

所以，对这类用户最有价值的结论不是“如何躲避检测”，而是：

- 哪条路径出了问题
- 这条路径保护的到底是什么边界
- 应该保留什么证据
- 应该走哪条恢复/支持链

## 7. 从第一性原理和苏格拉底追问再压一层

### 问：为什么平台不统一选择 fail-closed？

答：因为那会把网络波动、画像延迟、短时服务故障都扩大成大面积误伤。

### 问：为什么平台也不能统一选择 fail-open？

答：因为组织越界、高权限远程会话、危险变更这类问题一旦错放，代价远高于单次中断。

### 问：什么才是更准确的抽象？

答：Claude Code 不是在做“统一风控”，而是在做“按边界对象选择失败语义”。

### 问：如果研究还想继续做得更好，最该补什么？

答：把这张矩阵继续压缩成对用户、支持、平台设计者都可直接执行的一页表。

## 8. 给平台构建者的设计启示

1. 不要设计一个全局统一的 failure policy。
2. 先定义每条路径在保护谁，再决定其 fail-open / fail-closed 倾向。
3. entitlement、组织边界、高安全会话、危险变更、计费真相、解释层应分开建模。
4. 能恢复的失败要被快速识别，不能恢复的失败要被快速解释。
5. 失败语义本身就是产品体验的一部分，不只是内部实现细节。

## 9. 一句话总结

Claude Code 风控最成熟的地方之一，不是“哪里都严格”，而是知道哪些路径要优先保护连续性，哪些路径必须优先保护边界；  
这正是它比“统一封禁开关”高级得多的原因。
