# 失效模式、封禁表现与合规使用边界

## 1. 先说结论

从源码看，Claude Code 的“封禁”更像一组失效模式，而不是单一处罚状态。

最常见的六类失效模式是：

1. 本地安全阻断：permission、classifier、sandbox。
2. token 失效或 OAuth 刷新失败。
3. 组织策略限制：policy limits / managed settings。
4. 组织绑定不匹配：force login org fail closed。
5. 远程高安全链路失效：trusted device / elevated session。
6. 账号能力域不可用：feature not enabled / environments unavailable。

这六类失效模式叠加起来，用户体感上就会形成“被风控了”“像被封了”。

## 2. 本地阻断不是封号，但经常是第一道可见症状

典型表现：

- 某个动作在 safety check 处要求人工批准
- auto mode classifier 直接 block
- denial limit 触发后回退到人工或 headless abort
- sandbox 因托管策略要求而变成硬门槛

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:526-548`
- `claude-code-source-code/src/utils/permissions/permissions.ts:818-911`
- `claude-code-source-code/src/utils/permissions/permissions.ts:980-1057`
- `claude-code-source-code/src/entrypoints/sandboxTypes.ts:95-119`

## 3. token 失效如果无法恢复，就会升级成账号不可用症状

普通 401 不一定等于账号出问题，因为系统会：

1. 清缓存
2. 重读 keychain
3. 尝试 refresh
4. 处理并发刷新竞争

证据：

- `claude-code-source-code/src/utils/auth.ts:1360-1561`

但如果这条链路恢复不了，最终表现就会变成：

- 功能长期不可用
- 远程链路 fatal
- 用户需要重新登录甚至切换身份源

## 4. 组织策略是最像“软封禁”的机制

对很多企业场景，真正重要的不是账号全局被封，而是某个能力被组织策略关掉：

- `allow_remote_control`
- managed permission rules only
- managed read/domain only

证据：

- `claude-code-source-code/src/services/policyLimits/index.ts:505-525`
- `claude-code-source-code/src/utils/permissions/permissionsLoader.ts:27-44`
- `claude-code-source-code/src/entrypoints/sandboxTypes.ts:18-24`
- `claude-code-source-code/src/entrypoints/sandboxTypes.ts:78-83`

这类限制对用户的体感非常像“被禁用”，但本质上是组织治理在生效。

## 5. 组织绑定失败是明确的 fail-closed

`validateForceLoginOrg()` 说明：

- 只要机器要求特定 org UUID，就必须用该组织 token。
- 取不到 profile 也不能放行。

证据：

- `claude-code-source-code/src/utils/auth.ts:1920-2000`

这是典型的“宁可拒绝可疑身份，也不容忍组织越界”。

## 6. remote/bridge 场景最接近真正的平台级限制

以下几种情况最像用户口中的“风控”：

- multi-session remote control 未开放
- remote environments 对该账号不可用
- trusted device / elevated auth 不成立
- 401/403 导致 bridge fatal

证据：

- `claude-code-source-code/src/bridge/bridgeMain.ts:2055-2075`
- `claude-code-source-code/src/bridge/bridgeMain.ts:2451-2466`
- `claude-code-source-code/src/bridge/trustedDevice.ts:18-27`
- `claude-code-source-code/src/bridge/bridgeMain.ts:1241-1266`

## 7. 从合规使用角度，源码给出的边界很明确

可以提炼出几条正向结论：

1. 不要把本地用户终端里的外部 key 与托管 OAuth 会话混用。
2. 不要假设本地设置一定能覆盖组织策略或 managed settings。
3. auto mode 只适合隔离环境，不适合把高风险动作交给生产环境直接执行。
4. Remote Control 不是普通同步功能，而是高安全能力域。
5. trusted device、org binding、policy limits 这些限制是体系设计的一部分，不是偶发报错。

这些结论都来自系统边界本身，不是额外的道德劝告。

## 8. 为什么这里不讨论“怎么绕过”

因为从源码已经能看出：

- 真正关键的限制多在服务端。
- 本地很多逻辑只是承接服务端决策。
- remote control 甚至引入了设备级 token 与 elevated auth。

讨论绕过并不会帮助理解系统本质，反而会误判控制中心所在位置。

更有价值的研究方向是：

- 哪些能力是本地阻断
- 哪些能力是组织封控
- 哪些能力是账号分桶
- 哪些能力属于远程高安全域

## 9. 一句话总结

Claude Code 的“封号/风控”从源码视角看，实质上是一套多层失效与能力撤回机制；理解它的关键不在于找一个 ban 变量，而在于分清每一层是本地阻断、组织治理，还是服务端能力域裁决。
