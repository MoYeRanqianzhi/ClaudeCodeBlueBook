# 高波动环境严格运行SOP：从日常纪律到升级求助的四阶段手册

## 1. 为什么还要再写一份 SOP

`28` 和 `47` 已经给了不少合规建议。  
但它们更像研究结论，  
还不够像一份可以照着执行的运行手册。

如果从高波动环境用户的真实处境出发，  
最重要的不是再听一遍抽象原则，  
而是把动作顺序固定下来：

1. 平时怎么运行
2. 开工前怎么检查
3. 故障窗口前 5 分钟怎么止损
4. 什么时候该升级求助，带什么材料

这章就是把前面的结论压成四阶段 SOP。

## 2. 先说边界

这份 SOP 只讨论合规路径：

- 不提供伪装、绕过、对抗检测的方法
- 不鼓励规避组织或平台策略

它的目标只有两个：

1. 降低误伤概率
2. 缩短误伤后的自证与恢复时间

## 3. 总原则

高波动环境用户最容易吃亏的地方，不是“技术一定更差”，  
而是：

- 连续性噪声更高
- 解释成本更高
- 故障窗口里更容易做过多动作

所以这份 SOP 的核心原则只有四条：

1. 主路径稳定
2. 变量冻结
3. 第一现场优先
4. 共享词汇升级

## 4. 第一阶段：平时运行纪律

### 4.1 固定主路径

长期只维持一套主路径：

1. 一个主力设备
2. 一个主力 `CLAUDE_CONFIG_DIR`
3. 一个主身份路径
4. 一个主工作环境

不要把备用路径当日常路径使用。

### 4.2 把工作环境、个人环境、测试环境彻底分开

不要在同一环境中混用：

- 个人账号
- 组织账号
- 临时 token
- 临时远端/容器环境

你越混用，  
越容易把主体链、资格链和组织边界搅成同一团。

### 4.3 每周做一次低成本体检

优先检查：

1. `claude auth status --json` 或文本输出是否仍和你的预期一致
2. 当前 `authMethod`、`orgId`、`subscriptionType` 是否仍正确
3. 最近是否换过 config root、机器、网络出口
4. 某些常用 MCP/connector 是否仍在健康状态

这一步的目的不是折腾配置，  
而是提早发现连续性债务。

## 5. 第二阶段：开工前 2 分钟体检

### 5.1 高价值任务前先看三件事

1. 身份是否正确
2. 组织是否正确
3. 当前任务是否依赖高安全远程/某个 connector

### 5.2 用低成本状态面，不要等出错了才第一次看

从源码看，`auth status` 和 MCP 连接通知面本来就是为这种场景设计的：

- `authStatus()` 能输出 `authMethod`、`apiProvider`、`orgId`、`subscriptionType`
- `useMcpConnectivityStatus.tsx` 会把 failed / needs-auth 按本地 server 和 claude.ai connector 区分出来

因此，真正成熟的运行方式应该是：

- 在开工前主动看状态面
- 而不是问题来了再猜

### 5.3 高波动窗口不做高风险启动

如果你已经知道当前：

- 网络异常波动
- 设备刚切换
- 组织路径刚变
- connector 刚进入 `needs-auth`

那就不要在这个窗口启动高价值长链路任务。

## 6. 第三阶段：故障窗口前 5 分钟止损

### 6.1 先停，不先试

首次出现异常后的第一动作，不是继续试，  
而是先停 30 秒确认：

1. 错误原文
2. 首次失败时间
3. 最近一次成功时间
4. 当前是否发生过设备/网络/身份/组织变化

### 6.2 冻结四类变量

故障窗口里禁止同时变动：

1. 身份变量
2. 组织变量
3. 设备变量
4. 网络变量

否则你等于在主动破坏第一现场。

### 6.3 先分层，再动作

先问：

1. 这是 auth / org / quota / connector / remote 高安全链中的哪一层
2. 是全局问题还是局部问题
3. 是主体链、资格链还是会话链的问题

只有分完层，  
后续动作才值得做。

## 7. 第四阶段：升级求助与证据包

### 7.1 什么时候该升级求助

以下情况不要继续自试太久：

1. 同一错误跨越多个时间窗口仍存在
2. `auth status` 已与预期明显不一致
3. 组织闭锁/组织不匹配信息明确出现
4. 单个 connector/bridge 高安全链持续 `needs-auth` 或 `auth_failed`
5. 你已经无法在不增加新变量的情况下继续定位

### 7.2 升级时最少带六样东西

1. 首次失败绝对时间
2. 错误原文
3. 最近一次成功时间
4. `auth status` 关键信息
5. 当前受影响的是全局还是局部域
6. 故障窗口里是否有设备/网络/身份/组织变化

### 7.3 用共享词汇升级，不用情绪词升级

优先这样写：

- 当前更像 `needs-auth`
- 当前更像 `auth_error`
- 当前更像组织不匹配
- 当前更像 session limit / weekly limit
- 当前更像高安全远程链 `auth_failed`

不要只写：

- 我被封了

## 8. 对中国和高波动环境用户的特别补充

如果你处在中国或其他高波动环境里，  
再额外坚持三条：

1. 主力任务尽量固定在低波动时间段执行
2. 尽量减少容器、远端和本机之间的身份路径切换
3. 遇到异常时优先保存状态面和日志面，而不是继续尝试“碰运气恢复”

你真正要保护的不是“短期多试几次的机会”，  
而是长期可解释性。

## 9. 从第一性原理看，这份 SOP 为什么有效

因为它在做四件很朴素的事：

1. 降低连续性噪声
2. 降低误分流概率
3. 降低解释成本
4. 降低第一现场被覆盖的概率

这四件事，  
恰好就是高波动环境用户最容易失去的东西。

## 10. 用苏格拉底式追问再压一层

### 问：为什么 SOP 看起来比“恢复技巧”更保守？

答：因为恢复技巧常常依赖隐含假设，  
而 SOP 依赖的是可重复执行的纪律。

### 问：为什么先看状态面比先试更重要？

答：因为你需要先知道自己在哪一层失败，  
否则所有动作都可能做反。

### 问：为什么高波动环境用户更需要纪律而不是更多选择？

答：因为在高波动环境里，  
更多选择通常意味着更多变量，  
而更多变量直接等于更高自证成本。

## 11. 一句话总结

对高波动环境用户来说，最有效的合规自保不是更多技巧，而是更严格的四阶段运行纪律：平时稳定主路径、开工前做轻量体检、故障窗口先冻结变量、升级求助时用结构化证据和共享词汇。
