# 平台正义、误伤、公平性与可解释性

## 1. 为什么这一章必须存在

只研究“平台如何限制用户”，是不够的。  
真正成熟的研究还必须追问：

这套限制是否足够公平？  
误伤如何被控制？  
普通用户是否能理解自己为什么被挡住？

这就是平台正义问题。

## 2. 先说结论

从当前源码看，Claude Code 的风控设计有两个方向同时存在：

1. 它在尽量提高治理能力和平台可持续性。
2. 它也在若干关键位置尝试降低误伤并提升可解释性。

但这套系统仍然留下一个张力：

平台越擅长治理，普通用户越容易把复杂的分层拒绝体验成“我是不是被封了”。

所以，真正决定这套系统是否“正义”的，不只是它拦得准不准，还包括：

- 它是否给出足够好的 reason code
- 它是否让误伤用户有足够低成本的恢复和申诉路径
- 它是否区分资格缺失、组织治理和账号处罚

## 3. 源码里能看到哪些“朝向公平性”的设计

### 3.1 减少陈旧 `false` 误伤用户

某些 entitlement gate 会因为“陈旧 false 会不公平拦住用户”而走 blocking check。

这说明作者知道：

不是所有保守缓存都天然更安全。  
有时更安全的做法，反而是主动减少误伤。

### 3.2 给出更具可解释性的错误信息

例如 bridge / remote-control 的 unavailable reason，并不是统一一句“不可用”，而会区分：

- 需要订阅
- 缺少 full-scope token
- 无法确定组织
- 账号尚未开放

这类错误信息的价值，不只是更友好，而是更公正：

它让用户知道自己到底是资格问题、登录问题还是组织问题。

### 3.3 危险远程变更要求显式确认

managed settings 的危险变更不是静默应用，而是需要确认。  
这其实也是平台正义的一部分：

如果平台要重写用户本地执行边界，就应当让用户知道自己正在面对什么变化。

### 3.4 给出切换到已有订阅的提示

源码里甚至专门有提示逻辑，告诉用户：

你其实已经有 Max/Pro 订阅，只是当前没有走到正确的登录路径。

这类提示看似细小，但它体现了一种很重要的设计态度：

平台不只是“拒绝”，还试图引导用户回到正确权益路径。

## 4. 这套系统在哪些地方仍然会制造不公平体感

### 4.1 太多不同失败最终汇聚成“不能用”

从用户视角：

- `401`
- `403`
- `needs-auth`
- `policy denied`
- `not enabled for your account`
- `rate limit`

都可能被体验成“我被针对了”。

这会造成强烈的不公平体感，即使从平台角度看，它们完全不是一类事件。

### 4.2 高波动环境用户更容易承担解释成本

那些网络环境、设备连续性、支付和组织关系更复杂的用户，会更容易同时触发多层不稳定因素。  
结果就是：

- 平台的治理逻辑未必更针对他们
- 但他们更容易承担更高的解释成本和误伤成本

这也是为什么很多中国用户会主观上感到自己更容易“像被封”。

### 4.3 高安全链路的失败更容易被误读成处罚

Remote Control、trusted device、高权限远程环境一旦失败，用户非常容易把这种“高安全链路不成立”理解成“账号被针对”。  
这说明系统在高安全能力上，仍然需要更强的解释层。

## 5. 从第一性原理看，平台正义意味着什么

如果回到第一性原理，平台正义至少包括四个条件：

1. 同样的错误要有稳定一致的语义。
2. 不同层的失败要能被区分。
3. 用户要有恢复和申诉路径。
4. 平台不能让解释成本全部压在用户头上。

如果只做限制而不做解释，系统会很强，但不够正义。  
如果只做解释而不做治理，系统会很柔，但不够可持续。  
真正成熟的设计必须把这两件事同时做到。

## 6. 用苏格拉底式追问再往下压一层

### 问：平台为了自保而提高门槛，是否天然不公正？

答：不天然。

只要这些门槛：

- 针对的是风险后果
- 而不是任意地压给某类用户
- 并且保留了恢复和解释路径

它就可能是正当治理，而不是任性管制。

### 问：如果用户主观上感觉被针对，这种感觉就一定是错的吗？

答：也不一定。

即使平台的内部逻辑是分层的、合理的，  
只要外部体验把太多不同失败压成“不能用”，  
用户的“被针对感”就是真实存在的。

这说明平台正义不仅是内部判定合理，也包括外部体验是否可解释。

### 问：最该改进的不是风控强度，而是什么？

答：是 reason code、解释层和支持链路。

因为很多平台冲突，根本不是出在“该不该管”，而是出在“用户根本不知道自己到底为什么被管”。

## 7. 对研究者自己的警告

研究风控时，也要避免两种不公正：

1. 过度把所有限制都写成“平台恶意封号”
2. 过度替平台合理化每一种用户痛感

更好的写法是：

- 承认平台有治理必要性
- 也承认误伤和解释成本是真问题
- 用分层模型把二者同时写清

## 8. 一句话总结

Claude Code 的风控系统在工程上已经相当成熟，但是否真正“好”，最终不只取决于它能不能挡住风险，还取决于它能不能在挡住风险的同时，给普通用户足够的公平感、解释性和恢复路径。
