# 苏格拉底附录：如果要把误伤再降一半，系统该追问什么

## 1. 为什么还要再写这一章

前面的章节已经回答了很多“它现在是怎么做的”。  
但如果目标不是描述现状，而是继续把风控、误伤处置和平台公平性做得更好，就必须继续追问：

- 哪些默认前提其实没有被充分证明？
- 哪些拒绝虽然合理，却仍然解释得不够好？
- 哪些高波动环境用户承担了过多的连续性成本？
- 哪些问题不是风控太弱或太强，而是 reason code 太粗？

这一章不是重复总结，而是用第一性原理和苏格拉底诘问法，逼迫研究和系统一起提高标准。

## 2. 先回到第一性原理

如果一个强能力 agent 平台真的要长期存在，它至少要同时优化五件事：

1. 不让高风险权限被错误授予。
2. 不让正常用户因为证明链脆弱而频繁误伤。
3. 不让组织边界被静默绕开。
4. 不让失败风暴把偶发故障放大成系统性惩罚体感。
5. 不让支持与申诉成本高到只有最强用户才能自救。

因此，成熟风控追求的不能只是“拦住更多”，而必须是“总伤害更低”。

这里的总伤害至少包括：

- 滥用带来的伤害
- 误伤带来的伤害
- 恢复失败带来的伤害
- 解释不足带来的伤害
- 支持成本带来的伤害

## 3. 用苏格拉底式追问拆开这套系统

### 问：如果用户只能看到 `not enabled`，那系统真的完成了解释义务吗

答：不一定。

因为 `not enabled` 可能来自：

- entitlement 缺失
- rollout 未开放
- 组织画像缺失
- full-scope token 不足
- build 不支持

如果这些差异在用户面前仍被压缩成一句话，那么系统在治理上也许是正确的，但在平台正义上仍然是不充分的。

### 问：如果组织验证失败只是因为网络波动，fail-closed 仍然完全公平吗

答：从边界保护看，它是合理的；从用户体验看，它仍然欠一个更强的恢复入口。

`forceLoginOrgUUID` 这类设计说明平台把组织边界放在高优先级。  
但如果“无法证明”与“证明失败”在体验上没有被充分区分，用户就会承担过多解释成本。

所以真正的问题不是“该不该 fail-closed”，而是：

- fail-closed 之后是否给了足够清晰的 preflight 和 recovery
- 是否让用户知道自己错在 scope、网络还是组织本身

### 问：如果高安全会话依赖 fresh login、trusted device 和稳定网络，平台是不是该先做会前体检

答：应该。

当前源码已经能看出系统很重视：

- full-scope token
- trusted device
- elevated session
- 401/409 恢复

但研究越深入，越能发现一个现实问题：

很多错误发生在“用户已经开始干活之后”。

这说明平台还可以更进一步，把“会前体检”前置成正式产品层，而不是让用户在工作中途才意识到自己少了某个安全前提。

### 问：如果高波动环境用户更容易连续性断裂，平台是否应该把这只理解成用户问题

答：不应该。

更准确的理解是：

- 用户确实需要更稳定地使用系统
- 但平台也应承认自己的连续性假设存在环境偏置

如果某类用户因为网络、设备或身份路径现实约束，更容易落入“无法证明主体连续”的状态，那平台就应投入更多解释和恢复能力，而不只是要求用户“更规范”。

### 问：如果 diagnostics 严格避免 PII，支持系统还能不能足够强

答：可以，但前提是 reason code 和证据包设计必须更好。

`logForDiagnosticsNoPII(...)` 代表了正确边界：不能为了可观测性无限采集。  
但这也意味着另一项责任会变重：

- 必须给用户更强的结构化错误码
- 必须给支持团队更强的无 PII 调查索引
- 必须让用户自己能导出更有用的证据包

## 4. 如果要把误伤再降一半，平台该做什么

### 4.1 把 reason code 继续细化成用户可操作的诊断语义

至少要继续把以下几类情况分开：

- entitlement absence
- org policy denial
- full-scope token missing
- step-up auth required
- trusted device missing
- continuity broken by session/network
- billing or usage exhaustion

只要这些还经常共享一个表面提示，误伤体感就难以下降。

### 4.2 给高安全能力增加 preflight health check

在真正启动 Remote Control、bridge 或高安全会话前，平台完全可以先明确告诉用户：

- 订阅是否满足
- 当前 token scope 是否满足
- 组织画像是否完整
- trusted device 是否就绪
- 当前网络/会话是否适合继续

这会比事后再报错更公平。

### 4.3 给误伤用户自动生成结构化证据包

最理想的状态不是让用户自己手工收集一切，而是平台直接导出：

- 时间线
- reason code
- 最近一次成功时间
- 身份路径类型
- 组织/订阅状态摘要
- 是否发生网络或设备切换

而且全程不暴露 prompts、仓库名和文件路径。

### 4.4 把“高波动环境连续性破坏”单独当成产品问题

如果连续性断裂在特定环境下更常见，平台不应只把这理解为支持问题。  
它也应当被视为产品设计问题和公平性问题。

更好的方向包括：

- 更清晰的网络/会话健康提示
- 更少歧义的恢复文案
- 更强的“这是认证问题，不是处罚”显式提示

## 5. 如果要把研究本身做得更好，我们也该追问什么

### 5.1 我们是不是把客户端证据写得太像服务端事实

必须持续克制。

客户端源码能强有力地说明：

- 哪些信号被采集
- 哪些门槛被执行
- 哪些错误被翻译
- 哪些恢复路径被提供

但它不能直接证明：

- 服务端最终裁决权重
- 真实封禁策略全集
- 区域性动机

### 5.2 我们是不是把“高封号体感”写得太像“平台一定针对”

不能这样写。

更稳妥的研究表达必须始终保持：

- 对地域动机保持克制
- 对连续性破坏保持明确
- 对误伤体感保持同情但不越证据边界

### 5.3 我们是不是还不够重视“解释层就是风控的一部分”

前面的章节已经在强调这一点，但还可以更彻底。

未来如果继续深入，最值得继续补强的不是“还能再找到多少 gate”，而是：

- gate 如何被翻译
- 翻译如何影响误伤
- 误伤如何影响平台正义

## 6. 对用户而言，真正有用的反思是什么

用户最值得反复提醒自己的，不是“怎样更像正常用户”，而是：

1. 我有没有让自己的主体连续性尽量稳定。
2. 我是不是把个人、组织、设备、网络路径混在一起了。
3. 我是在解决问题，还是在制造更多变量。
4. 我遇到的是处罚，还是资格、授权、计费、组织、会话中的别的问题。
5. 我有没有留下足够保护自身权益的证据。

如果这五个问题能在异常发生的前五分钟就被问出来，很多“像被封了”的路径都会明显收敛。

## 7. 一句话总结

真正高水平的风控，不只是更会拒绝，而是更会证明、区分、解释、恢复和善后；真正高水平的研究，也不只是更会找 gate，而是更会追问系统是否把错误的成本公平地分配给了用户。
