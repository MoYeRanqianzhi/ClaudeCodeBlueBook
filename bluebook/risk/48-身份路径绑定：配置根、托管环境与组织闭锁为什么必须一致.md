# 身份路径绑定：配置根、托管环境与组织闭锁为什么必须一致

## 1. 为什么还要写这一章

前面已经写过：

- 认证连续性工程
- 仓库不是可信主体
- 高波动环境用户权益保护

但还有一条更细、也更硬的源码事实值得单独抽出来：

`Claude Code 不只是验证“你是谁”，还在验证“你是从哪条身份路径进入这个环境的”。`

这条结论来自几个看似分散的实现：

- `CLAUDE_CONFIG_DIR`
- keychain service name 的路径哈希
- `.credentials.json` 所在位置
- managed session 不允许回退到终端私有 API key
- `forceLoginOrgUUID` 的 fail-closed 验证

把它们放在一起后，会得到一个更明确的结论：

`身份不是抽象 token，而是被绑定在配置根、存储槽位、组织要求和运行场景上的。`

## 2. 先说结论

Claude Code 的认证和风控设计里，有一条很强的隐含原则：

`同一个受管环境，必须沿同一条可解释的身份路径进入；不能随意在配置根、存储后端、组织上下文和凭证来源之间跳转。`

因此，系统会做四件事：

1. 把凭证存储绑定到特定 config home
2. 把 keychain 槽位绑定到特定 config home
3. 在 managed context 下禁止回退到不相干的本地凭证
4. 对组织不匹配或不可验证的 token 直接 fail closed

这不是小题大做，  
而是为了防止“凭证存在”被误写成“身份路径正确”。

## 3. 源码为什么支持这个结论

### 3.1 config home 本身就是身份路径的一部分

`envUtils.ts` 的 `getClaudeConfigHomeDir()` 并不是普通路径函数。  
它把 Claude 的本地状态统一绑定到：

- `CLAUDE_CONFIG_DIR`
- 或默认的 `~/.claude`

随后：

- teams
- credentials
- 缓存
- 其他本地持久状态

都会沿着这条根路径派生。

这说明在 Claude Code 里，  
配置根不是无关紧要的本地细节，  
而是身份连续性的根目录。

### 3.2 keychain 槽位也绑定到 config home

`macOsKeychainHelpers.ts` 更进一步：

- 默认路径与自定义路径对应不同的 keychain service name
- 自定义 `CLAUDE_CONFIG_DIR` 还会参与哈希后缀

这意味着即使在同一台机器上：

- 不同 config home
- 也不是同一个凭证槽位

换句话说：

`切换 config home，本质上是在切换身份连续性的存储宇宙。`

### 3.3 plaintext fallback 也沿同一 config home 写入 `.credentials.json`

`plainTextStorage.ts` 显示：

- plaintext fallback 仍然写在当前 config home 下的 `.credentials.json`
- 权限还会被压到 `0600`

`fallbackStorage.ts` 又说明：

- 如果主存储失败并写入 secondary
- 还要尽量删除旧 primary，避免旧凭证继续遮蔽新凭证

这说明即使做 fallback，  
作者也在尽量维持“当前 config home 下只有一条权威身份路径”的语义。

### 3.4 managed session 明确禁止回退到终端私有凭证

`auth.ts` 里的 `isManagedOAuthContext()` 注释非常直白：

- CCR / Claude Desktop 这类 managed sessions
- 不应回退到用户自己 `~/.claude/settings.json` 里的 API key 配置

原因也写得很清楚：

- 用户终端上的私有 key
- 不等于受管环境应使用的 key
- 否则 stale / wrong-org key 会污染 managed session

这是这一章最关键的源码信号之一。  
它说明平台明确区分：

- 用户终端身份路径
- 受管环境身份路径

二者不能随意互借。

### 3.5 组织闭锁不是 advisory，而是 fail-closed

`validateForceLoginOrg()` 进一步把这条原则推到更强：

- 如果 `forceLoginOrgUUID` 存在
- 就必须去服务端 profile endpoint 校验组织
- 本地缓存 org UUID 不可信
- 验证失败或 profile 拉不到，就直接 fail closed

而且如果 token 来自 `CLAUDE_CODE_OAUTH_TOKEN` 或 FD，  
还会明确告诉用户：

- 这个环境变量 token 属于错误组织
- 请移除它，或换成正确组织的 token

这说明系统不接受一种含混状态：

- token 有
- 但组织路径不对

在受管环境里，这种状态直接被视为不可接受。

## 4. 从第一性原理看，这章真正讲的是什么

这一章真正讲的，不是路径配置技巧。  
而是：

`主体证明必须带来源证明。`

只有 token，  
不够。  
只有能用，  
也不够。

系统还要知道：

1. 这个 token 来自哪个配置根
2. 这个配置根对应哪个存储槽位
3. 这个运行场景是否允许使用这条凭证来源
4. 这个主体是否属于当前机器要求的组织

因此更本质的抽象不是：

- 我有 token

而是：

- 我正沿着被允许的身份路径，在正确组织边界内持有这个 token

## 5. 给用户的合规启示

### 5.1 不要把切 config home 当成无成本动作

如果你频繁切：

- `CLAUDE_CONFIG_DIR`
- 容器内外不同 config home
- 不同宿主映射的 `.claude`

你就在主动切换身份连续性的根。

### 5.2 受管环境不要混入终端私有凭证

如果当前是：

- 远程受管 session
- 桌面受管入口
- 明确要求组织闭锁的机器

就不要把终端私有 env token / API key 当成通用补丁。  
这类回退很可能只会制造 wrong-org 或 header mismatch。

### 5.3 组织要求明确时，先修正身份路径，再谈恢复

如果机器明确要求某个组织，  
那问题的核心就不是“先让它跑起来”，  
而是“让主体路径先回到正确组织边界里”。

## 6. 面向中国和高波动环境用户的合规结论

高波动环境用户更容易在以下动作里制造高噪声：

1. 临时切不同 config home
2. 容器、远端、宿主之间交叉挂载 `.claude`
3. 受管环境里塞入临时 env token
4. 组织不匹配时继续用错误路径硬试

这些动作看似是在“增加恢复机会”，  
但从源码逻辑看，更像是在破坏身份路径可解释性。

因此更有效的做法是：

1. 固定主 config home
2. 固定受管环境的凭证来源
3. 组织闭锁时先校正组织，再继续使用
4. 把路径一致性当成权益保护的一部分

## 7. 给平台构建者的启示

如果你在做自己的 agent 平台，  
这一章最值得迁移的是：

1. 让 config root 成为正式身份边界
2. 让不同 config root 对应不同 secure storage 槽位
3. 在 managed context 下显式禁止不相干的 credential fallback
4. 对组织路径不匹配做 fail-closed，而不是模糊运行

这样做会更严格，  
但能显著降低“凭证是对的、路径却是错的”这类高解释成本故障。

## 8. 用苏格拉底式追问再压一层

### 问：为什么 token 还不够？

答：因为 token 只证明“某种访问能力存在”，  
不证明它来自正确环境、正确组织、正确身份路径。

### 问：为什么要把 config home 绑定进身份语义？

答：因为本地连续性必须依附在一个稳定的状态根上。  
没有稳定状态根，就没有可解释连续性。

### 问：为什么受管环境不该回退到终端凭证？

答：因为那会把两个不同治理边界混成一个，  
最后谁都解释不清。

## 9. 一句话总结

Claude Code 的身份证明不是“拿到 token 就行”，而是把 token 绑定在 config home、secure storage 槽位、托管场景和组织边界上的一条完整身份路径；对用户来说，稳定这条路径本身就是降低误伤和保护权益的重要条件。
