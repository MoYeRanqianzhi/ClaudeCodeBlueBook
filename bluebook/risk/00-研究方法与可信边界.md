# 研究方法与可信边界

## 1. 这部分在研究什么

本专题研究的是 Claude Code 的账号治理、能力封控、组织策略、远程高安全链路和本地执行边界。

重点不是寻找一个“封号变量”，而是回答三个问题：

1. 客户端如何承接服务端的能力限制与风险治理。
2. 哪些“不可用”是账号资格问题，哪些是组织策略问题，哪些只是本地动作阻断。
3. 仅凭客户端源码，哪些结论可以稳定成立，哪些只能保持审慎推断。

## 2. 研究方法

本专题采用四步法：

### 2.1 先按治理面分组

优先研究五类模块，而不是先按功能菜单拆：

1. 身份与鉴权
2. 订阅、资格与能力域
3. 组织策略与远程下发
4. 遥测、实验开关与持续观测
5. 本地执行安全与高安全会话

### 2.2 再用用户可见现象回看

重点关注这些表象：

- `/login`
- `401/403`
- `not enabled for your account`
- `disabled by your organization's policy`
- 本地 risky action 被 block
- remote environment / trusted device 失效

只有把现象和治理模块一一对应，才不会把所有失败都误写成“封号”。

### 2.3 强结论必须由多模块共同支撑

例如：

- 账号资格结论，要同时看到鉴权路线、订阅门槛和能力文案。
- 组织治理结论，要同时看到 policy limits、managed settings 和运行时拒绝。
- 高安全会话结论，要同时看到 bridge、trusted device、fatal/auth_failed 路径。

### 2.4 证据只做支撑，不复制源码正文

本专题允许写证据，但遵循两个约束：

- 优先写模块、状态迁移和用户可见行为。
- 不直接搬运大段源码，不把蓝皮书写成源码镜像。

## 3. 当前可信边界

### 3.1 可以确认

- Claude Code 的风控不是单点开关，而是身份、资格、组织策略、执行边界与观测闭环的组合治理。
- 客户端具备承接服务端治理的正式控制面，包括 policy limits、managed settings、GrowthBook 和 telemetry。
- Remote Control / bridge 确实引入了更高等级的账号、设备和会话约束。

### 3.2 不能确认

- 服务端具体的封禁阈值、人工审核规则和 abuse score。
- 哪些遥测字段一定进入了处罚模型。
- Anthropic 服务端是否还有客户端不可见的额外治理链路。

## 4. 当前最重要的结论

到目前为止，最稳妥的一句话结论是：

Claude Code 的“封禁/风控”在客户端源码里更像一套多层能力撤回与会话治理系统，而不是一个显式的 ban switch。

## 5. 本专题的使用方式

- 想先看方法论和证据纪律，从本章开始。
- 想直接看结论，从 `01` 到 `06` 顺读。
- 想追更细的实现证据，再回到 `docs/development/research-log.md`。
