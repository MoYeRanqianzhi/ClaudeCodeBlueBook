# 批准链分析：谁有资格替用户说“可以”，以及这本身为何是风控问题

## 1. 为什么还要单独写“批准链”

前面的章节已经多次讨论：

- 权限规则
- deny / ask / allow
- 托管策略
- 支持与证据面

但还有一个更底层的问题没有被单独抽出来：

`在 Claude Code 里，究竟谁有资格代表用户批准一个高风险动作？`

这个问题非常关键。  
因为一旦把它想清楚，很多看似分散的设计会自动收束：

- 本地弹窗
- hook 决策
- classifier 决策
- config allow
- channel relay 批准
- allowlist / policy 限制

这些都不是“交互细节”，  
而是批准链本身。

## 2. 先说结论

从源码看，Claude Code 并不把“用户点击了是/某个服务器返回了是”当成批准链的全部。  
它更像是在维护一条分层批准链：

1. 谁发起请求
2. 谁拥有批准资格
3. 批准是通过哪一条信任面传来的
4. 这条信任面是否被 allowlist / policy 接纳
5. 批准结果是否被正式记录和归因

因此更准确的说法是：

`Claude Code 的权限系统，不只是在判断动作能否执行，也在判断“谁能合法代表用户说可以”。`

## 3. 源码里的批准链长什么样

### 3.1 批准不是一个按钮，而是一组来源

`permissionLogging.ts` 明确把批准和拒绝来源做了结构化建模：

- `config`
- `user`
- `hook`
- `classifier`

而且所有批准/拒绝都会被统一扇出到：

- analytics
- OTel
- code-edit metrics
- `toolUseContext.toolDecisions`

这说明在系统内部，“批准”从来不是无来源的。  
批准必须带着：

- 来源
- 时间
- 工具名
- 等待时长

一起进入正式事件面。

### 3.2 替代批准面也要经过单独信任边界

`channelPermissions.ts` 更进一步。  
它允许 Telegram、iMessage、Discord 这类 channel 参与 permission prompt relay，  
但这里的关键不是“手机上也能点同意”，  
而是：

- 必须走结构化 `notifications/claude/channel/permission`
- CC 不直接把普通文本当成批准
- 只有 allowlist 内、声明了相应 capability 的 channel server 才能成为批准面

这说明平台承认：

- 批准可以不发生在终端本地

但同时也明确要求：

- 替代批准面必须先被治理系统接纳

### 3.3 allowlist 才是真正的信任边界

`channelAllowlist.ts` 的逻辑非常清楚：

- channels 总开关由 GrowthBook 控制
- 具体插件是否允许成为 channel 面，由 allowlist 决定
- allowlist 是按 `{marketplace, plugin}` 比较，而不是“谁都能来”

`channelPermissions.ts` 甚至在注释里直接说明：

- 真正的 trust boundary 不是 terminal
- 而是 allowlist

这句话很重要。  
它意味着系统并不把“能转发消息”自动等同于“能替用户批准”。

## 4. 为什么这本身就是风控问题

很多人会把权限批准理解成纯交互问题：

- 提示一下
- 用户同意
- 就执行

但从源码看，Claude Code 认为更本质的问题是：

`批准权本身就是高风险能力。`

因为一旦某条面可以替用户批准，  
那条面就不再只是沟通通道，  
而成了治理入口。

所以系统必须进一步判断：

1. 这条批准面是否可信
2. 这条批准面是否在 policy 允许范围内
3. 如果它被攻破，会额外增加什么能力

这正是风控问题，而不是 UI 问题。

## 5. 从第一性原理看，平台到底在防什么

这套设计实际在防三种风险。

### 5.1 防“批准权漂移”

如果任意通道都能替用户批准，  
那么批准权就会从用户本人漂移到：

- 任意插件
- 任意 relay
- 任意共享服务器

这显然不可接受。

### 5.2 防“批准与执行脱钩”

如果批准结果不能被正式归因和记录，  
那系统后续就无法回答：

- 是谁批准的
- 通过哪条面批准的
- 这次批准是否属于正常路径

而 `permissionLogging.ts` 正是在防这个问题。

### 5.3 防“社交通道被误认成治理通道”

`channelPermissions.ts` 最有价值的一点在于它承认：

- 社交通道可以成为治理通道

但不会直接默认：

- 任何社交通道都是治理通道

这就是成熟系统和随手拼装系统的差异。

## 6. 对用户意味着什么

### 6.1 不是所有“我能回复 yes 的地方”都等于同一层批准

用户最该理解的一点是：

- 批准发生在哪条面上
- 决定了它的可信度和后续解释能力

本地终端、受控 hook、classifier、allowlisted channel relay，  
虽然都可能导致动作继续，  
但它们在治理意义上并不等价。

### 6.2 高波动环境用户更应减少批准面的复杂度

如果你本来就处在高噪声环境里，  
再同时引入过多替代批准面，  
往往只会放大：

- 解释成本
- 信任边界复杂度
- 故障排查难度

更好的合规做法是：

1. 尽量明确主批准面
2. 明确哪些替代面是你真正信任的
3. 出问题时能说清批准到底通过哪条面发生

## 7. 对平台构建者的技术启示

如果你在做自己的 agent 平台，  
最值得迁移的不是某个具体 allowlist 字段，  
而是下面这套批准链设计：

1. 把批准来源结构化建模
2. 把批准事件正式接入 telemetry
3. 把替代批准面单独设为 trust boundary
4. 让 allowlist / policy 决定谁有资格成为批准面
5. 让批准权和普通消息权严格区分

这套结构会明显提高系统复杂度，  
但也会把“谁有资格替用户说可以”这件事真正做清楚。

## 8. 用苏格拉底式追问再压一层

### 问：为什么批准链本身也属于风控？

答：因为批准权就是一种能力。  
谁能批准，本身就决定了谁能推动高风险动作通过。

### 问：为什么不能只相信“最终是人回复了 yes”？

答：因为系统还必须知道：

- yes 是怎么被解释成批准的
- 通过哪条信任面进入系统
- 这条面是否在治理上被允许

### 问：为什么要把批准结果做成正式事件？

答：因为不这样做，  
后续就无法归因、无法审计、无法优化误伤。

## 9. 一句话总结

Claude Code 的权限系统不仅在判断“能不能做”，也在判断“谁有资格替用户说可以”；替代批准面之所以要被 allowlist、telemetry 和 policy 包围，是因为批准权本身就是风控对象。
