# 冻结语义与单链恢复：为什么故障窗口越乱试越像被封了

## 1. 为什么还要单独写“冻结语义”

前面的章节已经讨论过：

- 恢复预算
- 治理时钟
- 故障窗口纪律
- 连续性成本

但还有一个更底层、也更容易被忽视的事实：

`Claude Code 在很多关键面上，并不是自由回滚和自由重试的系统。`

它在不断冻结一些先前事实：

- 已见过的 tool result 决策
- 已写入的 transcript 链头
- 已进入 cooldown 的速度状态
- 已被标记为局部失效的连接域

这意味着故障窗口里“多试几次”经常不是中性的。  
你不是在原地重试，  
而是在一个已经冻结了部分历史决策的系统里继续叠加新状态。

## 2. 先说结论

从源码看，Claude Code 很多关键恢复路径的本质是：

`优先保持单链一致性和模型可见前缀稳定，再在这个前提下做恢复。`

所以对用户来说，最重要的合规启示是：

1. 故障窗口里减少变量切换
2. 不要并发重放同一条会话链
3. 优先保留第一现场
4. 把“重试”理解为受预算、受链头、受冻结语义约束的动作

这也是为什么很多问题会越试越像“被封了”。

## 3. 源码为什么支持这个结论

### 3.1 transcript 远程追加是严格的单链顺序

`sessionIngress.ts` 有非常清楚的单链逻辑：

- 每个 session 用 sequential wrapper 串行追加
- 使用 `Last-Uuid` 维持链头
- 409 时采纳服务端链头再继续
- 401 直接失败，不继续浪费预算

这说明远程 transcript 不是“多条写入最后合并”的宽松模型，  
而是“单链 append”模型。

对用户而言，  
这意味着同一故障窗口里并发重放、切端续写、重复提交同一链头，  
会更容易把问题推进到冲突和耗尽重试预算。

### 3.2 本地 transcript 也在尽量避免 sidechain 污染主链

`sessionStorage.ts` 对 sidechain / main thread 的处理非常谨慎：

- sidechain local writes 可以单独存
- 但 remote persistence 仍共享同一 `Last-Uuid` 链
- 如果把 sidechain UUID 提前写进主链，会造成 409、悬挂 parentUuid、resume 断链

这说明系统非常明确地在维护一个原则：

`局部并发可以存在，但权威链必须保持单线可追踪。`

### 3.3 tool result 一旦被模型看到，决策就会被冻结

`toolResultStorage.ts` 是这章最关键的证据之一。

源码明确区分：

1. previously replaced
2. previously seen but unreplaced
3. never seen

而且一旦某个结果已经被模型看过：

- 之前没替换的，之后也不能随便再替换
- 之前替换过的，resume 后还要 byte-identical 地重新应用

原因不是形式主义，  
而是 prompt cache stability。

这说明系统把“模型已经见过的前缀”当成冻结事实。  
从支持角度看，这也是为什么第一现场证据特别值钱。

### 3.4 fast mode cooldown 也是一种冻结状态

`withRetry.ts` 与 `fastMode.ts` 表明：

- 短等待时，尽量保持 fast mode 不变，保护 prompt cache
- 长等待或未知等待时，进入 cooldown，显式记录 `resetAt` 和 `reason`

也就是说，  
系统并不是每次报错都回到一个中性起点。  
它会把“当前处于过载或限流冷却期”持久化成运行时事实。

这再次说明：

`失败窗口不是空白窗口，而是带状态的窗口。`

### 3.5 bridge / remote 会主动打碎死状态，而不是无限等

`replBridge.ts` 里对 heartbeat fatal 的注释非常说明问题：

- 如果继续抱着 stale transport/work state 不放
- poll loop 可能进入长 backoff
- 最后形成 10 分钟甚至更长的 dead window

所以它会主动 tear down transport 和 work state，  
让系统回到可快速重分发的状态。

这意味着平台自己也承认：

- 某些“继续撑着等恢复”的状态其实比直接打碎重建更糟

用户故障窗口里最不该做的，  
恰恰就是一边叠加新变量，一边幻想老状态会自动好起来。

### 3.6 resume consistency 说明“恢复后一致性”本身也是被监控的

`sessionStorage.ts` 的 `tengu_resume_consistency_delta` 说明：

- 系统会比较恢复后链条与原记录的差异
- 恢复前后不一致不是边缘体验问题
- 它是正式被监控的恢复质量指标

这进一步支持一个更深的结论：

`恢复不是把界面拉起来就算完成，而是要尽量回到同一条历史链。`

## 4. 从第一性原理看，系统到底在保护什么

这一整套设计保护的不是“多试几次的自由”，  
而是三样东西：

1. 会话链的单线可追踪性
2. 模型已见上下文的稳定性
3. 故障状态的可解释性

如果这三样被打碎，  
系统就会越来越像：

- 前后不一致
- 刚刚还能用现在不行
- 越恢复越乱

而用户体感上最容易把这种状态压缩成一句：

- 像被封了

## 5. 给用户的合规启示

### 5.1 第一现场优先于第十次重试

一旦出现：

- 409
- 恢复异常
- 前后状态不一致
- 局部组件突然进入 `needs-auth`

先保留第一现场的时间、错误原文、当前环境，  
比马上进行多轮重试更有价值。

### 5.2 不要并发重放同一条问题会话

如果同一 session 在多个终端、多个恢复动作里同时推进，  
你制造的往往不是更多机会，  
而是更多链头冲突。

### 5.3 区分“短等保稳”和“旧状态已经该打碎”

有些错误窗口适合短暂等待，  
例如明确的短 `retry-after`。  
但有些窗口里，旧状态本身已成问题，  
此时最值钱的是：

- 不再叠加变量
- 明确停止旧链
- 用更少动作做下一步重建

## 6. 面向中国和高波动环境用户的合规结论

这里仍然不讨论规避或对抗检测。  
更有效的做法是：

1. 在高波动环境里减少并发恢复
2. 不把多个故障动作压进同一个时间窗口
3. 一旦出现恢复异常，优先固定网络、设备、身份路径
4. 把 409、resume drift、长 cooldown 当成结构性信号，而不是普通卡顿

对高波动环境用户来说，  
最常见的损失不是“立刻被封”，  
而是会话链、连接链和恢复链被同时搅乱后，  
问题越来越难解释、越来越难自证。

## 7. 给平台构建者的启示

如果你在做自己的 agent 平台，  
这一章最值得迁移的是：

1. 把已见前缀当成冻结事实
2. 把 session append 做成严格单链
3. 对冲突采用链头采纳而不是无限分叉
4. 把 cooldown 做成显式状态
5. 对已死旧状态果断 teardown，不做长时间假等待

这些设计会让系统更“硬”，  
但也更容易恢复到可解释状态。

## 8. 用苏格拉底式追问再压一层

### 问：为什么不能把所有恢复都做成“再试一次”？

答：因为很多失败不是一次请求失败，  
而是链头、前缀和状态机已经发生变化。

### 问：为什么系统要冻结已见结果？

答：因为如果模型已见上下文可在之后随意改写，  
恢复就会变成另一场对话，而不是原会话的延续。

### 问：为什么用户越乱试越像被封了？

答：因为你不断在一个已经带状态的失败窗口里追加新状态，  
最终把局部故障扩展成跨层不一致。

## 9. 一句话总结

Claude Code 的许多恢复路径首先保护的是单链一致性和已见前缀稳定，而不是让你无限自由地重试；因此故障窗口里最有效的合规动作，不是多试，而是少变、少并发、先保留第一现场。
