# 身份、订阅、组织与策略限制

## 1. 先说结论

Claude Code 对账号可用性的第一层判断，不是“你有没有安装 CLI”，而是：

1. 你当前用的是哪种身份来源。
2. 这个身份是否允许进入 first-party 能力域。
3. 这个 token 是否属于正确组织。
4. 这个账号的订阅类型是否满足某项功能资格。

因此，“封禁”在客户端最常见的前置表现，不是 UI 上写着 ban，而是：

- token 失效
- org mismatch
- subscription ineligible
- feature/account not enabled
- managed settings 要求更强组织绑定

关键证据：

- `claude-code-source-code/src/utils/auth.ts:91-149`
- `claude-code-source-code/src/utils/auth.ts:153-205`
- `claude-code-source-code/src/services/policyLimits/index.ts:167-210`
- `claude-code-source-code/src/utils/auth.ts:1923-2000`

## 2. Claude Code 先区分“你是谁从哪里来的”

`auth.ts` 明确区分多种认证来源：

- `ANTHROPIC_AUTH_TOKEN`
- `CLAUDE_CODE_OAUTH_TOKEN`
- `CLAUDE_CODE_OAUTH_TOKEN_FILE_DESCRIPTOR`
- `apiKeyHelper`
- `claude.ai`
- `/login managed key`

这不是实现细节，而是控制逻辑的基础。不同来源对应不同的能力域、刷新逻辑和组织可信度。

证据：

- `claude-code-source-code/src/utils/auth.ts:153-205`

特别重要的是 `isManagedOAuthContext()`：

- CCR 和 Claude Desktop 里的 OAuth 不能随意退回用户本地 `~/.claude/settings.json` 的 API key。
- 否则托管会话可能错误地使用用户终端里的旧 key 或错误组织的 key。

证据：

- `claude-code-source-code/src/utils/auth.ts:83-96`

这说明账号风险控制从一开始就在避免“身份源串线”。

## 3. 订阅类型不是展示字段，而是能力准入字段

在 `policyLimits` 与 `remoteManagedSettings` 的资格判断里，OAuth 用户只有 `enterprise` / `team` 才被认为属于可接收组织级策略下发的人群。

证据：

- `claude-code-source-code/src/services/policyLimits/index.ts:190-210`
- `claude-code-source-code/src/services/remoteManagedSettings/syncCache.ts:35-45`
- `claude-code-source-code/src/services/remoteManagedSettings/syncCache.ts:87-111`

这意味着：

- 并非所有登录用户都处在同一治理平面里。
- 团队版/企业版用户会进入更强的“组织策略控制面”。
- 账号限制并不只是个人行为问题，还与组织级治理模型绑定。

## 4. token 过期只是表象，真正问题是“身份连续性”

`handleOAuth401Error()` 和 `checkAndRefreshOAuthTokenIfNeeded()` 展现出一套严格的身份连续性维护：

1. 401 先清缓存并异步重读 keychain。
2. 如果别的进程已经刷新过 token，则直接恢复。
3. 否则获取锁，避免多进程并发刷新。
4. 刷新失败则继续返回不可恢复状态。

证据：

- `claude-code-source-code/src/utils/auth.ts:1338-1391`
- `claude-code-source-code/src/utils/auth.ts:1427-1561`

这套逻辑说明客户端不把 401 简化为“网络抖动”，而把它当作身份边界破裂后的严肃事件。

## 5. 组织绑定是显式的 fail-closed 设计

`validateForceLoginOrg()` 是这套系统里最接近“组织级硬约束”的代码之一。

它的行为很清楚：

1. 如果 `policySettings.forceLoginOrgUUID` 存在，就必须验证当前 token 所属组织。
2. 即便本地缓存里记着 org UUID，也不信任，仍然去 profile endpoint 拉权威结果。
3. 如果 profile 拉取失败，就 fail closed。
4. 如果 token 属于错误组织，就给出明确错误并拒绝继续。

证据：

- `claude-code-source-code/src/utils/auth.ts:1916-2000`

这是一个非常强的信号：  
Claude Code 的组织治理边界不依赖本地配置自述，而依赖服务端权威身份信息。

## 6. 账号限制很多时候是“能力域不可达”

Bridge / Remote Control 帮我们看清了“账号限制”的另一种形态：

- `remote-control` 入口要求已有登录和订阅。
- 多 session remote control 如果没对该账号放开，会直接报 `not enabled for your account yet`。
- 注册远程环境时如果返回 404，会提示 `Remote Control environments are not available for your account.`。

证据：

- `claude-code-source-code/src/bridge/bridgeMain.ts:1941-1948`
- `claude-code-source-code/src/bridge/bridgeMain.ts:2055-2075`
- `claude-code-source-code/src/bridge/bridgeMain.ts:2451-2466`

这类限制非常像产品化风控里的“capability withdrawal”：

- 不是把整个客户端禁掉。
- 而是把高风险或高成本能力域从该账号上撤掉。

## 7. policy limits 是组织级封控的正式入口

`policyLimits` 直接说明：

- 它取的是 organization-level policy restrictions。
- API key 用户都可查询。
- OAuth 用户只有 Team / Enterprise 才进入这条路径。
- 本地用缓存与轮询承接这些限制。

证据：

- `claude-code-source-code/src/services/policyLimits/index.ts:1-13`
- `claude-code-source-code/src/services/policyLimits/index.ts:167-210`
- `claude-code-source-code/src/services/policyLimits/index.ts:556-645`

这里很关键的一点是：  
策略的决策来源在服务端，客户端只是读取、缓存并执行。

## 8. 一句话总结

Claude Code 的账号约束，本质上是身份源、订阅类型、组织归属和服务端能力域共同定义的准入问题；所谓“封号”在很多场景下就是这些准入条件被撤销或不再满足。
