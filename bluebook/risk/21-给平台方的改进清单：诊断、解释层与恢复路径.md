# 给平台方的改进清单：诊断、解释层与恢复路径

## 1. 为什么要写给平台方

如果研究只停留在“系统做得很强”，那其实还不够。  
真正成熟的研究还应该回答：

既然我们已经知道它的设计哲学和误伤痛点，平台方下一步最应该改什么？

## 2. 先说结论

从当前源码和文档结构反推，我认为平台方最值得优先补强的不是“更严的限制”，而是以下五类能力：

1. 更细的 reason code
2. 更强的诊断包导出
3. 更好的身份/组织/资格解释层
4. 更清晰的高安全远程会话状态可视化
5. 更低成本的误伤支持路径

## 3. 第一类改进：错误语义进一步产品化

当前系统已经能内部区分：

- 401
- 403
- insufficient_scope
- needs-auth
- not enabled
- policy denied
- rate limit

但普通用户并不总能稳定感知这些区别。

### 建议

- 为这些状态设计稳定、外显的 reason code
- 在 UI / CLI 层提供更一致的分类文案

### 价值

- 减少“所有失败都像封号”的误解
- 降低支持成本
- 提高用户信任

## 4. 第二类改进：支持导向的诊断包

源码已经有 `logForDiagnosticsNoPII` 这类机制，说明团队知道诊断很重要。  
但对于普通用户来说，诊断能力仍然不够产品化。

### 建议

提供一个显式的“支持诊断包”能力，至少包含：

- 错误语义
- 身份路径类型
- 组织/资格状态摘要
- 远程高安全链路状态
- 最近一次关键失败时间线

### 价值

- 把支持解释成本从用户脑力转移到系统结构化导出

## 5. 第三类改进：资格与处罚的清晰分离

从平台正义角度看，最危险的混淆是：

- entitlement 缺失
- 组织治理
- 本地动作阻断
- 真正处罚

都被用户体验成一种“被封”。

### 建议

产品层明确区分至少四种状态：

1. 账号/身份问题
2. 套餐/资格问题
3. 组织策略问题
4. 本地执行安全问题

### 价值

- 让用户知道下一步该去哪里，而不是只能继续试错

## 6. 第四类改进：高安全远程链路状态面可视化

Remote Control / bridge / trusted device 这部分实现已经很成熟，但普通用户看到的仍然过于黑盒。

### 建议

给用户一个最小可视状态面：

- 当前是否具备 remote-control entitlement
- 当前组织是否允许
- 当前设备信任状态是否成立
- 当前环境注册是否有效
- 当前 token / session 是否处于 reconnect / auth_failed / fatal

### 价值

- 高安全链路失败不会再被误读成“账号突然被针对”

## 7. 第五类改进：高波动环境用户的专门支持策略

从研究看，高波动环境并不一定意味着更高恶意，但确实更容易放大误伤体感。

### 建议

- 提供更明确的“连续性破坏”诊断提示
- 给远程能力单独的稳定性检查工具
- 给误伤用户更低成本的申诉入口

### 价值

- 降低特定环境用户承担的解释成本
- 提升平台正义感

## 8. 用苏格拉底式追问再压一下

### 问：为什么不是继续把风控做得更严？

答：因为很多痛点已经不主要来自“拦不住”，而来自“用户不知道自己为什么被拦、能去哪里恢复”。

### 问：为什么 reason code 和诊断包这么重要？

答：因为治理系统越复杂，解释层越不能省。  
没有解释层，复杂治理只会被用户体验成黑箱管制。

### 问：平台方最该追求的是什么？

答：不是单纯降低风险，也不是单纯提升可用性，而是：

让风险治理、误伤控制和支持恢复三者同时成立。

## 9. 一句话总结

Claude Code 的下一步成熟，不在于再叠一层限制，而在于把已有的强治理能力转化成更清晰的解释层、诊断层和恢复层。
