# 源码模块地图：风控相关代码入口与职责分层

## 1. 为什么要做源码地图

前面的章节已经把结论讲得很完整。  
但如果研究要继续深入，还需要一张“从问题回到源码”的地图：

- 遇到某类限制时，应该先看哪组模块
- 哪些模块在做身份、资格、组织治理、动作阻断或高安全会话
- 哪些模块更偏“观测与解释层”，哪些更偏“真正的 gating”

这一章就是为后续深入提供一张索引。

## 2. 先说结论

Claude Code 风控相关源码，大致可以按八组入口来理解：

1. 身份与组织
2. 资格与订阅
3. 组织策略与远程下发
4. 遥测与实验开关
5. 本地动作治理
6. 高安全远程会话
7. 连接认证与子系统恢复
8. 解释层、诊断层与支持层

这八组模块共同组成了我们前面反复提到的“分层治理系统”。

## 3. 模块总表

| 模块群 | 主要职责 | 用户可见表现 | 更接近哪一层 |
| --- | --- | --- | --- |
| `utils/auth` | token source、刷新、组织校验、订阅判定 | 401、重登、组织不匹配 | 主体层 |
| `services/oauth/*` | profile、订阅、rate limit tier 补全 | 登录后资格变化 | 主体/资格层 |
| `bridgeEnabled` | remote-control entitlement 与解释性错误文案 | requires subscription / full-scope / not enabled | 资格层 |
| `services/policyLimits` | 组织限制拉取、缓存、轮询 | policy denied | 组织层 |
| `services/remoteManagedSettings` | 远程治理配置下发与安全确认 | 配置变更、能力边界变化 | 组织层 |
| `utils/managedEnv` | policySettings.env 的运行时应用 | 远程配置影响本地行为 | 组织/执行层 |
| `services/analytics/growthbook` | entitlement gate、实验分桶、动态 gate | not enabled / feature bucket 变化 | 资格/观测层 |
| `utils/user` + `utils/telemetryAttributes` | 用户画像与 session/device/org 绑定 | 用户无感，但影响 gate 与追踪 | 观测层 |
| `permissions/*` | permission rules、classifier、denial tracking | risky action blocked | 动作层 |
| `entrypoints/sandboxTypes` | sandbox 能力边界 | sandbox unavailable / hard block | 动作层 |
| `bridge/*` | remote-control / bridge / trusted device / reconnect | auth_failed / fatal / env unavailable | 高安全会话层 |
| `remote/*` + `sessionIngress` | remote session、持久化、401 非重试语义 | remote session 异常 | 高安全会话层 |
| `services/mcp/client` + `services/mcp/auth` | needs-auth、step-up auth、subsystem auth | needs-auth / insufficient_scope | 连接认证层 |
| `services/rateLimitMessages` | usage / extra usage / overage 文案 | usage limit / out of extra usage | 计费层 |
| `diagLogs` + 相关 `logForDiagnosticsNoPII` 调用 | 无 PII 诊断日志 | 支持与调查能力增强 | 解释/支持层 |
| `hooks/notifs/useCanSwitchToExistingSubscription` | 引导用户切到已有权益路径 | /login to activate | 权益恢复层 |

## 4. 从研究问题回看源码入口

### 4.1 如果问题是“是不是账号被封了”

优先看：

- `utils/auth`
- `services/oauth/*`
- `bridgeEnabled`
- `services/policyLimits`

### 4.2 如果问题是“为什么高安全远程能力不能用”

优先看：

- `bridge/*`
- `trustedDevice`
- `bridgeEnabled`
- `sessionIngress`

### 4.3 如果问题是“为什么动作被挡住”

优先看：

- `permissions/*`
- `sandboxTypes`
- 相关 Tool 权限实现

### 4.4 如果问题是“为什么用户会觉得自己像被封了”

优先看：

- 错误语义层：`bridgeEnabled`、`bridgeApi`、`mcp/client`、`rateLimitMessages`
- 解释层：`diagLogs`、通知 hooks、支持提示

## 5. 哪些模块更像“守门人”，哪些更像“翻译器”

### 5.1 更像守门人

- `utils/auth`
- `services/policyLimits`
- `services/remoteManagedSettings`
- `permissions/*`
- `bridge/*`
- `services/mcp/auth`

这些模块决定“边界是否成立”。

### 5.2 更像翻译器

- `bridgeEnabled`
- `rateLimitMessages`
- `useCanSwitchToExistingSubscription`
- `diagLogs`
- 各类通知 hooks

这些模块决定“用户看到什么解释和恢复路径”。

这一区分很重要，因为研究如果只盯守门人模块，很容易忽略解释层在平台正义中的意义。

## 6. 从源码地图得出的一个关键结论

真正决定用户体验的，从来不是单个模块。  
用户体感往往来自“守门人模块的结果”加上“翻译器模块的解释方式”。

这也是为什么我们前面不断强调：

- 风控不仅是判定
- 也是解释
- 也是恢复
- 也是支持路径

## 7. 一句话总结

如果你要继续研究 Claude Code 风控，不要再问“哪个文件负责封号”，而要问“哪组模块在负责主体、资格、组织、动作、高安全会话和解释层的哪一环”。
