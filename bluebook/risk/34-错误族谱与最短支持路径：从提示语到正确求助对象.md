# 错误族谱与最短支持路径：从提示语到正确求助对象

## 1. 为什么要单独写“错误族谱”

前面的章节已经不断强调：

- 不是所有失败都叫封号
- 不是所有恢复都该从 `/login` 开始
- 不是所有问题都该找同一条支持路径

但如果没有一张错误族谱，用户和支持团队仍然会反复犯同一个错误：

把不同错误家族压缩成同一个“不能用了”。

这一章的目标就是把源码里的错误语义重新整理成：

1. 这类错误更接近什么治理层。
2. 最短恢复动作是什么。
3. 最该找谁，而不是最先慌什么。

## 2. 先说结论

从当前源码看，Claude Code 至少有七类彼此不同的错误家族：

1. 身份与认证家族
2. 组织与准入家族
3. entitlement/订阅激活家族
4. 子系统连接与 `needs-auth` 家族
5. 计费与额度家族
6. 高安全远程会话家族
7. 模型/部署/连接基础设施家族

真正高效的支持路径，不是“先试试重登”，  
而是先判断自己落在哪个家族，再走最短恢复动作。

## 3. 源码里已经存在的错误家族

### 3.1 身份与认证家族

`services/api/errors.ts` 已经拆出多类认证相关语义：

- `invalid_api_key`
- `token_revoked`
- `auth_error`

而且对应文案也并不完全相同：

- 无效外部 API key
- OAuth token revoked
- 通用 401/403 认证失败

这说明“认证失败”并不是一个统一原因。  
对用户来说，这个家族的共同点只是：

- 主体连续性当前不足以支撑请求

但具体恢复动作并不完全一样。

### 3.2 组织与准入家族

源码里至少两处很明确：

- `oauth_org_not_allowed`
- `validateForceLoginOrg()`

前者说明：

- 某个组织当前就不允许 OAuth authentication for Claude Code

后者说明：

- 某台机器被要求绑定某组织
- 无法证明组织归属时会 fail-closed

这个家族和普通 auth error 的区别在于：

- 重点不在“你有没有凭证”
- 而在“你的主体是否被允许进入这个组织边界”

### 3.3 entitlement/订阅激活家族

这类问题在体验层最容易像封号，但其实常常不是。

源码里有两个典型入口：

- `getBridgeDisabledReason()`
- `useCanSwitchToExistingSubscription()`

它们能拆出的情况包括：

- 你没有 claude.ai 订阅
- 你有订阅，但当前没激活到这个身份路径
- 你不是 full-scope token
- 你没有组织画像
- gate 没有命中

这整个家族的关键点是：

- 不是处罚
- 而是资格、画像、激活路径没有正确对齐

### 3.4 子系统连接与 `needs-auth` 家族

`mcp/client.ts` 和 `useManageMCPConnections.ts` 说明：

- 某个连接器进入 `needs-auth`
- 不等于整套账号或整套平台都坏了

系统甚至会：

- 缓存该状态
- 跳过必败重连
- 等你只在该域内做恢复

这意味着这个家族的支持归属应当是：

- 该连接域
- 而不是全局账号

### 3.5 计费与额度家族

`rateLimitMessages.ts` 把这块写得非常清楚：

- `session limit`
- `weekly limit`
- `Opus limit`
- `Sonnet limit`
- `out of extra usage`
- `resets ...`

这说明平台并没有试图把计费限制伪装成处罚。  
只是用户在高压场景下，仍然可能先把“不能继续”感受到为“被限制”。

但支持路径上，这个家族应直达：

- 套餐
- 额度
- overage
- reset time

### 3.6 高安全远程会话家族

`bridgeMain.ts` 里至少分出：

- `auth_failed`
- `fatal`
- `failed`

这说明远程高安全链路也不是统一失败。

典型区别是：

- `auth_failed` 更接近身份链可恢复问题
- `fatal` 更接近环境生命周期终结或不可恢复断裂

它们在支持归属上，显然更接近：

- bridge / Remote Control / trusted device / environment lifecycle

而不是普通账号支持。

### 3.7 模型/部署/基础设施家族

`errors.ts` 还明确拆出了：

- `bedrock_model_access`
- `model not available`
- `connection_error`
- `ssl_cert_error`
- `server_error`

这个家族更像：

- 部署不支持
- 模型未开放
- 网络/TLS/连接问题

它和“账号处罚”本来就不是一层。

## 4. 最短支持路径矩阵

| 错误家族 | 典型提示/分类 | 最短恢复动作 | 最该找谁 | 最不该做什么 |
| --- | --- | --- | --- | --- |
| 身份与认证 | `invalid_api_key` `token_revoked` `auth_error` | 先核对当前认证源，再做最小重认证 | 登录/账号支持 | 同时换多种 token 和账号 |
| 组织与准入 | `oauth_org_not_allowed` `forceLoginOrg` 失败 | 先确认组织要求与当前身份是否一致 | 管理员 / 组织 IT / 账号支持 | 把组织问题写成普通账号失效 |
| entitlement/订阅激活 | `not enabled` `/login to activate` | 先激活正确身份路径或核对 full-scope/画像 | 订阅/能力开放支持 | 误写成处罚、盲目换号 |
| 子系统连接 | `needs-auth` | 只恢复该 connector / server | 该连接域支持 | 全局重置整套账号 |
| 计费与额度 | `session limit` `weekly limit` `out of extra usage` | 先看 reset time、额度、overage | 计费/套餐支持 | 把它描述成封号 |
| 高安全远程会话 | `auth_failed` `fatal` | 先分可恢复认证问题和环境终结问题 | Remote Control / bridge 支持 | 把普通 CLI 也一起重置 |
| 模型/部署/连接 | `model not available` `bedrock_model_access` `ssl_cert_error` | 切模型、核对部署或网络/TLS | 部署管理员 / 网络支持 | 误判成风控处罚 |

## 5. 为什么中国和高波动环境用户更需要这张族谱

因为高波动环境更容易同时触发多种家族：

1. 认证问题
2. 高安全会话问题
3. 组织画像不连续
4. 子系统连接不稳定

如果没有错误族谱，  
这些问题会在体验层再次被压缩成统一的“又像被封了”。

于是用户最容易出现两种高成本动作：

- 误走错误支持路径
- 为了自救而制造更多变量

所以对这类用户来说，错误家族化并不是文档修辞，  
而是实际的止损工具。

## 6. 从第一性原理看，为什么错误族谱本身属于风控设计

如果一个平台不能把“不同失败”翻译成“不同恢复动作”，  
那它即使内部分类做得再细，也会在现实里重新塌缩成同一种高成本治理。

换句话说：

- 错误家族不是附属 UX
- 它本身就是风控控制平面的一部分

因为它决定：

- 用户会不会误判
- 支持会不会误分流
- 故障会不会被放大

## 7. 用苏格拉底式追问再压一层

### 问：统一错误入口会不会更简单？

答：入口可以统一，但恢复路径不能统一。

如果入口统一、恢复也统一，  
那复杂度只会被转嫁给用户和支持团队。

### 问：是不是只要分类足够细就够了？

答：还不够。

分类只是第一步。  
真正重要的是：

- 哪类错误对应哪类最短恢复路径
- 哪类错误归属哪条支持链

### 问：为什么这章也关系到平台正义？

答：因为谁更容易分清错误家族，谁就更容易避免误伤。  
如果平台不帮助用户分家族，理解能力更弱、环境更复杂的用户就会承担更高成本。

## 8. 给平台构建者的启示

1. 先做 error family，再做文案，不要反过来。
2. 每个错误家族都应绑定一条最短恢复路径。
3. 每个错误家族都应绑定一条支持归属路径。
4. 对高波动环境用户，最重要的不是更多提示，而是更少误分流。

## 9. 一句话总结

真正成熟的风控解释层，不是告诉用户“出了错”，而是尽快告诉用户“这是哪一类错、最该找谁、最不该做什么”。
