# 可证明性产品化：把 `auth status`、预检与证据包收敛成同一张用户仪表盘

## 1. 为什么要单独补这一章

前面的章节已经说明了很多“应该看什么”“应该怎么理解”。  
但如果这些知识只能靠读者自己拼出来，平台仍然把过多复杂度留给了用户。

换句话说，问题不只是：

- 平台内部有没有 reason family
- 平台内部有没有 preflight 零件

还在于：

- 这些零件有没有真正产品化

这一章研究的，就是如何把源码里已经存在的若干证明零件，  
收敛成一张真正对用户有用的仪表盘。

## 2. 先说结论

从当前源码看，Claude Code 已经拥有不少“可证明性产品化”的零件：

1. `auth status` 能输出当前登录方法、组织、订阅类型等核心事实。
2. `getBridgeDisabledReason()` 能把高安全能力的前提缺口拆成可执行诊断。
3. `useCanSwitchToExistingSubscription()` 能提示“你已有权益，但当前没激活到这个路径”。
4. `useMcpConnectivityStatus()` 能提示连接域的 `failed` 与 `needs-auth`。
5. `diagLogs.ts` 能为支持体系生成低敏感度调查线索。

这说明平台并不是缺少零件，  
而是还没有把这些零件合并成统一、低摩擦、用户可见的证明面。

## 3. 源码里已经存在的仪表盘零件

### 3.1 `auth status` 已经能回答“我现在到底是谁”

`cli/handlers/auth.ts` 的 `authStatus()` 已经能输出：

- `loggedIn`
- `authMethod`
- `apiProvider`
- `email`
- `orgId`
- `orgName`
- `subscriptionType`

这说明平台并不是完全没有“自证面”。  
只是这个自证面目前还偏命令式、偏诊断式。

从用户角度看，这类信息极其关键，因为它直接回答了：

- 当前到底是 claude.ai 登录、第三方、还是 API key
- 当前组织到底是谁
- 当前订阅到底有没有挂到这条路径上

### 3.2 Bridge preflight 已经部分产品化

`commands/bridge/bridge.tsx` 的 `checkBridgePrerequisites()` 很说明问题：

- 先看组织 policy
- 再看 `getBridgeDisabledReason()`
- 再看版本要求
- 最后看 access token 是否存在

这实际上就是一条完整的高安全能力 preflight。

也就是说，平台已经知道：

- 对 Remote Control 这种高价值能力
- 先做 prerequisites check 比用户撞上错误后再解释更好

### 3.3 启动通知已经在做“轻量证明提示”

`useCanSwitchToExistingSubscription()` 做的是：

- 检测你是否已有 Claude Pro/Max
- 但当前并未正确使用订阅身份
- 给出 `/login to activate`

`useMcpConnectivityStatus()` 做的是：

- 提示某些 MCP server/connector `failed`
- 或 `needs-auth`

这说明平台已经在产品层面尝试做“最少量但高价值”的提示。  
它并不是完全把一切都埋在内部日志里。

### 3.4 `diagLogs.ts` 是支持仪表盘的后端零件

`diagLogs.ts` 明确要求：

- 记录事件
- 但不允许带 PII

这说明平台已经在为支持体系构建一种“低敏感度证据面”。  
它不是直接给用户看的，但它决定了：

- 支持团队能不能快速确认用户到底落在哪类故障里

## 4. 为什么这些零件还不够

因为它们目前仍然分散在不同入口：

1. 有的在 CLI 命令里
2. 有的在 bridge 启动路径里
3. 有的在 startup notification
4. 有的在 supportside diagnostics

而用户真正需要的，往往不是四五个分散入口，  
而是一张统一回答下面问题的状态面：

1. 我当前身份路径是什么？
2. 我当前订阅/权益是否激活到这条路径？
3. 我当前组织画像是否完整？
4. 我当前高安全能力缺的是哪一个前提？
5. 我当前 connector/连接域是否需要单独 auth？
6. 我当前问题应走哪条支持路径？

## 5. 为什么中国和高波动环境用户尤其需要这张仪表盘

因为这类用户最容易承担额外的证明成本。

如果用户所处环境更容易发生：

- 网络切换
- 设备切换
- 身份路径切换
- 会话链断裂

那他就更需要一张能快速确认“现在到底是哪一层出问题”的状态面。

否则就会出现典型高成本场景：

- 明明只是订阅未激活，却开始怀疑封号
- 明明只是 connector `needs-auth`，却去全局重登
- 明明只是组织画像不完整，却开始切换多个账号和网络

所以对这类用户来说，统一仪表盘不只是方便，  
而是减少误伤、减少举证成本的基础设施。

## 6. 从第一性原理看，这为什么是平台正义的一部分

一个系统如果要求用户持续证明自己：

- 是谁
- 属于哪个组织
- 当前会话是否可信
- 当前能力是否已被激活

那么它就有责任提供一条低成本自证路径。

否则就会发生一件事：

- 平台内部掌握了大部分真相
- 用户却要自己从多个零散入口拼回这些真相

这不只是产品体验问题，  
也是证明责任分配问题。

## 7. 用苏格拉底式追问再压一层

### 问：用户真的需要这么多状态信息吗？

答：不需要全量状态，但需要足够支撑最短恢复路径的状态。

重点不是“把所有内部细节公开”，  
而是“给足够的信息让用户不再误走错误支持路径”。

### 问：会不会做成一个很复杂、没人看的诊断面？

答：会，如果做成内部工程视图。  
不会，如果只回答最关键的六个问题：

1. 我现在怎么登录的？
2. 我的组织和订阅是否对齐？
3. 我当前有没有高安全能力前提缺口？
4. 有没有单独 needs-auth 的连接域？
5. 当前是计费/认证/组织/连接哪一类问题？
6. 最短动作和支持归属是什么？

### 问：为什么说这会减少“像被封了”的体感？

答：因为统一仪表盘可以直接减少语义压缩税。

当用户不必再靠猜测去区分：

- entitlement
- auth
- org
- needs-auth
- rate limit
- fatal

“像被封了”的统一痛感就会显著下降。

## 8. 给平台构建者的启示

这一章最重要的设计启示有五条：

1. 把 `auth status`、资格 preflight、startup notification、support diagnostics 统一到同一张证明面。
2. 这张证明面不必暴露全部内部细节，但必须覆盖最短恢复路径所需事实。
3. 把高安全能力前提缺口显式化，而不是等失败后再让用户自己猜。
4. 把连接域问题和全局账号问题分开展示。
5. 把支持体系也视作仪表盘的一部分，而不是单独的人工补丁。

## 9. 一句话总结

真正成熟的平台，不只是在内部知道你为什么失败，还会把“你是谁、你缺什么、最该找谁”产品化成一张低摩擦、可执行的证明面。
