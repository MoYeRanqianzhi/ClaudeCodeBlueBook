# 局部撤权优于全局封号：能力撤回、连接降级与主体保全的治理哲学

## 1. 为什么还要单独写“局部撤权”

用户最容易把很多不同问题都压成一句：

- 我是不是被封了

但源码反复显示，Claude Code 真正做的经常不是“封整个人”，  
而是：

- 撤掉一条连接
- 暂停一个高风险能力
- 把某类连接打成 `needs-auth`
- 把当前窗口推入 cooldown
- 把某个组织外能力收口

如果不把这一层单独写出来，  
读者会继续把很多局部撤权误判成全局封禁。

## 2. 先说结论

从源码整体看，Claude Code 的治理哲学并不偏向“先把主体整体赶出去”。  
它更偏向：

`尽量保住主体身份，再按层撤回能力、连接、时间窗口或组织外权限。`

这样做的好处是：

1. 总伤害更小
2. 恢复路径更短
3. 解释更容易分层
4. 支持归属更清楚

所以更本质的判断不是：

- 你是不是被封了

而是：

- 当前到底是哪一层能力被撤回了

## 3. 源码里的直接证据

### 3.1 MCP `needs-auth` 是最典型的局部撤权

`mcp/client.ts` 在远程认证失败时，不会把主体整体打成失效。  
它会：

- 发事件 `tengu_mcp_server_needs_auth`
- 写入 `needs-auth` cache
- 返回该 server 的 `needs-auth` 连接状态

`toolExecution.ts` 也会在 MCP auth error 时只把对应 client 从 `connected` 改成 `needs-auth`。

这说明平台明确允许一种状态：

- 你还是你
- 但这个连接域现在不被允许继续工作

这就是典型的局部撤权，而不是全局封号。

### 3.2 `auth_failed` 与 `fatal` 被严格区分

`bridgeMain.ts` 和 `replBridge.ts` 反复区分：

- `auth_failed`
- `fatal`

其中：

- 401/403 更偏向 `auth_failed`
- 404/410 更偏向 `fatal`

这说明 even 在高安全远程链路里，  
系统也不把所有失败都当成同一种终局。

它在问的是：

- 这条链还能不能重新证明
- 还是这条链本身已经终止

这也是一种局部化、语义化的撤权，而不是粗暴整体封禁。

### 3.3 rate limit / extra usage / session limit 也是时间窗口撤权

`errors.ts` 与 `rateLimitMessages.ts` 明确把这些情况单独建模：

- session limit
- weekly limit
- out of extra usage
- reset time
- 429 capacity vs quota header presence

而且源码里甚至明确说明：

- 没有 quota headers 的 429，不应简单写成 quota limit
- 有些 429 可能是 infra capacity 或 entitlement rejection

这说明平台很努力避免一种糟糕压缩：

- 不能继续
- 就等于被处罚

实际上更多时候，  
它是在撤回某个时间窗口里的使用资格。

### 3.4 组织托管经常收回的是“谁能允许”，而不是“谁还是主体”

`mcp/config.ts` 里：

- `allowManagedMcpServersOnly` 可以让 allowlist 只剩 managed settings
- 但 denylist 仍允许用户保留自我保护权

`permissionsLoader.ts` 里：

- `allowManagedPermissionRulesOnly` 会把普通来源的 allow 面直接收窄

这说明组织治理经常收回的不是用户主体资格，  
而是：

- 用户自行扩张能力边界的权力

这同样是局部撤权。

### 3.5 “not enabled” 常常是资格未命中，不是处罚

`bridgeEnabled.ts` 的 `getBridgeDisabledReason()` 很典型：

- 没 claude.ai subscription
- 没 full-scope login token
- 组织画像不可判定
- gate 未命中

最终都可能落成：

- not enabled

如果只看表面，  
用户很容易把它理解成“平台把我封了”。  
但从结构上看，这通常只是：

- 高能力路径未放权

而不是主体被封。

## 4. 从第一性原理看，为什么成熟系统更喜欢局部撤权

因为局部撤权在很多情况下同时降低四类损失：

1. 降低平台安全损失
2. 降低用户误伤损失
3. 降低不可恢复损失
4. 降低支持解释损失

如果系统每次都直接封整个人，  
那会造成：

- 总伤害过大
- 恢复路径过长
- 问题分层被毁掉
- 支持归属混乱

局部撤权虽然让用户体验更碎片化，  
但从治理角度通常更优。

## 5. 为什么用户体感上却更像“被封了”

因为用户看到的是：

- 不能继续
- 不能连接
- 不能远程
- 不能高上下文

但用户往往看不到：

- 撤回的是哪一层
- 原因属于哪一族
- 是否有 reset time
- 是否只影响单个连接域

一旦这些语义被压缩，  
体感就会从“局部撤权”变成：

- 像被封了

这正是前面“语义压缩税”一章讲的现象。

## 6. 对用户的合规启示

### 6.1 先问“哪一层被撤了”，再问“是不是被封”

优先区分：

1. 主体链
2. 资格链
3. 会话链
4. 连接域
5. 时间窗口

只要这一步不做，  
后续动作很容易全错。

### 6.2 局部问题局部修，不做全局惊慌动作

如果只是：

- 某个 MCP server `needs-auth`
- 某个高安全链 `auth_failed`
- 当前额度窗口结束
- 某个能力 `not enabled`

那就先在对应层内修。  
不要一上来：

- 换号
- 换网
- 换设备
- 全局重置

### 6.3 中国和高波动环境用户尤其要避免“局部问题升级为全局问题”

高波动环境里最危险的不是单一错误，  
而是：

- 局部撤权
- 被错误重试
- 再被全局化
- 最后演化成真的跨层混乱

因此更要坚持：

1. 冻结变量
2. 先分层
3. 再动作

## 7. 对平台构建者的技术启示

这章最值得迁移的，不是某个 `needs-auth` 字段。  
而是下面这条治理原则：

`能局部撤，就不要全局封；能按层回收，就不要按主体整体清退。`

这条原则需要一整套配套：

1. 细 reason family
2. 细状态语义
3. 细恢复路径
4. 细归属方

没有这些，  
局部撤权只会让系统显得更乱。  
有了这些，它才会变成更成熟的治理系统。

## 8. 用苏格拉底式追问再压一层

### 问：为什么成熟系统不直接一刀切？

答：因为一刀切虽然简单，  
但总损害通常更大。

### 问：为什么用户却更容易误判局部撤权？

答：因为系统知道自己在分层，  
用户常常只看到结果表面。

### 问：平台为什么不把这些都解释得更明显？

答：有些地方已经在努力解释；  
但解释面仍然会被词汇、产品展示和支持路径进一步压缩。

## 9. 一句话总结

Claude Code 很多看似“像封号”的现象，本质上其实是能力、连接、时间窗口或自扩权力的局部撤回；理解这一点，是把问题从情绪判断拉回结构诊断的关键一步。
