# 资产保全与退出策略：账号风控窗口里真正该保护的不是面子而是工作连续性

## 1. 为什么还要单独写“资产保全”

很多用户一遇到“像被封了”的体验，  
第一反应是：

- 先证明自己没问题
- 先把权限弄回来

但如果从用户利益保护角度看，  
更应该先问的是：

- 当前最值钱的资产是什么？

在 Claude Code 这种系统里，真正的资产往往不是一时能不能继续跑，  
而是：

1. transcript
2. session metadata
3. worktree / 当前工作目录状态
4. 错误日志与证据面
5. 可恢复的会话链
6. 可复盘的录屏或输出记录

这一章就是把“利益不受损失”进一步落实成资产保全逻辑。

## 2. 先说结论

在风控、误伤、局部撤权或恢复异常窗口里，  
用户最应该优先保护的不是“立即继续运行的体面”，  
而是：

`工作连续性资产。`

这包括：

- 当前 transcript 是否仍可 resume
- 当前日志证据是否仍完整
- 当前录制/输出是否仍可回放
- 当前 worktree / metadata 是否仍被正确绑定到会话

如果这些资产保住了，  
权限恢复晚一点，损失仍可控。  
如果这些资产丢了，  
即便权限一时恢复，损失也可能已经不可逆。

## 3. 源码为什么支持这个结论

### 3.1 transcript 路径本身就是一项资产

`sessionStorage.ts` 里：

- `getTranscriptPath()`
- `getTranscriptPathForSession()`

都非常强调 session 与 project dir 的绑定关系。  
注释甚至直接指出：

- 如果 sessionId 和 sessionProjectDir 失配
- hook 会看到错误 transcript path
- resume / branch 也会漂移

这说明 transcript 文件不是普通缓存，  
而是恢复主链的关键资产。

### 3.2 metadata 会被重新附着到 transcript 尾部，防止恢复时丢失

`reAppendSessionMetadata()` 的设计很有代表性：

- title / tag / last prompt 等 metadata 会被重新附着到 transcript 尾部
- 还会吸收 SDK 写入的更新值
- 目的就是让 progressive loading / resume 仍能看到这些值

这说明平台明确承认：

- 元数据本身也是会话资产
- 如果不持续保尾部可见，它们就会在恢复时失效

### 3.3 resume 逻辑会显式接管 resumed session 的资产绑定

`sessionRestore.ts` 在 resume 时不只是读消息，  
还会：

- `switchSession(...)`
- `renameRecordingForSession()`
- `resetSessionFilePointer()`
- `restoreCostStateForSession(...)`
- `restoreSessionMetadata(...)`
- `restoreWorktreeForResume(...)`
- `adoptResumedSessionFile()`

这非常关键。  
它说明系统理解的“恢复”不是把对话内容读回来就结束，  
而是重新接管整套资产绑定关系。

### 3.4 录屏文件被视作可随 session 迁移的证据资产

`asciicast.ts` 里：

- 录制文件放在 project transcript 旁边
- `renameRecordingForSession()` 会在 `--resume/--continue` 后把文件重命名到当前 session ID

这说明终端录屏不是独立玩具，  
而被当作会话资产的一部分。

即使它现在受功能条件限制，  
设计意图也已经非常明确：

`录制应跟着会话资产一起走。`

### 3.5 错误日志与 MCP 日志是结构化资产

`errorLogSink.ts` 会把：

- timestamp
- cwd
- sessionId
- version

写进 JSONL 错误日志和 MCP 日志。

这意味着这些日志不是“顺便打出来的调试输出”，  
而是天然可供：

- 支持
- 复盘
- 自证

使用的证据资产。

### 3.6 恢复一致性本身已经被当成资产完整性检查

`tengu_resume_consistency_delta` 的存在说明：

- 恢复前后有没有漂移
- 本身就是被正式监控的完整性问题

这意味着平台自己也承认：

- 会话资产不是“在就行”
- 还必须“能一致地恢复”

## 4. 从第一性原理看，用户真正该保护什么

可以把用户资产分成四类：

### 4.1 生产资产

- 当前 worktree
- 当前项目目录状态
- 当前生成/编辑的工作成果

### 4.2 会话资产

- transcript
- session metadata
- content replacement records
- cost state

### 4.3 证据资产

- 错误日志
- MCP 日志
- diagnostics
- 首次失败时间与错误原文

### 4.4 复盘资产

- 录屏
- 终端输出
- resume 一致性线索

如果这四类资产都还在，  
你就还有很强的恢复和自证能力。

## 5. 用户在风控窗口里最容易做错的事

### 5.1 为了“马上恢复”而破坏会话资产

例如：

- 反复切 session
- 乱切 project root
- 重建新环境时不保留旧 transcript / 旧日志

### 5.2 为了“证明自己没问题”而覆盖证据资产

例如：

- 连续重试直到最初错误消失
- 不记首次失败时间
- 在多个终端反复跑不同恢复动作

### 5.3 为了“先继续工作”而打碎 worktree 关联

如果 resume / session file / worktreeSession 的绑定被打散，  
后续恢复难度会急剧上升。

## 6. 一个更合理的退出与止损顺序

当你怀疑自己正进入风控或误伤窗口时，  
更好的顺序是：

1. 停止继续叠加新变量
2. 保住 transcript、日志、录屏和当前 project dir
3. 记录首次失败时间与原始错误
4. 再决定是局部恢复、等待、还是升级求助

这和普通用户的直觉相反。  
但从资产保护角度看，这是更优顺序。

## 7. 对中国和高波动环境用户的特别结论

高波动环境用户更容易同时失去：

- 连接连续性
- 会话连续性
- 证据连续性

所以他们更应该把下面几件事当成长期纪律：

1. 让 transcript 和工作目录关系尽量稳定
2. 保留关键日志和第一现场
3. 出现异常时优先保护会话资产，而不是先做“多方案恢复实验”
4. 把“工作成果不丢”排在“马上继续跑”之前

## 8. 对平台构建者的技术启示

如果平台真的想保护用户利益，  
就不能只想着：

- 怎么更快挡住异常

还要继续强化：

1. transcript 与 project 绑定稳定性
2. session metadata 可恢复性
3. 证据面导出能力
4. 会话录制/复盘能力
5. 恢复一致性检测

因为这些东西决定的是：

- 用户在误伤后还能不能继续证明自己
- 还能不能把工作成果完整接回来

## 9. 用苏格拉底式追问再压一层

### 问：为什么“继续工作”不一定是最优先？

答：因为如果继续工作的代价是打碎恢复资产，  
那你只是把短期继续换成长期更大损失。

### 问：为什么 transcript 和日志也算“利益”？

答：因为它们决定你还能否：

- 恢复上下文
- 证明问题性质
- 缩短支持路径

### 问：什么叫真正保护自身利益？

答：不是单次成功继续，  
而是在异常窗口里最大化工作连续性、会话可恢复性和证据可复用性。

## 10. 一句话总结

在 Claude Code 的风控和误伤窗口里，用户真正该优先保护的不是一时能不能继续跑，而是 transcript、worktree、日志、录屏和 resume 绑定所构成的工作连续性资产；这些资产保住了，恢复与自证才有真正抓手。
