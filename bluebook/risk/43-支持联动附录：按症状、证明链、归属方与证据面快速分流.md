# 支持联动附录：按症状、证明链、归属方与证据面快速分流

## 1. 为什么要再写一份“联动附录”

前面已经有：

- 错误族谱
- 支持路径
- 共享词汇
- 证据包

但在真实故障里，用户、管理员、平台支持最常失败的地方，  
不是完全没有信息，  
而是三方不知道：

1. 这次到底是哪条证明链断了
2. 这次应该先找谁
3. 这次最有价值的证据到底是什么

这一章的目标，就是把这些内容压成一张更能执行的运行手册。

## 2. 先说结论

恢复最快的路径，往往不是“先做最多动作”，  
而是：

1. 先正确分层
2. 先正确归属
3. 先保住第一现场证据

从源码看，Claude Code 已经提供了不少证据面：

- 认证与会话状态
- `needs-auth` 这类局部状态
- 会话链 409/401 语义
- JSONL 错误日志与 MCP 日志
- 无 PII 诊断事件
- 可选终端录制
- resume consistency 之类的链路一致性信号

所以关键不是“有没有证据”，  
而是“有没有把证据送到正确的人手里”。

## 3. 先记住最小分流规则

出现“像被封了”的体验时，先问六个问题：

1. 这是主体链问题，还是资格链问题？
2. 这是全局问题，还是某个连接域的局部问题？
3. 这是组织策略，还是平台策略？
4. 这是会话链断裂，还是计费/额度窗口？
5. 这是普通能力，还是高安全远程能力？
6. 这是第一次出现，还是在已有失败窗口里继续放大？

如果这六个问题没有先回答，  
后续动作大概率会变成噪声。

## 4. 分流总表

| 用户症状 | 更可能断裂的链 | 更可能的错误家族 | 第一归属方 | 用户先做什么 | 最值钱的证据 |
| --- | --- | --- | --- | --- | --- |
| 明明登录了，但还反复要求 `/login` | 主体链 | `auth_error`、`token_revoked`、主体未闭合 | 平台认证支持 | 冻结登录路径，不混用多终端刷新 | 首次报错时间、最近一次成功时间、当前身份路径、错误原文 |
| 有订阅但提示未激活、功能没开 | 资格链 | entitlement 未命中、`/login to activate`、`not enabled` | 订阅/平台支持 | 先确认当前走的是哪条登录路径 | 订阅类型、当前 auth 路径、提示原文、是否曾成功用过同能力 |
| 组织环境里突然不认组织或能力消失 | 主体链 + 资格链 | `oauth_org_not_allowed`、`forceLoginOrg`、policy restriction | 组织管理员 + 平台支持 | 不切个人/组织账号来回试 | 组织归属、是否受托管、当前机器、管理员策略变更时间 |
| 某个 MCP/连接器突然失效，但其他功能正常 | 局部连接链 | `needs-auth`、connector unavailable | 该连接域支持 / 用户自身 | 只在该连接域内恢复，不做全局重置 | server 名、transport、最近一次成功连接时间、MCP 日志 |
| Remote Control / bridge 突然掉线、`auth_failed`、`fatal` | 会话链 + 高安全链 | `auth_failed`、`fatal`、高安全会话失效 | 平台远程/高安全支持 | 先区分是认证失效还是环境终局失效 | sessionId、错误原文、最近一次成功时间、设备/网络变化 |
| 刚刚还能用，后来持续出现 409/恢复异常/简历后不一致 | 会话链 | append conflict、resume drift、链头错位 | 平台会话/恢复支持 | 停止并发重放，保留当前会话现场 | sessionId、409/恢复语义、resume consistency 信号、当前 transcript 状态 |
| 提示额度、冷却、reset time | 额度链，不一定是风控 | `rate_limit`、`session limit`、`weekly limit` | 计费/额度支持 | 看 reset time，不要写成封号 | reset 时间、模型、是否 fast mode、套餐状态 |
| 原来本地允许的规则或 MCP 服务器突然失效 | 组织治理链 | 托管策略接管、managed-only allowlist | 组织管理员 | 先问策略是否收口，不先怪账号 | 是否启用 managed-only 策略、最近策略变更、受影响范围 |

## 5. 用户、管理员、平台支持分别该看什么

### 5.1 用户先看“我改了什么”

用户最该先回忆的，不是“平台是不是针对我”，  
而是：

1. 最近是否换过登录路径
2. 最近是否换过组织或账号
3. 最近是否换过设备、容器、远端环境
4. 最近是否切过网络出口
5. 最近是否只在某个连接域里出问题

如果这些变量在故障窗口里同时变化，  
支持成本会急剧上升。

### 5.2 管理员先看“策略是不是已经接管”

从 `settings.ts`、`permissionsLoader.ts`、`mcp/config.ts` 看，  
管理员最该先确认的不是“用户会不会操作”，  
而是：

1. 当前是否已有更高优先级 policy source 生效
2. 是否启用了 managed-only permission rules
3. 是否启用了 managed-only MCP allowlist
4. 是否最近下发过危险 managed settings 变更
5. 当前问题是策略收口，还是平台级认证/会话问题

管理员真正要避免的，是把平台认证问题错接成组织问题，  
或者把组织收口问题误写成平台处罚。

### 5.3 平台支持先看“证据链是否完整”

平台支持最值钱的不是一句“不能用”，  
而是能否拿到：

1. 错误家族
2. 会话或连接域标识
3. 最近成功时间
4. 失败开始时间
5. 失败窗口里发生过哪些变量变化

只要这些事实足够清楚，  
很多问题根本不需要走到“是否封禁”的叙事。

## 6. 证据面总表

| 证据面 | 源码启示 | 适合回答什么问题 | 备注 |
| --- | --- | --- | --- |
| 错误原文与首次时间 | `errors.ts`、`rateLimitMessages.ts`、`withRetry.ts` | 是 auth、policy、quota 还是局部连接问题 | 必须保留绝对时间 |
| sessionId / version / cwd 相关日志 | `errorLogSink.ts` | 这是不是同一会话、同一环境里的连续问题 | JSONL 日志更利于支持排查 |
| 无 PII 诊断事件 | `diagLogs.ts`、`sessionIngress.ts` | 是否发生 409 采纳链头、重试耗尽、坏 token | 更适合跨角色共享 |
| MCP server 日志 | `errorLogSink.ts`、`mcp/client.ts` | 这是某个连接域的局部认证问题吗 | 避免把局部问题写成全局 |
| resume consistency 信号 | `sessionStorage.ts` | 恢复后为什么前后不一致 | 适合会话/恢复问题 |
| 可选终端录制 | `asciicast.ts` | 故障是如何一步步出现的 | 不是默认总有，存在时很有价值 |
| transcript 追加链状态 | `sessionIngress.ts` | 是否发生 409 adopt / stale state / bad token | 对远程会话和恢复特别关键 |
| 已见结果冻结语义 | `toolResultStorage.ts` | 为什么第一现场证据比反复重试更值钱 | 已见内容后续会被冻结为决策边界 |

## 7. 为什么“第一现场证据”尤其重要

`sessionIngress.ts` 说明会话追加是单链顺序化的，  
409 时会采纳服务端链头继续重试。  
`toolResultStorage.ts` 又说明，一旦某段结果已经被模型看到，  
后续处理会尽量冻结先前决策，避免破坏前缀稳定性。

这两点合在一起，意味着：

1. 故障刚出现时的那一批状态最接近原始原因
2. 后续每一次重试、切换、恢复，都可能把现场改写成更难解释的状态

所以合规自保最重要的支持纪律之一，就是：

`先保存第一现场，再决定是否继续动作。`

## 8. 面向中国和高波动环境用户的合规操作建议

这里仍然只讨论合规路径。

对高波动环境用户，最值得执行的是：

1. 第一次失败后立即记下绝对时间
2. 不在同一故障窗口里连续切换网络、设备、账号、组织
3. 先判断是不是局部连接域问题
4. 优先保留错误原文、最近成功时间、当前环境变化
5. 向支持描述问题时直接使用共享词汇，而不是只说“被封了”

这不是为了迎合平台，  
而是为了减少你自己的解释成本和误分流成本。

## 9. 给管理员的一页运行手册

管理员收到报障时，建议按这个顺序问：

1. 当前故障是否只影响某个用户，还是一组用户
2. 当前故障是否只影响某类高能力路径
3. 最近是否下发过 remote managed settings / policy 变更
4. 当前是否启用了 managed-only permission 或 MCP allowlist
5. 当前机器是否存在 forceLoginOrg 或组织归属约束

如果这些问题有明确答案，  
管理员就能快速决定：

- 该自己处理
- 还是立刻转平台支持

## 10. 用苏格拉底式追问再压一层

### 问：为什么支持流程最怕用户做太多动作？

答：因为动作会改变证据面。  
系统并不是静态物体，而是会持续刷新、重试、采纳新链头、冻结旧决策的动态控制面。

### 问：为什么管理员不能只说“不是我的问题”？

答：因为很多问题的第一分流就决定了恢复时间。  
管理员虽然不一定是最终处理者，但往往是最快能识别“这是组织治理还是平台治理”的人。

### 问：为什么高波动环境用户更需要这张表？

答：因为他们更容易同时遭遇主体链、资格链、会话链的边界抖动；如果没有一张分流表，最容易被错误语义和错误动作放大伤害。

## 11. 一句话总结

“像被封了”的问题，最快恢复路径不是最多重试，而是先用证明链给问题分层、再把证据送到正确归属方，并尽可能保护第一现场证据不被后续噪声覆盖。
