# 结构化求助模板库：用户、管理员与平台支持的最短高质量文本

## 1. 为什么还要单独写“模板库”

前面已经有：

- 共享词汇表
- 支持联动附录
- 高波动环境 SOP

但在真实故障里，  
仍然有一个巨大损耗点：

- 用户知道该找谁
- 但不知道该怎么写

而“怎么写”这件事，  
实际上直接影响：

1. 是否被正确分流
2. 是否被快速理解
3. 是否需要来回补充信息
4. 是否会被误读成情绪化投诉

所以这里给出一份真正可复制的模板库。

## 2. 先说结论

高质量求助文本并不需要很长。  
它只需要稳定地包含五类信息：

1. 时间
2. 错误原文
3. 当前层级判断
4. 环境变化
5. 期望动作

如果这五类信息齐全，  
支持效率会比“我被封了怎么办”高很多。

## 3. 用户给平台支持的最短模板

### 3.1 通用模板

```text
问题开始时间：
最近一次成功时间：
当前错误原文：
我当前判断更像：
当前影响范围：全局 / 某个连接域 / 远程高安全链
故障窗口内是否有变更：设备 / 网络 / 身份路径 / 组织（是/否，若有请列出）
我已做的最小动作：
我希望得到的帮助：
```

### 3.2 `auth_error` / 登录态异常模板

```text
我在 {绝对时间} 首次遇到认证异常。
最近一次成功使用时间为 {绝对时间}。
当前错误原文为：{原文}
当前 auth status 关键信息：authMethod={...}, orgId={...}, subscriptionType={...}
我当前判断更像主体链/认证异常，而不是额度问题。
故障窗口内我没有/有进行以下变更：{设备、网络、身份路径、组织}
请协助判断这是认证连续性问题、组织约束问题，还是需要进一步重新登录/支持处理。
```

### 3.3 `needs-auth` / connector 局部失效模板

```text
在 {绝对时间} 起，{connector/server 名} 出现 needs-auth / unavailable。
其他功能 {正常/异常}。
最近一次成功连接该 connector 的时间为 {绝对时间}。
当前错误原文为：{原文}
我判断这是局部连接域问题，而不是全局账号问题。
请协助确认该连接域下一步应由我本地重认证、管理员处理，还是平台侧排查。
```

### 3.4 quota / reset time 模板

```text
在 {绝对时间} 起我无法继续使用。
当前提示原文为：{原文}
当前可见 reset time / limit 信息为：{原文中的时间或额度}
我判断这更像 session limit / weekly limit / extra usage / 429 capacity，而不是账号处罚。
请协助确认这是额度窗口问题还是其他限制。
```

## 4. 用户给管理员的最短模板

### 4.1 组织策略疑问模板

```text
我在 {绝对时间} 起遇到能力变化。
当前错误或提示原文：{原文}
我当前判断更像组织策略 / policy 接管，而不是全局账号异常。
当前受影响能力：{Remote Control / MCP / 某类功能}
请帮我确认近期是否有 managed settings / policy limit / allowlist 相关变更。
```

### 4.2 channels / plugin / allowlist 模板

```text
我在 {绝对时间} 尝试使用 {plugin/channel/server} 时被阻止。
当前提示为：{原文}
我判断可能与 org allowlist / channelsEnabled / plugin-only 限制有关。
请帮我确认这是否属于组织层允许列表或主权收口问题。
```

## 5. 管理员转平台支持的最短模板

管理员最有价值的不是“转发用户抱怨”，  
而是补足组织层事实。

```text
用户在 {绝对时间} 起出现 {问题类型}。
用户原始错误：{原文}
我们已确认以下组织层事实：
- 是否 team / enterprise
- 是否存在相关 policySettings
- 是否近期下发过 managed settings / allowlist / channelsEnabled 变更
- 当前问题更像组织策略、平台认证还是局部连接域问题
请平台侧协助判断剩余平台层原因。
```

## 6. 平台支持回用户的理想模板

如果平台要进一步降低误伤成本，  
理想回复不应只是：

- 请重新登录

而应至少包含：

1. 当前判断属于哪一类问题
2. 当前是否更像局部域问题还是全局问题
3. 当前建议的最小动作
4. 当前不建议做什么

例如：

```text
从当前信息看，这更像 {认证连续性 / 组织限制 / connector needs-auth / 配额窗口}。
当前影响范围更像 {局部/全局}。
建议你先执行的最小动作是：{动作}
在此之前请不要：{动作}
如果执行后仍存在，请带上 {证据项} 回复。
```

## 7. 为什么这些模板有效

因为它们强制把问题压回到结构层：

1. 时间轴
2. 证明链
3. 影响范围
4. 环境变化
5. 责任归属

而这五件事，  
恰好就是 Claude Code 这套控制平面最关心的事实。

## 8. 对中国和高波动环境用户的特别建议

高波动环境用户更应避免以下写法：

```text
今天突然全都不行了，是不是把我封了？
```

更好的写法是：

```text
我在 {时间} 首次遇到 {错误原文}。
最近一次成功时间是 {时间}。
当前只有 {某连接域/高安全远程/额度窗口/主体链} 受影响。
故障窗口内我 {没有/有} 更换 {设备、网络、身份路径、组织}。
请协助确认当前最短恢复路径。
```

这不是为了“说得更官方”，  
而是为了减少你自己的解释损失。

## 9. 用苏格拉底式追问再压一层

### 问：为什么模板比自由描述更好？

答：因为自由描述更容易遗漏最关键的结构事实。

### 问：为什么支持最需要的是时间和范围？

答：因为这两者直接决定它该走哪条诊断链和恢复链。

### 问：为什么高质量求助文本也属于风控研究的一部分？

答：因为误伤损失并不只发生在判定时，  
也发生在求助和恢复过程中。

## 10. 一句话总结

高质量求助文本不是礼貌问题，而是治理效率问题；只要把时间、错误原文、层级判断、环境变化和期望动作写清楚，很多“像被封了”的问题都会更快被拉回正确层级和正确支持路径。
