# 治理主权与恢复主动权：谁能关、谁能开、谁能替你说 yes

## 1. 为什么还要单独写“治理主权”

做到这个阶段，已经能看出 Claude Code 不是简单的用户端工具。  
它更像是一个多角色共同作用的治理系统。

但如果不把“谁拥有最后决定权”单独抽出来，  
读者仍然容易混淆几件事：

- 谁能关闭某项能力
- 谁能重新开启某项能力
- 谁能替用户批准高风险动作
- 谁只能提供线索，不能提供裁决

这章的目标，就是把这些主权边界压成一张清晰地图。

## 2. 先说结论

从源码看，Claude Code 的治理主权大致被分成五层：

1. 平台主权
2. 组织主权
3. 用户主权
4. 替代批准面主权
5. 自动恢复主权

而且这五层并不对等。  
真正值得记住的结论是：

`不是每个能发出信号的人，都拥有裁决权；不是每个能触发动作的人，都拥有最后仲裁权。`

这也正是风控哲学的核心之一：

`谁有资格发言，谁有资格批准，谁有资格最终裁决，必须分开。`

## 3. 第一层：平台主权

平台主权主要体现在：

- feature gates
- service-side eligibility
- API auth / quota / bridge / session ingress 语义

从 `bridgeEnabled.ts`、`withRetry.ts`、`sessionIngress.ts` 到前面的各章都能看到：

- 某些能力是否存在
- 某些 quota 是否允许
- 某些高安全链路是否仍可续

最终都要回到平台端权威真相。

所以平台主权最核心的一点是：

`平台决定哪些证明链最终足够，客户端更多是在组装和表达这些证明。`

## 4. 第二层：组织主权

组织主权在源码中非常强，而且往往是“先来源、后内容”。

### 4.1 policySettings 的 first source wins

`settings.ts` 对 `policySettings` 的处理不是“都合并”，  
而是：

1. remote
2. MDM
3. managed file
4. HKCU

先命中哪个高优先级来源，  
就由哪个来源主导这一面。

这说明组织主权不是建议层，  
而是真正的上位裁决层。

### 4.2 strictPluginOnlyCustomization 进一步限制谁能发言

`pluginOnlyPolicy.ts` 表明：

- 某些 customization surface 可以被锁到 plugin-only
- admin-trusted source 才可继续生效

这说明组织主权甚至不只在“决定内容”，  
而在“决定哪些来源根本还有资格发声”。

### 4.3 channels 也必须先经过组织同意

`channelNotification.ts` 说明：

- team / enterprise org 若未显式 `channelsEnabled: true`
- channel 能力直接不开放

这意味着即使平台支持、用户也想用，  
组织仍然可以作为更上层主权阻止它成为正式治理面。

## 5. 第三层：用户主权

用户当然不是没有主权。  
但用户主权在源码里主要体现为：

1. 局部 deny 权
2. 会话级 opt-in / opt-out
3. 明示批准权
4. 故障窗口里的恢复主动权

### 5.1 用户仍保留自我保护型否决权

`mcp/config.ts` 明确说明：

- 即便 `allowManagedMcpServersOnly` 打开
- denylist 仍会从所有来源合并

这说明组织可以垄断“谁能被允许”，  
但用户仍保留“我自己不要连这个”的主权。

这是一种非常成熟的非对称设计。

### 5.2 用户也拥有会话级选择权，但不是无限权

例如 `--channels` 是会话级 opt-in，  
但它仍需：

- 平台总开关
- 组织允许
- allowlist 通过

所以用户主权存在，  
但它不是无限主权。

更准确地说：

`用户拥有在被允许边界内做明确选择的主权，而不是重写上位治理边界的主权。`

## 6. 第四层：替代批准面主权

这一层最容易被误解。

### 6.1 替代批准面能执行“批准”，但不天然拥有批准主权

`channelPermissions.ts` 和 `channelAllowlist.ts` 已经说明：

- channel relay 可以承载批准信号
- 但必须先通过 capability + gate + allowlist + session opt-in

所以这些面更像：

- 被授予批准代行能力的面

而不是：

- 自然拥有批准主权的面

### 6.2 hook / classifier / config 也是同类问题

`permissionLogging.ts` 把这些来源拆开记录，  
本质上就是在说：

- 这些来源都可能影响“是否继续”
- 但它们的合法性和解释性并不相同

这再次说明，批准面和主权面不是同一个东西。

## 7. 第五层：自动恢复主权

系统还有一类很特别的主权：

`自动恢复主权`

它不是谁“批准”了什么，  
而是谁能在不等待用户操作的情况下，决定下一步如何恢复。

这一层体现在：

- OAuth 401 自动再确认 / refresh
- bridge re-dispatch
- cooldown 到期自动恢复
- session 409 adopt server UUID

自动恢复主权的特点是：

- 它可以主动做恢复动作
- 但通常不能越权放开新的治理边界

换句话说：

`系统可以自动帮你恢复连续性，但不能自动替你重写主权边界。`

## 8. 从第一性原理看，这张主权图为什么重要

如果没有这张图，  
用户会很容易出现三种误判：

1. 把“能发出批准信号”误当成“拥有最终批准权”
2. 把“能触发恢复”误当成“能恢复一切”
3. 把“本地能改”误当成“最终真相也会跟着改”

而 Claude Code 当前源码恰恰是在不断防止这些误判。

## 9. 对用户的合规启示

### 9.1 先分清谁有最后决定权

当你遇到问题时，  
先问：

1. 这是平台级问题吗
2. 这是组织级问题吗
3. 这是我自己当前会话级可修问题吗

这比马上做动作更重要。

### 9.2 不要把替代批准面误当成全能绕路

替代批准面只是被治理后允许代行某一类批准动作。  
它不是脱离平台和组织主权的万能旁门。

### 9.3 保护自己的主权，主要靠拒绝权和证据权

对普通用户来说，最强的稳定主权通常不是“强行放开”，  
而是：

- 明示拒绝
- 固定主路径
- 保护证据
- 正确升级求助

## 10. 对平台构建者的技术启示

真正成熟的治理系统，  
必须把下面几件事分开建模：

1. 谁能发信号
2. 谁能提出建议
3. 谁能代行批准
4. 谁能做最终裁决
5. 谁能自动恢复

如果这五件事被混成一个“有权限/没权限”开关，  
系统迟早会失去可解释性。

## 11. 用苏格拉底式追问再压一层

### 问：为什么用户主权不能无限扩张？

答：因为系统还要同时满足组织主权、平台主权和高风险边界。

### 问：为什么自动恢复不等于自动放权？

答：因为恢复连续性和授予新能力是两件不同的事。

### 问：为什么替代批准面必须被强治理？

答：因为批准权本身就是高价值能力。  
能代你说 yes 的面，必须先被系统确认“有资格说 yes”。

## 12. 一句话总结

Claude Code 的深层治理不是“谁能不能做”，而是“谁能发声、谁能代批、谁能裁决、谁能恢复”的主权分层；看清这张主权图，很多看似混乱的权限和风控设计就会重新变得清晰。
