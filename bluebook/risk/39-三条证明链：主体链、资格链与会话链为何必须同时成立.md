# 三条证明链：主体链、资格链与会话链为何必须同时成立

## 1. 为什么还要单独写“证明链总论”

前面的章节已经分别讨论了：

- 身份与组织
- entitlement 与策略
- 会话失效与高安全链路
- 误伤与解释成本

但如果不把这些内容收束到一个更底层的框架里，  
读者仍然容易把它们看成很多平行主题。

从源码实际结构看，这些主题并不平行。  
它们更像三条必须同时成立的证明链：

1. 主体链
2. 资格链
3. 会话链

只要其中任意一条不能稳定成立，用户就更容易感受到“像被封了”。

## 2. 先说结论

Claude Code 当前风控设计的本质，不只是“判断你能不能用”，  
而是持续证明三件事：

1. 你是不是同一个可识别主体。
2. 这个主体现在是否仍有对应资格。
3. 当前会话是否仍然是该主体的可信连续延伸。

我把这三层分别称为：

- 主体链
- 资格链
- 会话链

这里的说法是对源码结构的抽象，不是源码里的原始命名。  
但它能更清楚地解释很多看似分散的机制为什么会同时出现。

## 3. 第一条：主体链

主体链回答的问题是：

`你现在到底是谁？`

### 3.1 它至少由三部分构成

从源码看，主体链至少需要以下事实能同时成立：

1. 当前凭证是否仍有效
2. 当前登录路径是否仍是你以为的那条路径
3. 当前账号画像是否仍能被映射到正确账户与组织

`auth.ts`、`oauth/client.ts`、`auth status` 都在服务于这条链。

### 3.2 为什么主体链不是“有 token 就行”

`isClaudeAISubscriber()`、`hasProfileScope()`、`getOauthAccountInfo()` 已经说明：

- token 存在
- 不等于 profile scope 存在
- 不等于账号画像完整
- 更不等于组织归属已被权威确认

因此主体链的最低要求不是“手里有凭证”，  
而是“凭证、来源、画像三者能共同指向同一个主体”。

### 3.3 主体链断裂时的典型体感

这类问题用户最常感受到的是：

- 明明登录了
- 但为什么还让我 `/login`
- 为什么这里能识别我，那里又像不认我

这不是单纯的认证失败，  
而是主体链未能稳定闭合。

## 4. 第二条：资格链

资格链回答的问题是：

`即使你还是你，你现在有没有这项能力？`

### 4.1 资格链至少包含四种来源

从源码看，资格链并不是单一 gate，而是多源叠加：

1. 订阅类型
2. 组织画像
3. GrowthBook / feature gate
4. 组织 policy 与 remote managed settings

这解释了为什么“不是封号”的很多情况会仍然被体验成限制：

- 主体链可能没问题
- 但资格链没有闭合

### 4.2 为什么资格链经常被误判成处罚

因为从体验层看，  
资格链失败往往表现为：

- `not enabled`
- policy denied
- requires subscription

而用户自己往往看不到资格链内部到底是哪一段断了。

### 4.3 资格链断裂时最常见的误动作

- 把 entitlement 缺失误判成封号
- 把组织限制误判成账号异常
- 把订阅未激活误判成能力被撤销

这就是为什么资格链需要被单独抽出来理解。

## 5. 第三条：会话链

会话链回答的问题是：

`当前运行中的这条链路，是否还是同一个主体的可信延续？`

### 5.1 会话链和主体链不是同一回事

用户常见误区是：

- 只要主体恢复了，会话就一定恢复

但 `sessionIngress.ts`、`bridgeMain.ts`、`remoteBridgeCore.ts` 都说明：

- 账号恢复
- 不等于会话连续性已经恢复

会话链有自己的事实面：

- `Last-Uuid`
- `401`
- `409 adopt server UUID`
- `auth_failed`
- `fatal`

### 5.2 会话链为什么这么重要

因为 Claude Code 并不是纯 stateless 请求系统。  
它有：

- transcript 持久化
- remote session
- bridge worker
- ingress token
- heartbeat

在这种系统里，  
风控和封控就不可能只在“你是谁”这一层完成。  
它还必须在“这条会话链是否还能被持续证明”这一层完成。

### 5.3 会话链断裂的典型体感

用户最常见的主观描述是：

- 刚才还能用
- 我明明重新登录了
- 为什么远程能力还是像被踢下来了

这不是主体链还没恢复，  
而经常是会话链仍未重新闭合。

## 6. 为什么这三条链必须同时成立

可以把它们理解成三个门：

1. 主体链：你是不是这个人
2. 资格链：这个人有没有这项能力
3. 会话链：当前这条运行中的链路还能不能继续算作这个人的可信延伸

只要缺一条，就会出现看起来很像封禁的阻断体验。

这也是为什么我们前面不断强调：

- 不要把所有问题都写成单点封号

因为很多时候，  
根本不是“封不封”的问题，  
而是三条证明链里有一条没闭合。

## 7. 中国和高波动环境用户为什么更容易同时丢两条链

高波动环境最糟糕的地方，不只是更容易失败，  
而是更容易让两条甚至三条链一起失稳。

例如：

1. 网络波动让主体链和会话链同时不稳
2. 组织画像不完整让主体链和资格链同时不稳
3. 高安全远程路径失败让资格链和会话链一起失稳

这时用户就会明显感受到：

- 不是一个小问题
- 而是整套系统都像在卡自己

从结构上看，这就是多链同时断裂的体感。

## 8. 从第一性原理看，为什么三条链比“一个封号开关”更本质

如果你在做一个真实组织环境中的 agent 平台，  
你最终必须回答的不是：

- 有没有一个 ban switch

而是：

1. 主体如何被证明
2. 资格如何被证明
3. 会话如何被持续证明

这三件事都必须成立，  
系统才敢继续放行高能力行为。

所以更本质的抽象，不是“风控是封禁系统”，  
而是“风控是持续证明系统”。

## 9. 用苏格拉底式追问再压一层

### 问：为什么不把资格链和主体链合并？

答：因为“你还是你”并不等于“你现在仍有资格”。

如果把两者混成一层，  
entitlement 缺失就会不断被误写成账号异常。

### 问：为什么不把会话链当成主体链的一部分？

答：因为会话有自己独立的连续性和时序事实。  
账号恢复并不自动意味着会话恢复。

### 问：用户最该学会的不是更多 gate，而是更多证明链吗？

答：是。  
因为只要先知道自己是哪一条链断了，  
很多“像被封了”的体感都会立刻被拆开。

## 10. 给用户的合规启示

遇到限制时，最先问自己三个问题：

1. 我现在的主体链是不是稳定的？
2. 我现在的资格链是不是闭合的？
3. 我当前这条会话链是不是还连续可信？

这三个问题一旦先问出来，  
你就不会再把所有异常都写成同一种问题。

## 11. 给平台构建者的启示

1. 显式区分主体链、资格链、会话链。
2. 不要用同一类提示词覆盖三条不同证明链的失败。
3. 支持和恢复路径也应按三条链分流。

## 12. 一句话总结

Claude Code 风控真正保护的不是一个抽象账号，而是主体链、资格链和会话链三条证明链的同时成立；很多“像被封了”的体验，本质上是其中一条或多条链没有闭合。
