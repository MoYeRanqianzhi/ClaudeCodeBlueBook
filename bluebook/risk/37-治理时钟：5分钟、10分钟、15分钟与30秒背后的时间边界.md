# 治理时钟：5分钟、10分钟、15分钟与30秒背后的时间边界

## 1. 为什么要单独讨论“治理时钟”

很多人把风控理解成空间边界：

- 你能不能用
- 你有没有权限
- 你是不是被限制

但从源码看，Claude Code 的治理并不只依赖空间边界。  
它还深度依赖一组时间边界：

- 多久前刚登录
- token 还剩多久过期
- 某条失败状态缓存多久
- 心跳多久续租一次
- 多久之后才值得再试

也就是说，这套系统不只在问“你是谁”，  
还在问“你是在什么时间状态下成为这个主体的”。

## 2. 先说结论

Claude Code 当前源码已经暴露出一组很清晰的治理时钟：

1. `5` 分钟：预刷新和部分 preflight 的缓冲时间
2. `10` 分钟：trusted device enrollment 的 fresh login 窗口
3. `15` 分钟：MCP `needs-auth` 缓存窗口
4. `20/30` 秒：heartbeat / keepalive 级时间粒度
5. `3` 次：桥接失败熔断与 dead token 记忆阈值
6. `6` 小时：persistent retry / reset cap 上限

这说明平台并不是“静态检查一次就完”。  
它在用不同时间尺度治理不同风险对象。

## 3. 5 分钟：预刷新缓冲

### 3.1 MCP token 在到期前 5 分钟进入主动刷新窗口

`mcp/auth.ts` 明确写到：

- token 若在 5 分钟内到期
- 并且有 refresh token
- 则主动刷新

这说明平台不满足于“等 401 再修”。  
它更倾向于：

- 只要能提前避免失败，就提前避免

这是一种很成熟的时间边界设计：

- 把潜在失败从请求时刻，前移到更平滑的维护窗口

### 3.2 env-less bridge 也把 JWT 刷新缓冲做成 300_000 ms

`envLessBridgeConfig.ts` 里默认：

- `token_refresh_buffer_ms = 300_000`

也就是 5 分钟。

这说明 5 分钟不是某个局部魔法数字，  
而是一条跨模块复用的时间哲学：

- 不要等高安全链路真的断了再恢复

## 4. 10 分钟：fresh login 窗口

### 4.1 trusted device enrollment 依赖 fresh session

`trustedDevice.ts` 里非常关键的一句是：

- server gates enrollment on `account_session.created_at < 10min`

也就是说，trusted device 并不是随时都能优雅补上的。  
它依赖：

- 刚完成登录不久
- 会话还足够新鲜

这意味着对高安全链路而言：

- “你现在是谁”还不够
- “你多久前以这个身份登录的”也很重要

### 4.2 这对用户的直接启示

如果你是在：

- 登录很久以后
- 切过网络以后
- 切过设备以后
- 高压故障窗口里

才第一次进入高安全链路，  
那么你更容易遇到不是处罚、但很像被挡住的体验。

## 5. 15 分钟：短期必败记忆窗口

### 5.1 MCP `needs-auth` 会缓存 15 分钟

`mcp/client.ts` 里有：

- `MCP_AUTH_CACHE_TTL_MS = 15 * 60 * 1000`

这意味着系统明确承认：

- 某些路径不是“马上重试就可能好”
- 而是“在这段时间里高度可能继续失败”

这类时间边界的作用，不只是节流。  
它还在保护：

- 用户不再无意义地反复试
- 平台不再反复制造噪声

### 5.2 对 claude.ai connector 尤其重要

源码还特别提到：

- stale token mass-401 可能会把所有 claude.ai connector 一起拖进这 15 分钟缓存

这说明短期记忆窗口既是恢复策略，  
也是“错误风暴隔离层”。

## 6. 20 秒与 30 秒：心跳级治理时间

### 6.1 CCRClient 默认 20 秒 heartbeat，对应 60 秒 TTL

`ccrClient.ts` 和 `envLessBridgeConfig.ts` 都说明：

- 默认 heartbeat 间隔大约 20 秒
- 服务器 TTL 是 60 秒

这是一种典型的 3× 安全余量设计。

它体现的不是业务逻辑，而是：

- 会话连续性需要被周期性再证明

### 6.2 unattended retry 的 keepalive 用 30 秒颗粒切片

`withRetry.ts` 里：

- `HEARTBEAT_INTERVAL_MS = 30_000`

持久重试不会直接长睡过去，  
而是每 30 秒切片一次，以避免 host 误把会话判成 idle。

这说明时间结构不仅服务于认证，  
也服务于宿主对“活性”的判断。

## 7. 3 次：熔断阈值

### 7.1 bridge dead token 会在 3 次后进入跨进程熔断

`initReplBridge.ts` 和 `useReplBridge.tsx` 都体现了这个逻辑：

- 连续失败达到 3 次
- 认为这条路径已经足够证明为“当前会继续坏”

这条规则非常重要，因为它告诉我们：

- 平台并不追求无限恢复
- 平台允许“达到阈值后先停手”

### 7.2 这对用户的实际意义

如果系统都已经开始熔断，  
用户自己继续高频切换变量、继续暴力重试，  
只会让恢复更远而不会更近。

## 8. 6 小时：长等待上限

### 8.1 persistent retry 有 6 小时 reset cap

`withRetry.ts` 里：

- `PERSISTENT_RESET_CAP_MS = 6 * 60 * 60 * 1000`

即便在 unattended/persistent retry 模式里，  
系统也不会无上限等待某些 `Retry-After`。

这说明它承认：

- 有些恢复值得长期等待
- 但长期等待也需要上界

这是另一种时间边界的理性：

- 给恢复足够时间
- 但不让它变成无限延期

## 9. 为什么高波动环境用户更容易踩中这些治理时钟

因为高波动环境不是只增加一次错误，  
而是更容易让多个时间窗口错位：

1. 登录已过去很久，trusted device 窗口消失
2. token 快过期，网络又刚好波动
3. 某连接域进入 15 分钟 `needs-auth` 缓存
4. bridge 已接近 3 次熔断阈值
5. 用户还在高频试错

这就是为什么很多中国用户的主观体验会更像：

- “刚刚还能用，突然就像被封”

从结构上看，这其实常常不是新的处罚，  
而是多个时间边界同时错位后的连续性崩塌。

## 10. 从第一性原理看，为什么时间边界是风控哲学的一部分

如果强能力 agent 平台只能在空间上判定边界，  
它就会很粗。

成熟平台必须同时治理：

1. 资格边界
2. 组织边界
3. 会话边界
4. 时间边界

因为现实里的身份和会话都不是静态对象。  
它们只能在时间里持续被再证明。

## 11. 用苏格拉底式追问再压一层

### 问：为什么不把所有时间窗口都做得更长，让用户更轻松？

答：因为更长窗口意味着：

- 更弱的实时性
- 更高的滥用余量
- 更慢的边界收口

### 问：为什么不把所有时间窗口都做得更短，更安全？

答：因为那会：

- 增加误伤
- 增加用户摩擦
- 增加支持成本

所以真正成熟的系统，  
不是统一缩短或统一拉长，而是按对象定义不同治理时钟。

## 12. 给用户的合规启示

对用户最有价值的不是记住所有数字，  
而是记住一个原则：

- 很多问题不是“有没有资格”，而是“是否还处在正确时间窗口内”

因此更好的做法是：

1. 高价值能力尽量在刚完成正规登录、环境稳定时建立
2. 不要在长时间漂移后再临时做高安全能力首次尝试
3. 识别已经进入 cooldown / cache / fuse 的路径，不要继续高频重试

## 13. 给平台构建者的启示

1. 时间边界应显式建模，而不是埋在注释里。
2. 用户应能看见关键时间窗口是否仍然成立。
3. 高波动环境用户最需要的，常常不是更松，而是更早看到窗口快失效。

## 14. 一句话总结

Claude Code 的风控不是只在空间上画边界，也在时间上画边界；很多“像被封了”的体验，本质上来自这些治理时钟在高波动环境中的错位叠加。
