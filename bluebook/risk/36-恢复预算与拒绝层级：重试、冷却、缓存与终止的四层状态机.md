# 恢复预算与拒绝层级：重试、冷却、缓存与终止的四层状态机

## 1. 为什么要单独讨论“恢复预算”

很多用户直觉上会把失败理解成二选一：

1. 还能继续试
2. 已经彻底不行

但从源码看，Claude Code 远不是这样。  
它对失败至少使用了四种不同的治理动作：

1. 重试
2. 冷却
3. 缓存拒绝
4. 终止

这说明它的失败处理不是简单报错，而是一套带预算的恢复系统。

这里的“恢复预算”不是源码里的原始术语，而是我对这些机制的归纳。  
它指的是：系统允许某类失败在多大范围、多久时间、以什么方式被继续尝试，而不是立刻终止或永久放弃。

## 2. 先说结论

从当前源码看，Claude Code 并不会把所有失败都当成同一种东西。  
更准确的说，它在比较四类动作的成本：

1. 继续重试的成本
2. 暂时冷却的成本
3. 缓存拒绝的成本
4. 直接终止的成本

真正的风控成熟度，不只是“会不会拦”，还包括：

- 什么该立刻重试
- 什么该先停一下
- 什么该短期记忆为必败
- 什么已经不值得继续恢复

## 3. 第一层：重试

### 3.1 API retry 不只针对网络，还针对一部分 auth 与 capacity 问题

`withRetry.ts` 里可以看到：

- 408 可重试
- 409 可重试
- 某些 401/403 可重试
- 429/529 在不同前提下也可重试

这说明系统并不把“认证失败”都当终局。  
某些 401/403 在它看来更像：

- 凭证刚过期
- token 已被别处刷新
- CCR 模式下的短暂基础设施抖动

### 3.2 `handleOAuth401Error()` 体现的是“先检查是不是别人已经帮你恢复了”

`auth.ts` 的 401 恢复并不是直接暴力刷新。  
它会先：

- 看 keychain 里是不是已经有别的进程刷新好的新 token
- 对同一失败 token 去重并发处理

这说明“重试”不是盲重试。  
它首先在控制重复成本和跨进程竞争成本。

## 4. 第二层：冷却

### 4.1 fast mode 被 429/overloaded 打到时，不一定直接永久关闭

`fastMode.ts` 把 fast mode runtime state 建模成：

- `active`
- `cooldown`

这说明某些失败的正确动作不是“继续上”，  
而是“进入一段明确的冷却期”。

### 4.2 冷却是治理动作，不只是用户体验细节

触发 cooldown 时，源码会：

- 记录 reset 时间
- 记录原因
- 发事件

这表明 cooldown 并不是 UI 上的小提示，  
而是系统正式承认：

- 这类失败继续立刻重试只会更糟

## 5. 第三层：缓存拒绝

### 5.1 `needs-auth` 的 15 分钟缓存，本质上是短期“必败记忆”

`mcp/client.ts` 明确有：

- `MCP_AUTH_CACHE_TTL_MS = 15 * 60 * 1000`
- cached `needs-auth`

这说明当某连接域进入 `needs-auth` 后，系统不会每次都再去硬连一次。  
它会短期记住：

- 这条路径当前大概率必败

这是一种非常典型的“缓存拒绝”。

### 5.2 跨进程死 token 也会进入拒绝记忆

`initReplBridge.ts` 会把“同一个死 token 已经连续失败多次”记到全局配置里。  
达到阈值后，新的进程会直接跳过注册尝试。

也就是说：

- 不是只有 MCP 在做短期必败记忆
- bridge 高安全链路也在做

## 6. 第四层：终止

### 6.1 `fatal` 不是“报错严重一点”，而是恢复预算已经结束

`bridgeMain.ts` 和 `replBridge.ts` 都把 `auth_failed` 与 `fatal` 分开。

其中 `fatal` 更接近：

- 环境被删
- 环境已过期
- 某条链路已经不值得继续同样方式恢复

这说明 `fatal` 本质上不是情绪化文案，  
而是恢复预算被正式用尽。

### 6.2 非终止失败与终止失败的区别，在于是否还值得投入更多尝试

源码里最成熟的地方之一就是：

- 不把所有失败都一路试到死
- 也不把所有失败都过早判死

而是在重试、冷却、缓存拒绝、终止之间做层级切换。

## 7. 为什么这对中国和高波动环境用户更重要

高波动环境的一个典型风险是：

- 用户把“应该冷却”的问题继续猛试
- 把“应该走单域恢复”的问题扩大成全局变量切换
- 把“已经进入缓存拒绝”的路径继续频繁触发

于是本来系统已经在帮你降噪，  
用户自己又把噪声加了回去。

所以对这类用户来说，真正重要的不是“如何再试得更巧”，  
而是：

- 先判断当前失败属于哪一层恢复预算

## 8. 从第一性原理看，这套层级为什么先进

如果一个平台只有“放行”和“拒绝”两个状态，  
那它在真实世界里会很粗糙。

成熟系统必须至少回答四个问题：

1. 这个错误值得马上重试吗？
2. 这个错误值得先冷却吗？
3. 这个错误值得短期记忆成必败吗？
4. 这个错误是否已经应被视为终止？

Claude Code 当前源码最成熟的地方之一，  
就是它已经在按这四层思考。

## 9. 用苏格拉底式追问再压一层

### 问：用户多试几次，总有一次成功，不是更好吗？

答：不一定。

如果当前错误落在“冷却”或“缓存拒绝”层，  
多试只会增加噪声和误判成本。

### 问：为什么不把所有问题都缓存拒绝，省得系统忙？

答：因为那会把可恢复错误也过早冻结，误伤成本反而升高。

### 问：为什么不把所有错误都无限重试？

答：因为那会把会话、认证、capacity 问题放大成风暴，  
平台和用户都会更糟。

## 10. 给用户的合规启示

遇到失败时，先问自己：

1. 这是该重试的问题吗？
2. 这是该冷却的问题吗？
3. 这是已经进入短期拒绝缓存的问题吗？
4. 这是已接近终止的问题吗？

对中国和高波动环境用户尤其重要的是：

- 不要对明显需要冷却的路径做高频重试
- 不要对明显是单域 `needs-auth` 的问题做全局重置
- 不要把 `fatal` 误当成再多试一次就会好的问题

## 11. 给平台构建者的启示

1. 把失败分层，比统一 retry policy 成熟得多。
2. cooldown、cache、fatal 都应是正式状态，而不是散落的局部技巧。
3. 对用户可见的解释层，应尽量反映失败当前处在哪一层预算里。

## 12. 一句话总结

Claude Code 的失败治理成熟度，不只在于会恢复，更在于知道什么时候该重试、什么时候该冷却、什么时候该短期拒绝、什么时候该正式终止。
