# 共享词汇表：用户、管理员与支持团队如何减少语义错配

## 1. 为什么要单独写一章“共享词汇表”

很多误伤和争议，并不是因为系统完全没有给出线索，  
而是因为不同角色在用不同语言描述同一件事。

例如：

- 用户说“我被封了”
- 管理员说“这是组织策略”
- 平台支持说“这是 entitlement / auth / needs-auth / rate_limit”

如果三方没有共用词汇，  
再好的源码分层也会在沟通层重新塌缩成混乱。

这一章讨论的就是：

- 同一个失败，用户、管理员、支持团队各自该怎么叫它
- 为什么这件事对高波动环境用户尤其重要

## 2. 先说结论

从当前源码看，Claude Code 已经在内部逐步形成一套 reason family：

1. `token_revoked`
2. `oauth_org_not_allowed`
3. `auth_error`
4. `needs-auth`
5. `rate_limit`
6. entitlement / `not enabled`
7. policy denied / admin-governed restriction

但这些语义在用户体验里仍常常被压成：

- 不能用
- 被卡了
- 像被封了

因此，真正降低误伤成本的关键之一，不只是写更多文档，  
而是建立一套让用户、管理员、支持团队都能共用的词汇表。

## 3. 源码里已经存在哪些词汇层

### 3.1 支持词汇层：reason family

`errors.ts` 已经把多类认证问题拆成不同分类：

- `token_revoked`
- `oauth_org_not_allowed`
- `auth_error`

这说明平台内部其实已经有比较稳定的支持词汇层。  
也就是说，支持团队并不是只能看一句模糊报错。

### 3.2 用户词汇层：字面提示和启动通知

用户直接看到的往往是：

- `Please run /login`
- `Use your existing Claude Pro/Max plan · /login to activate`
- `MCP server needs auth`
- `claude.ai connector unavailable`
- `Remote Control is not yet enabled for your account`

这些提示已经比通用 “Error” 强很多，  
但它们仍然是字面语言，不是完整的 reason family。

### 3.3 管理员词汇层：组织限制与本地危险变更

管理员更应该熟悉的是：

- policy denied
- `forceLoginOrg`
- dangerous managed settings

这些词在用户嘴里经常会变成：

- 我突然不能用了
- 它怎么不认我这个组织
- 这是不是把我限制了

所以管理员必须能把用户的自然语言重新映射回组织治理词汇。

## 4. 为什么没有共享词汇，就会放大误伤

### 4.1 用户会误判问题所在层级

例如：

- entitlement 缺失被误判成封号
- `needs-auth` 被误判成全局账号异常
- 组织不允许被误判成平台针对个人

### 4.2 管理员会误接并不属于组织层的问题

例如：

- 其实只是用户没走对订阅激活路径
- 但用户先找管理员说“平台封了我”

### 4.3 支持团队会消耗更多翻译成本

因为支持团队必须先做一件本不该那么昂贵的事：

- 把用户的自然语言抱怨翻译回内部 reason family

这就是语义错配带来的额外税。

## 5. 共享词汇表最小版

下面这张表不是产品规范，而是研究建议。

| 用户常说的话 | 更准确的共享词汇 | 主要归属方 | 最短动作 |
| --- | --- | --- | --- |
| 我被封了 | entitlement 未命中 / auth_error / policy denied / rate_limit / needs-auth / fatal 中的一种 | 先分流，不直接归因 | 先按错误家族分层 |
| 我登录了但还是不行 | `auth_error` 或主体链未闭合 | 用户 + 平台支持 | 核对登录路径与 profile scope |
| 我有订阅但功能没了 | 订阅未激活到当前路径 / entitlement 未命中 | 用户 + 订阅支持 | `/login to activate` 或核对 gate |
| 组织账号怎么也不认 | `oauth_org_not_allowed` / `forceLoginOrg` 不匹配 | 用户 + 管理员 | 核对组织绑定与机器要求 |
| 某连接器突然坏了 | `needs-auth` / connector unavailable | 用户 + 该连接域支持 | 在该域内恢复，不做全局重置 |
| 今天突然不能继续用了 | `rate_limit` / `out_of_extra_usage` / model unavailable | 用户 + 计费/部署支持 | 看 reset time、额度或模型可用性 |

## 6. 为什么中国和高波动环境用户更需要共享词汇

因为这类用户更容易同时遇到多层问题：

1. 身份连续性波动
2. 组织画像不稳定
3. 连接域认证失效
4. 高安全会话不连续

在这种环境里，如果没有共享词汇，  
最容易发生的就是：

- 用户用情绪化词汇描述
- 管理员按组织词汇理解
- 支持团队按系统词汇处理

三方看的是同一个问题，  
但却像在说三件不同的事。

## 7. 从第一性原理看，为什么共享词汇也是风控基础设施

如果风控系统的目标之一，是降低总伤害，  
那共享词汇就不是附属文案工作，而是基础设施。

因为它决定：

1. 问题是否会被正确归层
2. 恢复路径是否会被正确选择
3. 支持资源是否会被正确分流

这直接关系到：

- 用户是否继续误伤自己
- 管理员是否被错误打扰
- 支持团队是否能快速闭环

## 8. 用苏格拉底式追问再压一层

### 问：是不是只要给用户更多报错文字就够了？

答：不够。

因为共享词汇不是“更多话”，  
而是“同一件事在三方语境里能不能对齐”。

### 问：为什么这件事不只是支持优化？

答：因为它反过来影响风控本身的成本。

如果用户不断误判并做错动作，  
风控系统就会被迫处理更多人为噪声。

### 问：为什么平台正义也和这件事有关？

答：因为谁更懂内部词汇，谁就更容易成功自证。  
而高波动环境用户往往恰恰最缺这层优势。

## 9. 给用户的合规启示

如果你想降低自己在误伤中的损失，  
最有价值的一件事，是尽量用共享词汇描述问题。

例如，不要只说：

- 我被封了

更好的说法是：

- 当前提示更像 `needs-auth`
- 当前更像订阅未激活
- 当前像组织绑定不匹配
- 当前像高安全会话 `auth_failed`

这不是为了迎合平台，  
而是为了让自己更快进入正确支持路径。

## 10. 给平台和管理员的启示

1. 平台应把内部 reason family 继续稳定化。
2. 管理员界面和用户提示应尽量共用同一族词汇。
3. 支持模板应鼓励用户按 reason family 描述问题，而不是只给一句“不能用”。

## 11. 一句话总结

很多误伤不是因为系统完全没有答案，而是因为用户、管理员和支持团队没有共用同一套词汇；一旦词汇失配，恢复路径就会立刻变长。
