# 连续性自检、故障窗口纪律与证据包：高波动环境用户的合规自保手册

## 1. 为什么还要单独写这一章

`07`、`13`、`17`、`20` 已经给了不少建议。  
但这些建议还比较分散，读者遇到真实故障时，未必能在五分钟内做对动作。

这一章把前面零散的建议压成一套可执行手册：

1. 出问题前如何做连续性自检。
2. 出问题时哪些变量必须冻结。
3. 出问题后该如何保留证据并保护自身权益。

这里讨论的只是不绕过平台策略的合规自保，  
不提供伪装、规避检测或对抗平台治理的方法。

## 2. 先说结论

从当前源码设计看，最有效的合规自保，不是“想办法让系统看不见你”，而是：

1. 让系统更稳定地把你识别为同一个主体。
2. 让系统更稳定地把你放在正确的组织和能力边界里。
3. 在故障窗口里减少高噪声试错，避免把普通问题放大成跨层混乱。
4. 在误伤发生时保留结构化证据，降低解释和支持成本。

对中国和其他高波动网络环境用户尤其如此。  
真正要对抗的不是“检测器”，而是“连续性破坏”。

## 3. 源码为什么支持这个结论

### 3.1 entitlement gate 本身就在努力减少“陈旧 false”误伤

`checkGate_CACHED_OR_BLOCKING(...)` 的设计很有代表性：

- 如果本地缓存已经是 `true`，直接放行
- 如果本地缓存是 `false` 或缺失，就阻塞拉一次最新服务端值

这说明作者明确知道：

- 陈旧的 `false` 更容易误伤用户
- 对用户主动触发的能力检查，宁可多等一次，也不要轻易把人挡在外面

### 3.2 remote managed settings 资格预检宁可接受假阳性，也不想把治理漏掉

`isRemoteManagedSettingsEligible()` 允许某些 externally injected token 先进入远程查询，  
即使它们本地看不出完整 `subscriptionType`。

这背后的取舍很清楚：

- 错误地多做一次查询，成本很低
- 错过真正存在的组织治理，成本更高

对用户来说，这意味着：

- 某些“明明画像不全，系统为什么还在继续查”的现象，不必理解为系统在针对你
- 更合理的理解是平台正在努力避免把治理面完全漏掉

### 3.3 高安全远程会话把 `auth_failed` 和 `fatal` 拆开

`bridgeMain.ts` 的 `heartbeatActiveWorkItems()` 明确区分了：

- `401/403` 导致的 `auth_failed`
- `404/410` 导致的 `fatal`

这类区分非常关键。

它说明在平台语义里：

- token 失效仍被视为可恢复连续性问题
- 环境过期或被删除则更接近终局失败

所以用户遇到远程失败时，最先要做的不是“是不是被封了”，  
而是先判断自己到底落在哪一类失败语义里。

### 3.4 `needs-auth` 只在“曾经连上过”的 claude.ai connector 上提醒

`useMcpConnectivityStatus.tsx` 的一个细节很说明问题：

- 对 claude.ai connector 的 `failed` / `needs-auth`
- 只有“曾经成功连接过”的连接器才会被高亮提醒

这是一种典型的 anti-nag 设计：

- 平台不想把“从来没真正连通过的配置”不断制造成高优先级告警
- 它更关注“状态发生了实质变化”的对象

这对用户的启示是：

- “曾经稳定可用，现在突然不行”是很关键的证据
- 最近一次成功时间，本身就是支持和诊断里非常高价值的事实

### 3.5 计费与处罚被故意拆开

`rateLimitMessages.ts` 明确把以下状态单独建模：

- `usage limit`
- `session limit`
- `weekly limit`
- `extra usage`
- `out of extra usage`
- `resets ...`

这意味着很多“突然不能继续”的情况，在系统内部根本不是处罚语义。  
如果用户自己先把它写成“封号”，后续只会增加沟通噪声。

## 4. 会前连续性自检

这一节的目标只有一个：

在你真正去用高风险或高价值能力之前，先确认自己没有处在明显脆弱的连续性状态。

### 4.1 身份路径

优先确认：

1. 你当前主力环境只保留一条主身份路径。
2. 你知道自己现在到底在用 `/login`、环境变量 token、长期 token 还是 API key。
3. 你没有在同一环境里混用多个组织账号和多个 token 来源。

### 4.2 组织路径

优先确认：

1. 当前机器是否受组织管理。
2. 当前账号是否应该属于该组织。
3. 当前操作到底是个人用途还是组织用途。

### 4.3 设备与网络路径

优先确认：

1. 高安全能力尽量只在主力设备上使用。
2. 故障高发窗口里不要频繁切换网络出口。
3. 不要把“短时可达”误当成“连续性稳定”。

### 4.4 功能路径

优先确认：

1. 你要用的是普通 CLI 能力，还是 Remote Control / bridge / MCP / 高安全远程能力。
2. 如果是高安全能力，就不要用“普通功能能用”来推断“高安全能力也一定应该能用”。

## 5. 故障窗口纪律

所谓“故障窗口”，就是：

系统刚开始连续报错，而你还没有确认失败语义的那段时间。

这时最重要的不是“继续试”，而是“冻结变量”。

### 5.1 必须冻结的四类变量

1. 身份变量：不要同时换账号、换 token 来源、换登录方式。
2. 组织变量：不要一会儿用个人，一会儿用组织，再一会儿换另一组织。
3. 设备变量：不要一边失败一边切主机、切容器、切远程环境。
4. 网络变量：不要在同一诊断窗口里频繁切出口或切网络策略。

### 5.2 先分语义，再动作

推荐顺序：

1. 先看错误原文是什么。
2. 先判断它更像身份、授权、组织、连接、计费还是高安全会话问题。
3. 只在该层内做最小动作。

### 5.3 明确不要做的事

不要做这些高噪声动作：

- 一口气重试很多次
- 同时换多个账号
- 同时换多种 token
- 远程高安全链路失败时，把所有普通能力也一起重置
- 没看清错误语义就先做“全局恢复”

这些动作最容易把一个本来可解释的问题，升级成跨层混乱。

## 6. 证据包最小规范

如果你确实怀疑自己被误伤，最少应保留下面八类信息：

1. 首次报错的精确时间
2. 错误原文
3. 最近一次成功使用的时间
4. 当前身份路径类型
5. 当前是否属于组织环境
6. 当前设备是否发生切换
7. 当前网络是否发生切换
8. 当前问题是否涉及 Remote Control / bridge / MCP / rate limit

如果能多保留，优先再加：

1. 套餐和付款状态
2. 最近一次登录或重认证时间
3. 故障窗口内是否做过变量切换

## 7. 面向中国和高波动环境用户的合规结论

这里必须把边界说清：

- 不能根据当前客户端源码断言平台在地域上一定针对你
- 但可以明确说，高波动环境更容易破坏主体连续性

因此，更有效的自保方式不是“规避检测”，而是：

1. 固定主力设备
2. 固定主力网络
3. 固定主力身份路径
4. 把工作和个人边界分开
5. 把支持证据保留得更完整

从保护自身利益的角度，这些动作比任何“乱试直到成功”都更有价值。

## 8. 给平台构建者的启示

如果你在做自己的 agent 平台，这一章最值得迁移的是：

1. 在高价值能力前提供连续性 preflight，而不是把所有失败留到中途爆发。
2. 把故障窗口内最常见的错误动作反过来设计成产品提示。
3. 给用户可导出的最小证据包，降低支持摩擦。
4. 对高波动环境用户，重点优化解释层，而不是单纯加更多 gate。

## 9. 一句话总结

对用户最有效的合规自保，不是隐藏自己，而是让身份、组织、设备、网络和证据都尽可能连续、稳定、可解释。
