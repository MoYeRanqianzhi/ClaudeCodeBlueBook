# 如果要把误伤再降一半：平台必须把哪些现有能力前置成产品

## 1. 为什么还要再写一章“平台改进”

前面已经多次写过平台侧改进：

- `21` 平台改进清单
- `26` 苏格拉底附录
- `35` 可证明性产品化

但随着 `41-53` 这一长段的深入，  
现在已经能更明确地说：

Claude Code 并不缺少很多底层能力。  
它更缺的是：

`把这些已存在的状态面、证据面和提醒面真正前置成用户可操作产品。`

这一章要回答的就是：

- 如果平台真的想把误伤再降一半
- 最应该先把哪些已有零件收敛成产品

## 2. 先说结论

从当前源码看，平台并不需要先发明一套全新风控系统。  
它更需要把已经存在的五类能力前置：

1. 主体状态面
2. 连接状态面
3. 故障分流面
4. 证据导出面
5. 升级求助面

换句话说：

`误伤的主要短板，不再是“底层完全没能力”，而是“用户太难看到和使用这些能力”。`

## 3. 已经存在的产品零件

### 3.1 `auth status` 已经是一个低成本主体状态面

`authStatus()` 已经能给出：

- `loggedIn`
- `authMethod`
- `apiProvider`
- `orgId`
- `orgName`
- `subscriptionType`

这已经足够支持一个非常有价值的主体状态卡片。  
但现在它主要还是命令行输出，  
还不是持续前置的产品面。

### 3.2 MCP 状态其实已经有一张小型连接健康面

`useMcpConnectivityStatus.tsx` 和 `status.tsx` 已经说明：

- `needs-auth`
- `failed`
- `connected`
- `pending`

都能被分别统计和展示。  
`MCPListPanel.tsx` 也已经把 enterprise / user / local / project / claude.ai connector 分层展示。

这说明连接健康面其实已经存在，  
只是还没有完全被提升成“误伤前置诊断入口”。

### 3.3 notification 逻辑已经在区分“值得提醒”的状态变化

例如 claude.ai connector 只有在：

- 以前连通过
- 现在 failed / needs-auth

时才更值得被提醒。

这意味着平台已经掌握了一个很重要的产品原则：

`状态变化比静态坏状态更值得提示。`

如果把这个原则继续推进，  
误伤前置能力会明显变强。

### 3.4 tracing / diagnostics / permission logging 已经具备产品化基础

前面章节已经证明：

- permission decisions 已记录来源
- diagnostics 已能低敏感记录事件
- tracing 已能表达 interaction / tool / request

从底层能力看，  
平台已经具备做“用户可导出的结构化证据包”的条件。

真正缺的是：

- 一个统一出口
- 一个统一术语层
- 一个统一升级动作

## 4. 如果要把误伤再降一半，平台最先该做什么

### 4.1 做一张统一的主体-资格-会话卡

这张卡不需要泄露内部实现，  
只需要把用户最关心的三件事并排显示：

1. 主体：你当前是怎样登录的
2. 资格：你当前能否使用相关能力
3. 会话：当前这条运行链路是否健康

现在这些信息分散在：

- `auth status`
- MCP 状态
- bridge/remote 报错
- rate limit 提示

用户必须自己脑内合并。  
这正是误伤成本的来源之一。

### 4.2 做“先分层、后动作”的诊断向导

平台目前已经有足够多的 reason family，  
但用户仍然容易只看到：

- 不能用

更好的产品化应该先问用户：

1. 是全局问题还是局部连接问题
2. 是认证、组织、额度还是高安全会话问题
3. 最近是否发生过身份/网络/设备变化

这会把错误动作前置拦住。

### 4.3 把证据导出变成正式动作，而不是隐式能力

当前源码已经具备：

- auth 状态
- diagnostics 事件
- JSONL 错误日志
- MCP 日志
- 部分 tracing

如果平台真想把误伤再降一半，  
这些能力就不该散落着。  
它们应当收敛成一个正式动作：

- 导出最小证据包

这样用户就不必在故障窗口里边慌边找。

### 4.4 把“下一步找谁”做成系统输出，而不是文档知识

现在的痛点之一不是没有支持路径，  
而是支持路径在用户脑子里。

如果系统已经能区分：

- `needs-auth`
- org policy
- rate limit
- bridge `auth_failed`
- `fatal`

那它就更应该进一步输出：

- 先找自己
- 先找管理员
- 先等 reset time
- 还是找平台支持

这一步会显著降低解释损失。

## 5. 为什么这些改进对中国和高波动环境用户尤其重要

因为高波动环境用户更容易同时遭遇：

1. 主体状态不稳
2. 会话状态不稳
3. 局部连接域状态不稳
4. 更高的误分流概率

也就是说，他们不是更需要“隐藏自己”，  
而是更需要：

- 更低成本看到正确状态
- 更低成本导出正确证据
- 更低成本进入正确支持路径

如果平台能把这三步前置，  
误伤损失会明显下降。

## 6. 从第一性原理看，平台为什么还没做到

因为现在很多能力仍然停留在“工程能力”层，  
还没有完全上升为“用户产品能力”。

这是两个不同问题：

1. 系统内部能不能知道
2. 用户能不能及时看到并正确使用

Claude Code 在第一问上已经做了很多。  
在第二问上，仍然有明显空间。

## 7. 给平台构建者的技术启示

一个成熟平台若想继续降低误伤，  
优先级不应只是：

- 再多加一个 gate
- 再多采一个指标

而应是：

1. 把现有状态面统一
2. 把现有 reason family 用户化
3. 把现有日志/诊断面导出化
4. 把现有分流规则产品化

这比继续增加隐性复杂度，更可能直接降低总伤害。

## 8. 用苏格拉底式追问再压一层

### 问：如果底层能力已经很多，为什么误伤还会重？

答：因为底层知道，不等于用户知道；  
支持能查，不等于用户能先做对。

### 问：为什么“产品化”也属于风控改进？

答：因为误伤并不只来自判定错误，  
还来自恢复路径太长、解释成本太高。

### 问：最该先做的一件事是什么？

答：把主体、资格、会话和连接健康收敛成一张统一状态面。

## 9. 一句话总结

如果平台真想把误伤再降一半，最优先的工作不是再发明一套新风控，而是把现有的 `auth status`、连接状态、reason family、diagnostics 和证据导出能力真正前置成用户可见、可用、可升级的产品面。
