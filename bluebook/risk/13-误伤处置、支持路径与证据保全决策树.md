# 误伤处置、支持路径与证据保全决策树

## 1. 为什么需要这一章

当用户觉得自己“像被封了”，最常见的问题不是缺少情绪，而是缺少正确的诊断顺序和证据保全方式。

这章的目标不是教你绕过系统，而是：

在真正出错时，用更低成本、更高成功率的方式保护自己。

## 2. 先说结论

误伤处置最重要的不是“继续尝试更多方法”，而是三件事：

1. 先识别错误语义。
2. 再冻结变量。
3. 最后用证据包走支持路径。

如果顺序反了，很多问题会被你自己越试越乱。

## 3. 第一步：先识别错误语义

### 3.1 如果你看到的是 `401`

优先判断：

- token 是否过期
- 是否发生了身份源混用
- 是否刚刚切换了设备、环境或组织

### 3.2 如果你看到的是 `403`

优先判断：

- 是 access denied 还是 insufficient scope
- 是组织权限问题还是高安全链路问题

### 3.3 如果你看到的是 `not enabled for your account`

优先判断：

- 这是资格缺失还是 rollout 未开放

### 3.4 如果你看到的是 `disabled by your organization's policy`

优先判断：

- 这是组织治理结果，不是本地设置异常

### 3.5 如果你看到的是 `needs-auth`

优先判断：

- 是某个连接域需要重新认证，不一定是全局账号问题

### 3.6 如果你看到的是 rate limit / extra usage exhausted

优先判断：

- 这是计费和消耗问题，不是风控处罚

## 4. 第二步：冻结变量

一旦出现疑似误伤，先停止以下动作：

- 继续切换账号
- 继续切换 token 来源
- 继续切换 provider
- 继续切换设备和网络环境
- 在同一故障窗口里疯狂重试远程环境、bridge 或 MCP 登录

原因很简单：

这些动作会把原本还能解释的问题，变成一个更难解释的混合故障。

## 5. 第三步：建立最小证据包

至少保留以下信息：

1. 发生时间
2. 当前账号类型
3. 当前组织信息
4. 当前设备和环境
5. 当前错误语义
6. 故障前是否有账号/网络/设备切换
7. 是否涉及 remote-control / bridge / MCP / 高权限自动化

如果是订阅或权益相关，还应加上：

8. 付款凭证
9. 套餐类型
10. 是否曾提示切换到已有订阅或重新登录

## 6. 第四步：按层走支持路径

### 6.1 身份层问题

更适合走：

- 登录与账号支持
- 身份恢复
- 组织绑定确认

### 6.2 资格层问题

更适合走：

- 套餐与 entitlement 支持
- 功能开放状态确认

### 6.3 组织层问题

更适合走：

- 管理员
- IT / 组织策略负责人

### 6.4 本地执行层问题

更适合走：

- 本地配置排查
- 工作区 trust
- 权限与 sandbox 诊断

### 6.5 高安全远程会话问题

更适合走：

- Remote Control / bridge / device trust 相关支持

## 7. 一个实用决策树

### 情况 A：只有某个高安全远程能力失效

更像：

- capability / session / device 问题

不宜直接判断成：

- 整个账号被封

### 情况 B：所有 first-party 能力都突然失效

更像：

- 身份问题
- 组织问题
- 重大账号状态变化

### 情况 C：只是危险动作被 block

更像：

- 本地动作治理

### 情况 D：只是达到配额或 extra usage 用尽

更像：

- 计费/消耗问题

## 8. 对高风险地区用户更重要的一点

高波动网络环境用户更容易出现：

- 身份连续性破坏
- 远程高安全会话不稳定
- 跨层错误语义叠加

所以对这类用户来说，证据保全的重要性更高，不是更低。

真正能保护自身利益的，不是“再试十种方法”，而是：

- 让支持能快速看见你遇到的是哪一层问题
- 让你的主体与使用环境尽量保持稳定可解释

## 9. 一句话总结

误伤发生时，最正确的动作不是继续试错，而是先诊断语义、冻结变量、保存证据，再沿正确层级走支持路径。
