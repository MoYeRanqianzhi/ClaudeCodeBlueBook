# 隔离优先于并发

这一章想回答一个常被误解的问题：

为什么 Claude Code 的多 Agent 看起来在追求并发，但源码真正反复强调的却是隔离。

## 1. 先说结论

Claude Code 的多 Agent 设计强点，并不在“同时跑更多 agent”。

它更深的设计原则是：

先隔离，再并发。

这里的隔离至少包括：

1. 上下文隔离
2. 权限隔离
3. 文件系统隔离
4. transcript 隔离
5. 远程环境隔离

如果没有这些隔离层，并发只会更快地放大状态污染、权限泄漏、文件冲突和恢复困难。

关键证据：

- `claude-code-source-code/src/utils/forkedAgent.ts:345-462`
- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:430-685`
- `claude-code-source-code/src/tools/AgentTool/runAgent.ts:368-410`
- `claude-code-source-code/src/tools/AgentTool/runAgent.ts:412-479`
- `claude-code-source-code/src/tools/AgentTool/runAgent.ts:732-859`
- `claude-code-source-code/src/tools/EnterWorktreeTool/EnterWorktreeTool.ts:77-119`

## 2. 为什么“并发更多”本身不成立

先做三个最直接的苏格拉底式追问。

### 2.1 如果并发本身就够了，为什么还要克隆 `readFileState`

因为共享同一份 read cache 会带来两个问题：

- 子代理会把父代理的读写校验链弄乱。
- 上下文裁剪、snip、resume 后，Edit 校验可能误把旧观察当成新观察。

所以 Claude Code 让子代理默认克隆 `readFileState`，而不是共享。

### 2.2 如果并发本身就够了，为什么还要 sidechain transcript

因为平铺日志会丢掉：

- 哪条消息属于哪个子代理
- 哪段链是 fork 出来的
- 哪些状态可以独立恢复

因此 subagent transcript 被单独持久化，并保持自己的 parent chain。

### 2.3 如果并发本身就够了，为什么还要 worktree / remote isolation

因为共享同一个代码现场会让并发直接变成冲突源：

- 文件竞争
- git 状态混杂
- cleanup 时误删
- 结果归属不清

所以 Claude Code 宁愿多建一层 worktree，甚至把任务送到远端，也不愿把所有 agent 堆在同一工作目录里抢写。

## 3. Claude Code 实际在隔离什么

### 3.1 隔离可变状态

`createSubagentContext(...)` 的默认策略几乎可以概括成一句话：

把父上下文里所有可能被污染的可变状态，都先隔离出去。

这包括：

- `readFileState`
- `abortController`
- `contentReplacementState`
- `toolDecision` 相关状态
- 大多数 mutation callback

所以子代理默认更像“受控分支”，而不是“共享线程”。

### 3.2 隔离权限

`runAgent(...)` 不只是把父权限搬过去。

它会按 agent 自己的：

- permission mode
- async / sync 形态
- allowed tools
- bubble / avoid prompts

重新塑造 permission context。

这说明 Claude Code 不把“父代理已被允许”自动等同于“子代理也应被允许”。

### 3.3 隔离文件系统

worktree 的意义不是“多一个目录”，而是：

- 给子代理一个独立代码现场
- 允许路径重写与重新读取
- 让 cleanup 可按变更状态决定 keep 或 remove

`ExitWorktreeTool` 的 fail-closed 策略尤其说明，系统把“保住工作成果”放在“收拾干净临时目录”之前。

### 3.4 隔离环境

remote isolation 进一步表明本地隔离并不是终点。

当本机环境、网络边界、长时任务或执行信任模型不再适合本地运行时，Claude Code 会把 agent 整体送到另一执行环境，而不是只换一个 transport。

### 3.5 隔离历史

subagent transcript、agent metadata、output file、task registry 共同形成了一套“子任务也可恢复”的历史结构。

这与很多只展示最终答案的多 Agent 系统不同。

Claude Code 更像在说：

并发结果不够，结果的来源链也必须可追踪。

## 4. 苏格拉底式继续追问

### 4.1 为什么不让所有 subagent 都直接共享主会话状态

因为那会立刻破坏三件事：

1. prompt cache 稳定性
2. 编辑校验的因果链
3. 恢复时的归属关系

所以 Claude Code 宁可多做克隆、多写 metadata，也不直接共享。

### 4.2 为什么 changed worktree 默认保留，而不是自动回收

因为在工程场景里，工作成果的安全性比临时目录的整洁更重要。

这是典型的工程系统价值排序：

状态保全优先于资源回收。

### 4.3 为什么远程 agent 不是“高级 transport”而是“高级隔离”

因为它改变的不只是传输方式，而是：

- 执行环境
- 生命周期
- 状态恢复路径
- 本地与远端责任边界

所以 remote 不是通道增强，而是隔离等级增强。

### 4.4 为什么要把并发问题讲成隔离问题

因为并发只是速度问题；
隔离才是正确性、可恢复性、可追责性问题。

而在软件工程现场，后者比前者更基础。

## 5. 回到第一性原理：隔离不是第七个点缀，而是协作与恢复的前提

蓝皮书主线把 coding agent 还原成六个不可约问题：

1. 观察
2. 决策
3. 行动
4. 记忆
5. 协作
6. 恢复

隔离并不一定要独立列成“第七个问题”，但它显然是后两项的前提约束：

- 没有隔离，协作会变成共享状态污染。
- 没有隔离，恢复会变成无法区分该恢复谁的状态。

所以更准确的说法是：

隔离是协作与恢复能够成立的工程条件。

## 6. 这对蓝皮书本身意味着什么

继续写蓝皮书时，必须持续警惕一个误区：

把“多 Agent”写成“高并发”。

更准确的目录组织应该让读者看到：

1. AgentTool 讲的是编排，而不只是 spawn。
2. worktree / remote 讲的是隔离等级，而不只是执行位置。
3. sidechain transcript 讲的是恢复边界，而不只是日志存储。
4. permission shaping 讲的是状态收口，而不只是安全规则。

这也是为什么本轮需要把：

- `architecture/10-AgentTool与隔离编排.md`
- `api/09-会话与状态API手册.md`

与本章一起连成一条新的阅读线。

## 7. 一句话总结

Claude Code 并不是先学会并发，再补隔离；
它恰恰是先把上下文、权限、文件系统、历史和环境分开，再让多 Agent 这件事变得可持续。
