# 显式远程失败优于模糊在线状态

这一章回答四个问题：

1. 为什么 Claude Code 不把远程问题写成“在线/离线”二元状态。
2. 为什么 `401`、`4001`、`4003`、`1002`、timeout、env reconnect 必须被看成不同失败。
3. 为什么恢复期间主动丢消息，反而比“尽量不丢”更诚实。
4. 从第一性原理看，远程 Agent runtime 为什么必须偏向显式失败而不是模糊连续性。

## 1. 先说结论

Claude Code 在远程恢复这一层传达出一条很硬的设计哲学：

- 显式远程失败优于模糊在线状态

也就是说，系统宁可明确告诉你：

- 这是 unauthorized
- 这是 session not found
- 这是 protocol error
- 这是 auth recovery
- 这是 reconnect budget exhausted

也不愿把这一切都模糊成：

- 连接有点问题，再等等看

这种哲学很重要，因为远程 Agent runtime 最危险的不是失败本身，而是：

- 看起来还在线，实际控制语义已经失效

## 2. 为什么“在线/离线”二元状态不够

如果只用二元状态叙述远程系统，那么下面这些情况都会被混成一类：

1. 权限被拒
2. session 暂时不可见
3. transport 协议失效
4. token 过期但可恢复
5. token 已死且根本不该继续发请求

这会直接制造三种坏结果：

1. 系统把不可恢复错误误判成暂时波动。
2. 系统把暂时波动误判成彻底失败。
3. 宿主和用户都不知道当前还能不能相信控制环。

所以 Claude Code 的远程状态观不是：

- 连上了没有

而是：

- 当前控制语义还是否成立

## 3. 为什么恢复期间主动丢消息反而更诚实

很多系统在恢复窗口会本能地想：

- 尽量别丢消息

Claude Code 在 `authRecoveryInFlight` 期间的选择恰好相反：

- 直接 drop `control_request`
- drop `control_response`
- drop `control_cancel_request`
- drop `result`

这看起来激进，但其实更诚实。

因为旧 transport 的 epoch 已经陈旧，而新 transport 还没建好。
这时继续写，最糟糕的情况不是“消息丢了”，而是：

- 消息看起来发出去了
- 状态看起来还在推进
- 实际控制环已经不可信

所以这里真正被保护的不是“每条消息”，而是：

- 控制语义的真实性

## 4. 为什么 401 不是换 token，而是重建语义

Claude Code 对 401 的处理非常有代表性。

它不是：

1. 收到 401
2. 刷 token
3. 继续

而是：

1. 抢占 recovery 运行权
2. 切状态到 `reconnecting`
3. refresh OAuth
4. 重新拉 remote credentials
5. rebuild transport
6. 让新的 epoch 接管后续控制环

这说明它对远程失败的理解不是：

- 某个凭证失效了

而是：

- 当前 transport 语义已经过期

所以 401 的哲学本体不是 auth，而是：

- 旧控制宇宙已经失效，必须显式切换到新的宇宙

## 5. 为什么 preflight failure avoidance 也属于失败哲学

`initReplBridge.ts` 里主动避免 expired-and-unrefreshable token 继续发请求，这一点很值得写进蓝皮书。

因为它表明 Claude Code 不满足于：

- 出错后再解释

它还要求：

- 对 guaranteed-fail path 直接 fail closed

这是一种很成熟的失败观：

- 失败不该只在崩掉之后被识别
- 还应在确定必败时就阻断

## 6. 苏格拉底式追问

### 6.1 如果一个系统看起来在线，但控制消息已经不该被信任，它真的算在线吗

不算。
它只是还没把失败显式说出来。

### 6.2 如果 401 之后继续在陈旧 transport 上发控制消息，是在提高可用性，还是在制造假连续性

更像后者。

### 6.3 为什么主动丢消息有时比努力不丢更正确

因为远程控制里最危险的不是“少做了一步”，而是：

- 误以为那一步已经成功

### 6.4 为什么 expired token 应该在 preflight 就被挡住

因为 guaranteed-fail 请求不会给系统带来任何新信息，只会把状态拖进更模糊的失败循环。

## 7. 一句话总结

Claude Code 的远程设计值得学的，不是“断了还能连”，而是它始终优先维护失败语义的清晰度，拒绝模糊在线状态制造的假连续性。
