# 统一配置语言优于扩展孤岛

这一章想回答一个比“插件系统强不强”更根本的问题：

为什么 Claude Code 没有把扩展能力做成一个单点插件中心，而是分散在 frontmatter、目录约定、plugin manifest、MCP、hooks、agents 这些表面上看很碎的地方。

## 1. 先说结论

Claude Code 的扩展观更接近：

用统一配置语言装配 runtime。

而不是：

把所有能力都塞进一个插件孤岛里。

这里的“统一配置语言”并不意味着只有一种文件格式，而是意味着：

1. 多种扩展对象共享一套高度相似的声明语义。
2. 不同信任等级的扩展入口被刻意分层。
3. 运行时最终关心的是统一对象模型，而不是作者最初写的是 Markdown 还是 JSON。

## 2. 从第一性原理看，agent runtime 为什么需要统一配置语言

如果一个 coding agent runtime 真的想持续增长能力，它至少要同时解决四个问题：

1. 新能力必须能快速接入，而不必每次都改核心 loop。
2. 能力必须能按项目、用户、组织、安装包来源分层信任。
3. 不同能力要能被统一装配进 prompt、tool pool、hook、MCP、agent runtime。
4. 作者写扩展的方式要足够低摩擦，否则生态只会停留在内建功能。

如果只做一个“插件 API”，通常会失败在两头：

- 对轻量能力太重。
- 对高权限能力又太不安全。

Claude Code 的答案是把扩展拆成多个入口，但让它们共享一套声明风格和运行时对象模型。

## 3. 为什么 frontmatter 反复出现

### 3.1 因为它天然适合“内容 + 元数据”共存

skills、commands、agents 的本体都不是纯配置对象，它们都包含：

- 一段面向模型的正文
- 一组控制 runtime 的元字段

Markdown + frontmatter 恰好适合这种结构：

- 正文仍然是 prompt / guide / instructions
- 元字段则声明模型、工具、hooks、agent、effort、paths、shell

### 3.2 因为作者真正写的是能力意图，而不是底层对象实例

同一个 frontmatter 字段在不同对象上会映射到不同 runtime 语义：

- `allowed-tools` 对 command/skill 是授权边界
- `tools` 对 agent 是 worker tool pool 约束
- `paths` 对 skill 是条件激活
- `memory` 对 agent 会触发工具注入与 memory prompt

这说明 Claude Code 设计的重点不是“让作者构造 JS 对象”，而是“让作者声明能力意图，由 loader 解释为 runtime 对象”。

## 4. 为什么 plugin manifest 不是 frontmatter 的替代品

### 4.1 plugin manifest 解决的是“打包与安装信任”

plugin manifest 把：

- commands
- skills
- agents
- hooks
- MCP
- LSP
- userConfig
- channels

收进一个安装单位。

这解决的是：

- 如何分发
- 如何启用/禁用
- 如何声明安装时所需配置
- 如何在 marketplace 上发布

它解决的不是“所有能力都必须改成 manifest”。

### 4.2 本地 `.claude/` 配置解决的是“作者即操作者”

本地 `commands/skills/agents` 的高信任假设是：

- 写配置的人就是当前项目的作者或操作者。
- 因此可以允许更强的 agent-level 权限表达。

plugin 则不同：

- 安装 plugin 的人与写 plugin 的人不是同一个人。
- 所以 manifest 可以声明 bundle 级能力，但 plugin 内部单个 agent 不能再偷偷提权。

这也是为什么 plugin agent 当前故意忽略：

- `permissionMode`
- `hooks`
- `mcpServers`

## 5. 苏格拉底式追问

### 5.1 为什么不把所有能力都做成 plugin

如果这样做，会有两个问题：

1. 项目级轻量定制被迫走安装与分发流程，成本过高。
2. 高信任本地配置与第三方安装包的边界被抹平，安全上过于粗糙。

所以 Claude Code 保留了 `.claude/*` 这条高信任、低摩擦路径。

### 5.2 为什么不把 plugin agent 做成与本地 agent 完全同权

因为那等于允许第三方包把更高权限藏在单个 agent 文件里，绕过用户对 bundle 级能力的理解。

Claude Code 的做法很克制：

- plugin manifest 级能力在安装时批准
- plugin 内部 agent 不允许局部升权

这是一种很典型的分层信任设计。

### 5.3 为什么不把命令、技能、agent 各做一套完全独立的配置语法

因为那会立刻带来三种成本：

1. 作者学习成本飙升。
2. loader / validator / UI 呈现逻辑重复。
3. 能力之间更难互相迁移和重组。

共享 frontmatter 语义，本质上是在降低系统的扩展熵。

### 5.4 为什么不直接让 runtime 读取 JS/TS 插件代码来定义一切

因为声明式配置有两个显著优势：

1. 更容易做静态验证与策略限制。
2. 更容易根据 source / trust / gate 做有边界的裁切。

换句话说，Claude Code 优先选择了“可治理的扩展性”，而不是“最自由的可编程性”。

## 6. 这背后的设计内涵是什么

Claude Code 并不是在说：

- “插件比内建高级”

它更像在说：

- “能力应该先被抽象成统一语言，再按信任、分发、运行时语义映射到不同入口”

所以它的扩展哲学不是 plugin-first，而是 language-first。

这里的 language-first 至少带来三种好处：

1. 低摩擦：项目作者用 Markdown/frontmatter 就能表达复杂能力。
2. 高治理：第三方 plugin 仍被 manifest、schema、trust boundary 限制。
3. 高组合：命令、skill、agent、hook、MCP 能共享对象模型和装配流程。

## 7. 反过来问蓝皮书：我们之前哪里写得还不够

如果继续用苏格拉底式反思蓝皮书本身，一个明显不足是：

我们前几轮已经能讲清：

- 工具协议
- 状态面
- 隔离编排

但对扩展面，仍然容易写成“插件、技能、agent 分别有哪些功能”。

这还不够。

更好的写法应该先问：

1. 这些扩展对象共享了什么语法和对象模型。
2. 哪些差异来自运行时语义，而不是格式差异。
3. 哪些边界来自信任分层，而不是功能缺失。

只有这样，读者才不会把 Claude Code 误解成“功能很多，但子系统很散”。

## 8. 一句话总结

Claude Code 的扩展能力之所以显得既灵活又不完全失控，不是因为它有一个万能插件中心，而是因为它先把扩展收敛成统一配置语言，再把不同信任等级的入口分层开放。
