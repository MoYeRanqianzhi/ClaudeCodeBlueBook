# 安装状态、启用状态与策略状态必须分层叙述

这一章回答四个问题：

1. 为什么插件系统最容易被误写成一个“开/关”问题。
2. 为什么 Claude Code 坚持把安装、启用、scope 所有权、策略阻断分层。
3. 为什么这种分层不是啰嗦，而是防止插件系统出现半真相的必要条件。
4. 从第一性原理看，这对 Agent 平台的扩展系统意味着什么。

## 1. 先说结论

Claude Code 在插件系统里传达出一条很重要的设计哲学：

- 安装状态、启用状态与策略状态必须分层叙述

这意味着系统不接受下面这种偷懒写法：

- 插件有一个 `enabled`，其他都围着它转

因为从第一性原理看，插件系统至少同时面对三种完全不同的问题：

1. 资产问题：
   - 插件有没有被安装
2. 运行时问题：
   - 当前项目里它是不是生效
3. 治理问题：
   - 即使用户想开，系统是否允许它开

再往前走一步，还应再拆出第四种：

4. 所有权问题：
   - 如果用户要改，应该改哪个 scope

这三件事如果被压成一个 bool，结果就一定是：

- 看起来简单
- 实际到处半真相

## 2. 为什么“启用”本身就不是单一问题

当人们说“一个插件启用了”，其实可能在混说至少四件事：

1. 它已经装在机器上
2. 当前项目设置里把它写成了 true
3. 合并后的 settings 结果里它仍然生效
4. 组织策略没有把它 force-disabled

Claude Code 恰恰拒绝把这几件事混成一句话。

这背后的哲学很成熟：

- 一个名字相同的状态，不代表它在所有层里是同一种事实

这也说明插件系统真正该问的不是：

- enabled 了吗

而是：

- 你现在问的是哪一层真相

## 3. 为什么策略状态必须独立于用户意图

`pluginPolicy.ts` 的价值就在于它明确告诉你：

- 用户意图不是最高真相

如果组织策略把某个插件 force-disabled，那么即使：

- userSettings 想开
- projectSettings 想开
- localSettings 想开

系统也不该把“用户想开”伪装成“当前真的 enabled”。

这正是很多扩展系统容易犯的错误：

- 把用户意图当成运行时真相

Claude Code 则把它们拆开：

- 用户可以想开
- 但 policy 仍然可以说不

而且策略本身也不是一条轴。

插件级阻断与 marketplace 级阻断分别对应：

- 某个具体插件能不能用
- 这个来源整体允不允许进入系统

这说明治理状态从来就不该被压成单层开关。

## 4. 为什么 scope 所有权也必须单独叙述

`getPluginEditableScopes()` 之所以重要，是因为“谁能改它”不是“它当前是否生效”的同义句。

一个插件当前可能因为 merged settings 而 enabled，但真正应该写回的是：

- local
- project
- user

三者之一。

这说明插件系统里的 scope 不是显示装饰，而是：

- 修改权真相

如果修改权真相和运行时真相被混成一层，就会出现：

- 用户改了不该改的地方
- 系统却以为自己完成了 enable / disable

所以更稳的扩展系统应该接受：

- editable truth 也是正式真相

## 5. 为什么安装状态也不该被吞进启用状态

`installedPluginsManager.ts` 里最值得学的，不是 migration 细节，而是设计立场：

- 安装是全局资产面
- 启用是项目运行面

这意味着 Claude Code 不愿意用“已安装且启用”去偷换：

- 安装存在

因为这会让系统失去一种很重要的表达能力：

- 已有资产，但当前项目不使用它

而这种状态在多项目、多 scope、策略治理存在的系统里，是完全正常且必要的。

甚至还应继续追问：

- 当前会话正在使用的 live state，和磁盘上的持久化安装元数据，是不是同一层真相

Claude Code 的回答也是：

- 不是

## 6. 苏格拉底式追问

### 6.1 如果一个插件被安装了，但当前项目没启用，它究竟算存在还是不存在

要先问你说的是资产存在，还是运行时存在。

### 6.2 如果一个插件在 project scope 是 true，但 policySettings 是 false，它究竟算 enabled 吗

不能只拿 project scope 回答。

### 6.3 如果一个插件当前生效，但用户应该写回 local scope，这个“当前生效”能指导写回吗

不能完全指导。
因为写回真相和运行时真相不是同一层。

### 6.4 为什么 Claude Code 不把这些都藏进一个 helper 里

因为藏起来之后，系统会更快回到：

- 一堆看起来方便的半真相

### 6.5 为什么 loader 能加载到插件，不等于安装元数据已经同步

因为 runtime materialization 与 persistence bookkeeping 不是同一层事实。

## 7. 一句话总结

Claude Code 的插件系统之所以更成熟，不是因为它把问题简化成一个开关，而是因为它承认安装、启用与策略本来就是不同真相，并坚持把它们分层叙述。
