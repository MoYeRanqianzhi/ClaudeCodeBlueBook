# 产品实验与演化方法

## 1. 先说结论

Claude Code 的一个很少被单独讨论的设计内涵是：  
它不是一块固定产品，而是一个被持续实验、按 build 和 runtime 裁切的能力空间。

这件事在源码里几乎是明写的。

## 2. 它为什么不像“发布一次就定型”的工具

从多个入口都能看到两类开关并存：

1. 编译期开关：`feature('...')`
2. 运行期开关：GrowthBook / env / policy / entitlement

这意味着 Claude Code 的能力不是单一静态集合，而是：

- 先在 build 时裁掉一部分
- 再在 runtime 按账号、计划、组织、环境继续裁切

## 3. 编译期开关说明什么

`skills/bundled/index.ts` 是一个很好的例子。  
它先注册一批稳定 bundled skills，然后按 `feature()` 决定是否把：

- `dream`
- `hunter`
- `loop`
- `scheduleRemoteAgents`
- `claudeApi`
- `runSkillGenerator`

接进当前构建。

证据：

- `claude-code-source-code/src/skills/bundled/index.ts:15-79`

这说明在 Claude Code 的设计里，“功能是否存在于构建产物中”本身就是产品策略的一部分。

## 4. 运行期开关说明什么

`bridgeEnabled.ts` 把 runtime experimentation 写得非常直白：

- `isBridgeEnabled()`
- `isBridgeEnabledBlocking()`
- `getBridgeDisabledReason()`
- `isEnvLessBridgeEnabled()`
- `isCseShimEnabled()`
- `getCcrAutoConnectDefault()`
- `isCcrMirrorEnabled()`

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-36`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:38-87`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:118-147`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:175-201`

这不是简单的 `if enabled then show`。  
而是在表达：

- entitlement 是否存在
- fast-path 是否允许
- v1/v2 路径如何切换
- 兼容 shim 是否继续保留
- auto-connect / mirror 是否 rollout

这是一种非常工程化的产品演化方式。

## 5. plugin 与 skill 的关系也体现了演化策略

`plugins/bundled/index.ts` 明确写着：

- 不是所有 bundled feature 都该变成 built-in plugin
- 对复杂 setup 或 auto-enable 的功能，应该继续放在 `skills/bundled/`
- 当前 built-in plugin 还只是 scaffolding

证据：

- `claude-code-source-code/src/plugins/bundled/index.ts:1-23`

这说明 Claude Code 团队不是简单地把所有东西塞进 plugin，而是在刻意设计迁移路径。

## 6. 从第一性原理看，这种实验方法在解决什么问题

如果 Claude Code 要同时面对：

- 普通用户
- 企业用户
- 远程/bridge/CCR 环境
- 多种 provider
- 内外部不同 build

那么它不可能靠“一个定型产品面”解决所有问题。

它必须让能力可以：

1. 被裁掉
2. 被灰度
3. 被 entitlement 限制
4. 被兼容 shim 过渡
5. 被 runtime policy 接管

所以从第一性原理看，Claude Code 不是单一产品，而是实验化产品平台。

## 7. 苏格拉底式追问：这说明它还“不稳定”吗

表面看，好像很多开关意味着不稳定。  
但继续追问：

1. 如果不做 gate，怎么在复杂运行面里灰度远程能力。
2. 如果不做 shim，怎么把旧会话 ID 语义平滑迁到新系统。
3. 如果不把 plugin/skill 分层，怎么逐步迁移生态能力。

答案反而是：  
这些开关不是“不成熟”的证据，而是“大系统正在谨慎演化”的证据。

## 8. 对蓝皮书的启发

这也意味着蓝皮书写法必须升级：

1. 不能只写“有哪些能力”，还要写“这些能力处在哪个演化阶段”。
2. 不能把 feature gate 和 runtime gate 混写。
3. 要把“公开主路径”和“实验性路径”分开描述。

否则读者会误以为源码是一张平面地图，实际上它更像一张分层 rollout 地图。

## 9. 一句话总结

Claude Code 的设计内涵之一，是它把产品演化本身也工程化了：编译期开关负责裁剪形态，运行期开关负责灰度与切换，兼容层负责迁移，最终把“变化”也纳入 runtime 结构。
