# 闭环状态机优于单向请求

这一章回答四个问题：

1. 为什么 Claude Code 的宿主协议不能被理解成单向 request/response。
2. `control_request`、`control_response`、`SDKMessage`、状态回写共同构成了怎样的闭环。
3. 这种闭环思维为什么比“单次请求成功”更接近 Agent runtime 的本质。
4. 蓝皮书后续应该如何继续用这条原则反思自己。

## 1. 先说结论

Claude Code 的宿主协议真正押注的不是：

- 单向请求成功

而是：

- 闭环状态机成立

也就是说，一个动作真正“完成”通常不是在 `control_response` 返回那一刻，而是在：

1. 请求被接受
2. 状态被更新
3. 后续 `SDKMessage` 被发出
4. 宿主与 runtime 对当前状态重新达成一致

## 2. 从第一性原理看，为什么单向请求不够

只要系统真的要长期工作，它就不能只回答：

- 这个请求收到了没有

它还必须继续回答：

- 现在系统处于什么状态
- 这个请求之后触发了什么执行
- 是否还卡在 requires_action
- 是否已经回到 running / idle

否则宿主只能得到一个空洞的“成功”，却不知道这次成功把系统带到了哪里。

对聊天应用来说，这种缺陷也许还能忍；
对 Agent runtime 来说，这会直接毁掉控制能力。

## 3. Claude Code 在哪里体现了闭环观

### 3.1 `initialize`

不是：

- 请求成功 -> 结束

而是：

- `control_response(success)`
- 再加 `system:init`

这样宿主才能拿到 runtime 装配态。

### 3.2 `set_permission_mode`

不是：

- 请求成功 -> 结束

而是：

- `control_response(success/error)`
- 再加 `system:status`

这样宿主才能知道 mode 真正变成了什么。

### 3.3 `can_use_tool`

更不是：

- 用户点一次批准 -> 结束

而是：

- permission request
- success/cancel/error
- `requires_action -> running`
- 继续 tool/task 消息链

这整个闭环才叫“权限处理完成”。

### 3.4 远程恢复

也一样。

系统并不把“socket 连上了”当成恢复完成，而是继续要求：

- epoch 正确
- transport rebuild 完成
- flush state 正常
- worker_status 回写正确

这就是典型的闭环思维。

## 4. 为什么这比单向请求更成熟

单向请求的世界观是：

- 我发一个动作
- 对方回一个结果
- 这事就结束

闭环状态机的世界观则是：

- 我发一个动作
- 系统进入新状态
- 新状态还会持续产生日志、事件、阻塞、恢复
- 我必须继续观测，直到状态闭环

后者更复杂，但也更接近真实工程系统。

Claude Code 显然站在后者。

## 5. 这反过来解释了它为什么更像 runtime

runtime 的本质，从来不是“提供一个函数调用”。

runtime 的本质是：

- 维持一个持续演化的状态机

只要你把 Claude Code 的宿主协议、任务、权限、远程恢复、session state 放在一起看，这一点就越来越明显。

所以它不像：

- 一个 ask/answer API

更像：

- 一个持续被观测、被控制、被恢复的运行时

## 6. 苏格拉底式反问：蓝皮书还会怎么把闭环写丢

很容易在三处写丢：

1. 只列 subtype，不写 follow-on SDKMessage。
2. 只列 success path，不写状态回写。
3. 只列 transport 成功，不写恢复后的状态一致性。

只要出现这些情况，蓝皮书就会再次退回成“接口字典”，而不是“运行时手册”。

## 7. 后续写作约束

以后只要写到宿主协议或远程恢复，就必须先问：

1. 这个请求的 response 是什么。
2. 这个 response 之后还有哪些后续消息。
3. 哪一个信号才是真正的“闭环完成”。

如果第三问答不出来，这一段分析就仍然停留在表层。

## 8. 一句话总结

Claude Code 之所以比普通“请求-响应工具箱”更像 Agent runtime，正是因为它真正维护的是闭环状态机，而不是一堆彼此孤立的单向请求。
