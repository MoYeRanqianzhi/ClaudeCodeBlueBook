# 外化状态优于推断状态

这一章回答四个问题：

1. 为什么 Agent runtime 不能只让宿主靠消息回放推断状态。
2. Claude Code 为什么要把 `worker_status`、`pending_action`、`permission_mode`、`model` 外化出来。
3. 为什么“外化状态”不等于“把一切都写进 metadata”。
4. 用这条原则研究状态面时，最容易重新掉回哪些误读。

## 1. 先说结论

Claude Code 在状态面上的一个深层判断是：

- 重要状态应该被外化，而不是让宿主自己推断

这里的“外化”不是指多打一份日志，而是指：

1. 状态可查询
2. 状态可恢复
3. 状态可 merge
4. 状态可在 attach-late / reconnect 后继续成立

这就是为什么源码里不仅有 `SDKMessage`，还有：

- `worker_status`
- `requires_action_details`
- `external_metadata.pending_action`
- `external_metadata.permission_mode`
- `external_metadata.model`

## 2. 从第一性原理看，为什么“让宿主自己猜”不够

如果一个系统真的要跨：

- 进程
- 网络
- 远端 UI
- 恢复窗口

长期工作，那么宿主仅靠“回放已经发过的消息”去猜当前状态，会立刻遇到四个问题：

1. 它可能是中途 attach 进来的。
2. 它可能错过过往部分事件。
3. 它消费的是窄适配子集，而不是 schema 全集。
4. 它真正关心的是“现在卡在哪”，而不是“过去发生过哪些细节”。

所以对 Agent runtime 来说，纯事件推理永远不够稳。

## 3. Claude Code 在哪里体现了“外化状态”

### 3.1 把 requires-action 外化

源码没有满足于：

- 发一个 `control_request(can_use_tool)`
- 再发若干 follow-on 消息

它还把阻塞点外化成：

- `worker_status=requires_action`
- `requires_action_details`
- `external_metadata.pending_action`

这说明作者不接受“前端去扫消息自己猜当前阻塞原因”。

### 3.2 把 permission mode 外化

mode 变化不只通过 `system:status` 广播，还会进入：

- `external_metadata.permission_mode`
- `external_metadata.is_ultraplan_mode`

这意味着 mode 不是“只给当前会话看一眼”的瞬时 UI 状态，而是远端宿主需要持续知道的运行时真相。

### 3.3 把 model 外化

`set_model` / `apply_flag_settings` 改变 model 后，源码会主动：

- `notifySessionMetadataChanged({ model })`

这说明 model 也被视为应该恢复、同步、回看的宿主真相。

## 4. 为什么这不等于“metadata 一统天下”

Claude Code 没有把一切都塞进 metadata。

像下面这些信号，仍然主要留在事件流里：

- `task_progress`
- `auth_status`
- `rate_limit_event`
- `prompt_suggestion`

原因也很清楚：

1. 它们更强调时间顺序。
2. 它们变化频繁。
3. 它们更像脉搏，而不是最新快照。

所以 Claude Code 押注的不是“metadata 替代事件流”，而是：

- 该外化的状态外化
- 该保留时间性的信号继续走事件流

这是比“只做事件流”或“只做快照”都更成熟的状态观。

## 5. 为什么这条原则很重要

一旦承认“外化状态优于推断状态”，很多设计就会变得更容易理解：

1. 为什么 `WorkerStateUploader` 要做 merge、retry 和 last-value 语义。
2. 为什么 worker 初始化时要主动清掉 `pending_action` 和 `task_summary`。
3. 为什么 `onChangeAppState` 要变成统一 choke point，而不是分散手动通知。
4. 为什么 remote recovery 不只修连接，还要修 `worker_status` 与 metadata。

这些都不是零散技巧，而是在保护“宿主看到的真相”。

## 6. 苏格拉底式反问：研究者还会怎样把这条原则写丢

至少会在三处写丢：

1. 只写 `SDKMessage`，不写 `worker_status` / `external_metadata`。
2. 只写事件时间线，不写恢复后如何重建最新真相。
3. 只写 schema，不写 consumer subset 与兼容层。

只要犯这三类错误，蓝皮书就会再次把 Claude Code 写成：

- 一个消息流产品

而不是：

- 一个维护宿主真相的一体化 runtime

## 7. 判断状态面时的最小检查表

只要研究状态面，就必须先问四个问题：

1. 这段描述的是时间线，还是当前快照。
2. 这个状态是否已被源码显式外化。
3. 这个状态在恢复后如何被重新建立。
4. 哪些 consumer 实际会消费它，哪些不会。

如果这四问答不完整，这段分析就仍然停留在“宿主自己猜”的层次。

## 8. 一句话总结

Claude Code 比普通消息流工具更像 Agent runtime，一个重要原因就在于：它不把关键状态留给宿主去猜，而是主动把它们外化成可查询、可恢复、可同步的真相。
