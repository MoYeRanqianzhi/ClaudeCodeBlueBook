# 单一真相入口优于多处半真相实现

这一章回答四个问题：

1. 为什么 Claude Code 会把“单一真相入口”反复做成显式设计原则。
2. 为什么多处局部正确的实现，最后反而更容易制造系统性错误。
3. 为什么这条原则和 prompt 魔力、安全治理、恢复正确性都直接相关。
4. 从第一性原理看，Agent runtime 为什么比普通应用更需要权威真相面。

## 1. 先说结论

Claude Code 很稳定的一条设计哲学是：

- 单一真相入口优于多处半真相实现

这句话听起来像普通工程常识，但在 Claude Code 这里，它不是 DRY 口号，而是：

- runtime epistemology

也就是：

- 系统到底通过什么位置知道“现在真实发生了什么”

对 Agent runtime 来说，这件事尤其关键，因为它同时面对：

- 模型真相
- 用户真相
- 宿主真相
- 恢复真相
- 治理真相

如果这些真相分别由不同调用点各自维护，就算每个地方都“局部合理”，全局也很容易变成：

- 看起来有很多状态，实际上没有一个状态值得真正相信

更进一步说，Claude Code 正在贯彻一条更细的结构法则：

- 名词真相进入 schema 或 pure leaf
- 动词真相收口到 chokepoint 或 state machine

## 2. 多处半真相为什么比单点错误更危险

单点错误至少容易暴露。

多处半真相更危险，因为它常常表现为：

1. 每个模块都像对的。
2. 每个模块都只错一点点。
3. 但一旦跨平面联动，就开始互相打架。

例如：

- UI 以为 mode 已经变了。
- `external_metadata` 还保留旧 mode。
- 宿主又根据最后一条消息猜第三种状态。

这时真正的问题已经不是“哪个地方写错了”，而是：

- 系统失去了唯一可信的叙述位置

所以半真相的危险不只是 bug 变多，而是：

- 你开始不知道该信谁

## 3. 为什么 Claude Code 在这些位置反复收口真相

### 3.1 `mode` 必须有统一外化面

`onChangeAppState(...)` 之所以关键，不是因为它写法漂亮，而是因为 mode 变化牵动：

- permission engine
- CCR / SDK metadata
- 宿主前台显示
- plan / auto / default 等交互语义

这种状态如果允许不同入口各自通知一遍，迟早会漂。

### 3.2 tool pool 必须有统一组合面

`getAllBaseTools()`、`assembleToolPool(...)`、`mergeAndFilterTools(...)` 之所以重要，是因为工具池同时决定：

- 模型能力边界
- 策略可见性
- prompt 前缀稳定性

如果不同路径给模型不同工具列表，就不是“小差异”，而是：

- 模型正在看见不同世界

### 3.3 schema 必须有统一定义面

`coreSchemas.ts`、`sandboxTypes.ts` 之所以要成为 single source of truth，是因为一旦：

- IDE 类型
- settings 校验
- runtime payload

三者不共源，系统就会不断制造“文档正确、运行失败”或“运行接受、宿主不知情”的断层。

### 3.4 transcript / worktree 必须有统一恢复面

`sessionStorage.ts` 对 transcript path 与 worktree state 的处理说明：

- 恢复系统最怕的不是缺信息，而是信息来自多个互相不一致的位置

所以 Claude Code 宁可把这些真相压到更少入口，也不接受多个“都差不多对”的推断面。

## 4. 从第一性原理看，为什么 Agent runtime 特别需要单一真相入口

普通应用里，半真相常常先表现成：

- 某个按钮状态不对

Agent runtime 里，半真相更容易表现成：

- 模型看见的能力面和宿主看见的能力面不一致
- 恢复到的会话位置和真实落盘位置不一致
- 当前阻塞原因和用户以为的阻塞原因不一致

这些问题之所以更严重，是因为 Agent runtime 的核心不是一个静态页面，而是：

- 持续演进的执行轨迹

执行轨迹一旦建立在半真相上，后续每一步推理、每一步恢复、每一步治理判断都会叠加误差。

所以从第一性原理看，Agent runtime 的首要问题之一不是：

- 如何做更多功能

而是：

- 谁来定义权威当前态

## 5. 这和 prompt 魔力有什么关系

很多人会把 prompt 魔力理解成：

- system prompt 写得更强

Claude Code 的源码反而在提醒另一件事：

- prompt 强度首先依赖 runtime 各平面是否共享同一真相

因为如果：

- mode 漂移
- tool pool 漂移
- metadata 漂移
- worktree / session 路径漂移

那么模型拿到的上下文、宿主看到的状态、恢复后继续的世界，都会变成不同版本。

这时就算 prompt 文案本身不变，系统表现也会变弱。

所以更精确的说法是：

- prompt 魔力依赖于权威真相面，而不只是文字本身

## 6. 为什么这也和安全、治理、恢复有关

安全和治理同样依赖单一真相入口。

如果 policy blocking、sandbox config、permission mode、channel 输入边界没有各自清晰权威面，系统就会出现：

- 一个路径放行
- 另一个路径拒绝
- 第三个路径根本不知道前两者发生过什么

恢复系统也是一样。
resume、hook、subagent、worktree 一旦对当前会话位置使用不同推断，就会出现 split-brain。

所以“单一真相入口”不是某个局部编码风格，而是同时服务于：

- prompt 稳定
- 治理一致
- 恢复正确

的底层原则。

## 7. 苏格拉底式追问

### 7.1 如果一个状态有八条修改路径，而只有两条会同步给宿主，它还是被真正管理了吗

不是。
它只是被局部修改了。

### 7.2 如果 REPL、worker、adapter 各自拼一份工具池，只差一点点，模型看到的世界还是同一个世界吗

不是。
因为 prompt 前缀和能力边界已经不同了。

### 7.3 如果 transcript path 可以由不同地方各自猜测，恢复系统还能被称为“恢复”吗

很难。
那更像在多个近似世界之间碰运气。

### 7.4 为什么 Claude Code 宁可做 choke point，也不愿继续散点修补

因为散点修补解决的是这次调用路径。
choke point 解决的是未来仍然能保持同一真相。

## 8. 一句话总结

Claude Code 的成熟之处之一，在于它反复把关键状态收口成单一真相入口，因为 Agent runtime 最怕的不是单点出错，而是多处半真相同时成立。
