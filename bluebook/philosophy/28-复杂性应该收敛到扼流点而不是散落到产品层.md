# 复杂性应该收敛到扼流点而不是散落到产品层

这一章回答四个问题：

1. 为什么 Claude Code 明明有大文件，却仍然能表现出结构成熟度。
2. 什么样的复杂性应该集中到 chokepoint，而不是分散在 feature 层。
3. 为什么扼流点不是坏味道，而是 runtime 的责任中心。
4. 这对 agent 平台设计有什么第一性原理启发。

## 0. 代表性源码锚点

- `claude-code-source-code/src/state/onChangeAppState.ts:43-92`
- `claude-code-source-code/src/tools.ts:193-367`
- `claude-code-source-code/src/bridge/bridgeMessaging.ts:243-320`
- `claude-code-source-code/src/utils/task/framework.ts:48-71`
- `claude-code-source-code/src/services/api/promptCacheBreakDetection.ts:28-158`

## 1. 先说结论

Claude Code 的一条深层设计原则可以概括成：

- 全局复杂性应该收敛到少数扼流点，而不是散落到每个产品入口

因为在 agent runtime 里，真正需要全局一致的东西包括：

- mode sync
- tool visibility
- control request handling
- task state patching
- cache break 原因判断

这些逻辑如果被 feature 层各自实现，短期看灵活，长期只会变成：

- 多套半真的真相

## 2. 什么叫“应该收口的复杂性”

不是所有复杂性都该集中。

真正该集中的是那些一旦分散就会制造全局错觉的东西，比如：

1. 外部状态同步
2. 工具池真相
3. 协议请求分流
4. 状态 patch 规则
5. cache break 原因判断

这些对象的共同特征是：

- 它们不是某个功能私有的局部逻辑
- 它们会影响整个 runtime 的一致性

## 3. 为什么扼流点不是坏味道

很多工程讨论一看到 chokepoint 就会联想到：

- 单点过重
- 难维护
- 文件变大

但这只在一种情况下成立：

- chokepoint 收的是局部功能细节

如果它收的是全局不变量，情况就相反。

比如：

- `onChangeAppState(...)` 收的是 mode externalization 真相
- `assembleToolPool(...)` 收的是“模型到底看见什么工具”真相
- `handleServerControlRequest(...)` 收的是 inbound control 的语义真相

这类 chokepoint 的价值，不在“让文件更漂亮”，而在：

- 给全局不变量一个正式负责人

## 4. 为什么这和产品体验直接相关

如果 chokepoint 消失，最先坏掉的通常不是代码风格，而是产品体验：

- 宿主会看到 stale mode
- 模型会看到不该看的工具
- 服务端会等不到合法响应
- 任务状态会被旧快照覆盖

所以扼流点不是后台实现偏好，而是：

- 用户体验稳定性的隐藏前提

## 5. 苏格拉底式追问

### 5.1 为什么不把这些逻辑分别放回各个功能模块

因为那样做的代价不是“多写一点代码”，而是：

- 同一真相被写成多份版本

### 5.2 为什么这不会自动为大文件开脱

因为扼流点只能为“全局不变量收口”辩护，不能为“局部细节无限堆积”辩护。

所以判断标准不是：

- 文件大不大

而是：

- 它集中维护的是不是全局不变量

### 5.3 对平台构建者最直接的启发是什么

先问自己：

- 哪些逻辑一旦分散实现，就会让系统出现多套真相

然后先为这些逻辑找 chokepoint，再谈 feature 速度。

## 6. 一句话总结

Claude Code 的成熟不只来自更多模块，也来自它承认某些复杂性必须被集中维护；比起把真相散落到产品层，它更愿意让全局不变量收敛到少数扼流点。
