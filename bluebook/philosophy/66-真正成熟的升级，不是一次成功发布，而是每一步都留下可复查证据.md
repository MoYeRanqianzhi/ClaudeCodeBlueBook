# 真正成熟的升级，不是一次成功发布，而是每一步都留下可复查证据

这一章回答五个问题：

1. 为什么成熟升级不该被写成“一次成功发布”。
2. 为什么 Diff、阶段、观测与回退必须共同构成证据，而不是散落在不同文档里。
3. 为什么 Prompt、治理与结构三条线最终都需要同一类 rollout 证据接口。
4. 为什么没有可复查证据的升级，最终会退化成作者记忆与团队传说。
5. 这对 agent runtime 设计者意味着什么。

## 0. 代表性源码锚点

- `claude-code-source-code/src/utils/systemPrompt.ts:28-127`
- `claude-code-source-code/src/query/stopHooks.ts:84-120`

- `claude-code-source-code/src/utils/task/framework.ts:160-248`

这些锚点共同说明：

- Claude Code 的强，不在于某次发布“终于成功了”，而在于 prompt、治理、结构三条线都能被拆成渐进升级、可观测判断与可回退动作。

## 1. 先说结论

真正成熟的升级，不是：

- 一次看起来成功的上线

而是：

1. 每一层改动都能被点名。
2. 每一个阶段都有进入、退出与停止条件。
3. 每一次判断都能回到明确指标与证据。
4. 每一次失败都知道回退到哪一层。
5. 每一份记录都足够让后来者复跑同样的判断。

从第一性原理看，升级的真正成熟度不取决于：

- 最终是不是上线成功

而取决于：

- 中间每一步是否都留下可复查证据

## 2. 第一层：没有最小 diff，就没有真正的升级

很多团队把升级写成：

- 从旧系统到新系统

这听起来完整，却没有真正留下证据。

因为真正可复查的问题不是：

- 你想去哪里

而是：

- 这一阶段到底改了哪一小步

没有最小 diff，任何成功都无法被归因，任何失败也无法被精准回退。

## 3. 第二层：没有阶段主权，升级就只是时间顺序

成熟升级的第二个特征不是排了甘特图，而是每个阶段都能回答：

1. 谁在判断这一步是否能切。
2. 谁在接管这一步失败后的 fallback。
3. 哪些信号说明应该继续。
4. 哪些信号说明必须停止。

没有这些字段，所谓分阶段 rollout 只是：

- 把动作排在不同日期

而不是：

- 把风险排进不同制度层

## 4. 第三层：没有观测，成功只是叙事

一个升级最容易自欺的地方，是把结果写成：

- 看起来更稳了
- 成本似乎更低了
- 结构明显更清晰了

Claude Code 式更成熟的做法不是拒绝这些判断，而是要求它们先落到：

- cache、handoff、wrong deny、approval race、split truth、reconnect、stale state 这类正式观测对象

所以成熟升级的核心，不是更会讲结果，而是：

- 不允许结果脱离观测字段单独存在

## 5. 第四层：没有回退记录，就没有真正的成熟

很多系统把回退视为失败后的补救。

但更成熟的理解是：

- 回退不是 rollout 的尾巴，而是 rollout 的正式组成部分

因为一旦系统真的在做渐进升级，它就必须预先知道：

1. 哪些成果应被保留。
2. 哪一层可以单独回退。
3. 哪些症状一出现就不该再赌。

没有这一步，所谓升级成熟度最终只是：

- 对成功的自信

而不是：

- 对失败的诚实准备

## 6. 第五层：证据接口保护的，是后来者而不只是当前作者

当一套升级只存在于作者脑中的“这次大概是这么判断的”，它就还不成熟。

真正成熟的升级，会把判断路径外化成正式证据接口，让后来者一进来就能看到：

1. 这次升级的对象是谁。
2. 这次改了什么。
3. 这一步是怎么被批准的。
4. 这一步实际观测到了什么。
5. 这一步如果失败，应该退回哪里。

所以证据接口真正服务的不是：

- 一次项目同步

而是：

- 时间尺度上的维护连续性

## 7. 苏格拉底式追问

### 7.1 为什么“最终成功上线”不等于“升级成熟”

因为成功上线只能证明：

- 这次没炸

却不能证明：

- 你知道为什么没炸
- 你知道哪里还能单独回退
- 你知道后来者如何复跑同样判断

### 7.2 为什么 Prompt、治理与结构最终都需要同一类证据接口

因为它们都不是静态文档，而是在做同一件事：

- 渐进改变系统真相，同时不丢失继续判断与回退能力

### 7.3 为什么真正成熟的升级必然更偏爱 ABI，而不是更偏爱故事

因为故事只能帮助理解，ABI 才能帮助：

- 复查
- 回归
- 回退
- 迁移
- 接手

## 8. 对 Agent 设计者的启发

如果想学 Claude Code，最该抄走的不是：

- 更会写成功复盘

而是：

1. 让每次升级都先定义对象与最小 diff。
2. 让每个阶段都有明确主权、停止条件与 fallback。
3. 让每个结果判断都必须绑定正式观测字段。
4. 让回退成为升级设计的一部分，而不是事故补丁。
5. 把升级证据写成后来者也能消费的 ABI，而不是作者自己的现场记忆。
