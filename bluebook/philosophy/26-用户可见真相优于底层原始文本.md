# 用户可见真相优于底层原始文本

这一章回答四个问题：

1. 为什么 Claude Code 的前台搜索、选择、sticky prompt 都在围绕“用户可见真相”设计。
2. 为什么 raw transcript truth 并不总等于 render truth。
3. 这条原则怎样同时影响前台体验、宿主状态和 prompt 约束。
4. 对 agent IDE 的前台设计有什么一般化启发。

## 1. 先说结论

Claude Code 的前台做对了一件特别关键的事：

- 当 raw text 与用户实际看见的内容不一致时，前台语义服从“用户可见真相”，而不是服从底层原始文本。

这条原则同时出现在：

- transcript search
- selection / copy
- sticky prompt
- 特殊来源消息包装

所以它不是 UI 小技巧，而是前台控制面的根约束。

## 2. 为什么 raw truth 会和 render truth 分叉

在 Claude Code 里，底层消息并不总能直接等于用户看到的东西。

典型例子包括：

- 中断消息在 UI 里被替换成更短的标记
- `tool_result` 的模型侧序列化不等于前台渲染出的 native output
- `<system-reminder>` 会进入上下文，但不应污染前台搜索
- `task-notification` 和 channel message 有不同的来源语义

这说明前台如果直接索引原始字符串，就会制造伪命中与语义错位。

## 3. 为什么这条原则这么重要

因为一旦前台真相和用户感知脱节，系统会迅速出现三种破坏：

1. 搜索错位
2. 复制错位
3. 主语错位

这三种错位都会直接破坏人类对 agent runtime 的控制感。

## 4. 为什么这也和 prompt 设计有关

因为“用户可见真相”不仅是前台问题，它还会反向约束 prompt 结构。

只有当系统清楚区分：

- 什么是模型上下文中的辅助文本
- 什么是用户真正应该看到和操作的内容

前台与 prompt contract 才不会互相污染。

## 5. 苏格拉底式追问

### 问：为什么不能为了简单，直接让前台等于底层原始文本

答：因为底层原始文本里混入了大量只服务模型、缓存或恢复的结构。

### 问：为什么这不是“帮用户美化一下显示”

答：因为这里处理的不是美化，而是认知真相。用户搜索、复制、回看、判断来源时，依赖的都应该是同一套真相。

## 6. 一句话总结

Claude Code 的前台之所以可信，不是因为显示得更好看，而是因为它把 render truth 明确放在 raw transcript truth 之前。
