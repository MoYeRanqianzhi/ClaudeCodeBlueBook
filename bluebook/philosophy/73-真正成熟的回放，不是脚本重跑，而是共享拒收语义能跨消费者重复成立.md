# 真正成熟的回放，不是脚本重跑，而是共享拒收语义能跨消费者重复成立

回放系统最容易被误解成：

- 让脚本再跑一遍

但 Claude Code 式系统真正要保护的不是：

- 脚本是否可执行

而是：

- 同一 reject reason 是否能在宿主、CI、评审与交接里跨消费者重复成立

所以当蓝皮书走到 Evaluator Harness / Replay Lab 这一层时，真正该收束的判断是：

- 好回放不是更会重跑，而是更能证明共享拒收语义在不同消费者里仍然是同一个对象判断

## 1. 第一性原理

如果一个系统已经有：

- rule packet
- sample kit
- evaluator interface

却仍没有 replay harness，

那么它依旧只是：

- 会给出一次判定

而不是：

- 会证明这个判定在不同消费者处指向的是同一个 shared object

这意味着它仍然可能出现：

- 一次测试通过

但：

- 宿主、CI、review 与 handoff 其实在重复不同的局部真相

## 2. Prompt 的魔力，为什么最后要落到 replay

Prompt 最容易被误写成：

- 文案强

但更接近 Claude Code 真相的说法是：

- Prompt 的魔力来自 shared continuation 能在不同消费者里重复成立

如果同一 `prompt_object_id` 断裂时，宿主、CI、review 与 handoff 在回放里不能重复给出同一 reject reason，

那么所谓的“魔力”就会重新退回：

- 原文崇拜
- 绿灯崇拜
- 总结崇拜
- 摘要崇拜

所以真正成熟的 Prompt replay，不是：

- 再跑一次规则

而是：

- 再次证明 continuation 断裂时，所有消费者仍在围绕同一对象拒绝继续

## 3. 安全与省 Token，为什么最后要落到 replay

安全最容易被误写成：

- 拦截更强

省 token 最容易被误写成：

- 节省更多

但更接近 Claude Code 真相的说法是：

- 两者都在重放同一条“没有决策增益就不该继续”的语义

如果同一 `decision_window` 消失时，宿主、CI、review 与 handoff 在回放里不能重复给出同一 reject / upgrade semantics，

那么系统就会重新退回：

- 状态色
- 计数
- verdict
- 状态摘要

而不是共享：

- rollback object
- failure semantics
- next action

## 4. 源码先进性，为什么最后也要落到 replay

源码先进性最容易被误写成：

- 结构更清楚

但更接近 Claude Code 真相的说法是：

- 先进结构会让 split-brain 与 stale-writer 风险在不同消费者的回放里重复暴露成同一失败

如果同一 `structure_object_id` 断裂时，宿主、CI、review 与 handoff 不能重复给出同一 reject reason，

那么系统就会重新退回：

- 目录图
- 恢复成功率
- 作者说明

而不是共享：

- authoritative path
- recovery asset
- anti-zombie evidence
- rollback object

## 5. 为什么 Harness 比 Evaluator 更成熟

Evaluator 只能说明：

- 规则能判

Harness 才能说明：

- 判定能跨消费者重复成立

两者的差别，就像：

- 有一个裁决

和：

- 不同法院、不同复议、不同执行环节都在围绕同一部法律裁决

Claude Code 值得学的地方，不在于 gate 更多，

而在于：

- 它不断把“单点判断”推进成“跨消费者的重复成立”

## 6. 苏格拉底式自反

在你准备宣布“我们已经有成熟 replay harness 了”前，先问自己：

1. 我重放的是同一 reject semantics，还是几条相似但不同源的规则。
2. 如果删掉状态色、绿灯、目录图和作者说明，回放还能命中吗。
3. 宿主、CI、review 与 handoff 是否真的围绕同一 shared object 重复做出同一拒收。
4. drift 回归样本暴露的是 shared object 断裂，还是只暴露了几条局部告警。
5. 我是在证明脚本会跑，还是在证明共享拒收语义会跨消费者重复成立。
