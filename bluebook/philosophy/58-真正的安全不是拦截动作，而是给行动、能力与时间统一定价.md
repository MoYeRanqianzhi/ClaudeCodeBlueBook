# 真正的安全不是拦截动作，而是给行动、能力与时间统一定价

这一章回答四个问题：

1. 为什么 Claude Code 的安全系统不该被理解成“做更多拦截”。
2. 为什么 permission mode、审批、能力可见性、结果外置与 token budget 本质上是同一秩序。
3. 为什么这套秩序追求的是有效自由，而不是最大裸露面。
4. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 的安全观真正成熟的地方，不在于：

- 它更会拦截动作

而在于：

- 它给行动、能力、上下文和时间统一定价

换句话说，它在问的不是：

- 能不能做

而是：

1. 这件事该以什么价格被允许。
2. 这项能力该以什么价格进入当前可见面。
3. 这份结果该以什么价格留在主上下文里。
4. 这一轮继续下去是否还值得继续支付时间与 token。

## 2. 为什么这不是“更保守”

如果安全只是拦截器，那么它和行动力天然冲突。

但 Claude Code 不是这样做的。

它会同时安排：

- mode 挡位
- ask / deny / bypass 的不同费率
- classifier / hook / channel relay 的并行协商
- deferred tools 的按需暴露
- tool result externalization 的上下文外置
- token continuation / compact 的时间预算停损

所以它真正优化的是：

- 在有秩序的前提下，保住最长的高行动力区间

## 3. 苏格拉底式追问

### 3.1 `bypass` 等于最大自由吗

不是。

它只是提高了某些动作的通行级别，但 ask rule、content-specific ask、`requiresUserInteraction` 和 safety check 这些更高价格仍然存在。

所以 bypass 不是无价许可，而是：

- 受许可证约束的高阶通道

### 3.2 审批是不是纯打断

也不是。

Claude Code 里的审批更像多路径协商：

- 本地 UI
- SDK host
- bridge
- hook
- channel relay
- classifier

它要做的不是让系统停下来，而是让高价动作尽快被合法决定。

### 3.3 deferred / externalize 是在拿走能力吗

不是。

它们是在说：

- 能力不是免费进入可见面
- 结果也不是免费长期占据主上下文

这不是削弱系统，而是在保护主线程的行动空间。

### 3.4 为什么说这是一部资源宪法

因为 Claude Code 真正在分配的不是单个按钮，而是四类稀缺资源：

1. 能力
2. 时间
3. 注意力
4. 批准权

模型不是这些资源的主人，runtime 才是。

## 4. 对 Agent 设计者的启发

如果想学 Claude Code，最该优化的不是：

- 怎么把限制减到最少

而是：

1. 怎么给动作分级定价。
2. 怎么把协商做成低延迟正式协议。
3. 怎么让能力按需出现，而不是默认全量暴露。
4. 怎么让长结果和长回合及时退出最昂贵的上下文席位。

## 5. 一句话总结

Claude Code 的安全之所以成熟，不在于它更会阻止动作，而在于它把行动、能力、上下文和时间统一纳入定价秩序，让约束不再天然伤害高行动力。
