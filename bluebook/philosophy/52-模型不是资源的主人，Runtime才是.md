# 模型不是资源的主人，Runtime才是

这一章回答四个问题：

1. 为什么 Claude Code 不把能力、时间、注意力与权威的分配权交给模型。
2. 为什么真正成熟的 agent runtime 必须先有资源宪法。
3. 为什么安全、治理与省 token 在这里是主权问题，而不是优化问题。
4. 这对 agent 设计者意味着什么。

## 1. 先说结论

Claude Code 的一个根本立场是：

- 模型不是资源的主人，runtime 才是

模型可以在被授予的资源内推理和行动，但它不负责决定：

- 自己能看见多大的世界
- 自己能调用多少能力
- 自己还能继续多久
- 谁有资格改写这些边界

这些权力都属于 runtime。

## 2. 为什么这条原则很关键

如果把资源主权交给模型，系统很容易退化成：

- 先全给
- 再让模型自己节制

Claude Code 明显不相信这套。

它更相信：

1. runtime 先分配资源。
2. 模型在既定资源内工作。
3. 资源不足时，由 runtime 决定缩、延后、外置还是停止。

## 3. 苏格拉底式追问

### 3.1 如果模型更聪明，它就该拥有更大的资源主权吗

未必。

更聪明不等于更适合担任资源宪法的制定者。

### 3.2 为什么“让模型自己节制”不够

因为那会把：

- 安全
- 稳定性
- 成本
- 治理

全都外包给一次性推理。

### 3.3 Claude Code 更成熟的地方在哪里

在于它把资源分配写进了 runtime 结构，而不是写成对模型的希望。

### 3.4 这意味着什么

意味着真正强的 agent，不只是模型强，而是：

- runtime 对资源有主权

## 4. 对 Agent 设计者的启发

如果想学 Claude Code，最该学的不是：

- 如何让模型更自由

而是：

1. 如何让 runtime 明确谁能分配资源。
2. 如何把资源暴露做成正式协议。
3. 如何让 stop / continue / ask / deny 都成为宪法动作。
4. 如何避免把主权外包给单轮推理。

## 5. 一句话总结

Claude Code 的安全、治理和省 token 之所以能统一，是因为它坚持资源主权属于 runtime，而不是模型。
