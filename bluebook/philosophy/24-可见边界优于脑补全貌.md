# 可见边界优于脑补全貌

这一章回答四个问题：

1. 为什么研究 Claude Code 这样的公开源码镜像时，最重要的不是“补完缺失部分”，而是守住可见边界。
2. 为什么 workflow、channels、remote、前台控制面这些能力，即使内核未完全展开，也仍然能做出高质量判断。
3. 真正值得迁移的到底是隐藏实现，还是公开可见的 contract 与对象模型。
4. 这条原则对蓝皮书写法本身有什么约束。

## 1. 先说结论

研究公开源码镜像时，最危险的两种错误是：

1. 看见缺口，就用脑补把它补满。
2. 因为缺口存在，就把可见部分一并写得很浅。

Claude Code 最值得学习的，恰恰往往已经体现在那些公开可见的边界上：

- task type
- control schema
- transcript 路径
- worktree slug
- managed settings
- channel gating
- selection / search / sticky scroll 这样的前台语义

也就是说，真正的设计先进性常常首先体现在：

- 边界怎么被对象化

而不是：

- 隐藏代码里可能还有多厉害

## 2. 为什么“可见边界”本身就是高质量证据

一个系统如果真的成熟，即使你看不到全部内核，也应该已经能在边界上看见：

1. 对象模型
2. 失败语义
3. 恢复语义
4. 外化状态
5. 治理挂点

Claude Code 公开镜像正是如此。

例如：

- workflow 内核未完全展开，但 `local_workflow` 已经进入 task type、SDK event、transcript 路径与 worktree 清理语义。
- channels 的完整产品面仍受策略与 entitlements 约束，但 gate 顺序、allowlist 模型、session opt-in 语义都已公开可见。
- REPL 的全部前台行为未必一眼能看懂，但 transcript search、selection、sticky scroll 的 contract 已经足够说明它不是普通聊天壳。

所以边界不是“剩下来的碎片”，而是系统设计是否成熟的第一现场。

## 3. 为什么脑补全貌反而会破坏判断

因为一旦开始脑补，研究就会退化成：

- 用自己熟悉的实现方式，替别人填空

这会带来三个问题：

1. 误把缺失模块写成已证实实现。
2. 误把产品承诺和架构可能性混在一起。
3. 低估那些其实已经清楚可见、但不够显眼的边界设计。

换句话说，脑补全貌会让人错过真正可迁移的东西。

## 4. 真正值得迁移的往往不是“缺失内核”，而是“公开 contract”

对于平台构建者来说，最有价值的通常不是：

- 某个没公开的内部 helper

而是这些已经公开的设计骨架：

- `TaskType` 怎样定义后台对象
- control subtype 怎样定义宿主控制面
- `get_settings` 怎样定义配置三重真相
- channels 怎样定义组织治理与会话 opt-in 的层级关系
- selection / search / scroll 怎样定义前台可见真相

这些东西之所以有价值，是因为它们已经足够抽象，足够稳定，也足够说明作者的设计取向。

## 5. 从第一性原理看，为什么边界比内核更值得先研究

如果从第一性原理看，一个复杂系统最终必须回答的不是：

- 它内部有多少聪明技巧

而是：

- 它如何与外部世界接壤

因为真正决定系统可用性的，往往是这些接壤处：

- 人能否理解它
- 宿主能否接入它
- 策略能否治理它
- 状态能否恢复它

所以边界并不是次要层，而是系统存在方式本身。

## 6. 苏格拉底式追问

### 6.1 如果内核没公开，为什么还值得研究

因为研究不是为了获得“实现占有感”，而是为了提炼可迁移原则。

### 6.2 如果边界已经足够清楚，为什么还要强调缺口

因为边界清楚不等于内核已证实。  
严谨的写法必须同时做到：

- 充分解释可见部分
- 明确承认不可见部分

### 6.3 对蓝皮书来说，最稳的姿势是什么

最稳的姿势不是“替源码补完”，而是：

- 先写可见事实
- 再写保守推断
- 最后写哪些地方必须停止推断

## 7. 对蓝皮书写法的直接约束

后续每当遇到缺失模块，都应先问三次：

1. 可见边界已经能证明什么。
2. 这些边界是否已经足够支撑一个设计判断。
3. 哪些地方如果继续写下去，就会越过证据边界。

只有这样，蓝皮书才不会变成“替作者脑补源码”的二创文档。

## 8. 一句话总结

研究 Claude Code 这样的公开源码镜像时，真正值得信的不是脑补出来的全貌，而是那些已经公开可见、并且足以暴露设计取向的边界、contract 与对象模型。
