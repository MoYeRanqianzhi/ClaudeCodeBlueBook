# 统一第一性原理不等于单一预算实现

这一章回答四个问题：

1. 为什么 Claude Code 可以同时说“安全与省 token 是同一优化”，又不能被概括成“只有一个总预算器”。
2. 为什么真正统一的是第一性原理，而不是具体实现数量。
3. 为什么把 `policySettings`、sandbox、tool result replacement、autocompact、turn continuation 混成一个 budget，反而会失真。
4. 这对 Agent runtime 的设计叙述有什么启发。

## 1. 先说结论

Claude Code 这里最值得升级的认知是：

- 统一第一性原理不等于单一预算实现

它确实在用同一个第一性原理处理很多问题：

- 安全
- 治理
- 省 token
- 连续性

这个第一性原理可以压成一句话：

- 限制无序扩张，维护可治理边界

但真正运行时里，并不是：

- 一个预算器统治一切

而是：

- 多套预算机制服务于同一个第一性原理

## 2. 为什么“同一优化”不等于“同一个计数器”

把“安全与省 token 是同一优化”误听成“它们只有一个 budget system”，会立刻导致三个错误：

1. 把治理边界问题误写成 token 计数问题。
2. 把工具结果替换问题误写成 compaction 问题。
3. 把 turn continuation 问题误写成上下文管理问题。

Claude Code 的源码反而提醒你：

- 不同阶段会遇到不同对象
- 不同对象需要不同预算器

真正统一它们的不是变量名，而是：

- 它们都在防止系统进入无序膨胀

## 3. 为什么 `policySettings` 本身就是一种预算思想

`policySettings` 采用：

- first-source-wins

这说明它不是“给 merge pipeline 多加一层输入”，而是在说：

- 最高治理来源必须能直接定义边界真相

这已经是一种预算思想：

- 并不是所有输入都值得平等进入系统

如果连配置来源都不被治理，后面的：

- sandbox
- permissions
- token

都只是在更低层补救。

## 4. 为什么工具结果替换、autocompact 与 continuation 不是一回事

### 4.1 工具结果替换预算

它解决的是：

- 最肥的内容不该直接进入 prompt

### 4.2 autocompact / headroom 预算

它解决的是：

- 当前工作集必须给 compaction 和继续执行留足空间

### 4.3 turn continuation 预算

它解决的是：

- 这一轮该不该继续干，而不是转向总结或停机

这三者如果被混成一个“budget 数字”，系统很快就会失去解释力。

所以 Claude Code 的成熟之处恰恰在于：

- 它没有强行把所有预算折叠成一个机制

## 5. 从第一性原理看，为什么这更合理

如果系统真正要治理的是：

1. 输入面
2. 动作面
3. 工作集
4. 续写深度

那么天然就会需要不同的控制器。

强行单一化，反而会出现：

- 描述统一了
- 运行时不准确了

所以更成熟的表述应该是：

- Claude Code 有统一的反扩张原则
- 但保留了多个针对不同对象的预算实现

## 6. 苏格拉底式追问

### 6.1 如果一个危险远端设置能改代理、证书或 hooks，这个问题该交给 token 计数器吗

不应该。
它首先是输入治理问题。

### 6.2 如果单条工具输出就足够撑爆 prompt，为什么还假设上下文压缩器是第一责任人

不应该。
更早的对象级替换预算才是第一责任人。

### 6.3 如果当前问题是“要不要继续本轮”，为什么让 autocompact 来回答

也不对。
那是 turn continuation budget 的职责。

### 6.4 为什么 Claude Code 不把这些硬统一成一个机制

因为运行时里真正需要统一的是原则，不是形式。

## 7. 一句话总结

Claude Code 的成熟之处，不是用一个总预算器解释所有问题，而是用同一个第一性原理驱动多套预算实现，各自约束不同对象的无序扩张。
