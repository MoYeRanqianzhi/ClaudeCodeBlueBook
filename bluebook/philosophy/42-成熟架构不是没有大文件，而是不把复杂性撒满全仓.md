# 成熟架构不是没有大文件，而是不把复杂性撒满全仓

这一章回答四个问题：

1. 为什么“小文件很多”不等于成熟架构。
2. 为什么 Claude Code 的一些热点大文件反而证明了它对时序复杂度足够诚实。
3. 为什么伪模块化常常比显式 kernel 更危险。
4. 这对 agent runtime 的工程美学意味着什么。

## 1. 先说结论

Claude Code 的源码给出了一条很值得学的反常识结论：

- 成熟架构不是没有大文件，而是不把复杂性撒满全仓

很多工程讨论天然会把：

- 小文件
- 多模块
- 更多目录层级

当成成熟的近义词。

Claude Code 更像在提醒你：

- 真正危险的不是“复杂度被看见”
- 而是“复杂度被伪装后四处蔓延”

## 2. 为什么“大”本身不是问题

如果一个文件承担的是：

- query kernel
- 前台 orchestration shell
- 策略控制平面

那它天然就会比普通叶子模块更大。

问题不在于它大不大，而在于：

- 它是不是承担了一个应当被集中管理的运行时平面

`query.ts`、`REPL.tsx`、`services/mcp/config.ts` 这些文件之所以可以成立，就是因为它们处理的不是散装业务，而是：

- 少数合法的复杂性中心

## 3. 为什么伪模块化常常更糟

对 agent runtime 来说，很多复杂度是：

- 时序耦合
- 恢复耦合
- 协议耦合

这类东西如果被强行拆碎，常见结果不是更清晰，而是：

1. state 被分散到更多地方
2. continue / stop / recover reason 被隐藏
3. 真相入口从一个变成很多个
4. 边界不再显式，只剩隐式依赖

所以“看起来更模块化”，并不总是更成熟。

很多时候它只是：

- 把 kernel 伪装成了碎片

## 4. Claude Code 为什么反而显得成熟

因为它做了一个更诚实的选择：

- 把必须复杂的地方承认为 kernel / shell / chokepoint
- 把能纯化的地方继续压成 leaf modules 与 single source of truth

于是你会同时看到两件事：

1. 少数热点文件很大。
2. 大量叶子模块非常干净。

这正是成熟系统常见的形态。

## 5. 工程美学应该从“尺寸审美”升级到“不变量审美”

Claude Code 值得学的工程美学不是：

- 文件越小越好

而是：

- 不变量应该待在什么地方

更准确地说：

1. kernel 负责时序不变量
2. shell 负责编排不变量
3. chokepoint 负责策略与协议不变量
4. leaf module 负责纯 contract

只要这个分工是清楚的，大文件并不可耻。

相反，如果为了满足“看起来更小”的审美，把这些不变量拆散，系统反而会更脆。

## 6. 苏格拉底式追问

### 6.1 如果一个大文件真的是系统唯一的时序中心，拆碎它一定更好吗

未必。
很多时候只会让时序关系变得更隐蔽。

### 6.2 如果一个仓库完全没有热点文件，是不是就一定更成熟

也未必。
它也可能只是把复杂度均匀撒进了很多地方。

### 6.3 真正该怕的是什么

该怕的是：

- 半真相到处都是
- 没人知道谁是 single source of truth

不是“某个文件有 4000 行”本身。

## 7. 一句话总结

Claude Code 的架构成熟度，不在于消灭所有大文件，而在于它敢把合法复杂度集中承认为 kernel、shell 与 chokepoint，同时拒绝让这种复杂度无名无姓地撒满全仓。
