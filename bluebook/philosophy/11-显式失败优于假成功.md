# 显式失败优于假成功

这一章回答四个问题：

1. 为什么 Claude Code 在宿主控制面里反复选择“明确报错”，而不是“看起来像成功”。
2. 这种设计选择和第一性原理有什么关系。
3. 这对多宿主 Agent runtime 的稳定性意味着什么。
4. 蓝皮书后续应该如何用这条原则约束自己。

## 1. 先说结论

Claude Code 在宿主控制面上反复体现出同一条偏好：

宁可显式失败，也不要假成功。

这条偏好体现在很多地方：

- bridge outbound-only 模式对 mutable request 明确回 error
- bridge 对未知 control subtype 明确回 error
- direct connect 对未知 control request 明确回 error
- `RemoteSessionManager` 对未知 control request 明确回 error
- `StructuredIO` 在输入流关闭或 request abort 时明确 reject

这不是“编码风格”，而是运行时一致性原则。

## 2. 从第一性原理看，为什么假成功比失败更坏

一个 coding agent runtime 最怕的不是失败，而是状态错觉。

失败至少意味着：

- 宿主知道动作没有完成
- 用户还能重试
- 系统状态仍然可解释

假成功则意味着：

- 用户以为状态已经改变
- 宿主和 worker 对世界真相理解不一致
- 后续动作建立在错误前提之上

在本地聊天程序里，这种问题已经很危险；
在有 bridge、remote session、direct connect、MCP、subagent 的系统里，它会被放大成全局不一致。

## 3. Claude Code 在哪里体现了这条偏好

### 3.1 bridge outbound-only

如果 outbound-only 还假装成功处理 `interrupt`、`set_model`、`set_permission_mode`，远端 UI 就会误以为本地已经接受控制。

作者宁可回 error，也不制造这种错觉。

### 3.2 未知 subtype

bridge、direct connect、`RemoteSessionManager` 都不支持把未知 subtype 静默丢掉。

原因很直接：

- 服务端会一直等待 reply
- 最终连接会被杀掉

所以“明确失败”不只是用户体验问题，也是时序完整性问题。

### 3.3 cancel / abort

`control_cancel_request` 和本地 promise reject 的组合说明：

- 作者不想让宿主继续抱着一个已经失效的请求
- 也不想让 CLI 继续抱着一个已经失效的等待态

这同样是在阻止“看起来还在等，其实已经无效”的假成功。

## 4. 这条原则反过来解释了 Claude Code 的设计内涵

Claude Code 团队显然更重视：

- 状态真相
- 时序真相
- 边界真相

而不是：

- 表面上别报错
- 用户感觉流程没断
- 先糊弄过去

这其实非常符合 Agent runtime 的第一性原理。

因为一旦系统真的要长期工作，它最需要的不是“每一步都顺滑”，而是“每一步都可解释”。

## 5. 为什么这对蓝皮书也很重要

文档作者也很容易犯和系统相反的错误：

- 为了让结构看起来整齐，把边界写平
- 为了让能力看起来完整，把未支持之处轻轻带过
- 为了让叙述看起来顺，把 error 和 no-op 写得像已实现

这其实就是文档层面的“假成功”。

所以蓝皮书后续也应该遵循同一原则：

- 宁可明确写“当前未见实现”
- 宁可明确写“仅 schema 可见”
- 宁可明确写“仅某宿主支持”

也不要为了漂亮而把边界抹平。

## 6. 苏格拉底式反问：我们还会在哪里犯这种错

会，尤其在三类地方：

1. 看到 schema 很全时，把未实现子集写成已支持。
2. 看到 transport 通了时，把业务状态同步写成已成立。
3. 看到某路径能跑时，把 error / cancel / reconnect 边界写成理所当然。

如果不持续反问这些点，蓝皮书就会逐渐从工程研究变成“整洁但危险的概述”。

## 7. 后续写作约束

以后只要写到宿主协议、远程路径、权限仲裁，就必须先标三件事：

1. 成功路径是什么。
2. 显式失败路径是什么。
3. 如果没有这条失败路径，会产生什么假成功错觉。

只要第三问答不上来，这一段分析就还不够好。

## 8. 一句话总结

对 Agent runtime 来说，最危险的不是失败，而是假成功；Claude Code 之所以显得更成熟，正是因为它在很多地方都明确地站在“显式失败优于假成功”这一边。
