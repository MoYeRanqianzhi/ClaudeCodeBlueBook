# 可观测预算优于经验调优

这一章回答四个问题：

1. 为什么 Claude Code 的调优逻辑不该被理解成“prompt 写法经验”。
2. 为什么 token、block、工具膨胀、memory 膨胀都应该先被观测再被处理。
3. `ContextSuggestions` 为什么是设计哲学，而不是辅助 UI。
4. 这对 Agent 平台设计者有什么第一性原理启发。

## 1. 先说结论

Claude Code 还有一条值得单独抽出来的设计原则：

- 可观测预算优于经验调优

很多系统的调优方式是：

1. 觉得模型变笨了
2. 觉得上下文太长了
3. 于是开始压 prompt、删规则、换模型

Claude Code 更接近另一种方法：

1. 先看预算花在哪。
2. 先看系统卡在哪。
3. 再根据结构化观测决定改哪里。

这就是为什么它更像 runtime，而不像 prompt 手艺。

## 2. 为什么经验调优常常会越调越糟

因为经验调优通常把很多不同问题混成一个表象：

- token 快满
- 工具结果太吵
- memory 太胖
- 能力面太宽
- 系统正在等待批准

如果把这些都误认成：

- prompt 不够短

就会出现一种典型坏结果：

- 规则被砍薄了
- 真正的噪音还在
- block 也没被解除

所以经验调优的问题不只是“不够科学”，而是：

- 它会把不同层的问题误伤到同一个位置

## 3. 观测型预算为什么更成熟

Claude Code 选择先把预算对象显式暴露出来：

- `systemPromptSections`
- `systemTools`
- `mcpTools`
- `memoryFiles`
- `messageBreakdown`
- `attachmentsByType`
- `pending_action`
- `session_state_changed`

这意味着调优的第一步不是猜，而是分层定位。

所以系统真正成熟的地方不在：

- 有没有一套神秘好用的 prompt

而在：

- 你能不能知道哪一层正在制造噪音、哪一层正在制造阻塞

## 4. 为什么 `ContextSuggestions` 也是设计哲学

很多人会把建议系统看成辅助 UI。

Claude Code 这里更深一层的价值在于：

- 它把观测转译成动作

这背后有一条很重要的哲学：

- 观测如果不能导向动作，就仍然只是诊断

而 Claude Code 明显希望的是：

- 预算观测成为用户和宿主下一步决策的输入

这就是为什么 `ContextSuggestions` 不只是“贴心提示”，而是：

- 运行时把自己的约束翻译成人类可执行的建议

## 5. 为什么 block 也应纳入预算观

很多系统把预算只理解成 token 预算。

Claude Code 明显更宽：

- `requires_action`
- `pending_action`
- `permission_mode`

也属于当前 runtime 的预算真相。

因为一旦系统正在等待批准，真正稀缺的往往已经不是 token，而是：

- 审批
- 输入
- 模式变更

这意味着 Claude Code 的预算哲学本质上不是：

- 看上下文还剩多少

而是：

- 看当前什么资源正在成为系统瓶颈

## 6. 苏格拉底式追问

### 6.1 为什么不直接把 prompt 写短一点试试

因为你可能正在修错层。

### 6.2 为什么系统要自己给建议，而不是把图丢给用户

因为用户真正需要的不是更多图，而是：

- 下一步该改什么

### 6.3 对平台构建者最重要的启发是什么

不要只造预算器。

还要造：

- 预算的观测面
- 预算的解释面
- 预算到动作的翻译面

## 7. 一句话总结

Claude Code 的调优之所以更稳，不是因为作者更会写 prompt，而是因为它把调优从经验主义推进成了可观测预算驱动的决策过程。
