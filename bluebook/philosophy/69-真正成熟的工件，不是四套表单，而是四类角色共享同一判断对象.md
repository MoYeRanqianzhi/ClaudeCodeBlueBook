# 真正成熟的工件，不是四套表单，而是四类角色共享同一判断对象

这一章回答五个问题：

1. 为什么卡片、附件、评审卡与交接包都已经存在，仍不代表制度已经成熟。
2. 为什么真正成熟的工件，必须让宿主、CI、评审与交接继续围绕同一判断对象行动。
3. 为什么 Prompt、安全与省 token、源码先进性会在 artifact contract 层重新会合。
4. 为什么好的工件协议会减少未来判断分歧，而不是制造新的表单装饰。
5. 这对 agent runtime 设计者意味着什么。

这些锚点共同说明：

- Claude Code 更值得学的地方，不是它会生成更多工件，而是它持续把不同角色重新压回同一判断对象、同一窗口、同一边界。

## 1. 先说结论

真正成熟的工件，不是：

- 宿主有宿主卡
- CI 有 CI 附件
- 评审有评审卡
- 交接有 handoff package

而是：

1. 四类工件都从同一判断对象起手。
2. 四类工件都引用同一组权威字段。
3. 四类工件只在展开深度上不同，而不在真相来源上不同。
4. 原作者离开之后，这些工件仍然足以支撑继续、停止、回退与接手判断。

从第一性原理看，成熟工件真正做的是：

- 约束不同角色的未来判断

## 2. 第一层：工件存在，不等于判断对象已经共享

一个卡片存在，只说明：

- 有人生成了卡片

一个附件存在，只说明：

- 有人导出了附件

一个 handoff package 存在，只说明：

- 有人打包过材料

这些都不自动推出：

1. 当前对象仍然被同一 authority source 定义。
2. 当前窗口仍然被同一判断链消费。
3. 当前回退边界仍然被不同角色共同理解。

真正成熟的工件，不是检查点存在，而是：

- shared object 仍然存在

## 3. 第二层：四类角色不是四份真相，而是同一判断对象的四个投影

宿主真正该拿到的，不是：

- 一套宿主独有真相

而是：

- shared object 的当前状态投影

CI 真正该拿到的，不是：

- 一套 CI 独有真相

而是：

- shared object 的 hard-fail 与 drift 投影

评审真正该拿到的，不是：

- 一套 reviewer 独有真相

而是：

- shared object 的 judgement 投影

交接真正该拿到的，不是：

- 一套 handoff 独有真相

而是：

- shared object 的 continuation 投影

这四种差异都不应该制造：

- 四份局部真相

## 4. 第三层：Prompt、安全与源码先进性都在这里重新会合

Prompt 线最终共享的是：

- compiled request object

安全与省 token 线最终共享的是：

- governance decision object

结构线最终共享的是：

- structure recovery object

所以三条线虽然表面不同，但在 artifact contract 层，它们都在回答同一个问题：

- 不同角色能否沿同一对象继续判断

这就是为什么真正成熟的工件协议，会让 Prompt 魔力、安全定价与源码先进性重新会合。

## 5. 第四层：好的工件协议会减少未来分歧，而不是增加新的表单

如果一套工件协议只能做到：

- 让栏目更多

它的价值是有限的。

真正好的工件协议应该做到：

1. 让后来者无需脑补当前对象是谁。
2. 让不同角色无需争执权威字段来自哪里。
3. 让继续、停止、升级、回退这些动作有共同前提。
4. 让交接从“讲故事”退回“交对象”。
5. 让审读结论更稳定，而不是让表格更多。

换句话说：

- 工件协议的真正产出不是表单，而是更少的未来分歧

## 6. 第五层：工件协议本身就是运行时治理的一部分

Claude Code 最值得学的地方，不是：

- 它把材料整理得多漂亮

而是：

- 它持续把真相压成可继续消费的对象

因此工件协议不是事后文档层，而是运行时治理的一部分：

1. 工件头部字段决定团队先以什么为真。
2. 工件 hard contract 决定哪些缺失会直接阻断继续。
3. 工件 projection 决定不同角色能否沿同一对象展开。
4. 工件 handoff 决定后来者是否还能继续行动。

如果这些东西仍然依赖作者脑中的排序，制度就没有真正外化。

## 7. 苏格拉底式追问

### 7.1 为什么工件越多，制度反而可能越脆弱

因为如果它们不共享同一判断对象，工件越多，只会制造越多局部真相。

### 7.2 为什么成熟工件必须先审 shared header

因为 shared header 决定了：

- 大家到底是不是在围绕同一个对象说话

### 7.3 为什么这会成为新的高级工程判断

因为一旦工件开始约束不同角色的未来判断，系统就不再只是“生成了很多材料”，而是真正拥有了可演化的制度外壳。

## 8. 对 Agent 设计者的启发

如果想学 Claude Code，最该抄走的不是：

- 更多导出格式
- 更复杂工件中心
- 更漂亮仪表盘

而是：

1. 先让所有工件共享同一判断对象。
2. 再让不同角色只在展开深度上不同。
3. 再把最关键字段写成 hard contract。
4. 最后才考虑页面、导出形式与周边装饰。
