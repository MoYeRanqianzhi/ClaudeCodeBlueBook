# 真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、下一次重构仍有路

这一章回答五个问题：

1. 为什么公开镜像里的源码质量，不该先按目录整洁度和文件大小下判断。
2. 为什么合法复杂度中心比“更小文件”更接近 Claude Code 的真实先进性。
3. 为什么 `contract / registry / authoritative surface / adapter subset` 的边界可证，比“模块看起来更合理”更值钱。
4. 为什么真正成熟的源码，不只服务当前实现，还要保留下一次重构整套系统的可能性。
5. 这对 agent runtime 设计者意味着什么。

这些锚点共同说明：

- Claude Code 的源码质量真正强的地方，不是“静态结构更像教科书”，而是复杂度有合法席位、边界有正式证据、未来修改仍有继续抽离的路。

## 1. 先说结论

真正成熟的源码质量判断，不是：

- 文件更小
- 目录更碎
- 规则更多

而是：

1. 复杂度有没有被安放在合法复杂度中心。
2. 关键边界能不能被正式证明，而不是靠作者脑补。
3. later maintainer 能不能独立反对错误实现。
4. 系统每次增长后，是否仍保留下一次重构自己的可能性。

从第一性原理看，Claude Code 值得学的不是：

- “把仓库修剪得更漂亮”

而是：

- “把复杂度治理成可反驳、可演化、可继续收束的正式对象”

## 2. 第一性原理：先问复杂度有没有合法席位

源码质量最容易被误写成：

- 复杂度越少越好

但 Claude Code 更接近的真相是：

- 某些复杂度必须存在，而且必须被集中治理

这类复杂度包括：

1. query lifecycle
2. permission / control transaction
3. compaction / continuation
4. recovery / ingress / transport boundary

如果这些复杂度被硬拆散，系统不会更简单，只会变成：

- 真相分散在更多文件里
- 风险更晚暴露
- 故障更难被 later maintainer 反驳

所以成熟判断的第一问不该是：

- 这个文件为什么还这么大

而该是：

- 这份复杂度有没有合法席位，它是否在集中维护统一 invariant

## 3. 合法复杂度中心 vs 散落式复杂性

`query.ts`、`main.tsx`、这类热点文件不能机械判坏。

更稳的区分方式是：

### 合法复杂度中心

- 它承接的是 runtime kernel 级 invariant。
- 新能力沿着正式 contract 进入。
- 周围有 seam、single-source、state machine 把 blast radius 围住。
- later maintainer 能说清它为什么大、在保护什么。

### 散落式复杂性

- 它只是不断吸纳局部补丁。
- 真相、恢复、边界和副作用同时混在一起。
- 你只能说“这边好像很复杂”，却说不清它在维护什么制度对象。

真正成熟的源码质量判断，首先要学会把这两类东西分开。

否则“大文件批评”会退回成：

- 结构体感

而不是：

- 结构判断

## 4. 边界可证优于模块美学

如果一个系统只有：

- 好看的目录树
- 很细的模块分层

却没有先保住：

1. `contract truth`
2. `registry truth`
3. `authoritative surface truth`
4. `adapter subset truth`

那么它依旧只是在展示：

- 模块美学

而不是：

- 可验证边界

Claude Code 更成熟的地方在于：

- 对象先被 schema / type union / single-source file 承认；
- 能力再被 registry 暴露；
- 真相再被 authoritative surface 宣布；
- 最后才由某个 adapter 消费这个子集。

这意味着 later maintainer 不必先脑补：

- 哪一层才是真相

因为结构已经替他把证据梯度排好了。

更硬一点说，`authoritative surface` 不只包括 schema、registry 与 single-source file。

它还包括：

- 谁负责把当前状态显式回写给 host
- 哪些 event 只负责时间线，而不负责宣布 present truth

如果宿主必须靠回放事件自己拼 `permission_mode / worker_status / pending_action`，那权威面就还没有被真正固定。

## 5. 源码质量必须双审：dependency honesty 与 temporal honesty

很多系统只审：

- `dependency honesty`

也就是：

- import 图说不说真话
- seam 有没有被打穿
- leaf module / single-source 是否仍然薄而硬

但 Claude Code 这类 runtime 还必须同时审：

- `temporal honesty`

也就是：

- stale snapshot 会不会重写 fresh state
- recovery asset 会不会篡位为 authority
- stale writer 是否仍可达
- 过去对象会不会重新写坏现在
- event stream 与 state writeback 会不会被错误混成同一条真相通道

如果只看第一条，你会高估静态整洁度。
如果只看第二条，你会忽视依赖图已经开始撒谎。

真正成熟的源码质量判断必须同时回答：

1. 依赖图有没有说真话。
2. 时间图有没有说真话。
3. present truth 有没有被显式写回，而不是留给外围消费者自己猜。

更稳的 `temporal honesty` 通道矩阵至少应被压成下面这样：

| 通道 | 职责 | 不能越权做什么 |
|---|---|---|
| `writeEvent` | 写前台可见 timeline | 不能宣布 current truth |
| `writeInternalEvent` | 写 resume / internal-only history | 不能冒充 host state |
| `reportState / reportMetadata` | 写 present truth 与 host-facing state | 不能退回日志回放 |
| `flushInternalEvents before idle` | 在宣布 idle 前先结清 internal timeline | 不能让“看起来结束了”先于真实结清 |

## 6. 把这种证据梯度继续压成蓝皮书目录结构

如果蓝皮书目录结构也要服务“边界可证、复杂度有合法席位”，那它就不能按主题库存或写作先后分层，而应按证据梯度分层：

1. `philosophy/`
   - 只负责不可约判断，解释“为什么必须这么设计”。
2. `architecture/`
   - 只负责对象链、chokepoint 与 present-state protection 机制。
3. `api/`
   - 只负责 contract、schema、host-facing truth 与可接入边界。
4. `guides/`
   - 只负责把判断压成模板、清单与审读动作。
5. `playbooks/`
   - 只负责准入、收口、回归与拒收判决。
6. `casebooks/`
   - 只负责失真样本、伪证信号与误判机制。
7. `navigation/`
   - 只负责跨目录 judgment map，不重新兼任库存页或过程档案。
8. `docs/`
   - 只负责长时记忆、变更记录与研究过程，不回写正文。

这不是整理癖，而是同一个质量原则在文档结构里的延伸：

- 让判断去判断
- 让合同去承认边界
- 让手册去宣布 verdict
- 让案例去保存 distortion
- 让记忆去保存过程

## 7. Future Maintainer Rejectability 才是高阶质量信号

Claude Code 最稀缺的质量信号，不是：

- 作者现在写得更聪明

而是：

- 未来维护者被当成正式消费者

这会直接改变“好代码”的定义：

1. 风险必须在命名阶段就被看见。
2. 制度理由必须在注释里留下来。
3. seam 必须告诉后来者该从哪里继续抽离。
4. state machine / snapshot 必须让后来者能直接拒绝危险实现。
5. event stream 与 state writeback 的分工必须让后来者一眼看出“谁在讲时间线、谁在宣布当前真相”。

如果 later maintainer 仍然必须：

- 去问作者
- 追全仓实现
- 靠“经验感觉”猜哪里不能改

那么这份源码还远谈不上成熟。

## 8. 真正好的源码，必须让“下一次重构仍有路”

源码质量最容易被误写成：

- 这次重构做得很漂亮

但 Claude Code 更成熟的一层在于：

- 每次增长后，它仍保留下一次重构自己的可能性

这体现在：

1. `config / deps` seam 先把未来提纯 reducer / step extraction 的路留出来。
2. risk-bearing naming 先把 blast radius 写在代码表面。
3. single-source / leaf module 先把共享规则压成可继续抽离的小真相文件。
4. state machine / snapshot 先把时间边界正式化，避免以后只能围着布尔值修补。

换句话说，真正成熟的源码不是：

- 已经重构完成

而是：

- 仍然可继续重构

## 9. 公开镜像下为什么更该这样判断

公开镜像研究尤其容易退回两种坏解法：

1. 因为缺模块，所以脑补全貌。
2. 因为看见热点文件，所以直接判坏。

更稳的做法正好相反：

1. 先承认 gap，不替缺失部分说话。
2. 先对可见证据做分级，不让声明、注册、实装和缺口互相偷换。
3. 只对“当前可见且可辩护”的复杂度下判断。

这会让源码质量研究重新回到：

- 可验证、可反驳、可迁移

而不是：

- 体感式赞美

## 10. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的源码成熟度，最该先复制的是：

1. 让复杂度有合法席位，而不是把复杂度赶得到处都是。
2. 让边界有正式证据，而不是让读者靠追实现猜真相。
3. 同时治理 dependency honesty 与 temporal honesty。
4. 把 future maintainer 当成正式消费者，而不是等他来后再补注释。
5. 让每次增长都保留下一次重构的可能性，而不是把系统一次次锁死在当前形态。

而不是先复制：

- 更碎的目录
- 更多 helper
- 更漂亮的结构图

## 11. 苏格拉底式追问

在你准备宣布“这个仓库源码质量很高”前，先问自己：

1. 我看到的是合法复杂度中心，还是散落式复杂性。
2. 我能正式指出 contract、registry、authoritative surface 与 adapter subset 的边界吗。
3. 我是否同时审了依赖图诚实性和时间图诚实性。
4. later maintainer 如果拿不到当前作者，是否仍能独立反对错误实现。
5. 这次增长之后，系统是否仍保留下一次重构自己的路。
6. 我写下的是证据判断，还是目录和体量体感。

## 12. 一句话总结

真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、未来维护者可独立 reject，而且系统每次增长后仍保留下一次重构自己的可能性。
