# 真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、下一次重构仍有路

这一章回答五个问题：

1. 为什么公开镜像里的源码质量，不该先按目录整洁度和文件大小下判断。
2. 为什么合法复杂度中心比“更小文件”更接近 Claude Code 的真实先进性。
3. 为什么先判 authority 是否唯一、再谈模块美学，比“模块看起来更合理”更值钱。
4. 为什么真正成熟的源码，不只服务当前实现，还要保留下一次重构整套系统的可能性。
5. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

真正成熟的源码质量判断，不是：

- 文件更小
- 目录更碎
- 规则更多

而是：

1. 复杂度有没有被安放在合法复杂度中心。
2. 关键边界能不能被正式证明，而不是靠作者脑补。
3. `failure semantics` 是否已经进入 architecture doctrine，而不是只留在下游异常说明。
4. later maintainer 能不能独立反对错误实现。
5. 系统每次增长后，是否仍保留下一次重构自己的可能性。

从第一性原理看，Claude Code 值得学的不是：

- “把仓库修剪得更像教科书”

而是：

- “把复杂度治理成可反驳、可演化、可继续收束的正式对象”

如果只先记源码质量前门的一句话，也只记这句：

- 源码质量判断不是文件更小，而是把复杂度收进合法复杂度中心，正式保护 `one writable present` 这条 claim，让 later maintainer 仍能独立 `reject` 错误实现并保留下一次重构之路。

再压一层，成熟源码质量至少同时守住两种诚实：`dependency honesty` 让依赖图不撒谎，`temporal honesty` 则保证旧时间证据在重新经过正式确认前不能越权写回 `present`。

这里还应再多记一句：

- 旧 `contract truth / registry truth / orchestration shell / danger surface / authoritative surface` 这些词只保留为检索兼容标签。
- `102` 继续独占 evidence ladder、降格规则、speaking-right protocol 与 `change-risk record` 字段；`87` 只回答为什么合法复杂度中心、失败诚实、later-maintainer rejectability 与 refactorability 才是源码质量的第一性原理。
- 若一条 advancedness reading 还做不到一跳落到 `architecture/README` 的 landing card，或落到像 `architecture/84` 那样已填过的对象卡，本页就只保留 why，不把 why 读成已完成对象验证。

如果一个源码质量解释还不能同时回答“复杂度为什么有合法席位、现在为什么只有一个可写现在、后来者为什么还能独立拒绝错误实现”，它就还停在目录美学或模块体感层。

如果把 advancedness reading 再压成 later maintainer 用来防 overread 的最小 review standard，也只先记三句：

- 手里只有目录观感、文件大小或模块体感时，只配停在 `layout praise`；先回本页补 why，不要先夸 advanced。
- why-proof 已站住、但对象承接仍缺时，只配停在 `provisional why / provisional reading`；这说明 why 已站住，不等于 object proof 已成立。
- why 已经足够把你送到对应对象卡时，才配离场去对象层继续证明；why 页自己不代做对象检查。

更硬一点说，这组 review standard 真正保护的不是谦虚语气，而是 speaking-right discipline：没有 why-proof 就不配讲 advanced；有 why 但对象承接未明时，也不能越级读成 object truth。
在 `public-evidence only` 条件下，这条 speaking-right discipline 还应再收紧一层：本页最强只到 why-proof 与 review standard；凡想继续判断 `writer truth / effect ceiling / first retreat layer / fail-closed seam` 是否已成立，都应离开本页，转去 `guides/102` 或 `architecture/README`。
更稳一点说，`87` 的离场条件不是“已经讲够 advancedness”，而是 reader 开始追问 object truth；一旦问题从“为什么值得这样判断”切到“哪张对象卡已经站住”，本页就该停止，不能在 why owner 里偷补 object proof。

## 2. 第一性原理：先问复杂度有没有合法席位

源码质量最容易被误写成：

- 复杂度越少越好

但 Claude Code 更接近的真相是：

- 某些复杂度必须存在，而且必须被集中治理

这类复杂度包括 query lifecycle、permission / control transaction、compaction / continuation 与 recovery / ingress / transport boundary 这类必须集中治理的 runtime 不变量。

只有同时仲裁 `dependency / temporal / failure / current-truth writeback` 多轴不变量的热点，才配被叫作合法复杂度中心；只是很多需求碰巧堆在一起的大文件，不算中心，只算补丁池。

更硬一点说，合法复杂度中心之所以合法，不是因为它集中，而是因为它同时守住三件事：现在仍只有一个可写现在，错误会先在低签字权层失败，未来维护者仍能据此发动局部 veto 并继续抽离。做不到这三点的热点，只是债务汇流点，不是成熟内核。

如果这些复杂度被硬拆散，系统不会更简单，只会变成：

- 真相分散在更多文件里
- 风险更晚暴露
- 故障更难被 later maintainer 反驳

所以成熟判断的第一问不该是：

- 这个文件为什么还这么大

而该是：

- 这份复杂度有没有合法席位，它是否在集中维护统一 invariant

## 3. 合法复杂度中心 vs 散落式复杂性

`query.ts`、`main.tsx`、这类热点文件不能机械判坏。

更稳的区分方式是：

### 合法复杂度中心

- 它承接的是 runtime kernel 级 invariant。
- 新能力沿着正式 contract 进入。
- 周围有 seam、single-source、state machine 把 blast radius 围住。
- later maintainer 能说清它为什么大、在保护什么。

### 散落式复杂性

- 它只是不断吸纳局部补丁。
- 真相、恢复、边界和副作用同时混在一起。
- 你只能说“这边好像很复杂”，却说不清它在维护什么制度对象。

真正成熟的源码质量判断，首先要学会把这两类东西分开。

否则“大文件批评”会退回成：

- 结构体感

而不是：

- 结构判断

## 4. 边界可证优于模块美学

如果一个系统只有：

- 好看的目录树
- 很细的模块分层

却没有先保住“谁配写现在、谁只配消费现在、later maintainer 怎样继续独立拒绝错误实现”这条更稳的判断：

那么它依旧只是在展示：

- 模块美学

而不是：

- 可验证边界

Claude Code 更成熟的地方在于：later maintainer 不必先脑补哪一层才是真相，因为结构已经把 later-maintainer veto 写成了可辩护的边界。

更硬一点说，`authoritative surface` 只保留为公开镜像里的兼容检索词；真正先判的仍是“现在是否只有一个可写现在”，而不是在 why 页展开对象层通道、回写责任与 truth plane 矩阵。

## 5. 源码质量必须三审：dependency honesty、temporal honesty 与 failure honesty

很多系统只审 `dependency honesty`，但 Claude Code 这类 runtime 还必须同时审 `temporal honesty`。这里真正值钱的原则只有一句：旧时间证据可以存在，但在重新经过正式确认前，不能直接跨时写回 `present`。
更硬一点说，成熟源码质量还要求错误先在低签字权层失败：projection、恢复资产与 provisional claim 必须先于 present writeback 暴露失真，让 later maintainer 的第一条 veto 发生在降格层，而不是发生在当前真相已被改坏之后。
更稳一点说，这条 veto 默认就该先落在 `projection / provisional claim / recovery asset / stale capability` 这类降格层；只有先把 promotion 资格打掉，later maintainer 才不必等 current truth 真被写坏后才补救。
同理，公开 artifact 的 overclaim 也该先在低签字权层失败：只要 `signer`、`effect ceiling` 或 `release-surface shaping` 还分不清，就先按 `public artifact ceiling` 降格，而不是等 current truth 或产品承诺被误写后再补救。

如果只看第一条，你会高估静态整洁度。
如果只看第二条，你会忽视依赖图已经开始撒谎。
如果不看第三条，你会把 architecture doctrine 错写成“异常最后总能被补救”。

真正成熟的源码质量判断必须同时回答：依赖图有没有说真话、时间图有没有说真话、失败语义有没有被写成正式对象、present truth 有没有被显式保护。更稳一点说，本页不再展开具体通道与对象矩阵；那是 `architecture/README` 的对象层职责。这里只保留判断句：时间线、现在与展示层必须严格拆开，否则权威面就会重新退回日志回放和 UI 猜测。

## 6. why-proof 为什么必须可交接

源码质量线真正该共享给 later maintainer 的，不是第二套目录法，而是一份可交接的 why-proof。

`87` 的正职不是代行 promotion gate，也不是替对象层发摘要卡；它只负责把 later maintainer 最该先守住的判断写成可复述、可反对、可继续继承的三条命题。

如果 later maintainer 变更前还答不出这三条，他拿到的就还不是质量判断，只是阅读体感。

这份 why-proof 也只必答三条：

1. `合法复杂度中心`
   - 这条风险到底落在哪个 hotspot kernel，而不是漂在目录观感里
2. `one writable present`
   - 这次变更是否仍只承认一个 present writer，而不是让恢复资产或投影面越权
3. `later-maintainer veto`
   - 拿不到作者时，later maintainer 是否仍能只凭结构指出当前判断为什么该被拒收

如果把 `87` 这一步继续压成最短产物，也只剩一份 `why-proof`：

- 为什么这个风险属于合法复杂度中心
- 为什么这里仍只有一个可写现在
- 为什么 later maintainer 仍有正式 veto
- 可交接的 `why-proof` 还必须让 later maintainer 一跳回到对应的 object owner（例如 `hotspot kernel / writer claim state / first retreat layer` 所在页）；如果这三点仍要跨多页追作者意图，交出去的就还是阅读体感，不是源码质量证明。
- 更硬一点说，这份 `why-proof` 还必须让 later maintainer 在 first no 之后立刻知道该去哪个 `fail-closed seam` owner，而不是先把“为什么危险”重新翻译成修复计划；答不出这一点，why-proof 仍停在解释层，没有真正进入 change control。

## 7. Future Maintainer Rejectability 才是高阶质量信号

Claude Code 最稀缺的质量信号，不是：

- 作者现在写得更聪明

而是：

- 未来维护者被当成正式消费者

这会直接改变“好代码”的定义：

1. 风险必须在命名阶段就被看见。
2. 制度理由必须在注释里留下来。
3. seam 必须告诉后来者该从哪里继续抽离。
4. state machine / snapshot 必须让后来者能直接拒绝危险实现。
5. event stream 与 state writeback 的分工必须让后来者一眼看出“谁在讲时间线、谁在宣布当前真相”。

这里的 `future maintainer rejectability` 也不只发生在热点代码里：凡是会改变运行时行为、权限裁决、上下文注入或代理装配的 `operator artifacts`，例如 `CLAUDE.md`、hooks、skills、agent defs 与 permission schemas，也都至少该被 later maintainer 视作需要标注 `surface candidate / first local veto cue` 的高风险 review surface。若这些工件只能被阅读，不能触发第一条局部拒收，它们就还只是作者体感，不算成熟源码质量。
更稳一点说，在当前 `public-evidence only` 条件下，这里只先提醒 later maintainer 不要把 foundation/context 工件误读成 signer 或 `current-truth writer`；更细的 `effect ceiling` 与 authority truth 仍需 owner 页补证。
公开 docs 至少已经给出这组工件的部分 `effect ceiling` 线索：managed settings 最高且不可被 user/project 覆盖，`CLAUDE.md` 与 auto memory 只被当作 context，subagents 拥有独立 context window 与独立 permissions，async hooks 不能回头阻断已发生的动作；同一仓库下所有 worktrees 还共享同一个 auto memory 目录，且 `autoMemoryDirectory` 不能由 project settings 指定。但在当前 worktree，这些仍只构成 public evidence；若缺 live mirror 或 object-level writer proof，它们都只先算 `surface candidate`，不自动 promotion 为 formal veto surface。
再压一层，源码先进性还要求显式暴露 `negative capability`：later maintainer 不只要知道哪里还能扩展，也要知道哪些 foundation/context artifacts 至少不该被顺手升格成 signer 或 `current-truth writer`。答不出这条非升级边界，advancedness 仍只是“可扩展体感”，不是可拒收结构。

这里也该把最低标准写得更硬：

- `future maintainer rejectability` 的最低标准，不是“看懂作者设计”，而是拿不到作者时仍能只凭结构给出正式 veto。
- 这不是阅读协议，而是 change control：结构必须给 later maintainer 保留局部 veto，而不是逼他接受作者体感。

如果 later maintainer 仍然必须：

- 去问作者
- 追全仓实现
- 靠“经验感觉”猜哪里不能改

那么这份源码还远谈不上成熟。

如果继续把 `future maintainer rejectability` 再压成更硬的对象级酸测试，也只剩三问：

1. 改坏 current truth 的第一条 `local veto cue`，later maintainer 能不能不问作者就点出来。
2. 第一退回层能不能直接指向最近的 `fail-closed seam`，而不是先把 why 重新翻译成修法。
3. 这条 advancedness claim 若删掉 `one writable present / first retreat layer` 两项 noun，是否还剩下除“结构好看”之外的东西。

任一问答不上，就先把 advancedness 降格成 `layout praise / provisional reading`，不继续夸成熟。

## 8. 真正好的源码，必须让“下一次重构仍有路”

源码质量最容易被误写成：

- 这次重构做得很漂亮

但 Claude Code 更成熟的一层在于：

- 每次增长后，它仍保留下一次重构自己的可能性

这体现在：

1. `config / deps` seam 先把未来提纯 reducer / step extraction 的路留出来。
2. risk-bearing naming 先把 blast radius 写在代码表面。
3. single-source / leaf module 先把共享规则压成可继续抽离的小真相文件。
4. state machine / snapshot 先把时间边界正式化，避免以后只能围着布尔值修补。

若一个 seam 还答不出第一条局部 veto 和第一退回层，它就还只是阅读入口，不配被叫作可继承的 `next-refactor entry`。
更稳一点说，later maintainer 若还答不出第一条 `local veto cue` 落在哪、第一退回层先回哪层，就该先把 advancedness claim 降格成 `layout praise`，而不是继续宣称这里已经保住了下一次重构入口。

换句话说，真正成熟的源码不是：

- 已经重构完成

而是：

- 仍然可继续重构

这本质上是在保护未来变更的 option value，而不是赞美这次重构本身。

## 9. 公开镜像下为什么更该这样判断

公开镜像研究尤其容易退回两种坏解法：

1. 因为缺模块，所以脑补全貌。
2. 因为看见热点文件，所以直接判坏。

更稳的做法正好相反：

1. 先承认 gap，不替缺失部分说话。
2. 先对可见证据做分级，不让声明、注册、实装和缺口互相偷换。
3. 只对“当前可见且可辩护”的复杂度下判断。

这会让源码质量研究重新回到：

- 可验证、可反驳、可迁移

而不是：

- 体感式赞美

## 10. 对 Agent Runtime 设计者的直接启发

如果你想复用 Claude Code 的源码判断，最该先复用的是：

1. 让复杂度有合法席位，而不是把复杂度赶得到处都是。
2. 让边界有正式证据，而不是让读者靠追实现猜真相。
3. 同时治理 dependency honesty 与 temporal honesty。
4. 把 future maintainer 当成正式消费者，而不是等他来后再补注释。
5. 让每次增长都保留下一次重构的可能性，而不是把系统一次次锁死在当前形态。

而不是先复制：

- 更碎的目录
- 更多 helper
- 更漂亮的结构图

## 11. 苏格拉底式追问

在你准备下源码质量判断前，先问自己：

1. 我看到的是合法复杂度中心，还是散落式复杂性。
2. 我能正式指出 authority 边界与证据分层吗。
3. 我是否同时审了依赖图诚实性、时间图诚实性和 failure honesty。
4. 第一条 reject signal 会先从哪一层暴露；是 contract、writeback、failure semantics 还是 `hotspot kernel / mirror gap discipline`。
5. later maintainer 如果拿不到当前作者，是否仍能独立反对错误实现。
6. 这次增长之后，系统是否仍保留下一次重构自己的路。
7. 我写下的是证据判断，还是目录和体量体感。

## 12. 一句话总结

真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、未来维护者可独立 reject，而且系统每次增长后仍保留下一次重构自己的可能性。
