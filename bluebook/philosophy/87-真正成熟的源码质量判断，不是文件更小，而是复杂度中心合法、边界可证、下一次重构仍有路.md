# 真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、下一次重构仍有路

这一章回答五个问题：

1. 为什么公开镜像里的源码质量，不该先按目录整洁度和文件大小下判断。
2. 为什么合法复杂度中心比“更小文件”更接近 Claude Code 的真实先进性。
3. 为什么 `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline` 的边界可证，比“模块看起来更合理”更值钱。
4. 为什么真正成熟的源码，不只服务当前实现，还要保留下一次重构整套系统的可能性。
5. 这对 agent runtime 设计者意味着什么。

这些锚点共同说明：

- Claude Code 的源码质量真正值钱的地方，不在“静态结构更像教科书”，而在复杂度有合法席位、边界有正式证据、未来修改仍有继续抽离的路。

## 1. 先说结论

真正成熟的源码质量判断，不是：

- 文件更小
- 目录更碎
- 规则更多

而是：

1. 复杂度有没有被安放在合法复杂度中心。
2. 关键边界能不能被正式证明，而不是靠作者脑补。
3. `failure semantics` 是否已经进入 architecture doctrine，而不是只留在下游异常说明。
4. later maintainer 能不能独立反对错误实现。
5. 系统每次增长后，是否仍保留下一次重构自己的可能性。

从第一性原理看，Claude Code 值得学的不是：

- “把仓库修剪得更像教科书”

而是：

- “把复杂度治理成可反驳、可演化、可继续收束的正式对象”

如果只先记源码质量前门的一句话，也只记这句：

- 源码质量判断不是文件更小，而是把复杂度收进合法复杂度中心，用 `canonical ladder + failure semantics` 正式保护 `one writable present`，让 later maintainer 仍能独立 `reject` 错误实现并保留下一次重构之路。

这里还应再多记一句：

- 旧 `contract truth / registry truth / orchestration shell / danger surface / authoritative surface` 这些词只保留为检索兼容标签；源码质量判断的主语仍固定为 `canonical ladder + failure semantics`。

如果把源码质量前门继续压成最短公式，只剩两句：

1. `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`
2. `dependency honesty + temporal honesty + failure honesty`

这条线最短的 reject trio 也只认：

1. `layout-first drift`
2. `recovery-sovereignty leak`
3. `surface-gap blur`

如果一个源码质量解释还没压到这两句，就还停在目录美学或模块体感层。

## 2. 第一性原理：先问复杂度有没有合法席位

源码质量最容易被误写成：

- 复杂度越少越好

但 Claude Code 更接近的真相是：

- 某些复杂度必须存在，而且必须被集中治理

这类复杂度包括：

1. query lifecycle
2. permission / control transaction
3. compaction / continuation
4. recovery / ingress / transport boundary

只有同时仲裁 `dependency / temporal / failure / current-truth writeback` 多轴不变量的热点，才配被叫作合法复杂度中心；只是很多需求碰巧堆在一起的大文件，不算中心，只算补丁池。

如果这些复杂度被硬拆散，系统不会更简单，只会变成：

- 真相分散在更多文件里
- 风险更晚暴露
- 故障更难被 later maintainer 反驳

所以成熟判断的第一问不该是：

- 这个文件为什么还这么大

而该是：

- 这份复杂度有没有合法席位，它是否在集中维护统一 invariant

## 3. 合法复杂度中心 vs 散落式复杂性

`query.ts`、`main.tsx`、这类热点文件不能机械判坏。

更稳的区分方式是：

### 合法复杂度中心

- 它承接的是 runtime kernel 级 invariant。
- 新能力沿着正式 contract 进入。
- 周围有 seam、single-source、state machine 把 blast radius 围住。
- later maintainer 能说清它为什么大、在保护什么。

### 散落式复杂性

- 它只是不断吸纳局部补丁。
- 真相、恢复、边界和副作用同时混在一起。
- 你只能说“这边好像很复杂”，却说不清它在维护什么制度对象。

真正成熟的源码质量判断，首先要学会把这两类东西分开。

否则“大文件批评”会退回成：

- 结构体感

而不是：

- 结构判断

## 4. 边界可证优于模块美学

如果一个系统只有：

- 好看的目录树
- 很细的模块分层

却没有先保住这条更稳的 truth ladder：

1. `contract`
2. `registry`
3. `current-truth surface`
4. `consumer subset`
5. `hotspot kernel`
6. `mirror gap discipline`

那么它依旧只是在展示：

- 模块美学

而不是：

- 可验证边界

Claude Code 更成熟的地方在于：

- 对象先被 schema / type union / single-source file 承认；
- 能力与装配顺序再被 registry 与 hotspot kernel / chokepoint 组织；
- 当前真相再被 `current-truth surface` 宣布；
- 最后才由 host / adapter 只消费自己配看的 consumer subset；
- 危险改动面与 reject path 则继续暴露在 `hotspot kernel -> mirror gap discipline` 的投影里，而不是藏在目录观感后面。

这意味着 later maintainer 不必先脑补：

- 哪一层才是真相

因为结构已经替他把证据梯度排好了。

更硬一点说，`authoritative surface` 只保留为公开镜像里的兼容检索词；真正先判的仍是 `current-truth surface` 与 `writeback seam`。

它还包括：

- 谁负责把当前状态显式回写给 host
- 哪些 event 只负责时间线，而不负责宣布 present truth

如果宿主必须靠回放事件自己拼 `permission_mode / worker_status / pending_action`，那权威面就还没有被真正固定。

## 5. 源码质量必须三审：dependency honesty、temporal honesty 与 failure honesty

很多系统只审：

- `dependency honesty`

也就是：

- import 图说不说真话
- seam 有没有被打穿
- leaf module / single-source 是否仍然薄而硬

但 Claude Code 这类 runtime 还必须同时审：

- `temporal honesty`

也就是：

- stale snapshot 会不会重写 fresh state
- recovery asset 会不会篡位为 authority
- stale writer 是否仍可达
- 过去对象会不会重新写坏现在
- event stream 与 state writeback 会不会被错误混成同一条真相通道

这里还要再多记一句：

- 时间诚实不是旧 `snapshot / pointer / recovery asset` 不能存在，而是它们在重新经过 `fresh-read / adopt / relink` 之前，只能作为 `evidence / projection` 存在，不能直接跨时写回 `present`

如果只看第一条，你会高估静态整洁度。
如果只看第二条，你会忽视依赖图已经开始撒谎。
如果不看第三条，你会把 architecture doctrine 错写成“异常最后总能被补救”。

真正成熟的源码质量判断必须同时回答：

1. 依赖图有没有说真话。
2. 时间图有没有说真话。
3. 失败语义有没有被写成正式对象，而不是散落在异常文案、UI 告警与口头约定里。
4. present truth 有没有被显式写回，而不是留给外围消费者自己猜。

更稳一点说，本页在这里不再展开 `writeEvent / reportState / writeback seam` 的对象矩阵；那是 `architecture/README -> 60 / 84` 的对象层职责。`87` 只保留判断句：时间诚实至少要把下面三种 truth 严格拆开：

1. `event truth`
   - 只回答“发生了什么”。
2. `current truth`
   - 只回答“现在是什么”。
3. `display truth`
   - 只回答“人现在看到什么”。

三者一旦互相代写，权威面就会重新退回日志回放和 UI 猜测。

如果三者发生冲突，更稳的复核顺序也只该是：

1. `display truth`
   - 只负责触发怀疑。
2. `current truth`
   - 只负责给出 present verdict。
3. `event truth`
   - 只负责补证，不负责代写现在。

如果 UI 或日志回放直接越过这条顺序，later maintainer 看到的就不再是 current truth，而是第二叙述者；若要继续核对具体通道、回写责任与 first retreat layer，统一回 `architecture/README`。

## 6. 文档分工只复用同一条证据顺序

如果蓝皮书文档分工也要服务“边界可证、复杂度有合法席位”，那它就只能复用同一条证据顺序；目录只是承载体，不是源码质量判断对象。

更准确地说，本页在文档结构层只保留一句提醒：同一源码质量判断只允许由同一证据层首答；`87` 在这里只保留三条源码质量式自校：

1. 同一源码质量判断是否仍由同一证据层首答，而不是同一问题在多个目录首答。
2. `contract / method / verdict / sample / memory` 是否仍各守自己的证据职责。
3. 兼容标签与研究过程是否已经被降格为附注，而没有改写正文判据。

这不是目录治理，而是防止不同证据层互相代写同一个源码质量判断：

- 定义对象的层先定义对象
- 方法层只给审读步骤
- verdict 层只给准入结论
- 样本与记忆层只保存观察与过程

更硬一点说，证据层不互相代写，不是写作卫生，而是 `future maintainer rejectability` 在证据层的实现。

## 7. Future Maintainer Rejectability 才是高阶质量信号

Claude Code 最稀缺的质量信号，不是：

- 作者现在写得更聪明

而是：

- 未来维护者被当成正式消费者

这会直接改变“好代码”的定义：

1. 风险必须在命名阶段就被看见。
2. 制度理由必须在注释里留下来。
3. seam 必须告诉后来者该从哪里继续抽离。
4. state machine / snapshot 必须让后来者能直接拒绝危险实现。
5. event stream 与 state writeback 的分工必须让后来者一眼看出“谁在讲时间线、谁在宣布当前真相”。

这里也该把最低标准写得更硬：

- `future maintainer rejectability` 的最低标准，不是“看懂作者设计”，而是拿不到作者时仍能只凭结构指出哪条 `event / snapshot / transport` 越权成了 authority，并立刻说出第一退回层。
- 这不是阅读协议，而是 change control：结构必须给 later maintainer 保留局部 veto 与第一退回层。

如果 later maintainer 仍然必须：

- 去问作者
- 追全仓实现
- 靠“经验感觉”猜哪里不能改

那么这份源码还远谈不上成熟。

## 8. 真正好的源码，必须让“下一次重构仍有路”

源码质量最容易被误写成：

- 这次重构做得很漂亮

但 Claude Code 更成熟的一层在于：

- 每次增长后，它仍保留下一次重构自己的可能性

这体现在：

1. `config / deps` seam 先把未来提纯 reducer / step extraction 的路留出来。
2. risk-bearing naming 先把 blast radius 写在代码表面。
3. single-source / leaf module 先把共享规则压成可继续抽离的小真相文件。
4. state machine / snapshot 先把时间边界正式化，避免以后只能围着布尔值修补。

换句话说，真正成熟的源码不是：

- 已经重构完成

而是：

- 仍然可继续重构

这本质上是在保护未来变更的 option value，而不是赞美这次重构本身。

## 9. 公开镜像下为什么更该这样判断

公开镜像研究尤其容易退回两种坏解法：

1. 因为缺模块，所以脑补全貌。
2. 因为看见热点文件，所以直接判坏。

更稳的做法正好相反：

1. 先承认 gap，不替缺失部分说话。
2. 先对可见证据做分级，不让声明、注册、实装和缺口互相偷换。
3. 只对“当前可见且可辩护”的复杂度下判断。

这会让源码质量研究重新回到：

- 可验证、可反驳、可迁移

而不是：

- 体感式赞美

## 10. 对 Agent Runtime 设计者的直接启发

如果你想复用 Claude Code 的源码判断，最该先复用的是：

1. 让复杂度有合法席位，而不是把复杂度赶得到处都是。
2. 让边界有正式证据，而不是让读者靠追实现猜真相。
3. 同时治理 dependency honesty 与 temporal honesty。
4. 把 future maintainer 当成正式消费者，而不是等他来后再补注释。
5. 让每次增长都保留下一次重构的可能性，而不是把系统一次次锁死在当前形态。

而不是先复制：

- 更碎的目录
- 更多 helper
- 更漂亮的结构图

## 11. 苏格拉底式追问

在你准备下源码质量判断前，先问自己：

1. 我看到的是合法复杂度中心，还是散落式复杂性。
2. 我能正式指出 canonical ladder：`contract / registry / current-truth surface / consumer subset / hotspot kernel / mirror gap discipline` 的边界吗。
3. 我是否同时审了依赖图诚实性、时间图诚实性和 failure honesty。
4. 第一条 reject signal 会先从哪一层暴露；是 contract、writeback、failure semantics 还是 `hotspot kernel / mirror gap discipline`。
5. later maintainer 如果拿不到当前作者，是否仍能独立反对错误实现。
6. 这次增长之后，系统是否仍保留下一次重构自己的路。
7. 我写下的是证据判断，还是目录和体量体感。

## 12. 一句话总结

真正成熟的源码质量判断，不是文件更小，而是复杂度中心合法、边界可证、未来维护者可独立 reject，而且系统每次增长后仍保留下一次重构自己的可能性。
