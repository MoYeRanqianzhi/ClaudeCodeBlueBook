# 提示词魔力来自运行时而非咒语

这一章回答四个问题：

1. 为什么 Claude Code 的 prompt 会让人感觉“有魔力”。
2. 这种魔力为什么不来自某个神秘 system prompt，而来自 runtime 结构。
3. prompt、tool、cache、角色、上下文经济学是怎样一起工作的。
4. 研究 prompt 魔力时，如何避免重新落回“神 prompt 故事”。

## 1. 先说结论

Claude Code 的 prompt 之所以强，不是因为某一句话写得神，而是因为 prompt 在这里已经不再只是文本，而是被写成了运行时合同。

这份合同至少同时约束六类对象：

1. 主语合同：
   - coordinator、worker、fork、proactive agent 不是语气风格，而是不同主语的正式席位。
2. 动作合同：
   - tool、hook、MCP、prompt command / skill 共同定义当前回合哪些动作配进入世界。
3. 升级与拒收合同：
   - 何时 fork、何时 ask user、何时必须验证、何时禁止偷看子代理结果，都是显式升级路径。
4. 缓存与继续合同：
   - attachment、deferred tools、prompt cache sharing 决定同一个世界如何跨 turn 继续，而不是每轮重写。
5. 交接合同：
   - compact、summary、memory、handoff 之后，后续消费者拿到的仍应是同一个世界，而不是二次转述出来的平行版本。
6. 谱系与投影合同：
   - `display truth`、`protocol truth` 与 `handoff truth` 不是三份彼此独立的文本，而是同一条 message lineage 在不同消费者前的三种投影。

所以它的“魔力”更准确地说，不是“更会说”，而是：

- `display truth`、`protocol truth` 与 `handoff truth` 仍沿同一条 `message lineage` 继续成立；`request object` 更像一次消费外壳，而不是跨显示、协议与交接都持续存活的真相核

## 2. 从第一性原理看，为什么 prompt 本来就不该是单段文案

一个能长期工作的 coding agent，至少要被告知：

1. 目标
2. 边界
3. 工具
4. 协作方式
5. 成本约束

如果这些东西都靠一段统一文案硬塞进去，就会立刻遇到三个问题：

1. 动态信息太多，prompt 很快失稳
2. 不同角色需要的约束根本不同
3. 当前回合、继续回合与交接回合会各自长出不同主语

Claude Code 的做法是反过来：

- 把不稳定的内容拆成动态 section 和 attachment
- 把不同角色分成不同 system prompt
- 把行动约束写进工具 prompt 与角色 prompt
- 把 cache、compact 与 handoff 的约束提前写进同一条继续合同

这比“写一段更长的神 prompt”成熟得多。

## 3. Claude Code 在哪里体现了这种魔力

### 3.1 coordinator prompt

`coordinatorMode.ts` 不是在鼓励“多开几个 agent”，而是在定义：

- 你是 coordinator，不是 worker
- worker 结果不是对话对象
- 你必须先综合理解，再写后续 worker prompt

这让 `role contract` 与 `epistemic boundary` 一起成立：

- coordinator 有综合责任
- worker 只对分配给自己的世界片段负责
- 主线程不能把未验证的子代理输出提前冒充当前真相

### 3.2 AgentTool prompt

`AgentTool/prompt.ts` 里最关键的内容，不是语气，而是：

- 何时 fork
- 何时 fresh subagent
- prompt 要写背景，不要写“based on your findings”
- 不要偷看 output_file

这直接把 delegation 写成了 `handoff contract`：

- 交给子代理的上下文必须自包含
- 交接时不能偷渡未验证事实
- 子代理结果必须沿正式回链进入主线程，而不是靠作者记忆口述回灌

### 3.3 compact prompt

`services/compact/prompt.ts` 开头就先写：

- TEXT ONLY
- DO NOT call tools
- 工具调用会浪费唯一 turn

这说明 compact prompt 管理的不是“摘要语气”，而是 `lawful forgetting` 与 `continuation pricing`：

- 哪些信息该迁出主工作集
- 哪些动作在这个回合根本不配被重新收费
- 怎样把世界压缩后仍交给同一个后续消费者

## 4. 为什么这种 prompt 会显得“更聪明”

“更聪明”其实只是表象。更准确的说法是，它做了三件普通 prompt 很少同时做到的事：

1. 把误区提前显式写出来
2. 把不同任务形态切成不同 prompt
3. 把成本模型写进 prompt

例如：

- fork 共享 prompt cache
- agent list 不内联，避免 cache bust
- MCP instructions 用 delta attachment，避免 late connect 污染 prefix
- queued command / attachment 在回流时仍保住同一个 `source_uuid`

这意味着模型接收到的不只是 instructions，还接收到：

- 当前谁有资格定义世界
- 当前哪些动作配进入上下文席位
- 当前哪些历史可以继续、哪些必须被合法遗忘

## 5. 这反过来解释了“为什么不是 prompt-first”

Claude Code 表面上当然很依赖 prompt。

更稳的读法还必须承认：

- prompt 本身是被 runtime 装配的
- tools 决定可行动作
- permission 决定可行动边界
- cache 决定 prompt 怎样拆
- attachment 决定动态信息怎样进入

所以真正 first 的不是 prompt，而是 runtime；prompt 只是 runtime 的可读合同，而不是 runtime 的替代品。

## 6. 苏格拉底式反问：研究者还会怎样把 prompt 魔力写坏

至少会在六处写坏：

1. 只摘 system prompt 句子，不写角色分层。
2. 只说 prompt 强，不写 tools / permissions / cache / attachment。
3. 把 prompt 的效果归因于“文案聪明”，不写结构约束。
4. 把 fork/coordinator/worker 的 prompt 混成一套。
5. 不追问 `display truth / protocol truth / handoff truth` 是否还活在同一个 `request object` 里。
6. 以为三份 truth 的正确性来自措辞一致，而不是来自 message lineage 没有脱链。

只要这样写，蓝皮书就会再次回到“玄学 prompt 评论”。

## 7. 一句话总结

Claude Code 的 prompt 魔力，不来自某句神秘咒语，而来自它把 prompt 写成了 `message lineage + role contract + cache boundary + handoff truth` 共同约束的运行时合同。
