# 真正先进的源码，会把批评路径提前编码进结构

这一章回答五个问题：

1. 为什么源码先进性不该只被写成“结构更好看”。
2. 为什么 authoritative surface、contract-first、recovery asset 与 anti-zombie 协议都在服务同一件事。
3. 为什么先进源码会把未来的反对意见、修改路径与退出路径提前写进结构。
4. 为什么未来维护者如果没有被当成正式消费者，所谓先进性迟早会退化。
5. 这对 agent runtime 设计者意味着什么。

## 0. 代表性源码锚点

- `claude-code-source-code/src/utils/QueryGuard.ts:1-121`
- `claude-code-source-code/src/query/config.ts:1-45`
- `claude-code-source-code/src/query/deps.ts:1-40`
- `claude-code-source-code/src/utils/plugins/pluginPolicy.ts:1-20`
- `claude-code-source-code/src/bridge/replBridgeTransport.ts:13-116`
- `claude-code-source-code/src/utils/cleanupRegistry.ts:1-21`
- `claude-code-source-code/src/utils/task/framework.ts:160-248`

这些锚点共同说明：

- Claude Code 的源码先进性，不只是减少错误，而是把“哪里该被反对、哪里该被收口、哪里该被恢复”提前编码进结构。

## 1. 先说结论

真正先进的源码，不是：

- 更少人会吐槽它

而是：

1. 关键真相有明确的 authoritative surface。
2. 新能力先进入 contract，再进入实现。
3. 共享规则先进入 leaf module 或 single source file。
4. transport 差异先被收进 shell。
5. stale write、recovery failure 与 zombie risk 先被命名成正式故障模型。
6. 未来维护者一进来就知道哪里该质疑、哪里不能随便改。

从第一性原理看，源码先进性的更深层本质不是：

- 当前结构更优雅

而是：

- 批评路径已经被结构提前编码

## 2. 第一层：权威面让反对意见有落点

如果一个系统没有明确的 authoritative surface，任何人想质疑“当前真相到底是什么”时，都只能：

- 在多处半真相里来回猜

先进源码更成熟的地方在于，它会主动告诉后来者：

- 如果要怀疑一处真相，请先来这里

这并不是让结构更中心化，而是让：

- 反对意见有正式落点

## 3. 第二层：合同优先让批评先指向制度，不先指向实现

当一个新能力先被写进 contract、schema、registry 或权威面，后来者在质疑它时，首先能问的是：

- 这条制度是否成立

而不是：

- 这一堆散落实现里哪一个 if 写错了

所以 contract-first 的价值不只是阅读更快，而是：

- 让批评先落在制度对象，而不是落在局部实现噪音

## 4. 第三层：恢复与 zombie 治理让批评覆盖时间维度

很多源码看上去结构整齐，但只要跨 await、跨进程、跨恢复路径就开始说谎。

Claude Code 更高级的地方在于，它会把：

- fresh-state merge
- generation / epoch
- stale drop
- recovery asset

都写成正式结构要求。

这意味着后来者在批评一段写法时，不只会问：

- 它现在能不能跑

还会问：

- 它会不会在未来某次恢复里把旧对象复活

## 5. 第四层：命名、注释与 seam 让批评不依赖作者本人

如果关键反对意见只存在于作者脑中，那么所谓先进性只是：

- 作者暂时还没离开

Claude Code 值得学的地方正在于，它经常把风险直接写进：

- 命名
- 注释
- seam
- 小而硬的单一真相文件

这说明它的先进性不是“作者更聪明”，而是：

- 批评路径已经脱离作者本人

## 6. 第五层：未来维护者消费者是批评路径的正式使用者

所谓“未来维护者也是正式消费者”，不是一句温柔口号，而是很硬的一条工程判断：

- 如果未来维护者进来时看不出哪里该先质疑、哪里该先核对、哪里该先回滚，那么当前结构还不够先进

真正先进的源码，应该在第一眼就让后来者知道：

1. 哪些对象是权威面。
2. 哪些边界是 transport shell。
3. 哪些状态属于 recovery asset。
4. 哪些 finally、merge 或 snapshot 写法可能导致 zombie。

## 7. 苏格拉底式追问

### 7.1 为什么先进源码不能只被写成“更漂亮”

因为漂亮只能帮助欣赏，不能帮助：

- 反对错误设计
- 约束危险修改
- 指导恢复与回滚

### 7.2 为什么 authoritative surface 与 contract-first 都在服务“批评路径”

因为它们都在回答：

- 如果系统开始说谎，后来者该先怀疑哪里

### 7.3 为什么未来维护者消费者是高级工程判断，而不是团队文化口号

因为只有把批评路径正式编码进结构，维护者才不需要依赖作者记忆去猜：

- 哪些改动会破坏制度真相

## 8. 对 Agent 设计者的启发

如果想学 Claude Code，最该抄走的不是：

- 更漂亮的目录树

而是：

1. 给关键真相建立明确权威面。
2. 让新能力先进入合同面，再进入实现面。
3. 把 transport、恢复与 stale-state 风险写成正式结构对象。
4. 用命名、注释、seam 与单一真相文件把批评路径外化。
5. 把未来维护者当成第一个会质疑你设计的人，并让源码提前回答这些质疑。
