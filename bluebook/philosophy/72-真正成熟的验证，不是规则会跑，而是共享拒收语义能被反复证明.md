# 真正成熟的验证，不是规则会跑，而是共享拒收语义能被反复证明

验证系统最容易被误解成：

- 规则终于能跑了

但 Claude Code 式系统真正要保护的不是：

- 是否有一个 evaluator 能返回结果

而是：

- 不同消费者是否能在重复执行中反复证明自己共享的是同一条拒收语义

所以当蓝皮书走到 Rule Sample Kit / Evaluator 这一层时，真正该收束的判断是：

- 好验证不是规则跑通一次，而是 shared reject semantics 能在宿主、CI、评审与交接里被重复证实

## 1. 第一性原理

如果一个系统已经有：

- shared contract
- validator
- rule ABI

却仍没有 sample kit / evaluator，

那么它依旧只是：

- 会定义规则

而不是：

- 会证明规则在不同消费者那里仍然指向同一个对象

这意味着它仍然可能出现：

- 宿主告警
- CI 通过
- reviewer 放行
- handoff 拒收

四种结果各自合理，却不共享同一对象判断。

## 2. Prompt 的魔力，为什么最后要落到可重复验证

Prompt 最容易被误写成：

- 文案更强

但更接近 Claude Code 真相的说法是：

- Prompt 的魔力来自 `compiled request continuity` 可以被不同消费者反复证明

如果同一 `prompt_object_id` 在宿主、CI、review 与 handoff 里不能稳定触发同一 reject reason，

那么所谓的“魔力”就会重新退回：

- 原文崇拜
- 绿灯崇拜
- 总结崇拜
- 摘要崇拜

所以真正成熟的 Prompt 验证，不是：

- 规则跑过

而是：

- 同一 continuation 断裂时，不同消费者会反复给出同一拒收

## 3. 安全与省 Token，为什么最后要落到可重复验证

安全最容易被误写成：

- 规则更严

省 token 最容易被误写成：

- 规则更省

但更接近 Claude Code 真相的说法是：

- 两者都在验证“没有决策增益就不该继续”

如果同一 `decision_window` 在宿主、CI、review 与 handoff 里不能稳定触发同一 reject reason，

那么系统就会重新分裂成：

- 状态色替代判断
- 计数替代仲裁
- verdict 替代失败语义
- 状态摘要替代 rollback object

所以真正成熟的治理验证，不是：

- 有一个 object-upgrade gate

而是：

- 同一 decision gain 消失时，不同消费者会反复做出同一拒收

## 4. 源码先进性，为什么最后也要落到可重复验证

源码先进性最容易被误写成：

- 结构漂亮

但更接近 Claude Code 真相的说法是：

- 先进结构会让 split-brain、anti-zombie 与 handoff 风险在不同消费者那里重复暴露为同一失败

如果同一 `structure_object_id` 在宿主、CI、review 与 handoff 里不能稳定触发同一 reject reason，

那么系统就会重新退回：

- 目录图
- 恢复成功率
- 作者讲解

而不是：

- authoritative path
- recovery asset
- stale writer 证据
- rollback object

## 5. 为什么 Sample Kit / Evaluator 比“规则会跑”更成熟

规则会跑，只能说明：

- 语法被执行了

反复验证，才能说明：

- 语义被共享了

两者的差别，就像：

- 有一条法律

和：

- 不同法院、不同执法者、不同复议流程都在执行同一部法律

Claude Code 值得学的地方，不在于它有更多 gate，

而在于：

- 它不断把“规则定义”推进成“规则可被不同消费者重复证明”

## 6. 苏格拉底式自反

在你准备宣布“我们已经有成熟 evaluator 了”前，先问自己：

1. 这些样例共享的是同一 reject reason，还是只是几份相似配置。
2. 如果删掉状态色、绿灯、图表和作者说明，evaluator 还会命中吗。
3. evaluator 输出的 `reject_reason` 是否真的能被宿主、CI、review 与 handoff 共享。
4. rewrite hint 修复的是 shared object continuity，还是只修格式和措辞。
5. 我是在证明规则能跑，还是在证明共享拒收语义真的存在。
