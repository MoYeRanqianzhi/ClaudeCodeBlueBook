# 当前真相必须可恢复，而不是事后可观测

这一章回答四个问题：

1. 为什么 Claude Code 不把状态回写理解成遥测或调试附属层。
2. 为什么“能解释过去发生了什么”仍然不够，还必须“能恢复当前真相”。
3. 为什么清旧状态、丢弃 stale 写入、坚持 last-value 语义反而是更成熟的做法。
4. 这对 agent runtime 的第一性原理意味着什么。

## 1. 先说结论

Claude Code 在状态设计上的更深原则是：

- 当前真相必须可恢复，而不是事后可观测

也就是说，它不满足于：

- 事后能在日志里看懂发生了什么

它更要求：

- 崩溃后、恢复后、换 transport 后，系统仍能继续站在正确的当前状态上

这就是为什么它会如此认真地维护：

- `worker_status`
- `requires_action_details`
- `external_metadata`

这些对象。

## 2. 观测和恢复不是同一件事

observability 主要解决的是：

- 我们能不能解释过去

durability 解决的是：

- 系统重来一次时，还能不能继续正确运行

这两者当然相关，但绝对不等价。

一个系统完全可能：

- 日志很全
- 指标很多
- 仪表盘很漂亮

但一旦断线、崩溃、恢复，就完全不知道：

- 当前是否仍在阻塞
- 上次卡在哪个权限请求
- permission mode 现在究竟是什么
- 当前 model 有没有被远程切走

Claude Code 明显不接受这种“会解释过去，但恢复不了现在”的设计。

## 3. 为什么清旧状态和丢弃 stale 写入更成熟

很多系统天然会偏好一种表面连续性：

- 尽量不断
- 尽量不丢
- 尽量维持看起来一直在线

Claude Code 更成熟，因为它知道：

- 假连续性比短暂中断更危险

所以它会主动做三件事。

### 3.1 重启时清理旧阻塞真相

旧 worker crash 后残留的：

- `pending_action`
- `task_summary`

如果不在新 worker 初始化时清掉，就会把旧阻塞伪装成当前阻塞。

### 3.2 恢复窗口宁可丢弃 stale 控制消息

`401` 恢复期间，旧 epoch 的：

- `control_request`
- `control_response`
- `control_cancel_request`
- `result`

继续写出，只会制造“看起来还在线”的假象。

### 3.3 状态必须按最后值收敛

`WorkerStateUploader` 的 last-value-wins 和 RFC 7396 merge，说明系统真正要保护的是：

- 当前真相最终收敛正确

而不是：

- 每一条旁路上报都尽量完整保留

## 4. 苏格拉底式追问

### 4.1 如果一个状态只存在于日志里，系统恢复时真的拥有它吗

没有。

它最多只是“事后看得见”。

### 4.2 如果恢复期间保留 stale 写入能让界面更连续，这就是更好吗

不一定。

如果代价是假在线、假阻塞、假运行中，那反而更差。

### 4.3 如果一个状态不能被显式清除，它还能被称为权威真相吗

很难。

因为旧值会持续污染当前判断。

### 4.4 一个 agent runtime 最该优先保护的是日志完整，还是当前真相正确

显然应该是：

- 当前真相正确

## 5. 对 Agent 设计者的启发

如果想做可恢复的 agent runtime，最该学的不是：

- 多打点
- 多留历史

而是：

1. 把当前状态写成可回读的权威快照。
2. 让旧状态能被显式清除。
3. 让恢复窗口拒绝 stale 写入继续污染当前真相。
4. 把“能恢复现在”放在“能解释过去”之前。

## 6. 一句话总结

Claude Code 的状态设计之所以成熟，不在于它观测得更细，而在于它坚持让当前真相可恢复，让系统在失败之后仍然拥有同一份可信的现在。
