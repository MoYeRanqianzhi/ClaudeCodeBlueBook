# 反竞争条件意识优于局部功能正确

这一章回答四个问题：

1. 为什么 agent runtime 里的“局部正确”常常仍然会产生全局错误。
2. Claude Code 怎样把 race condition 当成设计对象，而不是事后 bug。
3. 为什么 duplicate、stale、replay、双来源消息都属于同一个问题。
4. 这对多 Agent、远程宿主与长会话系统有什么一般化启发。

## 1. 先说结论

Claude Code 反复体现的一条更深原则是：

- 在长会话、多 Agent、宿主桥接系统里，反竞争条件意识比局部功能正确更重要

因为一个局部看似正确的操作，很可能在全局上造成：

- duplicate response
- stale overwrite
- replay delivery
- 双来源消息污染
- 假成功

所以真正成熟的代码，不只是“这次调用返回了对的值”，而是：

- 它预先假设其他路径也可能同时在写同一真相

## 2. 什么叫“局部正确但全局错误”

最典型的例子就是：

- 我拿到一个 task 快照，把它更新后写回去

局部看完全合理。

但如果中间发生过：

- 任务完成
- 任务恢复
- 状态替换

那这次写回就会把旧快照重新盖到新状态上。

Claude Code 因此选择：

- 只回 offset patch
- 再对 fresh state 重新检查 status / notified / retain

这就是典型的“放弃局部方便，换全局正确”。

## 3. 为什么 duplicate、replay、双来源消息属于同一个问题

它们表面上长得不一样：

- duplicate control response
- bridge echo / replay
- file mailbox + AppState inbox 重复消息

但本质上都在问同一个问题：

- 同一事实是否会从多条路径再次进入系统

Claude Code 的回答非常一致：

1. 给它稳定 ID
2. 维护已处理集合
3. 第二次来就忽略
4. 不让“也许重复”再进入主真相

这说明作者关注的不是某个 bug patch，而是：

- 可重入、可重放、可重复到达世界里的系统真相

## 4. 为什么这和宿主体验直接相关

如果没有反竞争条件意识，最先受伤的往往不是内部状态，而是宿主和用户的理解：

- 用户看到双份结果
- 服务端一直等待回复
- UI 还停留在旧 mode
- teammate transcript 混入了 leader inbox

所以 race-aware 设计并不是底层优化，它直接决定：

- 用户看到的真相是不是可信

## 5. 苏格拉底式追问

### 5.1 为什么不能先让功能跑通，race 以后再补

因为在 agent runtime 里，很多 race 不是边角情况，而是主路径常态：

- 网络会重放
- 多来源会并发
- 状态会跨 await 变化
- 宿主与本地 UI 会同时参与仲裁

### 5.2 为什么显式 error 也属于反竞争条件设计

因为 silence 会让系统保留一个“也许还在处理”的幻觉。

显式 error 的作用，不只是报错，而是：

- 及时终止错误竞争

### 5.3 对平台构建者最重要的启发是什么

当你写一段“看起来正确”的更新逻辑时，先问：

- 这份快照是否可能已经过时
- 这条消息是否可能已被别处处理
- 这次回复是否可能第二次到达

如果答案是“可能”，那你的设计对象就不再是函数，而是竞争条件。

## 6. 一句话总结

Claude Code 的可靠性来自一种更成熟的工程直觉：在 agent runtime 里，真正危险的往往不是明显错误，而是那些局部看似正确、全局却会制造重复、过时与错觉的代码路径。
