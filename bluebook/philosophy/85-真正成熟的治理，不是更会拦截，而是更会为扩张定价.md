# 真正成熟的治理，不是更会拦截，而是更会为扩张定价

这一章回答五个问题：

1. 为什么 Claude Code 的安全设计与省 Token 设计最后会合成同一条治理母线。
2. 为什么 `governance key`、`externalized truth chain`、`typed ask`、`decision window`、`continuation pricing` 与 `durable-vs-transient cleanup` 其实都在处理“扩张值不值钱”。
3. 为什么很多系统明明看起来更严，却仍然比 Claude Code 更脆。
4. 怎样用苏格拉底式追问审一个新 runtime 是否真的拥有这种治理成熟度。
5. 这对 Agent runtime 设计者意味着什么。

## 1. 先说结论

真正成熟的治理，不是：

- 更会 deny
- 更会弹窗
- 更会省字数

而是：

- 更会给扩张定价

如果只先记治理前门的一句话，也只记这句：

- 治理不是更会拦截，而是先由 governance key 外化当前真相，再给动作、可见性、上下文与 continuation 的一切扩张定价；安全、省 Token 与恢复都只是这条收费链的外观。

这里被统一定价的至少有四个空间：

1. 动作空间
2. 权威空间
3. 上下文空间
4. 时间空间

但这四者的顺序也不是并列的。

更稳的写法应先固定：

1. `governance key`
   - 谁先有资格改 provider、policy、permission mode 与 trust 前环境
2. `typed ask / decision window`
   - 哪些动作能执行、哪些动作必须 ask 或 deny，当前窗口还能承受什么
3. `continuation pricing`
   - 哪些能力、结果与时间占用配继续收费
4. `durable-vs-transient cleanup`
   - 哪些资产配被保留，哪些旧主权必须被清退

这里真正更高阶的判断还可以再压一层：

- `governance key` 不只是第一价格维度，它决定谁有资格给后面的 typed ask / window / continuation / cleanup 继续收费

所以 Claude Code 的安全设计与省 Token 设计之所以同构，不是因为它们都在“限制”，而是因为它们都在回答：

- 这笔扩张值不值得进入当前世界

更稳的说法甚至不是“权限 + 压缩”，而是一整套治理动作在不同资产上的投影：

- `allow / ask / deny`
- `defer visibility`
- `persist + preview`
- `compact / halt`
- `step-up to human`

如果把治理哲学继续压成最短公式，只剩三句：

1. `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`
2. `Narrow / Later / Outside + stable bytes + working set`
3. `没有决策增益的检查，本身就是免费扩张`

这条线最短的 reject trio 也只认：

1. `decision-window collapse`
2. `projection usurpation`
3. `free-expansion relapse`

如果一个治理解释还没压到这三句，就还停在机制对象总结，不算最硬的第一性原理。

## 2. 第一性原理：治理首先处理的不是风险，而是免费扩张

如果一个系统把治理理解成：

- 多几道检查

把省 Token 理解成：

- 压短一点回答

那它仍然停留在局部补丁层。

Claude Code 更深的一层是：

- 它首先拒绝一切未经正式定价的扩张

这包括：

1. 未经定价的动作
2. 未经定价的能力暴露
3. 未经定价的上下文占位
4. 未经定价的 continuation

而这套定价秩序也不是从动作开始。

它先问：

- 谁配改边界

然后才问：

- 哪些动作、可见性、外置结果与 continuation 配进入当前世界

所以真正成熟的治理首先不是：

- 更会阻止

而是：

- 更会收费

更准确地说，治理不只给动作收费，也给失败结算方式收费：

- `asset-typed failure semantics`

也就是说，不同资产失败后该 `deny / ask / cleanup / rollback / halt / escalate`，从一开始就是治理对象，而不是尾部异常处理。

### 更硬一点的源码证据

这条定价链在源码里不是抽象口号，而是四个连续动作：

1. `managedEnv.ts` 先把 `userSettings / flagSettings / policySettings` 认定为 trusted sources，并明确拒绝让 project-scoped settings 在 trust 建立前改 provider-routing 环境。
2. `permissionSetup.ts` 在 auto / classifier 路径下主动剥离会绕过分类器的危险 allow 规则，说明“高权限”也不能免费跳过正式仲裁。
3. `interactiveHandler.ts` 用 resolve-once、后台检查与用户交互竞速，说明权限批准不是 UI 装饰，而是一场 typed transaction，只能有一个合法胜者。
4. `tokenBudget.ts` 在 continuation 不足与 diminishing returns 之间做正式判定，说明“继续多久”本身也是被治理定价的时间资产，而不是默认免费续费。
5. `Tool.ts` 让 `checkPermissions` 与 `maxResultSizeChars` 共存于同一工具抽象，说明“能不能做”与“结果能否继续留在工作集”从一开始就是同一治理对象。
6. `query.ts` 在真正送入模型前先走 `applyToolResultBudget -> snip -> microcompact -> contextCollapse -> autocompact`，说明 token 设计不是事后压缩，而是请求编译前的准入裁剪。

所以安全与省 token 之所以会合流，不是因为它们都在“限制”：

- 一个管风险、一个管成本

而是因为它们都在做同一件事：

- 拒绝未经定价的能力暴露、上下文占位与时间延长

## 3. 最容易被误写成什么

这一层最容易被误写成七种坏解法：

1. permission = modal
   - 以为批准框就是治理本体
2. bypass = unlimited freedom
   - 以为更高权限等于没有边界
3. token saving = shorter outputs
   - 以为字更少就更省
4. trusted input = 谁配置得晚谁说了算
   - 以为治理只是一层 settings merge
5. more visible tools = more capable system
   - 以为模型看见得越多越强
6. dashboard = governance truth
   - 以为上下文统计就是 decision truth
7. host replay = governance truth
   - 以为宿主可以靠事件回放自己拼出当前 mode、pending action 与 context truth

更稳的写法必须承认：

- `policySettings` 不是普通 merge layer，而是带 first-source-wins 与 admin-trusted 语义的 sovereign tier

这些误写的共同问题在于：

- 它们都在把“定价系统”误写成“界面或指标系统”

## 4. 为什么这类误写会让系统既不安全也不省

一旦系统退回上述坏解法，通常会同时出现三种失真：

1. 看起来更安全
   - 但高风险与低风险动作没有被真正分价
2. 看起来更节省
   - 但高体积对象仍在主 prompt 里免费占位
3. 看起来更自由
   - 但自由其实建立在未定价扩张之上

这也是为什么 Claude Code 值得学的不是：

- 某个权限模式名称

而是：

- 反扩张治理的统一价格逻辑

### 反证法：最常伪装成“体验优化”的免费扩张

最危险的扩张，往往不是以“更危险”出现，而是以“更方便”出现：

1. 让 project settings 提前改 provider-routing
   - 看起来是更灵活，实质是低信任来源越权改边界。
2. 保留会绕过 classifier 的 allow 规则
   - 看起来是更省确认，实质是把治理仲裁偷换成历史特权。
3. 只压短回答，却不迁出大结果对象
   - 看起来是更省 token，实质是继续让高体积真相免费霸占主上下文。
4. 把 continuation 当默认继续
   - 看起来是更顺滑，实质是让时间扩张脱离正式定价。

### first reject signal 比 deny 次数更值钱

治理最先失稳时，第一条信号通常不是“漏拦了一次”，而是下面四件事先发生：

1. `Context Usage` 被直接读成成本面板，而不是 `decision window`。
2. 没有决策增益的检查，仍在继续占用主工作集与人类注意力。
3. 大结果对象没有被 `Later / Outside`，还在主上下文里免费占位。
4. host 重新开始靠事件回放自己拼当前真相，而不是消费 runtime 已外化的 truth chain。

## 5. 苏格拉底式追问

如果要审一个新 runtime 是否真的拥有这种治理成熟度，先问：

1. 谁配扩边界；低信任来源能不能自我扩权。
2. 权限决策是弹窗，还是一场 typed transaction。
3. 工具存在与工具当前可见，我有没有分开。
4. 大结果是不是已经被迁出主上下文，而不是继续霸占主席位。
5. continuation 在我这里是默认继续，还是正式收费。
6. fail-open / fail-closed / rollback / cleanup 的选择，是按资产类型，还是按团队情绪。
7. 第一条 reject signal 会先在哪里暴露；是窗口、工作集、truth chain 还是 continuation pricing 先失真。
8. 如果把所有 UI 都删掉，这套治理逻辑还是否成立。
9. host 消费的是 runtime 已外化的 authority/status，还是自己从事件流回放拼当前真相。

## 6. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的治理成熟度，先复制：

1. authority-source admission
2. typed ask arbitration
3. deferred visibility
4. deterministic externalization
5. continuation pricing
6. asset-typed failure semantics
7. host-truth consumption
8. transient authority requalification

而不是先复制：

- 一个更复杂的 approval 界面

## 7. 一句话总结

真正成熟的治理，不是更会拦截，而是更会为扩张定价；它先决定谁配改边界，再决定哪些动作、上下文与时间配进入当前世界。
