# 真正成熟的治理，不是更会拦截，而是更会为扩张定价

这一章回答五个问题：

1. 为什么 Claude Code 的 `decision window` 与 `continuation pricing` 最后会收回同一条治理收费链。
2. 为什么 `governance key`、`externalized truth chain`、`typed ask`、`decision window`、`continuation pricing` 与 `durable-vs-transient cleanup` 其实都在处理“扩张值不值钱”。
3. 为什么很多系统明明看起来更严，却仍然比 Claude Code 更脆。
4. 怎样用苏格拉底式追问审一个新 runtime 是否真的拥有这种治理成熟度。
5. 这对 Agent runtime 设计者意味着什么。

## 1. 先说结论

真正成熟的治理，不是：

- 更会 deny
- 更会弹窗
- 更会省字数

而是：

- 更会给扩张定价

如果只先记治理入口判定的一句话，也只记这句：

- 治理成熟度不先看 deny 次数、弹窗门数、usage 面板或统一预算器，而是先问最早 `unpaid expansion` 是什么，再核对 `repricing proof`、`lease checkpoint` 与 `cleanup`；canonical chain 只是这四问在对象层的正式落点。

更稳的写法不是把治理拆成几个空间，而是先固定六个收费节点：

1. `governance key`
   - 谁有资格定价、谁先能改 provider、policy、permission mode 与 trust 前环境
2. `externalized truth chain`
   - 宿主承认哪条当前真相，以及哪些 truth-surface 能继续签字
3. `typed ask`
   - 哪些扩张必须显式仲裁
4. `decision window`
   - 当前世界还能承受什么
5. `continuation pricing`
   - 是否继续为能力、结果与时间占用续费
6. `durable-transient cleanup`
   - 恢复后哪些资产可留、哪些旧主权必须被清退

这里要先把 speaking-rights 钉死：

- 上面六个收费节点是对象层 canonical chain，不是入口第一问。
- 入口第一问永远是 `earliest unpaid expansion -> repricing proof -> lease checkpoint -> cleanup`。
- 若这四问还答不上，stage name 越全越像第二前门，而不是更成熟的治理说明。

这里还要再多记一句：

- 这条收费链最早落地的地方，常常不是动作执行，而是 `visible world admission`；对象、schema 与 host truth 一旦先进入模型世界，后续 deny 也只能阻止手脚，阻止不了视野、风险与 token 已经完成的扩张。
- 换句话说，对象进入模型世界，与工具获得执行许可，都是同一种扩张定价；前者给视野续租，后者给动作续租。

这里真正更高阶的判断还可以再压一层：

- `governance key` 不只是第一价格维度，它决定谁有资格给后面的 typed ask / window / continuation / cleanup 继续收费
- `governance key` 也不只是第一道检查，它是下游收费权的主权对象；谁先能改 `provider / policy / trust`，谁就在定义后续价格的合法来源
- 更稳一点说，`governance key` 还应直接被读成 `pricing-right signer / pricing-right lease`：谁能定价，谁能见证价格，谁在 lease 失效后仍敢继续收费，这三件事必须属于同一主权对象链。

所以这条治理链真正回答的，不是“现在更安全还是更省 Token”，而是：

- 这笔扩张值不值得进入当前世界、还能不能继续、最后由谁收口

更稳的说法不是把 `allow / ask / deny`、`defer visibility`、`persist + preview`、`compact / halt` 与 `step-up to human` 并列成动作 bucket，而是把它们逐一挂回 canonical chain：它们分别只是 `typed ask verdict`、`visible-set control`、`externalization ABI`、`decision-window / continuation pricing verdict` 与 `human signer fallback`。

这里还要再多记一句：

- 真正值钱的不是 permission 投影、classifier 或某一格按钮，而是 `verdict seam`：围绕 `request_id / tool_use_id`，谁能出 verdict、谁能接管、谁能撤销、谁负责 rollback，都必须先被写成治理事务。

如果把治理哲学继续压成最短公式，只剩三句：

1. `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`
2. 同一笔扩张不是先被分进 `Narrow / Later / Outside`，而是先沿 canonical chain 判断边界、真相、ask、继续资格与 cleanup
3. `没有决策增益的检查，本身就是免费扩张`

如果要把这三句再直接对齐到执行层 visible noun，也只该再多记一句：

- 执行层无论出现 status、approval、rollback 还是 handoff 名词，都只能被解释为 canonical chain 的接口名，不能再另造一层入口别名。

如果一个治理解释还没压到这三句，就还停在机制对象总结，不算最硬的第一性原理。进入 `reject` 命名、第一失稳信号或 execution tail 时，统一回 `10 / security / playbooks`，本页不再自带 `reject trio`。

## 2. 第一性原理：治理首先处理的不是风险，而是免费扩张

如果一个系统把治理理解成：

- 多几道检查

把省 Token 理解成：

- 压短一点回答

那它仍然停留在局部补丁层。

Claude Code 更深的一层是：

- 它首先拒绝一切未经正式定价的扩张

这包括：

1. 未经定价的动作
2. 未经定价的能力暴露
3. 未经定价的上下文占位
4. 未经定价的 continuation

而这套定价秩序也不是从动作开始。

它先问：

- 谁配改边界

然后才问：

- 哪些动作、可见性、外置结果与 continuation 配进入当前世界

所以真正成熟的治理首先不是：

- 更会阻止

而是：

- 更会收费

更准确地说，治理不只给动作收费，也给失败结算方式收费：

- `asset-typed failure semantics`

也就是说，不同资产失败后该 `deny / ask / cleanup / rollback / halt / escalate`，从一开始就是治理对象，而不是尾部异常处理。

### 更硬一点的源码证据

这条定价链在源码里不是抽象口号，而是四个连续动作：

1. `managedEnv.ts` 先把 `userSettings / flagSettings / policySettings` 认定为 trusted sources，并明确拒绝让 project-scoped settings 在 trust 建立前改 provider-routing 环境。
2. `permissionSetup.ts` 在 auto / classifier 路径下主动剥离会绕过分类器的危险 allow 规则，说明“高权限”也不能免费跳过正式仲裁。
3. `interactiveHandler.ts` 用 resolve-once、后台检查与用户交互竞速，说明权限批准不是交互外观，而是一场 typed transaction，只能有一个合法胜者。
4. `tokenBudget.ts` 在 continuation 不足与 diminishing returns 之间做正式判定，说明“继续多久”本身也是被治理定价的时间资产，而不是默认免费续费。
5. `Tool.ts` 让 `checkPermissions` 与 `maxResultSizeChars` 共存于同一工具抽象，说明“能不能做”与“结果能否继续留在工作集”从一开始就是同一治理对象。
6. `query.ts` 在真正送入模型前先走 `applyToolResultBudget -> snip -> microcompact -> contextCollapse -> autocompact`，说明 token 设计不是事后压缩，而是请求编译前的准入裁剪。

如果再把这组证据压得更短，还应补一句：

- `classifier` 不是治理控制面外的免费裁判，而是只有在还能改变 `verdict` 时才值得调用的定价节点。
- 更具体地说，真正值钱的不是批准交互外观，而是 `verdict seam`：围绕 `request_id / tool_use_id`，谁能出 verdict、谁能接管、谁能撤销、谁负责 rollback，都必须先被写成治理事务。

所以动作定价、上下文定价与时间定价之所以会合流，不是因为它们都在“限制”：

- 一个管风险、一个管成本

而是因为它们都在做同一件事：

- 拒绝未经定价的能力暴露、上下文占位与时间延长

## 3. 最容易被误写成什么

这一层最容易被误写成七种坏解法：

1. permission = 单次弹窗
   - 以为一次批准交互就是治理本体
2. bypass = unlimited freedom
   - 以为更高权限等于没有边界
3. token saving = shorter outputs
   - 以为字更少就更省
4. trusted input = 谁配置得晚谁说了算
   - 以为治理只是一层 settings merge
5. more visible tools = more capable system
   - 以为模型看见得越多越强
6. usage readback = governance truth
   - 以为上下文读数就是 decision truth
7. host replay = governance truth
   - 以为宿主可以靠事件回放自己拼出当前 permission 投影、pending action 与 context truth

更稳的写法必须承认：

- `policySettings` 不是普通 merge layer，而是带 first-source-wins 与 admin-trusted 语义的 sovereign tier

这些误写的共同问题在于：

- 它们都在把“定价系统”误写成“界面或指标系统”

## 4. 为什么这类误写会让系统既不安全也不省

一旦系统退回上述坏解法，通常会同时出现三种失真：

1. 看起来更安全
   - 但高风险与低风险动作没有被真正分价
2. 看起来更节省
   - 但高体积对象仍在主 prompt 里免费占位
3. 看起来更自由
   - 但自由其实建立在未定价扩张之上

这也是为什么 Claude Code 值得学的不是：

- 某个权限模式名称

而是：

- 反扩张治理的统一价格逻辑

### 反证法：最常伪装成“体验优化”的免费扩张

最危险的扩张，往往不是以“更危险”出现，而是以“更方便”出现：

1. 让 project settings 提前改 provider-routing
   - 看起来是更灵活，实质是低信任来源越权改边界。
2. 保留会绕过 classifier 的 allow 规则
   - 看起来是更省确认，实质是把治理仲裁偷换成历史特权。
3. 只压短回答，却不迁出大结果对象
   - 看起来是更省 token，实质是继续让高体积真相免费霸占主上下文。
4. 把 continuation 当默认继续
   - 看起来是更顺滑，实质是让时间扩张脱离正式定价。

### 同题坏解最常先在哪里露出

治理最先露出坏解时，通常不是“漏拦了一次”，而是下面几件事先发生：

1. usage 读数被直接读成成本面板，而不是 `decision window`。
2. 没有决策增益的检查，仍在继续占用主工作集与人类注意力。
3. 大结果对象没有被 `Later / Outside`，还在主上下文里免费占位。
4. host 重新开始靠事件回放自己拼当前真相，而不是消费 runtime 已外化的 truth chain。
5. classifier、批准交互或默认重试已经失去决策增益，却仍在继续烧 token 购买“看起来更谨慎”的幻觉。

## 5. 苏格拉底式追问

如果要审一个新 runtime 是否真的拥有这种治理成熟度，先问：

1. 谁配扩边界；低信任来源能不能自我扩权。
2. 权限决策是弹窗，还是一场 typed transaction。
3. 工具存在与工具当前可见，我有没有分开。
4. 大结果是不是已经被迁出主上下文，而不是继续霸占主席位。
5. continuation 在我这里是默认继续，还是正式收费。
6. 旧 authority 的收口、回退与停止续租，是按正式定价对象，还是按团队情绪。
7. 最早 `unpaid expansion` 会先在哪一环暴露；是窗口、工作集、truth chain 还是 continuation pricing 先失真。
8. 如果把所有交互外观都删掉，这套治理逻辑还是否成立。
9. host 消费的是 runtime 已外化的 authority/status 读回，还是自己从事件流回放拼当前真相。
10. classifier 现在还能改变 verdict，还是已经只是更贵的谨慎表演。
11. 我这里的 continuation 是“再来一轮”，还是“再续一段可撤销的 authority 租期”。

## 6. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的治理成熟度，先复制：

1. authority-source admission
2. typed ask arbitration
3. deferred visibility
4. deterministic externalization
5. continuation pricing
6. asset-typed failure semantics
7. host-truth consumption
8. transient authority requalification

而不是先复制：

- 一个更复杂的批准交互外观

## 7. 一句话总结

真正成熟的治理，不是更会拦截，而是更会为扩张定价；它先决定谁配改边界，再决定哪些动作、上下文与时间配进入当前世界。
