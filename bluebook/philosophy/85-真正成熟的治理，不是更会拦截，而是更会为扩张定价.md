## 1. 先说结论

真正成熟的治理，不是：

- 更会 deny
- 更会弹窗
- 更会省字数

而是：

- 更会给扩张定价

如果只先记治理判断的一句话，也只记这句：

- 治理成熟度不先看 deny 次数、弹窗门数、usage 面板或统一预算器，而是先问最早 `unpaid expansion` 是什么，再核对 `repricing proof`、`lease checkpoint` 与 `cleanup`。

这句话之所以比任何机制名更值钱，是因为它先固定了“真正被治理的对象”：

- 不是单次拦截动作
- 不是某个交互外观
- 也不是某个成本数字
- 而是一次未经定价的扩张能不能进入当前世界、能不能继续、最后由谁收口

这里还要再多记一句：

- 这条收费链最早落地的地方，常常不是动作执行，而是世界准入、能力暴露、上下文占位或继续续租本身；一旦这些扩张先免费进入，后续 deny 只能阻止后半程，阻止不了风险、视野与 token 已经先完成的扩张。

所以治理真正回答的，不是“现在更安全还是更省 Token”，而是：

- 这笔扩张有没有被正式定价、还能不能继续收费、谁负责最终结算

更硬一点说，真正被续租的不是一次动作或一次 token 花费，而是一条 `authority lease`：世界准入、能力暴露、上下文占位与 continuation 都只是这条 lease 的收费节点；安全在阻止危险扩张免费续租，省 token 在阻止昂贵扩张免费续租。

这也解释了为什么治理尾链上的 `weak readback` 必须保持为弱：一旦 readback surface 还能重判 `repricing proof`、`lease checkpoint` 或 `cleanup`，它就把已经结算的 `authority lease` 重新打开成新的 `unpaid expansion` 入口。允许被读回的因此只能是 product promise、reopen eligibility 与 evidence receipt 这类结算后对象，而不是再次改判当前治理真相的第二裁决面。

这也意味着治理尾链的硬边界，不看它长得像不像答案，而只看它有没有资格改变 `verdict`：`/status / /usage` 之类 `weak readback` 只配暴露 drift 与 pressure；`host replay` 若靠事件回放拼当前 permission 与上下文真相，就已经在自造第二裁决面；`reopen eligibility` 只能读 residual liability 的门槛，不能从 readback 面板推断资格；`cleanup receipt` 即使未来做强，也只配签 closure truth，而不配签“已安全 / 可默认继续”的世界真相。
同一条治理秩序在执行面还必须再分成两层：`permissions` 回答“这次扩张是否配被尝试”，`sandboxing` 回答“即便被允许，也最多只能在什么边界内发生”。更硬一点说，成熟治理默认应是 `deny-first / boundary-first`：先把不该尝试的扩张拒在裁决面外，再把即便允许的扩张收进可承受的文件系统与网络边界内；把两者误写成“多一道安全检查”，会把治理重新退回弹窗与模式词表。
官方公开面把 `sandboxing` 说得更直白：它不是再加一层逐条许可，而是用预先划定的边界替代每条命令的 permission prompt。顺着这条 why 再压一层，治理真正处理的就不是“多一次 ask”，而是 `起租 -> 圈界 -> 回单`：先裁定扩张配不配被尝试，再把允许的扩张锁进边界，最后把读回降格成结算回单而不是新的裁决入口。
Anthropic 公开介绍 auto mode 时，也把同一条 why 暴露得更直白：既然绝大多数 permission prompts 最终都只是批准，成熟治理就不该让人类反复为同一种安全扩张重判，而应把高频安全批准压进 `classifier + sandbox boundary` 这类组合里，减少 `approval fatigue`，同时把真正的 `repricing / deny` 资格继续留在正式裁决面。
同一条治理秩序在配置层也不能被误写成普通 merge：managed settings 若能改写危险执行边界，就更接近待审签的治理命令通道，而不是静默并入的偏好补丁。更稳的判断因此不是“谁的设置写得更晚”，而是“哪条策略现在有资格改边界、且危险 delta 是否已经过单独审签”。
同一条 why 继续压到时间轴上，也会直接落到 `/compact / /resume / /memory`：它们不是 continuity 便利按钮，而是这条 `authority lease` 的 continuation consumer，真正被追问的是这次 `decision window` 是否仍值得续租，而不是如何默认把继续免费伪装成顺滑体验。
治理真正要再写死的一层，也正是 signer ceiling：managed settings 先签主权边界，auto mode 只配压缩高频 approval，continuation consumer 与 weak readback 只配消费或回单既有 verdict；凡会改写价格、继续资格或终局解释的动作，都必须回到正式 repricing 面。
这条治理链的第一条 `local veto cue` 因而也不是“面板看起来更吓人”，而是 `weak readback`、host replay、default continue 或 reopen tail evidence 开始越权代签 `repricing proof / continue qualification / cleanup truth`；第一退回层则应先回 `decision window / repricing seam / cleanup boundary`，而不是继续比较 mode、modal 或 usage 词表。
如果把这条秩序再压成一句，也只剩：`permissions` 定价起租，`sandboxing` 预先圈界，`weak readback` 只出回单；少一次重判，就少一次免费扩张。

更稳的 why 说法也因此只剩三句：

1. 治理先处理免费扩张，不先处理交互外观。
2. 安全与省 token 只是同一条定价秩序的不同投影。
3. 没有决策增益的检查，本身就是免费扩张。

如果一个治理解释还必须靠 stage name、对象 inventory 或界面词表才能站住，它就还停在机制巡览，不算 why-proof。

## 2. 第一性原理：治理首先处理的不是风险，而是免费扩张

如果一个系统把治理理解成：

- 多几道检查

把省 Token 理解成：

- 压短一点回答

那它仍然停留在局部补丁层。

Claude Code 更深的一层是：

- 它首先拒绝一切未经正式定价的扩张

这包括：

1. 未经定价的动作
2. 未经定价的能力暴露
3. 未经定价的上下文占位
4. 未经定价的 continuation

而这套定价秩序也不是从动作细目开始，而是先问：

- 谁配扩边界，谁只配消费边界

然后才问：

- 哪些动作、可见性、上下文与继续资格配进入当前世界

所以真正成熟的治理首先不是更会阻止，而是更会收费。
更准确地说，治理不只给动作收费，也给失败之后谁负责结算收费；不同失败之所以不该共享同一处理方式，不是因为分类更漂亮，而是因为它们对应的是不同的未定价扩张。

### 更硬一点的源码证据

这条定价链在源码里最值得保留的不是长串机制巡览，而是两个原则级信号：

1. 低信任路径不能免费改边界，说明“谁配扩张”被正式对象拦住了。
2. 大体积对象与长时间继续不能免费续租，说明“哪些扩张还值得继续收费”被正式检查住了。

如果再把这组证据压得更短，也只补一句：任何检查都只有在还能改变 `verdict` 时才值得存在；一旦只剩交互外观、状态面板或结果展示，它就已经从定价节点退回昂贵表演。
没有决策增益的检查，也因此不只是体验噪音，而是新的 `unpaid expansion`：它扩大了 readback 面、占用了工作集，并让人类或宿主继续为不能改价的对象付费。

所以动作定价、上下文定价与时间定价之所以会合流，不是因为它们都在“限制”：

- 一个管风险、一个管成本

而是因为它们都在做同一件事：

- 拒绝未经定价的能力暴露、上下文占位与时间延长

## 3. 最容易被误写成什么

这一层最容易被误写成七种坏解法：

1. permission = 单次弹窗
   - 以为一次批准交互就是治理本体
2. bypass = unlimited freedom
   - 以为更高权限等于没有边界
3. token saving = shorter outputs
   - 以为字更少就更省
4. trusted input = 谁配置得晚谁说了算
   - 以为治理只是一层 settings merge
5. more visible tools = more capable system
   - 以为模型看见得越多越强
6. usage readback = governance truth
   - 以为上下文读数就是 decision truth
7. host replay = governance truth
   - 以为宿主可以靠事件回放自己拼出当前 permission 投影、pending action 与 context truth

这些误写的共同问题在于：

- 它们都在把“定价系统”误写成“界面或指标系统”

## 4. 为什么这类误写会让系统既不安全也不省

一旦系统退回上述坏解法，通常会同时出现三种失真：

1. 看起来更安全
   - 但高风险与低风险动作没有被真正分价
2. 看起来更节省
   - 但高体积对象仍在主 prompt 里免费占位
3. 看起来更自由
   - 但自由其实建立在未定价扩张之上

这也是为什么 Claude Code 值得学的不是：

- 某个权限模式名称或交互外观

而是：

- 反扩张治理的统一价格逻辑

### 反证法：最常伪装成“体验优化”的免费扩张

最危险的扩张，往往不是以“更危险”出现，而是以“更方便”出现：

1. 让低信任来源提前改边界
   - 看起来是更灵活，实质是免费扩权。
2. 让历史特权绕过重定价
   - 看起来是更省确认，实质是把治理仲裁偷换成旧许可。
3. 只压短回答，却不迁出大结果对象
   - 看起来是更省 token，实质是继续让高体积真相免费霸占主上下文。
4. 把 continuation 当默认继续
   - 看起来是更顺滑，实质是让时间扩张脱离正式定价。

### 同题坏解最常先在哪里露出

治理最先露出坏解时，通常不是“漏拦了一次”，而是下面几件事先发生：

1. usage 读数被直接读成成本面板，而不是“这次定价是否仍成立”的读回。
2. 没有决策增益的检查，仍在继续占用主工作集与人类注意力。
3. 大结果对象没有被迁出主上下文，还在免费占位。
4. host 重新开始靠事件回放自己拼当前真相，而不是消费 runtime 已外化的读回。
5. 批准交互或默认重试已经失去决策增益，却仍在继续烧 token 购买“看起来更谨慎”的幻觉。

## 5. 苏格拉底式追问

如果要审一个新 runtime 是否真的拥有这种治理成熟度，先问：

1. 谁配扩边界；低信任来源能不能自我扩权。
2. 当前的权限决定，究竟是在给扩张定价，还是只是在表演谨慎。
3. 工具存在、工具当前可见与工具当前可继续，我有没有分开。
4. 大结果是不是已经迁出主上下文，而不是继续免费霸占主席位。
5. continuation 在我这里是默认继续，还是正式收费。
6. 旧 authority 的收口、回退与停止续租，是按正式对象结算，还是按团队情绪结算。
7. 最早 `unpaid expansion` 先在哪一环暴露：世界准入、动作暴露、工作集占位，还是继续续租。
8. 如果把所有交互外观都删掉，这套治理逻辑还是否成立。
9. host 消费的是 runtime 已外化的读回，还是自己从事件流回放拼当前真相。
10. 当前检查还能改变 verdict，还是已经只是更贵的谨慎表演。

## 6. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的治理成熟度，先复制“收费顺序先于交互外观”这条原则，而不是先复制一个更复杂的批准交互外观。
更细的机制对象、边界名词与执行顺序统一留给入口卡、对象层与执行页；why 页只负责证明：为什么 expansion 不能免费进入、免费继续、免费收尾。

## 7. 一句话总结

真正成熟的治理，不是更会拦截，而是更会为扩张定价；它先决定谁配改边界，再决定哪些动作、上下文与时间配进入当前世界。
