# 真正成熟的治理，不是更会拦截，而是更会为扩张定价

这一章回答五个问题：

1. 为什么 Claude Code 的安全设计与省 Token 设计最后会合成同一条治理母线。
2. 为什么 `trusted inputs`、权限决策、能力可见性、结果外置与 continuation 其实都在处理“扩张值不值钱”。
3. 为什么很多系统明明看起来更严，却仍然比 Claude Code 更脆。
4. 怎样用苏格拉底式追问审一个新 runtime 是否真的拥有这种治理成熟度。
5. 这对 Agent runtime 设计者意味着什么。

## 1. 先说结论

真正成熟的治理，不是：

- 更会 deny
- 更会弹窗
- 更会省字数

而是：

- 更会给扩张定价

这里被统一定价的至少有四个空间：

1. 动作空间
2. 权威空间
3. 上下文空间
4. 时间空间

但这四者的顺序也不是并列的。

更稳的写法应先固定：

1. `authority source`
   - 谁先有资格改 provider、policy、permission mode 与 trust 前环境
2. `action pricing`
   - 哪些动作能执行、哪些动作必须 ask 或 deny
3. `context pricing`
   - 哪些能力和结果配占当前可见集
4. `time pricing`
   - 这轮是否仍值得继续耗时和耗 token

这里真正更高阶的判断还可以再压一层：

- `authority source` 不只是第一价格维度，它决定谁有资格给后面的 action / context / time 三个空间收费

所以 Claude Code 的安全设计与省 Token 设计之所以同构，不是因为它们都在“限制”，而是因为它们都在回答：

- 这笔扩张值不值得进入当前世界

## 2. 第一性原理：治理首先处理的不是风险，而是免费扩张

如果一个系统把治理理解成：

- 多几道检查

把省 Token 理解成：

- 压短一点回答

那它仍然停留在局部补丁层。

Claude Code 更深的一层是：

- 它首先拒绝一切未经正式定价的扩张

这包括：

1. 未经定价的动作
2. 未经定价的能力暴露
3. 未经定价的上下文占位
4. 未经定价的 continuation

而这套定价秩序也不是从动作开始。

它先问：

- 谁配改边界

然后才问：

- 哪些动作、可见性、外置结果与 continuation 配进入当前世界

所以真正成熟的治理首先不是：

- 更会阻止

而是：

- 更会收费

## 3. 最容易被误写成什么

这一层最容易被误写成七种坏解法：

1. permission = modal
   - 以为批准框就是治理本体
2. bypass = unlimited freedom
   - 以为更高权限等于没有边界
3. token saving = shorter outputs
   - 以为字更少就更省
4. trusted input = 谁配置得晚谁说了算
   - 以为治理只是一层 settings merge
5. more visible tools = more capable system
   - 以为模型看见得越多越强
6. dashboard = governance truth
   - 以为上下文统计就是 decision truth
7. host replay = governance truth
   - 以为宿主可以靠事件回放自己拼出当前 mode、pending action 与 context truth

更稳的写法必须承认：

- `policySettings` 不是普通 merge layer，而是带 first-source-wins 与 admin-trusted 语义的 sovereign tier

这些误写的共同问题在于：

- 它们都在把“定价系统”误写成“界面或指标系统”

## 4. 为什么这类误写会让系统既不安全也不省

一旦系统退回上述坏解法，通常会同时出现三种失真：

1. 看起来更安全
   - 但高风险与低风险动作没有被真正分价
2. 看起来更节省
   - 但高体积对象仍在主 prompt 里免费占位
3. 看起来更自由
   - 但自由其实建立在未定价扩张之上

这也是为什么 Claude Code 值得学的不是：

- 某个权限模式名称

而是：

- 反扩张治理的统一价格逻辑

## 5. 苏格拉底式追问

如果要审一个新 runtime 是否真的拥有这种治理成熟度，先问：

1. 谁配扩边界；低信任来源能不能自我扩权。
2. 权限决策是弹窗，还是一场 typed transaction。
3. 工具存在与工具当前可见，我有没有分开。
4. 大结果是不是已经被迁出主上下文，而不是继续霸占主席位。
5. continuation 在我这里是默认继续，还是正式收费。
6. fail-open / fail-closed 的选择，是按资产类型，还是按团队情绪。
7. 如果把所有 UI 都删掉，这套治理逻辑还是否成立。
8. host 消费的是 runtime 已外化的 authority/status，还是自己从事件流回放拼当前真相。

## 6. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的治理成熟度，先复制：

1. authority-source admission
2. typed ask arbitration
3. deferred visibility
4. deterministic externalization
5. continuation pricing
6. asset-typed failure semantics
7. host-truth consumption
8. transient authority requalification

而不是先复制：

- 一个更复杂的 approval 界面

## 7. 一句话总结

真正成熟的治理，不是更会拦截，而是更会为扩张定价；它先决定谁配改边界，再决定哪些动作、上下文与时间配进入当前世界。
