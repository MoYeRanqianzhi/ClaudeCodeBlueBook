# 先进源码不是一开始就完美分层，而是增长时仍能守住不变量

这一章回答四个问题：

1. 为什么真正成熟的源码先进性不该只看“当前结构漂不漂亮”。
2. 为什么 Claude Code 更值得学的是它的增长方法，而不是某一次静态分层快照。
3. 为什么 config、deps、state machine、leaf module 这些看似局部的动作，透露出更高层方法论。
4. 这对大型 agent runtime 的工程审美意味着什么。

## 1. 先说结论

Claude Code 的源码先进性，更准确地说是：

- 它不是一开始就追求完美分层

而是：

- 它在持续增长时，始终试图把关键不变量收回固定落点

如果用更高一层的话说，它真正做的是：

- 控制增长带来的熵

所以真正该看的不是：

- 现在有多少文件
- 现在模块长什么样

而是：

- 新功能继续长进来时，哪些东西还能保持不乱

## 2. 为什么“增长时不乱”比“当前看着整齐”更值钱

很多代码库在静态截图上都能显得整齐。

真正难的是：

1. 新状态进来时，状态机会不会裂。
2. 新入口进来时，协议真相会不会裂。
3. 新异步路径进来时，时序会不会裂。
4. 新依赖进来时，图结构会不会裂。

Claude Code 值得学的地方，就在它明显知道这些裂缝最危险，所以先把相关边界建成基础设施。

## 3. 苏格拉底式追问

### 3.1 一个仓库今天看起来拆得很细，就一定更成熟吗

未必。

如果新功能进来时真相和状态还会四处漂，那只是“细”，不是“稳”。

### 3.2 为什么 config / deps / guard 这些局部动作很重要

因为它们在提前回答：

- 未来增长进来时，哪些东西不应该被再次揉在一起

### 3.3 如果今天不把不变量的落点固定下来，以后还能靠大重构补救吗

理论上能，实践里很贵。

而且往往已经被历史路径绑死。

### 3.4 真正先进的源码更像什么

更像：

- 一套持续把增长风险收回底盘的系统

## 4. 对 Agent 系统设计者的启发

如果想学 Claude Code，最该学的不是：

- 一次性设计出最优目录树

而是：

1. 先识别哪个不变量最容易被增长打碎。
2. 先把那个地方做成权威面或状态机。
3. 再让结构慢慢围着这些不变量重排。
4. 接受局部热点，但拒绝全仓半真相。

更硬一点说，先问四个问题，再谈 feature 速度：

1. authority 在哪里。
2. transition 在哪里。
3. boundary 在哪里。
4. dependency 是怎么被管住的。

## 5. 一句话总结

Claude Code 的源码之所以显得先进，不在于它从第一天就完美分层，而在于它增长时仍不断把关键不变量收回固定落点。
