# 前台交互不是 UI 皮肤，而是认知控制面

这一章回答四个问题：

1. 为什么 Claude Code 的前台交互不能被简化成“终端聊天 UI”。
2. sticky prompt、transcript mode、message actions 为什么不是装饰。
3. 这套前台设计揭示了怎样的 Agent 交互观。
4. 这对我们理解 Agent IDE 有什么第一性原理启发。

## 1. 先说结论

Claude Code 的前台交互本质上不是表现层，而是认知控制面。

它持续在帮助用户做四件事：

1. 保持任务锚点
2. 在长历史里导航
3. 在多种操作模式间切换
4. 在并行执行体之间切换主语

这四件事和普通聊天 UI 的目标完全不同。

## 2. 为什么普通聊天 UI 不够

普通聊天 UI 默认假设：

1. 你总在看最新一条
2. 历史主要用来回看，不用来操作
3. 输入框始终是唯一控制入口
4. 对话主体始终只有一个

Claude Code 的前台显然不是这样：

- 用户会向上滚动很多
- transcript 会很长
- message 本身可被操作
- queued command 会在未来进入
- teammate view 会切换“当前谁是主语”

所以它必须把前台做成控制面，而不是聊天皮肤。

## 3. sticky prompt：前台的第一认知装置

sticky prompt 的价值不是视觉效果，而是：

- 当用户离开底部时，仍知道自己当前围绕哪条 prompt 在看
- 点击可跳回 prompt
- 与 unseen divider 一起构成“离现场有多远”的知觉

这说明作者已经意识到一个关键事实：

- 长会话里，用户最容易丢失的不是消息内容，而是任务锚点

## 4. transcript mode：前台的第二认知装置

transcript mode 的核心不是“看全量消息”，而是把会话变成可导航文档：

- `/` 搜索
- `n/N` 跳转
- anchor 回弹
- dump 到 scrollback
- `v` 打开外部编辑器

这更像：

- pager
- 阅读器
- 检索器

而不是聊天窗口。

## 5. message actions：前台的第三认知装置

message actions 进一步表明：

- 消息不是只拿来看
- 某些消息是可操作工件

例如：

- edit user prompt
- copy text
- copy tool primary input
- expand/collapse grouped tool use

这说明 Claude Code 的前台把“消息”从聊天气泡提升成了：

- 可导航、可复用、可变换的工作对象

## 6. teammate view：前台的第四认知装置

teammate view 解决的是：

- 多 Agent 系统里，用户当前到底在看谁的世界

这和普通多标签页很不一样，因为它不只是切页面，而是在切：

- 主语
- in-progress tool state
- sticky/pill 可见性
- 可输入对象

所以多 Agent 前台真正难的部分，不是“列出几个 agent”，而是：

- 如何让用户不在多条执行链之间迷路

## 7. 从第一性原理看，为什么 Agent IDE 前台必须是控制面

如果一个 Agent IDE 想真的长期可用，前台至少要解决：

1. 注意力会漂移
2. 会话会变长
3. 同时会有多条执行链
4. 用户既要读，也要控，也要回溯

因此它的前台必须至少同时承担：

- 锚点
- 检索
- 主语切换
- 工件操作

这就是“认知控制面”的含义。

## 8. 苏格拉底式追问

### 问：为什么不能只做一个更漂亮的终端 UI？

答：因为漂亮解决不了长会话里的认知迷航。

### 问：为什么 transcript 搜索和 sticky prompt 值得做成正式机制？

答：因为用户失败的主要原因常常不是模型不够聪明，而是人已经失去现场感。

### 问：为什么 message actions 会成为产品能力？

答：因为一旦消息被当成工件，用户和 agent 的协作方式就从“聊天”变成“共同操作执行轨迹”。

## 9. 一句话总结

Claude Code 的前台交互之所以显得强，不是因为它像一个好看的聊天界面，而是因为它把用户的注意力、导航、主语切换和工件操作做成了正式的认知控制面。
