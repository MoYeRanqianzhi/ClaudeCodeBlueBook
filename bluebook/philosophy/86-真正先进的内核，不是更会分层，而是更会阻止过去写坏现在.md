# 真正先进的内核，不是更会分层，而是更会阻止过去写坏现在

这一章回答五个问题：

1. 为什么 Claude Code 的源码先进性最终要回到“现在不能被过去写坏”。
2. 为什么 authority surface、recovery asset、anti-zombie 与 release shaping 共同决定内核成熟度。
3. 为什么很多看起来更整洁的仓库，长期反而更难维护。
4. 怎样用苏格拉底式追问审一个新 runtime 是否真的拥有这种结构先进性。
5. 这对 Agent runtime 设计者意味着什么。

## 1. 先说结论

真正先进的内核，不是：

- 文件更小
- 目录更碎
- 依赖更抽象

而是：

- 它更会阻止过去写坏现在

这里至少有四类正式保护：

1. single-writer authority surface
2. recovery asset 不得篡位
3. anti-zombie 写回约束
4. compile-time / runtime / artifact 三层边界塑形

更硬一点说，这四类保护现在已经下沉成更具体的五类 freshness discipline：

1. fresh-read 才能写
2. validator worldview stale 时统一降级
3. per-host authority 不跨 host 继承
4. 危险增量必须 confirm-or-exit
5. ghost capability / ghost auth 必须主动驱逐

所以 Claude Code 的源码先进性首先不是：

- 分层能力

而是：

- 现在时态保护能力

更硬一点说，旧对象如果还想影响 present，通常只剩四种合法命运：

1. 被 `invalidate`
2. 被 `relink`
3. 被 `adopt` 到当前权威头
4. 被降格成 `projection / metadata`

## 2. 第一性原理：先进结构首先处理的是时间，不只是空间

很多结构分析都过度偏爱空间问题：

- 模块怎么分
- 目录怎么排
- 依赖怎么切

Claude Code 更深的一层提醒是：

- 时间维度同样会制造第二真相

也就是说，真正危险的不是只有坏依赖，还有：

1. 旧 finally
2. 旧 snapshot
3. 旧 pointer
4. 旧 head assumption
5. 旧文件快照
6. 旧 validator 视角
7. 旧 capability 目录

所以先进内核真正首先要问的不是：

- 代码分得美不美

而是：

- 过去会不会在未来某个时刻把现在重写掉

### 更硬一点的源码证据

这条 present-state protection 在源码里已经下沉成几类很硬的 choke point：

1. `QueryGuard.end(generation)` 与 `forceEnd()` 明确把 stale finally 挡在门外，旧查询即使晚回来，也不能替当前查询做清理决定。
2. `onChangeAppState()` 把 permission mode externalization 收到单一 diff choke point，防止不同 mutation path 各自宣布对外真相。
3. `sessionIngress.ts` 遇到 `409` 时先尝试 adopt server last UUID，再决定是否重试，说明远端 append chain 才是当前真相，旧本地假设必须先让位。
4. `bridgePointer.ts` 用 mtime TTL、schema 校验和跨 worktree 取 freshest pointer，说明恢复资产只能帮助重连 present state，不能靠陈旧 pointer 篡位。

所以“防过去写坏现在”不是一个漂亮比喻，而是：

- stale writer 不得清理现在
- 旧外部视角不得宣布现在
- 恢复资产不得冒充现在
- 过期 pointer 不得重新绑定现在

这也意味着系统并不信任“原始历史”本身；历史只有在被重新编译成当前仍可行动的对象之后，才配继续进入 present。

## 3. 最容易被误写成什么

这一层最容易被误写成八种坏解法：

1. smaller files = better architecture
2. more reuse = healthier dependency graph
3. recovery asset = second authority
4. remote / bridge = add-on transport
5. release shaping = 打包尾巴
6. anti-zombie = 某个局部并发 bug 修复
7. host / domain 变化后旧授权还能继续沿用
8. event timeline = current truth

这些误写的共同问题在于：

- 它们都把“时态保护结构”误写成“静态组织结构”

## 4. 为什么这类误写会直接毁掉源码先进性

一旦系统退回上述坏解法，马上会出现三种更贵的后果：

1. 当前真相不再有单一写入面
2. 恢复资产开始偷偷宣布自己最新
3. 旧对象在跨 await、跨进程、跨恢复路径后重新覆盖 present state

所以 Claude Code 值得学的不是：

- 哪个目录更漂亮

而是：

- 谁能写当前真相、谁只能帮助恢复、谁必须在代际检查后闭嘴

### 当前真相面不是事件时间线

很多系统最危险的偷换，是把“发生过什么”误写成“现在是什么”。

更稳的分法必须承认：

1. event stream 负责讲时间线
2. state writeback 负责宣布当前真相
3. recovery asset 负责帮助回到当前真相
4. freshness gate 负责阻止旧对象再次写入

如果这四件事混在一起，代码看起来也许更短，但 current-truth surface 已经消失了。

## 5. 苏格拉底式追问

如果要审一个新 runtime 是否真的拥有这种结构先进性，先问：

1. 当前哪一个 surface 有资格宣布当前真相。
2. recovery asset 在帮助恢复，还是已经偷偷篡位。
3. stale finally、stale snapshot、stale pointer 有没有被正式阻止。
4. compile-time gate、runtime gate 与 artifact surface 我有没有混成一层。
5. 如果作者今天离开，后来维护者还能不能直接看出哪里最危险。
6. 我现在夸的是更好看的分层，还是更硬的 present-state protection。
7. 写路径是不是要求 fresh-read 才能写，还是旧快照仍可越权落盘。
8. 如果 host / domain / capability 目录变化了，旧授权会不会继续假装有效。
9. 如果删掉漂亮的目录图，这套先进性还剩下什么。

## 6. 对 Agent Runtime 设计者的直接启发

如果你想复制 Claude Code 的源码成熟度，先复制：

1. single-writer surface
2. anti-zombie invariant
3. recovery asset non-sovereignty
4. policy-ordered bootstrap
5. release-surface shaping
6. freshness gate
7. ghost-capability eviction

而不是先复制：

- 更多 helper
- 更碎的目录

## 7. 一句话总结

真正先进的内核，不是更会分层，而是更会阻止过去写坏现在；如果 present state 仍能被旧对象轻易重写，再漂亮的结构都只是当前快照更顺眼。
