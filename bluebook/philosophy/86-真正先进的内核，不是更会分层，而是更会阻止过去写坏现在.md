# 真正先进的内核，不是更会分层，而是更会阻止过去写坏现在

这一章回答五个问题：

1. 为什么 Claude Code 的源码质量判断最终要回到“现在不能被过去写坏”。
2. 为什么 `one writable present`、recovery asset non-sovereignty、freshness gate 与 later maintainer rejectability 共同决定内核成熟度。
3. 为什么很多看起来更整洁的仓库，长期反而更难维护。
4. 怎样用苏格拉底式追问审一个新 runtime 是否真的拥有这种结构成熟度。
5. 这对 Agent runtime 设计者意味着什么。

## 1. 先说结论

真正先进的内核，不是：

- 文件更小
- 目录更碎
- 依赖更抽象

而是：

- 它把复杂度收进合法复杂度中心，并正式保护 `one writable present`

更完整一点说，这条源码质量判断必须同时成立三件事：

1. 复杂度中心合法：
   - 真正危险的编排、写回与 upgrade choke point 没被伪装成“更好看的分层”。
2. `one writable present` 被正式保护：
   - 当前真相只能通过 current-truth surface 与 freshness gate 被宣布。
3. later maintainer 仍能独立 reject：
   - 后来的维护者不需要作者口头补充，也能指出哪里在越权写现在。

这里还应再多记一句：

- later maintainer rejectability 不是“读完能复述设计”，而是“不追全仓、不追作者，也能点名哪条 `event / snapshot / pointer / recovery asset` 正在越权，且知道第一退回层落在哪”。

所以这页不再把“防过去写坏现在”写成半个源码质量总论，而只回答：

- 合法复杂度中心怎样保护 `one writable present`

## 2. 外层 canonical ladder 应该怎么读

如果要从外层一路走到内核对象，最稳的 canonical ladder 仍然是：

- `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`

这条顺序不是另一套命名包装，而是 later maintainer 复核源码质量时必须遵守的证据顺序：

1. 先看 contract 有没有写清谁配宣布真相。
2. 再看 registry 有没有列出当前有哪些权威对象。
3. 再看 current-truth surface 有没有单一写回面。
4. 再看 consumer subset 有没有把不同消费者分开。
5. 最后才看 hotspot kernel 是否把危险复杂度合法收拢。
6. 公开镜像条件下，再补 mirror gap discipline 判断证据断口。

所以这条顺序是外层证据路径；这页内层真正要看的仍然只有：

- `one writable present`

## 3. 为什么先进结构首先处理的是时间，而不只是空间

很多结构分析都过度偏爱空间问题：

- 模块怎么分
- 目录怎么排
- 依赖怎么切

Claude Code 更深的一层提醒是：

- 时间维度同样会制造第二真相

真正危险的不是只有坏依赖，还有：

1. 旧 finally
2. 旧 snapshot
3. 旧 pointer
4. 旧 head assumption
5. 旧 validator 视角
6. 旧 capability 目录

所以先进内核真正首先要问的不是：

- 代码分得美不美

而是：

- 过去会不会在未来某个时刻把现在重写掉

## 4. Claude Code 怎样把 one writable present 写硬

这条 present-state protection 在源码里至少落成了四类硬 choke point：

1. `QueryGuard.end(generation)` 与 `forceEnd()`
   - stale finally 被挡在门外，旧查询即使晚回来，也不能替当前查询做清理决定。
2. `onChangeAppState()`
   - permission mode externalization 被收到单一 diff choke point，防止不同 mutation path 各自宣布对外真相。
3. `sessionIngress.ts` 的 `409 adoption`
   - 远端 append chain 才是当前真相，旧本地假设必须先让位。
4. `bridgePointer.ts` 的 freshness gate
   - 恢复资产只能帮助重连 present state，不能靠陈旧 pointer 篡位。

这些例子真正共同说明的不是“架构很细”，而是：

- stale writer 不得清理现在
- 恢复资产不得宣布现在
- 旧外部视角不得覆盖现在
- 过期 pointer 不得重新绑定现在

也因此，旧对象如果还想影响 present，通常只剩四种合法命运：

1. `invalidate`
2. `relink`
3. `adopt`
4. 降格成 `projection / metadata`

这里还要再多看一层：

- `remote session client / bridge host / direct connect / ssh / viewer / transport shell`

这些在成熟源码里首先不是功能域，而是主权域。它们分别在保护 ownership、reconnect、permission、flush gate、sequence resume 与 exit contract；如果把它们重新看成功能分组，later maintainer 就会再次把 transport、session 与 current truth 混成一团。

所以结构先进性不只是在防 stale write，也是在坚持：

- 换 transport，不换高层语义

## 5. 这页最该防的三种失真

源码质量母线在这一页真正该守住的 reject trio 是：

1. `layout-first drift`
   - 还在先谈目录更整齐、文件更小、分层更漂亮。
2. `recovery-sovereignty leak`
   - replay、pointer、恢复资产开始偷偷篡位 current-truth surface。
3. `surface-gap blur`
   - current-truth surface、consumer subset、hotspot kernel 说不清边界。

这页的主反证应始终是：

- `recovery-sovereignty leak`

因为只要恢复资产开始篡位，`one writable present` 就已经失效。

源码质量线还应再固定一个 counterfeit test：

- 若一页还在先称赞目录观感、文件大小、层次漂亮，再补 `one writable present`
- 那它仍是在用 layout 美学偷换 local opposability
- 更稳的顺序应始终是：先问 present 是否单写，再问 later maintainer 是否能正式反对，最后才看 layout

另一个必须前置的失真也该记住：

- `truth-plane collapse`
  `event truth / current truth / display truth` 被重新混写，宿主或 UI 开始靠 replay transcript 猜现在。

一旦出现这类 collapse，哪怕没有立刻写坏文件，current-truth surface 也已经在制度上失效。

## 6. 苏格拉底式追问

如果要审一个新 runtime 是否真的把“过去不得写坏现在”写成制度对象，先问：

1. 当前哪一个 current-truth surface 有资格宣布当前真相。
2. recovery asset 在帮助恢复，还是已经偷偷篡位。
3. stale finally、stale snapshot、stale pointer 有没有被正式阻止。
4. later maintainer 能不能直接指出第一条 reject path。
5. 写路径是不是要求 fresh-read 才能写，还是旧快照仍可越权落盘。
6. 如果删掉漂亮的目录图，你还能否指出 sole writer、fresh-read、reject stale writer 与第一退回层。
7. remote、session 与 transport 在这里是功能组，还是主权边界。
8. 现在宣告 present 的，到底是 `event truth`、`current truth` 还是 `display truth`；如果三者混写，谁在越位。

## 7. 对 Agent Runtime 设计者的直接启发

如果你想复用 Claude Code 的源码判断，先复用：

1. 合法复杂度中心
2. current-truth surface
3. recovery asset non-sovereignty
4. freshness gate / anti-zombie invariant
5. later maintainer reject path

如果继续把这组启发再压成最短动作链，也只该剩：

- `surface -> split writeback -> fresh-read -> reject stale writer -> retreat`

而不是先复制：

- 更多 helper
- 更碎的目录
- 更顺眼的层级图

## 8. 一句话总结

真正先进的内核，不是更会分层，而是把复杂度收进合法复杂度中心，用 `one writable present` 与 later maintainer rejectability 正式阻止过去写坏现在。
