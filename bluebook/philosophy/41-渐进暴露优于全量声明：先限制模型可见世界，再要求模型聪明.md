# 渐进暴露优于全量声明：先限制模型可见世界，再要求模型聪明

这一章回答四个问题：

1. 为什么 Claude Code 不喜欢一开始就把全部工具、全部 instructions、全部 agents 都暴露给模型。
2. 为什么“先限制模型可见世界，再要求模型聪明”比“先全量声明再事后拦截”更成熟。
3. 为什么这既是安全设计，也是 token 设计、prompt 设计与治理设计。
4. 这对 agent runtime 的构建者有什么一般化启发。

## 1. 先说结论

Claude Code 在能力设计上有一条非常强的偏好：

- 渐进暴露优于全量声明

更具体地说，它宁可：

- 先只暴露一部分工具
- 先只暴露稳定前缀
- 先只暴露已经发现过的 deferred tools
- 先把变化迁成 delta attachments

也不愿：

- 把所有可能性一开始全摊给模型

然后再期望：

- 模型自己别乱用
- 权限系统最后能兜住
- cache 不要碎
- token 不要爆

这条偏好本质上不是“节省一点 token”。

它是在做更深的一件事：

- 控制模型一开始能看见的世界

## 2. 为什么“全量声明”在 agent runtime 里往往更差

很多人会天然觉得：

- 先把所有工具和说明都给模型，模型才最强

但 Claude Code 的源码反复暗示相反结论。

因为一旦全量声明，至少会同时增加四种成本：

1. schema token 成本
2. cache churn 成本
3. 行动熵成本
4. 治理复杂度成本

举例来说：

- 工具越多，tool schema block 越肥
- 前缀越肥，任何字节变化击穿的损失越大
- 能力越全量暴露，模型越可能在不该调用时调用
- 策略越多源同时暴露，越难回答“谁说了算”

所以“全量声明”看起来像开放，实际常常更像：

- 把无序扩张提前

## 3. Claude Code 如何做渐进暴露

### 3.1 deferred tools 不是一开始全量暴露

`ToolSearch` 的设计已经说明：

- deferred tools 可以先不进主工具池
- 模型先通过 ToolSearch 发现
- runtime 再根据 discovered set 回填

这代表的不是一个小功能，而是一种哲学：

- 能力可以先存在，但不必先全量暴露

### 3.2 tool schema 被会话级锁定

`toolSchemaCache.ts` 明确说明：

- schema 位于请求高位
- 中途变化会击穿大量下游缓存

这意味着 Claude Code 并不接受：

- 运行中任何能力变化都实时映射到 schema 前缀

它更偏好：

- 能力世界在一次会话里尽量稳定

### 3.3 高波动说明被迁到 delta attachments

deferred tools delta、agent listing delta、MCP instructions delta 共同说明：

- Claude Code 会刻意把“变化的能力描述”从主前缀迁到尾部

这其实也是渐进暴露：

- 先给稳定壳
- 后给变化增量

### 3.4 治理层也在做渐进暴露

`strictPluginOnlyCustomization`、`allowManagedPermissionRulesOnly` 这类设置表明：

- 不是所有来源都能同时参与定义模型可见世界
- 一部分扩权输入只能来自更高信任来源

也就是说，渐进暴露不只发生在 tool 层，也发生在：

- source 层
- authority 层

## 4. 为什么这同时是安全设计、token 设计和 prompt 设计

因为“模型可见世界”一旦扩大，三类问题会同时发生：

### 4.1 安全问题

暴露越多能力，模型的行动熵越大，权限系统的末端拦截压力越大。

### 4.2 token 问题

暴露越多 schema 和高波动说明，prompt 越肥，缓存越容易碎。

### 4.3 prompt 问题

暴露越多变化项，前缀越难稳定，side-loop 越难共享。

所以 Claude Code 的成熟之处不是把这三类问题拆开，而是用同一个动作处理：

- 先减少初始暴露

## 5. 苏格拉底式追问

### 5.1 如果所有工具都值得先告诉模型，那为什么还需要 ToolSearch 和 deferred tools

这本身就是反证。

### 5.2 如果变化说明都放在主前缀里，为什么还要发明 delta attachments

因为主前缀太贵，不能拿来承载所有变化。

### 5.3 如果管理策略允许所有来源并列参与定义能力边界，系统还能稳定解释自己的行为吗

很难。

### 5.4 如果一个系统总是先全暴露，再事后拦截，它真的在治理复杂性吗

更像是在把复杂性推迟到账单末尾。

## 6. 对 Agent 构建者的启发

如果想复制 Claude Code 的强点，最值得学的不是：

- 如何把更多能力更快塞给模型

而是：

1. 先决定哪些能力必须立即暴露。
2. 让其余能力按发现、权限、上下文预算逐步暴露。
3. 把变化放到尾部或增量，而不是主前缀。
4. 把“限制模型可见世界”当作第一步，而不是最后一步。

## 7. 一句话总结

Claude Code 真正高明的地方，不是一开始给模型更多能力，而是先限制模型可见世界，再通过渐进暴露、delta 回写与高信任来源收口，让能力在更低熵的条件下逐步展开。
