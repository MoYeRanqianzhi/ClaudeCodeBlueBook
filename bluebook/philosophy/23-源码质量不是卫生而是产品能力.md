# 源码质量不是卫生而是产品能力

这一章回答四个问题：

1. 为什么 Claude Code 的源码质量值得被上升为产品能力讨论。
2. Agent runtime 里的“高质量源码”到底指什么，而不只是“函数短、命名好”。
3. 为什么显式 contract、失败语义、恢复路径和共享基础设施会直接决定用户体验。
4. 从公开源码镜像里，我们该如何严谨地判断先进性与技术债。

## 1. 先说结论

Claude Code 的源码质量之所以值得单独讨论，不是因为“代码写得干净”这种卫生层意义，而是因为：

- 结构质量本身正在决定产品能力上限

如果没有：

- 显式 contract
- 显式失败语义
- 显式恢复语义
- 显式 cache 语义
- 显式角色子集

那么最终用户感受到的就会是：

- prompt 会抖
- 宿主会猜
- 多 Agent 会乱
- 远程会失真

在 Agent runtime 里，源码质量不是附属美德，而是能力来源。

## 2. Agent runtime 里的高质量，首先是 contract quality

传统应用里，很多复杂性可以藏在调用点后面。

Agent runtime 不行，因为它必须持续协调：

- 模型
- 工具
- 任务
- 宿主
- 状态
- 远程连接

所以真正高质量的第一标志不是“文件漂不漂亮”，而是：

- 关键约束有没有被做成显式 contract

Claude Code 可见源码里，这类 contract 随处可见：

- 命令元数据合同
- 工具子集合同
- task type / task state 合同
- control request / response 合同
- cache break detection 合同

这意味着新能力不是随手接进来，而必须沿着既有 contract 进入系统。

## 3. 第二标志：失败语义被显式写成产品行为

在 Agent runtime 里，失败不是例外，而是常态的一部分：

- permission 被拒
- control response 重复
- token 预算超限
- prompt cache 被打碎
- remote 会话断连

高质量源码的关键，不是消灭失败，而是让失败具备：

1. 明确语义
2. 明确外化
3. 明确恢复路径

这正是为什么 Claude Code 的显式失败语义能直接变成产品能力：

- 宿主不需要猜
- 用户不容易被假成功欺骗
- 恢复逻辑能回到稳定分支

## 4. 第三标志：恢复路径和共享基础设施被统一收口

很多系统的复杂性死在“每个功能各写一套恢复逻辑”。

Claude Code 更像是反过来：

- 先把恢复、自循环、预算、状态同步做成共享基础设施
- 再让具体能力沿着这些基础设施运行

`query.ts`、task runtime、control protocol、cache 检测都体现出同一偏好：

- 把复杂性收进统一层，而不是在产品层复制粘贴

这也是为什么即使某些热点文件很大，它看起来仍然像“有底盘的系统”，而不是“聪明技巧的堆积”。

## 5. 为什么热点大文件不自动等于架构差

热点文件当然是成本：

- 阅读成本更高
- 局部改动风险更集中
- 职责继续拆薄仍有空间

但热点文件本身不等于架构差。

更关键的问题是：

1. 关键 invariants 是否被集中管理。
2. 新能力是否沿统一 contract 接入。
3. 失败与恢复是否有统一语义。
4. 状态是否被外化给宿主。

如果这些答案都是“是”，那它就已经越过了卫生层，进入了产品层。

## 6. 公开镜像里的真实边界

这份判断也必须保持克制。

因为公开源码镜像明确不是完整 monorepo，所以更严谨的说法应当是：

- 当前可见的结构质量很高
- 当前可见的 contract / failure / recovery 设计很先进
- 但我们不能把它夸大成“完整测试闭环已经被公开证明”

也就是说：

- 先进性可判断
- 全貌不可臆测

## 7. 苏格拉底式追问

### 7.1 如果没有完整源码，为什么还敢谈先进性

因为我们谈的是：

- 当前可见代码里的结构判断

而不是：

- 对内部全貌的想象

### 7.2 如果热点文件很大，为什么不直接下结论说架构不好

因为大文件和结构好坏不是同一维度。

更关键的问题是：

- invariants 是否被集中管理
- 失败与恢复是否有统一语义
- 新能力是否沿共享基础设施接入

### 7.3 如果没有可见测试，为什么还能说它是高质量源码

更准确的说法应是：

- 当前可见的结构质量高，但测试闭环在公开镜像里不可见

这比直接说“高质量且已被充分测试”更严谨。

## 8. 一句话总结

在 Claude Code 这样的 Agent runtime 里，源码质量不是工程卫生，而是用户最终感知到的产品能力上限。
