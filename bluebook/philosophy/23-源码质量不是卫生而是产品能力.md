# 源码质量不是卫生而是产品能力

Claude Code 的源码质量之所以值得单独讨论，不是因为“代码写得干净”这种卫生层意义，而是因为：

- 结构质量本身正在决定产品能力上限

如果没有：

- 显式 contract
- 显式失败语义
- 显式恢复语义
- 显式 cache 语义

那么：

- prompt 会抖
- 宿主会猜
- 多 Agent 会乱
- 远程会失真

也就是说，在 Agent runtime 里，源码质量不是附属美德，而是用户最终感知到的能力来源。

这就是为什么 Claude Code 的先进性不能只写成功能多：

- 真正难抄走的，是把复杂性变成可持续结构的能力

从苏格拉底式反问看，判断一段 Agent 源码是否高质量，不该先问“函数短不短”，而该先问：

1. 这里的约束有没有被显式写出。
2. 这里的失败有没有明确语义。
3. 这里的恢复有没有稳定路径。
4. 这里的复杂性有没有被收进共享基础设施。

如果答案都是“有”，那这段质量就已经越过了卫生层，进入了产品层。
