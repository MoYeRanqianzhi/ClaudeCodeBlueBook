# 真正的设计单位不是功能，而是运行时平面

这一章回答五个问题：

1. Claude Code 为什么很难被功能清单准确解释。
2. prompt 魔力、安全设计、省 token、源码质量，为什么其实是同一个更深原则的不同投影。
3. 这个更深原则到底是什么。
4. 从第一性原理看，为什么 Agent IDE 必须这样设计。
5. 蓝皮书接下来应该如何用苏格拉底式追问避免再次写浅。

## 1. 先说结论

Claude Code 真正的设计单位不是 feature，而是 runtime plane。

也就是说，作者反复在设计的不是：

- 新增一个能力点

而是：

- 给某一类长期问题建立一个可持续运行的平面

至少可以看到六个平面：

1. prompt/context plane
2. permission/safety plane
3. token/context economics plane
4. state/recovery plane
5. collaboration plane
6. host/transport plane

prompt 魔力、安全设计、省 token、源码质量之所以会彼此呼应，是因为它们都在服务同一个目标：

- 把原本容易漂移、爆炸或失真的开放世界问题，压成边界清晰、可恢复、可组合的运行时平面

## 2. 为什么功能清单总是解释不完 Claude Code

如果只按功能列举，你当然可以写出：

- MCP
- skills
- subagents
- remote
- permissions
- resume
- compact

但这样仍然很难解释两个事实：

1. 为什么这些能力加在一起时没有完全互相打架。
2. 为什么它在 prompt、安全、token、工程质量上呈现出同一种“克制但强”的气质。

这是因为功能清单描述的是：

- 表层形态

而 Claude Code 真正稳定的东西是：

- 平面边界
- 平面之间的契约

## 3. 六个平面是怎样互相映射的

### 3.1 prompt plane

prompt plane 解决的不是“写一句好话”，而是：

- 角色合同
- 动态注入
- cache 稳定性
- 附件与提醒的边界

### 3.2 permission plane

它解决的不是“弹窗确认”，而是：

- typed decision
- rule / mode / automation / relay / hard boundary 的分层

### 3.3 token plane

它解决的不是“上下文快满了怎么办”，而是：

- 什么该成为稳定前缀
- 什么该被目录化/外移
- 什么只该作为输出文件
- 什么最后才值得 compact

### 3.4 state plane

它解决的不是“保存聊天记录”，而是：

- 当前真相
- 会话恢复
- file history
- context collapse
- sidechain / task output / metadata

### 3.5 collaboration plane

它解决的不是“能不能多开几个 agent”，而是：

- ownership
- 隔离
- mailbox / task bus
- coordinator / worker / fork / teammate 的角色合同

### 3.6 host plane

它解决的不是“CLI 怎么接 SDK”，而是：

- control loop
- event stream
- snapshot
- transport / adapter / subset

因此所谓“Claude Code 很强”，更准确地说是：

- 它把这些平面做得彼此相容

## 4. 这也解释了 prompt 魔力、安全设计、省 token、源码质量为什么会合流

### 4.1 prompt 魔力

因为 prompt plane 被做成了 contract，不是文案。

### 4.2 安全设计

因为 permission plane 被做成了 engine，不是 modal。

### 4.3 省 token

因为 token plane 被做成了经济系统，不是 compact 按钮。

### 4.4 源码质量

因为这些 plane 之间的 invariant 被写成了可复用基础设施，而不是散落技巧。

从更抽象的一层看，这四件事其实在重复同一个动作：

- 把隐性复杂度外化成显性结构

## 5. 从第一性原理看，Agent IDE 为什么必须这样设计

一个真正可持续工作的 Agent IDE，至少要同时面对：

1. 目标会变
2. 上下文会漂
3. 权限会失控
4. 输出会爆炸
5. 状态会中断
6. 协作会互相踩踏
7. 宿主会各自只消费一个子集

如果没有 runtime plane 的思维，这七个问题最后都会退化成：

- 局部补丁
- 一次性技巧
- 很快互相冲突

所以 Claude Code 的成熟，本质上不是“功能很多”，而是：

- 它拒绝用 feature 拼盘处理本该用平面处理的问题

## 6. 苏格拉底式追问

### 问：为什么 Claude Code 的 prompt 会有魔力？

答：因为 prompt 不是单独存在，它被 state、tool、cache、role 一起约束。

继续追问：

- 如果拿掉 tool ABI，只留 prompt，还会有同样魔力吗？
- 如果拿掉 attachment / cache 稳定性，只保留文案，还会一样吗？

答案都是否定的。

### 问：为什么它的安全设计看起来不靠大量 UX 提示？

答：因为真正起作用的是 decision plane 和 hard boundary。

继续追问：

- 如果只保留 modal，不保留 rule/sandbox/policy，会安全吗？

答案也是否定的。

### 问：为什么它省 token 但又不显得抠？

答：因为它先设计 plane，再设计压缩。

继续追问：

- 如果没有 deferred catalog、output externalization、stable prefix，光靠 compact 会够吗？

也不够。

### 问：为什么源码质量会直接转化成产品能力？

答：因为 plane 之间的 invariant 一旦写坏，产品就不是“稍差一点”，而是直接失真。

## 7. 对蓝皮书自身的反思

如果蓝皮书以后还只按功能写，就会反复犯三个错误：

1. 把同一设计原则拆成很多不相干的小结论。
2. 把 runtime plane 误写成 feature pile。
3. 把最有解释力的母题藏到细节下面。

更好的写法应该是：

- 主线先写 plane
- 专题再写 feature
- 最后再把 feature 放回 plane 里解释

## 8. 一句话总结

Claude Code 真正的设计智慧，不是做了很多功能，而是把 prompt、安全、token、状态、协作、宿主这些长期问题各自做成了可持续运行的平面。
