# 先让依赖关系诚实，再让抽象显得优雅

这一章回答四个问题：

1. 为什么 Claude Code 的模块化更像“依赖关系治理”，而不是“拆分文件数量竞赛”。
2. 为什么真正成熟的抽象，先要让依赖边诚实。
3. 为什么为了切断 runtime cycle，适度牺牲 DRY 往往是更高级的选择。
4. 这对 agent runtime 设计者有什么第一性原理启发。

## 1. 先说结论

Claude Code 的模块化哲学，可以压成一句话：

- 先让依赖关系诚实，再让抽象显得优雅

也就是说，它并不把“更漂亮的抽象”放在第一位。

它更优先关心：

1. 这个共享入口会不会把整条重依赖链拖进来。
2. 这个类型定义会不会反咬实现层。
3. 这个 helper 一旦 import，会不会闭合 runtime cycle。

所以它的优雅不是表面的“抽得多”，而是更硬的：

- 图更诚实

## 2. 为什么“抽象优先”有时反而会破坏系统

很多工程习惯天然鼓励：

- 先复用
- 先抽 helper
- 先统一入口

但在 agent runtime 这种高耦合系统里，这条路经常会意外制造：

- 隐性初始化顺序
- 大范围传染依赖
- 类型层与实现层互咬
- runtime edge 闭合

Claude Code 的源码反复证明：

- 抽象不是越早越好
- 先保住依赖图健康，抽象才值得成立

## 3. 为什么 tiny single-source file 是更成熟的优雅

真正值得跨层共享的东西，Claude Code 往往把它们压成：

- `pluginPolicy.ts`
- `normalization.ts`
- `systemPromptType.ts`

这种很小、很窄、职责很单一的文件。

这说明作者并不追求：

- 一个共享模块包打天下

而更偏爱：

- 共享真相尽量小

这种做法的价值在于：

- 它让复用成本和依赖代价一起下降

## 4. 为什么 anti-cycle seam 比“绝不允许脏边”更成熟

更幼稚的做法通常是：

- 一看到跨层依赖就强行重构

Claude Code 更成熟。

它承认：

- 有些依赖确实高位、确实脏、确实难消失

于是做法不是假装它不存在，而是：

- 把它关进 `queryContext` 这种受控 seam
- 限制调用者范围
- 在注释里明确写出 cycle path 和代价

这是一种更高级的诚实。

## 5. 为什么“适度不 DRY”有时是更强的工程选择

`teammateViewHelpers.ts` 一类代码最能说明这点。

为了切断 runtime edge，它宁可：

- 内联一个类型判断

也不愿为了“更 DRY”把 helper 拉进来。

这看起来像在退步，但其实更接近真正的系统工程：

- 不要为局部复用，付出全局依赖图失真的代价

所以 Claude Code 的很多选择都在提醒你：

- DRY 不是最高原则
- 依赖图健康才是

## 6. 苏格拉底式追问

### 6.1 如果一个共享入口被 200 多个文件依赖，它还应该继续长胖吗

显然不该。

### 6.2 如果一个类型定义会让实现层互相 import，对抽象优雅的坚持还值钱吗

不值。
它已经开始破坏图结构了。

### 6.3 如果一个 helper 的“复用”代价是闭合 runtime cycle，这种复用还叫进步吗

往往不叫。

### 6.4 真正成熟的模块化，优先保证的到底是什么

不是文件小，而是：

- 边界真
- 图不撒谎

## 7. 对 Agent 设计者的启发

如果想学 Claude Code，最值得抄的不是：

- 更多 util
- 更多 base class

而是：

1. 高扇入入口尽量做薄。
2. 共享判定尽量做单点小文件。
3. 不可避免的脏边显式隔离。
4. 愿意为图健康牺牲一点抽象洁癖。

## 8. 一句话总结

Claude Code 的模块化之所以显得成熟，不是因为抽象得更漂亮，而是因为它先让依赖关系诚实，再允许抽象发生。
