# 先把世界压缩成可治理语法，再让模型变聪明

这一章回答四个问题：

1. 为什么 Claude Code 的 prompt 魔力首先来自“世界表达方式”，而不是“模型说话方式”。
2. 为什么真正强的 agent runtime，优先做的是工作语法，而不是提示词文案。
3. 为什么 protocol transcript、tool schema、delta attachments 本质上都在服务同一目标。
4. 这对 agent 设计者意味着什么。

## 1. 先说结论

Claude Code 在 prompt 上最深的一条原则是：

- 先把世界压缩成可治理语法，再让模型变聪明

也就是说，它并不假设：

- 原始软件工程现场
- 原始聊天记录
- 原始工具宇宙

可以直接拿给模型。

它更相信：

- 世界必须先被整理成可行动、可缓存、可恢复、可审计的语法面

模型的聪明，是在这个语法面上释放出来的。

## 2. 为什么“世界先被整理”比“prompt 写得更好”更重要

一个 coding agent 真正难的地方，从来不是多写几句指导词。

真正难的是：

1. 模型现在看见的世界是不是稳定。
2. 模型可调用的动词是不是清楚。
3. 高波动对象是不是被隔离。
4. 对话历史是不是已经被翻译成可继续工作的协议。

如果这些事没先做好，再好的 prompt 文案也会被脏上下文、漂移能力面和破碎 transcript 抵消掉。

所以 Claude Code 的优先级非常成熟：

- 先治理世界的表达
- 再让模型在这个表达里推理

## 3. 苏格拉底式追问

### 3.1 如果 UI transcript 直接等于模型 transcript，会更真实吗

未必。

它更可能只是把 UI 真相误当协议真相。

### 3.2 如果所有工具一开始都暴露给模型，会更强吗

未必。

它也可能只是让模型面对更混乱的动词表。

### 3.3 如果高波动知识直接混进主前缀，是不是上下文更全

表面更全，实际更碎。

因为稳定语法被打坏了。

### 3.4 真正的 prompt 魔力更像什么

更像：

- 运行时把世界整理成模型容易做对事的形状

而不是：

- 运行时替模型多说了几句漂亮的话

## 4. 对 Agent 设计者的启发

如果想复制 Claude Code 的强点，最该先学的不是：

- system prompt 长什么样

而是：

1. 怎么切分稳定前缀和动态补丁。
2. 怎么把能力表写成清晰语法。
3. 怎么把高波动信息迁出主工作集。
4. 怎么把 UI 世界重编译成协议世界。

## 5. 一句话总结

Claude Code 的 prompt 魔力之所以强，不在于它更会措辞，而在于它先把工作世界压缩成可治理语法，再让模型在这个语法里变聪明。
