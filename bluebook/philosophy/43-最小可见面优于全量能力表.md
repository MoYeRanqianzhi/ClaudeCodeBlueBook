# 最小可见面优于全量能力表

这一章回答四个问题：

1. 为什么“给模型更多能力”不总等于“让系统更强”。
2. 为什么 Claude Code 更偏爱最小必要可见面，而不是全量能力表。
3. 为什么这条原则同时改进安全、成本、稳定性与解释性。
4. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 很深的一条设计原则可以压成一句话：

- 最小可见面优于全量能力表

这里的关键词不是：

- 能力少

而是：

- 当前轮只暴露必要能力

这和“全量能力表”最大的差异在于：

- 全量能力表默认把一切选择先交给模型
- 最小可见面先由 runtime 决定哪些选择值得被看见

这是一种更成熟的 agent 观：

- 不是先相信模型足够聪明
- 而是先把模型放进一个更低熵的世界

## 2. 为什么全量能力表看起来强，实际常常更脆

全量暴露最诱人的地方在于：

- 好像模型知道得越多越强

但 Claude Code 的源码恰好说明，代价会同时发生在四个地方：

1. 前缀更肥
2. cache 更脆
3. 行动熵更高
4. 治理更难解释

所以全量能力表不是单纯“开放”。

它常常意味着：

- 把无序复杂度提前

## 3. 最小可见面为什么更符合第一性原理

从第一性原理看，一个 agent runtime 首先要回答的不是：

- 系统理论上会什么

而是：

- 当前轮什么值得进入模型世界

只有这个问题先回答好，后面的：

- 执行
- 恢复
- 压缩
- 调优

才有稳定基础。

所以 Claude Code 真正高明的地方，不是把能力表做大，而是：

- 把能力暴露做成控制动作

## 4. 为什么这条原则不是保守，而是更积极的工程进攻

表面上看，最小可见面像是在克制。

但更深地看，它其实是在主动争取四种更稀缺的东西：

1. 更低的 token 常驻成本
2. 更高的缓存复用率
3. 更低的错误行动熵
4. 更清楚的运行时解释力

也就是说，Claude Code 不是在“少给模型一点”，而是在：

- 为更长期的系统稳定性抢预算

## 5. 苏格拉底式追问

### 5.1 如果更多能力总是更好，为什么还要 deferred tools

因为“更多能力先暴露”本身也是成本。

### 5.2 如果所有变化都值得写进主前缀，为什么还要 delta attachments

因为并不是所有变化都配得上主前缀的位置。

### 5.3 如果所有来源都值得同时定义模型世界，为什么还要 managed-only 输入裁剪

因为 authority 本身也必须收口。

### 5.4 真正强的系统，应该先给模型更多选择，还是先给模型更低熵的选择空间

Claude Code 显然押注后者。

## 6. 对 Agent 设计者的启发

如果你想复制 Claude Code 的强点，最值得学的不是：

- 如何更快地把全部能力展示给模型

而是：

1. 先定义最小立即可见面。
2. 把其余能力改成发现式、增量式、权限式进入。
3. 把能力暴露本身当作安全与成本的一部分。
4. 先降低模型的世界熵，再要求模型聪明。

## 7. 一句话总结

Claude Code 之所以显得更强，不是因为它把能力表做得更全，而是因为它更认真地控制了模型此刻真正能看见的世界。
