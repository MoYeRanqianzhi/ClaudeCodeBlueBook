# 宿主控制平面优于聊天外壳

这一章回答四个问题：

1. 为什么 Claude Code 需要显式的宿主控制平面，而不是把终端输出交给外部 UI 自己猜。
2. `StructuredIO`、`RemoteIO`、control request/response 暗示了怎样的设计哲学。
3. 这对“什么是 Agent IDE / Agent runtime”有什么第一性原理上的启发。
4. 研究者在这一点上最容易犯什么错。

## 1. 先说结论

Claude Code 的一个关键设计押注是：

宿主应该接入 runtime control plane，而不是只消费聊天外壳。

因为只要系统真的要长期工作，它就必须让宿主显式参与：

- 权限裁决
- 中断与取消
- 状态查询
- MCP 与插件控制
- 远程保活与重连
- 环境变量刷新

这些动作都不可能通过“读 assistant 文本”稳定完成。

关键证据：

- `claude-code-source-code/src/entrypoints/sdk/controlSchemas.ts:57-173`
- `claude-code-source-code/src/entrypoints/sdk/controlSchemas.ts:175-519`

## 2. 从第一性原理看，为什么“聊天外壳”不够

假设我们从零开始设计一个 coding agent，并且它要被接进：

- 终端
- IDE
- Web
- 远程会话
- bridge 环境

那么宿主至少要解决四个不可约问题：

1. 它怎么知道当前动作是否需要权限批准。
2. 它怎么中断一个正在运行的执行链。
3. 它怎么查询当前上下文、设置、连接状态。
4. 它怎么在网络抖动、远程重连、长任务挂起时保持一致性。

如果系统只提供“assistant 文本输出”，这四件事全都会变成脆弱的猜测。

因此只要目标真的是 agent runtime，控制平面就不是可选项，而是基础设施。

## 3. Claude Code 在这里押注了什么

### 3.1 押注协议，而不是抓 stdout

`control_request` / `control_response` / `control_cancel_request` / `keep_alive` 的存在说明，作者没有把 stdout 当成“顺便传一点机器可读信息”的地方，而是显式定义了一条协议。

### 3.2 押注关联，而不是事件乱飞

`request_id`、`pendingRequests`、schema parse、duplicate/orphan 防护说明，作者不接受“宿主自己想办法配对”这种松散集成方式。

### 3.3 押注取消，而不是阻塞弹窗

`control_cancel_request`、hook 与 host 的竞速、桥接注入 response，都说明宿主不是被动看客，而是参与仲裁的节点。

### 3.4 押注统一语义，而不是每个宿主一套 API

`RemoteIO extends StructuredIO` 暗示得很清楚：

- 远程不是新产品
- bridge 不是新协议
- direct connect 也不是新语义

它们只是同一控制平面的不同投射。

## 4. 这反过来说明 Claude Code 不是终端应用，而是 host-integrated runtime

如果把 Claude Code 当终端应用，会自然得到这样的错觉：

- UI 才是主角
- 协议只是辅助
- 远程只是 transport
- 权限只是弹窗

但源码显示恰好相反：

- runtime 才是主角
- 协议是宿主接入 runtime 的正式边界
- 远程是在搬运控制平面与状态面
- 权限是执行链中的可仲裁步骤

所以 Claude Code 的本体更像：

- 一个可被不同宿主接入的 agent runtime

而不是：

- 一个“在终端里看起来比较强”的聊天程序

## 5. 为什么这件事重要

这直接影响我们如何评价 Claude Code 的强弱。

如果把它看成聊天外壳，评价标准会退化成：

- 回复质量高不高
- 文本流顺不顺
- UI 好不好看

如果把它看成宿主控制平面，评价标准就会变成：

- 协议是否清晰
- 权限与取消是否可组合
- 宿主能否接到状态真相
- 远程与本地是否共享语义
- 连接中断与迟到响应是否被工程化处理

后者才更接近工程系统的真实难度。

## 6. 苏格拉底式反问：研究者最容易在哪里把宿主控制平面写坏

### 6.1 我们是否仍把 Claude Code 写成 terminal shell

很容易。

一旦我们只强调：

- REPL
- slash command
- assistant 输出

就会低估 control protocol、host callbacks、remote session 的地位。

### 6.2 我们是否把 transport 与 control plane 混成一层

也很容易。

`WebSocket`、`SSE`、`HTTP POST`、`keep_alive` 这些是 transport 语义；
`can_use_tool`、`interrupt`、`get_context_usage`、`mcp_set_servers` 才是 control plane 语义。

不把这两层拆开，就会误以为“换一种网络连接方式”就等于“换了一套产品能力”。

### 6.3 我们是否高估了所有宿主的能力对等性

也需要警惕。

当前可见源码里：

- `StructuredIO` 是最完整的控制平面
- direct connect 和 remote session manager 只实现了较窄子集

所以“schema 支持存在”不等于“所有宿主已完全支持”。

## 7. 研究宿主接入时的最小检查表

只要研究 SDK、远程、bridge、宿主接入，就必须先问自己：

1. 这里描述的是 UI 壳，还是 runtime control plane。
2. 这里描述的是 transport，还是 control semantics。
3. 这里描述的是完整宿主能力，还是某个窄化适配器。

如果这三问不先答清，后面的分析几乎一定会漂。

## 8. 一句话总结

Claude Code 之所以不像普通“AI 编程聊天框”，不是因为它会输出更多文本，而是因为它把宿主、权限、状态、取消和远程都纳入了统一控制平面。
