# 真正成熟的验证底盘，不是更多实验室，而是回放、改写与台账共享同一持续执行语义

实验室层最容易让人误以为：

- 只要 replay case 足够多，系统就成熟了

但 Claude Code 式系统真正要保护的不是：

- 某次 replay 是否通过

而是：

- 同一 shared object 的回放、改写与台账是否已经进入同一条持续执行语义

所以当蓝皮书走到 Harness Runner / Drift Ledger 这一层时，真正该收束的判断是：

- 好验证底盘不是更会做实验室，而是更会把 replay、rewrite adoption 与 drift ledger 写成下一次继续执行的输入

## 1. 第一性原理

如果一个系统已经有：

- rule packet
- sample kit
- evaluator harness
- replay lab

却仍没有 runner / ledger，

那么它依旧只是：

- 会证明一次

而不是：

- 会让这次证明继续约束下一次执行

这意味着它仍然可能出现：

- 每次实验室都能复现同一个问题

但：

- 生产路径依然在重复忘记同一 drift

## 2. Prompt 的魔力，为什么最后要落到 runner / ledger

Prompt 最容易被误写成：

- 某次回放命中了正确 reject reason

但更接近 Claude Code 真相的说法是：

- Prompt 的魔力来自 continuation 语义能被持续保存、持续比较、持续恢复

如果同一 `prompt_object_id` 的 drift 只存在于某次 replay，而没有进入：

- replay queue
- stable bytes ledger
- rewrite adoption

那么所谓的“Prompt 魔力”就会重新退回：

- 原文崇拜
- 绿灯崇拜
- 摘要崇拜

所以真正成熟的 Prompt runner，不是：

- 让实验室一直重放

而是：

- 让 continuation 的 drift 和修复都变成下一轮继续执行前必须面对的正式对象

## 3. 安全与省 Token，为什么最后要落到 runner / ledger

安全最容易被误写成：

- 拦得更严

省 token 最容易被误写成：

- 停得更早

但更接近 Claude Code 真相的说法是：

- 两者都在持续维护同一 `decision_window` 是否仍然值得继续

如果同一 `decision_window` 的消失只存在于某次 replay，而没有进入：

- arbitration ledger
- rollback object
- upgrade adoption

那么系统就会重新退回：

- 状态色
- 计数
- verdict
- 摘要

所以真正成熟的治理 runner，不是：

- 把更多 gate 编排起来

而是：

- 让“没有决策增益就不继续”的语义在每次继续前都被重新消费

## 4. 源码先进性，为什么最后也要落到 runner / ledger

源码先进性最容易被误写成：

- 目录更清楚

但更接近 Claude Code 真相的说法是：

- 先进结构会把 split-brain、stale writer 与恢复边界持续写进 runtime

如果同一 `structure_object_id` 的断裂只存在于某次 anti-zombie replay，而没有进入：

- recovery asset ledger
- dropped stale writers
- rollback boundary

那么系统就会重新退回：

- 目录图
- 恢复成功率
- 作者说明

所以真正成熟的结构 runner，不是：

- 再跑一次 anti-zombie case

而是：

- 让恢复资产、权威路径与 stale writer 清退证据成为每次继续执行的正式条件

## 5. 为什么 Runner 比 Replay Lab 更成熟

Replay Lab 只能说明：

- 这套 reject semantics 能被重放

Runner 才能说明：

- 这套 reject semantics 会继续约束下一次执行

两者的差别，就像：

- 法院判过一次

和：

- 这条判决已经进入后续执法、复议、归档与修法

Claude Code 值得学的地方，不在于实验室更多，

而在于：

- 它总在把“一次判断”推进成“下一次继续执行前必须面对的正式对象”

## 6. 苏格拉底式自反

在你准备宣布“我们已经有成熟验证底盘了”前，先问自己：

1. 我保存的是 replay 结果，还是保存了下一次继续执行的判断条件。
2. drift ledger 记录的是 shared object 断裂，还是只记录了症状统计。
3. rewrite adoption 恢复的是正式对象，还是只恢复了文档表面。
4. 如果删掉仪表盘、摘要、目录图和成功率，这套 runner 是否还能成立。
5. 我是在扩建实验室，还是在把运行时逐步变成会记住自己为什么该停、该改、该回退的系统。
