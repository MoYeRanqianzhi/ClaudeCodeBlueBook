# 安全、成本与体验只是同一治理收费链的三种外观

这一章回答四个问题：

1. 为什么 Claude Code 的安全设计和省 token 设计应该一起看。
2. 权限系统、治理系统与上下文预算系统本质上在共同控制什么。
3. 为什么体验并不是第三个独立目标，而是治理收费链是否设计正确的外部表现。
4. 这条统一治理收费链对 Agent 平台设计者有什么启发。

## 1. 先说结论

很多系统把三件事拆开看：

1. 安全
2. 成本
3. 体验

Claude Code 更像是在说：

- 这三件事只是同一条治理收费链的三种外观

更准确地说，这三件事共用的不是抽象预算器，而是同一条治理收费链：

- `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`

原因很简单：

- `governance key` 决定谁配改边界
- `externalized truth chain` 决定宿主承认什么当前真相
- `typed ask` 与 `decision window` 决定当前扩张是否可执行
- `continuation pricing` 与 `durable-transient cleanup` 决定时间与收尾是否继续收费

这三者表面不同，本质上都在限制系统的无序扩张。

所以更准确的说法不是：

- “安全和成本要权衡，体验放后面补”

而是：

- “安全、成本与体验共同决定 runtime 是否仍然可治理”

## 2. 动作空间如何沿治理收费链收费

Claude Code 在动作空间上至少有五道闸门：

1. permission mode
2. allow / deny rules
3. tool-specific safety
4. classifier / auto mode 校验
5. policy / org / managed settings

更关键的是，这些闸门并不是只在工具调用那一刻才出现。

它们会提前影响：

- 工具是否进入模型可见工具池
- 某些运行模式是否允许切换
- remote / channels / managed surface 是否允许启用
- auto / bypass 是否会被异步撤回或改写

这说明 Claude Code 的安全系统并不是“最后弹个确认框”，而是一条持续收束动作空间的治理收费链。

## 3. 上下文空间如何沿治理收费链收费

在上下文空间上，Claude Code 也有完整预算系统：

1. tool result budget 先控制大块输出。
2. compact / microcompact / autocompact 再控制历史消息。
3. reactive compact 负责异常恢复。
4. token budget continuation 把“用户要求多花 token”变成受约束继续。
5. token estimation 在 API usage 不完整时保守估算，避免过晚爆窗。

这里最关键的不是“省 token”三个字，而是：

- 让上下文始终保持在可运行、可恢复、可解释的工作集范围内

所以 token 经济也不是成本优化器那么简单，它本质上同样在维护系统治理能力。

## 4. 为什么体验并不是独立第三项

从用户视角看，体验常常表现成：

- 为什么这个动作被拦住了
- 为什么这个任务还能继续
- 为什么这里突然 compact
- 为什么多 Agent 没有乱套

这些问题表面是体验问题，根子上却都是治理收费链设计问题。

如果预算器设计错误，常见后果就是：

- 更安全，但卡到不可用
- 更省 token，但上下文断裂
- 更流畅，但边界失控

所以体验不是第三个独立优化目标，而是动作空间与上下文空间是否沿同一治理收费链被正确外化、恢复与续费的外部结果。

## 5. 为什么治理层也属于同一张图

很多系统把治理写成后台设置页。

Claude Code 的做法更接近：

- 治理直接进入主路径并塑造当前轮 runtime

它会影响：

- trust 建立前后能做什么
- remote / channels / MCP surface 是否允许暴露
- permission mode 是否会被策略撤回
- 当前状态是否要同步给宿主和远端

也就是说，治理不是安全层外面再套一层，而是这条治理收费链的高阶来源。

## 6. 苏格拉底式追问

### 6.1 为什么安全系统不能只靠 prompt 约束

因为 prompt 只能建议，不能：

- 从工具池里移除能力
- 阻止策略不允许的 remote path
- 保证 mode 变化被宿主看见

### 6.2 为什么 token 经济不能只靠 `/compact`

因为等到 `/compact` 才动作，通常已经太晚。

真正有效的系统必须更早就做：

- 输出外置
- delta 注入
- 保守估算
- 运行中恢复

### 6.3 为什么这两套系统必须和宿主状态同步一起写

因为治理收费链如果不外化，宿主就只能猜：

- 当前为什么不能做
- 当前为什么还能继续做
- 当前为什么切 mode 或 compact

而 Claude Code 显然不接受这种“让宿主自己猜”的设计。

## 7. 一句话总结

Claude Code 的安全、成本与体验之所以能同时成立，不是因为三者被单独优化，而是因为它用同一条治理收费链同时收束动作空间、上下文空间和认知负担。
