# 安全、成本与体验只是同一治理收费链的外显

这一章回答四个问题：

1. 为什么 Claude Code 的安全设计和省 token 设计应该一起看。
2. 权限系统、治理系统与上下文预算系统本质上在共同控制什么。
3. 为什么体验并不是第三个独立目标，而是治理收费链是否设计正确的外部表现。
4. 这条统一治理收费链对 Agent 平台设计者有什么启发。

## 1. 先说结论

很多系统把三件事拆开看：

1. 安全
2. 成本
3. 体验

Claude Code 更像是在说：

- 这三件事只是同一条治理收费链的外显

这条链就是：

- `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`

更准确地说：

- `governance key` 决定谁配改边界
- `externalized truth chain` 决定宿主当前被允许承认什么真相
- `typed ask` 决定扩张能否进入协商路径
- `decision window` 决定当前继续是否仍有决策增益
- `continuation pricing` 决定时间和 token 是否还配继续付费
- `durable-transient cleanup` 决定什么该留下，什么必须退场

所以更准确的说法不是：

- “安全和成本要权衡，体验放后面补”

而是：

- “安全、成本与体验共同暴露同一条治理链有没有写对”

## 2. 为什么体验不是第三项目标

从用户视角看，体验常常表现成：

- 为什么这个动作被拦住了
- 为什么这个任务还能继续
- 为什么这里突然 compact
- 为什么宿主现在看得懂 mode、pending action 和上下文压力

这些问题表面是体验问题，根子上却分别对应：

1. 被拦住：
   - `governance key -> typed ask`
   - 谁配改边界、当前 ask 到底是 `deny / ask / allow` 哪一种。
2. 还能继续：
   - `decision window -> continuation pricing`
   - 当前还有没有继续资格，以及继续会不会开始变成免费扩张。
3. 突然 compact：
   - `continuation pricing -> durable-transient cleanup`
   - working set 已不配继续免费霸占席位，系统开始结算和清理。
4. 宿主看得懂当前状态：
   - `externalized truth chain`
   - host 不再自己猜 mode、tool visibility、pending action 和 usage，而是消费被正式外化的当前真相。

所以体验不是独立第三项，而是：

- 同一治理收费链有没有被正确外化、解释与结算

## 3. 动作空间与上下文空间为什么必须沿同一条链收费

Claude Code 在动作空间上至少会提前塑形：

1. policy / org / managed settings
2. permission mode
3. allow / deny rules
4. tool-specific safety
5. classifier / auto mode 校验

在上下文空间上又会继续塑形：

1. tool result budget
2. compact / microcompact / autocompact
3. reactive compact
4. token budget continuation
5. conservative token estimation

表面看像两套系统，底层其实都在回答同一个问题：

- 当前扩张是否已经被正式定价

也因此，安全不是“最后弹个确认框”，省 token 也不是“最后做个 compact”。

它们都在保护：

- runtime 是否仍然可治理

## 4. 为什么治理层必须进入主路径

很多系统把治理写成后台设置页。

Claude Code 的做法更接近：

- 治理直接进入主路径并塑造当前轮 runtime

它会影响：

- trust 建立前后能做什么
- remote / channels / MCP surface 是否允许暴露
- permission mode 是否会被策略撤回
- 当前状态是否要同步给宿主和远端

所以治理不是安全层外面再套一层，而是：

- 同一收费链的主键来源

## 5. 这页真正要防的三种失真

治理母线在这一页真正该守住的 reject trio 是：

1. `projection usurpation`
   - mode 条、token 条、弹窗、流畅感这些投影开始冒充治理真相本身。
2. `decision-window collapse`
   - `Context Usage` 只剩成本面板，不再进入下一步决策。
3. `free-expansion relapse`
   - 动作、可见性、上下文或时间重新获得免费扩张资格。

这页的主 reject 应始终是：

- `projection usurpation`

因为一旦投影开始篡位，体验就会再次被误写成独立主题。

## 6. 苏格拉底式追问

### 6.1 为什么安全系统不能只靠 prompt 约束

因为 prompt 只能建议，不能：

- 从工具池里移除能力
- 阻止策略不允许的 remote path
- 保证 host 看到的当前真相仍然可信

### 6.2 为什么 token 经济不能只靠 `/compact`

因为等到 `/compact` 才动作，通常已经太晚。

真正有效的系统必须更早就做：

- 外化真相
- 定义 ask
- 判断 decision window
- 给 continuation 正式定价

### 6.3 为什么体验不能被单独优化

因为一旦安全、成本和体验各自独立优化，最后得到的通常是：

- 更安全，但不可用
- 更省 token，但断裂
- 更流畅，但边界失控

## 7. 一句话总结

Claude Code 的安全、成本与体验之所以能同时成立，不是因为三者被单独优化，而是因为它们只是 `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup` 这条治理收费链的不同外显。
