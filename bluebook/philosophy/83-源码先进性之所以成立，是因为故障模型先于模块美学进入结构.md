# 源码先进性之所以成立，是因为故障模型先于模块美学进入结构

这一章回答五个问题：

1. 为什么 Claude Code 的源码先进性不该被理解成“目录更合理”。
2. 为什么 authority surface、dependency seam、transition legality、recovery asset 与 anti-zombie 会共同决定结构上限。
3. 为什么真正先进的结构会先把故障模型写进去，而不是先追求模块美学。
4. 为什么未来维护者之所以重要，是因为他们首先会消费这套故障模型。
5. 这对 agent runtime 设计者意味着什么。

- `claude-code-source-code/src/state/onChangeAppState.ts:43-92`

这些锚点共同说明：

- Claude Code 的源码先进性，不只是“结构看起来不错”，而是“故障模型已经被提前写进结构，所以后来者能更快拒绝错误实现”。

## 1. 先说结论

真正先进的源码，不是：

- 没有大文件
- 目录很漂亮

而是：

1. 真相有明确 authority surface。
2. 依赖图通过 seam 主动暴露危险改动面。
3. 跨时间写回通过 generation 与 fresh merge 主动防 zombie。
4. recovery asset 被严格限定为恢复资产，而不是第二权威面。
5. worktree、dirty git 与 transport 也被当成正式故障面，而不是外围脚手架。
6. 后来维护者一眼就知道哪些改动会破坏制度对象。

从第一性原理看，源码先进性的真正本质不是：

- 模块美学

而是：

- 故障模型已经被提前编码

## 2. 第一性原理：协议优先于 UI，故障优先于美学

如果一个系统已经有：

- 分层
- 模块
- 接口

却还没有先保住：

- 单一权威入口
- 单一共享真相来源
- await 之后的 fresh-safe 写回
- 未来维护者可见的危险改动面

那么它依旧只是：

- 结构化的当前实现

而不是：

- 可演化的长期内核

Claude Code 值得学的地方在于，它不断把“可能怎么坏”写进结构，而不是只把“现在怎么分层”写进目录。

## 3. 第一层：authority surface 先回答“哪里会说谎”

如果一个系统没有明确 authority surface，那么后来维护者面对问题时只能：

- 到处找状态
- 到处猜入口

Claude Code 更成熟的地方在于，它让后来者先问：

- 如果系统开始说谎，应该先怀疑哪一个权威面

这会把结构先进性从“入口更整齐”提升成：

- 反对意见有正式落点

## 4. 第二层：dependency seam 先回答“哪里最危险”

好的 seam 不只是帮助阅读，它还帮助拒绝错误扩张。

像 `query/config`、`query/deps`、`queryContext` 这样的文件之所以值钱，不是因为它们小，而是因为它们明确告诉你：

- 哪些依赖可以进
- 哪些依赖进来就会打穿结构诚实性

也就是说，源码先进性真正保护的不是：

- 轻量文件本身

而是：

- 依赖图继续说真话

## 5. 第三层：mode choke point 与命名空间先回答“谁能扩张世界”

成熟架构的一个关键标志，不是模式切换多优雅，而是：

- 外部消费者看到的模式，永远由单一 choke point 投影出来

同样，工具命名空间也不是便利约定，而是：

- 防止模型、插件或 hook 靠猜名字意外 reach 到本不该触达的能力面

这意味着真正成熟的结构，连“谁有资格说自己是什么模式”“谁有资格暴露自己是什么工具”都先被结构化治理了。

## 6. 第四层：transition legality 先回答“过去会不会写坏现在”

很多结构只关心静态依赖，不关心时间维度。

Claude Code 更高级的地方在于，它把：

- QueryGuard generation
- fresh-state merge
- stale writer drop
- terminal re-check

都写成正式结构要求。

这意味着它在问的不是：

- 代码现在能不能跑

而是：

- 过去的对象会不会在未来某个时刻写坏现在

这正是故障模型优先于模块美学的体现。

## 7. 第五层：recovery asset 先回答“恢复会不会篡位”

恢复设计最容易自欺的地方是：

- 只要能恢复回来就算好设计

Claude Code 更接近的真相是：

- recovery asset 必须帮助回到主权对象，但永远不能自己变成主权对象

这会把恢复系统从“功能补丁”改写成：

- 权威边界管理

真正先进的结构，会提前防止：

- pointer、ledger、resume file 长成第二真相

## 8. 第六层：worktree、dirty git 与终端协议也属于源码先进性

很多团队会把：

- worktree
- dirty git 保护
- structured IO / SSE / NDJSON 传输

看成外围工程细节。

Claude Code 更接近的真相是：

1. `canonical root + slug 校验 + dirty/unpushed fail-closed` 是防副作用扩张的结构设计。
2. 终端协议里对重复响应、重放、分隔、重试和 fire-and-forget 的处理，是把 I/O 当正式协议面，而不是打印层。
3. 保守并行、可取消、顺序不乱，本身就是架构正确性的一部分。

所以源码先进性不只存在于模块树，也存在于它怎样防止真实工作区被误伤、怎样防止 transport 层制造第二真相。

## 9. 第七层：anti-zombie 让未来维护者真正拥有反对能力

如果 anti-zombie 只剩注释，那后来维护者拿到的只是：

- 作者的提醒

而不是：

- 结构给出的正式反对能力

Claude Code 值得学的地方在于，它经常把危险写法提前变成：

1. generation guard
2. stale-safe merge
3. delta patch
4. fresh-state re-check

这说明源码先进性最深的地方不是：

- 作者当下写得更聪明

而是：

- 后来者不用求助作者，也能拒绝错误实现

## 10. 苏格拉底式追问

### 10.1 为什么源码先进性不是目录美学

因为目录美学能帮助浏览，却不能直接帮助你：

- 找到权威面
- 识别危险 seam
- 拒绝 zombie 写回

### 10.2 为什么故障模型写进结构比“代码风格统一”更值钱

因为系统真正昂贵的失败，几乎都来自：

- 权威漂移
- 依赖失真
- stale write
- recovery 篡位

### 10.3 为什么未来维护者是正式消费者

因为只有他们真正会检验：

- 这套结构能不能在作者缺席时继续说真话

### 10.4 为什么 worktree 与 transport 也属于源码质量

因为真正昂贵的 bug，常常不是功能没写出来，而是：

- 错把真实工作区当测试场
- 错把重复事件当新真相
- 错把旧状态写回现在

### 10.5 为什么 QueryGuard 比“代码风格统一”更值钱

因为它把时间维度上的故障模型从经验文化变成了正式结构。

## 11. 对 Agent 设计者的启发

如果想学 Claude Code，最该抄走的不是：

- 更漂亮的目录树

而是：

1. 给关键真相建立明确 authority surface。
2. 用 seam 把危险依赖面主动暴露出来。
3. 把 generation、fresh merge 与 stale drop 变成正式结构要求。
4. 严格限制 recovery asset 的主权边界。
5. 把 anti-zombie 写成可执行故障模型，而不是口头文化。
