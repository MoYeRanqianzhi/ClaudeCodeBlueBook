# 真正成熟的规则，不是更多检查，而是不同消费者共享同一拒收语义

规则系统最容易被误解成：

- 多加几条检查

但 Claude Code 式系统真正要保护的不是检查数量，而是：

- 不同消费者在不同时间点，是否仍然共享同一条拒收语义

所以当蓝皮书走到 Artifact Rule ABI 这一层时，真正该收束的判断是：

- 好规则不是更会挑错，而是更早阻止 shared object 被不同角色各自重新解释

## 1. 第一性原理

如果一个系统已经有：

- shared contract
- samplebook
- drift casebook
- validator / linter

却仍没有 machine-readable rule ABI，

那么它依旧只是：

- 会在不同地方重复判断同一件事

而不是：

- 会让不同消费者共享同一判断后果

这意味着它仍然允许：

- reviewer 通过
- CI 放行
- handoff 拒收

三者围绕同一对象却说出三种不同的话。

## 2. Prompt 的魔力，为什么最后要落到共享拒收语义

Prompt 最容易被神化成：

- 文案更强

但更接近 Claude Code 真相的说法是：

- Prompt 的魔力来自同一 `compiled request object` 在不同消费者那里拥有同一条拒收边界

如果宿主卡、CI 附件、评审卡与交接包只是共享字段，却不共享拒收语义，

那么它们迟早会重新分裂成：

- 原文崇拜
- 绿灯崇拜
- 总结崇拜
- 摘要崇拜

所以成熟的 Prompt rule ABI 真正统一的，不是：

- 都知道哪些字段重要

而是：

- 都会在同一 shared object 断裂时做出同一种拒收

## 3. 安全与省 Token，为什么最后要共享同一拒收语义

安全最容易被退回成：

- 拦住危险动作

省 token 最容易被退回成：

- 少花一点预算

但更接近 Claude Code 真相的说法是：

- 两者都在决定：当前这轮继续是否还值得

如果治理规则不共享同一拒收语义，

那么系统就会重新分裂成：

- 安全面说“先拦住再说”
- 成本面说“再省一点看看”
- 交接面说“先继续试试”

而不是共享同一句话：

- 没有 decision gain，就不该继续；没有 rollback object，就不该交接；没有 failure semantics，就不该宣称判断已成立

## 4. 源码先进性，为什么最后也要共享同一拒收语义

源码先进性最容易被退回成：

- 结构漂亮

但更接近 Claude Code 真相的说法是：

- 先进结构会让 split-brain、stale writer 与口头交接在不同消费者那里同时失去合法性

如果结构规则不共享同一拒收语义，

那么系统就会重新分裂成：

- 宿主说“目录图挺清楚”
- CI 说“恢复通过了”
- reviewer 说“结构更清晰”
- handoff 说“作者会解释”

而真正成熟的结构规则应该共享同一句话：

- authoritative path 不清、recovery asset 不明、anti-zombie evidence 不足、rollback object 缺失时，不同消费者都该拒绝继续

## 5. 为什么 Rule ABI 比“更多规则”更成熟

更多规则，只能说明：

- 各处都在做判断

共享拒收语义，才能说明：

- 各处都在围绕同一对象做同一种判断

两者的差别，就像：

- 多个部门各自写制度

和：

- 多个部门共用同一部宪法

Claude Code 值得学的地方，不在于它有很多 gate、很多 mode、很多 trace，

而在于：

- 它不断把“分散判断”压回“共享拒收语义”

## 6. 苏格拉底式自反

在你准备宣布“我们已经有成熟规则 ABI 了”前，先问自己：

1. 这套规则共享的是同一拒收语义，还是只是很多长得相似的本地规则。
2. 如果删掉原作者解释，CI、reviewer 与 handoff 是否仍会在同一条件下拒绝继续。
3. 当前 `rewrite hint` 修的是 shared object continuity，还是只修展示格式。
4. 规则通过后，不同消费者是否仍在围绕同一对象、同一失败语义、同一回退对象说话。
5. 我是在让系统更会挑错，还是在让系统更不容易重新说谎。
