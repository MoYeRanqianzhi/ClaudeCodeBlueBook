# 真正成熟的源码地图，不是目录列得更细，而是更快暴露权威入口、消费者子集与危险改动面

源码地图最容易被误写成：

- 把目录树展开得更长

但 Claude Code 式系统真正需要保护的不是：

- 读者看到更多文件名

而是：

- 读者更快知道哪里在宣布真相、哪里只是消费它、哪里一改就会破坏系统级不变量

所以当蓝皮书走到源码地图层时，真正该收束的判断是：

- 好的源码地图不是更细的目录清单，而是更快暴露权威入口、消费者子集与危险改动面

如果再往下压一层，它还必须继续暴露三件事：

1. 证据梯度从哪里开始变硬
2. 时间上的第二真相最容易从哪里潜入
3. future maintainer 如果不同意当前实现，应先沿哪条入口链拒收它

这里还应再多记一句：

- 源码地图页只配做定位与跳转，不重签 present-truth frontdoor；源码质量线真正的 canonical ladder 仍只认 `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`。

## 1. 第一性原理

如果一份地图已经列出了：

- `commands/`
- `tools/`
- `services/`

却仍然没有说明：

- 谁是权威入口
- 谁只是 consumer subset
- 谁是 internal-only
- 谁是危险改动面

那么它依旧只是：

- 一个更长的目录树

而不是：

- 一张能帮助后来者降低误改成本的源码地图

更稳的阅读顺序应先变成：

1. `contract`
2. `registry`
3. `current-truth surface`
4. `consumer subset`
5. `hotspot kernel`
6. `mirror gap discipline`

如果把源码地图前门继续压成最短判断句，它也只该优先暴露权威入口、消费者子集、危险改动面与镜像缺口说明；地图页定位句不再单独写成一条公式链，免得 later maintainer 把地图词误读成源码质量线 canonical formula。

## 2. Prompt 效力，为什么也需要源码地图

Prompt 最容易被误解成：

- system prompt 写得好

但更接近 Claude Code 真相的说法是：

- Prompt 的效力散布在 `query`、`services/api`、`services/compact`、`tools` 的多个目录里

如果地图不能更快暴露：

- 哪些文件在编译 request truth
- 哪些文件在维护 stable bytes
- 哪些文件在维护 lawful forgetting

那么读者就会重新退回：

- 盯某个 prompt 文件

而错过真正的协作语法底盘。

## 3. 安全与省 Token，为什么也需要源码地图

安全最容易被误解成：

- 某个权限模块更严

省 token 最容易被误解成：

- 某个 compact 模块更会压缩

但更接近 Claude Code 真相的说法是：

- 两者都散布在 `permissions`、`tools.ts`、`structuredIO`、`sessionState`、`services/compact` 这些不同目录里

如果地图不能更快暴露：

- 哪些 authority file / truth-surface 在宣布控制平面真相
- 哪些对象只是 host projection
- 哪些文件只是在承载工具定义层

那么读者就会重新退回：

- 只看某个单模块

而错过真正的一体化定价秩序。

## 4. 源码先进性，为什么尤其体现在地图质量上

源码先进性最容易被误解成：

- 目录更整齐

但更接近 Claude Code 真相的说法是：

- 真正先进的源码，会让后来者更快看出哪里能改、哪里不能乱改、哪里只是子集投影

这就是为什么好的源码地图最终要回答的不是：

- 目录里有什么

而是：

1. 哪个 authority entry / entrypoint 真的在说话。
2. 哪个 current-truth surface 更接近 sole writer 或 choke point。
3. 哪些对象只是 consumer subset 或消费者暴露。
4. 哪些 hotspot kernel / mirror gap discipline 一改最可能放大 blast radius。

还要继续回答：

5. 哪些 stale write / recovery object / projection 最容易在时间上撒谎。
6. 哪些 `failure semantics` 其实已经是 architecture doctrine，而不是尾部错误处理。
7. future maintainer 离开作者记忆后，先靠什么独立反驳当前实现。

如果继续用苏格拉底式自校这页，也应先追三句：

1. 我是不是把地图定位词重新升成了 present-truth frontdoor。
2. 我是不是在用“更快找到文件”偷换 `current-truth surface / hotspot kernel / mirror gap discipline` 的制度判断。
3. 我是不是忘了这页只负责定位与跳转，而不是重签 canonical ladder。
4. 我是不是又把目录当 authority 单位，而不是只把它当 locator。

## 5. 为什么这会降低误改成本

一旦源码地图能暴露上述四件事，后来者会自然获得四种能力：

1. 他会先改权威入口，而不是在投影层打补丁。
2. 他会先看 consumer subset，再判断产品承诺边界。
3. 他会先看危险改动面，再决定是否需要更高层审查。
4. 他会先看 internal-only / gated surface，再决定某段代码是否能拿来当公开能力。

这就是为什么好的源码地图本身就是治理工具。

因为它不是在帮你：

- 更快找到文件

而是在帮你：

- 更快找到正确的真相层
- 更快找到时间上最危险的第二真相
- 更快保住后来维护者的 reject 能力

## 6. 苏格拉底式自反

在你准备宣布“这份源码地图已经足够好了”前，先问自己：

1. 我暴露的是权威入口，还是只暴露了更多目录名。
2. 我是不是仍先沿 `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline` 校过这张地图，而不是另写一套地图定位规则。
3. 我区分了 consumer subset 与正式承诺，还是默认代码存在就算公开能力。
4. 我标出了危险改动面与 failure semantics，还是让后来者继续靠直觉踩雷。
5. 这份地图在帮读者更快理解系统，还是只在帮读者更快迷路。
