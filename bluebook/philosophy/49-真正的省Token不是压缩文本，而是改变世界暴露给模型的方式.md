# 真正的省Token不是压缩文本，而是改变世界暴露给模型的方式

这一章回答四个问题：

1. 为什么 Claude Code 的省 token 设计不能被理解成“更会压缩文本”。
2. 为什么安全和成本控制共享同一条第一性原理。
3. `Narrow / Later / Outside` 为什么比“多打一层摘要”更接近本体。
4. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 对省 token 的更深理解是：

- 真正的省 token，不是把同一个世界压得更短

而是：

- 改变这个世界暴露给模型的宽度、时间和位置

所以它真正偏爱的不是：

- 更短文本

而是：

- 更窄的可达世界
- 更晚的暴露时机
- 更外部的位置安放

## 2. 为什么这和安全是同一条原则

安全要解决的是：

- 模型不该立刻获得过大的行动空间

省 token 要解决的是：

- 模型不该立刻背负过大的上下文空间

这两件事的底层损失函数是一样的：

- 世界一旦无序扩张，系统就会同时更危险、更贵、更难稳定

所以 Claude Code 才会反复偏爱：

- 先裁剪
- 再暴露
- 能外置就外置

## 3. 苏格拉底式追问

### 3.1 如果一个对象真的有用，就应该立刻进入主 prompt 吗

不一定。

“有用”和“应该现在进入主工作集”不是一回事。

### 3.2 如果通过外置和延迟暴露省下 token，这是不是在削弱能力

不一定。

很多时候是在保护能力不被噪声和漂移稀释。

### 3.3 为什么“先全给模型，再靠模型自己节制”不够好

因为那相当于把 runtime 治理外包给模型自律。

Claude Code 明显不信这一套。

### 3.4 真正成熟的省 token 更像什么

更像：

- runtime 重新安排世界出现的顺序、范围与位置

## 4. 对 Agent 设计者的启发

如果想学 Claude Code，最该先问的不是：

- 怎么把文本再压短一点

而是：

1. 哪些东西本来就不该现在出现。
2. 哪些东西应该先以搜索/引用方式出现。
3. 哪些东西应该迁到外部表面。
4. 哪些东西一旦进入前缀就会破坏稳定性。

## 5. 一句话总结

Claude Code 真正的省 token，不是更会压缩文本，而是更会重新安排世界暴露给模型的方式。
