# 协议全集不等于适配器子集

这一章回答四个问题：

1. 为什么看到 schema 全集，不代表每个宿主都已经拥有同样能力。
2. 这种“全集 vs 子集”的区分，为什么是 Agent runtime 研究里最容易犯错的地方。
3. Claude Code 在这方面透露了怎样的设计内涵。
4. 研究多宿主能力边界时，应如何持续保持严谨。

## 1. 先说结论

在 Claude Code 里，协议全集和适配器子集是两个完全不同的层次：

- `controlSchemas.ts` 描述的是协议全集
- `StructuredIO` 实现的是完整控制平面主路径
- bridge、direct connect、`RemoteSessionManager` 只是其中不同宽度的适配器子集

如果不把这三层拆开，分析就会立刻滑向两个错误：

1. 误把“schema 存在”写成“宿主已完整支持”
2. 误把“某个适配器很薄”写成“整个系统能力很薄”

## 2. 从第一性原理看，为什么一定会出现“子集”

一个真正可扩展的 runtime，只要要接多种宿主，就几乎必然出现这种结构：

1. 先定义统一协议全集。
2. 再定义核心执行路径。
3. 最后由不同宿主按自身需要接入其中一部分。

否则系统会陷入两种坏结果之一：

- 每个宿主一套私有协议，系统碎裂。
- 强行要求所有宿主支持全集，接入成本失控。

所以“有全集，也允许子集”不是缺陷，而是多宿主 runtime 的自然结果。

## 3. Claude Code 在这里的真实押注

Claude Code 并没有选择：

- 每个宿主自己定义一套消息语义

而是选择：

- 先有统一 control plane
- 再允许 bridge / direct-connect / remote-session manager 做不同宽度的投射

这背后反映的是一种成熟的工程观：

- 统一语义优先于局部完备

也就是说，系统宁可让某些宿主先只支持一个诚实的子集，也不愿意把不同宿主变成语义彼此漂移的孤岛。

## 4. 为什么研究者尤其容易在这里失真

因为文档作者最容易犯的错，恰好就是把这层边界抹平。

### 4.1 第一种错：把 schema 当产品现实

看到 control subtype 列表后，很容易顺手写成：

- Claude Code 支持 A、B、C、D、E……

但更准确的问题应该是：

- Claude Code 协议层声明了 A、B、C、D、E……
- 当前这个宿主适配器真的实现了哪些

### 4.2 第二种错：把薄适配器当系统本体

看到 direct connect 或 `RemoteSessionManager` 很窄，又容易反过来写成：

- Claude Code 的远程面其实只支持权限和 interrupt

这同样错误，因为它把窄适配器误当成完整控制平面。

### 4.3 第三种错：把“诚实子集”误判成“不成熟”

其实一个适配器支持子集，本身不是问题。

真正的问题是：

- 子集是否被诚实暴露
- 边界是否清晰
- 未支持能力是否会回 error 而不是假成功

bridge 在 outbound-only 模式下主动回 error，就是一个很好的反例：它宁可明确拒绝，也不制造“看起来成功但本地没生效”的错觉。

## 5. 苏格拉底式反问：我们会不会再次把两层写混

会，而且概率很高。

因为“全集”天生比“子集”更整齐，写起来也更爽。
但真正负责任的写法，必须持续追问：

1. 这是 schema 事实，还是适配器事实。
2. 这是完整控制平面，还是某个宿主子集。
3. 这是 transport 差异，还是能力差异。

只要这三问没答清，分析就很容易从工程文档退化成漂亮但危险的总览。

## 6. 这条原则反过来解释了 Claude Code 的设计内涵

Claude Code 在这里体现出的，不只是“接口多”，而是一种更深的系统态度：

- 先统一协议
- 再允许宿主按需裁剪
- 但始终维持语义边界清晰

这也是为什么它更像 runtime，而不是聊天应用。

聊天应用通常更偏好：

- 每个前端各写各的

而 runtime 更偏好：

- 统一协议
- 分层适配
- 子集诚实

Claude Code 明显站在后者。

## 7. 研究能力迁移时的最小检查表

只要研究 host / remote / bridge / direct connect / control protocol，就必须先标三层：

1. 协议全集
2. 控制平面主路径
3. 当前适配器子集

如果这三层没有同时出现，就说明写法仍然不够严谨。

## 8. 一句话总结

在多宿主 Agent runtime 里，真正成熟的不是“所有宿主立刻支持全集”，而是“协议全集清晰、适配器子集诚实、两者边界始终不被混写”。
