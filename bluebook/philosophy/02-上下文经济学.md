# 上下文经济学

## 1. 先说结论

Claude Code 的一个核心哲学是：  
上下文不是垃圾桶，而是预算。

它不是试图“永远把更多东西塞进 prompt”，而是在不停做三件事：

1. 减少不必要进入上下文的内容。
2. 把必要内容以更便宜的形式进入上下文。
3. 在必须回收时，优先保留行动价值而不是表面完整性。

## 2. 为什么说它把上下文当预算

从源码看，这种思路不是一句口号，而是到处可见：

- 动态 agent list 改成 attachment，避免 cache bust
- tool result budget 先限制 aggregate result size
- snip / microcompact / autocompact / reactive compact 多层回收
- prompt-too-long 错误先 withheld，再尝试恢复
- fork subagent 时明确强调“不要偷看输出文件”，避免把噪音拉回主上下文

证据：

- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-64`
- `claude-code-source-code/src/tools/AgentTool/prompt.ts:76-95`
- `claude-code-source-code/src/query.ts:369-468`
- `claude-code-source-code/src/query.ts:1065-1175`

## 3. 它如何让“昂贵信息”延迟进入上下文

### 3.1 动态 agent list 不再内联到 tool 描述

`AgentTool/prompt.ts` 明确写了一个非常工程化的理由：

- agent list 动态变化会导致 tool description 变化
- tool description 变化会导致 prompt cache 失效

所以系统选择在某些模式下把 agent list 作为 attachment message 注入，而不是放进工具描述正文。

证据：

- `claude-code-source-code/src/tools/AgentTool/prompt.ts:48-64`

这不是语义优化，而是 token economics。

### 3.2 Deferred tools / ToolSearch

Claude Code 不默认把所有工具细节都堆给模型，而是允许先 ToolSearch，再按需加载。

这也是一种上下文节约：  
不把“可能用不到的工具 schema”预先塞进会话。

## 4. 它如何回收已经进入上下文的内容

回收链至少包括：

- tool result budget
- snip
- microcompact
- context collapse
- autocompact
- reactive compact

证据：

- `claude-code-source-code/src/query.ts:369-468`
- `claude-code-source-code/src/query.ts:592-648`
- `claude-code-source-code/src/query.ts:1065-1175`

其中最值得注意的不是“有很多 compact”，而是顺序：

- 先做细粒度削减
- 再做更重的摘要化
- 最后才 surface error

这说明 Claude Code 在默认偏好上是“尽量保留可操作上下文”，而不是“尽快生成总结”。

## 5. 为什么 fork 也是上下文经济学的一部分

`AgentTool` 对 fork 的指导里有两句非常关键：

- fork yourself when the intermediate output isn't worth keeping in your context
- don't peek at output_file unless the user explicitly asks

证据：

- `claude-code-source-code/src/tools/AgentTool/prompt.ts:83-95`

这说明 fork 的本质不是并行，而是上下文隔离。  
并行只是副产品，真正目标是把中间噪音留在 worker 那边，不污染主线程。

## 6. 为什么它偏爱“文件、技能、配置”而不是纯聊天记忆

如果上下文是预算，那么长期记忆就不能只靠 transcript。  
Claude Code 更偏好把长期状态外化到：

- skill files
- CLAUDE.md / memory files
- settings
- transcript / session metadata

这是一种经济学意义上的转移：  
把长期信息从昂贵的上下文窗口，搬到更便宜、可重复读取的文件系统里。

## 7. 从第一性原理看，这在解决什么问题

如果 agent 的目标是长期工作，那么上下文真正需要承载的不是“所有发生过的事”，而是“下一步决策所需的最小充分信息”。

Claude Code 的上下文经济学，本质上是在回答：

1. 什么信息必须留在工作集里。
2. 什么信息可以外化。
3. 什么信息应该延迟加载。
4. 什么信息在必要时可以摘要化。

这和传统操作系统里的 cache / working set 管理，本质是同一种问题。

## 8. 一句话总结

Claude Code 的上下文观不是“大窗口神话”，而是工作集管理。它真正强的不是记得更多，而是知道什么该留下、什么该外化、什么该回收。
