# 可读性不是修辞，而是防误改、防误扩展、防失忆的系统能力

这一章回答四个问题：

1. 为什么 Claude Code 的可读性不该被理解成代码风格好看。
2. 为什么命名、注释和显式边界在这里本质上是治理工具。
3. 为什么源码同时服务机器执行和人类维护是更高级的工程目标。
4. 这对 agent runtime 设计者意味着什么。

## 1. 先说结论

Claude Code 的可读性，更准确地说是：

- 防误改、防误扩展、防失忆的系统能力

它的价值不止在：

- 读起来舒服

更在：

- 更难被错误理解
- 更难被错误修改
- 更难把制度性约束丢回团队口头记忆

## 2. 为什么这不只是审美

在简单代码里，可读性也许更像审美。

在 Claude Code 这种高耦合 runtime 里，可读性直接影响：

1. 未来维护者是否知道哪里不能动。
2. 新功能是否会无意击穿某条边界。
3. 依赖图纪律是否还能被长期守住。
4. snapshot / authority / cache 语义是否还能被正确延续。

所以这里的可读性更像：

- 运行时制度的可见外壳

## 3. 苏格拉底式追问

### 3.1 为什么要把 `DANGEROUS`、`authoritative`、`single source` 直接写进名字里

因为风险应该在阅读阶段就被看见。

### 3.2 为什么要把 cycle、snapshot、future extraction 这些理由写进注释

因为不该只让老作者记得。

### 3.3 为什么 dependency-free leaf module 如此重要

因为它让规则更短、更可审计，也更难被误扩展。

### 3.4 真正高级的可读性更像什么

更像：

- 让系统自己把治理边界写给后来者看

## 4. 对 Agent 设计者的启发

如果想学 Claude Code，最该学的不是：

- prettier 配得多整齐

而是：

1. 哪些风险应该写进命名。
2. 哪些制度理由应该写进注释。
3. 哪些共享规则必须压成 dependency-free leaf module。
4. 怎样让代码既能被机器执行，也能被人类治理。

## 5. 一句话总结

Claude Code 的可读性之所以成熟，不在于它更漂亮，而在于它把治理边界也写进了代码本身。
