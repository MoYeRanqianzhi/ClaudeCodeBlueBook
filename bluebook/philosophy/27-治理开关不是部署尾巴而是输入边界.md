# 治理开关不是部署尾巴而是输入边界

这一章回答四个问题：

1. 为什么 `channelsEnabled`、allowlist、managed settings、OAuth gate 不能被视为部署尾巴。
2. 这些治理开关实际上在改写什么。
3. 为什么治理层和 prompt / 安全 / token 经济属于同一运行时问题。
4. 这对平台设计和蓝皮书写法有什么约束。

## 1. 先说结论

Claude Code 的治理开关不是后部署时代码外壳，而是运行时输入边界的一部分。

它们至少在改变三件事：

1. 什么输入能进入当前会话
2. 什么配置能覆盖当前运行时
3. 什么外部对象被系统视为可信来源

所以治理开关不是外围配置，而是 runtime contract 的一部分。

## 2. 为什么 channels 是最典型例子

channels 表面上像“多一个 MCP 能力”，但从源码看，它真正受控的是：

- capability
- claude.ai OAuth
- org policy
- `allowedChannelPlugins`
- session `--channels`
- dev channel bypass

也就是说，channels 不是“多一个工具”，而是：

- 让外部消息进入当前会话的输入边界

输入边界一变，prompt、权限、上下文预算和用户信任都会一起变。

## 3. 为什么 managed settings 也不是普通配置

`remoteManagedSettings` 与安全确认链说明：

- 某些托管设置一旦变化，会触发阻断式安全确认

这表明系统自己也承认：管理策略变更可能重写执行边界，它不是 UI 偏好，而是运行时本体变化。

## 4. 为什么这也和 token 经济、体验设计有关

治理层不只影响安全，还影响：

1. 体验
2. 成本
3. 协作

因此治理开关和 token 预算、前台控制面并不是平行问题，而是共享同一约束系统。

## 5. 苏格拉底式追问

### 问：为什么很多系统会把治理层做得很糟

答：因为它们把治理层视为“部署后再补的限制器”，而不是系统一开始就该承认的输入边界。

### 问：为什么这会连带拖垮 prompt 与宿主体验

答：因为一旦输入边界不清楚，系统就无法稳定解释为什么某些消息能进来、为什么某些插件能用、为什么某些策略会覆盖本地配置。

## 6. 一句话总结

Claude Code 的治理开关之所以重要，不是因为企业功能多，而是因为它们直接定义了什么输入和什么信任可以进入当前 runtime。
