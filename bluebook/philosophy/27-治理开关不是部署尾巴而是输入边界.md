# 治理开关不是部署尾巴而是输入边界

这一章回答四个问题：

1. 为什么 `channelsEnabled`、allowlist、managed settings、OAuth gate 不能被视为部署尾巴。
2. 这些治理开关实际上在改写什么。
3. 为什么治理层和 prompt / 安全 / token 经济属于同一运行时问题。
4. 这对平台设计和蓝皮书写法有什么约束。

## 1. 先说结论

Claude Code 的治理开关不是后部署时代码外壳，而是运行时输入边界的一部分。

它们至少在改变三件事：

1. 什么输入能进入当前会话
2. 什么配置能覆盖当前运行时
3. 什么外部对象被系统视为可信来源

所以治理开关不是外围配置，而是 runtime contract 的一部分。

## 2. 为什么 channels 是最典型例子

channels 表面上像“多一个 MCP 能力”，但从源码看，它真正受控的是：

- capability
- claude.ai OAuth
- org policy
- `allowedChannelPlugins`
- session `--channels`
- dev channel bypass

也就是说，channels 不是“多一个工具”，而是：

- 让外部消息进入当前会话的输入边界

输入边界一变，prompt、权限、上下文预算和用户信任都会一起变。

## 3. 代表性源码锚点

- `claude-code-source-code/src/services/mcp/channelAllowlist.ts:1-16` 把 channel trust 明确绑定到可热更新的 allowlist，而不是仅靠本地安装事实
- `claude-code-source-code/src/services/mcp/channelAllowlist.ts:46-75` 区分总开关 `tengu_harbor` 与纯 allowlist 预判，说明治理先于功能注册
- `claude-code-source-code/src/services/mcp/channelPermissions.ts:1-23` 直接把 channel permission relay 解释为新的信任边界，而不是普通消息转发
- `claude-code-source-code/src/services/mcp/channelPermissions.ts:169-193` 只有同时满足连接、session allowlist 与 capability opt-in 的客户端才成为权限面
- `claude-code-source-code/src/services/mcp/channelNotification.ts:175-315` gate 顺序明确是 capability -> runtime gate -> auth -> org policy -> session -> allowlist
- `claude-code-source-code/src/services/remoteManagedSettings/securityCheck.tsx:15-30` 托管设置危险变更会触发阻断式安全检查
- `claude-code-source-code/src/utils/settings/types.ts:896-920` `channelsEnabled` 与 `allowedChannelPlugins` 被正式纳入组织级 settings schema
- `claude-code-source-code/src/utils/plugins/pluginPolicy.ts:11-19` 插件还能被 policy 强制禁用，说明输入边界与扩展边界本就同层

这些锚点共同说明，治理层在改写“谁能进入会话、谁能改写运行时、谁被视为可信来源”。

## 4. 为什么 managed settings 也不是普通配置

`remoteManagedSettings` 与安全确认链说明：

- 某些托管设置一旦变化，会触发阻断式安全确认

这表明系统自己也承认：管理策略变更可能重写执行边界，它不是 UI 偏好，而是运行时本体变化。

## 5. 为什么这也和 token 经济、体验设计有关

治理层不只影响安全，还影响：

1. 体验
2. 成本
3. 协作

因此治理开关和 token 预算、前台控制面并不是平行问题，而是共享同一约束系统。

## 6. 这对蓝皮书写法提出什么约束

以后所有治理相关章节都应先交代：

1. 这是在改 capability、auth、policy、session 还是 allowlist。
2. 这一层改变的到底是功能可见性，还是输入边界。
3. 它是否会连带改写权限、prompt、上下文预算与用户信任。

如果只写“企业配置项有哪些”，就仍然没有写到治理的本体。

## 7. 苏格拉底式追问

### 问：为什么很多系统会把治理层做得很糟

答：因为它们把治理层视为“部署后再补的限制器”，而不是系统一开始就该承认的输入边界。

### 问：为什么这会连带拖垮 prompt 与宿主体验

答：因为一旦输入边界不清楚，系统就无法稳定解释为什么某些消息能进来、为什么某些插件能用、为什么某些策略会覆盖本地配置。

## 8. 一句话总结

Claude Code 的治理开关之所以重要，不是因为企业功能多，而是因为它们直接定义了什么输入和什么信任可以进入当前 runtime。
