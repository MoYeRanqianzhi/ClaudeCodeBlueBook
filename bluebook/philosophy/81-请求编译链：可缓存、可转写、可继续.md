# Prompt 的效力来自把 same-world request object 编译成可缓存、可转写、可继续的工作对象

这一章回答五个问题：

1. 为什么 Claude Code 的 Prompt 看起来更有效，但答案并不在文案修辞。
2. 为什么优先顺序、section registry、dynamic boundary 与 stable prefix 会决定 Prompt 的真实上限。
3. 为什么 `message lineage` 的三键内核与 projection consumer 分层不是消息层细节，而是 Prompt 效力的一部分。
4. 为什么 lawful forgetting、compact、continue 与 threshold liability 属于同一条 Prompt 编译链。
5. 这对 agent runtime 设计者意味着什么。

## 0.1 Canonical Witness Ownership Table

如果问题已经变成“哪一跳 witness 到底由谁签、谁生产、谁可 first veto、谁只是消费”，单源回答统一留在这里，不再回 `30 / 54 / 06` 这些下游页局部补语义：

| witness hop | 签什么 | owner | producer | local veto cue | consumer | 不是 signer 的 carrier |
|---|---|---|---|---|---|---|
| `message_lineage_ref` | 当前工作对象仍挂在哪条历史骨架上 | same-world 会话主语 | message assembly / tool pairing | `parentUuid / logicalParentUuid / message.id / tool_use_id` 任一失配 | transcript builder、resume、review | display history、summary prose |
| `section_registry_ref` | 哪组 stable rules 仍在定义同一世界 | stable prompt constitution | section compiler / loader | section 缺席、重排、越过 dynamic boundary | request compiler、stable-prefix assembler | 零散 prompt 片段、复制粘贴规则 |
| `stable_prefix_ref` | 哪批稳定字节仍是同一前缀资产 | shared prefix custody | prompt assembly / cache-safe loader | prefix 无法解释 cache break，或频繁失稳 | model API、cache reuse、re-entry | sticky prompt、handoff note |
| `protocol_transcript_ref` | 哪份 transcript 仍是可执行协议对象 | normalized protocol truth | message normalization / tool-result pairing | pairing 失效、消息 ABI 失真、display transcript 冒充 protocol | model API、replay、forensics | UI transcript、human-readable chat log |
| `continuation_object_ref` | lawful forgetting 后还剩哪份可行动对象 | continuation object identity | compact / resume / handoff rebinding | `required witness` 缺失、已排除分支回流、rollback boundary 不明 | handoff、resume、later consumer | narrative summary、task recap |
| `continue_qualification_verdict` | 当前到底还能 continue、必须 reject，还是只剩 provisional | current continuation authority | continuation judge across stop hooks / cache break / budget / recovery surfaces | 第一声 `no` 落在 action、prefix、budget 或 re-entry boundary | host、human handoff、worker coordinator | “应该还能继续”的体感 |

更短地说，owner 决定哪一跳配签世界连续性，producer 只负责把它正式产出来，consumer 只负责按同一对象读取，`local veto` 决定第一声 `no` 应该落在哪里。凡是需要靠 carrier 重新解释 owner 或补签 verdict 的地方，都说明 witness chain 还没有真正站住。

更硬一点说，若 later consumer 追问的是“哪一跳该回哪一层补 witness”，这张表就是 canonical first answer；`30` 只消费它做审读 gate，`54` 只消费它做 lawful-consumer review，`06` 只消费它做自校，不再各自重发半套 ownership 语义。

## 1. 先说结论

Prompt 真正的效力，不是：

- 让模型当前这一轮更像懂你

而是：

1. 先把 system prompt 写成可分段、可缓存、可审计的 section registry。
2. 先把优先顺序压成同一条 request assembly order，而不是让多份并列文本互相争主语。
3. 先把 `message lineage` 写成可恢复的三键内核，而不是顺手拼出的聊天历史。
4. 先把 transcript 重写成合法协议对象，再让模型消费。
5. 先把动态变化赶到 stable boundary 之后，保住共享前缀资产。
6. 先规定删掉什么后系统还能继续，而不是一味保留更多历史。
7. 先让 cache break、continue 资格与失稳原因都能被正式解释。

从第一性原理看，Claude Code 的 Prompt 效力真正像的是：

- 一条编译链

而不是：

- 一段文案

## 2. 第一性原理：先固定谁能定义世界，再决定怎样描述世界

Claude Code 的 Prompt 有一个常被忽略的前提：

- 它不是简单的 `system + user`

而更接近：

- `systemPrompt + userContext + systemContext + protocol transcript`

这件事的重要性在于：

1. Prompt 的优先顺序先被固定。
2. 风格、流程、边界与工具法律被编译进 system 主链。
3. 人类 later 接手者消费的也不是裸原文，而是同一条被治理过的 request assembly result。

所以 Prompt 效力首先不是：

- 写得更强势
- 文风更顺滑

而是：

- 谁能定义当前世界、谁只能提供补充上下文，这条秩序先被钉死

## 3. 第二层：lineage kernel 优于“历史还在”

很多系统默认“只要历史没丢，继续就还能成立”。

Claude Code 更接近的真相是：

- 真正该被保住的不是一条更长的 transcript，而是 `message lineage` 的三键内核

也就是：

1. `parentUuid / logicalParentUuid`
   - 接回当前历史骨架
2. `message.id`
   - 让 assistant 片段仍被认成同一条消息
3. `tool_use_id / sourceToolAssistantUUID`
   - 让 tool use / tool result 与 findings 回到同一个行动主语

这意味着 Prompt 效力不只是“记得说过什么”，而是：

- 还知道现在这句话属于哪个世界、哪个动作、哪个继续主语

## 4. 第三层：协议转写优于显示历史

很多系统以为 Prompt 负责 instruction，messages 只是上下文容器。

Claude Code 更接近的真相是：

- 模型真正消费的是 `prompt + normalized transcript + paired tool protocol`

这意味着 Prompt 效力不只来自 system prompt 本身，还来自：

1. UI 历史不会直接越权进入模型路径。
2. `tool_use / tool_result` pairing 会先被修正。
3. 非法消息形态不会混入继续链。
4. 用户可见 transcript 与模型可执行 transcript 被明确分层。

所以它真正保护的不是：

- 文案纯度

而是：

- 世界进入模型前的协议纯度

## 5. 第四层：projection consumer 分层优于“所有历史都一样”

真正难的不是“有几份 transcript”，而是不同消费者是否仍在消费同一条 lineage。

Claude Code 至少稳定区分了四类 projection consumer：

1. display consumer
   - 关心人类可读历史
2. model API consumer
   - 关心 `protocol transcript` 与合法消息 ABI
3. SDK / control consumer
   - 关心宿主协议、tool progress 与状态投影
4. handoff / compact / resume consumer
   - 关心 `continuation object` 是否仍可行动

所以它真正保护的不是：

- 让所有观察面看起来都差不多

而是：

- 让不同 consumer 的差异只停留在读法，不升级成真相分叉

## 6. 第五层：可缓存前缀优于更长文案

如果 Prompt 只是长文案，那么每一次变化都会重新把世界整体打散。

Claude Code 更成熟的地方在于，它先问：

- 哪些字节必须稳定到足以成为共享前缀资产

这会改变 Prompt 设计的目标函数：

- 不再追求“尽量写全”
- 而是追求“真正稳定的部分尽量不被重编”

所以它的效力不是：

- 每次都重新讲清世界

而是：

- 尽量不重复讲已经成立的世界

也正因为如此，`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`、section registry 与 stable prefix 不只是性能技巧，而是 Prompt stable-prefix order 的一部分。
更稳地说，stable prefix 真正替 later consumer 省下的，不只是重述世界的 token，而是重新扫描工具约束、输出法与候选动作的冷启动搜索；前缀一旦频繁失稳，先膨胀的通常不是文本长度，而是下一步动作空间。

更进一步，section registry 也不是一份静态目录，而是会在 `/compact` 之后被重新整理、重置与续接的运行时对象。这意味着 Prompt stable-prefix order 不是“永远附着在顶部的一段总纲”，而是有自己生命周期的正式运行时资产。

Anthropic 官方文档又把这件事往外表钉死了一层：

1. nested `CLAUDE.md`、path-scoped rules、skills 与 MCP 定义都属于按需加载对象；
2. startup context、on-demand expansion 与 `/compact` 后 reload 被公开写成不同 load fate；
3. `InstructionsLoaded / PreCompact / PostCompact` 这种 hook 生命周期进一步说明，Boundary 不只是在管 stable bytes，还在管 `eager load / lazy load / compact reload` 的正式装配法。

## 7. 第六层：工具 ABI 与写作法同属 Prompt runtime order

Claude Code 的 Prompt 高密度，不只因为 instruction 多，还因为：

1. 工具命名、专用工具优先级、并行策略与替代禁令被直接写进 Prompt。
2. 长度、格式、文件引用、列表层级与 channels 分工也被直接写进 Prompt。
3. 显示层会继续把 thinking 与 tool 噪音蒸馏掉，保证用户消费的是 Prompt 指定的结果形态。

这意味着 Prompt 效力的一大部分，其实来自两层协同：

- Prompt frontdoor 固定行为与写法要求
- Runtime 明确规定哪些内容能被用户看见

更稳地说，工具 ABI、输出法与用户可见面之所以属于同一条 Prompt runtime order，不只是因为它们都在管格式，而是因为它们共同把下一步动作空间压成少数已授权分支；若 later consumer 仍要从工具全集和结果形态里重新猜该做什么，所谓 Prompt 效力就还只是长上下文幻觉。
所以它强的地方不是：

- 模型临场发挥得更好

而是：

- 模型能发挥的空间本身已经被编排成高行动语义密度的受治理世界

## 8. 第七层：合法遗忘优于无限记忆

如果一个 Prompt 只能靠“尽量别忘”维持效力，它就永远只是短任务技巧。

Claude Code 更高级的地方在于，它同样认真规定：

- 忘掉什么之后，下一轮仍能合法继续

这会把 compact 从“压缩历史”改写成：

- 续写对象重建

真正保住 Prompt 效力的不是：

- 还留了多少上下文

而是：

- 继续工作的判断条件有没有留下来

这也会把 summary 从“更好读”改写成：

- `lawful forgetting boundary` 是否仍然成立

如果继续把这组对象再压成 `lawful forgetting witness ABI`，更稳的最小分层也只该剩四类：`required`、`derivable`、`narrative-only`、`forbidden-as-sole-witness`。这里真正要保住的 `required` 也只是一份最小 `continuation object`：`current work / next-step guard / required assets / rollback boundary / continue qualification / threshold liability`。

更稳地说，`continue qualification` 不是对摘要质量的感受分，而是 later consumer 重新读取这组 `lawful forgetting witness ABI` 时，仍能在不重谈 `authority / boundary / lineage` 的前提下复现同一条 `continue-or-reject verdict` 的判定。compact / resume / handoff 共用的也不是“摘要格式”，而是这组 witness ABI 能否仍被 parse 并复现同一条 continuation verdict；字段可以扩展，但 `required` 与 `forbidden-as-sole-witness` 这两层不能在没有明确 rebinding 的情况下被静默改写。

第一条 `first retreat layer` 也因此不是补写更顺的摘要，而是先回 `protocol transcript / stable prefix / continuation object`，补齐哪一项 `required witness` 已经断掉；若退不回这层，lawful forgetting 就已经退回 narrative compression。更快一点的拒收线也只剩一句：只要 `narrative-only` 或 `forbidden-as-sole-witness` 被迫独自承担 continuation witness，就直接判 `continue qualification` 失效，不得再把摘要包装成 lawful forgetting。

更硬一点说，`next-step guard` 不只是在说“接下来做什么”，还必须保住“哪些看似可做的分支已经被当前 `authority / boundary` 排除”；若 compact 之后只剩正向建议而没有负向剪枝，later consumer 仍会重新打开全量动作搜索，`continuation object` 也就退回成叙事提示。若连“这些分支为什么仍该被排除”都要重新生成，later consumer 消耗掉的就不只是 token，而是一轮没有新增决策增益的 branch re-adjudication；这时失稳的不是摘要长度，而是 `continue qualification` 已退回冷启动。换句话说，如果 compact 之后只剩“更短的故事”，Prompt 就只剩叙事压缩，不再是合法继续。

如果把 compact / resume / handoff 再压成 builder-facing 机制，也只剩三件事：哪一份 `required witness` 仍被显式携带；哪个 surface 先给出第一声 `no`；失败后第一退回层究竟回哪一环补 witness。说不清这三件事，later consumer 接到的就不是 `same-world request object`，而只是一个“看起来还像同一任务”的 narrative package。

## 9. 第八层：继续资格暴露在哪些 surface

很多系统把 continue 视为运行时补丁。Claude Code 更接近的设计是：stop hooks、cache break explainability、budget continuation 与 prompt-too-long recovery 共同暴露 `continue qualification` 是否仍成立。builder 真正要看的也不是“还能不能继续”的情绪判断，而是第一条 `no` 先落在哪个 surface。

更具体地说，这四个 surface 各自只回答一件事：

1. `stop hooks` 暴露当前动作是否还能留在同一条 `authority / boundary` 里继续执行。
2. `cache break explainability` 暴露当前 break 是否仍能被 `stable prefix / protocol transcript` 正式解释。
3. `budget continuation` 暴露这次延长是否仍值得挂在同一份 `continuation object` 上继续持有。
4. `prompt-too-long recovery / re-entry` 暴露当前 carrier 是否已经装不下 `required witness`，必须先 externalize 再 rebind。

因此继续资格的 builder-facing 问法也只剩三层：`continue qualification` 还挂在哪个 surface；第一条 `no` 先落在 `current action / stable prefix / budgeted continuation / re-entry boundary` 的哪一格；第一退回层该回 `message lineage`、`protocol transcript` 还是 `continuation object` 补 witness。多 Agent 场景也一样：真正强的地方不是“会给 worker 写指令”，而是 coordinator 先理解 findings，再把 worker boundary 与 continuation qualification 一起写成自包含 prompt，而不是把理解外包给 worker。

## 10. 第九层：失稳必须先能定位

如果一个链路只能成功、不能定位 first no，它仍然只是幸运。builder-facing diagnostics 至少要同时落盘四件事：缺了哪一份 `required witness`、哪个 surface 先抛出 `local veto cue`、哪一批已排除分支重新回流、第一退回层该回哪一环。

`prompt cache break`、`stable bytes` 与 `continue qualification` 这些词真正值钱的地方，不在名词本身，而在它们能把这四件事固定成 later consumer 可复查的对象。若失稳时只会说“上下文变少了”或“摘要质量不够”，却说不清 veto 落点、witness 缺口与 reopened branches，那么 runtime 仍是在事后讲故事，不是在工程化继续资格。

## 11. 苏格拉底式追问

1. 哪个 `required witness` 缺了。
2. 是不是 current carrier 被迫单独代签继续资格。
3. 哪一批已排除分支重新回流候选集。
4. 第一条 `local veto cue` 究竟落在 `section registry`、`protocol transcript` 还是 `continue qualification`。

## 12. 对 Agent 设计者的三条直接约束

如果想学 Claude Code，最该抄走的不是某一段 system prompt 文案，而是三条工程约束：

1. 把 witness ABI 写成 versioned object，而不是靠 prose 续命。
2. 把 continue qualification 外化成 later consumer 可直接复查的正式对象。
3. 让 prompt 失稳原因可以被正式解释，而不是继续写回不可解释成功。
