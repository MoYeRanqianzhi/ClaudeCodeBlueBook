# 生态成熟度必须与协议支持分开叙述

这一章回答四个问题：

1. 为什么“源码里有支持”不等于“产品已经成熟”。
2. Claude Code 的生态面为什么特别需要把协议支持和成熟度分开。
3. 这条原则如何帮助我们避免误写蓝皮书。
4. 对产品与平台设计有什么一般化启发。

## 1. 先说结论

Claude Code 的生态面最容易被误判，因为它同时存在：

- bundled skill
- builtin plugin
- marketplace plugin
- MCPB / DXT
- LSP
- channels

这些对象虽然都和“扩展”有关，但完全不处在同一成熟度、同一信任层、同一治理层。

因此，蓝皮书必须坚持一个原则：

- 协议支持与产品成熟度必须分开叙述

代表性证据：

- `claude-code-source-code/src/plugins/builtinPlugins.ts:1-18`
- `claude-code-source-code/src/utils/plugins/marketplaceManager.ts:154-191`
- `claude-code-source-code/src/utils/dxt/helpers.ts:5-33`
- `claude-code-source-code/src/tools/LSPTool/LSPTool.ts:127-160`
- `claude-code-source-code/src/services/mcp/channelAllowlist.ts:1-16`
- `claude-code-source-code/src/services/mcp/channelNotification.ts:191-280`

## 2. 为什么“有 schema”不等于“有产品”

一个能力在源码中出现，通常只说明下面几件事之一：

1. 有类型或 schema
2. 有解析逻辑
3. 有某条 build 路径
4. 有某条运行时路径
5. 有某些 gate 打开时的产品行为

这距离“普通用户稳定可用”之间，还隔着很多层：

- auth
- org policy
- allowlist
- entitlement
- rollout
- UI/UX 完整性

所以“协议支持”只能说明：

- 运行时为这种对象留了位置

不能直接说明：

- 它已经成为成熟公共承诺

## 3. Claude Code 为什么更需要这条原则

因为 Claude Code 的生态并不是单一插件系统，而是叠了多层：

### 3.1 builtin plugin

这是：

- 随 CLI 发货
- 可在 UI 中启停
- 带用户偏好治理

### 3.2 marketplace plugin

这是：

- 来源声明
- 本地缓存
- 安装位置
- fallback source

### 3.3 MCPB / DXT

这是：

- 打包格式
- manifest 校验
- 解包与安全检查

### 3.4 LSP

这是：

- 本地基础设施
- 有条件地暴露给模型

### 3.5 channels

这是：

- capability
- auth
- org policy
- session opt-in
- allowlist

叠加后的结果是：

- 你不能只看到一个“支持入口”，就把它写成同一级生态能力

## 4. 苏格拉底式追问：为什么蓝皮书这么容易把生态写错

因为人脑天然更容易做两种偷懒：

1. 看见代码入口，就把它当成产品入口。
2. 看见一组相关名词，就把它们放进同一层分类。

继续追问就会发现，这两种偷懒都站不住：

1. 有入口不代表默认启用。
2. 有格式不代表有分发体系。
3. 有工具不代表有治理模型。
4. 有 capability 不代表能通过 auth/policy/allowlist。

所以真正严谨的写法应该始终同时标注：

- 代码支持
- gate 边界
- 产品边界

## 5. 这条原则对平台设计也成立

更一般地说，一个平台能力至少有三种成熟度：

1. 结构成熟：
   - schema、类型、接口位置已经存在
2. 运行时成熟：
   - 有完整执行路径、错误处理、兼容层
3. 产品成熟：
   - 有明确的启用方式、治理方式、用户承诺

Claude Code 的生态面恰好反复提醒我们：

- 结构成熟最快
- 产品成熟最慢

蓝皮书如果不把这三层拆开，就会不断夸大实际可用性。

## 6. 蓝皮书应该如何书写生态边界

更稳的写法是：

1. 先写“代码支持了什么对象和路径”。
2. 再写“哪些路径需要 feature gate / auth / policy / allowlist”。
3. 最后才写“哪些能力已经足以被用户当成正式表面来依赖”。

在 Claude Code 上，这条规则尤其适用于：

- marketplace
- MCPB / DXT
- LSP
- channels

## 7. 一句话总结

Claude Code 的生态之所以值得认真写，不是因为它“对象很多”，而是因为它迫使我们把协议支持、运行时路径和产品成熟度明确区分开来。
