# 治理必须落到字节级确定性：上下文准入优于功能堆叠

这一章回答五个问题：

1. 为什么 Claude Code 的 prompt 魔力更像“上下文准入编译器”，而不是功能堆叠或文案技巧。
2. 为什么治理如果不能落到字节级确定性，就很难真正生效。
3. 为什么安全、prompt 稳定性、省 token 与权限分类会共同收束到同一条上下文准入链。
4. 为什么 compaction 的目标不是简单变短，而是保住意图连续性。
5. 这套设计为什么比“再多一点能力”更难抄。

## 1. 先说结论

Claude Code 的一个深层秘密是：

- 它真正强的不是“有很多能力”

而是：

- 它有一条把上下文准入编译到字节级确定性的链

这条链会持续回答四个问题：

1. 哪些来源有资格进入系统真相。
2. 哪些内容该进静态前缀，哪些必须后移到动态区。
3. 哪些 tool schema / beta header / attachment 会破坏稳定字节，必须被锁定或延后。
4. 在必须压缩时，哪些语义绝不能丢。

所以 Claude Code 的 prompt 魔力，更准确地说并不是：

- 文案强

而是：

- 上下文准入被治理
- 运行时字节足够稳定
- 压缩策略仍保留意图连续性

## 2. 为什么治理首先是来源优先级，而不是后台设置页

`settings.ts` 的 merge 顺序已经说明了：

- 治理不是“配置系统的一部分”
- 治理是在决定哪些来源有资格先写进运行时真相

`strictPluginOnlyCustomization` 的 schema 描述更进一步说明：

- Claude Code 不只管工具执行
- 还管 skills、hooks、commands、agents 这类 customization surfaces 的输入来源

`permissions.ts` 里 `allowManagedPermissionRulesOnly` 打开时，会主动清空：

- `userSettings`
- `projectSettings`
- `localSettings`
- `cliArg`
- `session`

这些来源上的 permission rules。

这很关键。

因为它说明治理的真正形式不是：

- 某个后台开关被打开

而是：

- 一条准入编译链把低优先级来源直接排除出有效上下文

这就是“治理必须落到字节级确定性”的第一层含义：

- 不是抽象上谁更重要
- 而是最终请求里谁根本进不去

## 3. Prompt 不是字符串，而是编译产物

`buildEffectiveSystemPrompt()` 清楚地展示了：

- override
- coordinator
- agent
- custom
- default
- append

这些来源如何按确定优先级组成 effective system prompt。

随后，`prompts.ts` 又做了第二层治理：

- 会打碎全局缓存前缀的 session-variant guidance，被明确后移到 dynamic boundary 之后
- dynamic sections 统一通过 `systemPromptSection(...)` registry 装配

再往后，`splitSysPromptPrefix()` 与 `buildSystemPromptBlocks()` 会继续把 system prompt 编成：

- attribution
- prefix
- static content
- dynamic content
- 对应 cache scope

所以 Claude Code 真正的 prompt 主路径不是：

- 拼一堆字符串

而是：

- 来源排序
- 区块分层
- cache scope 分配
- block 级定稿

一旦理解成“编译产物”，很多现象就都变清楚了：

- 为什么动态 guidance 要后移
- 为什么 block 数量会被严格控制
- 为什么 cache_control 属于 prompt 正文的一部分而不是外围元数据

## 4. Prompt 魔力为什么依赖字节级稳定，而不只是语义正确

`toolSchemaCache.ts` 几乎把这件事写在注释里了：

- tool schema 处在请求前缀的高位
- 任意字节变化都会击穿后面的大块 cache

所以 Claude Code 不是只在想：

- 当前工具正确不正确

而是在想：

- 当前工具 schema 的字节是否稳定

`claude.ts` 对 beta header 的 sticky-on latch 也是同一种思路：

- 一旦某个动态 header 首次发出，后续会话内尽量继续保持
- 目的不是逻辑方便，而是避免 mid-session 改写 server-side cache key

这说明 Claude Code 真正把“稳定性”视为：

- 会话级资产

而不是：

- 某轮恰好命中缓存的好运气

也正因此，prompt 魔力更接近：

- 可持续复用的字节级确定性

不是：

- 一次性的高质量措辞

## 5. 为什么安全分类也必须共享同一准入链

`yoloClassifier.ts` 有个很深的信号：

- `CLAUDE.md` 会被显式包装成 classifier 输入
- 还会带 cache_control

这说明 Claude Code 不接受这种分裂：

- 主 prompt 认为 `CLAUDE.md` 是用户意图
- 安全分类器却把它当外部噪音

相反，它要求：

- 行为判断使用的上下文
- 主线程运行时使用的上下文

必须尽量共享同一准入真相。

这再次证明，安全不是附在 prompt 外面的第二系统。

它本身也是上下文准入编译器的消费者。

## 6. 为什么 compaction 的本质不是“变短”，而是“保住意图连续性”

Claude Code 的 compaction 设计也在反复证明这一点。

`compact/prompt.ts` 的 base prompt 并没有追求最短摘要，而是要求保留：

- Primary Request and Intent
- Files and Code Sections
- Pending Tasks
- Current Work
- Optional Next Step

这说明 compaction 不是“删除历史”，而是：

- 重新编译可继续工作的上下文

`compact.ts` 里又能看到两条很关键的纪律：

1. 图片与文档不会直接蒸发，而是变成 marker。
2. 下一轮本来就会重注入的 attachment 会先被剔除，避免重复为同一信息付费。

`autoCompact.ts` 则进一步表明：

- 当 context-collapse 正在接管头寸管理时，autocompact 要主动让路

这意味着 Claude Code 的原则并不是：

- 只要能省 token 就上

而是：

- 省 token 不能破坏更高优先级的上下文连续性机制

## 7. 苏格拉底式追问

### 7.1 如果组织策略只能停留在设置页里，而不能改变最终请求字节，治理真的生效了吗

很难说生效了。

### 7.2 如果 prompt 会因为 session-variant guidance、tool schema、beta header 的随意抖动而持续改写，你看到的是模型不稳定，还是你自己把前缀改碎了

更接近后者。

### 7.3 如果 compaction 只负责缩短，而不负责保住用户最新意图、当前工作与下一步，压缩后还能继续做事吗

未必。
可能只是更短，但不再连续。

### 7.4 如果安全分类器和主线程读取的是两套不同的“用户意图”，权限判断还能稳定吗

不能。
它会开始分叉。

### 7.5 如果一个系统没有上下文准入编译器，只是不断往 prompt 里加更多功能块，最后得到的是更强系统，还是更碎的系统

更常见的是后者。

## 8. 对 Agent 设计者的启发

如果想抄 Claude Code，最难抄走的其实不是某一段 prompt，而是这四条纪律：

1. 先治理来源优先级，再治理内容本身。
2. 让 prompt 成为 block-level、cache-aware 的编译产物，而不是字符串拼接结果。
3. 让安全分类、权限系统与主线程共享同一上下文准入真相。
4. 让压缩保住意图连续性，而不是只追求 token 下降。

## 9. 一句话总结

Claude Code 的 prompt 魔力，真正强在它把治理、稳定性、安全与压缩统一写成了一条“上下文准入编译器”主路径，而这条路径最终落到的不是抽象理念，而是请求字节的确定性。
