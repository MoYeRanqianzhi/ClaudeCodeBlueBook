# 真正先进的源码，会先替未来维护者保留反对当前实现的能力

源码先进性最容易被误写成：

- 目录更整齐
- 抽象更优雅

但 Claude Code 更接近的真相是：

- 真正先进的源码，会先替未来维护者保留反对当前实现的能力

这里谈的也不是 continuity 的第四条结构主题；`recovery asset / anti-zombie / freshness gate` 只是源码质量线在时间轴上保护局部可反对性的方式。

## 1. 第一性原理

如果一个系统已经有：

- 分层
- 模块
- 接口

却还没有先保住：

- 单一权威入口
- 单一共享真相来源
- await 之后的 fresh-safe 写回
- 未来维护者可见的危险改动面

那么它依旧只是：

- 结构化的当前实现

而不是：

- 可演化的长期内核

## 2. 为什么这会改变“好结构”的定义

真正成熟的结构不会只帮助今天写代码的人，它还会提前帮助后来的人：

1. 更快找到权威入口
2. 更快定位共享真相
3. 更快识别危险改动面
4. 更快拒绝陈旧写回与第二套真相

所以它保护的不是：

- 当前实现更省事

而是：

- 系统未来仍然可以被正确反驳、正确修复、正确继续

更硬一点说，这里真正值钱的不是“later maintainer 最终能看懂”，而是：

- later maintainer 是否拥有局部可反对性

也就是不追完整仓实现、不等作者口头补充，仍能只凭当前位置附近的命名、注释、state machine、truth surface 与 danger surface，直接指出：

1. 哪条写回在越权。
2. 哪个恢复资产在篡位。
3. 哪层 truth plane 在代写现在。
4. 第一退回层应该落到哪里。

## 3. 为什么 anti-zombie 也属于结构先进性

成熟结构不只关心静态依赖，还关心动态命运：

- 旧 finally 不应清理新状态
- 旧快照不应覆盖新对象
- 恢复资产不应长成第二权威面

也就是说，真正先进的结构会把“不要把过去写回现在”提前编码进日常写法。

## 3.1 局部可反对性优于全仓追读

如果 later maintainer 还必须：

- 追完整仓实现
- 找作者解释旧事故
- 靠经验猜哪层更接近 authority

那么所谓“未来维护者是正式消费者”就还只是态度，不是制度。

真正成熟的结构更像在给 later maintainer 预留一组局部拒收面：

1. `authority surface`
   - 告诉他谁在定义现在。
2. `truth planes`
   - 告诉他谁只在讲事件、谁在讲 current truth、谁只在讲 display。
3. `danger surface`
   - 告诉他哪里最容易把过去重新写成现在。
4. `retreat layer`
   - 告诉他这次先退 `authority / contract / seam / shell / recovery / eviction` 哪一层。

如果这些拒收面不能在局部被点名，目录再整齐也还只是 author-facing 结构，而不是 later-maintainer-facing 结构。

## 4. 苏格拉底式自反

在你准备宣布“这个代码库已经很先进”前，先问自己：

1. 我保护的是目录整洁，还是权威真相。
2. 我是否真的只有一个共享来源，而不是多处半真相。
3. await 之后的写回是否仍然会僵尸化新状态。
4. 后来维护者能否直接从代码里看见反对当前实现的理由。
5. 后来维护者是否能在不追全仓的情况下，直接指出第一条越权写回与第一退回层。
6. 我是不是把恢复成功率误写成源码先进性，而没有先判 `recovery asset` 是否已经越权 current truth。
