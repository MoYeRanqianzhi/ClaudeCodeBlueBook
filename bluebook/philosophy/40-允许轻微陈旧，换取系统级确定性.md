# 允许轻微陈旧，换取系统级确定性

这一章回答四个问题：

1. 为什么 Claude Code 会故意接受一些“看起来不够实时”的信息。
2. 为什么这种轻微陈旧不是偷懒，而是换取系统级确定性的工程选择。
3. 为什么 prompt 魔力、缓存命中、side-loop 复用与权限/治理稳定，都依赖这条取舍。
4. 这对 agent runtime 设计者有什么启发。

## 1. 先说结论

Claude Code 在 prompt/runtime 设计里有一个非常克制、但极其重要的原则：

- 允许轻微陈旧，换取系统级确定性

这句话的意思不是：

- 信息越旧越好

而是：

- 不是所有实时变化，都值得立刻打碎整段前缀

在 Claude Code 看来，很多“实时更新”如果直接进入主前缀，代价会远大于收益：

1. cache 立即击穿
2. side-loop 共享前缀失效
3. tool schema / header / section 字节漂移
4. 宿主与用户更难解释当前系统为什么突然变了

所以它宁可对一部分信息采用：

- 会话级固定
- section 级 memoize
- sticky latch
- tail-side delta

也不愿把“实时”错用成“持续抖动”。

## 2. 为什么“轻微陈旧”不是错误，而是设计

`getSessionStartDate()` 是最直白的例子。

它明确选择：

- 会话内日期固定

原因不是作者没想到午夜会变，而是源码直接承认：

- 午夜自动刷新会打碎缓存前缀

`getSystemContext()` 与 `getUserContext()` 也都被缓存到：

- conversation duration

这说明 Claude Code 在问的不是：

- 有没有可能更实时

而是：

- 这份实时性值不值得付出全前缀重算的代价

一旦从这个角度看，“轻微陈旧”反而变成一种更成熟的设计：

- 它是在主动控制系统变化率

## 3. 这个原则如何贯穿整个 prompt runtime

### 3.1 section 默认缓存，volatile section 必须显式自认危险

`systemPromptSections.ts` 的命名非常诚实：

- 默认 section 会缓存
- 不缓存的 section 必须叫 `DANGEROUS_uncachedSystemPromptSection`

这说明 Claude Code 的默认立场是：

- 稳定是默认
- 破坏稳定要被显式承认

### 3.2 tool schema 宁可会话内固定，也不要 mid-session 漂移

`toolSchemaCache.ts` 里直说了：

- schema 位于请求高位
- 中途变化会击穿整段下游 cache

所以哪怕 GrowthBook 翻转、MCP reconnect、动态 prompt 变化，schema 仍倾向于先保持会话内稳定。

### 3.3 beta headers 采用 sticky latch

`claude.ts` 对 AFK/fast/cache-editing 等 header 采用：

- once sent, keep sending

这就是典型的“允许轻微陈旧，换取确定性”。

它要保护的不是某个布尔值的实时性，而是：

- 当前会话的请求拓扑尽量保持一致

## 4. 为什么这条原则反而会让系统显得更聪明

表面上看，允许轻微陈旧像是在妥协。

但实际效果恰好相反。

因为 agent 系统最怕的不是：

- 某一处信息晚几分钟更新

而是：

- 整条运行时前缀无规律抖动

一旦前缀抖动，模型体验到的不是“更实时”，而是：

- 每轮条件都在变
- side loops 无法复用主线程资产
- cache break 更难解释
- 调优手段越来越像碰运气

从这个角度看，轻微陈旧其实在换一种更高阶的智能：

- 跨轮一致性

而跨轮一致性，恰恰是 Claude Code 看起来“更稳”“更像懂现场”的关键来源之一。

## 5. Claude Code 如何补偿这份“轻微陈旧”

它并不是把变化完全压掉，而是把变化放到更安全的位置。

最典型的补偿方式就是：

- delta attachments

例如：

- deferred tools delta
- agent listing delta
- MCP instructions delta

这些信息没有消失，只是被改成：

- 尾部回写
- 局部变化
- 不污染主前缀

所以 Claude Code 的真实策略不是：

- 不更新

而是：

- 不要用最贵的方式更新

## 6. 苏格拉底式追问

### 6.1 如果日期在午夜自动刷新，但整段系统前缀和 side-loop cache 全部击穿，这真的是“更对”吗

未必。

### 6.2 如果 tool schema 因 feature gate 或连接状态中途变化而频繁重排，模型得到的是更强能力，还是更碎的条件

更常见的是后者。

### 6.3 如果 volatile section 可以随时无标记地进入前缀，系统还能解释自己为什么突然失稳吗

很难。

### 6.4 如果变化可以通过 delta attachments 局部回写，为什么还要让它污染整个前缀

没有必要。

## 7. 对 Agent 设计者的启发

如果想复制 Claude Code 的稳定感，最该学的不是“再加一点实时信息”，而是：

1. 先决定哪些信息必须稳定到会话级。
2. 让破坏稳定的 section 被显式标记出来。
3. 把高波动变化迁到尾部或旁路，而不是主前缀。
4. 把“轻微陈旧”看成可接受的工程预算，而不是设计失败。

## 8. 一句话总结

Claude Code 之所以显得稳定，不是因为它追求绝对实时，而是因为它敢于接受轻微陈旧，用来换取更高层次的系统级确定性。
