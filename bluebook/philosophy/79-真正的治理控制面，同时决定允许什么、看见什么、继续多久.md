# 真正的治理控制面，同时决定允许什么、看见什么、继续多久

治理最容易被误写成：

- 安全是一套规则，省 token 是另一套优化

但 Claude Code 更接近的真相是：

- 真正的治理控制面，同时决定允许什么、看见什么、继续多久

## 1. 第一性原理

如果系统只会：

- 审批动作
- 统计 token

却还不会同时决定：

- 哪些能力先隐藏
- 哪些大对象先外置
- 哪些 continuation 先停止

那么它依旧只有：

- 局部门禁

而不是：

- 成立的治理控制面

## 2. 为什么安全与省 token 会回到同一问题

因为两者都在反对同一件事：

- 免费扩张

动作免费扩张，会伤安全。

能力免费扩张，会伤判断质量。

上下文免费扩张，会伤成本与稳定性。

时间免费扩张，会伤继续条件。

所以更接近 Claude Code 真相的说法不是：

- 安全和省 token 的平衡

而是：

- 对扩张的统一定价

## 3. 为什么 host / settings / MCP / Context Usage 必须一起看

因为这些平面分别负责：

1. 谁能改边界
2. 当前边界是什么
3. 外部能力是否可接入
4. 当前成本与继续条件如何被解释

把它们拆开看，就会重新退回：

- 宿主猜状态
- 模型猜能力
- 团队猜为什么变贵

## 4. 苏格拉底式自反

在你准备宣布“我们的治理系统已经成熟”前，先问自己：

1. 我治理的是边界，还是只治理了交互形式。
2. 模型看见的是最小必要世界，还是几乎全量世界。
3. 当前成本是被解释出来的，还是只是被统计出来的。
4. continuation 是正式时间预算，还是默认免费继续。
