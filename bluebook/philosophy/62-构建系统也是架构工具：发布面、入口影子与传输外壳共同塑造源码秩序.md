# 构建系统也是架构工具：发布面、入口影子与传输外壳共同塑造源码秩序

这一章回答五个问题：

1. 为什么源码先进性不该只从目录分层和大文件治理来判断。
2. 为什么 external stub、portable shadow entry、transport shell、薄 registry 与恢复治理应被放在同一条线上。
3. 为什么构建系统本身也在塑造模块图、发布面和入口安全。
4. 为什么恢复路径会反向塑造日常代码的写法。
5. 这对 agent runtime 设计者意味着什么。

这些锚点共同说明：

- Claude Code 的高级工程不只体现在运行时机制，还体现在它如何用构建、入口、传输和恢复制度主动塑造源码秩序。

## 1. 先说结论

更成熟的源码先进性，不是：

- 只有运行时设计先进

而是：

1. 构建系统在主动塑造模块图和发布面。
2. 不同入口拿到的是各自可承受的 implementation shadow。
3. transport 分叉被收进统一 shell，而不是蔓延成全仓分叉。
4. 高扇入 registry 只做注册和转运，不顺手长第二业务层。
5. 恢复路径先被设计清楚，日常写法才敢简化。

从第一性原理看，这条线说明 Claude Code 的源码先进性不只在：

- 当前结构是否优雅

更在：

- 仓库如何主动限制自己未来失控的方式

## 2. 第一层：构建系统在主动塑造模块图

很多项目把构建视作：

- 代码写完后的打包步骤

Claude Code 更成熟的地方在于：

- 构建系统本身就是架构工具

例如：

1. `useMoreRight.tsx` 提供 external stub；
2. `AppState.tsx` 用 ant-only DCE passthrough 保持外部图谱干净；
3. feature / external build 的设计不只是控制功能是否出现，也在控制哪些依赖边根本不该进入公开图谱。

这说明 Claude Code 在做的不是“少打一个包”，而是：

- 主动给不同发布面塑形

从源码先进性的角度看，这是很高阶的自律，因为它承认：

- 模块图不是写完自然形成的，而是需要被发布制度主动约束。

## 3. 第二层：portable shadow entry 优于入口全量直通

`listSessionsImpl.ts` 和 `emptyUsage.ts` 这类文件最有价值的地方，不是复用，而是：

- 让不同入口拿到各自可承受的实现影子

也就是说，Claude Code 并不要求：

- 所有入口都共享同一份完整实现

它更关心：

- 这个入口能否安全导入；
- 会不会因为拿错实现而触发 CLI 初始化、重依赖或副作用链。

这代表一种更成熟的入口设计：

- portable shadow entry

它承认“同一能力”在不同入口上可以拥有：

1. 同一份语义；
2. 不同厚度的实现影子。

这比强求“全仓所有入口都 import 同一个真身”更诚实，因为后者往往只会把重依赖和初始化副作用扩散出去。

## 4. 第三层：transport 分叉要被收进 shell，不要蔓延成平行系统

`replBridgeTransport.ts` 明确把 v1/v2 选择收口到 construction site；transport 变化被限制在统一抽象表面之内。

这说明 Claude Code 处理协议分叉的方式不是：

- 让不同 transport 各写一套逻辑

而是：

- 让 transport 成为 shell，语义层继续稳定

从第一性原理看，这和 prompt Constitution 很相似：

- 载体可以变；
- 不变语义必须稳定。

这也是为什么 Claude Code 能在桥接、远程、SDK、REPL 之间保住更高层的控制面真相，而不是每出现一种 transport 就复制一套 runtime。

## 5. 第四层：高扇入 registry 必须薄

`cleanupRegistry.ts` 和 swarm backend `registry.ts` 共同说明一个很强的判断：

- registry 最好只做注册和转运

因为一旦 registry 节点开始顺手长业务，就会立刻变成：

- 高扇入
- 高耦合
- 高误改风险

Claude Code 刻意让这些节点保持：

1. 只保存 class / function registration；
2. 只承担最小构造逻辑；
3. 注释中直接写出切环动机。

这说明真正成熟的“中心节点”不是：

- 更强大的公共层

而是：

- 更薄的 chokepoint

## 6. 第五层：恢复路径先被设计清楚，日常代码才敢简化

`bridgePointer.ts`、`sessionIngress.ts`、`task/framework.ts` 体现的是另一层高级关系：

- 恢复不是附属功能，而是会反向塑造平时写法

例如：

1. pointer 设计成按 repo 作用域、按 mtime 刷新、worktree-aware fanout；
2. ingress 通过 `Last-Uuid` 和 409 恢复 stale client state；
3. task framework 明确要求 against fresh state merge，防止 stale snapshot clobber terminal transition。

这说明 Claude Code 真正在治理的不是“某个 race”，而是：

- 跨 await、跨进程、跨 transport 的对象命运

从第一性原理看，成熟的恢复系统带来的最大好处不是“出了错还能接”，而是：

- 平时写状态更新时，开发者已经知道哪些写法未来一定会把恢复搞坏

## 7. 苏格拉底式追问

### 7.1 为什么构建系统也算架构工具

因为构建系统决定了：

- 哪些依赖边能进入发布面；
- 哪些入口能安全导入；
- 哪些功能必须以影子实现或外部 stub 形式存在。

### 7.2 为什么入口影子比“全入口共享同一实现”更高级

因为更高级的问题不是“复用更多”，而是：

- 在不污染模块图与初始化副作用的前提下，保持语义一致。

### 7.3 为什么恢复路径能反向塑造日常代码

因为一旦系统认真面对：

- stale state
- epoch mismatch
- crash recovery
- cross-await clobber

很多看似方便的写法就会立刻暴露为：

- 未来一定出事的写法

## 8. 对 Agent 设计者的启发

如果想学 Claude Code，最该抄的不是：

- 再画一张更漂亮的目录树

而是：

1. 把构建系统也当成塑形模块图和发布面的架构工具。
2. 给不同入口设计安全可导入的 implementation shadow。
3. 把 transport 分叉收进统一 shell，而不是复制运行时。
4. 让高扇入 registry 只做注册和转运。
5. 先把恢复路径设计清楚，再反过来约束日常状态写法。

## 9. 一句话总结

Claude Code 的源码先进性不仅在运行时机制，还在于它让构建系统、入口影子、传输外壳、薄 registry 与恢复制度一起塑造源码秩序，从而让仓库在增长中更不容易失控。
