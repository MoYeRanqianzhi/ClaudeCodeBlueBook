# 第一性原理与苏格拉底反思

这一章不直接回答“Claude Code 有什么功能”，而是把判断收回第一性问题，并持续做苏格拉底式自校。

先明确定位：本章是“总控自校页”，不是第二入口页。它不新增同题首答，只把 canonical source 已固定的判断压成可复核、可反驳、可迁移的问题梯子。

进入正文前只固定三条判断：

1. Prompt 线
   - 消息血缘、协议转录与继续资格，是否先被编译成同一个可复核世界
2. 治理线
   - 一切扩张是否仍按同一价格秩序合法进入当前世界
3. 源码质量线
   - 当前真相是否被正式保护，不让过去与失控复杂度写坏现在

`continuity` 不是第四条母线，也不是第九个问题；它只是三条判断在时间轴上的联合验收。

最小拒错顺序也只保留三条：

1. Prompt 线
   - 我是否又把 Prompt 的 object / witness / qualification 压成了第二条前门，而不是老老实实回 `84 / 51`
2. 治理线
   - 我是否又在自校页重发治理 canonical chain、failure ladder 或 signer / reopen 机制口径，而不是老老实实回 `10 / security / risk`
3. 源码质量线
   - 我是否仍沿 `guides/102 -> philosophy/87` 的 owner 顺序审读 current-truth protection，而没有把 ladder、降格规则或 later-maintainer 信号写成第二条前门

这页的职责边界同样要先钉死：

1. Prompt 顶层定义与 `first-reject path` 在 `philosophy/84`，same-world witness 与继续资格在 `guides/51`；本章只做第一性追问与失稳解释。
2. 根 `README` 与 `navigation/README` 负责进入路径，本章不代行路由法。
3. `06` 只解释“为什么这套分工不失真”，不重复发放入口答案。

如果要把本章继续写稳，苏格拉底自校至少持续追问十二件事：

1. 我们有没有把对象链偷换成结果词。
2. 我们有没有给出同题坏解，而不只是在讲正解。
3. 我们有没有写出第一条反证信号，而不只是在写高阶判断。
4. 我们有没有给出 `shadow -> qualify -> cutover -> rollback` 的渐进改写顺序。
5. 我们有没有让入口层与第一性判断共用同一组主语与最小排查顺序，而不是重复发放同题首答。
6. 我们有没有把 `谁定义现在 / 谁只消费现在 / 谁只能帮助恢复现在` 写成目录与源码的共同桥接句。
7. 我们有没有把治理失稳重新写成完整机制顺序，而不是先问自己是否已经回到治理 owner 页。
8. 我们有没有把证据层级写清：`runtime-core evidence` 与 `operator-governance evidence` 不能混用。
9. 我们有没有先判这次缺的是 `formula / order / evidence ceiling / verdict`，而不是把所有目录调整都叫“优化结构”。
10. 我们有没有把只需 `route trim / label fix / appeal chain` 的重复，误判成必须新增入口页。
11. 当前 worktree 镜像缺席时，我们有没有先暴露 `evidence mode`，而不是让目录话术替 runtime-core 缺口补 certainty。
12. 并行深挖的新结论，第一次究竟该回 `09 / 05 / 15 / 41` 哪一层，而不是同时散落到 `README / navigation / guide`。

如果把这组自校继续压短，最后都应回到四个跨平面函数：`compile / price / externalize / evict`。

文档层对应的默认动作顺序也应固定成：`label fix -> route trim -> speaking rights / appeal chain -> new entry`；只要前三级还没做完，就不应跳到第四级。

目录分工也只用这四个函数来校准：`userbook` 保留 plane/object/entry order，`philosophy` 保留不可约判断，`architecture` 保留对象与 choke point，`api` 承认 host contract，`guides` 负责问题梯子与模板，`playbooks` 负责 verdict 与 rollback，`casebooks` 保留 distortion，`docs` 保留 memory。

目录优化的成功标准不是“更快找到页”，而是“更少页承担第一次解释责任”。只要入口层与深页不能单源复用同一组最小顺序，目录就仍在制造重复首答。

后文的“八个不可约问题”，都应被读成这三条判断在操作面的展开，而不是另一套并列总纲。

这些锚点共同支撑一个判断：Claude Code 的核心不是单点模型调用，而是一个带装配、控制、恢复与远程能力的 runtime。

## 1. 如果从零开始，coding agent 的最小问题是什么

先把一切包装拿掉，coding agent 至少要面对八个不可约问题:

1. 观察: 它如何看见文件、命令结果、外部系统状态。
2. 决策: 它如何决定下一步动作。
3. 行动: 它如何安全地执行动作。
4. 记忆: 它如何在长任务里保留有效状态。
5. 协作: 它如何把大任务拆给多个工作单元。
6. 恢复: 它如何在超时、中断、上下文过长、权限阻塞时继续工作。
7. 治理: 它如何被账号、组织、策略、遥测与信任边界约束。
8. 经济: 它如何压缩 token、输出噪音与认知负担。

如果一个系统没有认真回答这八个问题，它最多是一个能写代码的聊天机器人，而不是一个能长期工作的工程代理。

## 2. Claude Code 对这八个问题的回答

### 2.1 观察

Claude Code 不只依赖聊天文本，而是把观察能力做成统一工具层:

- 文件读写
- shell
- MCP
- tool progress
- task output
- context usage

### 2.2 决策

它的决策核心仍然是 `query()` loop，但真正的决策质量来自外围 harness:

- system prompt 组装
- skill/command/tool 装配
- permission suggestion
- deferred tool search
- context compaction

### 2.3 行动

它不让模型直接“想做什么就做什么”，而是先进入 tool schema，再进入 permission/runtime policy。

### 2.4 记忆

它把 transcript、session、memory files、scratchpad、context usage 都纳入系统真相，而不是把对话历史当唯一记忆。

### 2.5 协作

它通过 AgentTool、task runtime、background、worktree、remote 把单线程代理变成可编排系统。

更硬一点看，协作对象至少包括：

- `local_workflow`
- `workflow_progress`
- `independent transcript`
- `worktree`
- `cleanup semantics`

命令只负责实例化这些对象，生命周期、观测、归档与恢复都应落在对象自身，而不是继续藏在命令分支里。

### 2.6 恢复

它认真处理:

- prompt too long
- compact/reactive compact
- fallback model
- interrupted turn
- late duplicate control response
- resume / rewind / session restore

因此 Claude Code 的差异不在“看起来更有效”，而在它针对这八个不可约问题都给出了工程化答案。

### 2.7 治理

Claude Code 从来不是“纯本地 agent”。

它持续处在治理平面里：

- account / org / subscription
- GrowthBook / feature gate
- remote managed settings
- trust / sandbox / allowlist
- remote control / trusted device

这意味着 Claude Code 的能力边界，不只由模型与工具决定，也由治理层实时塑形。

### 2.8 经济

Claude Code 也不只是“尽量多塞上下文”。

它在主动优化三种成本：

- token 成本
- 状态混乱成本
- 用户认知成本

因此它会把：

- 动态工具目录外移
- 大块工具输出外置
- prompt cache 稳定性
- compact / microcompact / reactive compact
- sticky prompt / transcript mode / message actions

共同纳入同一个 runtime 设计问题。

如果把这条线压成更可排查的对象，更稳的 why 句只剩一句：省 token 本质上是在压制无序扩张，而不是在做文本压缩；具体收费对象与顺序统一回治理 owner 页。

所以安全、治理、省 token 与 prompt 稳定性，并不是四个并列专题，而是同一价格秩序在不同资产上的投影。

### 2.9 隔离不是额外卖点，而是协作与恢复的前提

多 Agent 线继续往下压，会看到一个更硬的贯穿性约束：

- 子代理默认隔离上下文，而不是共享上下文
- worktree 默认隔离代码现场，而不是共享文件系统
- remote agent 默认隔离执行环境，而不是只换 transport
- sidechain transcript 默认隔离历史链，而不是平铺进主日志

所以“协作”并不是多开几个 worker 就结束了。
真正能让协作成立的，是隔离先被做好。

### 2.10 Prompt 的稳定性来自 same-world compiler contract

如果继续追问“为什么 Prompt 能维持同一世界”，更稳的回答不是：

- 它有更聪明的话术

而是：

- 真正值钱的不是 prompt 文案本身，而是 runtime 先保证“现在到底是什么世界”这件事不必在每一层重新协商；如果 `06` 在这里继续枚举 object、witness 或 qualification 名字，就说明这页已经越位复述了 `84 / 51`

所以多 Agent prompt、coordinator prompt、teammate addendum 真正强的地方，不是句子本身，而是它们被嵌进了一个有 task-notification、team context、tool ABI 和恢复链的 runtime。

如果要把这件事压成最短可检验对象，这里最该反复追问的不是 object 名单，而是：我是不是又把 `84 / 51` 本该定义的 Prompt owner 术语搬回了自校页。

### 2.11 真正的设计单位不是功能，而是运行时平面

Claude Code 最深的一层设计并不是：

- 今天又多了一个功能

而是：

- 某个长期问题有没有被收束成稳定的 runtime plane

这能同时解释：

- Prompt 的同一世界判据
- permission engine
- token 经济
- state/recovery
- host integration
- 多 Agent 协作

因为这些都不是彼此孤立的功能，而是同一 runtime 的不同平面。

更进一步，这几条线其实都在回答同一个更高阶的问题：

1. 当前世界怎样进入模型
2. 哪些扩张配进入当前世界
3. 过去怎样不得把自己重新写回当前世界

也就是说，Prompt 的 same-world test、统一定价治理与源码质量判断，并不是三条互不相干的专题。
它们首先共同服务的是：

- 当前世界的准入资格
- 当前世界的持续身份
- 当前世界不被陈旧对象篡位的能力

如果把 turn runtime 再压一层，至少还要分清：

- `turn state`
- `recovery self-loop`
- `stop-hook continuation`
- `token-budget continuation`
- `worker_status / requires_action_details / pending_action`

时间线可以走事件流，但当前真相不能靠消息回放去猜；它必须被显式回写给宿主。

### 2.12 前台交互为什么也属于核心设计

如果只从后端或 prompt 看 Claude Code，会继续低估一件事：

- 前台本身也在承担认知控制功能

sticky prompt、transcript mode、message actions、teammate view 的价值，不是“交互更顺滑”，而是：

- 让用户在长会话、多 Agent、后台任务里仍能保持对现场的控制

更硬一点说，前台真相面至少由这些对象组成：

- `sticky prompt`
- `transcript mode`
- `message actions`
- `search-selection-scroll semantics`
- `UI transcript`

其中 `UI transcript` 只服务人类可见投影，`protocol transcript` 服务模型侧协议合法性；两者相关，但不能互相冒充。

### 2.13 为什么三条母线必须压成三句对称判断

继续压缩 `philosophy/84-87`、`guides/99-102` 与 `playbooks/77-79` 后，会发现最稳的高阶压缩，不再只是“三张控制面图”，而是三句彼此对称的判断：

这三句的规范表述以 `09-三张控制面总图：世界进入模型、扩张定价与防过去写坏现在.md` 为准；本节只回答它们为什么从第一性原理上站得住。

1. Prompt 线
   - 消息血缘、协议转录与继续资格必须对同一个世界保持诚实
2. 治理线
   - 一切扩张必须按同一价格秩序合法进入当前世界
3. 源码质量线
   - 当前真相必须被正式保护，不让过去写坏现在

这三句一起看，Claude Code 最深的一层就不再是：

- Prompt 更强
- 安全更严
- 代码更漂亮

而是：

- 它同时治理世界进入模型、世界怎样被定价，以及 stale writer、恢复篡位与第二真相怎样不得偷走现在

## 3. 苏格拉底式追问一: 如果模型更强，还需要这些外层系统吗

表面答案似乎是“模型更强，外层系统就没那么重要”。

但继续追问:

1. 更强的模型能自动解决权限边界吗。
2. 更强的模型能自动把 shell 长任务后台化吗。
3. 更强的模型能自动维持 transcript 可恢复性吗。
4. 更强的模型能自动区分 public/gated/internal 功能边界吗。

答案都是否定的。

所以更强模型提升的是“单步推理质量”，不是“系统级可靠性”。Claude Code 真正押注的是后者。

## 4. 苏格拉底式追问二: 为什么 Claude Code 强，不代表“工具越多越强”

继续问:

1. 如果只是堆更多工具，会更强吗。
2. 如果工具没有统一权限、统一上下文、统一渲染、统一恢复，会不会只是更乱。

答案也是否定。

所以 Claude Code 强，不在“工具多”，而在:

- 工具被纳入统一协议
- 工具能被权限系统与控制协议管理
- 工具结果能进入 transcript 与恢复链

## 5. 苏格拉底式追问三: 为什么它不像传统 IDE 插件

因为它解决的问题不是“在 IDE 里加一个 AI 面板”，而是“让一个行动体在软件工程现场长期工作”。

从第一性原理看，这要求:

- 它必须是 runtime
- 它必须能持久化
- 它必须能中断与恢复
- 它必须能扩展
- 它必须能协作

所以 Claude Code 更像 agent operating environment，而不是 IDE widget。

## 6. 这类制度分析最常会在哪里失真

如果把这类制度分析本身拿来做苏格拉底式反思，至少有这些高频失真。

### 6.1 为什么接口支持面常被低估

是的。

只会说明“为什么强”，却不能回答“用户到底能通过哪些接口调它”时，接口支持面就会被系统性低估。
更稳的主线必须同时保留能力矩阵与 API 文档这类接口入口。

### 6.2 为什么“存在代码”与“公开可用”常被混写

还不够。

虽然已经拆出公开能力与隐藏能力一章，但后续每个专题都要继续明确:

- build-time gate
- runtime gate
- public build
- internal monorepo

### 6.3 为什么目录组织常退回作者视角，而不是任务视角

部分是。

如果分析目标是“接入 SDK”“理解命令面”“做多 Agent 编排”，需要的就不是泛泛总览，而是直接可查的索引。
所以目录结构需要把 `bluebook/`、`api/`、`architecture/` 的角色分得更清楚。

### 6.4 为什么结论必须保持可反驳

还可以更好。

更好的制度分析不只给观点，还应附:

- 锚点
- 反例边界
- 何处只是推论
- 何处是源码直接事实

### 6.5 为什么“主线章节”和“可检索手册”必须分载体

这一层最容易反复失稳。

如果读者想知道：

- compact 到底怎么做
- slash command 字段到底有哪些
- build gate 和 compat shim 到底如何分层

那么最优路径不应该是继续把 `bluebook/` 写得越来越厚，而应该是：

- 让主线继续保持一跳能读懂
- 把具体职责矩阵继续收回各目录 README 与 `navigation/README`

否则蓝皮书会越来越像一本难检索的大长文，而不是工程手册。

### 6.6 为什么状态面常被低估

如果只把：

- 命令面
- 工具面
- 权限面
- 前台交互面

写清楚，还不够。

Claude Code 还有另一半本体：

- transcript
- session metadata
- layered memory
- session memory
- file checkpoints
- rewind / resume / fork

### 6.7 为什么功能全集与检索路径必须分层

当章节越来越多时，读者首先需要的不是更多结论，而是：

- 我该从哪读起
- 某个问题该去哪查
- 这章属于主线、机制、接口还是哲学

所以更稳的目录秩序必须把 `navigation/` 单独立出来，专门承担阅读地图与检索职责。

### 6.8 为什么设计解释必须直接连到治理边界

如果继续只解释 prompt、runtime、tool orchestration，却不把：

- account / org / policy
- trust / device / remote control
- product maturity / protocol support

一起写进去，蓝皮书仍然会低估 Claude Code 的真实系统边界。

如果不把这组状态面单独立起来，读者会误以为 Claude Code 主要强在“控制与执行”，而不是“可恢复与可持续”。

### 6.9 为什么隔离先于并发

如果只写：

- 为什么 Claude Code 强在工具统一协议
- 为什么它强在状态面与恢复链

但还没有足够明确地区分：

- 并发
- 隔离

如果不把这两件事拆开，读者会误以为 `AgentTool` 的价值主要是“并行更多 agent”；
实际上它更深的价值是让这些 agent 在上下文、权限、文件系统和历史链上彼此分离，避免把主状态搞坏。

### 6.10 为什么扩展面应被写成统一配置语言

如果只列：

- skills
- plugins
- MCP
- agents

但还没有足够明确地指出：

- 它们共享 frontmatter / manifest 语言
- 它们最终会收敛成统一对象模型
- 它们的关键差异更多来自信任边界与运行语义，而不是“各有一套完全独立系统”

如果不把这一点点破，读者会把 Claude Code 的扩展面误解成“功能越来越多的拼盘”，而不是“配置驱动的 runtime 语言”。

### 6.11 为什么 `query.ts` 的 continue 语义其实是 turn runtime

如果继续把 `query.ts` 写成“流式请求 + 工具执行”，就会漏掉它真正更深的一面：

- turn state
- recovery self-loop
- stop hook continuation
- token budget continuation
- abort consistency

也就是说，我们过去低估的不是某个细节，而是整台 turn runtime。

### 6.12 为什么 permission prompt 只是权限系统的投影

更稳的对象顺序是：

1. 先写 permission context 与 mode transition
2. 再写 rule / safety / tool-specific checks
3. 再写 classifier 与 hook
4. 最后才写 dialog、SDK `can_use_tool`、permission prompt tool

否则蓝皮书会把“决策引擎”误写成“弹窗体验”。

### 6.13 为什么多 Agent prompt 的工程含量常被低估

如果后续只摘 coordinator prompt 里的句子，而不写“协作 prompt 背后还有协作 runtime”，那我们仍然是在做 prompt 评论，而不是在分析协作 runtime；若讨论已进入 team context、mailbox、notification 或 worker prompt 约束，统一回 `guides/51`。

### 6.14 为什么 `services` / `utils` 的分层其实在暴露 invariant

如果只说“这个仓库 utils 很多”，那其实什么都没说。
更关键的问题应该是：

- 哪些是产品域
- 哪些是跨层 invariant
- 为什么 schema、cache、worker state、markdown loader 这些东西必须成为共享基础设施

### 6.15 为什么恢复面与输出面必须分层

如果后续继续把：

- transcript
- sidechain transcript
- task output
- file history
- context collapse restore

混成一个“日志/状态系统”，蓝皮书就会继续写不清楚：

- 哪些东西服务语义恢复
- 哪些东西服务过程观察
- 哪些东西服务状态重建

### 6.16 为什么前台交互属于认知控制面

如果后续只把 REPL 当架构图里的一个 UI 壳，就会继续漏掉：

- sticky prompt 的认知锚点作用
- transcript mode 的 pager 语义
- message actions 的工件化意义
- teammate view 的主语切换作用

换句话说，我们会继续低估“前台如何让人类保持对 agent runtime 的控制”。

### 6.17 为什么 SDK 输出首先是事件流

只写 SDK 消息字典，还不够强调：

- `stream_event`
- `task_started`
- `task_progress`
- `tool_progress`
- `session_state_changed`
- `files_persisted`
- `rate_limit_event`

这些都说明宿主接入的不是一个简单回答接口，而是一条完整 runtime event stream。

如果不把这层结构讲透，读者会误以为 Claude Code SDK 的设计目标是“把模型回复嵌进外部 UI”，而不是“让外部宿主接入 Claude Code 的执行轨迹与状态脉搏”。

### 6.18 为什么 MCP 是连接平面

如果不把：

- config scope
- transport 类型
- needs-auth / pending / disabled / failed 状态
- reconnect / toggle / mcp_set_servers

写成统一状态机，读者会继续低估 MCP 在 Claude Code 中的地位。

它并不是一个“给模型多几个工具”的附属模块，而是 runtime 的正式连接平面。

### 6.19 为什么 Claude Code 应被写成 host-integrated runtime

如果我们只盯着：

- REPL
- slash command
- assistant 输出
- terminal 交互

就会低估：

- `control_request / control_response`
- `control_cancel_request`
- `interrupt`
- `get_context_usage`
- `get_settings`
- `mcp_set_servers`
- `update_environment_variables`

这些宿主控制语义。

更稳的写法应直接承认：Claude Code 不是“终端里一个更强的聊天程序”，而是可以被 IDE、bridge、remote session 与 direct-connect 接入的 host-integrated runtime。

### 6.20 为什么三张控制面图必须反过来约束主线

如果 `09`、`philosophy/84-87`、`guides/99-102` 与 `playbooks/77-79` 都已经成立，但 `03/04/05/08` 与 `navigation/` 入口仍然不受它们约束，那么“三张控制面图”就还只是高级总结，不是主线宪法。

更严格的要求应当是：

1. 任何主线结论都要能回到“世界如何进入模型 / 扩张如何被定价 / 当前如何不被过去写坏”三问。
2. 任何目录优化都要服务这三问的进入顺序，而不是继续制造新的并列入口。
3. 任何深专题如果不能反过来压缩成主线判断，就说明它还没有真正收束。
4. 任何主线章节如果继续各自重写三条母线，也说明 `09` 还没有成为真正的宪法层。

### 6.21 为什么论证正文和推进记录必须持续分界

正文最容易重新混入的，不是明显的 TODO，而是：

- “这一轮补了什么”
- “下一步该补什么”
- “当前最值得继续补强什么”

这些句子一旦重新混进 `bluebook/`，读者读到的就不再是对象、判断与边界，而是临时推进语气。

因此更稳的载体纪律至少包括：

1. 正文只保留主线判断、第一性原理、稳定阅读顺序与苏格拉底自检。
2. 路由页只回答“从哪里进入”，不承载推进叙事。
3. 推进记录、候选方向与外置记忆必须放入独立附录载体，而不是继续占据论证正文。

### 6.22 为什么“协议全集”常被误写成“每个宿主都已完整支持”

也很容易。

只要我们看见：

- `controlSchemas.ts`
- 一长串 control subtype
- `StructuredIO` 的完整请求能力

就很容易顺手写成“Claude Code 的每个宿主都支持这些控制能力”。

但更稳的写法必须拆开三层：

1. 协议全集
2. 控制平面主路径
3. 宿主适配器子集

否则就会把 bridge、direct connect 与 `RemoteSessionManager` 这些不同厚度的实现边界写平。

### 6.23 为什么“安全、治理、省 token 与 prompt 稳定性”常被写成四条互不相干的线

也仍然会。

如果把：

- 安全写成权限拦截
- 治理写成企业设置
- 省 token 写成压缩技巧
- prompt 稳定性写成 cache 性能

就会低估 Claude Code 更深的一层统一性：

- 它真正持续压制的是模型可达世界的无序扩张

更成熟的写法不该在这里重发四个空间、pricing chain、execution contract 或 reject path；这页只保留 why：安全、治理、省 token 与 prompt 稳定性不是四条线，而是同一秩序在不同资产上的投影。若这里已经需要靠 signer surface、弱读回面或 reopen tail-chain 才能站住，统一回治理 owner 页。

### 6.24 为什么安全常被写成“执行前多几道检查”，而不是“输入边界控制平面”

也是。

如果继续只盯：

- tool permission
- sandbox warning
- 安全对话框

就会漏掉真正更深的一层：

- 哪些输入源有资格扩张能力边界
- 哪些扩权输入只能来自 managed authority
- 哪些 contract 会继续被编译成 runtime hard boundary

也就是说，更成熟的写法不该是“工具调用前多了哪些限制”，而应该是 governance key、source-gated expansion 与 runtime adapter 怎样共同保护控制平面本身不被 agent 回写篡改。

### 6.25 为什么 workflow 常被写成“命令变体”，而不是后台对象

仍然很容易。

如果只看到：

- `kind: 'workflow'`
- workflow command
- workflow prompt

就会误以为 workflow 主要是命令系统的一个分支。

更稳的对象写法应直接交出：

- `local_workflow`
- `workflow_progress`
- `independent transcript`
- `worktree`
- `cleanup semantics`

这样 workflow 才会被看成受治理的后台对象，而不是多步 prompt 的命令皮肤。

### 6.26 为什么 search / selection / scroll 的理论价值常被低估

也是。

如果只把前台写成：

- transcript mode
- sticky prompt
- message actions

还不够。

真正让前台变强的是：

- 搜索只索引“用户真正看到的文本”
- 选择会追踪 drag-to-scroll、soft-wrap 与虚拟锚点
- 滚动会显式区分用户滚动与程序滚动

这些都不是 UI 抛光，而是在精确定义什么才算前台真相。

### 6.27 为什么治理实践和治理理论常被分得太开

是的。

如果只在哲学层写：

- policy
- allowlist
- managed settings
- channelsEnabled

却不把它们在实际团队里的启用顺序、职责分层与 opt-in 节奏写出来，蓝皮书仍然会停留在“知道机制”而不是“会用机制”。

所以像 channels 这样的能力，必须同时写：

- 协议边界
- 产品边界
- 组织实践

### 6.28 为什么“补完全貌”常被高估，而“可见边界”被低估

也是。

尤其在 workflow、部分 remote path、某些 gated 模块上，这个问题最明显。

更稳的写法应该是：

1. 先说明公开可见边界已经证明了什么。
2. 再说明哪些地方只能做保守推断。
3. 最后明确哪些地方不该再继续脑补。

这比“因为没看到全部内核，所以什么都不敢说”更好；也比“先凭经验把缺失实现补满再解释”更可靠。

### 6.29 为什么“显式失败”的价值常被低估

也会。

当我们只看 happy path 时，很容易把：

- outbound-only mode 明确报错
- unknown subtype 明确回 error
- cancel request 明确发送
- aborted promise 明确 reject

这些行为当成“实现细节”。

但它们其实是维持系统一致性的硬约束。宿主路径一旦在这些地方追求“看起来没报错”，系统就会迅速滑向假成功、状态错觉与时序不一致。

### 6.30 为什么宿主协议常被写成“请求-响应”，而不是“闭环状态机”

也仍然会。

如果我们只写：

- `control_request`
- `control_response`
- 宿主返回 success / error

就会漏掉同样关键的后半段：

- `system:init`
- `system:status`
- `system:session_state_changed`
- `task_*`
- `tool_progress`
- `elicitation_complete`

而这些 follow-on 消息，往往才决定宿主何时真正知道“这次动作完成了”。更成熟的写法必须把宿主协议理解成闭环，而不是单向 RPC。

### 6.31 为什么运行时真相常被写成“单通道消息流”，而不是“事件流 + 状态回写”

也还是会。

如果我们只写：

- `SDKMessage`
- `stream_event`
- `task_*`
- `status`

却不写：

- `worker_status`
- `requires_action_details`
- `external_metadata.pending_action`
- `external_metadata.permission_mode`
- `external_metadata.model`

就会让读者误以为宿主只能靠消息回放自己猜状态。

但 Claude Code 明显不接受这种设计：

- 时间线走事件流
- 当前真相走状态回写
- 恢复时还会主动清旧 metadata、防止 stale state

如果蓝皮书不把这两条通道写清，读者就会继续低估 Claude Code 在远程宿主、恢复和 attach-late 场景里的成熟度。

### 6.32 为什么 Prompt 稳定性常被误写成“强文案”，而不是“same-world compiler 合同”

也会。

如果我们只摘几段 system prompt 句子来评论：

- 语气很强
- 约束很严
- 角色很清楚

而不写 Prompt 线其实先依赖一套同一世界的装配与约束顺序，就会把 Claude Code 最深的一层 prompt 能力写丢。

真正起作用的，不是某句话，而是“同一世界不必在每一层重新协商”这件事先被写成制度；若讨论已经进入具体 assembly / witness nouns，统一回 `84 / guides/51`。

### 6.33 为什么“源码质量本身也是能力”常被低估

也会。

如果我们只写：

- 功能很多
- 状态机很多
- 协议很完整

却不写：

- `lazySchema`
- `systemPromptSection`
- `WorkerStateUploader`
- prompt cache break detection
- 依赖图拆分与 dead code elimination

就会错过 Claude Code 之所以显得先进的另一半原因。

它不是只是“功能全”，而是把 cache、边界、恢复、依赖图也工程化了。

### 6.34 为什么前台常被写成“显示层”，而不是“用户可见真相层”

也会。

如果只说：

- transcript search
- sticky prompt
- selection
- scroll

是“交互体验”，就还是写浅了。

更稳的对象顺序是：

- 前台在主动定义什么才算用户真正看见的真相
- 搜索、复制、导航与来源判断都应服从这套真相

否则系统就会出现搜到看不见的内容、复制结果和高亮不一致、以及无法区分消息主语来源这些失真。

### 6.35 为什么 UI transcript 常被当成模型看到的 transcript

也很容易。

如果把：

- 前台消息顺序
- attachment 展示顺序
- tool reference 边界
- virtual messages

直接等同于模型请求里的消息序列，就会继续低估 Claude Code 更深的一层协议意识。

更稳的写法应直接承认：前台真相与协议真相相关，但并不相同。若讨论已进入 `message lineage`、projection 规则或协议编译 chokepoint，统一回 `guides/51 / architecture/README`。

### 6.36 为什么多入口页必须被纳入规范入口治理

这类入口失真很常见。

如果根目录同时存在：

- 规范主线入口
- 历史别名页
- 兼容命名页

但目录与导航又不明确告诉读者：

- 哪些是规范入口
- 哪些只是兼容别名

那目录结构本身就会先制造理解噪音。

更成熟的做法不是删除一切旧页，而是：

- 保留兼容
- 明确规范
- 让导航层显式承担目录治理职责

否则文档会在入口层先失真，再把这种失真传导到正文层。

### 6.37 为什么“小文件更多”不是成熟架构的充分条件

这种尺寸迷信很常见。

如果后续继续用文件大小、目录层级或 helper 数量直接替代架构判断，这种分析就会继续把“看起来更碎”误写成“真的更好”；若讨论已经进入合法复杂度中心、leaf module 或时序耦合这些 owner nouns，统一回 `philosophy/87 / architecture/README`。

### 6.38 为什么热点大文件必须被点名其真实职责

这种回避同样常见。

如果我们继续只说某个文件很大、很复杂，却不肯继续追问“这里到底在承受什么合法复杂度”，那分析就仍停留在尺寸判断，而没有进入结构判断；若讨论已经进入 kernel / shell / chokepoint、seams 或 leaf modules 这些 owner nouns，统一回 `philosophy/87 / architecture/README`。

### 6.39 为什么“能力存在”不等于“能力应先暴露给模型”

这种说明书式误写很常见。

如果后续继续把：

- 工具全集
- agent 列表
- MCP instructions
- 高波动能力说明

顺手写成“模型应该一开始都看见”，那我们就仍然在用说明书思维写 agent runtime。

更成熟的写法必须先问：

- 当前轮最小必要可见面是什么
- 哪些能力应该 deferred
- 哪些变化应该走 delta
- 哪些来源甚至没有资格参与定义模型可见世界

否则这种分析会继续低估“可见性控制平面”这条主线。

### 6.40 为什么“更多能力暴露”不等于“系统更强”

这种乐观错觉也很常见。

如果后续继续默认：

- 知道得更多 = 更聪明

就会漏掉 Claude Code 更深的一层工程智慧：

- 先降低模型可见世界的熵，再要求模型发挥能力

更成熟的分析不该只问：

- Claude Code 会什么

还要问：

- Claude Code 何时让模型看见什么

因为真正被治理的，不只是能力本身，还有：

- 能力出现的时机
- 能力出现的来源
- 能力出现的位置

否则“安全”“token”“prompt 稳定性”三条线就会重新被写散。

### 6.41 为什么 observability 是正式运行时合同，而不是调试辅助层

这种降格处理很常见。

如果后续继续把 context usage、pending_action、worker_status 或 cache break reason 都写成“辅助观测”或“开发者调试接口”，这种分析就会继续低估 Claude Code 的运行时自觉；若讨论已经进入 observability 的真相分层，统一回对应 owner 页，而不是在自校段重发结构卡。

### 6.42 为什么“还能看见更多图表”不等于可解释性已经成立

这种图表崇拜也很常见。

真正的可解释运行时，不只是数据更多，而是观测仍能把你带回正确动作；若讨论已经进入输入分层、状态快照、稳定性因果或动作导向这些 owner nouns，统一回对应 owner 页。

### 6.43 为什么“拆出更多小文件”不等于依赖图更健康

这种模块化幻觉很常见。

如果后续继续把 helper 更多、文件更碎、目录更深直接等同于“模块化更成熟”，这种分析就会继续把“尺寸审美”误写成“结构质量”；若讨论已经进入依赖诚实、高扇入面或脏边隔离这些 owner nouns，统一回 `philosophy/87 / architecture/README`。

### 6.44 为什么 DRY 不能凌驾于依赖图健康之上

这种默认同样常见。

如果后续继续把更少重复、更多共享 helper 自动当作更优设计，就会把依赖图健康写浅；若讨论已经进入 runtime edge、tiny file、复用收益或依赖代价这些 owner nouns，统一回 `philosophy/87 / architecture/README`。

### 6.45 为什么 same-world compiler 不等于“提示词更完整”

这种文案化误读仍然高频出现。本节只保留 why，不再自己补一条 Prompt 审读位。

如果后续继续把：

- system prompt 更好
- 提示词更完整
- 角色设定更聪明

当成主解释，就会继续低估 Claude Code 更深的一层设计；若讨论已经进入具体工作语法对象，统一回 `84 / guides/51`。

### 6.46 为什么“省 token”不等于“压缩文本”

这种成本化浅解很常见。

如果后续继续把 compact、更短文本和更省上下文当作主解释，这种分析就会继续把“反扩张运行时”误写成“成本优化技巧”。更稳的 why 应先问：这里省掉的是文本，还是免费暴露、免费继续与不该留下的 transient authority；更细的工作集边界与外置对象统一回治理 owner 页。

### 6.47 为什么源码质量判断不在“结构更好看”

这种静态快照式判断也很常见。

如果后续继续把目录更细、文件更碎、模块更多当作主解释，这种静态快照式评价，就会继续把“可演化内核”误写成“当前分层截图更漂亮”。更稳的自校问句应是：新增之后，我们是否仍保留正式反对、局部重构与低成本回退的可能性，而不是只多了一张更整齐的结构截图。

### 6.48 为什么“语义压缩”不等于“摘要更强”

这种摘要化误写很常见。

如果后续继续把这条线写成一组独立功能，而不先问是否仍保住可继续工作的行动语义，这种分析就会继续把它误写成“摘要能力更强”；更细的 continuation 资产对象统一回 `guides/51`。

### 6.49 为什么资源分配不能被写成多个局部开关

这种特性拼盘式理解很常见。

如果后续继续把 permissions、settings、token budget 与 context suggestions 分别写成各自独立的 feature，就会继续把“资源宪法”误写成“几组碰巧相关的设置”；更细的资源分配轴统一回治理 owner 页。

### 6.50 为什么“保留重构可能性”不是团队软技能

这种人格化归因也很常见。

如果后续继续把这种能力写成“作者比较有经验”或“团队习惯较好”，就会继续把“演化制度设计”误写成“工程师个人修养”。更稳的自校问句应是：这次设计增长后，重构可能性是否被制度化保住，而不是继续寄托于个人经验。

### 6.51 为什么“人类-模型协调成本”不只是 UI 细节

这种体验化降格很常见。

如果后续继续把这条线写成前台功能或体验优化，而不先问是否仍保住人类接手 runtime 的成本边界，这种分析就会继续低估 prompt 作为协作接口的那一面。

### 6.52 为什么自由不等于约束更少

这种自由想象很常见。

如果后续继续把：

- bypass 更多
- ask 更少
- 暴露更全

自动当成更高自由，就会继续误解 Claude Code 更成熟的判断：

- 真正的自由是约束不再破坏行动

### 6.53 为什么可读性不等于代码更漂亮

这种审美化理解也很常见。

如果后续继续把：

- 命名更优雅
- 注释更丰富
- 结构更清晰

只写成审美收益，而不写它们如何降低误改、误扩展与制度失忆成本，这种分析就会继续漏掉源码作为治理界面的那一层。更稳的自校问句应是：这段代码是否降低了误改与制度失忆成本，并保留第一条正式反对路径，而不是只更好看。

### 6.54 为什么 Prompt 首先要证明同一主语，而不是比较提示词顺滑度

这种文案化理解很常见。

如果后续继续只比较：

- 某段 system prompt 更强
- 某种措辞更顺滑
- 某条 instruction 更贴近专家文风

而不先追问“这里还是不是同一主语”，这种分析就会继续低估 Claude Code 把意图编译成协作语法的那一层。如果必须落到更细的 Prompt 术语才能说明白，是否说明本段还没有先证明同一主语？

### 6.55 为什么安全首先要回答扩张如何被定价，而不是“拦得更严”

这种拦截器想象很常见。

如果后续继续只写：

- deny 更多
- ask 更严
- bypass 更少

而不写“扩张为什么必须先被同一秩序定价”，这种分析就会继续把资源秩序误写成单点拦截器；若这里已经需要靠 signer / 弱读回面 / reopen 责任的不对称才能说明白，统一回 `10 / security / risk`。

### 6.56 为什么源码质量判断不能退回“结构截图更漂亮”

这种截图式赞美很常见。

如果后续继续只写：

- 模块更多
- 目录更整齐
- 文件更小

而不写“为什么 later maintainer 仍保有正式反对与继续重构的能力”，这种分析就会继续高估静态结构，低估可演化内核的真正价值；若讨论已经进入 quality verdict 或 object/seam nouns，统一回 `philosophy/87 / architecture/README`。

### 6.57 为什么 Prompt Constitution 只是 same-world compiler 的制度边界

这种秘诀化误读很常见。这一节只补“它为什么不是秘诀”的边界，不再重复解释 Prompt 强度本身。

如果后续继续只讨论：

- system prompt 更聪明
- 某一段 instruction 更强
- 某种措辞更像秘诀

而不写“它为什么只是制度边界，而不是第二套 Prompt 魔法”，

这种分析就会继续低估 Claude Code 把 `same-world compiler` 写成制度边界那一层。如果必须靠 Constitution nouns 才能成立，是否说明本段还没有把它写成制度边界，而只是换一种 Prompt 说法？

### 6.58 为什么安全与省 token 不能只停在“统一预算器”

这种中层概括仍然常见。

如果后续继续只写：

- 都在定价
- 都在反扩张

而不写“为什么统一预算器仍不足以解释治理成熟度”，这种分析就会继续漏掉 Claude Code 最成熟的制度设计部分。如果必须展开检查顺序、失败语义、人工回退、弱读回面不能代签或 cleanup 后仍剩哪些 liability，才能站住，是否说明本段还没有说明“统一预算器为什么不够”？

### 6.59 为什么构建系统是架构工具，而不是打包尾巴

这种后处理想象很常见。

如果后续继续只写：

- build-time gate
- external build
- public mirror 缺口

而不写“构建如何反向塑造入口与恢复边界”，这种分析就会继续低估构建、入口与恢复对源码秩序的塑形作用；若讨论已进入具体 build / shell / registry nouns，统一回 `architecture/README`。

### 6.60 为什么结论必须继续压成可自校的问题梯子

这种结论先行的惯性很常见。

如果后续继续只写：

- Prompt 为什么强
- 安全为什么成熟
- 源码为什么先进

而不把这些判断继续压成：

- 设计前该先反问什么
- 失稳前该先质疑什么
- 迁移时该先核对什么

这种分析就会继续停留在“会解释”，而没有进入“会自我校准”。

### 6.61 为什么“同题坏解”对照不能缺席

这种缺席同样常见。

如果后续继续只写：

- Claude Code 式正解是什么

而不写：

- 同一个问题最常见的坏解法是什么
- 为什么那种坏解法看起来也很合理
- 应该从哪一步开始改写

这种分析就会停留在“告诉你正解是什么”，却无法帮助你识别自己已经走错了哪条路。

### 6.62 为什么“渐进改写顺序”不能缺席

这种缺席也很常见。

如果后续继续只写终态、正解与坏解，而不继续追问第一条可逆改写信号仍在哪里，这种分析就会更像理论地图，因为它没有证明增长如何在不损坏当前真相、当前能力与恢复路径的前提下继续发生。

### 6.63 为什么三条母线必须压成三句彼此对称的判断

这种高阶压缩缺口也很常见。

如果后续继续只写：

- Prompt 为什么强
- 安全为什么成熟
- 源码为什么先进

而不继续逼自己回答：

- Prompt 线到底在保护哪一种诚实
- 治理线到底在保护哪一种合法准入秩序
- 源码质量线到底在保护哪一种可继续重构的判断标准

这种分析就会继续拥有很多高阶结论，却仍缺少最短的高阶压缩。

## 7. 持续审读协议

只要继续扩写这类制度分析，就应持续通过下面八张判据卡反复审读，而不是让判断退回某一轮作者记忆。

### 7.1 事实判据

先问：

1. 这段结论是事实、解释，还是推论。
2. 如果源码明天升级，最可能先失效在哪里。
3. 这里写的是可见边界，还是脑补出来的全貌。

### 7.2 入口判据

先问：

1. 这段内容是否又在新增同题首答入口，而不是继续服务已有主线。
2. 它是在回答第一问题，还是在提前替另一个页面预支答案。
3. 目录结构是否又让同题入口彼此重叠，或把同一对象链拆成多个前门。

### 7.3 对象判据

先问：

1. 这里描述的是正式对象模型，还是表面入口清单。
2. 这里写的是 answer stream，还是 event stream；写的是快照，还是时间线。
3. 宿主、MCP、workflow、channel、remote 是单一对象，还是控制面、连接面、状态面共同组成的平面。

### 7.4 Prompt 判据

先问：

1. 这里写的是某段文案，还是同一世界为何仍能保持单源。
2. 这里是不是又把 Prompt 写回了替身，而不是单源判断本身。
3. 如果删掉最后一条回复和摘要，我是不是仍能判断这里写的是同一主语，而不是靠补写机制才成立。

### 7.5 治理判据

先问：

1. 这里写的是治理主语，还是 mode、审批弹窗、usage 条这类 projection 替身。
2. 这里写的是更严拦截，还是扩张为什么必须先被统一定价。
3. 我是不是又把弱读回面、continuation consumer 或 cleanup 结果词写成 signer，或还在这里重发具体治理链段与机制顺序，而不是老老实实停在自校层。

### 7.6 结构判据

先问：

1. 这里写的是代码很多很复杂，还是当前真相为什么不能退回 replay / 目录体感判断。
2. 这里写的是静态结构更漂亮，还是 later maintainer 为什么仍保有正式反对与继续重构的能力。
3. 如果必须落到具体 object/seam nouns 才能成立，是否说明这里还停在术语索引，而没有说清结构理由。

### 7.7 反例判据

先问：

1. 同题坏解法是什么。
2. 它为什么更贵、更脆或更难恢复。
3. 我是否已经给出最先失稳点和改写起点，而不是只在重复 owner 页术语。

### 7.8 迁移判据

先问：

1. 我是不是又把升级写成一次性重写，而没有证明它仍保留可逆边界。
2. 当前真相、当前能力与恢复路径是否在增长过程中被同时保住。
3. 第一条失稳信号出现时，我是否知道该先收回哪类判断，而不是等到全局失控。

### 7.9 迭代判据

先问：

1. 我是不是又把三条母线写回坏解，而没有回到各自 owner 页。
2. 我是不是只给了正解，没有点名同题坏解为什么看起来也像正解。
3. 我是不是没有给出第一条反证信号，而只是继续堆更细的术语。
4. 我是不是没有交代渐进改写顺序，而只是把迁移词表重新抄了一遍。

### 7.10 并行判据

先问：

1. 这次拆给不同 agent 的，是不同第一问题，还是同一问题的不同页面切片。
2. 合并时保住的是同一个问题主语，还是只是在拼接更长摘要。
3. 哪个结论已经被过早升级成全局公式，哪个还只是局部经验。
4. 目录入口层有没有因为这次并行深挖重新长回新的第二前门。

## 8. 最后一句

从第一性原理看，Claude Code 的价值不在“替你写代码”，而在“替你组织软件工程中的观察、决策、行动、记忆、协作与恢复”。  
从苏格拉底式反思看，更稳的写法也很明确：少一点结果词和概念重复，让主线更短；多一点对象链、同题坏解、第一条反证信号与渐进改写顺序，让专题不仅会解释，还会持续自我校准。
