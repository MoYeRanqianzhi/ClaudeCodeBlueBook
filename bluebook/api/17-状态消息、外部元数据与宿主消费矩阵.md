# 状态消息、外部元数据与宿主消费矩阵

这一章回答五个问题：

1. Claude Code 的 runtime 状态真相通过哪些正式通道暴露给宿主。
2. 哪些状态走 `SDKMessage` 事件流，哪些还会镜像到 `worker_status` / `external_metadata`。
3. 哪些信号是双通道同步，哪些只存在于单一通道。
4. 为什么不能把 schema 全集直接等同于每个 consumer 都完整消费。
5. 宿主接入时，应该把哪个信号当成时间线，哪个信号当成最新快照。

## 1. 先说结论

Claude Code 的宿主状态面至少有两条正式通道：

1. `SDKMessage` 事件流：负责时间顺序、执行脉搏、进度和补充信号。
2. `worker_status` / `requires_action_details` / `external_metadata`：负责可查询快照、恢复后真相和远端 UI 同步。

但这还不够，宿主还必须再接受第三个现实约束：

3. 不是每个 adapter / consumer 都完整消费 schema 全集，一些消息会被忽略、降级或过滤。

所以 Claude Code 的 API 现实不是“一个统一消息流”，而是：

- 一条事件时间线
- 一组外化状态快照
- 若干消费者子集与兼容层

- `claude-code-source-code/src/utils/sessionState.ts:1-149`
- `claude-code-source-code/src/state/onChangeAppState.ts:19-91`

- `claude-code-source-code/src/cli/transports/ccrClient.ts:645-662`
- `claude-code-source-code/src/remote/sdkMessageAdapter.ts:223-267`

## 2. 通道一：`SDKMessage` 负责时间顺序与执行脉搏

`coreSchemas.ts` 里可见的状态类 / 补充类消息至少包括：

- `system:init`
- `system:status`
- `system:compact_boundary`
- `system:api_retry`
- `system:local_command_output`
- `system:task_started`
- `system:task_progress`
- `system:task_notification`
- `system:session_state_changed`
- `auth_status`
- `rate_limit_event`
- `tool_use_summary`
- `prompt_suggestion`

其中几类尤其关键：

1. `system:status` 主要承载 `compacting|null` 与 `permissionMode`，不是 turn lifecycle 真相本身。
2. `session_state_changed` 才暴露 `idle | running | requires_action`。
3. `task_*` 暴露 background / workflow 生命周期。
4. `auth_status`、`rate_limit_event`、`prompt_suggestion` 暴露补充运行信号，而不是模型正文。

这说明 `SDKMessage` 的职责不是给宿主一个“最后答案”，而是给宿主一条按时间展开的 runtime 时间线。
同时也说明不能把所有 `system` 类消息都误读成“当前总状态”。
进一步说，`auth_status`、`rate_limit_event` 甚至都不是 `type:'system'`，而是平级消息类型。

证据：

- `claude-code-source-code/src/entrypoints/sdk/coreSchemas.ts:1533-1806`
- `claude-code-source-code/src/utils/sdkEventQueue.ts:1-121`
- `claude-code-source-code/src/cli/print.ts:1051-1135`
- `claude-code-source-code/src/QueryEngine.ts:940-966`

## 3. 通道二：`worker_status` 与 `external_metadata` 负责最新快照

### 3.1 `worker_status` 与 `requires_action_details`

`CCRClient.reportState(...)` 会把当前状态写回到远端 worker state：

- `worker_status`
- `requires_action_details`

其中 `requires_action_details` 保留：

- `tool_name`
- `action_description`
- `request_id`

这不是事件日志，而是供远端 UI / webhook / 查询接口直接读取的当前快照。

证据：

- `claude-code-source-code/src/cli/transports/ccrClient.ts:645-658`
- `claude-code-source-code/src/utils/sessionState.ts:4-25`

### 3.2 `external_metadata`

`sessionState.ts` 与 `onChangeAppState.ts` 共同说明，至少有五类状态会镜像到 `external_metadata`：

- `pending_action`
- `permission_mode`
- `is_ultraplan_mode`
- `model`
- `task_summary`

它们的共同特点是：

- 不是为了重放完整时间线
- 而是为了让后接入的 consumer 直接读到当前事实

尤其 `pending_action` 很典型。源码注释写得很直白：

- typed 的 `RequiresActionDetails` 走 proto / webhook
- full object 走 `external_metadata.pending_action`

这样前端不需要回扫整条事件流，就能知道当前阻塞在什么动作上。

证据：

- `claude-code-source-code/src/utils/sessionState.ts:8-25`
- `claude-code-source-code/src/utils/sessionState.ts:92-115`
- `claude-code-source-code/src/state/onChangeAppState.ts:43-91`
- `claude-code-source-code/src/cli/print.ts:2918-2941`
- `claude-code-source-code/src/cli/print.ts:3746-3752`

### 3.3 初始化与恢复时还会清理旧快照

`CCRClient.initialize()` 在 worker 注册时会主动写：

- `worker_status: 'idle'`
- `external_metadata.pending_action: null`
- `external_metadata.task_summary: null`

这说明作者明确知道：快照通道如果不主动修复，就会把崩溃前的旧状态错误地带到新进程。

证据：

- `claude-code-source-code/src/cli/transports/ccrClient.ts:476-489`

## 4. 双通道矩阵：哪些状态走哪里

| 状态真相 | `SDKMessage` 通道 | 快照通道 | 说明 |
| --- | --- | --- | --- |
| permission mode 变化 | `system:status(permissionMode)` | `external_metadata.permission_mode`、`is_ultraplan_mode` | 同时满足实时 UI 与恢复后查询 |
| requires action | `system:session_state_changed(requires_action)` | `worker_status=requires_action`、`requires_action_details`、`pending_action` | 状态机与阻塞细节一起外化 |
| model 切换 | 通常没有专属即时 system message | `external_metadata.model` | 主要为了恢复与远端一致性 |
| task 生命周期 | `task_started`、`task_progress`、`task_notification` | `task_summary` | 事件流更细，metadata 更粗 |
| auth / rate limit | `auth_status`、`rate_limit_event` | 无 | 这类高频临时态更适合事件流 |
| prompt suggestion | `prompt_suggestion` | 无 | 引导信号，不是稳定快照 |

这里最容易看错的一点是：

- 不是所有状态都双写
- 也不是所有状态都只写一边

Claude Code 明显在区分：

1. 需要时间顺序的
2. 需要可查询快照的
3. 两者都需要的

## 5. schema 全集不等于 consumer 全集

即使 `coreSchemas.ts` 已经声明了消息类型，也不能假设每个 consumer 都完整消费。

### 5.1 remote adapter 会主动忽略一批消息

`sdkMessageAdapter.ts` 明确忽略：

- `auth_status`
- `tool_use_summary`
- `rate_limit_event`

而 `system` 里也只挑：

- `init`
- `status`
- `compact_boundary`

其他 subtype 记录日志后直接忽略。

证据：

- `claude-code-source-code/src/remote/sdkMessageAdapter.ts:223-267`

### 5.2 `local_command_output` 甚至会被降级成 assistant

`utils/messages/mappers.ts` 里专门写了兼容层：

- 不直接发 `system:local_command_output`
- 而是转成 `assistant`

原因也在注释里写明了：

- 某些下游 consumer 不认识这个 subtype

这说明“schema 已存在”和“生态都已消费”仍是两件事。

证据：

- `claude-code-source-code/src/utils/messages/mappers.ts:145-196`

### 5.3 direct connect 也会过滤内部 / 窄适配消息

`DirectConnectSessionManager` 在转发时显式过滤：

- `streamlined_text`
- `streamlined_tool_use_summary`
- `system:post_turn_summary`

证据：

- `claude-code-source-code/src/server/directConnectManager.ts:100-109`

## 6. 宿主接入时应如何选权威信号

最稳妥的做法不是“永远只相信一条通道”，而是：

1. 需要完整时间线时，消费 `SDKMessage`。
2. 需要当前快照时，优先读 `worker_status` / `external_metadata`。
3. 需要兼容多 consumer 时，假设部分消息会被过滤或降级。

换句话说：

- 事件流回答“刚才发生了什么”
- 快照通道回答“现在到底是什么状态”

如果把这两件事混成一件事，宿主就会在 attach-late、重连、过滤或恢复场景下迅速失真。
