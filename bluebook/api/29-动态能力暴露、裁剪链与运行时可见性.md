# 动态能力暴露、裁剪链与运行时可见性

> Evidence stamp
> - `evidence mode`: `public-evidence only`
> - `rung`: `consumer subset / callability analysis`
> - `verdict`: 本页的“三层可见性”只是 `102` 已锁定后的 subset / callability 拆解，不是另一张支持度 classifier
> - `retreat`: 若还要先判 host-facing truth claim-state 是否成立，先回 `api/README` 与 `guides/102`

这一章回答四个问题：

1. 为什么 Claude Code 源码里能看到的命令和工具，不会全部出现在同一个会话里。
2. 命令面和工具面分别经过哪些裁剪链。
3. “代码里存在”“当前可见”“当前可调用”为什么必须分开。
4. 这条裁剪链如何反过来塑造 prompt、权限和 token 成本。

## 1. 先说结论

Claude Code 的能力表面不是静态清单，而是动态可见性系统；但这套“源码全集 / 当前会话可见集 / 当前调用可行集”只是在 API owner 路径里继续拆 `consumer subset / runtime gate`，不代替 `102` 与 `api/README` 重新定义支持 truth。

至少要区分三层：

1. 源码全集：
   - `commands.ts` 和 `tools.ts` 里定义的能力全集。
2. 当前会话可见集：
   - 受 feature、`USER_TYPE`、plugin、skill、mode、环境变量、session state 影响后，真正暴露给模型或 UI 的集合。
3. 当前调用可行集：
   - 再经过 permission deny、tool pool、host policy、runtime context 后，实际还能被执行的集合。

如果把这三层写成一层，就会不断误把“存在代码”写成“稳定能力”。

## 3. 命令面：先定义全集，再叠加多层来源

`commands.ts` 里最重要的结构有两个：

1. `INTERNAL_ONLY_COMMANDS`
2. `COMMANDS()`

这意味着内置命令首先就分成：

- 默认公开命令
- internal-only 命令
- 条件启用命令
- 非交互变体命令

随后还会继续叠加：

- bundled skills
- builtin plugin skills
- skill dir commands
- plugin commands
- dynamic skills

所以命令 API 真正暴露给模型或用户前，已经是一个合成结果，而不是某个静态数组。

## 4. 命令面的裁剪链

命令面至少经过六层裁剪：

1. 构建 / 用户类型裁剪
2. feature / 条件命令裁剪
3. 第三方服务环境裁剪
4. plugin / skill 裁剪
5. `isEnabled` / availability 裁剪
6. 运行时 mode 裁剪

所以命令面更像“声明式装配面”，不是“硬编码菜单”。

## 5. 工具面：`getAllBaseTools()` 也不是最终可见集

工具面有两个最关键的函数：

1. `getAllBaseTools()`
2. `getTools(permissionContext)`

前者回答：

- 当前环境下“理论上可能存在”的基础工具全集是什么。

后者回答：

- 当前 runtime context 下“应该真正交给模型看的工具集”是什么。

这两者之间不是小差异，而是 Claude Code 工具系统最重要的收口层。

## 6. 工具面的裁剪链

工具面至少经过八层裁剪：

1. 环境与 feature 裁剪
2. optimistic catalog 裁剪
3. simple mode 裁剪
4. REPL mode 裁剪
5. coordinator mode 裁剪
6. permission deny 预裁剪
7. MCP server-prefix 裁剪
8. call-time permission 裁剪

还要再补两层，否则对“动态可见性”的理解仍然偏浅：

9. prompt-cache-stable 排序
10. deferred tool / tool-search / model-compat 裁剪

所以 Claude Code 的 tool catalog 从一开始就不是“尽可能多地展示工具”，而是“尽可能早地剥离当前无权或无意义的动作空间”。

### 6.1 动态能力暴露同时是 token 策略

`query.ts` 会在主循环真正调用模型前先做 tool result budget、snip、microcompact、autocompact；`claude.ts` 又会根据当前模型能力剥离 stale tool-search 字段、把 deferred tool 说明尽量外移到 delta attachment。

这说明 Claude Code 的动态能力暴露不只服务权限和正确性，还直接服务：

- prompt cache 稳定
- token 成本收缩
- 恢复分支可控

也就是说，“显示哪些能力”本身就是 token 预算设计的一部分。

## 7. 为什么这条裁剪链本身就是 API 的一部分

这不是内部实现细节，原因有三：

1. 它决定 prompt 里会看到哪些工具描述。
2. 它直接影响 context usage 和 token 预算。
3. 它改变模型对当前运行时“可行动空间”的认知。

`promptCacheBreakDetection.ts` 甚至会把 tools hash、per-tool schema hash 和 global cache strategy 一起纳入追踪，这说明能力暴露面一旦变化，作者就把它当成：

- 可观测的 cache break 事件

换句话说，能力可见性本身就是一种 runtime API。

## 8. 最容易写错的三种说法

### 8.1 “源码里有这个工具，所以 Claude Code 支持它”

不够准确。  
更准确的说法应该是：

- 源码里存在该工具定义，但它是否进入当前 catalog 仍受环境、mode、flag 与 permission context 约束。

### 8.2 “命令列表就是用户实际能看到的命令”

也不准确。  
用户实际能看到的是：

- 静态命令
- skill / plugin 注入命令
- availability / enablement 过滤后的结果

### 8.3 “permission 只在调用时发生”

不准确。  
Claude Code 明确在调用前就做了 deny 预裁剪，这会改变模型的推理空间，而不仅仅是执行结果。

## 9. 对接入方和蓝皮书作者的直接启发

如果你是接入方：

1. 不要把 catalog 当作永久能力表。
2. 要接受“同一宿主、不同会话”暴露面不同。
3. 要把 permission deny 和 tool visibility 一起设计，而不是分开处理。

4. 要接受“少暴露能力”本身就是性能优化，而不只是安全限制。

如果你在继续写蓝皮书：

1. 命令和工具都必须写“全集”和“当前暴露面”的区别。
2. 不能把 feature-gated / internal-only / plugin-injected 混写成一个层级。
3. 不能把“可见”和“可执行”混写成一个词。

## 10. 一句话总结

Claude Code 的命令面和工具面不是静态功能列表，而是被 feature、mode、plugin、policy 与 permission context 联合裁剪出的动态能力暴露面。
