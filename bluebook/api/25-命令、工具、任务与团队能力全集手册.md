# 命令、工具、任务与团队能力全集手册

这一章的目标不是继续拆字段，而是把 Claude Code 的“能力全集”放回一张统一地图。

核心结论：

- Claude Code 的能力面不是“slash command + 几个 tools”，而是“命令控制平面 + 工具执行平面 + 任务/团队协作平面 + remote/worktree/trigger 自动化平面”的组合体。

## 2. 先把能力全集分四层

### 2.1 命令控制平面

`commands.ts` 同时定义了三层命令来源：

1. `COMMANDS()` 内置主表
2. `INTERNAL_ONLY_COMMANDS` 会被外部构建裁掉的内部门类
3. 运行时继续叠加 skills、plugins、workflow、dynamic skills

这意味着 slash command 不是静态清单，而是会话级控制平面。

### 2.2 工具执行平面

`getAllBaseTools()` 定义了当前环境下的 builtin tool 全集，`getTools()` 则继续按：

- simple mode
- REPL mode
- deny rules
- `isEnabled()`

做可见性收口。

所以 Claude Code 的 tool truth 也不是静态 truth，而是运行态 truth。

### 2.3 任务与团队协作平面

任务/团队能力并没有被塞进单一的 `AgentTool`，而是拆成：

- `TaskCreateTool`
- `TaskGetTool`
- `TaskUpdateTool`
- `TaskListTool`
- `TaskStopTool`
- `TaskOutputTool`
- `SendMessageTool`
- `TeamCreateTool`
- `TeamDeleteTool`

这说明任务对象、团队对象与通信对象已经进入正式 runtime。

### 2.4 自动化与执行环境平面

除了“本地起子代理”，还存在另外几类能力：

- `EnterWorktreeTool` / `ExitWorktreeTool`
- `RemoteTriggerTool`
- scheduler / cron 相关工具
- `SendMessageTool` 对现有 agent/task 的继续协作

所以 Claude Code 不是一次性 tool runner，而是长寿命任务 runtime。

## 3. 命令面全景

### 3.1 命令不是只有最终对用户可见的 slash

在当前源码快照里：

- `src/commands/` 下可见 `86` 个一级命令目录
- `COMMANDS()` 主表收拢 builtin 命令
- 运行时继续叠加 bundled skills、builtin plugin skills、skill dir commands、workflow commands、plugin commands、plugin skills、dynamic skills

这四层叠加之后，Claude Code 的命令面已经更接近“运行时能力索引”。

### 3.2 命令域可以按六类看

| 命令域 | 代表能力 |
| --- | --- |
| 会话与状态 | clear、compact、resume、session、rewind、memory、context |
| 配置与模式 | config、permissions、plan、fast、model、output-style、sandbox |
| 扩展与连接 | mcp、skills、plugin、reload-plugins、remote-env |
| 开发与交付 | diff、review、security-review、branch、tag |
| 运营与诊断 | doctor、usage、stats、feedback、upgrade |
| 协作与实验 | agents、tasks、bridge、voice、workflow、fork、buddy |

这里最重要的不是枚举本身，而是命令面已经覆盖：

- runtime mode
- policy surface
- extension surface
- delivery workflow
- session lifecycle

## 4. 工具面全景

### 4.1 基础工作集工具

`getAllBaseTools()` 把下列工具视为 builtin 正式面：

- 文件：`FileReadTool`、`FileEditTool`、`FileWriteTool`、`NotebookEditTool`
- Shell：`BashTool`、`PowerShellTool`
- 搜索：`GlobTool`、`GrepTool`、`WebFetchTool`、`WebSearchTool`
- 认知与控制：`TodoWriteTool`、`AskUserQuestionTool`、`BriefTool`
- 扩展：`SkillTool`、`ListMcpResourcesTool`、`ReadMcpResourceTool`

### 4.2 编排与协作工具

这一层是 Claude Code 容易被低估的部分：

- `AgentTool`
- `SendMessageTool`
- `TaskCreateTool` / `Get` / `Update` / `List` / `Stop`
- `TaskOutputTool`
- `TeamCreateTool` / `TeamDeleteTool`

它们共同把“起一个子代理”扩展为：

- 创建任务
- 查询任务
- 持续通信
- 查看输出
- 建立团队
- 回收团队

### 4.3 执行环境与自动化工具

再往外还有：

- `EnterPlanModeTool` / `ExitPlanModeV2Tool`
- `EnterWorktreeTool` / `ExitWorktreeTool`
- `RemoteTriggerTool`
- cron 系工具

它们说明 Claude Code 已经把“如何组织执行现场”也上升为正式能力面。

## 5. 任务、团队与 mailbox 不是附属功能

### 5.1 任务能力有独立 CRUD 面

当 `TodoWriteTool` 之外，又出现 `TaskCreate/Get/Update/List/Stop/Output` 这一整组工具时，说明 Claude Code 已不再把任务当作 UI 上的一条 todo，而是可被编排、可被查询、可被外部消费的 runtime object。

### 5.2 团队能力有独立对象面

`TeamCreateTool` / `TeamDeleteTool` 表明“team”已经不是 prompt 中的隐喻，而是系统里的正式对象。

### 5.3 mailbox 是多 Agent 运行时的通信面

`SendMessageTool` 的存在说明 Claude Code 的协作不是“子代理只跑一次然后结束”，而是：

- 任务可继续
- agent 可恢复
- mailbox 可持续通信
- 用户可中途介入

## 6. 能力全集里的边界

这张图必须和三层边界一起阅读：

1. `COMMANDS()` / `getAllBaseTools()` 看到的是“源码存在面”
2. `getCommands()` / `getTools()` 看到的是“当前运行态可见面”
3. 用户最终能稳定使用什么，还要再经过 public build、feature gate、账号/组织策略和产品成熟度过滤

## 7. 使用与设计启发

### 7.1 对使用者

如果你想更高效地用 Claude Code，不要只学 prompt。  
你需要同时学：

- slash command
- task / send-message
- worktree / remote
- session / resume / rewind

### 7.2 对平台构建者

如果你想复刻 Claude Code 的能力面，不要只做：

- 一套工具列表
- 一个子代理按钮

你还需要补上：

- 任务对象
- mailbox 通信
- 团队对象
- 执行环境对象
- 能力可见性收口

## 8. 一句话总结

Claude Code 的能力全集不是“更多命令和工具”，而是一组被正式对象化的命令面、工具面、任务面、团队面与执行环境面。
