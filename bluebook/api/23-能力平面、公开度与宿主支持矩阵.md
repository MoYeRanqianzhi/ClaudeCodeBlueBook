这一章回答四个问题：

1. Claude Code 的能力面到底应分成哪几层。
2. 哪些是公共主路径，哪些是宿主主路径，哪些只是声明、gate 或适配器子集。
3. 宿主开发者最容易误判哪些“看起来支持”的入口。
4. 蓝皮书今后该怎样用统一矩阵继续扩 API 支持。

## 1. 先说结论

Claude Code 的 API 支持不能只按“有没有入口”判断；更稳的写法是把支持度压成矩阵，而不是按一句“支持/不支持”写。对本页来说，这张矩阵只消费 `102` 与 `api/README` 已经锁定的 rung result，不再重开 frontdoor classifier；本页只解释矩阵列怎么读、标签怎么写。

## 2. 先固定四列，再保留五种结果标签

本页矩阵只继续保留四列读法；它们是 `102` 已锁 rung 之后的报告维度，不是本页重新发放的前门顺序：

1. `existence`
2. `implementation visibility`
3. `current admission / consumer subset`
4. `product promise`

为了避免误写，本文统一用五个最终标签：

1. `公共主路径`
   - 面向用户或 CLI 主路径稳定出现。
2. `宿主主路径`
   - 面向 SDK / host / control plane 稳定出现。
3. `声明先行`
   - 有清晰入口、类型或文档语义，但当前提取树里实现未完全闭合。
4. `gate / internal`
   - 需要 feature gate、内部模块或编译期残缺分支才完整成立。
5. `产品受限`
   - 功能真实存在，但实际使用还受 allowlist、policy、org 或账号状态约束。

还要先记一句：这五个最终标签是写法结果，不是新的 classifier pipeline；它们只是在四列都已经写清后，对当前 support width 的压缩写法。若某一行同时带有 CLI / host / gate / product 的混合事实，也统一先把混合性写回四列，不在结果栏并排摆两个标签。

## 3. 能力矩阵

| 能力平面 | 代表入口 | existence | implementation visibility | current admission / consumer subset | product promise | 结果标签 |
| --- | --- | --- | --- | --- | --- | --- |
| 命令面 | `commands.ts`、`types/command.ts` | 正式 command/type surface 存在 | builtin 与 runtime assembly 可见 | CLI 主路径稳定，动态来源继续叠加 | 公共命令面可承诺，动态项逐条写 | `公共主路径` |
| 工具面 | `Tool.ts`、`tools.ts` | Tool ABI 与工具池存在 | 基础工具闭合，扩展工具分层可见 | 主路径稳定，但 deny / mode / MCP 继续裁切 | 基础工具面可承诺，扩展工具逐项写 | `公共主路径` |
| 权限面 | `types/permissions.ts`、permission setup | 权限类型与治理入口存在 | permission setup / classifier / sandbox 可见 | 真实行为持续受 mode / policy 条件化 | 可承诺的是治理控制面，不是固定放行结果 | `公共主路径` |
| Session API | `agentSdkTypes.ts` session 函数族 | session 函数族已声明存在 | 多个函数体仍未闭合 | 宿主主路径未完全成立 | 不能写成稳定完整 SDK 面 | `声明先行` |
| 控制协议 | `controlSchemas.ts`、`StructuredIO` | control protocol 存在 | schema 与宿主实现主链可见 | 宿主主路径稳定，但 adapter 宽度不一 | 可承诺宿主控制面，不可假定每个 adapter 同宽 | `宿主主路径` |
| 状态真相 | `SDKMessage`、`worker_status`、`external_metadata` | 状态对象与快照存在 | 回写与快照实现可见 | consumer subset 与 adapter 过滤持续存在 | 可承诺状态真相面，字段粒度需分 consumer 写 | `宿主主路径` |
| 任务对象面 | `TaskCreate/Get/Update/List/Stop/Output`、`task_*` | task 对象与事件存在 | CRUD 与生命周期事件主链可见 | UI 消费与事件粒度仍按 subset 收窄 | 任务 runtime 可承诺，消费粒度逐项写 | `宿主主路径` |
| 团队与 mailbox 面 | `TeamCreate/Delete`、`SendMessageTool`、mailbox | 团队协作对象存在 | mailbox / team / worker 协议大体可见 | swarms、team UI、部分 worker 细节仍受 gate 影响 | 协作面可承诺，但 team/product 细粒度不能一次写满 | `gate / internal` |
| 执行环境对象面 | `EnterWorktreeTool`、`ExitWorktreeTool`、`RemoteTriggerTool`、`Cron*` | 执行环境对象存在 | worktree / trigger / cron 实现可见 | 受环境、账号、策略与 gate 联合裁切 | 只能写成受限产品面 | `产品受限` |
| 远程面 | bridge / remote session / direct connect | 远程对象与桥接路径存在 | 主要适配器可见 | 远程策略、trusted device、组织 policy 决定宽度 | 不能写成无条件公共面 | `产品受限` |
| 生态面 | plugins / channels / MCPB / LSP | 生态对象存在 | schema 与入口较宽，成熟度不齐 | 渠道、策略与适配器差异持续裁切 | 只能逐项承认成熟度，不可一笔抹平 | `产品受限` |
| 多 Agent 面 | AgentTool / mailbox / coordinator | 多 Agent 协作对象存在 | 协调器、mailbox 与 worker 协议可见 | 部分 worker 细节仍受 gate 与内部模块影响 | 协作 runtime 可承诺，细节面逐项写 | `gate / internal` |
| Prompt / 知识面 | system prompt / frontmatter / attachments / memory | 注入与记忆对象存在 | prompt / attachment / memory 实现可见 | 某些注入策略与远程记忆仍受 gate 影响 | 可承诺知识注入面，远程与策略差异另写 | `公共主路径` |

对矩阵里已经被降格的行，本页也只再补最小 `downgrade note`：

- `Session API`
  - 入口与意图清楚，但宿主主路径仍未闭合；当前更稳地按 `声明先行 + unresolved-authority note` 读。
- `远程面`
  - 远程对象与桥接路径可见，但真实宽度仍被 trusted device、组织 policy 与远程策略持续裁切；当前更稳地按 `产品受限 + promise boundary note` 读。
- `生态面`
  - 生态对象已进入统一 runtime 语言，但渠道、策略与适配器差异仍让成熟度不齐；当前更稳地按 `产品受限 + consumer-subset note` 读。

这张矩阵里最容易被低估的是三类“对象升级”：

1. `task` 已经不是 todo UI，而是有 CRUD、输出、生命周期事件的运行时对象。
2. `team` 已经不是 prompt 话术，而是可创建、可删除、可持续通信的协作对象。
3. `worktree`、`remote trigger`、`cron` 已经把“执行环境”提升成了能力对象，而不是临时参数。

## 4. 如果真要写全支持面，还必须再补对象 truth

仅靠上表还不够。对“全部功能和 API”来说，至少还要补下面几类对象 truth；但这些对象 truth 只负责补证，不再回到前门抢答：

1. 命令 truth：
   - `src/commands/` 目录
   - `COMMANDS()` 主表
   - runtime command assembly
   - 若要继续分清“源码全集 / 当前会话可见集 / 当前调用可行集”，统一下钻到 `29`
2. 工具 truth：
   - `getAllBaseTools()`
   - deny rules 过滤后的 visible tools
   - async agent / in-process teammate / coordinator 子集
   - 若要继续分清 visible catalog 与 callable set 的关系，也统一下钻到 `29`
3. 任务 truth：
   - `TaskType` 联合
   - `getAllTasks()` 注册表
   - `task_started` / `task_progress` / `task_notification`
   - 若要继续看 object runtime 与事件宽度，回 `../architecture/README.md`
4. 服务 truth：
   - `src/services/` 约 `20` 个一级子域
   - `api`、`compact`、`mcp`、`plugins`、`SessionMemory`、`remoteManagedSettings` 等 subsystem planes
   - 若要继续判 host-facing truth claim-state / consumer subset，回 `./README.md`
5. 宿主 truth：
   - `SDKMessage` 时间线
   - `worker_status` / `external_metadata` 快照
   - session / transcript / recovery 资产面
   - 若要继续判 object-level writer / writeback seam，回 `../architecture/README.md`

只有把这些对象 truth 补齐，矩阵才不会停留在“理念正确、覆盖不够”。

## 5. 宿主开发者最容易误判的五类入口

### 5.1 把 `query()` 当成宿主全集

误判：

- 觉得嵌入 `query(prompt)` 就等于接入 Claude Code

更准确的判断：

- Claude Code 的宿主主路径还包括 control request、event stream、状态回写与恢复闭环。

### 5.2 把 session API 当成“当前都已实现”

误判：

- 看见 `getSessionMessages`、`listSessions`、`forkSession` 就默认认为 SDK 实现完整闭环

更准确的判断：

- 当前提取树里它们更多体现的是接口意图，不能直接写成“现成稳定 SDK 功能”。

### 5.3 把 `controlSchemas.ts` 当成每个 adapter 的真实支持面

误判：

- schema 全集 = bridge / direct-connect / remote-session 全部支持

更准确的判断：

- schema 全集只是协议全集，adapter 仍有明显宽窄差异。

### 5.4 把 channels / marketplace / MCPB / LSP 当成同一成熟度

误判：

- 既然都有入口，那就都能按同一强度对外承诺

更准确的判断：

- 这里更多体现的是“生态对象已经进入统一 runtime 语言”，而不是“产品成熟度完全一致”。

### 5.5 把 task / team / worktree / remote trigger 当成 UI convenience

误判：

- 觉得这些只是 CLI 上的高级玩法，不属于正式 API 支持面

更准确的判断：

- 一旦它们已经有独立 tool、独立状态对象、独立事件或独立执行环境语义，就应被写入正式能力矩阵，而不是藏在“技巧”章节里。

## 6. 写矩阵时只补什么

以后每新增一类 API 支持，本页也只补四列和一个最终标签；若 mixed-width 还想再解释，就回四列，不在标签栏重开组合写法：

1. `existence`
2. `implementation visibility`
3. `current admission / consumer subset`
4. `product promise`
5. 最终结果标签

## 8. 一句话总结

Claude Code 的 API 支持必须按“能力平面 + 公开度/成熟度”双层矩阵书写，否则就会把 schema、实现、适配器子集和产品承诺持续写混。
