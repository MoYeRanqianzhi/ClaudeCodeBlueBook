# 能力平面、公开度与宿主支持矩阵

这一章回答四个问题：

1. Claude Code 的能力面到底应分成哪几层。
2. 哪些是公共主路径，哪些是宿主主路径，哪些只是声明、gate 或适配器子集。
3. 宿主开发者最容易误判哪些“看起来支持”的入口。
4. 蓝皮书今后该怎样用统一矩阵继续扩 API 支持。

## 1. 先说结论

Claude Code 的 API 支持不能只按“有没有入口”判断，而必须同时看五件事：

1. 是否有正式类型、schema、tool、command 或 protocol 入口。
2. 当前公开源码里是否有可见实现。
3. 当前实现是否进入主路径，而不是只停留在 adapter 子集。
4. 当前能力是否受 feature gate、policy、GrowthBook、账号或组织限制。
5. 当前产品是否真的承诺了稳定可用，而不是仅仅“源码里能看到”。

所以后续所有“支持情况”都应按矩阵写，而不是按一句“支持/不支持”写。

## 2. 五种判断标签

为了避免误写，本文统一用五种标签：

1. `公共主路径`
   - 面向用户或 CLI 主路径稳定出现。
2. `宿主主路径`
   - 面向 SDK / host / control plane 稳定出现。
3. `声明先行`
   - 有清晰入口、类型或文档语义，但当前提取树里实现未完全闭合。
4. `gate / internal`
   - 需要 feature gate、内部模块或编译期残缺分支才完整成立。
5. `产品受限`
   - 功能真实存在，但实际使用还受 allowlist、policy、org 或账号状态约束。

## 3. 能力矩阵

| 能力平面 | 代表入口 | 当前主判断 | 主要边界 |
| --- | --- | --- | --- |
| 命令面 | `commands.ts`、`types/command.ts` | `公共主路径` | 仍有动态 skills / plugins / workflow 叠加 |
| 工具面 | `Tool.ts`、`tools.ts` | `公共主路径` | MCP tools、internal tools、deny rules 会继续裁切 |
| 权限面 | `types/permissions.ts`、permission setup | `公共主路径` | mode、classifier、policy、sandbox 决定真实行为 |
| Session API | `agentSdkTypes.ts` session 函数族 | `声明先行` | 多个函数体当前仍是 `not implemented` |
| 控制协议 | `controlSchemas.ts`、`StructuredIO` | `宿主主路径` | 不等于每个 adapter 都完整支持 |
| 状态真相 | `SDKMessage`、`worker_status`、`external_metadata` | `宿主主路径` | consumer subset 与 adapter 过滤必须单独写 |
| 远程面 | bridge / remote session / direct connect | `产品受限` | 远程策略、组织 policy、trusted device、adapter 宽度 |
| 生态面 | plugins / channels / MCPB / LSP | `产品受限` | schema 支持不等于产品成熟度 |
| 多 Agent 面 | AgentTool / mailbox / coordinator | `公共主路径 + gate 边界` | worker 某些实现细节仍受 gate 与内部模块影响 |
| Prompt / 知识面 | system prompt / frontmatter / attachments / memory | `公共主路径` | 某些注入策略与远程记忆仍可能受 gate 影响 |

## 4. 宿主开发者最容易误判的四类入口

### 4.1 把 `query()` 当成宿主全集

误判：

- 觉得嵌入 `query(prompt)` 就等于接入 Claude Code

更准确的判断：

- Claude Code 的宿主主路径还包括 control request、event stream、状态回写与恢复闭环。

### 4.2 把 session API 当成“当前都已实现”

误判：

- 看见 `getSessionMessages`、`listSessions`、`forkSession` 就默认认为 SDK 实现完整闭环

更准确的判断：

- 当前提取树里它们更多体现的是接口意图，不能直接写成“现成稳定 SDK 功能”。

### 4.3 把 `controlSchemas.ts` 当成每个 adapter 的真实支持面

误判：

- schema 全集 = bridge / direct-connect / remote-session 全部支持

更准确的判断：

- schema 全集只是协议全集，adapter 仍有明显宽窄差异。

### 4.4 把 channels / marketplace / MCPB / LSP 当成同一成熟度

误判：

- 既然都有入口，那就都能按同一强度对外承诺

更准确的判断：

- 这里更多体现的是“生态对象已经进入统一 runtime 语言”，而不是“产品成熟度完全一致”。

## 5. 最小判断算法

以后每新增一类 API 支持，都先按下面顺序判断：

1. 入口：
   - 它的 command / tool / schema / control subtype 在哪里。
2. 实现：
   - 当前提取树里是否能看到主路径实现。
3. 适配器：
   - 它是协议全集，还是某个 adapter 子集。
4. 产品：
   - 它是否会被 gate、policy、账号或组织限制。
5. 写法：
   - 最终是写成 `公共主路径`、`宿主主路径`、`声明先行`、`gate/internal` 还是 `产品受限`。

Claude Code 的 API 支持必须按“能力平面 + 公开度/成熟度”双层矩阵书写，否则就会把 schema、实现、适配器子集和产品承诺持续写混。
