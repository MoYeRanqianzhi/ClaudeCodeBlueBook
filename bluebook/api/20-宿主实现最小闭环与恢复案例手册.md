# 宿主实现最小闭环与恢复案例手册

这一章回答五个问题：

1. 如果你要做 Claude Code host，最小必须吃哪几类对象。
2. 一个可工作的宿主闭环最少包含哪些 request / response / state。
3. `initialize`、`can_use_tool`、`resume` 三个路径该怎么理解。
4. `SDKMessage`、snapshot、recovery 三面如何一起消费。
5. 常见误实现是什么。

## 1. 先说结论

一个最小可工作的 Claude Code host，至少要正确消费四条通道：

1. control：
   - `control_request`
   - `control_response`
   - `control_cancel_request`
2. event stream：
   - `assistant`
   - `stream_event`
   - `result`
   - `system:init`
   - `task_*`
   - `session_state_changed`
3. snapshot：
   - `worker_status`
   - `requires_action_details`
   - `external_metadata`
4. recovery：
   - transcript / ingress / internal-events

如果缺其中任何一条，宿主都只能拿到 Claude Code 的一个投影，而不是完整 runtime。

## 2. 案例一：`initialize`

最小闭环应理解成：

1. 宿主发 `control_request(initialize)`
2. CLI 回 `control_response(success)`
3. 再发 `system:init`

其中：

- `control_response` 负责 ack + capability reply
- `system:init` 负责 runtime 实际装配态

所以如果宿主只看 `control_response` 不看 `system:init`，会错过：

- `tools`
- `mcp_servers`
- `model`
- `permissionMode`
- `slash_commands`
- `skills`
- `plugins`

## 3. 案例二：`can_use_tool`

这是最核心的运行时仲裁闭环。

最小闭环是：

1. CLI 发 `control_request(can_use_tool)`
2. 宿主根据 UI / policy / host state 回 `control_response`
3. 若另一条仲裁路径先赢，CLI 或宿主还可能收到 `control_cancel_request`
4. 后续再观察：
   - `session_state_changed`
   - `worker_status`
   - `external_metadata.pending_action`
   - `tool_progress` / `task_*`

这说明 `can_use_tool` 的实现不能只做一个“同步弹窗返回布尔值”的函数。
它必须允许：

- 异步
- cancel
- 后续状态变化

## 4. 案例三：`resume`

`resume` 不只是重新拿消息，而是：

1. 选择来源：
   - sessionId
   - 最近会话
   - JSONL
2. hydrate transcript：
   - 本地 JSONL
   - v1 ingress
   - v2 internal-events
3. 再恢复：
   - file history
   - attribution
   - context collapse
   - todos
   - mode / worktree / agent

因此宿主如果要做强恢复，不应只缓存最后几条 message，而应理解 Claude Code 的恢复本体是：

- `loadConversationForResume(...)`
- `restoreSessionStateFromLog(...)`

## 5. 宿主的最小消费清单

如果要做一个“够用”的宿主，最低建议吃这些对象：

### 5.1 控制面

- `initialize`
- `can_use_tool`
- `interrupt`
- `get_context_usage`
- `get_settings`

### 5.2 时间线面

- `assistant`
- `stream_event`
- `result`
- `task_started`
- `task_progress`
- `task_notification`
- `system:init`
- `system:session_state_changed`

### 5.3 快照面

- `worker_status`
- `requires_action_details`
- `external_metadata.permission_mode`
- `external_metadata.model`
- `external_metadata.pending_action`

## 6. 最容易犯的四个错误

### 6.1 只消费 `assistant` / `result`

这样会把 Claude Code 误当成答案流，而看不见：

- 执行脉搏
- 任务进度
- 权限阻塞
- 当前运行态

### 6.2 用 `system:status` 代替 `worker_status`

`system:status` 很重要，但它不是最稳定的 authoritative running/idle 信号。
更该依赖的是：

- `worker_status`
- `session_state_changed`

### 6.3 把 `can_use_tool` 做成无 cancel 的同步 RPC

这样会在真正竞速场景里出现：

- 过期弹窗
- 双重批准
- 已取消请求仍在等待

### 6.4 忽略 recovery 面

如果宿主完全不关心 transcript / resume / internal-events，那么它做出来的只能是：

- 在线运行时壳

而不是：

- 可恢复 host

## 7. 对 API 写作的启发

后续写宿主接入时，必须总是把下面三层并列写：

1. request/response
2. follow-on event stream
3. snapshot / recovery

只写第一层，就会把 Claude Code 错写成普通 RPC SDK。

## 8. 一句话总结

Claude Code 宿主接入的最小正确姿势，不是“能发消息并拿到结果”，而是“同时吃下 control、event stream、snapshot 与 recovery 四条闭环通道”。
