## 1. 这本蓝皮书要回答什么

这套文档重点回答四个问题：

1. Claude Code 的执行主链是什么？
2. 它如何把一个最小 Agent Loop 包装成可长期工作的工程系统？
3. 用户应该如何正确使用它，而不是把它当成普通聊天框？
4. 它的设计哲学是什么，为什么这种哲学会让开发体验显著变强？

## 2. 当前观察到的源码骨架

从 `claude-code-source-code` 可见：

- CLI 入口位于 `src/entrypoints/cli.tsx`
- 完整引导位于 `src/main.tsx`
- 初始化位于 `src/entrypoints/init.ts`
- 非交互 Agent 生命周期封装在 `src/QueryEngine.ts`
- 真正的主循环在 `src/query.ts`
- 工具接口与默认行为在 `src/Tool.ts`
- 工具池装配与过滤在 `src/tools.ts`
- 命令总表在 `src/commands.ts`
- 技能、插件、远程、桥接、协调器分别位于 `src/skills/`、`src/plugins/`、`src/remote/`、`src/bridge/`、`src/coordinator/`

这说明 Claude Code 不是“一个 CLI 文件 + 一个 API 调用”。
它更像一个终端内的操作系统外壳：入口、权限、工具、状态、压缩、恢复、远程会话、子代理、技能系统都是一级公民。

## 3. 源码分层心智模型

可以先用下面这张图理解全局：

```text
用户输入
  -> CLI 快路径 / 完整主程序
  -> 初始化环境、配置、遥测、策略
  -> 进入 REPL 或 QueryEngine
  -> query() 主循环
  -> 模型流式输出 tool_use / text
  -> 工具执行、权限判定、结果回填
  -> 压缩、持久化、恢复、并行子代理
  -> 再次进入下一轮
```

最重要的结论是：

- “问模型”只是中间一段
- 让 Claude Code 强大的，是主循环外层那一整圈 harness

## 5. 方法论

本阶段采用“三条并行分析链”：

1. 启动与 CLI 链：看系统如何快速进入正确模式
2. 主循环与工具链：看系统如何稳定执行复杂任务
3. 扩展与远程链：看系统如何突破单进程、单工具、单上下文限制

## 6. 已确认的研究边界

根据 `package.json`、`QUICKSTART.md` 与源码中的 `feature()` 使用方式：

- 该源码是面向研究的还原版本，不保证可直接完整构建
- Bun 编译期能力对源码形态有重要影响
- 一部分 Anthropic 内部模块在公开包中被编译期裁剪

所以蓝皮书里的原则是：

- 对可见代码，做确定性分析
- 对 feature-gated 的缺口，只做边界清晰的保守推断
