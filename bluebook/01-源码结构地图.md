# 源码结构地图

这页不该被读成“目录树展开图”。

更稳的读法是：

1. 先看谁在宣布真相
2. 再看谁只是消费真相
3. 再看哪些恢复资产、旧写入者与投影层可能在时间上撒谎
4. 最后才看目录体感、文件大小与模块数量

如果把源码地图前门再压成最短公式，也只剩三句：

1. Prompt 线
   - 地图先服务 `same-world compiler`，先暴露哪些对象在维持 `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability`。
2. 治理线
   - 地图先服务 `decision window`，先暴露哪些目录在外化 truth chain、哪些目录只在消费投影。
3. 源码质量线
   - `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`
   - `one writable present` 与 later maintainer reject path 也属于 architecture doctrine，不是尾部补充说明。

如果你在别页看到相近但不完全同名的梯子，例如：

- `contract truth -> registry truth -> orchestration shell -> current-truth surface -> consumer subset -> danger surface`

也不要把它读成第二条 canonical ladder。更稳的读法是：

- 前者才是入口层 canonical ladder；
- 后者只是把 `registry / hotspot kernel / danger surface` 放到 orchestration 热点里展开的投影变体。
- `contract / registry` 仍然是前两级，不因 orchestration 视角而换名；
- `orchestration shell` 只是热点读法，不是新 rung；
- `danger surface` 只是 `hotspot kernel + mirror gap discipline` 的 atlas 投影，后文若借用投影词，也应顺手标出自己投影的是哪一级。

如果一张地图还没压到这三句，它就还停在目录清单层。

更稳的桥接句也不该再是“源码按功能这样组织”，而该是：

- 这些目录之所以这样分布，不是在先做功能分层，而是在先把 `谁定义现在、谁只消费现在、谁只能帮助恢复现在` 这三种角色分开落到源码里；目录只是这套主权分配的投影。

如果继续把源码先进性的前门再压成 later maintainer 更容易复用的三条桥接句，也只该剩：

1. `remote / session / transport` 在源码里首先是主权域，不是功能域。
2. `event truth / current truth / display truth` 必须分层，宿主与 UI 都不能靠 replay transcript 猜现在。
3. command plane 可以变薄，但 tool plane 不能跟着一起失去装配主权；可见性与执行主权必须继续分开设计。

如果地图还看不出这三条，它就仍在带读者看目录树，而不是在暴露设计思路。

如果你还没先经过 `09 / 05 / 15 / 41` 这组高阶前门，也不要急着把这页读成“运行时库存图”；这里回答的是对象、边界与危险改动面，不重做模仿对象校正、失稳前追问或第一性原理收束。

## 1. 从入口看全局

Claude Code 的入口不是一个简单的 commander 程序，而是一个“先快路径分流，再按模式加载主程序”的引导器。

`src/entrypoints/cli.tsx` 做了三件事:

1. 提前设置必须在模块加载前生效的环境变量，例如 `COREPACK_ENABLE_AUTO_PIN` 与 CCR 场景下的 `NODE_OPTIONS`。
2. 用一串 fast path 处理特殊模式，避免加载整个 CLI:
   - `--version`
   - `--dump-system-prompt`
   - `--claude-in-chrome-mcp`
   - `--computer-use-mcp`
   - `daemon`
   - `remote-control`
   - `ps/logs/attach/kill`
   - `environment-runner`
   - `self-hosted-runner`
   - `--worktree --tmux`
3. 若没有命中特殊分支，再动态 import `main.js` 进入完整 CLI。

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:28-33`
- `claude-code-source-code/src/entrypoints/cli.tsx:36-48`
- `claude-code-source-code/src/entrypoints/cli.tsx:95-185`
- `claude-code-source-code/src/entrypoints/cli.tsx:224-299`

这说明它把“首屏启动时间”和“按模式剪裁加载成本”看得非常重。

更短的判断句也只该剩一句：

- 合法复杂度中心是资产，不是债务；入口主权面、orchestration layer 与安全初始化边界，都是系统为时序复杂度保留的正式席位，而不是等待目录洁癖抹平的偶然大文件

但对源码地图来说，更重要的不是这里“功能很多”，而是：

- `cli.tsx` 是入口主权面
- fast path 在控制哪些模式配更早进入运行时
- main program 的真正权威装配要等更后面的安全与模式判断成立后才发生

## 2. main.tsx 才是真正的总控室

`src/main.tsx` 不只是 UI 入口，它是合法复杂度中心里的主控热点。

它负责:

- 早期重写特殊 argv，例如 `cc://` 深链、`assistant`、`ssh`
- 判断本次运行是 interactive 还是 headless
- 设置 entrypoint/clientType
- 提前加载 settings
- 初始化 warning handler 与 trust 相关的流程
- 并行准备 tools、commands、agent definitions、MCP config
- 在 trust 建立后启动 LSP、plugins、telemetry、hooks、remote/session 逻辑
- 最终进入 REPL、remote REPL 或 print 模式

证据:

- `claude-code-source-code/src/main.tsx:517-540`
- `claude-code-source-code/src/main.tsx:585-855`
- `claude-code-source-code/src/main.tsx:1919-2029`
- `claude-code-source-code/src/main.tsx:2239-2607`
- `claude-code-source-code/src/main.tsx:3337-3502`

## 3. 初始化不是“开机动作”，而是安全边界的一部分

`src/entrypoints/init.ts` 的关键点不是“做初始化”，而是严格区分:

- trust 建立前，只能做哪些安全动作
- trust 建立后，哪些能力才允许启动

它在早期只应用 safe env vars、CA cert、proxy/mTLS、graceful shutdown、remote settings loading promise 等；而 telemetry 的完整初始化被显式延后到 trust 建立之后。

证据:

- `claude-code-source-code/src/entrypoints/init.ts:62-88`
- `claude-code-source-code/src/entrypoints/init.ts:120-183`
- `claude-code-source-code/src/entrypoints/init.ts:240-303`

这和很多 CLI 的区别非常大: Claude Code 把“是否可在当前目录执行潜在危险动作”当成运行时一等公民。

## 4. 运行时主链路

从用户输入到工具执行，大致链路如下:

```text
CLI / REPL
  -> processUserInput
  -> QueryEngine.submitMessage
  -> query()
  -> callModel(streaming)
  -> tool_use / tool_result
  -> transcript + app state + follow-up turns
```

这条链路在地图上的价值，不是告诉你“流程经过了几个函数”，而是先区分四件事：

1. 谁在决定这次 turn 是否配进入模型
2. 谁把当前 turn 写成第一份事件线真相
3. 谁在决定这次继续是否还值得被定价
4. 哪些恢复与回写路径最容易把过去重新写成第二真相

更稳的源码地图判断，最终都要回到同一条顺序：

1. `contract`
2. `registry`
3. `current-truth surface`
4. `consumer subset`
5. `hotspot kernel`
6. `mirror gap discipline`

### 4.1 输入预处理先决定这次 turn 是否配进入模型

`processUserInput()` 先判断这次输入到底是什么:

- 普通 prompt
- 本地 slash command
- prompt 型 skill / slash command
- 带附件、图片、hook 附加上下文的输入

如果 slash command 本地就能解决，它可以直接 `shouldQuery = false`，根本不走模型。

因此这一步在地图上更像 admission gate，而不是普通格式化辅助层。

证据:

- `claude-code-source-code/src/utils/processUserInput/processUserInput.ts:85-176`
- `claude-code-source-code/src/utils/processUserInput/processTextPrompt.ts:19-99`

### 4.2 QueryEngine 把 turn 写成第一份会话真相

`QueryEngine.submitMessage()` 负责:

- 拼 system prompt 与 user/system context
- 包装 `canUseTool`，记录 permission denials
- 调 `processUserInput()` 处理 slash commands / attachments
- 先写 transcript，再进入 query loop
- 构造 system init message，把 tools / skills / plugins / agents 一次性交给 SDK 消费端

证据:

- `claude-code-source-code/src/QueryEngine.ts:180-207`
- `claude-code-source-code/src/QueryEngine.ts:209-333`
- `claude-code-source-code/src/QueryEngine.ts:430-463`
- `claude-code-source-code/src/QueryEngine.ts:529-551`
- `claude-code-source-code/src/QueryEngine.ts:675-686`

这意味着 `QueryEngine` 在地图上不只是“会话外壳”，而是当前 turn 的第一份 `registry truth` 写入点：哪些上下文被编进请求、哪些工具被宣布为当前可见集、哪些拒绝被正式记账，都先在这里落档。

但它还不是宿主当前真相本身。更稳的分法是：

- `QueryEngine + transcript`
  负责事件线真相
- `state writeback / worker_status / external_metadata`
  负责 host-facing current truth

### 4.3 query.ts 负责 continuation pricing 与恢复真相

`src/query.ts` 体现的是 Claude Code 的真正“魔法来源”之一: 极简 loop 外加大量生产级恢复机制。

它在每轮 query 前后会做:

- skill discovery prefetch
- tool result budget 控制
- snip / microcompact / context collapse / autocompact
- 流式调用模型
- 流式消费 `tool_use`
- streaming tool execution
- fallback model retry
- interrupt / withheld error / reactive compact 恢复

证据:

- `claude-code-source-code/src/query.ts:323-337`
- `claude-code-source-code/src/query.ts:365-467`
- `claude-code-source-code/src/query.ts:551-568`
- `claude-code-source-code/src/query.ts:659-865`
- `claude-code-source-code/src/query.ts:893-950`
- `claude-code-source-code/src/query.ts:1011-1085`

所以 `query.ts` 在地图上的关键身份不是“循环更复杂”，而是：

- continuation eligibility 在这里被定价
- compact / snip / collapse 在这里决定哪些历史还配留下
- interrupt / retry / reactive compact 在这里决定系统怎样避免把失败写成假连续性

## 5. tools / commands / skills 不是三张图鉴，而是能力发布协议

Claude Code 表面上有三类东西:

- tool
- slash command
- skill

但源码里它们被做成了可互相装配的统一能力层。

### 5.1 Tool 是能力原语

`Tool.ts` 定义了统一工具接口:

- `call`
- `description` / `prompt`
- `inputSchema`
- `checkPermissions`
- `renderToolUseMessage`
- `renderToolResultMessage`
- `toAutoClassifierInput`

并通过 `buildTool()` 提供 fail-closed 默认值，例如:

- `isConcurrencySafe = false`
- `isReadOnly = false`
- `isDestructive = false`
- `checkPermissions = allow but defer to general permission system`

因此 `Tool.ts` 在地图上更像权威动作 ABI：它决定一个动作要以什么 schema、什么失败语义和什么权限合同进入当前世界。

证据:

- `claude-code-source-code/src/Tool.ts:362-695`
- `claude-code-source-code/src/Tool.ts:743-792`

### 5.2 tools.ts 负责运行时装配与过滤

`src/tools.ts` 不是一个静态列表，而是按当前运行环境拼装工具池:

- 基础工具: Bash / Read / Edit / Write / Skill / Agent / WebFetch / Todo 等
- feature-gated 工具: Workflow / Sleep / WebBrowser / TerminalCapture / Monitor 等
- 再根据 permission rules、mode、MCP tools 继续过滤

因此 `tools.ts` 的角色不是“列工具大全”，而是把正式存在的动作原语继续裁成当前可见子集。

证据:

- `claude-code-source-code/src/tools.ts:179-249`
- `claude-code-source-code/src/tools.ts:267-349`

### 5.3 commands.ts 是命令汇流层

`getCommands()` 把多个来源拼成统一命令列表:

- bundled skills
- builtin plugin skills
- skill dir commands
- workflows
- plugin commands
- plugin skills
- 内置 slash commands

证据:

- `claude-code-source-code/src/commands.ts:445-469`
- `claude-code-source-code/src/commands.ts:476-517`

所以 `commands.ts` 在地图上代表的是显式控制面发布协议，而不只是 slash 列表。

## 6. 技能系统不是“命令别名”，而是轻量 prompt 包管理器

## 7. MCP、plugin、remote 都是扩张边界对象，不是外围插件

### 7.1 MCP

MCP 支持多种 transport:

- `stdio`
- `sse`
- `http`
- `ws`
- `sdk`
- `claudeai-proxy`

配置来源也有多层:

- `project`
- `user`
- `local`
- `enterprise`
- `claudeai`
- `dynamic`

并且会做:

- env var expansion
- plugin server dedup
- claude.ai connector dedup
- policy filtering
- auth cache / reconnect / cleanup

证据:

- `claude-code-source-code/src/services/mcp/types.ts:9-27`
- `claude-code-source-code/src/services/mcp/types.ts:124-177`
- `claude-code-source-code/src/services/mcp/config.ts:223-310`
- `claude-code-source-code/src/services/mcp/config.ts:888-960`
- `claude-code-source-code/src/services/mcp/config.ts:1211-1290`
- `claude-code-source-code/src/services/mcp/client.ts:1743-1833`

这说明 MCP 在地图上不只是“扩展协议”，还是扩张如何被声明、过滤、认证、重连与去重的定价对象。

更稳的读法不是继续背 transport 和 config source，而是连续追问：

1. 谁在宣布外部 server 资格
2. 谁在做 merge / dedup / policy / auth 裁切
3. 哪一层只是 adapter，哪一层才是 authority
4. stale auth、reconnect 与 projection 最容易从哪里开始撒谎

### 7.2 Remote / Bridge

源码里至少存在两条远程相关路径:

- remote session: 本地 REPL 连接 CCR session
- remote control bridge: 把本机作为 bridge 环境暴露给远端客户端

`RemoteSessionManager` 负责 remote session 的 WebSocket 订阅、permission request、HTTP message send。

证据:

- `claude-code-source-code/src/remote/RemoteSessionManager.ts:87-141`
- `claude-code-source-code/src/remote/RemoteSessionManager.ts:186-250`

`bridgeEnabled.ts` 和 `bridgeMain.ts` 则体现了 Remote Control 的另一套体系:

- 依赖 claude.ai OAuth 与 org entitlement
- 需要版本检查、policy limit、GrowthBook gate
- 可以长期轮询 work、spawn child session、做 heartbeat 与 reconnect

证据:

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-55`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:57-87`
- `claude-code-source-code/src/bridge/bridgeMain.ts:85-99`
- `claude-code-source-code/src/bridge/bridgeMain.ts:141-205`

这两条路径不应混成一个“remote 功能”。在地图上更稳的读法是：

- remote session 更接近既有会话真相的远端消费路径
- remote control bridge 更接近执行半径被继续放大的另一套 authority domain

## 8. 多 Agent 是委托资格与隔离对象协议

`AgentTool`、`Task`、`LocalAgentTask`、`RemoteAgentTask` 共同构成了 Claude Code 的 delegation runtime。

关键结构:

- `AgentTool` 可以启动同步子 agent、后台 agent、teammate、worktree isolation、remote isolation
- `Task.ts` 把 `local_agent`、`remote_agent`、`local_bash`、`dream` 等统一成任务类型
- `coordinatorMode.ts` 明确把“coordinator 负责调度、worker 负责执行”写进系统 prompt

- 谁有资格签发 child execution
- 哪个对象才是任务真相
- coordinator 哪些只是调度投影
- local / remote / worktree 各自在防哪类 stale write 与恢复谎言

## 9. 一句话结构总结

Claude Code 的源码地图真正该留下的不是目录名单，而是五句拒收检查：

1. 我改的是 authority，还是 projection / subset。
2. 我碰到的是 turn truth，还是恢复后才出现的第二真相。
3. 我改动的是动作 ABI，还是当前可见集的裁切逻辑。
4. 我是在扩大执行半径，还是只在 transport shell 上做投影。
5. 如果我不同意当前实现，第一条该沿哪条入口链拒收它。
