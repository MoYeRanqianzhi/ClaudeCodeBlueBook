# 公开能力与隐藏能力

这一章专门回答一个容易被误读的问题: “源码里出现了，就等于外部用户一定能用吗？”

答案是否定的。

## 0. 先带着三条判断读公开度

在继续看 public runtime / shipped but gated / internal 之前，先记住三件事：

1. Prompt 线提醒你：
   - 代码里出现了某个能力，不等于它已经通过 `same-world test` 合法进入模型当前可见世界。
2. 治理线提醒你：
   - 能力“存在”、能力“当前可见”、能力“当前被允许”是三层不同的准入判断；`Context Usage`、mode 条和 dashboard 投影都不能直接替代 `decision window`。
3. 源码质量线提醒你：
   - 对公开镜像的公开度判断必须服从 `truth ladder`，不能把声明、注册、宿主子集和缺口脑补混成一层。

所以这章真正想解决的不是“有没有这个功能”，而是：

- 这份公开源码到底允许你对外稳当地说到哪一层
- 更具体地说，要持续把“能力存在”“当前可见”“当前被允许”“对外承诺”拆开

如果把公开度前门继续压成最短公式，也只剩三条：

1. `existence -> current visibility -> current admission -> product promise`
2. `externalized truth chain -> decision window -> current admission -> product promise`
3. `truth ladder` 负责约束你到底在拿哪一层证据说这句话

如果一个公开度判断还没压到这三条，它就还停在“源码里有 / 产品好像有”的结果词层。

这也意味着，`04` 不替 `09 / 05 / 15 / 41` 重做高阶前门：

1. `09` 先定义三张控制面。
2. `05` 先校正你到底在模仿功能表、治理表，还是控制面。
3. `15` 先暴露第一条反证信号。
4. `41` 再把“公开能力判断”为何成立压回第一性原理。

更稳的最小判断顺序应固定为：

1. `existence`
2. `current visibility`
3. `current admission`
4. `product promise`

如果继续往治理线压，这四层也不能跳过：

- `externalized truth chain`
- `decision window`
- `current admission`
- `product promise`

一旦又把 mode 条、usage dashboard 或 build-time 痕迹直接写成外部承诺，就说明入口层已经把治理和公开度重新混成一层。

更硬一点说，公开度的 first reject signal 也只该剩三条：

1. UI 可见开始冒充 current admission
2. build-time 痕迹开始冒充 product promise
3. `Context Usage` 或 mode 条开始冒充治理真相

## 1. 先分清判断顺序，再看三类结果桶

下面的三类能力不是前门判断顺序，而是四层问题被压缩后的结果桶。

更稳的顺序仍应先问：

1. `existence`
2. `current visibility`
3. `current admission`
4. `product promise`

### 1.1 public runtime

代码存在，且公开包大概率会实际包含主路径实现。

典型例子:

- CLI / REPL / `--print`
- File/Bash/Edit/Read/Write 等基础工具
- skills / commands / tools 主装配逻辑
- MCP 基础框架
- remote session 主路径

### 1.2 shipped but gated

代码存在于这份源码中，但要不要出现、何时出现，取决于:

- `feature('...')`
- GrowthBook gate
- env var
- policy / entitlement

典型例子:

- Remote Control entitlement
- 部分 bundled skills
- ToolSearch deferred tool 行为
- 部分 agent / coordinator / KAIROS 相关逻辑

### 1.3 internal / DCE'd

在 README 和代码里能看到引用，但 npm 发布物中并不包含完整实现，或者被 build-time dead code elimination 裁掉。

这类能力只能说“在 Anthropic 内部形态里存在过接口或痕迹”，不能说“外部可用”。

## 2. 如何识别 build-time gate

最强烈的信号是 `feature('...')`。

例如:

- `cli.tsx` 用它裁掉 daemon / bridge / environment-runner / self-hosted-runner 等路径
- `skills/bundled/index.ts` 用它裁掉 dream / loop / scheduleRemoteAgents / claudeApi 等技能
- `bridgeEnabled.ts` 用它裁掉 bridge 相关代码路径

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:53-178`
- `claude-code-source-code/src/skills/bundled/index.ts:35-78`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:28-55`

如果某个模块只在 `feature('X') ? require(...) : null` 下被引用，就不能默认它对外可用。

## 3. 明显属于公开主路径的能力

基于当前源码，最稳妥可视为公开主路径的有:

- 交互式与 headless 双运行面
- transcript / resume / session 体系
- 工具系统与权限系统
- skills 与 slash commands 装配
- MCP config / connection / tool wrapping 基础设施
- subagent 基础能力

证据:

- `claude-code-source-code/src/QueryEngine.ts:209-333`
- `claude-code-source-code/src/tools.ts:193-249`
- `claude-code-source-code/src/commands.ts:445-517`
- `claude-code-source-code/src/services/mcp/client.ts:1743-1833`

## 4. 明显属于 shipped but gated 的能力

### 4.1 Remote Control

不是所有用户都能直接用。它至少受以下条件约束:

- build 要带 `BRIDGE_MODE`
- 用户必须是 claude.ai subscriber
- token 要有 profile scope
- organization 要满足 entitlement gate
- policy 要允许 remote control

证据:

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-87`
- `claude-code-source-code/src/entrypoints/cli.tsx:108-161`

### 4.2 部分技能与高级工具

比如:

- `dream`
- `loop`
- `scheduleRemoteAgents`
- `claudeApi`
- `WorkflowTool`
- `MonitorTool`

都需要对应 gate。

证据:

- `claude-code-source-code/src/skills/bundled/index.ts:35-78`
- `claude-code-source-code/src/tools.ts:120-136`

### 4.3 多 Agent 高阶形态

`AgentTool` 本身在公开运行时中很核心，但 coordinator workers、teammate swarms、remote isolation 等更高阶形态不应默认对所有外部用户开放。

证据:

- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:90-101`
- `claude-code-source-code/src/tools/AgentTool/builtInAgents.ts:35-69`
- `claude-code-source-code/src/coordinator/coordinatorMode.ts:36-41`

## 5. 明显属于 internal / incomplete 的能力

从仓库 README 可直接确认:

- 公开 npm 包缺失大量 `feature()` 门控模块
- 这些模块仅存在于内部 monorepo 或被 DCE 后无法恢复

代表性方向:

- daemon worker registry
- proactive system
- assistant/KAIROS 完整实现
- internal agents platform
- workflows 某些任务实现
- transcript classifier 相关内部服务

证据:

- `claude-code-source-code/README_CN.md:42-145`

这里的正确姿势不是“宣布产品已经有这个功能”，而是“确认产品架构预留了这个方向，或内部构建里存在对应能力”。

## 6. plugin 系统当前状态的判断

plugin 基础设施很明确已经在架构里留好位置:

- 可承载 skills、hooks、MCP servers
- 有 `/plugin` UI 语义
- 有 builtin plugin registry

但当前 `src/plugins/bundled/index.ts` 里没有真正注册任何 built-in plugin，只是 scaffolding。

证据:

- `claude-code-source-code/src/plugins/builtinPlugins.ts:1-14`
- `claude-code-source-code/src/plugins/builtinPlugins.ts:57-121`
- `claude-code-source-code/src/plugins/bundled/index.ts:17-23`

所以对 plugin 系统更准确的描述是:

- 架构已经成熟
- 生态层还在扩张
- 某些能力更像“已经接上线的基础设施”，而不是“已经形成丰富市场”

## 7. Buddy、Voice、KAIROS 应该怎么表述

这三类很容易被误写成“已经发布”。

更准确的说法应该是:

- Buddy: 有完整实现痕迹，但属于 gated/未普遍开放能力
- Voice: 源码有明确入口与门控，但不能直接推断公开可用
- KAIROS: 是强烈的产品方向与内部能力线索，不宜等同当前公开默认能力

证据:

- `claude-code-source-code/src/commands.ts:70-80`
- `claude-code-source-code/src/buddy/companion.ts:15-133`
- `claude-code-source-code/docs/zh/05-未来路线图.md:31-83`

## 8. 如何稳地使用这章

后续阅读和写作都必须坚持三条:

1. 代码存在，不等于用户可用。
2. `feature()` gate 与 GrowthBook gate 要分开写。
3. 只要 README 明示“缺失模块”，就不能把推测写成事实。
