# 治理迁移工单：从规则堆叠到统一定价秩序的改写顺序与停止条件

这一章把 `guides/31` 和 `casebooks/11` 继续推进到迁移执行层。

它主要回答五个问题：

1. 怎样把旧式规则堆叠、假统一预算器系统迁到 Claude Code 式治理顺序系统。
2. 为什么这类迁移不该从“再加一层规则”开始，而应从资产清单与顺序图开始。
3. 怎样安排输入边界、稳定字节、决策增益、自动化回收与 continuation 的迁移顺序。
4. 哪些停止条件和回退条件必须先于 rollout 写出来。
5. 怎样用苏格拉底式追问避免把治理升级写成治理膨胀。

## 1. 第一性原理

这类迁移真正要解决的，不是：

- 规则数量不够

而是：

- 系统在错误顺序上花了错误的 token，得出错误结论，还没人知道为什么

所以迁移的第一原则不是：

- 继续堆检查器

而是：

- 先回答系统到底在保护什么、谁先判断、何时不再值得继续判断

## 2. 迁移步骤

### 步骤 1：先把规则堆叠改写成资产清单

目标：

- 先回答系统到底在保护什么，再决定用哪些规则；至少拆成能力、输入、注意力、时间、权威五类资产。

不要做什么：

- 不要一上来继续加 deny list、更多 classifier、更多 prompt 提醒。

验收信号：

- 团队已经能把每条现有规则映射到某类资产；出现问题时，不再只会说“安全规则触发了”。

### 步骤 2：把“统一预算器”拆成“治理顺序”

目标：

- 给每类资产定义判断顺序，明确谁先检查、谁有否决权、谁只能补充说明。

不要做什么：

- 不要把所有检查并排挂在同一层，再用“统一预算器”掩盖顺序错误。

验收信号：

- 对任一动作，团队可以画出固定判定链；`wrong allow` 与 `wrong deny` 能被追溯到某个顺序失效点。

### 步骤 3：先收输入边界，再谈更强判断

目标：

- 把高风险、高波动、高体积输入先拦在主 prompt 之外，改为 deferred、delta、attachment、对象外置或按需暴露。

不要做什么：

- 不要默认全量可见，再靠模型“自己懂得别用”。

验收信号：

- 主 prompt 更稳定；高波动说明不再频繁打碎前缀；问题定位从“模型乱用”转成“暴露面放错了”。

### 步骤 4：把 stable bytes 单独立法

目标：

- 明确哪些字节一旦漂移就会同时影响安全、成本和解释力，并把它们冻结成制度资产。

不要做什么：

- 不要把 prompt 文案、tool schema、betas、effort、cache control 混成“实现细节”。

验收信号：

- 团队能点名 stable bytes；每次改动都知道会不会触发 cache break、行为漂移或成本突增。

### 步骤 5：用“决策增益”替换“检查越多越安全”

目标：

- 每条检查链都要回答“继续花 token 还能改变什么决定”；不能改变决定的检查应停止或回退人工。

不要做什么：

- 不要为了看起来更稳，持续 classifier、持续 ask、持续 retry、持续 compact。

验收信号：

- cost spike 明显减少；长链路上的 token 花费更能对应真实决策变化；无增益重试被主动切断。

### 步骤 6：把自动化改成可撤销租约

目标：

- 所有 auto mode、fast-path、classifier bypass、自动 continuation 都必须有租约、撤销条件和人工接管点。

不要做什么：

- 不要把自动化当永久授权；不要只设计“怎么开”，不设计“怎么收回”。

验收信号：

- 环境变化、deny 累积、上下文过长、策略变更时，自动链路能体面退回人工或停机。

### 步骤 7：continuation 先对象化，再自动化

目标：

- 把 continuation 从“继续对话”升级成“继续哪种对象”；区分 session、task、worktree、compact 各自的合法条件。

不要做什么：

- 不要把 `continue` 当默认权利；不要在对象已不适合承载任务时继续硬续。

验收信号：

- 错误 continuation 明显减少；任务升级为 task / worktree 的时机更早；compact 后继续失败不再靠补更多 prompt 修。

### 步骤 8：最后接观测，不把观测当附属层

目标：

- 把 prompt / state / cost / governance 一起纳入正式运行时合同；至少能观测前缀、消息分解、cache break、失败语义和回退路径。

不要做什么：

- 不要上线后只看总 token 和错误率；不要把 observability 留给 debug 模式。

验收信号：

- 出现 `cache break`、`split truth`、`wrong deny`、`stale state` 时，团队能在一次排查中区分 Prompt / 治理 / 结构主线。

## 3. Rollout 顺序

```text
1. 先做资产清单与顺序图，不改行为
2. 再收输入边界与最小可见面
3. 然后冻结 stable bytes，并接 cache break 观测
4. 再上线 decision-gain stop logic
5. 再把自动化改成可撤销租约
6. 最后重写 continuation 策略与对象升级逻辑
7. 全程用演练模板记录灰度、停机与回退
```

## 4. 停止条件

```text
[ ] 连续两轮演练仍无法清楚区分 Prompt / 治理 / 结构主线
[ ] 引入新顺序后 wrong allow 或 wrong deny 显著上升
[ ] stable bytes 未冻结前就开始推广自动化
[ ] 观测面还不能解释 cost spike，却已准备做大规模 rollout
[ ] continuation 仍默认依赖长 prompt，而对象升级策略尚未定义
```

## 5. 回退条件

```text
[ ] cache break 原因无法归因，且命中率 / 成本明显恶化
[ ] approval race 退化为更慢的串行路径
[ ] 自动化回收失灵，出现不可撤销的 continue 或错误放行
[ ] compact / resume 后连续出现 replay mismatch、split truth 或 stale state
[ ] 团队开始通过“再加一条规则 / 再补一句 prompt”修系统性问题
```

## 6. 回退动作

更稳的回退顺序是：

1. 先回退自动化租约与 continuation 改写。
2. 再回退 decision-gain stop logic，仅保留资产清单、顺序图和 stable bytes 观测。
3. 最后才回退输入边界裁剪，因为那一层本身就是治理资产，不应轻易放弃。

## 8. 苏格拉底式追问

在你准备宣布“治理系统已升级”前，先问自己：

1. 我是在增强治理，还是在增强治理开销。
2. 这次上线修的是顺序，还是只增加了又一层检查器。
3. 哪些 token 还在花，但已经不再改变决定。
4. 自动化何时必须体面退场，我是否已写成制度。
5. 如果今天必须回退，我回退的是哪条链路，而不是回退整个治理理念。
