# 治理宿主迁移演练手册：统一定价控制面交接包、灰度记录与回退演练

这一章不再解释治理宿主迁移工单该怎样编排，而是把 Claude Code 式治理迁移压成一张可持续执行的交接、灰度与回退演练手册。

它主要回答五个问题：

1. 团队怎样知道治理宿主迁移真的围绕同一个统一定价控制面在运行，而不是只完成了 mode 与弹窗迁移。
2. 哪些交接字段最能保护 authority source、typed decision、decision window、continuation gate 与 rollback object 不被重新打散。
3. 灰度应该按什么顺序开放，才能避免先开放权力、后解释为什么该用这份权力。
4. 哪些回退演练最能暴露宿主又重新退回 mode 崇拜、pending action 文案与文件级回退。
5. 怎样用苏格拉底式追问避免把这层写成“权限迁移日报”。

## 1. 第一性原理

治理宿主迁移真正要守住的不是：

- mode 可以切换
- 弹窗可以出现
- token 条已经可见

而是：

- 交接、灰度与回退仍围绕同一个统一定价控制面成立

所以这层 playbook 最先要看的不是：

- 控制请求能不能跑通

而是：

1. authority source 是否仍是唯一主权面。
2. typed decision 是否仍是正式决策对象。
3. Context Usage 是否仍在解释 decision window。
4. continuation 是否仍是正式时间 gate。
5. rollback 是否仍围绕治理对象，而不是围绕文件。

## 2. 交接包最小样例

每次治理宿主迁移交接，至少应留下：

1. `governance_object_id`
2. `authority_source`
3. `effective_settings_projection`
4. `typed_decision`
5. `pending_action_surface`
6. `pending_permission_ledger`
7. `decision_window`
8. `context_usage_breakdown`
9. `continuation_gate`
10. `rollback_object`
11. `next_action`
12. `reject_reason`

如果交接只剩这些内容以外的东西：

- 当前 mode 名字
- 一张弹窗截图
- 一句“这里现在卡住了”

就说明团队还没有真正把治理迁过去。

## 3. 灰度发布顺序

更稳的治理灰度顺序是：

1. `authority source shadow`
   - 先让宿主只读消费 source / effective / applied 的差异。
2. `typed decision shadow`
   - 再灰度 ask / allow / deny 与 reason 对象。
3. `decision window read-only`
   - 再让宿主看见 Context Usage 如何解释当前窗口。
4. `continuation gate`
   - 再灰度 stop / continue / object upgrade 的正式时间边界。
5. `rollback object`
   - 最后才把正式回退对象与恢复入口放给宿主。

每个阶段至少观测：

1. 当前 authority source 是否单一。
2. typed decision 是否仍无 orphan / duplicate 漂移。
3. Context Usage 是否仍在解释 decision window，而不是重新变成仪表盘。
4. 回退入口是否仍指向 rollback object，而不是文件动作。

## 4. 回退演练集

每轮至少跑下面五个治理迁移演练：

1. `mode/settings divergence`
   - 确认宿主不会把 mode 投影当 authority source。
2. `pending_action enter/clear`
   - 确认宿主消费的是对象，而不是文案。
3. `duplicate/orphan control_response`
   - 确认宿主没有把陈旧或重复审批结果重新接回主状态机。
4. `Context Usage spike without gain`
   - 确认宿主能把窗口解释成 stop、compact 或对象升级。
5. `headless/background deny`
   - 确认宿主不会把无弹窗环境误判成默认允许。
6. `rewind_files rollback`
   - 确认宿主能把动作链回 rollback object，而不是文件列表。

## 5. 直接终止条件

出现下面情况时，应直接停止灰度并回退：

1. 宿主重新把 `permission_mode` 当全部 authority source。
2. `pending_action` 重新只剩文案。
3. `pending_permission_ledger` 重新缺席，开始吞掉 duplicate/orphan response。
4. Context Usage 重新只剩仪表盘，不解释 decision window。
5. continuation 重新默认免费，只看能不能继续。
6. rollback 重新围绕文件动作，而不是治理对象。

## 6. 复盘记录最少字段

每次治理迁移回退或停止灰度，至少记录：

1. `governance_object_id`
2. `stage`
3. `authority_gap`
4. `decision_gap`
5. `window_gap`
6. `continuation_gap`
7. `rollback_gap`
8. `rollback_action`
9. `re-entry_condition`

## 7. 苏格拉底式检查清单

在你准备宣布“治理宿主迁移已经稳定运行”前，先问自己：

1. 交接包里留下的是治理对象，还是只剩 mode 与弹窗记忆。
2. 当前 ask / allow / deny 在我的系统里还是 typed decision，还是交互副作用。
3. Context Usage 是否真的还在解释 decision window。
4. continuation 在我的系统里还是正式时间 gate，还是默认继续。
5. rollback 在我的系统里还是对象，还是已经重新退回文件回退。

## 8. 一句话总结

真正成熟的治理宿主迁移演练，不是让控制界面跑起来，而是持续证明交接、灰度与回退仍围绕同一个统一定价控制面。
