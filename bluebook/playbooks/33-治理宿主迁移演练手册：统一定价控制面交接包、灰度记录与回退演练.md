# 治理宿主迁移演练手册：governance key、typed ask、decision window、continuation pricing 与回退演练

治理宿主迁移演练真正要守住的，不是 mode 可以切换、弹窗可以出现、token 条已经可见，而是：

1. `governance key`
2. `externalized truth chain`
3. `typed ask`
4. `decision window`
5. `continuation pricing`
6. `rollback / durable-transient cleanup`

交接、灰度与回退都必须围绕这同一条链工作。

## 1. 第一性原理

治理宿主迁移真正要守住的不是：

- mode 可以切换
- 弹窗可以出现
- token 条已经可见

而是：

- 交接、灰度与回退仍围绕同一个 `governance key` 成立，而不是退回 mode 崇拜、pending action 文案与文件级回退

所以这层 playbook 最先要看的不是控制请求能不能跑通，而是：

1. `governance key` 是否仍是唯一主权面。
2. `typed ask` 是否仍是正式决策对象。
3. `decision window` 是否仍在解释当前阻塞点。
4. `continuation pricing` 是否仍在给时间扩张收费。
5. cleanup 是否仍围绕治理对象，而不是围绕文件列表。

## 2. 交接包最小样例

每次治理宿主迁移交接，至少应留下：

1. `governance_object_id`
2. `governance_key`
3. `externalized_truth_chain_ref`
4. `typed_ask_ref`
5. `pending_action_surface`
6. `decision_window_ref`
7. `context_usage_breakdown`
8. `continuation_pricing_ref`
9. `rollback_object_ref`
10. `durable_assets_after`
11. `transient_authority_cleared`
12. `reject_reason`

如果交接只剩这些内容以外的东西：

- 当前 mode 名字
- 一张弹窗截图
- 一句“这里现在卡住了”

就说明团队还没有真正把治理迁过去。

## 3. 灰度发布顺序

更稳的治理灰度顺序是：

1. `governance_key_shadow`
   - 先让宿主只读消费 source / effective / applied / externalized 的差异。
2. `typed_ask_shadow`
   - 再灰度 ask / allow / deny 与 reason 对象。
3. `decision_window_read_only`
   - 再让宿主看见 Context Usage 如何解释当前窗口。
4. `continuation_pricing`
   - 再灰度 stop / continue / object upgrade 的正式时间边界。
5. `rollback_cleanup`
   - 最后才把正式回退对象与 cleanup 语义放给宿主。

每个阶段至少观测：

1. 当前 `governance key` 是否单一。
2. `typed ask` 是否仍无 orphan / duplicate 漂移。
3. Context Usage 是否仍在解释 `decision window`，而不是重新变成仪表盘。
4. 回退入口是否仍指向 `rollback object`，而不是文件动作。

## 4. 回退演练集

每轮至少跑下面六个治理迁移演练：

1. `mode_settings_divergence`
   - 确认宿主不会把 mode 投影当 `governance key`。
2. `pending_action_enter_clear`
   - 确认宿主消费的是对象，而不是文案。
3. `duplicate_orphan_control_response`
   - 确认宿主没有把陈旧或重复审批结果重新接回主状态机。
4. `context_usage_spike_without_gain`
   - 确认宿主能把窗口解释成 stop、compact 或对象升级。
5. `headless_background_deny`
   - 确认宿主不会把无弹窗环境误判成默认允许。
6. `rewind_files_rollback`
   - 确认宿主能把动作链回 `rollback object`，而不是文件列表。

## 5. 直接终止条件

出现下面情况时，应直接停止灰度并回退：

1. 宿主重新把 `permission_mode` 当全部 `governance key`。
2. `pending_action` 重新只剩文案。
3. `permission ledger` 重新缺席，开始吞掉 duplicate / orphan response。
4. Context Usage 重新只剩仪表盘，不解释 `decision window`。
5. continuation 重新默认免费，只看能不能继续。
6. rollback 重新围绕文件动作，而不是治理对象。

## 6. 复盘记录最少字段

每次治理迁移回退或停止灰度，至少记录：

1. `governance_object_id`
2. `stage`
3. `governance_gap`
4. `ask_gap`
5. `window_gap`
6. `pricing_gap`
7. `cleanup_gap`
8. `rollback_action`
9. `re_entry_condition`

## 7. 苏格拉底式检查清单

在你准备宣布“治理宿主迁移已经稳定运行”前，先问自己：

1. 交接包里留下的是治理对象，还是只剩 mode 与弹窗记忆。
2. 当前 ask / allow / deny 在我的系统里还是 `typed ask`，还是交互副作用。
3. Context Usage 是否真的还在解释 `decision window`。
4. continuation 在我的系统里还是正式 pricing，还是默认继续。
5. rollback 在我的系统里还是对象，还是已经重新退回文件回退。

## 8. 一句话总结

真正成熟的治理宿主迁移演练，不是让控制界面跑起来，而是持续证明交接、灰度与回退仍围绕同一个 `governance key` 成立。
