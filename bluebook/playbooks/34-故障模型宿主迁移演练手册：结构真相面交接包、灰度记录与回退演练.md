# 故障模型宿主迁移演练手册：结构真相面交接包、灰度记录与回退演练

这一章不再解释故障模型宿主迁移工单该怎样编排，而是把 Claude Code 式结构迁移压成一张可持续执行的交接、灰度与回退演练手册。

它主要回答五个问题：

1. 团队怎样知道故障模型宿主迁移真的围绕同一个结构真相面在运行，而不是只完成了恢复按钮迁移。
2. 哪些交接字段最能保护 authority state、transition legality、anti-zombie evidence、recovery boundary 与 writeback 主路径不被重新打散。
3. 灰度应该按什么顺序开放，才能避免先开放恢复控制、后建立真相面。
4. 哪些回退演练最能暴露宿主又重新退回 spinner、pointer 与成功率崇拜。
5. 怎样用苏格拉底式追问避免把这层写成“恢复迁移日报”。

## 1. 第一性原理

故障模型宿主迁移真正要守住的不是：

- 恢复按钮已经可用
- pointer 还能找到
- 成功率看起来不错

而是：

- 交接、灰度与回退仍围绕同一个结构真相面成立

所以这层 playbook 最先要看的不是：

- 恢复流程是不是能跑完

而是：

1. authority state 是否仍来自单一状态面。
2. 合法转移是否仍可见。
3. anti-zombie 结果面是否仍对宿主可见。
4. recovery asset 是否仍只是资产。
5. writeback 是否仍围绕同一主路径成立。

## 2. 交接包最小样例

每次故障模型宿主迁移交接，至少应留下：

1. `kernel_object_id`
2. `authority_state_surface`
3. `transition_legality_snapshot`
4. `anti_zombie_projection`
5. `stale_writer_evidence`
6. `recovery_boundary`
7. `recovery_asset_inventory`
8. `bridge_pointer_health`
9. `writeback_delivery_state`
10. `rollback_object`
11. `degraded_path`
12. `reject_reason`

如果交接只剩这些内容以外的东西：

- 当前恢复成功率
- 一份 pointer 路径
- 一句“这次应该恢复好了”

就说明团队还没有真正把结构真相面迁过去。

## 3. 灰度发布顺序

更稳的结构灰度顺序是：

1. `writeback + state projection shadow`
   - 先只读灰度 authority state 与 writeback 真相面。
2. `resume/read path shadow`
   - 再灰度 recovery boundary 与 reentry condition。
3. `anti-zombie projection`
   - 再让宿主看见 stale writer / rejected outcome。
4. `remote control path`
   - 最后才放开 ingress / bridge / resume 等控制动作。
5. `legacy heuristic retirement`
   - 最后下线 spinner、按钮状态与成功率主导的旧面板。

每个阶段至少观测：

1. authority state 是否仍只从正式状态面宣布。
2. transition legality 是否仍可被 later maintainer 消费。
3. anti-zombie 结果面是否仍无缺席。
4. recovery asset 是否仍没有篡位。
5. writeback 主路径是否仍是唯一远端真相入口。

## 4. 回退演练集

每轮至少跑下面五个结构迁移演练：

1. `requires_action -> idle`
   - 确认宿主围绕 authority state 投影切换，而不是 UI heuristic。
2. `stale task patch drop`
   - 确认宿主能看到 anti-zombie 结果面。
3. `pointer stale/schema invalid`
   - 确认宿主把 pointer 当资产，而不当真相。
4. `resume order replay`
   - 确认 `switch/reset -> metadata -> worktree -> adopt` 顺序没有被宿主打乱。
5. `409 adopt / 401 fail-fast`
   - 确认宿主能区分 append legality 与 auth failure。
6. `epoch rebuild + writeback lag`
   - 确认宿主仍围绕统一 writeback 主路径观测恢复窗口。

## 5. 直接终止条件

出现下面情况时，应直接停止灰度并回退：

1. 宿主重新主要靠 UI heuristic 猜 authority state。
2. pointer 或恢复文件重新被神化成当前真相。
3. `bridge_pointer_health` 缺席，团队说不清 pointer 是 crash 残留还是合法恢复资产。
4. anti-zombie 结果面重新缺席。
5. writeback 重新退回日志式 best-effort，而不是正式主路径。
6. later maintainer 仍必须问作者“这次到底算不算恢复好了”。

## 6. 复盘记录最少字段

每次结构迁移回退或停止灰度，至少记录：

1. `kernel_object_id`
2. `stage`
3. `authority_gap`
4. `transition_gap`
5. `anti_zombie_gap`
6. `recovery_boundary_gap`
7. `writeback_gap`
8. `rollback_action`
9. `re-entry_condition`

## 7. 苏格拉底式检查清单

在你准备宣布“故障模型宿主迁移已经稳定运行”前，先问自己：

1. 交接包里留下的是结构真相面，还是恢复经验。
2. authority state 在我的系统里还是单一来源，还是又被 UI 猜测夺权。
3. recovery asset 还只是资产，还是已经篡位。
4. 我看到的是 anti-zombie 结果面，还是只看到成功率。
5. later maintainer 是否真的可以只靠交接包与宿主投影理解当前故障边界。

## 8. 一句话总结

真正成熟的故障模型宿主迁移演练，不是让恢复流程走通，而是持续证明交接、灰度与回退仍围绕同一个结构真相面。
