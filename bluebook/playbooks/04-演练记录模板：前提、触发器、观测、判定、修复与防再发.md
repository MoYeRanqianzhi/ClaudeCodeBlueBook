# 演练记录模板：前提、触发器、观测、判定、修复与防再发

这一章不再新增制度判断，而是提供一张统一记录模板，用来把运营层与样本层接起来。

它主要回答五个问题：

1. 为什么演练和事故复盘必须共享同一记录 ABI。
2. 怎样让 prompt、治理、源码演化三条线都能用同一张记录表。
3. 怎样保证一次记录既能服务当前复盘，又能回填到案例库。
4. 怎样用第一性原理判断这次记录是否真的有决策价值。
5. 怎样用苏格拉底式追问防止记录模板退化成空表格。

## 0. 第一性原理

真正强的运营制度，不只是会写手册，而是会把：

- 触发条件
- 观察信号
- 最终判定
- 修复动作
- 防再发动作

写成同一套最小记录 ABI。

没有这层 ABI，案例库和手册很快就会分裂成两套语言。

## 1. 统一记录模板

```text
记录名称:
时间窗口:
对象类型:
- prompt / governance / structure

authority source:
- 当前由谁给出最终权威判断

assembly path:
- 这次对象是经哪条组装/适配路径进入运行时

前提:
- 当前配置
- 当前入口
- 当前 build / transport / mode

触发器:
- 变更
- 故障
- 演练动作

期望不变量:
- 本次本应守住什么

观测信号:
- token / latency / cache
- state / cursor / pointer
- approval / race / replay

decision-gain judgement:
- 这次额外花费是否改变了最终决策

最终判定:
- 正常
- 漂移
- 失效
- split truth
- zombie risk

根因标签:
- 从 `casebooks/04` 选择

修复动作:
- rollback / invalidate / rebuild / drop / merge / retire

防再发动作:
- 新增回归
- 新增样本
- 新增标签
- 新增 stop logic

evidence schema:
- request id / tool_use_id / source / timing / cache / cursor 等最小证据字段
```

## 2. Prompt 记录补充字段

```text
- 哪个 section 变了
- 是否跨 boundary
- 哪条 sovereignty chain 受影响
- lawful forgetting ABI 是否仍成立
- invalidation 是否正确触发
```

## 3. 治理记录补充字段

```text
- 哪条规则所在顺序节点
- 是否命中 hard guard
- 哪些 approval source 参与了 race
- stable bytes 是否漂移
- stop logic 是否正确触发
```

## 4. 结构记录补充字段

```text
- 进入哪个 build surface
- 是否新增 shadow / stub / compat
- transport 差异是否仍被困在 shell
- recovery asset 是否受影响
- 是否出现 zombie 风险
```

## 5. 回填协议

每次记录完成后，都要问三件事：

1. 这次是否应回填到 `casebooks/01-03` 的某一类事故。
2. 这次是否暴露了新的标签，需要更新 `casebooks/04`。
3. 这次是否说明现有 playbook 缺少步骤，需要回写 `playbooks/01-03`。

这样记录模板才不是静态表格，而是：

- 手册层和样本层之间的回流协议

## 6. 苏格拉底式追问

在你准备填写这张表前，先问自己：

1. 这次记录是在保存事实，还是在提前解释事实。
2. 我列的是观测信号，还是主观感受。
3. 这次最终判定能否映射回统一标签体系。
4. 这次记录填完后，未来团队能否据此复现、复盘、检索。
5. 这张表是否真的帮助做决策，而不是只增加文书工作。
