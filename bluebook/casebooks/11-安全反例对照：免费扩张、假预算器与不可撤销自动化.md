# 安全反例对照：免费扩张、假预算器与不可撤销自动化

这一章不再收集“事故发生后”的治理样本，而是收集最常见的伪安全、伪省 token 方案。

它主要回答五个问题：

1. 为什么很多团队明明想学 Claude Code 的治理成熟度，最后却只学到更多规则和更多弹窗。
2. 为什么安全与省 token 的坏解法经常伪装成“更稳”“更自由”“更智能”。
3. 哪些反例最能说明免费扩张才是共同根因。
4. 怎样把这些坏解法改写成 Claude Code 式正解。
5. 怎样用苏格拉底式追问避免把这一章读成“更保守就是更好”。

## 0. 第一性原理

安全与省 token 的共同敌人，不是：

- 风险本身

而是：

- 未经定价的扩张

动作、能力、上下文与时间，只要有一项被系统免费放大，就会同时伤害：

- 安全边界
- 成本结构
- 解释能力

## 1. 免费扩张 vs 有资格进入主上下文的东西

### 坏解法

- 把更多工具、更多历史、更多记忆、更多说明默认塞进主 prompt，认为“信息越全越安全、越聪明”。

### 为什么坏

- 稳定前缀被高波动信息污染。
- token 成本随输入面线性上涨。
- 风险边界后移，系统开始依赖模型自行分辨哪些信息该忽略。

### Claude Code 式正解

- 先限制扩张，再谈能力。
- 稳定前缀、动态尾部、delta attachment、deferred tool、对象外置，共同控制什么有资格进入主上下文。

### 改写路径

1. 先把高波动内容从主 prompt 迁到 attachment / delta。
2. 把工具面改成按需暴露，而不是默认全开。
3. 把长期状态沉到 memory / task / worktree，而不是继续堆提示词。

## 2. 假统一预算器 vs 统一定价秩序

### 坏解法

- 宣称“我们有统一预算器”，但实际上只统一了 token 计数，没统一治理顺序、失败语义、自动化回收与 stable bytes。

### 为什么坏

- 安全检查顺序错了。
- 继续执行逻辑和成本逻辑脱节。
- prompt 漂移、工具漂移、审批漂移无法共用一套解释。

### Claude Code 式正解

- 不是一个大预算数字，而是一套统一定价秩序。
- 能力、注意力、时间、权威一起被 runtime 分配；token 只是其中一种表现。

### 改写路径

1. 先列出系统真正稀缺的四类资源。
2. 明确每类资源由谁分配、何时回收。
3. 最后才决定哪些指标放进同一仪表盘。

## 3. 全量可见面 vs 最小可见面

### 坏解法

- 默认让模型看到全部工具、全部协议说明、全部辅助状态、全部历史细节，再靠 prompt 提醒它“谨慎使用”。

### 为什么坏

- 模型需要先理解更多世界，成本更高。
- 错误调用面的机会更大。
- cache 更脆，解释更难。

### Claude Code 式正解

- 最小可见面优于全量能力表。
- 先限制模型可见世界，再逐步暴露可行动面。

### 改写路径

1. 区分“可存在能力”和“当前可见能力”。
2. 把发现式能力做成 deferred / search / listing，而不是常驻正文。
3. 把用户可见真相与模型可见真相分层维护。

## 4. 无决策增益检查 vs 在无增益处停止

### 坏解法

- 只要还有风险，就继续 classifier、继续审批、继续重试、继续压缩、继续总结。

### 为什么坏

- 检查已经不能改变结论。
- 系统仍在重复支付推理和协调成本。
- 最终形成“又慢又烦还没更安全”的治理噪音。

### Claude Code 式正解

- 只在仍有决策增益时花 token。
- 不能改变结论的检查，应立即停掉或回退人工。

### 改写路径

1. 给每条检查链定义“还能改变什么决定”。
2. 一旦决策增益归零，就终止自动链。
3. 把“停止判断”视为正式能力，而不是失败。

## 5. 不可撤销自动化 vs 可撤销租约

### 坏解法

- 自动化一旦开启，就一路升级：更多自动放行、更多自动续行、更多自动恢复，默认不回头。

### 为什么坏

- 人类无法及时接管。
- 环境变化后旧租约继续生效。
- 错误状态会被自动化放大。

### Claude Code 式正解

- 自动化必须是可撤销租约，不是永久授权。
- 系统要持续检查当前环境、权限、上下文、失败信号是否仍支持继续自动执行。

### 改写路径

1. 给自动化加显式 lease。
2. 为 lease 设计撤销触发器。
3. 把“回退人工”做成第一等路径，而不是兜底异常。

## 6. 错误失败语义 vs 资产分型失败语义

### 坏解法

- 把所有失败都统一处理：要么一律 fail-open，要么一律 fail-closed，要么一律“稍后重试”。

### 为什么坏

- 身份与治理边界失败，不能和普通提示失败同语义。
- 远程不可用、分类器不可用、上下文过长、组织策略未到位，也不该一刀切。
- 统一失败语义只会制造假安全或假可用。

### Claude Code 式正解

- 失败语义必须按资产分类。
- 哪些显式失败，哪些退回人工，哪些允许轻微陈旧，哪些必须硬收口，都应明示。

### 改写路径

1. 先按资产类型分组。
2. 再为每组定义失败语义。
3. 最后才设计 UI 与重试策略，避免先有表现层、后补制度。

## 7. 错误 continuation vs 对象升级

### 坏解法

- 把 continuation 当默认权利：上下文不够就 compact，任务没完就 continue，状态不清就 resume，反正先续上。

### 为什么坏

- 没有判断当前对象是否还适合承载任务。
- 历史被压缩后可能已经失去合法继续条件。
- continuation 会掩盖该升级为 task / worktree / 新对象的问题。

### Claude Code 式正解

- continuation 是受条件约束的临时资源。
- 系统必须先问：当前对象还能继续吗，还是应该升级承载对象。

### 改写路径

1. 区分 session continuation、task continuation、worktree continuation。
2. 为每种 continuation 定义合法条件和停止条件。
3. 把“升级对象”视为正解，不把它当失败退路。

## 8. 苏格拉底式追问

在你读完这些对照样例后，先问自己：

1. 我是在增加规则，还是在减少免费扩张。
2. 我保住的是制度边界，还是只是在制造更多摩擦。
3. 这轮检查还会改变决定吗。
4. 自动化是租约，还是已经偷偷变成永久主权。
5. 这套治理如果出事，团队能否沿着制度词汇而不是个人直觉复盘。
