# 治理宿主修复稳态纠偏执行反例：假纠偏、假再托管与假阈值重绑

这一章不再回答“治理宿主修复稳态纠偏执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 correction card、correction verdict order、capability recustody 与 threshold rebinding drill，仍会重新退回假纠偏、假再托管、免费继续回魂与假阈值重绑。

它主要回答五个问题：

1. 为什么治理宿主修复稳态纠偏执行最危险的失败方式不是“没有 correction card”，而是“correction card 存在，却仍围绕交互投影与保守建议工作”。
2. 为什么假纠偏最容易把 `authority reassertion`、`ledger reseal` 与 `window refreeze` 重新退回 mode、dashboard 与安静感。
3. 为什么假再托管最容易把 `capability recustody` 重新退回运维备注与主观谨慎。
4. 为什么免费继续回魂与假阈值重绑最容易把 `continuation repricing` 与 `threshold rebinding` 重新退回默认继续与以后再说。
5. 怎样用苏格拉底式追问避免把这些反例读成“把 correction card 再做清楚一点就好”。

## 1. 第一性原理

治理宿主修复稳态纠偏执行最危险的，不是：

- 没有 correction card
- 没有 capability recustody
- 没有 threshold rebinding drill

而是：

- 这些东西已经存在，却仍然围绕 mode、usage dashboard、保守建议与“现在应该还能继续”运作

一旦如此，团队就会重新回到：

1. 看 mode 是不是恢复正常
2. 看 usage 图表是不是又平了
3. 看审批历史是不是已经结束
4. 看现在能不能先继续自动化

而不再围绕：

- 同一个治理对象

## 2. 假纠偏：correction by dashboard

### 坏解法

- 宿主、CI 与评审只要看到 mode 名字没变、面板状态还绿、`correction_verdict=steady_state_restored`，就默认 `authority reassertion`、`ledger reseal` 与 `window refreeze` 已经成立，不再检查 `authority_source_after`、`permission_ledger_state` 与 `decision_window` 是否仍属于同一个治理对象。

### 为什么坏

- correction 不是“终于恢复正常”，而是“同一个治理对象重新接管定价、尾账与窗口”。
- 一旦纠偏退回 dashboard 平静感，团队就会重新容忍：
  - authority drift 被 mode 投影掩盖
  - ledger residue 被安静感掩盖
  - `correction_verdict` 被面板绿灯提前消费
- 这会让平静感直接取代治理主权。

### Claude Code 式正解

- correction verdict 应先绑定 authority source、ledger object 与 window object，再宣布 restored。


## 3. 假再托管：custody by ops note

### 坏解法

- 团队虽然写了 `capability recustody`，但真正执行时只要看到工单里写着“能力已重新托管”“现在先保守点”，就默认 capability 已被正式托管，不再检查 `capability_release_scope`、`rollback_object` 与 `custody_owner` 是否仍在同一个 formal object 里被执行。

### 为什么坏

- recustody 保护的不是“建议大家谨慎一点”，而是“扩张 scope 仍由同一个对象正式托管”。
- 一旦再托管退回表演，团队就会最先误以为：
  - “主流程没问题就行”
  - “真有事 later 会处理”
- 这会把安全设计与省 token 设计同时拆回一套不再可定价的背景噪音。

### Claude Code 式正解

- capability recustody 应继续被当作正式对象，而不是背景告警与值班注释。


## 4. 免费继续回魂与假阈值重绑：repricing by inertia

### 坏解法

- 只要没有硬错误、token 看起来还够、用户没明确拒绝，宿主就默认 capability 仍可继续，不再围绕 `continuation repricing`、`settled_price`、`budget_policy_generation` 与 `threshold_rebinding` 判断未来继续是否仍值得付费、未来 reopen 是否仍可追责。

### 为什么坏

- continuation repricing 本质上是在持续判断“继续是否仍值得付费”，不是给“再试一次”找借口。
- threshold rebinding 本质上是在恢复 future reopen 的正式追责能力，不是补一条更礼貌的说明。
- 一旦继续回魂与阈值重绑退回惯性，治理 correction execution 就会重新退回：
  - free continuation by habit
  - reopen without liability
  - custody without owner
- 这正是安全设计与省 token 设计在纠偏执行层重新脱钩的地方。

### Claude Code 式正解

- continuation 与 threshold 必须同时重新生效；没有 threshold，就只能 `hard_reject`、`liability_hold`、`reentry_required` 或 `reopen_required`。


## 5. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是治理对象，还是 mode / dashboard 的平静感。
2. 我现在重封的是 formal ledger 与 window，还是只是在接受“最近没人再报错”。
3. 我现在再托管的是 capability 边界，还是把责任礼貌地外包给 later。
4. 我现在重绑的是 future reopen 的正式 threshold，还是一句“后面有问题再看”。
5. 我现在保护的是治理对象的长期延续，还是一套更制度化的纠偏感觉。

## 6. 一句话总结

真正危险的治理宿主修复稳态纠偏执行失败，不是没跑 correction card，而是跑了 correction card 却仍在围绕假纠偏、假再托管、免费继续回魂与假阈值重绑消费治理世界。
