# 治理宿主迁移反例：权限账本缺席、假窗口对齐与免费继续回退幻觉

这一章不再回答“治理宿主迁移演练该怎样执行”，而是回答：

- 为什么团队明明已经写了统一定价控制面的交接包、灰度记录与回退演练，仍会重新退回 mode 面板、弹窗截图、仪表盘对齐与文件级回退。

它主要回答五个问题：

1. 为什么治理宿主迁移最危险的失败方式不是“没做演练”，而是“做了演练却仍围绕 mode、弹窗与通过率运作”。
2. 为什么权限账本一旦缺席，治理交接就会重新退回 UI 记忆。
3. 为什么把 `Context Usage` 误当成对齐过的仪表盘，会重新让 decision window 失语。
4. 为什么把 continuation 写成默认继续，会重新免费化时间与输出扩张。
5. 怎样用苏格拉底式追问避免把这些反例读成“再补几张控制面截图就好了”。

- 治理宿主迁移真正最容易失真的地方，不在控制面有没有搭起来，而在交接、灰度与回退是否仍围绕同一个治理对象运行。

## 1. 第一性原理

治理宿主迁移最危险的，不是：

- 没有交接模板
- 没有灰度记录
- 没有回退 drill

而是：

- 这些东西已经存在，却仍然围绕 mode、弹窗、颜色与 token 曲线运作

一旦如此，团队就会重新回到：

1. 看当前 mode 名字
2. 看审批弹窗出现没有
3. 看 token 条是不是转绿
4. 看文件是不是回退了

而不再围绕：

- 同一个 `governance control plane object`

## 2. 权限账本缺席：交接包退回 mode 与弹窗记忆

### 坏解法

- 交接包里只有当前 mode、几张审批截图和一句“这里现在卡住了”，没有 `pending_permission_ledger` 与 typed decision 对象。

### 为什么坏

- Claude Code 的权限请求天然是多路仲裁，`request_id + tool_use_id` 是最小权威键。
- 一旦账本缺席，later 接手人根本不知道哪些审批还悬而未决、哪些 `control_response` 已经重复或过期。
- 交接会重新退回“刚刚应该批过了吧”的 UI 记忆。

### Claude Code 式正解

- 交接应围绕 `authority_source + typed_decision + pending_permission_ledger + pending_action_surface`。

### 改写路径

1. 把 mode 和弹窗截图降为辅助材料。
2. 把 `pending_permission_ledger` 提升为正式交接字段。
3. 任何缺席权限账本的治理交接都判为 drift。

## 3. 假窗口对齐：Context Usage 仪表盘冒充 decision window

### 坏解法

- 灰度记录只写 token 百分比、剩余空间或压缩次数，不写 `Context Usage + pending_action + state` 的窗口对象。

### 为什么坏

- 这样会让灰度看起来像对齐了，实际却没有说明“为什么现在应该停、应该 compact、应该升级对象”。
- 宿主若自己另算一份 usage，很可能与 runtime 的真实口径漂移。
- 最后灰度面板会变成一组仪表盘，而不是治理窗口。

### Claude Code 式正解

- 灰度记录必须围绕 `decision_window + context_usage_breakdown + pending_action_surface`。

### 改写路径

1. 把 `Context Usage` 从漂亮仪表盘拉回 decision window。
2. 把 `why_now` 和 `diminishing_return_reason` 提升为正式记录项。
3. 任何只看百分比、不看窗口对象的治理灰度都判为 drift。

## 4. 免费继续回退幻觉

### 坏解法

- 只要没报硬错误、用户没叫停、token 似乎还够，宿主就默认继续。

### 为什么坏

- 这会把 continuation gate 重新退回“再来一轮看看”的默认行为。
- `currentTurnTokenBudget`、`continuationCount`、`completionEvent` 与 `diminishing returns` 会统统失语。
- 时间与输出扩张会重新变成免费资源。

### Claude Code 式正解

- 回退与继续必须围绕 `continuation_gate + tokenBudget result + decision_window`。

### 改写路径

1. 禁止“暂时还能跑”充当继续资格。
2. 把 `continuation_gate` 提升成正式回退门。
3. 任何默认继续的治理宿主都判为 drift。

## 5. rollback object 重新退回文件动作

### 坏解法

- 回退 drill 只关心回退了哪些文件、撤销了哪些 patch，不再关心 `rollback_object`。

### 为什么坏

- 这会把治理对象级回退重新降格成文件级补救。
- later 维护者会知道“怎么撤回”，却不知道“为何撤回、撤回到哪个治理对象”。

### Claude Code 式正解

- 回退应围绕 `rollback_object`，文件动作只是其执行路径。

### 改写路径

1. 把 `rollback_object` 写成回退记录主对象。
2. 把 `rewind_files` 链回当前 governance object。
3. 任何只剩文件回退的治理 drill 都判为 drift。

## 6. authority source 重新退回 mode 投影

### 坏解法

- 灰度看起来全绿，就默认 authority source 已经对齐，实际上仍只在看 `permission_mode`。

### 为什么坏

- `permission_mode` 只是投影，不是 authority source 全部。
- 一旦 `sources / effective / applied` 缺席，宿主就看不到谁改了边界、哪些危险规则已被 strip、当前外露模式是否只是一种外化方言。

### Claude Code 式正解

- authority source 必须围绕 source/effective/applied 与 external mode 一起消费。

### 改写路径

1. 把 mode 从主对象降为投影。
2. 把 authority source ledger 提升为灰度记录必填。
3. 任何只看 mode 的治理对齐都判为 drift。

## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在交接的是治理对象，还是 mode 与弹窗记忆。
2. 我现在灰度的是 decision window，还是仪表盘幻觉。
3. 我现在继续的是正式 gate，还是默认行为。
4. 我现在回退的是 rollback object，还是文件动作。
5. 我现在保护的是 authority source，还是 mode 投影。

## 8. 一句话总结

真正危险的治理宿主迁移失败，不是没做交接和灰度，而是做了这些动作却仍在围绕 mode、弹窗、仪表盘与文件回退幻觉消费治理世界。
