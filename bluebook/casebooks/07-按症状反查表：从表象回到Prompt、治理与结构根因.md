# 按症状反查表：从表象回到Prompt、治理与结构根因

这一章把参考层继续压成症状优先的诊断表。

它主要回答四个问题：

1. 现场先看见症状而不是标签时，应该先走哪条主线。
2. 怎样从表象更快回到样本、手册与解释层。
3. 为什么强诊断不是先取一个漂亮标签，而是缩短到制度根因的路径。
4. 怎样用苏格拉底式追问避免把症状表写成另一份现象目录。

## 0. 第一性原理

值班现场最先出现的通常不是：

- 根因标签

而是：

- 某种让人不安的表象

所以症状优先反查表的作用，不是给表象起学名，而是把：

- 眼前现象
- 优先主线
- 应查正文
- 容易误判的方向

重新压成一跳可用的诊断入口。

## 1. 反查总顺序

更稳的顺序是：

1. 先用本页判断优先主线落在 Prompt、治理还是结构。
2. 再回到 `04` 给这次问题补阶段、资产与根因标签。
3. 再进入对应 `playbooks/01-03` 做回归、复盘或演练。
4. 最后需要源码入口时，再用 `06` 继续反查。

## 3. 使用协议

当你只能描述“发生了什么”而还说不出“为什么发生”时：

1. 先用本页选主线。
2. 再用 `04` 把症状压回统一标签体系。
3. 再决定是回 `01-03` 看失败样本，还是回 `../playbooks/01-03` 看运营手册。

## 4. 苏格拉底式追问

在你准备靠经验直觉跳源码前，先问自己：

1. 我现在看到的是表象，还是已经足够命名根因。
2. 这次症状更像 Prompt 稳定性问题、治理顺序问题，还是结构命运问题。
3. 我是在缩短定位路径，还是只是在给现象换一个更复杂的名字。
4. 如果把这条症状写进团队知识库，未来值班者能否据此少走一轮弯路。
