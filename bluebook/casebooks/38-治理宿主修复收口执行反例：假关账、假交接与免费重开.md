# 治理宿主修复收口执行反例：假关账、假交接与免费重开

这一章不再回答“治理宿主修复收口执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 closeout card、completion verdict order、handoff warranty 与 reopen drill，仍会重新退回假关账、假交接与免费重开。

它主要回答五个问题：

1. 为什么治理宿主修复收口执行最危险的失败方式不是“没有收口卡”，而是“收口卡存在，却仍围绕交互投影工作”。
2. 为什么 authority 假收口最容易把统一定价治理对象重新退回 mode 标签、面板状态与审批完成感。
3. 为什么 ledger 未排空与 window 假关闭最容易把安全设计与省 token 设计一起降格成仪表盘刷新。
4. 为什么假交接与免费重开最容易把 continuation warranty 重新退回默认继续。
5. 怎样用苏格拉底式追问避免把这些反例读成“把审批完成页再做清楚一点就好”。

这些锚点共同说明：

- 治理宿主修复收口执行真正最容易失真的地方，不在 closeout card 有没有 authority、ledger、window 与 continuation 字段，而在完成、交接与 reopen 是否仍围绕同一个治理对象消费安全、成本、时间与回退。

## 1. 第一性原理

治理宿主修复收口执行最危险的，不是：

- 没有 authority settlement
- 没有 ledger seal
- 没有 continuation warranty

而是：

- 这些东西已经存在，却仍然围绕 mode、审批完成感、token 仪表盘与“还能再来一轮”运作

一旦如此，团队就会重新回到：

1. 看当前 mode 是不是恢复了
2. 看审批是不是结束了
3. 看 token 图表是不是降下来了
4. 看现在还能不能继续

而不再围绕：

- 同一个治理对象

## 2. authority 假收口：authority settlement 退回 mode 标签

### 坏解法

- 宿主、CI 与评审只要看到 mode 名字对了、审批结束了、页面状态转绿，就默认 authority settlement 已经完成，不再检查 `authority_source_before/after`、`effective_settings_projection` 与 `writer_chokepoint` 是否仍一致。

### 为什么坏

- Claude Code 的治理控制面真正强大之处，在于 authority source 决定允许什么、看见什么、继续多久，而不是 mode 标签本身。
- 一旦 authority settlement 退回 mode 恢复，就会重新容忍：
  - source 漂移却没人看见
  - effective settings 只是投影却被误当主权
  - writer chokepoint 漂移却只剩页面成功感
- 这会让“已经关账”的完成感直接取代治理主权。

### Claude Code 式正解

- authority settlement 应先恢复治理主权，再恢复 mode 投影。

### 改写路径

1. 把 mode 恢复降为投影级信号。
2. 把 `authority_source_before/after + effective_settings_projection + writer_chokepoint` 提升为通过前提。
3. 任何只看 mode 恢复的治理 closeout 都判为 drift。

## 3. ledger 未排空：审批历史冒充 ledger seal

### 坏解法

- 团队虽然写了 `permission_ledger`，但真正 closeout 时只看审批是否点完、请求是否消失，不再检查 `typed_decision`、`pending_permission_requests`、duplicate response 与 orphan state 是否仍被同一个 ledger 收口。

### 为什么坏

- ledger seal 保护的不是“审批做没做完”，而是“扩张理由是否仍可追溯并且没有尾账”。
- 一旦 ledger 退回审批历史，团队就会最先误以为：
  - “记录还在，应该没问题”
  - “弹窗关了，说明已经 close”
- 这会把安全设计与省 token 设计同时拆回一套无法定价的点击历史。

### Claude Code 式正解

- ledger seal 应恢复 formal reason，而不是恢复交互完成感。

### 改写路径

1. 把审批历史降为辅助材料。
2. 把 `typed_decision + permission_ledger + pending_permission_requests` 提升为恢复主对象。
3. 任何只恢复审批历史、不恢复 ledger 主权的治理 closeout 都判为 drift。

## 4. window 假关闭：仪表盘刷新冒充 window closure

### 坏解法

- 执行层只要看到 usage 百分比刷新、context bar 下降、compact 跑过，就默认 `window_closure` 已经完成，不再检查当前 request、pending action、generation 与 context usage 是否仍属于同一关闭窗口。

### 为什么坏

- Claude Code 的安全设计与省 token 设计共用同一个治理窗口；它不是“一个管权限、一个管成本”，而是同一对象在统一定价一切扩张。
- 一旦窗口关闭退回面板刷新：
  - 安全会退回审批完成感
  - 成本会退回数字变小
  - 继续会退回“似乎还能跑”
- 这会让安全与省 token 一起失语。

### Claude Code 式正解

- window closure 应先恢复同一窗口对象，再谈仪表盘变化。

### 改写路径

1. 把 usage 百分比与 compact 次数降为窗口投影。
2. 把 `decision_window + pending_action + context_usage_snapshot + late_response_quarantined` 提升为正式对象。
3. 任何只看面板刷新的治理 closeout 都判为 drift。

## 5. 假交接：handoff warranty 退回保守 mode 建议

### 坏解法

- 交接系统虽然写了 `handoff_warranty`，但真正发放交接时只给 later 团队一段“先用更保守的 mode”“先别继续自动化”的建议，而不交付同一个 governance object 的 closeout 证据。

### 为什么坏

- handoff warranty 保护的是 later 仍能围绕同一 authority source、ledger seal 与 reopen 条件继续判断，而不是收到一套更谨慎的经验建议。
- 一旦交接只剩 mode 建议与注意事项，later 团队会同时失去：
  - authority source 的正式边界
  - 统一定价对象的 completion verdict
  - continuation 是否应重新定价的依据
- 这会把治理交接退回经验传帮带。

### Claude Code 式正解

- handoff warranty 应交付治理对象，而不是交付一份更谨慎的操作建议。

### 改写路径

1. 把 mode 建议降为 handoff 注释。
2. 把 `governance_object_id + continuation_warranty + rollback_clearance + handoff_ready` 提升为交接硬条件。
3. 任何只交接经验、不交接治理对象的治理 closeout 都判为 drift。

## 6. 免费重开：reopen 退回默认继续

### 坏解法

- 只要没有硬错误、用户没明确拒绝、token 看起来还够，宿主就默认 reopen，不再围绕 `continuation_warranty`、`token_budget_result`、`re_entry_condition` 与 reopen trigger。

### 为什么坏

- Claude Code 的 reopen 本质上是在给时间与输出继续定价，而不是给“再试一次”找借口。
- 一旦 reopen 退回默认继续，就会同时做坏：
  - 安全边界，因为过期决定继续生效
  - 成本边界，因为 diminishing returns 失语
  - 交接边界，因为 later 会继承一条免费时间线
- 这正是安全设计与省 token 设计在收口层重新脱钩的地方。

### Claude Code 式正解

- reopen 应围绕 repricing 结果，而不是围绕“目前没被阻止”。

### 改写路径

1. 禁止“还没报错”充当重开资格。
2. 把 `reopen_trigger + continuation_warranty + token_budget_result + re_entry_condition` 提升为正式对象。
3. 任何默认重开的治理 closeout 都判为 drift。

## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority source，还是 mode 名字。
2. 我现在消费的是 ledger seal，还是审批历史。
3. 我现在消费的是关闭窗口，还是 usage 仪表盘。
4. 我现在交接的是治理对象，还是一套更保守的经验建议。
5. 我现在重开的是正式 repricing，还是默认继续。

## 8. 一句话总结

真正危险的治理宿主修复收口执行失败，不是没写 closeout card，而是写了 closeout card 却仍在围绕假关账、假交接与免费重开消费治理世界。
