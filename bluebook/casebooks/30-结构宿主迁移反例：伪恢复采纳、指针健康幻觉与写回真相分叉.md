# 结构宿主迁移反例：伪恢复采纳、指针健康幻觉与写回真相分叉

这一章不再回答“故障模型宿主迁移演练该怎样执行”，而是回答：

- 为什么团队明明已经写了结构真相面的交接包、灰度记录与回退演练，仍会重新退回 pointer、成功率、恢复口述与分叉写回。

它主要回答五个问题：

1. 为什么结构宿主迁移最危险的失败方式不是“没做演练”，而是“做了演练却仍围绕恢复错觉运作”。
2. 为什么伪恢复采纳最容易把 `resume/adopt` 顺序重新打乱，并制造 split-brain。
3. 为什么把 `bridgePointer` 健康度误当恢复真相，会重新神化恢复资产。
4. 为什么把 writeback 看成日志而不是真相主路径，会重新制造双真相面。
5. 怎样用苏格拉底式追问避免把这些反例读成“再多展示几个 debug 字段就好了”。

- 结构宿主迁移真正最容易失真的地方，不在恢复功能有没有跑通，而在交接、灰度与回退是否仍围绕同一个结构真相面运作。

## 1. 第一性原理

结构宿主迁移最危险的，不是：

- 没有交接模板
- 没有灰度记录
- 没有回退 drill

而是：

- 这些东西已经存在，却仍然围绕 pointer、成功率、spinner 与作者口述运作

一旦如此，团队就会重新回到：

1. 看 pointer 有没有找到
2. 看恢复最后成没成功
3. 看 UI 是否恢复正常
4. 看作者怎么解释

而不再围绕：

- 同一个结构真相面

## 2. 伪恢复采纳：resume 顺序被打乱却看起来还能跑

### 坏解法

- 宿主只要恢复出了 transcript 或某个 session id，就认为 resume/adopt 已经成立，不再关心 `switch/reset -> metadata -> worktree -> adopt` 的顺序。

### 为什么坏

- 结构真相面真正危险的不是“完全恢复失败”，而是“恢复得像成功，其实顺序错了”。
- 一旦 adopt 顺序被打乱，就可能出现 worktree 泄漏、旧 metadata 残留、resume file 未接管、sidecar 复活旧任务。
- 这类问题最像对，最难被普通成功率指标发现。

### Claude Code 式正解

- resume/adopt 必须围绕正式恢复顺序成立，而不是围绕“最后能跑起来”。


## 3. 指针健康幻觉：`bridgePointer` 被神化成恢复真相

### 坏解法

- 只要 pointer 还在、mtime 还新、路径还能找到，团队就认为结构恢复已经成立。

### 为什么坏

- `bridgePointer` 是 crash breadcrumb，不是 authority surface。
- 它会过期、会 fanout、会被 clean exit 清掉，也会因 schema 问题失效。
- 把 pointer 健康度神化成恢复真相，会直接把恢复资产篡位。

### Claude Code 式正解

- pointer 只能证明“还有恢复入口”，不能证明“当前恢复已经成立”。


## 4. 写回真相分叉：worker/session 重新各说各话

### 坏解法

- 灰度记录只看 UI 是否正常、远端请求是否成功，不检查 transcript、external metadata、worker_status 是否仍指向同一主路径。

### 为什么坏

- `WorkerStateUploader` 的语义是有界写回，不是普通日志；`print.ts` 的 hydrate 顺序也先保证 transcript，再把 metadata 回灌本地状态。
- 一旦宿主把写回当普通 telemetry，就会重新制造“本地一套真相，远端一套真相”的 split-brain。

### Claude Code 式正解

- 灰度必须围绕 `writeback_delivery_state + external_metadata + worker_status` 的统一真相面。


## 5. 成功率崇拜：anti-zombie 结果面缺席

### 坏解法

- 团队只看恢复成功率、任务最终完成率，不看 stale writer、rejected outcome、epoch rebuild 期间 dropped control 等结果面。

### 为什么坏

- 结构先进性真正保护的是 anti-zombie，不是“最后大多数时候能好”。
- 一旦只看成功率，最危险的 stale adoption、late control、zombie turn 就会被伪装成偶发噪声。

### Claude Code 式正解

- 结构迁移的交接、灰度与回退都应围绕 anti-zombie 的结果投影，而不是只围绕成功率。


## 6. sidecar 与 remote restore 口头化

### 坏解法

- 交接只说“远端任务会自己恢复”，不记录 sidecar 何时保留、何时删除、何时必须重新拉活体状态。

### 为什么坏

- `RemoteAgentTask` 已经明确区分了 sidecar 身份持久化与 live status 校验。
- 一旦这些清理与保留条件口头化，later 维护者就会最先在 404、archived、401 路径上被假恢复骗过。

### Claude Code 式正解

- 交接包必须明确 sidecar 只保存身份，不等于保存了当前真相。


## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在交接的是结构真相面，还是恢复经验和成功率。
2. 我现在灰度的是 writeback 对齐，还是 UI 看起来正常。
3. 我现在回退的是 authority state 与 recovery boundary，还是 pointer 与截图。
4. 我现在保护的是 anti-zombie 结果面，还是“最后通常会成功”的感觉。
5. 我现在消费的是正式恢复顺序，还是作者口头说明。

## 8. 一句话总结

真正危险的结构宿主迁移失败，不是没做交接和灰度，而是做了这些动作却仍在围绕伪恢复采纳、指针健康幻觉与写回真相分叉消费结构世界。
