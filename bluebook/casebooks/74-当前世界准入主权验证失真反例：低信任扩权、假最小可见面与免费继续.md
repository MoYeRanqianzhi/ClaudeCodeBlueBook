# 当前世界准入主权验证失真反例：低信任扩权、假最小可见面与免费继续

这一章不再回答“当前世界准入主权该怎样持续验证”，而是回答：

- 为什么团队明明已经有 `playbooks/78` 这一类验证手册，仍会把 mode 更安静、usage 更平、弹窗出现过与预算似乎还够误当成世界准入主权仍然成立。

它主要回答五个问题：

1. 为什么世界准入主权最危险的失败方式不是“没有门禁”，而是“门禁还在，但扩张已经不再被同一个对象定价”。
2. 为什么低信任输入最容易在 calm dashboard 下面偷偷扩权。
3. 为什么 ask 一旦退回 modal 事件，就会把 typed decision transaction 偷换成交互痕迹。
4. 为什么假最小可见面与假 externalization 最容易把未定价扩张伪装成开放能力。
5. 怎样用苏格拉底式追问避免把这些反例读成“再多加几个审批弹窗”。

- 世界准入主权真正要保护的，不是“系统现在更谨慎”，而是 trusted inputs、typed ask、visibility、externalization 与 continuation 是否仍围绕同一个统一定价对象工作。

## 1. 第一性原理

世界准入主权验证最危险的，不是：

- 没有弹窗
- 没有 token 统计

而是：

- 所有这些东西都还在，却已经不再围绕同一个 admission sovereignty object 定价扩张

一旦如此，团队就会重新回到：

1. 看 mode 有没有恢复正常。
2. 看 ask 有没有出现过。
3. 看 usage 图有没有转绿。
4. 看工具是不是“似乎都还能用”。
5. 看这轮能不能先继续再说。

而不再围绕：

- 同一个 `trusted_input_chain + typed_ask + visible_capability_set + externalized_object_set + continuation_gate`

## 2. 低信任扩权 vs trusted inputs continuity

### 坏解法

- 低信任来源、临时上下文、旁路说明或宿主局部状态开始偷偷扩边界，但团队只看 mode 是否更保守。

### 为什么坏

- `authority_leak` 最危险的地方，不是系统不再 deny，而是低信任来源开始宣布谁配进入世界。
- 一旦 trusted input chain 被 mode calmness 替代，later 团队就会不知道该先信 settings merge、host sync 还是某个局部面板。
- 世界边界会退回“谁先出现谁说了算”。

### Claude Code 式正解

- trusted inputs 必须继续由高信任主权链宣布，其他来源只能自我收缩，不能自我扩权。
- 治理 calmness 不能替代 authority provenance。

## 3. ask 退回 modal 事件 vs typed decision continuity

### 坏解法

- 只要弹窗出现、approve 点过、classifier 跑过，就把这轮视为 ask 已完成。

### 为什么坏

- `modalized_ask` 会把 typed decision transaction 偷换成“有人看过”。
- 一旦 deny / ask / allow / reason 不再被当作正式对象，治理就会退回局部交互痕迹。
- later fail-open、fail-closed、headless deny 与 asset-typed failure 都会被抹平。

### Claude Code 式正解

- ask 必须继续作为正式仲裁事务存在，而不是 UI 事件的余波。
- “审批出现过”只能是证据的一部分，不能替代 typed decision 本身。

## 4. 假最小可见面 vs 免费可见世界

### 坏解法

- 工具“存在”被直接当成“现在可见”；高体积对象名义上 externalize，实际上仍回灌主上下文或无法稳定重放。

### 为什么坏

- `free_visibility` 会把 capability registry 偷换成 runtime visibility。
- `free_context` 与 `externalization_became_cosmetic` 会让高波动对象重新霸占主 prompt。
- 团队会以为世界更开放，实际上只是把未定价扩张免费放进来了。

### Claude Code 式正解

- 可见面必须继续是按对象、按时机、按代价显式准入的，而不是“工具本来就在那里”。
- externalization 必须能稳定回放；不能只在文案上“说自己已迁出”。

## 5. 免费继续 vs continuation pricing 崩塌

### 坏解法

- 只要预算还没爆、pending action 消失、当前对话还没被叫停，就默认继续。

### 为什么坏

- `free_continuation` 会让时间重新成为免费资源。
- `windowless_pricing` 会把 decision window 退回 token 条与颜色。
- `flattened_failure_semantics` 会让 fail-open / fail-closed / revocable / irrevocable 的资产分型全部变平。

### Claude Code 式正解

- continuation 必须继续由正式 gate 决定，而不是由“现在还能跑”决定。
- 失败语义必须继续按资产与后果分型，不能再退回统一的情绪化开关。

## 6. 从更多角度看它为什么迷惑

这类假象之所以迷惑，至少有五个原因：

1. 它借用了真正严格治理的外观：mode、弹窗、usage、classifier、pending action 都还在工作。
2. 它满足了不同角色最容易满足的局部需求：宿主要安静，CI 要不过线，评审要能解释，交接要能接上。
3. 它把统一定价控制面偷换成了多个各自合理的治理投影。
4. 它把“拒绝未定价扩张”偷换成了“现在看起来更保守”。
5. 它把安全设计与省 token 设计的共同第一性原理偷换成了更谨慎的语气和界面。

## 7. 苏格拉底式追问

在你准备宣布“当前世界准入主权验证仍然健康”前，先问自己：

1. 现在到底是谁配宣布世界边界。
2. ask 还是不是一场 typed decision transaction。
3. 模型看见的是不是最小必要可见面，而不是免费开放的世界。
4. 高体积对象有没有真的迁出主上下文，而不是只换了存放位置。
5. continuation 还有没有正式价格、停止条件与责任归属。
6. 如果删掉所有 UI，这套准入主权是否仍然成立。
