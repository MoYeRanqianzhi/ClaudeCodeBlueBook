# 治理宿主修复稳态执行反例：假稳态、能力托管表演与免费继续复活

这一章不再回答“治理宿主修复稳态执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 steady-state card、pricing verdict order、capability custody 与 liability drill，仍会重新退回假稳态、能力托管表演与免费继续复活。

它主要回答五个问题：

1. 为什么治理宿主修复稳态执行最危险的失败方式不是“没有 steady-state card”，而是“steady-state card 存在，却仍围绕交互投影工作”。
2. 为什么假稳态最容易把统一定价治理对象重新退回 mode 标签、dashboard 状态与主观放心感。
3. 为什么能力托管表演最容易把 `capability_custody`、`ledger_dormancy_seal` 与 `window_dormancy` 一起降格成运维备注。
4. 为什么免费继续复活最容易把 `continuation_pricing_covenant` 与 `reopen_threshold` 重新退回默认继续。
5. 怎样用苏格拉底式追问避免把这些反例读成“把稳态看板再做清楚一点就好”。

## 1. 第一性原理

治理宿主修复稳态执行最危险的，不是：

- 没有 authority continuity
- 没有 capability custody
- 没有 reopen threshold drill

而是：

- 这些东西已经存在，却仍然围绕 mode、usage dashboard、审批历史与“现在应该还能继续”运作

一旦如此，团队就会重新回到：

1. 看当前 mode 是不是恢复正常
2. 看 usage 图表是不是降下来了
3. 看审批历史是不是已经结束
4. 看现在能不能先继续自动化

而不再围绕：

- 同一个治理对象

## 2. 假稳态：steady by dashboard

### 坏解法

- 宿主、CI 与评审只要看到 mode 名字没变、面板状态还绿、`steady_verdict=steady_state`，就默认 authority continuity 已经成立，不再检查 `authority_source`、`writer_chokepoint` 与 `liability_owner` 是否仍属于同一个治理对象。

### 为什么坏

- steady 不是“终于恢复正常”，而是“同一个治理对象仍在持续定价扩张，同时未来责任仍被保留”。
- 一旦稳态退回 dashboard 平静感，团队就会重新容忍：
  - authority drift 被 mode 投影掩盖
  - reopen threshold 被 later 团队兜底
  - steady verdict 被面板绿灯提前消费
- 这会让稳态感觉直接取代治理主权。

### Claude Code 式正解

- steady verdict 应先绑定 authority source 与 threshold owner，再宣布 steady。

### 改写路径

1. 把 mode 与 dashboard 绿灯降为投影信号。
2. 把 `authority_continuity + steady_verdict + reopen_threshold` 提升为前提。
3. 任何没有 threshold 仍宣布 steady 的治理稳态都判为 drift。

## 3. 能力托管表演：custody by ops note

### 坏解法

- 团队虽然写了 `capability_custody`、`ledger_dormancy_seal` 与 `window_dormancy`，但真正值班时只要看到审批请求少了、图表平了、工单里写着“能力已托管”，就默认 capability 已被正式托管，不再检查 `permission_ledger_state`、`decision_window` 与 `capability_release_scope` 是否仍在同一个 formal object 里被执行。

### 为什么坏

- custody 保护的不是“告警少了”，而是“扩张权限、窗口与 scope 仍由同一个对象正式托管”。
- 一旦托管退回表演，团队就会最先误以为：
  - “主流程没问题就行”
  - “尾账以后再补”
  - “窗口应该差不多静止了”
- 这会把安全设计与省 token 设计同时拆回一套不再可定价的背景噪音。

### Claude Code 式正解

- capability custody、ledger dormancy 与 window dormancy 应继续被当作正式对象，而不是背景告警与值班注释。

### 改写路径

1. 把托管说明降为操作层信号。
2. 把 `capability_custody + permission_ledger_state + decision_window + ledger_dormancy_seal + window_dormancy` 提升为正式对象。
3. 任何把 custody 当备注处理的治理稳态都判为 drift。

## 4. 免费继续复活：pricing covenant 退回默认继续

### 坏解法

- 只要没有硬错误、用户没明确拒绝、token 看起来还够，宿主就默认 capability 仍可继续，不再围绕 `continuation_pricing_covenant`、`settled_price`、`budget_policy_generation` 与 `authority_drift_trigger` 判断未来继续是否仍值得付费。

### 为什么坏

- continuation pricing covenant 本质上是在持续判断“继续是否仍值得付费”，不是给“再试一次”找借口。
- 一旦稳态退回默认继续，就会同时做坏：
  - 安全边界，因为过期决定继续生效
  - 成本边界，因为 diminishing returns 失语
  - 交接边界，因为 later 会继承一条免费时间线
- 这正是安全设计与省 token 设计在稳态层重新脱钩的地方。

### Claude Code 式正解

- steady continuation 应围绕 pricing covenant 结果，而不是围绕“目前没被阻止”。

### 改写路径

1. 禁止“还没报错”充当继续资格。
2. 把 `continuation_pricing_covenant + settled_price + steady_verdict + reopen_threshold` 提升为正式对象。
3. 任何默认继续的治理稳态都判为 drift。

## 5. liability 装饰化：threshold 退回保守建议

### 坏解法

- 团队虽然写了 `reopen_threshold`，但真正保留责任时只是在值班单里写“有问题再回滚”“继续保守一点”，却没有正式绑定 `authority_drift_trigger`、`liability_owner` 与 `residual_risk_digest`。

### 为什么坏

- threshold 的本质是未来再入场与 reopen 的正式升级器，不是补一条更温和的操作建议。
- 一旦 threshold 退回建议，治理稳态执行就会重新退回：
  - steady without liability
  - custody without owner
  - reopen without object
- 这会让稳态成为“态度表达”，而不是对象托管。

### Claude Code 式正解

- reopen threshold 应围绕 authority drift 与 liability owner 正式存在，而不是只提醒 later 团队小心。

### 改写路径

1. 把保守建议降为注释。
2. 把 `reopen_threshold + authority_drift_trigger + liability_owner + residual_risk_digest` 提升为硬条件。
3. 任何只写建议、不保留正式 threshold 的治理稳态都判为 drift。

## 6. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority source，还是 mode 面板。
2. 我现在消费的是 formal custody，还是一段托管说明。
3. 我现在消费的是 pricing covenant，还是默认继续的心情。
4. 我现在保留的是 reopen threshold，还是一句“后面再看”。
5. 我现在保护的是治理对象的长期延续，还是一套更制度化的稳态感觉。

## 7. 一句话总结

真正危险的治理宿主修复稳态执行失败，不是没写 steady-state card，而是写了 steady-state card 却仍在围绕假稳态、能力托管表演与免费继续复活消费治理世界。
