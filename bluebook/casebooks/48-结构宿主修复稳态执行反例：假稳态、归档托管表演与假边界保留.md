# 结构宿主修复稳态执行反例：假稳态、归档托管表演与假边界保留

这一章不再回答“结构宿主修复稳态执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 steady-state card、steady verdict order、archive custody、re-entry threshold 与 reopen reservation drill，仍会重新退回假稳态、归档托管表演与假边界保留。

它主要回答五个问题：

1. 为什么结构宿主修复稳态执行最危险的失败方式不是“没有 steady-state card”，而是“steady-state card 存在，却仍围绕入口感与监控工作”。
2. 为什么假稳态最容易把 `authority continuity`、`resume continuity seal` 与 `writeback custody` 重新退回 pointer、telemetry 与作者记忆。
3. 为什么归档托管表演最容易把 `anti-zombie dormancy`、`archive truth` 与 `writeback custody` 退回复盘文本与作者说明。
4. 为什么假边界保留最容易把 `reopen reservation boundary` 重新退回 reconnect、重试与旧入口循环。
5. 怎样用苏格拉底式追问避免把这些反例读成“把 steady-state 看板再做清楚一点就好”。

## 1. 第一性原理

结构宿主修复稳态执行最危险的，不是：

- 没有 authority continuity
- 没有 archive truth
- 没有 reopen reservation drill

而是：

- 这些东西已经存在，却仍然围绕 pointer、telemetry 转绿、日志繁荣与作者口述运作

一旦如此，团队就会重新回到：

1. 看 pointer 还在不在
2. 看 reconnect 是不是成功了
3. 看监控是不是转绿了
4. 看作者是不是说没问题

而不再围绕：

- 同一个结构真相面

## 2. 假稳态：steady by pointer

### 坏解法

- 宿主、CI 与评审只要看到 pointer 还在、telemetry 还绿、`steady_verdict=steady_state`，就默认 `authority_continuity`、`resume_continuity_seal` 与 `writeback_custody` 已成立，不再检查 `authority_object_id`、`writer_chokepoint` 与 `worktree_resume_path` 是否仍围绕同一个 authority object。

### 为什么坏

- steady 不是“入口还活着”，而是“同一个结构真相面仍持续成立”。
- 一旦稳态退回 pointer 健康感，团队就会重新容忍：
  - authority drift 被入口存在感掩盖
  - resume regression 被最终结果掩盖
  - writeback drift 被监控转绿掩盖
- 这会让“看起来已经稳定”的感觉直接取代结构真相面。

### Claude Code 式正解

- steady verdict 应先证明 authority、resume 与 writeback 仍围绕同一对象被持续约束，再宣布 steady。


## 3. 归档托管表演：archive by narrative

### 坏解法

- 团队虽然写了 `anti_zombie_dormancy` 与 `archive_truth`，但真正归档时只要产出一份复盘文字、几条日志截图与作者说明，就默认 zombie 风险已被托管，不再交付 `anti_zombie_projection`、`stale_writer_blocked`、`archive_pointer` 与 `retired_surface_hash`。

### 为什么坏

- archive 保护的是“过去不会再写回现在”，不是“我们已经写了复盘”。
- 一旦归档退回表演，团队就会最先误以为：
  - “结论说清楚了，应该就托管了”
  - “日志很多，later 应该能理解”
- 这会把 anti-zombie 从证据对象退回一段更会解释的文本。

### Claude Code 式正解

- archive custody 应首先是可审计证据，其次才是复盘说明。


## 4. 假边界保留：reservation 退回 reconnect 提示

### 坏解法

- 团队虽然写了 `reopen_reservation_boundary`，但真正保留重开时只是在工单里写“以后有问题再 reconnect”“原恢复入口还在”，却没有明确 `recovery_boundary`、`reservation_owner` 与 `threshold_retained_until` 的正式回跳条件。

### 为什么坏

- reservation 不是“以后再试一次”，而是“未来失稳时正式回到哪个边界组合”。
- 一旦 reservation 退回 reconnect 提示，later 团队会同时失去：
  - 正式的 recovery boundary
  - 对 stale path 的清退保证
  - 对 writeback 漂移的隔离保证
- 这会把重开退回 retry 循环。

### Claude Code 式正解

- reopen reservation 应围绕 authority、writeback 与 boundary，而不是围绕 reconnect 与旧入口。


## 5. side-write 式健康：旁路写回冒充 custody

### 坏解法

- 团队只要看到 sidecar 文件还在更新、日志里还有回写痕迹、结果也没报错，就默认 `writeback_custody` 仍成立，不再检查 `writeback_path`、`worker_status` 与 `external_metadata` 是否仍走唯一主 chokepoint。

### 为什么坏

- 写回健康不是“还有东西在写”，而是“写回仍从同一条主路径回灌真相”。
- 一旦 side write 在稳态里复活，团队就会最先误以为：
  - “结果没坏就行”
  - “旁路写回也算写回”
- 这会把 steady 从对象级托管退回幸运结果。

### Claude Code 式正解

- writeback custody 应先证明唯一主路径仍被托管，再允许任何周边资产存在。


## 6. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority object，还是 pointer、监控与作者说明。
2. 我现在验证的是对象连续性，还是健康感。
3. 我现在归档的是 anti-zombie 证据，还是一份更会解释的故事。
4. 我现在保留的是 reopen reservation boundary，还是一句“以后再 reconnect”。
5. 我现在保护的是结构边界的长期延续，还是一套更制度化的稳态幻觉。

## 7. 一句话总结

真正危险的结构宿主修复稳态执行失败，不是没写 steady-state card，而是写了 steady-state card 却仍在围绕假稳态、归档托管表演与假边界保留消费结构世界。
