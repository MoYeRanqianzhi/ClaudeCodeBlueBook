# 故障模型宿主消费反例：权威状态猜测、恢复指针神化与成功率崇拜

这一章不再回答“宿主该怎样消费故障模型支持面”，而是回答：

- 为什么宿主、SDK、CI 与交接系统明明已经拿到了 `authority state`、`generation evidence`、`recovery boundary` 与 `anti-zombie projection` 的正式支持面，仍会重新退回状态猜测、pointer 神化、恢复成功率崇拜与作者说明。

它主要回答五个问题：

1. 为什么故障模型宿主消费最危险的失败方式不是“没接恢复接口”，而是“把正式故障模型重新消费成 UI 猜测和恢复玄学”。
2. 为什么宿主自己猜 `authority state` 最容易重新长出第二真相。
3. 为什么把 generation / request 内部细节绑成宿主主真相，会重新把内部防抖机制误当业务契约。
4. 为什么恢复成功率一旦替代 recovery boundary 与 stale writer 证据，宿主就会重新误判“已经恢复好了”。
5. 怎样用苏格拉底式追问避免把这些反例读成“再多展示几个 debug 字段”。

## 1. 第一性原理

故障模型宿主消费最危险的，不是：

- 没有恢复按钮
- 没有 pointer 文件

而是：

- 已经有了回退 contract、状态投影与恢复资产，宿主却仍然用 spinner、最近一条消息、恢复成功率与作者说明去猜当前结构状态

一旦如此，宿主就会重新回到：

1. 看 spinner 还在不在
2. 看 pointer 有没有找到
3. 看这次恢复最后成没成功
4. 看作者怎么解释

而不再围绕：

- 同一个故障模型对象

## 2. 自己猜 `authority state` vs 正式状态面

### 坏解法

- 宿主用最近一条消息类型、本地按钮状态、spinner 或自建状态机去猜当前 authority state。

### 为什么坏

- 这会绕过已经正式外化的 `session_state_changed`、`pending_action` 与 `permission_mode`。
- 宿主会重新制造一套与 CLI / worker status 不一致的第二真相。
- later requires_action 丢失、mode 漂移或状态分叉时，宿主无法证明自己到底跟谁不同步了。

### Claude Code 式正解

- 宿主应围绕外化后的 authority state 投影消费当前状态，而不是重新猜状态。
- `state + pending_action + permission_mode` 应是最小正式状态面。


## 3. 把 generation / request 内部细节绑成宿主真相 vs anti-zombie 结果面

### 坏解法

- 宿主直接依赖 `_generation`、竞速输赢、去重集合或内部 request lineage 细节来组织自己的主逻辑。

### 为什么坏

- 这些对象的职责是防 stale finally、late duplicate 与 orphan response，不是给宿主当业务契约。
- 宿主一旦把这些内部防抖机制当主真相，就会把被丢弃的内部路径重新消费成外部状态。
- later 内部重构时，宿主会首先炸在最不该绑定的实现细节上。

### Claude Code 式正解

- 宿主应消费 anti-zombie 的结果面，而不是 generation 的全部内部实现。
- internal lineage 只适合调试，不适合成为宿主主逻辑。


## 4. 恢复指针神化 vs recovery boundary

### 坏解法

- 宿主只要找到了 `bridgePointer` 或类似恢复资产，就把它当成“恢复真相已经成立”。

### 为什么坏

- pointer 只是 best-effort recovery asset，不是 authority surface。
- 它会过期、会因 schema 错误被清掉、会在 worktree 间 fanout 找 freshest candidate。
- 宿主如果把“找到了 pointer”当成“当前恢复已成立”，就会把辅助资产神化成主真相。

### Claude Code 式正解

- recovery asset 只能证明“存在恢复入口”，不能证明“当前恢复已经成立”。
- 宿主应消费 recovery boundary，而不是神化 pointer 本体。


## 5. 恢复成功率崇拜 vs stale writer 证据

### 坏解法

- 宿主只看“恢复成功率”“最后成功没”，不看 stale writer 被拒绝、append-chain adopt、recovery boundary 是否发生漂移。

### 为什么坏

- 恢复不是布尔事件，而是一条对象级收敛路径。
- 只看成功率会把 split-brain、并发写入、恢复期间丢弃的 control 流统统藏起来。
- later 团队会误以为“重试后能好”就代表故障模型健康。

### Claude Code 式正解

- 宿主应围绕 stale writer evidence、append-chain adoption 与 recovery boundary 消费恢复结果。
- 成功率只能是最终投影，不能替代故障模型证据。


## 6. anti-zombie 结果面缺席 vs later maintainer 消费

### 坏解法

- 宿主只消费“任务完成了”“恢复成功了”，完全看不到 stale merge skipped、generation mismatch、stale writer rejected 这类 anti-zombie 结果面。

### 为什么坏

- later maintainer 会以为结构只是“运气好没出事”。
- 宿主与 CI 会重新退回恢复成功率崇拜和作者说明。
- anti-zombie 会重新从正式机制退回团队默契。

### Claude Code 式正解

- 宿主不一定要看到所有内部细节，但必须看到 anti-zombie 的正式结果投影。
- 这才是 later maintainer 真正能消费的结构先进性证据。


## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority state，还是自己猜出来的状态。
2. 我绑定的是故障模型结果面，还是内部 generation 细节。
3. recovery pointer 在我的系统里还是资产，还是已经被神化成真相。
4. 我看的是真正的恢复边界，还是恢复成功率。
5. later maintainer 能否只靠宿主投影看见 anti-zombie 证据。
