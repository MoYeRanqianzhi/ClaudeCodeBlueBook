这一章收集最能暴露 Claude Code 治理系统本质的失败样本。

它主要回答五个问题：

1. 为什么安全与省 token 的真义在事故里比在原则里更清楚。
2. 哪些样本最能说明治理不是规则堆叠，而是顺序制度。
3. 为什么审批竞速、auto-mode 回收与 stable-bytes drift 应放进同一张样本库。
4. 怎样从事故样本反推 Claude Code 的治理哲学。
5. 怎样用苏格拉底式追问避免把案例读成零散异常。

## 1. 案例一：危险 allow rule 抢在 classifier 前生效

### 现象

系统本来想依赖 classifier 或 safety check 决策，但某条宽放行规则提前短路了后续治理。

### 为什么重要

这不是“规则写得太松”，而是：

- 检查顺序错了

### 暴露的制度边界

- 高危 fast-path 必须被单独 inventory
- 顺序错位会同时破坏安全和成本

### 反推哲学

成熟治理最先追求的不是更多规则，而是：

- 先后顺序的正确性

## 2. 案例二：审批系统还在，但 approval race 已退化

### 现象

本地 UI、hook、bridge、classifier 都还存在，但实际只剩串行等待或 split-brain 竞速。

### 为什么重要

用户感知到的是：

- “怎么突然更慢、更烦”

真实问题却是：

- approval-race degradation

### 暴露的制度边界

- 审批是并发治理协议，不是弹窗组件
- 谁能 claim、谁能修改输入、谁该取消其他通道，都是正式制度

### 反推哲学

强治理不是让更多人来批，而是：

- 让正确的人更快给出最终决定

## 3. 案例三：auto mode 继续跑，但早已不该继续

### 现象

model、mode、环境或 denial 状态已经改变，但自动化路径仍在沿用旧租约。

### 为什么重要

这类事故不是“自动化太激进”，而是：

- automation lease 没有被合法撤销

### 暴露的制度边界

- 自动化必须是可撤销主权
- stop logic 与 revalidation 不是附属功能

### 反推哲学

强系统的自动化更像：

- 可回收的特许经营权

## 4. 案例四：stable bytes 漂移，系统突然又贵又不稳

### 现象

prompt sections、tool visibility、preview marker 或 headers 变了，系统行为和成本突然异常。

### 为什么重要

表面看像性能或体验波动，底层其实是：

- 制度字节漂移

### 暴露的制度边界

- 安全和成本共用同一批关键字节
- 字节漂移不被审计，治理系统就会变成黑箱

### 反推哲学

真正成熟的治理，不只管动作，还管：

- 哪些字节有资格决定动作

## 5. 案例五：系统还在继续花 token，但结论已不可能改变

### 现象

自动链路继续 classifier、继续重试、继续 ask，但其实没有任何新决策增益。

### 为什么重要

这不是“有点浪费”，而是：

- stop-logic failure

### 暴露的制度边界

- token 花费必须服从 decision gain
- 不能改变结论的推理，就是治理噪音

### 反推哲学

省 token 的本质不是压缩文本，而是：

- 停止无效判断

## 6. 苏格拉底式追问

在你读完这些事故样本后，先问自己：

1. 这是规则不够，还是顺序错了。
2. 审批慢是更安全了，还是 race 退化了。
3. 自动化继续跑是更智能了，还是 stop logic 缺席了。
4. 成本波动是更难了，还是 stable bytes 漂移了。
5. 我现在是在看治理结果，还是在看治理制度本身是否还活着。

## 7. 一句话总结

安全与省 token 的真义，在事故里最清楚地暴露为：顺序、竞速、撤销、字节和停止条件共同构成同一套治理制度。
