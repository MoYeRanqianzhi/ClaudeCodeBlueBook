# 治理宿主修复监护执行反例：假观察、假隔离与免费重开

这一章不再回答“治理宿主修复监护执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 watch card、drift verdict order、quarantine order 与 reopen drill，仍会重新退回假观察、假隔离与免费重开。

它主要回答五个问题：

1. 为什么治理宿主修复监护执行最危险的失败方式不是“没有 watch card”，而是“watch card 存在，却仍围绕交互投影工作”。
2. 为什么假观察最容易把统一定价治理对象重新退回 mode 标签、面板状态与审批历史。
3. 为什么假隔离最容易把 ledger residue、window reopening 与 rollback quarantine 一起降格成运维备注。
4. 为什么免费重开最容易把 continuation repricing watch 重新退回默认继续。
5. 怎样用苏格拉底式追问避免把这些反例读成“把观察看板再做清楚一点就好”。

## 1. 第一性原理

治理宿主修复监护执行最危险的，不是：

- 没有 authority watch
- 没有 ledger residue
- 没有 quarantine order

而是：

- 这些东西已经存在，却仍然围绕 mode、usage dashboard、审批历史与“先保守一点”运作

一旦如此，团队就会重新回到：

1. 看当前 mode 是不是还绿
2. 看 usage 图表是不是还稳定
3. 看审批历史是不是还在
4. 看现在能不能先继续

而不再围绕：

- 同一个治理对象

## 2. 假观察：authority watch 退回 mode 看板

### 坏解法

- 宿主、CI 与评审只要看到 mode 名字没变、面板状态还绿、usage 没爆，就默认 authority watch 仍然成立，不再检查 `authority_source_snapshot`、`effective_settings_projection` 与 `writer_chokepoint` 是否仍一致。

### 为什么坏

- authority watch 观察的是“谁还在定价”，不是“面板看起来是否平静”。
- 一旦 authority watch 退回 mode 看板，就会重新容忍：
  - authority drift 被投影掩盖
  - effective settings 漂移却没人看见
  - writer chokepoint 漂移却只剩 dashboard 成功感
- 这会让“正在观察”的治理感直接取代治理主权。

### Claude Code 式正解

- authority watch 应先观察治理主权，再观察 mode 投影。

### 改写路径

1. 把 mode 与 dashboard 降为投影级信号。
2. 把 `authority_source_snapshot + effective_settings_projection + writer_chokepoint` 提升为监护前提。
3. 任何只看 mode 与 dashboard 的治理 watch 都判为 drift。

## 3. ledger residue 噪声化：尾账被当成告警噪音

### 坏解法

- 团队虽然写了 `ledger_residue`，但真正值班时只把残留请求、late response 与 orphan state 当成背景噪音，不再检查它们是否仍在同一个 ledger 里被正式记账与排空。

### 为什么坏

- residue 保护的不是“还有没有小告警”，而是“扩张理由是否还有尾账没有清算”。
- 一旦 residue 被当成噪音，团队就会最先误以为：
  - “主流程没问题就行”
  - “尾账以后再看”
- 这会把安全设计与省 token 设计同时拆回一套不再可定价的告警噪音。

### Claude Code 式正解

- ledger residue 应继续被当作正式对象，而不是背景告警。

### 改写路径

1. 把告警噪音降为操作层信号。
2. 把 `typed_decision_digest + permission_ledger_state + pending_permission_requests` 提升为正式对象。
3. 任何把 residue 当噪音处理的治理 watch 都判为 drift。

## 4. 假隔离：quarantine 退回运维备注

### 坏解法

- 团队虽然写了 `rollback_quarantine`，但真正隔离时只是在值班单里写“先别动自动化”“先别继续放量”，却没有正式冻结 capability expansion、排空 outstanding request 或阻断 later handoff。

### 为什么坏

- quarantine 的本质是隔离余波扩张，不是补一条更谨慎的操作建议。
- 一旦 quarantine 退回备注，治理监护执行就会重新退回：
  - expansion without gate
  - residue without drain
  - handoff without quarantine
- 这会让隔离成为“态度表达”，而不是对象阻断。

### Claude Code 式正解

- quarantine 应隔离对象级扩张，而不是只提醒 later 团队小心。

### 改写路径

1. 把保守建议降为注释。
2. 把 `rollback_quarantine + quarantine_scope + handoff_status` 提升为隔离硬条件。
3. 任何只写备注、不冻结扩张的治理 watch 都判为 drift。

## 5. window 观察面板化：dashboard 刷新冒充 window watch

### 坏解法

- 值班系统只要看到 usage 百分比刷新、context bar 下降、compact 跑过，就默认 `window_watch` 仍然健康，不再检查当前 request、pending action 与 late response 是否仍属于同一观察窗口。

### 为什么坏

- window watch 观察的是“时态是否仍受控”，不是“图表是否仍漂亮”。
- 一旦 window watch 退回 dashboard：
  - 安全会退回审批完成感
  - 成本会退回数字变小
  - 继续会退回‘似乎还能跑’
- 这会让安全与省 token 一起失语。

### Claude Code 式正解

- window watch 应先观察同一窗口对象，再接受图表变化。

### 改写路径

1. 把 usage 百分比与 compact 次数降为窗口投影。
2. 把 `decision_window + pending_action + context_usage_snapshot + late_response_quarantined` 提升为正式对象。
3. 任何只看 dashboard 刷新的治理 watch 都判为 drift。

## 6. 免费重开：repricing watch 退回默认继续

### 坏解法

- 只要没有硬错误、用户没明确拒绝、token 看起来还够，宿主就默认 reopen，不再围绕 `continuation_gate`、`token_budget_result`、`repricing_due_at` 与 quarantine 状态。

### 为什么坏

- continuation repricing watch 本质上是在持续判断‘继续是否仍值得付费’，不是给‘再试一次’找借口。
- 一旦 reopen 退回默认继续，就会同时做坏：
  - 安全边界，因为过期决定继续生效
  - 成本边界，因为 diminishing returns 失语
  - 交接边界，因为 later 会继承一条免费时间线
- 这正是安全设计与省 token 设计在监护层重新脱钩的地方。

### Claude Code 式正解

- reopen 应围绕 repricing 结果，而不是围绕“目前没被阻止”。

### 改写路径

1. 禁止“还没报错”充当重开资格。
2. 把 `repricing_due_at + continuation_gate + token_budget_result + reopen_gate` 提升为正式对象。
3. 任何默认重开的治理 watch 都判为 drift。

## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority source，还是 mode 面板。
2. 我现在消费的是 ledger residue，还是告警噪音。
3. 我现在消费的是观察窗口，还是 usage dashboard。
4. 我现在隔离的是治理对象，还是一串保守建议。
5. 我现在重开的是正式 repricing，还是默认继续。

## 8. 一句话总结

真正危险的治理宿主修复监护执行失败，不是没写 watch card，而是写了 watch card 却仍在围绕假观察、假隔离与免费重开消费治理世界。
