# 治理宿主修复解除监护执行反例：免责式出监、假清账与免费继续

这一章不再回答“治理宿主修复解除监护执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 release card、release verdict order、capability release 与 reopen liability drill，仍会重新退回免责式出监、假清账与免费继续。

它主要回答五个问题：

1. 为什么治理宿主修复解除监护执行最危险的失败方式不是“没有 release card”，而是“release card 存在，却仍围绕交互投影工作”。
2. 为什么免责式出监最容易把统一定价治理对象重新退回 mode 标签、dashboard 状态与主观放心感。
3. 为什么假清账最容易把 ledger clearance、window exit 与 quarantine release 一起降格成运维备注。
4. 为什么免费继续最容易把 continuation settlement 与 reopen liability 重新退回默认继续。
5. 怎样用苏格拉底式追问避免把这些反例读成“把 release 看板再做清楚一点就好”。

## 1. 第一性原理

治理宿主修复解除监护执行最危险的，不是：

- 没有 authority release
- 没有 ledger clearance
- 没有 reopen liability drill

而是：

- 这些东西已经存在，却仍然围绕 mode、usage dashboard、审批历史与“现在应该可以放行了”运作

一旦如此，团队就会重新回到：

1. 看当前 mode 是不是恢复正常
2. 看 usage 图表是不是降下来了
3. 看审批历史是不是已经结束
4. 看现在能不能先继续自动化

而不再围绕：

- 同一个治理对象

## 2. 免责式出监：release without liability

### 坏解法

- 宿主、CI 与评审只要看到 mode 名字没变、面板状态还绿、`capability_release_verdict=capability_released`，就默认 authority release 已经完成，不再检查 `authority_source`、`writer_chokepoint` 与 `reopen_liability` 是否仍属于同一个治理对象。

### 为什么坏

- capability release 不是“终于恢复正常”，而是“同一个治理对象可以重新放开扩张，同时未来责任仍被保留”。
- 一旦 release 退回免责式出监，团队就会重新容忍：
  - authority drift 被 mode 投影掩盖
  - reopen liability 被 later 团队兜底
  - release verdict 被面板绿灯提前消费
- 这会让 released 感直接取代治理主权。

### Claude Code 式正解

- capability release 应先绑定 authority source 与 reopen liability，再宣布 released。

### 改写路径

1. 把 mode 与 dashboard 绿灯降为投影信号。
2. 把 `authority_release + capability_release_verdict + reopen_liability` 提升为 release 前提。
3. 任何没有 liability 仍宣布 capability released 的治理 release 都判为 drift。

## 3. 假清账：ledger clearance 退回运维备注

### 坏解法

- 团队虽然写了 `ledger_clearance` 与 `window_exit`，但真正值班时只要看到审批请求少了、图表平了、工单里写着“尾账已清”，就默认 ledger 已经 clearance，不再检查 `typed_decision_digest`、`permission_ledger_state` 与 `pending_permission_requests` 是否仍在同一个 formal ledger 里被正式排空。

### 为什么坏

- clearance 保护的不是“告警少了”，而是“扩张理由的正式尾账已被清算”。
- 一旦清账退回备注，团队就会最先误以为：
  - “主流程没问题就行”
  - “尾账以后再补”
  - “窗口退出应该差不多成立了”
- 这会把安全设计与省 token 设计同时拆回一套不再可定价的背景噪音。

### Claude Code 式正解

- ledger clearance 与 window exit 应继续被当作正式对象，而不是背景告警与值班注释。

### 改写路径

1. 把尾账说明降为操作层信号。
2. 把 `typed_decision_digest + permission_ledger_state + pending_permission_requests + window_exit` 提升为正式对象。
3. 任何把 clearance 当备注处理的治理 release 都判为 drift。

## 4. 免费继续：continuation settlement 退回默认继续

### 坏解法

- 只要没有硬错误、用户没明确拒绝、token 看起来还够，宿主就默认 capability release，不再围绕 `continuation_settlement`、`token_budget_result`、`repricing_due_at` 与 `quarantine_scope` 判断未来继续是否仍值得付费。

### 为什么坏

- continuation settlement 本质上是在持续判断“继续是否仍值得付费”，不是给“再试一次”找借口。
- 一旦 release 退回默认继续，就会同时做坏：
  - 安全边界，因为过期决定继续生效
  - 成本边界，因为 diminishing returns 失语
  - 交接边界，因为 later 会继承一条免费时间线
- 这正是安全设计与省 token 设计在出监层重新脱钩的地方。

### Claude Code 式正解

- capability release 应围绕 settlement 结果，而不是围绕“目前没被阻止”。

### 改写路径

1. 禁止“还没报错”充当 capability release 资格。
2. 把 `continuation_settlement + token_budget_result + capability_release_verdict + reopen_liability` 提升为正式对象。
3. 任何默认继续的治理 release 都判为 drift。

## 5. quarantine 装饰化：scope 退回保守建议

### 坏解法

- 团队虽然写了 `quarantine_release`，但真正解除隔离时只是在值班单里写“可以逐步放开”“先谨慎一点”，却没有正式冻结或分层恢复 capability scope，也没有检查 `quarantine_scope` 与 `rollback_object` 是否仍一致。

### 为什么坏

- quarantine release 的本质是解除对象级冻结，不是补一条更温和的操作建议。
- 一旦 quarantine 退回建议，治理解除监护执行就会重新退回：
  - release without scope
  - handoff without boundary
  - liability without object
- 这会让隔离成为“态度表达”，而不是对象放行。

### Claude Code 式正解

- quarantine release 应按 scope 正式解除，而不是只提醒 later 团队小心。

### 改写路径

1. 把保守建议降为注释。
2. 把 `quarantine_release + quarantine_scope + rollback_object + handoff_status` 提升为解除隔离硬条件。
3. 任何只写建议、不解除对象边界的治理 release 都判为 drift。

## 6. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 authority source，还是 mode 面板。
2. 我现在消费的是 formal ledger，还是一段清账说明。
3. 我现在消费的是 settlement，还是默认继续的心情。
4. 我现在解除的是 quarantine scope，还是一串保守建议。
5. 我现在保留的是 reopen liability，还是 later 团队的兜底义务。

## 7. 一句话总结

真正危险的治理宿主修复解除监护执行失败，不是没写 release card，而是写了 release card 却仍在围绕免责式出监、假清账与免费继续消费治理世界。
