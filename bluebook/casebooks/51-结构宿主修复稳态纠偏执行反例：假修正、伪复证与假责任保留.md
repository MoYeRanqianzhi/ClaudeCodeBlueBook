# 结构宿主修复稳态纠偏执行反例：假修正、伪复证与假责任保留

这一章不再回答“结构宿主修复稳态纠偏执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 correction card、correction verdict order、archive restitution、re-entry threshold 与 reopen responsibility drill，仍会重新退回假修正、假恢复顺序、伪复证与假责任保留。

它主要回答五个问题：

1. 为什么结构宿主修复稳态纠偏执行最危险的失败方式不是“没有 correction card”，而是“correction card 存在，却仍围绕 pointer、telemetry 与 archive prose 工作”。
2. 为什么假修正最容易把 `authority rebind` 重新退回 pointer 健康感与作者说明。
3. 为什么假恢复顺序最容易把 `resume lineage rebuild` 重新退回幸运 reconnect 与结果导向。
4. 为什么伪复证与假责任保留最容易把 `writeback recustody`、`anti_zombie reproof`、`archive truth restitution` 与 `reservation rebinding` 重新退回复盘文本、监控转绿与“以后再 reconnect”。
5. 怎样用苏格拉底式追问避免把这些反例读成“把 correction card 再做清楚一点就好”。

## 1. 第一性原理

结构宿主修复稳态纠偏执行最危险的，不是：

- 没有 authority rebind
- 没有 archive restitution
- 没有 reopen responsibility drill

而是：

- 这些东西已经存在，却仍然围绕 pointer、telemetry 转绿、archive prose 与作者口述运作

一旦如此，团队就会重新回到：

1. 看 pointer 还在不在
2. 看 reconnect 是不是成功了
3. 看监控是不是转绿了
4. 看作者是不是说没问题

而不再围绕：

- 同一个结构真相面

## 2. 假修正：correction by pointer

### 坏解法

- 宿主、CI 与评审只要看到 pointer 还在、telemetry 还绿、`correction_verdict=steady_state_restored`，就默认 `authority rebind` 已成立，不再检查 `authority_object_id`、`authority_rebind_state` 与 `writer_chokepoint` 是否仍围绕同一个 authority object。

### 为什么坏

- correction 不是“入口还活着”，而是“同一个结构真相面已重新夺回主权”。
- 一旦纠偏退回 pointer 健康感，团队就会重新容忍：
  - authority drift 被入口存在感掩盖
  - `correction_verdict` 被健康感提前消费
- 这会让“看起来已经恢复”的感觉直接取代结构真相面。

### Claude Code 式正解

- correction verdict 应先证明 authority object 仍唯一、仍被重新绑定，再宣布 restored。


## 3. 假恢复顺序：resume by lucky reconnect

### 坏解法

- 团队虽然写了 `resume lineage rebuild`，但真正执行时只要 reconnect 成功、最终结果没坏、later 团队能接着做，就默认顺序已经 rebuild，不再检查 `resume_lineage_ref`、`return_lineage`、`late_resume_detected` 与 `generation_regression_detected` 是否仍被正式约束。

### 为什么坏

- resume rebuild 保护的是同一 lineage 的合法恢复顺序，不是“反正连上了”。
- 一旦恢复顺序退回好运气，团队就会最先误以为：
  - “能继续就说明 lineage 没问题”
  - “结果没坏就说明顺序也没坏”
- 这会把结构纠偏从顺序真相退回结果崇拜。

### Claude Code 式正解

- resume rebuild 必须围绕 lineage object 与顺序闭合，而不是围绕 lucky reconnect 与最终结果。


## 4. 伪复证：reproof by archive prose

### 坏解法

- 团队虽然写了 `anti_zombie reproof` 与 `archive truth restitution`，但真正归档时只要产出一份复盘文字、几条日志截图与 archive note，就默认 zombie 风险已被复证，不再交付 `anti_zombie_evidence_ref`、`stale_writer_evidence`、`archive_truth_ref` 与 `archive_pointer`。

### 为什么坏

- reproof 保护的是“过去不会再写回现在”，不是“我们已经写了复盘”。
- archive restitution 保护的是结构真相仍可审计，不是“解释已经完整”。
- 一旦复证退回 prose，团队就会最先误以为：
  - “结论说清楚了，应该就恢复了”
  - “later 应该能理解”
- 这会把 anti-zombie 从证据对象退回一段更会解释的文本。

### Claude Code 式正解

- anti-zombie reproof 与 archive restitution 应首先是可审计证据，其次才是复盘说明。


## 5. 假责任保留：responsibility by reconnect hint

### 坏解法

- 团队虽然写了 `reservation rebinding` 与 `reopen responsibility drill`，但真正保留责任时只是在工单里写“以后有问题再 reconnect”“原恢复入口还在”，却没有明确 `reopen_reservation_boundary`、`reservation_owner`、`threshold_retained_until` 与 `return_writeback_path` 的正式回跳条件。

### 为什么坏

- responsibility 不是“以后再试一次”，而是“未来失稳时正式回到哪个边界组合并由谁负责”。
- 一旦责任保留退回 reconnect 提示，later 团队会同时失去：
  - 正式的 recovery boundary
  - 对 stale path 的清退保证
  - 对 writeback 漂移的隔离保证
- 这会把 reopen 退回 retry 循环。

### Claude Code 式正解

- reservation 与 responsibility 应围绕 authority、writeback 与 boundary，而不是围绕 reconnect 与旧入口。


## 6. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在修回的是唯一 authority object，还是只把入口感重新包装了一遍。
2. 我现在保住的是 `resume lineage`，还是几次幸运 reconnect 的结果。
3. 我现在托管的是唯一 writeback 主路径，还是监控转绿后的安心感。
4. 我现在归档的是 anti-zombie 证据，还是一段更会解释的 archive prose。
5. later 团队明天如果必须 reopen，依赖的是正式 boundary，还是一句“以后再试一次”。

## 7. 一句话总结

真正危险的结构宿主修复稳态纠偏执行失败，不是没跑 correction card，而是跑了 correction card 却仍在围绕假修正、假恢复顺序、伪复证与假责任保留消费结构世界。
