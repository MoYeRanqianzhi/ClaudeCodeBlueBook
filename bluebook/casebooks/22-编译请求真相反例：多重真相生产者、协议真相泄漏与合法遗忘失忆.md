# 编译请求真相反例：多重真相生产者、协议真相泄漏与合法遗忘失忆

这一章不再回答“怎样验证 `compiled request truth` 仍然成立”，而是回答：

- 为什么 Prompt 机制明明已经被写成 `section law + stable prefix + protocol rewrite + lawful forgetting`，团队却仍会重新退回原文 prompt 崇拜与摘要崇拜。

它主要回答五个问题：

1. 为什么 Prompt 机制最危险的失败方式不是“没写 prompt”，而是“长出多个都像真的 request truth”。
2. 为什么协议真相一旦泄漏回 UI 历史与显示层消息，Prompt 魔力就会立刻失真。
3. 为什么 side loop、compact、resume 与 handoff 最容易重新发明自己的半真相世界。
4. 为什么 lawful forgetting 一旦退回摘要崇拜，系统就会忘掉继续条件而不是忘掉噪音。
5. 怎样用苏格拉底式追问避免把这些反例读成“Prompt 文案写坏了”。

这些锚点共同说明：

- Prompt 魔力真正脆弱的地方，不在措辞，而在“唯一 request truth 是否仍被同一条编译链生产、同一条协议链消费、同一条继续链继承”。

## 1. 第一性原理

编译请求真相层最危险的，不是：

- prompt 不够长
- section 不够多

而是：

- 系统已经有 Prompt 机制，却重新长出多个都像最终真相的生产者

一旦如此，团队就会重新回到：

1. 看原文 prompt
2. 看 UI transcript
3. 看 compact summary
4. 看 reviewer 总结

而不再围绕：

- 同一个 `compiled request truth`

## 2. 多重真相生产者 vs 唯一 request object

### 坏解法

- system prompt section、动态追加段、tool schema、resume 补边各自产生一份“最终 prompt 解释”，没有单一 request object。

### 为什么坏

- Prompt 魔力会从“编译对象”退回“多份相似文本竞争”。
- 团队会开始争论哪份描述更完整，而不是哪条编译链才是权威。
- later cache break 发生时，没有人能指出到底是哪一份真相变了。

### Claude Code 式正解

- `section registry -> dynamic boundary -> stable prefix split -> request state record` 必须仍是同一条主权链。
- 任何侧路径都只能消费这条主权链留下的对象，不能再造一份“差不多的世界”。

### 改写路径

1. 把“最终 request object 在哪里生产”显式写出来。
2. 拒绝多个入口各自宣布最终 prompt 真相。
3. 让所有 debug、review 与 handoff 都回链同一 request object。

## 3. 协议真相泄漏 vs protocol rewrite

### 坏解法

- UI 历史、显示层消息、孤儿 tool_result 或格式不合法的 transcript 直接进入模型消费主路径。

### 为什么坏

- 模型看到的世界重新退回“谁先把文本塞进去”。
- Prompt 魔力会被误以为来自措辞，而不是来自合法消息 ABI。
- 一旦 transcript 内部 pairing 失真，继续链会开始围绕错误历史工作。

### Claude Code 式正解

- `normalizeMessagesForAPI()` 与 `ensureToolResultPairing()` 必须先把历史编译成合法协议对象，再让 Prompt 工作。
- UI transcript 只能是显示层事实，不得替代 protocol transcript。

### 改写路径

1. 把“显示层历史”和“模型消费历史”强制分开叙述。
2. 让每次 transcript 修复都留下正式 pairing 证据。
3. 任何 display-only 消息进入主路径都直接判为 drift。

## 4. 旁路循环自行重建世界 vs shared prefix

### 坏解法

- side question、compact fork、resume、handoff 各自重建一套 prompt 前缀，只求“看起来差不多”。

### 为什么坏

- 共享前缀资产会退回“多份背景副本”。
- Prompt 魔力会表现成偶发稳定，而不是制度性稳定。
- later side path 与 main path 一旦分叉，团队只会感觉“怎么忽然不聪明了”。

### Claude Code 式正解

- 旁路循环必须尽量复用 `cache-safe params` 与主线程前缀，只允许重建同一个世界，不允许重建另一个世界。
- fallback 只能是同一主权链的补写，不是新世界生成器。

### 改写路径

1. 明确谁是唯一 stable prefix producer。
2. 让所有 side path 先回链 parent prefix，再决定是否补边。
3. 任何没有合法前缀就自行开工的 side loop 都直接拒收。

## 5. 合法遗忘退回摘要崇拜 vs continuation object

### 坏解法

- compact 只留下 task summary，却丢掉 next step、rollback boundary、tool pairing 或继续条件。

### 为什么坏

- 系统忘掉的不是噪音，而是继续工作的语法。
- lawful forgetting 会退回“更短的故事”，而不是“仍可继续的最小对象”。
- later resume 看起来像成功恢复，实际上只是拿着摘要重新猜世界。

### Claude Code 式正解

- compact 必须保住 `current work + next step + preserved boundary + required assets`。
- summary 只是 continuation object 的投影，不能替代 continuation object 本身。

### 改写路径

1. 强制 compact 输出先回答“接下来怎样继续”。
2. 把 rollback boundary 与 next step 视为必须保留资产。
3. 任何只剩摘要、不剩继续条件的 compact 都判为失忆。

## 6. cache break 退回玄学 vs stable bytes explanation

### 坏解法

- cache miss 发生后，只会说“最近变贵了”“感觉不稳定了”，却说不清是 section、tool schema、beta header 还是 body param 变了。

### 为什么坏

- 可解释稳定性会重新退回玄学稳定性。
- 团队会把 Prompt 魔力误读成“偶尔灵、偶尔不灵”的黑箱。
- later 调优只会更依赖猜测和咒语。

### Claude Code 式正解

- cache break 必须能被解释到具体稳定字节层。
- “为什么失稳”必须和“当前 request object 是什么”一样可追责。

### 改写路径

1. 强制记录 stable bytes inventory。
2. 把 break reason 从日志散文提升为正式字段。
3. 任何不能解释 break 的稳定性讨论都视为退回玄学。

## 7. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在保护的是同一个 compiled request truth，还是几份长得相似的 prompt 解释。
2. 模型消费的到底是 protocol transcript，还是显示层历史的替身。
3. 旁路循环是不是还活在同一个 stable prefix 宇宙里。
4. compact 之后系统留下的是 continuation object，还是一段更短的故事。
5. 我能否正式解释这次 cache break，而不是把 Prompt 魔力继续神秘化。
