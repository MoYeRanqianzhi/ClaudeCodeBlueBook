# 结构宿主修复监护执行反例：authority object 假稳定、freshness gate 假冻结与假 reopen

这一章不再回答“结构宿主修复监护执行该怎样运行”，而是回答：

- 为什么团队明明已经写了 watch card、drift verdict order、handoff freeze 与 reopen drill，仍会重新退回假稳定、假冻结与假 reopen。

它主要回答五个问题：

1. 为什么结构宿主修复监护执行最危险的失败方式不是“没有 watch card”，而是“watch card 存在，却仍围绕入口感、监控工作与旧投影判断 current truth”。
2. 为什么假稳定最容易把 `authority object`、`per-host authority width` 与 `state writeback` 重新退回 pointer、监控与作者记忆。
3. 为什么假冻结最容易把 `freshness gate` 与 boundary quarantine 退回备注与截图。
4. 为什么假 reopen 最容易把对象级重开退回 reconnect、重试与旧入口循环。
5. 怎样用苏格拉底式追问避免把这些反例读成“把观察看板再做细一点就好”。

## 1. 第一性原理

结构宿主修复监护执行最危险的，不是：

- 没有 `authority object / per-host authority width` 守卫
- 没有 `event-stream-vs-state-writeback` 守卫
- 没有 reopen drill

而是：

- 这些东西已经存在，却仍然围绕 pointer、监控转绿、日志繁荣与作者口述运作

一旦如此，团队就会重新回到：

1. 看 pointer 还在不在
2. 看 reconnect 是不是成功了
3. 看监控是不是转绿了
4. 看作者是不是说没问题

而不再围绕：

- 同一个 `authority object -> per-host authority width -> event-stream-vs-state-writeback -> freshness gate -> stale worldview / ghost capability -> reopen boundary`

在 watch 现场里最常见的五个替身也正好对应这条链：

1. pointer / sidecar
   - 冒充 `authority object`
2. 监控平静感
   - 冒充 `per-host authority width`
3. telemetry 繁荣
   - 冒充 `state writeback`
4. 作者说明
   - 冒充 `stale worldview / ghost capability` 证据
5. reconnect 成功
   - 冒充 `reopen boundary`

## 2. 假稳定：pointer、监控与作者记忆冒充 authority width

### 坏解法

- 宿主、CI 与评审只要看到 pointer 文件还在、monitoring 还绿、作者说“应该稳定了”，就默认 `authority object / per-host authority width` 守卫与 `event-stream-vs-state-writeback` guard 仍然成立。

### 为什么坏

- 监护保护的是对象连续性，不是入口存在感、监控健康感与作者保证。
- 一旦稳定感退回 pointer、监控与作者记忆，团队就会重新容忍：
  - authority drift 被入口感掩盖
  - resume regression 被最终结果掩盖
  - writeback drift 被监控转绿掩盖
- 这会让“stable under watch”直接取代 `per-host authority width + state writeback` 的正式对象链。

### Claude Code 式正解

- 稳定 verdict 应先证明 `authority object`、`per-host authority width` 与 `state writeback` 仍围绕同一对象被 watch，再接受监控上的平静感。

## 3. breadcrumb 复活：旧入口重新充当 authority watch

### 坏解法

- 团队虽然写了 `authority_watch`，但真正值班时只要看到旧路径还能找到、sidecar 还在、恢复入口仍可点击，就默认 authority 仍唯一。

### 为什么坏

- breadcrumb 只能证明入口还在，不能证明 live truth 仍唯一。
- 一旦 breadcrumb 在 watch 阶段复活，团队就会最先误以为：
  - “还能找到就说明还稳定”
  - “入口没丢就说明 authority 没漂”
- 这会把监护从对象级观察退回入口级观察。

### Claude Code 式正解

- authority watch 应先证明 live truth 仍只有一个 `authority object`，再允许 breadcrumb 存在。

## 4. 假冻结：freeze 退回备注与截图

### 坏解法

- 团队虽然写了 `handoff_freeze` 与 `boundary_quarantine`，但真正冻结时只是在工单里写“先别动”，附几张监控截图，却没有正式阻断 later handoff、reconnect 或 side write。

### 为什么坏

- freeze 的本质是让 `freshness gate` 生效，阻断错误 continuation，不是补一条更谨慎的说明。
- 一旦 freeze 退回备注与截图，结构监护执行就会重新退回：
  - handoff without object gate
  - quarantine without boundary
  - reconnect without authority control
- 这会让冻结成为“态度表达”，而不是对象隔离。

### Claude Code 式正解

- freeze 应阻断 authority / writeback / boundary 的错误消费资格，而不是只告诉 later 团队小心。

## 5. telemetry 充当 writeback watch：event stream 冒充 present truth

### 坏解法

- 团队只要看到 PUT 次数多、状态回写频率高、日志很繁荣，就默认 `writeback_watch` 仍然健康，不再检查 `worker_status + external_metadata` 是否仍经由唯一主路径 merge/delete。

### 为什么坏

- writeback watch 保护唯一主写点与 merge/delete 语义，不保护“系统很忙”。
- 一旦稳定感退回 telemetry，团队就会重新容忍：
  - 多路旁写
  - metadata 全量覆盖
  - 局部成功掩盖最终状态错误
- 这会把 `event stream` 的忙碌感错当成 `state writeback` 的 present truth。

### Claude Code 式正解

- writeback watch 应围绕唯一 writer、merge 语义与最终 authority state 对齐，而不是围绕请求数量与日志丰富度。

## 6. anti-zombie 口头化：stale worldview 与 ghost capability 被作者说明替位

### 坏解法

- 团队在值班单里写“stale writer 已处理”，却不交付 `anti_zombie_projection`、`stale_writer_evidence`、`stale_worldview_evidence` 与 `ghost_capability_eviction_state`。

### 为什么坏

- anti-zombie 是持续 witness，不是一次性口头保证。
- 一旦 `stale worldview` 与 `ghost capability` 的驱逐只剩作者说明，团队就会重新把先进性理解成：
  - 文件分层更好看
  - 监控更丰富
  - 作者更有把握
- 这会把真正值钱的 stale-safe、generation fence 与 anti-zombie watch 直接蒸发掉。

### Claude Code 式正解

- 源码先进性应首先被验证为 `stale worldview` 与 `ghost capability` 仍被正式驱逐，而不是作者说明仍然可信。

## 7. 假 reopen：reconnect 冒充对象级重开

### 坏解法

- reopen 看起来存在，实际只是“再连一次”“再试一次恢复”“等监控回绿”，并没有回到上一个可验证的 `authority object / writeback path / generation floor` 组合。

### 为什么坏

- reopen 必须回到正式对象边界，不是重复旧入口。
- 一旦 reconnect 冒充 reopen，later 团队会同时失去：
  - 正式的 recovery boundary
  - 对 stale path 的清退保证
  - 对 writeback 漂移的隔离保证
- 这会把重开退回 retry 循环。

### Claude Code 式正解

- reopen 应先回到 `authority object`、`state writeback` 与 `generation floor`，再操作 reconnect 与 UI。

## 8. 苏格拉底式追问

在你读完这些反例后，先问自己：

1. 我现在消费的是 `authority object` 与 `per-host authority width`，还是 pointer、监控与作者说明。
2. 我现在验证的是对象连续性，还是稳定感。
3. 我现在冻结的是 `freshness gate` 与 boundary，还是只是 later 的操作冲动。
4. 我现在保护的是 `stale worldview / ghost capability` 的驱逐面，还是一套更会解释的故事。
5. 我现在重开的是结构边界，还是旧入口与 retry。

## 9. 一句话总结

真正危险的结构宿主修复监护执行失败，不是没写 watch card，而是写了 watch card 却仍在围绕 authority width 假稳定、假冻结与假 reopen 消费结构世界。
