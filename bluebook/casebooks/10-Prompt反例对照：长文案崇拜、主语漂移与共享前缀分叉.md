# Prompt反例对照：长文案崇拜、主语漂移与共享前缀分叉

这一章不再收集“事故发生后”的样本，而是收集“设计时就埋雷”的同题坏解对照。

它主要回答五个问题：

1. 为什么很多团队明明想抄 Claude Code 的 Prompt 魔力，最后却只抄到了更长文案。
2. 为什么 Prompt 的失败常常先以“看起来更完整”出现，而不是先以“明显错误”出现。
3. 哪些坏解法最容易把 Prompt Constitution 退回成普通提示词。
4. 怎样把同一个问题从坏解法改写成 Claude Code 式正解。
5. 怎样用苏格拉底式追问避免把这章读成“作者讨厌长 prompt”。

## 0. 第一性原理

Prompt 设计里最危险的，不是：

- 没有任何规则

而是：

- 用看起来更全面、更聪明、更稳妥的写法，把制度边界悄悄抹掉

所以这一章的价值不在于说：

- Claude Code 做对了什么

而在于说：

- 同一个问题最容易怎样被做错

## 1. 长文案崇拜 vs Section 宪法

### 坏解法

- 把规则、风格、工具说明、边界提醒、异常处理全塞进一段越来越长的总提示词里，认为“信息越全越强”。

### 为什么坏

- 没有 section 槽位、没有主权链、没有稳定前缀与动态边界之分。
- 任何小改动都会变成全局漂移。
- 最后只能靠感觉调 prompt。

### Claude Code 式正解

- 把 prompt 视为 constitution，而不是长文案。
- 先拆成 section，再定义每个 section 的职责、优先级、失效条件和可观测差异。


## 2. 主语漂移 vs 固定说话人链

### 坏解法

- 一会儿以“系统”口吻下命令，一会儿以“用户”口吻追加偏好，一会儿又让工具说明和任务目标混在一起，靠上下文猜谁在说话。

### 为什么坏

- 主语一漂，优先级就漂。
- 模型会把不同权威源混成同一层。
- 恢复和接手时无法重建当前真相。

### Claude Code 式正解

- 先固定主语，再写内容。
- 系统、宿主、用户、任务对象、子代理分别说什么，必须有明确主权链。


## 3. 共享前缀分叉 vs 主线程唯一前缀生产者

### 坏解法

- 主线程、摘要线程、建议线程、旁路查询、子代理各自拼一份“差不多知道现场”的 prompt，觉得只是轻微冗余。

### 为什么坏

- 同一工作现场被分裂成多份世界模型。
- 缓存命中下降只是表象，更深的问题是不同路径会产出不同判断。
- 最终出现 path parity split。

### Claude Code 式正解

- 主线程负责生产共享前缀，旁路循环默认只读前缀、只加窄差异、只隔离副作用。


## 4. 边界错放 vs Stable Prefix / Dynamic Boundary 分层

### 坏解法

- 把会频繁变化的状态、临时观测、一次性工具结果直接写进稳定提示头部，或把核心规则延后到动态尾部才补。

### 为什么坏

- 前者会不断击穿缓存并污染制度字节。
- 后者会让真正的治理边界变成可丢失附注。
- 系统既贵又不稳。

### Claude Code 式正解

- 稳定前缀只放长期有效、跨回合复用的制度字节。
- 动态边界只承载本轮必须变化的事实。
- 两者之间必须有明确编译边界。


## 5. 非法遗忘 vs 合法遗忘

### 坏解法

- compact 或 summary 时只保留“聊过什么”，把继续行动所需的 ABI、对象状态、未完成约束、工具结果命运一起压掉，事后再让模型脑补。

### 为什么坏

- 这类遗忘不是压缩，而是丢合同。
- 恢复后看似还有摘要，实际已失去继续工作的接口。
- resume 变成半失忆重启。

### Claude Code 式正解

- 允许遗忘叙事冗余，不允许遗忘继续行动所依赖的协议位点。
- 忘掉之后，任务仍必须知道自己是谁、在做什么、下一步受什么约束。


## 6. 接手断裂 vs Handoff Continuity

### 坏解法

- 把 prompt 设计成只对当前生成瞬间有效，不考虑下一位人类、下一次 resume、下一条宿主事件如何接手。

### 为什么坏

- 一旦需要人工批准、远程恢复、跨端继续或排查事故，没有权威快照就只能回翻 transcript。
- 状态优先系统退化回聊天记录考古。

### Claude Code 式正解

- prompt 同时服务模型行动与人类接手。
- 当前真相要能被 snapshot、writeback、external metadata 和摘要共同继承。


## 7. Cache-Aware Assembly 失真 vs 可解释组装

### 坏解法

- 把 prompt 组装视为字符串拼接，谁需要就在哪拼一点；header、betas、工具面、模型差异、query path 临时决定，出了 cache break 再补解释。

### 为什么坏

- assembly path 本身会成为隐性故障源。
- 团队不知道哪些字节应稳定，也不知道哪次 drift 是制度变化、哪次只是拼接噪音。

### Claude Code 式正解

- 把 assembly 当作正式控制面。
- 组装顺序、cache-safe section、latched header、tool schema 可见性、diff 诊断都要进入同一套可解释机制。


## 8. 苏格拉底式追问

在你读完这些对照样例后，先问自己：

1. 我现在写的是 Prompt Constitution，还是越来越长的说明书。
2. 这个系统是否已经只有一个共享前缀生产者。
3. 我删掉的是冗余，还是删掉了继续行动的 ABI。
4. handoff 到来时，后来者能靠结构化真相继续，还是只能回看全文。
5. 我是在修 Prompt，还是在修 Prompt 的制度体。
