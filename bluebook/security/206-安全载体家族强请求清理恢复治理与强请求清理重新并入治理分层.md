# 安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层：为什么artifact-family cleanup stronger-request cleanup-recovery-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reintegration-governor signer

## 1. 为什么在 `205` 之后还必须继续写 `206`

`205-安全载体家族强请求清理连续性治理与强请求清理恢复治理分层` 已经回答了：

`stronger-request cleanup 线即便已经拿到 fresh credential proof、fresh reconnect verdict 与 fresh tool fetch，也还需要 recovery governor 去决定对象是否真的 recovered。`

但如果继续往下追问，
还会碰到另一层很容易被偷写成“既然 reconnect result 已经是 connected，或者 OAuth flow 已经闭环，那它当然已经回到当前世界了”的错觉：

`只要 stronger-request cleanup recovery governor 已经宣布对象 recovered，它就自动拥有了决定这份 recovered truth 是否已经重新进入当前 tool pool、status plane、handler plane 与下一轮 consumer world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/services/mcp/client.ts:2137-2163` 的 `reconnectMcpServerImpl()` 只返回 recovery result，本身不更新 consumer surfaces
2. `src/services/mcp/useManageMCPConnections.ts:293-332` 的 `updateServer()` / `onConnectionAttempt()` 与 `registerElicitationHandler()`
3. `src/cli/print.ts:3148-3204,3255-3288` 的 `mcp_reconnect` / `mcp_toggle`：先写回 `appState` / `dynamicMcpState`，再按 `connected` 决定是否注册 handlers 与返回 success
4. `src/cli/print.ts:3391-3441` 的 auth-complete background path：reconnect 之后还要把 client/tools/commands/resources 写回 `appState` / `dynamicMcpState`
5. `src/cli/print.ts:1471-1485,1610-1627` 的 `buildAllTools()` 与 `buildMcpServerStatuses()`：tool plane 与 status plane 都依赖 reintegration state
6. `src/cli/print.ts:1253-1280,4786-4835` 的 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`
7. `src/tools/McpAuthTool/McpAuthTool.ts:134-165` 的 pseudo-tool -> real tools swap

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup recovery governance` 负责决定 recovered truth 是否成立
2. `stronger-request cleanup reintegration governance` 负责决定这份 recovered truth 何时、以什么方式、进入哪些当前 consumer planes

也就是说：

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经恢复。`

后者才配说：

`这个恢复后的对象现在已经重新并入当前 tool world、status world、event/handler world，并可被后续 consumer 当成 current world 的一部分继续消费。`

所以 `205` 之后必须继续补的一层就是：

`安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`

也就是：

`stronger-request cleanup recovery governor 决定恢复是否成立；stronger-request cleanup reintegration governor 才决定这份恢复真相何时真正并回 current consumer world。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-reintegration-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reintegration manager，
而是在说：

1. repo 已经把 reconnect result 与 `appState` / `dynamicMcpState` 的更新拆开
2. repo 已经把 recovered client 与 real tools/resources 的重回 consumer plane 拆开
3. repo 已经把 `connected` 与 handler re-registration 拆开

第二条：

这里说的 `reintegration governance`

不是在声称：

`一切 recovery success 都必须立即让所有 consumer plane 同步完成。`

当前更稳的说法只能是：

`reintegration governance 首先回答的是 recovered truth 何时分别回到 state plane、tool plane、status plane 与 event plane。`

第三条：

这里也不是把：

`dynamicMcpState`

单独当成全部 reintegration grammar。

真正关键的是：

1. state writeback
2. tool/status read model
3. live handler rebind
4. pseudo-tool replacement

第四条：

这里尤其不能过度声称：

`reintegration governance`

已经等于：

`reprojection governance`

当前更稳的说法只能是：

`reintegration governance 回答 recovered truth 何时进入 current world；reprojection governance 才回答这份 current truth 随后怎样被不同 reader surface 重新讲述。`

第五条：

这里也不能过度声称：

`connected`

就天然等于：

`next-turn visible`

当前更稳的说法只能是：

`raw recovery result 只是 reintegration 输入材料；只有写回 current state、更新 dynamic tool pool 并重挂 handlers 后，consumer 才真正看见它。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类“stronger-request cleanup-recovery-governor signer 仍不等于 stronger-request cleanup-reintegration-governor signer”证据：

1. `reconnectMcpServerImpl()` 只返回 `client/tools/commands/resources`，说明 recovery result 本身不自动进入 consumer planes
2. `onConnectionAttempt()` 要先 `updateServer()` 再在 `connected` 情况下注册 elicitation handler；这说明恢复成立与 UI/event plane 回归仍是两步
3. `mcp_reconnect` / `mcp_toggle` 会显式写回 `appState.mcp`、`dynamicMcpState`，并单独执行 `registerElicitationHandlers()` / `reregisterChannelHandlerAfterReconnect()`；这说明 reconnect success 不自动等于 reintegration complete
4. `buildAllTools()` 与 `buildMcpServerStatuses()` 只从 `appState` / `dynamicMcpState` 读当前世界；这说明 consumer 看到的是 reintegrated truth，不是 raw recovery result
5. `McpAuthTool` 在 OAuth 完成后还要再把 pseudo-tool 换成 real tools；这说明 auth+reconnect 的 recovered proof 仍不自动等于 tool-world reintegration

因此这一章的最短结论是：

`stronger-request cleanup recovery governor 最多能说恢复真相已经成立；stronger-request cleanup reintegration governor 才能说这份恢复真相现在是否已经重新并入当前 consumer world。`

再压成一句：

`recovered，不等于 reintegrated。`

## 4. 第一性原理：recovery governance 回答“恢复是否成立”，reintegration governance 回答“恢复后的真相怎样重新进入当前世界”

从第一性原理看，
强请求清理恢复治理与强请求清理重新并入治理处理的是两个不同主权问题。

stronger-request cleanup recovery governor 回答的是：

1. 新凭据是否已经闭环
2. 新 callback / token exchange 是否完成
3. reconnect result 是否已经回到 `connected`
4. 当前 failure grammar 是否可撤销
5. recovered truth 本身是否成立

stronger-request cleanup reintegration governor 回答的则是：

1. 恢复后的 client 是否已写回 `appState`
2. 恢复后的 tools / commands / resources 是否已并回 current tool pool
3. `dynamicMcpState` 是否已更新，供下一轮 run 读取
4. elicitation handler / channel handler 是否已重新挂回
5. status plane、tool plane、event plane 是否已重新把它视为 current world 的一部分

如果把这两层压成一句“反正已经恢复了”，
系统就会制造五类危险幻觉：

1. connected-means-integrated illusion  
   只要 reconnect 回来 `connected`，就误以为所有 consumer plane 已并回
2. auth-done-means-tool-visible illusion  
   只要 auth flow 结束，就误以为真实工具已经回到模型可见世界
3. state-update-means-handler-ready illusion  
   只要写回状态，就误以为 event plane 也已恢复
4. recovered-means-next-turn-visible illusion  
   只要恢复成立，就误以为下一轮 run 一定已经看见它
5. raw-result-means-current-world illusion  
   只要函数返回了恢复结果，就误以为当前世界已经完成再并入

所以从第一性原理看：

`recovery governance` 管的是恢复真相是否成立；  
`reintegration governance` 管的是这份真相怎样重新进入 current world 的各个消费面。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup recovery governance 已经等于 reintegration governance，为什么 `reconnectMcpServerImpl()` 还只返回 raw result，而不直接写回 `appState`、`dynamicMcpState` 并重挂 handlers？
   因为 “恢复成立” 与 “重新并回当前世界” 根本不是同一句制度。
2. 如果 auth+reconnect 一旦闭环就自动被所有 consumer 看见，为什么 `buildAllTools()` 与 `buildMcpServerStatuses()` 还要依赖 `dynamicMcpState` / `appState`？
   因为 consumer 读取的是 reintegrated truth，不是 raw recovery result。
3. 如果写回状态就已经足够，为什么 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()` 还要单独执行？
   因为 event plane / channel plane 的 re-entry 也是单独的 reintegration authority。

## 5. `reconnectMcpServerImpl()` 先证明：恢复结果本身并不会自动把对象并回当前世界

`client.ts:2137-2163` 很硬。

这里的 `reconnectMcpServerImpl()` 做的是：

1. 清掉 keychain cache
2. 清掉 server cache
3. 重新 `connectToServer()`
4. 若成功则抓 `tools/commands/resources`
5. 返回 result object

但它并没有：

1. 写回 `appState`
2. 更新 `dynamicMcpState`
3. 注册 elicitation handler
4. 重新注册 channel handler

这说明 repo 自己已经把：

`recovery result exists`

和

`recovery result re-enters current world`

拆成了两步。

从技术角度看，这非常值钱。
因为它拒绝让 raw transport-level recovery verdict 越级冒充完整 reintegration。

## 6. `onConnectionAttempt()` 再证明：standard path 上，reintegration 是单独的 projection choreography

`useManageMCPConnections.ts:293-332` 特别值钱。

这里 `onConnectionAttempt()` 在收到 connection result 后，
先做的不是直接宣告一切完成，
而是：

1. `updateServer({ ...client, tools, commands, resources })`
2. 然后只在 `client.type === 'connected'` 时注册真正的 elicitation handler

这条证据很硬。

它说明即便在标准 hook path 里，
repo 也明确承认：

1. recovery result 要先投影成 state update
2. event plane side effect 仍要单独挂回

也就是说，
recovery 和 reintegration 不是同一个动词。

## 7. `print.ts` 证明：control plane 上，reintegration 至少拆成 `appState`、`dynamicMcpState` 与 handler 三层

`print.ts:3148-3204` 的 `mcp_reconnect` 很值钱。

这里在 `reconnectMcpServerImpl()` 之后，
系统继续做了三层 reintegration：

1. 写回 `appState.mcp.clients/tools/commands/resources`
2. 写回 `dynamicMcpState.clients/tools`
3. 只有 `connected` 时才 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`

这说明 reconnect result 并不自动等于：

`LLM next turn 已经看见它`
`status control plane 已经看见它`
`event plane 已经看见它`

`3255-3288` 的 `mcp_toggle` 也是同样制度：
enable 之后不是 “recovered 就算完成”，
而是要继续做回填与 handler rebind。

所以这里最值钱的技术启示不是：

`print 有几段重复代码`

而是：

`repo 明确把恢复后的对象重新并入当前世界做成了单独 choreography`

## 8. `dynamicMcpState`、`buildAllTools()` 与 `buildMcpServerStatuses()` 再证明：reintegration 直接决定下一轮 consumer 会不会真的看见恢复结果

`print.ts:1471-1485` 明确写出：

`buildAllTools()` 会读取 `dynamicMcpState.tools`

而 `1610-1627` 又明确写出：

`buildMcpServerStatuses()` 会读取 `dynamicMcpState.clients`

这说明对 print/headless 世界来说，
某个 recovered object 即便已经存在，
如果没有被正确 reintegrate 进 `dynamicMcpState`，
那么：

1. 下一轮 tool pool 可能仍看不见它
2. status surface 可能仍看不见它

这条证据很关键。
它说明 reintegration 不是 UI 装饰，
而是当前消费世界是否真的收到 recovered truth 的制度入口。

## 9. `registerElicitationHandlers()`、`reregisterChannelHandlerAfterReconnect()` 与 `McpAuthTool` 再证明：live interaction plane 与 real tool world 都有单独的 re-entry grammar

`print.ts:1253-1280` 的 `registerElicitationHandlers()` 很硬。

它只对：

1. `connected`
2. 未注册
3. 非 SDK

client 才重新接回 request handler。

`4786-4835` 的 `reregisterChannelHandlerAfterReconnect()` 也同样严格：

1. 非 `connected` 直接返回
2. 未通过 gate 直接返回
3. 通过后才重新挂回 channel notification handler

而 `McpAuthTool.ts:134-165` 又把 tool plane 的 reintegration 写得很清楚：

1. `clearMcpAuthCache()`
2. `reconnectMcpServerImpl()`
3. 清掉旧 pseudo-tool
4. 把 real tools / commands / resources swap 回 `appState`

这说明 repo 对 reintegration 的理解不是：

`recovered proof exists -> current world naturally absorbs it`

而是：

`不同 consumer plane 要各自完成自己的 re-entry choreography`

这也是 Claude Code 多重安全术真正先进的地方：
它不把“恢复”偷写成单一事件，
而是把恢复后的 world re-entry 再拆成 state、tool、status、event 四个相互校验的面。

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 线只要补出 recovery governance，就已经足够成熟。`

而是：

`repo 在 reconnectMcpServerImpl() 的 raw recovery result、useManageMCPConnections.ts 的 state projection + handler attach、print.ts 对 appState / dynamicMcpState 的回填、buildAllTools() / buildMcpServerStatuses() 的 consumer read model、reregisterChannelHandlerAfterReconnect() 的 live interaction re-entry，以及 McpAuthTool 的 pseudo-tool -> real tools swap 上，已经明确展示了 reintegration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-recovery-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

因此：

`stronger-request cleanup recovery governance 能回答“它已经恢复了吗”；stronger-request cleanup reintegration governance 才能回答“它现在已经重新成为当前世界的一部分了吗”。`
