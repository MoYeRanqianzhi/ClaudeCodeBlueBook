# 宿主账本投影速查表：不同宿主能看到哪些账、缺哪些账、最容易误读什么

这一页服务于 [36-安全账本投影原则：为什么不同宿主只能看到账本子集，而不能冒充完整安全控制台](../36-%E5%AE%89%E5%85%A8%E8%B4%A6%E6%9C%AC%E6%8A%95%E5%BD%B1%E5%8E%9F%E5%88%99%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E4%B8%8D%E5%90%8C%E5%AE%BF%E4%B8%BB%E5%8F%AA%E8%83%BD%E7%9C%8B%E5%88%B0%E8%B4%A6%E6%9C%AC%E5%AD%90%E9%9B%86%EF%BC%8C%E8%80%8C%E4%B8%8D%E8%83%BD%E5%86%92%E5%85%85%E5%AE%8C%E6%95%B4%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%8F%B0.md)。

如果 `36` 的长文解释的是为什么不同宿主只能看到账本子集，  
那么这一页只做一件事：

`把宿主类型、可见账本、可写账本、明显缺口与典型误读压成一张矩阵。`

## 2. 宿主账本投影矩阵

| 宿主类型 | 主要可见账本 | 主要可写账本 | 明显缺口 | 最危险的误读 |
| --- | --- | --- | --- | --- |
| 默认 `StructuredIO` 宿主 | 基础消息面、基础控制面 | outbound 控制/消息写口 | 默认没有恢复账回读，也没有 internal-event 持久面 | 把“接上协议”误读成“已有完整控制台” |
| `RemoteIO + CCR v2` | 语义账镜像、复制账、可见账、恢复账读写面都更完整 | `reportState(...)`、`reportMetadata(...)`、`writeEvent(...)`、`writeInternalEvent(...)` | 仍受 CCR v2、transport 和初始化成功条件限制 | 把“较完整投影”误读成“任何时候都全知全能” |
| `DirectConnectSessionManager` | 权限请求面 | permission response、interrupt | 没有完整恢复账、复制账、控制全集 | 把权限型窄宿主误读成完整宿主 |
| `RemoteSessionManager` | 远程会话消息面、权限请求面 | send message、permission response、interrupt | control 子类型很窄，缺完整恢复账和完整复制账 | 把“能发消息”误读成“看到了全部安全状态” |
| `remoteBridgeCore` | 更宽的 SDK message / control 投影 | `writeSdkMessages(...)`、control req/resp/cancel、result | 仍受 recovery window、drop、去重和状态纪律约束 | 把“桥接更宽”误读成“桥接无损且无条件完整” |

## 3. 最重要的三条判断

| 判断 | 为什么重要 |
| --- | --- |
| 协议全集不等于宿主全集 | schema 很宽，但具体宿主可能只实现很窄子集 |
| 更宽投影不等于完整控制台 | bridge / remote IO 仍然有条件、窗口和 drop 纪律 |
| 窄宿主不等于差宿主 | 很多窄子集恰恰是更稳定、更安全的接入方式 |

## 4. 最短用法

- 先确认当前宿主属于哪一类投影
- 再确认它看到的是哪几本账，而不是默认它看全了
- 如果某个宿主缺恢复账或缺完整 control 子集，就不要让它承担完整解释责任

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是宿主能力有差异，  
而是：

`明明只拿到账本子集，却还按完整控制台的口气给出判断。`
