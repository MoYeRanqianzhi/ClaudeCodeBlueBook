# 安全恢复留痕纪律速查表：失败场景、留痕对象、再进入入口与清除条件

## 1. 这一页服务于什么

这一页服务于 [56-安全恢复留痕纪律：为什么auto-close失败、transient failure与人工确认挂起都必须保留可重试痕迹](../56-%E5%AE%89%E5%85%A8%E6%81%A2%E5%A4%8D%E7%95%99%E7%97%95%E7%BA%AA%E5%BE%8B%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88auto-close%E5%A4%B1%E8%B4%A5%E3%80%81transient%20failure%E4%B8%8E%E4%BA%BA%E5%B7%A5%E7%A1%AE%E8%AE%A4%E6%8C%82%E8%B5%B7%E9%83%BD%E5%BF%85%E9%A1%BB%E4%BF%9D%E7%95%99%E5%8F%AF%E9%87%8D%E8%AF%95%E7%97%95%E8%BF%B9.md)。

如果 `56` 的长文解释的是：

`为什么未闭环恢复必须保留可重试、可回访、可解释的痕迹，`

那么这一页只做一件事：

`把失败场景、留痕对象、再进入入口与清除条件压成一张可直接用于评审的矩阵。`

## 2. 恢复留痕矩阵

| 失败场景 | 留痕对象 | 再进入入口 | 何时允许清除 | 关键证据 |
| --- | --- | --- | --- | --- |
| plugin auto-refresh 失败 | `plugins.needsRefresh=true` | `/reload-plugins` | 只有 `refreshActivePlugins()` 真正消费并把 `needsRefresh=false` 后 | `PluginInstallationManager.ts:146-165`、`refresh.ts:123-138` |
| plugin 更新待应用 | `needsRefresh` + `plugin-reload-pending` 通知 | `/reload-plugins` | 不能靠自动刷新或通知超时清除，必须由 `/reload-plugins` 关环 | `useManagePlugins.ts:288-303` |
| plugin hooks 局部加载失败 | `errors` 合并保留、`hook_load_failed` 计入 `error_count` | plugin 刷新 / 诊断路径 | 只有后续刷新成功并错误计数回落后 | `refresh.ts:149-187` |
| MCP automatic reconnection 进行中 | `pending`、`reconnectAttempt`、`maxReconnectAttempts` | 系统自动 backoff；必要时用户回 `/mcp` | 成功收敛到 connected 或最终落成 failed 后再转态 | `useManageMCPConnections.ts:387-460` |
| MCP automatic reconnection 放弃 | server `type='failed'` | `/mcp` | 只有新的 reconnect 成功并能力重新可见后 | `useManageMCPConnections.ts:417-442` |
| bridge transient reconnect failure | `pointer` 保留、重试提示保留 | 重跑 bridge / `/remote-control` 路径 | 只有后续 reconnect 真正成功或被证明 fatal 后 | `bridgeMain.ts:2524-2539` |
| crash-recovery / 长会话风险 | bridge pointer 立即写入并定时刷新 | `--continue` / resume / `/remote-control` 相关恢复链 | 在正常 archive+deregister 或 fatal clear 时清除 | `bridgeMain.ts:2700-2728` |
| 表层通知未被正式 invalidated | notification key 仍在 current/queue | 对应默认修复入口 | 只能靠 `invalidates`、timeout、或显式 `removeNotification` | `notifications.tsx:78-116,193-212` |

## 3. 最短判断公式

评审任一恢复失败场景时，先问四句：

1. 当前未闭环恢复有没有被落成显式状态对象
2. 这个对象是不是比 toast 更持久
3. 它有没有直接绑定再次进入入口
4. 没有哪个新事实到来之前绝不能清除

## 4. 最常见的四类留痕错误

| 留痕错误 | 会造成什么问题 |
| --- | --- |
| auto-close 失败后不回退到挂起态 | 用户看不到问题仍未闭环 |
| transient failure 不保留 pointer / pending / retry count | 用户失去再次进入恢复链的锚点 |
| 只在通知层留痕 | 通知一消失，问题就像被系统忘了 |
| 先清痕迹再等后续修复 | 未解决问题被伪装成已结束 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是失败本身，  
而是：

`失败已经发生、恢复还没闭环，但系统却没有留下任何仍可被再次进入和再次解释的痕迹。`
