## 2. 恢复跳转纪律矩阵

| 状态 / 中间态 | 当前缺口归属 | 默认跳转 | 次级跳转 | 禁止歧义 | 关键证据 |
| --- | --- | --- | --- | --- | --- |
| flagged plugins | plugin 风险治理 | `/plugins` | 无，除非进入插件页后再细分 | 不能和 `/reload-plugins` 并列成两个同级默认入口 | `useManagePlugins.ts:62-67` |
| plugin activation pending | plugin 激活 / 热刷新 | `/reload-plugins` | `/plugins` 仅作为次级治理入口 | 不能把“要刷新”和“要治理”压成同一条默认路径 | `useManagePlugins.ts:294-300` |
| MCP failed / connector unavailable | MCP 连接域 | `/mcp` | `/login` 仅在 `/mcp` 内确认是 auth bottleneck 时再显式升级 | 不能和 `/status` 并列成主入口 | `useMcpConnectivityStatus.tsx:36-63` |
| MCP needs-auth | MCP auth within MCP domain | `/mcp` | `/login` 作为领域页内动作，不应在外层抢主入口 | 不能直接把所有 MCP auth 问题都从通知层改路由到 `/login` | `useMcpConnectivityStatus.tsx:50-63` |
| channels auth bottleneck | 全局认证缺口 | `/login` | `/status` 仅用于补充诊断 | 不能和 `/mcp`、`/status` 并列给用户盲选 | `ChannelsNotice.tsx:73` |
| global not logged in | 全局认证缺口 | `/login` | 无 | 不能提示“去看看别的设置页” | `PromptInput/Notifications.tsx:306-309` |
| IDE plugin not connected | IDE 诊断不清 / 先看总状态 | `/status` | 后续再进入对应 IDE / bridge 细项 | 不能直接把用户送去 unrelated repair command | `useIDEStatusIndicator.tsx:119-123` |
| IDE extension install failed | IDE 诊断与安装失败 | `/status` | 后续在状态页内再细分修复 | 不能直接把安装失败和登录/MCP/plugin 路径混为一谈 | `useIDEStatusIndicator.tsx:147-152` |
| bridge attach / enable / disconnect | Remote Control / bridge 域 | `/remote-control` | bridge 详情或会话详情作为次级入口 | 不能把 bridge 半恢复状态默认送去 `/status` 或 `/mcp` | `RemoteCallout.tsx:36-42`、`bridgeMain.ts:2125-2128` |
| `mirror-only` / outbound-only | bridge 入站控制缺口 | `/remote-control` | bridge 详情作为解释层 | 不能用 generic “active” 文案但不给 bridge 专属入口 | `replBridgeTransport.ts:142-147,336-345` |
| `write-ready / read-blind` | bridge 读链缺口 | bridge 详情 / `/remote-control` | `/status` 仅作次级诊断 | 不能默认丢给 unrelated 领域页 | `replBridgeTransport.ts:289-299`、`bridgeStatusUtil.ts:123-140` |
| `restored-but-not-fresh` | state freshness 缺口 | `/status` | 相关领域页仅在状态页内再细分 | 不能跳过状态页直接宣称“去继续操作” | `ccrClient.ts:474-522`、`print.ts:5048-5058` |

## 3. 最短判断公式

给任一半恢复状态设计默认跳转时，先问四句：

1. 当前缺口真正属于认证、MCP、plugin、bridge，还是跨域状态 freshness
2. 现在最短路径是先诊断，还是直接修
3. 有没有第二条入口虽然也相关，但会让 dominant path 失去默认性
4. 如果把两条命令并列展示，用户最可能误修到哪里

## 4. 最常见的四类路由歧义错误

| 歧义错误 | 会造成什么问题 |
| --- | --- |
| 把 plugin 风险和 plugin 激活都指向同一入口 | 用户会把治理问题误当成刷新问题 |
| 把 MCP auth 与全局 auth 都直接扔给 `/login` | 用户绕开了 MCP 领域页，丢失 server 级上下文 |
| 把 bridge 半恢复统一送去 `/status` | 用户看见摘要，却进不到真正的 bridge 修复入口 |
| 同时提示 `/status`、`/mcp`、`/login` 却不给主次 | 默认路径失效，用户开始盲选 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是默认跳转太少，  
而是：

`系统明明知道当前缺口属于哪一条修复链，却仍把多个“看起来都有关”的命令并列抛给用户自己猜。`
