# 安全未知语义速查表：哪些情况必须显式说不知道、不支持或当前不可判定

如果 `38` 的长文解释的是为什么安全控制面必须把“不知道”做成显式语义，

`把“不支持”“没有这本账”“当前不可判定”压成一张矩阵。`

## 2. 未知语义矩阵

| 场景 | 系统当前真正缺什么 | 正确做法 | 错误做法 | 如果硬猜会坏什么 |
| --- | --- | --- | --- | --- |
| 宿主不支持某个 control subtype | capability | 显式回 unsupported / error | 静默等待、装作会处理 | 上游 hang、错误期待、伪解释 |
| 默认宿主没有恢复账 / internal-event 面 | ledger | 显式 `null`、no-op、零 pending | 伪造默认恢复成功态 | 把“没接上”误读成“已经没问题” |
| `RemoteIO` 前置条件未成立 | precondition | fail loudly、停止伪完整模式 | 偷偷退成“差不多可用” | 用户与宿主误以为完整能力仍在 |
| bridge 处于 recovery window | timing | 显式 drop、收窄解释权 | 继续假装发送/解释完整成立 | 在不可判定窗口里输出假确定性 |
| authoritative signal 尚未到来 | authority boundary | 暂不授予完整解释权 | 过早宣布 turn over / state settled | 近似状态冒充正式裁决 |

## 3. 最短判断公式

遇到安全状态空洞时，先问三句：

1. 现在缺的是能力、账本、条件，还是时序基底
2. 这个缺口有没有被显式表达出来
3. 如果没有显式表达，当前结论是不是其实已经在硬猜

## 4. 最常见的三类伪解释

| 伪解释 | 为什么危险 |
| --- | --- |
| “没报错就是支持” | unsupported 若未显式回错，就会被误读成可能支持 |
| “空就是正常” | `null` / `no-op` 可能表达的是“没有这本账”，不是“已经完成” |
| “暂时看不到但应该差不多” | recovery / init / precondition 窗口里最怕这种近似想象 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是不知道，  
而是：

`明明不知道，却还输出了像知道一样的结论。`
