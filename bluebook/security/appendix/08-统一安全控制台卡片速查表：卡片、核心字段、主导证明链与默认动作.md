# 统一安全控制台卡片速查表：卡片、核心字段、主导证明链与默认动作

## 1. 这一页服务于什么

这一页服务于 [24-统一安全控制台卡片设计：如何把字段、证明链与最短动作做成真正可用的界面](../24-%E7%BB%9F%E4%B8%80%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%8F%B0%E5%8D%A1%E7%89%87%E8%AE%BE%E8%AE%A1%EF%BC%9A%E5%A6%82%E4%BD%95%E6%8A%8A%E5%AD%97%E6%AE%B5%E3%80%81%E8%AF%81%E6%98%8E%E9%93%BE%E4%B8%8E%E6%9C%80%E7%9F%AD%E5%8A%A8%E4%BD%9C%E5%81%9A%E6%88%90%E7%9C%9F%E6%AD%A3%E5%8F%AF%E7%94%A8%E7%9A%84%E7%95%8C%E9%9D%A2.md)。

如果 `24` 回答的是：

`统一安全控制台最少应该由哪几张卡组成？`

那么这一页回答的是：

`每张卡最少要带哪些字段，它主要服务哪条证明链，又默认应该给出什么动作。`

## 2. 卡片矩阵

| 卡片 | 核心字段 | 主导证明链 | 最应该回答的问题 | 默认动作类型 |
| --- | --- | --- | --- | --- |
| 主体与资格卡 | account type、token source、auth status、profile scope、organization UUID | 主体链 | “我现在到底是不是一个可被正确识别并可继续授予资格的主体？” | 重新登录、刷新身份、检查 scope |
| 主权与来源卡 | active setting sources、policy origin、managed-only/plugin-only、host-managed env | 主权链 | “谁在定义我现在所处的运行现实？” | 打开来源详情、跳转配置源、联系管理员 |
| 仲裁与权限卡 | permission mode、decision behavior、decisionReason.type、rule source、classifier、safetyCheck | 授权链 | “为什么这次动作会被 allow / ask / deny？” | 查看原因详情、调整模式、修正规则 |
| 包络与执行壳卡 | sandbox enabled、fs/network restrictions、excluded commands、unix sockets | 包络链 | “即便允许，这个动作实际还能扩散到哪里？” | 查看包络详情、调整可写目录、调整网络限制 |
| 外部能力卡 | MCP status、MCP scope、tool annotations、bridge session state、plugin marketplace state | 外部能力链 | “当前开放的是哪一种外部能力语义？” | 跳转 MCP / bridge / plugin 深度页、重新连接、最短修复 |
| 增量审批卡 | dangerous settings changed、approval required、approval result、approval subject | 增量审批链 | “最近有什么边界变化要求我重新确认？” | 接受 / 拒绝 / 查看变更详情 |
| 断链诊断卡 | current leading boundary、broken proof chain、shortest corrective action、responsible party | 派生解释层 | “所以现在到底是哪条链断了，我最短该做什么？” | 直接执行最短动作、跳到负责卡片 |

## 3. 设计时最容易犯的三个错误

1. 把七张卡做成七个孤岛，而不是让断链诊断卡把它们重新串起来。
2. 只展示原始字段，不展示派生结论，例如“当前主导边界”“当前断裂证明链”。
3. 把所有动作都做成“查看更多”，而不是给出真正可执行的最短动作。

## 4. 最短用法

- 想做控制台首页：先实现主体卡、仲裁卡、断链诊断卡
- 想做深度页：再补主权卡、包络卡、外部能力卡
- 想做事件流：最后补增量审批卡

## 5. 最终用途

这张表的作用，是把 `24` 的长文设计压成可直接拿去做信息架构、卡片拆分和迭代优先级排序的实现速查表。
