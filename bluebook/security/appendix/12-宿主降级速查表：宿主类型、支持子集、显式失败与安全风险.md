# 宿主降级速查表：宿主类型、支持子集、显式失败与安全风险

那么这一页只做一件事：

`把不同宿主的降级方式压成一张宿主对照矩阵。`

## 2. 宿主降级矩阵

| 宿主类型 | 当前支持子集 | 对不支持能力的处理 | 当前优点 | 当前风险 | 最低改进方向 |
| --- | --- | --- | --- | --- | --- |
| bridge / REPL bridge | 比 direct connect 更宽，支持 `initialize`、`set_model`、`set_max_thinking_tokens`、`set_permission_mode`、`interrupt` 等控制子集 | 未知 control subtype 显式 error；但某些 ingress path 对 non-user message 直接忽略 | control path 更成熟，显式失败语义较好 | 观察面仍可能遗漏非 user 对象，子集边界不总是前台可见 | 把“哪些对象不可见”显式做成 degraded 状态 |
| direct connect | 极窄控制面，核心只处理 `can_use_tool`；主动动作主要是 `interrupt` 和 permission response | 对其他 control subtype 显式 error | 非常诚实，边界清晰，不会假装支持协议全集 | 宽度很窄，用户容易把窄宿主误认为系统本体 | 前台显式暴露“仅支持权限与中断”的宿主身份 |
| remote session manager | 极窄控制面，主要处理 `can_use_tool`、`control_cancel_request`、`interrupt` | 不支持的 control subtype 显式 error | permission 路径明确，降级相对诚实 | 很容易被误解成“远程完整版宿主” | 显式声明这是会话管理子集，不是完整控制台宿主 |
| `sdkMessageAdapter` 路径上的 REPL 适配层 | 会投影一部分 message，但会把 `status` 文本化，并忽略 `auth_status`、`tool_use_summary`、`rate_limit_event` 等 | 当前主要是静默忽略或仅调试日志 | 适配简单、显示轻 | 这是最危险的隐性子集类型，容易把宿主盲区伪装成系统事实 | 先声明对象缺失，再决定是否压成文本 |

## 3. 最重要的两类区别

| 区别维度 | 相对更好 | 相对更危险 |
| --- | --- | --- |
| 动作路径 | 显式 error 的窄宿主 | 假成功或不声明 unsupported |
| 观察路径 | 保留对象但投影变窄 | 静默忽略对象或直接文本化后不声明损失 |

## 4. 这张表最重要的结论

宿主并不需要都一样强。  
真正要统一的，不是宿主宽度，  
而是：

`宿主对自己宽度的诚实程度。`

因此判断一个宿主是否“安全”，  
最先不该问：

`它支持多少功能？`

而该先问：

`它有没有把自己的降级边界显式说出来？`

## 5. 最短用法

- 先判定当前宿主属于哪一类子集
- 再看它对不支持能力是显式 error、显式 degraded，还是静默忽略
- 如果是静默忽略，优先把它视为高风险宿主，不要把它输出的“正常”当成系统真相

## 6. 一条硬结论

在多宿主控制面里，  
最该被优先修复的不是最窄的宿主，  
而是：

`那些已经发生降级，却没有把降级显式写出来的宿主。`
