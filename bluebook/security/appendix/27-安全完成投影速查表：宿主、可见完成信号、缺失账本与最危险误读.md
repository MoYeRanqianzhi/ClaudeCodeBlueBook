## 2. 完成投影矩阵

| 宿主 / 路径 | 它能看到的完成信号 | 它明确缺什么 | 最安全的完成说法上限 | 最危险的误读 | 关键证据 |
| --- | --- | --- | --- | --- | --- |
| SDK 流消费者 | `session_state_changed(idle)`、普通 system/result 流 | 更底层 flush / 恢复 / 复制账细节 | “主 turn 的 authoritative 结束事件已到达” | 把事件流视图误当成全部完成账本 | `coreSchemas.ts:1738-1746`、`sdkEventQueue.ts:56-61` |
| RemoteIO / CCR 宿主 | `worker_status`、`requires_action_details`、`external_metadata`、`restoredWorkerState`、`flushInternalEvents()` | 仍不是所有内部事实的无限解释权 | “我看到的是更厚的完成复制态与恢复态投影” | 把更厚投影误说成完整内部真相 | `remoteIO.ts:126-167`、`remoteIO.ts:217-223`、`ccrClient.ts:644-662` |
| 默认 StructuredIO 宿主 | 常规输入输出流 | `restoredWorkerState = null`、`flushInternalEvents() = no-op`、`internalEventsPending = 0` | “我没有恢复账和 internal-event flush 视角” | 假装自己拥有和 RemoteIO 同等完成视角 | `structuredIO.ts:139-196` |
| bridge 正常窗口 | `reportState('requires_action'/'running'/'idle')` 等较厚状态投影 | 仍受 transport / recovery 条件限制 | “当前窗口下我能投影较完整完成状态” | 把 bridge 正常窗口能力误写成恒定能力 | `remoteBridgeCore.ts:824-878` |
| bridge recovery 窗口 | 只能看到收缩后的状态面，且会主动 drop control/result | 稳定 result/control completion 投影 | “当前完成解释权正在收窄，不应给满结论” | 在 recovery 中继续宣布已完成 | `remoteBridgeCore.ts:824-878` |
| DirectConnectManager | `can_use_tool` 审批闭环 + 转发 SDK messages | 广泛 control subtype 支持、完整 host 级 completion control | “我只在窄审批路径里消费完成信号” | 把自己当完整 SDK host | `directConnectManager.ts:81-99` |
| RemoteSessionManager | `can_use_tool` 审批闭环 + 远程消息转发 | 广泛 control subtype 支持、完整 completion control 面 | “我只持有窄控制子集上的完成投影” | 把远程 session 管理器误当完整完成解释器 | `RemoteSessionManager.ts:192-213` |

## 3. 最短判断公式

看到某个宿主在说“已经完成”时，先问四句：

1. 它看到的是 authoritative signer 还是 signer 的投影
2. 它缺不缺恢复账、flush 视角或更厚 metadata
3. 它的 control surface 是全集还是窄子集
4. 当前窗口是否正处于 recovery / 降级态

## 4. 最常见的四类投影误读

| 误读方式 | 会造成什么问题 |
| --- | --- |
| 把 SDK 事件流视图当成全部完成账本 | 忽略了更底层 flush / 恢复边界 |
| 把 RemoteIO 的厚投影当成无限解释权 | 复制态与 authoritative 真相被混写 |
| 把 StructuredIO 的默认空视角当成“也许一样全” | 宿主输出过满完成结论 |
| 把窄 control 宿主当成完整 host | 局部审批闭环被冒充成全链完成解释 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是宿主看到的太少，  
而是：

`宿主明明只看到了完成签字链的局部投影，却还按完整完成链在说话。`
