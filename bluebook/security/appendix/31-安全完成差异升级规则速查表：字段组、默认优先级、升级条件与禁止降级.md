# 安全完成差异升级规则速查表：字段组、默认优先级、升级条件与禁止降级

## 1. 这一页服务于什么

这一页服务于 [47-安全完成差异升级规则：哪些宿主盲区必须抢占界面，哪些只能下钻显示](../47-%E5%AE%89%E5%85%A8%E5%AE%8C%E6%88%90%E5%B7%AE%E5%BC%82%E5%8D%87%E7%BA%A7%E8%A7%84%E5%88%99%EF%BC%9A%E5%93%AA%E4%BA%9B%E5%AE%BF%E4%B8%BB%E7%9B%B2%E5%8C%BA%E5%BF%85%E9%A1%BB%E6%8A%A2%E5%8D%A0%E7%95%8C%E9%9D%A2%EF%BC%8C%E5%93%AA%E4%BA%9B%E5%8F%AA%E8%83%BD%E4%B8%8B%E9%92%BB%E6%98%BE%E7%A4%BA.md)。

如果 `47` 的长文解释的是哪些宿主盲区必须抢占界面，哪些只能下钻显示，  
那么这一页只做一件事：

`把字段组、默认优先级、升级条件和禁止降级规则压成一张可直接用于评审的矩阵。`

## 2. 升级规则矩阵

| 字段组 / 信号 | 默认优先级 | 升级到更高优先级的条件 | 绝不应降到哪一层 | 关键证据 |
| --- | --- | --- | --- | --- |
| `completion_claim_ceiling` | `high` | 当前 ceiling 被动态拉低并正在推翻用户刚得出的结论时升 `immediate` | `low` | `47` 主线归纳 |
| `recovery_window_open` / projection suspended | `high` | 当前 result/control 投影已暂停时升 `immediate` | `medium` / `low` | `remoteBridgeCore.ts:824-878` |
| `host_kind` | `medium` | 当当前宿主与用户默认假定的宿主类别冲突时升 `high` | `low` | `45`、`46` 主线归纳 |
| `projection_grade` | `medium` | 从厚投影降到薄投影且会误导完成判断时升 `high` | `low` | `43`、`45` 主线归纳 |
| `missing_ledgers[]` | `medium` | 缺失账本直接导致当前动作或完成解释失效时升 `high` | `low` | `structuredIO.ts:139-196`、`directConnectManager.ts:81-99` |
| `write_ready/read_ready` 分裂 | `medium` | 表面仍显示 connected/active 且用户正可能误解为读写都可用时升 `high` | `low` | `replBridgeTransport.ts:289-299` |
| `supported_control_subtypes[]` | `medium` | 用户当前正要执行宿主不支持的动作时升 `high` | `low` | `directConnectManager.ts:81-99`、`RemoteSessionManager.ts:192-213` |
| MCP failed / needs-auth | `medium` | 当前失败正在阻断用户下一步动作链时可升 `high` | `low` | `useMcpConnectivityStatus.tsx:36-63` |
| IDE disconnected | `medium` | 若当前任务强依赖 IDE selection / IDE bridge 时可升 `high` | `low` | `useIDEStatusIndicator.tsx:87-92` |
| plugin reload pending | `low` | 若当前插件变化正在影响控制面可见度时可升 `medium` | `immediate` | `useManagePlugins.ts:295-300` |
| plugin delisted flagged / channels blocked | `high` | 若正实时推翻当前完成结论可升 `immediate`，否则维持 `high` | `low` | `useManagePlugins.ts:62-67`、`useManageMCPConnections.ts:604-608` |

## 3. 最短判断公式

做优先级评审时，先问四句：

1. 这个盲区是不是正在推翻用户刚刚获得的确定性
2. 它是不是已经改写了用户下一步动作路径
3. 它是显著退化还是仅仅是维护提醒
4. 如果把它再往下压一级，最先会误导哪个动作

## 4. 最常见的四类降级错误

| 降级错误 | 会造成什么问题 |
| --- | --- |
| 把 recovery 窗口压成普通 medium 提示 | 用户继续相信当前完成解释权仍完整 |
| 把 `completion_claim_ceiling` 压成低优先级文案 | 过满结论先赢得注意力 |
| 把 `missing_ledgers[]` 压成低优先级维护提示 | 用户不知道当前动作为什么实际不可解释 |
| 把 control subtype 缺口压成静态详情 | 用户会继续尝试宿主根本不支持的动作 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是不知道该显示什么，  
而是：

`明明知道某个宿主盲区会立刻误导下一步动作，却仍把它降成了不会打断用户的低优先级提示。`
