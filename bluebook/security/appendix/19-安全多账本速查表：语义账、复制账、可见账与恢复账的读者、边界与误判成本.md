# 安全多账本速查表：语义账、复制账、可见账与恢复账的读者、边界与误判成本

## 1. 这一页服务于什么

这一页服务于 [35-安全多账本原则：为什么语义、复制、可见与恢复不能共用同一本账](../35-%E5%AE%89%E5%85%A8%E5%A4%9A%E8%B4%A6%E6%9C%AC%E5%8E%9F%E5%88%99%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E8%AF%AD%E4%B9%89%E3%80%81%E5%A4%8D%E5%88%B6%E3%80%81%E5%8F%AF%E8%A7%81%E4%B8%8E%E6%81%A2%E5%A4%8D%E4%B8%8D%E8%83%BD%E5%85%B1%E7%94%A8%E5%90%8C%E4%B8%80%E6%9C%AC%E8%B4%A6.md)。

如果 `35` 的长文解释的是为什么 Claude Code 必须维护多本职责不同的账，  
那么这一页只做一件事：

`把每一本账的读者、写入口、提交边界和误判成本压成一张矩阵。`

## 2. 多账本矩阵

| 账本 | 它主要回答什么 | 主要读者 | 主要写入口 | 提交边界 | 最危险的误读 |
| --- | --- | --- | --- | --- | --- |
| 语义账 | 系统当前在语义上是什么状态 | 本地状态机、权限仲裁链 | `notifySessionStateChanged(...)` | `currentState` 与语义 listener 已更新 | 把镜像延迟误读成语义尚未成立 |
| 复制账 | 远端 worker 面当前拥有什么状态 | CCR worker 面、跨端查询、远端恢复入口 | `reportState(...)`、`reportMetadata(...)`、`WorkerStateUploader` | uploader send success | 把 enqueue 或 retry 中误读成远端已同步 |
| 可见账 | 外部消费者现在应该看见什么 | 前端 UI、SSE 消费者、SDK 宿主 | `writeEvent(...)`、`sdkEventQueue`、`heldBackResult` 路径 | flush、hold-back 释放、输出顺序成立 | 把消息已生成误读成用户已看见 |
| 恢复账 | 下一次 resume 是否能把状态重建回来 | session resume、subagent transcript hydrate | `writeInternalEvent(...)`、`flushInternalEvents()`、`hydrateFromCCRv2InternalEvents(...)` | internal events 已落并可被 hydrate | 把前端已见误读成恢复一定可重建 |

## 3. 为什么 UI 文案不算正式账本

| 看起来像账本的东西 | 为什么不算 |
| --- | --- |
| status line / footer / pill 文案 | 它们是投影层，不是正式写入面 |
| 临时摘要或颜色状态 | 它们适合阅读，不适合恢复和仲裁 |
| 宿主本地压缩后的提示语 | 它们会丢字段，也会滞后于源账本 |

## 4. 最重要的顺序口诀

可以把这张表压成一句话：

`先分清读者，再分清账本；先分清账本，再讨论“已生效”。`

## 5. 最短用法

- 如果争论的是“系统当前到底算什么状态”，优先看语义账
- 如果争论的是“远端到底有没有拿到”，优先看复制账
- 如果争论的是“用户现在到底看见了没有”，优先看可见账
- 如果争论的是“下次 resume 能不能还原”，优先看恢复账

## 6. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是账本多，  
而是：

`把不同读者、不同边界、不同失败后果的记录面，误当成同一本账。`
