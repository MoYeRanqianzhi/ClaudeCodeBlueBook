| recovery window 打开 / projection suspended | `recovery_window_open`、remote projection 暂停 | 立刻降低 `completion_claim_ceiling`，在完成卡和资格卡显式显示“当前解释权暂挂” | `immediate` | 禁用或替换依赖完整回读与完整完成解释的动作 | recovery 关闭后仍需一次新鲜 authoritative 回读，才能恢复主结论并撤销旧告警 | `useReplBridge.tsx:102-111`、`notifications.tsx:78-116` |

## 3. 一条最短路由公式

评审任一宿主盲区触发器时，按下面五句判断：

1. 它有没有立刻改写 `completion_claim_ceiling`
2. 它有没有新增 `missing_ledgers[]` 或放大 host blindspot
3. 它是否要求动作入口立刻收口，而不是等点击后失败
4. 它是不是只该抢当前通知槽位，而不该独占全部解释
5. 它的清除条件是不是绑定了 fresh 回读，而不是表面恢复

## 4. 最常见的四类交互路由错误

| 错误 | 后果 |
| --- | --- |
| 只发通知，不先改主卡结论 | 用户仍会先相信更醒目的旧结论 |
| 只改卡片，不收动作 | 用户会继续点击旧路径，把解释分裂转成行为误导 |
| 盲区恢复时先清通知、后等回读 | 系统会比真实恢复更乐观 |
| 把 connected 当成动作授权，而不是把 ceiling 当成动作授权 | 局部就绪会被误读成全链就绪 |

## 5. 一条硬结论

对 Claude Code 这类多宿主安全控制面来说，  
最危险的不是缺一张卡或少一个字段，  
而是：

`同一个宿主盲区已经改变了完成上限和动作路径，但卡片、通知、动作和清除条件却没有在同一时刻一起改口。`
