# 安全真相仲裁速查表：冲突场景、优先级、胜出真相与禁止误读

## 1. 这一页服务于什么

这一页服务于 [31-安全真相仲裁：当实时事件、复制状态、本地状态与界面摘要冲突时谁说了算](../31-%E5%AE%89%E5%85%A8%E7%9C%9F%E7%9B%B8%E4%BB%B2%E8%A3%81%EF%BC%9A%E5%BD%93%E5%AE%9E%E6%97%B6%E4%BA%8B%E4%BB%B6%E3%80%81%E5%A4%8D%E5%88%B6%E7%8A%B6%E6%80%81%E3%80%81%E6%9C%AC%E5%9C%B0%E7%8A%B6%E6%80%81%E4%B8%8E%E7%95%8C%E9%9D%A2%E6%91%98%E8%A6%81%E5%86%B2%E7%AA%81%E6%97%B6%E8%B0%81%E8%AF%B4%E4%BA%86%E7%AE%97.md)。

如果 `31` 的长文解释的是多层真相冲突时谁说了算，  
那么这一页只做一件事：

`把最关键的仲裁场景压成一张优先级矩阵。`

## 2. 真相仲裁矩阵

| 冲突场景 | 胜出真相 | 为什么它优先 | 败方为什么不能赢 | 禁止误读 |
| --- | --- | --- | --- | --- |
| 恢复时 `restoredWorkerState` 与本地 fresh default 冲突 | 恢复到的复制真相 | 否则 SSE catchup 会落在错误基底上 | fresh default 只是启动占位，不是恢复真相 | 不能把“本地刚启动的默认值”当成真实当前状态 |
| 回合结束时 `session_state_changed(idle)` 与普通最后一条消息冲突 | 语义事件真相 | 它被明确标为 authoritative turn-over signal | 普通消息只是展示序列的一部分 | 不能把 UI 最后一条消息当成 turn-over 证明 |
| 复制层里的旧 crash residue 与新的 init 清理冲突 | 明确初始化清理 | stale `pending_action` / `task_summary` 必须先被清掉 | 旧值只是上次异常残留 | 不能把“恢复到了旧 metadata”理解成旧 metadata 全都应保留 |
| 本地 `AppState` 私下变化与 chokepoint 后的对外镜像冲突 | 经 choke point 输出的真相 | 它才会进入 CCR / SDK / 外部消费面 | 私下本地变化没有完成对外同步 | 不能把宿主本地感知直接当成跨宿主统一真相 |
| UI 摘要与更高层语义/复制真相冲突 | 更高层真相 | UI 只负责投影 | UI 天然会压缩或滞后 | 不能把文案颜色、pill 标签或摘要句子当成最终裁决 |

## 3. 最重要的优先级口诀

可以把这张表压成一句话：

`恢复先于默认，语义先于消息，清理先于残留，镜像先于私改，界面永远最后。`

## 4. 这张表最重要的结论

真正危险的不是系统里有很多真相层，  
而是：

`开发者或用户在冲突时不知道该信哪一层。`

所以这张表的用途，不是增加概念数量，  
而是把复杂系统重新压成一套可操作的冲突处理规则。

## 5. 最短用法

- 先识别当前属于哪一类冲突
- 再查哪一层真相应当胜出
- 最后用“禁止误读”一栏校正直觉，避免把方便读到的表面当成裁决者

## 6. 一条硬结论

一套安全控制面如果不能回答：

`冲突时谁说了算？`

那它就还没有真正拥有控制面，  
而只是：

`把多层状态同时暴露出来，却没有给出裁决纪律。`
