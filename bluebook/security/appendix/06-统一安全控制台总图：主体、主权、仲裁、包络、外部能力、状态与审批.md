# 统一安全控制台总图：主体、主权、仲裁、包络、外部能力、状态与审批

## 1. 这一页的目的

这一页是把下面几章重新压成同一张单页总图：

- `18-安全检测技术内核`
- `19-安全不变量`
- `20-边界失真理论`
- `21-安全状态面源码解剖`
- `22-安全证明链`

它不再补新机制，  
而是回答：

`如果真要把 Claude Code 做成一张统一安全控制台，这张控制台最少应该包含哪些面。`

## 2. 单页总图

```text
用户进入系统
  |
  v
[主体面]
  证明：我是谁，我拿着什么凭据，我属于哪个资格主体
  典型源码：status.tsx / bridgeEnabled.ts
  典型状态：subscription, token source, profile scope, organization UUID
  失败感知：明明登录了，但某些高阶能力仍不可用
  |
  v
[主权面]
  证明：谁有资格定义当前运行现实
  典型源码：settings.ts / managedEnv.ts / pluginOnlyPolicy.ts / mcp/config.ts
  典型状态：policy source, managed-only, plugin-only, host-managed env
  失败感知：值看似合法，但系统仍拒绝覆盖或接管
  |
  v
[仲裁面]
  证明：动作是否经过了正确的 rule / mode / safetyCheck / classifier 顺序
  典型源码：permissionSetup.ts / permissions.ts / shadowedRuleDetection.ts
  典型状态：allow, ask, deny, safetyCheck, classifier, unreachable rule
  失败感知：规则明明写了却不生效，或系统看起来放得过宽/过窄
  |
  v
[包络面]
  证明：允许后的动作还能扩散到哪里
  典型源码：pathValidation.ts / sandbox-adapter.ts / SandboxConfigTab.tsx
  典型状态：fs read/write restrictions, network restrictions, unix sockets
  失败感知：动作被允许了，但某些路径/主机仍然碰不到
  |
  v
[外部能力面]
  证明：当前到底开放的是读、连、执行、身份中的哪一种能力语义
  典型源码：WebFetch preapproved / mcpValidation.ts / mcp/config.ts / PluginSettings.tsx / bridgeEnabled.ts
  典型状态：preapproved GET, MCP policy filter, output truncation, managed-only marketplace, bridge entitlement
  失败感知：既然能 fetch/connect，为什么不能顺便获得另一类外部能力
  |
  v
[增量审批面]
  证明：边界变化是否被重新确认
  典型源码：remoteManagedSettings/securityCheck.tsx / ManagedSettingsSecurityDialog.tsx
  典型状态：dangerous settings changed, approved / rejected / no_check_needed
  失败感知：为什么这次突然弹审批，而以前没有
  |
  v
[状态面]
  作用：把上面各面的一部分事实投影给用户
  典型源码：Settings/Status.tsx / SandboxConfigTab.tsx / PluginSettings.tsx / bridgeEnabled.ts
  当前问题：摘要足够多，但跨面的因果联结仍不统一
```

## 3. 如果把它做成真正的“统一安全控制台”，最少要有哪些区块

1. 主体区：账号、凭据来源、资格主体、组织归属
2. 主权区：哪些 source 正在定义关键运行时，哪些被 managed-only / plugin-only 收口
3. 仲裁区：当前 mode、规则命中、safetyCheck、classifier 状态
4. 包络区：文件系统、网络、Unix sockets、排除命令、危险路径
5. 外部能力区：MCP、WebFetch、bridge、plugin marketplace、输出预算
6. 增量审批区：最近哪些高风险边界变化需要重新确认
7. 解释区：把上面六区串成“为什么当前会得到这个结果”的因果句

## 4. 这张总图最大的启发

Claude Code 现有问题不是：

`没有状态。`

而是：

`状态已经够多，但还没有被组织成同一张因果图。`

换句话说，它离统一安全控制台已经不是“从零到一”的距离，  
而更像是：

`从多块可信局部视图，走向一个跨面的总装配层。`

## 5. 最短用法

- 想快速解释“为什么系统现在拦我”：按这六个面依次排查
- 想设计下一代控制台：直接把这六个面做成六个区块
- 想读长文：这张图是 `18-22` 的压缩入口
