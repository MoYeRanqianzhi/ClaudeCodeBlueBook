# 安全完成差异字段速查表：字段、来源、展示层级与缺失后果

## 1. 这一页服务于什么

这一页服务于 [45-安全完成差异字段：统一安全控制台最少该用哪些字段显式展示宿主盲区](../45-%E5%AE%89%E5%85%A8%E5%AE%8C%E6%88%90%E5%B7%AE%E5%BC%82%E5%AD%97%E6%AE%B5%EF%BC%9A%E7%BB%9F%E4%B8%80%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%8F%B0%E6%9C%80%E5%B0%91%E8%AF%A5%E7%94%A8%E5%93%AA%E4%BA%9B%E5%AD%97%E6%AE%B5%E6%98%BE%E5%BC%8F%E5%B1%95%E7%A4%BA%E5%AE%BF%E4%B8%BB%E7%9B%B2%E5%8C%BA.md)。

如果 `45` 的长文解释的是统一安全控制台最少该用哪些字段显式展示宿主盲区，  
那么这一页只做一件事：

`把这些最小字段、源码来源、推荐展示层级和缺失后会导致的误读压成一张矩阵。`

## 2. 字段矩阵

| 字段 | 主要来源 | 推荐展示层级 | 缺失后最可能的误导 | 关键证据 |
| --- | --- | --- | --- | --- |
| `host_kind` | `AppStateStore` bridge/remote 槽位 + 宿主实现身份 | 卡片顶部 / 总览摘要 | 用户默认把所有宿主当成同一类 host | `AppStateStore.ts:122-155`、`structuredIO.ts:139-196`、`remoteIO.ts:126-167` |
| `projection_grade` | 由恢复账、复制态、事件态能力轻度派生 | 总览摘要 + 诊断卡 | 薄投影被误读成厚投影 | `structuredIO.ts:139-196`、`remoteIO.ts:217-223`、`ccrClient.ts:644-662` |
| `missing_ledgers[]` | `restoredWorkerState` / flush / control 子集缺口 | 诊断卡正文 | 用户不知道当前到底缺哪本账 | `structuredIO.ts:139-196`、`directConnectManager.ts:81-99`、`RemoteSessionManager.ts:192-213` |
| `write_ready` | transport / bridge readiness | 状态 pill 下钻 | “connected” 被误读成读写都完整 | `replBridgeTransport.ts:289-299` |
| `read_ready` | transport / stream 可读性与回读能力派生 | 状态 pill 下钻 | 用户看不到写读分裂 | `replBridgeTransport.ts:289-299` |
| `label_basis` | 连接标签是 synthesized/debug-only 还是 authoritative | 调试展开层 / 高级详情 | debug label 被误当正式 completion label | `replBridgeTransport.ts:294-299` |
| `supported_control_subtypes[]` | 宿主当前正式支持的 control 子集 | 能力卡 / 宿主详情 | 协议全集被误当宿主全集 | `directConnectManager.ts:81-99`、`RemoteSessionManager.ts:192-213` |
| `restored_worker_state_supported` | `StructuredIO` vs `RemoteIO` | 宿主详情 | 无恢复账宿主被误以为可做恢复级解释 | `structuredIO.ts:139-196`、`remoteIO.ts:126-128` |
| `internal_event_flush_supported` | `flushInternalEvents()` 是否真实实现 | 宿主详情 | queue / persisted 视图差异被吞掉 | `structuredIO.ts:189-196`、`remoteIO.ts:217-223` |
| `internal_events_pending_visible` | `internalEventsPending` 是否可见 | 诊断层 / 调试层 | 用户不知道“还在收口中” | `structuredIO.ts:194-196`、`remoteIO.ts:221-223` |
| `recovery_window_open` | bridge recovery / authRecoveryInFlight | 高优先级警告 | recovery 中的局部成功被说成正常完成 | `remoteBridgeCore.ts:824-878` |
| `result_projection_suspended` | recovery 时 result 被 drop | 高优先级警告 | 当前结果链收缩却仍显示 completed | `remoteBridgeCore.ts:873-878` |
| `control_projection_suspended` | recovery 时 control 被 drop | 高优先级警告 | 用户被误导去执行当前不会生效的动作 | `remoteBridgeCore.ts:824-871` |
| `completion_claim_ceiling` | 由宿主身份、账本缺口、readiness、recovery 共同派生 | 卡片摘要 / 结论栏 | UI 继续输出过满的“已完成” | `43`、`44`、`45` 主线归纳 |

## 3. 最短判断公式

设计一个面向用户的状态卡时，先问四句：

1. 这张卡有没有先告诉用户当前是哪个宿主
2. 这张卡有没有告诉用户当前缺哪本账
3. 这张卡有没有把 readiness 与 completion ceiling 分开
4. 如果这些字段都不显示，用户会自动把哪种局部视图补成完整视图

## 4. 最常见的四类字段缺失错误

| 缺失类型 | 会造成什么后果 |
| --- | --- |
| 缺 `host_kind` | 所有宿主被当成一个统一观察面 |
| 缺 `missing_ledgers[]` | 用户不知道当前盲区到底在哪里 |
| 缺 `write_ready/read_ready` | “connected” 吞掉读写差异 |
| 缺 `completion_claim_ceiling` | 最终文案重新膨胀成过满结论 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是字段太多，  
而是：

`那些真正表达宿主盲区的关键字段没有被做成对象，最后只能被文案再次压平。`
