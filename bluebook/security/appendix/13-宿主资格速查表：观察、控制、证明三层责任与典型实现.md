# 宿主资格速查表：观察、控制、证明三层责任与典型实现

## 1. 这一页服务于什么

这一页服务于 [29-宿主资格等级：观察宿主、控制宿主、证明宿主为何必须分层定义](../29-%E5%AE%BF%E4%B8%BB%E8%B5%84%E6%A0%BC%E7%AD%89%E7%BA%A7%EF%BC%9A%E8%A7%82%E5%AF%9F%E5%AE%BF%E4%B8%BB%E3%80%81%E6%8E%A7%E5%88%B6%E5%AE%BF%E4%B8%BB%E3%80%81%E8%AF%81%E6%98%8E%E5%AE%BF%E4%B8%BB%E4%B8%BA%E4%BD%95%E5%BF%85%E9%A1%BB%E5%88%86%E5%B1%82%E5%AE%9A%E4%B9%89.md)。

如果 `29` 的长文解释的是为什么宿主必须按观察、控制、证明三层分级，  
那么这一页只做一件事：

`把这三层资格压成一张宿主责任矩阵。`

## 2. 宿主资格矩阵

| 资格层 | 最低责任 | 典型实现 | 为什么能归到这一层 | 还不能声称什么 |
| --- | --- | --- | --- | --- |
| 观察宿主 | 保留安全对象，明确对象可见/不可见，不把缺失伪装成正常 | `sdkMessageAdapter` 路径上的投影宿主，理想观察宿主还应优于它 | 至少能把部分对象投影出来 | 不能自动声称对象完整，更不能声称已完成修复证明 |
| 控制宿主 | 对对象发起动作，对 unsupported subtype 显式失败，把动作结果回到对象上 | `DirectConnectManager`、`RemoteSessionManager`、更宽的 bridge control path | 能处理 `can_use_tool`、`interrupt` 等控制动作，并对未知 subtype 显式 error | 不能声称自己维持了完整恢复纪律、重复响应治理和状态证明 |
| 证明宿主 | 处理 cancel / abort / duplicate / orphan / 恢复窗口 / 状态回写，使动作后真相仍一致 | `StructuredIO`、`RemoteIO`、`remoteBridgeCore` | 已经显式承担一致性治理、状态恢复、delivery/state/metadata 回写与恢复窗口纪律 | 不能把窄子集宿主混成同等级真相来源 |

## 3. 最关键的识别问题

| 问题 | 如果回答“是”，通常说明什么 |
| --- | --- |
| 这个宿主能不能告诉你某对象根本不可见，而不是假装正常？ | 至少具备观察宿主底线 |
| 这个宿主能不能对不支持的 control subtype 显式报错？ | 至少具备控制宿主底线 |
| 这个宿主能不能处理 duplicate / orphan / cancel / 恢复窗口中的真相一致性？ | 才开始接近证明宿主 |

## 4. 这张表最重要的结论

统一安全控制台最危险的误判，不是：

`宿主太窄。`

而是：

`把控制宿主误认成证明宿主，把投影宿主误认成完整观察宿主。`

因此，任何安全 UI、宿主接入文档或产品说明，  
都应该先标：

1. 当前宿主属于哪一层资格
2. 它承担到哪一层责任
3. 它明确不承担什么

## 5. 最短用法

- 先看当前宿主属于观察、控制还是证明层
- 如果它不是证明宿主，就不要把它的“成功”当作最终安全证明
- 如果它不是控制宿主，就不要期望它承担真正修复动作
- 如果它连观察宿主底线都达不到，就不要把它当成统一安全控制台的一部分

## 6. 一条硬结论

统一安全控制台真正要统一的，  
不是让所有宿主都一样强，  
而是：

`让每个宿主都诚实地声明自己属于哪一层资格。`
