## 2. 裂脑防御矩阵

| 裂脑场景 | 分叉在哪两层之间 | 收口闸门 / 守卫 | 为什么有效 | 如果没有它会怎样 |
| --- | --- | --- | --- | --- |
| mode 变了但 CCR / SDK 还不知道 | 本地 `AppState` vs 外部 metadata / status stream | `onChangeAppState` single choke point | 所有 mode 变更统一从一个出口镜像出去 | 不同 mutation path 会留下 stale `permission_mode`，UI 和 CLI 失去一致性 |
| `requires_action` 已经解除但复制层还挂着旧 `pending_action` | 语义状态 vs `external_metadata` | `notifySessionStateChanged` 中的镜像与显式 `null` 清除 | 同一语义状态源主动管理镜像的生灭 | 外部查询面会继续以为系统仍然 blocked |
| permission response 迟到两次 | 当前 permission 结果 vs 迟到 duplicate / orphan response | `StructuredIO` 的 resolved tool_use_id 跟踪与 orphan handler | 防止同一控制事实被重复落地 | duplicate assistant message、API 400、会话真相污染 |
| worker 重启后还保留 crash residue | 当前 worker 真相 vs prior worker stale metadata | `ccrClient.initialize()` 主动清 `pending_action` / `task_summary` | 先切断无主旧真相，再建立新真相 | 新 worker 会继承旧 crash 残留，恢复面继续分叉 |
| proactive refresh 与 SSE 401 同时触发 rebuild | 当前 transport vs 并发恢复路径 | `authRecoveryInFlight` 串行化恢复 | 防止双 `/bridge` fetch 和双 epoch bump | stale epoch、409、恢复窗口自我打架 |
| rebuild 期间旧 transport 还在写 | 新 transport 真相 vs stale transport 写入 | `flushGate` + stale transport guard | 新旧 transport 不能同时写真相 | silent message loss、旧 epoch 心跳、旧路径错误宣告 connected |

## 3. 最重要的共同点

这些场景表面上分别属于：

1. 本地状态同步
2. metadata 镜像
3. permission control
4. crash 恢复
5. transport rebuild

但它们真正共同防的，都是：

`同一安全事实出现两个同时有效的写入口。`

## 4. 这张表最重要的结论

Claude Code 的很多“保守”实现，  
例如：

- 单一 choke point
- 显式 `null`
- duplicate/orphan 去重
- stale cleanup
- recovery lock

本质上都不是细枝末节，  
而是：

`防止双写、双真相、双解释同时成立的反裂脑措施。`

## 5. 最短用法

- 先判断当前问题属于哪一种裂脑场景
- 再看它是靠收口闸门、镜像清除、去重，还是恢复锁来防
- 如果某条路径绕开了这类守卫，就应优先把它视为高风险改动

## 6. 一条硬结论

对这类安全控制面来说，  
最危险的改动通常不是“少一个提示”，  
而是：

`悄悄新增了第二个更新出口。`
