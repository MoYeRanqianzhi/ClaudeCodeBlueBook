# 统一安全控制台字段速查表：字段组、现有来源、缺失派生字段与展示层级

## 1. 这一页服务于什么

这一页服务于 [23-统一安全控制台字段设计：源码里已经有哪些状态字段，还缺哪些跨面字段](../23-%E7%BB%9F%E4%B8%80%E5%AE%89%E5%85%A8%E6%8E%A7%E5%88%B6%E5%8F%B0%E5%AD%97%E6%AE%B5%E8%AE%BE%E8%AE%A1%EF%BC%9A%E6%BA%90%E7%A0%81%E9%87%8C%E5%B7%B2%E7%BB%8F%E6%9C%89%E5%93%AA%E4%BA%9B%E7%8A%B6%E6%80%81%E5%AD%97%E6%AE%B5%EF%BC%8C%E8%BF%98%E7%BC%BA%E5%93%AA%E4%BA%9B%E8%B7%A8%E9%9D%A2%E5%AD%97%E6%AE%B5.md)。

如果 `23` 回答的是：

`控制台最少应该有哪些字段组，字段到底从哪里来？`

那么这一页回答的是：

`每组字段现在已经落在哪些源码对象里，还缺哪些派生字段，以及它更适合放在首页、深度页还是解释层。`

## 3. 从这张表最该得出的结论

统一安全控制台并不缺字段。  
它真正缺的是三类派生字段：

1. `当前主导边界`
2. `当前断裂证明链`
3. `当前最短纠正动作`

也就是说，  
系统最缺的不是更多原始状态，  
而是把现有字段重新加工成可以直接解释的派生状态。

## 4. 最短用法

- 想知道某组字段现在散落在哪：查“现有字段来源”
- 想知道下一代控制台最该补什么：查“仍缺的关键派生字段”
- 想决定字段应该放首页还是深度页：查“最适合展示层级”

## 5. 最终用途

这张表的最终作用，是把 `23` 里的长文判断变成一张可以直接指导控制台信息架构、卡片拆分和字段实现优先级的速查表。
