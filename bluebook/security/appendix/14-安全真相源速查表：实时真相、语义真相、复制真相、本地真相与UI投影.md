# 安全真相源速查表：实时真相、语义真相、复制真相、本地真相与UI投影

## 1. 这一页服务于什么

这一页服务于 [30-安全真相源层级：为什么 `worker_status`、`external_metadata`、`AppState` 与界面文案不能互相替代](../30-%E5%AE%89%E5%85%A8%E7%9C%9F%E7%9B%B8%E6%BA%90%E5%B1%82%E7%BA%A7%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%20%60worker_status%60%E3%80%81%60external_metadata%60%E3%80%81%60AppState%60%20%E4%B8%8E%E7%95%8C%E9%9D%A2%E6%96%87%E6%A1%88%E4%B8%8D%E8%83%BD%E4%BA%92%E7%9B%B8%E6%9B%BF%E4%BB%A3.md)。

如果 `30` 的长文解释的是为什么安全控制面必须按真相层级来理解，  
那么这一页只做一件事：

`把五层真相压成一张真相矩阵。`

## 2. 五层真相矩阵

| 真相层 | 它回答什么问题 | 典型实现 | 优势 | 局限 | 绝不能被误读成什么 |
| --- | --- | --- | --- | --- | --- |
| 实时协议真相 | 现在这一刻发生了什么 | `control_request`、`control_response`、SDK `status` / `auth_status` / tool progress | 最及时 | 不是长期恢复面 | 不能被误读成稳定持久真相 |
| 语义会话真相 | 当前会话语义上是 idle / running / requires_action 吗 | `sessionState.ts`、`RequiresActionDetails` | 语义最清楚 | 仍需要被镜像和复制 | 不能被误读成完整宿主状态树 |
| 复制 worker 真相 | 远程/跨端如何恢复、查询和重建这份状态 | `worker_status`、`requires_action_details`、`external_metadata`、`WorkerStateUploader`、`ccrClient` | 可恢复、可查询、可跨端 | 是 mergeable / retryable 复制层，不是原始事件本体 | 不能被误读成原始协议真相 |
| 本地宿主真相 | 当前宿主如何交互、渲染和执行动作 | `AppState`、`onChangeAppState`、恢复后灌回的本地状态 | 最适合交互 | 不是天然跨宿主一致 | 不能被误读成全局统一真相 |
| UI 投影真相 | 当前界面决定显示什么 | footer、status line、settings 摘要、适配器压缩文案 | 最适合人类阅读 | 信息最容易损失 | 不能被误读成完整安全事实 |

## 3. 最常见的三种误读

| 误读 | 为什么错 |
| --- | --- |
| 把 UI 文案当成全部真相 | UI 只是最后一层投影，天然会压缩和丢字段 |
| 把 `external_metadata` 当成原始事件真相 | 这是一层 mergeable、可恢复的复制真相，不是原始事件流 |
| 把本地 `AppState` 当成跨宿主统一真相 | 它适合交互，但并不天然代表别的宿主已看到同样内容 |

## 4. 这张表最重要的结论

安全控制面里最危险的直觉，不是：

`没有真相。`

而是：

`随手抓到哪一层，就把那一层当成全部真相。`

真正成熟的做法应当是先问：

1. 这是哪一层真相
2. 它负责什么
3. 它不负责什么

## 5. 最短用法

- 先判定当前信息来自哪一层真相
- 如果要判断“现在正在发生什么”，优先看实时协议真相和语义会话真相
- 如果要判断“恢复后系统还记得什么”，优先看复制 worker 真相
- 如果只是看到界面摘要，不要直接把它当作全部安全事实

## 6. 一条硬结论

在 Claude Code 这类分层安全控制面里，  
真正的专业判断能力不在于背多少字段，  
而在于：

`永远先分清自己当前看到的是哪一层真相。`
