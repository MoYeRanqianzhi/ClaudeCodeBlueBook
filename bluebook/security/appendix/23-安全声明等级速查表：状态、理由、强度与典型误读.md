# 安全声明等级速查表：状态、理由、强度与典型误读

## 1. 这一页服务于什么

这一页服务于 [39-安全声明等级：为什么控制面不该只输出 yes/no，而要把状态、理由与证据强度分层编码](../39-%E5%AE%89%E5%85%A8%E5%A3%B0%E6%98%8E%E7%AD%89%E7%BA%A7%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8D%E8%AF%A5%E5%8F%AA%E8%BE%93%E5%87%BA%20yes-no%EF%BC%8C%E8%80%8C%E8%A6%81%E6%8A%8A%E7%8A%B6%E6%80%81%E3%80%81%E7%90%86%E7%94%B1%E4%B8%8E%E8%AF%81%E6%8D%AE%E5%BC%BA%E5%BA%A6%E5%88%86%E5%B1%82%E7%BC%96%E7%A0%81.md)。

如果 `39` 的长文解释的是为什么控制面必须把结论按强度和理由分层编码，  
那么这一页只做一件事：

`把不同声明族的等级、典型字段和最常见误读压成一张矩阵。`

## 2. 声明等级矩阵

| 声明族 | 典型等级 | 典型字段 | 这类等级真正表达什么 | 最危险的误读 |
| --- | --- | --- | --- | --- |
| 允许类 | `allowed` / `allowed_warning` / `rejected` | `status`、`utilization`、`overageDisabledReason` | 不只是能不能用，还包括离边界有多近 | 把 warning 误读成普通 allowed |
| 连接类 | `connected` / `failed` / `needs-auth` / `pending` / `disabled` | `type`、`error`、server info | 不只是在线与否，还包括恢复路径属于哪一类 | 把所有非 connected 都误读成同一类失败 |
| 运行类 | `blocked` / `waiting` / `completed` / `review_ready` / `failed` | `status_category`、`status_detail`、`is_noteworthy` | 不只是结束与否，还包括完成到哪一类、是否值得优先关注 | 把 completed 和 review_ready 混成同一种“结束” |
| 重试类 | typed error + `error_status` + retry budget | `attempt`、`max_retries`、`retry_delay_ms`、`error_status` | 不只是失败，还包括失败形态与下一步策略 | 把可重试失败误读成终局失败 |
| 初始化 / 校验类 | typed reason 或 descriptive message | `reason`、`message` | 不只是通过 / 不通过，还包括失败在第几层 | 把所有 fail 混成同一个“初始化失败” |

## 3. 最短判断公式

看到一条系统声明时，先问四句：

1. 它属于哪一类声明
2. 它给的是布尔量、枚举还是原因对象
3. 当前等级比相邻等级强还是弱
4. 如果把它压扁成 yes/no，会丢掉什么动作意义

## 4. 最常见的三类压扁错误

| 压扁方式 | 会丢掉什么 |
| --- | --- |
| 把 warning 压成 allowed | 丢掉边界逼近信号 |
| 把 pending / needs-auth / failed 压成 disconnected | 丢掉恢复路径差异 |
| 把 typed reason 压成 generic failure | 丢掉正确支持和排障入口 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是字段太多，  
而是：

`本来已经被分级表达的结论，又在下游被重新压扁成了二元判断。`
