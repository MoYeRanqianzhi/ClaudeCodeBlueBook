# 安全提交边界速查表：语义、复制、可见、持久与恢复五层提交

## 1. 这一页服务于什么

这一页服务于 [34-安全提交语义：关键安全事实何时算已提交，何时只是排队、可见或可恢复](../34-%E5%AE%89%E5%85%A8%E6%8F%90%E4%BA%A4%E8%AF%AD%E4%B9%89%EF%BC%9A%E5%85%B3%E9%94%AE%E5%AE%89%E5%85%A8%E4%BA%8B%E5%AE%9E%E4%BD%95%E6%97%B6%E7%AE%97%E5%B7%B2%E6%8F%90%E4%BA%A4%EF%BC%8C%E4%BD%95%E6%97%B6%E5%8F%AA%E6%98%AF%E6%8E%92%E9%98%9F%E3%80%81%E5%8F%AF%E8%A7%81%E6%88%96%E5%8F%AF%E6%81%A2%E5%A4%8D.md)。

如果 `34` 的长文解释的是为什么安全控制面必须把“已提交”拆成多层边界，  
那么这一页只做一件事：

`把五层提交边界压成一张矩阵。`

## 2. 五层提交边界矩阵

| 提交层 | 何时算提交 | 还不算提交的状态 | 典型源码入口 | 最危险的误读 |
| --- | --- | --- | --- | --- |
| 语义提交 | 本地语义状态已经被正式作者改写并通知本地语义监听面 | 还只是旧语义、或只有消费层猜测变化 | `sessionState.ts` 的 `notifySessionStateChanged(...)` | 把镜像层更新延迟误读成语义还没变 |
| 复制提交 | 远端 worker 复制层已经成功接收该状态 | 只是 `enqueue(...)`、只是 pending、只是 retry 中 | `ccrClient.ts` 的 `reportState(...)` / `reportMetadata(...)`，`WorkerStateUploader.ts` | 把“已入队”误读成“远端已拥有当前真相” |
| 可见提交 | 外部消费者现在应当看到它，且顺序约束已经满足 | 结果刚生成但仍被 hold-back、stream buffer 还没 flush | `ccrClient.ts` 的 `writeEvent(...)`，`print.ts` 的 `heldBackResult` 路径 | 把“代码已经生成消息”误读成“用户已经看见消息” |
| 持久提交 | 恢复账本已经 flush，下一次 resume 具备重建基础 | 事件只发给前端了，但 internal events 还没落 | `ccrClient.ts` 的 `writeInternalEvent(...)` / `flushInternalEvents()`，`print.ts` finally | 把“屏幕上看到了”误读成“恢复系统一定能还原” |
| 恢复提交 | 新进程已经能用 transcript + metadata 把状态重建回来 | 只恢复了 transcript，或只恢复了 metadata，或 fresh default 抢先落位 | `sessionStorage.ts` 的 `hydrateFromCCRv2InternalEvents(...)`，`print.ts` resume 路径 | 把单边恢复误读成完整恢复 |

## 3. 最关键的顺序纪律

这张表最想强调的不是“五层都存在”，  
而是它们的顺序不能被乱写：

1. 先有语义提交
2. 再谈复制提交
3. 再谈可见提交
4. 再谈持久提交
5. 最后才谈恢复提交

## 4. 最常见的三类误判

| 误判 | 为什么错 |
| --- | --- |
| “已经 enqueue 了，所以远端已经同步” | enqueue 只是进入提交流水线，不等于 PUT 成功 |
| “result 已经出来了，所以这一轮已经结束” | `heldBackResult` 和 `idle` 事件说明 turn-over 还要等更多边界跨过 |
| “前端已经看见了，所以 resume 肯定能恢复” | internal events 账本和 client events 不是同一套持久化面 |

## 5. 最短用法

- 先问当前讨论的是哪一层提交
- 再问这层边界是靠哪个模块正式跨过
- 如果争论双方说的“已提交”不是同一层，就先不要继续争结论

## 6. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是提交边界太多，  
而是：

`大家都说“已经提交了”，但说的不是同一个提交层。`
