# 安全解释权限速查表：不同宿主可以解释什么、不应解释什么

## 1. 这一页服务于什么

这一页服务于 [37-安全解释权限：为什么看到账本子集的宿主不应替完整控制面下结论](../37-%E5%AE%89%E5%85%A8%E8%A7%A3%E9%87%8A%E6%9D%83%E9%99%90%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E7%9C%8B%E5%88%B0%E8%B4%A6%E6%9C%AC%E5%AD%90%E9%9B%86%E7%9A%84%E5%AE%BF%E4%B8%BB%E4%B8%8D%E5%BA%94%E6%9B%BF%E5%AE%8C%E6%95%B4%E6%8E%A7%E5%88%B6%E9%9D%A2%E4%B8%8B%E7%BB%93%E8%AE%BA.md)。

如果 `37` 的长文解释的是为什么解释权必须受账本可见性约束，  
那么这一页只做一件事：

`把不同宿主“可以解释什么、不能解释什么”压成一张矩阵。`

## 2. 解释权限矩阵

| 宿主类型 | 它真实看到了什么 | 它明显没看到什么 | 它可以诚实说什么 | 它绝不能替系统说什么 |
| --- | --- | --- | --- | --- |
| 默认 `StructuredIO` 宿主 | 基础消息流、基础控制交互 | 恢复账、internal-event 持久面、完整远端复制状态 | “我看到了当前控制/消息交互” | “整套控制面现在已经完整对齐/可恢复” |
| `RemoteIO + CCR v2` | 更宽的语义账镜像、复制账、可见账、恢复账接口 | 在 init 失败、条件不成立或 transport 不符时的完整视野 | “在当前条件成立时，我能看到更宽的状态面” | “我在任何时刻都能代表完整控制面下结论” |
| `DirectConnectSessionManager` | `can_use_tool` 权限请求面、interrupt | 恢复账、完整复制账、控制全集 | “当前权限请求怎样、是否可答复” | “系统整体安全状态、恢复状态、控制面对齐状态如何” |
| `RemoteSessionManager` | 远程消息发送面、权限请求面 | 完整恢复账、完整复制账、宽 control 子集 | “这条远程消息/权限请求当前怎样” | “我已经看到了完整安全状态” |
| `remoteBridgeCore` | 更宽的 SDK message / control 投影 | recovery window 中被主动 drop 的那部分事实 | “当前窗口下我能转发这部分更宽事实” | “bridge 永远无损、永远全视角、永远能完整解释” |

## 3. 最常见的三类越权解释

| 越权方式 | 表面上像什么 | 本质问题 |
| --- | --- | --- |
| 局部事实硬推全局状态 | “我看到了 permission，所以系统整体就是这样” | 子集宿主越权替全集裁决 |
| 窄宿主代替恢复面说话 | “前端能看到，所以 resume 肯定没问题” | 把可见账误读成恢复账 |
| 条件化宽宿主冒充绝对全知 | “bridge/remote IO 接上了，所以任何时候我都知道全部” | 忽略初始化条件、drop 窗口和降权状态 |

## 4. 最短用法

- 先问当前宿主拿到了哪几本账
- 再问当前时序窗口有没有让解释权限缩窄
- 如果两步中任何一步答案不完整，就只说局部真相，不替整套控制面下结论

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正成熟的宿主不是“解释得最多”的宿主，  
而是：

`最清楚自己在哪些地方应该停止解释的宿主。`
