## 2. 差异显化矩阵

| 当前界面 / 适配层 | 当前压缩写法 | 被隐藏的盲区 | 最少必须显化的字段 | 最不该继续复用的文案 | 关键证据 |
| --- | --- | --- | --- | --- | --- |
| `/status` 的 MCP 总览 | 计数摘要：`connected / need auth / pending / failed` | 看不到宿主有没有恢复账、flush 视角、completion scope | `host_type`、`completion_visibility_grade`、`missing_ledgers[]` | “connected = 一切完整可用” | `status.tsx:89-115` |
| remote `sdkMessageAdapter` | `Status: ...` generic informational text | 结构化状态被压成泛文本，丢失宿主边界 | `source_host`、`status_authority_level`、`missing_context_reason` | “Status: X” 足以表达完成强度 | `sdkMessageAdapter.ts:88-103` |
| bridge 总体状态 | `Remote Control active / reconnecting / failed / connecting` | active 不说明读写 readiness 差异、也不说明 completion thickness | `write_ready`、`read_ready`、`recovery_in_flight`、`projection_grade` | “Remote Control active” | `bridgeStatusUtil.ts:124-140` |
| repl bridge transport | `connected / init / connecting` synthesized label | `connected` 只是 write-readiness，不是 read-readiness | `label_basis`、`readiness_kind`、`debug_only` | “connected” 直接给用户看 | `replBridgeTransport.ts:289-299` |
| 默认 StructuredIO 宿主 | 表面上仍像普通 host | 没有 `restoredWorkerState`、没有 flush、没有 pending depth | `restored_worker_state_supported`、`internal_event_flush_supported`、`recovery_ledger_supported` | “与 RemoteIO 同等完成视角” | `structuredIO.ts:139-196` |
| RemoteIO / CCR 宿主 | 看起来只是另一个 remote transport | 实际拿到更厚 completion projection | `has_restored_worker_state`、`has_internal_event_reader`、`has_internal_event_flush`、`replicated_state_visible` | “只是另一个连接方式” | `remoteIO.ts:126-167`、`remoteIO.ts:217-223` |
| DirectConnectManager | 远端接入看似统一 | control surface 极窄，几乎只围绕 `can_use_tool` | `supported_control_subtypes[]`、`completion_claim_ceiling` | “完整 SDK host” | `directConnectManager.ts:81-99` |
| RemoteSessionManager | 远端 session 看似完整 | 同样只有窄 control 子集 | `supported_control_subtypes[]`、`completion_claim_ceiling` | “完整 remote completion controller” | `RemoteSessionManager.ts:192-213` |
| bridge recovery 窗口 | 可能仍沿用同一 status shell | completion 投影正在动态收缩，result/control 被 drop | `recovery_window_open`、`result_projection_suspended`、`control_projection_suspended` | “仍然 active / completed” | `remoteBridgeCore.ts:824-878` |

## 3. 最短判断公式

看到一个用户界面上的完成或连接状态时，先问四句：

1. 这个标签是不是把多个宿主的不同可见度压成了一句通用文案
2. 当前界面有没有显式告诉用户缺了哪本账
3. 当前界面有没有说明这是事件级、复制级还是恢复级投影
4. 如果沿用这句文案，用户最容易高估哪种完成能力

## 4. 最常见的四类显化失败

| 失败方式 | 会造成什么误导 |
| --- | --- |
| 只显示状态，不显示观察边界 | 用户会自动把薄投影视作厚投影 |
| 只显示 connected，不显示 readiness 维度 | 写可用会被误读成读写都完整 |
| 只显示摘要计数，不显示宿主类型 | 不同 host 的完成上限被抹平 |
| 只显示 generic informational text | 结构化 completion authority 消失 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是界面太复杂，  
而是：

`底层已经承认的宿主完成差异，在展示层里又被重新压成了同一句状态文案。`
