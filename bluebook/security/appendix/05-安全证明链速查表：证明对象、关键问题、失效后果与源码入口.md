# 安全证明链速查表：证明对象、关键问题、失效后果与源码入口

如果 `22` 回答的是：

`系统凭什么认为当前可安全执行？`

那么这一页回答的是：

`到底是哪一条证明链在回答哪个问题，一旦断掉会出现什么后果。`

## 2. 六条证明链总表

| 证明链 | 它在证明什么 | 它回答的关键问题 | 主要源码入口 | 失效后最典型的后果 | 用户侧常见症状 |
| --- | --- | --- | --- | --- | --- |
| 主体链 | 当前主体可被正确识别与归属 | “现在到底是谁在请求这项能力？” | `utils/status.tsx`、`bridge/bridgeEnabled.ts` | 高阶能力没有合法承载主体，资格无法授予 | “我明明登录了，为什么 Remote Control 还是不可用？” |
| 主权链 | 当前运行时由合法来源定义 | “谁有资格定义当前运行现实？” | `settings.ts`、`managedEnv.ts`、`pluginOnlyPolicy.ts`、`mcp/config.ts` | 授权判断发生在错误前提下；低信任来源悄悄改写边界 | “这个值看起来没问题，为什么系统仍然不让我覆盖？” |
| 授权链 | 当前动作经过了正确仲裁 | “这次动作是否真的走过了应走的仲裁顺序？” | `permissionSetup.ts`、`permissions.ts`、`shadowedRuleDetection.ts` | allow rule、mode 或假规则绕过仲裁层 | “我写了规则，为什么还是 prompt / deny，或者反过来为什么 auto 放得太宽？” |
| 包络链 | 允许后的扩散半径仍被收口 | “即便允许，这个动作实际还能碰到什么？” | `pathValidation.ts`、sandbox 相关实现、`SandboxConfigTab.tsx` | 动作合法但破坏半径失控；路径/网络壳层被误判 | “它明明能执行，为什么某些路径或网络目标还是不让碰？” |
| 外部能力链 | 当前开放的是哪一种外部能力语义 | “现在开放的是读、连、执行，还是身份能力？” | `WebFetchTool/preapproved.ts`、`mcpValidation.ts`、`mcp/config.ts`、`PluginSettings.tsx`、`bridge/bridgeEnabled.ts` | 读取、连接、输出、执行、资格语义被混写 | “既然能 fetch / connect，为什么不能顺便把另一类外部能力也当成可信？” |
| 增量审批链 | 边界变化已经被重新确认 | “今天的边界与昨天不同时，这个变化是否被重新接受？” | `remoteManagedSettings/securityCheck.tsx`、`ManagedSettingsSecurityDialog.tsx` | 边界漂移以“曾经同意过”为借口静默进入系统 | “为什么这次突然弹一个信任确认，而不是直接沿用以前的状态？” |

## 3. 从失效后果回看这六条证明链

把上表反过来看，就会发现：

- 主体链失败，意味着“没人能合法承载能力”
- 主权链失败，意味着“系统活在被错误来源定义的现实里”
- 授权链失败，意味着“程序结构本身被提前架空”
- 包络链失败，意味着“允许之后没有边界壳层”
- 外部能力链失败，意味着“不同外部能力语义被混写”
- 增量审批链失败，意味着“边界漂移没有重新取得正当性”

所以这六条链不是并列装饰，  
而是六种不同类型的证明责任。

## 4. 最适合怎样用这张表

- 如果你先遇到的是一个用户症状，就先判断是哪条证明链在断
- 如果你先遇到的是一个源码模块，就先判断它属于哪条证明链
- 如果你在设计别的 Agent 平台，就用这张表检查自己是否把六类证明偷懒压成了一个总开关

## 5. 最短压缩

把这张表再压成一句话：

`Claude Code 不是在单点证明“动作安全”，而是在并行证明“主体、现实、裁决、包络、外部能力与边界更新”同时成立。`
