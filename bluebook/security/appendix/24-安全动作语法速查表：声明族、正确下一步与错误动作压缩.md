## 2. 动作语法矩阵

| 声明族 / 信号 | 典型状态或字段 | 正确下一步动作 | 最危险的错误动作压缩 | 关键证据 |
| --- | --- | --- | --- | --- |
| 权限请求 | `can_use_tool` + `permission_suggestions` + `decision_reason` | 按 suggestions 做 `addRules / setMode / addDirectories` 等受限修法，或显式 deny | 压成“弹个确认框就行” | `controlSchemas.ts:106-121`、`coreSchemas.ts:263-299` |
| 权限请求报错 | `pending_permission_requests` | 继续保留未完成权限链，避免一次报错吃掉动作上下文 | 压成“报错 = 本次所有待处理权限都消失” | `controlSchemas.ts:594-603` |
| requires-action 权限流 | session 进入 `requires_action`，宿主只回 `allow / deny / interrupt` | 把状态落成显式审批动词，而不是停留在被动描述 | 压成“系统卡住了，随便刷新或重试” | `structuredIO.ts:93`、`structuredIO.ts:585`、`print.ts:808`、`ccrClient.ts:644` |
| turn 摘要阻断态 | `status_category = blocked` + `needs_action` | 去解堵，不把 blocked 当 finished | 压成“本轮失败，重新跑一遍” | `coreSchemas.ts:1544-1569` |
| turn 摘要等待态 | `status_category = waiting` + `needs_action` | 等待外部前提或异步结果，不抢跑下一轮 | 压成“waiting 也是失败，立刻重复提交” | `coreSchemas.ts:1544-1569` |
| turn 摘要审查态 | `status_category = review_ready` + `artifact_urls` | 去 review 产物、审代码或审结果 | 压成“已经 completed，无需人工看” | `coreSchemas.ts:1544-1569` |
| API 重试态 | `api_retry` + `retry_delay_ms` + `attempt/max_retries` | 等待系统按既定窗口自动重试 | 压成“用户再手工猛点一次重试” | `coreSchemas.ts:1572-1587`、`QueryEngine.ts:943` |
| MCP 可执行连接 | `connected` 且带 live `client / capabilities / instructions` | 才允许列工具、拉资源、采纳 instructions | 压成“只要不是 failed 就当成可调用连接” | `types.ts:180-192`、`client.ts:1743`、`client.ts:2000`、`prompts.ts:579` |
| MCP 认证缺失 | `needs-auth` | 去认证 / 重认证，再回连接链 | 压成“继续 reconnect 或反复重试连接” | `types.ts:201-205`、`client.ts:340-360`、`toolExecution.ts:1599` |
| MCP 重连进行中 | `pending` + `reconnectAttempt/maxReconnectAttempts` | 等待指数退避重连，展示进度，必要时允许取消或禁用 | 压成“pending 就是 failed，立刻让用户重配” | `types.ts:207-213`、`useManageMCPConnections.ts:387` |
| MCP 被禁用 | `disabled` | 先 enable，再回到 pending / reconnect | 压成“disabled 和 failed 一样，直接重试连接” | `types.ts:215-219`、`useManageMCPConnections.ts:1093` |
| MCP 连接失败 | `failed` + `error` | 看错误、修配置、手工 reconnect，必要时转 `/mcp` | 压成“failed 与 needs-auth 一样，都去登录” | `types.ts:194-199`、`client.ts:1605`、`useMcpConnectivityStatus.tsx:36` |
| rewind 预演可回滚 | `canRewind = true` + `filesChanged` + `insertions/deletions` | 展示受影响文件和改动规模，再决定是否执行恢复 | 压成“能回滚就直接回滚，不用先看影响面” | `coreSchemas.ts:1217-1226`、`controlSchemas.ts:318-327`、`print.ts:4520` |
| rewind 无代码改动 | `filesChanged` 为空或未给出 | 不提供 code restore，最多保留 conversation restore 或仅给结果说明 | 压成“所有 rewind success 都应该有代码恢复按钮” | `MessageSelector.tsx:313` |
| rewind 不可执行 | `canRewind = false` + `error` | 显式停在不可回滚结论，不假装有修复路径 | 压成“先点一下执行，说不定能过” | `controlSchemas.ts:318-327`、`print.ts:2995`、`print.ts:736` |
| 组织校验失败 | `valid: false; message` | 按 message 分流到 retry、`claude auth login`、移除错误 env var、换正确组织 token | 压成“所有 auth fail 都提示同一条重新登录” | `auth.ts:1911-1999` |
| CCR 初始化失败 | `no_auth_headers / missing_epoch / worker_register_failed` | 分别走补认证头、补协议基底、重注册 worker 等不同修复路径 | 压成“初始化失败就统一再试一次” | `ccrClient.ts:49-57` |

## 3. 最短判断公式

看到一条安全声明时，先问四句：

1. 它对应的下一步动作到底是谁来做
2. 这条动作是等待、审批、认证、修配置还是执行修复
3. 如果把它压扁成另一种相近状态，会误导到哪条错误动作
4. 这条动作有没有被 schema 明确写成 typed field，而不是散落在文案里

## 4. 最常见的四类动作压缩错误

| 压缩方式 | 会导致什么错动作 |
| --- | --- |
| 把 `needs-auth / pending / failed / disabled` 全压成“连接异常” | 用户和宿主会对完全不同的问题走同一条修复路径 |
| 把 `blocked / waiting / review_ready` 全压成“还没完成” | 会错过解堵、等待和审查三种完全不同的后续动作 |
| 把 `permission_suggestions` 压成普通说明文字 | 宿主失去受限修法集合，容易退化成过宽 allow |
| 把 `canRewind` 成功态全部理解成“现在就能恢复代码” | 会误给不存在的恢复按钮或跳过 dry-run 审核 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是状态太多，  
而是：

`本来已经区分好的状态，被下游重新压扁，最终把用户和宿主推向了错误动作。`
