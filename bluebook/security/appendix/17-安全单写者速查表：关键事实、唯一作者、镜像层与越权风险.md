# 安全单写者速查表：关键事实、唯一作者、镜像层与越权风险

## 1. 这一页服务于什么

这一页服务于 [33-安全单写者原则：为什么关键安全事实必须有唯一作者，其他层只能镜像](../33-%E5%AE%89%E5%85%A8%E5%8D%95%E5%86%99%E8%80%85%E5%8E%9F%E5%88%99%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E5%85%B3%E9%94%AE%E5%AE%89%E5%85%A8%E4%BA%8B%E5%AE%9E%E5%BF%85%E9%A1%BB%E6%9C%89%E5%94%AF%E4%B8%80%E4%BD%9C%E8%80%85%EF%BC%8C%E5%85%B6%E4%BB%96%E5%B1%82%E5%8F%AA%E8%83%BD%E9%95%9C%E5%83%8F.md)。

如果 `33` 的长文解释的是为什么关键安全事实必须维持唯一作者，  
那么这一页只做一件事：

`把关键事实、正式作者、镜像层和多写者风险压成一张矩阵。`

## 2. 安全单写者矩阵

| 关键安全事实 | 正式作者 | 允许存在的镜像层 | 为什么必须单写 | 如果镜像层越权成作者会怎样 |
| --- | --- | --- | --- | --- |
| 当前 `permission mode` 对外应是什么 | `onChangeAppState` 中的 mode diff 出口 | `external_metadata.permission_mode`、SDK status stream、UI 展示 | 所有 mode 变更必须从同一出口对外同步，外部消费者才会收敛到同一事实 | 不同 mutation path 会留下 stale mode，CLI、CCR、SDK、UI 各自说不同答案 |
| 当前会话语义是 `idle`、`running` 还是 `requires_action` | `notifySessionStateChanged(...)` | state listener、`external_metadata.pending_action`、`task_summary` 清理、SDK `session_state_changed` | 语义状态、阻塞细节和复制层清理必须一起推进，才能避免“状态已变但镜像未收口” | 外部层会继续以为系统 blocked、忙碌或仍带着上轮残留 |
| 当前控制面消息的真实输出顺序 | `StructuredIO` outbound drain loop | `sendRequest()`、`print.ts`、宿主 transport 只是入队者 | 时序事实只能由一个写出口决定，否则权限请求和流式事件会互相超车 | `control_request` 乱序、duplicate/orphan 重落地、会话时序失真 |
| 当前 worker 复制真相的最新版本 | `WorkerStateUploader` 的 `1 in-flight + 1 pending` 串行写入 | pending patch、retry 吸收、服务端 `/worker` 状态 | 复制层必须只有一个正式上传通道，才能保证 top-level current version 唯一 | 多个并发 patch 会把当前真相写成竞争版本，恢复面无法判断谁是最新 |
| 关键资格/能力布尔量是否成立 | 预先计算好的单一真相源，例如 `kairosEnabled` | 各消费面只读消费、界面投影、功能 gating | 资格类布尔量一旦由多面重复推导，就会出现“一个面开、另一个面关” | 能力开关解释失效，资格边界从确定条件退化成宿主偶然性 |
| 当前恢复动作是否正在进行、谁有权重建 transport | `authRecoveryInFlight` 护栏下的单恢复路径 | `flushGate` 排队写入、新 transport 接管、旧 transport 只允许退场 | 恢复动作本身也是关键安全事实，不能由两个恢复器同时写 | 双 `/bridge` fetch、双 epoch bump、stale transport 写入和 silent loss 会一起出现 |

## 3. 共同规律

表面上这些事实分别属于：

1. mode 同步
2. 会话语义
3. 输出时序
4. worker 复制
5. 能力资格
6. transport 恢复

但它们真正共同遵守的都是同一条纪律：

`正式作者负责提交真相，镜像层只负责传播真相。`

## 4. 最常见的越权方式

| 越权方式 | 看起来像什么 | 实际上破坏了什么 |
| --- | --- | --- |
| 镜像层自己补写 | UI、bridge、handler “顺手再同步一次” | 正式作者的唯一出口 |
| 消费层重复推导 | 多个宿主各自算资格布尔量 | 资格事实的唯一判定面 |
| 重试层并发补发 | 多个 uploader / rebuild 路径同时发出当前版本 | 当前版本的唯一提交面 |
| 展示层代替裁决 | 以最后一条文案、标签、状态色代表最终真相 | 真相层级与作者权边界 |

## 5. 最短用法

- 先问当前争议的是哪一种关键事实
- 再问哪一个模块才是正式作者
- 如果某个镜像层也开始写同一事实，就应把它视为高风险设计

## 6. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正危险的往往不是“镜像太多”，  
而是：

`镜像层忘了自己只是镜像，开始偷偷把自己当作者。`
