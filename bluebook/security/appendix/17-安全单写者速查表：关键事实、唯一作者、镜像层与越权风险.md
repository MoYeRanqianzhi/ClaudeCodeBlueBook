# 安全单写者速查表：关键事实、唯一作者、镜像层与越权风险

## 1. 这一页服务于什么

这一页服务于 [33-安全单写者原则：为什么关键安全事实必须有唯一作者，其他层只能镜像](../33-%E5%AE%89%E5%85%A8%E5%8D%95%E5%86%99%E8%80%85%E5%8E%9F%E5%88%99%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E5%85%B3%E9%94%AE%E5%AE%89%E5%85%A8%E4%BA%8B%E5%AE%9E%E5%BF%85%E9%A1%BB%E6%9C%89%E5%94%AF%E4%B8%80%E4%BD%9C%E8%80%85%EF%BC%8C%E5%85%B6%E4%BB%96%E5%B1%82%E5%8F%AA%E8%83%BD%E9%95%9C%E5%83%8F.md)。

如果 `33` 的长文解释的是为什么关键安全事实必须维持成单写者结构，  
那么这一页只做一件事：

`把“谁能正式写、谁只能镜像、越权后会坏什么”压成一张矩阵。`

## 2. 单写者矩阵

| 关键安全事实 | 正式作者 | 合法镜像 / 投影层 | 为什么必须单写 | 一旦出现并列作者会坏什么 |
| --- | --- | --- | --- | --- |
| 对外 `permission_mode` 真相 | `onChangeAppState` 中的 mode diff choke point | `external_metadata`、SDK status stream、宿主 UI | mode 变化路径很多，必须统一从一个出口外化 | stale `permission_mode`、CLI 与外部 UI 不一致、仲裁失真 |
| 会话语义状态 `idle/running/requires_action` | `notifySessionStateChanged(...)` | `pending_action`、`task_summary` 清理、SDK `session_state_changed` | 语义状态必须和镜像状态同向变化 | blocked 已解除但外部仍显示阻塞，或旧摘要残留到下一轮 |
| mode 变化后的下游广播 | `notifyPermissionModeChanged(...)` | CCR metadata listener、SDK status listener | 下游消费面必须共享同一广播出口 | 某条 mode mutation path 静默绕过，消费面各自看到不同 mode |
| outbound control / event 写出顺序 | `StructuredIO` drain loop | `sendRequest()`、`print.ts`、transport stdout 写口 | 时序本身就是安全语义的一部分 | `control_request` 超车、事件乱序、权限交互失真 |
| `worker_status` / `external_metadata` 复制真相 | `WorkerStateUploader` 的单 in-flight drain | pending patch、重试吸收、服务端 worker 面 | 复制真相必须只有一个当前版本在路上 | 多个 patch 并发竞争、旧值回写覆盖、新旧版本互相打架 |
| 资格 / 功能布尔真相 | `main.tsx` 预计算后的 `kairosEnabled` | 各消费组件、状态读取面 | 资格判断不能由消费面各自重复推导 | 一个面认为 enabled、另一个面认为 disabled，没人能解释谁是权威 |
| 恢复窗口中的 transport 真相 | `authRecoveryInFlight` + `rebuildTransport(...)` 串行恢复路径 | `flushGate` 队列、重建后的新 transport、后续 reconnect 逻辑 | 恢复动作自身也必须有唯一作者 | 双 `/bridge` fetch、双 epoch bump、stale transport 继续写、silent loss |

## 3. 最常见的三类越权

| 越权类型 | 表面看起来像什么 | 实际上在做什么 | 风险 |
| --- | --- | --- | --- |
| 镜像层偷偷补写 | “顺手把 metadata 也改了” | 让镜像层升级成并列作者 | 真相层级坍塌，恢复与仲裁无法稳定 |
| UI 层反向定义真相 | “界面已经显示成这样了，状态就应该是这样” | 让投影层冒充作者层 | 文案压缩被误读成正式裁决 |
| 恢复层并发自救 | “两个恢复路径一起跑更稳” | 让恢复动作自身长出多个作者 | 重建窗口裂脑，越恢复越混乱 |

## 4. 最短判断公式

遇到关键安全字段时，先连问四句：

1. 这个事实的正式作者是谁
2. 其他层是在镜像它、恢复它，还是投影它
3. 哪一层如果开始改写它，就会变成越权作者
4. 一旦出现双作者，首先坏掉的是时序、镜像、恢复还是解释

## 5. 这张表最重要的结论

很多团队会把“同步做全”当成目标。  
这张表给出的更深结论是：

`真正成熟的安全控制面，不是让每一层都能写，而是先定义谁有写权，再要求其他层承认自己只是镜像。`

## 6. 最短用法

- 先找到当前字段的正式作者
- 再检查其他层是否只做镜像、排队、恢复或投影
- 如果某个适配层、UI 层或恢复层开始直接改写该事实，应优先视为高风险改动

## 7. 一条硬结论

对 Claude Code 这类分层安全控制面来说，  
最危险的工程幻觉不是“缺少更多同步”，  
而是：

`让本该只读的层，误以为自己也有资格写关键安全事实。`
