## 3. 最短判断公式

做统一安全控制台设计评审时，先问四句：

1. 哪个字段必须先于 connected/active 出现
2. 哪个字段决定当前整张卡的解释上限
3. 哪个字段只是下钻解释，不该抢占顶层
4. 哪个字段如果埋进详情区，就等于没有显化

## 4. 最常见的四类挂载错误

| 挂载错误 | 会造成什么问题 |
| --- | --- |
| 把 `completion_claim_ceiling` 放到详情层 | 用户先看到过满结论，再也不会回头修正 |
| 把 `missing_ledgers[]` 压成一句提醒 | 用户不知道到底缺了哪本账 |
| 把 `write_ready/read_ready` 合并回 `connected` | readiness 分裂被界面吞掉 |
| 把 `supported_control_subtypes[]` 放进静态状态区 | 动作上限信息无法约束用户操作预期 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是字段不够多，  
而是：

`那些本该抢占高优先级位置的宿主盲区字段，被界面重新埋回了低优先级详情区。`
