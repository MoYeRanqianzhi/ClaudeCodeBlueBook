如果 `25` 解释的是统一安全控制台应如何形成行动闭环，  
那么这一页只做一件事：

`把这条闭环压成一张可以直接查的动作矩阵。`

## 2. 四段闭环矩阵

| 当前状态 | 断裂证明链 | 最短控制动作 | 默认回读验证 | 典型源码入口 |
| --- | --- | --- | --- | --- |
| bridge 不可用，提示 scope / org / subscription 问题 | 主体链 | 重新登录、刷新主体信息 | 再读 bridge entitlement / auth status | `bridge/bridgeEnabled.ts`、`SDKAuthStatusMessageSchema` |
| setting source 不符合预期，managed-only / plugin-only 收口 | 主权链 | 打开来源详情、跳转正确 source、联系管理员 | `get_settings` / per-source settings 回读 | `settings.ts`、`pluginOnlyPolicy.ts`、`controlSchemas.ts` |
| 当前动作被 ask / deny / safetyCheck 卡住 | 授权链 | 调整 mode、修正规则、接受人工审批 | 回读 `decisionReason` / 当前 permission mode | `permissions.ts`、`types/permissions.ts` |
| 动作允许但路径 / 网络仍受限 | 包络链 | 打开 sandbox 深度页、调整 allowOnly / network restriction | 回读 sandbox / path 包络 | `pathValidation.ts`、`SandboxConfigTab.tsx` |
| MCP failed / disabled / need auth | 外部能力链 | `mcp_status`、`mcp_reconnect`、`mcp_toggle`、`reload_plugins` | 再读 `mcp_status` | `controlSchemas.ts`、`McpServerStatusSchema` |
| managed dangerous settings 新增或变化 | 增量审批链 | accept / reject | 回读 active settings 是否变化 / 是否沿用 cached settings | `securityCheck.tsx`、`remoteManagedSettings/index.ts` |

## 3. 这张表最重要的启示

统一安全控制台的目标不是只把问题分类。  
它必须进一步把每一类问题都绑定到：

1. 一条断裂证明链
2. 一个最短动作
3. 一个默认回读验证

否则“解释”不会自动变成“闭环”。

## 4. 最短用法

- 看到一个状态，先查它最可能对应哪条证明链
- 再执行对应的最短动作
- 最后按表里的默认回读路径验证是否真的修复

## 5. 最终用途

这张表的用途，是把 `25` 的长文压成工程实现和产品交互都能直接复用的最短诊断矩阵。
