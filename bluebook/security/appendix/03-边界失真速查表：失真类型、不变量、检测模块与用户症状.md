## 2. 五类边界失真总表

| 失真类型 | 典型诱因 | 被破坏的不变量 | 主要检测 / 修复模块 | 第一硬拦截层 | 用户侧典型感知 | 最短结论 |
| --- | --- | --- | --- | --- | --- | --- |
| 规则失真 | 危险 allow rule、tool-wide ask/deny 阴影化、坏 rule 污染 settings | 结构性危险高于当前模式便利；局部坏值不应毒化全局配置 | `dangerousPatterns.ts`、`permissionSetup.ts`、`shadowedRuleDetection.ts`、`validation.ts` | auto mode 入口前的规则剥离；不可达规则检测；局部过滤 | “我明明写了 allow，为什么还是不生效？” | 写在配置里的规则，不等于真正仍在仲裁链里生效的规则 |
| 指称失真 | shell expansion、UNC、tilde 变体、glob 写路径、连接真实对象被包装 | 语义等价高于文本等价 | `pathValidation.ts`、`mcp/config.ts` | 路径输入预处理；canonicalization；真实连接签名比较 | “我批准的是这个路径，为什么最后碰的是另一个对象？” | 最危险的不是字符串复杂，而是批准对象和执行对象脱钩 |
| 主权失真 | low-trust settings/env 覆盖高风险主链；连接定义权从 managed 滑向 user | 来源合法性高于配置值本身 | `managedEnv.ts`、`settings.ts`、`pluginOnlyPolicy.ts`、`mcp/config.ts` | source gate；first-source-wins；managed-only / plugin-only 收口 | “为什么这值看起来合法却还是不允许生效？” | 真正高风险的是谁有资格定义运行时，而不是值表面像不像合法 |
| 接口失真 | 把读取例外误当成通用外联；把连接入口误当成可信能力；把输出预算当成纯性能问题 | 外部能力必须按能力语义治理 | `WebFetchTool/preapproved.ts`、`mcpValidation.ts`、`mcp/config.ts` | WebFetch 与 sandbox 网络分离；MCP policy filter；输出截断/压缩 | “既然能 fetch，为什么不能把同站点当网络白名单？” | 外部入口的风险不在“有没有接口”，而在接口语义会不会被偷换 |
| 解释失真 | 真实边界存在，但用户看不懂哪一层在生效 | 解释能力是边界的一部分 | `permissionExplainer.ts`、状态展示组件、managed security dialog | 结构化解释与状态面呈现 | “系统为什么挡我？到底是权限、来源、还是策略问题？” | 不可解释的安全最终会在用户侧退化成黑箱，并诱发新的绕路行为 |

## 3. 从“失真”回看“检测链”

把这五类失真反过来看，就能更清楚地理解为什么 Claude Code 要把安全检测拆成多条链：

1. 规则链处理规则失真
2. 路径链处理指称失真
3. 来源链处理主权失真
4. 外部入口链处理接口失真
5. 解释层处理解释失真

换句话说，安全检测链不是按文件夹随便长出来的。  
它们是在不同节点，分别修复不同类型的边界变形。

## 4. 从“失真”回看“用户症状”

用户侧最常见的困惑，往往正是失真被系统纠正后的感知结果：

1. “规则写了但不生效”通常对应规则失真
2. “明明是这个路径，为什么系统还不让我做”通常对应指称失真
3. “值合法却不允许覆盖”通常对应主权失真
4. “同样是外部能力，为什么这个能读那个不能发”通常对应接口失真
5. “系统挡了我，但我不知道是哪一层在挡”通常对应解释失真

所以速查时，不一定非要从源码入口开始。  
也可以先从用户症状倒推失真类型，再倒推检测模块。

## 5. 最短用法

- 想从源码入口倒推系统为什么这样设计：先读 `18`，再对照这张表
- 想从设计原则倒推哪些东西绝不能被局部优化打破：先读 `19`，再对照这张表
- 想从第一性原理理解这些现象为什么能统一：先读 `20`，再回到这张表
