# 安全检测速查表：风险对象、检测模块与不可绕过层

## 1. 这一页怎么用

这一页不是完整论证，  
而是把安全检测专题里最常复用的问题压成一张速查表。

适合两种场景：

1. 已经知道自己在查哪类风险对象，只想快速定位源码模块
2. 想先判断“这一层到底能不能被 classifier、allow rule 或 mode 绕过去”

## 2. 速查矩阵

| 风险对象 | 典型场景 | 主要检测模块 | 第一硬拦截层 | classifier 是否能替代 | 最短结论 |
| --- | --- | --- | --- | --- | --- |
| 危险 allow rule | `Bash(*)`、`Bash(python:*)`、`PowerShell(iex:*)`、`Agent(*)` | `dangerousPatterns.ts`、`permissionSetup.ts` | auto mode 入口前的规则剥离/识别 | 否 | classifier 之前先防仲裁层被 allow rule 架空 |
| 工具级 deny / ask 阴影化 | 写了 `Bash(ls:*)`，同时存在 `Bash` deny 或 ask | `shadowedRuleDetection.ts`、`permissions.ts` | rule 可达性检测 | 否 | 系统治理的不只是坏规则，还有永远到不了的假规则 |
| 敏感路径写入 | `.git/`、`.claude/`、shell config、危险编辑目标 | `pathValidation.ts`、相关 filesystem safety check | `safetyCheck` | 部分可以，部分不可以 | 有些 ask 属于 bypass-immune，也不该交给 classifier 吞掉 |
| 路径语义漂移 | `~root`、`$HOME`、`$(cmd)`、UNC、写入 glob | `pathValidation.ts` | 输入预处理与路径语义校验 | 否 | 真正危险的是校验对象与执行对象不是同一路径语义 |
| 高不可逆删除 | 根目录、家目录、盘符根、通配删除 | `pathValidation.ts` | 删除危险目标判定 | 否 | 可写不等于可安全破坏，系统单独治理不可逆性 |
| 工作目录便利绕过安全检查 | “在 cwd 里，所以应该直接放行” | `pathValidation.ts`、`permissions.ts` | `safetyCheck` 位于 working-dir allow 之前 | 否 | 安全检查先于目录便利，先于 `acceptEdits` |
| WebFetch 读取白名单被误当网络白名单 | 文档站点可读，但站点同时支持上传/POST | `WebFetchTool/preapproved.ts` | WebFetch 与 sandbox 网络权限分离 | 否 | GET 读取白名单绝不等于外联写权限白名单 |
| MCP 输出污染半径过大 | 外部工具返回超长文本或图像 | `mcpValidation.ts` | token 预算、截断与压缩 | 否 | 输出规模本身就是安全面，因为它决定上下文污染半径 |
| MCP 连接定义权失控 | 用户私配 server、enterprise 独占、plugin-only 锁定 | `services/mcp/config.ts`、`pluginOnlyPolicy.ts` | policy filter / enterprise exclusive control | 否 | MCP 的核心风险不是“能不能连”，而是谁定义“连什么” |
| provider / auth 主链被配置污染 | settings.env 覆盖 host-managed provider 或宿主关键 env | `managedEnv.ts` | env 过滤与来源分层 | 否 | 真正高风险的是低信任来源悄悄改写运行时主链 |
| 坏 permission rule 毒化整份 settings | 一个非法 rule 让整个文件失效 | `validation.ts`、`settings.ts` | 局部过滤后再 schema parse | 否 | 系统偏好缩小坏值破坏半径，而不是一次配置错误导致全局瘫痪 |
| 定制面主权漂移 | user/project source 试图接管 plugin-only customization surface | `pluginOnlyPolicy.ts` | managed policy 的 source gate | 否 | “谁能扩展系统”本身就是正式安全策略 |

## 3. 最常见的四个误判

### 误判一：有 allow rule 就等于后面都不会再挡

错。  
如果这条 allow rule 本身会提前架空仲裁层，  
系统会先把它当成风险对象治理。

### 误判二：classifier 是最终裁判

错。  
classifier 是中段裁判，  
前面还有 rule、tool-specific check、`safetyCheck` 与来源过滤。

### 误判三：在工作目录里的路径天然更安全

错。  
路径是否在 cwd，只是便利条件；  
它排在敏感路径安全检查之后。

### 误判四：能读某外部站点，就说明系统信任那个站点

错。  
Claude Code 允许的是某种读取语义，  
不是把整个站点升级成通用可信网络主体。
