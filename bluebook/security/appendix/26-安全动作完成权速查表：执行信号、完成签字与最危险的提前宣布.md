# 安全动作完成权速查表：执行信号、完成签字与最危险的提前宣布

## 1. 这一页服务于什么

这一页服务于 [42-安全动作完成权：为什么动作执行不等于动作完成，必须由 authoritative 回读签字](../42-%E5%AE%89%E5%85%A8%E5%8A%A8%E4%BD%9C%E5%AE%8C%E6%88%90%E6%9D%83%EF%BC%9A%E4%B8%BA%E4%BB%80%E4%B9%88%E5%8A%A8%E4%BD%9C%E6%89%A7%E8%A1%8C%E4%B8%8D%E7%AD%89%E4%BA%8E%E5%8A%A8%E4%BD%9C%E5%AE%8C%E6%88%90%EF%BC%8C%E5%BF%85%E9%A1%BB%E7%94%B1authoritative%E5%9B%9E%E8%AF%BB%E7%AD%BE%E5%AD%97.md)。

如果 `42` 的长文解释的是为什么“动作已执行”不能直接等于“动作已完成”，  
那么这一页只做一件事：

`把不同动作链条里的执行信号、真正完成签字信号，以及最危险的提前宣布压成一张矩阵。`

| 动作 / 场景 | 你先看到的执行信号 | 真正的完成签字信号 | 明确还不够的信号 | 最危险的提前宣布 | 关键证据 |
| --- | --- | --- | --- | --- | --- |
| 主 turn 结束 | 局部结果已生成、输出暂时停住 | `session_state_changed(idle)` | “现在看起来没输出了” | 把暂时空闲误写成 turn 已完成 | `coreSchemas.ts:1738-1746`、`sdkEventQueue.ts:56-61` |
| going idle 之前 | finally 已开始收尾 | `flushInternalEvents()` 完成后才能发 idle | 本地 finally block 已进入 | 在内部事件未 flush 前切 idle | `print.ts:2454-2466` |
| transcript / internal event 交付 | queue 已 enqueue / flush 已返回 | 还需 server state separately 确认 | 本地 queue drain | 把 delivery confirmation 当成远端已承认 | `ccrClient.ts:824-829` |
| 文件持久化 | 普通 result 或动作输出已出现 | `files_persisted` 事件 | 结果文本、turn 摘要 | 把“已写过”误写成“已持久化成功” | `coreSchemas.ts:1672-1692`、`print.ts:2260-2269` |
| bridge 初始 flush | 新消息已产生 | flush gate 结束并 drain 完 queued items | 消息已进 gate 队列 | 把 queued 当 committed | `flushGate.ts:1-15` |
| 401 recovery 窗口中的控制动作 | 本地似乎已发出 request/response/result | 恢复结束后新的 authoritative 状态 | recovery 期间的局部发送尝试 | 在不可靠窗口里宣布动作已完成 | `remoteBridgeCore.ts:824-878` |
| 权限请求进入审批 | `requires_action` 已上报 | 后续 running / idle 与控制链收口 | 只看到 request 已发出 | 把“已进入审批”误写成“审批已完成” | `coreSchemas.ts:1738-1746`、`structuredIO.ts:585`、`ccrClient.ts:644-657` |

| rewind 真正执行 | 恢复逻辑已触发 | 文件层后续状态与工作区回读 | “按钮已点”“函数已跑” | 把执行尝试误写成工作区已稳定回滚 | `print.ts:2996-3010` |

## 3. 最短判断公式

看到一条“完成了”的声明时，先问四句：

1. 这只是执行痕迹，还是 authoritative completion signal
2. 中间还有没有 queue、flush、persist、recovery 这些未收口层
3. 这条完成信号有没有在源码里被明确命名为 authoritative
4. 如果我现在就宣布完成，会不会把局部成功误写成全链成功

## 4. 最常见的四类提前宣布错误

| 提前宣布方式 | 会造成什么错觉 |
| --- | --- |
| 把“没输出了”当成 `idle` | turn 还没真正收尾就被当成完成 |
| 把 queue drain 当成 server state committed | 远端未承认的事实被说成已落成 |
| 把 result 当成 persisted | 文件层失败被 UI 文案吞掉 |
| 把 recovery 窗口里的局部发送当成完成 | 不可靠状态里出现伪成功、伪关闭、伪回写 |

## 5. 一条硬结论

对 Claude Code 这类安全控制面来说，  
最危险的不是完成信号太少，  
而是：

`本来还在收口中的动作，被下游提前宣布成了已经完成。`
