## 2. 恢复中间态矩阵

| 中间态 | 典型来源 | 当前可见账本 / 事实 | 当前可用动作 | 禁止文案 | 默认跳转 / 默认动作 | 关键证据 |
| --- | --- | --- | --- | --- | --- | --- |
| `mirror-only` | `outboundOnly`，只开写链不上读流 | 仅能确认 write path 与事件转发；无 inbound prompt / control 证明 | 低风险写出、镜像转发、状态观察 | `Remote Control active`、`Fully connected` | 跳转 bridge 详情，提示“附着但仅镜像” | `replBridgeTransport.ts:142-147,336-345` |
| `write-ready / read-blind` | `isConnectedStatus()` 仅签 write-readiness | 写链就绪；read path 未被同等证明 | 只依赖写链的动作 | `Connected`、`Recovered`、`Interactive ready` | 显示读链未证成，禁止依赖入站控制的动作 | `replBridgeTransport.ts:289-299` |
| `restored-but-not-fresh` | `state_restored`、`restoredWorkerState`、restore+hydrate 正在收敛 | 旧状态已接回；fresh state 尚未说满；避免 fresh default | 查看已恢复状态、等待 fresh 回读、允许低风险诊断 | `Up to date`、`Fully restored` | 跳转状态页或等待 fresh state，同步保留降级提示 | `ccrClient.ts:474-522`、`print.ts:5048-5058`、`structuredIO.ts:139-142` |
| `control-narrow` | 仅支持 `can_use_tool`，其他 subtype unsupported | 权限请求路径存在；控制协议全集不存在 | 处理权限请求、有限控制动作 | `Remote control ready`、`Control channel restored` | 跳转能力说明，禁用不支持 subtype 的动作 | `directConnectManager.ts:81-98`、`RemoteSessionManager.ts:192-213` |
| `result-ready / effects-pending` | result 已出，`files_persisted` 未到 | 结果已生成；持久化副作用未获签字 | 查看结果、等待落盘确认、避免宣称“已完成全部变更” | `Completed`、`All changes applied` | 等待 `files_persisted` 或等效后效应 signer | `print.ts:2261-2267` |
| `state-ready / explanation-pending` | `idle` / state signer 已到，但账本或盲区未补齐 | 会话阶段已回到 idle；解释 ceiling 仍未恢复说满 | 浏览状态、执行低风险不依赖完整解释权的动作 | `All clear`、`No blindspots` | 跳转完成卡 / 资格卡，继续补账本或换宿主 | `sdkEventQueue.ts:56-66`、`49`、`50` 主线归纳 |

## 3. 最短判断公式

看到任一“绿色”状态时，先问四句：

1. 它到底是活性恢复、restore 恢复、state 恢复、effect 恢复，还是 explanation 恢复
2. 它现在缺的是读链、控制面、fresh 性，还是账本完整性
3. 现在还能做什么，不能做什么
4. 如果继续沿用统一“已恢复”文案，最先误导哪一步

## 4. 最常见的四类中间态压平错误

| 压平错误 | 后果 |
| --- | --- |
| 把 mirror-only 压成 active | 用户会以为入站 prompt / control 也已恢复 |
| 把 write-ready/read-blind 压成 connected | 用户会把写链恢复误读成读链恢复 |
| 把 restored-but-not-fresh 压成 fully restored | 用户会把旧状态接回误读成当前状态已 fresh |
| 把 result-ready/effects-pending 压成 completed | 用户会误以为所有持久化后效应都已完成 |

## 5. 一条硬结论

对 Claude Code 这类多宿主、多账本控制面来说，  
最危险的不是中间态太多，  
而是：

`明明系统只恢复到了某一半，却仍被统一压成一句没有缺口信息的“已恢复”。`
