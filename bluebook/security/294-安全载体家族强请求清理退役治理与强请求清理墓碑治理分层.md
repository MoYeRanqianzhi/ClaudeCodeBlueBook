# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `293` 之后还必须继续写 `294`

`293-安全载体家族强请求清理迁移治理与强请求清理退役治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样过渡与何时正式结束，也仍需要 sunset governor 去决定兼容世界在哪一刻不再算 current truth。`

但如果继续往下追问，
还会撞上一层更隐蔽、也更容易被偷写成“宣布退役就够了”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界不再算 current truth，它就自动拥有决定退役后还应留下什么最小残留事实的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 自己已经落地的安全机制，
会发现它在多个子系统里都把：

`old world 何时结束`

和

`结束之后还留下什么最小残留标记，继续约束未来解释`

拆成了两层不同治理。

最硬的证据至少有七组：

1. `src/query.ts:712-719`
   遇到 orphaned assistant messages 时不会只静默丢弃旧对象，
   而会显式 `yield { type: 'tombstone' }`
2. `src/utils/messages.ts:2954-2957`
   与
   `src/QueryEngine.ts:757-760`
   证明 `tombstone` 不是普通内容，而是后续消费者必须识别的 control signal
3. `src/screens/REPL.tsx:2646-2648`
   与
   `src/utils/sessionStorage.ts:121-123,925-945,1470-1473`
   又证明 tombstone 会继续驱动 UI 与 transcript 删除链，甚至需要专门的 slow-path rewrite 上限
4. `src/utils/plugins/cacheUtils.ts:56-72,149-176`
   与
   `src/services/plugins/pluginOperations.ts:523-547`
   证明 `.orphaned_at` 不是 sunset 时点本身，而是对象退役后留下的最小 marker
5. `src/utils/plugins/orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
   与
   `src/main.tsx:2544-2568`
   进一步证明 marker 还会被翻译成 grep / glob exclusion grammar，继续约束未来读取世界
6. `src/utils/config.ts:427-438`
   与
   `src/hooks/notifs/useModelMigrationNotifications.tsx:5-8,21-33,49-50`
   说明对象世界已经变化之后，系统有时仍要保留最小历史注脚，向当前 runtime 解释“为什么现在变了”
7. `src/services/plugins/pluginOperations.ts:543-545`
   甚至直接把 `tombstones` 当成应避免的 teardown fallout 来讨论，
   公开承认 post-retirement residue 不是副现象，而是受治理对象

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request cleanup 世界从什么时候起不再算当前正式世界。`

后者才配说：

`旧世界结束之后，还要留下什么最小残留事实，让后来的 runtime、搜索面、恢复链与审计者不会把“已经退役”重新读成“从未发生”或“仍可继续援引”。`

所以 `293` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责宣布结束；stronger-request cleanup tombstone governor 才负责治理结束之后留下的最小残留事实。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开 tombstone manager，
而是在说：

1. repo 在 message、plugin 与 model 线已经展示出真实的 tombstone-like grammar
2. 这些 grammar 不等于 sunset deadline 本身
3. 它们承担的是 post-retirement minimal residue governance

第二条：

这里说的 `tombstone governance`
不是在声称：

`退役之后一定要保留完整旧对象。`

更稳的说法只能是：

`墓碑治理回答的是退役后保留多少最小事实，既足以约束未来解释，又不把旧对象重新抬回 current world。`

第三条：

这里也不是把：

`migration timestamp`

一概硬写成字面 tombstone。

当前更稳的说法只能是：

`有些残留物虽然源码里不直接叫 tombstone，但功能上承担了 post-transition trace artifact 的角色，因此属于 tombstone-like marker。`

第四条：

这里尤其不能过度声称：

`marker 还在`

就等于：

`旧世界仍然有效。`

更稳的说法只能是：

`marker presence` 说明系统还在保留“旧世界已经失效”这一最小事实，
而不是在恢复旧世界的 current validity。`

第五条：

这里也不能把：

`warning surface`

偷换成：

`完整 tombstone grammar`。

更稳的说法只能是：

`warning` 主要面向人类解释，
而 tombstone 还要继续回答 marker type、marker consumer、future effect 与 final cleanup responsibility。`

第六条：

这里还不能过度声称：

`Claude Code 既然已经在 message/plugin/model 线展示 tombstone grammar，stronger-request cleanup 线未来就必须逐字照抄。`

更稳的说法只能是：

`不同对象家族可以有不同最小残留形式，但“谁来决定退役后留下什么最小机器可消费事实”必须被正式回答；不能因为 marker 形式可变，就把 tombstone 主权偷写成 sunset 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer`

证据：

1. `query.ts`
   遇到 orphaned assistant message 时会显式产出 `tombstone`
2. `utils/messages.ts`
   与
   `QueryEngine.ts`
   证明 tombstone 是 control signal，而不是普通消息内容
3. `REPL.tsx`
   与
   `sessionStorage.ts`
   证明 tombstone 会继续驱动显示层与 transcript 存储层改口
4. `.orphaned_at`
   不是 sunset 时点本身，而是对象退役后留在磁盘上的最小 marker
5. marker 还会继续塑造 future search / visibility world
6. migration timestamp 证明对象世界变化之后，系统有时还要保留一个最小历史注脚去解释“为什么当前世界刚变”
7. stronger-request cleanup 当前虽已开始显露 sunset-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup path、旧 promise vocabulary 与旧 receipt semantics 退役后还应留下什么最小 marker

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留事实来约束未来解释。`

再压成一句：

`会退役，不等于会留下可治理的最小残留事实。`

再压成更哲学一点的话：

`制度知道何时关门，不等于制度知道门后还要挂什么告示。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
这两层治理处理的是两个不同真相面的主权问题。

`sunset governor`

回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

`tombstone governor`

回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. marker 将怎样约束未来读取、恢复、审计与解释
5. marker 何时再被进一步清除，谁对这次二次清除负责

所以这两层的哲学本质不同：

`sunset`

决定旧对象什么时候停止作为 current truth；

`tombstone`

则决定它停止之后，
系统还保留什么最小事实来约束后来者。

Claude Code 在这里真正先进的地方，
不是它“会在对象无效后把它删掉”，
而是它已经明确承认：

`删除不是全部，最小残留事实同样需要被制度化。`

这是一种很成熟的安全哲学：

`对象停止有效，不等于它留下的解释义务也随之消失。`

因为一旦把 sunset 与 tombstone 压成一层，
系统就容易出现三种危险：

1. 旧对象虽然不再有效，但未来读者不知道它为什么消失
2. 旧对象虽然已经失效，但机器世界缺乏最小 marker 去防止误复活
3. 删除动作虽然完成，但搜索、恢复、通知与审计面仍可能继续引用旧残影

## 5. 从技术角度看，Claude Code 的先进性在于把“退出真相面后的最小残留”做成可继续消费的制度，而不是把删除当唯一答案

### 5.1 message 线：对象退役之后，系统可能需要显式 tombstone 来告诉后续消费者“该删谁”

`query.ts:712-719`
很关键。
当 streaming fallback 产生 orphaned messages 时，
系统不是只把旧消息从内存数组里粗暴过滤掉。

它会：

1. 为每个 orphaned message 产生 `type: 'tombstone'`
2. 记录 `tengu_orphaned_messages_tombstoned`
3. 清空与旧尝试关联的临时结构

随后，
`utils/messages.ts:2954-2957`
把 tombstone 路由到 `onTombstone`；
`QueryEngine.ts:757-760`
明确把 tombstone 视为 control signal；
`REPL.tsx:2646-2648`
与
`sessionStorage.ts:925-945,1470-1473`
再把它落实为 UI / transcript 删除动作。

更进一步，
`sessionStorage.ts:121-123`
甚至要为 tombstone slow path 单独设定 `MAX_TOMBSTONE_REWRITE_BYTES`，
因为这不是可有可无的附带语义，
而是必须被认真治理的后续动作链。

这条链条显示出一个关键先进性：

`失效判定`

`残留表达`

`后续消费`

被拆成了连续但不等价的三层。

### 5.2 plugin 线：`.orphaned_at` 不是垃圾残留，而是被治理的机器墓碑

`cacheUtils.ts:56-72`
的 `markPluginVersionOrphaned()` 与注释已经说明：

1. 插件被卸载或更新到新版本后，旧版本先被打上 `.orphaned_at`
2. 这个 marker 会被后续 orphan cleanup 继续消费

`pluginOperations.ts:523-547`
也明确把最后一个 scope 卸载后的 install path 交给 `markPluginVersionOrphaned()`，
并且在注释里直写：

`Blocking creates tombstones`

这说明在 plugin 线里，
系统已经把 tombstone-like fallout 当成正式治理对象，
而不是副现象。

更重要的是
`cacheUtils.ts:149-176`
里的 `processOrphanedPluginVersion()`：

1. 如果 marker 不存在，先写 marker
2. 如果 marker 超过 7 天，再删目录

这说明 `.orphaned_at` 的职责不是：

`对象还活着`

而是：

`对象已经退役，但系统仍需留下一个最小残留标记来决定后续怎么处理它。`

### 5.3 marker-driven exclusion：墓碑不是只服务历史解释，它还会主动约束未来读取世界

`orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
与
`main.tsx:2544-2568`
特别值钱。

这里的流程是：

1. 先完成 orphan GC
2. 再扫描 `.orphaned_at`
3. 基于 marker 生成 grep / glob exclusion patterns
4. 把 exclusion cache 冻结在当前 session

这意味着 `.orphaned_at` 的作用不只是：

`证明旧版本退役过`

它还会继续决定：

`未来的搜索与读取不应再触及哪些目录。`

这把墓碑治理从：

`历史注脚`

提升成了：

`未来约束`

也就是说：

`成熟的 tombstone，不只是留下痕迹，还会继续约束后来者别再走回旧世界。`

### 5.4 model 迁移时间戳：有些墓碑不长得像 tombstone，但功能上就是“变化已经发生过”的最小残留事实

`config.ts:427-438`
定义了：

1. `opusProMigrationComplete`
2. `opusProMigrationTimestamp`
3. `legacyOpusMigrationTimestamp`
4. `sonnet45To46MigrationTimestamp`

这些字段都不是迁移动作本身，
也不是 retirement date 本身。
它们是迁移之后留在 `GlobalConfig` 里的最小事实残留。

`useModelMigrationNotifications.tsx:5-8,21-33,49-50`
再说明这些字段的用途：

1. 判断这次启动是否刚刚发生过模型迁移
2. 只发一次高优先级提示
3. 在 `legacyOpusMigrationTimestamp` 场景里额外给出 opt-out 说明

这里必须谨慎说：
源码没有把这些字段字面命名成 tombstone。
但从功能上看，
它们确实承担了 tombstone-like marker 的职责：

`它们不是迁移动作，而是迁移动作完成后保留下来的最小历史事实，用于向当前系统解释“为什么现在变了”。`

## 6. 对 stronger-request cleanup 而言，当前真正缺的不是“何时退役”，而是“退役后留下什么最小 anti-resurrection residue”

当前 stronger-request cleanup 已经开始显露 sunset-governance，
但仍缺少 tombstone 治理回答：

1. 旧 startup wording 退役后，系统是否需要留下一个最小 marker 去说明“这不是从未承诺过，而是旧承诺已失效”
2. 旧 home-root `plans` cleanup law 退出 current world 之后，是否还需要 tombstone-like residue 去约束未来恢复、搜索或解释
3. 旧 `CleanupResult` / local count semantics 退役后，是否要留下一个最小 post-retirement trace，防止后来者把“当前无 receipt”误读成“历史上没有过 cleanup fact”
4. 旧 promise vocabulary 与旧 covered-family contract 退出 current truth 后，系统是否需要保留 anti-resurrection residue，避免未来文档、运行时或 operator 再把它们误援引

这些都不是：

`何时不再有效`

能回答的。

它们已经进入：

`tombstone governance`

层。

也就是说，
stronger-request cleanup 今天真正缺的，
不是再补一个 sunset cutoff，
而是：

`谁来决定旧 cleanup truth 退场后还要留下什么最小残留事实，来防止误恢复、误解释与误复活。`

## 7. 苏格拉底式自反诘问：我是不是又把“已经会退役”误认成了“已经会处理退役后的残留事实”

如果对这一层做更严格的自我审查，
至少要连续追问六句：

1. 如果旧世界已经不再有效，为什么还要留下 marker？
   因为“失效”只解决 current validity，不解决未来消费者如何知道该删谁、避开谁、解释什么。
2. 如果 marker 还在，这是否意味着旧世界仍然有效？
   不是。marker 的职责恰恰是把“旧世界已经失效”以机器还能继续消费的形式留下来。
3. 如果系统已经给过 warning 或 notification，是否就等于拥有完整 tombstone governance？
   还不等于。完整墓碑治理至少还要回答 marker 是什么、谁消费 marker、marker 继续影响哪些 future surface、marker 何时再被清除。
4. 如果 migration 已经完成，为什么还要继续区分 sunset 与 tombstone？
   因为 migration 解决世界如何切换，sunset 解决旧世界何时停止，tombstone 解决停止之后留下什么最小残留事实。
5. 如果删除动作已经存在，为什么还说 tombstone 仍是独立层？
   因为删除只是一次当前动作；墓碑治理回答的是删除前后哪些最小事实仍要继续约束后来者。
6. 如果 cleanup 线当前还没有显式 tombstone code，是不是说明这层还不值得写？
   恰恰相反。越缺正式 tombstone grammar，越容易把“旧世界已结束”误读成“旧世界从未存在”或“旧世界随时可被误复活”。

这一串反问最后逼出一句更稳的判断：

`tombstone 的关键，不在系统会不会多写一个标记文件，而在系统能不能把“退役后的最小残留事实”治理成一个持续约束未来解释的制度。`

## 8. 从安全设计哲学看，这一层真正回答的是“制度是否承认历史残留也是安全对象”

`sunset`
更像制度终止 current truth 的主权。

`tombstone`
则更像制度处理 historical residue 的主权。

一个系统可以知道：

`我现在怎样让旧世界退出 current world。`

但如果它还不能明确回答：

`退出以后到底留下什么最小可消费事实、谁继续消费它、它怎样继续约束未来搜索 / 恢复 / 解释 / 审计、以及何时才能被二次清除。`

那么这套制度仍然更接近：

`会关停旧世界的制度`

而不是：

`会治理旧世界残留义务的制度`

Claude Code 当前 stronger-request cleanup 线，
恰好就停在这个哲学分界点上：

1. 它已经不再只是不会自述、不会兑现、不会自检、不会纠偏、不会终止兼容世界的 cleanup world
2. 它已经开始学会正式宣布旧 world 的结束
3. 但它还没有 equally-strong 地展示旧 cleanup truth 结束后要留下什么 anti-resurrection residue

所以这一章真正要钉死的，
不仅是一个工程判断，
也是一个更深的安全设计判断：

`制度知道何时关门，不等于制度已经知道门后还要挂什么告示。`

## 9. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

因为：

1. sunset 负责终止 current validity
2. tombstone 负责治理 post-retirement minimal residue
3. message tombstones、plugin markers、marker-driven exclusion 与 migration timestamps 都证明 repo 已把这两层正式拆开
4. stronger-request cleanup 当前虽已开始理解旧世界何时结束，却还没有对等地回答结束后留下什么最小 marker 来防止误复活与误解释

因此，
这一章最终必须压成一句：

`会退役，不等于会留下可治理的最小残留事实。`
