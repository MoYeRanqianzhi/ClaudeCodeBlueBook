# 安全状态面与可解释性产品化：系统已经有零件，但还缺一张统一仪表盘

## 1. 为什么还要单独写这一章

前面的 `00-09` 章已经把：

- 安全控制面的层次
- 权限状态机
- 配置主权
- 外部入口风险

都拆得很细了。

但如果站在真正的产品视角，还会剩下一个更尖锐的问题：

`Claude Code 内部已经有大量安全边界与解释零件，为什么用户仍然容易把系统体验成“黑箱”？`

这一章回答的正是这个问题。

## 2. 先说结论

Claude Code 在安全产品化上已经具备很多关键零件：

1. 有权限请求解释文案
2. 有按 source 呈现配置来源的能力
3. 有 permission explainer 这种按需解释器
4. 有 managed settings 危险变更审批对话框
5. 有高阶 entitlement 的具体失败原因文本
6. 有 notifications 队列可以承接 gate 变化

但这些零件目前仍然是分散的。  
它们还没有完全汇成一张统一的安全状态面。

所以这套系统当前最值得继续进化的方向不是：

- 再加一个 deny rule

而是：

`把已经存在的解释零件收束成一张用户能持续读取的安全仪表盘。`

## 3. 源码已经有哪些“解释层零件”

### 3.1 权限请求解释文案

`permissions.ts` 里的 `createPermissionRequestMessage()` 已经不是简单输出“需要权限”。  
它会按不同决策原因生成不同文案，例如：

- hook 阻断
- rule 命中
- subcommand results
- permission prompt tool
- sandbox override
- working directory
- safety check
- 当前 permission mode

这意味着系统内部已经承认：

`用户需要知道“为什么问我”，而不只是“现在来点允许/拒绝”。`

### 3.2 按需权限解释器

`permissionExplainer.ts` 与 `PermissionExplanation.tsx` 又往前走了一步：

- 用户可按需触发解释
- 系统会生成风险等级、解释、原因和风险提示
- UI 里还有 lazy promise、loading shimmer、不可用回退

这说明 Claude Code 已经不满足于：

- 只给结果

它开始尝试给：

- 风险解释
- 行动理由
- 面向用户的短文本摘要

### 3.3 配置来源可见性

`status.tsx` 里的 `buildSettingSourcesProperties()` 能把当前设置来源以更可读的名称呈现出来，  
包括：

- User
- Project
- Local
- Enterprise managed settings（还区分 remote / plist / file / HKLM 等来源）

这说明系统内部已经具备把“来源”转译给用户的能力。  
这点非常重要，因为安全专题前面已经反复证明：

`来源本身就是一级安全变量。`

### 3.4 managed settings 危险变更审批

`remoteManagedSettings/securityCheck.tsx` 与 `ManagedSettingsSecurityDialog.tsx` 已经形成了一个很强的安全产品化原型：

- 只在 dangerous settings 变化时弹出
- 非交互模式跳过
- 交互模式下要求显式接受或退出
- 文案直接告诉用户，这些设置可能允许任意代码执行或截获 prompts / responses

这实际上已经是一张非常明确的“安全告警面”。

### 3.5 高阶 entitlement 的可解释失败原因

`bridgeEnabled.ts` 里的 `getBridgeDisabledReason()` 也很关键。  
它不会只返回一个布尔值，而是区分：

- 没有 claude.ai subscription
- token scope 不够
- 无法确定 organization
- account 还没被 gate 放行

这说明 Claude Code 在高阶资格面上已经具备：

`从“不可用”走向“为什么不可用”的解释能力。`

### 3.6 gate 变化的通知通道

`bypassPermissionsKillswitch.ts` 说明：

- auto mode gate 的变化已经可以通过 notifications queue 告知用户

这意味着安全边界变化不是只能藏在日志里。  
系统已经有向用户显式通知的机制。

## 4. 这些零件为什么还不足以构成完整安全状态面

因为它们目前大多是分散、局部、按需触发的。

### 4.1 解释散落在不同交互节点

现在的解释层更像：

- 问权限时解释一点
- Remote Control 失败时解释一点
- managed settings 变化时解释一点

而不是：

- 持续可读的一张总面板

### 4.2 用户仍然缺一张“当前安全语义总览”

用户依然难以在同一处直接看到：

- 当前 permission mode
- 当前 bypass / auto 是否可用
- 当前有哪些 managed-only 收口在生效
- 当前 settings source 到底有哪些
- 当前 high-risk entitlement 为什么不可用
- 当前 hooks / MCP / gateway 的主权归属是什么

而这些恰恰是 `security/07-09` 证明过最重要的边界信息。

### 4.3 解释层还没有完全和风险矩阵打通

源码里已有：

- 风险等级
- source 展示
- gate disabled reason
- managed settings dangerous change approval

但这些仍未汇成一套统一的“风险状态语言”。

用户很难直接从界面得到类似下面的结论：

- 这是 source 风险
- 这是 entitlement 风险
- 这是 external-input 风险
- 这是 execution risk

## 5. 如果要产品化成一张统一安全仪表盘，最值得放哪些字段

基于现有源码结构，最自然的一张安全状态面至少应包含六组字段。

### 5.1 当前 mode 与仲裁状态

- `permissionMode`
- `isAutoModeAvailable`
- `isBypassPermissionsModeAvailable`
- 当前是否存在 stripped dangerous rules

### 5.2 当前设置来源与主权归属

- 当前启用的 setting sources
- policySettings 的来源类型
- 是否存在 host-managed provider vars
- 是否处于 plugin-only / managed-only customization 策略下

### 5.3 当前高阶 entitlement 状态

- bridge / Remote Control 是否可用
- 若不可用，`getBridgeDisabledReason()` 的原因
- 当前登录方法与账户类型

### 5.4 当前外部入口收口状态

- MCP 是否受 managed-only 约束
- hooks 是否受 managed-only 约束
- sandbox network 是否处于 managed domains only
- gateway / provider 是否由 host 管理

### 5.5 当前危险变更审批状态

- 最近一次 managed settings 危险变更是否已被用户接受
- 当前是否存在未解释的危险设置变化

### 5.6 当前解释层可用性

- permission explainer 是否启用
- 当前 permission request 的解释来源是什么
- always-allow 选项是否因 managed-only 被隐藏

## 6. 从技术角度看，这一步为什么值钱

因为 Claude Code 现在已经拥有足够多的内部状态和解释零件。  
再往前一步，收益会非常大：

1. 减少“黑箱安全感”
2. 减少用户误解 mode / managed-only / entitlement
3. 减少支持成本
4. 把安全从“源码里存在”变成“产品里可见”

这是一条非常现实的产品化路径：

`先有内部边界，再把边界可视化。`

## 7. 从第一性原理看，为什么安全系统必须可解释

如果从第一性原理压缩，安全系统至少要同时满足两件事：

1. 真正限制危险动作
2. 让合法用户理解限制为何存在

如果只满足第一条，  
系统会在用户眼里退化成：

- 任意
- 不透明
- 像被针对

Claude Code 的源码其实已经显露出一个相当成熟的方向：

- 它并不满足于只是 deny
- 它开始解释 deny、解释 ask、解释 gate、解释 managed change

这说明它已经在向“可解释安全”迈进。  
只是目前还没完全汇成统一产品形态。

## 8. 用苏格拉底式追问再压一层

### 问：为什么我说 Claude Code 已经有安全状态面的零件？

答：因为源码里已经有 source display、permission explainer、bridge disabled reason、managed settings security dialog、notification queue 这些解释层原件。

### 问：为什么这些零件还不够？

答：因为它们仍然分散在不同交互节点，没有汇成一张“持续可读”的总面。

### 问：为什么统一安全状态面比再加一条规则更值钱？

答：因为很多现实损耗不是规则缺失，而是用户不理解当前边界到底是谁在生效。

### 问：为什么这件事在 Claude Code 里尤其重要？

答：因为它的安全性本来就不是单点机制，而是多层控制面的叠加；没有统一可见性，用户很难区分当前到底被哪一层收口。

## 9. 给平台构建者的技术启示

最值得借鉴的不是某个具体 dialog 组件，  
而是这条产品化顺序：

1. 先把安全边界做成结构
2. 再为每个边界留下解释接口
3. 最后把解释接口收束成统一状态面

如果反过来做，只会得到漂亮但不真实的安全 UI。

## 10. 一句话总结

Claude Code 在安全产品化上已经具备大量解释层零件，但它们目前仍然散落在 permission prompt、managed settings 审批、entitlement 诊断和 source 展示等局部节点；下一步最值得做的，不是再加更多安全规则，而是把这些现有零件汇成一张统一的安全状态仪表盘，让用户真正看见当前是谁在定义运行时、谁在批准动作、谁在收口外部入口。
