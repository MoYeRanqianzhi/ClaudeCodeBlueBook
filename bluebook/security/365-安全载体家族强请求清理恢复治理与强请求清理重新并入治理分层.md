# 安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层：为什么artifact-family cleanup stronger-request cleanup-recovery-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reintegration-governor signer

## 1. 为什么在 `364` 之后还必须继续写 `365`

`364-安全载体家族强请求清理连续性治理与强请求清理恢复治理分层`
已经回答了：

`stronger-request cleanup 线即便已经拿到 fresh credential proof、fresh reconnect verdict 与 fresh tool fetch，也还需要 recovery governor 去决定对象是否真的 recovered。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然 reconnect result 已经是 connected，或者 OAuth flow 已经闭环，那它当然已经回到当前世界了”
的错觉：

`只要 stronger-request cleanup recovery governor 已经宣布对象 recovered，它就自动拥有了决定这份 recovered truth 是否已经重新进入当前 tool pool、status plane、handler plane 与下一轮 consumer world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的重新并入正对照，
会发现 repo 其实明确拆开了七件事：

1. raw recovery result
2. current state writeback
3. dynamic tool-state writeback
4. tool read model re-entry
5. status read model re-entry
6. event / channel handler rebind
7. pseudo-tool replacement

最硬的证据至少有七组：

1. `client.ts:2137-2198` 的 `reconnectMcpServerImpl()` 只返回 raw `client/tools/commands/resources`，本身不更新 consumer surfaces
2. `useManageMCPConnections.ts:296-332,1048-1122` 的 `updateServer()` / `onConnectionAttempt()` / `reconnectMcpServer()` / `toggleMcpServer()`
3. `print.ts:3133-3295` 的 `mcp_reconnect`：先写回 `appState` / `dynamicMcpState`，再按 `connected` 决定是否注册 handlers 与返回 success
4. `print.ts:3206-3295,3310-3445` 的 `mcp_toggle` enable path 与 auth-complete background path：recovery 之后仍要单独 reintegrate 到 `appState` 与 `dynamicMcpState`
5. `print.ts:1472-1485` 的 `buildAllTools()`：tool plane 只读取 reintegrated tool pool
6. `print.ts:1612-1626` 的 `buildMcpServerStatuses()`：status plane 只读取 reintegrated current state
7. `print.ts:1263-1285,4786-4835` 的 `registerElicitationHandlers()` / `reregisterChannelHandlerAfterReconnect()`，以及 `McpAuthTool.ts:124-165` 的 pseudo-tool -> real tools swap

这说明：

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经恢复。`

后者才配说：

`这个恢复后的对象现在已经重新并入当前 tool world、status world、event / handler world，并可被后续 consumer 当成 current world 的一部分继续消费。`

所以 `364` 之后必须继续补的一层就是：

`安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`

也就是：

`stronger-request cleanup recovery governor 决定恢复是否成立；stronger-request cleanup reintegration governor 才决定这份恢复真相何时真正并回 current consumer world。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reintegration manager，
而是在说：

1. repo 已经把 reconnect result 与 `appState` / `dynamicMcpState` 的更新拆开
2. repo 已经把 recovered client 与 real tools / resources 的重回 consumer plane 拆开
3. repo 已经把 `connected` 与 handler re-registration 拆开

第二条：

这里说的 `reintegration governance`

不是在声称：

`一切 recovery success 都必须立即让所有 consumer plane 同步完成。`

当前更稳的说法只能是：

`reintegration governance 首先回答的是 recovered truth 何时分别回到 state plane、tool plane、status plane 与 event plane。`

第三条：

这里也不是把：

`dynamicMcpState`

单独当成全部 reintegration grammar。

真正关键的是：

1. state writeback
2. dynamic tool-state writeback
3. tool / status read model re-entry
4. live handler rebind
5. pseudo-tool replacement

第四条：

这里尤其不能过度声称：

`reintegration governance`

已经等于：

`reprojection governance`

当前更稳的说法只能是：

`reintegration governance 回答 recovered truth 何时进入 current world；reprojection governance 才回答这份 current truth 随后怎样被不同 reader surface 重新讲述。`

第五条：

这里也不能过度声称：

`connected`

就天然等于：

`next-turn visible`

当前更稳的说法只能是：

`raw recovery result 只是 reintegration 输入材料；只有写回 current state、更新 dynamic tool pool 并重挂 handlers 后，consumer 才真正看见它。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-recovery-governor signer 仍不等于 stronger-request cleanup-reintegration-governor signer`

证据：

1. `reconnectMcpServerImpl()` 只返回 `client/tools/commands/resources`，说明 recovery result 本身不自动进入 consumer planes
2. `onConnectionAttempt()` 要先 `updateServer()` 再在 `connected` 情况下注册 elicitation handler；这说明恢复成立与 UI / event plane 回归仍是两步
3. `mcp_reconnect` / `mcp_toggle` 会显式写回 `appState.mcp`、`dynamicMcpState`，并单独执行 `registerElicitationHandlers()` / `reregisterChannelHandlerAfterReconnect()`；这说明 reconnect success 不自动等于 reintegration complete
4. `buildAllTools()` 与 `buildMcpServerStatuses()` 只从 `appState` / `dynamicMcpState` 读当前世界；这说明 consumer 看到的是 reintegrated truth，不是 raw recovery result
5. `McpAuthTool` 在 OAuth 完成后还要再把 pseudo-tool 换成 real tools；这说明 auth + reconnect 的 recovered proof 仍不自动等于 tool-world reintegration

因此这一章的最短结论是：

`stronger-request cleanup recovery governor 最多能说恢复真相已经成立；stronger-request cleanup reintegration governor 才能说这份恢复真相现在是否已经重新并入当前 consumer world。`

再压成一句：

`recovered，不等于 reintegrated。`

再压成更哲学一点的话：

`制度知道它已经回来，不等于制度已经决定谁现在看得见它、用得着它、收得到它。`

## 4. 第一性原理：recovery governance 回答“恢复是否成立”，reintegration governance 回答“恢复后的真相怎样重新进入当前世界”

从第一性原理看，
强请求清理恢复治理与强请求清理重新并入治理处理的是两个不同主权问题。

`stronger-request cleanup recovery governor`
回答的是：

1. 新凭据是否已经闭环
2. 新 callback / token exchange 是否完成
3. reconnect result 是否已经回到 `connected`
4. 当前 failure grammar 是否可撤销
5. recovered truth 本身是否成立

`stronger-request cleanup reintegration governor`
回答的则是：

1. 恢复后的 client 是否已写回 `appState`
2. 恢复后的 tools / commands / resources 是否已并回 current tool pool
3. `dynamicMcpState` 是否已更新，供下一轮 run 读取
4. elicitation handler / channel handler 是否已重新挂回
5. status plane、tool plane、event plane 是否已重新把它视为 current world 的一部分

如果把这两层压成一句：

`反正已经恢复了`

系统就会制造五类危险幻觉：

1. `connected-means-integrated`
2. `auth-done-means-tool-visible`
3. `state-update-means-handler-ready`
4. `recovered-means-next-turn-visible`
5. `raw-result-means-current-world`

所以从第一性原理看：

`recovery governance` 管的是恢复真相是否成立；
`reintegration governance` 管的是这份真相怎样重新进入 current world 的各个消费面。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup recovery governance 已经等于 reintegration governance，为什么 `reconnectMcpServerImpl()` 还只返回 raw result，而不直接写回 `appState`、`dynamicMcpState` 并重挂 handlers？
   因为“恢复成立”与“重新并回当前世界”根本不是同一句制度。
2. 如果 auth + reconnect 一旦闭环就自动被所有 consumer 看见，为什么 `buildAllTools()` 与 `buildMcpServerStatuses()` 还要依赖 `dynamicMcpState` / `appState`？
   因为 consumer 读取的是 reintegrated truth，不是 raw recovery result。
3. 如果写回状态就已经足够，为什么 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()` 还要单独执行？
   因为 event plane / channel plane 的 re-entry 也是单独的 reintegration authority。

## 5. `reconnectMcpServerImpl()` 先证明：恢复结果本身并不会自动把对象并回当前世界

`client.ts:2137-2198`
很硬。

这里的 `reconnectMcpServerImpl()`
做的是：

1. 清掉 keychain cache
2. 清掉 server cache
3. 重新 `connectToServer()`
4. 若成功则抓 `tools/commands/resources`
5. 返回 result object

但它并没有：

1. 写回 `appState`
2. 更新 `dynamicMcpState`
3. 注册 elicitation handler
4. 重新注册 channel handler

这说明 repo 自己已经把：

`recovery result exists`

和

`recovery result re-enters current world`

拆成了两步。

从技术角度看，
这非常值钱。
因为它拒绝让 raw transport-level recovery verdict 越级冒充完整 reintegration。

## 6. `onConnectionAttempt()` 再证明：standard path 上，reintegration 是单独的 projection choreography

`useManageMCPConnections.ts:296-332`
特别值钱。

这里 `onConnectionAttempt()` 在收到 connection result 后，
先做的不是直接宣告一切完成，
而是：

1. `updateServer({ ...client, tools, commands, resources })`
2. 然后只在 `client.type === 'connected'` 时注册真正的 elicitation handler

这条证据很硬。

它说明即便在标准 hook path 里，
repo 也明确承认：

1. recovery result 要先投影成 state update
2. event plane side effect 仍要单独挂回

也就是说，
recovery 和 reintegration 不是同一个动词。

`useManageMCPConnections.ts:1048-1122`
又继续把这层主权写给 operator path：

1. `reconnectMcpServer()` 先拿 raw `result`
2. 再交给 `onConnectionAttempt(result)` 做 reintegration
3. `toggleMcpServer()` enable path 也先标 `pending`
4. 然后 reconnect
5. 最后仍回到 `onConnectionAttempt(result)`

这说明连 hook 世界内部也拒绝把：

`reconnect 成功`

偷写成：

`当前世界已经自动重新接回它`

## 7. `print.ts` 证明：control plane 上，reintegration 至少拆成 `appState`、`dynamicMcpState` 与 handler 三层

`print.ts:3133-3295`
的 `mcp_reconnect`
很值钱。

这里在 `reconnectMcpServerImpl()` 之后，
系统继续做了三层 reintegration：

1. 写回 `appState.mcp.clients/tools/commands/resources`
2. 写回 `dynamicMcpState.clients/tools`
3. 只有 `connected` 时才 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`

这说明 reconnect result 并不自动等于：

1. `LLM next turn 已经看见它`
2. `status plane 已经接回它`
3. `event plane 已经接回它`

`print.ts:3206-3295`
的 `mcp_toggle` enable path 也是同样制度：

1. re-enable 不是直接假装重新并入
2. 而是 reconnect
3. 然后显式写回状态
4. 最后再按 `connected` 决定 handler rebind 与 control success

`print.ts:3310-3445`
的 auth-complete background path 更值钱：

1. auth promise resolve 还不够
2. reconnect result 出来还不够
3. 仍要 `setAppState(...)`
4. 仍要 `dynamicMcpState = ...`

这说明 print / headless 世界里，
至少存在三层不同 reintegration 问题：

1. state plane
2. tool / status visibility plane
3. event / channel plane

任何一层没接回，
都不该被偷写成：

`当前世界已经重新接回对象`

## 8. `buildAllTools()` 与 `buildMcpServerStatuses()` 证明：consumer 真正看见的是 reintegrated truth，不是 raw recovery result

`print.ts:1472-1485`
很值钱。

这里 `buildAllTools()`
会把：

`dynamicMcpState.tools`

并入下一轮真实工具池。

`print.ts:1612-1626`
又说明 `buildMcpServerStatuses()`
会把：

1. `currentAppState.mcp.clients`
2. `dynamicMcpState.clients`

并入当前 status response。

这意味着：

`recovered object`

如果还没被 reintegrate 进 `dynamicMcpState` 或 `appState`，
那么：

1. next-turn tool pool 不一定看见它
2. status plane 不一定看见它

这条证据很关键。
它说明 reintegration 不是附属 UI 操作，
而是当前 consumer world
是否真的收到恢复结果的 authoritative gate。

## 9. `registerElicitationHandlers()`、`reregisterChannelHandlerAfterReconnect()` 与 `McpAuthTool` 再证明：live interaction plane 与 placeholder world 都有单独的 re-entry grammar

`print.ts:1263-1285`
的 `registerElicitationHandlers()`
很硬。

它只对：

1. `connected`
2. 未注册
3. 非 SDK

client 才重新接回 request handler。

`print.ts:4786-4835`
的 `reregisterChannelHandlerAfterReconnect()`
也同样严格：

1. 非 `connected` 直接返回
2. 未通过 gate 直接返回
3. 通过后才重新挂回 channel notification handler

这说明 event plane / channel plane 的 re-entry，
并不是 raw recovery result 自动携带的。
它是单独的 reintegration authority。

`McpAuthTool.ts:124-165`
则给出最强 tool-plane 正例：

1. auth promise resolve
2. `reconnectMcpServerImpl()`
3. prefix-based 清理 pseudo-tool
4. real tools / commands / resources swap 回 `appState`

这说明 repo 对 tool plane 的理解不是：

`auth 成功了，所以真实工具自然就在了`

而是：

`恢复成立后，real tool world 仍要被单独 reintegrate 回来`

## 10. 更深一层的技术先进性：Claude Code 把“恢复是否成立”和“恢复后的真相何时再次成为当前世界的一部分”继续拆开

这组源码给出的技术启示至少有五条：

1. raw result 不应自动变成 reader-visible truth
2. state plane、tool plane、status plane 与 event plane 的 re-entry 需要分层治理
3. dynamic mutable state 是 next-turn world visibility 的正式 gate，不是杂项缓存
4. placeholder -> real world swap 本身是恢复后的一次单独主权动作
5. control success grammar 必须绑定 reintegration choreography，而不能只绑定 reconnect effort

从技术哲学看，
Claude Code 在这里先进的地方不是“恢复能力很多”，
而是它拒绝把：

`已经回来`

直接偷写成：

`已经重新属于当前世界`

## 11. 苏格拉底式自反诘问：我是不是又把“恢复成立”误认成了“重新并入已经完成”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 recovery grammar 已经足够强，为什么还要再拆 reintegration？
   因为恢复是否成立，不等于恢复后的真相怎样重新进入当前世界。
2. 如果 reconnect 已经回到 `connected`，是不是就说明下一轮 consumer 已经看见它？
   不是。next-turn visibility 还取决于 `dynamicMcpState` 与 `appState` 的 writeback。
3. 如果状态已经写回了，是不是就说明 event plane 也回来了？
   不是。handler rebind 仍是单独步骤。
4. 如果 auth flow 结束并返回了成功，是不是就说明 pseudo-tool 已经退出世界？
   不是。placeholder -> real tools 的 swap 仍需单独执行。
5. 如果 control plane 已经返回 success，是不是就能自动推出所有消费面已完全一致？
   不能。success grammar 必须重新检查它到底绑定了哪些 reintegration choreography。
6. 如果 cleanup 线现在还没有显式 reintegration 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确 grammar，越容易把“它已经回来”偷写成“它已经重新属于当前世界”。`

这一串反问最终逼出一句更稳的判断：

`reintegration 的关键，不在对象是不是已经 recovered，而在系统能不能正式决定这份 recovered truth 何时重新成为 current world 的一部分。`

## 12. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 recovery governance，就已经足够成熟。`

而是：

`Claude Code 在 raw reconnect result、state writeback、dynamic tool-state writeback、tool/status read model re-entry、event/channel handler rebind 与 pseudo-tool replacement 上已经明确展示了 reintegration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-recovery-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“它已经回来”，而是“它什么时候重新属于当前世界”。`
