# 安全载体家族强请求清理迁移治理与强请求清理退役治理分层：为什么artifact-family cleanup stronger-request cleanup-migration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-sunset-governor signer

## 1. 为什么在 `260` 之后还必须继续写 `261`

`260-安全载体家族强请求清理修复治理与强请求清理迁移治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样过渡，也还需要 migration governor 去决定 rewrite、remap、grace window、notice 与 continuity policy。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“迁完就算退役”的错觉：

`只要 stronger-request cleanup migration governor 已经安排了新旧世界如何并存，它就自动拥有宣布旧 stronger-request 世界正式退场的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 自己最成熟的 sunset 正对照，
会发现它其实已经把：

`migration clock`

和

`sunset clock`

拆成了两种不同制度。

最硬的证据有六组：

1. `src/utils/model/deprecation.ts:28-100`
   已为 deprecated models 维护 provider-specific retirement dates，
   说明 repo 区分
   `已经有迁移路径`
   和
   `旧模型在哪个 provider、哪一天正式退役`
2. `src/hooks/notifs/useModelMigrationNotifications.tsx:5-50`
   只负责一次性宣布
   `这次启动刚刚把你带到新默认值`
3. `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-42`
   与
   `src/main.tsx:2872-2895`
   则负责持续提示
   `旧模型即将在某日 retired`
4. `src/migrations/migrateLegacyOpusToCurrent.ts:13-57`
   即便已把 `userSettings` 迁到 `opus`，
   `src/utils/model/model.ts:549-554`
   仍保留 legacy remap opt-out，
   说明 migration 已发生时旧显式字符串仍可能继续以兼容形式活着
5. `src/utils/plugins/cacheUtils.ts:23-24,52-74,89-105`
   说明 orphan plugin version 先进入 grace world，再在 7 天后删除；
   这不是 migration 本身，而是 sunset deadline
6. `src/utils/plugins/orphanedPluginFilter.ts:1-15,35-38`
   与
   `src/main.tsx:2544-2568`
   又进一步给出 visibility sunset：
   旧 plugin version 可能先从当前 session 的 search / grep / glob world 退役，
   再从 storage world 真正消失

这说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界先怎样被带过渡。`

后者才配说：

`过渡什么时候结束，旧 stronger-request law 从哪一个日期、哪一个 runtime、哪一个 visibility surface 起正式不再算当前世界。`

所以 `260` 之后必须继续补的一层就是：

`安全载体家族强请求清理迁移治理与强请求清理退役治理分层`

也就是：

`stronger-request cleanup migration governor 负责 coexistence；stronger-request cleanup sunset governor 才负责宣布 coexistence 结束。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-sunset-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 retirement manager，
而是在说：

1. repo 在 model 与 plugin 线已经展示出正式 sunset grammar
2. sunset grammar 回答的是 retirement date、grace-window 截止、旧世界 visibility 何时被切断
3. 任何 stronger-request cleanup grammar 只要想自称完整，就必须回答这些“何时正式结束”的问题

第二条：

这里说的 `sunset governance`
不是在声称：

`旧世界一旦退役，就必须立刻从磁盘彻底消失。`

真正要说的是：

`退役治理首先回答的是“从什么时候起它不再算当前世界”，而这可以先发生在 visibility、search、warning semantics 上，再发生在 storage delete 上。`

第三条：

这里也不是把：

`plugin orphan grace window`

单独当成全部 sunset grammar。

真正关键的是：

1. model line 证明 migration notice 与 deprecation warning 是两张时钟
2. legacy remap 证明 migration 已完成时旧世界仍可能继续被兼容
3. plugin line 证明 visibility sunset 与 storage sunset 可以继续拆开
4. stronger-request cleanup 线当前还没有谁正式决定旧 startup wording、旧 plan path、旧 coverage covenant 与旧 receipt semantics 何时不再算当前世界

第四条：

这里尤其不能过度声称：

`migrationCompleted`、timestamp flag 或 one-time notification 已经等于 sunset 完成。

当前更稳的说法只能是：

`这些信号主要回答“迁移动作刚刚发生过”，并不自动回答“旧世界何时正式退役”。`

第五条：

这里也不能过度声称：

`sunset governance` 一定意味着用户立刻失去所有兼容路径。

当前更稳的说法只能是：

`sunset 可以先切 warning semantics、再切 visibility、最后切 storage；关键不在快慢，而在谁来正式宣布旧世界从何时起不再算 current truth。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-migration-governor signer 仍不等于 stronger-request cleanup-sunset-governor signer`

证据：

1. `deprecation.ts:28-100`
   已为 deprecated models 维护 provider-specific retirement dates，
   说明 repo 已把 retirement date 当成正式 truth
2. `useModelMigrationNotifications.tsx:5-50`
   只负责 one-time migration notification，
   而
   `useDeprecationWarningNotification.tsx:6-42`
   与
   `main.tsx:2872-2895`
   负责持续 deprecation warning
3. `migrateLegacyOpusToCurrent.ts:13-57`
   即便已把 `userSettings` 迁到 `opus`，
   `model.ts:549-554`
   仍保留 legacy remap opt-out，
   说明 migration 已发生但 sunset 尚未 hard-close
4. `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days`
   说明 orphan plugin version 先进入 grace world，再在期限后删除
   `cacheUtils.ts:23-24,52-74,89-105`
5. `orphanedPluginFilter.ts:1-15,35-38`
   与
   `main.tsx:2544-2568`
   说明旧 plugin version 可以先从当前 session 的 visibility world 退役，再从 storage world 消失
6. migration timestamp、completion flag、notice signal 只能证明迁移动作已发生，
   并不自动签出“旧世界已经正式退役”
7. stronger-request cleanup 当前虽然已开始显露 migration-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup law、旧 covered-family contract 与旧 receipt semantics 何时停止作为 current world 继续存在

因此这一章的最短结论是：

`stronger-request cleanup migration governor 最多能说旧世界怎样过渡；stronger-request cleanup sunset governor 才能说旧世界何时停止作为正式世界继续存在。`

再压成一句：

`会过渡，不等于会退役。`

## 4. 第一性原理：migration governance 回答“如何并存”，sunset governance 回答“何时终止并存”

从第一性原理看，
这两层治理处理的是两个不同时间面的主权问题。

`migration governor`

回答的是：

1. 旧值是 rewrite、remap，还是 dual-read
2. 哪些 source 可以改写，哪些必须保持原样
3. 兼容期间怎样提示用户
4. 哪些旧 artifact 要进入 grace window
5. 怎样保持幂等与 continuity

`sunset governor`

回答的则是：

1. 兼容窗口在哪个日期、哪次启动或哪类版本后结束
2. 旧值何时不再被 runtime 接受
3. 旧 artifact 何时不再被搜索、引用或恢复
4. 旧 warning 何时从“建议切换”进入“正式不可再用”
5. 双世界共存何时必须切成单世界真相

所以这两层的哲学本质不同：

`migration`

是在管理共存；

`sunset`

则是在管理共存的终止。

Claude Code 在这方面真正先进的地方，
不是它“会在旧世界不对时马上换成新世界”，
而是它已经明确承认：

`旧世界何时不再算当前世界，本身也是正式安全问题。`

这是一种很成熟的安全哲学：

`更正当前真相，不等于有权立即宣布旧真相已经作废。`

因为一旦把 migration 与 sunset 压成一层，
系统就容易出现三种危险：

1. compatibility path 被无限拖延，旧世界永不正式结束
2. migration 刚发生就被误写成 sunset 完成，旧世界被过早切断
3. visibility、warning 与 storage 三个层面的退役节奏失去秩序，系统开始在不同 surface 上说不同的话

## 5. 模型线给出最强正例：migration notification 与 deprecation warning 是两种不同制度

`useModelMigrationNotifications.tsx:5-50`
只在 migration timestamp 刚写入 config 时弹一次通知。

它回答的是：

`this startup moved you to the new default`

但
`deprecation.ts:28-100`
与
`useDeprecationWarningNotification.tsx:6-42`
回答的是另一件事：

`the old model will be retired on a provider-specific date`

更重要的是，
`main.tsx:2872-2895`
还会把 deprecation warning 放进 initial notification queue，
说明 sunset warning 不是 migration 副产物，
而是单独进入 runtime surface 的正式控制面。

## 6. stronger-request cleanup 当前真正缺的，是 formal end-of-compatibility policy

当前 stronger-request cleanup 线已经开始显露 migration choice：

1. 旧 startup-delete wording 是否继续 coexist
2. 旧 `~/.claude/plans` cleanup law 是否继续与 override world 并存
3. 旧 diagnostics uncovered reality 是否先被 notice、再被吸收到 family covenant
4. 旧 local count / local receipt truth 是否继续被默许为“勉强可用”

但即便这些 coexistence path 都被安排好了，
仍然会立刻碰到 sunset 问题：

### 6.1 旧 wording 何时不再被允许继续说

如果 deferred cleanup / validation skip explainability 被正式加强，
旧 `deleted at startup` 的绝对表述何时必须被硬关停。

### 6.2 旧 path 何时不再被允许继续当 current law

如果 `plansDirectory` cleanup 最终完成迁移，
旧 home-root-only plan cleanup law 何时正式不再算 current policy。

### 6.3 旧 coverage 何时不再被允许继续保持例外

如果 diagnostics family 最终被纳入 stronger-request cleanup family，
旧 uncovered status 何时正式结束，而不再只是“下次再说”。

### 6.4 旧 receipt 何时不再被允许继续作为可接受解释

如果 family receipt 被正式 surface，
旧 local-count-only truth 何时不再能被拿来充当 current explanation。

这些问题都不是 migration 自己能回答的。

它们已经进入：

`sunset-governance`

层。

## 7. 用苏格拉底式反问再压一层

1. 如果 stronger-request cleanup migration governance 已经等于 sunset governance，为什么模型线还要同时维护 migration timestamp 和 deprecation retirement date？
   因为“刚完成迁移”和“旧世界正式退役”根本不是同一时钟。
2. 如果旧 plugin version 还在磁盘上就说明它还算 current world，为什么 repo 还要先切 grep/glob visibility，再等 7 天才真正删除？
   因为 storage presence 与 current-world validity 不是同一层。
3. 如果 diagnostics 以后被纳入 cleanup covenant，为什么不能说“迁进去就算结束”？
   因为迁进去只回答 coexistence，没回答旧 uncovered exception 何时 hard-close。
4. 如果旧 wording 只是“过渡期里还在兼容”，为什么不能让它永远兼容？
   因为没有 sunset，compatibility 就会滑成永久双世界。
5. 如果 warning 已经出现，为什么还不够宣布 sunset 完成？
   因为 warning surface 只能提示 impending retirement，不自动等于 hard-close authority。

## 8. 硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 migration governance，就已经足够成熟。`

而是：

`Claude Code 在 model deprecation clock、migration clock、plugin orphan grace window 与 session-level visibility cutoff 上已经明确展示了 sunset governance 的存在；因此 artifact-family cleanup stronger-request cleanup-migration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

换句话说：

`cleanup 线真正缺的不是“怎么把旧世界带着走一段路”，而是“谁来宣布这段路何时结束”。`
