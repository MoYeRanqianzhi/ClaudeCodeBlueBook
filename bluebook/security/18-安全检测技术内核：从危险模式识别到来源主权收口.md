# 安全检测技术内核：从危险模式识别到来源主权收口

## 1. 这一章研究什么

前面的安全专题已经说明：Claude Code 的安全不是单点沙箱，也不是单点分类器。  
这一章继续往源码底层压一层，专门回答一个更技术的问题：

`Claude Code 到底靠哪些“检测链”防止能力越界、来源污染和外部入口失控？`

对应的核心源码模块主要包括：

- `src/utils/permissions/dangerousPatterns.ts`
- `src/utils/permissions/permissionSetup.ts`
- `src/utils/permissions/permissions.ts`
- `src/utils/permissions/pathValidation.ts`
- `src/utils/permissions/shadowedRuleDetection.ts`
- `src/utils/permissions/permissionExplainer.ts`
- `src/tools/WebFetchTool/preapproved.ts`
- `src/utils/mcpValidation.ts`
- `src/services/mcp/config.ts`
- `src/utils/managedEnv.ts`
- `src/utils/settings/settings.ts`
- `src/utils/settings/validation.ts`
- `src/utils/settings/pluginOnlyPolicy.ts`

## 2. 最短总论

Claude Code 的“安全检测”不是单个检测器，  
而是四条彼此串联的检测链：

1. 规则越权检测链
2. 路径与文件系统检测链
3. 外部入口检测链
4. 来源主权检测链

如果继续压缩，它的真实目标也不是“尽量多拦”，  
而是四个更具体的工程目标：

1. 不让危险 allow rule 绕过仲裁层
2. 不让路径语义在校验前后发生漂移
3. 不让外部入口把读权限偷换成执行权、写权或身份权
4. 不让低信任来源改写高风险运行时主链

## 3. 第一条链：规则越权检测链

### 3.1 `dangerousPatterns.ts` 的真正作用

`src/utils/permissions/dangerousPatterns.ts` 不是在列举“坏命令大全”，  
而是在列举：

`哪些 allow rule 一旦放进自动模式，就会把“先仲裁、后放行”改写成“先放行、后补救”。`

它把重点放在几类模式上：

- 解释器与脚本运行入口，例如 `python`、`node`、`deno`
- package runner 与 shell 包装入口，例如 `npm run`、`yarn run`、`bash`
- 远程命令或二次执行入口，例如 `ssh`
- PowerShell 下的二次求值、进程拉起与脚本块执行入口

这说明系统定义“危险”的标准不是工具名本身，  
而是：

`该规则是否会把任意代码执行能力提前变成默认允许。`

### 3.2 `permissionSetup.ts` 真正在做什么

`src/utils/permissions/permissionSetup.ts` 进一步把这些模式转化成可执行策略。

它最关键的几组函数是：

- `isDangerousBashPermission()`
- `isDangerousPowerShellPermission()`
- `isDangerousTaskPermission()`
- `findDangerousClassifierPermissions()`
- `findOverlyBroadBashPermissions()`
- `findOverlyBroadPowerShellPermissions()`

这里最值得写进蓝皮书的结论不是“它会告警”，  
而是：

`它会在进入 auto mode 之前，主动识别那些会提前绕过 classifier 的 allow 规则。`

也就是说，Claude Code 不是在 classifier 失败后再补锅，  
而是在 classifier 还没上场前，先把可能架空 classifier 的许可规则清掉或标出来。

这是非常成熟的一层设计：  
它把风险定义成“仲裁层被前置绕过”，而不是“危险行为发生以后再解释”。

### 3.3 `permissions.ts` 的检测顺序比检测器本身更重要

`src/utils/permissions/permissions.ts` 才是真正的总仲裁链。

从 `hasPermissionsToUseToolInner()` 和 `hasPermissionsToUseTool()` 看，  
它大致按下面的顺序工作：

1. 工具级 deny rule
2. 工具级 ask rule
3. tool-specific `checkPermissions()`
4. content-specific ask rule
5. `safetyCheck`
6. bypass mode / plan-bypass
7. tool-wide allow rule
8. ask 结果在 `dontAsk` / `auto` / headless 场景下再做二次转换

最重要的工程结论是：

`safetyCheck` 被明确放在 bypass 之前，并且在 auto mode 里还有一层“是否允许 classifier 介入”的二次判断。

这意味着：

1. 不是所有 ask 都能交给 classifier
2. 不是所有 ask 都能被 fast-path 吃掉
3. 有些 ask 被系统定义成“必须保留人工仲裁”

这正是安全控制面成熟的标志。  
系统没有把 classifier 神化成万能裁判，而是把它放在一条已经分层过的仲裁链中间。

### 3.4 这里最容易被误解的两个细节

第一，`permissionExplainer.ts` 不是拦截器。  
`src/utils/permissions/permissionExplainer.ts` 做的是解释层，把风险、原因和上下文整理给用户看。  
它很重要，但它是：

`可解释性组件，而不是执行授权组件。`

第二，safe-tool allowlist 也不是“永久免检”。  
它只是在 auto mode 里减少 classifier API 调用成本。  
真正越过不了的节点，依然是前面的 deny、content-specific ask、`safetyCheck` 和 tool-specific permission result。

### 3.5 `shadowedRuleDetection.ts` 说明系统在检测“坏规则”，也在检测“假规则”

`src/utils/permissions/shadowedRuleDetection.ts` 很关键，因为它说明系统不只关心危险规则，  
还关心：

`看起来存在、实际上永远到不了的规则。`

它会检测两类情况：

1. allow rule 被 tool-wide deny rule 完全压住
2. allow rule 被 tool-wide ask rule 阴影化，导致永远会先弹审批

这里最成熟的细节是它对 Bash sandbox auto-allow 的特殊判断：

- 如果 ask rule 来自个人设置，且 sandbox auto-allow 生效，就不一定把具体 allow rule 判成不可达
- 如果 ask rule 来自共享设置，则仍然告警，因为团队成员未必拥有相同 sandbox 条件

这说明它检测的不是抽象语法冲突，  
而是：

`在真实共享环境里，这条规则到底有没有可达性。`

## 4. 第二条链：路径与文件系统检测链

### 4.1 `pathValidation.ts` 的核心不是路径合法性，而是路径语义一致性

`src/utils/permissions/pathValidation.ts` 是这组源码里最值得单独重看的文件之一。

它最有价值的地方不是“会做 path 校验”，  
而是它持续在解决一个更底层的问题：

`校验时看到的路径，是否和真正执行时落到的路径是同一个语义对象。`

因此它专门处理了几类容易产生语义漂移的输入：

1. UNC 路径
2. `~user`、`~+`、`~-` 这类 tilde 变体
3. `$VAR`、`%VAR%`、`$(...)`、`=cmd` 这类 shell expansion
4. glob 写路径
5. symlink 解析与 canonical path

这背后的安全哲学非常清楚：

`真正危险的不是路径复杂，而是验证对象和执行对象不是同一个东西。`

### 4.2 `isPathAllowed()` 的顺序本身就是一套安全哲学

`isPathAllowed()` 的顺序大致是：

1. deny rule
2. internal editable path
3. `checkPathSafetyForAutoEdit()`
4. working directory / `acceptEdits`
5. internal readable path
6. sandbox write allowlist
7. allow rule

这里最关键的结论只有一句：

`安全检查先于工作目录便利，先于 acceptEdits，先于 allow rule。`

这意味着即使路径在工作目录里，  
只要它先撞上安全检查，便利路径就不能吃掉这次仲裁。

这也是 Claude Code 安全设计反复出现的同一条原则：

`先确认边界是否合法，再讨论当前模式愿不愿意放行。`

### 4.3 删除操作被单独看待

`isDangerousRemovalPath()` 还单独把下面这些目标视为危险删除路径：

- 根目录
- 家目录
- 根目录直接子项
- Windows 盘符根与一级子目录
- 通配删除

这说明系统并不满足于“用户有写权限”。  
它还会继续判断：

`这个写操作是否属于高不可逆性的破坏型动作。`

所以 Claude Code 的路径检测链，不是 ACL 风格的“你能不能碰”，  
而是更接近：

`你现在碰的这个对象，会不会把可恢复性直接打穿。`

## 5. 第三条链：外部入口检测链

### 5.1 WebFetch 的预批准域名不是网络白名单

`src/tools/WebFetchTool/preapproved.ts` 极其关键。

它明确写出一条容易被忽视的原则：

`WebFetch 预批准域名只适用于 GET 类读取，不应自动继承到 sandbox 网络放行。`

这实际上是在明确区分两种完全不同的权限语义：

1. 读取公共文档
2. 拥有对外网络写能力、上传能力或任意 HTTP 交互能力

源码甚至特意强调某些文档/模型站点虽然适合读取，  
但并不适合直接变成通用网络权限。

这背后的技术结论很硬：

`读取型外部入口的允许，不能被偷换成传输型或外发型权限。`

### 5.2 `mcpValidation.ts` 表明输出本身也是攻击面

`src/utils/mcpValidation.ts` 关注的是 MCP 输出截断与 token 预算。

这看起来像性能或 UX 问题，  
但从安全角度看，它表达的是另一条很成熟的判断：

`外部工具的风险不只在输入和连接，也在输出体积、输出形态和上下文吞吐。`

它通过：

- 粗粒度 token 估算
- API 级 token counting
- 文本截断
- 图像压缩
- 截断提示注入

把 MCP 输出控制在预算内。

这说明 Claude Code 对外部入口的治理不是“只要连上就行”，  
而是：

`连接、内容规模、上下文污染半径，都属于同一条安全控制链。`

### 5.3 `services/mcp/config.ts` 体现的是“连接主权”，不是简单配置读取

`src/services/mcp/config.ts` 的成熟点主要有四个：

1. denylist 绝对优先于 allowlist
2. allowlist 不只按 name，还可按 command 和 URL 模式匹配
3. enterprise MCP config 可以直接取得独占控制权
4. `strictPluginOnlyCustomization` 可以把 MCP 入口锁到 plugin / managed 来源

最关键的判断是：

`MCP 的真正风险不只是谁能连，而是谁有资格定义“要连什么”。`

尤其是：

- `filterMcpServersByPolicy()` 说明用户入口会被政策过滤
- `doesEnterpriseMcpConfigExist()` 说明在更高治理层，平台甚至允许“企业配置独占全部 MCP 主权”
- `isRestrictedToPluginOnly('mcp')` 说明系统已经把“来源是否有资格提供外部连接定义”做成正式策略

这不是插件管理，  
而是：

`外部能力装配主权管理。`

## 6. 第四条链：来源主权检测链

### 6.1 `managedEnv.ts` 的核心不是读 env，而是过滤 env

`src/utils/managedEnv.ts` 是整个安全专题里最容易被低估的文件之一。

它最关键的工作不是“把 settings.env 写进 process.env”，  
而是连续做了三层过滤：

1. 过滤 SSH tunnel 相关变量，防止远程转发主链被 settings 覆盖
2. 过滤 host-managed provider-routing 变量，防止用户把推理由 host 控制改写成自定义 provider
3. 过滤 Claude Desktop spawn env 快照中的关键变量，防止宿主注入被运行期配置污染

这背后对应的不是“配置兼容性”，  
而是：

`运行时主链主权不能被低信任来源静默重写。`

### 6.2 trust 前和 trust 后是两套不同的 env 治理模型

`applySafeConfigEnvironmentVariables()` 与 `applyConfigEnvironmentVariables()` 的分层非常重要。

trust 前：

- 只允许 trusted setting sources 进入
- project/local source 只允许 safe env vars 生效
- policy source 还被放在一个显式更高的位置

trust 后：

- 才允许更完整的 settings env 生效
- 仍然要经过 host-managed provider var 过滤

这说明 Claude Code 对来源主权的理解非常成熟：

`不是所有配置都等到加载后再决定是否可信，而是在进入运行时之前就先按来源分层。`

### 6.3 `settings.ts` 与 `validation.ts` 把“坏配置的破坏半径”压小

`src/utils/settings/validation.ts` 的 `filterInvalidPermissionRules()` 值得重点记住。

它体现的判断是：

`一条坏规则不应该毒化整份 settings 文件。`

而 `src/utils/settings/settings.ts` 进一步把这个原则落到加载链：

- 解析 settings 文件前先过滤无效 permission rules
- `policySettings` 采用 first source wins
- 优先级明确是 `remote > MDM(HKLM/plist) > managed file > HKCU`

这里的哲学非常鲜明：

1. 来源优先级比“谁先被读到”更重要
2. 单个坏值不应拖垮整份配置
3. 管理面必须拥有正式且可解释的优先级

### 6.4 `pluginOnlyPolicy.ts` 说明“谁能定制系统”本身也是安全问题

`src/utils/settings/pluginOnlyPolicy.ts` 很短，但意义很大。

它把一个经常被产品文档淡化的问题正式化了：

`定制面本身也是攻击面。`

一旦 `strictPluginOnlyCustomization` 生效：

- user/project/local 这类用户控制来源会被排除在某些定制面之外
- plugin、policySettings、built-in / bundled 则被视为 admin-trusted source

这说明系统在问的不是“能不能扩展”，  
而是：

`扩展定义权究竟属于谁。`

## 7. 最值得写入蓝皮书的 12 条源码级结论

1. `dangerousPatterns.ts` 与 `permissionSetup.ts` 说明，Claude Code 真正警惕的是“allow rule 绕过 classifier”，而不是命令名字难看。
2. `permissions.ts` 说明 deny、content-specific ask 与 `safetyCheck` 都被故意放在 bypass 和大多数 fast-path 前面。
3. auto mode classifier 不是第一裁判，而是已经经过预过滤后的中段裁判。
4. `shadowedRuleDetection.ts` 说明系统连“看起来存在但实际上不可达”的权限规则也会治理。
5. `pathValidation.ts` 的核心价值是防止验证对象与执行对象出现语义漂移，而不是简单做路径正则校验。
6. 写操作 glob 禁止、shell expansion 禁止、UNC 限制、tilde 变体限制，说明路径输入被当成潜在解释器边界，而不是普通字符串。
7. `WebFetchTool/preapproved.ts` 明确证明 GET 读取白名单不等于网络放行白名单。
8. `mcpValidation.ts` 证明外部工具输出规模本身就是安全与控制面问题，因为它决定上下文污染半径。
9. `services/mcp/config.ts` 证明 MCP 治理的核心不是“是否支持”，而是“连接定义权和连接主权属于谁”。
10. `managedEnv.ts` 证明 Claude Code 首先保护的是运行时主链不被配置来源污染。
11. `settings/validation.ts` 与 `settings/settings.ts` 证明系统偏好局部过滤和 first-source-wins，而不是把所有错误都升级成全局失效。
12. `pluginOnlyPolicy.ts` 证明平台已经把“谁有资格扩展系统”上升为正式安全策略，而不是非正式约定。

## 8. 这套检测技术先进在哪里

如果只从技术先进性角度评价，我会给出五条结论：

1. 它把检测前移到了 mode 切换和来源加载阶段，而不是只在执行点拦截。
2. 它把“坏规则”和“假规则”都纳入治理对象。
3. 它把路径、输出规模、连接定义权和 env 来源都视为同一控制面的不同侧面。
4. 它反复区分读取语义、写入语义、执行语义和身份语义，避免权限偷换。
5. 它允许 classifier 参与，但拒绝把 classifier 神化成唯一安全基础。

这说明 Claude Code 的先进性不是“模型更聪明”，  
而是：

`它把安全做成了多层、可组合、可回退、可解释的运行时结构。`

## 9. 第一性原理与苏格拉底式追问

### 问：为什么它不把所有判断都交给 classifier？

答：因为 classifier 再强，也不应该负责来源主权、路径语义一致性和配置优先级这类结构性问题。

### 问：为什么要单独检测危险 allow rule？

答：因为真正危险的不是用户显式批准某次动作，而是系统把未来一整类高风险动作提前变成“默认允许”。

### 问：为什么连 settings 的坏规则都要局部过滤？

答：因为安全系统的目标不是借一次配置错误制造全局瘫痪，而是把坏值的破坏半径压缩到最小。

### 问：为什么 WebFetch 白名单不能直接变成 network allowlist？

答：因为“能读文档”和“能向外发数据”不是同一种能力，把两者混写就等于把读权限偷换成外发权限。

### 问：为什么说这套系统的核心不是 classifier，而是主权？

答：因为一旦低信任来源可以改写 provider、auth、MCP、hooks 或 runtime env，后面再强的 classifier 都是在错误边界内工作。

## 10. 给 Agent 平台构建者的技术启示

1. 不要只做行为检测，要先做来源检测。
2. 不要只检测危险动作，要检测会让危险动作永久默认化的危险规则。
3. 不要只验证路径字符串，要验证路径在校验时与执行时是否还是同一语义对象。
4. 不要把所有外部入口统一成“插件问题”，要按读取、写入、执行、身份、上下文污染分别治理。
5. 不要把输出预算当成成本优化，它同时是安全和上下文主权问题。
6. 不要只治理“谁能调用工具”，还要治理“谁有资格定义工具与连接”。
7. 不要把解释层当作装饰，它决定用户是否能真正理解边界为何生效。

## 11. 最短结论

Claude Code 的安全检测技术，本质上不是在寻找“坏行为”本身，  
而是在寻找四类更深层的结构性风险：

1. 仲裁层是否被 allow rule 提前架空
2. 校验对象与执行对象是否发生语义漂移
3. 外部入口是否把读取能力偷换成执行、写入或身份能力
4. 低信任来源是否正在改写高风险运行时主链

所以这套系统真正先进的地方，不是某个检测器特别强，  
而是：

`它把危险模式、路径语义、外部入口和来源主权，全部纳入同一套安全控制面。`
