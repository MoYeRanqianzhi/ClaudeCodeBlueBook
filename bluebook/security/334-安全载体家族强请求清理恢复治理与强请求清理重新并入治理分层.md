# 安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层：为什么artifact-family cleanup stronger-request cleanup-recovery-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reintegration-governor signer

## 1. 为什么在 `333` 之后还必须继续写 `334`

`333-安全载体家族强请求清理连续性治理与强请求清理恢复治理分层`
已经回答了：

`stronger-request cleanup 线即便已经拿到 fresh credential proof、fresh reconnect verdict 与 fresh tool fetch，也还需要 recovery governor 去决定对象是否真的 recovered。`

但如果继续往下追问，
还会碰到另一层很容易被偷写成
“既然 reconnect result 已经是 connected，或者 OAuth flow 已经闭环，那它当然已经回到当前世界了”
的错觉：

`只要 stronger-request cleanup recovery governor 已经宣布对象 recovered，它就自动拥有了决定这份 recovered truth 是否已经重新进入当前 tool pool、status plane、handler plane 与下一轮 consumer world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的重新并入正对照，
会发现 repo 其实明确拆开了七件事：

1. `raw recovery result`
2. `current state writeback`
3. `dynamic tool-state writeback`
4. `tool read model re-entry`
5. `status read model re-entry`
6. `event / channel handler rebind`
7. `pseudo-tool replacement`

最硬的证据至少有七组：

1. `src/services/mcp/client.ts:2137-2194`
   的 `reconnectMcpServerImpl()`
   只返回 raw `client / tools / commands / resources`，
   本身不更新 consumer surfaces
2. `src/services/mcp/useManageMCPConnections.ts:293-332`
   的 `updateServer()` / `onConnectionAttempt()`
   与 `registerElicitationHandler()`
3. `src/cli/print.ts:3146-3204`
   的 `mcp_reconnect`：
   先写回 `appState` / `dynamicMcpState`，
   再按 `connected` 决定是否注册 handlers 与返回 success
4. `src/cli/print.ts:3255-3288`
   的 `mcp_toggle` enable path：
   recovery 之后仍要单独 reintegrate 到 `appState` 与 handler plane
5. `src/cli/print.ts:3391-3445`
   的 auth-complete background path：
   reconnect 之后还要把 client / tools / commands / resources 写回 `appState` / `dynamicMcpState`
6. `src/cli/print.ts:1474-1490,1612-1626`
   的 `buildAllTools()` 与 `buildMcpServerStatuses()`：
   tool plane 与 status plane 都只依赖 reintegrated state
7. `src/cli/print.ts:1263-1285,4786-4835`
   的 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`，
   以及 `src/tools/McpAuthTool/McpAuthTool.ts:134-168`
   的 pseudo-tool -> real tools swap

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup recovery governance`
   负责决定 recovered truth 是否成立
2. `stronger-request cleanup reintegration governance`
   负责决定这份 recovered truth 何时、以什么方式、进入哪些当前 consumer planes

也就是说：

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经恢复。`

后者才配说：

`这个恢复后的对象现在已经重新并入当前 tool world、status world、event / handler world，并可被后续 consumer 当成 current world 的一部分继续消费。`

所以 `333` 之后必须继续补的一层就是：

`安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`

也就是：

`stronger-request cleanup recovery governor 决定恢复是否成立；stronger-request cleanup reintegration governor 才决定这份恢复真相何时真正并回 current consumer world。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reintegration manager，
而是在说：

1. repo 已经把 reconnect result 与 `appState` / `dynamicMcpState` 的更新拆开
2. repo 已经把 recovered client 与 real tools / resources 的重回 consumer plane 拆开
3. repo 已经把 `connected` 与 handler re-registration 拆开

第二条：

这里说的 `reintegration governance`

不是在声称：

`一切 recovery success 都必须立即让所有 consumer plane 同步完成。`

当前更稳的说法只能是：

`reintegration governance 首先回答的是 recovered truth 何时分别回到 state plane、tool plane、status plane 与 event plane。`

第三条：

这里也不是把：

`dynamicMcpState`

单独当成全部 reintegration grammar。

真正关键的是：

1. state writeback
2. dynamic tool-state writeback
3. tool / status read model re-entry
4. live handler rebind
5. pseudo-tool replacement

第四条：

这里尤其不能过度声称：

`reintegration governance`

已经等于：

`reprojection governance`

当前更稳的说法只能是：

`reintegration governance 回答 recovered truth 何时进入 current world；reprojection governance 才回答这份 current truth 随后怎样被不同 reader surface 重新讲述。`

第五条：

这里也不能过度声称：

`connected`

就天然等于：

`next-turn visible`

当前更稳的说法只能是：

`raw recovery result 只是 reintegration 输入材料；只有写回 current state、更新 dynamic tool pool 并重挂 handlers 后，consumer 才真正看见它。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-recovery-governor signer 仍不等于 stronger-request cleanup-reintegration-governor signer`

证据：

1. `reconnectMcpServerImpl()`
   只返回 `client / tools / commands / resources`，
   说明 recovery result 本身不自动进入 consumer planes
2. `onConnectionAttempt()`
   要先 `updateServer()` 再在 `connected` 情况下注册 elicitation handler；
   这说明恢复成立与 UI / event plane 回归仍是两步
3. `mcp_reconnect` / `mcp_toggle`
   会显式写回 `appState.mcp`、`dynamicMcpState`，
   并单独执行 `registerElicitationHandlers()` / `reregisterChannelHandlerAfterReconnect()`；
   这说明 reconnect success 不自动等于 reintegration complete
4. `buildAllTools()` 与 `buildMcpServerStatuses()`
   只从 `appState` / `dynamicMcpState` 读当前世界；
   这说明 consumer 看到的是 reintegrated truth，
   不是 raw recovery result
5. `McpAuthTool`
   在 OAuth 完成后还要再把 pseudo-tool 换成 real tools；
   这说明 auth + reconnect 的 recovered proof 仍不自动等于 tool-world reintegration

因此这一章的最短结论是：

`stronger-request cleanup recovery governor 最多能说恢复真相已经成立；stronger-request cleanup reintegration governor 才能说这份恢复真相现在是否已经重新并入当前 consumer world。`

再压成一句：

`recovered，不等于 reintegrated。`

再压成更哲学一点的话：

`制度知道它已经回来，不等于制度已经让它重新成为当前世界的一部分。`

## 4. 第一性原理：recovery governance 回答“恢复是否成立”，reintegration governance 回答“恢复后的真相怎样重新进入当前世界”

从第一性原理看，
强请求清理恢复治理与强请求清理重新并入治理处理的是两个不同主权问题。

`stronger-request cleanup recovery governor`
回答的是：

1. 新凭据是否已经闭环
2. 新 callback / token exchange 是否完成
3. reconnect result 是否已经回到 `connected`
4. 当前 failure grammar 是否可撤销
5. recovered truth 本身是否成立

`stronger-request cleanup reintegration governor`
回答的则是：

1. 恢复后的 client 是否已写回 `appState`
2. 恢复后的 tools / commands / resources 是否已并回 current tool pool
3. `dynamicMcpState` 是否已更新，供下一轮 run 读取
4. elicitation handler / channel handler 是否已重新挂回
5. status plane、tool plane、event plane 是否已重新把它视为 current world 的一部分

如果把这两层压成一句
`反正已经恢复了`，
系统就会制造五类危险幻觉：

1. `connected-means-integrated`
2. `auth-done-means-tool-visible`
3. `state-update-means-handler-ready`
4. `recovered-means-next-turn-visible`
5. `raw-result-means-current-world`

所以从第一性原理看：

`recovery governance` 管的是恢复真相是否成立；
`reintegration governance` 管的是这份真相怎样重新进入 current world 的各个消费面。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup recovery governance 已经等于 reintegration governance，
   为什么 `reconnectMcpServerImpl()` 还只返回 raw result，
   而不直接写回 `appState`、`dynamicMcpState` 并重挂 handlers？
   因为“恢复成立”与“重新并回当前世界”根本不是同一句制度。
2. 如果 auth + reconnect 一旦闭环就自动被所有 consumer 看见，
   为什么 `print.ts` 还要单独更新 `dynamicMcpState`，
   `buildAllTools()` 与 `buildMcpServerStatuses()` 也还要只从 reintegrated state 读取？
   因为 next-turn visibility 与 raw recovery result 不是同一层。
3. 如果 `connected` 一成立就天然回到 live interaction world，
   为什么 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()` 还要单独重绑？
   因为 event plane 的回归同样是单独主权，不是 recovery 的尾气。

## 5. `reconnectMcpServerImpl()` 先证明：recovery result 是原材料，不是完成态

`src/services/mcp/client.ts:2137-2194`
很值钱。

这里 `reconnectMcpServerImpl()`
负责做的是：

1. fresh reconnect
2. fresh fetch tools / commands / resources
3. 返回 raw result object

它不负责：

1. 写回 `appState`
2. 更新 `dynamicMcpState`
3. 重绑 elicitation handler
4. 重绑 channel handler

这条证据非常硬。

它说明 repo 自己已经拒绝把：

`recovery proof exists`

偷写成：

`reintegration complete`

也就是说，
raw recovery result 只是 reintegration 的输入材料，
不是 completed current-world verdict。

## 6. `onConnectionAttempt()` 再证明：standard path 上，reintegration 至少拆成“state projection + event re-entry”两步

`src/services/mcp/useManageMCPConnections.ts:293-332`
很值钱。

`onConnectionAttempt()`
在拿到 result 后，
先做：

`updateServer({ ...client, tools, commands, resources })`

然后只在 `client.type === 'connected'` 时：

`registerElicitationHandler(client.client, client.name, setAppState)`

这说明即便在标准 TUI path，
repo 也明确承认：

1. result 要先投影进 current state
2. 事件交互面还要单独重新接回

所以 reintegration 不是 recovery 的同义词，
而是 recovery 之后的一整段 projection choreography。

## 7. `print.ts` 证明：control plane 上，reintegration 至少拆成 state plane、tool plane 与 event plane 三层

`src/cli/print.ts:3146-3204`
的 `mcp_reconnect`
很硬。

这里在 `reconnectMcpServerImpl()` 后，
系统继续做了四件事：

1. 写回 `appState.mcp.clients`
2. 写回 `appState.mcp.tools / commands / resources`
3. 更新 `dynamicMcpState`
4. 只有 `connected` 时才 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`

`src/cli/print.ts:3255-3288`
的 `mcp_toggle`
也是同样制度。

`src/cli/print.ts:3391-3445`
的 auth-complete background path 更值钱：

1. recovery result 已经存在
2. 仍然要单独 `setAppState`
3. 仍然要单独更新 `dynamicMcpState`

这说明 print / headless 世界里，
至少存在三层不同 reintegration 问题：

1. state plane
2. tool / status visibility plane
3. event / channel plane

任何一层没接回，
都不该被偷写成：

`当前世界已经重新接回对象`

## 8. `dynamicMcpState`、`buildAllTools()` 与 `buildMcpServerStatuses()` 证明：reintegration 决定下一轮 consumer 到底看见什么

`src/cli/print.ts:1474-1490`
很值钱。

这里 `buildAllTools()`
会把：

`dynamicMcpState.tools`

并入下一轮真实工具池。

`src/cli/print.ts:1612-1626`
又说明 `buildMcpServerStatuses()`
会把：

`dynamicMcpState.clients`

并入当前 status response。

这意味着：

`recovered object`

如果还没被 reintegrate 进 `dynamicMcpState`，
那么：

1. next turn tool pool 不一定看见它
2. status plane 不一定看见它

这条证据很关键。
它说明 reintegration 不是附属 UI 操作，
而是当前 consumer world
是否真的收到恢复结果的 authoritative gate。

## 9. `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()` 再证明：live interaction plane 也有单独的 re-entry grammar

`src/cli/print.ts:1263-1285`
的 `registerElicitationHandlers()`
很硬。

它只对：

1. `connected`
2. 未注册
3. 非 SDK

client 才重新接回 request handler。

`src/cli/print.ts:4786-4835`
的 `reregisterChannelHandlerAfterReconnect()`
也同样严格：

1. 非 `connected` 直接返回
2. 未通过 gate 直接返回
3. 通过后才重新挂回 channel notification handler

这说明 event plane / channel plane 的 re-entry，
并不是 raw recovery result 自动携带的。
它是单独的 reintegration authority。

如果 stronger-request cleanup 线未来没有这类 live interaction reintegration grammar，
那它就仍然回答不了：

`恢复后的旧对象现在有没有重新回到那些真正会在当前世界里发生活动的面。`

## 10. `McpAuthTool` 给出最强 tool-plane 正例：恢复完成后还要把 placeholder world 换回 real world

`src/tools/McpAuthTool/McpAuthTool.ts:134-168`
特别值钱。

这里在 auth promise resolve 之后，
系统还要继续：

1. `clearMcpAuthCache()`
2. `reconnectMcpServerImpl()`
3. 把旧 pseudo-tool 按 prefix 清掉
4. 把 real tools / commands / resources swap 回 `appState`

这说明 repo 对 tool plane 的理解不是：

`auth 成功了，所以恢复完成了，所以真实工具自然就在了`

而是：

`恢复成立后，real tool world 仍要被单独 reintegrate 回来`

这条证据把 reintegration
从“state bookkeeping”提升成了真正的
model-visible world re-entry。

## 11. 对 stronger-request cleanup 而言，当前真正缺的不是“恢复是否成立”，而是“恢复后的真相何时重新进入当前 consumer world”

当前 stronger-request cleanup 已经开始显露 recovery-governance，
但仍缺少 reintegration 治理回答：

1. recovered path / promise / receipt truth 何时真正写回 current state
2. 何时真正回到 current tool / status / event world
3. 哪些下游 consumer 现在已经能看见它，哪些还没有
4. 哪些 placeholder / pseudo-world 必须被 real world 替换
5. 哪些 live handler 仍必须单独重新接回

这些都不是：

`什么新的证据足以宣布恢复成立`

能回答的。

它们已经进入：

`reintegration governance`

层。

## 12. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

因为前者只能决定：

`恢复真相是否已经成立。`

而后者才有资格决定：

`这份恢复真相何时真正重新成为当前 consumer world 的一部分。`

再压成最后一句：

`制度知道它已经恢复，不等于制度已经让所有当前读者重新看见它。`
