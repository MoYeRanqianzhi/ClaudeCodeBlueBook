# 安全完成差异交互路由：为什么卡片、通知、动作可用性与清除条件必须同时联动

## 1. 为什么在 `47` 之后还要继续写“交互路由”

`45-安全完成差异字段` 回答的是：

`到底要把哪些宿主盲区字段做成正式对象。`

`46-安全完成差异卡片` 回答的是：

`这些字段该挂到哪张卡、按什么层级出现。`

`47-安全完成差异升级规则` 回答的是：

`哪些差异必须抢占界面，哪些只适合中层提示或下钻。`

但如果继续往实现层再压一层，  
还会剩下一个更关键的问题：

`同一个宿主盲区一旦变化，卡片、通知、动作可用性和清除条件是否会在同一时刻一起变化？`

如果答案是否定的，  
那么前面三章即使都成立，  
用户仍然会遇到一种更隐蔽也更危险的错觉：

1. 通知已经在说“这里有差异”
2. 卡片还保留着过满结论
3. 动作按钮还允许继续走旧路径
4. 旧提示消失后，界面又像什么都没发生过

这意味着真正成熟的统一安全控制台还必须再补一层：

`宿主盲区交互路由。`

也就是：

`当盲区变宽或变窄时，哪些界面对象要一起更新，谁先更新，谁负责清除旧结论。`

## 2. 最短结论

从源码看，Claude Code 已经拥有三种成熟但分散的控制能力：

1. 通知优先级与抢占队列  
   `src/context/notifications.tsx:45-116,230-239`
2. 摘要型状态页与桥接对话框  
   `src/components/Settings/Status.tsx:19-53`、`src/components/BridgeDialog.tsx:137-256`
3. 若干局部状态提示与低维宿主标签  
   `src/hooks/useIdeConnectionStatus.ts:11-32`、`src/components/IdeStatusIndicator.tsx:18-55`

但这些能力今天还没有被统一成同一条宿主盲区状态机。  
它们更像：

`局部分工明确，但没有共同路由器。`

所以这一章的最短结论是：

`字段、卡片和升级规则都还不够，统一安全控制台还必须把“卡片状态、通知优先级、动作可用性、清除条件”绑定成一次原子路由。`

换句话说：

`宿主盲区如果只更新文案，不同步收缩动作与旧结论，它仍然会在交互层制造假完整。`

## 3. 源码已经证明：系统拥有“注意力路由”，但主要还停留在通知层

## 3.1 `notifications.tsx` 已经不是样式系统，而是注意力仲裁器

`src/context/notifications.tsx:78-116` 非常关键。  
`immediate` 通知到来时，系统会：

1. 清掉当前 timeout
2. 直接替换当前通知
3. 把旧通知重新排队
4. 按 `invalidates` 清理失效旧项

而 `src/context/notifications.tsx:230-239` 又把：

1. `immediate`
2. `high`
3. `medium`
4. `low`

做成正式优先级，并用 `getNext()` 按优先级选出当前通知。

这说明 Claude Code 底层已经知道：

`不同事实应该竞争同一个当前注意力槽位。`

这非常先进。  
因为很多系统虽然也会发 toast，  
但并没有把：

`哪个事实配抢当前认知通道`

做成正式语义。

## 3.2 Prompt 区只显示一个当前通知，这使“当前赢者”尤其关键

`src/components/PromptInput/Notifications.tsx:286-292` 说明当前通知最终被渲染在 prompt input 区，  
用户真正第一眼看到的是：

`notifications.current`

而不是一个平权列表。

这意味着一旦宿主盲区变化被正确发成通知，  
它理论上已经可以占据最关键的交互位置。  
问题在于：

`通知赢了当前槽位，不等于卡片和动作也跟着一起改口。`

## 4. 源码同样证明：其他界面层仍然主要是“摘要层”，不是“联动层”

## 4.1 `/status` 更像属性摘要，而不是宿主盲区控制台

`src/components/Settings/Status.tsx:19-53` 会把 `/status` 组织成两段属性列表，  
其中 IDE、MCP、sandbox、setting sources 都被压成 `Property[]`。

而 `src/utils/status.tsx:89-114` 更进一步把 MCP 状态压成：

1. connected 数量
2. needs auth 数量
3. pending 数量
4. failed 数量

再附一个 `/mcp` 跳转提示。

这种设计很适合：

`总览。`

但并不适合：

`在宿主盲区突然扩大时立即改写动作语义。`

因为属性列表天生擅长“静态展示”，  
不擅长“强制同步收口”。

## 4.2 BridgeDialog 已经能展示状态、错误、环境和 footer，但还不是完成差异路由器

`src/components/BridgeDialog.tsx:137-256` 会展示：

1. bridge status label
2. error
3. environment id
4. session id
5. footer text

这说明桥接视图已经有了：

`状态摘要 + 调试上下文 + 辅助说明`

的成熟零件。

但它没有把下面这些对象正式做成同一时刻联动的控制字段：

1. `completion_claim_ceiling`
2. `missing_ledgers[]`
3. `write_ready/read_ready`
4. 当前动作是否应禁用或改写

所以 BridgeDialog 今天更像：

`桥接状态面板`

而不是：

`桥接盲区路由器`

## 4.3 IDE 与 host 资格仍在被压成低维状态

`src/hooks/useIdeConnectionStatus.ts:11-32` 把 IDE 连接压成：

1. `connected`
2. `pending`
3. `disconnected`
4. `null`

`src/components/IdeStatusIndicator.tsx:18-55` 则只在 connected 且存在 selection 时显示：

1. 多少行被选中
2. 当前在哪个文件里

这套设计很适合：

`告诉用户 IDE selection 能不能拿到。`

但它不回答：

`IDE 通道变薄之后，控制面里哪些解释、哪些动作、哪些完成结论必须一起降级。`

## 4.4 某些底层已经承认“connected 不等于全链 ready”，但这件事还没被统一路由出来

`src/bridge/replBridgeTransport.ts:289-299` 写得非常直接：

`isConnectedStatus()` 是 write-readiness，  
不是 read-readiness。

这是一条非常珍贵的源码证据。  
因为它说明底层已经清楚知道：

`连接标签本身可能只代表部分能力。`

可如果这一点不被同步路由到卡片和动作层，  
用户看到 `connected` 时仍会天然误以为：

`写和读都稳定，结果和状态都能回来。`

## 5. 第一性原理：统一安全控制台真正要路由的不是“状态词”，而是“可安全行动的界面组合”

如果从第一性原理追问：

`用户为什么会被宿主盲区误导？`

答案通常不是“完全没有信息”，  
而是：

`信息分布在不同界面对象上，但它们没有一起改口。`

因此统一安全控制台真正要维护的，不是单个字段，  
而是一个交互组合：

1. 标题或主卡给出的当前结论
2. 通知系统给出的当前注意力优先级
3. 按钮、命令入口或默认动作给出的可执行路径
4. 清除旧结论的条件

我会把这条原则压成一句话：

`安全控制面最终保护的不是“知道了什么”，而是“用户接下来被允许按什么理解去行动”。`

所以一旦宿主盲区变化：

1. 只改通知，不改卡片，不够
2. 只改卡片，不改动作，不够
3. 只改动作，不清旧结论，也不够

真正成熟的做法必须是：

`结论、提醒、动作、清除四件事同一时刻联动。`

## 6. 我对“安全完成差异交互路由”的正式定义

我会把这一层定义成：

`当宿主盲区字段发生变化时，把它映射成卡片更新、通知级别、动作收口和清除条件的一次原子更新。`

它至少要回答五个问题：

1. 这是盲区扩大还是盲区缩小
2. 当前最该被改写的是哪张卡
3. 当前是否要抢占通知槽位
4. 哪些动作必须立刻禁用、替换或降级
5. 旧结论要到什么条件满足后才允许被清除

## 7. 这套交互路由最少应有六类触发器

## 7.1 `completion_claim_ceiling` 被拉低

这是最关键的一类。  
因为它直接决定：

`当前界面还能把“完成”说到哪一步。`

一旦 ceiling 下降：

1. 主卡标题必须立刻改口
2. 当前说明卡必须补出缺失账本或资格原因
3. 若正在推翻用户刚得出的完成结论，应升级通知
4. 任何仍假定全链完成的动作都应禁用或改写

否则就会出现：

`通知说有风险，但主卡和动作还在说“你可以按完整完成来理解”。`

## 7.2 `missing_ledgers[]` 新增

这是第二关键类。  
因为很多宿主误导并不是纯连接失败，  
而是：

`少了一本解释账，却仍沿用旧结论。`

对于这类变化，  
路由器至少应做到：

1. 在资格/完成卡中显式新增缺失账本
2. 把默认动作从“继续使用当前结论”切成“去补账本或切宿主”
3. 只有当账本重新出现并完成 authoritative 回读时才清除旧告警

## 7.3 `write_ready/read_ready` 分裂或恢复

`src/bridge/replBridgeTransport.ts:289-299` 已经表明 write-readiness 和 read-readiness 不是同一回事。  
所以路由规则必须承认：

`connected` 只是一个局部事实，不能自动授权完整动作集。`

一旦发生分裂：

1. bridge 卡应保持连接信息，但明确声明读写能力不对称
2. 不应允许界面继续使用“远程控制 active”这类过满结论
3. 依赖回读、确认或完成解释的动作必须降级

一旦恢复：

1. 不能只去掉通知
2. 还要同步恢复主卡文案与动作可用性
3. 并要求一次新鲜回读来清除旧 ceiling 降级

## 7.4 `supported_control_subtypes[]` 不覆盖当前动作

前面章节已经说明某些窄宿主天然只支持少量 control subtype。  
这类问题最容易在交互层被误处理成：

`按钮还能点，只是点完报错。`

真正成熟的交互路由不该等点击后再报错，  
而应：

1. 在卡片上声明当前宿主支持边界
2. 在动作层直接隐藏、禁用或替换不支持的入口
3. 在通知层只负责提醒“为什么被收口”，而不是承担所有收口责任

## 7.5 recovery window 打开或关闭

这是最应触发联动的一类。  
因为 recovery window 不是普通降级，  
而是：

`当前完成解释权临时失效。`

这时如果只弹出通知，  
但主卡还保留旧结论，  
风险极大。

正确路由至少应包括：

1. 立即抢占通知槽位
2. 立刻降低 completion ceiling
3. 立刻收紧依赖远程结果完整性的动作
4. 只有恢复窗口关闭且回读完成后，才允许撤销该降级

## 7.6 gate / plugin / IDE / MCP 状态改写下一步动作路径

源码里已经有成熟优先级例子：

1. bridge fail 为 `immediate`  
   `src/hooks/useReplBridge.tsx:102-111`
2. channels blocked 为 `high`  
   `src/services/mcp/useManageMCPConnections.ts:596-610`
3. flagged plugins 为 `high`  
   `src/hooks/useManagePlugins.ts:59-67`
4. IDE disconnected 为 `medium`  
   `src/hooks/notifs/useIDEStatusIndicator.tsx:87-92`
5. MCP failed / needs-auth 为 `medium`  
   `src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`
6. plugin reload pending 为 `low`  
   `src/hooks/useManagePlugins.ts:294-300`

但这些例子今天主要还是：

`通知级别的成熟。`

统一安全控制台真正应该再往前推进一步：

`把这些优先级直接接到动作可用性和卡片主结论上。`

## 8. 我给下一代统一安全控制台的交互路由规则

我会把它压成下面四条硬规则。

### 8.1 宿主盲区一旦扩大，先收结论，再发通知，再留动作

顺序不能反。  
因为如果先发通知、后收主卡，  
用户仍然会先看见旧结论。

所以一旦盲区扩大，  
应当按以下顺序原子更新：

1. 降低主卡结论上限
2. 增补盲区字段
3. 升级当前通知
4. 收紧或替换动作

### 8.2 宿主盲区一旦缩小，先验证，再恢复结论，再清通知

恢复同样不能只靠“状态看起来回来了”。  
否则系统会反复在旧盲区和新乐观之间摆动。

正确顺序应是：

1. 先等 authoritative 回读或足够强的新鲜状态
2. 再恢复主卡结论
3. 再恢复动作
4. 最后清除旧通知和旧降级痕迹

### 8.3 通知负责赢得注意力，不负责独自承担全部解释

这一点非常重要。  
`notifications.tsx` 已经是成熟的注意力总线，  
但它不该替代结构化解释。

所以通知最适合承载的是：

1. 当前优先级
2. 最短原因
3. 最短跳转或动作

而不该承载全部宿主盲区语义。  
完整解释仍应回到卡片和字段对象。

### 8.4 动作入口必须服从 ceiling，而不是服从表面连接词

最危险的反模式是：

1. 标签显示 connected
2. 动作仍然全开
3. 真正的回读或解释资格其实已经变薄

成熟系统必须坚持：

`动作集由 completion ceiling 和 supported subtypes 决定，而不是由一个更好看的连接词决定。`

## 9. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 已经把三件最难的底层能力分别做出来了：优先级仲裁、局部状态面、宿主差异承认。`

它的先进之处不只是“通知分四级”，  
更在于：

1. 知道 `immediate` 可以抢占当前通知  
   `src/context/notifications.tsx:79-116`
2. 知道 prompt 区只有一个当前通知槽位  
   `src/components/PromptInput/Notifications.tsx:286-292`
3. 知道 `connected` 也可能只是 write-readiness  
   `src/bridge/replBridgeTransport.ts:289-299`
4. 知道不同通道退化应有不同优先级  
   `useReplBridge.tsx`、`useManagePlugins.ts`、`useIDEStatusIndicator.tsx`、`useMcpConnectivityStatus.tsx`

所以 Claude Code 的真正技术启示不是“再做几个状态词”，  
而是：

`把分散在通知、状态页、bridge view、host label 里的成熟局部机制，升级成同一条交互路由状态机。`

对其他 Agent 平台构建者，这里至少有四条直接可迁移的经验：

1. 优先级系统必须能真正抢当前注意力，而不是只改颜色
2. 摘要面板和动作面板不能各自演化，否则一定会出现假完整
3. “connected” 这类表面词必须让位于能力上限字段
4. 清除旧告警必须绑定新鲜回读，而不是绑定表面恢复

## 10. 哲学本质

这一章更深层的哲学不是“界面一致性”，  
而是：

`解释权一致性。`

因为宿主盲区真正危险的地方不在于它让系统短暂退化，  
而在于它让不同界面对象开始替同一事实给出不同答案。

一旦发生这种分裂：

1. 通知在说“别完全信”
2. 卡片在说“基本完成”
3. 按钮在说“继续执行”

那用户最终一定会相信：

`最顺手的那个界面对象。`

因此从第一性原理看，  
交互路由真正守护的是这条公理：

`同一个安全事实在同一时刻只能拥有一个可行动的官方解释。`

## 11. 苏格拉底式反思：这一章最容易犯什么错

### 11.1 会不会把一切都做成全局联动，结果造成界面过度抖动

会。  
所以联动不是“所有字段都同步刷新所有对象”，  
而是：

`只对会改写下一步动作或完成结论的字段建立强联动。`

### 11.2 会不会把通知系统当成完整控制台

也会。  
通知只负责抢注意力，  
不负责承载全部证据、资格和账本解释。

### 11.3 会不会只做升级，不做清除纪律

这是最大风险之一。  
如果清除条件没有被正式化，  
控制面就会在恢复时重新滑向乐观谎言。


## 12. 结语

`45` 回答的是字段。  
`46` 回答的是卡片。  
`47` 回答的是升级。  
这一章继续推进后的结论则是：

`真正成熟的统一安全控制台，还必须把宿主盲区变化路由成一次原子联动：卡片改口、通知抢位、动作收口、清除守纪。`

这意味着 Claude Code 更深层的安全启示之一是：

`安全控制面不是把真相摆在多个地方，而是保证多个地方在同一时刻只说同一种可行动的真相。`
