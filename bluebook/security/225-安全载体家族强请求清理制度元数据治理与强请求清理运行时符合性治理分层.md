# 安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层：为什么artifact-family cleanup stronger-request cleanup-metadata-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer

## 1. 为什么在 `224` 之后还必须继续写 `225`

`224-安全载体家族强请求清理制度理由治理与强请求清理制度元数据治理分层` 已经回答了：

`different stronger-request family 的制度理由当前虽然已经开始显影，但这些理由仍主要寄居在 helper、comment、schema、permission projection 与 hand-written dispatcher 中，还没有被完整提升成统一 metadata plane。`

但如果继续往下追问，
还会碰到下一层更危险的错觉：

`只要系统已经有 stronger-request cleanup metadata signal，runtime reality 就必然已经持续服从这份 signal。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看：

1. `src/utils/settings/types.ts:325-333` 与 `src/utils/settings/validationTips.ts:46-53` 把 `cleanupPeriodDays=0` 写成 “no transcripts are written and existing transcripts are deleted at startup”
2. 但 `src/utils/sessionStorage.ts:954-969` 的 `shouldSkipPersistence()` 立刻保证的首先只是 transcript write suppression，而不是 delete receipt
3. `src/screens/REPL.tsx:3903-3906`、`src/main.tsx:2811-2818` 与 `src/utils/backgroundHousekeeping.ts:28-58,77-80` 又把 cleanup 的真实 runtime path 切成 first-submit admission、headless deferred admission、10 分钟延迟与 recent-activity reschedule
4. `src/utils/cleanup.ts:575-584` 甚至会在 settings validation errors 且用户显式设置 `cleanupPeriodDays` 时直接 skip cleanup，以避免 accidental deletion
5. `src/utils/plans.ts:79-106`、`src/utils/settings/types.ts:824-830` 与 `src/utils/permissions/filesystem.ts:245-255,1487-1494,1644-1652` 明明已经让 `plansDirectory` 成为正式 metadata knob，并在读写权限平面跟随，但 `src/utils/cleanup.ts:300-303` 的 executor 仍只扫默认 `~/.claude/plans`
6. `src/utils/cleanup.ts:33-44,575-595` 虽然定义并局部返回了 `CleanupResult`，后台入口却没有把这些结果汇总、持久化或对外投影成 family-by-family conformance receipt

这说明：

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

和

`artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer`

仍然不是一回事。

前者最多能说：

`系统已经把 stronger-request cleanup 的某些 truth 写进了 schema、helper、comment、permission rule 或 dispatcher。`

后者才配说：

`当前 runtime 已经在这个 session、这个启动阶段、这个 validation 状态下真实地服从了这份 truth，而且这个服从结果被正式保存。`

所以 `224` 之后必须继续补的一层就是：

`安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层`

也就是：

`stronger-request cleanup-metadata governor 最多只配说“规则已经被写出来”；stronger-request cleanup-runtime-conformance governor 才能说“当前运行时已经按这份规则发生，而且这个发生结果被诚实保存”。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer` 仍是研究命名。
它不是在声称 repo 已经有一张现成的 conformance receipt 表却没人读取，
而是在说：

1. metadata truth 即使存在，也仍然必须通过 scheduler、executor、skip guard 与 result surfacing 被证成
2. 当前 stronger-request cleanup 线已经暴露出 temporal gap、propagation gap 与 receipt gap
3. 因而 runtime-conformance 已经是一个独立问题，而不是 metadata 的脚注

第二条：

这里说的 `runtime-conformance`

不是在声称：

`当前实现一定存在严重 bug。`

真正要说的是：

`当前源码已经足以证明 stronger-request cleanup 的 declared truth、scheduled truth、executed truth 与 surfaced truth 仍然不是同一层。`

第三条：

这里也不是把：

`cleanupPeriodDays=0`

单独当成全部 runtime-conformance 问题。

真正关键的是：

1. transcript write suppression 与 existing transcript deletion 不是同一步
2. delayed housekeeping 使 startup language 与 execution timing 分叉
3. validation skip 让 runtime 有时为了更高阶安全而故意不执行
4. plans family 则暴露出 metadata consumed in some planes but ignored in another
5. `CleanupResult` 说明 outcome vocabulary 已经存在，但还没有升格成正式 receipt

第四条：

这里尤其不能过度声称：

`runtime skip` 一定意味着设计失败。

当前更稳的说法只能是：

`skip 可能正是更高阶 safety semantics 的体现，但这恰恰更证明 metadata 与 runtime-conformance 不是同一层。`

第五条：

这里也不能过度声称：

`只要某个 cleanup function 返回 CleanupResult，就已经算 conformance receipt。`

当前更稳的说法只能是：

`局部返回值说明作者意识到了 outcome 值得计数；但如果后台入口没有汇总、保存或对外投影，它仍不是正式 conformance surface。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-metadata-governor signer 仍不等于 stronger-request cleanup-runtime-conformance-governor signer`

证据：

1. `cleanupPeriodDays=0` 的 metadata wording 很强，但 `shouldSkipPersistence()` 立刻保证的首先只是 future-write suppression
   `settings/types.ts:325-333`; `validationTips.ts:46-53`; `sessionStorage.ts:954-969`
2. interactive 与 headless world 对 cleanup 的 admission 都走 background housekeeping，而不是即时 delete receipt
   `REPL.tsx:3903-3906`; `main.tsx:2811-2818`; `backgroundHousekeeping.ts:28-58,77-80`
3. runtime 还会因为 validation state 故意 skip cleanup，以避免 accidental deletion
   `cleanup.ts:575-584`
4. plans family 证明 metadata 即使已存在，也不自动保证 executor 跟随
   `settings/types.ts:824-830`; `plans.ts:79-106`; `permissions/filesystem.ts:245-255,1487-1494,1644-1652`; `cleanup.ts:300-303`
5. `CleanupResult` 明明存在，后台入口却没有把 family outcome 汇总成正式 receipt
   `cleanup.ts:33-44,575-595`
6. stale worktrees 反而获得了单独 telemetry，说明 repo 已经知道某些 cleanup outcome 值得投影，但 stronger-request carrier family 当前还没有等强度 surfaced receipt
   `cleanup.ts:595-597`

因此这一章的最短结论是：

`stronger-request cleanup-metadata-governor signer 最多只能说“系统宣称应该如此”；stronger-request cleanup-runtime-conformance-governor signer 才能说“它已经在当前 runtime 中如此发生，并且这个发生结果被正式保存”。`

再压成一句：

`有元数据，不等于有符合性证明。`

## 4. 第一性原理：metadata 回答“系统自述了什么”，runtime-conformance 回答“运行时实际兑现了什么”

从第一性原理看，
强请求清理制度元数据治理与强请求清理运行时符合性治理处理的是两种完全不同的问题。

`stronger-request cleanup-metadata governor`

回答的是：

1. 这个 family 的 rule 被写在哪里
2. 哪些 knob、helper、comment、dispatcher 描述了它
3. 哪些路径或 override 是合法输入
4. 哪些 retention / storage truth 已被系统承认
5. family truth 在理论上应怎样传播

`stronger-request cleanup-runtime-conformance governor`

回答的则是：

1. scheduler 何时真正开始执行
2. validation / interactivity / environment state 是否改变执行结果
3. executor 是否真的消费了那份 metadata
4. suppress、delete、skip、receipt 是否同步成立
5. 当前 session 里到底发生了什么，而不只是应该发生什么

如果把两层压成一句“metadata 已经有了，所以 runtime 自然会照做”，
系统就会制造五类危险幻觉：

1. documented-means-done illusion
   只要 schema / comment 写了，就误以为 runtime 已经完成
2. suppression-means-deletion illusion
   只要 future writes 被 suppress，就误以为 past carriers 已被 delete
3. startup-language-means-immediate-execution illusion
   只要文案写 startup delete，就误以为 delete receipt 已在启动瞬间成立
4. metadata-consumed-somewhere-means-consumed-everywhere illusion
   只要某个 override 被 path helper 或 permission plane 读取，就误以为 executor 也已经跟随
5. return-type-means-receipt illusion
   只要 cleanup function 返回 `CleanupResult`，就误以为 runtime 已正式保存 conformance receipt

所以从第一性原理看：

`metadata` 是系统对规则的自述；
`runtime-conformance` 是系统对“这条规则此刻已被兑现”的证明。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup metadata 已经等于 runtime-conformance，为什么 `cleanupPeriodDays=0` 会被写成 startup delete，但 REPL 还要等第一次 submit 才开始 housekeeping？
   因为 declaration truth 不自动等于 temporal conformance truth。
2. 如果 metadata 被系统完整消费，为什么 `plansDirectory` 已经改写 storage / permission world，而 `cleanupOldPlanFiles()` 仍只扫默认 `~/.claude/plans`？
   因为 one-plane consumption 不自动等于 all-plane conformance。
3. 如果 conformance surface 已存在，为什么 `CleanupResult` 明明定义了 outcome vocabulary，后台入口却没有把 family-by-family result 汇总成正式 receipt？
   因为 local countability 不自动等于 surfaced conformance。

## 5. `cleanupPeriodDays=0` 先给出最典型样本：metadata 句子很强，但 runtime 实际先分裂成“停写”和“延迟删”

`src/utils/settings/types.ts:325-333` 与 `src/utils/settings/validationTips.ts:46-53` 都给出同一类强语义：

`no transcripts are written and existing transcripts are deleted at startup`

如果只看 metadata，
这已经是一句非常强的统一承诺。

但 `src/utils/sessionStorage.ts:954-969` 的 `shouldSkipPersistence()` 立刻保证的首先只有：

`future transcript writes are suppressed`

这说明 stronger-request cleanup metadata 一旦进入 runtime，
至少会先被拆成两层：

1. future-write suppression
2. past-artifact deletion

而这两层当前并没有被同一 receipt 统一证明。

## 6. `backgroundHousekeeping` 再说明：所谓 startup cleanup，在 runtime 里实际上是“进入一个可延迟、可重排、可后延的 housekeeping 世界”

`src/screens/REPL.tsx:3903-3906` 很关键：

1. 只有 `submitCount === 1`
2. 才会 `startBackgroundHousekeeping()`

这意味着 interactive world 下的 stronger-request cleanup 根本不是进程刚启动就立刻执行。

`src/main.tsx:2811-2818` 对 headless world 也只是：

1. 后台导入 housekeeping 模块
2. 启动 deferred bookkeeping

`src/utils/backgroundHousekeeping.ts:28-58,77-80` 更进一步把时间结构钉死：

1. 初次运行延迟 10 分钟
2. 最近一分钟用户有交互就继续后延
3. 只有 `needsCleanup` 仍为真时才真正跑 cleanup

所以 stronger-request cleanup 的 runtime truth 更接近：

`startup admits a deferred cleanup path`

而不是：

`startup has already produced a delete receipt`

## 7. `cleanup.ts` 的 validation guard 再说明：runtime 有时会为了更高阶安全而故意不符合纸面 metadata

`src/utils/cleanup.ts:575-584` 明确写着：

1. settings 有 validation errors
2. 且用户显式设置了 `cleanupPeriodDays`
3. 那就 skip cleanup entirely

这里非常关键，
因为它说明 runtime-conformance 不是一个机械层。

它还要服从：

`anti-accidental-deletion fail-safe principle`

所以 stronger-request cleanup runtime truth 至少还要区分：

1. executed
2. delayed
3. skipped for safer semantics

## 8. plans family 继续说明：metadata 被部分消费，不等于 runtime executor 已 conform

`plansDirectory` 已经被：

1. settings plane 承认
2. storage plane 消费
3. permission plane 跟随

但 `cleanupOldPlanFiles()` 仍只清默认 home-root plans dir。

这说明 stronger-request cleanup runtime-conformance 当前至少还存在：

`propagation gap`

也就是：

`metadata-consumed in one plane != executor-conformed in another plane`

从技术角度看，
这正是 runtime-conformance 比 metadata 更强的地方：

`它要求系统不只记得规则，还要真的在所有关键平面里按规则行动。`

## 9. `CleanupResult` 与 stale-worktree telemetry 最后说明：repo 已经拥有局部 outcome vocabulary，却还没有 stronger-request family 级 conformance receipt

`src/utils/cleanup.ts:33-44` 定义了 `CleanupResult`。

这说明作者知道 cleanup outcome 值得计数。

但 `src/utils/cleanup.ts:575-595` 的后台入口没有把这些 result 汇总、保存或投影；
反而只有 stale worktrees 获得了单独 telemetry。

这说明当前 repo 的状态不是：

`不会描述 outcome`

而是：

`只会局部描述 outcome，还不会把 stronger-request family outcome 签成统一 conformance receipt`

## 10. 从技术角度看 Claude Code 这一层设计的先进性与启示

如果把这一层抽成技术启示，
至少有五条：

1. 它没有把 skip 当成实现失败，而是显式保留了 fail-safe 语义，这让 runtime truth 比纸面规则更诚实
2. 它把 suppression、admission、execution 与 receipt 分成了不同层，这使系统更接近真实而不是宣传语
3. 它承认某些 metadata 只在部分平面被消费，这使 propagation gap 可以被明确指出，而不是被安全宣传覆盖
4. 它已经拥有局部 outcome vocabulary，这说明系统知道 receipt 层是值得存在的，只是还没把它提升到 stronger-request family plane
5. 它让 research 能够精确地区分 “规则已经写出” 和 “规则已经兑现”，这正是成熟安全工程的必要前提

## 11. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup metadata 一旦存在，runtime conformance 就自动成立。`

而是：

`cleanupPeriodDays=0` 的强文案、`shouldSkipPersistence()` 的即时停写、housekeeping 的延迟 admission、validation guard 的 intentional skip、plans family 的 propagation gap，以及 `CleanupResult` 未被升级成统一 receipt，已经共同展示出 stronger-request cleanup runtime-conformance 的独立存在；因此 artifact-family cleanup stronger-request cleanup-metadata-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer。`

因此：

`cleanup 线真正缺的，不只是“谁来把规则写成 metadata”，还包括“谁来证明这些 metadata 当前已经被运行时诚实兑现并签收”。`
