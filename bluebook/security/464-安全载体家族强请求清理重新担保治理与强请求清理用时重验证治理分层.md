# 安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层：为什么artifact-family cleanup stronger-request cleanup-reassurance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer

## 1. 为什么在 `463` 之后还必须继续写 `464`

`463-安全载体家族强请求清理重新投影治理与强请求清理重新担保治理分层`
已经回答了：

`stronger-request cleanup 线即便已经把 current truth 重新讲给不同 surface，也还要单独回答这些重讲里哪些现在配承载多强的正向 reassurance。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然某个 surface 已经给出了足够正向的 reassurance，真正 consumer 在实际使用那一刻当然可以直接继续依赖”
的错觉：

`只要 stronger-request cleanup reassurance governor 已经允许某个 surface 说出足够正向的话，真正的 use path 在实际依赖这一刻就自动拥有了继续依赖这份 truth 的资格，不再需要 fresh re-check。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的用时重验证正对照，
会发现 repo 其实明确拆开了七件事：

1. `fresh connection primitive`
2. `session-recovery retry`
3. `runtime contradiction revocation`
4. `tool roster freshness`
5. `current-tool hard filter`
6. `consumer-edge live gate`
7. `agent-edge actual-tools gate`

最硬的证据至少有七组：

1. `src/services/mcp/client.ts:1688-1703`
   的 `ensureConnectedClient()` 会 fresh `connectToServer()`，若结果不再是 `connected` 就直接抛错
2. `src/services/mcp/client.ts:1858-1922`
   的 `MCPTool.call()` 会在每次真正 tool use 前先 `ensureConnectedClient(client)`，并在 session expired 后只允许有限次 fresh retry
3. `src/services/mcp/client.ts:3196-3230`
   的 `callMCPTool()` 会把 `401` 与 session-expired 明确识别成 live contradiction，并触发 `clearServerCache()` 与 `McpSessionExpiredError`
4. `src/services/mcp/useManageMCPConnections.ts:618-656`
   会在 `tools/list_changed` 时主动清掉 `fetchToolsForClient` cache 并重抓 current roster
5. `src/services/tools/toolExecution.ts:337-394`
   只承认当前 `toolUseContext.options.tools` 里仍然存在的 tool；旧 tool 名若不再在 current pool 中，就直接报 `No such tool available`
6. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:80-100`、
   `src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:79-89`
   与 `src/services/mcp/client.ts:2073-2080`
   会在 read/list/prompt 的真实 consumer path 上继续 `ensureConnectedClient()`
7. `src/tools/AgentTool/AgentTool.tsx:369-408`
   明确把“required MCP server 已连接”进一步收紧成“required MCP server 现在真的有 tools”，拒绝让空连接或未鉴权连接冒充 live-use readiness

这说明：

`artifact-family cleanup stronger-request cleanup-reassurance-governor signer`

和

`artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer`

仍然不是一回事。

前者最多能说：

`当前有某个 surface 配给出多强的正向 reassurance。`

后者才配说：

`真正 consumer 在依赖发生这一刻，已经重新拿到 fresh enough 的 current-use proof，或在 proof 失效时能够立即撤销旧 reassurance。`

所以 `463` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层`

也就是：

`stronger-request cleanup reassurance governor 决定正向话术现在能强到什么程度；stronger-request cleanup use-time revalidation governor 才决定真正使用时这句正向话是否仍配被当真。`

## 2. 先做四条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 live gate manager，
而是在说：

1. repo 已经明确把 reassurance surface 与 live consumer edge 拆开
2. repo 已经明确要求真实依赖路径在 use-time 重新确认 current truth
3. repo 已经明确允许 runtime contradiction 即时撤销旧 reassurance

第二条：

这里说的 `use-time revalidation`
不是在声称：

`每次使用都一定重新走一遍昂贵的物理重连。`

当前更稳的说法只能是：

`repo 要求存在 canonical fresh-proof primitive；healthy path 可以是 memoized no-op，但依赖资格仍要经由这条 primitive，而不是直接沿用旧 reassurance label。`

第三条：

这里也不能过度声称：

`use-time revalidation`
已经等于：

`step-up reauthorization`

当前更稳的说法只能是：

`use-time revalidation 先回答“这份 current-use truth 此刻还配不配被当真”；step-up reauthorization 才回答“即便此刻还能用，它对更强请求是否仍然太弱”。`

第四条：

这里同样不能把：

`use-time revalidation`

偷窄成：

`只有真正 callTool() 才算 use-time。`

当前更稳的说法只能是：

`tool call、resource read、resource list、prompt get、agent prerequisite check、current tool roster filter 都属于 consumer-edge revalidation grammar。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-reassurance-governor signer 仍不等于 stronger-request cleanup-use-time revalidation-governor signer`

证据：

1. `ensureConnectedClient()` 会把 current-use proof 收束成 `connect-or-throw` primitive，说明 reassurance 不自动等于 live-use authorization
2. `MCPTool.call()` 会在每次真实 tool use 前先 `ensureConnectedClient()`，并在 session expired 后只允许 bounded retry，说明真正消费路径不盲信旧 reassurance
3. `callMCPTool()` 会把 `401` 与 session-expired 明确识别成 live contradiction，并清 cache 后抛出新的 recovery signal，说明 runtime contradiction 会正式撤销旧 assurance
4. `useManageMCPConnections()` 会在 `tools/list_changed` 上主动失效 cache 并重抓 current tools，说明旧 roster 也不配继续代表 live capability truth
5. `runToolUse()` 只承认当前 tool pool 里仍然存在的 tool，旧 tool 名如果已经掉出 current roster 就直接失败，说明历史可用性不等于当前可用性
6. `ReadMcpResourceTool`、`ListMcpResourcesTool` 与 prompt `getPromptForCommand()` 都在真实 consumer edge 再次 `ensureConnectedClient()`，说明 list/read/prompt 也拒绝盲信 reassurance
7. `AgentTool` 明确把 required MCP server 收紧成“现在真的有 tools”的服务器，而不是“看起来连接过”的服务器，说明 agent edge 也只认 actual-tools proof

因此这一章的最短结论是：

`stronger-request cleanup reassurance governor 最多能说现在可以把话说到多放心；stronger-request cleanup use-time revalidation governor 才能说真正依赖这一刻这句放心话是否仍配被当真。`

再压成一句：

`会背书，不等于会在用前再验票。`

再压成更硬的一句：

`被担保到可期待，不等于已经被重验到可依赖。`

再压成更哲学一点的话：

`制度知道现在可以把话说多满，不等于制度已经在你伸手这一刻重新核过这句话还真不真。`

## 4. 第一性原理：reassurance 回答“现在能说多放心”，use-time revalidation 回答“真正依赖这一刻还敢不敢继续相信这句话”

从第一性原理看，
强请求清理重新担保治理与强请求清理用时重验证治理处理的是两个不同主权问题。

`stronger-request cleanup reassurance governor`
回答的是：

1. 哪个 surface 现在配说正向话术
2. 这句正向话术允许多强
3. 是否必须保留 caveat
4. 是否宁可沉默也不发布 success
5. 当前 reassuring sentence 的 liability ceiling 是什么

`stronger-request cleanup use-time revalidation governor`
回答的则是：

1. 真正 consumer 在依赖前是否还要 fresh reconnect
2. 当前 tool / resource / prompt / server roster 是否仍是 current truth
3. runtime contradiction 出现时谁有权立刻撤销旧 reassurance
4. 哪些 stale label 应被 live consumer 直接无视
5. 哪些 positive surface 只能邀请继续尝试，哪些 current proof 才能真正授权继续依赖

如果把这两层压成一句：

`既然刚才已经说能用了`

系统就会制造七类危险幻觉：

1. `reassurance-means-usable-now`
2. `connected-copy-means-fresh-session`
3. `tool-once-existed-means-tool-still-exists`
4. `positive-roster-means-current-roster`
5. `health-or-dialog-speaks-for-consumer-edge`
6. `agent-sees-server-means-agent-has-tools`
7. `runtime-contradiction-is-just-local-noise`

所以从第一性原理看：

`reassurance governance`
管的是正向信号强度；

`use-time revalidation governance`
管的是实际依赖发生前后，旧 reassurance 是否仍配被当真。

## 5. Claude Code 多重安全技术的先进性：它把“正向担保”与“live-use 资格”强行拆成一条多层收口链

从技术角度看，
Claude Code 在这里最先进的地方，
不是“有一个 hard gate”这么简单，
而是它把 live-use 资格写成了一条多层收口链：

1. `fresh proof primitive over stale label reuse`
   不沿用旧 `connected` 标签，而是统一经过 `ensureConnectedClient()`
2. `session contradiction becomes structured recovery signal`
   401 与 session-expired 不被当成普通失败，而被转写成 `McpAuthError` / `McpSessionExpiredError`
3. `roster freshness is part of security, not just UX`
   `tools/list_changed` 会主动失效 cache 并刷新 current roster
4. `consumer edge stays sovereign`
   read/list/prompt/tool 都要在 use-time 重新拿 fresh proof
5. `agent edge also rejects empty connectivity`
   agent prerequisite 不看“连没连过”，只看“现在有没有 tools”
6. `runtime revocation is part of verification`
   验真不只发生在入场前，也发生在 live contradiction 出现时

这些设计背后的源码思路也很清楚：

1. 不让 reassuring sentence 越级冒充 current-use proof
2. 不让 stale roster 越级冒充 live capability roster
3. 不让历史 tool name 越级冒充当前可调用工具
4. 不让 consumer edge 把上游 comfort copy 当成自己的授权签名
5. 不让 runtime contradiction 被解释成局部例外而不是正式撤销

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. reassurance 之外，还必须单列 `fresh-at-use proof primitive`
2. tool roster freshness 也是 revalidation grammar，不只是缓存策略
3. live contradiction 必须有正式 demotion / retry / revoke path
4. consumer-edge gate 与 agent-edge gate 都不应消费旧 reassurance 残影
5. current execution context 也必须服从 connected-only discipline
6. “刚才看起来能用”与“此刻真正还能用”必须是两层不同 signer

## 6. 苏格拉底式自反诘问：我是不是又把“它刚才看起来能用”误认成了“它现在真的仍然属于可依赖的当前世界”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 `Authentication successful` 已经出现，为什么 `MCPTool.call()` 还要先 `ensureConnectedClient()`？
   因为 reassuring sentence 不拥有豁免 live-use proof 的资格。
2. 如果某个 tool 刚才还在工具列表里，为什么 `runToolUse()` 现在还会报 `No such tool available`？
   因为历史 roster 不等于当前 roster。
3. 如果 session 只是 runtime 中断一下，为什么 `callMCPTool()` 要把它升级成 cache 清理与 session recovery signal？
   因为 runtime contradiction 不是噪音，而是旧 reassurance 失真的正式证据。
4. 如果 list/read/prompt 只是外围入口，为什么它们也要重新 `ensureConnectedClient()`？
   因为 consumer edge 的主权不该外包给别的 surface。
5. 如果 agent 已经看到 server 配置存在，为什么它还只认当前真的有 tools 的 server？
   因为 configuration presence 不等于 live usable capability。
6. 如果 cleanup 线还没正式长出 use-time revalidation grammar，是不是说明这层只是实现细节？
   恰恰相反。越把它当细节，越容易让 reassuring story 冒充 current-use authorization。

这一串反问最终逼出一句更稳的判断：

`use-time revalidation 的关键，不在某个 surface 刚才说得多积极，而在真正依赖发生时系统能不能再次裁定那句积极话是否仍配被当真。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 线未来只要补出 reassurance grammar，就已经足够成熟。`

而是：

repo 已经在 `ReadMcpResourceTool` 与 `ListMcpResourcesTool` 的 consumer-edge hard gate、`ensureConnectedClient()` 的 fresh reconnect primitive，以及 `toolExecution.ts` 的 current-context filter 与 `McpAuthError` demotion 上，明确展示了 use-time revalidation governance 的存在；因此 `artifact-family cleanup stronger-request cleanup-reassurance-governor signer` 仍不能越级冒充 `artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer`。

因此：

`stronger-request cleanup 线真正缺的，不只是“谁来决定现在可以说多放心”，还包括“谁来决定真正消费这一刻，这份放心话是否仍配被当真”。`
