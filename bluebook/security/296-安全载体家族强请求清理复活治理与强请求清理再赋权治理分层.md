# 安全载体家族强请求清理复活治理与强请求清理再赋权治理分层：为什么artifact-family cleanup stronger-request cleanup-resurrection-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer

## 1. 为什么在 `295` 之后还必须继续写 `296`

`295-安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪些旧对象可以重新回到 current world，也还需要 resurrection governor 去决定它们是否能回来、凭什么回来、回来时进入哪一层 world。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“对象回来就说明它原来的资格也一起回来了”
的错觉：

`只要 stronger-request cleanup resurrection governor 已经让旧对象重新出现，它就自动拥有了决定旧 identity、旧配置、旧 secrets、旧 scope 与旧可用资格是否一起恢复的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 re-entitlement 正对照，
会发现它其实明确拆开了五件事：

1. `object return`
2. `enabled truth`
3. `config / secrets trust`
4. `policy veto`
5. `identity seat`

最硬的证据至少有七组：

1. `src/utils/plugins/pluginOptionsStorage.ts:201-208,210-272`
   证明 `deletePluginOptions()` 只在 last installation 卸载时才抹掉旧 `pluginConfigs` 与旧 `pluginSecrets`，
   说明 comeback 不自动带回旧配置与旧 secrets
2. `src/services/plugins/pluginOperations.ts:528-540`
   进一步证明 last-scope remove 与 options/secrets wipe 被显式联动，
   说明对象离场时 entitlement material 会被单独清算
3. `src/services/plugins/pluginOperations.ts:573-727`
   证明 `setPluginEnabledOp()` 把 enable / disable 写成单独的 settings action，
   说明对象回来不等于自动恢复 enabled truth
4. `src/utils/plugins/pluginPolicy.ts:11-19`
   与
   `src/services/plugins/pluginOperations.ts:650-657`
   证明更高层 policy 仍可 veto 旧资格恢复
5. `src/services/plugins/pluginOperations.ts:664-687`
   证明 even after return，
   scope / precedence 仍要经过 override legality 判断，
   不是对象一回来就自动恢复原 seat
6. `src/utils/plugins/installedPluginsManager.ts:827-830,859-861`
   证明 disk / materialized existence 与 effective installed / enabled truth 仍是两层不同真相
7. `src/utils/plans.ts:233-257`
   与
   `src/screens/REPL.tsx:1790-1797`
   证明内容回来时仍可能只拿到 new slug，
   而不是恢复 original identity

这说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

和

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象可以重新出现。`

后者才配说：

`它重新出现之后，算不算原来那个被授权、被配置、被允许继续工作的对象。`

所以 `295` 之后必须继续补的一层就是：

`安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`

也就是：

`stronger-request cleanup resurrection governor 负责让对象回来；stronger-request cleanup re-entitlement governor 才负责决定它回来之后恢复哪些旧资格。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 re-entitlement manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出对象回来不等于资格回来
2. `回来` 与 `恢复旧资格` 是两个不同问题
3. stronger-request cleanup 线如果真要把 resurrection world 做完整，也迟早要回答这些“回来后还算不算原来那个对象”的问题

第二条：

这里说的 `re-entitlement governance`
不是在声称：

`对象回来后必须恢复所有旧资格。`

当前更稳的说法只能是：

`re-entitlement governance 首先回答的是哪些资格可以回来、哪些必须重新授予、哪些根本不该回来。`

第三条：

这里也不是把：

`enable state`

单独当成全部 re-entitlement grammar。

真正关键的是：

1. 配置 / secret 会不会回来
2. enabled truth 会不会回来
3. policy 是否允许回来
4. identity / scope 是否还是原来的那个 seat

第四条：

这里尤其不能过度声称：

`deletePluginOptions()`

就等于完整资格治理。

更稳的说法只能是：

`资格治理` 还要继续回答 post-return enable、policy、scope 与 identity fallback；option/secrets wipe 只是其中一层。`

第五条：

这里也不能过度声称：

`new identity` 一定比 `old identity` 弱。

当前更稳的说法只能是：

`new slug / new seat` 的关键不在强弱，而在系统拒绝把“内容回来”偷写成“原资格原封不动回来”。`

第六条：

这里还不能过度声称：

`re-entitlement` 只是恢复体验之外的附带管理。

更稳的说法只能是：

`它本质上是在治理 post-return qualification grammar；体验只描述回来，资格才决定回来之后还能做什么。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-resurrection-governor signer 仍不等于 stronger-request cleanup-re-entitlement-governor signer`

证据：

1. `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`，
   说明对象回来不自动带回旧配置与旧 secrets
2. `setPluginEnabledOp()` 把 enable / disable 做成单独的 settings write，
   说明对象可 install、materialize、refresh，但是否恢复 enabled truth 仍要经过独立写入
3. `pluginPolicy.ts`
   与
   `pluginOperations.ts`
   明确写出 org policy 可以继续 veto install / enable
4. `pluginOperations.ts`
   继续要求 cross-scope override legality，
   说明 current seat 与 old seat 不是自然延续关系
5. `installedPluginsManager.ts`
   直接证明 materialized existence 与 effective installed / enabled entitlement 不是同一层 truth
6. `copyPlanForFork()`
   与
   `REPL.tsx`
   证明内容回来时仍可能只获得 new identity，而不是 old identity
7. `deletePluginOptions()` 的 best-effort 失败不回滚 uninstall，
   又说明对象离场事实与资格清算成败也被继续分层

因此这一章的最短结论是：

`stronger-request cleanup resurrection governor 最多能说对象能回来；stronger-request cleanup re-entitlement governor 才能说它回来之后恢复什么旧资格。`

再压成一句：

`会回来，不等于还是原来那个被授权的它。`

再压成更哲学一点的话：

`制度知道怎样让对象重新出现，不等于制度已经知道该把哪些旧权利重新还给它。`

## 4. 第一性原理：resurrection governance 回答“能否回来”，re-entitlement governance 回答“回来后恢复哪些资格”

从第一性原理看，
强请求清理复活治理与强请求清理再赋权治理处理的是两个不同主权问题。

`stronger-request cleanup resurrection governor`
回答的是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回到的是哪一层 world
4. 是否沿原 lineage 回来
5. 是否应该避免 clobber

`stronger-request cleanup re-entitlement governor`
回答的则是：

1. 对象回来后是否恢复旧 enabled state
2. 是否恢复旧 secrets
3. 是否恢复旧 pluginConfigs / options
4. 是否恢复旧 scope / precedence 位置
5. 是否恢复旧 identity，还是只给它一个新的 current seat

如果把这两层压成一句
`反正它已经回来了`，
系统就会制造五类危险幻觉：

1. `return-means-enabled`
2. `return-means-configured`
3. `return-means-same-scope`
4. `return-means-same-identity`
5. `existence-means-entitlement`

所以这两层的哲学本质也不同：

`resurrection`

治理的是：

`对象能不能重新进入当前世界。`

`re-entitlement`

治理的是：

`它重新进入之后，还配不配继续拿着原来的权利说话。`

Claude Code 在这里真正先进的地方，
不是它让 comeback path 看起来顺滑，
而是它明确承认：

`重新出现`

和

`重新被授权`

不是同一件事。

这是一种很成熟的安全哲学：

`当前世界可以重新容纳一个旧对象，但不能因此默认它仍配拥有旧权利。`

## 5. 从技术角度看，Claude Code 的先进性在于把 comeback 之后的 seat、资格、配置、policy 与 identity 全部拆开治理，而不是把“回来”误写成一键恢复

### 5.1 plugin 线先证明：对象回来并不自动恢复旧配置、旧 secrets 与旧 enabled truth

`pluginOptionsStorage.ts:201-208`
先把 `deletePluginOptions()` 的制度边界写死：

1. 只在 LAST installation 被移除时调用
2. 不是每次 uninstall 都删
3. 因为 plugin 可能仍在其他 scope 存活
4. keychain cleanup 失败也不回滚 uninstall 本身

继续看
`pluginOptionsStorage.ts:210-272`：

1. settings side 会删 `pluginConfigs[pluginId]`
2. secure storage side 会删 `pluginSecrets[pluginId]`
3. 还会删 `${pluginId}/${server}` 这类 per-server composite keys
4. 最后 clear options cache

这说明 plugin comeback 之后，
系统不会自动假定：

`旧配置还有效`

或

`旧 secrets 还应继续被信任`

再看
`pluginOperations.ts:528-540`
last-scope remove 时会联动 `deletePluginOptions(pluginId)`，
这进一步说明：

`对象离场`

和

`旧资格材料是否继续保留`

被明确写成独立清算语法。

### 5.2 enabled truth 不是 comeback 的自然副作用

`pluginOperations.ts:573-727`
把 `setPluginEnabledOp()` 写成完整独立动作：

1. 先 resolve pluginId 与 scope
2. 再过 policy guard
3. 再过 cross-scope legality
4. 再判断当前 state / idempotency
5. 最后才写 `enabledPlugins`

这说明：

`对象存在`

和

`对象被重新赋予 enabled truth`

是两层不同动作。

系统宁可要求一笔新的 settings write，
也不允许从：

`它回来了`

直接偷推出：

`它又是 enabled 的了`

### 5.3 policy veto 证明更高层主权仍能拒绝旧资格恢复

`pluginPolicy.ts:11-19`
把 policy guard 写得很直白：

1. `enabledPlugins[pluginId] === false`
2. 即视为 blocked
3. install chokepoint、enable op 与 UI filters 共用这份 truth

`pluginOperations.ts:650-657`
又在 enable 前直接挡住：

`Plugin "...\" is blocked by your organization's policy and cannot be enabled`

这说明：

`能回来`

和

`回来后还被允许`

不是同一问题。

### 5.4 scope legality 与 divergence guard 证明 current seat 不是 object return 的自然延续

`pluginOperations.ts:664-687`
在 explicit scope given but plugin is elsewhere 时，
继续做 cross-scope legality 判断：

1. absent from requested scope but present at another scope
2. 这次又不是合法 higher-precedence override
3. 则直接返回 wrong-scope guidance

这说明对象即便回来了，
系统也不会自动假定：

`它还是原来那个 scope 上的有效 seat`

更进一步，
`installedPluginsManager.ts:827-830,859-861`
直接写出：

1. plugins are loaded from `settings.enabledPlugins`
2. 如果 `settings.enabledPlugins` 与 `installed_plugins.json` diverge
3. 就 treat as not-installed so the user can re-enable

这等于直接证明：

`disk / materialized existence`

和

`effective installed / enabled entitlement`

不是同一层 truth。

### 5.5 plan 线给出第二个强正例：内容回来仍可能只拿到 new identity，而不是原来的资格位

`plans.ts:233-257`
的 `copyPlanForFork()` 很清楚：

1. 复制 original content
2. 生成 `newSlug`
3. 写到 `newPlanPath`
4. 避免 original 与 forked sessions clobber

`REPL.tsx:1790-1797`
又把 fork / resume 两条路明确拆开：

1. fork path 调 `copyPlanForFork()`
2. regular resume path 才调 `copyPlanForResume()`

这直接证明：

`内容回来`

不等于

`原身份回来`

从再赋权角度看，
这非常重要。
因为 identity 本身就是 entitlement 的一部分。

如果系统连对象的名字 / seat 都重新分配了，
那它显然没有默认把旧资格整包带回。

## 6. 对 stronger-request cleanup 而言，当前真正缺的不是“能不能回来”，而是“回来后还算不算原来那个被授权的它”

当前 stronger-request cleanup 已经开始显露 resurrection-governance，
但仍缺少 re-entitlement 治理回答：

1. 旧 path / old promise / old receipt world comeback 之后，是否恢复旧 enabled truth，还是只恢复可见存在
2. 旧 cleanup promise comeback 之后，是否恢复旧资格上限，还是必须重新授予
3. 旧对象 comeback 之后，是否继续沿用旧 identity / old seat，还是必须 new identity
4. 旧配置、旧 secrets、旧 policy exception 会不会一起回来
5. 哪些 comeback 只代表 materialization return，不代表 effective entitlement return

这些都不是：

`对象能否回来`

能回答的。

它们已经进入：

`re-entitlement governance`

层。

也就是说，
stronger-request cleanup 今天真正缺的，
不是再补一个 resurrection gate，
而是：

`谁来决定旧对象回来之后恢复哪些旧资格、哪些 seat、哪些 current rights。`

## 7. 苏格拉底诘问：怎样把这一层做得更严格

### 诘问一

`如果对象已经回到 current world，为什么还不能直接说它恢复了原来的资格？`

因为 comeback 只证明：

`它又出现了。`

并不自动证明：

`旧 enabled truth、旧 config trust、旧 scope seat 与旧 identity seat 也一起回来了。`

### 诘问二

`如果系统还保留了一部分旧配置，是否就意味着旧资格仍然有效？`

不意味着。
保留配置可能只是为了别的 scope 仍在使用它，
或者只是为了让系统不误删仍在别处生效的 truth。

### 诘问三

`如果重新授予资格太繁琐，会不会损伤恢复体验？`

会增加摩擦，
但这是必要代价。
否则系统得到的是一种更危险的“平滑”：

`任何旧对象只要回来，就顺手拿回所有旧权利。`

### 诘问四

`那怎样才算更好的 re-entitlement governance？`

至少要继续把五件事制度化：

1. enabled truth
2. config / secrets trust
3. policy veto
4. scope legality
5. identity seat

缺任何一层，
都容易把：

`对象已回来`

误写成：

`它仍是原来那个被授权的它`

## 8. 技术启示与哲学本质

这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 re-entitlement。`

而是更具体地说：

1. 旧对象回来不自动带回旧配置与旧 secrets
2. 旧对象回来不自动带回 enabled truth
3. 旧对象回来仍可能被 policy veto
4. 旧对象回来仍可能不占回旧 seat
5. 旧对象回来仍可能只拿到 new identity

Claude Code 在这里真正先进的地方，
不是它把 comeback 之后的所有资格都自动恢复，
而是它已经在多个 subsystem 里承认：

`重新进入当前世界`

和

`重新拿回旧资格`

不是同一件事。

这是一种很成熟的安全哲学：

`存在可以回来，但权利必须重新被判断。`

## 9. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

因为：

1. resurrection 负责对象回来
2. re-entitlement 负责资格回来
3. config wipe、explicit enable、policy veto、scope legality、divergence guard 与 new-identity fork policy 都证明 repo 已把这两层正式拆开
4. stronger-request cleanup 当前虽已开始理解旧对象如何回来，却还没有对等地回答回来后恢复哪些旧资格

因此，
这一章最终必须压成一句：

`会回来，不等于还是原来那个被授权的它。`
