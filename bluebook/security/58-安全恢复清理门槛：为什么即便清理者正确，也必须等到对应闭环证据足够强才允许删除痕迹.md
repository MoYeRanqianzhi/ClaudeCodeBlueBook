# 安全恢复清理门槛：为什么即便清理者正确，也必须等到对应闭环证据足够强才允许删除痕迹

## 1. 为什么在 `57` 之后还要继续写“清理门槛”

`57-安全恢复清理权限边界` 已经回答了：

`不是任何层都能删除恢复痕迹，必须由对应闭环所有者清理。`

但如果继续往下一层追问，  
还会出现一个更难也更重要的问题：

`即便清理者是对的人，他是不是现在就能清？`

因为“谁能清”与“何时能清”不是同一个问题。  
一个成熟控制面即使已经找对了 owner，  
仍然可能在时机上犯错：

1. owner 还没拿到足够强的 fresh 证据，就提前删除痕迹
2. owner 只看到局部成功，就把全链恢复说成已经完成
3. owner 确实掌管该 trace，但当前状态仍然处在 retry window、manual confirmation 或 resumable trail 中

所以在 `57` 之后，  
统一安全控制台还必须再补一条制度：

`恢复清理门槛。`

也就是：

`正确 clearer 不等于立即 clearer；即便对象归你清，也必须等到对应闭环证据强到足以支持“这条恢复链已可安全删除”为止。`

## 2. 最短结论

从源码看，Claude Code 已经在多个关键对象上把这条门槛做成了状态机约束：

1. notification key 只有在 timeout、`invalidates` 或显式 `removeNotification(key)` 时才消失  
   `src/context/notifications.tsx:78-116,176-212`
2. `plugins.needsRefresh` 只有在 `refreshActivePlugins()` 完成全量重建并写回后才会被清成 `false`  
   `src/utils/plugins/refresh.ts:81-138`
3. plugin 更新场景即便仍属于 plugin 域，也会故意停在 `needsRefresh`，等待用户显式运行 `/reload-plugins`  
   `src/services/plugins/PluginInstallationManager.ts:166-177`、`src/hooks/useManagePlugins.ts:287-303`
4. MCP reconnect trace 会贯穿整个 retry budget，只有成功收敛、最终失败或手工域内动作介入时才清理  
   `src/services/mcp/useManageMCPConnections.ts:363-460,1055-1122`
5. bridge pointer 会跨 SIGINT、跨 transient reconnect failure、跨长会话刷新保留，只有 archive+deregister、stale 或 fatal 才清  
   `src/bridge/bridgeMain.ts:1515-1577,2384-2403,2524-2533,2700-2728`

所以这一章的最短结论是：

`恢复痕迹的删除不仅受权限约束，还受证据强度约束。`

我会把它再压成一句话：

`owner 解决的是“谁有资格忘记”，threshold 解决的是“什么时候才配忘记”。`

## 3. 源码已经证明：Claude Code 在多个地方把“清理时机”收得比“清理权限”更严

## 3.1 notification 系统说明：就算对象属于你，也得等显式失效条件触发

`src/context/notifications.tsx:78-116,176-212` 显示 notification 域确实拥有 notification key 的清理权。  
但它并不是想清就清，  
而是必须满足下面几类触发条件：

1. timeout 到期
2. 新 notification 通过 `invalidates` 明确宣告旧提示失效
3. 显式调用 `removeNotification(key)`

这意味着即便在最表层的提示对象上，  
系统也已经在坚持一条纪律：

`owner 的存在，不自动授予任意时刻删除的自由。`

这条纪律虽然看上去只发生在 UI 层，  
但它实际是在给更高层 trace 立一个模板：

`删除必须由明确的失效事件触发，而不是由“差不多好了”的感觉触发。`

## 3.2 `needsRefresh` 说明：plugin 域 owner 也必须等完整重建闭环后才能清

`src/utils/plugins/refresh.ts:81-138` 很关键。  
`refreshActivePlugins()` 不是一开始就把 `needsRefresh` 清掉，  
而是先做完这些事：

1. 清 plugin caches 与 orphan exclusions
2. 重新加载 plugins
3. 重新拿 commands 与 agentDefinitions
4. 预热 plugin MCP / LSP 相关状态
5. bump `pluginReconnectKey`

直到这些步骤完成后，  
它才在 `setAppState` 里把：

`needsRefresh: false`

真正写回去。

这说明对 plugin 域来说：

1. `needsRefresh` 的 owner 是 plugin refresh 闭环
2. 但就算 owner 正确，也必须等 full rebuild 完成
3. “开始 reload 了”不够
4. “局部组件更新了”也不够

所以这里保护的不是权限边界，  
而是：

`完成强度门槛。`

## 3.3 plugin updates only 场景更进一步说明：同一 owner 也会主动退回人工门槛

`src/services/plugins/PluginInstallationManager.ts:166-177` 与 `src/hooks/useManagePlugins.ts:287-303` 组合在一起尤其重要。

这里最值得注意的不是：

`plugin 域能清理自己的 trace`

而是：

`plugin 域明明就是 owner，却仍然在某些场景拒绝立即清理。`

对新 marketplace 安装，系统允许 auto-refresh；  
但对已有 marketplace 更新，系统只把状态推到：

`needsRefresh=true`

并要求用户显式运行 `/reload-plugins`。

这说明 Claude Code 的设计者其实已经在源码里承认：

`ownership 完整，不代表 threshold 已达标。`

也就是说，  
同一个 owner 会根据当前证据强度，把清理门槛继续细分成：

1. 可以 auto-clear 的场景
2. 必须等人工显式确认的场景

这是非常成熟的控制面设计，  
因为它把“谁负责”与“何时收尾”拆成了两层治理。

## 3.4 MCP reconnect 说明：retry budget 期间 owner 也必须继续保留 trace

`src/services/mcp/useManageMCPConnections.ts:363-460` 显示，  
在 automatic reconnection 过程中，系统会反复写：

1. `pending`
2. `reconnectAttempt`
3. `maxReconnectAttempts`

并把 timer 保留下来，  
直到出现三类更强事件之一：

1. reconnect 成功并收敛
2. 已达上限，最终失败
3. 用户手工 reconnect / enable / disable，明确接管当前域内状态机  
   `src/services/mcp/useManageMCPConnections.ts:1055-1122`

这说明 reconnect 域 owner 并不会因为自己已经拥有清理权，  
就把每次中间失败都当成可删除时刻。  
它真正遵守的是：

`中间尝试不配删除恢复痕迹，只有收敛、终止或显式接管才配。`

换句话说，  
retry window 本身就是清理门槛的一部分。

## 3.5 bridge pointer 说明：高价值恢复锚点的 threshold 比 owner 更重要

`src/bridge/bridgeMain.ts:1515-1577,2384-2403,2524-2533,2700-2728` 把这条原则推到了最高强度。

bridge 域当然是 pointer 的 owner，  
但源码仍然明确区分：

1. resumable SIGINT shutdown  
   -> 不清 pointer
2. transient reconnect failure  
   -> 不清 pointer
3. crash recovery / 长会话运行  
   -> 立即写 pointer，且每小时刷新
4. archive+deregister、pointer stale、fatal reconnect failure  
   -> 才清 pointer

这说明 pointer 的本质不是普通状态位，  
而是：

`高价值恢复入口。`

对于这种对象，  
控制面真正保护的不是“谁能删”，  
而是：

`在仍有恢复价值时，谁都不能删；即便 owner 本人也不行。`

## 4. 第一性原理：清理门槛本质上是在回答“这条恢复链现在还有没有未来价值”

如果从第一性原理追问：

`为什么控制面还要把“时机”单独做成一条原则？`

因为删除一个恢复痕迹时，  
系统不只是在做界面清扫，  
而是在做一个更重的判断：

`这条恢复链从现在开始，已经不需要继续被进入、被重试、被回访、被解释。`

这显然不是单纯的 owner 问题。  
它本质上是在判断：

1. 当前是不是已经拿到足够强的完成证据
2. 当前是不是已经确定没有继续恢复价值
3. 当前是不是已经跨过 retry / resume / manual confirmation 的窗口

所以清理门槛保护的是更底层的公理：

`删除 trace 的前提，不是“有人能删”，而是“继续保留这条 trace 已经不再产生控制价值”。`

## 5. 我给统一安全控制台的“清理门槛模型”

我会把它分成四条判断。

## 5.1 owner 正确，不等于 threshold 达标

先问：

`当前 clearer 是不是 owner 或更高层 signer。`

再问：

`它现在是否已经拿到足够强的 fresh / terminal / manual-confirmed 证据。`

两者缺一不可。

## 5.2 中间成功不配删高价值 trace

以下这些通常都还不够：

1. 某次局部 reconnect 成功
2. 某个 timer 被 cancel
3. 某个提示被替换
4. 某个动作已经被触发

这些都只是：

`过程信号`

而不是：

`收尾信号`

## 5.3 retry window 与 resume window 本身就是安全对象

如果系统仍处于：

1. backoff retry 中
2. resume pointer 仍可用
3. 用户尚未完成显式 `/reload-plugins`
4. restart / `/status` 复核仍未发生

那么“等待窗口”本身就说明：

`清理门槛还没过。`

## 5.4 terminal proof 与 manual proof 才配结束遗留

我会把允许清理的强证据压成三类：

1. full rebuild proof  
   例如完整重建后再清 `needsRefresh`
2. terminal proof  
   例如 archive+deregister、最终失败、确认 stale/fatal
3. manual proof  
   例如必须由用户显式触发并完成确认的 repair path

## 6. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于它限制谁能改状态，还在于它把“何时允许删状态”编码进了状态机。`

这对其他 Agent 平台构建者的直接启示有五条：

1. 清理规则必须同时编码 authority 和 threshold，不能只做 ACL
2. 中间态、retry budget 与 resume window 应被当成正式安全对象，而不是实现细节
3. trace 删除应由强证据触发，而不是由弱成功信号触发
4. 自动清理与人工清理应共享同一条 threshold grammar，而不是各写一套特例
5. 高价值恢复锚点应显式区分 transient failure 与 terminal failure

## 7. 哲学本质

这一章更深层的哲学是：

`真正克制的系统，不只限制谁有遗忘权，还限制遗忘发生的时刻。`

很多系统在权限模型上并不差，  
真正失败的地方在于：

`把“我有资格处理这件事”误写成“我现在就可以把它抹掉”。`

Claude Code 这条恢复治理链到这里，其实已经逼近一条很强的哲学结论：

`遗忘不是所有权的自然延伸，而是闭环证据足够强之后才被授予的终局动作。`

## 8. 苏格拉底式反思：这一章最需要自我追问什么

### 8.1 如果 clearer 已经是 owner，为什么还不能让它自己决定马上清理

因为 owner 可能只拥有对象主权，  
却还没拥有足够强的 closing evidence。

### 8.2 为什么“再等等”不是拖延，而是安全控制的一部分

因为 retry window、resume window、manual confirmation window 本身就在保护再次进入能力。

### 8.3 真正危险的错误是什么

不是痕迹留得久一点，  
而是：

`在仍有恢复价值时过早删除高价值 trace。`


## 9. 结语

`57` 回答的是：谁有权清理恢复痕迹。  
这一章继续推进后的结论则是：

`即便是正确的人，也必须等到正确的证据强度，才配清理恢复痕迹。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不仅要限制谁有权忘记失败，还要限制失败何时才允许被忘记。`
