# 显式降级理论：为什么窄宿主不是问题，隐性子集才是安全问题

## 1. 为什么在 `27` 之后还要继续写“显式降级”

`27-安全对象协议` 已经把一个关键事实说清楚了：

`统一安全控制台要成立，必须先定义哪些安全对象跨宿主必须保真。`

但这还会继续引出一个更具体的问题：

`如果某些宿主天生就是窄的，那它们是不是天然不合格？`

答案并不是。  
真正的问题从来不在于：

`宿主是否支持协议全集。`

真正的问题在于：

`宿主是不是把自己的子集身份说清楚了。`

所以这一章的核心问题是：

`为什么 Claude Code 的多宿主世界里，窄宿主本身不是安全问题，隐性子集才是安全问题？`

## 2. 最短结论

从源码看，Claude Code 的宿主现实本来就是：

`协议全集 + 多种宽度不同的消费子集。`

这并不糟糕。  
真正好的部分是：

1. 某些路径会对未知 control subtype 显式报错
2. 某些路径会明确只处理 `can_use_tool`
3. 某些路径会明确只承担 `interrupt`

真正危险的部分则是：

1. 某些对象被静默忽略
2. 某些对象被文本化后不再声明丢失了哪些语义
3. 某些宿主看起来像“都支持”，其实只支持一部分

所以 `28` 的核心判断是：

`安全控制面的宿主降级没有问题，问题在于降级如果不是显式的，就会把“看不见”伪装成“没问题”。`

## 3. 源码已经清楚表明：不同宿主就是不同宽度的子集

## 3.1 `DirectConnectManager` 是一个非常窄但很诚实的子集

`src/server/directConnectManager.ts:81-99` 说明它处理 inbound `control_request` 时：

1. 只正式处理 `can_use_tool`
2. 其他 subtype 直接回 error

`src/server/directConnectManager.ts:102-112` 又说明它对 SDK message 的转发也有过滤：

- `control_response`
- `keep_alive`
- `control_cancel_request`
- `streamlined_text`
- `streamlined_tool_use_summary`
- `post_turn_summary`

都不继续走普通消息转发。

再往下，`170-200` 说明它额外只主动发送：

- `interrupt`
- permission response
- error response

这是一种很典型的“窄宿主”：

`支持面很窄，但边界非常明确。`

## 3.2 `RemoteSessionManager` 同样是窄子集，而且同样显式

`src/remote/RemoteSessionManager.ts:153-214` 也给出几乎相同的事实：

1. `control_request` 只正式处理 `can_use_tool`
2. 其他 subtype 直接发 `control_response.error`
3. `control_cancel_request` 被单独处理
4. 普通 `control_response` 只做 acknowledgment
5. 其他 `SDKMessage` 再交给 callback

`src/remote/RemoteSessionManager.ts:247-297` 继续说明它主动承担的控制动作也很有限：

- permission response
- `interrupt`

这再次证明：

`窄宿主不是异常，而是系统设计现实的一部分。`

## 3.3 `bridgeMessaging.ts` 则代表更宽的宿主，但仍然不是协议全集

`src/bridge/bridgeMessaging.ts:243-389` 比 direct connect / remote session manager 宽得多，  
它显式支持：

- `initialize`
- `set_model`
- `set_max_thinking_tokens`
- `set_permission_mode`
- `interrupt`

未知 subtype 同样回 error。  
这是一种更宽的、但仍然明确的宿主子集。

与此同时，`src/bridge/bridgeMessaging.ts:132-208` 也说明在这条 ingress path 上：

1. `control_request` / `control_response` 会被专门分流
2. `SDKMessage` 里只有 `user` 会被继续转发
3. non-user inbound message 会被显式忽略

所以 bridge 并不是“完整协议宿主”，  
而是：

`一条更宽、但仍然有边界的子集宿主。`

## 4. 好的降级和坏的降级，差别到底在哪里

## 4.1 好的降级：能力窄，但边界明示

好的降级有三个特点：

1. 它会显式说明只支持哪些 subtype
2. 它会对不支持的 subtype 返回 error，而不是假成功
3. 它不会让用户误以为对象本来就不存在

direct connect 和 remote session manager 在 control path 上，  
大体就属于这一类。

## 4.2 坏的降级：对象没了，但没有人承认它没了

最典型的坏降级，是 `sdkMessageAdapter.ts` 这一类路径：

1. `status` 被压成普通文案
2. `auth_status` 被静默忽略
3. `tool_use_summary`、`rate_limit_event` 被直接忽略

这不是“边界窄”本身的问题，  
而是：

`对象损失没有被显式提升成系统事实。`

这样一来，宿主就会把自己的盲区伪装成：

`世界本来就是这样。`

## 4.3 为什么后者比前者危险得多

因为显式 error 至少会告诉用户：

`这条路径这里不支持。`

而静默忽略会让用户以为：

`系统根本没有这类状态。`

这会直接造成三类安全后果：

1. 错误归因
2. 错误动作
3. 错误升级对象

所以从安全角度看，  
真正危险的不是“宿主不够强”，  
而是：

`宿主把自己的看不见伪装成系统真相。`

## 5. 从第一性原理看，宿主降级必须守住哪三条公理

## 5.1 子集必须显式命名

宿主必须说清楚：

1. 我消费哪一类消息
2. 我处理哪一类 control subtype
3. 我保留哪几类安全对象

否则统一控制台就无法知道：

`当前缺失的是系统状态，还是宿主能力。`

## 5.2 不支持必须显式失败

最好的默认值不是 silent ignore，  
而是：

`显式 error 或显式 degraded。`

前者适用于动作路径，  
后者适用于观察路径。

## 5.3 文本化之前必须先完成对象保留

宿主可以最后只显示一句文案，  
但系统内部必须先保留：

1. 原对象
2. 对象类别
3. 丢失了哪些字段
4. 当前降级级别

否则文本化就会直接变成真相丢失。

## 6. 宿主应该如何把“降级”做成正式语义

我会把正式降级语义压成四段。

## 6.1 `supported`

宿主完整保留对象并支持动作。

## 6.2 `supported_with_projection`

宿主保留对象，但显示层只投影一部分。  
这是可接受的窄显示。

## 6.3 `observable_but_non_actionable`

宿主能看见对象，但不能对其执行完整动作。  
这类状态必须显式说明“可观察，不可控制”。

## 6.4 `degraded_unavailable`

宿主不保留该对象。  
这时必须明确告诉系统和用户：

`这不是正常，而是不可见。`

## 7. 技术先进性与设计启示

这一章最重要的技术判断是：

`Claude Code 的先进性不在于它真的有一个统一宿主，而在于它已经拥有把“多宿主、多宽度”治理成同一控制面的潜力。`

这个潜力来自两点：

1. 协议层已经够结构化
2. 很多窄宿主其实已经在控制路径上坚持“显式失败优于假成功”

因此它下一步要补的，不是更多兼容 hack，  
而是：

`把显式降级从局部工程习惯提升成正式宿主语义。`

对其他 Agent 平台构建者的启示也很直接：

1. 多宿主不是问题
2. 多子集不是问题
3. 隐性子集才是问题

## 8. 哲学本质

这章更深层的哲学是：

`诚实比强大更重要。`

一个窄宿主如果诚实，  
它仍然是系统的一部分。  
一个更强宿主如果不诚实，  
它反而会制造更大的安全黑箱。

所以从第一性原理看，  
多宿主安全控制面的真正伦理不是“谁最全”，  
而是：

`谁最清楚地告诉你自己看见了什么、没看见什么、能做什么、不能做什么。`

## 9. 苏格拉底式反思：这章最容易犯什么错

### 9.1 我们是不是把“显式报错”夸大成万能解

不能。  
显式报错只是底线，  
不是终局。  
更好的状态仍然是对象保真。

### 9.2 宿主过滤是不是都不合理

也不是。  
过滤本身很正常。  
真正要防的是：

`过滤后不说明自己过滤了什么。`

### 9.3 这是不是纯工程风格问题

不是。  
一旦降级语义不显式，  
用户就会把宿主缺口误判成系统事实。  
这已经是安全解释性和控制性的正式问题。

## 10. 结语

`27` 回答的是：统一安全控制台必须先有安全对象协议。  
`28` 继续推进后的结论则是：

`在多宿主世界里，窄宿主本身不是问题；真正的问题，是宿主把自己的子集身份藏起来。`

因此统一安全控制台下一步真正该补的，不只是对象协议，  
还包括：

`正式的显式降级语义。`
