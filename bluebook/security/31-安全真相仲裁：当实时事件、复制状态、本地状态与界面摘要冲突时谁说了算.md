# 安全真相仲裁：当实时事件、复制状态、本地状态与界面摘要冲突时谁说了算

## 1. 为什么在 `30` 之后还要继续写“真相仲裁”

`30-安全真相源层级` 已经说明：

`安全真相不是一个点，而是一套分层复制的真相系统。`

但只把层级列出来还不够。  
因为一旦系统运行起来，真正困难的问题不是：

`有哪些层。`

而是：

`这些层冲突时谁说了算。`

例如：

1. 实时事件说现在是 `running`
2. 复制层的 `worker_status` 还是旧值
3. 本地 `AppState` 还没来得及回灌
4. UI 上已经显示了另一个摘要

这时系统必须有一套正式的仲裁逻辑，  
否则“多层真相”就会退化成“多层互相打架”。

所以这一章的核心问题是：

`Claude Code 的安全控制面里，当实时事件、语义状态、复制状态、本地状态与界面摘要冲突时，究竟谁说了算？`

## 2. 最短结论

从源码看，Claude Code 并不是让所有真相层平权竞争。  
它事实上已经形成了一套隐含仲裁顺序：

1. `当前回合的语义状态更新` 优先于旧复制状态
2. `恢复时拉回的复制状态` 优先于本地 fresh default
3. `明确标注为 authoritative 的 turn-over signal` 优先于一般 UI 末尾消息
4. `本地交互状态` 必须通过 choke point 再镜像出去，不能自封为最终权威
5. `界面摘要` 只能跟随，不得仲裁

所以 `31` 的核心判断是：

`安全真相层级不是静态目录，而是一套带优先级的仲裁系统。`

## 3. 第一原则：恢复时旧真相优先于本地默认值

## 3.1 `print.ts` 明确避免“用 fresh default 覆盖 restored state”

`src/cli/print.ts:5048-5057` 有一句非常关键的注释：

`Await restore alongside hydration so SSE catchup lands on restored state, not a fresh default.`

这句话几乎就是仲裁原则本身。

它说明恢复时系统不接受这样一种顺序：

1. 先以本地 fresh default 启动
2. 再让实时事件慢慢把它纠正回来

相反，它强制采用：

1. 同时等待 transcript hydrate 与 `restoredWorkerState`
2. 先把 metadata 回灌进 `AppState`
3. 再让后续实时流落在这个恢复好的基底上

这说明恢复时的仲裁原则非常明确：

`复制真相优先于本地默认值。`

## 3.2 `externalMetadataToAppState(...)` 说明恢复不是全量覆盖，而是选择性仲裁

`src/state/onChangeAppState.ts:23-40` 又显示，  
复制真相灌回本地时并不是全量覆盖一切，  
而是只恢复：

- `permission_mode`
- `is_ultraplan_mode`

这说明恢复仲裁不是“远程一定全赢”，  
而是：

`只有被认定为跨进程关键语义的字段，才有资格覆盖本地默认状态。`

因此真正的仲裁不是粗暴覆盖，  
而是：

`按语义重要性做选择性恢复。`

## 4. 第二原则：会话语义真相优先于普通界面消息

## 4.1 `session_state_changed` 被明确标注为 authoritative turn-over signal

`src/entrypoints/sdk/coreSchemas.ts:1739-1745` 对 `session_state_changed` 的描述非常重要：

`'idle' fires after heldBackResult flushes and the bg-agent do-while exits — authoritative turn-over signal.`

`src/utils/sdkEventQueue.ts:56-66` 也重复了同样的意思：

`the authoritative "turn is over" signal`

这说明系统在“回合是否真的结束”这个问题上，  
并不让普通 message 自行竞争解释权。  
它明确设立了一个更高优先级的语义事件。

也就是说，在“turn 是否完成”这个问题上，  
仲裁顺序不是：

`谁最后显示出来谁算。`

而是：

`session_state_changed(idle) 说了算。`

## 4.2 `print.ts` 还专门防止这些系统事件污染 lastMessage 语义

`src/cli/print.ts:887-914` 明确把：

- `session_state_changed`
- `task_notification`
- `task_started`
- `task_progress`
- `post_turn_summary`

排除在普通 `lastMessage` 竞争之外。

这说明系统已经知道：

`UI 的“最后一条消息”并不等于系统的“最终语义状态”。`

所以在这里，仲裁又非常明确：

`语义事件优先于普通转录消息。`

## 5. 第三原则：复制真相允许延迟一致，但不允许失去清理权

## 5.1 `WorkerStateUploader` 说明复制层不追求瞬时一致，而追求可收敛一致

`src/cli/transports/WorkerStateUploader.ts:4-17`、`39-47`、`69-84` 显示：

1. `PUT /worker` 是 coalescing uploader
2. patch 可以合并
3. 失败会退避重试
4. metadata 走 RFC 7396 merge

这说明复制层天然接受：

`延迟一致。`

但它并不接受：

`无边界脏状态。`

因为同一个文件又明确规定了：

`null` 用于服务端删除。

所以复制层的仲裁哲学是：

`允许晚一点对齐，但不允许永远不清。`

## 5.2 `ccrClient.initialize()` 说明 stale metadata 必须先被清掉

`src/cli/transports/ccrClient.ts:473-489` 在 init PUT 时明确写入：

- `worker_status: 'idle'`
- `external_metadata.pending_action: null`
- `external_metadata.task_summary: null`

注释也明确说这是为了清掉 prior worker crash 留下的 stale 状态。

这意味着在复制层，仲裁顺序不是：

`旧状态只要存在就永远比新状态优先。`

而是：

`可恢复旧状态优先于默认值，但 stale crash residue 低于明确初始化清理。`

这是一条非常成熟的真相仲裁原则。

## 6. 第四原则：本地状态可以驱动交互，但必须通过 choke point 出口

## 6.1 `onChangeAppState.ts` 明确拒绝“本地状态偷偷自封为真相”

`src/state/onChangeAppState.ts:50-92` 这段很关键。

它明确说明过去很多 mode 变化路径只改了本地 `AppState`，  
却没有把变更告诉 CCR 和 SDK，  
导致外部面看到的仍是 stale mode。

因此作者把 mode diff 统一收口到这里，  
让任何 setAppState 只要真的改了 mode，  
就必须继续：

1. `notifySessionMetadataChanged(...)`
2. `notifyPermissionModeChanged(...)`

这说明在“本地状态 vs 对外真相”冲突时，  
系统的原则不是：

`本地先改了就算。`

而是：

`本地状态只有经过统一出口镜像出去后，才有资格进入更高层真相。`

## 6.2 这也是为什么 `AppState` 不能被当成单独的终局权威

它可以非常真实地代表：

`当前宿主正在怎么交互。`

但它不能单独代表：

`其他宿主、恢复面、复制面也已经知道同样的事实。`

所以本地状态的仲裁位置是：

`中间层，不是最终层。`

## 7. 第五原则：UI 投影永远没有仲裁权

如果说前四层都多少可以参与真相仲裁，  
那么 UI 投影层最重要的性质就是：

`它没有仲裁权。`

它只能：

1. 跟随
2. 投影
3. 压缩

但不能决定：

1. 当前到底是哪个状态
2. 哪条证明链成立
3. 恢复是否已经完成

这正是为什么：

- `sdkMessageAdapter` 的文本化结果
- footer pill 的状态标签
- settings 摘要页的最终文案

都不能被直接当成安全真相本体。

## 8. 把这套仲裁顺序压成一句话

我会把它压成一句话：

`恢复时，复制真相赢过默认值；运行时，语义状态赢过普通消息；传播时，choke point 赢过私下本地改动；显示时，UI 永远不赢。`

这其实已经非常接近一条可操作的系统原则。

## 9. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只是它有多层真相，而是这些层之间已经隐含了很强的仲裁纪律。`

这种纪律体现在：

1. 恢复顺序被明确写进注释和代码
2. authoritative turn-over signal 被明确命名
3. stale metadata 会在 init 时被主动清理
4. 本地状态必须通过统一出口再进入复制/事件面

这对其他 Agent 平台构建者的启示非常直接：

1. 多层真相不可避免
2. 真相层级一旦存在，就必须再定义冲突仲裁
3. 最危险的不是层级多，而是没有优先级

## 10. 哲学本质

这一章更深层的哲学是：

`真相层级如果没有仲裁规则，就不是真相系统，只是多份意见。`

真正的安全控制面，不是“把更多状态暴露出来”就结束，  
而是：

`要明确什么时候该相信历史，什么时候该相信实时，什么时候该相信复制，什么时候只把界面当作表象。`

## 11. 苏格拉底式反思：这章最容易犯什么错

### 11.1 我们是不是把复制层说得太弱了

不能。  
复制层非常强，  
只是它不该被误解成原始事件面。

### 11.2 实时真相是不是永远最高

也不是。  
恢复时如果没有 restored state，  
实时流可能会先落在错误默认值上。  
所以“实时”不是在所有语境下都无条件最高。

### 11.3 UI 真的一点仲裁作用都没有吗

严格说没有。  
它可以帮助人理解，  
但它不该决定系统语义。

## 12. 结语

`30` 回答的是：安全真相为什么必须分层。  
`31` 继续推进后的结论则是：

`分层之后还必须有仲裁。`

只有当恢复顺序、语义事件、复制清理、本地镜像和 UI 跟随之间的优先级被明确下来，  
安全控制台才不会从“多层真相”重新退化成：

`多层互相争吵。`
