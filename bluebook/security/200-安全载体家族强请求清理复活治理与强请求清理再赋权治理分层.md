# 安全载体家族强请求清理复活治理与强请求清理再赋权治理分层：为什么artifact-family cleanup stronger-request cleanup-resurrection-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer

## 1. 为什么在 `199` 之后还必须继续写 `200`

`199-安全载体家族强请求清理墓碑治理与强请求清理复活治理分层` 已经回答了：

`stronger-request cleanup 线即便知道哪些旧对象可以重新回到 current world，也还需要 resurrection governor 去决定它们是否能回来、凭什么回来、回来时进入哪一层 world。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“对象回来就说明它原来的资格也一起回来了”的错觉：

`只要 stronger-request cleanup resurrection governor 已经让旧对象重新出现，它就自动拥有了决定旧 identity、旧配置、旧 secrets、旧 scope 与旧可用资格是否一起恢复的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/utils/plugins/pluginOptionsStorage.ts:210-267` 的 `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`
2. `src/services/plugins/pluginOperations.ts:573-734` 的 `setPluginEnabledOp()` 把 enable/disable 做成单独的 settings write，说明对象回来不等于自动恢复 enabled state
3. `src/utils/plugins/pluginPolicy.ts:1-18` 的 policy guard 说明更高主权仍可拒绝恢复旧资格
4. `src/utils/plugins/installedPluginsManager.ts:820-861` 的 settings divergence guard 说明 disk/materialized return 不等于 effective entitlement return
5. `src/utils/plans.ts:234-258` 的 `copyPlanForFork()` 会用 new slug，而不复用 original slug，说明内容回来不等于 old identity 恢复

这说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

和

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象可以重新出现。`

后者才配说：

`它重新出现之后，算不算原来那个被授权、被配置、被允许继续工作的对象。`

所以 `199` 之后必须继续补的一层就是：

`安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`

也就是：

`stronger-request cleanup resurrection governor 负责让对象回来；stronger-request cleanup re-entitlement governor 才负责决定它回来之后恢复哪些旧资格。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 re-entitlement manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出对象回来不等于资格回来
2. “回来”与“恢复旧资格”是两个不同问题
3. stronger-request cleanup 线如果真要把 resurrection world 做完整，也迟早要回答这些“回来后还算不算原来那个对象”的问题

第二条：

这里说的 `re-entitlement governance`

不是在声称：

`对象回来后必须恢复所有旧资格。`

当前更稳的说法只能是：

`re-entitlement governance 首先回答的是哪些资格可以回来、哪些必须重新授予、哪些根本不该回来。`

第三条：

这里也不是把：

`enable state`

单独当成全部 re-entitlement grammar。

真正关键的是：

1. 配置/secret 会不会回来
2. enabled truth 会不会回来
3. policy 是否允许回来
4. identity / scope 是否还是原来的那个 seat

第四条：

这里尤其不能过度声称：

`只要旧对象重新 materialize，就已经恢复旧 effective entitlement。`

当前更稳的说法只能是：

`repo 明确把 object existence 与 effective entitlement 分开；前者只是 comeback，后者还要经过更高层治理。`

第五条：

这里也不能过度声称：

`new identity` 一定比 `old identity` 弱。`

当前更稳的说法只能是：

`new slug / new seat 的关键不在强弱，而在系统拒绝把“内容回来”偷写成“原资格原封不动回来”。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类“stronger-request cleanup-resurrection-governor signer 仍不等于 stronger-request cleanup-re-entitlement-governor signer”证据：

1. `pluginOptionsStorage.ts:210-267` 的 `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`，说明对象回来不自动带回旧配置与旧 secrets
2. `pluginOperations.ts:573-734` 的 `setPluginEnabledOp()` 说明 enable/disable 是单独 settings write，对象可 install、materialize、refresh，但是否恢复 enabled truth 仍要经过独立写入
3. `pluginPolicy.ts:1-18` 明确写出 org-blocked plugins 不能 install 或 enable，说明对象即便能回来，也未必恢复旧 entitlement
4. `installedPluginsManager.ts:820-861` 的 divergence guard 说明 disk/materialized return 不等于 effective installed/enabled truth
5. `plans.ts:234-258` 的 `copyPlanForFork()` 用 new slug 而不复用 original slug，说明内容可以回来，但不自动恢复旧 identity

因此这一章的最短结论是：

`stronger-request cleanup resurrection governor 最多能说对象能回来；stronger-request cleanup re-entitlement governor 才能说它回来之后恢复什么旧资格。`

再压成一句：

`会回来，不等于还是原来那个被授权的它。`

## 4. 第一性原理：resurrection governance 回答“能否回来”，re-entitlement governance 回答“回来后恢复哪些资格”

从第一性原理看，
强请求清理复活治理与强请求清理再赋权治理处理的是两个不同主权问题。

stronger-request cleanup resurrection governor 回答的是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回到的是哪一层 world
4. 是否沿原 lineage 回来
5. 是否应该避免 clobber

stronger-request cleanup re-entitlement governor 回答的则是：

1. 对象回来后是否恢复旧 enabled state
2. 是否恢复旧 secrets
3. 是否恢复旧 pluginConfigs / options
4. 是否恢复旧 scope / precedence 位置
5. 是否恢复旧 identity，还是只给它一个新的 current seat

如果把这两层压成一句“反正它已经回来了”，
系统就会制造五类危险幻觉：

1. return-means-enabled illusion  
   只要对象回来，就误以为它自动恢复旧可用资格
2. return-means-configured illusion  
   只要对象回来，就误以为旧配置与 secrets 也自动回来
3. return-means-same-scope illusion  
   只要对象回来，就误以为它自动回到原来的 scope 和 precedence
4. return-means-same-identity illusion  
   只要内容回来，就误以为对象还是旧 identity
5. existence-means-entitlement illusion  
   只要对象存在于 disk/materialized world，就误以为它有 current effective entitlement

所以从第一性原理看：

`resurrection governor` 决定对象能否重新出现；  
`re-entitlement governor` 决定对象回来之后恢复什么旧资格，或者只给它一个更弱的新资格。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup resurrection governance 已经等于 re-entitlement governance，为什么 `deletePluginOptions()` 还要在 last-scope uninstall 时主动抹掉旧配置与旧 secrets？
   因为“对象回来”与“旧配置继续值得信任”根本不是同一句制度。
2. 如果对象一旦回来就自动恢复旧 enabled truth，为什么 `setPluginEnabledOp()` 还要成为独立写入动作，并且继续受 policy guard 约束？
   因为 comeback 与 entitlement return 是两层主权。
3. 如果内容回来就等于 old identity 回来，为什么 `copyPlanForFork()` 还要明确生成 new slug？
   因为 resurrection 只回答内容能否重现，re-entitlement 还要回答它是否仍算原 seat。

## 5. plugin 线给出最强正例：插件可以回来，但旧 options / secrets / enabled state 不会自动一起回来

`src/utils/plugins/pluginOptionsStorage.ts:210-267` 非常关键。
`deletePluginOptions()` 明确写出：

1. last installation 卸载时会清 `settings.pluginConfigs[pluginId]`
2. 同时清 `secureStorage.pluginSecrets[pluginId]`
3. 包括 per-server composite keys 也一起删

这说明插件对象即便后来重新 install、重新 materialize、重新 refresh，
也不自动等于：

`旧配置回来`

或

`旧 secrets 回来`

继续看 `src/services/plugins/pluginOperations.ts:573-734`：
`setPluginEnabledOp()` 明确把 enable/disable 做成单独的 settings write。

这里的关键不是“能不能 enable”，
而是：

`对象存在`

和

`对象被重新赋予 enabled entitlement`

是两层动作。

从技术角度看，这非常值钱。
它把：

1. resurrection of object
2. re-entitlement of object

拆成了两个 chokepoint，
避免系统把“东西回来了”误写成“它还能继续按旧资格工作”。

## 6. policy guard 与 divergence guard 再证明：回来不等于重新被允许，也不等于重新有效

`src/utils/plugins/pluginPolicy.ts:1-18` 的判断非常直白：

`policySettings.enabledPlugins[pluginId] === false`

就视为 blocked。

而 `setPluginEnabledOp()` 也直接在 enable 前挡住它。

这说明：

1. resurrection 不是 entitlement 的最高主权
2. 更高层的 policy 仍可拒绝 entitlement

继续看 `src/utils/plugins/installedPluginsManager.ts:820-861`：
这里明确写出：

1. plugins are loaded from `settings.enabledPlugins`
2. 如果 `settings.enabledPlugins` 与 `installed_plugins.json` diverge，就返回 `false`
3. 甚至要 `treat as not-installed so the user can re-enable`

这等于直接证明：

`disk/materialized existence`

和

`effective installed/enabled entitlement`

不是同一层 truth。

这条设计并不追求“更方便”，
它追求的是：

`不让 stale installation metadata 直接偷签 current entitlement。`

## 7. plan 线给出第二组强正例：内容回来仍可能只拿到新 identity，而不是原来的资格位

`src/utils/plans.ts:234-258` 的 `copyPlanForFork()` 很清楚：

1. 复制 original content
2. 生成 new slug
3. 明确避免 clobber original session

这说明 plan line 已经明确写出：

`内容回来`

不等于

`原身份回来`

从再赋权角度看，这非常重要。
因为 identity 本身就是 entitlement 的一部分。
一个对象即便回来了，也可能只被赋予：

`new seat, not old seat`

这正是 re-entitlement governance 的本体。

## 8. 技术先进性与哲学本质：Claude Code 真正先进的地方，是它把“允许历史对象回来”继续拆成资格、配置与身份三层

从技术角度看，
Claude Code 在这条线上的先进性，
不只是它允许某些旧对象回来，
而是它已经在多个子系统里承认：

1. return != configured
2. return != enabled
3. return != allowed
4. return != same identity

这背后的哲学本质是：

`真正成熟的系统，不只知道怎样让历史对象回来，还知道回来之后它是不是仍配占据原来的资格位。`

这也带来三条直接技术启示：

1. 任何 stronger-request cleanup grammar 若只定义 resurrection gate、不定义旧资格恢复规则，就仍缺 post-return entitlement discipline
2. plugin 线的 `delete options/secrets -> explicit enable -> policy veto -> divergence guard` 四段式说明，成熟系统不会把 comeback 直接偷写成 effective entitlement
3. plan 线的 `fork -> new slug` 说明，identity 本身就是 entitlement 的一部分，不应被内容恢复自动冒充

## 9. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要定义了 resurrection governance，就已经自动拥有了决定旧对象回来后恢复哪些旧资格的主权。`

而是：

`repo 在 config/secret wipe、explicit enable entitlement、policy veto、existence-vs-entitlement guard 与 new-identity fork policy 上已经明确展示了 re-entitlement governance 的存在；因此 artifact-family cleanup stronger-request cleanup-resurrection-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

再压成最后一句：

`复活负责让对象回来；再赋权，才负责决定它回来后还算不算原来那个被授权的它。`
