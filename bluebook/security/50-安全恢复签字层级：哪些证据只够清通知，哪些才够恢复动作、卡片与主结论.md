# 安全恢复签字层级：哪些证据只够清通知，哪些才够恢复动作、卡片与主结论

## 1. 为什么在 `49` 之后还要继续写“恢复签字层级”

`49-安全完成差异清除纪律` 已经回答了：

`宿主盲区恢复时不能先删旧告警，必须等足够强的 fresh 证据。`

但如果继续往下一层追问，  
还会出现一个更精确也更工程化的问题：

`不同强度的 fresh 证据，到底分别有资格恢复什么？`

因为在真实系统里，  
“恢复”从来不是一个单一动作。  
它至少可以拆成四层：

1. 清除某条通知
2. 恢复某张卡的乐观文案
3. 放开某组动作
4. 恢复主结论与完成解释权

而这些层显然不该由同一类证据统一签字。  
如果把它们都交给同一种低层 signal，  
控制面就会重新退化成：

`表面恢复 = 全面恢复`

所以在 `49` 之后，  
统一安全控制台还必须再补一条更底层的制度：

`恢复签字层级。`

也就是：

`哪些恢复信号只配清提示，哪些信号可以恢复动作，哪些信号才配恢复主结论。`

## 2. 最短结论

从源码看，Claude Code 已经把不同强度的恢复证据分散做在不同层里：

1. `connected / active / stateLabel` 一类 transport 可见性  
   `src/bridge/replBridgeTransport.ts:289-299`、`src/components/BridgeDialog.tsx:137-156`
2. `state_restored / restoredWorkerState` 一类会话恢复与旧元数据回读  
   `src/cli/transports/ccrClient.ts:474-522`、`src/cli/remoteIO.ts:126-167`、`src/cli/structuredIO.ts:139-142`
3. `reportState(...)` / `session_state_changed(...)` 一类会话状态签字  
   `src/cli/remoteIO.ts:159-167`、`src/utils/sdkEventQueue.ts:56-66`、`src/cli/print.ts:818`
4. `files_persisted` 一类后效应签字  
   `src/cli/print.ts:2261-2267`
5. 账本补齐、读写链恢复、host blindspot 缩小一类解释资格恢复  
   `src/bridge/replBridgeTransport.ts:289-299`、`49` 主线归纳

所以这一章的最短结论是：

`恢复证据必须分层签字，低层信号只能恢复低层对象，高层结论必须等待更强 signer。`

最危险的错误则是：

`把 connected、active、no error 这类低层活性信号，错当成高层恢复签字。`

## 3. 源码里已经存在至少五种不同强度的恢复 signer

## 3.1 第一层 signer：活性 signer，只能证明“通道看起来活着”

`src/bridge/replBridgeTransport.ts:289-299` 写得非常直接：

1. `isConnectedStatus()` 只代表 write-readiness
2. 不代表 read-readiness
3. `getStateLabel()` 只是合成的观察标签

这说明 `connected`、`init`、`connecting` 这类东西，  
本质上都只是：

`活性 signer`

它们最多能签什么？

1. transport 当前没完全死
2. 某些写路径也许可用
3. debug/状态行可以显示“当前大致连着”

它们绝不能签什么？

1. 读链已经恢复
2. 完成解释权已经恢复
3. 依赖完整回读的动作可以放开
4. 主卡可以回到“已恢复 / 已完成”

我会把这一层压成一句话：

`活性 signer 只配清除“完全断了”的表层恐慌，不配恢复高层结论。`

## 3.2 第二层 signer：恢复 signer，证明“旧状态已经被重新接回”

`src/cli/transports/ccrClient.ts:474-522` 很关键。  
这里同时做了三件事：

1. 取回旧 worker state
2. 成功初始化新 worker
3. 之后才记录 `cli_worker_state_restored`

`src/cli/remoteIO.ts:126-167` 又说明：

1. `CCRClient.initialize()` 之后把 `restoredWorkerState` 暴露出来
2. 同时把 `reportState` / `reportMetadata` 监听器接上

而 `src/cli/structuredIO.ts:139-142` 则明确写出：

`restoredWorkerState` 在不支持 restore 的 transport 上就是 `null`

这说明 `state_restored` / `restoredWorkerState` 这一类 signer 的真实含义是：

`旧状态已重新接回，不是 fresh default。`

这层 signer 比活性 signer 强，  
因为它已经能防：

`空白冒充恢复`

但它仍然不够强到直接恢复主结论。  
因为它证明的只是：

`旧状态接回来了`

而不是：

`当前解释链和后效应都已经 fresh。`

## 3.3 第三层 signer：状态 signer，证明“当前会话语义处在什么状态”

`src/cli/remoteIO.ts:159-167` 会把：

1. `setSessionStateChangedListener`
2. `setSessionMetadataChangedListener`

接到：

1. `ccrClient.reportState(...)`
2. `ccrClient.reportMetadata(...)`

`src/cli/print.ts:808-818` 又说明 permission prompt 到来时，  
会调用：

`notifySessionStateChanged('requires_action', details)`

而 `src/utils/sdkEventQueue.ts:56-66` 更明确把：

`session_state_changed(idle)`

定义成 authoritative 的 “turn is over” signal。

这说明 `running / requires_action / idle` 一类信号，  
本质上是：

`状态 signer`

它们比 restore signer 更强。  
因为它们已经不是在说“旧状态接回来了”，  
而是在说：

`当前会话语义目前处在什么阶段。`

这一层 signer 最适合恢复什么？

1. 某些会话态卡片
2. 某些依赖 `requires_action` / `running` / `idle` 的动作可用性
3. 某些“当前阶段”相关告警

但它依然不自动等于：

`文件已落盘`
`所有后效应都完成`
`宿主盲区已经消失`

## 3.4 第四层 signer：后效应 signer，证明“结果之外的副作用也完成了”

`src/cli/print.ts:2261-2267` 单独把：

`files_persisted`

做成 system event。  
这件事极其重要，  
因为它说明作者明确拒绝：

`result 一到 = 后效应一切完成`

因此 `files_persisted` 这一类 signer 本质上是：

`后效应 signer`

它们最适合恢复什么？

1. 与文件落盘有关的卡片结论
2. 与持久化副作用有关的动作放开
3. 那些不能只靠 result 或 idle 来恢复的更高层断言

这层 signer 也很强，  
但仍不是最顶层。  
因为哪怕副作用完成了，  
如果宿主账本仍缺、读链仍薄、解释 ceiling 仍低，  
主结论依然不该恢复说满。

## 3.5 第五层 signer：解释 signer，证明“宿主盲区真的缩小到了可恢复主结论”

前面几章已经建立过一条连续链：

1. `missing_ledgers[]`
2. `completion_claim_ceiling`
3. `write_ready/read_ready`
4. `supported_control_subtypes[]`
5. host projection width

这些对象共同决定的是：

`解释权限`

所以最顶层的恢复 signer 其实不是单一事件，  
而是：

`解释 signer`

它的真实含义是：

1. 宿主盲区已缩小
2. 必要账本已补齐
3. 相关读链 / 持久链 / 状态链已经恢复到当前结论所需强度

只有这一层 signer 到位，  
控制面才有资格：

1. 恢复主结论
2. 恢复完整动作集
3. 把“当前仍有宿主盲区”的告警真正撤销

## 4. 第一性原理：恢复不是一个点，而是一串逐层变强的签字

如果从第一性原理追问：

`为什么“恢复签字层级”必须被单独写出来？`

因为系统真正恢复的，  
从来不是一个单点对象。  
它同时恢复的是：

1. 通道
2. 会话状态
3. 持久副作用
4. 解释资格

而这四者的恢复速度天然不同。  
所以成熟系统的正确姿势不该是：

`等最强证据来了再统一恢复所有东西`

也不该是：

`最弱证据一来就统一恢复所有东西`

而应当是：

`每一层对象由恰好足够强的 signer 恢复。`

这条原则可以压成一句话：

`恢复权限必须和证据强度同阶。`

## 5. 我给统一安全控制台的恢复签字层级

我会把它正式分成五层。

### 5.1 L0 活性 signer

代表：

1. `connected`
2. `active`
3. `stateLabel`
4. “当前没报错”

它只能恢复：

1. 某些极表层存活提示
2. 某些 transport 类低风险通知

它绝不能恢复：

1. 主卡乐观文案
2. 依赖回读的动作
3. 主结论

### 5.2 L1 restore signer

代表：

1. `state_restored`
2. `restoredWorkerState`
3. `restore + hydrate` 已排到正确顺序

它可以恢复：

1. 某些“旧状态已重新接上”的卡片
2. 某些 stale residue 告警

但仍不该恢复：

1. 完整动作集
2. 最高层完成结论

### 5.3 L2 state signer

代表：

1. `reportState(running / requires_action / idle)`
2. `session_state_changed(...)`

它可以恢复：

1. 会话阶段类卡片
2. 某些阶段相关动作
3. 某些 `requires_action` / `idle` 类告警

但仍不自动恢复：

1. 与持久化副作用有关的结论
2. 与解释账本完整性有关的主结论

### 5.4 L3 effect signer

代表：

1. `files_persisted`
2. 其他明确独立于 result 的副作用完成信号

它可以恢复：

1. 持久层相关文案
2. 依赖副作用已完成的动作

但依然不自动恢复：

1. 所有宿主盲区告警
2. 所有解释权限

### 5.5 L4 explanation signer

代表：

1. 账本补齐
2. host blindspot 收缩
3. `completion_claim_ceiling` 恢复
4. 必要读链 / 持久链 / 状态链都达到当前结论所需强度

只有这一层才配恢复：

1. 主结论
2. 全量动作集
3. “当前仍有宿主盲区”的最高层告警

## 6. 为什么窄宿主天然拿不到高层 signer

`src/server/directConnectManager.ts:81-98` 和 `src/remote/RemoteSessionManager.ts:192-213` 都说明：

1. 它们几乎只正式支持 `can_use_tool`
2. 对其他 control subtype 会显式报 unsupported

这意味着某些窄宿主从协议宽度上就不具备：

`完整恢复高层解释权`

的资格。  
因为它们连完整控制语义都看不到，  
更不可能替整条恢复链签最高层字。

这和前面的 host blindspot 章节完全一致：

`宿主能参与恢复，不等于宿主有资格恢复主结论。`

## 7. 为什么 `connected / active` 永远不够当最高层 signer

这是整条链里最需要被反复强调的点。

因为一旦系统把：

1. `connected`
2. `active`
3. `no error`

这类词错当成最高层 signer，  
控制面就会出现三种典型谎言：

1. 读链没恢复，但动作先恢复了
2. 账本没补齐，但主卡先改口了
3. 副作用没完成，但“已完成”先显示了

所以对于统一安全控制台，我会给一条硬公理：

`connected 只能签活性，不能签解释。`

## 8. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于它有很多状态，而在于这些状态天然已经分布在不同强度的语义层上。`

它至少已经做对了五件事：

1. 把 write-readiness 和 read-readiness 分开  
   `replBridgeTransport.ts:289-299`
2. 把 restore 放到 fresh default 之前  
   `print.ts:5050-5055`
3. 把 stale metadata 清理与 state_restored 分离  
   `ccrClient.ts:474-522`
4. 把会话状态 signer 单独通过 `reportState` / `session_state_changed` 输出  
   `remoteIO.ts:159-167`、`sdkEventQueue.ts:56-66`
5. 把副作用 signer 单独做成 `files_persisted`  
   `print.ts:2261-2267`

对其他 Agent 平台构建者，这里的直接启示是：

1. 恢复不应只有一个绿灯
2. signer hierarchy 应先于 UI hierarchy
3. transport signer、state signer、effect signer、explanation signer 不应混用
4. 窄宿主应诚实暴露自己拿不到哪些高层 signer

## 9. 哲学本质

这一章更深层的哲学是：

`真实恢复是一种层级秩序，而不是一块统一变绿的面板。`

很多系统的问题不是不会恢复，  
而是太想把恢复压成一个更好看的统一词。  
Claude Code 源码里那些看似分散的细节，  
其实共同指向同一个更成熟的方向：

`每一种恢复对象都只接受与自己同阶的证据签字。`

从第一性原理看，  
这保护的是下面这条公理：

`如果最低层活性信号可以替最高层解释权签字，系统最终一定会把局部恢复说成全面恢复。`

## 10. 苏格拉底式反思：这一章最容易犯什么错

### 10.1 会不会把层级分得太细，导致系统难以实现

会。  
所以重点不是无限细分，  
而是先把最危险的混用断开：

1. 活性 vs 解释
2. restore vs fresh
3. state vs effect

### 10.2 会不会反过来把所有恢复都推迟到最高层 signer

也会。  
这会让系统看起来永远不恢复。  
正确做法不是“只认最高层”，  
而是：

`每层对象使用对应层级 signer。`

### 10.3 这一章之后还缺什么

还缺一张更压缩的 signer 矩阵：

`signer -> 可恢复对象 -> 不可恢复对象 -> 典型误读`

也就是说，  
下一步最自然的延伸就是：

`appendix/34-安全恢复签字层级速查表`

## 11. 结语

`49` 回答的是：旧结论不能被过早删除。  
这一章继续推进后的结论则是：

`不同层级的恢复对象，必须等待不同强度的 signer；低层 signal 只能恢复低层对象，高层结论必须等待高层签字。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不是只会谨慎地说“还没恢复”，而是能精确地区分“到底恢复到了哪一层”。`
