# 安全载体家族强请求清理隔离治理与强请求清理家族宪法治理分层：为什么artifact-family cleanup stronger-request cleanup-isolation-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-constitution-governor signer

## 1. 为什么在 `221` 之后还必须继续写 `222`

`221-安全载体家族强请求清理保留期执行诚实性治理与强请求清理隔离治理分层` 已经回答了：

`谁来证明 stronger-request cleanup 当前不会误伤同项目里仍在运行、仍在读取或仍在写入旧 carrier 的 live peer。`

但如果继续往下追问，
还会碰到另一层更容易被偷写的错觉：

`既然系统现在已经把 cleanup truth 说得更诚实，也已经知道 cleanup 不能误伤 same-project live peer，那么是不是就已经等于所有 stronger-request old cleanup carriers 都生活在同一套 cleanup law 里？`

Claude Code 当前源码给出的答案仍然是否定的。

因为继续看：

1. `src/utils/task/diskOutput.ts:31-49` 把 task outputs 提升到 `project temp dir / sessionId / tasks`，并把 same-project concurrent-session clobber 直接写进路径注释
2. `src/utils/permissions/filesystem.ts:1688-1700` 又明确写出 project temp dir 是 intentionally cross-session readable shared space
3. `src/utils/sessionStorage.ts:198-220`、`src/utils/toolResultStorage.ts:94-118` 与 `src/utils/cleanup.ts:155-257` 则把 transcripts 与 persisted tool-results 留在 `projects/<project>/<session>` 的 sweep world
4. `src/utils/plans.ts:79-106`、`src/utils/fileHistory.ts:951-957`、`src/utils/sessionEnvironment.ts:14-21`、`src/utils/debug.ts:229-249` 与 `src/utils/cleanup.ts:300-428` 又把 plans、file-history、session-env 与 debug logs 分别放进不同形态的 home-root retention worlds
5. `src/utils/diagLogs.ts:27-60` 最后再给出一条更强的异质面：diagnostics carrier 直接走 `CLAUDE_CODE_DIAGNOSTICS_FILE` 选路，repo-local 只显露 append writer，看不到对等的 repo-local cleanup constitution
6. `src/utils/cleanup.ts:575-595` 还把这些不同 family 用 hardcoded dispatcher 串起来，说明当前 visible cleanup plane 知道 family 有别，却没有公开展示一条统一的 stronger-request cleanup constitution plane

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup isolation governance` 负责回答“这次 cleanup 会不会误伤 live peer”
2. `stronger-request cleanup constitution governance` 负责回答“为什么不同 stronger-request old cleanup carriers 可以合法地活在不同 storage root、different delete gate、different ownership surface 与 different host boundary 下”

也就是说：

`artifact-family cleanup stronger-request cleanup-isolation-governor signer`

和

`artifact-family cleanup stronger-request cleanup-constitution-governor signer`

仍然不是一回事。

前者最多能说：

`这次 stronger-request cleanup 当前没有显露出会误伤 same-project live peer 的 noninterference failure。`

后者才配说：

`task outputs、transcripts、tool-results、plans、file-history、session-env、debug logs 与 diagnostics 为什么分别服从不同的 cleanup law，而且这些差异究竟是被制度化解释过的家法，还是仍未收口的历史堆叠。`

所以 `221` 之后必须继续补的一层就是：

`安全载体家族强请求清理隔离治理与强请求清理家族宪法治理分层`

也就是：

`stronger-request cleanup-isolation governor 决定“这次删会不会伤人”；stronger-request cleanup-constitution governor 才决定“这些旧 carrier 本来生活在哪种清理法律世界里”。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-constitution-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-constitution-governor signer` 仍是研究命名。
它不是在声称 repo 已经有现成的 family policy registry，
而是在说：

1. stronger-request old cleanup carriers 已明显分布在多种 storage world
2. 这些 world 的 cleanup root、delete gate、owner surface 与 host boundary 并不相同
3. 因而 stronger-request cleanup constitution 已经是一层独立可命名的问题，而不是研究者凭空追加的新名词

第二条：

这里说的 `cleanup constitution`

不是在声称：

`当前所有 stronger-request carriers 都必须被统一成同一条最强法律。`

更稳的说法只能是：

`当前源码已经足以证明 stronger-request old cleanup carriers 并不生活在同一种 cleanup world；因此“这次删会不会伤人”与“这些 carrier 本来该活在什么法律世界里”必须分开。`

第三条：

这里也不是把：

`task outputs`

当成全部 stronger-request cleanup constitution world。

真正关键的是：

1. task outputs 已被提升进 repaired session-scoped temp constitution
2. transcripts 与 tool-results 仍留在 project-session sweep constitution
3. plans、file-history、session-env 与 debug logs 又分散在 plural home-root constitutions
4. diagnostics 更直接落在 env-selected external constitution

正是这种 plural constitutions，
构成了 stronger-request cleanup constitution 的核心问题。

第四条：

这里尤其不能过度声称：

`plans`、`file-history`、`session-env` 与 `debug logs` 已经服从同一条完整 home-root policy。

当前更稳的说法只能是：

`它们都被 visible cleanup plane 归入 home-root retention world，但其中仍可见 single-directory、per-session-dir、preserve-latest-symlink 与 project-root override 等不同子宪法。`

第五条：

这里也不能过度声称：

`diagnostics carrier 完全不受治理。`

当前更稳的说法只能是：

`在 repo-local 可见范围内，diagnostics carrier 的 cleanup constitution 没有与 task outputs、projects tree 或 home-root cleanup 对等地被公开展开；它更像一条 host/env-selected external constitution。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-isolation-governor signer 仍不等于 stronger-request cleanup-constitution-governor signer`

证据：

1. task outputs 已被提升为 `project temp dir / sessionId / tasks` 的 repaired session-scoped temp constitution  
   `diskOutput.ts:31-49`
2. project temp dir 仍 intentionally cross-session readable，说明这个 constitution 的关键不是“别人看不见”，而是“谁能删、删什么”  
   `permissions/filesystem.ts:1688-1700`
3. transcripts 与 persisted tool-results 仍主要服从 `projects/<project>/<session>` 的 mtime sweep constitution  
   `sessionStorage.ts:198-220`; `toolResultStorage.ts:94-118`; `cleanup.ts:155-257`
4. plans、file-history、session-env 与 debug logs 又进入不同形态的 home-root constitutions  
   `plans.ts:79-106`; `fileHistory.ts:951-957`; `sessionEnvironment.ts:14-21`; `debug.ts:229-249`; `cleanup.ts:300-428`
5. diagnostics carrier 直接落在 env-selected external logfile constitution  
   `diagLogs.ts:27-60`
6. `cleanupOldMessageFilesInBackground()` 只是顺次 dispatch 不同 cleanup family，而不是从统一 stronger-request constitution plane 读取 policy  
   `cleanup.ts:575-595`

因此这一章的最短结论是：

`stronger-request cleanup-isolation-governor signer 最多只能说“现在看不出 live-peer 误伤”；stronger-request cleanup-constitution-governor signer 才能说“为什么不同 stronger-request old cleanup carriers 服从不同 cleanup law，以及这些 law 是否已被正式解释。”`

再压成一句：

`非干扰已追问，不等于家法已收口。`

## 4. 第一性原理：cleanup isolation 回答“这次删会不会伤人”，cleanup constitution 回答“这个载体本来生活在哪个法律世界里”

从第一性原理看，
强请求清理隔离治理与强请求清理家族宪法治理处理的是两个不同主权问题。

`stronger-request cleanup-isolation governor`

回答的是：

1. live peer 是否仍在依赖这个 carrier
2. 当前 delete 是否 consult 了 ownership / liveness / lease
3. shared readable surface 会不会被误当成 safe delete surface
4. 这次 cleanup 是否已经被证明 noninterfering
5. 当前执行会不会伤到 still-live world

`stronger-request cleanup-constitution governor`

回答的则是：

1. 这个 carrier 属于 temp world、project-session world、home-root world，还是 host-external world
2. 它的 cleanup root 为什么在那里
3. 它的 delete gate 为什么按 scoped path、mtime、symlink preservation、session-dir rm 或 env-selected path 工作
4. 它为什么要和别的 carrier family 服从不同的家法
5. 这些差异究竟是制度化设计，还是未完成收口的历史堆叠

如果把这两层压成一句“已经开始防误伤，所以 stronger-request cleanup world 已经安全且统一”，
系统就会制造五类危险幻觉：

1. noninterference-means-one-law illusion
   只要当前不误伤，就误以为所有 stronger-request carriers 已共享同一条 cleanup constitution
2. repaired-family-means-general-constitution illusion
   只要 task outputs 修过，就误以为 transcripts、tool-results、debug logs 与 diagnostics 也已同样建制
3. same-dispatcher-means-same-policy illusion
   只要 `cleanupOldMessageFilesInBackground()` 顺次调用这些 family，就误以为它们背后是同一法律
4. home-root-means-unified-constitution illusion
   只要东西都落在 `~/.claude/*`，就误以为 plan/debug/file-history/session-env 天然共享同一 delete constitution
5. env-external-means-not-our-problem illusion
   只要 diagnostics 被 env 选到外部文件，就误以为 stronger-request cleanup constitution 已自动由别处签完

所以从第一性原理看：

`stronger-request cleanup isolation governance` 管的是 local noninterference；
`stronger-request cleanup constitution governance` 管的是 family-specific legal world。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup isolation 已经等于 stronger-request cleanup constitution，为什么 task outputs 要被抬进 `project temp dir / sessionId / tasks`，而 transcripts / tool-results 却仍活在 `projects/<project>/<session>` 的 sweep world？
   因为“不会误伤 live peer”与“这个 family 该活在哪个清理世界里”不是同一个问题。
2. 如果 repo 已经有统一 stronger-request cleanup constitution，为什么 plans、file-history、session-env 与 debug logs 会继续分裂成 single-directory、per-session-dir 与 preserve-latest-symlink 三种 home-root 形态？
   因为 home-root 也不是单一世界，它内部仍有不同家法。
3. 如果 stronger-request cleanup constitution 已被完全收口，为什么 diagnostics 仍只显露 `CLAUDE_CODE_DIAGNOSTICS_FILE` writer，而没有 repo-local 对等 cleanup path？
   因为 host-external carrier 的法律边界并不自动等于 repo-local family constitution。

## 5. task-output family 先说明：stronger-request old cleanup 已有一条 repaired session-scoped temp constitution

`src/utils/task/diskOutput.ts:31-49` 很硬。

这里 repo 不只是把 task outputs 放进一个目录，
而是把 constitution 直接写进路径与注释：

1. 路径是 `getProjectTempDir()/sessionId/tasks`
2. 目的明确是避免 same-project concurrent sessions clobber each other's output files
3. 同时还把 `/clear` 之后 sessionId 变化导致 path drift 的风险写进注释

这说明 task-output family 当前服从的法律不是普通 retention sweep，
而是：

`repaired session-scoped temp constitution`

它的哲学本质是：

`先把 still-live runtime carrier 的世界做小，再让 cleanup 进入。`

而 `permissions/filesystem.ts:1688-1700` 又提醒我们，
project temp dir 仍 intentionally allows reading files from all sessions in this project。

这使 task-output constitution 更清楚：

`它不是靠“别人绝对碰不到”成立，而是靠 shared-readable world 里的 scoped write/delete discipline 成立。`

## 6. transcripts 与 persisted tool-results 再说明：stronger-request old cleanup 仍有一条 project-session sweep constitution

`src/utils/sessionStorage.ts:198-220` 与 `src/utils/toolResultStorage.ts:94-118` 说明：

1. transcript 落在 `projects/<project>/<session>.jsonl`
2. tool-results 落在 `projects/<project>/<session>/tool-results`

而 `src/utils/cleanup.ts:155-257` 说明 visible cleanup 会：

1. 先进入 `~/.claude/projects/<project>`
2. 清顶层 `.jsonl/.cast`
3. 再继续清 session dir 下的 `tool-results`
4. 整个 gate 主要仍按 cutoff date 与 mtime 工作

这说明 transcripts 与 persisted tool-results 服从的是另一条法律：

`project-session sweep constitution`

它的哲学本质不是：

`让 live runtime output 尽可能只待在当前 session temp 里`

而更接近：

`把 session persistence 做成 project-rooted history bucket，再按 retention horizon 扫除。`

所以 stronger-request cleanup constitution 在这里已经和 task outputs 彻底分叉。

## 7. plans、file-history、session-env 与 debug logs 再说明：home-root world 自己内部也不是一条法律

`src/utils/plans.ts:79-106`、`src/utils/fileHistory.ts:951-957`、`src/utils/sessionEnvironment.ts:14-21`、`src/utils/debug.ts:229-249` 与 `src/utils/cleanup.ts:300-428` 共同说明：

1. plans 默认在 `~/.claude/plans`，而且可被 `plansDirectory` 改到 project root 内
2. file-history 在 `~/.claude/file-history/<sessionId>/`
3. session-env 在 `~/.claude/session-env/<sessionId>/`
4. debug log 在 `~/.claude/debug/<sessionId>.txt`，还维护 `latest` symlink
5. cleanup 对这些 family 的动作分别是 single-directory cleanup、per-session-dir rm 与 preserve-latest-symlink age-based delete

这意味着 stronger-request old cleanup 一旦离开 project/session tree，
进入的也不是一个统一的“home-root cleanup law”，
而至少是三种不同子宪法：

1. plan single-directory constitution
2. file-history / session-env per-session-dir constitution
3. debug operational-log constitution with latest-symlink preservation

其中 `plans.ts` 最值得额外警惕。

因为这里还显露出一个制度张力：

1. storage root 可以被 `plansDirectory` 改到 project root
2. 但 visible cleanup 仍只在 `cleanupOldPlanFiles()` 里清默认 `~/.claude/plans`

这再次证明 cleanup constitution 关心的不是“当前有没有非干扰删除”，
而是：

`这个 family 的 legal world 究竟是什么，以及写入世界与清理世界有没有继续保持同一部法律。`

## 8. diagnostics 最后给出最强边界：有些 stronger-request old cleanup carrier 根本不在 repo-local cleanup constitution 内

`src/utils/diagLogs.ts:27-60` 很短，
但信息量极高。

它告诉我们：

1. diagnostics 只看 `CLAUDE_CODE_DIAGNOSTICS_FILE`
2. visible action 是 append line
3. repo-local 看不到与 `cleanup.ts` 对等的 diagnostics cleanup function

这说明 diagnostics carrier 当前不能被稳妥地归入：

1. repaired temp constitution
2. project-session sweep constitution
3. home-root retention constitution

更稳的技术判断是：

`diagnostics currently appears as a host/env-selected external constitution from the repo-local viewpoint`

这使 stronger-request cleanup constitution 的边界一下子变得更硬：

`有些 old cleanup carrier 连法律世界都不在 repo-local visible cleanup plane 里。`

## 9. `cleanupOldMessageFilesInBackground()` 让缺口更具体：repo 知道 family plurality，但现在主要是 stitched plural laws

`src/utils/cleanup.ts:575-595` 现在做的是：

1. `cleanupOldMessageFiles()`
2. `cleanupOldSessionFiles()`
3. `cleanupOldPlanFiles()`
4. `cleanupOldFileHistoryBackups()`
5. `cleanupOldSessionEnvDirs()`
6. `cleanupOldDebugLogs()`
7. image caches、pastes、stale worktrees 等

这条 dispatcher 非常值钱。

因为它说明 repo 并不是完全不知道 family 有别。
相反，它非常知道。

问题恰恰在于：

`visible cleanup plane knows family plurality operationally, but does not yet expose an equally explicit stronger-request constitution plane that explains why these laws differ.`

换句话说，
当前显露出来的是：

`plural cleanup executors`

但还不是：

`fully signed plural cleanup constitutions`

这也是这一章最重要的技术启示之一：

`真正先进的安全设计，不是把一切都硬压进单一清理器，而是先承认法律世界本来就多元，再明确解释每个载体为什么属于那条法律。`

## 10. 安全设计的哲学本质：安全不是“所有东西都交给同一把扫帚”，而是“每种载体都知道自己受哪部法律约束”

如果把这一章再往哲学本质压一次，
可以得到一句更硬的话：

`isolation 防的是误伤，constitution 防的是误统。`

误伤的危险是：

`删错对象`

误统的危险则是：

`把本来生活在不同法律世界里的载体，偷写成同一种 cleanup object。`

从安全哲学看，
后一种危险并不比前一种小。

因为一旦误统，
系统就会：

1. 错配 delete gate
2. 错配 ownership model
3. 错配 host boundary
4. 错配 storage rationale
5. 最终把看似统一的政策施加到不该被同法处理的 carrier 上

所以这一章真正钉死的是一条更深的原则：

`一个成熟的安全系统不只要知道什么时候不能删错人，还要知道不同载体本来就不应被同一部法律粗暴处理。`

## 11. 从技术角度看 Claude Code 这层设计的先进性与启示

如果把 Claude Code 当前这层设计抽成技术启示，
至少有五条：

1. 它没有把所有旧 carrier 强行压扁成一个 storage law，而是允许 family-sensitive storage constitution 先存在
2. 它知道 high-risk live runtime output 需要更强的 session-scoped temp repair，而不是继续塞回 project-session sweep
3. 它把 shared-readable surface 直接写明，这比靠隐含约定更成熟，因为它逼迫 delete boundary 被单独思考
4. 它让 dispatcher operationally 知道 family plurality，这为未来把 plural laws 提升成显式 constitution plane 留下了接缝
5. 它把 diagnostics 留在 env-selected external path，这虽然带来 repo-local constitution gap，但也证明系统承认 host boundary 本来就可能跨出 repo

这些启示共同说明：

`Claude Code 的先进性不只是“有好多安全机制”，而是它已经在若干关键点上承认并暴露了多重法律世界，而不是假装一个清理器能解释一切。`

## 12. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要不误伤 live peer，就已经 constitutionally complete。`

而是：

`repaired temp world、project-session sweep world、home-root plural worlds 与 host-external diagnostics world 已经共同展示出 stronger-request old cleanup carriers 的 constitutional plurality；因此 artifact-family cleanup stronger-request cleanup-isolation-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-constitution-governor signer。`

因此：

`cleanup 线真正缺的，不只是“谁来证明 stronger-request cleanup 没伤到 live peer”，还包括“谁来正式解释这些 old cleanup carriers 为什么分别生活在不同 cleanup law 里”。`
