# 安全载体家族强请求清理复活治理与强请求清理再赋权治理分层：为什么artifact-family cleanup stronger-request cleanup-resurrection-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer

## 1. 为什么在 `327` 之后还必须继续写 `328`

`327-安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪些旧对象可以重新回到 current world，也还需要 resurrection governor 去决定它们是否能回来、凭什么回来、回来时进入哪一层 world。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“对象回来就说明它原来的资格也一起回来了”
的错觉：

`只要 stronger-request cleanup resurrection governor 已经让旧对象重新出现，它就自动拥有了决定旧 identity、旧配置、旧 secrets、旧 scope 与旧可用资格是否一起恢复的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 re-entitlement 正对照，
会发现它其实明确拆开了五件事：

1. `object return`
2. `enabled truth`
3. `config / secrets trust`
4. `policy veto`
5. `identity seat`

最硬的证据至少有七组：

1. `pluginOptionsStorage.ts` 证明 `deletePluginOptions()` 只在 last installation 卸载时才抹掉旧 `pluginConfigs` 与旧 `pluginSecrets`，说明 comeback 不自动带回旧配置与旧 secrets  
   `src/utils/plugins/pluginOptionsStorage.ts:201-208,210-272`
2. `pluginOperations.ts:528-540` 进一步证明 last-scope remove 与 options/secrets wipe 被显式联动，说明对象离场时 entitlement material 会被单独清算
3. `setPluginEnabledOp()` 把 enable / disable 写成单独的 settings action，说明对象回来不等于自动恢复 enabled truth  
   `src/services/plugins/pluginOperations.ts:573-727`
4. `pluginPolicy.ts` 与 `pluginOperations.ts:650-657` 证明更高层 policy 仍可 veto 旧资格恢复
5. `pluginOperations.ts:664-687` 证明 even after return，scope / precedence 仍要经过 override legality 判断，不是对象一回来就自动恢复原 seat
6. `installedPluginsManager.ts:827-830,859-861` 证明 disk / materialized existence 与 effective installed / enabled truth 仍是两层不同真相
7. `plans.ts:233-257` 与 `REPL.tsx:1790-1797` 证明内容回来时仍可能只拿到 `newSlug`，而不是恢复 original identity

这说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

和

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象可以重新出现。`

后者才配说：

`它重新出现之后，算不算原来那个被授权、被配置、被允许继续工作的对象。`

所以 `327` 之后必须继续补的一层就是：

`安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`

也就是：

`stronger-request cleanup resurrection governor 负责让对象回来；stronger-request cleanup re-entitlement governor 才负责决定它回来之后恢复哪些旧资格。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 re-entitlement manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出对象回来不等于资格回来
2. `回来` 与 `恢复旧资格` 是两个不同问题
3. stronger-request cleanup 线如果真要把 resurrection world 做完整，也迟早要回答这些“回来后还算不算原来那个对象”的问题

第二条：

这里说的 `re-entitlement governance`

不是在声称：

`对象回来后必须恢复所有旧资格。`

当前更稳的说法只能是：

`re-entitlement governance 首先回答的是哪些资格可以回来、哪些必须重新授予、哪些根本不该回来。`

第三条：

这里也不是把：

`enable state`

单独当成全部 re-entitlement grammar。

真正关键的是：

1. 配置 / secret 会不会回来
2. enabled truth 会不会回来
3. policy 是否允许回来
4. identity / scope 是否还是原来的那个 seat

第四条：

这里尤其不能过度声称：

`deletePluginOptions()`

就等于完整资格治理。

更稳的说法只能是：

`资格治理` 还要继续回答 post-return enable、policy、scope 与 identity fallback；option/secrets wipe 只是其中一层。`

第五条：

这里也不能过度声称：

`new identity` 一定比 `old identity` 弱。

当前更稳的说法只能是：

`new slug / new seat` 的关键不在强弱，而在系统拒绝把“内容回来”偷写成“原资格原封不动回来”。`

第六条：

这里还不能过度声称：

`re-entitlement` 只是恢复体验之外的附带管理。

更稳的说法只能是：

`它本质上是在治理 post-return qualification grammar；体验只描述回来，资格才决定回来之后还能做什么。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-resurrection-governor signer 仍不等于 stronger-request cleanup-re-entitlement-governor signer`

证据：

1. `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`，说明对象回来不自动带回旧配置与旧 secrets  
   `pluginOptionsStorage.ts:201-208,210-272`
2. `setPluginEnabledOp()` 把 enable / disable 做成单独的 settings write，说明对象可 install、materialize、refresh，但是否恢复 enabled truth 仍要经过独立写入  
   `pluginOperations.ts:573-727`
3. `pluginPolicy.ts` 与 `pluginOperations.ts` 明确写出 org policy 可以继续 veto install / enable  
   `pluginPolicy.ts:11-19`; `pluginOperations.ts:650-657`
4. `pluginOperations.ts` 继续要求 cross-scope override legality，说明 current seat 与 old seat 不是自然延续关系  
   `pluginOperations.ts:664-687`
5. `installedPluginsManager.ts` 直接证明 materialized existence 与 effective installed / enabled entitlement 不是同一层 truth  
   `installedPluginsManager.ts:827-830,859-861`
6. `copyPlanForFork()` 与 `REPL.tsx` 证明内容回来时仍可能只获得 new identity，而不是 old identity  
   `plans.ts:233-257`; `REPL.tsx:1790-1797`
7. `deletePluginOptions()` 的 best-effort 失败不回滚 uninstall，又说明对象离场事实与资格清算成败也被继续分层

因此这一章的最短结论是：

`stronger-request cleanup resurrection governor 最多能说对象能回来；stronger-request cleanup re-entitlement governor 才能说它回来之后恢复什么旧资格。`

再压成一句：

`会回来，不等于还是原来那个被授权的它。`

再压成更哲学一点的话：

`制度知道怎样让对象重新出现，不等于制度已经知道该把哪些旧权利重新还给它。`

## 4. 第一性原理：resurrection governance 回答“能否回来”，re-entitlement governance 回答“回来后恢复哪些资格”

从第一性原理看，
强请求清理复活治理与强请求清理再赋权治理处理的是两个不同主权问题。

`stronger-request cleanup resurrection governor`
回答的是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回到的是哪一层 world
4. 是否沿原 lineage 回来
5. 是否应该避免 clobber

`stronger-request cleanup re-entitlement governor`
回答的则是：

1. 对象回来后是否恢复旧 enabled state
2. 是否恢复旧 secrets
3. 是否恢复旧 pluginConfigs / options
4. 是否恢复旧 scope / precedence 位置
5. 是否恢复旧 identity，还是只给它一个新的 current seat

如果把这两层压成一句：

`反正它已经回来了`

系统就会制造五类危险幻觉：

1. `return-means-enabled`
2. `return-means-configured`
3. `return-means-same-scope`
4. `return-means-same-identity`
5. `existence-means-entitlement`

所以这两层的哲学本质也不同：

`resurrection`

治理的是：

`对象能不能重新进入当前世界。`

`re-entitlement`

治理的是：

`它重新进入之后，还配不配继续拿着原来的权利说话。`

Claude Code 在这里真正先进的地方，
不是它把“回来”和“可继续用”压成一个词，
而是它已经明确承认：

`重新出现，不等于重新被授权。`

这是一种很成熟的安全哲学：

`current-world presence` 不是 `current-world qualification`。

## 5. 从技术角度看，Claude Code 的先进性在于把 post-return qualification 拆成 config/secrets、enabled truth、policy veto、scope legality 与 identity fallback，而不是把“回来”压成资格总包

### 5.1 config/secrets wipe：对象回来并不自动恢复旧配置、旧 secrets 与旧信任材料

`pluginOptionsStorage.ts:201-208`
先把 `deletePluginOptions()` 的制度边界写死：

1. 只在 LAST installation 被移除时调用
2. 不是每次 uninstall 都删
3. 因为 plugin 可能仍在其他 scope 存活
4. 即便 keychain cleanup 失败，也不把 uninstall itself 回滚成失败

继续看
`pluginOptionsStorage.ts:210-272`：

1. settings side 会删 `pluginConfigs[pluginId]`
2. secure storage side 会删 `pluginSecrets[pluginId]`
3. 还会删 `${pluginId}/${server}` 这类 per-server composite keys
4. 最后 clear options cache

再看
`pluginOperations.ts:528-540`
last-scope remove 时会显式联动：

1. `deletePluginOptions(pluginId)`
2. `deletePluginDataDir(pluginId)`

这说明 plugin comeback 之后，
系统不会自动假定：

`旧配置还有效`

或

`旧 secrets 还应继续被信任`

### 5.2 enabled truth：对象回来不等于自动恢复 enabled entitlement

`setPluginEnabledOp()`
把 enable / disable 写成完整独立动作：

1. 先 resolve pluginId 与 scope
2. 再过 policy guard
3. 再过 cross-scope legality
4. 再判断当前 state / idempotency
5. 最后才写 `enabledPlugins`

这说明：

`对象存在`

和

`对象被重新赋予 enabled truth`

是两层不同动作。

### 5.3 policy veto 与 scope legality：回来不等于重新被允许，也不等于重新占回旧位置

`pluginPolicy.ts:11-19`
把 policy guard 写得很直白：

1. `enabledPlugins[pluginId] === false`
2. 即视为 blocked
3. install chokepoint、enable op 与 UI filters 共用这份 truth

`pluginOperations.ts:650-657`
又在 enable 前直接挡住：

`Plugin "...\" is blocked by your organization's policy and cannot be enabled`

而
`pluginOperations.ts:664-687`
在 explicit scope given but plugin is elsewhere 时，
会继续做 cross-scope legality 判断：

1. 若 absent from requested scope but present at another scope
2. 且这次不是合法 higher-precedence override
3. 则直接返回 wrong-scope guidance

这说明：

`能回来`

和

`回来后还被允许、还占原来的 seat`

不是同一问题。

### 5.4 divergence guard：materialized existence 与 effective entitlement 不是同一层 truth

`installedPluginsManager.ts:827-830,859-861`
直接写出：

1. plugins are loaded from `settings.enabledPlugins`
2. 如果 `settings.enabledPlugins` 与 `installed_plugins.json` diverge
3. 就 treat as not-installed so the user can re-enable

这等于直接证明：

`disk / materialized existence`

和

`effective installed / enabled entitlement`

不是同一层 truth。

### 5.5 identity fallback：内容回来时仍可能只拿到 new identity，而不是原资格位

`plans.ts:233-257`
的 `copyPlanForFork()` 很清楚：

1. 复制 original content
2. 生成 `newSlug`
3. 写到 `newPlanPath`
4. 避免 original 与 forked sessions clobber

`REPL.tsx:1790-1797`
又把 fork / resume 两条路明确拆开：

1. fork path 调 `copyPlanForFork()`
2. regular resume path 才调 `copyPlanForResume()`

这直接证明：

`内容回来`

不等于

`原身份回来`

## 6. 对 stronger-request cleanup 的技术启示：真正缺的不是 comeback，本质上缺的是 qualification-return grammar

对 stronger-request cleanup 来说，
上一层 `resurrection governance` 主要还在回答：

1. 哪些带碑的旧对象还能回来
2. 回来靠什么 evidence
3. 回来后进入哪一层 active world

但这一层已经必须继续追问：

1. 回来之后它还算不算原来那个被授权的对象
2. 旧 stronger promise 是否继续有效
3. 旧解释席位、旧配置席位、旧 receipt 席位是否原样恢复
4. 哪些资格必须被重新授予，哪些根本不该回来

这些都不是“再多做一点 re-entry”能回答的。

它们已经进入：

`re-entitlement governance`

层。

## 7. 苏格拉底式自反诘问：我是不是又把“对象回来了”误认成了“它原来的全部资格也回来了”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 resurrection grammar 已经足够强，为什么还要再拆 re-entitlement？  
   因为对象能回来，不等于它原来的资格也一起回来。
2. 如果 disk/materialized world 已经恢复，是不是就说明 enabled truth 也恢复了？  
   不是。enabled truth 仍要经过独立 settings write。
3. 如果 plan 内容能恢复，是不是就说明 old identity 也恢复了？  
   不是。`copyPlanForFork()` 明确证明 content return 与 identity return 可以分开。
4. 如果 policy 没有阻止 comeback，是不是就说明它自动允许 old entitlement 恢复？  
   也不对。policy veto 只是更高层拒绝条件，不等于 entitlement 已被重新签发。
5. 如果旧 config/secrets 还没被完全删干净，是不是就说明它们仍然可信？  
   不能这样推。成熟系统恰恰会把 trust material 的回归单独治理。
6. 如果 cleanup 线现在还没有显式再赋权代码，是不是说明这层还不值得写？  
   恰恰相反。越是缺这层明确 grammar，越容易把“对象回来”偷写成“旧资格整包归位”。

这一串反问最终逼出一句更稳的判断：

`re-entitlement 的关键，不在对象会不会回来，而在系统能不能正式决定回来之后哪些旧权利还配继续属于它。`

## 8. 从安全设计哲学看，这一层真正回答的是“制度不仅会让对象回来，而且配重新签发它的资格”

`resurrection governance`
更像制度会不会让对象重新出现。

`re-entitlement governance`
更像制度配不配重新签发资格。

一个系统可以决定：

1. 对象能否回来
2. 回来靠什么证据
3. 回来进入哪一层 current world

但如果它没有继续决定：

1. 旧 enabled truth 是否恢复
2. 旧 config / secrets 是否恢复
3. 旧 identity / scope / seat 是否恢复
4. 哪些资格必须被重新授予

那么它仍然更接近：

`会让对象回来的制度`

而不是：

`会重新授予对象资格的制度`

Claude Code 当前 stronger-request cleanup 线，
恰好停在这个哲学分界点上：

1. 它已经不再只是“没决定谁能回来”
2. 它已经开始能把 re-entry authority 单独治理
3. 但它还没有 equally-strong 地回答回来后哪些旧资格还算它的

所以这一章真正要钉死的，
不仅是一个工程判断，
也是一个更深的安全设计判断：

`制度知道怎样让对象重新出现，不等于制度已经知道该把哪些旧权利重新还给它。`

## 9. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 resurrection governance，就已经足够成熟。`

而是：

`Claude Code 在 old config/secrets wipe、explicit enabled write、policy veto、divergence guard 与 new-slug identity fallback 上已经明确展示了 re-entitlement governance 的存在；因此 artifact-family cleanup stronger-request cleanup-resurrection-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“怎么让旧对象回来”，而是“谁来决定回来之后它还算不算原来那个被授权的对象”。`
