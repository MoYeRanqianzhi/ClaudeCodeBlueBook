# MCP、WebFetch、hooks 与外部能力收口：外部世界不是默认可信上下文

## 1. 先说结论

Claude Code 对外部能力最成熟的判断是：

`外部世界不是默认可信上下文。`

因此无论是：

- MCP
- WebFetch
- hooks
- gateway / proxy

系统都不是简单“连上就用”，  
而是分别解决三件事：

1. 外部输入能不能进来
2. 进来多少
3. 谁有资格定义这条外部入口

## 2. 为什么 MCP 不是普通插件机制

很多系统把 MCP 看成“多接几个 server”。  
但从 `mcpValidation.ts`、`QueryEngine.ts`、`toolSearch.ts` 这些入口看，Claude Code 更像把它当作：

`可把外部世界注入上下文和工具面的高权限扩展接口。`

因此它最关心的不是“能否接入”，  
而是：

- 输出体积
- token 开销
- 上下文污染
- 兼容语义

## 3. MCP 收口的第一个重点：输出预算

`mcpValidation.ts` 最直接的安全含义，不是节省 token，  
而是：

- 限制外部工具对上下文的注入规模
- 限制外部结果对主会话的资源挤压
- 在截断时显式告诉模型“输出不完整”

这很重要，因为如果只做 silent truncation，  
模型会在错误前提下继续推理。

Claude Code 这里采用的是更成熟的做法：

- 截断
- 说明已截断
- 建议分页或过滤

这不只是性能设计，  
也是：

`外部输入完整性语义的安全设计。`

## 4. MCP 收口的第二个重点：beta 语义与代理边界

`toolSearch.ts` 明确指出：

- `tool_reference` 是 beta content type
- 很多 `ANTHROPIC_BASE_URL` 代理通常不支持
- 在非第一方 host 时会更保守地关闭默认 optimistic 路径

这说明 Claude Code 很清楚一个现实：

`外部入口接得上，不等于扩展协议语义也完整。`

从安全角度看，这一点很关键。  
因为协议能力不完整时，系统若仍假装完整，会带来：

- 误判
- 假成功
- 错误的权限期望

它宁愿更保守，也不愿把不完整能力包装成完整能力。

## 5. WebFetch 收口为什么体现了很强的安全成熟度

`WebFetchTool/preapproved.ts` 是一个很容易被低估的文件。

它明确区分了两件事：

1. WebFetch 的预批准域名
2. sandbox 的网络权限

而且注释把理由说得很清楚：

- 某些代码相关站点也允许上传
- GET 可读，不代表任意网络写入、上传、POST 就安全

这条分离极其重要。  
因为很多系统最容易犯的错误正是：

`把“允许访问站点”误等于“允许对站点做任何网络动作”。`

Claude Code 没有这样做。

## 6. hooks 收口为什么本质上是在处理主权问题

`hooksConfigSnapshot.ts` 揭示出 hooks 的真正问题，不是“脚本方便不方便”，  
而是：

`到底谁有资格在会话里插入额外行为。`

源码支持几种非常清晰的治理：

- managed settings 可要求只允许 managed hooks
- managed settings 可彻底禁用 hooks
- 非 managed settings 不能反向禁掉 managed hooks

这说明 hooks 在 Claude Code 里不是随手拼接的小扩展，  
而是：

- 用户主权
- 项目主权
- 管理员主权

三者之间的冲突面。

## 7. file watcher / snapshot 细节为什么也属于安全设计

`hooksConfigSnapshot.ts` 与 `setup.ts` 的组合表明：

- 启动时会捕获 hooks 配置快照
- 后续更新需要显式刷新缓存

其目的不是“让设置更快生效”这么简单，  
更深一层是：

`避免隐藏 hook 修改在用户未察觉时悄悄改变会话行为。`

这说明 Claude Code 甚至把“配置的可观察性”也纳入了安全设计。

## 8. gateway / proxy 为什么也属于外部能力收口的一部分

从 `apiPreconnect.ts`、`providers.ts`、`toolSearch.ts` 与前面安全专题的认证章节连起来看，可以得到一个更准确的结论：

- gateway / proxy 不是纯网络细节
- 它也是外部能力入口

因为一旦改写：

- 上游 host
- 支持的 beta 语义
- 认证方式

整个系统的能力面和安全语义都会跟着变化。

这也是为什么安全专题必须把 gateway 放进同一章。  
它与 MCP、WebFetch、hooks 一样，本质上都在回答：

`外部世界通过什么边界进入会话。`

## 9. 从技术角度看，这部分先进性在哪里

Claude Code 在外部能力收口上的先进性，至少体现在四点：

1. 把不同外部入口视为不同攻击面，而不是统一按“扩展”处理
2. 不仅限制是否接入，还限制输出体积与语义完整性
3. 明确区分读取语义和写入语义
4. 把 hooks / gateway 的问题上升到主权和治理，而不是只看 DX

## 10. 从第一性原理看，这部分设计究竟在防什么

如果从第一性原理压缩，外部能力安全设计真正防的是四类失控：

1. 外部内容无限注入上下文
2. 外部域名被错误等价成任意网络写权限
3. 隐形 hooks 改变运行时行为
4. 不完整代理语义被误写成完整官方语义

Claude Code 的答案不是一把大锁，  
而是：

`按入口类型分别收口。`

## 11. 给平台构建者的技术启示

如果你也在做 agent 平台，这一章最值得借鉴的是：

1. 扩展能力应按攻击面拆开治理，不要只做统一插件系统
2. 外部输出预算本身就是安全控制，不只是性能优化
3. 预批准读域名不应自动继承为网络写权限
4. hook 治理的核心不是脚本执行，而是谁拥有插入点主权

## 12. 一句话总结

Claude Code 对 MCP、WebFetch、hooks 和 gateway 的安全处理之所以成熟，不在于它一概拒绝外部能力，而在于它清楚外部世界不是默认可信上下文，因此每一种入口都要按自己的注入面、语义面和主权面分别收口。
