# 安全载体家族强请求清理重新并入治理与强请求清理重新投影治理分层：为什么artifact-family cleanup stronger-request cleanup-reintegration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reprojection-governor signer

## 1. 为什么在 `429` 之后还必须继续写 `430`

`429-安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`
已经回答了：

`stronger-request cleanup 线即便已经拿到 recovered truth，也还要单独回答这份 truth 何时重新并入当前 tool / status / event world。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然对象已经并回当前世界，那系统当然知道该怎样把它讲给所有读者”
的错觉：

`只要 stronger-request cleanup reintegration governor 已经让对象重新进入 current world，它就自动拥有了决定这份 current truth 该怎样被 control protocol、status pane、notification surface、health surface 与 reconnect dialog 重新讲述的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的 reprojection 正对照，
会发现 repo 其实明确拆开了七件事：

1. `structured control projection`
2. `manager status grammar`
3. `panel status grammar`
4. `selective notification policy`
5. `health glyph projection`
6. `action-oriented reconnect copy`
7. `control envelope payload discipline`

最硬的证据至少有七组：

1. `src/cli/print.ts:1610-1698,2957-2960,3127-3128`
   的 `buildMcpServerStatuses()` 与 `mcp_status` / `reload_plugins` control responses 会把同一份 current truth 重新编码成结构化 payload
2. `src/commands/plugin/ManagePlugins.tsx:512-519`
   的 `getMcpStatus()` 会把同一 truth 压成 manager-facing enum
3. `src/components/mcp/MCPListPanel.tsx:307-337`
   的 `statusText` 又把同一 truth 压成 panel-facing label 与 reconnect wording
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
   只为 `failed / needs-auth` 发通知，不对称发布 `connected / pending`
5. `src/cli/handlers/mcp.tsx:26-35`
   的 `/mcp` health surface 只给 glyph-level summary
6. `src/components/mcp/MCPReconnect.tsx:40-63`
   把同一 truth 压成 operation-local success/auth/failure copy
7. `src/cli/print.ts:2736-2758,3148-3204,3258-3295`
   又说明 control plane 会对同一 truth 继续裁成 success-only、error-only 或 success+payload 三种 envelope

这说明：

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reprojection-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经重新成为当前世界的一部分。`

后者才配说：

`这份当前真相现在该怎样被 control response、health、notification、panel status 与 reconnect copy 重新讲述给不同读者。`

所以 `429` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新并入治理与强请求清理重新投影治理分层`

也就是：

`stronger-request cleanup reintegration governor 决定对象何时重新进入当前世界；stronger-request cleanup reprojection governor 才决定进入之后这份真相怎样被不同 surface 重新讲述。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reprojection-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reprojection manager，
而是在说：

1. repo 已经把 reintegrated truth 重新讲述给多个 surface
2. 这些 surface 使用不同的枚举、文案、选择性提示与结构化 payload
3. repo 已经拒绝让某一份投影自动代表所有 reader 的 current truth

第二条：

这里说的 `reprojection governance`
不是在声称：

`每个读者都应该看到同样多、同样强的 current truth。`

当前更稳的说法只能是：

`reprojection governance 首先回答的是不同 reader 应看到哪种粒度、哪种词法强度、哪种 publication policy 的 current truth。`

第三条：

这里也不是把：

`buildMcpServerStatuses()`

单独当成全部 reprojection grammar。

真正关键的是：

1. structured control projection
2. manager/panel status grammar
3. selective notification policy
4. health glyph projection
5. action-oriented reconnect copy
6. control envelope payload discipline

第四条：

这里尤其不能过度声称：

`reprojection governance`

已经等于：

`reassurance governance`

当前更稳的说法只能是：

`reprojection governance 回答 current truth 怎样被重讲；reassurance governance 才回答这些重讲里哪些现在配承载多强的继续依赖担保。`

第五条：

这里也不能过度声称：

`某一份 status text`

就天然代表：

`所有 consumer 的 current truth`

当前更稳的说法只能是：

`同一份 reintegrated truth 在 control、health、notification、dialog 与 panel world 里本来就需要不同的 reader-specific projection。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-reintegration-governor signer 仍不等于 stronger-request cleanup-reprojection-governor signer`

证据：

1. `buildMcpServerStatuses()` 会把同一份 current truth 重新编码成 control protocol 用的结构化 `McpServerStatus[]`
2. `getMcpStatus()` 与 `MCPListPanel` 又把同一 truth 压成 plugin manager / panel status enum 与 label
3. `useMcpConnectivityStatus()` 只为 `failed / needs-auth` 发 toast，不为 `connected / pending` 发“恢复成功”通知；这说明 reprojection 有选择性 publication grammar
4. `handlers/mcp.tsx` 把同一 truth 压成 `✓ Connected / ! Needs authentication / ✗ Failed to connect`；这说明 health surface 的词法主权独立于 reintegration
5. `MCPReconnect` 把同一 truth 压成 `Successfully reconnected`、`requires authentication` 与 generic failure copy；这说明 dialog/reconnect surface 的叙事强度也不是 reintegration 本身能回答的
6. control response envelope 会继续裁决读者拿到 success-only、error-only 还是 success+payload；这说明 payload strength 也是单独的 reprojection gate
7. `getMcpStatus()`、panel label、health glyph、toast 与 dialog copy 彼此并不互为真值源，而是同一 truth 的不同 reader retelling

因此这一章的最短结论是：

`stronger-request cleanup reintegration governor 最多能说对象已经重新回到当前世界；stronger-request cleanup reprojection governor 才能说这份 current truth 现在应当怎样被不同 reader surface 重新讲述。`

再压成一句：

`reintegrated，不等于已经被所有读者按正确方式重新看见。`

再压成更硬的一句：

`会回席，不等于会改口。`

再压成更哲学一点的话：

`制度知道它已经并回当前世界，不等于制度已经知道该让谁看到哪一版当前真相。`

## 4. 第一性原理：reintegration governance 回答“对象回到当前世界了吗”，reprojection governance 回答“这份当前真相该怎样被重新讲述”

从第一性原理看，
强请求清理重新并入治理与强请求清理重新投影治理处理的是两个不同主权问题。

`stronger-request cleanup reintegration governor`
回答的是：

1. recovered client 是否已进入 current state
2. recovered tools / commands / resources 是否已回到 current tool pool
3. event plane 是否重新接回
4. next turn consumer 是否已经能读到它
5. 对象是否重新成为 current world 的一部分

`stronger-request cleanup reprojection governor`
回答的则是：

1. 哪些 reader 该看到结构化 status
2. 哪些 reader 该看到 glyph / label / copy
3. 哪些 reader 只该看到 failure/auth warning，而不该看到 success toast
4. 同一 truth 在不同 surface 上该被压成怎样的词法强度
5. 哪些 surface 该重讲 current truth，哪些 surface 该保持沉默

如果把这两层压成一句：

`反正已经并回来了`

系统就会制造六类危险幻觉：

1. `integrated-means-announced`
2. `one-surface-speaks-for-all`
3. `success-toast-is-mandatory`
4. `health-string-equals-control-truth`
5. `reconnect-copy-equals-status-plane`
6. `payload-shape-is-incidental`

所以从第一性原理看：

`reintegration governance`
管的是 world membership；

`reprojection governance`
管的是 reader-facing retelling。

如果再压缩一点，
两者的本质差别其实就是：

1. `reintegration` 负责 `truth has rejoined current world`
2. `reprojection` 负责 `truth is retold per reader`

一个 object 即便已经并回 current world，
也完全不等于所有读者都已经拿到正确粒度、正确词法和正确发布策略的 current truth。

## 5. Claude Code 多重安全技术的先进性：它把“当前真相的传播”绑定到 reader-specific grammar，而不是绑定到单一通用文案

Claude Code 在这一层真正先进的地方，
不是“让所有 surface 共享一句成功或失败文案”，
而是：

`它把 current truth 的对外传播继续拆成 control projection、UI status grammar、notification publication policy、health glyph、dialog copy 与 payload envelope 六种 reader-specific grammar。`

这种先进性至少体现在六个方面：

1. `control truth is structured, not improvised`
   control reader 拿到的是重组后的 `McpServerStatus[]`，不是某个 UI 标签的随手复用。
2. `panel truth is compressed deliberately`
   `getMcpStatus()` 与 `MCPListPanel` 明确承认 manager reader 与 panel reader 需要不同压缩级别。
3. `notification truth is selective by design`
   `useMcpConnectivityStatus()` 用沉默来防止过度广播正向恢复叙事。
4. `health truth is narrow on purpose`
   health reader 只拿 glyph-level summary，不拿完整 control payload。
5. `dialog truth is action-oriented`
   `MCPReconnect` 给 operator 的是本次 repair path 的局部叙事，而不是全局 current truth 的完整镜像。
6. `payload strength is governed`
   `sendControlResponseSuccess()` / `sendControlResponseError()` 说明“给多少 truth”本身就是制度问题。

这些设计背后的源码思路也很清楚：

1. 不让 `reintegrated` 越级冒充 `correctly reprojected`
2. 不让某个局部 reader 抢夺全部 reader 的解释权
3. 不让 success publication 成为默认强制动作
4. 不让 health glyph 冒充 full control truth
5. 不让 dialog 文案冒充系统级 current narrative

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. current-world membership 之外，还必须单列 reader-facing reprojection grammar。
2. 不同 reader 应有不同字段集、不同词法强度、不同 publication policy。
3. success 不一定值得主动广播，沉默本身可以是更诚实的投影策略。
4. payload shape 不能被视为 incidental formatting，它本身就是控制面设计。
5. health、panel、dialog、control 这些 surface 不能共享一个万能 truth sentence。
6. 同一份 current truth 必须允许多种 honest projection，而不是被单一 surface 垄断。 

## 6. 苏格拉底式自反诘问：我是不是又把“它已经回到当前世界”误认成了“所有读者已经按正确方式看见它”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 reintegration governance 已经等于 reprojection governance，为什么 `buildMcpServerStatuses()` 还要重建一套 structured status，而不是直接把 raw state 送给 control reader？
   因为“对象已经回到当前世界”与“control reader 该怎么看它”根本不是同一句制度。
2. 如果同一份 current truth 天然适合所有读者，为什么 `useMcpConnectivityStatus()` 只发布 `failed / needs-auth`，却不对称发布 `connected / pending`？
   因为 publication policy 也是单独的 reader-facing governance。
3. 如果 panel status 已经足够，为什么 health surface 还要单独使用 glyph grammar？
   因为不同 reader 需要的不是同一份语言复制，而是 reader-specific projection honesty。
4. 如果 reconnect dialog 的 success/failure copy 就已经代表全部 current truth，为什么 control response 还要有 structured payload 与 success/error envelope？
   因为 operation-local copy 不是系统级 current narrative。
5. 如果 payload 只是格式问题，为什么 control plane 要反复裁决 success-only、error-only 与 success+payload？
   因为给多少 truth、本次给哪版 truth，本来就是强主权问题。
6. 如果 cleanup 线现在还没把 reprojection 单列出来，是不是说明这层只属于 UI 修辞？
   恰恰相反。越是把它当修辞，越容易让弱 surface 偷签强 current truth。

这一串反问最终逼出一句更稳的判断：

`reprojection 的关键，不在对象是否已经回到当前世界，而在系统能不能正式决定不同读者现在该被怎样告知这份 current truth。`

如果再追问“怎样做得更好”，
更好的方法也不是只增加更多 UI 文案，
而是继续把四个问题拆开：

1. 谁决定哪类读者拿 structured payload
2. 谁决定哪类读者只拿 compressed label
3. 谁决定哪类 truth 应被主动发布
4. 谁决定哪类 success 更诚实的动作其实是保持沉默

只有把这四个问题分开，
`reintegration` 才不会再次偷走 `reprojection` 的主权。

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 reintegration governance，就已经足够成熟。`

而是：

`Claude Code 在 buildMcpServerStatuses() 的 structured status builder、getMcpStatus() / MCPListPanel 的 status grammar、useMcpConnectivityStatus() 的 selective notification policy、handlers/mcp.tsx 的 health glyph grammar、MCPReconnect 的 action-oriented copy，以及 control response envelope 的 success/error payload discipline 上，已经明确展示了 reprojection governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reintegration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reprojection-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“它回到当前世界了吗”，而是“不同读者现在该怎样被告知这份 current truth”。`
