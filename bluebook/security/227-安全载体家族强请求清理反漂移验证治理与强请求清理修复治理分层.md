# 安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层：为什么artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer不能越级冒充artifact-family cleanup stronger-request cleanup-repair-governor signer

## 1. 为什么在 `226` 之后还必须继续写 `227`

`226-安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层` 已经回答了：

`stronger-request cleanup 线即便某次运行看起来已经 conform，也还不等于系统已经具备长期 anti-drift verification；repo 在别处已经明确展示了 test-based、structure-based 与 live re-verification 三种 verifier grammar，而 cleanup 线当前还没接上。`

但如果继续往下追问，
还会碰到下一层更容易被误写成同一动作的错觉：

`只要 verifier 已经抓到 stronger-request cleanup drift，它就自然拥有了修复主权。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看 repo 自己最成熟的 repair 正对照，
会发现它其实已经把：

`truth exposure`

和

`governance consequence`

拆成了不同层。

最硬的证据有五组：

1. `src/utils/permissions/permissionSetup.ts:1078-1157`
   的 `verifyAutoModeGateAccess()` 负责 fresh read dynamic config、识别 stale-vs-live divergence、产出 `updateContext` 与 `notification`
2. 但真正把 verifier 结果施加到 CURRENT app state、并把 warning 推进 notification queue 的，
   是 `src/utils/permissions/bypassPermissionsKillswitch.ts:74-116`
   的 `checkAndDisableAutoModeIfNeeded()`
3. `src/utils/permissions/getNextPermissionMode.ts:11-17`
   也只负责在 local mode cycling 前暴露 divergence 风险，
   并不拥有 circuit breaker 或 consequence application 主权
4. plugin 线里
   `src/utils/plugins/dependencyResolver.ts:161-233`
   的 `verifyAndDemote(...)`
   与
   `src/utils/plugins/pluginLoader.ts:3189-3195`
   的 session-local demotion
   更直接展示了 repo 的成熟 grammar：
   drift 被抓住后，
   系统会明确决定 capability consequence，
   而且注释还特意写明
   `does NOT write settings`
5. 与这些正对照相比，
   stronger-request cleanup 线当前虽然已经暴露出 temporal drift、propagation drift 与 receipt drift，
   却还没有谁正式决定：
   是修 scheduler、修 executor、补 receipt、降级 wording，还是把某条 promise 临时降到 session-local safe mode

这说明：

`artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer`

和

`artifact-family cleanup stronger-request cleanup-repair-governor signer`

仍然不是一回事。

前者最多能说：

`这里已经 drift 了，或者当前还没有足够证据证明它没有 drift。`

后者才配说：

`面对这个 drift，应该修哪一层、修到什么范围、由谁承担修复代价、哪些 truth 可以临时降级、哪些 user intent 不能被静默重写。`

所以 `226` 之后必须继续补的一层就是：

`安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层`

也就是：

`stronger-request cleanup-anti-drift verifier 只能发现并揭露 drift；stronger-request cleanup-repair governor 才能决定如何把世界重新拉回可接受状态。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-repair-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-repair-governor signer` 仍是研究命名。
它不是在声称仓库里已经有一张现成的 cleanup repair 接口表，
而是在说：

1. repo 其他子系统已经显式展示出 repair-governance grammar
2. stronger-request cleanup 线当前还没看到对等 grammar
3. 因而更高阶的问题已经不再只是“谁发现 drift”，而是“谁有权决定怎么修”

第二条：

这里说的 `repair governance`
不是在声称：

`一旦 drift 出现，答案就一定是补代码。`

真正要说的是：

`repair` 可能意味着修 executor、修 scheduler、补 receipt、改 wording、缩 promise、或做 session-local demotion；关键不在“修不修”，而在“谁来正式决定修哪种东西”。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部 repair problem。

真正关键的是：

1. temporal drift 可能要求修 scheduler 或降级 startup-delete 承诺
2. propagation drift 可能要求修 executor 或限制 override semantics
3. receipt drift 可能要求补 surfaced receipt 或降低 conformance 语言
4. validation skip 甚至可能要求增加 explainability surface，而不是扩大 destructive scope

第四条：

这里尤其不能过度声称：

`verifier 输出是纯中性的，不会隐含治理方向。`

当前更稳的说法只能是：

`一旦 verifier 报出 drift，系统就必须在不同 repair scope 之间做选择；repair scope 本身就是治理主权，而不是验证结果的自动副产物。`

第五条：

这里也不能过度声称：

`repair governance` 一定意味着持久修改 settings。

当前更稳的说法只能是：

`plugin 线已经明确展示出 session-local demotion 且 does NOT write settings；因此 cleanup 线未来的 repair 也可能是 session-local、receipt-local 或 promise-language-local，而不是只能落到持久状态。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-anti-drift-verifier signer 仍不等于 stronger-request cleanup-repair-governor signer`

证据：

1. `verifyAutoModeGateAccess()` 负责发现 stale-vs-live divergence
   `permissionSetup.ts:1078-1157`
2. `checkAndDisableAutoModeIfNeeded()` 才负责把 verifier 结果施加到 CURRENT app state 与 notification queue
   `bypassPermissionsKillswitch.ts:74-116`
3. `getNextPermissionMode.ts:11-17` 只负责 local precheck，并不拥有 repair sovereignty
4. `verifyAndDemote(...)` 的命名本身就把 verify 与 demote consequence 并列写出
   `dependencyResolver.ts:161-233`
5. `pluginLoader.ts:3189-3195` 还明确把 demotion 约束为 session-local consequence，且 `does NOT write settings`
6. stronger-request cleanup 当前虽已暴露 temporal / propagation / receipt drift，却没有一处可见代码正式决定这些 drift 应该修 executor、修 scheduler、补 receipt，还是缩 wording
   `cleanup.ts:33-45,575-597`; `plans.ts:79-110`; `cleanup.ts:300-303`
7. 因而 stronger-request cleanup 当前最缺的不是新 detector，而是 drift 暴露后的 repair grammar

因此这一章的最短结论是：

`stronger-request cleanup-anti-drift-verifier signer 最多能说“这里不对了”；stronger-request cleanup-repair-governor signer 才能说“该由哪一层付出代价把它重新拉回正确世界”。`

再压成一句：

`发现偏离，不等于拥有纠偏主权。`

## 4. 第一性原理：verifier 回答“哪里错了”，repair governor 回答“谁来承担修复代价”

从第一性原理看，
这两层治理处理的是两种根本不同的安全问题。

`anti-drift verifier`

回答的是：

1. 有没有 drift
2. drift 在哪一层
3. 证据是否足够强
4. 当前 truth 是否已分叉
5. 是否还应继续允许系统说“这里没问题”

`repair governor`

回答的则是：

1. 应该修 metadata，还是修 runtime
2. 应该修 scheduler，还是修 executor
3. 应该补 receipt，还是缩小承诺语言
4. 应该做 session-local consequence，还是持久修复
5. user intent、current safety 与 promise honesty 这三者谁先保

因此这两层的哲学本质也不同。

`verifier`

属于认识论层：

`它回答“真相是什么”。`

`repair governor`

则属于治理论层：

`它回答“为了让世界恢复可接受状态，谁来付代价、付哪种代价”。`

Claude Code 在这里真正先进的地方，
不是它“会在发现问题后做点什么”，
而是它已经显式展示出：

`truth exposure`

和

`consequence application`

可以是不同主权。

这是一种很成熟的安全哲学：

`检测者不自动拥有改世界的权力。`

因为一旦把检测与修复混成一层，
系统就容易出现三种危险：

1. verifier 自己偷偷重写用户意图
2. 修复 scope 被不小心放大成持久状态污染
3. 本该修 wording 的问题被误修成 destructive executor 扩张

## 5. 从技术角度看，Claude Code 的先进性在于“consequence application 也被当成受治理对象”

很多系统的修复逻辑只停在：

`发现问题 -> 直接改世界`

Claude Code 更成熟的地方在于，
它连 consequence application 本身都做了治理。

### 5.1 `checkAndDisableAutoModeIfNeeded()`：修复应用到 CURRENT state，而不是 stale snapshot

`bypassPermissionsKillswitch.ts:89-115`
最值钱的地方，
不是它会调用 verifier，
而是它在 apply consequence 时明确坚持：

`Apply the transform to CURRENT context, not the stale snapshot`

这说明 repo 连 repair application 自己都当作一类 anti-drift 风险来处理。

也就是说：

`不仅 verifier 可能漂，repair itself 也可能因为异步等待而漂。`

这是一种非常成熟的技术认识：

`修复后果的应用，也必须对抗 stale state。`

### 5.2 `verifyAndDemote(...)` + `pluginLoader`：修复不仅有动作，还有明确 scope 和边界

`dependencyResolver.ts:161-233`
把 verify 与 demote 写进一个明确 grammar，
而
`pluginLoader.ts:3189-3195`
进一步把 demotion 的边界钉死为：

1. `enabled -> false`
2. session-local
3. `does NOT write settings`

这说明 repair governance 的成熟度，
不只看有没有 consequence，
还看系统能否明确回答：

1. consequence 作用于哪一层 truth
2. consequence 持续多久
3. 哪些层绝不能被静默篡改

这对 stronger-request cleanup 非常关键。

因为 cleanup 修复如果没有这些边界，
最容易出现的不是“不修”，
而是“修过头”。

## 6. stronger-request cleanup 当前真正缺的，是 repair choice grammar

当前 stronger-request cleanup 已暴露出至少四类 repair choice：

### 6.1 temporal drift

`cleanupPeriodDays=0` 的 startup-delete wording
与
`backgroundHousekeeping` 的 delayed admission
之间存在时间错位。

这类 drift 的 repair 可能是：

1. 调整 scheduler
2. 提前 executor
3. 保留 scheduler 但降级 wording
4. 增加 “pending cleanup / deferred cleanup” receipt

### 6.2 propagation drift

`plansDirectory`
已被 storage / permission plane 消费，
但 `cleanupOldPlanFiles()`
仍只扫默认 `~/.claude/plans`。

这类 drift 的 repair 可能是：

1. 让 executor 跟随 override
2. 限缩 override 语义
3. 区分 storage plan root 与 cleanup plan root
4. 明确把这件事写成 session-local warning 而不是默默保留

### 6.3 receipt drift

`CleanupResult`
已有 local countability，
但未形成 family surfaced receipt。

这类 drift 的 repair 可能是：

1. 把 local count 提升为 family receipt
2. 增加 background cleanup explainability
3. 降低 conformance 话语强度

### 6.4 abstention drift

validation errors + explicit `cleanupPeriodDays`
时系统会 deliberate skip cleanup。

这类情形的 repair 未必是扩大 destructive path，
也可能是：

1. 补 stronger warning
2. 提供 explicit “not cleaned because …” receipt
3. 暂时熔断强承诺 wording

这些都不是 verifier 自己能决定的。

它们已经进入：

`repair-governance`

层。

## 7. 用苏格拉底式反问再压一层

1. 如果 verifier 发现 stronger-request cleanup drift 就已经等于 repair governance，为什么 `verifyAutoModeGateAccess()` 还要把后果应用交给 `checkAndDisableAutoModeIfNeeded()`？
   因为发现 divergence 与改变世界是两种主权。
2. 如果 repair scope 是显然的，为什么 plugin 线还要特意写明 demotion 是 session-local 且 `does NOT write settings`？
   因为 repair 不只是在决定“做什么”，还在决定“不能做什么”。
3. 如果 stronger-request cleanup 现在已经知道 `plansDirectory` drift，为什么它还没有自动长出正确修法？
   因为知道哪里不一致，不等于已经决定应由 executor、metadata 还是 promise wording 承担修复代价。
4. 如果 `CleanupResult` 已存在，为什么它不能自动等于 repair governance？
   因为 local count 只暴露 outcome，不决定是否要提升 receipt、缩 wording，还是改变 scheduling policy。
5. 如果 repo 已经有成熟 verifier 文化，为什么 cleanup 线仍停在 repair-governance 之前？
   因为成熟系统真正稀缺的从来不是 detector，而是 detector 之后谁有权动世界。

## 8. 硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要把 verifier 补齐，修复问题就会自然解决。`

而是：

`Claude Code 已经在 auto-mode 与 plugin 两条线上明确展示了 detection、consequence application 与 repair scope 的分层治理；stronger-request cleanup 线今天真正缺的，不是再多一个 detector，而是把 drift 暴露后的 repair sovereignty 正式分配出去。`

因此：

`artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-repair-governor signer。`

换句话说：

`系统知道哪里漂了，还不等于系统已经决定由谁、以什么代价、在什么范围内把它拉回正确世界。`
