# 安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层：为什么artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer

## 1. 为什么在 `264` 之后还必须继续写 `265`

`264-安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`
已经回答了：

`stronger-request cleanup 线即便知道旧对象回来后恢复哪些旧 seat / enable / identity truth，也还需要 re-entitlement governor 去决定这些资格究竟是否重新成立。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“资格回来就说明配置也一起回来了”
的错觉：

`只要 stronger-request cleanup re-entitlement governor 已经让旧对象重新获得 current seat，它就自动拥有了决定旧 options、旧 secrets、旧 channel config 与旧 payload 应怎样重新填写、重存、重刷新的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 线已经成熟写出的 reconfiguration 正对照，
会发现它其实明确拆开了四件事：

1. `seat truth`
2. `config truth`
3. `save truth`
4. `apply truth`

最硬的证据至少有七组：

1. `src/utils/plugins/pluginOptionsStorage.ts:90-193`
   的 `savePluginOptions()` 说明 top-level option truth 要按敏感性 split-save，
   并且做双向 scrub
2. `src/utils/plugins/pluginOptionsStorage.ts:282-309`
   的 `getUnconfiguredOptions()` 说明对象已经存在，
   仍可能被单独判定为 unconfigured
3. `src/utils/plugins/mcpbHandler.ts:141-339`
   的 `loadMcpServerUserConfig()` / `saveMcpServerUserConfig()` 说明 per-server / per-channel payload 也有自己的 split-save、secure-first 与 partial reconfigure grammar
4. `src/utils/plugins/mcpbHandler.ts:742-757,894-902`
   说明 validation failed 或 force reconfigure 时，
   系统不会把 seat 恢复偷写成 ready truth，
   而是显式返回 `needs-config`
5. `src/utils/plugins/mcpPluginIntegration.ts:290-317`
   的 `getUnconfiguredChannels()` 说明 channel config completeness 也是单独 decision surface
6. `src/commands/plugin/PluginOptionsFlow.tsx:53-133`
   的 `configured / skipped / error` outcome grammar 说明 config closure 不是 enabled truth 的附属尾气
7. `src/commands/plugin/ManagePlugins.tsx:1645-1668,1700-1707`
   明确把结果句法拆成 `Enabled and configured`、`Enabled` 与 `Configuration saved. Run /reload-plugins for changes to take effect.`

这说明：

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象重新获得了 current authority seat。`

后者才配说：

`它重新获得 seat 之后，哪些 current config truth 必须被重新填写、重新保存、重新验证，它才配开始工作。`

所以 `264` 之后必须继续补的一层就是：

`安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层`

也就是：

`stronger-request cleanup re-entitlement governor 负责让对象重新拿回资格；stronger-request cleanup reconfiguration governor 才负责决定它拿回资格后如何重新具备可运行的配置真相。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reconfiguration manager，
而是在说：

1. repo 已经在 plugin 线上明确把 `enabled` 与 `configured` 拆开
2. repo 已经在 config storage 线上明确把 `seat exists` 与 `config truth is valid` 拆开
3. stronger-request cleanup 线如果真要把 re-entitlement world 做完整，也迟早要回答这些“它拿回 seat 之后，凭什么按哪组 current config 重新工作”的问题

第二条：

这里说的 `reconfiguration governance`
不是在声称：

`一切旧配置都必须原样恢复。`

当前更稳的说法只能是：

`reconfiguration governance 首先回答的是当前哪组 config truth 被继续承认、哪组必须重填、哪组必须 scrub。`

第三条：

这里也不是把：

`enabled`

单独当成全部配置真相。

真正关键的是：

1. options 是否齐全
2. secrets 是否仍可信
3. per-server / per-channel payload 是否满足 schema
4. 保存后的配置是否已经真正 take effect

第四条：

这里尤其不能过度声称：

`配置已保存` 就等于 `配置已生效`。

当前更稳的说法只能是：

`repo 明确把 saved truth 与 applied truth 继续拆开；后者仍可能要求额外 refresh / reload choreography。`

第五条：

这里也不能过度声称：

`needs-config` 一定意味着对象无资格存在。

当前更稳的说法只能是：

`needs-config` 恰恰说明对象 seat 已经可见，但 config truth 尚未闭合。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-re-entitlement-governor signer 仍不等于 stronger-request cleanup-reconfiguration-governor signer`

证据：

1. `savePluginOptions()` 会把 `sensitive` 与 `non-sensitive` option 分仓保存，
   并在两侧做反向 scrub；
   这说明 seat 已经存在，
   不等于当前 options truth 已经自动成立
2. `getUnconfiguredOptions()` 会对保存值重新做 schema 校验，
   只把失败字段重新交回 dialog；
   这说明“已 enable”与“仍缺配置”可以同时成立
3. `saveMcpServerUserConfig()` 对 channel / server config 复现同样的 split-save、schema-flip scrub 与 partial reconfigure 保留逻辑；
   这说明 per-server payload 也有自己的 reconfiguration governance
4. `loadMcpbFile()` 在 `forceConfigDialog` 或 validation 失败时直接返回 `needs-config`，
   而不是把对象存在或 seat 恢复偷写成 ready truth
5. `PluginOptionsFlow` 明确把 outcome 写成 `configured / skipped / error`，
   说明 config closure 自身就是单独治理层
6. `ManagePlugins` 继续区分 `Enabled and configured`、`Enabled` 与 `Configuration saved. Run /reload-plugins...`，
   说明 config truth、saved truth 与 applied truth 仍是不同事实

因此这一章的最短结论是：

`stronger-request cleanup re-entitlement governor 最多能说对象重新拿回资格；stronger-request cleanup reconfiguration governor 才能说它拿回资格后按哪组 current config truth 重新工作。`

再压成一句：

`资格回来，不等于配置也回来。`

## 4. 第一性原理：re-entitlement governance 回答“它还配不配拿 seat”，reconfiguration governance 回答“这个 seat 凭哪组输入重新工作”

从第一性原理看，
强请求清理再赋权治理与强请求清理重配置治理处理的是两个不同主权问题。

`stronger-request cleanup re-entitlement governor`
回答的是：

1. 对象回来后是否重新获得 enabled truth
2. 是否重新获得当前 scope / precedence 位置
3. 是否重新获得原 identity，还是只拿到 new seat
4. 是否重新被 policy 允许
5. 是否重新进入可声明的 authority world

`stronger-request cleanup reconfiguration governor`
回答的则是：

1. 哪些 options 仍被视作缺失
2. 哪些 secrets 必须重填，哪些应继续放在 secure storage
3. 哪些 plaintext 配置必须被 scrub
4. 哪些 per-server / per-channel config 仍需单独保存
5. 配置保存后，当前运行时是否已经按这组新 truth 生效

如果把这两层压成一句
“反正它已经重新 enable 了”，
系统就会制造五类危险幻觉：

1. `enabled-means-configured`
2. `return-means-secret-restored`
3. `partial-save-means-full-schema-valid`
4. `saved-means-applied`
5. `authority-seat-means-operable`

再用苏格拉底式反问压一次：

1. 如果 re-entitlement governance 已经等于 reconfiguration governance，
   为什么 `savePluginOptions()` 还要把 secret / non-secret 分仓、双向 scrub，
   并且显式处理 partial reconfigure？
   因为“seat 回来”与“current config truth 已闭合”根本不是同一句制度。
2. 如果对象一旦重新 enable 就已具备完整 config truth，
   为什么 `getUnconfiguredOptions()` 与 `getUnconfiguredChannels()` 还要持续把失败字段重新暴露出来？
   因为 enable truth 与 config validity 是两层主权。
3. 如果配置一旦保存就等于正在生效，
   为什么 `ManagePlugins` 还要继续写
   `Run /reload-plugins for changes to take effect`？
   因为 saved truth 与 applied truth 仍不是同一件事。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“重配置治理”是一层独立制度

### 5.1 `savePluginOptions()` 给出最强正例：top-level option truth 不是 enable 的尾气，而是单独的重配置主权

`pluginOptionsStorage.ts:90-193`
很硬。

这段代码至少做了四件事情：

1. 按 `schema[key].sensitive` 把值拆到 `secureStorage` 与 `settings.pluginConfigs[pluginId].options`
2. 先写 `secureStorage`，失败就先不动 `settings.json`
3. 把本次应转去另一侧的 key 从旧 store scrub 掉
4. 只 scrub 本次 save 覆盖到的 key，避免 partial reconfigure 误删其他字段

这说明 Claude Code 并不把配置视作：

`对象重新被允许之后，顺手附带的一包数据。`

相反，
它把 config truth 当成另一套需要被精确治理的 current state：

1. secret 去哪一侧存
2. 旧 plaintext 何时必须撤销
3. schema 翻转时哪一边拥有当前真相
4. 局部重配置时哪些旧值仍可保留

这里最先进的设计不是“秘密单独存”，
而是：

`双侧当前真相如何保持不自相矛盾。`

尤其
`secureStorage FIRST`
这一条非常关键。
它不是一般的实现顺序，
而是一种安全哲学声明：

`宁可暂时保留旧 plaintext fallback，也不要因为 keychain 失败把 current config truth 弄成双侧都不可信。`

### 5.2 `getUnconfiguredOptions()` 再证明：配置完整性是单独 decision surface

`getUnconfiguredOptions():282-309`
的价值经常被低估。

它并不只是一个 UI helper。
它本质上在做一件治理性的事情：

`判定 current seat 是否已经具备足够 config truth。`

这段代码先读当前保存值，
再跑 `validateUserConfig()`，
只把失败字段重新打包回 schema slice。

这说明 repo 清楚承认：

1. 对象已经存在
2. schema 仍然不满足
3. 这不是异常，而是正式可继续处理的状态

也就是说，
在 Claude Code 里：

`configured`

不是对象存在的自然副产物，
而是一种需要被单独验证的 truth class。

### 5.3 `saveMcpServerUserConfig()` 把同一治理逻辑推进到 per-server / per-channel payload

`mcpbHandler.ts:141-339`
让这条线变得更硬。

`loadMcpServerUserConfig():141-162`
先把：

1. `settings.pluginConfigs[pluginId].mcpServers[serverName]`
2. `secureStorage.pluginSecrets[pluginId/serverName]`

合并成当前视图。

然后
`saveMcpServerUserConfig():193-339`
再精确规定：

1. sensitive / non-sensitive split
2. secure-first write order
3. schema flip 时的反向 scrub
4. partial reconfigure 的最小影响面

这一套设计说明 repo 并不把：

`channel / server config`

看成 top-level option 的附属细节。
它承认：

`不同 surface 的 payload truth 也需要独立治理。`

更强的是
`mcpbHandler.ts:742-757,894-902`。
这里在两条路径上都明确返回：

`status: 'needs-config'`

而且同时带上：

1. `configSchema`
2. `existingConfig`
3. `validationErrors`

这等于直接证明：

`object visible`

和

`payload configuration ready`

不是同一件事。

### 5.4 `PluginOptionsFlow` 与 `ManagePlugins` 再证明：`enabled`、`configured`、`applied` 是三层不同事实

`getUnconfiguredChannels():290-317`
很硬。
它对每个 channel：

1. 读取保存值
2. 跑 `validateUserConfig()`
3. 只要失败，就把该 channel 放进 `unconfigured` 列表

这使
`PluginOptionsFlow.tsx:53-133`
可以在 mount 时直接构造 step list：

1. top-level unconfigured options
2. per-channel unconfigured config

如果没有 step，
立刻 `onDone('skipped')`；
全部 step 完成后，
才 `onDone('configured')`。

这说明 flow 层对配置真相的理解，
不是：

`要么 enabled，要么 disabled`

而是至少三段：

1. object seat restored
2. config truth unresolved
3. config truth closed

`ManagePlugins.tsx:1645-1668,1700-1707`
甚至把这套分层直接公开写成结果句法：

1. `Enabled and configured`
2. `Enabled`
3. `Configuration saved. Run /reload-plugins for changes to take effect.`

一旦这三句话同时存在，
整个 repo 就在公开承认：

1. `enabled != configured`
2. `configured != applied`
3. `saved != take effect`

## 6. 更深一层的技术先进性：Claude Code 连“配置已保存”与“配置已生效”都继续拆开

`ManagePlugins.tsx:1663-1668,1700-1707`
明确写出：

`Configuration saved. Run /reload-plugins for changes to take effect.`

这说明 repo 在 config line 上甚至没有止步于：

`seat` vs `config`

它还继续承认：

`config saved`

和

`config applied to active world`

也不是同一件事。

从技术角度看，
这非常先进。
它避免系统把：

1. authority restoration
2. config persistence
3. active application

压成一句虚假的“已经可以用了”。

## 7. 哲学本质：再赋权是规范位阶的回归，重配置是输入真相的重建

如果把 `re-entitlement` 与 `reconfiguration` 抽到哲学层，
两者治理的不是同一类真相。

`re-entitlement`
首先回答的是规范位阶问题：

`这个对象回来后，当前制度还给不给它 seat。`

`reconfiguration`
回答的则是输入真相问题：

`它即便拿回了 seat，现在究竟按哪组 current inputs 重新工作。`

所以这两层的根本差异是：

1. 前者在恢复可被承认的资格
2. 后者在恢复可被信任的输入
3. 前者防的是无 seat 的误用
4. 后者防的是错配置的误用

Claude Code 在这里真正先进的地方，
不是它把配置流程做得复杂，
而是它默认承认：

`seat 的恢复，不足以推出 work-ready truth 的恢复。`

这是一种很成熟的安全哲学：

`规范地位可以回归，但输入真相必须重建。`

## 8. 苏格拉底诘问：怎样把这一层做得更严格

### 诘问一

`如果对象已经重新 enable，为什么还要继续追问 unconfigured / needs-config？`

因为 enable 只证明：

`它重新被允许存在。`

并不证明：

`它现在知道该用哪组 current inputs 工作。`

### 诘问二

`如果配置已经保存，为什么还不能直接宣布 ready？`

因为保存只说明：

`有一组新 truth 被写入了某些 store。`

它不证明：

`当前 active world 已按这组 truth 重新加载。`

### 诘问三

`为什么 partial reconfigure 反而更难做？`

因为它最容易制造“局部更新带坏全局 truth”的幻觉。
所以系统必须额外治理：

1. 哪些旧值保留
2. 哪些旧值必须 scrub
3. 哪些新值优先

### 诘问四

`那怎样才算更好的 reconfiguration governance？`

至少要继续把四件事制度化：

1. config completeness grammar
2. save ordering grammar
3. saved-vs-applied distinction
4. payload-specific validation grammar

少任何一层，
都容易把“对象有 seat”偷写成“对象已按正确输入工作”。

## 9. 技术启示与下一层约束

对 stronger-request cleanup 线而言，
这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 reconfiguration。`

而是更具体地说：

1. 旧 path、旧 promise、旧 receipt 即便恢复了 seat，也要重新回答它们按哪组 current options / secrets / payload 工作
2. 配置被重新保存，不等于它已经重新生效
3. current input truth 的闭合必须独立于 comeback 与 enable truth 被治理
4. `needs-config` 一类中间态不应被过早压平为 ready / not-ready 二值

只有这样，
`stronger-request cleanup re-entitlement-governance`
才不会继续被误写成一种过度强大的总主权：

`既负责把 seat 还给对象，又自动负责保证对象已经按正确配置开始工作。`
