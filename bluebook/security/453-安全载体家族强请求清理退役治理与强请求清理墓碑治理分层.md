# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `452` 之后还必须继续写 `453`

`452-安全载体家族强请求清理迁移治理与强请求清理退役治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样过渡、何时正式退役，也仍需要 sunset governor 去决定旧 law 从什么时候起不再算 current truth。`

但如果继续往下追问，
还会撞上一层更隐蔽、也更容易被偷写成“会退役就等于会把后事管好”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界结束，它就自动拥有决定结束后还该留下什么最小残留事实的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 自己已经落地的成熟机制，
会发现它在多个子系统里都把：

`old world 何时结束`

和

`结束之后还要留下什么最小机器可消费事实，继续约束未来解释`

拆成了两层不同治理。

最硬的证据至少有七组：

1. `src/query.ts:706-722`
   遇到 orphaned assistant messages 时不会只静默丢弃旧对象，
   而会显式 `yield { type: 'tombstone' }`
2. `src/utils/messages.ts:2948-2960`
   与
   `src/QueryEngine.ts:757-760`
   证明 `tombstone` 不是普通内容，
   而是后续消费者必须识别的 control signal
3. `src/screens/REPL.tsx:2646-2648`
   与
   `src/utils/sessionStorage.ts:121-123,925-945,1470-1473`
   证明 tombstone 会继续驱动 UI 与 transcript 删除链，
   而且这条链还需要专门的 rewrite 上限来保护资源边界
4. `src/utils/plugins/cacheUtils.ts:23-24,149-176`
   与
   `src/services/plugins/pluginOperations.ts:523-547`
   证明 `.orphaned_at` 不是 sunset 时点本身，
   而是对象退役后留在磁盘上的最小 marker；
   后者甚至直说 teardown fallout 会制造 `tombstones`
5. `src/utils/plugins/orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
   与
   `src/main.tsx:2544-2568`
   进一步证明 marker 还会被翻译成 grep / glob exclusion grammar，
   继续约束 future visibility world
6. `src/utils/config.ts:427-438`
   与
   `src/hooks/notifs/useModelMigrationNotifications.tsx:5-8,21-33,49-50`
   说明对象世界已经变化之后，系统有时仍要保留一个最小历史注脚，
   向当前 runtime 解释“为什么现在变了”
7. `MAX_TOMBSTONE_REWRITE_BYTES`、orphan grace marker 与 one-time migration trace
   这些设计放在一起看，
   说明 Claude Code 不只是会宣布旧世界结束，
   还会针对不同风险面决定应该留下哪一种最小残留物，以及谁继续消费它

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request cleanup 世界从什么时候起正式不再算当前世界。`

后者才配说：

`旧世界结束之后，系统还要留下什么最小机器可消费事实，来防止未来误读、误恢复、误审计与误复活。`

所以 `452` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责关门；stronger-request cleanup tombstone governor 才负责决定门后还要挂什么最小告示。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经存在一个未导出的墓碑 manager，
而是在说：

1. repo 在 message、plugin 与 model trace 三条线上已经展示出 post-retirement residue grammar
2. 这层 grammar 回答的是 `结束之后还要留下什么最小事实`
3. stronger-request cleanup 若要自称完整，也必须回答这些问题

第二条：

这里说的 `tombstone`
不能被狭义理解成“只有名字就叫 tombstone 的消息”。

更稳的说法是：

`tombstone governance` 研究的是 post-retirement minimal residue。

它可以是：

1. `tombstone` message
2. `.orphaned_at` marker
3. migration timestamp
4. 其他最小、可持续约束未来解释的 marker

第三条：

这里也不能把：

`系统里仍然留了一点旧痕迹`

直接当成：

`系统已经拥有完整墓碑治理。`

更稳的说法只能是：

`真正的 tombstone governance 不只是留下痕迹，而是正式规定 marker type、consumer set、future effect 与 second-stage responsibility。`

第四条：

这里尤其不能把：

`warning、note 或注释里提到 tombstone`

单独当成全部墓碑治理。

更稳的说法只能是：

`human-facing explanation 只是其中一层；真正更硬的是 machine-consumable control signal 是否存在、是否有明确消费者、是否约束未来读取。`

第五条：

这里也不能把：

`marker 还在`

误写成：

`旧世界仍然有效。`

更稳的说法只能是：

`marker 恰恰可能在表达“旧世界已经失效”这一最小事实；残留物的职责不是恢复 current truth，而是约束后来者如何理解无效对象。`

第六条：

这里还不能过度声称：

`Claude Code 在 message / plugin / model 线上怎样做 tombstone，stronger-request cleanup 线未来就必须完全照抄。`

更稳的说法只能是：

`不同旧世界可以有不同 marker grammar，但“谁来签 post-retirement residue、谁消费它、它能约束什么未来行为、谁负责二次清除或 resurrection gate”必须被正式回答；不能因为实现形式可变，就把 tombstone 主权偷写成 sunset 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer`

证据：

1. `query.ts` 遇到 orphaned assistant messages 时会显式产出 `tombstone`
   `query.ts:706-722`
2. `utils/messages.ts` 与 `QueryEngine.ts` 证明 tombstone 是 control signal，
   而不是普通消息内容
   `utils/messages.ts:2948-2960`; `QueryEngine.ts:757-760`
3. `REPL.tsx` 与 `sessionStorage.ts` 证明 tombstone 会继续驱动显示层与 transcript 存储层改口，
   而且 slow-path rewrite 还受专门上限控制
   `REPL.tsx:2646-2648`; `sessionStorage.ts:121-123,925-945,1470-1473`
4. `.orphaned_at` 不是 sunset 时点本身，
   而是对象退役后留在磁盘上的最小 marker
   `cacheUtils.ts:23-24,149-176`; `pluginOperations.ts:523-547`
5. marker 还会继续塑造 future search / visibility world
   `orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`; `main.tsx:2544-2568`
6. migration timestamp 证明对象世界变化之后，
   系统有时还要保留一个最小历史注脚去解释“为什么当前世界刚变”
   `config.ts:427-438`; `useModelMigrationNotifications.tsx:5-8,21-33,49-50`
7. stronger-request cleanup 当前虽已开始显露 sunset-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup path、旧 promise vocabulary 与旧 receipt semantics 退役后还应留下什么最小 marker

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留事实来约束未来解释。`

再压成一句：

`会退役，不等于会留下可治理的最小残留事实。`

再压成更哲学一点的话：

`制度知道何时关门，不等于制度知道门后还要挂什么告示。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
这两层治理处理的是两个不同真相面的主权问题。

`sunset governor`

回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

`tombstone governor`

回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. marker 将怎样约束未来读取、恢复、审计与解释
5. marker 何时再被进一步清除，谁对这次二次清除负责

所以这两层的哲学本质不同：

`sunset`

决定旧对象什么时候停止作为 current truth；

`tombstone`

则决定它停止之后，
系统还保留什么最小事实来约束后来者。

如果再压缩一点，
两者的本质差别其实是：

1. `sunset` 治理的是 `absence of validity`
2. `tombstone` 治理的是 `persistence of invalidity as an interpretable fact`

换句话说，
一个制度成熟不成熟，
不仅看它会不会宣布旧东西失效，
还要看它会不会把“它为什么失效、之后谁不能再误用它”写成最小而坚硬的机器事实。

## 5. Claude Code 多重安全技术的先进性：它不是只会删旧对象，而是会设计“残留物如何继续约束未来”

Claude Code 在这一层真正先进的地方，
不是“对象旧了就删掉”，
而是：

`它把 post-retirement residue 本身做成安全控制面。`

这种先进性至少体现在五个方面：

### 5.1 显式控制信号

orphaned message 不是静默过滤，
而是显式 `tombstone`。
这让后续 UI、transcript、QueryEngine 都能围绕同一控制信号收敛，
而不是各自猜测“旧对象是不是该没了”。

### 5.2 资源边界诚实性

tombstone slow-path rewrite 受 `MAX_TOMBSTONE_REWRITE_BYTES` 约束。
系统不把“理论上该删”伪装成“任何体量下都能安全重写”，
而是承认 destructive rewrite 也受成本上限治理。

### 5.3 延迟删除与阶段化退场

`.orphaned_at` 先把对象写成 orphan，
再等待 7 天后删除。
这把 `logical retirement` 与 `physical erasure` 拆开，
允许并发会话、恢复窗口与后续治理共存。

### 5.4 可见性与存储分层

orphan marker 不只用于以后删盘，
还会立刻塑造 grep / glob exclusion。
也就是说，
系统先治理“后来者还能不能看到它”，
再治理“磁盘上还有没有它”。

### 5.5 最小历史注脚

migration timestamp 不是为怀旧存在，
而是为当前 runtime 提供一次性解释：
世界已经变了，而且这次变化刚刚发生。
它把“变化已发生”压成最小可消费历史事实。

这些设计背后的源码思路也很清楚：

1. 不把“沉默”当成安全，宁可留下可消费的最小 marker
2. 不把“还留痕迹”当成失败，而是把残留物制度化
3. 不把“未来怎样解释这个失效对象”外包给调用方猜测
4. 不把 destructive cleanup 伪装成零成本动作
5. 不把 visibility、storage、history、recovery 四个世界混成一个世界

## 6. 为什么 stronger-request cleanup 现在仍停在“world-end boundary”，而不是“post-retirement marker grammar”

和这些成熟 tombstone 正对照相比，
stronger-request cleanup 当前最明显的缺口已经不是：

`旧世界何时结束。`

而是：

`旧世界结束后还留下什么最小机器事实。`

更具体地说，
现在仍没有哪一层正式回答：

1. 旧 startup wording 退场后是否应留下一个 machine-consumable rejection marker
2. 旧 home-root cleanup path 结束后是否需要留下一个 future-audit trace
3. 旧 promise vocabulary 失效后应由哪个 consumer 继续识别它只是已退役 residue
4. 旧 receipt / local-count truth 退场后应以何种最小 marker 防止 future recovery 误认它们仍有效
5. 这些 marker 由谁消费、何时再清除、哪些 resurrection gate 需要显式承认它们

换句话说，
stronger-request cleanup 当前虽然已经开始显露 sunset-governance，
却还没有长出足够正式的：

1. marker type plane
2. consumer set plane
3. future effect plane
4. rewrite / rewrite-budget plane
5. second-stage cleanup / resurrection responsibility

这意味着它现在能说的是：

`旧世界已经关门。`

却还不能更强地说：

`门后还留下什么最小告示，谁继续看这个告示，它还能约束哪些未来行为。`

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界已经正式结束”误认成了“旧世界结束后的解释义务也结束了”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 sunset boundary 已经明确，为什么还要再拆 tombstone？
   因为宣布结束，不等于决定结束后还留下什么最小事实。
2. 如果已经删掉旧对象，是否就等于后事都治理完了？
   不等于。删除动作本身不能替代未来解释、恢复和审计约束。
3. 如果 marker 还在，是否说明旧世界仍然有效？
   也不对。marker 常常正是在表达“它已经失效”。
4. 如果 warning 或 note 已经留下，是否就等于墓碑治理完整？
   不等于。human-facing note 不等于 machine-consumable control signal。
5. 如果对象已经从 visibility world 消失，是否就不再需要任何 residue？
   也不等于。history / recovery / audit 仍可能需要一个最小 marker。
6. 如果 cleanup 线现在还没有显式 tombstone 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层正式 marker grammar，越容易把“已经结束”的世界误恢复、误解释、误审计。

这一串反问最后逼出一句更稳的判断：

`tombstone 的关键，不在系统能不能给旧对象立一个名字，而在系统能不能正式决定哪些最小残留事实必须继续约束未来读者。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 sunset governance，就已经足够成熟。`

而是：

`Claude Code 在 message tombstone、bounded transcript rewrite、plugin orphan marker、marker-driven visibility exclusion 与 migration trace artifact 上已经明确展示了 tombstone governance 的存在；因此 artifact-family cleanup stronger-request cleanup-sunset-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁来宣布旧世界结束”，而是“谁来决定结束后还要留下什么最小机器事实继续约束后来者”。`
