# 宿主资格等级：观察宿主、控制宿主、证明宿主为何必须分层定义

## 1. 为什么在 `28` 之后还要继续写“宿主资格等级”

`28-显式降级理论` 已经回答了一个关键问题：

`窄宿主本身不是问题，隐性子集才是问题。`

但如果继续往前推进，  
还会剩下一个更根的问题：

`既然不同宿主天然宽度不同，那它们到底有资格承担哪一种安全责任？`

这不是措辞问题。  
因为统一安全控制台一旦落地成产品，  
宿主并不只是“一个显示端”，  
它会被用户默认理解成至少三种角色之一：

1. 告诉我现在发生了什么
2. 允许我对问题采取动作
3. 证明动作后问题真的被修复了

而这三件事的技术门槛并不相同。

所以这一章的核心问题是：

`Claude Code 的多宿主世界里，为什么必须正式区分观察宿主、控制宿主与证明宿主？`

## 2. 最短结论

从源码看，Claude Code 的宿主不应再被粗糙地分成“完整”和“不完整”，  
更合理的划分应当是三层资格：

1. `观察宿主`
2. `控制宿主`
3. `证明宿主`

这三层的区别不在于谁页面更多，  
而在于谁能承担更重的安全责任。

更具体地说：

- 观察宿主负责保留并展示安全对象
- 控制宿主负责对对象执行动作并显式处理不支持能力
- 证明宿主负责在竞态、重复响应、取消、恢复和状态回写里维持真相一致性

所以 `29` 的核心判断是：

`统一安全控制台不是“每个宿主都做同一件事”，而是“每个宿主只声明自己有资格承担哪一层责任”。`

## 3. 为什么“证明宿主”是新的关键层

很多系统只区分：

1. 能不能看
2. 能不能点

但 Claude Code 的源码已经说明，这还不够。

因为安全控制并不是按钮一按就算完，  
它会经过：

1. permission request / response
2. cancel / abort
3. duplicate response 去重
4. orphan response 处理
5. 状态回写
6. 恢复窗口保护

如果一个宿主不能在这些环节里维持一致性，  
那它就没有资格说：

`问题已经被证明修复。`

这正是“证明宿主”这一层必须被单独命名的原因。

## 4. 源码里的三层资格，其实已经隐约存在

## 4.1 `StructuredIO` 是“证明宿主”基线

`src/cli/structuredIO.ts:275-309` 很关键。  
这里不是简单地“收一个 response 再继续”，  
而是在做真正的控制一致性维护：

1. `injectControlResponse(...)` 会把 bridge 注入的结果接回原 permission flow
2. 同时主动发出 `control_cancel_request`
3. 防止外部 callback 挂死

这已经不是普通控制面，  
而是：

`双边竞争中的一致性维护。`

更关键的是 `src/cli/structuredIO.ts:362-429`：

1. 每个 `control_response` 都会进入统一闭环
2. duplicate response 会被识别并忽略
3. orphan response 也会被统一处理
4. 已解决的 `toolUseID` 会被跟踪，避免重复 assistant message 污染会话

这说明 `StructuredIO` 真正在做的不是“转发控制消息”，  
而是：

`证明控制链仍然一致。`

这正是证明宿主的最低门槛。

## 4.2 `RemoteIO` 说明高资格宿主不仅处理控制，还处理状态回写和恢复

`src/cli/remoteIO.ts:111-167` 又进一步表明：

1. `CCRClient` 初始化后会恢复 `restoredWorkerState`
2. internal events 会被写入、读取
3. command lifecycle 会映射成 delivery 状态
4. session state 和 metadata 会被主动上报

这意味着它并不只是“远程传个消息”，  
而是在做：

`状态持久化 + 生命周期回写 + worker 状态恢复。`

再看 `174-240`：

1. keep_alive 被主动维持
2. bridge 模式下 `control_request` 会被特殊回显
3. 传输层行为和上层语义被明确对齐

所以 `RemoteIO` 代表的不是一个普通远程宿主，  
而是一种更高资格的宿主：

`它有能力把对象、动作、恢复和状态回写同时连成一条链。`

## 4.3 `remoteBridgeCore` 说明证明宿主还必须有恢复窗口纪律

`src/bridge/remoteBridgeCore.ts:284-372` 暴露了另一个很关键的事实：

1. `authRecoveryInFlight` 会串行化恢复路径
2. 双重 rebuild 会被刻意避免
3. 恢复失败会进入明确状态

而 `src/bridge/remoteBridgeCore.ts:802-878` 更进一步说明：

1. `worker_status` / `reportState(...)` 会被显式维护
2. `can_use_tool` 会把状态推到 `requires_action`
3. `control_response` / `control_cancel_request` 会把状态推回 `running`
4. 401 恢复窗口里会主动丢弃 control_request / control_response / result

这说明一个宿主若想承担“证明”责任，  
它不仅要能收发控制消息，  
还必须能回答：

`在恢复窗口里，哪些动作应被视为不可信，哪些状态还能继续作为真相。`

没有这层纪律，  
就没有证明宿主。

## 4.4 `DirectConnectManager` 和 `RemoteSessionManager` 更接近“控制宿主”

相比之下：

- `src/server/directConnectManager.ts:82-97`
- `src/remote/RemoteSessionManager.ts:192-213`

都表明它们能明确处理：

1. `can_use_tool`
2. `interrupt`
3. 不支持时显式 error

这是很有价值的。  
但它们没有像 `StructuredIO` / `RemoteIO` / `remoteBridgeCore` 那样，  
展示出完整的：

1. duplicate / orphan 一致性治理
2. worker state 恢复
3. metadata 回写
4. 恢复窗口纪律

所以更准确地说，它们属于：

`控制宿主`

而不是证明宿主。

## 4.5 `sdkMessageAdapter` 甚至还没稳定达到“观察宿主”的理想形态

`src/remote/sdkMessageAdapter.ts:168-266` 说明它会：

1. 把 `status` 文本化
2. 忽略 `auth_status`
3. 忽略 `tool_use_summary`
4. 忽略 `rate_limit_event`

所以这条路径最多只能算：

`投影型观察宿主`

而且还是风险较高的那一类。  
因为它的观察并不是对象保真观察，  
而是：

`部分对象投影 + 部分对象静默丢弃。`

## 5. 三层宿主资格到底分别要求什么

## 5.1 观察宿主

最低职责是：

1. 能看见安全对象
2. 能区分对象存在与对象不可见
3. 不能把对象缺失伪装成系统正常

如果连这三件事都做不到，  
那它甚至没有资格做统一安全控制台的一部分。

## 5.2 控制宿主

在观察宿主之上，还必须：

1. 能对对象发起动作
2. 能显式处理 unsupported subtype
3. 能把动作结果回到正确对象上

也就是说，控制宿主的关键不是“按钮多”，  
而是：

`动作不会说谎。`

## 5.3 证明宿主

在控制宿主之上，还必须：

1. 处理 cancel / abort / duplicate / orphan
2. 维持 before / after 的一致性
3. 维持恢复窗口里的真相纪律
4. 回写 lifecycle、state、metadata 或其等价物

证明宿主的核心不是“更强”，  
而是：

`更能承担证明责任。`

## 6. 从第一性原理看，为什么必须分层定义资格

如果不分层，系统就会把所有宿主混成一句话：

`都在接同一个协议。`

但这在安全上是不够的。  
因为用户真正要知道的是：

1. 你能不能看见全部关键对象
2. 你能不能对这些对象采取有效动作
3. 你能不能证明动作后对象真的恢复

所以从第一性原理看，  
宿主资格分层不是架构洁癖，  
而是：

`责任分层。`

谁没有承担哪一层责任，  
就不该冒充那一层宿主。

## 7. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只是协议多、路径多、宿主多，而是这些源码已经足够支持一套正式的宿主资格模型。`

这对其他 Agent 平台构建者的启示非常明确：

1. 不要只写“宿主接入了协议”
2. 要写“宿主接入后有资格承担哪层责任”
3. 观察、控制、证明必须分层标注

否则系统最后就会出现一种危险错觉：

`所有接了协议的宿主都一样可信。`

而源码已经说明，这不是真的。

## 8. 哲学本质

这章更深层的哲学是：

`能力不是资格。`

一个宿主能点按钮，  
不代表它有资格证明。  
一个宿主能收消息，  
不代表它有资格代表系统真相。

所以从第一性原理看，  
统一安全控制台真正该治理的不是“功能列表”，  
而是：

`资格边界。`

## 9. 苏格拉底式反思：这章最容易犯什么错

### 9.1 我们是不是把 `StructuredIO` 神化成唯一正确宿主

不能。  
这里不是说其他宿主都没价值，  
而是说：

`它们承担的责任层级不同。`

### 9.2 控制宿主是不是就“不安全”

也不是。  
控制宿主可以非常安全。  
它只是不能声称自己已经完成证明。

### 9.3 证明宿主是不是一定更复杂

通常是。  
因为证明本身就意味着要承担更多一致性和恢复责任。  
复杂不是缺点，  
无声明地省掉这些复杂度才是缺点。

## 10. 结语

`28` 回答的是：显式降级为什么比隐性子集重要。  
`29` 继续推进后的结论则是：

`不同宿主不该再被粗糙理解成“完整/不完整”，而应该被正式划分为观察宿主、控制宿主与证明宿主。`

只有这样，  
统一安全控制台才不会把不同责任层的宿主混成同一种真相来源。
