# 安全恢复清理权限边界：为什么不是任何层都能删除恢复痕迹，必须由对应闭环所有者清理

## 1. 为什么在 `56` 之后还要继续写“清理权限边界”

`56-安全恢复留痕纪律` 已经回答了：

`未闭环恢复必须留下可重试、可回访、可解释的痕迹。`

但如果继续往下一层追问，  
还会出现一个更精确的问题：

`这些痕迹一旦存在，到底谁有资格把它们清掉？`

因为控制面最大的风险之一并不是：

`没有留痕`

而是：

`留了痕，但被错误层级顺手删了。`

例如：

1. 低层 UI 提示层把高层未闭环状态一并抹掉
2. 某个局部成功信号把不属于自己的挂起痕迹清掉
3. 低层 reconnect / refresh 成功后，顺手越权撤掉解释层告警

所以在 `56` 之后，  
统一安全控制台还必须再补一条制度：

`恢复清理权限边界。`

也就是：

`每一种恢复痕迹都必须由与其闭环强度相匹配的所有者清理，不能由更低层成功信号越权删除。`

## 2. 最短结论

从源码看，Claude Code 已经在多个地方事实上建立了这种边界：

1. 通知层只能删除自己的通知 key  
   `src/context/notifications.tsx:78-116,193-212`
2. plugin 的 `needsRefresh` 只能被 `refreshActivePlugins()` 真正消费并清零  
   `src/utils/plugins/refresh.ts:123-138`、`src/hooks/useManagePlugins.ts:301-302`
3. plugin auto-refresh 失败会重新把 `needsRefresh=true` 落回去，而不是假清理  
   `src/services/plugins/PluginInstallationManager.ts:146-177`
4. MCP reconnect trace 的删除依赖成功、最终失败或显式取消，不是任意地方随便清  
   `src/services/mcp/useManageMCPConnections.ts:387-442,1055-1064,1086-1122`
5. bridge pointer 只会在归档注销、resume stale/fatal 等特定高权场景才清  
   `src/bridge/bridgeMain.ts:1520-1577,2385-2403,2524-2533`

所以这一章的最短结论是：

`恢复痕迹不是普通 UI 状态，而是带有清理权限边界的控制对象。`

我会把这条结论再压成一句话：

`删除恢复痕迹，本身就是一种受限权限。`

## 3. 源码已经证明：不同痕迹由不同所有者清理

## 3.1 通知 key 的清理权限是最窄的，只能删除“提示”，不能删除“事实”

`src/context/notifications.tsx:78-116,193-212` 很关键。  
当前 notification 系统允许：

1. timeout 到期清当前提示
2. `invalidates` 清被替代的提示
3. `removeNotification(key)` 显式清某个 key

但这里真正被清掉的只是：

`notification queue/current`

它从来不负责清掉：

1. `needsRefresh`
2. `pending`
3. `failed`
4. pointer
5. signer 所支撑的更高层恢复状态

这说明通知层在当前系统里已经被自然限制成：

`最窄清理权限层`

它只能清自己，  
不能清更高层恢复痕迹。

## 3.2 `needsRefresh` 的清理权限被严格交给 `refreshActivePlugins()`

`src/utils/plugins/refresh.ts:123-138` 明确把：

`needsRefresh: false`

放在 `refreshActivePlugins()` 这一条真正完成重建的路径里。  
而 `src/hooks/useManagePlugins.ts:301-302` 又特地写出：

`Do NOT reset needsRefresh — /reload-plugins consumes it via refreshActivePlugins().`

这几乎就是一条教科书级的清理权限边界：

1. 可以很多地方设置 `needsRefresh=true`
2. 但不能很多地方把它清掉
3. 真正有资格清它的，只有那条把插件事实重新读回并写回 AppState 的闭环路径

所以这里的 `needsRefresh` 不是简单布尔位，  
而是：

`受限清理权痕迹。`

## 3.3 plugin auto-refresh 失败不会偷清状态，反而会把清理权退回高层挂起态

`src/services/plugins/PluginInstallationManager.ts:146-177` 说明：

1. auto-refresh 成功时可以推进闭环
2. 一旦失败，不会假装清理完毕
3. 会把 `needsRefresh=true` 再次明确写回

这说明作者已经意识到：

`自动化路径即便参与了恢复，也不代表它在失败后还能保留清理权。`

失败时，  
它必须把清理权让回给：

`显式人工 repair path`

也就是 `/reload-plugins`。

## 3.4 MCP reconnect timer 的清理权限也不是全局开放的

`src/services/mcp/useManageMCPConnections.ts:387-442,453-460` 显示：

1. automatic reconnection 会把 server 标成 `pending`
2. 写入 `reconnectAttempt`
3. 只有在成功、最终结果写回或到上限失败时，才删 timer、更新状态

同文件 `src/services/mcp/useManageMCPConnections.ts:1055-1064,1086-1122` 又说明：

1. 手工 reconnect / enable / disable 前会先 cancel 现有 timer
2. 这类取消发生在明确的域内控制动作里
3. 之后仍要重新走 `onConnectionAttempt(result)` 或更新 `pending/disabled`

这说明 reconnect timer、pending 和 failed 这些对象也有非常清晰的清理边界：

1. transport 域自己可以清自己的 timer
2. 但必须伴随明确状态转移
3. 不能由别的无关成功信号顺手抹掉

## 3.5 bridge pointer 的清理权限被故意收得很高

`src/bridge/bridgeMain.ts:1520-1577` 显示：

1. 正常 archive+deregister 后才清 pointer
2. 可恢复 SIGINT shutdown 会刻意不清

`src/bridge/bridgeMain.ts:2385-2403` 又显示：

1. resume stale / session gone / env missing 时可以清 pointer

`src/bridge/bridgeMain.ts:2524-2533` 最关键：

1. transient reconnect failure 不清 pointer
2. fatal reconnect failure 才清 pointer

这说明 bridge pointer 在当前设计里被视作一种非常高权的恢复痕迹。  
它的清理者不能是：

1. 某条普通通知
2. 某个短暂 reconnect success signal
3. 某个低层 UI 动作完成事件

它只能被：

1. 正常闭环终结
2. 确认 stale / fatal 的高权路径

清掉。

## 4. 第一性原理：清理权本质上是“宣布这条恢复链已无继续价值”的权力

如果从第一性原理追问：

`为什么“谁有权删除痕迹”必须单独上升成一条原则？`

因为删除一个恢复痕迹，  
实际在做的不是“界面清扫”，  
而是在宣布：

`这条恢复链不再需要继续进入了。`

而这显然是一种高权判断。  
因为一旦判断错了，  
系统就会把：

`仍有价值的修复入口`

当成：

`可安全遗忘的旧状态`

所以清理权限边界真正保护的是这条公理：

`只有能证明这条恢复链已经闭环，或已被确认不可继续的那一层，才有资格删除它的痕迹。`

## 5. 我给统一安全控制台的清理权限边界

我会把它分成四层。

## 5.1 L0 提示清理权

适用对象：

1. notification key
2. toast
3. prompt footer 提示

权限范围：

1. 只能清提示
2. 不能清底层恢复状态对象

## 5.2 L1 域内挂起态清理权

适用对象：

1. `needsRefresh`
2. reconnect timer
3. pending marker

权限范围：

1. 仅域内真正闭环路径或域内明确状态转移可以清
2. 不能跨域顺手清

## 5.3 L2 恢复锚点清理权

适用对象：

1. pointer
2. resume anchor
3. 可重试路径锚点

权限范围：

1. 只在正常终结、确认 stale、确认 fatal 时清
2. 低层 transient success / failure 均不配清

## 5.4 L3 解释层痕迹清理权

适用对象：

1. 高层盲区告警
2. 主结论降级痕迹
3. 解释权限相关痕迹

权限范围：

1. 只能由对应 signer 或人工明确确认闭环后清除
2. 绝不能由 L0/L1/L2 越权清掉

## 6. 我给清理权限的最低语法

我会把控制面里的清理语法压成：

`trace object -> owner -> allowed clearer -> forbidden clearer`

例如：

1. `notification key`
   -> owner: 通知系统
   -> clearer: timeout / invalidates / removeNotification
   -> forbidden clearer: 任何把它当成高层恢复已完成的逻辑
2. `needsRefresh`
   -> owner: plugin refresh 闭环
   -> clearer: `refreshActivePlugins()`
   -> forbidden clearer: 任意提示消失或低层 auto-refresh 尝试
3. `bridge pointer`
   -> owner: bridge resume 闭环
   -> clearer: archive+deregister / stale/fatal
   -> forbidden clearer: transient reconnect failure

## 7. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于它会保留痕迹，更在于它已经在多个关键对象上体现出“清理权必须受限”的设计。`

这对其他 Agent 平台构建者的直接启示有四条：

1. 痕迹删除应像状态写入一样被设计成有权限边界
2. 不同层级的痕迹应有不同清理者
3. 低层 UI 系统不应被允许清高层控制面事实
4. 清理错误本质上是一种越权解释

## 8. 哲学本质

这一章更深层的哲学是：

`真正克制的系统，不只限制谁能说“已经恢复”，也限制谁能说“这件事可以忘了”。`

很多系统失败的地方不在判断错误，  
而在清理过早。  
Claude Code 现有源码里那些 `needsRefresh` 只能被 `refreshActivePlugins()` 清、pointer 只在正常终结或 fatal/stale 时清、reconnect trace 只由域内状态机推进时清掉的细节，  
其实共同指向一条非常成熟的哲学：

`遗忘权必须比行动权更稀缺。`

从第一性原理看，  
这保护的是下面这条公理：

`控制面不得让更低层的成功信号越权删除更高层未闭环痕迹。`

## 9. 苏格拉底式反思：这一章最容易犯什么错

### 9.1 会不会把清理权收得太紧，导致系统永远堆积旧痕迹

会。  
所以关键不是“尽量不清”，  
而是：

`谁能清、在什么条件下清，要和 trace 对象同阶。`

### 9.2 会不会反过来把所有清理都塞给高层 signer，导致低层提示体验变差

也会。  
所以提示层仍应有自己的窄清理权，  
只是不能越权清更高层对象。


## 10. 结语

`56` 回答的是：未闭环恢复必须留下痕迹。  
这一章继续推进后的结论则是：

`这些痕迹还必须有清理权限边界，不能被错误层级顺手抹掉。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不仅要会保留失败，还要会限制谁有权忘记失败。`
