# 安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层：为什么artifact-family cleanup stronger-request cleanup-reactivation-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-readiness-governor signer

## 1. 为什么在 `234` 之后还必须继续写 `235`

`234-安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层` 已经回答了：

`stronger-request cleanup 线即便知道哪组 current truth 何时真正接管 active world，也还需要 reactivation governor 去决定 stale world 何时被 current world 替换。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“active world 已切换就说明现在已经能用了”
的错觉：

`只要 stronger-request cleanup reactivation governor 已经把新的 truth 接进 current session，它就自动拥有了决定对象现在是否真的 connected、authenticated、capable、ready-for-use 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 MCP 线已经成熟写出的 readiness 正对照，
会发现它其实明确拆开了四件事：

1. `state machine`
2. `surface grammar`
3. `consumer gate`
4. `runtime revocation`

最硬的证据至少有七组：

1. `src/services/mcp/types.ts:179-203`
   的 `MCPServerConnection` 联合类型把 client 明确拆成
   `connected / pending / needs-auth / failed / disabled`
2. `src/services/mcp/useManageMCPConnections.ts:149-153,765-853`
   说明 `/reload-plugins` 之后不是直接 ready，
   而是重新进入 `pending` lifecycle
3. `src/services/mcp/useManageMCPConnections.ts:1109-1123`
   说明 even after explicit enabling，
   client 也仍要先进入 `pending`
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
   只把 `failed / needs-auth` 当成 readiness 降级 surface，
   而不是把 active truth 直接冒充成 ready truth
5. `src/commands/plugin/ManagePlugins.tsx:512-519`
   与
   `src/cli/handlers/mcp.tsx:26-35`
   继续把 `connected / pending / needs-auth / failed / disabled`
   分开显化
6. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:78-95`
   对 downstream tool 明确要求 `client.type === 'connected'`
7. `src/services/tools/toolExecution.ts:1599-1628`
   会把一个原本 `connected` 的 client 在运行时降成 `needs-auth`

这说明：

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

和

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

仍然不是一回事。

前者最多能说：

`新的 truth 已经被接进当前世界。`

后者才配说：

`这个当前世界现在真的 ready，可以被后续 reader、tool 与 user-facing surface 当成可用事实继续消费。`

所以 `234` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层`

也就是：

`stronger-request cleanup reactivation governor 负责把 truth 接进 active world；stronger-request cleanup readiness governor 才负责决定这个 active world 现在是否真的可用。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-readiness-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 readiness manager，
而是在说：

1. repo 已经明确把 MCP 连接状态拆成 `connected / pending / needs-auth / failed / disabled`
2. repo 已经把 `activate`、`connected`、`needs auth`、`failed` 写成不同 surface
3. repo 已经让工具调用只接受 `connected`，而不是“刚重载过就算 ready”

第二条：

这里说的 `readiness governance`
不是在声称：

`只有完全无错误的系统才算 ready。`

当前更稳的说法只能是：

`readiness governance 首先回答的是下游 consumer 现在可不可以继续安全消费它，而不是回答系统是否永远不会再出错。`

第三条：

这里也不是把：

`connected`

单独当成全部 readiness grammar。

真正关键的是：

1. `pending` 还没闭合
2. `needs-auth` 是 readiness 未闭合
3. `failed` 是 readiness 失败
4. capability gate 仍会继续限制下游消费

第四条：

这里尤其不能过度声称：

`对象曾经连通过一次，就说明它现在仍 ready。`

当前更稳的说法只能是：

`readiness 可以被运行时证据持续撤销；它不是一次性永久 verdict。`

第五条：

这里也不能过度声称：

`用户看见 reload summary，就已经得到 capability proof。`

当前更稳的说法只能是：

`summary 只回答某次 reactivation choreography 已经运行，不自动回答所有 downstream consumer 都已经可以继续用。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-reactivation-governor signer 仍不等于 stronger-request cleanup-readiness-governor signer`

证据：

1. `MCPServerConnection`
   联合类型直接把 client state 拆成
   `connected / pending / needs-auth / failed / disabled`，
   说明 active world 已有对象，不等于对象已 ready
2. `useManageMCPConnections()`
   在 `pluginReconnectKey` 触发后，
   会先把新 client 置为 `pending`，
   再尝试连接，
   说明 reactivation 之后还存在单独的 readiness 过渡层
3. `useMcpConnectivityStatus()`、`getMcpStatus()` 与 `/mcp` health
   都把 `failed`、`needs-auth`、`pending` 单独显化，
   说明 repo 不把 active truth 偷写成 ready truth
4. `ReadMcpResourceTool`
   明确要求 `client.type === 'connected'`，
   否则直接报错，
   说明可用性消费面只接受 readiness truth
5. `toolExecution.ts`
   在 `McpAuthError` 时会把一个原本 `connected` 的 client 降成 `needs-auth`，
   说明 readiness 不是一次性 verdict，
   而是可被运行时持续撤销的主权
6. `toggleMcpServer()`
   就算是显式 enable，
   也仍先把对象写成 `pending` 再 reconnect，
   说明“重新接入”不自动等于“立即 ready”

因此这一章的最短结论是：

`stronger-request cleanup reactivation governor 最多能说新的 truth 已经进入当前世界；stronger-request cleanup readiness governor 才能说这个当前世界现在到底能不能被继续使用。`

再压成一句：

`重新激活，不等于已经就绪。`

## 4. 第一性原理：reactivation governance 回答“这组 truth 是否已接入当前世界”，readiness governance 回答“这个当前世界现在能不能被继续消费”

从第一性原理看，
强请求清理重新激活治理与强请求清理就绪治理处理的是两个不同主权问题。

`stronger-request cleanup reactivation governor`
回答的是：

1. stale world 是否已被 current world 替换
2. 哪个时刻完成 Layer-3 swap
3. 哪些 active components 已经换入
4. 当前 session 是否已经接到新的 config truth
5. 不同 mode 下谁来触发 active takeover

`stronger-request cleanup readiness governor`
回答的则是：

1. 当前 client 是否已 `connected`
2. 是否仍在 `pending`
3. 是否进入 `needs-auth`
4. 是否已经 `failed`
5. 下游工具、资源读取和 health check 是否可以继续消费它

如果把这两层压成一句
“反正已经 reload 了”，
系统就会制造五类危险幻觉：

1. `active-means-ready`
2. `pending-means-soon-enough`
3. `auth-gap-is-incidental`
4. `connected-once-means-ready-now`
5. `activation-summary-means-capability-proof`

再用苏格拉底式反问压一次：

1. 如果 reactivation governance 已经等于 readiness governance，
   为什么 `pluginReconnectKey` 触发后系统还要先把 client 加成 `pending`，
   而不是直接视为 ready？
   因为“truth 已接进 current world”与“现在可以继续消费”根本不是同一句制度。
2. 如果当前世界一旦 reactivated 就已经 ready，
   为什么 `ReadMcpResourceTool` 还要坚持 `client.type === 'connected'` 的 hard gate？
   因为 downstream consumer 只接受 readiness truth，不接受 mere reactivation。
3. 如果 readiness 是一次性 verdict，
   为什么 `toolExecution.ts` 还会把一个原本 `connected` 的 client 降成 `needs-auth`？
   因为 readiness 会被运行时证据持续撤销，它不是一张永久通行证。

## 5. `MCPServerConnection` 与 `useManageMCPConnections()` 先证明：active world 里也允许正式存在 non-ready 对象

`src/services/mcp/types.ts:179-203`
很硬。

这里没有把 MCP client 简化成：

`present / absent`

而是显式拆成：

1. `connected`
2. `pending`
3. `needs-auth`
4. `failed`
5. `disabled`

只要这种联合类型存在，
系统就已经公开承认：

`active world has object`

并不自动等于：

`object is ready`

`src/services/mcp/useManageMCPConnections.ts:765-853`
进一步把这种分层做成运行时制度。
文件注释明确写出：

`Re-runs on session change (/clear) and on /reload-plugins (pluginReconnectKey).`

而在 effect 内，
新 client 会先被加成：

`type: 'pending'`

再继续进入连接流程。

这说明 `pluginReconnectKey` 的意义不是：

`ready`

而是：

`readiness evaluation should begin again`

也就是说，
reactivation 只是在重新开启 readiness 管道，
不是直接签发 readiness 本身。

## 6. `pending / needs-auth / failed` 的 surface grammar 再证明：repo 明确拒绝把激活后的当前世界冒充成可用世界

`src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
很值钱。

这里只在两类状态上发通知：

1. `failed`
2. `needs-auth`

这不是细枝末节。
它说明系统继续追问的是：

`现在能不能用`

而不是：

`它是不是已经在 current world 里`

`src/commands/plugin/ManagePlugins.tsx:512-519`
又把 status 显式枚举为：

`connected / disabled / pending / needs-auth / failed`

而
`src/cli/handlers/mcp.tsx:26-35`
则把 health grammar 压成：

1. `✓ Connected`
2. `! Needs authentication`
3. `✗ Failed to connect`

当 notification、管理界面与 CLI health 都保留这组分层时，
repo 就已经在多个 consumer surface 上明确承认：

`reactivation`

之后仍然有一整层：

`readiness adjudication`

没有被偷掉。

## 7. `ReadMcpResourceTool` 给出最强消费侧正例：可用性消费面只接受 `connected`

`src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:78-95`
很硬。

它找到 client 后，
先检查：

`if (client.type !== 'connected') throw ...`

再检查：

`if (!client.capabilities?.resources) throw ...`

这说明 tool consumer 的制度不是：

`只要对象已经在 active world 里，就可以继续消费`

而是：

`只有 readiness verdict 已经强到 connected + capability-present，我才消费它`

这条证据非常值钱。
因为它把 readiness 从 UI 提示提升成了真正的 downstream gate。

从 cleanup 研究角度看，
这意味着将来旧 path、旧 promise、旧 receipt 即便已经 reactivated，
也仍不自动说明：

`下游 reader 已经可以安全消费它。`

## 8. `toolExecution.ts` 再证明：readiness 不是静态标签，而是会被运行时证据持续撤销的主权

`src/services/tools/toolExecution.ts:1599-1628`
直接给出一个更强反例。

这里在 `McpAuthError` 时，
会把一个原本 `connected` 的 client 更新成：

`type: 'needs-auth'`

而且只在原状态确实是 `connected` 时才改。

这等于说明：

1. readiness 并不是一旦连上就永远成立
2. 运行时证据可以把 ready truth 撤销掉
3. 系统会把这次撤销传播回 `/mcp` display 与后续 consumer

这正是 readiness governance 的本体。

它处理的不是“对象在不在”，
而是：

`这个对象当前还能不能继续被安全消费`

## 9. 这层分工的技术先进性与设计启示

Claude Code 在这层设计上真正先进的地方，
不是“对象能被重新连上”，
而是它已经把
`present`
和
`ready`
拆成至少四个不能互相偷写的技术问题：

1. `state machine`
2. `surface grammar`
3. `consumer gate`
4. `runtime revocation`

这四层给 stronger-request cleanup 线的技术启示也非常直接：

1. 旧 cleanup truth 即便已经重新接入 current world，
   也不应自动被后续 consumer 当成可用 truth
2. cleanup readiness 的 verdict
   不应由 reactivation signer 顺手代签，
   而应由掌握 runtime state machine 与 consumer gate truth 的更强 signer 单独签发
3. cleanup line 如果只补 `active takeover` grammar，
   却不补 `pending / needs-auth / failed` 这些 state grammar，
   就会把 present truth 错写成 ready truth
4. 真正成熟的 readiness grammar，
   必须允许 runtime revocation，
   否则系统会把一次性的 connected history 误写成持续有效的可用性

所以这一章最后可以压成一句硬结论：

`stronger-request cleanup reactivation governor 负责把新的 truth 接进当前世界；stronger-request cleanup readiness governor 才负责决定这个当前世界现在是否真的 ready-for-use。`
