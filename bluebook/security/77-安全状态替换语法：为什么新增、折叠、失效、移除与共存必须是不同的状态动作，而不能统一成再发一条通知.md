# 安全状态替换语法：为什么新增、折叠、失效、移除与共存必须是不同的状态动作，而不能统一成再发一条通知

## 1. 为什么在 `76` 之后还要继续写“状态替换语法”

`76-安全能力显式让位` 已经回答了：

`更高风险、更具体的新状态必须主动驱逐旧的弱状态。`

但如果继续往下追问，  
还会碰到一个更底层、也更接近控制面协议设计的问题：

`那是不是所有状态变化都该驱逐旧状态？`

显然不是。

因为在 Claude Code 里，  
不同状态变化其实对应完全不同的动作：

1. 有些应该共存
2. 有些应该折叠
3. 有些应该失效旧项
4. 有些应该由拥有者直接移除

如果把这些动作全部压成一句：

`再发一条通知`

系统就会立刻退化成两种坏状态：

1. 该共存的状态互相挤掉
2. 该作废的旧状态继续赖着不走

所以在 `76` 之后，  
安全专题必须再补一条更协议级的原则：

`状态替换语法。`

也就是：

`成熟控制面不能只知道何时更新状态，还必须知道当前这次更新究竟属于新增、折叠、失效、移除还是共存。`

## 2. 最短结论

从源码看，Claude Code 的通知与状态面已经隐含了一套小型 replacement grammar：

1. 基础通知协议同时定义了 `fold` 与 `invalidates`，说明“折叠”和“失效”是两种不同的一级语义  
   `src/context/notifications.tsx:16-23`
2. 同 key 的重复事件如果提供 `fold`，系统会把它们合并成一个更高阶状态，而不是继续堆叠  
   `src/context/notifications.tsx:121-169`
3. 互斥的新状态如果提供 `invalidates`，系统会显式清掉旧状态，而不是让它自己过期  
   `src/context/notifications.tsx:78-117,172-191`
4. 状态拥有者还可以直接 `removeNotification(key)`，说明“撤回自己名下状态”是独立动作，不等于被别人废止  
   `src/context/notifications.tsx:193-213`
5. 重复 teammate 生命周期事件走的是 `fold`，fast mode started / expired 走的是对称 `invalidates`，settings error / external editor hint / ultrathink-active 走的是 owner-driven `remove`  
   `src/hooks/notifs/useTeammateShutdownNotification.ts:18-46,49-77`  
   `src/hooks/notifs/useFastModeNotification.tsx:101-121`  
   `src/hooks/notifs/useSettingsErrors.tsx:37-48`  
   `src/components/PromptInput/Notifications.tsx:150-162`  
   `src/components/PromptInput/PromptInput.tsx:748-769,787-790`
6. 还有一些状态根本不该互相替代，而应该共存展示，例如当前通知槽、IDE 状态、token warning、memory usage 与 sandbox hint  
   `src/components/PromptInput/Notifications.tsx:286-330`

所以这一章的最短结论是：

`Claude Code 真正先进的地方，不只是会替换状态，而是知道不同状态变化该使用不同的替换动词。`

我把它再压成一句：

`安全控制面不是在追加消息，而是在执行一套状态编辑语法。`

## 3. 源码已经说明：Claude Code 把状态更新分成至少四种不同动词

## 3.1 `fold` 与 `invalidates` 并列出现在协议里，已经说明“替换”不是单一动作

`src/context/notifications.tsx:16-23` 非常值得单独写。

这里 `BaseNotification` 不是只描述：

1. key
2. priority
3. timeout

还同时描述：

1. `fold`
2. `invalidates`

这意味着作者从一开始就知道：

`重复出现的新状态`  
和  
`互斥到必须作废旧状态的新状态`

不是同一种事情。

换句话说，  
Claude Code 的通知协议不是：

`append-only plus timeout`

而是：

`state-editable`

这正是控制面系统与普通 toast 系统的分水岭。

## 3.2 `fold` 解决的不是“消重”，而是“把重复事件提升成更高阶语义”

`src/context/notifications.tsx:121-169` 定义了折叠路径：

1. 如果当前通知同 key，则 fold into current
2. 如果队列里已有同 key，则 fold into queued notification
3. fold 后会继承新的结果与 timeout

而 `src/hooks/notifs/useTeammateShutdownNotification.ts:18-46,49-77` 则给出了最干净的业务例子：

1. `teammate-spawn`
2. `teammate-shutdown`
3. 重复事件不会堆成一串
4. 而是折叠成 `3 agents spawned`、`2 agents shut down`

这说明 `fold` 的真正作用不是“少占空间”，  
而是：

`把离散重复事件提升成更高层次的聚合事实。`

如果这里误用 `invalidates`，  
系统只会保留最后一个事件；  
如果误用简单 append，  
系统又会把同一语义刷成噪声。

因此 `fold` 保护的是：

`重复中的语义聚合。`

## 3.3 `invalidates` 解决的是“互斥真相替换”，不是“拥有者撤回”

`src/context/notifications.tsx:78-117,172-191` 已经说明：

1. `invalidates` 可以驱逐队列项
2. 也可以驱逐当前项
3. immediate 和 non-immediate 都支持这种语义

而 `src/hooks/notifs/useFastModeNotification.tsx:101-121` 给出了一个非常标准的对称互斥例子：

1. cooldown started invalidates expired
2. cooldown expired invalidates started

这说明 `invalidates` 的本质是：

`我来了以后，你那句旧话就不再成立。`

它并不要求两个状态来自同一个 hook 生命周期，  
也不要求“拥有者”主动回来清理自己。  
它更像：

`新的真相正式宣布旧真相失效。`

这与 `removeNotification` 有本质区别。

## 3.4 `removeNotification` 代表的是“拥有者撤回”，而不是“新状态推翻旧状态”

`src/context/notifications.tsx:193-213` 提供了一个独立能力：

`removeNotification(key)`

这个动作的语义不是：

`我来证明旧状态错了`

而是：

`我作为该状态的持有者，确认它的适用条件已经消失。`

这个区别在业务层非常清楚：

1. `src/hooks/notifs/useSettingsErrors.tsx:37-48`  
   当 settings errors 归零时，拥有者自己撤回 `settings-errors`
2. `src/components/PromptInput/Notifications.tsx:150-162`  
   当 external editor hint 条件不成立时，拥有者自己撤回 `external-editor-hint`
3. `src/components/PromptInput/PromptInput.tsx:748-769,787-790`  
   当 ultrathink / ultraplan 触发器不再成立或 stash hint 应被取消时，拥有者主动移除对应提示

这说明 `remove` 的哲学不是冲突，  
而是：

`生命周期结束。`

也就是说：

`invalidates` 是跨状态裁决，remove 是同状态收尾。`

## 3.5 共存不是遗漏替换，而是刻意保留正交状态

如果只看到 `fold / invalidates / remove`，  
很容易误以为成熟控制面就该尽量压成一个当前状态。

但 `src/components/PromptInput/Notifications.tsx:286-330` 正好说明相反情况。

在这里：

1. `IdeStatusIndicator`
2. `notifications.current`
3. overage 提示
4. apiKeyHelper slow
5. auth error
6. token warning
7. auto updater
8. memory usage
9. sandbox hint

可以同时出现。

这说明作者并没有盲目追求“单状态面”。  
相反，  
他们已经在隐含地承认：

`有些状态是正交维度，应当共存，而不是互相替换。`

所以真正成熟的 replacement grammar 至少包含第四个动词：

`coexist`

它保护的是：

`不同维度事实不被错误压平。`

## 3.6 第一性原理：状态替换语法真正保护的，是“不同变化类型不被误判成同一种编辑”

如果从第一性原理追问：

`为什么一定要把 fold、invalidate、remove、coexist 分这么细？`

因为它们解决的是四类完全不同的问题：

1. `fold`  
   重复事件如何升格成聚合语义
2. `invalidate`  
   互斥新真相如何正式废止旧真相
3. `remove`  
   状态拥有者如何在条件消失后自己收尾
4. `coexist`  
   正交维度如何避免被压成单线叙述

如果把这四类编辑压成同一种“再发一条”，  
系统就会失去两个最关键的能力：

1. 正确表达状态之间的关系
2. 正确路由用户的下一步动作

因此 replacement grammar 不是 UI 优化，  
而是：

`状态关系的语义编码。`

## 3.7 技术先进性：Claude Code 在用极少量协议字段换来高密度状态治理能力

从技术角度看，  
这套设计的先进性恰恰在于它并不臃肿。

核心原语其实很少：

1. `key`
2. `priority`
3. `fold`
4. `invalidates`
5. `removeNotification`

但这几件原语组合起来，  
已经足以表达：

1. 竞争
2. 聚合
3. 作废
4. 撤回
5. 共存

这类设计的价值在于：

`用很小的协议面，换来了很强的状态治理密度。`

对别的 Agent 平台来说，  
这是一条很值得抄的工程启示：

`不要急着造复杂状态系统，先把状态替换动词定义清楚。`

## 4. 状态替换语法的最短规则

把这一章压成最短规则，就是下面六句：

1. 重复事件若属于同一语义，应优先 `fold`
2. 互斥新状态若使旧状态不再成立，应使用 `invalidate`
3. 状态拥有者在自身适用条件消失时，应主动 `remove`
4. 正交状态不该互相替换，而应允许 `coexist`
5. immediate 只解决抢位，不自动回答折叠还是作废
6. 如果系统分不清这四类动作，屏幕上的状态关系一定会越来越乱

## 5. 苏格拉底式追问：这套语法还能怎样再向前推进

如果继续自问，  
这套语法仍然还可以再向前走一步：

1. 现在 `fold / invalidates / remove` 已经存在，但 `coexist` 仍是隐式约定，尚未被显式编码为关系类型
2. 不同 hook 仍在手工决定自己属于哪一种替换动词；未来可把这些选择提升为 declarative relation table
3. 当前协议没有显式记录“为什么这次是 fold 而不是 invalidate”；若后续要做统一安全控制台，最好把 relation reason 也做成机器可读字段

所以这一章最终沉淀出的设计启示是：

`成熟控制面不是状态多，而是状态之间的编辑关系被说清楚了。`
