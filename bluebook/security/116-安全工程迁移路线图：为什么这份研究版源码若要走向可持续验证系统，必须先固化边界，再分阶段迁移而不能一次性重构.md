`真实可持续演进的工程路径`

## 2. 最短结论

从源码与仓库结构看，这份仓库最合理的演进方式绝不是“一步到位重构”，  
而必须是分阶段迁移：

1. 仓库定位首先是“研究版 extracted source”，不是完整 monorepo  
   `README.md:1-6`
2. 构建层自己也承认只能做到 best-effort，且仍依赖 Bun compile-time intrinsics 与缺失模块问题  
   `QUICKSTART.md:3-5,58-70,102-103`
3. 当前已有 `prepare-src/build/check/start` 这些基础脚手架，说明可以逐步加制度，而不必先推翻  
   `package.json:7-11`
4. 但测试入口、测试目录、测试类型配置、CI 宿主都还未形成制度化接口  
   `package.json:7-11`; `tsconfig.json:16-18,28-35`; `.github => no`

所以这一章的最短结论是：

`最优路线不是“大重构”，而是“先固化研究边界，再沿现有 build/check 骨架分阶段接入验证制度，最后才考虑更高层的统一状态机与验证体系”。`

再压成一句：

`先把边界说清，再把制度接上，最后才谈全面工程化。`

## 3. 第一性原理：为什么不能一次性重构

因为这份仓库并不是普通源码仓库，  
而是一份：

`不完整、带预处理、带 feature-gate 缺口、以研究为目的的源码研究版。`

这意味着如果贸然一次性推进：

1. 测试制度
2. 状态机重构
3. ledger 抽象
4. 统一 dispatch
5. CI 自动化

很容易把“研究边界”和“工程边界”混为一谈。  
而这是最危险的错误之一。

因为一旦混淆，  
系统就会出现两种假象：

1. 误把研究版当成可生产演进仓库
2. 误把暂时缺失的内部模块当成局部实现缺口

所以迁移路线图的第一原则不是“快”，  
而是：

`先正确理解当前对象是什么。`

`research-extracted, guard-rich, build-check-capable, test-thin, feature-incomplete source study repo`

1. `research-extracted`

   `README.md:15`; `QUICKSTART.md:60-70,102-103`

## 5. 最合理的五阶段迁移路线

## 5.1 Phase 0：固化研究边界

目标：

1. 明确这是研究版、非完整上游工程
2. 明确缺失模块与 Bun intrinsics 边界
3. 明确哪些结论可工程化，哪些结论仍只能停留在研究层

为什么必须先做：

因为如果这一步不先完成，  
后面所有工程化动作都会面对一个根本问题：

`到底是在修这个仓库，还是在假装替一个不存在的完整上游工程设计制度。`

Phase 0 的退出标准：

1. 仓库定位被明确
2. 缺失边界被明确
3. 工程化目标被限定在“研究版可持续验证能力”而非“完整产品复刻”

## 5.2 Phase 1：固化安全制度知识

这一阶段其实就是我们已经在做的事情：

1. 生命周期协议
2. ledger
3. dispatch
4. constitution
5. invariants
6. verification pyramid
7. priority roadmap

其作用是：

`先把隐含规则变成显式知识资产。`

为什么重要？

因为没有这一步，  
后面的脚本、测试、接口都不知道到底在守什么。

Phase 1 的退出标准：

1. 高风险制度被明确 catalog 化
2. 首批验证套件已定义
3. 制度化接口已定义

## 5.3 Phase 2：接入最小验证制度

这是下一步最自然的工程动作。

目标：

1. 加入测试脚本入口
2. 建立 `tests/security/*` 骨架
3. 建立测试配置入口
4. 落首批五组最贵失真护栏

这一步不是追求完整覆盖，  
而是追求：

`让仓库第一次拥有真正可执行的安全验证制度。`

为什么它排在这里而不是更后？

因为在研究版源码这种环境里，  
最紧迫的不是大重构，  
而是先防未来修改把已知 guard 轻易破坏掉。

Phase 2 的退出标准：

1. 有统一测试入口
2. 有最小目录骨架
3. 首批高优先级 suite 已落地
4. 开发者能用单一命令执行安全验证

## 5.4 Phase 3：局部制度收口为统一工程接口

在最小验证制度建立后，  
再推进：

1. 更多 centralized transition
2. 更明确的 lifecycle ledger
3. invariant catalog
4. 更系统的 test layers

这一步的目标不是“立刻重写”，  
而是：

`在已有护栏下，让局部成熟规则开始收口成统一接口。`

为什么不先做这个？

因为如果先抽象再补护栏，  
重构期本身就可能把旧制度打碎。

Phase 3 的退出标准：

1. 至少一两个高风险子系统完成从 local guard -> centralized transition 的升级
2. ledger / dispatch / tests 开始共享同一批制度定义

## 5.5 Phase 4：自动执行与长期维持

最后才是：

1. CI 挂载位
2. pre-merge 规则
3. suite owner
4. regression policy
5. 失败 triage 流程

这一步的目标是：

`把安全验证从可执行提升成可持续。`

只有到这一步，  
系统才算真正从：

`研究型制度资产`

走到：

`工程型制度资产`

## 6. 为什么“先制度、后抽象、再自动化”是最合理顺序

这是整章最核心的工程判断。

很多团队会本能地想：

1. 先把抽象做漂亮
2. 再补测试
3. 最后加 CI

但对这份仓库来说，  
最合理顺序恰恰相反：

1. 先固化制度知识
2. 再接最小验证制度
3. 再做更高层抽象收口
4. 最后上自动执行

原因很简单：

`研究版源码最怕的不是抽象不美，而是制度先天脆弱。`

所以必须先让制度有落脚点，  
再让抽象变漂亮。

## 7. 从技术角度看，每一阶段各自守什么风险

Phase 0 守的是：

1. 边界误判
2. 仓库角色误判

Phase 1 守的是：

1. 规则失语
2. 制度只存在于实现里

Phase 2 守的是：

1. 已知高价值 guard 被未来改坏
2. 最贵时序漏洞没有自动化护栏

Phase 3 守的是：

1. 多子系统重复造轮子
2. 规则分散导致语义飘移

Phase 4 守的是：

1. 本地能跑、自动流程不跑
2. 团队记得、仓库不强制

所以这不是普通 roadmap，  
而是一张：

`按风险种类分层收敛的迁移路线图。`

## 8. 为什么这份路线图对研究版源码尤其重要

因为研究版源码与正常产品仓库的最大区别在于：

1. 它更容易被当成“只读资料”
2. 它更容易缺少正式工程入口
3. 它更容易陷入“结论很多、制度很少”

而一旦走到我们现在这一步，  
如果还不补路线图，  
整份安全研究就很可能停在：

`概念体系非常强，但工程演化路径不清晰`

这会让后续任何真实落地都陷入重复争论：

1. 先补什么
2. 先不先上 CI
3. 先抽象还是先测试
4. 这份仓库到底该不该工程化

路线图的价值就在于：

`提前结束这些高层歧义。`

## 9. 苏格拉底式反思：如果要把这份路线图做得更好，还应继续追问什么

可以继续追问八个问题：

1. 哪些阶段应以 commit 粒度推进，哪些阶段应以专题里程碑推进
2. 哪些验证制度最适合先在研究版仓库内演示，哪些必须等待更完整上游环境
3. `prepare-src` 是否应被视为所有未来验证入口的前置门
4. 首批 suite 落地后，第二批应优先扩面积还是加深随机化
5. CI 缺席的情况下，是否应先定义本地“准 CI”入口
6. 哪些结论若未来被证伪，会影响整张路线图的排序
7. 这份路线图里哪些阶段最可能被过早跳过
8. 如何避免“路线图写得很完整，但没有任何阶段真正开始”

这些问题共同逼问的是：

`这份研究究竟是在提供一套持续演进路径，还是只是在把更长的结论继续堆上去。`

## 10. 一条硬结论

对这份 Claude Code 研究版源码来说，  
真正成熟的下一步不是：

`假设自己已经是完整上游工程，然后一次性大重构，`

而是：

`先承认自己是 research-extracted、feature-incomplete、guard-rich but test-thin 的起点，再按“固化边界 -> 固化制度 -> 接入验证 -> 收口抽象 -> 自动执行”五阶段稳步迁移。`

因为只有这样，  
这份安全研究才会从：

`越来越完整的分析文本`

进一步升级到：

`越来越可持续的安全工程迁移方案。`
