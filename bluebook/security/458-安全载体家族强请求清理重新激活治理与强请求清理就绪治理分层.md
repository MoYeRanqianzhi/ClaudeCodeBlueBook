# 安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层：为什么artifact-family cleanup stronger-request cleanup-reactivation-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-readiness-governor signer

## 1. 为什么在 `457` 之后还必须继续写 `458`

`457-安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪组 current truth 何时真正接管 active world，也还需要 reactivation governor 去决定 stale world 何时被 current world 替换。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
`active world 已切换就说明现在已经能用了`
的错觉：

`只要 stronger-request cleanup reactivation governor 已经把新的 truth 接进 current session，它就自动拥有了决定对象现在是否真的 connected、authenticated、capable、ready-for-use 的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 在 MCP 线已经成熟写出的 readiness 正对照，
会发现它其实明确拆开了五件事：

1. `state machine`
2. `surface grammar`
3. `consumer gate`
4. `runtime revocation`
5. `operator intent`

最硬的证据至少有七组：

1. `src/services/mcp/types.ts:179-226`
   的 `MCPServerConnection` 联合类型把 client 明确拆成
   `connected / pending / needs-auth / failed / disabled`
2. `src/services/mcp/useManageMCPConnections.ts:149-153,765-853`
   说明 `/reload-plugins` 之后不是直接 ready，而是重新进入 `pending` lifecycle
3. `src/services/mcp/useManageMCPConnections.ts:1109-1123`
   说明 even after explicit enabling，client 也仍要先进入 `pending`
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-80`
   只把 `failed / needs-auth` 当成 readiness 降级 surface，
   而不是把 active truth 直接冒充成 ready truth
5. `src/commands/plugin/ManagePlugins.tsx:513-519`
   与
   `src/cli/handlers/mcp.tsx:26-35`
   继续把 `connected / pending / needs-auth / failed / disabled` 分开显化
6. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:78-95`
   对 downstream tool 明确要求 `client.type === 'connected'`
7. `src/services/tools/toolExecution.ts:1599-1628`
   会把一个原本 `connected` 的 client 在运行时降成 `needs-auth`

这说明：

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

和

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

仍然不是一回事。

前者最多能说：

`新的 truth 已经被接进当前世界。`

后者才配说：

`这个当前世界现在真的 ready，可以被后续 reader、tool 与 user-facing surface 当成可用事实继续消费。`

所以 `457` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层`

也就是：

`stronger-request cleanup reactivation governor 负责把 truth 接进 active world；stronger-request cleanup readiness governor 才负责决定这个 active world 现在是否真的可用。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-readiness-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 readiness manager，
而是在说：

1. repo 已经明确把 MCP 连接状态拆成 `connected / pending / needs-auth / failed / disabled`
2. repo 已经把 `activate`、`connected`、`needs auth`、`failed` 写成不同 surface
3. repo 已经让工具调用只接受 `connected`，而不是“刚重载过就算 ready”

第二条：

这里说的 `readiness governance`

不是在声称：

`只有完全无错误的系统才算 ready。`

当前更稳的说法只能是：

`readiness governance 首先回答的是下游 consumer 现在可不可以继续安全消费它，而不是回答系统是否永远不会再出错。`

第三条：

这里也不是把：

`connected`

单独当成全部 readiness grammar。

真正关键的是：

1. `pending` 还没闭合
2. `needs-auth` 是 readiness 未闭合
3. `failed` 是 readiness 失败
4. capability gate 仍会继续限制下游消费

第四条：

这里尤其不能过度声称：

`对象曾经连通过一次，就说明它现在仍 ready。`

当前更稳的说法只能是：

`readiness 可以被运行时证据持续撤销；它不是一次性 verdict。`

第五条：

这里也不能过度声称：

`用户看见 reload summary，就已经得到 capability proof。`

当前更稳的说法只能是：

`summary 只回答某次 reactivation choreography 已经运行，不自动回答所有 downstream consumer 都已经可以继续用。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-reactivation-governor signer 仍不等于 stronger-request cleanup-readiness-governor signer`

证据：

1. `MCPServerConnection` 联合类型直接把 client state 拆成 `connected / pending / needs-auth / failed / disabled`，
   说明 active world 已有对象，不等于对象已 ready
2. `useManageMCPConnections()` 在 `pluginReconnectKey` 触发后，会先把新 client 置为 `pending`，
   再尝试连接，说明 reactivation 之后还存在单独的 readiness 过渡层
3. `toggleMcpServer()` 就算是显式 enable，也仍先把对象写成 `pending` 再 reconnect，
   说明“重新接入”不自动等于“立即 ready”
4. `useMcpConnectivityStatus()`、`getMcpStatus()` 与 `/mcp` health 都把 `failed`、`needs-auth`、`pending` 单独显化，
   说明 repo 不把 active truth 偷写成 ready truth
5. `ReadMcpResourceTool` 明确要求 `client.type === 'connected'`，否则直接报错，
   说明可用性消费面只接受 readiness truth
6. `toolExecution.ts` 在 `McpAuthError` 时会把一个原本是 `connected` 的 client 降成 `needs-auth`，
   说明 readiness 不是一次性 verdict，而是可被运行时持续撤销的主权
7. `connected / pending / needs-auth / failed / disabled`
   在类型、通知、命令与工具面都被反复复用，
   说明 readiness grammar 不是 UI 装饰，而是跨平面制度事实

因此这一章的最短结论是：

`stronger-request cleanup reactivation governor 最多能说新的 truth 已经进入当前世界；stronger-request cleanup readiness governor 才能说这个当前世界现在到底能不能被继续使用。`

再压成一句：

`重新激活，不等于已经就绪。`

再压成更哲学一点的话：

`制度知道怎样把对象重新接回当前世界，不等于制度已经知道这个世界现在敢不敢继续信它。`

## 4. 第一性原理：reactivation governance 回答“这组 truth 是否已接入当前世界”，readiness governance 回答“这个当前世界现在能不能被继续消费”

从第一性原理看，
强请求清理重新激活治理与强请求清理就绪治理处理的是两个不同主权问题。

`stronger-request cleanup reactivation governor`
回答的是：

1. stale world 是否已被 current world 替换
2. 哪个时刻完成 Layer-3 swap
3. 哪些 active components 已经换入
4. 当前 session 是否已经接到新的 config truth
5. 不同 mode 下谁来触发 active takeover

`stronger-request cleanup readiness governor`
回答的则是：

1. 当前 client 是否已 `connected`
2. 是否仍在 `pending`
3. 是否进入 `needs-auth`
4. 是否已经 `failed`
5. 下游工具、资源读取和 health check 是否可以继续消费它

如果把这两层压成一句：

`反正已经 reload 了`

系统就会制造六类危险幻觉：

1. `active-means-ready`
2. `pending-means-soon-enough`
3. `auth-gap-is-incidental`
4. `connected-once-means-ready-now`
5. `activation-summary-means-capability-proof`
6. `visible-means-consumable`

所以这两层的哲学本质也不同：

`reactivation`

治理的是：

`这组 truth 是否已接入当前世界。`

`readiness`

治理的是：

`这个当前世界现在敢不敢继续被使用。`

如果再压缩一点，
两者的本质差别其实是：

1. `reactivation` 负责 `world is updated`
2. `readiness` 负责 `world is trusted for use`

一个 object 即便已经重新进入 current world，
也完全不等于 consumer 现在就该继续依赖它。

## 5. Claude Code 多重安全技术的先进性：它把 ready-for-use 判定写成跨平面的硬门，而不是把 reload 结果冒充 capability proof

Claude Code 在这一层真正先进的地方，
不是“只要对象已经在当前世界里，就默认为可用”，
而是：

`它把 ready-for-use 判定写成类型、通知、CLI、工具与运行时降级共同维护的硬边界。`

这种先进性至少体现在六个方面：

### 5.1 typed state machine

readiness 先在类型层被表达，
不依赖后续调用方自由解释。

### 5.2 pending is first-class

重新接入后的过渡层不是被隐藏，
而是被正式保留为制度状态。

### 5.3 consumer-side hard gate

`ReadMcpResourceTool` 只接受 `connected`。
这意味着 readiness 不是提示，而是能力消费前提。

### 5.4 runtime revocation exists

`connected` 不是永久通行证；
新证据可以把它撤回成 `needs-auth`。

### 5.5 surface grammar stays honest

UI 和 CLI 都不会把 `pending / needs-auth / failed` 压成一条模糊“已激活”。

### 5.6 operator intent is weaker than proof

显式 enable 也只能把对象送回 `pending`，
不能直接把它送进 `connected`。

这些设计背后的源码思路也很清楚：

1. 不让 `reactivated` 越级冒充 `ready`
2. 不让 `visible` 越级冒充 `consumable`
3. 不让 `connected once` 越级冒充 `connected now`
4. 不让 `operator wanted it` 越级冒充 `runtime has proved it`
5. 不让 `activation summary` 越级冒充 `capability proof`

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. 旧 cleanup truth comeback 后，不能因为它已进入 active world 就宣称 downstream 现在可继续用它。
2. readiness 需要独立状态机，而不是由 active status 兼任。
3. pending 应是正式制度状态，而不是实现细节。
4. consumer gate 必须只接受更强 readiness fact。
5. runtime revocation 需要被设计进系统，而不是假定 ready verdict 永不失效。
6. 就绪判断的核心不是“对象现在是否在场”，而是“对象现在是否足够可信以支撑继续行动”。

## 6. 为什么 stronger-request cleanup 现在仍停在“active-world takeover grammar”，而不是“ready-for-use adjudication grammar”

和这些成熟 readiness 正对照相比，
stronger-request cleanup 当前最明显的缺口已经不是：

`这组 truth 何时真正接管 current world。`

而是：

`接回来之后此刻到底敢不敢继续用。`

更具体地说，
现在仍没有哪一层正式决定：

1. 旧 startup wording reactivated 之后何时算 ready、何时仍算 pending
2. 旧 cleanup law reactivated 之后哪些状态等价于 `needs-auth` 或 `failed`
3. 旧 promise vocabulary reactivated 之后哪些 downstream reader 现在可以继续消费
4. 旧 receipt objects reactivated 之后 readiness verdict 能否被运行时证据撤销
5. 哪些 user-facing surface 只配宣布 active，哪些才配宣布 ready

换句话说，
stronger-request cleanup 当前虽然已经开始显露 reactivation-governance，
却还没有长出足够正式的：

1. ready-state machine
2. readiness surface grammar
3. consumer hard gate
4. runtime revocation rule
5. operator-intent / proof distinction
6. active-vs-ready separation

这意味着它现在能说的是：

`新的 truth 已经接回当前世界。`

却还不能更强地说：

`这个当前世界现在足够可信，可以继续被依赖。`

## 7. 苏格拉底式自反诘问：我是不是又把“对象已经重新接进当前世界”误认成了“它现在一定能用”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 reactivation governance 已经等于 readiness governance，为什么 `pluginReconnectKey` 触发后系统还要先把 client 加成 `pending`，而不是直接视为 ready？
   因为“truth 已接进 current world”与“现在可以继续消费”根本不是同一句制度。
2. 如果当前世界一旦 reactivated 就已经 ready，为什么 `ReadMcpResourceTool` 还要坚持 `client.type === 'connected'` 的 hard gate？
   因为 downstream consumer 只接受 readiness truth，不接受 mere reactivation。
3. 如果 readiness 是一次性 verdict，为什么 `toolExecution.ts` 还会把一个原本 `connected` 的 client 降成 `needs-auth`？
   因为 readiness 会被运行时证据持续撤销，它不是一张永久通行证。
4. 如果用户能在 UI 里看见对象，是不是就说明它已经足够可信？
   不是。visible 只是 current-world presence，不是 capability proof。
5. 如果用户明确执行了 enable，是不是就说明系统该立刻相信它？
   不是。operator intent 只能触发重新证明流程，不能越级签出 ready verdict。
6. 如果 cleanup 线现在还没有显式 readiness 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确 grammar，越容易把“已经接回 current world”偷写成“现在可以继续信”。

这一串反问最终逼出一句更稳的判断：

`readiness 的关键，不在对象有没有回到当前世界，而在系统能不能正式决定这个当前世界现在是否足够可信以继续被使用。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 reactivation governance，就已经足够成熟。`

而是：

`Claude Code 在 MCP readiness state machine、consumer hard gate、runtime revocation 与 honest status grammar 上已经明确展示了 readiness governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reactivation-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“如何把对象接回当前世界”，而是“接回来之后此刻到底敢不敢继续用”。`
