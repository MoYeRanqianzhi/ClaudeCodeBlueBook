# 安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层：为什么artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer

## 1. 为什么在 `257` 之后还必须继续写 `258`

`257-安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层`
已经回答了：

`即便 stronger-request cleanup metadata 已经存在，runtime reality 也仍然会因为 suppression/deletion 分裂、deferred housekeeping、validation skip、plans executor propagation gap、diagnostics uncovered gap 与 receipt gap 而不自动符合。`

但如果继续往下追问，
还会碰到下一层更危险的错觉：

`只要这一次 runtime 看起来已经 conform，就等于系统已经拥有了长期 anti-drift verification。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看 repo 自己最成熟的 verifier 正对照，
会发现它其实已经展示出至少三种不同的反漂移技术：

1. `src/services/compact/microCompact.ts:32-36`
   直接把 derived truth 与 source truth 的一致性绑到 test culture 上
2. `src/bootstrap/state.ts:456-466`
   直接把 `sessionId` 与 `sessionProjectDir` 做成 atomic anti-drift structure
3. `src/utils/permissions/getNextPermissionMode.ts:11-17` 与 `src/utils/permissions/permissionSetup.ts:1078-1157`
   直接把 stale cache vs live truth divergence 提升成 runtime re-verification

与这些正对照相比，
stronger-request cleanup 当前可见的仍主要是：

1. `src/utils/sessionStorage.ts:954-969` 的 suppress guard
2. `src/utils/backgroundHousekeeping.ts:31-80` 的 delayed admission / reschedule
3. `src/utils/cleanup.ts:575-597` 的 skip guard 与 executor dispatch
4. `src/utils/plans.ts:79-110` 与 `src/utils/cleanup.ts:300-303` 的 propagation drift 样本
5. `src/utils/cleanup.ts:33-45` 的 local `CleanupResult`
6. `src/utils/diagLogs.ts:11-20,27-60` 的 uncovered diagnostics writer contract

这些都很重要，
但它们回答的仍主要是：

`这一次 runtime 做了什么、没做什么、为什么没做。`

它们还没有回答：

`如果以后 metadata、scheduler、executor、coverage set 与 receipt surface 再次分叉，谁会第一时间把这种分叉抓出来，并阻止系统继续把局部符合冒充成长期可信。`

所以 `257` 之后必须继续补的一层就是：

`安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层`

也就是：

`stronger-request cleanup-runtime-conformance governor 最多只能说“当前看起来已兑现”；stronger-request cleanup-anti-drift verifier 才能说“如果以后再次漂移，系统会主动发现”。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer。`

这里的 `artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer` 仍是研究命名。
它不是在声称 repo 里已经有一个现成的 `verifyCleanupPolicyAlignment()` 只是没人接线，
而是在说：

1. repo 其他区域已经明确展示出成熟 verifier 文化
2. stronger-request cleanup 线当前还没看到同等级 verifier plane
3. 因而“这次 conform”与“未来再漂移时谁报警”已经是两个问题

第二条：

这里说的 `anti-drift verification`
不是在声称：

`当前 stronger-request cleanup 一定已经发生严重回归。`

真正要说的是：

`当前源码已经足以证明 stronger-request cleanup 存在 temporal drift、propagation drift、coverage drift 与 receipt drift 的结构机会，但还没有看到与这些风险对称的显式 verifier grammar。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部漂移问题。

真正关键的是：

1. `cleanupPeriodDays=0` 暴露 declaration truth 与 execution truth 可分裂
2. housekeeping admission 暴露 timing truth 可分裂
3. `plansDirectory` 暴露 cross-plane propagation truth 可分裂
4. diagnostics 暴露 coverage truth 可分裂
5. `CleanupResult` 暴露 outcome truth 与 receipt truth 可分裂

第四条：

这里尤其不能过度声称：

`skip guard` 已经等于 anti-drift verifier。

当前更稳的说法只能是：

`skip guard` 只是在某些危险条件下选择不执行，
它回答的是 safe abstention，
不是 drift detection。

第五条：

这里也不能过度声称：

`anti-drift verifier` 只能长成 test。

当前更稳的说法只能是：

`Claude Code 已经展示出三种 verifier 形态：source-of-truth test、atomic anti-drift structure、live re-verification against divergence。`

真正的问题不是形式单一，
而是 stronger-request cleanup 当前还没把其中任一形态正式接上。

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-runtime-conformance-governor signer 仍不等于 stronger-request cleanup-anti-drift-verifier signer`

证据：

1. repo 已明确展示 source-of-truth verifier 正例
   `microCompact.ts:32-36`
2. repo 已明确展示 atomic anti-drift structure 正例
   `state.ts:456-466`
3. repo 已明确展示 stale-vs-live runtime re-verification 正例
   `getNextPermissionMode.ts:11-17`; `permissionSetup.ts:1078-1157`
4. stronger-request cleanup 当前只有 suppress / delay / skip / dispatch cluster，而没有对等 verifier surface
   `sessionStorage.ts:954-969`; `backgroundHousekeeping.ts:31-80`; `cleanup.ts:575-597`
5. plans family 仍持续暴露 strongest propagation drift
   `plans.ts:79-110`; `cleanup.ts:300-303`
6. diagnostics family 仍持续暴露 uncovered coverage drift
   `diagLogs.ts:11-20,27-60`; `cleanup.ts:575-595`
7. `CleanupResult` 说明 outcome 已可计数，但仍未升格为 machine-comparable family receipt
   `cleanup.ts:33-45`
8. 当前可见源码命名面里看不到 `verifyCleanup*` 或 `reverifyCleanup*` 这类 stronger-request cleanup live verifier surface

因此这一章的最短结论是：

`stronger-request cleanup-runtime-conformance-governor signer 最多能说“这次运行看起来已经兑现”；stronger-request cleanup-anti-drift-verifier signer 才能说“如果将来再偏离，系统会主动发现并拒绝继续沉默”。`

再压成一句：

`一次符合，不等于长期可信。`

## 4. 第一性原理：runtime-conformance 回答“此刻是否成立”，anti-drift verification 回答“偏离时谁会报警”

从第一性原理看，
这两层治理处理的是两种完全不同的安全问题。

`runtime-conformance`

处理的是：

1. 这次 stronger-request cleanup 当前是否执行
2. 没执行时是否被明确 skip
3. skip 是 deliberate safety 还是 accidental miss
4. executor 当前是否消费了正确 truth
5. 当前 outcome 有没有被保存

`anti-drift verification`

处理的则是：

1. 如果明天 helper 改了而 executor 没跟上，谁会失败得足够响
2. 如果 metadata 被某个 plane 消费、另一个 plane 忘记跟进，谁会主动报警
3. 如果 runtime 因为异步、缓存或切换而持有 stale truth，谁会 fresh read 并修正
4. 如果系统因为 safe abstention 而没做事，谁来区分“故意不做”与“悄悄漂移”
5. 如果 family outcome 迟早要被长期信任，谁来把它投影成 machine-comparable receipt

所以这两层的哲学本质不同：

`runtime-conformance` 是对当前事实的回答；
`anti-drift verification` 是对未来谎言的预防。

Claude Code 在这方面真正先进的地方，
不是“它有一个万能安全开关”，
而是：

`它已经知道不同 drift class 需要不同 verifier grammar。`

这就是它的技术哲学本质：

1. derived truth 漂移，用 test 抓
2. pairwise state 漂移，用 atomic structure 封死
3. stale cache vs live truth 漂移，用 runtime re-verification 修正

这不是抽象审美，
而是一套对失真类型做分类治理的工程认识论。

## 5. 从技术角度看，Claude Code 的多重安全技术先进性不在“多”，而在“不同失真用不同证明方式”

很多系统的安全控制只会重复一种动作：

`更多判断、更多 guard、更多 deny。`

Claude Code 更成熟的地方在于，
它没有把所有安全问题都压成同一种检查器，
而是按失真来源选择证明方式。

### 5.1 `microCompact`：当风险是“派生真相偏离源真相”，最优技术不是运行时猜，而是 source-of-truth equality test

`src/services/compact/microCompact.ts:32-36` 的注释很短，
但技术含量很高：

`Drift is caught by a test asserting equality with the source-of-truth.`

它意味着：

1. 系统承认 derived truth 会漂
2. 系统承认漂移风险应该被预先制度化
3. verifier 不靠运行时偶遇症状，而靠 source truth 对 derived truth 的持续校验

这是测试型 anti-drift verifier。

### 5.2 `switchSession`：当风险是“两份状态必须同时变化”，最优技术不是事后校验，而是结构性原子绑定

`src/bootstrap/state.ts:456-466` 直接说：

`sessionId` 与 `sessionProjectDir` `cannot drift out of sync`

这里没有再额外发明一层观察者，
而是直接把 drift 的表达机会从结构上拿掉。

这是结构型 anti-drift verifier。

### 5.3 `verifyAutoModeGateAccess`：当风险是“stale cache 与 live truth 会动态分叉”，最优技术不是静态配置，而是 fresh read + current-context transform

`src/utils/permissions/getNextPermissionMode.ts:11-17`
明确承认：

`cached isAutoModeAvailable` 与 live `isAutoModeGateEnabled()` 会 diverge

`src/utils/permissions/permissionSetup.ts:1078-1157`
更进一步把 fresh read、circuit breaker 更新与 current-context transform 绑在一起，
避免异步等待期间把旧模式写回当前世界。

这是运行时重验证型 anti-drift verifier。

### 5.4 对 stronger-request cleanup 的技术启示

这三种正对照说明：

`Claude Code 已经不是“会不会做 verifier”的问题，而是“cleanup 线还没把正确的 verifier grammar 接上”。`

所以 stronger-request cleanup 当前的不足，
不是文化不足，
而是 verifier 分配不足。

## 6. stronger-request cleanup 当前可见的是“符合性控制簇”，还不是“反漂移验证簇”

当前 stronger-request cleanup 这条线已经拥有不少安全控制：

1. `shouldSkipPersistence()` 会在 `cleanupPeriodDays=0`、`--no-session-persistence`、测试环境等条件下 suppress transcript writes
   `sessionStorage.ts:954-969`
2. `backgroundHousekeeping` 会 defer housekeeping，并在 recent interaction 下持续后延
   `backgroundHousekeeping.ts:31-80`
3. `cleanupOldMessageFilesInBackground()` 会在 settings validation errors + explicit `cleanupPeriodDays` 时主动 skip cleanup
   `cleanup.ts:575-584`
4. `CleanupResult` 说明局部 outcome 已可计数
   `cleanup.ts:33-45`

这些控制都是真实安全设计，
而且并不低级。

但它们回答的仍是：

`执行要不要发生、什么时候发生、为什么没发生、发生后怎样局部计数。`

它们没有回答：

`如果 stronger-request cleanup truth 在多个 plane 之间再次分叉，谁会把这种分叉从“研究者需要自己读源码才能看见”提升成“系统自己会显式发现”。`

`plansDirectory` 就是最硬的反例样本。

`src/utils/plans.ts:79-110`
已经把 plans truth 提升成正式 metadata knob，
并要求 project-root-contained path。

但 `src/utils/cleanup.ts:300-303`
仍让 `cleanupOldPlanFiles()` 只扫默认 `~/.claude/plans`。

`diagLogs.ts`
则给出另一种更尖锐的样本：

1. diagnostics carrier 明确存在
2. writer contract 与 no-PII discipline 都清楚
3. 但 visible cleanup dispatcher 根本不覆盖这一家

这说明 stronger-request cleanup 当前最危险的不是：

`系统没有 policy。`

而是：

`policy 已有，局部 plane 已消费，但还没有 verifier 负责持续检查 executor plane 与 covered-family set 是否仍与 policy 同步。`

## 7. 用苏格拉底式反问再压一层

1. 如果 stronger-request cleanup runtime-conformance 已经等于 anti-drift verification，为什么 repo 还要在 `microCompact` 明确把 drift 绑到 source-of-truth test 上？
   因为“一次运行看起来没问题”从来不等于“未来不会悄悄变坏”。
2. 如果 cleanup 线已经拥有对等 verifier，为什么 `plansDirectory` 这种最明显的 cross-plane drift 到今天仍主要靠人工阅读源码识别？
   因为 conformance symptom 已经显影，但 verifier grammar 还没有接线。
3. 如果 delayed housekeeping 与 validation skip 已经足够成熟，为什么它们不能自动升级成 anti-drift verification？
   因为 scheduling discipline 与 safe abstention 只回答“现在做不做”，不回答“以后漂了谁来抓”。
4. 如果 `CleanupResult` 已经存在，为什么它还不能自动充当 verifier receipt？
   因为 local count 不自动等于 machine-comparable family receipt。
5. 如果 diagnostics family 连 visible cleanup coverage 都没有，为什么还能假装 cleanup 线已经被长期 anti-drift-safe？
   因为 coverage blind spot 本身就是 drift class，而不是小尾注。

## 8. 硬结论

这组源码真正说明的不是：

`stronger-request cleanup 还没有 verifier，只是因为 Claude Code 不懂怎样做 verifier。`

而是：

`Claude Code 明明已经在 microCompact、switchSession 与 verifyAutoModeGateAccess 上展示了测试型、结构型与运行时重验证型三种成熟 anti-drift 技术；stronger-request cleanup 线今天真正缺的，不是安全意识，而是把这三种 grammar 中至少一种正式接到 cleanup metadata、scheduler、executor、coverage set 与 receipt surface 之间。`

因此：

`artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer。`

换句话说：

`当前 repo 已经证明自己会做反漂移验证；它尚未证明 stronger-request cleanup 这条线也已经被同等级验证文化正式接管。`
