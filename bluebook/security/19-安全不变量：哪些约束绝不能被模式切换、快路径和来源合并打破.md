# 安全不变量：哪些约束绝不能被模式切换、快路径和来源合并打破

## 1. 为什么在“检测链”之后还要写“不变量”

`security/18` 已经回答了一个问题：  
Claude Code 靠哪些检测链识别危险模式、路径语义漂移、外部入口越权和来源污染。

但如果继续往下问，一个更本质的问题会出现：

`这些检测链背后，到底有哪些绝不能被局部优化打破的约束？`

这就是“不变量”的意义。

很多系统不是败在没有安全机制，  
而是败在下面这些局部优化慢慢掏空了边界：

1. 为了方便，把 mode 做成万能 override
2. 为了性能，把 fast-path 放到真正的安全检查之前
3. 为了兼容，把低信任来源也并进高风险主链
4. 为了容错，让坏配置把整份配置面一起拖死
5. 为了易用，把读取权限、连接权限、写权限和身份权限混成一个“能用”

Claude Code 的成熟之处，在于它已经在源码里隐含了一组相对稳定的不变量。

## 2. 最短结论

如果把这套安全设计进一步压缩，Claude Code 真正守住的不是“某条规则”，  
而是六条更深的不变量：

1. 结构性危险高于当前模式便利
2. 来源合法性高于配置值本身
3. 语义等价高于文本等价
4. 外部能力必须按能力语义治理
5. 局部坏值不应毒化全局配置
6. 解释能力不是附属 UX，而是边界的一部分

理解了这六条，  
前面很多具体源码设计就会变得非常清楚。

## 3. 第一条不变量：结构性危险高于当前模式便利

这一条是 Claude Code 权限系统最硬的底层原则。

### 3.1 证据在哪里

最关键的证据链在：

- `src/utils/permissions/permissionSetup.ts`
- `src/utils/permissions/permissions.ts`
- `src/utils/permissions/pathValidation.ts`

具体表现为：

1. `permissionSetup.ts` 会先识别会绕过 classifier 的危险 allow rule
2. `permissions.ts` 把 `safetyCheck` 放在 bypass 和多数 fast-path 之前
3. `pathValidation.ts` 把路径级安全检查放在 working directory / `acceptEdits` 之前

### 3.2 这条不变量的真实含义

它不是说 mode 不重要，  
而是说：

`mode 只能在结构性危险之外调节便利度，不能推翻结构性危险本身。`

这意味着：

1. bypass 不是万能许可
2. auto mode 不是万能 classifier
3. 工作区路径不自动等于安全路径
4. allow rule 也不能抹掉某些更高阶危险

### 3.3 为什么这条不变量很高级

很多系统把“当前模式”做成最大的开关，  
看起来简洁，实际上很脆。

Claude Code 的做法更成熟，因为它知道：

`真正的边界不应该随着当前 UX 模式一起松动。`

## 4. 第二条不变量：来源合法性高于配置值本身

这是整个安全专题里最值得反复强调的一条。

### 4.1 证据在哪里

最关键的证据链在：

- `src/utils/managedEnv.ts`
- `src/utils/settings/settings.ts`
- `src/utils/settings/pluginOnlyPolicy.ts`
- `src/services/mcp/config.ts`

这些模块反复表达同一个思想：

1. 不是所有 settings source 都能定义同样的运行时
2. 不是所有 env 都有资格进入高风险主链
3. 不是所有 customization surface 都属于默认用户自治空间
4. 对于高风险 MCP allowlist，managed source 可以独占定义权

### 4.2 这条不变量的真实含义

很多配置系统默认先问：

`这值是什么？`

Claude Code 更常先问：

`这值是谁写的，它有没有资格写这个值？`

这就是“来源先于值”的真正技术版定义。

### 4.3 为什么这条不变量很重要

因为在高风险主链上，  
低信任来源最常见的攻击方式不是写出明显恶意值，  
而是：

`悄悄把本来不该由自己决定的值改掉。`

比如：

1. provider routing
2. SSH/host 运行时关键 env
3. enterprise 受管 MCP allowlist
4. plugin-only customization surface

所以真正的安全设计不是“过滤更多坏值”，  
而是先做来源主权治理。

## 5. 第三条不变量：语义等价高于文本等价

这是 `security/18` 里最技术化、也最容易被低估的一层。

### 5.1 证据在哪里

主要证据在：

- `src/utils/permissions/pathValidation.ts`
- `src/services/mcp/config.ts`

`pathValidation.ts` 关心的是：

1. shell expansion
2. tilde 变体
3. UNC
4. glob
5. symlink / canonical path

`mcp/config.ts` 关心的是：

1. stdio server 的命令签名
2. remote server 的 URL 模式
3. CCR proxy URL 还原后的真实 vendor URL
4. 去重时真正比较的是底层连接签名，不是显示名称

### 5.2 这条不变量的真实含义

系统不满足于“字符串看起来一样或不一样”。  
它更关心：

`这两个表述在执行层、连接层或文件系统层是不是同一个对象。`

这就是为什么：

1. 路径要 realpath / canonicalize
2. 写 glob 会被拒绝
3. CCR proxy URL 要 unwrap
4. MCP server 不能只按名字去治理

### 5.3 为什么这条不变量重要

因为很多越权和绕过都不发生在表面文本层，  
而发生在：

`文本被 shell、文件系统、连接代理或宿主解释之后。`

如果系统只做文本比较，  
就无法真正守住边界。

## 6. 第四条不变量：外部能力必须按能力语义治理

Claude Code 不把外部入口视作一个统一大类。  
这点非常关键。

### 6.1 证据在哪里

最关键的证据在：

- `src/tools/WebFetchTool/preapproved.ts`
- `src/utils/mcpValidation.ts`
- `src/services/mcp/config.ts`

它们分别治理的是：

1. WebFetch 的读取例外
2. MCP 输出预算与截断
3. MCP 连接定义、allow/deny 与主权来源

### 6.2 这条不变量的真实含义

Claude Code 实际上在说：

`读文档、定义连接、承接巨大输出、拥有写网络或身份语义，不是同一种能力。`

因此它不会因为一个 host 对读操作可接受，  
就自动把同一 host 升级成：

1. 任意外联
2. 任意输出注入
3. 任意连接定义来源

### 6.3 为什么这条不变量先进

因为很多系统会犯一个常见错误：

`把“外部能力”打包成一个统称。`

一旦打包，  
安全边界就只能跟着最粗粒度治理。  
Claude Code 明显更细。

## 7. 第五条不变量：局部坏值不应毒化全局配置

这一条体现的是系统韧性，而不是表面“宽容”。

### 7.1 证据在哪里

主要证据在：

- `src/utils/settings/validation.ts`
- `src/utils/settings/settings.ts`

`filterInvalidPermissionRules()` 和 `parseSettingsFile()` 的组合说明：

1. 单个坏 permission rule 会被局部过滤并记录 warning
2. 系统尽量避免让整份 settings 因一条坏规则整体失效
3. 只有在 schema 级无法继续时，才整体拒绝

### 7.2 这条不变量的真实含义

好的安全系统不应该轻易把自己搞成脆系统。  
否则攻击者或误配置者只要制造一个局部坏值，  
就能把整个配置面拖进异常状态。

Claude Code 的做法更稳：

`尽量把坏值隔离在局部，而不是让全局一起中毒。`

### 7.3 为什么这条不变量值得专门写

因为它说明 Claude Code 的安全观不是“越硬越好”，  
而是：

`在不损坏主边界的前提下，尽量维持系统可运行、可诊断、可恢复。`

这是工程成熟度的重要标志。

## 8. 第六条不变量：解释能力是边界的一部分

这一条很容易被误解成产品文案，  
其实不是。

### 8.1 证据在哪里

主要证据在：

- `src/utils/permissions/shadowedRuleDetection.ts`
- `src/utils/permissions/permissionExplainer.ts`
- 以及前面已研究过的状态面相关组件

`shadowedRuleDetection.ts` 不是简单返回布尔值，  
而是给出：

1. 哪条规则被谁压住
2. 为什么不可达
3. 如何修复

`permissionExplainer.ts` 也不是胡乱生成提示，  
而是把工具、输入、上下文、风险等级、原因和风险摘要结构化。

### 8.2 这条不变量的真实含义

如果用户长期看不懂为什么被 ask、deny 或收口，  
那系统在产品层面就相当于没有边界。

因为用户会把它体验成：

1. 任意
2. 不可预测
3. 无法复盘
4. 难以协同

所以解释层不是“锦上添花”，  
而是边界能否被真实采用的一部分。

## 9. 六条不变量合起来意味着什么

把前面六条放在一起，就能看出 Claude Code 的安全哲学：

1. 它不把安全理解成“禁止更多”
2. 它把安全理解成“强能力在正确不变量之内流动”
3. 它不允许便利度、兼容层和局部优化偷偷改写这些不变量

因此它真正守住的，不是某条 deny rule，  
而是下面这条更深的控制逻辑：

```text
结构危险先收口
  -> 只有合法来源能定义高风险值
  -> 只有语义上仍然等价的对象才能沿用原先判断
  -> 外部能力按能力语义分别治理
  -> 局部坏值被隔离而非放大全局失效
  -> 结果必须能被用户和管理员解释
```

## 10. 这对平台构建者的启示

如果把这章抽象成给别的 Agent 平台的法则，我会保留六句：

1. 不要让 mode 成为最高主权来源。
2. 不要让 fast-path 先于结构性危险检测。
3. 不要只比较文本，要比较执行语义。
4. 不要把所有外部能力打包成同一种“插件风险”。
5. 不要让一条坏规则毒化整份配置。
6. 不要把解释层留到最后做。

这六句基本就是 Claude Code 安全系统里最值得迁移的骨架。

## 11. 第一性原理再压缩

如果回到第一性原理，可以把这些不变量再压成三条更底层的判断：

1. 边界必须先于能力
2. 主权必须先于配置
3. 语义必须先于文本

而所有检测链、状态机、收口策略和解释层，  
本质上都在服务这三条更底层的判断。

## 12. 苏格拉底式反思

### 问：这套系统最怕什么？

答：最怕的不是单个危险动作，而是某种局部优化把不变量慢慢掏空。

### 问：为什么“模式切换”会成为风险来源？

答：因为如果 mode 能直接压过结构性危险，安全边界就会退化成 UX toggle。

### 问：为什么“来源主权”要反复强调？

答：因为高风险链路里，最危险的往往不是值长得坏，而是本来没有资格定义这个值的人突然拥有了定义权。

### 问：为什么“语义等价”这么关键？

答：因为攻击者和误配置最常利用的，正是“看起来是 A，执行时其实变成 B”的缝隙。

### 问：如果还要继续增强，这套系统最该补什么？

答：最该补的是统一的不变量仪表盘，让用户能一眼看到：

1. 当前有哪些不变量正在起作用
2. 哪一次判断是被哪条不变量拦下的
3. 哪个快路径因为哪条不变量而失效
4. 哪个来源因为没有主权被剥离

这会让整套系统从“源码上成熟”走向“产品上成熟”。

## 13. 一句话总结

Claude Code 的安全之所以不是脆弱拼装件，是因为它背后有一组相当稳定的不变量：结构性危险不能被模式切换覆盖，来源合法性高于配置值，语义等价高于文本等价，外部能力必须按能力语义治理，局部坏值不能毒化全局，而解释层本身也属于边界；真正高级的地方，不是它拦得多，而是它知道哪些约束绝不能被任何局部便利打破。
