# 安全工程举证责任协议：为什么想把结论说得更强的人必须补齐缺失层证明，而不能把沉默与默认推定外包给读者

## 1. 为什么在 `122` 之后还必须继续写“举证责任协议”

`122-安全工程沉默与缺席语义` 已经回答了：

`文本命名、文件存在、脚本入口与自动化宿主是四层不同的存在，不能被混成同一种“有”。`

但如果继续追问，  
还会碰到一个更实务化、也更尖锐的问题：

`既然存在有层级，那么当有人想把结论升级到更强层级时，缺的那几层到底由谁负责补？`

这是一个很多文档都会逃避的问题。  
最常见的写法是：

1. 作者只展示自己手上已有的部分证据
2. 对缺失层保持沉默
3. 然后默认让读者自己脑补：
   “也许这些层只是暂时没展开”

但这在安全工程里是不成立的。

因为只要：

1. artifact 层缺失
2. entry 层缺失
3. host 层缺失

就不应由读者承担“帮你把缺失层补成立”的责任。

所以 `122` 之后必须继续补出的下一层就是：

`安全工程举证责任协议`

也就是：

`谁想把结论说得更强，谁就必须负责补齐对应层级的证明；不能把沉默、默认推定和存在跳跃外包给读者。`

## 2. 最短结论

对这份研究版 Claude Code 源码来说，  
最重要的写作纪律之一不是“尽量多列证据”，  
而是：

`结论升级的举证责任必须由升级主张者承担。`

当前源码给出的场景非常典型：

1. `README.md:167-174` 里有 `verify/*` 的 lexical existence
2. 但当前树里 `verify/*` 实际不存在
3. `package.json:7-11` 也没有 `verify/test` 入口
4. `.github dirs: 0`、`test/spec files: 0` 说明也没有显式宿主与测试骨架

在这种情况下，  
如果有人还想主张：

`仓内已存在某种 formal verify capability`

那举证责任就不在读者身上，  
而在提出这个更强主张的人身上。

所以这一章的最短结论是：

`安全工程里，强结论不是默认可说，而是默认需要补证。`

再压成一句：

`谁要升级结论，谁就要补齐缺层。`

## 3. 第一性原理：为什么举证责任不能外包

因为 stronger claim 本质上是在新增承诺。

例如：

1. 从“README 提到 verify”升级到“verify files 存在”
2. 从“files 存在”升级到“formal verify entry 存在”
3. 从“entry 存在”升级到“长期自动化宿主存在”

每升级一步，  
都在引入新的结构性承诺。

而任何新增承诺都意味着：

`有人必须为它补证。`

如果没有这条规则，  
系统就会默认：

1. 沉默 = 也许成立
2. 缺席 = 先不算反证
3. 命名 = 暂且视为制度

而这三条正是安全研究最危险的滑坡。

所以从第一性原理看，  
举证责任协议的作用就是：

`阻止 stronger claim 把缺失成本转嫁给读者。`

## 4. 当前仓库里的举证责任应该如何分配

## 4.1 若主张 “verify 只是没展开”，你必须补 artifact proof

当前已知：

1. `README.md:167-174` 只证明 lexical existence
2. `rg --files ... | rg 'verify/' => 0`
3. `find ... -path '*/verify/*' -type f => 0`

所以如果有人想说：

`verify 文件其实存在，只是当前分析没翻到`

那这句话的提出者就必须拿出：

1. 当前树中的真实文件路径
2. 对应内容
3. 与 README 命名之间的可核对映射

否则这只是：

`把缺失 artifact 的成本转嫁给读者。`

## 4.2 若主张 “verify 能力其实已制度化”，你必须补 entry proof

当前已知：

1. `package.json:7-11` 没有 `verify/test` script

所以如果有人想说：

`验证能力虽然没写在 README 这一段，但其实已经制度化`

那他至少必须补出：

1. 正式入口名
2. 调用方式
3. 它如何接入当前 script chain

否则这属于：

`把 formal entry 的举证责任外包给读者。`

## 4.3 若主张 “验证体系虽然没写在 package 里，但长期会自动执行”，你必须补 host proof

当前已知：

1. `.github dirs: 0`
2. `test/spec files: 0`

所以如果有人想说：

`虽然脚本和文件不全，但整体验证体系大概率有宿主`

那他必须补出：

1. 自动化宿主在哪
2. 它如何调用
3. 它如何持续承载

否则这就是：

`把 host existence 的举证责任外包给读者。`

## 4.4 若主张 “build/check/smoke 已足够代表验证”，你必须补 equivalence proof

这是最常见也最危险的一类偷换。

当前已知：

1. `check` 只签静态通过
2. `build` 只签 best-effort artifact
3. `smoke` 只签最小启动
4. `README` / `QUICKSTART` 还明确存在对象上限与能力上限 veto

所以如果有人还想说：

`现有 build/check/smoke 已经足够代表验证`

那他就必须补出：

1. 为什么这些局部信号足以替代 formal verify
2. 为什么缺失 tests/CI 不影响 stronger claim
3. 为什么对象上限与能力上限不再构成 veto

否则这仍然是：

`把 stronger claim 的证明成本转嫁给读者接受。`

## 5. 技术先进性：成熟系统会把 burden of proof 正式写出来

很多系统只定义：

1. 哪些证据算信号
2. 哪些信号能升级结论

但更成熟的系统还会定义第三层：

`谁负责补足升级所需的缺失层。`

这一步其实非常重要。  
因为没有 burden rule，  
系统就总会滑向：

`先大胆说，再让别人来反驳。`

但安全工程不是辩论赛。  
它要求的是：

`先补齐必要证明，再获得升级资格。`

从这个角度看，  
Claude Code 这份研究版源码最值得借鉴的一个思想，不只是 gate、多层证据、veto，  
而是它逼我们看到：

`只要缺失层还在，升级主张就天然带着未履行的举证债务。`

## 6. 苏格拉底式反思：当我准备把一句话说得更强时，我该先质问什么

可以先问八句：

1. 我现在要升级的是哪一层存在
2. 这层存在当前由谁签字
3. 这个签字人是否真的到场
4. 如果没到场，我是不是在让读者替我补缺
5. 我是不是把“可能有”写成了“可以先按有处理”
6. 我是不是把默认推定当成了正式证据
7. 若把所有缺层逐条列出，我的 stronger claim 还剩下什么
8. 如果我不能补证，最保守却仍然真实的表述应是什么

这些问题最终收敛到一条非常硬的原则：

`安全工程里，结论强度与举证责任严格成正比。`

## 7. 对下一代安全控制台的技术启示

如果把这一章继续产品化，  
最值得做成系统功能的不是更多提示文案，  
而是：

1. 显式显示当前 stronger claim 缺哪一层证明
2. 明确标注“谁负有补证责任”
3. 把默认推定改写成显式的“未证明”
4. 在 UI 上禁止从 lexical/artifact 直接跳到 entry/host 结论

这会让安全控制台从：

`展示信号`

升级成：

`管理举证责任`

## 8. 一条硬结论

对这份 Claude Code 研究版源码来说，  
真正成熟的写法不是：

`我先把结论说得更强，如果你不同意再来反驳，`

而是：

`如果我要把结论升级，我就先承担把缺失层补齐的举证责任。`

因为只有这样，  
这份蓝皮书才不会落入：

`作者说得很强，读者负责兜底`

而会继续升级成：

`谁主张、谁补证、谁负责结论升级成本的安全工程分析。`
