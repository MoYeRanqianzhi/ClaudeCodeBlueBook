# 安全恢复绿色词租约：为什么active、connected、ready都只能被视作可撤销租约，而不是稳定承诺

## 1. 为什么在 `63` 之后还要继续写“绿色词租约”

`63-安全恢复词法失效` 已经回答了：

`当更强的新证据出现时，旧的弱词必须被显式 invalidated，不能继续并存。`

但如果继续往下一层追问，  
还会出现一个更抽象、也更接近第一性原理的问题：

`为什么这些正向词本来就这么容易被撤销？`

如果只是把它理解成：

`系统实现里凑巧加了几条 remove 和 invalidates`

那这个结论仍然不够稳。  
更深的一层应该是：

`这些绿色词从一开始就不应被理解为永久承诺，而应被理解为可撤销租约。`

也就是说：

1. `active`
2. `connected`
3. `ready`
4. `installed`

这些词都不是一次说出就永久生效的真理。  
它们都必须持续依赖：

1. 当前条件仍然成立
2. 更强反证尚未出现
3. 更高层 signer 还没有收回它

所以在 `63` 之后，  
统一安全控制台还必须再补一条制度：

`绿色词租约。`

也就是：

`正向恢复词不是永久产权，而是带有撤销条件、时效条件和上层覆盖条件的可撤销租约。`

## 2. 最短结论

从源码看，Claude Code 的很多正向词其实都已经在按“租约”而不是“永久承诺”运作：

1. notification 的 `immediate` 提示会 timeout，到时自动退出当前前景  
   `src/context/notifications.tsx:80-104`
2. `useTasksV2.ts` 不会因为任务刚完成就立刻隐藏，而是先给一个 hide delay，再确认仍然 completed 才真正清掉  
   `src/hooks/useTasksV2.ts:123-170`
3. `bridgeStatusUtil.ts` 的 `active` 实际依赖 `sessionActive || connected`，一旦 `error` 或 `reconnecting` 出现就立刻被更强词覆盖  
   `src/bridge/bridgeStatusUtil.ts:123-140`
4. `PromptInputFooter.tsx` 明确把 failed 从 footer pill 撤出，不让弱状态词继续占位  
   `src/components/PromptInput/PromptInputFooter.tsx:173-189`
5. `/status` 里的 `Connected to ...` 与 `Installed ...` 本身也只是在当前聚合视图下暂时成立，并不带最终 resolved 承诺  
   `src/utils/status.tsx:53-85,97-114`

所以这一章的最短结论是：

`Claude Code 里的大多数绿色词，本质上都是临时授权给当前界面的可撤销租约，而不是永久真相。`

我会把这条结论再压成一句话：

`绿色词只能被续租，不能被一次性买断。`

## 3. 源码已经显示：很多正向词天然就是“有条件、可撤销、会到期”的

## 3.1 notification success/attention 词天然有 timeout 与被替换机制

`src/context/notifications.tsx:80-104` 显示，  
`immediate` notification 会：

1. 立刻抢占当前
2. 启动 timeout
3. 到点后退出当前前景

再加上 `invalidates` 和 `fold`，  
这说明 notification 层的正向词从来就不是“永久挂起”的。  
它们更像：

`当前回合的短期注意力租约`

只要时间过了、上下文变了或更强新词来了，  
这份租约就会被收回。

## 3.2 task 隐藏逻辑说明：就连“完成”也要先经过一个观望窗口

`src/hooks/useTasksV2.ts:123-170` 很关键。  
这里系统并没有在任务一旦全部 `completed` 时立刻把任务面板清空。  
它会：

1. 先启动 `hideTimer`
2. 过一段时间后再去重新检查
3. 只有 `allStillCompleted` 才真正 `resetTaskList()`

这是一种非常典型的租约思想：

`“已完成”不是一次观察就永久成立，而是需要在一个短窗口内持续成立，才值得被进一步隐藏。`

这条设计说明作者其实已经接受了下面这条公理：

`正向状态必须经得起短期撤销。`

## 3.3 bridge 的 `active` 更像运行中租约，而不是闭环结论

`src/bridge/bridgeStatusUtil.ts:123-140` 里，  
`Remote Control active` 的成立条件其实很宽：

`sessionActive || connected`

也就是说：

1. 只要还连着
2. 或者会话当前活跃
3. 它就会暂时被说成 `active`

但同一函数也说明：

1. `error` 出现时立刻改口为 `failed`
2. `reconnecting` 出现时立刻改口为 `reconnecting`

这正是租约而不是产权的典型表现：

`当前可以临时使用这个积极词，但它随时可能被上层条件撤销。`

## 3.4 footer 分流 failed，说明弱绿色词并没有稳定居留权

`src/components/PromptInput/PromptInputFooter.tsx:173-189` 里，  
作者明确写出：

`Failed state is surfaced via notification, not a footer pill.`

这句话的深层含义不是“UI 省一行”，  
而是：

`footer 上那类低占位正向词，本来就没有在高风险条件下继续占位的居留权。`

一旦更强风险出现，  
旧的弱绿色词必须把位置让出来。  
这本质上就是：

`租约到期或被收回`

## 3.5 `/status` 的连接词同样只是当前盘点租约

`src/utils/status.tsx:53-85,97-114` 里，  
无论是：

1. `Connected to ...`
2. `Installed ...`
3. `n connected`

它们都只是当前聚合视图下的暂时盘点结论。  
它们不附带：

1. 更高层 signer 的最终担保
2. 跨 surface 的永久优先级
3. “之后不会被撤销”的承诺

也就是说，  
这些词更接近：

`当前视角下可用的弱正向租约`

而不是：

`系统对未来稳定性的担保书`

## 4. 第一性原理：正向词之所以必须可撤销，是因为系统永远运行在未来会继续到来的世界里

如果从第一性原理追问：

`为什么成熟系统里的正向词不能被设计成永久承诺？`

因为控制面不是给过去下结论，  
而是在一个未来仍会继续产生新证据的系统里持续运行。

这意味着任何正向词都必须面对三类未来：

1. 更强反证可能马上出现
2. 当前条件可能随时消失
3. 更高层 signer 可能尚未真正到位

所以只要系统还活着，  
大多数绿色词都不可能拥有绝对永久性。  
它们天然就更像：

`在当前证据边界内暂时有效的语义租约`

## 5. 我给统一安全控制台的“绿色词租约模型”

我会把它压成四条规则。

## 5.1 每个绿色词都必须自带续租条件

例如：

1. 条件仍成立
2. 更强反证未出现
3. 上层 signer 未撤销

## 5.2 每个绿色词都必须自带撤租路径

也就是：

1. 被什么词覆盖
2. 被什么事件 invalidated
3. 被哪个上层 surface 收回

## 5.3 越低层的绿色词，租期越短

例如：

1. notification 级绿色词最短
2. summary / footer 级次之
3. higher signer 才能给更长租期的正向结论

## 5.4 resolved 是最接近长期租约的词，但也仍依赖更高层闭环

resolved 并不是随便任何 surface 都能签发的绿卡。  
它只能来自最严格的闭环。

## 6. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性在于它的很多局部实现已经默认正向词是可撤销的；真正值得继续系统化的，是把这种“可撤销租约”提升成统一设计语言。`

当前先进的地方：

1. timeout / invalidates / fold 已经天然体现短租约
2. hideTimer + recheck 体现了“完成也需续租”
3. footer 对 failed 的让位体现了撤租逻辑

当前仍待系统化的地方：

1. 不同绿色词的租期长短还没被统一命名
2. 续租条件与撤租条件还分散在局部实现里
3. 缺少一张正式的“绿色词租约表”

这对其他 Agent 平台构建者的直接启示有五条：

1. 把正向词看成 lease，而不是 property
2. 给每个正向词设计 renewal 条件
3. 给每个正向词设计 revocation trigger
4. 让低层绿色词默认短租
5. 只让最高层 signer 签发长期正向结论

## 7. 哲学本质

这一章更深层的哲学是：

`真正克制的系统，连自己的绿色词都不完全相信。`

很多系统之所以让人误判，  
不是因为它们有太多错误词，  
而是因为：

`它们把本该短租的绿色词，伪装成了永久承诺。`

Claude Code 当前源码里最值得肯定的一点，  
就是大量实现细节已经在反复提醒我们：

`正向词必须准备被收回。`

## 8. 苏格拉底式反思：这一章最需要追问什么

### 8.1 为什么用户会天然把绿色词当成永久承诺

因为大多数产品都默认绿色词意味着“已经稳定没事”。

### 8.2 为什么成熟控制面必须反着设计

因为现实系统永远会继续收到新证据，稳定性只能被暂时假设，不能被永久宣告。

### 8.3 真正危险的错误是什么

不是绿色词少一点，  
而是：

`系统把可撤销租约说成了永久产权。`


## 9. 结语

`63` 回答的是：新证据出现时旧词必须失效。  
这一章继续推进后的结论则是：

`这不是补丁式收词，而是因为绝大多数绿色词本来就应该被设计成可撤销租约。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不仅要会撤销旧解释，还要从一开始就把正向解释设计成可撤销的。`
