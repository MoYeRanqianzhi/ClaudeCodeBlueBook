# 安全工程联合签字协议：为什么更强结论必须由多类证据联立签发，而不能靠多个弱正向信号拼凑成假闭环

## 1. 为什么在 `119` 之后还必须继续写“联合签字协议”

`119-安全工程结论签字权` 已经回答了：

`每类源码证据都有严格的 claim ceiling，超过这个上限的句子就是 overclaim。`

但如果继续追问，  
还会碰到一个更难、也更重要的问题：

`如果单个证据都不够强，那么更强结论到底什么时候才允许成立？`

因为工程实践里最常见的自我欺骗，  
往往不是靠单条证据直接越权，  
而是靠：

`把多个弱正向信号堆在一起，伪装成一个强结论。`

例如：

1. `README` 说对象边界清楚
2. `check` 通过
3. `build` 成功
4. `smoke` 能跑

然后团队就很容易顺手写出一句：

`当前构建已经被验证。`

但这句话其实仍然是假的。  
因为这些证据虽然都是真的，  
却并没有形成：

`合法的联合签字。`

所以 `119` 之后必须继续补出的下一层就是：

`安全工程联合签字协议`

也就是：

`更强结论只有在满足明确的多证据联立条件、且没有被更高优先级缺口信号否决时，才允许升级。`

## 2. 最短结论

对这份研究版 Claude Code 源码来说，  
最危险的并不是单个错误句子，  
而是：

`多个真句子拼出一个假结论。`

从当前源码看，  
至少有四类常见联合签字：

1. `README + QUICKSTART` 可以联合签出“对象边界与能力上限”
2. `package.json + prepare-src + tsconfig + build.mjs` 可以联合签出“当前存在正式的 best-effort build/check pipeline”
3. `build + start/smoke` 可以联合签出“best-effort artifact 已存在且可最小启动”
4. `package missing verify + 无 tests + 无 .github` 可以联合签出“当前正式验证闭环缺失”

但它们仍然不能签出：

1. 官方等价构建完成
2. 运行时功能正确
3. 安全验证闭环已经成立

所以这一章的最短结论是：

`强结论不是“证据数量更多”，而是“签字结构合法”。`

再压成一句：

`不是所有证据相加都能升级；只有满足联合签字协议的证据，才有资格升级结论。`

## 3. 第一性原理：什么叫“联合签字”

联合签字不是把证据放在同一段里。  
它要求至少同时满足三件事：

1. 这些证据来自不同层，彼此互补，而不是同义重复
2. 它们共同支持的是同一个命题，而不是相邻命题
3. 没有更高优先级的 gap signal 正在否决这个升级

换句话说：

`联合签字 = 互补证据 + 同一命题 + 无否决信号。`

缺任何一项都不成立。

所以如果只有：

1. 多个都来自静态层的证据
2. 多个都只是“可跑”的证据
3. 多个都没回答“未证明部分”的证据

那它们并不构成 stronger claim，  
只构成：

`更热闹的弱证据集合。`

## 4. 当前仓库里哪些联合签字是合法的

## 4.1 对象边界联合签字：`README + QUICKSTART`

`README.md:72-74,190-193` 说明：

1. source is incomplete
2. missing modules 不可从公开 artifact 恢复
3. `feature()` 的 compile-time 机制导致公开产物缺失无法逆推

`QUICKSTART.md:3-5,89-103` 又说明：

1. full rebuild requires Bun
2. internal build configuration 不在公开包内

这两类证据来自：

1. 对象边界说明
2. 构建能力上限说明

它们互补，且共同支持同一命题：

`当前对象是 research-extracted、feature-incomplete、无法在公开仓内闭合成完整上游构建的研究版源码。`

这是一个合法联合签字。

## 4.2 best-effort pipeline 联合签字：`package.json + prepare-src + tsconfig + build.mjs`

`package.json:7-11` 签：

1. 正式入口存在
2. `build/check` 都依赖 `prepare-src`

`prepare-src.mjs:3-9,79-115` 签：

1. 语境正规化是必经前置门

`tsconfig.json:16-18,28-35` 签：

1. 当前静态边界的作用域

`build.mjs:52-170` 签：

1. copy / transform / entry / bundle 的实际 pipeline

这四类证据来自：

1. 正式入口
2. 前置语境归一化
3. 静态边界
4. 构建执行链

它们可以联合签出：

`当前仓库确实拥有一条被正式承认的、围绕 prepare-src 组织的 best-effort build/check pipeline。`

但仍然不能签出：

`这条 pipeline 已形成 formal verification closure。`

因为 verify 相关签字人仍然缺位。

## 4.3 artifact 可启动联合签字：`build + start/smoke`

`build.mjs` 可以签：

1. artifact 已生成

`package.json:start` 与 `QUICKSTART.md:43-45` 可以签：

1. artifact 可被最小启动

这两类证据可以联合签出：

`best-effort artifact exists and can minimally launch.`

这是一个合法升级。  
因为“产物存在”与“最小启动成立”是互补证据。

但它们仍然不能签出：

1. 完整功能正确
2. 权限/安全机制已验证
3. 行为与官方一致

所以这是：

`artifact + launch claim`

不是：

`validation claim`

## 4.4 缺口闭环联合签字：`package missing verify + 无 tests + 无 .github`

`package.json:7-11` 没有任何 verify/test script。  
本地检查显示：

1. `.github dirs: 0`
2. `test/spec files: 0`

这三类证据联合起来，  
可以合法签出：

`当前仓库内不存在正式的 verify/test/CI 闭环。`

注意这也是联合签字，  
只是它签的是：

`缺口成立`

而不是：

`能力成立`

这类联合签字非常关键，  
因为它能阻止前面的正向信号被误升级成闭环叙事。

## 5. 哪些联合签字是非法的

## 5.1 `README + smoke` 不是“官方等价构建”

很多人会想：

1. README 说明对象边界
2. smoke 证明产物能启动

于是得出：

`公开仓已经能较完整地还原官方构建。`

这是错的。  
因为对象边界说明与最小启动证据之间，  
仍然缺少：

1. 语义等价证明
2. 缺失模块补全证明
3. runtime 行为对照证明

所以这是非法联合签字。

## 5.2 `check + smoke` 不是“系统已验证”

`check` 签的是静态通过。  
`smoke` 签的是最小启动。

把这两者叠加，  
最多只能得到：

`当前静态视角下可通过，且 artifact 可最小启动。`

它仍然不等于：

1. 功能正确
2. 时序正确
3. 安全正确
4. 回归已被正式守住

所以 `check + smoke = verified` 是典型的非法升级。

## 5.3 `build + gap ignorance` 不是“工程化闭环”

很多团队最容易犯的错误是：

1. build 成功
2. output 能跑
3. 然后对 verify 缺失保持沉默

一旦 silence 被当成 neutral，  
系统就会自动滑向：

`工程化闭环已经基本成立。`

但 gap signal 不是中立项，  
而是 veto item。

只要：

1. 没有 verify script
2. 没有 tests
3. 没有 CI host

那么任何关于“闭环已完成”的升级都应被否决。

## 6. 技术先进性：真正先进的系统，不只定义 evidence ceiling，还定义 upgrade protocol

如果说 `119` 解决的是：

`单个证据最多能说到哪`

那么这一章解决的是：

`多个证据什么时候才允许一起说得更强。`

这一步其实更接近真正成熟的安全工程。

因为成熟系统不会只做：

1. 单点权限控制
2. 单点证据控制

它还会做第三层：

`升级控制`

也就是：

1. 哪些证据必须联立
2. 联立后最多升到哪一级
3. 哪些缺口信号拥有否决权

从这个角度看，  
Claude Code 这份研究版源码给出的技术启示是：

`安全工程真正难的不是收集信号，而是治理信号升级。`

## 7. 苏格拉底式反思：如果我想把一句弱结论升级成强结论，我该先问什么

可以先追问八句：

1. 这几份证据真的互补，还是只是同义重复
2. 它们共同支持的是同一个命题，还是几个相邻命题
3. 当前升级有没有被缺口信号否决
4. 这里是否还缺一个来自更高层的签字人
5. 我是不是把“能构建”偷换成了“能验证”
6. 我是不是把“可启动”偷换成了“可正确运行”
7. 如果删掉其中一份证据，这个更强结论还站得住吗
8. 如果换成更保守的句子，这些证据是否仍然完全够用

这些问题最终都会压缩到一句：

`更强结论不是默认可升级，而是默认不可升级，除非联合签字协议已经满足。`

## 8. 一条硬结论

对这份 Claude Code 研究版源码来说，  
真正严谨的安全工程方法不是：

`看到很多正向信号后自然觉得“差不多已经验证了”，`

而是：

`先给每个单项证据设 ceiling，再给多证据升级设联合签字协议，并把 gap signal 设成正式否决项。`

因为只有这样，  
这份研究才不会陷入：

`局部都对，整体却说得过满`

而会真正升级成：

`单项证据、联合证据与未证缺口三者同时受治理的安全工程分析。`
