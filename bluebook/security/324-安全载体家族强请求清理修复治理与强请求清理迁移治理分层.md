# 安全载体家族强请求清理修复治理与强请求清理迁移治理分层：为什么artifact-family cleanup stronger-request cleanup-repair-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-migration-governor signer

## 1. 为什么在 `323` 之后还必须继续写 `324`

`323-安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层`
已经回答了：

`stronger-request cleanup 线即便未来长出 repair governor，也仍需要独立主权去决定 drift 被发现后是修 scheduler、修 executor、补 coverage、补 receipt、加强 explainability，还是缩 wording / promise strength。`

但如果继续往下追问，
还会碰到下一层更容易被误写成“修完就结束”的错觉：

`只要 repair governor 已经决定怎么修，它就自动拥有把旧 stronger-request 世界迁到新世界的主权。`

Claude Code 当前源码研究仍然不能支持这种更强说法。

因为继续看 repo 自己最成熟的 migration 正对照，
会发现它其实已经把：

`fix direction`

和

`old world exit strategy`

拆成了不同层。

最硬的证据有六组：

1. `main.tsx` 直接维护 startup migration chain 与 `CURRENT_MIGRATION_VERSION`，说明 repo 不把“新规则更正确”偷写成“旧世界自然退场”  
   `src/main.tsx:325-345`
2. `migrateLegacyOpusToCurrent.ts` 明确只改 `userSettings`、保留 project/local/policy source、继续 runtime remap，并保存 timestamp 供 one-time notification  
   `src/migrations/migrateLegacyOpusToCurrent.ts:13-57`
3. `migrateReplBridgeEnabledToRemoteControlAtStartup.ts` 又说明旧 key 到新 key 的交接不是简单 rename，而要治理何时复制、何时删除旧 key、何时保持幂等  
   `src/migrations/migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-21`
4. `cacheUtils.ts` 的 `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days` 更进一步说明旧 plugin world 不会“新版本一来就立刻消失”；迁移治理要决定 grace window 与 phased retirement  
   `src/utils/plugins/cacheUtils.ts:23-24,69-105`
5. `plans.ts` 的 `copyPlanForResume()` / `copyPlanForFork()` 又把 continuity strategy 明确摆上台面：旧 plan artifact 如何被 reuse、recover、copy，而不是被粗暴抹平  
   `src/utils/plans.ts:164-255`
6. 与这些正对照相比，stronger-request cleanup 线当前虽然已开始显露 repair-governance，但还没有谁正式决定旧 startup-delete wording、旧 home-root cleanup law、旧 receipt / local-count truth 与旧 stronger-request promise 世界，是立即 cutover，还是应该 staged migration

这说明：

`artifact-family cleanup stronger-request cleanup-repair-governor signer`

和

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

仍然不是一回事。

前者最多能说：

`我们已经决定应该修哪一层。`

后者才配说：

`旧 law、旧 path、旧 coverage、旧 receipt、旧 wording 应怎样被带过渡、作废、保留兼容期，或者被重签发。`

所以 `323` 之后必须继续补的一层就是：

`安全载体家族强请求清理修复治理与强请求清理迁移治理分层`

也就是：

`stronger-request cleanup-repair governor 只能决定修什么；stronger-request cleanup-migration governor 才能决定旧世界怎样在不制造新谎言的前提下被平稳带离现场。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个 migration policy object 只差导出，
而是在说：

1. repo 在别的子系统里已明确展示出 migration governance 的存在
2. migration governance 回答的是 rewrite、remap、grace、notification 与 continuity
3. stronger-request cleanup 线今天若要真正成熟，也迟早要回答这些问题

第二条：

这里说的 `migration governance`

不是在声称：

`修复之后一律都要做批量重写。`

更稳的说法只能是：

`migration` 可能意味着 source rewrite、runtime remap、completion marker、one-time notification、orphan grace window 或 continuity-preserving copy / reuse / recovery；关键不在迁不迁，而在谁来决定旧世界怎样退场。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部迁移问题。

真正关键的是：

1. temporal drift 修完后，旧 wording 何时退场
2. propagation drift 修完后，旧 home-root law 是否还保留兼容窗口
3. coverage drift 修完后，旧 covered-family set 是否需要 grace / notice / dual-policy
4. receipt drift 修完后，旧 local truth 是否需要重签、dual-read 或 notice
5. stronger-request cleanup 线当前没有一处可见代码显式持有这些过渡决定

第四条：

这里尤其不能过度声称：

`migration governance` 一定意味着“旧世界马上消失”。

更稳的说法只能是：

`plugin orphan cleanup、model remap 与 plan continuity 已经说明：成熟系统更常见的做法恰恰是让旧世界按受治理的节奏退出，而不是被一次性粗暴抹平。`

第五条：

这里也不能过度声称：

`runtime remap` 就已经等于 migration governance 的全部。

更稳的说法只能是：

`runtime remap` 只是迁移语法之一；它仍需要与 source rewrite、grace window、notification 与 continuity policy 一起构成完整 migration grammar。`

第六条：

这里还不能把：

`migration`

偷换成：

`sunset`

因为 migration 的任务是让旧世界平滑过渡到新世界，
sunset 的任务才是决定兼容世界什么时候正式结束。

所以 migration 虽然比 repair 更强，
但仍然不是最后一层。

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-repair-governor signer 仍不等于 stronger-request cleanup-migration-governor signer`

证据：

1. `main.tsx` 直接维护 startup migration chain 与 migration version gate  
   `main.tsx:325-345`
2. `migrateLegacyOpusToCurrent.ts` 说明 model 修复之后还要继续治理哪些 source 被改、哪些 source 只 remap、是否通知用户  
   `migrateLegacyOpusToCurrent.ts:13-57`
3. `migrateReplBridgeEnabledToRemoteControlAtStartup.ts` 说明旧 key 到新 key 的过渡还要治理复制、删除时机与幂等  
   `migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-21`
4. `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days` 说明旧 plugin world 不是“新版本一来就立即消失”  
   `cacheUtils.ts:23-24,69-105`
5. `copyPlanForResume()` / `copyPlanForFork()` 说明 plan artifact 过渡不是单纯 repair，而要继续回答 resume reuse、missing-file recovery 与 fork continuity  
   `plans.ts:164-255`
6. stronger-request cleanup 当前虽已开始显露 repair-governance，但还没有谁正式决定旧 law、旧 path、旧 coverage、旧 receipt 与旧 wording 应怎样 phased exit
7. 因而 stronger-request cleanup 今天真正缺的不是又一个 fix，而是 repair 之后的 transition grammar

因此这一章的最短结论是：

`stronger-request cleanup-repair-governor signer 最多能说“该修哪里”；stronger-request cleanup-migration-governor signer 才能说“旧世界怎样有次序地退出，而不把兼容世界和新世界一起搞乱”。`

再压成一句：

`会修，不等于会迁。`

再压成更哲学一点的话：

`制度知道怎样改正当前世界，不等于制度知道怎样送走旧世界。`

## 4. 第一性原理：repair governor 回答“修哪层”，migration governor 回答“旧世界如何不被粗暴抹平”

从第一性原理看，
这两层治理处理的是两个不同时间面的主权问题。

`repair governor`

回答的是：

1. 应修 metadata、runtime、scheduler、executor、coverage、receipt 还是 promise
2. 哪条路径现在不可信
3. 哪类 drift 必须立即止血
4. 哪种 stronger claim 应先撤回
5. 哪一层对当前 bug 负主责

`migration governor`

回答的则是：

1. 旧值是立刻重写，还是先 runtime remap
2. 哪些 source 允许被改写，哪些必须保持原样
3. 兼容窗口多长
4. 旧缓存、旧目录、旧 wording 何时清理
5. 是否需要 one-time notification、completion flag、dual-read 或 continuity copy

所以这两层的哲学本质不同：

`repair`

属于纠偏主权：

`它回答“现在应把哪里拉回正确状态”。`

`migration`

则属于秩序交接主权：

`它回答“旧状态怎样体面让位，而不在交接中制造新的谎言”。`

Claude Code 在这方面真正先进的地方，
不是它“会在修完后顺手删掉旧东西”，
而是它已经在多个 subsystem 里明确展示：

`旧世界的退出本身也是正式治理对象。`

这是一种很成熟的安全哲学：

`更正当前真相，不等于有权立即抹掉旧真相。`

## 5. 从技术角度看，Claude Code 的先进性在于“迁移被做成多种 transition grammar，而不是单一 rewrite”

很多系统一谈迁移，
就只剩下一种动作：

`改旧值。`

Claude Code 更成熟的地方在于，
它会根据旧世界的类型选择不同的 transition grammar。

### 5.1 `runMigrations()`：迁移先被提升成 startup governance

`main.tsx:325-345`
最值钱的地方，
不是它调用了几段脚本，
而是它把迁移正式纳入：

1. startup path
2. version gate
3. ordered execution

这说明 repo 已明确承认：

`old world exit`

不是临时修补，
而是正式控制面的一部分。

### 5.2 `migrateLegacyOpusToCurrent()`：成熟系统不会把“更正新规则”直接写成“粗暴覆盖所有旧值”

`migrateLegacyOpusToCurrent.ts`
展示的不是简单“把旧模型改成新模型”，
而是：

1. 只重写 `userSettings`
2. 保留 project / local / policy source 不动
3. runtime 继续 remap
4. 记录 `legacyOpusMigrationTimestamp` 供 REPL 做 one-time notification

这说明 model migration governance 真正回答的是：

1. 哪些 source 被 rewrite
2. 哪些 source 只被 remap
3. 哪些用户需要显式 notice
4. 哪些旧 source 必须继续被尊重

而这些都不是 repair 自己能回答的。

### 5.3 old-key/new-key handoff：迁移治理并不等于“把旧字段替换掉”，而是决定新旧字段怎样交接

`migrateReplBridgeEnabledToRemoteControlAtStartup.ts`
非常值钱。

它不是暴力删旧 key，
而是：

1. 只有 old key 存在且 new key 未设置时才复制
2. 复制到新 key
3. 再删除 old key
4. 整个过程保持 idempotent

这说明 config-key migration 的技术本质不是 rename，
而是：

`handoff without double-write and without silent loss`

stronger-request cleanup 线未来如果修 startup-delete wording、
调整 `plansDirectory` cleanup world、
补 stronger receipt semantics、
或重划 covered-family covenant，
同样会遇到这类问题：

`旧 law、旧 knob、旧 wording、旧 contract 是双读一段时间，还是立即作废。`

### 5.4 `.orphaned_at` 与 7 天宽限：旧世界体面退出，往往意味着先被标记、再进入 grace window、最后才真正清理

`cacheUtils.ts`
最值钱的地方在于：

1. 旧版本先被标记 `.orphaned_at`
2. `CLEANUP_AGE_MS = 7 days`
3. 超过 7 天才真正删除
4. 重新安装时还会移除 stale marker

这里体现的就是最典型的：

`phased retirement grammar`

它对 stronger-request cleanup 的启示非常直接：

`旧 cleanup law、旧 path、旧 coverage、旧 receipt world 也未必应该立即蒸发；它们很可能需要自己的 orphan / grace / cutover strategy。`

### 5.5 plans continuity：有些 migration 甚至不是 rewrite，而是 continuity-preserving reuse / recovery / copy

`plans.ts` 的 `copyPlanForResume()` / `copyPlanForFork()`
说明：

1. resume 时复用原 slug
2. 文件缺失时先从 snapshot / message history 恢复
3. fork 时复制内容但换新 slug，避免 clobber

这说明 migration governance 有时并不是：

`rewrite old world`

而是：

`preserve continuity while changing lineage`

这对 stronger-request cleanup 尤其重要。

一旦 cleanup law 未来修到 plans / receipts / continuity surfaces，
就必须继续回答：

`旧 artifact 在 transition 期间怎样被 continue、怎样被 recover、何时停止被看见。`

## 6. 对 stronger-request cleanup 的技术启示：真正缺的不是 fix itself，而是 transition grammar

对 stronger-request cleanup 来说，
上一层 `repair governance` 主要还在回答：

1. temporal drift 要修 scheduler 还是修 wording
2. propagation drift 要修 executor 还是收窄 override
3. receipt/coverage drift 要补 receipt 还是补 covenant

但这一层已经必须继续追问：

1. 旧 startup-delete wording 何时停止作为 current promise
2. 旧 default home-root cleanup law 是否需要 grace 或 dual-read
3. 旧 `CleanupResult` local truth 是直接废弃、升格重签，还是兼容一段时间
4. uncovered diagnostics world 如果以后进入 cleanup covenant，是立即纳入还是 staged adoption

这些都不是“再多写一个修复函数”能回答的。

它们已经进入：

`migration governance`

层。

## 7. 苏格拉底式自反诘问：我是不是又把“知道该怎么修”误认成了“知道怎样让旧世界退场”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 repair scope 已经明确，为什么还要再拆 migration？  
   因为修法只决定新世界该怎样成立，不决定旧世界怎样退出。
2. 如果某个 drift 修完就能运行，为什么不能直接 cut over？  
   因为 source、cache、receipt、continuity 与用户意图可能仍生活在旧世界里。
3. 如果 plugin 线已有 orphan + grace window，cleanup 线是不是也必须照抄？  
   不能照抄。不同旧世界需要不同 transition grammar，但“是否需要 grace / continuity policy”必须被正式回答。
4. 如果某个过渡只需要 runtime remap，是否就不需要 migration governor？  
   仍需要。决定“只 remap 不 rewrite”本身就是迁移主权。
5. 如果 cleanup 线当前还没有显式迁移代码，是不是说明 migration 层还不值得写？  
   恰恰相反。越是缺这层显式语法，越容易在未来修复时把旧世界退场问题偷写成“自然发生”。
6. 如果新 law 明显更正确，旧 law 是否就应立即清零？  
   也不能这样推。更正确的新 law 不自动消灭兼容成本；迁移治理恰恰负责决定这些成本如何被显式承受。

这一串反问最后逼出一句更稳的判断：

`migration 的关键，不在系统会不会改旧值，而在系统能不能把“旧世界退场”本身治理成一个不制造新谎言的过程。`

## 8. 从安全设计哲学看，这一层真正回答的是“制度不仅会纠偏，而且配安排旧世界谢幕”

`repair governance`
更像制度会不会纠偏。

`migration governance`
更像制度配不配安排旧世界谢幕。

一个系统可以决定：

1. 哪里错了
2. 该修哪里
3. 该把哪一层拉回当前正确状态

但如果它没有继续决定：

1. 旧 law 何时失效
2. 旧 path 是否保留兼容窗口
3. 旧 receipt 是否需要 dual-read / re-sign / notice
4. continuity artifact 怎样在过渡期继续被体面消费

那么它仍然更接近：

`会修当前世界的制度`

而不是：

`会送走旧世界的制度`

Claude Code 当前 stronger-request cleanup 线，
恰好停在这个哲学分界点上：

1. 它已经不再只是“没决定怎么修”
2. 它已经开始能把 repair scope 问题单独治理
3. 但它还没有 equally-strong 地回答旧 stronger-request world 怎样 phased exit

所以这一章真正要钉死的，
不仅是一个工程判断，
也是一个更深的安全设计判断：

`制度知道怎样改正当前世界，不等于制度已经知道怎样送走旧世界。`

## 9. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 repair governor，就已经足够成熟。`

而是：

`repo 在模型、配置键、plugin orphan 宽限期与 plans continuity 上已经明确展示了 migration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-repair-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“修完”，而是“修完后旧 stronger-request 世界怎样被有秩序地带走”的最后一道治理设计。`
