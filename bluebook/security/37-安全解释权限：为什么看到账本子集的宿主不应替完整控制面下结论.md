# 安全解释权限：为什么看到账本子集的宿主不应替完整控制面下结论

## 1. 为什么在 `36` 之后还要继续写“解释权限”

`36-安全账本投影原则` 已经回答了：

`不同宿主通常只拿到账本子集。`

但如果继续往第一性原理再压一层，  
还会出现另一个更尖锐的问题：

`既然宿主只看到账本子集，它到底凭什么替完整安全控制面下结论？`

也就是说，  
真正需要被追问的已经不只是：

`它看到了什么`

而是：

`它有资格解释什么。`

这就是这一章的核心问题：

`Claude Code 为什么要求“解释权限”必须从账本子集和当前边界反推，而不能从“我接入了协议”直接推出？`

## 2. 最短结论

从源码看，Claude Code 的深层安全性不仅在限制写权限，  
也不仅在限制宿主视野，  
还隐含着一条更深的纪律：

`解释权限必须受可见账本边界约束。`

也就是说：

1. 看到了某一层，不等于能解释全部
2. 接入了某个协议，不等于有完整解释权
3. 更宽的宿主解释权更强，但依然受条件窗口限制
4. 窄宿主最需要的不是“大胆解释”，而是诚实承认未知

所以本章最短结论是：

`安全控制面里，解释权不是默认附赠品，而是从账本可见性、子集宽度和当前状态窗口推导出来的能力。`

## 3. 什么叫“解释权限”

这里说的“解释权限”，不是法律意义上的授权。  
在本专题里，它的定义是：

`某个宿主有资格就哪一层安全事实做出多大范围、多强确定性的解释。`

这至少包含三层：

1. 描述权限：我能描述当前我看到了什么
2. 推断权限：我能从局部事实推到多远
3. 裁决权限：我能不能替整套控制面宣布“系统现在就是这样”

成熟系统最容易出事的地方，  
往往不是没有数据，  
而是：

`局部读者越级行使了完整裁决权限。`

## 4. 为什么“接入了协议”不等于拥有完整解释权

## 4.1 `controlSchemas.ts` 给出的只是可接入全集，不是解释权全集

`src/entrypoints/sdk/controlSchemas.ts:157-308` 说明协议上可以有：

1. `mcp_status`
2. `get_context_usage`
3. 以及更多 control request

但这只能说明：

`系统允许存在这样的接口。`

它并不能说明：

`当前这个宿主真的实现了这些接口，也真的拿到了这些账本。`

所以从“schema 存在”直接推出“宿主能完整解释”本身就是错误推理。

## 4.2 `coreSchemas.ts` 的宽消息面同样不等于每个宿主都拥有同等解释能力

`src/entrypoints/sdk/coreSchemas.ts:1663-1750` 里有：

1. `auth_status`
2. `session_state_changed`
3. 以及很多 system message

这同样说明系统有丰富的解释材料，  
但并不说明：

`任何宿主都同时、完整、稳定地拿到了这些材料。`

所以“协议宽”与“解释权宽”绝不能直接画等号。

## 5. 默认 `StructuredIO` 宿主为什么不该替完整控制面下结论

## 5.1 它连恢复账和 internal-event 持久面都默认没有

`src/cli/structuredIO.ts:139-195` 已经很明确：

1. `restoredWorkerState` 默认是 `null`
2. `flushInternalEvents()` 默认 no-op
3. `internalEventsPending` 默认 `0`

这意味着默认宿主至多有资格解释：

- 基础消息面
- 当前控制交互面

却没有足够根据解释：

- 恢复账是否完整
- transcript 是否已真正可重建
- 当前远端恢复状态是否闭环

所以它不应越级给出：

`完整安全控制台已经怎样怎样`

这类结论。

## 5.2 这说明“我能看到部分控制消息”不等于“我能解释整套系统”

这是很多集成系统最容易犯的认识论错误：

`把有一条控制通道，误写成有完整解释权。`

Claude Code 的源码恰恰在不断避免这种误用。

## 6. `RemoteIO + CCR v2` 为什么解释权更大，但仍然是条件化的

## 6.1 它拿到了更多账，因此解释权确实更强

`src/cli/remoteIO.ts:88-140`、`205-252` 说明：

1. 它能拿到 `restoredWorkerState`
2. 能接入 `reportState(...)`
3. 能接入 `reportMetadata(...)`
4. 能写 `writeEvent(...)`
5. 能写 `writeInternalEvent(...)`
6. 能 `flushInternalEvents()`

所以和默认 `StructuredIO` 相比，  
它显然有更强的解释权限。

## 6.2 但它的解释权依然依赖初始化、CCR v2 和 transport 条件

同一段代码也说明：

1. 依赖 `CLAUDE_CODE_USE_CCR_V2`
2. 依赖 `SSETransport`
3. 依赖 `initialize()` 成功

因此更准确的结论不是：

`RemoteIO 拥有完整解释权`

而是：

`在条件成立时，RemoteIO 拥有更宽但仍然条件化的解释权。`

## 7. 窄宿主为什么尤其不能越级解释

## 7.1 `DirectConnect` 只正式支持 `can_use_tool`

`src/server/directConnectManager.ts:82-96`、`152-177` 已经足够说明：

1. 核心支持点是 `can_use_tool`
2. 其他 control subtype 会回 error
3. 主动控制也基本只有 `interrupt`

这种宿主最合理的解释范围是：

`权限请求当前如何`

而不是：

- 整体恢复状态如何
- 完整控制面是否一致
- 全局安全账本是否都对齐

## 7.2 `RemoteSessionManager` 同样只有消息面 + 权限型窄解释权

`src/remote/RemoteSessionManager.ts:188-206`、`224-241` 说明：

1. 它处理 `can_use_tool`
2. 其他 control subtype 会被视为 unsupported
3. 它能发送会话消息

因此它最合理的解释范围也是：

- 当前远程会话消息是否送达
- 当前权限请求是否存在

而不是替完整安全控制面做全局裁决。

## 7.3 窄宿主最成熟的表现不是“解释得更多”，而是“知道自己解释不了什么”

这正是这一章最重要的实践含义之一：

`解释权不足时，最专业的行为是显式保留未知，而不是拿局部事实硬凑全集解释。`

## 8. 更宽的 bridge 宿主为什么也不能滥用解释权

## 8.1 bridge 的视野更宽，但不代表任何时刻都能完整解释

`src/bridge/remoteBridgeCore.ts:813-878` 已经说明：

1. 它可以转发更宽的 SDK messages
2. 可以发 control request / response / cancel
3. 可以发 result

所以它比 direct connect / remote session manager 的解释权限更强。

## 8.2 但 recovery 窗口里它会主动 drop 一部分事实

同一段代码还说明：

1. recovery 中会 drop request
2. drop response
3. drop cancel
4. drop result

这意味着：

`bridge 的解释权限本身也会随状态窗口动态收窄。`

也就是说，  
解释权不是“宿主类型”这一维就能完全决定，  
还要继续看：

`宿主当前正处在哪个时序窗口。`

## 9. 把这些例子压成一条底层公理

我会把它压成一句话：

`宿主的解释权限，永远只能小于等于它当前真正可见的账本边界。`

因此成熟系统最需要的不是“多解释”，  
而是：

1. 先声明自己看到了哪些账
2. 再声明自己没看到哪些账
3. 最后只在可见边界内解释

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只在于允许子集宿主存在，还在于源码多处已经隐含“子集宿主不应越级解释”的纪律。`

这对其他 Agent 平台构建者的启示非常直接：

1. 宿主解释层应显式声明自己当前的账本可见性
2. 窄宿主应有标准化的“未知”语义，而不是乱猜
3. 更宽宿主也应显式暴露自己的条件窗口，例如 recovery 中降权
4. 支持体系与 UI 不应默认每个宿主都具备完整解释力
5. 解释权限应产品化，而不是只藏在源码约束里

## 11. 哲学本质

这一章更深层的哲学是：

`知道自己不知道什么，本身就是安全能力。`

很多系统把“解释得多”当成能力强。  
Claude Code 的源码更接近另一种成熟判断：

`真正可靠的解释，不来自胆子大，而来自边界清楚。`

所以解释权限的本质不是修辞能力，  
而是：

`认识论上的自限。`

## 12. 苏格拉底式反思：这章最容易犯什么错

### 12.1 我们是不是把解释权说得太抽象了

有这个风险。  
但源码里的 narrow subtype 支持、default no-op、conditional init 和 recovery drop 已经非常具体地说明：

`解释能力确实随账本可见性变化。`

### 12.2 宿主是不是只要更宽就应该默认解释更多

不一定。  
更宽只意味着“可解释上限更高”，  
不意味着任何时刻都应该把上限当成当前权限。

### 12.3 产品上真的需要把解释权限显式做出来吗

需要。  
否则系统就会把很多本来是“子集宿主的合理未知”，  
呈现成：

`看起来像黑箱，实际上是越级解释被禁止。`

## 13. 结语

`36` 回答的是：为什么不同宿主只拿到账本子集。  
`37` 继续推进后的结论则是：

`既然宿主只拿到子集，它也就不应替完整安全控制面下结论。`

这意味着 Claude Code 更深层的安全启示之一是：

`安全解释力不是默认权限，而是从当前可见账本边界反推出来的受限能力。`
