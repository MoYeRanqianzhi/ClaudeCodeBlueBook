# 安全载体家族强请求清理制度理由治理与强请求清理制度元数据治理分层：为什么artifact-family cleanup stronger-request cleanup-rationale-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-metadata-governor signer

## 1. 为什么在 `223` 之后还必须继续写 `224`

`223-安全载体家族强请求清理家族宪法治理与强请求清理制度理由治理分层` 已经回答了：

`不同 stronger-request old cleanup carriers 不只住在不同 cleanup law 里，而且服务不同 risk object、operational duty 与 host visibility。`

但如果继续往下追问，
还会碰到下一层更工程化、也更容易被偷写的错觉：

`只要系统已经存在不同 stronger-request cleanup rationale，就等于这些理由已经被正式提升成系统自己的显式 metadata plane。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看：

1. `src/utils/task/diskOutput.ts:31-49` 把 task-output family 的理由写进 path helper 注释，但这份 truth 主要仍寄居在 `getTaskOutputDir()` 与注释本身
2. `src/utils/toolResultStorage.ts:1-5,94-118` 只在文件头说明 “persisting large tool results to disk instead of truncating them”，并在 path helper 里落地；并没有和 transcript continuity、debug operational、diagnostics host-monitoring 一起收束成统一 family metadata
3. `src/utils/sessionStorage.ts:198-220`、`src/utils/plans.ts:79-106,164-233`、`src/utils/fileHistory.ts:951-957`、`src/utils/sessionEnvironment.ts:14-21,60-140` 各自只保存某一家的局部 truth
4. `src/utils/settings/types.ts:325-333,824-830` 只公开 `cleanupPeriodDays` 与 `plansDirectory` 这类局部 knob，而不是 family-wide metadata grammar
5. `src/utils/permissions/filesystem.ts:245-255,1487-1494,1644-1652` 只把计划文件的读写权限接到 `getPlansDirectory()` / `getPlanSlug()` 的局部判断上，说明 permission plane 消费的是 family truth 的局部投影，而不是统一 descriptor
6. `src/utils/debug.ts:30-68,91-111,229-249` 与 `src/utils/diagLogs.ts:11-20,27-60` 继续把 operational debugging、`/share`、bug-report 与 no-PII host monitoring 这些理由藏在注释、argv/env 与局部 helper 中
7. `src/utils/cleanup.ts:575-595` 仍通过 hand-written dispatcher 顺次调不同 cleaner，而不是从统一 stronger-request family registry 读取 policy metadata

这说明：

`artifact-family cleanup stronger-request cleanup-rationale-governor signer`

和

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

仍然不是一回事。

前者最多能说：

`我能解释为什么这些 stronger-request carrier family 应服从不同 law。`

后者才配说：

`系统已经把这些为什么正式保存成了自己可传播、可复用、可防漂移的 family metadata。`

所以 `223` 之后必须继续补的一层就是：

`安全载体家族强请求清理制度理由治理与强请求清理制度元数据治理分层`

也就是：

`stronger-request cleanup-rationale governor 最多只配解释理由；stronger-request cleanup-metadata governor 才配把理由升级成系统自己能重复读取的自描述。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-metadata-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-metadata-governor signer` 仍是研究命名。
它不是在声称 repo 里已经有一份未被接线的现成 family registry，
而是在说：

1. stronger-request family truth 已经广泛存在
2. 但这些 truth 主要散落在 helper、comment、schema、resume logic、permission rule 与 dispatcher 之间
3. 因而 stronger-request cleanup metadata 已经是一个独立问题，而不是 rationale 的同义词

第二条：

这里说的 `cleanup metadata`

不是在声称：

`当前系统完全没有 metadata。`

真正要说的是：

`当前系统有很多局部 metadata signal，但它们还没有被提升成统一的 family metadata grammar。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部 metadata 问题。

真正关键的是：

1. task-output family 的 truth 主要活在 comment + path helper
2. tool-result family 的 truth 主要活在 file header + path helper
3. plans family 的 truth 横跨 settings、storage、permission、resume 与 cleanup，最容易暴露 propagation gap
4. debug / diagnostics family 的 truth 还混着 argv/env 与 no-PII discipline

第四条：

这里尤其不能过度声称：

`只要 helper 足够多，就已经等于 metadata plane。`

当前更稳的说法只能是：

`helper composition 可以让人类研究者重建 family truth，但并不自动等于系统自己拥有一份统一、自描述、可传播的 metadata object。`

第五条：

这里也不能过度声称：

`metadata plane 一定意味着要做一个全新的中心类。`

当前更稳的说法只能是：

`问题的关键不在具体类名，而在这些理由能否被多平面共读、能否自动传播、能否被 drift invariant 校验。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-rationale-governor signer 仍不等于 stronger-request cleanup-metadata-governor signer`

证据：

1. task-output family 的关键 truth 主要活在 `diskOutput.ts` 的 helper 与注释里，而不是统一 metadata object
   `diskOutput.ts:31-49`
2. tool-results / transcripts 的 family truth 分别散落在 file header、path helper 与 persistence helper 中
   `toolResultStorage.ts:1-5,94-118`; `sessionStorage.ts:198-220`
3. plans family 的 truth 横跨 `plansDirectory` schema、`getPlansDirectory()`、`isSessionPlanFile()`、`copyPlanForResume()` 与 `cleanupOldPlanFiles()`，最容易暴露 propagation blind spot
   `settings/types.ts:824-830`; `plans.ts:79-106,164-233`; `permissions/filesystem.ts:245-255,1487-1494,1644-1652`; `cleanup.ts:300-303`
4. file-history / session-env 的 truth 虽相对稳定，但仍只是各自模块内的局部 family truth
   `fileHistory.ts:951-957`; `sessionEnvironment.ts:14-21,60-140`; `cleanup.ts:305-387`
5. debug / diagnostics 的理由分别寄居在 argv/env、comment 与局部 helper 中，没有被收束成统一 operational metadata grammar
   `debug.ts:30-68,91-111,229-249`; `diagLogs.ts:11-20,27-60`
6. `cleanupOldMessageFilesInBackground()` 当前更像 hardcoded call list，而不是 metadata-driven family policy plane
   `cleanup.ts:575-595`
7. `cleanupPeriodDays` 与 `plansDirectory` 当前暴露的是局部 knob，不是完整 family grammar
   `settings/types.ts:325-333,824-830`

因此这一章的最短结论是：

`stronger-request cleanup-rationale-governor signer 最多能解释“为什么应当如此”；stronger-request cleanup-metadata-governor signer 才能证明“系统已经把这份为什么正式保存成了自己会重复读取的同一份 truth”。`

再压成一句：

`有理由，不等于有自描述。`

## 4. 第一性原理：rationale 回答“为什么”，metadata 回答“这份为什么被系统存放在哪里、怎样被系统自己读回”

从第一性原理看，
强请求清理制度理由治理与强请求清理制度元数据治理处理的是不同层次的问题。

`stronger-request cleanup-rationale governor`

回答的是：

1. 这个 family 为什么存在
2. 它服务什么 risk object
3. 它保护什么 operational duty
4. 为什么它配住在 temp、project-session、home-root 或 host-external world
5. 为什么当前 law 不会破坏它的主要职责

`stronger-request cleanup-metadata governor`

回答的则是：

1. 上面这些理由被系统保存在哪里
2. 哪些平面消费的是同一份 family truth
3. settings、storage、permission、resume、cleanup 是否共享一套 descriptor
4. 一处 policy truth 的变化能否自动传播
5. 哪些 drift 可以由代码自己发现，而不是等研究者事后拼回

如果把两层压成一句“系统已经有不同理由，所以 metadata 自然也存在”，
系统就会制造五类危险幻觉：

1. explained-means-objectified illusion
   只要人类能解释，就误以为系统自己也保存了这份解释
2. helper-composition-means-policy illusion
   只要若干 helper 拼起来能讲通，就误以为那里已经等于 policy object
3. schema-field-means-metadata-plane illusion
   只要有 `cleanupPeriodDays` 或 `plansDirectory`，就误以为 family metadata grammar 已经存在
4. no-current-bug-means-no-metadata-gap illusion
   只要当前还能工作，就误以为没有 propagation blind spot
5. local-comment-means-global-truth illusion
   只要某处注释把理由写得很清楚，就误以为别的平面也都在读同一份 truth

所以从第一性原理看：

`rationale` 是制度说明；
`metadata` 是制度说明被系统自己持久保存并重复消费的方式。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup rationale 已经等于 stronger-request cleanup metadata，为什么 task outputs 的最硬 truth 主要仍活在 `diskOutput.ts` 的注释和 path helper 里，而不是被 cleanup dispatcher、permission plane 与 drift checker 共读？
   因为“理由存在”不等于“理由已对象化”。
2. 如果 plans family 已经有 metadata plane，为什么 `plansDirectory` 会改写 storage world，而 `cleanupOldPlanFiles()` 仍停在默认 `~/.claude/plans`？
   因为 propagation 只有在 metadata plane 存在时才更容易自动成立。
3. 如果 diagnostics 的 no-PII host-monitoring reason 已经被提升成统一 metadata，为什么它仍主要寄居在局部 comment 与 `CLAUDE_CODE_DIAGNOSTICS_FILE` env 选路里？
   因为 host-visible rationale 目前还没有和 repo-local family truth 合并进同一描述层。

## 5. task outputs、tool-results 与 diagnostics 先说明：有些 stronger-request family 已经有强理由，但这些理由仍主要停留在 local truth

`src/utils/task/diskOutput.ts:31-49` 让 task-output family 的 truth 非常清楚：

1. risk object 是 same-project live clobber
2. constitution 是 `project temp dir / sessionId / tasks`
3. path drift 也是这一家法的一部分

但它的保存方式主要仍是：

1. 注释
2. path helper
3. 下游调用者的隐式信任

`src/utils/toolResultStorage.ts:1-5,94-118` 对 tool-results 也是类似：

1. persisted inspection rationale 很清楚
2. 但这份 truth 主要活在 file header、`TOOL_RESULTS_SUBDIR` 与 `getToolResultsDir()` 中

`src/utils/diagLogs.ts:11-20,27-60` 对 diagnostics 也一样：

1. no-PII host monitoring rationale 很清楚
2. 但它主要活在 function comment 与 env-selected logfile helper 中

这三类 family 共同说明：

`reason can be explicit and still remain local`

也就是说，
“注释很清楚”与“metadata plane 已存在”之间仍然差着一个系统自描述层。

## 6. plans family 继续说明：metadata 缺口最容易在跨平面传播时被显影

`plans` 是这一层最硬的工程证据。

因为它的 truth 同时跨越：

1. `settings/types.ts:824-830` 的 `plansDirectory`
2. `plans.ts:79-106` 的 storage resolution
3. `permissions/filesystem.ts:245-255,1487-1494,1644-1652` 的 current-session read/write allowance
4. `plans.ts:164-233` 的 resume recovery
5. `cleanup.ts:300-303` 的 cleanup executor

这已经不是 local truth，
而是：

`cross-plane truth`

而当前最硬的现象是：

`storage plane 已接受 configurable world，permission plane 也跟着读写开放，cleanup plane 却仍主要活在 default world。`

这里暴露的不是孤立 bug，
而是更深的 metadata gap：

`当 family truth 横跨多个平面时，如果没有统一 metadata plane，传播就只能依赖实现纪律，而不是系统自描述。`

## 7. debug、file-history、session-env 与 cleanup dispatcher 再证明：系统当然知道 family 有别，但知道有别不等于把“有别”保存成统一 grammar

`src/utils/fileHistory.ts:951-957`、`src/utils/sessionEnvironment.ts:14-21,60-140` 与 `src/utils/debug.ts:30-68,91-111,229-249` 已经分别把：

1. restore backup
2. environment replay
3. operational debugging / `/share` / bug reports

这些理由公开出来。

`src/utils/cleanup.ts:575-595` 也明确知道不同 cleaner 要分开调度。

但这组代码合在一起暴露的事实是：

1. 系统知道 family 有别
2. 系统也知道这些 family 背后的理由不同
3. 但这些理由目前仍主要以
   helper
   comment
   schema field
   hand-written dispatch
   的形式存在

这说明当前显露出来的是：

`distributed family truth`

而不是：

`explicit stronger-request cleanup metadata grammar`

## 8. 技术先进性与哲学本质：Claude Code 真正先进的不是“已经把 metadata plane 完成”，而是已经逼到必须诚实面对 metadata plane 的地步

从技术上看，
Claude Code 当前实现真正先进的地方，
不是它已经把 stronger-request cleanup metadata 做成一份完美 registry。

真正先进的地方在于：

1. 它已经让不同 family 的理由明显到不能再假装“一份 generic cleanup sentence 就够了”
2. 它已经让跨平面传播问题真实显影，尤其是 plans family
3. 它已经让 host-visible no-PII diagnostics 这种异质面逼迫系统承认 metadata 不能只停在 repo-local cleanup helper 里

换句话说，
它的先进性在于：

`系统已经暴露出 metadata 层是真实缺口，而不是还能继续靠口头解释蒙混过去。`

从哲学上看，
这意味着：

`rationale` 解决的是解释义务，
`metadata` 解决的是记忆义务。

没有 metadata，
制度理由就只能停留在：

`知道的人知道，不知道的人就只能继续猜。`

而一个真正成熟的安全系统，
不能把自己的制度理由建立在“维护者记得住”之上。

## 9. 一条硬结论

这组源码真正说明的不是：

`Claude Code 还没有 family-level governance。`

而是：

`Claude Code 已经拥有大量 stronger-request family-level truth，但这些 truth 仍主要寄居在 cleanup、settings、helper、permission 与 comments 的分散组合里；因此 artifact-family cleanup stronger-request cleanup-metadata-governor signer 仍必须独立于 artifact-family cleanup stronger-request cleanup-rationale-governor signer 被追问。`

因此：

`cleanup 线真正缺的，不只是“谁来解释这些 law 为什么成立”，还包括“谁来把这些为什么保存成系统自己会重复读取、传播并校验的 metadata plane”。`
