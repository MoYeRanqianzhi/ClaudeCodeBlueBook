# 安全最小可执行验证蓝图：为什么下一步不是泛泛补测试，而是先建一套能持续守住最贵失真的首批验证套件

## 1. 为什么在 `113` 之后还必须继续写“最小可执行验证蓝图”

`113-安全验证覆盖缺口与优先补强路线` 已经回答了：

`测试投资不该平均分配，而应按历史前科、时序脆弱度、跨面失真成本与重构易损性排序。`

但如果继续往下追问，  
还会碰到一个更实际的问题：

`既然优先级已经排出来，那第一批到底该怎么落地？`

如果答案还是：

`以后逐步补一些 property/table/integration tests`

那这仍然不够。  
因为工程团队真正需要的不是抽象方向，  
而是：

1. 第一批先建什么
2. 每类测试的最小覆盖目标是什么
3. 需要先补哪种基础设施
4. 什么时候算“第一阶段闭环”

所以 `113` 之后必须继续补出的下一层就是：

`安全最小可执行验证蓝图`

也就是：

`下一步不是泛泛补测试，而是先建一套能持续守住最贵失真的首批验证套件。`

## 2. 最短结论

从源码看，这份 Claude Code 研究版源码当前更接近：

`guard-rich but test-thin`

而不是：

`verification-ready`

证据很直接：

1. `package.json` 只有 `prepare-src/build/check/start`，没有测试脚本  
   `package.json:7-11`
2. 本地检索 `__tests__/.test./.spec.` 结果为 `0`  
   本地检索：`rg --files ... | rg '(__tests__|\\.test\\.|\\.spec\\.)' | wc -l => 0`
3. 但高风险模块里已经明确留下了大量“值得先测”的回归信号：
   `useManagePlugins.ts:289-302`
   `refresh.ts:81-88,140-145`
   `useReplBridge.tsx:31-40,417-453,616-644`
   `onChangeAppState.ts:50-64`

所以这一章的最短结论是：

`第一阶段最值得做的，不是追求广覆盖，而是先建立最小测试骨架，并把最贵的五类失真各自钉上一枚自动化护栏。`

再压成一句：

`先做能守住制度边界的第一组测试，不先做看起来完整的一堆测试。`

## 3. 当前状态不是“没有验证意识”，而是“没有可执行测试骨架”

`package.json:7-11` 非常重要。

当前脚本只有：

1. `prepare-src`
2. `build`
3. `check`
4. `start`

这说明：

1. 源码整理链存在
2. 构建链存在
3. 类型检查链存在
4. 运行链存在

但：

`测试入口不存在。`

再结合本地检索结果：

`test/spec 文件数为 0`

可以得出一个很关键的判断：

`当前这份源码并不是没有验证思想，而是还没有把验证思想转化成可执行测试骨架。`

这意味着下一步最合理的动作，不是立刻铺满具体测试，  
而是先建：

1. 测试命令入口
2. 最小目录骨架
3. 第一批高优先级验证样本

## 4. 第一性原理：最小可执行蓝图应只回答三个问题

从第一性原理出发，  
首批验证蓝图不应追求“大而全”，  
而应只回答：

1. 系统最贵的失真是什么
2. 哪种测试最适合证明它不会复发
3. 最小做到什么程度，就足以让未来重构不再轻易破坏它

所以首批蓝图的目标不是“覆盖率好看”，  
而是：

`让未来任何对这些高风险路径的修改，都必须先穿过自动化护栏。`

## 5. 第一批验证套件应如何选题

基于前一章优先级排序，  
我建议第一批只做五组。

## 5.1 套件 A：Permission Mode Ordering Suite

优先级：`P0`

目标：

1. 守住 `guard-before-transition`
2. 守住 `external metadata sync`
3. 守住 `mode transition` 的 centralized path

为什么必须先做：

`onChangeAppState.ts:50-64` 已明确说明这里曾经有 8+ mutation paths 分叉；  
`useReplBridge.tsx:417-453` 又明确写出如果 guard 顺序错误会触发 `3-way invariant violation`。

最小测试类型：

1. integration ordering test
2. transition-table test
3. diff propagation test

最小断言：

1. guard 未通过时，`transitionPermissionMode` 不得推进到危险状态
2. mode 变化后，external metadata 一定同步
3. 相同 mode 不产生额外 transition

## 5.2 套件 B：Notification Truth-Integrity Suite

优先级：`P1`

目标：

1. 守住 stale timeout 不能清新 current
2. 守住 duplicate family 不并存
3. 守住 absent remove 是 no-op

为什么必须先做：

这些规则目前全靠局部 guard：  
`notifications.tsx:54-68,172-188,193-212`

而这类时序逻辑极容易在“无害重构”中退化。

最小测试类型：

1. property tests
2. table tests

最小断言：

1. 任意旧 timer 回调都不能清掉已被新 key 替换的 current
2. 相同 key 重复 add 最终 family cardinality 必须为 1
3. remove 不存在对象不会产生伪 retire

## 5.3 套件 C：Bridge Failure Lifecycle Suite

优先级：`P1`

目标：

1. 守住 repeated failure threshold
2. 守住 cancelled throw 不应污染错误态
3. 守住 cleared error 之后 timer 不再 disable bridge

为什么必须先做：

`useReplBridge.tsx:31-40` 已明确写出 401 风暴风险；  
`useReplBridge.tsx:616-644` 又明确留下 cancelled throw / stale error 前科。

最小测试类型：

1. threshold table tests
2. timer interleaving property tests
3. cancelled-path integration tests

最小断言：

1. failure count 达到阈值后必须熔断
2. cancelled path 不增加 failure 计数，也不写 stale error
3. error 已清时旧 timer 不得 disable bridge

## 5.4 套件 D：Plugin Refresh Completion Suite

优先级：`P2`

目标：

1. 守住 `needsRefresh` 只能由合法 refresh 路径消费
2. 守住 stale-cache bug 不复发
3. 守住 reconnect key bump 与 refresh completion 一起发生

为什么必须先做：

`useManagePlugins.ts:289-302` 明确写出 previous stale-cache bug；  
`refresh.ts:81-88,140-145` 又明确记录了 race 和 issue #15521。

最小测试类型：

1. transition-table tests
2. stale-cache regression tests

最小断言：

1. 没跑 full refresh 时，`needsRefresh` 不能被清
2. completion 后 `needsRefresh=false`
3. 同时触发 `pluginReconnectKey+1`

## 5.5 套件 E：Pointer Branch Matrix Suite

优先级：`P2`

目标：

1. 守住 invalid/stale pointer 不得 current
2. 守住 transient != fatal
3. 守住 resumable clean shutdown != retired/cleared

为什么必须先做：

pointer 是恢复资格对象，  
一旦退化就是直接的恢复主权损害。

最小测试类型：

1. schema + TTL boundary tests
2. branch matrix tests

最小断言：

1. invalid/stale -> `null`
2. transient reconnect failure 不 clear pointer
3. fatal reconnect failure 才 clear pointer
4. resumable clean shutdown 不 archive+deregister+clear

## 6. 第一批测试基础设施最小需要什么

如果只谈哲学，这一章就不够落地。  
所以还必须回答最小基础设施需求。

基于 `package.json:7-11` 当前状态，  
第一步至少需要加上三样东西：

## 6.1 一个统一测试入口

例如未来的：

1. `test`
2. `test:unit`
3. `test:integration`

作用：

`让验证从“研究建议”变成“可执行命令”。`

## 6.2 一个最小测试目录骨架

例如：

1. `tests/security/property/`
2. `tests/security/table/`
3. `tests/security/integration/`

作用：

`让测试优先级与测试类型在目录结构上就可见。`

## 6.3 一个最小 golden suite

也就是前面五组里至少先落每组一两个代表性测试。

作用：

`先把最贵失真钉住，而不是等完整体系建完。`

## 7. 第一阶段完成的判定标准

很多团队会在这里失焦，  
因为他们不知道“什么时候算第一阶段完成”。

我建议把第一阶段完成标准压成五条：

1. 有统一测试入口
2. 有最小测试目录骨架
3. 五个高优先级 area 各至少有一组自动化验证
4. 至少一条 property test 覆盖时序空间
5. 至少一条 integration ordering test 覆盖 centralized transition + propagation

满足这五条，  
就意味着系统第一次拥有了：

`不是口头上的测试规划，而是最小可执行验证蓝图。`

## 8. 为什么“最小可执行”比“完整蓝图”更重要

因为在安全工程里，  
最危险的状态不是“还没完全补齐”，  
而是：

`大家都知道该补，但永远停在计划阶段。`

所以这里真正需要的不是一张非常美的大图，  
而是一套：

`从明天就能开始落第一个测试的执行序列。`

这也是第一性原理上最务实的回答：

`先建立能持续变强的验证增量机制，再追求长期完整性。`

## 9. 技术启示：最好的安全验证路线图，应该像补丁序列，不该像口号墙

Claude Code 当前源码里留下的这些 bug/race/stale/invariant 前科，  
其实已经天然在告诉我们：

1. 哪些地方最脆
2. 哪些地方最贵
3. 哪些地方最适合先补

因此真正成熟的技术启示不是：

`以后要多写测试`

而是：

`把这些前科转译成一条可执行的补丁序列：先建入口，再建骨架，再钉五类最贵失真。`

## 10. 苏格拉底式反思：如果要把这一步做得更好，还应继续追问什么

可以继续追问八个问题：

1. 当前五个优先级 area 是否还应再细分出更小的 first test
2. 第一批 property tests 应优先用什么输入生成策略
3. 哪条 integration ordering test 最能证明 centralized transition 没被绕过
4. pointer suite 是否还应补 worktree-aware fanout 场景
5. plugin suite 是否应补 memoized loader 乱序场景
6. notification suite 是否应补 invalidates 与 fold 交错场景
7. 当前是否应把“历史前科必须伴随后续回归测试”写成开发制度
8. 第一阶段完成后，第二阶段是扩面积还是加深随机化

这些问题真正逼问的是：

`这套蓝图到底是在生成可执行护栏，还是又变成了一份漂亮但无执行力的研究文本。`

## 11. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正正确的下一步不是：

`泛泛地说应该补测试，`

而是：

`立刻建立最小测试入口与目录骨架，并优先把 permission ordering、metadata sync、notification stale actor、bridge failure timer、plugin refresh stale-cache、pointer branch matrix 这些最贵失真先钉死。`

因为只有这样，  
验证体系才会从：

`有方向`

进一步升级到：

`有第一批可执行护栏。`
