# 安全状态编辑主权：为什么状态替换语法之后，还必须继续回答谁有资格编辑谁的状态

## 1. 为什么在 `77` 之后还要继续写“状态编辑主权”

`77-安全状态替换语法` 已经回答了：

`成熟控制面必须区分 fold、invalidate、remove 与 coexist。`

但如果继续追问，  
还会碰到一个更根本、也更靠近真正安全问题的问题：

`这些编辑动词，到底谁有资格用？`

因为一套 replacement grammar 只回答了：

1. 有哪些编辑动作
2. 它们语义上有什么区别

却还没有回答：

1. 谁能折叠谁
2. 谁能作废谁
3. 谁能撤回谁
4. 谁无权碰谁

如果没有这层主权问题，  
那么再优雅的状态编辑语法也可能退化成：

`任何层都能随便改写别人的状态。`

所以在 `77` 之后，  
安全专题必须再补一条更接近第一性原理的原则：

`状态编辑主权。`

也就是：

`定义编辑动词还不够，系统还必须继续回答：哪个主体、哪个状态家族、在哪种关系下，才配对哪个状态执行 fold、invalidate、remove 或保持 coexist。`

## 2. 最短结论

从源码看，Claude Code 目前已经形成了明显的“局部状态家族主权”惯例，但还没有把它完全做成硬约束：

1. 通知上下文把 `removeNotification` 暴露成全局 `key: string` 接口，任何拿到 hook 的调用方理论上都能删任意 key  
   `src/context/notifications.tsx:31-40,193-213`
2. 但真实使用里，绝大多数 `removeNotification(...)` 都是状态拥有者自己撤回自己的 key  
   `src/hooks/notifs/useSettingsErrors.tsx:37-48`  
   `src/components/PromptInput/Notifications.tsx:150-162`  
   `src/components/PromptInput/PromptInput.tsx:747-769,787-790`
3. `invalidates` 在源码里只出现在 fast mode 这一类明确互斥、且属于同一状态家族的 paired keys 中  
   `src/hooks/notifs/useFastModeNotification.tsx:8-11,101-121`
4. `fold` 也只出现在 teammate lifecycle 这种明显由同一子系统维护的重复事件聚合里  
   `src/hooks/notifs/useTeammateShutdownNotification.ts:18-46,49-77`
5. IDE 状态 hook 会跨 sibling keys 做家族内编辑，比如 generic disconnected 会给 install error / JetBrains-specific 状态让位，但这仍发生在同一 IDE status family 内部  
   `src/hooks/notifs/useIDEStatusIndicator.tsx:78-152`
6. 也就是说，代码现实不是“谁都在乱改谁”，而是“大家大体遵守本地家族所有权纪律”；只是这种纪律更多靠约定与文件边界，而不是类型系统强制

所以这一章的最短结论是：

`Claude Code 已经形成了状态编辑主权的工程习惯，但还没有把它彻底提升为协议级能力边界。`

我把它再压成一句：

`它已经知道谁应该改口，但还没把“谁配改口”完全写进接口。`

## 3. 源码已经说明：状态编辑权目前主要依赖“家族所有者”而不是全局仲裁者

## 3.1 接口层其实很宽：`removeNotification(key)` 是 stringly-typed 的全局删改能力

`src/context/notifications.tsx:31-40` 先定义了：

1. `AddNotificationFn = (content: Notification) => void`
2. `RemoveNotificationFn = (key: string) => void`

而 `193-213` 又真正实现了：

`removeNotification(key)`

它只按 key 查：

1. current
2. queue

然后删掉命中的项。

这里没有：

1. 状态家族 token
2. 作用域限制
3. 所有者校验
4. capability handle

这说明从纯接口能力上看，  
当前系统仍是：

`全局可删，字符串寻址。`

如果从严格安全工程的标准看，  
这其实是一条重要事实：

`编辑主权尚未被类型化。`

## 3.2 但真实代码并没有滥用这条宽接口，而是基本遵守“自己增、自己撤”

虽然接口很宽，  
真实使用却很克制。

`src/hooks/notifs/useSettingsErrors.tsx:37-48`：

1. settings error hook 自己 add
2. errors 消失后自己 remove

`src/components/PromptInput/Notifications.tsx:150-162`：

1. external editor hint 由同一逻辑决定是否 add
2. 条件失效时由同一逻辑 remove

`src/components/PromptInput/PromptInput.tsx:747-769,787-790`：

1. `ultrathink-active`
2. `ultraplan-active`
3. `stash-hint`

也都由同一 PromptInput 逻辑负责撤回。

这说明团队虽然没有把主权硬编码进 API，  
却已经在工程上遵守一条默认规则：

`谁创建自己的状态，谁优先负责结束自己的状态。`

这是一种非常典型的：

`convention-enforced sovereignty`

## 3.3 `fold` 的使用也严格局限在本地家族内部，说明聚合权不是共享权

`src/hooks/notifs/useTeammateShutdownNotification.ts:18-46,49-77` 很值得注意。

这里：

1. `foldSpawn` 只折叠 `teammate-spawn`
2. `foldShutdown` 只折叠 `teammate-shutdown`
3. 聚合逻辑完全封装在同一个 teammate lifecycle hook 内

这说明 `fold` 虽然在协议层是一个开放字段，  
但在业务现实里它并没有变成“谁都能替别人汇总”的共享权限。

反而它表现出一条更细的主权原则：

`只有理解同一家族重复事件语义的人，才配定义它的聚合方式。`

因为如果 outsider 来 fold，  
最容易发生的不是删除错误，  
而是：

`聚合规则错误。`

所以 `fold` 的主权核心不是删除权，  
而是：

`语义压缩权。`

## 3.4 `invalidates` 只出现在 fast mode 这类成对互斥 key 上，说明废止权也被局部化了

`src/hooks/notifs/useFastModeNotification.tsx:8-11,101-121` 定义了：

1. `COOLDOWN_STARTED_KEY`
2. `COOLDOWN_EXPIRED_KEY`

并且让二者对称地彼此 `invalidates`。

这很关键。

因为它说明在现有代码里，  
`invalidates` 并没有被到处乱用成“我看你不顺眼就把你废掉”。

相反，  
它只出现在一种关系非常清楚的地方：

`同一家族、同一轴线、互斥、且双方语义边界都由同一个 subsystem 理解。`

这说明废止权在当前工程实践里其实也遵守着一条隐形约束：

`谁能宣布旧状态失效，前提是它同时理解旧状态与新状态的互斥关系。`

也就是说，  
`invalidate` 的真正主权不是拥有当前 key，  
而是：

`拥有互斥关系的解释权。`

## 3.5 IDE 状态 hook 展示了另一种更高级的主权：家族内 sibling 编辑

`src/hooks/notifs/useIDEStatusIndicator.tsx:78-152` 比前面几类都更复杂。

它并不是简单的“自己增、自己删一个 key”，  
而是在同一家族内部维护多种 sibling 状态：

1. `ide-status-disconnected`
2. `ide-status-jetbrains-disconnected`
3. `ide-status-install-error`

这里发生的不是 owner self-withdrawal，  
而是：

`family-level editorial control`

即：

1. 同一个 IDE status family owner
2. 同时掌握多条 sibling 状态
3. 并根据更具体条件决定谁该出现、谁该退场

这说明真正成熟的状态编辑主权不是只能“一条状态一个主人”，  
还可以是：

`一个家族有一个编辑者，负责家族内部的替换秩序。`

这比简单 owner model 更接近现实控制面。

## 3.6 第一性原理：编辑语法必须继续追问主语，否则动词再精细也会失控

如果从第一性原理追问：

`为什么 replacement grammar 之后还必须继续写 sovereignty？`

因为动词本身不能独立存在。  
每个状态编辑动作背后都隐含一个主语：

1. 谁在聚合
2. 谁在作废
3. 谁在撤回
4. 谁决定共存

如果没有主语，  
系统就只能依靠：

1. 命名碰巧不冲突
2. 团队成员记得别乱删别人的 key
3. 文件边界刚好把错误概率压住

这对当前代码规模也许还能工作，  
但对下一代统一安全控制台来说，  
这显然不够。

因此编辑主权真正保护的不是美观，  
而是：

`状态编辑关系不会随着系统变大而退化成全局字符串互殴。`

## 3.7 技术先进性与缺口并存：Claude Code 已经有主权直觉，但尚缺 capability-encoded interface

从技术角度看，  
Claude Code 当前状态编辑系统的优点很明确：

1. 家族边界大体清楚
2. 真正使用 `fold` / `invalidates` 的地方非常克制
3. 大部分 remove 行为都由状态拥有者执行
4. sibling editing 也主要发生在同一 subsystem 内

这说明作者明显已经具备：

`不要让任意层乱改任意状态`

的工程直觉。

但它的缺口也同样明确：

1. `removeNotification(key)` 仍是裸字符串接口
2. `invalidates` 仍是手写 key 列表
3. 没有 family scope 或 handle token
4. 没有静态约束“你只能移除自己家族里的 key”

所以如果从“下一代可证明安全控制面”的标准看，  
最值得补的一步不是再加更多提示文案，  
而是：

`把状态编辑主权从团队习惯升级成接口能力边界。`

## 4. 状态编辑主权的最短规则

把这一章压成最短规则，就是下面六句：

1. `fold` 应由理解同一事件家族聚合语义的拥有者执行
2. `invalidate` 应由掌握互斥关系解释权的家族编辑者执行
3. `remove` 应优先由状态拥有者在生命周期结束时执行
4. sibling 状态替换应局限在同一状态家族内部
5. 正交状态是否共存，不应由任意单个 key owner 擅自决定
6. 如果接口不编码这些主权，系统就只能靠纪律维持秩序

## 5. 苏格拉底式追问：这套系统还能怎样再向前推进

如果继续自问，  
下一代最值得做的三项升级会是：

1. 给通知返回 opaque handle，而不是只暴露 `string key`
2. 给 `invalidates` 引入 family-scoped relation declaration，而不是手写全局 key 名
3. 把 `owner remove`、`family invalidate`、`family fold`、`global coexist` 明确做成不同 capability interface

所以这一章最终沉淀出的设计启示是：

`状态系统真正成熟，不是编辑动词更多，而是每个编辑动词都有被承认的主语。`
