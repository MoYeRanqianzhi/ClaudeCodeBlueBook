# 安全载体家族强请求清理就绪治理与强请求清理连续性治理分层：为什么artifact-family cleanup stronger-request cleanup-readiness-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-continuity-governor signer

## 1. 为什么在 `362` 之后还必须继续写 `363`

`362-安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层`
已经回答了：

`stronger-request cleanup 线即便已经把旧 truth 重新接进 current world，也还需要 readiness governor 去决定这个当前世界现在到底能不能被继续消费。`

但如果继续往下看，
还会碰到另一层更容易被偷写成
“既然现在已经 ready，那就自然会继续可用”
的错觉：

`只要 stronger-request cleanup readiness governor 已经判定对象此刻 connected、capable、downstream-consumable，它就自动拥有了决定这份可用性在断连、退化、重试、放弃、人工停表与世界切换中怎样继续成立的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 MCP 线已经成熟写出的连续性正对照，
会发现 repo 其实明确拆开了六件事：

1. `continuity budget`
2. `disconnect 后的 retry lifecycle`
3. `stale retry line revocation`
4. `operator stop / restart control`
5. `runtime downgrade after current readiness`
6. `pool-level repair for current usable world`

最硬的证据至少有六组：

1. `useManageMCPConnections.ts:88-90` 先把 continuity constitution 明写成 `MAX_RECONNECT_ATTEMPTS = 5`、`INITIAL_BACKOFF_MS = 1000`、`MAX_BACKOFF_MS = 30000`
2. `useManageMCPConnections.ts:333-464` 把 disconnect 之后的制度写成 `pending -> reconnect attempt -> backoff -> give up / recover`
3. `useManageMCPConnections.ts:765-853` 明确清理 stale reconnect timer，并拒绝让旧 config、旧 session、旧 retry closure 继续冒充 current world
4. `useManageMCPConnections.ts:1043-1123` 把 manual reconnect、disable cancel 与 re-enable reconnect 写成显式的 continuity operator control
5. `toolExecution.ts:1599-1628` 会把一个原本 `connected` 的 client 在运行时降成 `needs-auth`
6. `print.ts:1392-1426` 会因为 `pending / failed` SDK clients 重新初始化整池 `setupSdkMcpClients()`

这说明：

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

和

`artifact-family cleanup stronger-request cleanup-continuity-governor signer`

仍然不是一回事。

前者最多能说：

`它现在能用。`

后者才配说：

`它一旦掉出 ready、需要重试、需要停表、需要撤销旧持续线或需要整体修池时，系统由谁决定继续争取、重新开线、暂停等待还是正式终止这份可用性。`

所以 `362` 之后必须继续补的一层就是：

`安全载体家族强请求清理就绪治理与强请求清理连续性治理分层`

也就是：

`stronger-request cleanup readiness governor 负责决定对象现在能不能用；stronger-request cleanup continuity governor 才负责决定这种可用性怎样跨时间维持、退化、重建或终止。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-continuity-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-continuity-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 continuity manager，
而是在说：

1. repo 已经把 “当前 ready” 与 “时间里继续 ready” 拆成两层
2. repo 已经把 retry、backoff、give-up、cancel、stale-line cleanup 与 pool repair 写成正式 grammar
3. repo 已经拒绝让一次 current-time readiness verdict 越级冒充 temporal continuity verdict

第二条：

这里说的 `continuity governance`

不是在声称：

`对象一旦 ready，就必须永远保持可用。`

当前更稳的说法只能是：

`continuity governance 首先回答的是 ready truth 中断后系统还要不要继续为它争取可用性，以及争取到什么程度。`

第三条：

这里也不是把：

`auto-reconnect`

单独当成全部 continuity grammar。

真正关键的是：

1. retry budget
2. stale continuity revocation
3. operator stop / restart control
4. runtime downgrade 后的重新裁决
5. pool-level repair

第四条：

这里尤其不能过度声称：

`continuity governance`

已经等于：

`recovery governance`

当前更稳的说法只能是：

`continuity governance 回答旧的 ready line 还要不要继续；recovery governance 才回答什么新的证据足以重新宣布 recovered truth 成立。`

第五条：

这里也不能过度声称：

`某一刻读成功一次`

就等于：

`后续时间里都可以继续依赖它。`

当前更稳的说法只能是：

`single successful read 只是 current-time consumption proof，不是 temporal continuity proof。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-readiness-governor signer 仍不等于 stronger-request cleanup-continuity-governor signer`

证据：

1. `useManageMCPConnections()` 把断连后的制度直接写成 `MAX_RECONNECT_ATTEMPTS`、指数 backoff 与最终 give-up，说明 ready truth 断掉后还存在单独的 continuity budget
2. stale reconnect timer cleanup 说明 repo 继续追问的不是“现在 ready 不 ready”，而是“哪条持续线仍配代表 current world”
3. manual reconnect、disable cancel 与 re-enable reconnect 说明 continuity 不是 background retry 的副作用，而是可被人类显式治理的 stop / restart 主权
4. `toolExecution.ts` 会把一个原本 `connected` 的 client 降成 `needs-auth`，说明 ready truth 会被运行时证据持续撤销，continuity 必须处理这种撤销后的时间线
5. `print.ts` 会因为 `pending / failed` SDK clients 整批 re-init 当前 pool，说明 continuity 不只属于单对象 state，还属于 current usable world 的持续维持

因此这一章的最短结论是：

`stronger-request cleanup readiness governor 最多能说对象现在 ready；stronger-request cleanup continuity governor 才能说这份 ready truth 在断连、退化、重试、放弃与人工干预中怎样继续成立。`

再压成一句：

`现在可用，不等于持续可用。`

再压成更哲学一点的话：

`制度知道此刻敢不敢用它，不等于制度已经知道这份可用性在时间里怎样继续成立。`

## 4. 第一性原理：readiness governance 回答“当前能不能消费”，continuity governance 回答“这种可消费性怎样跨时间维持”

从第一性原理看，
强请求清理就绪治理与强请求清理连续性治理处理的是两个不同主权问题。

`stronger-request cleanup readiness governor`
回答的是：

1. 当前 client 是否 `connected`
2. 当前 capability gate 是否闭合
3. 当前 consumer 能不能立刻读、调、调用
4. 当前 surface 应显示 `connected / pending / needs-auth / failed / disabled` 中的哪一档
5. 这一刻的 usable truth 是否成立

`stronger-request cleanup continuity governor`
回答的则是：

1. ready truth 一旦断掉，要不要继续重试
2. 重试最多试几次，以什么 backoff 节奏继续
3. 什么时候该从“继续争取可用”转成“正式 give up”
4. 旧 retry line、旧 timer、旧 closure 何时必须被 current world 撤销
5. manual reconnect、disable、re-enable 与 pool repair 怎样改写这条可用性时间线

如果把这两层压成一句：

`反正它刚才已经 connected`

系统就会制造五类危险幻觉：

1. `ready-means-stable`
2. `retry-is-incidental`
3. `stale-line-is-harmless`
4. `operator-intervention-is-ui-noise`
5. `one-read-means-dependable`

所以从第一性原理看：

`readiness governance` 处理现在时的可消费性；
`continuity governance` 处理时间轴上的可维持性。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup readiness governance 已经等于 continuity governance，为什么 repo 还要单独维护 `MAX_RECONNECT_ATTEMPTS`、指数 backoff 与 final give-up？  
   因为“当前 ready”与“ready truth 断掉后还要不要继续争取”根本不是同一句制度。
2. 如果 ready truth 一旦成立就天然连续，为什么 manual reconnect 前还要取消旧 pending reconnect，disable 时还要停掉重试线？  
   因为 continuity 不是自然延长，而是被 current authority 持续裁决的时间过程。
3. 如果当前 consumer 读成功一次就足够代表 continuity，为什么 `toolExecution.ts` 还能把一个已经 `connected` 的 client 降成 `needs-auth`，`print.ts` 还能因 `pending / failed` SDK clients 重新初始化整池？  
   因为 ready verdict 不是永久席位，continuity 处理的正是它被撤销之后谁继续负责。

## 5. `useManageMCPConnections()` 先证明：当前 ready 之后，repo 还单独维护一整条 continuity budget

`useManageMCPConnections.ts:88-90`
很值钱。

这里不是简单写一句“断了就重连”，
而是先定义三条 continuity constitution：

1. 最多重试五次
2. 从 1 秒开始指数退避
3. 最长退避不超过 30 秒

这说明 repo 追问的已经不是：

`此刻是不是 ready`

而是：

`ready truth 一旦中断，系统接下来还愿意为它维持多久的连续性。`

`useManageMCPConnections.ts:333-464`
则把这条制度真正落成时间语法。

这里在 `onclose` 之后：

1. 先判定对象是否已 disable
2. remote transport 才进入 automatic reconnect
3. 每次 attempt 前都显式改成 `pending`
4. 成功时回到 `connected`
5. 达到上限时正式 give up
6. 每次失败后按指数 backoff 再尝试下一次

从第一性原理看，
这已经不再是单纯的 readiness state machine，
而是：

`readiness breaks -> continuity process begins`

repo 公开拒绝把一个 ready verdict 当作天然可持续的真相。

## 6. stale reconnect timer 清理再证明：continuity 还要回答“哪条持续线才是 current”

`useManageMCPConnections.ts:765-853`
的价值不只在于它会把新对象置成 `pending`。

更值钱的是它在处理 stale plugin clients 时，
会优先：

1. 清掉旧 reconnect timer
2. 对旧 connected client 解除 `onclose`
3. 再按新的 configs 重建当前 pending world

这条证据很硬。
它说明 continuity 不是：

`只要某条重试线还在跑，就说明系统还在维持连续性`

而是：

`只有与 current config、current session、current plugin world 对齐的尝试线，才配继续代表 continuity truth`

这对 stronger-request cleanup 极其重要。
因为将来旧 path、旧 promise、旧 receipt 即便已经回到当前世界，
也仍不自动说明：

`之前那条 continuity attempt 仍然合法。`

如果 current truth 已换，
旧 continuity 线本身也必须被撤销。

## 7. manual reconnect 与 toggle path 再证明：continuity 是可被人类显式治理的操作主权

`useManageMCPConnections.ts:1043-1123`
继续把问题收紧。

这里的 `reconnectMcpServer()` 和 `toggleMcpServer()` 明确展示出：

1. manual reconnect 会先取消已有 pending reconnect，再开一条新的 continuity line
2. disable 不只是 UI 状态变化，而是要先停掉一切正在进行的 retry
3. re-enable 不是直接假装 continuity 已恢复，而是先进入 `pending`，再重新争取 ready truth

这说明 continuity governance 不是背景线程自己“慢慢修”的黑盒。
它已经是一个正式的操作面：

1. 谁有权继续试
2. 谁有权停下
3. 谁有权重启

从技术先进性看，
这里最值得学习的不是“提供了重连按钮”，
而是 repo 把按钮语义、timer 语义与 state transition 统一写进了同一条治理线。

## 8. `toolExecution.ts` 与 `print.ts` 证明：continuity 从 ready truth 被撤销的那一刻开始，并在 pool-level 上重建 current world

`toolExecution.ts:1599-1628`
很硬。

这里在 `McpAuthError` 时，
会把一个本来是 `connected` 的 client 降成：

`type: 'needs-auth'`

而且只在原状态确实是 `connected` 时才改。

这说明 repo 已经公开承认：

1. ready truth 不是永久席位
2. runtime evidence 可以撤销 ready truth
3. 撤销之后系统必须进入新的 continuity judgment

从技术角度看，
这是比“有 retry loop”更强的证据。

因为它说明 continuity 的起点不一定是 transport close，
也可以是：

`当前 ready 的对象被新的运行时证据判定为不再可靠。`

`print.ts:1392-1426`
特别值钱。

这里不只检查：

`有没有新 server 或被删掉的 server`

还继续检查：

1. `hasPendingSdkClients`
2. `hasFailedSdkClients`

只要这些条件成立，
就整批重新执行 `setupSdkMcpClients()`。

这说明 continuity 不只是：

`单个对象掉线了怎么办`

更是：

`当前 usable pool 还能不能继续被当成一个整体成立`

这一步把 continuity 从单连接鲁棒性提升成了系统级自我修复能力。

## 9. `ReadMcpResourceTool` 给出反向提醒：current-time hard gate 很硬，但它不是 continuity signer

`ReadMcpResourceTool.ts:78-95`
很有启发。

它对 consumer 说得非常硬：

`不是 connected 就别读`

这条 gate 非常先进，
因为它防止 consumer 用弱事实消费强对象。

但这条代码也恰好提醒我们：

`consumer hard gate 很硬，不等于 continuity question 已经被回答。`

它只验证：

`当前能不能读`

并不验证：

`接下来是否还能继续读`

所以 stronger-request cleanup continuity governor 不能被偷写成 consumer gate 的附属物。

## 10. 这篇源码剖面给主线带来的五条技术启示

1. repo 已经在 retry budget 上明确展示：`temporal dependability` 本身就是制度对象，不是 current-time readiness 的自然延伸。
2. repo 已经在 stale reconnect timer cleanup 上明确展示：`旧 continuity line` 本身也是风险对象，必须受 current-world revocation 约束。
3. repo 已经在 operator stop / restart control 上明确展示：continuity 是可被人类显式治理的操作主权，不是 background retry 的副作用。
4. repo 已经在 runtime downgrade 上明确展示：continuity 从 ready verdict 被撤销的那一刻开始，而不是等到对象彻底消失才开始。
5. repo 已经在 SDK pool re-init 上明确展示：continuity 既属于单对象，也属于当前 usable world 的整体维持。 

## 11. 苏格拉底式自反诘问：我是不是又把“对象现在已经 ready”误认成了“这份可用性在时间里会自然继续成立”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 readiness grammar 已经足够强，为什么还要再拆 continuity？  
   因为当前 ready，不等于 ready truth 断掉后系统还知道怎样继续、暂停或终止。
2. 如果对象已经 `connected`，是不是就说明后面自然会稳定？  
   不是。retry budget、backoff 与 give-up 都说明稳定性需要被继续治理。
3. 如果旧 retry line 还在跑，是不是就说明系统还在正确维护 continuity？  
   不能这样推。stale retry line 可能已经不再代表 current world。
4. 如果 manual reconnect 只是一个按钮，是不是不值得写成治理层？  
   恰恰相反。按钮语义本身就是 continuity authority 的显式入口。
5. 如果 `ReadMcpResourceTool` 当前能读成功一次，是不是就说明 continuity 已成立？  
   也不对。那只是 current-time proof，不是 temporal proof。
6. 如果 cleanup 线现在还没有显式 continuity 代码，是不是说明这层还不值得写？  
   恰恰相反。越是缺这层明确 grammar，越容易把“现在可用”偷写成“持续可用”。`

这一串反问最终逼出一句更稳的判断：

`continuity 的关键，不在对象此刻会不会用，而在系统能不能正式决定这份可用性在时间里怎样继续成立。`

## 12. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 readiness governance，就已经足够成熟。`

而是：

`Claude Code 在 continuity budget、指数 backoff、stale-line revocation、operator stop/restart control、runtime downgrade 与 pool-level repair 上已经明确展示了 continuity governance 的存在；因此 artifact-family cleanup stronger-request cleanup-readiness-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-continuity-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁来判断它现在能不能用”，而是“谁来决定这份可用性在时间里怎样继续成立”。`
