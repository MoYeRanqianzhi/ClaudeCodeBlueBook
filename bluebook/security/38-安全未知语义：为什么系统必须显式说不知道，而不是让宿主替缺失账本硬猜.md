# 安全未知语义：为什么系统必须显式说不知道，而不是让宿主替缺失账本硬猜

## 1. 为什么在 `37` 之后还要继续写“未知语义”

`37-安全解释权限` 已经回答了：

`看到账本子集的宿主不应替完整控制面下结论。`

但如果继续往第一性原理再压一层，  
还会剩下一个更具体、也更严苛的问题：

`当系统真的不知道时，它应该怎么做？`

换句话说，  
如果一个宿主：

1. 不支持某个 control subtype
2. 没接上某本账
3. 正处在恢复窗口
4. 当前条件还没成立

那它究竟应该：

1. 静默继续
2. 猜一个最像的答案
3. 假装成功
4. 还是明确说“我现在不知道 / 不支持 / 不成立”？

Claude Code 的源码在很多地方都给了同一种答案：

`成熟的安全控制面，必须把“不知道”做成显式语义。`

所以这一章的核心问题是：

`Claude Code 为什么宁可显式暴露“不知道”“不支持”“当前不能判定”，也不让宿主替缺失账本硬猜？`

## 2. 最短结论

从源码看，Claude Code 的很多高质量安全实现都在遵守同一条更深的纪律：

`未知必须被显式表达，不能被静默猜测替代。`

这里的“未知”至少包括四种：

1. capability unknown：当前宿主不支持
2. state unknown：当前账本没接上
3. timing unknown：当前窗口不允许判断
4. restore unknown：当前恢复基底还没成立

所以这一章的最短结论是：

`Claude Code 的安全性不只来自边界、账本和解释权限，也来自它把“未知”本身产品化成显式语义，而不是让局部宿主用猜测填平空洞。`

## 3. 什么叫“安全未知语义”

这里说的“未知语义”，不是普通的编程 `null`。  
我在这一章里的定义是：

`系统明确把“当前没有足够依据给出结论”编码成可传播、可处理、可停止推断的状态。`

这意味着：

1. 未知不是异常垃圾桶
2. 未知不是默认值伪装
3. 未知不是静默吞掉
4. 未知是控制面的一等输出

只有这样，  
宿主和用户才不会把：

`系统没看见`

误读成：

`系统已经看见并且没问题`

## 4. 第一类未知：不支持某种 control subtype，就必须显式回“不支持”

## 4.1 `DirectConnect` 明确对未知 subtype 回 error，而不是装作处理了

`src/server/directConnectManager.ts:82-96` 已经把这条纪律写得很明白：

1. 它只真正支持 `can_use_tool`
2. 其他 subtype 会进入 unsupported 分支
3. 直接发 error response
4. 注释明确说这样做是为了避免 server hang waiting for a reply

这里最关键的不是“回了个错”，  
而是它拒绝让系统落进一种危险假象：

`宿主虽然不支持，但上游以为它可能会处理。`

也就是说，  
它选择的是：

`显式不支持`

而不是：

`假装也许能支持。`

## 4.2 `RemoteSessionManager` 做了完全同一种选择

`src/remote/RemoteSessionManager.ts:192-212` 同样如此：

1. 只把 `can_use_tool` 当成正式支持路径
2. 其他 subtype 明确回 `control_response.error`
3. 同样强调不能让 server 一直等一个不会来的回复

这说明这不是某个模块的偶然写法，  
而是同一种控制面哲学：

`不支持时，必须显式给出“不支持”的语义。`

## 5. 第二类未知：没接上某本账，就必须显式暴露“没有”

## 5.1 `StructuredIO` 默认把恢复账缺口编码成 `null` 与 no-op

`src/cli/structuredIO.ts:139-195` 很关键。

它没有做下面这种危险操作：

`假装所有宿主都拥有 restored worker state 和 internal-event flush。`

相反，它明确给出：

1. `restoredWorkerState = Promise.resolve(null)`
2. `flushInternalEvents()` 对非 remote IO 是 no-op
3. `internalEventsPending = 0`

也就是说，  
它把：

`这个宿主当前没有这本账`

编码成了显式语义。

## 5.2 这比“返回一个像真的默认值”更安全

如果默认宿主去伪造一个“看起来像恢复成功”的占位状态，  
那上层很容易把：

`没有恢复账`

误读成：

`恢复账为空且一切正常`

所以这里的 `null` 和 no-op 不是偷懒，  
而是：

`对未知保持诚实。`

## 6. 第三类未知：条件还没成立时，必须显式拒绝解释

## 6.1 `RemoteIO` 对 CCR v2 条件失败给出显式失败，而不是降成伪完整模式

`src/cli/remoteIO.ts:118-140` 已经说明：

1. transport 不是 `SSETransport` 会直接 throw
2. `initialize()` 失败会打 diagnostics，并走 shutdown
3. `reason` 明确归到 typed/unknown failure reason

这说明系统没有选择：

`条件不成立时偷偷退成一个“差不多”的完整 remote 模式。`

它反而在强调：

`条件不成立 = 当前不能声称这套能力存在。`

## 6.2 这本质上也是一种“未知必须显式化”

也就是说，  
它不是在说“系统坏了所以退出”，  
更深层的意思是：

`在没有足够前提时，不应继续装作自己知道。`

## 7. 第四类未知：恢复窗口里的事实必须被显式降为“当前不可判定”

## 7.1 `remoteBridgeCore` 在 401 recovery 中直接 drop，而不是假装继续成立

`src/bridge/remoteBridgeCore.ts:824-878` 非常关键。

源码明确写出：

1. recovery 中 drop `control_request`
2. drop `control_response`
3. drop `control_cancel_request`
4. drop `result`

这里最重要的不是“丢了消息”，  
而是作者明确拒绝一种更危险的事情：

`在恢复窗口里继续假装控制面仍然完整可判定。`

## 7.2 这说明“当前窗口下不能说”本身就是一种安全语义

也就是说，  
恢复窗口不是普通降速窗口，  
而是：

`解释权、发送权和状态判定权都被临时收窄的窗口。`

因此 drop 的深层语义不是粗暴，  
而是：

`现在不能假装知道。`

## 8. 第五类未知：系统还会明确标出“哪些事实现在才算 authoritative”

## 8.1 `sdkEventQueue.ts` 把 `session_state_changed(idle)` 明确标成 authoritative turn-over signal

`src/utils/sdkEventQueue.ts:56-74` 也和这一章有关。

它实际上在做一件事：

`不给前面的近似信号过度解释权。`

只有当：

1. heldBackResult flush 完成
2. bg-agent do-while 退出

之后，  
才把 `idle` 明确标成 authoritative signal。

这说明系统在主动区分：

1. 现在还不能确信 turn over
2. 现在可以正式说 turn over 了

也就是说：

`在 authoritative signal 出现之前，系统宁可保留未知，也不提前下完整结论。`

## 9. 把这些例子压成一条底层公理

我会把它压成一句话：

`当系统缺少能力、缺少账本、缺少条件或缺少时序基底时，最安全的做法不是猜，而是显式输出“当前不能这样解释”。`

这条公理解释了前面的很多实现：

1. unsupported subtype -> error
2. no restore ledger -> null / no-op
3. init 条件不成立 -> fail loudly
4. recovery 中不可判定 -> drop
5. authoritative signal 未到 -> 暂不授予解释权

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只在于它会做很多判断，还在于它知道什么时候不该判断。`

这对其他 Agent 平台构建者的启示非常直接：

1. 不支持的能力要显式回 unsupported/error
2. 缺失账本不要伪造默认成功态
3. 条件未成立时不要偷偷降成伪完整模式
4. 恢复窗口里的事实要允许显式不可判定
5. authoritative signal 出现前，不要给近似状态过度解释权

如果没有这些纪律，  
系统最后最容易产生的不是 crash，  
而是更危险的东西：

`看起来一切都还在工作，其实解释已经开始脱离事实。`

## 11. 哲学本质

这一章更深层的哲学是：

`真正可靠的系统，不是永远给答案的系统，而是知道何时停止给答案的系统。`

很多系统把“输出更多结论”误当成更智能。  
Claude Code 的源码更接近另一种成熟判断：

`在不知道时明确说不知道，本身就是高阶安全能力。`

这不是保守主义，  
而是：

`对现实边界的忠诚。`

## 12. 苏格拉底式反思：这章最容易犯什么错

### 12.1 我们是不是把太多错误路径都解释成“未知”

不能混写。  
不是所有 error 都是未知，  
但很多高质量 error 的确承担了显式表达未知/不支持/不可判定的职责。

### 12.2 未知语义会不会让产品变得太“不会说话”

会有风险。  
但真正的问题不是未知太多，  
而是未知没有被组织成可理解、可操作的输出。

### 12.3 “明确不知道”会不会牺牲体验

短期可能会。  
但比起给出一个漂亮却假的解释，  
这类损失通常更小。

## 13. 结语

`37` 回答的是：为什么宿主解释权限必须受账本子集约束。  
`38` 继续推进后的结论则是：

`当系统没有足够根据时，它不应勉强解释，而应把“不知道”本身做成显式安全语义。`

这意味着 Claude Code 更深层的安全启示之一是：

`不知道不是失败的残渣，而是成熟安全控制面必须拥有的一等输出。`
