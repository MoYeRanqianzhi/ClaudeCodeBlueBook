# 终局总指南：Claude Code 安全研究的最佳最全版

## 1. 这份终局版在做什么

前面的 `security/00-16` 已经把 Claude Code 的安全专题拆得足够细。  
这一章的任务不是继续展开，  
而是把最重要的结论重新压缩成一份终局版。

如果只想记住一套最短而完整的判断，  
看这一章就够了。

## 2. 先说总论

Claude Code 的安全不是：

- 单点沙箱
- 单点分类器
- 单点权限弹窗

它更像一套分层安全控制面。

最短的全链结构可以压成：

1. trust
2. 配置来源主权
3. permission mode
4. 动作仲裁链
5. 环境隔离
6. 外部入口收口
7. 身份与 entitlement
8. 状态面与解释层

这八层连起来，才构成它的真实安全性。

## 3. 最重要的第一性原理

把整套安全专题继续压缩，最终只剩六条最重要的第一性原理：

1. 来源先于值
2. 仲裁先于放行
3. 外部世界不是默认可信上下文
4. entitlement 不等于连通
5. 高风险便利必须由更强边界补偿
6. 安全系统必须可解释

这六条已经足以解释前面大部分源码设计。

## 4. 权限系统最重要的结论

权限系统不是简单 allowlist。  
它至少由：

- mode
- rule
- tool.checkPermissions
- safetyCheck
- fast-path
- classifier
- deny tracking

共同组成。

其中最关键的一条，是：

`safetyCheck 的优先级高于 bypass 与大部分自动放行路径。`

这说明 Claude Code 真正保护的不是“某些工具不能用”，  
而是：

`结构性危险目标不能因为当前模式更激进就被抹平。`

## 5. 认证与受管环境最重要的结论

Claude Code 不是把 `process.env` 当作一个平面配置表。  
它明确把：

- settings source
- auth source
- provider source
- host-managed vars

做成分层控制面。

最重要的一条结论是：

`真正高风险的不是用户不会配，而是运行时会被错误来源悄悄改写。`

所以：

- trust 前只有受信来源和 safe env 才能影响运行时
- host 还能把 provider-routing 主链变量重新收口

## 6. 外部入口最重要的结论

MCP、WebFetch、hooks、gateway 并不是一种统一的“插件风险”。  
它们分别主要打开不同攻击面：

- MCP：上下文与工具面扩张
- WebFetch：读写语义混淆
- hooks：执行插入点
- gateway：身份 / 路由漂移与语义错觉

最重要的结论不是“它们都危险”，  
而是：

`它们危险的方式完全不同，因此收口也必须按攻击面分别设计。`

## 7. 主权结构最重要的结论

Claude Code 没有把权力简单交给：

- 用户
- 管理员
- 宿主

中的任何一方。

它真正做的是按风险分配主权：

- 越低风险，越保留用户灵活性
- 越接近运行时主链、执行主链、外部入口主链和身份主链，越容易把最终收口权推回 host、managed settings 或第一方 entitlement

最短一句话就是：

`越靠近主链，主权越集中。`

## 8. 技术先进性最重要的结论

Claude Code 的先进性，不在于它比别的系统“更保守”，  
而在于它把安全做成了 runtime 结构。

最值得保留的四点是：

1. 安全不是后置附录，而是主执行时序的一部分
2. 来源被当作一级安全变量
3. 真正的危险被定义为“绕过仲裁层”
4. 外部能力被按攻击面拆开治理

## 9. 最值得警惕的反模式

如果只记住几条反模式，也足够帮助避免大量误判。

最危险的几条是：

1. 把 mode 当 UX toggle
2. 把兼容入口当第一方身份等价物
3. 把 project/local settings 当成可以随意改写主链的地方
4. 把危险 allow rule 当成效率优化
5. 把外部入口统一当成插件问题
6. 把解释层当成可选 UX

这些做法共同的问题是：

`系统还能跑，但边界会开始偷偷漂移。`

## 10. 这套系统现在最缺什么

从机制上看，Claude Code 已经有很多成熟设计。  
从产品化上看，它最缺的不是再多一层 gate，  
而是更完整的可见性。

最值得继续做成产品的方向是：

1. 统一安全状态面
2. 配置来源血缘展示
3. 权限状态机可视化
4. managed-only 标识统一化
5. entitlement 差异解释层
6. 危险设置 diff 审批
7. 外部入口风险标签
8. 面向支持与管理员的统一证据包

## 11. 给平台构建者的最短法则

如果把整套安全专题只压成一组可迁移法则，我会保留下面这些：

1. 来源先于值
2. mode 即安全语义
3. 真正危险的是绕过仲裁层
4. 外部能力按攻击面治理
5. 兼容不等于 entitlement 等价
6. 解释层必须产品化
7. public build 边界必须与代码边界一起治理

这七条已经足以构成一个成熟 Agent 平台安全设计的骨架。

## 12. 如果只看一张总图，应该怎么理解整套系统

最短的文字总图可以这样记：

```text
来源进入系统
  -> trust 决定哪些来源先影响运行时
  -> settings / host / managed 决定谁拥有配置主权
  -> permission mode 决定仲裁门槛
  -> 权限状态机决定 ask / allow / deny 的真实时序
  -> sandbox 决定已允许动作还能碰到什么
  -> MCP / WebFetch / hooks / gateway 分别打开不同攻击面
  -> OAuth / entitlement 决定高阶远程能力是否真的成立
  -> 解释层决定用户能否看懂当前边界
```

## 13. 用苏格拉底式追问作最终收束

### 问：Claude Code 的安全到底最成熟在哪？

答：不在某个单点机制，而在它把安全做成了一条有顺序的控制链。

### 问：最关键的一条技术定义是什么？

答：真正危险的不是能力强，而是能力绕过了本该存在的仲裁层。

### 问：最关键的一条产品定义是什么？

答：没有解释层的结构化安全，最终仍会在用户侧退化成黑箱。

### 问：最关键的一条平台法则是什么？

答：越靠近主链，越要把主权重新收口；越复杂的边界，越要把它可视化。

## 14. 一句话总结

Claude Code 的安全研究最终可以压成这样一句话：它不是把能力做小，而是把来源、主权、仲裁、隔离、外部入口、身份资格和解释层做成同一套分层控制面；真正成熟的地方，不是禁止得更多，而是让强能力在正确边界内流动，并且让用户看得见这些边界正在如何生效。
