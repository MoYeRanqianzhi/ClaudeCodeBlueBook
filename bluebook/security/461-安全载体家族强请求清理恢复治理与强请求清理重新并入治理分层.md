# 安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层：为什么artifact-family cleanup stronger-request cleanup-recovery-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reintegration-governor signer

## 1. 为什么在 `460` 之后还必须继续写 `461`

`460-安全载体家族强请求清理连续性治理与强请求清理恢复治理分层`
已经回答了：

`stronger-request cleanup 线即便已经拿到 fresh credential proof、fresh reconnect verdict 与 fresh tool fetch，也还需要 recovery governor 去决定对象是否真的 recovered。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然 reconnect result 已经是 connected，或者 OAuth flow 已经闭环，那它当然已经回到当前世界了”
的错觉：

`只要 stronger-request cleanup recovery governor 已经宣布对象 recovered，它就自动拥有了决定这份 recovered truth 是否已经重新进入当前 tool pool、status plane、handler plane 与下一轮 consumer world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的重新并入正对照，
会发现 repo 其实明确拆开了六件事：

1. `raw recovery result`
2. `state writeback`
3. `dynamic tool/state reinsertion`
4. `live handler rebind`
5. `pseudo-tool replacement`
6. `consumer-world read model`

最硬的证据至少有七组：

1. `services/mcp/client.ts:2137-2198` 的 `reconnectMcpServerImpl()` 只返回 `client/tools/commands/resources`，本身不更新 consumer surfaces
2. `services/mcp/useManageMCPConnections.ts:310-332` 的 `onConnectionAttempt()` 仍要单独 `updateServer()`，并只在 `connected` 后注册真正的 `registerElicitationHandler()`
3. `cli/print.ts:3153-3204` 的 `mcp_reconnect` 会在 recovery 之后继续写回 `appState`、`dynamicMcpState`，并单独执行 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`
4. `cli/print.ts:3255-3288` 的 `mcp_toggle` enable path 也是先 recovery，再 reintegrate
5. `cli/print.ts:3391-3441` 的 auth-complete background path 说明 auth+reconnect 之后仍要单独把 result 写回 current state
6. `cli/print.ts:1471-1485,1610-1627` 的 `buildAllTools()` 与 `buildMcpServerStatuses()` 只从 `dynamicMcpState` / `appState` 读取当前世界
7. `cli/print.ts:1253-1280,4786-4835` 与 `tools/McpAuthTool/McpAuthTool.ts:134-165` 又把 handler re-entry 与 pseudo-tool -> real tools swap 写成单独 choreography

这说明：

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经恢复。`

后者才配说：

`这个恢复后的对象现在已经重新并入当前 tool world、status world、event/handler world，并可被后续 consumer 当成 current world 的一部分继续消费。`

所以 `460` 之后必须继续补的一层就是：

`安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`

也就是：

`stronger-request cleanup recovery governor 决定恢复是否成立；stronger-request cleanup reintegration governor 才决定这份恢复真相何时真正并回 current consumer world。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reintegration-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reintegration manager，
而是在说：

1. repo 已经把 reconnect result 与 `appState` / `dynamicMcpState` 的更新拆开
2. repo 已经把 recovered client 与 real tools/resources 的重回 consumer plane 拆开
3. repo 已经把 `connected` 与 handler re-registration 拆开

第二条：

这里说的 `reintegration governance`

不是在声称：

`一切 recovery success 都必须立即让所有 consumer plane 同步完成。`

当前更稳的说法只能是：

`reintegration governance 首先回答的是 recovered truth 何时分别回到 state plane、tool plane、status plane 与 event plane。`

第三条：

这里也不是把：

`dynamicMcpState`

单独当成全部 reintegration grammar。

真正关键的是：

1. state writeback
2. tool/status read model
3. live handler rebind
4. pseudo-tool replacement

第四条：

这里尤其不能过度声称：

`reintegration governance`

已经等于：

`reprojection governance`

当前更稳的说法只能是：

`reintegration governance 回答 recovered truth 何时进入 current world；reprojection governance 才回答这份 current truth 随后怎样被不同 reader surface 重新讲述。`

第五条：

这里也不能过度声称：

`connected`

就天然等于：

`next-turn visible`

当前更稳的说法只能是：

`raw recovery result 只是 reintegration 输入材料；只有写回 current state、更新 dynamic tool pool 并重挂 handlers 后，consumer 才真正看见它。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-recovery-governor signer 仍不等于 stronger-request cleanup-reintegration-governor signer`

证据：

1. `reconnectMcpServerImpl()` 只返回 `client/tools/commands/resources`，说明 recovery result 本身不自动进入 consumer planes
2. `onConnectionAttempt()` 要先 `updateServer()` 再在 `connected` 情况下注册 elicitation handler；这说明恢复成立与 UI/event plane 回归仍是两步
3. `mcp_reconnect` / `mcp_toggle` 会显式写回 `appState.mcp`、`dynamicMcpState`，并单独执行 `registerElicitationHandlers()` / `reregisterChannelHandlerAfterReconnect()`；这说明 reconnect success 不自动等于 reintegration complete
4. auth-complete background path 会在 recovery 之后继续 `setAppState` 与更新 `dynamicMcpState`；这说明 auth+reconnect 闭环不自动等于 current-world membership
5. `buildAllTools()` 与 `buildMcpServerStatuses()` 只从 `appState` / `dynamicMcpState` 读当前世界；这说明 consumer 看到的是 reintegrated truth，不是 raw recovery result
6. `McpAuthTool` 在 OAuth 完成后还要再把 pseudo-tool 换成 real tools；这说明 auth+reconnect 的 recovered proof 仍不自动等于 tool-world reintegration

因此这一章的最短结论是：

`stronger-request cleanup recovery governor 最多能说恢复真相已经成立；stronger-request cleanup reintegration governor 才能说这份恢复真相现在是否已经重新并入当前 consumer world。`

再压成一句：

`recovered，不等于 reintegrated。`

再压成更哲学一点的话：

`制度知道它已经回来，不等于制度已经知道它该何时重新成为当前世界的一部分。`

## 4. 第一性原理：recovery governance 回答“恢复是否成立”，reintegration governance 回答“恢复后的真相怎样重新进入当前世界”

从第一性原理看，
强请求清理恢复治理与强请求清理重新并入治理处理的是两个不同主权问题。

`stronger-request cleanup recovery governor`
回答的是：

1. 新凭据是否已经闭环
2. 新 callback / token exchange 是否完成
3. reconnect result 是否已经回到 `connected`
4. 当前 failure grammar 是否可撤销
5. recovered truth 本身是否成立

`stronger-request cleanup reintegration governor`
回答的则是：

1. 恢复后的 client 是否已写回 `appState`
2. 恢复后的 tools / commands / resources 是否已并回 current tool pool
3. `dynamicMcpState` 是否已更新，供下一轮 run 读取
4. elicitation handler / channel handler 是否已重新挂回
5. status plane、tool plane、event plane 是否已重新把它视为 current world 的一部分

如果把这两层压成一句：

`反正系统已经恢复了`

系统就会制造六类危险幻觉：

1. `connected-means-integrated`
2. `auth-done-means-tool-visible`
3. `state-writeback-means-handler-ready`
4. `recovered-means-next-turn-visible`
5. `raw-result-means-current-world`
6. `recovery-success-means-consumer-usable-now`

所以从第一性原理看：

`recovery governance` 管的是恢复真相是否成立；
`reintegration governance` 管的是这份真相怎样重新进入 current world 的各个消费面。

## 5. Claude Code 多重安全技术的先进性：它把“回到当前世界”绑定到 current-world choreography，而不是绑定到单一 `connected` bit

Claude Code 在这一层真正先进的地方，
不是“只要 reconnect 成功就宣告一切归位”，
而是：

`它把 recovered truth 重新进入当前世界的过程继续拆成 raw-result、state writeback、dynamic projection、handler rebind 与 tool-world swap 五层相互校验的 choreography。`

这种先进性至少体现在六个方面：

1. `raw recovery result is not overclaimed`
   `reconnectMcpServerImpl()` 明知自己已经拿到 fresh result，仍拒绝越权写 current world。
2. `state plane and event plane are split`
   `onConnectionAttempt()` 与 print control path 都说明写回状态不自动等于 live interaction 已重建。
3. `next-turn world is modeled explicitly`
   `dynamicMcpState` 不是多余缓存，而是 next-turn tool/status world 的 authoritative read model。
4. `consumer truth is derived from reintegrated state`
   `buildAllTools()` 与 `buildMcpServerStatuses()` 只认 current-world projection，不认 raw reconnect return。
5. `handler re-entry is separately governed`
   channel handler 和 elicitation handler 都有单独的 connected-only 重挂 gate。
6. `placeholder and real world are structurally separated`
   `McpAuthTool` 与真实工具不是一个状态位的不同展示，而是两个不同制度对象。

这些设计背后的源码思路也很清楚：

1. 不让 `recovered` 越级冒充 `already back in current world`
2. 不让 `state updated` 越级冒充 `event plane repaired`
3. 不让 `connected` 越级冒充 `consumer-visible`
4. 不让 `placeholder removed later` 越级冒充 `real tool already present now`
5. 不让 next-turn visibility 依赖隐式共享状态，而是依赖显式的 dynamic projection

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. cleanup recovery 之后，必须继续追问 recovered truth 何时被 current world 正式接纳。
2. reintegration 需要自己的 writeback grammar，而不应混在 recovery verdict 里一笔带过。
3. current state、next-turn tool pool 与 live handler plane 应分别建模，而不是假设一个 `connected` 位自动投影全部世界。
4. placeholder artifact 与 real artifact 应显式分层，防止过早暴露“已经回来了”的误导。
5. consumer-facing truth 应来自 reintegrated state，而不是来自内部 raw result object。
6. reintegration 的本体不是“对象已经好了”，而是“当前世界已经把它重新接纳为自己的一部分”。

## 6. 苏格拉底式自反诘问：我是不是又把“恢复成立”误认成了“当前世界已经重新接回它”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 recovery governance 已经等于 reintegration governance，为什么 `reconnectMcpServerImpl()` 还只返回 raw result，而不直接改 current state？
   因为 repo 自己已经承认 recovered truth 还只是输入材料。
2. 如果 `connected` 已经足够，为什么 `onConnectionAttempt()` 仍要 `updateServer()` 再单独注册 handler？
   因为 standard path 上的 current-world projection 也是单独步骤。
3. 如果 `setAppState` 已经足够，为什么 print control path 还要更新 `dynamicMcpState`？
   因为 next-turn tool/status visibility 不由单一 snapshot 自动担保。
4. 如果 dynamic state 已经更新，为什么 live interaction plane 还要 `registerElicitationHandlers()` 与 `reregisterChannelHandlerAfterReconnect()`？
   因为 event plane 的回归仍有自己的 gate。
5. 如果 auth+reconnect 闭环已经说明 everything is back，为什么 `McpAuthTool` 还要手动清 placeholder、换回 real tools？
   因为 pseudo-world 与 real world 从来不是同一句 truth。
6. 如果 stronger-request cleanup 线还没正式长出 reintegration grammar，是不是说明这层只属于工程细节？
   恰恰相反。越是把它当细节，越容易让“恢复”偷签“当前世界正式接纳它”。

这一串反问最终逼出一句更稳的判断：

`reintegration 的关键，不在 recovered proof 有没有出现，而在系统能不能正式决定这份 proof 何时重新成为当前世界的一部分。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 线未来只要补出 recovery governance，就已经足够成熟。`

而是：

repo 在 `reconnectMcpServerImpl()` 的 raw recovery result、`useManageMCPConnections.ts` 的 state projection + handler attach、`print.ts` 对 `appState` / `dynamicMcpState` 的回填、`buildAllTools()` / `buildMcpServerStatuses()` 的 consumer read model、`reregisterChannelHandlerAfterReconnect()` 的 live interaction re-entry，以及 `McpAuthTool` 的 pseudo-tool -> real tools swap 上，已经明确展示了 reintegration governance 的存在；因此 `artifact-family cleanup stronger-request cleanup-recovery-governor signer` 仍不能越级冒充 `artifact-family cleanup stronger-request cleanup-reintegration-governor signer`。

因此：

`stronger-request cleanup 线真正缺的不是“它已经恢复了吗”，而是“它现在已经重新成为当前世界的一部分了吗”。`
