# 安全验证制度化接口：为什么验证蓝图若不进入script、目录、tsconfig与CI挂载点，就仍然不是工程能力

## 1. 为什么在 `114` 之后还必须继续写“验证制度化接口”

`114-安全最小可执行验证蓝图` 已经回答了：

`下一步不该泛泛补测试，而应先建一套首批可执行验证套件。`

但如果继续往下追问，  
还会碰到一个更工程化的问题：

`这些套件如何正式进入仓库的工程流程？`

如果答案只是：

`文档里写清楚以后该补哪些测试`

那仍然不够。  
因为只要这些验证还没有进入：

1. script 入口
2. 目录骨架
3. tsconfig / runner 配置
4. CI 或检查挂载点

它们就仍然只是：

`研究建议`

而不是：

`工程能力`

所以 `114` 之后必须继续补出的下一层就是：

`安全验证制度化接口`

也就是：

`验证蓝图若不进入 script、目录、tsconfig 与 CI 挂载点，就仍然不是制度，而只是口头共识。`

## 2. 最短结论

从源码看，这份研究版 Claude Code 当前已经有构建接口和类型检查接口，  
但还没有验证接口：

1. `package.json` 只有 `prepare-src/build/check/start` 四类入口，没有任何测试脚本  
   `package.json:7-11`
2. 根目录有 `scripts/`，说明仓库已经接受“用脚本承载工程制度”这件事  
   根目录结构：`scripts/build.mjs`, `scripts/prepare-src.mjs`, `scripts/stub-modules.mjs`, `scripts/transform.mjs`
3. `prepare-src.mjs` 会对源码做前置整理，说明验证若要稳定落地，通常也应挂到这条入口链上  
   `scripts/prepare-src.mjs:1-112`
4. `build.mjs` 已把整个构建流程写成 phase pipeline，说明仓库天然适合把验证也组织成 phase / script chain  
   `scripts/build.mjs:1-260`
5. `tsconfig.json` 的 `rootDir` 是 `src`，`include` 只有 `src/**/*` 与 `stubs/**/*`，说明如果将测试放到 `tests/`，现有 TS 配置并不会自动承接  
   `tsconfig.json:16-18,28-35`
6. 根目录不存在 `.github/`，说明当前没有显式 CI 挂载位；这使“未来如何把验证接进自动流程”本身就需要制度化设计  
   本地检查：`.github => no`

所以这一章的最短结论是：

`Claude Code 研究版当前已经有 build/check 的制度化入口，但还没有 test/verify 的制度化入口。下一步最重要的不是再扩写验证蓝图，而是把蓝图正式变成工程接口。`

再压成一句：

`没有接口的验证，不算制度。`

## 3. 当前仓库已经具备“制度化”的骨架，只是验证还没接上

## 3.1 `package.json` 已经是工程制度入口，但目前只承载 build/check

`package.json:7-11` 当前只有：

1. `prepare-src`
2. `build`
3. `check`
4. `start`

这说明仓库已经默认接受一条重要原则：

`工程流程应该通过稳定脚本名暴露给外部。`

所以问题并不是“要不要制度化”，  
而是：

`为什么验证仍未被赋予同样的制度入口。`

## 3.2 `scripts/` 目录已经证明：复杂流程在这里是被正式承认的

根目录当前已有：

1. `scripts/build.mjs`
2. `scripts/prepare-src.mjs`
3. `scripts/stub-modules.mjs`
4. `scripts/transform.mjs`

这说明作者并不抵触：

1. 多阶段脚本链
2. 预处理步骤
3. 专门的构建辅助工具

所以把验证制度化进 `scripts/`，  
并不会违背现有仓库哲学，  
而恰恰是：

`顺着已有工程风格自然延伸。`

## 3.3 `prepare-src` 说明验证入口不能脱离源码预处理链单独想象

`scripts/prepare-src.mjs:1-112` 明确告诉我们：

1. 这份源码需要预处理才能在非 Bun 环境下稳定参与构建/检查
2. 它会 patch `bun:bundle`、`MACRO.*`、stub 等

这意味着未来的验证入口若要稳定，  
通常也应回答：

`测试是在原始源码上跑，还是在 prepare-src 之后跑。`

所以验证制度化不是简单加一个 `npm test`，  
而是要先决定：

`它与 prepare-src 的制度关系。`

## 3.4 `build.mjs` 又说明“phase pipeline”已是仓库认可的组织方式

`scripts/build.mjs:1-260` 已经把 build 写成：

1. copy source
2. transform
3. create entry
4. iterative stub + bundle

这说明未来验证同样适合按 phase 组织，例如：

1. prepare
2. unit/property
3. transition/integration
4. security aggregate

换句话说：

`验证制度化最自然的形式不是一堆零散命令，而是一条分阶段脚本链。`

## 4. `tsconfig.json` 暴露了最关键的接口缺口：测试目录尚未被纳入类型制度

`tsconfig.json:16-18,28-35` 当前非常关键：

1. `rootDir = src`
2. `include = src/**/*, stubs/**/*`
3. `exclude = node_modules, dist`

这直接意味着一件事：

`如果未来把测试按研究建议放到 tests/ 目录，当前 tsconfig 并不会自动承接它们。`

所以制度化接口至少还缺：

1. `tsconfig.tests.json`
2. 或 runner 自带 TS/transpile 配置
3. 或新的 include 组织方式

这也是为什么“测试蓝图”之后必须补这一章。  
因为没有类型层接口，  
测试目录只是：

`摆在仓库里的一堆孤儿文件。`

## 5. 当前没有 `.github/`，说明验证制度还没有自动执行宿主

本地检查结果：

`.github => no`

这不代表未来必须上 GitHub Actions，  
但它至少说明：

`当前仓库没有显式 CI 宿主层。`

于是验证若要制度化，  
就必须先明确：

1. 本地如何跑
2. 在未来有 CI 时，该挂载到哪里
3. 哪个脚本应被视为单一事实入口

否则就会出现非常典型的问题：

1. 文档里说要跑很多测试
2. 本地没有统一命令
3. 将来也不知道 CI 应该调用哪一个

## 6. 从第一性原理看，制度化接口到底在解决什么问题

不是“方便运行”这么简单。  
它真正解决的是三件事：

## 6.1 它把责任从“人记得要做”转成“系统提供入口”

只要没有 `script` 入口，  
验证就是：

`需要有人想起它`

一旦有了统一入口，  
验证才会变成：

`系统默认能做它`

## 6.2 它把结构从“测试想法”转成“仓库契约”

只要没有：

1. 目录骨架
2. tsconfig / runner 配置
3. hook / CI 挂载位

测试就只是研究想法。  
一旦这些接口存在，  
它就变成：

`仓库对未来维护者作出的契约。`

## 6.3 它把验证从“补丁行为”转成“日常流程”

成熟系统不会把验证当作：

`出现事故后再补一条`

而是当作：

`构建、检查、发布之前的常规过程`

所以制度化接口的本质是：

`把验证嵌回日常工程节律。`

## 7. 下一步最合理的接口设计

如果沿着当前仓库风格最自然地往前走，  
我会建议做四类接口。

## 7.1 脚本接口

最小建议：

1. `test`
2. `test:security`
3. `test:property`
4. `test:integration`
5. `check:security`

其中最重要的是：

`check:security = prepare-src -> check -> test:security`

原因是这能把验证纳回现有 build/check 文化，而不是另起体系。

## 7.2 目录接口

最小建议：

1. `tests/security/property/`
2. `tests/security/table/`
3. `tests/security/integration/`
4. `tests/security/fixtures/`

这样做的意义不只是分类，  
而是让“验证优先级与验证类型”在目录层就被制度化。

## 7.3 类型/运行接口

最小建议：

1. `tsconfig.tests.json`
2. 或单独的测试 runner 配置文件

因为当前 `tsconfig.json` 并不会自然承接 `tests/`。  
如果不先补这一层，  
测试体系很快就会陷入：

`能写，但不能稳地检查和演进`

的状态。

## 7.4 自动执行接口

即便当前没有 `.github/`，  
也应预先定义：

1. 哪个脚本是 CI 单一入口
2. 本地开发者应跑哪个命令
3. pre-merge 应至少跑到哪一层

最小建议：

1. 本地日常：`npm run test:security`
2. 预提交或预合并：`npm run check:security`
3. 将来 CI：统一调用 `check:security`

这样就不会在未来出现：

`CI 有自己的命令，本地有另一套命令，文档又写第三套命令`

## 8. 为什么“制度化接口”比“再补几组测试”更优先

因为没有接口时，  
任何新测试都很容易再次变成：

1. 无法统一运行
2. 无法被团队记住
3. 无法被未来 CI 接管
4. 无法成为稳定的工程流程

也就是说，  
没有制度化接口时，  
即便你写出几组很好的测试，  
它们也仍然像：

`散落的防御工事`

而不是：

`连成体系的防线`

## 9. 技术启示：成熟安全工程的下一个台阶，不是更多规则，而是让规则有稳定入口

Claude Code 当前最值得学习的不是“还缺测试”，  
而是：

`一旦规则已经被发现、优先级已经被排出，下一步最重要的工作就是给它们建立制度入口。`

这对任何 Agent 平台都成立：

1. 没有入口，规则靠记忆
2. 有入口，规则变流程
3. 有统一入口，流程才可被自动执行

所以制度化接口的哲学本质是：

`把安全原则从知识，变成程序。`

## 10. 苏格拉底式反思：如果要把这一步做得更好，还应继续追问什么

可以继续追问八个问题：

1. `prepare-src` 与测试入口的关系，应该是前置强依赖还是按层拆分
2. `tests/` 是否应继续拆成 `security/` 与其他未来业务测试
3. `tsconfig.tests.json` 是否应继承当前 tsconfig，还是单独最小化
4. 未来若引入 CI，是否应把 `check:security` 设为唯一入口
5. 当前哪些测试最需要 fixture 层支持
6. 是否应给每个高优先级 suite 绑定显式 owner
7. 何时应把“没有测试入口”本身视为安全债务
8. 研究版源码若未来转成可演化仓库，验证接口是否应先于功能扩写落地

这些问题共同逼问的是：

`系统到底只是知道自己该如何验证，还是已经为验证留出了稳定制度入口。`

## 11. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正成熟的下一步不是：

`继续把验证蓝图停留在文档里，`

而是：

`把它正式接进 script、目录、tsconfig 与 CI 挂载位，让高优先级验证从研究建议变成仓库的日常工程接口。`

因为只有这样，  
这套安全研究才会从：

`会说应该怎么做`

进一步升级到：

`会让系统默认按这个方式去做。`
