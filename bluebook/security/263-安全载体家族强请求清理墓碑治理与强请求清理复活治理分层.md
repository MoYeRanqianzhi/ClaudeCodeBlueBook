# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `262` 之后还必须继续写 `263`

`262-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`
已经回答了：

`stronger-request cleanup 线即便知道旧世界何时正式结束，也还需要 tombstone governor 去决定结束后留下什么最小残留事实。`

但如果继续往下追问，
还会撞上一层更容易被偷写成
“只要墓碑已经留下或清掉，后面怎样回来也就自动清楚了”
的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定旧对象退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、回来时算谁、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 resurrection 正对照，
会发现它其实明确拆开了四件事：

1. `authoritative clearing`
2. `re-entry gate`
3. `identity policy`
4. `active-layer admission`

最硬的证据至少有六组：

1. `src/utils/plugins/cacheUtils.ts:67-73,82-92,122-130`
   证明 `.orphaned_at` 的清除由 authoritative installed set 决定，
   而不是 marker 自己决定
2. `src/utils/plugins/refresh.ts:1-18,72-79,123-137`
   证明 active plugin return 是专门的 Layer-3 refresh primitive，
   不是 marker 清掉后的自动副作用
3. `src/commands/reload-plugins/reload-plugins.ts:37-56`
   证明 `/reload-plugins` 是显式 re-entry signal，
   系统不会把“磁盘世界变了”偷写成“active world 已接纳”
4. `src/services/plugins/PluginInstallationManager.ts:56-59,135-179`
   与
   `src/hooks/useManagePlugins.ts:287-303`
   证明 repo 还会继续区分：
   哪些 return path 可以 auto-refresh，
   哪些 return path 必须停在 `needsRefresh`
5. `src/utils/plans.ts:164-231,239-257`
   证明 plan resurrection 不只是 undelete，
   还要治理 lineage、evidence 与 identity policy
6. `src/utils/conversationRecovery.ts:539-547`
   与
   `src/utils/sessionStorage.ts:1468-1473`
   共同证明 resurrection grammar 必须进入正式恢复链，
   而 deletion grammar 本身不自动携带 re-entry grammar

这说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 identity policy 与什么 active-layer admission 下，这个旧对象还能重新回到 current world。`

所以 `262` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`
不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk / materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery` 就等于 resurrection 的全部语义。

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线是否完全复用这套 grammar，仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer`

证据：

1. `cacheUtils.ts:67-73,82-92,122-130`
   证明 `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
2. `refresh.ts:1-18,72-79,123-137`
   与
   `reload-plugins.ts:37-56`
   证明 marker clearing / disk presence 不等于 active components 已重新进入当前 session
3. `PluginInstallationManager.ts:56-59,135-179`
   与
   `useManagePlugins.ts:287-303`
   证明 repo 明确要求某些 update path 停在 `needsRefresh`，
   而不是默认自动接纳
4. `plans.ts:164-231,239-257`
   证明 plan resurrection 需要 lineage、evidence 与 identity policy，
   而不是简单 undelete
5. `conversationRecovery.ts:539-547`
   把 resurrection 接进恢复主线，
   说明这不是局部 helper，而是正式 recovery grammar
6. `sessionStorage.ts:1468-1473`
   的 tombstone path 只回答怎样删除旧对象，
   不回答它何时、以什么身份重新回来

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据、何种身份政策与何种激活层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

## 4. 第一性原理：tombstone governance 回答“留什么最小残留事实”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

`stronger-request cleanup tombstone governor`
回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

`stronger-request cleanup resurrection governor`
回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk / materialized world，还是 active / runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句“反正 marker 还在或 marker 已清掉”，
系统就会制造五类危险幻觉：

1. `marker-clear-means-resurrection`
2. `disk-return-means-active-return`
3. `content-recovered-means-identity-restored`
4. `tombstone-is-bidirectional`
5. `any-old-object-can-come-back`

再用苏格拉底式反问压一次：

1. 如果 tombstone governance 已经等于 resurrection governance，
   为什么 repo 还要把 `removeOrphanedAtMarker()`、`refreshActivePlugins()` 与 `/reload-plugins` 拆成多层？
   因为“marker 清掉”与“active world 接纳它”根本不是同一个时刻。
2. 如果对象内容恢复就等于对象身份恢复，
   为什么 `copyPlanForFork()` 还要显式生成 new slug？
   因为 resurrection 还要治理 identity，不只是治理 bytes。
3. 如果 tombstone path 天然自带复活逻辑，
   为什么 `removeTranscriptMessage()` 只回答怎样删，不回答怎样回来？
   因为 deletion grammar 与 re-entry grammar 是两层不同主权。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“复活治理”是一层独立制度

### 5.1 authoritative clearing 不是 resurrection

`cleanupOrphanedPluginVersionsInBackground()` 的 Pass 1 会只对 authoritative installed set 清 `.orphaned_at`。
这说明 repo 已经知道：

`谁配撤销“你已退役”这条 marker`

本身就是一种制度。

但 `removeOrphanedAtMarker()` 只删 marker，
并不负责任何 active components return。
这正好证明：

`能够撤销墓碑`

和

`能够重新接纳对象`

并不是同一主权。

### 5.2 Layer-3 refresh 是显式 re-entry gate

`refresh.ts` 直接把 active plugin swap 定义为 `Layer-3 refresh primitive`，
同时明确区分：

1. Layer 1: intent
2. Layer 2: materialization
3. Layer 3: active components

这意味着 resurrection 在 Claude Code 里不是一句“对象回来了”。
真正被治理的是：

`它回到了哪一层世界。`

`/reload-plugins` 返回 `Reloaded: ...`
更是把 re-entry 写成显式命令事实，
而不是磁盘状态变化的自动影子。

### 5.3 `needsRefresh` 把回归路径分级

`performBackgroundPluginInstallations()` 会对 new installs 与 updates only 分流：

1. 新安装可尝试 auto-refresh
2. 更新则停在 `needsRefresh`
3. auto-refresh 失败也回退到 `needsRefresh`

`useManagePlugins()` 又显式发出：

`Plugins changed. Run /reload-plugins to activate.`

并附带：

`Do NOT auto-refresh`

这说明 Claude Code 的 advanced point
不只是“有复活”，
而是它还在继续治理：

`哪一种 return path 配自动生效，哪一种必须显式接纳。`

### 5.4 plan 线证明 resurrection 还要治理 lineage 与 identity

`copyPlanForResume()` 的技术含义不是简单的“缺文件就补回来”。
它真正表达的是：

1. 先绑定 original session lineage
2. 先读真实文件
3. 只有 `ENOENT` 才进入 recovery
4. 只在 remote sessions 里做恢复
5. 先信 file snapshot，再退回 message history

这说明 resurrection 不是内容回归本身，
而是：

`lineage + evidence + eligibility`

的合取。

而 `copyPlanForFork()` 直接生成 new slug，
进一步证明：

`能回来`

不等于

`以原身份回来`

### 5.5 deletion-only path 反证 resurrection 不能被偷渡

`removeTranscriptMessage()` 的注释和实现都极其单向：

1. 目标是 remove
2. 载体是 orphaned message tombstone
3. 操作是按 UUID 删除

它完全不回答 restore、
rebind、
replay admission、
identity policy。

这条反证很重要，
因为它说明：

`成熟系统不会把 re-entry 语义偷塞进 deletion helper 里。`

它宁可显式缺席，
也不允许你从“能删”直接推成“会让它回来”。

## 6. 哲学本质：墓碑是历史残留的治理，复活是当前世界接纳权的治理

如果把 `tombstone` 和 `resurrection` 抽到哲学层，
两者治理的不是同一类存在论问题。

`tombstone`
治理的是：

`一个对象退出当前真相之后，世界还欠后来者什么最小解释义务。`

`resurrection`
治理的是：

`一个历史对象若想重新进入当前真相，谁有权为它重新签发在场资格。`

所以二者的本质差别是：

1. 前者面向历史残留
2. 后者面向当前接纳
3. 前者在防蒸发
4. 后者在防误复活

Claude Code 在这里真正先进的地方，
不是它把 every return path 都自动做通，
而是它已经默认承认：

`重新出现，不等于重新被承认。`

这恰恰是一种很成熟的安全哲学：

`对象可以重新物化，但 current world 的接纳权不能因此被偷走。`

## 7. 苏格拉底诘问：怎样把这一层做得更严格

### 诘问一

`如果 marker 被 authoritative set 清掉，为什么还不能直接宣布对象已经回来？`

因为 marker clear 只证明：

`“你不再被视作 orphan/retired”这条历史判断被撤销了。`

它并不自动证明：

`你已经重新被当前 runtime 接纳。`

### 诘问二

`如果对象内容已经恢复，为什么还要单独治理 identity policy？`

因为对象的风险从来不只在内容，
还在：

1. 谁拥有它
2. 谁为它担保
3. 谁可以继续援引它的旧资格

内容回来而身份不重建，
很多安全错误就会被偷渡成“只是恢复内容”。

### 诘问三

`如果 resurrection grammar 太严格，会不会导致系统难以恢复？`

会增加恢复门槛，
但这是必要代价。
否则系统获得的是一种更危险的“容易回来”：

`任何旧对象只要重新出现，就可能被误当作 current truth。`

### 诘问四

`那怎样才算更好的 resurrection governance？`

至少要继续把四件事制度化：

1. marker clear authority
2. re-entry gate
3. identity policy
4. active-layer admission

缺任何一层，
都容易把“历史对象可见”误写成“当前世界已接纳”。 

## 8. 技术启示与下一层约束

对 stronger-request cleanup 线而言，
这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 resurrection。`

而是更具体地说：

1. 旧 path、旧 promise、旧 receipt 若想回来，必须回答谁有权清掉其墓碑
2. 墓碑清掉后，还要回答是否存在显式 re-entry gate
3. 即便内容可恢复，也还要回答它是否保留旧 identity
4. 即便物化回来，也还要回答它进入的是 disk world 还是 active world

只有这样，
`stronger-request cleanup tombstone-governance`
才不会继续被误写成一种过度强大的总主权：

`既负责留下历史残留，又自动负责把历史对象重新接纳回当前世界。`
