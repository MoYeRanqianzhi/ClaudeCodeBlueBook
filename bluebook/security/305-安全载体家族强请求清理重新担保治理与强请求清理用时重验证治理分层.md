# 安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层：为什么artifact-family cleanup stronger-request cleanup-reassurance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer

## 1. 为什么在 `304` 之后还必须继续写 `305`

`304-安全载体家族强请求清理重新投影治理与强请求清理重新担保治理分层`
已经回答了：

`stronger-request cleanup 线即便已经把 current truth 重新讲给不同 readers，也还要单独回答这些讲述里哪些现在配承载多强的继续依赖担保。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然某个 surface 已经给出了足够正向的 reassurance，真正 consumer 在实际使用这一刻当然就可以直接依赖”
的错觉：

`只要 stronger-request cleanup reassurance governor 已经允许某个 surface 给出足够正向的 reassurance，真正的 consumer 在实际 read / list / execute 这一刻就自动拥有了继续依赖这份 truth 的资格，不再需要 fresh re-check。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的 use-time revalidation 正对照，
会发现 repo 其实明确拆开了六件事：

1. `consumer-edge hard gate`
2. `fresh reconnect primitive`
3. `batch list live-use discipline`
4. `connected-only execution metadata`
5. `runtime contradiction revocation`
6. `current-use truth as reconnect-or-throw`

最硬的证据至少有五组：

1. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:75-100`
   的 `client.type !== 'connected'` hard gate
   与 `ensureConnectedClient(client)`
2. `src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:79-89`
   关于 `ensureConnectedClient()` 的 fresh reconnect 注释与调用
3. `src/services/mcp/client.ts:1688-1703`
   的 `ensureConnectedClient()`
4. `src/services/tools/toolExecution.ts:308-335`
   的 `getMcpServerType()` /
   `getMcpServerBaseUrlFromToolName()`
5. `src/services/tools/toolExecution.ts:1599-1627`
   的 `McpAuthError -> needs-auth` demotion

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup reassurance governance`
   负责决定不同 surface 现在最多配说多强的正向 reassurance
2. `stronger-request cleanup use-time revalidation governance`
   负责决定真正 consumer 在实际依赖之前，
   是否已经重新拿到 fresh enough 的 current-use proof，
   才能把这份 reassurance 转成真实可用资格

也就是说：

`artifact-family cleanup stronger-request cleanup-reassurance-governor signer`

和

`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`

仍然不是一回事。

前者最多能说：

`当前某个 surface 现在配给出多强的正向 reassurance。`

后者才配说：

`真正 consumer 在实际调用这一刻，已经重新完成 fresh connection check / current capability check / current authorization check，因此这份 reassurance 现在可以被转译成 current-use truth。`

所以 `304` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层`

也就是：

`stronger-request cleanup reassurance governor 决定现在可以说多放心；stronger-request cleanup use-time-revalidation governor 才决定真正使用时是否已经 fresh enough to rely。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 live-use manager，
而是在说：

1. repo 已经明确把正向 reassurance 与真实 consumer edge 拆开
2. repo 已经明确要求某些 use path 在实际依赖前重拿 fresh proof
3. repo 已经明确允许 runtime contradiction 当场撤销旧 reassurance

第二条：

这里说的 `use-time revalidation governance`
不是在声称：

`系统必须把 every surface 的 positive wording 都重算一遍。`

当前更稳的说法只能是：

`use-time revalidation governance 首先回答的是 read / list / execute 这些真正 consumer edge 在依赖前要不要重新验真。`

第三条：

这里也不是把：

`ensureConnectedClient()`

单独当成全部 revalidation grammar。

真正关键的是：

1. `connected` hard gate
2. reconnect-or-throw primitive
3. cache-invalidated list freshness
4. connected-only execution metadata
5. runtime demotion on live contradiction

第四条：

这里尤其不能过度声称：

`use-time revalidation governance`

已经等于：

`step-up reauthorization governance`

当前更稳的说法只能是：

`use-time revalidation governance 回答 consumer 此刻是不是 fresh enough to use；step-up reauthorization governance 才回答这份 fresh proof 对当前更强请求的 authority level 是否已经足够。`

第五条：

这里还不能过度声称：

`当前还显示 connected`

就天然等于：

`当前使用资格仍然成立`

当前更稳的说法只能是：

`stale label 最多还能支撑某种 reassurance；真正 current-use truth 仍要在 use boundary 被重新验真。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-reassurance-governor signer 仍不等于 stronger-request cleanup-use-time-revalidation-governor signer`

证据：

1. `ReadMcpResourceTool`
   即便已经看到 `connected` client，
   仍继续 `ensureConnectedClient(client)`
2. `ListMcpResourcesTool`
   明确写出 cache invalidation 与 `onclose` 后 fresh reconnect 的制度
3. `ensureConnectedClient()`
   把真实 current-use truth 压成
   `reconnect or throw`
4. `toolExecution`
   连 transport / base-url 这种辅助 execution metadata
   也只承认 current connected truth
5. `McpAuthError -> needs-auth`
   又进一步说明 live contradiction 会即时撤销旧 reassurance

因此这一章的最短结论是：

`stronger-request cleanup reassurance governor 最多能说当前某个 surface 现在配说多放心；stronger-request cleanup use-time-revalidation governor 才能说真正 consumer 在使用瞬间已经重新拿到 fresh enough 的 current-use proof。`

再压成一句：

`被担保到可期待，不等于已经被验证到可在使用瞬间当真。`

再压成更哲学一点的话：

`制度已经愿意让你相信一点，不等于制度已经替你的这一次依赖重新验过真。`

## 4. 第一性原理：reassurance governance 回答“你现在最多可以放心到什么程度”，use-time revalidation governance 回答“你这一刻是否已经重新验真到足以真的去用”

从第一性原理看，
强请求清理重新担保治理与强请求清理用时重验证治理处理的是两个不同主权问题。

`stronger-request cleanup reassurance governor`
回答的是：

1. 这句正向话现在配有多强
2. 这句成功话是否还必须带 caveat
3. 哪些 surface 应保持 positive silence
4. control reader 是否仍该收到 `requiresUserAction`
5. 当前话术最多能替依赖背书到什么上限

`stronger-request cleanup use-time-revalidation governor`
回答的则是：

1. 真实 read consumer 此刻是否 still connected
2. 真实 list consumer 在 `onclose` 之后是否已 fresh reconnect
3. execution metadata 是否仍建立在 current connected truth 上
4. live contradiction 出现时旧 reassurance 是否必须立即撤销
5. 这一刻的 current-use truth 究竟是什么

换句话说：

`reassurance governance` 处理的是
`dependence-bearing wording ceiling`

而
`use-time revalidation governance` 处理的是
`consumer-edge truth refresh`

这两层如果不拆开，
系统最容易犯的错就是：

`把“现在听起来可以继续信一点”偷写成“现在就已经可以真的去用”。`

## 5. `ReadMcpResourceTool` 与 `ListMcpResourcesTool` 先证明：consumer edge 拒绝把 reassurance 当成免检通行证

`src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:75-100`
很值钱。

这里的制度顺序非常严格：

1. 找到目标 server
2. 若 `client.type !== 'connected'`，直接失败
3. 即便当前看起来已是 `connected`，仍继续 `ensureConnectedClient(client)`
4. 然后才真正发起 `resources/read`

这说明 repo 对 read consumer 的制度不是：

`既然前面某个 surface 刚说可用，现在就先信再说`

而是：

`真正 read 之前，再确认一次它现在是否仍然 connected`

`src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:79-89`
更值钱。

它不仅复用 `ensureConnectedClient(client)`，
还公开写出：

1. resource cache `onclose` / `resources/list_changed` 时失效
2. healthy 时 `ensureConnectedClient` 是 no-op
3. after `onclose` it returns a fresh connection so the re-fetch succeeds

这条注释直接说明：

`真正 consumer 在使用前重新拿 fresh proof，不是 incidental implementation detail，而是 repo 明确承认的制度语义`

也就是说，
consumer edge 的世界观是：

`positive reassurance can invite use; only fresh revalidation can authorize use`

## 6. `ensureConnectedClient()` 再证明：current-use truth 的 canonical 形态就是 reconnect-or-throw

`src/services/mcp/client.ts:1688-1703`
很硬。

`ensureConnectedClient()` 的制度非常简单：

1. SDK client 原样返回
2. 其他 client 一律重新 `connectToServer(client.name, client.config)`
3. 若 fresh 结果不再是 `connected`，直接抛
   `MCP server "<name>" is not connected`
4. 只有 fresh 结果成立时，才返回 connected client

这说明 repo 并没有把：

`旧的 connected label`

当成 durable live-use truth。

它真正承认的是：

`当前使用资格只有两种状态：要么 fresh enough，所以继续；要么 fresh proof 不成立，所以立刻失败`

这条证据非常值钱。

因为它把“真正使用时的 current truth”从正向文案、health glyph、dialog success 里彻底抽出来，
单独交给了一个 fresh proof primitive。

## 7. `toolExecution` 再证明：执行平面连辅助上下文都要 current connected truth，而且会在 live failure 上立即撤销旧 reassurance

`src/services/tools/toolExecution.ts:308-335`
先给出一组很硬的正例。

这里：

1. `getMcpServerType()` 只有在 `serverConnection?.type === 'connected'` 时才返回 transport type
2. `getMcpServerBaseUrlFromToolName()` 只有在 `serverConnection?.type === 'connected'` 时才返回 base URL

这说明 repo 连“给工具执行补什么 transport / base-url 背景”这类辅助信息，
都不愿意沿用 stale reassurance。

换句话说，
execution plane 不是只在最核心的 tool body 里严格，
它在外围 metadata 上也坚持：

`current connected truth first`

`src/services/tools/toolExecution.ts:1599-1627`
更关键。

一旦真实 tool use 遇到 `McpAuthError`，
repo 会立刻把 client 从 `connected` 降成 `needs-auth`。

这里最值钱的不是它会报错，
而是它会主动撤销旧的 connected reassurance，
把 live contradiction 写回 current state。

这说明 use-time revalidation 不是：

`验证失败了就算了，旧 reassurance 还能挂着`

而是：

`验证失败了，旧 reassurance 必须立即降级`

## 8. 更深一层的技术先进性：Claude Code 把“可以放心一点”和“这一刻可以真的去用”继续拆开

这组源码给出的技术启示至少有五条：

1. positive wording 永远不该直接冒充 consumer-edge authorization
2. current-use truth 的 canonical 形态应尽量收束成少数 fresh proof primitive
3. helper metadata 也要 obey current truth，而不是只让核心操作严格
4. runtime contradiction 不是局部异常，而应成为正式 revocation signal
5. 对 consumer edge 的谨慎，比在 surface 上堆更多正向 reassurance 更值钱

从源码设计思路看，
Claude Code 的先进性不在于“它已经给了足够正向的话，所以用户自然能继续用”，
而在于：

`它知道真正的使用边界必须重新验真，而且验真失败时必须比正向话术更有主权。`

## 9. 苏格拉底式自我反思：如果我把 `305` 写得更强，我会在哪些地方越级

可以先问五个问题：

1. 如果 `Authentication successful` 已经足够，为什么 `ReadMcpResourceTool` 还要继续 `ensureConnectedClient()`？
2. 如果一次 `Successfully reconnected` 已经足够，为什么 `ListMcpResourcesTool` 还要在 `onclose` 后重新拿 fresh connection？
3. 如果 `connected` label 已经等于 current-use truth，为什么 `getMcpServerType()` 和 `getMcpServerBaseUrlFromToolName()` 对 disconnected path 直接返回 `undefined`？
4. 如果旧 reassurance 还能继续挂着，为什么 `McpAuthError` 会当场把 client 降成 `needs-auth`？
5. 如果 reassurance 已经等于 use-proof，为什么 repo 还需要一个统一的 reconnect-or-throw primitive？

这些反问逼着我们承认：

`reassurance`
只回答
`现在最多可以放心到什么程度`

而
`use-time revalidation`
真正回答的是：

`你这一刻是否已经重新验真到足以真的去用`

这也是本章最应该记住的压缩句：

`制度已经愿意让你更放心，不等于制度已经替你的这一次使用重新验了真。`
