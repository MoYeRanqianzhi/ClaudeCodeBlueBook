# 安全恢复跳转纪律：为什么半恢复状态必须绑定唯一默认修复路径，而不能让用户在多个命令之间盲选

## 1. 为什么在 `52` 之后还要继续写“恢复跳转纪律”

`52-安全恢复文案禁令` 已经回答了：

`半恢复状态下不能继续使用过满绿色词，必须改用受约束语言。`

但如果继续往下一层追问，  
还会出现一个更偏交互工程的问题：

`当系统已经诚实地承认“我只恢复到一半”之后，它接下来到底应该把用户送去哪里？`

因为一个成熟控制面不只要会说：

`我现在还缺哪一半`

还要会说：

`你现在该走哪一条默认修复路径。`

如果这件事做不好，  
系统就会出现另一种更隐蔽的安全缺陷：

1. 状态说得已经够诚实
2. 但用户不知道该去 `/status`、`/mcp`、`/plugins`、`/reload-plugins`、`/login` 还是 `/remote-control`
3. 于是用户开始盲选命令
4. 最终把半恢复状态变成误修复、过度修复或绕路修复

所以在 `52` 之后，  
统一安全控制台还必须再补一条制度：

`恢复跳转纪律。`

也就是：

`每种半恢复状态都必须绑定一个唯一默认修复路径，而不是把用户扔进多个命令入口之间自己猜。`

## 2. 最短结论

从源码看，Claude Code 其实已经在很多局部场景里偷偷做了这件事：

1. 插件风险 -> `/plugins`  
   `src/hooks/useManagePlugins.ts:62-67`
2. 插件变更待激活 -> `/reload-plugins`  
   `src/hooks/useManagePlugins.ts:294-300`
3. IDE 插件断开 / 安装失败 -> `/status`  
   `src/hooks/notifs/useIDEStatusIndicator.tsx:119-123,147-152`
4. MCP failed / needs-auth -> `/mcp`  
   `src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`
5. channels 认证问题 / 未登录 -> `/login`  
   `src/components/LogoV2/ChannelsNotice.tsx:73`、`src/components/PromptInput/Notifications.tsx:306-309`
6. Remote Control 启停 -> `/remote-control`  
   `src/components/RemoteCallout.tsx:36-42`、`src/bridge/bridgeMain.ts:2125-2128`

所以这一章的最短结论是：

`Claude Code 现有源码已经证明“默认修复路径”是控制面的一等语义，只是它还分散在不同通知和局部组件里；下一代统一安全控制台必须把这件事正式化成跳转纪律。`

我会把这条结论再压成一句话：

`半恢复状态如果没有唯一默认修复路径，诚实状态最终仍会在交互上输给歧义。`

## 3. 源码已经暴露出一套分散但真实存在的“默认修复语言”

## 3.1 插件路径已经有两条非常清楚的默认修复入口

`src/hooks/useManagePlugins.ts:62-67` 对 flagged plugins 直接给出：

`Check /plugins`

而 `src/hooks/useManagePlugins.ts:294-300` 对插件变化待激活则给出：

`Run /reload-plugins to activate.`

这两条其实已经体现出一种很成熟的区分：

1. 风险与治理问题 -> 去 `/plugins`
2. 激活与热刷新问题 -> 去 `/reload-plugins`

也就是说，  
源码已经知道：

`不同半恢复状态即使同属 plugin 域，也不该共用一条默认修复路径。`

## 3.2 IDE 相关问题当前更倾向于把用户送去 `/status`

`src/hooks/notifs/useIDEStatusIndicator.tsx:119-123` 会在 JetBrains plugin not connected 时提示：

`/status for info`

`src/hooks/notifs/useIDEStatusIndicator.tsx:147-152` 又会在 IDE extension install failed 时提示：

`see /status for info`

这说明 IDE 相关问题在当前设计里更像：

`先走诊断页`

而不是：

`直接走修复动作页`

这很重要，  
因为它说明恢复跳转纪律不总是“直接修”，  
有时正确默认路径是：

`先去总诊断页收敛事实。`

## 3.3 MCP 问题已经有非常明确的领域内入口 `/mcp`

`src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63` 对：

1. MCP failed
2. claude.ai connector unavailable
3. MCP needs auth
4. claude.ai connector needs auth

都统一在通知尾部放了：

`· /mcp`

这说明对 MCP 域来说，  
源码已经在事实上做了一条非常干净的纪律：

`只要问题属于 MCP server / connector 连接域，默认跳转就先收敛到 /mcp。`

## 3.4 认证瓶颈已经在多个地方被指向 `/login`

`src/components/LogoV2/ChannelsNotice.tsx:73` 明确写出：

`Channels require claude.ai authentication · run /login, then restart`

`src/components/PromptInput/Notifications.tsx:306-309` 也在未登录时给出：

`Not logged in · Run /login`

这说明认证相关的半恢复状态在源码里已经有一个非常强的一致性：

`auth bottleneck -> /login`

从控制面角度看，  
这其实已经是一条正式的默认修复路径规则。

## 3.5 Remote Control 也已经有自己的专属入口 `/remote-control`

`src/components/RemoteCallout.tsx:36-42` 直接把：

`Enable Remote Control for this session`

做成了用户可选动作，  
同时说明：

`You can always enable it later with /remote-control.`

`src/bridge/bridgeMain.ts:2125-2128` 也明确告诉用户：

Remote Control 可以通过 `/remote-control` 再次启用或断开。

这说明 bridge / remote-control 域在源码里同样已经有自己的专属入口，  
不需要用户自己猜。

## 3.6 这套默认路径是存在的，但仍然分散、局部且不完全一致

现在的问题不在于没有路径，  
而在于：

1. 有的状态跳去治理页
2. 有的状态跳去总诊断页
3. 有的状态跳去领域页
4. 有的状态跳去动作命令
5. 有的状态根本没明确给出唯一默认路径

也就是说，  
源码已经有了“修复路由碎片”，  
但还没有：

`统一恢复跳转纪律。`

## 4. 第一性原理：修复歧义本身就是一种安全风险

如果从第一性原理追问：

`为什么“默认修复路径”也要上升为安全问题？`

因为当系统处于半恢复状态时，  
用户最需要的不是更多状态词，  
而是：

`一条不会把问题越修越歪的最短路径。`

如果控制面不给这条路径，  
用户往往会出现四类风险行为：

1. 去错页面
2. 跑错命令
3. 把诊断页当修复页
4. 把治理页当状态页

这四类错误的共同点是：

`它们并不改变底层事实，却会延长错误状态存在时间。`

所以恢复跳转纪律真正保护的是这条公理：

`半恢复状态必须把用户送去最能缩小当前缺口的那个入口，而不是送去一个“看起来也许相关”的入口。`

## 5. 我给统一安全控制台的恢复跳转纪律

我会把这条纪律压成四条规则。

## 5.1 一种半恢复状态，只允许一个 dominant repair path

也就是说，  
每个状态都可以有：

1. 一个主路径
2. 零到一个次路径

但不能给用户并列三个看似都差不多的入口。

因为一旦并列：

`默认路径就不再默认。`

## 5.2 默认路径必须跟“缺口归属”绑定，而不是跟“表面症状”绑定

比如：

1. 认证缺口 -> `/login`
2. MCP 连接缺口 -> `/mcp`
3. plugin 风险治理 -> `/plugins`
4. plugin 激活刷新 -> `/reload-plugins`
5. bridge attach / reconnect / disconnect -> `/remote-control`
6. 跨域诊断不清 -> `/status`

也就是说，  
默认跳转应优先回答：

`当前哪条链真正断了`

而不是：

`哪条命令名字看起来比较接近`

## 5.3 诊断页和修复页必须显式分工

这在现有源码里已经部分存在：

1. `/status` 更像总诊断页
2. `/mcp`、`/plugins`、`/remote-control` 更像领域页
3. `/reload-plugins`、`/login` 更像直接动作命令

所以统一安全控制台应正式承认：

1. 有些状态的默认路径是“先诊断”
2. 有些状态的默认路径是“直接修”

而不能把所有入口混成一个“都去看看”。

## 5.4 默认路径必须和受约束语言一起出现

也就是说，  
正确文案不应只是：

`MCP server failed`

而应更像：

`MCP server failed · /mcp`

或：

`State restored; awaiting fresh sync · stay on status`

这说明恢复跳转纪律不是独立于文案系统的，  
而是：

`受约束语言的最后一段。`

## 6. 我给半恢复状态的最低跳转语法

我会把统一安全控制台的修复语法压成：

`当前缺口 + 唯一默认入口`

必要时可扩展为：

`当前缺口 + 唯一默认入口 + 次级条件`

例如：

1. `Plugin risk flagged · /plugins`
2. `Plugins changed; activation pending · /reload-plugins`
3. `MCP connector unavailable · /mcp`
4. `Auth missing · /login`
5. `Bridge attached but inbound control unavailable · /remote-control`
6. `IDE install state ambiguous · /status`

这类句式的价值在于：

1. 说清楚缺口
2. 不给用户分叉选择
3. 允许控制面继续保持诚实

## 7. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只是底层状态建模，而是它已经在多个局部模块中显露出“默认修复路径”这一控制面意识。`

这套设计对其他 Agent 平台最直接的启示有四条：

1. 修复入口应按缺口归属路由，而不是按 UI 模块命名路由
2. 总诊断页和领域修复页必须严格分工
3. 通知如果不给唯一入口，最终就会把用户送去错误命令
4. 默认路径本身应被看作控制面协议的一部分

## 8. 哲学本质

这一章更深层的哲学是：

`真正关心用户的系统，不只告诉用户哪里坏了，还替用户缩小修复选择空间。`

很多系统看起来“已经提示得很完整”，  
但用户仍然修不好，  
不是因为信息不够，  
而是因为：

`系统把解释责任给了用户，把路由责任也给了用户。`

Claude Code 现有源码里那些 `/plugins`、`/reload-plugins`、`/mcp`、`/login`、`/remote-control`、`/status` 提示，  
其实已经在说明一条更成熟的哲学：

`控制面不仅要治理状态，也要治理用户下一步最短动作。`

从第一性原理看，  
这保护的是下面这条公理：

`当系统知道哪一条修复路径最短时，就不应把路径选择成本重新外包给用户。`

## 9. 苏格拉底式反思：这一章最容易犯什么错

### 9.1 会不会把所有状态都强行绑定到一个命令，反而过度简化

会。  
所以关键不是“每个域只有一个命令”，  
而是：

`每个具体半恢复状态只有一个 dominant repair path。`

### 9.2 会不会把默认路径做成黑箱，用户看不到为什么跳那里

也会。  
所以默认路径必须总和当前缺口一起出现，  
不能只给一个神秘按钮。

### 9.3 这一章之后还缺什么

还缺一张真正可执行的修复路由表：

`状态 / 中间态 -> 默认跳转 -> 次级跳转 -> 禁止歧义`

也就是说，  
下一步最自然的延伸就是：

`appendix/37-安全恢复跳转纪律速查表`

## 10. 结语

`52` 回答的是：恢复语言必须受约束。  
这一章继续推进后的结论则是：

`受约束语言还必须带出唯一默认修复路径，避免把用户扔进多个命令之间盲选。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不仅会限制错误动作，也会把正确修复动作路由成最短路径。`
