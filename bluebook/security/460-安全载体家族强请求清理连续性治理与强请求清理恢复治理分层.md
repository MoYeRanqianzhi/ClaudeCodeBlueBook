# 安全载体家族强请求清理连续性治理与强请求清理恢复治理分层：为什么artifact-family cleanup stronger-request cleanup-continuity-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-recovery-governor signer

## 1. 为什么在 `459` 之后还必须继续写 `460`

`459-安全载体家族强请求清理就绪治理与强请求清理连续性治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道对象此刻 ready，也还需要 continuity governor 去决定这份可用性在时间里怎样继续成立。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然系统已经开始 retry、已经给出 auth URL、已经重新 connect 过一次，那它当然算恢复了”
的错觉：

`只要 stronger-request cleanup continuity governor 已经让旧线继续重试、等待重鉴权、重新发起 reconnect，或者已经拿到某个 auth callback，它就自动拥有了宣布对象“已经恢复”的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的恢复正对照，
会发现 repo 其实明确拆开了六件事：

1. `auth-failure demotion`
2. `fresh reconnect verdict`
3. `new credential proof collection`
4. `auth-only callback 与 reconnect completion 的拆层`
5. `control-plane success grammar`
6. `consumer-side real-tool return discipline`

最硬的证据至少有七组：

1. `services/mcp/client.ts:335-360,1105-1137,2287-2345` 会把 401 / `UnauthorizedError` 明确降成 `needs-auth`，写入 auth cache，并用 `createMcpAuthTool()` 代替真实工具
2. `services/mcp/client.ts:2137-2198` 的 `reconnectMcpServerImpl()` 会先 `clearKeychainCache()`、`clearServerCache()`，再只接受 fresh `connected` 结果
3. `services/mcp/auth.ts:575-616,847-1065,1568-1805` 的 `performMCPOAuthFlow()` 与 XAA silent / interactive 路径把“新凭据”写成正式证据门槛
4. `cli/print.ts:3133-3295` 的 `mcp_reconnect` / `mcp_toggle` 只在真正 `connected` 时返回 success
5. `cli/print.ts:3310-3505` 的 `mcp_authenticate` / `mcp_oauth_callback_url` 明确把 auth phase、callback phase 与 reconnect phase 拆开
6. `tools/McpAuthTool/McpAuthTool.ts:37-197` 只在 reconnect 真正闭环后才让 real tools 回来
7. `components/mcp/MCPReconnect.tsx:15-63` 只在 reconnect 真正返回 `connected` 时才显示 success message

这说明：

`artifact-family cleanup stronger-request cleanup-continuity-governor signer`

和

`artifact-family cleanup stronger-request cleanup-recovery-governor signer`

仍然不是一回事。

前者最多能说：

`这条旧的可用性时间线现在还要不要继续抢救。`

后者才配说：

`抢救之后，或者 continuity 已断裂之后，现在到底有没有足够新的凭据、新的连接结果与新的工具世界，正式宣布它已经回来。`

所以 `459` 之后必须继续补的一层就是：

`安全载体家族强请求清理连续性治理与强请求清理恢复治理分层`

也就是：

`stronger-request cleanup continuity governor 决定旧线要不要继续；stronger-request cleanup recovery governor 才决定什么新的证据足以重新宣布 recovered truth 成立。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-recovery-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-recovery-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 recovery manager，
而是在说：

1. repo 已经把 `needs-auth`、`pending`、`connected`、`failed` 与 consumer-side tool swap 拆成不同恢复阶段
2. repo 已经把 retry、auth、callback、reconnect、tool / world restoration 拆成不同判定层
3. repo 已经拒绝让“正在恢复”自动篡位成“恢复完成”

第二条：

这里说的 `recovery governance`

不是在声称：

`对象只要重新连上一次，就算彻底痊愈。`

当前更稳的说法只能是：

`recovery governance 首先回答的是哪组新的凭据、新的连接结果与新的工具世界，足以重新签出 usable truth。`

第三条：

这里也不是把：

`performMCPOAuthFlow()`

单独当成全部 recovery grammar。

真正关键的是：

1. auth-failure demotion
2. fresh reconnect verdict
3. auth-only callback 与 reconnect completion 的拆层
4. control-plane success grammar
5. consumer-side real-tool return discipline

第四条：

这里尤其不能过度声称：

`recovery governance`

已经等于：

`reintegration governance`

当前更稳的说法只能是：

`recovery governance 回答什么新证据足以宣布 recovered；reintegration governance 才回答 recovered truth 何时重新并入 current world 并被不同读者重新消费。`

第五条：

这里也不能过度声称：

`silent re-auth`

就天然比 interactive re-auth 更弱或更不正式。

当前更稳的说法只能是：

`无论 silent 还是 interactive，只有拿到新的 credential evidence 并闭合 reconnect / tool restoration，才配进入 recovery verdict。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-continuity-governor signer 仍不等于 stronger-request cleanup-recovery-governor signer`

证据：

1. `handleRemoteAuthFailure()` 与 auth cache 会把 401 落成 `needs-auth`，并改为暴露 `McpAuthTool`；这说明 continuity 中断后，系统不会假装 recovered truth 仍在
2. `reconnectMcpServerImpl()` 会先清 keychain cache 和 connection cache，再重新 `connectToServer()`；只有 `client.type === 'connected'` 才继续抓 tools / commands / resources，这说明 recovery 需要 fresh evidence，而不是旧 continuity 的延长
3. `mcp_reconnect` / `mcp_toggle` 只有在 reconnect result 真的是 `connected` 时才返回 success；这说明手工重试不等于恢复 verdict
4. `mcp_authenticate` 与 `mcp_oauth_callback_url` 把 `auth url ready`、`auth promise done` 与 `reconnect complete` 拆开；这说明 auth 完成不等于 recovery 完成
5. `McpAuthTool` 与 `MCPReconnect` 只在 reconnect 真正返回 `connected` 后才把 real tools 或 success message 交还给用户；这说明 consumer surface 也拒绝偷签 recovery truth
6. `needs-auth` 伪工具与 real tools 的 prefix-based replacement 机制说明 repo 不只是“告诉你要认证”，而是在制度上暂时收回真实工具世界，直到恢复成立

因此这一章的最短结论是：

`stronger-request cleanup continuity governor 最多能说旧的可用性线还要不要继续；stronger-request cleanup recovery governor 才能说新的凭据与新的连接结果是否已经足以重新建立 usable truth。`

再压成一句：

`还在恢复，不等于已经恢复。`

再压成更哲学一点的话：

`制度知道旧线还要不要继续抢救，不等于制度已经知道什么新证据足以宣布它真的回来了。`

## 4. 第一性原理：continuity governance 回答“旧线还要不要继续”，recovery governance 回答“什么新的证据足以重新成立”

从第一性原理看，
强请求清理连续性治理与强请求清理恢复治理处理的是两个不同主权问题。

`stronger-request cleanup continuity governor`
回答的是：

1. transport 断掉之后还要不要继续重连
2. `pending` 还要维持多久
3. 什么时候从 retry 转成 give-up
4. `needs-auth` 或 `failed` 之后还要不要继续等待旧线自己回来
5. 哪些 stale continuity attempt 必须被停掉

`stronger-request cleanup recovery governor`
回答的则是：

1. 什么新的 credential proof 算足够
2. 什么新的 callback / code / token exchange 算完成
3. 什么新的 `connectToServer()` 结果算恢复成立
4. 什么时候 real tools、commands、resources 才能重新回到 current world
5. 哪些 surface 配宣布 `recovered`，哪些只配签 `recovering`、`needs-auth` 或 `still failed`

如果把这两层压成一句：

`反正系统已经开始恢复`

系统就会制造六类危险幻觉：

1. `retry-means-recovered`
2. `auth-url-means-recovered`
3. `callback-means-recovered`
4. `fresh-attempt-means-recovered`
5. `auth-done-means-usable`
6. `pseudo-tool-means-real-service-is-back`

所以从第一性原理看：

`continuity governance` 管的是旧线是否继续；
`recovery governance` 管的是新证据是否足够重新成立。

## 5. Claude Code 多重安全技术的先进性：它把“恢复成立”绑定到 fresh proof，而不是绑定到 good intention

Claude Code 在这一层真正先进的地方，
不是“只要系统愿意继续重试，就算恢复已经在发生”，
而是：

`它把 recovered truth 绑定到 fresh credential proof、fresh connection verdict 和 fresh tool-world restoration 的闭环。`

这种先进性至少体现在六个方面：

1. `auth-failure is an explicit demotion`
   continuity 一旦不够，系统就把对象降到 `needs-auth`，而不是继续用旧 ready truth 混过去。
2. `fresh reconnect invalidates stale proof first`
   `clearKeychainCache()` 和 `clearServerCache()` 说明恢复不是“接着旧证据往下拖”，而是先切断旧闭环。
3. `auth proof collection is formalized`
   OAuth flow 不论 silent 还是 interactive，都必须产出新的认证证据。
4. `control-plane honesty is strong`
   `mcp_authenticate`、`mcp_oauth_callback_url`、`mcp_reconnect` 各签各的事实，不互相越权。
5. `consumer return is gated by recovered reality`
   只有真正 reconnect 成功，real tools 和 success message 才能回到用户面前。
6. `pseudo-service and real-service are structurally separated`
   `McpAuthTool` 的存在说明 repo 会把“可开始恢复”与“已恢复可用”拆成两个不同对象。

这些设计背后的源码思路也很清楚：

1. 不让 `trying to recover` 越级冒充 `recovered`
2. 不让 `auth started` 越级冒充 `auth completed`
3. 不让 `auth completed` 越级冒充 `service restored`
4. 不让 `fresh attempt` 越级冒充 `fresh proof`
5. 不让 consumer 从较弱控制事实里误读出较强可用性结论

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. continuity 需要被显式治理，但 recovery 还需要一套更强的 fresh-proof grammar。
2. 旧 cleanup truth 断裂后，必须允许系统明确降到 `needs-auth`、`recovering` 或等价状态，而不是让旧 ready truth 延迟死亡。
3. recovery 不能只靠重新尝试动作本身，必须依赖新的 credential / callback / reconnect 结果。
4. auth-start、auth-done、reconnect-success、tool-world-restored 应该分开签字，不应压成单一“恢复中”文案。
5. consumer-facing 的假工具与真工具应显式分层，防止“恢复入口”被误读成“恢复完成”。
6. recovery 的本体不是“系统还在努力”，而是“系统已经拿到足够新的证据重新担保它”。

## 6. 苏格拉底式自反诘问：我是不是又把“系统已经开始恢复”误认成了“对象已经恢复成立”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 continuity grammar 已经足够强，为什么还要再拆 recovery？
   因为旧线还能不能继续，不等于什么新证据足以重新宣布 usable truth 成立。
2. 如果系统已经在 retry，是不是就说明它其实已经回来了？
   不是。retry 只说明系统还没放弃，不说明 fresh proof 已经闭合。
3. 如果 auth URL 已经拿到，是不是就说明恢复很快就能算成立？
   不是。auth start 只是恢复入口，不是恢复 verdict。
4. 如果 callback 已经到达，是不是就说明对象已经恢复？
   不是。callback completion 仍不等于 reconnect / tool-world restoration。
5. 如果 pseudo-tool 已经告诉用户去认证，是不是就说明真实工具已经可以被继续信任？
   不是。pseudo-tool 的存在恰恰说明真实工具仍被制度性收回。
6. 如果 cleanup 线现在还没显式 recovery 代码，是不是可以先把 continuity 写得更强来替代？
   不能。越是缺 recovery grammar，越容易把“还在抢救”偷写成“已经回来”。

这一串反问最终逼出一句更稳的判断：

`recovery 的关键，不在系统愿不愿意继续尝试，而在系统有没有拿到足够新的证据重新替对象签字。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 continuity governance，就已经足够成熟。`

而是：

`Claude Code 在 auth-failure demotion、fresh reconnect verdict、OAuth proof collection、control-plane honesty 与 consumer-side real-tool return 上已经明确展示了 recovery governance 的存在；因此 artifact-family cleanup stronger-request cleanup-continuity-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-recovery-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“旧线还要不要继续”，而是“什么新证据才配宣布它已经回来”。`
