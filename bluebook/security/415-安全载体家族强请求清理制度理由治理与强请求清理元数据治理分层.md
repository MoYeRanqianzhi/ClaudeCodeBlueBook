# 安全载体家族强请求清理制度理由治理与强请求清理元数据治理分层：为什么artifact-family cleanup stronger-request cleanup-rationale-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-metadata-governor signer

## 1. 为什么在 `414` 之后还必须继续写 `415`

`414-安全载体家族强请求清理家族宪法治理与强请求清理制度理由治理分层`
已经回答了：

`不同 stronger-request old cleanup carrier family 为什么值得活在不同 cleanup law 里。`

但如果继续往下追问，
还会碰到一层更工程化、也更危险的错觉：

`只要系统已经知道这些 family 的 primary risk object、operational duty 与 host visibility，这些理由就已经自动变成系统自己的元数据。`

Claude Code 当前源码并不能支持这种更强说法。

继续看会发现，
repo 当前最值得重视的事实不是：

`它没有 cleanup metadata。`

而是：

`它已经长出了多套贴着不同 carrier family 运行的 metadata microgrammar，但这些 microgrammar 还没有被提升成统一、可跨平面共读的 canonical descriptor。`

最硬的证据至少有八组：

1. `src/utils/cleanup.ts:575-598` 明确按 family 手写 dispatch cleanup helpers，说明系统知道 family 有别，但这种知道主要仍以 hand-written call order 的形式存在
2. `src/utils/settings/types.ts:325-332,824-830` 对外公开的是 `cleanupPeriodDays` 与 `plansDirectory` 这类局部 knob，而不是完整 family descriptor
3. `src/utils/task/diskOutput.ts:33-55` 不只写出 non-clobber rationale，还把 `sessionId` 固定进 `getProjectTempDir()/sessionId/tasks`，并强调 `FIRST CALL` capture 以抵抗 `/clear` drift，这已经是一种 very local metadata grammar
4. `src/utils/toolResultStorage.ts:26-31,94-116` 把 tool-results family 编成 `tool-results` subdir、`<persisted-output>` tag 与 `id.json|txt` 文件名语法，说明 inspection truth 已部分物化成 carrier grammar
5. `src/utils/sessionStorage.ts:202-225,247-258` 又把 transcripts / subagent transcripts 编成 `${sessionId}.jsonl`、`sessionId/subagents/.../agent-${agentId}.jsonl`，并用 `sessionProjectDir` 原子耦合来抑制 path drift
6. `src/utils/plans.ts:32-48,79-110,119-129,164-229` 与 `src/utils/permissions/filesystem.ts:244-255,1487-1495,1644-1652` 共同把 plans family 编成 `plansDirectory`、word slug、`-agent-` 后缀与 current-session permission prefix，但 `src/utils/cleanup.ts:300-303` 仍只清默认 `~/.claude/plans`，说明 cross-plane metadata carrier 仍不统一
7. `src/utils/fileHistory.ts:951-957`、`src/utils/sessionEnvironment.ts:15-31,72-100,146-150` 与 `src/utils/debug.ts:44-56,91-101,230-249` 继续展示 `file-history/<sessionId>`、`session-env/<sessionId>/setup-hook-0.sh`、`debug/<sessionId>.txt + latest` 这些 family-local naming grammar
8. `src/utils/diagLogs.ts:5-12,14-20,27-60` 最后再给出最强的 host-external metadata：`DiagnosticLogEntry` schema、`CLAUDE_CODE_DIAGNOSTICS_FILE` 与 no-PII contract

这说明：

`artifact-family cleanup stronger-request cleanup-rationale-governor signer`

和

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

仍然不是一回事。

前者最多能说：

`为什么这些 stronger-request old cleanup carrier family 值得活在不同 law 里。`

后者才配说：

`这些 why 现在究竟住在哪些名字、路径、slug、tag、symlink、permission projection 与 env contract 里，哪些平面在消费它，哪些传播仍靠人工纪律维持。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-metadata-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-metadata-governor signer`
仍是研究命名。
它不是在声称 repo 已有一张现成的 registry table，
而是在说：

1. family truth 已经明显存在
2. 这些 truth 已经以 comment、helper、slug、path grammar、permission projection、argv/env contract 与 dispatcher literal 的形式分布存在
3. 但这些 truth 还没有被提升成同一张 canonical descriptor

第二条：

这里说的 `metadata`
不是在声称：

`当前系统完全没有 metadata。`

更稳的说法只能是：

`当前系统已经有相当多 cleanup metadata，但它们更像 family-local metadata microgrammar，而不是统一 stronger-request cleanup metadata plane。`

第三条：

这里也不是把：

`global registry`

直接当成唯一正确答案。

更稳的说法只能是：

`不是所有 local grammar 都必须上升成单张总表，但凡一个 family truth 要跨 settings、storage、permissions、recovery 与 cleanup executor 传播，就至少需要某种 canonical carrier 来抑制 drift。`

第四条：

这里尤其不能过度声称：

`comment 很清楚，所以 metadata 已完成。`

当前更稳的说法只能是：

`comment 只是在某个 module 内显式写出 truth；module-local explicitness 不自动等于 multi-plane shared metadata。`

第五条：

这里也不能过度声称：

`plans family 的 propagation gap 就等于全部 metadata 问题。`

更稳的说法只能是：

1. plans family 是最硬的 stress test
2. task outputs、tool-results、transcripts、session-env、debug 与 diagnostics 同样都暴露出不同形态的 metadata microgrammar
3. 真正的问题是 plural local grammars 还没有被制度化成 plural-yet-shareable descriptor plane

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-rationale-governor signer 仍不等于 stronger-request cleanup-metadata-governor signer`

证据：

1. 系统已经知道不同 family 为什么存在不同 law，但这种知道常常还停在 local helper、comment 与 path grammar
2. 有些 family 已经长出很强的 metadata microgrammar，例如 plan slug、debug `latest` 指针、hook-file priority、diagnostic log schema
3. 这些 microgrammar 仍大多是 module-local truth，不是全链 canonical descriptor
4. permission plane 经常只能读到 partial projection，而不是 source-of-truth
5. cleanup dispatcher 知道 plural families，却仍主要按字面函数顺序执行，而不是从 shared metadata plane 取 policy
6. plans family 继续暴露最强的 cross-plane propagation gap
7. host-external diagnostics world 更说明 repo 并不是只有 repo-local metadata 问题，而是 boundary-local metadata contract 问题

因此这一章的最短结论是：

`stronger-request cleanup-rationale-governor signer 最多只能说“这些 law 为什么合理”；stronger-request cleanup-metadata-governor signer 才能说“这些 why 有没有被系统自己登记成可传播、可投影、可防漂移的制度自我记忆”。`

再压成一句：

`理由已知，不等于理由已被系统登记成制度自我记忆。`

再压成更硬的一句：

`制度会解释自己，不等于制度会记住自己。`

## 4. 第一性原理：rationale 回答“为什么这样设计”，metadata 回答“系统如何把这个为什么记给自己”

从第一性原理看，
强请求清理制度理由治理与强请求清理元数据治理处理的是两种不同主权。

`stronger-request cleanup-rationale governor`

回答的是：

1. 这个 family 的 primary risk object 是什么
2. 它主要服务 inspection、continuity、planning recovery、restore、environment replay、operational debugging，还是 host monitoring
3. 为什么它应该活在当前这条 cleanup law 里
4. 当前 law 是否还忠于这个 why
5. 哪里已经出现 rationale drift

`stronger-request cleanup-metadata governor`

回答的则是：

1. 这些 why 现在住在哪些名字、路径、slug、tag、symlink、permission rule、argv/env contract 与 schema 字段里
2. 哪些平面真正消费同一份 truth
3. 哪些平面消费的只是 partial projection
4. 这些 truth 变化后会不会自动传播
5. 哪里因为没有 canonical carrier 而开始 drift

换句话说：

1. rationale 是 `why this law deserves to exist`
2. metadata 是 `where that why is carried, named, read and remembered`

如果把两层压成一句：

`系统已经知道为什么这样设计，所以元数据自然也已经存在`

系统就会制造五类危险幻觉：

1. `known-reason-means-shared-metadata`
2. `local-helper-means-canonical-descriptor`
3. `permission-projection-means-source-of-truth`
4. `settings-knob-means-full-family-grammar`
5. `dispatch-order-means-policy-registry`

所以从第一性原理看：

`rationale` 负责制度自我正当化；
`metadata` 负责制度自我记忆。

其哲学本质可以压成一句：

`知道为什么配有这条法，不等于知道系统靠什么把这条法记给自己。`

## 5. Claude Code 多重安全技术的先进性：它已经发明了多套贴身的 metadata microgrammar

从技术角度看，
Claude Code 在这里最先进的地方，
并不是它已经拥有一张完美的 cleanup family registry，
而是它已经发明出多套贴着不同安全边界运行的 metadata microgrammar。

至少有七种值得单独点名：

1. `task-output microgrammar`
   `diskOutput.ts` 把 `sessionId` 固定进 `project-temp/sessionId/tasks`，并明确用 first-call capture 抵抗 `/clear` path drift
2. `tool-result microgrammar`
   `toolResultStorage.ts` 把 inspection carrier 编成 `tool-results` subdir、`persisted-output` tag 与 `id.ext` 命名法
3. `transcript microgrammar`
   `sessionStorage.ts` 把 main transcript、other-session transcript 与 subagent transcript 分成不同 path grammar，并用 `sessionProjectDir` 原子耦合抵抗 resumed-path drift
4. `plan microgrammar`
   `plans.ts` 把 plan family 编成 `plansDirectory`、word slug、`-agent-` suffix、resume recovery 与 fork copy policy
5. `session-env microgrammar`
   `sessionEnvironment.ts` 把 replay family 编成 `setup-hook-0.sh` 等 hook grammar，并用 priority order 固定 load sequence
6. `debug microgrammar`
   `debug.ts` 把 argv/env 开关、default file path 与 `latest` symlink 编成 operational current-pointer grammar
7. `diagnostics microgrammar`
   `diagLogs.ts` 把 host-visible carrier 编成 no-PII JSON entry schema 与 env-selected external logfile contract

这其实揭示了 Claude Code 多重安全技术的一个先进点：

`它没有把所有 old cleanup carrier 强行压进同一套元数据语言，而是让不同边界在最接近运行事实的地方生长出各自的贴身语法。`

这带来三个好处：

1. local truth 离 runtime 实现很近，不容易被抽象层误写
2. 每个 family 可以把自己的特殊风险直接折进 naming / path / contract 层
3. 很多安全属性可以在 helper 级别被立即消费，而不必先穿过一张全局大表

但这套先进性也带来另一面：

`局部微语法越多，canonical carrier 越缺，cross-plane drift 就越容易变成治理问题。`

plans family 就是最硬的例子：

1. settings 知道 `plansDirectory`
2. path helper 知道 resolved plans root
3. permissions 知道 current-session plan prefix
4. recovery 知道 slug 与 missing-file restore
5. cleanup executor 却仍主要只知道默认 `~/.claude/plans`

所以技术判断必须更精确：

`Claude Code 在 metadata 层不是“太弱”，而是“已经有 plural local grammar，但还没有 equally-strong canonical carrier”。`

## 6. 为什么 local metadata microgrammar 仍不等于 metadata governor

把上面的事实进一步压缩，
local metadata microgrammar 至少还缺四层：

1. `canonicality`
   当前谁是 source-of-truth，常常仍要靠研究者跨文件判断
2. `propagation guarantee`
   某一平面更新后，其他平面是否自动跟上，没有统一承诺
3. `projection honesty`
   permission、settings、cleanup 与 host contract 经常各自只读到局部 truth
4. `drift detectability`
   如果 truth 失配，系统常常没有显式 metadata-plane receipt 告诉你哪里失配

因此：

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`
真正要签的不是
`某个 helper 里有没有一个好名字`
，
而是：

`这些名字、路径、合同与局部语法能否共同构成系统自己的长期自描述。`

## 7. 苏格拉底式回问：如果我还要把这一层做得更好，我该先怀疑什么

第一问：

`如果制度理由已经清楚，为什么机器还需要 metadata？`

因为机器不消费哲学，
机器消费的是名字、路径、字段、tag、contract 与 projection。
没有这些载体，
理由就只能被研究者读懂，不能被系统稳定复读。

第二问：

`既然 local microgrammar 已经很多，是否只要再造一张总表就好了？`

不一定。
总表可以统一，
也可能把 local truth 抽空。
真正更好的问题不是“要不要总表”，
而是“哪些 truth 必须跨平面共享同一 canonical carrier”。
plans family 已经明确回答：
凡是横跨 settings、storage、permissions、recovery 与 cleanup executor 的 truth，都不能只靠实现纪律维持。

第三问：

`如果我把 metadata 理解成制度自我记忆，这一层最容易被忽略的危险是什么？`

最容易被忽略的不是没有词，
而是：

`制度在很多地方会说自己，却没有任何地方对这些说法负责把它们保持一致。`

第四问：

`这一轮相比旧版本分析，真正往前推进了什么？`

真正往前推进的不是再说一遍
`truth is scattered`
，
而是更精确地把当前状态命名为：

`distributed metadata microgrammar without an equally-strong canonical carrier`

这比“truth 分散”更贴近源码，
也更能解释为什么 Claude Code 同时显得先进又显得容易漂移。

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 的制度理由只要被解释清楚，metadata 问题就已经结束。`

而是：

`Claude Code 已经让不同 carrier family 各自长出贴身 metadata microgrammar；正因如此，metadata governance 才必须被单独提升出来，专门回答这些局部语法何时只是局部 truth，何时才算系统自己的长期自描述。`

因此：

`artifact-family cleanup stronger-request cleanup-rationale-governor signer`
仍不能越级冒充
`artifact-family cleanup stronger-request cleanup-metadata-governor signer`。

最后把这一章压成一句：

`制度会解释自己，不等于制度会记住自己。`
