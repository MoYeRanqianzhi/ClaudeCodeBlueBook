# 安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层：为什么artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer

## 1. 为什么在 `328` 之后还必须继续写 `329`

`328-安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`
已经回答了：

`stronger-request cleanup 线即便知道旧对象可以回来、也重新拿回了一部分 current seat / enabled truth，仍然还需要 re-entitlement governor 去决定它究竟恢复哪些旧资格。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“资格回来就说明配置也一起回来了”
的错觉：

`只要 stronger-request cleanup re-entitlement governor 已经让旧对象重新拿回 current seat，它就自动拥有了决定旧 options、旧 secrets、旧 payload、旧 channel config 是否仍可被当前系统继续承认的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 线已经成熟写出的 reconfiguration 正对照，
会发现它其实明确拆开了五件事：

1. `seat truth`
2. `config source truth`
3. `config closure truth`
4. `save truth`
5. `apply truth`

最硬的证据至少有七组：

1. `src/utils/plugins/pluginOptionsStorage.ts:90-193`
   的 `savePluginOptions()` 说明 top-level option truth 不跟着 seat 自动继承，
   而是要按 sensitive / non-sensitive split-save、双向 scrub 与 partial reconfigure grammar 被重新写定
2. `src/utils/plugins/pluginOptionsStorage.ts:282-309`
   的 `getUnconfiguredOptions()` 说明对象已经存在，
   仍可能被单独判定为 config 未闭合
3. `src/utils/plugins/mcpbHandler.ts:141-339`
   的 `loadMcpServerUserConfig()` / `saveMcpServerUserConfig()` 说明 per-server / per-channel payload truth 同样有自己的 source-of-truth grammar
4. `src/utils/plugins/mcpbHandler.ts:742-757,894-902`
   明确把 forced reconfigure 或 validation failed 继续写成 `needs-config`
5. `src/utils/plugins/mcpPluginIntegration.ts:290-317`
   的 `getUnconfiguredChannels()` 继续证明 channel config completeness 是单独 decision surface
6. `src/commands/plugin/PluginOptionsFlow.tsx:53-133`
   把 `configured / skipped / error` 写成正式 config-closure grammar
7. `src/commands/plugin/ManagePlugins.tsx:1645-1668,1700-1707`
   明确把 `Enabled and configured`、`Enabled` 与 `Configuration saved. Run /reload-plugins for changes to take effect.` 拆成三种不同事实

这说明：

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象重新拿回了 current authority seat。`

后者才配说：

`它拿回 seat 之后，当前究竟按哪组 options / secrets / payload / channel config 重新工作。`

所以 `328` 之后必须继续补的一层就是：

`安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层`

也就是：

`stronger-request cleanup re-entitlement governor 负责决定对象回来后还配不配继续占 seat；stronger-request cleanup reconfiguration governor 才负责决定它占住 seat 之后按哪组 current config truth 重新运转。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reconfiguration manager，
而是在说：

1. repo 已经在 plugin 线成熟写出 reconfiguration grammar
2. repo 已经把 seat truth 与 config truth 拆开
3. stronger-request cleanup 线如果真要把 post-return world 做完整，也迟早要回答“回来后它究竟按哪组 current config truth 继续工作”

第二条：

这里说的 `reconfiguration governance`
不是在声称：

`所有旧配置都必须原样回来。`

当前更稳的说法只能是：

`reconfiguration governance 首先回答的是哪些 current config truth 仍被承认、哪些必须重填、哪些必须 scrub、哪些只能作为 stale residue 被丢弃。`

第三条：

这里也不是把：

`enabled`

单独当成全部 configuration truth。

真正关键的是：

1. 哪些输入属于 current source-of-truth
2. 哪些字段仍是 missing / invalid
3. 哪些值只是 saved，哪些已经 active
4. 哪些 secret reuse 仍被当前 schema 承认

第四条：

这里尤其不能过度声称：

`savePluginOptions()`

就等于完整 reconfiguration governance。

更稳的说法只能是：

`top-level options 只是 config truth 的一层；per-server payload、channel closure、save/apply choreography 仍要被单独治理。`

第五条：

这里也不能过度声称：

`配置已保存`

就等于：

`当前运行时已经按这组 truth 工作。`

更稳的说法只能是：

`saved truth` 和 `applied truth` 在 Claude Code 里被显式拆开；`/reload-plugins` 就是这条分界被公开说出的证据。`

第六条：

这里还不能过度声称：

`reconfiguration` 只是恢复体验里补填表单的交互步骤。

更稳的说法只能是：

`它本质上是在治理 returned object 的 current config truth grammar；表单只是输入界面，真正被治理的是 source、closure、save 与 apply 之间的制度分层。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-re-entitlement-governor signer 仍不等于 stronger-request cleanup-reconfiguration-governor signer`

证据：

1. `savePluginOptions()` 把 top-level option truth 写成 split-save + secure-first + cross-store scrub grammar
2. `getUnconfiguredOptions()` 把 schema-failed field 继续收束回 unconfigured surface
3. `loadMcpServerUserConfig()` / `saveMcpServerUserConfig()` 把 per-server / per-channel payload truth 提升成独立 source-of-truth plane
4. `needs-config` 明确拒绝把 visible object 偷写成 ready object
5. `getUnconfiguredChannels()` 证明 channel completeness 是独立 decision surface
6. `PluginOptionsFlow` 把 `configured / skipped / error` 写成正式 closure outcome
7. `ManagePlugins` 把 `Enabled`、`Enabled and configured` 与 `Configuration saved...take effect` 写成三种不同句法

因此这一章的最短结论是：

`stronger-request cleanup re-entitlement governor 最多能说对象回来后仍配占哪张 seat；stronger-request cleanup reconfiguration governor 才能说它占住 seat 之后当前该按哪组 config truth 工作。`

再压成一句：

`资格回来，不等于配置也回来。`

再压成更哲学一点的话：

`制度知道谁还能坐回座位，不等于制度已经知道这个座位现在该按哪组输入重新运转。`

## 4. 第一性原理：re-entitlement governance 回答“它还配不配占这个 seat”，reconfiguration governance 回答“这个 seat 凭哪组 current truth 运转”

从第一性原理看，
强请求清理再赋权治理与强请求清理重配置治理处理的是两个不同主权问题。

`stronger-request cleanup re-entitlement governor`
回答的是：

1. 对象回来后是否重新获得 enabled truth
2. 是否重新获得某个 scope / precedence 位置
3. 是否仍被 policy 允许
4. 是否保留 old identity，还是只拿到 new seat
5. 是否重新进入可声明 authority 的世界

`stronger-request cleanup reconfiguration governor`
回答的则是：

1. 哪组 options / secrets 仍被视作 current source truth
2. 哪些字段当前仍缺失或失效
3. 哪些 payload / channel config 必须重填
4. 哪些值只是 saved、哪些已经 active
5. returned object 何时才配说自己 work-ready

如果把这两层压成一句
`反正它已经重新 enable 了`，
系统就会制造五类危险幻觉：

1. `enabled-means-configured`
2. `return-means-old-secret-still-valid`
3. `seat-restored-means-schema-closed`
4. `saved-means-applied`
5. `authority-seat-means-operable`

所以这两层的哲学本质也不同：

`re-entitlement`

治理的是：

`它还配不配占这个 seat。`

`reconfiguration`

治理的是：

`它占住这个 seat 之后，当前究竟凭哪组可被承认的输入真相运转。`

Claude Code 在这里真正先进的地方，
不是它把 comeback 之后的配置过程包进一个模糊“已恢复”里，
而是它公开承认：

`资格归来`

和

`当前输入真相被重新确立`

不是同一件事。

## 5. 从技术角度看，Claude Code 的先进性在于把 config source、config closure、save effect 与 apply effect 全部拆开治理

### 5.1 `savePluginOptions()` 先证明：top-level option truth 不是 authority seat 的尾气，而是单独的 source-of-truth plane

`pluginOptionsStorage.ts:90-193`
最值钱的地方不是“能把值存进去”，
而是它明确拒绝以下偷换：

`seat 回来了 -> option truth 自然也一起回来`

这段代码至少同时治理四个问题：

1. `sensitive` key 必须进 `secureStorage`
2. `non-sensitive` key 进入 `settings.pluginConfigs[pluginId].options`
3. 本次 save 已迁到另一侧的 key 必须把旧侧 scrub 掉
4. partial reconfigure 不得因局部 save 误删未改字段

这里最先进的设计不是“机密单独存”，
而是：

`双侧 current truth 如何保持不互相打架。`

尤其
`secureStorage FIRST`
这一条非常关键。
它不是一般的实现顺序，
而是一种安全哲学声明：

`宁可暂时保留旧 fallback，也不要因为新写失败把 current config truth 变成双侧都不可信。`

### 5.2 `getUnconfiguredOptions()` 再证明：config closure 是单独 decision surface

`getUnconfiguredOptions():282-309`
的价值经常被低估。

它不只是 UI helper。
它本质上在做一件治理性的事情：

`判定 current seat 是否已经具备足够 config truth。`

这段代码先读当前保存值，
再跑 `validateUserConfig()`，
最后只把失败字段重新回收进 schema slice。

这说明 repo 清楚承认：

1. 对象已经存在
2. 配置仍可能不闭合
3. 这不是异常，而是正式的 current state

也就是说，在 Claude Code 里：

`configured`

不是 seat existence 的自然副产物，
而是一种需要被单独验证的 truth class。

### 5.3 `saveMcpServerUserConfig()` 把同一治理逻辑推进到 payload / channel 面

`mcpbHandler.ts:141-339`
让这条线更硬。

`loadMcpServerUserConfig()` 先把 settings 与 secure storage 合并成当前读取视图，
`saveMcpServerUserConfig()` 再明确规定：

1. sensitive / non-sensitive split
2. secure-first write order
3. schema flip 时的反向 scrub
4. partial reconfigure 的最小影响面

这说明 repo 并不把：

`server / channel payload`

看成 top-level option 的附属细节。
它承认：

`不同 surface 的 config truth 也需要独立治理。`

更强的是
`mcpbHandler.ts:742-757,894-902`
在两条路径上都明确返回：

`status: 'needs-config'`

而且同时带上：

1. `configSchema`
2. `existingConfig`
3. `validationErrors`

这等于直接证明：

`object visible`

和

`payload ready`

不是同一件事。

### 5.4 `PluginOptionsFlow` 与 `ManagePlugins` 再证明：`configured`、`saved`、`applied` 是三层不同事实

`getUnconfiguredChannels():290-317`
很硬。
它对每个 channel：

1. 读取保存值
2. 跑 `validateUserConfig()`
3. 只要失败，就把该 channel 放回 `unconfigured` 列表

这使
`PluginOptionsFlow.tsx:53-133`
可以在 mount 时直接构造 step list：

1. top-level unconfigured options
2. per-channel unconfigured config

如果没有 step，
立刻 `onDone('skipped')`；
全部 step 完成后，
才 `onDone('configured')`。

这说明 flow 层对配置真相的理解，
不是：

`要么 enabled，要么 disabled`

而是至少三段：

1. object has seat
2. config truth unresolved
3. config truth closed

`ManagePlugins.tsx:1645-1668,1700-1707`
甚至把这套分层直接公开写成结果句法：

1. `Enabled and configured`
2. `Enabled`
3. `Configuration saved. Run /reload-plugins for changes to take effect.`

一旦这三句话同时存在，
整个 repo 就在公开承认：

1. `enabled != configured`
2. `configured != saved-and-applied`
3. `saved != take effect`

## 6. 对 stronger-request cleanup 而言，当前真正缺的不是“回来后还配不配占 seat”，而是“回来后究竟按哪组 current config truth 工作”

当前 stronger-request cleanup 已经开始显露 re-entitlement-governance，
但仍缺少 reconfiguration 治理回答：

1. 旧 path / old promise / old receipt comeback 之后，哪组 current config truth 仍被承认
2. 哪些 old inputs 只说明历史存在，不说明 current validity
3. 哪些字段即便还在，也必须被 force reconfigure
4. 哪些 truth 只是 saved，哪些已经真正 take effect
5. 哪些 returned object 虽然重新 allowed，但仍不配宣布自己 ready for work

这些都不是：

`它还配不配拿回 authority seat`

能回答的。

它们已经进入：

`reconfiguration governance`

层。

也就是说，
stronger-request cleanup 今天真正缺的，
不是再补一个“资格有没有回来”的判断，
而是：

`谁来决定 returned object 现在应按哪组 current config truth 继续工作。`

## 7. 苏格拉底诘问：如果要把这一层做得更好，还缺什么

### 诘问一

`如果对象已经重新 enable，为什么不能直接说它现在能工作了？`

因为 enable 只证明：

`它被允许占住某张 seat。`

并不自动证明：

`它已经拿到当前可运行的 options / secrets / payload truth。`

### 诘问二

`如果旧 secret 还留在 secureStorage，为什么不直接继续用？`

因为 old secret presence
不自动等于
current truth validity。

schema 可能变了，
payload shape 可能变了，
channel 可能变了，
系统也可能显式要求 `needs-config`。

### 诘问三

`如果配置已经保存，为什么还不能直接说 returned object 已经恢复？`

因为 saved truth 仍不等于 applied truth。
当前运行世界是否真的接管这组 truth，
还要看后续的 active-layer choreography。

### 诘问四

`那怎样才算更好的 cleanup reconfiguration governance？`

至少要继续把五件事制度化：

1. returned carrier 的 current config ledger
2. missing-config / invalid-config surfacing
3. secret reuse 与 stale residue 的明确边界
4. save truth 与 apply truth 的分离回执
5. ready / not-ready 的正式状态语法

### 诘问五

`如果没有这些层，最坏会怎样？`

系统就会把：

`对象重新获得资格`

误写成：

`对象已经按当前真相正确工作`

而这恰恰是 stronger-request cleanup 最不能偷写的一句话。

## 8. 技术启示与哲学本质

这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 reconfiguration。`

而是更具体地说：

1. comeback object 仍可能 unconfigured
2. old secret 仍可能只是 residue，不是 current trust
3. current config truth 必须有 source-of-truth grammar
4. config closure 必须有 formal state，而不是隐含推断
5. save truth 与 apply truth 必须继续分层

Claude Code 在这里真正先进的地方，
不是它把配置过程藏到“恢复成功”里，
而是它已经在多个 subsystem 里明确承认：

`恢复资格`

和

`恢复当前可运行真相`

不是同一件事。

这是一种很成熟的安全哲学：

`权利可以先回来，但运行真相必须重新被确立。`

## 9. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

因为前者只能决定：

`returned object 还配不配继续占 current seat。`

而后者才有资格决定：

`它占住 seat 之后，当前究竟按哪组可被承认的 config truth 继续工作。`

再压成最后一句：

`制度知道谁还能回来继续坐着，不等于制度已经知道他回来以后该按哪份现行说明书运转。`
