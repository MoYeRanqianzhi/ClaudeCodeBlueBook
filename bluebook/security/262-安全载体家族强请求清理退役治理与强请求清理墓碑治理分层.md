# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `261` 之后还必须继续写 `262`

`261-安全载体家族强请求清理迁移治理与强请求清理退役治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样迁完，也还需要 sunset governor 去决定兼容与共存究竟在哪一刻正式结束。`

但如果继续往下追问，
还会撞上一层更隐蔽、也更容易被偷写成“宣布退役就够了”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界不再算 current truth，它就自动拥有决定退役后还该留下什么最小残留事实的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 自己已经落地的安全机制，
会发现它在多个子系统里都把：

`old world 何时结束`

和

`结束之后还留下什么最小残留标记，继续约束未来解释`

拆成了两层不同治理。

最硬的证据至少有六组：

1. `src/query.ts:712-719`
   遇到 orphaned assistant messages 时不会只静默丢弃旧对象，
   而会显式 `yield { type: 'tombstone' }`
2. `src/utils/messages.ts:2954-2957`
   与
   `src/QueryEngine.ts:758-760`
   证明 `tombstone` 不是普通内容，而是后续消费者必须识别的 control signal
3. `src/screens/REPL.tsx:2646-2648`
   与
   `src/utils/sessionStorage.ts:1470-1473,925-945`
   又证明 tombstone 会继续驱动 UI 与 transcript 删除链
4. `src/utils/plugins/cacheUtils.ts:56-72,145-171`
   与
   `src/services/plugins/pluginOperations.ts:523-547`
   证明 `.orphaned_at` 不是 sunset 时点本身，而是对象退役后留下的最小 marker
5. `src/utils/plugins/orphanedPluginFilter.ts:1-15,35-38`
   与
   `src/main.tsx:2544-2568`
   进一步证明 marker 还会被翻译成 grep / glob exclusion grammar，继续约束未来读取世界
6. `src/utils/config.ts:427-438`
   与
   `src/hooks/notifs/useModelMigrationNotifications.tsx:5-50`
   说明对象世界已经变化之后，系统有时仍要保留最小历史注脚，向当前 runtime 解释“为什么现在变了”

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request cleanup 世界从什么时候起不再算当前正式世界。`

后者才配说：

`旧世界结束之后，还要留下什么最小残留事实，让后来的 runtime、搜索面、恢复链与审计者不会把“已经退役”重新读成“从未发生”或“仍可继续援引”。`

所以 `261` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责宣布结束；stronger-request cleanup tombstone governor 才负责治理结束之后留下的最小残留事实。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开 tombstone manager，
而是在说：

1. repo 在 message / plugin / model 线已经展示出真实的 tombstone-like grammar
2. 这些 grammar 不等于 sunset deadline 本身
3. 它们承担的是 post-retirement minimal residue governance

第二条：

这里说的 `tombstone governance`
不是在声称：

`退役之后一定要保留完整旧对象。`

更稳的说法只能是：

`墓碑治理回答的是退役后保留多少最小事实，既足以约束未来解释，又不把旧对象重新抬回 current world。`

第三条：

这里也不是把：

`migration timestamp`

一概硬写成字面 tombstone。

当前更稳的说法只能是：

`有些残留物虽然源码里不直接叫 tombstone，但功能上承担了 post-transition trace artifact 的角色，因此属于 tombstone-like marker。`

第四条：

这里尤其不能过度声称：

`只要对象退役，就一定先有统一形态的 tombstone artifact。`

当前更稳的说法只能是：

`repo 展示了多种 marker grammar：message tombstone、plugin marker、migration timestamp；它们形态不同，但都在回答“结束后最小残留物是什么”。`

第五条：

这里也不能过度声称：

`只要 repo 某处已经会用 tombstone，stronger-request cleanup 线就自动拥有完整墓碑制度。`

当前更稳的说法只能是：

`repo 已在别的子系统里展示 tombstone governance 的思想；stronger-request cleanup 线当前缺的是把这套思想正式映射到旧 cleanup path、旧 promise、旧 receipt semantics 与旧解释面上。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer`

证据：

1. `query.ts:712-719`
   orphaned messages 不只被判定为失效，还会显式产出 `tombstone`
2. `utils/messages.ts:2954-2957`
   与
   `QueryEngine.ts:758-760`
   证明 tombstone 是 control signal，而不是普通 message
3. `REPL.tsx:2646-2648`
   与
   `sessionStorage.ts:1470-1473,925-945`
   证明 tombstone 会继续驱动显示层与 transcript 存储层改口
4. `.orphaned_at`
   不是 sunset 时点本身，而是对象退役后留在磁盘上的最小 marker
   `cacheUtils.ts:56-72,145-171`; `pluginOperations.ts:523-547`
5. marker 还会继续塑造 future search / visibility world
   `orphanedPluginFilter.ts:1-15,35-38`; `main.tsx:2544-2568`
6. migration timestamp 证明对象世界变化之后，系统有时还要保留一个最小历史注脚去解释“为什么当前世界刚变”
   `config.ts:427-438`; `useModelMigrationNotifications.tsx:5-50`
7. stronger-request cleanup 当前虽已开始显露 sunset-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup path、旧 promise vocabulary 与旧 receipt semantics 退役后还应留下什么最小 marker

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留事实来约束未来解释。`

再压成一句：

`会退役，不等于会留下可治理的最小残留事实。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
这两层治理处理的是两个不同真相面的主权问题。

`sunset governor`

回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

`tombstone governor`

回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. marker 将怎样约束未来读取、恢复、审计与解释
5. marker 何时再被进一步清除，谁对这次二次清除负责

所以这两层的哲学本质不同：

`sunset`

决定旧对象什么时候停止作为 current truth；

`tombstone`

则决定它停止之后，
系统还保留什么最小事实来约束后来者。

Claude Code 在这里真正先进的地方，
不是它“会在对象无效后把它删掉”，
而是它已经明确承认：

`删除不是全部，最小残留事实同样需要被制度化。`

这是一种很成熟的安全哲学：

`对象停止有效，不等于它留下的解释义务也随之消失。`

因为一旦把 sunset 与 tombstone 压成一层，
系统就容易出现三种危险：

1. 旧对象虽然不再有效，但未来读者不知道它为什么消失
2. 旧对象虽然已经失效，但机器世界缺乏最小 marker 去防止误复活
3. 删除动作虽然完成，但搜索、恢复、通知与审计面仍可能继续引用旧残影

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“墓碑治理”是一层独立制度

### 5.1 message 线

`query.ts` 在 streaming fallback 场景里显式产出 `tombstone`，
说明系统并不满足于：

`旧消息无效了。`

它还必须额外表达：

`旧消息虽然无效，但后续消费者仍要知道该删的是谁。`

随后 `utils/messages.ts` 把 tombstone 路由到 `onTombstone`，
`QueryEngine.ts` 明确把它视为 control signal，
`REPL.tsx` 与 `sessionStorage.ts` 再把它落实为 UI / transcript 删除动作。

这条链条显示出一个关键先进性：

`失效判定`、`残留表达`、`后续消费`

被拆成了连续但不等价的三层。

### 5.2 plugin 线

`.orphaned_at` 的价值不在于它是一个隐藏文件，
而在于它是对象退役之后留下的最小机器可解释残留物。

`markPluginVersionOrphaned()` 负责把旧版本推进到 orphan world，
`processOrphanedPluginVersion()` 负责读取 marker 并决定是否真正删除，
`pluginOperations.ts` 又把最后一个 scope 被卸载时的 install path 明确送去打标。

这说明在 plugin 线里，
系统并不是：

`旧版本不再 current -> 立刻抹平一切痕迹`

而是：

`旧版本不再 current -> 先留下最小 marker -> 再依据 marker 决定后续删除与可见性命运`

### 5.3 search / visibility 线

`orphanedPluginFilter.ts`
最值钱的地方在于它证明：

`墓碑不只是为了记过去，还会继续约束未来读取。`

系统扫描 `.orphaned_at`，
把 marker 转成 grep / glob exclusion patterns，
再在 `main.tsx` 启动阶段把 exclusion cache 预热并冻结到当前 session。

这意味着 tombstone-like marker 的职责不只是：

`告诉人“旧东西退役了”。`

它还要进一步告诉机器：

`既然它退役了，以后搜索、索引、浏览时就别再把它当作当前候选。`

### 5.4 model 迁移线

`config.ts` 里的 migration timestamp 与 completion flag，
再加上 `useModelMigrationNotifications.tsx` 的一次性通知链，
提供了另一类更克制的 tombstone-like grammar。

这里保留下来的不是旧模型对象本身，
而是：

`世界刚刚切换过`

这一最小历史事实。

它的作用不是让旧模型继续生效，
而是让当前 runtime 能解释：

`为什么你这次启动看到的世界和上次不一样。`

这类机制很重要，
因为它说明 tombstone governance 的核心并不依赖统一外形。
真正稳定的定义不是“长得像墓碑”，
而是：

`它是否在对象退役后承担最小残留事实的表达与消费责任。`

## 6. 苏格拉底诘问：怎样避免把“退役”误写成“墓碑已经自动成立”

### 诘问一

`如果旧世界已经退役，为什么不直接删干净，为什么还要保留最小残留事实？`

因为系统面对的不是单点删除，
而是并发 session、延迟消费、未来搜索、历史解释与误复活风险。
没有最小残留事实，
删除就只是一次当前动作，
而不是一条可被后来者持续遵守的制度。

### 诘问二

`如果 marker 还在，这是否意味着旧世界仍然有效？`

不是。
marker 的职责恰恰是把：

`旧世界已经失效`

这件事，
以机器还能继续消费的形式留下来。

marker presence 不是 current-world validity，
而是 anti-resurrection residue。

### 诘问三

`如果系统已经给过 warning 或 notification，是否就等于拥有完整 tombstone governance？`

还不等于。
完整墓碑治理至少还要回答：

1. marker 是什么
2. 谁消费 marker
3. marker 继续影响哪些 future surface
4. marker 何时再被清除

只给人看一次通知，
还不构成完整的 tombstone grammar。

### 诘问四

`如果 migration 已经完成，为什么还要继续区分 sunset 与 tombstone？`

因为 migration 解决的是世界如何切换，
sunset 解决的是旧世界何时停止，
tombstone 解决的是停止之后留下什么最小残留事实。

三者分别治理：

`过渡`

`结束`

`结束后的最小事实残留`

少一层，
就会把一个时间面的问题误写成另一个时间面的问题。

## 7. 技术启示与哲学本质

Claude Code 多重安全技术的先进性，
不在于它把每个子系统都做成同一种统一框架，
而在于它已经在不同对象家族上反复证明一条更底层的原则：

`安全治理不是只有准入和删除，还包括对象退出真相面之后，最小残留事实如何继续被制度化消费。`

这条原则的哲学本质可以压成三句话：

1. `current truth` 需要边界
2. `historical residue` 也需要边界
3. 两种边界不能由同一个时间点粗暴代管

因此，对 stronger-request cleanup 线而言，
下一层最值得坚持的不是继续堆更多“何时退役”的说法，
而是正式回答：

`一旦旧 cleanup truth 退出 current world，系统究竟留下什么最小残留事实，来防止它在未来被误引用、误恢复、误解释或误复活。`
