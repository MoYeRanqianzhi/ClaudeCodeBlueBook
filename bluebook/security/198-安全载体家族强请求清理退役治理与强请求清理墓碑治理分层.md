# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `197` 之后还必须继续写 `198`

`197-安全载体家族强请求清理迁移治理与强请求清理退役治理分层` 已经回答了：

`stronger-request cleanup 线即便知道旧世界怎样过渡，也还需要 sunset governor 去决定兼容期何时正式结束。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“宣布退役就够了”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界不再算当前 truth，它就自动拥有了决定退役后还要留下什么最小残留标记的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/query.ts:713-719` 会为 orphaned messages 显式产出 `tombstone`
2. `src/utils/messages.ts:2954-2957`、`src/QueryEngine.ts:756-759`、`src/screens/REPL.tsx:2646-2648` 与 `src/utils/sessionStorage.ts:1466-1473,923-944` 证明 `tombstone` 不是普通正文，而是会继续驱动 UI / transcript 删除的 control signal
3. `src/utils/plugins/cacheUtils.ts:56-72,145-171` 与 `src/services/plugins/pluginOperations.ts:526-547` 证明 `.orphaned_at` 不是 sunset 时点本身，而是退役后留在磁盘上的最小 marker
4. `src/utils/plugins/orphanedPluginFilter.ts:1-79` 与 `src/main.tsx:2546-2568` 又进一步证明 marker 还会被转译成 grep / glob exclusion grammar，继续约束未来读取世界
5. `src/utils/config.ts:428-438` 的 migration timestamps 与 `src/hooks/notifs/useModelMigrationNotifications.tsx:1-33` 证明对象世界已经变化之后，系统有时还要保留一个最小历史注脚，向当前 runtime 解释“为什么现在变了”
6. `197` 已经钉住 visibility sunset 与 storage sunset 不是同一件事；而这里继续看到的 `tombstone` / `.orphaned_at` / migration timestamp 则说明  
   `退役之后留下什么最小残留物`  
   也不是 `sunset governance` 自动回答的同一个问题

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界从什么时候起不再算当前正式世界。`

后者才配说：

`它退役之后还要留下什么最小残留标记，让后来的 runtime、搜索面、恢复链与读者不会把“已经退役”重新读成“从未发生”或“仍可继续援引”。`

所以 `197` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责宣布结束；stronger-request cleanup tombstone governor 才负责治理结束之后留下的最小残留事实。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 tombstone manager，
而是在说：

1. repo 在 message / plugin / model 线上已经展示出真实的 marker grammar
2. 这些 marker grammar 不等于 sunset deadline 本身
3. 它们承担的是退役后的最小残留事实表达

第二条：

这里说的 `tombstone governance`

不是在声称：

`退役之后一定要保留完整旧对象。`

真正要说的是：

`墓碑治理回答的是退役后保留多少最小事实，既足以约束未来解释，又不把旧对象重新抬回 current world。`

第三条：

这里也不是把：

`migration timestamp`

一概硬写成字面 tombstone。

当前更稳的说法只能是：

`有些残留物虽然源码里不直接叫 tombstone，但功能上承担了 post-transition trace artifact 的角色，因此属于 tombstone-like marker。`

第四条：

这里尤其不能过度声称：

`只要对象退役，就一定先有 tombstone message。`

当前更稳的说法只能是：

`repo 展示了多种 marker grammar：message tombstone、plugin marker、migration timestamp；它们形态不同，但都在回答“结束后最小残留物是什么”。`

第五条：

这里也不能过度声称：

`只要有 marker，就代表 cleanup 线已经拥有完整墓碑制度。`

当前更稳的说法只能是：

`repo 已在别的子系统里展示 tombstone governance 的思想；stronger-request cleanup 线当前缺的是把这套思想正式映射到旧 plan path、旧 startup wording 与旧 receipt semantics 上。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类“stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer”证据：

1. `query.ts:713-719` 遇到 orphaned messages 时不会只静默丢弃旧消息，而是显式 `yield { type: 'tombstone' }`
2. `utils/messages.ts:2954-2957`、`QueryEngine.ts:756-759`、`REPL.tsx:2646-2648` 与 `sessionStorage.ts:1466-1473,923-944` 证明 tombstone 会继续驱动显示层与 transcript 存储层改口
3. `cacheUtils.ts:56-72,145-171` 与 `pluginOperations.ts:526-547` 证明 `.orphaned_at` 是对象退役后留在磁盘上的最小 marker，而不是 sunset 时点本身
4. `orphanedPluginFilter.ts:1-79` 与 `main.tsx:2546-2568` 证明 marker 还会继续塑造 future search / visibility world
5. `config.ts:428-438` 与 `useModelMigrationNotifications.tsx:1-33` 证明对象世界已经变化之后，系统有时还要保留一个最小历史注脚去解释“为什么刚刚变了”
6. `197` 已经证明 `visibility sunset` 与 `storage sunset` 仍不是同一件事；而这里继续看到的 marker grammar 又进一步证明  
   `何时结束`  
   和  
   `结束后留下什么最小残留物`  
   也不是同一层主权

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留标记来约束未来解释。`

再压成一句：

`会退役，不等于会留下正确的退役痕迹。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
强请求清理退役治理与强请求清理墓碑治理处理的是两个不同真相面的主权问题。

stronger-request cleanup sunset governor 回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

stronger-request cleanup tombstone governor 回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. 这个 marker 何时再被进一步清除
5. 它究竟是给未来读者解释历史，还是给 runtime 防止误复活，还是两者兼有

如果把这两层压成一句“反正已经退役”，
系统就会制造五类危险幻觉：

1. retired-means-silent-disappearance illusion  
   只要对象已退役，就误以为应该无痕消失
2. no-marker-needed illusion  
   只要 hard close 成立，就误以为未来读者和 runtime 不需要最小历史标记
3. deletion-equals-explanation illusion  
   只要删掉旧对象，就误以为已经解释了它为何消失
4. residue-equals-trash illusion  
   只要还有残留物，就误以为它只是垃圾，而不是受治理的 marker
5. marker-equals-sunset illusion  
   只要看到 marker，就误以为 marker 自己就是退役时点，而不是退役之后的约束媒介

所以从第一性原理看：

`sunset governor` 决定对象什么时候失效；  
`tombstone governor` 决定对象失效后，系统还保留什么最小事实来避免未来继续说谎。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup sunset governance 已经等于 tombstone governance，为什么 `query.ts` 还要专门产出 `tombstone` control signal，而不是只靠“旧对象已无效”自动蒸发？
   因为“已经失效”与“后续消费者该删谁、避开谁”根本不是同一句制度。
2. 如果对象一旦退役就不需要最小残留物，为什么 plugin 线还要写 `.orphaned_at`，并让它继续驱动 7 天删除与 exclusion grammar？
   因为 post-retirement world 仍需要一个机器可消费的最小 marker。
3. 如果 migration timestamp 只是脚本副产品，为什么 `useModelMigrationNotifications()` 还要继续消费它来解释“这次启动为什么刚变”？
   因为对象世界变化之后，系统仍需要一个最小历史注脚向当前 reader 说明变化已经发生过。

## 5. message 线给出最强正例：对象退役之后，Claude Code 并不会只让它静默蒸发，而是显式产出 `tombstone`

`src/query.ts:713-719` 非常关键。
当 streaming fallback 造成 orphaned assistant messages 时，
系统并不是简单地把旧 partial messages 从内存里扔掉。

它会：

1. 为每个 orphaned message `yield { type: 'tombstone' }`
2. 记录 `tengu_orphaned_messages_tombstoned`
3. 清空旧 attempt 遗留的 `assistantMessages`、`toolResults` 与 `toolUseBlocks`

这说明对消息系统来说：

`对象已不再有效`

和

`应留下一个最小控制信号告诉下游怎样处理它`

是两件事。

继续看 `src/utils/messages.ts:2954-2957`：
stream handling 遇到 `message.type === 'tombstone'` 时，不是把它当正文内容渲染，而是调用 `onTombstone`。

再看 `src/QueryEngine.ts:756-759`：

`Tombstone messages are control signals for removing messages`

而 `src/screens/REPL.tsx:2646-2648` 与 `src/utils/sessionStorage.ts:1466-1473,923-944` 又把 tombstone 接到 UI / transcript 删除链上。

这组代码合在一起说明：

`tombstone` 不是描述性注脚，也不是普通通知。
它是退役后最小残留控制面的正式成员。

## 6. plugin 线给出第二组强正例：`.orphaned_at` 不是退役时钟，而是退役后的机器墓碑

`src/utils/plugins/cacheUtils.ts:56-72` 的 `markPluginVersionOrphaned()` 和注释非常值钱。

这里的 `.orphaned_at` 不回答：

`这个版本哪一天应该退役。`

因为那个问题已经由 orphan 进入 grace world 与 `CLEANUP_AGE_MS = 7 days` 回答了。
`.orphaned_at` 回答的是另一件事：

`它既然已经进入退役世界，系统接下来靠什么最小残留物持续记住这一事实。`

继续看 `processOrphanedPluginVersion():145-171`：

1. 如果没有 `.orphaned_at`，就先写入 marker
2. 如果 marker 已存在且超过 7 天，再真正删除

再看 `src/services/plugins/pluginOperations.ts:526-547`：
最后一个 scope 卸载后的 install path 会交给 `markPluginVersionOrphaned()`，
说明 `.orphaned_at` 不是清理附言，而是卸载后 post-retirement world 的正式入口。

这意味着：

1. sunset governance 决定旧版本已进入 orphan world
2. tombstone governance 决定 orphan world 通过什么最小残留物被机器持续看见

## 7. marker-driven exclusion 再证明：墓碑不只服务历史解释，它还会主动约束未来读取世界

`src/utils/plugins/orphanedPluginFilter.ts:1-79` 与 `src/main.tsx:2546-2568` 特别值钱。

这里的流程是：

1. 先完成 orphan GC
2. 再扫描 `.orphaned_at`
3. 基于 marker 生成 grep / glob exclusion patterns
4. 把 exclusion cache 冻结在当前 session

这意味着 `.orphaned_at` 的作用不只是：

`证明旧版本退役过。`

它还会继续决定：

`未来的搜索与读取不应再触及哪些目录。`

这把墓碑治理从“历史注脚”提升成了“未来约束”。

对 stronger-request cleanup 线的启示非常直接：

`旧 cleanup world 退役后是否还要留下 marker，不只关系解释历史，也关系后来者会不会重新踩回旧路径。`

## 8. model 迁移时间戳给出第三类正对照：有些墓碑不长得像 tombstone，但功能上就是“变化已经发生过”的最小残留事实

`src/utils/config.ts:428-438` 定义了：

1. `opusProMigrationTimestamp`
2. `legacyOpusMigrationTimestamp`
3. `sonnet45To46MigrationTimestamp`

这些字段都不是 migration 脚本本身，
也不是 sunset 日期本身。
它们更接近一种：

`post-transition trace artifact`

也就是：

`这次变化已经发生过，系统需要留下一个最小历史注脚，供后续解释当前世界为什么变成这样。`

`src/hooks/notifs/useModelMigrationNotifications.tsx:1-33` 正是靠这些时间戳决定：

1. 是否在本次启动中发一次 notification
2. 是否提示用户现在已经迁到新模型
3. 是否展示 opt-out 说明

这里必须谨慎说：
这些 timestamps 在源码里并没有被字面叫作 tombstone。
但从功能上看，
它们确实承担了 tombstone-like marker 的职责：

`不再重做迁移，而是留下一个最小残留事实，告诉当前系统这场变化刚刚发生过。`

所以 repo 在模型线上也已经说明：

`世界改变之后，常常还需要一个比“已经改完”更小、但又不能没有的历史标记。`

## 9. 技术先进性与哲学本质：Claude Code 真正先进的地方，是它在多个子系统里把“对象结束”与“结束后最小残留物”继续拆成两层

从技术角度看，
Claude Code 在这条线上的先进性，
不只是它会宣布 retirement，
而是它已经在多个子系统里承认：

1. invalid 之后不一定是 silent disappearance
2. marker 不等于正文，不等于普通通知，而是 machine-consumable control artifact
3. 最小残留物既可以驱动删除，也可以驱动隐藏，也可以驱动解释
4. 受治理的 residue 与未治理的 stale junk 不是同一回事

这背后的哲学本质是：

`真正成熟的治理，不只知道何时宣布旧世界结束，还知道结束后必须留下多少历史痕迹，才足够防止未来继续说谎。`

这也带来三条直接技术启示：

1. 任何 stronger-request cleanup grammar 若只定义 sunset clock、不定义旧 path、旧 promise、旧 receipt 的 tombstone grammar，就仍缺 post-retirement truth discipline
2. plugin 的 `.orphaned_at -> exclusion -> delayed delete` 三段式说明，墓碑不是单纯留痕，而是会继续塑造 future read world
3. 只让旧世界退役、却不给后来者留下最小合法残留标记，系统就会停在另一种半治理状态：
   `大家知道旧世界不算数，但没人知道该靠什么机器信号来证明它已经不算数`

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要决定了旧世界何时退役，就已经自动拥有了治理退役后最小残留标记的主权。`

而是：

`repo 在 tombstone messages、.orphaned_at markers、marker-driven exclusion grammar 与 migration timestamps 上已经明确展示了 tombstone governance 的存在；因此 artifact-family cleanup stronger-request cleanup-sunset-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

再压成最后一句：

`退役负责宣布对象结束；墓碑，才负责让后来者知道它为何结束、并且别再把它当回事。`
