# 安全完成差异升级规则：哪些宿主盲区必须抢占界面，哪些只能下钻显示

## 1. 为什么在 `46` 之后还要继续写“完成差异升级规则”

`46-安全完成差异卡片` 已经回答了：

`宿主盲区字段该挂到哪张卡、按什么优先级显示。`

但如果继续往交互层再压一层，  
还会出现一个更动态的问题：

`哪些宿主盲区必须立即抢占界面，哪些只需要中等提示，哪些适合下钻显示？`

因为“显示在前面”和“抢占当前界面”仍然不是一回事。  
很多安全问题真正危险的地方在于：

1. 用户正在按一个过满结论行动
2. 控制面虽然知道这个结论不该说满
3. 但这个修正信号如果不及时抢占，用户就已经做错下一步

所以统一安全控制台真正还差一条规则：

`什么情况下必须立刻打断用户，什么情况下只该发高优先级警示，什么情况下只需保持可见但不抢占。`

## 2. 最短结论

从源码看，Claude Code 已经在多处建立了非常明确的优先级语义：

1. `immediate`
2. `high`
3. `medium`
4. `low`

而且这些优先级不是纯 UI 偏好，  
它们已经隐含着一套成熟判断：

`当前问题是否正在实时破坏用户对系统状态的理解，是否会立即误导下一步动作，是否需要抢占当前认知通道。`

因此下一代统一安全控制台最合理的做法，不是所有宿主盲区都统一显示，  
而是把它们分成四层升级规则：

1. `immediate`: 当前过满结论必须马上被打断
2. `high`: 当前动作路径已被结构性改写，必须强提醒
3. `medium`: 当前能力显著退化，但不必抢断
4. `low`: 当前存在差异或刷新需求，但不宜打断主任务

所以这一章的最短结论是：

`宿主盲区显化不只需要字段和卡片，还需要一套基于误导风险的升级规则。`

## 3. 现有源码已经有一套成熟的优先级机制

## 3.1 `notifications.tsx` 直接把优先级做成一等控制语义

`src/context/notifications.tsx:5-33` 把优先级定义成：

1. `low`
2. `medium`
3. `high`
4. `immediate`

更关键的是，  
`src/context/notifications.tsx:78-116` 明确说明：

1. `immediate` 会立刻显示
2. 会清掉当前 timeout
3. 会把原当前通知重新排队
4. 允许通过 `invalidates` 清理旧通知

这说明优先级在源码里根本不是样式字段，  
而是：

`注意力仲裁字段。`

也就是说，  
系统已经具备：

`把更危险的事实抢到用户眼前`

的正式能力。

## 4. 第一层规则：`immediate` 只该用于“当前过满结论必须被立刻推翻”

## 4.1 bridge 失败就是最典型的 immediate

`src/hooks/useReplBridge.tsx:102-111` 很关键。  
当 bridge fail 时，它直接发：

`Remote Control failed`

并且优先级是：

`immediate`

这里的设计非常成熟。  
因为 bridge failure 不是普通能力退化，  
而是在说：

`你刚刚可能还以为这条远程完成链是成立的，但现在这个前提已经被立刻推翻。`

因此它必须抢占界面，  
而不是排队等一会再说。

## 4.2 从第一性原理看，只有“正在实时推翻当前结论”的盲区才配 immediate

对于统一安全控制台，这意味着：

`immediate` 不该滥用。  
它只适用于那种如果不立刻打断，用户就会继续在错误完成语义上操作的场景。

例如：

1. recovery 窗口刚打开
2. result/control projection 突然被暂停
3. 之前可用的远程完成链突然失效

这类问题不是“告诉用户一下”，  
而是：

`立即撤销用户刚刚获得的确定性。`

## 5. 第二层规则：`high` 用于“当前动作路径已被结构性改写”

## 5.1 插件被 flagged 属于高优先级而非 immediate

`src/hooks/useManagePlugins.ts:59-67` 会在插件被 flagged 时发：

`Plugins flagged. Check /plugins`

优先级：

`high`

这说明作者认为这里需要强提醒，  
但还不需要像 bridge hard fail 一样瞬时打断。

它的语义更像：

`你的环境正在发生需要尽快处理的结构变化，但不一定要中断你当前这一秒的动作。`

## 5.2 channels blocked 也被归到 high

`src/services/mcp/useManageMCPConnections.ts:596-608` 对：

1. `disabled`
2. `auth`
3. `policy`

几种 gate 都发：

`priority: 'high'`

这说明：

1. 当前路径确实被结构性改变
2. 用户的下一步行为必须被纠正
3. 但这个纠正不一定要以 immediate 抢断形式发生

对于宿主盲区显化，这意味着：

`high` 适合那种如果不尽快纠正，用户接下来几步会持续沿错误路径走下去，但当前并没有一条正在实时崩塌的强完成链。`

## 6. 第三层规则：`medium` 用于“能力明显退化，但不必立刻抢断”

## 6.1 IDE disconnect 被放在 medium

`src/hooks/notifs/useIDEStatusIndicator.tsx:87-92` 对：

`ide disconnected`

使用：

`priority: 'medium'`

这很值得注意。  
因为 IDE 断开当然重要，  
但它并不等于当前整条安全完成链已经被推翻。

它更像是在说：

`你失去了一条观察/交互辅助通道，但这不必一定中断主任务。`

## 6.2 MCP failed / needs-auth 也多被归到 medium

`src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`  
对 MCP failed、claude.ai connector unavailable、needs-auth 都给：

`priority: 'medium'`

这说明作者已经在做一种很成熟的区分：

`不是所有“完成差异”都值得抢到最前。`

有些问题更适合：

1. 保持可见
2. 让用户知道能力受损
3. 给出 `/mcp` 跳转
4. 但不抢断当前主任务

对于统一安全控制台来说，  
这意味着很多宿主盲区字段如果只是说明：

`当前投影较薄，但没有立即推翻当前主任务的正确性`

它们更适合 medium 层，而不是 high/immediate。

## 7. 第四层规则：`low` 用于“真实但不宜打断的状态变化”

## 7.1 插件刷新待执行是 low

`src/hooks/useManagePlugins.ts:294-300` 对：

`Plugins changed. Run /reload-plugins to activate.`

使用：

`priority: 'low'`

这里的判断非常成熟。  
因为它在说：

`存在变化，但这不是立刻会把你当前理解推翻的差异。`

## 7.2 status line trust blocked 也是 low

`src/components/StatusLine.tsx:277-283` 对：

`statusline skipped · restart to fix`

也用：

`priority: 'low'`

这说明低优先级的适用条件非常清楚：

1. 系统确实存在差异或欠账
2. 但不需要现在强行改写用户的完成预期
3. 只需要留一个可恢复的提醒即可

## 8. 把这套优先级重新压回宿主盲区显化

我会把宿主完成差异的升级规则压成下面四句：

### 8.1 `immediate`

用于：

`当前宿主盲区正在实时推翻用户刚刚获得的完成结论。`

典型情况：

1. recovery 窗口开启
2. bridge fail
3. 完成投影从可解释突然降成不可解释

### 8.2 `high`

用于：

`当前宿主盲区已经改写了用户接下来的动作路径。`

典型情况：

1. policy block
2. channels blocked
3. 高风险插件被 flagged
4. completion ceiling 显著低于界面当前默认说法

### 8.3 `medium`

用于：

`当前宿主能力或完成投影明显变薄，但不必抢断。`

典型情况：

1. IDE disconnected
2. MCP failed / needs-auth
3. 当前 host 只有 event/replica 级 completion，但主任务未立即受损

### 8.4 `low`

用于：

`当前宿主差异真实存在，但更像配置维护、刷新或补齐动作。`

典型情况：

1. plugin reload pending
2. 某些非当前主任务关键的视图受限
3. 可延后处理的盲区补齐提示

## 9. 这对统一安全控制台意味着什么

这意味着宿主盲区字段不该只按“是什么卡片”来组织，  
还必须按：

`何时抢占`

来组织。

也就是说，  
在 `46` 的卡片结构之上，  
还要再套一层升级语义：

1. `completion_claim_ceiling` 正常情况下放标题层
2. 一旦 ceiling 被动态拉低，就可能升级为 high
3. 一旦 recovery 打开并撤销当前完成解释权，就应升级为 immediate
4. `missing_ledgers[]` 多数时候放正文，但在它直接导致当前动作无效时应提升

这说明统一安全控制台最终不只是：

`静态卡片布局`

而是：

`卡片 + 升级规则`

## 10. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`宿主盲区显化如果没有升级规则，就仍然会在关键时刻输给更好看的默认状态。`

这条公理几乎能统一前面所有证据：

1. 通知系统已有正式优先级仲裁
2. bridge 失败会 immediate 抢断
3. channels / flagged plugins 会 high 强提醒
4. IDE / MCP 连接退化通常只到 medium
5. refresh / trust 这类欠账可留在 low

所以在 `46` 之后，  
更进一步的产品化结论已经很清楚：

`不是所有宿主盲区都该平等上屏，而是必须按“会不会立刻误导下一步动作”来排序。`

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 当前连“注意力仲裁”这层基础设施都已经具备了，统一安全控制台真正缺的只是把宿主盲区接入这套优先级语义。`

这对其他 Agent 平台构建者的启示也很直接：

1. 盲区显化如果没有优先级，用户最终只会看到最好看的默认状态
2. `immediate` 应严格保留给“正在推翻当前结论”的问题
3. `high` 更适合“正在改写下一步动作路径”的问题
4. `medium` 适合能力显著退化但不必抢断的盲区
5. `low` 适合真实但可延后处理的维护型差异

## 12. 哲学本质

这一章更深层的哲学是：

`诚实不只是把限制说出来，还包括让更重要的限制赢得注意力。`

很多系统的问题不是不知道边界，  
而是边界在注意力竞争里总是输给“状态看起来没问题”的那条默认叙事。  
Claude Code 的底层已经给出了一个更成熟的方向：

`把注意力分配本身做成控制面的一部分。`

从第一性原理看，  
这保护的是一条非常根本的公理：

`如果更危险的边界信息不能先于更好看的默认状态出现，那它最终仍然等于没出现。`

## 13. 苏格拉底式反思：这一章最容易犯什么错

### 13.1 会不会把所有重要问题都提升成 high/immediate

这是最大风险。  
一旦滥用高优先级，  
系统会重新退化成噪声。

### 13.2 为什么不只靠用户自己点开详情

因为用户只有在已经怀疑时才会点开详情。  
而最危险的问题恰恰是：

`用户目前还没有意识到自己的完成理解已经不成立。`

### 13.3 这一章之后还缺什么

还缺最后一层更压缩的设计材料：

`把这些升级规则压成一张“字段组 -> 默认优先级 -> 升级条件 -> 禁止降级”的矩阵。`

也就是说，  
下一步最自然的延伸就是：

`appendix/31-安全完成差异升级规则速查表`

## 14. 结语

`46` 回答的是：宿主盲区字段该挂到哪张卡。  
这一章继续推进后的结论则是：

`这些字段还必须遵守一套基于误导风险的升级规则，决定何时抢占界面、何时只做下钻显示。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正成熟的控制面，不只会展示诚实边界，还会让更危险的边界先赢得用户注意力。`
