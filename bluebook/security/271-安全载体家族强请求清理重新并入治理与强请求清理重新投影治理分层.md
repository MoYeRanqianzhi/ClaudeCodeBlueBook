# 安全载体家族强请求清理重新并入治理与强请求清理重新投影治理分层：为什么artifact-family cleanup stronger-request cleanup-reintegration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reprojection-governor signer

## 1. 为什么在 `270` 之后还必须继续写 `271`

`270-安全载体家族强请求清理恢复治理与强请求清理重新并入治理分层`
已经回答了：

`stronger-request cleanup 线即便已经拿到 recovered truth，也还要单独回答这份 truth 何时重新并入当前 tool / status / event world。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然对象已经并回当前世界，那系统当然知道该怎样把它讲给所有读者”
的错觉：

`只要 stronger-request cleanup reintegration governor 已经让对象重新进入 current world，它就自动拥有了决定这份 current truth 该怎样被 control protocol、status pane、notification surface、health surface 与 reconnect dialog 重新讲述的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的重新投影正对照，
会发现 repo 其实明确拆开了六件事：

1. `control payload projection`
2. `manager / panel status grammar`
3. `selective notification publication`
4. `health glyph grammar`
5. `reconnect dialog copy`
6. `control envelope strength`

最硬的证据至少有七组：

1. `src/cli/print.ts:1610-1630,2957-2960,3127-3128`
   的 `buildMcpServerStatuses()` 与 `mcp_status` / `reload_plugins`
   control responses
2. `src/commands/plugin/ManagePlugins.tsx:512-519`
   的 `getMcpStatus()`
3. `src/components/mcp/MCPListPanel.tsx:306-337`
   的 `statusText` projection
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
   的 selective notification grammar
5. `src/cli/handlers/mcp.tsx:26-35`
   的 health glyph grammar
6. `src/components/mcp/MCPReconnect.tsx:40-63`
   的 action copy
7. `src/cli/print.ts:2736-2758`
   的 `sendControlResponseSuccess()` / `sendControlResponseError()`
   envelope discipline

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup reintegration governance`
   负责决定恢复后的对象何时重新进入 current world
2. `stronger-request cleanup reprojection governance`
   负责决定这份 current truth 怎样以不同粒度、不同句法、不同选择性被重新投影给不同 reader surface

也就是说：

`artifact-family cleanup stronger-request cleanup-reintegration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reprojection-governor signer`

仍然不是一回事。

前者最多能说：

`对象已经重新成为当前世界的一部分。`

后者才配说：

`这份当前真相现在该怎样被 control response、health、notification、panel status 与 reconnect copy 重新讲述给不同读者。`

所以 `270` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新并入治理与强请求清理重新投影治理分层`

也就是：

`stronger-request cleanup reintegration governor 决定对象何时重新进入当前世界；stronger-request cleanup reprojection governor 才决定进入之后这份真相怎样被不同 surface 重新讲述。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reprojection-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-reprojection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reprojection manager，
而是在说：

1. repo 已经把 reintegrated truth 重新讲述给多个 surface
2. 这些 surface 使用不同的枚举、文案、选择性提示与结构化 payload
3. repo 已经拒绝让某一份投影自动代表所有 reader 的 current truth

第二条：

这里说的 `reprojection governance`
不是在声称：

`每个读者都应该看到同样多、同样强的 current truth。`

当前更稳的说法只能是：

`reprojection governance 首先回答的是不同 reader 应看到哪种粒度、哪种词法强度、哪种 publication policy 的 current truth。`

第三条：

这里也不是把：

`buildMcpServerStatuses()`

单独当成全部 reprojection grammar。

真正关键的是：

1. `structured control projection`
2. `manager / panel status grammar`
3. `selective notification policy`
4. `health glyph projection`
5. `action-oriented reconnect copy`
6. `success / error control envelope`

第四条：

这里尤其不能过度声称：

`reprojection governance`

已经等于：

`reassurance governance`

当前更稳的说法只能是：

`reprojection governance 回答 current truth 怎样被重讲；reassurance governance 才回答这些重讲里哪些现在配承载多强的继续依赖担保。`

第五条：

这里也不能过度声称：

`某一份 status text`

就天然代表：

`所有 consumer 的 current truth`

当前更稳的说法只能是：

`同一份 reintegrated truth 在 control、health、notification、dialog 与 panel world 里本来就需要不同的 reader-specific projection。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-reintegration-governor signer 仍不等于 stronger-request cleanup-reprojection-governor signer`

证据：

1. `buildMcpServerStatuses()`
   会把同一份 current truth
   重新编码成 control protocol 用的结构化 `McpServerStatus[]`
2. `getMcpStatus()` 与 `MCPListPanel`
   又把同一 truth 压成 plugin manager / panel status enum 与 label
3. `useMcpConnectivityStatus()`
   只为 `failed / needs-auth` 发 toast，
   不为 `connected / pending` 发“恢复成功”通知；
   这说明 reprojection 有选择性 publication grammar
4. `handlers/mcp.tsx`
   把同一 truth 压成
   `✓ Connected / ! Needs authentication / ✗ Failed to connect`；
   这说明 health surface 的词法主权独立于 reintegration
5. `MCPReconnect`
   把同一 truth 压成
   `Successfully reconnected`、
   `requires authentication`
   与 generic failure copy；
   这说明 dialog / reconnect surface 的叙事强度也不是 reintegration 本身能回答的
6. `sendControlResponseSuccess()` / `sendControlResponseError()`
   又进一步说明 control plane 还要决定给读者 success only、error only 还是 success + payload

因此这一章的最短结论是：

`stronger-request cleanup reintegration governor 最多能说对象已经重新回到当前世界；stronger-request cleanup reprojection governor 才能说这份 current truth 现在应当怎样被不同 reader surface 重新讲述。`

再压成一句：

`reintegrated，不等于已经被所有读者按正确方式重新看见。`

## 4. 第一性原理：reintegration governance 回答“对象回到当前世界了吗”，reprojection governance 回答“这份当前真相该怎样被重新讲述”

从第一性原理看，
强请求清理重新并入治理与强请求清理重新投影治理处理的是两个不同主权问题。

`stronger-request cleanup reintegration governor`
回答的是：

1. recovered client 是否已进入 current state
2. recovered tools / commands / resources 是否已回到 current tool pool
3. event plane 是否重新接回
4. next turn consumer 是否已经能读到它
5. 对象是否重新成为 current world 的一部分

`stronger-request cleanup reprojection governor`
回答的则是：

1. 哪些 reader 该看到结构化 status
2. 哪些 reader 该看到 glyph / label / copy
3. 哪些 reader 只该看到 failure / auth warning，而不该看到 success toast
4. 同一 truth 在不同 surface 上该被压成怎样的词法强度
5. 哪些 surface 该重讲 current truth，哪些 surface 该保持沉默

如果把这两层压成一句
“反正已经并回来了”，
系统就会制造五类危险幻觉：

1. `integrated-means-announced`
2. `one-surface-speaks-for-all`
3. `success-toast-is-mandatory`
4. `health-string-equals-control-truth`
5. `reconnect-copy-equals-status-plane`

再用苏格拉底式反问压一次：

1. 如果 reintegration governance 已经等于 reprojection governance，
   为什么 `buildMcpServerStatuses()` 还要把同一份 truth
   重新编码成专门给 control reader 看的结构化 status？
   因为“对象已经回到当前世界”与“control reader 该怎么看它”根本不是同一句制度。
2. 如果同一份 current truth 天然适合所有读者，
   为什么 `useMcpConnectivityStatus()` 只发布 `failed / needs-auth`，
   却不对称发布 `connected / pending`？
   因为 publication policy 也是单独的 reader-facing governance。
3. 如果 reconnect dialog 的 success / failure copy 就已经代表全部 current truth，
   为什么 panel、health 与 control response 还要各自保留不同 grammar？
   因为不同读者需要的不是同一份语言复制，
   而是 reader-specific projection honesty。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“重新投影治理”是一层独立制度

### 5.1 `buildMcpServerStatuses()` 先证明：reintegrated truth 进入 control protocol 仍要经过单独的结构化投影

`src/cli/print.ts:1610-1630`
很值钱。

这里 `buildMcpServerStatuses()` 不只是回读当前对象，
而是在 `appState` / `sdkClients` / `dynamicMcpState` 之上重新组织出：

1. `status`
2. `serverInfo`
3. `error`
4. `config`
5. `scope`
6. `tools`
7. `capabilities`

随后
`2957-2960`
与
`3127-3128`
又分别把这份结构投影到：

1. `mcp_status`
2. `reload_plugins`

这说明 control protocol 自己并不直接消费 raw reintegration state。
它要的是真相的专门投影版本。

这条证据非常硬。
因为它说明：

`current-world membership`

并不自动等于：

`control-reader-ready publication`

### 5.2 `ManagePlugins` 与 `MCPListPanel` 再证明：同一份 reintegrated truth 还会被压成不同 UI status grammar

`src/commands/plugin/ManagePlugins.tsx:512-519`
的 `getMcpStatus()`
把同一 truth 压成：

`connected / disabled / pending / needs-auth / failed`

这是一个 manager-facing normalized enum。

`src/components/mcp/MCPListPanel.tsx:306-337`
又继续把同一 truth 压成：

1. `disabled`
2. `connected`
3. `reconnecting (n/m)…`
4. `connecting…`
5. `needs authentication`
6. `failed`

也就是说，
同一份已经 reintegrated 的 truth，
在 UI world 里也不是自动等于同一句 current statement。

repo 已经明确让：

`what this reader needs to know`

参与决定如何重新讲 truth。

### 5.3 `useMcpConnectivityStatus()` 证明：重新投影还包括“沉默治理”

`src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
特别值钱。

这里只发：

1. `failed`
2. `needs-auth`

两类 notification。

它不会对：

1. `connected`
2. `pending`

发出对称的 success / recovering toast。

从第一性原理看，
这非常先进。

它说明 reprojection governance 的本体不只包含：

`怎么讲`

还包含：

`哪些 truth 此刻根本不该被主动讲`

这正是 reader-facing projection honesty。

### 5.4 `/mcp` health 与 `MCPReconnect` 再证明：health surface 和 dialog surface 各自拥有不同的重新讲述权

`src/cli/handlers/mcp.tsx:26-35`
把同一 truth 压成：

1. `✓ Connected`
2. `! Needs authentication`
3. `✗ Failed to connect`

这是 health-check reader 的 glyph grammar。

`src/components/mcp/MCPReconnect.tsx:40-63`
则把同一 truth 压成：

1. `Successfully reconnected to ...`
2. `... requires authentication. Use /mcp to authenticate.`
3. `Failed to reconnect to ...`

这里最值钱的不是“有两套文案”，
而是 repo 公开承认：

1. health reader 需要的是简短状态符号
2. dialog reader 需要的是 action-oriented operation story

这说明 reprojection governance 是对读者类型的正式分配，
不是对同一句 current truth 的机械复制。

### 5.5 `sendControlResponseSuccess()` / `sendControlResponseError()` 再证明：control plane 还要单独决定投影粒度

`src/cli/print.ts:2736-2758`
很值钱。

这里 control plane 本身就强制 truth 以：

1. `success`
2. `error`
3. optional `response payload`

的 envelope 重新输出。

再看
`3148-3204`
与
`3258-3295`，
就会发现同一 reintegrated truth
在 control plane 里仍要继续被裁成：

1. success with no payload
2. error with `Server status: ...`
3. structured status payload

这说明即便 control protocol 自己，
也没有把 reintegrated truth 当成
“天然可直接发给读者”的东西。

它还要单独决定：

`这个读者现在该拿到多少 truth`

## 6. 安全设计的哲学本质：Claude Code 在这里保护的不是“它回来了没有”，而是“谁配决定不同读者现在该看到哪种版本的它”

这组源码在哲学上最值钱的地方，
不是它有 status、toast、health、dialog 和 control response，
而是它拒绝把：

`已经并回 current world`

偷写成：

`所有读者已经被正确告知`

从第一性原理看，
它保护的是四件更根本的东西：

1. 同一 truth 的读者并不相同
2. 不同读者需要的 payload 厚度并不相同
3. 主动发布与保持沉默同样是治理动作
4. 一份 current truth 不能靠单一 surface 垄断解释权

这意味着 Claude Code 的先进性不只在于
“它能把对象接回来”，
而在于
“它知道接回来之后还要重新分配讲述权”。

真正成熟的安全系统，
不只问：

`它现在是不是 current world 的一部分`

还问：

`哪位读者现在该听到哪一种版本的 current truth`

## 7. 苏格拉底式自校：如果把 reprojection 说早了，会在哪些地方立即露馅

可以用五个问题反过来检查自己有没有把 reprojection 说早：

1. 如果 reintegrated 已经等于 reprojection 完成，
   那么 `buildMcpServerStatuses()` 为什么还要额外重组结构化 payload？
2. 如果一份 status 足够所有 reader，
   那么 `ManagePlugins` 和 `MCPListPanel` 为什么还要保留不同 grammar？
3. 如果 success 天然该被主动广播，
   那么 `useMcpConnectivityStatus()` 为什么只广播 failure / auth？
4. 如果 health glyph 已经足够，
   那么 reconnect dialog 为什么还要保留 action-oriented copy？
5. 如果 control truth 已经完整，
   那么 `sendControlResponseSuccess()` / `sendControlResponseError()` 为什么还要继续选择 envelope 与 payload 强度？

只要这五个问题里任何一个答不上来，
就说明自己已经把：

`reintegration truth`

越级偷写成了：

`reprojection truth`

## 8. 一条硬结论

这一层真正说明的不是：

`cleanup 线未来只要补出 reintegration governance，就已经足够成熟。`

而是：

`repo 在 buildMcpServerStatuses() 的 structured status builder、getMcpStatus() / MCPListPanel 的 status grammar、useMcpConnectivityStatus() 的 selective notification policy、handlers/mcp.tsx 的 health glyph grammar、MCPReconnect 的 action-oriented copy，以及 control response envelope 的 success / error payload discipline 上，已经明确展示了 reprojection governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reintegration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reprojection-governor signer。`

因此：

`stronger-request cleanup 线真正缺的不是“它回到当前世界了吗”，而是“不同读者现在该怎样被告知这份 current truth”。`
