# 安全载体家族强请求清理迁移治理与强请求清理退役治理分层：为什么artifact-family cleanup stronger-request cleanup-migration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-sunset-governor signer

## 1. 为什么在 `292` 之后还必须继续写 `293`

`292-安全载体家族强请求清理修复治理与强请求清理迁移治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样被修后带过渡，也仍需要 migration governor 去决定 rewrite、remap、grace window、notice 与 continuity policy。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“会迁就等于会退役”的错觉：

`只要 stronger-request cleanup migration governor 已经安排了新旧世界怎样并存，它就自动拥有宣布旧 stronger-request 世界正式结束的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 自己最成熟的 sunset 正对照，
会发现它其实已经把：

`migration clock`

和

`sunset clock`

拆成了两种不同制度。

最硬的证据至少有六组：

1. `src/utils/model/deprecation.ts:21-25,33-61,66-100`
   已为 deprecated models 维护 provider-specific retirement dates，
   说明 repo 区分
   `已经存在迁移路径`
   和
   `旧模型在哪个 provider、哪一天正式 retired`
2. `src/hooks/notifs/useModelMigrationNotifications.tsx:5-8,21-33,49-50`
   只负责一次性宣布
   `这次启动刚刚把你带到新默认值`
3. `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-27,29-42`
   与
   `src/main.tsx:2872-2895`
   则负责持续提示
   `旧模型即将在某日退役`
4. `src/migrations/migrateLegacyOpusToCurrent.ts:13-27,29-57`
   即便已把 `userSettings` 迁到 `opus`，
   `src/utils/model/model.ts:549-554`
   仍保留 legacy remap opt-out，
   说明 migration 已发生时旧显式字符串仍可能继续以兼容形式活着
5. `src/utils/plugins/cacheUtils.ts:23-24,66-74,89-105,149-176`
   说明 orphan plugin version 先进入 grace world，再在 7 天后删除；
   这不是 migration 本身，而是 sunset deadline
6. `src/utils/plugins/orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
   与
   `src/main.tsx:2544-2568`
   又进一步给出 visibility sunset：
   旧 plugin version 可能先从当前 session 的 grep / glob world 退役，
   再从 storage world 真正消失

更关键的是，
`src/main.tsx:323-345`
的 `CURRENT_MIGRATION_VERSION`
虽然明确展示了 startup migration registry，
但它回答的只是：

`哪些迁移动作要重新跑`

而不是：

`旧兼容世界从哪一天、哪个 surface、哪个 provider 起不再算 current truth`

这说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界先怎样被带过渡。`

后者才配说：

`过渡什么时候结束，旧 stronger-request law 从哪一个日期、哪一个 runtime、哪一个 visibility surface 起正式不再算当前世界。`

所以 `292` 之后必须继续补的一层就是：

`安全载体家族强请求清理迁移治理与强请求清理退役治理分层`

也就是：

`stronger-request cleanup migration governor 负责 coexistence grammar；stronger-request cleanup sunset governor 才负责宣布 coexistence 结束。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-sunset-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经存在一个未导出的退役 manager，
而是在说：

1. repo 在 model 与 plugin 线已经展示出正式 sunset grammar
2. sunset grammar 回答的是 retirement date、warning phase、visibility cutoff 与 hard-close boundary
3. stronger-request cleanup 若要自称完整，就必须回答这些“何时正式结束”的问题

第二条：

这里说的 `sunset governance`
不是在声称：

`旧世界一旦退役，就必须立刻从磁盘彻底消失。`

更稳的说法只能是：

`sunset 首先回答的是“从什么时候起它不再算当前世界”，这可以先发生在 warning、visibility、acceptance semantics 上，再发生在 storage deletion 上。`

第三条：

这里也不是把：

`provider-specific retirement date`

单独当成全部 sunset grammar。

真正关键的是：

1. model line 证明 migration notice 与 deprecation warning 是两张时钟
2. remap opt-out 证明 migration 已发生时旧 world 仍可兼容存在
3. plugin line 又证明 visibility sunset 与 storage sunset 还能继续拆层
4. `CURRENT_MIGRATION_VERSION` 更进一步说明 migration registry 也不等于 sunset registry

第四条：

这里尤其不能过度声称：

`warning surface` 就已经等于 `sunset governance`。

更稳的说法只能是：

`warning surface` 只是退役语法的一种表达；它仍需要与 retirement date、visibility cutoff、runtime acceptance cutoff 与 storage cutoff 一起构成完整 sunset grammar。`

第五条：

这里也不能把：

`runtime remap 仍然存在`

误写成：

`sunset 尚不存在。`

更稳的说法只能是：

`runtime remap 的继续存在恰恰证明 migration 与 sunset 被拆开了；旧世界可以被兼容一段时间，而不是迁移一发生就被自动 hard-close。`

第六条：

这里还不能过度声称：

`Claude Code 在 model / plugin 线上怎样 sunset，stronger-request cleanup 线未来就必须完全照抄。`

更稳的说法只能是：

`不同旧世界可以有不同 cutoff grammar，但“谁来宣布兼容期结束”必须被正式回答；不能因为实现语法可变，就把 sunset 主权偷写成 migration 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-migration-governor signer 仍不等于 stronger-request cleanup-sunset-governor signer`

证据：

1. `deprecation.ts`
   已为 deprecated models 维护 provider-specific retirement dates，
   说明 repo 已把 retirement date 当成正式 truth
2. `useModelMigrationNotifications.tsx`
   只负责 one-time migration notification，
   而
   `useDeprecationWarningNotification.tsx`
   与
   `main.tsx`
   负责持续 deprecation warning
3. `migrateLegacyOpusToCurrent.ts`
   即便已把 `userSettings` 迁到 `opus`，
   `model.ts`
   仍保留 legacy remap opt-out，
   说明 migration 已发生但 sunset 尚未 hard-close
4. `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days`
   说明 orphan plugin version 先进入 grace world，再在期限后删除
5. `orphanedPluginFilter.ts`
   与
   `main.tsx`
   说明旧 plugin version 可以先从当前 session 的 visibility world 退役，再从 storage world 消失
6. `CURRENT_MIGRATION_VERSION`
   只能证明哪些迁移动作需要重跑，
   并不自动签出“旧世界已经正式退役”
7. stronger-request cleanup 当前虽然已开始显露 migration-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup law、旧 covered-family contract 与旧 receipt semantics 何时停止作为 current world 继续存在

因此这一章的最短结论是：

`stronger-request cleanup migration governor 最多能说旧世界怎样过渡；stronger-request cleanup sunset governor 才能说旧世界何时停止作为正式世界继续存在。`

再压成一句：

`会迁，不等于会退役。`

再压成更哲学一点的话：

`制度知道怎样带旧世界过桥，不等于制度知道何时封桥。`

## 4. 第一性原理：migration governance 回答“如何并存”，sunset governance 回答“何时终止并存”

从第一性原理看，
这两层治理处理的是两个不同时间面的主权问题。

`migration governor`

回答的是：

1. 旧值是 rewrite、remap，还是 dual-read
2. 哪些 source 可以改写，哪些必须保持原样
3. 兼容期间怎样提示用户
4. 哪些旧 artifact 要进入 grace window
5. 怎样保持幂等与 continuity

`sunset governor`

回答的则是：

1. 兼容窗口在哪个日期、哪次启动或哪类 provider 后结束
2. 旧值何时不再被 runtime 接受
3. 旧 artifact 何时不再被搜索、引用或恢复
4. 旧 warning 何时从“建议切换”进入“正式不可再用”
5. 双世界共存何时必须切成单世界真相

所以这两层的哲学本质不同：

`migration`

是在管理共存；

`sunset`

则是在管理共存的终止。

Claude Code 在这方面真正先进的地方，
不是它“会在旧世界不对时马上带去新世界”，
而是它已经明确承认：

`旧世界何时不再算当前世界，本身也是正式安全问题。`

这是一种很成熟的安全哲学：

`更正当前真相，不等于有权立即宣布旧真相已经作废。`

因为一旦把 migration 与 sunset 压成一层，
系统就容易出现三种危险：

1. compatibility path 被无限拖延，旧世界永不正式结束
2. migration 刚发生就被误写成 sunset 完成，旧世界被过早切断
3. warning、visibility 与 storage 三个层面的退役节奏失去秩序，系统开始在不同 surface 上说不同的话

## 5. 从技术角度看，Claude Code 的先进性在于把 sunset 做成多张时钟，而不是把退役压成一次性删除

很多系统一谈退役，
就只剩一种动作：

`删旧东西。`

Claude Code 更成熟的地方在于，
它已经把 sunset 做成至少四种相互配合但互不偷换的技术语法。

### 5.1 provider-specific retirement clock：同一个旧模型，不同 provider 可以有不同 sunset date

`deprecation.ts:21-25,33-61,66-100`
最强的地方，
不是它会提示旧模型 deprecated，
而是它把：

`retirement truth`

写成了 provider-sensitive data model。

也就是说，
repo 不假装 sunset 是一个全局同时刻事件。

这很先进，
因为真实系统里不同 provider 的 rollout、contract 与可用窗口本来就不同。

安全设计启示是：

`退役不是抽象价值判断，而是与具体运行世界绑定的时钟制度。`

### 5.2 migration clock != warning clock：一次性迁移通知与持续退役警告被明确拆开

`useModelMigrationNotifications.tsx:5-8,21-33,49-50`
只在 timestamp 刚写入后的极短窗口内弹一次：

`Model updated to ...`

而
`useDeprecationWarningNotification.tsx:6-27,29-42`
与
`main.tsx:2872-2895`
则把 deprecation warning 做成另一个 surface。

前者回答：

`这次启动发生了迁移动作`

后者回答：

`你当前仍在使用一个已经进入退役倒计时的世界`

这说明 Claude Code 不是把 transition event 偷写成 retirement fact，
而是承认：

`发生过迁移`

和

`旧世界已经进入终止倒计时`

是两件事。

### 5.3 compatibility still alive：migration 已发生时，旧世界仍可能继续被 runtime remap

`migrateLegacyOpusToCurrent.ts:13-27,29-57`
已经做了三件迁移动作：

1. 改 `userSettings`
2. 写 `legacyOpusMigrationTimestamp`
3. 让 REPL 有机会展示 one-time notification

但
`model.ts:549-554`
仍保留：

`CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP`

这等于公开承认：

1. migration 已发生
2. compatibility 仍在继续
3. sunset 还没有 hard-close

这正是比“会迁”更强的一层：

`系统是否知道兼容何时结束`

### 5.4 storage sunset != visibility sunset：plugin orphan 先失去当前会话可见性，再晚一些失去磁盘存在

`cacheUtils.ts:23-24,66-74,89-105,149-176`
说明 orphan plugin version 的生命周期不是：

`new version appears -> old version deleted immediately`

而是：

1. old version becomes orphan
2. `.orphaned_at` marker 存在
3. 进入 7 天 grace world
4. 到点后再删

更进一步，
`orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
与
`main.tsx:2544-2568`
又把 grep/glob exclusion cache 做成：

1. orphan GC 先整理 marker state
2. 再 warm exclusions
3. 当前 session 先不再把 orphan world 当成 search current truth
4. cache 在 session 级冻结，直到显式 `/reload-plugins`

这极其先进，
因为它展示出：

`是不是当前世界`

与

`是不是还留在磁盘上`

本来就不是同一个问题。

### 5.5 migration registry != sunset registry：知道哪些迁移动作要跑，不等于知道兼容什么时候结束

`main.tsx:323-345`
的 `CURRENT_MIGRATION_VERSION`
说明 repo 已把 migration 做成正式 startup registry。

但它没有宣称：

`只要 version gate 已推进，旧世界就必然已经退役。`

这说明 Claude Code 识别出：

1. migration registry 负责重放过渡动作
2. sunset registry 则应负责宣布哪些兼容世界已经不再有效

这是一条很深的技术启示：

`制度知道怎样重放变更，不等于制度知道何时取消旧制度的合法性。`

## 6. 对 stronger-request cleanup 而言，当前真正缺的不是“要不要迁移”，而是“旧 stronger-request 世界何时停止作为当前世界”

当前 stronger-request cleanup 已经开始显露 migration-governance，
但仍缺少 sunset 治理回答：

1. 旧 startup-delete wording 何时不再能被继续当作 current promise
2. 旧 home-root `plans` cleanup law 何时从兼容世界正式退出
3. 旧 `CleanupResult` / local count semantics 何时不再被允许充当 current receipt truth
4. diagnostics、history、tool-result 这些 uncovered / differently-covered worlds 何时从“暂时例外”进入“正式另立制度”或“正式纳入新 covenant”
5. 旧 covered-family contract 是什么时候从“过渡中保留”变成“再引用即 overclaim”

这些都不是“怎样带过去”能回答的。

它们已经进入：

`sunset governance`

层。

也就是说，
stronger-request cleanup 今天真正缺的，
不是再补一条 migration grammar，
而是：

`谁来正式宣布兼容 grammar 何时终止。`

## 7. 苏格拉底式自反诘问：我是不是又把“已经会迁”误认成了“已经会退役”

如果对这一层做更严格的自我审查，
至少要连续追问六句：

1. 如果 migration object 已经定义，为什么还要再拆 sunset？
   因为“怎样带过去”不等于“何时结束带着走”。
2. 如果 migration timestamp 已经写进 config，为什么还不算退役完成？
   因为 timestamp 只能证明动作发生过，不能证明兼容世界已经失效。
3. 如果 warning 已经在 UI 上出现，为什么还说 sunset 未完成？
   因为 warning surface 只是提醒，不自动等于 hard-close authority。
4. 如果 plugin orphan 已经从 grep / glob 看不见了，为什么还不算一切结束？
   因为 visibility sunset 不等于 storage sunset，更不等于制度上所有引用都已失效。
5. 如果 model line 已有 provider-specific retirement date，cleanup 线是不是直接照抄就行？
   不能照抄。时钟形式可以借鉴，但 cleanup 线必须自己回答哪类 carrier、哪类 promise、哪类 receipt 要被哪个 cutoff 裁决。
6. 如果 cleanup 线当前还没有显式 sunset code，是不是说明这层还不值得写？
   恰恰相反。越缺正式 sunset grammar，越容易在未来把兼容世界无限拖延，或者把旧世界过早切断。

这一串反问最后逼出一句更稳的判断：

`sunset 的关键，不在系统会不会删旧东西，而在系统能不能把“旧世界何时不再算当前世界”治理成一个不说两套话的制度。`

## 8. 从安全设计哲学看，这一层真正回答的是“制度有没有终止例外世界的主权”

`migration`
更像制度的交接能力。

`sunset`
则更像制度终止例外世界的主权。

一个系统可以知道：

`我现在怎样让旧世界和新世界先并存。`

但如果它还不能明确回答：

`这段并存从什么时候起不再合法、哪些兼容路径只是临时让步而不是永久权利、哪些旧 receipt/old wording 从什么时候起再被引用就构成 overclaim。`

那么这套制度仍然更接近：

`会过渡的制度`

而不是：

`会终止例外世界的制度`

Claude Code 当前 stronger-request cleanup 线，
恰好就停在这个哲学分界点上：

1. 它已经不再只是不会自述、不会兑现、不会自检、不会纠偏的 cleanup world
2. 它已经开始学会把旧世界带入受治理的 transition grammar
3. 但它还没有 equally-strong 地展示旧 stronger-request 世界何时被正式取消 current status

所以这一章真正要钉死的，
不仅是一个工程判断，
也是一个更深的安全设计判断：

`制度知道怎样带旧世界过桥，不等于制度已经知道何时封桥。`

## 9. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

因为：

1. migration 负责共存设计
2. sunset 负责共存终止
3. provider-specific retirement date、one-time migration notice、persistent deprecation warning、visibility cutoff 与 delayed deletion 都证明 repo 已把这两层正式拆开
4. stronger-request cleanup 当前虽已开始理解如何把旧世界带过渡，却还没有对等地回答旧 world 何时正式不再算 current truth

因此，
这一章最终必须压成一句：

`会迁，不等于会退役。`
