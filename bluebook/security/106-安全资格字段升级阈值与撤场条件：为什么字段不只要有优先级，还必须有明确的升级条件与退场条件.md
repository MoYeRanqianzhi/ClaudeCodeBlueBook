# 安全资格字段升级阈值与撤场条件：为什么字段不只要有优先级，还必须有明确的升级条件与退场条件

## 1. 为什么在 `105` 之后还必须继续写“升级阈值与撤场条件”

`105-安全资格字段优先级与抢占规则` 已经回答了：

`同一个 surface 上的多个 signal 不能平权竞争，强断点必须抢占弱信号。`

但如果继续追问，  
还会碰到一个更深的问题：

`强信号到底在什么条件下才配升级为强信号？又在什么条件下必须退场？`

如果没有这层规则，  
仅靠优先级排序仍然不够。  
因为系统马上会落入另外两种失真：

1. 弱证据过早升级，制造假紧急态
2. 旧强信号退场过晚，制造过期恐慌态

也就是说：

`优先级只回答“谁先说”，并不回答“谁什么时候配说、什么时候必须闭嘴”。`

所以 `105` 之后必须继续补出的下一层就是：

`安全资格字段升级阈值与撤场条件`

也就是：

`字段治理不是静态排序，而是带证据门槛、保留条件与退场条件的生命周期协议。`

## 2. 最短结论

从源码看，Claude Code 已经在多个子系统里显式实现了这种“先过门槛、再占通道、再按条件退场”的控制逻辑：

1. notification 不是一来就永久在场，而是受 `priority`、`timeoutMs`、`invalidates` 与 `fold` 共同治理  
   `src/context/notifications.tsx:8-22,45-68,78-116,121-146,172-212,230-238`
2. bridge failed 不是普通红字，而是只有在明确进入 failed 分支或 init 异常时才升级为 `immediate`；随后再按 `BRIDGE_FAILURE_DISMISS_MS` 自动撤场并关闭持续重试  
   `src/hooks/useReplBridge.tsx:28-40,102-127,339-362,623-644`
3. plugin reload pending 不允许被局部提示层随手清掉，而必须等 `refreshActivePlugins()` 真正完成全量刷新后，才把 `needsRefresh` 置回 `false`  
   `src/hooks/useManagePlugins.ts:287-303`; `src/utils/plugins/refresh.ts:59-67,123-138`
4. crash-recovery pointer 也不是想留就留、想删就删，而是依赖 freshness、resume 结果、fatality 与 clean shutdown 语义差异化处理  
   `src/bridge/bridgePointer.ts:57-60,77-112`; `src/bridge/bridgeMain.ts:1515-1577,2316-2325,2384-2403,2473-2534`
5. MCP stale clients 的撤场也不是“一键全删”，而是先解除旧 timer、阻断旧 closure，再只对真正 `connected` 的 stale client 做 cleanup  
   `src/services/mcp/useManageMCPConnections.ts:791-825`

所以这一章的最短结论是：

`真正成熟的安全控制面，不只给字段排强弱，还给字段规定升级门槛、在场条件与退场条件。`

再压成一句：

`优先级决定谁赢，阈值决定谁配赢，撤场条件决定谁何时必须退出。`

## 3. notification 子系统已经把字段生命周期写成了协议

## 3.1 升级不是“被创建”就够了，而是要跨过 admission gate

`src/context/notifications.tsx:78-116` 里最重要的不是 `immediate` 更高，  
而是它一旦出现就会做三件事：

1. 清掉当前 timeout
2. 立即占据 `current`
3. 把被它 `invalidates` 的旧通知从当前与队列里驱逐

这意味着：

`能升级成当前解释通道的前提，不是通知存在，而是它满足了“足够强”这一 admission gate。`

反过来，  
`src/context/notifications.tsx:172-191` 又说明非 immediate 通知不会直接霸占当前通道，  
而是进入 queue，等待 `getNext()` 根据优先级出队。

也就是说：

1. immediate 走的是“越级直升”路径
2. non-immediate 走的是“入队等待仲裁”路径

这已经不是单纯 UI 效果，  
而是：

`字段升级路径本身已经被制度化。`

## 3.2 在场也不是永久在场，而是受保留条件约束

`src/context/notifications.tsx:45-68` 和 `131-146` 很关键：

1. 当前通知不是永久存活
2. 它只在 timeout 窗口内占据 current
3. 同 key 的 fold 会刷新当前通知的 timeout

这说明 notification 的“在场资格”至少绑定两类条件：

1. 计时还没耗尽
2. 同家族新信号没有把它折叠成新版本

换句话说：

`字段在场不是产权，而是租约。`

这和前文对绿色词 lease 的分析完全一致，  
只是现在我们把这条原理推进到了：

`所有强弱字段都应被当成有续期条件的解释租约。`

## 3.3 退场不是“用户看烦了”，而是满足了明确的 retire trigger

notification 的 retire trigger 至少有四种：

1. `timeoutMs` 到期  
   `src/context/notifications.tsx:52-68,88-104,131-146`
2. 更强新信号通过 `invalidates` 驱逐  
   `src/context/notifications.tsx:8-13,98-99,113,176-185`
3. 同 key 信号通过 `fold` 吸收旧版本  
   `src/context/notifications.tsx:16-22,121-169`
4. 显式 `removeNotification(key)`  
   `src/context/notifications.tsx:193-212`

这四类 trigger 共同说明：

`一个字段的撤场不该是任意消失，而必须对应一条可解释的退出因果。`

## 3.4 `invalidates` 证明“升级条件”和“撤场条件”是一对互补协议

`src/hooks/notifs/useFastModeNotification.tsx:101-121` 给了最干净的互补样本：

1. cooldown started invalidates expired
2. cooldown expired invalidates started

这里最值得注意的不是两条消息都很重要，  
而是：

`谁升级，谁就必须同时定义谁退场。`

这是一条很深的控制面公理：

`没有 retire target 的 upgrade，往往会制造双重真相并存。`

## 4. bridge 失败链展示了“升级阈值”和“撤场条件”如何绑定到安全成本

## 4.1 bridge failed 的升级门槛并不低

`src/hooks/useReplBridge.tsx:102-127` 与 `339-362` 表明：

1. bridge failed 只有在明确进入 failed 分支时才升级成 `immediate`
2. 连续初始化失败达到 `MAX_CONSECUTIVE_INIT_FAILURES = 3` 时，才进一步触发 fuse
3. fuse 触发后，不只是显示错误，还直接把 `replBridgeEnabled` 关掉

这说明 Claude Code 并没有把每次小抖动都升级成永久性大故障。  
它在区分：

1. 失败 signal 的第一次成立
2. 会话级“再重试已经不值”的熔断门槛

所以 bridge 这里的升级协议至少有两层：

1. `failed` 升级到前台断点
2. repeated failure 升级到会话级撤销

## 4.2 bridge failed 的保留条件也不是无限期

`src/hooks/useReplBridge.tsx:28-29` 明确给出 `BRIDGE_FAILURE_DISMISS_MS = 10_000`。  
而 `339-362`、`504-515`、`633-644` 又把这个窗口真正绑到 failed / init-failure 路径上。

也就是说，  
bridge failed 不会无限期赖在前台，  
它的在场条件是：

1. 当前失败结论仍成立
2. 自动撤场计时尚未结束
3. 尚未被更高层状态覆盖

这说明作者在这里并不追求“永远提醒”，  
而是追求：

`在高风险窗口内强力拦截，然后把控制面从持续抖动中解放出来。`

## 4.3 真正的退场动作不是隐藏文案，而是关闭错误世界的自我重试

`src/hooks/useReplBridge.tsx:350-362`、`504-515`、`633-644` 最重要的一点是：

错误提示计时结束后，  
系统做的不是单纯 `error toast disappear`，  
而是：

1. `replBridgeEnabled = false`
2. `replBridgeError = undefined`

这说明 bridge 失败的撤场，本质上不是 UI 清屏，  
而是：

`把旧失败世界从“持续重试中”正式退役。`

所以这里的哲学是：

`字段退场不是为了界面整洁，而是为了结束一段已经不值得继续维持的控制面策略。`

## 4.4 成功路径同样有自己的升级阈值

`src/hooks/useReplBridge.tsx:518-614` 也同样重要。  
只有在 `handle_0` 成功初始化后，系统才会：

1. `consecutiveFailuresRef.current = 0`
2. 写入 `replBridgeConnected`
3. 写入 `replBridgeSessionId/sessionUrl/environmentId`
4. 清除 `replBridgeError`

所以成功并不是“用户点了开启 remote control”就成立，  
而是必须等：

`真实 handle + 会话 URL + 边界 ID bundle` 到位

之后，  
前面的失败或 pending 叙事才配退场。

这再次说明：

`字段升级的本质不是用户意图变化，而是真实证据 bundle 过线。`

## 5. plugin 子系统说明“低优先级字段”同样必须受严肃的退场纪律约束

很多人会误以为：

`只有高风险红字才需要严格生命周期；低优先级提示随便清就行。`

源码恰恰反过来证明不是这样。

## 5.1 `plugin-reload-pending` 的升级阈值是“磁盘状态已变，但运行态尚未全量重建”

`src/hooks/useManagePlugins.ts:287-303` 里，  
当 `needsRefresh` 成立时，系统会发出：

`Plugins changed. Run /reload-plugins to activate.`

这说明它升级到前台 low-priority 提示的门槛不是：

1. 有人打开过 `/plugin`
2. 用户主观怀疑插件可能变了

而是：

`系统已经知道磁盘态和运行态发生了结构性分离。`

## 5.2 这条提示被显式禁止“假退场”

同一段注释写得非常直接：

1. `Do NOT auto-refresh`
2. `Do NOT reset needsRefresh`
3. `refreshActivePlugins()` 才是唯一消费者

这意味着：

`低优先级不等于低纪律。`

它只是说明这件事不必打断全局注意力，  
并不说明它可以在没有 verifier 的情况下自行消失。

## 5.3 真正的退场条件是 full swap 完成，而不是“用户点过某个按钮”

`src/utils/plugins/refresh.ts:59-67,123-138` 说明：

1. 先清 cache
2. 先重新 load plugins / commands / agents
3. 重新填充 MCP/LSP 相关状态
4. `pluginReconnectKey + 1`
5. 最后才 `needsRefresh: false`

这说明 plugin pending 提示的正式退场条件不是：

`用户表示我知道了`

而是：

`对应的运行态重建确实已经完成。`

这正是安全控制面的一个关键美德：

`哪怕是“维护性提醒”，退场权也必须绑定到真正完成闭环的动作。`

## 6. pointer 生命周期把“撤场条件”推进到了更深的安全语义

如果 notification 和 plugin 还容易被理解成 UI 层规则，  
那么 pointer 相关代码已经非常清楚地告诉我们：

`字段生命周期其实在治理边界连续性本身。`

## 6.1 pointer 的升级条件是 freshness 持续成立

`src/bridge/bridgePointer.ts:57-60` 表明 pointer 会被重复写入以刷新 mtime。  
这意味着它不是静态工件，  
而是：

`随时间续保的恢复资格声明。`

也就是说，  
pointer 之所以还能代表“可恢复边界”，  
不是因为文件还在，  
而是因为 freshness 仍然过线。

## 6.2 stale/invalid pointer 的退场条件是明确且自动的

`src/bridge/bridgePointer.ts:77-112` 说明：

1. schema invalid -> clear
2. `mtime > TTL` -> clear
3. clear 后不再反复提示用户 resume

这说明 pointer 的退场不是“下次有人顺手清理”，  
而是：

`一旦失去资格，立刻撤回它继续代表当前边界的发言权。`

## 6.3 fresh start、本地 resume failure 与 clean shutdown 采用不同撤场策略

`src/bridge/bridgeMain.ts:2316-2325` 表明：

1. 没有 `--continue` 时，历史 leftover pointer 会先被清掉

`src/bridge/bridgeMain.ts:2384-2403` 表明：

1. session not found -> clear pointer
2. no environment_id -> clear pointer

`src/bridge/bridgeMain.ts:1515-1577` 表明：

1. 若是可恢复的 single-session clean shutdown 且 `!fatalExit`
2. 就跳过 archive + deregister + clear
3. 保留 pointer 以支撑 `--continue`
4. 反之在 archive + deregister 后清掉 pointer

这三组逻辑合在一起说明：

`同一个字段的撤场条件，不是单点 if/else，而是依赖整个边界生命周期语义。`

## 6.4 transient failure 与 fatal failure 的差异，说明最先进的地方在“分辨何时不该退场”

`src/bridge/bridgeMain.ts:2473-2534` 非常关键：

1. reconnect 失败后，不是所有失败都清 pointer
2. 只有 `isFatal` 才 clear
3. transient failure 要保留 pointer，让下一次启动还能 retry

这里最先进的地方其实不是异常处理本身，  
而是：

`系统知道某些失败会削弱边界，但并未彻底撤销边界。`

所以它治理的不只是“错误”，  
而是：

`错误是否足以构成撤销当前资格的充分证据。`

这是典型的第一性原理思维：

`不要问发生没发生错误，而要问该错误是否已经满足资格撤回门槛。`

## 7. MCP stale cleanup 进一步证明：退场应只撤掉“旧世界”，不能误杀“新世界”

`src/services/mcp/useManageMCPConnections.ts:791-825` 给出一个很成熟的 stale retirement 样本：

1. 先清掉旧 reconnect timer，避免旧 config 的延迟动作回流
2. 只对真正 `connected` 的 stale client 做 cleanup
3. never-connected 的 `disabled/pending/failed` 不做同样 cleanup，避免为清理而误触发新连接
4. 新 config 下的 client 会重新以 `pending/disabled` 进入

这说明 stale retirement 的目标不是“全部删干净”，  
而是：

`只撤掉旧世界仍然持有的发言权和副作用通道。`

这与前文 pointer 的哲学完全一致：

`撤场不是清扫，而是精准回收旧边界仍握着的控制权。`

## 8. 从第一性原理看，字段生命周期至少包含四个问题

如果把前面的源码压成最小模型，  
每个安全字段都应回答四个问题：

## 8.1 它凭什么升级

也就是：

`upgrade threshold`

例如：

1. immediate notification arrival
2. explicit failed branch
3. `needsRefresh === true`
4. pointer freshness within TTL

## 8.2 它靠什么继续留场

也就是：

`stay condition`

例如：

1. timeout 未到
2. 未被 invalidates
3. 闭环尚未完成
4. freshness 仍有效
5. fatal revocation 尚未发生

## 8.3 它何时必须退场

也就是：

`retire trigger`

例如：

1. timeout expire
2. stronger contradictory signal
3. real completion
4. stale / invalid / fatal
5. clean shutdown with no more resumability obligation

## 8.4 谁拥有它的退场权

也就是：

`retire owner`

例如：

1. notification queue manager
2. refreshActivePlugins full swap path
3. bridge resume / shutdown controller
4. MCP stale cleanup owner

如果这四个问题答不出来，  
系统就几乎一定会出现：

1. 过早升级
2. 过晚退场
3. 错主语清理
4. 新旧真相并存

## 9. 这套设计的哲学本质：安全不是“会不会拦”，而是“会不会在正确时刻改口”

从更深层看，  
这章真正揭示的是：

`安全控制面的成熟度，不在于它能发多少告警，而在于它能否在正确时刻改变自己的说法。`

因为任何安全系统都至少会犯两种错误：

1. 太早说“坏了”
2. 太早说“好了”

而字段生命周期协议恰好就是在治理这两种“改口错误”。

所以这里的哲学本质可以压成三句：

1. 字段不是装饰，它们是在租用解释权
2. 升级不是视觉加重，而是解释权升级
3. 撤场不是视觉消失，而是解释权撤回

换句话说：

`安全控制面真正治理的不是颜色，而是解释权在时间轴上的合法流转。`

## 10. 技术先进性：Claude Code 真正先进的不是单个 guard，而是对“stale truth”的系统性处理

如果只从功能角度看，  
这些代码像是在管理：

1. 提示
2. reconnect
3. pointer
4. plugin reload

但从架构角度看，  
它们共同解决的其实是更难的问题：

`如何防止过期真相继续占据当前控制面。`

这比“检测坏动作”更高级，  
因为它要求系统具备：

1. freshness 概念
2. failure tier 概念
3. explicit invalidation 概念
4. conditional retirement 概念
5. owner-specific cleanup 概念

很多系统会做第一步，  
少数系统会做第二步，  
更少的系统会把后面三步串起来。

Claude Code 在这些细节上的先进性，  
恰恰体现在它知道：

`stale truth 本身就是一种安全风险。`

## 11. 对下一代统一安全控制台的直接启示

如果要把这套设计再向前推进一代，  
我会建议把每个前台字段都显式协议化为：

1. `priority`
2. `upgrade_threshold`
3. `stay_condition`
4. `retire_trigger`
5. `retire_owner`
6. `expires_at` 或 `freshness_basis`
7. `invalidates`
8. `handoff_route`

这样控制台就不只知道：

`现在该显示什么`

还知道：

`为什么现在能显示它、为什么还没撤、以及谁有资格把它撤掉。`

这会把当前散落在各子系统里的成熟直觉，  
升级成真正统一的字段生命周期协议。

## 12. 苏格拉底式反思：如果要把这一层做得更好，还该继续追问什么

可以继续追问六个问题：

1. 当前每个字段的 `upgrade_threshold` 是否都已显式编码，还是仍有不少散落在注释和局部 if 里
2. 当前每个字段的 `retire_trigger` 是否可被别的 surface 复用，还是仍被困在局部组件生命周期里
3. 当前是否存在“会显示但说不清为何出现”的字段
4. 当前是否存在“会消失但说不清谁让它消失”的字段
5. 当前是否存在“弱证据可以错误升级、强证据不能及时撤场”的非对称漏洞
6. 当前是否应该把 freshness、fatality、completion、staleness 统一抽象成控制面通用 retirement semantics

这些追问的重要性在于：

`一套控制面只要还不能自我解释字段为何出现与为何退场，它就还没有真正完成可治理化。`

## 13. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正成熟的不是：

`给字段排个优先级，`

而是：

`让每个字段都只能在足够证据下升级、在真实条件下留场、并在正确触发器出现后被正确的主语撤场。`

所以 `105` 之后真正必须加上的那句话就是：

`字段治理不是排序问题，而是生命周期问题。`
