# 安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层：为什么artifact-family cleanup stronger-request cleanup-reassurance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer

## 1. 为什么在 `399` 之后还必须继续写 `400`

`399-安全载体家族强请求清理重新投影治理与强请求清理重新担保治理分层`
已经回答了：

`stronger-request cleanup 线即便已经把 current truth 重新讲给不同 surface，也还要单独回答这些重讲里哪些现在配承载多强的正向 reassurance。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然某个 surface 已经给出了足够正向的 reassurance，真正 consumer 在实际使用那一刻当然可以直接依赖”
的错觉：

`只要 stronger-request cleanup reassurance governor 已经允许某个 surface 给出足够正向的 reassurance，真正的 consumer 在实际使用这一刻就自动拥有了继续依赖这份 truth 的资格，不再需要 fresh re-check。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的用时重验证正对照，
会发现 repo 其实明确拆开了五件事：

1. `resource read hard gate`
2. `batch list fresh reconnect`
3. `fresh proof primitive`
4. `execution-context current filter`
5. `runtime revocation`

最硬的证据至少有五组：

1. `tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:80-100` 会在真正读取资源前，既要求 `client.type === 'connected'`，又继续调用 `ensureConnectedClient(client)`
2. `tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:79-92` 明确把 `ensureConnectedClient` 注释成 healthy 时 no-op、onclose 后 fresh reconnect
3. `services/mcp/client.ts:1688-1705` 的 `ensureConnectedClient()` 会 fresh `connectToServer()`，若结果不再是 `connected` 就直接抛出 not connected
4. `services/tools/toolExecution.ts:308-335` 的 `getMcpServerType()` / `getMcpServerBaseUrlFromToolName()` 只有在 current connection 仍是 `connected` 时才继续派生 execution context
5. `services/tools/toolExecution.ts:1599-1624` 在 `McpAuthError` 上会把原本 `connected` 的 client 即时降成 `needs-auth`

这说明：

`artifact-family cleanup stronger-request cleanup-reassurance-governor signer`

和

`artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer`

仍然不是一回事。

前者最多能说：

`当前有某个 surface 配给出多强的正向 reassurance。`

后者才配说：

`真正的 consumer 在实际调用这一刻，已经重新完成 fresh connection check / current capability check / current authorization check，可以把这份 reassurance 转成可依赖的 current-use truth。`

所以 `399` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层`

也就是：

`stronger-request cleanup reassurance governor 决定正向话术强度；stronger-request cleanup use-time revalidation governor 才决定真正使用时是否已经 fresh enough to rely。`

## 2. 先做三条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 live gate manager，
而是在说：

1. repo 已经明确把 positive surface reassurance 与 real consumer use path 拆开
2. repo 已经明确要求 consumer 在真正依赖前重新检查 `connected` / current truth
3. repo 已经明确允许 runtime 把旧的 `connected` 立即降回 `needs-auth`

第二条：

这里的 `use-time revalidation`

不是更早那条反漂移验证链里的：

`anti-drift verification`

当前更稳的说法只能是：

`反漂移验证回答未来若再分叉，谁会主动发现；用时重验证回答真正 consumer 在这次依赖发生的瞬间，谁会要求重新验真。`

第三条：

这里也不能过度声称：

`use-time revalidation governance`

已经等于：

`step-up reauthorization governance`

当前更稳的说法只能是：

`use-time revalidation 先回答“现在这份 reassurance 还配不配被当真”；step-up reauthorization 才回答“即便 current-use truth 成立，它对更强请求是否仍不够强”。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-reassurance-governor signer 仍不等于 stronger-request cleanup-use-time revalidation-governor signer`

证据：

1. `ReadMcpResourceTool` 在真正读取资源前，既要求 `client.type === 'connected'`，又继续调用 `ensureConnectedClient(client)`；这说明 positive reassurance 不自动等于 read path can rely now
2. `ListMcpResourcesTool` 明确把 `ensureConnectedClient` 注释成 healthy 时 no-op、onclose 后 fresh reconnect；这说明 list path 也拒绝盲信旧 reassurance
3. `ensureConnectedClient()` 本身会 fresh `connectToServer()`，若结果不再是 `connected` 就直接抛出 `MCP server "<name>" is not connected`；这说明真正依赖前仍要重新验真
4. `toolExecution.ts` 的 `getMcpServerType()` / `getMcpServerBaseUrlFromToolName()` 只有在 current connection 仍是 `connected` 时才继续派生 execution context；这说明连辅助执行上下文也不盲信旧 reassurance
5. `toolExecution.ts` 在 `McpAuthError` 上会把原本 `connected` 的 client 即时降成 `needs-auth`；这说明 stale reassurance 会在真实使用的第一时间被撤销

因此这一章的最短结论是：

`stronger-request cleanup reassurance governor 最多能说当前有某个 surface 配给出多强的正向 reassurance；stronger-request cleanup use-time revalidation governor 才能说真正 consumer 在使用瞬间是否已经重新拿到 fresh enough 的 current-use proof。`

再压成一句：

`被正向担保，不等于已经通过 live-use 重验证。`

再压成更哲学一点的话：

`制度知道现在可以说多放心，不等于制度已经知道真正依赖这一刻这份放心话仍然值得被相信。`

## 4. 第一性原理：reassurance 回答“能放心到什么程度”，use-time revalidation 回答“真正依赖这一刻还能不能把刚才那句放心话继续当真”

从第一性原理看，
强请求清理重新担保治理与强请求清理用时重验证治理处理的是两个不同主权问题。

`stronger-request cleanup reassurance governor`
回答的是：

1. 哪个 surface 现在配说正向话术
2. 这句正向话术允许多强
3. 是否必须保留 caveat
4. 是否宁可沉默也不发布正向信息
5. 当前 surface 的 reassurance ceiling 是什么

`stronger-request cleanup use-time revalidation governor`
回答的则是：

1. 真正 consumer 在调用前是否还要 fresh reconnect
2. 当前 `connected` truth 是否要在 use-time 重新检查
3. capability / resources / transport 是否仍然 current
4. 如果刚才的 reassurance 已经过时，谁会在使用瞬间把它撤销
5. 哪些 positive surface 只是 local confidence，哪些能够通过 live gate 转成 actual reliance

如果把这两层压成一句：

`既然刚才已经说能用了`

系统就会制造六类危险幻觉：

1. `reassurance-means-usable-now`
2. `dialog-success-means-resource-readable`
3. `connected-copy-means-fresh-proof`
4. `positive-surface-speaks-for-consumer`
5. `reassurance-once-means-revalidation-forever`
6. `surface-confidence-equals-consumer-authorization`

所以从第一性原理看：

`reassurance governance` 管的是正向信号强度；
`use-time revalidation governance` 管的是实际依赖发生前，旧 reassurance 是否仍配被当真。

## 5. Claude Code 多重安全技术的先进性：它把“正向话术”与“实际依赖资格”强行拆开

从技术角度看，
Claude Code 在这里最先进的地方，不是它“有 hard gate”，
而是它把多重安全技术写成了一条从语言到依赖资格的逐层收口链：

1. `positive reassurance is cheap, live-use proof is expensive`
   正向话术可以存在，但真正使用资格仍要单独拿 fresh proof
2. `fresh proof primitive over stale label reuse`
   用 `ensureConnectedClient()` 明确拒绝沿用旧 `connected` 标签
3. `execution metadata also obeys current-truth discipline`
   连 transport type/base-url 这种辅助信息都不搭 stale 顺风车
4. `live contradiction revokes old reassurance immediately`
   真实使用一旦撞上 `McpAuthError`，系统立刻把旧 reassurance 降回 `needs-auth`
5. `consumer edge stays sovereign`
   read/list/execution 这些真实消费边缘不接受任何上游 surface 越级代签
6. `runtime revocation is part of verification`
   revalidation 不只是入场前再查一次，还包括使用中即时撤销旧 truth

这些设计背后的源码思路也很清楚：

1. 不让 `reassured truth` 越级冒充 `currently consumable truth`
2. 不让某个 positive surface 抢夺真实 consumer edge 的主权
3. 不让旧 `connected` 标签继续污染当前依赖资格
4. 不让 runtime contradiction 被解释成局部例外而不是制度性撤销
5. 不让一次 reassuring story 变成永久通行证

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. reassurance 之外，还必须单列 fresh-at-use proof grammar。
2. 真正 consumer edge 应该拥有自己的 hard gate，而不是只消费上游 positive story。
3. fresh reconnect / fresh check 必须有 canonical primitive，不能分散成暗含假设。
4. 执行辅助上下文也应服从 current-truth discipline。
5. runtime contradiction 出现时，旧 reassurance 应被即时撤销。
6. “能说多放心”与“能不能真正用”必须是两层不同 signer。

## 6. 苏格拉底式自反诘问：我是不是又把“它刚才被说成可用”误认成了“它现在真的还能被安全地使用”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 `MCPReconnect` 已经说 `Successfully reconnected`，为什么 `ReadMcpResourceTool` 还要再次 `ensureConnectedClient(client)`？
   因为 operation-local reassurance 不拥有替真实 read path 免除 fresh proof 的资格。
2. 如果 `/mcp` 已经显示 `✓ Connected`，为什么 `toolExecution` 仍会在 `McpAuthError` 上立刻把它降回 `needs-auth`？
   因为 health reassurance 不是 immutable use authorization。
3. 如果 `McpAuthTool` 刚说 `should now be available`，为什么 list/read tool 仍会在真正调用时拒绝 non-connected client？
   因为 positive wording 只配描述 current expectation，不配越级免除 consumer-side live gate。
4. 如果旧 `connected` 标签已经足够，为什么 `ensureConnectedClient()` 还要 fresh reconnect-or-throw？
   因为 stale label 不等于 current-use truth。
5. 如果 reassurance 真能一路沿用，为什么 execution metadata 也坚持 connected-only filter？
   因为连外围上下文都不愿意沿用旧 reassurance 的残影。
6. 如果 cleanup 线还没正式长出 use-time revalidation grammar，是不是说明这层只是实现细节？
   恰恰相反。越是把它当细节，越容易让 surface-level comfort 冒充 live-use authorization。

这一串反问最终逼出一句更稳的判断：

`use-time revalidation 的关键，不在某个 surface 刚才说得多积极，而在真正依赖发生时系统能不能再次裁定那句积极话是否仍配被当真。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 reassurance grammar，就已经足够成熟。`

而是：

`Claude Code 在 ReadMcpResourceTool 与 ListMcpResourcesTool 的 consumer-edge hard gate、ensureConnectedClient() 的 fresh reconnect primitive，以及 toolExecution.ts 的 current-context filter 与 McpAuthError demotion 上，已经明确展示了 use-time revalidation governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reassurance-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-use-time revalidation-governor signer。`

再压成一句：

`stronger-request cleanup reassurance governance 能回答“现在可以说多放心”；stronger-request cleanup use-time revalidation governance 才能回答“真正消费这一刻，这份放心话是否仍配被当真”。`
