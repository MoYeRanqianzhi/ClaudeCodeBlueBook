# 研究方法与可信边界

## 1. 这部分在研究什么

本专题研究的是 Claude Code 的安全控制面。

重点不是简单判断它“安不安全”，  
而是回答四个更可操作的问题：

1. 它把哪些能力视为高风险能力
2. 这些能力在什么边界内被允许、询问、拒绝或降级
3. 哪些安全结论可以仅凭公开源码稳定成立
4. 哪些看起来像“安全特性”的东西，其实只是 ant-only / feature-gated / internal 痕迹

## 2. 研究方法

本专题采用四步法。

### 2.1 先按安全层分组，而不是按功能菜单分组

优先研究五层：

1. 启动与信任层
2. 权限与沙箱层
3. 认证、密钥与受管环境层
4. 外部能力与扩展入口层
5. 远程资格、组织策略与宿主边界层

这样更容易看清一件事：  
Claude Code 的安全不是一个功能点，而是一串前后相接的边界。

### 2.2 再用用户可见表象回看

重点观察这些表象：

- 为什么某些工具会 ask / deny
- 为什么 `--dangerously-skip-permissions` 不是随时都能开
- 为什么某些 `ANTHROPIC_BASE_URL` 场景能推理但不等于拥有第一方高阶资格
- 为什么 hooks、MCP、WebFetch 都有额外的收口逻辑
- 为什么 project-scoped settings 在 trust 前不能任意注入危险 env

只有把表象和安全层对应起来，  
才不会把所有安全现象都误读成“一个总开关”。

### 2.3 强结论必须由多模块共同支撑

例如：

- 权限系统结论，要同时看到 `permissionSetup.ts`、`filesystem.ts`、`PermissionMode.ts`
- 受管环境结论，要同时看到 `managedEnv.ts`、`managedEnvConstants.ts`、`setup.ts`
- 外部能力收口结论，要同时看到 `toolSearch.ts`、`mcpValidation.ts`、`WebFetchTool/preapproved.ts`
- 认证与高阶资格结论，要同时看到 `auth.ts`、`bridgeEnabled.ts`、secure storage 实现

### 2.4 证据用于压实结论，不用于复制源码

本专题仍然遵循蓝皮书总约束：

- 写模块、边界、状态迁移和设计结论
- 不直接搬运大段源码
- 遇到 public build 不可见或 ant-only stub 时，必须标出可信边界

## 3. 当前可信边界

### 3.1 可以稳定确认

- Claude Code 的安全性是分层控制面，不是单点沙箱。
- 权限模式、危险规则剥离、受管环境过滤、外部能力收口和远程资格门槛都是真实存在的源码结构。
- project-scoped 配置并不能在 trust 前自由接管危险 env。
- Remote Control 一类高阶能力并不等于“任意兼容入口都能等价替代”。

### 3.2 需要保守表述

- classifier 在公开提取树里存在明显 ant-only / external build stub 边界，不能把内部分类能力细节写成所有外部构建的稳定事实。
- GrowthBook、Statsig 与部分 gate 的具体线上值不可从源码静态推出。
- 不能仅凭客户端代码断言服务端的全部安全阈值或审计策略。

## 4. 当前最重要的结论

到目前为止，最稳妥的一句话结论是：

Claude Code 的安全设计不是“先默认全开，再用提示词补救”，而是把信任、权限、配置、认证、外部工具和远程能力都前置成运行时边界。

## 5. 本专题的使用方式

- 想先看方法论和证据纪律，从本章开始。
- 想先建立整体安全判断，从 `01` 顺读。
- 想追更细的源码入口，再回到 `docs/development/research-log.md`。
