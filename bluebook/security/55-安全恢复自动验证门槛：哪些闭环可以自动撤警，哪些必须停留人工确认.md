# 安全恢复自动验证门槛：哪些闭环可以自动撤警，哪些必须停留人工确认

## 1. 为什么在 `54` 之后还要继续写“自动验证门槛”

`54-安全恢复验证闭环` 已经回答了：

`repair action 只能产生修复意图，真正恢复必须等待 verifier readback 和 signer 关环。`

但如果继续往下一层追问，  
还会出现一个更关键的治理问题：

`既然都要闭环，那么哪些闭环可以由系统自动完成，哪些闭环必须停在人工确认态？`

因为控制面不可能对所有修复都一视同仁。  
有些路径系统自己就掌握：

1. 动作执行权
2. 读回权
3. signer 权

而另一些路径天然跨出了系统主权范围：

1. 用户重新登录
2. 外部应用重启
3. 宿主或组织策略变化
4. 需要人工再次观察或确认

如果不把这条边界正式写出来，  
系统就会在自动化上犯两个相反的错误：

1. 自动化过度：把本不该自动清的告警提前撤掉
2. 自动化不足：明明系统可以自己完成闭环，却还把责任丢给用户

所以在 `54` 之后，  
统一安全控制台还必须再补一条制度：

`恢复自动验证门槛。`

也就是：

`只有当系统拥有足够的主权去执行动作、读回结果并拿到 signer 时，才配自动撤警；一旦跨出这条主权边界，就必须停留在人工确认。`

## 2. 最短结论

从源码看，Claude Code 已经明显区分了两大类恢复闭环：

1. 系统可自证闭环  
   例如：
   - 新 marketplace 安装后的 plugin auto-refresh  
     `src/services/plugins/PluginInstallationManager.ts:136-165`
   - 远程 MCP transport automatic reconnection with backoff  
     `src/services/mcp/useManageMCPConnections.ts:354-464`
2. 系统不可自证闭环  
   例如：
   - plugin updates 只提示 `/reload-plugins`，不自动应用  
     `src/services/plugins/PluginInstallationManager.ts:166-177`、`src/hooks/useManagePlugins.ts:288-303`
   - `/login` 之后仍要求 `then restart` 或 `/status` 复核  
     `src/components/LogoV2/ChannelsNotice.tsx:73`、`src/utils/teleport.tsx:441`
   - bridge reconnect 失败时保留 pointer 并要求用户再次尝试  
     `src/bridge/bridgeMain.ts:2521-2539`

所以这一章的最短结论是：

`自动撤警的真正门槛不是“有没有修复命令”，而是“系统是否同时拥有动作执行权、读回权和关闭解释的 signer 权”。`

我会把这条结论再压成一句话：

`自动化只能走到系统主权的边界，不能越过边界替人类和外部世界签字。`

## 3. 源码已经在多个地方做了“自动 vs 人工”分流

## 3.1 plugin 新安装可以 auto-refresh，因为系统掌握全链主权

`src/services/plugins/PluginInstallationManager.ts:136-165` 很关键。  
这里对：

`New marketplaces were installed`

采取的策略是：

1. 清 marketplace cache
2. 调 `refreshActivePlugins(setAppState)`
3. 若成功，直接完成刷新闭环
4. 若失败，再 fallback 到 `needsRefresh`

而 `src/utils/plugins/refresh.ts:60-138` 又说明：

1. 它能清所有 plugin caches
2. 重新装载 plugin state
3. 重新拉 commands / agents
4. 递增 `pluginReconnectKey`
5. 清掉 `needsRefresh`

这说明在这个场景里，  
系统其实具备了完整主权：

1. 可以自己执行 repair action
2. 可以自己做 verifier readback
3. 可以自己拿到状态回写 signer

因此：

`允许自动撤警`

是合理的。

## 3.2 plugin 更新不 auto-refresh，因为系统明确承认这里会越过安全门槛

同一个文件在 `src/services/plugins/PluginInstallationManager.ts:166-177` 对：

`Existing marketplaces updated`

却采取了相反策略：

1. 不自动 refresh
2. 只设置 `needsRefresh`

`src/hooks/useManagePlugins.ts:288-303` 也再次强调：

1. 之前的 auto-refresh 有 stale-cache bug
2. 还不完整
3. 现在必须由用户显式运行 `/reload-plugins`

这里的设计非常成熟。  
因为作者并不是“不会自动化”，  
而是已经明确承认：

`这个场景下，自动化闭环的可信门槛不够。`

也就是说，  
虽然表面上 plugin 安装和 plugin 更新都叫“插件变了”，  
但控制面已经在底层区分出：

1. 一类可自动闭环
2. 一类必须停在人工确认

## 3.3 MCP remote transport 允许 automatic reconnection，因为系统拥有 transport 级闭环

`src/services/mcp/useManageMCPConnections.ts:354-464` 展示了另一种成熟自动化：

1. 只对 remote transports 自动重连
2. 排除 `stdio` 和 `sdk`
3. 使用 exponential backoff
4. 最多 `MAX_RECONNECT_ATTEMPTS`
5. 成功则 `onConnectionAttempt(result)` 关环
6. 失败到上限再给出最终状态

这说明这里的自动化之所以合理，  
是因为系统掌握了：

1. transport 生命周期
2. 重连逻辑
3. verifier readback
4. 最终状态落盘

也就是说，  
对这类 transport 级闭环，  
系统具备完整主权，  
所以：

`自动验证是合理的。`

## 3.4 `/login` 之后要求 restart，说明系统明确知道自己失去自动验证主权

`src/components/LogoV2/ChannelsNotice.tsx:73` 直接说：

`run /login, then restart`

`src/utils/teleport.tsx:441` 又说：

`Please run /login to authenticate, or check your authentication status with /status.`

这里已经非常清楚地显示出：

1. `/login` 不是自动闭环
2. 之后仍需 restart 或复核
3. 控制面不会替用户把这一步签掉

这意味着在 auth / restart 这类路径上，  
系统已经天然承认：

`自动验证门槛未达标。`

因为一旦涉及：

1. 进程外状态
2. 新 token 真正生效
3. restart 后新进程重新读配置

控制面就已经越出了自己单轮主权的范围。

## 3.5 bridge reconnect 失败保留 pointer 并提示重试，本质上也是人工确认态

`src/bridge/bridgeMain.ts:2521-2539` 写得很清楚：

1. transient reconnect failure 不清 pointer
2. fatal 才清 pointer
3. transient 时直接提示：

`The session may still be resumable — try running the same command again.`

这说明 bridge 域也不是“只要有 reconnect API 就一定自动闭环”。  
源码已经明确把它分成：

1. 系统能继续自动推进的部分
2. 必须由用户再次确认和触发的部分

换句话说，  
一旦进入 transient uncertainty，  
控制面就会主动退回：

`manual-confirmation state`

## 4. 第一性原理：自动验证能力的本质是控制面主权范围

如果从第一性原理追问：

`为什么有些恢复路径能自动关环，有些却不能？`

答案不是“实现难度”，  
而是：

`主权范围。`

只有当系统同时拥有下面三种权力时，  
它才配自动撤警：

1. action ownership  
   它自己能发起修复动作
2. verification ownership  
   它自己能读回新事实
3. interpretation ownership  
   它自己能拿到对应 signer 并更新主结论

只要缺其中任意一项，  
系统就不该自动宣布恢复。

这条原则可以压成一句话：

`自动验证不是便利功能，而是控制面主权的副产物。`

## 5. 我给统一安全控制台的自动验证门槛

我会把它分成三档。

## 5.1 A 档：全自动闭环

适用条件：

1. repair action 由系统发起
2. verifier 由系统直接读回
3. signer 由系统直接拿到

典型例子：

1. remote MCP transport auto reconnect
2. 某些新安装 plugin 的 auto-refresh

这种情况下，  
允许：

1. 自动更新卡片
2. 自动撤掉对应告警
3. 必要时自动恢复部分动作

## 5.2 B 档：半自动闭环

适用条件：

1. repair action 可以由系统辅助发起
2. 但 verifier 或 signer 有一部分仍需用户协作

典型例子：

1. `/reload-plugins`
2. 某些 bridge reconnect retry

这种情况下，  
系统可以：

1. 记录动作已执行
2. 显示正在等待哪类 readback
3. 暂不撤掉高层告警

## 5.3 C 档：人工确认闭环

适用条件：

1. 修复必须跨出当前进程
2. 或必须依赖用户、外部应用、组织策略、系统设置的再次确认

典型例子：

1. `/login` 之后的 restart
2. 外部环境中某些权限授权
3. bridge reconnect 的手工重试

这种情况下，  
控制面最多只能说：

`repair step started / waiting for manual confirmation`

绝不能自动清掉核心告警。

## 6. 我给自动验证闭环的最低语法

我会把统一安全控制台里的自动验证语法压成：

`repair path -> verifier ownership -> signer ownership -> auto/manual gate`

例如：

1. 如果三者都在系统手里  
   -> `auto-close allowed`
2. 如果 verifier 或 signer 不全在系统手里  
   -> `manual confirmation required`

这样一来，  
自动验证不再是拍脑袋规则，  
而是：

`主权检查结果。`

## 7. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只是做了很多自动恢复，而是已经在多个地方表现出“只在拥有足够主权时才自动闭环”的克制。`

这对其他 Agent 平台构建者的直接启示有四条：

1. 自动化与否应由 ownership matrix 决定，不应由用户焦虑决定
2. 系统若缺 verifier，就不配 auto-close
3. 系统若缺 signer，就不配 auto-clear
4. 人工确认态不是失败，而是对主权边界的诚实表达

## 8. 哲学本质

这一章更深层的哲学是：

`真正克制的系统，不会因为自己“看起来能帮一点忙”就越权替世界签字。`

很多系统的最大问题不是不会自动化，  
而是：

`自动化冲动超过了主权边界。`

Claude Code 现有源码里自动刷新、自动重连、手动 `/reload-plugins`、`/login` 后 restart、bridge retry 这些差异化策略，  
其实已经共同表达了一条很成熟的哲学：

`自动化应服务于真相，而不是服务于把告警尽快弄没。`

从第一性原理看，  
这保护的是下面这条公理：

`系统只能自动关闭自己真正拥有闭环主权的问题。`

## 9. 苏格拉底式反思：这一章最容易犯什么错

### 9.1 会不会把太多问题都归到人工确认，导致系统过于保守

会。  
所以关键不是一味保守，  
而是：

`把 ownership 说清楚。`

### 9.2 会不会反过来把所有 verifier 都做进系统里，造成复杂度爆炸

也会。  
所以应先优先收敛那些：

1. 风险高
2. 误清成本高
3. 已经有现成 readback 的路径

### 9.3 这一章之后还缺什么

还缺一张真正可执行的门槛矩阵：

`repair path -> ownership 完整度 -> auto-close allowed / manual-required`

也就是说，  
下一步最自然的延伸就是：

`appendix/39-安全恢复自动验证门槛速查表`

## 10. 结语

`54` 回答的是：修复命令不等于恢复闭环。  
这一章继续推进后的结论则是：

`并非所有闭环都该自动完成，只有系统真正拥有动作、读回和 signer 主权的路径，才配自动撤警。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟控制面不仅要会自动恢复，更要知道哪些恢复不该由自己自动宣布完成。`
