# 安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层：为什么artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer

## 1. 为什么在 `423` 之后还必须继续写 `424`

`423-安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`
已经回答了：

`stronger-request cleanup 线即便知道旧对象可以回来、也重新拿回了一部分 current seat / enabled truth，仍然还需要 re-entitlement governor 去决定它究竟恢复哪些旧资格。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“资格回来就说明配置也一起回来了”
的错觉：

`只要 stronger-request cleanup re-entitlement governor 已经让旧对象重新拿回 current seat，它就自动拥有了决定旧 options、旧 secrets、旧 payload、旧 channel config 是否仍可被当前系统继续承认的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 在 plugin 线已经成熟写出的 reconfiguration 正对照，
会发现它其实明确拆开了六件事：

1. `seat truth`
2. `config source truth`
3. `config closure truth`
4. `save truth`
5. `apply truth`
6. `work-ready truth`

最硬的证据至少有八组：

1. `src/utils/plugins/pluginOptionsStorage.ts:90-193`
   的 `savePluginOptions()` 说明 top-level option truth 不跟着 seat 自动继承，而是要按 sensitive / non-sensitive split-save、双向 scrub 与 partial reconfigure grammar 被重新写定
2. `src/utils/plugins/pluginOptionsStorage.ts:282-309`
   的 `getUnconfiguredOptions()` 说明对象已经存在，仍可能被单独判定为 config 未闭合
3. `src/utils/plugins/mcpbHandler.ts:141-339`
   的 `loadMcpServerUserConfig()` / `saveMcpServerUserConfig()` 说明 per-server / per-channel payload truth 同样有自己的 source-of-truth grammar
4. `src/utils/plugins/mcpbHandler.ts:742-757,894-902`
   明确把 forced reconfigure 或 validation failed 继续写成 `needs-config`
5. `src/utils/plugins/mcpPluginIntegration.ts:290-317`
   的 `getUnconfiguredChannels()` 继续证明 channel config completeness 是单独 decision surface
6. `src/commands/plugin/PluginOptionsFlow.tsx:53-133`
   把 `configured / skipped / error` 写成正式 config-closure grammar
7. `src/commands/plugin/ManagePlugins.tsx:1645-1668,1700-1707`
   明确把 `Enabled and configured`、`Enabled` 与 `Configuration saved. Run /reload-plugins for changes to take effect.` 拆成三种不同事实
8. `saveMcpServerUserConfig()` 和 `savePluginOptions()` 都坚持 secure-first、scrub-stale、partial-save-not-destructive，说明 current config truth 的设计目标从来不是“把旧值尽量留着”，而是“让当前被承认的值彼此不矛盾”

这说明：

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象重新拿回了 current authority seat。`

后者才配说：

`它拿回 seat 之后，当前究竟按哪组 options / secrets / payload / channel config 重新工作。`

所以 `423` 之后必须继续补的一层就是：

`安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层`

也就是：

`stronger-request cleanup re-entitlement governor 负责决定对象回来后还配不配继续占 seat；stronger-request cleanup reconfiguration governor 才负责决定它占住 seat 之后按哪组 current config truth 重新运转。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reconfiguration manager，
而是在说：

1. repo 已经在 plugin 线成熟写出 reconfiguration grammar
2. repo 已经把 seat truth 与 config truth 拆开
3. stronger-request cleanup 线如果真要把 post-return world 做完整，也迟早要回答“回来后它究竟按哪组 current config truth 继续工作”

第二条：

这里说的 `reconfiguration governance`
不是在声称：

`所有旧配置都必须原样回来。`

当前更稳的说法只能是：

`reconfiguration governance 首先回答的是哪些 current config truth 仍被承认、哪些必须重填、哪些必须 scrub、哪些只能作为 stale residue 被丢弃。`

第三条：

这里也不是把：

`enabled`

单独当成全部 configuration truth。

真正关键的是：

1. 哪些输入属于 current source-of-truth
2. 哪些字段仍是 missing / invalid
3. 哪些值只是 saved，哪些已经 active
4. 哪些 secret reuse 仍被当前 schema 承认

第四条：

这里尤其不能过度声称：

`savePluginOptions()`

就等于完整 reconfiguration governance。

更稳的说法只能是：

`top-level options 只是 config truth 的一层；per-server payload、channel closure、save / apply choreography 仍要被单独治理。`

第五条：

这里也不能过度声称：

`配置已保存`

就等于：

`当前运行时已经按这组 truth 工作。`

更稳的说法只能是：

`saved truth` 和 `applied truth` 在 Claude Code 里被显式拆开；`/reload-plugins` 就是这条分界被公开说出的证据。`

第六条：

这里还不能过度声称：

`reconfiguration` 只是恢复体验里补填表单的交互步骤。

更稳的说法只能是：

`它本质上是在治理 returned object 的 current config truth grammar；表单只是输入界面，真正被治理的是 source、closure、save 与 apply 之间的制度分层。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-re-entitlement-governor signer 仍不等于 stronger-request cleanup-reconfiguration-governor signer`

证据：

1. `savePluginOptions()` 把 top-level option truth 写成 split-save + secure-first + cross-store scrub grammar
2. `getUnconfiguredOptions()` 把 schema-failed field 继续收束回 unconfigured surface
3. `loadMcpServerUserConfig()` / `saveMcpServerUserConfig()` 把 per-server / per-channel payload truth 提升成独立 source-of-truth plane
4. `needs-config` 明确拒绝把 visible object 偷写成 ready object
5. `getUnconfiguredChannels()` 证明 channel completeness 是独立 decision surface
6. `PluginOptionsFlow` 把 `configured / skipped / error` 写成正式 closure outcome
7. `ManagePlugins` 把 `Enabled`、`Enabled and configured` 与 `Configuration saved...take effect` 写成三种不同句法
8. secure-first + stale-scrub + partial-save discipline 共同证明：当前配置真相的核心不是“保住旧值”，而是“让被承认的值保持一致”

因此这一章的最短结论是：

`stronger-request cleanup re-entitlement governor 最多能说对象回来后仍配占哪张 seat；stronger-request cleanup reconfiguration governor 才能说它占住 seat 之后当前该按哪组 config truth 工作。`

再压成一句：

`资格回来，不等于配置也回来。`

再压成更硬的一句：

`制度知道谁还能坐回座位，不等于制度已经知道这个座位现在该按哪组输入重新运转。`

## 4. 第一性原理：re-entitlement governance 回答“它还配不配占这个 seat”，reconfiguration governance 回答“这个 seat 凭哪组 current truth 运转”

从第一性原理看，
强请求清理再赋权治理与强请求清理重配置治理处理的是两个不同主权问题。

`stronger-request cleanup re-entitlement governor`
回答的是：

1. 对象回来后是否重新获得 enabled truth
2. 是否重新获得某个 scope / precedence 位置
3. 是否仍被 policy 允许
4. 是否保留 old identity，还是只拿到 new seat
5. 是否重新进入可声明 authority 的世界

`stronger-request cleanup reconfiguration governor`
回答的则是：

1. 哪组 options / secrets 仍被视作 current source truth
2. 哪些字段当前仍缺失或失效
3. 哪些 payload / channel config 必须重填
4. 哪些值只是 saved、哪些已经 active
5. returned object 何时才配说自己 work-ready

如果把这两层压成一句：

`反正它已经重新 enable 了`

系统就会制造六类危险幻觉：

1. `enabled-means-configured`
2. `return-means-old-secret-still-valid`
3. `seat-restored-means-schema-closed`
4. `saved-means-applied`
5. `authority-seat-means-operable`
6. `old-config-presence-means-current-config-truth`

所以这两层的哲学本质也不同：

`re-entitlement`

治理的是：

`它还配不配占这个 seat。`

`reconfiguration`

治理的是：

`它占住这个 seat 之后，当前究竟凭哪组可被承认的输入真相运转。`

如果再压缩一点，
两者的本质差别其实是：

1. `re-entitlement` 负责 `authorization to stand`
2. `reconfiguration` 负责 `truth to operate`

一个 returned object 即便再次被承认为“它可以在场”，
也完全不等于系统已经承认“它现在知道该怎么工作”。

## 5. Claude Code 多重安全技术的先进性：它把 comeback 后的输入真相写成分层账本，而不是默认继承旧值

Claude Code 在这一层真正先进的地方，
不是“旧对象回来后把旧配置直接沿用”，
而是：

`它把 returned object 的 current config truth 写成 source、closure、save、apply 四层不同账本。`

这种先进性至少体现在六个方面：

1. `source truth split by sensitivity`
   top-level options 与 per-server payload 都按 sensitive / non-sensitive 分仓，拒绝把“值存在”偷写成“值在正确的真相源里存在”。
2. `stale truth is actively scrubbed`
   schema 翻转或 partial reconfigure 时，旧侧残留会被主动 scrub。系统不默认“旧值留下来反正也没坏处”。
3. `closure is validated, not assumed`
   `getUnconfiguredOptions()` 和 `getUnconfiguredChannels()` 都把 missing / invalid fields 单独收束出来，拒绝把 visible seat 偷写成 schema-closed seat。
4. `flow outcome is explicit`
   `configured / skipped / error` 是正式 outcome，不是 UI 细节。这让 config closure 成为一个可治理事实。
5. `saved truth and applied truth are split`
   `Configuration saved. Run /reload-plugins for changes to take effect.` 把保存和生效分成两张真相面，避免“刚写盘就算运行中”的错觉。
6. `ready truth is stricter than enabled truth`
   `Enabled` 与 `Enabled and configured` 并列存在，说明 runnability 的标准高于 mere seat occupancy。

这些设计背后的源码思路也很清楚：

1. 不让 `seat truth` 越级冒充 `config truth`
2. 不让 `config source truth` 越级冒充 `config closure truth`
3. 不让 `save truth` 越级冒充 `apply truth`
4. 不让 `old value still exists` 越级冒充 `current value still counts`

这对 stronger-request cleanup 的技术启示很直接：

1. 旧 startup wording 如果回来，是否仍按旧 wording payload 工作，不能由 re-entitlement 自己决定
2. 旧 cleanup law 如果回来，哪些 path / retention / scope 参数算 current config truth，必须被单独签字
3. 旧 receipt semantics 如果回来，哪些字段仍被 schema 承认、哪些必须重签或 scrub，不能由 mere seat occupancy 偷推
4. 旧 covered-family covenant 如果回来，save truth 与 active apply truth 是否仍分层，也必须被正式回答

Claude Code 当前 stronger-request cleanup 线还没有把这些 comeback-after-config grammar 正式接到 cleanup artifact family 上。

因此这里真正被钉住的不是：

`cleanup 线还缺一个 current seat 开关。`

而是：

`cleanup 线还缺一个 current config truth grammar。`

## 6. 苏格拉底式自反诘问：我是不是又把“对象拿回资格”误认成了“它已经按正确配置工作”

如果对这组代码做更严格的自我审查，
至少要追问八句：

1. 如果 re-entitlement grammar 已经足够强，为什么还要再拆 reconfiguration？
   因为能坐回 seat，不等于知道该按哪组输入运转。
2. 如果对象已经 enable，是不是就说明配置一定已齐？
   不是。`Enabled` 与 `Enabled and configured` 被明确拆开。
3. 如果旧 secret 还在 secure storage 里，是不是就说明 current truth 还承认它？
   不能这样推。secure-first 和 scrub-stale 恰恰说明存在不等于当前有效。
4. 如果 flow 走完并保存成功，是不是就说明运行时已经在用这组值？
   也不对。`saved` 与 `take effect` 被公开拆成两层。
5. 如果 channel config 只有少数字段缺失，是不是可以默认剩下的旧值都继续算 current truth？
   不能这样推。partial reconfigure 恰恰要求系统明确知道哪些字段仍闭合、哪些已失效。
6. 如果 closure-level config 只是 plugin 线细节，cleanup 线是不是可以忽略 relation bundle？
   不行。relation bundle 越存在，越说明单对象 config truth 不足以描述当前世界。
7. 如果 cleanup 线还没有显式 reconfiguration 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层 grammar，越容易把“它又能看见了”误说成“它现在知道该怎么工作”。
8. 如果我继续把 reconfiguration 当成 re-entitlement 的副产品，会漏掉什么？
   会漏掉 current source truth、config closure、save/apply 分层与 stale scrub 这些真正决定 returned object 如何工作的制度对象。

这一串反问最终逼出一句更稳的判断：

`reconfiguration 的关键，不在对象有没有回来、也不在对象有没有 seat，而在系统能不能正式决定它现在究竟按哪组 current truth 运转。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 re-entitlement governance，就已经足够成熟。`

而是：

`Claude Code 在 split-save source truth、missing-config closure、needs-config gate、configured/skipped outcome 与 save-vs-apply distinction 上已经明确展示了 reconfiguration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁还能回来并坐下”，而是“谁来决定它坐下以后现在按哪组输入真相重新工作”。`
