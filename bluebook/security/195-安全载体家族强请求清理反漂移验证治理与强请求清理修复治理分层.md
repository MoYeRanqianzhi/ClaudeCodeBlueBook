# 安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层：为什么artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer不能越级冒充artifact-family cleanup stronger-request cleanup-repair-governor signer

## 1. 为什么在 `194` 之后还必须继续写 `195`

`194-安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层` 已经回答了：

`stronger-request cleanup 线即便某次运行看起来已经 conform，也还不等于系统已经具备长期 anti-drift verification；repo 在别处已经有 test-based、atomic 与 live re-verification 三种 verifier 文化，而 cleanup 线当前还没接上。`

但如果继续往下追问，
还会碰到下一层更容易被偷写成同一动作的错觉：

`只要 verifier 已经发现 stronger-request cleanup drift，它就等于自动拥有了修复主权。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看：

1. `src/utils/permissions/permissionSetup.ts:1078-1152` 的 `verifyAutoModeGateAccess()` 负责发现 stale-vs-live divergence
2. 但真正把结果施加到 app state、通知用户并改变 mode world 的是 `src/utils/permissions/bypassPermissionsKillswitch.ts:74-116` 的 `checkAndDisableAutoModeIfNeeded()`
3. `src/utils/permissions/getNextPermissionMode.ts:7-17` 也只负责在 local cycle 前揭露 divergence 风险，并不拥有 circuit breaker 或 promise demotion 主权
4. plugin 线里 `src/utils/plugins/dependencyResolver.ts:157-233` 的 `verifyAndDemote(...)` 与 `src/utils/plugins/pluginLoader.ts:3188-3198` 的 session-local disable 更进一步展示了另一种成熟 grammar：发现 dependency drift 之后，repo 会明确决定 capability consequence，而且注释还特意写明 `does NOT write settings`，也就是 repair scope 被有意限定
5. 与这些正对照相比，stronger-request cleanup 线当前虽然已经暴露出 temporal gap、propagation gap 与 receipt gap，但还没有谁正式决定：
   应该修 metadata、修 scheduler、修 executor、扩大 receipt，还是降低 `deleted at startup` 之类承诺语言
6. 也就是说，stronger-request cleanup 当前不只缺 verifier grammar，还缺 drift 被抓住之后的 repair governance grammar

这说明：

`artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer`

和

`artifact-family cleanup stronger-request cleanup-repair-governor signer`

仍然不是一回事。

前者最多能说：

`这里已经 drift 了，或者现在缺少足够证据证明它没有 drift。`

后者才配说：

`面对这个 drift，系统应该修 metadata、修 scheduler、修 executor、修 receipt plane，还是缩窄承诺、临时熔断某条 capability，并由哪一层承担这些修复代价。`

所以 `194` 之后必须继续补的一层就是：

`安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层`

也就是：

`stronger-request cleanup-anti-drift verifier 只能发现与揭露 drift；stronger-request cleanup-repair governor 才能决定如何修、修到哪一层、代价由谁承担。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-repair-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-repair-governor signer` 仍是研究命名。
它不是在声称仓库里已经有一套 cleanup governor 接口只差文档，
而是在说：

1. repo 其他子系统已经展示出 verification 与 governance consequence 的分层
2. stronger-request cleanup 线今天真正缺的不是风险意识，而是 drift 被 verifier 报警后谁来行使 repair sovereignty
3. 所以更高阶的问题已经不是“谁来发现 drift”，而是“发现以后谁来做制度性处置”

第二条：

这里说的 `repair governance`

不是在声称：

`一旦 drift 出现，答案一定是修代码。`

真正要说的是：

`repair governance 可能意味着修 executor、修 metadata、修 scheduler、补 receipt、降级 promise wording，甚至暂时禁止某条路径；关键不在“修不修”，而在“谁来正式决定怎么修”。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部 repair problem。

真正关键的是：

1. temporal gap 可能要求修 scheduler 或改文案
2. propagation gap 可能要求修 executor 或限缩 override semantics
3. receipt gap 可能要求补 surfaced conformance plane
4. stronger-request cleanup 线当前没有一处可见代码显式持有这类 repair choice

第四条：

这里尤其不能过度声称：

`verifier 的输出天然中立，不会隐含治理方向选择。`

当前更稳的说法只能是：

`一旦 verifier 报出 drift，系统必然要在“修哪一层、谁付代价、是否降级承诺”之间做治理选择；这正是 repair governance 的独立价值。`

第五条：

这里也不能过度声称：

`repair governance 一定意味着持久化修改 settings。`

当前更稳的说法只能是：

`plugin 线已经展示出 session-local demotion、not writing settings 也是一种 repair governance；因此 cleanup 线未来的 repair 也可能是 session-local、receipt-local 或 wording-local，而不是只能改持久状态。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类“stronger-request cleanup-anti-drift-verifier signer 仍不等于 stronger-request cleanup-repair-governor signer”证据：

1. `verifyAutoModeGateAccess()` 负责发现 divergence，`checkAndDisableAutoModeIfNeeded()` 才负责把结果施加到 app state 与 notification queue  
   `permissionSetup.ts:1078-1152`; `bypassPermissionsKillswitch.ts:74-116`
2. `getNextPermissionMode.ts:7-17` 只负责在本地切换前承认 divergence 风险，并不拥有 circuit breaker 或 repair sovereignty  
   `getNextPermissionMode.ts:7-17`
3. plugin 线里 `verifyAndDemote(...)` 的命名本身就把 verify 与 demote governance 并列写出  
   `dependencyResolver.ts:157-233`
4. `pluginLoader.ts:3188-3198` 还明确把 demotion 约束为 session-local consequence，`does NOT write settings`，说明 repair scope 本身就是治理决策  
   `pluginLoader.ts:3188-3198`
5. stronger-request cleanup 线当前虽已暴露 temporal / propagation / receipt gap，却还没有谁正式决定这些 gap 应修 executor、修 metadata 还是降级 promise wording  
   `cleanup.ts:575-595`; `plans.ts:79-106`; `cleanup.ts:300-303`
6. 因此 stronger-request cleanup 今天最缺的不是又一个 detector，而是 drift 暴露后的 repair grammar

因此这一章的最短结论是：

`stronger-request cleanup-anti-drift-verifier signer 最多能说“这里不对了”；stronger-request cleanup-repair-governor signer 才能说“该由哪一层付出代价把它重新拉回正确世界”。`

再压成一句：

`发现偏离，不等于拥有纠偏主权。`

## 4. 第一性原理：verifier 回答“哪里错了”，repair governor 回答“谁来承担修复代价”

从第一性原理看，
强请求清理反漂移验证治理与强请求清理修复治理处理的是两类根本不同的问题。

stronger-request cleanup-anti-drift verifier 回答的是：

1. 有没有 drift
2. drift 在哪一层
3. 当前证据是否充分
4. 哪些 truth 已经分叉
5. 当前是否应拒绝继续宣称“没问题”

stronger-request cleanup-repair governor 回答的则是：

1. 应该修 metadata，还是修 runtime
2. 应该修 scheduler，还是修 executor
3. 应该扩大 receipt，还是缩小承诺语言
4. 应该做 session-local demotion，还是持久修补
5. 修复成本由哪个 control plane 承担

如果把这两层压成一句“检测到问题就修”，
系统就会制造五类危险幻觉：

1. detection-means-authority illusion  
   能发现问题，就误以为自动拥有修复主权
2. failing-check-means-obvious-fix illusion  
   verifier 一报警，就误以为修哪一层是不言自明的
3. stronger-promise-means-keep-promise-at-all-costs illusion  
   文案已经写强了，就误以为只能修执行器，不能考虑降级承诺
4. one-gap-one-fix illusion  
   看见一处 drift，就误以为只对应一个局部代码补丁
5. verifier-neutrality illusion  
   误以为 verifier 产出的 truth 不会隐含治理方向选择

所以从第一性原理看：

`verifier` 是 truth exposure；
`repair governor` 是 repair sovereignty。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup anti-drift verification 已经等于 repair governance，为什么 `verifyAutoModeGateAccess()` 还要把治理后果交给 `checkAndDisableAutoModeIfNeeded()`？
   因为发现 divergence 与决定如何改变世界本来就不是同一主权。
2. 如果 verifier 发现问题后修法是显然的，为什么 plugin 线还要特意把 demotion 限定为 session-local 且 `does NOT write settings`？
   因为 repair scope 本身就是治理选择，而不是算法副产物。
3. 如果 stronger-request cleanup 现在已经知道 temporal、propagation 与 receipt gap，为什么还没有一处可见代码回答“该修文案、修 scheduler，还是修 executor”？
   因为 gap exposure 还没有被正式提升成 repair governor grammar。

## 5. `verifyAutoModeGateAccess` 与 `checkAndDisableAutoModeIfNeeded` 先给出最强正例：发现 divergence 和施加治理后果已经被显式拆层

`src/utils/permissions/permissionSetup.ts:1078-1152` 的 `verifyAutoModeGateAccess()` 负责：

1. fresh read config
2. compare stale cache and live gate
3. produce `updateContext` and `notification`

它是典型 verifier。

但真正的治理后果并不在这里结束。
`src/utils/permissions/bypassPermissionsKillswitch.ts:74-116` 的 `checkAndDisableAutoModeIfNeeded()` 才会：

1. 调用 verifier
2. 把结果施加到 current app state
3. 把 notification 推进 queue
4. 决定用户是否还留在当前 mode world

这说明 repo 的治理哲学非常清楚：

`先验证，再决定 consequence；先暴露 truth，再决定谁付代价改变世界。`

这正是 stronger-request cleanup 当前缺的第二步。

## 6. plugin 线再给出更硬的正例：`verifyAndDemote` 说明 repair governance 不只存在，而且已经被正式命名为 capability consequence

`src/utils/plugins/dependencyResolver.ts:157-233` 暴露出一个很值钱的命名：

`verifyAndDemote`

这说明两件事已经被工程上显式承认：

1. verify  
   发现 dependency truth 是否成立
2. demote  
   对不再满足约束的 plugin 施加治理后果

`src/utils/plugins/pluginLoader.ts:3188-3198` 还会真正消费这个后果：

1. `if (demoted.has(p.source)) p.enabled = false`
2. 并且注释明确写出  
   `Demotion is session-local: does NOT write settings (user fixes intent via /doctor).`

这里的信息量极大。

因为它说明 repo 不只知道 repair governance 存在，
还知道 repair governance 至少要回答三件事：

1. 后果是什么  
   `enabled -> false`
2. 作用域是什么  
   session-local
3. 不做什么  
   does NOT write settings

这正是 stronger-request cleanup 线目前还没有明确回答的。

## 7. 对照这些正例，stronger-request cleanup 线当前尚未回答的不是“哪里 drift”，而是“谁来决定修哪一层”

stronger-request cleanup 当前已知至少有三类 drift：

1. temporal gap  
   `cleanupPeriodDays=0` 文案强于 runtime admission
2. propagation gap  
   `plansDirectory` 被某些 plane 消费，但 executor 未跟随
3. receipt gap  
   `CleanupResult` 未升格成 family surfaced receipt

但当前仍没有一处可见代码回答：

1. temporal gap 该修 scheduler，还是改 wording
2. propagation gap 该修 executor，还是限制 `plansDirectory`
3. receipt gap 该补 receipt plane，还是缩小“已经符合”的宣称

这些问题都不是 verifier 本身能决定的。
它们已经进入：

`repair governance`

层。

所以 `195` 的核心判断就是：

`stronger-request cleanup 线真正缺的不只是 verifier grammar，而是 verifier 报警后由谁行使治理性修复主权。`

## 8. 技术先进性与哲学本质：Claude Code 真正先进的不是“检测器很多”，而是它在别处已经承认“发现问题”和“改变世界”必须分层

从技术上看，
Claude Code 在这条线上真正先进的地方，
不是它检测器多。

更值钱的是它已经在别处承认：

1. `verifyAutoModeGateAccess()` 不是 `checkAndDisableAutoModeIfNeeded()`
2. `verifyAndDemote` 不是“只报错不处置”
3. session-local demotion、not writing settings、/doctor user fix 这些都是 repair scope choice

这背后的哲学本质是：

`真正成熟的安全系统，不把“知道哪里错了”偷写成“自然知道怎么修”；它把修复代价、修复作用域和承诺降级权都当成独立主权。`

这也带来三条直接技术启示：

1. stronger-request cleanup 下一步最值钱的补强，不只是 verifier，更是明确 repair grammar
2. plans family 是 cleanup repair-governance 最好的报警器，因为它已经把 metadata / executor 分裂具体化
3. cleanup 线未来很可能需要像 plugin 线那样区分 session-local consequence、persistent change 与 user-fix-required path，而不是只会“自动修”

## 9. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要把 anti-drift verifier 补出来，剩下的治理问题就会自然解决。`

而是：

`repo 在 auto-mode 与 plugin 线已经明确把 verification 与 governance consequence 分层；stronger-request cleanup 线当前缺的正是同等级的 repair-governor plane。`

因此：

`artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer` 仍不能越级冒充 `artifact-family cleanup stronger-request cleanup-repair-governor signer`。`
