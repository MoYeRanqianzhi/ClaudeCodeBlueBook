# 配置来源优先级矩阵与受管环境主权：谁有资格定义运行时

## 1. 为什么还要单独写这一章

前面的 `03` 章已经解释了：

- 为什么运行时会被配置污染
- 为什么认证来源和 provider 来源必须分层
- 为什么 `managedEnv.ts` 是核心安全入口

但如果想继续深挖，还需要把下面这个问题单独压实：

`Claude Code 到底按什么优先级合并配置来源，又在哪些地方允许 host 或 managed settings 把用户配置反压回去？`

这一章回答的是“谁有资格定义运行时”。

## 2. 先说结论

Claude Code 的配置系统不是简单后者覆盖前者。  
更准确地说，它由三层机制叠加而成：

1. 普通 settings source 覆盖顺序
2. trust 前后不同的 env 生效范围
3. managed / host 对部分高风险语义的反向收口权

这意味着配置主权不是平面分布，  
而是：

`默认可覆盖 + 例外收口 + 高风险字段再治理` 的分层结构。`

## 3. 第一层：普通设置来源的基础覆盖顺序

`settings/constants.ts` 里把顺序写得很清楚：

1. `userSettings`
2. `projectSettings`
3. `localSettings`
4. `flagSettings`
5. `policySettings`

而且注释直接说明：

- later sources override earlier ones

这条顺序的安全含义很大：

- 用户级默认值可以被项目改写
- 项目共享值可以被 gitignored local 覆盖
- CLI flag 还能临时盖掉前面的三层
- managed policy 最后再统一收口

换句话说：

`它先承认灵活性，再把最终主权留给 managed 层。`

## 4. 第二层：并不是所有来源都总是启用

`getEnabledSettingSources()` 又加了一层现实约束：

- 允许的 sources 会受 runtime 允许列表影响
- 但 `policySettings` 和 `flagSettings` 永远被包含

这说明系统对配置灵活性的理解不是：

- 所有人都随时都能进来

而是：

- 某些来源可以被整体关掉
- 但管理面和显式 CLI 面仍然保留主权

从安全角度看，这是一种很成熟的最小面原则：

`不该启用的来源可以整体拿掉，但管理员和显式用户操作仍保留。`

## 5. 第三层：trust 前后，env 生效范围并不一样

`managedEnv.ts` 是这一章最重要的核心。

系统把 env 分成了两次应用：

### 5.1 trust 前：只应用可信来源 + safe env

在 trust 建立前：

- global config / trusted setting sources 的 env 可以应用
- project/local settings 只能有 safe allowlist 中的 env 生效

这样做的目的非常明确：

- 防止恶意项目在用户还没建立 trust 前，就用高风险 env 改写运行时

### 5.2 trust 后：才允许全量 settings.env 生效

在 trust 之后：

- 才会应用完整 merged settings 的 env
- 同时刷新 CA、mTLS、proxy 等缓存与全局 agent 配置

这说明 Claude Code 把 trust 看成一个真正的边界事件，  
而不是一个纯 UI 提示。

## 6. 第四层：并非所有 env 都能被自由覆盖

`managedEnvConstants.ts` 进一步说明：

- provider-routing env 被单独列成一组
- host 若声明 `CLAUDE_CODE_PROVIDER_MANAGED_BY_HOST`，这些变量会从 settings-sourced env 中被剥离

这组变量包括：

- provider 选择
- base URL
- auth token / API key
- model defaults
- region / resource / project routing

其设计意图非常清楚：

`宿主如果拥有推理路由主权，用户配置就不应反向接管这条主链。`

## 7. 第五层：safe env allowlist 本身就是一张风险分类表

`SAFE_ENV_VARS` 的注释其实已经把风险分层讲清了。

源码把“不安全 env”大致分成了几类：

1. 把请求重定向到攻击者控制的服务
2. 信任攻击者控制的证书或代理
3. 切换到攻击者控制的 project / resource / auth 语义

这意味着 SAFE_ENV_VARS 不是随手拼出来的白名单，  
而是：

`一张关于运行时攻击面的分类表。`

从工程角度看，这非常成熟。  
因为它不是只问“这个变量常不常用”，  
而是问：

`这个变量会不会改变控制权。`

## 8. 第六层：managed settings 对规则系统还有二次主权

安全主权不只体现在 env。  
`permissionsLoader.ts`、`hooksConfigSnapshot.ts` 和 `sandbox-adapter.ts` 表明，managed settings 还可以对其他安全面进行再收口。

### 8.1 permission rules 的 managed-only

`permissionsLoader.ts` 支持：

- `allowManagedPermissionRulesOnly`

一旦开启：

- 只有 managed permission rules 被尊重
- always-allow 一类用户便捷写法会被整体收紧

### 8.2 hooks 的 managed-only

`hooksConfigSnapshot.ts` 支持：

- `allowManagedHooksOnly`
- `disableAllHooks`

这意味着 hooks 的执行主权可以被管理员重新收口。

### 8.3 sandbox 域名与读路径的 managed-only

`sandbox-adapter.ts` 又进一步支持：

- `allowManagedDomainsOnly`
- `allowManagedReadPathsOnly`

这说明 managed 层不只是“最后覆盖一个值”，  
而是可以：

`改变整类安全策略的允许范围。`

## 9. 从技术角度看，这套优先级设计先进在哪里

它至少有五点成熟之处：

1. 配置来源顺序被显式声明，而不是隐式散落
2. trust 前后使用不同 env 生效策略
3. host 可对 provider 主链实施反向收口
4. managed 层可以把 permission / hooks / sandbox 从“可混用”切到“只认 managed”
5. SAFE_ENV_VARS 不是方便性清单，而是风险语义清单

很多系统只能做到第一点。  
Claude Code 基本把这五点都做了。

## 10. 一张压缩后的运行时主权矩阵

| 层 | 默认逻辑 | 更高主权的收口方式 | 安全含义 |
| --- | --- | --- | --- |
| settings source | 后者覆盖前者 | `policySettings` 最后覆盖 | 默认承认灵活性 |
| enabled sources | 受 allowed sources 控制 | `policySettings` / `flagSettings` 永远保留 | 关键来源不可整体失联 |
| trust 前 env | 只允许可信来源 + safe env | 项目/本地危险 env 先被压住 | 防仓库污染宿主 |
| provider routing | 可被 env 配置 | host-managed provider vars 反向剥离 | 防用户配置破坏宿主主链 |
| permission / hooks / sandbox | 默认可混合来源 | managed-only flags 收口 | 防本地灵活性压过组织安全策略 |

这张表最想说明的一点是：

`Claude Code 的配置系统不是“谁后写谁赢”，而是“谁更接近主权边界，谁在关键层面赢”。`

## 11. 从第一性原理看，这套设计究竟在防什么

如果从第一性原理压缩，这一章其实在防三类典型失控：

1. 项目内容反向接管宿主运行时
2. 用户局部设置反向破坏 host/enterprise 管理语义
3. 看似只是“配置覆盖”的行为，实质上变成身份、路由或扩展主权被偷走

因此它真正要守的不是“配置文件”，  
而是：

`配置背后的主权边界。`

## 12. 给平台构建者的技术启示

如果要借鉴 Claude Code，这一章最值得学的是：

1. 配置系统必须显式声明覆盖顺序
2. trust 前后应采用不同的 env 策略
3. 对 provider / auth / proxy / cert 这类主链变量，要单独建风险模型
4. managed-only 不只是“只读”，而应能切换整类安全策略的主权
5. “safe env allowlist” 应来自攻击面分析，而不是方便性直觉

## 13. 一句话总结

Claude Code 的配置与受管环境设计之所以成熟，不在于它有更多设置项，而在于它把运行时主权做成了分层结构：默认覆盖顺序承认灵活性，trust 前后的 env 应用重新划边界，host 与 managed 层则在 provider、permission、hooks 和 sandbox 等关键面上保留最终收口权。
