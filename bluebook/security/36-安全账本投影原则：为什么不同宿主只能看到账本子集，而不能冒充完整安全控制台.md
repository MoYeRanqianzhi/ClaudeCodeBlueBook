# 安全账本投影原则：为什么不同宿主只能看到账本子集，而不能冒充完整安全控制台

## 1. 为什么在 `35` 之后还要继续写“账本投影原则”

`35-安全多账本原则` 已经回答了：

`语义、复制、可见与恢复本来就是不同的账。`

但如果继续往第一性原理再压一层，  
还会出现另一个同样关键的问题：

`既然账本不同，为什么不同宿主不能都声称自己看到了“完整安全状态”？`

因为 Claude Code 的现实并不是：

`每个宿主都拿到同一份完整控制面。`

源码反而在不断说明另一件事：

`不同宿主通常只被允许看到、消费或写入其中一部分账本。`

所以这一章的核心问题是：

`Claude Code 为什么必须把宿主理解成“账本投影的读者”，而不是“天然拥有完整控制台的全知端”？`

## 2. 最短结论

从源码看，Claude Code 对宿主采取的不是“统一完整视图”设计，  
而是：

`协议全集 + 宿主子集 + 投影纪律`

也就是说：

1. 协议层给出一个可能很宽的全集
2. 具体宿主只实现其中的一个子集
3. 这个子集通常只覆盖部分账本
4. 因此宿主只能诚实地把自己声明为某种投影，而不能冒充完整安全控制台

所以本章最短结论是：

`成熟的安全控制面，不只要维护多账本，还要维护“不同宿主只能看到哪一本账的哪一部分”这一层投影纪律。`

## 3. 什么叫“账本投影”

这里说的“账本投影”，指的是：

`某个宿主真正能读取、写入或消费的安全账本子集。`

这至少包含三件事：

1. 它看得到哪些账
2. 它能影响哪些账
3. 它看不到哪些账，却又很容易误以为自己看到了全部

因此，“宿主能力”不能只按功能多少来理解，  
还必须按：

`它到底拿到了哪几本账的哪一层投影`

来理解。

## 4. 源码先给出的是“协议全集”，不是“每个宿主都能实现的全集”

## 4.1 `coreSchemas.ts` 和 `controlSchemas.ts` 明确暴露了一个很宽的协议面

`src/entrypoints/sdk/coreSchemas.ts:1457-1750` 与  
`src/entrypoints/sdk/controlSchemas.ts:157-308` 已经很清楚：

协议面至少包含：

1. `system:init`
2. `status`
3. `auth_status`
4. `session_state_changed`
5. `task_notification`
6. `get_context_usage`
7. `mcp_status`
8. 以及更多 control request / response

这说明 schema 首先定义的是：

`理论上可以被宿主接入的一张大图。`

## 4.2 但 schema 的存在不等于每个宿主都拥有同样宽的账本视野

这是这一章最重要的前提：

`协议全集不是宿主视野全集。`

否则后面那些 narrow host、bridge-only host 和恢复期 drop 逻辑就完全没必要存在。

## 5. 第一类投影：`StructuredIO` 的默认宿主投影其实很窄

## 5.1 `StructuredIO` 默认没有恢复账本，也没有 internal-event 持久能力

`src/cli/structuredIO.ts:135-195` 很关键。

这里源码明确写出：

1. `restoredWorkerState` 默认是 `Promise.resolve(null)`
2. 注释直接说 transport may not restore
3. `flushInternalEvents()` 对非 remote IO 是 no-op
4. `internalEventsPending` 默认就是 `0`

这说明默认 `StructuredIO` 宿主并不天然拥有：

- 恢复账读取面
- internal event 持久面
- 远端复制账的完整回读能力

也就是说，  
默认宿主视角本身就是：

`一个窄投影。`

## 5.2 这意味着“能跑 control protocol”不等于“拥有完整多账本视野”

很多人会误以为：

`只要接上 StructuredIO，就等于拿到了完整控制台。`

源码恰恰在反驳这一点：

`默认 StructuredIO 只保证一条可用的控制/消息通道，不保证恢复账和持久账天然存在。`

## 6. 第二类投影：`RemoteIO + CCR v2` 是更完整的宿主投影

## 6.1 `RemoteIO` 明确把更多账本接进来

`src/cli/remoteIO.ts:88-140`、`205-252` 说明：

1. `RemoteIO` 在 CCR v2 下初始化 `CCRClient`
2. 会把 `restoredWorkerState` 绑定到 init 结果
3. 注册 internal event writer
4. 注册 internal event reader
5. 注册 `reportState(...)`
6. 注册 `reportMetadata(...)`
7. 覆盖 `flushInternalEvents()`
8. 覆盖 `internalEventsPending`
9. 写消息时优先走 `ccrClient.writeEvent(...)`

这说明 `RemoteIO` 比默认 `StructuredIO` 拿到的不是同一层投影，  
而是：

`一个包含复制账、可见账、恢复账更多接口的厚投影。`

## 6.2 但它依然不是“无条件完整控制台”

即使在 `RemoteIO` 里，  
很多能力也依赖：

- CCR v2 是否启用
- transport 是否是 SSETransport
- 当前是否真的初始化成功

这说明更厚的投影仍然是：

`条件成立后的较完整投影`

而不是“任何时候都全知全能”的绝对控制台。

## 7. 第三类投影：`DirectConnect` 是典型的权限型窄宿主

## 7.1 `DirectConnectSessionManager` 只认真处理 `can_use_tool`

`src/server/directConnectManager.ts:82-96`、`152-177` 已经足够说明：

1. 它识别 `control_request`
2. 真正支持的核心 subtype 是 `can_use_tool`
3. 其他 subtype 会直接回 error
4. 主动发出的 request 也只是 `interrupt`

这说明 direct connect 宿主并不是在实现完整控制协议，  
而是在实现一个非常窄的子集：

`权限请求 / 权限响应 / 中断`

## 7.2 这类宿主看到的是权限账投影，不是完整安全账本集合

换句话说，  
它的主要视野是：

`当前 permission 流程是否成立`

而不是：

- 恢复账是否完整
- transcript 是否已持久
- 多层状态是否已对齐
- 整个控制台是否可解释

所以把 direct connect 写成“完整宿主”是明显过度断言。

## 8. 第四类投影：`RemoteSessionManager` 也是窄宿主，只是多了远程会话发送面

## 8.1 它同样只把 `can_use_tool` 当成正式支持的 control subtype

`src/remote/RemoteSessionManager.ts:188-206` 已经写得很清楚：

1. `can_use_tool` 会被记录成 pending permission request
2. 其他 control subtype 会被当成 unsupported 并返回 error

所以它在 control 面上依旧是：

`窄子集宿主`

## 8.2 `sendMessage(...)` 说明它还有消息投递面，但这不等于完整账本面

`src/remote/RemoteSessionManager.ts:224-241` 说明：

1. 它可以把 message content 发到 remote session
2. 可以回 permission response
3. 可以发 interrupt

但这些能力仍不足以让它自动拥有：

- 完整恢复账
- 完整复制账
- 完整可见账 crosswalk
- 全套 control schema 能力

因此更准确的表述是：

`RemoteSessionManager = 远程会话消息面 + 权限型窄控制面`

而不是完整安全控制台。

## 9. 第五类投影：`remoteBridgeCore` 是更宽的宿主投影，但仍然只是投影

## 9.1 bridge 能转发更宽的消息与控制面

`src/bridge/remoteBridgeCore.ts:813-878` 说明 bridge 至少可以：

1. `writeSdkMessages(...)`
2. `sendControlRequest(...)`
3. `sendControlResponse(...)`
4. `sendControlCancelRequest(...)`
5. `sendResult()`

相比 direct connect / remote session manager，  
它明显是更宽的宿主投影。

## 9.2 但 bridge 依然会在恢复期主动 drop 一部分写入

同一段代码还说明：

1. 401 recovery 中会 drop control request
2. 也会 drop control response
3. 还会 drop control cancel request 和 result

这意味着 bridge 不是“完整真相无损搬运”，  
而是：

`一个受恢复窗口、去重和状态纪律约束的较宽投影。`

也就是说，  
就算是更强的宿主，  
也仍然必须承认自己：

`看到的是某种条件化投影，不是无条件全视角。`

## 10. 把这些宿主压成一个层级

如果把这几类宿主按账本投影宽度压缩，大概可以得到：

1. 默认 `StructuredIO`：基础控制/消息投影
2. `DirectConnect`：权限型窄投影
3. `RemoteSessionManager`：消息面 + 权限型窄投影
4. `remoteBridgeCore`：更宽的事件/控制投影，但受恢复纪律约束
5. `RemoteIO + CCR v2`：最接近完整多账本宿主投影

这张层级图最重要的不是给谁贴“强弱”标签，  
而是说明：

`不同宿主根本不在看同一组账。`

## 11. 把这些例子压成一条底层公理

我会把它压成一句话：

`宿主不是安全控制面的完整真相源，而是多账本系统上的条件化读者。`

因此成熟系统不该问：

`这个宿主能不能接上协议？`

而应该继续追问：

1. 它到底拿到了哪些账
2. 它缺哪几本账
3. 它能不能诚实声明自己的子集身份
4. 它有没有把子集误装成全集

## 12. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只在于给出宽协议，还在于允许窄宿主诚实地只实现子集。`

这对其他 Agent 平台构建者的启示非常直接：

1. 协议设计要允许子集实现
2. 子集宿主必须显式失败，而不是假装支持
3. 宿主应声明自己缺哪些账本
4. UI 和支持体系要按宿主子集给解释，不能默认每个宿主都全知
5. 更强宿主也要声明自己的条件化窗口，例如恢复期 drop

否则系统就会反复制造一种很坏的假象：

`接了协议 = 拥有完整控制台`

而这恰恰是很多误判和错误期待的来源。

## 13. 哲学本质

这一章更深层的哲学是：

`看到一部分，不等于拥有全部。`

很多系统喜欢把“接入能力”误写成“完整理解能力”。  
Claude Code 的源码反过来说明：

`真正成熟的控制面，会让每个宿主诚实地承认自己只是某种投影。`

这不是保守，  
而是认识论上的自律。

## 14. 苏格拉底式反思：这章最容易犯什么错

### 14.1 我们是不是把“窄宿主”写成了低级宿主

不应该。  
窄不等于差。  
很多时候窄子集恰恰是更安全、更稳定的接入方式。

### 14.2 更宽的宿主是不是就一定更好

也不是。  
更宽意味着更强解释力，  
但也意味着更高实现和恢复纪律要求。

### 14.3 既然投影这么重要，为什么不让每个宿主都显式声明子集

这恰恰是本章反推出的产品改进方向之一。  
源码已经有很多子集事实，  
但还缺统一、可读、产品化的宿主子集声明层。

## 15. 结语

`35` 回答的是：为什么安全控制面必须有多本账。  
`36` 继续推进后的结论则是：

`既然账本本来就不止一本，宿主也就不可能天然都看到完整账本集合。`

这意味着 Claude Code 更深层的安全启示之一是：

`不要把宿主当成完整安全控制台，而要把它当成多账本系统上的诚实投影。`
