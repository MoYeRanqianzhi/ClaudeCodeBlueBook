# 安全载体家族强请求清理复活治理与强请求清理再赋权治理分层：为什么artifact-family cleanup stronger-request cleanup-resurrection-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer

## 1. 为什么在 `263` 之后还必须继续写 `264`

`263-安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪些旧对象可以重新回到 current world，也还需要 resurrection governor 去决定它们是否能回来、凭什么回来、回来时进入哪一层 world。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“对象回来就说明它原来的资格也一起回来了”
的错觉：

`只要 stronger-request cleanup resurrection governor 已经让旧对象重新出现，它就自动拥有了决定旧 identity、旧配置、旧 secrets、旧 scope 与旧可用资格是否一起恢复的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 re-entitlement 正对照，
会发现它其实明确拆开了五件事：

1. `object return`
2. `enabled truth`
3. `config / secrets trust`
4. `policy veto`
5. `identity seat`

最硬的证据至少有七组：

1. `src/utils/plugins/pluginOptionsStorage.ts:201-204,210-272`
   证明 `deletePluginOptions()` 只在 last installation 卸载时才抹掉旧 `pluginConfigs` 与旧 `pluginSecrets`，
   说明 comeback 不自动带回旧配置与旧 secrets
2. `src/services/plugins/pluginOperations.ts:560-571,610-648,721-727`
   证明 `setPluginEnabledOp()` 把 enable / disable 写成单独的 settings action，
   说明对象回来不等于自动恢复 enabled truth
3. `src/services/plugins/pluginOperations.ts:650-657`
   与
   `src/utils/plugins/pluginPolicy.ts:11-19`
   证明更高层 policy 仍可 veto 旧资格恢复
4. `src/services/plugins/pluginOperations.ts:664-687`
   证明 even after return，
   scope / precedence 仍要经过 override legality 判断，
   不是对象一回来就自动恢复原 seat
5. `src/utils/plugins/installedPluginsManager.ts:827-830,859-861`
   证明 disk / materialized existence 与 effective installed / enabled truth 仍是两层不同真相
6. `src/utils/plans.ts:239-257`
   与
   `src/screens/REPL.tsx:1790-1797`
   证明内容回来时仍可能只拿到 new slug，
   而不是恢复 original identity
7. `src/utils/plugins/pluginOptionsStorage.ts:206-208`
   还进一步说明即便 cleanup side-effect 失败，
   uninstall 结果也不被回滚，
   这说明“对象离场”与“旧资格擦除是否完全成功”也被谨慎拆层

这说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

和

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象可以重新出现。`

后者才配说：

`它重新出现之后，算不算原来那个被授权、被配置、被允许继续工作的对象。`

所以 `263` 之后必须继续补的一层就是：

`安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`

也就是：

`stronger-request cleanup resurrection governor 负责让对象回来；stronger-request cleanup re-entitlement governor 才负责决定它回来之后恢复哪些旧资格。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 re-entitlement manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出对象回来不等于资格回来
2. “回来”与“恢复旧资格”是两个不同问题
3. stronger-request cleanup 线如果真要把 resurrection world 做完整，也迟早要回答这些“回来后还算不算原来那个对象”的问题

第二条：

这里说的 `re-entitlement governance`
不是在声称：

`对象回来后必须恢复所有旧资格。`

当前更稳的说法只能是：

`re-entitlement governance 首先回答的是哪些资格可以回来、哪些必须重新授予、哪些根本不该回来。`

第三条：

这里也不是把：

`enable state`

单独当成全部 re-entitlement grammar。

真正关键的是：

1. 配置 / secret 会不会回来
2. enabled truth 会不会回来
3. policy 是否允许回来
4. identity / scope 是否还是原来的那个 seat

第四条：

这里尤其不能过度声称：

`只要旧对象重新 materialize，就已经恢复旧 effective entitlement。`

当前更稳的说法只能是：

`repo 明确把 object existence 与 effective entitlement 分开；前者只是 comeback，后者还要经过更高层治理。`

第五条：

这里也不能过度声称：

`new identity` 一定比 `old identity` 弱。

当前更稳的说法只能是：

`new slug / new seat 的关键不在强弱，而在系统拒绝把“内容回来”偷写成“原资格原封不动回来”。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-resurrection-governor signer 仍不等于 stronger-request cleanup-re-entitlement-governor signer`

证据：

1. `pluginOptionsStorage.ts:201-204,210-272`
   的 `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`，
   说明对象回来不自动带回旧配置与旧 secrets
2. `pluginOperations.ts:560-571,610-648,721-727`
   的 `setPluginEnabledOp()` 把 enable / disable 做成单独的 settings write，
   说明对象可 install、materialize、refresh，
   但是否恢复 enabled truth 仍要经过独立写入
3. `pluginPolicy.ts:11-19`
   与
   `pluginOperations.ts:650-657`
   明确写出 org policy 可以继续 veto install / enable
4. `pluginOperations.ts:664-687`
   与
   `installedPluginsManager.ts:827-830,859-861`
   说明 current seat 与 effective entitlement 仍受 scope precedence 与 divergence guard 约束
5. `plans.ts:239-257`
   与
   `REPL.tsx:1790-1797`
   证明内容回来时仍可能只获得 new identity，而不是恢复 original seat
6. `pluginOptionsStorage.ts:206-208`
   说明即便旧资格擦除 side-effect 失败，系统也不会倒推出“旧资格仍应被保留”，
   反而继续把 uninstall 成功与 qualification cleanup 分开记账

因此这一章的最短结论是：

`stronger-request cleanup resurrection governor 最多能说对象能回来；stronger-request cleanup re-entitlement governor 才能说它回来之后恢复什么旧资格。`

再压成一句：

`会回来，不等于还是原来那个被授权的它。`

## 4. 第一性原理：resurrection governance 回答“能否回来”，re-entitlement governance 回答“回来后恢复哪些资格”

从第一性原理看，
强请求清理复活治理与强请求清理再赋权治理处理的是两个不同主权问题。

`stronger-request cleanup resurrection governor`
回答的是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回到的是哪一层 world
4. 是否沿原 lineage 回来
5. 是否应该避免 clobber

`stronger-request cleanup re-entitlement governor`
回答的则是：

1. 对象回来后是否恢复旧 enabled state
2. 是否恢复旧 secrets
3. 是否恢复旧 pluginConfigs / options
4. 是否恢复旧 scope / precedence 位置
5. 是否恢复旧 identity，还是只给它一个新的 current seat

如果把这两层压成一句
“反正它已经回来了”，
系统就会制造五类危险幻觉：

1. `return-means-enabled`
2. `return-means-configured`
3. `return-means-same-scope`
4. `return-means-same-identity`
5. `existence-means-entitlement`

再用苏格拉底式反问压一次：

1. 如果 resurrection governance 已经等于 re-entitlement governance，
   为什么 `deletePluginOptions()` 还要在 last-scope uninstall 时主动抹掉旧配置与旧 secrets？
   因为“对象回来”与“旧配置继续值得信任”根本不是同一句制度。
2. 如果对象一旦回来就自动恢复旧 enabled truth，
   为什么 `setPluginEnabledOp()` 还要成为独立写入动作，并且继续受 policy 与 scope guard 约束？
   因为 comeback 与 entitlement return 是两层主权。
3. 如果内容回来就等于 old identity 回来，
   为什么 `copyPlanForFork()` 还要明确生成 new slug？
   因为 resurrection 只回答内容能否重现，re-entitlement 还要回答它是否仍算原 seat。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“再赋权治理”是一层独立制度

### 5.1 plugin 线先证明：对象回来并不自动恢复旧配置、旧 secrets 与旧 enabled truth

`pluginOptionsStorage.ts:201-204`
先把 `deletePluginOptions()` 的制度边界写死：

1. 只在 LAST installation 被移除时调用
2. 不是每次 uninstall 都删
3. 因为 plugin 可能仍在其他 scope 存活

这说明 repo 在资格擦除上先做的不是“一刀切”，
而是：

`只有当旧 entitlement 世界真的彻底结束时，才抹掉旧配置与旧 secrets。`

继续看
`pluginOptionsStorage.ts:210-272`：

1. settings side 会删 `pluginConfigs[pluginId]`
2. secure storage side 会删 `pluginSecrets[pluginId]`
3. 还会删 `${pluginId}/${server}` 这类 per-server composite keys
4. 最后 clear options cache

这说明 plugin comeback 之后，
系统不会自动假定：

`旧配置还有效`

或

`旧 secrets 还应继续被信任`

再看
`pluginOptionsStorage.ts:206-208`：
即便 keychain cleanup 失败，
repo 也明确不把 uninstall itself 回滚成失败。

这说明：

`对象离场`

和

`旧资格痕迹是否已被完全擦除`

本来就不是同一层事实。

接着看
`pluginOperations.ts:560-571`
与
`pluginOperations.ts:610-648`：

1. `setPluginEnabledOp()` 先 resolve `pluginId` 与 scope
2. 明确说明 settings declares intent
3. 不把 installed metadata 当成 entitlement truth 的最高主权

继续看
`pluginOperations.ts:721-727`：
真正的 enable / disable 是一笔单独的 settings write。

这说明：

`对象存在`

和

`对象被重新赋予 enabled truth`

是两层不同动作。

### 5.2 policy guard 与 seat legality 再证明：回来不等于重新被允许，也不等于重新占回旧位置

`pluginPolicy.ts:11-19`
把 policy guard 写得很直白：

1. `policySettings.enabledPlugins[pluginId] === false`
2. 即视为 blocked
3. install chokepoint、enable op 与 UI filters 都共用这份 truth

再看
`pluginOperations.ts:650-657`：
enable 前会直接挡住：

`Plugin "${pluginId}" is blocked by your organization's policy and cannot be enabled`

这说明：

`能回来`

和

`回来后还被允许`

不是同一问题。

但 re-entitlement 不只回答 allow / deny。
它还回答：

`回来后是不是仍占原来的 scope / precedence seat。`

`pluginOperations.ts:664-687`
在 explicit scope given but plugin is elsewhere 时，
会继续做 cross-scope legality 判断：

1. 若 absent from requested scope but present at another scope
2. 且这次不是合法 higher-precedence override
3. 则直接返回 wrong-scope guidance

这说明对象即便回来了，
系统也不会自动假定：

`它还是原来那个 scope 上的有效 seat。`

### 5.3 divergence guard 再证明：current entitlement 不是单看 object 在不在

`installedPluginsManager.ts:827-830`
写得很明白：

1. plugins are loaded from `settings.enabledPlugins`
2. 如果 `settings.enabledPlugins` 与 `installed_plugins.json` diverge
3. 则 `isPluginInstalled()` 返回 `false`

继续看
`installedPluginsManager.ts:859-861`：

1. same settings divergence guard as `isPluginInstalled`
2. if enabledPlugins was clobbered
3. treat as not-installed so the user can re-enable

这等于直接证明：

`disk / materialized existence`

和

`effective installed / enabled entitlement`

不是同一层 truth。

这条设计并不追求“更方便”，
它追求的是：

`不让 stale installation metadata 直接偷签 current entitlement。`

### 5.4 plan 线给出第二个强正例：内容回来仍可能只拿到新 identity，而不是原来的资格位

`plans.ts:239-257`
的 `copyPlanForFork()` 很清楚：

1. 复制 original content
2. 生成 `newSlug`
3. 写到 `newPlanPath`
4. 避免 original 与 forked sessions clobber

再看
`REPL.tsx:1790-1797`：

1. fork path 调 `copyPlanForFork()`
2. regular resume path 才调 `copyPlanForResume()`

这直接证明：

`内容回来`

不等于

`原身份回来`

从再赋权角度看，
这非常重要。
因为 identity 本身就是 entitlement 的一部分。

一个对象即便回来了，
也可能只被赋予：

`new seat, not old seat`

这正是 re-entitlement governance 的本体。

## 6. 哲学本质：复活是存在论问题，再赋权是规范论问题

如果把 `resurrection` 和 `re-entitlement` 抽到哲学层，
两者治理的其实不是同一类判断。

`resurrection`
首先回答的是存在论问题：

`这个历史对象能不能重新出现在当前世界。`

`re-entitlement`
回答的则是规范论问题：

`它重新出现之后，当前世界还欠不欠它原来的 seat、trust 与行动资格。`

所以这两层的根本差异是：

1. 前者判断是否在场
2. 后者判断是否仍被授权
3. 前者在防蒸发式误删
4. 后者在防回归式误信

Claude Code 在这里真正先进的地方，
不是它把 comeback 做得顺滑，
而是它已经默认承认：

`回到场内，不等于拿回原来的规范地位。`

这是一种很成熟的安全哲学：

`存在可以恢复，资格必须重签。`

## 7. 苏格拉底诘问：怎样把这一层做得更严格

### 诘问一

`如果对象已经被允许回来，为什么不让它自动恢复旧配置与旧 secrets？`

因为回来只证明：

`你重新被允许出现。`

它不证明：

`你旧时携带的 trust material 仍然可信。`

### 诘问二

`如果对象内容和 lineage 都成立，为什么还不能默认恢复 enabled truth？`

因为 enabled truth 是当前世界的积极担保，
不是历史事实的自然延伸。
它必须由当前规则重新签发。

### 诘问三

`如果 fork 时内容相同，为什么还要强行 new slug？`

因为系统要保护的不是 bytes，
而是 seat 的清晰边界。
不重新分 seat，
就容易把“复制内容”误写成“复制资格”。

### 诘问四

`那怎样才算更好的 re-entitlement governance？`

至少要继续把五件事制度化：

1. config / secrets return policy
2. enabled truth return policy
3. policy veto precedence
4. scope seat legality
5. identity fallback policy

少任何一层，
都容易把“对象回来”偷写成“原资格自动回来”。

## 8. 技术启示与下一层约束

对 stronger-request cleanup 线而言，
这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 re-entitlement。`

而是更具体地说：

1. 旧 path、旧 promise、旧 receipt 即便被允许回来，也要重新回答它们是否恢复旧 enable truth
2. 回来的对象是否继续携带旧配置与旧解释承诺，不能由 comeback 事实自动推导
3. 旧对象若以 new identity materialize，也不能再被偷写成 old seat 的自然延续
4. 更高层 policy 对 comeback 后的资格恢复仍应保留 veto 主权

只有这样，
`stronger-request cleanup resurrection-governance`
才不会继续被误写成一种过度强大的总主权：

`既负责把对象带回来，又自动负责把旧资格原封不动一并带回来。`
