# 安全提交语义：关键安全事实何时算已提交，何时只是排队、可见或可恢复

## 1. 为什么在 `33` 之后还要继续写“安全提交语义”

`33-安全单写者原则` 已经把一个更深层的问题说清楚了：

`关键安全事实必须有唯一作者。`

但如果继续往下一层追问，  
还会出现另一个同样重要的问题：

`唯一作者什么时候才算真的把这件事“写成了”？`

因为在 Claude Code 这类安全控制面里，  
“写了”至少可能有四种完全不同的含义：

1. 本地语义已经变了
2. 远端复制已经排队
3. 外部消费者已经看见
4. 恢复系统已经能重建

如果把这四种状态混成一种“已提交”，  
系统就会反过来制造新的误判：

1. 把排队误当成已落地
2. 把可见误当成可恢复
3. 把本地已改误当成跨宿主已收敛
4. 把镜像成功误当成持久化完成

所以这一章的核心问题是：

`Claude Code 到底把哪些时刻视为“已提交”，又把哪些时刻明确定义成只是排队、镜像、可见或待恢复？`

## 2. 最短结论

从源码看，Claude Code 的关键安全事实并没有一个单一的“提交瞬间”。  
它更像一个分层提交模型：

1. 语义提交
2. 复制提交
3. 可见提交
4. 持久提交
5. 恢复提交

这意味着一条更底层的判断：

`成熟的安全控制面，不只定义谁能写，还必须定义何时算真的写成。`

所以本章的核心结论是：

`Claude Code 的先进性，不只在单写者结构，还在它没有把“本地已改”“远端已见”“可恢复已落盘”混成同一种提交。`

## 3. 什么叫“安全提交语义”

这里说的“提交语义”，不是数据库术语的照搬。  
在本专题里，我会把它定义为：

`系统明确区分一条关键安全事实在不同层分别何时算本地成立、何时算对外可见、何时算远端复制完成、何时算恢复层可重建。`

也就是说，本章讨论的不是“有没有写”，  
而是：

`写入何时跨过了哪一道边界。`

## 4. 第一层提交：语义提交

## 4.1 `notifySessionStateChanged(...)` 先提交本地语义，再推动镜像

`src/utils/sessionState.ts:92-133` 很能说明问题。

这里的顺序非常清楚：

1. 先写 `currentState`
2. 立刻通知 `stateListener`
3. 再镜像 `pending_action`
4. 再清 `task_summary`
5. 最后才按条件把事件放进 SDK 队列

这说明 Claude Code 明确承认：

`语义提交先于镜像提交。`

也就是说，  
当 `currentState` 和 `stateListener` 已经更新时，  
本地语义上这件事已经成立；  
但这还不等于远端复制层、外部消费者或恢复系统都已经同步完成。

## 4.2 这也是为什么它不把镜像写成语义真相本体

同一段代码还说明：

- `pending_action` 是镜像到 `external_metadata`
- `task_summary` 是复制层清理
- `session_state_changed` 是事件流镜像

这三者都依赖语义提交，  
但都不等于语义提交本身。

所以这里的第一条硬判断是：

`语义真相先成立，镜像层随后跟进。`

## 5. 第二层提交：复制提交

## 5.1 `reportState(...)` / `reportMetadata(...)` 并不等于远端已经提交

`src/cli/transports/ccrClient.ts:645-662` 很关键。

这两处代码做的事情其实非常克制：

1. 接收语义状态与元数据
2. 调用 `workerState.enqueue(...)`
3. 不等待网络成功
4. 不把本次 enqueue 误写成“远端已落地”

这说明作者明确区分：

`排队不是提交。`

`reportState` 和 `reportMetadata` 的语义更接近：

`把复制请求送进提交流水线。`

而不是：

`宣布远端已经拥有了这份真相。`

## 5.2 `WorkerStateUploader` 进一步把“排队”和“真正提交”拆开

`src/cli/transports/WorkerStateUploader.ts:3-84` 把这层边界写得非常直白：

1. `enqueue(...)` 只是把 patch 放进 pending
2. 同时最多只有 `1 in-flight + 1 pending`
3. 重试期间新 patch 会被吸收到当前版本
4. 真正成功要等 `send(current)` 返回 `ok`

这意味着复制层里至少有三个状态：

1. 已入队
2. 正在提交
3. 远端确认成功

而系统拒绝把这三者压扁成一个“已同步”。

## 5.3 `last value wins` 其实也是提交语义的一部分

很多人会把 `last value wins` 只看成覆盖规则。  
但在这里它更像是在声明：

`pending 中的旧版本只是未完成提交的候选值，不配和最新候选值并列拥有“当前真相”地位。`

所以这里不只是单写者，  
还是：

`单提交面。`

## 6. 第三层提交：注册提交

## 6.1 `CCRClient.initialize()` 先完成 worker 注册，之后才承认恢复结果有效

`src/cli/transports/ccrClient.ts:459-525` 是本章最重要的证据之一。

这里的顺序非常值得注意：

1. 先并发发起 `getWorkerState()`
2. 再执行 `PUT /worker`
3. 只有 `PUT /worker` 成功，才设置 `currentState = 'idle'`
4. 只有在这之后，才记录 `state_restored`

注释已经把原因说得很清楚：

如果 GET 先返回，而 PUT 最后失败，  
系统不能一边说“恢复成功”，  
一边说“初始化失败”。

这说明 Claude Code 明确坚持：

`恢复读取不构成提交，成功注册后再承认恢复结果才构成有效提交。`

## 6.2 这是一种非常强的提交纪律

也就是说，  
即便旧状态已经读到了，  
只要当前 worker 还没成功注册，  
那份旧状态就还不能被正式纳入当前提交面。

这背后的哲学是：

`先确立当前作者身份，再接收旧真相。`

## 7. 第四层提交：可见提交

## 7.1 `writeEvent(...)` 把“生成”与“对外可见”明确分开

`src/cli/transports/ccrClient.ts:725-833` 说明：

1. `stream_event` 先进入 buffer
2. 非 `stream_event` 写入前会先 flush buffer
3. `close()` 会直接丢弃未 flush 的队列
4. `flush()` 才是在需要交付确认时显式推进提交

这说明对外可见性并不是在消息生成的瞬间成立，  
而是在：

1. 顺序约束满足之后
2. buffer 已冲刷之后
3. uploader 真正排空之后

才更接近“对外可见提交”。

换句话说：

`写进消息对象，不等于外部已经看见。`

## 7.2 `print.ts` 甚至把 result 的可见提交继续后移

`src/cli/print.ts:2216-2244` 与 `2408-2468` 更进一步：

1. `result` 生成后可能先被放进 `heldBackResult`
2. 只要后台 agent 仍在运行，它就不立即出流
3. 直到后台任务结束后，才真正 enqueue 到 output
4. 而 `idle` 事件又要等 `flushInternalEvents()` 完成后才发

这说明系统明确不把：

`模型已经算出 result`

等同于：

`外部现在就应该把这一轮视为已经完成。`

所以这里的第二条硬判断是：

`可见提交晚于结果生成，也晚于部分局部状态完成。`

## 7.3 `session_state_changed(idle)` 被当成“turn is over”信号，本身就是提交边界声明

`src/utils/sdkEventQueue.ts:56-61` 与 `src/utils/sessionState.ts:118-133` 直接说明：

1. `idle` 事件在 `heldBackResult` flush 之后才成立
2. 它要等后台 agent 的 drain 结束
3. 它被 SDK 消费者视为 authoritative turn-over signal

这其实已经不是普通注释，  
而是在定义：

`这一轮对外何时算真的结束。`

这就是典型的可见提交边界。

## 8. 第五层提交：持久提交

## 8.1 `writeInternalEvent(...)` 明确把“可见”与“可恢复”拆开

`src/cli/transports/ccrClient.ts:788-822` 非常关键。

它明确区分：

1. `writeEvent(...)` 是给前端看的 client event
2. `writeInternalEvent(...)` 是给恢复系统用的 internal event
3. `flushInternalEvents()` 的目标是保证 transcript 持久化

这说明 Claude Code 没有把“用户看得见”当成“恢复一定做得到”。

它明确承认：

`可见提交和持久提交是两回事。`

## 8.2 `print.ts` 在发 `idle` 之前先 flush internal events

`src/cli/print.ts:2453-2468` 则把这条纪律真正闭环：

1. `finally` 先进入 `finally_flush`
2. 先 `await structuredIO.flushInternalEvents()`
3. 再 `notifySessionStateChanged('idle')`
4. 再把 `idle` 事件 drain 到 output

这说明作者明确在维护一条顺序：

`先让恢复层可重建，再让外部世界知道这一轮结束。`

这是一种非常成熟的提交观：

`恢复能力先于完成宣告。`

## 9. 第六层提交：恢复提交

## 9.1 `hydrateFromCCRv2InternalEvents(...)` 说明恢复提交依赖内部事件而不是普通可见事件

`src/utils/sessionStorage.ts:1632-1710` 写得很清楚：

1. 恢复读取的是 internal events
2. 前景 transcript 会被重建到本地文件
3. subagent transcript 也会分别重建
4. 这条链服务的是 session resume

这说明系统认为“可恢复提交”的主要依据不是：

- UI 看到了什么
- 普通 client event 发了什么

而是：

`恢复面有没有拿到专门的内部事件账本。`

## 9.2 `print.ts` 在 resume 时还要求 transcript hydrate 与 metadata restore 并行完成

`src/cli/print.ts:5048-5060` 又补了一层纪律：

1. `hydrateFromCCRv2InternalEvents(...)` 与 `restoredWorkerState` 一起等待
2. 目的是让 SSE catchup 落在 restored state 上，而不是 fresh default 上

这说明恢复提交也不是“把 transcript 写回来”就够了。  
它至少要求：

1. 可恢复事件账本就位
2. worker metadata 也回到对应状态
3. 本地默认值不能抢先冒充当前事实

所以这里的结论是：

`恢复提交是 transcript 与 metadata 的联合提交。`

## 10. 把这些边界压成一张最小模型

我会把 Claude Code 的提交语义压成五层：

1. 语义提交：本地语义真相已经改变
2. 复制提交：远端复制请求已经成功写入
3. 可见提交：外部消费者现在应该看到它
4. 持久提交：恢复账本已经落好
5. 恢复提交：新进程已经能把它重建回来

这五层里最重要的不是名词，  
而是它们不被混写。

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只在于它有很多状态，而在于它把不同层的“提交完成”拆成了不同边界。`

这对其他 Agent 平台构建者的启示非常直接：

1. 不要把 enqueue 当成 committed
2. 不要把 visible 当成 durable
3. 不要把 local truth 当成 cross-host truth
4. 不要把 resume read 当成 current registration
5. 不要把 result ready 当成 turn finished

如果这几层不拆开，  
系统最后会出现一种很常见但很危险的假象：

`每一层都在说“已经好了”，但它们说的根本不是同一件事。`

## 12. 哲学本质

这一章更深层的哲学是：

`安全系统最危险的谎言之一，就是把不同层的“已生效”说成同一个词。`

Claude Code 的源码反过来说明：

`真正成熟的控制面，会给“已提交”加前缀。`

例如：

1. 语义上已提交
2. 输出上已提交
3. 复制上已提交
4. 持久化上已提交
5. 恢复上已提交

这不是术语洁癖，  
而是为了防止系统把尚未跨过的边界假装已经跨过。

## 13. 苏格拉底式反思：这章最容易犯什么错

### 13.1 我们是不是把状态层次写得太复杂了

有这个风险。  
但复杂并不来自分析者，  
而是来自源码本身已经明确在区分这些边界。

### 13.2 能不能把“复制提交”和“持久提交”合成一个词

不建议。  
因为 `writeEvent(...)`、`writeInternalEvent(...)`、`flushInternalEvents()` 和 resume hydrate 已经说明：

`可见给用户看的东西，和给恢复系统重建的东西，不是同一种账本。`

### 13.3 这是不是只适用于远程 CCR v2 场景

不是。  
CCR v2 把这些边界写得更清楚，  
但“本地语义”“外部可见”“恢复可重建”这三层区分，对任何严肃的 Agent 控制面都成立。

## 14. 结语

`33` 回答的是：关键安全事实为什么必须有唯一作者。  
`34` 继续推进后的结论则是：

`就算作者只有一个，系统仍必须继续回答：它什么时候才算真的写成。`

这意味着 Claude Code 更深层的安全启示之一是：

`不要只治理写权限，还要治理提交边界。`
