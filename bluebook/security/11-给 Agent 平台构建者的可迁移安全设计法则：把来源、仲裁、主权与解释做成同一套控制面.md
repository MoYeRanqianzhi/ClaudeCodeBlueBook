# 给 Agent 平台构建者的可迁移安全设计法则：把来源、仲裁、主权与解释做成同一套控制面

## 1. 为什么还要单独写这一章

前面的 `00-10` 基本已经把 Claude Code 的安全专题拆开了：

- 方法论
- 总论
- 权限
- 认证与受管环境
- 外部入口
- 哲学
- 第一性原理反思
- 状态机
- 主权矩阵
- 可解释性产品化

但如果这套蓝皮书要对平台构建者真正有用，  
最后还必须补一层：

`哪些结论只是 Claude Code 的具体实现，哪些结论其实是下一代 Agent 平台都该继承的安全设计法则？`

这一章专门做这个提炼。

## 2. 先说结论

如果把整个安全专题继续压缩，Claude Code 最值得迁移的不是某个具体模块，  
而是八条设计法则：

1. 把来源当一级安全变量
2. 把 mode 当安全语义切换，而不是 UX toggle
3. 把真正的危险定义为“绕过仲裁层”
4. 把高风险便利用更强外部边界补偿
5. 把外部能力按攻击面拆开治理
6. 把 entitlement 和传输层连通明确分开
7. 把安全解释层做成产品，而不是附录
8. 把 public build 边界与 internal capability 严格分开叙述

这八条合起来，才构成 Claude Code 安全性的真正可迁移内核。

## 3. 法则一：来源先于值

Claude Code 反复证明：

- settings source 很重要
- token source 很重要
- plugin / policy / user / project source 都很重要

这条法则的抽象形式是：

`不要只判断值危险不危险，还要判断这个值来自谁。`

如果一个系统只看值、不看来源，  
它就很难真正处理：

- 仓库污染宿主
- 用户设置污染 host 管理语义
- managed settings 与本地自定义的主权冲突

## 4. 法则二：mode 不是体验模式，而是安全模式

在很多产品里，mode 只是 UX 术语。  
Claude Code 的启示是：

- `default`
- `acceptEdits`
- `auto`
- `bypassPermissions`
- `plan`

这些 mode 的真正意义，是改变：

- 谁可以自动放行
- 谁必须经过仲裁
- 哪些规则需要先被剥离

可迁移法则就是：

`只要 mode 会改变安全语义，就不要把它设计成单纯 UI 状态。`

## 5. 法则三：真正的危险是绕过仲裁层

前面专题已经反复论证：

- 危险的不是 Bash 本身
- 危险的不是 Agent 本身
- 危险的是 `Bash(*)`、`Agent(*)` 这类让它们在仲裁前被直接放行的规则

这条法则是整套安全系统里最有价值的抽象之一：

`与其穷举危险能力，不如优先识别哪些路径绕过了本该存在的仲裁层。`

因为能力列表会变，  
但仲裁层被绕过这件事的危险性是稳定的。

## 6. 法则四：高风险便利必须由更强边界补偿

Claude Code 对 `--dangerously-skip-permissions` 的处理，非常适合被提升成一般法则：

- 允许更大胆模式
- 但要求 sandbox / no-internet / no-root 等更强环境前提

这其实是一条通用原则：

`你可以给便利，但必须用更强约束来支付便利的安全成本。`

如果没有这一层补偿，  
所谓“专家模式”往往只是“无边界模式”的委婉说法。

## 7. 法则五：外部能力不是统一扩展面，而是多种攻击面

Claude Code 对：

- MCP
- WebFetch
- hooks
- gateway

的治理方式非常值得抄。  
它没有说：

- 扩展都是插件，统一一套权限就够了

而是承认它们分别打开了：

- 上下文污染
- 网络写入
- 执行插入点
- 身份/路由漂移

因此更好的法则是：

`外部能力必须按攻击面建模，不要按功能分类建模。`

## 8. 法则六：entitlement 不等于连通，兼容不等于等价

Claude Code 非常清楚一件事：

- token 能用
- API 能回

并不自动等于：

- 第一方身份仍成立
- 高阶资格仍成立
- Remote Control entitlement 仍成立

这条法则对所有多提供商 Agent 平台都适用：

`把“调用成功”误写成“身份语义等价”，是一种系统级安全误导。`

## 9. 法则七：安全系统必须有解释层

前面的 `10` 章已经说明：

- Permission explainer
- Source display
- Managed settings security dialog
- Bridge disabled reason

这些零件的价值不只是用户体验，  
而是：

`没有解释层，安全系统会在用户侧退化成黑箱。`

因此可迁移法则是：

1. 先把安全边界做成结构
2. 再为每一层边界预留解释接口
3. 最后把解释接口产品化

## 10. 法则八：public build 边界必须显式写入设计纪律

Claude Code 的公开提取树里，  
最值得学习的自律不是某个安全功能，  
而是分析时必须持续强调：

- 哪些是 public build 可确认结构
- 哪些是 ant-only / gated / internal 留口

如果一个平台构建者自己都不区分：

- 结构留口
- 稳定公共能力

那么安全文档最终一定会变成：

- 神话
- 误导
- 错误承诺

所以这条法则其实是文档法则，也是工程法则：

`安全系统必须同时治理代码边界和叙事边界。`

## 11. 这些法则为什么能被称为“可迁移”

因为它们并不依赖于：

- Anthropic 的具体服务端
- Claude Code 的具体 UI
- 某一个特定 classifier

它们依赖的是更高层的系统事实：

1. Agent 一定会接触真实执行面
2. Agent 一定会接触多来源配置
3. Agent 一定会接触外部世界
4. Agent 一定会面对“自动化便利”和“主权边界”的冲突

只要一个平台满足这四点，  
这些法则就仍然成立。

## 12. 从第一性原理再压缩一次

如果还要再压缩，我会把这八条法则进一步压成三条元法则。

### 元法则一：先问谁拥有主权

无论是 mode、settings、hooks、gateway 还是 entitlement，  
第一问都不应是“它怎么配”，  
而应是：

`谁有资格定义它。`

### 元法则二：先问哪一层在仲裁

真正危险的不是动作本身，  
而是动作绕过了哪一层本该存在的仲裁。

### 元法则三：先问用户是否看得见当前边界

如果用户看不见：

- 当前是谁在管
- 当前为什么被拒
- 当前什么是 managed-only
- 当前什么只是兼容不是等价

那么系统即使结构正确，也仍然会被体验成不可信。

## 13. 用苏格拉底式追问收束

### 问：为什么这一章不再继续谈具体源码文件？

答：因为前面的十章已经给出足够多的结构证据；这一章的任务不是再举例，而是把例子压成可迁移法则。

### 问：为什么“来源优先”能排在第一条？

答：因为很多真实安全失控，不是能力本身危险，而是危险能力或危险配置来自了错误来源。

### 问：为什么“绕过仲裁层”比“危险工具名单”更重要？

答：因为名单永远会过时，而仲裁层是否被绕过是更稳定的上位抽象。

### 问：为什么“解释层”要被放进安全法则，而不是 UX 法则？

答：因为不解释的安全最终只会制造不信任，而不信任会反过来诱发规避、误操作和错误支持路径。

### 问：为什么这一章要把“public build 边界”也算进安全设计？

答：因为错误叙事本身就会伤害安全决策。对能力边界的误解，会直接导致错误配置、错误信任和错误依赖。

## 14. 一句话总结

Claude Code 给 Agent 平台构建者最值得迁移的安全启示，不是某个具体沙箱或分类器实现，而是一整套“来源优先、mode 即语义、绕过仲裁才是真危险、外部能力按攻击面治理、兼容不等于等价、解释层必须产品化、叙事边界必须与代码边界同时治理”的设计法则；真正成熟的安全平台，不只是能挡住风险，还能清楚说明当前到底是谁在定义边界。
