# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `356` 之后还必须继续写 `357`

`356-安全载体家族强请求清理迁移治理与强请求清理退役治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样过桥、怎样退役，也仍需要 sunset governor 去决定旧 law 从什么时候起不再算 current truth。`

但如果继续往下追问，
还会撞上一层更隐蔽、也更容易被偷写成“会退役就等于会把后事管好”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界结束，它就自动拥有决定结束后还该留下什么最小残留事实的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 自己已经落地的成熟机制，
会发现它在多个子系统里都把：

`old world 何时结束`

和

`结束之后还要留下什么最小机器可消费事实，继续约束未来解释`

拆成了两层不同治理。

最硬的证据至少有七组：

1. `query.ts` 遇到 orphaned assistant messages 时不会只静默丢弃旧对象，而会显式 `yield { type: 'tombstone' }`  
   `src/query.ts:712-719`
2. `utils/messages.ts` 与 `QueryEngine.ts` 证明 `tombstone` 不是普通内容，而是后续消费者必须识别的 control signal  
   `src/utils/messages.ts:2954-2957`  
   `src/QueryEngine.ts:757-760`
3. `REPL.tsx` 与 `sessionStorage.ts` 证明 tombstone 会继续驱动 UI 与 transcript 删除链，而且这条链还需要专门的 rewrite 上限来保护资源边界  
   `src/screens/REPL.tsx:2646-2648`  
   `src/utils/sessionStorage.ts:121-123,925-945,1470-1473`
4. `cacheUtils.ts` 与 `pluginOperations.ts` 证明 `.orphaned_at` 不是 sunset 时点本身，而是对象退役后留在磁盘上的最小 marker  
   `src/utils/plugins/cacheUtils.ts:23-24,149-176`  
   `src/services/plugins/pluginOperations.ts:523-547`
5. `orphanedPluginFilter.ts` 与 `main.tsx` 进一步证明 marker 还会被翻译成 grep / glob exclusion grammar，继续约束 future visibility world  
   `src/utils/plugins/orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`  
   `src/main.tsx:2544-2568`
6. `config.ts` 与 `useModelMigrationNotifications.tsx` 说明对象世界已经变化之后，系统有时仍要保留一个最小历史注脚，向当前 runtime 解释“为什么现在变了”  
   `src/utils/config.ts:427-438`  
   `src/hooks/notifs/useModelMigrationNotifications.tsx:5-8,21-33,49-50`
7. `pluginOperations.ts` 甚至直接把 `tombstones` 当成 teardown fallout 来讨论，公开承认 post-retirement residue 不是副现象，而是受治理对象  
   `src/services/plugins/pluginOperations.ts:543-545`

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request cleanup 世界从什么时候起不再算 current truth。`

后者才配说：

`旧世界结束之后，还要留下什么最小残留事实，让后来的 runtime、搜索面、恢复链与审计者不会把“已经退役”重新读成“从未发生”或“仍可继续援引”。`

所以 `356` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责宣布结束；stronger-request cleanup tombstone governor 才负责治理结束之后留下的最小残留事实。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开 tombstone manager，
而是在说：

1. repo 在 message、plugin 与 model 线已经展示出真实的 tombstone-like grammar
2. 这些 grammar 不等于 sunset deadline 本身
3. 它们承担的是 post-retirement minimal residue governance

第二条：

这里说的 `tombstone governance`

不是在声称：

`退役之后一定要保留完整旧对象。`

更稳的说法只能是：

`墓碑治理回答的是退役后保留多少最小事实，既足以约束未来解释，又不把旧对象重新抬回 current world。`

第三条：

这里也不是把：

`migration timestamp`

一概硬写成字面 tombstone。

当前更稳的说法只能是：

`有些残留物虽然源码里不直接叫 tombstone，但功能上承担了 post-transition trace artifact 的角色，因此属于 tombstone-like marker。`

第四条：

这里尤其不能过度声称：

`marker 还在`

就等于：

`旧世界仍然有效。`

更稳的说法只能是：

`marker presence` 说明系统还在保留“旧世界已经失效”这一最小事实，而不是在恢复旧世界的 current validity。`

第五条：

这里也不能把：

`warning surface`

偷换成：

`完整 tombstone grammar`。

更稳的说法只能是：

`warning` 主要面向人类解释，而 tombstone 还要继续回答 marker type、marker consumer、future effect 与 final cleanup responsibility。`

第六条：

这里还不能过度声称：

`Claude Code 既然已经在 message/plugin/model 线展示 tombstone grammar，stronger-request cleanup 线未来就必须逐字照抄。`

更稳的说法只能是：

`不同对象家族可以有不同最小残留形式，但“谁来决定退役后留下什么最小机器可消费事实”必须被正式回答；不能因为 marker 形式可变，就把 tombstone 主权偷写成 sunset 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer`

证据：

1. `query.ts` 遇到 orphaned assistant messages 时会显式产出 `tombstone`  
   `query.ts:712-719`
2. `utils/messages.ts` 与 `QueryEngine.ts` 证明 tombstone 是 control signal，而不是普通消息内容  
   `utils/messages.ts:2954-2957`; `QueryEngine.ts:757-760`
3. `REPL.tsx` 与 `sessionStorage.ts` 证明 tombstone 会继续驱动显示层与 transcript 存储层改口，而且 slow-path rewrite 还受专门上限控制  
   `REPL.tsx:2646-2648`; `sessionStorage.ts:121-123,925-945,1470-1473`
4. `.orphaned_at` 不是 sunset 时点本身，而是对象退役后留在磁盘上的最小 marker  
   `cacheUtils.ts:23-24,149-176`; `pluginOperations.ts:523-547`
5. marker 还会继续塑造 future search / visibility world  
   `orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`; `main.tsx:2544-2568`
6. migration timestamp 证明对象世界变化之后，系统有时还要保留一个最小历史注脚去解释“为什么当前世界刚变”  
   `config.ts:427-438`; `useModelMigrationNotifications.tsx:5-8,21-33,49-50`
7. stronger-request cleanup 当前虽已开始显露 sunset-governance，却还没有谁正式决定旧 startup wording、旧 home-root cleanup path、旧 promise vocabulary 与旧 receipt semantics 退役后还应留下什么最小 marker

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留事实来约束未来解释。`

再压成一句：

`会退役，不等于会留下可治理的最小残留事实。`

再压成更哲学一点的话：

`制度知道何时关门，不等于制度知道门后还要挂什么告示。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
这两层治理处理的是两个不同真相面的主权问题。

`sunset governor`

回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

`tombstone governor`

回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. marker 将怎样约束未来读取、恢复、审计与解释
5. marker 何时再被进一步清除，谁对这次二次清除负责

所以这两层的哲学本质不同：

`sunset`

决定旧对象什么时候停止作为 current truth；

`tombstone`

则决定它停止之后，系统还保留什么最小事实来约束后来者。

Claude Code 在这里真正先进的地方，
不是它“会在对象无效后把它删掉”，
而是它已经明确承认：

`删除不是全部，最小残留事实同样需要被制度化。`

这是一种很成熟的安全哲学：

`对象停止有效，不等于它留下的解释义务也随之消失。`

## 5. 从技术角度看，Claude Code 的先进性在于把“退出真相面后的最小残留”做成可继续消费的制度，而不是把删除当唯一答案

### 5.1 message 线：对象退役之后，系统需要显式 tombstone 来告诉后续消费者“该删谁”

`query.ts:712-719`
最关键的事实是：
当 streaming fallback 产生 orphaned messages 时，
系统不是只把旧消息从内存数组里粗暴过滤掉。

它会：

1. 为每个 orphaned message 产出 `type: 'tombstone'`
2. 记录 `tengu_orphaned_messages_tombstoned`
3. 清空与旧尝试关联的临时结构

`utils/messages.ts:2954-2957`
进一步说明：
stream handler 遇到 tombstone 时不当内容渲染，而是走 `onTombstone` 分支。

`QueryEngine.ts:757-760`
则继续把 tombstone 明写成：

`control signals for removing messages`

这意味着在 message 线里：

`对象结束`

和

`留下一个最小控制信号去驱动正确删除`

根本不是同一件事。

### 5.2 transcript 线：墓碑不是一句说明，而是驱动 UI / storage 改口的正式控制面

`REPL.tsx:2646-2648`
收到 tombstoned message 时，
会立刻：

1. 从当前消息列表中过滤该消息
2. 调 `removeTranscriptMessage(tombstonedMessage.uuid)`

而
`sessionStorage.ts:1470-1473`
则明确把这件事定义成：

`Used when a tombstone is received for an orphaned message`

更进一步，
`sessionStorage.ts:925-945`
专门为 tombstone removal 维护 slow-path rewrite，
而
`sessionStorage.ts:121-123`
还要用 `MAX_TOMBSTONE_REWRITE_BYTES`
避免 tombstone rewrite 把超大 transcript 文件拖进 OOM。

这说明 tombstone 的语义不是：

`写一条历史说明，给人随便看看。`

而是：

`给系统一个继续消费的控制信号，让后续存储与显示世界一起改口。`

### 5.3 plugin 线：`.orphaned_at` 不是垃圾残留，而是会继续塑造 future visibility world 的机器墓碑

`cacheUtils.ts:23-24,149-176`
与
`pluginOperations.ts:523-547`
表明：

1. 插件被卸载或更新到新版本后，旧版本先被打上 `.orphaned_at`
2. orphan marker 先进入 7 天 grace world，再决定是否物理删除
3. dependency teardown 还要显式避免制造新的 tombstone fallout

更强的一层来自
`orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
与
`main.tsx:2544-2568`：

1. 系统会基于 `.orphaned_at` 生成 grep / glob exclusion grammar
2. exclusion cache 会在 session 里冻结，直到显式 `/reload-plugins`
3. 也就是说 marker 不只记史，还继续决定 future search / visibility world

这就是 Claude Code 的技术先进性所在：

`它不只知道旧对象已经死了，还知道死亡之后哪些读取世界必须跟着改口。`

### 5.4 model 线：不是所有墓碑都长成 tombstone message，但 trace artifact 仍属于 post-retirement residue

`config.ts:427-438`
定义了：

1. `opusProMigrationTimestamp`
2. `legacyOpusMigrationTimestamp`
3. `sonnet45To46MigrationTimestamp`

这些字段都不是迁移动作本身，
也不是 retirement date 本身。
但
`useModelMigrationNotifications.tsx:5-8,21-33,49-50`
继续消费它们来判断：

1. 这次启动是否刚刚发生过模型迁移
2. 当前 runtime 是否需要一次性的历史解释

这说明有些残留物虽然源码里不直接叫 tombstone，
但功能上确实承担了 tombstone-like marker 的职责：

`它们是变化已经发生过的最小历史事实，用于向当前系统解释为什么现在变了。`

## 6. 对 stronger-request cleanup 设计的四条直接启示

1. 不能只写 sunset clock，还要单独写 post-retirement marker grammar。
2. 不能只写“删不删”，还要写 marker type、consumer set、future effect 与 second-stage cleanup responsibility。
3. 不能只给人类 warning，还要给机器可消费的最小残留事实，否则未来读取、恢复与审计会重新发明各自版本的“墓碑”。
4. 不能把 resurrection 提前偷写进 tombstone；`留下残留事实` 与 `允许对象回来` 必须继续拆层。

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界已经正式结束”误认成了“旧世界结束后的一切解释义务也结束了”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 sunset time 已经明确，为什么还要再拆 tombstone？  
   因为知道何时结束，不等于知道结束后还要留下什么最小事实。
2. 如果对象已经不再有效，为什么不直接删干净？  
   因为未来读取、恢复、审计与解释仍可能需要一个最小 marker 防止误复活或误叙述。
3. 如果 marker 还在，是不是说明旧世界仍有效？  
   不是。marker 正是在表达“旧世界已经失效”这一最小事实。
4. 如果 warning 或 timestamp 已经解释过一次，是否就不再需要 tombstone grammar？  
   不够。warning 面向人类，tombstone 还必须面向机器后续消费。
5. 如果 `.orphaned_at` 这类 marker 只存在于 plugin 线，cleanup 线是不是可以不需要？  
   不能这样推。不同 marker 形式可以不同，但“谁来签 post-retirement residue”必须被回答。
6. 如果 cleanup 线还没有显式 tombstone 代码，是不是说明这层还不值得写？  
   恰恰相反。越是缺这层明确 grammar，越容易在未来把退役后的残留义务偷写成“反正已经结束了”。

这一串反问最终逼出一句更稳的判断：

`tombstone 的关键，不在系统会不会保留一点旧痕迹，而在系统能不能正式决定哪些最小残留事实必须留下来继续约束后来者。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 sunset governance，就已经足够成熟。`

而是：

`Claude Code 在 message tombstone、plugin orphan marker、marker-driven visibility control 与 migration timestamp trace artifact 上已经明确展示了 tombstone governance 的存在；因此 artifact-family cleanup stronger-request cleanup-sunset-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁来宣布旧世界结束”，而是“谁来决定结束后还留下什么最小机器可消费事实”。`
