# 外部入口风险分级矩阵：MCP、WebFetch、hooks、gateway 分别在打开哪一种攻击面

## 1. 为什么还要单独写这一章

前面的 `04` 章已经说明：

- 外部世界不是默认可信上下文
- MCP、WebFetch、hooks、gateway 需要分别收口

但如果只停在这一步，  
还不够支持更精细的设计和选型。  
还必须继续追问：

`这些外部入口到底分别在打开哪一种攻击面？`

因为如果不把风险面拆开，  
讨论就会退化成一句没用的话：

- 外部能力有风险

这一章专门把它压成一张风险分级矩阵。

## 2. 先说结论

Claude Code 对外部入口的真正成熟之处，在于它并没有把所有外部能力都视为同一种风险。

更准确地说，不同入口主要打开的是不同的五种风险：

1. 读取风险
2. 写入 / 外流风险
3. 执行风险
4. 身份 / 路由风险
5. 上下文污染与语义漂移风险

而 Claude Code 的收口策略，恰恰是围绕这五类风险分别布置的。

## 3. 先把五种风险定义清楚

### 3.1 读取风险

也就是：

- 外部世界能否把信息带进会话
- 带进来的是不是超出预期范围

### 3.2 写入 / 外流风险

也就是：

- 系统能否对外发出更高风险动作
- 包括上传、POST、隐式写入、远端副作用

### 3.3 执行风险

也就是：

- 外部入口是否能直接或间接执行代码、命令或脚本

### 3.4 身份 / 路由风险

也就是：

- 入口是否能改变上游身份语义、provider 路由或认证解释

### 3.5 上下文污染与语义漂移风险

也就是：

- 外部结果虽然“成功进入”，但因为输出过大、协议不完整或语义不等价，让后续推理建立在错误前提上

## 4. 一张压缩后的风险分级矩阵

| 入口 | 读取风险 | 写入/外流风险 | 执行风险 | 身份/路由风险 | 上下文污染/语义漂移风险 |
| --- | --- | --- | --- | --- | --- |
| MCP | 高 | 中 | 中到高 | 低 | 高 |
| WebFetch | 中 | 中到高 | 低 | 低 | 中 |
| hooks | 低 | 中 | 高 | 低到中 | 中 |
| gateway / proxy | 中 | 中 | 低 | 高 | 高 |

这张表最重要的意义不是“谁最危险”，  
而是：

`它们危险的方式根本不同。`

## 5. MCP 为什么最像“上下文与工具面双高风险入口”

`mcpValidation.ts`、`toolSearch.ts` 与 `QueryEngine.ts` 一起看，  
MCP 最主要的风险不是“插件多”，  
而是：

1. 它能把大量外部结果注入主上下文
2. 它还可能把新工具面带进系统
3. 它的输出若不被限制，会直接改变后续决策质量

因此 MCP 的最高风险主要在两类：

- 上下文污染
- 工具面扩张

Claude Code 的收口方式也正对应这两点：

- `mcpValidation.ts` 限制输出预算
- `toolSearch.ts` 对代理与 beta 语义保持保守

## 6. WebFetch 为什么主要不是“执行风险”，而是“读写语义混淆风险”

`WebFetchTool/preapproved.ts` 把问题说得非常清楚：

- 某些站点虽然是文档或代码站点
- 但它们同样可能允许上传

所以 WebFetch 的核心风险不是“执行本地命令”，  
而是：

`用户容易把“允许读取某域名”误解成“可以对这个域名做任意网络动作”。`

Claude Code 的成熟点在于它没有犯这个错误。  
它刻意分离了：

- 预批准 GET
- sandbox 网络权限

这就是在把读取风险和写入/外流风险拆开治理。

## 7. hooks 为什么是典型的高执行风险入口

hooks 的核心不是上下文注入，  
而是：

- 额外执行点
- 隐形执行点
- 主权冲突执行点

`hooksConfigSnapshot.ts` 显示：

- managed settings 可限定只跑 managed hooks
- 也可完全禁用 hooks
- 快照与更新机制还在防“隐藏修改”

这说明 hooks 在 Claude Code 里的真正风险标签应当是：

`高执行风险 + 中等上下文污染风险。`

因为它最危险的不是“带来文字”，  
而是：

`带来额外动作。`

## 8. gateway / proxy 为什么是最高的身份与语义风险入口

gateway / proxy 表面上看只是“换个 base URL”，  
但从：

- `auth.ts`
- `providers.ts`
- `bridgeEnabled.ts`
- `toolSearch.ts`

连起来看，它真正打开的是两种高风险：

1. 身份 / 路由风险
2. 语义漂移风险

原因在于：

- 请求虽然能发出
- 但第一方 OAuth、provider entitlement、beta 形态支持与高阶能力语义可能已经变化

因此 gateway 在安全上最危险的地方，不是“能不能联通”，  
而是：

`用户可能在不自知的情况下，以为自己仍处于原来的身份和能力语义里。`

## 9. 为什么 hooks 和 gateway 不能被归到同一类

很多人会把它们都叫“扩展”或“外部能力”。  
但从风险结构看，它们完全不是同一种东西。

### hooks 更像：

- 本地执行插入点
- 执行动作与主权冲突面

### gateway 更像：

- 上游身份与协议语义重写点
- 路由与解释层漂移面

也就是说：

`hooks 是执行主权风险，gateway 是身份语义风险。`

## 10. 为什么 MCP 和 WebFetch 也不能混成同一类

它们都能“带外部信息进来”，  
但重点不同。

### MCP 的重点：

- 上下文规模
- 工具扩张
- 输出结构与协议兼容

### WebFetch 的重点：

- 域名边界
- 读取与写入语义分离
- 网络副作用约束

所以：

`MCP 更像高权限结构化入口，WebFetch 更像受限网络读取入口。`

## 11. Claude Code 的收口策略为什么体现了先进性

它至少做对了四件事：

1. 不用同一把锁处理所有外部入口
2. 对每个入口都抓住了最核心的风险维度
3. 允许功能存在，但把高风险语义单独抬高优先级
4. 把“语义漂移”也当作安全问题，而不是只把“直接破坏”当问题

第四点尤其重要。  
很多系统能防直接攻击，  
但防不了：

- 假成功
- 假等价
- 假兼容

Claude Code 至少已经意识到这一点。

## 12. 从第一性原理看，外部入口治理真正要回答什么

如果从第一性原理压缩，任何外部入口治理其实都在回答五个问题：

1. 外部世界能带什么进来
2. 外部世界能让系统带什么出去
3. 这个入口能不能触发执行
4. 这个入口会不会改变身份或上游语义
5. 这个入口会不会让模型在错误前提上继续推理

Claude Code 的价值不只是“做了限制”，  
而是它已经开始按这五个问题分别建模。

## 13. 对平台构建者的技术启示

如果要从这一章提炼方法学，最值得留下六条：

1. 先按风险维度拆入口，不要先按功能名拆入口
2. 读取风险和写入风险必须分开治理
3. 执行风险要和上下文污染风险分开治理
4. 身份与路由漂移本身就是安全问题
5. “协议不完整但还能跑”要被当作风险，而不是兼容成功
6. 风险矩阵应直接指导产品层的权限与解释设计

## 14. 一句话总结

Claude Code 对外部入口的安全成熟度，不在于它把 MCP、WebFetch、hooks、gateway 一概视为“危险扩展”，而在于它清楚这些入口分别打开的是读取、写入、执行、身份和上下文污染等不同攻击面，因此收口策略也必须按风险维度而不是按功能标签分别构造。
