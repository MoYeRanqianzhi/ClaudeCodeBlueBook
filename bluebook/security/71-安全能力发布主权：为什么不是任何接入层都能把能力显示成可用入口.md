# 安全能力发布主权：为什么不是任何接入层都能把能力显示成可用入口

## 1. 为什么在 `70` 之后还要继续写“能力发布主权”

`70-安全多重窄门` 已经回答了：

`能力必须连续穿过环境门、信任门、内容门、决议门与发布门，才会被借出。`

但如果继续追问，  
还会出现一个更贴近产品表面、却同样关键的问题：

`到底谁有资格把一项能力显示成“现在可用”？`

这不是纯 UI 问题。  
因为在 Claude Code 这类系统里，  
一旦某个入口被显示成可用，  
用户会自然读出三层含义：

1. 这项能力存在
2. 这项能力当前可达
3. 这项能力当前由这个表面合法承载

所以“可见入口”本身就是一种承诺。  
如果错误层级把能力过早显示出来，  
那不是普通交互瑕疵，  
而是：

`安全承诺越权。`

因此在 `70` 之后，  
安全专题必须再补一条更高阶原则：

`能力发布主权。`

也就是：

`不是任何接入层、任何宿主、任何 adapter 都有资格把能力发布成可见入口；只有掌握对应 gate 结果的 authoritative 层，才有资格把“可用”变成前台事实。`

## 2. 最短结论

从源码看，Claude Code 已经在多个位置把“谁有资格发布能力可见性”写成了硬规则：

1. REPL bridge 发给 remote 的 `system/init` 会主动删去 `tools`、`mcpClients` 与 `plugins`，只发布 bridge-safe commands  
   `src/hooks/useReplBridge.tsx:297-318`
2. `system/init` 本身就是远端 picker 和 gate UI 的能力发布载体，因此上游传什么、下游就会承诺什么  
   `src/utils/messages/systemInit.ts:41-53,57-85`
3. remote 模式在真正连接前就先 `filterCommandsForRemoteMode()`，避免本地命令短暂暴露  
   `src/main.tsx:3337-3345,3484-3492`  
   `src/commands.ts:678-686`
4. REPL 收到远端 init 后，还会再按远端实际宣告的 `slash_commands` 把本地命令继续收窄  
   `src/screens/REPL.tsx:1379-1384`
5. project MCP server 只有 `approved` 才进入最终 config merge，因此“写进项目配置”不自动转化为“前台可见能力”  
   `src/services/mcp/utils.ts:351-372`  
   `src/services/mcp/config.ts:1164-1188,1231-1238`
6. channel capability 只有在 approved allowlist 条件下才会出现在 schema / output 中，IDE 才据此显示 enable prompt  
   `src/entrypoints/sdk/coreSchemas.ts:210-217`  
   `src/cli/print.ts:1666-1676`

所以这一章的最短结论是：

`能力可见性不是观测结果，而是受主权约束的发布动作。`

我把它再压成一句：

`谁发布可用入口，谁就在替系统作安全承诺。`

## 3. 源码已经说明：Claude Code 把“可见入口”当成安全对象在管理

## 3.1 remote 看到什么，不由 remote 自己决定，而由 bridge-authoritative init 决定

`src/utils/messages/systemInit.ts:41-53` 明确说明：

`system/init`

是远端 client 用来渲染 picker 和 gate UI 的第一条能力发布消息。

而 `src/hooks/useReplBridge.tsx:297-318` 更关键。  
这里 bridge 在发送 `system/init` 时不是简单转发本地全量状态，  
而是主动做了三层删减：

1. `tools: []`
2. `mcpClients: []`
3. `plugins: []`

同时只发布：

`commands: commandsRef.current.filter(isBridgeSafeCommand)`

注释写得非常直白：

1. MCP-prefixed tool names 会泄露已接入的 integration
2. plugin paths 会泄露 raw filesystem path
3. remote client 只能调用 bridge-safe commands

这说明 bridge 在这里做的不是“UI 优化”，  
而是：

`能力发布主权裁决。`

也就是说，  
哪怕本地真实拥有更多能力，  
remote 也没有资格因为“那边确实存在”就直接看见。

## 3.2 预过滤比事后报错更高级：remote mode 一开始就不该看见本地不安全命令

`src/commands.ts:678-686` 定义了：

`filterCommandsForRemoteMode(commands)`

它的用途也写得很清楚：

`preventing local-only commands from being briefly available before the CCR init message arrives`

`src/main.tsx:3337-3345` 与 `3484-3492` 则进一步说明：  
进入 remote mode 时，  
命令列表在真正 launch REPL 前就已经被收窄成：

`remoteCommands`

这非常重要。

因为如果系统先把全量命令展示出来，  
再在稍后初始化里说“这些其实不能用”，  
那短暂暴露本身已经构成了错误承诺。

Claude Code 显然不接受这种模式。  
它宁可前置删减，  
也不让“本来就不该拥有的入口”短暂出现。

这条设计可以压成一句：

`错误的入口，不应先出现再撤回。`

## 3.3 remote init 到达后，REPL 还要再用远端实际宣告做第二次收窄

`src/screens/REPL.tsx:1379-1384` 说明：

当 REPL 收到远端 init 后，  
本地会：

1. 读取 `remoteSlashCommands`
2. 建立 `remoteCommandSet`
3. 只保留 `remoteCommandSet` 中存在的命令
4. 以及本地 `REMOTE_SAFE_COMMANDS`

这说明 capability publish sovereignty 并不是一次性动作，  
而是一条持续校正链：

1. 本地先按 remote-safe 预过滤
2. 远端 authoritative init 再回写一份“当前真实发布集”
3. REPL 再按这份发布集继续删减

这套设计表达的是一个很强的哲学：

`可见入口应服从最窄、最新、最 authoritative 的发布集。`

而不是服从“本地推测它也许可用”。

## 3.4 project MCP server 的存在，不自动转化为前台可见能力

`src/services/mcp/utils.ts:351-372` 把 project MCP server 状态正式分成：

1. `approved`
2. `rejected`
3. `pending`

`src/services/mcp/config.ts:1164-1188` 又进一步说明：

只有：

`approvedProjectServers`

才会被并入后续 `enabledManualServers` 计算链。

同文件 `1231-1238` 最终 merge 时也只合并：

`approvedProjectServers`

这说明 project config 的语义不是：

`声明即发布`

而是：

`声明 -> 审批 -> 才能进入可连接、可呈现、可交互的能力集合`

因此 project MCP server 的“是否可见”  
也不属于普通读取权限，  
而属于：

`能力发布主权`

的范围。

## 3.5 schema 里 capability 字段的出现本身，就是发布许可的一部分

`src/entrypoints/sdk/coreSchemas.ts:210-217` 有一条很容易被忽略、但非常有力量的描述：

`experimental['claude/channel'] is only present if the server's plugin is on the approved channels allowlist`

也就是说，  
schema 并不是无条件映射底层对象全部 capability。  
某个 capability 是否“出现在 schema 里”，  
本身就已经被 allowlist 结果收口了一层。

这意味着：

`能力字段的可见性，也是一种授权动作。`

只有当系统确认某个 channel server 已获批准，  
对应 capability 才会被描述成“现在可见、现在可被前台利用”。

## 3.6 CLI/IDE 侧的 prompt 可见性，同样服从上游发布主权

`src/cli/print.ts:1666-1676` 很值得写进这一章。

这里作者明确写到：

1. IDE 会读取 `experimental['claude/channel']`
2. 以决定是否显示 Enable-channel prompt
3. 但这个 capability 在输出前就先做 allowlist pre-filter
4. 目的就是避免 dead buttons

注释甚至直接说：

`Not a security boundary ... just avoids dead buttons.`

这句话表面上在降级，  
其实更能说明设计层次：

1. 真正的 security boundary 仍在 handler 内部
2. 但“要不要把按钮显示出来”本身已经要服从发布主权

因为一个不该出现的 dead button，  
虽然未必立刻越权，  
却已经在替系统说出一个错误承诺：

`你现在似乎可以启用这项能力。`

成熟系统连这种“伪承诺按钮”都要尽量消除。

## 3.7 第一性原理：入口不是观察值，而是承诺值

如果从第一性原理追问：

`为什么能力可见性要被如此严格地管控？`

因为在安全系统里，  
入口从来不是普通观察值，  
而是：

`承诺值。`

一个入口一旦出现在用户面前，  
用户默认读到的不是：

`底层某处也许存在这个对象`

而是：

`系统现在允许我通过这个表面去触碰这段能力`

所以 capability publish sovereignty 的第一性原理可以压成一句：

`visibility is a promise.`

进一步展开，就是：

1. 不是所有存在的能力都该被显示
2. 不是所有可调用的对象都该被宣传成入口
3. 不是所有 adapter 都配替上游主权层对用户作承诺

## 4. 苏格拉底式自问：为什么不能让每个宿主各自决定显示什么入口

### 4.1 如果某项能力底层确实存在，为什么不直接显示

因为“存在”不等于“当前可由此表面合法承载”。

remote client 可能没有本地 Ink UI；  
channel server 可能没进 allowlist；  
project MCP server 可能还在 pending；  
bridge 也可能故意不暴露 tools / plugins / mcpClients。

存在只是对象事实，  
可见入口则是主权事实。

### 4.2 如果显示了但点了会被拒绝，问题有多大

问题很大。

因为安全系统不只管理“最终是否成功”，  
也管理：

`用户对系统边界的正确心智模型。`

先显示、后拒绝，  
会把边界教育成本转嫁给用户反复试错。

### 4.3 预过滤会不会让系统显得过于保守

会更克制，  
但不会更不准确。

真正危险的不是“少显示了一点能力”，  
而是：

`多显示了一项本不该由此表面承诺的能力。`

### 4.4 这和上一章的“多重窄门”有什么区别

`70` 讨论的是：

`能力要经过哪些门才会被借出`

而这一章讨论的是：

`谁有资格把借出的结果宣布成前台入口`

前者是放行链，  
后者是发布权。

## 5. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于它会拦截危险动作，还在于它把“能力是否可见”也纳入安全主权治理。`

它先进的地方有六点：

1. 用 `system/init` 把 capability publish 做成正式协议对象
2. remote mode 在连接前就做预过滤，不允许短暂过度暴露
3. bridge init 会主动删去高泄露、高误导的能力字段
4. REPL 收到 authoritative remote init 后还会再收窄一次本地入口
5. project MCP server 必须先审批，才能从配置事实升级为可见能力
6. schema / print 层都会对 capability 字段做前置 allowlist 收口

对其他 Agent 平台构建者的直接启示有六条：

1. 把 capability publish 当成独立安全层，而不是 UI 层细节
2. 让“入口可见”服从 authoritative publish set，而不是底层对象存在性
3. 远端宿主必须默认看见能力子集，而不是完整能力面
4. 所有 capability field 的暴露条件都要显式写进协议或 schema
5. dead button 不是小问题，它是错误承诺的前端症状
6. 设计评审时必须单独问：这项能力由谁发布、谁有权撤回、谁无权自报

## 6. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正危险的不是某项能力底层存在，  
而是：

`错误层级提前把它发布成了“现在可用”的前台入口。`

所以比“多重窄门”更深一层的原则是：

`能力可见性本身就是安全主权，不是任何接入层都能随便宣布。`
