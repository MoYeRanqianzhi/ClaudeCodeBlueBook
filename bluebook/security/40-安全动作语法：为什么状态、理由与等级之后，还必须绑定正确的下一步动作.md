# 安全动作语法：为什么状态、理由与等级之后，还必须绑定正确的下一步动作

## 1. 为什么在 `39` 之后还要继续写“动作语法”

`39-安全声明等级` 已经回答了：

`成熟控制面不能只说 yes/no，而要把状态、理由和强度分层编码。`

但如果继续往第一性原理再压一层，  
还会出现一个更苛刻的问题：

`系统说完之后，下一步到底该让谁做什么？`

换句话说，  
如果系统已经给出：

1. `needs-auth`
2. `pending`
3. `review_ready`
4. `allowed_warning`
5. `canRewind = false`

那它到底是在输出：

1. 一条静态描述
2. 一条诊断结论
3. 还是一条已经隐含下一步动作的控制指令

Claude Code 的源码更接近第三种。  
它反复在做的，不只是把结论分级，  
而是把：

`结论和正确下一步动作绑定成同一条协议语句。`

所以这一章的核心问题是：

`Claude Code 为什么不满足于“说清楚状态”，而还要把状态、理由、等级继续压成动作语法？`

## 2. 最短结论

从源码看，Claude Code 的安全控制面依赖的不是孤立状态，  
而是一套更完整的输出纪律：

`重要安全声明必须同时告诉宿主和用户：现在是什么、为什么如此、下一步该做什么。`

这至少体现在五类地方：

1. 权限请求会直接携带 `permission_suggestions`
2. turn 摘要会同时给出 `recent_action`、`needs_action` 和 `artifact_urls`
3. 回滚结果会明确区分 `canRewind`、`error` 和 `filesChanged`
4. 连接与认证状态会把恢复路径拆成 `needs-auth`、`pending`、`failed`、`disabled`
5. 校验和初始化失败会给出 typed reason 或面向动作的 message

所以这一章的最短结论是：

`Claude Code 的成熟之处，不只是让控制面会分级说话，还在于它让声明天然带着正确动作路径。`

## 3. 什么叫“安全动作语法”

这里说的“动作语法”，不是 UI 按钮文案。  
我在这一章里的定义是：

`系统把一个安全结论与其对应的正确下一步动作，编码成同一个可传播、可消费、可审计的结构化输出。`

一个成熟的动作语法至少要回答四件事：

1. 当前结论是什么
2. 为什么是这个结论
3. 哪条动作是正确下一步
4. 哪条看似合理的动作其实不该做

如果缺最后两条，  
那再精细的声明等级也容易退化成另一种失败：

`系统说得很细，但用户和宿主还是做错了下一步。`

## 4. 第一类动作语法：权限请求不是“拦住”，而是附带建议修复路径

## 4.1 `PermissionUpdateSchema` 本身就是动作集合，而不是注释集合

`src/entrypoints/sdk/coreSchemas.ts:263-299` 很关键。  
这里 `PermissionUpdateSchema` 不是模糊提示，  
而是明确枚举了：

1. `addRules`
2. `replaceRules`
3. `removeRules`
4. `setMode`
5. `addDirectories`
6. `removeDirectories`

这说明权限系统并不是只想说：

`当前不行。`

它更想说的是：

`如果你要继续，允许的修正动作属于这几类。`

## 4.2 `SDKControlPermissionRequestSchema` 把结论与动作提示直接绑在一起

`src/entrypoints/sdk/controlSchemas.ts:106-121` 进一步说明了这一点。  
`can_use_tool` request 除了工具名和输入之外，还主动携带：

1. `permission_suggestions`
2. `blocked_path`
3. `decision_reason`
4. `title`
5. `display_name`
6. `description`

这不是单纯为了“更好解释”。  
更深层的含义是：

`权限仲裁输出的不是裸阻断，而是带修复候选路径的阻断。`

而且 `src/entrypoints/sdk/controlSchemas.ts:594-603` 的  
`ControlErrorResponseSchema` 在 error response 里还保留：

`pending_permission_requests`

这说明作者连报错场景都不愿让动作上下文丢失。  
也就是说，  
控制面不仅在发出动作建议，  
还在维护：

`动作链不能因为一次错误响应而断掉。`

## 4.3 宿主消费层也没有把这些字段当装饰

`src/hooks/useDirectConnect.ts:101-107`、  
`src/hooks/useRemoteSession.ts:345-350`、  
`src/hooks/useSSHSession.ts:105-110`  
都把 `request.permission_suggestions` 直接放进 `PermissionAskDecision.suggestions`。

这说明动作语法并没有停在 schema 层。  
它被稳定地穿透到了多种宿主消费层。

也就是说，  
Claude Code 的设计不是：

`底层给状态，上层自己猜怎么修。`

而是：

`底层把可接受修法一并编码，上层负责呈现和执行。`

## 4.4 hook 与内部权限类型也把“理由 + 动作”当成同一个对象

`src/entrypoints/sdk/coreSchemas.ts:425-432` 的  
`PermissionRequestHookInputSchema` 会把 `permission_suggestions` 暴露给 hook。  
`src/entrypoints/sdk/coreSchemas.ts:875-890` 的  
`PermissionRequestHookSpecificOutputSchema` 又允许 hook 返回：

1. `allow` + `updatedPermissions`
2. `deny` + `message`

也就是说，  
hook 管线本身就是动作语法的一部分。

更进一步，  
`src/types/permissions.ts:199-226` 的 `PermissionAskDecision`  
把 `message`、`decisionReason`、`suggestions`、`blockedPath`、`pendingClassifierCheck`  
放进同一个 ask decision；  
`src/types/permissions.ts:271-320` 的 `PermissionDecisionReason`  
又把原因拆成 `rule`、`mode`、`hook`、`classifier`、`workingDir`、`safetyCheck` 等 typed 分支。

这说明源码内部模型早就不是：

`先解释原因，再让别的层自己发明动作。`

它更接近：

`把原因、动作建议和后续异步判定资格一起打包成权限决定对象。`

## 5. 第二类动作语法：turn 摘要不是结果卡片，而是下一步动作包

## 5.1 `post_turn_summary` 同时输出“刚做了什么”和“接下来该做什么”

`src/entrypoints/sdk/coreSchemas.ts:1544-1569` 很关键。  
`SDKPostTurnSummaryMessageSchema` 不只给：

1. `status_category`
2. `status_detail`
3. `is_noteworthy`

还继续给：

1. `recent_action`
2. `needs_action`
3. `artifact_urls`

这说明作者并不满足于生成一条：

`本轮结束状态说明。`

他们要的是一条更完整的东西：

`本轮做了什么、现在卡在哪、接下来应做什么、相关产物在哪里。`

## 5.2 这说明摘要本身就是“动作包络”

很多系统把 summary 理解成摘要文本。  
Claude Code 这里更接近：

`把状态说明、注意力排序和下一步动作压成同一条消息。`

这意味着：

1. `blocked` 不是普通失败，而是需要明确解堵动作
2. `waiting` 不是普通空转，而是应等待某个前提成立
3. `review_ready` 不是普通完成，而是把下一步转交给审查动作

也就是说，  
这些状态的真实语义不是名词，  
而是：

`名词 + 动作方向`

## 6. 第三类动作语法：修复与回滚能力被编码成可执行分支，而不是抽象能力

## 6.1 `rewind_files` 的结果直接告诉调用方哪条修复动作存在、哪条不存在

`src/entrypoints/sdk/controlSchemas.ts:318-327` 明确给出：

1. `canRewind`
2. `error`
3. `filesChanged`
4. `insertions`
5. `deletions`

这说明“回滚”不是一个含糊的 capability label。  
系统在每次响应里都在明确区分：

1. 能不能回滚
2. 为什么不能
3. 如果能，影响面有多大

这其实就是动作语法：

`不要从“系统支持 rewind”直接跳到“当前这次一定能 rewind”。`

## 6.2 `api_retry` 也是一种动作绑定，而不是单纯错误广播

`src/entrypoints/sdk/coreSchemas.ts:1572-1587` 的 `SDKAPIRetryMessageSchema` 也很典型。  
它输出：

1. `attempt`
2. `max_retries`
3. `retry_delay_ms`
4. `error_status`
5. typed `error`

这说明它不是只在说：

`请求失败了。`

它更像是在说：

`这次失败已被系统归类，当前正确动作不是盲目重试或乱切状态，而是按既定重试窗口等待。`

## 7. 第四类动作语法：连接与认证状态的真正价值，在于把恢复路径分流

## 7.1 MCP 状态被拆开，本质上是在拆动作路径

`src/services/mcp/types.ts:179-226` 与  
`src/entrypoints/sdk/coreSchemas.ts:167-190`  
都把 MCP 相关状态分成：

1. `connected`
2. `failed`
3. `needs-auth`
4. `pending`
5. `disabled`

而且 `PendingMCPServer` 还带：

1. `reconnectAttempt`
2. `maxReconnectAttempts`

这说明作者非常清楚：

`同样是不“可立即使用”，其正确下一步动作完全不同。`

例如：

1. `needs-auth` 的动作是补认证
2. `pending` 的动作是等待重连而不是误报失败
3. `disabled` 的动作是承认当前策略关闭，而不是继续重试
4. `failed` 才更接近真正的错误排查路径

## 7.2 源码里还会把动作提示继续压成更短的人类说明

`src/services/mcp/client.ts:335-360` 在认证失败时直接返回 `needs-auth`。  
`src/services/mcp/channelNotification.ts:222-227` 又把 auth gate 的结论写成：

`channels requires claude.ai authentication (run /login)`

`src/services/mcp/useManageMCPConnections.ts:593-603` 则继续把不同 gate 映射成不同短通知：

1. auth -> `run /login`
2. policy -> 请管理员开启 `channelsEnabled`
3. disabled -> 当前功能不可用

这说明系统并没有把动作路径藏在支持文档里。  
它在协议层、连接层和通知层都反复维护同一件事：

`不同状态必须收敛到不同下一步。`

## 8. 第五类动作语法：typed failure 的意义，不只是分类，而是防止错误补救

## 8.1 `OrgValidationResult` 不是“校验失败”，而是“该怎么修”的分流器

`src/utils/auth.ts:1911-1999` 很有代表性。  
这个类型不是异常，而是：

1. `{ valid: true }`
2. `{ valid: false; message: string }`

注释还明确说：

`returns a result object rather than throwing so callers can choose how to surface the error`

更关键的是，  
不同失败消息对应的动作并不一样：

1. profile 拉取失败时，建议重试或重新用 `claude auth login` 获取 full-scope token
2. 环境变量 token 组织不匹配时，建议移除环境变量或换正确 token
3. 普通组织不匹配时，建议重新登录到正确组织

也就是说，  
这里真正被编码的不是：

`验证没过`

而是：

`哪一种补救动作才不会继续扩大错误。`

## 8.2 `CCRInitFailReason` 也是同一套哲学

`src/cli/transports/ccrClient.ts:49-57` 把初始化失败拆成：

1. `no_auth_headers`
2. `missing_epoch`
3. `worker_register_failed`

这说明作者不接受：

`远程初始化失败，重试看看。`

因为这三类失败的合理动作明显不同：

1. 认证头缺失更像入口资格问题
2. `missing_epoch` 更像协议或状态基底未成立
3. `worker_register_failed` 更像工作端注册链失效

typed reason 在这里的价值，  
并不是让日志更好看，  
而是防止系统和用户走上错误修复路径。

## 9. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`安全声明如果不能约束正确下一步动作，就仍然是不完整声明。`

这条公理几乎能统一前面所有例子：

1. 权限拒绝要附带允许的修法集合
2. turn 摘要要附带 `needs_action`
3. 回滚结果要显式区分当前能不能修
4. 连接状态要按恢复路径拆开
5. typed failure 要把补救动作分流

所以在 `38` 和 `39` 之后，  
安全控制面的更深层真相已经不只是：

1. 未知要显式
2. 已知要分级

还包括：

3. 已知后的正确动作也要显式

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的控制面不是状态机外面再挂一层帮助文案，而是在协议层就把“结论 -> 动作”绑定起来。`

这带来几个很强的工程收益：

1. 宿主不必发明自己的补救逻辑
2. 支持团队不会把不同失败形态混成同一个 FAQ
3. 用户不容易在 `pending`、`needs-auth`、`disabled` 之间误走同一条动作
4. 权限系统可以把“允许的修法”直接沿协议下发给不同宿主
5. 观测与产品层能围绕同一批 typed action signal 建模

这对其他 Agent 平台构建者的启示也很直接：

1. 不要只设计状态字段，也要设计下一步动作字段
2. 不要把补救逻辑只留给文档和客服
3. 不要让宿主在缺乏 authority 的情况下自己发明解释和动作
4. 让错误类型、状态等级和动作路径共用同一套协议语言

## 11. 哲学本质

这一章更深层的哲学是：

`安全控制面的真相不是静态名词，而是带方向的动词。`

很多系统以为把“当前是什么状态”说出来就够了。  
Claude Code 的源码更接近另一种成熟认识：

`如果一个声明不能把人带到正确下一步，那它仍然没有真正完成自己的职责。`

这意味着安全控制面的成熟，  
不只在于它会判断，  
还在于它会：

`把判断约束成正确动作。`

## 12. 苏格拉底式反思：这一章最容易犯什么错

### 12.1 我们是不是把一些文案字段过度解释成“动作协议”

有风险。  
但这里不是只看到个别文案，而是看到：

1. schema 层有 `permission_suggestions`
2. summary 层有 `needs_action`
3. 状态层有 `needs-auth / pending / disabled`
4. 消费层会直接读取这些字段驱动交互

这已经不是偶然文案，而是重复出现的结构选择。

### 12.2 动作语法会不会让协议太重

会更重。  
但如果不这样做，  
复杂性只是被转移到宿主、用户和支持团队身上。

### 12.3 这套动作语法还有什么没做到最好

还有。  
例如 `needs_action` 目前仍是字符串，  
未来如果继续推进，  
完全可以再把动作主体、动作紧急度和动作前置条件 typed 化。

也就是说，  
这一章看到的是很成熟的方向，  
但还不是动作语法的终点形态。

## 13. 结语

`38` 回答的是：不知道时必须显式说不知道。  
`39` 回答的是：知道时也不能只说 yes/no。  
这一章继续推进后的结论则是：

`即使已经把结论分级了，系统仍然要把正确下一步动作绑定进去。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟安全控制面不是静态声明面板，而是一台会把状态、理由、等级与动作一起编码出来的控制语法机。`
