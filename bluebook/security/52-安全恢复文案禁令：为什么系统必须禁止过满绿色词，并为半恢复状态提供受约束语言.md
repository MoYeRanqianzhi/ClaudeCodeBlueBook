# 安全恢复文案禁令：为什么系统必须禁止过满绿色词，并为半恢复状态提供受约束语言

## 1. 为什么在 `51` 之后还要继续写“恢复文案禁令”

`51-安全恢复中间态` 已经回答了：

`系统恢复到一半时，必须把半恢复状态正式命名出来。`

但如果继续往更细的一层追问，  
还会出现一个更具体的问题：

`即使系统已经知道自己处在半恢复状态，它到底应该被允许说哪些词，又绝不能说哪些词？`

因为很多控制面的问题，  
并不是完全不知道当前状态，  
而是：

`知道是半恢复，却仍然沿用全面恢复时代的绿色词。`

这意味着真正危险的东西，  
已经不再只是状态模型，  
而是：

`状态模型被翻译成什么语言。`

所以在 `51` 之后，  
统一安全控制台还必须继续补一条制度：

`恢复文案禁令。`

也就是：

`当系统处于半恢复、窄恢复、只恢复到某层 signer 的状态时，哪些绿色词必须被禁止，哪些受约束语言必须被启用。`

## 2. 最短结论

从源码看，Claude Code 当前展示层已经存在若干典型压缩词：

1. `Remote Control active`  
   `src/bridge/bridgeStatusUtil.ts:123-140`
2. `Connected to ... extension` / `Installed ... extension` / `Not connected ...`  
   `src/utils/status.tsx:54-84`
3. MCP 汇总里的 `connected / need auth / pending / failed`  
   `src/utils/status.tsx:95-114`
4. IDE、MCP 通知里的 `disconnected`、`not connected`、`failed`、`needs auth`  
   `src/hooks/notifs/useIDEStatusIndicator.tsx:79-152`、`src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`

这些词并不全错。  
真正的问题是：

`它们没有和 signer hierarchy、恢复中间态、宿主盲区宽度建立强绑定。`

所以这一章的最短结论是：

`下一代统一安全控制台必须把文案本身提升成受约束语言系统：在半恢复状态下，过满绿色词应被禁止，系统必须改用能显式暴露缺口的受约束表达。`

我会把这条结论再压成一句话：

`安全控制面不只要有正确状态，还要有禁止把错误乐观翻译成文字的语言护栏。`

## 3. 源码已经暴露出“文案压缩”与“状态真实厚度”之间的张力

## 3.1 `Remote Control active` 是最典型的过满绿色词

`src/bridge/bridgeStatusUtil.ts:123-140` 当前逻辑非常短：

1. error -> `Remote Control failed`
2. reconnecting -> `Remote Control reconnecting`
3. `sessionActive || connected` -> `Remote Control active`

这条映射的危险不在于它错误，  
而在于：

`它把多个厚度不同的恢复状态，统一翻译成了同一个成功词。`

而前一章已经证明，  
下面这些状态都可能被它吃掉：

1. `mirror-only`
2. `write-ready / read-blind`
3. `restored-but-not-fresh`
4. `control-narrow`

所以 “active” 这个词在这里最大的风险不是过于抽象，  
而是：

`它会让用户自动脑补成 fully interactive / fully recovered。`

## 3.2 IDE 状态词也在暗示一种过早完成

`src/utils/status.tsx:54-84` 里，IDE 相关文案主要是：

1. `Connected to ... extension`
2. `Installed ... extension`
3. `Not connected to ...`

这些词本身很易懂，  
但它们默认在说的是：

`安装状态` 或 `连接状态`

而不是：

`IDE 侧当前到底恢复到了哪种交互厚度`

也就是说，  
就算宿主现在只恢复到 selection 可见、  
或只有 plugin 在、本体没连、  
或只是某种窄交互就绪，  
控制面也没有足够语言把这些差异说出来。

## 3.3 MCP 汇总词虽然诚实，但仍天然会压平能力差异

`src/utils/status.tsx:95-114` 会把 MCP 汇总成：

1. `connected`
2. `need auth`
3. `pending`
4. `failed`

它在总览页上很实用，  
但也天然存在一种风险：

`用户会把“有几个 connected”理解成“当前外部能力整体可恢复使用”。`

而真实情况可能只是：

1. 某些关键 server 仍没恢复
2. 某条动作路径仍被 auth / policy 卡住
3. 总览 connected 数量无法直接推出“当前主任务链已恢复”

所以这里的问题也不是用词不够友好，  
而是：

`摘要词如果没有禁止解释范围，就会被读成超出自身证据强度的结论。`

## 3.4 通知层已经更诚实，因为它天生偏“缺口语言”

`src/hooks/notifs/useIDEStatusIndicator.tsx:79-152` 和  
`src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`

反而展示了一个更成熟的方向。  
这些通知词大多是：

1. `disconnected`
2. `not connected`
3. `failed`
4. `needs auth`

也就是说，  
通知层天然更擅长“缺口表达”，  
而不是“成功表达”。

这恰恰说明：

`真正危险的，不是系统不会说问题，而是系统在恢复态里太爱说绿色好词。`

## 4. 第一性原理：语言不是展示皮肤，而是解释权限的最后一道边界

如果从第一性原理追问：

`为什么文案本身要被当成安全对象治理？`

因为用户最终不会直接消费：

1. ledger
2. signer
3. host blindspot
4. ready split

用户最终消费的是：

`一句话。`

而一句话一旦超过了它背后的证据强度，  
系统即使底层全做对了，  
在用户那一侧仍然等价于：

`说错了。`

所以控制面最后一道边界其实不是字段，  
也不是卡片，  
而是：

`语言是否越权。`

我会把这条原则压成一句话：

`过满文案，就是展示层的越权解释。`

## 5. 我给统一安全控制台的文案禁令体系

我会把它分成三层。

## 5.1 第一层：禁止词

这类词在半恢复状态下绝不应出现。  
典型包括：

1. `active`
2. `connected`
3. `ready`
4. `restored`
5. `completed`
6. `all clear`
7. `fully`
8. `up to date`

它们的问题不在于单词本身，  
而在于它们隐含：

`没有关键缺口`

因此只要系统还处于：

1. `mirror-only`
2. `write-ready / read-blind`
3. `restored-but-not-fresh`
4. `control-narrow`
5. `result-ready / effects-pending`

这些词就必须被禁止。

## 5.2 第二层：受约束词

这类词允许使用，  
但必须和缺口对象一起出现。  
例如：

1. `attached (mirror-only)`
2. `write path ready; read path unproven`
3. `restored; awaiting fresh state`
4. `permission-only control host`
5. `result delivered; persistence pending`

这类表达的共同特点是：

1. 它承认当前已经恢复了某一部分
2. 它同时显式说明还缺哪一部分

所以它们是：

`受约束绿色词`

## 5.3 第三层：禁止跳级词

有些词单独看似乎没问题，  
但一旦用于更高层对象就会越权。  
比如：

1. 用 `connected` 给主结论签字
2. 用 `restored` 给 fresh state 签字
3. 用 `idle` 给 effects complete 签字
4. 用 `files_persisted` 给 explanation 完整性签字

这类情况最危险的不是词本身，  
而是：

`词和对象层级不匹配`

所以统一安全控制台必须同时审查：

1. 说了哪个词
2. 这个词挂在了哪张卡上
3. 它替哪一层对象在说话

## 6. 我给恢复态文案的最低语法规范

我会把所有恢复态的正确语言压成一个最小句式：

`已恢复的部分 + 当前缺口 + 默认下一步`

例如：

1. `Bridge attached; inbound control unavailable · open bridge details`
2. `State restored; awaiting fresh sync · stay on status`
3. `Result delivered; persistence pending · wait for file confirmation`
4. `Permission path available; broader control unsupported · use permission flow only`

这类句式的好处是：

1. 不会伪装成 fully recovered
2. 不会只说坏消息
3. 会直接告诉用户下一步去哪

## 7. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于底层已经区分了多种恢复厚度，还在于它已经有足够证据让展示层建立一套受约束语言系统。`

具体说，它至少已经具备：

1. 可用于禁止 `active` 泛化的中间态证据  
   `replBridgeTransport.ts:142-147,289-299,336-345`
2. 可用于禁止把 restore 说成 fresh 的恢复证据  
   `ccrClient.ts:474-522`、`print.ts:5048-5058`
3. 可用于禁止把窄 control 宿主说成 fully ready 的协议证据  
   `DirectConnectManager.ts`、`RemoteSessionManager.ts`
4. 可用于禁止把副作用未完成说成 completed 的 effect 证据  
   `print.ts:2261-2267`

对其他 Agent 平台构建者，这里的直接启示有四条：

1. UI copy 应像 API schema 一样被约束
2. 绿色词需要 capability gate，不应默认可用
3. “好消息”必须绑定缺口说明，否则会越权
4. 产品语言应服从 signer hierarchy，而不是服从营销冲动

## 8. 哲学本质

这一章更深层的哲学是：

`真正诚实的系统，不只限制动作，也限制语言。`

很多系统的失败并不是因为底层没做对，  
而是因为它们在最后一步忍不住给自己加一句更好看的总结。  
Claude Code 源码里那些已经存在的中间态和 signer 层级，  
如果最后仍被翻译成：

`active`
`connected`
`completed`

那控制面最终还是会在语言层滑回假完整。

从第一性原理看，  
这保护的是下面这条公理：

`系统不得使用超出自身证据强度的词汇来描述自己的恢复状态。`

## 9. 苏格拉底式反思：这一章最容易犯什么错

### 9.1 会不会把所有文案都写得太保守，最后什么都不敢说

会。  
所以禁令不是为了禁止表达，  
而是为了：

`把表达从过满绿色词，改造成受约束绿色词。`

### 9.2 会不会让文案体系过于复杂，最终没人维护

也会。  
所以最少只需要维护三类东西：

1. 禁止词
2. 允许的受约束句式
3. 各层对象对应的最高可说词


## 10. 结语

`51` 回答的是：半恢复状态必须有名字。  
这一章继续推进后的结论则是：

`这些名字还必须被一套语言禁令保护，确保系统不会再用超出自身证据强度的绿色词把它们重新说满。`

这意味着 Claude Code 更深层的安全启示之一是：

`成熟安全控制面最后一道边界，不只是状态机和动作机，也是文案机。`
