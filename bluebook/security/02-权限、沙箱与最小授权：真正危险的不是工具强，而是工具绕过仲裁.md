# 权限、沙箱与最小授权：真正危险的不是工具强，而是工具绕过仲裁

## 1. 先说结论

Claude Code 的权限设计最成熟的一点，不是“默认更严格”，  
而是它非常清楚：

`真正危险的不是工具能力强，而是强能力在没有经过正确仲裁层时被自动放行。`

因此这套权限系统的重点不是简单减少工具，  
而是四件事：

1. 先定义 mode
2. 再加载 rule
3. 再做 safety check
4. 最后才决定 allow / ask / deny

## 2. 权限系统为什么不是一个简单 allowlist

`permissions.ts`、`permissionSetup.ts`、`PermissionMode.ts`、`permissionsLoader.ts` 放在一起看，会发现 Claude Code 的权限系统至少有五层输入：

1. permission mode
2. settings / CLI / session / command 来源的规则
3. sandbox 运行时状态
4. safety check 与 path validation
5. 部分 feature-gated 的 classifier / denial tracking

这意味着权限结果不是单点判断，  
而是：

`多来源规则 + 运行时上下文 + 安全边界检查` 的合成结果。

## 3. 为什么 mode 先于 rule

从 `PermissionMode.ts` 与 `permissionSetup.ts` 看，  
mode 不是“展示给用户看的风格”，而是直接参与安全语义。

例如：

- `acceptEdits` 会改变编辑类动作的自动放行条件
- `bypassPermissions` 会触发额外环境安全前提检查
- feature-gated `auto` 进入时还会 strip dangerous rules

这说明 Claude Code 的权限系统不是：

`先有规则，再看模式。`

而是：

`先确立当前允许什么级别的信任语义，再解释这些规则该怎么执行。`

## 4. 为什么说“危险 allow rule”比“危险工具”更可怕

`dangerousPatterns.ts` 和 `permissionSetup.ts` 的组合非常值得注意。

系统明确把下面这类规则视为危险：

- `Bash(*)`
- `Bash(python:*)`
- `PowerShell(iex:*)`
- `Agent(*)`

原因不是这些工具“天然不能用”，  
而是这些规则会让：

- 任意解释器
- 任意嵌套 shell
- 任意子代理扩张

在 classifier 或交互确认前被直接放行。

所以 Claude Code 的最小授权原则不是：

`少给工具。`

而是：

`不给无边界的工具入口。`

## 5. path validation 为什么是这套系统的关键防线

`filesystem.ts` 和 `pathValidation.ts` 显示出另一个成熟点：

- 安全检查先于 working directory 的便利放行
- 内部可编辑路径和可读路径被单独处理
- `.claude`、`.git`、`.vscode` 等敏感目录被额外保护
- 大小写归一化、UNC 风险路径、路径遍历都被纳入检查

尤其关键的是：

`pathValidation.ts` 把 safety check 放在“当前目录内自动允许”之前。`

这意味着系统非常清楚一个常见失败模式：

`不能因为用户已经在项目目录里，就默认一切写入都安全。`

否则：

- 配置文件污染
- hook 注入
- 编辑器配置接管
- 敏感内部目录修改

都会被错误地降格成“普通项目编辑”。

## 6. 沙箱在这里究竟扮演什么角色

`sandbox-adapter.ts` 与 `setup.ts` 一起看，沙箱不是“万能护城河”，  
而是权限系统的一层外部边界。

它承担三类任务：

1. 文件系统读写限制
2. 网络访问限制
3. 高风险模式的环境前提约束

但更重要的是：  
Claude Code 没有把沙箱当成唯一答案。

相反，它做了三层叠加：

- permission rule 决定是否允许某类动作
- path validation 决定目标路径是否安全
- sandbox 决定运行时环境里这类动作还能接触到什么

这是一种很成熟的设计：

`能力允许` 和 `环境隔离` 被拆成两个问题，各自独立治理。

## 7. `--dangerously-skip-permissions` 为什么仍然要受限制

`setup.ts` 很清楚地说明：

- root / sudo 场景下不能随意开启
- 即使开启，也要求位于 Docker / sandbox 之类更可控环境
- 同时还要求没有互联网

这说明 Claude Code 不承认一种简单逻辑：

`只要用户愿意承担风险，就可以完全跳过安全设计。`

它更接近下面这条原则：

`绕过交互权限，至少要用更强的环境边界补回来。`

这也是为什么这套系统看起来比很多本地 agent 更“麻烦”，  
但其实更成熟。

## 8. deny tracking 和 classifier 的真正价值是什么

公开提取树里的 classifier 存在 ant-only / stub 边界，  
不能把内部分类细节写成稳定公共事实。

但即便只看公开可确认的结构，也能稳定下一个结论：

- Claude Code 并不把 classifier 当作唯一裁判
- deny tracking、fallback、prompting、mode 共同参与权限收口

也就是说，这套系统并不是：

`丢给一个安全小模型去判断。`

而是：

`把分类器嵌进一个更大的状态机。`

## 9. 从第一性原理看，这套权限设计在防什么

如果从第一性原理上重写开发代理的权限系统，  
真正需要防的不是“模型会不会出错”这一句废话，  
而是更具体的四种失控：

1. 无边界的命令执行
2. 无边界的路径写入
3. 无边界的子代理扩张
4. 无边界的规则提前放行

Claude Code 的答案很清楚：

- 强工具可以存在
- 但强工具必须经过 mode、rule、path safety 和 sandbox 的多层边界

## 10. 给平台构建者的技术启示

如果要借鉴 Claude Code，最值得学的不是某个具体规则字符串，  
而是四条实现原则：

1. 把 mode 当安全语义，不要当 UI 外观
2. 把危险点定义为“绕过仲裁层”，不要只盯着“危险工具名单”
3. 路径安全检查必须先于便利路径放行
4. 即使允许 bypass，也要用更强的环境约束来补偿

## 11. 一句话总结

Claude Code 的权限与沙箱设计之所以成熟，不在于它“更爱拒绝”，而在于它明确把最小授权理解为“任何强能力都必须经过正确仲裁层与外部边界”，真正危险的不是工具强，而是工具在没有边界时变成了无条件入口。
