# 统一安全控制台卡片设计：如何把字段、证明链与最短动作做成真正可用的界面

## 1. 为什么在 `23` 之后要写“卡片设计”

`23-统一安全控制台字段设计` 已经回答了：

`源码里已经有哪些状态字段，还缺哪些跨面字段。`

但如果继续往产品落地层推进，仅有字段清单还不够。  
因为真实用户并不会消费“字段表”，  
他们只会消费：

`卡片、面板、状态块、诊断区和下一步动作。`

因此这一章继续追问的，是另一个更具体的问题：

`这些字段应该如何被组织成真正可用的统一安全控制台界面？`

我的判断是，Claude Code 的下一代安全控制台不应该直接把所有字段平铺。  
它应该采用：

`证明链驱动的卡片设计。`

也就是：

`每张卡片都围绕一条证明链、一类边界问题和一个最短纠正动作来组织。`

## 2. 最短结论

从源码看，统一安全控制台最合理的实现方式不是“一个超长状态页”，  
而是七张核心卡片：

1. 主体与资格卡
2. 主权与来源卡
3. 仲裁与权限卡
4. 包络与执行壳卡
5. 外部能力卡
6. 增量审批卡
7. 断链诊断卡

其中前六张卡负责展示局部真相，  
最后一张卡负责把局部真相重新压成：

`当前哪条证明链断了，以及最短纠正动作是什么。`

这就是为什么我认为下一代统一安全控制台的关键，不在更多字段，  
而在：

`把字段、证明链和最短动作绑定成可操作卡片。`

## 3. 当前最大的技术瓶颈不是字段不足，而是状态投影太薄

### 3.1 `AppState` 与 SDK schema 里已经有大量结构化字段

从源码看，控制台原材料其实并不缺：

- `AppStateStore.ts` 已经保存 bridge、mcp、plugins、toolPermissionContext、remote status 等状态
- `coreSchemas.ts` 已经定义 `SDKStatusMessageSchema`、`SDKAuthStatusMessageSchema`、`McpServerStatusSchema`
- `controlSchemas.ts` 已经能结构化返回 MCP status
- `types/permissions.ts` 已经定义了丰富的 `PermissionDecisionReason`

这意味着：

`统一安全控制台需要的大部分原始字段，系统早就有了。`

### 3.2 真正的问题是这些字段在展示层被压扁了

最有代表性的源码是 `remote/sdkMessageAdapter.ts`：

- `convertStatusMessage()` 把结构化 status 压成非常薄的文本消息
- `auth_status` 被直接忽略，不转成显示消息

这说明当前宿主适配层在做一件对 REPL 简洁度合理、  
但对统一控制台非常不利的事情：

`把结构化安全状态过早压扁成文本，甚至直接丢弃。`

所以从技术上讲，统一控制台最大的瓶颈不是“采集不到字段”，  
而是：

`字段存在，但在跨宿主、跨视图传输时没有被保留为同一套安全语义对象。`

### 3.3 从第一性原理看，控制台的对象应该是“卡片”，不是“消息文本”

消息文本适合：

- 提示
- 通知
- 转瞬即逝的局部反馈

但控制台不同。  
控制台要解决的是：

`用户如何在一个稳定视图里持续理解当前边界状态。`

这就要求控制台的基本对象不是字符串，  
而应该是：

`带字段、带因果、带责任主体、带下一步动作的状态卡片。`

## 4. 为什么“卡片”是最合适的组织形式

因为 Claude Code 的安全状态天然是半异构、半分层的：

- 主体字段和资格字段很适合摘要卡
- 包络字段和外部能力字段很适合深度卡
- 审批字段和断链字段更适合事件卡 / 诊断卡

如果强行做成一个全局表，  
用户会看到大量低层细节却抓不住重点。  
如果强行做成单一总状态，  
又会失去真正有用的结构化信息。

卡片设计恰好能同时满足两件事：

1. 每张卡保持局部语义纯度
2. 多张卡又能通过证明链重新串成统一解释

所以卡片不是 UI 喜好问题，  
而是：

`与 Claude Code 当前安全数据形态最匹配的组织方式。`

## 5. 卡片一：主体与资格卡

### 5.1 这张卡要回答什么

`现在是谁在请求能力，以及这个主体是否具备继续证明资格的起点。`

### 5.2 应收纳的核心字段

- account type
- token source
- API key source
- auth status
- profile scope
- organization UUID
- bridge entitlement 基础条件

### 5.3 最适合从哪些源码装配

- `utils/status.tsx`
- `bridge/bridgeEnabled.ts`
- `SDKAuthStatusMessageSchema`

### 5.4 这张卡最重要的不是状态，而是断链提示

例如：

- 主体可识别，但 scope 不足
- 有账号，但缺 organization 归属
- token 是 inference-only，不足以支撑高阶远程能力

这张卡的关键输出不应只是“已登录”，  
而应是：

`主体链当前成立到哪一步。`

## 6. 卡片二：主权与来源卡

### 6.1 这张卡要回答什么

`谁在定义你现在所处的运行现实。`

### 6.2 应收纳的核心字段

- active setting sources
- policy origin
- host-managed env flags
- managed-only / plugin-only surfaces
- 当前 MCP / plugin source scope

### 6.3 最适合从哪些源码装配

- `settings.ts`
- `managedEnv.ts`
- `pluginOnlyPolicy.ts`
- `mcp/config.ts`
- `PluginSettings.tsx`

### 6.4 这张卡必须显式区分“来源存在”和“来源主导”

这是主权卡最关键的设计点。  
不能只显示：

- userSettings
- policySettings
- localSettings

还必须进一步显示：

1. 当前有哪些来源有值
2. 谁在主导关键表面
3. 哪些来源虽然存在，但没有最终主权

否则用户看到的只是配置罗列，  
而不是主权结构。

## 7. 卡片三：仲裁与权限卡

### 7.1 这张卡要回答什么

`当前动作为什么会被 allow / ask / deny。`

### 7.2 应收纳的核心字段

- permission mode
- decision behavior
- decisionReason.type
- rule source / rule value
- classifier reason
- safetyCheck reason
- classifierApprovable

### 7.3 最适合从哪些源码装配

- `types/permissions.ts`
- `permissions.ts`
- `permissionSetup.ts`
- `shadowedRuleDetection.ts`

### 7.4 这张卡的哲学本质是“仲裁因果卡”

它不应该只显示：

- mode = auto
- result = ask

它必须能继续说：

`为什么 ask，ask 是由哪一层引发，是否允许 classifier 接手，以及还有没有更高优先级的结构性危险。`

否则统一控制台仍然只是更大的状态页，  
不是更好的解释器。

## 8. 卡片四：包络与执行壳卡

### 8.1 这张卡要回答什么

`即便允许动作，它还被哪一层执行壳收住。`

### 8.2 应收纳的核心字段

- sandbox enabled
- fs read restrictions
- fs write restrictions
- network restrictions
- allowed Unix sockets
- excluded commands
- 当前路径 safetyCheck 命中情况

### 8.3 最适合从哪些源码装配

- `pathValidation.ts`
- sandbox adapter
- `SandboxConfigTab.tsx`

### 8.4 这张卡最关键的是“边界半径”，不是“开没开沙箱”

很多产品会把这张卡做成：

`Sandbox: Enabled`

这基本没有解释力。  
真正有用的是展示：

1. 允许动作后还能写哪
2. 不能碰哪
3. 网络包络长什么样
4. 是否有 managed network restriction

也就是说，这张卡应当围绕：

`执行半径`

而不是围绕“功能开关”。

## 9. 卡片五：外部能力卡

### 9.1 这张卡要回答什么

`当前到底开放了哪一种外部能力。`

### 9.2 应收纳的核心字段

- MCP server statuses
- MCP scope / config
- MCP tools / capabilities
- bridge connected / session active / reconnecting
- plugin marketplace 主权与错误
- WebFetch 的能力语义提示

### 9.3 最适合从哪些源码装配

- `McpServerStatusSchema`
- `controlSchemas.ts`
- `mcp/config.ts`
- `bridge/bridgeEnabled.ts`
- `PluginSettings.tsx`
- `WebFetchTool/preapproved.ts`

### 9.4 这张卡最关键的是“能力语义标签”

不能只显示：

- connected
- failed
- available

还要显示：

`这是读能力、连接能力、执行能力，还是身份能力。`

否则用户还是会把不同外部能力误当成同一回事。

## 10. 卡片六：增量审批卡

### 10.1 这张卡要回答什么

`今天的边界与昨天不同时，系统要求你重新确认了什么。`

### 10.2 应收纳的核心字段

- dangerous settings changed
- approval required
- approval result
- approval subject
- approval source

### 10.3 最适合从哪些源码装配

- `remoteManagedSettings/securityCheck.tsx`
- `ManagedSettingsSecurityDialog.tsx`

### 10.4 这张卡不是“告警卡”，而是“边界变更卡”

它真正证明的不是危险本身，  
而是：

`危险边界变化有没有重新取得正当性。`

所以这张卡必须是事件导向的，  
不能只是静态状态的一部分。

## 11. 卡片七：断链诊断卡

### 11.1 这张卡为什么必须独立存在

前六张卡都在讲局部真相。  
但用户最终真正关心的是：

`所以到底是哪一条证明链断了，我现在最短该做什么？`

这就是为什么统一控制台必须有单独的断链诊断卡。

### 11.2 这张卡应如何生成

它不是某个子系统直接提供的原始字段。  
它是一个派生层，  
要综合：

- 主体卡
- 主权卡
- 仲裁卡
- 包络卡
- 外部能力卡
- 增量审批卡

产出三条最关键的派生结论：

1. 当前主导边界
2. 当前断裂证明链
3. 当前最短纠正动作

### 11.3 这张卡才是真正的“统一控制台心脏”

没有这张卡，  
控制台只是多个漂亮卡片并排。  
有了这张卡，  
系统才能从“有很多可见性”变成：

`能把可见性压成行动。`

## 12. 从技术角度看，为什么这是下一步最自然的产品化路径

因为这条路径不要求重写底层机制。  
它主要要求三件事：

1. 保留结构化字段，不在宿主适配层过早压扁
2. 建立跨面的字段归一化层
3. 建立一个派生解释层，负责输出断链诊断卡

这三件事里，最难的不是 UI，  
而是：

`状态投影和派生解释。`

而 Claude Code 现有源码已经给了非常好的起点：

- `PermissionDecisionReason` 提供仲裁因果
- `McpServerStatusSchema` 提供外部能力状态
- `AppState` 提供运行时状态基座
- `bridgeEnabled.ts` 提供资格断链解释
- security dialog 提供边界变更事件

所以这一设计不是空想，  
而是对现有结构最顺理成章的上提。

## 13. 苏格拉底式追问

### 问：为什么不用一个总表替代卡片？

答：因为安全状态天然异构，总表会压平语义层级，用户只会看到噪音。

### 问：为什么断链诊断卡必须单独存在？

答：因为局部真相不会自动变成行动建议，必须有一层派生解释把它们压成最短纠正动作。

### 问：最大的技术风险是什么？

答：过早把结构化状态压扁成字符串消息，导致后续控制台只能重新猜测因果。

### 问：最值得先做哪张卡？

答：主体与资格卡、仲裁与权限卡、断链诊断卡。这三张最直接影响用户是否知道“为什么被拦”和“该找谁解决”。

## 14. 本章结论

统一安全控制台真正可行的实现方式，不是：

- 一个更长的 `/status`
- 一个更复杂的 settings 页
- 更多零散弹窗

而是：

`用七张围绕证明链组织的卡片，把字段、因果与最短动作绑定起来。`

这样 Claude Code 才能从：

`已有很多安全零件`

走到：

`拥有一个真正可解释、可诊断、可行动的统一安全控制台。`
