# 边界失真理论：为什么 Claude Code 检测的不是坏动作而是结构失真

## 1. 为什么在“不变量”之后还要写“失真理论”

`security/19` 已经把 Claude Code 安全设计里最重要的六条不变量压了出来。  
但如果继续往第一性原理追问，一个更高层的问题会出现：

`系统为什么需要这些不变量？`

我的判断是：

这些不变量并不是任意挑出来的工程偏好，  
它们其实都在共同对抗同一个更深的问题：

`边界失真。`

所谓边界失真，指的是：

`系统名义上宣称的安全边界，和运行时真实生效的安全边界，开始不再是同一件事。`

从这个角度回看 Claude Code，会发现它检测的并不只是“坏动作”，  
而是在持续修复各种让边界在传播过程中变形的结构性失真。

## 2. 最短定义

如果把这一章压成一句话：

`Claude Code 的安全检测，本质上是在守住“声明的边界”和“实际运行的边界”之间的等价关系。`

一旦这种等价关系被打破，风险就出现了。  
而这种破坏，通常不是一次性爆炸，而是分五类慢慢发生：

1. 规则失真
2. 指称失真
3. 主权失真
4. 接口失真
5. 解释失真

## 3. 第一类：规则失真

规则失真的含义是：

`纸面上的 permission rule，和真正会参与仲裁的 permission rule，不再是同一套东西。`

这类失真最典型的场景有三种：

1. 危险 allow rule 让 classifier 还没来得及看到动作，动作就被提前默认放行
2. tool-wide ask / deny 让更具体的 allow rule 变成不可达规则
3. 坏 permission rule 如果不被局部过滤，会反过来污染整份 settings

这些现象表面不同，底层却完全一致：

`规则的文本存在，不等于规则的安全语义仍然存在。`

因此 Claude Code 的权限系统不会只读取规则本身。  
它还要继续追问：

1. 这条规则是否合法
2. 这条规则是否可达
3. 这条规则是否会架空更高一层仲裁

这正是为什么 `dangerousPatterns.ts`、`permissionSetup.ts`、`shadowedRuleDetection.ts` 与 `validation.ts` 会在同一专题里反复出现。  
它们治理的是同一类问题的不同侧面。

## 4. 第二类：指称失真

指称失真的含义是：

`系统正在批准的对象，和系统最终真正作用到的对象，不再是同一个对象。`

路径安全是这类问题最典型的载体。  
`pathValidation.ts` 里之所以反复处理：

- UNC
- tilde 变体
- `$VAR` / `%VAR%`
- `$(...)`
- `=cmd`
- glob 写路径

并不是因为这些语法“花哨”，  
而是因为它们会制造一种非常危险的错觉：

`校验看到的是一个字符串，shell 执行到的是另一个真实目标。`

这就是指称失真。

从第一性原理看，任何授权行为都隐含一个前提：

`被授权对象必须和被执行对象保持同一指称。`

一旦这个前提失效，  
批准关系就已经在结构上被掏空了。

所以 Claude Code 在这里防的不是“路径复杂”，  
而是：

`授权对象和执行对象脱钩。`

## 5. 第三类：主权失真

主权失真的含义是：

`名义上谁应该定义运行时，和实际上谁正在定义运行时，不再一致。`

这类问题贯穿：

- `managedEnv.ts`
- `settings.ts`
- `pluginOnlyPolicy.ts`
- `services/mcp/config.ts`

在这些模块里，系统反复问的不是“值是什么”，  
而是：

`这个值、这条连接、这个扩展定义，究竟是谁有资格提供。`

比如：

1. host-managed provider env 不能被 settings.env 静默覆盖
2. `policySettings` 采用 first-source-wins，而不是“谁最后 merge 谁赢”
3. 某些 customization surface 会被收口到 plugin-only / managed-only
4. enterprise MCP config 存在时，其它来源甚至可以整体失去定义权

这些都说明 Claude Code 保护的不是“配置整洁”，  
而是：

`控制面主权不能从高信任主体滑向低信任主体。`

所以它的来源治理，本质上不是配置管理，  
而是：

`运行时主权保全。`

## 6. 第四类：接口失真

接口失真的含义是：

`系统原本只打算开放某种语义的入口，但这个入口在运行中被偷换成了另一种更危险的能力。`

最典型的例子来自外部入口：

1. WebFetch 允许的是 GET 读取公共内容
2. sandbox network access 允许的是更一般的网络交互
3. MCP 允许的是外部能力接入，但其风险还要区分 command、URL、SDK placeholder、enterprise exclusive

这里最危险的不是“外部入口存在”，  
而是：

`读取语义、连接语义、执行语义、身份语义被混写。`

一旦混写，接口就发生失真：

- 文档读取入口，可能被误当作通用网络信任
- MCP 名称配置，可能掩盖真实进程或真实端点
- 插件或连接定义入口，可能偷换成连接主权入口

这就是为什么 Claude Code 不愿意把“外部能力”当成一个统一标签。  
它更倾向于问：

`当前这个入口，到底在开放哪一种能力语义。`

这是非常成熟的安全设计。  
因为真正可靠的控制面，不相信接口标签，只相信接口语义。

## 7. 第五类：解释失真

解释失真的含义是：

`系统内部边界其实仍然存在，但用户已经无法知道当前到底是哪一层边界在生效。`

这类问题最容易被低估，  
但在复杂控制面里非常致命。

一旦解释失真出现，会发生三件事：

1. 用户把 fail-closed 误解成随机故障
2. 用户把局部撤权误解成全局封禁
3. 用户为了“恢复工作”开始盲目绕路，反而进一步冲撞边界

所以 `permissionExplainer.ts`、状态展示组件、managed security dialog 这些看似偏 UX 的零件，  
其实也属于安全系统本体的一部分。

它们修复的不是“审美问题”，  
而是：

`用户感知边界和系统真实边界之间的认知脱节。`

从第一性原理看，安全如果不可解释，  
就会在用户行为层面重新退化成黑箱。  
而黑箱会诱发新的风险动作。  
因此解释层不是装饰，而是防止边界再次失真的认知控制面。

## 8. 为什么这五类失真可以统一成一套理论

因为它们虽然出现在不同模块，  
但结构上都在问同一个问题：

`某个中间层是否让原始边界定义，在传递到运行时的过程中变形了。`

可以把它压成下面这条链：

```text
边界声明
  -> 规则装配
  -> 来源装配
  -> 模式切换
  -> 输入解释
  -> 外部入口接入
  -> 输出回流
  -> 用户感知
```

每一层都有可能引入新的失真：

- 规则装配会引入规则失真
- 输入解释会引入指称失真
- 来源装配会引入主权失真
- 外部入口接入会引入接口失真
- 用户感知会引入解释失真

这就是为什么 Claude Code 看起来像有很多“零散安全检测”，  
但实际并不零散。  
它们是在不同节点，防守同一条边界传递链。

## 9. 这套系统真正先进在哪里

如果用这套理论回看 Claude Code，它的先进性并不只是“拦截更早”。

更关键的是三点：

1. 它把安全问题从“动作危险性”提升成“边界传递完整性”
2. 它知道不同失真必须由不同层处理，不能全部丢给 classifier
3. 它承认用户感知也是控制面的一部分，因此解释层必须产品化

这比传统的 rule engine 更成熟，  
也比把安全全押给模型判断更稳。

因为后两种做法都有同一个问题：

`它们只盯着动作本身，没有盯住边界在系统内部传播时是否已经变形。`

## 10. 从损失函数看，平台到底在最小化什么

如果把这套系统写成损失函数，它最小化的并不是单一的“攻击成功概率”。

它更像同时在压低五种损失：

1. 规则绕过仲裁的损失
2. 授权对象与执行对象脱钩的损失
3. 低信任来源篡改主链的损失
4. 外部入口语义漂移的损失
5. 用户认知边界失真的损失

这也解释了为什么 Claude Code 经常愿意在局部牺牲一点流畅度：

- 多一层来源过滤
- 多一层 path precheck
- 多一层 MCP policy filter
- 多一层解释层展示

因为从平台损失函数看，  
这些摩擦换来的是：

`边界在传播过程中的完整性。`

## 11. 苏格拉底式追问

### 问：如果一条规则写在配置里，但永远不会被命中，它还算边界吗？

答：不算。那只是边界幻觉。

### 问：如果系统批准了某个路径字符串，但 shell 实际访问的是另一个对象，谁被批准了？

答：没有对象真正被批准；批准关系已经失真。

### 问：如果用户能通过低信任来源改写 provider、auth 或 MCP 入口，后面的 classifier 还能算在正确边界里工作吗？

答：不能。它只是在错误主权前提下继续推理。

### 问：如果一个只读入口能被误当成通用外联信任，风险发生在哪里？

答：不是发生在请求之后，而是发生在接口语义被偷换的那一刻。

### 问：如果系统机制完全正确，但用户始终不知道是哪一层边界在挡他，这套系统算真正完成了吗？

答：还没有。解释失真会反过来制造新的风险动作。

## 12. 给 Agent 平台构建者的最短启示

1. 先检测边界是否失真，再检测动作是否危险。
2. 规则系统必须治理“假规则”，不是只治理“坏规则”。
3. 所有需要 shell、path、URL、env 解释的输入，都应被视为潜在指称失真点。
4. 安全设计必须单独回答“谁有资格定义运行时”。
5. 外部入口的风险不在有没有接口，而在接口语义会不会被偷换。
6. 可解释性不是附属层，而是防止认知失真的安全层。

## 13. 本章结论

Claude Code 的安全检测之所以显得复杂，  
不是因为它堆了太多规则，  
而是因为它在多个传播节点上持续修复同一个问题：

`防止边界从声明走向运行时的过程中发生结构失真。`

因此，这套系统真正先进的地方不只是：

- 有 deny
- 有 ask
- 有 classifier
- 有 sandbox
- 有 managed settings

而是它已经隐含形成了一种更高阶的安全本体论：

`安全不是多拦一些动作，而是持续守住边界定义、边界指称、边界主权、边界接口和边界解释之间的等价关系。`
