# 安全载体家族强请求清理迁移治理与强请求清理退役治理分层：为什么artifact-family cleanup stronger-request cleanup-migration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-sunset-governor signer

## 1. 为什么在 `196` 之后还必须继续写 `197`

`196-安全载体家族强请求清理修复治理与强请求清理迁移治理分层` 已经回答了：

`stronger-request cleanup 线即便知道旧世界该怎样过渡，也还需要 migration governor 去决定兼容窗口、过渡写法、旧值保留与新值接管。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“迁完就算退役”的错觉：

`只要 stronger-request cleanup migration governor 已经安排了新旧世界如何并存，它就自动拥有了宣布旧 stronger-request 世界正式退场的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/utils/model/deprecation.ts:29-100` 已经为 deprecated models 维护 provider-specific retirement dates，说明 repo 区分  
   `已经有迁移路径`  
   和  
   `旧模型在哪一天正式退役`
2. `src/hooks/notifs/useModelMigrationNotifications.tsx:7-33` 只负责一次性宣布  
   `模型刚刚被迁到新默认值`
3. `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-31` 与 `src/main.tsx:2872-2892` 负责持续提示  
   `旧模型即将在某日退役`
4. `src/migrations/migrateLegacyOpusToCurrent.ts:13-51` 即便已把 `userSettings` 迁到 `opus`，  
   `src/utils/model/model.ts:552-554` 仍保留 legacy remap opt-out，说明 migration 已发生时旧显式字符串仍可能继续活着
5. `src/utils/plugins/cacheUtils.ts:20-99` 说明 orphan plugin version 先进入 grace world，再在 7 天后删除；  
   这不是 migration 本身，而是 sunset deadline
6. `src/utils/plugins/orphanedPluginFilter.ts:1-79` 与 `src/main.tsx:2546-2568` 又进一步给出 visibility sunset：旧 plugin version 可能先从当前 session 的 search world 退役，再从 storage world 真正消失

这说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界先怎样被带过渡。`

后者才配说：

`过渡什么时候结束，旧 stronger-request law 从哪一天、哪一个 runtime、哪一个 visibility surface 起正式不再有效。`

所以 `196` 之后必须继续补的一层就是：

`安全载体家族强请求清理迁移治理与强请求清理退役治理分层`

也就是：

`stronger-request cleanup migration governor 负责 coexistence；stronger-request cleanup sunset governor 才负责宣布 coexistence 结束。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-sunset-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 retirement manager，
而是在说：

1. repo 在 model 与 plugin 线已经展示出正式的 sunset grammar
2. sunset grammar 回答的是 retirement date、grace-window 截止、旧世界 visibility 何时被切断
3. 任何 stronger-request cleanup grammar 只要想自称完整，就必须回答这些“何时正式结束”的问题

第二条：

这里说的 `sunset governance`

不是在声称：

`旧世界一旦退役，就必须立刻从磁盘彻底消失。`

真正要说的是：

`退役治理首先回答的是“从什么时候起它不再算当前世界”，而这可以先发生在 visibility、search、warning semantics 上，再发生在 storage delete 上。`

第三条：

这里也不是把：

`plugin orphan grace window`

单独当成全部 sunset grammar。

真正关键的是：

1. model line 证明 migration notice 与 deprecation warning 是两张时钟
2. legacy remap 证明 migration 已完成时旧世界仍可能继续被兼容
3. plugin line 证明 visibility sunset 与 storage sunset 可以继续拆开
4. stronger-request cleanup 线当前还没有谁正式决定旧 plan path、旧 startup wording 与旧 receipt semantics 何时不再算当前世界

第四条：

这里尤其不能过度声称：

`migrationCompleted` 或 timestamp flag 已经等于 sunset 完成。`

当前更稳的说法只能是：

`这些 flag 主要回答“迁移脚本跑没跑过”或“刚迁完要不要提醒”，并不自动回答“旧世界何时正式退役”。`

第五条：

这里也不能过度声称：

`sunset governance` 一定意味着用户立刻失去所有兼容路径。`

当前更稳的说法只能是：

`sunset 可以先切 warning semantics、再切 visibility、最后切 storage；关键不在快慢，而在谁来正式宣布旧世界从何时起不再算 current truth。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类“stronger-request cleanup-migration-governor signer 仍不等于 stronger-request cleanup-sunset-governor signer”证据：

1. `deprecation.ts:29-100` 已为 deprecated models 维护 provider-specific retirement dates，说明 repo 区分  
   `已经有迁移路径`  
   和  
   `旧模型在哪一天正式退役`
2. `useModelMigrationNotifications.tsx:7-33` 只负责一次性宣布  
   `模型刚刚被迁到新默认值`，  
   而 `useDeprecationWarningNotification.tsx:6-31` 与 `main.tsx:2872-2892` 负责持续提示  
   `旧模型即将在某日退役`
3. `migrateLegacyOpusToCurrent.ts:13-51` 即便已把 `userSettings` 迁到 `opus`，  
   `model.ts:552-554` 仍保留 legacy remap opt-out，说明 migration 已发生但 sunset 尚未 hard-close
4. `cacheUtils.ts:20-99` 说明 orphan plugin version 先进入 grace world，再在 7 天后删除；  
   这不是 migration 本身，而是 sunset deadline
5. `orphanedPluginFilter.ts:1-79` 与 `main.tsx:2546-2568` 说明旧 plugin version 在 grace 期内虽然仍在磁盘上，却已先从当前 session 的可见世界里退役；  
   这是一种 visibility sunset，而不只是 file migration
6. `migrationCompleted` / timestamp flag 一类信号说明迁移动作已跑过，  
   但并不自动签出 “旧世界已经正式退役”

因此这一章的最短结论是：

`stronger-request cleanup migration governor 最多能说旧世界怎样过渡；stronger-request cleanup sunset governor 才能说旧世界何时停止作为正式世界继续存在。`

再压成一句：

`会过渡，不等于会退役。`

## 4. 第一性原理：migration governance 回答“如何并存”，sunset governance 回答“何时终止并存”

从第一性原理看，
强请求清理迁移治理与强请求清理退役治理处理的是两个不同时间面的主权问题。

stronger-request cleanup migration governor 回答的是：

1. 旧值是 rewrite、remap，还是 dual-read
2. 哪些 source 可以改写，哪些必须保持原样
3. 兼容期间如何提示用户
4. 哪些旧 artifact 要进入 grace window
5. 怎样保持幂等与 continuity

stronger-request cleanup sunset governor 回答的则是：

1. 兼容窗口在哪个日期、哪次启动或哪类版本后结束
2. 旧值何时不再被 runtime 接受
3. 旧 artifact 何时不再被搜索、引用或恢复
4. 旧 warning 何时从“建议切换”进入“正式不可再用”
5. 双世界共存何时必须切成单世界真相

如果把这两层压成一句“迁移策略已经存在就够了”，
系统就会制造五类危险幻觉：

1. coexistence-means-forever illusion  
   只要兼容窗口已经存在，就误以为它可以无限拖延
2. completion-means-sunset illusion  
   只要迁移脚本跑过一次，就误以为旧世界已经正式退役
3. warning-means-governance illusion  
   只要有提醒语，就误以为退役时点已经被治理
4. data-still-on-disk-means-still-valid illusion  
   只要旧 artifact 还在磁盘上，就误以为它仍应被当前世界继续消费
5. migration-covers-hard-close illusion  
   把“新旧怎么并存”误当成“旧世界何时不再被允许存在”

所以从第一性原理看：

`migration governor` 决定怎样带着旧世界前进；  
`sunset governor` 决定何时不再继续背着它前进。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup migration governance 已经等于 sunset governance，为什么 `useModelMigrationNotifications()` 和 `useDeprecationWarningNotification()` 还要分别存在？
   因为“刚迁完”与“即将退役”根本不是同一句制度。
2. 如果旧世界一旦被带过渡就算退役，为什么 legacy opus 迁完后仍允许 runtime remap 与 opt-out？
   因为 migration 已完成时，旧世界仍可能继续以兼容形式活着。
3. 如果 sunset 只是 hard delete，为什么 plugin 线还要先做 visibility cutoff，再做 7-day storage cleanup？
   因为退役先回答的是“从什么时候起不再算当前世界”，而不是“何时物理消失”。`

## 5. 模型线给出最强正例：migration notification 与 deprecation warning 是两种不同制度

`src/hooks/notifs/useModelMigrationNotifications.tsx:7-33` 非常清楚。
它只在 migration timestamp 刚写入 config、也就是“这次启动刚发生迁移”时弹一次通知。

它回答的是：

`你刚刚被带到了新模型。`

但 `src/utils/model/deprecation.ts:29-100` 与 `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-31` 回答的是另一件事：

`某个旧模型将在哪个 provider、哪个具体日期上被 retired。`

更重要的是，
`src/main.tsx:2872-2892` 还会把 deprecation warning 加入 initial notification queue，
说明退役提示不是迁移脚本副作用，
而是单独进入 runtime surface 的正式控制面。

这意味着模型线已经清楚地拆开了两张时钟：

1. migration clock  
   什么时候把用户带到新默认值
2. sunset clock  
   什么时候旧模型在 provider 侧正式退役

这正是 stronger-request cleanup 线当前仍缺的清晰度。

## 6. `legacy opus` 更进一步证明：migration 已完成，旧世界也可能仍以兼容 remap 的形式继续活着

`migrateLegacyOpusToCurrent.ts:13-51` 做了几件典型 migration-governor 的事：

1. 只改 `userSettings`
2. 记录 `legacyOpusMigrationTimestamp`
3. 让启动后可以弹一次 migration notification

但这还不是 sunset。

因为 `src/utils/model/model.ts:552-554` 的 `isLegacyModelRemapEnabled()` 仍允许 runtime remap 继续存在，
甚至保留 `CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP` 这样的 opt-out。

这说明：

1. migration 已发生
2. 旧世界仍被兼容
3. sunset 还没有被彻底 hard-close

这组代码非常适合照 stronger-request cleanup 线。
即便只把问题压回 cleanup 语义本身，
也仍然要继续回答：

1. 旧 startup wording 还能被引用多久
2. 旧 home-root plan law 还能活多久
3. 旧 receipt semantics 何时不再可被当成 current truth

这些都已经不是 migration 本身，
而是 sunset governance。

## 7. plugin 线给出第二组强正例：grace window 不是迁移附带品，而是退役制度本身

`src/utils/plugins/cacheUtils.ts:20-99` 最值钱的地方在于：

1. 旧版本先被标记 `.orphaned_at`
2. `CLEANUP_AGE_MS = 7 days`
3. 超过 7 天才真正删除旧版本

这已经不再只是“迁到新版本”。
它回答的是：

`旧版本什么时候只算兼容残影，什么时候连残影也不再允许继续留着。`

更进一步，
`src/utils/plugins/orphanedPluginFilter.ts:1-79` 与 `src/main.tsx:2546-2568` 还展示了另一种更细的 sunset grammar：

1. orphan GC 先调整磁盘状态
2. 然后 warm Grep/Glob exclusion cache
3. exclusion list 在 session 内冻结，除非 `/reload-plugins`

这意味着旧 plugin version 至少会经历两次退役：

1. visibility sunset  
   先从搜索/匹配世界消失
2. storage sunset  
   grace window 到期后再从磁盘上真正删除

这套拆法非常值得 stronger-request cleanup 学习。

## 8. 技术先进性与哲学本质：Claude Code 真正先进的不是“会迁移”，而是它在别处已经知道“旧世界何时才算真正结束”

从技术角度看，
Claude Code 在这条线上真正先进的地方，
不是它会写迁移函数。

更值钱的是它已经在多个子系统里承认：

1. 迁移完成不等于旧世界退役
2. warning clock 与 migration clock 不是同一时钟
3. visibility cutoff 与 storage delete 不是同一步
4. 兼容 remap 可以在 sunset 前继续存在
5. grace window 是正式治理，而不是临时拖延

这背后的哲学本质是：

`真正成熟的治理，不只会决定新规则是什么，还会决定旧规则从什么时候起不再被承认为当前世界。`

这也带来三条直接技术启示：

1. 任何 stronger-request cleanup grammar 若只补 migration 细节、不定义旧 promise、旧 path、旧 receipt 的 sunset clock，就仍缺 formal retirement authority
2. plugin 的 visibility sunset / storage sunset 双时钟很适合成为 stronger-request cleanup sunset grammar 的设计镜子
3. 如果 stronger-request cleanup 只补过渡、不补 hard-close authority，旧世界就会继续以“还在磁盘上、还在提示里、还在某些 runtime remap 中活着”的方式制造长期模糊地带

## 9. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 migration governance，就已经足够成熟。`

而是：

`repo 在 model deprecation、migration notifications、plugin orphan grace window 与 session-level exclusion 上已经明确展示了 sunset governance 的存在；因此 artifact-family cleanup stronger-request cleanup-migration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

因此：

`cleanup 线真正缺的不是“怎么带旧世界走一段路”，而是“谁来宣布这段路何时结束”。`
