# 安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层：为什么artifact-family cleanup stronger-request cleanup-metadata-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer

## 1. 为什么在 `447` 之后还必须继续写 `448`

`447-安全载体家族强请求清理制度理由治理与强请求清理元数据治理分层`
已经回答了：

`Claude Code 已经让不同 stronger-request cleanup family 长出 plural metadata microgrammar，但这些 microgrammar 还没有被提升成 equally-strong canonical carrier。`

但如果继续往下追问，
还会碰到下一层更危险、也更容易被误判为“系统既然已经记住规则，就一定已经按规则执行”的错觉：

`只要 stronger-request cleanup 的规则已经被系统自己记住，运行时就自然已经持续忠于这份自我记忆。`

Claude Code 当前源码仍然不能支持这种更强说法。

继续看会发现，
repo 当前最值得重视的事实不是：

`它没有 runtime cleanup 行为。`

而是：

`它已经把 stronger-request cleanup 做成一条分层运行链：declaration、admission、execution、skip、coverage 与 receipt 各自是不同问题。`

最硬的证据至少有八组：

1. `src/utils/settings/types.ts:325-332` 与 `src/utils/settings/validationTips.ts:47-54` 把 `cleanupPeriodDays=0` 写成很强的 metadata 句子：`no transcripts are written and existing transcripts are deleted at startup`
2. `src/utils/sessionStorage.ts:953-969` 的 `shouldSkipPersistence()` 当场首先证明的却只是 future transcript writes suppression
3. `src/screens/REPL.tsx:3903-3906` 说明 interactive world 只有在 first submit 后才启动 housekeeping
4. `src/main.tsx:2811-2818` 又说明 headless world 也只是 deferred import housekeeping，而且注释明确写出 `scripted calls don't need bookkeeping — the next interactive session reconciles`
5. `src/utils/backgroundHousekeeping.ts:28-80` 继续把 cleanup admission 写成 `10 minutes after start`、recent-activity reschedule 与 one-shot `needsCleanup`
6. `src/utils/cleanup.ts:575-584` 明确在 validation errors + explicit `cleanupPeriodDays` 时 intentional skip cleanup，以避免 accidental deletion
7. `src/utils/plans.ts:79-110,164-255`、`src/utils/permissions/filesystem.ts:244-255,1487-1495,1644-1652` 与 `src/utils/cleanup.ts:300-303` 继续给出最硬的 propagation gap：plan metadata 已被多个平面消费，但 cleanup executor 仍只认默认 `~/.claude/plans`
8. `src/utils/cleanup.ts:33-45,575-598` 说明 repo 已经有 `CleanupResult` 这种 local outcome vocabulary，但后台入口只为 stale worktrees 发 telemetry，并没有把 stronger-request family results 升格成统一 receipt；`src/utils/diagLogs.ts:14-20,27-60` 又继续说明 diagnostics carrier 连 visible cleanup coverage 都还没进入

这说明：

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

和

`artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer`

仍然不是一回事。

前者最多能说：

`系统已经把 stronger-request cleanup 的规则登记成了自己的某种自我记忆。`

后者才配说：

`在当前 mode、当前 timing、当前 validation 状态、当前 executor 与当前 coverage 下，这些规则已经被运行时诚实兑现，并且兑现结果被正式签收。`

所以 `447` 之后必须继续补的一层就是：

`安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层`

也就是：

`stronger-request cleanup-metadata governor 只能说明“系统说自己应该如此”；stronger-request cleanup-runtime-conformance governor 才能说明“系统现在真的如此发生”。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer。`

这里仍是研究命名。
它不是在声称 repo 已有一张现成的 conformance ledger 却无人接线，
而是在说：

1. metadata truth 即使存在
2. 仍必须穿过 admission、execution、skip、coverage 与 receipt
3. 才配被宣布成 runtime-conformance

第二条：

这里说的 `runtime-conformance`
不是在声称：

`当前实现一定有严重 bug。`

更稳的说法只能是：

`当前 stronger-request cleanup 已经足以证明 declared truth、admitted truth、executed truth、skipped truth、covered truth 与 receipted truth 仍不是同一层。`

第三条：

这里也不是把：

`cleanupPeriodDays=0`

单独当成全部运行时符合性问题。

真正关键的是：

1. declaration 不等于 admission
2. suppression 不等于 deletion
3. partial metadata consumption 不等于 all-plane execution
4. local return value 不等于 surfaced receipt
5. visible cleanup coverage 也不等于 retained carrier total coverage

第四条：

这里尤其不能过度声称：

`runtime skip 一定意味着实现失败。`

更稳的说法只能是：

`validation-driven skip 很可能恰恰体现了更高阶的 fail-safe semantics，而这本身就证明 metadata 与 runtime-conformance 不是一层。`

第五条：

这里也不能过度声称：

`只要函数返回 CleanupResult，就已经有 conformance receipt。`

当前更稳的说法只能是：

`local outcome count 说明作者意识到结果值得记录，但如果这份结果没有被汇总、投影、保存或签字，它仍不是 stronger-request family receipt。`

第六条：

这里还不能把：

`runtime-conformance`

偷换成：

`anti-drift verification`

即使某次运行真的 conform，
也还要继续问：

`谁来持续证明它明天不会重新漂移。`

所以 runtime-conformance 虽然比 metadata 更强，
但仍不是这条链的终点。

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-metadata-governor signer 仍不等于 stronger-request cleanup-runtime-conformance-governor signer`

证据：

1. `cleanupPeriodDays=0` 的 startup-delete wording 很强，但 `shouldSkipPersistence()` 立刻保证的首先只是 future-write suppression
2. interactive 与 headless mode 对 cleanup 的 admission 都走 deferred housekeeping，而不是 immediate delete receipt
3. `backgroundHousekeeping` 把 cleanup 进一步写成 delay / recent-activity reschedule
4. validation-error + explicit setting 会 intentional skip cleanup，runtime truth 明确可能偏离纸面 metadata
5. `plansDirectory` 证明 metadata 即使已被多个平面消费，也不自动保证 executor 跟随
6. `CleanupResult` 证明 outcome vocabulary 已存在，但 stronger-request family receipt 仍未形成
7. diagnostics carrier 继续说明并非每个 retained carrier 都已进入 visible cleanup conformance world

因此这一章的最短结论是：

`stronger-request cleanup-metadata-governor signer 最多只能说“系统宣称应该如此”；stronger-request cleanup-runtime-conformance-governor signer 才能说“它已经在当前 runtime 中如此发生，并且这个发生结果被正式保存”。`

再压成一句：

`制度会记住自己，不等于制度会执行并签收自己。`

再压成更硬的一句：

`制度已经把自己写下来，不等于制度已经把自己活出来。`

## 4. 第一性原理：metadata 回答“系统说了什么”，runtime-conformance 回答“系统做了什么”

从第一性原理看，
强请求清理制度元数据治理与强请求清理运行时符合性治理回答的是两种不同问题。

`stronger-request cleanup-metadata governor`

回答的是：

1. 规则被写在哪里
2. 哪些 schema、helper、path grammar、permission rule 与 env contract 描述了它
3. 哪些平面理论上应该消费这份 truth
4. 这份 truth 怎样被系统自己记住
5. 哪里已经出现 metadata drift

`stronger-request cleanup-runtime-conformance governor`

回答的则是：

1. scheduler 何时真正 admission 这条规则
2. executor 是否真的消费了这份 metadata
3. validation / mode / interaction / environment 是否改变了执行结果
4. suppress、delete、skip、coverage 与 receipt 是否同步成立
5. 当前 runtime 到底发生了什么，而不只是它本来应该发生什么

换句话说：

1. metadata 是 `制度自我记忆`
2. runtime-conformance 是 `制度运行兑现`

如果把两层压成一句：

`制度已经记住自己，所以运行时自然已经照做`

系统就会制造五类危险幻觉：

1. `documented-means-done`
2. `suppression-means-deletion`
3. `startup-language-means-immediate-execution`
4. `partial-consumption-means-full-conformance`
5. `return-value-means-receipt`

所以从第一性原理看：

`metadata` 负责制度自述，
`runtime-conformance` 负责制度兑现。

其哲学本质可以压成一句：

`制度已经把自己写下来，不等于制度已经把自己活出来。`

## 5. Claude Code 多重安全技术的先进性：它把 cleanup 真相拆成 declaration、admission、execution 与 abstention

从技术角度看，
Claude Code 在这一层最先进的地方，
不在于它已经形成完美的 cleanup receipt ledger，
而在于它没有落入“写了规则就算完成”的幼稚控制哲学。

它至少做了五个很成熟的拆分：

1. `declaration vs immediate effect`
   `cleanupPeriodDays=0` 写成强声明，但 `shouldSkipPersistence()` 只先承担停写，不冒充删除 receipt
2. `admission vs execution`
   housekeeping 被安排成 deferred path，而不是把所有 cleanup 承诺压进同步副作用
3. `interactive vs headless mode policy`
   REPL 与 headless 共享 housekeeping，但注释公开承认 scripted calls 不需要立刻 bookkeeping
4. `execution vs safer abstention`
   validation-error + explicit setting 场景下，系统选择 skip cleanup，以避免 accidental deletion
5. `local outcome vs surfaced receipt`
   repo 已经用 `CleanupResult` 表明结果值得计数，但仍克制地没有把局部计数冒充 family-wide receipt

这套技术路线很先进，
因为它体现的是：

`系统宁可把运行真相拆层，也不把一句强文案伪装成同步兑现。`

但它的先进性也暴露出下一层问题：

`拆层之后，谁来替这些层之间的距离签字。`

尤其是以下三种距离：

1. declaration 到 admission 的距离
2. metadata consumption 到 executor behavior 的距离
3. local outcome 到 official receipt 的距离

## 6. 为什么 runtime-conformance 仍不能被 metadata 冒充

把当前证据再压缩一次，
runtime-conformance 至少比 metadata 多出四层义务：

1. `timing honesty`
   系统必须回答“现在是否真的已发生”，而不是“以后大概率会发生”
2. `mode honesty`
   系统必须回答 interactive / headless / scripted worlds 的 admission 是否相同
3. `abstention honesty`
   系统必须回答 skip 是否发生、为什么发生、谁批准了 skip
4. `receipt honesty`
   系统必须回答结果是否已被汇总、投影并留下正式证据

所以：

`artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer`

天然比

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

更强。

因为前者不只要理解规则，
还要理解 runtime 时间、mode、coverage 与结果证据。

## 7. 苏格拉底式自反诘问：我是不是又把“制度已经记住自己”误当成了“制度已经兑现自己”

如果对这组代码做更严格的自我审查，
至少要追问五句：

1. 如果 `cleanupPeriodDays=0` 已经写得这么强，为什么我还说 conform 不成立？
   因为 declaration truth 不自动等于 execution truth。
2. 如果 `shouldSkipPersistence()` 已经在运行，为什么还不够？
   因为 write suppression 只证明 future path，被 delete 的 old carriers 仍需单独 receipt。
3. 如果 housekeeping 最终会跑，为什么还说 runtime 有裂缝？
   因为 delay、reschedule 与 mode-sensitive admission 意味着“会跑”不等于“此刻已经完成”。
4. 如果 validation-error 下 intentional skip 是更安全的，为什么这不算已经符合？
   因为这恰恰说明 metadata 与 runtime 之间还需要一层更强的行为解释与签收。
5. 如果 `CleanupResult` 已经存在，为什么还说没有 receipt？
   因为 local return value 不是 surfaced stronger-request family receipt。

这组反问最终逼出一句更稳的判断：

`runtime-conformance 的关键，不在系统有没有动作，而在系统能不能诚实证明动作已经在正确时机、正确平面、正确覆盖范围内发生。`

这也是这一章最后必须保留的哲学边界：

`制度会记住自己，不等于制度会执行并签收自己。`
