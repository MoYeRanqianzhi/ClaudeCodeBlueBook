# 安全多账本原则：为什么语义、复制、可见与恢复不能共用同一本账

## 1. 为什么在 `34` 之后还要继续写“多账本原则”

`34-安全提交语义` 已经回答了：

`关键安全事实要按不同提交层来理解。`

但如果继续往第一性原理再压一层，  
还会出现一个更本质的问题：

`既然提交层不同，为什么系统不能把它们都记进同一本账？`

也就是说，  
为什么 Claude Code 没有追求一种看似更简单的设计：

`所有状态都进同一套事件流，所有消费者都读同一份记录。`

源码给出的答案其实非常明确：

`因为语义、复制、可见与恢复面对的对象、时序、持久要求和误判成本根本不是一回事。`

所以这一章的核心问题是：

`Claude Code 为什么必须把关键安全事实拆进多本账，而不是强行压成一套统一账本？`

## 2. 最短结论

从源码看，Claude Code 的安全控制面并不是单账本系统。  
它更像一套多账本系统：

1. 语义账本
2. 复制账本
3. 可见账本
4. 恢复账本

这些账本会互相镜像、互相引用、互相校验，  
但不能互相替代。

所以本章最短结论是：

`成熟的安全控制面，不只需要多层提交边界，还需要多本职责不同的账。`

## 3. 什么叫“安全账本”

这里说的“账本”，不是一定指某个数据库表。  
我在这里的定义是：

`一组面向特定消费对象、拥有特定提交边界和特定恢复语义的事实记录面。`

这意味着四件事：

1. 它记录什么
2. 谁读取它
3. 何时算写成
4. 丢了之后系统会坏在哪

只要这四件事不同，  
它就不该被强行和别的账本合并。

## 4. 第一类账本：语义账本

## 4.1 `sessionState.ts` 维护的是会话语义账本

`src/utils/sessionState.ts:92-133` 很关键。

这段代码最先维护的不是：

- 远端复制
- 前端可见
- 恢复 transcript

而是：

`当前会话在语义上到底是 idle、running 还是 requires_action。`

这里的核心动作是：

1. 更新 `currentState`
2. 通知 `stateListener`
3. 再把部分语义镜像到 metadata 和 SDK event

这说明 `sessionState.ts` 首先在维护一份：

`面向本地会话语义仲裁的账本。`

它的责任不是“让所有人都看到”，  
而是先回答：

`系统此刻在语义上处于什么状态。`

## 4.2 语义账本不能被复制账本或 UI 账本替代

同一段代码已经说明：

- `pending_action` 只是对外镜像
- SDK event 只是对宿主广播
- `task_summary` 清理也是镜像纪律的一部分

这意味着：

`语义账本是源，其他账本是派生面。`

如果把语义账本直接交给 UI 或复制层代理，  
系统迟早会把“别人看到什么”误读成“语义上是什么”。

## 5. 第二类账本：复制账本

## 5.1 `reportState(...)` / `reportMetadata(...)` 面向的是远端复制账本

`src/cli/transports/ccrClient.ts:645-662` 已经把复制账本的职责写得很清楚：

1. `reportState(...)` 把 `worker_status` 和 `requires_action_details` 送进 worker 上传链
2. `reportMetadata(...)` 把 `external_metadata` 送进同一条复制链

这本账不是给本地语义仲裁用的，  
也不是给恢复 transcript 直接用的。  
它首先服务的是：

`远端 worker 状态复制与跨端查询。`

## 5.2 `WorkerStateUploader` 说明复制账本有自己的提交纪律

`src/cli/transports/WorkerStateUploader.ts:4-75` 进一步说明：

1. 复制账本允许 pending
2. 允许 in-flight
3. 允许 retry
4. 允许 top-level last value wins

这些特征都说明：

`复制账本本质上是一个面向远端一致性的复制账，不是一个面向本地语义解释的账。`

如果你把它当成语义账本，  
你会误把：

`还在复制途中`

理解成：

`语义上还没成立。`

## 6. 第三类账本：可见账本

## 6.1 `writeEvent(...)` 明确面向前端可见账本

`src/cli/transports/ccrClient.ts:735-840` 的注释很直接：

1. `writeEvent(...)` 写的是 worker events
2. 这些 events 是 visible to frontend clients via SSE stream
3. stream_event 会 buffer、flush、按顺序出流

这说明这本账首先服务的是：

`外部消费者现在应该看到什么。`

它不承诺：

- transcript 一定能恢复
- 本地语义一定只靠它就能重建
- 复制层一定已经 PUT 成功

它只承诺一件事：

`该给外部看的，现在按正确顺序给出去。`

## 6.2 `sdkEventQueue.ts` 和 `print.ts` 又补了一条本地可见链

`src/utils/sdkEventQueue.ts:37-41`、`77-96` 与 `src/cli/print.ts:2218-2244`、`2456-2466` 一起说明：

1. 有些 SDK event 会直接 drain 到 output stream
2. `heldBackResult` 会推迟 result 对外可见
3. `idle` 事件要在正确时序下才对外释放

也就是说，  
Claude Code 的“可见账本”甚至不只一条输出通路，  
但它们共同服务的仍然是同一个问题：

`外部现在应该看见什么，而不是系统内部已经完成了什么。`

## 7. 第四类账本：恢复账本

## 7.1 `writeInternalEvent(...)` 明确声明有一套“不对前端可见”的恢复账本

`src/cli/transports/ccrClient.ts:793-820` 非常关键。

作者直接写明：

1. internal events are NOT visible to frontend clients
2. 它们存的是 worker-internal state
3. 它们 needed for session resume

这几句话几乎已经把“多账本原则”写成源码注释了：

`给恢复系统用的账，不应该拿前端可见账来代替。`

## 7.2 `hydrateFromCCRv2InternalEvents(...)` 说明恢复账本的消费者完全不同

`src/utils/sessionStorage.ts:1632-1710` 又把这件事落实得更彻底：

1. 它读的是 internal events
2. 它写的是 foreground transcript 与 subagent transcript 文件
3. 它服务的是 session resume

这说明恢复账本的消费者不是：

- 当前在线前端
- 当前这次 PUT 复制链
- 当前 UI 状态机

而是：

`未来的下一次恢复。`

所以这本账天然就不该和“当前谁看见了什么”共用同一语义。

## 8. 为什么不能共用同一本账

我会把原因压成四条。

## 8.1 消费对象不同

语义账本给本地状态机看。  
复制账本给远端 worker 面和跨端查询看。  
可见账本给前端与宿主消费面看。  
恢复账本给未来的 resume 路径看。

既然读者都不同，  
把它们强行压成一本账，  
就会让每个读者都读到一堆不该由自己解释的字段。

## 8.2 提交边界不同

语义账本在 `currentState` 变化时成立。  
复制账本在 uploader send success 后成立。  
可见账本在 flush / hold-back / output drain 后成立。  
恢复账本在 internal events flush 并可被 hydrate 时成立。

既然提交边界不同，  
单账本会不断制造一句危险的话：

`到底是“写了”，还是“哪一层写了”？`

## 8.3 时序要求不同

可见账本非常关心顺序和用户体感。  
恢复账本非常关心重建完整性。  
复制账本非常关心远端一致性与 retry。  
语义账本非常关心当前仲裁结果。

这四者的时序优化方向根本不同，  
所以强行共账，  
只会让每一类目标都被另一类目标拖坏。

## 8.4 误判成本不同

把语义账本读错，  
你会误判权限或状态。

把复制账本读错，  
你会误判远端是否已同步。

把可见账本读错，  
你会误判用户是否真的看到结果。

把恢复账本读错，  
你会误判下次 resume 是否还能重建。

既然坏法都不同，  
它们就不该共用一套“已生效”话语。

## 9. 把这些账本压成一条底层公理

我会把它压成一句话：

`关键安全事实可以跨账本镜像，但不能失去“这是哪一本账”的身份。`

也就是说，  
同一个事实可以在多个地方出现，  
但它在每本账里的含义不同：

1. 在语义账里，它回答当前状态
2. 在复制账里，它回答远端是否拥有
3. 在可见账里，它回答外部是否该看到
4. 在恢复账里，它回答未来是否能重建

如果开发者把这些副本当成“完全等价”，  
系统就会回到 `30-34` 一直在试图避免的那种混乱：

`多层事实被错误压扁。`

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只是把状态分层，还在于它敢于维护多本职责不同的账。`

这对其他 Agent 平台构建者的启示很直接：

1. 不要为了“统一”而合并语义账和恢复账
2. 不要用前端可见事件冒充恢复账本
3. 不要用复制状态冒充本地语义仲裁
4. 不要让 UI 文案承担账本职责
5. 每本账都应当明确自己的读者和提交边界

真正的工程节制不是“所有事一张表搞定”，  
而是：

`只在真正同语义、同读者、同边界的事实之间做合并。`

## 11. 哲学本质

这一章更深层的哲学是：

`不是所有“记录”都在回答同一个问题。`

很多系统喜欢迷信统一账本，  
仿佛“所有状态都在一处”天然更真。

Claude Code 的源码反过来说明：

`更成熟的做法，是承认不同事实面向不同时间尺度、不同读者和不同风险。`

因此，多账本不是冗余，  
而是：

`对不同问题保持诚实。`

## 12. 苏格拉底式反思：这章最容易犯什么错

### 12.1 多账本是不是天然更复杂、更难维护

是，会更复杂。  
但如果这些账本本来就在回答不同问题，  
把它们假装成一本账只会把复杂性藏起来，  
不会真的消灭它。

### 12.2 有没有可能以后把它们重新统一

只有在以下三件事同时成立时才可能：

1. 读者相同
2. 提交边界相同
3. 误判成本相同

否则“统一”通常只是把语义差异压扁。

### 12.3 多账本会不会重新制造多真相源冲突

会有这个风险。  
但 Claude Code 前面几章已经说明，  
它不是靠假装只有一本账解决冲突，  
而是靠：

1. 真相层级
2. 仲裁规则
3. 单写者
4. 提交边界

来解决冲突。

## 13. 结语

`34` 回答的是：关键安全事实何时算已提交。  
`35` 继续推进后的结论则是：

`不同层之所以必须有不同提交边界，根本原因是它们本来就在记不同的账。`

这意味着 Claude Code 更深层的安全启示之一是：

`不要追求虚假的单账本统一，而要让每一本账都诚实地服务自己的读者、边界与恢复语义。`
