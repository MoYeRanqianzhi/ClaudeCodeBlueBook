# 统一安全控制台最短诊断路径：如何把状态、证明断点、控制动作与回读验证闭成一条链

## 1. 为什么在 `24` 之后还要写“最短诊断路径”

`24-统一安全控制台卡片设计` 已经把统一安全控制台压成了七张核心卡片。  
但如果继续往实现落地层推进，仍然会剩下一个真正决定用户体验的问题：

`用户看懂了以后，下一步该怎么做？`

如果控制台只能做到：

1. 展示状态
2. 展示因果
3. 展示卡片

却不能把用户带到最短动作，  
那它仍然只是更高级的观察面，  
不是更高级的控制面。

所以这一章的核心问题是：

`Claude Code 的统一安全控制台，如何把状态、证明断点、控制动作和回读验证闭成一条真正可执行的最短诊断路径？`

## 2. 最短结论

从源码看，Claude Code 现在离“统一安全控制台”真正还差的一步，不只是卡片设计，  
而是：

`动作闭环设计。`

也就是每次安全问题出现时，系统应当能把用户带过四步：

1. 看见当前状态
2. 知道哪条证明链断了
3. 获得最短控制动作
4. 动作执行后立即回读验证新的状态

这四步缺一不可。  
因为只有这样，控制台才不是静态仪表盘，而是：

`一个可诊断、可纠正、可复核的安全控制回路。`

## 3. 当前源码已经有很多“动作”，只是还没和状态卡闭环

### 3.1 控制 schema 已经暴露出不少可操作入口

`src/entrypoints/sdk/controlSchemas.ts` 已经定义了不少和安全控制直接相关的动作：

- `mcp_status`
- `reload_plugins`
- `mcp_reconnect`
- `mcp_toggle`
- `get_settings`
- `apply_flag_settings`

这些动作非常关键。  
因为它们说明系统并不缺“可执行按钮”。

它真正缺的是：

`哪种状态对应哪个动作，哪个动作做完以后该回读哪一组状态。`

### 3.2 `get_settings` 尤其重要，因为它已经天然适合做“主权回读”

`SDKControlGetSettingsResponseSchema` 不只返回 `effective`，  
还返回：

- `sources`
- `applied`

这意味着系统其实已经有能力在动作后重新告诉用户：

1. 合并后的配置是什么
2. 原始 per-source settings 是什么
3. 真正会作用于运行时的 resolved values 是什么

这不是普通“设置查看”，  
而是天然适合做：

`主权链与来源链的回读验证。`

### 3.3 `mcp_status`、`reload_plugins`、`mcp_reconnect`、`mcp_toggle` 已经接近完整的外部能力修复回路

这一组 schema 说明外部能力面其实已经有很完整的控制动作：

1. 读当前连接状态
2. 重新加载插件与其附带的 MCP
3. 对失败或断开的 server 执行 reconnect
4. 对 server 执行 enable / disable

这已经很接近一个成熟的“外部能力卡”动作闭环。  
缺的不是动作，  
而是：

`动作应该从哪个卡片发起，以及动作执行后默认回读哪张卡。`

## 4. 当前最大的缺口：展示层和适配层把动作闭环切断了

### 4.1 `sdkMessageAdapter.ts` 说明结构化状态在宿主边界被压扁

`src/remote/sdkMessageAdapter.ts` 里最值得注意的，不是某条消息被忽略，  
而是它展现了一种更普遍的问题：

1. `status` 被压成很薄的文本
2. `auth_status` 被直接忽略
3. 复杂子域状态没有被保留下来作为可操作对象

这意味着一旦跨到某些宿主或适配层，  
系统会从：

`结构化状态 + 可进一步派生动作`

退化成：

`一条文本提示`

而一旦退化成文本，  
后续统一控制台就不得不重新猜：

1. 当前到底是什么状态
2. 哪条证明链断了
3. 应该给用户什么动作

所以技术上最关键的改进之一不是更多 UI，  
而是：

`不要在适配层过早丢掉结构化状态。`

### 4.2 `PromptInputFooter` 和 `bridgeStatusUtil` 说明很多状态目前只是“轻提示”

`bridgeStatusUtil.ts` 会把 bridge 状态压成：

- `Remote Control failed`
- `Remote Control reconnecting`
- `Remote Control active`
- `Remote Control connecting…`

而 `PromptInputFooter.tsx` 里的 Bridge pill 只在很有限的条件下展示这些标签。

这很适合 footer，  
但它说明现在的 bridge 状态主要被设计成：

`轻量状态提示`

而不是：

`驱动用户完成最短诊断路径的操作对象。`

这再次说明统一控制台真正缺的是“状态到动作的闭环层”。

### 4.3 `remoteManagedSettings/index.ts` 暴露出另一类动作闭环：审批后的状态回退 / 沿用缓存

`src/services/remoteManagedSettings/index.ts` 里可以看到一个非常重要的模式：

1. 拉取新设置
2. 检查危险变更
3. 用户拒绝则保留 cached settings
4. 用户接受则应用新设置

这说明系统其实已经具备一种很成熟的闭环：

`变化 -> 审批 -> 应用或回退 -> 继续运行`

这几乎就是增量审批卡的天然动作模型。

问题在于，当前这个闭环主要存在于逻辑链里，  
还没有被升格成控制台里的显式“最短动作路径”。

## 5. 从第一性原理看，真正的安全控制台必须形成四段闭环

我会把统一安全控制台的最小闭环压成四段：

### 5.1 第一段：状态识别

控制台先回答：

`现在发生了什么。`

这对应：

- status fields
- state cards
- bridge labels
- MCP status
- permission result

### 5.2 第二段：断点定位

接着控制台必须回答：

`哪条证明链断了。`

例如：

- 主体链没过
- 主权链冲突
- 授权链被 safetyCheck 卡住
- 外部能力链语义不成立
- 增量审批链待确认

### 5.3 第三段：最短动作

然后控制台不能只说“请检查设置”。  
它必须给出：

`这一类断点当前最短、最可执行、最不容易误操作的动作。`

例如：

- 重新登录
- 刷新 scope
- 打开来源详情
- reconnect MCP
- reload plugins
- accept / reject managed changes
- apply flag settings

### 5.4 第四段：回读验证

最后，控制台必须再告诉用户：

`动作之后该看哪里，来确认问题是否真的修复。`

这一步非常关键。  
没有回读验证，用户只会在动作和结果之间来回猜。

而源码里其实已经有不少天然适合做回读的结构：

- `get_settings`
- `mcp_status`
- bridge status
- permission reason

所以真正的控制台不应止于“有个按钮”，  
它必须提供：

`动作后默认回读哪张卡、哪组字段、哪条证明链。`

## 6. 七张卡应如何被接成最短诊断路径

### 6.1 主体与资格卡

最短路径应是：

1. 显示主体字段
2. 判断主体链是否断
3. 如果断，默认动作是 `claude auth login` 或刷新 account info
4. 动作后回读 auth status 和 bridge entitlement

### 6.2 主权与来源卡

最短路径应是：

1. 显示当前 active sources 与 policy origin
2. 判断当前主导 source 是否符合预期
3. 如果不符合，默认动作是打开来源详情或跳到可编辑 source
4. 动作后回读 `get_settings`

### 6.3 仲裁与权限卡

最短路径应是：

1. 显示 decisionReason.type 和当前 mode
2. 判断是 rule、mode、classifier 还是 safetyCheck 主导
3. 如果可修，默认动作是改规则 / 调 mode / 走人工审批
4. 动作后回读相同动作的 decisionReason

### 6.4 包络与执行壳卡

最短路径应是：

1. 显示 fs/network 壳层
2. 判断当前阻断是路径壳、写壳还是网络壳
3. 默认动作是调整 sandbox 或允许范围
4. 动作后回读 SandboxConfigTab 或等价结构化壳层数据

### 6.5 外部能力卡

最短路径应是：

1. 显示连接状态与能力语义
2. 判断当前问题是 connect、scope、disabled 还是 failed
3. 默认动作是 reconnect / toggle / reload / 打开对应子页
4. 动作后回读 `mcp_status` 或 bridge state

### 6.6 增量审批卡

最短路径应是：

1. 显示变更是什么
2. 判断是否需要重新确认
3. 默认动作是 accept / reject
4. 动作后回读当前 active settings 是否变化

### 6.7 断链诊断卡

这张卡不直接拥有全部原始状态，  
但它拥有最关键的派生逻辑：

1. 汇总哪条链断了
2. 给出责任主体
3. 给出最短动作
4. 指定动作后的默认回读卡片

因此它本质上是：

`控制台里的路由器。`

## 7. 技术先进性：Claude Code 离“行动型控制台”其实已经很近

从技术角度看，它已经具备四个非常强的基础：

1. 结构化状态字段已经存在
2. 局部控制动作已经存在
3. 很多因果 reason 已经存在
4. 某些边界变化已经具备审批闭环

这意味着它不需要先发明一套全新的安全系统。  
真正要做的是：

`把现有结构化状态和现有控制动作，用证明链与回读验证重新绑起来。`

换句话说，Claude Code 已经很接近：

`从观察型控制台升级成行动型控制台。`

## 8. 这套设计的哲学本质

从第一性原理看，真正成熟的安全系统不应只回答：

`现在是什么状态？`

它还必须回答：

1. 为什么是这个状态
2. 这个状态意味着哪条证明链断了
3. 最短纠正动作是什么
4. 动作后如何验证新状态

这四件事加起来，才是完整的安全控制闭环。

所以统一安全控制台的哲学本质，不是“更漂亮的状态页”，  
而是：

`把系统知道的真相，压成用户能执行的最短诊断路径。`

## 9. 苏格拉底式追问

### 问：为什么统一控制台不能停留在“解释层”？

答：因为如果没有动作和回读，解释只是更高级的旁观。

### 问：为什么要强调“最短动作”而不是“所有动作”？

答：因为控制台的首要任务不是展示完备性，而是减少误操作和盲目试错。

### 问：为什么回读验证必须和动作绑定？

答：因为没有回读，用户无法判断动作究竟修复了主体链、主权链，还是只改了表象状态。

### 问：这章最重要的技术启示是什么？

答：不要在状态适配层过早把结构化状态压扁成文本，否则后续控制台只能重新猜因果和动作。

## 10. 本章结论

统一安全控制台真正要做成产品，关键不在：

- 再多一张状态页
- 再多一层提示
- 再多一个总开关

而在于：

`把状态、证明断点、控制动作和回读验证闭成一条最短诊断路径。`

Claude Code 现有源码已经提供了这条路径所需的大部分零件。  
下一代最值得做的，不是继续堆信息，  
而是把这些零件组织成：

`一个真正能带用户完成诊断、修复和复核的行动型安全控制台。`
