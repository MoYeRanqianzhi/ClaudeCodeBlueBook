# 安全资格生命周期账本：为什么统一控制台不能只保存当前结果，还必须保存字段为何升级、为何留场、为何退场的时间性依据

## 1. 为什么在 `107` 之后还必须继续写“生命周期账本”

`107-安全资格字段生命周期协议化` 已经回答了：

`upgrade-threshold、stay-condition 与 retire-trigger 不应继续散落在局部实现里，而应升级成统一 field-lifecycle protocol。`

但如果继续往下追问，  
还会碰到一个更底层的问题：

`这套协议到底应该存放在哪里？`

如果答案只是：

`继续放在当前 AppState 的结果字段里，或者运行时临时算出来`

那统一协议仍然无法真正变成：

1. 可审计的
2. 可追责的
3. 可解释的
4. 可跨 surface 复用的

原因很简单：

`安全生命周期治理的是时间轴上的合法变化，而不仅是某一时刻的显示结果。`

所以 `107` 之后必须继续补出的下一层就是：

`安全资格生命周期账本`

也就是：

`统一控制台不能只保存 current result，还必须保存字段为何升级、为何留场、为何退场的时间性依据。`

## 2. 最短结论

从源码看，Claude Code 当前的状态基础设施更像一套“快照存储”，  
而不是一套“生命周期账本”：

1. AppState 主要保存的是当前 bridge/plugin/MCP/notification 的结果字段  
   `src/state/AppStateStore.ts:133-225`
2. 默认状态初始化同样只初始化这些结果位，而不初始化任何 transition provenance  
   `src/state/AppStateStore.ts:485-535`
3. 底层 `createStore()` 只是通用 `setState(prev => next)` 容器，没有事件日志、transition record 或 causal ledger  
   `src/state/store.ts:10-33`
4. `useSetAppState()` 暴露的是裸 `setState` 写口，说明调用方通常直接写最终结果，而不是提交结构化 lifecycle event  
   `src/state/AppState.tsx:165-178`
5. `onChangeAppState()` 只处理少量横切副作用，例如 permission mode、model、config persistence，并没有承接字段生命周期的审计责任  
   `src/state/onChangeAppState.ts:43-170`

所以这一章的最短结论是：

`Claude Code 当前已经具备生命周期治理逻辑，但它主要依附于快照状态和局部控制路径，尚未沉淀为可审计的生命周期账本。`

再压成一句：

`快照足以渲染当前，账本才足以解释当前。`

## 3. 当前 AppState 基础设施本质上是一套结果快照系统

## 3.1 状态树保存的是“现在是什么”，而不是“为什么成了这样”

`src/state/AppStateStore.ts:133-225` 当前对安全相关对象的保存方式非常典型：

1. bridge 用的是：
   `replBridgeEnabled/Connected/SessionActive/Reconnecting/Error/...`
2. plugin 用的是：
   `needsRefresh`
3. MCP 用的是：
   `clients/tools/commands/resources/pluginReconnectKey`
4. notifications 用的是：
   `current/queue`

这些字段都很有价值，  
但它们主要回答的是：

`现在的投影结果是什么`

却没有回答：

1. 哪个 evidence bundle 使它升级
2. 谁拥有这次 upgrade 的签发权
3. 它现在为何仍可留场
4. 它下一次会由谁在什么条件下撤场

所以当前 AppState 更像：

`current projection snapshot`

而不是：

`lifecycle explanation ledger`

## 3.2 默认状态初始化再次证明：系统默认假设“结果字段”就是核心持久对象

`src/state/AppStateStore.ts:485-535` 初始化时设置了：

1. `replBridgeConnected = false`
2. `replBridgeError = undefined`
3. `pluginReconnectKey = 0`
4. `needsRefresh = false`
5. `notifications.current = null`
6. `notifications.queue = []`

这些初始值说明系统的最小状态模型目前是：

`字段当前有没有、是什么`

而不是：

`字段刚刚经历了哪种 transition、该不该被解释为新的边界事实`

这对一般 UI 足够，  
但对安全控制面来说仍然偏薄。  
因为安全系统真正难的，  
恰恰不是静态状态，  
而是：

`状态改变的合法性。`

## 4. `createStore` 暴露了更深的结构事实：底层容器本身没有时间语义

`src/state/store.ts:10-33` 非常干净：

1. `getState`
2. `setState(updater)`
3. `subscribe`

`setState` 做的事情也非常直接：

1. 读 `prev`
2. 算 `next`
3. `Object.is(next, prev)` 则跳过
4. 否则整体替换并通知 listeners

这意味着底层容器本身不知道：

1. 是什么事件触发了这次变化
2. 这次变化是 upgrade 还是 retire
3. 是 threshold 过线还是 timeout 过期
4. 是 explicit invalidation 还是 true completion

也就是说，  
当前 store 是：

`state snapshot container`

而不是：

`temporal governance container`

这不是 bug，  
但它解释了为什么上一章说的 protocolization 到这里还没有完成：

`协议即便被定义出来，也还缺一个承载时间依据的账本层。`

## 5. `useSetAppState` 进一步说明：当前写口是“结果导向写入”，不是“事件导向写入”

## 5.1 写口暴露的是裸 updater

`src/state/AppState.tsx:165-178` 直接把 `store.setState` 作为 `useSetAppState()` 暴露出来。

这意味着调用方通常做的是：

1. 读旧状态
2. 直接写出新结果

例如：

1. 把 `replBridgeConnected` 改成 `true/false`
2. 把 `replBridgeError` 改成某个字符串或 `undefined`
3. 把 `needsRefresh` 改成 `true/false`
4. 把 `notifications.current` 换成另一个对象

这种接口很灵活，  
但其代价也很清楚：

`写入动作本身并不强制携带“为什么写”的结构化语义。`

## 5.2 这会把 lifecycle legality 继续外包给调用方自觉

换句话说，  
当前系统默认调用方自己知道：

1. 何时配升级
2. 何时不该退场
3. 何时只是 timeout
4. 何时属于 fatal revocation

这恰恰就是前一章所谓：

`规则已在，协议未成`

在状态写口层的直接表现。

## 6. `onChangeAppState` 说明当前的横切层只做少量副作用同步，并不承担生命周期审计

`src/state/onChangeAppState.ts:43-170` 已经展示了一个非常成熟的思路：

1. mode 变化走统一 diff choke point
2. model 变化持久化到 settings
3. verbose / expandedView / tungstenPanelVisible 等同步到 config
4. settings 变化触发 auth cache 与 env 应用

这证明 Claude Code 作者已经知道：

`某些重要状态变化，需要在统一 diff 入口被横切处理。`

但与此同时，  
这一层并没有承接：

1. notification upgrade/retire 审计
2. bridge failure threshold/fuse/retire 审计
3. plugin pending -> full refresh complete 的 completion 审计
4. pointer keep/clear 的 retire reason 审计

所以现状可以概括成一句：

`统一 diff choke point 已经存在，但安全字段生命周期还没有被接入这条统一审计链。`

## 7. 源码里已经存在一个重要提示：作者知道“事件源”在某些对象上是必要的

`src/state/AppStateStore.ts:127-132` 有一句非常重要的注释：

`remoteBackgroundTaskCount` 是 event-sourced from WS events。

这其实是在告诉我们两件事：

1. 作者并不排斥 event-sourced / ledger-like 建模
2. 当对象的意义依赖时间序列时，作者愿意显式承认“当前值来自事件流”

这正好反过来强化了本章判断：

`安全字段生命周期本来就比背景任务计数更依赖时间序列，因此更应进入账本化建模。`

## 8. 从第一性原理看，为什么安全控制面必须有 lifecycle ledger

原因并不复杂：

`安全系统的关键对象不是静态值，而是合法转换。`

换句话说，  
真正被保护的不是：

1. `current = failed`
2. `needsRefresh = true`
3. `pointer exists`

而是：

1. 这个 failed 是如何成立的
2. 这个 pending 为什么还没被撤销
3. 这个 pointer 为什么仍然合法
4. 这个绿色词为什么现在还能继续说

所以如果系统只保存快照，  
它最多能回答：

`现在是什么`

却很难回答：

`为什么现在仍然配是这样`

而安全控制面的本质，  
恰恰就在第二句。

## 9. 生命周期账本至少应该记录什么

如果把前面所有章节压成一个最小 ledger schema，  
我会建议每条 lifecycle record 至少包含：

## 9.1 对象身份

1. `field_id`
2. `field_family`
3. `object_scope`
4. `surface_scope`

## 9.2 发生了什么变化

1. `transition_kind`
   例如 `upgrade / renew / invalidate / timeout / retire / complete`
2. `from_state`
3. `to_state`
4. `priority_at_transition`

## 9.3 为什么发生

1. `threshold_kind`
2. `evidence_bundle_id`
3. `evidence_owner`
4. `reason_code`

## 9.4 谁授权了这次变化

1. `transition_owner`
2. `retire_owner`
3. `completion_signer`
4. `forbidden_earlier_owner`

## 9.5 时间依据

1. `emitted_at`
2. `expires_at`
3. `freshness_basis`
4. `renewal_mechanism`

## 9.6 用户解释与审计

1. `hidden_truth_hint`
2. `handoff_route`
3. `overclaim_ceiling`
4. `premature_retire_risk`

一旦有了这层账本，  
系统就能同时服务三类需求：

1. UI 渲染
2. 审计与 review
3. 面向用户的自解释

## 10. 为什么“账本”比“继续加字段”更关键

很多系统在这里容易犯一个错：

`发现信息不够，就继续往当前 snapshot 上多挂几个字段。`

但这还不够。  
因为 snapshot field 再多，  
它也只是：

`当前一帧的结果`

而 lifecycle ledger 记录的是：

`这一帧是怎么合法地来到这里的`

所以两者服务的是不同层级：

1. snapshot 回答“看见什么”
2. ledger 回答“凭什么看见这个”

这也是为什么下一代统一安全控制台不该只做成更大的对象树，  
而应做成：

`snapshot + lifecycle ledger` 的双层架构。

## 11. 技术先进性：Claude Code 现在最值得学习的，不是已经有账本，而是已经具备账本化升级的条件

Claude Code 的先进之处在于：

1. 它已经有很多明确的 lifecycle rule
2. 它已经有统一 store
3. 它已经有统一 diff hook
4. 它已经在部分对象上承认 event-sourced 价值

也就是说，  
它并不是离 lifecycle ledger 很远，  
而是：

`只差把已有的成熟实现抽象提升一层。`

这对平台构建者的启示非常大：

`真正好的架构演进，不是推翻重写，而是把已经反复证明正确的局部控制逻辑升级成全局账本协议。`

## 12. 苏格拉底式反思：如果要把这一层做得更好，还应继续追问什么

可以继续追问七个问题：

1. 当前哪些安全字段最需要时间性依据，却仍只保存了当前值
2. 当前哪些高风险 retire 动作还无法被统一审计
3. 当前统一控制台若想解释“为什么这个字段现在还在”，还需要从多少处分支重新推理
4. 当前哪些 transition 是 deterministic 的，已经适合提炼成账本事件
5. 当前哪些 transition 仍太依赖局部上下文，还需要先做 protocolization 再账本化
6. 当前是否应该让 `useSetAppState` 之外再增加一层显式 lifecycle dispatch
7. 当前哪些 overclaim/underclaim 风险，本质上是“没有时间性依据可查”导致的

这些问题的核心只有一句：

`如果系统只能展示结果、不能追溯结果，它就还没有真正掌握自己的安全时间轴。`

## 13. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正成熟的下一步不是：

`继续把生命周期规则藏在快照字段和局部 setState 里，`

而是：

`建立统一的 lifecycle ledger，把字段为何升级、为何留场、为何退场的时间性依据正式纳入控制面账本。`

因为只有这样，  
系统才不只是：

`知道当前是什么，`

而且能正式回答：

`为什么当前仍然配是这样。`
