# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `198` 之后还必须继续写 `199`

`198-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层` 已经回答了：

`stronger-request cleanup 线即便知道旧世界结束后该留下什么最小残留标记，也还需要 tombstone governor 去决定这些 marker 的形态、消费方与后续约束。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“有墓碑就等于以后怎么回来也清楚了”的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/utils/plugins/cacheUtils.ts:69-92,122-171` 的 `.orphaned_at` 清除与 orphan GC 证明 marker clearing 受 authoritative installed state 治理，而不是由 marker 自己决定
2. `src/utils/plugins/refresh.ts:1-85` 与 `src/commands/reload-plugins/reload-plugins.ts:1-42` 的 active plugin refresh grammar 证明就算 marker 清掉、对象回到磁盘世界，也不自动回到 active world
3. `src/services/plugins/PluginInstallationManager.ts:47-113,149-176` 与 `src/hooks/useManagePlugins.ts:288-303` 的 `needsRefresh` / `/reload-plugins` 设计，证明 repo 明确拒绝把“对象重新存在”偷写成“当前会话已经重新接纳它”
4. `src/utils/plans.ts:164-279` 的 `copyPlanForResume()`、`copyPlanForFork()` 与 `recoverPlanFromMessages()` 证明 plan resurrection 不只是 undelete，还要治理 lineage、evidence 与 identity policy
5. `src/utils/conversationRecovery.ts:540-555` 把 `copyPlanForResume()` 接进恢复主线，说明 resurrection 不是局部 helper 偶发行为，而是正式恢复制度的一层
6. `src/utils/sessionStorage.ts:1466-1473` 的 tombstone 删除接口只回答怎样删，不回答怎样复活；这反过来证明 tombstone consumption 本身不自动携带 resurrection grammar

这说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 surface、什么 active layer 下，这个旧对象还能重新回到 current world。`

所以 `198` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`

不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk/materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery` 就等于 `plan resurrection` 的全部语义。`

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线是否完全复用这套 grammar，仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类“stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer”证据：

1. `cacheUtils.ts:69-92,122-171` 证明 `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
2. `refresh.ts:1-85` 与 `/reload-plugins` command 证明 marker clearing / disk presence 不等于 active components 已重新进入当前 session
3. `PluginInstallationManager.ts:47-113,149-176` 与 `useManagePlugins.ts:288-303` 证明 repo 明确要求 `/reload-plugins` 作为 re-entry signal，而不是默认自动接纳
4. `plans.ts:164-279` 证明 plan resurrection 需要 lineage、evidence 与 identity policy，而不是简单 undelete
5. `conversationRecovery.ts:540-555` 把 resurrection 接进恢复主线，说明这不是偶发 helper，而是正式的 recovery grammar
6. `sessionStorage.ts:1466-1473` 的 tombstone path 只回答怎样删除旧对象，不回答它何时、以什么身份重新回来

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据与何种层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

## 4. 第一性原理：tombstone governance 回答“留什么残留物”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

stronger-request cleanup tombstone governor 回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

stronger-request cleanup resurrection governor 回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk/materialized world，还是 active/runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句“反正 marker 还在或 marker 已清掉”，
系统就会制造五类危险幻觉：

1. marker-clear-means-resurrection illusion  
   只要墓碑被清掉，就误以为对象已经正式回到 current world
2. disk-return-means-active-return illusion  
   只要对象重新出现在磁盘上，就误以为运行时已经重新接纳它
3. content-recovered-means-identity-restored illusion  
   只要历史内容被恢复，就误以为原 identity 和原 ownership 也被恢复
4. tombstone-is-bidirectional illusion  
   误以为同一 marker grammar 既定义退役，也自动定义复活
5. any-old-object-can-come-back illusion  
   误以为只要历史对象曾经存在，就都可以不经额外授权重新成为 current truth

所以从第一性原理看：

`tombstone governor` 决定对象退役后留下什么最小痕迹；  
`resurrection governor` 决定对象凭什么、以什么身份、在什么层面重新回来。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup tombstone governance 已经等于 resurrection governance，为什么 repo 还要把 `removeOrphanedAtMarker()`、`refreshActivePlugins()` 与 `/reload-plugins` 拆成多层？
   因为“marker 清掉”与“active world 接纳它”根本不是同一个时刻。
2. 如果对象内容恢复就等于对象身份恢复，为什么 `copyPlanForFork()` 还要显式生成 new slug？
   因为 resurrection 还要治理 identity，不只是治理 bytes。
3. 如果 tombstone path 天然自带复活逻辑，为什么 `sessionStorage.ts` 的 tombstone 删除接口只回答怎样删、不回答怎样回来？
   因为删除 grammar 与 re-entry grammar 是两层不同主权。

## 5. plugin 线先证明：`.orphaned_at` 被清掉，并不等于插件已经重新回到当前 active world

`src/utils/plugins/cacheUtils.ts:69-92` 很清楚。
orphan cleanup 的 Pass 1 会：

1. 先读取 `installed_plugins.json`
2. 形成 authoritative `installedVersions`
3. 只对这个集合里的 version path 执行 `removeOrphanedAtMarker()`

这说明：

`谁配清墓碑`

不是 marker 自己决定的，
而是由 authoritative installed state 决定。

但这只是第一步。
`src/utils/plugins/refresh.ts:1-85` 又明确把 plugin world 拆成三层：

1. intent
2. materialization
3. active components

而 `refreshActivePlugins()` 被直接命名为：

`Layer-3 refresh primitive`

`src/commands/reload-plugins/reload-plugins.ts:1-42` 也进一步说明：
用户显式运行 `/reload-plugins`，才会触发 `refreshActivePlugins(context.setAppState)`。

这说明 plugin resurrection 至少包含两层不同问题：

1. 墓碑能不能清
2. 清掉后是否重新进入 active components world

源码明确告诉我们：
这两件事不是同一个主权。

## 6. `needsRefresh` 再证明：repo 明确拒绝把“对象重新存在”偷写成“当前会话已经重新接纳它”

`src/services/plugins/PluginInstallationManager.ts:47-113,149-176` 特别重要。

它写得很直白：

1. new installs 才可能 auto-refresh
2. updates only 则设置 `needsRefresh`
3. `src/hooks/useManagePlugins.ts:288-303` 再提示用户：`Plugins changed. Run /reload-plugins to activate.`
4. 注释明确要求：`Do NOT auto-refresh`

也就是说，
Claude Code 在 plugin 线上明确承认：

`对象已经在 disk world 更新`

和

`对象已经被当前 active world 接纳`

不是同一件事。

这条设计非常先进。
它避免系统因为“墓碑清了”或“新对象到盘了”就提前声称当前运行时已恢复正确。

## 7. plan 线给出第二个强正例：对象复活时，系统还要治理它是沿原 identity 回来，还是以新 identity 回来

`src/utils/plans.ts:164-229` 的 `copyPlanForResume()` 很值钱。

它不是看到 plan file 缺失就粗暴恢复。
它要求：

1. log 里确实有 slug
2. 把 slug 绑定回 original session ID
3. 只在 remote session 才尝试 recovery
4. 优先用 file snapshot
5. 失败后才回到 message history

这说明 plan resurrection 的本体不是：

`内容还在，就给你拉回来`

而是：

`只有在 lineage 和 evidence 都成立时，才允许对象重新进入 current session world。`

更关键的是 `copyPlanForFork():234-258`：

1. 不复用 original slug
2. 生成 new slug
3. 写入 new file 避免 clobber

这直接证明：

`内容回来`

不等于

`原身份回来`

也就是说 resurrection governance 还必须回答：

`re-entry 后对象到底算 old identity 还是 new identity。`

## 8. conversationRecovery 与 tombstone 删除路径一起证明：复活不是 tombstone 的反向按钮，而是独立的恢复制度

`src/utils/conversationRecovery.ts:540-555` 并不是随手调用 `copyPlanForResume()`。
它把 plan recovery 接进了恢复主线，
说明 resurrection 不是局部 helper 的偶发行为，而是正式恢复制度的一部分。

与此同时，
`src/utils/sessionStorage.ts:1466-1473` 的 tombstone 删除接口只回答：

`从 transcript 中删掉被 tombstone 的消息。`

它并没有再回答：

`这些对象在什么条件下还能回来。`

这说明：

1. tombstone consumption 是 deletion grammar
2. resurrection governance 是 re-entry grammar
3. 前者的存在不自动携带后者

对 stronger-request cleanup 线的启示非常直接：

`旧 cleanup marker 留下来了，不等于旧 cleanup object 就获得了重新进入 current world 的资格。`

## 9. 技术先进性与哲学本质：Claude Code 真正先进的地方，是它把“允许历史对象回来”继续拆成身份、证据与激活三层

从技术角度看，
Claude Code 在这条线上的先进性，
不只是它允许某些旧对象回来，
而是它已经在多个子系统里承认：

1. marker clearing 不等于 active return
2. disk return 不等于 runtime acceptance
3. content recovery 不等于 identity restoration
4. recovery 不是一个单点动作，而是 evidence、identity 与 activation 三层 choreography

这背后的哲学本质是：

`真正成熟的系统，不只知道怎样让历史对象留下痕迹，也知道什么样的历史对象配回来、凭什么回来、回来后算谁。`

这也带来三条直接技术启示：

1. 任何 stronger-request cleanup grammar 若只定义 tombstone marker、不定义 resurrection gate，就仍缺 re-entry authority
2. plugin 的 `remove marker -> needsRefresh -> /reload-plugins -> active refresh` 四段式说明，复活不是墓碑的自然反面，而是单独的 current-world admission
3. plan 的 `resume` 与 `fork` 双路径说明，成熟的 resurrection governance 必须同时治理 lineage 与 identity，而不是只治理字节级恢复

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要定义了墓碑，就已经自动拥有了决定旧对象怎样回来的主权。`

而是：

`repo 在 authoritative marker clearing、active refresh、explicit re-entry gate 与 lineage-based plan recovery 上已经明确展示了 resurrection governance 的存在；因此 artifact-family cleanup stronger-request cleanup-tombstone-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

再压成最后一句：

`墓碑负责让对象退出后不被误读；复活，才负责决定它凭什么重新被接纳。`
