# 安全完成投影：为什么不同宿主只能看到完成签字链的子集，不能把局部完成冒充全链完成

## 1. 为什么在 `42` 之后还要继续写“完成投影”

`42-安全动作完成权` 已经回答了：

`动作执行不等于动作完成，必须由 authoritative 回读链签字。`

但如果继续往第一性原理再压一层，  
还会剩下一个更现实的问题：

`不同宿主到底能看到这条完成签字链里的哪一部分？`

因为在 Claude Code 的源码里，  
“完成”并不是只有一种对外表面。  
不同路径会看到完全不同的投影：

1. 有的宿主能看到 `session_state_changed(idle)`
2. 有的宿主能看到 `worker_status` 与 `requires_action_details`
3. 有的宿主能读回 `restoredWorkerState`
4. 有的宿主根本没有 internal-event flush 与恢复账
5. 有的宿主只支持极窄的 `can_use_tool` 控制面

这意味着即使系统内部已经有较完整的完成签字链，  
宿主拿到的也往往只是它的：

`一个条件化投影。`

所以这一章的核心问题是：

`Claude Code 为什么不允许不同宿主把自己看到的局部完成投影，冒充成完整的完成签字链？`

## 2. 最短结论

从源码看，Claude Code 的安全控制面依赖另一条很深的纪律：

`完成签字权可以存在，但不同宿主通常只能看到它的子集投影；看见局部完成，不等于拥有完整完成解释权。`

这至少体现在五类地方：

1. SDK 事件流把 `session_state_changed(idle)` 单独作为 authoritative turn-over signal 暴露给消费者
2. CCR/RemoteIO 路径除了状态事件，还拥有 `worker_status`、`requires_action_details`、`external_metadata`、恢复读回与 internal-event flush
3. 默认 `StructuredIO` 明确没有恢复账，也没有 internal-event flush 视角
4. `DirectConnectManager` 与 `RemoteSessionManager` 的 control surface 极窄，不能把自己当成完整 SDK host
5. bridge 在 recovery 窗口会主动 drop control/result，说明更厚宿主的完成投影也会随时序条件收缩

所以这一章的最短结论是：

`成熟安全控制面不仅要定义完成签字权，还要继续限制不同宿主如何消费、解释和投影这条签字链。`

## 3. 什么叫“安全完成投影”

这里说的“完成投影”，不是 UI 上显示了一个状态。  
我在这一章里的定义是：

`某个宿主基于自己能接触到的状态、事件、恢复账和控制面子集，对“动作是否已完成”形成的局部可见面。`

一个成熟的完成投影至少要回答四件事：

1. 这个宿主看到了哪一种完成信号
2. 它没看到哪一层完成账本
3. 它最多能对完成说到哪一步
4. 一旦越过这条边界，最危险的伪完成长什么样

如果缺这四句，  
宿主最容易把：

`我看见了一个完成影子`

误读成：

`我拥有了完整完成解释权`

## 4. 第一类投影：SDK 流消费者能看到 authoritative turn-over signal，但仍只是事件视图

## 4.1 `session_state_changed(idle)` 是明确暴露给 SDK 消费者的完成信号

`src/entrypoints/sdk/coreSchemas.ts:1738-1746` 与  
`src/utils/sdkEventQueue.ts:56-61`  
都把 `session_state_changed(idle)` 明确写成：

`authoritative turn-over signal`

这说明面向 SDK 流消费者时，  
系统愿意授予一条很强的完成信号。

## 4.2 但这仍然只是“事件流视图”的完成投影

因为同一套源码也说明，  
完成签字背后还依赖：

1. heldBackResult flush
2. bg-agent do-while 退出
3. internal events drain

SDK 消费者看到的是结果化事件，  
不是整条底层账本本身。

也就是说，  
它拿到的是：

`一个高质量的完成事件投影`

而不是：

`整条完成链的全部内部细节`

## 5. 第二类投影：CCR / RemoteIO 宿主拿到的是更厚的完成投影

## 5.1 `RemoteIO` 明确接上了恢复读回、状态回报、metadata 回报与 internal-event flush

`src/cli/remoteIO.ts:126-167` 很关键。  
这里 `RemoteIO` 做了几件默认 `StructuredIO` 没做的事：

1. `this.restoredWorkerState = init.catch(() => null)`
2. 注册 internal event writer
3. 注册 internal event reader
4. `reportState(state, details)`
5. `reportMetadata(metadata)`

而 `src/cli/remoteIO.ts:217-223` 又继续重写：

1. `flushInternalEvents()`
2. `internalEventsPending`

这说明 RemoteIO 宿主拿到的不是单纯的事件流。  
它拿到的是一条更厚的完成投影：

`状态 + metadata + 恢复基底 + flush 可见性`

## 5.2 `ccrClient.reportState()` 进一步说明这是一套“复制态完成投影”

`src/cli/transports/ccrClient.ts:644-662` 把：

1. `worker_status`
2. `requires_action_details`
3. `external_metadata`

分开上报。

这意味着 RemoteIO / CCR 路径对完成的理解，  
并不只是：

`我看见 idle 事件了`

而更像是：

`我还同时看见了 worker 复制态和 action 上下文快照。`

所以它确实比普通 SDK 流消费者更厚。  
但它依然只是：

`一条复制与回报路径上的完成投影`

而不是所有账本的无限制解释权。

## 6. 第三类投影：默认 `StructuredIO` 被明确设计成“没有这本账”

`src/cli/structuredIO.ts:139-196` 很关键。  
默认 `StructuredIO` 直接把：

1. `restoredWorkerState = Promise.resolve(null)`
2. `flushInternalEvents()` = no-op
3. `internalEventsPending = 0`

写死成默认语义。

这说明作者非常清楚：

`不是每个宿主都配拥有恢复账与持久化 flush 视角。`

这里最重要的不是功能缺失，  
而是它把这种缺失写成了显式语义。

换句话说，  
默认 StructuredIO 不是一个“稍后也许能看全”的宿主，  
而是被明确定义为：

`当前只拥有较薄完成投影的宿主。`

## 7. 第四类投影：窄 control surface 的宿主更不该冒充完整完成解释器

## 7.1 `DirectConnectManager` 只正式支持 `can_use_tool`

`src/server/directConnectManager.ts:81-99` 已经说明：

1. 只真正处理 `can_use_tool`
2. 其他 control subtype 直接 error

## 7.2 `RemoteSessionManager` 也做了同一种收窄

`src/remote/RemoteSessionManager.ts:192-213` 同样如此：

1. 正式支持 `can_use_tool`
2. 其余 subtype 明确回 unsupported

这说明这两类宿主在 control plane 上拿到的是窄子集。  
因此它们最合理的角色，不是完整完成解释器，  
而是：

`在极窄审批闭环里消费完成信号的局部宿主。`

如果它们把自己看到的状态直接升级成“完整任务已完成”或“整条控制面已收口”，  
那就是典型的过度解释。

## 8. 第五类投影：更厚的 bridge 也会在 recovery 窗口失去完整完成投影

`src/bridge/remoteBridgeCore.ts:824-878` 非常关键。  
在 `authRecoveryInFlight` 时，bridge 会主动：

1. drop `control_request`
2. drop `control_response`
3. drop `control_cancel_request`
4. drop `result`

但同一段代码又会在正常路径上调用：

1. `transport.reportState('requires_action')`
2. `transport.reportState('running')`
3. `transport.reportState('idle')`

这说明 bridge 确实拥有比窄宿主更厚的完成投影。  
但这条投影不是恒定资格。  
它会随着 recovery 条件收缩。

因此真正成熟的结论不是：

`bridge 更强，所以永远能解释完成。`

而是：

`bridge 只有在当前时序窗口仍完整时，才配投影较完整的完成语义。`

## 9. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`完成签字权是一回事，宿主能看到的完成投影又是另一回事；后者永远不能自动升级成前者。`

这条公理几乎能统一前面所有例子：

1. SDK 流消费者看到的是 authoritative 事件投影
2. RemoteIO/CCR 看到的是更厚的复制态与恢复态投影
3. StructuredIO 默认没有恢复账与 flush 视角
4. DirectConnect / RemoteSessionManager 的控制面天生更窄
5. bridge 在 recovery 窗口也会主动丢掉一部分完成解释资格

所以在 `42` 之后，  
更深层的控制面主线已经变成：

`完成签字权存在，但不同宿主只会拿到它的条件化投影。`

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只是定义了 authoritative completion signal，还继续把不同宿主能看到的完成子集设计成显式差异。`

这会带来几个很强的工程收益：

1. 接入方不容易误以为所有宿主都拥有同样完成语义
2. 恢复账、复制态和事件态不会被静默合并成同一层
3. 窄 control 宿主不会默认被抬升成完整 host
4. recovery 窗口里的解释权收缩会更容易被保留
5. 后续做统一安全控制台时，能更清楚地写出“这个宿主缺哪条完成链”

这对其他 Agent 平台构建者的启示也很直接：

1. 不要把一条内部 authoritative signal 直接假设为所有宿主都可见
2. 要把“宿主缺哪本账”显式做成产品语义
3. 不要让窄宿主在缺乏恢复账或 flush 视图时输出过满结论
4. 要允许更厚宿主在 recovery 窗口动态降级
5. 要把 completion signer 和 completion projection 明确分层

## 11. 哲学本质

这一章更深层的哲学是：

`真相的拥有者与真相的观看者，不是同一回事。`

很多系统一旦把某条完成信号对外暴露，  
就默认所有消费者都能同样地理解它。  
Claude Code 的源码更接近另一种成熟认识：

`即使完成已经被签字，不同观看者也仍然可能只拿到签字链的一部分影子。`

这意味着控制面的成熟，  
不只在于它会规定谁来签字，  
还在于它会规定：

`不同观察者最多能从这张签字上读出多少内容。`

从第一性原理看，  
它保护的是另一条根本公理：

`局部可见性不应自动升级成完整解释权。`

## 12. 苏格拉底式反思：这一章最容易犯什么错

### 12.1 我们会不会把“宿主差异”写得比源码实际更大

有风险。  
所以这一章里尽量只抓那些源码明确写出的差异：

1. 是否有恢复账
2. 是否有 internal-event flush
3. control subtype 是否受限
4. recovery 窗口会不会 drop 完成相关消息

### 12.2 这会不会和前面的宿主资格、账本投影章节重叠

会有交叉。  
但前面更偏总体宿主资格与账本子集；  
这一章只把这些差异重新压回“完成”这一条具体语义上。


## 13. 结语

`42` 回答的是：完成必须由 authoritative 回读链签字。  
这一章继续推进后的结论则是：

`即使这条签字链存在，不同宿主也往往只能看到它的子集投影，不能因此冒充完整完成解释器。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正成熟的控制面，不只定义谁能签字，还定义不同宿主最多能看到多少签字内容。`
