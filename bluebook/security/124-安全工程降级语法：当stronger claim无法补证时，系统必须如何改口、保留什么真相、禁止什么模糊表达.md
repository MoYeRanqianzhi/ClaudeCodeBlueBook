# 安全工程降级语法：当stronger claim无法补证时，系统必须如何改口、保留什么真相、禁止什么模糊表达

## 1. 为什么在 `123` 之后还必须继续写“降级语法”

`123-安全工程举证责任协议` 已经回答了：

`谁想把结论说得更强，谁就必须补齐缺失层证明。`

但如果继续追问，  
还会碰到一个更接地气的问题：

`如果补不齐呢？`

也就是：

`当 stronger claim 的举证责任没有履行完，系统到底该怎么说？`

这件事看起来像文案问题，  
但实际上它是安全工程问题。

因为只要降级语法不明确，  
系统就会自然滑向三种坏写法：

1. 继续维持强说法，不改口
2. 表面降级，但保留暗示性词汇
3. 完全沉默，让读者自己理解“应该问题不大”

所以 `123` 之后必须继续补出的下一层就是：

`安全工程降级语法`

也就是：

`当 stronger claim 无法补证时，系统必须明确改口到哪个层级、保留哪些仍然成立的真相，并禁止哪些模糊表达继续制造过满结论。`

## 2. 最短结论

对这份研究版 Claude Code 源码来说，  
真正成熟的安全写法不只是：

1. 识别 stronger claim
2. 确认它缺证
3. 指出责任在谁

还必须继续做到第四步：

`把语言降回真实层级。`

例如：

1. 不能说“verify 存在”，只能说“README 中存在 verify 命名”
2. 不能说“已形成验证体系”，只能说“当前仅存在 build/check/start 入口”
3. 不能说“已验证”，只能说“best-effort artifact 可构建且可最小启动”

所以这一章的最短结论是：

`举证责任若未完成，语言必须随之降级。`

再压成一句：

`不能补证，就要改口。`

## 3. 第一性原理：为什么降级语法是安全问题，不只是修辞问题

因为安全工程里最危险的从来不是“说得短”，  
而是：

`说得比证据强。`

而 stronger claim 一旦未被撤回，  
它就会继续在系统里产生三个效果：

1. 误导后续判断
2. 抬高读者默认预期
3. 为后续更强升级制造虚假前提

所以当举证责任没有被履行时，  
系统若不降级表达，  
就等于继续允许：

`未被证明的强结论留在场上。`

这正是安全工程应当禁止的。

所以降级语法的第一性原理就是：

`语言必须跟随证据层级退回，而不能停留在旧强度。`

## 4. 当前仓库里哪些句子必须如何降级

## 4.1 从“verify 存在”降到“verify 命名存在”

当前可成立的事实只有：

1. `README.md:167-174` 中出现了 `verify/*`

但不可成立的事实包括：

1. 当前树中存在 `verify/*` 文件
2. 当前仓内存在 formal verify capability

所以正确改口应是：

1. 不说：`verify 存在`
2. 改说：`README 中存在 verify 命名回声`
3. 再补一句：`当前提取树未观察到对应 artifact/entry/host`

这是从：

`institutional claim`

降回：

`lexical claim`

## 4.2 从“已有验证体系”降到“仅有 build/check/start 制度”

当前已知：

1. `package.json:7-11` 只有 `prepare-src/build/check/start`
2. 无 `verify/test` script

所以不应写：

`当前仓内已有验证体系`

而应写：

`当前仓内仅有围绕 prepare-src 组织的 build/check/start 入口制度，尚无 formal verify/test 入口。`

这是从：

`verification-system claim`

降回：

`partial engineering-interface claim`

## 4.3 从“已验证”降到“best-effort artifact 可构建且可最小启动”

当前已知：

1. `build` 可以产出 best-effort artifact
2. `start` / `--version` 可以最小启动
3. 但对象上限、能力上限、验证缺口 veto 仍在

所以不应写：

`当前构建已验证`

而应写：

`当前公开提取树下的 best-effort artifact 已可构建并可最小启动，但这不构成 formal verification closure。`

这是从：

`validated-system claim`

降回：

`artifact-and-smoke claim`

## 4.4 从“只是还没展开”降到“当前未证明”

很多句子最危险的地方不是强词，  
而是模糊词。

例如：

1. `只是还没展开`
2. `大概率有`
3. `应该已经接上`
4. `可能在别处`

这些词看似谨慎，  
但它们实际上仍在替 stronger claim 保留空间。

所以当证据缺失时，  
不应写：

`可能只是没展开`

而应写：

`当前未证明，且不应按已存在处理。`

这是最关键的降级语法。

## 5. 技术先进性：成熟系统会定义 downgrade path，而不是只定义 success path

很多系统只会定义：

1. 什么时候升级
2. 什么时候通过
3. 什么时候成功

但真正成熟的系统还会定义：

`当升级失败时，语言该退到哪里。`

这正是下一代安全控制台最该继承的能力。

因为如果没有 downgrade path，  
系统即便识别出缺口，也只会：

1. 停在尴尬的半强表述
2. 用模糊词缓冲
3. 让旧强结论赖在界面上

而这会直接腐蚀前面辛苦建立的：

1. gate discipline
2. claim ceiling
3. joint-signature protocol
4. veto constitution
5. absence semantics
6. burden-of-proof rule

所以从技术角度看，  
降级语法其实是这一整套制度的：

`执行面`

## 6. 苏格拉底式反思：如果我发现 stronger claim 证据不足，我该如何逼自己改口

可以先追问八句：

1. 这句话当前最强还能保留到哪一层
2. 哪些词已经超过了当前证据层
3. 我是否仍在用“可能”“应该”“大概”替强结论续命
4. 如果删掉所有模糊词，这句话还剩什么
5. 我能否明确写出“当前未证明”
6. 我能否明确写出“不能按已存在处理”
7. 当前应保留的真相最小子集是什么
8. 我是否已经把旧强结论彻底撤回，而不是只是把它说软一点

这些问题最终都会压成一句：

`降级不是把话说虚，而是把话说回真实层级。`

## 7. 对统一安全控制台的产品启示

如果把这一层做成产品能力，  
最值得实现的是：

1. 当 stronger claim 被 veto 时，自动生成降级表述
2. 把“仍成立的真相子集”与“已撤回的强结论”明确分栏
3. 禁止继续显示暗示已存在/已验证的残留词
4. 把 `unproven`、`not observed`、`not wired`、`no host` 这类降级语法做成标准词表

这会让控制台从：

`发现问题`

升级成：

`正确改口`

## 8. 一条硬结论

对这份 Claude Code 研究版源码来说，  
真正严谨的安全研究不是：

`一旦 stronger claim 站不住，就把话说得模糊一点，`

而是：

`一旦补证失败，就明确降回真实层级，保留真实子集，并撤掉所有暗示性强词。`

因为只有这样，  
这份蓝皮书才不会变成：

`证据不够，但语言还在硬撑`

而会继续升级成：

`证据层级、举证责任与表达强度同步收缩的安全工程分析。`
