# 统一安全控制台字段设计：源码里已经有哪些状态字段，还缺哪些跨面字段

## 1. 为什么在 `22` 之后要写“字段设计”

`22-安全证明链` 已经把 Claude Code 的安全压成六条必须同时成立的证明链。  
但如果继续往产品落地层推进，还会出现一个非常具体的问题：

`如果真要把这套安全证明系统做成统一控制台，屏幕上到底该放哪些字段？`

没有这一步，  
“统一安全控制台”仍然只是概念图。  
而一旦进入字段级，就能看出 Claude Code 现有源码其实已经提供了大量原材料。

因此这一章的目标不是新增安全理论，  
而是回答：

1. 源码里已经有哪些状态字段
2. 这些字段现在分散在哪些对象和消息里
3. 为什么它们尚未构成统一控制台
4. 如果要产品化，最少应该如何分组

## 2. 最短结论

Claude Code 当前的问题不是没有安全字段，  
而是：

`安全字段已经存在于 AppState、SDK 消息、子系统状态页和权限原因枚举里，但它们还没有被归一化成同一张控制台的数据模型。`

从源码看，至少已经存在五组高价值字段源：

1. 会话 / 主体字段
2. 主权 / 来源字段
3. 仲裁 / 权限字段
4. 包络 / 外部能力字段
5. 增量审批与解释字段

因此下一代统一安全控制台真正要做的，不是发明全新信息，  
而是：

`把已有字段做成可聚合、可关联、可解释的同一套安全状态模型。`

## 3. 第一组字段源：`AppState` 已经是一半控制台模型

### 3.1 `AppStateStore.ts` 已经包含大量安全态字段

`src/state/AppStateStore.ts` 的 `AppState` 定义里，已经能看到不少直接适合控制台的字段：

- `mainLoopModel`
- `toolPermissionContext`
- `remoteConnectionStatus`
- `replBridgeConnected`
- `replBridgeSessionActive`
- `replBridgeReconnecting`
- `replBridgeConnectUrl`
- `replBridgeSessionUrl`
- `replBridgeError`
- `mcp.clients`
- `plugins.errors`
- `plugins.installationStatus`

这说明统一控制台缺的不是底层状态，  
因为系统进程内部已经持有了大量可视化所需数据。

### 3.2 但这些字段目前仍然按功能子域组织

同一个 `AppState` 里，状态被分在：

- bridge
- mcp
- plugins
- permission
- remote session

这对运行时实现是自然的，  
但对安全控制台来说会有一个问题：

`字段是按实现所有权组织的，不是按安全问题组织的。`

于是用户想问：

`我现在为什么不能安全地继续？`

系统给出的字段却是：

- MCP 在哪一组
- bridge 在哪一组
- plugin 在哪一组
- permission mode 在哪一组

这就是字段层面的碎片化。

### 3.3 从第一性原理看，控制台字段不应只按模块归档

如果安全控制台的目标是解释“当前边界为什么成立或不成立”，  
那字段就不应该只按模块归档，  
而更应该按：

1. 主体
2. 主权
3. 仲裁
4. 包络
5. 外部能力
6. 增量审批

来重新归组。

所以 `AppState` 的真正价值不在于“已经有很多字段”，  
而在于：

`它已经证明统一控制台的数据并不缺，只是还没有被按安全语义重组。`

## 4. 第二组字段源：SDK 状态消息已经暴露了“可外传状态面”，但仍然过碎

### 4.1 `SDKStatusSchema` 太薄，说明“总状态”目前仍很克制

`src/entrypoints/sdk/coreSchemas.ts` 里的 `SDKStatusSchema` 目前只有：

- `compacting`
- `null`

而 `SDKStatusMessageSchema` 额外带上了：

- `permissionMode`

这说明一个重要现象：

`系统确实已经有“状态消息”概念，但这个总状态消息目前只承载了最小跨宿主状态，而没有承载完整安全状态。`

也就是说，它更像：

`最小兼容状态面`

而不是：

`统一安全状态面`。

### 4.2 `SDKAuthStatusMessageSchema` 和 `McpServerStatusSchema` 又各自形成独立状态流

同一文件里，`SDKAuthStatusMessageSchema` 会单独发：

- `isAuthenticating`
- `output`
- `error`

而 `McpServerStatusSchema` 又有另一套字段：

- `name`
- `status`
- `serverInfo`
- `error`
- `config`
- `scope`
- `tools`
- `capabilities`

这说明系统已经知道：

1. auth 状态需要单独流
2. MCP 连接状态需要结构化对象
3. 不同安全子域天然需要不同字段密度

但也说明统一控制台的下一步难题是：

`如何把这些天然异构的状态流重新映射到一张统一安全卡片模型上。`

### 4.3 `controlSchemas.ts` 证明外部宿主已经能请求子系统状态

`SDKControlMcpStatusRequestSchema` 和相关 response 进一步说明，  
宿主侧已经可以按需拉取 MCP 状态。

这非常关键，因为它证明：

`统一控制台不是缺“拿不到字段”，而是缺“怎样把拿到的字段组织成同一张语义图”。`

## 5. 第三组字段源：权限原因枚举已经接近“仲裁因果字段”

### 5.1 `PermissionDecisionReason` 是控制台最容易被忽略的字段宝库

`src/types/permissions.ts` 里的 `PermissionDecisionReason` 很重要。  
它已经枚举出多种仲裁原因：

- `rule`
- `mode`
- `subcommandResults`
- `permissionPromptTool`
- `hook`
- `asyncAgent`
- `sandboxOverride`
- `classifier`
- `workingDir`
- `safetyCheck`
- `other`

这其实已经非常接近统一控制台中的“为什么这次动作会得到当前裁决”字段。

### 5.2 它的真正价值不在权限系统内部，而在解释层外部化

如果把这些 reason 只留在权限系统内部，  
它们只是实现细节。

但一旦把它们字段化，就能直接变成控制台里的因果解释：

- 当前由哪条 rule 触发
- 当前是 mode 约束还是 classifier 约束
- 当前是 safetyCheck 且是否 `classifierApprovable`
- 当前是否是 sandboxOverride 或 workingDir 约束

这说明 Claude Code 其实已经拥有了：

`仲裁解释字段。`

只是这些字段目前主要服务局部提示，不服务全局控制台。

### 5.3 从第一性原理看，控制台的核心不是状态字段，而是“因果字段”

很多控制台只展示：

- enabled / disabled
- connected / disconnected
- allowed / denied

这在安全系统里远远不够。  
真正高质量的安全控制台，还必须展示：

`为什么是 allowed / denied / disconnected。`

从源码看，Claude Code 已经有了相当丰富的因果字段原材料，  
尤其是 `PermissionDecisionReason` 这类类型定义。

## 6. 第四组字段源：状态页与子系统页已经定义了“用户可消费字段”

### 6.1 `status.tsx` 给的是摘要字段

`src/utils/status.tsx` 已经输出了多类适合直接面向用户的字段：

- account information
- token source / api key source
- API provider
- proxy / mTLS / CA cert
- sandbox enabled
- setting sources
- MCP counts by state

这些字段的特点是：

`可读性很高，但关联性较弱。`

它们适合做控制台首页，  
不适合单独承担完整诊断。

### 6.2 `SandboxConfigTab` 给的是包络字段

它输出的是：

- excluded commands
- fs read denyOnly
- fs write allowOnly / denyWithinAllow
- network restrictions
- allowed Unix sockets
- glob warnings

这类字段的特点是：

`信息密度高，适合深度页，不适合首页摘要。`

### 6.3 `PluginSettings` 给的是主权归因字段

它已经把：

- editableSources
- `isInPolicy`
- `managed-only`
- action routing

做成了 UI 可消费字段。

这说明统一控制台最关键的一个设计启示：

`不同字段不是一视同仁平铺，而应按摘要字段、深度字段、归因字段分层。`

## 7. 第五组字段源：增量审批和资格诊断已经定义了“断链字段”

### 7.1 `securityCheck.tsx` 定义的是边界变化字段

这一组字段的特点不在长期状态，  
而在：

`边界变化事件。`

例如：

- 是否有 dangerous managed settings
- 这些 dangerous settings 是否 changed
- 当前是 interactive 还是 non-interactive
- 最终是 `approved` / `rejected` / `no_check_needed`

这是统一控制台里非常容易缺失的一类字段：  
事件型字段，而不是静态字段。

### 7.2 `getBridgeDisabledReason()` 定义的是资格断链字段

bridge disabled reason 已经可以分解出：

- 主体不符
- scope 不足
- organization 归属缺失
- gate 未开
- build 不支持

这类字段的价值在于：

`它们不是普通状态，而是“哪条证明链断了”的结构化提示。`

统一控制台如果没有这类字段，  
就只能告诉用户“不行”，  
而不能告诉用户“哪一段证明没成立”。

## 8. 如果把这些源码字段真正做成统一安全控制台，最少应有哪几组字段

我会把控制台字段压成七组。

### 8.1 主体字段

- account type
- token source
- auth status
- profile scope
- organization UUID

### 8.2 主权字段

- active setting sources
- policy origin
- managed-only / plugin-only surfaces
- host-managed env flags
- current MCP scope / source

### 8.3 仲裁字段

- permission mode
- last decision behavior
- decision reason type
- matched rule / rule source
- safetyCheck reason
- classifierApprovable

### 8.4 包络字段

- sandbox enabled
- fs read restrictions
- fs write restrictions
- network restrictions
- allowed Unix sockets
- excluded commands

### 8.5 外部能力字段

- MCP server statuses
- MCP scopes
- tool annotations
- bridge connected / session active / reconnecting
- WebFetch read allowance class

### 8.6 增量审批字段

- dangerous settings changed
- approval required
- approval result
- approval subject

### 8.7 解释字段

- current top blocking reason
- broken proof chain
- shortest next action
- correct responsible party

这一组是最关键的。  
因为前六组都是原材料，  
第七组才是“统一控制台”真正成立的地方。

## 9. 当前还缺哪些跨面字段

虽然源码里已经有很多字段，  
但如果真做统一安全控制台，仍有几类字段还缺。

### 9.1 缺“当前主导边界”字段

系统现在能分别告诉你：

- source
- mode
- sandbox
- entitlement

但它还不太能直接告诉你：

`当前到底是哪一层边界在主导最终结果。`

### 9.2 缺“跨面因果句”字段

例如：

`因为当前主体缺少 profile scope，所以 bridge entitlement 不成立，因此 remote 能力面被整体收口。`

这种因果句并不是单个模块天然拥有的，  
需要跨面汇总。

### 9.3 缺“证明链完整度”字段

统一控制台如果足够成熟，  
应该可以直接显示：

- 主体链：已成立 / 未成立
- 主权链：已成立 / 部分成立
- 授权链：当前动作待裁决
- 包络链：已收口
- 外部能力链：局部开放
- 增量审批链：待确认

而源码现在更多只给出局部事实，  
还没有把这些事实汇总成证明链完整度。

## 10. 从技术角度看，这套字段设计的先进性与下一步

Claude Code 当前已经做对了三件事：

1. 它在不同子系统里保留了足够丰富的结构化字段
2. 它没有把所有状态粗暴压成一个 total status
3. 它已经拥有不少因果字段，而不只是结果字段

下一步真正要做的不是再造字段，  
而是：

`建立一个跨面的字段归一化层。`

这个归一化层至少要把：

- AppState 字段
- SDK status / auth status / MCP status 消息字段
- permission reason 字段
- 子系统深度页字段
- 增量审批结果字段

重新汇成统一的安全状态模型。

## 11. 苏格拉底式追问

### 问：为什么要专门写字段，而不是继续写哲学？

答：因为没有字段，哲学就无法变成控制台；没有字段归一化，控制台就无法解释系统。

### 问：为什么说 Claude Code 已经离统一控制台很近？

答：因为大部分字段已经存在，只是分散在不同消息、类型和子系统页里。

### 问：为什么现有状态页还不够？

答：因为它们展示字段，但没有把字段重新组织成跨面的安全语义。

### 问：统一控制台最关键的新增字段是什么？

答：不是更多原始状态，而是“当前主导边界”“当前断裂证明链”“当前最短纠正动作”这类派生字段。

## 12. 本章结论

Claude Code 现在最缺的，并不是新的安全字段。  
它最缺的是：

`把已有字段从“按模块分别可见”提升成“按安全语义统一可见”的归一化层。`

这就是为什么我会认为，统一安全控制台的下一步核心工作不是继续堆页面，  
而是先把：

- 状态字段
- 原因字段
- 主权字段
- 证明链字段

收束成同一套数据模型。

只有这样，安全控制面才会真正从“系统知道很多”变成“用户也能看懂系统到底知道什么”。
