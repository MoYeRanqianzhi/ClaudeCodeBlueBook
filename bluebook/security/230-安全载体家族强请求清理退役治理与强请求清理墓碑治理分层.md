# 安全载体家族强请求清理退役治理与强请求清理墓碑治理分层：为什么artifact-family cleanup stronger-request cleanup-sunset-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-tombstone-governor signer

## 1. 为什么在 `229` 之后还必须继续写 `230`

`229-安全载体家族强请求清理迁移治理与强请求清理退役治理分层` 已经回答了：

`stronger-request cleanup 线即便知道旧世界怎样过渡，也还需要 sunset governor 去决定兼容期何时正式结束。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成“宣布退役就够了”的错觉：

`只要 stronger-request cleanup sunset governor 已经宣布旧世界不再算 current truth，它就自动拥有了决定退役后还要留下什么最小残留标记的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 自己最成熟的 tombstone 正对照，
会发现它其实已经把：

`when an old world stops counting`

和

`what minimal residue must still survive after it stops counting`

拆成了不同层。

最硬的证据有六组：

1. `src/query.ts:712-719`
   遇到 orphaned assistant messages 时不会只静默丢弃旧对象，
   而会显式产出 `type: 'tombstone'`
2. `src/utils/messages.ts:2954-2957`
   与
   `src/QueryEngine.ts:758-760`
   证明 tombstone 不是普通内容，而是 control signal
3. `src/screens/REPL.tsx:2646-2648`
   与
   `src/utils/sessionStorage.ts:1470-1473,925-945`
   又证明 tombstone 会继续驱动 UI / transcript 删除链
4. `src/utils/plugins/cacheUtils.ts:56-72,145-171`
   与
   `src/services/plugins/pluginOperations.ts:523-547`
   证明 `.orphaned_at` 不是 sunset 时钟本身，而是退役后留在磁盘上的最小 marker
5. `src/utils/plugins/orphanedPluginFilter.ts:1-15,35-38`
   与
   `src/main.tsx:2544-2568`
   进一步证明 marker 还会被转译成 grep / glob exclusion grammar，继续约束未来读取世界
6. `src/utils/config.ts:427-438`
   与
   `src/hooks/notifs/useModelMigrationNotifications.tsx:5-50`
   则说明对象世界已经变化之后，系统有时还要保留一个最小历史注脚，向当前 runtime 解释“为什么现在变了”

这说明：

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

和

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界从什么时候起不再算当前正式世界。`

后者才配说：

`它退役之后还要留下什么最小残留标记，让后来的 runtime、搜索面、恢复链与读者不会把“已经退役”重新读成“从未发生”或“仍可继续援引”。`

所以 `229` 之后必须继续补的一层就是：

`安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`

也就是：

`stronger-request cleanup sunset governor 负责宣布结束；stronger-request cleanup tombstone governor 才负责治理结束之后留下的最小残留事实。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-tombstone-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 tombstone manager，
而是在说：

1. repo 在 message / plugin / model 线上已经展示出真实的 marker grammar
2. 这些 marker grammar 不等于 sunset deadline 本身
3. 它们承担的是退役后的最小残留事实表达

第二条：

这里说的 `tombstone governance`
不是在声称：

`退役之后一定要保留完整旧对象。`

真正要说的是：

`墓碑治理回答的是退役后保留多少最小事实，既足以约束未来解释，又不把旧对象重新抬回 current world。`

第三条：

这里也不是把：

`migration timestamp`

一概硬写成字面 tombstone。

当前更稳的说法只能是：

`有些残留物虽然源码里不直接叫 tombstone，但功能上承担了 post-transition trace artifact 的角色，因此属于 tombstone-like marker。`

第四条：

这里尤其不能过度声称：

`只要对象退役，就一定先有 tombstone message。`

当前更稳的说法只能是：

`repo 展示了多种 marker grammar：message tombstone、plugin marker、migration timestamp；它们形态不同，但都在回答“结束后最小残留物是什么”。`

第五条：

这里也不能过度声称：

`只要有 marker，就代表 cleanup 线已经拥有完整墓碑制度。`

当前更稳的说法只能是：

`repo 已在别的子系统里展示 tombstone governance 的思想；stronger-request cleanup 线当前缺的是把这套思想正式映射到旧 cleanup path、旧 promise 与旧 receipt semantics 上。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-sunset-governor signer 仍不等于 stronger-request cleanup-tombstone-governor signer`

证据：

1. `query.ts:712-719`
   遇到 orphaned messages 时显式 `yield { type: 'tombstone' }`
2. `utils/messages.ts:2954-2957`
   与
   `QueryEngine.ts:758-760`
   证明 tombstone 是 control signal，而不是普通 message
3. `REPL.tsx:2646-2648`
   与
   `sessionStorage.ts:1470-1473,925-945`
   证明 tombstone 会继续驱动显示层与 transcript 存储层改口
4. `.orphaned_at`
   不是 sunset 时点本身，而是对象退役后留在磁盘上的最小 marker
   `cacheUtils.ts:56-72,145-171`; `pluginOperations.ts:523-547`
5. marker 还会继续塑造 future search / visibility world
   `orphanedPluginFilter.ts:1-15,35-38`; `main.tsx:2544-2568`
6. migration timestamp 证明对象世界变化之后，系统有时还要保留一个最小历史注脚去解释“为什么当前世界刚变”
   `config.ts:427-438`; `useModelMigrationNotifications.tsx:5-50`
7. stronger-request cleanup 当前虽已开始显露 sunset-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup path 与旧 receipt semantics 退役后还应留下什么最小 marker

因此这一章的最短结论是：

`stronger-request cleanup sunset governor 最多能说旧世界何时结束；stronger-request cleanup tombstone governor 才能说结束之后留下什么最小残留标记来约束未来解释。`

再压成一句：

`会退役，不等于会留下正确的退役痕迹。`

## 4. 第一性原理：sunset governance 回答“何时失效”，tombstone governance 回答“失效后还保留什么最小事实”

从第一性原理看，
这两层治理处理的是两个不同真相面的主权问题。

`sunset governor`

回答的是：

1. 旧 world 从哪一天、哪次启动或哪类版本后不再有效
2. 兼容窗口是否结束
3. dual-world 是否切成 hard close
4. 旧 artifact 是否不再被 runtime 接受
5. 哪些 visibility / warning / storage plane 已进入 post-retirement world

`tombstone governor`

回答的则是：

1. 旧对象退役后是否要留一个最小 control signal
2. 这个 signal 是 tombstone message、marker、timestamp、deprecation note 还是其他最小残留物
3. 哪些子系统仍应继续消费这个 marker
4. 这个 marker 何时再被进一步清除
5. 它究竟是给未来读者解释历史，还是给 runtime 防止误复活，还是两者兼有

所以这两层的哲学本质不同：

`sunset`

决定旧对象什么时候停止作为 current truth；

`tombstone`

则决定它停止之后，系统还保留什么最小事实来约束后来者。

Claude Code 在这方面真正先进的地方，
不是它“会在对象无效后把它删掉”，
而是它已经明确承认：

`删除不是全部，最小残留事实同样需要被制度化。`

这是一种很成熟的安全哲学：

`对象停止有效，不等于它留下的解释义务也随之消失。`

因为一旦把 sunset 与 tombstone 压成一层，
系统就容易出现三种危险：

1. 旧对象虽然不再有效，但未来读者不知道它为什么消失
2. 旧对象虽然已退役，但未来 runtime 没有 marker 去阻止误读或误复活
3. 旧对象虽然被删掉了，但系统失去了“它曾经为何被结束”的最小历史约束

## 5. message 线给出最强正例：对象退役之后，Claude Code 并不会只让它静默蒸发，而是显式产出 `tombstone`

`src/query.ts:712-719`
非常关键。
当 streaming fallback 造成 orphaned assistant messages 时，
系统并不是简单把旧 partial messages 从内存里扔掉。

它会：

1. 为每个 orphaned message `yield { type: 'tombstone' }`
2. 记录 `tengu_orphaned_messages_tombstoned`
3. 清空旧 attempt 遗留的 `assistantMessages`、`toolResults` 与 `toolUseBlocks`

这说明对消息系统来说：

`对象已不再有效`

和

`应留下一个最小控制信号告诉下游怎样处理它`

是两件事。

继续看
`src/utils/messages.ts:2954-2957`：
stream handling 遇到 `message.type === 'tombstone'` 时，不是把它当正文内容渲染，而是调用 `onTombstone`。

再看
`src/QueryEngine.ts:758-760`：

`Tombstone messages are control signals for removing messages`

而
`src/screens/REPL.tsx:2646-2648`
与
`src/utils/sessionStorage.ts:1470-1473,925-945`
又把 tombstone 接到 UI / transcript 删除链上。

这组代码合在一起说明：

`tombstone` 不是描述性注脚，也不是普通通知。
它是退役后最小残留控制面的正式成员。

## 6. plugin 线给出第二组强正例：`.orphaned_at` 不是垃圾残留，而是被治理的机器墓碑

`src/utils/plugins/cacheUtils.ts:56-72`
的 `markPluginVersionOrphaned()` 与注释已经说明：

1. 插件被卸载或更新到新版本后，旧版本先被打上 `.orphaned_at`
2. 这个 marker 会被后续 orphan cleanup 继续消费

`src/services/plugins/pluginOperations.ts:523-547`
也明确把最后一个 scope 卸载后的 install path 交给 `markPluginVersionOrphaned()`。

更重要的是
`processOrphanedPluginVersion()` 所在的
`cacheUtils.ts:145-171`：

1. 如果 marker 不存在，先写 marker
2. 如果 marker 超过 7 天，再删目录

这说明 `.orphaned_at` 的职责不是：

`对象还活着`

而是：

`对象已经退役，但系统仍需留下一个最小残留标记来决定后续怎么处理它。`

所以 `.orphaned_at` 更像一块真正的“机器墓碑”，
而不是普通元数据文件。

## 7. marker-driven visibility 再证明：墓碑不是只服务历史解释，它还会主动约束未来读取世界

`src/utils/plugins/orphanedPluginFilter.ts:1-15,35-38`
与
`src/main.tsx:2544-2568`
特别值钱。

这里的流程是：

1. 先完成 orphan GC
2. 再扫描 `.orphaned_at`
3. 基于 marker 生成 grep / glob exclusion patterns
4. 把 exclusion cache 冻结在当前 session

这意味着 `.orphaned_at` 的作用不只是：

`证明旧版本退役过`

它还会继续决定：

`未来的搜索与读取不应再触及哪些目录。`

这把墓碑治理从“历史注脚”提升成了“未来约束”。

也就是说：

`成熟的 tombstone，不只是留下痕迹，还会继续约束后来者别再走回旧世界。`

## 8. model 迁移时间戳给出第三类正对照：有些墓碑不长得像 tombstone，但功能上就是“变化已经发生过”的最小残留事实

`src/utils/config.ts:427-438`
定义了：

1. `opusProMigrationComplete`
2. `opusProMigrationTimestamp`
3. `legacyOpusMigrationTimestamp`
4. `sonnet45To46MigrationTimestamp`

这些字段都不是迁移动作本身，
也不是 retirement date 本身。
它们是迁移之后留在 `GlobalConfig` 里的最小事实残留。

`src/hooks/notifs/useModelMigrationNotifications.tsx:5-50`
再说明这些字段的用途：

1. 判断这次启动是否刚刚发生过模型迁移
2. 只发一次高优先级提示
3. 在 `legacyOpusMigrationTimestamp` 场景里额外给出 opt-out 说明

这里必须谨慎说：
源码没有把这些字段字面命名成 tombstone。
但从功能上看，
它们确实承担了 tombstone-like marker 的职责：

`它们不是迁移动作，而是迁移动作完成后保留下来的最小历史事实，用于向当前系统解释“为什么现在变了”。`

## 10. 硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 sunset governance，就已经足够成熟。`

而是：

`Claude Code 在 tombstone messages、.orphaned_at 与 migration timestamps 上已经明确展示了 tombstone governance 的存在；因此 artifact-family cleanup stronger-request cleanup-sunset-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-tombstone-governor signer。`

换句话说：

`cleanup 线真正缺的不是“何时宣布旧世界结束”，而是“结束之后靠什么最小残留事实约束后来者”。`
