# 安全载体家族强请求清理复活治理与强请求清理再赋权治理分层：为什么artifact-family cleanup stronger-request cleanup-resurrection-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer

## 1. 为什么在 `390` 之后还必须继续写 `391`

`390-安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪些旧对象可以重新回到 current world，也仍需要 resurrection governor 去决定它们是否能回来、凭什么回来、回来时进入哪一层 world。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“对象回来就说明它原来的资格也一起回来了”
的错觉：

`只要 stronger-request cleanup resurrection governor 已经让旧对象重新出现，它就自动拥有了决定旧 identity、旧配置、旧 secrets、旧 scope 与旧可用资格是否一起恢复的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 re-entitlement 正对照，
会发现它其实明确拆开了六件事：

1. `object return`
2. `enabled truth`
3. `config / secrets trust`
4. `policy veto`
5. `entitlement closure`
6. `identity seat`

最硬的证据至少有八组：

1. `pluginOptionsStorage.ts` 证明 `deletePluginOptions()` 只在 last installation 卸载时才抹掉旧 `pluginConfigs` 与旧 `pluginSecrets`，说明 comeback 不自动带回旧配置与旧 secrets  
   `src/utils/plugins/pluginOptionsStorage.ts:201-272`
2. `pluginOperations.ts:523-540` 进一步证明 last-scope remove 与 options / secrets wipe 被显式联动，说明对象离场时 entitlement material 会被单独清算  
   `src/services/plugins/pluginOperations.ts:523-540`
3. `setPluginEnabledOp()` 把 enable / disable 写成单独 settings action，说明对象回来不等于自动恢复 enabled truth  
   `src/services/plugins/pluginOperations.ts:560-727`
4. `pluginPolicy.ts` 与 `pluginOperations.ts:650-657` 证明更高层 policy 仍可 veto 旧资格恢复  
   `src/utils/plugins/pluginPolicy.ts:11-19`  
   `src/services/plugins/pluginOperations.ts:650-657`
5. `pluginOperations.ts:664-687` 证明 even after return，scope / precedence 仍要经过 override legality 判断，不是对象一回来就自动恢复原 seat  
   `src/services/plugins/pluginOperations.ts:664-687`
6. `installedPluginsManager.ts:827-830,859-861` 证明 disk / materialized existence 与 effective installed / enabled truth 仍是两层不同真相  
   `src/utils/plugins/installedPluginsManager.ts:827-830,859-861`
7. `pluginInstallationHelpers.ts` 说明再赋权甚至可能是 closure-level 的统一 settings write，而不是某个对象“回来后自然继续拥有旧资格”  
   `src/utils/plugins/pluginInstallationHelpers.ts:330-339,361-367,414-437`
8. `plans.ts:233-257` 与 `REPL.tsx:1790-1797` 证明内容回来时仍可能只拿到 `newSlug`，而不是恢复 original identity  
   `src/utils/plans.ts:233-257`  
   `src/screens/REPL.tsx:1790-1797`

这说明：

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

和

`artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`

仍然不是一回事。

前者最多能说：

`这个旧对象可以重新出现。`

后者才配说：

`它重新出现之后，算不算原来那个被授权、被配置、被允许继续工作的对象。`

所以 `390` 之后必须继续补的一层就是：

`安全载体家族强请求清理复活治理与强请求清理再赋权治理分层`

也就是：

`stronger-request cleanup resurrection governor 负责让对象回来；stronger-request cleanup re-entitlement governor 才负责决定它回来之后恢复哪些旧资格。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 re-entitlement manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出对象回来不等于资格回来
2. `回来` 与 `恢复旧资格` 是两个不同问题
3. stronger-request cleanup 线如果真要把 resurrection world 做完整，也迟早要回答这些“回来后还算不算原来那个对象”的问题

第二条：

这里说的 `re-entitlement governance`

不是在声称：

`对象回来后必须恢复所有旧资格。`

当前更稳的说法只能是：

`re-entitlement governance 首先回答的是哪些资格可以回来、哪些必须重新授予、哪些根本不该回来。`

第三条：

这里也不是把：

`enable state`

单独当成全部 re-entitlement grammar。

真正关键的是：

1. 配置 / secret 会不会回来
2. enabled truth 会不会回来
3. policy 是否允许回来
4. identity / scope 是否还是原来的那个 seat
5. entitlement 是否要按 closure 重新签发

第四条：

这里尤其不能过度声称：

`deletePluginOptions()`

就等于完整资格治理。

更稳的说法只能是：

`资格治理` 还要继续回答 post-return enable、policy、closure、scope 与 identity fallback；options / secrets wipe 只是其中一层。`

第五条：

这里也不能过度声称：

`new identity`

一定比 `old identity` 弱。

当前更稳的说法只能是：

`new slug / new seat` 的关键不在强弱，而在系统拒绝把“内容回来”偷写成“原资格原封不动回来”。`

第六条：

这里还不能过度声称：

`re-entitlement` 只是恢复体验之外的附带管理。

更稳的说法只能是：

`它本质上是在治理 post-return qualification grammar；体验只描述回来，资格才决定回来之后还能做什么。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-resurrection-governor signer 仍不等于 stronger-request cleanup-re-entitlement-governor signer`

证据：

1. `deletePluginOptions()` 会在 last-scope uninstall 时删除旧 `pluginConfigs` 与旧 `pluginSecrets`，说明对象回来不自动带回旧配置与旧 secrets  
   `pluginOptionsStorage.ts:201-272`
2. `setPluginEnabledOp()` 把 enable / disable 做成单独 settings write，说明对象可 install、materialize、refresh，但是否恢复 enabled truth 仍要经过独立写入  
   `pluginOperations.ts:560-727`
3. `pluginPolicy.ts` 与 `pluginOperations.ts` 明确写出 org policy 可以继续 veto install / enable  
   `pluginPolicy.ts:11-19`; `pluginOperations.ts:650-657`
4. `pluginOperations.ts` 继续要求 cross-scope override legality，说明 current seat 与 old seat 不是自然延续关系  
   `pluginOperations.ts:664-687`
5. `installedPluginsManager.ts` 直接证明 materialized existence 与 effective installed / enabled entitlement 不是同一层 truth  
   `installedPluginsManager.ts:827-830,859-861`
6. `pluginInstallationHelpers.ts` 证明 entitlement 有时要按 dependency closure 一次性写入，而不是跟着某个单对象 comeback 自动恢复  
   `pluginInstallationHelpers.ts:330-339,361-367,414-437`
7. `copyPlanForFork()` 与 `REPL.tsx` 证明内容回来时仍可能只获得 new identity，而不是 old identity  
   `plans.ts:233-257`; `REPL.tsx:1790-1797`
8. `deletePluginOptions()` 的 best-effort 失败不回滚 uninstall，又说明对象离场事实与资格清算成败也被继续分层

因此这一章的最短结论是：

`stronger-request cleanup resurrection governor 最多能说对象能回来；stronger-request cleanup re-entitlement governor 才能说它回来之后恢复什么旧资格。`

再压成一句：

`会回来，不等于还是原来那个被授权的它。`

再压成更哲学一点的话：

`制度知道怎样让对象重新出现，不等于制度已经知道该把哪些旧权利重新还给它。`

## 4. 第一性原理：resurrection governance 回答“能否回来”，re-entitlement governance 回答“回来后恢复哪些资格”

从第一性原理看，
强请求清理复活治理与强请求清理再赋权治理处理的是两个不同主权问题。

`stronger-request cleanup resurrection governor`
回答的是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回到的是哪一层 world
4. 是否沿原 lineage 回来
5. 是否应该避免 clobber

`stronger-request cleanup re-entitlement governor`
回答的则是：

1. 对象回来后是否恢复旧 enabled state
2. 是否恢复旧 secrets
3. 是否恢复旧 pluginConfigs / options
4. 是否恢复旧 scope / precedence 位置
5. 是否恢复旧 identity，还是只给它一个新的 current seat
6. entitlement 是逐对象重签，还是要按 closure / dependency bundle 统一重签

如果把这两层压成一句：

`反正它已经回来了`

系统就会制造六类危险幻觉：

1. `return-means-enabled`
2. `return-means-configured`
3. `return-means-same-scope`
4. `return-means-same-identity`
5. `existence-means-entitlement`
6. `one-object-return-means-the-whole-old-closure-is-back`

所以这两层的哲学本质也不同：

`resurrection`

治理的是：

`对象能不能重新进入当前世界。`

`re-entitlement`

治理的是：

`对象重新进入之后，当前世界愿意重新把哪些可行动资格交还给它。`

如果再压缩一点，
两者的本质差别其实是：

1. `resurrection` 负责 `reappearance`
2. `re-entitlement` 负责 `re-authorization`

可见性与可行动性，
从来不是同一个问题。

## 5. Claude Code 多重安全技术的先进性：它把 comeback 后的资格恢复写成独立签发，而不是默认继承

Claude Code 在这一层真正先进的地方，
不是“对象一回来，系统就默认它还是原来那个被授权的对象”，
而是：

`它把 post-return qualification 写成一套独立、可 veto、可分层、可按身份回退的签发问题。`

这种先进性至少体现在六个方面：

1. `trust material does not auto-return`  
   old config / secrets 在 last-scope uninstall 时被显式清算。系统不默认“只要对象回来，旧 trust material 就仍可继续使用”。
2. `enabled truth is a fresh write`  
   `setPluginEnabledOp()` 把 enable / disable 写成 settings-first 独立动作。资格恢复不是对象返回的隐含尾巴。
3. `higher governance can still veto`  
   org policy 对 install / enable 都保留否决权。回来不等于获得豁免。
4. `seat legality remains checked`  
   current scope / precedence 仍要重新过 legality gate。系统不承认“曾经在哪，就天然还能回哪”。
5. `closure-level re-entitlement exists`  
   `installResolvedPlugin()` 会把 dependency closure 一次性写进 `enabledPlugins`。这说明再赋权有时不是单对象判断，而是关系网络的统一重签。
6. `identity fallback is explicit`  
   forked plan comeback 会拿 `newSlug`。系统宁可给你新 seat，也不假装原资格自然延续。

这些设计背后的源码思路也很清楚：

1. 不让 `presence` 越级冒充 `permission`
2. 不让 `return` 越级冒充 `enablement`
3. 不让 `content continuity` 越级冒充 `identity continuity`
4. 不让 `came back once` 越级冒充 `all old trust material still valid`
5. 不让 `one object reappears` 越级冒充 `its old dependency closure is automatically re-entitled`

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. 旧 cleanup object comeback 之后，不应自动恢复旧 wording / path / receipt entitlement。
2. config / secret / policy / scope / identity 需要被分开判断，不能压成一句“又能用了”。
3. enabled truth 必须是 fresh grant，而不是 existence side-effect。
4. 更高层 policy veto 必须保留在 comeback 之后，不能因为对象回来了就失效。
5. comeback 若涉及多对象关系网，也需要 closure-level qualification grammar。
6. 有些对象即便允许内容回来，也应只给 new seat，而不恢复 old seat。

## 6. 苏格拉底式自反诘问：我是不是又把“对象回来了”误认成了“它原来的全部资格也回来了”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 resurrection grammar 已经足够强，为什么还要再拆 re-entitlement？  
   因为对象能回来，不等于它原来的资格也一起回来。
2. 如果 disk / materialized world 已经恢复，是不是就说明 enabled truth 也恢复了？  
   不是。enabled truth 仍要经过独立 settings write。
3. 如果 plan 内容能恢复，是不是就说明 old identity 也恢复了？  
   不是。`copyPlanForFork()` 明确证明 content return 与 identity return 可以分开。
4. 如果 policy 没有阻止 comeback，是不是就说明它自动允许 old entitlement 恢复？  
   也不对。policy veto 只是更高层拒绝条件，不等于 entitlement 已被重新签发。
5. 如果旧 config / secrets 还没被完全删干净，是不是就说明它们仍然可信？  
   不能这样推。成熟系统恰恰会把 trust material 的回归单独治理。
6. 如果 cleanup 线现在还没有显式再赋权代码，是不是说明这层还不值得写？  
   恰恰相反。越是缺这层明确 grammar，越容易把“对象回来”偷写成“旧资格整包归位”。

这一串反问最终逼出一句更稳的判断：

`re-entitlement 的关键，不在对象会不会回来，而在系统能不能正式决定回来之后哪些旧权利还配继续属于它。`

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 resurrection governance，就已经足够成熟。`

而是：

`Claude Code 在 old config / secrets wipe、explicit enabled write、policy veto、divergence guard、closure-level entitlement write 与 new-slug identity fallback 上已经明确展示了 re-entitlement governance 的存在；因此 artifact-family cleanup stronger-request cleanup-resurrection-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-re-entitlement-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“怎么让旧对象回来”，而是“谁来决定回来之后它还算不算原来那个被授权的对象”。`
