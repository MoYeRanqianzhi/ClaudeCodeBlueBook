# 安全对象协议：为什么统一安全控制台必须先有宿主保真契约，再谈页面设计

## 1. 为什么在 `26` 之后还要继续写“安全对象协议”

`26-统一安全控制台交互状态机` 已经把问题推进到一个很关键的结论：

`统一安全控制台离真正落地，只差统一交互状态机与宿主适配契约。`

但如果继续往更底层追问，  
还会剩下一个比“页面如何切换”更根的问题：

`宿主之间到底在传什么东西？`

如果这个问题没有被单独说清楚，  
那么“统一控制台”“统一交互状态机”“统一诊断路径”都会在实现阶段重新碎掉。

因为页面、卡片、动作、回读这些东西，本质上都只是：

`某种对象的投影。`

真正决定系统能不能跨宿主不失真的，  
不是页面设计得多漂亮，  
而是：

`安全状态是否先以结构化对象存在。`

所以这一章的核心问题是：

`Claude Code 的统一安全控制台，为什么必须先定义一套安全对象协议，才能真正拥有宿主保真契约？`

## 2. 最短结论

从源码看，Claude Code 现在的下一层产品缺口，不是缺页面，  
而是缺：

`安全对象协议。`

更具体地说，系统已经具备：

1. 结构化协议消息
2. 结构化宿主状态字段
3. 结构化控制动作
4. 局部成熟的审批闭环

但它还没有把这些东西正式收束成：

`一组必须跨宿主保真的安全对象。`

于是当前实现就会在不同边界发生三种退化：

1. 对象被压成文本
2. 对象被拆成零散布尔量
3. 对象被直接忽略

所以 `27` 的核心判断是：

`统一安全控制台真正还缺的不是更多状态，而是把已有状态正式对象化，并把“允许如何降级”写成宿主契约。`

## 3. 协议层本来就已经不是“纯文本消息流”

## 3.1 `SDKStatusMessageSchema` 已经是结构化状态，而不是一句提示

`src/entrypoints/sdk/coreSchemas.ts:1533-1541` 定义的 `SDKStatusMessageSchema` 包含：

- `type`
- `subtype`
- `status`
- `permissionMode`
- `uuid`
- `session_id`

这很关键。  
因为这说明协议作者并不是把 status 当成“给 UI 看一眼的字符串”，  
而是把它当成：

`可被识别、带上下文、带会话身份的状态对象。`

尤其 `permissionMode` 出现在这里，  
进一步说明这类状态不是普通转场文案，  
而是和安全语义直接相关的控制面对象。

## 3.2 `SDKAuthStatusMessageSchema` 进一步证明认证过程本来就是对象

`src/entrypoints/sdk/coreSchemas.ts:1661-1669` 定义的 `SDKAuthStatusMessageSchema` 包含：

- `isAuthenticating`
- `output`
- `error`
- `uuid`
- `session_id`

这意味着认证过程也不是一句“正在登录”文案，  
而是一份明确带有：

1. 当前状态
2. 输出轨迹
3. 错误信息
4. 会话归属

的结构化对象。

也就是说，  
如果某个宿主最后只剩一句“auth failed”或者根本不展示 auth 过程，  
那不是协议天然做不到，  
而是：

`宿主自己把对象压扁了。`

## 3.3 协议整体已经在朝“对象流”而不是“答案流”设计

`src/entrypoints/sdk/coreSchemas.ts:1544-1569` 里的 `post_turn_summary`、  
`1648-1658` 里的 `tool_progress`，  
都进一步说明同一套协议在表达的是：

`结构化运行时对象。`

也就是说，Claude Code 的协议层已经不是“模型回你一句话”这种窄接口，  
而是：

`一个不断产出状态对象、控制对象和执行对象的运行时事件面。`

这正是统一安全控制台能够成立的前提。

## 4. 宿主本地其实也已经有很多对象槽位

## 4.1 `AppStateStore.ts` 已经保留了安全控制台最需要的骨架

`src/state/AppStateStore.ts:89-157` 很重要。  
这里直接保留了多组与安全控制台高度相关的状态：

- `toolPermissionContext`
- `remoteConnectionStatus`
- `replBridgeEnabled`
- `replBridgeExplicit`
- `replBridgeOutboundOnly`
- `replBridgeConnected`
- `replBridgeSessionActive`
- `replBridgeReconnecting`
- `replBridgeConnectUrl`
- `replBridgeSessionUrl`
- `replBridgeEnvironmentId`
- `replBridgeSessionId`
- `replBridgeError`

这说明宿主本地状态树并不是空的，  
也不是只能存一行 footer 文案。

它已经非常接近一套：

`可被统一控制台直接消费的状态对象池。`

## 4.2 `AppStateStore.ts` 还保留了外部能力对象和插件对象

`src/state/AppStateStore.ts:173-215` 继续说明宿主里本来就有：

- `mcp.clients`
- `mcp.tools`
- `mcp.commands`
- `mcp.resources`
- `mcp.pluginReconnectKey`
- `plugins.enabled`
- `plugins.disabled`
- `plugins.errors`
- `plugins.installationStatus`
- `plugins.needsRefresh`

这意味着外部能力链也不是只能靠“连接成功/失败”一句话被理解。

协议层和宿主层，其实都已经拥有不少对象化基础。  
真正缺的是：

`一层正式命名、正式约束的安全对象协议。`

## 4.3 默认状态也说明作者预设的是“结构化持有”，不是“即时拼字串”

`src/state/AppStateStore.ts:475-518` 里的默认值同样很说明问题：

1. bridge 各面默认都有独立字段
2. `toolPermissionContext` 被独立初始化
3. `mcp` 有完整对象容器，而不是单个标签值

这意味着系统在内部建模时，本来就倾向于：

`先持有对象，再渲染视图。`

所以当前问题不是内部世界没有对象，  
而是对象还没有被正式提升为：

`跨宿主必须保真的协议对象。`

## 5. 当前失真主要发生在适配器和子集化边界

## 5.1 `sdkMessageAdapter.ts` 会把部分对象压成文本，或直接忽略

`src/remote/sdkMessageAdapter.ts:168-266` 是这一章最关键的证据之一。

它说明：

1. `system/status` 被转换成普通 message
2. `auth_status` 被直接忽略
3. `tool_use_summary` 被直接忽略
4. `rate_limit_event` 被直接忽略
5. 其他多数 `system` subtype 也会被忽略

这意味着一旦走入这层适配器，  
原本在协议层是对象的东西，  
就会被重写成三种命运：

1. 文本化
2. 子集化
3. 丢弃

所以问题从来不只是“少了一条提示”，  
而是：

`对象身份在宿主边界丢失了。`

## 5.2 `bridgeMessaging.ts` 的 ingress 路径还进一步展示了对象子集化

`src/bridge/bridgeMessaging.ts:132-208` 说明在这条 ingress path 上：

1. `control_response` 单独分流
2. `control_request` 单独分流
3. `SDKMessage` 中只有 `user` 会被继续转发
4. 其他 non-user inbound message 会被显式忽略

这件事本身并不一定是 bug。  
但它非常清楚地暴露了一个事实：

`宿主消费面天然就是协议全集的子集。`

因此如果没有正式对象契约，  
不同宿主就会各自决定：

1. 哪些对象值得保留
2. 哪些对象可以缩略
3. 哪些对象可以完全不看

这正是统一安全控制台最危险的裂口。

## 5.3 `bridgeMessaging.ts` 对 control request 的处理再次证明“显式子集”是现实

`src/bridge/bridgeMessaging.ts:243-389` 又进一步显示：

这条路径当前显式处理的主要是：

- `initialize`
- `set_model`
- `set_max_thinking_tokens`
- `set_permission_mode`
- `interrupt`

未知 subtype 会返回 error，  
而不是静默成功。

这里有两层结论。

第一层是好的：

`显式失败优于假成功。`

第二层则说明现实：

`宿主当前支持面明确小于协议全集。`

所以统一安全控制台若要跨宿主成立，  
就不能只依赖“协议里存在什么”，  
而必须额外规定：

`宿主至少要保留哪些安全对象，允许以什么方式降级。`

## 6. 从第一性原理看，什么叫“安全对象”

安全对象不是“任何和安全有关的字段堆”。  
它至少要同时满足四个条件：

1. 能描述当前边界状态
2. 能指向某条证明链
3. 能绑定动作
4. 能支持 before / after 验证

从这个标准出发，我会把统一安全控制台真正需要的对象压成六族。

## 6.1 主体对象

回答：

`当前到底是谁在请求能力。`

最少应包含：

- auth 状态
- entitlement / connection 资格
- 会话归属
- 当前认证错误或进度

## 6.2 主权对象

回答：

`谁有资格定义当前运行时。`

最少应包含：

- `effective`
- `sources`
- `applied`
- 当前来源冲突和覆盖结果

## 6.3 授权对象

回答：

`当前动作为何被允许、阻断或待审批。`

最少应包含：

- `toolPermissionContext`
- `permissionMode`
- 断链原因
- 建议动作

## 6.4 包络对象

回答：

`动作现在被包在哪个执行边界里。`

最少应包含：

- sandbox 深度
- path / network 限制
- allow only / deny 语义

## 6.5 外部能力对象

回答：

`外部世界的哪一面当前可用、不可用、待恢复。`

最少应包含：

- MCP 状态
- plugin 状态
- bridge 状态
- reconnect / reload 能力

## 6.6 交互证明对象

回答：

`当前处于交互状态机的哪一段，以及动作后是否真的修复。`

最少应包含：

- 当前阶段
- 动作 id
- 默认刷新源
- before / after 差分
- 升级归属方

## 7. 宿主保真契约最少应写成哪四条硬约束

## 7.1 对象先于视图

宿主可以显示 pill、summary、dialog 或全页控制台，  
但这些都必须是对象投影，  
不能反过来只保留文案。

## 7.2 子集允许，但必须显式声明

某宿主可以不支持完整对象面，  
但它必须显式告诉系统：

1. 哪个对象被保留
2. 哪个对象被压缩
3. 哪个对象完全不可见

没有这一步，  
统一控制台就无法区分：

`真实正常`

和

`宿主看不见。`

## 7.3 动作必须绑定对象，不绑定字符串

动作应该直接作用于：

- 某个 MCP server 对象
- 某个 settings 主权对象
- 某个审批对象

而不是作用于一条模糊的“错误描述”。

## 7.4 验证必须比较对象差分，而不是比较提示文案

真正的安全验证不是：

`提示从红色变成绿色。`

而是：

`哪条对象字段发生了什么变化，因此哪条证明链重新成立。`

## 8. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不仅在于它已经有很多安全机制，还在于它离“对象化安全控制台”只差最后一层建模纪律。`

很多系统的安全 UI 做不起来，  
并不是因为它们不会写页面，  
而是因为它们从协议开始就没有把安全真相对象化。

Claude Code 不是这样。  
它已经有：

1. 结构化消息 schema
2. 结构化控制 schema
3. 结构化宿主状态树
4. 局部成熟的审批状态机

所以它下一步真正该做的不是“再做一堆页面”，  
而是：

`把哪些对象必须保真、哪些对象可以降级、降级后如何显式呈现，写成正式契约。`

这对所有 Agent 平台构建者的技术启示非常直接：

1. 安全设计最终会变成对象建模问题
2. 对象建模最终会变成宿主契约问题
3. 宿主契约如果不严，页面再好也会失真

## 9. 哲学本质

这一章背后真正更深的哲学，不是“对象比字符串高级”。  
而是：

`安全不是一句判断，而是一份可运输、可保留、可验证的真相。`

如果真相不能跨宿主运输，  
那它就不是系统真相，  
而只是局部界面的观感。

所以从第一性原理看，安全控制台的本质不是仪表盘，  
而是：

`真相运输系统。`

## 10. 苏格拉底式反思：这章最容易犯的错误是什么

### 10.1 我们是不是把“对象化”神化了

不能。  
对象化不是目的，  
它只是为了保证：

1. 因果不丢
2. 动作可绑
3. 验证可做

如果对象化不能带来这三件事，  
那只是另一种形式主义。

### 10.2 宿主子集是不是天然错误

也不是。  
窄宿主、轻宿主、只读宿主都可以成立。  
错误不在于子集，  
而在于：

`子集是隐性的。`

### 10.3 这是不是纯产品问题，而不是安全问题

不是。  
一旦安全真相在适配层被压扁，  
用户就会误判因果、误试动作、误归责对象。  
这已经不是纯体验问题，  
而是：

`安全控制本身的失真问题。`

### 10.4 如果只能做一件改进，最该做什么

不是先做新页面。  
最该做的是：

`先定义统一安全对象协议和宿主降级规则。`

因为没有这一步，  
任何控制台都只会在不同宿主里重新碎裂。

## 11. 结语

`26` 回答的是：为什么统一安全控制台必须有交互状态机。  
`27` 继续推进后的结论则是：

`统一安全控制台真正的最小前提，不是页面，而是安全对象协议。`

只有当主体、主权、授权、包络、外部能力和交互证明这些对象，  
被正式规定为跨宿主必须保真的安全真相，  
统一控制台才不会在实现阶段重新退化成：

`一组互相看不懂的提示页面。`
