# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `230` 之后还必须继续写 `231`

`230-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层` 已经回答了：

`stronger-request cleanup 线即便知道旧世界何时正式结束，也还需要 tombstone governor 去决定退役后留下什么最小 marker。`

但如果继续往下追问，
还会撞上一层更容易被偷写成
“只要墓碑已经留下，后面怎样回来也就自动清楚了”
的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定旧对象退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、回来时算谁、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 resurrection 正对照，
会发现它其实明确拆开了四件事：

1. `authoritative clearing`
2. `re-entry gate`
3. `identity policy`
4. `active-layer admission`

最硬的证据至少有六组：

1. `src/utils/plugins/cacheUtils.ts:67-73,82-92,122-130`
   证明 `.orphaned_at` 的清除由 authoritative installed set 决定，
   而不是 marker 自己决定
2. `src/utils/plugins/refresh.ts:1-18,72-79,123-137`
   证明 active plugin return 是专门的 Layer-3 refresh primitive，
   不是 marker 清掉后的自动副作用
3. `src/commands/reload-plugins/reload-plugins.ts:37-56`
   证明 `/reload-plugins` 是显式 re-entry signal，
   系统不会把“磁盘世界变了”偷写成“active world 已接纳”
4. `src/services/plugins/PluginInstallationManager.ts:56-59,135-179`
   与
   `src/hooks/useManagePlugins.ts:287-303`
   证明 repo 还会继续区分：
   哪些 return path 可以 auto-refresh，
   哪些 return path 必须停在 `needsRefresh`
5. `src/utils/plans.ts:164-231,239-257`
   证明 plan resurrection 不只是 undelete，
   还要治理 lineage、evidence 与 identity policy
6. `src/utils/conversationRecovery.ts:539-547`
   与
   `src/utils/sessionStorage.ts:1468-1473`
   共同证明 resurrection grammar 必须进入正式恢复链，
   而 deletion grammar 本身不自动携带 re-entry grammar

这说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 identity policy 与什么 active-layer admission 下，这个旧对象还能重新回到 current world。`

所以 `230` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`
不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk / materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery` 就等于 resurrection 的全部语义。

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线是否完全复用这套 grammar，仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer`

证据：

1. `cacheUtils.ts:67-73,82-92,122-130`
   证明 `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
2. `refresh.ts:1-18,72-79,123-137`
   与
   `reload-plugins.ts:37-56`
   证明 marker clearing / disk presence 不等于 active components 已重新进入当前 session
3. `PluginInstallationManager.ts:56-59,135-179`
   与
   `useManagePlugins.ts:287-303`
   证明 repo 明确要求某些 update path 停在 `needsRefresh`，
   而不是默认自动接纳
4. `plans.ts:164-231,239-257`
   证明 plan resurrection 需要 lineage、evidence 与 identity policy，
   而不是简单 undelete
5. `conversationRecovery.ts:539-547`
   把 resurrection 接进恢复主线，
   说明这不是局部 helper，而是正式 recovery grammar
6. `sessionStorage.ts:1468-1473`
   的 tombstone path 只回答怎样删除旧对象，
   不回答它何时、以什么身份重新回来

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据、何种身份政策与何种激活层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

## 4. 第一性原理：tombstone governance 回答“留什么最小残留事实”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

`stronger-request cleanup tombstone governor`
回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

`stronger-request cleanup resurrection governor`
回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk / materialized world，还是 active / runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句“反正 marker 还在或 marker 已清掉”，
系统就会制造五类危险幻觉：

1. `marker-clear-means-resurrection`
2. `disk-return-means-active-return`
3. `content-recovered-means-identity-restored`
4. `tombstone-is-bidirectional`
5. `any-old-object-can-come-back`

再用苏格拉底式反问压一次：

1. 如果 tombstone governance 已经等于 resurrection governance，
   为什么 repo 还要把 `removeOrphanedAtMarker()`、`refreshActivePlugins()` 与 `/reload-plugins` 拆成多层？
   因为“marker 清掉”与“active world 接纳它”根本不是同一个时刻。
2. 如果对象内容恢复就等于对象身份恢复，
   为什么 `copyPlanForFork()` 还要显式生成 new slug？
   因为 resurrection 还要治理 identity，不只是治理 bytes。
3. 如果 tombstone path 天然自带复活逻辑，
   为什么 `removeTranscriptMessage()` 只回答怎样删，不回答怎样回来？
   因为 deletion grammar 与 re-entry grammar 是两层不同主权。

## 5. plugin 线先证明：`.orphaned_at` 被清掉，并不等于插件已经重新回到当前 active world

`src/utils/plugins/cacheUtils.ts:67-73`
先把 orphan cleanup 的制度边界写死：

1. Pass 1 清理 installed versions 的 `.orphaned_at`
2. Pass 2 才处理真正 orphaned 的版本

而
`src/utils/plugins/cacheUtils.ts:82-92`
又继续把 authoritative clearing 写得很明白：

1. 先从 `installed_plugins.json` 读出 authoritative `installedVersions`
2. 只对这个集合里的 version path 执行 `removeOrphanedAtMarker()`

再看
`src/utils/plugins/cacheUtils.ts:122-130`，
`removeOrphanedAtMarker()` 只负责删 marker，
并不负责任何 active return。

这说明：

`谁配清墓碑`

不是 marker 自己决定的，
而是由 authoritative installed state 决定。

但这只是第一步。

`src/utils/plugins/refresh.ts:1-18`
又明确把 plugin world 拆成三层：

1. intent
2. materialization
3. active components

而
`src/utils/plugins/refresh.ts:72-79`
与
`src/utils/plugins/refresh.ts:123-137`
进一步证明：

1. `refreshActivePlugins()` 是专门的 Layer-3 refresh primitive
2. 它会 clear caches
3. 它会把 `needsRefresh` 消费为 `false`
4. 它会 bump `pluginReconnectKey`

也就是说，
即便 marker 已经被清掉，
对象也还没有自动回到 running session 的 active world。

`src/commands/reload-plugins/reload-plugins.ts:37-56`
则把这个结论再钉死一遍：
显式运行 `/reload-plugins`，
才会触发 `refreshActivePlugins(context.setAppState)`，
然后系统才返回
`Reloaded: ...`
这样的 active-world receipt。

所以 plugin 线已经很成熟地展示出：

`authoritative marker clearing != active-layer admission`

这恰恰就是 resurrection governance 与 tombstone governance 的分层。

## 6. `needsRefresh` 再证明：repo 明确拒绝把“对象重新存在”偷写成“当前会话已经重新接纳它”

`src/services/plugins/PluginInstallationManager.ts:56-59`
先把 background reconciliation 之后的世界拆成两类：

1. new installs
2. updates only

继续看
`src/services/plugins/PluginInstallationManager.ts:135-179`：

1. 新 marketplace 安装完成时，
   repo 会尝试 `refreshActivePlugins()`
2. 但如果 auto-refresh 失败，
   会回退成 `needsRefresh`
3. 如果只是 updates only，
   则根本不自动接纳，
   而是直接设置 `needsRefresh`

再看
`src/hooks/useManagePlugins.ts:287-303`：

1. `needsRefresh` 触发的不是 silent auto-refresh
2. 而是通知：
   `Plugins changed. Run /reload-plugins to activate.`
3. 注释还明确写出：
   `Do NOT auto-refresh`

这三组证据非常关键。

它们说明 Claude Code 不只是把 resurrection governance 写成“回来 / 不回来”二选一，
而是继续拆成：

1. 哪些 return path 可以自动接纳
2. 哪些 return path 必须停在 pending-like state
3. 哪些 return path 要由用户显式触发 `/reload-plugins`

这是一种很成熟的安全设计：

`系统不把“对象重新存在”偷写成“当前会话已经重新承认它”。`

也就是说，
resurrection governance 真正回答的不是：

`对象还能不能回来`

而是：

`它凭什么回来，又应该在什么 admission policy 下回来。`

## 7. plan 线给出第二个强正例：对象复活时，系统还要治理 lineage、evidence 与 identity policy

`src/utils/plans.ts:164-231` 的 `copyPlanForResume()` 很值钱。

它不是看到 plan file 缺失就粗暴恢复。
它要求：

1. log 里确实能抽出 slug
2. 把 slug 重新绑定到 target session ID
3. 先尝试直接读取当前 plan file
4. 只有在 `ENOENT` 且处于 remote session 时才尝试 recovery
5. 优先用 file snapshot，
   失败后才回到 message history

这说明 plan resurrection 的本体不是：

`内容还在，就给你拉回来`

而是：

`只有在 lineage 和 evidence 都成立时，才允许对象重新进入 current session world。`

更关键的是
`src/utils/plans.ts:239-257`
里的 `copyPlanForFork()`：

1. 不复用 original slug
2. 生成 new slug
3. 把原计划内容写进新文件

这直接证明：

`内容回来`

不等于

`原身份回来`

也就是说 resurrection governance 还必须回答：

`re-entry 后对象到底算 old identity 还是 new identity。`

再看
`src/utils/conversationRecovery.ts:539-547`：
恢复主线会主动调用 `copyPlanForResume(log, asSessionId(sessionId))`，
说明 resurrection 不是局部 helper 的偶发行为，
而是正式恢复制度的一部分。

这对 stronger-request cleanup 的技术启示很强：

`真正的 resurrection governance 不会只停在局部 helper，它必须进入正式恢复链。`

## 8. tombstone 删除路径反证：删除 grammar 的存在不自动携带复活 grammar

`src/utils/sessionStorage.ts:1468-1473`
把 tombstone 删除路径写得非常克制：

1. 注释只说：
   `Remove a message from the transcript by UUID.`
2. 语义只说：
   `Used when a tombstone is received for an orphaned message.`
3. 实现只做：
   `removeMessageByUuid(targetUuid)`

它没有再回答：

1. 这些对象在什么条件下还能回来
2. 回来时是否还是原 UUID
3. 回来时谁来宣布它重新成为 current truth

这恰恰是一条反向强证据：

`deletion grammar`

和

`re-entry grammar`

本来就不是同一层主权。

如果 stronger-request cleanup 线未来只补出了 marker grammar，
却没有同时补出：

1. authoritative clearing
2. re-entry gate
3. identity policy
4. active-layer admission

那么旧 cleanup 世界仍会以
“marker 留下了，但谁配回来没人正式回答”
的方式继续制造歧义。

## 9. 这层分工的技术先进性与设计启示

Claude Code 在这层设计上真正先进的地方，
不是“对象坏了还能恢复”，
而是它已经把 resurrection grammar 压成四个彼此不能偷写的技术问题：

1. `authoritative clearing`
   不是谁看到 marker 都能删，
   而是要由 authoritative installed state 决定
2. `re-entry gate`
   不是对象回到磁盘就自动恢复，
   而是要经过 `/reload-plugins` 或正式恢复链
3. `identity policy`
   不是内容回来就等于旧身份回来，
   而是要明确 old identity / new identity 的边界
4. `active-layer admission`
   不是 materialized return 就等于 runtime acceptance，
   而是还要决定谁配让它重新进入 active world

这四层给 stronger-request cleanup 线的设计启示也非常直接：

1. 旧 cleanup path / promise / receipt 如果要允许 resurrection，
   就不能只补一个 marker-clearing 操作，
   还必须补正式 re-entry gate
2. resurrected cleanup object 如果只是借旧内容重新生成 current object，
   就不应自动继承 old identity 与 old entitlement
3. cleanup resurrection 的 receipt 不应由 tombstone 消费方顺手代签，
   而应由掌握 current admission truth 的更强 signer 单独签发
4. 真正可靠的 resurrection grammar
   必须进入正式恢复主线，
   而不是散落在若干局部 helper 的偶发回填里

所以这一章最后可以压成一句硬结论：

`stronger-request cleanup tombstone governor 负责决定旧对象退役后留下什么最小痕迹；stronger-request cleanup resurrection governor 才负责决定这些带着痕迹的旧对象凭什么、以什么身份、在什么层面重新进入 current world。`
