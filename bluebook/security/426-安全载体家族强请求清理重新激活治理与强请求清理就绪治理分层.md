# 安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层：为什么artifact-family cleanup stronger-request cleanup-reactivation-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-readiness-governor signer

## 1. 为什么在 `425` 之后还必须继续写 `426`

`425-安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道 returned object 应按哪组 current truth 工作，也还需要 reactivation governor 去决定这组 truth 何时真正接管当前运行世界。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“当前世界既然已经完成 takeover，就说明现在已经能放心继续用”
的错觉：

`只要 stronger-request cleanup reactivation governor 已经让新的 truth 接管 current world，它就自动拥有了决定对象现在是否真的 connected、authenticated、capable、ready-for-use 的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 在 MCP 线已经成熟写出的 readiness 正对照，
会发现它其实明确拆开了八件事：

1. `typed state machine`
2. `reactivation-triggered pending`
3. `explicit enable still pending`
4. `notification honesty`
5. `manager status grammar`
6. `CLI health grammar`
7. `consumer hard gate`
8. `runtime revocation`

最硬的证据至少有八组：

1. `src/services/mcp/types.ts:179-226`
   直接把 `MCPServerConnection` 写成
   `connected / failed / needs-auth / pending / disabled`
2. `src/services/mcp/useManageMCPConnections.ts:149-153,765-853`
   说明 `/reload-plugins` 之后新 client 会重新进入 `pending`
3. `src/services/mcp/useManageMCPConnections.ts:1109-1123`
   说明 even explicit enable 也仍只会先回到 `pending`
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
   继续把 `failed` 与 `needs-auth` 单独显化
5. `src/commands/plugin/ManagePlugins.tsx:513-519`
   明确保留 `connected / disabled / pending / needs-auth / failed`
6. `src/cli/handlers/mcp.tsx:26-35`
   把 health grammar 压成 `Connected / Needs authentication / Failed to connect`
7. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:78-95`
   对 consumer 明确要求 `client.type === 'connected'` 且 capability 存在
8. `src/services/tools/toolExecution.ts:1599-1628`
   会把一个原本 `connected` 的 client 在运行时降成 `needs-auth`

这说明：

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

和

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

仍然不是一回事。

前者最多能说：

`新的 truth 已经接管当前世界。`

后者才配说：

`这个当前世界现在真的足够可信，可以被后续 reader、tool 与 user-facing surface 继续消费。`

所以 `425` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层`

也就是：

`stronger-request cleanup reactivation governor 负责决定 truth 何时接管 current world；stronger-request cleanup readiness governor 才负责决定这个 current world 何时真正配被继续使用。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 readiness manager，
而是在说：

1. repo 已经把 current-world object 的 not-ready 状态写成正式联合类型
2. repo 已经把 `reactivated` 与 `ready-for-use` 拆成两层
3. repo 已经让 surface、consumer 与 runtime downgrade 共同维护 ready verdict

第二条：

这里说的 `readiness governance`
不是在声称：

`只有永远不会再出错的对象才算 ready。`

当前更稳的说法只能是：

`readiness governance 首先回答的是下游 consumer 现在能不能继续安全消费它，而不是回答它未来是否绝不会再失败。`

第三条：

这里也不是把：

`connected`

单独当成全部 readiness grammar。

真正关键的是：

1. `pending` 是 readiness 未闭合
2. `needs-auth` 是 readiness 被阻断
3. `failed` 是 readiness 失败
4. capability gate 仍会继续限制 downstream consumption

第四条：

这里尤其不能过度声称：

`对象曾经 connected 过一次，就说明它现在仍然 ready。`

当前更稳的说法只能是：

`readiness 可以被新的运行时证据持续撤销；它不是一次性判词。`

第五条：

这里也不能过度声称：

`reload summary 已经给了用户，所以 capability proof 也已经给了。`

当前更稳的说法只能是：

`summary 只回答 takeover choreography 已经运行，不自动回答所有 consumer 现在都可继续依赖它。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-reactivation-governor signer 仍不等于 stronger-request cleanup-readiness-governor signer`

证据：

1. `MCPServerConnection` 联合类型直接把 client 拆成 `connected / failed / needs-auth / pending / disabled`
2. `useManageMCPConnections()` 在 `pluginReconnectKey` 触发后会把新 client 重新置为 `pending`
3. 显式 enable 也仍只会先进入 `pending -> reconnect`
4. `useMcpConnectivityStatus()` 继续只把 `failed / needs-auth` 作为 readiness 降级 signal
5. `ManagePlugins.tsx` 明确保留五态 status grammar
6. `/mcp` health 继续把 `Connected / Needs authentication / Failed to connect` 分开
7. `ReadMcpResourceTool` 只接受 `connected` + capability-present 的 client
8. `toolExecution.ts` 在 `McpAuthError` 上会把 `connected` 运行时撤销成 `needs-auth`

因此这一章的最短结论是：

`stronger-request cleanup reactivation governor 最多能说新的 truth 已经接管当前世界；stronger-request cleanup readiness governor 才能说这个世界现在是否真的配被继续消费。`

再压成一句：

`重新激活，不等于已经就绪。`

再压成更硬的一句：

`会接管，不等于会就绪。`

再压成更哲学的一句：

`制度知道怎样让新 truth 接管当前世界，不等于制度已经知道这个世界现在配不配被继续消费。`

## 4. 第一性原理：reactivation governance 回答“谁拥有当前世界”，readiness governance 回答“这个世界此刻能不能成为行动依据”

从第一性原理看，
强请求清理重新激活治理与强请求清理就绪治理处理的是两个完全不同的主权问题。

`stronger-request cleanup reactivation governor`
回答的是：

1. stale world 是否已被 current world 替换
2. 哪个时刻完成 active-world takeover
3. 哪些 active components 已经按新 truth 接线
4. 当前 session 是否已经切到新的 runtime world
5. 不同 mode 下谁来触发这种 takeover

`stronger-request cleanup readiness governor`
回答的则是：

1. 当前对象究竟是 `connected`、`pending`、`needs-auth`、`failed` 还是 `disabled`
2. 用户能否把它当成当前可信对象继续依赖
3. downstream tool / reader 是否有权继续消费它
4. 新的运行时证据是否足以撤销旧的 ready verdict
5. 哪些 non-ready truth 必须继续被显化，而不能被 takeover success 淹掉

如果把这两层压成一句：

`反正它已经 reload 回来了`

系统就会制造六类危险幻觉：

1. `active-means-ready`
2. `pending-is-just-a-transition-noise`
3. `operator-intent-equals-proof`
4. `visible-means-consumable`
5. `connected-once-means-connected-now`
6. `reload-summary-equals-capability-proof`

所以这两层的哲学本质也不同：

`reactivation`

治理的是：

`谁拥有当前世界。`

`readiness`

治理的是：

`这个世界此刻能不能被托付为行动依据。`

如果再压缩一点，
两者的本质差别其实就是：

1. `reactivation` 负责 `world is updated`
2. `readiness` 负责 `world is trusted for use`

一个 object 即便已经重新进入 current world，
也完全不等于 downstream action 现在就该继续压在它身上。

## 5. Claude Code 多重安全技术的先进性：它把“能否继续用”写成跨平面的硬制度，而不是让“已经接回当前世界”偷签 capability proof

Claude Code 在这一层真正先进的地方，
不是“它也有状态机”这么简单，
而是：

`它把就绪判定写成类型、连接流程、状态显化、consumer gate 与运行时撤销共同维护的跨平面制度。`

这种先进性至少体现在六个方面：

1. `typed state machine`
   readiness 先在类型层被表达，不依赖后续 consumer 自由脑补。
2. `pending is first-class`
   reactivation 之后的等待层不是被隐藏，而是被正式承认为制度状态。
3. `surface grammar stays honest`
   notification、manager 与 CLI 都不会把 `pending / needs-auth / failed` 压成一条模糊“已恢复”。
4. `consumer gate is stricter than visibility`
   `ReadMcpResourceTool` 只接受 `connected` + capability-present，而不是“系统里看得见”。
5. `runtime revocation exists`
   `connected` 不是永久通行证；`McpAuthError` 这样的新证据可以把它撤回。
6. `operator intent is weaker than proof`
   显式 enable 也只能把对象送回 `pending`，不能越级签发 ready verdict。

这些设计背后的源码思路也很清楚：

1. 不让 `reactivated` 越级冒充 `ready`
2. 不让 `visible` 越级冒充 `consumable`
3. 不让 `operator wanted it` 越级冒充 `runtime has proved it`
4. 不让 `old verdict` 越级冒充 `current verdict`
5. 不让 `takeover success` 越级冒充 `capability proof`

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. returned cleanup object 即便已接回 current world，也不能因此跳过 readiness adjudication。
2. pending 应是正式制度状态，而不是实现细节。
3. non-ready truth 必须继续显化，不能被“对象已经回来”掩盖。
4. consumer gate 必须只接受更强 readiness fact，而不是 mere presence。
5. runtime evidence 需要有权持续撤销 ready verdict。
6. 就绪判断的核心不是“对象现在在不在场”，而是“对象现在够不够可信，足以支撑继续行动”。

## 6. 苏格拉底式自反诘问：我是不是又把“当前世界已切换”偷换成了“当前行动可被托付”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 reactivation governance 已经等于 readiness governance，为什么 `pluginReconnectKey` 触发后系统还要把新 client 重新送回 `pending`，而不是直接判成 ready？
   因为“truth 已接进 current world”与“它现在配被继续使用”根本不是同一句制度。
2. 如果对象一旦被当前世界接纳就已经 ready，为什么 `ReadMcpResourceTool` 还要坚持 `client.type === 'connected'` 的 hard gate？
   因为 downstream consumer 只接受 readiness truth，不接受 mere reactivation。
3. 如果 explicit enable 已经足以表达系统意图，为什么 enable path 还要先 `pending -> reconnect`？
   因为 operator intent 只能触发重新证明流程，不能越级替 runtime 签出 ready verdict。
4. 如果 readiness 是一次性 verdict，为什么 `toolExecution.ts` 还要在 `McpAuthError` 上把 `connected` 降成 `needs-auth`？
   因为 readiness 不是永久席位，而是会被新证据持续改写的主权。
5. 如果用户已经能在 UI 里看见对象，是不是就说明它已经足够可信？
   不是。visible 只是 current-world presence，不是 capability proof。
6. 如果 cleanup 线现在还没有显式 readiness grammar，是不是意味着这层可以晚一点再说？
   恰恰相反。越是缺这层 formal grammar，越容易把“世界已切换”误说成“行动可托付”。

这串反问最终逼出一句更稳的判断：

`readiness 的关键，不在对象有没有回到当前世界，而在系统能不能正式决定这个当前世界此刻是否足够可信，足以支撑继续行动。`

如果再追问“怎样做得更好”，
更好的方法也不是继续堆叠 `ready` 的同义词，
而是继续把四个问题分开问清：

1. 谁签字说它 ready
2. 谁消费这份 ready
3. 谁显化 non-ready truth
4. 谁能在新证据到来时撤销旧 verdict

只有把这四个问题分开，
`reactivation` 才不会再次偷走 `readiness` 的主权。

## 7. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 reactivation governance，就已经足够成熟。`

而是：

`Claude Code 在 MCP typed state machine、pending re-adjudication、honest surface grammar、consumer hard gate 与 runtime revocation 上已经明确展示了 readiness governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reactivation-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“如何让对象重新接管当前世界”，而是“接管之后此刻到底敢不敢继续用”。`
