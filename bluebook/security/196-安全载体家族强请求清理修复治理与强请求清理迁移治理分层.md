# 安全载体家族强请求清理修复治理与强请求清理迁移治理分层：为什么artifact-family cleanup stronger-request cleanup-repair-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-migration-governor signer

## 1. 为什么在 `195` 之后还必须继续写 `196`

`195-安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层` 已经回答了：

`stronger-request cleanup 线即便未来长出 verifier，也还需要 repair governor 去决定到底修 metadata、修 scheduler、修 executor、补 receipt，还是降级 promise wording。`

但如果继续往下追问，
还会碰到下一层更容易被误写成“修完就结束”的错觉：

`只要 repair governor 已经决定怎么修，它就等于自动拥有了把旧 stronger-request 世界平滑迁到新世界的主权。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看：

1. `src/main.tsx:323-341` 直接维护启动迁移链，说明 repo 并不把“修法确定”偷写成“旧世界自然退场”
2. `src/migrations/migrateLegacyOpusToCurrent.ts:13-51` 说明 model repair 之后还要决定  
   哪些 source 被重写、哪些 source 只在 runtime remap、是否记 timestamp 供一次性通知
3. `src/migrations/migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-18` 又说明旧 key 到新 key 的过渡并不是简单 rename，而要治理何时复制、何时删除旧 key、何时保持 idempotent
4. `src/utils/plugins/cacheUtils.ts:20-99` 里的 `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days` 进一步说明旧插件版本不会“新版本一来就立刻消失”；迁移治理要决定 grace window 与 orphan cleanup 节奏
5. `src/utils/plans.ts:150-248` 的 `copyPlanForResume()` / `copyPlanForFork()` 又把 continuity strategy 明确摆上台面：旧 plan artifact 怎样被 current session reuse、recover、copy，而不是被粗暴抹平
6. 与这些正对照相比，stronger-request cleanup 线当前虽然已暴露出 temporal、propagation 与 receipt gap，并且 repair governor 已开始变得可命名，但还没有谁正式决定：  
   旧 startup-delete wording 何时退场  
   旧 home-root `plans` cleanup law 如何 phase out  
   旧 receipt / old echo law 是否需要 grace window、dual-read、runtime remap 或 one-time notice

这说明：

`artifact-family cleanup stronger-request cleanup-repair-governor signer`

和

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

仍然不是一回事。

前者最多能说：

`我们已经决定应该修哪一层。`

后者才配说：

`旧 stronger-request law、旧 path、旧 receipt、旧 promise 应怎样被带过渡、作废、保留兼容期，或者被重签发。`

所以 `195` 之后必须继续补的一层就是：

`安全载体家族强请求清理修复治理与强请求清理迁移治理分层`

也就是：

`stronger-request cleanup-repair governor 只能决定修什么；stronger-request cleanup-migration governor 才能决定旧世界如何在不制造新谎言的前提下被迁到新世界。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-migration-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个 migration policy object 只差导出，
而是在说：

1. repo 在其他子系统里已经明确展示出 migration governance 的存在
2. migration governance 回答的是兼容期、旧值重写、旧路径保留、旧资产回收时机与用户通知
3. stronger-request cleanup 线今天若要真正进入成熟世界，也迟早要回答这些问题

第二条：

这里说的 `migration governance`

不是在声称：

`修复之后一律都要做批量重写。`

真正要说的是：

`迁移治理可能意味着 source rewrite、runtime remap、completion flag、one-time notification、orphan grace window，或 continuity-preserving copy/reuse/recovery；关键不在迁不迁，而在谁来决定旧世界怎样退场。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部迁移问题。

真正关键的是：

1. temporal gap 修完后，旧 wording 何时退场
2. propagation gap 修完后，旧 home-root law 是否仍保留一段兼容窗口
3. receipt gap 修完后，旧 surfaced truth 是否需要重签或 grace period
4. stronger-request cleanup 线当前没有一处可见代码显式持有这些过渡决定

第四条：

这里尤其不能过度声称：

`migration governance` 一定意味着“旧世界马上消失”。`

当前更稳的说法只能是：

`plugin orphan cleanup、model remap 与 plan continuity 已经说明：成熟系统更常见的做法恰恰是让旧世界按受治理的节奏退出，而不是被一次性粗暴抹平。`

第五条：

这里也不能过度声称：

`runtime remap` 就已经等于 migration governance 的全部。

当前更稳的说法只能是：

`runtime remap 只是迁移语法之一；它仍需要与 source rewrite、grace window、notification、idempotent flag 等共同构成完整 migration grammar。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类“stronger-request cleanup-repair-governor signer 仍不等于 stronger-request cleanup-migration-governor signer”证据：

1. `main.tsx:323-341` 直接维护一组启动迁移函数，说明 repo 已把“旧世界怎样退场”当成正式 startup governance
2. `migrateLegacyOpusToCurrent.ts:13-51` 说明 model 修复之后还要决定  
   只改 `userSettings` 还是保留 project/local/policy pin  
   是否 runtime remap 但不重写某些来源  
   是否保存 timestamp 供 one-time notification
3. `migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-18` 说明旧 key 到新 key 的过渡还要决定  
   何时复制值、何时删除旧 key、何时保持幂等
4. `cacheUtils.ts:20-99` 的 `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days` 说明旧插件版本不是“新版本一来就立即消失”；迁移治理要决定 grace window 和 orphan cleanup 节奏
5. `plans.ts:150-248` 的 `copyPlanForResume()` / `copyPlanForFork()` 说明 plan artifact 的过渡也不是单纯 repair；还要决定 resume reuse、missing-file recovery 与 fork copy/new-slug policy
6. 与这些正例相比，stronger-request cleanup 线当前虽已开始显露 repair-governance，但还没有谁正式决定旧 law、旧 path、旧 receipt 与旧 wording 应怎样 phased exit

因此这一章的最短结论是：

`stronger-request cleanup-repair-governor signer 最多能说“该修哪里”；stronger-request cleanup-migration-governor signer 才能说“旧世界怎样有次序地退出，而不把兼容世界和新世界一起搞乱”。`

再压成一句：

`会修，不等于会迁。`

## 4. 第一性原理：repair governor 回答“修哪层”，migration governor 回答“旧世界如何不被粗暴抹平”

从第一性原理看，
强请求清理修复治理与强请求清理迁移治理处理的是两个不同时间面的主权问题。

stronger-request cleanup-repair governor 回答的是：

1. 应修 metadata、runtime、scheduler、executor 还是 promise
2. 哪条路径现在不可信
3. 哪类 drift 必须立即止血
4. 哪种 stronger claim 应先撤回
5. 哪一层对当前 bug 负主责

stronger-request cleanup-migration governor 回答的则是：

1. 旧值是立刻重写，还是先 runtime remap
2. 哪些 source 允许被改写，哪些必须保持原样
3. 兼容窗口多长
4. 旧缓存、旧目录、旧版本何时清理
5. 是否需要 one-time notification、completion flag 或 grace period

如果把这两层压成一句“决定修复方案就行”，
系统就会制造五类危险幻觉：

1. fix-means-cutover illusion  
   只要修法确定，就误以为可以立刻切到新世界
2. one-write-fits-all illusion  
   只要一个 source 被重写，就误以为所有 source 都该被重写
3. no-grace-needed illusion  
   只要新规则更正确，就误以为旧世界不需要兼容窗口
4. migration-equals-rewrite illusion  
   只把 migration 理解成“改文件”，忽略 runtime remap、orphan retention、通知与幂等控制
5. artifact-world-is-stateless illusion  
   忽略旧 receipts、旧 path、旧 slug、旧 cache 仍在现实中继续活着

所以从第一性原理看：

`repair governor` 决定修复方向；
`migration governor` 决定旧世界如何安全让位。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup repair governance 已经等于 migration governance，为什么 `migrateLegacyOpusToCurrent()` 还要只改 `userSettings`、保留其他 source、不用 completion flag 而改用 timestamp notification？
   因为“知道该修什么”与“决定旧世界怎样退出”本来就不是同一主权。
2. 如果修好 `plansDirectory` propagation gap 就已经足够，为什么 `copyPlanForResume()` / `copyPlanForFork()` 还要回答 reuse、recover、copy 与 new-slug policy？
   因为 plan artifact 的过渡世界本来就需要 continuity grammar，而不是只要新路径正确就结束。
3. 如果 migration 只是 rewrite，为什么 plugin 线还要先 `.orphaned_at` 再等 7 天 cleanup？
   因为旧世界体面退出，本来就需要 grace window 与 staged retirement。`

## 5. 模型迁移先给出最强正例：成熟系统不会把“更正新规则”直接写成“粗暴覆盖所有旧值”

`src/main.tsx:323-341` 直接维护了一组 migration calls。
这本身就说明：

`新规则进入系统，不等于旧设置自然消失。`

更具体地看，
`src/migrations/migrateLegacyOpusToCurrent.ts:13-51` 做了几件极其 migration-governor 的事：

1. 只改 `userSettings`
2. project/local/policy settings 仍保留原样
3. runtime 继续 remap
4. 记录 `legacyOpusMigrationTimestamp` 供 REPL 做 one-time notification

这些决定没有一条是在回答“要不要修模型字符串”。  
它们回答的是：

`修完之后，旧世界哪些来源应该被尊重、哪些来源只在运行时兼容、哪些用户应被告知。`

这就是 migration governance 的本体。

## 6. 配置键迁移再给出第二个正例：迁移治理并不等于“把旧字段替换掉”，而是决定新旧字段怎样并存多久

`src/migrations/migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-18` 很关键。

它不是暴力删旧 key，
而是：

1. 只有 old key 存在且 new key 未设置时才复制
2. 复制到新 key
3. 再删除旧 key
4. 保持 idempotent

这说明 config-key 迁移治理回答的是：

`新旧 truth 怎样交接，才能既不重复写坏，也不让旧世界突然蒸发。`

stronger-request cleanup 线未来若要修 startup-delete wording、repair `plansDirectory` 传播，或补 stronger receipt semantics，
同样会遇到这类问题：

`旧 promise、旧 path、旧 law 是双读一段时间，还是立即作废。`

## 7. plugin 线给出第三个更强正例：真正成熟的 migration governor 会管理旧缓存世界的延迟退场，而不是只管新结构能不能工作

`src/utils/plugins/cacheUtils.ts:20-99` 最值钱的地方在于：

1. 旧版本先被标记 `.orphaned_at`
2. `CLEANUP_AGE_MS = 7 days`
3. 超过 7 天才真正删除
4. 重新安装时又会移除 stale marker

这里体现的就是最典型的 migration-governor 主权：

`旧世界何时退场，不由 repair 逻辑瞬间决定，而由兼容窗口治理来决定。`

这对 stronger-request cleanup 线是极其强的技术启示。
一旦未来要修 `plansDirectory` propagation gap、receipt semantics 或 startup wording，
就一定会面对同样的问题：

1. 旧 cleanup semantics 立即失效还是先进入 grace window
2. 旧 home-root artifact 保留多久
3. 过渡期内是 dual-read、dual-clean 还是 one-way migrate

## 8. plans 自己也已经悄悄暴露出 migration 问题：resume / fork 不是 repair 逻辑，而是 continuity-preserving migration grammar

`src/utils/plans.ts:150-248` 的 `copyPlanForResume()` / `copyPlanForFork()` 很关键。

它们处理的不是 repair；  
它们处理的是：

1. 旧 session plan 如何被 new/current session reuse
2. 缺失文件时怎样从 file snapshot / message history 恢复
3. fork 时怎样复制旧世界但避免 clobber

这说明 migration governance 有时并不是：

`rewrite old data`

而是：

`preserve continuity while changing ownership / path / session lineage`

这对 stronger-request cleanup 尤其重要。
未来若修 `plansDirectory` law，
问题绝不只是“新路径是什么”，
还包括：

`旧 plan artifact 怎样继续被 resume world 看见，以及何时停止继续被看见。`

## 9. 技术先进性与哲学本质：Claude Code 真正先进的不是“修复后立刻切断旧世界”，而是它在别处已经知道如何用幂等、范围限制、兼容 remap 与 grace window 管理过渡

从技术角度看，
Claude Code 在这条线上真正先进的地方，
不是它会写 migration 脚本。

更值钱的是它已经在多个子系统里承认：

1. 不是所有 source 都该被重写
2. 不是所有旧值都该立即失效
3. 有些过渡要靠 runtime remap
4. 有些过渡要靠 one-time notification
5. 有些过渡要靠 grace window 与 orphan cleanup
6. 有些过渡甚至要靠 continuity-preserving copy / reuse / recovery

这背后的哲学本质是：

`真正成熟的治理，不只会决定新规则是什么，还会决定旧世界怎样体面退出。`

这也带来三条直接技术启示：

1. stronger-request cleanup 下一步最值钱的补强，不只是 repair grammar，而是把 phased exit、grace window、source rewrite / runtime remap choice 变成正式控制面
2. plans family 是 stronger-request cleanup migration-governance 最好的实验场，因为它天然同时跨越 current-session active world、resume world 与 forked world
3. cleanup 线未来若补 stronger promise 修复，必须连同旧 promise 退场策略一起设计，否则新 law 会和旧世界一起并存成更大的谎言

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 repair governor，就已经足够成熟。`

而是：

`repo 在模型、配置键、插件缓存与 plan continuity 上已经明确展示了 migration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-repair-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

因此：

`cleanup 线真正缺的不是“修完”，而是“修完后旧世界怎样被体面带走”的最后一道治理设计。`
