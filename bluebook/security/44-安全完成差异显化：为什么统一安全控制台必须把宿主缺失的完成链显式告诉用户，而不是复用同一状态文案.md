# 安全完成差异显化：为什么统一安全控制台必须把宿主缺失的完成链显式告诉用户，而不是复用同一状态文案

## 1. 为什么在 `43` 之后还要继续写“完成差异显化”

`43-安全完成投影` 已经回答了：

`不同宿主只能看到完成签字链的子集，不能把局部完成冒充全链完成。`

但如果继续往产品层再压一层，  
还会出现一个更现实的问题：

`既然不同宿主看到的是不同子集，这件事该不该被明确展示给用户？`

也就是说，  
就算系统内部已经知道：

1. 这个宿主只有事件流视图
2. 那个宿主有恢复账和 flush 视角
3. 另一个宿主只拥有极窄 control 子集
4. 当前 bridge 只是 write-ready，不等于 read-ready

如果最终 UI 和状态面还是统一写成：

1. connected
2. active
3. completed
4. status ok

那控制面仍然会在产品层犯同一种旧错误：

`把不同强度、不同可见度、不同完成资格的宿主，再次压成同一句话。`

所以这一章的核心问题是：

`Claude Code 为什么不该继续复用同一套状态文案，而必须把宿主缺失的完成链显式展示给用户？`

## 2. 最短结论

从源码看，Claude Code 已经在很多底层结构里承认：

`不同宿主看到的是不同厚度的完成投影。`

但现有产品表面仍会在若干地方把这些差异压缩成摘要型文案。  
这说明下一代统一安全控制台最该补的一层，不是再加一个状态字段，  
而是：

`把宿主当前缺哪本账、当前看到的是哪一级完成信号、为什么不能说得更满，显式告诉用户。`

这至少体现在五类地方：

1. `/status` 会把 MCP 状态压成计数摘要
2. `sdkMessageAdapter` 会把 status 压成 generic informational 文本
3. `bridgeStatusUtil` 会把 bridge 状态压成 `active / reconnecting / failed / connecting`
4. `replBridgeTransport` 明确说 connected 只是 write-readiness，不是 read-readiness
5. 但底层另一方面又已经明确建模了恢复账、flush 视角、control 子集和 recovery 收缩

所以这一章的最短结论是：

`成熟安全控制面不只要在内部承认宿主差异，还要在产品层显式暴露这些差异；否则底层诚实会在展示层重新失真。`

## 3. 什么叫“安全完成差异显化”

这里说的“差异显化”，不是多加几个提示词。  
我在这一章里的定义是：

`系统把“当前宿主看到了什么、缺了什么、因此最多能把完成说到哪一步”明确展示给用户，而不是让这些限制默默藏在实现细节里。`

一个成熟的差异显化至少要回答四件事：

1. 当前宿主属于哪一类完成投影
2. 当前宿主缺哪条完成链
3. 当前界面的“完成”到底是事件级、复制级还是恢复级
4. 哪条更满的说法当前不应说

如果缺这四句，  
用户最容易把：

`当前界面说完成了`

误读成：

`系统内部整条完成链都已经闭合了`

## 4. 第一类压缩：`/status` 已经在把异质宿主状态压成摘要

## 4.1 `buildMcpProperties()` 主动把服务器状态压成计数摘要

`src/utils/status.tsx:89-115` 很关键。  
这里它并不展示每个 server 的详细 completion context，  
而是把非 IDE 的 MCP 状态汇总成：

1. `connected`
2. `need auth`
3. `pending`
4. `failed`

再统一拼成：

`MCP servers: ... · /mcp`

这当然很实用，  
但从安全完成差异的角度看，  
它也说明了一个问题：

`展示层已经开始主动做语义压缩。`

## 4.2 压缩本身没错，问题是压缩后如果不说明“还缺什么”，用户就会过度信任

计数摘要适合总览，  
但它不会告诉用户：

1. 这是哪类宿主的视图
2. 当前有没有恢复账
3. 当前 connected 到底是哪一级 completedness
4. 当前 pending 是系统自动恢复中，还是宿主自己其实缺完整完成链

也就是说，  
摘要不是罪过，  
真正的风险是：

`摘要被误读成完整说明。`

## 5. 第二类压缩：remote 适配层会把 richer status 压成 generic informational text

## 5.1 `sdkMessageAdapter` 直接把 status 转成 `Status: ...`

`src/remote/sdkMessageAdapter.ts:88-103` 非常有代表性。  
这里 `convertStatusMessage()` 在绝大多数情况下会输出：

`Status: ${msg.status}`

只有 `compacting` 被特别翻成：

`Compacting conversation…`

这说明 remote adapter 的一个真实职责就是：

`把结构化状态压成适合通用系统消息消费的文本。`

## 5.2 这是一种必要压缩，但也天然带来“完成语义损失”

因为一旦状态被压成 generic informational message，  
就更难继续表达：

1. 这是哪类宿主给出的状态
2. 这条状态背后缺失了哪些账本
3. 这条状态的 completion authority 到底有多强

所以这段源码其实在提醒我们：

`只要统一控制台未来仍允许结构化状态在适配层被压成泛文案，就必须同时补一层宿主能力和完成可见度的显式说明。`

## 6. 第三类压缩：bridge 状态标签已经明显比内部真相更薄

## 6.1 `bridgeStatusUtil` 只给四种总体标签

`src/bridge/bridgeStatusUtil.ts:124-140` 把 bridge 的总体状态压成：

1. `Remote Control failed`
2. `Remote Control reconnecting`
3. `Remote Control active`
4. `Remote Control connecting…`

这是典型的产品级状态归并。  
但它也意味着很多底层语义被主动收掉了。

## 6.2 `active` 尤其危险，因为它太像“已经完整可用”

如果用户只看到 `Remote Control active`，  
很容易自然地理解成：

`读写、状态、恢复、完成投影都已经完整。`

可源码别处并没有给出这么强的保证。  
这就说明：

`产品标签一旦不揭示自己的观察边界，就会天然鼓励过度解释。`

## 7. 第四类压缩：源码甚至已经明确提醒“connected”只代表写可用，不代表读可用

`src/bridge/replBridgeTransport.ts:289-299` 非常关键。  
注释直接写出：

1. `isConnectedStatus()` 是 write-readiness
2. 不是 read-readiness
3. `getStateLabel()` 只是从当前可观察事实合成出来
4. replBridge 也只把它用在 debug logging

这段证据非常强，  
因为它几乎把我们这章的论点直接说出来了：

`同一个“connected”标签，其实只是在某个观察面上成立。`

也就是说，  
作者已经明确知道：

`连接、写可用、读可用、完整完成投影，并不是同一件事。`

如果未来产品层仍然把这些东西复用成同一种“已连接/已激活”文案，  
那就等于把底层明明已经分开的语义，  
重新在展示层里合并回去了。

## 8. 第五类压缩：与这些摘要相对，底层其实已经有充分证据证明差异必须显式化

`src/cli/structuredIO.ts:139-196` 明确说默认宿主：

1. 没有 `restoredWorkerState`
2. 没有 internal-event flush
3. 没有 pending depth

`src/cli/remoteIO.ts:126-167` 和 `217-223` 又明确说明 RemoteIO 多了：

1. 恢复读回
2. internal event read/write
3. state / metadata report
4. flush 与 pending 可见性

`src/server/directConnectManager.ts:81-99` 与  
`src/remote/RemoteSessionManager.ts:192-213`  
又说明窄宿主只正式支持：

`can_use_tool`

而 `src/bridge/remoteBridgeCore.ts:824-878` 则说明更厚宿主还会在 recovery 窗口主动收缩。

这意味着源码底层已经明确告诉我们：

`宿主差异不是猜测，也不是 UI 风格问题，而是结构事实。`

既然如此，  
真正成熟的下一步就不该是继续隐藏这些差异，  
而应该是：

`把差异做成正式展示对象。`

## 9. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`如果宿主完成差异不被显式展示，底层已经被分开的完成语义最终仍会在产品层重新塌缩。`

这条公理几乎能统一前面所有例子：

1. `/status` 把异质状态压成摘要
2. remote adapter 把结构化 status 压成通用文案
3. bridge 用总体标签代替内部差异
4. connected 本身都只代表某一维 readiness
5. 但底层又早已明确建模恢复账、flush、control 子集和 recovery 收缩

所以在 `43` 之后，  
更深层的产品化结论已经很清楚：

`不是再发明更多状态，而是把当前宿主“看不到什么”正式告诉用户。`

## 10. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 真正先进的地方，不只在于底层把宿主差异设计对了，还在于这些差异已经具备被产品层显式化的原材料。`

这会带来几个很强的工程收益：

1. 用户不会把薄投影误读成厚投影
2. 支持团队更容易判断“这个界面为什么不能下更满结论”
3. 宿主接入方更难把 debug label、summary label、completion signer 混成同一层
4. 统一安全控制台可以把“当前宿主缺失账本”做成正式字段
5. 底层诚实语义不容易在适配层和摘要层被重新掩盖

这对其他 Agent 平台构建者的启示也很直接：

1. 设计状态面时，不只展示“当前状态”，还要展示“当前观察边界”
2. 不要让同一个 connected/active 标签跨不同宿主复用而不加限定
3. 把“缺恢复账”“缺 flush 视角”“仅有窄 control 子集”做成正式产品语义
4. 把 debug-only label 和 user-facing completion label 分开
5. 把宿主差异显式化当成安全设计，而不是 UX 附属优化

## 11. 哲学本质

这一章更深层的哲学是：

`诚实不仅是内部知道自己的边界，还包括把边界告诉观察者。`

很多系统在内部其实知道自己看不全，  
但对外仍然选择用统一、干净、好懂的状态词汇把差异抹平。  
Claude Code 的源码已经走到了更成熟的一半：

`它在底层承认差异。`

下一步真正该完成的，则是另一半：

`把这种差异公开地、结构化地呈现出来。`

从第一性原理看，  
这保护的是一条很根本的公理：

`观察边界如果不被显式揭示，就会被用户自动补全成“系统应该已经看全了”。`

## 12. 苏格拉底式反思：这一章最容易犯什么错

### 12.1 我们会不会把状态摘要的存在本身妖魔化

不会。  
问题不是摘要，  
而是摘要后不再说明“这个摘要来自哪一类宿主、缺什么视角”。

### 12.2 这会不会只是产品优化，不算安全问题

不是。  
因为一旦不同完成投影被统一文案抹平，  
用户就会在错误的 certainty level 上行动。

### 12.3 这一章之后还缺什么

还缺一层更落地的东西：

`统一安全控制台最少该用哪几个字段，把宿主完成差异显式展示出来。`

也就是说，  
下一步最自然的延伸就是把这章压成一张：

`宿主完成差异显化字段矩阵`

## 13. 结语

`43` 回答的是：不同宿主只能看到完成签字链的子集。  
这一章继续推进后的结论则是：

`这种差异不能只留在底层实现里，而必须被统一安全控制台显式告诉用户。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正成熟的控制面，不只在内部诚实地承认宿主盲区，还会把这种盲区公开成用户可见的产品语义。`
