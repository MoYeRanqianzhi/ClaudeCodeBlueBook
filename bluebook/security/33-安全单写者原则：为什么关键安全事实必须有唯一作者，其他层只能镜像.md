# 安全单写者原则：为什么关键安全事实必须有唯一作者，其他层只能镜像

## 1. 为什么在 `32` 之后还要继续写“单写者原则”

`32-安全裂脑防御` 已经把一个更深层的问题说清楚了：

`Claude Code 的很多安全工程细节，本质上都在防止同一安全事实分叉成多份互相打架的真相。`

但如果继续往第一性原理再压一层，  
还会剩下一个更简洁、也更强的判断：

`为什么系统总是在努力让关键安全事实只有一个正式写入口？`

换句话说，  
为什么看起来分散在不同文件里的很多实现，  
都反复在回到同一种结构：

1. 单一 choke point
2. 单一 writer
3. last value wins
4. dedupe
5. 外部只读镜像

这说明系统真正坚持的，  
其实不是零散的“同步技巧”，  
而是一条更底层的公理：

`关键安全事实必须有唯一作者。`

所以这一章的核心问题是：

`Claude Code 的安全控制面，为什么会自然收敛到“单写者原则”？`

## 2. 最短结论

从源码看，Claude Code 的很多高质量安全实现都可以压成一句话：

`关键安全事实只能有一个正式作者，其他所有层都应该是镜像、投影、复制或恢复，而不应成为并列作者。`

这条原则一旦被破坏，  
系统就会同时遭遇四类问题：

1. 状态分叉
2. 仲裁困难
3. 恢复歧义
4. 界面误导

所以 `33` 的核心判断是：

`Claude Code 的深层安全设计，不只是边界收口，也是在努力维持关键安全事实的单写者结构。`

## 3. 什么叫“安全单写者”

这里说的“单写者”，不是所有字段都只能在一个文件里赋值。  
真正的定义是：

`同一安全语义的最终决定权，只能有一个被正式承认的更新入口。`

例如：

- permission mode 的外部同步不该有多个散点作者
- session state 的语义变化不该由多个模块各自猜测后上报
- worker 真相不该允许多个上传器并列竞争
- outbound control/request 排序不该允许多个 writer 乱序插队

也就是说，  
这里的“单写者”关心的是：

`语义作者权。`

## 4. 源码里的第一类单写者：mode 同步作者权

## 4.1 `onChangeAppState.ts` 明确把 mode 同步收口到单一 choke point

`src/state/onChangeAppState.ts:50-92` 已经把这件事写得非常直白：

1. 过去有很多 mutation path 会改 mode
2. 其中不少路径不会告诉 CCR
3. 结果就是 `external_metadata.permission_mode` stale
4. web UI 和 CLI 真实 mode 失去一致性

这里最重要的不是“又补了一次同步”，  
而是作者明确选择：

`mode 这件事的外部真相作者只能有一个出口。`

这个出口就是：

`onChangeAppState` 的 diff 块。

一旦所有 mode 变化都必须从这里出站，  
外部元数据和 SDK status stream 才可能重新收敛。

## 4.2 这也说明“单写者”不是为了代码整洁，而是为了安全一致性

如果 mode 同步仍然分散在不同 UI、不同命令和不同 bridge handler 里，  
那系统最后看到的就不是一个事实，  
而是：

`谁碰巧记得同步、谁碰巧没忘。`

这就是最典型的多写者失控。

## 5. 源码里的第二类单写者：会话语义作者权

## 5.1 `notifySessionStateChanged(...)` 本质上在扮演会话语义唯一作者

`src/utils/sessionState.ts:92-149` 做了几件关键事：

1. 更新 `currentState`
2. 写 `pending_action`
3. 清 `pending_action`
4. 清 `task_summary`
5. 视情况镜像到 SDK event stream

这些动作看起来像“一组配套逻辑”，  
但从单写者原则看，它们其实都在说明：

`会话语义状态的作者权，应该集中在 notifySessionStateChanged。`

因为只有这样，  
语义状态、复制状态和事件状态三条链才能保持同向。

## 5.2 `notifyPermissionModeChanged(...)` 更是直接被作者写成了单写者声明

`src/utils/sessionState.ts:142-149` 直接说：

`both wired through this single choke point so no mode-mutation path can silently bypass them.`

这实际上已经不是普通注释，  
而是作者在代码里明确写下的系统法则：

`关键安全事实必须有单一出口，不能被静默绕过。`

## 6. 源码里的第三类单写者：outbound 顺序作者权

## 6.1 `StructuredIO` 直接说 drain loop 是 only writer

`src/cli/structuredIO.ts:160-162` 的注释非常关键：

`sendRequest() and print.ts both enqueue here; the drain loop is the only writer.`

这说明作者清楚地知道：

对控制消息和流式事件来说，  
“谁最后往 stdout / transport 写”本身就是一种语义作者权。

如果多个模块都直接写，  
就会产生：

1. `control_request` 超车
2. `stream_event` 乱序
3. 权限请求和普通事件交错失真

因此这里的 single writer 不是性能优化，  
而是：

`控制面时序的作者权收口。`

## 6.2 这也是为什么它与安全直接相关

一旦 control request 可以绕开唯一 writer，  
系统就会把“先发生什么、后发生什么”写乱。  
而权限、审批、状态切换恰恰极度依赖时序。

所以这里 again 不是普通架构洁癖，  
而是安全控制面在保护：

`时序真相的唯一作者。`

## 7. 源码里的第四类单写者：worker 真相作者权

## 7.1 `WorkerStateUploader` 明确采用 1 in-flight + 1 pending

`src/cli/transports/WorkerStateUploader.ts:4-17`、`39-75` 说明：

1. 只有 1 个 in-flight PUT
2. 只有 1 个 pending patch
3. top-level keys last value wins
4. retry 时吸收 pending

这本质上是在防止：

`多个作者并发地把 worker 真相写成互相竞争的版本。`

这里最值得注意的，不是它会重试，  
而是它明确拒绝“无限并发 patch 乱飞”的写法。

也就是说，  
它在维护：

`worker 复制真相的单写者结构。`

## 7.2 last value wins 也是一种单写者哲学

很多人会把 “last value wins” 当成简单覆盖规则。  
但在这里，它真正表达的是：

`顶层真相不能并列存在多个当前版本。`

也就是说，  
系统宁愿丢掉中间过时版本，  
也不允许多个“当前 worker_status”并列竞争。

## 8. 源码里的第五类单写者：功能开关与资格开关作者权

## 8.1 `AppStateStore.ts` 对 `kairosEnabled` 直接写成 Single source of truth

`src/state/AppStateStore.ts:113-116` 这里非常有代表性：

`Single source of truth - computed once in main.tsx before option mutation`

虽然这是 assistant mode 相关状态，  
但它体现的是同一种作者权哲学：

`功能或资格相关的关键布尔量，不能由多个消费面重复推导。`

否则你迟早会遇到：

1. 一个面认为开了
2. 另一个面认为没开
3. 最后没有人能解释到底谁是权威

所以单写者原则并不限于安全告警和 permission，  
它同样适用于：

`关键能力资格状态。`

## 9. 源码里的第六类单写者：恢复窗口作者权

## 9.1 `remoteBridgeCore` 通过 `authRecoveryInFlight` 明确禁止多恢复作者

`src/bridge/remoteBridgeCore.ts:330-372` 和 `477-526` 已经说明：

1. proactive refresh 与 SSE 401 可能同时触发恢复
2. 双 `/bridge` fetch 会导致双 epoch bump
3. 旧 transport 继续写会造成 silent loss
4. 所以恢复路径必须先 claim flag，再串行进入

这其实是单写者原则在恢复面上的直接体现：

`同一时刻只能有一个恢复作者。`

否则系统就会在“修复状态”的动作本身上再次分叉。

## 10. 把这些例子压成一条底层公理

我会把它压成一句话：

`关键安全事实必须只有一个正式作者；其他所有层要么镜像它、要么恢复它、要么投影它，但不能与它并列争夺作者权。`

这条公理几乎可以解释前面多章里出现的大部分高质量实现。

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性，不只在于它做了很多同步，而在于它常常先决定“谁有资格写”，再决定“其他层如何看”。`

这对其他 Agent 平台构建者的启示很直接：

1. 先决定单写者是谁
2. 再决定镜像链怎么走
3. 最后才决定 UI 怎么展示

如果顺序反过来，  
系统通常会走向：

1. 多点写入
2. 事后对账
3. 最终解释越来越复杂

## 12. 哲学本质

这一章更深层的哲学是：

`解释不是第一道防线，作者权才是。`

很多系统喜欢把安全问题理解成“状态多了以后要做更好的解释”。  
Claude Code 的源码反过来说明：

`更高阶的做法是先减少真相作者的数量。`

因为一旦作者权混乱，  
再好的解释也只是对裂脑结果做事后修辞。

## 13. 苏格拉底式反思：这章最容易犯什么错

### 13.1 单写者是不是意味着集中式脆弱

有风险。  
但对关键安全事实来说，  
多作者竞争通常更危险。

### 13.2 所有字段都值得做单写者吗

不一定。  
只有那些一旦分叉就会造成权限、状态、恢复或资格误判的事实，  
才值得如此严格对待。

### 13.3 我们是不是把“镜像”低估了

也不是。  
镜像很重要。  
但镜像的前提是：

`它先承认自己不是作者。`

## 14. 结语

`32` 回答的是：系统为什么不断在防裂脑。  
`33` 继续推进后的结论则是：

`防裂脑之所以成立，是因为关键安全事实被尽量维持成单写者结构。`

这意味着 Claude Code 真正深层的安全启示之一是：

`让关键安全事实只有一个正式作者，比在多位作者打完架之后做更漂亮的解释更重要。`
