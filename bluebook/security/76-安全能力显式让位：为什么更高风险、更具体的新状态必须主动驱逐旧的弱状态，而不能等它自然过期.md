# 安全能力显式让位：为什么更高风险、更具体的新状态必须主动驱逐旧的弱状态，而不能等它自然过期

## 1. 为什么在 `75` 之后还要继续写“显式让位”

`75-安全能力声明仲裁` 已经回答了：

`notification 赢的是注意力，/status 赢的是解释权，footer 只配保留连续性弱词。`

但如果继续往下追问，  
还会碰到一个更工程化、也更决定用户体感的问题：

`当一个更高风险、更具体的新状态到来时，系统到底是让旧状态自己慢慢超时消失，还是必须主动把它赶走？`

这不是交互细节。  
因为只要旧的弱状态继续挂着，  
哪怕只多挂几秒，  
用户看到的也不是“多一个提示”，  
而是：

`两个互不兼容的世界观同时存在。`

比如：

1. footer 还在说 `active`
2. notification 已经说 `failed`
3. generic disconnected 还挂着
4. 更具体的 install failed 已经成立

这时候如果系统只是等旧提示自然 timeout，  
那就不是温和过渡，  
而是：

`语义裂脑。`

所以在 `75` 之后，  
安全专题必须再补一条更硬的纪律：

`显式让位。`

也就是：

`更高风险、更具体、更新鲜的新状态一旦成立，系统必须主动清除、撤回或失效旧的弱状态；不能把旧状态留给时间自己处理。`

## 2. 最短结论

从源码看，Claude Code 已经把这条原则做成了显式机制，而不是靠文案约定：

1. notification 基础协议直接定义了 `invalidates?: string[]`，语义就是“把被失效的通知从队列移除，如果当前正在展示则立刻清掉”  
   `src/context/notifications.tsx:6-14`
2. `immediate` 通知到来时，不只会抢占当前位，还会把 `invalidates` 指定的旧通知从当前位与队列里一起清掉  
   `src/context/notifications.tsx:78-117`
3. 非 `immediate` 通知同样支持失效当前通知：如果 `invalidatesCurrent` 成立，就清 timeout 并把 current 置空  
   `src/context/notifications.tsx:172-191`
4. IDE 状态提示不是并列累积，而是按状态条件主动 `removeNotification(...)`：generic disconnected 一旦不再最合适，就立即撤掉  
   `src/hooks/notifs/useIDEStatusIndicator.tsx:78-152`
5. bridge failed 走 `priority: 'immediate'` 抢位，同时 footer 明确拒绝承担 failed 的残留显示  
   `src/hooks/useReplBridge.tsx:102-111`  
   `src/components/PromptInput/PromptInputFooter.tsx:173-189`
6. `invalidates` 不是一次性黑魔法，而是通用状态替换原语；fast mode 的 cooldown started / expired 就是对称驱逐关系  
   `src/hooks/notifs/useFastModeNotification.tsx:101-121`

所以这一章的最短结论是：

`Claude Code 的安全状态面不是“谁更重要谁排前面”这么简单，而是“新的强状态必须显式驱逐旧的弱状态”。`

我把它再压成一句：

`安全控制面宁可主动改口，也不能让旧真相继续赖在屏幕上。`

## 3. 源码已经说明：显式让位是协议能力，不是界面补丁

## 3.1 `invalidates` 被写进基础通知协议，说明“失效旧状态”是一级公民

`src/context/notifications.tsx:6-14` 非常关键。

这里 `BaseNotification` 不是只定义：

1. `key`
2. `priority`
3. `timeoutMs`

还专门定义了：

`invalidates?: string[]`

注释写得非常清楚：

1. 被 invalidated 的通知会从 queue 移除
2. 如果它当前正在展示，也会立刻清掉

这说明作者理解的通知系统，不是单纯 toast 队列，  
而是一套最小状态机。

也就是说，  
在 Claude Code 里，  
通知不是：

`只能新增，不能正式撤销`

而是：

`新增、折叠、失效、移除都被视作正式语义动作。`

这正是“显式让位”能成立的前提。

## 3.2 `immediate` 并不只是插队，它还是强制清场机制

`src/context/notifications.tsx:78-117` 进一步证明：

当 `priority === 'immediate'` 时，  
系统做的不是“先显示我，稍后再说”，  
而是：

1. 清掉当前 timeout
2. 直接把新通知放进 `current`
3. 重排旧的 `current + queue`
4. 过滤掉所有被 `invalidates` 命中的旧项

这里最重要的不是“快”，  
而是：

`带着清场能力的快。`

如果没有第 4 步，  
那 immediate 只是在视觉上压住旧状态；  
一旦新通知超时，旧状态还会弹回来。

Claude Code 显然不接受这种“旧真相复活”。  
所以它把：

`抢位`

和

`驱逐旧项`

做成了同一动作。

## 3.3 非 immediate 也能驱逐当前状态，说明显式让位不是紧急态专属特权

`src/context/notifications.tsx:172-191` 更值得注意。

很多系统即便支持高优先级抢位，  
也只会在 emergency 情况下替换当前项。  
但这里的实现更成熟：

1. 即便不是 `immediate`
2. 只要 `invalidatesCurrent` 成立
3. 系统也会先清掉当前 timeout
4. 再把 `current` 置空

这说明 Claude Code 对“显式让位”的理解不是：

`只有最紧急的错误才配改口`

而是：

`只要新的语义与旧的语义互斥，就应当正式让位。`

也就是说，  
显式让位保护的不是高优先级本身，  
而是：

`语义不共存。`

## 3.4 IDE 状态提示已经在手工执行这条纪律：generic 症状必须给更具体原因让位

`src/hooks/notifs/useIDEStatusIndicator.tsx:78-152` 是这一章最直接的业务证据。

这里 generic 的：

`{ideName} disconnected`

并不会顽固保留。  
只要下面任一条件成立：

1. `showIDEInstallError`
2. `showJetBrainsInfo`
3. `ideStatus !== "disconnected"`
4. `!ideName`

系统就立刻：

`removeNotification("ide-status-disconnected")`

与此同时，  
更具体的通知又被独立挂起：

1. `IDE plugin not connected · /status for info`
2. `IDE extension install failed (see /status for info)`

这说明作者并没有把 generic disconnected 当成“总保底提示”。  
相反，  
它被当成一个有严格适用域的临时占位状态。

一旦更具体原因成立，  
它就必须退场。

因此这里体现的不是简单优先级，  
而是一条更底层的哲学：

`原因态必须驱逐症状态。`

## 3.5 bridge 再次说明：高风险否定词不能只盖过去，还要阻止旧的弱绿色词继续占位

`src/hooks/useReplBridge.tsx:102-111` 证明：

bridge 失败时，  
系统会发出：

`key: 'bridge-failed'`

并且是：

`priority: 'immediate'`

而 `src/components/PromptInput/PromptInputFooter.tsx:173-189` 又明确写明：

`Failed state is surfaced via notification (useReplBridge), not a footer pill.`

这两段源码连起来，  
形成了非常清楚的设计意图：

1. 失败必须抢到前景
2. footer 不得继续扮演 failed 的保留层
3. 旧的低占位连续反馈要主动让路

这说明在 bridge 场景里，  
作者最担心的不是“失败不够醒目”，  
而是：

`旧的绿色或黄色连续性词还留着，用户会以为系统只是抖了一下。`

所以这里的显式让位，本质上是在防：

`危险的假连续性。`

## 3.6 fast mode 的成对 `invalidates` 证明：这是一种可复用的状态替换原语

`src/hooks/notifs/useFastModeNotification.tsx:101-121` 虽然不直接属于能力安全本体，  
但它非常能说明团队的抽象水平。

这里：

1. cooldown started 会 `invalidates: [COOLDOWN_EXPIRED_KEY]`
2. cooldown expired 会 `invalidates: [COOLDOWN_STARTED_KEY]`

这是非常干净的双向驱逐关系。

它说明作者没有把“显式让位”写死在某一个安全 hook 里，  
而是已经把它抽象成：

`互斥状态之间应显式地彼此废止。`

这对我们理解 Claude Code 的安全设计很重要。

因为这意味着：

`安全状态面的成熟，不是某个页面偶然写对了，而是底层已经存在一套可复用的状态替换语法。`

## 3.7 第一性原理：显式让位真正保护的，是“当前行动方向”而不是界面整洁

如果从第一性原理追问：

`为什么旧提示多留几秒会这么严重？`

因为用户不是在读数据库，  
而是在根据当前表面选择下一步动作。

一旦旧弱状态没被驱逐，  
用户就可能在两个动作之间摇摆：

1. 继续等待
2. 重新连接
3. 去 `/status`
4. 去 `/mcp`
5. 误以为问题已恢复

所以显式让位保护的不是审美，  
而是：

`动作路由的一致性。`

可以把它压成一句：

`成熟控制面最怕的不是一瞬间变红，而是红了之后屏幕上还残留着旧绿词。`

## 4. 显式让位的最短规则

把这一章压成最短规则，就是下面五句：

1. 新状态如果与旧状态互斥，就必须显式失效旧状态
2. 更具体的原因态必须驱逐更泛化的症状态
3. 更高风险的否定词必须驱逐旧的连续性弱词
4. 抢位如果不伴随驱逐，旧状态就会在稍后复活
5. timeout 只适合处理“已经过时但不冲突”的提示，不适合处理互斥真相

## 5. 苏格拉底式追问：这套设计还能怎样再向前推进

如果继续自问，  
这套设计仍然还可以再前进一步：

1. 当前 `invalidates` 已经很强，但大多还是 key 级手工维护；若下一代控制台继续产品化，最好把互斥关系提升为表驱动声明
2. 现在很多显式让位仍分散在具体 hook 中；若未来 surface 更多，最好把“症状态 vs 原因态”“连续性词 vs 高风险词”做成统一 eviction policy
3. `/status`、`/mcp` 与 notification 已有 handoff，但旧弱词被驱逐后还缺更统一的“你现在该去哪里”的结构化字段

所以这一章最终沉淀出的技术启示是：

`安全状态系统真正高级的地方，不是能显示很多状态，而是知道哪些旧状态必须被正式宣布作废。`
