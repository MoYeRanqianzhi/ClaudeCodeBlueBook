# 安全能力闭包绑定：为什么句柄真正承载的不是方法集合，而是创建时上下文

## 1. 为什么在 `80` 之后还要继续写“闭包绑定”

`80-安全状态句柄化` 已经回答了：

`名字只能指向状态，句柄才能承载状态主权。`

但如果继续追问，  
还会碰到一个更深入、也更接近句柄本质的问题：

`句柄为什么比名字更安全？`

如果只是把一堆方法塞进对象里，  
那还不足以解释它的安全价值。  
真正关键的是：

`这些方法到底绑定了什么上下文。`

所以在 `80` 之后，  
安全专题必须再补一条更底层的原则：

`闭包绑定。`

也就是：

`真正有安全价值的 handle，不只是“有方法的对象”，而是“把创建时的 session、transport、token 语义、生命周期与回调账本一起封进闭包的能力对象”。`

## 2. 最短结论

从源码看，Claude Code 在 bridge 子系统里已经把这件事写得非常清楚：

1. `ReplBridgeHandle` 提供的不是裸 id，而是一组直接依赖创建时上下文的方法  
   `src/bridge/replBridge.ts:70-80`
2. 这些方法内部都直接使用闭包里的 `currentSessionId`、`transport` 与 `doTeardownImpl`，而不是让调用方重新传入  
   `src/bridge/replBridge.ts:1779-1834`
3. `replBridgeHandle.ts` 的注释明确说：handle 的闭包捕获了创建 session 的 `sessionId` 和 `getAccessToken`，独立重建这些值会带来 token divergence 风险  
   `src/bridge/replBridgeHandle.ts:5-13`
4. `useReplBridge.tsx` 里导出的 `permissionCallbacks` 也不是裸函数集合，而是围绕 `handle_0` 和 `pendingPermissionHandlers` 构造出的局部能力面  
   `src/hooks/useReplBridge.tsx:369-380,538-586`
5. `BridgeDebugHandle` 同样是受限方法集，而不是把整个 bridge 内部状态敞开给外部  
   `src/bridge/bridgeDebug.ts:38-52`

所以这一章的最短结论是：

`Claude Code 句柄化真正保护的，不只是方法权限，而是上下文重建权。`

我把它再压成一句：

`handle 的安全价值，在于后来者拿到的是能力，不是配方。`

## 3. 源码已经说明：真正的 handle 总是在避免“后来者自己重建上下文”

## 3.1 `ReplBridgeHandle` 的方法根本不是通用 RPC 包装，而是 creator-bound capability

`src/bridge/replBridge.ts:70-80` 里的 `ReplBridgeHandle` 很直白：

1. `writeMessages`
2. `writeSdkMessages`
3. `sendControlRequest`
4. `sendControlResponse`
5. `sendControlCancelRequest`
6. `sendResult`
7. `teardown`

如果这只是普通对象封装，  
外部理论上也可以自己拿：

1. session id
2. transport
3. token

再拼出同样的能力。

但后面的实现明确说明，  
作者并不希望调用者这么做。

## 3.2 真正的关键在实现：方法内部直接吃闭包里的 `currentSessionId` 和 `transport`

`src/bridge/replBridge.ts:1779-1834` 是本章最重要的实现证据。

这里：

1. `sendControlRequest` 直接把 `session_id: currentSessionId` 写进事件
2. `sendControlResponse` 同样把 `currentSessionId` 编入响应
3. `sendControlCancelRequest` 也复用同一个 `currentSessionId`
4. `sendResult` 直接调用 `makeResultMessage(currentSessionId)`
5. `teardown()` 直接走当前实例的 `unregister()` 与 `doTeardownImpl`

调用方并没有机会传入：

1. “我想用哪个 session id”
2. “我想用哪个 transport”
3. “我想 teardown 哪个实例”

这说明 handle 真正做的是：

`把调用能力绑死在创建该 handle 的上下文上。`

这就是 capability object 与 helper function 的根本区别。

## 3.3 `replBridgeHandle.ts` 的注释几乎把“为什么不能重建”写成了设计宣言

`src/bridge/replBridgeHandle.ts:5-13` 里最关键的一句是：

handle 的闭包捕获了创建 session 的：

1. `sessionId`
2. `getAccessToken`

并且注释直接说：

`re-deriving those independently ... risks staging/prod token divergence`

这非常有力量。  
因为它把 handle 的价值从“更方便调用”提升成了：

`防止后来者在错误上下文里自以为拿到了同样的能力。`

也就是说，  
句柄真正封装的不是 API 形式，  
而是：

`上下文正确性。`

## 3.4 `permissionCallbacks` 进一步证明：句柄还可以把“关系型状态”一起绑进来

`src/hooks/useReplBridge.tsx:369-380,538-586` 说明了另一层更微妙的价值。

这里作者不是把 bridge permission surface 暴露成：

1. 一个全局 request sender
2. 一个全局 response listener

而是先在 effect 局部创建：

1. `pendingPermissionHandlers` map
2. 基于 `handle_0` 的 `sendRequest`
3. 基于 `handle_0` 的 `sendResponse`
4. 基于 `handle_0` 的 `cancelRequest`
5. 基于局部 map 的 `onResponse`

这说明句柄并不只绑定静态上下文，  
它还可以绑定：

`当前这轮关系账本`

即：

1. 谁发出的 request
2. 谁还在等 response
3. 谁能取消
4. 谁有权把 handler 从 map 里删掉

换句话说，  
handle 真正厉害的地方不是“封装方法”，  
而是：

`把对象、上下文和关系账本一起封装。`

## 3.5 `BridgeDebugHandle` 说明：就算是 debug 面，也还是受限句柄而不是全量内部状态导出

`src/bridge/bridgeDebug.ts:38-52` 的 `BridgeDebugHandle` 也很能说明问题。

即便是 debug 能力，  
作者也没有选择暴露：

1. 整个 bridge 内部状态结构
2. 任意 transport 读写能力
3. 任意 session 改写入口

而是只给出受限方法：

1. `fireClose`
2. `forceReconnect`
3. `injectFault`
4. `wakePollLoop`
5. `describe`

这说明 Claude Code 团队很明确知道：

`哪怕是内部 debug 能力，也应该先做成受限 capability surface，再决定是否暴露。`

这和 notifications 当前的裸 key 模型形成了非常鲜明的对照。

## 3.6 第一性原理：闭包绑定真正回答的是“谁能在正确上下文里行动”

如果从第一性原理追问：

`为什么句柄一定要绑住创建时上下文？`

因为真正危险的从来不是：

`别人不知道对象名字`

而是：

`别人知道名字后，在错误上下文里成功执行了看似正确的动作。`

所以一个真正安全的 capability object 必须回答三件事：

1. 你能做什么
2. 你是对谁做
3. 你是在什么上下文里做

前两件事靠方法名和对象身份还能凑合表达，  
第三件事如果不靠闭包绑定，  
就只能让调用方自己重建。

而一旦让调用方自己重建，  
主权边界就开始松动。

## 3.7 技术先进性与启示：Claude Code 已经在高风险控制面里证明“闭包优于重推导”

从技术角度看，  
Claude Code 在 bridge 里的先进性，不只是用了 handle。

更关键的是，  
它已经明确站在了这样一边：

`creator-bound closure > caller-side reconstruction`

这条哲学非常值得迁移到统一安全控制台。

因为 notifications / state 面如果未来也要承载：

1. family scope
2. owner
3. allowed operators
4. relation metadata

那么最稳妥的做法同样不是让调用方拿名字去重新推理这些属性，  
而是：

`直接把这些能力绑进 scoped handle。`

## 4. 安全能力闭包绑定的最短规则

把这一章压成最短规则，就是下面六句：

1. handle 不只是对象壳，必须绑定创建时上下文
2. session、transport、token 语义不该交给后来者重建
3. 关系型账本也应尽量与 handle 一起封装
4. debug surface 也应优先暴露受限方法，而不是全量内部状态
5. 一旦上下文重建成为调用方责任，能力边界就开始变脆
6. 句柄真正成熟的标志，是“拿到它就拿到正确上下文”，而不是“拿到它再去猜上下文”

## 5. 苏格拉底式追问：这套系统还能怎样再向前推进

如果继续自问，  
下一步最值得做的三项升级会是：

1. 把 notifications future handle 设计成 creator-bound scoped object，而不是只加更多 metadata 字段
2. 把 family relation、owner 与 allowed operators 都封入 handle，而不是让调用方二次查询
3. 把“禁止重建上下文”正式写成统一安全控制台的设计公理，而不只是一段 bridge 注释

所以这一章最终沉淀出的设计启示是：

`真正成熟的 capability system，最后保护的不是名字，而是上下文。`
