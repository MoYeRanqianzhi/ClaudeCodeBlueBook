# 安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层：为什么artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer

## 1. 为什么在 `416` 之后还必须继续写 `417`

`416-安全载体家族强请求清理制度元数据治理与强请求清理运行时符合性治理分层`
已经回答了：

`即便 stronger-request cleanup metadata 已经成立，runtime reality 也仍会因为 declaration / admission / execution / abstention / coverage / receipt 分裂，而不自动符合。`

但如果继续往下追问，
还会碰到下一层更危险、也更容易被误判成“已经解决”的错觉：

`只要这一次 runtime 看起来已经 conform，系统就已经拥有长期 anti-drift verification。`

Claude Code 当前源码并不能支持这种更强说法。

继续看 repo 自己最成熟的 verifier 正对照，
会发现它其实已经展示出至少三种不同的反漂移技术：

1. `src/services/compact/microCompact.ts:32-36`
   直接把 drift 绑定到 source-of-truth equality test
2. `src/bootstrap/state.ts:456-479`
   直接把 `sessionId` 与 `sessionProjectDir` 做成 atomic anti-drift structure
3. `src/utils/permissions/getNextPermissionMode.ts:11-17`
   与
   `src/utils/permissions/permissionSetup.ts:1078-1157`
   直接把 stale cache vs live truth divergence 提升成 fresh read + current-context transform 的 runtime re-verification

与这些正对照相比，
stronger-request cleanup 当前可见的仍主要是：

1. `shouldSkipPersistence()` 的 suppress guard
   `src/utils/sessionStorage.ts:953-969`
2. `REPL.tsx` / `main.tsx` / `backgroundHousekeeping.ts` 的 first-submit admission、headless deferred admission 与 10 分钟 delay / recent-activity reschedule
   `src/screens/REPL.tsx:3903-3906`
   `src/main.tsx:2811-2818`
   `src/utils/backgroundHousekeeping.ts:28-80`
3. `cleanupOldMessageFilesInBackground()` 的 validation-driven safe skip 与 hand-written dispatch
   `src/utils/cleanup.ts:575-598`
4. `plansDirectory` 的 propagation drift 样本
   `src/utils/plans.ts:79-110,164-229`
   `src/utils/cleanup.ts:300-303`
5. `CleanupResult` 的 local countability
   `src/utils/cleanup.ts:33-45`
6. diagnostics 的 uncovered coverage gap
   `src/utils/diagLogs.ts:14-60`

这些都很重要，
但它们回答的仍主要是：

`这一次 runtime 做了什么、没做什么、为什么没做。`

它们还没有回答：

`如果明天 metadata、scheduler、executor、coverage set 与 receipt surface 再次分叉，谁会第一时间把这种分叉抓出来，并阻止系统继续把局部符合冒充成长期可信。`

所以 `416` 之后必须继续补的一层就是：

`安全载体家族强请求清理运行时符合性治理与强请求清理反漂移验证治理分层`

也就是：

`stronger-request cleanup-runtime-conformance governor 最多只能说“当前看起来已兑现”；stronger-request cleanup-anti-drift verifier 才能说“如果以后再次漂移，系统会主动发现”。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer。`

这里仍是研究命名。
它不是在声称 repo 里已经有一个现成的 `verifyCleanupPolicyAlignment()` 却没人接线，
而是在说：

1. repo 其他区域已经明确展示出成熟 verifier 文化
2. stronger-request cleanup 线当前还没看到同等级 verifier plane
3. 因而“这次 conform”与“将来再漂移时谁报警”已经是两个独立问题

第二条：

这里说的 `anti-drift verification`
不是在声称：

`当前 stronger-request cleanup 一定已经发生严重回归。`

更稳的说法只能是：

`当前源码已经足以证明 stronger-request cleanup 存在 temporal drift、propagation drift、coverage drift 与 receipt drift 的结构机会，但还没有看到与这些 drift class 对称的显式 verifier grammar。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部漂移问题。

真正关键的是：

1. `cleanupPeriodDays=0` 暴露 declaration truth 与 execution truth 可分裂
2. housekeeping admission 暴露 timing truth 可分裂
3. `plansDirectory` 暴露 cross-plane propagation truth 可分裂
4. diagnostics 暴露 coverage truth 可分裂
5. `CleanupResult` 暴露 outcome truth 与 receipt truth 可分裂

第四条：

这里尤其不能过度声称：

`skip guard` 已经等于 anti-drift verifier。`

更稳的说法只能是：

`skip guard` 回答的是 safe abstention，不是 drift detection。`

第五条：

这里也不能过度声称：

`anti-drift verifier` 只能长成测试。`

更稳的说法只能是：

`Claude Code 已经展示出至少三种 verifier 形态：source-of-truth test、atomic anti-drift structure、live re-verification against divergence。`

第六条：

这里还不能把：

`anti-drift verification`

偷换成：

`repair`

因为 verifier 的任务是发现漂移，
repair 的任务才是决定如何改正漂移。

所以 anti-drift verification 虽然比 runtime-conformance 更强，
但仍然不是最后一层。

## 3. 最短结论

Claude Code 当前源码至少给出了九类

`stronger-request cleanup-runtime-conformance-governor signer 仍不等于 stronger-request cleanup-anti-drift-verifier signer`

证据：

1. repo 已明确展示 source-of-truth verifier 正例
   `microCompact.ts:32-36`
2. repo 已明确展示 atomic anti-drift structure 正例
   `state.ts:456-479`
3. repo 已明确展示 stale-vs-live runtime re-verification 正例
   `getNextPermissionMode.ts:11-17`; `permissionSetup.ts:1078-1157`
4. repo 已明确展示“truth exposure”与“governance consequence”是拆层的
   `bypassPermissionsKillswitch.ts:74-116`
5. stronger-request cleanup 当前主要仍是 suppress / delay / skip / dispatch cluster，而没有对等 verifier surface
   `sessionStorage.ts:953-969`; `backgroundHousekeeping.ts:28-80`; `cleanup.ts:575-598`
6. interactive 与 headless world 的 cleanup admission 都仍是 deferred，而不是 verifier-led continuous re-check
   `REPL.tsx:3903-3906`; `main.tsx:2811-2818`
7. plans family 仍持续暴露 strongest propagation drift
   `plans.ts:79-110,164-229`; `cleanup.ts:300-303`
8. diagnostics family 仍持续暴露 uncovered coverage drift
   `diagLogs.ts:14-60`; `cleanup.ts:587-595`
9. `CleanupResult` 说明 outcome 已可计数，但仍未升格为 machine-comparable family receipt
   `cleanup.ts:33-45`

因此这一章的最短结论是：

`stronger-request cleanup-runtime-conformance-governor signer 最多能说“这次运行看起来已经兑现”；stronger-request cleanup-anti-drift-verifier signer 才能说“如果将来再偏离，系统会主动发现并拒绝继续沉默”。`

再压成一句：

`一次符合，不等于长期可信。`

再压成更硬的一句：

`制度这次说真话，不等于制度以后不会再说错话。`

## 4. 第一性原理：runtime-conformance 回答“此刻是否成立”，anti-drift verification 回答“偏离时谁会报警”

从第一性原理看，
这两层治理处理的是两种完全不同的安全问题。

`runtime-conformance`

处理的是：

1. 这次 stronger-request cleanup 当前是否执行
2. 没执行时是否被明确 skip
3. skip 是 deliberate safety 还是 accidental miss
4. executor 当前是否消费了正确 truth
5. 当前 outcome 有没有被保存

`anti-drift verification`

处理的则是：

1. 如果明天 helper 改了而 executor 没跟上，谁会失败得足够响
2. 如果 metadata 被某个 plane 消费、另一个 plane 忘记跟进，谁会主动报警
3. 如果 runtime 因为异步、缓存或 mode 切换而持有 stale truth，谁会 fresh read 并修正
4. 如果系统因为 safe abstention 而没做事，谁来区分“故意不做”与“悄悄漂移”
5. 如果 family outcome 迟早要被长期信任，谁来把它投影成 machine-comparable receipt

所以这两层的哲学本质不同：

`runtime-conformance` 是对当前事实的回答；
`anti-drift verification` 是对未来谎言的预防。

其最短表达是：

`当前的真，不等于将来的真；将来的真，需要 verifier 守住。`

## 5. Claude Code 多重安全技术的先进性，不在“多做检查”，而在“不同失真用不同证明方式”

很多系统的安全控制只会重复一种动作：

`更多判断、更多 guard、更多 deny。`

Claude Code 更成熟的地方在于，
它已经在别处明确展示出：

1. 当风险是 derived truth 偏离 source truth 时，用 source-of-truth test
2. 当风险是几个字段绝不能脱钩时，用 atomic structure
3. 当风险是 stale cache 与 live truth 会动态分叉时，用 live re-verification
4. 当 verifier 输出要作用到当前世界时，再把 consequence application 单独拆层

这说明 Claude Code 的多重安全技术先进性，
不是“安全动作很多”，
而是：

`它会按 drift class 选择不同 proof form。`

所以对 cleanup 子链最硬的技术判断不是：

`这里 guard 还不够多。`

而是：

`这里还没有把 drift class 与对应 proof form 正式绑定。`

## 6. 为什么 stronger-request cleanup 现在仍停在 conformance cluster，而不是 verifier plane

把当前 cleanup 子链压缩起来，
它已经拥有的是：

1. suppress guard
2. delayed admission
3. validation-driven abstention
4. partial executor
5. local outcome count
6. uncovered carrier evidence

这些都是真实的安全设计，
而且已经很有层次。

但它们的共同边界也同样清楚：

`它们在表达当前 runtime 做了什么，却还没有表达未来若再漂移谁会主动报警。`

这正是为什么：

1. `shouldSkipPersistence()` 不是 verifier
2. `backgroundHousekeeping` 的 delay / reschedule 不是 verifier
3. `CleanupResult` 不是 verifier
4. `plansDirectory` gap 只是 drift symptom，不是 drift detector

它们都在告诉你：

`这里可能偏`

却还没有谁在说：

`一旦偏了，我会负责抓出来。`

## 7. 苏格拉底式回问：我是不是又把“当前看起来没出事”误认成了“系统已经会抓漂移”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果这次 stronger-request cleanup 看起来已经 conform，为什么我还说 verifier 不存在？
   因为一次 conforming run 只能回答现在，不足以回答以后再分叉时谁会报警。
2. 如果 `shouldSkipPersistence()` 已经在保护系统，为什么这还不算 anti-drift？
   因为 guard 只限制动作，不对 drift class 做主动验证。
3. 如果 `backgroundHousekeeping` 已经有 reschedule 逻辑，为什么这还不算 verifier？
   因为 reschedule 是调度策略，不是 source-truth comparison 或 stale-vs-live re-verification。
4. 如果 `CleanupResult` 已经在计数，为什么这还不算 verifier surface？
   因为 local countability 不等于 machine-comparable family receipt。
5. 如果 `REPL` 与 headless 入口都已经会启动 cleanup，为什么 admission 还不算 anti-drift？
   因为 admission 说明“会开始做”，不说明“以后每次偏离都会被重新检查并报警”。
6. 如果 repo 别处已经有 verifier 文化，cleanup 线是不是以后自然也会得到保护？
   不能这样推。verifier culture 的存在不自动等于 verifier plane 已经接线。

这组反问最终逼出一句更稳的判断：

`anti-drift verification 的关键，不在系统有没有 guard，而在系统能不能把“再次偏离”转化成不会继续沉默的事件。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 目前没有 verifier，只是因为 Claude Code 安全体系还不成熟。`

而是：

`repo 安全体系恰恰已经很成熟，成熟到它在别处明确展示了三种 anti-drift grammar；也正因为如此，stronger-request cleanup 当前仍缺对等 verifier plane 这件事，才不是小缺口，而是一个被成熟对照系明确照亮的结构缺口。`

因此：

`artifact-family cleanup stronger-request cleanup-runtime-conformance-governor signer`
仍不能越级冒充
`artifact-family cleanup stronger-request cleanup-anti-drift-verifier signer`。

最后把这一章压成一句：

`制度这次说真话，不等于制度以后不会再说错话。`
