# 安全完成差异卡片：统一安全控制台应把宿主盲区字段挂到哪张卡、按什么优先级显示

## 1. 为什么在 `45` 之后还要继续写“完成差异卡片”

`45-安全完成差异字段` 已经回答了：

`统一安全控制台最少该用哪些字段显式展示宿主盲区。`

但如果继续往产品落地层再压一层，  
还会立刻冒出另一个更具体的问题：

`这些字段应该挂到哪张卡、以什么优先级出现，才不会再次被界面压平？`

因为控制台设计从来不只是“有没有字段”。  
它还总是隐含着更强的判断：

1. 哪些字段该最先打断用户
2. 哪些字段该做卡片标题
3. 哪些字段只能放进高级详情
4. 哪些字段如果埋得太深，就等于根本没显化

换句话说，  
就算我们已经知道必须补：

1. `host_kind`
2. `projection_grade`
3. `missing_ledgers`
4. `write_ready / read_ready`
5. `supported_control_subtypes`
6. `completion_claim_ceiling`

如果最后把它们统统塞进一个“更多信息”折叠区，  
那控制面还是会回到老问题：

`底层已经诚实，展示层仍然不诚实。`

所以这一章的核心问题是：

`统一安全控制台应该把宿主盲区字段挂到哪张卡、按什么优先级显示，才能真正阻止用户把局部完成误读成全链完成？`

## 2. 最短结论

从源码和现有展示层看，  
下一代统一安全控制台不应再把宿主差异当成“附加提示”。  
它至少应形成一套三层展示结构：

1. 顶层断言层：先限制当前最多能说多满
2. 中层宿主能力层：解释当前宿主看到了什么、缺了什么
3. 下层调试细节层：给需要深挖的人看具体字段来源和 basis

换句话说，  
这批字段最合理的卡片挂载方式不是“单独再加一页状态”，  
而是：

`把 completion ceiling 抬到最上层，把 host/projection/missing ledger 放到宿主能力卡，把 write/read readiness 与 subtype 宽度放到展开层。`

所以这一章的最短结论是：

`宿主盲区字段如果不按优先级重新挂到卡片结构里，统一安全控制台仍然会在最后一层交互里把重要差异埋掉。`

## 3. 为什么当前界面组织方式天然会压平这些差异

## 3.1 `/status` 当前更像摘要列表，不像因果控制台

`src/components/Settings/Status.tsx:19-55` 与  
`src/utils/status.tsx:38-115` 非常关键。  
当前 `Status` 页面主要由：

1. primary section
2. secondary section
3. diagnostics

组成。  
而其中 MCP、IDE、sandbox、setting sources 都被压成 property 列表或摘要行。

例如 `buildMcpProperties()` 最后直接汇总成：

`MCP servers: X connected, Y need auth, Z pending, ...`

这说明当前 `/status` 面在结构上天然偏向：

`摘要归档`

而不是：

`优先级驱动的控制面`

这类组织方式很容易把宿主盲区字段埋在同一层 property 列表里，  
无法第一时间拦住用户的过满解读。

## 3.2 `BridgeDialog` 当前会显示状态、错误和少量 debug id，但不会解释 completion ceiling

`src/components/BridgeDialog.tsx:137-257` 显示，  
BridgeDialog 目前会组织：

1. status label
2. error
3. environment / session id（verbose）
4. footer text

这是一个很典型的“当前连接状态 + 辅助信息”对话框。  
但它不会继续显式回答：

1. 当前 `active` 到底是 write-ready 还是 full completion projection
2. 当前 recovery 是否正在收缩结果解释权
3. 当前宿主缺哪本账

也就是说，  
它已经证明 UI 有地方放这些信息，  
但当前并没有把最关键的完成差异字段抬到显眼位置。

## 3.3 IDE 指示器和 MCP 通知也都在做单维摘要

`src/hooks/useIdeConnectionStatus.ts:11-32` 先把 IDE 状态压成：

1. `connected`
2. `pending`
3. `disconnected`

`src/components/IdeStatusIndicator.tsx:18-56` 再只在 connected 时显示“选中了哪些代码/文件”。  
`src/hooks/notifs/useMcpConnectivityStatus.tsx:29-63` 则继续把 MCP 问题压成：

1. 几个 failed
2. 几个 needs-auth

这说明当前多个用户可见表面都采用了同一策略：

`先压成单维状态，再给一个去 /mcp 或看选区的轻提示。`

这种组织方式对轻交互很好，  
但对统一安全控制台来说有个致命问题：

`它没有给宿主盲区字段留下优先级位置。`

## 4. 从第一性原理看，这批字段不该形成“第八张卡”，而应重构现有卡片层次

`24-统一安全控制台卡片设计` 已经把控制台压成七张卡。  
我现在的判断是：

`宿主完成差异字段不应再新造一张孤立卡，而应重构至少三张已有卡的层次。`

最关键的是这三张：

1. 主体与资格卡
2. 外部能力卡
3. 断链诊断卡

因为宿主盲区并不只属于某一个子系统，  
它更像是：

`当前整个控制台说话资格的元数据。`

## 5. 第一挂载点：主体与资格卡顶部必须先声明 `completion_claim_ceiling`

## 5.1 为什么第一眼最该看到的不是 connected，而是“当前最多能说到哪一步”

用户进入控制台时，  
最容易犯的错并不是不知道当前连没连上，  
而是：

`默认以为当前界面已经代表了完整系统真相。`

因此主体与资格卡顶部最该先展示的，不是：

1. Connected
2. Active
3. Remote ready

而应是：

`completion_claim_ceiling`

例如：

1. `当前仅能确认事件级完成`
2. `当前可确认复制态完成`
3. `当前可确认恢复级完成`
4. `当前处于降级窗口，不应下完整完成结论`

## 5.2 这张顶层字段必须抢占标题区，而不是放进详情区

原因很简单：  
只要它不在标题区，  
用户就会先读到旧式 connected/active 文案，  
然后再很难回头修正自己的 certainty level。

也就是说，  
这不是普通说明字段，  
而是：

`当前整张控制台的解释上限字段。`

## 6. 第二挂载点：外部能力卡里必须显式展示 `host_kind + projection_grade + missing_ledgers`

## 6.1 为什么这三者应该挂在同一张卡里

因为用户真正想知道的是一整句完整回答：

1. 我现在在哪个宿主上
2. 这个宿主看到了多厚的完成投影
3. 它又明确缺哪本账

这三者天然是一个因果簇。  
如果拆开放在不同卡，  
用户就得自己拼接：

`host_kind -> projection_grade -> missing_ledgers`

而这正是统一控制台不该再把负担留给用户的地方。

## 6.2 外部能力卡目前最接近这个位置

因为它本来就处理：

1. MCP
2. IDE
3. bridge
4. 外部入口连通性

也就是说，  
它天然已经是“当前你通过哪条宿主/外部通路在看系统”的卡。  
把：

1. `host_kind`
2. `projection_grade`
3. `missing_ledgers[]`

挂到这里，  
比另起一张盲区卡更符合用户直觉。

## 7. 第三挂载点：readiness 分裂字段必须挂在状态 pill 下钻层，而不是藏进 debug

## 7.1 `write_ready / read_ready` 不是纯调试字段

`replBridgeTransport.ts:289-299` 虽然说 `getStateLabel()` 只用于 debug logging，  
但这恰恰说明当前实现已经知道：

1. 写准备度
2. 读准备度
3. synthesized label

是不同层。

如果统一控制台仍把这些差异继续埋到 debug 展开区，  
那它就还是在重复旧问题：

`内部知道，用户看不到。`

## 7.2 最合理的方式是“主标签 + 下钻解释”

也就是：

1. 表层仍可显示一个简化标签
2. 但标签下必须能立即展开：
   - `write_ready`
   - `read_ready`
   - `label_basis`

这样既保留表层简洁，  
又不牺牲关键差异的可见性。

## 8. 第四挂载点：`supported_control_subtypes` 应挂在动作区，而不是状态区

## 8.1 为什么 control 宽度字段不适合放在状态摘要卡

因为这个字段本质上回答的是：

`当前宿主能执行哪些控制动作。`

它不是纯观察态字段，  
而是：

`动作能力上限字段`

因此它更适合挂在：

1. 动作区
2. 能力详情区
3. 断链诊断卡的“为什么当前不给这个动作”区

## 8.2 这能直接防止一种常见误解

用户最常见的误解之一是：

`既然协议里有这个动作，这个宿主现在应该也能做。`

而 `supported_control_subtypes[]` 如果被明确放在动作区，  
就能把这条误解直接掐断。

## 9. 第五挂载点：`recovery_window_open` 和 suspension 字段必须抢占警告层

## 9.1 这类字段一旦埋深，伤害最大

像：

1. `recovery_window_open`
2. `result_projection_suspended`
3. `control_projection_suspended`

这种字段，本质上都不是普通详情。  
它们是在说：

`当前你正在看的完成解释权本身就在收缩。`

这类字段一旦只放在高级详情区，  
用户几乎注定会先按正常路径理解状态，  
然后做出错误动作。

## 9.2 因此它们必须进入高优先级警告层

也就是控制台里那种：

1. 一眼可见
2. 抢占普通摘要之上
3. 先于 connected/active 解释的层

这在产品上看像 warning banner，  
但在语义上其实是：

`当前解释权收缩告警。`

## 10. 把这些挂载点压成一套最小展示优先级

我会把它压成四层：

### 10.1 第一层：结论上限层

放：

1. `completion_claim_ceiling`
2. `recovery_window_open`
3. `result_projection_suspended`
4. `control_projection_suspended`

这层先回答：

`当前这张界面最多敢说到哪一步。`

### 10.2 第二层：宿主能力层

放：

1. `host_kind`
2. `projection_grade`
3. `missing_ledgers[]`

这层回答：

`你现在到底站在哪个观察面上。`

### 10.3 第三层：状态分裂层

放：

1. `write_ready`
2. `read_ready`
3. `label_basis`

这层回答：

`当前状态词是由哪种 readiness 维度合成出来的。`

### 10.4 第四层：动作上限层

放：

1. `supported_control_subtypes[]`
2. 关联的默认动作 / 不给动作原因

这层回答：

`当前宿主到底能做什么，为什么别的动作现在不给。`

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 当前已经拥有做“诚实卡片控制台”的全部关键字段与状态槽位，真正缺的是把这些字段按照解释上限优先级重新挂回界面。`

这对其他 Agent 平台构建者的启示很直接：

1. 字段设计完成后，不要默认字段平铺就等于完成产品化
2. 解释上限字段必须先于状态字段展示
3. 宿主盲区字段应作为卡片骨架，不应只做 tooltip
4. readiness 分裂不应只在 debug 页面出现
5. capability ceiling 字段应进入动作区而不是埋在设置详情里

## 12. 哲学本质

这一章更深层的哲学是：

`诚实不只是说出限制，还包括让限制抢占足够靠前的位置。`

很多系统的失败并不是没有把限制写出来，  
而是把限制放在了用户几乎不会先读到的地方。  
Claude Code 的下一步如果要继续把安全做深，  
真正该完成的不是再补一条提示语，  
而是：

`让“我现在不能把话说满”的字段先于“我现在看起来还不错”的字段出现。`

从第一性原理看，  
这保护的是一条很根本的公理：

`被埋深的边界，最终等于不存在的边界。`

## 13. 苏格拉底式反思：这一章最容易犯什么错

### 13.1 我们会不会把太多信息推到界面最前面

有风险。  
所以这里强调的是四层最小结构，  
不是让每张卡都膨胀成字段清单。

### 13.2 为什么不直接做一张“宿主盲区卡”

因为宿主盲区不是某个附属问题，  
而是每张卡解释上限的元信息。  
把它单独隔离，反而更容易被忽略。


## 14. 结语

`45` 回答的是：统一安全控制台最少该补哪些字段。  
这一章继续推进后的结论则是：

`这些字段不能被平铺处理，而必须按解释上限优先级挂到正确的卡片层次上。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正成熟的控制面，不只会设计诚实字段，还会让这些字段出现在用户最不容易误解的位置。`
