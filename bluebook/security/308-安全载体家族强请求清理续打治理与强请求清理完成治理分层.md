# 安全载体家族强请求清理续打治理与强请求清理完成治理分层：为什么artifact-family cleanup stronger-request continuation-governor signer不能越级冒充artifact-family cleanup stronger-request completion-governor signer

## 1. 为什么在 `307` 之后还必须继续写 `308`

`307-安全载体家族强请求清理step-up重授权治理与强请求清理续打治理分层`
已经回答了：

`系统现在已经把更高 authority 还给你`

不等于

`那个先前被挡下来的 stronger cleanup request 现在已经以同一请求名义继续。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成强结论的错觉：

`既然系统已经允许这条 old blocked stronger cleanup request 继续，那它现在当然就算完成了。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/services/mcp/client.ts:2813-3025` 的 `callMCPToolWithUrlElicitationRetry()`
2. `src/services/mcp/client.ts:1848-1895` 的 started / completed progress emission
3. `src/services/mcp/client.ts:3029-3238` 的 `callMCPTool()`
4. `src/services/mcp/client.ts:2662-2728` 的 `transformMCPResult()` 与 `processMCPResult()`
5. `src/tools/FileWriteTool/FileWriteTool.ts:418-430`
6. `src/tools/TaskCreateTool/TaskCreateTool.ts:130-135`

会发现 repo 已经很明确地把两层制度拆开了：

1. `stronger-request continuation governance`
   负责决定这条 old blocked stronger cleanup request 是否仍配继续尝试，以及继续时的 actor、budget、consent 与 waiting grammar
2. `stronger-request completion governance`
   负责决定这次继续之后，当前请求究竟是 `could not complete`、`completed`、`error`、`abort`，以及谁有权把结果结算成 completion-grade output

也就是说：

`artifact-family cleanup stronger-request continuation-governor signer`

和

`artifact-family cleanup stronger-request completion-governor signer`

仍然不是一回事。

前者最多能说：

`这条旧 stronger cleanup request 现在仍配被当成同一请求继续，且已经拿到继续所需的 trigger / consent / retry budget。`

后者才配说：

`这条旧 stronger cleanup request 现在已经真正产出 completion-grade result，或者被正式签成未完成/失败/中止，而不再停留在“继续尝试中”。`

所以 `307` 之后必须继续补的一层就是：

`安全载体家族强请求清理续打治理与强请求清理完成治理分层`

也就是：

`stronger-request continuation governor 决定旧强请求能否继续；stronger-request completion governor 才决定它是否真的已经完成。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request completion-governor signer。`

这里的
`artifact-family cleanup stronger-request completion-governor signer`
仍是研究命名。
它不是在声称 repo 里藏着一个未公开的 completion manager，
而是在说：

1. repo 已经把 retry permission 与 result settlement 写成不同 grammar
2. repo 已经把 `could not complete`、`status: 'completed'`、`completed successfully`、`tool_result` success block 写成不同 ceiling
3. stronger-request cleanup 线未来若只补 continuation grammar，而不补 completion grammar，仍会留下“谁来正式宣布 continued stronger cleanup request 现在已经完成”的缺口

第二条：

这里的 `completion`
不是在声称：

`只要当前有一句 success string，就等于 durable finality 已经成立。`

`149-安全完成与终局分层`
与后续 stronger-request finality 链已经回答过：

`completion`

不等于

`finality`

`308` 这里只问：

`在 old blocked stronger cleanup request 已经被允许继续的语境里，谁有权把系统从“还在尝试”推进到“当前请求已经完成/未完成”的结果层。`

第三条：

这里也不是在说：

`continuation grammar 绝不能同时返回结果。`

更稳的说法是：

`即便 continuation path 最终会导向某个结果，它也不能因此越级替 completion signer 签字。`

继续尝试是一段 control grammar；
结果结算才是 completion grammar。

第四条：

这里尤其不能把：

`accepted to continue`

偷写成：

`already completed`

源码里 `accept`
只负责把 old blocked request 接回执行链；
真正的 completion 还要等待 `callMCPTool()` 成功或失败、结果被 normalize、progress 被正确发出，甚至 tool 自己产出 result block。

第五条：

这里还不能把：

`工具重新返回了内容`

偷写成：

`系统已经合法结案`

`transformMCPResult()` / `processMCPResult()` 明明还要继续检查：

1. 结果是 `toolResult`、`structuredContent` 还是 `contentArray`
2. format 是否 unexpected
3. abort 时是否只允许留下 `undefined content`

这说明 raw return 也不等于 completion-grade settlement。

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request continuation-governor signer 仍不等于 stronger-request completion-governor signer`

证据：

1. `callMCPToolWithUrlElicitationRetry()` 在 decline / cancel 时会明确返回 `could not complete`，说明 continuation path 可以合法停在 non-completion
2. outer MCP tool path 只在 `mcpResult` 返回之后才发 `status: 'completed'`，说明 completed progress 不是 continuation 的内建属性
3. `callMCPTool()` 只有在底层 `client.callTool()` 成功、`result.isError` 被排除且 `processMCPResult()` 正常完成后，才记录 `completed successfully`
4. 同一函数在 `AbortError` 上只返回 `{ content: undefined }`，拒绝让 aborted execution 冒充 completion
5. `transformMCPResult()` / `processMCPResult()` 只关心 settled result shape，而不关心 retry permission
6. `FileWriteTool` / `TaskCreateTool` 的 `tool_result` success block 直接以结果所有者身份说话，不再说“你可以继续试”

因此这一章的最短结论是：

`stronger-request continuation governor 最多能说旧 stronger cleanup request 现在仍配再试；stronger-request completion governor 才能说这条 stronger cleanup request 现在已经产出了 completion-grade result，或者被正式签成未完成/失败/中止。`

再压成一句：

`已经被允许继续，不等于已经真正完成。`

## 4. 第一性原理：continuation 回答“这条被打断的强请求还该不该继续”，completion 回答“继续之后有没有真的被结算”

从第一性原理看，
这两层处理的是不同主权问题。

`stronger-request continuation governance`
回答的是：

1. 旧 stronger cleanup request 是否仍应被视为同一请求
2. 哪一类中断允许继续尝试
3. 继续者是谁
4. retry budget 是多少
5. 用户 / hook 是否已经同意进入下一轮尝试

`stronger-request completion governance`
回答的则是：

1. 这轮继续之后是否已经返回 result，而不是仍停在等待 / 重试 / 拒绝状态
2. 结果是 success、error、could-not-complete 还是 aborted
3. 哪个 progress、哪个 normalized result、哪个 explicit result block 才配被视为 completion-grade output
4. 哪些 control-plane success 仍不能被偷写成请求完成
5. 当前这条 stronger cleanup request 是否已经拿到 request-level closure

如果把这两层压成一句：

`反正已经 retry 了`

系统就会制造至少六类危险幻觉：

1. retried-means-finished illusion
2. accepted-means-completed illusion
3. elicitation-completed-means-tool-completed illusion
4. progress-started-means-success illusion
5. tool-returned-means-settled illusion
6. reauthorized-and-resumed-means-result-owned illusion

所以从第一性原理看：

`continuation` 管的是 causal reopening；
`completion` 管的是 result settlement。

其哲学本质也可以压成一句：

`制度已经允许旧动作重新进入因果链，不等于制度已经替这个动作签发结案书。`

## 5. `callMCPToolWithUrlElicitationRetry()` 先证明：续打路径里明明白白地存在“继续了，但仍未完成”

`src/services/mcp/client.ts:2813-3025`
非常硬。

这段代码最值钱的地方不只是它有 retry loop，
而是它在 continuation path 内部，
就已经把：

`继续`

和

`未完成`

拆开了。

具体说：

1. 只有 `ErrorCode.UrlElicitationRequired` 才能进入这条 continuation path
2. continuation 被明确限制在 `MAX_URL_ELICITATION_RETRIES = 3`
3. hook 不是 `accept` 时，会直接返回
   `The tool "<tool>" could not complete because it requires the user to open a URL.`
4. 用户 / result hook 最终不是 `accept` 时，同样返回 `could not complete`
5. 只有 `accept` 成立时，才 loop back 到下一轮 `callToolFn`
6. queue path 还把 `Retry now`、`showCancel` 与 `onWaitingDismiss` 拆成两阶段等待语法

这条证据非常关键。

因为它公开说明：

`same-request continuation`

即便已经成立，
仍然可以合法地落在：

`same-request non-completion`

而不是自动升格成：

`same-request completion`

这意味着 continuation signer 的权力上限非常清楚：

它能说：

`继续`

却不能说：

`已经完成`

## 6. `status: 'completed'`、`callMCPTool()` 与 result normalization 再证明：真正的 completion signer 只在结果回来之后说话

`src/services/mcp/client.ts:1848-1895`
很值钱。

outer MCP tool path 先发：

`status: 'started'`

随后进入：

`callMCPToolWithUrlElicitationRetry()`

只有拿到 `mcpResult` 之后，
才发：

`status: 'completed'`

这说明：

1. continuation path 可以跑很久
2. 其中可能经历多次 retry / waiting / accept
3. 但 completed progress 不是 continuation grammar 的内建属性
4. completed progress 要等结果真正回来之后才配发

`src/services/mcp/client.ts:3029-3238`
给出第二组强证据。

`callMCPTool()` 的逻辑非常干净：

1. `client.callTool()` 真正发起底层调用
2. 若 `result.isError`，就提取错误并抛出 `McpToolCallError`
3. 只有真正成功后，才记录
   `Tool '<tool>' completed successfully`
4. 接着调用 `processMCPResult()` 处理结果
5. 最后才返回 `{ content, _meta, structuredContent }`
6. 若是 `AbortError`，就只返回 `{ content: undefined }`

这说明 completion 在这里不是一种“继续执行的姿态”，
而是一种“结果已经被结算并可返回”的状态。

`src/services/mcp/client.ts:2662-2728`
再给出第三组强证据。

`transformMCPResult()` 与 `processMCPResult()` 继续追问的是：

1. 结果是 `toolResult`
2. 还是 `structuredContent`
3. 还是 `contentArray`
4. unexpected response format 是否应直接报错

也就是：

`完成后的结果究竟长什么样`

这和 continuation layer 关心的：

`为什么能继续`

完全不是一个问题。

所以 repo 在代码职责上已经公开做出切分：

1. continuation grammar 负责 retry control
2. completion grammar 负责 result settlement / normalization

## 7. `tool_result` 与 abort discipline 再证明：真正的 completion signer 会直接对结果负责

`src/tools/FileWriteTool/FileWriteTool.ts:418-430`
与
`src/tools/TaskCreateTool/TaskCreateTool.ts:130-135`
是很好的正对照。

它们在成功时分别返回：

1. `File created successfully at: <path>`
2. `The file <path> has been updated successfully.`
3. `Task #<id> created successfully: <subject>`

这些 `tool_result` block 的关键不在于它们都是 success string，
而在于：

它们已经在以结果所有者的身份说话。

它们不再说：

`你可以继续`

而是在说：

`请求已经完成，而这就是完成后的结果`

同时，
`callMCPTool()` 在 `AbortError` 上只返回 `{ content: undefined }` 也非常值钱。

因为它说明 repo 宁可留下一个明确的未完成空洞，
也不愿意让被中断执行伪装成 completion。

再往下一层看，
`src/tools/MCPTool/MCPTool.ts:70-75`
把 MCP 成功结果映射成的只是：

`{ type: 'tool_result', tool_use_id, content }`

而
`src/services/tools/toolExecution.ts:1717-1723`
在失败 path 上才显式补：

`is_error: true`

这说明 success 在协议层往往是隐式的，
error 才是显式标记的。

更进一步，
`src/utils/toolResultStorage.ts:280-294`
还会把空 `tool_result` 内容统一改写成：

`(<toolName> completed with no output)`

这条证据非常有价值。

因为它说明 completion grammar 不只是“有没有 payload”，
还包括：

1. 空结果怎样被诚实命名
2. 协议层怎样避免空输出破坏后续推理
3. friendly completion copy 何时只是 renderer/storage 的兼容性补丁，而不是更强的语义完成证据

这正是 completion signer 与 continuation signer 的最大区别：

前者必须对结果负责，
后者只需要对“能否继续”负责。

## 8. 更深一层的技术先进性：Claude Code 不只把“能否续打”做成控制语法，还把“何时算完成”做成结算语法

从技术角度看，
Claude Code 在这一层最先进的地方，
不是它“会 retry”，
而是它拒绝把：

`retry permission`

偷写成：

`result settlement`

这套设计至少体现了六个成熟点：

1. `non-completion is a first-class outcome`
   continuation path 可以合法终止于 `could not complete`
2. `progress and settlement are layered`
   `started / completed / progress` 与 retry loop 明确拆层
3. `result ownership is explicit`
   success 结果要么由 `completed successfully`，要么由 `tool_result` block 负责
4. `raw return is not enough`
   还要经过 `transformMCPResult()` / `processMCPResult()` 的格式与内容结算
5. `abort does not impersonate completion`
   aborted execution 会留下 `undefined content`
6. `shape validation is part of completion honesty`
   unexpected format 不会被温柔吞掉，而是直接报错
7. `friendly copy is not identical to semantic proof`
   `completed with no output` 这类 storage/rendering 兼容文案也要被放回 completion grammar，而不能被误当成更强语义的替身

它的哲学本质是：

`安全不只问“还能不能接着做”，还问“这次做完之后到底算不算已经完成”。`

对 stronger-request cleanup 线的技术启示也因此非常清楚：

如果未来只补出：

`谁来决定 old blocked stronger cleanup request 还能不能继续`

却没有补出：

`谁来决定 could-not-complete / completed / aborted`

`谁来负责 result settlement`

`谁来产出 completion-grade result block`

那么这条线依然只是 continuation grammar，
还不是 completion grammar。

## 9. 苏格拉底式自我反思：如果我把 `308` 写得更强，我会在哪些地方越级

可以先问六个问题：

1. 如果 `accept` 已经等于 completion，为什么 `callMCPToolWithUrlElicitationRetry()` 还会在非 `accept` 时明确落到 `could not complete`？
2. 如果 old request 只要 loop back 一次就算完成，为什么 `status: 'completed'` 还要等到 `mcpResult` 返回后才发？
3. 如果底层工具只要有返回就够，为什么 `transformMCPResult()` 还要继续区分 `toolResult`、`structuredContent` 与 `contentArray`？
4. 如果 interrupted execution 也能算完成，为什么 `AbortError` 只允许留下 `undefined content`？
5. 如果 continuation signer 已经拥有结果主权，为什么 `FileWriteTool` 与 `TaskCreateTool` 还要单独产出 explicit success `tool_result`？
6. 如果 stronger-request cleanup 线未来只补 retry grammar，不补 result settlement，谁来正式负责“这条 continued stronger cleanup request 已经完成”这句重话？

这些反问逼着我们承认：

`continuation`
只回答
`old request 是否仍可继续`

而
`completion`
真正回答的是：

`这次继续之后有没有拿到可以负责的结果`

这也是本章最应该记住的压缩句：

`制度已经允许旧请求回到执行链，不等于制度已经替这个旧请求结案。`

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 线只要补出 continuation grammar，就已经足够成熟。`

而是：

repo 已经在 `callMCPToolWithUrlElicitationRetry()` 的 `could not complete` 分流、outer MCP tool path 的 `status: 'completed'` progress、`callMCPTool()` 的 `completed successfully` 与 abort-not-complete discipline、`transformMCPResult()` / `processMCPResult()` 的 normalized result path，以及 `FileWriteTool` / `TaskCreateTool` 的 explicit success `tool_result` 上，清楚展示了 stronger-request completion governance 的独立存在；因此 `artifact-family cleanup stronger-request continuation-governor signer` 仍不能越级冒充 `artifact-family cleanup stronger-request completion-governor signer`。

因此：

`旧 stronger cleanup request 现在仍配继续`

最多只配说明：

`the request may continue trying`

而不是：

`the request has already completed`
