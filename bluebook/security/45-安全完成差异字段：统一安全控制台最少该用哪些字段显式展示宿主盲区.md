# 安全完成差异字段：统一安全控制台最少该用哪些字段显式展示宿主盲区

## 1. 为什么在 `44` 之后还要继续写“完成差异字段”

`44-安全完成差异显化` 已经回答了：

`宿主缺失的完成链不能只藏在实现里，而必须被统一安全控制台显式告诉用户。`

但如果继续往产品实现层再压一层，  
还会立刻遇到一个更具体的问题：

`到底该显示哪些字段，才能把这种差异说清楚？`

因为只要还停留在抽象层，  
“显式展示宿主盲区”就仍然只是一个方向。  
而一旦进入字段级，问题就会变得非常严苛：

1. 哪些字段源码已经有
2. 哪些字段只差重组
3. 哪些字段必须新派生
4. 缺少哪个字段，用户就会被哪一种伪完成误导

所以这一章的核心问题是：

`如果统一安全控制台真要把宿主完成差异显化出来，最少必须有哪些字段？`

## 2. 最短结论

从源码看，下一代统一安全控制台并不缺原材料。  
真正缺的是一组专门服务于“宿主完成差异显化”的字段模型。

我会把这组最小字段压成六类：

1. 宿主身份字段
2. 完成可见度字段
3. 缺失账本字段
4. readiness 维度字段
5. 控制面宽度字段
6. 结论上限字段

也就是说，  
如果未来控制台只继续显示：

1. connected
2. active
3. completed
4. failed

那它仍然会把不同宿主的完成差异压平。  
但如果至少补上：

1. `host_kind`
2. `projection_grade`
3. `missing_ledgers`
4. `write_ready` / `read_ready`
5. `supported_control_subtypes`
6. `completion_claim_ceiling`

那么同一条状态就第一次有机会在产品层说得诚实。

## 3. 第一类字段：宿主身份字段

## 3.1 为什么第一件事不是状态，而是 `host_kind`

只要界面不先告诉用户“你当前面对的是哪类宿主”，  
后面几乎所有 completion 语义都会被默认当成同一层。

从源码看，这个字段并不需要凭空发明。  
不同宿主的结构事实已经非常明确：

1. `StructuredIO` 默认无恢复账、无 flush 视角
2. `RemoteIO` 有恢复读回、internal event read/write、state/metadata 回报
3. bridge 有更厚状态投影，但会在 recovery 窗口收缩
4. `DirectConnectManager` 与 `RemoteSessionManager` 是窄 control 宿主

所以统一控制台最少应直接显式一个字段：

`host_kind`

例如可以是：

1. `sdk_stream_consumer`
2. `structured_io`
3. `remote_io_ccr`
4. `bridge`
5. `direct_connect`
6. `remote_session_manager`

## 3.2 证据来自哪里

源码里已有足够多的身份线索：

- `AppStateStore.ts:122-155` 的 remote / bridge 状态槽位
- `structuredIO.ts:139-196` 的默认空恢复视角
- `remoteIO.ts:126-167` 与 `217-223` 的增强完成投影
- `directConnectManager.ts:81-99`
- `RemoteSessionManager.ts:192-213`

这说明 `host_kind` 不是猜测字段，  
而是：

`把当前宿主结构身份从隐式实现细节提升成显式产品字段。`

## 4. 第二类字段：完成可见度字段

## 4.1 为什么必须有 `projection_grade`

仅仅知道“你在 bridge 上”仍然不够。  
同一类宿主内部，也可能因为 recovery、flush、恢复账接入与否而拥有不同厚度的完成投影。

因此控制台最少还需要一个总结字段：

`projection_grade`

它回答的问题不是：

`你是谁`

而是：

`你现在看到了多厚的完成投影`

最小可用分级可以是：

1. `event_only`
2. `event_plus_replica`
3. `event_plus_replica_plus_recovery`
4. `degraded_recovery_window`

## 4.2 这不是主观评分，而是对源码差异的压缩编码

例如：

1. `sdkEventQueue.ts` 暴露的是 authoritative turn-over event
2. `ccrClient.reportState()` / `reportMetadata()` 暴露的是复制态
3. `restoredWorkerState` 与 internal event reader 暴露的是恢复基底
4. `remoteBridgeCore.ts` 的 recovery drop 说明这条投影会动态降级

所以 `projection_grade` 的意义，是把原本分散在多处的结构差异，  
压成一个用户能立即理解的字段。

## 5. 第三类字段：缺失账本字段

## 5.1 为什么只说“较薄视图”还不够，必须明确写出 `missing_ledgers`

真正危险的不是宿主较薄，  
而是用户不知道它究竟薄在哪里。

因此控制台需要直接列出：

`missing_ledgers`

最少应允许显式列出：

1. `recovery_ledger`
2. `internal_event_flush_ledger`
3. `replica_metadata_ledger`
4. `full_control_ledger`

## 5.2 源码里这类缺失已经被明确编码

例如：

- `StructuredIO` 直接把 `restoredWorkerState = null`
- `flushInternalEvents()` 默认 no-op
- `internalEventsPending = 0`
- `DirectConnectManager` / `RemoteSessionManager` 对未知 control subtype 显式 error

这些都不是“也许没有”，  
而是：

`当前明确缺这本账。`

所以最成熟的界面做法不是写：

`Some features may be unavailable`

而是写成结构字段：

`missing_ledgers = ['recovery_ledger', 'full_control_ledger']`

## 6. 第四类字段：readiness 维度字段

## 6.1 `connected` 不够，必须拆成 `write_ready` 和 `read_ready`

`replBridgeTransport.ts:289-299` 的证据已经足够强：

1. `isConnectedStatus()` 是 write-readiness
2. 不是 read-readiness

这意味着统一安全控制台如果继续只显示：

`connected`

就仍然是在复用一个已经被源码明确拆开的词。

所以最少需要两个字段：

1. `write_ready`
2. `read_ready`

必要时再加：

3. `label_basis`

用于标明当前这个状态词到底是从哪种可见性推导出来的。

## 6.2 这类字段的产品价值非常高

因为很多用户的直觉会把：

`能写`

自动补全成：

`也能稳定读、能稳定回读、能稳定解释完成`

而 readiness 分裂字段正是用来阻断这种自动补全。

## 7. 第五类字段：控制面宽度字段

## 7.1 为什么必须显示 `supported_control_subtypes`

一个宿主到底能不能对完成说满，  
和它拥有多宽的 control surface 直接相关。

`DirectConnectManager` 与 `RemoteSessionManager` 当前都主要正式支持：

`can_use_tool`

这意味着它们不是没有价值，  
而是：

`它们的完成解释权天然只能覆盖极窄控制闭环。`

所以控制台必须最少暴露：

`supported_control_subtypes`

这能让用户和接入方马上看见：

1. 这是完整 host
2. 还是窄审批 host
3. 还是只读观察 host

## 7.2 这也是防止“schema 全集 = 宿主全集”的关键字段

很多系统最容易误把：

`协议里声明过`

误读成：

`这个宿主当前都支持`

而 `supported_control_subtypes` 正是用来反击这种误读的最小字段。

## 8. 第六类字段：结论上限字段

## 8.1 为什么最后还必须有 `completion_claim_ceiling`

就算界面已经展示了：

1. `host_kind`
2. `projection_grade`
3. `missing_ledgers`
4. `write_ready`
5. `supported_control_subtypes`

用户依然可能继续问：

`那你现在最多敢把“已完成”说到什么程度？`

因此控制台最少还需要一个总结字段：

`completion_claim_ceiling`

例如：

1. `event_only_completion`
2. `replica_visible_completion`
3. `recovery_backed_completion`
4. `degraded_no_completion_claim`

这个字段的意义，是把前面所有差异最终收束成：

`当前这块界面最多能说到哪一步。`

## 8.2 这是控制台最重要的诚实字段之一

因为很多 UI 最后都会回到一句人话。  
而这句人话如果没有 ceiling 约束，  
就会重新膨胀成：

`completed`

这种过满结论。

## 9. 第七类字段：哪些字段源码已经有，哪些只是轻度派生

从实现难度看，这批字段并不都需要全新底层改造。

已经有较强直接来源的包括：

1. `host_kind`
2. `write_ready`
3. `recovery_window_open`
4. `supported_control_subtypes`
5. `restored_worker_state_supported`
6. `internal_event_flush_supported`

需要轻度派生与重组的包括：

1. `projection_grade`
2. `missing_ledgers`
3. `completion_claim_ceiling`
4. `label_basis`

这说明下一代统一安全控制台真正缺的不是基础设施，  
而是：

`把已有结构事实重组为一套诚实字段。`

## 10. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`宿主完成差异如果不能被字段化，就只能继续被文案压平。`

这条公理几乎能统一前面所有证据：

1. 状态摘要会天然压平异质宿主
2. generic informational 文案会天然丢掉边界
3. connected/active 这种单词会天然吞掉 readiness 分裂
4. 窄 control 宿主会天然被误写成完整 host
5. 只有字段化，才能让这些差异稳定进入控制台和协议评审

所以在 `44` 之后，  
下一步最自然的产品化结论已经很清楚：

`不是再发明更多故事，而是先把宿主盲区做成正式字段模型。`

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 已经具备做“诚实字段控制台”的所有关键原材料，只差把它们从模块内事实提升成跨面字段。`

这对其他 Agent 平台构建者的启示也很直接：

1. 不要只为“状态是什么”设计字段，也要为“当前看不到什么”设计字段
2. 不要只为“连接状态”设计字段，也要为 readiness 分裂设计字段
3. 不要只为“宿主类型”设计字段，也要为完成结论上限设计字段
4. 如果一个差异无法进入字段模型，它最后几乎一定会在文案层被压平
5. 统一控制台真正的高级之处，不是看起来信息很多，而是字段能诚实表达自己的盲区

## 12. 哲学本质

这一章更深层的哲学是：

`诚实的系统，不只是会说真话，还会把自己不能说满的原因做成对象。`

很多系统直到最后都停留在文案层：  
看起来很人性化，  
但无法稳定表达自己的观察边界。  
Claude Code 的源码已经在更深一层给出了答案：

`把边界做成字段。`

从第一性原理看，  
这保护的是一条极其根本的公理：

`只要盲区不是对象，盲区最终就会在产品层消失。`

## 13. 苏格拉底式反思：这一章最容易犯什么错

### 13.1 我们会不会设计了太多字段

有风险。  
所以这里强调的是最小字段集，  
不是把所有底层细节都端给用户。

### 13.2 为什么不直接只显示一条“当前视图不完整”

因为那仍然太粗。  
用户需要知道是：

1. 缺恢复账
2. 缺 flush 视角
3. 缺 control 子集
4. 还是 readiness 本身不完整

### 13.3 这一章之后还缺什么

还缺最后一层：

`这些字段在统一安全控制台里应该分别挂到哪张卡、以什么优先级展示。`

也就是说，  
下一步最自然的延伸就是把这章压成一张：

`宿主完成差异字段速查表`

## 14. 结语

`44` 回答的是：宿主差异必须被显式展示。  
这一章继续推进后的结论则是：

`要做到这件事，统一安全控制台最少必须拥有一组专门表达宿主盲区和完成上限的字段。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正成熟的控制面，不只是会展示状态，还会把“我为什么不能把这条状态说得更满”变成正式字段。`
