# 安全载体家族强请求清理制度理由治理与强请求清理制度元数据治理分层：为什么artifact-family cleanup stronger-request cleanup-rationale-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-metadata-governor signer

## 1. 为什么在 `287` 之后还必须继续写 `288`

`287-安全载体家族强请求清理家族宪法治理与强请求清理制度理由治理分层`
已经回答了：

`不同 stronger-request old cleanup carriers 不只活在不同 cleanup law 里，而且服务不同 risk object、operational duty、reader world 与 host visibility。`

但如果继续往下追问，
还会碰到下一层更工程化、也更容易被偷换的幻觉：

`只要这些 family 的制度理由已经存在，系统就一定已经把这些理由保存成了自己会重复读取的 metadata plane。`

Claude Code 当前研究证据仍然不能支持这种更强说法。

因为继续看：

1. task-output family 的 non-clobber truth 虽然已经非常清楚，但主要仍寄居在 `diskOutput.ts` 的 comment 与 path helper 局部真相里
2. tool-results 与 transcripts 虽同处 project-session persistence world，却分别承载 inspection 与 continuity / replay duty，但两者并没有被系统提升成统一 family descriptor
3. plans family 的 truth 已经横跨 settings、storage、permissions、resume 与 cleanup 五个平面，却仍暴露出 `plansDirectory` 可以改写前四层、cleanup 仍停在默认 home-root 的传播断层
4. file-history、session-env、debug 与 diagnostics 都展示了十分明确的 family-specific rationale，但这些 truth 依旧主要靠 helper、comment、argv / env contract 与局部 rule 被人读出，而不是被系统自己统一复述
5. `cleanupOldMessageFilesInBackground()` 已经知道 stronger-request cleanup world 是 plural family dispatch，却仍主要以 hand-written call order，而不是 metadata-driven policy plane 运行

这说明：

`artifact-family cleanup stronger-request cleanup-rationale-governor signer`

和

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

仍然不是一回事。

前者最多能说：

`我能解释为什么不同 carrier family 应该服从不同清理哲学。`

后者才配说：

`系统已经把这些为什么保存成了自己会重复读取、可多平面传播、可防漂移核对的自描述。`

所以 `287` 之后必须继续补的一层就是：

`安全载体家族强请求清理制度理由治理与强请求清理制度元数据治理分层`

也就是：

`stronger-request cleanup-rationale governor 只能解释“为什么”；stronger-request cleanup-metadata governor 才能决定“这份为什么现在究竟被系统放在哪里、谁在读、谁没读、哪里会漂移”。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-metadata-governor signer。`

这里仍是研究命名。
它不是在声称 repo 里已经有一份未接线的统一 registry，
而是在说：

1. stronger-request cleanup family truth 已经大量存在
2. 这些 truth 已经足以构成独立治理问题
3. 但它们仍未稳定长成统一 metadata plane

第二条：

这里说的 `cleanup metadata`

不是在声称：

`当前系统完全没有 metadata。`

更稳的说法只能是：

`当前系统已经有大量局部 metadata signal，但这些 signal 还没有被提升成一套跨 settings、storage、permissions、resume、cleanup 与 host contract 共读的 family grammar。`

第三条：

这里也不是把：

`plansDirectory` propagation gap

当成全部 metadata 问题。

更准确地说，
它只是最硬、最容易被观测到的一种症状。
真正的问题是：

1. task outputs 展示了 local explicit truth
2. tool-results / transcripts 展示了 cross-file puzzle truth
3. plans 展示了 cross-plane propagation failure
4. debug / diagnostics 展示了 argv / env / comment local truth

第四条：

这里尤其不能过度声称：

`只要人类研究者能把多处 helper 拼起来，metadata plane 就已经存在。`

更稳的说法只能是：

`helper composition 可以让研究者重建制度真相，但不等于系统自己已经拥有一份会被多个平面复用的自描述对象。`

第五条：

这里也不能过度声称：

`metadata plane 一定意味着新增一个中心类或一个总表。`

当前更稳的说法只能是：

`关键不在类名，而在 family truth 能否被多平面共读、自动传播并被 drift invariant 核对。`

第六条：

这里还不能把：

`metadata`

偷换成：

`runtime conformance`

即使系统已经真的保存了一份 descriptor，
也还要继续问：

`运行时到底是不是按这份 descriptor 在做事。`

所以 metadata 仍只是比 rationale 更强的一层，
但还不是最强的一层。

## 3. 最短结论

Claude Code 当前研究证据至少给出了七类

`stronger-request cleanup-rationale-governor signer 仍不等于 stronger-request cleanup-metadata-governor signer`

证据：

1. task-output family 的关键 truth 主要活在 `diskOutput.ts` comment + path helper 中，而不是 shared family descriptor
2. tool-results / transcripts 的 truth 分别散落在 file header、path helper 与 persistence helper 中，虽同居一条 constitution，仍无统一 metadata grammar
3. plans family 的 truth 横跨 `plansDirectory` schema、`getPlansDirectory()`、permission allowlist、resume helper 与 cleanup helper，是最硬的 propagation stress test
4. file-history / session-env 的 truth 虽相对稳定，但仍只是各自模块内部的 local restore/replay truth
5. debug logs 的 operational truth 与 diagnostics 的 host-visible no-PII truth 都很强，却仍主要寄居在 argv / env、comment 与 helper 局部世界
6. `cleanupOldMessageFilesInBackground()` 当前更像 hardcoded family dispatch，而不是 metadata-driven family policy plane
7. 当前系统已经展示出 family plurality、rationale plurality 与 partial metadata signals，但还没有展示 equally-strong machine-readable self-description

因此这一章的最短结论是：

`stronger-request cleanup-rationale-governor signer 最多能解释“为什么应当如此”；stronger-request cleanup-metadata-governor signer 才能证明“系统已经把这份为什么保存成了自己会重复读取的同一份 truth”。`

再压成一句：

`有理由，不等于有自描述。`

再压得更哲学一点：

`制度能被研究者解释，不等于制度已经被系统自己记住。`

## 4. 第一性原理：rationale 回答“为什么这条法律成立”，metadata 回答“这份为什么现在究竟住在哪里、如何被系统自己反复读回”

从第一性原理看，
强请求清理制度理由治理与强请求清理制度元数据治理处理的是不同层次的问题。

`stronger-request cleanup-rationale governor`

回答的是：

1. 这个 family 为什么存在
2. 它的 primary risk object 是什么
3. 它的 operational duty 是什么
4. 为什么它配住在 temp、project-session、home-root 或 host-external world
5. 为什么当前 law 没有立刻背叛这项 duty

`stronger-request cleanup-metadata governor`

回答的则是：

1. 上面这些理由被系统保存在哪里
2. 哪些平面消费的是同一份 family truth
3. settings、storage、permissions、resume、cleanup 是否共享同一份 descriptor
4. 一处 family truth 变化后，其他平面会不会自动传播
5. 哪些 drift 现在仍只能靠研究者事后拼出来

如果把两层压成一句
“系统既然已经有理由，所以 metadata 自然也有”，
系统就会制造五类危险幻觉：

1. explained-means-objectified illusion
   只要人能解释，就误以为系统也已经把解释对象化
2. helper-composition-means-policy illusion
   只要几个 helper 能拼成一条故事，就误以为 policy object 已经存在
3. local-comment-means-global-truth illusion
   只要某处注释写得硬，就误以为所有平面都在消费同一份 truth
4. schema-field-means-family-grammar illusion
   只要有 `cleanupPeriodDays` 或 `plansDirectory`，就误以为 family metadata grammar 已经形成
5. no-observed-bug-means-no-gap illusion
   只要现在还能跑，就误以为没有传播断层

所以从第一性原理看：

`rationale` 是制度的意义，
`metadata` 是制度意义被系统自己保存和复读的语法。

## 5. 从技术角度看，这一层恰好揭示了 Claude Code 多重安全技术的先进性与边界

Claude Code 在 stronger-request cleanup 线上最先进的地方，
不是它已经拥有一份完美 metadata registry，
而是它已经展示出一种比“单点沙箱”更成熟的多平面安全技术雏形：

1. 它知道不同 artifact family 面对的是不同 risk object，而不是把所有 cleanup 都压成同一条 retention loop
2. 它已经把部分 family truth 投影到不同技术层：comment、path helper、schema、permission rule、resume helper、env contract、dispatcher
3. 它已经让某些高风险 family 显露出 stronger design intent，比如 task outputs 的 anti-clobber repair、diagnostics 的 host-visible no-PII discipline
4. 它甚至已经出现跨平面 partial alignment，例如 plans family 的 storage / permission / resume world 已经共享一部分 truth

这其实说明 Claude Code 的多重安全技术已经从“单点阻断”走向“分层语义协调”：

1. settings plane 决定哪些 knobs 被正式公开
2. storage plane 决定 family truth 住在哪里
3. permission plane 决定哪些路径在当前 session 被合法看见
4. resume plane 决定哪些旧 truth 值得恢复
5. cleanup plane 决定哪些旧 family 会被背景收口
6. env / host plane 决定哪些 truth 进入宿主可见世界

这比“只有一个 cleaner”要先进得多。

但它的边界也恰恰因此被放大：

`只要多平面开始同时消费 family truth，metadata plane 不存在的代价就会急剧上升。`

换句话说，
Claude Code 的先进性和它的缺口来自同一事实：

`它已经不是一个简单系统。`

正因为它已经拥有多层安全语义，
所以“系统是否把自己的制度理由保存成自描述”才变成一个真正高价值的问题。

## 6. task outputs、tool-results、transcripts 与 diagnostics 先证明：truth 可以很强，但仍然只是 local truth

task-output family 的 non-clobber reason 很强。
它已经不是模糊的“旧文件清理”，
而是明确的：

1. same-project concurrent sessions 不得互相 clobber
2. startup cleanup 不得让 live output reader 读到 ENOENT
3. `/clear` path drift 必须被 repaired temp architecture 吸收

但这份 truth 主要仍保存为：

1. comment
2. path helper
3. 调用方对 helper 的隐式信任

tool-results 与 transcripts 也一样。

它们已经显露出两种不同 duty：

1. tool-results 偏向 persisted inspection
2. transcripts 偏向 continuity / replay

但这份分层仍主要散落在：

1. file header
2. path helper
3. persistence helper
4. cleanup traversal

diagnostics 更明显。
它的 no-PII host-monitoring truth 很硬，
但主要仍保存在：

1. function comment
2. `CLAUDE_CODE_DIAGNOSTICS_FILE` env contract
3. local logfile helper

这三类 family 共同证明了一件事：

`truth can be explicit and still remain local.`

也就是说，
“理由写得清楚”与“系统已经形成 metadata plane”之间，
仍隔着一整层系统自描述工程。

## 7. plans family 再证明：真正暴露 metadata 问题的，不是 local truth，而是 cross-plane truth

plans family 是这一层最硬的工程证据。

因为它的 truth 同时横跨：

1. `plansDirectory` schema
2. storage resolution
3. current-session permission allowance
4. resume recovery
5. cleanup executor

这已经不是 local truth，
而是：

`cross-plane truth`

而当前最有价值的观察正是：

`storage、permissions 与 resume 已经部分承认 configurable plan world；cleanup plane 却仍主要停在 default home-root world。`

这暴露的不是一个偶然 bug，
而是 metadata 缺口最经典的形式：

`同一制度真相已经被多个平面部分消费，但没有一份被所有平面稳定共读的 descriptor。`

这也是为什么 plans family 在安全设计上格外有启发性。

它说明：

1. 安全设计的难点不只是把路径选对
2. 也不只是把权限开对
3. 而是让多平面都知道自己正在服从同一条制度真相

从技术史角度说，
这正是很多成熟安全系统会从 implicit conventions 走向 explicit policy object 的原因。

## 8. file-history、session-env、debug 与 dispatcher 继续证明：系统已经知道 family plurality，但还没有把 plurality 提升成 family grammar

file-history、session-env、debug 与 diagnostics 并不是“没有理由”的家族。
恰恰相反，
它们的理由都很清楚：

1. file-history 服务 restore backup
2. session-env 服务 environment replay
3. debug logs 服务 operational debugging、`/share` 与 bug report
4. diagnostics 服务 host-visible no-PII monitoring

问题不在理由缺失，
而在这些理由目前仍主要以

1. module-local helper
2. comment
3. argv / env contract
4. cleanup helper
5. hardcoded dispatcher

被分布式保存。

`cleanupOldMessageFilesInBackground()` 最能说明这一点。

它已经知道 stronger-request cleanup world 是 plural family dispatch，
也知道不同 cleaner 应按不同 order 被执行。

但它当前显露出来的仍然更像：

`operationally plural executors`

而不是：

`metadata-driven plural family governance`

这说明系统已经看见了 plurality，
但还没有 equally-strong 地把 plurality 保存成 grammar。

## 9. 苏格拉底式自反诘问：我是不是又把“能解释”误当成了“已制度化”

如果要对这一层做更严格的自我审查，
至少要连续追问六句：

1. 如果 task outputs 的 non-clobber truth 已经写得这么清楚，为什么我还说 metadata plane 不存在？
   因为清楚的 local truth 不等于被 cleanup、permissions、settings 与 drift-check plane 共读的 shared truth。
2. 如果 plans family 只是一个 propagation bug，为什么还要上升到 metadata 治理？
   因为真正暴露出来的不是某一处写漏，而是“跨平面真相缺少统一载体”这一更强问题。
3. 如果 diagnostics 已经有 env contract，为什么这还不算 metadata？
   因为 env contract 只说明某一平面知道怎么写，不说明其他平面也能按同一制度真相解释、约束或核对它。
4. 如果我能从 helper 和注释里重建全部 family truth，为什么系统不能算已经拥有自描述？
   因为研究者的重建能力不是系统的自描述能力；前者依赖外部解释者，后者依赖内部可复读对象。
5. 如果当前没有更明显的运行时事故，metadata gap 是否被夸大了？
   不能这样说。absence of failure 只能说明今天还没撞上，不足以证明多平面已经在服从同一条真相。
6. 如果 metadata plane 将来真的形成，是否就等于问题解决了？
   还不够。那时还必须继续追问运行时是否真的按那份 metadata 在做事。

这一串反问最后逼出一句更稳的判断：

`metadata 问题不是“系统有没有留下线索”，而是“系统有没有把线索提升成自己会反复复述的制度真相”。`

## 10. 从安全设计哲学看，这一层真正回答的是“制度有没有自知之明”

`rationale`
更像制度的哲学理由。

`metadata`
更像制度的自知之明。

一个系统可以有制度，
也可以有制度理由，
但如果它不能把这些理由保存成自己会重复读取的对象，
那么这套制度仍然更接近：

`被专家解释的秩序`

而不是：

`被系统自己记住的秩序`

Claude Code 当前 stronger-request cleanup 线的状态，
恰好就停在这个哲学分界点上：

1. 它已经不是一堆无理由的清理函数
2. 它已经显露出多家族、多职责、多世界的制度哲学
3. 但这套哲学还没有被 equally-strong 地提升成系统自己的统一自描述层

所以这一章真正要钉死的，不只是一个工程判断，
也是一个更深的安全设计判断：

`没有自描述的制度，仍然更依赖解释者，而不是依赖制度自身。`

## 11. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-rationale-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-metadata-governor signer`

因为前者最多回答：

`为什么这条 law 值得存在。`

后者才配回答：

`这份为什么现在究竟被系统保存在哪里、哪些平面共读、哪里还只是局部线索、哪里已经开始传播漂移。`

因此这一层的最终哲学边界是：

`有理由，不等于有自描述。`

再压成一句更硬的话：

`制度被说清楚，不等于制度已经学会自己说明自己。`
