# 安全证明链：主体、主权、授权、包络、外部能力与增量审批为何必须同时成立

## 1. 为什么在 `21` 之后还要写“安全证明链”

`21` 已经解释了 Claude Code 现有安全状态面为什么是碎片化的。  
但再往前追一步，还会出现一个更根本的问题：

`系统到底凭什么认为“当前这个动作可以安全发生”？`

如果没有这个答案，  
状态面就只是碎片化展示，  
检测链也只是碎片化拦截。

我认为 Claude Code 背后其实已经隐含了一套更深的结构：

`安全证明链。`

也就是：

`只有当若干独立但彼此关联的证明同时成立，系统才真正拥有“可安全执行”的理由。`

## 2. 最短结论

如果把这套结构压到最短，Claude Code 的安全并不是建立在某一个总开关上，  
而是建立在六条必须同时成立的证明链上：

1. 主体链：你是谁
2. 主权链：谁有资格定义当前运行时
3. 授权链：当前动作是否经过正确仲裁
4. 包络链：即便动作被允许，它还能碰到什么
5. 外部能力链：当前外部入口到底开放了哪一种能力
6. 增量审批链：高风险边界变化是否被重新确认

这六条链里任何一条断掉，  
“当前可以安全执行”这个命题就不成立。

这也是为什么 Claude Code 看起来像有很多独立机制，  
但本质上它们都在回答同一个问题：

`你现在是不是已经完成了足够多的证明，足以让系统承担这次执行。`

## 3. 第一条证明链：主体链

### 3.1 主体链证明的不是“有人在用”，而是“当前主体可被系统识别”

主体链最基本的问题是：

`现在做动作的，到底是谁。`

在 Claude Code 里，这条链并不是抽象存在。  
它至少会被投影到：

- `utils/status.tsx` 的账户与 token source 状态
- `bridge/bridgeEnabled.ts` 的 subscription / profile scope / organization UUID 判断

这说明系统真正关心的不是“有没有登录”，  
而是：

1. 这是哪一类主体
2. 它拿着哪一类凭据
3. 它是否具备被后续资格链继续识别的条件

### 3.2 主体链一旦不成立，后面的高阶能力就没有合法承载体

`getBridgeDisabledReason()` 最能说明这一点。  
它不是只检查一个 gate，  
而是连续检查：

1. 是否是 claude.ai subscription
2. 是否有 full profile scope
3. 是否能拿到 organization UUID
4. 是否真的通过远端 gate

这说明 bridge entitlement 的真正前提不是“用户想用”。  
而是：

`系统能不能先证明当前主体是一个可被正确识别、可被正确归属、可被正确授予资格的主体。`

因此主体链的本质不是身份装饰，  
而是：

`整个安全控制面的承载前提。`

## 4. 第二条证明链：主权链

### 4.1 主权链证明的不是“配置存在”，而是“当前运行时到底由谁说了算”

主权链对应的不是权限按钮，  
而是：

`谁有资格定义当前运行时。`

这条链至少横跨：

- `settings.ts`
- `managedEnv.ts`
- `pluginOnlyPolicy.ts`
- `mcp/config.ts`

系统在这里持续追问：

1. 当前值来自哪个 source
2. 这个 source 是否有资格影响该表面
3. 是否存在 managed-only / plugin-only / host-managed 的最终收口

### 4.2 主权链不成立时，后面所有“授权”都可能发生在错误前提下

这一点非常容易被低估。  
很多系统会以为只要最后 ask/allow 做对了就行。  
Claude Code 的源码恰恰相反，它一再说明：

`如果低信任来源已经改写了主链运行时，后面的授权判断就是在错误世界里做出的正确判断。`

所以主权链比授权链更前。  
不是因为它更“高级”，  
而是因为它定义了：

`授权到底在谁设定的现实里发生。`

### 4.3 从第一性原理看，主权链是“现实定义权”的证明

主体链回答“你是谁”，  
主权链回答“谁在定义你现在所处的运行现实”。

没有这条链，  
安全系统就会悄悄退化成：

`谁能写到配置，谁就能改写边界。`

Claude Code 显然不接受这一点。

## 5. 第三条证明链：授权链

### 5.1 授权链证明的是“这次动作是否经过了正确仲裁”

授权链主要由：

- `permissionSetup.ts`
- `permissions.ts`
- `shadowedRuleDetection.ts`

来支撑。

这条链真正关心的不是“有没有 allow”，  
而是：

1. 规则本身是否合法
2. 规则是否可达
3. 当前 mode 是否有资格改变仲裁强度
4. `safetyCheck` 是否先于便利路径
5. classifier 是否只能接手可接手的 ask

### 5.2 授权链不是在证明“动作安全”，而是在证明“动作被正确裁决”

这点非常重要。  
Claude Code 不把授权链等同于风险评估链。  
它更像是在证明：

`这个动作已经走过了本该走的仲裁顺序，没有在中途偷偷绕开。`

这就是为什么危险 allow rule 会在 auto mode 入口前被剥离，  
也是为什么 shadowed rule 会被单独治理。

授权链的本质不是告诉你“动作绝对安全”，  
而是告诉你：

`系统没有在程序结构上被提前架空。`

## 6. 第四条证明链：包络链

### 6.1 包络链证明的是“即便允许动作，它实际还能碰到什么”

授权通过，不等于动作无限自由。  
真正执行前，还要问：

`它被允许以后，实际会落在怎样的环境包络里。`

这条链主要体现在：

- `pathValidation.ts`
- sandbox 相关能力
- `SandboxConfigTab.tsx`

它处理的不是“可不可以执行”，  
而是：

1. 路径实际能碰到哪些对象
2. 哪些写目标会被 safetyCheck 拦住
3. 文件系统包络和网络包络具体长什么样
4. 已允许动作是否仍被 sandbox 限制在某个壳层里

### 6.2 包络链是对授权链的第二次现实收口

这一点很关键。  
授权链证明的是“你可以做这个动作”。  
包络链证明的是：

`你可以做这个动作，但只能在这一层壳里做。`

这意味着 Claude Code 的安全不是一次性 yes/no。  
而是：

`先决定动作是否合法，再决定合法动作实际能扩散到哪里。`

### 6.3 从第一性原理看，包络链在证明“破坏半径”

主体链证明身份，  
主权链证明现实定义权，  
授权链证明仲裁顺序，  
包络链则在证明：

`哪怕你现在能动，破坏半径是否仍被收口。`

这正是沙箱和路径安全检查存在的更深理由。

## 7. 第五条证明链：外部能力链

### 7.1 外部能力链证明的不是“能不能连”，而是“当前到底开放了哪一种外部能力”

这条链贯穿：

- `WebFetchTool/preapproved.ts`
- `mcpValidation.ts`
- `mcp/config.ts`
- `PluginSettings.tsx`
- bridge 相关 entitlement

Claude Code 在这里没有采用“外部能力”这个粗标签。  
它更像在分开证明：

1. 当前是读取型能力，还是写/连/执行型能力
2. 当前是 host 信任，还是 name-only 伪装
3. 当前只是连接建立，还是连同输出污染半径一起被接受
4. 当前连接定义权归谁

### 7.2 这条链的哲学本质是“能力语义证明”

很多系统会说：

`这个入口是可信的。`

Claude Code 更像在问：

`这个入口当前到底在开放哪一种能力语义。`

也就是说，它不轻易证明“入口整体可信”，  
它只证明：

`在当前边界内，这种特定能力语义是否成立。`

这就是为什么：

- WebFetch 的读取白名单不会自动升级成网络写白名单
- MCP 不能只按名字治理
- 输出预算会被当成安全问题

## 8. 第六条证明链：增量审批链

### 8.1 增量审批链证明的是“边界变化已经被重新确认”

前面几条链，更多是在回答静态或当前状态的问题。  
增量审批链回答的是另一个问题：

`如果边界正在变化，系统有没有重新取得同意。`

这里最明显的源码就是：

- `remoteManagedSettings/securityCheck.tsx`
- `ManagedSettingsSecurityDialog.tsx`

系统不会因为 managed settings 一直危险就一直弹。  
它只会在：

`危险设置发生新增或变化`

时要求重新确认。

### 8.2 这条链证明的是“边界漂移的合法性”

这比单纯的“第一次接受条款”更成熟。  
因为系统知道：

`真正危险的，不是你一直活在已知边界里，而是边界在你没重新表态的情况下被改写。`

所以增量审批链本质上是在证明：

`新的边界不是偷偷长出来的，而是经过重新确认后进入系统。`

### 8.3 从第一性原理看，增量审批链在证明“边界更新的正当性”

如果主体链证明“谁在这里”，  
主权链证明“谁定义现实”，  
授权链证明“动作如何被裁决”，  
包络链证明“破坏半径”，  
外部能力链证明“当前到底开放了哪种能力”，

那么增量审批链证明的就是：

`今天的边界和昨天不一样时，这个差异是不是正当进入系统的。`

## 9. 为什么这六条链必须同时成立

因为它们回答的是六个不同的问题，  
任何一条都不能替代另一条。

可以把它压成这样：

```text
主体链：谁在这里
主权链：谁定义现实
授权链：动作如何被裁决
包络链：允许后还能扩散到哪里
外部能力链：到底开放了哪一种外部能力
增量审批链：边界变化是否被重新确认
```

少掉任何一条，都会产生系统性盲区：

1. 没有主体链，资格无法归属
2. 没有主权链，授权在错误前提下发生
3. 没有授权链，动作可能绕过仲裁
4. 没有包络链，允许后破坏半径失控
5. 没有外部能力链，读取/连接/执行语义被混写
6. 没有增量审批链，边界漂移会悄悄进入系统

因此“安全执行”不是某个按钮给出来的，  
而是：

`六条证明同时成立后，系统才有资格承担这次执行。`

## 10. 这套证明链的先进性在哪里

我会给出八条判断：

1. Claude Code 不把安全归约成单一 allow/deny 结果。
2. 它知道身份、主权、授权、包络、外部能力和增量确认是不同维度。
3. 它不允许 mode 或 classifier 单独冒充整个安全证明。
4. 它知道 source governance 比 value correctness 更前置。
5. 它把包络控制视为授权后的第二层证明，而不是可选附件。
6. 它把外部能力证明做成能力语义证明，而不是接口标签证明。
7. 它把边界更新单独做成增量审批，而不是靠一次初始同意吃一辈子。
8. 它已经拥有这些证明链的大部分零件，只是还没把它们统一可视化。

这就是为什么 Claude Code 的安全性显得复杂却不混乱。  
它复杂，是因为它拒绝把不同证明维度偷懒压成一个总开关。

## 11. 从第一性原理看，安全系统到底在证明什么

如果把这一章再压到最深一层，安全系统真正要证明的不是：

`这个动作是不是看起来危险。`

而是：

`当前主体、当前现实、当前裁决、当前包络、当前外部能力和当前边界更新，是否共同构成了一个仍然可承担后果的执行场景。`

这比“危险动作检测”高一个层级。  
因为它在证明：

`系统现在是否仍然处在一个可治理、可归因、可收口、可解释的执行世界。`

## 12. 苏格拉底式追问

### 问：如果授权链成立，但主权链不成立，会怎样？

答：系统只是“在错误的现实里做了正确裁决”。

### 问：如果主权链成立，但包络链缺失，会怎样？

答：动作虽然合法，但破坏半径没有被控制。

### 问：如果包络链成立，但外部能力链缺失，会怎样？

答：系统可能把读取、连接、执行、身份等不同能力语义混成一团。

### 问：如果前五条都成立，但增量审批链缺失，会怎样？

答：边界漂移会以“已经同意过”为借口静默进入系统。

### 问：为什么说这套系统还没完全产品化？

答：因为证明链已经隐含存在，但还没有以统一视图暴露给用户。

## 13. 本章结论

Claude Code 的安全设计，最终不应被理解成：

- 一套规则
- 一个沙箱
- 一个 classifier
- 一堆状态页

而应被理解成：

`一套由主体链、主权链、授权链、包络链、外部能力链和增量审批链共同组成的安全证明系统。`

这也是它真正先进的地方。

它不是在问：

`这个动作能不能做？`

它更深地在问：

`系统现在是否已经完成了足够多的证明，足以让这次动作在可治理、可收口、可归因的边界内发生。`
