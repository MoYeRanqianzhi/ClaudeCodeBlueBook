# 安全工程否决权宪法：为什么缺口信号、未证事实与能力上限必须压过build、check与smoke的正向信号

## 1. 为什么在 `120` 之后还必须继续写“否决权宪法”

`120-安全工程联合签字协议` 已经回答了：

`更强结论只有在 required evidence set 已齐、minimum signer quorum 已满足、且 veto signal 未触发时，才允许升级。`

但如果继续追问，  
还会碰到一个更根本的问题：

`为什么 veto signal 的优先级必须这么高？`

换句话说：

`为什么缺口、未证事实、能力上限这些“负信号”，必须压过一串正向信号？`

因为如果这一层不继续说清楚，  
团队很容易在情绪上滑回旧习惯：

1. `build` 成了
2. `check` 过了
3. `smoke` 跑了

然后就会本能地觉得：

`这些正向信号加起来，怎么也该比那几个负信号更重要吧。`

但安全工程恰恰相反。  
在高不确定性系统里，  
真正决定结论上限的往往不是正向信号有多少，  
而是：

`还有哪些关键事实仍然未被证明，或已被明确证明为缺失。`

所以 `120` 之后必须继续补出的下一层就是：

`安全工程否决权宪法`

也就是：

`在强结论的升级过程中，某些缺口信号、未证事实与能力上限必须拥有比局部正向结果更高的优先级。`

## 2. 最短结论

对这份研究版 Claude Code 源码来说，  
最容易被误读的不是单个信号，  
而是：

`信号之间的优先级。`

从当前仓库的源码与结构看，  
至少有三类高优先级 veto：

1. `README.md:72-74` 明确签出 source incomplete 与 missing modules 不可恢复
2. `QUICKSTART.md:60-70` 明确签出 Bun intrinsics 与 missing modules 造成的公开构建能力上限
3. `package.json:7-11` 加上本地检查 `.github dirs: 0`、`test/spec files: 0` 明确签出 verify/test/CI 宿主缺失

这些信号一旦出现，  
就会直接否决下列升级：

1. `best-effort build -> 官方等价构建`
2. `check + smoke -> 系统已验证`
3. `局部工程入口存在 -> 工程化闭环已成立`

所以这一章的最短结论是：

`安全工程里，负信号不是附注，而是宪法级约束。`

再压成一句：

`只要关键缺口仍在，正向信号再多，也只能在缺口之下说话。`

## 3. 第一性原理：为什么 veto 必须高于正向信号

因为正向信号通常证明的是：

`某件事在某个局部视角下成立。`

而 veto signal 证明的是：

`更强结论所必需的一块前提根本不存在，或者仍未被证明。`

这两者不是同一层命题。

例如：

1. `build` 成功证明 artifact exists
2. `smoke` 成功证明 artifact can minimally launch

但如果同时存在：

1. missing modules 无法恢复
2. Bun 语义无法完整复现
3. verify/tests/CI 闭环缺失

那这三个缺口并不是“另外三个普通事实”，  
而是直接在说明：

`某些更强结论的前提条件尚未满足。`

所以从第一性原理看，  
veto signal 的本质不是悲观，  
而是：

`限制错误升级。`

## 4. 当前仓库里的三类宪法级 veto

## 4.1 对象缺口 veto：`source is incomplete`

`README.md:72-74` 写得非常硬：

1. this source is incomplete
2. 108 modules are not included
3. they cannot be recovered from published artifacts

这类信号一旦成立，  
就直接否决了三种强结论：

1. “公开仓可以逐步补成完整产品”
2. “当前 build 成功意味着核心语义已经被完整保留”
3. “缺失模块只是普通待实现项”

为什么它优先级这么高？

因为它不是某个局部步骤失败，  
而是在定义：

`对象本身的上限。`

对象上限若不先被尊重，  
后面所有局部成功都会被系统性误读。

## 4.2 能力上限 veto：`cannot be fully replicated with esbuild`

`QUICKSTART.md:60-70` 继续把上限钉死：

1. Bun compile-time intrinsics cannot be fully replicated
2. 108 missing modules still exist as hard gap
3. generated type files are also incomplete

这意味着即便：

1. `prepare-src` 正常
2. `build` 正常
3. `start` 正常

也仍然不能升级到：

`官方等价构建`

因为能力上限 veto 说的是：

`你当前这条路径的上限本来就不是 full-fidelity rebuild。`

所以它否决的不是局部动作，  
而是：

`过满的命名。`

## 4.3 验证缺口 veto：`no verify script / no tests / no CI host`

`package.json:7-11` 当前只承认：

1. `prepare-src`
2. `build`
3. `check`
4. `start`

再加上本地检查：

1. `.github dirs: 0`
2. `test/spec files: 0`

这三者联合起来构成另一类更直接的 veto：

`当前正式验证闭环不存在。`

只要这一 veto 仍然存在，  
以下升级都必须被打回：

1. “当前系统已验证”
2. “工程化闭环已成立”
3. “当前行为已被正式测试体系守住”

因为这里缺的不是小优化，  
而是：

`formal verification host`

## 5. 为什么 `build/check/smoke` 必须在 veto 之下发言

这是这一章最核心的工程纪律。

`build/check/smoke` 都是有价值的正向信号，  
但它们的价值必须被限定在：

1. `build` 只证明产物存在
2. `check` 只证明当前静态配置下可通过
3. `smoke` 只证明最小启动成立

一旦更高层 veto 已经存在，  
它们就不能继续升级自己的语义。

所以正确写法永远应该是：

1. `build` 成功，但对象缺口 veto 仍在
2. `check` 通过，但验证缺口 veto 仍在
3. `smoke` 成立，但能力上限 veto 仍在

而不是：

1. `build` 成功，所以闭环差不多了
2. `check` 通过，所以系统大体正确
3. `smoke` 成立，所以已经可视作“已验证”

这就是为什么我说：

`正向信号必须在 veto 之下说话。`

## 6. 技术先进性：真正成熟的安全工程，不只是定义升级条件，还定义“谁能一票否决”

很多团队的工程治理只做到两层：

1. 什么信号算正向
2. 什么条件下允许升级

但更成熟的系统还会做到第三层：

`明确定义哪些信号拥有 veto power。`

这才是高级治理。

因为如果没有 veto power，  
系统就会默认：

`所有信号都是加分项。`

而安全工程里并不是这样。  
有些信号不是加分项，  
而是：

`天花板`

`刹车`

`红线`

从这个角度看，  
Claude Code 这份研究版源码最重要的启示是：

`能力上限、对象缺口、验证闭环缺失，这些都不应被写成“待补充说明”，而应被写成“当前结论升级的宪法性约束”。`

## 7. 苏格拉底式反思：如果我看到很多正向信号，我该先怎样质问自己

可以先追问八句：

1. 当前有没有任何对象缺口 veto 仍然存在
2. 当前有没有任何能力上限 veto 仍然存在
3. 当前有没有任何验证闭环 veto 仍然存在
4. 我是不是在把局部成功错当成整体升级许可
5. 我是不是把“未证明”默默当成了“暂时视为成立”
6. 如果把所有 veto 逐条写出来，这个更强结论还站得住吗
7. 当前这句话是建立在正向证据上，还是建立在对缺口的遗忘上
8. 若把表达改成“在以下 veto 仍存在的前提下”，这句话还是否成立

这些问题最终都会逼出一条硬纪律：

`安全工程不是先看什么成功了，而是先看什么仍然不允许你说得更强。`

## 8. 一条硬结论

对这份 Claude Code 研究版源码来说，  
真正稳健的安全研究方法不是：

`把所有正向信号列出来，再看能不能乐观升级，`

而是：

`先把所有宪法级 veto 列出来，再让正向信号在这些 veto 之下说话。`

因为只有这样，  
这份蓝皮书才不会滑向：

`局部信号很多，整体结论过满`

而会继续升级成：

`由对象边界、能力上限与验证缺口共同约束的安全工程研究。`
