# 安全动作归属：为什么正确下一步还必须绑定正确主体、作用域与持久层

## 1. 为什么在 `40` 之后还要继续写“动作归属”

`40-安全动作语法` 已经回答了：

`成熟控制面不能只给状态和理由，还必须给出正确下一步动作。`

但如果继续往第一性原理再压一层，  
还会剩下一个更关键的问题：

`这条动作到底该由谁来做，写到哪里，以及它会持续多久？`

换句话说，  
即使系统已经知道下一步应该是：

1. 登录
2. 等待重连
3. 审批允许
4. 修改权限规则
5. 回滚文件

它仍然还缺三件事：

1. 这条动作属于系统、宿主、用户还是管理员
2. 这条动作应该写进 `session`、`cliArg`、`localSettings`、`projectSettings` 还是 `userSettings`
3. 这条动作是瞬时的、会话级的，还是持久化的

Claude Code 的源码在很多地方都在悄悄维护这三件事。  
它不是只在说：

`该做什么。`

它更在说：

`谁有资格做、该把结果写到哪一层、以及这条动作能不能越过当前边界长期生效。`

所以这一章的核心问题是：

`Claude Code 为什么不满足于把“正确动作”写出来，而还要继续把动作的主体、作用域与持久层编码进去？`

## 2. 最短结论

从源码看，Claude Code 的安全控制面依赖另一条更深的纪律：

`正确动作如果没有绑定正确主体、正确写入作用域和正确持久层，就仍然是不安全动作。`

这至少体现在五类地方：

1. 权限更新显式区分 `userSettings / projectSettings / localSettings / session / cliArg`
2. 只有部分目的地支持持久化，`session` 和 `cliArg` 不能被偷偷写成长期规则
3. 权限审批输出只允许 `allow / deny`，并要求宿主准确反映用户到底是临时允许、永久允许还是拒绝
4. `needs-auth / pending / failed / disabled / policy` 对应的修复主体并不相同
5. `api_retry`、`rewind dry_run`、组织校验失败等都在区分“系统该自动做什么”和“必须由人来做什么”

所以这一章的最短结论是：

`Claude Code 的安全性不只来自它会给动作，还来自它会把动作和责任归属一起编码。`

## 3. 什么叫“安全动作归属”

这里说的“动作归属”，不是产品经理意义上的流程负责人。  
我在这一章里的定义是：

`系统把一条安全相关动作与其合法执行主体、允许写入的配置层，以及应该持续多久，一起编码成可约束的控制面语义。`

一个成熟的动作归属至少要回答四件事：

1. 这条动作是谁来执行
2. 这条动作写入哪一层
3. 这条动作会持续多久
4. 哪个主体最不应该越权替别人执行这条动作

如果缺这四句，  
系统就很容易出现一种比“动作给错了”更隐蔽的失败：

`动作本身没错，但被错误主体在错误层级上执行了。`

## 4. 第一类归属：权限修复不只是“改规则”，而是必须写到正确目的地

## 4.1 `PermissionUpdateDestinationSchema` 直接把写入层做成协议字段

`src/entrypoints/sdk/coreSchemas.ts:242-299` 很关键。  
`PermissionUpdateDestinationSchema` 并不是可有可无的附注，  
而是明确枚举出五种目标层：

1. `userSettings`
2. `projectSettings`
3. `localSettings`
4. `session`
5. `cliArg`

而 `PermissionUpdateSchema` 的每一种更新动作，  
无论是：

1. `addRules`
2. `replaceRules`
3. `removeRules`
4. `setMode`
5. `addDirectories`
6. `removeDirectories`

都必须同时带 `destination`。

这说明作者不接受一种粗糙模型：

`只要动作是“加一条 allow 规则”，它写到哪里都差不多。`

Claude Code 的源码更接近另一种成熟认识：

`同一个动作名，写到不同目的地，就是不同安全动作。`

## 4.2 这意味着“动作内容”还不够，必须继续问“写到哪”

例如：

1. 把规则写进 `session` 是一次性放宽
2. 写进 `cliArg` 是命令行上下文里的局部覆盖
3. 写进 `localSettings`、`projectSettings`、`userSettings` 则会改变更长时间、更大范围的运行时

也就是说，  
真正的安全语义不是：

`allow this tool`

而是：

`谁允许、允许多久、影响到哪一层上下文。`

## 5. 第二类归属：并不是每个目的地都允许持久化

## 5.1 `supportsPersistence()` 明确把可持久层与瞬时层分开

`src/utils/permissions/PermissionUpdate.ts:208-223` 很关键。  
这里 `supportsPersistence()` 只接受：

1. `localSettings`
2. `userSettings`
3. `projectSettings`

并且 `persistPermissionUpdate()` 一开始就直接做：

`if (!supportsPersistence(update.destination)) return`

这说明作者非常清楚：

`不是每一个动作目的地都配拥有持久化资格。`

## 5.2 这是一条非常深的安全边界

因为如果系统把：

1. `session`
2. `cliArg`

这种本来应当短期生效的动作，  
偷偷升级成长期配置写入，  
那临时例外就会被错误固化成结构性放宽。

所以这里真正被保护的不是“写盘”这件小事，  
而是：

`动作承诺的时间边界。`

也就是说，  
Claude Code 的动作归属不只绑定“谁做”，  
还绑定：

`这条动作允许持续多久。`

## 6. 第三类归属：宿主并不是动作主体，而是用户决策的见证与转译层

## 6.1 `PermissionPromptTool` 的输出非常克制

`src/utils/permissions/PermissionPromptToolResultSchema.ts:65-77` 说明，  
宿主对权限请求的正式输出只有两大类：

1. `allow`
2. `deny`

同时可带：

1. `updatedPermissions`
2. `decisionClassification`
3. `interrupt`

这说明宿主并不是一个可以自由发挥的解释者。  
它更像是一个被严格约束的动作转译器。

## 6.2 `PermissionDecisionClassificationSchema` 继续要求宿主忠实报告用户到底做了什么

`src/entrypoints/sdk/coreSchemas.ts:301-313` 把分类写成：

1. `user_temporary`
2. `user_permanent`
3. `user_reject`

并明确说明：

`SDK hosts that prompt users ... should set this to reflect what actually happened`

这句很重要。  
它说明宿主不能把用户的一次临时允许，  
伪装成永久允许；  
也不能把永久允许，  
伪装成一次性点击。

所以这里更深的结论是：

`权限动作真正的主体是用户，宿主负责如实上报，而不是代替用户重写承诺级别。`

## 6.3 这背后的哲学是“见证层不能冒充授权层”

很多系统在宿主 UI 层最容易犯的错就是：

`用户点了一次通过，宿主顺手把它写成了长期规则。`

Claude Code 的 schema 设计恰恰在防这个。  
它要求把：

1. 行为本身
2. 用户承诺强度
3. 可能的权限更新

都显式编码出来。

这意味着：

`宿主最多是授权动作的见证者和转译者，而不应升级成授权的真实拥有者。`

## 7. 第四类归属：同样是“不能继续”，不同状态对应的责任主体完全不同

## 7.1 `pending` 是系统责任，不是用户责任

`src/services/mcp/types.ts` 与  
`src/entrypoints/sdk/coreSchemas.ts:167-193`  
把 MCP 状态明确拆成五种。  
其中 `pending` 还会带：

1. `reconnectAttempt`
2. `maxReconnectAttempts`

这说明 `pending` 的语义不是：

`用户该开始乱点修复。`

而是：

`系统已经进入自动恢复路径，当前动作主体首先是系统自己。`

如果此时宿主把 `pending` 压成 generic failure，  
就会把系统责任错误转嫁给用户责任。

## 7.2 `needs-auth` 是用户责任，不是系统责任

`src/services/mcp/useManageMCPConnections.ts:596-603`  
把 auth gate 的提示明确写成：

`Channels require claude.ai authentication · run /login`

这里动作主体非常清楚：

`是用户去登录。`

系统可以检测、提示、缓存状态，  
但它不能替用户完成登录。

## 7.3 `policy` 是管理员责任，不是普通用户责任

同一段源码又把 policy gate 写成：

`have an administrator set channelsEnabled: true in managed settings`

这比一般错误提示成熟得多。  
因为它没有让普通用户去做一条自己根本无权做的动作。

换句话说，  
它不是在说：

`功能不可用。`

而是在说：

`这条修复路径的合法主体不是你，而是管理员。`

## 7.4 `disabled` 甚至可能没有可执行修复主体

`gate.kind === 'disabled'` 时，  
源码只给出：

`Channels are not currently available`

这说明某些状态下，  
系统故意不伪造一个“你现在就能做点什么”的动作。

也就是说，  
成熟控制面不仅要指出可行动作，  
还要在没有合法主体时承认：

`当前不存在你可执行的修复动作。`

## 8. 第五类归属：有些动作属于系统自动执行，有些必须停在人工审查

## 8.1 `api_retry` 明确属于系统自动动作

`src/entrypoints/sdk/coreSchemas.ts:1572-1587` 与  
`src/QueryEngine.ts:943-954` 都说明，  
`api_retry` 不是建议动作，  
而是系统已经决定要做的动作：

1. 第几次重试
2. 最多重试几次
3. 多久后重试

这里用户不是主动作执行者。  
真正的主体是运行时本身。

因此如果宿主看到 `api_retry` 后，  
又额外引导用户立刻手工重复提交，  
就等于把系统动作和用户动作错误叠加。

## 8.2 `rewind_files` 则故意把人工确认保留下来

`src/entrypoints/sdk/controlSchemas.ts:308-327`  
和 `src/cli/print.ts:2996-3010` 很关键。  
`rewind_files` 不仅有 `canRewind`，  
还有 `dry_run` 路径。

这说明作者并不想让“回滚能力存在”自动等于“现在就执行回滚”。  
系统先允许：

1. 预演
2. 看 `filesChanged`
3. 再决定是否真正执行

`src/components/MessageSelector.tsx:313` 更进一步说明：  
只有 `filesChanged.length > 0` 才允许 code restore 动作。

这说明回滚链条的动作主体并不是纯系统，  
而是：

`系统提供能力与预演，用户承担最终确认。`

## 9. 第六类归属：同一类“认证失败”也会继续区分不同的人类动作

`src/utils/auth.ts:1958-1999` 很有代表性。  
`OrgValidationResult` 的失败 message 并没有把所有情况都压成一句：

`Please log in again`

它继续做了分流：

1. profile 拉取失败：`Try again, or obtain a full-scope token via 'claude auth login'.`
2. 环境变量 token 组织不匹配：`Remove the environment variable or obtain a token for the correct organization.`
3. 普通组织不匹配：`Please log in with the correct organization: claude auth login`

这说明它不仅在绑定动作，  
还在绑定：

`人类该对哪一个控制点动手。`

例如：

1. 有的是重试网络
2. 有的是清理环境变量
3. 有的是重新登录正确组织

三者看起来都像 auth 问题，  
但动作主体和动作入口已经完全不同。

## 10. 把这些例子压成一条底层公理

我会把这一章压成一句话：

`安全动作如果没有绑定正确主体、正确作用域和正确持久层，就仍然不是成熟动作。`

这条公理几乎能统一前面所有例子：

1. 权限更新要写到对的 destination
2. 会话级例外不能偷偷变成持久规则
3. 宿主不能把用户临时允许升级成永久允许
4. `pending`、`needs-auth`、`policy`、`disabled` 分别属于不同主体
5. 自动重试和人工回滚确认不能混成一条动作
6. 认证失败的修复入口也必须继续分流

所以在 `40` 之后，  
安全控制面的更深层真相已经不只是：

`状态 -> 动作`

而是：

`状态 -> 动作 -> 主体 -> 写入层 -> 持续时间`

## 11. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 把动作责任本身做成了控制面的一部分，而不是留给下游产品经理和宿主 UI 临时发挥。`

这会带来几个很强的工程收益：

1. 临时允许与永久允许不会被宿主轻易混写
2. 会话级例外不容易偷偷泄漏成长期策略
3. 管理员动作不会被误派发给普通用户
4. 系统自动恢复不会被误包装成人工操作建议
5. 权限和认证修复可以围绕同一套 actor/scope 语言设计

这对其他 Agent 平台构建者的启示也很直接：

1. 设计动作字段时，一定要把 actor 一起设计进去
2. 设计动作字段时，一定要把 destination / scope 一起设计进去
3. 设计动作字段时，一定要把 persistence horizon 一起设计进去
4. 不要让宿主把“见证用户操作”偷偷升级成“替用户做长期授权”
5. 不要把系统恢复动作、用户修复动作和管理员配置动作混成同一类按钮

## 12. 哲学本质

这一章更深层的哲学是：

`安全边界不仅是能力边界，也是责任边界。`

很多系统只关心：

`谁能调用哪个能力。`

Claude Code 的源码更接近另一种成熟认识：

`谁能修改边界、谁能承诺更长时间、谁只能临时批准、谁只能等待系统自己恢复，这些也都属于安全问题。`

这意味着控制面的成熟，  
不只在于它知道下一步该做什么，  
还在于它知道：

`谁不该替别人做。`

从第一性原理看，  
这其实是在保护一条很根本的公理：

`如果错误主体可以执行正确动作，控制面最终仍会被掏空。`

因为那意味着：

1. 宿主可以偷改授权
2. 临时动作可以越权持久化
3. 普通用户会被诱导去做管理员动作
4. 系统不可修复状态会被包装成伪可修复状态

## 13. 苏格拉底式反思：这一章最容易犯什么错

### 13.1 我们是不是把一些 destination 和文案过度上纲成哲学原则

有风险。  
但这里不是个别命名巧合，  
而是 schema、持久化函数、通知文案、重试消息和 UI 开关一起反复收敛到同一个结构。

### 13.2 “动作归属”会不会只是产品设计问题，不算安全问题

不会。  
因为只要错误主体能越权替别人执行动作，  
安全边界就会被真实改写。


## 14. 结语

`40` 回答的是：成熟控制面必须把状态和动作绑在一起。  
这一章继续推进后的结论则是：

`成熟控制面还必须把动作与正确主体、正确作用域和正确持久层绑在一起。`

这意味着 Claude Code 更深层的安全启示之一是：

`真正可靠的控制面，不只是会告诉你做什么，还会约束谁能做、写到哪、以及这条动作能持续多久。`
