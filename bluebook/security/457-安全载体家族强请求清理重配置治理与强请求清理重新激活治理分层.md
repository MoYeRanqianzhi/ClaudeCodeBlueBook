# 安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层：为什么artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reactivation-governor signer

## 1. 为什么在 `456` 之后还必须继续写 `457`

`456-安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层`
已经回答了：

`stronger-request cleanup 线即便知道 returned object 现在应按哪组 current config truth 重新工作，也仍然还需要 reconfiguration governor 去决定这些 truth 长什么样。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
`配置写好了就说明当前运行世界已经按它工作了`
的错觉：

`只要 stronger-request cleanup reconfiguration governor 已经决定了 current options、current secrets、current payload，它就自动拥有了决定这些 truth 何时进入 active session、何时真的接管 commands / agents / hooks / MCP / LSP 的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 在 plugin 线已经成熟写出的 reactivation 正对照，
会发现它其实明确拆开了五件事：

1. `persisted truth`
2. `active truth`
3. `mode-sensitive policy`
4. `take-effect contract`
5. `stale signal`

最硬的证据至少有八组：

1. `src/commands/plugin/ManagePlugins.tsx:1645-1668,1700-1707`
   明确把 `Configuration saved` 与 `take effect` 拆成两层句法
2. `src/utils/plugins/refresh.ts:1-17,59-79,123-145`
   把 `refreshActivePlugins()` 直接命名为 `Layer-3 refresh primitive: swap active plugin components in the running session`
3. `src/commands/reload-plugins/reload-plugins.ts:10-56`
   把 `/reload-plugins` 写成正式 `Reloaded: ...` 契约，
   说明 reactivation 可以是一个单独的用户命令主权
4. `src/hooks/useManagePlugins.ts:287-303`
   明确写出 `Do NOT auto-refresh. Do NOT reset needsRefresh`
5. `src/services/plugins/PluginInstallationManager.ts:52-59,135-180`
   区分 `new installs -> auto-refresh` 与 `updates only -> needsRefresh`
6. `src/state/AppStateStore.ts:209-215`
   直接把 `needsRefresh` 注释成
   `interactive mode, user runs /reload-plugins to consume; headless mode, refreshPluginState() auto-consumes via refreshActivePlugins()`
7. `src/cli/print.ts:1749-1769`
   以 `refreshPluginState()` 明示 headless 的 Layer-3 auto-consume path
8. `src/cli/print.ts:3071-3080`
   又在 control-request reload 路径上先 re-pull user settings，再跑 `refreshActivePlugins()`，
   说明 remote/headless take-effect 也有独立 choreography

这说明：

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

仍然不是一回事。

前者最多能说：

`对象现在该按哪组 current config truth 工作。`

后者才配说：

`这组 current config truth 现在是否已经真正接管运行中的 current session。`

所以 `456` 之后必须继续补的一层就是：

`安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层`

也就是：

`stronger-request cleanup reconfiguration governor 负责改写 current config truth；stronger-request cleanup reactivation governor 才负责决定这组 truth 何时真正进入 active world。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reactivation-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-reactivation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reactivation manager，
而是在说：

1. repo 已经把 Layer 3 明确命名为 active components world
2. repo 已经把 `Configuration saved` 与 `take effect` 拆开
3. repo 已经把 interactive mode、background install 与 headless mode 的激活路径做成不同制度

第二条：

这里说的 `reactivation governance`

不是在声称：

`一切配置变化都必须立刻自动生效。`

当前更稳的说法只能是：

`reactivation governance 首先回答的是谁配触发 active takeover、在哪个 mode 下触发、触发失败后怎样降级。`

第三条：

这里也不是把：

`/reload-plugins`

单独当成全部 reactivation grammar。

真正关键的是：

1. Layer-3 refresh primitive
2. interactive stale-state discipline
3. new install 与 update 的不同激活策略
4. headless auto-consume path

第四条：

这里尤其不能过度声称：

`配置已保存`

就等于：

`当前运行世界已经按这组 persisted truth 运转`。

当前更稳的说法只能是：

`repo 明确把 persisted truth 与 active truth 拆开；后者还要经过显式接管 choreography。`

第五条：

这里也不能过度声称：

`同一类 config change 在所有模式下都应采用同一种 reactivation policy。`

当前更稳的说法只能是：

`reactivation policy 本身就是 mode-sensitive governance，而不是单一自动规则。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-reconfiguration-governor signer 仍不等于 stronger-request cleanup-reactivation-governor signer`

证据：

1. `ManagePlugins.tsx` 明确写出 `Configuration saved. Run /reload-plugins for changes to take effect.`，
   说明 config persistence 不自动等于 active effect
2. `refresh.ts` 直接把 `refreshActivePlugins()` 命名为
   `Layer-3 refresh primitive: swap active plugin components in the running session`
3. `reload-plugins.ts` 把 `/reload-plugins` 写成显式 `Reloaded: ...` 的 take-effect command
4. `useManagePlugins.ts` 明确写出 `Do NOT auto-refresh` 与 `Do NOT reset needsRefresh`
5. `PluginInstallationManager.ts` 区分 `new installs -> auto-refresh` 与 `updates only -> needsRefresh`
6. `AppStateStore.ts` 直接写出 interactive / headless 的不同 consume policy
7. `print.ts` 给出 headless `refreshPluginState()` 的 auto-reactivation 路径
8. remote/headless reload 还要求先拉新 settings 再接管 active world，
   说明 take-effect choreography 独立于 mere persistence

因此这一章的最短结论是：

`stronger-request cleanup reconfiguration governor 最多能说当前配置真相已经改写；stronger-request cleanup reactivation governor 才能说这组真相是否已经真正成为 running session truth。`

再压成一句：

`配置写好了，不等于已经重新生效。`

再压成更哲学一点的话：

`制度知道现在该按哪组现行说明书工作，不等于制度已经让这份说明书接管当前世界。`

## 4. 第一性原理：reconfiguration governance 回答“现在该按哪组 truth 工作”，reactivation governance 回答“这组 truth 何时真正接管当前世界”

从第一性原理看，
强请求清理重配置治理与强请求清理重新激活治理处理的是两个不同主权问题。

`stronger-request cleanup reconfiguration governor`
回答的是：

1. 哪些 options / secrets / payload 属于当前真相
2. 哪些值必须被 scrub
3. 哪些 channel / server config 仍算缺失
4. 哪些 store 持有 current truth
5. schema 当前是否已经闭合

`stronger-request cleanup reactivation governor`
回答的则是：

1. 这组 truth 是否已经进入 running session
2. 哪个时刻触发 active component swap
3. commands / agents / hooks / MCP / LSP 是否已按新 truth 重载
4. interactive / background / headless 各自采用哪种激活策略
5. stale active world 何时失效、何时被 current world 接管

如果把这两层压成一句：

`反正配置已经保存`

系统就会制造六类危险幻觉：

1. `saved-means-live`
2. `disk-change-means-active-swap`
3. `one-policy-fits-all-modes`
4. `schema-valid-means-running-valid`
5. `notification-means-reactivation`
6. `persisted-means-operable`

所以这两层的哲学本质也不同：

`reconfiguration`

治理的是：

`现在该按哪组 current truth 工作。`

`reactivation`

治理的是：

`这组 truth 何时正式接管当前运行世界。`

如果再压缩一点，
两者的本质差别其实是：

1. `reconfiguration` 负责 `truth to use`
2. `reactivation` 负责 `truth becoming live`

一个 returned object 即便已经知道自己“理论上应按哪组 truth 运转”，
也不等于当前世界已经真的开始按它运转。

## 5. Claude Code 多重安全技术的先进性：它把 take-effect 写成独立治理，而不是把 live 更新偷写成持久化副作用

Claude Code 在这一层真正先进的地方，
不是“配置写盘后世界自然跟上”，
而是：

`它把 persisted truth、stale signal、take-effect trigger 与 active-world swap 分开治理。`

这种先进性至少体现在六个方面：

### 5.1 saved truth is linguistically bounded

UI 不会说“已生效”，
而是说 `saved` 或 `configured`，
并要求 `/reload-plugins` 才 `take effect`。

### 5.2 active takeover has a named primitive

`refreshActivePlugins()` 明确承认 active-world swap 不是普通 helper，
而是 Layer-3 主权动作。

### 5.3 interactive mode preserves stale-state visibility

`needsRefresh` 不是 UX 噪音，
而是对 stale active world 的显式承认。

### 5.4 same disk change can map to different activation rules

new install 可 auto-refresh，
updates-only 则等待显式 consume。
系统不追求一刀切。

### 5.5 headless mode can auto-consume under explicit policy

`refreshPluginState()` 说明自动激活也可以是受治理的正规路径，
而不是偷偷刷新。

### 5.6 remote/headless choreography separates re-pull from reactivation

settings 先同步，
再 Layer-3 swap。
truth arrival 与 truth takeover 继续拆层。

这些设计背后的源码思路也很清楚：

1. 不让 `saved truth` 越级冒充 `active truth`
2. 不让 `disk mutation` 越级冒充 `running-session takeover`
3. 不让 `mode differences` 被压平为“都自动”或“都手动”
4. 不让 stale world 被静默吞掉，而是把它当成显式制度状态
5. 不让一次刷新失败被伪装成“反正已经生效了”

对 stronger-request cleanup 线最有价值的技术启示至少有六条：

1. 旧 cleanup truth comeback 后，不能因为新 truth 已写入就宣称 current world 已按它工作。
2. take-effect 需要独立 trigger，而不是 persistence side-effect。
3. stale active world 应被显式 surfaced，而不是被偷偷自动覆盖。
4. 不同模式下的 activation rule 可以不同，但必须被制度化。
5. 自动接管也必须是明示路径，而不是隐藏副作用。
6. 保存、接管、可用三者必须分账，否则“能看到配置”会冒充“已经在运行”。

## 6. 为什么 stronger-request cleanup 现在仍停在“current input truth grammar”，而不是“active-world takeover grammar”

和这些成熟 reactivation 正对照相比，
stronger-request cleanup 当前最明显的缺口已经不是：

`对象回来后究竟按哪组 current truth 运转。`

而是：

`这组 current truth 何时真正接管 running session。`

更具体地说，
现在仍没有哪一层正式决定：

1. 旧 startup wording comeback 后何时从 saved truth 进入 live truth
2. 旧 cleanup law comeback 后哪些 config change 只是 persisted、哪些已 active
3. 旧 promise vocabulary comeback 后是否需要显式 trigger 才能接管 current world
4. 旧 receipt objects comeback 后 interactive、background、headless 是否采用不同 activation policy
5. 哪些 stale active surface 应继续显化、哪些可以 auto-consume

换句话说，
stronger-request cleanup 当前虽然已经开始显露 reconfiguration-governance，
却还没有长出足够正式的：

1. Layer-3 swap primitive
2. take-effect contract
3. stale-signal discipline
4. mode-sensitive activation policy
5. headless auto-consume choreography
6. persisted-vs-active distinction

这意味着它现在能说的是：

`returned seat 现在应按哪组 current truth 运转。`

却还不能更强地说：

`这组 truth 现在已经真正接管当前运行世界。`

## 7. 苏格拉底式自反诘问：我是不是又把“配置已经保存”误认成了“当前运行世界已经接管了这组 truth”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 reconfiguration grammar 已经足够强，为什么还要再拆 reactivation？
   因为 config truth 已闭合，不等于 running session 已按它工作。
2. 如果配置已保存，是不是就说明当前 active world 已替换？
   不是。saved 与 active swap 在 repo 里被显式拆开。
3. 如果用户已经看见 `Enabled and configured`，是不是就说明新 truth 已经 live？
   不是。UI 同时保留了 `Run /reload-plugins to apply.` 这条更强 gate。
4. 如果同一 truth change 在某些路径下会 auto-refresh，是不是就说明所有路径都该自动？
   不是。reactivation policy 本来就按 mode 与 scenario 分配。
5. 如果 headless 会自动 consume，是不是就说明 interactive 也该自动？
   不是。headless 与 interactive 的风险、时机和用户心智不同。
6. 如果 cleanup 线现在还没有显式 reactivation 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确 grammar，越容易把“truth 已写好”偷写成“world 已换好”。

这一串反问最终逼出一句更稳的判断：

`reactivation 的关键，不在系统会不会让新 truth 最终生效，而在系统能不能正式决定它何时、以什么方式、在什么模式下接管当前运行世界。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 reconfiguration governance，就已经足够成熟。`

而是：

`Claude Code 在 Layer-3 refresh primitive、interactive stale-signal discipline、scenario-sensitive activation policy 与 headless auto-consume path 上已经明确展示了 reactivation governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-reactivation-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“现在该按哪组 truth 工作”，而是“这组 truth 何时真正接管当前世界”。`
