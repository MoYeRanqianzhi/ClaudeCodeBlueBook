# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `294` 之后还必须继续写 `295`

`294-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界何时正式结束，也还需要 tombstone governor 去决定结束后留下什么最小残留事实。`

但如果继续往下追问，
还会撞上一层更容易被偷写成
“只要墓碑已经留下或清掉，后面怎样回来也就自动清楚了”
的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定旧对象退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、回来时算谁、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 resurrection 正对照，
会发现它其实明确拆开了四件事：

1. `authoritative clearing`
2. `re-entry gate`
3. `identity policy`
4. `active-layer admission`

最硬的证据至少有七组：

1. `src/utils/plugins/cacheUtils.ts:67-72,82-92,122-130`
   证明 `.orphaned_at` 的清除由 authoritative installed set 决定，
   而不是 marker 自己决定
2. `src/utils/plugins/refresh.ts:1-18,72-80,123-137`
   证明 active plugin return 是专门的 Layer-3 refresh primitive，
   不是 marker 清掉后的自动副作用
3. `src/commands/reload-plugins/reload-plugins.ts:37-56`
   证明 `/reload-plugins` 是显式 re-entry signal，
   系统不会把“磁盘世界变了”偷写成“active world 已接纳”
4. `src/services/plugins/PluginInstallationManager.ts:56-59,135-179`
   与
   `src/hooks/useManagePlugins.ts:28-34,287-303`
   证明 repo 还会继续区分：
   哪些 return path 可以 auto-refresh，
   哪些 return path 必须停在 `needsRefresh`
5. `src/utils/plans.ts:164-231,233-257`
   证明 plan resurrection 不只是 undelete，
   还要治理 lineage、evidence 与 identity policy
6. `src/utils/conversationRecovery.ts:539-547`
   证明 resurrection grammar 被接进正式恢复主线，
   不是局部 helper 的偶发行为
7. `src/utils/sessionStorage.ts:871-945,1468-1474`
   共同证明 deletion grammar 只回答怎样删旧对象，
   不回答它何时、以什么身份重新回来

这说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 identity policy 与什么 active-layer admission 下，这个旧对象还能重新回到 current world。`

所以 `294` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`
不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk / materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery`

就等于 resurrection 的全部语义。

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线未来的复活 grammar 仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

第六条：

这里还不能过度声称：

`Claude Code 的 resurrection 只是恢复体验优化。`

更稳的说法只能是：

`它本质上是在治理 current world 的重新接纳权；体验只是外观，资格与接纳才是本体。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer`

证据：

1. `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
2. `refreshActivePlugins()` 与 `/reload-plugins` 证明 marker clearing / disk presence 不等于 active components 已重新进入当前 session
3. repo 明确要求某些 update path 停在 `needsRefresh`，而不是默认自动接纳
4. `copyPlanForResume()` 证明 plan resurrection 需要 lineage 与 evidence
5. `copyPlanForFork()` 证明“回来”不等于“以原身份回来”
6. `conversationRecovery.ts` 把 resurrection 接进恢复主线，说明它是正式 recovery grammar
7. `sessionStorage.ts` 的 tombstone path 只回答怎样删除旧对象，不回答它何时、以什么身份重新回来

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据、何种身份政策与何种激活层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

再压成更哲学一点的话：

`制度知道怎样留下历史痕迹，不等于制度已经知道谁配重新被当前世界承认。`

## 4. 第一性原理：tombstone governance 回答“留什么最小残留事实”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

`stronger-request cleanup tombstone governor`
回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

`stronger-request cleanup resurrection governor`
回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk / materialized world，还是 active / runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句
`反正 marker 还在或 marker 已清掉`，
系统就会制造五类危险幻觉：

1. `marker-clear-means-resurrection`
2. `disk-return-means-active-return`
3. `content-recovered-means-identity-restored`
4. `tombstone-is-bidirectional`
5. `any-old-object-can-come-back`

所以这两层的哲学本质也不同：

`tombstone`

治理的是：

`一个对象退出当前真相之后，世界还欠后来者什么最小解释义务。`

`resurrection`

治理的是：

`一个历史对象若想重新进入当前真相，谁有权为它重新签发在场资格。`

Claude Code 在这里真正先进的地方，
不是它把 every return path 都自动做通，
而是它已经默认承认：

`重新出现，不等于重新被承认。`

这恰恰是一种很成熟的安全哲学：

`对象可以重新物化，但 current world 的接纳权不能因此被偷走。`

## 5. 从技术角度看，Claude Code 的先进性在于把 resurrection 拆成 authoritative clearing、re-entry gate、identity policy 与 active-layer admission，而不是把“回来”压成一个动作

### 5.1 authoritative marker clearing 不是 resurrection

`cacheUtils.ts:67-72,82-92,122-130`
先把 orphan cleanup 写成两段：

1. Pass 1 清已安装版本的 `.orphaned_at`
2. Pass 2 才处理真正 orphaned 的版本

而且它不是任意清，
而是：

1. 先读取 `installed_plugins.json`
2. 形成 authoritative `installedVersions`
3. 只对这个集合里的 version path 执行 `removeOrphanedAtMarker()`

这说明：

`谁配撤销“你已退役”这条 marker`

本身就是一种制度。

但 `removeOrphanedAtMarker()` 只删 marker，
并不负责任何 active components return。
这正好证明：

`能够撤销墓碑`

和

`能够重新接纳对象`

并不是同一主权。

### 5.2 Layer-3 refresh 是显式 re-entry gate

`refresh.ts:1-18`
直接把 active plugin swap 定义为：

`Layer-3 refresh primitive`

同时明确区分：

1. Layer 1: intent
2. Layer 2: materialization
3. Layer 3: active components

`refresh.ts:72-80,123-137`
继续把 `refreshActivePlugins()` 固定成 Layer-3 refresh primitive：

1. clear plugin caches
2. clear orphan exclusions
3. reload active commands / agents / hooks / MCP / LSP
4. consume `needsRefresh`
5. bump `pluginReconnectKey`

再看
`reload-plugins.ts:37-56`，
显式运行 `/reload-plugins`，
系统才会返回：

`Reloaded: ...`

也就是说 plugin 线已经把：

`marker clearing`

和

`active-layer admission`

写成了两个不同主权。

### 5.3 `needsRefresh` 证明 repo 明确拒绝把“对象重新存在”偷写成“当前会话已经重新接纳它”

`PluginInstallationManager.ts:56-59`
先给出大前提：

1. `New installs -> auto-refresh plugins`
2. `Updates only -> set needsRefresh`

继续看
`PluginInstallationManager.ts:135-179`：

1. 新 marketplace 安装完成时，repo 会尝试 `refreshActivePlugins()`
2. 但 auto-refresh 失败时，会回退为 `needsRefresh`
3. 如果只是 updates only，系统直接设置 `needsRefresh`

再看
`useManagePlugins.ts:28-34,287-303`：

1. 初始 Layer-3 load 之后，后续 refresh 统一走 `/reload-plugins`
2. `needsRefresh` 的 effect 只发通知：
   `Plugins changed. Run /reload-plugins to activate.`
3. 注释明确写着：
   `Do NOT auto-refresh`

这说明 Claude Code 并不把 resurrection grammar 压成一个粗糙的
`回来 / 不回来`
开关。

它实际上在治理三种不同 world：

1. 可以 auto-admit 的 world
2. 必须 explicit admit 的 world
3. 还没被接纳、只能停在 `needsRefresh` 的 world

### 5.4 plan 线给出第二个强正例：对象复活时，系统还要治理 lineage、evidence 与 identity policy

`plans.ts:164-231`
的 `copyPlanForResume()` 很值钱。

它不是看到 plan file 缺失就粗暴恢复。
它要求：

1. log 里确实有 slug
2. 把 slug 绑定回 original session ID
3. 先尝试直接读取 plan file
4. 只有在 `ENOENT` 且 remote session 才尝试 recovery
5. 优先用 file snapshot，失败后再回到 message history

这说明 plan resurrection 的本体不是：

`内容还在，就给你拉回来`

而是：

`只有在 lineage 和 evidence 都成立时，才允许对象重新进入 current session world。`

更关键的是
`plans.ts:233-257`
里的 `copyPlanForFork()`：

1. 不复用 original slug
2. 生成 new slug
3. 把原计划内容写进新文件

这直接证明：

`内容回来`

不等于

`原身份回来`

也就是说 resurrection governance 还必须回答：

`re-entry 后对象到底算 old identity 还是 new identity。`

`conversationRecovery.ts:539-547`
又把这件事接进正式恢复主线，
说明 resurrection 不是局部 helper 的偶发行为，
而是正式恢复制度的一部分。

### 5.5 tombstone 删除路径反证：删除 grammar 的存在不自动携带复活 grammar

`sessionStorage.ts:871-945,1468-1474`
把 tombstone 删除路径写得极其单向：

1. 目标是 remove
2. 载体是 orphaned message tombstone
3. 操作是按 UUID 删除
4. slow path 只回答如何安全重写大 transcript 文件

它完全不回答：

1. 这些对象在什么条件下还能回来
2. 回来时是否还是原 UUID
3. 回来时谁来宣布它重新成为 current truth

这条反证很重要，
因为它说明：

`成熟系统不会把 re-entry 语义偷塞进 deletion helper 里。`

它宁可显式缺席，
也不允许你从：

`能删`

直接推成：

`会让它回来`

## 6. 对 stronger-request cleanup 而言，当前真正缺的不是“有没有墓碑”，而是“带着墓碑的旧对象凭什么回来”

当前 stronger-request cleanup 已经开始显露 tombstone-governance，
但仍缺少 resurrection 治理回答：

1. 旧 path / old promise / old receipt world 的 marker 若被撤销，谁来宣布那不是单纯 marker clear，而是 current world re-entry
2. 旧 cleanup object 回来时，是原 identity 恢复，还是新 identity 接管旧内容
3. 回来的是 disk-world visibility，还是 runtime-world authority
4. 哪些 return path 可以 auto-admit，哪些必须显式 re-entry gate
5. 哪些历史对象根本不配回来，即便 marker 仍可被清掉

这些都不是：

`退役后留下什么`

能回答的。

它们已经进入：

`resurrection governance`

层。

也就是说，
stronger-request cleanup 今天真正缺的，
不是再补一个 tombstone marker，
而是：

`谁来决定带着这些 marker 的旧对象凭什么重新进入 current world。`

## 7. 苏格拉底诘问：怎样把这一层做得更严格

### 诘问一

`如果 marker 被 authoritative set 清掉，为什么还不能直接宣布对象已经回来？`

因为 marker clear 只证明：

`“你不再被视作 orphan/retired”这条历史判断被撤销了。`

它并不自动证明：

`你已经重新被当前 runtime 接纳。`

### 诘问二

`如果对象内容已经恢复，为什么还要单独治理 identity policy？`

因为对象的风险从来不只在内容，
还在：

1. 谁拥有它
2. 谁为它担保
3. 谁可以继续援引它的旧资格

内容回来而身份不重建，
很多安全错误就会被偷渡成“只是恢复内容”。

### 诘问三

`如果 resurrection grammar 太严格，会不会导致系统难以恢复？`

会增加恢复门槛，
但这是必要代价。
否则系统获得的是一种更危险的“容易回来”：

`任何旧对象只要重新出现，就可能被误当作 current truth。`

### 诘问四

`那怎样才算更好的 resurrection governance？`

至少要继续把四件事制度化：

1. marker clear authority
2. re-entry gate
3. identity policy
4. active-layer admission

缺任何一层，
都容易把“历史对象可见”误写成“当前世界已接纳”。

## 8. 技术启示与哲学本质

这一层真正重要的技术启示不是抽象地说：

`cleanup 线也许还需要 resurrection。`

而是更具体地说：

1. 旧 path、旧 promise、旧 receipt 若想回来，必须回答谁有权清掉其墓碑
2. 墓碑清掉后，还要回答是否存在显式 re-entry gate
3. 即便内容可恢复，也还要回答它是否保留旧 identity
4. 即便物化回来，也还要回答它进入的是 disk world 还是 active world

Claude Code 在这里真正先进的地方，
不是它把 comeback path 做成 one-shot convenience，
而是它已经在多个 subsystem 里承认：

`重新出现`

和

`重新被承认`

不是同一件事。

这是一种很成熟的安全哲学：

`历史对象可以再次可见，但 current world 的接纳权不能因此被偷走。`

## 9. 一条硬结论

Claude Code 当前源码研究已经足以说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

仍然不能越级冒充

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

因为：

1. tombstone 负责治理 post-retirement marker
2. resurrection 负责治理 current-world re-entry
3. authoritative clearing、Layer-3 refresh、needsRefresh、lineage-based recovery 与 deletion-only path 都证明 repo 已把这两层正式拆开
4. stronger-request cleanup 当前虽已开始理解旧世界结束后要留下什么 marker，却还没有对等地回答带着这些 marker 的旧对象凭什么回来

因此，
这一章最终必须压成一句：

`会留墓碑，不等于会让它回来。`
