# 安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层：为什么artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-reactivation-governor signer

## 1. 为什么在 `297` 之后还必须继续写 `298`

`297-安全载体家族强请求清理再赋权治理与强请求清理重配置治理分层`
已经回答了：

`stronger-request cleanup 线即便知道旧对象重新拿回 seat 后该按哪组 current config truth 工作，也还需要 reconfiguration governor 去决定这组 truth 长什么样。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“配置写好了就说明当前运行世界已经按它工作了”
的错觉：

`只要 stronger-request cleanup reconfiguration governor 已经决定了 current options、current secrets、current payload 与 current channel truth，它就自动拥有了宣布这些 truth 已经接管 running session 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 plugin 线已经成熟写出的 reactivation 正对照，
会发现它其实明确拆开了五件事：

1. `persisted truth`
2. `active truth`
3. `stale signal`
4. `mode-sensitive activation policy`
5. `takeover contract`

最硬的证据至少有七组：

1. `src/commands/plugin/ManagePlugins.tsx:1631-1707`
   明确把
   `Enabled and configured`
   、
   `Enabled`
   与
   `Configuration saved. Run /reload-plugins for changes to take effect.`
   拆成三层句法
2. `src/utils/plugins/refresh.ts:1-18,59-160`
   把 `refreshActivePlugins()` 直接命名为
   `Layer-3 refresh primitive: swap active plugin components in the running session`
3. `src/commands/reload-plugins/index.ts:1-16`
   与
   `src/commands/reload-plugins/reload-plugins.ts:1-56`
   把 `/reload-plugins` 写成正式
   `Activate pending plugin changes in the current session`
   / `Reloaded: ...`
   契约
4. `src/hooks/useManagePlugins.ts:287-303`
   明确写出
   `Do NOT auto-refresh. Do NOT reset needsRefresh`
5. `src/services/plugins/PluginInstallationManager.ts:52-60,123-180`
   明确区分
   `new installs -> auto-refresh`
   与
   `updates only -> needsRefresh`
6. `src/state/AppStateStore.ts:179-215`
   直接把 `needsRefresh` 注释成：
   interactive mode 由用户运行 `/reload-plugins` 消费，
   headless mode 由 `refreshPluginState()` 自动消费
7. `src/cli/print.ts:1733-1765,1881-1912`
   以 `before the first ask()` 的明确注释，
   把 headless auto-reactivation path 写成单独 choreography

这说明：

`artifact-family cleanup stronger-request cleanup-reconfiguration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

仍然不是一回事。

前者最多能说：

`对象现在该按哪组 current config truth 工作。`

后者才配说：

`这组 current config truth 现在是否已经真正接管 running session，并开始统治 commands / agents / hooks / MCP / LSP 的 active world。`

所以 `297` 之后必须继续补的一层就是：

`安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层`

也就是：

`stronger-request cleanup reconfiguration governor 负责改写 current config truth；stronger-request cleanup reactivation governor 才负责决定这组 truth 何时真正进入 active world。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-reactivation-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 reactivation manager，
而是在说：

1. repo 已经把 Layer 3 明确命名为 active components world
2. repo 已经把 `saved` 与 `take effect` 拆开
3. repo 已经把 interactive、background 与 headless 的接管策略做成不同制度

第二条：

这里说的 `reactivation governance`
不是在声称：

`一切配置变化都必须立刻自动生效。`

当前更稳的说法只能是：

`reactivation governance 首先回答的是谁配触发 active takeover、在哪个 mode 下触发、触发失败后怎样降级。`

第三条：

这里也不是把：

`/reload-plugins`

单独当成全部 reactivation grammar。

真正关键的是：

1. Layer-3 refresh primitive
2. stale signal
3. interactive no-auto-refresh discipline
4. background auto-refresh / manual-refresh split
5. headless before-first-ask auto-consume path

第四条：

这里尤其不能过度声称：

`配置已保存`

就等于

`当前 running session 已按这组 truth 运行。`

当前更稳的说法只能是：

`repo 明确把 persisted truth 与 active truth 拆开；后者还要经过显式 takeover choreography。`

第五条：

这里也不能过度声称：

`同一类 config change 在所有模式下都应采用同一种 reactivation policy。`

当前更稳的说法只能是：

`reactivation policy 本身就是 mode-sensitive governance，而不是单一自动规则。`

第六条：

这里还不能过度声称：

`只要重新激活成功，就说明对象已经 ready。`

当前更稳的说法只能是：

`reactivation 先回答的是 active world 是否切换；切换以后是否真正 ready、connected、healthy，仍是下一层问题。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-reconfiguration-governor signer 仍不等于 stronger-request cleanup-reactivation-governor signer`

证据：

1. `ManagePlugins.tsx`
   明确写出
   `Configuration saved`
   与
   `Run /reload-plugins for changes to take effect`
   不是同一句制度
2. `refresh.ts`
   直接把 `refreshActivePlugins()` 命名为
   `Layer-3 refresh primitive`
3. `/reload-plugins`
   明确把激活写成正式用户命令契约
4. `useManagePlugins.ts`
   明确允许
   `current truth changed but active world stays stale`
   的中间态存在
5. `PluginInstallationManager.ts`
   明确把不同变化类型分配给不同 reactivation policy
6. `print.ts` 与 `AppStateStore.ts`
   又给出 headless `before first ask()` 的 auto-consume path

因此这一章的最短结论是：

`stronger-request cleanup reconfiguration governor 最多能说当前配置真相已经改写；stronger-request cleanup reactivation governor 才能说这组真相是否已经真正成为 running session truth。`

再压成一句：

`配置写好了，不等于已经重新生效。`

再压成更哲学一点的话：

`制度知道这个对象现在该按哪组输入工作，不等于制度已经允许这组输入接管当前世界。`

## 4. 第一性原理：reconfiguration governance 回答“现在该按哪组 truth 工作”，reactivation governance 回答“这组 truth 何时真正接管当前世界”

从第一性原理看，
强请求清理重配置治理与强请求清理重新激活治理处理的是两个不同主权问题。

`stronger-request cleanup reconfiguration governor`
回答的是：

1. 哪些 options / secrets / payload 属于当前真相
2. 哪些字段必须被 scrub
3. 哪些 schema 仍未闭合
4. 哪些 store 持有 current truth
5. 配置真相现在长什么样

`stronger-request cleanup reactivation governor`
回答的则是：

1. 这组 truth 是否已经进入 running session
2. 谁有权触发 active component swap
3. commands / agents / hooks / MCP / LSP 是否已按新 truth 重载
4. interactive / background / headless 各自采用哪种 takeover policy
5. 旧 active world 何时被正式判定为 stale 并被 current world 替换

如果把这两层压成一句
“反正配置已经保存”，
系统就会制造六类危险幻觉：

1. `saved-means-live`
2. `disk-change-means-active-swap`
3. `one-policy-fits-all-modes`
4. `schema-valid-means-running-valid`
5. `notification-means-activation`
6. `activation-means-readiness`

再用苏格拉底式反问压一次：

1. 如果 reconfiguration governance 已经等于 reactivation governance，
   为什么 `ManagePlugins` 还要把 `Configuration saved` 与 `take effect` 拆成两句话？
   因为“truth 已写入”与“truth 已接管当前世界”根本不是同一句制度。
2. 如果 config truth 一旦闭合就自动生效，
   为什么 `refreshActivePlugins()` 还要作为独立的 Layer-3 primitive 存在？
   因为 persisted truth 与 active truth 是两层不同主权。
3. 如果 reactivation policy 与 mode 无关，
   为什么 interactive mode 明确 `Do NOT auto-refresh`，
   而 headless mode 却允许 `before first ask()` 自动 consume？
   因为谁配宣布 stale world 被 current world 接管，本来就随 operational context 变化。
4. 如果 stale signal 已经等于 active takeover，
   为什么 repo 还要保留 `needsRefresh`？
   因为系统必须诚实承认“我知道它变了”，但不冒领“我已经切过去了”。
5. 如果 active takeover 已经等于 ready truth，
   为什么重连、connected、health、read gate 还要存在？
   因为 active world 切换只是接管，不是全链 readiness verdict。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“重新激活治理”是一层独立制度

### 5.1 `ManagePlugins` 先证明：`Configuration saved` 与 `take effect` 被明确写成两层事实

`src/commands/plugin/ManagePlugins.tsx:1631-1707`
很硬。

这段代码至少并列暴露了三种不同句法：

1. `Enabled and configured ... Run /reload-plugins to apply.`
2. `Enabled ... Run /reload-plugins to apply.`
3. `Configuration saved. Run /reload-plugins for changes to take effect.`

这说明 repo 公开承认：

1. enable truth
2. config truth
3. active-world effect truth

不是同一层。

这里最值钱的不是提示语存在，
而是提示语拒绝偷写 stronger claim。

它没有说：

`Configuration saved and activated`

而是明确说：

`for changes to take effect`

这是一种非常成熟的安全设计：

`只承认已经写入的 truth，不冒领尚未接管的 truth。`

### 5.2 `refreshActivePlugins()` 再证明：current active world 的接管不是顺手刷新，而是单独 Layer-3 choreography

`src/utils/plugins/refresh.ts:1-18`
第一行就把自己的角色写死：

`Layer-3 refresh primitive: swap active plugin components in the running session.`

更关键的是它做的事并不是“再读一遍配置”而已，
而是完整的 active-world takeover choreography：

1. `clearAllCaches()`
2. `clearPluginCacheExclusions()`
3. 重新 `loadAllPlugins()`
4. 重新装填 commands / agents
5. 预热 MCP / LSP cache slot
6. 写回 `AppState.plugins`
7. 消费 `needsRefresh`
8. bump `mcp.pluginReconnectKey`
9. `reinitializeLspServerManager()`
10. `loadPluginHooks()`

这已经不是“保存后的顺手刷新”，
而是完整的 Layer-3 主权动作。

Claude Code 在这里展示的先进性是：

`它没有把“配置真相存在”偷写成“运行中世界已经替换”，而是给 active takeover 单独分配了可见 primitive。`

### 5.3 `/reload-plugins` 再证明：reactivation 可以是一个单独的用户命令主权

`src/commands/reload-plugins/index.ts:1-16`
与
`src/commands/reload-plugins/reload-plugins.ts:1-56`
一起很硬。

这两处共同做了四件事：

1. 文档层把命令写成
   `Activate pending plugin changes in the current session`
2. 运行层显式调用 `refreshActivePlugins()`
3. 回执层明确返回 `Reloaded: ...`
4. remote mode 下还会先 `redownloadUserSettings()`，
   再通知 settings change detector，让 mid-session current truth 先被重拉

这里最重要的不是命令存在，
而是它的 contract 被明确写成：

`activate`

与

`Reloaded`

也就是说 repo 明确承认：

1. settings truth 可以先改
2. running session 仍可能停留在旧 active world
3. 必须再走一次单独命令，
   才能把 pending truth 升格为 current running truth

而且它还采用了
`one attempt + fail-open`
策略，
允许用户重试，
说明 reactivation governance 的哲学不是“无论如何偷偷替你完成”，
而是“把 takeover 主权留在可见的、可重试的控制面上”。

### 5.4 `useManagePlugins` 与 `PluginInstallationManager` 一起证明：同一 persisted truth，在不同 mode / scenario 下可被分配不同接管策略

`src/hooks/useManagePlugins.ts:287-303`
价值非常高。

这里只做两件事情：

1. 当 `needsRefresh` 为真时发通知：
   `Plugins changed. Run /reload-plugins to activate.`
2. 明确写注释：
   `Do NOT auto-refresh. Do NOT reset needsRefresh`

也就是说，
在 interactive mode 下，
repo 明确允许一种中间状态存在：

`current truth has changed, active world is stale`

而
`src/services/plugins/PluginInstallationManager.ts:123-180`
又把制度写得很清楚：

1. `new installs -> auto-refresh plugins`
2. `updates only -> set needsRefresh`
3. auto-refresh 失败后再 fallback 到 `needsRefresh`

这说明 repo 已经公开承认：

`reactivation policy is scenario-sensitive`

也就是说，
即便底层都属于 `plugin state changed on disk`，
系统仍继续追问：

1. 这次是新安装还是更新
2. 现在更该自动接管，还是更该等待用户显式确认时机
3. 自动路径失败后如何安全降级到手动路径

这种设计的先进性在于：

`它把自动化能力约束在适合自动化的场景里，而不是把一切变化都粗暴压成同一条自动接管规则。`

### 5.5 `print.ts` 与 `AppStateStore` 给出第三个强正例：headless mode 的 reactivation governance 可以与 interactive mode 完全不同

`src/cli/print.ts:1733-1765,1881-1912`
写得非常清楚：

1. `CLAUDE_CODE_SYNC_PLUGIN_INSTALL=true` 时，
   installation 会在 first ask 前 resolve
2. resolve 后显式调用 `refreshPluginState()`
3. `refreshPluginState()` 再显式调用 `refreshActivePlugins()`
4. 注释直接写出：
   `plugins are guaranteed available on the first ask()`

而
`src/state/AppStateStore.ts:206-214`
更直接把制度压成一句：

`interactive mode, user runs /reload-plugins to consume; headless mode, refreshPluginState() auto-consumes`

这等于直接证明：

1. same Layer-3 primitive
2. different operational context
3. different activation authority

这不是不一致，
恰恰是成熟治理：

`同一 takeover primitive，可以被不同 mode 的 governor 以不同政策消费。`

## 6. 安全设计的哲学本质：真正被治理的不是“配置”，而是“哪一层配宣布当前世界已经被改写”

如果把这章压到最深处，
Claude Code 在这里展示的并不是单纯的插件热重载技巧，
而是一种更硬的安全哲学：

`世界是否已经改变，必须由有权宣布 current-world takeover 的层来签字。`

这套哲学至少包含四个原则：

1. `honesty over convenience`
   系统宁可承认 `needsRefresh`，
   也不把 pending truth 冒充成 live truth
2. `stale visibility over hidden magic`
   系统宁可公开暴露 active world 仍旧 stale，
   也不在 interactive path 上偷偷替用户切世界
3. `mode-sensitive governance over one-size-fits-all automation`
   同样的 truth change，
   在 headless、interactive、background 三种模式下可有不同接管策略
4. `takeover authority over mere persistence`
   保存只是把 truth 写在磁盘；
   生效才是让这组 truth 接管当前权力世界

所以这一层的哲学本质不是：

`系统会不会刷新`

而是：

`系统把“宣布当前世界已经换了”这件事交给谁。`

## 7. 苏格拉底式自我反思：如果要把这层分析做得更严，最该防什么偷换

第一问：

`我是不是把 save path 误判成 apply path 了？`

如果是，
`ManagePlugins` 的 `Configuration saved ... take effect`
就已经明确反驳我。

第二问：

`我是不是把 stale signal 误判成 activation verdict 了？`

如果是，
`needsRefresh` 的存在本身就说明
“系统知道它旧了”
不等于
“系统已经切过去了”。

第三问：

`我是不是把 Layer-3 refresh 当成普通 helper，而没把它当成治理 primitive？`

如果是，
`refresh.ts` 的文件头注释已经把这条误判直接封死。

第四问：

`我是不是把 interactive path、background path 与 headless path 压成了同一规则？`

如果是，
`useManagePlugins`、
`PluginInstallationManager`
与
`print.ts`
三组代码已经说明这是错的。

第五问：

`我是不是又在 reactivation 层偷写 readiness、continuity 或 recovery 的 stronger claim？`

如果是，
我就需要立即收手。
因为这一章最严格的边界正是：

`只证明 current truth 是否已接管 active world，不越级证明后续运行一定健康、持续、可恢复。`

所以，对这层分析最严的自律不是写得更强，
而是写得更准：

`配置写好了，不等于已经重新生效；重新生效了，也还不等于一切后续治理问题都已经被回答。`
