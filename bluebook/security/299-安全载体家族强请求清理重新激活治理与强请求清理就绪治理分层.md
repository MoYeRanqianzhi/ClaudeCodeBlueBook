# 安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层：为什么artifact-family cleanup stronger-request cleanup-reactivation-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-readiness-governor signer

## 1. 为什么在 `298` 之后还必须继续写 `299`

`298-安全载体家族强请求清理重配置治理与强请求清理重新激活治理分层`
已经回答了：

`stronger-request cleanup 线即便知道哪组 current config truth 何时真正接管 active world，也还需要 reactivation governor 去决定 stale world 何时被 current world 替换。`

但如果继续往下追问，
还会碰到另一层更容易被偷写成
“active world 已切换就说明现在已经能用了”
的错觉：

`只要 stronger-request cleanup reactivation governor 已经把新的 truth 接进 current session，它就自动拥有了决定对象现在是否真的 connected、authenticated、capable、ready-for-use 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 repo 在 MCP 线已经成熟写出的 readiness 正对照，
会发现它其实明确拆开了五件事：

1. `state machine`
2. `reactivation-triggered pending`
3. `surface grammar`
4. `consumer gate`
5. `runtime revocation`

最硬的证据至少有七组：

1. `src/services/mcp/types.ts:179-226`
   的 `MCPServerConnection` 联合类型把 client 明确拆成
   `connected / pending / needs-auth / failed / disabled`
2. `src/services/mcp/useManageMCPConnections.ts:149-153,765-853`
   说明 `/reload-plugins` 之后不是直接 ready，
   而是重新进入 `pending` lifecycle
3. `src/services/mcp/useManageMCPConnections.ts:1109-1123`
   说明 even after explicit enabling，
   client 也仍要先进入 `pending`
4. `src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
   只把 `failed / needs-auth` 当成 readiness 降级 surface，
   而不是把 active truth 直接冒充成 ready truth
5. `src/commands/plugin/ManagePlugins.tsx:513-519`
   与
   `src/cli/handlers/mcp.tsx:26-35`
   继续把 `connected / pending / needs-auth / failed / disabled`
   分开显化
6. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:86-95`
   对 downstream tool 明确要求 `client.type === 'connected'`
7. `src/services/tools/toolExecution.ts:1599-1628`
   会把一个原本 `connected` 的 client 在运行时降成 `needs-auth`

这说明：

`artifact-family cleanup stronger-request cleanup-reactivation-governor signer`

和

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

仍然不是一回事。

前者最多能说：

`新的 truth 已经被接进当前世界。`

后者才配说：

`这个当前世界现在真的 ready，可以被后续 reader、tool 与 user-facing surface 当成可用事实继续消费。`

所以 `298` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层`

也就是：

`stronger-request cleanup reactivation governor 负责把 truth 接进 active world；stronger-request cleanup readiness governor 才负责决定这个 active world 现在是否真的可用。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-readiness-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-readiness-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 readiness manager，
而是在说：

1. repo 已经明确把 MCP 连接状态拆成 `connected / pending / needs-auth / failed / disabled`
2. repo 已经把 `activate`、`connected`、`needs-auth`、`failed` 写成不同 surface
3. repo 已经让工具调用只接受 `connected`，而不是“刚重载过就算 ready”

第二条：

这里说的 `readiness governance`
不是在声称：

`只有完全无错误的系统才算 ready。`

当前更稳的说法只能是：

`readiness governance 首先回答的是下游 consumer 现在可不可以继续安全消费它，而不是回答系统是否永远不会再出错。`

第三条：

这里也不是把：

`connected`

单独当成全部 readiness grammar。

真正关键的是：

1. `pending` 仍未闭合
2. `needs-auth` 是 readiness 未闭合
3. `failed` 是 readiness 失败
4. capability gate 仍会继续限制下游消费

第四条：

这里尤其不能过度声称：

`对象曾经连通过一次，就说明它现在仍 ready。`

当前更稳的说法只能是：

`readiness 可以被运行时证据持续撤销；它不是一次性永久 verdict。`

第五条：

这里也不能过度声称：

`用户看见 reload summary，就已经得到 capability proof。`

当前更稳的说法只能是：

`summary 只回答某次 reactivation choreography 已经运行，不自动回答所有 downstream consumer 都已经可以继续用。`

第六条：

这里还不能过度声称：

`只要 ready，就说明后续一定连续、稳定、可恢复。`

当前更稳的说法只能是：

`readiness 先回答“现在能不能继续消费”，但它还不替 continuity、recovery 或 reintegration 签字。`

## 3. 最短结论

Claude Code 当前源码至少给出了六类

`stronger-request cleanup-reactivation-governor signer 仍不等于 stronger-request cleanup-readiness-governor signer`

证据：

1. `MCPServerConnection`
   联合类型直接把 client state 拆成
   `connected / pending / needs-auth / failed / disabled`
2. `useManageMCPConnections()`
   在 `pluginReconnectKey` 触发后，
   会先把新 client 置为 `pending`
3. `toggleMcpServer()`
   就算是显式 enable，
   也仍先把对象写成 `pending` 再 reconnect
4. `useMcpConnectivityStatus()`、`getMcpStatus()` 与 `/mcp` health
   都把 `failed`、`needs-auth`、`pending` 单独显化
5. `ReadMcpResourceTool`
   明确要求 `client.type === 'connected'`
6. `toolExecution.ts`
   在 `McpAuthError` 时会把一个原本 `connected` 的 client 降成 `needs-auth`

因此这一章的最短结论是：

`stronger-request cleanup reactivation governor 最多能说新的 truth 已经进入当前世界；stronger-request cleanup readiness governor 才能说这个当前世界现在到底能不能被继续使用。`

再压成一句：

`重新激活，不等于已经就绪。`

再压成更哲学一点的话：

`制度已经把对象接回当前世界，不等于制度已经愿意让后续读者继续相信它。`

## 4. 第一性原理：reactivation governance 回答“这组 truth 是否已接入当前世界”，readiness governance 回答“这个当前世界现在能不能被继续消费”

从第一性原理看，
强请求清理重新激活治理与强请求清理就绪治理处理的是两个不同主权问题。

`stronger-request cleanup reactivation governor`
回答的是：

1. stale world 是否已被 current world 替换
2. 哪个时刻完成 Layer-3 swap
3. 哪些 active components 已经换入
4. 当前 session 是否已经接到新的 config truth
5. 不同 mode 下谁来触发 active takeover

`stronger-request cleanup readiness governor`
回答的则是：

1. 当前 client 是否已 `connected`
2. 是否仍在 `pending`
3. 是否进入 `needs-auth`
4. 是否已经 `failed`
5. 下游工具、资源读取和 health check 是否可以继续消费它

如果把这两层压成一句
“反正已经 reload 了”，
系统就会制造六类危险幻觉：

1. `active-means-ready`
2. `pending-means-soon-enough`
3. `auth-gap-is-incidental`
4. `connected-once-means-ready-now`
5. `activation-summary-means-capability-proof`
6. `ready-means-stable`

再用苏格拉底式反问压一次：

1. 如果 reactivation governance 已经等于 readiness governance，
   为什么 `pluginReconnectKey` 触发后系统还要先把 client 加成 `pending`，
   而不是直接视为 ready？
   因为“truth 已接进 current world”与“现在可以继续消费”根本不是同一句制度。
2. 如果当前世界一旦 reactivated 就已经 ready，
   为什么 `ReadMcpResourceTool` 还要坚持 `client.type === 'connected'` 的 hard gate？
   因为 active truth 与 consumable truth 是两层不同主权。
3. 如果 readiness 是一次性 verdict，
   为什么 `toolExecution.ts` 还要在 `McpAuthError` 时把 `connected` 降成 `needs-auth`？
   因为 readiness 不是历史荣誉，而是会被新证据撤销的当前判断。
4. 如果用户看见 `/reload-plugins` 的 summary 就已经拿到就绪证明，
   为什么 `/mcp` health、notification 与管理界面还要继续显化 `needs-auth / failed / pending`？
   因为 activation receipt 不等于 readiness receipt。
5. 如果 ready 就已经等于后续稳定，
   为什么后面还会有 continuity、recovery 等更高层治理？
   因为“现在能用”与“之后继续能用”同样不是同一制度。

## 5. 技术机制剖面：Claude Code 的多重安全机制为什么能证明“就绪治理”是一层独立制度

### 5.1 `MCPServerConnection` 先证明：repo 公开承认 active world 里可以合法存在 non-ready object

`src/services/mcp/types.ts:179-226`
非常值钱。

这里没有把对象状态写成：

`present / absent`

而是明确拆成：

1. `connected`
2. `failed`
3. `needs-auth`
4. `pending`
5. `disabled`

一旦 union type 本身已经这么写，
系统就已经在类型层承认：

`object is in current world`

并不自动等于：

`object is ready`

从第一性原理看，
这是非常先进的安全设计。

它拒绝把“对象已经被当前世界看见”偷写成“对象已经被当前世界信任并可继续消费”。

### 5.2 `pluginReconnectKey` 与 `initializeServersAsPending()` 再证明：reactivation 之后仍要经过 readiness 过渡层

`src/services/mcp/useManageMCPConnections.ts:149-153`
明确把 `_pluginReconnectKey` 挂进 hook 依赖里，
而
`src/services/mcp/useManageMCPConnections.ts:765-853`
的 effect 注释又直接写出：

`Re-runs on session change (/clear) and on /reload-plugins (pluginReconnectKey).`

更关键的是 re-run 后系统做的不是：

`mark all ready`

而是：

1. 清除 stale plugin clients，
   防止旧配置与 ghost tools 继续污染当前世界
2. 对新 client 按 disabled / pending 初始化
3. 让连接流程重新开始

这里最硬的点在 `819-825`：

不是 disabled 的新 client 会被直接加成：

`type: 'pending'`

这说明 `pluginReconnectKey` 的制度含义不是：

`readiness granted`

而是：

`readiness must be re-adjudicated`

换句话说，
reactivation 只把对象带回 current lifecycle，
并没有替它把 readiness 一并签了。

### 5.3 `toggleMcpServer()` 再加一条强证据：显式 enable 也只是重新开启 readiness 流程

`src/services/mcp/useManageMCPConnections.ts:1109-1123`
再给出一个更直接的正例。

在 disabled -> enabled 的路径上，
系统做的不是：

`set connected`

而是：

1. 先 `setMcpServerEnabled(serverName, true)`
2. 再 `updateServer({ type: 'pending' })`
3. 然后才 `reconnectMcpServerImpl(...)`
4. 最后交给 `onConnectionAttempt(result)`

这说明 even explicit enable 也不自动等于 ready。

也就是说，
就算用户已经明确表达了：

`我要让它回来`

系统仍继续坚持：

`回来之后还要再证明你现在可用`

这正是 readiness governance 比 reactivation governance 更强的地方。

### 5.4 `useMcpConnectivityStatus()`、`getMcpStatus()` 与 `/mcp` health 一起证明：surface 层也拒绝把 active truth 冒充成 ready truth

`src/hooks/notifs/useMcpConnectivityStatus.tsx:25-63`
在 user-facing notification 上只盯两种状态：

1. `failed`
2. `needs-auth`

这不是细枝末节。
它说明系统继续追问的是：

`现在能不能用`

而不是：

`它是不是已经在 current world 里`

`src/commands/plugin/ManagePlugins.tsx:513-519`
又把 status 显式枚举为：

`connected / disabled / pending / needs-auth / failed`

而
`src/cli/handlers/mcp.tsx:26-35`
则把 health grammar 压成：

1. `✓ Connected`
2. `! Needs authentication`
3. `✗ Failed to connect`

当 notification、管理界面与 CLI health 都保留这组分层时，
repo 就已经在多个 consumer surface 上明确承认：

`reactivation`

之后仍然有一整层：

`readiness adjudication`

没有被偷掉。

### 5.5 `ReadMcpResourceTool` 给出最强消费侧正例：可用性消费面只接受 `connected`

`src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:86-95`
很硬。

它找到 client 后，
先检查：

`if (client.type !== 'connected') throw ...`

再检查：

`if (!client.capabilities?.resources) throw ...`

这说明 tool consumer 的制度不是：

`只要对象已经在 active world 里，就可以继续消费`

而是：

`只有 readiness verdict 已经强到 connected + capability-present，我才消费它`

这条证据非常值钱。
因为它把 readiness 从 UI 提示提升成了真正的 downstream gate。

从 cleanup 研究角度看，
这意味着将来旧 path、旧 promise、旧 receipt 即便已经 reactivated，
也仍不自动说明：

`下游 reader 已经可以安全消费它。`

### 5.6 `toolExecution.ts` 再证明：readiness 不是静态标签，而是会被运行时证据持续撤销的主权

`src/services/tools/toolExecution.ts:1599-1628`
直接给出一个更强反例。

这里在 `McpAuthError` 时，
会把一个原本 `connected` 的 client 更新成：

`type: 'needs-auth'`

而且只在原状态确实是 `connected` 时才改。

这等于说明：

1. readiness 不是 once-connected forever-connected
2. runtime evidence 可以推翻旧 readiness verdict
3. consumer world 必须接受“刚才还是 ready，现在又不 ready 了”的制度现实

Claude Code 在这里的先进性非常高：

`它不把“历史上曾经连通过”当成永久通行证，而是允许当前证据随时撤销旧的 ready truth。`

## 6. 安全设计的哲学本质：真正被治理的不是“对象是否在场”，而是“对象现在值不值得被继续信任”

如果把这章压到最深处，
Claude Code 在这里展示的并不是普通的连接状态管理，
而是一种更硬的安全哲学：

`presence is not trust, activation is not usability.`

这套哲学至少包含四个原则：

1. `typed honesty over vague availability`
   系统宁可把 `pending / needs-auth / failed` 全部保留下来，
   也不把它们压成“差不多能用”
2. `consumer gate over optimistic inference`
   真正的消费面只接受 `connected`，
   而不是“刚刚 reload 过”
3. `revocable readiness over historical reputation`
   一次 ready verdict 可以被新证据撤销
4. `surface pluralism over single receipt`
   notification、manager、CLI、tool gate 各自显化 readiness，
   而不是只靠一次 activation summary 统治一切读者

所以这一层的哲学本质不是：

`对象有没有回来`

而是：

`回来之后，制度现在愿不愿意继续把它交给别人用。`

## 7. 苏格拉底式自我反思：如果要把这层分析做得更严，最该防什么偷换

第一问：

`我是不是把 active-world fact 误判成了 ready-for-use fact？`

如果是，
`MCPServerConnection` 的 union type 已经直接反驳我。

第二问：

`我是不是把 pending 当成了“马上就好，所以可以忽略”？`

如果是，
`ReadMcpResourceTool` 的 `connected` hard gate 已经直接反驳我。

第三问：

`我是不是把用户触发过 reload 或 enable，就误判成 readiness 已签发？`

如果是，
`initializeServersAsPending()` 与 `toggleMcpServer()` 已经直接反驳我。

第四问：

`我是不是把一次曾经 connected 过，当成了现在仍然 ready？`

如果是，
`toolExecution.ts` 在 `McpAuthError` 时的 runtime downgrade 已经直接反驳我。

第五问：

`我是不是又在 readiness 层偷偷带入了 continuity 或 recovery 的 stronger judgment？`

如果是，
我就需要立刻收手。
因为这一章最严格的边界正是：

`只证明“现在能不能被继续消费”，不越级证明“之后一定会持续可用或一定可被恢复”。`

所以，对这层分析最严的自律不是写得更强，
而是写得更准：

`重新激活，不等于已经就绪；已经就绪，也还不等于后续一切连续性与恢复性问题都已经被回答。`
