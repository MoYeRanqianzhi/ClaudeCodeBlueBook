# 安全能力恢复主权：为什么不是任何层都能把已撤回入口重新恢复为可见可用

## 1. 为什么在 `72` 之后还要继续写“能力恢复主权”

`72-安全能力撤回主权` 已经回答了：

`能力撤回和能力发布一样都是主权动作，不是任何镜像层都能越权决定。`

但如果继续往下追问，  
还会出现一条同样重要、并且和发布/撤回对称的问题：

`那谁有资格把一个曾经被撤回的入口重新恢复为“现在可见、现在可用”？`

这不是普通刷新问题。  
因为入口一旦被恢复，  
系统实际上是在重新对用户作出一条承诺：

`之前那段能力现在又重新成立了。`

所以在 `72` 之后，  
安全专题必须再补一条闭环原则：

`能力恢复主权。`

也就是：

`不是任何局部状态变化、任何镜像层回暖、任何 adapter 直觉都能让入口重新出现；只有掌握原始发布条件的 authoritative publisher，才有资格把“重新可用”变成前台事实。`

## 2. 最短结论

从源码看，Claude Code 已经在多个位置把“恢复入口”的主权写成了硬规则：

1. MCP server 被重新启用时，不是界面直接改口，而是 `toggleMcpServer()` 先写 disk enabled state，再标成 `pending`，再走 reconnect  
   `src/services/mcp/useManageMCPConnections.ts:1073-1123`
2. config / plugin / claude.ai server 重新进入有效配置集时，`useManageMCPConnections()` 会把它们作为 `newClients` 加回 `mcp.clients`，并以 `pending` 重新出现在 UI  
   `src/services/mcp/useManageMCPConnections.ts:814-836,923-940`
3. project MCP server 只有重新回到 `approved` 并通过 policy filter，才会重新进入最终 merged configs  
   `src/services/mcp/utils.ts:351-372`  
   `src/services/mcp/config.ts:1164-1188,1231-1248`
4. plugin 变化后的入口恢复也不能靠界面自己 reset，而必须由 `/reload-plugins -> refreshActivePlugins()` 正式消费 `needsRefresh`，并重建 commands / agents / MCP / LSP  
   `src/hooks/useManagePlugins.ts:287-303`  
   `src/utils/plugins/refresh.ts:72-152`
5. channel capability 的恢复也不是按钮自己亮起来，而是 allowlist entry 写入后还要再次通过 `gateChannelServer(...)`；失败则 rollback  
   `src/cli/print.ts:1666-1688,4708-4728`

所以这一章的最短结论是：

`能力恢复不是“又看见了”这么简单，而是一次重新授予。`

我把它再压成一句：

`谁有资格发布入口、谁有资格撤回入口、谁就有资格重新恢复入口。`

## 3. 源码已经说明：Claude Code 把“能力恢复”也当成安全主权在管理

## 3.1 MCP re-enable 不是界面乐观更新，而是受控的重新授予

`src/services/mcp/useManageMCPConnections.ts:1073-1123` 很关键。

这里 `toggleMcpServer()` 的 enabling 分支并不是：

`UI 先把 server 说成又能用了`

而是严格按三步走：

1. `setMcpServerEnabled(serverName, true)`
2. `updateServer({ type: 'pending' })`
3. `reconnectMcpServerImpl(...)`

这说明 capability 的恢复在这里被表达成：

`重新进入受控连接流程`

而不是简单把旧入口翻回绿色。

换句话说，  
恢复入口的条件不是“有人想重新打开它”，  
而是：

`负责原始能力发布的 config + reconnect authority 已重新批准并重新接线。`

## 3.2 新配置重新出现时，谁能把 client 放回 UI，也是主权动作

`src/services/mcp/useManageMCPConnections.ts:814-836` 说明：

当新的 configs 出现时，  
系统会计算：

`newClients`

并把它们作为：

1. `disabled`
2. 或 `pending`

重新并入 `mcp.clients`。

同文件 `923-940` 又说明：

claude.ai servers 出现时也不是等完全连通后才出现，  
而是：

`Add ... as pending immediately so they show up in UI`

这很重要。  
因为它表明 capability restoration 不是随意出现在任意层，  
而是：

`由配置主权层先正式把对象重新纳入“当前有效候选集合”，再由连接层推进到 connected。`

也就是说，  
恢复入口时前台说的第一句合法话往往不是：

`已经恢复`

而是：

`重新进入了受控恢复流程`

这比很多系统常见的乐观“已回来了”要严谨得多。

## 3.3 project approval 决定了 server 何时才配“重新存在”

`src/services/mcp/utils.ts:351-372` 定义的 `approved / rejected / pending`，  
在 `src/services/mcp/config.ts:1164-1188` 与 `1231-1248` 里真正被转换成了配置事实：

1. 只有 `approvedProjectServers` 才会进入 merge
2. merge 之后还要再过 policy filter

这说明 project MCP server 的恢复不是：

`配置文件还在，所以它又算活了`

而是：

`审批重新通过，并且当前 policy 也重新允许，它才配回到最终能力集合。`

所以这里最该强调的哲学是：

`恢复不是回忆旧状态，而是重新经过授权。`

## 3.4 plugin 能力的恢复必须由 `/reload-plugins` 正式消费 `needsRefresh`

`src/hooks/useManagePlugins.ts:287-303` 已经明确写出：

1. disk state changed 后只发通知
2. `Do NOT auto-refresh`
3. `Do NOT reset needsRefresh`
4. 真正消费者是 `/reload-plugins`

对应的真正恢复动作在 `src/utils/plugins/refresh.ts:72-152`：

1. 清 caches
2. 重新 `loadAllPlugins()`
3. 重建 plugin commands
4. 重建 agentDefinitions
5. 重新填充 plugin MCP / LSP
6. `needsRefresh: false`
7. bump `pluginReconnectKey`

这里非常清楚地说明：

`插件能力的恢复不是靠通知层说一句“现在好了”，而是要由正式的 refresh authority 完成整套重建后才能改口。`

这也是一种典型的 capability restoration sovereignty：

`只有那个真正负责重建能力全集的路径，才配把入口恢复回来。`

## 3.5 channel capability 的恢复也必须服从 allowlist authority，并允许 rollback

`src/cli/print.ts:1666-1688` 说明：

如果 allowlist 不成立，  
`exp['claude/channel']` 会被删除，  
按钮不该出现。

而 `4708-4728` 更能说明恢复过程：

1. 先尝试把 entry 写入 `allowedChannels`
2. 再调用 `gateChannelServer(...)`
3. 如果 gate 失败，则 rollback 到 prior
4. 只有 gate 通过，才正式注册 channel notifications

这说明 channel capability 的恢复也不是：

`有个按钮、点一下、希望它亮`

而是：

`allowlist authority 先临时授予，再由 gate authority 复核，失败立即回滚。`

这是一种很成熟的恢复设计：

`恢复动作必须天然支持 rollback。`

因为恢复本身也是一种高风险主权变更。

## 3.6 第一性原理：恢复不是回到旧真相，而是重新获得新真相

如果从第一性原理追问：

`为什么恢复入口也不能交给任意层自由处理？`

因为一项能力一旦被撤回，  
它之前的合法性已经失效。  
此后它要重新出现，  
并不是在“延续旧授权”，  
而是在：

`重新获得一份新的、当前时刻的授权。`

所以 capability restoration sovereignty 的第一性原理可以压成一句：

`restoration is re-authorization.`

进一步展开，就是：

1. 能力撤回后，旧凭据不再自动延续
2. 能力恢复必须重新经过对应 gate
3. 局部回暖信号不能直接等价于正式恢复
4. 能力恢复要么由 authoritative publisher 完成，要么就不该被宣布

## 4. 苏格拉底式自问：为什么不能让界面自己“看情况恢复一下”

### 4.1 如果某个对象重新出现了，为什么不直接把按钮放回来

因为“对象重新出现”只说明观测面变了，  
不等于发布主权已经重新授予。

例如：

1. config 可能回来了，但 approval 未重新通过
2. plugin 文件可能变了，但 `/reload-plugins` 还没消费
3. channel allowlist entry 可能加上了，但 gate 还没过

观测恢复不等于授权恢复。

### 4.2 如果先把入口恢复出来，失败了再撤回，不也可以吗

这会把“恢复”退化成试错。

对高安全控制面来说，  
这和“先发布、后撤回”的问题完全对称：

`它会制造错误承诺。`

### 4.3 为什么恢复路径里经常先出现 `pending`

因为真正成熟的系统知道：

`正式恢复之前，还有一段受控恢复中的中间态。`

先说 `pending`，  
比直接说 `available` 更诚实。

### 4.4 为什么 rollback 这么重要

因为恢复动作本身就在尝试重新把一段能力借给前台。

如果无法 rollback，  
那一旦 gate 判断失误，  
错误入口就会立刻形成事实。

## 5. 技术先进性与技术启示

这一章最重要的技术判断是：

`Claude Code 的先进性不只在于它会保护能力不被误发布，也在于它会保护能力不被误恢复。`

它先进的地方有六点：

1. 恢复入口先进入 `pending`，而不是直接说 fully available
2. re-enable 路径总是绑定正式 reconnect / refresh，而不是局部状态翻转
3. project server 的恢复继续服从 approval + policy 双重门
4. plugin 能力恢复必须由统一 refresh authority 消费 `needsRefresh`
5. channel capability 恢复支持 gate 失败后的即时 rollback
6. 恢复被建模成重新授权，而不是旧授权自动续命

对其他 Agent 平台构建者的直接启示有六条：

1. 把 capability restoration 当成独立于 publish / revoke 的第三个主权动作
2. 恢复前优先输出 `pending` 这类诚实中间态
3. 不要让任何局部镜像层仅凭“对象又看见了”就恢复入口
4. 恢复路径必须重新经过对应 gate，而不是复用旧结果
5. 允许恢复失败后 rollback，而不是半恢复悬空
6. 设计评审时必须单独问：谁有权恢复、凭什么恢复、失败后谁负责收回

## 6. 一条硬结论

对 Claude Code 这类安全控制面来说，  
真正危险的不是入口短暂不可用，  
而是：

`没有重新经过对应主权层授权，就把一个曾经撤回的入口又重新说成“现在可用”。`

所以比“发布主权 + 撤回主权”更完整的一条原则是：

`能力的发布、撤回与恢复，必须由同一条 authoritative sovereignty chain 连续签字。`
