# 统一安全控制台交互状态机：页面切换、动作触发、回读刷新与宿主适配为什么必须统一设计

## 1. 为什么在 `25` 之后还要继续下钻到“交互状态机”

`25-统一安全控制台最短诊断路径` 已经把统一安全控制台压成了：

1. 状态
2. 断链
3. 动作
4. 回读验证

但如果继续往真正可实现的产品层落，仍然还差一层：

`这些动作和回读，如何在不同页面、不同宿主、不同适配器里保持同一条因果链？`

这不是界面润色问题，  
而是安全控制面能不能真正落地的问题。

因为用户一旦开始：

1. 从总览页进入某张卡片
2. 在卡片里触发控制动作
3. 等待刷新
4. 切回别的页面或换一个宿主查看结果

如果系统不能持续保留同一个“结构化安全对象”，  
那么前面所有状态解释都会再次退化成：

`一串零散提示。`

所以这一章的核心问题是：

`Claude Code 的统一安全控制台，为什么必须进一步被设计成一台跨页面、跨动作、跨宿主的交互状态机？`

## 2. 最短结论

从源码看，Claude Code 现在真正还缺的，不是更多安全零件，  
也不是更多按钮，  
而是：

`统一交互状态机。`

更准确地说，是缺：

`从结构化状态 -> 页面投影 -> 动作触发 -> 刷新回读 -> 结果验证 这一整条链的统一保真。`

当前系统已经有：

1. 足够多的控制动作
2. 足够多的状态字段
3. 足够多的局部安全界面
4. 某些已经成熟的审批闭环

但它仍然容易在宿主边界和轻量提示层发生一件事：

`结构被过早压扁。`

一旦压扁，控制台就不得不重新猜测：

1. 当前到底是什么状态
2. 这条状态对应哪条证明链
3. 应该给什么动作
4. 动作之后又该回读什么

所以 `26` 的核心判断是：

`统一安全控制台的下一个缺口，不是字段，也不是卡片，而是交互状态机。`

## 3. 源码已经给出了动作面和刷新面

## 3.1 `controlSchemas.ts` 已经暴露出足够完整的控制动作

`src/entrypoints/sdk/controlSchemas.ts:157-173`、`405-519` 已经给出一组很关键的控制请求和返回结构：

- `mcp_status`
- `reload_plugins`
- `mcp_reconnect`
- `mcp_toggle`
- `apply_flag_settings`
- `get_settings`

这说明系统并不缺“控制动作层”。  
它真正缺的是：

`哪类状态应该绑定哪类动作，以及动作之后应该自动回读哪组事实。`

## 3.2 `get_settings` 本身已经是天然的“主权回读接口”

`SDKControlGetSettingsResponseSchema` 返回的不只是 `effective`，  
还返回：

- `sources`
- `applied`

也就是说，动作完成后系统完全可以再回答三件事：

1. 合并后的设置长什么样
2. 每个来源各自说了什么
3. 运行时最终真正会使用什么

这已经非常接近：

`动作后的主权链回读。`

所以统一安全控制台真正缺的不是“读设置”能力，  
而是：

`把动作和主权回读预先绑定。`

## 3.3 `reload_plugins` 和 MCP 动作已经接近完整的外部能力闭环

`reload_plugins` 会返回刷新后的：

- commands
- agents
- plugins
- `mcpServers`

`mcp_status`、`mcp_reconnect`、`mcp_toggle` 则分别对应：

1. 读当前连接状态
2. 尝试恢复失败连接
3. 显式切换启停状态

这已经几乎构成一套完整的“外部能力卡交互闭环”。  
它距离统一安全控制台只差一步：

`把这些动作放进同一个交互状态机，而不是散落成协议能力。`

## 4. 当前真正的问题：结构化状态在宿主边界被过早压扁

## 4.1 `sdkMessageAdapter.ts` 把 `status` 压成薄文本

`src/remote/sdkMessageAdapter.ts:88-103` 的 `convertStatusMessage(...)` 会把 `SDKStatusMessage` 投影成：

- `Compacting conversation…`
- `Status: ${msg.status}`

这不是错误。  
但这说明一个更深的问题：

`结构化状态对象在进入某些宿主前，就已经被削成了文本。`

而 `src/remote/sdkMessageAdapter.ts:228-256` 还进一步显示：

1. `status` 只保留为轻量信息文本
2. `auth_status` 被直接忽略

所以一旦跨到某些适配器边界，  
原本可以驱动控制台动作和验证链的结构，  
就退化成了：

`可显示但不可推理的提示语。`

## 4.2 `bridgeStatusUtil.ts` 和 `PromptInputFooter.tsx` 进一步证明状态被做成了轻提示

`src/bridge/bridgeStatusUtil.ts:113-140` 把 bridge 连接状态压成四个标签：

- `Remote Control failed`
- `Remote Control reconnecting`
- `Remote Control active`
- `Remote Control connecting…`

`src/components/PromptInput/PromptInputFooter.tsx:157-189` 又说明：

1. footer 只是读取几项 bridge 布尔状态
2. 失败态不经 footer pill，而经 notification 暴露
3. implicit remote 甚至只显示 reconnecting 状态

这说明 bridge 状态当前主要被设计成：

`轻量 footer 提示。`

而不是：

`统一安全控制台中的结构化控制对象。`

所以这里暴露出的不是“bridge 没状态”，  
而是：

`bridge 状态已有，但没有被保真地接入统一交互状态机。`

## 5. 一个已经成形的正例：受管 settings 其实有真正的交互状态机

## 5.1 `remoteManagedSettings/index.ts` 已经实现了完整闭环

`src/services/remoteManagedSettings/index.ts:420-547` 很重要，  
因为这里已经不是零散按钮，  
而是一条完整交互链：

1. 读取 cached settings
2. 带 checksum 发起拉取
3. `304` 时沿用 cache
4. 有新内容时先做危险变更安全检查
5. 用户拒绝则回退到 cache 或退出
6. 用户接受则写入并应用
7. 最后触发 `notifyChange('policySettings')` 做热刷新

这已经是成熟的：

`拉取 -> 判定 -> 审批 -> 应用/回退 -> 刷新`

状态机。

## 5.2 `securityCheck.tsx` 说明它不仅有动作，还有阻断式用户决议

`src/services/remoteManagedSettings/securityCheck.tsx:22-72` 继续证明：

1. 无危险变化时不打断
2. 非交互模式跳过对话框
3. 交互模式弹出阻断式确认
4. 接受和拒绝都被记录为显式结果
5. 拒绝会触发 `gracefulShutdownSync(1)`

这意味着系统已经掌握了一种非常成熟的安全交互原则：

`不是所有状态都只该提示，有些状态必须进入显式决议。`

也就是说，统一安全控制台未来并不需要发明一套全新哲学，  
它更需要做的是：

`把这种局部成功模式推广成跨域统一模式。`

## 6. 从第一性原理看，统一安全控制台最少要有六个交互状态

我会把统一安全控制台的最小交互状态机压成六段。

## 6.1 `Observe`

系统先暴露一份结构化安全快照，  
回答：

`现在发生了什么。`

这里不只是显示文案，  
而是保留：

- 状态对象
- 来源对象
- 当前证明链位置
- 推荐动作集合

## 6.2 `Diagnose`

系统接着回答：

`是哪条证明链断了。`

这一步把状态压回：

- 主体链
- 主权链
- 授权链
- 包络链
- 外部能力链
- 增量审批链

没有这一步，  
动作就会失去因果基础。

## 6.3 `Act`

系统给出最短动作，  
动作必须是：

1. 可执行
2. 低误操作
3. 与当前断链直接绑定

也就是把 `controlSchemas.ts` 里的协议能力、  
本地设置页跳转、  
审批对话框，  
统一当成安全控制动作看待。

## 6.4 `Refresh`

动作执行后，系统必须自动回读真相源。  
这一步对应：

- `mcp_status`
- `get_settings`
- bridge / auth 状态重新读取
- managed settings 的热刷新

没有刷新，  
控制台就只能告诉用户：

`动作已经发出。`

却不能告诉用户：

`动作是否真的生效。`

## 6.5 `Verify`

刷新之后，系统必须显式比较 before / after，  
回答：

`哪条证明链已经修复，哪条还没修复。`

这一步决定控制台是不是“证明面”，  
而不只是“操作面”。

## 6.6 `Escalate`

如果动作后仍未修复，  
控制台必须把问题升级到正确归属方：

- 用户本地
- 组织管理员
- 平台侧 entitlement / account 面
- 宿主适配器实现方

也就是说，真正成熟的统一安全控制台并不以“按钮按完”为终点，  
而以：

`证明修复或正确升级`

为终点。

## 7. 页面切换、动作触发、回读刷新与宿主适配为什么必须统一设计

## 7.1 页面切换如果不共享同一状态对象，诊断就会被迫重来

如果总览页、详情页、动作面板和验证页各自重新解释状态，  
那系统就会在不同页面重复做同一件事：

`重新猜测原因。`

这会直接制造解释漂移。

所以页面切换的本质不是导航问题，  
而是：

`同一结构化安全对象能不能被连续携带。`

## 7.2 动作触发如果没有默认回读，就会制造“假成功”

用户最常见的错觉不是“没有按钮”，  
而是：

`我已经点了，为什么还是不清楚有没有修好。`

这正是因为很多系统把“动作发出”误当成“问题解决”。  
而 Claude Code 的源码已经证明，这种设计其实可以更好：

- MCP 可回读状态
- settings 可回读来源
- managed settings 可热刷新

所以问题不是技术做不到，  
而是交互层还没统一。

## 7.3 回读刷新如果不和动作绑定，就无法形成安全证明

安全控制台和普通设置页最大的区别是：

`它必须负责证明。`

也就是说，每个动作都不只要有“执行”语义，  
还要有：

1. 默认刷新源
2. 默认对比字段
3. 默认成功判据

否则控制台只能说：

`大概好了。`

而不能说：

`哪条链已经恢复成立。`

## 7.4 宿主适配如果在前面就压扁结构，后面所有页面都会失真

这正是 `sdkMessageAdapter.ts` 暴露出的根问题：

`适配器过早决定了状态只能以文本存在。`

一旦如此，  
后续统一控制台即使页面设计得再好，  
也只能在压扁后的残余信号上做二次推断。

所以真正该统一的不是某个具体 UI，  
而是：

`宿主适配契约。`

## 8. 统一交互状态机真正该守住的四条适配器公理

如果把这一章进一步压成产品实现原则，我会得到四条公理。

## 8.1 先保留结构，再决定怎样显示

宿主可以因为空间受限只显示 pill 或短文案，  
但内部不能先把结构化安全对象销毁。

## 8.2 每个动作都必须绑定默认刷新源

动作不是孤立按钮。  
动作必须天然知道：

- 去哪里回读
- 该比较哪些字段

## 8.3 每次刷新都必须保留前后差分

没有 before / after，  
就没有真正的“验证”。

## 8.4 不支持的宿主必须显式降级，而不是静默忽略

`auth_status` 这类信号如果某个宿主不能完整展示，  
更合理的做法是显式声明降级，  
而不是把它静默抹去。

否则用户感受到的就不是“窄宿主”，  
而是：

`状态黑箱。`

## 9. 技术先进性与设计启示

这章最重要的技术判断是：

`Claude Code 的先进性不只在于安全机制多，而在于它离“可恢复、可证明、可跨宿主保真”的安全控制台已经很近。`

它已经具备：

1. 协议层动作
2. 来源回读能力
3. 局部审批闭环
4. 连接状态面
5. 宿主事件流

它现在真正缺的是：

`把这些零件变成统一因果链的产品层。`

这对其他 Agent 平台构建者的启示非常直接：

1. 不要把安全 UI 做成设置页集合
2. 不要把状态消息做成纯文本提示
3. 不要把动作和刷新拆开
4. 不要把宿主适配当成可以任意压缩结构的显示层

真正成熟的安全产品，不是“能检测”，  
而是：

`能把检测、动作、刷新和证明连成一条不丢因果的链。`

## 10. 苏格拉底式反思：这章最容易自欺的地方是什么

### 10.1 我们是不是把“页面多”误当成“系统成熟”

不是。  
页面数量并不重要。  
重要的是：

`同一个结构化安全对象能否跨页面流动。`

### 10.2 窄宿主是不是就只能牺牲解释性

也不是。  
窄宿主可以牺牲展示密度，  
但不该牺牲结构保真。

### 10.3 统一控制台是不是等于把所有状态都集中到一个大页面

不是。  
统一的首先应是状态机和数据契约，  
不是视觉上只有一页。

### 10.4 动作越多是不是越好

也不是。  
动作只有在绑定断链原因和默认回读时才真正有价值。  
否则只是噪声。

## 11. 结语

`25` 解决的是最短诊断闭环。  
`26` 继续推进后的关键结论是：

`统一安全控制台真正离落地只差一层交互状态机。`

它要求系统不再把安全状态当成散落在不同页面的提示，  
而要把它们当成：

`可跨宿主携带、可驱动动作、可回读验证、可显式升级的结构化证明对象。`
