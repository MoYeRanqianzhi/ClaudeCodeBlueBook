# 安全载体家族强请求清理就绪治理与强请求清理连续性治理分层：为什么artifact-family cleanup stronger-request cleanup-readiness-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-continuity-governor signer

## 1. 为什么在 `203` 之后还必须继续写 `204`

`203-安全载体家族强请求清理重新激活治理与强请求清理就绪治理分层` 已经回答了：

`stronger-request cleanup 线即便已经把 truth 接进当前世界，也还需要 readiness governor 去决定这个当前世界现在到底能不能被继续消费。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成“既然现在已经 ready，那接下来当然也会继续可用”的错觉：

`只要 stronger-request cleanup readiness governor 已经判定对象此刻 connected、capable、downstream-consumable，它就自动拥有了决定这份可用性在断连、退化、重试、放弃、人工重启与时间流逝中怎样继续成立的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看：

1. `src/services/mcp/useManageMCPConnections.ts:87-90,333-466` 的 automatic reconnection、指数退避、最大重试次数与最终 give-up 语义
2. `src/services/mcp/useManageMCPConnections.ts:765-853` 的 stale reconnect timer 清理、pending 重建与 `/reload-plugins` continuity reseed
3. `src/services/mcp/useManageMCPConnections.ts:1043-1123` 的 manual reconnect、disable cancel 与 re-enable reconnect path
4. `src/services/tools/toolExecution.ts:1599-1628` 的 `connected -> needs-auth` runtime downgrade
5. `src/cli/print.ts:1392-1426` 的 `hasPendingSdkClients / hasFailedSdkClients` re-init path
6. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:78-95` 的 `connected` hard gate

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup readiness governance` 负责回答对象现在是不是 ready for use
2. `stronger-request cleanup continuity governance` 负责回答 ready truth 一旦断裂、退化、失鉴权、超时、被人工干预或被新世界替换时，这份可用性怎样继续维持、重启、放弃或被正式撤销

也就是说：

`artifact-family cleanup stronger-request cleanup-readiness-governor signer`

和

`artifact-family cleanup stronger-request cleanup-continuity-governor signer`

仍然不是一回事。

前者最多能说：

`它现在能用。`

后者才配说：

`它接下来若掉出 ready、需要重试、被人工停表、被 stale timer 污染，系统由谁决定继续争取、重新开线还是正式终止这份可用性。`

所以 `203` 之后必须继续补的一层就是：

`安全载体家族强请求清理就绪治理与强请求清理连续性治理分层`

也就是：

`stronger-request cleanup readiness governor 决定对象现在能不能用；stronger-request cleanup continuity governor 才决定这种可用性怎样跨时间维持、退化、重建或终止。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-continuity-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-continuity-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 continuity manager，
而是在说：

1. repo 已经把 “当前 ready” 与 “时间里继续 ready” 拆成两层
2. repo 已经把 retry、backoff、give-up、cancel、reseed 与 pool re-init 写成正式 grammar
3. repo 已经拒绝让一次 current-time ready verdict 越级冒充 temporal continuity verdict

第二条：

这里说的 `continuity governance`

不是在声称：

`对象一旦 ready，就必须永远保持可用。`

当前更稳的说法只能是：

`continuity governance 首先回答的是 ready truth 中断后系统还要不要继续为它争取可用性，以及争取到什么程度。`

第三条：

这里也不是把：

`auto-reconnect`

单独当成全部 continuity grammar。

真正关键的是：

1. retry budget
2. stale continuity revocation
3. operator stop / restart control
4. runtime downgrade 后的重新裁决
5. pool-level repair

第四条：

这里尤其不能过度声称：

`continuity governance`

已经等于：

`recovery governance`

当前更稳的说法只能是：

`continuity governance 回答旧的 ready line 还要不要继续；recovery governance 才回答什么新的证据足以重新宣布 recovered truth 成立。`

第五条：

这里也不能过度声称：

`某一刻读成功一次`

就等于：

`后续时间里都可以继续依赖它。`

当前更稳的说法只能是：

`single successful read 只是 current-time consumption proof，不是 temporal continuity proof。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类“stronger-request cleanup-readiness-governor signer 仍不等于 stronger-request cleanup-continuity-governor signer”证据：

1. `useManageMCPConnections()` 把断连后的制度直接写成 `MAX_RECONNECT_ATTEMPTS`、指数 backoff 与最终 give-up，说明 ready truth 断掉后还存在单独的 continuity budget
2. stale reconnect timer cleanup 与 stale connected cleanup 说明 repo 继续追问的不是“现在 ready 不 ready”，而是“哪条持续线仍配代表 current world”
3. manual reconnect、disable cancel 与 re-enable reconnect 说明 continuity 不是 background retry 的副作用，而是可被人类显式治理的 stop / restart 主权
4. `toolExecution.ts` 会把一个原本 `connected` 的 client 降成 `needs-auth`，说明 ready truth 会被运行时证据持续撤销，continuity 必须处理这种撤销后的时间线
5. `print.ts` 会因为 `pending / failed` SDK clients 整批 re-init 当前 pool，说明 continuity 不只属于单对象 state，还属于 current usable world 的持续维持

因此这一章的最短结论是：

`stronger-request cleanup readiness governor 最多能说对象现在 ready；stronger-request cleanup continuity governor 才能说这份 ready truth 在断连、退化、重试、放弃与人工干预中怎样继续成立。`

再压成一句：

`现在可用，不等于持续可用。`

## 4. 第一性原理：readiness governance 回答“当前能不能消费”，continuity governance 回答“这种可消费性怎样跨时间维持”

从第一性原理看，
强请求清理就绪治理与强请求清理连续性治理处理的是两个不同主权问题。

stronger-request cleanup readiness governor 回答的是：

1. 当前 client 是否 `connected`
2. 当前 capability gate 是否闭合
3. 当前 consumer 能不能立刻读、调、调用
4. 当前 surface 应显示 `connected / pending / needs-auth / failed / disabled` 中的哪一档
5. 这一刻的 usable truth 是否成立

stronger-request cleanup continuity governor 回答的则是：

1. ready truth 一旦断掉，要不要继续重试
2. 重试最多试几次，以什么 backoff 节奏继续
3. 什么时候该从 “继续争取可用” 转成 “正式 give up”
4. 旧 retry line、旧 timer、旧 closure 何时必须被 current world 撤销
5. manual reconnect、disable、re-enable 与 pool re-init 怎样改写这条可用性时间线

如果把这两层压成一句“反正它刚才已经 connected”，
系统就会制造五类危险幻觉：

1. ready-means-stable illusion  
   只要当前 ready，就误以为接下来也会继续 ready
2. retry-is-incidental illusion  
   误以为 backoff / give-up 只是实现细节，而不是正式治理语义
3. stale-line-is-harmless illusion  
   误以为旧 continuity attempt 留着也没关系，不会冒充 current world
4. operator-intervention-is-ui-noise illusion  
   误以为 manual reconnect 或 disable 只是按钮，不算制度主权
5. one-read-means-dependable illusion  
   误以为一次 current read 成功，就说明它之后仍可持续依赖

所以从第一性原理看：

`readiness governance` 处理现在时的可消费性；  
`continuity governance` 处理时间轴上的可维持性。

再用苏格拉底式反问压一次：

1. 如果 stronger-request cleanup readiness governance 已经等于 continuity governance，为什么 repo 还要单独维护 `MAX_RECONNECT_ATTEMPTS`、指数 backoff 与 final give-up？
   因为 “当前 ready” 与 “ready truth 断掉后还要不要继续争取” 根本不是同一句制度。
2. 如果 ready truth 一旦成立就天然连续，为什么 manual reconnect 前还要取消旧 pending reconnect，disable 时还要停掉重试线？
   因为 continuity 不是自然延长，而是被 current authority 持续裁决的时间过程。
3. 如果当前 consumer 读成功一次就足够代表 continuity，为什么 `toolExecution.ts` 还能把一个已经 `connected` 的 client 降成 `needs-auth`，`print.ts` 还能因 `pending / failed` SDK clients 重新初始化整池？
   因为 ready verdict 不是永久席位，continuity 处理的正是它被撤销之后谁继续负责。

## 5. `useManageMCPConnections()` 先证明：当前 ready 之后，repo 还单独维护一整条 continuity budget

`src/services/mcp/useManageMCPConnections.ts:87-90` 很硬。

这里不是简单写一句“断了就重连”，
而是先定义三条 continuity constitution：

1. `MAX_RECONNECT_ATTEMPTS = 5`
2. `INITIAL_BACKOFF_MS = 1000`
3. `MAX_BACKOFF_MS = 30000`

这已经说明 repo 追问的不是：

`对象现在是不是 ready`

而是：

`这份 ready truth 一旦断掉，系统接下来还要为它争取多久。`

`333-466` 更硬。

这里在 `client.client.onclose` 后做的不是直接宣告“现在不 ready 了”就结束，
而是继续进入一条正式 continuity path：

1. 先检查对象是否已被 disable；若已 disable，则跳过 continuity attempt
2. remote transport 才进入 automatic reconnect
3. 每次 attempt 前把状态更新成 `pending`
4. 每次尝试都可能回到 `connected`
5. 最终可以因为达到上限而进入 give-up
6. backoff 以指数方式增长，而不是平铺重试

这里最值钱的，不是代码里有 retry。

而是 repo 明确拒绝把：

`当前 connected`

偷写成：

`未来仍然可持续 connected`

它要求系统在时间上持续重新裁决。

这正是 stronger-request cleanup continuity governance 的本体。

## 6. stale reconnect timer 清理、manual reconnect 与 toggle path 再证明：continuity 不只是自动重试，而是 current-world revocation 与 operator control

`useManageMCPConnections.ts:765-853` 非常关键。

这里在 `/reload-plugins` 和 session change 场景下，
不仅会把新对象初始化为 `pending`，
还会优先：

1. 清理 stale reconnect timer
2. 对旧 connected client 解除 `onclose`
3. 再按新的 configs 重建当前 pending world

这说明 continuity 不是：

`谁还在自动重试，谁就算继续代表 current world`

而是：

`只有与 current config、current session、current plugin world 对齐的那条持续线，才配继续代表 continuity truth`

`1043-1123` 又把 continuity 主权进一步显化成人工操作：

1. manual reconnect 会先取消已有 pending reconnect，再显式触发新的 reconnect attempt
2. disable 会先取消 pending reconnect，再把对象转成 `disabled`
3. re-enable 会先置成 `pending`，再重新连一次

这条证据非常值钱。
它说明 continuity governance 不是 background retry 的副作用，
而是正式回答：

`继续试、停止试、重新试，分别由谁触发、何时重启、何时停表。`

如果 stronger-request cleanup 线未来只有 readiness grammar，
没有这种 “cancel / reseed / restart / give-up” grammar，
那它就仍然无法正式回答：

`旧 path、旧 promise、旧 receipt 一旦掉出 ready，后面由谁维持它的连续可用性。`

## 7. `toolExecution.ts` 与 `print.ts` 再证明：continuity 处理的是 ready truth 的退化与 world-level repair

`toolExecution.ts:1599-1628` 给出一个特别强的反例。

这里在 `McpAuthError` 时，
会把一个原本 `connected` 的 client 更新成：

`type: 'needs-auth'`

而且只在原状态真的还是 `connected` 时才改。

这说明 repo 已经公开承认：

1. ready truth 会在运行时被撤销
2. 被撤销后，系统不能继续假装 continuity 仍成立
3. continuity governance 必须回答撤销后是等待重鉴权、等待重连，还是正式退出 usable service

`print.ts:1392-1426` 又把这层治理推进到 SDK client pool。

这里不是只检查 “有没有新增 server”，
而是进一步追问：

1. 有没有 `pending` SDK clients 还没升格
2. 有没有 `failed` SDK clients 已经卡死
3. 如果有，就整批重新初始化当前 SDK MCP servers

这说明 continuity 并不只是某个单一连接的故事。
它还回答：

`当前 usable pool 还能不能继续作为一个整体成立。`

从技术角度看，这非常先进。
它把安全从瞬时 gate 提升成了时间中的 system responsibility，
也把多重安全术从 “单次拦截” 提升成了 “持续裁决、持续撤销、持续重建”。

## 8. `ReadMcpResourceTool` 再次提醒：consumer hard gate 只验证 readiness，不自动替 continuity 背书

`ReadMcpResourceTool.ts:78-95` 仍然很硬。

它看到 `client.type !== 'connected'` 就直接拒绝消费。

这说明 consumer gate 的制度只到这里：

`当前 ready，我才读。`

它并不会进一步说：

`既然这一刻能读，所以接下来也一定持续能读。`

换句话说，
consumer gate 证明了 readiness 很硬，
却也反过来暴露出 continuity 需要由别的层单独负责。

这正是本章最关键的一点：

`downstream-safe now`

不等于：

`temporally dependable over time`

## 9. 从技术先进性看：Claude Code 把安全写成多重时间控制面，而不是单一时刻的通过/拒绝

从技术角度看，
Claude Code 在这里最先进的地方，不是它“有重连逻辑”，
而是它把多重安全技术写成了四层互相制约的时间控制面：

1. `state machine as security grammar`  
   它不把对象压成 present/absent，而是明确区分 `connected / pending / needs-auth / failed / disabled`
2. `retry budget as governance`  
   它不把 backoff / give-up 当成 incidental implementation，而是公开写成制度参数
3. `stale-line revocation as anti-impersonation control`  
   它不允许旧 reconnect timer、旧 closure、旧 client 在 current world 里继续冒充合法 continuity truth
4. `runtime downgrade + pool repair as continuous adjudication`  
   它不把安全写成“一次过闸终身有效”，而是允许 ready truth 被持续撤销，并在 pool-level 上重建 current usable world

这套设计的哲学本质是：

`安全不是一次性的许可事件，而是一个在时间中不断被重申、被撤销、被修复、被终止的治理过程。`

对 stronger-request cleanup 线的技术启示也因此非常清楚：

如果未来只补出 “旧 truth 现在 ready 了”，
却没有补出 “ready truth 断掉后谁来继续争取、谁来停表、谁来撤销旧线、谁来重建当前世界”，
那么这条线依然只是 current-time safety，
还不是 temporal safety。

## 10. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 线只要补出 readiness governance，就已经足够成熟。`

而是：

`repo 在 useManageMCPConnections 的 auto-reconnect / backoff / give-up / cancel path、stale reconnect timer cleanup、manual reconnect 与 toggle control、toolExecution 对 connected client 的 runtime downgrade、print 对 pending / failed SDK clients 的 pool-level re-init，以及 ReadMcpResourceTool 的 current-time hard gate 上，已经明确展示了 continuity governance 的存在；因此 artifact-family cleanup stronger-request cleanup-readiness-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-continuity-governor signer。`

因此：

`stronger-request cleanup readiness governance 能回答“它现在能不能用”；stronger-request cleanup continuity governance 才能回答“这种可用性怎样在时间里继续成立”。`
