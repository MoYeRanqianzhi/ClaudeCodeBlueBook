# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `453` 之后还必须继续写 `454`

`453-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界何时正式结束，也仍需要 tombstone governor 去决定结束后留下什么最小残留事实。`

但如果继续往下追问，
还会撞上一层更容易被偷写成
`只要墓碑已经留下或清掉，后面怎样回来也就自动清楚了`
的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定旧对象退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、回来时算谁、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 resurrection 正对照，
会发现它其实明确拆开了五件事：

1. `authoritative clearing`
2. `re-entry gate`
3. `mode-sensitive admission`
4. `identity policy`
5. `active-layer admission`

最硬的证据至少有八组：

1. `src/utils/plugins/cacheUtils.ts:67-72,82-92,122-130`
   证明 `.orphaned_at` 的清除由 authoritative installed set 决定，
   而不是 marker 自己决定
2. `src/utils/plugins/refresh.ts:1-17,59-79,123-137`
   证明 active plugin return 是专门的 `Layer-3 refresh primitive`，
   而不是 marker 清掉后的自动副作用
3. `src/commands/reload-plugins/reload-plugins.ts:37-56`
   证明 re-entry 需要显式 signal，
   系统不会把“磁盘世界变了”偷写成“active world 已接纳”
4. `src/hooks/useManagePlugins.ts:28-35,288-303`
   证明 interactive 模式下 `needsRefresh` 只会通知用户运行 `/reload-plugins`，
   而且明确 `Do NOT auto-refresh`
5. `src/services/plugins/PluginInstallationManager.ts:53-58,135-160`
   进一步证明 repo 会按 return path 区分 admission grammar：
   新安装可 auto-refresh，updates-only 则只停在 `needsRefresh`
6. `src/utils/plans.ts:164-231,233-257`
   证明 plan resurrection 不只是 undelete，
   还要治理 lineage、evidence 与 identity policy
7. `src/utils/conversationRecovery.ts:539-547`
   证明 resurrection grammar 被接进正式恢复主线，
   而不是局部 helper 的偶发行为
8. `src/utils/sessionStorage.ts:925-945,1470-1473`
   的 tombstone path 只回答怎样删旧对象，
   不回答它何时、以什么身份重新回来

这些源码放在一起看，
已经非常清楚地说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 identity policy 与什么 active-layer admission 下，这个旧对象还能重新回到 current world。`

所以 `453` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`

不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk / materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery`

就等于 resurrection 的全部语义。

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线未来的复活 grammar 仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

第六条：

这里还不能过度声称：

`Claude Code 的 resurrection 只是恢复体验优化。`

更稳的说法只能是：

`它本质上是在治理 current world 的重新接纳权；体验只是外观，资格与接纳才是本体。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer`

证据：

1. `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
   `cacheUtils.ts:67-72,82-92,122-130`
2. `refreshActivePlugins()` 与 `/reload-plugins` 证明 marker clearing / disk presence
   不等于 active components 已重新进入当前 session
   `refresh.ts:1-17,59-79,123-137`; `reload-plugins.ts:37-56`
3. `needsRefresh` 明确展示 repo 不会把 interactive 场景下的 comeback 偷写成自动接纳
   `useManagePlugins.ts:28-35,288-303`
4. repo 还会根据 return path 区分 auto-refresh 与 explicit-refresh
   `PluginInstallationManager.ts:53-58,135-160`
5. `copyPlanForResume()` 证明 resurrection 需要 lineage 与 evidence
   `plans.ts:164-231`
6. `copyPlanForFork()` 证明“回来”不等于“以原身份回来”
   `plans.ts:233-257`
7. `conversationRecovery.ts` 把 resurrection 接进恢复主线，
   说明它是正式 recovery grammar
   `conversationRecovery.ts:539-547`
8. `sessionStorage.ts` 的 tombstone path 只回答怎样删除旧对象，
   不回答它何时、以什么身份重新回来
   `sessionStorage.ts:925-945,1470-1473`

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据、何种身份政策与何种激活层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

再压成更哲学一点的话：

`制度知道怎样留下历史痕迹，不等于制度已经知道谁配重新被当前世界承认。`

## 4. 第一性原理：tombstone governance 回答“留什么最小残留事实”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

`stronger-request cleanup tombstone governor`
回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

`stronger-request cleanup resurrection governor`
回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk / materialized world，还是 active / runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句：

`反正 marker 还在或 marker 已清掉`

系统就会制造五类危险幻觉：

1. `marker-clear-means-resurrection`
2. `disk-return-means-active-return`
3. `content-recovered-means-identity-restored`
4. `tombstone-is-bidirectional`
5. `any-old-object-can-come-back`

所以这两层的哲学本质也不同：

`tombstone`

治理的是：

`过去的无效性如何继续被最小地保存。`

`resurrection`

治理的则是：

`当前的有效性怎样被重新签发。`

如果再压缩一点，
两者的本质差别其实是：

1. `tombstone` 负责 `preserving the fact of invalidity`
2. `resurrection` 负责 `reissuing the right to count as current`

过去已经失效这一事实，
永远不能自动推出现在重新有效这一资格。

## 5. Claude Code 多重安全技术的先进性：它把 comeback 写成分层接纳，而不是把 return 偷写成副作用

Claude Code 在这一层真正先进的地方，
不是“旧对象又出现了，所以系统顺手把它当 current”，
而是：

`它把 comeback 写成一个分层、分模式、分身份的 current-world admission problem。`

这种先进性至少体现在六个方面：

### 5.1 authoritative state clears the marker

`.orphaned_at` 不是自己消失，
也不是被任意读取者顺手忽略，
而是由 installed set 的 authoritative truth 决定何时清除。

### 5.2 three-layer re-entry model

`refresh.ts` 把意图层、物化层、active components 层单列出来，
拒绝把 disk materialization 偷写成 active admission。

### 5.3 explicit admission signal

交互式回归必须通过 `/reload-plugins`，
而不是“磁盘变了所以会话自动理解为已经回来了”。

### 5.4 mode-sensitive return policy

同样是 comeback，
新安装与更新不会走一条 admission path。
repo 明确允许某些 return path auto-admit，
同时坚持另一些 path 必须 explicit-admit。

### 5.5 lineage and evidence gating

plan resume 不是看到旧内容就直接塞回来，
而是先检查 slug、session、ENOENT 条件、remote context 与 recoverable evidence。

### 5.6 identity split

`copyPlanForFork()` 直接把 resurrection 问题切开：
内容可以回来，
但身份未必回来。

这些设计背后的源码思路也很清楚：

1. 不把 marker-clear 当成 active-world admission
2. 不把 disk presence 当成 runtime legitimacy
3. 不把 content continuity 当成 identity continuity
4. 不把所有 return path 写成同一种 admission policy
5. 不把 resurrection 伪装成 deletion helper 的副作用

## 6. 为什么 stronger-request cleanup 现在仍停在“post-retirement marker grammar”，而不是“re-entry authority grammar”

和这些成熟 resurrection 正对照相比，
stronger-request cleanup 当前最明显的缺口已经不是：

`旧世界结束后留下什么最小痕迹。`

而是：

`这些带碑对象凭什么重新回来。`

更具体地说，
现在仍没有哪一层正式回答：

1. 哪些旧 startup wording 配重新进入 current world
2. 旧 cleanup law 回来时算 old identity 还是 new identity
3. 旧 promise vocabulary 是 content-only return，还是资格也一起 return
4. 旧 receipt objects 回来时需要什么 evidence、什么 lineage
5. 哪些 comeback path 可以 auto-admit，哪些只能 pending 再 explicit-admit

换句话说，
stronger-request cleanup 当前虽然已经开始显露 tombstone-governance，
却还没有长出足够正式的：

1. authoritative clearing plane
2. explicit re-entry gate
3. mode-sensitive admission policy
4. lineage / evidence gate
5. identity policy
6. active-layer admission rule

这意味着它现在能说的是：

`旧世界结束后留下什么最小机器事实。`

却还不能更强地说：

`哪些带碑对象在什么资格、什么证据与什么 admission layer 下可以重新被当前世界承认。`

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界结束后留下了 marker”误认成了“系统已经知道谁配回来”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 tombstone grammar 已经足够强，为什么还要再拆 resurrection？
   因为会留下标记，不等于知道谁配重新进入 current world。
2. 如果 `.orphaned_at` 会被清掉，是不是就说明对象已经完全回来了？
   不是。marker clearing 只是撤销退役标记，不等于 active world 已接纳。
3. 如果磁盘上对象已经恢复，是不是就说明当前 session 可以继续用？
   也不对。disk/materialized return 和 active/runtime return 被明确拆层。
4. 如果 plan 内容成功恢复，是不是就说明还是原来的 plan？
   也不对。resume 与 fork 已明确拆成 old slug 和 new slug 两种身份政策。
5. 如果 deletion helper 已经很完善，是不是 comeback grammar 也就自然成立？
   不是。删除的单向语义恰恰反证 resurrection 必须被单独签字。
6. 如果 cleanup 线现在还没有显式 resurrection 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层正式 admission grammar，越容易把“可恢复的历史对象”误写成“已被当前世界重新承认”。

这一串反问最后逼出一句更稳的判断：

`resurrection 的关键，不在旧对象能不能重新出现，而在系统能不能正式决定谁配重新被当前世界承认、以什么身份承认、以及承认到哪一层。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 tombstone governance，就已经足够成熟。`

而是：

`Claude Code 在 authoritative marker clearing、Layer-3 refresh primitive、explicit /reload-plugins gate、needsRefresh pending world、lineage-based plan recovery 与 new-slug identity policy 上已经明确展示了 resurrection governance 的存在；因此 artifact-family cleanup stronger-request cleanup-tombstone-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁来给旧世界立碑”，而是“谁来决定哪些带碑对象还能回来、回来时算谁、回到哪一层当前世界”。`
