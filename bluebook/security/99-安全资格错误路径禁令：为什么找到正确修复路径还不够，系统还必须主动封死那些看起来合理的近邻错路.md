# 安全资格错误路径禁令：为什么找到正确修复路径还不够，系统还必须主动封死那些看起来合理的近邻错路

## 1. 为什么在 `98` 之后还必须继续写“错误路径禁令”

`98-安全资格默认动作路由` 已经回答了：

`每一种资格状态都应绑定唯一 dominant route。`

但如果继续往下追问，  
还会遇到一个更尖锐的问题：

`系统把正确路径指出来之后，是否就已经足够安全了？`

答案是否定的。

因为现实里最危险的往往不是“完全乱来的动作”，  
而是：

`那些看起来很像正确路径、却会偷偷破坏资格真相的近邻动作。`

例如：

1. retryable resume 失败后立刻清 pointer
2. plugin changed 后自动 refresh 一部分状态
3. stale MCP client 继续沿旧 timer / 旧 config 回连
4. bridge 缺 OAuth 时先给用户 policy 错误而不是 `/login`
5. CCR 模式 transient auth error 时错误地指向 `/login`

所以 `98` 之后必须继续补出的下一层就是：

`安全资格错误路径禁令`

也就是：

`仅有正确默认动作还不够，系统还必须显式封死那些极易被误选的近邻错路。`

## 2. 最短结论

从源码看，Claude Code 在多个子系统里都不只是提供正确修复路径，  
还在主动阻止错误近邻动作：

1. retryable resume 明确禁止 `deregister` 与 premature `clear pointer`，否则 retry path 会被自己打断  
   `src/bridge/bridgeMain.ts:2524-2534`
2. plugin `needsRefresh` 明确禁止 auto-refresh 与自动 reset `needsRefresh`，否则用户会被伪装成“已激活”  
   `src/hooks/useManagePlugins.ts:287-303`
3. MCP stale cleanup 明确禁止让旧 timer、旧 onclose、旧 config 跟新连接竞争，并禁止对 never-connected servers 做无意义 cleanup  
   `src/services/mcp/useManageMCPConnections.ts:791-812`
4. bridge no OAuth 明确把 `/login` 放在 policy check 之前，以避免把 stale/wrong-org cache 误说成 policy error  
   `src/bridge/initReplBridge.ts:144-150`
5. CCR mode 下 transient auth error 明确要求提示 retry，而不是 `/login`，因为该模式 auth 由 infrastructure JWT 处理  
   `src/services/api/errors.ts:212-216`
6. channels 认证缺口不只要求 `/login`，还要求 `then restart`，说明“只登录不重启”也是一条近邻错路  
   `src/components/LogoV2/ChannelsNotice.tsx:73`; `src/services/mcp/channelNotification.ts:222-227`

所以这一章的最短结论是：

`Claude Code 的安全控制面不只是在给出正确路径，它还在主动禁止那些会把系统重新带回错误状态解释的近邻动作。`

再压成一句：

`真正成熟的修复设计，不是给路，而是把错路一并封死。`

## 3. bridge 子系统已经明确：retryable resume 最危险的错路是“自己抹掉自己的重试资产”

## 3.1 transient reconnect failure 时，`deregister` 就是一条看似合理但本质错误的近邻动作

`src/bridge/bridgeMain.ts:2524-2526` 写得非常直白：

1. `Do NOT deregister on transient reconnect failure`
2. 因为当前 `environmentId` 仍是该 session 自己的 environment
3. `Deregistering would make retry impossible`

这说明在作者眼里，  
`deregister` 不是一个“保守清理动作”，  
而是一条明确的错路。

为什么它危险？

因为它会把：

`当前仍存在 retry value 的资格对象`

提前送入：

`不可恢复`

所以这不是“清理太激进”，  
而是：

`把 retryable 误写成 terminal-invalid`

## 3.2 `clear pointer` 也只允许在 fatal 上发生，说明“清得干净”不是第一优先级

同段 `2528-2534` 又进一步说明：

1. `Clear pointer only on fatal reconnect failure`
2. transient failures 应 keep the pointer
3. 因为 next launch re-prompts，that IS the retry mechanism

这说明 Claude Code 很清楚：

`pointer` 在 retryable 情况下不是脏状态，  
而是恢复资产的一部分。

所以“顺手清掉旧东西”这条常见近邻动作，  
在这里恰恰是错路。

从第一性原理看：

`凡是仍承载 retry value 的对象，都不应被 housekeeping 语义误杀。`

## 4. plugin 子系统已经明确：`needsRefresh` 最危险的错路是“半刷新假完成”

## 4.1 作者直接写了禁令：Do NOT auto-refresh. Do NOT reset needsRefresh

`src/hooks/useManagePlugins.ts:287-303` 是本章最直的证据之一。

这里不仅说了正确路径：

`Run /reload-plugins to activate`

还直接补上了错误路径禁令：

1. `Do NOT auto-refresh`
2. `Do NOT reset needsRefresh`

这说明在 plugin 域里，  
错误路径不是抽象风险，  
而是源码作者已经亲自踩过并显式封禁的实现路线。

## 4.2 为什么 auto-refresh 是错路，而不是更丝滑的体验

注释里已经说明原因：

1. 旧 auto-refresh 有 stale-cache bug
2. 只 cleared loadAllPlugins
3. downstream memoized loaders 仍返回 old data
4. 不完整，缺 MCP、agentDefinitions

这意味着 auto-refresh 这条近邻动作之所以危险，  
不是因为它会 crash，  
而是因为它更糟糕：

`它会看起来像成功，但实际只是半签发。`

所以在安全控制面里，  
最该禁止的往往不是粗暴失败，  
而是：

`成功表象过强、制度完成度却不足的近邻动作。`

## 5. MCP 子系统已经明确：stale cleanup 最危险的错路是让旧配置继续参加当前竞争

## 5.1 stale cleanup 前必须先拆掉旧 timer、旧 onclose 与旧 config closure

`src/services/mcp/useManageMCPConnections.ts:791-812` 的三条 hazard 注释非常有力量：

1. pending reconnect timer 会带着 OLD config 重新触发
2. onclose 会用 OLD config 开始 reconnectWithBackoff
3. never-connected servers 的 clearServerCache 会平白触发真实 connect attempt

这说明 MCP 子系统真正要禁止的，不只是“旧对象还在”，  
而是：

`旧对象沿着旧闭包、旧定时器、旧回连逻辑继续偷偷参与 current 竞争。`

这是一种更高级的错误路径禁令：

`禁止旧路径以异步残留形式重新入场。`

## 5.2 “对所有 stale server 都统一 cleanup” 也是错路

同一段里还有一个很关键的限制：

`Only connected servers need cleanup.`

这说明看似更整齐的动作：

`把所有 stale server 统统清一遍`

其实也是错路。

因为对 `disabled/pending/failed` 而言，  
那样做会：

1. 触发空 cache 下的真实 connect attempt
2. spawn/OAuth
3. 立即 kill

也就是制造一次无意义又带副作用的伪修复。

所以 MCP 域已经证明：

`错路不只是“去错地方”，也包括“用错力度”。`

## 6. bridge auth / channels / CCR 进一步说明：即便动作表面相同，领域错配也是错路

## 6.1 bridge no OAuth 时先给 policy error，就是一种有害错路

`src/bridge/initReplBridge.ts:144-150` 的注释非常重要：

1. no OAuth check runs before policy check
2. 这样 console-auth users 才会得到 actionable `/login` hint
3. 否则会拿到 misleading policy error from stale/wrong-org cache

这说明“先做 policy check”在这里不是中性的顺序问题，  
而是一条错误路径。

因为它把：

`auth prerequisite 缺失`

错翻译成

`policy denied`

本质上是把用户送进了错误的解释域。

## 6.2 CCR mode transient auth error 时让用户去 `/login`，也是领域错配的错路

`src/services/api/errors.ts:212-216` 明确写出：

1. in CCR mode, auth is handled via infrastructure JWTs
2. transient auth errors should suggest retrying, not logging in

这说明即使 `/login` 在很多地方是正确动作，  
它在某些域里也会变成错路。

所以安全系统不能只记住：

`这个动作通常有用`

而必须记住：

`这个动作在当前资格制度里是否归属正确。`

## 6.3 channels auth 缺口若只说 `/login` 而不说 restart，也是不完整的近邻错路

`src/components/LogoV2/ChannelsNotice.tsx:73` 直接给出：

`run /login, then restart`

`src/services/mcp/channelNotification.ts:222-227` 也把 channels 缺口原因写成：

`channels requires claude.ai authentication (run /login)`

两者合起来说明：

1. `/login` 是必要路径
2. 但它并不自动等于完成
3. restart 也是协议的一部分

这说明“只做前半步”也是一种近邻错路。  
安全控制面如果不把后半步说清，  
就会制造半修复错觉。

## 7. 这套设计背后的哲学本质

这一章最重要的哲学结论是：

`正确路径的价值，取决于系统是否敢于同时否定看起来差不多的错路径。`

也就是说，  
安全控制面真正成熟的地方不在于：

`我知道怎么修`

而在于：

`我也知道哪些修法虽然看起来相近，却会把资格真相破坏掉。`

所以安全设计在更深一层上，是：

`对近邻错误动作的主动排斥。`

这比一般的“给用户建议”更强。  
因为它已经从 recommendation 进入：

`normative governance`

## 8. 第一性原理：为什么没有错路禁令的系统仍然不安全

如果从第一性原理问：

`为什么给出 dominant route 之后，系统仍然必须再写错路禁令？`

答案是：

`因为用户和实现者都天然会偏向选择那些更省事、更自动、更像成功的近邻动作。`

例如：

1. 自动 refresh 一下看起来更顺
2. 顺手 clear 一下看起来更干净
3. 统一 retry 一下看起来更省脑
4. 统一让用户 /login 看起来更通用

但这些看似省事的动作，  
恰恰最容易破坏资格制度的真实性。

所以安全系统必须把以下两件事同时做完：

1. 指出正路
2. 禁止错路

只做前者，  
系统仍会被“近似正确”的动作慢慢掏空。

## 9. 苏格拉底式自我反思

可以继续问六个问题：

1. 如果 plugin 域已经明确禁 auto-refresh，是否还有别的子系统仍在偷偷接受“半签发假完成”？
2. 如果 retryable bridge resume 明确禁 `deregister`，是否还有其他恢复资产也应拥有类似的“不得清理”保护？
3. 如果 stale MCP cleanup 已显式拆 timer / onclose / config closure，是否还有别的异步残留路径未被治理？
4. 如果 `/login` 在 CCR transient auth error 中是错路，当前所有 auth 相关 surface 都能区分这种例外吗？
5. 如果 channels 要求 `/login` then restart，哪些提示还只说了前半步？
6. 如果错误路径禁令这么重要，下一代安全控制台是否应把 forbidden adjacent path 做成显式字段，而不是只埋在注释与零散文案里？

这些追问说明：

Claude Code 已经在多个局部实现了错误路径禁令，  
但还可以进一步把它上升成统一协议。

## 10. 对目录结构与后续研究的启发

写到这里，这条资格安全链已经推进成六层：

1. 资格对象
2. 重签发路径
3. 中间态语法
4. 承诺上限
5. 默认动作路由
6. 错误路径禁令

所以下一步最自然的深化方向会是：

`安全资格动作完成权`

也就是继续回答：

`即便用户已经走上正确路径，谁有资格宣布这条路径已经完成，而不是仍停在进行中。`

## 11. 给系统设计者的技术启示

最后把这一章压成六条工程启示：

1. 给出 dominant route 后，仍要把近邻错路显式封死
2. 最危险的错路通常不是离谱动作，而是“看起来差不多”的省事动作
3. 不要把 housekeeping 语义误用到仍有资格价值的对象上
4. 不要把通用修复动作滥用到领域不匹配的状态上
5. 半修复、半刷新、半签发通常比显式失败更危险
6. 若不治理 forbidden adjacent path，默认路由最终会被近邻错路稀释掉

这一章最终要守住的一条硬结论是：

`对 Claude Code 这类安全控制面来说，真正成熟的不是总能告诉用户正确路径，而是能在正确路径旁边立起足够清楚的“此路不通”。`
