# 安全载体家族强请求清理修复治理与强请求清理迁移治理分层：为什么artifact-family cleanup stronger-request cleanup-repair-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-migration-governor signer

## 1. 为什么在 `227` 之后还必须继续写 `228`

`227-安全载体家族强请求清理反漂移验证治理与强请求清理修复治理分层` 已经回答了：

`stronger-request cleanup 线即便未来长出 repair governor，也还需要独立主权去决定 drift 被发现后是修 scheduler、修 executor、补 receipt、加强 explainability，还是缩 wording / promise strength。`

但如果继续往下追问，
还会碰到下一层更容易被误写成“修完就结束”的错觉：

`只要 repair governor 已经决定怎么修，它就自动拥有把旧 stronger-request 世界迁到新世界的主权。`

Claude Code 当前源码仍然不能支持这种更强说法。

因为继续看 repo 自己最成熟的 migration 正对照，
会发现它其实已经把：

`fix direction`

和

`old world exit strategy`

拆成了不同层。

最硬的证据有五组：

1. `src/main.tsx:323-345`
   直接维护启动迁移链与 `CURRENT_MIGRATION_VERSION`，
   说明 repo 不把“新规则更正确”偷写成“旧世界自然退场”
2. `src/migrations/migrateLegacyOpusToCurrent.ts:13-57`
   明确只改 `userSettings`、保留 project/local/policy source、继续 runtime remap，并保存 timestamp 供 one-time notification
3. `src/migrations/migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-21`
   又说明旧 key 到新 key 的交接不是简单 rename，
   而要治理何时复制、何时删除旧 key、何时保持幂等
4. `src/utils/plugins/cacheUtils.ts:23-24,52-74,89-105`
   的 `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days`
   更进一步说明旧插件世界不会“新版本一来就立刻消失”；
   迁移治理要决定 grace window 与 staged retirement
5. `src/utils/plans.ts:153-255`
   的 `copyPlanForResume()` / `copyPlanForFork()`
   又把 continuity strategy 明确摆上台面：
   旧 plan artifact 如何被 reuse、recover、copy，而不是被粗暴抹平

与这些正对照相比，
stronger-request cleanup 线当前虽然已开始显露 repair-governance，
但还没有谁正式决定：

1. 旧 startup-delete wording 何时退场
2. 旧 home-root `plans` cleanup law 如何 phase out
3. 旧 receipt / local count truth 是否需要 dual-read、grace window 或 one-time notice
4. 旧 stronger-request promise 世界是立即 cutover，还是 staged migration

这说明：

`artifact-family cleanup stronger-request cleanup-repair-governor signer`

和

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

仍然不是一回事。

前者最多能说：

`我们已经决定应该修哪一层。`

后者才配说：

`旧 law、旧 path、旧 receipt、旧 wording 应怎样被带过渡、作废、保留兼容期，或者被重签发。`

所以 `227` 之后必须继续补的一层就是：

`安全载体家族强请求清理修复治理与强请求清理迁移治理分层`

也就是：

`stronger-request cleanup-repair governor 只能决定修什么；stronger-request cleanup-migration governor 才能决定旧世界怎样在不制造新谎言的前提下被平稳带离现场。`

## 2. 先做五条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-migration-governor signer` 仍是研究命名。
它不是在声称 cleanup 线已经有一个 migration policy object 只差导出，
而是在说：

1. repo 在别的子系统里已明确展示出 migration governance 的存在
2. migration governance 回答的是 rewrite、remap、grace、notification 与 continuity
3. stronger-request cleanup 线今天若要真正成熟，也迟早要回答这些问题

第二条：

这里说的 `migration governance`
不是在声称：

`修复之后一律都要做批量重写。`

真正要说的是：

`migration` 可能意味着 source rewrite、runtime remap、completion marker、one-time notification、orphan grace window 或 continuity-preserving copy/reuse/recovery；关键不在迁不迁，而在谁来决定旧世界怎样退场。`

第三条：

这里也不是把：

`plansDirectory`

单独当成全部迁移问题。

真正关键的是：

1. temporal drift 修完后，旧 wording 何时退场
2. propagation drift 修完后，旧 home-root law 是否还保留兼容窗口
3. receipt drift 修完后，旧 local truth 是否需要重签、dual-read 或 notice
4. stronger-request cleanup 线当前没有一处可见代码显式持有这些过渡决定

第四条：

这里尤其不能过度声称：

`migration governance` 一定意味着“旧世界马上消失”。

当前更稳的说法只能是：

`plugin orphan cleanup、model remap 与 plan continuity 已经说明：成熟系统更常见的做法恰恰是让旧世界按受治理的节奏退出，而不是被一次性粗暴抹平。`

第五条：

这里也不能过度声称：

`runtime remap` 就已经等于 migration governance 的全部。

当前更稳的说法只能是：

`runtime remap` 只是迁移语法之一；它仍需要与 source rewrite、grace window、notification 与 continuity policy 一起构成完整 migration grammar。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-repair-governor signer 仍不等于 stronger-request cleanup-migration-governor signer`

证据：

1. `main.tsx:323-345`
   直接维护启动迁移链与 migration version gate
2. `migrateLegacyOpusToCurrent.ts:13-57`
   说明 model 修复之后还要继续治理哪些 source 被改、哪些 source 只 remap、是否通知用户
3. `migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-21`
   说明旧 key 到新 key 的过渡还要治理复制、删除时机与幂等
4. `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days`
   说明旧 plugin world 不是“新版本一来就立即消失”
   `cacheUtils.ts:23-24,52-74,89-105`
5. `copyPlanForResume()` / `copyPlanForFork()`
   说明 plan artifact 过渡不是单纯 repair，而要继续回答 resume reuse、missing-file recovery 与 fork continuity
   `plans.ts:153-255`
6. stronger-request cleanup 当前虽已开始显露 repair-governance，但还没有谁正式决定旧 law、旧 path、旧 receipt 与旧 wording 应怎样 phased exit
7. 因而 stronger-request cleanup 今天真正缺的不是又一个 fix，而是 repair 之后的 transition grammar

因此这一章的最短结论是：

`stronger-request cleanup-repair-governor signer 最多能说“该修哪里”；stronger-request cleanup-migration-governor signer 才能说“旧世界怎样有次序地退出，而不把兼容世界和新世界一起搞乱”。`

再压成一句：

`会修，不等于会迁。`

## 4. 第一性原理：repair governor 回答“修哪层”，migration governor 回答“旧世界如何不被粗暴抹平”

从第一性原理看，
这两层治理处理的是两个不同时间面的主权问题。

`repair governor`

回答的是：

1. 应修 metadata、runtime、scheduler、executor 还是 promise
2. 哪条路径现在不可信
3. 哪类 drift 必须立即止血
4. 哪种 stronger claim 应先撤回
5. 哪一层对当前 bug 负主责

`migration governor`

回答的则是：

1. 旧值是立刻重写，还是先 runtime remap
2. 哪些 source 允许被改写，哪些必须保持原样
3. 兼容窗口多长
4. 旧缓存、旧目录、旧 wording 何时清理
5. 是否需要 one-time notification、completion flag、dual-read 或 continuity copy

所以这两层的哲学本质不同：

`repair`

属于纠偏主权：

`它回答“现在应把哪里拉回正确状态”。`

`migration`

则属于秩序交接主权：

`它回答“旧状态怎样体面让位，而不在交接中制造新的谎言”。`

Claude Code 在这方面真正先进的地方，
不是它“会在修完后顺手删掉旧东西”，
而是它已经在多个 subsystem 里明确展示：

`旧世界的退出本身也是正式治理对象。`

这是一种很成熟的安全哲学：

`更正当前真相，不等于有权立即抹掉旧真相。`

因为一旦把 repair 与 migration 压成一层，
系统就容易出现三种危险：

1. 旧 source 被过早重写，用户意图静默蒸发
2. 新旧规则交接没有兼容窗口，运行时开始说两套互相冲突的话
3. continuity artifact 被粗暴清空，恢复链、回放链与审计链一起断掉

## 5. 从技术角度看，Claude Code 的先进性在于“迁移被做成多种 transition grammar，而不是单一 rewrite”

很多系统一谈迁移，
就只剩下一种动作：

`改旧值。`

Claude Code 更成熟的地方在于，
它会根据旧世界的类型选择不同的 transition grammar。

### 5.1 `runMigrations()`：迁移本身先被提升成 startup governance

`main.tsx:323-345`
最值钱的地方，
不是它调用了几段脚本，
而是它把迁移正式纳入：

1. startup path
2. version gate
3. ordered execution

这说明 repo 已明确承认：

`old world exit`

不是临时修补，
而是正式控制面的一部分。

### 5.2 `migrateLegacyOpusToCurrent()`：source-selective rewrite + runtime remap + timestamp notice

`migrateLegacyOpusToCurrent.ts:13-57`
展示的不是简单“把旧模型改成新模型”，
而是：

1. 只重写 `userSettings`
2. 保留 project/local/policy source 不动
3. runtime 继续 remap
4. 保存 timestamp 供 one-time notification

这说明成熟 migration governance 不只问：

`是不是应该换值`

还要问：

`哪类 source 可改、哪类 source 只能兼容、用户要不要被显式告知。`

### 5.3 `migrateReplBridgeEnabledToRemoteControlAtStartup()`：idempotent key handoff

`migrateReplBridgeEnabledToRemoteControlAtStartup.ts:3-21`
展示的则是另一种 transition grammar：

1. old key 存在且 new key 未设置时才复制
2. 复制后删除 old key
3. 整个过程保持 idempotent

这说明 config-key migration 的技术本质不是 rename，
而是：

`handoff without double-write and without silent loss`

### 5.4 `.orphaned_at` + 7-day grace：staged retirement

`cacheUtils.ts:23-24,52-74,89-105`
展示的则是：

1. 先标记 orphan
2. 再进入 7 天 grace window
3. 之后才真正 cleanup

这说明成熟系统不会把：

`new world is preferred`

直接压成：

`old world must disappear now`

### 5.5 `copyPlanForResume()` / `copyPlanForFork()`：continuity-preserving migration

`plans.ts:153-255`
更进一步说明：

有些迁移甚至不是 rewrite，
而是：

1. resume reuse 原 slug
2. 文件丢失时 recovery
3. fork 时 copy content 但 new slug，避免 clobber

这是一种 continuity grammar，
说明迁移治理有时真正要守住的，
不是“删旧换新”，
而是：

`在 lineage 变化时保住连续性。`

## 6. stronger-request cleanup 当前真正缺的，是 old world exit strategy

当前 stronger-request cleanup 线已经开始显露 repair choice：

1. temporal drift 可能要求修 scheduler 或降级 wording
2. propagation drift 可能要求让 executor 跟随 `plansDirectory` 或限缩 override semantics
3. receipt drift 可能要求把 `CleanupResult` 提升成 surfaced receipt
4. abstention drift 可能要求增强 explainability 而不是扩大 destructive scope

但即便这些 repair direction 都被决定了，
仍然会立刻碰到 migration 问题：

### 6.1 旧 wording 世界怎样退场

如果 startup-delete wording 被修弱，
旧 stronger wording 是立即消失，
还是需要一个过渡期来解释“deferred cleanup still pending”。

### 6.2 旧 path world 怎样退场

如果 `plansDirectory` 最终让 cleanup executor 跟随 override，
旧 home-root `plans` cleanup law 是立即作废，
还是在一段兼容窗口里继续 dual-read / dual-clean。

### 6.3 旧 receipt world 怎样退场

如果 `CleanupResult` 被提升成 family receipt，
旧 local-count truth 是立即不再被信任，
还是先进入 “local only / not surfaced” 的过渡说明。

### 6.4 旧 promise world 怎样退场

如果 validation skip 与 delayed housekeeping 被正式提升成 stronger explainability surface，
旧 “deleted at startup” 的绝对 promise
是否需要 one-time notice 或 migration marker。

这些都不是 repair 自己能回答的。

它们已经进入：

`migration-governance`

层。

## 7. 用苏格拉底式反问再压一层

1. 如果 stronger-request cleanup repair governance 已经等于 migration governance，为什么 `migrateLegacyOpusToCurrent()` 还要只改 `userSettings`、保留其他 source，并额外写 timestamp notification？
   因为“知道该修什么”与“决定旧世界怎样退出”本来就不是同一主权。
2. 如果修好 `plansDirectory` propagation gap 就已经足够，为什么 `copyPlanForResume()` / `copyPlanForFork()` 还要继续回答 reuse、recover、copy 与 new-slug policy？
   因为 continuity 本身就是 transition grammar，而不是 fix 的副产品。
3. 如果 migration 只是 rewrite，为什么 plugin 线还要先 `.orphaned_at` 再等 7 天 cleanup？
   因为旧世界体面退出，本来就需要 grace window 与 staged retirement。
4. 如果旧 wording 只是“说得不够准”，为什么不能修完后立即把它抹掉？
   因为当用户已经根据旧 promise 理解系统时，旧 promise 的退场本身也需要被治理。
5. 如果 receipt surface 被补齐就算完成迁移，为什么 source-selective rewrite、runtime remap 与 continuity-preserving copy 还同时存在？
   因为不同旧世界需要不同过渡语法，没有一种单一 cutover 能安全覆盖全部场景。

## 8. 硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 repair governor，就已经足够成熟。`

而是：

`Claude Code 在模型、配置键、plugin orphan 宽限期与 plans continuity 上已经明确展示了 migration governance 的存在；因此 artifact-family cleanup stronger-request cleanup-repair-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-migration-governor signer。`

换句话说：

`cleanup 线真正缺的不是“修完”，而是“修完后旧 stronger-request 世界怎样被有秩序地带走”的最后一道治理设计。`
