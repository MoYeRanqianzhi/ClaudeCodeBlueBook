# 安全载体家族强请求清理墓碑治理与强请求清理复活治理分层：为什么artifact-family cleanup stronger-request cleanup-tombstone-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-resurrection-governor signer

## 1. 为什么在 `357` 之后还必须继续写 `358`

`357-安全载体家族强请求清理退役治理与强请求清理墓碑治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界何时正式结束，也仍需要 tombstone governor 去决定结束后留下什么最小残留事实。`

但如果继续往下追问，
还会撞上一层更容易被偷写成
“只要墓碑已经留下或清掉，后面怎样回来也就自动清楚了”
的错觉：

`只要 stronger-request cleanup tombstone governor 已经决定旧对象退役后留下什么 marker，它就自动拥有了决定哪些旧对象可以回来、何时回来、回来时算谁、以及回来后进入哪一层 current world 的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 在 plugin 与 plan 两条线上已经成熟写出的 resurrection 正对照，
会发现它其实明确拆开了四件事：

1. `authoritative clearing`
2. `re-entry gate`
3. `identity policy`
4. `active-layer admission`

最硬的证据至少有七组：

1. `cacheUtils.ts` 证明 `.orphaned_at` 的清除由 authoritative installed set 决定，而不是 marker 自己决定
   `src/utils/plugins/cacheUtils.ts:67-72,82-92,122-130`
2. `refresh.ts` 证明 active plugin return 是专门的 Layer-3 refresh primitive，不是 marker 清掉后的自动副作用
   `src/utils/plugins/refresh.ts:1-18,72-80,123-137`
3. `/reload-plugins` 证明 re-entry 需要显式 signal，系统不会把“磁盘世界变了”偷写成“active world 已接纳”
   `src/commands/reload-plugins/reload-plugins.ts:37-56`
4. `PluginInstallationManager.ts` 与 `useManagePlugins.ts` 证明 repo 还会继续区分：哪些 return path 可以 auto-refresh，哪些 return path 必须停在 `needsRefresh`
   `src/services/plugins/PluginInstallationManager.ts:56-59,135-179`
   `src/hooks/useManagePlugins.ts:28-34,287-303`
5. `plans.ts` 证明 plan resurrection 不只是 undelete，还要治理 lineage、evidence 与 identity policy
   `src/utils/plans.ts:164-231,233-257`
6. `conversationRecovery.ts` 证明 resurrection grammar 被接进正式恢复主线，而不是局部 helper 的偶发行为
   `src/utils/conversationRecovery.ts:539-547`
7. `sessionStorage.ts` 的 tombstone path 只回答怎样删旧对象，不回答它何时、以什么身份重新回来
   `src/utils/sessionStorage.ts:925-945,1468-1474`

这说明：

`artifact-family cleanup stronger-request cleanup-tombstone-governor signer`

和

`artifact-family cleanup stronger-request cleanup-resurrection-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界结束后给它留一个 marker。`

后者才配说：

`在什么证据、什么 lineage、什么 identity policy 与什么 active-layer admission 下，这个旧对象还能重新回到 current world。`

所以 `357` 之后必须继续补的一层就是：

`安全载体家族强请求清理墓碑治理与强请求清理复活治理分层`

也就是：

`stronger-request cleanup tombstone governor 负责留下退役痕迹；stronger-request cleanup resurrection governor 才负责治理旧对象怎样从历史世界重新回到当前世界。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-resurrection-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 resurrection manager，
而是在说：

1. repo 在 plugin 与 plan 线上已经展示出真实的 resurrection grammar
2. 这些 grammar 回答的是“怎样让历史对象重新回到 current world”
3. stronger-request cleanup 线如果真要把 tombstone world 做完整，也迟早要回答这些“谁配让它回来”的问题

第二条：

这里说的 `resurrection governance`

不是在声称：

`只要对象回来，就一定沿原 identity 原样复活。`

当前更稳的说法只能是：

`resurrection governance 还要回答回来时是 old identity、new identity，还是只恢复内容不恢复身份。`

第三条：

这里也不是把：

`marker clearing`

直接等同于：

`current world 已重新接纳该对象。`

repo 在 plugin 线上明确拆开了：

1. marker clear
2. disk / materialized return
3. active components return

第四条：

这里尤其不能过度声称：

`plan recovery`

就等于 resurrection 的全部语义。

当前更稳的说法只能是：

`plan line 给出了 lineage / evidence / identity policy 的强正例，但 stronger-request cleanup 线未来的复活 grammar 仍不能提前写死。`

第五条：

这里也不能过度声称：

`有 resurrection grammar 就意味着任何旧对象都可以回来。`

当前更稳的说法只能是：

`成熟的 resurrection governance 恰恰首先回答“哪些对象不配回来”，以及“谁配决定它不能回来”。`

第六条：

这里还不能过度声称：

`Claude Code 的 resurrection 只是恢复体验优化。`

更稳的说法只能是：

`它本质上是在治理 current world 的重新接纳权；体验只是外观，资格与接纳才是本体。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-tombstone-governor signer 仍不等于 stronger-request cleanup-resurrection-governor signer`

证据：

1. `.orphaned_at` 只有在 authoritative installed set 重新声明该版本时才会被清除
   `cacheUtils.ts:67-72,82-92,122-130`
2. `refreshActivePlugins()` 与 `/reload-plugins` 证明 marker clearing / disk presence 不等于 active components 已重新进入当前 session
   `refresh.ts:1-18,72-80,123-137`; `reload-plugins.ts:37-56`
3. repo 明确要求某些 update path 停在 `needsRefresh`，而不是默认自动接纳
   `PluginInstallationManager.ts:56-59,135-179`; `useManagePlugins.ts:28-34,287-303`
4. `copyPlanForResume()` 证明 plan resurrection 需要 lineage 与 evidence
   `plans.ts:164-231`
5. `copyPlanForFork()` 证明“回来”不等于“以原身份回来”
   `plans.ts:233-257`
6. `conversationRecovery.ts` 把 resurrection 接进恢复主线，说明它是正式 recovery grammar
   `conversationRecovery.ts:539-547`
7. `sessionStorage.ts` 的 tombstone path 只回答怎样删除旧对象，不回答它何时、以什么身份重新回来
   `sessionStorage.ts:925-945,1468-1474`

因此这一章的最短结论是：

`stronger-request cleanup tombstone governor 最多能说旧世界结束后留下什么；stronger-request cleanup resurrection governor 才能说哪些退役对象在何种证据、何种身份政策与何种激活层面下可以重新回来。`

再压成一句：

`会留墓碑，不等于会让它回来。`

再压成更哲学一点的话：

`制度知道怎样留下历史痕迹，不等于制度已经知道谁配重新被当前世界承认。`

## 4. 第一性原理：tombstone governance 回答“留什么最小残留事实”，resurrection governance 回答“凭什么重新接纳它”

从第一性原理看，
强请求清理墓碑治理与强请求清理复活治理处理的是两个不同方向的主权问题。

`stronger-request cleanup tombstone governor`
回答的是：

1. 对象退役后是否留 marker
2. marker 是 tombstone、timestamp、warning 还是 dotfile
3. 哪些读取面要继续消费这个 marker
4. marker 何时再被进一步清除
5. 它怎样避免未来误读旧世界

`stronger-request cleanup resurrection governor`
回答的则是：

1. 哪些历史对象允许重新回到 current world
2. 需要什么 evidence 才能回来
3. 回来的是同一 identity，还是 new identity
4. 回来的是 disk / materialized world，还是 active / runtime world
5. 谁来决定 re-entry 后的可见性、冲突边界与 lineage

如果把这两层压成一句：

`反正 marker 还在或 marker 已清掉`

系统就会制造五类危险幻觉：

1. `marker-clear-means-resurrection`
2. `disk-return-means-active-return`
3. `content-recovered-means-identity-restored`
4. `tombstone-is-bidirectional`
5. `any-old-object-can-come-back`

所以这两层的哲学本质也不同：

`tombstone`

治理的是：

`一个对象退出当前真相之后，世界还欠后来者什么最小解释义务。`

`resurrection`

治理的是：

`一个历史对象若想重新进入当前真相，谁有权为它重新签发在场资格。`

Claude Code 在这里真正先进的地方，
不是它把 every return path 都自动做通，
而是它已经默认承认：

`重新出现，不等于重新被承认。`

这恰恰是一种很成熟的安全哲学：

`对象可以重新物化，但 current world 的接纳权不能因此被偷走。`

## 5. 从技术角度看，Claude Code 的先进性在于把 resurrection 拆成 authoritative clearing、re-entry gate、identity policy 与 active-layer admission，而不是把“回来”压成一个动作

### 5.1 authoritative marker clearing 不是 resurrection

`cacheUtils.ts:67-72,82-92,122-130`
先把 orphan cleanup 写成两段：

1. Pass 1 清已安装版本的 `.orphaned_at`
2. Pass 2 才处理真正 orphaned 的版本

而且它不是任意清，
而是：

1. 先读取 `installed_plugins.json`
2. 形成 authoritative `installedVersions`
3. 只对这个集合里的 version path 执行 `removeOrphanedAtMarker()`

这说明：

`谁配撤销“你已退役”这条 marker`

不是 marker 自己决定的，
而是由 authoritative installed state 决定。

但这还不是 resurrection。

因为 `removeOrphanedAtMarker()` 自己只删 marker，
不负责 active return。

### 5.2 Layer-3 refresh 才是 current-world admission，不是 disk 状态的自动尾气

`refresh.ts:1-18`
直接把 active plugin return 定义成：

`Layer-3 refresh primitive: swap active plugin components in the running session`

并显式区分：

1. Layer 1: intent
2. Layer 2: materialization
3. Layer 3: active components

`refresh.ts:72-80,123-137`
继续把 `refreshActivePlugins()` 写成当前会话里的真正激活入口：

1. clear all caches
2. clear orphan exclusions
3. 重建 enabled / disabled / commands / agents / hooks / MCP / LSP
4. 把 `needsRefresh` 消费为 `false`
5. bump `pluginReconnectKey`

而
`reload-plugins.ts:37-56`
则把这件事公开成一个显式命令事实：

`Reloaded: ...`

这说明 repo 明确拒绝把：

`marker clearing`

偷写成：

`current session 已接纳该对象`

### 5.3 `needsRefresh` 证明 comeback path 还要再分 auto-admit、explicit-admit 与 pending worlds

`PluginInstallationManager.ts:56-59`
先给出高层策略：

1. new installs -> auto-refresh
2. updates only -> set `needsRefresh`

继续看
`PluginInstallationManager.ts:135-179`：

1. 新 marketplace 安装完成时，repo 会尝试 `refreshActivePlugins()`
2. auto-refresh 失败时，会回退为 `needsRefresh`
3. 如果只是 updates only，系统直接保持显式 pending 状态

再看
`useManagePlugins.ts:28-34,287-303`：

1. 初始 Layer-3 load 之后，后续 refresh 一律走 `/reload-plugins`
2. `needsRefresh` 只会弹通知：
   `Plugins changed. Run /reload-plugins to activate.`
3. 注释明确写着：
   `Do NOT auto-refresh`

这说明 repo 实际上在治理三种不同的 return world：

1. auto-admit world
2. explicit-admit world
3. not-yet-admitted world

真正成熟的 resurrection governance，
不是一句“回来了”，
而是继续回答：

`哪种 return path 可以自动被 current world 接纳，哪种必须保持显式。`

### 5.4 plan 线给出第二个强正例：对象复活时，系统还要治理 lineage、evidence 与 identity policy

`plans.ts:164-231`
的 `copyPlanForResume()` 很值钱。

它不是看到 plan file 缺失就粗暴恢复。
它要求：

1. log 里确实有 slug
2. 把 slug 绑定回 original session ID
3. 先尝试直接读取 plan file
4. 只有在 `ENOENT` 且 remote session 才尝试 recovery
5. 优先用 file snapshot，失败后再回到 message history

这说明 plan resurrection 的本体不是：

`内容还在，就给你拉回来`

而是：

`只有在 lineage 和 evidence 都成立时，才允许对象重新进入 current session world`

更关键的是
`plans.ts:233-257`
里的 `copyPlanForFork()`：

1. 不复用 original slug
2. 生成 new slug
3. 把原计划内容写进新文件

这直接证明：

`内容回来`

不等于

`原身份回来`

也就是说 resurrection governance 还必须回答：

`re-entry 后对象到底算 old identity 还是 new identity。`

`conversationRecovery.ts:539-547`
又把这件事接进正式恢复主线，
说明 resurrection 不是局部 helper，
而是正式 recovery grammar 的一部分。

## 6. 对 stronger-request cleanup 设计的四条直接启示

1. 不能把 marker clearing 写成 comeback 完成；`撤碑` 与 `接纳` 必须拆层。
2. 不能把 disk/materialized return 写成 active/runtime return；re-entry gate 必须显式化。
3. 不能把内容 continuity 偷写成身份 continuity；identity policy 需要被正式签字。
4. 不能把 resurrection 提前偷写进 tombstone；`如何留下痕迹` 与 `谁配回来` 必须继续拆层。

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界结束后留下了 marker”误认成了“系统已经知道谁配回来”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 tombstone grammar 已经足够强，为什么还要再拆 resurrection？
   因为会留下标记，不等于知道谁配重新进入 current world。
2. 如果 `.orphaned_at` 会被清掉，是不是就说明对象已经完全回来了？
   不是。marker clearing 只是撤销退役标记，不等于 active world 已接纳。
3. 如果磁盘上对象已经恢复，是不是就说明当前 session 可以继续用？
   也不对。disk/materialized return 和 active/runtime return 被明确拆层。
4. 如果 plan 内容能恢复，是不是就说明原身份也恢复了？
   不是。`copyPlanForFork()` 明确证明 content return 与 identity return 可以分开。
5. 如果 resurrection grammar 已存在，是不是就意味着任何旧对象都该能回来？
   恰恰相反。成熟 resurrection 首先治理“谁不配回来”。
6. 如果 cleanup 线还没有显式 resurrection 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确 grammar，越容易把“还能回来吗”偷写成“反正 marker 清掉就算回来”。`

这一串反问最终逼出一句更稳的判断：

`resurrection 的关键，不在系统会不会让某些旧对象重新出现，而在系统能不能正式决定哪些历史对象配重新被当前世界承认。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 未来只要补出 tombstone governance，就已经足够成熟。`

而是：

`Claude Code 在 authoritative marker clearing、Layer-3 refresh、mode-sensitive admission、lineage-based plan recovery 与 identity-splitting fork policy 上已经明确展示了 resurrection governance 的存在；因此 artifact-family cleanup stronger-request cleanup-tombstone-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-resurrection-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“谁来留下历史痕迹”，而是“谁来决定哪些带碑对象可以重新回到当前世界”。`
