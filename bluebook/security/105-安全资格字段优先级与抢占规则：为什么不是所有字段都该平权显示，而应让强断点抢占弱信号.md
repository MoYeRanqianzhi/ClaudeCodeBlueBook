# 安全资格字段优先级与抢占规则：为什么不是所有字段都该平权显示，而应让强断点抢占弱信号

## 1. 为什么在 `104` 之后还必须继续写“字段优先级与抢占规则”

`104-安全资格投影字段落位` 已经回答了：

`不同字段应按 surface 强弱分层落位，而不是平均撒开。`

但如果继续往下追问，  
还会碰到一个更细的问题：

`即便字段已经落在了正确的 surface 上，它们彼此之间要不要再分优先级？`

答案是必须分。

因为在真实运行时，  
同一个 surface 里经常会同时出现多个竞争性的信号：

1. failed
2. needs-auth
3. reconnecting
4. connected
5. plugin updated

如果这些字段被平权显示，  
结果往往不是更完整，  
而是更混乱：

1. 强断点被弱信号稀释
2. 用户看见“成功提示”和“失败提示”并存
3. 修复优先级被视觉平均化

所以 `104` 之后必须继续补出的下一层就是：

`安全资格字段优先级与抢占规则`

也就是：

`不是所有字段都该平权显示，系统必须让强断点抢占弱信号，让高风险 truth 覆盖低风险投影。`

## 2. 最短结论

从源码看，Claude Code 已经在多个地方显式实现了优先级与抢占：

1. notification 队列定义了 `immediate > high > medium > low` 的严格优先级  
   `src/context/notifications.tsx:230-239`
2. `immediate` notification 会直接抢占当前 notification，并可通过 `invalidates` 驱逐旧通知  
   `src/context/notifications.tsx:78-118`
3. bridge failure notification 使用 `priority: 'immediate'`，说明它应抢占普通状态提示  
   `src/hooks/useReplBridge.tsx:102-112`
4. MCP failed / needs-auth 只给 `medium`，说明它们重要但不应压过 bridge fatal / fuse 类断点  
   `src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63`
5. plugin updated 只给 `low`，说明“可稍后处理”的 activation 提示不应抢占更强安全断点  
   `src/hooks/notifs/usePluginAutoupdateNotification.tsx:60-64`
6. PromptInputFooter 在 implicit remote 下只允许 `Remote Control reconnecting` 存活，且 failed 被直接剥离到 notification surface，说明 footer 自身也有字段抢占纪律  
   `src/components/PromptInput/PromptInputFooter.tsx:173-185`; `src/hooks/useReplBridge.tsx:339-349`

所以这一章的最短结论是：

`Claude Code 不只在决定字段出现在哪里，也在决定字段出现时谁该压过谁。`

再压成一句：

`强断点应当先说话，弱信号必须让位。`

## 3. notification 子系统已经把抢占规则写成了显式协议

## 3.1 `immediate/high/medium/low` 不是颜色装饰，而是控制面优先级

`src/context/notifications.tsx:230-239` 非常直接：

1. `immediate: 0`
2. `high: 1`
3. `medium: 2`
4. `low: 3`

`getNext(queue)` 总是选择数值更低者。

这说明 notification priority 在这里不是视觉标签，  
而是：

`谁有资格先占据当前通知位`

也就是说，  
字段优先级在 Claude Code 里已经是协议级概念，  
不是 UI cosmetics。

## 3.2 `immediate` 的语义是“直接抢占当前解释通道”

`src/context/notifications.tsx:78-118` 更进一步：

1. immediate 到来时会清掉当前 timeout
2. 直接设置自己为 `current`
3. 可通过 `invalidates` 移除队列中被它淘汰的通知

这说明 immediate 通知的本质不是：

`更红一点`

而是：

`拥有解释通道抢占权`

从控制面哲学看，  
这非常重要。  
因为它明确承认：

`不同 truth 的解释权不是平权的。`

## 3.3 `invalidates` 说明抢占还不够，旧叙事还必须被显式驱逐

`src/hooks/notifs/useFastModeNotification.tsx:106-121` 给了一个很干净的例子：

1. cooldown started invalidates expired
2. cooldown expired invalidates started

这说明优先级系统不只决定谁先显示，  
还决定：

`哪些旧叙事已经不配继续存在。`

从安全语义看，  
这与我们前面讨论的：

1. 状态替换
2. 词法失效
3. 完成投影

其实是一条线：

`更强新信号到来后，旧弱信号必须退场，而不是并排共存。`

## 4. 源码已经自然形成了一个“安全字段优先级金字塔”

## 4.1 第一层：立即性失败断点

`src/hooks/useReplBridge.tsx:102-112` 里，bridge failure notification 直接是：

`priority: 'immediate'`

这说明 bridge failed 被视为：

`必须立即打断当前注意力的高优先级安全断点`

为什么合理？

因为 bridge failed 往往意味着：

1. 当前 remote path 已断
2. 原来的 active / reconnecting 叙事不再成立
3. 用户若继续按旧世界模型操作，会立刻误判

所以它理应位于优先级金字塔顶层。

## 4.2 第二层：仍需动作但不必立即打断全局上下文的断点

`src/hooks/notifs/useMcpConnectivityStatus.tsx:36-63` 中，  
MCP failed / needs-auth 都使用：

`priority: "medium"`

这说明作者有意把它们放在 bridge fatal 之下。  
原因也很清楚：

1. MCP failed 需要动作
2. 但通常不等于整个主控制面已经崩掉
3. 它更像局部 capability breakage

也就是说，  
MCP 问题很重要，  
但不应无条件抢占比它更强的全局断点。

## 4.3 第三层：信息性、延后可处理的提示

`src/hooks/notifs/usePluginAutoupdateNotification.tsx:60-64` 里，  
plugin updated 提示是：

`priority: "low"`

这说明：

1. 这是需要动作的提示
2. 但它不应与 failed / needs-auth / fatal breakage 平权竞争
3. 它属于“可稍后处理”的弱信号

这正是成熟控制面的优先级判断：

`所有需要动作的提示都不等强。`

## 4.4 由此可归纳出优先级梯度

Claude Code 当前隐含的梯度大致是：

1. `immediate`
   立即性失败断点 / 高风险控制面断裂
2. `high/medium`
   明确需要动作的局部安全问题
3. `low`
   可延后处理的信息性维护提示

这说明安全字段优先级，本质上是在编码：

`风险强度 + 行动紧迫度`

## 5. footer 域也在做自己的“局部抢占规则”

## 5.1 failed 被抢占性地移出 footer，说明 footer 并不是平权陈列面

`src/components/PromptInput/PromptInputFooter.tsx:173-185` 很关键：

1. failed state is surfaced via notification
2. footer 不显示 failed
3. implicit remote 下只显示 reconnecting

这说明 footer 也有自己的局部优先级：

1. failed 不应占 footer
2. reconnecting 可在 implicit remote 场景下成为唯一可见状态

也就是说，  
footer 不是一个把所有 bridge state 平铺开的面，  
而是一个经过抢占裁剪后的：

`最小操作投影面`

## 5.2 为什么这里只保留 reconnecting

因为在 implicit remote 场景下，  
如果仍平权展示 active / connected / failed / reconnecting，  
这个 pill 会立刻越界成半个控制台。

作者显然不想让它承担这种责任，  
所以只保留：

`最需要即时提醒用户的 narrow operational signal`

这也是一种优先级治理：

`在窄 surface 上，字段抢占比字段完整更重要。`

## 6. 第一性原理：为什么字段平权显示反而更容易失真

如果从第一性原理问：

`为什么不把所有字段都展示出来，让用户自己判断哪个更重要？`

答案是：

`因为用户界面不是数据库，平权并排显示会把风险层级压平。`

一旦压平：

1. 用户会把 low-signal 与 high-breakage 等权对待
2. 更强断点失去注意力优势
3. 旧叙事与新叙事容易并排共存
4. “应该立刻修”的问题与“可以稍后处理”的问题混在一起

所以平权显示并不更客观，  
反而更容易制造解释失真。

## 7. 哲学本质：优先级是控制面对注意力的治理

这一章最重要的哲学结论是：

`安全控制面的优先级系统，本质上是在治理用户注意力，而不仅是在治理排版顺序。`

谁被立即显示，  
谁被降级到 medium，  
谁只在 low 里排队，  
谁甚至被从 footer 剥离，  
这些设计都在回答同一个问题：

`当前哪一种真相最值得先占据用户的认知窗口。`

所以优先级规则在更深一层上，其实是：

`认知资源分配协议`

而不是简单的视觉技巧。

## 8. 苏格拉底式自我反思

可以继续问六个问题：

1. 为什么 bridge failed 是 `immediate`，而 MCP failed 只是 `medium`，这个分层是否还可以更细？
2. plugin updated 一律 `low` 是否足够，还是某些高风险 plugin refresh failure 应被升级？
3. `invalidates` 机制目前只在少数通知里显式使用，是否应更广泛用于安全状态替换？
4. footer 只保留 reconnecting 的策略，是否也应扩展到其他窄 surface？
5. 下一代统一控制台是否应把“字段优先级”和“通知优先级”统一建模，而不是分散在不同系统里？
6. 若 projection fields 已字段化，优先级是否也应成为这些字段的结构属性，而不是仅靠 notification queue 实现？

这些问题说明：

Claude Code 已经在多个局部实现了优先级纪律，  
下一步最值得做的是把这种纪律统一成跨 surface 的 field-priority protocol。

`统一安全控制台的字段升级阈值与撤场条件`

也就是继续回答：

`哪些字段在什么条件下必须升级显示，哪些字段在什么条件下必须让位或退场。`

## 10. 给系统设计者的技术启示

最后把这一章压成六条工程启示：

1. 不要让所有字段平权竞争用户注意力
2. 让高风险断点优先于维护性提示
3. 用 `invalidates` 这类机制显式驱逐旧弱叙事
4. 在窄 surface 上优先治理抢占，而不是追求字段完整
5. 将通知优先级、字段优先级和 surface 带宽统一思考
6. 真正成熟的安全控制面，必须把“谁先说话”治理成协议

这一章最终要守住的一条硬结论是：

`对 Claude Code 这类安全控制面来说，字段有没有被展示只是第一步，更关键的是它是否在该抢占时抢占、该让位时让位。`
