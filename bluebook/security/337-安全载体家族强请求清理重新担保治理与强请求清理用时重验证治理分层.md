# 安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层：为什么artifact-family cleanup stronger-request cleanup-reassurance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer

## 1. 为什么在 `336` 之后还必须继续写 `337`

`336-安全载体家族强请求清理重新投影治理与强请求清理重新担保治理分层`
已经回答了：

`stronger-request cleanup 线即便已经允许某些 surface 给出更强的正向 reassurance，也还要单独回答这些 reassurance 现在究竟敢签到什么程度。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然刚才已经有了足够正向的 reassurance，真正 consumer 在实际使用这一刻当然可以直接依赖”
的错觉：

`只要 stronger-request cleanup reassurance governor 已经允许某个 surface 给出足够正向的 reassurance，它就自动拥有了替真正 consumer 在依赖瞬间免除 fresh re-check 的主权。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的 use-time revalidation 正对照，
会发现 repo 其实明确拆开了五件事：

1. `consumer-edge hard gate`
2. `fresh reconnect primitive`
3. `batch list live-use discipline`
4. `connected-only execution context`
5. `runtime revocation on contradiction`

最硬的证据至少有五组：

1. `src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:86-97`
   的
   `client.type !== 'connected'`
   hard gate
   与
   `ensureConnectedClient(client)`
2. `src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:81-88`
   关于
   `ensureConnectedClient`
   的 no-op / fresh reconnect 注释与调用
3. `src/services/mcp/client.ts:1688-1699`
   的
   `ensureConnectedClient()`
4. `src/services/tools/toolExecution.ts:308-326`
   的
   `getMcpServerType()` /
   `getMcpServerBaseUrlFromToolName()`
5. `src/services/tools/toolExecution.ts:1599-1618`
   的
   `McpAuthError -> needs-auth`
   demotion

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup reassurance governance`
   负责决定某个 surface 现在配说多强的正向话
2. `stronger-request cleanup use-time revalidation governance`
   负责决定真正 consumer 在实际依赖之前，
   还必须重新拿到什么 fresh enough 的 current-use proof

也就是说：

`artifact-family cleanup stronger-request cleanup-reassurance-governor signer`

和

`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`

仍然不是一回事。

前者最多能说：

`当前有某个 surface 配给出多强的正向 reassurance。`

后者才配说：

`真正的 consumer 在 read / list / execute 这一刻，已经重新拿到 fresh connection、current capability 与可继续依赖的 live-use truth。`

所以 `336` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层`

也就是：

`stronger-request cleanup reassurance governor 决定正向 reassurance 的 ceiling；stronger-request cleanup use-time-revalidation governor 才决定真正使用这一刻刚才那句 reassurance 是否仍配被当真。`

## 2. 先做四条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 live-use gate manager，
而是在说：

1. repo 已经明确把 reassurance surface 与 consumer edge 拆开
2. repo 已经明确要求真正依赖前重新检查 `connected` / current truth
3. repo 已经明确允许 runtime contradiction 即时撤销旧 reassurance

第二条：

这里的 `use-time revalidation`
不是前面那条：

`anti-drift verification`

当前更稳的说法只能是：

`反漂移验证回答未来若再分叉，谁会主动发现；用时重验证回答这次 read/list/execute 真正发生的瞬间，谁会要求重新验真。`

第三条：

这里也不能过度声称：

`use-time revalidation governance`

已经等于：

`step-up reauthorization governance`

当前更稳的说法只能是：

`use-time revalidation 先回答“现在还配不配直接用”；step-up reauthorization 才回答“即便现在能用，对更强请求的 authority level 是否仍然不够”。`

第四条：

这里最后也不能把：

`ensureConnectedClient()`

误读成普通实现细节。

当前更稳的说法只能是：

`reconnect-or-throw`
本身就是 repo 对 current-use truth 的制度化 primitive。

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-reassurance-governor signer 仍不等于 stronger-request cleanup-use-time-revalidation-governor signer`

证据：

1. `ReadMcpResourceTool`
   在真正读资源前，
   既要求 `client.type === 'connected'`，
   又继续调用 `ensureConnectedClient(client)`；
   这说明 positive reassurance 不自动等于 read path may rely now
2. `ListMcpResourcesTool`
   明确把 `ensureConnectedClient`
   注释成 healthy 时 no-op、
   after `onclose` 要拿 fresh connection；
   这说明 browse/list 也拒绝盲信旧 reassurance
3. `ensureConnectedClient()`
   本身会 fresh `connectToServer()`，
   若结果不再是 `connected` 就直接抛错；
   这说明真正依赖前仍要重新验真
4. `toolExecution.ts`
   的 `getMcpServerType()` /
   `getMcpServerBaseUrlFromToolName()`
   只有在 current connection 仍是 `connected` 时才继续派生 execution context；
   这说明连辅助执行上下文也不盲信旧 reassurance
5. `toolExecution.ts`
   在 `McpAuthError` 上会把原本 `connected` 的 client 即时降成 `needs-auth`；
   这说明 stale reassurance 会在真实使用的第一时间被撤销

因此这一章的最短结论是：

`stronger-request cleanup reassurance governor 最多能说当前有某个 surface 配给出多强的正向 reassurance；stronger-request cleanup use-time-revalidation governor 才能说真正 consumer 在使用瞬间是否已经重新拿到 fresh enough 的 current-use proof。`

再压成一句：

`被正向担保，不等于已经通过 live-use 重验证。`

再压成更硬的一句：

`reassured truth，不等于 current-use truth。`

## 4. 第一性原理：reassurance 回答“能放心到什么程度”，use-time revalidation 回答“真正依赖这一刻还能不能继续把刚才那句放心话当真”

从第一性原理看，
强请求清理重新担保治理与强请求清理用时重验证治理处理的是两个不同主权问题。

`stronger-request cleanup reassurance governor`
回答的是：

1. 哪个 surface 现在配说正向话
2. 这句正向话允许多强
3. 是否仍必须保留 caveat
4. 是否宁可沉默也不发布 success
5. 当前 surface 的 reassurance ceiling 是什么

`stronger-request cleanup use-time-revalidation governor`
回答的则是：

1. 真正 consumer 在 read/list/execute 前是否还要 fresh reconnect
2. 当前 `connected` truth 是否要在 use-time 重新检查
3. capability / transport / execution context 是否仍然 current
4. 如果刚才的 reassurance 已经过时，谁会在使用瞬间把它撤销
5. 哪些 positive surface 只是 expectation，哪些能够通过 live gate 转成 actual reliance

如果把这两层压成一句
`既然刚才已经说能用了`，
系统就会制造五类危险幻觉：

1. `reassurance-means-usable-now`
2. `dialog-success-means-resource-readable`
3. `connected-copy-means-fresh-proof`
4. `positive-surface-speaks-for-consumer`
5. `reassurance-once-means-revalidation-forever`

所以从第一性原理看：

`reassurance governance` 管的是 belief strength；
`use-time revalidation governance` 管的是 actual reliance gate。

## 5. `ReadMcpResourceTool` 与 `ListMcpResourcesTool` 先证明：真正 consumer path 信任的是此刻的 fresh truth，而不是刚才那句 reassurance

`src/tools/ReadMcpResourceTool/ReadMcpResourceTool.ts:86-97`
很硬。

这里的顺序非常严格：

1. 找到目标 server
2. 若 `client.type !== 'connected'`，直接失败
3. 即便当前看起来已是 `connected`，仍继续 `ensureConnectedClient(client)`
4. 然后才真正发 `resources/read`

这说明 resource read consumer 明确拒绝把：

`surface 上一个正向 reassurance`

偷写成：

`actual read authorization right now`

`src/tools/ListMcpResourcesTool/ListMcpResourcesTool.ts:81-88`
更值钱。

repo 在注释里直接写出：

1. resource cache 已由 `resources/list_changed` 与 `onclose` 处理
2. healthy 时 `ensureConnectedClient` 是 no-op
3. after `onclose` it returns a fresh connection so the re-fetch succeeds

这说明 list path 的制度不是：

`反正只是列一下资源，旧 reassurance 足够`

而是：

`只要进入真正的 consumer path，就要重新问 current-use proof 现在还在不在`

从安全设计内涵看，
这非常先进。

因为它说明 use-time revalidation
不是只在最高风险的 write/execute path 才存在，
而是连看起来较弱的 browse/list consumer 也同样 obey live truth discipline。

## 6. `ensureConnectedClient()` 再证明：repo 把真正可依赖的 current-use truth 收束成 reconnect-or-throw primitive

`src/services/mcp/client.ts:1688-1699`
特别值钱。

这里 `ensureConnectedClient()` 做的事情非常干净：

1. SDK client 直接返回
2. 非 SDK client 重新 `connectToServer(client.name, client.config)`
3. 若 fresh result 不再是 `connected`，直接抛出
   `MCP server "<name>" is not connected`
4. 只有 fresh result 仍成立时，才返回 connected client

这条证据很硬。

因为它明确把 repo 真正承认的 current-use truth 收束成：

`old connected label -> reconnect -> connected or throw`

它不接受第三种模糊状态。

从第一性原理看，
这正是更成熟的安全设计：

`真正依赖前，系统信任的是 fresh proof，不是上一秒钟看起来还像真的标签。`

## 7. `toolExecution.ts` 再证明：执行平面连辅助 metadata 都要 obey current connected truth

`src/services/tools/toolExecution.ts:308-326`
先给出一组很硬的正例。

这里：

1. `getMcpServerType()` 只有在 `serverConnection?.type === 'connected'` 时才返回 transport type
2. `getMcpServerBaseUrlFromToolName()` 只有在 `serverConnection?.type === 'connected'` 时才返回 base URL

这说明 repo 连“给一次 tool execution 补什么 transport / base-url 背景”这类辅助信息，
都不愿意沿用 stale reassurance。

换句话说，
execution plane 不是只在最核心 request path 严格，
它在外围 metadata 上也坚持：

`connected now or nothing`

这比“只让 tool body 严格、其余上下文宽松”要先进得多。

因为一旦 helper context 允许 stale truth 残留，
系统就会在真正调用前就被污染。

## 8. `McpAuthError -> needs-auth` 再证明：live contradiction 在 repo 里拥有撤销旧 reassurance 的更高主权

`src/services/tools/toolExecution.ts:1599-1618`
更关键。

一旦真实 tool use 遇到 `McpAuthError`，
repo 会立刻：

1. 找到对应 server
2. 只在原状态仍是 `connected` 时执行改写
3. 把 client 降成 `needs-auth`

这里最值钱的不是它会报错，
而是它会主动撤销旧的 connected reassurance，
把 live contradiction 写回 current state。

这说明 use-time revalidation 不是：

`验证失败了就算了，旧 reassurance 还能挂着`

而是：

`验证失败了，旧 reassurance 必须让位于当前反证事实`

这是一条非常成熟的安全设计原则：

`运行时反证比既有正向话术更有主权。`

## 9. Claude Code 在这一层的先进性与技术启示

这组源码至少给出六条值得单独记住的技术启示：

1. 真正 consumer edge 的 fresh proof
   永远不该被 earlier reassurance 冒充
2. current-use truth
   最好收束成少数 canonical primitives，
   例如 reconnect-or-throw
3. helper metadata
   也应 obey current truth，
   而不只是让核心操作严格
4. list / browse
   这类较弱 consumer path
   同样需要 freshness discipline
5. runtime contradiction
   应该拥有正式的 revocation 主权，
   而不是只留下日志
6. 安全系统真正治理的
   不只是“能不能继续说好话”，
   而是“这句好话在实际依赖发生前是否还配被当真”

把这些启示压成更高一层的技术哲学，
就是：

`Claude Code 不只管理 reassurance，还管理 reliance conversion。`

更具体地说：

`它不只管理系统此刻敢让你相信什么，还管理这份相信何时才配转化成真实使用资格。`

## 10. 回到 stronger-request cleanup 线当前仍缺的制度缝

与这套成熟 use-time revalidation grammar 对照，
stronger-request cleanup 线当前虽然已经有了 reassurance grammar，
却还没有谁正式决定：

1. 旧 startup wording
   在真正 cleanup consumer 读取这一刻还要不要 fresh re-check
2. 旧 home-root cleanup law
   在真正被依赖前要由谁重新证明 current applicability
3. 旧 promise vocabulary
   哪些只是 reassurance，
   哪些在 live use 前还必须重新验真
4. 旧 `CleanupResult`
   与其他 local receipt objects
   何时只算正向信号，
   何时才配被转成 current-use proof
5. uncovered diagnostics / coverage gap
   出现时谁有权正式撤销刚才那句“现在可以继续信”的话

这些证据共同说明：

`stronger-request cleanup-reassurance-governance`
仍不能越级冒充
`stronger-request cleanup-use-time-revalidation-governance`

因为 cleanup 线现在最缺的，
已经不是
`这句正向话现在敢说多强`，
而是
`真正使用这一刻这句正向话是否仍配被当真。`

## 11. 用苏格拉底式反问强迫自己不把 reassurance 偷写成 current-use proof

如果我想继续把这一层做得更硬，
至少要反复问自己五句：

1. 我现在说的是
   `某个 surface 配说多强`
   还是
   `真正 consumer 现在可以直接用`？
   如果这两句混了，
   我就在偷把 reassurance 写成 live authorization。
2. `MCPReconnect`
   已经说
   `Successfully reconnected`
   之后，
   为什么 `ReadMcpResourceTool` 还要继续 `ensureConnectedClient()`？
   因为 operation-local reassurance 没有替真实 read path 免检的资格。
3. `/mcp`
   已经显示
   `✓ Connected`
   之后，
   为什么 `toolExecution.ts` 还会在 `McpAuthError` 上立刻把它降回 `needs-auth`？
   因为 health reassurance 不是 immutable current-use proof。
4. `ensureConnectedClient()`
   如果只是实现细节，
   为什么 repo 要把真正 current-use truth 压成 reconnect-or-throw？
   因为 without fresh proof，
   connected label 只是上一刻的记忆，不是这一刻的制度事实。
5. 就算 use-time revalidation 已经成立，
   我能否因此直接跳过下一层“更强请求还需不需要 step-up 重授权”？
   不能。
   因为
   `fresh enough to use`
   仍不等于
   `authorized enough to upscope`。

所以这一章最后必须把自己压回一句最朴素的话：

`被正向担保，不等于已经通过 live-use 重验证；通过 live-use 重验证，也不等于已经拿到更高等级的权限。`
