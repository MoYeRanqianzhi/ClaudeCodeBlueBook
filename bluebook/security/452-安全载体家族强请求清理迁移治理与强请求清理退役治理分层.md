# 安全载体家族强请求清理迁移治理与强请求清理退役治理分层：为什么artifact-family cleanup stronger-request cleanup-migration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-sunset-governor signer

## 1. 为什么在 `451` 之后还必须继续写 `452`

`451-安全载体家族强请求清理修复治理与强请求清理迁移治理分层`
已经回答了：

`stronger-request cleanup 线即便已经知道旧世界怎样被修、怎样被带过渡，也仍需要独立 migration governor 去决定 rewrite、remap、grace window、notice 与 continuity policy。`

但如果继续往下追问，
还会撞上另一层更容易被误写成“会迁就等于会退役”的错觉：

`只要 stronger-request cleanup migration governor 已经安排了新旧世界怎样并存，它就自动拥有宣布旧 stronger-request 世界正式结束的主权。`

Claude Code 当前源码并不能支持这种更强说法。

因为继续看 repo 自己最成熟的 sunset 正对照，
会发现它其实已经把：

`migration clock`

和

`sunset clock`

拆成了两种不同制度。

最硬的证据至少有七组：

1. `src/main.tsx:323-346`
   直接维护 `CURRENT_MIGRATION_VERSION` 与 startup migration chain，
   但它回答的只是
   `哪些迁移动作这次启动需要重跑`
   而不是
   `旧世界从什么时候起不再算 current truth`
2. `src/utils/model/deprecation.ts:21-25,33-100`
   已把 deprecated models 写成 provider-specific retirement-date plane，
   说明 repo 明确区分
   `已经存在迁移路径`
   和
   `旧模型在哪个 provider、哪一天正式 retired`
3. `src/hooks/notifs/useModelMigrationNotifications.tsx:5-8,21-33,49-50`
   只负责 one-time migration timestamp notice，
   说明“刚迁完”本来就是一张独立时钟
4. `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-42`
   与
   `src/main.tsx:2872-2895`
   又把持续 deprecation warning 做成另一张 runtime surface，
   说明“即将退役”不是 migration 的副产物
5. `src/migrations/migrateLegacyOpusToCurrent.ts:13-57`
   即便已经改写 `userSettings`，
   `src/utils/model/model.ts:549-554`
   仍保留 `CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP`；
   这说明 migration 已发生时，compatibility residue 仍可继续活着
6. `src/utils/plugins/cacheUtils.ts:23-24,69-105,149-176`
   把 orphan plugin version 先放进 `7 days` grace world，再决定删除；
   这不是 migration 本身，而是 storage sunset clock
7. `src/utils/plugins/orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
   与
   `src/main.tsx:2544-2568`
   又把 visibility sunset 单独做成 session-scoped exclusion cache，
   说明旧 plugin world 可以先退出 grep/glob current world，再晚一点退出 storage world

这说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界先怎样被带过渡。`

后者才配说：

`这段过渡什么时候结束，旧 stronger-request law 从哪一个日期、哪一个 runtime、哪一个 visibility surface 起正式不再算当前世界。`

所以 `451` 之后必须继续补的一层就是：

`安全载体家族强请求清理迁移治理与强请求清理退役治理分层`

也就是：

`stronger-request cleanup migration governor 负责 coexistence grammar；stronger-request cleanup sunset governor 才负责宣布 coexistence 结束。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

这里的 `artifact-family cleanup stronger-request cleanup-sunset-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经存在一个未导出的退役 manager，
而是在说：

1. repo 在 model 与 plugin 线上已经展示出正式 sunset grammar
2. sunset grammar 回答的是 retirement date、warning phase、visibility cutoff 与 hard-close boundary
3. stronger-request cleanup 若要自称完整，就必须回答这些“何时正式结束”的问题

第二条：

这里说的 `sunset governance`

不是在声称：

`旧世界一旦退役，就必须立刻从磁盘彻底消失。`

更稳的说法只能是：

`sunset 首先回答的是“从什么时候起它不再算当前世界”，这可以先发生在 warning、visibility、runtime acceptance 上，再发生在 storage deletion 上。`

第三条：

这里也不是把：

`provider-specific retirement date`

单独当成全部 sunset grammar。

真正关键的是：

1. model line 证明 migration notice 与 deprecation warning 是两张时钟
2. remap opt-out 证明 migration 已发生时旧 world 仍可兼容存在
3. plugin line 又证明 visibility sunset 与 storage sunset 还能继续拆层
4. `CURRENT_MIGRATION_VERSION` 更进一步说明 migration registry 也不等于 sunset registry

第四条：

这里尤其不能过度声称：

`warning surface` 就已经等于 `sunset governance`。

更稳的说法只能是：

`warning surface` 只是退役语法的一种表达；它仍需要与 retirement date、visibility cutoff、runtime acceptance cutoff 与 storage cutoff 一起构成完整 sunset grammar。`

第五条：

这里也不能把：

`runtime remap 仍然存在`

误写成：

`sunset 尚不存在。`

更稳的说法只能是：

`runtime remap 的继续存在恰恰证明 migration 与 sunset 被拆开了；旧世界可以被兼容一段时间，而不是迁移一发生就被自动 hard-close。`

第六条：

这里还不能过度声称：

`Claude Code 在 model / plugin 线上怎样 sunset，stronger-request cleanup 线未来就必须完全照抄。`

更稳的说法只能是：

`不同旧世界可以有不同 cutoff grammar，但“谁来宣布兼容期结束”必须被正式回答；不能因为实现语法可变，就把 sunset 主权偷写成 migration 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了七类

`stronger-request cleanup-migration-governor signer 仍不等于 stronger-request cleanup-sunset-governor signer`

证据：

1. `deprecation.ts:21-25,33-100`
   已为 deprecated models 维护 provider-specific retirement dates，
   说明 repo 已把 retirement date 当成正式 truth
2. `useModelMigrationNotifications.tsx:5-8,21-33,49-50`
   只负责 one-time migration notification，
   而 `useDeprecationWarningNotification.tsx:6-42` 与 `main.tsx:2872-2895`
   负责持续 deprecation warning
3. `migrateLegacyOpusToCurrent.ts:13-57`
   即便已把 `userSettings` 迁到 `opus`，
   `model.ts:549-554`
   仍保留 legacy remap opt-out，
   说明 migration 已发生但 sunset 尚未 hard-close
4. `.orphaned_at` 与 `CLEANUP_AGE_MS = 7 days`
   说明 orphan plugin version 先进入 grace world，再在期限后删除
   `cacheUtils.ts:23-24,149-176`
5. `orphanedPluginFilter.ts:1-15,25-26,35-37,47-87`
   与 `main.tsx:2544-2568`
   说明旧 plugin version 可以先从当前 session 的 visibility world 退役，再从 storage world 消失
6. `CURRENT_MIGRATION_VERSION`
   只能证明哪些迁移动作需要重跑，并不自动签出“旧世界已经正式退役”
   `main.tsx:323-346`
7. stronger-request cleanup 当前虽然已开始显露 migration-governance，
   却还没有谁正式决定旧 startup wording、旧 home-root cleanup law、旧 covered-family covenant、旧 receipt / local-count truth 与旧 stronger-request promise 何时停止作为 current world 继续存在

因此这一章的最短结论是：

`stronger-request cleanup migration governor 最多能说旧世界怎样过渡；stronger-request cleanup sunset governor 才能说旧世界何时停止作为正式世界继续存在。`

再压成一句：

`会迁，不等于会退役。`

再压成更哲学一点的话：

`制度知道怎样带旧世界过桥，不等于制度知道何时封桥。`

## 4. 第一性原理：migration governance 回答“如何并存”，sunset governance 回答“何时终止并存”

从第一性原理看，
这两层治理处理的是两个不同时间面的主权问题。

`migration governor`

回答的是：

1. 旧值是 rewrite、remap，还是 dual-read
2. 哪些 source 可以改写，哪些必须保持原样
3. 兼容期间怎样提示用户
4. 哪些旧 artifact 要进入 grace window
5. 怎样保持幂等与 continuity

`sunset governor`

回答的则是：

1. 兼容窗口在哪个日期、哪次启动或哪类 provider 后结束
2. 旧值何时不再被 runtime 当成 current truth
3. 旧 artifact 何时不再被搜索、引用或恢复
4. 旧 warning 何时从“建议切换”进入“正式不可再用”
5. 双世界共存何时必须切成单世界真相

所以这两层的哲学本质不同：

`migration`

是在管理共存；

`sunset`

则是在管理共存的终止。

Claude Code 在这方面真正先进的地方，
不是它“会在旧世界不对时马上带去新世界”，
而是它已经明确承认：

`旧世界何时不再算当前世界，本身也是正式安全问题。`

这是一种很成熟的安全哲学：

`制度会安排交接，不等于制度已经有权宣告前任死亡。`

## 5. Claude Code 多重安全技术的先进性，不只在会迁旧世界，还在会把旧世界的终止拆成多张时钟

很多系统一谈退役，
就只剩下一句：

`deprecated`

Claude Code 更成熟的地方在于，
它会把同一个旧世界拆成多张不同的时钟。

### 5.1 provider-specific retirement clock：旧世界何时结束，本来就可能因 provider 而异

`deprecation.ts`
最硬的事实是：

1. retirement date 是 provider-specific
2. 同一旧模型在不同 provider 上可以有不同 retired date
3. `null` 本身也表达“这个 provider 上还不 deprecated”

这说明 sunset 不是一个抽象布尔值，
而是有环境维度的正式 truth。

### 5.2 migration notice clock：刚发生的迁移和已经发生的退役不是一件事

`useModelMigrationNotifications.tsx`
回答的是：

`this startup moved you`

而
`useDeprecationWarningNotification.tsx`
与
`main.tsx` 初始通知队列
回答的是另一件事：

`this old thing will soon stop being valid`

这意味着：

1. 刚被带过渡
2. 即将被退役

本来就是两种不同时间话语。

### 5.3 compatibility residue：旧世界被 remap 兼容时，恰恰证明 sunset 还没结束

`model.ts:549-554`
保留 legacy remap opt-out，
说明旧字符串虽已被迁移触发，
仍然还能在 runtime 世界里继续被兼容。

这说明 migration 成功，
并不会自动结束 sunset。

### 5.4 visibility sunset 与 storage sunset：旧世界先退出“可见世界”，再退出“存在世界”

`orphanedPluginFilter.ts`
与
`cacheUtils.ts`

共同说明：

1. orphan 版本可先从 grep/glob visibility 世界退役
2. 再在 grace window 结束后从 storage 世界消失

这说明：

`是否还在磁盘上`

和

`是否还算 current world`

不是同一个问题。

## 6. 为什么 stronger-request cleanup 现在仍停在“transition grammar”，而不是“current-truth cutoff grammar”

和这些成熟 sunset 正对照相比，
stronger-request cleanup 当前最明显的缺口已经不是：

`旧世界怎么过渡。`

而是：

`旧世界什么时候不再被承认为当前真相。`

更具体地说，
现在仍没有哪一层正式回答：

1. 旧 startup-delete wording 到哪一次启动、哪一类 surface 为止还算被 tolerated
2. 旧 home-root cleanup law 何时只是 retained residue，何时必须不再被 current policy 承认
3. 旧 covered-family covenant 何时退出 visibility world，何时退出 storage / continuity world
4. 旧 receipt / local-count truth 何时只是解释历史，何时不再能被当成 current result
5. 旧 stronger-request promise 到何时仍处在 grace window，何时必须被 hard-close

换句话说，
stronger-request cleanup 当前虽然已开始显露 migration-governance，
却还没有长出足够正式的：

1. retirement date / cutoff plane
2. sustained warning plane
3. visibility sunset plane
4. storage sunset plane
5. hard-close authority

这意味着它现在能说的是：

`旧世界先怎样被带过去。`

却还不能更强地说：

`从什么时候起，旧世界不再配继续被承认为 current truth。`

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界已经被过渡处理”误认成了“旧世界已经正式结束”

如果对这一层做更严格的自我审查，
至少要追问六句：

1. 如果 migration strategy 已经明确，为什么还要再拆 sunset？
   因为会并存，不等于知道并存何时结束。
2. 如果 warning 已经显示，是否就等于 sunset 已经发生？
   不等于。warning 只是 pressure，不是 hard-close。
3. 如果 runtime remap 仍在，是否说明 sunset 不存在？
   也不对。恰恰说明 sunset 还没结束，因而 migration 与 sunset 被分开治理。
4. 如果 orphan 版本还在磁盘上，是否说明它仍算当前世界？
   也不对。visibility world 和 storage world 可以先后退役。
5. 如果旧 law 先从 visibility 世界退役，是不是就等于一切都结束了？
   也不等于。storage / recovery / continuity 可能还保留残留事实。
6. 如果 cleanup 线现在还没有显式 sunset 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确时钟，越容易把“还在兼容”误说成“已经正式结束”，或反过来。

这一串反问最后逼出一句更稳的判断：

`sunset 的关键，不在系统能不能让旧世界先共存一段时间，而在系统能不能正式决定这段共存从什么时候起不再被承认为当前世界。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 migration governance，就已经足够成熟。`

而是：

`Claude Code 在 migration registry、model deprecation clock、migration timestamp、warning surface、plugin orphan grace window 与 session-level visibility cutoff 上已经明确展示了 sunset governance 的存在；因此 artifact-family cleanup stronger-request cleanup-migration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“怎么把旧世界带着走一段路”，而是“谁来宣布这段路何时结束”。`
