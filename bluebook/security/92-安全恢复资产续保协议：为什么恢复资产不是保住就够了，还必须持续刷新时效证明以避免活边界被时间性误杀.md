# 安全恢复资产续保协议：为什么恢复资产不是保住就够了，还必须持续刷新时效证明以避免活边界被时间性误杀

## 1. 为什么在 `91` 之后还要继续写“恢复资产续保协议”

`91-安全恢复资产保全义务` 已经回答了：

`在恢复资格未被正式撤回前，某些恢复资产必须被制度性保留。`

但如果继续往下追问，  
还会遇到一个更时间化、也更接近运行时现实的问题：

`资产被保留下来，就等于它仍然新鲜、仍然能托底恢复资格吗？`

显然不等于。

因为恢复资产除了会被错误清掉，  
还会发生另一种更隐蔽的退化：

`资产还在，但已经因为缺乏续保而时间性失效。`

所以在 `91` 之后，  
安全专题还必须再补一章，把“保全义务”继续推进成一条动态协议：

`恢复资产续保协议。`

也就是：

`成熟控制面不仅要保留恢复资产，还必须持续刷新这些资产的时效证明，确保它们在需要被恢复时仍然是活资产，而不是被时间悄悄判死的旧资产。`

## 2. 最短结论

从源码看，Claude Code 已经把恢复资产的新鲜度治理写成了一套明确协议：

1. `bridgePointer.ts` 明确把 staleness anchor 绑定到 file `mtime`，而不是嵌入式时间戳，并把 TTL 设为 4h，以对齐 backend rolling `BRIDGE_LAST_POLL_TTL`  
   `src/bridge/bridgePointer.ts:22-40,76-113`
2. pointer write 明确支持“同内容重写只为 bump staleness clock”，说明续保是正式动作，不是 incidental side effect  
   `src/bridge/bridgePointer.ts:56-74`
3. REPL bridge 在 work dispatch 时刷新 pointer mtime，保证高活跃会话持续拥有 crash-recovery freshness  
   `src/bridge/replBridge.ts:1101-1109`
4. perpetual 模式下还额外引入 hourly refresh timer，因为仅靠 prompt 驱动的刷新不足以保护 idle 但仍活着的边界  
   `src/bridge/replBridge.ts:1505-1524`
5. suspend 路径也会再次刷新 pointer，显式避免长时会话在下次启动时被误判 stale  
   `src/bridge/replBridge.ts:1605-1611`
6. standalone bridge 同样在初写后启动 hourly refresh，表明续保不是 REPL 特例，而是跨 runtime 的统一原则  
   `src/bridge/bridgeMain.ts:2700-2728`
7. `readBridgePointerAcrossWorktrees(...)` 最终只挑 freshest pointer，说明 freshness 不仅决定真假，还决定候选优先级  
   `src/bridge/bridgePointer.ts:166-181`

所以这一章的最短结论是：

`Claude Code 不只在保护恢复资产的存在性，它还在保护这些资产的新鲜度证明。`

我把它再压成一句：

`活资产必须持续被证明还活着。`

## 3. 源码已经说明：恢复资产必须拥有持续续保的 freshness proof

## 3.1 staleness 被正式锚定在 mtime 上，而不是只靠“最近看起来像是活的”

`src/bridge/bridgePointer.ts:22-40` 已经把 pointer 语义说得很直白：

1. pointer 会被定期刷新
2. staleness 检查依赖 file `mtime`
3. 这样做是为了对齐 backend 的 rolling TTL

这说明 Claude Code 团队没有把 freshness 理解成模糊直觉，  
而是把它设计成：

`有物理锚点的时效证明。`

一旦 `mtime` 超过 4h，  
pointer 就会被视为 stale 并被清除。

所以恢复资产不是：

`存在就算活`

而是：

`存在且带着足够新的时效证明才算活`

## 3.2 同内容重写的意义不是写文件，而是续保

`src/bridge/bridgePointer.ts:56-74` 很关键。

这里作者明确说：

1. 用相同 IDs 重写 pointer 也是合理的
2. 因为这会 bump staleness clock

这说明在作者眼里，  
同内容写入不是低价值噪音，  
而是一种正式协议动作：

`续保`

从第一性原理看，  
这也说明系统真正维护的对象不是 pointer payload 本身，  
而是：

`payload + freshness proof`

后者一旦消失，  
前者也会退化成伪资产。

## 3.3 高活跃边界靠真实工作流自动续保，说明活跃度本身可以成为 freshness signal

`src/bridge/replBridge.ts:1101-1109` 说明 REPL bridge 在 work dispatch 时会刷新 pointer mtime。

这意味着：

1. 只要当前会话仍在正常接收和处理工作
2. 它就会自然地产生 freshness proof

这是一种很成熟的设计。  
因为它让：

`真实活跃`

直接变成：

`时效证明自动续保`

所以高活跃边界不需要额外心智负担，  
其正常行为本身就在保护其恢复资格。

## 3.4 idle-but-live 的边界不能靠“最近有消息”来碰运气，所以必须有独立续保协议

`src/bridge/replBridge.ts:1505-1524` 是这一章最重要的证据之一。

作者直接写出：

1. 仅靠 `onWorkReceived` 刷新还不够
2. daemon idle 超过 4h 会导致 pointer stale
3. 下次 restart 就会 clear it -> fresh session
4. 因此 perpetual 模式下需要 hourly timer

这说明 Claude Code 已经清楚区分：

1. active and self-refreshing assets
2. idle but still live assets

后一类如果没有独立续保协议，  
会被时间误杀。

所以 hourly timer 的真正意义不是后台小优化，  
而是：

`保护 idle-but-live 边界不被时间性判死`

## 3.5 suspend 前再补一笔 freshness proof，说明续保也属于生命周期切换协议的一部分

`src/bridge/replBridge.ts:1605-1611` 进一步说明：

系统在进入 perpetual suspend 前，  
还会显式刷新一次 pointer。

这说明续保协议不只是周期任务，  
还包括：

`生命周期切换前的 checkpoint refresh`

也就是说，  
当系统从活跃运行态切到 suspend 可恢复态时，  
它会先补一笔最新时效证明，  
防止边界带着旧 freshness 进入新阶段。

这是一种非常成熟的设计：  
不仅持续保，  
还在关键制度切换点补保。

## 3.6 standalone bridge 的同类 hourly refresh 说明：续保不是 REPL 补丁，而是统一原则

`src/bridge/bridgeMain.ts:2700-2728` 和 `src/bridge/replBridge.ts:1505-1524` 放在一起看很有力量。

standalone bridge 同样：

1. 初写 pointer
2. 启动 hourly refresh
3. 解释这样做是为了让 5h+ 会话 crash 后仍有 fresh pointer

这说明恢复资产续保不是：

`某个局部 runtime 的临时战术`

而是：

`跨 runtime 一致的控制面原则`

从工程成熟度看，  
这意味着团队已经把 freshness governance 抽象成可迁移设计法则。

## 3.7 freshest 选择说明：freshness 不只决定真假，还决定仲裁优先级

`src/bridge/bridgePointer.ts:166-181` 揭示了更深一层。

worktree fanout 最终会选择：

`lowest ageMs`

也就是 freshest pointer。

所以 freshness proof 在这里承担了两种职责：

1. validity threshold  
   够不够新
2. arbitration priority  
   多个候选里该先信谁

这使得续保协议不只是资产维护，  
更是：

`恢复候选仲裁机制的一部分`

## 3.8 TTL 对齐 backend 语义，说明续保协议必须和远端时间制度同构

`src/bridge/pollConfigDefaults.ts:20-29` 与 `src/bridge/bridgePointer.ts:30-34` 一起看很有价值。

这里作者明确让 pointer TTL 与 backend rolling `BRIDGE_LAST_POLL_TTL = 4h` 对齐。

这说明一个很成熟的原则：

`本地恢复资产的新鲜度证明不能自说自话，它必须和远端的存活制度同构。`

否则会出现：

1. 本地觉得资产还新鲜
2. 远端其实已经把边界视为过期

这类制度分裂。

Claude Code 明显在主动避免这种错配。

## 3.9 第一性原理：恢复资产如果不被持续续保，就会从“活资产”退化成“时间性伪资产”

从第一性原理看，  
恢复资产真正怕的并不只是被删除。

它更怕一种更隐蔽的退化：

1. 文件还在
2. 名字还在
3. 用户还能看见它
4. 但系统已经没有足够新的证据证明它仍值得相信

这时资产就会从：

`recoverable asset`

退化成：

`time-decayed pseudo-asset`

而续保协议的本质，  
就是防止这种退化。

## 3.10 技术先进性：Claude Code 已经把 freshness proof 变成恢复安全的一等对象

从技术角度看，  
很多系统会把 freshness 当后台 housekeeping。

但 Claude Code 更成熟的地方在于：

1. freshness proof 有明确物理锚点：mtime
2. freshness 与 backend rolling TTL 对齐
3. active path 和 idle path 有不同续保机制
4. suspend 切换时会显式补写 freshness proof
5. freshest 选择让 freshness 进入了候选仲裁

这说明它真正治理的不是“文件多久刷一次”，  
而是：

`恢复资产的时效证明连续性`

## 4. 安全恢复资产续保协议的最短规则

把这一章压成最短规则，就是下面七句：

1. 恢复资产不仅要存在，还要持续拥有 freshness proof
2. freshness proof 应绑定明确物理锚点，而不是只靠内存猜测
3. 活跃边界应尽量由真实工作流自动续保
4. idle-but-live 边界必须有独立续保协议
5. 生命周期切换前应补写最新 freshness proof
6. freshness 不仅决定资产真假，还决定候选优先级
7. 成熟控制面治理的不只是恢复资产的存在性，还包括其时效证明的连续性

## 5. 苏格拉底式追问：这套系统还能怎样再向前推进

如果继续自问，  
下一步最值得补的三项升级会是：

1. 给恢复资产显式区分 `fresh`、`aging`、`stale` 三态，而不是只在超过 TTL 后瞬时翻转
2. 把 freshness proof 的最近来源显式暴露出来，例如 `from work dispatch`、`from hourly timer`、`from suspend checkpoint`
3. 在统一安全控制台中加入 recoverability freshness 卡片，让用户看到资产为何仍新鲜、多久后会衰减

所以这一章最终沉淀出的设计启示是：

`成熟安全系统不只要保住恢复资产，还必须持续续保这些资产的时效证明，避免活边界被时间悄悄判死。`
