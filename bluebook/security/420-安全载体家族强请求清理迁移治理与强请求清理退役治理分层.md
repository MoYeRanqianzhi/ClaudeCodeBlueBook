# 安全载体家族强请求清理迁移治理与强请求清理退役治理分层：为什么artifact-family cleanup stronger-request cleanup-migration-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-sunset-governor signer

## 1. 为什么在 `419` 之后还必须继续写 `420`

`419-安全载体家族强请求清理修复治理与强请求清理迁移治理分层`
已经回答了：

`stronger-request cleanup 线即便未来知道旧世界怎样 rewrite、remap、grace、notice 与 continuity-preserve，也仍需要独立 migration governor 去决定“旧世界怎样被带离现场”。`

但如果继续往下追问，
还会撞上另一层更容易被误写成“会迁就等于会退役”的错觉：

`只要 stronger-request cleanup migration governor 已经安排了新旧世界怎样并存，它就自动拥有宣布旧 stronger-request 世界正式结束的主权。`

Claude Code 当前源码不能支持这种更强说法。

因为继续看 repo 自己最成熟的 sunset 正对照，
会发现它其实已经把：

`coexistence grammar`

和

`coexistence end condition`

拆成了不同层。

最硬的证据至少有八组：

1. `src/main.tsx:323-346`
   直接维护 `CURRENT_MIGRATION_VERSION` 与 startup migration chain，
   说明 repo 很清楚“哪些迁移动作要重跑”，
   但这仍不是“旧世界何时不再算当前世界”
2. `src/utils/model/deprecation.ts:21-25,33-100`
   已把 deprecated model 的 retirement date 写成 provider-specific truth，
   说明 sunset clock 是正式制度对象，而不是 migration 的尾注
3. `src/hooks/notifs/useModelMigrationNotifications.tsx:5-50`
   只在 migration timestamp 刚写入时弹一次 notification，
   这说明 migration clock 回答的是
   `刚发生了过渡`
   而不是
   `旧世界已经结束`
4. `src/hooks/notifs/useDeprecationWarningNotification.tsx:6-42`
   与 `src/screens/REPL.tsx:744-762`
   又把 deprecation warning 做成持续 runtime surface，
   说明 repo 单独维护
   `尚在兼容，但退役时钟已开始计时`
5. `src/main.tsx:2872-2895`
   继续在 initial notification queue 注入 deprecation warning，
   说明 sunset grammar 甚至有 startup-projected 入口，
   而不是只靠 migration hook 顺手附带
6. `src/migrations/migrateLegacyOpusToCurrent.ts:13-57`
   即便已经把 `userSettings` 改成 `opus` 并写入 `legacyOpusMigrationTimestamp`，
   `src/utils/model/model.ts:549-554`
   仍保留 `CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP` 这一 compatibility residue，
   证明 migration 已发生时 sunset 仍未 hard-close
7. `src/utils/plugins/cacheUtils.ts:23-24,66-176`
   已给 orphan plugin version 写出 `7 days` grace window，
   说明 storage sunset 自己就有单独时钟
8. `src/utils/plugins/orphanedPluginFilter.ts:1-91`
   与 `src/main.tsx:2544-2568`
   又进一步把 visibility cutoff 做成 session-scoped frozen exclusion list，
   说明旧 world 可以先退出 grep/glob current world，
   再晚一点才退出磁盘世界

这说明：

`artifact-family cleanup stronger-request cleanup-migration-governor signer`

和

`artifact-family cleanup stronger-request cleanup-sunset-governor signer`

仍然不是一回事。

前者最多能说：

`旧 stronger-request 世界先怎样被过渡、重映射、并存和解释。`

后者才配说：

`这段并存从什么时候起必须停止，旧 stronger-request law 从哪一个日期、哪一个 runtime surface、哪一个 visibility surface 起正式不再算 current truth。`

所以 `419` 之后必须继续补的一层就是：

`安全载体家族强请求清理迁移治理与强请求清理退役治理分层`

也就是：

`stronger-request cleanup migration governor 负责把旧世界带过桥；stronger-request cleanup sunset governor 才负责决定什么时候封桥。`

## 2. 先做六条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

这里仍是研究命名。
它不是在声称 cleanup 线已经有一个只差导出的退役 manager，
而是在说：

1. repo 在 model / plugin 线上已经明白展示出 sunset grammar
2. sunset grammar 回答的是 retirement date、warning phase、visibility cutoff 与 hard-close boundary
3. stronger-request cleanup 若要真正成熟，就迟早要回答“兼容期何时正式结束”

第二条：

这里说的 `sunset governance`
不是在声称：

`旧世界一退役，就必须立刻从所有 carrier 彻底消失。`

更稳的说法只能是：

`sunset 首先回答的是“从什么时候起它不再算当前世界”，这可以先发生在 warning、visibility 与 runtime acceptance 上，再发生在 storage deletion 上。`

第三条：

这里也不是把：

`provider-specific retirement date`

单独当成全部 sunset grammar。

真正关键的是：

1. `migration timestamp` 与 `retirement date` 是两张时钟
2. `warning surface` 与 `hard-close signal` 不是一回事
3. `visibility cutoff` 与 `storage cutoff` 也不是一回事
4. `CURRENT_MIGRATION_VERSION` 更不是 sunset registry 的替身

第四条：

这里尤其不能过度声称：

`warning surface` 就已经等于 `sunset governance`。

更稳的说法只能是：

`warning surface` 只是退役制度的一个用户可见表达；它仍需要与 retirement date、runtime acceptance cutoff、visibility cutoff 与 storage cutoff 一起构成完整 sunset grammar。`

第五条：

这里也不能把：

`runtime remap 仍然存在`

误写成：

`sunset 不存在。`

更稳的说法只能是：

`runtime remap 继续存在，恰恰证明 migration 与 sunset 被拆开了；旧世界可以被兼容一段时间，而不是迁移一发生就自动 hard-close。`

第六条：

这里还不能过度声称：

`Claude Code 在 model / plugin 线上怎样 sunset，stronger-request cleanup 线未来就必须逐字照抄。`

更稳的说法只能是：

`不同旧世界可以有不同 cutoff grammar，但“谁来宣布兼容期结束”必须被正式回答；不能因为实现语法可变，就把 sunset 主权偷写成 migration 的副产品。`

## 3. 最短结论

Claude Code 当前源码至少给出了八类

`stronger-request cleanup-migration-governor signer 仍不等于 stronger-request cleanup-sunset-governor signer`

证据：

1. `CURRENT_MIGRATION_VERSION` 只回答 migration set 是否需要重跑，不回答 retirement date
   `main.tsx:323-346`
2. `deprecation.ts` 已把 provider-specific retirement dates 升格成正式 truth
   `deprecation.ts:21-25,33-100`
3. `useModelMigrationNotifications.tsx` 只做 one-time migration notification
   `useModelMigrationNotifications.tsx:5-50`
4. `useDeprecationWarningNotification.tsx` 与 `REPL.tsx` 做的是持续 warning surface，而不是 migration replay
   `useDeprecationWarningNotification.tsx:6-42`; `REPL.tsx:744-762`
5. `main.tsx` 还会在 initial notification queue 再注入一次 deprecation warning，说明 sunset 有自己的 startup-projection plane
   `main.tsx:2872-2895`
6. `migrateLegacyOpusToCurrent.ts` 已迁移用户设置，但 `model.ts` 仍保留 legacy remap opt-out，证明 compatibility residue 仍活着
   `migrateLegacyOpusToCurrent.ts:13-57`; `model.ts:549-554`
7. `.orphaned_at + 7 days`
   说明 storage sunset 有自己的 grace clock
   `cacheUtils.ts:23-24,66-176`
8. `orphanedPluginFilter.ts` 与 `main.tsx`
   说明 visibility sunset 甚至可以先于 storage sunset 发生
   `orphanedPluginFilter.ts:1-91`; `main.tsx:2544-2568`

因此这一章的最短结论是：

`stronger-request cleanup migration governor 最多能说旧世界怎样被过渡；stronger-request cleanup sunset governor 才能说旧世界何时停止作为正式世界继续存在。`

再压成一句：

`会迁，不等于会退役。`

再压成更硬的一句：

`制度知道怎样带旧世界过桥，不等于制度知道何时封桥。`

## 4. 第一性原理：migration governance 管状态转换，sunset governance 管当前真相的截止条件

从第一性原理看，
这两层治理不是在回答同一个时间问题。

`migration governor`

处理的是：

1. 哪些旧值要 rewrite
2. 哪些旧值只 remap
3. 哪些 source 保持原样
4. 兼容期间怎样通知、怎样双读、怎样 continuity-preserve
5. 旧世界怎样在新世界旁边继续活一段时间而不立刻造成断裂

`sunset governor`

处理的则是：

1. 兼容窗口在哪个日期、哪次启动、哪类 provider 或哪类 surface 后结束
2. 旧值何时不再被 runtime 接受
3. 旧 artifact 何时不再被搜索、引用、恢复或当作当前世界继续显示
4. warning 何时从“建议切换”进入“旧世界正式不再算 current truth”
5. storage residual 何时只是历史残留，而不再拥有 current-world legitimacy

所以两层的哲学本质不同：

`migration`
是在管理状态转换；

`sunset`
是在管理合法性的截止。

换句话说，

`migration` 修改的是世界怎样变；

`sunset` 修改的是旧世界从何时起不再有资格被叫做“现在”。

这就是为什么 `CURRENT_MIGRATION_VERSION`
再正式，
也仍然不等于 retirement registry。

因为前者只回答：

`这次要不要把迁移动作跑一遍。`

后者回答的却是：

`跑完之后，旧世界还能算多久。`

## 5. Claude Code 多重安全技术的先进性，在于它用多张时钟治理同一个旧世界

很多系统一谈 sunset，
就把所有问题压成一句：

`给旧世界一个截止日期。`

Claude Code 更成熟的地方在于，
它没有把退役问题压成单时钟，
而是把不同 cutoff 分散到不同控制面：

1. `migration version clock`
   决定哪些过渡动作需要重跑
2. `migration timestamp clock`
   决定这次启动是否显示 one-time migration notice
3. `provider-specific retirement clock`
   决定旧模型在哪个 provider 上何时正式 retired
4. `runtime warning surface`
   决定当前用户什么时候被持续提醒
5. `visibility cutoff`
   决定旧 plugin version 什么时候先退出 grep/glob current world
6. `storage cutoff`
   决定旧 orphan version 什么时候才真正被删除

这套设计的先进性，
不只在“层多”，
而在它对风险类型的区分很清楚：

1. 对 `用户理解风险`
   它给持续 warning
2. 对 `配置误导风险`
   它给 migration timestamp 与 one-time notice
3. 对 `搜索污染风险`
   它给 session-scoped visibility exclusion cache
4. 对 `并发 session 误伤风险`
   它给 `7 days` grace window
5. 对 `跨 provider 现实差异`
   它给 provider-specific retirement dates

这意味着 Claude Code 的安全设计内涵，
不是“找到一个最强 cutover 动作，把所有旧世界一次砍掉”，
而是：

`先把旧世界拆成不同风险面，再给每个风险面配不同的结束语法。`

这背后对应的哲学本质其实很强：

`退役不是删除动作，而是多种合法性逐步收回的过程。`

系统越成熟，
越不会把：

`用户可见性`

`runtime 接受性`

`磁盘存留性`

混写成同一个布尔值。

## 6. 从源码设计思路看，Claude Code 为什么值得学

把这组 sunset 代码重新压缩成设计思路，
至少能得到七条稳定启示：

1. `迁移注册表` 应和 `退役注册表` 拆开
   否则系统永远只能回答“是否重跑动作”，却回答不了“何时正式封桥”
2. `one-time migration notice` 应和 `sustained deprecation warning` 拆开
   否则“刚迁完”和“旧世界将停止计数”会被用户误认成同一句话
3. `compatibility residue` 应被显式承认
   例如 `CLAUDE_CODE_DISABLE_LEGACY_MODEL_REMAP` 这种 opt-out，本身就在证明 sunset 尚未 hard-close
4. `visibility cutoff` 应和 `storage deletion` 拆开
   搜索面污染风险常常早于磁盘残留风险，先切 visibility 往往更安全
5. `warning surface` 应有 scope discipline
   `useDeprecationWarningNotification()` 在 remote mode 下直接退出，说明不是所有 runtime 都需要同一类退役提示
6. `session-scoped freeze` 很重要
   orphan exclusion cache 一旦生成就冻结到 `/reload-plugins`，这不是偷懒，而是在保证当前 session 的 current world 不被并发磁盘突变来回改写
7. `sunset` 不是“迁移之后的尾巴”，而是 `current truth authority` 的单独层
   只有这样，系统才不会在 coexistence 仍然存在时假装旧世界已经死了，或者在旧世界已不该被看见时继续让它污染当前 surface

这些启示放回 stronger-request cleanup，
就会逼出同样的问题：

1. 旧 startup-delete wording 何时不再允许继续代表当前清理法律
2. 旧 home-root cleanup law 何时不再算 current cleanup truth
3. 旧 covered-family covenant 何时只剩历史兼容意义，而不再能支撑 stronger promise
4. 旧 receipt / local-count semantics 何时不再能被当作 current-world receipt
5. 哪些旧 world 先从 visibility 上退役，哪些要等到 storage / recovery 面也关掉才算结束

Claude Code 当前 stronger-request cleanup 线还没有把这些问题正式写出来。

因此这里真正被钉住的不是：

`cleanup 线还缺一个更强 delete。`

而是：

`cleanup 线还缺一个更强“current-world legitimacy cutoff” grammar。`

## 7. 苏格拉底式自反诘问：我是不是又把“旧世界已被安排”误认成了“旧世界已被终止”

如果对这一层做更严格的自我审查，
至少要追问八句：

1. 如果 migration path 已经完整，为什么还要再拆 sunset？
   因为会并存，不等于知道并存何时结束。
2. 如果 migration notice 已经弹出，是否就等于旧世界已退役？
   不等于。那只说明这次启动发生了过渡，不说明兼容期已经封桥。
3. 如果 deprecation warning 已在 REPL 和 startup queue 出现，是否就等于 hard-close 已发生？
   也不等于。warning 是制度性施压，不是制度性封禁。
4. 如果 runtime remap 仍在，是否说明 sunset 不存在？
   反而相反。remap 继续存在，恰恰说明 sunset 尚未完成。
5. 如果旧 plugin version 还在磁盘上，是否说明它仍算 current world？
   不一定。visibility world 可能已经先把它切掉。
6. 如果旧 world 先退出 visibility，是否就代表全部问题已经结束？
   也不代表。storage、recovery、continuity 可能仍保留历史残留。
7. 如果 stronger-request cleanup 线今天还没有显式 sunset 代码，是不是说明这层暂时不值得写？
   恰恰相反。越缺这层正式 cutoff grammar，越容易把兼容世界误叫成 current truth，或把 current truth 过早宣布结束。
8. 如果我坚持把 sunset 当成 migration 的附属物，会造成什么后果？
   会让系统只会说“已经在迁”，却永远说不清“旧 stronger-request 世界从什么时候起不再被允许继续代表现在”。

这一串反问最后逼出一句更稳的判断：

`sunset 的关键，不在系统能不能让旧世界先活一段时间，而在系统能不能正式决定这段共存从什么时候起不再被承认为当前世界。`

## 8. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 migration governance，就已经足够成熟。`

而是：

`Claude Code 在 migration version、migration timestamp、provider-specific retirement date、runtime warning、session visibility cutoff 与 storage grace window 上已经明确展示了 sunset governance 的存在；因此 artifact-family cleanup stronger-request cleanup-migration-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-sunset-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“怎么把旧世界带着走”，而是“谁来宣布这段路何时结束”。`
