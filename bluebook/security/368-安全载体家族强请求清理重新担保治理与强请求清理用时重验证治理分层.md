# 安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层：为什么artifact-family cleanup stronger-request cleanup-reassurance-governor signer不能越级冒充artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer

## 1. 为什么在 `367` 之后还必须继续写 `368`

`367-安全载体家族强请求清理重新投影治理与强请求清理重新担保治理分层`
已经回答了：

`stronger-request cleanup 线即便已经把 current truth 重新讲给不同 surface，也还要单独回答这些重讲里哪些现在配承载多强的正向 reassurance。`

但如果继续往下追问，
还会碰到另一层同样容易被偷写成
“既然某个 surface 已经给出了足够正向的 reassurance，真正 consumer 在实际使用那一刻当然可以直接依赖”
的错觉：

`只要 stronger-request cleanup reassurance governor 已经允许某个 surface 给出足够正向的 reassurance，真正的 consumer 在实际使用这一刻就自动拥有了继续依赖这份 truth 的资格，不再需要 fresh re-check。`

Claude Code 当前源码同样不能支持这种更强说法。

因为继续看 MCP 线上已经成熟写出的用时重验证正对照，
会发现 repo 其实明确拆开了五件事：

1. consumer-edge hard gate
2. fresh reconnect primitive
3. batch list live-use discipline
4. execution-context filter
5. runtime revocation on contradiction

最硬的证据至少有五组：

1. `ReadMcpResourceTool.ts:80-97` 的 `client.type !== 'connected'` hard gate 与 `ensureConnectedClient(client)`
2. `ListMcpResourcesTool.ts:79-89` 关于 `ensureConnectedClient` 的 fresh reconnect 注释与调用
3. `client.ts:1688-1701` 的 `ensureConnectedClient()`
4. `toolExecution.ts:308-336` 的 `getMcpServerType()` / `getMcpServerBaseUrlFromToolName()`
5. `toolExecution.ts:1599-1624` 的 `McpAuthError -> needs-auth` demotion

会发现 repo 已经清楚展示出：

1. `stronger-request cleanup reassurance governance`
   负责决定不同 surface 配说多强的正向话术
2. `stronger-request cleanup use-time revalidation governance`
   负责决定真正 consumer 在实际依赖之前，
   还必须重新拿到什么 fresh proof，
   才能把这份 reassurance 转成实际可用资格

也就是说：

`artifact-family cleanup stronger-request cleanup-reassurance-governor signer`

和

`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`

仍然不是一回事。

前者最多能说：

`当前有某个 surface 配给出多强的正向 reassurance。`

后者才配说：

`真正的 consumer 在实际调用这一刻，已经重新完成 fresh connection check / current capability check / current authorization check，可以把这份 reassurance 转成可依赖的 current-use truth。`

所以 `367` 之后必须继续补的一层就是：

`安全载体家族强请求清理重新担保治理与强请求清理用时重验证治理分层`

也就是：

`stronger-request cleanup reassurance governor 决定正向话术强度；stronger-request cleanup use-time-revalidation governor 才决定真正使用时是否已经 fresh enough to rely。`

## 2. 先做三条谨慎声明

第一条：

`Claude Code 当前源码里并没有一个字面存在的类型叫 artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer。`

这里的
`artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer`
仍是研究命名。
它不是在声称 cleanup 线已经有一个未公开的 live gate manager，
而是在说：

1. repo 已经明确把 positive surface reassurance 与 real consumer use path 拆开
2. repo 已经明确要求 consumer 在真正依赖前重新检查 `connected` / current truth
3. repo 已经明确允许 runtime 把旧的 `connected` 立即降回 `needs-auth`

第二条：

这里的 `use-time revalidation`

不是更早那条反漂移验证链里的：

`anti-drift verification`

当前更稳的说法只能是：

`反漂移验证回答未来若再分叉，谁会主动发现；用时重验证回答真正 consumer 在这次依赖发生的瞬间，谁会要求重新验真。`

第三条：

这里也不能过度声称：

`use-time revalidation governance`

已经等于：

`step-up reauthorization governance`

当前更稳的说法只能是：

`use-time revalidation 先回答“现在这份 reassurance 还配不配被当真”；step-up reauthorization 才回答“即便 current-use truth 成立，它对更强请求是否仍不够强”。`

## 3. 最短结论

Claude Code 当前源码至少给出了五类

`stronger-request cleanup-reassurance-governor signer 仍不等于 stronger-request cleanup-use-time-revalidation-governor signer`

证据：

1. `ReadMcpResourceTool` 在真正读取资源前，既要求 `client.type === 'connected'`，又继续调用 `ensureConnectedClient(client)`；这说明 positive reassurance 不自动等于 read path can rely now
2. `ListMcpResourcesTool` 明确把 `ensureConnectedClient` 注释成 healthy 时 no-op、onclose 后 fresh reconnect；这说明 list path 也拒绝盲信旧 reassurance
3. `ensureConnectedClient()` 本身会 fresh `connectToServer()`，若结果不再是 `connected` 就直接抛出 `MCP server "<name>" is not connected`；这说明真正依赖前仍要重新验真
4. `toolExecution.ts` 的 `getMcpServerType()` / `getMcpServerBaseUrlFromToolName()` 只有在 current connection 仍是 `connected` 时才继续派生 execution context；这说明连辅助执行上下文也不盲信旧 reassurance
5. `toolExecution.ts` 在 `McpAuthError` 上会把原本 `connected` 的 client 即时降成 `needs-auth`；这说明 stale reassurance 会在真实使用的第一时间被撤销

因此这一章的最短结论是：

`stronger-request cleanup reassurance governor 最多能说当前有某个 surface 配给出多强的正向 reassurance；stronger-request cleanup use-time-revalidation governor 才能说真正 consumer 在使用瞬间是否已经重新拿到 fresh enough 的 current-use proof。`

再压成一句：

`被正向担保，不等于已经通过 live-use 重验证。`

再压成更哲学一点的话：

`制度知道你现在能放心多少，不等于制度已经知道你这一刻真的还能去用。`

## 4. 第一性原理：reassurance 回答“能放心到什么程度”，use-time revalidation 回答“真正依赖这一刻还能不能把刚才那句放心话继续当真”

从第一性原理看，
强请求清理重新担保治理与强请求清理用时重验证治理处理的是两个不同主权问题。

`stronger-request cleanup reassurance governor`
回答的是：

1. 哪个 surface 现在配说正向话术
2. 这句正向话术允许多强
3. 是否必须保留 caveat
4. 是否宁可沉默也不发布正向信息
5. 当前 surface 的 reassurance ceiling 是什么

`stronger-request cleanup use-time-revalidation governor`
回答的则是：

1. 真正 consumer 在调用前是否还要 fresh reconnect
2. 当前 `connected` truth 是否要在 use-time 重新检查
3. capability / resources / transport 是否仍然 current
4. 如果刚才的 reassurance 已经过时，谁会在使用瞬间把它撤销
5. 哪些 positive surface 只是 local confidence，哪些能够通过 live gate 转成 actual reliance

如果把这两层压成一句：

`既然刚才已经说能用了`

系统就会制造五类危险幻觉：

1. `reassurance-means-usable-now`
2. `dialog-success-means-resource-readable`
3. `connected-copy-means-fresh-proof`
4. `positive-surface-speaks-for-consumer`
5. `reassurance-once-means-revalidation-forever`

所以从第一性原理看：

`reassurance governance` 管的是正向信号强度；
`use-time revalidation governance` 管的是实际依赖发生前，旧 reassurance 是否仍配被当真。

用苏格拉底式反问再压一次：

1. 如果 `MCPReconnect` 已经说 `Successfully reconnected`，为什么 `ReadMcpResourceTool` 还要再次 `ensureConnectedClient(client)`？
   因为 operation-local reassurance 不拥有替真实 read path 免除 fresh proof 的资格。
2. 如果 `/mcp` 已经显示 `✓ Connected`，为什么 `toolExecution` 仍会在 `McpAuthError` 上立刻把它降回 `needs-auth`？
   因为 health reassurance 不是 immutable use authorization。
3. 如果 `McpAuthTool` 刚说 `should now be available`，为什么 list / read tool 仍会在真正调用时拒绝 non-connected client？
   因为 positive wording 只配描述 current expectation，不配越级免除 consumer-side live gate。

## 5. `ReadMcpResourceTool` 与 `ListMcpResourcesTool` 先证明：真实 consumer path 不信任刚才那句 reassurance，而信任此刻的 fresh truth

`ReadMcpResourceTool.ts:80-97`
很硬。

这里的逻辑是：

1. 先找到目标 server
2. 如果 `client.type !== 'connected'`，直接报错
3. 即便已经是 `connected`，仍继续 `ensureConnectedClient(client)`
4. 然后才真正发 `resources/read`

这说明 resource read consumer 明确拒绝把：

`surface 上一个正向 reassurance`

偷写成：

`actual read authorization right now`

`ListMcpResourcesTool.ts:79-89`
更值钱。

它不仅也调用 `ensureConnectedClient(client)`，
还在注释里写得非常直白：

1. healthy 时 `ensureConnectedClient` 是 no-op
2. after `onclose` it returns a fresh connection so the re-fetch succeeds

这条注释把 use-time revalidation 的本体直接说出来了：

`consumer surface 不是拿旧 reassurance 直接去用，而是在真正依赖前确认连接仍然 fresh`

所以这两条 tool path 一起说明：

`reassurance can open expectation; it cannot waive live-use revalidation`

## 6. `ensureConnectedClient()` 再证明：真正的 current-use proof 不是沿用旧 `connected`，而是 fresh connect 或失败

`client.ts:1688-1701`
特别值钱。

这里 `ensureConnectedClient()` 做的事情非常干净：

1. SDK client 直接返回
2. 非 SDK client 重新 `connectToServer(client.name, client.config)`
3. 若 fresh result 不再是 `connected`，直接抛出
   `MCP server "<name>" is not connected`
4. 只有 fresh result 仍然成立时，才返回 connected client

这条证据很硬。

它说明 repo 真正承认的 current-use truth 不是：

`它之前看起来是 connected`

而是：

`它在真正使用这一刻重新被 fresh enough 地证明为 connected`

这条 primitive 把 use-time revalidation 从抽象哲学直接落成了具体工程语法。

## 7. `toolExecution` 再证明：执行平面连辅助 metadata 都要 obey current connected truth

`toolExecution.ts:308-336`
先给出一组很硬的正例。

这里：

1. `getMcpServerType()` 只有在 `serverConnection?.type === 'connected'` 时才返回 transport type
2. `getMcpServerBaseUrlFromToolName()` 只有在 `serverConnection?.type === 'connected'` 时才返回 base URL

这说明 repo 连“给一次 tool execution 补什么 transport / base-url 背景”这类辅助信息，
都不愿意沿用 stale reassurance。

换句话说，
execution plane 不是只在最核心的 request path 严格，
它在外围 metadata 上也坚持：

`connected now or nothing`

这比“只让 tool body 严格、其余上下文宽松”要先进得多。

因为一旦 helper context 允许 stale truth 残留，
系统就会在真正调用前就被污染。

## 8. `McpAuthError -> needs-auth` 再证明：live contradiction 在 repo 里拥有撤销旧 reassurance 的更高主权

`toolExecution.ts:1599-1624`
更关键。

一旦真实 tool use 遇到 `McpAuthError`，
repo 会立刻：

1. 找到对应 server
2. 只在原状态仍是 `connected` 时执行改写
3. 把 client 降成 `needs-auth`

这里最值钱的不是它会报错，
而是它会主动撤销旧的 connected reassurance，
把 live contradiction 写回 current state。

这说明 use-time revalidation 不是：

`验证失败了就算了，旧 reassurance 还能挂着`

而是：

`验证失败了，旧 reassurance 必须让位于当前矛盾事实`

这是一条非常成熟的安全设计原则：

`运行时反证比既有正向话术更有主权`

## 9. 更深一层的技术先进性：Claude Code 把“让你放心多少”和“你这一刻是否真的还配去用”继续拆开

这组源码给出的技术启示至少有五条：

1. 真正 consumer edge 的 fresh proof 永远不该被 earlier reassurance 冒充
2. current-use truth 最好收束成少数 canonical primitives，例如 reconnect-or-throw
3. helper metadata 也应 obey current truth，而不只是让核心操作严格
4. list / browse 这类较弱 consumer path 同样需要 freshness discipline，不能因为“看起来风险小”就放宽
5. runtime contradiction 应该拥有正式的 revocation 主权，而不是只留下日志或局部异常

用苏格拉底式反问压缩，
可以得到五个自检问题：

1. 如果 earlier reassurance 已经足够，为什么 `ReadMcpResourceTool` 还要继续 `ensureConnectedClient()`？
2. 如果 browse path 不需要 fresh proof，为什么 `ListMcpResourcesTool` 明写 `onclose` 后要拿 fresh connection？
3. 如果旧 `connected` label 已经等于 current-use truth，为什么 repo 还需要 reconnect-or-throw primitive？
4. 如果 helper context 可以复用旧结论，为什么 disconnected path 的 transport / base-url 直接变成 `undefined`？
5. 如果 runtime contradiction 只是局部异常，为什么 `McpAuthError` 会直接把 client 降成 `needs-auth`？

## 10. 苏格拉底式自反诘问：我是不是又把“刚才那句放心话”误认成了“这一刻真实 consumer 已经获准依赖”

如果对这组代码做更严格的自我审查，
至少要追问六句：

1. 如果 reassurance grammar 已经足够强，为什么还要再拆 use-time revalidation？
   因为会被正向担保，不等于使用瞬间仍然 fresh enough to rely。
2. 如果某个 surface 已经说 `connected` 或 `available`，是不是就说明资源读取、资源列举和实际执行都不必再重查？
   不是。真正 consumer edge 仍要 fresh re-check。
3. 如果 health glyph 还在绿色，为什么 runtime 一出 auth error 就要立刻降级？
   因为 probe reassurance 不能压过 live contradiction。
4. 如果 list / read 只是较弱 consumer path，是不是可以放宽 fresh proof 纪律？
   不能。弱消费面同样会把 stale reassurance 转成错误依赖。
5. 如果 helper metadata 只是附属信息，是不是没必要遵守 connected-only filter？
   不能。污染上下文本身就会污染后续真实使用。
6. 如果 cleanup 线现在还没有显式 use-time revalidation 代码，是不是说明这层还不值得写？
   恰恰相反。越是缺这层明确 grammar，越容易把“你现在可以放心一点”偷写成“你这一刻已经真的还能去用”。`

这一串反问最终逼出一句更稳的判断：

`use-time revalidation 的关键，不在系统之前给了你多强的 reassurance，而在系统能不能正式决定你这一刻是否仍然 fresh enough to rely。`

## 11. 一条硬结论

这组源码真正说明的不是：

`stronger-request cleanup 只要补出 reassurance governance，就已经足够成熟。`

而是：

`Claude Code 在 consumer-edge hard gate、fresh reconnect primitive、batch list live-use discipline、execution-context filter 与 runtime revocation on contradiction 上已经明确展示了 use-time revalidation governance 的存在；因此 artifact-family cleanup stronger-request cleanup-reassurance-governor signer 仍不能越级冒充 artifact-family cleanup stronger-request cleanup-use-time-revalidation-governor signer。`

再压成一句：

`cleanup 线真正缺的不是“它刚才让你放心多少”，而是“你这一刻到底还能不能真的去用”。`
