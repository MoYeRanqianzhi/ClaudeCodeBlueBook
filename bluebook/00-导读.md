# 导读

## 1. 研究对象

> Evidence mode
> - `mirror present`：当前 worktree 含 `claude-code-source-code/` 时，优先按本地镜像核对版本锚点、源码锚点与运行时对象。
> - `public-evidence only`：当前 worktree 不含该目录时，本页只承认蓝皮书正文、`docs/` 归档锚点与已公开材料，不把缺席镜像写成当前在场事实。

本蓝皮书研究的对象设计上对应 `claude-code-source-code/` 镜像；当本地镜像在场时，它通常以 `package.json`、源码树与 README 提供版本锚点与运行时证据。若当前 worktree 缺席该目录，先回 [../docs/README.md](../docs/README.md) 与 `docs/development/02-证据索引` 查看已归档锚点，不在入口层假装已经完成本地核证。

研究方法、版本锚点与证据索引单独归档；本页只保留 preface 与 evidence mode，不再承担第二套导航与研究工作台职责。
更直白地说，`00` 不是 `00-09` 线性教程的第一站，而是 preface / locator；若你已经定清工作对象，正式入口只看根 `README`。

## 2. 为什么值得读

如果只先带一句判断进入蓝皮书，也只记这一句：

- 这里研究的不是功能清单，而是世界准入、扩张定价与当前真相保护三条制度设计如何在同一 runtime 里共同成立。

更硬的 canonical formula、artifact gate 与 first reject signal 统一回 [README](README.md)；本页不代签入口升级。

## 3. 为什么要先画结构图

Claude Code 的强，不是单点 prompt 技巧，而是很多工程机制叠加出来的:

- 启动阶段极度在意 fast path。
- 核心 agent loop 足够小，但外围 harness 很厚。
- 工具、技能、插件、MCP 不是孤立系统，而是共享一套运行时装配逻辑。
- 多 Agent、远程、上下文压缩、权限系统都不是外挂，而是主路径的一部分。

如果先不把结构图画清，后面写“使用指南”和“设计哲学”会不可避免地漂浮。

## 4. 阅读后的第一判断

Claude Code 的本质不是“会写代码的聊天框”，而是一个把 conversation、tooling、permission、compaction、subagent、remote transport 与 writeback 一起纳入同一 runtime 的系统。
如果再往上压缩一层，这套 runtime 最难被抄走的，不是功能数量，而是三条制度判断：世界怎样进入模型、扩张怎样进入当前世界、过去与失控复杂度怎样不得写坏当前世界。

正式入口看 [README](README.md)，跨目录缺口反查看 [navigation/README.md](navigation/README.md)，方法附录与证据归档另见 [../docs/README.md](../docs/README.md)。
