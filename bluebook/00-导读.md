# 导读

## 1. 研究对象

本蓝皮书研究的是仓库中的 `claude-code-source-code/`，其 `package.json` 标明源码对应 `@anthropic-ai/claude-code-source` `v2.1.88`，并注明这是面向研究的解包源码，而不是可直接生产构建的原始 monorepo。

证据:

- `claude-code-source-code/package.json:2-8`
- `claude-code-source-code/README.md:1-8`

## 2. 研究方法

当前采用“五层拆解法”：

1. 入口层: CLI 如何分流不同运行模式。
2. 核心循环层: 用户输入如何变成 system prompt、model stream、tool use、transcript。
3. 能力装配层: tools / commands / skills / plugins / MCP 如何在运行时拼装。
4. 扩展层: multi-agent、remote session、remote control、feature gate 如何影响最终产品形态。
5. 接口层: CLI、slash command、SDK control protocol、MCP transport、remote IO 如何把 runtime 暴露给外部。

## 2.1 先带着三句判断再继续读

如果只带一个最短判断进入整本蓝皮书，请先记住三句：

1. Prompt 线
   - Claude Code 真正保护的是：主权链、历史链与继续链对同一个世界保持诚实。
2. 治理线
   - Claude Code 真正保护的是：一切扩张按同一价格秩序合法进入当前世界。
3. 源码质量线
   - Claude Code 真正保护的是：复杂度中心合法、边界可证、下一次重构仍有路。

后面的架构、接口、哲学、指南与案例，其实都只是这三句判断的展开。

## 3. 为什么要先画结构图

Claude Code 的强，不是单点 prompt 技巧，而是很多工程机制叠加出来的:

- 启动阶段极度在意 fast path。
- 核心 agent loop 足够小，但外围 harness 很厚。
- 工具、技能、插件、MCP 不是孤立系统，而是共享一套运行时装配逻辑。
- 多 Agent、远程、上下文压缩、权限系统都不是外挂，而是主路径的一部分。

如果先不把结构图画清，后面写“使用指南”和“设计哲学”会不可避免地漂浮。

这说明 Claude Code 不是“一个 prompt + 一个 shell tool”，而是一个已经产品化的 agent runtime。

## 5. 源码阅读建议

最推荐的顺序不是从最大文件 `query.ts` 直接硬读，而是:

1. `src/entrypoints/cli.tsx`
2. `src/main.tsx`
3. `src/commands.ts`

7. 再进入 `skills/`、`services/mcp/`、`tools/AgentTool/`、`remote/`、`bridge/`

这个顺序更符合运行时真实链路。

## 6.1 为什么目录要继续分层

随着专题变多，蓝皮书必须持续区分：

1. 主线结论
2. 机制细节

更具体地说：

- `bluebook/` 根目录只保留主线章节

否则读者会被迫在一篇文档里同时切换“顺读、查表、回溯证据”三种阅读模式，成本会越来越高。

## 7. 阅读后的第一判断

Claude Code 的本质不是“会写代码的聊天框”，而是一个把以下对象统一纳入一个 runtime 的系统:

- conversation transcript
- tool registry
- permission context
- slash commands and skills
- dynamic MCP connections
- background tasks and subagents
- compaction and resume
- remote and bridge transports

后续章节都围绕这个统一 runtime 展开。

如果再往上压缩一层，这套 runtime 最难被抄走的，不是功能数量，而是三条制度判断：

1. 世界怎样进入模型
2. 扩张怎样进入当前世界
3. 过去与失控复杂度怎样不得写坏当前世界
