# 权限、沙箱、审批与安全边界

## Claude Code 的核心不是“能不能做”，而是“这次扩张有没有先拿到 pricing-right”

从 `src/utils/permissions/*`、`src/types/permissions.ts`、`src/commands/permissions/index.ts`、`src/tools/BashTool/bashPermissions.ts` 可以看到，权限系统贯穿整个运行时，而不是某个弹窗组件。

更稳的读法不是先盯 mode 或审批弹窗，而是先看四层合同：

1. 当前 visible set / mode 先裁掉什么。
2. rules、hooks 与 `typed ask` 如何决定这次动作能否进入当前世界。
3. sandbox / filesystem / network boundary 如何限制副作用世界。
4. host-facing truth 与 `rollback ceiling` 如何说明这次批准到底覆盖了什么。

审批框因此只配当 `typed ask` 的一个 consumer，不是治理本体；真正先定义 verdict family 的，仍是 `pricing-right -> truth-surface attestation -> asset-rollback ABI`。

## 计划模式不是产品姿态，而是权限模式切换

`EnterPlanModeTool` 做的事情非常直接：

- 切换权限模式到 `plan`。
- 要求只读探索。
- 要求在真正改文件前先提交计划并退出计划模式。

这不是“提示工程建议”，而是运行时治理动作。

对用户的意义：

- 当任务复杂、风险高、需要先摸清结构时，应主动进入计划模式。
- 计划模式的价值不在于“多想一会儿”，而在于用运行时约束先禁掉实现冲动。

## 沙箱与审批不是一回事

从 Bash/PowerShell 工具实现能看出，系统同时关心：

- 这个命令本身是否危险。
- 这个路径是否越界。
- 是否涉及网络。
- 是否是只读。
- 是否需要用户确认。
- 当前 permission mode 是否允许自动通过。

也就是说，“在沙箱里”并不等于“自动放行”，反过来也一样。

官方 `Security` 页把这件事写得更直白：

1. sandbox 处理的是 filesystem / network isolation。
2. approval 处理的是这次动作是否被用户、规则或宿主批准。
3. write access 默认只限启动目录及其子目录。
4. network requests 默认仍需审批。
5. WebFetch 还会使用 separate context window，避免把可疑网页内容直接注入主对话。

所以更稳的用户心智模型不是：

- 开了沙箱 = 自动安全

而是：

- 沙箱在缩边界，审批在签动作，separate context 在隔离 payload；三者是正交控制，不是一把总开关。

## allow/deny 规则的作用比你以为的大

`src/tools.ts` 中 `filterToolsByDenyRules()` 会在模型看到工具前就先裁掉一部分工具。

这很关键，因为它说明权限并不是只在调用时判断。系统会：

- 先改写模型可见的工具宇宙。
- 再在调用时做更细粒度判断。

用户含义：

- `/permissions` 不是事后补救界面。
- 你在这里调整的规则，会直接影响模型的规划空间。

官方 `Configure permissions` 页还把顺序写得很清楚：

1. rules 按 `deny -> ask -> allow` 评估；
2. `PreToolUse` hook 可以在 prompt 前置筛选；
3. 但 hook 的 `"allow"` 并不会跳过 deny / ask 规则；
4. unmatched commands 默认回 manual approval。

所以更稳的理解不是：

- “允许某个命令”只是少一次弹窗

而是：

- 你在修改模型可见世界、裁决顺序和默认失败语义；这既会改安全边界，也会改 token 去向。

## 远程策略与托管设置是更高一层控制面

`src/services/remoteManagedSettings/*`、`src/services/analytics/growthbook.ts`、`src/utils/permissions/bypassPermissionsKillswitch.ts` 说明，发布构建还存在更高层的治理面：

- 托管设置。
- feature gate。
- killswitch。
- 远程策略。

因此功能“是否出现”不只由本地版本决定，还取决于：

- 账户类型。
- 组织策略。
- 运行环境。
- 远程开关。

## 用户实践

### 低风险任务

可以直接在普通模式推进，让工具审批与规则去兜底。

### 高风险任务

优先进入计划模式，必要时切到 worktree，再开始实施。

### 频繁重复的安全例外

不要反复临时授权，应通过 `/permissions` 或配置面收敛成明确规则。

### 组织环境中

要预期某些能力会因策略而消失、降级或强制弹出确认。
