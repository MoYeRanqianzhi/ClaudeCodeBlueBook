# 任务对象、输出回流、通知与恢复

## 第一性原理：Claude Code 的运行态任务不是 Todo，而是可被治理的执行对象

这里讨论的不是待办清单，而是后台 shell、子代理、远程代理这些真正会持续运行、产出结果、等待回收的对象。

主链可以压成一句话：

`启动任务 -> 写入 AppState.tasks -> 绑定 outputFile -> 各任务类型自管终态 -> 生成 <task-notification> -> 回流给模型 / SDK / UI`

所以任务系统优化的不是“列出还有什么没做”，而是“让异步执行仍然可见、可中断、可恢复、可回流”。

## `TaskStateBase` 是统一骨架

稳定的运行态任务对象至少共享这些字段：

- `status`
- `startTime`
- `outputFile`
- `outputOffset`
- `notified`

这意味着 Claude Code 把任务首先当成“有生命周期和输出文件的对象”，然后才按类型分化。

## 三类核心稳定任务

### `local_bash`

本地 shell 长任务。

### `local_agent`

本地子代理任务。

### `remote_agent`

远程执行任务。

如果一页 userbook 要写默认并行工作面，应首先围绕这三类展开，而不是先写更条件化的 workflow、monitor、ultraplan。

## 输出文件不是附属物，而是回流协议的一部分

`getTaskOutputPath(taskId)` 提供了统一的 output file 语义。

但不同任务的 output file 含义并不一样：

- `local_agent` 的 output file 常常是指向 subagent transcript 的 symlink。
- `remote_agent` 的 output file 是本地持续追加的真实文件。

这就是为什么“直接读 output file”越来越成为主路径，而旧的 `TaskOutputTool` 被标成兼容层。

## `TaskOutputTool` 仍然有用，但已经不该当主入口

源码已经明确提示：

- 更推荐直接 `Read output_file`
- `TaskOutputTool` 更像兼容接口
- 它还能做 block / non-blocking 查询
- 成功取回后还会把任务标成 `notified=true`

所以更稳的用户心智模型是：

- output file 是正式结果载体
- `TaskOutputTool` 是兼容查询器

## 真正的结果回流不是 toast，而是 `<task-notification>`

最重要的一条运行时事实是：

- 任务完成后不是只弹个 UI 提示
- 而是通过 `enqueuePendingNotification()` 把 `<task-notification>` 注入消息队列
- 之后再由 `query.ts` / `print.ts` 送回模型或 SDK

这解释了为什么后台任务能在主线程下一轮里“自动被知道”。

换句话说，Claude Code 真正维护的是结果回流协议，而不是普通前端提示。

## 停止任务也是统一协议，不是各写各的按钮

`TaskStopTool` 只是统一入口。

真正的 stop 逻辑在 `tasks/stopTask.ts`：

- 先查任务对象
- 检查是否处于 running
- 再分发到对应 task implementation 的 `kill()`

这让停止动作也纳入同一治理框架，而不是由每个任务面板各自处理。

## 恢复语义是不对称的

### 本地代理

更偏向按需恢复：

- transcript + metadata 保持现场
- 需要时再通过相关入口恢复后台子代理

### 远程代理

更偏向自动恢复：

- `remote-agents/*.meta.json` 持久化身份
- 恢复时再向远端拉状态并继续轮询

这两类恢复语义不能写成同一种“后台任务自动接回”。

## 稳定面与条件面

### 稳定工作面

- `TaskStateBase`
- `local_bash`
- `local_agent`
- `remote_agent`
- output file 路径语义
- `<task-notification>` 回流链

### 条件或灰度面

- `TaskOutputTool` 作为兼容接口
- 远程隔离、远程 review、ultraplan、workflow、monitor
- 只在特定入口自动恢复的远程任务链

## 对用户的使用结论

### 结论 1

并行任务真正要盯的是任务对象、output file 和通知回流，不是只盯 UI 面板。

### 结论 2

如果你需要最稳地取后台结果，优先读 output file；不要把 `TaskOutputTool` 当成唯一主入口。

### 结论 3

远程任务与本地子代理都属于任务系统，但它们的输出语义和恢复方式并不对称。
