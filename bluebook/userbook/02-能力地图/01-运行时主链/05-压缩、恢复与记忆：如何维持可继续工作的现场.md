# 压缩、恢复与记忆：如何维持可继续工作的现场

这页关注的是“连续性工具如何协同工作”，默认你已经区分：

- 实时状态层
- transcript 恢复账本
- compaction / memory / resume 这些连续性机制

## 第一性原理：Claude Code 维护的不是“完整历史”，而是“可继续工作的现场”

`services/compact/*`、`services/SessionMemory/*`、`commands/resume/*` 连起来看，Claude Code 对连续性的定义非常工程化：

- 不追求无限保留所有原始消息。
- 追求在有限预算内保住继续行动所需的工作态。

## Compact 是一个子系统，不是一个按钮

`compact.ts`、`microCompact.ts`、`postCompactCleanup.ts` 说明 compaction 至少包含：

- 正式总结压缩。
- microcompact / cached microcompact。
- compaction 后缓存和状态的统一清理。

它关心的不只是 token，还关心：

- prompt cache 稳定性。
- 附件与技能再注入。
- 主线程与子代理之间的状态污染。

官方 `Explore the context window` 与 `Manage costs effectively` 两页把这条线又钉死了一层：

1. `/compact` 不是把一切清掉，而是把 conversation 换成 structured summary。
2. 多数 startup content 会自动 reload。
3. skill listing 是被明确点名的例外。
4. custom compact instructions 本身也是正式控制面，而不是一次性提示技巧。

所以 `/compact` 更稳的心智模型其实是：

- 先把高噪音历史压成 summary，再把仍属于 constitution 的 startup content 按同一 load law 重新装回去。

## Session Memory 是“长期工作态账本”，不是普通摘要

`sessionMemory.ts` 有几个关键特征：

- 独立 feature gate。
- 按阈值触发，不是每轮执行。
- 用 forked agent 维护。
- 只能精确编辑那一个 memory file。
- 明确服务于 compaction 后连续性。

推论：Session Memory 在设计上更像“受约束的状态写回系统”，而不是自由发挥的总结器。

## Resume 恢复的是工作现场，不只是会话标题

`resume.tsx` 对“同目录 / 同 repo worktree / 跨目录”的区别处理，说明恢复最重要的不是文本检索，而是当前工作面与旧会话工作面的兼容性。

推论：Claude Code 把会话当成工作对象，而不是消息对象。

这里还要再多记一句边界：

- Agent SDK 的 file checkpointing 只回滚 `Write / Edit / NotebookEdit` 造成的文件变更，不回滚 Bash 副作用，也不回滚 conversation 本身。

这意味着用户不该把：

- `/resume`
- `/rewind`
- file checkpointing

想成同一种恢复。

更稳的区分是：

1. `resume`
   - 回到旧工作现场
2. `compact`
   - 压短当前工作现场
3. file checkpointing / rewind files
   - 回滚特定文件写入

只有把这三者分开，连续性工具才不会被误读成“万能时光倒流”。

## 为什么连续性机制必须分层

因为它们解决不同时间尺度的问题：

- `Compact`：当前长任务还要继续。
- `Resume`：用户离开后回来继续。
- `Memory`：长期稳定约束跨任务保留。
- `Export`：把当前内部状态翻译给会话外部。

## 对用户的使用结论

### 结论 1

`/compact` 的正确心智模型是“整理工作态后继续”，不是“清空聊天”。

### 结论 2

`/memory` 的正确心智模型是“把长期约束抽出短上下文”，不是“多一个备注文件”。

### 结论 3

`/resume` 的本质是恢复工作现场，所以目录不匹配、worktree 变化等问题会影响它能否平滑继续。
