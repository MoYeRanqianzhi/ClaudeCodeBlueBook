# 压缩、恢复与记忆：如何维持可继续工作的现场

这页关注的是“连续性工具如何协同工作”，默认你已经区分：

- 实时状态层
- transcript 恢复账本
- compaction / memory / resume 这些连续性机制

## 第一性原理：Claude Code 维护的不是“完整历史”，而是“可继续工作的现场”

`services/compact/*`、`services/SessionMemory/*`、`commands/resume/*` 连起来看，Claude Code 对连续性的定义非常工程化：

- 不追求无限保留所有原始消息。
- 追求在有限预算内保住继续行动所需的工作态。

如果把这一层继续压成最短公式，也只该剩：

1. `lawful forgetting -> stable bytes externalize -> reload law -> continuation qualification`
2. `summary != witness`
3. `memory != transient state`
4. `resume != unconditional restore`

更硬一点说，`Compact / Memory / Resume` 共同消费的是 continuity law，但它们都不是 continuation qualification signer；时间本身就是扩张面，继续前必须重新证明当前工作、next-step guard、required assets 与 stable witness 仍配续租。

更硬一点说，这页真正值钱的不是再列几条连续性命令，而是回答：

1. 哪些内容还配留在当前主账里。
2. 哪些内容应被外置成 stable bytes，再按同一 load law 带回。
3. 哪些对象只是帮助恢复连续性，不单独证明可以继续。

## Compact 是一个子系统，不是一个按钮

`compact.ts`、`microCompact.ts`、`postCompactCleanup.ts` 说明 compaction 至少包含：

- 正式总结压缩。
- microcompact / cached microcompact。
- compaction 后缓存和状态的统一清理。

它关心的不只是 token，还关心：

- prompt cache 稳定性。
- 附件与技能再注入。
- 主线程与子代理之间的状态污染。

官方 `Explore the context window` 与 `Manage costs effectively` 两页把这条线又钉死了一层：

1. `/compact` 不是把一切清掉，而是把 conversation 换成 structured summary。
2. 多数 startup content 会自动 reload。
3. skill listing 是被明确点名的例外。
4. custom compact instructions 本身也是正式控制面，而不是一次性提示技巧。

所以 `/compact` 更稳的心智模型其实是：

- 先把高噪音历史压成 summary，再把仍属于 constitution 的 startup content 按同一 load law 重新装回去。

真正省 token 的也不是“更短”，而是：

- 把不再配占用当前世界的高波动 bytes 逐出主账。
- 把仍会反复被需要的稳定约束外置成可 reload 资产。
- 把继续资格仍然需要的 witness 留在场内。

## Session Memory 是“长期工作态账本”，不是普通摘要

`sessionMemory.ts` 有几个关键特征：

- 独立 feature gate。
- 按阈值触发，不是每轮执行。
- 用 forked agent 维护。
- 只能精确编辑那一个 memory file。
- 明确服务于 compaction 后连续性。

推论：Session Memory 在设计上更像“受约束的状态写回系统”，而不是自由发挥的总结器。

它真正值钱的不是“多存一点内容”，而是：

1. 只在阈值满足时提取，避免每轮都重写长期层。
2. 只允许把长期稳定约束沉到 memory file，不鼓励把 transient authority、旧 blocker 或当前 working set 直接写死。
3. 明确服务于 compaction 后连续性，而不是抢当前主账的主语。

## Resume 恢复的是工作现场，不只是会话标题

`resume.tsx` 对“同目录 / 同 repo worktree / 跨目录”的区别处理，说明恢复最重要的不是文本检索，而是当前工作面与旧会话工作面的兼容性。

推论：Claude Code 把会话当成工作对象，而不是消息对象。

这里还要再多记一句边界：

- Agent SDK 的 file checkpointing 只回滚 `Write / Edit / NotebookEdit` 造成的文件变更，不回滚 Bash 副作用，也不回滚 conversation 本身。

这意味着用户不该把：

- `/resume`
- `/rewind`
- file checkpointing

想成同一种恢复。

更稳的区分是：

1. `resume`
   - 回到旧工作现场
2. `compact`
   - 压短当前工作现场
3. file checkpointing / rewind files
   - 回滚特定文件写入

只有把这三者分开，连续性工具才不会被误读成“万能时光倒流”。

## continuity system 的第一预算器是 reload law

Compact、Resume、Memory 共同守的不是“怎么把更多历史塞下”，而是：

1. 这批 bytes 是否还值得继续留在主账。
2. 若不值得，它们能不能被降格成 stable bytes，在需要时按同一规则 reload。
3. 若既不值得留在主账、也不值得 reload，就该被 lawful forgetting 正式逐出。

所以安全设计与省 token 设计在这里其实是同一个问题：

- 不该继续留在当前世界的对象，既不该继续占 token，也不该继续保有恢复和写回资格。

## 为什么连续性机制必须分层

因为它们解决不同时间尺度的问题：

- `Compact`：当前长任务还要继续。
- `Resume`：用户离开后回来继续。
- `Memory`：长期稳定约束跨任务保留。
- `Export`：把当前内部状态翻译给会话外部。

## 进入这一层前的 first reject signal

1. 你还把 `/compact` 理解成“把聊天变短”，而不是 `lawful forgetting + reload law`。
2. 你还把 `summary`、`memory`、`resume`、checkpointing 混成同一种恢复，而不是分清 `继续工作 / 恢复工作对象 / 长期规则外置 / 文件级回滚`。
3. 你还把“故事还说得通”当成 continue qualification，而点不出 `current work / next-step guard / required assets / stable witness`。
4. 你还把 `resume` 当恢复便利，而没有先问这次继续是不是仍值得被编译、被定价、被安全写回。

## 对用户的使用结论

### 结论 1

`/compact` 的正确心智模型是“整理工作态后继续”，不是“清空聊天”。

### 结论 2

`/memory` 的正确心智模型是“把长期约束抽出短上下文”，不是“多一个备注文件”。

### 结论 3

`/resume` 的本质是恢复工作现场，所以目录不匹配、worktree 变化等问题会影响它能否平滑继续。

### 结论 4

`/compact`、`/memory`、`/resume` 真正共享的是同一条 continuity law，不是同一种摘要格式；它们都在服务 `stable bytes externalize + reload law + continuation qualification`。
