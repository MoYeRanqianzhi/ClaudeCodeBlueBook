# 查询循环、工具池与行动编排

这一页默认你已经知道命令空间不是静态列表，而是一条会持续装配的总线。

## 第一性原理：Claude Code 优化的不是“回答质量”，而是“工作推进质量”

`query.ts`、`QueryEngine.ts`、`tools.ts`、`services/tools/toolExecution.ts` 合起来构成 Claude Code 的行动底盘。核心不是一次回答，而是重复执行下面的循环：

1. 组装请求。
2. 让模型决定是否调用工具。
3. 执行工具。
4. 生成 `tool_result`。
5. 继续推进直到得到可交付结果。

所以 Claude Code 的主循环更接近一个工作推进器，而不是聊天输出器。

## 工具池是在模型看到之前就被治理过的

`tools.ts` 先收集所有潜在工具，再依次处理：

- feature gate。
- 平台与环境条件。
- deny 规则。
- REPL / simple mode 特例。
- MCP 工具合并。

这说明系统的第一个治理动作不是“拦截已发生的调用”，而是“先改变模型能够规划的工具宇宙”。

## 这比事后拦截更重要

如果一个工具压根不在当前工具池里，模型不会围绕它规划实施路径。

推论：Claude Code 的稳定性很大程度上来自对规划空间的预裁剪，而不仅仅来自弹窗审批。

## 查询循环为什么这么在意 `tool_result` 配对

`query.ts` 和 `StreamingToolExecutor.ts` 里多处处理 orphan tool result、synthetic tool_result、executor 切换问题，说明系统非常在意：

- 每个工具调用都要有协议上可追溯的结果。
- 并发和流式执行不能破坏 tool-use/tool-result 成对关系。
- 旧执行器不能把新轮次消息链弄乱。

这表明 Claude Code 在维护一种严格的行动语法，而不是把工具执行当成松散副作用。

## `toolExecution.ts` 是抽象意图到真实副作用的扼流点

这里统一处理：

- 查找工具。
- 兼容别名与旧转录。
- MCP 工具解析。
- 权限检查。
- 调用与失败。
- 标准化结果。

推论：Claude Code 真正的可演化性，不只来自工具多，而来自它有单一执行扼流点。

## ToolSearch 暗示的未来

工具数变多后，系统已经不再假设“所有工具都该直接暴露给模型”。ToolSearch 的存在意味着 Claude Code 正在往“可搜索工具面”演化。

这和 MCP、插件、工作流命令、外部资源面增长是同一方向。

## 对用户的使用结论

### 结论 1

看到模型“不会用某工具”，不一定是模型不聪明，也可能是当前工具池根本没把那条能力暴露给它。

### 结论 2

相同 prompt 在不同会话里出现不同规划路径，很多时候是工具池差异而不是模型随机。

### 结论 3

MCP 是外部执行面的增量拼接，不是内核工具的等价替代，因此它天然更依赖当前会话条件。
