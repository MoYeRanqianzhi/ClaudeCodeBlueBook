# 输入分流、命令分发与入口编译

## 第一性原理：Claude Code 先判定输入的“制度身份”，再决定是否进入模型

`handlePromptSubmit.ts`、`processUserInput.ts`、`processSlashCommand.tsx` 串起来看，Claude Code 的第一个核心动作不是回答问题，而是给输入分配制度身份：

- 普通提示词。
- 斜杠命令。
- 本地 UI 命令。
- prompt 命令/技能。
- bridge/远程来源输入。
- `isMeta` 的系统注入输入。

这意味着 Claude Code 的起点不是文本框，而是一个输入编译器。

## 为什么说它在“编译”而不是“解析”

因为它做的不是简单识别命令前缀，而是在决定：

- 这段内容是否应该进入模型。
- 如果进入模型，应以什么消息形态进入。
- 如果不进入模型，应由本地命令还是本地 JSX 控制面处理。
- 如果来自桥接环境，哪些命令必须继续封闭。
- 如果是系统生成结果，是否要作为用户不可见的 meta prompt 重新回流。

## `Command` 类型其实是运行时合同

`types/command.ts` 里的 `Command` 定义了很多表面看起来不像“命令”的字段：

- `immediate`
- `supportsNonInteractive`
- `isSensitive`
- `userInvocable`
- `disableModelInvocation`
- `hooks`
- `context`
- `agent`

这说明命令不是 UI 糖，而是带执行、治理、可见性、敏感性和上下文策略的正式对象。

## `local`、`local-jsx`、`prompt` 三分法的意义

### `local`

适合立刻修改本地会话状态，不必进入模型。

### `local-jsx`

适合“本地管理面板”类操作，把配置和治理交给本地 UI，而不是交给模型猜测。

### `prompt`

适合把一段正式工作流送进模型。它是受控 prompt 注入，而不是自由聊天。

推论：Claude Code 之所以能容纳很多复杂能力而不把所有事情都变成 prompt 工程，是因为它把本地控制面和模型控制面分开了。

## `bridgeOrigin` 和 `skipSlashCommands` 透露了什么

`processUserInput.ts` 里专门处理 bridge-safe slash command，说明远程/桥接前端并不天然拥有本地终端的全部入口权。

推论：Claude Code 把“不同来源的输入”也当成治理变量，而不是默认一视同仁。

## `isMeta` 是 Claude Code 很容易被忽视的高级能力

一部分系统生成结果会以 `isMeta` 的形式重新进入队列：

- 用户看不到。
- 模型能继续看见。
- 可避免把系统内部调度细节直接刷进 transcript。

这对后台子代理、计划后续动作和系统回流结果非常关键。

## 对用户最关键的使用结论

### 结论 1

如果目标是改运行时状态，优先用命令，而不是自然语言。

### 结论 2

如果目标是让 Claude Code 帮你推进工作，优先走普通提示词或 `prompt`/skill 路径，而不是手工拼接大量命令。

### 结论 3

同一个“看似一样的输入”在不同前端、不同模式、不同来源下，执行路径可能不同；这不是随机，而是制度分流。
