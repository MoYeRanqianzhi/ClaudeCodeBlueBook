# 专题深潜

这一层专门处理“用户真正会反复遇到的复杂工作”，避免把手册停留在功能罗列。

它和 `05-控制面深挖` 的区别是：

- 这里按“我要完成什么工作”来组织。
- `05-控制面深挖` 按“哪些控制面决定了工作质量与边界”来组织。

如果把这一层继续压成最短作用，也只剩一句：

- `04` 先回答“我要推进哪个工作对象”，再把你送去对应控制面，而不是先把你塞回功能表。

更硬一点说，`04` 在 `userbook/` 里的发言权也只该剩：

1. `工作对象翻译权`
   - 把已承认的对象链翻成“我要推进什么工作”。
2. `工作顺序提示权`
   - 把最小顺序翻成真实工作动作。
3. `无机制改判权`
   - 它不替控制面页改判哪条 truth 在说话，也不替正文发新的 canonical formula。

所以它不该被读成一排长文章列表，而应被读成：

1. 工作目标入口
2. 对应的最小排查顺序
3. 需要时再下钻到控制面深挖

这里也要先压住一个常见误读：`连续性与记忆` 不是第五类工作对象；它只是这四组工作对象在时间轴上的继续条件，失真时要么退回 Prompt `Continuation`，要么退回治理 `continuation pricing`，要么退回当前真相收口。

如果专题页不能先回答这三件事，它就还只是“更长的目录”，不是工作对象入口。

## 先按工作对象，不按功能名

进入这一层前，先问自己：

1. 我是在推进代码主线、维持连续性、扩张能力、并行隔离，还是在做评审/交付/远程接入。
2. 我现在最缺的是同一个工作对象、治理判断，还是当前工作现场边界。
3. 如果删掉命令名和 UI 面板，我还能不能说清这次要推进的是哪个工作对象。

如果还说不清这些问题，就先回 `userbook/00` 与 `01-主线使用`，不要在专题层靠目录名碰运气。

## 工作对象版三问

到了专题深潜，最值钱的不是再抄一遍抽象顺序，而是先把它们翻译成三句工作问法：

1. 这还是同一个工作对象吗
   - 先确认当前任务、附件、文件和 handoff 还在服务同一件事。
2. 这是继续推进，还是该先回 `05`
   - 若卡住的是继续、收口、恢复现场或升级给人这类控制面判断，就先退回 `05-控制面深挖`。
3. 这能不能外推成系统真相
   - 先过 `public artifact ceiling`；若 sole writer、writeback seam 与 freshness guard 还没锁定，就只配写 `current-truth surface candidate`。

如果某篇专题长文只会继续列入口、列功能，却没有把这三问翻译成对应的工作动作，那它就还停在“扩展版目录”，还不是专题入口。

更稳一点说，`04` 不只要和根 `userbook/README` 共享 nouns，也要和 `01 / 05` 共享同一组 first-answer order；否则专题层仍会把用户送回“按标题试错”。

## 进入专题前的 first reject signal

看到下面迹象时，应先退回主线使用或控制面页，而不是继续在专题里硬读：

1. 你还在靠 slash 命令名字决定路径，而不是先按工作对象决定路径。
2. 你还在靠 mode、usage 条、长摘要决定下一步，而不是先判当前该落在哪个 verdict。
3. 你还把恢复理解成“找旧聊天记录”，而不是恢复同一个工作对象。
4. 你还把“多 Agent / worktree / 远程”理解成功能开关，而不是边界与主权安排。

## 按工作对象分组，不按专题名平铺

如果继续把这一层压成最短工作路由，最稳的四组入口只该剩：

1. 把同一工作对象带进场并持续推进。
2. 给当前工作对象扩权、接外部能力或并行隔离。
3. 让当前工作对象跨宿主、跨时间、跨现场继续存在。
4. 让当前工作对象被评审、交付、运营、回退或重新开工。
