# 安全放大工作面

这个物理入口只处理一个高频误判：

- 都跟权限、计划、目录、隔离有关，
- 但 `/permissions`、`/plan`、EnterWorktree、`/add-dir` 其实不是四个互不相干的按钮。

如果这几层不先拆开，

最容易把下面这些动作重新混成同一种“治理功能”：

- 调整长期 allow / deny 规则
- 先探索再实施
- 把实现面隔离到新 worktree
- 在当前 session 里正式扩张工作面

如果你已经确认自己的问题就落在这条线上，不是来做第一跳分诊，而是想按顺序一次读通这整条使用线，再看 [../../00-阅读路径.md](../../00-%E9%98%85%E8%AF%BB%E8%B7%AF%E5%BE%84.md) 里的“路径 14：我想安全放大工作面，而不是一上来把行动半径开到最大”。

## 第一性原理

更稳的问法不是：

- “现在该不该让 Claude Code 改代码？”

而是：

- 当前行动半径该放多大，
- 当前工作区是不是正确实现面，
- 我需要的是隔离，还是扩张。

按这条线，更稳的分层是：

1. 权限层：`/permissions` 决定当前工具宇宙和长期例外规则。
2. 计划层：`/plan` / EnterPlanMode 决定当前是否先保持只读探索。
3. 隔离层：EnterWorktree 决定是否把实现搬到独立代码面。
4. 扩张层：`/add-dir` 决定是否在当前 session 里正式把更多目录纳入工作面。

## 稳定面、条件面与内部面

- 稳定可见：`/permissions`、`/plan`、EnterPlanMode / ExitPlanMode、EnterWorktree / ExitWorktree、`/add-dir`，以及 `/add-dir` 的 session / local-settings 两种持久度选择
- 条件公开：teammate 里的计划审批联动、某些 channel / Git / worktree 上下文前提
- 内部/灰度：更激进的 auto / coordinator / swarm 治理链、leader approval handoff、remote managed settings、fetch 跳过策略、symlink 大目录优化等实现细节

## 苏格拉底式自审

### 问：我现在缺的是更强权限，还是更小实施面？

答：如果只是要隔离实现，不该先调大权限；先问 worktree 是否更合适。

### 问：我需要的是隔离，还是扩张？

答：如果仍在当前 session 和代码面里，只是要正式纳入更多目录，那是 `/add-dir`，不是 EnterWorktree。

### 问：我是不是把计划模式当成礼貌开场，而不是正式治理闸门？

答：如果没有真正把实施推迟到计划收敛之后，那就是。
