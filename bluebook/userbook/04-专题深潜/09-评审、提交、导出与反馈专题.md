# 评审、提交、导出与反馈专题

## 用户目标

把当前会话里的工作对象升级成外部可消费对象：

- 本地 diff
- 评审结论
- commit
- PR
- 导出文件
- 产品反馈

这不是一个“顺手收尾”的阶段，而是 Claude Code 把内部工作翻译给团队、文件系统和产品回路的阶段。

## 第一性原理

Claude Code 不只帮你完成任务，还帮你把任务结果对象化。

更精确地说，这组入口并不是几套互不相干的业务，而是同一命令框架的三种执行形态：

- `prompt`：先把本地对象编码成模型任务，再交给主查询循环。
- `local`：直接在本地返回文本结果。
- `local-jsx`：先走本地交互或 UI，再产出结果。

这里至少有五种对象升级：

- 工作树差异 -> 本地事实面
- 差异或 PR -> 审查对象
- 工作树变化 -> commit
- commit / branch -> PR
- 会话现场 -> 导出文件或反馈材料

如果不把这些对象升级路径区分开，很容易把 `/diff`、`/review`、`/commit`、`/export`、`/feedback` 都误写成“不同按钮而已”。

## 正确入口

### `/diff`

适合先站到本地事实面上，而不是让模型复述你改了什么。

这一步的价值是：

- 差异来自宿主侧真实工作树
- 不是模型二次概括

### `/review`

默认本地审查路径。

它更适合：

- 快速看当前 PR 或分支
- 在本地仓库与 `gh` 生态内完成代码审查

### `/ultrareview`

不是 `/review` 的“更强版本”，而是另一条远程审查链。

它适合：

- 账户、配额和入口条件都满足时
- 明确要走远程 bughunter / CCR 路径

### `/pr-comments`

它更像 PR 评论工作流的插件化补口，而不是普通用户默认稳定主线。

这类入口的提醒意义很强：

- 有些评审后续动作并不属于 builtin command
- 而是插件化延伸能力

### `/commit`

适合把当前工作树变化收口成一个受控提交。

它的真正价值不是“帮你敲 git 命令”，而是把高频高风险提交动作压成窄协议。

它并不是“只提交已 staged 内容”的壳；它会先看 `git diff HEAD` 再决定相关文件的收口动作。

### `/commit-push-pr`

适合把当前本地变化升级成 branch + PR 这个团队协作对象。

它关心的不是只多做几条 `git` / `gh` 命令，而是：

- 分支是否合理
- PR 是否已存在
- 默认分支策略
- reviewer / changelog / 协作元信息

### `/export`

适合把当前会话正式外化给文件系统或别人。

它不是 `/copy` 的放大版，而是把内部会话对象翻译成正式文本表示。

### `/release-notes`

适合把产品变化重新带回当前 CLI，会影响你如何解释“为什么这版行为不同了”。

### `/feedback`

适合把当前摩擦点连同现场证据一起回流到产品侧，而不是脱离上下文重述问题。

## 隐藏的运行时合同

### `/commit` 是受控提交协议

它会固定拉取仓库事实，并限制允许调用的 git 动作，而不是把整套 git 自由开放给模型。

更准确地说，它是“预烘焙上下文型命令”：

- 先在本地执行 `git status/diff/...`
- 把结果写进 prompt
- 再让模型在窄工具白名单内完成提交

### `/commit-push-pr` 是协作对象升级器

它处理的是 branch、PR、远程、现有 PR 与协作元信息，不是“一键发布”。

### `/review` 与 `/ultrareview` 属于不同宿主

前者更本地，后者更远程。深浅不是最重要的区别，宿主和成本模型才是。

而 `/pr-comments` 处于另一条边界：

- 它更像迁移到插件后的兼容入口
- 不是普通用户默认稳定主线

### `/export` 是正式外化，不是内部连续性

它把会话渲染成纯文本并落盘，目标是离开系统，而不是继续在系统内工作。

### `/feedback` 带的是证据，不只是表单内容

它回流的不只是用户一句抱怨，而是 transcript、环境信息、日志与其他现场材料。

### `/release-notes` 与 `/export` 更多在本地完成

这两者并不依赖主对话里的常规推理链：

- `/release-notes` 读取 Claude Code 产品版本说明
- `/export` 把会话渲染成纯文本并导出

## 最容易误用的点

### 把 `/review` 写成自动远程深审

默认不是。

### 把 `/ultrareview` 写成稳定增强版 `/review`

也不是。它有单独配额、门禁和远程任务链。

### 把 `/commit` 当成自动全仓库提交

它仍然假设你在做受控收口，而不是无差别吞并所有变化。

### 把 `/commit-push-pr` 当成只开 PR

它通常会连带处理分支、commit、push 与 PR，而不是只做最后一步。

### 把 `/export` 当复制工具

它的职责是正式对外交付。

### 把 `/feedback` 当所有环境下都同样开放的按钮

它会受 provider、privacy、policy 与组织数据保留策略影响。

### 把 `/release-notes` 当仓库 release note 生成器

它看的不是你的项目版本，而是 Claude Code 产品本身的变更证据。

## 稳定面与条件面

### 稳定主线

- `/diff`
- `/review`
- `/commit`
- `/commit-push-pr`
- `/export`
- `/release-notes`

### 条件或灰度面

- `/ultrareview`
- `/pr-comments`
- 更远程化的 PR 审查和评论处理路径
- `/feedback` 在部分部署、策略或隐私前提下的限制
- attribution、undercover 与远程会话相关的提交差异

## 用户策略

更稳的外部化顺序通常是：

1. 先用 `/diff` 看事实面。
2. 再用 `/review` 判断质量。
3. 需要远程深审时，明确区分 `/ultrareview` 是另一条路径。
4. 用 `/commit` 收口提交。
5. 用 `/commit-push-pr` 升级为团队协作对象。
6. 用 `/export` 或 `/feedback` 把结果或问题正式带出会话。
