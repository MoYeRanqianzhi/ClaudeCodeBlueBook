# 等待输入、权限提问与阻塞态投影专题

## 用户目标

不是笼统地问“为什么系统说在等输入”，而是更稳地处理这些真实场景：

- 为什么同样看起来都像“权限提问”，有时你只能看到一个 waiting-for-input 状态，有时却能看到更厚的上下文。
- 为什么同样都叫 `can_use_tool`，并不是每一次都会把整条会话推成完整 blocked session。
- 为什么 bridge / remote-control 这条线很多时候只会告诉远端“现在被卡住了”，却不会把完整 pending context 一起发出去。
- 为什么 worker store 里已经有 `pending_action`，CLI 前台本地恢复后却不一定重建出同样的 blocked UI。

如果你的误判已经滑到更下游：

- “审批 verdict 已经回来了，为什么系统后面还在收尾、重判、扫尾或写设置”

先读 [21-桥接审批后处理与权限尾部分诊专题.md](./21-%E6%A1%A5%E6%8E%A5%E5%AE%A1%E6%89%B9%E5%90%8E%E5%A4%84%E7%90%86%E4%B8%8E%E6%9D%83%E9%99%90%E5%B0%BE%E9%83%A8%E5%88%86%E8%AF%8A%E4%B8%93%E9%A2%98.md)。

这页处理的是更靠前的一类误判：

- permission ask 还没判完时，系统为什么已经进入 blocked state，以及这层 blocked 信息为什么会有厚薄差异。

## 第一性原理

“在等输入 / 在等审批”不是一个对象，而是至少沿着六条轴线分裂：

1. `Ask Transport`：
   当前这次 permission 问题只是怎么走协议，还是已经变成会话级 blocked state。
2. `Blocked Phase`：
   当前系统只是在说 “requires_action / waiting”，还是已经有更细的阻塞上下文。
3. `Typed Block Context`：
   当前是否已经生成一份窄而结构化的 `requires_action_details`。
4. `Queryable Pending JSON`：
   当前是否还保留了一份可查询、可被外部 frontend 继续读的 `pending_action`。
5. `Publish Ceiling`：
   当前这条链最多会把 blocked 信息发布到哪一层，是完整 context，还是只有 blocked bit。
6. `Foreground Restore`：
   当前本地前台恢复时，会不会把这份 blocked 信息重建回来。

如果这六轴不先拆开，最容易写错的就是：

- 把 `can_use_tool` 直接写成完整 blocked session。
- 把 `requires_action_details` 和 `pending_action` 写成同一张面。
- 把 bridge 的 waiting bit 写成完整 pending context。
- 把 worker store 里有账写成 CLI 前台已经消费。

## 隐藏的运行时合同

### `can_use_tool` 先回答的是“怎么问”，不是“blocked session 现在长什么样”

所以最稳的一句不是：

- 发了 permission ask，就一定进入完整 waiting context

而是：

- ask transport 和 blocked-state promotion 不是同一层

### `stdio` 是 blocked-state 升级门槛，不是所有 permission path 的共性

只有：

- `stdio`
- `onPermissionPrompt`
- `notifySessionStateChanged('requires_action', details)`

这条链成立时，

你才会同时看到：

- typed blocked context
- queryable pending JSON

### bridge 只签裸 blocked bit，不签完整 pending context

所以远端很多时候只能知道：

- 现在 blocked 了

却未必拿到：

- 完整 `pending_action`
- 原始 input
- 更厚的 prompt context

### worker store 有账，不等于本地 foreground 一定恢复

因为 restore path 当前只回填：

- `permission_mode`
- `is_ultraplan_mode`

而不会自动重建：

- `pending_action`

## 不该误用的邻近能力

### 不要把这页写成 21 的前传缩写版

21 讨论的是：

- verdict 已经回来了之后
- 各种尾巴为什么继续分叉

这页讨论的是：

- verdict 还没回来时
- blocked-state 为什么已经有厚薄差异

### 不要把这页写成 20 的缩写版

20 先分 continuation / host control / ingress input / permission verdict。

这页默认你已经过了那一关，

只处理：

- permission ask 怎么升级成 blocked-state

### 不要把 `requires_action_details` 和 `pending_action` 写成技术术语换壳

对用户来说，它们分别对应的是：

- 一份更窄、更结构化的阻塞上下文
- 一份更宽、可被外部 frontend 继续读的 pending JSON

如果重新压回一句“反正都表示正在等输入”，

就会重新丢掉专题页的价值。

## 稳定 / 条件 / 灰度保护

| 类型 | 对象 |
| --- | --- |
| 稳定可见 | `22` 这一层更稳的是 ask transport、typed blocked context、queryable pending JSON 与 bridge blocked bit 不是同一种等待输入状态 |
| 条件公开 | blocked 信息厚薄取决于 permission host path、是否走 `stdio`、是否只是 bridge publish、以及 foreground restore 是否真的接上 |
| 内部/灰度层 | `pending_action` 的具体存储位置、`requires_action_details` 的字段形状、bridge `reportState(...)` 的内部调用链与 restore 缺口都属于证据层 |

## 对用户的策略结论

如果你的问题是：

- “为什么系统都说在等输入，但有时只有 waiting bit，有时却能拿到更厚的上下文？”

更稳的拆法是：

1. 先问这次 ask 只是 transport，还是已经升级成 blocked-state。
2. 再问当前拿到的是 typed blocked context，还是 queryable pending JSON。
3. 再问这份 JSON 只是进了 worker store，还是 foreground 已恢复。
4. 最后再问 bridge 这条线是不是天生就只签裸 blocked bit。

而不是一句：

- “permission prompt 有时信息多，有时信息少。”

因为这几句分别落在不同层。
