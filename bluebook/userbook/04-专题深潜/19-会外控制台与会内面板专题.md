# 会外控制台与会内面板专题

## 用户目标

不是只记住几个相似名字，而是看清 Claude Code 里有一整类容易混写的双层入口：

- `claude auth login/status/logout` 与 `/login` `/logout`
- `claude mcp ...` 与 `/mcp`
- `claude plugin ...` 与 `/plugin` `/reload-plugins`
- `claude agents` 与 `/agents`
- `claude doctor`、`setup-token` 与 `/doctor` `/status`

这些入口经常长得像同一个动作的两套语法，但源码显示它们根本不在同一层。

## 第一性原理

Claude Code 至少同时维护两种控制面：

- 会外控制台：你还没进入默认 REPL，或你要直接对宿主、安装、认证、配置、插件市场、MCP 配置做生命周期管理。
- 会内面板：你已经在当前会话里，要基于当前 `AppState`、当前权限上下文、当前工具池，对这条会话继续操作。

所以同一个名词在两层里可能分别代表：

- 持久化配置与生命周期管理
- 当前会话的可见状态与即时操作

最危险的误判不是“名字记错了”，而是：

- 把 root command 当 slash command 的外壳
- 把 slash command 当 root command 的 UI 版

如果把这一页再压成更硬的对象链，只剩两段：

- `compile same world -> bootstrap plane -> session materialization -> authority/context assembly -> runtime handoff`
- `state truth -> trustworthiness -> decision window -> continuation path -> session tuning / display`

第一段是会外 root plane 的责任：

- 先把一个可工作的世界装起来
- 再把控制权交给 print 流、交互 REPL、后台会话或 bridge

第二段才是会内 session plane 的责任：

- 在当前世界里投影 present truth
- 判断这套 truth 值不值得信任
- 决定还能不能继续，以及继续后怎样调参与显示

所以更稳的判断顺序不是“先看名字像不像”，而是：

1. 我现在是在装世界，还是在当前世界里继续工作。
2. 我改的是 lifecycle / config source / install state / inventory，还是当前 `AppState` / permission context / tool pool。
3. 我需要的是动作完成即退出，还是让当前会话继续活着并立刻吃到状态变化。
4. 如果找不到对偶项，就直接承认 root-only 或 session-only。

## 这两层在源码里长什么样

### 会外 root commands

`main.tsx` 里的 root commands 通过 Commander 注册，按需 lazy-import handler，执行完成后通常直接退出进程。

典型特征：

- `claude auth ...` 进入 `src/cli/handlers/auth.ts`
- `claude mcp ...` 进入 `src/cli/handlers/mcp.js`
- `claude plugin ...` 进入 `src/cli/handlers/plugins.js`
- `claude agents` 进入 `src/cli/handlers/agents.ts`
- `claude doctor`、`setup-token` 进入 `src/cli/handlers/util.tsx`

这类入口处理的是：

- 认证生命周期
- MCP 配置增删查改
- 插件和 marketplace 生命周期
- agent inventory
- 宿主诊断和 token setup

### 会内 slash commands

`commands.ts` 里的 slash commands 大量是 `local-jsx`，会在当前会话里加载一个面板、弹窗或即时动作。

典型特征：

- `/login` 会在当前 session 中触发登录流程，并刷新当前会话依赖的 auth state
- `/mcp` 会读当前 app state 里的 MCP clients，并做 enable/disable/reconnect 这类即时动作
- `/plugin` 直接进入 `PluginSettings`
- `/agents` 会拿当前 `toolPermissionContext` 算出当前会话可管理的 tools，再渲染 `AgentsMenu`
- `/doctor` 直接渲染 `Doctor` screen

这类入口处理的是：

- 当前会话正在看见什么
- 当前会话现在能改什么
- 当前会话如何立刻吃到状态变化

## 五组最容易混写的入口

### `claude auth ...` 与 `/login` `/logout`

这组最容易因为词面相同而被写扁。

更稳的区分是：

- `claude auth login/status/logout` 是会外认证控制台
- `/login` `/logout` 是当前会话里的认证切换入口

关键差异：

- root `auth login` 直接跑 OAuth/browser 流，成功后写入本地认证状态并退出
- slash `/login` 不只是拿 token，还会在当前 session 内刷新 cost、GrowthBook、policy limits、remote managed settings、trusted device enrolment、authVersion 等状态
- root `auth status` 是独立的会外认证查看命令，slash 层并没有一个真正等价的 `/auth status`
- `/logout` 在会话里做的是“把当前身份现场清干净后退出”，root `auth logout` 则是会外注销入口

所以 `/login` 更像“在当前会话里重建身份现场”，而 `claude auth ...` 更像“在会话外管理认证生命周期”。

### `claude mcp ...` 与 `/mcp`

这组最容易被误写成“一个是命令行版，一个是菜单版”。

其实对象不同：

- `claude mcp ...` 管的是 MCP 配置本身
- `/mcp` 管的是当前会话里看到的 MCP 服务器与连接状态

关键差异：

- root `claude mcp add/remove/list/get/add-json/...` 处理的是配置源、server registration 和 `mcp serve`
- slash `/mcp` 是 immediate `local-jsx`，在当前会话里支持 reconnect、enable、disable，默认打开 `MCPSettings`
- slash `/mcp` 明确依赖当前 `AppState` 里的 MCP clients
- root `claude mcp list/get` 更接近“配置健康检查”，会从配置出发主动检查 server，而不是读取当前 session 里的 MCP 真相
- root `mcp serve` 属于“让 Claude Code 充当 MCP server”的会外入口，绝不能和用户面的 `/mcp` 管理页混成一回事

所以：

- 想管理配置，走 `claude mcp ...`
- 想看当前 session 的连接状态和即时开关，走 `/mcp`

### `claude plugin ...` 与 `/plugin` `/reload-plugins`

这组最能暴露“生命周期管理”和“当前会话生效”不是同一件事。

更稳的区分是：

- `claude plugin ...` 管插件包与 marketplace 生命周期
- `/plugin` 看和改当前会话里的插件设置面
- `/reload-plugins` 让当前 session 吃到磁盘上的变更

关键差异：

- root `claude plugin validate/list/install/uninstall/enable/disable/update/...` 是会外插件控制台
- slash `/plugin` 只是把你带进 `PluginSettings`
- 插件刚装好，不代表当前 session 已经加载到了；这时常常还需要 `/reload-plugins`
- `/reload-plugins` 的对象是当前 session refresh，不是安装流程

所以“装了插件为什么这轮还没生效”，很可能不是安装失败，而是你还停在会外生命周期层，没有回到会内刷新层。

### `claude agents` 与 `/agents`

这组名字最像，但对象几乎完全不同。

更稳的区分是：

- `claude agents`：列出当前 cwd 下解析到的 agent inventory、来源与 shadowing 关系
- `/agents`：基于当前会话 permission context 和 tool pool 打开的 agent 配置面板

关键差异：

- root `claude agents` 输出的是清单与覆盖来源，明显是 inventory/report
- slash `/agents` 要先拿当前 `toolPermissionContext`，再算当前会话能管理的 tools，最后渲染 `AgentsMenu`

所以：

- 想知道当前工作目录下到底解析出了哪些 agent，走 `claude agents`
- 想在当前会话里看和调 agent 配置，走 `/agents`

### `claude doctor`、`setup-token` 与 `/doctor` `/status`

这组里最容易出现两种错：

- 把 `claude doctor` 写成 `/doctor` 的 CLI 别名
- 给 `setup-token` 硬找一个 slash 同类项

更稳的区分是：

- `claude doctor` 是会外宿主诊断命令，会自行搭一套 root render 栈并退出
- `/doctor` 是当前会话里的诊断 screen
- `/status` 是当前会话状态投影，不是 `auth status` 或 `doctor` 的别名
- `setup-token` 是 root-only 入口，没有自然的 slash 等价物

尤其 `setup-token` 要单独记住：

- 它解决的是长生命周期 token setup
- 它不是 `/login` 的“更高级版本”
- 它也不是当前 session 里的状态面板

## 隐藏的运行时合同

### session plane 默认消费的是 root plane 已装好的世界

这一层最容易被漏写。

slash commands 之所以能直接谈：

- 当前 `AppState`
- 当前 permission context
- 当前 tool pool
- 当前 session refresh

前提不是“它们名字和 root command 很像”，而是：

- root plane 已经先把这轮工作的世界装出来了
- 当前 session 已经有一套可继续消费的 authority/context assembly

所以 `/status`、`/mcp`、`/plugin`、`/agents` 这类入口，不是在和 root command 重复造世界，而是在已装好的世界里消费 present truth。

### 会外命令多数直接退出，会内命令继续留在当前线程

这不是小实现差异，而是对象差异：

- root commands 常常完成动作后 `process.exit`
- slash commands 常常是让当前会话继续活着，并刷新当前状态

### 会内命令比会外命令更依赖当前 `AppState`

例如：

- `/login` 会刷新当前 session 的 authVersion
- `/mcp` 直接读取 `appState.mcp.clients`
- `/agents` 直接读取当前 permission context 和 tools

这说明 slash 面板不是“同名 CLI 的图形壳”，而是当前会话上下文的一部分。

### `claude plugin install` 和 `/reload-plugins` 不解决同一个问题

前者回答：

- 包装好了没有
- 安装到了哪里
- scope 是什么

后者回答：

- 当前这条 session 吃到了没有

### `/status` 没有真正的 root 对等物

`claude auth status` 看的是认证状态。

`/status` 看的是当前 Claude Code 会话的宿主状态、provider、模型、MCP、settings 等投影。

两者不能混写成“状态命令”。

## 最容易误用的点

### 把 `claude auth login` 当 `/login`

一个在会话外建或切换身份，一个在会话里重建当前身份现场。

### 把 `claude mcp serve` 当 `/mcp`

一个是 server 侧入口，一个是当前 session 的管理面。

### 插件装完就认为当前 session 已经加载

很多时候还差 `/reload-plugins`。

### 把 `claude agents` 当 `/agents` 的只读版

前者是 inventory，后者是当前会话的配置面板。

### 给 `setup-token` 硬找 slash 对等物

它就是 root-only。

## 稳定面、条件面与 root-only

### 稳定公开，且确实成对出现

- `claude auth login/logout` 与 `/login` `/logout`
- `claude mcp ...` 与 `/mcp`
- `claude plugin ...` 与 `/plugin`
- `claude agents` 与 `/agents`
- `claude doctor` 与 `/doctor`

### 稳定公开，但不应强行配对

- `claude auth status`
- `setup-token`
- `/status`
- `/reload-plugins`

### 产品面或条件面仍要单独标注

- `/privacy-settings`
- `/upgrade`
- `/passes`

它们和认证入口相邻，但不是 root/slash 对偶关系本身。

## 用户策略

更稳的实际使用顺序通常是：

1. 先问自己：我要改的是宿主生命周期，还是当前会话状态。
2. 需要登录、安装、列清单、改配置源时，优先走 root commands。
3. 已经在会话里，要看当前暴露面、即时启停、刷新当前 session 时，优先走 slash commands。
4. 看见同名入口时，不要先假设它们只是两套语法。
5. 遇到 root-only 入口时，直接承认它没有 slash 对等物，不要硬凑。
