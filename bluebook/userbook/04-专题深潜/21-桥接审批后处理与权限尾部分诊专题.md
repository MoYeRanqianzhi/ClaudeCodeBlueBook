# 桥接审批后处理与权限尾部分诊专题

## 用户目标

不是笼统地问“为什么桥接审批回来后系统还在变”，而是更稳地处理这些真实场景：

- 为什么远端已经 allow / deny 了，本地 prompt 却还会撤场、退订、吃晚到 response。
- 为什么改了 permission mode 之后，有些等待中的 ask 会自动重判，有些不会。
- 为什么 sandbox network 的一次 verdict 会顺手扫掉同 host 的一整组请求。
- 为什么用户点了 `persistToSettings` 之后，看到的不只是一次“写设置”，而是本地 context、settings 和 live sandbox 都跟着变。

如果你的误判还停在更上游的层级：

- “远端输入怎么进来”
- “bridge ingress 和会话接续是不是一回事”

先读 [20-远端输入注入、桥接审批与会话接续专题.md](./20-%E8%BF%9C%E7%AB%AF%E8%BE%93%E5%85%A5%E6%B3%A8%E5%85%A5%E3%80%81%E6%A1%A5%E6%8E%A5%E5%AE%A1%E6%89%B9%E4%B8%8E%E4%BC%9A%E8%AF%9D%E6%8E%A5%E7%BB%AD%E4%B8%93%E9%A2%98.md)。

这页处理的是更窄的一类误判：

- 审批 verdict 已经回到本地之后，为什么后面还会继续分叉成几种完全不同的尾部问题。

## 第一性原理

“桥接审批回来了”不是一个对象，而是至少沿着五条轴线分裂：

1. `Verdict Ledger`：
   当前这次审批到底还记在哪张本地账上。
2. `Request Closeout`：
   prompt 撤场、unsubscribe、late response 消费与 queue recheck 到底是不是同一种收口。
3. `Local Re-evaluation`：
   本地 permission context 一变，哪些 ask 会收到重判广播，哪些不会。
4. `Host-level Sweep`：
   sandbox network 为什么会按 `hostPattern.host` 批量 settle。
5. `Persist Write Surfaces`：
   点下 `persistToSettings` 后，为什么会同时碰到 context、settings 和 live runtime。

如果这五轴不先拆开，最容易写错的就是：

- 把 prompt 撤场写成 verdict ledger 的一部分。
- 把本地重判广播写成 request closeout 的附录。
- 把 sandbox network 的 same-host sweep 写成普通 permission race。
- 把 `persistToSettings` 写成单层“写设置”。

## 隐藏的运行时合同

### verdict 回来，不等于所有尾巴都已经结束

`196` 和 `198` 连起来之后，最稳的句子不是：

- verdict 有了，permission race 就完了

而是：

- verdict 只是根页
- closeout 是它的第一个后继问题

### mode change 后的自动重判，不是 verdict closeout 的副产品

`199` 回答的是：

- 本地 context 变了，哪些等待 ask 要不要重判

它属于：

- re-evaluation

不是：

- closeout

### sandbox network 的 same-host settle，不是普通 request cleanup

`201` 回答的是：

- host family 怎么一起 settle

它属于：

- host-level sibling sweep

不是：

- request-level permission closeout

### persist-to-settings 不是“顺手写个规则”

`202` 回答的是：

- 为什么用户手势会跨 context / settings / runtime 三层传播

它属于：

- write-surface propagation

不是：

- UI 小副作用

## 不该误用的邻近能力

### 不要把这页写成 20 的缩写版

20 先分 continuation / host control / ingress input / permission verdict。

这页默认你已经过了那一关，

只处理：

- permission verdict 回来以后
- 为什么还会继续在本地分叉

### 不要把这页写成 197 的缩写版

197 解决的是 ingress 深线的六层阅读顺序。

这页不再回去讲：

- `handleIngressMessage(...)`
- user transcript adapter
- inbound normalization

它只讲：

- permission tail 的用户问题分诊

### 不要把 198/199/201/202 写成“四种实现细节”

对用户来说，它们分别对应的是：

- 收尾
- 重判
- 同 host 扫尾
- 持久化传播

如果把这四件事重新压回一句“审批回来后的细节”，

就会重新丢掉专题页的价值。

## 对用户的策略结论

如果你的问题是：

- “桥接审批已经回来了，为什么系统后面还会有这些不同动作？”

更稳的拆法是：

1. 先问这是不是 verdict ledger 问题。
2. 再问这是不是 request-level closeout。
3. 再问这是不是本地 re-evaluation。
4. 再问这是不是 sandbox network 的 host-level sweep。
5. 最后再问这是不是 persist 的多层写面传播。

而不是一句：

- “桥接审批回来以后有时会取消 prompt，有时会自动重判，有时会顺手写设置。”

因为这几句分别落在不同层。
