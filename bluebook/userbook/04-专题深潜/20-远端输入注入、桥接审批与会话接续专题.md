# 远端输入注入、桥接审批与会话接续专题

## 用户目标

不是笼统地问“Claude Code 能不能远程继续用”，而是更稳地处理这些真实场景：

- 为什么同样看起来都像“远端继续当前工作”，`/desktop`、`/mobile`、`/session`、`/remote-control` 其实不是同一种能力。
- 为什么远端前端里的输入、权限审批、会话接续看起来都从同一个 web / mobile 面来，但在本地运行时里分属不同链路。
- 当你在 Claude.ai / Mobile / Desktop 上看到远端输入、审批 prompt、mode 变化或会话接续时，应该先排查哪一层，而不是跳进一堆彼此相邻的源码细节页。

如果你的问题只是：

- “哪条命令能继续到 Desktop / Mobile / remote viewer”

先读 [16-IDE、Desktop、Mobile 与会话接续专题.md](./16-IDE%E3%80%81Desktop%E3%80%81Mobile%20%E4%B8%8E%E4%BC%9A%E8%AF%9D%E6%8E%A5%E7%BB%AD%E4%B8%93%E9%A2%98.md)。

这页处理的是更难的一类误判：

- 远端接续、bridge ingress 输入注入、桥接审批与本地权限重判为什么不是同一种“远端能力”。

## 第一性原理

“远端继续当前工作”至少沿着五条轴线分裂：

1. `Viewer Continuation`：
   当前会话要不要交给别的前端继续看/继续用。
2. `Host Control`：
   本机 CLI 是否变成远端宿主，由 bridge / remote-control 接管连接状态。
3. `Ingress Input Injection`：
   远端 user input 怎样进入本地 prompt queue。
4. `Permission Verdict Chain`：
   远端审批如何回到本地 pending verdict / closeout / re-evaluation。
5. `Context / Payload Normalization`：
   输入在进 prompt sink 之前，哪些前处理只修 content，哪些是在做附件或策略层的改写。

如果这五轴不先拆开，最容易写错的就是：

- 把 `/session` 当成 bridge host。
- 把远端审批写成通用 control bus。
- 把 `SDKMessage` 宽类型写成通用 ingress event 面。
- 把 mode change 的本地重判写成所有 pending ask 的统一重判。

### 4. 我卡在“桥接接续”和“桥接输入/审批”为什么会一起误判

先把这三层分开：

- `06-前端与远程专题.md`
- `16-IDE、Desktop、Mobile 与会话接续专题.md`
- `05-控制面深挖/197-...六层阅读链.md`

因为最常见的误判不是不会读源码，

而是把：

- viewer continuation
- host control
- ingress input injection
- permission verdict / closeout

四层对象写成一句“远端前端能力”。

## 隐藏的运行时合同

### `/session` 与 `/desktop` 回答的是“把工作对象交给另一个前端继续”

所以它们先属于：

- continuation / handoff

不是：

- ingress adapter
- permission race

### `/remote-control` 回答的是“本机 CLI 要不要变成 bridge host”

所以它首先属于：

- host control

不是：

- 远端 user input 已怎样进入本地 prompt

### `handleIngressMessage(...)` 与 `extractInboundMessageFields(...)` 回答的是“远端 user input 怎样变成本地 prompt”

所以这条线首先属于：

- input injection
- transcript adapter

不是：

- viewer continuation

### `pendingPermissionHandlers` 与 `cancelRequest(...)` 回答的是“远端审批如何回到本地权限队列”

所以这条线首先属于：

- permission verdict / closeout

不是：

- prompt adapter

## 不该误用的邻近能力

### 不要把 `/session` 当作 bridge ingress 的“前台版本”

它交付的是接续地址，不是输入注入链。

### 不要把 bridge permission prompt 当作 viewer continuation 的副作用

它是一条独立的 permission race / verdict chain。

### 不要把 `SDKMessage` 宽类型当成“远端前端里所有东西最终都走这条链”

它在 ingress 上只落成 user transcript adapter，而不是所有 remote capability 的统一汇合点。

### 不要把 mode change 的本地重判广播当成“远端 mode 一改，所有 ask 都自动重判”

它只触达 leader-local queue，不共享 remote / inbox ask 的 surface。

## 稳定 / 条件 / 灰度保护

| 类型 | 对象 |
| --- | --- |
| 稳定可见 | `20` 这一层更稳的是四分法本身：viewer continuation、host control、ingress input injection、permission verdict / closeout 不是同一种远端能力 |
| 条件公开 | Desktop / Mobile / viewer handoff、bridge host、remote approval 与 input injection 都依赖当前入口、宿主与运行时前提，不写成一条默认远端主线 |
| 内部/灰度层 | 具体 ingress helper、`SDKMessage` 宽类型、mode-change 本地重判与若干桥接 callback 只是证据，不该被写成用户稳定能力名 |

## 对用户的策略结论

如果你的问题是：

- “为什么我在 web / mobile / Desktop 里看到的远端行为这么像，但在 CLI 里却落成不同运行时？”

更稳的拆法是：

1. 先问这是不是 continuation / handoff。
2. 再问这是不是 host control。
3. 再问这是不是 ingress input injection。
4. 再问这是不是 permission verdict / closeout / re-evaluation。

而不是一句：

- “远端功能有时候能继续会话，有时候能审权限，有时候还能把输入发回来。”

因为这四句分别落在不同层。
