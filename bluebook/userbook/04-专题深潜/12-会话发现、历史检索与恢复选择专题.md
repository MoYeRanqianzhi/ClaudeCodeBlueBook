# 会话发现、历史检索与恢复选择专题

## 用户目标

不是“当前这轮怎么继续”，而是：

- 以前的工作到底在哪里
- 我该恢复哪个会话
- 该找旧 prompt，还是找旧 session
- 该直接接回，还是只借旧内容

这条线关注的是“找到过去的工作对象”，而不是“运营当前工作对象”。

## 第一性原理

会话运营和会话发现不是一回事。

更稳的拆法是：

- 当前会话对象怎么命名、分叉、回退、导出
- 过去的会话对象如何被搜索、预览、筛选、恢复

前者已经在“会话运营、分叉与回退专题”里，后者应该单独成立。

## 正确入口

### `/resume`

这是会话发现与恢复的主入口。

它不只是“恢复上一次”，而是：

- 打开 picker
- 按同目录、同 repo worktree、跨目录做分流
- 必要时只给出应执行的恢复命令

### custom title 搜索

当你已经给会话命名后，title search 比翻时间线更稳。

源码里 `searchSessionsByCustomTitle()` 明确会：

- 跨同 repo worktrees 搜索
- 不区分大小写
- 按 recency 排序
- 以 `sessionId` 去重

这说明命名会话并不是 UI 美化，而是会直接改变恢复效率。

### prompt 历史检索

适合找“我以前怎么问过”，而不是找“我以前在哪个会话里做过”。

它与 `/resume` 分工不同：

- `/resume` 找工作对象
- prompt history 找输入模式

### session preview

在真正恢复前，预览最近会话是重要的降误选机制。

因为恢复的核心不是“标题像不像”，而是“这是不是你要接回的那个工作现场”。

## 隐藏的运行时合同

### `/resume` 默认是 worktree-aware

它不是只看当前目录里的历史，而是会把同 repo worktree 一起纳入候选范围。

### 跨目录不会盲目接管

如果会话来自别的目录，系统更倾向于给出应执行的命令，而不是直接把当前会话硬切过去。

### custom title 搜索不是全局裸搜索

它建立在已经 enrich 过的 session metadata 上，因此更接近“恢复账本中的正式字段”，而不是对全文做模糊 grep。

### prompt history 和 session history 是两本账

一个面向输入重用，一个面向工作对象恢复。

## 最容易误用的点

### 把 prompt 历史搜索当 `/resume`

它不会恢复工作现场，只会帮你找回输入。

### 把 `/resume` 当全文聊天搜索

它找的是 resumable session，而不是对所有历史消息做自由检索。

### 不给会话命名，却期待以后很好找

如果长任务经常要中断再回来，custom title 是高价值能力，不是附带美化。

### 把同 repo worktree 看成完全割裂的会话岛

源码明确把它们纳入恢复候选，所以读 userbook 时也不该把它们写成互相隔离的历史空间。

## 稳定面与条件面

### 稳定主线

- `/resume`
- resume picker
- custom title 搜索
- 同 repo worktree 感知
- prompt 历史检索

### 条件或边界面

- ant-only `/tag`
- 需要跨目录手动恢复命令的路径
- sidechain、team 等不应混入普通恢复主线的会话类型

## 用户策略

更稳的发现与恢复路径通常是：

1. 长任务开始后尽早命名会话。
2. 想接回旧工作时先用 `/resume`，不要先翻原始 transcript。
3. 只想复用提问方式时，用 prompt 历史检索，而不是恢复整个会话。
4. 看到同 repo worktree 的候选时，不要误以为那一定是“别的项目”。
