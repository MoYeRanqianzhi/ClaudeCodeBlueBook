# 治理与安全专题

## 用户目标

在保持行动力的同时，避免：

- 在错误目录里直接改动。
- 在未想清方案时过早实施。
- 在高风险命令上无边界扩张。
- 把稳定能力与灰度能力混写进同一流程。

如果你现在不是在看“治理总览”，而是在问“当前到底该收权限、进计划、切 worktree，还是只扩工作面”，先读 [安全放大工作面/README.md](./%E5%AE%89%E5%85%A8%E6%94%BE%E5%A4%A7%E5%B7%A5%E4%BD%9C%E9%9D%A2/README.md)。

## 正确入口

### 权限规则

用 `/permissions` 调整 allow / deny 规则，而不是让模型在主对话里“记住以后允许什么”。

因为规则不只是显示给用户看，还会前置影响工具池过滤。

### 计划模式

用 `/plan` 或 `EnterPlanMode` 进入只读探索阶段。

从工具实现可见，计划模式不是软建议，而是权限模式切换：

- 明确切到 `plan`。
- 明确要求暂不写文件。
- 明确退出计划模式后再开始实施。

而 `ExitPlanModeV2Tool` 又进一步说明：

- 计划不是口头说“我想好了”，而是会和 plan file、审批、allowed prompts、teammate 审批等机制连接。
- 在 teammate 场景下，退出计划模式甚至会转成 leader 审批协议，而不是本地弹窗。

### worktree

用 `EnterWorktree` 在需要隔离时切出工作面。

`worktree.ts` 透露出设计重点：

- slug 校验避免路径逃逸。
- 尽量复用已存在 worktree。
- 平铺嵌套 slug，避免 git ref / 目录冲突。
- 在大仓库里尽量跳过不必要 fetch。
- 允许 symlink 大目录以减少重复。

这说明 worktree 在 Claude Code 里是正式隔离机制，不只是 git 附件。

### 远程治理

治理并不止于本地权限。源码里还有更高一层的控制面：

- remote managed settings
- killswitch
- feature gate
- bridge / remote-control 前置检查

这些能力提醒读者：当前用户是否能看到某个入口、能否直接启用某种模式，并不只取决于本地版本。

## 不该误用的邻近能力

### 不要把 plan mode 当成“多想一会儿”

它是治理闸门，不是心态建议。

### 不要把 `/permissions` 当审批历史页面

它实际上会改变模型能看到什么工具。

### 不要把 worktree 只当多人协作技巧

单人高风险任务同样应该使用 worktree 隔离。

### 不要把远程控制当纯 UI 开关

`/remote-control` 连接前会做 preflight，已有连接时会进入另一套 disconnect / continue 对话逻辑。它是桥接状态机入口，不是单态按钮。

## 稳定面与灰度面

### 稳定面

- `/permissions`
- `/plan`
- `EnterPlanMode`
- `ExitPlanMode`
- `EnterWorktree`
- `ExitWorktree`
- `/add-dir`

### 灰度或条件面

- 更激进的 auto mode / coordinator / swarm 配套治理。
- 远程托管设置与组织策略。
- 一些 feature-gated 的背景自治能力。

## 对用户的策略结论

高风险任务的稳健路径通常是：

1. 先进计划模式。
2. 再视情况切 worktree。
3. 再开始真正改动。
4. 让 `/permissions` 收敛长期例外规则。
