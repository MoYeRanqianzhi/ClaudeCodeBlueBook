# 治理与安全专题

## 先按治理对象，不按按钮

这一页最容易被误读成“权限命令列表”。

但从第一性原理看，治理先处理的不是按钮，而是：

- 当前能力申请到底落在哪个 `governance key`。
- 当前 working set、额度、权限和环境还配不配继续裁决，也就是当前 `decision window` 是否仍然成立。
- 当前问题该落 `拒收 / 降级 / 停止 / 清理后恢复 / 升级给人` 的哪一格。

如果把这页压成用户侧最小顺序，只该先做五步：

1. 先认这次要治理的到底是什么能力、例外或继续资格。
2. 再认当前 `decision window` 是否仍有决策增益，而不是默认还能继续。
3. 再分清哪些是治理真相，哪些只是 mode、modal、usage 条、`/compact` 或 default continue 这类投影。
4. 再给当前状态落一个明确 verdict：`拒收 / 降级 / 停止 / 清理后恢复 / 升级给人`。
5. 最后才决定该走 `/permissions`、`/plan`、worktree，还是远程治理入口。

## 用户目标

在保持行动力的同时，避免：

- 在错误目录里直接改动。
- 在未想清方案时过早实施。
- 在高风险命令上无边界扩张。
- 把稳定能力与灰度能力混写进同一流程。

如果你现在不是在看“治理总览”，而是在问“当前到底该收权限、进计划、切 worktree，还是只扩工作面”，先读 [安全放大工作面/README.md](./%E5%AE%89%E5%85%A8%E6%94%BE%E5%A4%A7%E5%B7%A5%E4%BD%9C%E9%9D%A2/README.md)。

## 进入本页前的 first reject signal

看到下面迹象时，应先回到治理判断，不要直接选命令：

- 你还在用 mode、modal、usage 条或“应该还能继续”解释当前状态，而没有先点名 `governance key`。
- 你还把 `Context Usage` 当账单面板，而没有把它当当前 `decision window` 的诚实投影。
- 你还把 `/compact`、default continue 或长摘要当成继续资格本身，而不是继续后的投影结果。
- 你还没给当前状态落下 `拒收 / 降级 / 停止 / 清理后恢复 / 升级给人` 之一，就已经开始选按钮。

## 正确入口

### 工作面扩张

如果当前真正的问题不是“要不要改长期权限规则”，也不是“要不要隔离实现面”，而是“要不要把另一片目录正式纳入当前工作面”，先用 `/add-dir`。

这一步先回答的是：

- 世界边界要不要扩大。
- sandbox / permission context 要不要同步刷新。

它不是 `cd`，也不是 worktree 的轻量替代。

### 权限规则

用 `/permissions` 调整 allow / deny 规则，而不是让模型在主对话里“记住以后允许什么”。

因为规则不只是显示给用户看，还会前置影响工具池过滤。

### 计划模式

用 `/plan` 或 `EnterPlanMode` 进入只读探索阶段。

从工具实现可见，计划模式不是软建议，而是权限模式切换：

- 明确切到 `plan`。
- 明确要求暂不写文件。
- 明确退出计划模式后再开始实施。

它对应的不是“多想一会儿”，而是先把当前 verdict 暂时落到“只读裁决”，避免在治理未收敛前提前进入执行。

而 `ExitPlanModeV2Tool` 又进一步说明：

- 计划不是口头说“我想好了”，而是会和 plan file、审批、allowed prompts、teammate 审批等机制连接。
- 在 teammate 场景下，退出计划模式甚至会转成 leader 审批协议，而不是本地弹窗。

### worktree

用 `EnterWorktree` 在需要隔离时切出工作面。

`worktree.ts` 透露出设计重点：

- slug 校验避免路径逃逸。
- 尽量复用已存在 worktree。
- 平铺嵌套 slug，避免 git ref / 目录冲突。
- 在大仓库里尽量跳过不必要 fetch。
- 允许 symlink 大目录以减少重复。

这说明 worktree 在 Claude Code 里是正式隔离机制，不只是 git 附件。

从治理顺序看，它对应的是“当前能力和目录边界需要被重新定价”，而不只是“我想开个分支”。

### 远程治理

治理并不止于本地权限。源码里还有更高一层的控制面：

- remote managed settings
- killswitch
- feature gate
- bridge / remote-control 前置检查

这些能力提醒读者：当前用户是否能看到某个入口、能否直接启用某种模式，并不只取决于本地版本。

换句话说，治理真相不只在本地 UI，也在更高一层的外部化真相链里。

## 不该误用的邻近能力

### 不要把 plan mode 当成“多想一会儿”

它是治理闸门，不是心态建议。

### 不要把 `/permissions` 当审批历史页面

它实际上会改变模型能看到什么工具。

### 不要把 worktree 只当多人协作技巧

单人高风险任务同样应该使用 worktree 隔离。

### 不要把远程控制当纯 UI 开关

`/remote-control` 连接前会做 preflight，已有连接时会进入另一套 disconnect / continue 对话逻辑。它是桥接状态机入口，不是单态按钮。

### 不要把 mode、usage、compact 当治理真相

它们最多只是投影或结果页。真正决定当前能不能继续、该不该隔离、是否需要升级给人的，是 `governance key`、`decision window` 与当前 failure verdict。

## 稳定面与灰度面

### 稳定面

- `/permissions`
- `/plan`
- `EnterPlanMode`
- `ExitPlanMode`
- `EnterWorktree`
- `ExitWorktree`
- `/add-dir`

### 灰度或条件面

- 更激进的 auto mode / coordinator / swarm 配套治理。
- 远程托管设置与组织策略。
- 一些 feature-gated 的背景自治能力。

## 对用户的策略结论

高风险任务的稳健路径通常是：

1. 先给当前状态落 `拒收 / 降级 / 停止 / 清理后恢复 / 升级给人` 的 verdict。
2. 如果新目录还没正式纳入当前工作面，先用 `/add-dir` 扩正式工作面。
3. 需要继续裁决时，再先进计划模式。
4. 需要隔离目录、风险或并行路径时，再切 worktree。
5. 需要稳定长期例外时，再让 `/permissions` 收敛规则。
6. 发现主权已不在本地时，直接升级给人或进入远程治理，而不是在本地反复试错。
