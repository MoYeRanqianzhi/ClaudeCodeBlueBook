# 连续性与记忆专题

## 用户目标

让一次开发工作不是“聊完即散”，而是能够：

- 在长任务中不中断地继续。
- 在离开后恢复现场。
- 在上下文变长后压缩继续。
- 把稳定偏好沉到记忆层。

## 正确入口

### 上下文观察

先看 `/context` 和 `/files`。

这是为了确认“当前共同工作现场”到底包含什么，而不是靠感觉判断是否该继续喂信息。

### 记忆管理

用 `/memory` 处理长期有效的偏好、规则和项目事实。

源码里的 `claudemd.ts` 说明记忆层不是单文件，而是分层加载：

- managed memory
- user memory
- project memory
- local memory

而且支持 `@include`。

`memory.tsx` 还显示一个很实际的使用事实：

- `/memory` 不只是打开固定文件，而是先列出 memory file，再必要时创建文件并调用外部编辑器。
- 它会根据 `$VISUAL` / `$EDITOR` 决定编辑器来源，并把这一层行为显式反馈给用户。

所以 `/memory` 更像“记忆对象管理入口”，而不只是某个路径快捷方式。

### 压缩

用 `/compact`。

重要的不是“减少字数”，而是 Claude Code 会先做：

- compact boundary 处理。
- session memory compaction 尝试。
- microcompact。
- 再进入传统或 reactive compact 路径。

也就是说，它在尽量保留继续行动所需的结构，而不是随便生成一段摘要。

### 恢复

用 `/resume`。

`resume.tsx` 说明恢复逻辑考虑了：

- 同仓库工作树。
- 跨项目限制。
- 自定义标题匹配。
- session id 直达。
- lite log 与 full log。

所以 `/resume` 本质上是一个“恢复工作对象”的 UI，不只是聊天历史搜索器。

### 邻近但不属于 continuity 本体的动作

`/rename`、`/export` 会紧挨着 continuity 出现，但它们承担的是“给当前工作对象加身份”与“把当前工作对象做正式外化”的责任，不属于本专题主对象。

对长任务来说，这些动作不是整理癖，而是在降低未来恢复与交接的认知成本。

如果你主要卡在当前工作对象该怎样被命名、外发、分叉、回退和清空，应转去读：

- [07-会话运营、分叉与回退专题.md](./07-%E4%BC%9A%E8%AF%9D%E8%BF%90%E8%90%A5%E3%80%81%E5%88%86%E5%8F%89%E4%B8%8E%E5%9B%9E%E9%80%80%E4%B8%93%E9%A2%98.md)

需要额外纠偏的是：

- `/tag` 在当前源码里是 ant-only / 内部能力，不应继续写成普通用户稳定主线。

## 不该误用的邻近能力

### 不要把记忆当聊天备忘录

记忆适合长期稳定约束，不适合每回合临时任务状态。

### 不要把 `/compact` 当作清屏

它是连续性工具，不是 UI 整理工具。

### 不要把 `/resume` 当跨项目万能跳板

源码明确区分同目录、同仓库 worktree、不同项目三类恢复语义。

### 不要把 `/export` 当成连续性替代品

导出适合交付和外发，不适合替代系统内部的 compact / resume / memory 机制。

## 稳定面与灰度面

### 稳定面

- `/context`
- `/files`
- `/memory`
- `/compact`
- `/resume`

### 条件或内部面

- `/tag`
- 自动 session memory。
- 某些 reactive compact 或 cached microcompact 分支。
- 远程环境中的恢复补写逻辑。

## 对用户的策略结论

最稳的连续性工作流通常是：

1. 用 `/context` 看现场。
2. 用 `/memory` 沉淀长期规则。
3. 用 `/compact` 处理长任务。
4. 用 `/resume` 返回任务，而不是新开会话重讲。
