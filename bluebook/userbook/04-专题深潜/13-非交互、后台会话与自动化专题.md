# 非交互、后台会话与自动化专题

## 用户目标

不是“打开终端聊一会儿”，而是把 Claude Code 接进更长、更稳的执行链：

- 用脚本或管道一次性拿结果
- 让长任务脱离当前终端继续跑
- 之后再回来查看日志或重新接上
- 把结果接到外部宿主或协议流

## 第一性原理

Claude Code 的自动化面至少分成三类问题：

- 这次调用是否需要交互式 UI
- 会话是否要落盘并可恢复
- 这次任务结束后，是直接退出，还是继续在后台活着

所以“非交互”“后台会话”“远程协议流”不能混成一个入口。

更稳的拆法是：

- 一次性输出：`claude -p/--print`
- 后台继续跑、以后再接回：`--bg` 与 `ps/logs/attach/kill`
- 宿主协议流：`--sdk-url` + `stream-json`

## 正确入口

### `claude -p/--print`

这是稳定主线上的非交互入口。

它的公开合同在 `main.tsx` 里非常明确：

- `-p/--print` 直接把会话切到非交互输出
- `--output-format` 只有 `text`、`json`、`stream-json`
- `--input-format` 只有 `text`、`stream-json`
- `--max-turns`、`--max-budget-usd`、`--allowed-tools`、`--disallowed-tools` 等约束都围绕 print 路径工作

适合：

- shell 管道
- CI / 本地自动化
- 一次性脚本
- 需要结构化输出的宿主

### `--bare`

这是公开稳定的最小化宿主模式，不应被误判成灰度面。

它解决的不是“后台运行”，而是“把 Claude Code 压到最小副作用面”：

- 跳过 hooks、LSP、plugin sync、auto-memory、背景预取等
- 不自动发现 `CLAUDE.md`
- 要求你显式提供 system prompt、`--add-dir`、`--mcp-config`、`--settings` 等上下文

适合高可控脚本，不适合默认当成交互主线。

### `--continue`、`--resume`、`--fork-session`

这条线不是交互式专属。

在 print 路径里，你依然可以：

- `--continue` 接当前目录最近一次会话
- `--resume` 指定 session ID 恢复
- `--fork-session` 从旧会话分叉，而不是复用原 session ID

这意味着自动化不必总是“无状态一次性调用”，也可以建立可恢复的长链条。

### `ps`、`logs`、`attach`、`kill` 与 `--bg`

这是“后台会话”线，不等于一般的 print 线。

源码能稳定确认的是：

- 它们在 CLI 启动链里有独立 fast-path
- 会围绕 `~/.claude/sessions/` 下的注册表工作
- `claude --bg` 的退出不是结束进程，而是 `tmux detach`
- 以后可以再用 `claude attach` 接回

但这一支的 handler 源文件在当前源码树里缺失，因此不应把更细行为写成确定事实。

更稳的理解是：

- `-p/--print` 解决“这次不要交互 UI”
- `--bg` 解决“这次先从当前终端脱身，但任务继续活着”

### `--sdk-url`

这是内部/托管宿主使用的协议流入口。

它会自动把会话推到：

- `print`
- `input-format=stream-json`
- `output-format=stream-json`
- `verbose`

然后用 `StructuredIO` / `RemoteIO` 把消息、状态、权限请求、会话元数据变成流式协议。

但这不等于：

- 所有消息都按同一宽度直通
- 所有元数据都自动以同一宿主合同暴露给外部消费者

更适合宿主集成，而不是人手直接敲。

## 隐藏的运行时合同

### print 路径默认跳过 trust dialog

`-p/--print` 的 help 文本已经把这一点写明了：工作区 trust dialog 会被跳过。

这不是小注释，而是硬边界。

所以非交互脚本应只在可信目录里运行。

### `stream-json` 不是“任意 JSON 开关”

它有严格组合约束：

- `--input-format=stream-json` 需要 `--output-format=stream-json`
- `--include-partial-messages` 只能跟 print + `stream-json` 组合
- `--sdk-url` 也要求 `stream-json`

所以把它理解成“随便开个 JSON 模式”会很危险。

### transcript 落盘主干和后台注册表不是同一本账

可恢复会话的 transcript 主干在：

- `~/.claude/projects/<sanitize(cwd)>/<session-id>.jsonl`

后台会话注册表主干在：

- `~/.claude/sessions/<pid>.json`

前者回答“怎么恢复这条工作历史”，后者回答“现在有哪些后台会话活着”。

### print 恢复比交互恢复更硬

在 print 路径里：

- `--resume` 需要合法 session ID
- `--resume-session-at`、`--rewind-files` 必须依赖 `--resume`
- `--no-session-persistence` 只能用于 print

这说明自动化恢复的合同比交互 picker 更窄、更明确。

## 最容易误用的点

### 把所有自动化入口当成一回事

`-p`、`--bg`、`--sdk-url` 解决的是不同问题。

### 把后台会话注册表当 transcript 存储

这会直接导致你误判恢复机制和日志机制。

### 以为 `/exit` 一定会结束后台任务

在 `claude --bg` 会话里，它会优先 detach，而不是 kill。

### 以为 print 路径和交互路径有同一套可见命令面

print 模式会跳过整套子命令注册，重点是 prompt + structured I/O，不是 REPL 菜单面。

## 稳定面与条件面

### 稳定主线

- `claude -p/--print`
- `--output-format`
- `--input-format`
- `--continue`
- `--resume`
- `--fork-session`
- `--max-budget-usd`
- `--bare`

### 条件公开或隐藏集成面

- `ps`
- `logs`
- `attach`
- `kill`
- `--bg` / `--background`
- `--sdk-url`
- `--max-turns`
- `--resume-session-at`
- `--rewind-files`

### 内部、托管或不宜写成普通主线

- `daemon`
- `--daemon-worker`
- `environment-runner`
- `self-hosted-runner`

## 用户策略

更稳的自动化路径通常是：

1. 只要你的目标是一次性输出或脚本化，先从 `claude -p/--print` 开始。
2. 需要机器消费时，再决定 `text`、`json` 还是 `stream-json`，不要默认全上 `stream-json`。
3. 需要长任务脱离当前终端时，再考虑 `--bg`，不要把它和 print 混用成同一种模式。
4. 需要宿主协议流时，用 `--sdk-url`，并把它视为集成接口而不是普通命令行参数。
5. 想恢复旧自动化会话时，先分清自己要找的是 transcript，还是活着的后台会话。
