# 上下文接入、附件与提示编译专题

## 用户目标

不是“把更多字塞进 prompt”，而是把真正有价值的事实稳定送进 Claude Code：

- 当前任务到底要做什么。
- 哪些文件、图片、选区和外部文本是事实载体。
- 哪些目录真的属于当前工作面。
- 哪些系统回写信息是补充证据，而不是新的主任务。

## 第一性原理

Claude Code 首先是输入编译器，其次才是回答器。

它先判断输入的制度身份，再决定这些内容以什么形式进入系统：

- 用户主任务。
- 斜杠命令。
- 附件与图片。
- IDE 选区。
- 本地命令回流。
- hooks 注入。

所以“补上下文”这件事，本质上不是润色 prompt，而是选择正确输入通道。

更进一步看，真正进入系统的是两层编译结果：

1. 固定前缀：`CLAUDE.md`、日期、git 状态等系统上下文。
2. 本轮消息图：`user / assistant / attachment / system`。

只有被编进 system prompt、`isMeta` user message 或 `tool_result` 的内容，才算真正进入模型。

## 正确入口

### 普通提示词

适合声明任务目标、成功标准、限制条件。

更稳的写法不是长篇背景铺陈，而是：

- 目标是什么。
- 约束是什么。
- 哪些事实已经通过附件、文件、IDE 选区或工作目录提供。

### 附件、图片与大块粘贴

适合承载事实本体，而不是让你把这些内容重新口述一遍。

更具体地说：

- 文件内容、日志片段、协议文本，优先作为附件或粘贴块。
- 图像信息走图像输入链，而不是转写成含混描述。
- 多行文本是证据，提示词是任务说明。

### IDE 连接与选区

当问题强依赖当前编辑现场时，应优先用 IDE 集成和选区，而不是手工描述“我现在看到哪里”。

这条路径能让 Claude Code 直接获得：

- 当前编辑器状态。
- 选中区域。
- 跳转与差异查看能力。

### `/files` 与 `/context`

它们不解决“怎么补”，而解决“我现在到底已经补进去了多少”。

更稳的用法是：

- 先用 `/files` 看共享事实面。
- 再用 `/context` 看上下文压力与组成。
- 决定是继续塞信息，还是先压缩或外化。

### `/add-dir`

当你需要把新的目录正式纳入当前工作面时，应该用 `/add-dir`，而不是只做一个 shell 层面的 `cd`。

`/add-dir` 真正改变的是：

- session permission context
- sandbox 可访问目录集合
- 当前工作面对该目录的制度承认

### hooks 与本地命令回流

这两条都是补充通道，但地位不同：

- hooks 适合在既有事件点注入结构化补充上下文。
- 本地命令回流更多是系统生成的 caveat 信息，不应与用户主任务混成一条自然语言。

更稳的理解是：

- hook 输出可能被重新编译成模型可见的 `<system-reminder>`。
- 本地 bash / local slash 输出多数先落会话，通常到下一次真正 query 才进入编译链。

## 隐藏的运行时合同

### 输入先分流，再进模型

`processUserInput` 先判断输入种类，说明 Claude Code 不会把所有输入都平铺成一段文本。

而 prompt 型 slash 也不会把 `/foo` 原样送进模型。真正进入模型的是：

- 命令元信息
- 技能正文
- 可编译附件
- 临时权限约束下的后续查询

### 附件不是附庸，而是输入对象

附件、图片、粘贴块和 IDE 选区都被单独建模，这意味着它们在系统里拥有比“文字描述”更稳定的地位。

但要注意几条边界：

- IDE“选中”主要送截断后的选中文本。
- 仅“打开文件”更多只提供文件名与相关发现线索。
- `/context` 本身不注入上下文，它只是按真实 API 视图重算当前消耗。

### `/add-dir` 同时改权限面和沙箱面

它不只是更新一个可见目录列表，还会刷新 sandbox 配置，因此这是“工作面扩张”而不是“路径切换”。

它也不会立刻把整个目录内容塞进模型；它改变的是后续可发现范围。

### shell 回流有明确 caveat

系统会标记本地命令生成的消息，提醒模型不要把它们误当成新的用户意图。这解释了为什么“让 shell 打印一些东西再继续”与“用户重新提需求”不是一回事。

## 最容易误用的点

### 把所有背景写成长 prompt

这会把“任务说明”和“事实载体”混在一起，也更容易挤爆上下文。

### 以为 `cd` 等于新目录已进入工作面

在 Claude Code 里，路径、权限、sandbox 和共享上下文不是同一件事。

### 以为 `/context` 或 `/add-dir` 本身就会“喂上下文”

前者更多是在看账，后者更多是在改未来可发现范围。

### 以为 IDE 选区总会自动进入系统

它依赖 IDE 已连接、当前有可用状态、以及宿主正确把这些信息送进会话。

### 把 hook 输出当成主任务

更稳的理解是：hook 输出是补充证据或治理插针，不是新的主线需求。

## 稳定面与条件面

### 稳定主线

- 普通提示词
- 附件、粘贴块、图片输入
- IDE 集成与选区
- `/files`
- `/context`
- `/add-dir`

### 条件或强依赖环境

- hook 注入的额外上下文
- 特定宿主下的浏览器、语音、远端上下文
- 因文件触达激活的动态技能与条件规则
- 本地命令输出在下一次 query 前的延迟进入

## 用户策略

更稳的补上下文顺序通常是：

1. 用提示词声明任务。
2. 用附件、文件、图片、选区承载事实。
3. 用 `/files` 和 `/context` 确认共享事实面。
4. 需要扩大工作面时，用 `/add-dir` 正式扩张。
5. 上下文已重时，优先压缩或外化，而不是继续堆文字。
