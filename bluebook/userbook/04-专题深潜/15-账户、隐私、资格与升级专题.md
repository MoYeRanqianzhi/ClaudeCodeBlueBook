# 账户、隐私、资格与升级专题

## 用户目标

不是只会“登上去”或“退出去”，而是看清 Claude Code 账户面的真实边界：

- 我现在到底是 Claude 订阅身份，还是 API 计费身份
- 哪些入口只对 `claude.ai` 消费者订阅开放
- 隐私、资格、额度升级为什么不是一套统一菜单
- hit rate limit 之后，系统到底会给出什么分流

## 第一性原理

Claude Code 的账户面至少包含四个不同维度：

- 认证方式
- 产品面归属
- 订阅 / 计费层级
- 组织与策略限制

所以 `/login`、`/privacy-settings`、`/passes`、`/upgrade` 不能被写成一个平面的“账户设置页”。

更稳的拆法是：

- 身份切换：`/login`、`/logout`
- 消费者产品设置：`/privacy-settings`
- 资格与激励：`/passes`
- 升级与限流分流：`/upgrade`、`rate-limit-options`

## 正确入口

### `/login`

这是稳定公开的身份入口。

它的公开语义会随着当前认证状态变化：

- 没有 Anthropic API key auth 时：`Sign in with your Anthropic account`
- 已经有 Anthropic API key auth 时：`Switch Anthropic accounts`

更关键的是，它并不只是“拿到 token”。

登录成功后，源码会继续刷新和重置：

- cost state
- remotely managed settings
- policy limits
- GrowthBook / 用户缓存
- trusted device enrolment
- auth-dependent MCP / app state

所以 `/login` 的真实作用是“重建当前会话的身份现场”。

### `/logout`

这也是稳定公开主线。

它不是简单删 token，而是按顺序做一整套清理：

- 先 flush telemetry
- 删除 API key / secure storage
- 清 trusted device、Grove、policy limits、remote managed settings 等缓存
- 清 `oauthAccount`
- 最后退出进程

这说明 `/logout` 更像一次身份上下文清场，而不是局部配置切换。

### `/privacy-settings`

这条线必须明确写成产品面依赖能力。

源码能稳定确认：

- 它只对 `claude.ai` 消费者订阅开放，也就是 `Pro` / `Max`
- 还要满足 Grove 资格
- 如果无资格、API 失败或配置拉取失败，会直接回退到网页链接

因此它不能被写成“所有 Anthropic 用户都能在 CLI 里管理的隐私设置”。

还要再补一条边界：

- 它管理的是 consumer 侧的 Grove / “Help improve Claude”
- 不是全局 telemetry kill switch

### `/passes`

这是资格驱动的 guest passes / referral 入口。

它不只是“某个营销页面”。

源码里它至少依赖：

- `claude.ai` subscriber
- `Max`
- `organizationUuid`
- 本地 eligibility cache

而且命令显隐本身就跟 cache 绑定：

- 没有 cache 时，即使真实有资格，也可能先隐藏

所以“我现在看不到 `/passes`”不等于“我绝对没有资格”。

### `/upgrade`

这条线属于产品面依赖，但公开合同清晰。

源码能确认：

- 只面向 `claude-ai`
- Enterprise 被直接排除
- 它会打开 Max 升级页
- 然后要求重新登录，以刷新当前身份

如果用户已经是最高 `Max 20x`，它不会继续升级，而是提示切换到 API usage-billed 账户。

### `rate-limit-options`

这条命令必须明确标成隐藏分流入口。

它虽然注册在命令系统里，但：

- `isHidden: true`
- 只在 `isClaudeAISubscriber()` 时启用
- 主要在 hit rate limit 时由系统内部唤起

它的作用不是“常规账户菜单”，而是限流后帮你分流：

- 升级套餐
- 切 extra usage
- 或者停下来等待 reset

## 隐藏的运行时合同

### 3P provider 会直接改写命令面

`commands.ts` 里 `login/logout` 只在 `!isUsing3PServices()` 时注册。

这意味着：

- Bedrock
- Vertex
- Foundry

这类模式下，`/login`、`/logout` 本身就不会出现。

### “有 API key”不等于“有 claude.ai 订阅能力”

一旦用户走的是外部 API key、helper 或 3P provider，很多消费者订阅语义就会失效。

受影响的典型入口包括：

- `/privacy-settings`
- `/passes`
- `/upgrade`
- 部分 rate-limit 分流

### `/privacy-settings` 的网页回退不是次要细节

它说明这条能力不是纯本地状态页，而是强依赖远端资格与 API 可用性的产品面能力。

### telemetry 抑制面和 `/privacy-settings` 不是一回事

真正稳定的全局遥测抑制面在环境变量：

- `DISABLE_TELEMETRY`
- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`

`/privacy-settings` 处理的是消费者产品隐私选项，不应被写成这两类环境开关的 UI 替身。

### `/passes` 的显隐依赖缓存

所以它更像“资格被本地缓存后显露出来的入口”，而不是永远可见的固定菜单项。

### `/upgrade` 本质上是“升级后重新鉴权”

打开网页只是前半段，后半段是重新 `/login` 让新身份进入当前会话。

## 最容易误用的点

### 把 `/login` 当所有 provider 的统一入口

它只覆盖 Anthropic 自身账户面，不覆盖 3P provider 模式。

### 把 `/privacy-settings` 写成通用隐私命令

它只覆盖 `claude.ai` 消费者订阅，并且还受 Grove 资格影响。

### 把 `/privacy-settings` 当全局 telemetry 开关

真正的全局抑制面是环境变量，不是这个消费者产品设置页。

### 看到 `/passes` 不显示，就断言自己没有资格

源码明确把 cache 作为显隐条件的一部分。

### 把 `/upgrade` 理解成任意套餐都能继续往上叠

Enterprise 不走这条线；已经在 `Max 20x` 顶层时，也不会继续升级。

### 把 `rate-limit-options` 当公开主菜单

它本质上是隐藏分流器，不是用户日常主动调用的账户页。

## 稳定面与条件面

### 稳定主线

- `/login`
- `/logout`

### 产品面依赖或资格依赖

- `/privacy-settings`
- `/passes`
- `/upgrade`

### 隐藏或不应写成公开主线

- `rate-limit-options`

## 用户策略

更稳的账户操作路径通常是：

1. 先分清自己当前是 `claude.ai` 订阅身份，还是 API / 3P provider 身份。
2. 需要重建当前会话的身份现场时，用 `/login`；需要彻底清场时，用 `/logout`。
3. 想改隐私设置前，先默认它是消费者产品能力，而不是所有账户都可见。
4. 看到 `/passes` 时，把它理解为“资格与缓存都满足后显露出来的入口”。
5. hit rate limit 以后，先分清是升级、extra usage 还是等待 reset，不要把所有限流都归结成同一种解决方案。
