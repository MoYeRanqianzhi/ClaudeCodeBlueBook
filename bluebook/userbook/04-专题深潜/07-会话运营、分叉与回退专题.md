# 会话运营、分叉与回退专题

## 第一性问题

Claude Code 不只是让你“在一个对话里继续说话”。

如果你当前更关心的是“怎么找回过去的会话、过去的 prompt 和过去的工作对象”，应转去读：

如果你当前更关心的是“长任务怎样靠 `/memory`、`/compact`、`/resume` 继续续上”，而不是当前工作对象怎样被复制、命名、导出、分叉、回退和清空，应转去读：

这一页处理的是长文专题，不是索引速查，所以重点不是“命令列表”，而是“为什么这些命令共同构成会话运营层”。

默认只讨论：

- 普通用户
- 默认宿主
- 当前会话对象上的稳定运营动作

因此 `/session`、`/remote-control` 这类宿主/接续入口不在这页主线，`/tag` 也仍只应作为 ant-only / 内部边界提醒出现。

它还假设：

- 会话本身是工作对象。
- 这个工作对象需要被命名、复制、导出、分叉、回退和清空。
- 这些动作既影响对话时间线，也可能影响代码恢复面。

因此 `/copy`、`/rename`、`/export`、`/branch`、`/rewind`、`/clear` 不应被当成杂项命令。它们解决的是“如何运营当前工作对象”。

## 先分清三种动作

### 1. 把当前工作对象变得可识别

这里的代表入口是 `/rename``。

它让一次会话从临时聊天升级成可恢复、可辨认、可跨前端继续的工作对象。

### 2. 把当前工作对象变得可交付

这里的代表入口是 `/copy` 和 `/export`。

- `/copy` 解决“如何把当前结果快速带出终端”。
- `/export` 解决“如何把整个会话正式外化到文件系统”。

### 3. 改写当前时间线，或切出新的工作面

这里的代表入口是 `/branch`、`/rewind`、`/clear`。

- `/branch` 解决“从当前点分出一条新时间线”。
- `/rewind` 解决“回到更早的对话/代码状态前”。
- `/clear` 解决“生成一个新的空工作面”。

前两者仍在改写当前会话对象的时间线；`/clear` 则是切到新的空 session/frontstage，不属于时间线分叉或回退。

## `/copy` 不是简单剪贴板命令

从实现看，`/copy` 会向后回看最近 assistant 输出，支持：

- 复制整段回复。
- 只复制某个代码块。
- 记住“始终复制完整回复”的偏好。
- 在剪贴板之外，再写一份临时文件作为兜底。

这说明它的目标不是“发一个 OSC 52 就结束”，而是：

- 尽量把结果带出当前会话。
- 在终端不支持剪贴板时仍保留可交付出口。

最常见误用：

- 以为它只能复制最后一次完整回复。
- 以为剪贴板失败就等于命令没价值。

## `/export` 不是 `/copy` 的放大版

`/export` 的目标更正式：把整个会话渲染成可保存、可交接、可归档的文本对象。

它和 `/copy` 的关系不是“一个大，一个小”，而是：

- `/copy` 面向快速搬运当前结果。
- `/export` 面向完整外化当前工作现场。

最常见误用：

- 用 `/export` 代替 `/compact` 或 `/resume`。
- 用 `/copy` 代替正式交付和归档。

## `/branch` 的核心不是“另开一个聊天”

从 `branch.ts` 看，它会复制当前 transcript 的主线消息，把当前会话改造成一个新的 fork，并保留 `forkedFrom` 追踪关系。

它还会：

- 为 fork 生成新的 session id。
- 自动保存 branch 名称。
- 处理同名 branch 的标题冲突。
- 在成功后直接把当前工作切进新的 branch。

这说明 `/branch` 的核心价值是：

- 保留原时间线。
- 在当前点创建一个可恢复、可命名、可继续的替代分支。

最常见误用：

- 把它当 Git branch。
- 以为它只是 UI 快捷方式，不会改 session identity。

## `/rewind` 的核心不是“撤销上一句”

`/rewind` 表面上只是本地命令，但它真正做的是打开消息选择器，让用户从某个历史点之前恢复代码和/或对话。

`MessageSelector.tsx` 透露出两个关键事实：

- 它可以恢复代码、对话，或两者一起恢复。
- 它并不承诺恢复所有外部改动；手工编辑或 bash 改动有明确边界提示。

也就是说，`/rewind` 不是 Ctrl+Z，而是一个受约束的时间线恢复入口。

最常见误用：

- 把它当 Git reset。
- 以为所有文件变化都会无条件被带回。

## `/clear` 不是轻量版 `/compact`

`/clear` 直接清空当前会话，并生成新的 session id。

更关键的是，源码显示它会同时处理：

- session end hooks 与 session start hooks。
- cache eviction hint。
- plan slug 清理。
- session metadata 清理。
- MCP 状态重置。
- worktree state 重持久化。
- 前景任务清理与部分后台任务保留。

这说明 `/clear` 的职责是“切出一个新的空工作面”，而不是“把当前长对话整理一下”。

最常见误用：

- 把 `/clear` 当成上下文不够时的第一选择。
- 用它代替 `/compact`，结果把原会话的连续性直接打断。

## `/tag` 只应作为边界提醒出现

如果一定要在会话运营专题里提到 `/tag`，也只能提醒一件事：

- 它当前是 ant-only / 内部能力，不应继续写成普通用户稳定主线。

## 用户层结论

从使用角度看，最稳的会话运营路径通常是：

1. 用 `/rename` 让当前会话成为可识别对象。
2. 需要快速搬运局部结果时用 `/copy`。
3. 需要正式外化整个会话时用 `/export`。
4. 需要试验替代路径时用 `/branch`，不要污染原时间线。
5. 需要恢复较早状态时用 `/rewind`，而不是把它当 shell 撤销。
6. 只有在真的要切出新空工作面时才用 `/clear`。
