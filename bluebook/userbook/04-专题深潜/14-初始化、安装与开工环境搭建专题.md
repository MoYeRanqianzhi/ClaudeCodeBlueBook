# 初始化、安装与开工环境搭建专题

## 用户目标

不是“知道几个命令名”，而是把 Claude Code 的开工面搭好：

- 给仓库建立最小但有用的 `CLAUDE.md`
- 安装或更新 Claude Code 本体
- 把终端输入环境调顺
- 把 GitHub / Slack 这类外部入口接好
- 给账户准备长期可用的认证方式

## 第一性原理

Claude Code 的“开工准备”不是一个动作，而是三层不同对象：

- 仓库对象：`/init`
- 宿主对象：`claude install`、`/terminal-setup`
- 外部平台对象：`/install-github-app`、`/install-slack-app`、`setup-token`

如果把这三层混写，用户最容易犯的错就是：

- 以为所有 setup 都会改仓库
- 以为所有安装都在改本机
- 以为所有集成都只是“打开一个网页”

如果你还想分清 `setup-token`、`claude doctor` 这类会外 root command 与 `/doctor`、`/status` 这类会内面板的层级差异，应继续看 `19-会外控制台与会内面板专题.md`。

## 正确入口

### `/init`

这是仓库初始化主线。

稳定能确认的公开合同是：

- 它会为仓库生成或改进 `CLAUDE.md`
- 核心目标是写出“Claude 否则会犯错的信息”，不是写一份泛泛的项目说明书

当前源码里还存在一条更丰富的 `NEW_INIT` 路径，会把问题拆成：

- 项目级 `CLAUDE.md`
- 个人级 `CLAUDE.local.md`
- skills / hooks
- 代码库访谈与提案确认

但这条 richer flow 受 `NEW_INIT` / `CLAUDE_CODE_NEW_INIT` 控制，不能把它写成所有构建都稳定可见的主线。

### `/terminal-setup`

这是终端输入摩擦治理入口，不是仓库初始化入口。

它会根据终端环境做不同处理：

- Apple Terminal：安装 Option+Enter / visual bell 相关设置
- VSCode / Cursor / Windsurf / Zed / Alacritty：安装 Shift+Enter 绑定
- iTerm2 / WezTerm / Ghostty / Kitty 等原生支持 CSI u 的终端：直接隐藏或提示“无需设置”

所以它解决的是“多行输入怎么更顺”，不是“项目规范怎么更顺”。

### `claude install [target]`

这是 CLI 根命令层的安装入口，不是斜杠命令。

它的职责是安装 Claude Code native build，本质上属于宿主环境准备。

适合：

- 新机器装环境
- 切到 `stable`、`latest` 或特定版本
- 修复本机安装态

### `/doctor`

这是稳定公开的开工前总诊断入口。

它不只是“看版本”，而是聚合检查：

- 安装形态
- PATH / alias
- settings
- sandbox
- MCP
- keybindings
- context 体积

更准确地说，它是“宿主健康检查”，不是项目代码调试器。

### `setup-token`

这也是 CLI 根命令层入口，不是斜杠命令。

它会引导用户创建长期有效的 OAuth token，并且 help 文本直接写明：

- 这是长生命周期 token setup
- 需要 Claude subscription

所以它不是 provider-neutral 的通用鉴权，也不是普通 `/login` 的替代。

更稳的理解是：

- 它提供的是长期 token setup
- 不是把完整账户产品面都切换成“已登录”

### `/install-github-app`

这是 GitHub 集成安装入口。

它不是“给 Claude 登录 GitHub”，而是引导你为仓库接上 Claude GitHub Actions / App 工作流。

源码能确认它会检查：

- `gh` 是否安装
- `gh` 是否已认证
- `repo`、`workflow` scopes 是否齐全
- 当前仓库是否可识别
- workflow 文件和 secrets 是否已经存在
- 目标仓库是否有 admin 权限

这说明它是“GitHub 工作流接入器”，不是普通 OAuth 向导。

### `/install-slack-app`

这是最轻量的外部入口。

它的实现非常窄：

- 记录一次点击
- 打开 Slack app marketplace 页面

因此它更像“外部应用跳转入口”，不是深度配置器。

### `/init-verifiers`

这条线必须单独标边界。

它在源码里确实存在，但被放进 `INTERNAL_ONLY_COMMANDS`，不应写成稳定公开主线。

更准确的定位是：

- 内部 / ant-only 的 verifier-skill 生成辅助能力

## 隐藏的运行时合同

### `/init` 的稳定底座和 richer flow 不是一回事

稳定能写死的是“初始化 `CLAUDE.md` 这件事”。

不能写死的是“所有用户都能看到项目/个人双文件、skills/hooks 提案流”。

### `/terminal-setup` 改的是宿主，不是仓库

它会直接触到：

- 本机终端偏好
- IDE keybindings
- 全局配置

所以它不属于项目目录结构的一部分。

### 远程 IDE / SSH 会改变 `/terminal-setup` 的处理方式

源码明确区分 VSCode Remote SSH 场景：

- keybinding 必须装在本地机器，而不是远端 server

这说明“我在远端仓库里敲 `/terminal-setup`”不等于它就能直接改好本地终端。

### `/install-github-app` 是强前提型命令

只要 `gh` 缺失、未认证、scope 不够、repo 权限不足、workflow 冲突，流程就会停在检查或报错层。

### `/install-slack-app` 不是非交互入口

它明确 `supportsNonInteractive: false`，因此不应被写进自动化主线。

## 最容易误用的点

### 把 `/init` 写成“生成完整知识库”

从提示词设计看，它追求的是“最小但关键”的仓库指导，不是堆满常识的文档仓。

### 把 richer `NEW_INIT` 当稳定公开合同

这会直接污染正文边界。

### 把 `/init-verifiers` 当公开功能

它属于内部能力，不该混入普通用户主线。

### 把 `/install-github-app` 理解成 GitHub 登录

它真正解决的是 GitHub Actions / App 落地，不是账号切换。

### 把 `/terminal-setup` 当仓库级配置

它首先是本机输入体验配置。

## 稳定面与条件面

### 稳定主线

- `/init`
- `/terminal-setup`
- `claude install`
- `/doctor`
- `setup-token`
- `/install-github-app`
- `/install-slack-app`

### 条件依赖或环境强相关

- `NEW_INIT` richer flow
- `/terminal-setup` 的终端可见性与具体效果
- `/install-github-app` 对 `gh`、scopes、repo admin 的依赖
- `/install-slack-app` 需要交互式浏览器跳转

### 内部或不应写成公开主线

- `/init-verifiers`

## 用户策略

更稳的开工准备路径通常是：

1. 先决定你是在准备仓库、准备宿主，还是准备外部平台入口。
2. 仓库层优先用 `/init`，并把 richer flow 当“可能出现的增强版”，不要预设它必在。
3. 宿主层优先解决 `claude install` 和 `/terminal-setup`，避免一开始就被输入摩擦拖慢。
4. 需要把结果接进 GitHub 工作流时再跑 `/install-github-app`，并先确认 `gh`、scope、权限。
5. 只想获得长期 token 时用 `setup-token`，不要和仓库初始化或 GitHub/Slack 安装混成一次操作。
