# 远端前台运行、会话存在面与桥接后台分诊专题

## 用户目标

不是笼统地问“Claude Code 的 remote 到底是哪一种 remote”，而是更稳地处理这些真实场景：

- 为什么有时你面对的是一个正在前台运行的远端会话，有时你拿到的却只是一个可被别的前端继续消费的 session URL。
- 为什么有时本地 REPL 只是把远端 runtime 当成 `send/cancel` 交互壳，有时又会额外出现 bridge pill、bridge dialog、session URL、mirror 之类的后台覆盖层。
- 为什么同样都看起来像 “remote mode”，`direct connect`、`ssh remote`、`--remote`、assistant viewer、bridge / remote-control 实际在本地运行时里分属不同对象。

如果你的问题还停在更宽口径的层级：

- “多前端与远程总览是什么”
- “Desktop / Mobile / session continuation 是什么”

先读：

- [16-IDE、Desktop、Mobile 与会话接续专题.md](./16-IDE%E3%80%81Desktop%E3%80%81Mobile%20%E4%B8%8E%E4%BC%9A%E8%AF%9D%E6%8E%A5%E7%BB%AD%E4%B8%93%E9%A2%98.md)

这页处理的是更窄的一类误判：

- 同样都像 remote surface，但为什么本地 REPL 有时在托管一个前台 remote runtime，有时只是挂着一层 remote presence / bridge overlay。

## 第一性原理

“我现在在用远端能力”不是一个对象，而是至少沿着五条轴线分裂：

1. `Foreground Runtime`：
   当前前台真正负责 `sendMessage/cancelRequest` 的是谁。
2. `Shared Interaction Shell`：
   当前 REPL 顶层有没有把不同 transport 压进一个最小交互壳。
3. `Presence Ledger`：
   当前本地有没有一张可被 footer / `/session` / brief line 继续消费的 authoritative 存在账。
4. `Bridge Overlay`：
   当前桥是不是只是在本地 transcript 之外再挂一层连接态、URL、dialog、permission callback。
5. `Global Remote Bit`：
   当前系统是不是只需要一个 coarse remote behavior 开关，而不是完整 presence truth。

如果这五轴不先拆开，最容易写错的就是：

- 把 `activeRemote` 当成 remote truth。
- 把 `remoteSessionUrl` 写成所有 remote 壳共享的自然结果。
- 把 bridge overlay 写成第四种前台 runtime。
- 把 `getIsRemoteMode()` 写成 authoritative presence ledger。

## 隐藏的运行时合同

### foreground runtime 先回答的是“谁在前台跑 turn”

所以最稳的一句不是：

- 现在是 remote，就说明我已经拥有了一张远端存在面

而是：

- 先看谁拥有 `sendMessage/cancelRequest`
- 再看它有没有 authoritative presence ledger

### `activeRemote` 只是 shared interaction shell，不是 remote truth

它统一的是：

- `isRemoteMode`
- `sendMessage(...)`
- `cancelRequest()`
- `disconnect()`

不是：

- session URL
- footer ledger
- bridge overlay

### `remoteSessionUrl` 这种字段回答的是 presence，不是 transport

所以 direct connect / ssh remote 没写这张账，

并不等于：

- 它们只是“少做了几项 UI”

更像：

- 它们本来就不是这张 presence ledger 的 owner

### bridge 先是后台 overlay，再谈 gray runtime

它不是第四种前台 runtime，

而是：

- transcript mirror
- inbound prompt 注回
- 本地状态回写
- 若干连接态 / dialog / URL surface

组合出来的一层 overlay

## 对用户的策略结论

如果你的问题是：

- “同样都像 remote，为什么有时我面对的是前台会话，有时却只看到 URL、pill 或别的桥接状态？”

更稳的拆法是：

1. 先问当前是谁拥有 foreground runtime。
2. 再问当前有没有 authoritative presence ledger。
3. 再问 bridge 现在是不是只在本地挂了一层 overlay。
4. 最后再问系统是不是还额外保留了一层 coarse remote bit。

而不是一句：

- “remote 有时像会话，有时像桥，有时又像状态面板。”

因为这几句分别落在不同层。
