# 前端与远程专题

## 用户目标

在不同前端、不同设备、不同环境之间保持同一个工作对象，而不是把 Claude Code 锁死在单个终端窗口里。

如果你更关心稳定用户面上的 IDE / Desktop / Mobile / session viewer 分工，优先继续读 [16-IDE、Desktop、Mobile 与会话接续专题.md](./16-IDE%E3%80%81Desktop%E3%80%81Mobile%20%E4%B8%8E%E4%BC%9A%E8%AF%9D%E6%8E%A5%E7%BB%AD%E4%B8%93%E9%A2%98.md)。本页保留的是更宽口径的前端与远程总览，条件面会更多。

如果你卡住的是“远端前端里的输入、审批与会话接续为什么看起来像一件事，落到本地却分成不同链路”，继续读 [20-远端输入注入、桥接审批与会话接续专题.md](./20-%E8%BF%9C%E7%AB%AF%E8%BE%93%E5%85%A5%E6%B3%A8%E5%85%A5%E3%80%81%E6%A1%A5%E6%8E%A5%E5%AE%A1%E6%89%B9%E4%B8%8E%E4%BC%9A%E8%AF%9D%E6%8E%A5%E7%BB%AD%E4%B8%93%E9%A2%98.md)。

## 正确入口

### IDE

`/ide` 不是“打开某个编辑器”，而是在建立 IDE 连接关系。

从 `ide.tsx` 可以看到它关注：

- 检测可用 IDE。
- 区分可用与不可用实例。
- 自动连接。
- 动态 MCP 配置。
- 当前 worktree 会话。

这意味着 IDE 入口是“连接开发环境”，不是单纯“跳转编辑器”。

### session / remote

`/session` 会展示 remote session URL 和 QR code。它不是普通信息页，而是多前端接续的桥面。

具体实现上：

- 如果当前不在 remote mode，会直接提示“先用 `claude --remote` 启动”。
- 如果 remote session URL 存在，会生成 QR code；即便 QR 生成失败，也仍把 URL 保留为主要接续入口。

这说明 `/session` 的主对象不是“二维码”，而是“可被别的前端继续消费的连接面”。

### remote-control / bridge

`/remote-control` 的真实语义是桥接连接控制，而不是单纯“开启远程”：

- 若已经连接，会进入 disconnect / continue 对话。
- 若尚未连接，会先做 preflight。
- 还会受 remote callout、bridge prerequisites、版本与策略检查影响。

因此它更像“宿主和控制面的联结器”，而不是按钮开关。

### voice

`/voice` 的开启逻辑比表面复杂得多：

- 先检查 kill-switch 和 auth。
- 再检查录音环境。
- 再检查 voice stream 可用性。
- 再检查依赖与麦克风权限。
- 最后才写设置启用。

这说明语音在 Claude Code 里已被当成正式产品能力，而不是单个热键。

更进一步：

- `/voice` 不是纯 UI toggle，而是一条预检查链。
- 关闭 voice 很简单；开启 voice 则是逐层验证 auth、录音、依赖、权限、settings。

### chrome

`/chrome` 也不只是“浏览器彩蛋入口”，而是说明 Claude Code 在尝试把浏览器会话纳入正式前端/集成面。

### desktop / mobile / chrome

这些命令共同表明 Claude Code 正在把会话迁移和多前端接续产品化：

- 桌面端继续。
- 移动端接续。
- Chrome 内交互。

## 隐藏的运行时合同

### `/session`

它真正暴露的是“当前 remote 会话的接续地址”，不是状态面板。

### `/remote-control`

它控制的是 bridge 状态机，因此已连接和未连接时的行为完全不同。

### `/voice`

它开启的是一条预检查链，而不是纯设置翻转。

## 为什么 Claude Code 不用更简单方案

### 为什么 `/session` 不是简单打印一个 URL

因为它服务的是多前端接续，二维码、URL、remote mode 检查都属于同一个交付对象。

### 为什么 `/remote-control` 不只是一个 toggle

因为桥接连接涉及 entitlement、版本、策略、对话框和断开语义。

### 为什么 `/voice` 不只是快捷键开关

因为它依赖 auth、依赖、设备权限和远端语音流能力。

## 使用设计事实

### 1. `/session` 的核心是“把同一工作对象带到别的前端”

因此它展示 QR 和 URL，而不是堆叠状态字段。

### 2. `/remote-control` 的核心是“把本地 CLI 变成远程宿主”

它不是普通信息命令，而是桥接执行环境。

### 3. IDE 入口本质上是在接入开发现场

一旦 IDE 连接建立，Claude Code 拿到的是更丰富的上下文，而不只是“打开文件”的能力。

### 4. Voice / Chrome 都是条件前端

源码显示它们都受较强的环境、auth 和 gate 约束，因此不应写进无条件主线。

## 不该误用的邻近能力

### 不要把 `/session` 当作状态页

它实际上是远程接续入口。

### 不要把 `/voice` 当作单纯 UI 开关

它依赖 auth、依赖、系统权限和 feature gate。

### 不要把 IDE 连接当附带体验

它显著改变 Claude Code 能拿到的工作现场信息。

### 不要把 `/remote-control` 当成普通信息命令

它实际上会改 bridge 状态，而且既有连接中和未连接时的行为完全不同。

## 稳定面与灰度面

### 稳定面

- `/ide`
- `/desktop`
- `/mobile`
- 键位、主题、Vim 相关终端前端能力

### 灰度或条件面

- `/session`
- `/voice`
- `/chrome`
- `/remote-control`
- `/remote-env`
- 某些 web / CCR 相关路径

## 对用户的策略结论

如果你把 Claude Code 视为多前端工作对象，而不是“当前终端里的聊天框”，很多看似零散的命令会突然统一起来：

- `/ide` 是连接开发宿主。
- `/session` 是接续桥。
- `/voice` 是输入通道扩展。
- `/remote-control` 是执行环境与控制面的分离。
