# 状态、额度、模型与节奏运营专题

## 用户目标

不是“会几个命令”，而是让 Claude Code 在长任务里持续可用：

- 当前状态是否可信
- 环境是否健康
- 预算是否够
- 模型和 effort 是否匹配当前阶段
- 是否该加速、降速、压缩、暂停或切换节奏

## 第一性原理

Claude Code 运行失败，很多时候不是“模型不会”，而是：

- 宿主状态不对
- 环境失真
- 额度不够
- 模型/effort/fast mode 选择不匹配

所以 `/status`、`/doctor`、`/usage`、`/cost`、`/stats`、`/model`、`/effort`、`/fast`、`/statusline` 这些入口，本质上都在运营“任务推进节奏”。

如果你还想分清这些入口分别属于 `Settings` tab、独立诊断屏、session 调参命令还是预算分流入口，应继续看 `../05-控制面深挖/10-设置面板、诊断屏与运营命令：会内控制面的三层分工.md`。

更精确的分层是：

- 观测层：`/status`、`/usage`、`/doctor`、`/stats`
- 调参层：`/model`、`/effort`、`/fast`、`/statusline`
- 预算与限流层：`/cost`、rate limit、extra usage

## 正确入口

### `/status`

先确认当前到底站在哪个宿主状态里：

- 账户
- provider
- cwd
- 模型
- IDE / MCP / sandbox 摘要

它适合做开工前和接手旧会话时的第一检查。

### `/doctor`

适合当 `/status` 只能告诉你“现在不对劲”，但还不知道错在哪一层时。

它更像环境可信度诊断器，而不是代码调试器。

### `/usage`

适合在长任务和高成本任务开始前看清预算面，而不是撞到上限后才开始调整策略。

### `/cost` 与 `/stats`

这两者更偏运行中观测：

- 当前会话已经花了多少
- 当前会话以什么节奏在消耗与产出

它们适合回答“这个任务值得继续深挖，还是该收束”。

但不要把它们和 `/usage` 混成一件事：

- `/usage` 看额度窗口
- `/cost` 看本地会话估算
- `/stats` 看长期行为画像

### `/model`

不是简单换模型，而是切换整套能力面与预算面。

从源码看，模型切换还会联动：

- fast mode 兼容性
- extra usage 计费提示
- session override 与 base model 的区分

### `/effort` 与 `/fast`

这两者控制的不是同一维度：

- `effort` 更像思考投入度
- `fast` 更像吞吐策略

它们应该跟任务阶段一起调整，而不是固定不变。

### `/statusline`

它不直接改变策略，但会改变你如何低摩擦观察当前状态，因此更像“持续运营的显示层”。

更准确地说，它不是普通查询命令，而是一个 prompt 命令：会调用专门 agent 去写 `statusLine` 配置，把成本、上下文占用、rate limits、worktree 等信息持续投到终端状态栏。

## 隐藏的运行时合同

### `/status` 与 `/usage` 都是 Settings 面板的一部分

这说明它们不是临时输出，而是长期运行时运营面。

### `/doctor` 是独立诊断屏

这代表环境可信度被视为单独问题，不被塞进普通状态页里。

### `/model` 会自动处理部分节奏降级

例如：

- 新模型不支持 fast mode 时，系统会自动关掉 fast mode
- 某些模型会触发 extra usage 提示
- 某些 1M 上下文模型有额外访问门槛
- plan mode 还可能给当前会话单独覆写模型

### `/usage`、`/cost`、`/stats` 不是财务尾巴

它们会直接影响你是否继续深挖、是否压缩上下文、是否切换模型。

### rate limit 不是 429 之后才需要关心

限流窗口会预取、更新，并在“当前消耗速度相对时间进度过快”时提前给出 warning。

## 最容易误用的点

### 把 `/status` 当 about 页

它更接近运行时状态镜像。

### 把 `/doctor` 当安装出问题时才会用的命令

更稳的理解是：只要怀疑环境失真，就该先看它。

### 把 `/usage` 和 `/cost` 当事后账单

它们更应该进入任务前和任务中的决策。

### 把 `/model` 理解成“换个更强的”

实际变化往往是：

- 预算变了
- fast mode 是否兼容变了
- 可见能力和工作节奏也变了

### 把 `effort` 和 `fast` 当重复功能

它们治理的是不同维度。

### 把 `/statusline` 当永远可靠的状态源

它依赖配置、hooks 与 trust 条件；配置存在并不等于一定会亮。

## 稳定面与条件面

### 稳定主线

- `/status`
- `/doctor`
- `/usage`（稳定可见，但仍依赖 Claude AI 产品面与对应套餐 / provider 语境）
- `/model`
- `/effort`
- `/fast`
- `/statusline`

### 条件或与产品面强相关

- `/cost`
- `/stats`
- `extra usage`
- 1M context 访问
- 不同 provider / plan 下的 usage 展示与模型可见面
- rate limit 预警与 fast mode cooldown

## 用户策略

更稳的节奏通常是：

1. 开工前先看 `/status`
2. 感到异常时先跑 `/doctor`
3. 长任务前看 `/usage`
4. 任务进入重构、审查、验证阶段时再提高 `effort` 或切模型
5. 发现节奏失衡时看 `/cost` 与 `/stats`
6. 用 `/statusline` 保持低摩擦观测，而不是不断中断工作流
