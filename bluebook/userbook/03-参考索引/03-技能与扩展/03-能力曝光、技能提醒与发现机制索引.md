## 1. 五条不同的曝光链

| 曝光链 | 主要受众 | 核心内容 | 典型来源 | 最容易误判 |
| --- | --- | --- | --- | --- |
| `/skills` 菜单 | 用户 | 当前会话里某些来源类别的 prompt 技能菜单 | `SkillsMenu` + `context.options.commands` | 误写成技能总库 |
| `system/init.slash_commands` | remote / SDK 客户端 | 用户可见 slash commands 名称 | `inputs.commands` | 误写成模型 prompt |
| `system/init.skills` | remote / SDK 客户端 | remote 可见 skills 名称 | `getSlashCommandToolSkills()` | 误写成 `/skills` 菜单 |
| `skill_listing` | 模型 | broad inventory：当前可用 SkillTool 技能库存 | `getSkillToolCommands()` + `getMcpSkillCommands()` | 误写成 UI 列表 |
| `skill_discovery` | 模型 | 和当前任务相关的技能子集提醒 | 实验搜索 / prefetch | 误写成全量技能表 |

## 2. 三张最容易混淆的“技能表”

| 表 | 主要函数 / UI | 筛法 | 更准确的理解 |
| --- | --- | --- | --- |
| 用户菜单表 | `/skills` -> `SkillsMenu` | `prompt` 且 `loadedFrom in {skills, commands_DEPRECATED, plugin, mcp}` | 给人翻的菜单 |
| remote 元数据表 | `buildSystemInitMessage()` | `userInvocable !== false` 的 commands / skills 名称 | 给客户端渲染 picker 和 gate 的元数据 |
| 模型 broad inventory 表 | `skill_listing` | `getSkillToolCommands()` + `getMcpSkillCommands()`，并带去重/预算/搜索裁剪 | 给 Claude 的广义库存 |

## 3. 相关性提醒链

| 对象 | 作用 | 条件 | 对用户意味着什么 |
| --- | --- | --- | --- |
| `skill_discovery` | 每回合提醒和当前任务相关的技能子集 | `EXPERIMENTAL_SKILL_SEARCH` | Claude 不一定先看全量，而可能先看相关子集 |
| DiscoverSkills guidance | 告诉 Claude 何时继续做技能发现 | `EXPERIMENTAL_SKILL_SEARCH` 且工具开启 | 这是实验搜索链的一部分，不应写成默认稳定主线 |
| subagent discover guidance | 给子代理补同样的 framing | 子代理路径 + 搜索链开启 | 子代理不是简单复制主线程曝光面 |

## 4. UI 可见面 vs 模型可见面

| 对象 | 用户在 transcript 里看到什么 | 模型真正收到什么 |
| --- | --- | --- |
| `skill_listing` 初始批次 | 通常不显示 | 完整 `system-reminder` broad inventory |
| `skill_listing` 后续增量 | `N skills available` 这种短提示 | 带完整内容的 meta reminder |
| `skill_discovery` | `N relevant skills: ...` 的短行 | `Skills relevant to your task:` 的 meta reminder |

## 5. SkillTool 运行时硬门

| 边界 | 含义 | 常见误判 |
| --- | --- | --- |
| `disableModelInvocation` | SkillTool 运行时直接拒绝调用 | 误写成“只是建议模型别用” |
| `userInvocable` | 用户能否直接 `/skill-name` | 误写成“模型也不能用” |
| `loadedFrom === 'mcp'` 过滤 | SkillTool 只并入 MCP skills，不并 plain MCP prompts | 误写成“MCP prompt 猜到名字也能正常用” |

## 6. 搜索开启时的裁剪逻辑

| 场景 | 曝光行为 |
| --- | --- |
| 普通 broad inventory | local SkillTool commands + MCP skills |
| 搜索开启后的 subagent / turn-0 保护 | 优先裁到 bundled + MCP |
| 仍然过大时 | 再退到 bundled-only |

这部分应写成：

- 条件公开 / 实现可见的裁剪策略

而不是：

- 无条件长期承诺

## 7. 五个最有价值的判断问题

- 现在讨论的是用户菜单、remote 元数据，还是模型 inventory？
- 这条技能是“看不见”，还是“看得见但模型不能调”？
- 这是 broad inventory，还是 current-task relevance reminder？
- 这条行为是默认主线，还是实验搜索链下才成立？
- 这是 UI 上的短提示，还是模型实际上看到的完整 meta reminder？
