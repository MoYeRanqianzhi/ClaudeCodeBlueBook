## 1. 技能来源矩阵

| 来源类别 | 如何进入运行时 | `/skills` 是否默认可见 | SkillTool 是否可见 | 是否可能延迟激活 | 备注 |
| --- | --- | --- | --- | --- | --- |
| file-based `/skills` | `getSkillDirCommands(cwd)` | 通常是 | 通常是 | 可以，若是 `paths:` 条件技能或嵌套动态目录 | 受 settings source、policy、bare mode 影响 |
| legacy `commands`-as-skills | `getSkillDirCommands(cwd)` -> `commands_DEPRECATED` | 是 | 是 | 否 | 仍在运行时链中，但不是推荐长期心智 |
| bundled skills | `getBundledSkills()` | 否 | 是，若未 `disableModelInvocation` | 否 | 官方工作流对象，不等于 `/skills` 菜单 |
| built-in plugin skills | `getBuiltinPluginSkillCommands()` | 否 | 通常是 | 否 | 当前实现按 `bundled` 身份处理，更像 provenance 细节 |
| plugin skills | `getPluginSkills()` | 是 | 是，若未 `disableModelInvocation` | 否 | 前提是插件已启用 |
| MCP skills | `AppState.mcp.commands` + `getMcpSkillCommands()` | 消费侧有 mcp 分支 | 是，若 feature 开且未 `disableModelInvocation` | 取决于 MCP server 连接与 prompts | 不走 `getCommands()` 主装配链 |

## 2. 四张不同的“技能表”

| 投影 | 主要入口 | 核心筛法 | 回答的问题 |
| --- | --- | --- | --- |
| 来源投影 | `getCommands()` / `getSkills()` / `getDynamicSkills()` | 多路装配 + `availability`/`isEnabled()` 过滤 | 运行时有哪些技能对象被装进命令链 |
| 用户投影 | `/skills` -> `SkillsMenu` | `prompt` 且 `loadedFrom in {skills, commands_DEPRECATED, plugin, mcp}` | 用户现在在菜单里能翻到哪些技能 |
| 模型投影 | SkillTool / `getSkillToolCommands()` / `getMcpSkillCommands()` | `prompt`、`!disableModelInvocation`、`source != builtin` 等 | Claude 现在能自动调用哪些技能 |
| 激活投影 | `conditionalSkills` / `dynamicSkills` | 触碰文件、命中 `paths:`、发现嵌套目录 | 还有哪些技能暂时潜伏，未来可能出现 |

## 3. 三个最关键的边界字段

| 字段 | 控制什么 | 不控制什么 | 最容易误判 |
| --- | --- | --- | --- |
| `userInvocable` | 用户能否直接 `/skill-name` | 模型能否通过 SkillTool 调用 | 误写成“技能不存在” |
| `disableModelInvocation` | 模型能否自动调用 | 用户能否手动 `/skill-name` | 误写成“用户也不能用” |
| `paths` | 技能是否先潜伏、命中路径后再激活 | `/skills` 菜单是否永远显示 | 误写成普通隐藏技能 |

## 4. 动态触发矩阵

| 触发方式 | 发生了什么 | 对用户意味着什么 |
| --- | --- | --- |
| 触碰匹配 `paths:` 的文件 | skill 从 `conditionalSkills` 移到 `dynamicSkills` | 某些技能会在会话中途出现 |
| 触碰 cwd 下更深层目录的文件 | 系统向上发现嵌套 `.claude/skills` 并动态加载 | 靠近文件的局部技能可以后出现、且优先级更高 |
| bare mode | 跳过常规 user/project/managed 发现，只保留显式 `--add-dir` 路径和另一条 bundled 注册链 | 技能面被主动收窄，不是“同一世界的纯净版” |
| plugin-only policy / skillsLocked | 用户和项目技能链、legacy commands-as-skills、动态发现会被压缩 | “为什么目录里有技能却不出现”可能是策略，不是加载 bug |

## 5. 五个最有价值的判断问题

- 这条技能是来源问题，还是投影问题？也就是“没加载”还是“加载了但不在这张表里”。
- 它是用户不能直打，还是模型不能自动调？先区分 `userInvocable` 和 `disableModelInvocation`。
- 它是不是 bundled / built-in plugin skill，所以不该拿 `/skills` 当唯一证据。
- 它是不是 `paths:` 条件技能，或者来自尚未触碰到的嵌套 `.claude/skills`。
- 当前是不是 bare mode、plugin-only policy、project settings disabled，把发现链直接收窄了。

## 6. 最容易混淆的对照

| 容易混淆 | 真正差异 |
| --- | --- |
| `/skills` vs SkillTool | 前者是用户菜单投影，后者是模型可调用投影 |
| bundled skills vs `/skills` | bundled 更偏官方工作流对象，默认不出现在 `/skills` |
| `userInvocable: false` vs 不存在 | 前者是模型面有、用户直调面没有 |
| `disableModelInvocation: true` vs 隐藏 | 前者主要影响模型集合，不必然影响用户直打 |
| dynamic skills vs 插件安装 | 前者可能只因文件触达而出现，不一定发生安装动作 |
| MCP skills vs 本地 skill dir | 前者是外部 server 线程，不走同一条本地装配链 |
