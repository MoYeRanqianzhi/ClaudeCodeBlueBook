# 任务到入口速查矩阵

下表不是按源码模块排，而是按用户任务排。目的是帮助你在真实工作中快速选入口。
它只负责把任务路由到更低协调成本的正式入口，不负责替正文签 truth 或直接给 verdict。
如果任务本身是 continuity 问题，也应先把 `/compact / /resume / /memory` 读成同一工作对象在时间轴上的动作，而不是三个并列功能桶。
更稳一点说，矩阵里的 continuity 任务不构成第五类任务簇；它们只是现有工作对象在时间轴上继续、压缩或恢复的不同动作。

## 这张矩阵的用法

### 看 first-answer order

先按 `问题分型 -> 工作对象 -> 控制面 -> 入口` 排，再用表选入口；如果顺序倒过来，这张矩阵也会退回“按标题碰运气”。

### 看主线

如果目标属于稳定列，就优先走稳定面，不要先上灰度面。

### 看边界

如果某项功能落在“条件公开但已成形”或“灰度更重”，就应在文档、团队约定和自动化中单独标注。

### 看替代路径

Claude Code 很多能力并不是“有或没有”，而是“有推荐路径和退化路径”。

## 用户层结论

Claude Code 的真正强项不是单个命令有多炫，而是：

- 几乎每种工作目标都存在一个更低协调成本的正式入口。
- 当正式入口不可用时，通常还能退化到 prompt、shell、手工工作流。
