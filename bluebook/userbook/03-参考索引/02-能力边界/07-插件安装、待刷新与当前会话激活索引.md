## 1. 插件的三层不同状态

| 层 | 核心问题 | 典型对象 | 最容易误判 |
| --- | --- | --- | --- |
| 意图层 | 用户想启用什么插件 | settings、enable/disable/update 标记 | 误写成已经生效 |
| 物化层 | plugin / marketplace 是否已落到磁盘缓存 | 安装结果、marketplace reconcile | 误写成当前 session 已刷新 |
| 激活层 | 当前 session 是否已换成新的插件结果 | `/reload-plugins`、refreshActivePlugins | 误写成安装流程本身 |

## 2. 四个相邻但不同的入口

| 入口 | 主要对象 | 更准确的理解 |
| --- | --- | --- |
| `claude plugin ...` | 配置 / 物化 | 会外安装与配置动作 |
| `/plugin` | 插件包管理 UI | 会内管理插件包与选项 |
| `/reload-plugins` | 当前 session 激活 | 把 pending plugin changes 应用到当前会话 |
| SDK `reload_plugins` | 控制面刷新请求 | 与 `/reload-plugins` 同语义、不同宿主接口 |

## 3. 为什么会不断看到 “Run /reload-plugins to apply”

| 场景 | 真实含义 |
| --- | --- |
| 安装成功 | 插件包已落盘，但当前 session 还没激活 |
| 启用 / 禁用 / 配置保存 | 意图层已改，当前 session 结果仍旧 |
| `needsRefresh` 通知 | 磁盘状态变了，但交互主线不自动刷新 |

## 4. `/reload-plugins` 真正会刷新什么

| 面 | 说明 |
| --- | --- |
| plugin commands / skills | 当前会话可见技能结果会刷新 |
| agent definitions | 当前会话可见 agent 结果会刷新 |
| hooks | plugin hooks 的注册结果会刷新 |
| plugin MCP / LSP | 插件携带的 MCP / LSP 结果会刷新 |

## 5. 自动刷新与手动刷新不是同一主线

| 路径 | 当前语义 |
| --- | --- |
| 交互 REPL | 只通知，用户手动跑 `/reload-plugins` |
| background install | 某些新安装路径会自动 refresh |
| headless / CCR | 某些路径会自动 refresh 或用专门安装/刷新流程 |
| SDK | 不发文本命令，走 control request |

## 6. 六个高价值判断问题

- 现在变的是意图层、物化层，还是当前 session 激活层？
- 这是插件包已经装好，还是当前 session 真的换栈完成？
- 我看到的是 pending / needsRefresh，还是已激活结果？
- 当前宿主是交互 REPL、SDK、headless，还是 remote 模式？
- 这里说的刷新是 `/reload-plugins`，还是自动 background refresh？
- 我是不是把 “Run /reload-plugins to apply” 当成了普通提示文案，而不是对象层级证据？
