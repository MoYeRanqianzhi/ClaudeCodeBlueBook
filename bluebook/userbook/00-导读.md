# 导读

## 先回答四个读者问题

大多数人打开这本手册时，真正想解决的通常不是“目录怎么分”，而是下面四个问题：

1. 我现在该从哪个入口开始。
2. Claude Code 到底能做哪些事。
3. 哪些能力是稳定主线，哪些只是条件公开或内部影子。
4. 为什么系统会推荐这条路径，而不是看起来相邻的另一条路径。

后面的结构都是围绕这四个问题组织的。

这里也要先压住一个常见误读：`continuity` 不是第五类读者问题；它只是同一工作对象在 `Continuation / continuation pricing / cleanup-before-resume` 上的共同时间轴。

## 这本手册和蓝皮书有什么不同

同在 `bluebook/` 体系里，蓝皮书主干更偏“源码结构研究”，而 `bluebook/userbook/` 更偏“面向使用者的系统说明书”。

- 蓝皮书关心模块怎样拆、为什么先进。
- 本手册关心用户何时该用哪条路径，以及每条路径的真实边界。

因此，这里采用四个固定视角：

1. 入口视角：用户从哪里进入系统。
2. 行动视角：模型到底能调用哪些执行面。
3. 治理视角：哪些动作会被权限、模式、worktree、远程策略约束。
4. 演化视角：哪些是稳定面，哪些是灰度面，哪些只是内部影子。

在此基础上再增加一个实战视角：

5. 专题视角：把高频工作流单独拆开，判断它们各自该走哪条能力路径。

这里的专题再分成两层：

- `04-专题深潜`：按真实工作目标组织。
- `05-控制面深挖`：按高价值控制面组织。

如果你只想知道“先看哪里”：

- 主线使用：回答“第一次怎样稳定用起来”。
- 参考索引：回答“到底有哪些能力、入口和边界”。
- 专题深潜：回答“围绕一个真实工作目标，完整该怎么走”。
- 控制面深挖：回答“为什么不能把看似相邻的入口混着用”。
- 能力地图：回答“系统在源码里究竟怎样组织这些能力”。

## 源码证据怎么读

最重要的注册点只有少数几类：

- 命令注册：`src/commands.ts`
- 命令类型合同：`src/types/command.ts`
- 工具注册：`src/tools.ts`
- 输入分发：`src/utils/processUserInput/processUserInput.ts`
- 斜杠命令执行：`src/utils/processUserInput/processSlashCommand.tsx`
- 主循环与会话状态：`src/main.tsx`、`src/query.ts`、`src/QueryEngine.ts`

这意味着读 Claude Code 不要从某个 UI 组件开始，而要先抓住这五个事实：

1. 任何用户输入先被分流成“普通提示词”还是“斜杠命令”。
2. 斜杠命令内部又分成 `local`、`local-jsx`、`prompt` 三类。
3. 普通提示词进入主查询循环后，模型拿到的是一组被权限和模式过滤后的工具池。
4. 技能、插件、MCP、工作流命令都不是旁枝，而是命令/工具总线的增量扩展。
5. 计划模式、worktree、任务、子代理不是附加玩具，而是“如何安全放大行动范围”的治理手段。

## 本手册怎样组织

为避免把所有信息混成一层，正文按六个部分组织：

1. 导读与阅读路径。
2. 主线使用。
3. 能力地图。
4. 参考索引。
5. 专题深潜。
6. 控制面深挖。

更简化地说：

- `01` 给你主线。
- `03` 给你总览和速查。
- `04` 给你围绕工作目标的长文。
- `05` 给你控制面判断。
- `02` 给你机制总图。

## 本版本的关键判断

从源码看，Claude Code 不是“聊天界面 + 几个工具”。

它更接近一个带治理层的编程运行时：

- 上层是输入编译器：把用户输入、附件、IDE 选区、记忆、钩子、命令、技能重新整理成模型可消费的请求对象。
- 中层是行动总线：命令、工具、MCP、插件、工作流、子代理。
- 下层是治理面：权限模式、计划模式、worktree、远程策略、灰度开关、上下文压缩和恢复。

所以正确使用 Claude Code，不是“尽量多说”，而是“把任务送到正确入口，让系统替你管预算、边界和恢复”。

如果把使用者第一跳再压成最短判断，只剩三句：

1. 提问前先过 `same-world test`
   - 你的目标、附件和继续条件，是否还让当前、下一步与 handoff 共享同一条 `message lineage`。
2. 看状态前先认 `decision window`
   - `Context Usage`、pending action、worker_status 和 session state 共同在告诉你“当前还配不配继续”，不是只在告诉你“已经花了多少”。
3. 省 token 前先分清 `stable bytes` 与 `working set`
   - 文件、附件、memory 和外置结果承载 stable bytes；当前对话只保留真正还要继续裁决的 working set。

更稳一点说，`00-导读 -> 01-主线使用 -> 04 / 05 -> 02 / 03` 也必须继续共享 `问题分型 -> 工作对象 -> 控制面 -> 入口` 这组 first-answer order；如果顺序不共享，入口页再多也只是在放大标题试错。

如果你在入口层最先关心的是 token 百分比、长摘要或 slash 面板长什么样，而不是这三句，后面越努力越容易把系统用浅。

如果继续把用户第一跳压成更稳的动作顺序，它也只该再多三行：

1. Prompt
   - 先确认目标、附件、working set、下一步和 handoff 还共享同一个工作对象。
2. 治理
   - 先确认这次是该继续，还是该拒收、降级、清理后恢复或升级给人。
3. 当前真相
   - 先确认旧目录、旧恢复资产和旧权限没有在冒充当前工作现场。

Prompt 这行在用户侧继续只该翻译同一张 ABI，不该再偷缩成三桶：

1. `目标`
   - 对应 `message_lineage_ref`
2. `附件`
   - 对应 `section_registry_ref + stable_prefix_ref`
3. `working set`
   - 对应 `protocol_transcript_ref + continuation_object_ref`
4. `next step`
   - 对应 `continue_qualification_verdict`

如果需要正式 owner 版本，不在这里补第二套 Prompt 前门，直接回 `bluebook/philosophy/84 -> guides/51`。

这三行不是蓝皮书深页里的抽象术语，而是“现在先做哪一步”的用户侧最小顺序。手册如果不给出这层顺序，读者就仍会把入口页只当目录，不当工作起点。
