# 设置面板、诊断屏与运营命令：会内控制面的三层分工

## 用户目标

不是只知道这些命令都“和状态有关”，而是先看清它们到底属于哪种运行时对象：

- `/status`、`/usage`、`/config` 这类 Settings tab
- `/doctor`、`/stats` 这类独立 screen
- `/model`、`/effort`、`/fast` 这类 session 调参命令
- `/theme`、`/color` 这类显示层命令
- `/statusline` 这类受控 prompt 工作流
- `/cost`、`/extra-usage`、`/rate-limit-options` 这类预算与限流分流入口

如果把它们都写成“状态运营命令”，读者最容易丢掉对象边界。

## 第一性原理

Claude Code 里至少存在四种不同的“运营面”：

- Settings 面板：把当前宿主状态、配置与预算集中投影到同一个多 tab UI。
- 独立诊断/统计屏：不是 tab，而是单独 screen。
- 会话调参命令：直接改当前 session 的行为参数。
- 预算与限流分流入口：回答“还能不能继续工作”，而不是“现在是什么状态”。

所以 `/status`、`/doctor`、`/usage`、`/cost`、`/stats`、`/model`、`/effort`、`/fast`、`/statusline` 不能按一个平面理解。

如果把这一页压成用户侧最小顺序，只该先做五步：

1. 先认 `state truth`
   - 当前 `/status` 看到的到底是不是同一个宿主现场。
2. 再认 `trustworthiness`
   - 当前 `/doctor` 看到的环境还值不值得继续信任。
3. 再认 `decision window`
   - 当前 `/usage`、`/cost`、`/stats` 指向的是额度窗口、会话 burn rate，还是长期画像。
4. 再认 `continuation path`
   - 当前真正要解决的是还能不能继续工作、继续走哪条产品路径。
5. 最后才认 `session tuning / display`
   - 当前真正要调的是模型、effort、fast、显示层，还是状态线配置工作流。

更短的前门公式就是：

- `state truth -> trustworthiness -> decision window -> continuation path -> session tuning / display`

还要再补一条治理边界：

- `/config` 不是另一套系统，而是同一 authority surface 在 Settings 容器里的配置投影。

## 进入本页前的 first reject signal

如果你现在问的是下面这些问题，也不该只停在这页：

- 你把 `/usage` 或 `Context Usage` 当 token 条，而不是 decision window。
- 你把 `/doctor` 当成 `/status` 的补充 tab。
- 你把 `/model`、`/effort`、`/fast` 写成同一类调参。
- 你把 `/statusline` 写成“显示当前状态”的只读命令。

## 先分清四种对象

### 1. Settings tab

源码能直接确认三条命令只是 `Settings` 面板的不同 tab：

- `/status` -> `defaultTab="Status"`
- `/config` -> `defaultTab="Config"`
- `/usage` -> `defaultTab="Usage"`

这说明它们不是三种完全独立的 UI，而是同一块 Settings 控制面的不同切口。

更准确地说：

- `/status` 解决“当前宿主现场是什么”
- `/config` 解决“当前配置面怎么改”
- `/usage` 解决“当前额度窗口和 extra usage 状态是什么”

### 2. 独立 screen

这组命令不进入 Settings tabs，而是单独打开自己的 screen：

- `/doctor` -> `Doctor`
- `/stats` -> `Stats`

它们和 Settings tab 的最大区别不是“界面长得不同”，而是：

- 不是同一容器
- 不是在切 tab
- 解决的问题也不是“当前设置长什么样”

### 3. 会话调参命令

这组命令更接近“改当前 session 的运行方式”：

- `/model`
- `/effort`
- `/fast`

共同特征：

- 都会直接作用到当前会话
- 会改 `AppState`、settings 或两者
- 它们回答的是“接下来怎么跑”，不是“当前看到了什么”

### 4. 显示层与受控 prompt 工作流

这一组最容易被误判成普通查询命令：

- `/theme`、`/color`：显示层
- `/statusline`：受控 prompt 工作流

尤其 `/statusline` 很特殊：

- 它不是 Settings tab
- 也不是独立 screen
- 它是 `prompt` 类型命令，会强制创建 `statusline-setup` 子代理，并限定可用工具

所以它本质上不是“显示当前状态”，而是“配置状态线如何显示状态”。

## 各入口的真实对象

### `/status`

这是 Settings 的 `Status` tab。

它会聚合：

- 版本
- Session 名称与 ID
- cwd
- account / provider
- model
- IDE / MCP / sandbox
- settings sources
- diagnostics summary

还要额外记住：

- 没有真正的 root `claude status`
- 与它名字最像的是 `claude auth status`，但后者只是认证投影，不是运行态总览

因此 `/status` 更像“当前会话宿主真相层”的投影，而不是 about 页。

### `/config`

这也是 Settings tab，只是切到 `Config`。

所以它和 `/status`、`/usage` 共享同一容器，但对象不同：

- `/status` 看当前真相
- `/config` 改当前配置面

不要把 `/config` 写成独立控制台。

### `/usage`

这同样是 Settings tab，只是切到 `Usage`。

它的对象不是“本地会话已经花了多少钱”，而是：

- 额度窗口
- 五小时与七天 utilization
- extra usage 状态
- overage / credit 相关分流

还要额外记住一个非常容易写错的点：

- UI 里的 `Current session` 实际映射的是 `five_hour` 配额窗
- 它不是当前 `sessionId` 的本地账本
- 这条命令本身也更偏 `claude.ai` 产品面；不是所有登录形态下都稳定可见可用

这就是为什么 `/usage` 应被写成预算面，而不是账单尾巴。

### `/doctor`

这是独立的 `Doctor` screen，不在 Settings tabs 里。

它解决的是环境可信度，而不是当前 tab 切换。

更稳的理解是：

- `/status` 告诉你“当前会话像什么”
- `/doctor` 告诉你“这套环境值不值得信任”

还要补一条跨层边界：

- `/doctor` 与 `claude doctor` 共用同一个 `Doctor` screen
- 但 root 版会自己搭 root render 栈，并且 help 文案明确写了 trust dialog / `.mcp.json` health check 的额外边界

### `/stats`

这也是独立 screen。

它和 `/usage`、`/cost` 的区别是：

- `/usage` 看额度窗口
- `/cost` 看当前会话成本 / 订阅或 overage 语义
- `/stats` 看长期使用统计与活动画像

更关键的是：

- `/stats` 主要基于本地 transcript 与 stats cache 聚合
- 它不是远端 billing 真相，也不是 `/usage` 的长期版

所以 `/stats` 不属于 Settings，也不属于当前会话即时调参。

### `/cost`

`/cost` 是 `local` 命令，不是 JSX screen。

它解决的是“当前会话成本/当前计费语义”，而不是长期统计，也不是额度窗口。

还要注意一条产品边界：

- 对 `claude.ai` subscriber，它优先解释“当前在用 subscription 还是 overages”
- 对非 subscriber，它才直接返回本地会话成本格式

所以 `/cost` 更像本地当前会话账本，而不是订阅用户的实时美元账单。

所以 `/cost` 和 `/usage` 不能合写成“费用命令”。

### `/model`

这是 session 调参面。

它不只是换一个模型名，而是会联动：

- `mainLoopModel`
- `mainLoopModelForSession`
- fast mode 兼容性
- extra usage 计费提示

因此 `/model` 的真实对象是“当前会话的能力面和成本面”。

### `/effort`

这也是 session 调参面，但维度不是模型，而是推理投入度。

关键点：

- 它支持 `low/medium/high/max/auto`
- 会读写 `userSettings.effortLevel`
- 还会受 `CLAUDE_CODE_EFFORT_LEVEL` 环境变量覆盖

还要再补一条边界：

- `/effort` 走的是 effort 通道
- 它不等于 thinking 配置

所以 `/effort` 不是 UI 装饰，而是明确的行为参数。

### `/fast`

这也是 session 调参面，但对象更接近吞吐策略。

关键点：

- 只在 fast mode 启用且可见时出现
- 开关时会更新 `userSettings.fastMode`
- 必要时还会顺带切到 fast mode 对应模型
- 会受可用性、冷却与组织状态影响

所以 `/fast` 不是 `/effort` 的同义词，它们控制的是不同维度。

### `/theme` 与 `/color`

这一组只该写成显示层：

- `/theme` 改主题
- `/color` 改 prompt bar color

更细一点：

- `/theme` 更接近全局 palette / theme
- `/color` 更接近当前 session 的身份色

它们不解决宿主状态、预算或 session 推理参数。

### `/statusline`

这是这一组里最容易被错写的一条。

它不是“当前状态查询命令”，而是一个 `prompt` 命令：

- 允许的工具被限制死
- 会创建 `statusline-setup` 子代理
- 默认去读 shell 配置并写 `~/.claude/settings.json`

所以它的真实对象是：

- 状态线配置工作流

而不是：

- 当前状态本身

### `/extra-usage` 与 `/rate-limit-options`

这两条属于预算与限流分流面，不属于单纯观测层。

更稳的区分是：

- `/extra-usage`：显式配置 overage / extra usage，必要时还会引导重新登录
- `/rate-limit-options`：隐藏分流器，只在 hit rate limit 且 `claude.ai` subscriber 时出现

它们回答的是：

- 还能不能继续工作
- 继续工作该走哪条产品路径

而不是：

- 当前状态长什么样

所以这两条更接近 `continuation path`，而不是观测层。

## 最容易误用的点

### 把 `/status`、`/config`、`/usage` 当三个独立系统

它们其实都只是 `Settings` 的不同 tab。

### 把 `/doctor` 当成 Settings 里的另一个 tab

它是独立 screen。

### 把 `/usage`、`/cost`、`/stats` 当同一类“花费命令”

它们分别对应：

- 额度窗口
- 当前会话成本 / 当前计费语义
- 长期统计画像

### 把 `/model`、`/effort`、`/fast` 当成同一件事

它们都在调参，但调的是不同层：

- model：能力面 / 成本面
- effort：推理投入度
- fast：吞吐与模型兼容策略

还要避免另一种误写：

- `/fast` 不是独立模型档位
- `/effort` 也不是 thinking 的另一套说法

### 把 `/statusline` 当成只读命令

它是受控 prompt 工作流，不是面板查询。

### 把 `/extra-usage` 写成预算查询页

它更像“预算续航动作”。

## 稳定面与条件面

### 稳定主线

- `/status`
- `/config`
- `/usage`
- `/doctor`
- `/model`
- `/effort`
- `/theme`
- `/color`
- `/stats`
- `/statusline`

### 条件、资格或产品面强相关

- `/fast`
- `/cost`
- `/extra-usage`
- `/rate-limit-options`

其中：

- `/rate-limit-options` 还是隐藏分流入口
- `/cost` 对 subscriber 与非 subscriber 的语义不同
- `/extra-usage` 明显依赖产品资格

## 用户策略

更稳的读法和用法通常是：

1. 先分清自己现在是要“看真相”“改配置”“做诊断”“调参数”还是“续预算”。
2. 看当前宿主现场时，用 `/status`。
3. 改配置面时，用 `/config`。
4. 环境可信度有疑问时，用 `/doctor`。
5. 看额度窗口时，用 `/usage`；看当前会话成本时，用 `/cost`；看长期画像时，用 `/stats`。
6. 调 session 行为时，再决定 `/model`、`/effort`、`/fast`。
7. 想把状态长期外显时，用 `/statusline`，但把它当配置工作流而不是查询命令。
