# 命令、提示词、技能、工具：入口选择决策树

## 第一性原理

Claude Code 最容易被误用的地方，不是工具不会用，而是入口选错。

同一任务可以从四种入口进入：

- 普通提示词
- 斜杠命令
- 技能
- 工具调用

用户真正该学会的不是“多写提示词”，而是“把任务送到最省协调成本的入口”。

如果把这页压成用户侧最小顺序，只该先做四步：

1. 先认任务对象
   - 你要推进的是开放探索、管理动作、协议化工作流，还是执行面边界控制。
2. 再认 projection
   - 这件事应该先以自然语言任务、slash 命令、技能对象，还是工具池边界的形式出现。
3. 再认 runtime gate
   - 当前真正需要控制的是入口歧义、能力边界，还是执行权限。
4. 最后才在提示词、命令、技能、工具之间做选择。

更短的前门公式就是：

- `task object -> capability projection -> runtime gate -> execution surface`

## 进入本页前的 first reject signal

看到下面迹象时，应先回到任务对象判断，而不是直接挑入口名字：

- 你其实已经知道要打开管理面，却还在写长 prompt。
- 你其实只是在重复一套协议化工作法，却还把它留在普通提示词里。
- 你真正想控制的是工具边界，却把问题写成“先找一个更像按钮的入口”。
- 你把“工具”理解成用户第一心智，而不是模型的执行面。

## 入口 1：普通提示词

适合：

- 问题还没完全定型。
- 希望模型先探索再决定工具路径。
- 任务需要综合文件、命令、测试、修改和总结。

不适合：

- 你其实已经知道要打开某个管理面。
- 你只是想切配置。
- 你只是想调用现成工作流。

## 入口 2：斜杠命令

适合：

- 本地状态管理。
- 需要 UI 面板的管理动作。
- 想避免自然语言歧义。

典型命令：

- `/config`
- `/permissions`
- `/mcp`
- `/plugin`
- `/agents`
- `/help`

## 入口 3：技能

适合：

- 一套会反复发生的工作方法。
- 想让方法变成正式命令对象。
- 想约束工具、模型或执行上下文。

如果你发现自己经常重复一段长 prompt，而它又不是底层系统设置，那通常应该升格为技能。

## 入口 4：工具

工具本身通常不是用户直接手打的入口，而是模型在任务执行中调用的行动面。

用户真正需要做的是：

- 理解某类任务大致会落到哪些工具。
- 通过 `/permissions`、计划模式、worktree 等治理手段决定这些工具的边界。

## 决策树

### 我只是想切某个设置或打开管理界面

用斜杠命令。

### 我想让 Claude 按某套固定工作法做事

先考虑技能。

### 我的问题还没定型，想让 Claude 自己探索

用普通提示词。

### 我想扩展到外部系统

不是直接写 prompt，而应进入 MCP/插件路径。

### 我真正纠结的是入口选错会不会放大协调成本

先回到四问：

1. 这是探索对象、管理对象、协议对象，还是执行对象。
2. 它应该先给谁看：用户、本地宿主，还是模型执行面。
3. 这一步真正要过的 gate 是歧义、治理，还是权限。
4. 哪种入口最少引入额外协调层。

## 为什么“工具”不应该成为用户的第一心智

因为 Claude Code 的设计意图是让用户在更高层描述任务，让模型自己在工具池里行动。工具是执行面，不是人类主要交互面。

用户真正该掌握的是：

- 入口选择。
- 边界控制。
- 长任务连续性。
- 并行与隔离。
