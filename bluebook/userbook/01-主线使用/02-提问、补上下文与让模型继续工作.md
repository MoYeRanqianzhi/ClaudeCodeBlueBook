# 提问、补上下文与让模型继续工作

## Claude Code 先分流输入，再决定是否进入模型

`src/utils/processUserInput/processUserInput.ts` 是用户输入的总入口。它首先做的不是“直接问模型”，而是判断：

- 这是不是普通提示词。
- 这是不是斜杠命令。
- 这次输入有没有附件、图片、IDE 选区、粘贴块、系统生成的 meta prompt。

这一步很重要，因为 Claude Code 的真正强项不是自由聊天，而是“把你的输入编译成更适合行动的对象”。

如果把这一页压成最短使用判断，只剩三条：

1. 先确认你现在问的还是同一件事
   - 任务目标、附件和下一步安排别中途换主语。
2. 先看这轮对话还值不值得继续展开
   - `Context Usage` 不是装饰，它在提醒你现在留下来的内容是不是还值得继续处理。
3. 先把精确事实放到文件、附件或记忆里
   - 对话里只保留还要你和模型一起决定的部分。

如果继续把 Prompt 线压成用户能直接执行的最小顺序，也只该再补成：

1. 先确认任务目标还没变。
2. 再确认附件、文件、选区和记忆都还在补同一件事。
3. 再确认当前对话里留下的，真的是还要继续处理的部分。
4. 最后才确认下一步是不是已经足够明确。

只要这四步里有任何一步开始依赖长摘要、最后一条回复或 `systemPrompt` 想象，这一页就在教你把 Prompt 用浅。

## 别只会 compile，还要会 verify witnesses

更硬一点说，Prompt 真正值钱的不是“把输入送进去”，而是“送进去以后还能不能证明这还是同一个世界”。
只有 later consumer 无需重答“谁在定义世界、哪些事实还在场、下一步什么仍合法”就能继承的东西，才算 witness；只能帮人回忆的长摘要、标题、最后一条消息或想象中的 `systemPrompt`，最多只算 reminder，不算 witness。

对用户更稳的最小核查是：

1. `目标 witness`
   - 你要推进的任务目标有没有换主语。
2. `附件 witness`
   - 附件、文件、选区和 memory 还在不在补同一件事。
3. `working-set witness`
   - 当前对话里留下来的，是不是仍然只剩要继续裁决的部分。
4. `next-step witness`
   - 下一步动作与 handoff 还沿着同一条工作对象继续，而不是只剩故事或摘要。

如果这四个 witness 里任何一个已经只能靠长摘要、标题、最后一条消息或想象中的 `systemPrompt` 来维持，这轮 Prompt continuity 就已经失真，因为这时留下来的不是可采 witness，而只是更会提醒人的替身。

这四个 user-facing witness 也只是在翻译同一张 Prompt owner 页，不该在这里继续变成字段表：

1. `目标 witness`
   - 先确认你还在同一件事上。
2. `附件 witness`
   - 先确认文件、选区、附件和 memory 还在给这件事补事实。
3. `working-set witness`
   - 先确认当前对话里留下的仍然只是这件事还要继续判断的部分。
4. `next-step witness`
   - 先确认下一步与 handoff 没被摘要、标题或最后一条消息偷换。

如果需要 owner 级 object chain，不在这一页补第二套 Prompt doctrine，直接回 `../../philosophy/84`。

## 普通提示词适合什么

普通提示词适合：

- 提交真实开发任务。
- 让模型自己决定要不要读文件、搜索、改代码、跑命令。
- 把多步任务交给查询循环自动推进。

如果你已经知道该用哪个命令或技能，再用普通提示词有时反而更绕。源码里命令和技能之所以被保留为一级入口，就是为了减少这类额外协调。

## 斜杠命令适合什么

`src/utils/processUserInput/processSlashCommand.tsx` 说明斜杠命令是一个专门的执行面。它能做三种事：

- 直接本地执行，不进模型。
- 打开一个本地 JSX 界面，让用户在 UI 中完成配置或确认。
- 变成一个 prompt command，把技能内容或命令内容送进模型。

所以你应该把斜杠命令理解为“绕过自然语言歧义的精确入口”。

典型场景：

- `/model`、`/theme`、`/config`、`/permissions` 这类配置类操作。
- `/mcp`、`/plugin`、`/agents`、`/hooks` 这类管理类操作。
- `/compact`、`/resume`、`/review`、`/diff`、`/memory` 这类高价值工作流操作。

## 附件与上下文不是补充品，而是输入一部分

从 `src/utils/attachments.ts`、`src/utils/imageResizer.ts`、`src/types/textInputTypes.ts` 可以看出，Claude Code 会系统性地吸纳：

- 粘贴的大块文本。
- 图像输入。
- 文件附件。
- IDE 选区。
- 钩子回写的额外上下文。

因此高质量使用方式不是“把所有背景写成长段文字”，而是：

- 把精确上下文挂成附件或文件。
- 把任务目标写成简洁提示词。
- 让系统自己读文件、合并上下文、决定调用工具。

更短的 token 经济版本就是：

- `Narrow / Later / Outside`

也就是说：

- 先缩小真正要裁决的 working set。
- 晚绑定高波动事实。
- 把大结果对象外置，只留下 stable bytes 的引用和必要摘要。

## 如何让模型继续工作，而不是被你打断

源码里有几条很强的设计倾向：

- `processSlashCommand.tsx` 会把某些命令 fork 到子代理。
- 计划模式明确要求先探索后实现。
- 任务、后台代理和通知系统支持长任务不中断继续。
- 紧急插话、侧问题和队列消息都被单独建模。

对用户来说，这意味着：

1. 不要每一步都手工指挥。
2. 在任务主线稳定后，让系统自己读、改、测、汇报。
3. 需要插入控制时，用命令入口，而不是改写整段任务。

更短的用户侧判断是：

- 不要把“继续工作”理解成“继续聊天”。
- 真正要保住的是同一件事、同一批已确定事实，以及还值得继续处理的当前部分。

## 最有价值的四个补上下文方式

### `/context`

查看当前上下文的占用与组成。它不是装饰，也不是纯成本面板，而是在帮你判断这轮对话里还有多少内容值得继续保留。

### `/files`

快速确认哪些文件已经进入上下文面，避免你和模型对“当前共同事实”理解不一致。

### `/memory`

当某些偏好、约束或项目事实会反复出现时，不要每回合重写，应该沉到记忆层。

### `/compact`

当任务已经积累大量历史，而你还想继续推进时，应优先压缩，而不是重开会话。

更准确地说，`/compact` 的价值不是“把故事缩短”，而是尽量把已确定的内容留在场外，让当前对话只保留还要继续处理的部分。

还要再补一句边界：

- summary 只是 carrier，不是 signer。
- `/compact` 可以换表示，不能替 `continuation qualification` 签字。

## 什么时候该转去读长文专题

如果你已经不满足于“知道有哪些入口”，而是要判断：

- 附件、图片、选区、目录和 hooks 该怎样配合。
- 为什么 `cd`、`/add-dir`、`/files`、`/context` 不是一回事。
- 哪些上下文属于主任务，哪些只是系统回写证据。
