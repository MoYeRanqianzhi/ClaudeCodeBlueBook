# 三张控制面总图：世界进入模型、扩张定价与防过去写坏现在

Claude Code 最难被抄走的不是功能全集，而是这三条设计母线。

## 0. 三条母线

如果把 Claude Code 继续压缩到最少，它真正持续回答的只有三件事：

1. 世界如何进入模型
2. 扩张如何被定价
3. 当前如何不被过去写坏

这三件事分别对应三张控制面图：

1. 请求装配控制面
   - 对象镜像见 `architecture/82`
2. 反扩张治理控制面
   - 对象镜像见 `architecture/83`
3. 当前真相与反僵尸控制面
   - 对象镜像见 `architecture/84`

也就是说，Claude Code 真正的高级感不是：

- 它会更多功能

而是：

- 它先把这三个问题制度化了

如果再往上压一层，这三张图现在也各自形成了稳定的判断标准：

1. 请求装配线
   - `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability` 是否仍对同一个世界保持诚实
2. 治理线
   - 扩张是否仍被同一条治理收费链单源定价
3. 当前真相线
   - 现在是否仍只有一条合法 current-truth chain 在阻止过去写坏现在

这里还要再补一条入口纪律：

- `evidence gradient -> current-truth surface -> temporal honesty -> future maintainer rejectability` 只继续保留为 crosswalk / 解释层压缩句
- 不再和源码质量线 canonical ladder 并列抢首答权

这里也应再固定一条 continuity crosswalk：

- `continuity` 不是第四张控制面图；它只是三张图在时间维度上的共同验收面。
- 第一张图负责 `Continuation` 仍是否沿同一 witness chain 成立。
- 第二张图负责 `continuation pricing` 是否仍愿意为这次继续收费。
- 第三张图负责恢复资产、旧写入者与 replay 路径是否还没篡位 current truth。

第三张图在本页也只保留一条宪法钩子：当前必须先被正式保护；更细的 evidence ceiling、canonical ladder 与 object-level certainty 统一先回根入口与 `guides/102`，对象镜像才继续回 `architecture/84`。

如果继续把这三张图压成最短动作链，它们还可以再写成：

1. 请求装配线
   - `compile -> verify -> enforce -> continue -> explain`
2. 治理线
   - `price -> externalize -> arbitrate -> continue-or-stop -> cleanup`
3. 当前真相线
   - `surface -> writeback -> freshness -> evict`

一旦高阶总论页不能再压成这三条动作链，说明解释还停在对象层总结，没有真正回到设计宪法。

如果再往目录入口压一层，这三张图也只该继续暴露三条 first-answer hooks：

1. 请求装配线
   - 只认 `same-world compiler`；更细的 `入口顺序 / canonical witness order` 统一回 `84`
2. 治理线
   - 只认“扩张必须先被单源定价”；更细顺序统一回治理归属链
3. 当前真相线
   - 只认“当前必须先被正式保护”；更细证据层统一先见根入口与 `guides/102`

如果 later maintainer 第一次进入 `09` 时还看不见这三条，解释就还停在对象层，总图还没有真正变成入口宪法。

如果 later maintainer 已经开始寻找每条母线的具体 reject path、failure order 或排查顺序，本页就已经不该继续下沉；那些顺序统一回 `15` 与各归属页，本页只保留首答 hook 与 cross-line mirror。

## 0.1 这三张图不是高级观察，而是设计宪法

这三张图不是读后总结，而是设计宪法。

1. 它们不是专题并列关系，而是后续一切功能与目录都要回来的宪法层。
2. 少了第一张图，Prompt 会退回更像真相的文案。
3. 少了第二张图，安全与省 token 会退回彼此分裂的局部优化。
4. 少了第三张图，源码质量会退回更漂亮的分层截图与目录体感。

所以这三张图真正值钱的地方，不是解释功能，而是规定：

- 什么问题配先成为先验约束
- 什么扩张配先进入当前世界
- 什么旧状态、旧入口与旧写入者永远不配重新篡位当前世界

这三张图共享的制度敌人也只剩一个：重判资格回流。凡是展示层、消费面、恢复资产或局部文本又重新拿回改判世界、免费续租扩张或篡位当前的资格，系统就会从制度能力退回协商。

## 1. 第一张图：世界如何进入模型

Claude Code 第一张图真正强的不是：

- prompt 更会说话

而是：

- `world entry / request assembly / six-stage assembly chain` 先把世界合法编译进模型

Anthropic 官方文档常把 Claude Code 写成 `gather context -> take action -> verify results` 的 agentic loop；但在总图页里更稳的结论只有一条：

- 第一张图只认 `same-world compiler`；若你已经开始点 `witness` 对象、canonical order 或 builder-facing implementation order，说明这页又在重开 `84` 的归属页账本。

离开这一步，Prompt 效力就只剩“会开场”，不再是可闭环的 same-world runtime。

如果继续把第一张图压得更硬，也只保留一句：Prompt 的本体不是更长文案，而是 `world entry / request assembly / six-stage assembly chain` 让世界在进入、压缩、分叉与交接后仍保持同一现场；具体对象与资格顺序统一回 `84`。

如果这条流水线不存在，系统再会写 Prompt，也只是在堆更像真相的说明书。

## 2. 第二张图：扩张如何被定价

Claude Code 真正成熟的安全设计，也不是：

- 更会拦截

它更接近：

- 先阻止未定价扩张进入当前世界，再为合法扩张声明价格与顺序

这里还应先钉死一条 `public-evidence ceiling`：

- `approval / modal / status / usage / compact` 都只是治理消费面，不是治理说话面。
- 安全与省 token 只是同一反免费扩张秩序的两种可见读法。
- 如果你已经开始点 `pricing-right / truth-surface / asset-rollback ABI` 或四类被定价空间，说明这页又在重开 `10 / philosophy/61 / security` 的 owner 链。

所以安全与 token 效率共享同一治理目标。

它们共同反对的是：

- 免费扩张

这也意味着：

- “更安全”不是多几道门
- “更省 token”不是回答更短

更稳一点说：

- “省 token”的本体不是 compact 技巧，而是未定价的字节、上下文与续租都不配免费进入当前世界

在总图页，这里也只保留一句治理 mirror：安全与 token 效率共享同一反免费扩张秩序；具体 `governance pricing chain / failure semantics / human fallback` 统一回 `10 / philosophy/61 / security`，不在 `09` 第二次发布治理模板或排查顺序。

它们只是同一价格秩序在不同资产上的可见读法

这也意味着第一张图和省 token 并不冲突；`stable prefix / dynamic boundary` 本来就是把世界准入写进同一价格秩序。

也就是说，Claude Code 真正在做的不是“限制模型”，而是：

- 让模型只能在已授权、已外化、已定价的当前世界里行动

## 3. 第三张图：当前如何不被过去写坏

Claude Code 的源码先进性，也不是：

- 目录更整齐
- 文件更碎

它真正先进的地方，在于先把“谁配宣布现在、过去为何不配篡位现在”写成结构约束。

所以第三张图真正保护的是时间诚实性：

- 过去的写入者不得偷走现在
- 恢复资产不得假装自己就是当前真相
- future maintainer 也必须能只凭权威入口、危险改动面与拒收能力继续保护这条边界

所以这里真正被保护的不是当前分层截图，而是未来维护者继续拒绝错误实现的正式能力。

这也是为什么“目录更整齐”“文件更小”本身都不构成源码先进性的证据；真正的证据是复杂度中心是否合法、边界是否可证、时间上的第二真相能否被系统性拒收。

如果你已经开始点 `current-truth surface / consumer subset / danger surface / reject path`，说明这页又在重开 `guides/102 / philosophy/86 / philosophy/87 / architecture/README` 的归属链。

在总图页，这里也只保留一句 current-truth mirror：未来维护者必须继续拥有正式拒收错误实现的能力；具体 current-truth why、source-quality why、evidence ceiling、current-truth surface、recovery non-sovereignty、freshness / eviction 与 later reject path 统一回 `guides/102 / philosophy/86 / philosophy/87 / architecture/README`，不在 `09` 第二次发布 source-quality 模板或排查顺序。

## 4. 为什么这三张图必须先于功能解释

如果把它们降成后置总结，Claude Code 就容易继续被误读成：

1. Prompt 更强
2. 安全更严
3. 代码更漂亮

更成熟的解释顺序应该先声明：

1. 它首先在编译世界
2. 它首先在为扩张定价
3. 它首先在防止过去写坏现在
4. 它还分别给出了这三件事各自的判断标准，而不是只给出三组实现细节

只有把这三件事前置，后面的架构、API、哲学、指南、案例和演练才不会重新写散。

这也是为什么三张控制面之后还需要 `navigation/05 / navigation/15 / navigation/41`：

1. `navigation/05` 负责先防止读者抄错对象。
2. `navigation/15` 负责先点名第一条反证信号。
3. `navigation/41` 负责最后把对象链压回第一性原理。

如果少了这三步，三张图本身也会重新退回漂亮总结。

## 5. 如果从第一性原理重新设计 Agent Runtime

如果今天从零开始设计一台 Agent runtime，最先问的应当不是：

- 有哪些工具
- 能不能多 Agent
- 支不支持远程

而应当先问：

1. 现在的世界是怎样被合法编译进模型的
2. 哪些扩张要立刻收费、哪些扩张必须延后、哪些扩张必须迁出主工作集
3. 哪些旧写入者、旧状态、旧入口绝不能重新写坏现在

如果这三件事没有先成立，后续功能再多，也只是更厚的外观。

## 6. 苏格拉底式追问

在你准备宣布“我已经抓住了 Claude Code 的设计内涵”前，先问自己：

1. 我描述的是功能，还是世界进入模型的方式。
2. 我描述的是更多限制，还是扩张如何被正式定价。
3. 我描述的是分层更漂亮，还是过去怎样被正式阻止写坏现在。
4. 这些问题能不能直接拿去审一个新 runtime，而不是只拿来赞美 Claude Code。
5. 如果把功能都先删掉一半，这三张控制面还在不在。
6. 我复制的是这三张图里的约束链与判断顺序，还是只复制了措辞、面板与目录外观。

## 7. 一句话总结

Claude Code 最难被抄走的部分，不是更多功能，而是三张控制面图：它先规定世界怎样进入模型，再规定扩张怎样被定价，最后规定过去怎样不得写坏现在。
