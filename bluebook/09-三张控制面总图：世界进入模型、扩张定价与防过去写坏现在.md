# 三张控制面总图：世界进入模型、扩张定价与防过去写坏现在

Claude Code 最难被抄走的不是功能全集，而是这三条设计母线。

## 0. 三条母线

如果把 Claude Code 继续压缩到最少，它真正持续回答的只有三件事：

1. 世界如何进入模型
2. 扩张如何被定价
3. 当前如何不被过去写坏

这三件事分别对应三张控制面图：

1. 请求装配控制面
   - 见 `architecture/82`
2. 反扩张治理控制面
   - 见 `architecture/83`
3. 当前真相与反僵尸控制面
   - 见 `architecture/84`

也就是说，Claude Code 真正的高级感不是：

- 它会更多功能

而是：

- 它先把这三个问题制度化了

如果再往上压一层，这三张图现在也各自形成了稳定的判断标准：

1. 请求装配线
   - `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability` 是否仍对同一个世界保持诚实
2. 治理线
   - `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup` 是否仍在为同一类扩张收费
3. 当前真相线
   - `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline` 是否仍阻止过去写坏现在

这里还要再补一条入口纪律：

- `evidence gradient -> current-truth surface -> temporal honesty -> future maintainer rejectability` 只继续保留为 crosswalk / 解释层压缩句
- 不再和源码质量线 canonical ladder 并列抢首答权

这里也应再固定一条 continuity crosswalk：

- `continuity` 不是第四张控制面图；它只是三张图在时间维度上的共同验收面。
- 第一张图负责 `Continuation` 仍是否沿同一 witness chain 成立。
- 第二张图负责 `continuation pricing` 是否仍愿意为这次继续收费。
- 第三张图负责恢复资产、旧写入者与 replay 路径是否还没篡位 current truth。

如果继续把源码质量线往 later maintainer 可直接复用的顺序再压一层，还应先过：

1. `public artifact ceiling`
2. `contract`
3. `registry`
4. `current-truth surface`
5. `consumer subset`
6. `hotspot kernel`
7. `mirror gap discipline`

若 sole writer、writeback seam 与 freshness gate 还没锁定，就只配先写 `current-truth surface candidate`，不配提前把 present truth 写死。

后续解释都应回到这三句判断，把它们当成共同校准点，而不是各自再造一版定义。

如果继续把这三张图压成最短动作链，它们还可以再写成：

1. 请求装配线
   - `compile -> verify -> enforce -> continue -> explain`
2. 治理线
   - `price -> externalize -> arbitrate -> continue-or-stop -> cleanup`
3. 当前真相线
   - `surface -> writeback -> freshness -> evict`

一旦高阶总论页不能再压成这三条动作链，说明解释还停在对象层总结，没有真正回到设计宪法。

如果再往目录入口压一层，这三张图还应先暴露三条 canonical formula：

1. 请求装配线
   - 只认 `same-world compiler`；更细的 `入口顺序 / canonical witness order` 统一回 `84`
2. 治理线
   - `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`
   - `Narrow / Later / Outside` 只是用户侧助记，不是第二条治理链
3. 当前真相线
   - `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`

如果继续给第三张图保留一条 crosswalk，也只该再补一句：

- `evidence gradient -> current-truth surface -> temporal honesty -> future maintainer rejectability` 只配做 later-maintainer 视角下的解释层压缩句，不再和 canonical ladder 并列抢首答权。

如果 later maintainer 第一次进入 `09` 时还看不见这三条，解释就还停在对象层，总图还没有真正变成入口宪法。

如果继续把三张图压成 later maintainer 可直接拿来排查的最小顺序，它们还应各自产出：

1. 请求装配线
   - 本页只把 Prompt 宪法压成 `same-world compiler` 这句总钩子；更细的 `入口顺序` 继续回 `84`
   - later maintainer 在这里先问世界准入、协议消费与继续资格是否仍单源成立；具体 witness object 与 builder-facing implement / audit 顺序继续回 `84`
2. 治理线
   - `pricing-right mismatch -> truth-surface demotion -> asset-rollback ABI sealing -> shared reject verdict -> reopen qualification / human fallback`
   - 第二张图真正先判的不是“回滚物件在不在”，而是 `pricing-right signer / lease` 是否先合法成立、`truth-surface attestation` 是否把 `reject / reopen` 外化清楚；`asset-rollback ABI` 只负责把收口动作做成可恢复 contract。
3. 当前真相线
   - `contract mismatch -> registry drift -> current-truth split -> stale-writer eviction -> mirror-gap demotion`

如果继续把 continuity 的最小顺序也压成 later maintainer 可直接拿来排查的一行，它也只该剩：

- `continue qualification -> continuation pricing -> recovery non-sovereignty / anti-zombie`
- 一旦这条顺序被拆成独立命令目录或独立第四母线，later maintainer 学到的就不再是三张图的时间轴，而是按钮库存。

没有这三组顺序，三张图就仍然更像高阶总结，而不是能指导 builder、reviewer 与 later maintainer 同时落地的宪法对象。

## 0.1 这三张图不是高级观察，而是设计宪法

这三张图不是读后总结，而是设计宪法。

1. 它们不是专题并列关系，而是后续一切功能与目录都要回来的宪法层。
2. 少了第一张图，Prompt 会退回更像真相的文案。
3. 少了第二张图，安全与省 token 会退回彼此分裂的局部优化。
4. 少了第三张图，源码质量会退回更漂亮的分层截图与目录体感。

所以这三张图真正值钱的地方，不是解释功能，而是规定：

- 什么问题配先成为先验约束
- 什么扩张配先进入当前世界
- 什么旧状态、旧入口与旧写入者永远不配重新篡位当前世界

## 1. 第一张图：世界如何进入模型

Claude Code 第一张图真正强的不是：

- prompt 更会说话

而是：

- `world entry / request assembly / six-stage assembly chain` 先把世界合法编译进模型

Anthropic 官方文档还把 Claude Code 直接写成 `gather context -> take action -> verify results` 的 agentic loop；所以第一张图如果只剩 `compile`，还不够硬。更稳的读法是：

- `compile` 先决定世界怎样合法进入模型；
- `verify` 再持续检查 `world-entry / continue verdict` 是否仍沿同一条 witness chain 成立。

离开这一步，Prompt 效力就只剩“会开场”，不再是可闭环的 same-world runtime。

这里最稳的入口不是再开两栏 inventory，而是承认第一张图必须同时具备 compile inputs 与 verify witnesses 这两类对象；具体对象清单统一回 `84`。本页只保留宪法层判断，不再第二次库存化 Prompt object list。

如果继续把第一张图压得更硬，还应先承认：

- Prompt 不是单段 `systemPrompt`
- 而是 `system sections / tool descriptions / agent prompts / attachment deltas` 共同组成的 multi-surface world-entry object

而 `same-world test` 真正最小可验的对象，也不能只剩一句判断；如果 `compact / resume / fork / handoff` 之后已经回不到 `84` owner chain 承认的 canonical witness order 与 builder-facing implementation order，第一张图就已经从“同一个世界”退回“长得还像同一个世界”的叙事成功。
这些具体 witness object 也只是在具体层复写 `Authority -> Boundary -> Transcript -> Lineage -> Continuation`；真正不能越位的是末端 `Explainability`，它只能解释哪一环断了，不能把断掉的资格重新说成还在。

这意味着 Prompt 的本体不是：

- 一段文案

而是：

- 一条请求装配流水线

再往上压一层，这条流水线真正保护的也不是“这一轮回答更顺”，而是：

1. 当前回合活在什么世界里
2. 下一步继续时还能否留在同一个世界里
3. compact / handoff 之后是否仍能把同一个世界交给后续消费者

所以最难抄走的从来不是某段 system prompt，而是：

- stable prefix 里只留下真正稳定的 law，高波动事实只从晚绑定层进入
- `display transcript`、`protocol transcript` 与 `continuation object` 仍沿同一条 `message lineage` 投影
- `/btw`、suggestion、memory、summary 与 subagent forks 仍 cache-safe 复用同一世界，而不是各自重造现场

更短的判断句只有一句：

- 第一张图不是文案吸引力，而是 `world entry / request assembly / six-stage assembly chain` 让世界在进入、压缩、分叉与交接后仍保持同一现场

如果继续把第一张图写成 first reject signal，最先暴露的不是回答质量变差，而是：

- `world-defining winner` 不清
- `display transcript` 开始冒充 `protocol transcript`
- compact 后只剩摘要，不剩 `continuation qualification`

而 later maintainer 真正最短的 Prompt 排查顺序，也不该先回 `systemPrompt` 截图，而应先回 `84` 的 `入口顺序` 与 canonical witness order，再回 owner chain 内部的 builder-facing implementation order。

如果这条流水线不存在，系统再会写 Prompt，也只是在堆更像真相的说明书。

## 2. 第二张图：扩张如何被定价

Claude Code 真正成熟的安全设计，也不是：

- 更会拦截

它更接近：

- 先阻止未定价扩张进入当前世界，再为合法扩张声明价格与顺序

这里还应先钉死一条 `public-evidence ceiling`：

- `approval / modal / status / usage / compact` 都只配做 `typed ask / runtime readback / continuation` consumer，不能定义治理 verdict。
- 真正配定义治理主语的，只能是 `pricing-right -> truth-surface attestation -> asset-rollback ABI` 这条执行链。

这里被统一定价的至少有四个空间：

1. 动作空间
2. 权威空间
3. 上下文空间
4. 时间空间

更具体一点看，就是：

1. `mode / permission` 在给动作定价
2. `governance key / typed ask` 在给权威定价
3. `visibility / externalization / context seat` 在给上下文定价
4. `decision window / continuation pricing / durable-transient cleanup` 在给时间定价

如果把这张图写成更可排查的模板，可以直接固定成三格：

- `governance pricing chain`
  - `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`
- `execution contract`
  - runtime 决定收费顺序，模型不能自己免费扩张动作、权威、上下文与时间；host 只消费 runtime 外化的 governance truth chain 与 window/pricing readback，不自己回放拼 mode、tool pool 或 pending action；人类只在升级点与拒收点介入
- `reject path`
  - 一旦叙事又退回“安全 vs token”的双话题，或者退回 mode、modal、dashboard 与 default-continue 这些可见替身，就先回四类扩张与收费对象

再往执行层压一层，这张图还必须最终交出：

- failure semantics
- escalation thresholds
- human fallback / halt conditions

否则“统一定价”仍会退回抽象口号，而不会变成可验证的治理判决。

所以安全与 token 效率共享同一治理目标。

它们共同反对的是：

- 免费扩张

这也意味着：

- “更安全”不是多几道门
- “更省 token”不是回答更短

而 failure semantics 也必须沿这条治理线一起被写硬：

1. `deny`
2. `ask`
3. `rollback`
4. `cleanup`
5. `halt`
6. `step-up / human fallback`

如果第二张图只会给出 `allow / ask / deny` 这类前半段 verdict，却不交代继续失败、恢复失败、对象升级失败之后谁配 `rollback / cleanup / halt`，它就仍停在“更成熟的权限系统”，还不是完整的治理定价控制面。
更硬一点说，rollback 也必须带 ceiling：它只封住可恢复 contract surface，不回滚治理真相，也不天然覆盖 Bash、network 或 operator side effects 的整世界可逆。

更稳一点说：

- “省 token”的本体不是 compact 技巧，而是 stable bytes：被外置的大对象必须在 turn、fork 与 resume 后重放同一 replacement bytes，否则省下来的不是 token，而是系统诚实性

如果继续把第二张图写成 first reject signal，最先暴露的通常也不是“漏拦了一次”，而是：

- `Context Usage` 被直接读成成本面板
- 没有决策增益的检查继续占用 working set
- 大结果对象没有被 `Later / Outside`

而 later maintainer 真正最短的治理排查顺序，也应先回：

1. `pricing-right mismatch`
2. `truth-surface demotion`
3. `asset-rollback ABI sealing`
4. `shared reject verdict`
5. `reopen qualification / human fallback`

只要系统还在用 mode、modal、usage、compact、default continue 这些可见替身替代这组 failure verdict，它就还没有真正进入第二张图。
审批框、status 条和 usage 面板都只配做 `typed ask / runtime readback` consumer；谁先有资格定义治理 verdict，仍只取决于 `pricing-right` 与 `truth-surface attestation` 是否先成立。

它们只是同一价格秩序在不同资产上的可见读法

这也意味着第一张图和省 token 并不冲突；`stable prefix / dynamic boundary` 本来就是把世界准入写进同一价格秩序。

也就是说，Claude Code 真正在做的不是“限制模型”，而是：

- 让模型只能在已授权、已外化、已定价的当前世界里行动

## 3. 第三张图：当前如何不被过去写坏

Claude Code 的源码先进性，也不是：

- 目录更整齐
- 文件更碎

它真正先进的地方在于，它会先把这些问题结构化：

1. 当前哪一个 surface 有资格宣布真相
2. 哪些恢复资产只能帮助恢复，不能篡位
3. 哪些陈旧 finally、陈旧 snapshot、陈旧 pointer 不得写回 present state
4. 哪些编译期 / 运行期边界根本不该进入公开图谱

所以第三张图真正保护的是时间诚实性：

- 过去的写入者不得偷走现在
- 恢复资产不得假装自己就是当前真相
- future maintainer 也必须能只凭权威入口、危险改动面与拒收能力继续保护这条边界

所以这里真正被保护的不是：

- 当前分层截图

而是：

- 未来维护者继续拒绝错误实现的正式能力

这也是为什么“目录更整齐”“文件更小”本身都不构成源码先进性的证据；真正的证据是复杂度中心是否合法、边界是否可证、时间上的第二真相能否被系统性拒收。

因此目录结构也不是阅读菜单，而是 future maintainer 的证据梯度：它必须更快暴露 current-truth surface、consumer subset、danger surface 与 reject path，而不是只把文件列得更细。

第三张图也应继续补上 later maintainer 的最小 reject 规则：

1. 看见 shim 没有退出条件，就先拒绝把它当长期结构。
2. 看见 snapshot、pointer、recovery asset 能直接回写主状态，就先拒绝它继续冒充 present truth。
3. 看见旧 capability token、旧 authority width、旧 display pin 还能继续放权，就先把这次结构升级退回到 freshness / eviction 层。

如果把这张图也压成更可排查的模板，可以固定成三格：

- `current-truth surface`
  - `current-truth writeback / recovery assets / stale snapshot-finally-pointer / danger surface / second-truth risk`
- `authority contract`
  - 只有 present surface 有资格宣布当前真相；恢复资产可以帮助恢复，但不能篡位；future maintainer 是正式守门人，而不是事后旁观者
- `reject path`
  - 一旦源码质量讨论退回目录美学，或者旧状态开始重写 present state，就先回 current-truth surface 与 second-truth guard

如果继续把第三张图写成 first reject signal，最先暴露的通常不是“结构不好看”，而是：

- host 只能靠事件回放自己拼当前真相
- `failure semantics` 还停在异常文案里
- danger surface 被目录观感盖住

而 later maintainer 真正最短的当前真相排查顺序，也应先回：

1. shim 是否仍有退出条件
2. recovery asset 是否还守 non-sovereignty
3. stale authority / ghost capability 是否已正式 eviction
4. 一旦失真，当前该先退 `authority / contract / seam / shell / recovery / eviction` 哪一层

## 4. 为什么这三张图必须先于功能解释

如果把它们降成后置总结，Claude Code 就容易继续被误读成：

1. Prompt 更强
2. 安全更严
3. 代码更漂亮

更成熟的解释顺序应该先声明：

1. 它首先在编译世界
2. 它首先在为扩张定价
3. 它首先在防止过去写坏现在
4. 它还分别给出了这三件事各自的判断标准，而不是只给出三组实现细节

只有把这三件事前置，后面的架构、API、哲学、指南、案例和演练才不会重新写散。

这也是为什么三张控制面之后还需要 `05 / 15 / 41`：

1. `05` 负责先防止读者抄错对象。
2. `15` 负责先点名第一条反证信号。
3. `41` 负责最后把对象链压回第一性原理。

如果少了这三步，三张图本身也会重新退回漂亮总结。

## 5. 如果从第一性原理重新设计 Agent Runtime

如果今天从零开始设计一台 Agent runtime，最先问的应当不是：

- 有哪些工具
- 能不能多 Agent
- 支不支持远程

而应当先问：

1. 现在的世界是怎样被合法编译进模型的
2. 哪些扩张要立刻收费、哪些扩张必须延后、哪些扩张必须迁出主工作集
3. 哪些旧写入者、旧状态、旧入口绝不能重新写坏现在

如果这三件事没有先成立，后续功能再多，也只是更厚的外观。

## 6. 苏格拉底式追问

在你准备宣布“我已经抓住了 Claude Code 的设计内涵”前，先问自己：

1. 我描述的是功能，还是世界进入模型的方式。
2. 我描述的是更多限制，还是扩张如何被正式定价。
3. 我描述的是分层更漂亮，还是过去怎样被正式阻止写坏现在。
4. 这些问题能不能直接拿去审一个新 runtime，而不是只拿来赞美 Claude Code。
5. 如果把功能都先删掉一半，这三张控制面还在不在。
6. 我复制的是这三张图里的约束链与判断顺序，还是只复制了措辞、面板与目录外观。

## 7. 一句话总结

Claude Code 最难被抄走的部分，不是更多功能，而是三张控制面图：它先规定世界怎样进入模型，再规定扩张怎样被定价，最后规定过去怎样不得写坏现在。
