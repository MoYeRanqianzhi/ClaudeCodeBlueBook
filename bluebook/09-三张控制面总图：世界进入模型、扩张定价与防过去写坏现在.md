# 三张控制面总图：世界进入模型、扩张定价与防过去写坏现在

Claude Code 最难被抄走的不是功能全集，而是这三条设计母线。

## 0. 三条母线

如果把 Claude Code 继续压缩到最少，它真正持续回答的只有三件事：

1. 世界如何进入模型
2. 扩张如何被定价
3. 当前如何不被过去写坏

这三件事分别对应三张控制面图：

1. 请求装配控制面
   - 见 `architecture/82`
2. 反扩张治理控制面
   - 见 `architecture/83`
3. 权威面与反僵尸控制面
   - 见 `architecture/84`

也就是说，Claude Code 真正的高级感不是：

- 它会更多功能

而是：

- 它先把这三个问题制度化了

如果再往上压一层，这三张图现在也各自形成了稳定的判断标准：

1. Prompt 线
   - 消息血缘、协议转录与继续资格是否仍对同一个世界保持诚实
2. 治理线
   - 一切扩张是否仍按同一价格秩序合法进入当前世界
3. 源码质量线
   - 当前真相是否仍被正式保护，不让过去与失控复杂度写坏现在

后续解释都应回到这三句判断，把它们当成共同校准点，而不是各自再造一版定义。

## 0.1 这三张图不是高级观察，而是设计宪法

这三张图不是读后总结，而是设计宪法。

1. 它们不是专题并列关系，而是后续一切功能与目录都要回来的宪法层。
2. 少了第一张图，Prompt 会退回更像真相的文案。
3. 少了第二张图，安全与省 token 会退回彼此分裂的局部优化。
4. 少了第三张图，源码质量会退回更漂亮的分层截图与目录体感。

所以这三张图真正值钱的地方，不是解释功能，而是规定：

- 什么问题配先成为先验约束
- 什么扩张配先进入当前世界
- 什么旧状态、旧入口与旧写入者永远不配重新篡位当前世界

## 1. 第一张图：世界如何进入模型

Claude Code 的 Prompt 魔力，真正强的不是：

- 写得更会说话

而是：

- 世界进入模型之前，已经先被编译过一轮

这里至少包含六个关键编译环节：

1. `message lineage`
2. `projection consumer`
3. `section registry / dynamic boundary`
4. `protocol transcript`
5. `continuation object / continuation qualification`
6. `cache-safe forks`

如果把第一张图压成更短的制度名，可以直接记成：

- `Request Surface Constitution`

它指的不是“Prompt 更像一部宪法”这种泛称，而是：

- 一切 request-facing surfaces 如何共同构成模型可见世界的宪法层

更稳的最小槽位至少包括：

1. `section registry`
2. `dynamic boundary`
3. `protocol transcript`
4. `continuation qualification`
5. `cache-safe fork reuse`

如果把它们按装配顺序写硬一点，可以继续压成一条更短的流水线：

1. `message lineage / section registry`
2. `projection consumer / dynamic boundary`
3. `protocol transcript normalization`
4. `lawful forgetting / continuation qualification`
5. `cache-safe forks / handoff continuation`

最短记法是：

- Prompt 魔力 = 让 message lineage 穿过 projection consumer、protocol transcript、lawful forgetting、continuation qualification 与 cache-safe forks 时，仍不丢失同一个世界

这意味着 Prompt 的本体不是：

- 一段文案

而是：

- 一条请求装配流水线

再往上压一层，这条流水线真正保护的也不是“这一轮回答更顺”，而是：

1. 当前回合活在什么世界里
2. 下一步继续时还能否留在同一个世界里
3. compact / handoff 之后是否仍能把同一个世界交给后续消费者

所以最难抄走的从来不是某段 system prompt，而是：

- display truth、protocol truth 与 handoff truth 是否仍只是同一条 message lineage 的不同投影
- 主线程产出的 prefix asset 能否继续被 `/btw`、suggestion、memory、summary 这些辅助循环复用，而不是各自重造平行世界
- fresh worker、fork worker、coordinator 与 teammate prompt 是否仍各自拿到合法但不同的 cognitive contract，而不是共用一份模糊世界

再压一层，这张图真正要保护的不是“prompt 组件很多”，而是：

- `message lineage` 不被拆散
- `projection consumer` 不被偷改
- `cache boundary` 不被免费击穿
- `handoff truth` 不被二次转述改写

如果这条流水线不存在，系统再会写 Prompt，也只是在堆更像真相的说明书。

## 2. 第二张图：扩张如何被定价

Claude Code 真正成熟的安全设计，也不是：

- 更会拦截

它更接近：

- 先阻止未定价扩张进入当前世界，再为合法扩张声明价格与顺序

这里被统一定价的至少有四个空间：

1. 动作空间
2. 权威空间
3. 上下文空间
4. 时间空间

更具体一点看，就是：

1. `mode / permission` 在给动作定价
2. `governance key / typed ask` 在给权威定价
3. `visibility / externalization / context seat` 在给上下文定价
4. `decision window / continuation pricing / durable-transient cleanup` 在给时间定价

如果把这张图写成更可排查的模板，可以直接固定成三格：

- `governance pricing chain`
  - `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable-transient cleanup`
- `role contract`
  - runtime 决定收费顺序，模型不能自己免费扩张动作、权威、上下文与时间；host 只消费 runtime 外化的 governance truth chain 与 window/pricing projections，不自己回放拼 mode、tool pool 或 pending action；人类只在升级点与拒收点介入
- `reject path`
  - 一旦叙事又退回“安全 vs token”的双话题，或者退回 mode、modal、dashboard 与 default-continue 投影，就先回四类扩张与收费对象

再往执行层压一层，这张图还必须最终交出：

- failure semantics
- escalation thresholds
- human fallback / halt conditions

否则“统一定价”仍会退回抽象口号，而不会变成可验证的治理判决。

所以安全与 token 效率共享同一治理目标。

它们共同反对的是：

- 免费扩张

这也意味着：

- “更安全”不是多几道门
- “更省 token”不是回答更短

它们只是同一价格秩序在不同资产上的投影

也就是说，Claude Code 真正在做的不是“限制模型”，而是：

- 让模型只能在已授权、已外化、已定价的当前世界里行动

## 3. 第三张图：当前如何不被过去写坏

Claude Code 的源码先进性，也不是：

- 目录更整齐
- 文件更碎

它真正先进的地方在于，它会先把这些问题结构化：

1. 当前哪一个 surface 有资格宣布真相
2. 哪些恢复资产只能帮助恢复，不能篡位
3. 哪些陈旧 finally、陈旧 snapshot、陈旧 pointer 不得写回 present state
4. 哪些编译期 / 运行期边界根本不该进入公开图谱

所以第三张图真正保护的是时间诚实性：

- 过去的写入者不得偷走现在
- 恢复资产不得假装自己就是当前真相
- future maintainer 也必须能只凭权威入口、危险改动面与拒收能力继续保护这条边界

所以这里真正被保护的不是：

- 当前分层截图

而是：

- 未来维护者继续拒绝错误实现的正式能力

这也是为什么“目录更整齐”“文件更小”本身都不构成源码先进性的证据；真正的证据是复杂度中心是否合法、边界是否可证、时间上的第二真相能否被系统性拒收。

如果把这张图也压成更可排查的模板，可以固定成三格：

- `current-truth surface`
  - `authoritative surface / recovery assets / stale snapshot-finally-pointer / danger surface / second-truth risk`
- `role contract`
  - 只有 present surface 有资格宣布当前真相；恢复资产可以帮助恢复，但不能篡位；future maintainer 是正式守门人，而不是事后旁观者
- `reject path`
  - 一旦源码质量讨论退回目录美学，或者旧状态开始重写 present state，就先回 authority surface 与 second-truth guard

## 4. 为什么这三张图必须先于功能解释

如果把它们降成后置总结，Claude Code 就容易继续被误读成：

1. Prompt 更强
2. 安全更严
3. 代码更漂亮

更成熟的解释顺序应该先声明：

1. 它首先在编译世界
2. 它首先在为扩张定价
3. 它首先在防止过去写坏现在
4. 它还分别给出了这三件事各自的判断标准，而不是只给出三组实现细节

只有把这三件事前置，后面的架构、API、哲学、指南、案例和演练才不会重新写散。

## 5. 如果从第一性原理重新设计 Agent Runtime

如果今天从零开始设计一台 Agent runtime，最先问的应当不是：

- 有哪些工具
- 能不能多 Agent
- 支不支持远程

而应当先问：

1. 现在的世界是怎样被合法编译进模型的
2. 哪些扩张要立刻收费、哪些扩张必须延后、哪些扩张必须迁出主工作集
3. 哪些旧写入者、旧状态、旧入口绝不能重新写坏现在

如果这三件事没有先成立，后续功能再多，也只是更厚的外观。

## 6. 苏格拉底式追问

在你准备宣布“我已经抓住了 Claude Code 的设计内涵”前，先问自己：

1. 我描述的是功能，还是世界进入模型的方式。
2. 我描述的是更多限制，还是扩张如何被正式定价。
3. 我描述的是分层更漂亮，还是过去怎样被正式阻止写坏现在。
4. 这些问题能不能直接拿去审一个新 runtime，而不是只拿来赞美 Claude Code。
5. 如果把功能都先删掉一半，这三张控制面还在不在。
6. 我复制的是这三张图里的约束链与判断顺序，还是只复制了措辞、面板与目录外观。

## 7. 一句话总结

Claude Code 最难被抄走的部分，不是更多功能，而是三张控制面图：它先规定世界怎样进入模型，再规定扩张怎样被定价，最后规定过去怎样不得写坏现在。
