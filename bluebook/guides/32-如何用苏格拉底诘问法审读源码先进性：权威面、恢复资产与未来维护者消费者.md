# 如何用苏格拉底诘问法审读源码先进性：current-truth surface、current-truth writeback 与 future maintainer 可独立 reject

这一章不再解释“为什么先进”，而是把“怎么审它是否真的先进”写成可执行的证据梯度。

它主要回答五个问题：

1. 怎样避免把源码先进性写回目录审美和大文件焦虑。
2. 怎样按顺序审读 current-truth surface、contract-first、writeback 边界、recovery assets 与 anti-zombie。
3. 怎样让 later maintainer 用同一套证据独立通过或拒绝。
4. 怎样在新增能力、迁移和事故复盘中复用同一条诘问链。
5. 怎样避免把“源码质量”退回术语表。

## 1. 第一性原理

更先进的源码，不是今天更顺眼，而是明天更难偷偷说谎。

所以苏格拉底诘问的目标不是“解释作者意图”，而是逼出两件事：

1. 可见证据是否足以支持当前结构。
2. later maintainer 是否拥有独立 reject 的最低权利。

## 2. 苏格拉底诘问顺序：六层证据梯度

### 2.1 G1：current-truth surface（先问真相入口）

核心问题：

1. 哪个入口定义 enabled / in-flight / transcript 边界的当前真相。
2. 有没有多个入口在并行定义同一状态。

必收证据：

- 单一入口函数、单一状态面、单一字段约束。

拒绝条件：

- 找不到唯一入口，或 UI/缓存推断正在“代替真相定义”。

### 2.2 G2：contract-first（再问是否先有合同）

核心问题：

1. 这条能力先进入 schema/union，还是先散落到调用点。
2. 消费者看到的是同一真相还是不同投影。

必收证据：

- contract 变更点、调用点映射、投影边界说明。

拒绝条件：

- 先补分支再补合同，或消费者被迫共享同一原始表示。

### 2.3 G3：current-truth writeback + adapter subset（再问谁能回写、入口实现哪段）

核心问题：

1. 谁有资格回写主状态，写回前后有什么 freshness gate。
2. 当前 adapter 是协议全集的哪个子集，差异是策略还是遗留。

必收证据：

- writeback gate、generation/fate freeze 规则、subset 对照表。

拒绝条件：

- 任意 consumer 都可回写，或 subset 边界只存在口头解释。

### 2.4 G4：leaf/seam/shell（再问复杂度是否被隔离）

核心问题：

1. 高复用模块是否真 leaf，还是隐藏控制面。
2. anti-cycle seam 是真切断，还是换形态延迟爆炸。
3. transport 差异是否已被收进 transport shell。

必收证据：

- 依赖图、跨层调用路径、transport 差异封装点。

拒绝条件：

- 重依赖 util、隐式回调维持切环、carrier 细节泄漏到语义层。

### 2.5 G5：recovery + anti-zombie（再问时间维度是否诚实）

核心问题：

1. 哪些状态必须跨进程/崩溃/resume 存活，它们是不是正式 recovery asset。
2. async gap 下 stale writer 会不会复活 zombie。

必收证据：

- pointer/snapshot/log/metadata 登记、fresh-state merge 规则。

拒绝条件：

- 恢复依赖“重新猜”，或 stale writer 没有被明确阻断。

### 2.6 G6：mirror gap + reject/retreat（最后问维护者权利）

核心问题：

1. 哪些是可见证据，哪些只是公开镜像下的保守推断。
2. later maintainer 的最小 reject 规则是否成文。
3. 结构失真时先退哪层：`current-truth surface / contract / seam / shell / recovery / writeback eviction`。

必收证据：

- reject 语句模板、退层顺序、shim/compat/stub/shadow 的退出条件。

拒绝条件：

- 只能“以后再重构”，没有可执行退层与退出条件。

## 3. 常见自欺

看到下面信号时应立刻停下并回到上一关补证据：

1. 用目录精细化掩盖多处半真相。
2. 用“统一复用”掩盖 writeback 越权和副作用污染。
3. 用“抽象优雅”掩盖 seam/shell 已失守。
4. 用“先兼容”掩盖 shim、stub、shadow 无退出路径。
5. 用“偶发 race”掩盖 recovery asset 与 anti-zombie 规则缺位。
6. 用“作者熟悉”替代 later maintainer 的制度化 reject 路径。

## 4. 更好的迭代顺序

当任一关答不清时，按这个顺序回修：

1. 先补 G1/G2：锁定 current-truth surface 与 contract 证据。
2. 再补 G3：收紧 writeback gate，并把 adapter subset 写成显式对照。
3. 再补 G4/G5：切 seam、收 shell、登记 recovery assets、补 anti-zombie。
4. 最后补 G6：写明最小 reject 语句与退层顺序。

## 6. 苏格拉底式检查清单

在继续“整理代码”前，先问自己：

1. 我是在收紧 current-truth surface，还是只在改善目录表面。
2. 我是在明确 writeback 资格，还是在默认所有人都能改主状态。
3. 我是在减少未来误改路径，还是把纠偏压力推给后人。
4. 这个结构出事时，团队能否按 G1-G6 证据反查，而不是靠作者记忆。
5. 我能否给 later maintainer 一句明确的 reject 理由和退层动作。
