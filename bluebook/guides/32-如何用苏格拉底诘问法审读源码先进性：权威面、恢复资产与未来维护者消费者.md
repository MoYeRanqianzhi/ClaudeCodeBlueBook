# 如何用苏格拉底诘问法审读源码先进性：current-truth surface、证据梯度与 future maintainer 可独立 reject

这一章不再解释 Claude Code 的源码为什么先进，而是把“怎么审它是否真的先进”压成一套 builder-facing 框架。

它主要回答五个问题：

1. 怎样避免把源码先进性写回目录审美和大文件焦虑。
2. 怎样用递进式问题审读 current-truth surface、contract-first、leaf module、transport shell、mirror gap discipline 与 anti-zombie 协议。
3. 怎样判断一段源码是否已经把批评路径提前编码进结构。
4. 怎样在新增能力、迁移和事故复盘中使用同一组问题自我校准。
5. 怎样用苏格拉底式追问避免把“源码质量”退回空泛评价。

这些锚点共同说明：

- Claude Code 的源码先进性不是“更漂亮”，而是把 current-truth surface、证据梯度、时间诚实与未来维护路径一起写进结构。

## 1. 第一性原理

更先进的源码，不是：

- 今天更好看

而是：

- 明天更不容易偷偷说谎

所以审读源码先进性时，最该反问的不是：

- 目录是否更像教科书

而是：

- 如果今天有人改错，这个结构有没有提前暴露反对意见与纠偏路径

## 2. 苏格拉底诘问链

### 2.1 这里真正定义当前真相的 current-truth surface 是什么

判断标准：

- 如果不能指出单一入口、单一状态面或单一 choke point，就仍在多处半真相上运行。

### 2.2 这条能力先被写成 contract，还是先被散落到多个调用点

判断标准：

- 如果新增能力主要靠补 if、补字段、补分支进入系统，而不是先进入统一合同面，就不可持续。

### 2.3 同一对象被哪些消费者看到，它们看到的应是同一真相还是不同投影

判断标准：

- 如果 model、UI、search、host 被迫共享一份原始表示，通常说明边界还没被编译干净。

### 2.4 这个模块是真 leaf module，还是只是被叫做 util 的隐藏控制面

判断标准：

- 高复用模块若仍携带重依赖、初始化副作用或 UI/runtime 背景包袱，就不是合格 leaf。

### 2.5 这里的 anti-cycle seam 是真切断，还是把循环依赖推迟到别处爆炸

判断标准：

- 如果切环后仍需靠隐式回调、共享全局、跨层偷拿对象维持运行，只是换了形状，没有降复杂度。

### 2.6 transport 差异是否被收进 transport shell，还是已蔓延到业务层

判断标准：

- `v1/v2`、`local/remote`、`structured/streaming` 的差异若到处可见，说明外壳失守，语义层已被 carrier 污染。

### 2.7 哪些状态必须跨进程、跨崩溃、跨 resume 存活，它们被明确做成 recovery asset 了吗

判断标准：

- 如果恢复依赖“重新猜”，而不是依赖 pointer、snapshot、replacement log、metadata 等正式资产，就不算 recovery-ready。

### 2.8 这里有没有 stale writer 把旧对象复活成 zombie 的可能

判断标准：

- 只要存在 async gap，就必须追问 generation、fresh-state merge、fate freeze 是否到位；否则 zombie 迟早出现。

### 2.9 如果删掉一个 shim、compat、stub 或 shadow，这个系统靠什么证明还能成立

判断标准：

- 没有退出条件、回归信号和演练路径的过渡层，最终都会从迁移工具变成永久地层。

### 2.10 这里哪些是可见证据，哪些仍只是公开镜像下的保守推断

判断标准：

- 如果 README、schema、registry、adapter subset、hotspot 与 mirror gap 被混成同一层真相，源码质量判断就会重新退回作者体感，而不是证据梯度。

### 2.11 未来维护者进入这里，依赖的是源码制度，还是依赖口头传承

判断标准：

- 如果关键判断只存在于 PR、Slack、作者记忆里，而不在命名、注释、台账和 checklist 中，就还没把维护者当正式消费者。

### 2.12 这个设计失败时，团队能按症状、阶段、资产反查到根因吗

判断标准：

- 如果故障只能靠“熟人直觉”定位，而不能通过统一标签和入口反查，先进性仍停留在个人能力，不是系统能力。

### 2.13 这段结构是在服务当前功能，还是在保留下一次重构的合法走线

判断标准：

- 如果它只解释今天为什么能跑，却不解释明天如何替换、收口、退出与演练，就不是成熟 runtime 结构。

### 2.14 这里真正防的是哪类 failure object

判断标准：

- 如果一段结构只能被解释成“更优雅”，却说不清它在防 `split truth / stale state / zombification / transport leakage / registry obesity` 里的哪一类对象，这段先进性仍停在 happy-path 工程法。

### 2.15 later maintainer 的最小 reject 规则是否已写清

判断标准：

- 如果 later maintainer 看见 shim 无退出条件、snapshot 可回写主状态、旧 capability token 还能继续放权，却没有明确理由拒绝这次结构升级，那 future maintainer 仍不是正式消费者。

### 2.16 这段结构坏掉时先退哪一层

判断标准：

- 如果一旦失真，只能说“之后再重构”，却说不清应先退 `authority / contract / seam / shell / recovery / eviction` 哪一层，可演化就仍是方向，不是协议。

## 3. 常见自欺

看到下面信号时，应提高警惕：

1. 用更细目录掩盖更散的真相。
2. 用“统一复用”掩盖重依赖和副作用污染。
3. 用“抽象更优雅”掩盖 anti-cycle seam 已经失效。
4. 用“先兼容一下”掩盖 shadow、stub、shim 没有退出路径。
5. 用“偶发 race”掩盖对象命运、恢复与 anti-zombie 设计尚未成立。
6. 用“工程习惯很好”掩盖 invariant、failure object 与 reject path 仍未成文。

## 4. 更好的迭代顺序

当这组问题里有任何一个答不清时，优先做下面四步：

1. 先回 `../philosophy/65` 与 `../philosophy/62`，判断自己缺的是 current-truth surface、批评路径，还是恢复与传输壳层。
2. 再回 `19`、`20`、`29`，检查 contract-first、dependency honesty、build surface 与 anti-zombie 协议是否已经松掉。
3. 再回 `../navigation/14` 与 `../casebooks/08-09`，确认当前失效是否能被阶段与资产正确定位。
4. 最后才决定是重写 contract、切 seam、收 transport shell，还是补 recovery asset 与演练路径。

## 6. 苏格拉底式检查清单

在你准备继续“把代码整理漂亮”前，先问自己：

1. 我是在改善真相结构，还是只是在改善目录表面。
2. 我保住的是可演化边界，还是只是短期可运行性。
3. 我是在减少未来误改路径，还是在制造更隐蔽的误改点。
4. 我是在提前编码批评路径，还是在把批评推给未来的人。
5. 这个结构如果出事，团队能否沿着制度对象而不是作者记忆复盘。
6. 我现在夸的是源码地图、证据梯度与时间诚实，还是又退回了目录体感。
7. 我能否说清这段结构替 later maintainer 多保留了哪条独立 reject 权。
