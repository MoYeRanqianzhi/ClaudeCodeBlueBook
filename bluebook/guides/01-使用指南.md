# Claude Code 使用指南

本指南不是基于营销文案，而是基于源码结构反推“怎么用更对”。

## 1. 先选运行形态，再开始工作

从入口代码可以看出，Claude Code 有明显的多形态设计。

建议按任务类型选择：

- 快速查询、脚本调用：非交互模式
- 长会话、探索式开发：REPL
- 长时间 shell / 测试 / 构建：后台任务
- 高风险改动：worktree 隔离
- 大任务拆分：subagent / coordinator
- 外部系统接入：MCP

错误用法通常是：把所有事都塞进一个前台会话里硬跑。

## 2. 学会把命令与对话混用

源码说明 slash commands 是一等公民，不是补充玩具。

实际心法：

- 对“控制工作流”的事情，优先想命令
- 对“表达目标与约束”的事情，优先想自然语言

比如：

- 配置、恢复、模型切换、权限调整、插件技能管理，用命令
- 需求、验收标准、设计取舍，用自然语言

## 3. 长任务不要硬阻塞

从 `BashTool` 和 `AgentTool` 看，Claude Code 明显鼓励后台化。

建议：

- 长 shell 命令使用后台执行
- 复杂多步骤任务交给子代理
- 让主会话保留为协调与决策层

这也是避免上下文污染的核心方法。

## 4. 把权限模式当成设计工具

权限不是负担，而是任务边界描述。

建议：

- 阅读/分析任务：用更保守的权限边界
- 确认后的实现任务：逐步放开编辑权限
- 高风险场景：优先 worktree，而不是直接提高主会话权限

好的使用方式不是一上来就最大权限，而是按任务阶段逐步扩大行动空间。

## 5. 把技能当成可复用流程，而不是收藏夹

从技能系统的 frontmatter 设计可知，技能应该承载：

- 触发场景
- 工具边界
- 参数提示
- 必要时的 agent/context 策略

所以技能适合沉淀：

- 固定审查流程
- 固定发布流程
- 固定代码生成模板
- 固定排障 SOP

## 6. 学会利用上下文压缩，而不是害怕它

源码里有 snip、microcompact、autocompact、context collapse。

这意味着：

- Claude Code 天生不是无限上下文容器
- 但它也不是超长会话就必死

正确做法是：

- 把关键约束写进持久文档
- 让技能、文件、计划、任务列表承载长期状态
- 不要指望所有细节永远只存在于聊天记录里

再补一条很实用的心法：

- 省 token 的正确方法，不是逼自己少说话，而是少让无关能力、无关输出和无关输入进入当前会话

这在实践上通常意味着：

- 长输出交给后台任务或子代理消化
- 无关工具不要硬塞进同一轮任务
- 外部 channel 输入只在确实需要时显式 opt-in

## 7. 多代理什么时候值得用

多代理适合三类任务：

1. 并行研究
2. 实现与验证解耦
3. 高风险任务隔离

不适合：

- 只需要读一个文件的小问题
- 只为了“显得高级”的机械拆分
- 多个代理抢同一批文件同时写

## 8. 推荐工作流

一个更贴近源码设计的工作流是：

1. 主线程澄清目标与约束
2. 并行子代理做研究
3. 主线程综合并产出实施方案
4. 单代理或有限代理做实现
5. 独立代理做验证
6. 重要结论回写文档

## 9. 一句话总结

Claude Code 最佳用法不是“多说一点提示词”，而是“用对运行形态、工具边界、后台任务和子代理分工”。

## 10. 团队场景下，不要把治理开关当附属配置

如果你在团队或企业环境里使用 Claude Code，还要额外记住一条：

- 托管策略、allowlist、会话 opt-in 也是运行时的一部分

尤其是 channels 这类会把外部消息带入当前会话的能力，更不该按“先接上再说”处理。

更稳的做法是：

1. 管理员先定义组织级边界
2. 用户再在单次会话里显式 opt-in
3. 把外部输入默认当作不可信来源
4. 把治理与 token 预算一起看，外部输入越多，越要控制主线程的上下文负担

这部分的实操路径见：

- `guides/04-Channels、托管策略与组织级治理实践.md`

## 11. 不要把 Prompt 写成一段话，而要写成五层合同

更接近 Claude Code 设计方式的写法是：

1. system 层写稳定原则
2. developer 层写宿主覆写和附加约束
3. user 层写本轮目标与验收标准
4. tool 层让能力目录和 delta 自己说话
5. channel 层显式区分普通输入、任务通知与外部协作消息

最差的做法则是把这些层混成一大段自然语言，既不稳缓存，也不利于长期复用。

## 12. 真正的省 token，不是少说话，而是稳定前缀和外置噪音

从源码看，更稳的省 token 方法是：

1. 把长期规则放进稳定文档或稳定 system 层
2. 让大块输出留在文件、任务输出或工具结果外置路径
3. 不要每轮都重述工具目录、团队成员和环境状态
4. 接受 compact / reactive compact 是运行时主路径，而不是异常

所以高质量使用方式，不是拼命缩短每句话，而是：

- 让真正会波动的内容少进入静态 prompt 主体

## 13. 研究或扩展 Claude Code 时，先找 contract，再找热点文件

如果你是在读源码、做宿主接入或设计同类系统，最稳的顺序是：

1. 先找 schema、union、task type、control request 这类正式合同
2. 再找 query、main、attachments 这类热点文件在维护什么 invariant
3. 最后再讨论公开镜像的缺口和技术债

这样更不容易掉进两种误区：

- 看见大文件就说架构差
- 看见缺模块就直接脑补完整内部实现
