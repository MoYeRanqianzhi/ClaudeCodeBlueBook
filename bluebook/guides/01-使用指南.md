# Claude Code 使用指南

本指南不是基于营销文案，而是基于源码结构反推“怎么用更对”。

## 1. 先选运行形态，再开始工作

从入口代码可以看出，Claude Code 有明显的多形态设计。

建议按任务类型选择：

- 快速查询、脚本调用：非交互模式
- 长会话、探索式开发：REPL
- 长时间 shell / 测试 / 构建：后台任务
- 高风险改动：worktree 隔离
- 大任务拆分：subagent / coordinator
- 外部系统接入：MCP

错误用法通常是：把所有事都塞进一个前台会话里硬跑。

## 2. 学会把命令与对话混用

源码说明 slash commands 是一等公民，不是补充玩具。

实际心法：

- 对“控制工作流”的事情，优先想命令
- 对“表达目标与约束”的事情，优先想自然语言

比如：

- 配置、恢复、模型切换、权限调整、插件技能管理，用命令
- 需求、验收标准、设计取舍，用自然语言

## 3. 长任务不要硬阻塞

从 `BashTool` 和 `AgentTool` 看，Claude Code 明显鼓励后台化。

建议：

- 长 shell 命令使用后台执行
- 复杂多步骤任务交给子代理
- 让主会话保留为协调与决策层

这也是避免上下文污染的核心方法。

## 4. 把权限模式当成设计工具

权限不是负担，而是任务边界描述。

建议：

- 阅读/分析任务：用更保守的权限边界
- 确认后的实现任务：逐步放开编辑权限
- 高风险场景：优先 worktree，而不是直接提高主会话权限

好的使用方式不是一上来就最大权限，而是按任务阶段逐步扩大行动空间。

## 5. 把技能当成可复用流程，而不是收藏夹

从技能系统的 frontmatter 设计可知，技能应该承载：

- 触发场景
- 工具边界
- 参数提示
- 必要时的 agent/context 策略

所以技能适合沉淀：

- 固定审查流程
- 固定发布流程
- 固定代码生成模板
- 固定排障 SOP

## 6. 学会利用上下文压缩，而不是害怕它

源码里有 snip、microcompact、autocompact、context collapse。

这意味着：

- Claude Code 天生不是无限上下文容器
- 但它也不是超长会话就必死

正确做法是：

- 把关键约束写进持久文档
- 让技能、文件、计划、任务列表承载长期状态
- 不要指望所有细节永远只存在于聊天记录里

## 7. 多代理什么时候值得用

多代理适合三类任务：

1. 并行研究
2. 实现与验证解耦
3. 高风险任务隔离

不适合：

- 只需要读一个文件的小问题
- 只为了“显得高级”的机械拆分
- 多个代理抢同一批文件同时写

## 8. 推荐工作流

一个更贴近源码设计的工作流是：

1. 主线程澄清目标与约束
2. 并行子代理做研究
3. 主线程综合并产出实施方案
4. 单代理或有限代理做实现
5. 独立代理做验证
6. 重要结论回写文档

## 9. 一句话总结

Claude Code 最佳用法不是“多说一点提示词”，而是“用对运行形态、工具边界、后台任务和子代理分工”。
