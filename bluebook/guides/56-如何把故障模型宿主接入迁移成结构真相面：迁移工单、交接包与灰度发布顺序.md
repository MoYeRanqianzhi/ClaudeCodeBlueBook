# 如何把故障模型宿主接入迁移成 current-truth surface：writeback 主路径、recovery asset non-sovereignty、freshness gate 与灰度发布顺序

这一章不再解释故障模型宿主接入该怎样审读，而是把 Claude Code 式结构迁移压成一张可执行的宿主迁移工单。

它主要回答五个问题：

1. 为什么故障模型迁移真正要迁的不是恢复成功率、目录图和 pointer 文件，而是 current-truth surface。
2. 怎样把 `current-truth surface`、`writeback path`、`recovery asset non-sovereignty`、`freshness gate` 与 `anti-zombie evidence` 写成固定实施顺序。
3. 怎样把交接从“这个模块负责恢复”升级成正式故障窗口交接包。
4. 怎样安排灰度发布，避免宿主先学会操纵恢复，却还没学会消费真相面。
5. 怎样用苏格拉底式追问避免把迁移写成“把恢复逻辑搬过来”。

## 1. 第一性原理

故障模型宿主迁移真正要迁的不是：

- spinner 状态
- pointer 文件
- 恢复成功率

而是：

- 围绕同一个 current-truth surface 成立的 writeback 主路径、合法转移、recovery asset non-sovereignty 与 anti-zombie 结果面

所以更稳的迁移目标不是：

- 让宿主“也能看到恢复过程”

而是：

1. 宿主知道当前 current-truth surface 从哪里宣布。
2. 宿主知道哪些状态转移是合法的。
3. 宿主知道恢复资产只是资产，不是第二权威面。
4. 宿主知道 stale writer / stale merge 的结果面是什么。
5. 宿主知道远端写回与恢复是否仍围绕同一主路径成立。

更进一步说，迁移顺序本身也是源码先进性的高阶判据：

- 如果团队总是先开放恢复控制、后统一 current truth，这套结构就还没有成熟到能安全迁移。

## 2. 先识别旧宿主还在消费什么

在开始迁移前，先判断旧宿主是否仍在消费这些假信号：

1. 用 `isLoading`、spinner、按钮状态猜当前 current-truth surface。
2. 把 pointer、resume file、breadcrumb 神化成当前真相。
3. 只看恢复成功率，不看 stale writer、rejected outcome 与 boundary 变化。
4. 交接仍围绕“哪个模块负责恢复”，而不是围绕故障窗口与对象边界。
5. 远端状态写回仍只是日志，不是正式恢复资产。

如果这些症状还在，迁移顺序就不能从恢复面板开始，而应先从 single-source current-truth surface 开始。

## 3. 迁移工单顺序

### 3.1 先冻结 current-truth surface 与 writeback 主路径

第一张工单要做的不是补更多恢复按钮，而是明确：

1. `QueryGuard + sessionState + onChangeAppState` 谁在宣布当前 current-truth surface。
2. 宿主如何只消费正式状态投影，而不是自己猜状态。
3. 本地状态怎样通过 writeback 主路径进入远端恢复资产。

最小交付物：

1. `current_truth_surface`
2. single-source writeback 图
3. 宿主只读状态投影

### 3.2 再补 transition legality 与 anti-zombie 证据

第二张工单要补的不是更多 debug 输出，而是：

1. `dispatching / running / idle / requires_action` 的合法转移。
2. `registerTask`、fresh-state patch、terminal legality 的正式证据。
3. stale writer / stale merge 被拒收的结果面。

最小交付物：

1. `transition_legality_snapshot`
2. `anti_zombie_projection`
3. `stale_writer_evidence`

### 3.3 再补 recovery asset non-sovereignty

第三张工单要补的是：

1. 哪些对象只是 recovery asset。
2. pointer、ledger、append chain 怎样只负责帮助回到主权对象。
3. 409 adopt、401 fail-fast、epoch rebuild 各自属于哪一类恢复资产非主权边界。

最小交付物：

1. `recovery_asset_non_sovereignty`
2. `recovery_asset_inventory`
3. `adopt_or_fail_reason`

### 3.4 再补 remote ingress 与 bridge 灰度面

第四张工单要补的是：

1. `sessionIngress` 对 stale local head / concurrent writer 的正式处理。
2. `remoteBridgeCore` 对 401、epoch mismatch、flush gate 的正式处理。
3. 远端 worker state 是否仍沿统一 writeback 主路径写回。

最小交付物：

1. `remote_recovery_window`
2. `epoch_rebuild_projection`
3. `writeback_delivery_state`

### 3.5 最后才放开恢复控制与旧路径下线

第五张工单要补的是：

1. 宿主何时可以真正触发 resume / rollback / adopt。
2. 旧的成功率面板、按钮 heuristics 何时下线。
3. later maintainer 是否只靠投影就能理解当前故障边界。

最小交付物：

1. `handoff_reentry_condition`
2. `rollback_object`
3. 旧路径下线清单

不要反过来做：

1. 不要先开放恢复控制，再补 current-truth surface。
2. 不要先展示 pointer，再补 recovery asset non-sovereignty。
3. 不要先报成功率，再补 anti-zombie 结果面。
4. 不要先下线旧路径，再补 writeback 主路径观测。

## 4. 交接包最小字段

故障模型宿主迁移后的交接包至少要有：

1. `kernel_object_id`
2. `current_truth_surface`
3. `transition_legality_snapshot`
4. `anti_zombie_projection`
5. `stale_writer_evidence`
6. `recovery_asset_non_sovereignty`
7. `recovery_asset_inventory`
8. `writeback_delivery_state`
9. `rollback_object`
10. `degraded_path`
11. `reject_reason`

少了这些字段，交接就仍会退回：

- 当前大概恢复好了
- pointer 还在

## 5. 灰度发布顺序

更稳的结构灰度顺序是：

1. `writeback + state projection shadow`
   - 先灰度 current-truth surface 与 writeback 真相面。
2. `resume/read path shadow`
   - 再灰度 recovery asset non-sovereignty 与 reentry condition。
3. `anti-zombie projection`
   - 再让宿主看见 stale writer / rejected outcome。
4. `remote control path`
   - 最后才放开 bridge / ingress / resume 等控制权。
5. `retire legacy heuristics`
   - 最后移除 spinner、按钮状态与成功率主导的旧面板。

最危险的错误顺序是：

1. 先开放 resume / rollback。
2. 先展示 pointer。
3. 最后才补 current-truth surface 与 anti-zombie 结果面。

这会直接制造：

- split-brain
- zombie outcome
- pointer 篡位

## 6. 直接拒收条件

出现下面情况时，应直接拒收迁移版本：

1. 宿主仍主要靠 UI heuristic 猜 current-truth surface。
2. 宿主把 pointer 或恢复文件当当前真相。
3. 宿主只展示恢复成功率，不展示 anti-zombie 结果面。
4. 远端状态写回仍只是日志，不是正式 writeback 主路径。
5. later maintainer 仍必须问作者“这次到底算不算恢复好了”。

## 7. 苏格拉底式检查清单

在你准备宣布“故障模型宿主迁移已经完成”前，先问自己：

1. 我迁移的是 current-truth surface，还是只迁移了恢复操作。
2. current-truth surface 在我的系统里还是正式单一来源，还是又被 UI 猜测夺权。
3. recovery asset 还只是资产，还是已经在篡位。
4. 我看到的是 anti-zombie 结果面，还是只看到成功率。
5. later maintainer 是否真的可以只靠交接包与宿主投影理解故障边界。

## 8. 一句话总结

真正成熟的故障模型宿主迁移，不是把恢复逻辑搬进宿主，而是把 current-truth surface、合法转移、anti-zombie 证据、recovery asset non-sovereignty 与 writeback 主路径一起迁成同一个 current-truth surface。
