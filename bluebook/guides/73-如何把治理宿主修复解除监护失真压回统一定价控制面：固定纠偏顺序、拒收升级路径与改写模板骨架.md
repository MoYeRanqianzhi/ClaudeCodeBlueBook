# 如何把治理宿主修复解除监护失真压回统一定价控制面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释治理宿主修复解除监护执行最常怎样失真，而是把 Claude Code 式治理 release correction 压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么治理 release correction 真正要救回的不是一套更谨慎的 capability release 流程，而是同一个统一定价治理对象。
2. 怎样把免责式出监、假清账与免费继续压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续补 release 文案。
4. 怎样把 release card、capability release 与 reopen liability 重新压回治理对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把看板与审批流程再做严格一点”。

## 1. 第一性原理

治理 release correction 真正要救回的不是：

- 更保守的 capability release checklist
- 更细的放行审批
- 更谨慎的 later 注意事项

而是：

- 围绕同一个治理对象成立的统一定价控制面

所以更稳的纠偏目标不是：

- 先把放行流程做得更谨慎

而是：

1. 先把 `authority_release` 从 mode、status 与 dashboard 里救出来。
2. 先把 `ledger_clearance` 从值班备注与背景噪音里救出来。
3. 先把 `window_exit` 从 usage 回落与图表平静感里救出来。
4. 先把 `continuation_settlement` 从默认继续里救出来。
5. 先把 `quarantine_release` 与 `reopen_liability` 从免责说明里救出来。

## 2. 固定纠偏顺序

### 2.1 先拆掉假 authority release

第一步不是补 release 备注，而是：

1. 把 mode、status、dashboard 降回 authority 的投影。
2. 恢复 `authority_source`、`effective_settings_projection` 与 `writer_chokepoint` 的单一权威。
3. 禁止 UI 恢复感充当 `authority_release`。

### 2.2 再恢复 ledger clearance

第二步要救回：

1. `typed_decision_digest`
2. `permission_ledger_state`
3. `pending_permission_requests`
4. duplicate / orphan / late residue 的正式清账
5. `ledger_cleared_at`

这一步不恢复，later 继承的就不是 release，而是一条未结算时间线。

### 2.3 再恢复 window exit

第三步要修的是：

1. `decision_window`
2. `pending_action`
3. `context_usage_snapshot`
4. `reserved_buffer`
5. `late_response_quarantined`
6. `window_exited_at`

不要把 usage 回落、compact 执行或图表转绿当成 `window_exit`。

### 2.4 再恢复 continuation settlement

第四步要把继续从默认行为拉回正式结算：

1. `continuation_gate`
2. `token_budget_result`
3. `completion_event`
4. `settlement_due_at`
5. `capability_release_scope`

这一步不恢复，继续就会重新免费。

### 2.5 最后恢复 quarantine release 与 reopen liability

最后才修：

1. `quarantine_scope`
2. `quarantine_release`
3. `reopen_liability`
4. `handoff_status`
5. `release_verdict`

不要反过来先放 capability、先交给 later、再补 liability。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_release` 仍由 mode、面板或经验判断推断。
2. `ledger_clearance` 缺席，或 duplicate / orphan / pending residue 仍被当成背景噪音。
3. `window_exit` 只有 usage 回落，没有 `late_response_quarantined`。
4. `continuation_settlement` 仍由“没报错、还能跑”裁定。
5. capability 已释放，但 `quarantine_scope` 与 `reopen_liability` 仍未写清。
6. `handoff_status` 不能让 later 团队仅凭 release artifact 重建同一治理判断。
7. reopen 仍允许沿用旧 ledger、旧 window、旧 continuation 直接续跑。
8. `release_verdict` 先于 authority / ledger / settlement 复核生效。

## 5. 苏格拉底式检查清单

在你准备宣布“治理 release distortion 已纠偏完成”前，先问自己：

1. 我救回的是治理对象，还是一套更严格的放行流程。
2. 我现在释放的是 authority，还是释放值班压力。
3. 我现在清掉的是 formal ledger，还是接受尾账常态化。
4. 我现在退出的是同一个 decision window，还是一张 usage 图。
5. 我现在结算的是 continuation，还是给默认继续换了个更体面的名字。
6. 我现在保留的是 reopen liability，还是把责任礼貌地外包给 later。

## 6. 一句话总结

真正成熟的治理宿主修复解除监护纠偏，不是把 capability release 写得更保守，而是把免责式出监、假清账与免费继续重新压回同一个统一定价控制面。
