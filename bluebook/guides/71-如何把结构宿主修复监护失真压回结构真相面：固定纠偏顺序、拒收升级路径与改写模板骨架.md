# 如何把结构宿主修复监护失真压回结构真相面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释结构宿主修复监护执行最常怎样失真，而是把 Claude Code 式结构监护纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构监护纠偏真正要救回的不是更顺手的看板，而是同一个结构真相面。
2. 怎样把假稳定、假冻结与假重开压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续观察。
4. 怎样把 watch card、handoff freeze 与 reopen ticket 重新压回结构对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“补几条更精致的观测日志”。

## 1. 第一性原理

结构监护纠偏真正要救回的不是：

- pointer 文件
- 监控转绿
- PUT 成功数
- 作者说明

而是：

- 围绕同一个结构真相面成立的 authority object、resume order、writeback path、anti-zombie watch 与 recovery boundary

所以更稳的纠偏目标不是：

- 先让恢复看起来更稳

而是：

1. 先把 authority watch 从 pointer 与 UI 活性里救出来。
2. 先把 resume regression watch 从 reconnect 成功感里救出来。
3. 先把 writeback watch 从 telemetry 繁荣里救出来。
4. 先把 anti-zombie watch 从作者说明与成功率里救出来。
5. 先把 freeze 与 reopen ticket 从旧路径、旧入口与 retry 幻觉里救出来。

这些纠偏之所以本质上是在恢复源码先进性的真正对象边界，是因为：

- Claude Code 的先进性不在“看起来更稳”，而在 authority、single-source、stale-safe 与 anti-zombie 被提前编码进结构。
- 一旦 breadcrumb、telemetry 与作者口述重新上位，源码先进性就会退回目录美学与监控运营。

## 2. 固定纠偏顺序

### 2.1 先 demote 假稳定

第一步不是补看板，而是：

1. demote pointer、sidecar、旧路径与监控转绿，让它们回到辅助信号。
2. 恢复 `authority_watch`、`authority_state_surface` 与 `seal_generation` 的统一真相面。
3. 禁止健康感再次充当 watch 主对象。

### 2.2 再恢复 resume regression watch

第二步要救回：

1. `resume_closure_order`
2. `late_resume_detected`
3. `stale_adoption_blocked`
4. `generation_regression_detected`
5. `order_regression_cleared`

只要回归被成功感掩盖，后面的稳定再高都仍是伪稳定。

### 2.3 再恢复 writeback 主路径

第三步要修的是：

1. `writeback_path`
2. `worker_status`
3. `external_metadata`
4. `merge_semantics`
5. `side_write_detected`

不要继续做的事：

1. 不要把写回当普通 telemetry。
2. 不要只看 PUT 成功率。
3. 不要把监控转绿当成真相对齐。

### 2.4 再恢复 anti-zombie watch

第四步要救回：

1. `anti_zombie_projection`
2. `stale_writer_evidence`
3. `duplicate_or_orphan_control_state_resolved`
4. `transition_legality_snapshot`
5. `regression_detected`

只要这一步没修好，later 团队就仍会继承一条假连续时间线。

### 2.5 最后恢复 freeze、boundary 与 reopen

最后才修：

1. `boundary_quarantine`
2. `recovery_boundary`
3. `handoff_status`
4. `re_entry_condition`
5. `reopen_trigger`

不要反过来：

1. 不要先修 pointer 展示。
2. 不要先修监控面板。
3. 不要先让 later 团队“再连一次试试看”。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_watch` 仍由 pointer、sidecar 与 dashboard 推断。
2. `resume_regression_watch` 缺席，或 reconnect 成功感继续替代顺序观察。
3. `writeback_watch` 仍只看 telemetry、PUT 成功数与监控转绿。
4. `anti_zombie_watch` 仍主要依赖作者说明与成功率。
5. `boundary_quarantine` 缺席，或 freeze 仍只停在备注。
6. `reopen_trigger` 缺席，而重开仍退回旧路径、旧 sidecar 与 retry。
7. `handoff_status` 不能让 later 团队仅凭 watch card 重建同一结构对象。
8. `watch_verdict` 继续被健康感、监控与作者保证主导。

## 5. 苏格拉底式检查清单

在你准备宣布“结构监护失真已经纠偏完成”前，先问自己：

1. 我救回的是结构真相面，还是稳定错觉。
2. 我现在消费的是 authority object，还是 pointer、spinner 与 sidecar。
3. writeback 在我的系统里是真相主路径，还是普通日志。
4. 我看到的是 anti-zombie watch，还是作者说明与成功率。
5. freeze 与 reopen 在我的系统里回的是 recovery boundary，还是旧路径与重试习惯。

## 6. 一句话总结

真正成熟的结构监护纠偏，不是把 pointer、日志和监控修得更漂亮，而是把假稳定、假冻结与假重开重新压回同一个结构真相面。
