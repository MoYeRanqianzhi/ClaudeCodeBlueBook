# 如何把治理宿主修复稳态纠偏执行失真压回统一定价控制面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释治理宿主修复稳态纠偏执行最常怎样失真，而是把 Claude Code 式治理 correction-of-correction 压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么治理 correction-of-correction 真正要救回的不是一套更严格的 correction 流程，而是同一个统一定价治理对象。
2. 怎样把假纠偏、假再托管、默认继续回魂与假阈值重绑压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续补 correction prose。
4. 怎样把 correction card、repricing rewrite block 与 custody / threshold ticket 重新压回对象级骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把审批与卡片再做严格一点”。

## 1. 第一性原理

治理 correction-of-correction 真正要救回的不是：

- 更保守的 correction checklist
- 更细的 capability 托管说明
- 更谨慎的 later 注意事项

而是：

- 无人继续盯防时，同一个治理对象仍在统一定价一切扩张

安全设计回答：

- 什么能继续放开

省 token 设计回答：

- 什么值得继续付费

在 Claude Code 里，这两者不是两套系统，而是同一条：

- `authority -> ledger -> window -> pricing -> custody -> threshold`

一旦 correction 退回 mode、dashboard、值班备注与“大家先保守一点”，统一定价控制面就会重新退回交互感受、背景告警与默认继续。

## 2. 固定纠偏顺序

### 2.1 先冻结假 restored verdict

第一步不是补 correction 文案，而是：

1. 把 mode、status、dashboard 降回 authority 的投影。
2. 禁止 `correction_verdict=restored` 在对象复核之前生效。
3. 禁止“目前没报错”继续充当 `steady_correction_object` 的替身。

### 2.2 再恢复 `authority_reassertion`

第二步要救回：

1. `authority_source`
2. `writer_chokepoint`
3. `liability_owner`
4. `authority_reasserted_at`
5. `authority_reassertion`

authority 不稳时，任何后续清账都可能清在错误对象上，只会制造假完成。

### 2.3 再恢复 `ledger_reseal`

第三步要修的是：

1. `typed_decision_digest`
2. `permission_ledger_state`
3. `pending_permission_requests`
4. duplicate / orphan / late residue 的正式清账
5. `ledger_reseal`

不要把“最近没人再点审批”当成 `ledger_reseal`。

### 2.4 再恢复 `window_refreeze`

第四步要把决策窗口从平静感拉回正式对象：

1. `decision_window`
2. `pending_action`
3. `context_usage_snapshot`
4. `reserved_buffer`
5. `late_response_quarantined`
6. `window_refreeze`

usage 回落、compact 执行或图表转绿都不能替代 `window_refreeze`。

### 2.5 再恢复 `continuation_repricing`

第五步要把继续从默认行为拉回正式定价：

1. `continuation_gate`
2. `budget_policy_generation`
3. `settled_price`
4. `completion_event`
5. `continuation_repricing`

没有这一步，继续就会重新免费。

### 2.6 再恢复 `capability_recustody`

第六步要把 capability 从运维备注拉回正式托管：

1. `capability_release_scope`
2. `custody_owner`
3. `quarantine_recall_handle`
4. `rollback_object`
5. `capability_recustody`

这一步的目标不是更保守，而是让 later 团队继承的不是一条免责时间线。

### 2.7 最后恢复 `threshold_rebinding`

最后才修未来重新反对当前 correction 的责任：

1. `reopen_threshold`
2. `authority_drift_trigger`
3. `liability_owner`
4. `residual_risk_digest`
5. `threshold_rebinding`

不要反过来：

1. 不要先放 capability，再补 threshold。
2. 不要先让 usage 图更平，再修 authority。
3. 不要先把 correction card 写完整，再修 pricing covenant。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `correction_verdict_early_promoted`
2. `authority_reassertion_missing_or_projected_by_mode`
3. `ledger_reseal_required`
4. `window_refreeze_required`
5. `default_continuation_detected`
6. `capability_recustody_missing`
7. `threshold_rebinding_missing`
8. `liability_owner_or_reopen_trigger_missing`

## 5. 苏格拉底式检查清单

在你准备宣布“治理 steady-state correction execution distortion 已纠偏完成”前，先问自己：

1. 我现在救回的是治理对象，还是更严格的修正流程。
2. 我现在消费的是 `authority_source`，还是 mode 面板给我的安全感。
3. 我现在封住的是 formal ledger 尾账，还是只是最近没人再点审批。
4. 我现在重新定价的是未来继续，还是把默认继续换了个更体面的名字。
5. 我现在保留的是 `capability_recustody + threshold_rebinding`，还是把责任礼貌地外包给 later。

## 6. 一句话总结

真正成熟的治理宿主修复稳态纠偏再纠偏，不是把 correction 看板写得更严谨，而是把假纠偏、假再托管、默认继续回魂与假阈值重绑重新压回同一个统一定价控制面。
