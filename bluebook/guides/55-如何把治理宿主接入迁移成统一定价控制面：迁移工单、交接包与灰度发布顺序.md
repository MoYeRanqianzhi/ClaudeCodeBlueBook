# 如何把治理宿主接入迁移成统一定价控制面：迁移工单、交接包与灰度发布顺序

这一章不再解释治理宿主接入该怎样审读，而是把 Claude Code 式治理迁移压成一张可执行的宿主迁移工单。

它主要回答五个问题：

1. 为什么治理迁移真正要迁的不是 mode、弹窗和 token 条，而是统一定价控制面。
2. 怎样把 `authority source`、`typed decision`、`decision window`、`continuation gate` 与 `rollback object` 写成固定实施顺序。
3. 怎样把交接从“当前卡在哪”升级成正式治理对象交接包。
4. 怎样安排灰度发布，避免宿主先学会放权，却还没学会为什么该放或不该放。
5. 怎样用苏格拉底式追问避免把迁移写成“更漂亮的权限面板”。

## 1. 第一性原理

治理宿主迁移真正要迁的不是：

- mode 名字
- 审批弹窗
- token 百分比

而是：

- 围绕同一个治理对象成立的统一定价控制面

所以更稳的迁移目标不是：

- 让宿主“也能控制权限”

而是：

1. 宿主知道谁是 authority source。
2. 宿主知道当前决策为什么是 ask / allow / deny。
3. 宿主知道 Context Usage 在解释哪个 decision window。
4. 宿主知道当前继续是否还有决策增益。
5. 宿主知道失败时回退到哪个治理对象，而不是回退几份文件。

## 2. 先识别旧宿主还在消费什么

在开始迁移前，先判断旧宿主是否仍在消费这些假信号：

1. `permission_mode` 被当成全部 authority source。
2. `pending_action` 只剩一句提示文案。
3. Context Usage 只剩 token 条或颜色。
4. continue 仍默认免费，只看“还能不能跑”。
5. rollback 仍主要围绕文件列表、commit 或补丁说明。

如果这些症状还在，迁移顺序就不能从界面控件开始，而应先从治理对象收口开始。

## 3. 迁移工单顺序

### 3.1 先冻结 authority source

第一张工单要做的不是补更多弹窗，而是明确：

1. `mode / settings / policy / host sync / session metadata` 谁拥有正式主权。
2. `permission_mode` 只是投影，不是 authority source 全部。
3. mode 变化必须经过统一 choke point。

最小交付物：

1. authority source 清单。
2. source / effective / applied 的统一投影。
3. 宿主与 SDK 的 single-source 说明。

### 3.2 再补 typed decision 与 pending action 对象

第二张工单要补的不是更多审批 UI，而是：

1. `can_use_tool / decision_reason / request_id / tool_use_id` 的正式对象。
2. `pending_action` 不再降格成文案。
3. duplicate / orphan / cancel path 都有明确治理语义。

最小交付物：

1. typed decision shadow diff。
2. `pending_action_surface`
3. headless / background deny 说明。

### 3.3 再补 decision window

第三张工单要补的是：

1. `state + pending_action + Context Usage` 的同屏判断窗口。
2. 当前哪些能力、上下文、时间与对象正在占用决策预算。
3. Context Usage 为什么会触发 stop / compact / object upgrade。

最小交付物：

1. `decision_window_snapshot`
2. `context_usage_breakdown`
3. `why_now` 解释面

### 3.4 再补 continuation gate

第四张工单要补的是：

1. 继续资格是否仍有决策增益。
2. 当前是否应该 stop、continue、compact 或升级对象。
3. headless deny、pending action 与 token budget 如何共同进入时间定价。

最小交付物：

1. `continuation_gate`
2. `object_upgrade_recommendation`
3. `diminishing_return_reason`

### 3.5 最后才放开 rollback object

第五张工单要补的是：

1. 哪一个治理对象才是回退对象。
2. 宿主、CI、交接是否围绕同一个 rollback object 工作。
3. `rewind_files`、cancel、resume 只是执行动作，不是治理真相本体。

最小交付物：

1. `rollback_object`
2. `recovery_action`
3. `handoff_reentry_condition`

不要反过来做：

1. 不要先做更漂亮的 mode 面板，再补 authority source。
2. 不要先做 token 仪表盘，再补 decision window。
3. 不要先开放自动继续，再补 continuation gate。
4. 不要先设计文件回退，再补 rollback object。

## 4. 交接包最小字段

治理宿主迁移后的交接包至少要有：

1. `governance_object_id`
2. `authority_source`
3. `effective_settings_projection`
4. `typed_decision`
5. `pending_action_surface`
6. `decision_window`
7. `context_usage_breakdown`
8. `continuation_gate`
9. `rollback_object`
10. `next_action`
11. `reject_reason`

少了这些字段，交接就仍会退回：

- 当前 mode 是什么
- 当前界面看起来卡在哪

## 5. 灰度发布顺序

更稳的治理灰度顺序是：

1. `authority source shadow`
   - 先 shadow 观测 mode/source/effective 的差异。
2. `typed decision shadow`
   - 先让宿主看见 ask / allow / deny 的正式对象。
3. `decision window read-only`
   - 先让宿主看见 Context Usage 如何解释当前窗口。
4. `continuation gate`
   - 再让宿主围绕正式 gate 管继续、compact 与对象升级。
5. `rollback object`
   - 最后才放开正式回退对象与恢复入口。

最危险的错误顺序是：

1. 先开放自动继续。
2. 先做 mode 面板。
3. 最后才补 authority source 与 rollback object。

这会直接制造：

- 免费扩张
- 默认继续
- 文件级伪回退

## 6. 直接拒收条件

出现下面情况时，应直接拒收迁移版本：

1. 宿主仍把 `permission_mode` 当全部 authority source。
2. `pending_action` 仍只是文案，不是对象。
3. Context Usage 仍只是仪表盘，不解释 decision window。
4. continue 仍默认免费，不解释是否还有决策增益。
5. rollback 仍主要围绕文件动作，而不是治理对象。

## 7. 苏格拉底式检查清单

在你准备宣布“治理宿主迁移已经完成”前，先问自己：

1. 我迁移的是治理对象，还是只迁移了权限界面。
2. 当前 ask / allow / deny 在我的系统里还是 typed decision，还是交互事件。
3. Context Usage 是否真的在解释 decision window，而不是只显示成本。
4. continuation 在我的系统里是正式 gate，还是默认继续。
5. 失败时我能否立刻指出 rollback object，而不是只会说回退哪些文件。

## 8. 一句话总结

真正成熟的治理宿主迁移，不是把更多审批控件搬进宿主，而是把 authority source、typed decision、decision window、continuation gate 与 rollback object 一起迁成同一个统一定价控制面。
