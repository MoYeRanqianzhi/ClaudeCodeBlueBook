# 如何把结构宿主修复稳态纠偏再纠偏执行失真压回结构真相面：固定纠偏顺序、拒收升级路径与authority、seam、evidence与liability改写模板骨架

这一章不再解释结构宿主修复稳态纠偏再纠偏执行最常怎样失真，而是把 Claude Code 式结构 correction-of-correction-of-correction 压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构 correction-of-correction-of-correction 真正要救回的不是“系统重新恢复秩序感”，而是同一个结构真相面。
2. 怎样把假 `authority surface`、伪 `single-source`、假 `lineage reproof`、伪 `anti-zombie evidence` 与假 `reopen liability` 压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续补 recorrection、archive 与责任文案。
4. 怎样把 `recorrection card`、`evidence restitution block` 与 `reopen liability ticket` 重新压回结构对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把系统再纠偏流程再做细一点”。

## 1. 第一性原理

结构 correction-of-correction-of-correction 真正要救回的不是：

- 更像样的 recorrection 流程
- 更完整的 archive 清单
- 更谨慎的 reopen 提示

而是：

- 无人继续盯防时，同一个结构真相面仍成立

结构先进性的真正前提不是目录更整洁，而是：

- `authority surface` 继续只有一个主语
- `single-source` 没有被第二来源篡位
- `resume lineage` 没有被好运气伪装成稳定
- `anti-zombie evidence` 仍然比 archive prose 更权威
- `reopen liability` 仍然保留未来推翻当前 recorrection 的正式边界

所以更稳的纠偏目标不是：

- 先把 recorrection 与 archive 文档写完整

而是：

1. 先把 pointer、side-write、旧路径与监控转绿降回投影。
2. 先把 authority、single-source、lineage 与 writeback 重新拉回同一个结构对象。
3. 先把 anti-zombie 与 archive 从 prose 拉回证据。
4. 先把 liability 从 reconnect 提示拉回正式 boundary。

## 2. 固定纠偏顺序

### 2.1 先冻结假 recorrection

第一步不是补 recorrection note，而是：

1. demote pointer、sidecar、旧路径与监控转绿，让它们回到辅助信号。
2. 恢复 `authority_object_id`、`authority_state_surface` 与 `recorrection_generation` 的统一真相面。
3. 禁止入口存在感再次充当 recorrection 的主对象。

### 2.2 再恢复 `authority_surface`

第二步要救回：

1. `authority_object_id`
2. `authoritative_path`
3. `authority_state_surface`
4. `writer_chokepoint`
5. `authority_surface_restitution`

只要 authority 还靠 pointer 与作者说明维持，就不能 restored。

### 2.3 再恢复 `single-source_seal`

第三步要修的是：

1. `single_source_refs`
2. `dependency_seam_status`
3. `heavy_import_regression_blocked`
4. `single_source_reseal`

不要把目录整洁度当作 `single-source`。

### 2.4 再恢复 `lineage_reproof`

第四步要把恢复顺序从好运气拉回正式对象：

1. `resume_lineage_ref`
2. `return_lineage`
3. `late_resume_detected`
4. `generation_regression_detected`
5. `lineage_reproof`

能 reconnect 不等于 lineage 已被复证。

### 2.5 再恢复 `writeback_custody`

第五步要把写回从结果感拉回唯一主路径：

1. `writeback_primary_path`
2. `worker_status_ref`
3. `external_metadata_ref`
4. `side_write_detected`
5. `writeback_custody_rebinding`

没有这一步，recorrection 仍只是“结果暂时没坏”。

### 2.6 再恢复 `anti-zombie_evidence`

第六步要把 zombie 清退从 archive prose 拉回正式证据：

1. `anti_zombie_evidence_ref`
2. `stale_writer_evidence`
3. `orphan_session_resolved`
4. `archive_truth_ref`
5. `anti_zombie_evidence_restitution`

只要这一步没修好，later 团队继承的就仍是一条假连续时间线。

### 2.7 最后恢复 `reopen_liability`

最后才修未来重新推翻当前 recorrection 的合法回跳边界：

1. `reopen_reservation_boundary`
2. `reservation_owner`
3. `threshold_retained_until`
4. `return_writeback_path`
5. `reopen_liability_rebinding`

不要反过来：

1. 不要先删临时痕迹，再修 authority。
2. 不要先写“以后再 reconnect”，再修 liability。
3. 不要先交付 archive 文本，再修 anti-zombie 证据。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `recorrection_object_missing`
2. `authority_surface_missing`
3. `pointer_based_restore_detected`
4. `single_source_breached`
5. `lineage_reproof_missing`
6. `writeback_custody_missing`
7. `anti_zombie_evidence_missing`
8. `reopen_liability_unbound`

## 5. 苏格拉底式检查清单

在你准备宣布“结构 steady-state correction-of-correction execution distortion 已纠偏完成”前，先问自己：

1. 我现在保护的是 authority object，还是一个还没失效的入口感。
2. 我现在验证的是 `single-source + lineage reproof`，还是几次幸运 reconnect 与更整洁的目录。
3. 我现在托管的是唯一 writeback 主路径，还是监控转绿带来的安心感。
4. 我现在归档的是 anti-zombie 证据，还是一段更会解释的复盘文本。
5. 如果明天必须 reopen，我保留下来的到底是正式 boundary，还是一句“以后再试一次”。

## 6. 一句话总结

真正成熟的结构宿主修复稳态纠偏再纠偏再纠偏，不是把 recorrection、archive 与责任流程写得更像制度，而是把假 `authority surface`、伪 `single-source`、假 `lineage reproof`、伪 `anti-zombie evidence` 与假 `reopen liability` 重新压回同一个结构真相面。
