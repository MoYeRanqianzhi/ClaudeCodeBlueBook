# 如何把结构宿主修复收口失真压回结构真相面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释结构宿主修复收口执行最常怎样失真，而是把 Claude Code 式结构收口纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构收口纠偏真正要救回的不是更顺手的 reconnect，而是同一个结构真相面。
2. 怎样把假 seal、假交接与假重开压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续观察。
4. 怎样把 closeout card、handoff block 与 reopen ticket 重新压回结构对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“补几条恢复日志和时序图”。

## 1. 第一性原理

结构收口纠偏真正要救回的不是：

- pointer 文件
- reconnect 通过感
- PUT 成功数
- 作者说明

而是：

- 围绕同一个结构真相面成立的 authority object、resume order、writeback path、anti-zombie witness 与 recovery boundary

所以更稳的纠偏目标不是：

- 先让恢复看起来更顺

而是：

1. 先把 authority seal 从 breadcrumb 与 UI 活性里救出来。
2. 先把 resume closure order 从 reconnect 成功感里救出来。
3. 先把 writeback seal 从 telemetry 繁荣里救出来。
4. 先把 anti-zombie witness 从作者说明与成功率里救出来。
5. 先把 handoff warranty 与 reopen ticket 从旧路径、旧入口与 retry 幻觉里救出来。

这些纠偏之所以本质上是在恢复源码先进性的真正对象边界，是因为：

- Claude Code 的先进性不在“拆得更好看”，而在 authority、single-source、stale-safe 与 anti-zombie 被提前编码进结构。
- 一旦 breadcrumb、telemetry 与作者口述重新上位，源码先进性就会退回目录美学与监控运营。

## 2. 固定纠偏顺序

### 2.1 先 demote breadcrumb

第一步不是补恢复按钮，而是：

1. demote pointer、sidecar 与旧路径，让它们回到入口信号。
2. 恢复 `authority_object_after`、`authority_state_surface` 与 `seal_generation` 的统一真相面。
3. 禁止 breadcrumb 再次充当 closeout 主对象。

### 2.2 再恢复 resume closure order

第二步要救回：

1. `switch_reset_done`
2. `restore_metadata_done`
3. `restore_worktree_done`
4. `adopt_resumed_session_done`
5. `clear_stale_done`

只要顺序错了，后面的成功率再高都仍是伪关闭。

### 2.3 再恢复 writeback 主路径

第三步要修的是：

1. `writeback_path`
2. `worker_status`
3. `external_metadata`
4. `merge_semantics`
5. `writeback_sealed`

不要继续做的事：

1. 不要把写回当普通 telemetry。
2. 不要只看 PUT 成功率。
3. 不要把日志繁荣当成真相对齐。

### 2.4 再恢复 anti-zombie witness

第四步要救回：

1. `anti_zombie_projection`
2. `stale_writer_evidence`
3. `duplicate_or_orphan_control_state_resolved`
4. `transition_legality_snapshot`
5. `witness_attested`

只要这一步没修好，later 团队就仍会继承一条假连续时间线。

### 2.5 最后恢复 boundary、handoff 与 reopen

最后才修：

1. `recovery_boundary`
2. `boundary_closed`
3. `handoff_warranty`
4. `re_entry_condition`
5. `reopen_ticket`

不要反过来：

1. 不要先修 pointer 展示。
2. 不要先修监控面板。
3. 不要先让 later 团队“再连一次试试看”。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_not_sealed`
2. `resume_order_not_closed`
3. `writeback_not_sealed`
4. `anti_zombie_not_witnessed`
5. `boundary_not_closed`
6. `reconnect_only_closeout`
7. `handoff_not_ready`
8. `reopen_required`
9. `breadcrumb_promoted_to_authority`
10. `telemetry_only_closeout`

## 5. 苏格拉底式检查清单

在你准备宣布“结构收口失真已经纠偏完成”前，先问自己：

1. 我救回的是结构真相面，还是恢复错觉。
2. 我现在消费的是 authority object，还是 breadcrumb、spinner 与 sidecar。
3. writeback 在我的系统里是真相主路径，还是普通日志。
4. 我看到的是 anti-zombie witness，还是作者说明与成功率。
5. handoff 与 reopen 在我的系统里回的是 recovery boundary，还是旧路径与重连通过感。

## 6. 一句话总结

真正成熟的结构收口纠偏，不是把 pointer、日志和监控修得更漂亮，而是把假 seal、假交接与假重开重新压回同一个结构真相面。
