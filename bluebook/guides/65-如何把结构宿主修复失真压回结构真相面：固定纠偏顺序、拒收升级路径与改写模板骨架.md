# 如何把结构宿主修复失真压回结构真相面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释结构宿主修复演练最常怎样失真，而是把 Claude Code 式结构修复纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构修复纠偏真正要救回的不是更顺手的 reconnect，而是同一个结构真相面。
2. 怎样把 breadcrumb 篡位、旁路写回与 anti-zombie 伪证明压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续观察。
4. 怎样把升级卡、rollback 模板与 re-entry ticket 重新压回结构对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“补几条恢复日志和时序图”。

## 1. 第一性原理

结构修复纠偏真正要救回的不是：

- pointer 文件
- reconnect 通过感
- PUT 成功数
- 恢复成功率

而是：

- 围绕同一个结构真相面成立的 authority object、resume order、writeback path 与 anti-zombie projection

所以更稳的纠偏目标不是：

- 先让恢复看起来更顺

而是：

1. 先把 authority object 从 breadcrumb 与 UI 活性里救出来。
2. 先把 resume / adopt 顺序恢复成正式顺序。
3. 先把 writeback 从 telemetry 成功感里救出来。
4. 先把 anti-zombie 结果面从成功率与目录美学里救出来。
5. 先把 rollback 从旧路径、旧 pointer 与监控绿灯里救出来。

这些纠偏之所以本质上是在恢复源码先进性的真正对象边界，是因为：

- Claude Code 的先进性不在“拆得更好看”，而在 authority、single-source、stale-safe 与 anti-zombie 被提前编码进结构。
- 一旦 breadcrumb、telemetry 与成功率重新上位，源码先进性就会退回目录美学与监控运营。

## 2. 固定纠偏顺序

### 2.1 先恢复 authority object

第一步不是补恢复按钮，而是：

1. 恢复 authority object 与 generation 的正式真相面。
2. 禁止 UI heuristic、spinner、pointer 与 task card 充当主状态机。
3. 恢复 single-source 状态投影。

### 2.2 再恢复 resume / adopt 顺序

第二步要救回：

1. `switch/reset`
2. `restore metadata`
3. `restore worktree`
4. `adopt resumed session file`
5. `clear stale`

只要顺序错了，后面的成功率再高都仍是伪恢复。

### 2.3 再恢复 recovery boundary

第三步要把恢复资产拉回边界：

1. `recovery_boundary`
2. `authority_state_surface`
3. `breadcrumb_assets`
4. `adopt_or_fail_reason`

这一步的目标不是更多入口，而是阻止 breadcrumb 篡位。

### 2.4 再恢复 writeback 主路径

第四步要修的是：

1. `writeback_path`
2. `worker_status`
3. `external_metadata`
4. merge / delete 的唯一语义

不要继续做的事：

1. 不要把写回当普通 telemetry。
2. 不要只看 PUT 成功率。
3. 不要把日志繁荣当成真相对齐。

### 2.5 最后恢复 anti-zombie 与 rollback object

最后才修：

1. `anti_zombie_projection`
2. `stale_writer_evidence`
3. `rollback_object`
4. `re_entry_condition`

不要反过来：

1. 不要先修 pointer 展示。
2. 不要先修成功率面板。
3. 不要先回到旧路径与绿灯。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. 宿主仍主要靠 pointer、spinner 或 sidecar 猜 authority object。
2. `resume/adopt/clear stale` 顺序没有被显式记录。
3. `recovery_boundary` 仍被旧路径与 reconnect 通过感替代。
4. writeback 仍只是日志或 telemetry。
5. `anti_zombie_projection` 缺席，而成功率充当主判断对象。
6. rollback 仍只回到旧 pointer、旧路径与监控绿灯。
7. 重入仍未证明 stale writer、late finally 与 orphan response 不会复活。

## 5. 苏格拉底式检查清单

在你准备宣布“结构修复失真已经纠偏完成”前，先问自己：

1. 我救回的是结构真相面，还是恢复错觉。
2. 我现在消费的是 authority object，还是 breadcrumb、spinner 与 sidecar。
3. writeback 在我的系统里是真相主路径，还是普通日志。
4. 我看到的是 anti-zombie 结果面，还是只看到成功率。
5. rollback 与重入在我的系统里回的是 recovery boundary，还是旧路径与重连通过感。

## 6. 一句话总结

真正成熟的结构修复纠偏，不是把 pointer、日志和监控修得更漂亮，而是把 breadcrumb 篡位、旁路写回与 anti-zombie 伪证明重新压回同一个结构真相面。
