# 如何把治理宿主修复稳态失真压回统一定价控制面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释治理宿主修复稳态执行最常怎样失真，而是把 Claude Code 式治理 steady-state correction 压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么治理 steady-state correction 真正要救回的不是一套更谨慎的稳态看板，而是同一个统一定价治理对象。
2. 怎样把假稳态、能力托管表演与免费继续复活压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续补 steady-state 文案。
4. 怎样把 steady-state card、pricing rewrite、capability custody 与 reopen threshold 重新压回治理对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把看板与审批流程再做严格一点”。

## 1. 第一性原理

治理 steady-state correction 真正要救回的不是：

- 更保守的稳态 checklist
- 更细的 capability 托管说明
- 更谨慎的 later 注意事项

而是：

- 无人继续盯防时，同一个治理对象仍在统一定价一切扩张

安全设计回答：

- 什么能继续放开

省 token 设计回答：

- 什么值得继续付费

在 Claude Code 里，这两者不是两套系统，而是同一条：

- `authority -> ledger -> window -> pricing -> custody -> threshold`

一旦 steady-state 退回 mode、dashboard、值班备注与“大家先保守一点”，统一定价控制面就会重新退回交互感受、背景告警与默认继续。

## 2. 固定纠偏顺序

### 2.1 先拆掉假 authority steady

第一步不是补 steady-state 备注，而是：

1. 把 mode、status、dashboard 降回 authority 的投影。
2. 恢复 `authority_source`、`writer_chokepoint` 与 `liability_owner` 的单一权威。
3. 禁止 UI 恢复感与“目前没报错”继续充当 `steady_verdict` 的前提。

### 2.2 再恢复 `ledger_dormancy_seal`

第二步要救回：

1. `typed_decision_digest`
2. `permission_ledger_state`
3. `pending_permission_requests`
4. duplicate / orphan / late residue 的正式清账
5. `ledger_dormancy_seal`

authority 不稳时，任何清账都可能清在错误对象上，只会制造假清账。

### 2.3 再恢复 `window_dormancy`

第三步要修的是：

1. `decision_window`
2. `pending_action`
3. `context_usage_snapshot`
4. `reserved_buffer`
5. `late_response_quarantined`
6. `window_dormancy`

不要把 usage 回落、compact 执行或图表转绿当成 `window_dormancy`。

### 2.4 再恢复 `continuation_pricing_covenant`

第四步要把继续从默认行为拉回正式定价：

1. `continuation_pricing_covenant`
2. `continuation_gate`
3. `budget_policy_generation`
4. `settled_price`
5. `completion_event`

没有这一步，继续就会重新免费。

### 2.5 再恢复 `capability_custody`

第五步要把 capability 从运维备注拉回正式托管：

1. `capability_custody`
2. `capability_release_scope`
3. `custody_owner`
4. `quarantine_recall_handle`
5. `rollback_object`

这一步的目标不是更保守，而是让 later 团队继承的不是一条免责时间线。

### 2.6 最后恢复 `reopen_threshold`

最后才修未来重新反对当前 steady 的责任：

1. `reopen_threshold`
2. `authority_drift_trigger`
3. `liability_owner`
4. `residual_risk_digest`
5. `threshold_retained_until`

不要反过来：

1. 不要先放 capability，再补 threshold。
2. 不要先让 usage 图更平，再修 authority。
3. 不要先把 steady-state card 写完整，再修 pricing covenant。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_source_projected_by_mode_or_dashboard`
2. `steady_verdict_early_promoted`
3. `ledger_dormancy_unsealed`
4. `window_dormancy_only_backed_by_usage_calmness`
5. `continuation_pricing_covenant_missing`
6. `capability_custody_missing`
7. `reopen_threshold_missing`
8. `default_continuation_or_no_error_as_reason`

## 5. 苏格拉底式检查清单

在你准备宣布“治理 steady-state distortion 已纠偏完成”前，先问自己：

1. 我现在消费的是 `authority_source`，还是 mode 面板给我的安全感。
2. 我现在封住的是 formal ledger 尾账，还是只是最近没人再点审批。
3. 我现在看到的是 `window_dormancy`，还是 usage 图暂时平静。
4. 我现在重新定价的是未来继续，还是把默认继续换了个更体面的名字。
5. 我现在保留的是 `capability_custody + reopen_threshold`，还是把责任礼貌地外包给 later。

## 6. 一句话总结

真正成熟的治理宿主修复稳态纠偏，不是把 steady-state 看板写得更严谨，而是把假稳态、能力托管表演与免费继续复活重新压回同一个统一定价控制面。
