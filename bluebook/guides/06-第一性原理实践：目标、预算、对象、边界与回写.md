# 第一性原理实践：目标、预算、对象、边界与回写

这一章回答五个问题：

1. 为什么复杂任务里最先该设计的不是更长的 prompt，而是 runtime 形态。
2. 怎样把 Claude Code 的强项压缩成一套可执行的使用方法。
3. 为什么 token 节省、本地隔离、前台真相与治理边界必须一起操作。
4. 怎样用苏格拉底式追问检查自己有没有把系统用错。
5. 怎样把一次会话沉淀成可恢复、可复用、可治理的长期资产。

这些锚点共同说明一件事：

- Claude Code 最值得学习的用法，不是“怎么把一句话写得更像咒语”，而是“怎么把目标、预算、对象、边界与回写装进同一条 runtime 链”。

## 1. 先说结论

如果只记一套最小方法，可以压成五个预算和四类对象。

五个预算：

1. 目标预算：这轮到底要交付什么，完成标准是什么。
2. 动作预算：这轮允许动哪些工具、哪些目录、哪些环境。
3. 上下文预算：哪些信息该放进稳定前缀，哪些只该临时注入。
4. 协作预算：哪些工作可以并行研究，哪些文件只能有一个主写入者。
5. 治理预算：哪些外部输入、托管策略和审批链可以进入当前会话。

四类对象：

1. session：让任务有恢复入口。
2. task / workflow：让长流程有生命周期与进度面。
3. worktree：让高风险修改有隔离现场。
4. 记忆文档：让稳定规则从聊天里外移。

这意味着更稳的使用方式不是：

- 一次性写更长 prompt

而是：

- 先分预算
- 再选对象
- 再让 prompt 只承担合同，而不是承担全部状态

## 2. 第一性原理使用法

### 2.1 先决定这次任务属于哪一种 runtime

不要一上来就问“prompt 怎么写”，先问：

1. 这次任务是单轮查询，还是长会话协作。
2. 这次任务是局部实现，还是跨文件、跨阶段的大任务。
3. 这次任务是否需要隔离目录、后台运行或多代理分工。
4. 这次任务是否会引入外部输入、远程环境或组织策略。

如果答案偏向：

- 短、窄、一次性：用单轮或轻量 REPL。
- 长、脏、可中断：用 session + plan + resume。
- 高风险改动：优先 worktree。
- 可拆成并行研究：起 subagent，但保持单一主写入者。
- 多阶段自动化：把它当 task / workflow，而不是“多发几次 continue”。

`Task.ts`、`framework.ts`、`sessionStorage.ts` 与 `worktree.ts` 已经表明，Claude Code 在设计上承认：

- 长任务应该拥有 task type、task_started、独立 transcript 与 worktree 语义。

所以正确用法应当顺着对象模型走，而不是逆着对象模型把一切都塞回聊天历史。

### 2.2 先分五种预算，再写 prompt

`buildEffectiveSystemPrompt(...)` 明确把 override、coordinator、agent、custom、default、append 分层装配。

这说明 prompt 更像合同栈，而不是一段孤立字符串。

实际写法可以先用下面这张最小模板：

```text
目标：
- 这轮必须交付什么。

验收：
- 交付物如何判断完成。

边界：
- 不要动哪些文件、目录、环境、远程面。

允许动作：
- 允许使用哪些工具或只允许读/分析/验证。

输出格式：
- 先给结论，再给证据，再给风险或待确认项。

失败时：
- 遇到权限、缺上下文、风险上升时先汇报，不要硬做。
```

关键不是文字优美，而是五个预算有没有被写清楚：

1. 目标预算决定系统往哪里收敛。
2. 动作预算决定工具空间有多宽。
3. 上下文预算决定什么该稳定，什么该晚绑定。
4. 协作预算决定谁研究、谁综合、谁写入。
5. 治理预算决定哪些输入和策略能改写当前轮。

### 2.3 把上下文预算理解成“稳定前缀 + 动态外移”

`checkTokenBudget(...)` 不是简单统计 token，而是在判断：

- 是否值得继续
- 是否已出现递减收益

`mergeAndFilterTools(...)` 则显式为了 prompt cache 稳定，把 built-in tools 保持成连续前缀；`promptCacheBreakDetection.ts` 又持续记录 system/tool hash、cache_control、betas、effort、extra body 变化。

这些事实共同说明：

- Claude Code 的省 token 主要不是“把一句话说短点”
- 而是“让稳定部分稳定，让高波动部分外移”

实践上应这样做：

1. 稳定规则写进 `CLAUDE.md`、skill、固定模板，而不是每轮重写。
2. 变化快的现场状态放在当前 prompt、附件、计划或任务对象里。
3. 大块输出让系统外置、摘要或分阶段消费，不要反复塞回主线程。
4. 如果一个任务开始靠连续 `continue` 才能维持，就该考虑 plan、task、session tag 或 worktree，而不是继续扩 prompt。

### 2.4 把长流程写成对象，不要写成“很多轮对话”

`TaskType` 已把 `local_workflow` 纳入正式枚举；`task_started` 事件也能带 `workflow_name`；`sessionStorage` 与 `worktree` 进一步把 workflow transcript 与临时 worktree 命名做成正式路径。

这意味着：

- 长流程的正确承载体是对象，而不是聊天泡泡

更稳的实践是：

1. 给大任务起清晰 description，而不是模糊的“继续做下去”。
2. 给长流程留 session 恢复点，不要只依赖当前上下文还没丢。
3. 高风险实现拆到独立 worktree，避免主现场和验证现场互相污染。
4. 并行时允许多代理研究，但同一批待写文件维持单一写入者。

如果一个任务已经跨回合、跨阶段、跨目录，你还把它当“多轮聊天”处理，通常就已经在逆用 Claude Code。

### 2.5 让“用户可见真相”优先于原始日志

`transcriptSearch.ts` 显式跳过不可见 sentinel、system reminder 和模型侧序列化的 tool result；`selection.ts` 则显式维护 anchor/focus、滚动丢失文本与复制一致性。

这说明前台的目标不是“忠实暴露所有底层字符串”，而是：

- 让搜索、复制、定位、回看都服从同一套用户可见真相

使用时应顺着这条原则：

1. 要求模型先给可见结论、差异与引用，再给原始大块输出。
2. 把后台任务输出当证据附件，而不是把所有日志都当最终答案。
3. 做 review 或复盘时，优先保留“用户真正看到并据以决策的真相”。
4. 如果需要原始日志，单独存放，不要让它污染主结论层。

很多“上下文越来越乱”的问题，本质上不是模型不够强，而是你把 raw log 和 render truth 混成了一层。

### 2.6 把治理变化当成运行时变化，而不是部署尾巴

`channelPermissions.ts` 显式把 channel permission relay 建成独立能力：不仅要 connected，还要在 allowlist 中，且必须声明 `claude/channel` 与 `claude/channel/permission` 两层 capability。`securityCheck.tsx` 又说明危险 managed settings 变化会触发阻断式对话，拒绝则直接退出。

这意味着：

1. channels 不是普通插件接入，而是外部输入边界。
2. managed settings 不是普通偏好同步，而是运行时边界变更。
3. 审批链不是 UI 装饰，而是预算器的一部分。

实践上应这样操作：

1. 团队场景先确定 allowlist、opt-in 与托管策略，再决定是否启用 channels。
2. 把远程下发的危险设置变更视为“运行时已变”，而不是“管理员小改了一下配置”。
3. 一旦外部输入面变宽，就同步收紧 prompt 范围、写入权限和结论置信度。

### 2.7 每个阶段都要回写稳定状态

Claude Code 的真正优势，不在“这一轮答得对”，而在“下一轮还能接得住”。

所以每做完一段工作，都应问：

1. 哪些规则应写回 `CLAUDE.md` 或技能文档。
2. 哪些结论应写回共享 artifact、决策日志或长期规则载体。
3. 哪些任务状态应落成 plan、session tag 或 task description。
4. 哪些风险与边界应明确写成“不要做什么”。

只要不回写，系统就会在后续把同一个结论重新推理一遍，既浪费 token，也浪费人类判断力。

## 3. 苏格拉底式检查清单

在开始前问自己：

1. 这轮真正的交付物是什么，还是我只是在继续堆上下文。
2. 这轮最紧的预算是动作、上下文、协作还是治理。
3. 这项任务是否已经大到应该升级成 task、workflow、worktree 或独立 session。
4. 哪些规则应进入稳定层，哪些只应出现在当前轮。
5. 这次是否真的需要并行，还是只需要一个主写入者。
6. 外部输入一旦进入，会不会改写当前信任边界。

在结束后再问自己：

7. 我留下的是用户可见真相，还是一堆未整理的原始日志。
8. 关键状态是否已经回写到长期可恢复的位置。
9. 这轮是否缩小了噪音，还是只是扩大了会话体积。
10. 如果现在中断，下一位接手者能否靠现有对象恢复工作。
11. 这轮有没有把治理变化、安全变化或权限变化明确说出来。
12. 我写下的结论里，哪些是事实，哪些是解释，哪些只是推论。

## 4. 最常见的五种误用

1. 把复杂任务全压进一个 prompt，结果目标预算和上下文预算一起失控。
2. 一上来就开最大权限，结果动作预算失去治理价值。
3. 让多个代理同时改同一批文件，结果协作预算被直接打爆。
4. 把 channels、remote policy、managed settings 当“接上再说”的附属能力，结果治理预算失真。
5. 只保留原始日志，不回写稳定结论，结果 session 价值无法累积。

## 5. 一句话总结

从第一性原理看，Claude Code 的最佳用法不是“把 prompt 写得更像咒语”，而是“先分预算、再选对象、再写合同、最后回写状态”。
