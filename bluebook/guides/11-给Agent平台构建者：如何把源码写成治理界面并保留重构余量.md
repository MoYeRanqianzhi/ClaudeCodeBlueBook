# 给Agent平台构建者：如何把源码写成治理界面并保留重构余量

这一章回答五个问题：

1. 如果你不是 Claude Code 用户，而是在做自己的 agent 平台，最该抄什么。
2. 怎样把“可读性”“注释”“小模块”从审美变成治理工具。
3. 怎样在持续增长时保留重构余量，而不是等失控后大重构。
4. 哪些模式适合先做成制度，再做成架构。
5. 怎样用苏格拉底式追问避免把“写得漂亮”误当“长期可治理”。

这些锚点共同说明：

- Claude Code 的代码不只为执行服务，也在为未来维护和未来重构服务。

## 1. 先说结论

如果只记一套最小方法，可以压成五条制度：

1. 先把未来会频繁读的共享规则压成 dependency-free 小模块。
2. 先把未来会频繁漂移的状态压成显式状态机。
3. 先把未来想重排的地方切出 config / deps / snapshot 边界。
4. 先把危险理由写进命名和注释，而不是留在团队口头记忆里。
5. 先让代码自己表达“哪里不能随便改”，再追求目录美感。

## 2. 制度一：把高扇入规则压成极薄小模块

值得学的不是“更多 util”，而是：

- 更薄、更少依赖、更容易审计的规则单元

实践上：

1. 类型 branding、名字规范、归一化规则优先做成 dependency-free leaf module。
2. 这些模块只承载一条共享真相，不顺手带业务逻辑。
3. 目标不是复用更多，而是让共享规则更容易被全仓安全引用。

## 3. 制度二：把状态迁移收成显式状态机

当一个对象会经历：

- 进入
- 中间态
- 结束
- stale cleanup

就不要继续靠散落 flag 管它。

更稳的做法：

1. 把状态枚举显式写出来。
2. 把“谁能结束它”“谁会 stale”写成正式转移。
3. 把读快照和写转移分开。

这样新增异步路径时，系统仍可推理。

## 4. 制度三：提前为未来重构留接口

如果你今天就知道未来可能要：

- 抽 step
- 注入 fake
- 切断 cycle
- 冻结快照

那今天就该先把这些边界切出来。

实践上：

1. 把 immutable data 先移到 config。
2. 把少量最痛的 I/O 先移到 deps。
3. 不追求一次性全抽离，先证明模式有效。
4. 把 why 写进注释，方便后来者沿正确方向继续抽。

## 5. 制度四：把风险理由写进命名和注释

更成熟的写法不是：

- 代码能跑就行

而是：

- 未来改动者一眼能看出风险

实践上：

1. 风险高的边界在名字里显式标出。
2. 注释优先写“为什么不能随便改”。
3. 对 cycle、cache、snapshot、authority 这类制度性理由，写得比算法细节还清楚。

## 6. 制度五：同时服务机器执行和人类维护

很多系统只优化其中一边：

- 对机器严格
- 对人类难懂

或者：

- 对人类顺眼
- 对机器边界模糊

更好的做法是两边一起做：

1. 机器有类型、状态机、协议、合并语义。
2. 人类有注释、命名、阶段边界、依赖动机。

只有这样，系统增长时才不会只靠少数老作者维持。

## 7. 苏格拉底式检查清单

在你设计自己的 agent runtime 时，先问自己：

1. 这一条共享规则，为什么必须被全仓引用。
2. 它能不能压成 dependency-free 小模块。
3. 这个状态对象是不是已经值得一台正式状态机。
4. 我现在写下的边界，是否为未来重构保留了路。
5. 哪些“为什么这样写”的理由，必须写进代码边上，而不是靠人记住。
6. 这段代码是否同时服务机器执行和人类治理。

如果这些问题答不清，结构再漂亮也很难长期稳。

## 8. 一句话总结

从 Claude Code 可迁移出来的更成熟方法，不是“照着它的目录抄一遍”，而是先把规则、小边界、状态机、快照和制度记忆写进代码，让仓库在增长时仍保留治理力和重构余量。
