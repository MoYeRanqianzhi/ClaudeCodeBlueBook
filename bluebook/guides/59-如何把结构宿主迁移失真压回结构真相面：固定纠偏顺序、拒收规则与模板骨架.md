# 如何把结构宿主迁移失真压回结构真相面：固定纠偏顺序、拒收规则与模板骨架

这一章不再解释结构宿主迁移最常怎样失真，而是把 Claude Code 式结构纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构纠偏真正要救回的不是 pointer、成功率或恢复按钮，而是同一个结构真相面。
2. 怎样把伪恢复采纳、指针健康幻觉与写回分叉压回固定纠偏顺序。
3. 哪些现象应被直接拒收，而不是继续观察。
4. 怎样把结构交接、灰度与回退重新压回模板骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“补几个 debug 指标”。

## 1. 第一性原理

结构纠偏真正要救回的不是：

- 恢复成功率
- pointer 文件
- UI 看起来已经正常

而是：

- 围绕同一个结构真相面成立的 authority state、合法恢复顺序、writeback 主路径与 anti-zombie 结果面

所以更稳的纠偏目标不是：

- 先让恢复看起来跑通

而是：

1. 先把 authority state 从 UI 猜测里救出来。
2. 先把 resume/adopt 顺序恢复成正式顺序。
3. 先把 `bridgePointer` 降回恢复资产。
4. 先把 writeback 从 telemetry 拉回真相主路径。
5. 先把 anti-zombie 结果面从成功率幻觉里救出来。

## 2. 固定纠偏顺序

### 2.1 先恢复 authority state

第一步不是补恢复按钮，而是：

1. 恢复 `session_state_changed + pending_action + permission_mode` 的正式状态面。
2. 禁止 UI heuristic 充当主状态机。
3. 恢复 single-source 状态投影。

### 2.2 再恢复 resume/adopt 顺序

第二步要救回：

1. `switch/reset`
2. `restore metadata`
3. `restore worktree`
4. `adopt resumed session file`

只要顺序错了，后面的 success 率再高都仍是伪恢复。

### 2.3 再恢复 recovery boundary

第三步要把恢复资产拉回边界：

1. `bridge_pointer_health`
2. `recovery_asset_inventory`
3. `recovery_boundary`
4. `adopt_or_fail_reason`

这一步的目标不是更多恢复入口，而是阻止恢复资产篡位。

### 2.4 再恢复 writeback 主路径

第四步要修的是：

1. `writeback_delivery_state`
2. `external_metadata`
3. `worker_status`
4. pending patch / merge 的单一路径

不要继续做的事：

1. 不要把写回当普通 telemetry。
2. 不要只看请求成功率。

### 2.5 最后恢复 anti-zombie 结果面

最后才修：

1. `anti_zombie_projection`
2. `stale_writer_evidence`
3. `transition_legality_snapshot`
4. `degraded_path`

不要反过来：

1. 不要先修 pointer 展示。
2. 不要先修成功率面板。
3. 不要先修远端恢复按钮。

## 3. 硬拒收规则

出现下面情况时，应直接拒收当前结构宿主实现：

1. 宿主仍主要靠 UI heuristic 猜 authority state。
2. `resume/adopt` 顺序没有被显式记录。
3. `bridgePointer` 仍在替代 recovery boundary。
4. writeback 仍只是日志或 telemetry。
5. anti-zombie 结果面仍缺席，而成功率充当主判断对象。

## 5. 苏格拉底式检查清单

在你准备宣布“结构失真已经纠偏完成”前，先问自己：

1. 我救回的是结构真相面，还是恢复错觉。
2. 我现在依赖的是正式恢复顺序，还是作者口述。
3. `bridgePointer` 在我的系统里还是资产，还是已经篡位。
4. writeback 在我的系统里是真相主路径，还是普通日志。
5. 我看到的是 anti-zombie 结果面，还是只看到成功率。

## 6. 一句话总结

真正成熟的结构纠偏，不是把 pointer 和恢复面板修得更顺，而是把伪恢复采纳、指针健康幻觉与写回分叉重新压回同一个结构真相面。
