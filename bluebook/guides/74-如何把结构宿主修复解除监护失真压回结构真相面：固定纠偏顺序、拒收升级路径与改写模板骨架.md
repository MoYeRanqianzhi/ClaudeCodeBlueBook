# 如何把结构宿主修复解除监护失真压回结构真相面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释结构宿主修复解除监护执行最常怎样失真，而是把 Claude Code 式结构 release correction 压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么结构 release correction 真正要救回的不是更像样的 retirement / archive 流程，而是同一个结构真相面。
2. 怎样把假退休、归档表演与假保留重开压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续补 archive 与 retirement 文案。
4. 怎样把 release card、archive、boundary retirement 与 reopen reservation 重新压回结构对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把 release 看板与复盘模板再做细一点”。

## 1. 第一性原理

结构 release correction 真正要救回的不是：

- 更像样的 retirement 流程
- 更完整的 archive 清单
- 更谨慎的 reopen 提示

而是：

- 同一个结构真相面：authority 仍唯一、resume 顺序已真正稳定、writeback 仍走唯一主写点、anti-zombie 证据已被正式归档、recovery boundary 可以退休但仍保留合法重开能力

所以更稳的纠偏目标不是：

- 先把 release 文档写完整

而是：

1. 先把 pointer、sidecar、旧路径与监控转绿降回 breadcrumb。
2. 先把 `authority_release_seal` 从入口感里救出来。
3. 先把 `resume_stability_seal` 与 `writeback_release_seal` 从成功感与 telemetry 里救出来。
4. 先把 `anti_zombie_archive` 从复盘文本与作者说明里救出来。
5. 先把 `boundary_retirement` 与 `reopen_reservation` 从 reconnect 提示与以后再说里救出来。

## 2. 固定纠偏顺序

### 2.1 先 demote 假退休信号

第一步不是补 retirement note，而是：

1. demote pointer、sidecar、旧路径与监控转绿，让它们回到辅助信号。
2. 恢复 `authority_object_after`、`authority_state_surface` 与 `release_generation` 的统一真相面。
3. 禁止入口存在感再次充当 release 主对象。

### 2.2 再恢复 `authority_release_seal`

第二步要救回：

1. `authority_release_seal`
2. `authority_object_after`
3. `authority_state_surface`
4. `writer_chokepoint`
5. `release_generation`

只要 authority 还靠 pointer 与作者说明维持，就不能 release。

### 2.3 再恢复 `resume_stability_seal` 与 `writeback_release_seal`

第三步要修的是：

1. `resume_closure_order`
2. `late_resume_detected`
3. `generation_regression_detected`
4. `writeback_path`
5. `writeback_release_seal`
6. `side_write_detected`

不要继续做的事：

1. 不要把 reconnect 成功感当作顺序稳定。
2. 不要把 telemetry 繁荣当作 writeback release。
3. 不要把监控转绿当成结构对齐。

### 2.4 再恢复 `anti_zombie_archive`

第四步要救回：

1. `anti_zombie_projection`
2. `stale_writer_evidence`
3. `duplicate_or_orphan_control_state_resolved`
4. `transition_legality_snapshot`
5. `archive_pointer`

只要这一步没修好，later 团队就仍会继承一条假连续时间线。

### 2.5 最后恢复 `boundary_retirement` 与 `reopen_reservation`

最后才修：

1. `boundary_retirement`
2. `recovery_boundary`
3. `reopen_reservation`
4. `return_authority_object`
5. `return_writeback_path`

不要反过来：

1. 不要先删临时痕迹，再修 authority。
2. 不要先写“以后再 reconnect”，再修 reservation。
3. 不要先交付 archive 文本，再修 anti-zombie 证据。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_release_seal` 仍主要由 pointer、sidecar 或旧路径可达来证明。
2. `resume_stability_seal` 仍建立在 reconnect 成功感上，而不是顺序闭合。
3. `writeback_release_seal` 仍只看监控、PUT 成功率与日志繁荣。
4. `anti_zombie_archive` 缺失 `stale_writer_evidence` 或 transition legality 证据。
5. `boundary_retirement` 缺失正式 `recovery_boundary` 绑定。
6. `reopen_reservation` 缺失，或 reservation 仍退回“真出问题再连一次”。
7. later 团队不能只凭 release card 与 archive 重建同一结构判断。
8. `release_verdict` 继续被健康感、监控与作者保证主导。

## 5. 苏格拉底式检查清单

在你准备宣布“结构 release distortion 已纠偏完成”前，先问自己：

1. 我救回的是结构真相面，还是一个更像样的 release 流程。
2. 我现在 release 的是 authority，对象是否唯一，还是只是在 release 稳定感。
3. 我现在 archive 的是 anti-zombie 证据，还是一段更能安抚人的复盘。
4. 我现在 retirement 的是 boundary，还是只是撤掉显眼的警示。
5. 我现在 reservation 的是合法重开闸门，还是一句含糊的“以后再看”。

## 6. 一句话总结

真正成熟的结构宿主修复解除监护纠偏，不是把假退休、归档表演与 pointer 式稳定写成更漂亮的流程，而是把它们重新压回 authority、resume、writeback、anti-zombie 与 boundary 共同成立的结构真相面。
