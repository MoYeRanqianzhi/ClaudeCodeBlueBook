# 如何设计有顺序的治理系统：检查顺序、失败语义分型、可撤销自动化与稳定字节资产

这一章回答五个问题：

1. 如果你在做自己的 agent runtime，怎样把安全和省 token 从“统一预算器”继续下沉成可执行制度。
2. 怎样设计检查顺序，而不是不断往系统里堆更多检查器。
3. 怎样为不同资产设计不同失败语义，并让自动化随时可撤销。
4. 怎样用“决策增益”决定是否继续花 token。
5. 怎样把 prompt / permission / replacement 一类关键字节当作稳定制度资产来治理。

## 1. 先说结论

更成熟的治理系统，不是：

- 有更多规则

而是：

1. 先决定哪些检查必须先发生。
2. 先为不同资产写清失败语义。
3. 先规定自动化何时必须退回人工。
4. 只在仍有决策增益时继续花 token。
5. 把关键治理字节当成稳定资产管理。

如果你把这五件事做了，安全和省 token 才真的会合成同一套：

- ordered governance system

## 2. 第一步：先画检查顺序图，而不是先堆检查器

Claude Code 的关键不是“检查很多”，而是检查顺序很清楚。

最值得抄的动作不是某条规则，而是一张顺序图：

```text
Governance Order
- source gate
- dangerous fast-path stripping
- safety check
- classifier / hook / relay
- human fallback / deny / abort
```

为什么要先画这张图：

1. 危险 allow rule 如果不先剔掉，后面的 classifier 形同虚设。
2. 极窄豁免如果不与普遍 allow 分开，系统会默默放大边界。
3. 某些 source gating 如果放错位置，会同时破坏安全和成本。

落地规则：

1. 每条治理路径都要写明“谁先判断、谁后判断”。
2. 高危 fast-path 要单独 inventory，不允许它们绕过后续治理。
3. 所有“特殊豁免”都要在顺序图上有单独节点，而不是混在一般 allow 里。

## 3. 第二步：做资产分型，而不是统一 fail-open / fail-closed

Claude Code 清楚地承认：

- 不同资产不该共享同一种失败哲学

所以你在设计自己的系统时，应先列一张资产表：

```text
Asset Classes
- 核心组织/身份边界
- 高危配置与策略
- 远程托管设置
- 自动化辅助路径
- 低价值优化路径
```

然后为每类资产定义默认失败语义：

1. 哪些必须 fail-closed。
2. 哪些可以 fail-open。
3. 哪些应回退人工。
4. 哪些应直接停机。

Claude Code 的启发不是“总是更保守”，而是：

- 失败要按资产目的分型

如果你没有这张分型表，系统很快就会退回：

- 某些本该阻断的路径被放开；
- 某些本可退化运行的路径被整体拉闸。

## 4. 第三步：把自动化当成可撤销租约，而不是永久主权

Claude Code 的 auto mode 很成熟的一点是：

- 自动化始终可被系统收回

对应到 builder 实践，可以直接做一个“自动化租约”模型：

```text
Automation Lease
- 允许自动判断的范围
- 撤销条件
- 人工承接条件
- headless 无人工承接时的终止条件
```

落地动作：

1. denial-limit 一类累计条件必须能打回人工 review。
2. 任何自动通道都要有 killswitch。
3. 无人工承接的运行环境必须比交互环境更早终止。
4. 自动化失败不能永远重试，要有显式退出条件。

这样做的收益是：

- 自动化不再是越堆越大的一次性升级，而是可收回的受控通道

## 5. 第四步：用“决策增益”决定何时继续花 token

Claude Code 给 builder 的最大启发之一是：

- 不是所有检查都值得继续花 token

你可以把这条原则直接写成团队政策：

```text
Decision Gain Policy
- 如果该检查已不能改变决策，就不再继续
- 如果继续检查只会重复已知结果，就停止
- 如果环境不再允许人工承接，就终止而不是盲重试
```

这条原则可以直接落到三种地方：

1. classifier：
   - transcript 过长或上下文已失真时，不要假装“再判一次也许更准”；
2. compaction：
   - 连续失败后必须熔断；
3. 审批：
   - 已经有更低成本确定性结论时，不再走高价评估路径。

这不是“优化成本附属项”，而是成熟治理系统的中心判断：

- token 应花在仍有制度收益的地方

## 6. 第五步：把稳定字节单独立法

Claude Code 反复提醒你：

- 真正危险的不是状态变化，而是关键治理字节不稳定

所以你在自己的系统里要单独列出：

```text
Stable Bytes Assets
- prompt sections / cache-control boundary
- sticky headers / latched feature bits
- replacement preview / externalized result marker
- 被观测和 diff 的关键 request shape
```

实践规则：

1. 一旦某种 preview / replacement 进入主路径，后续必须稳定重放。
2. 一旦某些 header / beta 进入会话，就不要中途偷偷改变。
3. 会影响 cache、权限、解释力的关键字节要被单独观测。
4. 任何 cache break 都要可归因，不能只当作“这轮变贵了”。

如果不把这层独立出来，团队最终只会讨论：

- 这轮怎么突然不稳了

而不是：

- 哪个制度字节漂移了

## 7. 有顺序治理系统设计卡

```text
1. Governance Order
- 谁先检查
- 谁后检查
- 哪些 fast-path 必须先剔除

2. Failure Semantics Matrix
- 每类资产的 fail-open / fail-closed / human-fallback / abort

3. Automation Lease
- 自动化范围
- 撤销条件
- 无人工承接时的终止条件

4. Decision Gain Policy
- 什么时候继续花 token
- 什么时候停止评估

5. Stable Bytes Assets
- 哪些字节必须稳定
- 哪些变化必须可解释
```

## 8. 苏格拉底式检查清单

在你准备继续加一层安全/优化机制前，先问自己：

1. 我解决的是顺序问题，还是只是在叠检查器。
2. 我已经区分资产类型了吗，还是仍在追求单一失败哲学。
3. 我的自动化有没有合法撤销路径。
4. 这次多花的 token 还能改变决策吗。
5. 我是否已经知道哪些字节一旦漂移，就会同时破坏安全、成本和解释力。

如果这些问题答不清，你的系统大概率仍然只有预算器，没有治理制度。

## 9. 一句话总结

Claude Code 值得学的，不是把安全和省 token 都写成预算器，而是怎样把检查顺序、失败语义分型、可撤销自动化、决策增益和稳定字节资产合成同一套有顺序的治理系统。
