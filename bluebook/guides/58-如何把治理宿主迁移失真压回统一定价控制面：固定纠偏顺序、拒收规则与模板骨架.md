# 如何把治理宿主迁移失真压回统一定价控制面：固定纠偏顺序、拒收规则与模板骨架

这一章不再解释治理宿主迁移最常怎样失真，而是把 Claude Code 式治理纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么治理纠偏真正要救回的不是 mode 面板，而是同一个统一定价控制面。
2. 怎样把权限账本缺席、假窗口对齐与免费继续幻觉压回固定纠偏顺序。
3. 哪些现象应被直接拒收，而不是继续观察。
4. 怎样把治理交接、灰度与回退重新压回模板骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“补几张控制面截图”。

## 1. 第一性原理

治理纠偏真正要救回的不是：

- 当前 mode 名字
- 一组弹窗截图
- 一条 token 进度条

而是：

- 围绕同一个治理对象成立的统一定价控制面

所以更稳的纠偏目标不是：

- 先把 UI 调到“看起来正常”

而是：

1. 先把 authority source 从 mode 投影里救出来。
2. 先把 `pending_permission_ledger` 从 UI 记忆里救出来。
3. 先把 decision window 从仪表盘里救出来。
4. 先把 continuation gate 从默认继续里救出来。
5. 先把 rollback object 从文件动作里救出来。

## 2. 固定纠偏顺序

### 2.1 先拆掉 mode 崇拜

第一步不是补弹窗，而是：

1. 把 `permission_mode` 降回 authority source 的投影。
2. 恢复 `sources / effective / applied` 的统一权威视图。
3. 明确 internal mode 与 external mode 的边界。

### 2.2 再恢复权限账本

第二步要救回：

1. `typed_decision`
2. `pending_permission_ledger`
3. `request_id + tool_use_id`
4. duplicate / orphan / cancel 的仲裁语义

这一步不恢复，交接就仍然只剩“刚刚应该批过了吧”。

### 2.3 再恢复 decision window

第三步要修的是：

1. `Context Usage + pending_action + state`
2. `context parity snapshot`
3. `why_now / diminishing_return_reason`

不要继续做的事：

1. 不要把 Context Usage 当漂亮仪表盘。
2. 不要把 token 百分比当治理对象。

### 2.4 再恢复 continuation gate

第四步要把继续从默认行为拉回：

1. `continuation_gate`
2. `currentTurnTokenBudget`
3. `continuationCount`
4. `completionEvent`

这一步不恢复，时间与输出扩张就仍然是免费的。

### 2.5 最后恢复 rollback object

最后才修：

1. `rollback_object`
2. `rewind_files` 与对象的映射
3. `re-entry_condition`

不要反过来：

1. 不要先恢复 mode 面板。
2. 不要先恢复 token 图表。
3. 不要先恢复文件回退。

## 3. 硬拒收规则

出现下面情况时，应直接拒收当前治理宿主实现：

1. `permission_mode` 仍在替代 authority source。
2. `pending_permission_ledger` 缺席。
3. `Context Usage` 仍只是仪表盘，不进入 decision window。
4. continuation 仍默认免费，只看“还能不能跑”。
5. rollback 仍围绕文件动作，而不是 `rollback_object`。

## 5. 苏格拉底式检查清单

在你准备宣布“治理失真已经纠偏完成”前，先问自己：

1. 我救回的是治理对象，还是 mode 面板。
2. 我的权限请求现在还是账本，还是 UI 记忆。
3. `Context Usage` 在我的系统里是 decision window，还是仪表盘。
4. continuation 在我的系统里是正式 gate，还是默认继续。
5. rollback 在我的系统里还是对象，还是文件回退。

## 6. 一句话总结

真正成熟的治理纠偏，不是把 mode、弹窗和图表修得更漂亮，而是把权限账本缺席、假窗口对齐与免费继续幻觉重新压回同一个统一定价控制面。
