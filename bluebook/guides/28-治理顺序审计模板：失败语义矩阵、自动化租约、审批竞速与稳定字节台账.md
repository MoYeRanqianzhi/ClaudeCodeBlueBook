# 治理顺序审计模板：失败语义矩阵、自动化租约、审批竞速与稳定字节台账

这一章把 `guides/25` 继续压成审计层，不再只是 builder-facing 理解，而是团队可执行的治理模板。

它主要回答五个问题：

1. 怎样用顺序图而不是规则堆叠来审安全与省 token。
2. 怎样把失败语义按资产分型，而不是统一 fail-open / fail-closed。
3. 怎样把自动化写成租约、把审批写成竞速协议。
4. 怎样识别哪些 token 仍有决策增益，哪些只是重复消耗。
5. 怎样把稳定字节、结果外置与停止条件一起纳入治理台账。

## 1. Governance Order Audit Sheet

任何新规则都先填顺序表：

```text
当前规则名称:
它属于哪一层:
- source gate
- dangerous strip
- cheap deterministic path
- safety / classifier
- human fallback
- deny / abort

为什么必须放在这里:
如果前移会破坏什么:
如果后移会破坏什么:
它与哪些 fast-path 冲突:
```

团队规则：

1. 没有顺序位置的规则，不算治理规则。
2. 高危 fast-path 必须单独 inventory，不能混在一般 allow 里。
3. 每次新增规则，都要回答“它减少了哪类错误决策”。

## 2. Failure Semantics Matrix

对不同资产先做分型，再决定失败语义：

```text
资产类别:
- 身份 / 组织边界
- 高危权限
- 托管策略
- 自动化辅助路径
- 低价值优化路径

失败类型:
- fail-open
- fail-closed
- human-fallback
- degrade
- abort
```

这张矩阵至少要写清：

1. 交互环境与 headless 环境是否同判。
2. 哪些失败允许退化运行，哪些必须立即停机。
3. 哪些失败看起来像“体验问题”，实际是边界问题。

更稳的母版列至少应固定为：

```text
控制对象:
默认失败语义:
threshold trigger:
escalation target:
headless divergence:
minimum legal degraded shape:
re-entry / rollback requirement:
durable assets kept:
transient authority cleared:
```

没有这些列，团队最后只会记住 reject code，而不会记住：

- 何时该升级
- 升级给谁
- 降级后最小合法形态是什么
- 哪些旧主权必须被清掉

## 3. Automation Lease

把自动化当成租约，而不是永久主权：

```text
自动化路径:
准入条件:
撤销条件:
再验证触发器:
无人工承接时的终止条件:
用户通知方式:
```

团队规则：

1. 每条自动化都必须有 killswitch。
2. model / mode / environment 变化要触发再验证。
3. 无人接管环境的停止条件要比交互环境更早。

## 4. Decision-Gain Ladder

token 只花在仍能改变决策的地方：

```text
Level 0:
- 纯确定 cheap path

Level 1:
- 低价模拟 / allowlist / replaceable check

Level 2:
- classifier / host race / extra inference

Level 3:
- 人工承接或直接 abort
```

每条高价判断前，必须回答：

1. 还有没有更便宜且不失真的路径。
2. 再花一次 token 是否可能改变结论。
3. 如果不会改变，为什么不立刻停。

## 5. Approval Race Matrix

把审批当协议竞速，而不是单个弹窗：

```text
决策源:
- local UI
- host
- bridge
- channel relay
- hook
- classifier

它能否 claim:
它能否附带反馈:
它是否能更新输入:
超时后由谁接管:
谁负责取消其他通道:
```

没有这张表，审批系统很容易在增长中退回：

- 多通道都存在，但没人知道谁最终是权威。

## 6. Stable Bytes Ledger

把制度字节单独立法：

```text
资产:
- prompt sections
- cache-control boundary
- sticky headers / feature bits
- tool visibility shape
- replacement preview marker

变化来源:
可观测手段:
回滚开关:
是否影响 cache key:
```

如果不单列这张台账，团队最终会说：

- “这轮怎么突然又贵又不稳了”

而不是：

- “是哪个制度字节变了”

## 7. Result Externalization Contract

大结果外置不能只看字节大小，还要看协议连续性：

```text
[ ] 这类结果是否允许 externalize
[ ] preview 规则是否稳定
[ ] replay 是否一致
[ ] 哪些结果绝不能替换
[ ] 替换后模型是否仍能继续正确判断
```

核心原则：

- 外置是在压缩重复字节成本和重复决策成本，而不是简单裁文本

## 8. Stop-Logic Checklist

任何自动链路都必须有正式停止条件：

```text
[ ] 连续失败阈值
[ ] transcript too long 时的行为
[ ] 无人工承接时的终止条件
[ ] hopeless retry 的熔断点
[ ] 降级运行的最小合法形态
```

成熟系统不是更会重试，而是更知道什么时候该停。

## 9. Governance Cost Dashboard

治理层的回归观测至少要看：

```text
- permission waiting time
- classifier token / latency / cost
- cache-break count
- approval race 的来源分布
- auto path 被撤销的频率
```

如果只看“功能是否通过”，你看不到治理系统是否正在偷偷变贵、变慢、变脆。

## 10. 苏格拉底式检查清单

在你准备加一条新治理逻辑前，先问自己：

1. 我解决的是顺序问题，还是只是在堆检查器。
2. 我已经分清不同资产的失败语义了吗。
3. 自动化有没有合法撤销路径。
4. 这次多花的 token 还能改变什么。
5. 哪些制度字节会因这次变更而漂移。
