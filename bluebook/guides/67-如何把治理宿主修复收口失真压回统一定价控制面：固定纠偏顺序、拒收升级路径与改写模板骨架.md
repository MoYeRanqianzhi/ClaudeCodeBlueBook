# 如何把治理宿主修复收口失真压回统一定价控制面：固定纠偏顺序、拒收升级路径与改写模板骨架

这一章不再解释治理宿主修复收口执行最常怎样失真，而是把 Claude Code 式治理收口纠偏压成一张可执行的 builder-facing 手册。

它主要回答五个问题：

1. 为什么治理收口纠偏真正要救回的不是更保守的 mode 建议，而是同一个统一定价控制面。
2. 怎样把假关账、假交接与免费重开压回固定纠偏顺序。
3. 哪些现象应被直接升级为硬拒收，而不是继续观察。
4. 怎样把 closeout card、handoff block 与 reopen packet 重新压回治理对象骨架。
5. 怎样用苏格拉底式追问避免把纠偏写成“把审批完成页再做清楚一点”。

## 1. 第一性原理

治理收口纠偏真正要救回的不是：

- 当前 mode 名字
- 一组审批历史
- 一块刷新后的 usage 面板

而是：

- 围绕同一个治理对象成立的统一定价控制面

所以更稳的纠偏目标不是：

- 先把页面调到“看起来正常”

而是：

1. 先把 authority settlement 从 mode 投影里救出来。
2. 先把 ledger seal 从审批完成感里救出来。
3. 先把 window closure 从 usage dashboard 里救出来。
4. 先把 continuation warranty 从默认继续里救出来。
5. 先把 rollback clearance 与 handoff warranty 从经验建议里救出来。

这些纠偏之所以同时在恢复安全设计与省 token 设计，是因为：

- 安全保护的是“允许什么”。
- 省 token 保护的是“继续多久、扩张多少”。
- Claude Code 把这两件事都压在同一个治理对象上统一定价。

## 2. 固定纠偏顺序

### 2.1 先拆掉 authority 假收口

第一步不是补 mode 建议，而是：

1. 把 `permission_mode` 降回 authority source 的投影。
2. 恢复 `authority_source_before/after`、`effective_settings_projection` 与 `writer_chokepoint` 的统一权威视图。
3. 禁止 internal mode 与 UI mode 充当 closeout 真相。

### 2.2 再排空 ledger

第二步要救回：

1. `typed_decision`
2. `permission_ledger`
3. `pending_permission_requests`
4. `request_id`
5. `tool_use_id`
6. `duplicate_or_orphan_state_resolved`

这一步不恢复，交接就仍然只剩“刚刚应该批过了吧”。

### 2.3 再关闭 decision window

第三步要修的是：

1. `decision_window`
2. `pending_action`
3. `context_usage_snapshot`
4. `reserved_buffer`
5. `late_response_quarantined`

不要继续做的事：

1. 不要把 usage 百分比当治理对象。
2. 不要把 compact 次数当窗口解释。
3. 不要把“看起来还能跑”当窗口结论。

### 2.4 再恢复 continuation repricing

第四步要把继续从默认行为拉回：

1. `continuation_warranty`
2. `continuation_gate`
3. `token_budget_result`
4. `completion_event`
5. `re_entry_condition`

这一步不恢复，时间与输出扩张就仍然是免费的。

### 2.5 最后恢复 rollback、handoff 与 reopen

最后才修：

1. `rollback_clearance`
2. `handoff_warranty`
3. `rollback_object`
4. `baseline_reset_required`
5. `reopen_packet`

不要反过来：

1. 不要先修 mode 面板。
2. 不要先修 usage 图表。
3. 不要先给 later 一份更保守的建议清单。

## 3. 硬拒收规则

出现下面情况时，应直接升级为硬拒收：

1. `authority_source` 仍由 mode 推断，而非正式 source / writer chokepoint。
2. `ledger_seal` 不含 `typed_decision`，或仍存在 stale pending request。
3. `window_closure` 仅表现为 dashboard 刷新，没有 late response quarantine。
4. continuation 仍默认免费，只看“还能不能跑”。
5. compact 仍被当成降本技巧，而不是窗口重写动作。
6. `rollback_clearance` 缺失 `baseline_reset_required` 或 `rollback_cleared`。
7. `handoff_warranty` 不能让 later 团队仅凭 closeout card 重建同一治理对象。
8. reopen 允许沿用旧 ledger、旧 window、旧 continuation 直接续跑。

## 5. 苏格拉底式检查清单

在你准备宣布“治理收口失真已经纠偏完成”前，先问自己：

1. 我救回的是治理对象，还是 mode 面板。
2. 我现在保护的是 authority source，还是审批完成感。
3. `Context Usage` 在我的系统里是 decision window，还是 usage dashboard。
4. continuation 在我的系统里是正式 repricing，还是默认行为。
5. rollback、handoff 与 reopen 在我的系统里回的是对象边界，还是经验建议。

## 6. 一句话总结

真正成熟的治理收口纠偏，不是把 mode、弹窗和图表修得更漂亮，而是把假关账、假交接与免费重开重新压回同一个统一定价控制面。
