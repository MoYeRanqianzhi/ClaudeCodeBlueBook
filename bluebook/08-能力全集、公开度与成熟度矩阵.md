# 能力全集、公开度与成熟度矩阵

这一章回答四个问题：

1. Claude Code 的“全部功能和 API”该如何写全，而不是越写越散。
2. 为什么“源码里有入口”不等于“对外稳定可用”。
3. 为什么能力全集、实现可见度、产品承诺必须分三层写。
4. 在蓝皮书目录里，这一章应当承担什么角色。

## 0. 代表性源码锚点

- `claude-code-source-code/package.json:2-19`
- `claude-code-source-code/README.md:70-74`
- `claude-code-source-code/README.md:250-280`
- `claude-code-source-code/src/entrypoints/agentSdkTypes.ts:103-272`
- `claude-code-source-code/src/tools.ts:253-367`
- `claude-code-source-code/src/types/permissions.ts:16-38`
- `claude-code-source-code/src/main.tsx:1497-1520`
- `claude-code-source-code/src/main.tsx:1635-1688`

这些锚点共同说明三件事：

1. 当前研究对象是公开源码基线，不是完整内部 monorepo。
2. 这个基线已经暴露出非常宽的能力表面。
3. 能力表面、实现可见度、产品成熟度彼此不能画等号。

## 1. 先说结论

Claude Code 的能力全集不能写成一张扁平功能表，而必须同时按两层拆：

1. 按能力平面拆：
   - 交互
   - 执行
   - 安全
   - 状态
   - 宿主
   - 远程
   - 生态
   - 协作
2. 按公开度与成熟度拆：
   - 公共主路径
   - 宿主主路径
   - 声明存在但当前实现未闭合
   - gated / internal 痕迹
   - 受组织、账号、策略约束的产品面

如果不同时做这两层拆分，蓝皮书就会持续在两个错误之间摇摆：

- 把所有代码痕迹都写成“公开支持”
- 或把真正稳定的能力和实验能力混成一团

## 3. 公开度与成熟度必须再拆一层

最稳妥的写法不是问“支不支持”，而是连续追问四次：

1. 源码里是否有正式 schema、type、command、tool 或 protocol 入口。
2. 当前公开源码里是否有可见实现，而不是只剩声明。
3. 当前实现是否进入主路径，而不是仅存在于子集适配器或门控分支。
4. 产品现实里是否还受 feature gate、GrowthBook、policy、org settings 或账号 entitlements 约束。

例如：

- `agentSdkTypes.ts` 已公开暴露 session 管理函数族，但当前提取树里多处函数体仍是 `not implemented`。
- `tools.ts` 与 `types/permissions.ts` 说明工具与权限是正式平台 ABI，而不是零散工具脚本。
- `main.tsx` 又进一步说明 MCP、channels、remote session 仍会被策略、allowlist 与组织政策裁切。
- `README.md` 则直接声明公开源码仍缺失 `108` 个 `feature()` 相关模块。

所以正确结论不是：

- “它全都公开稳定”

也不是：

- “这些都只是内部实验”

而是：

- Claude Code 已经公开暴露出很宽的 runtime 表面，但每个表面都必须再按实现可见度和产品成熟度二次分层。

## 4. 一张足够实用的能力矩阵

| 能力平面 | 代表入口 | 当前判断 |
| --- | --- | --- |
| 交互平面 | `commands.ts`、`types/command.ts` | 主路径稳定，仍有动态来源与 gate 叠加 |
| 执行平面 | `Tool.ts`、`tools.ts` | 主路径稳定，MCP / internal tools 需要继续分层 |
| 安全平面 | `types/permissions.ts`、permission setup | 主路径稳定，但 mode / classifier / policy 带明显条件化 |
| 状态平面 | session / transcript / `worker_status` | 主路径稳定，但宿主消费子集不同 |
| 宿主平面 | SDK、control schema、`StructuredIO` | 主路径稳定；部分 session API 仍是声明先行 |
| 远程平面 | bridge / remote session / direct connect | 能力真实存在，但适配器宽度差异很大 |
| 生态平面 | plugins / channels / MCPB / LSP | schema 支持较宽，产品成熟度不齐 |
| 协作平面 | AgentTool / mailbox / coordinator | 主路径清晰，但部分 worker 相关实现仍受 gate 影响 |

这张矩阵的目的不是给出最终评分，而是强制后续写作先区分层次，再下判断。

## 5. 如果要把“全部功能与 API”写全，至少要补齐五张对象表

仅有平面划分还不够。后续写作如果真要覆盖“全部功能与 API”，至少要持续补齐五张对象表：

1. 命令表：
   - `src/commands/` 约 `86` 个一级命令目录
   - `COMMANDS()` 约 `61` 个内置主表项
   - 运行时还会再叠加 skills / plugins / workflows / dynamic skills
2. 工具表：
   - `src/tools/` 约 `42` 个一级工具目录
   - `getAllBaseTools()` 是主线程工具 truth
   - `constants/tools.ts` 还给出 async agent、in-process teammate、coordinator 的子集 truth
3. 任务表：
   - `TaskType` 当前显式包含 `7` 类任务
   - `getAllTasks()` 当前显式注册 `4` 个基础 task 加 `2` 个 gated task
4. 服务表：
   - `src/services/` 当前可见约 `20` 个一级 service plane
   - 真正的“能力先进性”很大一部分就藏在这些 subsystem 里
5. 宿主接入表：
   - `agentSdkTypes.ts` 给 query / session / remote control 入口
   - `controlSchemas.ts` 给 control subtypes
   - 还必须叠加 event、snapshot、recovery 三层真相

如果没有这五张表，所谓“能力全集”依然只是摘要；如果只有这五张表而没有成熟度分层，又会重新滑回“源码里有就算支持”的错误。

## 7. 苏格拉底式追问

### 7.1 如果源码里有类型和 schema，为什么还不能直接写成公开支持

因为 schema 只能证明：

- 这个能力被设计过

却不能单独证明：

- 当前公开实现已经闭环
- 所有宿主都支持
- 产品已经稳定承诺

### 7.2 如果产品里能看到入口，为什么还要继续区分 gate

因为 Claude Code 的运行现实明显是多层裁切的：

- build gate
- runtime gate
- policy gate
- consumer subset

不写这层，就会把同一能力在不同环境下的真实差异抹平。

### 7.3 如果我们已经有导航层，为什么根目录还要再写这一章

因为导航层解决的是“怎么找”，而这一章解决的是“找到之后该怎么判断”。

二者不是重复关系，而是：

- 导航层解决检索问题
- 这一章解决判断标准问题

### 7.4 如果我们已经有能力矩阵，为什么还要继续补对象总表

因为“能力平面”回答的是系统结构问题，而“对象总表”回答的是覆盖率问题。

前者告诉你 Claude Code 是什么。
后者告诉你蓝皮书到底漏了什么。

## 8. 一句话总结

Claude Code 的能力全集不是一张功能表，而是一组运行时平面；而蓝皮书要把它写对，就必须把“能力存在、实现可见、产品承诺”三层持续分开。
