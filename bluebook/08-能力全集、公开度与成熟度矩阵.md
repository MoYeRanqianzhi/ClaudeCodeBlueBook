# 能力全集、公开度与成熟度矩阵

这一章回答四个问题：

1. Claude Code 的“全部功能和 API”该如何写全，而不是越写越散。
2. 为什么“源码里有入口”不等于“对外稳定可用”。
3. 为什么能力全集、实现可见度、当前准入与产品承诺必须分四层写。
4. 在蓝皮书目录里，这一章应当承担什么角色。

它也不替 `09 / 05 / 15 / 41` 重新发明高阶前门：

- `09` 先定义三张控制面。
- `05` 先校正你到底在模仿什么。
- `15` 先暴露第一条反证信号。
- `41` 再把高阶结论压回第一性原理。
- `08` 只负责阻止“能力存在 / 实现可见 / 当前准入 / 产品承诺”重新混成一层。
矩阵层也继续继承 `问题分型 -> 工作对象 -> 控制面 -> 入口`；它只负责 existence / visibility / admission / promise 的分层记账，不额外重签控制面 verdict。

这些锚点共同说明三件事：

1. 当前研究对象是公开源码基线，不是完整内部 monorepo。
2. 这个基线已经暴露出非常宽的能力表面。
3. 能力表面、实现可见度、产品成熟度彼此不能画等号。

如果把这一章前门继续压成最短公式，也只剩三条：

1. Prompt 线
   - 能力存在，不等于它已经通过 `same-world test` 合法进入模型当前世界。
2. 治理线
   - 能力存在，不等于它已经通过 `decision window` 合法进入当前世界。
3. 公开度线
   - `existence -> implementation visibility -> current admission -> product promise`

如果一个能力判断还压不回这三条，它就还停在“源码里有 / 好像可用”的结果词层。

这里还应再多记一句：

- `continuity` 在矩阵层也不是第五条公开度轴；它只是同一能力平面在 `Continuation / continuation pricing / reopen qualification` 上的时间轴读法。

## 0.1 这张矩阵还要继续挂到三条判断上

这一章不只是在做能力表，还在替三条高阶判断提供一个更稳的前门：

1. Prompt 线提醒你：
   - 能力存在，不等于它已经合法进入模型当前可见世界。
2. 治理线提醒你：
   - 能力存在，不等于它已经被当前价格秩序允许进入当前世界。
3. 源码质量线提醒你：
   - 能力存在、实现可见、产品承诺之间的区分，必须继续服从证据梯度和公开镜像克制原则。

因此这一章最该固定的不是“能力很多”，而是判断优先级：

1. 先给出 `governance / current admission / product promise` 的首答
2. 再把证据回填到 `existence / implementation visibility / current admission / product promise`
3. 最后才用能力平面标注“它属于哪一类对象”

更稳一点说，这张矩阵也该先暴露两条 first reject signal：

1. UI 可见、命令存在或类型声明开始冒充 `current admission`
2. build-time 痕迹、宿主子集或组织限定开始冒充 `product promise`

## 1. 先说结论

Claude Code 的能力全集不能再以 taxonomy-first 开场，首答必须先落在治理与承诺判断：

1. 先判 `current admission`：
   - 它是否已经通过 `decision window` 进入当前世界，而不是只在源码里“存在”。
2. 再判 `product promise`：
   - 它是否能被写成稳定产品承诺，而不是仍受 gate / policy / entitlement 裁切。
3. 再回填四层证据：
   - `existence -> implementation visibility -> current admission -> product promise`
4. 最后再用能力平面与标签做定位压缩：
   - 交互 / 执行 / 安全 / 状态 / 宿主 / 远程 / 生态 / 协作
   - 公共主路径 / 宿主主路径 / 声明先行 / 产品受限

如果不同时做这两层拆分，蓝皮书就会持续在两个错误之间摇摆：

- 把所有代码痕迹都写成“公开支持”
- 或把真正稳定的能力和实验能力混成一团

换句话说，这一章的职责不是列出更多能力，而是持续阻止“能力存在”和“外部可承诺”重新合并。

## 3. 公开度与成熟度：先判治理与承诺，再做分层

最稳妥的写法不是先问“它属于哪个平面”，而是先做四次判断：

1. 当前实现是否进入主路径并通过 `decision window`（`current admission`）。
2. 产品现实里是否可承诺，还是仍受 feature gate、policy、org settings 或 entitlement 裁切（`product promise`）。
3. 若不能承诺，问题是“仅有入口”还是“实现未闭合”（`existence / implementation visibility`）。
4. 最后才回填“它位于哪个能力平面”，用于覆盖率与索引。

例如：

- `agentSdkTypes.ts` 已公开暴露 session 管理函数族，但当前提取树里多处函数体仍是 `not implemented`。
- `tools.ts` 与 `types/permissions.ts` 说明工具与权限是正式平台 ABI，而不是零散工具脚本。
- `main.tsx` 又进一步说明 MCP、channels、remote session 仍会被策略、allowlist 与组织政策裁切。
- `README.md` 则直接声明公开源码仍缺失 `108` 个 `feature()` 相关模块。

所以正确结论不是：

- “它全都公开稳定”

也不是：

- “这些都只是内部实验”

而是：

- Claude Code 已经公开暴露出很宽的 runtime 表面，但每个表面都必须再按实现可见度和产品成熟度二次分层。

这也意味着判断顺序应固定为：

1. 先问“它现在是否被治理线允许进入当前世界（`current admission`）”
2. 再问“它是否已经达到可对外承诺的产品状态（`product promise`）”
3. 再问“证据处在 `existence / implementation visibility` 的哪一层”
4. 最后才问“它在哪个能力平面里，以及它为三条高阶判断贡献什么”

如果继续压成最短动作链，这里也应和三张控制面保持同构：

1. Prompt 线先问 `compile -> protocolize -> preserve -> continue -> explain`
2. 治理线先问 `price -> externalize -> arbitrate -> continue-or-stop -> cleanup`
3. 当前真相线先问 `surface -> writeback -> freshness -> evict`

换句话说：Prompt 线不该先按“可自定义程度”评分，治理线不该先按“拦截层数”评分，源码质量线不该先按“目录整洁度”评分；三者都要回到那三句高阶判断本身。

## 4. 一张足够实用的能力矩阵

| 能力平面 | 代表入口 | existence | implementation visibility | current admission | product promise |
| --- | --- | --- | --- | --- | --- |
| 交互平面 | `commands.ts`、`types/command.ts` | 正式命令与类型入口已公开 | CLI / REPL / command 装配闭合可见 | 主路径稳定，但动态来源与 gate 继续叠加 | 公共交互面可承诺，动态附加项逐条判断 |
| 执行平面 | `Tool.ts`、`tools.ts` | Tool ABI 与工具池正式存在 | 基础工具闭合，MCP / internal 分层可见 | 主路径稳定，但 mode / deny / MCP 继续裁切 | 基础执行面可承诺，扩展工具逐项判断 |
| 安全平面 | `types/permissions.ts`、permission setup | 权限类型与决策入口存在 | mode / classifier / sandbox / policy 实现可见 | 始终受当前策略与环境条件化 | 可承诺的是治理控制面，不是预写具体放行结果 |
| 状态平面 | session / transcript / `worker_status` | 状态对象、快照与恢复资产存在 | 状态回写与快照实现可见 | 宿主消费子集明显不同 | 状态真相面可承诺，字段粒度按 consumer subset 写 |
| 宿主平面 | SDK、control schema、`StructuredIO` | host control 与 SDK 入口存在 | 主路径实现可见，但部分 session API 未闭合 | 宿主主路径稳定，adapter 宽度不一 | 控制面可承诺，未闭合 session 函数不能提前承诺 |
| 远程平面 | bridge / remote session / direct connect | 远程与桥接对象真实存在 | 公开树能看到主要适配器 | 仍受策略、组织、设备与适配器裁切 | 只能写成受限产品面，不能写成无条件支持 |
| 生态平面 | plugins / channels / MCPB / LSP | 扩展与生态对象存在 | schema 与入口较宽，但成熟度不齐 | 受渠道、策略与适配器差异持续裁切 | 可承认其存在，成熟度必须逐项写 |
| 协作平面 | AgentTool / mailbox / coordinator | 多 Agent 与协作对象存在 | 协作协议与事件主路径可见 | worker 细节与部分 team surface 仍受 gate 影响 | 协作 runtime 可承诺，细粒度 worker/product 面逐项写 |

这张矩阵的目的不是给出最终评分，而是强制后续写作遵守 `judgment-first, taxonomy-second`：先看 `current admission / product promise`，再回查 `implementation visibility / existence`，最后再看能力平面。

如果后续还需要写 `公共主路径`、`宿主主路径`、`声明先行`、`产品受限`，它们只能是这四列之后的压缩结论，不能倒过来替代这四列。

## 5. 如果要把“全部功能与 API”读稳，至少要同时看五张对象表

仅有前门矩阵还不够。任何试图覆盖“全部功能与 API”的读法，至少都要同时经过五张对象表；但这些对象表只负责补证，不再充当前门骨架：

1. 命令表：
   - `src/commands/` 约 `86` 个一级命令目录
   - `COMMANDS()` 约 `61` 个内置主表项
   - 运行时还会再叠加 skills / plugins / workflows / dynamic skills
2. 工具表：
   - `src/tools/` 约 `42` 个一级工具目录
   - `getAllBaseTools()` 是主线程工具 truth
   - `constants/tools.ts` 还给出 async agent、in-process teammate、coordinator 的子集 truth
3. 任务表：
   - `TaskType` 当前显式包含 `7` 类任务
   - `getAllTasks()` 当前显式注册 `4` 个基础 task 加 `2` 个 gated task
4. 服务表：
   - `src/services/` 当前可见约 `20` 个一级 service plane
   - 真正的“能力先进性”很大一部分就藏在这些 subsystem 里
5. 宿主接入表：
   - `agentSdkTypes.ts` 给 query / session / remote control 入口
   - `controlSchemas.ts` 给 control subtypes
   - 还必须叠加 event、snapshot、recovery 三层真相

如果不同时看这五张表，所谓“能力全集”依然只是摘要；如果只看这五张表而不继续做成熟度分层，又会重新滑回“源码里有就算支持”的错误。

## 7. 苏格拉底式追问

### 7.1 如果源码里有类型和 schema，为什么还不能直接写成公开支持

因为 schema 只能证明：

- 这个能力被设计过

却不能单独证明：

- 当前公开实现已经闭环
- 所有宿主都支持
- 产品已经稳定承诺

### 7.2 如果产品里能看到入口，为什么还要继续区分 gate

因为 Claude Code 的运行现实明显是多层裁切的：

- build gate
- runtime gate
- policy gate
- consumer subset

不写这层，就会把同一能力在不同环境下的真实差异抹平。

### 7.3 如果我们已经有导航层，为什么根目录还要再写这一章

因为导航层解决的是“怎么找”，而这一章解决的是“找到之后该怎么判断”。

二者不是重复关系，而是：

- 导航层解决检索问题
- 这一章解决判断标准问题

### 7.4 如果我们已经有能力矩阵，为什么还要继续补对象总表

因为“能力平面”回答的是系统结构问题，而“对象总表”回答的是覆盖率问题。

前者告诉你 Claude Code 是什么。
后者告诉你蓝皮书到底漏了什么。

## 8. 一句话总结

Claude Code 的能力全集不是一张功能表，而是一组运行时平面；而蓝皮书要把它写对，就必须把“能力存在、实现可见、当前准入、产品承诺”四层持续分开。
