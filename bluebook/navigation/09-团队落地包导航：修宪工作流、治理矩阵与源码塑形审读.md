# 团队落地包导航：修宪工作流、治理矩阵与源码塑形审读

这一篇不新增新的制度判断，而是回答一个更务实的问题：

- 当 `philosophy/60-62`、`architecture/73-75`、`guides/24-26` 都已经成立后，团队该拿什么直接做评审、回归、事故复盘和设计迁移。

它主要回答四个问题：

1. Prompt Constitution 何时该从“理解”进入“修宪工作流”。
2. 安全与省 token 何时该从“原则”进入“审计矩阵与停止条件”。
3. 源码先进性何时该从“判断”进入“结构审读与反模式清单”。
4. 为什么第二序制度如果不继续下沉成工件层，很快就会重新退回口头方法。

## 1. Prompt 修宪工作流包

适合在这些问题下阅读：

- 怎样把 prompt 改动当成修宪，而不是文案微调。
- 怎样让 section、boundary、主权链、合法遗忘和 triage 进入同一套团队动作。
- 怎样在 prompt 失稳时先跑审计流程，而不是凭感觉猜。

稳定阅读顺序：

1. `philosophy/60`
2. `architecture/73`
3. `guides/24`
4. `guides/27`

这条线的核心交付物应是：

- section constitution card
- prompt amendment workflow
- lawful forgetting ABI checklist
- prompt invalidation ledger
- prompt triage runbook

## 2. 治理顺序与省 token 审计包

适合在这些问题下阅读：

- 新规则到底该进哪一层检查顺序。
- 哪些失败该 fail-open，哪些必须 fail-closed，哪些该停机。
- 自动化什么时候是租约，什么时候应该被系统收回。

稳定阅读顺序：

1. `philosophy/61`
2. `architecture/74`
3. `guides/25`
4. `guides/28`

这条线的核心交付物应是：

- governance order audit sheet
- failure semantics matrix
- automation lease
- decision-gain ladder
- approval race matrix
- stable bytes ledger
- stop-logic checklist

## 3. 源码塑形与反-zombie 审读包

适合在这些问题下阅读：

- 一个新能力该进主循环、旁路 service、shadow entry 还是 transport shell。
- 为什么 portable shadow、thin registry、recovery asset 和 anti-zombie protocol 应一起评审。
- 为什么“看起来更模块化”不等于“更适合演进”。

稳定阅读顺序：

1. `philosophy/62`
2. `architecture/75`
3. `guides/26`
4. `guides/29`

这条线的核心交付物应是：

- build surface matrix
- entry shadow card
- transport shell confinement checklist
- thin registry review
- recovery asset ledger
- anti-zombie protocol
- structure anti-pattern list

## 4. 一句话用法

如果 `08` 回答的是：

- 这三条制度为什么重要

那么 `09` 回答的是：

- 团队怎样把它们压成正式可执行工件

最稳的使用顺序是：

1. 先用 `08` 校准制度判断。
2. 再用 `24-26` 理解 builder-facing 动作。
3. 最后用 `27-29` 把动作固定成 review、回归与迁移模板。

## 5. 苏格拉底式自检

在你准备继续新增一篇“更深分析”前，先问自己：

1. 这条制度已经缺解释，还是已经缺工件。
2. 团队现在缺的是新观点，还是缺审读表、台账和 runbook。
3. 我是在帮助读者理解，还是在帮助团队执行。
4. 如果这套东西出了事故，今天有没有现成模板能复盘。
