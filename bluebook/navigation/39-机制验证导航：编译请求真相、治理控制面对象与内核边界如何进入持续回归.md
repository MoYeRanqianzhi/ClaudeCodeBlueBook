# 机制验证导航：message lineage、governance key 与 authority object 如何进入持续回归

持续回归真正要验证的，不是文档标题有没有更新，而是三条对象链有没有被替身化：

- Prompt: `message lineage -> projection consumer -> protocol transcript -> continuation object -> continuation qualification`
- Governance: `governance key -> externalized truth chain -> typed ask -> decision window -> continuation pricing -> durable/transient cleanup`
- Structure: `authority object -> per-host authority width -> event-stream-vs-state-writeback -> freshness gate -> stale worldview / ghost capability`

## 1. 请求装配控制面验证线

优先验证这些对象：

1. `message lineage` 的三键内核是否仍完整。
2. display、model API、SDK/control、handoff/resume 是否仍各吃各的 projection consumer。
3. `protocol transcript` 是否仍保护 tool pairing、attachment 迁移与合法消息 ABI。
4. compact、resume 与 handoff 是否仍围绕同一个 `continuation object`。
5. `continuation qualification` 与 `cache-safe fork reuse` 是否仍在阻止平行世界续写。

阅读顺序：

1. `../architecture/82`
2. `../guides/99`
3. `../playbooks/77`

如果验证开始只剩 display transcript、摘要 prose 或“字段还在”，转 `../navigation/40` 与 `../casebooks/73`。

## 2. 当前世界准入主权验证线

优先验证这些对象：

1. `governance key` 是否仍是统一主键，而不是 mode 名字的替身。
2. `externalized truth chain` 是否仍把 source、effective settings、pending action、worker state 与 context usage 串成同一条外化真相。
3. `typed ask` 是否仍是正式事务，而不是“弹窗出现过”。
4. `decision window` 是否仍在解释为什么当前值得继续、为什么当前应该拒收。
5. `continuation pricing` 是否仍把时间、能力、上下文与 cleanup 放在同一判断链上。

阅读顺序：

1. `../architecture/83`
2. `../guides/100`
3. `../playbooks/78`

如果验证开始只看 mode 历史、token 条、审批出现过或默认继续，转 `../navigation/40` 与 `../casebooks/74`。

## 3. one writable present 验证线

优先验证这些对象：

1. `authority object` 是否仍是唯一当前真相。
2. `per-host authority width` 是否仍限制各 consumer 只能读自己配读的宽度。
3. `event-stream-vs-state-writeback` 是否仍把时间线通道和 present-truth 通道分开。
4. `freshness gate` 是否仍在 stale writer 写回前先撤权。
5. `stale worldview / ghost capability` 是否仍会在过期后被正式驱逐。

阅读顺序：

1. `../architecture/84`
2. `../guides/101`
3. `../playbooks/79`

如果验证开始只看 pointer 健康感、reconnect 好运气、监控平静感或恢复成功率，转 `../navigation/40` 与 `../casebooks/75`。

## 4. 为什么验证层更适合落在 playbooks

因为这里真正要固定的不是理念，而是：

1. 哪些症状说明对象已经被替身化。
2. 哪些证据足够证明对象仍成立。
3. 哪些 drift 必须直接拒收。
4. 哪些失真需要进入长期复盘与防再发。

这比“再解释一遍机制”更接近长期运行中的回归与拒收。

## 5. 苏格拉底式自检

在你准备宣布“这三条机制已经稳定”前，先问自己：

1. 我现在点名的是正式对象，还是点名它的投影替身。
2. 一旦 drift 发生，团队是否知道第一条 reject path 要回到哪个对象。
3. 当前验证证据是在证明 present truth，还是只是在证明某个展示面看起来还平静。
4. 我保住的是同一条对象链，还是只保住了一组熟悉名词。
