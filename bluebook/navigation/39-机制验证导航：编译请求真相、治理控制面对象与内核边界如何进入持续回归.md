# 机制验证导航：编译请求真相、治理控制面对象与内核边界如何进入持续回归

这一篇不再回答“这些机制对象是什么”，而是回答：

- 当 `compiled request truth`、`governance control plane` 与 `evolvable kernel object boundary` 已经在机制层成立之后，团队该怎样持续验证它们仍然成立。

它主要回答五个问题：

1. 为什么机制回灌层之后，蓝皮书仍需要单独讨论“机制验证层”。
2. 为什么 Prompt 线如果不能持续验证 request continuity，就会重新退回原文 prompt 崇拜。
3. 为什么治理线如果不能持续验证 decision window 与 continuation gate，就会重新退回 KPI 与审批流程。
4. 为什么结构线如果不能持续验证 authority、transition 与 anti-zombie，就会重新退回目录审美与作者记忆。
5. 怎样用苏格拉底式追问避免把这层写成“又一组测试清单”。

## 1. 编译请求真相验证线

适合在这些问题下阅读：

- 怎样证明当前保护的仍是同一个 `compiled request truth`。
- 怎样让 section continuity、stable bytes、protocol rewrite 与 lawful forgetting 进入日常回归。

稳定阅读顺序：

1. `../architecture/82`
2. `../guides/99`
3. `../playbooks/77`

这条线的核心不是：

- 再讲一次 Prompt 设计方法

而是：

- 把 request object continuity 变成可重复验证、可复盘拒收的运行手册

## 2. 治理控制面对象验证线

适合在这些问题下阅读：

- 怎样证明当前治理对象仍围绕同一个 authority source、decision window 与 continuation gate 成立。
- 怎样避免治理系统重新退回只看 mode、只看阈值、只看最终结果。

稳定阅读顺序：

1. `../architecture/83`
2. `../guides/100`
3. `../playbooks/78`

这条线的核心不是：

- 再补一层治理规则

而是：

- 把 governance object 的边界、仲裁、继续条件与回退对象压成持续回归的检查点

## 3. 可演化内核对象验证线

适合在这些问题下阅读：

- 怎样证明 authority、single-source、transition legality、recovery asset 与 anti-zombie 仍围绕同一内核对象成立。
- 怎样避免结构先进性重新退回“有规则、但没人验证它是否还在工作”。

稳定阅读顺序：

1. `../architecture/84`
2. `../guides/101`
3. `../playbooks/79`

这条线的核心不是：

- 再做一次结构评审

而是：

- 把内核对象边界变成长期运行里的持续验证条件

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- 机制对象怎样成立

而是：

1. 哪些症状说明它已经不成立。
2. 哪些观测足够证明它还成立。
3. 哪些情况必须直接拒收并回退。
4. 哪些 drift 应该进入复盘与防再发。

这些都更接近：

- 运营层和回归层

## 5. 苏格拉底式自检

在你准备宣布“机制层已经写完了”前，先问自己：

1. 这套机制是否已经有明确回归症状。
2. 这套机制是否已经有明确拒收条件。
3. 团队是否知道它失效时先看哪个对象、哪个窗口、哪个边界。
4. 我保护的是一个还能持续验证的机制对象，还是一段已经写下来的架构解释。
