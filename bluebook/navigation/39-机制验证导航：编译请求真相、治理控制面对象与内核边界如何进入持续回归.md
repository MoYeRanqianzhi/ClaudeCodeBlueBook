# 机制验证导航：编译请求真相、治理控制面对象与内核边界如何进入持续回归

这一篇不再回答“这些机制对象是什么”，而是回答：

- 当 `compiled request truth`、`governance control plane` 与 `evolvable kernel object boundary` 已经在机制层成立之后，团队该怎样持续验证它们仍然成立。

它主要回答五个问题：

1. 为什么机制回灌层之后，蓝皮书仍需要单独讨论“机制验证层”。
2. 为什么请求装配控制面如果不能持续验证 authority chain、protocol transcript 与 continuation object，就会重新退回 Prompt 文案崇拜与平行世界续写。
3. 为什么当前世界准入主权如果不能持续验证 trusted inputs、最小可见面与 continuation gate，就会重新退回 mode、弹窗与默认继续。
4. 为什么 one writable present 如果不能持续验证 single-writer authority、recovery asset 与 anti-zombie，就会重新退回健康投影、恢复成功率与作者记忆。
5. 怎样用苏格拉底式追问避免把这层写成“又一组测试清单”。

## 1. 请求装配控制面验证线

适合在这些问题下阅读：

- 怎样证明当前保护的仍是同一个请求装配控制面对象，而不是一张更像真的 Prompt 卡片。
- 怎样让 authority chain、protocol transcript、continuation object 与 cache-safe fork 进入日常回归。

稳定阅读顺序：

1. `../architecture/82`
2. `../guides/99`
3. `../playbooks/77`

这条线的核心不是：

- 再讲一次 Prompt 设计方法

而是：

- 把 request object continuity 变成可重复验证、可复盘拒收的运行手册
- 如果这些验证开始退回字段存在性、display transcript 或 summary prose，转 `../navigation/40` 与 `../casebooks/73`

## 2. 当前世界准入主权验证线

适合在这些问题下阅读：

- 怎样证明当前治理对象仍围绕同一个 trusted input chain、typed ask、最小可见面与 continuation gate 成立。
- 怎样避免治理系统重新退回只看 mode、只看 token 条、只看审批出现过。

稳定阅读顺序：

1. `../architecture/83`
2. `../guides/100`
3. `../playbooks/78`

这条线的核心不是：

- 再补一层治理规则

而是：

- 把世界准入主权的边界、仲裁、可见面与继续条件压成持续回归的检查点
- 如果这些验证开始退回 calm dashboard、免费可见面或默认继续，转 `../navigation/40` 与 `../casebooks/74`

## 3. one writable present验证线

适合在这些问题下阅读：

- 怎样证明 single-writer authority、recovery asset 与 anti-zombie 仍围绕同一当前态成立。
- 怎样避免结构先进性重新退回 pointer 健康感、reconnect 好运气与作者说明。

稳定阅读顺序：

1. `../architecture/84`
2. `../guides/101`
3. `../playbooks/79`

这条线的核心不是：

- 再做一次结构评审

而是：

- 把 one writable present 的权威边界变成长期运行里的持续验证条件
- 如果这些验证开始退回 pointer、telemetry、author memory 或 health projection，转 `../navigation/40` 与 `../casebooks/75`

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- 机制对象怎样成立

而是：

1. 哪些症状说明它已经不成立。
2. 哪些观测足够证明它还成立。
3. 哪些情况必须直接拒收并回退。
4. 哪些 drift 应该进入复盘与防再发。

这些都更接近：

- 运营层和回归层

## 5. 苏格拉底式自检

在你准备宣布“机制层已经写完了”前，先问自己：

1. 这套机制是否已经有明确回归症状。
2. 这套机制是否已经有明确拒收条件。
3. 团队是否知道它失效时先看哪个对象、哪个窗口、哪个边界。
4. 我保护的是一个还能持续验证的机制对象，还是一段已经写下来的架构解释。
