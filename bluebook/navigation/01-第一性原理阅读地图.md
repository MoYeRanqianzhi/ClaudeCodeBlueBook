# 第一性原理审读地图

面对任一 Agent runtime，先用这张地图判断它究竟在回答什么。

核心判断不该再只是一句“它是个运行时”，而应先压成三句：

1. Prompt 线
   - 消息血缘、协议转录与继续资格在进入模型前，是否已经先被编译成同一个世界
2. 治理线
   - 一切扩张是否仍按同一价格秩序合法进入当前世界
3. 源码质量线
   - 当前真相是否仍被正式保护，不让过去与失控复杂度写坏现在

“观察、决策、行动、记忆、协作、恢复、治理、成本”这八个问题，只是这三句判断在工程现场里的第二层诊断框架。

这里还应再固定一条 continuity crosswalk：

- `记忆 / 恢复 / 交接` 不构成第四条母线；它们只是 Prompt `Continuation`、治理 `continuation pricing` 与源码质量 `recovery non-sovereignty / anti-zombie` 在不同时间尺度上的展开。
- 所以 continuity 的最短动作链不另起新线，只认 `continue qualification -> continuation pricing -> recovery non-sovereignty / anti-zombie`。

如果继续把这三句判断压成最短动作链，也只该剩下三行：

1. Prompt 线
   - `compile -> protocolize -> preserve -> continue -> explain`
2. 治理线
   - `price -> externalize -> arbitrate -> continue-or-stop -> cleanup`
3. 源码质量线
   - `surface -> writeback -> freshness -> evict`

## 1. 八个不可约问题是第二层诊断框架

从零设计一个 coding agent，至少要回答八个问题：

1. 观察：它如何得到文件、命令、远端系统与状态快照。
2. 决策：它如何在多轮里形成下一步动作。
3. 行动：它如何安全执行文件、shell、网络、远程与扩展调用。
4. 记忆：它如何维持长会话、局部知识与项目规则。
5. 协作：它如何拆任务、隔离上下文、让多个执行单元合作。
6. 恢复：它如何在超长上下文、中断、权限阻塞、重连后继续。
7. 治理：它如何被账号、组织、策略、遥测、信任边界约束。
8. 经济：它如何压缩 token、输出、状态噪音与认知负担。

如果一套系统没有同时回答这八个问题，它更像聊天外壳，而不是工程现场里的代理运行时。

但这八个问题里最容易被写浅的仍是 `记忆 / 恢复 / 成本`：

- 它们在第一性原理层不配自立母线。
- 只有回到 `Continuation / continuation pricing / freshness-eviction discipline`，它们才重新获得制度位置。

## 5. 苏格拉底式追问

### 5.1 如果模型更强，为什么还需要 runtime

因为更强模型提升的是“单步质量”，而不是：

- 权限判定
- host 闭环
- 会话恢复
- 长任务治理
- token 预算

### 5.2 如果工具更多，为什么不一定更强

因为没有统一协议、统一权限、统一恢复与统一状态写回时，更多工具只会制造更多无序自由度。

### 5.3 如果 prompt 更长，为什么不等于更聪明

因为 prompt 的真正上限受制于：

- 装配顺序
- cache 稳定性
- 附件晚绑定
- 工具可用性
- token 预算

### 5.4 如果宿主只能看文本流，为什么会误解 Claude Code

因为宿主真正需要吃的是：

- `control_request` / `control_response`
- `SDKMessage`
- `worker_status`
- `external_metadata`
- 恢复与 snapshot

所以 Claude Code 的对外面首先是 control plane，不只是 answer stream。

## 6. 用这张地图审任一 runtime 说明

面对任何一套 runtime 说明，先问五个问题：

1. 它主要在回答八个不可约问题里的哪一个。
2. 它究竟落在世界进入模型、扩张定价，还是当前真相保护。
3. 它是否明确区分了 public、gated、internal 三层。
4. 它是否给出了 first reject path，而不只是功能列表。
5. 它是否把目标、预算、承载体、边界与回写中的至少一项写成了可操作方法。
