# 第一性原理阅读地图

这一篇不是机制细拆，而是给整本蓝皮书建立统一入口。

核心判断只有一句：

- Claude Code 的本体不是“一个会写代码的模型”，而是“一个同时管理观察、决策、行动、记忆、协作、恢复、治理与成本的运行时”。

## 1. 八个不可约问题

从零设计一个 coding agent，至少要回答八个问题：

1. 观察：它如何得到文件、命令、远端系统与状态快照。
2. 决策：它如何在多轮里形成下一步动作。
3. 行动：它如何安全执行文件、shell、网络、远程与扩展调用。
4. 记忆：它如何维持长会话、局部知识与项目规则。
5. 协作：它如何拆任务、隔离上下文、让多个执行单元合作。
6. 恢复：它如何在超长上下文、中断、权限阻塞、重连后继续。
7. 治理：它如何被账号、组织、策略、遥测、信任边界约束。
8. 经济：它如何压缩 token、输出、状态噪音与认知负担。

如果一套系统没有同时回答这八个问题，它更像聊天外壳，而不是工程现场里的代理运行时。

## 5. 苏格拉底式追问

### 5.1 如果模型更强，为什么还需要 runtime

因为更强模型提升的是“单步质量”，而不是：

- 权限判定
- host 闭环
- 会话恢复
- 长任务治理
- token 预算

### 5.2 如果工具更多，为什么不一定更强

因为没有统一协议、统一权限、统一恢复与统一状态写回时，更多工具只会制造更多无序自由度。

### 5.3 如果 prompt 更长，为什么不等于更聪明

因为 prompt 的真正上限受制于：

- 装配顺序
- cache 稳定性
- 附件晚绑定
- 工具可用性
- token 预算

### 5.4 如果宿主只能看文本流，为什么会误解 Claude Code

因为宿主真正需要吃的是：

- `control_request` / `control_response`
- `SDKMessage`
- `worker_status`
- `external_metadata`
- 恢复与 snapshot

所以 Claude Code 的对外面首先是 control plane，不只是 answer stream。
