# 宿主验收执行导航：Prompt、治理与故障模型验收协议如何进入执行卡、拒收顺序与回退剧本

这一篇不再回答“宿主、SDK、CI 与交接系统到底该消费哪些正式验收对象”，而是回答：

- 当 `compiled request truth`、统一定价治理对象与结构真相面的验收协议已经被写出来之后，团队到底该按什么固定执行顺序验收、拒收、停止灰度与进入回退剧本，才能不重新退回表单化绿灯与经验判断。

它主要回答五个问题：

1. 为什么宿主验收协议层之后，蓝皮书仍需要单独讨论“宿主验收执行层”。
2. 为什么 Prompt 线如果不继续压成执行卡、拒收顺序与回退剧本，就会重新退回 prompt 截图、摘要交接与最后一条消息判断。
3. 为什么治理线如果不继续压成执行卡、拒收顺序与回退剧本，就会重新退回 mode 面板、审批弹窗与 token 仪表盘。
4. 为什么结构线如果不继续压成执行卡、拒收顺序与回退剧本，就会重新退回 pointer、成功率与作者口述。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的验收表单”。

## 1. Prompt 宿主验收执行线

适合在这些问题下阅读：

- 宿主、CI、评审与交接系统到底该按什么顺序执行 Prompt 宿主验收，而不是只检查卡片是否存在。
- 哪些 reject reason 一旦出现就必须停止继续、拒绝交接并进入回退。

稳定阅读顺序：

1. `../api/54`
2. `../playbooks/35`
3. `../casebooks/28`
4. `../guides/57`

这条线的核心不是：

- 再讲一次 Prompt 宿主验收协议

而是：

- 把 Prompt 宿主验收协议真正压成值班、CI、评审与交接可以共享执行的验收卡、拒收顺序与回退剧本

- 哪些 reject reason 一旦出现就说明统一定价控制面已经退回交互投影。

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- 该消费什么字段
- 该怎样纠偏
- 哪些现象最像坏样例

而是：

1. 现场先看哪一个对象。
2. 看见什么必须立刻拒收。
3. 谁有权宣布继续、停止灰度或进入回退。
4. 回退到底回到哪个对象边界，而不是回到哪几句说明。

这些都更接近：

- runtime execution layer、值班顺序与长期运行语义

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主验收执行层已经完整”前，先问自己：

1. 我写的是对象级执行顺序，还是一套更细的经验判断。
2. 这些 reject reason 能否被宿主、CI、评审与交接系统共享，而不是各自翻译一次。
3. 当前判断主体到底是谁，消费的到底是哪一个对象。
4. 当前拒收之后，回退的是正式对象边界，还是回退到某种 UI 成功感。
