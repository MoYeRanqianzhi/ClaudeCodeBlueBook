# 宿主迁移失真导航：request compiler、externalized truth chain 与 current-truth writeback 如何重新退回伪交接、假灰度与回退幻觉

这一篇不再回答“宿主迁移演练该怎样执行”，而是回答：

- 为什么团队明明已经把 `request compiler`、`externalized truth chain` 与 `current-truth writeback` 写成交接包、灰度记录与回退演练，仍会重新退回截图交接、假窗口对齐、成功率回退与作者口述。

它主要回答五个问题：

1. 为什么宿主迁移演练层之后，蓝皮书仍需要单独讨论“宿主迁移失真反例层”。
2. 为什么 Prompt 线如果不继续指出伪交接、黑箱灰度与继续资格回退幻觉，就会重新退回摘要 handoff。
3. 为什么治理线如果不继续指出权限账本缺席、假窗口对齐与免费继续回退幻觉，就会重新退回 mode 面板与弹窗记忆。
4. 为什么结构线如果不继续指出伪恢复采纳、指针健康幻觉与写回分叉，就会重新退回 pointer 崇拜与成功率崇拜。
5. 怎样用苏格拉底式追问避免把这层写成“演练做得还不够仔细”。

## 1. Prompt 宿主迁移失真线

适合在这些问题下阅读：

- 为什么 Prompt 迁移演练明明已经有交接包、灰度记录与回退 drill，团队仍会重新把它消费成截图、摘要与黑箱指标。
- 哪些“看起来已经交接好了”的信号，其实正在重新制造第二真相。

稳定阅读顺序：

1. `../casebooks/28`
2. `../playbooks/32`
3. `../guides/54`

这条线的核心不是：

- 再讲一次 Prompt 迁移怎么做

而是：

- 指出 Prompt 演练最常怎样死于伪交接、假灰度与继续资格幻觉

## 2. 治理宿主迁移失真线

适合在这些问题下阅读：

- 哪些“看起来治理已经对齐”的信号，其实正在重新免费化扩张。

稳定阅读顺序：

1. `../casebooks/29`
2. `../playbooks/33`
3. `../guides/55`

这条线的核心不是：

而是：

- 指出治理演练最常怎样死于权限账本缺席、假窗口对齐与免费继续回退幻觉

## 3. 故障模型宿主迁移失真线

适合在这些问题下阅读：

- 为什么 `current-truth writeback` 迁移演练明明已经有交接包、灰度记录与回退 drill，团队仍会重新把它消费成 pointer、成功率与作者说明。

稳定阅读顺序：

2. `../playbooks/34`
3. `../guides/56`

这条线的核心不是：

- 再讲一次 `current-truth surface`

## 4. 为什么这层更适合落在 casebooks

因为这一层最关键的问题已经不是：

- 工单是否完整
- 演练是否执行了

而是：

1. 哪些交接样例最像真的，其实只是更漂亮的摘要。
2. 哪些灰度记录最像对齐，其实只是把黑箱指标当正式对象。
3. 哪些回退动作最像成功，其实只是把对象级回退重新退回文件、截图或成功率。
4. later 维护者最容易被哪些“像成功”的信号骗过。

这些都更接近：

- 失真样本层与反例层

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主迁移已经稳定”前，先问自己：

1. 我看到的是 `request compiler / externalized truth chain / current-truth writeback` 的正式对象交接，还是更漂亮的截图与摘要。
2. 我看到的是对象级灰度信号，还是一组黑箱通过率。
3. 我看到的是对象级回退，还是文件回退、成功率与作者口述。
4. 这份成功感来自机制对象真的成立，还是来自我们换了一套更高级的第二真相。
