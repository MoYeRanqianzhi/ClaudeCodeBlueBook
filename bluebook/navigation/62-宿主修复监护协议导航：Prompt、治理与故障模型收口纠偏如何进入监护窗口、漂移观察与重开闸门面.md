# 宿主修复监护协议导航：Prompt、治理与故障模型收口纠偏如何进入监护窗口、漂移观察与重开闸门面

这一篇不再回答“宿主修复收口失真该怎样纠偏”，而是回答：

- 当 closeout correction 已经把假完成、假交接与假重开压回正式对象之后，团队到底该怎样把 `monitor_only`、handoff watch、drift observation 与 reopen gate 继续压成新的宿主可消费规则面，才能不让“继续观察”重新退回口头建议。

它主要回答五个问题：

1. 为什么宿主修复收口纠偏层之后，蓝皮书还必须继续长出一层“宿主修复监护协议层”。
2. 为什么 Prompt 线如果不继续把 watch window、baseline drift ledger 与 reopen gate 压成正式对象，就会重新把 Prompt 魔力退回 closeout note、handoff 故事与临场判断。
3. 为什么治理线如果不继续把 authority watch、ledger residue、window watch 与 continuation repricing watch 压成正式对象，就会重新把安全设计与省 token 设计退回 mode 面板、usage 仪表盘与“先保守点”。
4. 为什么结构线如果不继续把 authority watch、writeback watch、anti-zombie watch 与 boundary quarantine 压成正式对象，就会重新把源码先进性退回 pointer 健康感、监控转绿与作者口头说明。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的观察清单”。

## 1. Prompt 宿主修复监护协议线

如果问题是：

- 宿主、CI、评审与交接系统到底该消费哪些正式对象，才能在 Prompt closeout 之后继续围绕同一个 `compiled request truth` 观察漂移、阻断坏 handoff，并在必要时 reopen。
- 哪些 watch verdict 一旦出现，就说明 Prompt 世界虽然没有立即 reopen，但也绝不能被当成彻底结束。

建议顺序：

1. `../api/63`
2. `../guides/66`
3. `../casebooks/37`
4. `../playbooks/41`

这条线的核心不是：

- 再讲一次 Prompt 宿主修复收口怎么纠偏

而是：

- 把 Prompt closeout 之后的 watch、handoff 与 reopen gate 压成正式监护对象

## 2. 治理宿主修复监护协议线

如果问题是：

- 团队到底该消费哪些正式对象，才能在治理 closeout 之后继续围绕同一个统一定价对象观察 authority 漂移、ledger 尾账、window 复活与免费继续。
- 哪些 watch verdict 一旦出现，就说明治理对象虽然没有立即爆炸，但已经重新失去统一定价。

建议顺序：

1. `../api/64`
2. `../guides/67`
3. `../casebooks/38`
4. `../playbooks/42`

这条线的核心不是：

- 再讲一次治理宿主修复收口怎么纠偏

而是：

- 把治理 closeout 之后的 authority watch、window watch 与 repricing watch 压成正式监护对象

## 3. 结构宿主修复监护协议线

如果问题是：

- 团队到底该消费哪些正式对象，才能在结构 closeout 之后继续围绕同一个 authority object 观察 writeback 漂移、anti-zombie 回归与 boundary reopening。
- 哪些 watch verdict 一旦出现，就说明结构真相面虽然还没彻底失守，但已经重新长出假连续性。

建议顺序：

1. `../api/65`
2. `../guides/68`
3. `../casebooks/39`
4. `../playbooks/43`

这条线的核心不是：

- 再讲一次结构宿主修复收口怎么纠偏

而是：

- 把结构 closeout 之后的 authority watch、writeback watch、anti-zombie watch 与 boundary quarantine 压成正式监护对象

## 4. 为什么这层更适合落在 api

因为这一层最关键的问题已经不是：

- 先改哪个坏信号
- 先救哪个正式对象
- 先 hard reject 哪个 drift

而是：

1. 监护窗口到底是什么。
2. drift ledger 到底记录什么。
3. handoff watch 与 reopen gate 到底共享哪些字段。
4. 哪些 watch verdict 必须被宿主、CI、评审与交接共同消费。

这些都更接近：

- host-consumable rule surface、watch contract 与 post-closeout ABI

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复监护协议已经完整”前，先问自己：

1. 我写的是 post-closeout 的正式对象，还是一份更细的观察建议。
2. monitor_only 在我的系统里是正式 watch verdict，还是一句‘先盯着’。
3. later 团队拿到的是 drift object，还是一些经验提醒。
4. reopen 在我的系统里由正式 gate 裁定，还是由现场直觉裁定。
5. 当前监护协议保护的是单一真相的延续，还是只是在给完成叙事补售后。
