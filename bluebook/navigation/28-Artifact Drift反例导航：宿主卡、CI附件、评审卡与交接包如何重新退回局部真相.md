# Artifact Drift反例导航：宿主卡、CI附件、评审卡与交接包如何重新退回局部真相

这一篇不新增新的制度判断，而是回答一个更接近执行失真现场的问题：

- 当团队已经有 host artifact samplebook 之后，最常见的失败不再是“没有工件”，而是这些工件已经存在，却重新退回各自维护自己的局部真相。

它主要回答四个问题：

1. 为什么 Artifact Samplebook 层之后仍需要单独讨论 artifact drift casebook 层。
2. 为什么宿主卡、CI 附件、评审卡与交接包最常见的坏解法，不是完全缺失，而是“各自都有一份看起来像真的版本”。
3. 怎样把 Prompt、安全/省 token 与结构演化三条线分别压成 artifact drift 的正式反例。
4. 怎样用苏格拉底式追问避免把这层写成“抱怨别人不会填表”。

## 1. Prompt Artifact 反例

适合在这些问题下阅读：

- 为什么 Prompt 线明明已经有宿主卡、CI 附件、评审卡与交接包，团队却仍会退回原文 prompt、CI 绿灯、评审总结与摘要包四套局部真相。
- 为什么 Prompt 魔力进入 artifact contract 之后，仍会在工件层重新退回文案崇拜。

稳定阅读顺序：

1. `27`
2. `../casebooks/19`

这条线的核心不是：

- 再补一轮 Prompt 样例

而是：

- 看清 Prompt artifact 本身最常会怎样重新说谎

- 为什么安全与省 token 进入 artifact contract 之后，仍会在工件层重新退回事后解释。

- 为什么源码先进性进入 artifact contract 之后，仍会在工件层重新退回目录审美与作者权威。

1. `27`
2. `../casebooks/21`

- 再补一轮结构样例

## 4. 一句话用法

如果：

- `27` 回答“这套共享工件协议真正填出来长什么样”

那么：

- `28` 回答“这些工件最常会怎样重新退回局部真相”

## 5. 苏格拉底式自检

在你准备宣布“团队已经能靠统一 artifact 运转”前，先问自己：

1. 四类工件共享的是同一对象，还是只是名字很像的四份材料。
2. 当前失真是因为没填字段，还是因为 shared header 被不同角色重新各自解释。
3. 我是在记录工件缺失，还是在记录工件存在却重新说谎。
4. 如果拿掉原作者，这些反例是否足以让后来者看出哪一类局部真相正在冒充系统真相。
