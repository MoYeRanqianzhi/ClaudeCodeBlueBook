# 宿主验收纠偏导航：Prompt、治理与故障模型执行失真如何压回固定顺序、拒收升级路径与改写模板骨架

这一篇不再回答“宿主验收执行为什么会重新说谎”，而是回答：

- 当表单化绿灯、假 reject、伪 rollback、mode 投影、breadcrumb 篡位与监控绿灯已经被识别出来之后，团队到底该按什么固定顺序纠偏、升级拒收与改写模板骨架，才能不重新退回更制度化的第二真相。

它主要回答五个问题：

1. 为什么宿主验收失真层之后，蓝皮书仍需要单独讨论“宿主验收纠偏层”。
2. 为什么 Prompt 线如果不继续压成固定纠偏顺序、拒收升级路径与模板骨架，就会重新把 Prompt 魔力退回表单工艺。
3. 为什么治理线如果不继续压成固定纠偏顺序、拒收升级路径与模板骨架，就会重新把安全设计与省 token 设计退回 mode 和 dashboard。
4. 为什么结构线如果不继续压成固定纠偏顺序、拒收升级路径与模板骨架，就会重新把源码先进性退回目录美学、日志繁荣与成功率崇拜。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的整改清单”。

## 1. Prompt 宿主验收纠偏线

适合在这些问题下阅读：

- 明明已经识别出表单化绿灯、假 reject 与伪 rollback，Prompt 宿主验收到底该先纠什么、后纠什么，而不是继续靠 reviewer 心法做 patch。
- 哪些现象一旦出现应直接升级为硬拒收，而不是继续观察。

稳定阅读顺序：

1. `../casebooks/31`
2. `../guides/60`
3. `../playbooks/35`
4. `../api/54`

这条线的核心不是：

- 再讲一次 Prompt 宿主验收执行反例

而是：

- 把 Prompt 执行失真真正压回 `compiled request truth`、`protocol transcript`、`lawful forgetting object` 与 `continue qualification` 的固定纠偏顺序

## 4. 为什么这层更适合落在 guides

因为这一层最关键的问题已经不是：

- 哪些执行动作在说谎
- 哪个坏样例最像现在的问题

而是：

1. 遇到这些执行失真时，纠偏应该按什么顺序发生。
2. 哪些现象必须立即升级拒收，而不是继续灰度。
3. 哪些模板字段必须被替换、重命名或升级成正式对象。
4. 怎样把反形式主义要求真正写回团队可复用模板。

这些都更接近：

- builder-facing correction guide、整改顺序与模板改写层

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主验收纠偏层已经完整”前，先问自己：

1. 我写的是对象级纠偏顺序，还是在更细地描述坏现象。
2. 我升级的是共享 reject 语义，还是只升级了措辞。
3. 我改写的是对象模板骨架，还是只在表单上多加了几栏。
4. 这些纠偏动作是否真正让 later 维护者更难被绿灯幻觉骗过。
