# 宿主修复稳态纠偏执行导航：Prompt、治理与故障模型稳态纠偏协议如何进入修正卡、执行顺序与再入场责任演练

这一篇不再回答“宿主修复稳态纠偏协议该消费哪些正式对象”，而是回答：

- 当 correction object、reject semantics 与 threshold retention 已经被压成正式协议之后，团队到底该按什么固定执行顺序运行 correction card、给出 verdict、触发 re-entry / reopen drill，才能不让协议重新退回作者脑内顺序、值班经验与 prose 解释。

它主要回答五个问题：

1. 为什么宿主修复稳态纠偏协议层之后，蓝皮书仍需要单独讨论“宿主修复稳态纠偏执行层”。
2. 为什么 Prompt 线如果不继续把 correction object 压成 correction card、recovery verdict order、requalification drill 与 threshold reinstatement drill，就会重新把 Prompt 魔力退回 steady note、summary 与“最近没出事”。
3. 为什么治理线如果不继续把 correction object 压成 correction card、correction verdict order、capability recustody 与 threshold rebinding drill，就会重新把安全设计与省 token 设计退回 mode 面板、usage dashboard 与保守建议。
4. 为什么结构线如果不继续把 correction object 压成 correction card、correction verdict order、archive restitution 与 reopen responsibility drill，就会重新把源码先进性退回 pointer 健康感、telemetry 转绿、archive prose 与作者说明。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的纠偏值班表”。

## 1. Prompt 宿主修复稳态纠偏执行线

适合在这些问题下阅读：

- 宿主、CI、评审与交接系统到底该按什么顺序证明 Prompt 世界已经把假稳态纠偏回同一个 `compiled request truth`。
- 哪些 correction verdict 一旦出现就说明当前必须冻结 continuation、阻断 handoff 并进入 re-entry / reopen drill。

稳定阅读顺序：

1. `../playbooks/53`
2. `../api/72`
3. `../guides/75`
4. `../playbooks/50`

这条线的核心不是：

- 再讲一次 Prompt steady-state correction protocol 消费哪些字段

而是：

- 把 Prompt steady-state correction protocol 真正压成 correction card、recovery verdict order、requalification drill 与 threshold reinstatement drill

- 哪些 correction verdict 一旦出现就说明 capability expansion 不能继续放行，必须进入 liability hold、re-entry 或 reopen drill。

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- correction object 叫什么
- 哪些 reject / escalation 语义该被共享
- 哪些 threshold / reservation 字段该被长期保留

而是：

1. 现场先看哪一张 correction card。
2. 先验哪个 correction object。
3. 看见什么必须立刻降级为 `hard_reject`、`reentry_required`、`liability_hold` 或 `reopen_required`。
4. 谁有权宣布 restored，谁只能建议。
5. later 团队再入场时到底回到哪个正式对象边界，而不是回到哪种“现在看起来很稳”的感觉。

这些都更接近：

- runtime correction execution layer、固定执行顺序与 re-entry / reopen 运行语义

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复稳态纠偏执行层已经完整”前，先问自己：

1. 我写的是对象级纠偏执行顺序，还是一套更细的值班表单。
2. 这些 correction verdict 能否被宿主、CI、评审与交接共享，而不是各自翻译一次。
3. 当前 correction 一旦失效，回退的是正式对象边界，还是回到 steady note、dashboard、pointer 与作者记忆。
4. later 团队只看 correction card，能否重建同一判断并执行同一降级动作。
5. 当前执行保护的是无人继续盯防时的真相延续，还是只是在把纠偏协议运行成一套更制度化的售后流程。
