# Agent Runtime深化导航：上下文编译、治理控制面与可演化内核

这一篇不再回答“怎样构建 Agent Runtime”，而是回答：

- 当对象、预算、恢复与宿主模板已经固定之后，下一步该怎样继续把 Claude Code 的 Prompt 魔力、安全/省 token 同构与源码先进性压成三条更细的 builder-facing 方法线。

它主要回答五个问题：

1. 为什么 Runtime 构建层之后，还必须继续长出“上下文编译”“治理控制面”“可演化内核”三联层。
2. 为什么 Prompt 魔力最终要回到“世界如何进入模型”，而不是“prompt 怎样写得更强”。
3. 为什么安全与省 token 最终要回到同一张治理控制面，而不是两套局部优化。
4. 为什么源码先进性最终要回到“未来维护者如何继续反对当前实现”，而不是目录看起来更整齐。
5. 怎样用苏格拉底式追问避免把 Runtime 构建重新退回功能模仿。

## 1. 上下文编译线

如果问题是：

- 为什么 Claude Code 的 prompt 有魔力。
- 为什么真正强的 prompt 不是长文案，而是稳定前缀、动态边界、协议真相与合法遗忘共同成立。

建议顺序：

1. `../guides/48`
2. `../philosophy/78`
3. `../architecture/73`
4. `../architecture/70`

这条线的核心不是：

- 再补几条 prompt 经验法则

而是：

- 把 Prompt 理解成上下文准入编译器，先决定世界怎样进入模型，再决定模型怎样继续发言

## 2. 治理控制面线

如果问题是：

- 为什么安全设计与省 token 设计本质上是一回事。
- 为什么 host control、settings、MCP、Context Usage、continuation 最后都要落成同一张治理控制面。

建议顺序：

1. `../guides/49`
2. `../philosophy/79`
3. `../api/28`
4. `../api/32`
5. `../architecture/62`
6. `../architecture/71`

这条线的核心不是：

- 再补一个统一预算器

而是：

- 先决定允许什么、看见什么、外置什么、继续多久，再让动作、能力、上下文与时间围绕同一顺序被定价

## 3. 可演化内核线

如果问题是：

- 为什么 Claude Code 的源码结构看起来“更成熟”。
- 为什么真正先进的结构，不是目录分层漂亮，而是权威入口、single-source、anti-cycle seam 与 anti-zombie 写法共同成立。

建议顺序：

1. `../guides/50`
2. `../philosophy/80`
3. `../architecture/58`
4. `../architecture/63`

这条线的核心不是：

- 再强调一次工程整洁

而是：

- 让后来维护者拥有继续拒绝半真相、拒绝陈旧写回、拒绝错误扩张的正式结构条件

## 4. 三条线为什么必须一起看

因为它们其实都在回答同一件事：

1. 哪些对象有资格进入当前世界。
2. 哪些边界有资格修改当前世界。
3. 哪些消费者有资格继续消费当前世界。

所以更稳的联合阅读顺序是：

1. `../guides/48`
2. `../guides/49`
3. `../guides/50`
4. `../philosophy/78`
5. `../philosophy/79`
6. `../philosophy/80`

## 5. 为什么这层仍然落在 guides / philosophy

因为这一层最关键的问题已经不是：

- 源码里有什么

而是：

1. 我该怎样把世界编译成模型可行动的协议真相。
2. 我该怎样把安全、成本与继续条件写成同一控制面。
3. 我该怎样让结构先替未来维护者保留反驳和恢复能力。

这些都更接近：

- builder-facing 的方法与判断

## 6. 苏格拉底式自检

在你准备宣布“我已经把 Claude Code 的高级感迁移出来了”前，先问自己：

1. 我固定的是世界如何进入模型，还是只固定了 prompt 文案。
2. 我固定的是治理顺序，还是只固定了几条局部限制。
3. 我固定的是未来维护者的反驳能力，还是只固定了当前实现路径。
4. 这三条线最后是否仍然围绕同一组对象、边界、预算与恢复资产成立。
