# Host Artifact样例导航：宿主卡、CI附件、评审卡与交接包的最小共享样例

这一篇不新增新的制度判断，而是回答一个更接近使用现场的问题：

- 当团队已经有 host artifact contract 之后，怎样把这些正式工件协议继续落成最小共享样例，让宿主实现者、CI 维护者、评审者与交接者不再从空白表格起步。

它主要回答四个问题：

1. 为什么 Host Artifact Contract 层之后仍需要单独讨论 artifact samplebook 层。
2. 为什么 Prompt、治理与结构三条线都需要把 shared header 继续翻译成最小可填写样例。
3. 怎样让宿主卡、CI附件、评审卡与交接包先共享同一对象真相，再按角色分化展示。
4. 怎样用苏格拉底式追问避免把这层写成“更多示例截图”和“更多格式抄写”。

## 1. Prompt Artifact 样例库

适合在这些问题下阅读：

- 怎样把 Prompt 线的宿主卡、CI附件、评审卡与交接包真正填出来，而不是只知道 header 里应该有什么。
- 怎样避免四类 Prompt 工件再次退回原文 prompt、宿主卡片、CI 绿灯与摘要包四套局部真相。

稳定阅读顺序：

1. `26`
2. `../playbooks/17`

这条线的核心不是：

- 再补一轮 Prompt contract

而是：

- 看清同一 Prompt object contract 被四类工件真实填出来时长什么样

## 4. 一句话用法

如果：

- `26` 回答“怎样把统一审读对象压成正式工件协议”

那么：

- `27` 回答“这些正式工件协议真正填出来时长什么样”

## 5. 苏格拉底式自检

在你准备宣布“团队已经能按同一套 artifact contract 运转”前，先问自己：

1. 我展示的是共享对象的最小样例，还是只是四份格式相近的表单。
2. 这些样例是否真的能让后来者不靠作者继续判断。
3. 我是在降低未来工件分歧，还是在增加新的模板装饰。
4. 如果把字段名拿掉，只看内容，四类工件是否仍然围绕同一对象说话。
