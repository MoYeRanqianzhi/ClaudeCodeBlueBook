# 证据接口导航：从Rollout ABI到宿主消费、回退对象与复盘真相面

这一篇不新增新的制度判断，而是回答一个更接近落地深水区的问题：

- 当团队已经把 rollout 证据压成统一 ABI 之后，怎样让这些证据继续进入宿主消费、评审流程、回退对象边界与复盘真相面，而不是继续停在 playbook 模板层。

它主要回答四个问题：

1. 为什么 rollout ABI 层之后仍需要单独讨论“证据接口层”。
2. 为什么同一套证据必须同时被宿主、评审者、自动化与后来接手者消费。
3. 为什么回退必须按对象边界，而不是按文件回退。
4. 怎样用苏格拉底式追问避免把证据接口层写成“更多表单字段”。

## 1. Prompt 证据接口

适合在这些问题下阅读：

- 怎样让 Prompt Constitution 的 rollout 证据不只停在卡片里，而是变成宿主能消费、评审者能复核、后来者能接手的正式真相面。
- 怎样把 `speaker_chain`、`section_slots_changed`、`stable_prefix_surface`、`dynamic_boundary_surface`、`lawful_forgetting_abi` 这些字段接回 runtime 机制。

稳定阅读顺序：

1. `19`
2. `../playbooks/12`
3. `../architecture/76`
4. `../api/35`

这条线的核心不是：

- 再解释 prompt 为什么强

而是：

- 让 prompt 的升级证据真的能被系统消费

## 2. 治理与省 token 证据接口

适合在这些问题下阅读：

- 怎样让 `worker_status`、`external_metadata`、`Context Usage`、`continuation decision`、`rollback target` 这些信号共同构成治理 rollout 的正式消费面。
- 怎样把“为什么这轮更贵/更稳/该停止/该回退”从故事解释，推进成结构化证据。

稳定阅读顺序：

1. `19`
2. `../playbooks/12`
3. `../architecture/76`
4. `../api/35`
5. `../guides/34`

这条线的核心不是：

- 多看几张指标图

而是：

- 把成本控制、停止条件与回退动作变成宿主可消费的 ABI

## 3. 结构演化与回退对象边界

适合在这些问题下阅读：

- 怎样让 authoritative surface、recovery asset、anti-zombie gate 不只停在源码结构判断，而是进入 rollout 证据和回退对象边界。
- 怎样避免把结构升级写成“改了哪些文件”，却说不清“回退到底应该回退哪一个对象”。

稳定阅读顺序：

1. `19`
2. `../playbooks/12`
3. `../architecture/76`
4. `../api/35`
5. `../guides/34`

这条线的核心不是：

- 再强调结构优雅

而是：

- 让真相迁移、恢复资产与对象回退拥有正式证据面

## 4. 一句话用法

如果：

- `19` 回答“统一 rollout ABI 应该长什么样”

那么：

- `20` 回答“这套 ABI 怎样进入宿主消费、回退对象与复盘真相面”

## 5. 苏格拉底式自检

在你准备宣布“我们已经有统一 rollout 证据”前，先问自己：

1. 这些证据现在只是被写下来，还是已经被正式消费。
2. 宿主、评审者、自动化与后来接手者，看到的是不是同一套证据真相。
3. 如果现在要回退，我知道该回退哪个对象，而不是只知道改过哪些文件。
4. 我是在降低未来判断成本，还是只是在增加更多记录负担。
