# 宿主修复稳态纠偏再纠偏改写纠偏精修纠偏精修执行导航：Prompt、治理与结构 refinement correction repair protocol 如何进入 repair card、共同 reject 顺序与 reopen 演练

这一篇不再回答：

- refinement correction repair protocol 该共享哪些 repair 对象
- 哪些字段属于共同 reject 语义与长期 reopen 责任面

这一篇回答的是：

- 当 `api/90-92` 已经把 Prompt、治理与结构三条 refinement correction fixed order 压成正式 repair 对象协议之后，团队到底该怎样把这些协议继续运行成固定 `repair card`、共同 `reject order` 与正式 `reopen drill`，才不让这条线重新退回更像协议说明的 repair prose、更平静的 dashboard 感受与作者记忆。

它主要回答五个问题：

1. 为什么 `api/90-92` 之后，蓝皮书还必须继续长出一层“宿主修复稳态纠偏再纠偏改写纠偏精修纠偏精修执行层”。
2. 为什么 Prompt 线如果不把 refinement correction repair protocol 继续压成统一 `repair card`、共同 `reject order` 与 `reopen drill`，Prompt 魔力仍会重新退回 UI transcript、handoff packet 与未综合的子 Agent prose。
3. 为什么治理线如果不把 refinement correction repair protocol 继续压成统一 `repair card`、共同 `reject order` 与 `reopen liability drill`，安全设计与省 token 设计仍会重新退回 mode 面板、usage dashboard 与运营感觉。
4. 为什么结构线如果不把 refinement correction repair protocol 继续压成统一 `repair card`、共同 `reject order` 与 `fail-closed reopen drill`，源码先进性仍会重新退回 pointer、telemetry、archive prose 与作者说明。
5. 怎样用第一性原理与苏格拉底式追问，避免把这一层写成“把 repair protocol 再执行得更熟练”的值班化叙事。

- 怎样把 Prompt refinement correction repair protocol 运行成宿主、CI、评审与交接都能共同消费的 `repair card`、共同 `reject order` 与 `reopen drill`。
- 哪些顺序不能交换，否则 authority chain、compiled request lineage、registry-boundary repair、protocol truth 与 repair attestation 会重新退回解释稿。

建议顺序：

1. `../api/90`

## 4. 为什么这一层更适合落在 playbooks

因为这一层真正新增的问题已经不是：

1. 哪些 repair 对象必须共享。
2. 哪些 reject 语义必须跨消费者复用。
3. 哪些 reopen 责任字段必须长期保留。

而是：

1. 谁先看哪一张 `repair card`。
2. 哪个共同 `reject` 必须先触发。
3. 哪个 `re-entry / reopen` drill 不能跳过。
4. 哪些交接动作必须在冻结 continuation、冻结 capability 或冻结 reconnect 之前完成。

这些都更接近：

- execution card
- shared verdict order
- reentry / reopen drill

## 6. 第一性原理与苏格拉底式自检

在你准备宣布“refinement correction repair protocol 已经真正运行起来”前，先问自己：

1. 我现在运行的是同一个 shared repair object，还是四份彼此相像的导出包。
2. 我现在执行的是同一条共同 `reject order`，还是宿主、CI、评审与交接各自的流程感觉。
3. 我现在保留的是正式 `reopen` 责任边界，还是一句“以后有问题再看”。
4. later 团队接手时依赖的是 `repair card`、顺序与 drill，还是仍要回到 prose、dashboard、pointer 与作者记忆。
5. 当前执行保护的是作者退场后的真相延续，还是只是在把 repair protocol 写成更体面的现场 SOP。
