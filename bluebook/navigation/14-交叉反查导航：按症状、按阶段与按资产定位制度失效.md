# 交叉反查导航：按症状、按阶段与按资产定位制度失效

这一篇不新增新的制度判断，而是回答一个更贴近现场的问题：

- 当团队已经有标签字典、源码锚点和记录样例后，怎样从“眼前看见的问题”最快回到制度根因。

它主要回答四个问题：

1. 为什么参考层之后仍需要单独讨论多入口反查层。
2. 为什么症状、阶段与资产三条入口必须并存，而不能只靠标签字典。
3. 怎样把现场观察重新路由回样本、手册、章节与源码入口。
4. 怎样用苏格拉底式追问避免把反查层写成另一份目录堆。

## 1. 按症状进入

如果现场先看到的是：

- `cache break`
- `cost spike`
- `wrong allow / wrong deny`
- `split truth`
- `replay mismatch`
- `stale state`
- `zombie risk`

先读：

1. `casebooks/07`

它解决的是：

- 从表象先判断优先主线该落在 Prompt、治理还是结构
- 从症状先回到样本、手册与解释层

## 2. 按阶段进入

如果你更清楚问题出现在：

- `design`
- `assembly`
- `runtime`
- `recovery`
- `evolution`

先读：

1. `casebooks/08`

它解决的是：

- 这次失效究竟是设计时埋下、组装时带入、运行时触发、恢复时放大，还是演化时累积

## 3. 按资产进入

如果你已经知道受伤的是哪类制度资产，比如：

- `section`
- `boundary`
- `stable bytes`
- `shadow-stub`
- `transport shell`
- `recovery asset`
- `object state`

先读：

1. `casebooks/09`

它解决的是：

- 为什么这些对象应被当成正式制度资产，而不是零散实现细节
- 该回到哪组正文、手册与底盘章节继续查

## 4. 一句话用法

如果：

- `12` 回答“怎样建立索引”
- `13` 回答“怎样直接查定义、锚点与样例”

那么：

- `14` 回答“怎样从现场观察直接进入制度诊断”

## 5. 苏格拉底式自检

在你准备继续全文搜索前，先问自己：

1. 我眼前看到的是症状、阶段，还是某种受损资产。
2. 我缺的是标签定义，还是更短的诊断入口。
3. 我现在需要的是案例、手册、解释层，还是源码锚点。
4. 我是在帮助未来团队更快定位，还是又造了一层只能自己看懂的目录。
