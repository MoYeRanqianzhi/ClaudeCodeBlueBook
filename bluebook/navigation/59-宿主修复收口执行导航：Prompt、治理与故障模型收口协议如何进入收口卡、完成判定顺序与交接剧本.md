# 宿主修复收口执行导航：Prompt、治理与故障模型收口协议如何进入收口卡、完成判定顺序与交接剧本

这一篇不再回答“宿主、SDK、CI、评审与交接系统到底该消费哪些正式收口对象”，而是回答：

- 当 restored request object、authority settlement 与 authority seal 已经被压成正式收口协议之后，团队到底该按什么固定执行顺序宣布修复完成、释放交接、允许 reopen 或阻断 handoff，才能不重新退回‘看起来已经收口’的完成幻觉。

它主要回答五个问题：

1. 为什么宿主修复收口协议层之后，蓝皮书仍需要单独讨论“宿主修复收口执行层”。
2. 为什么 Prompt 线如果不继续压成收口卡、完成判定顺序与 reopen drill，就会重新退回事故总结、summary handoff 与默认继续。
3. 为什么治理线如果不继续压成收口卡、完成判定顺序与 reopen drill，就会重新退回 mode 面板、审批完成感与 token 图表转绿。
4. 为什么结构线如果不继续压成收口卡、完成判定顺序与 reopen drill，就会重新退回 pointer、重连通过感、日志繁荣与恢复成功率。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的完成表单”。

## 1. Prompt 宿主修复收口执行线

适合在这些问题下阅读：

- 宿主、CI、评审与交接系统到底该按什么顺序宣布 Prompt 修复已经正式收口，而不是只检查 closeout verdict 是否写出来了。
- 哪些 verdict reason 一旦出现就必须阻断 handoff、拒绝 closeout 并进入 reopen drill。

稳定阅读顺序：

1. `../api/60`
2. `../playbooks/41`
3. `../casebooks/34`
4. `../guides/63`

这条线的核心不是：

- 再讲一次 Prompt 宿主修复收口协议

而是：

- 把 Prompt 宿主修复收口协议真正压成值班、CI、评审与交接可以共享执行的收口卡、完成判定顺序与 reopen 剧本

## 2. 治理宿主修复收口执行线

适合在这些问题下阅读：

- 团队到底该按什么顺序宣布治理修复已经正式收口，而不是继续围绕 mode 恢复、审批结束与 token 图表判断。
- 哪些 verdict reason 一旦出现就说明统一定价控制面还不能被视为关闭，必须重开治理对象。

1. `../api/61`
2. `../playbooks/42`
3. `../casebooks/35`
4. `../guides/64`

这条线的核心不是：

- 再讲一次治理宿主修复收口协议

- 哪些 verdict reason 一旦出现就说明结构真相面还没有真正被 seal，必须阻断 handoff 并重开恢复。

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- 该消费什么收口字段
- 该怎样纠偏
- 哪些现象最像坏样例

而是：

1. 现场先看哪一个收口对象。
2. 看见什么必须立刻阻断 closeout。
3. 谁有权宣布 closed、handoff ready、monitor only 或 reopen required。
4. reopen 到底回到哪个对象边界，而不是回到哪几句说明。

这些都更接近：

- runtime closeout execution layer、值班顺序与长期运行语义

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复收口执行层已经完整”前，先问自己：

1. 我写的是对象级完成顺序，还是一套更细的完成表单。
2. 这些 verdict reason 能否被宿主、CI、评审与交接共享，而不是各自翻译一次。
3. 当前判断主体到底是谁，消费的到底是哪一个收口对象。
4. 当前阻断或 reopen 之后，回退的是正式对象边界，还是回到某种 UI 完成感。
