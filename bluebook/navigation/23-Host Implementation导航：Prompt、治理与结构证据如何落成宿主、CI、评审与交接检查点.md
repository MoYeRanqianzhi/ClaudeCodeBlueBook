# Host Implementation导航：Prompt、治理与结构证据如何落成宿主、CI、评审与交接检查点

这一篇不新增新的制度判断，而是回答一个更接近执行现场的问题：

- 当 shared evidence envelope 与 distortion casebook 都已经存在之后，怎样把它们继续压成真正可执行的 host implementation playbook，让宿主、CI、评审与交接都有明确检查点。

它主要回答四个问题：

1. 为什么 Evidence Envelope 反例层之后仍需要单独讨论 host implementation playbook 层。
2. 为什么 Prompt、治理与结构三条线都需要把“共享证据骨架”继续翻译成角色化检查点。
3. 怎样让宿主、CI、评审与交接围绕同一对象，但各自知道该检查什么。
4. 怎样用苏格拉底式追问避免把这层写成“更多 checklist”和“更多流程负担”。

## 1. Prompt host implementation playbook

适合在这些问题下阅读：

- 怎样让宿主、CI、评审与交接在 Prompt 线共同消费 compiled request truth、stable bytes、lawful forgetting ABI 与 handoff state。
- 怎样避免 Prompt 继续退回成“原文 prompt + 人工总结 + token 曲线”的拼贴。

稳定阅读顺序：

1. `22`
2. `../playbooks/14`

这条线的核心不是：

- 再解释 prompt 为什么强

而是：

- 把 Prompt shared envelope 真正落成可执行检查点

## 2. 治理 host implementation playbook

适合在这些问题下阅读：

- 怎样让宿主、CI、评审与交接在治理线共同消费 decision window、control arbitration、Context Usage 与 rollback boundary。
- 怎样避免治理继续退回成“只看 token”“只看审批次数”或“只看最终 allow/deny”。

稳定阅读顺序：

1. `22`
2. `../playbooks/15`

这条线的核心不是：

- 再强调统一定价秩序

而是：

- 把治理 envelope 翻译成统一硬门禁与软检查点

## 3. 结构 host implementation playbook

适合在这些问题下阅读：

- 怎样让宿主、CI、评审与交接在结构线共同消费 authority surface、recovery asset、anti-zombie gate 与 rollback object。
- 怎样避免结构升级继续退回文件级 diff、目录图和作者记忆。

稳定阅读顺序：

1. `22`
2. `../playbooks/16`

这条线的核心不是：

- 再赞美结构先进性

而是：

- 把结构 envelope 真正落成对象级 handoff gate 与 rollback gate

## 4. 一句话用法

如果：

- `22` 回答“最常见会怎样坏”

那么：

- `23` 回答“怎样把它重新落实成宿主、CI、评审与交接都能执行的统一检查点”

## 5. 苏格拉底式自检

在你准备宣布“团队已经能按同一套 envelope 落地”前，先问自己：

1. 每个角色现在看到的是同一对象，只是不同粒度，还是已经各自看回了不同材料。
2. 哪些检查点必须写成硬门禁，哪些只该保留为评审或交接提示。
3. 我是在降低未来判断分歧，还是在叠加更多流程摩擦。
4. 如果原作者离开，这套 playbook 还能不能独立支撑下一轮升级与回退。
