# 反例对照导航：坏解法、伪优化与改写路径

这一篇不新增新的制度判断，而是回答一个更贴近设计现场的问题：

- 当团队已经有自反审读问题后，怎样通过同题坏解与正确改写的对照，真正看懂 Claude Code 式设计为什么成立。

它主要回答四个问题：

1. 为什么自反审读层之后还必须继续长出反例对照层。
2. 为什么坏解法、伪优化与改写路径必须并排出现，而不能只给正解。
3. 怎样把 Prompt 魔力、安全定价与源码先进性分别压成可迁移对照样例。
4. 怎样用苏格拉底式追问避免把对照层写成“作者吐槽坏方案”。

## 1. Prompt 反例对照

如果问题是：

- 为什么“长文案更强”“多讲背景更稳”“多留一点历史更安全”这些直觉经常错。
- 为什么主语漂移、共享前缀分叉、边界错放与非法遗忘会一起打碎 Prompt 魔力。

建议顺序：

1. `../guides/30`
2. `../casebooks/10`

这条线的核心不是：

- 再总结一遍 Prompt 原理

而是：

- 看清最常见的坏写法是如何一步步把制度体退回文案堆

## 2. 安全与省 token 反例对照

如果问题是：

- 为什么“规则更多”“检查更多”“暴露更多信息”并不自动更安全。
- 为什么假统一预算器、全量可见面、无决策增益检查和不可撤销自动化会同时让系统更贵也更脆。

建议顺序：

1. `../guides/31`
2. `../casebooks/11`

这条线的核心不是：

- 再强调一次统一定价

而是：

- 看清免费扩张是怎样伪装成自由、便利与智能的

## 3. 源码先进性反例对照

如果问题是：

- 为什么“目录更碎”“registry 更全”“兼容层更多”不等于更先进。
- 为什么第二真相、transport 泄漏、恢复资产缺席与 zombie 温床常常伪装成“工程完整度更高”。

建议顺序：

1. `../guides/32`
2. `../casebooks/12`

这条线的核心不是：

- 再夸一次结构优雅

而是：

- 看清哪些看起来体面的重构，其实只是在制造下一轮结构债

## 4. 一句话用法

如果：

- `15` 回答“设计前该先反问自己什么”

那么：

- `16` 回答“同一个问题上，最常见的坏解法长什么样，以及 Claude Code 式改写为什么更稳”

## 5. 苏格拉底式自检

在你准备继续推广一个“看起来很合理”的设计前，先问自己：

1. 我现在看到的是正解，还是只是没见过它的坏解法。
2. 这个方案的失败样子是否已经被我正面写出来。
3. 我是在给团队可迁移的对照样例，还是只是在重复表扬 Claude Code。
4. 如果把作者名字拿掉，后来者还能否通过对照样例独立判断哪条路更稳。
