# 宿主修复协议导航：Prompt、治理与故障模型执行纠偏如何进入修复卡、拒收升级语义与重入规则面

这一篇不再回答“宿主验收执行失真该怎样纠偏”，而是回答：

- 当 Prompt、治理与结构的执行失真已经被压成固定纠偏顺序、hard reject 与模板改写骨架之后，宿主、SDK、CI、评审与交接系统到底该消费哪些正式修复对象、reject 升级语义与 rollback 重入规则，才能不重新退回经验修复。

它主要回答五个问题：

1. 为什么宿主验收纠偏层之后，蓝皮书仍需要单独讨论“宿主修复协议层”。
2. 为什么 Prompt 线如果不继续压成 repair object、reject escalation、rollback boundary 与 re-entry qualification 的正式修复协议，就会重新退回表单补丁与 reviewer 心法。
3. 为什么治理线如果不继续压成 authority repair、ledger rebuild、decision window reset、continuation repricing 与 rollback object 的正式修复协议，就会重新退回 mode 调参与审批补救。
4. 为什么结构线如果不继续压成 authority recovery、resume replay order、writeback restoration、anti-zombie verdict 与 boundary reset 的正式修复协议，就会重新退回 pointer 修修补补与重连碰运气。
5. 怎样用苏格拉底式追问避免把这层写成“更细的整改说明”。

## 1. Prompt 宿主修复协议线

适合在这些问题下阅读：

- 宿主、SDK、CI、评审与交接系统到底该围绕哪些正式对象消费 Prompt 执行纠偏，而不是继续围绕补过的表单与 reviewer 结论判断。
- 哪些 reject escalation 与 rollback boundary 一旦出现就说明 Prompt 世界已经重新长出第二真相。

稳定阅读顺序：

1. `../api/57`
2. `../guides/60`
3. `../casebooks/31`

这条线的核心不是：

- 再讲一次 Prompt 执行纠偏

而是：

- 把 Prompt 执行纠偏真正压成宿主可消费的修复卡、reject 升级语义与重入规则面

## 4. 为什么这层更适合落在 api

因为这一层最关键的问题已经不是：

- 先修什么
- 哪种执行幻觉在说谎

而是：

1. 哪些修复对象必须被宿主正式消费。
2. 哪些 escalation level 必须跨宿主、CI、评审与交接共享。
3. 哪些 rollback boundary 与 re-entry qualification 必须是正式规则，而不是建议项。
4. 哪些内部整改动作不应被当成公共 ABI。

这些都更接近：

- host-consumable repair contract、升级语义层与规则面层

## 5. 苏格拉底式自检

在你准备宣布“宿主修复协议已经完整”前，先问自己：

1. 我定义的是对象级 repair contract，还是一套更细的整改经验。
2. 这些 reject escalation 能否跨宿主、CI、评审与交接共享。
3. 当前消费者消费的是正式 repair object，还是某次临时修复留下的偶然细节。
4. 如果 later 内部重构发生，这套修复协议是否仍然成立。
