# 目录职责与入口协议

这一篇只回答三个入口问题：

1. 蓝皮书的规范入口是什么。
2. 为什么根目录存在若干“别名页/兼容入口页”。
3. 读者在主线、专题与研究记录之间该如何切换，而不把目录结构读乱。

## 1. 规范入口

当前蓝皮书的规范主线入口是：

1. `bluebook/README.md`
2. `bluebook/00-导读.md`
3. `bluebook/01-源码结构地图.md`
4. `bluebook/02-使用指南.md`
5. `bluebook/03-设计哲学.md`
6. `bluebook/04-公开能力与隐藏能力.md`
7. `bluebook/05-功能全景与API支持.md`
8. `bluebook/06-第一性原理与苏格拉底反思.md`
9. `bluebook/07-运行时契约、知识层与生态边界.md`
10. `bluebook/08-能力全集、公开度与成熟度矩阵.md`
11. `bluebook/09-三张控制面总图：世界进入模型、扩张定价与防过去写坏现在.md`

也就是说：

- `README.md` 是顶层入口
- `00-09` 是规范主线

更稳一点的高阶入口协议还应补一句：

- `09` 负责宪法总图
- `05` 负责模仿对象校正
- `15` 负责失稳前追问
- `41` 负责第一性原理收束
- 如果问题已经变成“这一层到底有没有权第一次回答”，本页就是 `speaking rights / appeal chain` 的唯一 charter，而不是再去各目录 README 拼目录政治。

如果只想从一套最短路径进入，不需要猜，也不需要碰运气。

## 2. 兼容别名页

当前根目录还有三类“兼容入口页”：

- `00-总览.md`
- `00-蓝皮书总览.md`
- `01-源码总地图.md`

这些页面保留的原因不是目录继续失控，而是：

- 兼容旧链接
- 兼容不同命名习惯下的入口搜索

但从目录治理角度，它们不再是规范主线。

更准确地说：

- 规范入口负责当前阅读路径
- 兼容别名页负责历史链接不失效

## 3. 各目录职责

### 3.1 `bluebook/`

根目录只承载：

- 主线结论
- 总判断标准
- 最短阅读路径

### 3.2 `navigation/`

导航目录只承载：

- 阅读地图
- 问题检索
- 目录治理与入口协议

其中高阶前门继续固定分工：

1. `05`
   - 模仿对象校正
2. `15`
   - 失稳前追问
3. `41`
   - 第一性原理收束

### 3.3 `architecture/`

承载：

- 运行时链路
- 状态机
- 装配链
- 恢复链
- 工程模式

### 3.4 `api/`

承载：

- 字段
- schema
- 控制协议
- 可接入性

### 3.5 `philosophy/`

承载：

- 第一性原理
- 设计内涵
- 演化方法

### 3.6 `guides/`

承载：

- 使用方法
- 接入策略
- 面向用户与宿主的实践路径

### 3.7 `playbooks/`

承载：

- 回归手册
- 事故复盘
- 演化演练
- 团队运行仪式

### 3.8 `casebooks/`

承载：

- 失败样本库
- 事故原型
- 结构反模式样本
- 具体案例复盘

它不承载整改待办或轮次级“改写路径”。

### 3.9 `security/`

承载：

- 安全控制面
- 权限与沙箱边界
- 恢复语义
- 工程化安全验证

### 3.10 `risk/`

承载：

- 风控
- 治理
- 账号/设备/策略

### 3.11 `docs/`

承载：

- 研究过程
- 证据索引
- 变更记录
- 目录治理记录

它不承载蓝皮书正文。
这里的 `docs/` 也始终指仓库根 `docs/`，不是 `bluebook/` 内部子目录；它是记忆层，不是正文层。

### 3.12 正文与研究记录分界

蓝皮书正文可以承载：

- 本篇回答什么、不再回答什么
- 概念边界、对象链与消费者顺序
- 第一性原理解释
- 苏格拉底式自检
- 稳定阅读顺序与跨目录跳转
- `09 / 05 / 15 / 41` 这种高阶前门分工

蓝皮书正文不应承载：

- 环境路径、版本标记与提交细节
- 轮次进度、候选优先级与待办排程
- 编号推进日志与临时过程记事

这些内容都应进入：

- `docs/README.md`
- `docs/development/long-term-memory.md`
- `docs/development/research-log.md`
- `docs/changelog.md`

### 3.14 发言权限与申诉链

如果继续把目录结构从“分层列表”压成“发言权限表”，更稳的写法也只该剩：

1. `philosophy/`
   - 判必要性：为什么必须这样设计。
2. `architecture/`
   - 证对象链：哪些 choke point / current-truth surface / writeback seam 真正在说话。
3. `api/`
   - 定承认面：宿主、schema 与 host-facing truth 到底正式承认了什么。
4. `guides/`
   - 出问题梯子与模板：怎样把判断压成可执行审读动作。
5. `playbooks/`
   - 出 verdict：准入、拒收、回退、闭环到底怎样被正式裁决。
6. `casebooks/`
   - 保留 distortion：哪些伪证、坏解与误判最常复活。
7. `navigation/`
   - 只路由与校正，不代言深页结论。
8. `docs/`
   - 只存记忆与变更，不重判正文。

如果某一层开始替另一层越权发言，仓库就会长出第二叙述者，later maintainer 也就不知道该在哪一层正式提出反对。

如果继续把申诉链压成 later maintainer 可直接执行的动作表，也只该剩：

1. 对“为什么必须这样设计”有异议
   - 回 `philosophy/`
2. 对对象链、主权面、writeback seam 或 choke point 有异议
   - 回 `architecture/`
3. 对宿主承认面、schema、consumer subset 或 host-facing truth 有异议
   - 回 `api/`
4. 对问题梯子、审读模板或落地动作有异议
   - 回 `guides/`
5. 对准入、拒收、回退、收口 verdict 有异议
   - 回 `playbooks/`
6. 对伪证、坏解、反例归档有异议
   - 回 `casebooks/`
7. 对入口路由、前门职责或兼容入口债有异议
   - 回 `navigation/`
8. 对轮次记忆、研究过程或变更台账有异议
   - 回仓库根 `docs/`

### 3.13 证据梯度分工

如果一个问题需要继续下钻，更稳的切换顺序不是“按主题找文件”，而是先问它属于哪一层证据职责：

1. 先验判断与不可约原则
   - 去 `philosophy/`
2. 机制对象、chokepoint 与当前真相保护
   - 去 `architecture/`
3. contract、schema 与 host-facing truth
   - 去 `api/`
4. 模板、审读清单与矩阵母版
   - 去 `guides/`
5. 准入、收口、回归与拒收 verdict
   - 去 `playbooks/`
6. distortion、伪证与误判样本
   - 去 `casebooks/`
7. judgment map 与跨目录反查
   - 去 `navigation/`
8. 研究过程、长时记忆与变更记录
   - 去 `docs/`

这条证据梯度一旦被打乱，目录结构就会重新混成：

- 总论页塞矩阵
- 导航页写正文
- 手册页写哲学
- 案例页写修复
- 记忆页抢正文

Atlas、template 与 verdict 的承载关系也应在这里固定下来：

1. atlas 不是第九层证据梯度，而是前五层证据压出来的编译地图。
2. atlas 模板放在 `guides/`，atlas 正文默认放在最接近真相暴露面的目录。
3. 对源码二级目录 atlas，默认正文宿主是 `api/46-50`，由 `navigation/35` 负责路由。
4. `playbooks/` 只消费 `atlas_ref` 绑定 verdict、rollback 与 reopen，不反向承担 atlas 生产职责。
5. `navigation/` 只声明这条分工，不替 atlas 正文或 verdict 正文重复展开。

同一对象如果继续往下拆，还要继续守住三层分工：

1. `architecture/`
   - 解释 runtime orchestration、状态机与 choke point
2. `api/`
   - 承认 contract definition、host-facing truth 与 consumer subset
3. `security / risk`
   - 分别解释 signer / ledger 机制前门，以及统一定价治理在用户侧的结算面

## 4. 目录切换协议

如果读者当前的问题是：

- “先理解 Claude Code 是什么”

就留在：

- `README.md + 00-09`

如果问题变成：

- “我是不是已经抄错对象”

就切到：

- `navigation/05`

如果问题变成：

- “第一条反证信号是什么、first reject path 在哪”

就切到：

- `navigation/15`

如果问题变成：

- “高阶结论是否已经压回第一性原理”

就切到：

- `navigation/41`

如果问题变成：

- “机制怎么实现”

就切到：

- `architecture/`

如果问题变成：

- “接口支不支持、字段是什么”

就切到：

- `api/`

如果问题变成：

- “为什么这样设计”

就切到：

- `philosophy/`

如果问题变成：

- “我应该怎么用”

就切到：

- `guides/`

如果问题变成：

- “我应该怎么运营、回归、复盘和演练”

就切到：

- `playbooks/`

如果问题变成：

- “安全控制面、权限边界、恢复语义怎样收口”

就切到：

- `security/`

如果问题变成：

- “真实失败长什么样、制度边界如何暴露”

就切到：

- `casebooks/`

如果问题变成：

- “我怎样把这些样本继续编目、标记并回流到演练体系”

就切到：

- `casebooks/04` 与 `playbooks/04`

如果问题变成：

- “风险、误伤、账号/设备约束与区域入口怎么理解”

就切到：

- `risk/`

如果问题变成：

- “研究过程、证据或变更记录该放哪里”

就切到：

- `docs/`

## 5. 一句话总结

蓝皮书当前的目录治理原则是：`README + 00-09` 是规范主线，`navigation/` 是检索层，`docs/` 是研究记录层，旧命名页只作为兼容别名保留，不再承担规范入口职责。
