# 目录职责与入口协议

这一篇只回答三个入口问题：

1. 蓝皮书的规范入口是什么。
2. 为什么根目录存在若干“别名页/兼容入口页”。
3. 读者在主线、专题与研究记录之间该如何切换，而不把目录结构读乱。

## 1. 规范入口

当前蓝皮书的规范主线入口是：

1. `bluebook/README.md`
2. `bluebook/00-导读.md`
3. `bluebook/01-源码结构地图.md`
4. `bluebook/02-使用指南.md`
5. `bluebook/03-设计哲学.md`
6. `bluebook/04-公开能力与隐藏能力.md`
7. `bluebook/05-功能全景与API支持.md`
8. `bluebook/06-第一性原理与苏格拉底反思.md`
9. `bluebook/07-运行时契约、知识层与生态边界.md`
10. `bluebook/08-能力全集、公开度与成熟度矩阵.md`
11. `bluebook/09-三张控制面总图：世界进入模型、扩张定价与防过去写坏现在.md`

也就是说：

- `README.md` 是顶层入口
- `00-09` 是规范主线

更稳一点的高阶入口协议还应补一句：

- `09` 负责宪法总图
- `05` 负责模仿对象校正
- `15` 负责失稳前追问
- `41` 负责第一性原理收束
- 如果问题已经变成“这一层到底有没有权第一次回答”，本页只负责把你路由到对应的 `README / guides / navigation` 节点，而不是再去各目录 README 拼目录政治。

如果只想从一套最短路径进入，不需要猜，也不需要碰运气。

## 2. 兼容别名页

当前根目录还有三类“兼容入口页”：

- `00-总览.md`
- `00-蓝皮书总览.md`
- `01-源码总地图.md`

这些页面保留的原因不是目录继续失控，而是：

- 兼容旧链接
- 兼容不同命名习惯下的入口搜索

但从目录治理角度，它们不再是规范主线。

更准确地说：

- 规范入口负责当前阅读路径
- 兼容别名页负责历史链接不失效

## 3. 各目录职责

### 3.1 `bluebook/`

根目录只承载：

- 主线结论
- 总判断标准
- 最短阅读路径

### 3.2 `navigation/`

导航目录只承载：

- 阅读地图
- 问题检索
- 目录治理与入口协议

其中高阶前门继续固定分工：

1. `05`
   - 模仿对象校正
2. `15`
   - 失稳前追问
3. `41`
   - 第一性原理收束

### 3.3 `architecture/`

承载：

- 运行时链路
- 状态机
- 装配链
- 恢复链
- 工程模式

### 3.4 `api/`

承载：

- 字段
- schema
- 控制协议
- 可接入性

### 3.5 `philosophy/`

承载：

- 第一性原理
- 设计内涵
- 演化方法

### 3.6 `guides/`

承载：

- 使用方法
- 接入策略
- 面向用户与宿主的实践路径

### 3.7 `playbooks/`

承载：

- 回归手册
- 事故复盘
- 演化演练
- 团队运行仪式

### 3.8 `casebooks/`

承载：

- 失败样本库
- 事故原型
- 结构反模式样本
- 具体案例复盘

它不承载整改待办或轮次级“改写路径”。

### 3.9 `security/`

承载：

- 安全控制面
- 权限与沙箱边界
- 恢复语义
- 工程化安全验证

### 3.10 `risk/`

承载：

- 风控
- 治理
- 账号/设备/策略

### 3.11 `docs/`

承载：

- 研究过程
- 证据索引
- 变更记录
- 目录治理记录

它不承载蓝皮书正文。
这里的 `docs/` 也始终指仓库根 `docs/`，不是 `bluebook/` 内部子目录；它是记忆层，不是正文层。

### 3.12 正文与研究记录分界

蓝皮书正文可以承载：

- 本篇回答什么、不再回答什么
- 概念边界、对象链与消费者顺序
- 第一性原理解释
- 苏格拉底式自检
- 稳定阅读顺序与跨目录跳转
- `09 / 05 / 15 / 41` 这种高阶前门分工

蓝皮书正文不应承载：

- 环境路径、版本标记与提交细节
- 轮次进度、候选优先级与待办排程
- 编号推进日志与临时过程记事

这些内容都应进入：

- `docs/README.md`
- `docs/development/long-term-memory.md`
- `docs/development/research-log.md`
- `docs/changelog.md`

### 3.13 证据梯度只做切层提示

如果一个问题需要继续下钻，本页只提示它该去哪个证据层，不再替深页展开正文：

1. 先验判断与不可约原则
   - 去 `philosophy/`
2. 机制对象、chokepoint 与当前真相保护
   - 去 `architecture/`
3. contract、schema 与 host-facing truth
   - 去 `api/`
4. 模板、审读清单与 canonical rung 降格规则
   - 去 `guides/`
5. 准入、收口、回归与拒收 verdict
   - 去 `playbooks/`
6. distortion、伪证与误判样本
   - 去 `casebooks/`
7. judgment map、projection crosswalk 与 route dispute
   - 去 `navigation/`
8. 研究过程、长时记忆与变更记录
   - 去仓库根 `docs/`

更稳一点说：

- `guides/102` 负责 canonical rung、candidate 降格与 gap note 模板。
- 各目录 `README` 负责各自边界，不替别的目录代签首答。
- `navigation/35` 负责 atlas projection 与 route dispute，不代签 atlas 正文。
- `navigation/04` 只提示“这类争议先去哪里”，不再预先指定某篇深页拥有目录宪法正文。

### 3.14 发言权限争议只做 next-hop

如果问题已经变成 `speaking rights / appeal chain` 争议，本页也只做路由，不再代签目录宪法正文：

1. 争“为什么必须这样设计”
   - 回 `philosophy/`
2. 争对象链、choke point、writeback seam 或 current-truth surface
   - 回 `architecture/`
3. 争 host-facing truth、schema 或 consumer subset
   - 回 `api/`
4. 争 canonical rung、降格规则或 gap note 模板
   - 回 `guides/102`
5. 争 verdict、rollback、reject 或 reopen 执行链
   - 回 `playbooks/`
6. 争 distortion、伪证或样本归档
   - 回 `casebooks/`
7. 争 repo-specific atlas、projection crosswalk 或跨目录 next-hop
   - 回对应目录 `README` 与 `navigation/35`
8. 争记忆、轮次过程或变更台账
   - 回仓库根 `docs/`

如果争议已经滑到“谁有权第一次回答”，先回对应目录 `README`；只有当问题确实是跨目录 next-hop 或 projection dispute 时，才继续停在 `navigation/`。

## 4. 目录切换协议

如果读者当前的问题是：

- “先理解 Claude Code 是什么”

就留在：

- `README.md + 00-09`

如果问题变成：

- “我是不是已经抄错对象”

就切到：

- `navigation/05`

如果问题变成：

- “第一条反证信号是什么、first reject path 在哪”

就切到：

- `navigation/15`

如果问题变成：

- “高阶结论是否已经压回第一性原理”

就切到：

- `navigation/41`

如果问题变成：

- “机制怎么实现”

就切到：

- `architecture/`

如果问题变成：

- “接口支不支持、字段是什么”

就切到：

- `api/`

如果问题变成：

- “为什么这样设计”

就切到：

- `philosophy/`

如果问题变成：

- “我应该怎么用”

就切到：

- `guides/`

如果问题变成：

- “我应该怎么运营、回归、复盘和演练”

就切到：

- `playbooks/`

如果问题变成：

- “安全控制面、权限边界、恢复语义怎样收口”

就切到：

- `security/`

如果问题变成：

- “真实失败长什么样、制度边界如何暴露”

就切到：

- `casebooks/`

如果问题变成：

- “我怎样把这些样本继续编目、标记并回流到演练体系”

就切到：

- `casebooks/04` 与 `playbooks/04`

如果问题变成：

- “风险、误伤、账号/设备约束与区域入口怎么理解”

就切到：

- `risk/`

如果问题变成：

- “研究过程、证据或变更记录该放哪里”

就切到：

- `docs/`

## 5. 一句话总结

蓝皮书当前的目录治理原则是：`README + 00-09` 是规范主线，`navigation/` 是检索层，`docs/` 是研究记录层，旧命名页只作为兼容别名保留，不再承担规范入口职责。
