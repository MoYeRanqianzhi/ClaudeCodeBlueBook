# 宿主修复纠偏导航：Prompt、治理与故障模型修复失真如何压回固定顺序、拒收升级路径与改写模板骨架

这一篇不再回答“宿主修复演练最常怎样失真”，而是回答：

- 当 repair drill 已经被证明会重新退回假修复、假回滚与假重入之后，团队到底该怎样把这些失真重新压回固定纠偏顺序、拒收升级路径与改写模板骨架，才能不让第二真相继续接管修复流程。

它主要回答五个问题：

1. 为什么宿主修复失真反例层之后，蓝皮书仍需要单独讨论“宿主修复纠偏层”。
2. 为什么 Prompt 线如果不继续把 repair object 伪绑定、摘要回滚与假重入压回固定顺序，就会重新把 Prompt 魔力退回事故叙事、摘要 handoff 与默认继续。
3. 为什么治理线如果不继续把 authority 假恢复、假窗口重置与免费重入压回固定顺序，就会重新把安全设计与省 token 设计拆成两套彼此脱钩的补救流。
4. 为什么结构线如果不继续把 breadcrumb 篡位、旁路写回与 anti-zombie 伪证明压回固定顺序，就会重新把源码先进性退回目录美学、日志繁荣与恢复成功率。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“把故障单流程再做细一点”。

- 哪些现象一旦出现就必须直接 hard reject，而不是继续观察。

1. `../guides/63`
2. `../casebooks/34`

## 4. 为什么这层更适合落在 guides

因为这一层最关键的问题已经不是：

- 哪些坏样例在说谎
- 哪些演练存在
- 哪些升级卡该怎样填写

而是：

1. 纠偏时先降级哪个假信号。
2. 再救回哪个正式对象。
3. 什么时候应直接 hard reject，而不是继续试错。
4. 模板骨架该如何改写，才能不再生产第二真相。

这些都更接近：

- correction guide、builder-facing 手册与制度化改写顺序

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复失真已经纠偏完成”前，先问自己：

1. 我救回的是单一真相，还是一套更严格的修复流程。
2. 我现在拒收的是对象边界漂移，还是只是在拒收观感不好。
3. 我现在回退的是正式对象边界，还是回到更熟悉的摘要、mode 或 pointer。
4. 我现在重入的是同一真相，还是只是系统又恢复响应。
5. 当前纠偏保护的是恢复能力本身，还是只是在重写一份更制度化的事故叙事。
