# Agent Runtime构建导航：第一性原理、宿主模板与统一蓝图

这一篇不再回答“Claude Code 源码里有什么”，而是回答一个更接近迁移与构建的问题：

- 当读者已经理解了协作语法、资源定价、恢复对象、runner 底盘与源码 atlas 之后，下一步就不该继续停在“理解 Claude Code”，而要知道怎样把这些原则编译成自己的 Agent Runtime。

它主要回答五个问题：

1. 为什么源码 atlas 层之后还必须继续长出 Agent Runtime 构建层。
2. 为什么真正成熟的构建，不是复刻功能，而是先固定对象、边界、预算、恢复与消费者。
3. 怎样把第一性原理构建法、宿主落地模板与统一蓝图分别压成可执行手册。
4. 为什么这层更适合落在 `guides/`，而不是继续扩写 `api/` 或 `architecture/`。
5. 怎样用苏格拉底式追问避免把“理解源码”误当“已经能构建同类系统”。

## 1. 第一性原理构建指南

如果问题是：

- 怎样从 Claude Code 里抽出一套可迁移的 Agent Runtime 构建顺序，而不是只复制某些功能或 prompt。
- 为什么 Claude Code 的 prompt 魔力、预算器、安全设计和恢复链，最后都要回到同一套构建顺序。

建议顺序：

1. `../guides/45`
2. `../guides/47`
3. `../philosophy/77`

这条线的核心不是：

- 再多写几段经验法则

而是：

- 先把对象、协作语法、资源定价、恢复闭环与消费者边界固定下来，再决定功能如何长出来

## 2. 宿主落地模板

如果问题是：

- 怎样把 Claude Code 的 control / event / snapshot / recovery / evidence envelope 编译成一个真正可接入的宿主模板。
- 为什么安全设计与省 token 设计，最后必须落成 host integration 的控制顺序，而不是落成更多策略文案。

建议顺序：

1. `../guides/46`
2. `../api/13`
3. `../api/20`
4. `../api/35`

这条线的核心不是：

- 再补一个 SDK 封装层

而是：

- 让 host 真的拥有 `control_request/control_response/control_cancel_request + worker_status/external_metadata + Context Usage + resume/recovery` 这一整套闭环

## 3. 统一蓝图手册

如果问题是：

- 怎样把 Prompt 语法、安全/省 token 设计与源码先进性合并成同一张 Agent Runtime 蓝图。
- 为什么 Claude Code 的高级感不在单点机制，而在这三条线被编译成同一套对象与边界治理。

建议顺序：

1. `../guides/47`
2. `../philosophy/77`

这条线的核心不是：

- 给三条线各写一张对照表

而是：

- 让协作语法、资源定价与可演化内核被理解成同一套设计单位的三个投影

## 4. 为什么这层落在 guides

因为这层要回答的核心问题不是：

- Claude Code 是什么

而是：

1. 我该先构建哪些对象。
2. 我该先给哪些面设权威入口。
3. 宿主该先接哪些协议，后接哪些辅助观测。
4. 我该怎样避免抄到半真相、投影层或 internal-only 设计。

这些都更接近：

- builder-facing 方法层

## 5. 一句话用法

如果：

- `35` 回答“源码二级目录应该怎样读”

那么：

- `36` 回答“理解完这些目录后，怎样把它们真正编译成自己的 Agent Runtime”

## 6. 苏格拉底式自检

在你准备宣布“我已经能照着做一个 Claude Code 风格系统了”前，先问自己：

1. 我固定的是对象与边界，还是只固定了 UI 和命令表面。
2. 宿主是否拥有正式闭环，还是只拥有答案输出流。
3. 我迁移的是协作语法、资源定价与恢复闭环，还是只迁移了某个单点功能。
4. 这套蓝图能否帮助后来者更快判断“哪里是真相，哪里只是投影”。
