# 迁移工单导航：从坏解法到制度改写的顺序、灰度与回退

这一篇不新增新的制度判断，而是回答一个更接近执行现场的问题：

- 当团队已经看清坏解法与 Claude Code 式正解的对照后，接下来到底该按什么顺序改、先灰度什么、什么情况下必须回退。

它主要回答四个问题：

1. 为什么反例对照层之后还必须继续长出迁移工单层。
2. 为什么迁移顺序、灰度策略与回退条件必须一起写，而不能只给“理想终态”。
3. 怎样把 Prompt、治理与源码先进性分别压成可执行迁移工单。
4. 怎样用苏格拉底式追问避免把迁移层写成“一次性大重构计划”。

## 1. Prompt 迁移工单

如果问题是：

- 怎样把旧式长文案 / 局部补丁 prompt 系统迁到 Prompt Constitution。
- 怎样在不打碎当前可用性的前提下，逐步固定主语、section、边界与合法遗忘。

建议顺序：

1. `../casebooks/10`
2. `../playbooks/06`

这条线的核心不是：

- 再写一次更强 prompt

而是：

- 把 prompt 从文本资产迁成制度资产

## 2. 治理与省 token 迁移工单

如果问题是：

- 怎样把旧式规则堆叠、全量可见面和假统一预算器迁到治理顺序系统。
- 怎样定义哪些检查先发生、哪些自动化必须回收、哪些 continuation 根本不该继续。

建议顺序：

1. `../casebooks/11`
2. `../playbooks/07`

这条线的核心不是：

- 再加一层规则

而是：

- 把免费扩张逐步收回到被定价的运行时秩序里

## 3. 源码先进性迁移工单

如果问题是：

- 怎样把旧式伪模块化、第二真相与 zombie 温床迁到 authoritative surface、recovery asset 与 anti-zombie 结构。
- 怎样避免“一次性拆分全仓”这种看似积极、实际高风险的伪迁移。

建议顺序：

1. `../casebooks/12`
2. `../playbooks/08`

这条线的核心不是：

- 立刻把目录改得更碎

而是：

- 先把真相、传输、恢复与未来修改路径重新收口

## 4. 一句话用法

如果：

- `16` 回答“哪些坏解法最常伪装成正解”

那么：

- `17` 回答“既然已经知道坏在哪，接下来该按什么顺序改，哪里先灰度，哪里必须保留回退”

## 5. 苏格拉底式自检

在你准备启动一次“制度升级”前，先问自己：

1. 我现在拥有的是对错判断，还是可执行迁移顺序。
2. 这次改写里，什么可以先 shadow，什么必须最后切。
3. 如果中途指标变坏，我知道该回退哪一层吗。
4. 我是在设计渐进改写路径，还是在设计一场没有回头路的重写。
