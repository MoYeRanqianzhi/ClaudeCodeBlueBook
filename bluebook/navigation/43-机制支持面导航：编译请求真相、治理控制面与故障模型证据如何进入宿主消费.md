# 机制支持面导航：编译请求真相、治理控制面与故障模型证据如何进入宿主消费

这一篇不再回答“这些机制该怎样实现”，而是回答：

- 当 Prompt 编译链、统一定价治理与故障模型编码已经被翻译成 builder-facing 手册之后，宿主、SDK、CI 与交接系统到底该消费哪些正式对象，才能不重新退回内部实现猜测。

它主要回答五个问题：

1. 为什么机制实现层之后，蓝皮书仍需要单独讨论“机制支持面层”。
2. 为什么 Prompt 线如果不能把编译请求真相压成宿主可消费支持面，就会重新退回原文 prompt 与黑箱稳定性。
3. 为什么治理线如果不能把 authority source、decision window 与 rollback object 压成宿主可消费支持面，就会重新退回 modal 与仪表盘崇拜。
4. 为什么结构线如果不能把 authority state、generation evidence、recovery boundary 与 anti-zombie 证据压成宿主可消费支持面，就会重新退回目录图与作者说明。
5. 怎样用苏格拉底式追问避免把这层写成“又一组字段表”。

## 1. 编译请求真相支持面线

适合在这些问题下阅读：

- 宿主该怎样消费 Prompt 编译链的正式结果，而不是直接依赖内部 prompt compiler。
- 哪些输入面、观测面与继续资格投影才是 Prompt 魔力的真正消费入口。

稳定阅读顺序：

1. `../api/49`
2. `../api/51`
3. `../guides/51`

这条线的核心不是：

- 再暴露更多内部 prompt 细节

而是：

- 把编译请求真相重新压成宿主真正应该写、应该看、绝不能直接绑定的分层支持面

## 2. 统一定价治理支持面线

适合在这些问题下阅读：

- 宿主该怎样消费 authority source、decision window、pending action、continuation gate 与 rollback object。
- 为什么治理控制面必须通过 control request、state writeback、Context Usage 与 rollback contract 共同被看见。

稳定阅读顺序：

1. `../api/50`
2. `../api/52`
3. `../guides/52`

这条线的核心不是：

- 再列一遍 control schema

而是：

- 把统一定价治理重新压成宿主、SDK、CI 与交接系统都能围绕同一对象判断的支持面

## 3. 故障模型证据支持面线

适合在这些问题下阅读：

- 宿主该怎样消费 authority state、generation evidence、recovery boundary 与 anti-zombie projection。
- 为什么结构先进性如果不能被外化成正式证据，后来维护者就仍只能依赖作者解释。

稳定阅读顺序：

1. `../api/53`
2. `../guides/53`
3. `../architecture/81`

这条线的核心不是：

- 再讲一次结构先进性

而是：

- 把故障模型重新压成宿主与后来维护者可以真正消费的状态面、回退面与恢复证据面

## 4. 为什么这层更适合落在 api

因为这一层最关键的问题已经不是：

- 原则是什么
- 实现顺序是什么

而是：

1. 哪些是宿主可写入口。
2. 哪些是宿主可读投影。
3. 哪些是 internal-only 机制，不该被绑成公共 ABI。
4. 当前消费者到底该围绕哪些对象组织自己的面板、CI、交接与复盘。

这些都更接近：

- 宿主可消费的支持面层

## 5. 苏格拉底式自检

在你准备宣布“宿主已经接好这些机制”前，先问自己：

1. 我消费的是正式支持面，还是内部实现偶然暴露出的细节。
2. 宿主看到的是对象级真相，还是字符串、颜色与日志替身。
3. later 内部重构发生时，宿主是否仍能继续正确工作。
4. 我是在暴露同一机制对象，还是在让不同消费者重新各说各话。
