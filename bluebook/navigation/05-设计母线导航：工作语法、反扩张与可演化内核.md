# 设计母线导航：Prompt 编译链、治理收费链与源码质量证据梯度

这一篇不新增新的机制判断，而是把 Claude Code 最近已经稳定下来的三条高阶判断压成可直接进入的阅读入口。

它主要回答三个问题：

1. 如果想理解 prompt 的效力来源，现在最稳的阅读入口是什么。
2. 如果想理解安全设计和省 token 设计为什么共享同一条治理收费链，应该从哪里进入。
3. 如果想真正学 Claude Code 的源码先进性，应该学结构本身，还是学它的演化方法。

## 0. 先问你是不是在抄错东西

在进入这三条判断前，先停一下，问自己准备模仿的到底是什么：

1. 如果你想抄的是一段更强的 system prompt，你已经把 Prompt 效力读浅了。
2. 如果你想抄的是更多 modal、allowlist 或 compact 技巧，你已经把安全与省 token 读浅了。
3. 如果你想抄的是更整齐的目录树或更多小文件，你已经把源码先进性读浅了。
4. 如果你想优化目录结构的方式只是再补一个 README 或更长的 route map，你已经把目录优化读浅了。
5. 如果你把 lawful forgetting 读成“更会总结”，把 classifier 读成“额外保险”，或把 `remote / session / transport` 读成“并列功能层”，你已经把三条母线都抄成了投影层。

这页真正要纠正的不是阅读顺序，而是模仿对象本身：Claude Code 最难学会的从来不是表面做法，而是三条判断各自保护什么不变量。
`05` 因而也只负责校正“别抄错东西”，不重发 Prompt / governance / source-quality owner 页已经拥有的首答公式、witness order 或 canonical ladder。

这里还应再固定一条 continuity 纪律：

- `continuity` 不是第四条母线，而是 Prompt `Continuation`、治理 `continuation pricing` 与源码质量 `recovery non-sovereignty / anti-zombie` 在时间轴上的共同交汇。
- 如果你现在还把 `/compact / /resume / /memory` 当成第四类设计对象，你抄到的就仍是按钮表，不是三条母线怎样处理“继续”。

## 0.2 第一刀先拒什么

如果想在进入长链前先做一次更狠的模仿对象校正，也只先拒三件事：

1. Prompt
   - 一旦 lawful forgetting 会改写 `reject / continue verdict`，或者 `card / verdict / prose` 越位成主语，你抄到的就不是 compiler，而是 narration。
2. 治理
   - 一旦 modal、token 面板或 classifier 被当成免费裁判，你抄到的就不是 pricing chain，而是治理 UI。
3. 源码质量
   - 一旦目录树观感、feature matrix 或 remote 模块分组先于 `current-truth surface / truth planes / later-maintainer reject path` 出现，你抄到的就不是结构治理，而是结构体感。

这三刀如果先拒不掉，后面的阅读顺序再对，也只是在更深的位置继续抄错东西。

如果还想先做一刀 continuity 去魅，也只先拒三件事：

1. 把 summary prose 当 continuity 主语。
2. 把 continuation pricing 当“默认免费续聊”的 UX 优化。
3. 把 recovery asset / replay transcript 当成 current truth signer。

## 0.1 并行深挖时不要按文件拆题

如果要并行多 Agent 深挖这三条判断，更稳的拆法也不是按文件或目录切块，而是先按第一问题拆；合并时也只回到同一组稳定 frontdoor，而不在 `05` 重新发明“该并回哪一页”的目录宪法。若争议已经上升到目录职责、入口升级或谁配第一次回答，统一回 `bluebook/README.md + 09 + docs/development/00-研究方法.md`。

如果继续把 continuity 的最小顺序再压成一行，它也只该剩：

- `continue qualification -> continuation pricing -> recovery non-sovereignty / anti-zombie`
- 这条顺序只配作为三条母线的时间轴，不配单独升级成第四套目录主语。

## 4. 三条判断如何互证

这三条判断不能拆开学。

1. 只学 Prompt 编译链，不学治理收费链，Prompt 效力就会退回免费扩张下的暂时效果。
2. 只学治理收费链，不学当前真相保护，`decision window / cleanup / continuation pricing` 就会失去可信宿主。
3. 只学结构优雅，不学请求装配与治理对象，源码先进性就会退回静态截图。

如果你准备继续写蓝皮书、迁移这套方法，或者拿别的 agent 系统对照它，先用同一组自校问题反复逼自己：

1. 我现在写的是对象链，还是结果词。
2. 我有没有点名同题坏解，而不只是在讲正解。
3. 我有没有写出第一条反证信号，而不只是在写高阶总结。
4. 我有没有给出 `shadow -> qualify -> cutover -> rollback` 的渐进改写顺序。

## 5. 如果把三条判断继续写成目录结构

更稳的目录优化，不该再理解为“补更多入口”，而应理解为“让同一条母线在不同目录只承担一种责任”。

这页因此不再重发目录发言权限表，只保留一条 frontdoor 纪律：

- 同一类第一问题只允许在一个稳定入口第一次被回答；若已经进入“哪一层配说这件事”的争议，统一回 `bluebook/README.md + 09 + docs/development/00-研究方法.md`，`05` 只继续校正“你是不是把文案、UI 或目录体感误抄成母线本体”。

目录一旦越权代言，仓库就会长出第二叙述者，later maintainer 也就不知道该在哪一层正式提出反对。

如果 `05 / 15 / 41` 重新变成长 route map，或者各目录 README 又开始同时兼做哲学解释、执行手册与记忆仓库，目录结构就会再次失去证据梯度。

更硬一点说，目录前门真正保护的不是排版纪律，而是 later maintainer 的正式反对位阶；一旦 frontdoor 页开始代签“谁配第一次回答”，局部可反对性就会立刻失效。

## 6. 一句话总纲

如果把 Claude Code 再压成一句最硬的总纲，它更像：

- 别抄表面做法；Prompt、治理与源码质量各自回 owner 页，真正要学的是它们各自保护什么不变量，以及为什么不能被免费改写。
