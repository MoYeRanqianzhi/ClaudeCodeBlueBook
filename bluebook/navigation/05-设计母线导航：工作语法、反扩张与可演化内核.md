# 设计母线导航：工作语法、反扩张与可演化内核

这一篇不新增新的机制判断，而是把 Claude Code 最近已经稳定下来的三条高阶母线压成可直接进入的阅读入口。

它主要回答三个问题：

1. 如果想理解 prompt 为什么会有“魔力”，现在最稳的阅读入口是什么。
2. 如果想理解安全设计和省 token 设计为什么是同一系统，应该从哪里进入。
3. 如果想真正学 Claude Code 的源码先进性，应该学结构本身，还是学它的演化方法。

## 1. 工作语法母线

如果问题是：

- 为什么 Claude Code 的 prompt 不是一段更好的文案。
- 为什么它更像上下文准入编译器、协议转译器和工作语法机。
- 为什么 protocol transcript、tool schema、delta attachments 都该放在同一条线上。

建议顺序：

1. `07-运行时契约、知识层与生态边界.md`
2. `architecture/28-提示词契约分层、知识注入与缓存稳定性.md`
3. `architecture/39-Prompt可重放前缀、可观测预算与Section编译器.md`
4. `architecture/53-稳定前缀、动态尾部与旁路Fork：Claude Code的Cache-Aware Prompt Assembly.md`
5. `architecture/54-从UI Transcript到Protocol Transcript：Prompt不是聊天记录的直接重放.md`
6. `architecture/61-工作语法机：Claude Code如何把软件工程世界编译成模型可行动的Protocol Surface.md`
7. `philosophy/18-Prompt不是文本技巧而是契约分层.md`
8. `philosophy/39-治理必须落到字节级确定性：上下文准入优于功能堆叠.md`
9. `philosophy/48-先把世界压缩成可治理语法，再让模型变聪明.md`
10. `architecture/64-语义压缩器：Claude Code如何把工作现场压成可继续行动的最小语义体.md`
11. `philosophy/51-真正强的Prompt不是信息更多，而是行动语义更密.md`

这条线的核心结论是：

- Claude Code 的 prompt 魔力来自 runtime 先把工作世界翻译成稳定语法，再把当前现场压成可继续行动的高密度语义

## 2. 反扩张母线

如果问题是：

- 为什么安全设计和省 token 设计应该一起看。
- 为什么 Claude Code 偏爱先裁剪、后暴露、再外置。
- 为什么 deferred、delta、tool result externalization、budget continuation 本质上同构。

建议顺序：

1. `architecture/21-消息塑形、输出外置与Token经济.md`
2. `architecture/37-统一预算器：能力裁剪、Token延续与状态外化.md`
3. `architecture/50-PolicySettings控制平面、Sandbox契约与三套预算器.md`
4. `architecture/51-安全即输入边界控制平面：Managed Authority、Trusted Sources与Runtime Boundary Compilation.md`
5. `architecture/56-能力可见性控制平面：Deferred、Delta与最小可见面.md`
6. `architecture/62-Narrow、Later、Outside：安全设计与省Token设计的统一反扩张运行时.md`
7. `philosophy/22-安全、成本与体验必须共用预算器.md`
8. `philosophy/38-安全、治理、Token与Prompt稳定性本质上是同一收口问题.md`
9. `philosophy/49-真正的省Token不是压缩文本，而是改变世界暴露给模型的方式.md`
10. `architecture/65-资源宪法：Claude Code如何统一分配能力、时间、注意力与权威.md`
11. `philosophy/52-模型不是资源的主人，Runtime才是.md`

这条线的核心结论是：

- Claude Code 真正在优化的不是 token 数本身，而是由 runtime 主权统一分配的能力、时间、注意力与权威

## 3. 可演化内核母线

如果问题是：

- Claude Code 的源码先进性到底该学什么。
- 为什么它不是“模块更多”就更强，而是“增长时仍能守住不变量”。
- 为什么 config、deps、state machine、leaf module 这些局部动作比目录树更值得看。

建议顺序：

1. `architecture/41-叶子模块、扼流点与循环依赖切断法.md`
2. `architecture/52-Chokepoint、Typed Decision、Authoritative Surface、Race-Aware Runtime与Contract-First：Claude Code源码先进性五法.md`
3. `architecture/58-让依赖图说真话：Leaf Module、Anti-Cycle Seam与Single-Source File.md`
4. `architecture/63-可演化内核：Claude Code如何在持续增长中维持不变量.md`
5. `philosophy/42-成熟架构不是没有大文件，而是不把复杂性撒满全仓.md`
6. `philosophy/45-先让依赖关系诚实，再让抽象显得优雅.md`
7. `philosophy/50-先进源码不是一开始就完美分层，而是增长时仍能守住不变量.md`
8. `architecture/66-演化制度设计：Claude Code如何在增长中保留重构可能性.md`
9. `philosophy/53-好架构不是更会重构，而是始终保留重构可能性.md`

这条线的核心结论是：

- Claude Code 值得学的不只是当前结构，而是它如何把增长风险不断收回到底盘，并持续保留下一次重构整套系统的可能性

## 4. 一句话总纲

如果把 Claude Code 再压成一句最硬的总纲，它更像：

- 先把工作世界写成可治理语法，再用反扩张运行时维持其稳定，最后用可演化内核让这套系统在增长中不失控
