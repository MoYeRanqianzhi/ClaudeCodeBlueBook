# 样本与演练导航：标签体系、交叉索引与记录模板

这一篇不新增新的制度判断，而是回答一个更偏“知识治理”的问题：

- 当团队已经有手册与样本之后，怎样避免它们继续长成不可检索的文档堆。

它主要回答四个问题：

1. 为什么样本层之后必须继续长出索引层。
2. 为什么标签体系、交叉索引与记录模板应一起设计，而不是各写一篇补丁。
3. Prompt、治理、源码演化三条线分别该用什么维度建索引。
4. 怎样用苏格拉底式追问避免把索引层写成又一轮目录美化。

## 1. 统一索引入口

如果问题是：

- 已经有 playbook 和 casebook，但团队不知道该用什么维度检索。
- 新增一次演练或事故后，不知道该回填到哪个案例。

建议顺序：

1. `playbooks/04`
2. `casebooks/04`
3. `navigation/12`

这层最重要的不是新案例，而是：

- 统一记录 ABI
- 统一标签字典
- 统一交叉索引

## 2. Prompt 索引维度

Prompt 样本最值得按下面五类标签组织：

- section
- boundary
- path parity
- lawful forgetting
- invalidation

因为这五类标签比“哪次改坏了”更接近 Prompt Constitution 的真实结构。

## 3. 治理索引维度

治理样本最值得按下面五类标签组织：

- order
- hard guard
- approval race
- stable bytes
- stop logic

因为这五类标签比“太慢 / 太贵 / 太松 / 太严”更接近治理系统的真实控制点。

## 4. 结构索引维度

结构演化样本最值得按下面五类标签组织：

- build surface
- shadow / stub
- transport shell
- recovery asset
- zombie risk

因为这五类标签比“哪个文件坏了”更接近源码先进性的真实边界。

## 5. 一句话用法

如果：

- `10` 回答“怎么运行”
- `11` 回答“坏会怎样”

那么：

- `12` 回答“怎样把运行和失败都编成可持续检索的知识结构”

## 6. 苏格拉底式自检

在你准备继续新增案例或手册前，先问自己：

1. 团队现在缺的是更多样本，还是缺更好的索引维度。
2. 这次新增内容能否被现有标签体系定位。
3. 这次演练记录能否无歧义回填到样本层。
4. 我是在优化知识检索，还是只是在继续堆目录。
