# 机制回灌导航：编译请求真相、治理控制面与可演化内核对象边界

这一篇不再回答“怎样照着 Claude Code 设计自己的 Runtime”，而是回答：

- 当 builder-facing 方法已经稳定之后，下一步该怎样把这些方法重新回灌成机制层，让对象边界、控制面与内核不变量直接落在源码结构里。

这里还应先钉死一条 speaking-rights 边界：本页只路由 mechanism owner，不代签当前真相 why 或源码质量 verdict；凡涉及 evidence ceiling、current-truth why、quality threshold 或全局 source-quality 结论，统一回 `guides/102 + philosophy/86 + philosophy/87 + architecture/README`。

它主要回答五个问题：

1. 为什么 `guides/48-50` 之后，蓝皮书仍需要单独讨论新的机制回灌层。
2. 为什么请求装配能力最终要落成 `compiled request truth` 与 `current-truth writeback`，而不是停在 Prompt Constitution。
3. 为什么安全与省 token 最终要落成 `governance control plane`，而不是停在治理方法。
4. 为什么源码先进性最终要落成 `current-truth surface` 与 `evolvable kernel object boundary`，而不是停在结构原则。
5. 怎样用苏格拉底式追问避免把机制层重新退回方法摘要层。

## 1. 编译请求真相线

适合在这些问题下阅读：

- 请求编译能力怎样继续落到源码机制层。
- 为什么 section law、stable prefix、protocol transcript 与 lawful forgetting 最终应该被理解成同一个请求对象。

稳定阅读顺序：

1. `../architecture/79`
2. `../architecture/73`
3. `../architecture/70`
4. `../api/37`
5. `../api/40`

这条线的核心不是：

- 再多讲一些 Prompt 写法

而是：

- 让模型、旁路 fork、宿主证据与恢复路径共同围绕同一份 `compiled request truth` 与同一条 `current-truth writeback` 工作

这条线的机制回灌顺序应固定成：

1. 先把 section law / stable prefix 编译成 `compiled request truth`。
2. 再固定谁拥有 `current-truth surface` 的写权限。
3. 再固定所有异步路径只能走 `current-truth writeback`，禁止旧快照直写。
4. 最后才允许 adapter、cache、summary 等消费投影出现。

## 2. 治理控制面线

适合在这些问题下阅读：

- 为什么安全设计、省 token 设计、宿主控制与继续语义最终是一张控制面。
- 为什么 `PolicySettings`、typed decision、deferred visibility、Context Usage 与 continuation window 必须一起理解。

稳定阅读顺序：

1. `../architecture/80`
2. `../architecture/74`
3. `../architecture/71`
4. `../api/28`
5. `../api/32`
6. `../api/35`

这条线的核心不是：

- 再加一个统一预算器

而是：

- 先固定谁能写边界、谁能宣布决定、谁能解释当前成本与继续条件，再让动作、能力、上下文与时间被统一定价

## 3. 可演化内核线

适合在这些问题下阅读：

- 为什么 Claude Code 的源码结构在增长中仍然能守住不变量。
- 为什么 `current-truth surface`、single-source、anti-cycle seam 与 `current-truth writeback` 最终应被理解成同一条可演化内核母线。

稳定阅读顺序：

1. `../architecture/81`
2. `../architecture/63`
3. `../architecture/58`
4. `../architecture/72`
5. `../api/39`
6. `../api/42`

这条线的核心不是：

- 再做一层结构抽象

而是：

- 让未来维护者继续拥有反驳半真相、拒绝陈旧写回、识别危险改动面的正式结构条件

## 4. 为什么这层要回到 architecture

因为这层要回答的核心问题已经不是：

- 我该怎么理解

而是：

1. 这些对象在源码里到底怎样被正式固定。
2. 哪些路径是 `current-truth surface`，哪些只是消费投影。
3. 哪些 drift、stale writer、cache break 与 permission race 会被机制层直接阻断。

这些问题都更接近：

- 运行时底盘的机制化解释

## 5. 联合回灌顺序

如果你想把上一层的三条方法线重新压回一组机制层对象，建议顺序固定成：

1. 固定请求对象真相：`../architecture/79`
2. 固定治理控制面真相：`../architecture/80`
3. 固定内核边界真相：`../architecture/81`

再按需要回跳：

1. 请求编译侧回 `../guides/48`
2. 治理侧回 `../guides/49`
3. 结构侧回 `../guides/50`

执行时不要换序。因为后两步都要消费前一步输出的当前真相对象：先有 `compiled request truth`，再有控制面定价与仲裁，最后才有可演化内核边界的长期维护责任。

## 6. 苏格拉底式自检

在你准备宣布“这些方法已经真正落到机制层”前，先问自己：

1. 请求编译这条线是否已经长成 `compiled request truth`，而不只是写作方法。
2. 治理这条线是否已经长成正式控制面，而不只是一些规则顺序。
3. 结构这条线是否已经长成 `current-truth surface` 与对象边界，而不只是结构美学。
4. 所有异步与恢复路径是否都经过 `current-truth writeback`，而不是各自补写局部真相。
5. 这些机制层对象是否还能被宿主、CI、评审与接手者消费成同一份当前真相。
