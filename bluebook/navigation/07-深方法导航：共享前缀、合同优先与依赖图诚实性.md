# 深方法导航：共享前缀、合同优先与依赖图诚实性

这一篇不新增新的正文判断，而是把最近已经稳定下来的三条“深方法”压成一个短入口。

它主要回答五个问题：

1. 如果已经接受 Claude Code 的高阶判断，下一步该怎么把这些判断转成操作方法。
2. 如果想从“如何做得更好”切入，而不是继续堆章节，应该优先进入哪条模板链。
3. 如果想做苏格拉底式自我反思，哪几组问题最能暴露自己是否仍在浅层理解 Claude Code。
4. 如果想在公开镜像条件下稳当地判断源码质量，该先分哪几层证据。
5. 如果想迁移到自己的 agent runtime，哪些内容该先变成模板，哪些内容仍只是解释。

## 1. 共享前缀方法线

适合在这些问题下阅读：

- 怎样把 prompt 真正设计成共享前缀资产网络，而不是很多相似小调用。
- 什么时候应该保存 snapshot，什么时候应该复用，什么时候应该 suppress。
- 为什么共享前缀不等于共享副作用。

稳定阅读顺序：

1. `philosophy/30-Prompt不是一次请求而是可复用前缀资产.md`
2. `architecture/42-辅助循环、侧问题与后回合Fork共享前缀.md`
3. `guides/18-如何把Prompt当成共享前缀资产网络：侧问题、suggestion、memory与summary共用主线程前缀.md`
4. `guides/21-共享前缀快照策略模板：何时保存、何时复用、何时suppress.md`

这条线的核心方法结论是：

- 主线程必须是共享前缀的唯一生产者，旁路循环默认只读前缀、只加窄差异、只隔离副作用

苏格拉底式自检：

1. 我设计的是共享前缀网络，还是很多“差不多知道现场”的平行 prompt。
2. 我是在保护 cache-safe ABI，还是只是希望功能“能跑就行”。
3. 没有可信 snapshot 时，我会 suppress，还是会默默制造世界模型漂移。

## 2. 合同优先方法线

适合在这些问题下阅读：

- 研究 Claude Code 源码时怎样减少脑补。
- 设计自己的 runtime 时怎样避免大文件先长出来、合同后补。
- 为什么必须先分清合同真相、registry 真相、权威真相入口与 adapter 子集。

稳定阅读顺序：

1. `architecture/38-Contract优先、运行时底盘与公开镜像缺口.md`
2. `api/34-单一真相入口、权威状态面与Chokepoint手册.md`
3. `guides/19-如何用Contract-First方法阅读和设计Agent Runtime：先找合同，再看热点文件.md`
4. `guides/22-Contract-First审读清单：如何系统读懂Agent Runtime的合同、权威面与热点文件.md`

这条线的核心方法结论是：

- 先看 schema / type union / state interface，再看 registry，再看 authoritative surface，最后才看热点 kernel

苏格拉底式自检：

1. 我现在看到的是协议全集、当前注册，还是某个 adapter 的线性子集。
2. 我已经找到权威真相入口了吗，还是还在从派生逻辑反推真相。
3. 我读热点文件时，是在看 invariant kernel，还是继续把实现噪音误当全局设计。

## 3. 依赖图诚实性方法线

适合在这些问题下阅读：

- 怎样把源码结构评审从“看起来整齐”推进到“长期不会撒谎”。
- 为什么 leaf module、anti-cycle seam、single-source file、风险命名和制度注释要放在一起看。
- 怎样把这套方法变成可执行 review checklist。

稳定阅读顺序：

1. `philosophy/45-先让依赖关系诚实，再让抽象显得优雅.md`
2. `architecture/58-让依赖图说真话：Leaf Module、Anti-Cycle Seam与Single-Source File.md`
3. `guides/20-如何工程化地维持依赖图诚实性：leaf module、anti-cycle seam与single-source file.md`
4. `guides/23-Dependency-Honesty Review Checklist：如何评审leaf module、anti-cycle seam与single-source file.md`

这条线的核心方法结论是：

- 允许少量不 DRY，只要它换来高扇入入口更薄、脏边更可控、共享真相更单一

苏格拉底式自检：

1. 这次抽象是在减少认知债，还是在偷偷拉深依赖图。
2. 这个共享入口是不是已经胖到不配继续做“公共层”。
3. 风险是否应该在命名和注释里提前暴露，而不是等事故后补文档。

## 4. 公开镜像证据分级方法线

适合在这些问题下阅读：

- 怎样在公开镜像条件下稳当地判断 Claude Code 的源码质量，而不是靠目录树、体量和作者光环脑补。
- 为什么必须分清 contract、registry、authoritative surface、adapter subset、hotspot kernel 与 mirror gap。
- 为什么“大文件”要区分成合法复杂度中心和散落式复杂性。

稳定阅读顺序：

1. `architecture/38-Contract优先、运行时底盘与公开镜像缺口.md`
2. `guides/19-如何用Contract-First方法阅读和设计Agent Runtime：先找合同，再看热点文件.md`
3. `guides/102-如何给公开镜像做源码质量证据分级：contract、registry、authoritative surface、adapter subset与hotspot gap discipline.md`
4. `philosophy/76-真正成熟的源码地图，不是目录列得更细，而是更快暴露权威入口、消费者子集与危险改动面.md`

这条线的核心方法结论是：

- 公开镜像里的源码质量，不该靠热点文件体感下判断，而应先做证据分级，再同时审 `dependency honesty` 与 `temporal honesty`

苏格拉底式自检：

1. 我现在看到的是 contract、registry、authoritative surface、adapter subset 还是 hotspot。
2. 这个大文件是在维护统一 invariant，还是在四处补丁式接线。
3. 我把哪些结论写成了证据，哪些只是脑补。

## 5. 如果你想继续把蓝皮书读深，而不是读散

更稳的顺序不是：

- 看到新文章就各读一篇

而是按三条方法线循环进入：

1. 先用 `07-运行时契约、知识层与生态边界.md` 与 `06-第一性原理与苏格拉底反思.md` 固定总问题。
2. 再按“共享前缀 / 合同优先 / 依赖图诚实性”选一条方法线深读。
3. 每读完一条，回到对应模板或 checklist，检验自己是否已经能迁移到别的 runtime。
4. 只有在模板层仍答不清时，才回到底盘层继续补机制。

## 5. 一句话总纲

如果把这一批深方法压成一句话，它更像：

- 先让主线程稳定生产共享前缀，再让阅读顺序服从正式合同，最后让依赖图与评审标准一起说真话
