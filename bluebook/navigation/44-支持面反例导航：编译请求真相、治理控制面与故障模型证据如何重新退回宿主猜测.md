# 支持面反例导航：message lineage、governance key 与 authority object 如何重新退回宿主猜测

这篇聚焦宿主误用本身。

- 核心问题不是支持面有哪些字段，而是宿主、SDK、CI 与交接系统明明已经拿到了 `message lineage / projection consumer / continuation qualification`、`governance key / externalized truth chain / decision window / continuation pricing` 与故障模型证据的正式支持面，为什么仍会重新退回字符串、仪表盘、成功率与口头说明。
- 读这篇时要抓住三件事：宿主最常拿哪种替身充当正式对象、哪种 consumer 绑定会重新制造第二真相、哪种误绑会在后续重构时最先拖垮宿主。

## 1. 编译请求真相宿主误用线

适合在这些问题下阅读：

- 为什么 Prompt 编译链明明已经外化成消息血缘、投影消费面、协议转录与继续资格投影，宿主仍会重新把它消费成一段 system prompt 字符串。
- 为什么 cache break、compact、continue qualification 明明都有正式投影，宿主仍会重新回到黑箱推断。

推荐阅读链：

1. `../casebooks/25`
2. `../api/51`
3. `../guides/51`

这条线的核心不是：

- 再解释一次 Prompt 编译链

而是：

- 解释宿主最常怎样把 Prompt 支持面重新消费成字符串、黑箱指标与“最后一条消息启发式”

## 2. 统一定价治理宿主误用线

适合在这些问题下阅读：

- 为什么治理支持面明明已经外化成 governance key、externalized truth chain、typed ask、decision window、continuation pricing 与 durable-vs-transient cleanup，宿主仍会重新消费成 mode、仪表盘与弹窗流程。
- 为什么治理对象明明已经存在，交接与 CI 仍然回到“当前看起来卡住了”。

推荐阅读链：

1. `../casebooks/26`
2. `../api/52`
3. `../guides/52`

这条线的核心不是：

- 再讲一遍治理控制面

而是：

- 解释宿主最常怎样把治理支持面重新退回 mode 投影、脱锚 ask/window prose、被仪表盘化的定价信号与文件级 cleanup 幻觉

## 3. 故障模型证据宿主误用线

适合在这些问题下阅读：

- 为什么结构故障模型明明已经外化成 authority object、freshness gate、recovery boundary 与 anti-zombie 结果面，宿主仍会重新退回目录图、恢复成功率与 pointer 崇拜。
- 为什么后来维护者明明拿到了恢复与回退 contract，依旧需要口头解释“现在到底算不算恢复好了”。

推荐阅读链：

1. `../casebooks/27`
2. `../api/53`
3. `../guides/53`

这条线的核心不是：

- 再夸一次故障模型编码

而是：

- 解释宿主最常怎样把 authority object 猜测、generation 误绑、recovery pointer 神化与 anti-zombie 缺席消费成新的第二真相

## 4. 这一层要抓住的宿主误绑

真正要命的不是字段没接全，而是 consumer 绑定错主语：

1. 宿主最常消费了哪些假信号。
2. 哪些“看起来接上了”的面板其实在重新制造第二真相。
3. 哪些宿主误绑会让后续 CI、交接与恢复继续说谎。
4. 后续重构发生时，哪些绑定会最先把宿主拖死。

## 5. 苏格拉底式自检

在你准备宣布“宿主已经把这些机制接好了”前，先问自己：

1. 我消费的是对象级支持面，还是内部实现偶然泄漏出来的细节。
2. 宿主看到的是同一真相，还是字符串、状态色、成功率与口头说明这些替身。
3. 后续内部重构发生时，我的宿主还会不会继续说真话。
4. 我是在消费正式对象，还是在重新发明自己的第二真相。
