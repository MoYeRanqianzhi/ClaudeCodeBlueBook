# 支持面反例导航：编译请求真相、治理控制面与故障模型证据如何重新退回宿主猜测

这一篇不再回答“这些机制支持面该怎样被宿主消费”，而是回答：

- 为什么宿主、SDK、CI 与交接系统明明已经拿到了 `compiled request truth`、`governance control plane object` 与故障模型证据的正式支持面，仍会重新退回字符串、仪表盘、成功率与作者说明。

它主要回答五个问题：

1. 为什么支持面层之后，蓝皮书还必须继续长出一层“支持面反例层”。
2. 为什么 Prompt 支持面如果被误消费成 `systemPrompt` 字符串、token 数字与最后一条消息，就会重新退回黑箱 prompt 崇拜。
3. 为什么治理支持面如果被误消费成 mode 名字、pending action 文案与 token 条，就会重新退回 modal 与默认继续。
4. 为什么故障模型支持面如果被误消费成恢复成功率、目录图与 pointer 文件，就会重新退回作者记忆与恢复玄学。
5. 怎样用苏格拉底式追问避免把这层写成“前端接错字段”。

## 1. 编译请求真相宿主误用线

如果问题是：

- 为什么 Prompt 编译链明明已经外化成输入面、section breakdown 与继续资格投影，宿主仍会重新把它消费成一段 system prompt 字符串。
- 为什么 cache break、compact、continue qualification 明明都有正式投影，宿主仍会重新回到黑箱推断。

建议顺序：

1. `../casebooks/25`
2. `../api/51`
3. `../guides/51`

这条线的核心不是：

- 再解释一次 Prompt 编译链

而是：

- 解释宿主最常怎样把 Prompt 支持面重新消费成字符串、黑箱指标与 last-message heuristic

## 2. 统一定价治理宿主误用线

如果问题是：

- 为什么治理支持面明明已经外化成 authority source、decision window、pending action、continuation gate 与 rollback object，宿主仍会重新消费成 mode、仪表盘与弹窗流程。
- 为什么治理对象明明已经存在，交接与 CI 仍然回到“当前看起来卡住了”。

建议顺序：

1. `../casebooks/26`
2. `../api/52`
3. `../guides/52`

这条线的核心不是：

- 再讲一遍治理控制面

而是：

- 解释宿主最常怎样把治理支持面重新退回 mode 投影、状态色、pending action 文案与文件级回退

## 3. 故障模型证据宿主误用线

如果问题是：

- 为什么结构故障模型明明已经外化成 authority state、recovery boundary 与 anti-zombie 结果面，宿主仍会重新退回目录图、恢复成功率与 pointer 崇拜。
- 为什么后来维护者明明拿到了恢复与回退 contract，依旧需要作者解释“现在到底算不算恢复好了”。

建议顺序：

1. `../casebooks/27`
2. `../api/53`
3. `../guides/53`

这条线的核心不是：

- 再夸一次故障模型编码

而是：

- 解释宿主最常怎样把 authority state 猜测、generation 误绑、recovery pointer 神化与 anti-zombie 缺席消费成新的第二真相

## 4. 为什么这层更适合落在 casebooks

因为这一层最关键的问题已经不是：

- 这些支持面有哪些字段
- 这些支持面该怎样接

而是：

1. 宿主最常消费了哪些假信号。
2. 哪些“看起来接上了”的面板其实重新制造了第二真相。
3. 哪些 consumer misuse 会让后续 CI、交接与恢复继续说谎。
4. later 重构发生时，哪些绑定会最先把宿主拖死。

这些都更接近：

- 支持面误用样本层和宿主漂移反例层

## 5. 苏格拉底式自检

在你准备宣布“宿主已经把这些机制接好了”前，先问自己：

1. 我消费的是对象级支持面，还是内部实现偶然泄漏出来的细节。
2. 宿主看到的是同一真相，还是字符串、状态色、成功率与作者说明这些替身。
3. later 内部重构发生时，我的宿主还会不会继续说真话。
4. 我是在消费正式对象，还是在重新发明自己的第二真相。
