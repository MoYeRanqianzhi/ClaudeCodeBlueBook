# 终局判断导航：协作语法、资源定价与未来维护者消费者

这一篇不新增新的机制判断，而是把最近已经稳定下来的三条终局收束压成一个直接可进入的导航入口。

它主要回答四个问题：

1. 如果不想再在章节海里来回跳，怎样最快进入 Claude Code 当前最硬的三条终局判断。
2. 如果想从第一性原理继续上推，57-59 该各自接回哪些机制层与实践层。
3. 如果想做自我反思，应该如何避免把 prompt、安全、源码先进性重新写浅。
4. 如果想把这套方法迁移到自己的 agent runtime，应先抄哪一层，而不是先抄目录。

- 为什么 UI transcript 与 protocol transcript 必须分离。
- 为什么 sticky prompt、suggestion、session memory 应被看成同一协作接口。

1. `architecture/54-从UI Transcript到Protocol Transcript：Prompt不是聊天记录的直接重放.md`
2. `architecture/61-工作语法机：Claude Code如何把软件工程世界编译成模型可行动的Protocol Surface.md`
3. `architecture/64-语义压缩器：Claude Code如何把工作现场压成可继续行动的最小语义体.md`
4. `architecture/67-协调成本控制面：Prompt如何同时组织模型推理与人类接手.md`
5. `guides/09-如何降低人类-模型协调成本：Sticky Prompt、Suggestion、Session Memory与接手连续性.md`
6. `guides/12-如何把Prompt当成人机协作接口：固定主语、反馈纠偏与低成本接手.md`
7. `philosophy/54-好的Prompt同时组织模型与人类接手路径.md`

- Claude Code 的 prompt 先组织协作语法，再组织模型续写

苏格拉底式自检：

3. 我描述的是聊天记录，还是可继续工作的 protocol transcript。
