# 苏格拉底审读导航：Prompt魔力、安全定价与源码先进性的自我校准

这一篇不新增新的制度判断，而是回答一个更难也更关键的问题：

- 当团队已经能诊断故障、查标签、看样本后，怎样在故障发生前就追问自己的设计是否真的成立。

它主要回答四个问题：

1. 为什么现场诊断层之后还必须继续长出苏格拉底审读层。
2. 为什么 Prompt 魔力、安全与省 token、源码先进性都需要自反式问题，而不只需要结论。
3. 怎样把“为什么强”继续压成“如果今天让我照着设计，我该先反问什么”。
4. 怎样用苏格拉底式追问避免把蓝皮书再次写回单向说教。

## 1. Prompt 魔力的自我校准

如果问题是：

- 为什么这套 prompt 在当前、下一步、接手后都像活在同一个现场里。
- 为什么一段 prompt 看起来更长，却未必更强。
- 为什么一个 Prompt Constitution 如果不能自我审读，最终会退回神秘文案。

建议顺序：

1. `../philosophy/63`
2. `../guides/30`

这条线的核心不是：

- 如何再多写几条 instruction

而是：

- 如何确认主语、共享前缀、section 宪法、合法遗忘与接手连续性都还成立

## 2. 安全与省 token 的自我校准

如果问题是：

- 为什么系统越“聪明”越需要限制免费扩张。
- 为什么安全与成本的深层共同点不是预算器名字，而是拒绝未定价输入。
- 为什么没有决策增益的检查，本身就是风险和浪费。

建议顺序：

1. `../philosophy/64`
2. `../guides/31`

这条线的核心不是：

- 怎样把规则叠得更多

而是：

- 如何确认动作、能力、上下文和时间都没有在免费扩张

## 3. 源码先进性的自我校准

如果问题是：

- 为什么有些仓库看起来更整齐，却更不适合长期演化。
- 为什么“源码先进性”最终一定要落实到权威面、合同、恢复资产与反 zombie 协议。
- 为什么未来维护者如果没有被当成正式消费者，先进性只会停留在作者视角。

建议顺序：

1. `../philosophy/65`
2. `../guides/32`

这条线的核心不是：

- 代码是不是更像教科书

而是：

- 结构里是否已经提前编码了批评路径、修改路径与退出路径

## 4. 一句话用法

如果：

- `14` 回答“出问题后怎样从现场观察回到制度根因”

那么：

- `15` 回答“在再次出问题前，怎样反问自己的设计是否早就埋下制度断裂点”

## 5. 苏格拉底式自检

在你准备继续新增一篇“为什么它很强”的分析前，先问自己：

1. 我写下的是结论，还是问题梯子。
2. 这些问题是否真的能帮助设计者自我校准，而不是只帮助读者点头。
3. 我是在解释 Claude Code 为什么强，还是在帮助团队避免用错它的思想。
4. 如果今天把这些问题拿去审一个新 Agent runtime，它们是否足够具体到能改设计。
