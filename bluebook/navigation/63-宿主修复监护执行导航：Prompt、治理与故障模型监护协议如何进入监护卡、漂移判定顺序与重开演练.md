# 宿主修复监护执行导航：Prompt、治理与故障模型监护协议如何进入监护卡、漂移判定顺序与重开演练

这一篇不再回答“宿主、SDK、CI、评审与交接系统到底该消费哪些正式监护对象”，而是回答：

- 当 watch window、authority watch 与 anti-zombie watch 已经被压成正式监护协议之后，团队到底该按什么固定执行顺序运行 watch card、宣布 stable under watch、冻结 handoff 或触发 reopen，才能不重新退回‘先观察一下’的口头治理。

它主要回答五个问题：

1. 为什么宿主修复监护协议层之后，蓝皮书还必须继续长出一层“宿主修复监护执行层”。
2. 为什么 Prompt 线如果不继续压成 watch card、drift verdict order 与 reopen drill，就会重新退回 closeout note、handoff 故事与临场判断。
3. 为什么治理线如果不继续压成 watch card、drift verdict order 与 reopen drill，就会重新退回 mode 面板、usage dashboard 与保守建议。
4. 为什么结构线如果不继续压成 watch card、drift verdict order 与 reopen drill，就会重新退回 pointer 健康感、监控转绿与作者口头说明。
5. 怎样用第一性原理与苏格拉底式追问避免把这层写成“更细的观察 checklist”。

## 1. Prompt 宿主修复监护执行线

如果问题是：

- 宿主、CI、评审与交接系统到底该按什么顺序运行 Prompt watch card，而不是只写下 `monitor_only` 或一句‘先盯着’。
- 哪些 drift verdict 一旦出现就必须冻结 handoff、拒绝继续观察并进入 reopen drill。

建议顺序：

1. `../api/63`
2. `../playbooks/44`
3. `../casebooks/37`
4. `../guides/66`

这条线的核心不是：

- 再讲一次 Prompt 宿主修复监护协议

而是：

- 把 Prompt 宿主修复监护协议真正压成值班、CI、评审与交接可以共享执行的 watch card、drift verdict order 与 reopen 剧本

## 2. 治理宿主修复监护执行线

如果问题是：

- 团队到底该按什么顺序运行治理 watch card，而不是继续围绕 mode 变化、token 图表与保守建议判断。
- 哪些 drift verdict 一旦出现就说明统一定价控制面已重新失配，必须冻结 capability expansion 并进入 reopen drill。

建议顺序：

1. `../api/64`
2. `../playbooks/45`
3. `../casebooks/38`
4. `../guides/67`

这条线的核心不是：

- 再讲一次治理宿主修复监护协议

而是：

- 把治理宿主修复监护协议真正压成值班、CI、评审与交接可以共享执行的 watch card、drift verdict order 与 reopen 剧本

## 3. 结构宿主修复监护执行线

如果问题是：

- 团队到底该按什么顺序运行结构 watch card，而不是继续围绕 pointer、重连、监控与作者口述判断。
- 哪些 drift verdict 一旦出现就说明结构真相面已经重新长出假连续性，必须冻结 handoff 并进入 reopen drill。

建议顺序：

1. `../api/65`
2. `../playbooks/46`
3. `../casebooks/39`
4. `../guides/68`

这条线的核心不是：

- 再讲一次结构宿主修复监护协议

而是：

- 把结构宿主修复监护协议真正压成值班、CI、评审与交接可以共享执行的 watch card、drift verdict order 与 reopen 剧本

## 4. 为什么这层更适合落在 playbooks

因为这一层最关键的问题已经不是：

- 该消费什么监护字段
- 该怎样纠偏
- 哪些现象最像坏样例

而是：

1. 现场先看哪一个监护对象。
2. 看见什么必须立刻冻结 handoff 或 capability expansion。
3. 谁有权宣布 stable under watch、handoff blocked 或 reopen required。
4. reopen 到底回到哪个对象边界，而不是回到哪几句观察建议。

这些都更接近：

- runtime watch execution layer、值班顺序与长期运行语义

## 5. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复监护执行层已经完整”前，先问自己：

1. 我写的是对象级监护顺序，还是一套更细的观察表单。
2. 这些 watch verdict 能否被宿主、CI、评审与交接共享，而不是各自翻译一次。
3. 当前判断主体到底是谁，消费的到底是哪一个监护对象。
4. 当前冻结或 reopen 之后，回退的是正式对象边界，还是回到某种 UI 健康感。
