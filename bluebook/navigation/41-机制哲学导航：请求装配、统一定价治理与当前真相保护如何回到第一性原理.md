# 机制哲学导航：世界如何合法进入模型、扩张如何先被定价、过去如何不得写坏现在

这一篇不再回答“这些机制对象怎样成立、怎样验证、怎样失真”，而是回答：

- 这一篇只回答三句不可约判断：世界如何合法进入模型，扩张如何先被定价，过去如何不得写坏现在。

更稳的校正方式不是停在旧对象门牌，而是先固定三条前门判断：世界如何合法进入模型，扩张如何先被定价，过去如何不得写坏现在。

1. 世界怎样进入模型
2. 扩张怎样进入当前世界
3. 过去怎样不得写坏现在

这些对象只是三条判断继续落成协议、对象和验证工件后的二级载体。

它主要压成五个不可约判断：

1. 为什么 Claude Code 的请求装配能通过 `same-world test`，答案不在文案，而在可缓存、可转写、可继续的编译链。
2. 为什么 `decision window` 与 `continuation pricing` 并不是两套权衡，而是治理控制面对一切扩张的统一定价。
3. 为什么 `current-truth protection` 并不是风格或洁癖，而是把故障模型提前编码进结构。
4. 为什么不能停在反例层，而必须继续把坏解法压回第一性原理收束。
5. 怎样用苏格拉底式追问避免把这一层写成新的口号集。

如果再往下压一层，这五个判断最终都要回到四个跨平面函数：

1. `compile`
   - 世界如何先被编进模型
2. `price`
   - 扩张如何先被定价
3. `externalize`
   - 当前真相如何先被 runtime 外化给 host
4. `evict`
   - stale writer、ghost capability 与旧 authority 如何被正式驱逐

如果一个解释还没有压到这四个函数，就还只是对象层总结，不算最硬的第一性原理收束。

这里还应再多记一句：

- `continuity` 不是第五个函数，也不是第四条母线；它只是 `compile / price / externalize / evict` 在时间维度上的联合验收。
- 更短地说，continuity 只配被写成：Prompt 保 `Continuation`，治理保 `continuation pricing`，源码质量保 `recovery non-sovereignty / anti-zombie`。

如果再往目录前门压一层，这一组哲学导航也只消费三条已承认的 shared formula，不在这里重写各目录的首答：

1. Prompt
   - 只认 `same-world compiler`；更细的 `first-reject order` 仍回 `philosophy/84`
2. 治理
   - `治理先外化当前真相，再批准当前扩张`
   - `Narrow / Later / Outside + stable bytes + working set`
3. 源码质量
   - `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline`

如果 later maintainer 第一次点进哲学前门时还看不见这三条，目录结构就还在迫使他先走对象层总结，再自己回压成第一性原理。

如果继续把 continuity 的最小哲学判断也压成一行，它也只该剩：

- `继续不只是还能发生，而是仍值得被编译、被定价、被安全写回。`

如果继续把目录结构优化也压成第一性原理层的一句判断，它最该先守住的不是“route 更短”，而是“第一次定义权唯一”：

- 前门只负责第一次定义三条母线
- 后续页只负责把它们落成对象、template、verdict、distortion 与 memory
- later page 若开始重写前门 nouns 或发明新的 route law，就已经在偷走第一次定义权

Prompt 与结构专题页因而都只配做专题侧门：前者承接 `continuation qualification / residual reopen / threshold liability`，后者承接 builder-facing template 与 later reject path；这篇不在这里库存化列出专题链路，它们也都不负责第一次定义 Prompt `same-world test`、统一定价治理或 `current-truth protection`。

## 1. 世界如何合法进入模型

先问：

- 为什么同一条 `message lineage -> section registry / stable prefix -> protocol transcript -> continue qualification verdict` witness chain 能同时约束当前、下一步与接手后的合法进入。
- 为什么一旦离开 `message lineage`、`stable prefix`、`protocol transcript`、`continue qualification verdict` 与 `cache-safe forks`，这条 witness chain 就会立刻失真。

成立证据：

1. `../philosophy/84`
2. `../architecture/82`
3. `../guides/99`
4. `../playbooks/77`

Anthropic 官方 engineering 还把这条线从 production 角度补硬了一层：prompt engineering 真正的杠杆是 collaboration framework、effort budget、tool heuristics、observability 与 tight feedback loops；这说明 `same-world compiler` 若只剩 nouns、没有 verify 与 delegation heuristics，就还没有压回 production first principles。

这条线的核心不是：

- 再停在 Prompt 显得有效这组结果词

而是：

- `41` 在 Prompt 线上只回答“为什么必须先有 same-world compiler contract”，不再重写 `84` owner chain 已拥有的公式、witness order 或实现顺序。
- 如果必须压成一句，只写：先确认谁定义世界、模型实际消费什么、同一身份如何延续，以及继续资格由谁裁定；更细的 `first-reject order` 与 object-level witness 继续回 `../philosophy/84`。
- continuity 在这条线上只配回答：`Continuation` 这一 rung 还是否沿同一 witness chain 成立；若仍停在旧对象名、桥接词或 tooling heuristics 层，先退回 `../09`、`../architecture/82` 与 owner page。

## 4. 为什么这层更适合落在 philosophy

因为这一层最关键的问题已经不是：

- 它怎么实现
- 它怎么回归
- 它怎么失真

而是：

1. 它为什么从一开始就必须这么设计。
2. 它保护的到底是哪一种不可约约束。
3. 如果拿掉具体实现，这些原则还能否独立成立。
4. 团队怎样把这些原则迁移到别的 agent runtime，而不是只会背 Claude Code 的做法。
