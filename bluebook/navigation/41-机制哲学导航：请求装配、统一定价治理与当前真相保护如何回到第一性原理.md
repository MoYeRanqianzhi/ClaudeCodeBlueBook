# 机制哲学导航：世界如何合法进入模型、扩张如何先被定价、过去如何不得写坏现在

这一篇不再回答“这些机制对象怎样成立、怎样验证、怎样失真”，而只回答一件事：

- 为什么 Claude Code 必须先回答世界如何合法进入模型、扩张如何先被定价、过去如何不得写坏现在。

更稳的校正方式不是停在旧对象门牌，而是先反问：这三个判断如果不先成立，后面的协议、对象、模板、排错和目录分工为什么都会失真。

这里还应再多记一句：

- `continuity` 不是第四条母线；它只是这三条判断在时间维度上的共同验收。
- 若一段解释必须靠第二套公式库、函数名或目录法才能成立，说明它还没压回第一性原理，只是在重写 owner page。

如果继续把目录结构优化也压成第一性原理层的一句判断，它最该先守住的不是“route 更短”，而是“第一次定义权唯一”：

- 前门只负责第一次定义三条母线
- 后续页只负责把它们落成对象、template、verdict、distortion 与 memory
- later page 若开始重写前门 nouns 或发明新的 route law，就已经在偷走第一次定义权

Prompt 与结构专题页因而都只配做专题侧门：前者承接 `continuation qualification / residual reopen / threshold liability`，后者承接 builder-facing template 与 later reject path；这篇不在这里库存化列出专题链路，它们也都不负责第一次定义 Prompt `same-world test`、统一定价治理或 `current-truth protection`。

## 1. 世界如何合法进入模型

先问：

- 为什么 Prompt 的效力不是措辞技巧，而是世界定义权、消费边界与继续资格仍被同一条 same-world contract 约束。
- 为什么一旦 later maintainer 需要重述现场而不是消费同一 contract，same-world property 就会立刻退回文案效果。

成立证据：

1. `../philosophy/84`
2. `../architecture/82`
3. `../guides/99`
4. `../playbooks/77`

Anthropic 官方 engineering 还把这条线从 production 角度补硬了一层；本页因此只保留一条验收问句：`same-world compiler` 若真成立，它是否同时让 verify、delegate、tool choice 与 feedback loop 共享同一现场；更完整的 why 统一回 `../philosophy/84`，不在本页重造 object chain。

这条线的核心不是：

- 再停在 Prompt 显得有效这组结果词

而是：

- `41` 在 Prompt 线上只回答“为什么必须先有 same-world compiler contract”，不再重写 `84` owner chain 已拥有的公式、witness order 或实现顺序。
- 如果必须压成一句，只写：先确认谁定义世界、模型实际消费什么、同一身份如何延续，以及继续资格由谁裁定；更细 owner fact 继续回 `../philosophy/84`。
- continuity 在这条线上只配回答：继续之后是否仍是同一现场；若仍停在旧对象名、桥接词或 tooling heuristics 层，直接回 owner page。

## 2. 统一定价治理哲学线

先问：

- 为什么安全设计与省 token 设计看似不同，却总是落到同一治理秩序。
- 为什么没有决策增益的检查，本身就构成免费扩张。

成立证据：

1. `../philosophy/85`
2. `../architecture/83`
3. `../guides/100`
4. `../playbooks/78`

这条线的核心不是：

- 再讲一遍治理控制面字段

而是：

- 把治理压回“拒绝一切免费扩张”的第一性原理
- 把“先裁 visible world，再谈动作许可”写成治理第一句，而不是让 mode、modal、token 面板抢先代表治理
- continuity 在这条线上只配回答：这次继续是否仍值得付费；更细 chain、readback 与 failure semantics 统一回治理 owner chain

## 3. 源码质量与可演化内核哲学线

先问：

- 为什么 Claude Code 的源码先进性不像目录美学，而像一套可证、可批评、可继续重构的复杂度治理协议。
- 为什么真正重要的不是“文件怎么排”，而是“谁配宣布现在、谁只能消费现在、谁只能帮助恢复现在”。

成立证据：

1. `../philosophy/76`
2. `../guides/102`
3. `../philosophy/86`
4. `../philosophy/87`
5. `../architecture/84`
6. `../playbooks/79`

这条线的核心不是：

- 再总结一遍结构优点

而是：

- 先确认谁配写现在，再让 ladder、writeback doctrine 与 failure honesty 一起保护 `one writable present`
- 把源码先进性压回 current-truth protection、时间诚实与 future maintainer rejectability，再谈模块美学
- 如果解释已经开始重发 ladder、对象链或 route law，说明它已经离开哲学收束层，回到了 `102 / architecture/README`
- continuity 在这条线上只配回答：恢复资产、旧写入者与 replay 路径有没有借“继续”之名重新篡位 present truth

## 4. 为什么这层更适合落在 philosophy

因为这一层最关键的问题已经不是：

- 它怎么实现
- 它怎么回归
- 它怎么失真

而是：

1. 它为什么从一开始就必须这么设计。
2. 它保护的到底是哪一种不可约约束。
3. 如果拿掉具体实现，这些原则还能否独立成立。
4. 团队怎样把这些原则迁移到别的 agent runtime，而不是只会背 Claude Code 的做法。

这些问题更接近：

- 第一性原理层和哲学收束层

## 5. 如果把第一性原理继续写成目录结构

更稳的目录结构，不该再按主题库存或写作先后分层，而应按 speaking-right ledger 分层：

- 前门先回答不可约判断
- 后续页再把这些判断落成对象、contract、template、verdict、distortion 与 memory
- 第一定义权必须唯一；later page 只负责实例化，不重新命名三条母线

这不是文档排版问题，而是同一条第一性原理在目录层继续外化后的权限分工。

如果继续写得更好，还要再问三件事：

1. later maintainer 只看目录前门、不点深页时，是否已经能先用对这三组主语。
2. 后续目录入口是否只承接 reopen、template、verdict 与 distortion 这类 question domain，而不重新发明顶层 route map。
3. 目录前门是否已经把第一次定义权写成唯一 speaking right，而不是把目录结构本身写成 ontology。

## 6. 苏格拉底式自检

在你准备宣布“一套制度解释已经足够深”前，先问自己：

1. 删掉“Prompt 很强”的措辞后，我是否仍能解释 why。
2. 删掉 mode、token 面板与审批界面后，我是否仍能解释 why。
3. 删掉目录树观感与作者说明后，我是否仍能解释 why。
4. 我是在描述 Claude Code 的做法，还是已经提炼出可迁移的第一性原理。
5. 我有没有把 owner page 的公式库、对象链或 route law 偷运进这篇哲学导航。
