# 机制哲学导航：世界如何合法进入模型、扩张如何先被定价、过去如何不得写坏现在

这一篇不再回答“这些机制对象怎样成立、怎样验证、怎样失真”，而只回答一件事：

- 为什么 Claude Code 必须先回答世界如何合法进入模型、扩张如何先被定价、过去如何不得写坏现在。

更稳的校正方式不是停在旧对象门牌，而是先反问：这三个判断如果不先成立，后面的协议、对象、模板、排错和目录分工为什么都会失真。

这里还应再多记一句：

- `continuity` 不是第四条母线；它只是这三条判断在时间维度上的共同验收。
- 若一段解释必须靠第二套公式库、函数名或目录法才能成立，说明它还没压回第一性原理，只是在重写 owner page。

Prompt、治理与结构专题页都只配承接 deeper mechanism；这篇只保留 why，不在这里继续讲 later page、专题链路或编辑规则。

## 1. 世界如何合法进入模型

先问：

- 为什么世界定义权、消费边界与继续资格必须仍被同一条世界准入 contract 约束。
- 为什么一旦 later maintainer 需要重述现场而不是消费同一 contract，世界准入能力就会立刻退回文案效果。
- 为什么晚绑定只有在填充预授权槽位、而不改写 `Authority / Boundary / Transcript / Lineage / Continuation` 时才合法；一旦改写其中任何一项，就已进入 renegotiation。

更完整的 owner fact、object chain、审读动作与执行收口统一回 `../philosophy/84 -> ../architecture/82 -> ../guides/99 -> ../playbooks/77`；本页只保留一个 why 问句：若世界定义权不是单源，为什么 `verify / delegate / tool choice / handoff` 会一起退回现场重谈。

这条线的核心不是：

- 再停在 Prompt 显得有效这组结果词

而是：

- `41` 在 Prompt 线上只回答“为什么世界定义权必须单源”，不再重写 `84` owner chain 已拥有的公式、witness order 或实现顺序。
- 如果必须压成一句，只写：谁配把当前世界合法编译进模型、谁只配消费这份编译结果、谁在继续时重新获得资格；更细 owner fact 继续回 `../philosophy/84`。
- continuity 在这条线上只配回答：继续之后是否仍是同一现场；若仍停在旧对象名、桥接词或 tooling heuristics 层，直接回 owner page。

## 2. 统一定价治理哲学线

先问：

- 为什么安全设计与省 token 设计看似不同，却总是落到同一治理秩序。
- 为什么没有决策增益的检查，本身就构成免费扩张。

更完整的 why、对象链、审读动作与执行收口统一回 `../philosophy/85 -> ../architecture/83 -> ../guides/100 -> ../playbooks/78`；本页只保留一个 why 问句：为什么未被重新定价的继续会同时延长 authority 与成本在场。

治理线也只认一条 bridge theorem：任何晚绑定增量只要改写了 `Authority / Boundary / Transcript / Lineage / Continuation`，或继续扩大了可见世界与 lease duration，就不再是“继续消费当前世界”，而是 renegotiation，必须重新开价。

这条线的核心不是：

- 再讲一遍治理控制面字段

而是：

- 把治理压回“拒绝一切免费扩张”的第一性原理
- 把“先裁 visible world，再谈动作许可”写成治理第一句，而不是让 mode、modal、token 面板抢先代表治理
- continuity 在这条线上只配回答：这次继续是否仍值得付费；更细 chain、readback 与 failure semantics 统一回治理 owner chain
- 两种最常见 counterfeit 是：投影词先代表治理、继续入口默认续租 authority
- 第一条 reject signal 也只该问：暴露已发生时，你还能否立刻点出谁重新定价、谁记账、谁负责结算

## 3. 源码质量与可演化内核哲学线

先问：

- 为什么 Claude Code 的源码先进性不像目录美学，而像一套可证、可批评、可继续重构的复杂度治理协议。
- 为什么真正重要的不是“文件怎么排”，而是“谁配宣布现在、谁只能消费现在、谁只能帮助恢复现在”。

更完整的 why、evidence ladder、对象层与执行样本统一回 `../philosophy/76 -> ../guides/102 -> ../philosophy/87 -> ../architecture/README -> ../playbooks/79`；本页只保留一个 why 问句：为什么 `one writable present` 必须先成立，模块美学才有资格出现。

这条线的核心不是：

- 再总结一遍结构优点

而是：

- 先确认谁配写现在，再让 ladder、writeback doctrine 与 failure honesty 一起保护 `one writable present`
- 把源码先进性压回 current-truth protection、时间诚实与 future maintainer rejectability，再谈模块美学
- 如果解释已经开始重发 ladder、对象链或编排规则，说明它已经离开哲学收束层，回到了 `102 / architecture/README`
- continuity 在这条线上只配回答：恢复资产、旧写入者与 replay 路径有没有借“继续”之名重新篡位 present truth
- 两种最常见 counterfeit 是：恢复资产越权成 present writer、写不出 `change-risk record` 却仍宣称看清了边界
- 第一条 reject signal 也只该问：later maintainer 能否立刻说出哪条对象越权，以及第一退回层先落哪

## 4. 为什么这层更适合落在 philosophy

因为这一层最关键的问题已经不是：

- 它怎么实现
- 它怎么回归
- 它怎么失真

而是：

1. 它为什么从一开始就必须这么设计。
2. 它保护的到底是哪一种不可约约束。
3. 如果拿掉具体实现，这些原则还能否独立成立。
4. 团队怎样把这些原则迁移到别的 agent runtime，而不是只会背 Claude Code 的做法。

这些问题更接近：

- 第一性原理层和哲学收束层
