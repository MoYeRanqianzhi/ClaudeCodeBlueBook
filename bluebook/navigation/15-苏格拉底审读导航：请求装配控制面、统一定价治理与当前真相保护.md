# 苏格拉底审读导航：请求装配控制面、统一定价治理与当前真相保护

这篇只做一件事：把苏格拉底审读压成失稳前问题，而不是故障后诊断。

- 核心问题不是“事后怎么解释”，而是“在再次出问题前，怎样反问这套制度是否已经埋下断裂点”。
- 请求装配、统一定价治理与当前真相保护在这里都只是 route shorthand；真正要追问的始终是 `same-world test`、`decision window` 与 `present-truth protection` 有没有在前门失真。

这里还应再多记一句：

- `continuity` 不是第四类审读对象；它只是这三类问题在时间维度上的共同失稳面。
- 所以继续相关问题先问 `continue qualification`、再问 `continuation pricing`、最后问 `recovery asset` 是否越权 current truth，而不是先按 `/compact / /resume / /memory` 这些命令名拆题。

## 1. 世界是否仍按同一 same-world order 合法进入模型

关键追问：

- 为什么 `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability` 这组 `first-reject order` 必须同时守住当前、下一步与接手后的合法进入，而不能被误写成对象级 witness chain。
- 为什么 instruction 更长，却未必更能通过 `same-world test`。
- 为什么一旦 later maintainer 分不清 `first-reject order` 与 canonical witness order，维护者就会把 Prompt 误写成文案效果。
- 为什么 later consumer 若还要先遍历“哪些动作可能成立”，甚至把已排除分支重新拉回候选集重判，而不是只在少数已授权分支里做选择，这条 `first-reject order` 就已经开始失效。

凡是 continue 相关审读，第一条 `first-reject` 追问都只问：当前是不是某个更易读的 `projection / summary / delegated context` 正在越权代签 `continue qualification`；如果是，就先停在 `same-world failure verdict`，不要直接跳去补 witness 细节或恢复技巧。

规范锚点：

1. `../philosophy/84`
2. `../architecture/82`
3. `../guides/99`
4. `../playbooks/77`

这条线的核心不是：

- 如何再多写几条 instruction

而是：

- `15` 只保留 Prompt 线的失稳前问题梯子：究竟是谁在定义世界、模型实际消费什么、同一身份如何延续，以及继续资格由谁裁定。
- 目录前门在这里仍只复用 `Authority -> Boundary -> Transcript -> Lineage -> Continuation -> Explainability` 这组 `first-reject order`；对象级 canonical witness order 继续只回 `philosophy/84`。
- 第一条反证信号也只先追三类：`systemPrompt` 截图或最后一条消息篡位成主语、display transcript 越级冒充 protocol truth、`compact / handoff / steady` 之后只剩 story 不剩 qualification。
- 这三类反证其实也只是一条信号的三种外观：某个更易读、更顺手的 surface 正在越权代签 `same-world / continue verdict`；第一动作一律不是补说明，而是先把它降回 `projection / evidence`。
- Prompt 线的第一退回层也只保留一条：先把越权 surface 降回 `projection`，再停在 `same-world failure verdict`；若这条退回层说不清，所谓 `continue` 就只是带病续命。
- 如果你已经开始点名 `message_lineage_ref / section_registry_ref / stable_prefix_ref / protocol_transcript_ref / continuation_object_ref / continue_qualification_verdict`，说明你缺的已不是问题梯子，而是归属页。

## 2. 统一治理定价的苏格拉底审读

关键追问：

- 为什么系统越“聪明”越需要限制免费扩张。
- 为什么安全与成本的深层共同点不是预算器名字，而是拒绝未定价输入。
- 为什么没有决策增益的检查，本身就是风险和浪费。

规范锚点：

1. `../philosophy/85`
2. `../architecture/83`
3. `../guides/100`
4. `../guides/31`

这条线的核心不是：

- 怎样把规则叠得更多

而是：

- 系统有没有先把扩张是否值得被定价说清，再允许动作、上下文与时间进入现场
- 第一条反证信号常常不是 token 升高，而是 mode 面板、审批弹窗、usage 条、default continue 或 compact 技巧又开始被当成治理真相
- 更硬的前置追问还应是：系统有没有先裁 `visible world`，还是先让过宽工具、schema 与上下文免费进入，再用动作审批补票
- 如果 classifier、审批或默认重试自己没有被定价，它们就在把治理控制面重新膨胀成新的免费扩张源
- 如果你已经开始点 `pricing-right / truth-surface / asset-rollback ABI / reopen qualification`，说明你缺的已不是问题梯子，而是归属页

这一刀区分也要先钉死：mode、审批、usage、default continue 与 compact 都只是投影或 consumer，不是治理真相。

如果排查已经进入 failure order、shared reject verdict 或 reopen qualification，就立刻离开 `15`，先回 `10`；若追 why-proof 再回 `85 / 61`，若已转成 signer / cleanup authority ambiguity 再回 `security/README`。`15` 只负责暴露第一条反证，不负责发布治理故障顺序。

## 3. 当前真相保护的苏格拉底审读

关键追问：

- 为什么有些仓库看起来更整齐，却更不适合长期演化。
- 为什么“源码先进性”最终一定要落实到 current-truth surface、recovery asset non-sovereignty、freshness gate 与 ghost capability。
- 为什么未来维护者如果没有被当成正式消费者，先进性只会停留在作者视角。

规范锚点：

1. `../philosophy/86`
2. `../architecture/84`
3. `../guides/101`
4. `../guides/102`
5. `../philosophy/76`
6. `../guides/32`

这条线的核心不是：

- 代码是不是更像教科书

而是：

- 结构里是否已经提前编码了 `current-truth surface / recovery asset non-sovereignty / freshness gate / ghost capability` 这条 present-truth 纪律，以及证据梯度、批评路径、修改路径与退出路径
- `truth ladder / orchestration shell / danger surface / atlas` 这些词只配做 projection / crosswalk，不再单独充当 present-truth frontdoor
- 第一条反证信号常常不是 bug 变多，而是 replay、pointer、snapshot 或作者说明又开始篡位成 `current-truth surface`
- 如果 `remote / session / transport` 仍被讲成功能域而不是主权域，later maintainer 就还看不见 ownership、reconnect、permission 与 exit contract 实际在保护什么
- 如果 `event truth / current truth / display truth` 重新混写，宿主与 UI 就会再次靠 transcript replay 猜现在，而不是消费正式 writeback surface
- `later maintainer rejectability` 的最低标准也不是“看懂作者设计”，而是拿不到作者时仍能只凭结构指出哪条 `event / snapshot / transport` 越权成了 authority，并立刻说出第一退回层
- continuity 的源码审读也不该先问“恢复成功了吗”，而应先问旧 summary、pointer、snapshot 与 replay transcript 有没有借恢复路径篡位成 present truth

如果排查已经开始给 `shim / snapshot / pointer / authority / seam / recovery / eviction` 排 later-maintainer reject 顺序，就先离开 `15` 回 `guides/102`；若追 why-proof 再回 `86 / 87`，若缺对象镜像再回 `architecture/README`。

## 4. 三条线共同的先验反证

在你继续扩写“更深解释”之前，先反问三件事：

1. 如果删掉漂亮措辞，这条 `world entry / request assembly / six-stage assembly chain` 还剩哪条合法世界准入链。
2. 如果删掉面板与预算图，这套治理还剩哪条拒绝免费扩张的价格链。
3. 如果删掉目录美学，这套结构还剩哪条当前真相保护链。
4. 如果删掉 `/compact / /resume / /memory` 这些命令名，这套系统还剩哪条 `continue qualification -> continuation pricing -> recovery non-sovereignty` 的时间轴。

只有这三问还能成立，后面的分析才不是新一层口号。

在你准备把这三条线分给多个 agent 同时研究前，先反问四件事：

3. 哪个分支在回答前门，哪个分支在回答对象层、模板层、verdict 层或反例层，是否已经混层。

## 7. 苏格拉底式自检

在你准备继续新增一篇“为什么它很强”的分析前，先问自己：

1. 我写下的是结论，还是问题梯子。
2. 这些问题是否真的能帮助设计者在失稳前改设计，而不是只帮助读者点头。
3. 我是在解释 Claude Code 为什么强，还是在帮助团队避免用错它的思想。
4. 如果今天把这些问题拿去审一个新 Agent runtime，它们是否足够具体到能改设计。
5. 如果并行多 Agent 同时研究，这些问题是否足够让各自答案在同一前门合流，而不是再产生四套新目录词。
6. 我有没有把归属页的 failure order 讲成了本页的制度顺序，而不是只保留失稳前问题。
7. 如果一段“为什么 Prompt 很强”的分析还需要先列命令名、artifact 名或加载来源，才能解释它为什么有效，却答不出“它替 later consumer 删掉了哪一段冷启动动作搜索”，那它讲的仍是配置库存，不是失稳前问题。
