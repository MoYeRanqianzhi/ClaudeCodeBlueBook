# 运营与复盘导航：修宪回归、治理事故与演化演练

这一篇不新增新的制度判断，而是回答一个更后置但更关键的问题：

- 当 `guides/27-29` 已经把第二序制度压成模板层后，团队该怎样继续把它们变成长期运营、事故复盘、回归与演练。

它主要回答四个问题：

1. 为什么强制度最终必须进入运营层，而不只是停在模板层。
2. 为什么 `playbooks/` 应单独成目录，而不是继续塞回 `guides/`。
3. prompt、安全与源码先进性分别该怎样进入回归、复盘与演练。
4. 怎样用第一性原理和苏格拉底式追问防止制度设计重新退回口头经验。

## 1. Prompt 修宪回归线

如果问题是：

- 一次 prompt 修宪之后，怎样确认 section、boundary、lawful forgetting 与 invalidation 仍然合法。
- 出现 prompt 失稳时，怎样按制度字节而不是按主观感觉复盘。

建议顺序：

1. `philosophy/60`
2. `architecture/73`
3. `guides/27`
4. `playbooks/01`

这条线的核心产物应是：

- prompt regression ritual
- constitution drift taxonomy
- invalidation review
- prompt incident postmortem

## 2. 治理事故运营线

如果问题是：

- approval race、auto mode 回收、stable bytes 漂移、stop-logic 失效该怎样进入运营与事故处理。
- 为什么安全与省 token 不是只靠矩阵，而要靠持续回归。

建议顺序：

1. `philosophy/61`
2. `architecture/74`
3. `guides/28`
4. `playbooks/02`

这条线的核心产物应是：

- governance incident taxonomy
- approval race postmortem
- automation revoke drill
- stable-bytes drift audit
- decision-gain regression panel

## 3. 源码演化演练线

如果问题是：

- shadow / stub / compat 该如何退出，而不是越积越厚。
- recovery asset、transport shell、anti-zombie protocol 怎样变成正式演练对象。

建议顺序：

1. `philosophy/62`
2. `architecture/75`
3. `guides/29`
4. `playbooks/03`

这条线的核心产物应是：

- build-surface drill
- shadow-stub exit ledger
- recovery drill
- anti-zombie postmortem
- structure anti-pattern library

## 4. 目录职责判断

如果 `09` 回答的是：

- 团队该有哪些模板与审读工件

那么 `10` 回答的是：

- 团队该何时运行这些模板、怎样复盘事故、怎样验证制度仍然成立

所以最稳的目录切分是：

- `guides/`：方法、模板、设计动作
- `playbooks/`：运营、回归、复盘、演练

## 5. 苏格拉底式自检

在你准备继续补一篇“更深分析”前，先问自己：

1. 团队现在缺的是新解释，还是缺回归与复盘手册。
2. 这个制度如果今天出事故，是否已经有正式 playbook。
3. 这个目录新增是否让职责更清楚，还是又把方法层与运营层混回一起。
4. 我是在帮助读者理解，还是在帮助团队持续正确地运行。
