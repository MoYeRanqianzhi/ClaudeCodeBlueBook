# Rollout样例导航：Diff、评审问题卡、灰度结果与回退记录

这一篇不新增新的制度判断，而是回答一个更接近实施尾端的问题：

- 当团队已经有迁移工单之后，怎样用完整 rollout 样例证明自己真的在按制度升级，而不是只在纸面上改写。

它主要回答四个问题：

1. 为什么迁移工单层之后仍需要单独讨论 rollout 样例层。
2. 为什么改写前后 diff、评审问题卡、灰度结果与回退记录必须一起出现。
3. 怎样把 Prompt、治理与源码先进性分别压成一套完整 rollout 证据样例。
4. 怎样用苏格拉底式追问避免把 rollout 样例写成“事后美化的成功故事”。

## 1. Prompt rollout 样例

适合在这些问题下阅读：

- 怎样证明一套旧式长文案系统正在被渐进迁到 Prompt Constitution。
- 怎样记录 section 拆分、边界切换、共享前缀收敛与 handoff 改善的真实过程。

稳定阅读顺序：

1. `../playbooks/06`
2. `../playbooks/09`

这条线的核心不是：

- 宣布 prompt 改好了

而是：

- 留下能复查的迁移证据

## 2. 治理与省 token rollout 样例

适合在这些问题下阅读：

- 怎样证明系统不是只加了规则，而是真的把免费扩张收回到了统一定价秩序里。
- 怎样记录顺序调整、stable bytes 冻结、decision gain stop-logic 与自动化回收的灰度结果。

稳定阅读顺序：

1. `../playbooks/07`
2. `../playbooks/10`

这条线的核心不是：

- 成本下降了多少

而是：

- 哪些制度动作带来了更稳定、更可解释的成本结构

## 3. 结构 rollout 样例

适合在这些问题下阅读：

- 怎样证明第二真相、transport 泄漏与 zombie 温床真的被渐进收口，而不是只是目录变漂亮。
- 怎样记录 authoritative surface、recovery asset 与 anti-zombie 规则的切换证据。

稳定阅读顺序：

1. `../playbooks/08`
2. `../playbooks/11`

这条线的核心不是：

- 结构更优雅了

而是：

- 结构是否已经留下可回退、可复盘、可复查的升级轨迹

## 4. 一句话用法

如果：

- `17` 回答“知道坏在哪以后，该按什么顺序改”

那么：

- `18` 回答“改了之后，哪些样例足以证明你不是在自我感觉良好”

## 5. 苏格拉底式自检

在你准备宣布一次迁移“已经成功”前，先问自己：

1. 我现在拥有的是理想终态描述，还是完整 rollout 证据。
2. 这次灰度里最关键的 diff、指标和回退记录是否已经留下。
3. 我是在记录真实演进，还是在事后编排一个好看的故事。
4. 如果半年后重跑这次迁移，后来者能否仅靠样例文档复现同样的判断过程。
