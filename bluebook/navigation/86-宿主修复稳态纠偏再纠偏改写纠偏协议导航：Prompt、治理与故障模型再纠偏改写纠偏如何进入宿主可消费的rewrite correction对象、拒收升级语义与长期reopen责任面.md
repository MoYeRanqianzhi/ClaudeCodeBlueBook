## 4. 为什么这层更适合落在 api

因为这一层最关键的问题已经不是：

- 先冻结什么假信号
- 再救回什么对象
- 什么时候应直接硬拒收

而是：

1. 哪些对象必须被四类消费者共同消费，才能保证 later 团队不再回到作者记忆。
2. 哪些 verdict 语义必须被四类消费者共同理解，才能保证 reject / reentry / reopen 不再各说各话。
3. 哪些 explainability / prose / projection 应该明确降级，不再被绑定成公共 ABI。
4. 哪些 reopen 条件必须被长期保留，才能让未来正式反对当前状态的能力继续存在。

这些都更接近：

- host-consumable protocol、shared object surface、shared reject semantics 与 reopen liability surface

## 6. 第一性原理与苏格拉底式自检

在你准备宣布“宿主修复稳态纠偏再纠偏改写纠偏协议已经完整”前，先问自己：

1. 我写下的是同一个共享对象，还是四份不同的解释模板。
2. 我共享的是正式 reject 语义，还是不同角色各自理解的工作习惯。
3. 我保留的是未来仍能推翻当前状态的正式 `reopen` 能力，还是一句“有问题再回来看看”。
4. later 团队接手时依赖的是对象、边界与责任，还是仍要回到 dashboard、pointer 与作者记忆。
5. 当前 protocol 保护的是无人继续盯防时的真相延续，还是只是在把 correction guide 重新格式化成字段表。
