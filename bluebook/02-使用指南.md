# 使用指南

本章不是根据营销文案，而是根据源码反推“怎样用 Claude Code 才最吃到它的真实能力”。
使用层也继续继承 `问题分型 -> 工作对象 -> 控制面 -> 入口`：先判这次失真的还是不是同一工作对象，再判控制面，最后才选命令、技能、工具、子代理或工作流入口。

## 0. 先带着三条判断再用 Claude Code

如果你只想记住最值钱的使用前提，先记住三句：

1. Prompt 不是更长文案，而是先过 `same-world test`：`world-definition source -> message lineage -> protocol transcript -> stable bytes -> continuation qualification` 仍在同一个世界里成立。
2. 安全与省 token 不是两套系统，而是先承认 `治理先外化当前真相，再批准当前扩张`；`Context Usage` 只是 `decision window` 的诚实投影。
3. 源码质量不是目录更漂亮，而是先认 `contract -> registry -> current-truth surface -> consumer subset -> hotspot kernel -> mirror gap discipline` 这条 canonical ladder 与对应 `failure semantics`。

这里还应再多记一句：

- `continuity` 在使用层也不是第四类技巧专题；它只是 `same-world test`、`continuation pricing` 与 `stable bytes externalize + reload law` 在时间轴上的同题展开。

这三句的权威短语统一以 `09-三张控制面总图：世界进入模型、扩张定价与防过去写坏现在.md` 为准；本章只从使用层复用它们，不再另起变体。
源码质量线的术语也统一以 `guides/102` 的 canonical rung 为准；atlas / projection 别名只在地图页做 locator，不进入使用层首答句。

所以高质量使用 Claude Code，不只是“会下 prompt”，而是先判断：

1. 这次任务最先在保护哪个世界对象。
2. 哪些扩张值得现在支付。
3. 哪些旁路、恢复或自动化可能偷偷制造第二真相。

如果继续把使用层压成最短动作句，只剩三条：

1. 先让任务通过 `same-world test`，再进入模型。
2. 先判断当前 `decision window` 还值不值得继续付费，再决定要不要扩张。
3. 先把 stable bytes 留在场外，只把 working set 留在场内。

如果这三问还回答不清，就不要急着继续看使用层技巧：

1. 先回 `09`
   - 定义你正在处理哪张控制面
2. 再回 `navigation/05`
   - 校正你到底在模仿什么
3. 再回 `navigation/15`
   - 先找第一条反证信号与 first reject path
4. 最后回 `navigation/41`
   - 确认高阶判断是否已经压回第一性原理

如果一个使用建议还答不上“它究竟在保护哪个工作对象、哪条控制面、哪个入口只是 consumer”，就说明它仍停在结果词层，而不是第一性原理层。

## 1. 先选对运行模式

### 1.1 交互式 REPL: 适合探索、实现、验证连续工作流

如果你的任务需要:

- 反复修改代码
- 看工具进度
- 临时切权限模式
- 继续同一个 session
- 启动 subagent / remote session / MCP

优先使用交互式 REPL。`launchRepl()` 只是很薄的一层，但整个 `main.tsx` 的大多数优化、setup screens、hooks、MCP warm-up、plugin init 都围绕它构建。

证据:

- `claude-code-source-code/src/replLauncher.tsx:12-21`
- `claude-code-source-code/src/main.tsx:2239-2607`

### 1.2 `--print` / headless: 适合自动化与流水线

如果你的目标是:

- 单轮执行
- 机器消费结果
- SDK / CI / pipeline 集成

优先 headless 模式。源码里 headless 路径显式支持:

- stream-json 输出
- structured output
- session persistence 开关
- headless MCP connect
- trust bypass 后的完整 env 应用

证据:

- `claude-code-source-code/src/main.tsx:1847-1896`
- `claude-code-source-code/src/main.tsx:2557-2619`
- `claude-code-source-code/src/QueryEngine.ts:529-551`

结论:

- 交互式 REPL 是“协作面”。
- headless 是“自动化面”。
- 两者共享底层 QueryEngine / query loop，而不是两套割裂系统。
- `Context Usage` 在这两条路径里都不该先被读成成本面板，而应先被读成这次 `decision window` 是否仍值得支付的诚实投影。
- 两条路径的 first reject signal 也一致：一旦 mode 条、token 百分比或长摘要开始冒充当前真相，就先回 runtime 已外化的 world / window / state projections。

## 2. 不要把 slash command、skill、tool 混为一谈

源码显示 Claude Code 的命令面分三层:

1. slash command: 用户显式输入 `/...`
2. skill: prompt 包，可被用户触发，也可被模型调用
3. tool: 结构化能力原语

### 2.1 什么该用 slash command

适合:

- 切换状态
- 本地 UI 操作
- 触发一段预封装 prompt

因为 `processUserInput()` 允许本地 slash command 在不触发 query 的情况下直接完成。

证据:

- `claude-code-source-code/src/utils/processUserInput/processUserInput.ts:153-176`

### 2.2 什么该用 skill

适合:

- 需要一段高质量操作手册式 prompt
- 希望顺带授予额外工具权限
- 希望挂 hooks / model / effort / path activation

skill 不是“快捷别名”，而是带 frontmatter 的 prompt artifact。调用 skill 时，系统会注入 `command_permissions` 附件，把 skill 声明的额外 allowed tools 一起带进去。

证据:

- `claude-code-source-code/src/utils/processUserInput/processSlashCommand.tsx:869-920`
- `claude-code-source-code/src/skills/loadSkillsDir.ts:185-261`

### 2.3 什么该直接交给 tool

适合:

- 文件、shell、MCP、subagent、task 这类结构化动作
- 需要明确 schema、权限、渲染与恢复语义的能力

tool 是 Claude Code 真正的执行原语。

## 3. 接受 ToolSearch 这个设计，不要假设模型一开始就看见全部工具

源码里很多工具都带 `shouldDefer: true`，而 `ToolSearch` 的职责就是按需拉取 deferred tools 的完整 schema。MCP tools 默认也会延迟加载，除非显式 `alwaysLoad`。

证据:

- `claude-code-source-code/src/tools/ToolSearchTool/prompt.ts:53-108`
- `claude-code-source-code/src/tools/ToolSearchTool/ToolSearchTool.ts:21-44`
- `claude-code-source-code/src/tools.ts:247-249`

实践建议:

- 不要要求模型记住所有冷门工具名。
- 给任务意图，让它先搜，再选，再调用。
- 如果做自动化 prompt 设计，主动允许“先 ToolSearch 再行动”的两跳过程。

这也是 `Narrow / Later / Outside` 的使用层版本：

- 先缩小当下真正要裁决的 working set
- 再晚绑定冷门能力细节
- 再把不该常驻的 schema 和大对象留在场外按需拉取

## 4. 权限策略不要图省事开太大

源码明确把以下规则视为危险权限:

- `Bash(*)`
- PowerShell 的宽泛解释器或执行器模式
- Agent 的宽泛放行

原因不是“保守”，而是这些规则会绕过 auto mode classifier 对危险执行的评估。

证据:

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:84-147`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:149-233`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:235-285`

实践建议:

- 尽量给细粒度 allow rule，而不是工具级通配。
- 对 shell 能力尤其如此。
- 如果你想要自动化，先把 task 拆得更明确，而不是先把权限全开。

## 5. 学会用 subagent，但不要乱并发

源码中的多 Agent 设计非常明确:

- 读任务可并行
- 写任务应按文件集合串行
- coordinator 负责综合，不把理解工作外包给 worker

这不只是经验，而是直接写进 `coordinatorMode.ts` 的系统 prompt。

证据:

- `claude-code-source-code/src/coordinator/coordinatorMode.ts:198-237`
- `claude-code-source-code/src/tools/AgentTool/AgentTool.tsx:261-316`

实践建议:

- 研究阶段: 同时起多个 agent 去扫不同方向。
- 实现阶段: 同一组文件只保留一个主写入者。
- 验证阶段: 独立 agent 做测试与 review，避免“自己改自己验”。

## 6. 技能系统的最佳打开方式

源码里 bundled skills 有几个很有代表性:

- `verify`: 把“运行起来验证”固化为技能
- `remember`: 整理 CLAUDE.md / CLAUDE.local.md / auto-memory
- `stuck`: 诊断卡死 session

证据:

- `claude-code-source-code/src/skills/bundled/verify.ts:12-29`
- `claude-code-source-code/src/skills/bundled/remember.ts:9-81`
- `claude-code-source-code/src/skills/bundled/stuck.ts:6-79`

实践建议:

- 重复出现的高价值工作流，优先抽成 skill，而不是每次重写 prompt。
- 需要参考资料的 skill，可以像 bundled skill 一样把文件随技能一起下发。
- 有明确适用文件范围的 skill，使用 `paths` frontmatter，让系统按文件触发。

## 7. MCP 的正确使用方式

源码表明 MCP 在 Claude Code 里不是“临时外挂”，而是正式扩展总线。配置支持:

- `.mcp.json` 逐级向上继承 project scope
- user scope
- local scope
- enterprise managed scope
- claude.ai connectors
- plugin 提供的 MCP servers

同时系统会做:

- schema 校验
- env var expansion
- policy filter
- duplicate server suppression

证据:

- `claude-code-source-code/src/services/mcp/config.ts:888-960`
- `claude-code-source-code/src/services/mcp/config.ts:1164-1250`
- `claude-code-source-code/src/services/mcp/config.ts:1258-1290`
- `claude-code-source-code/src/services/mcp/types.ts:9-27`

实践建议:

- 优先把稳定集成接到 MCP，而不是塞进 Bash 脚本。
- 同一个服务不要同时从 plugin、手写 `.mcp.json`、claude.ai connector 三路重复接入。
- 把 MCP 看成“能力边界扩展”，不是“万能远程 shell”。

## 8. 远程能力分两类，不要混淆

### 8.1 remote session

适合“把任务放到云端环境跑”，本地只做观察与控制。

- `claude-code-source-code/src/remote/RemoteSessionManager.ts:87-141`

### 8.2 remote control bridge

适合“把本机暴露成一个可被远端控制的 bridge 环境”。

证据:

- `claude-code-source-code/src/entrypoints/cli.tsx:108-161`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:57-87`

实践建议:

- 想把任务放云上跑: 用 remote session。
- 想让手机或网页来驱动当前机器: 用 remote control。

## 9. 把 session 当成工作对象，而不是聊天历史

源码显示 Claude Code 明确支持：

- `/resume`
- `/memory`
- `/rewind`
- session title / tag / PR link / agent metadata
- file checkpointing

证据:

- `claude-code-source-code/src/commands/resume/index.ts:3-10`
- `claude-code-source-code/src/commands/memory/index.ts:3-8`
- `claude-code-source-code/src/commands/rewind/index.ts:3-11`
- `claude-code-source-code/src/utils/sessionStorage.ts:2690-2817`
- `claude-code-source-code/src/utils/fileHistory.ts:45-78`

实践建议:

- 长任务不要只依赖“当前上下文还记得”，而要主动使用 `resume`、title、tag 形成可恢复入口。
- 对项目级稳定规则，用 `CLAUDE.md` / `CLAUDE.local.md`；对会话级沉淀，理解 session memory 会自动参与压缩与恢复。
- 把 `/rewind` 当成工程保险丝，而不是最后没办法时才想起的按钮。

## 10. 最后的高价值建议

如果只记三条:

1. 先问“我现在在保护哪个世界对象”，再决定 prompt 怎么写。
2. 先问“哪些扩张值得现在支付”，再决定权限怎么开、上下文怎么塞、会话要不要继续。
3. 先问“这次便利会不会制造第二真相”，再决定要不要并发、外置、恢复或自动化。

## 12. 复杂任务先分预算，再选对象

真正复杂的任务，不要一上来就写更长 prompt。

更稳的顺序是：

1. 先写清目标预算和验收标准。
2. 再写动作预算，限制工具、目录、远程面。
3. 再决定这次任务应该落在 session、task、workflow 还是 worktree。
4. 最后再让 prompt 只承担合同，而不是承担全部状态。

完整实践版继续看：

- `guides/06-第一性原理实践：目标、预算、对象、边界与回写.md`

## 13. 真正的省 token，不是少说话，而是稳定前缀与裁剪能力

从源码看，更稳的省 token 方式不是把回答压短，而是：

1. 让稳定规则留在稳定层。
2. 让大块输出外置。
3. 让无关能力不要进入当前工具池。
4. 接受 compact / reactive compact 是主路径，而不是异常。

## 14. 研究源码时先找 contract，再找热点文件

如果你是在读 Claude Code 源码，而不是单纯使用它，更稳的顺序是：

1. 先找 task type、control schema、permission union 这类正式合同。
2. 再看 `query.ts`、`main.tsx`、`attachments.ts` 这类热点文件在维护什么 invariant。
3. 最后再写公开镜像缺口与技术债。
