# hooks 四层漂移专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经写过：

- 扩展与集成专题中的 hooks 角色
- 扩展运维专题中的 `/hooks` 观察面
- 来源信任与 hooks 总闸

但仍缺一层专门回答：

- `/hooks` 为什么不是运行时真相
- settings hooks、plugin hooks、session hooks、frontmatter hooks 为什么不是同一对象
- `allowManagedHooksOnly`、`disableAllHooks`、plugin-only hooks policy 为什么作用在不同层

如果这层不拆，正文会继续犯两类错：

- 把 `/hooks` 写成“会执行的 hooks 总表”
- 把来源锁定、执行锁定、trust 前置条件写成一类开关

## 苏格拉底式自审

### 问：为什么不把这批继续塞进 14 号 trust/policy 长文？

答：因为 14 号页回答的是“来源信任怎样分级”，主要对象是 workspace trust、source trust 与执行门控的关系。hooks 四层漂移更窄，也更偏运行时对象语义：

- `/hooks` 看见什么
- snapshot 装了什么
- registered / session hooks 怎么合流
- 实际执行再被哪些门拦掉

继续往 14 号页堆，会把“来源信任”与“hooks 运行时对象模型”混成一页。

### 问：为什么不把这批继续塞进 17 号运维专题？

答：因为 17 号页回答的是“真实工作中怎么稳地管理扩展对象”，它可以说明 `/hooks` 是观察面，但不适合承载：

- UI 可见与真实执行脱钩
- plugin hooks / session hooks / frontmatter hooks 的不同入口
- `statusLine` / `fileSuggestion` 这种 hooks 邻域旁路

这批更适合单独成为控制面页。

### 问：正文里最该避免的假并列是什么？

答：

- `/hooks` UI vs 运行时执行总线
- settings snapshot vs registered hooks
- session hooks vs frontmatter hooks
- plugin-only hooks policy vs managed-only hooks policy
- Trust Dialog 提示面 vs hooks 的统一 trust 执行前置条件
- 普通 hooks 事件 vs `statusLine` / `fileSuggestion`

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- `hooksConfigManager` 如何把 registered plugin hooks 拼进 UI grouping
- `HooksConfigMenu` 的 disabled modal / restricted banner 的具体分支文案
- `statusLine` / `fileSuggestion` 的命令执行细节、超时和返回格式
- registered hooks / session function hooks 的内部数据结构

这些都是维护判断，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- hooks 至少有加载、注册、执行、UI 四层不同投影
- `/hooks` 是观察面，不是运行时真相
- `strictPluginOnlyCustomization['hooks']`、`allowManagedHooksOnly`、`disableAllHooks` 作用在不同层
- workspace trust 是交互模式下 hooks 的统一执行前置条件
- `statusLine` / `fileSuggestion` 应写成 hooks 邻域旁路，而不是普通 matcher hooks

### 不应写进正文的

- UI 具体通过哪些 grouping 函数把 registered plugin hooks 拼出来
- 某个 modal / banner 的即时文本
- session function hooks 与 registered hooks 的内部结构
- `statusLine` / `fileSuggestion` 的命令实现细节

这些继续只放 `docs/userbook-memory/`。

## 本轮从正文下沉回 memory 的实现细节

### `/hooks` UI 与执行语义的错位证据

- `HooksConfigMenu` 主要围绕 `disabledByPolicy` / `restrictedByPolicy` 两种 UI 状态组织界面。
- `getAllHooks()` 不检查 plugin-only hooks policy，只决定是否隐藏 editable settings hooks。
- `hooksConfigManager` 还会把 registered plugin hooks 拼进 UI。

所以“菜单可见”与“真实执行”天然错位，这个结论可以写入正文；但其精确拼装细节应留在 memory。

### managed-only 的真实执行外延

- `allowManagedHooksOnly` 不只是 settings snapshot 只留 managed hooks。
- 它还会让 `getHooksConfig(...)` 跳过 plugin hooks 和 session hooks。
- `SessionStart` / `Setup` 甚至直接跳过 `loadPluginHooks()`。

正文可以稳写：

- 它是执行层 managed-only 策略

但不应把每个函数级分支都抬进正文。

### `disableAllHooks` 的来源差异

- `policySettings.disableAllHooks === true` 才是全部 hooks 真停。
- non-managed `disableAllHooks` 只会把不少路径退化成 managed-only。
- UI 有时会把这两种状态合并成相似提示，这也是一个会误导读者的点。

正文应强调来源差异；具体 UI 偏差留在 memory。

### hooks 与 trust 的统一前置条件

- `hooks.ts` 里对交互模式 hooks 统一加了 `shouldSkipHookDueToTrust()`。
- 这条门不仅覆盖普通 hooks，也覆盖 outside-REPL hooks、`statusLine`、`fileSuggestion`。

正文应写：

- hooks 的执行统一要求 workspace trust

但不应堆叠历史漏洞注释与所有 callsite。

## 后续继续深入时的检查问题

1. 当前讨论的是 UI、加载、注册，还是执行层？
2. 这个 hooks 对象来自 settings、plugin、session，还是 frontmatter？
3. 当前策略在改写来源面，还是执行面？
4. 我是不是把 `/hooks` 菜单仍然当成了运行时真相？
5. 我是不是把 hooks 邻域旁路命令混写成普通 hooks 事件？

只要这五问没先答清，正文就会继续把 hooks 写成一张假的“统一配置总表”。
