# dual-trunk stable-gray hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经补完：

- `213` 的 scope guard
- `204` 的 stable / conditional / gray hardening

之后，

当前更值钱的下一刀，

不是继续抽更细的目录入口，

而是把：

- `213-owner-side recovery、transport stop rule 与 compaction boundary：为什么 122、123、124、125、126、127 不是线性六连，而是双主干加两个 zoom.md`

也补成和 `204/214/215/216/217` 一致的三层分账。

## 为什么这轮值得单独提交

### 判断一：当前缺口已经不在“213 有没有 dual-trunk map”，而在“213 会不会把条件公开和灰度实现继续写成同一桶”

`213` 现在已经有：

- 页首范围声明
- 双主干 + 两个 zoom 的结构图

但后段仍把：

- 宿主条件
- 具体 timeout / reconnect / compaction 常量
- 当前 hook 顺序与显示层投影

压在同一个：

- `条件 / 灰度证据`

桶里。

这会让读者虽然知道结构，

却还不容易区分：

- 哪些是当前仍可被外部观察到的条件公开行为
- 哪些只是内部实现或灰度证据

### 判断二：这轮补的不是新结构，而是合同厚度

本轮真正缺的不是：

- 新的 branch map
- 新的 subfamily hub

而是让 `213` 也承担：

- 保护稳定骨架
- 标出条件公开
- 隔离内部 / 灰度证据

的责任。

这和 `204/214/215/216/217` 的写法已经一致，

只是 `213` 还没补齐。

## 这轮具体怎么拆

这轮只做一件事：

1. 在 `213-...双主干加两个 zoom.md`
   的后段，
   把原来的“稳定主干 + 条件 / 灰度证据”改写成：
   - `稳定可见`
   - `条件公开`
   - `内部/灰度层`
   并把 `122/123/124` 与 `125/126/127` 重新放回这三层

这样：

- `213`
  就不只是一张 dual-trunk zoom map
- 它还会成为 remote recovery 这簇页面的稳定骨架页
- 后面继续读 `214/215/216/217` 时，也能保持同一套三层口径

## 苏格拉底式自审

### 问：为什么这轮不是继续抽更细的 `122-127` 子家族？

答：因为目录入口现在已经够细。当前真正还不够硬的是 `213` 自己还没把条件公开和内部灰度彻底分账。

### 问：为什么 `213` 需要补这三层，而不是只保留“稳定主干 + 条件/灰度证据”？

答：因为前者会让条件公开行为和内部实现证据重新混桶；后者则能和相邻成熟结构页保持一致，让读者知道哪些是可观察条件，哪些只是内部举证。

### 问：为什么这轮不再补页首？

答：因为页首范围声明已经有了。当前缺口在读完整页之后的层级分账，而不是结构图开头的误听切断。
