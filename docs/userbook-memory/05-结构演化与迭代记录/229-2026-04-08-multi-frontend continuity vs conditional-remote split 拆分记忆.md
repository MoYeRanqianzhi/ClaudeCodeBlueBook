# multi-frontend continuity vs conditional-remote split 拆分记忆

## 本轮继续深入的核心判断

“多前端接续”当前真正缺的不是：

- 更多 remote 叶子页
- 更细的 bridge 事件证明

而是：

- 一个能把稳定接续面和条件远端面拆开的前门

正文里已经有：

- `04-专题深潜/16` 负责 `IDE / Desktop / Mobile / /session`
- `04-专题深潜/06` 负责更宽口径的前端与远端总览
- `04-专题深潜/20/23` 负责 deeper remote runtime 分诊
- `02-能力地图/05-体验与入口/03` 已经把 `/session` 和 `/remote-control` 分成两种宿主语义

所以这轮最稳的做法不是：

- 再补一张 remote 总图

而是：

- 新增 `04-专题深潜/多前端接续与条件远端/README.md`
- 把 `README`、`04-专题深潜/README`、`00-阅读路径`、`04-专题深潜/16` 接到这个前门
- 顺手修正 `04-专题深潜/06` 里把 `/session` 压成条件面的口径漂移

## 为什么这轮该先补“多前端接续”的物理前门

### 判断一：读者先问的是“我是在继续工作对象，还是在进入条件远端”

只要用户已经分清：

- 这是多前端工作
- 这和远端 / bridge 有关

下一步最常见的误判就不是：

- “这些命令分别叫什么”

而是：

- “这些是不是都属于远端入口”

如果没有物理前门，

读者就会在：

- `/session`
- `/remote-control`
- `/mobile`
- `/voice`
- `/chrome`

之间自己重建边界。

所以当前缺的不是更多分析，

而是：

- 一个能把稳定接续和条件远端拆开的目录入口

## 为什么 `/session` 必须从条件面收回到稳定面

### 判断二：`/session` 的稳定性来自入口稳定，不是来自无前提

之前口径漂移的核心问题不是：

- `/session` 没有前提

而是：

- 它被写进了“灰度或条件面”

这会让读者自然把它和：

- `/remote-control`
- `/remote-env`
- `/voice`
- `/chrome`

压成同一类条件能力。

更稳的写法是：

- `/session` 属于稳定接续面
- 但它带有明确 runtime 前提：必须已经进入 remote mode

也就是说，稳定与否回答的是：

- 是否有明确公开入口与稳定对象

前提回答的是：

- 当前上下文能否使用

这两件事不能混成一层。

## 为什么这轮不直接去改 `20/23`

### 判断三：先把入口层分清，再让 deeper runtime 分诊继续承担它自己的问题

`20/23` 已经在处理：

- 远端输入注入
- bridge 审批
- presence ledger
- bridge 后台分诊

这些都已经属于：

- deeper runtime

如果这轮直接继续往 `20/23` 加叶子页或改模板，

读者仍然会在入口层复发同一个误判：

- 把稳定接续和条件远端写成同一类“远端能力”

所以更稳的顺序是：

1. 先把前门拆开
2. 再让 `20/23` 继续承接 deeper runtime 问题

## 为什么这轮顺手改 `04-专题深潜/06`

### 判断四：总览页必须和专题页、能力地图页在对象分层上对齐

`04-专题深潜/06` 之前把：

- `/session`

写进：

- 灰度或条件面

但 `04-专题深潜/16` 与 `02-能力地图/05-体验与入口/03`

都已经把它写成：

- 稳定接续面，但带 runtime 前提

如果不顺手纠偏，

新的前门只会继续把读者送进：

- 三种互相打架的口径

所以这轮必须一起修掉这个漂移。

## 本轮最关键的新判断

### 判断五：高频使用功能的目录优化，优先保护“对象层”而不是“技术相邻”

`/session`、`/remote-control`、`/voice`、`/chrome`

技术上都跟远端 / 多前端有关，

但对用户更重要的不是它们共享了什么实现，

而是：

- 它们到底是不是同一种工作对象

这轮说明入口层最该保护的，不是技术相邻，

而是：

- object separation

即：

- viewer continuity
- host control
- conditional frontends

必须先分账。

## 苏格拉底式自审

### 问：我现在是在继续同一工作对象，还是在把本机变成宿主？

答：如果是后者，就不该再把 `/session` 和 `/remote-control` 写成同一种入口。

### 问：我是不是因为这些能力都跨设备，就把稳定接续和条件远端压成了同一层？

答：跨设备不等于同层；先分 viewer 接续、host 宿主、条件前端，再谈 deeper runtime。

### 问：我是不是因为 `20/23` 已经存在，就想直接继续扩 remote 叶子页？

答：如果入口层还没分清，那继续扩叶子只会把入口误判重新放大。
