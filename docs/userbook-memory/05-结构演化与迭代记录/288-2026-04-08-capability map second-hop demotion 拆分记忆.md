# capability map second-hop demotion 拆分记忆

## 本轮继续深入的核心判断

这轮最值得先收的，

不是能力地图正文内容本身，

而是根层 front-door 对它的入口层级。

`02-能力地图`

一直被正文定义为：

- 机制总图
- 为什么这样实现
- 不是默认前门

但根层第一屏却仍把它放在：

- `01-主线使用`
- `03-参考索引`
- `04-专题深潜`
- `05-控制面深挖`

同一层的并列入口里。

## 为什么这轮落点选根层 `README.md` 与 `00-导读.md`

### 判断一：当前真正冲突的不是正文定义，而是 front-door 两层文案还没一起降层

根层第一屏文案说的是：

- 还没分清该走主线使用、参考索引、专题深潜还是控制面深挖

但并列列表里又把：

- `02-能力地图`

放进去。

这会让读者先在最前门吸收成：

- 能力地图也是和这四个“用户入口层”并列的 first-hop

而这和仓内已有定义直接冲突。

同时，

`00-导读.md`

里“如果你只想知道先看哪里”那组极简说明，

也还把：

- 能力地图

写成与：

- 主线使用
- 参考索引
- 专题深潜
- 控制面深挖

同层的并列“先看哪里”。

所以这轮要一起把：

- 根层 `README`
- `00-导读`

都改成 second-hop 口径。

## 为什么不是把 `02-能力地图` 从根层完全拿掉

### 判断二：这轮是降层，不是隐藏

能力地图仍然有价值，

只是它回答的是：

- 机制总图
- 实现组织

而不是：

- 用户刚进门时该先走哪条工作主线

所以这轮保留：

- 根层已有的 `03-参考索引/04-功能面七分法 -> 02-能力地图`

作为 second-hop 路径，

并把导读中的极简定义同步改成：

- `02` 给你 second-hop 的机制总图

而不是把：

- `02-能力地图`

完全从根层消失。

## 为什么这轮不顺手做 `/review` 与 `ultrareview`

### 判断三：那是另一条可独立提交的标题层澄清，不该混进 front-door 层级修正

并行分析里仍然有另一个很小的候选：

- `/review` 与 `ultrareview` 的主次标题

它成立，

但那条处理的是：

- 索引页里稳定入口和远程路径的主次关系

而这轮更应该单独提交的是：

- 根层入口层级

两者虽然都属于 front-door，

但一个是：

- 根层目录路由

另一个是：

- 具体合同页标题澄清

不应混成一刀。

## 苏格拉底式自审

### 问：我是不是把“能从根层点进去”误当成了“就该作为 first-hop 并列入口”？

答：能被找到不等于应该并列作为第一跳；能力地图更适合作为 second-hop。

### 问：我是不是因为它叫“能力地图”，就默认它应该排在所有前门入口旁边？

答：名字像总图，不等于阅读角色就是前门导航；仓内正文已经把它定义成机制总图层。

### 问：我是不是又想通过同一提交顺手多做一条 `/review` 标题澄清？

答：那会把“根层入口层级修正”和“合同页标题澄清”混成一刀，破坏提交边界。
