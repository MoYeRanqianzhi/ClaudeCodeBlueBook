# 插件自动物化专题拆分记忆

## 本轮继续深入的核心判断

上一批已经把：

- `/reload-plugins` 不是安装器
- 插件有意图、物化、当前会话激活三层

写出来了。

但仍缺一层专门回答：

- 交互 startup、background install、headless install 为什么不是同一条自动链
- startup trust 为什么会拦住交互 REPL 的自动物化
- 为什么有些路径会自动 refresh，有些只会留下 `needsRefresh`

如果这层不拆，正文会继续犯两类错：

- 把插件自动出现写成“系统总会自动安装并生效”
- 把 `needsRefresh` 写成失败，而不是自动链停在激活层前

- headless sync install vs 交互 REPL 自动链

## 正文落笔纪律

### 应写进正文的

- 交互 REPL 的 startup plugin path 先过 trust，再决定是否推进后台物化
- seed / background install 主要推进物化层，不天然等于当前 session 已激活
- headless 与交互 REPL 的自动刷新策略不同
- 自动刷新与手动 `/reload-plugins` 共享同一个 Layer-3 refresh primitive
- `needsRefresh` 主要表示自动链停在激活层之前

### 不应写进正文的

- seed marketplace 的覆写规则
- zip cache / unsupported source 的具体条件
- `applyPluginMcpDiff()`、prefetch race 规避与 timeout env 细节
- background install 各种低层缓存操作

这些继续只放 `docs/userbook-memory/`。

## 本轮从正文下沉回 memory 的实现细节

### startup trust 的精确位置

- `performStartupChecks()` 只在 trust accepted 后继续。
- 这让交互 REPL 的插件自动物化链天然晚于工作区信任建立。

正文应写 trust gate 的存在；具体 callsite 与注释细节放 memory。

### headless / sync install 的时序证据

- `print.ts` 可在 first query 前等待 `installPluginsForHeadless()`。
- `CLAUDE_CODE_SYNC_PLUGIN_INSTALL` 会改变等待时机，并在后面显式 `refreshPluginState()`。
- `setup.ts` 还会为此跳过 plugin prefetch，避免 race。

正文适合只写“headless 更强调首问前可用”；具体时序留在 memory。

### background install 的双分支策略

- 新安装倾向 auto-refresh。
- 更新已有 marketplace 更倾向只标记 `needsRefresh`。

这条分支非常适合作为正文结论，但实现条件与 fallback 行为仍应留在 memory。

## 后续继续深入时的检查问题

1. 当前讨论的是 trust、物化，还是激活层？
2. 当前宿主是交互 REPL、background install，还是 headless？
3. 这里的自动行为是“继续物化”，还是“已经替当前 session refresh”？
4. 我是不是把 `needsRefresh` 误写成失败？
5. 我是不是把自动安装和自动激活混写成了同一个动作？

只要这五问没先答清，正文就会继续把插件自动链写成一张假的“自动安装成功状态表”。
