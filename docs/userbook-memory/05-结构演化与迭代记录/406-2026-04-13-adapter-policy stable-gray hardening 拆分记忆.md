## 为什么这轮值得单独提交

### 判断一：目录层的边际收益已经低于正文合同厚度

当前读者已经能稳定走到：

- callback-visible
- adapter triad
- UI consumer policy

这条线。

继续新增 `112-113` 或 `114-115` 的目录入口，

收益会开始低于：

- 把 `115` 自己的 stable / conditional / gray 边界写硬

所以这轮真正该补的不是“更细的入口”，

而是“更硬的合同厚度”。

### 判断二：`115` 正好卡在最容易把稳定对象和宿主策略写平的地方

`115` 当前最容易被误写成：

- adapter 里一切都只是“显示/不显示”过滤

但它真正回答的是三种不同 consumer rationale：

- render parity
- replay completeness
- success-noise suppression

如果不把这三者按稳定合同、条件公开和内部灰度重新分账，

读者就会继续把：

- `convertToolResults`
- `convertUserTextMessages`
- success `result -> ignored`

写成同一种 policy。

### 判断三：这轮比继续拆 `114-115` 更贴近“保护稳定功能和灰度功能”

这页最需要保住的稳定层是：

- same adapter layer != same consumer rationale

最需要降到条件或灰度层的是：

- 哪些宿主会打开哪些 policy
- future host 会不会继续只保这三类 policy
- exact helper 顺序和 host wiring

这正是用户要求的：

- 保护稳定功能
- 隔离灰度功能

这样：

- `115`
  会和相邻成熟结构页共享同一套分层口径

## 苏格拉底式自审

### 问：为什么这轮不是继续去做 `114-115` 子家族入口？

答：因为当前更稀缺的不是 findability，而是合同厚度。读者已经能找到这条线，缺的是如何不把三种不同 policy 混成同一个过滤开关。

### 问：为什么这轮不是继续去做 `112-113`？

答：因为那条线虽然更干净，但边际收益已经更低；`115` 这页正处在 stable / conditional / gray 最容易串层的地方，先补它更值。

### 问：为什么这也算“优化相关目录结构”？

答：因为结构优化不只等于新建目录。当前选择不继续裂目录，本身就是在保护层级职责，把“找得到”与“说得稳”重新拉回平衡。
