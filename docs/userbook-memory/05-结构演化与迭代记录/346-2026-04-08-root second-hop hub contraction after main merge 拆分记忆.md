## 先检查并并入主线更新

本轮开始前，

当前 worktree 先执行：

- `git fetch origin main`
- `git pull --ff-only origin main`

结果仍然失败，

原因不是主线不可读，

而是：

- 当前 `userbook` 已经带着自己的提交，无法对 `origin/main` 快进

继续核对后确认：

- 本地 `main` 比 `origin/main` 更新
- 当前 `userbook` 也落后于本地 `main`
- `git merge-tree $(git merge-base HEAD main) HEAD main` 未显示冲突标记

所以只在当前 worktree 执行：

- `git merge --no-edit main`

把主线更新合进当前 `userbook`，

没有改动主分支和其他 worktree。

## 多 Agent 交叉结论

这一刀直接吸收了三条并行只读分析：

1. `Einstein` 判断，
   根前门现在最该保留的只有 5-8 个稳定入口，
   不该继续在根页展开 `04 / 05` 的叶子页直发；
   具体深页应回到：
   - `00-阅读路径`
   - `04-专题深潜/README`
   - `05-控制面深挖/README`
2. `Feynman` 判断，
   下一批大的结构收敛更适合先做：
   - `控制 vs 治理`
   - `恢复对象`
   而不是继续在入口层反复整理投影或反馈面；
   这反过来也要求根前门先退回 owner switchboard，而不是继续代替这些层发叶子页。
3. `Wegener` 判断，
   条件面、灰度面与稳定面最容易在前门/索引/矩阵页被混写成默认入口，
   所以根前门越长，
   越容易把：
   - 稳定二跳
   - 条件能力
   - 实现叶子
   混成同一层

这三条判断收敛成一句话就是：

- 根前门应该负责把用户送到正确的 owner，不该继续代替 owner 发叶子页

## 这轮具体怎么拆

这轮只做一件事：

1. 保留这段前面的原则说明
2. 把长尾深链压缩成 7 个稳定入口：
   - `00-导读`
   - `00-阅读路径`
   - `01-主线使用/README`
   - `04-专题深潜/README`
   - `05-控制面深挖/README`
   - `03-参考索引/README`
   - `03-参考索引/04-功能面七分法 -> 02-能力地图/README`
3. 再补三条“如果你已经知道自己卡在哪类对象，就回对应 owner”：
   - 正式路径
   - 专题 README
   - 控制面 README

这样：

- 根页重新只负责稳定 handoff
- 正式路径继续负责顺序编排
- 各子树 README 继续负责内部入口
- 叶子控制面与实现对象退出根前门

## 为什么这轮不顺手继续压 `00-导读`

因为这一轮最小有效修正的主源仍然在：

- `README` 的二跳段

`00-导读`

虽然也有一定前门职责，

但它目前仍比根 README 更克制，

first-hop owner 的主要越界点还在根页。

## 苏格拉底式自审

### 问：根页真的需要告诉用户这么少吗？

答：需要。根页值钱的不是覆盖面，而是少数稳定 owner。只要根页还在大量点叶子，它就还在和路径页争权。

### 问：删掉这些深链，会不会让用户更难找到具体问题？

答：不会。具体问题没有被删除，只是退回了正式路径页和各子树 README。根前门失去的是越级库存权，不是解释能力。

### 问：为什么不继续保留几个“看起来很常用”的深页？

答：因为一旦允许根页继续豁免几个深页，first-hop owner 又会慢慢回流成“根页兼任叶子目录”。这轮要收的是制度边界，不是只修几个链接。
