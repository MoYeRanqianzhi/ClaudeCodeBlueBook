# root path-90 route re-lift after main fast-forward 拆分记忆

## 本轮继续深入的核心判断

这一轮最值得修的，

不是继续让根 `README`

把：

- `在 IDE、Desktop、Mobile 之间继续同一工作`

直接落到：

- `04-专题深潜/多前端接续与条件远端/README.md`

而是重新把这条目标抬回正式的：

- `路径 90：我想在 IDE、Desktop 和移动端之间继续同一工作`

上。

## 为什么这轮值得单独提交

### 判断一：快进 main 后，这条目标从“路径入口 + 禁止局部直达”回流成了“只剩局部直达”

本轮开始前，

当前 worktree 先做了：

- `git fetch origin main`
- 对本地 `main` 的快进同步

快进之后，

根前门这里重新变成：

- 只给 `多前端接续与条件远端/README.md`

却不再先把读者交给：

- `路径 90`

这说明回流的不只是“多余的局部补句”，

而是更前面的职责层级本身：

- 根前门再次把路径页绕开了

### 判断二：`路径 90` 不是 `多前端接续与条件远端/README` 的别名

`路径 90`

的顺序是：

1. `多前端接续与条件远端/README.md`
2. `16-IDE、Desktop、Mobile 与会话接续专题.md`
3. `06-前端与远程专题.md`
4. `01-主线使用/01-进入系统：启动、会话与工作目录.md`

所以根前门如果只给第 1 步，

就会把：

- “这是这条路径的第一跳”

偷换成：

- “这就是这条目标的默认入口”

这会把路径页原本负责的完整顺序压扁成一个局部专题目录。

### 判断三：这条目标尤其需要先经过路径页，因为它天然容易把不同宿主、不同 viewer 与不同远端条件混成同一种“远程入口”

`路径 90`

真正先帮读者做的是：

- 区分 IDE 连接、Desktop handoff、移动端下载和 remote session viewer 的对象差异
- 避免把 `/session`、`/remote-control`、`/mobile` 混写成同一种“远程入口”

如果根前门直接把人送进多前端专题目录，

就会默认读者已经知道：

- 自己卡的是哪一种前端接续对象

而不是先通过路径页做对象分型。

## 这轮具体怎么拆

这轮只做一件事：

1. 把根 `README` 里的：
   - `在 IDE、Desktop、Mobile 之间继续同一工作 -> 多前端接续与条件远端/README`
   收成：
   - `在 IDE、Desktop、Mobile 之间继续同一工作 -> 00-阅读路径.md -> 路径 90`
2. 保留一条条件化下钻：
   - 只有在已经确认自己主要卡在多前端接续、条件远端或 viewer 对象差异时，才继续看 `多前端接续与条件远端/README`

这样：

- 根前门重新把目标 handoff 交回正式路径
- 专题目录仍保留为条件化二跳
- 之后如果路径 90 调整顺序，根页无需再同步局部路线

## 为什么这轮不顺手连同“交付”“自动化”等其它回流目标一起处理

因为当前最小有效边界仍然是：

- 一个被快进回流的根目标入口

如果把多条回流路径同时收，

提交粒度就会从：

- 单目标重抬升

变成：

- 主线快进后的成片回流清扫

不利于持续判断哪些目标最容易退回“根页直接打深页”。

## 苏格拉底式自审

### 问：为什么根前门不能直接给 `多前端接续与条件远端/README`，它明明很相关？

答：相关不等于适合做前门。只要正式路径已经承担对象分型与顺序编排，根前门就不该再把第一步专题页冒充成默认入口。

### 问：为什么还保留一条“如果你已经确认卡点”的深页下钻？

答：因为要收的是默认 handoff，不是抹掉深页价值。深页仍可作为条件化二跳，但不能越过路径页抢前门。

### 问：这轮是不是只是把旧修复再做一遍？

答：不是简单重复，而是在把“主线快进后最容易回流的入口形态”显式记忆化。之前清的是局部直达补句，这次清的是更前面的“根页直接打第一跳专题页”。
