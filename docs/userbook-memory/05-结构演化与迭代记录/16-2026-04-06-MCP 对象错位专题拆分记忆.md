# MCP 对象错位专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经写过：

- MCP 是外部系统边界
- `/mcp` 是稳定公开的 MCP 管理面
- plugin-only `mcp` 与 agent frontmatter MCP 会受来源信任影响

但仍缺一层专门回答：

- 全局配置总览、按名解析、当前 client、agent frontmatter 为什么不是同一对象
- 为什么 `/mcp` 菜单里看见的 server 不等于 agent string reference 一定可用
- 为什么 plugin-only `mcp` 在总览层与按名层会出现不同结果

如果这一层不拆，正文会继续犯两类错：

- 把 `/mcp` 写成 MCP 唯一真相
- 把 string 与 inline `mcpServers` 写成同一语义

## 正文落笔纪律

### 应写进正文的

- MCP 至少有配置、按名解析、连接、Agent 挂载四层不同对象
- `/mcp` 是连接视图，不是唯一真相
- `getClaudeCodeMcpConfigs()` 与 `getMcpConfigByName()` 不是同一集合
- string 与 inline `mcpServers` 必须分开写
- `mcp` surface 被 policy 锁窄时，agent 的 MCP 面可能被整体剥掉

### 不应写进正文的

- plugin MCP servers 的 namespaced 命名与 `dynamic` scope 归类
- claude.ai / connector 拉取与 dedup 实现
- UI 某个面板如何回填 config location 的具体代码路径
- proxy URL 展开与 signature 细节

这些继续只放 `docs/userbook-memory/`。

## 本轮从正文下沉回 memory 的实现细节

### plugin-only `mcp` 的双层错位证据

- `getClaudeCodeMcpConfigs()` 在 `mcpLocked` 时会跳过 user/project/local，但继续装 plugin servers。
- `getMcpConfigByName()` 在 `mcpLocked` 时却直接退化成只看 enterprise servers。

正文可以稳写：

- plugin-only `mcp` 在全局总览层与按名解析层不是同一语义

但不应在正文里堆所有 precedence 与 merge 细节。

### `/mcp` UI 与按名解析的错位证据

- `MCPSettings` 主要从当前 `mcp.clients` 派生普通 server 列表。
- `MCPListPanel` 还会单列 agent-specific servers。
- `MCPStdioServerMenu` 的 config location 又会回到 `getMcpConfigByName()` 去推 scope。

这说明 UI、连接对象与按名解析对象三者不等价。

正文应写这个结论；具体 UI 回填细节留在 memory。

### plugin agents 的特例

- plugin agent 文件里的 `mcpServers` 当前是故意忽略的。
- 这意味着“plugin source 是 admin-trusted”并不自动推导出“plugin agent 文件里写的 `mcpServers` 会生效”。

这条非常适合作为正文里的高级注记，但其实现理由和 PR 背景应留在 memory。

## 后续继续深入时的检查问题

1. 我现在讨论的是配置总览、按名解析、当前 client，还是 agent-specific server？
2. 这条 agent `mcpServers` 是 string 引用还是 inline 定义？
3. 当前 policy 锁的是来源层，还是已经影响到按名解析层？
4. 我是不是把 `/mcp` UI 误写成了 MCP 唯一真相？
5. 我是不是把 plugin servers、claude.ai servers、enterprise servers 和 agent inline servers 写成了同一类对象？

只要这五问没先答清，正文就会继续把 MCP 写成一张假的统一 server 总表。
