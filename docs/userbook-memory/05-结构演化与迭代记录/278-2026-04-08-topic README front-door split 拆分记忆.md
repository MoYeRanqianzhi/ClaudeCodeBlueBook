## 为什么这轮落点选 `04-专题深潜/README.md`

### 判断一：专题 README 首先应该让读者分清“我是在做 first-hop 分诊，还是已经准备进入长文主题”

这页上半段其实一直在做：

- first-hop 分诊

而下半段的长列表做的是：

- 完整专题目录

如果不把两者显式拆层，

读者就会继续把：

- 哪个前门切片先看

和：

- 所有长文主题都有哪些

当成同一种导航动作。

## 为什么这轮只加两个标题和一句过渡

### 判断二：当前缺的是层级信号，不是重写专题 README

这轮的问题不在于：

- 路由内容错误
- 主题列表错误

而在于：

- 缺少显式层级信号

所以最小且最稳的一刀不是：

- 重写整页结构

而是：

- 加 `前门切片`
- 加 `完整专题目录`
- 再补一句过渡说明

## 为什么要把“完整专题目录”单独标出来

### 判断三：如果不单列，长列表会反向吞掉上半段的 first-hop 角色

上半段负责的是：

- 带着问题选入口

下半段负责的是：

- 直接进入已知主题

如果不把下半段显式命名，

读者会继续把：

- 大段 first-hop 提示

和：

- 全量专题总表

读成同一张导航页。

## 为什么这轮不再继续补更多子目录 README

### 判断四：当前更缺的是把已有中层 hub 的角色说清，而不是继续扩文件数量

已经存在：

- 根 README
- `03-参考索引/README`
- `04-专题深潜/README`

这些层级化前门。

此时继续加更多 README，

不如先把：

- 这几个已有前门各自承担什么层级

写清楚。

## 苏格拉底式自审

### 问：我现在缺的是再加新路由，还是先把已有路由分清层级？

答：如果内容本身已经够用，当前更缺的是层级信号。

答：如果问题只是导航角色混在一起，先补层级标题比重写整页更稳。

答：如果它已经在承担完整目录角色，就应该显式被命名出来。
