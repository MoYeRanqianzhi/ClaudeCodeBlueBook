# continuation-bridge family hub uplift 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经连续补完：

- `218` / `219` 的 structure-page scope guard
- `203` / `204` 的 legacy scope guard

之后，

当前最自然的下一刀，

不是继续补新的单页 scope guard，

而是先在 memory 目录里给：

- `168-190`

这簇高密度接力链补一个 family hub。

## 为什么这轮值得单独提交

### 判断一：当前真正缺的是“家族入口”，不是更多叶页

`168-190` 已经同时拥有：

- `220` 对 `168-179` 的子树收束
- `221` 对 `180-190` 的三线收束
- `383` 与 `385` 对 `218/219` 的页首护栏
- `205/206` 对蓝皮书索引层的接力入口

这说明它缺的已经不是：

- 新的 leaf 页
- 新的 range-map 页

而是：

- 在 `docs/userbook-memory/05-结构演化与迭代记录/README.md` 的扁平时间流之外，
  先给这一整簇文件一个可浏览的家族入口

### 判断二：这轮更稳的是 README-only hub，而不是搬动旧文件

当前更脆弱的是既有链接，

不是编号本身。

所以这一刀更稳的最小改动是：

- 新增一个 `README-only hub`
- 不搬原始记忆文件
- 不修改原编号

这样既能提升检索效率，

又不会破坏已有引用。

## 这轮具体怎么拆

这轮只做三件事：

1. 新增：
   - `168-190-接续与桥接家族/README.md`
2. 在主 `README.md`
   里，
   给“快速索引”补一条 family hub 入口
3. 新增这条 `388` 记忆，
   记录为什么这轮优先做家族入口，而不是继续逐页补 guard

这样：

- `168-190` 这簇记忆会从“需要顺时间扫文件名”
  变成“先入家族，再跳子树 / 三线 / 索引 / scope guard”
- 目录结构会更像现有蓝皮书里的 README-only hub 模式
- 后续如果继续做 `191-206` 或 `100-149` family hub，也会有一条可复用的最小模板

## 苏格拉底式自审

### 问：为什么这轮不是继续补 `213` 或 `206`？

答：因为 `168-190` 现在已经有足够多的结构页和记忆护栏，真正的瓶颈变成“读者怎么在记忆层找到这一簇”，不是“再缺一张单页图”。

### 问：为什么要在 memory 目录层做 hub，而不是只靠主 `README` 再多加几条链接？

答：因为主 `README` 仍是总入口，家族内部的阅读关系需要一个能单独展开的二层入口；否则还是会回到平铺时间流。

### 问：为什么不直接搬 `168-190` 记忆文件进新目录？

答：因为当前最需要优化的是导航，不是物理路径。先加 hub，后谈搬迁，风险更低。
