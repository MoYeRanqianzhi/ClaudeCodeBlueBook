# frontend conditional-surface wording alignment 拆分记忆

## 本轮继续深入的核心判断

这一轮不继续改：

- `16-IDE、Desktop、Mobile 与会话接续专题`
- `控制 vs 治理`

而是先把：

- `04-专题深潜/06-前端与远程专题.md`

里最直接的稳定/条件公开边界对齐。

## 为什么这轮值得单独提交

### 判断一：`06` 当前的问题不是缺信息，而是条件面写得过稳

这一页已经在开头承认：

- 它是更宽口径的前端与远程总览
- 条件面会更多

但正文里仍有两类语气过稳的问题：

1. `/voice`
   被写成：
   - `正式产品能力`
2. `/chrome`
   被写成：
   - `纳入正式前端/集成面`

同时在稳定/条件分层里，

- `/desktop`
- `/session`
- `--remote`

也没有完全按“稳定公开 / 稳定但带前提 / 灰度或条件面”拆干净。

### 判断二：这刀比继续扩写 `16` 更小，也更接近稳定/灰度边界本身

`16`

更偏：

- 接续对象解释
- remote bit 与 URL 可用性的二元误读

而 `06`

这一页就是：

- 宽口径总览页
- 最容易把稳定面与条件面混写成一层

所以它更适合先收可见性语气。

## 这轮具体怎么拆

这轮只做三件事：

1. 把 `/voice`
   从：
   - `正式产品能力`
   收成：
   - `条件前端能力`
2. 把 `/chrome`
   从：
   - `正式前端/集成面`
   收成：
   - `条件前端/集成面`
3. 把稳定/条件分层改成：
   - `稳定主线`：`/ide`、`/mobile`、终端前端能力
   - `稳定但带前提`：`/desktop`、`/session`
   - `灰度或条件面`：`--remote`、`/voice`、`/chrome`、`/remote-control`、`/remote-env`

这样：

- `/desktop`
  不再和 `/ide`
  混成同样无前提的稳定面
- `/session`
  继续保留为稳定接续面，但明确受 remote mode 前提约束
- `--remote`
  从“稳定但带运行时前提”
  退回到更符合索引的灰度创建面

## 为什么这轮不同时改 `/session` 的 “remote bit != URL ready” 细节

因为那是：

- `16`
  更擅长承担的对象合同问题

当前这刀只收：

- `06`
  作为宽口径总览页的稳定/条件语气

## 苏格拉底式自审

### 问：为什么 `/voice` 和 `/chrome` 不能继续写成“正式能力”？

答：因为这会盖过它们仍受 gate、账户、环境、依赖与资格前提约束的事实。对宽口径总览页来说，这种误导比缺一句细节更严重。

### 问：为什么 `--remote` 不放在“稳定但带前提”？

答：因为它不只是运行时前提问题，而是可见性级别本身就不是稳定默认主线。把它放在“灰度或条件面”更能保护稳定功能与灰度功能的边界。
