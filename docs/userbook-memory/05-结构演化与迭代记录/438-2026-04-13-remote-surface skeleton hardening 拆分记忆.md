## 为什么这轮值得单独提交

### 判断一：`204` 已经在正文里承担 skeleton / anti-overlap map 职责，但尾部术语还停在半旧状态

`204`

正文前半已经明确在做：

- root page 与五条后继线的分叉梳理
- 不是 `20`
- 不是 `197`
- 不是 `00-阅读路径.md`

所以它实际承担的是：

- 局部结构页 / anti-overlap map

但尾部标题却还写成：

- `稳定层与灰度层`

而正文内部又已经出现：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

这说明它的尾部口径并没有完全跟上最近的结构页成熟格式。

### 判断二：这页最该保护的是 remote surface 分叉骨架本身，不是把中间节点名抬成稳定公开能力

和 `197/203/218/219` 一样，

`204`

真正稳定的不是：

- foreground runtime
- shared shell
- presence ledger
- gray runtime
- behavior bit

这些中间节点名本身，

而是：

- `132 -> 135 / 138 / 141 / 142 / 143`

这张分叉骨架的走法。

所以更合适的尾部口径应当显式写成：

- `稳定阅读骨架 / 条件公开 / 内部证据层`

### 判断三：这轮继续贯彻“正文结构统一优先于新增 hub”的目录策略

当前结构优化的高 ROI，

已经不在继续加 README / hub，

而在：

- 让同类型结构页共享同一套收束术语

`204`

作为 remote surface 分叉图页，

补这一刀比继续扩目录入口更值钱。

1. 修改 `204-...五种后继问题.md`

   - 记录为什么这轮优先补 `204` 的 skeleton 收束

   - 把最新批次索引推进到 `438`

## 苏格拉底式自审

### 问：为什么这轮不是继续回到 `194/195`？

答：因为那条 user-leg 支线刚完成 stable-gray hardening。当前更不统一的，是 remote surface 结构页 `204` 还保留着半旧的尾部术语。

### 问：为什么这轮不是继续去改 `206`？

答：因为 `206` 已经是成熟页。当前更明显的结构漂移点，在 `204` 这种 skeleton 页的尾部语言。

### 问：为什么这也算“优化相关目录结构”？

答：因为 `204` 本身就在承担 remote surface 的第二层导航与 anti-overlap map 职责；把它的尾部语言拉回结构页统一口径，本身就在降低目录系统里的结构表述漂移。
