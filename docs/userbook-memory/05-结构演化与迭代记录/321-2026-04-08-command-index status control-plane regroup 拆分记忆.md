## 本轮继续深入的核心判断

这一轮最值得收的，

不是继续大改整张命令索引的所有功能分桶，

而是把：

- `/status`

从 `会话与导航`

收回到更接近控制面的分组里。

## 为什么这轮值得单独提交

### 判断一：命令索引按“使用功能”分组时，先问它解决的第一性问题，而不是它有没有打开一个页面

`/status` 确实会打开一个界面，

但它解决的不是：

- 去哪里
- 切哪个会话
- 看帮助再继续走

而是：

- 当前宿主状态到底是什么

这和：

- `/config`
- `/usage`
- `/doctor`

共同落在运行时自检 / 控制面对象附近，

而不是导航语义里。

### 判断二：当前最小失配只出现在 `01-命令索引.md`

现有证据已经足够一致：

- `05-设置、状态、预算与调参入口索引.md` 把 `/status` 明确写成 `Settings tab`
- `05-控制面深挖/06-Status、Doctor、Usage：运行时自检、额度与诊断.md` 把它放进运行时自检控制面
- `04-根命令与斜杠命令对照索引.md` 也强调它是会话宿主状态投影

真正残留的错桶点，

只剩 `01-命令索引.md` 还把它放在：

- `## 会话与导航`

所以这轮只改这一页就够。

## 这轮具体怎么拆

### 判断三：不新开大类，只把它并入“状态与运行时配置”

这轮只做两件事：

1. 从 `## 会话与导航` 里移出 `/status`
2. 把 `## 运行时配置` 收敛成：
   - `## 状态与运行时配置`
   并在这里补上 `/status` 的对象说明

这样：

- `/status` 不再被读成导航页
- `/config` 与 `/status` 的 Settings 邻接关系变得可见
- 改动仍然限制在一个文件内

## 为什么这轮不顺手一起重排 `/doctor` `/usage`

### 判断四：那会把“修错桶”扩大成“整页控制面重组”

从第一性上看，

`/doctor`、`/usage` 也都与运行时自检紧邻。

但当前最明确、最局部、最不需要牵连其它说明的一刀，

只是：

- 先把 `/status` 从导航语义中移走

如果这一轮再把 `/doctor`、`/usage` 一起迁移，

就会变成整张索引表的再平衡，

超出本轮高信号小切片。

## 苏格拉底式自审

### 问：我是不是把“打开一个页面”误当成了“导航功能”？

答：是，这正是这轮要修的误判。页面只是表象，第一性问题才决定它该落在哪个桶里。

### 问：如果 `/status` 和 `/config` 都在 `Settings` 容器里，为什么过去我会把一个写成导航、另一个写成配置？

答：因为我被 UI 动作表面牵着走了，没有先问“它在帮用户确认什么运行时真相”。

### 问：我是不是应该一口气把整页所有控制面入口都重排？

答：不应该。当前最小有效改动，是先修掉一个最明显的错桶，再让后续重组有更干净的基线。
