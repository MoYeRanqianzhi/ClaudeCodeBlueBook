# bridge 补救动作与恢复建议拆分记忆

## 本轮继续深入的核心判断

第 34 页已经拆开：

- stale pointer
- deterministic invalid source
- env mismatch fallback
- transient retry

但 bridge host 还缺一层非常容易被继续混写的 remedial action semantics：

- `run claude first`
- `/login`
- restart `remote-control`
- fresh session fallback
- retry same command

如果不单独补这一批，正文会继续犯六种错：

- 把 workspace trust 与 account auth 写成同一条前提
- 把 `/login` 与 restart remote-control 写成同一种修复
- 把 fresh fallback 写成 resume success
- 把 retry 文案写成低信息量安慰句
- 把 `Session not found` 里的 login hint 写成纯 auth failure
- 把 failure object 与 remedy action 压成同一页

## 苏格拉底式自审

### 问：为什么这批不能塞回第 34 页？

答：第 34 页解决的是：

- 这次恢复失败到底坏在哪一层对象上

而本轮问题已经换成：

- 系统为什么给出这一类补救动作，而不是另一类

也就是从：

- failure object

继续下钻到：

- remedial action semantics

所以需要新起一页。

### 问：为什么这批也不能回塞到第 23 页？

答：第 23 页解决的是：

- trust
- eligibility
- trusted-device
- policy

那里主要在拆“钥匙”与资格边界。

本轮要拆的是：

- 当 bridge 真的失败或拒绝继续时，系统为什么要求用户做不同动作

也就是：

- operational remedy

而不是：

- static eligibility boundary

### 问：这批最该防的偷换是什么？

答：

- trust = auth
- `/login` = restart bridge
- fallback = resume success
- retry = generic comfort text
- login hint = pure auth branch

只要这五组没拆开，bridge 补救正文就还会继续糊。

## 正文落笔纪律

### 应写进正文的

- trust repair 解决的是 workspace prerequisite
- `/login` 解决的是 account prerequisite
- expiry restart 解决的是 expired host / environment
- fresh fallback 是 continuity downgrade
- retry same command 对应 pointer preserve 的用户面
- `Session not found` 的 login hint 不能被偷写成纯 auth failure

### 不应写进正文的

- 401 之前的 refresh retry 细节
- `errorType.includes('expired'|'lifetime')` 的实现判断
- suppressible 403 的噪音压制逻辑
- pointer 清理与保留的全部分支实现

这些只保留为作者判断依据。

## 本轮特别保留到记忆的源码发现

### `BRIDGE_LOGIN_INSTRUCTION` 是 bridge auth 错误的统一身份修复文案

`types.ts` 定义了统一的：

- `Please use \`/login\` to sign in with your claude.ai account`

`bridgeMain.ts` 无 token 时直接打印完整 `BRIDGE_LOGIN_ERROR`，`bridgeApi.ts` 的 `401` 也复用这条 instruction。

这意味着正文可以稳定写：

- `/login` 属于 account-plane remedy

但不必展开 refresh handler 的技术路径。

### expiry 型 `403` / `410` 与普通权限型 `403` 是不同 repair route

`bridgeApi.ts` 里：

- expiry / lifetime 语义的 `403`
- `410`

都指向 restart remote-control。

而普通 `403` 则提示检查组织权限。

所以正文必须写“不是所有 403 都能重启解决”，否则补救建议会被写糊。

### REPL 的 `session expired · /remote-control to reconnect` 与 CLI restart 文案属于同一 repair route

`replBridge.ts` 在 expiry fatal error 上，给的是短状态词：

- `session expired · /remote-control to reconnect`

它不是另一套策略，而是同一补救语义的压缩展示。

正文可以写成：

- CLI 是完整 repair instruction
- REPL 是短状态版

### `Session not found` 即使提到 login，也仍先清 pointer

`bridgeMain.ts` 在 session 不存在时会先清 pointer，再报：

- 可能 archived / expired
- 或 login lapsed

这意味着：

- login hint 只是可能原因
- 当前 failure object 仍然首先是 invalid resume source

正文必须 cross-link 第 34 页，不能直接合并到纯 auth 段落。

## 并行 agent 结果

本轮复用了现成 subagent 做并行核查。

当前已确认可用于本轮正文的结果：

- `401 -> /login` 属于身份/订阅前提修复
- expiry 型 `403` / `410` -> restart remote-control
- REPL 的 expiry 提示与 CLI repair route 对齐

延迟返回的结果若提供额外角度，只进入后续 memory，不回灌本轮正文主线。

## 后续继续深入时的检查问题

1. 我现在拆的是 broken plane，还是 remedy scope？
2. 这段话是在解释“为什么失败”，还是“为什么系统建议你这么修”？
3. 我是不是把 fallback 又写成了 resume success？
4. 我是不是把短状态词与完整 repair instruction 误写成两套策略？
5. 我是不是又让 `/login` 同时吞掉了 trust 问题和 invalid resume source 问题？

只要这五问没先答清，下一页继续深入就会重新滑回：

- 补救动作混写
- 或实现细节泄漏到正文

而不是用户真正可用的 remedial action 正文。
