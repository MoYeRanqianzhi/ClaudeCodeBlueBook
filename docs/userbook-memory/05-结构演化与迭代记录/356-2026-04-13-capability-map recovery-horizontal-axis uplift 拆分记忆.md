# capability-map recovery-horizontal-axis uplift 拆分记忆

## 本轮继续深入的核心判断

在上一刀把恢复对象 crosswalk 放进：

- `02-能力地图/01-运行时主链/04-会话真相层：实时状态、Transcript 与恢复账本.md`

之后，

当前最值钱的结构优化不再是继续扩 `04/05`，

而是把“连续性不是第六类目录”直接写回：

- `02-能力地图/README.md`

做成横向路由表。

## 为什么这轮值得单独提交

### 判断一：目录层面真正缺的是横坐标，不是新树枝

当前 `02-能力地图`

已经有五个纵向子层，

但当读者问：

- 恢复什么对象
- 消费什么资产
- 不恢复 / 不签什么
- 下一跳该去哪层

答案仍散在：

- 运行时主链
- 执行与工具
- 治理与边界
- 体验与入口

所以这轮最小有效修正，

不是把 continuity 再长成一类，

而是把它正式写成 capability map 的横向坐标。

### 判断二：这一刀比去改子 README 更稳

并行只读分析已经指出：

- `04` 适合承接恢复对象真相
- `05` 适合承接 continuity law
- 任务页适合承接 task recovery 的不对称语义
- 体验层适合承接 viewer / preview / history 这类消费面

所以当前最稳的落点，

就是在总 README 先加一张：

- `恢复对象 -> 消费资产 -> 不恢复什么 -> 下一跳`

的小表，

把四棵子树串起来。

## 这轮具体怎么拆

这轮只做两件事：

1. 在 `02-能力地图/README.md`
   增加 `## 恢复与连续性的横坐标`
2. 用 5 行压平：
   - `resume`
   - `compact`
   - `memory`
   - `task recovery`
   - `viewer / preview / history consumption`

这样：

- 连续性不再被误读成新目录
- 稳定恢复对象与消费面不再混层
- 读者能从总 README 一跳跳到正确子层

## 苏格拉底式自审

### 问：为什么不直接改 `01-运行时主链/README.md` 做横坐标？

答：因为那样横轴会被误读成运行时主链内部结构，而不是整个能力地图的跨层路由。当前要优化的是目录结构，不只是某个子层的解释力度。

### 问：为什么要把 `viewer / preview / history` 也放进表里？

答：因为它们正好提供一个反例坐标。没有这行，读者仍可能把“可见的 continuation 消费面”误听成“正式 restore”。加上这行，稳定恢复对象与消费面才真正分家。

### 问：为什么这轮不顺手去给各子 README 加 echo？

答：因为这会把“总 README 补横轴”和“各子树局部 handoff”混成一刀。先把总路由写清，再决定是否需要在子 README 做一行 echo，边界更稳。
