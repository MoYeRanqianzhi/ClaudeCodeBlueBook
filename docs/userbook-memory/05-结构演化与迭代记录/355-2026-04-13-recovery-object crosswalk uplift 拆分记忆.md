# recovery-object crosswalk uplift 拆分记忆

## 本轮继续深入的核心判断

这一轮先在当前 `userbook` worktree 内把分支快进到最新 `main`，

然后不扩成新的恢复总图，

只在：

- `02-能力地图/01-运行时主链/04-会话真相层：实时状态、Transcript 与恢复账本.md`

把已经存在的“恢复对象谱系”

补成一张最小横向 crosswalk。

## 为什么这轮值得单独提交

### 判断一：当前缺的不是新的对象，而是“同一坐标系”

前一轮已经把：

- `resume`
- `compact`
- `memory`
- `task recovery`

作为四类恢复对象点出来了。

但读者仍要自己从散点段落里拼出：

- 它主要消费什么资产
- 它恢复什么对象
- 它不恢复什么

所以当前最高价值的最小修正，

不是再扩写更多 why，

而是给这四类对象补一个并排可比的横轴。

### 判断二：这刀应落在“会话真相层”，而不是继续膨胀 continuity law 页

`05-压缩、恢复与记忆：如何维持可继续工作的现场.md`

更适合承担：

- `lawful forgetting`
- `reload law`
- `continuation qualification`

这一层的职责。

而“恢复对象分别消费什么、恢复什么、不恢复什么”

本质上仍是对象分类问题，

更应该落在：

- 恢复账与实时账如何分账的真相层页面

而不是把 continuity 页继续写成大总表。

## 这轮具体怎么拆

这轮只做一件事：

1. 在 `## 恢复对象谱系` 下先补一句总引导：
   - 最小 crosswalk 只看资产、恢复对象、不恢复什么
2. 再把四类对象横向压平：
   - `resume`
   - `compact`
   - `memory`
   - `task recovery`

这样：

- `resume` 不再只被读成“回来继续”
- `compact` 不再只被读成“压短上下文”
- `memory` 不再只被读成“附加备注”
- `task recovery` 也不再继续漂回 `resume` 的影子里

## 苏格拉底式自审

### 问：为什么不直接改成表格，甚至再拆一个“恢复对象总览”新页？

答：因为当前缺的是横向坐标，不是新的目录层。先把同一坐标系嵌回已有主文，收益高、风险低、提交边界也最清楚。

### 问：为什么不顺手改 `05` 或任务恢复页，把三页一起对齐？

答：因为这会把“对象横轴补齐”和“连续性法则页/任务页的局部叙事”混成一刀。当前最稳的做法是只在真相层补横轴，让其他页继续承担各自的纵深。

### 问：这条 crosswalk 真能提升稳定/灰度边界吗？

答：能。它至少防止三种常见混写：

- 把 `resume` 与 `task recovery` 写成同一种恢复
- 把 `memory` 写成当前 authority 的代签人
- 把 `compact` 写成 runtime 当前态的回填器
