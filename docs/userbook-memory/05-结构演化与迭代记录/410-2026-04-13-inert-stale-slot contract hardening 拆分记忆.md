在当前 worktree 已经把：

- `100`
- `101`
- `102`
- `103`
- `104`
- `208`

这条 `summary / tail / restore` 线的分叉关系写清之后，

这一支眼下更值钱的下一刀，

不是继续为 `100-104` 新建 hub，

而是回到：

- `104-headless print 的 deferred suggestion staging 会留下 inert stale slot.md`

把它对：

- `cleanup asymmetry`

- `visible protocol bug`

之间边界的证明再压硬一层。

同时，

`208`

- `稳定可见`

三层口径，

而不是留在更早一版的命名上。

## 为什么这轮值得单独提交

### 判断一：当前更缺的是 “孤立 slot 为什么还不能升级成外部后果” 的证据厚度

`104`

之前已经能证明：

- `pendingSuggestion` / `pendingLastEmittedEntry` 清理存在不对称

但还容易被读者夸大成：

- 只要 orphan slot 还在，就迟早会重新长成 ghost suggestion

这轮补的关键，

就是把同一段控制流拆成三个更底层的问题：

1. 它还有没有独立 promote gate？
2. 它还有没有独立 emit 路径？
3. 它还有没有独立 settlement sink？

这样才不会把：

- hygiene asymmetry

误写成：

- visible protocol corruption

### 判断二：目录结构此时已经不是瓶颈，继续裂 hub 的 ROI 太低

当前这条线已经同时有：

- 物理入口 README
- `208` 这种分叉拓扑页

再给 `100-104` 加一个新 hub，

只会重复导航，

不会增加正文的合同厚度。

所以这轮结构优化的动作，

不是新增入口，

而是继续压缩命名漂移，

让结构页和正文页共享同一套三层口径。

### 判断三：这轮也在保护“结构页负责分诊，正文页负责证明”的职责分工

`208`

这页值钱的地方，

是告诉读者：

- 哪些页是主干
- 哪些页是侧枝
- 哪些 helper 名只能当证据，不能被升级成稳定能力名

这轮把它的三层命名同步到：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

本质上是在继续收紧结构页的职责边界。

## 这轮具体怎么拆

这轮只做四件事：

1. 修改 `104-...inert stale slot.md`
   - 给“更像 inert，不像 visible bug”补一层 gate / emit / settlement 三问
2. 统一 `104` 的尾部命名
   - 从较早版本的“稳定、条件与内部边界”
   - 改成 `稳定层、条件层与灰度层`
   - 并补上“结论应停在这里”的收束句
3. 修改 `208-...100-104 不是并列细页...md`
   - 让结构页表格继续复用当前成熟的三层命名
4. 新增这条 `410` 记忆并更新 `memory/05 README`

## 苏格拉底式自审

### 问：为什么这轮不是继续去做 `206` 或别的专题出口页？

答：因为 `206` 当然仍值得做，但这轮上下文已经完整落在 `100-104/208` 这条链上；先把 `104` 这处最容易被夸大成外部 bug 的边界压硬，更适合独立提交。

### 问：为什么这轮不顺手给 `100-104` 再做一个子家族 hub？

答：因为读者现在更缺的不是怎么找到 `104`，而是怎样不把 orphan staging slot 直接写成 ghost suggestion。这里缺的是论证厚度，不是入口。

### 问：为什么只改 `208` 的口径也算结构优化？

答：因为结构优化不只等于新增 README。当前统一结构页和正文页的分层命名，本身就在减少目录系统里的术语漂移。
