## 为什么这轮该补专题层入口

### 判断一：`04-专题深潜/README` 已经承担“高频复杂工作 first-hop”职责

这页前面已经显式抬了：

- `安全放大工作面`
- `多前端接续与条件远端`
- `启动时决定什么，进会话后决定什么`

说明它并不是纯目录列表，

而是在做：

- 高频复杂工作对象的 first-hop routing

既然 `/status`、`/doctor`、`/usage`、`/config`、`/model`、`/effort`

这条线也已经具备：

- 独立阅读路径
- 专题主文
- 控制面页

它就应该在专题层得到同级 first-hop。

## 为什么这里先落到路径 16，而不是只给专题 10

### 判断二：专题层读者先需要分对象，再需要看节奏运营

`04-专题深潜/10`

擅长回答的是：

- 长任务如何看状态
- 模型与 effort 如何配合节奏
- 预算如何影响推进

但如果用户当前问题是：

- `/status`
- `/doctor`
- `/usage`
- `/config`
- `/model`
- `/effort`

到底是不是同一种对象，

直接送去专题 10 还是太晚。

所以专题层的 first-hop 仍应先给：

- 路径 16

然后再把：

- 专题 10
- 控制面 06

作为后继分流。

## 为什么这轮不改控制面 README

### 判断三：当前缺的是工作目标层，不是控制面目录层

这条线在控制面目录里本来就有：

- `06`
- `10`

控制面层的可发现性不算差。

真正缺的是：

- 从“我遇到一个复杂工作对象”这个角度进入时
- 专题层还没把它当成显式第一跳

所以这轮只补：

- `04-专题深潜/README`

不去同时改：

- `05-控制面深挖/README`
