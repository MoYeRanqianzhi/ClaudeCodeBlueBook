# safe action-radius stable-gray alignment 拆分记忆

## 本轮继续深入的核心判断

“安全放大工作面”前门补出来以后，

下一层最容易复发的问题不是：

- 有没有前门

而是：

- 前门页和深页对稳定面、条件面、灰度面的分桶是否一致

这轮只读复核说明，真正该补的不是新入口，

而是：

- 口径对齐

## 为什么 `/add-dir` 的 local-settings 持久度不能被写成条件能力

### 判断一：稳定合同里的“有两种选择”，不等于条件能力

`/add-dir` 已经是稳定可见入口。

它在实现上区分：

- 当前 session 生效
- 写入 local settings

这两种是稳定合同内的持久度选择，

不是：

- feature-gated 能力
- 某些环境才公开的隐藏分支

如果把这个选择写进“条件公开”，

读者就会把：

- “这是正式能力里的两个持久度”

误读成：

- “这可能不是稳定面”

所以这轮把它明确收回到稳定面。

## 为什么 teammate 审批联动和 leader handoff 不能落在同一桶里

### 判断二：条件前提与内部编排不是一回事

前门页只需要保留读者真正会遇到的条件前提：

- teammate 里的计划审批联动
- 某些 channel / Git / worktree 上下文前提

而：

- leader approval handoff
- remote managed settings

更接近内部编排或宿主治理实现。

如果把它们和条件前提混写，

读者会在入口层重新失去判断：

- 这到底是“公开但有前提”
- 还是“内部/灰度细节”

所以这轮把两者拆开。

## 为什么旧页也必须补 `/add-dir`

### 判断三：前门是四分，深页也不能退回三分

前门页已经把动作固定成：

1. `/permissions`
2. `/plan`
3. EnterWorktree
4. `/add-dir`

如果 `04-专题深潜/03` 与 `05-控制面深挖/01`

仍把稳定面只写成：

- permissions
- plan
- worktree

读者一进长文就会退回旧口径，

仿佛 `/add-dir` 仍只是边上的上下文技巧。

所以这轮至少要把稳定面名单对齐，

即便 `05-控制面深挖/01` 仍保持“三层治理链”的主叙事，

它的稳定面总结也必须承认：

- `/add-dir` 是同一条安全放大使用线里的稳定入口

## 本轮最关键的新判断

### 判断四：入口切片补完后，下一优先级是“稳定/条件/灰度口径一致”

这说明后续继续深入时，

每补一个物理前门，都要立即追问：

- 前门页的稳定/条件/灰度分桶
- 旧总览页的稳定/灰度总结
- 深页的稳定面总结

是不是仍在说同一种对象分层。

否则前门虽然出现了，

但读者一旦继续深入，

仍会在不同层级里撞到互相打架的口径。

## 苏格拉底式自审

### 问：我是不是把“稳定功能中的不同选择”写成了“条件功能”？

答：如果用户总能看到这个入口，只是运行时再选持久度，那它仍属于稳定面。

### 问：我是不是把“用户能遇到的前提”与“系统内部怎么编排”混成一桶了？

答：如果读者不需要靠它决定第一跳，它就不该占用前门页的条件面位置。

### 问：我是不是让前门说四分、深页说三分？

答：只要深页稳定面不承认 `/add-dir`，这个回退就会复发。
