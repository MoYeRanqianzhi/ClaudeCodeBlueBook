# root automation route narrowing 拆分记忆

## 本轮继续深入的核心判断

这一轮最值得收的，

不是继续手写根 `README` 里的自动化/协议流局部路线，

而是把：

- `把 Claude Code 接进脚本、后台任务或协议流`

这一条

收回到已经存在的：

- `路径 9`

上。

## 为什么这轮值得单独提交

### 判断一：当前问题不是缺路线，而是根前门又在重复维护一条局部自动化路线

根 `README` 当前这条写法是：

- `04-专题深潜/13-非交互、后台会话与自动化专题.md`
- `05-控制面深挖/非交互结果、summary 与协议流/README.md`

但 `00-阅读路径.md`

已经存在：

- `路径 9：我想把 Claude Code 接进脚本或后台任务`

而且它不只是这两跳，

还把：

- 启动入口
- headless / StructuredIO
- direct connect / 协议流后续页

串成一条更完整的顺序路径。

所以根前门这里真正的问题是：

- 手写了一条比正式路径更窄、也更容易滞后的局部路线

### 判断二：目标入口应只负责把读者送到权威路径，而不是再局部复刻路径中的几步

这条目标本身属于：

- 自动化 / 非交互 / 协议流

天然会跨：

- 主线进入
- 非交互专题
- 控制面深挖

如果根前门继续自己写两跳，

就会不断出现：

- 只列部分节点
- 漏掉正式路径中的前置或后续

更稳的做法是：

- 根前门只做目标分诊
- 正式顺序交给阅读路径页

## 这轮具体怎么拆

### 判断三：把手写两跳收成对 `路径 9` 的单点 handoff

这轮只做一件事：

1. 把根 `README` 里这条从：
   - `04-专题深潜/13 ... -> 05-控制面深挖/非交互结果、summary 与协议流/README.md`
   收成：
   - `00-阅读路径.md` 里的“路径 9：我想把 Claude Code 接进脚本或后台任务”

这样：

- 根前门不再重复维护自动化局部路线
- 自动化目标重新回到阅读路径层统一编排
- 后续如果 `路径 9` 调整，根前门不需要跟着手动改节点顺序

## 为什么这轮不顺手改“把结果交付给团队或外部流程”

### 判断四：那是另一条目标路径，不该和自动化入口混成同一刀

虽然：

- `把结果交付给团队或外部流程`

也可能继续拆成更精确路径，

但它和当前这条的对象主语不同：

- 一个偏自动化/协议流宿主
- 一个偏评审/提交/导出/反馈

如果这轮一起动，

就会把：

- 自动化路线去重

扩大成：

- 根前门多目标入口联动重写

超出当前最小提交边界。
