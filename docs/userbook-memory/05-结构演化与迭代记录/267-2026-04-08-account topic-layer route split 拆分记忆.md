## 为什么这轮该补专题层 route

### 判断一：顶层和路径层已经存在，但专题层还没把账户/隐私/升级边界抬成 first-hop

当前用户已经可以在：

- 顶层 README
- 路径 89

看到这条线，

但进入 `04-专题深潜`

之后，

它还只是平铺列表里的：

- `15-账户、隐私、资格与升级专题`

没有像：

- context intake
- delivery
- terminal efficiency
- init/setup

那样被抬成 first-hop。

所以这轮缺的是：

- topic-layer discoverability

不是：

- 再写正文

## 为什么 route 文案必须先保护“消费者订阅 / API 计费 / 3P provider”三分

### 判断二：账户线最容易被写平的，不是按钮名，而是身份主语

`15`

正文已经明确：

- `claude.ai` 消费者订阅身份
- API 计费身份
- 3P provider 身份

会直接决定：

- `/login`
- `/privacy-settings`
- `/passes`
- `/upgrade`
- `rate-limit-options`

这些入口的可见性与语义。

如果专题层 route 只说：

- “去看账户专题”

读者仍然会继续把：

- 身份切换
- 消费者产品设置
- 资格与激励
- 升级与限流分流

压成同一类“账户设置”。

所以这轮 route 的第一句必须先把问题压成：

- 我是 `claude.ai` 消费者订阅、API 计费身份，还是 3P provider 身份

## 为什么这轮先给路径 89，再给专题 15

### 判断三：专题层 first-hop 回答“先去哪”，路径层仍然负责“按什么顺序继续”

`15`

正文已经能回答对象边界，

但如果读者此时还没分清：

- 先理解身份与资格边界
- 再看额度、模型与节奏分流
- 最后再看哪些入口只是隐藏或资格依赖

直接只送专题 15，

仍会跳过路径 89 已经给出的顺序价值。

所以这轮更稳的顺序应当是：

- 先给路径 89
- 再给专题 15
- 再把后继分叉接到 `10` 和“最易误判的能力清单”

## 为什么这轮顺手接 `10` 和能力边界清单

### 判断四：账户线继续深入时，最常见的分叉就是“额度运营”与“资格/隐藏边界”

这条线真正继续深入时，

最常见的两个后继问题分别是：

- hit rate limit 之后该怎样分流到额度、模型与节奏运营
- 哪些入口只是资格依赖、产品面依赖或隐藏分流

所以这轮 route 不能只停在 `15`，

还应显式把：

- `10-状态、额度、模型与节奏运营专题`
- `03-参考索引/02-能力边界/04-最易误判的能力清单`

接出来。

## 为什么这轮不先补矩阵或合同速查

### 判断五：对已经进入专题层的读者来说，当前更直接的物理断点仍然是 first-hop 缺失

路径 89 和正文 15 都已经存在，

但在专题总览层还没有一个第一跳告诉读者：

- 这条线已经成形
- 它和其他高频工作一样值得单独 first-hop

所以这轮先补：

- `04-专题深潜/README`

不是：

- `03-参考索引/05`
- `03-参考索引/06`

答：对已经进入专题层的读者来说，first-hop 缺口更近，所以应先补 route。
