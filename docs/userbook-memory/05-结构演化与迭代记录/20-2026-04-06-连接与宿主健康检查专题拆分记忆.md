# 连接与宿主健康检查专题拆分记忆

## 本轮继续深入的核心判断

上一批已经把：

- headless 启动链
- `StructuredIO`
- 首问前 ready 合同

拆开了。

但仍缺一批专门回答：

- `open <cc-url>`
- `assistant [sessionId]`
- `doctor`
- `mcp list|get`

为什么它们虽然都在会外层，却不是同一类入口。

如果不拆，正文会继续犯三种错：

- 把连接入口写成普通用户主线
- 把 viewer client 写成 daemon
- 把健康检查入口写成纯只读命令

## 苏格拉底式自审

### 问：为什么这批不继续塞进 headless 那一页？

答：上一页回答的是：

- non-interactive host 如何装配
- 首问前为什么选择性等待
- `print` 为什么不是 REPL 去 UI

本轮的问题已经变成：

- 当前进程是不是在连接另一个宿主
- 当前 REPL 是不是 viewer client
- 当前 root command 是不是健康检查型读取

这已经不是 headless 主链，而是另一类“宿主对象边界”。

### 问：为什么最初放进 `04-专题深潜`，最后又改放 `05-控制面深挖`？

答：一开始我把这批理解成：

- 用户如何连接
- 用户如何接续
- 用户如何诊断

所以先放进了 `04`。

但继续看结构后发现，真正的核心问题并不是“我要做什么”，而是：

- `server`、`remote-control`、`assistant`、`doctor`、`mcp list|get`
- 虽然都在会外层
- 却属于完全不同的对象类型

这已经是典型的横切控制面问题：

- host
- viewer
- health-check

因此最终改放：

- `05-控制面深挖`

而不是继续放在 `04`。

### 问：正文里最该避免的假并列是什么？

答：

- `server` / `cc://` 的 direct-connect 公共面 vs bridge / remote-control
- `cc://` vs internal `open`
- `assistant [sessionId]` vs `--assistant`
- viewer client vs daemon
- `doctor` 文案里的 updater 健康 vs 实际 runtime 检查对象
- `mcp list|get` 的 config 读取 vs 实际连接健康检查
- `mcp list` / `doctor` 的 merged-config 视角 vs `mcp get` 的按名直取视角

## 正文落笔纪律

### 应写进正文的

- `cc://` 的交互与 headless 分流
- `open <cc-url>` 是 internal transport shim
- `assistant [sessionId]` 是 viewer client，不是 daemon
- `doctor` 与 `mcp list|get` 都带 trust skip / health check 副作用
- `doctor` 实际检查对象比帮助文案更宽
- `mcp list|get` 是“健康检查型读取”，不是纯静态读取

### 不应写进正文的

- `parseConnectUrl`、`sessionDiscovery` 等缺失文件的未证实细节
- Direct connect / bridge transport 的低层协议字段
- install wizard、chooser、remote history paging 的完整实现细节
- `useRemoteSession` 的所有超时与 reconnect 细节

这些应只作为作者判断依据，不应回流到用户正文。

## 本轮特殊注意

### 文件缺口必须降级表述

当前快照里：

- `parseConnectUrl.js`
- `assistant/sessionDiscovery.js`

都只有引用点，没有实现文件。

因此正文只能写：

- 可从注册点和调用点稳定确认的对象边界

不能写：

- 这些模块内部如何完整解析 URL 或发现 session

### `doctor` 的对象漂移值得保留

帮助文案写的是：

- auto-updater 健康

但 handler / screen 实际已经覆盖：

- plugin
- MCP
- context warnings
- sandbox

这是一条很高价值的“描述对象 vs runtime 对象”漂移证据，适合进入正文。

### `mcp list|get` 的副作用必须前置

它们最容易被用户误判成：

- 只读配置命令

正文必须把：

- 连接检查
- stdio server spawn
- cleanup / gracefulShutdown

放到前面写，而不是藏到小注释里。

## 后续继续深入时的检查问题

1. 我现在讲的是用户任务主线，还是宿主连接 / viewer / 健康检查对象？
2. 这里的入口是公开主线、条件公开，还是内部宿主实现？
3. 这里读取的是静态配置，还是已经触发了活对象检查？
4. 我是不是把 `assistant [sessionId]` 和 `--assistant` 混成了一层？
5. 我是不是把缺失文件里的实现脑补成了稳定事实？

只要这五问没先答清，正文就会把“连接、附着、健康检查”重新写回同一种普通 root command。
