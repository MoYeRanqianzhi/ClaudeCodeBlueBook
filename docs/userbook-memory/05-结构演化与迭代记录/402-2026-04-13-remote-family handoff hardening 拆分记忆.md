# remote-family handoff hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经补完：

- `memory/05` 根 `README` 的暴露层级收窄
- `122-149 -> 139-149 -> 144-149`
  的 truth-shell handoff
- `122-149 -> 128-138 -> 133-138`
  的 surface-shell handoff

之后，

当前最值钱的下一刀，

不是继续回根索引加入口，

也不是转去 `191-206` 再造更细导航，

而是把：

- `100-149-非交互、recovery 与 remote surface 家族/README.md`

补成一张真正承担：

- `100-149 -> 122-149`

family-level handoff 的父 hub。

## 为什么这轮值得单独提交

### 判断一：当前已经不是“子家族有没有”，而是上层 family hub 还没把 remote 半段交出去

`122-149`

这条子家族已经存在，

而且它下面的：

- `139-149 -> 144-149`
- `128-138 -> 133-138`

两条 handoff 也都补齐了。

这意味着 remote 这一整半段的树内连通，

其实已经足够细。

当前缺口反而在更上一层：

- `100-149`
  还没有明确告诉读者，
  当你已经确认自己在追的是 remote half，
  下一步就直接进入 `122-149`

于是读者虽然已经进了对的 family，

却还得自己猜：

- 现在是不是该再去找 `122-149`

### 判断二：目录优化在这一轮表现为“父层交接”，不是“继续开新层”

如果这时回根索引再挂更多入口，

会重新把已经收窄好的 root exposure 写宽。

如果这时转去给 `191-206` 再补更细导航，

则会跳过当前这条更明显的未闭合链：

- `100-149`
  还没有把 remote half 交给
  `122-149`

更稳的最小动作其实是：

1. 在 `100-149` 里明说：
   - 这里负责把读者分到 `100-115`、`116-127`、`128-149`
2. 在 `128-149` 这半段的小节里
   明说：
   - 如果你已经确认自己要在 remote recovery / surface / truth / shell 这半段继续下钻，就直接进 `122-149`

这样并没有增加目录层数，

只是把父层该做的 handoff 补完整。

### 判断三：`191-206` 当前仍更适合保持 family hub + 正文厚度，而不是抢这一刀

`191-206`

现在当然也是一条完整 family，

但它的结构更像：

- `197` ingress 阅读链
- `203` permission tail 分叉图
- `206` blocked-state ceiling

配一层：

- `222`
  stable-gray hardening

这条线当前更适合继续守住：

- family hub 已足够
- 正文厚度优先

而不是为了和 remote 链条对称，

机械地再去补新的入口动作。

所以本轮更值钱的，

还是把已经明显闭合到一半的：

- `100-149 -> 122-149`

补完整。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `100-149` family hub
   - 增加“从根索引或时间流进来后，这里负责完成 `100-115 / 116-127 / 128-149` 分流”的说明
2. 在 `128-149` 小节里
   - 增加直达 `122-149` 子家族的显式入口
3. 新增这条 `402` 记忆
   - 记录为什么这轮优先补 family-level handoff，而不是回根索引或跳到 `191-206`

这样：

- `root -> 100-149 -> 122-149 -> 139-149/128-138`
  会形成连续的 hub handoff 链
- 根入口的收窄不会损失 findability
- 目录结构优化也继续和第一性原理一致：上一层只把你送到下一层，不替下一层继续说话

## 苏格拉底式自审

### 问：为什么这轮不是继续让根索引重新挂更多 remote 子家族？

答：因为那会把根入口重新写宽。当前问题不是找不到更深子家族，而是 `100-149` 还没明确承担把 remote half 送到 `122-149` 的责任。

### 问：为什么这轮不是转去做 `191-206` 的新导航？

答：因为 `191-206` 当前没有同等级的 handoff 缺口。它更像“family hub 已够、正文厚度优先”的范围；而 `100-149` 的父层交接缺口已经很明确。

### 问：为什么这也算“加强对使用功能的剖析”？

答：因为 userbook 的使用剖析依赖逐层收窄。`100-149` 如果不能明确告诉读者什么时候该进 `122-149`，读者就会重新退回按时间流碰运气，而不是沿着 recovery / surface / truth / shell 的能力边界下钻。
