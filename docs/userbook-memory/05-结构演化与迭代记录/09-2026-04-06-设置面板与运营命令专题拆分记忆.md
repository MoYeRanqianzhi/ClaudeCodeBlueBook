# 设置面板与运营命令专题拆分记忆

## 本轮继续深入的核心判断

之前的 userbook 已经分别写过：

- 节奏运营
- 运行时自检
- root/slash 双层入口

但还缺一层专门回答“这些运营入口到底属于哪种 UI/控制面对象”。

如果这一层不拆，正文会持续把四种对象混成一团：

- Settings tab
- 独立诊断 / 统计 screen
- session 调参命令
- 预算 / 限流分流入口

## 苏格拉底式自审

### 问：为什么不继续往 `10-状态、额度、模型与节奏运营专题` 里加？

答：因为 `10` 的对象是“任务节奏如何运营”，不是“这些入口属于哪种 UI/控制面类型”。继续往里加，会让“对象分层”再次被“策略建议”淹没。

### 问：为什么不只在索引页修一下？

答：因为这里只做索引不够。读者真正会犯错的，是把 `/status`、`/usage`、`/config`、`/doctor`、`/stats`、`/statusline` 当成一类命令；这需要一页长文把对象差异讲透。

### 问：哪些地方最容易制造假并列？

答：

- `/status`、`/config`、`/usage`
- `/usage`、`/cost`、`/stats`
- `/model`、`/effort`、`/fast`
- `/statusline` 与所有“状态命令”
- `/extra-usage` 与 `/usage`

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- 为什么这批要按“UI/控制面对象”而不是按“用户策略”继续拆
- “还缺哪一层控制面没独立出来”的维护判断
- 作者如何决定哪些邻接入口只是邻接，而不是同一类对象

这些都是维护知识，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- 哪些入口共享同一容器
- 哪些入口是独立 screen
- 哪些入口在改 session 参数
- 哪些入口在做预算与限流分流

### 不应写进正文的

- 作者为何觉得旧结构“对象层还不干净”
- 下轮还想继续拆哪些控制面
- 目录命名的犹豫和维护策略

这些继续只放 `docs/userbook-memory/`。

## 本轮边界保护

### 应明确写成 Settings tabs 的

- `/status`
- `/config`
- `/usage`

### 应明确写成独立 screen 的

- `/doctor`
- `/stats`

### 应明确写成 session 调参的

- `/model`
- `/effort`
- `/fast`

### 应明确写成显示层或受控 prompt 工作流的

- `/theme`
- `/color`
- `/statusline`

### 应明确写成预算 / 限流分流的

- `/cost`
- `/extra-usage`
- `/rate-limit-options`

还要继续强调三条写作纪律：

- `/usage` 的 `Current session` 不要误写成当前 `sessionId`
- `/stats` 不要误写成 `/usage` 的长期版
- `/cost` 不要误写成订阅用户的实时美元账单

## 后续继续深入时的检查问题

1. 这条入口是在展示当前状态，还是在改变当前会话？
2. 它属于同一个容器里的 tab，还是完全独立的 screen？
3. 它是在回答“现在是什么”，还是在回答“还能不能继续工作”？

只要这三问没先答清，正文就会继续制造假并列。
