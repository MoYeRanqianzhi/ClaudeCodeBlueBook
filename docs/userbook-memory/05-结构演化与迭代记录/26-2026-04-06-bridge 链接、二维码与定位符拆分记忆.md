# bridge 链接、二维码与定位符拆分记忆

## 本轮继续深入的核心判断

第 25 页已经拆开了：

- 默认策略面
- 当前显式控制面
- 状态展示面
- 查看/断开面

但 bridge 相关正文里仍缺一层更细的对象边界：

- connect URL
- session URL
- environment ID
- session ID
- `remoteSessionUrl`

如果不补这一批，正文会继续犯六种错：

- 把 connect URL 写成“当前会话链接”
- 把 session URL 写成“唯一远控地址”
- 把 environment ID 与 session ID 当成一回事
- 把二维码目标变化写成 UI 偶然现象
- 把 `remoteSessionUrl` 与 `replBridgeSessionUrl` 写成同一对象
- 把 URL / QR / ID 三种展示形式重新压成一个“远控链接”

## 苏格拉底式自审

### 问：为什么这批不能塞回第 25 页？

答：第 25 页回答的是：

- 哪个 surface 在改什么对象

而本轮问题变成：

- 这些 surface 里展示出来的定位符，各自到底在指向什么

这已经从：

- surface boundary

继续下钻到：

- locator boundary

所以应该新起一页，而不是继续把 25 页拉厚。

### 问：为什么这批值得进入正文，而不是只留在 memory？

答：因为这是用户立即可感知的混淆：

- 为什么 dialog 有时给我环境入口，有时给我当前会话？
- 为什么二维码在不同模式下指向不同对象？
- 为什么 verbose 里有两个 ID？
- 为什么 `/session` 打开的链接和 bridge dialog 里的链接不是同一条？

这些都属于使用层误判，不是作者内部推理。

### 问：这批最该防的假并列是什么？

答：

- connect URL = session URL
- URL = QR = ID
- `remoteSessionUrl` = `replBridgeSessionUrl`

只要这三组没先拆开，bridge 的定位对象就会重新糊掉。

## 正文落笔纪律

### 应写进正文的

- `connectUrl` 指 environment，不指当前 session
- `sessionUrl` 指当前附着 session，不指 environment 入口
- single-session 与 multi-session 会改变 QR 的目标对象
- `environmentId` / `sessionId` 只应作为调试定位符出现
- `remoteSessionUrl` 属于 `--remote` 模式，不属于 always-on bridge 那组字段
- headless / control surface 同时返回 `connect_url` 与 `session_url`，说明这不是 UI 偏好而是对象边界

### 不应写进正文的

- 所有赋值时机的完整状态流
- v1 / v2 env-less 协议的完整细枝末节
- source-map 展开的编译噪音
- 所有 UI 渲染小细节

这些内容只作为作者判据，不回流正文。

## 本轮特殊注意

### “都是能点开的东西”不等于“是同一种定位符”

本轮最容易犯的偷换是：

- 既然用户最终都只是“打开一个链接”

于是误写成：

- connect URL、session URL、`remoteSessionUrl` 本质一样

但源码明确显示：

- 它们定位的对象不同
- 进入的 surface 也不同

正文必须保留这一层分界。

### “二维码显示的内容”也要按对象边界解释

如果正文只写：

- 可以扫码继续

就会遗漏最关键的问题：

- 扫的是 environment 入口，还是当前 session

这正是本轮专题的第一性原理，不应被 UI 描述吞掉。
