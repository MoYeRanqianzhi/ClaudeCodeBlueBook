`启动入口分流`

这条线在顶层入口、阅读路径和专题页都补出来以后，

当前最容易残留的不是目录缺口，

而是：

- 专题 18 仍默认把“同名入口错位”留给读者自己跳去专题 19

## 为什么 `01-进入系统` 必须把 `-p/--print` 抬到入口枚举里

### 判断一：非交互不是附属模式，而是和普通 CLI 并列的启动形态

`01-进入系统`

原来在第一段入口清单里列了：

- 普通 CLI / REPL
- `remote-control` / `bridge`
- `daemon`
- 后台会话管理
- runner
- `--worktree --tmux`

却没有把：

- `-p/--print`

直接列成同层入口。

这会产生一个轻微但稳定的误导：

- 启动入口首先是交互 / 宿主 / 后台分流
- 非交互只是后文补充的一种模式

但从源码和用户实际使用看，

`-p/--print` 明显是：

- first-class startup shape

所以这轮把它直接抬回第一段入口枚举。

## 为什么这里还要顺带补“回路径 14”

### 判断二：主线第一页不该自己承担启动层全部对象分诊

`01-进入系统`

是主线起点，

它应该承认：

- 这页负责建立入口对象感

但不该假装自己已经足够承接：

- root flags
- root commands
- slash commands

三层分流的全部分析。

所以更稳的做法是：

- 在这页最前面补一个很短的 handoff
- 把需要完整对象分流的读者送回 `路径 14`

## 为什么 `04-专题深潜/18` 还要回指 `19`

### 判断三：专题 18 已经讲清启动层，但没有把“同名入口错位”明确交给专题 19

专题 18 主要回答：

- 进程启动层
- 会话运行层
- root flags
- root commands
- 隐藏 / 灰度 flags

但不少用户真正卡住的具体问题是：

- `claude auth` 与 `/login`
- `claude mcp` 与 `/mcp`
- `claude plugin` 与 `/plugin`
- `claude doctor` 与 `/doctor` `/status`

这已经不是“启动 flags 分流”本身，

而是：

- 会外 root commands
- 会内面板

的同名错位。

专题 19 已经负责这个对象层，

所以专题 18 应该显式 handoff 过去，

不让读者在 18 里自己横跳。
