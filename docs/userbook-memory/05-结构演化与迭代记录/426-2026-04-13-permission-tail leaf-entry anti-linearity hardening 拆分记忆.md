# permission-tail leaf-entry anti-linearity hardening 拆分记忆

## 本轮继续深入的核心判断

在上一轮已经把 `203`

钉成：

- successor skeleton
- canonical topology

之后，

permission tail 这簇剩下的更细风险，

不再是总图有没有写出来，

而是：

- leaf 页页首会不会继续用“上一页已经……”的口气，
  把 sibling branch 重新误听成线性下一步

当前最明显的两处就是：

- `199`
- `201`

它们都借 `198` 起笔，

这在正文逻辑上是对照，

但在页首路由上仍可能被误听成：

- `198 -> 199`
- `198 -> 201`

## 为什么这轮值得单独提交

### 判断一：总图已经够硬，下一刀该保护 leaf 页入口句不要反向冲掉 canonical 拓扑

`203`

现在已经明确写成：

- `196 -> 198 / 199 / (201 -> 202)`

所以当前最高 ROI，

不再是回头继续补总图，

而是确保 leaf 页自己的页首不要继续制造：

- “上一页讲完，这一页顺着往下”

这种伪线性承接。

### 判断二：`199` 和 `201` 的问题不在正文主语，而在页首借反例起笔时没有先声明“这是对照，不是主链”

`199`

正文主语其实一直很稳：

- leader-local re-evaluation

`201`

正文主语也很稳：

- host-level sibling sweep

但它们页首都先说：

- `198` 已经……

如果缺一条显式提醒，

读者就仍可能把：

- `198`

当成编号主链上的必经上一页，

而不是：

- 一个被借来先压住的 closeout 对照页

### 判断三：这轮仍然是在做第一性原理式的位阶保护

从第一性原理看，

真正该先问的不是：

- 这些页前后都引用了谁

而是：

- 当前页首引用上一页时，是在声明 canonical 拓扑，还是在借反例划清主语

只要这层不先说清，

leaf 页就会继续把：

- 结构分叉
- 对照关系

重新压成同一种“顺着编号继续读”的错觉。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `199-...re-evaluation surface.md`
   - 在页首补一句：这不表示 `199` 是 `198` 的线性下一步
   - 明确 `198` 与 `199` 都回挂到 `196` 之后，只是这里借 `198` 当 closeout 反例
2. 修改 `201-...sibling cleanup.md`
   - 在页首补一句：这不表示 `201` 是 `198` 的 sandbox 版线性下一步
   - 明确 `201` 回挂到 `196` 之后的 sandbox-network 分支，只是借 `198` 当反例
3. 新增这条 `426` 记忆并更新 `README`
   - 记录为什么这轮开始保护 leaf-entry anti-linearity

## 苏格拉底式自审

### 问：为什么这轮不去改 `202`？

答：因为 `202` 现在作为 `201` 之后的 persist zoom 反而是合理承接；当前更容易制造伪线性的，是 `199/201` 这两个 sibling 分叉页。

### 问：为什么这轮不继续回改 `203`？

答：因为 `203` 的骨架与 canonical 公式已经够硬。继续加总图收益低于直接消掉 leaf 页页首对编号主链的误导。

### 问：为什么这也算“优化相关目录结构”？

答：因为结构优化不只在 README/hub。当前是在保护页级入口句和 canonical branch map 的一致性，避免 leaf 页自己把局部目录关系重新写歪。
