## 为什么这轮值得单独提交

### 判断一：当前问题不是缺 handoff，而是正式路径和局部直达入口同时存在

根 `README` 当前已经把这个目标正式路由到：

- `00-阅读路径.md`
- `路径 92：我想安全放大工作面，而不是一上来把行动半径开到最大`

而 `路径 92` 的第 1 步本来就是：

- `04-专题深潜/安全放大工作面/README.md`

所以根页后面再补一句：

- `如果你已经明确要进入这条主题线，再直接看 ...`

实际上是在同时维护：

- 正式路径入口
- 局部直达入口

这会制造两套前门信号。

### 判断二：局部直达入口会把“走完整条路径”和“点开第 1 步”混成同一动作

当前这句最容易制造的误读是：

- 只点 `安全放大工作面/README.md`
  就等于完成了这条目标路线

但正式的 `路径 92`

后面还有：

- 开工前自检
- Add-dir / Sandbox 刷新
- 治理与安全专题
- 权限、计划模式与 Worktree

所以更稳的做法是：

- 只保留路径 92 这一层入口

让完整顺序仍由阅读路径页承担。

## 这轮具体怎么拆

### 判断三：只删局部直达句，不动路径 92 或专题 README 本体

这轮只做一件事：

1. 删除根 `README` 里：
   - `如果你已经明确要进入这条主题线，再直接看 ...`

这样：

- 根前门不再同时维护正式路径和局部入口

- 改动仍保持在单句删除

## 为什么这轮不顺手改“建立稳定工作流”或其它路径型入口

### 判断四：当前最小有效改动只是封住一个重复直达口

这轮问题的关键不是根前门整体太厚，

而是：

- 某一条目标在已经有正式路径后

如果把其它路径型入口一起改，

就会把：

- 单个重复直达口清理

扩大成：

- 根前门整体重排

超出当前最小提交边界。
