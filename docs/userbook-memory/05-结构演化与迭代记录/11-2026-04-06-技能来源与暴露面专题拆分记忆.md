# 技能来源与暴露面专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经分别写过：

- 扩展层如何选
- 插件、MCP、skills、hooks、agents 的运维对象
- 命令对象、执行语义与可见性边界

但仍缺一层专门回答：

- 技能为什么不是一张表
- `/skills`、SkillTool、dynamic skills、MCP skills 为什么看到的是不同投影
- 为什么“当前菜单里没看到”不等于“Claude 不会用”

如果这一层不拆，正文就会持续把四种完全不同的事实混成一类“技能列表”：

- 技能来源
- 用户可见菜单
- 模型可调用集合
- 运行时延迟激活机制

## 苏格拉底式自审

### 问：为什么不把这些内容继续塞回 `17-插件、MCP、技能、Hooks 与 Agents 运维专题`？

答：因为 `17` 的对象是扩展运维工作流，不是技能暴露面的投影结构。继续往里加，会让“怎么运维扩展”与“技能如何进入并暴露到运行时”重新缠在一起。

### 问：为什么不只在 `03-技能与扩展/01-内置技能索引` 里补几句说明？

答：因为真正的误判不是“有哪些 bundled skills”，而是把 bundled、plugin、file-based、MCP、dynamic skills 当成同一种存在方式。只补一页 bundled 索引，解释不了 `/skills` 为什么不是能力全集。

### 问：为什么这一批还放在 `05-控制面深挖`？

答：因为这一批讨论的是“技能暴露面如何被制度性切开”，本质上是控制面问题，不是某个单一用户任务目标。

### 问：正文里最该避免的假并列是什么？

答：

- 技能来源 vs 技能菜单
- 用户可直打 vs 模型可自动调
- 常驻 skills vs `paths:` 条件技能
- 本地 skill dir vs MCP skill thread
- bundled skills vs `/skills` 菜单里的 file/plugin/mcp skills

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- 为什么作者选择“多重投影”而不是“技能分类表”来继续拆
- 哪些 provenance 字段更像实现细节，不应写成稳定产品承诺
- 当前 visible source 里哪些 skill consumer 明显比 producer 更完整

这些都是维护知识，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- 技能从哪些来源进入运行时
- `/skills` 菜单实际筛出来的是什么
- SkillTool 给模型看的集合与用户菜单有什么不同
- `userInvocable`、`disableModelInvocation`、`paths:` 分别切开什么边界
- 文件触达为什么会改变技能面

### 不应写进正文的

- built-in plugin skills 目前为什么要伪装成 bundled 的维护层原因
- provenance 字段未来是否会重命名或重构
- 哪些 consumer 比 producer 先暴露，导致当前 visible source 只能从消费侧反推
- `paths:` 条件技能与运行中动态发现的新目录技能当前不是同一条二次激活链
- bare mode 下 read/edit 与 write 对动态技能发现的守卫并不完全对称

这些继续只放 `docs/userbook-memory/`。

## 本轮边界保护

### 应明确写成来源层的

- file-based skills
- legacy commands-as-skills
- bundled skills
- plugin skills
- built-in plugin skills
- MCP skills

### 应明确写成投影层的

- `/skills` 菜单
- SkillTool 模型集合
- system/init 暴露面
- dynamicSkills / conditionalSkills

### 应明确写成激活层的

- `paths:` 条件技能
- 嵌套 `.claude/skills` 的动态发现
- bare mode
- plugin-only policy / settings source 限制

### 应继续只作为实现细节保留在 memory 的

- 启动期已知技能的 `paths:` 是先入 `conditionalSkills` 再转 `dynamicSkills`
- 运行中动态发现的新目录目前更直接地进入 `dynamicSkills`
- 这些差异足以解释运行时现象，但还不适合直接上升成稳定用户承诺

## 后续继续深入时的检查问题

1. 我现在写的是技能“来源”，还是技能“暴露菜单”？
2. 这是用户直调边界，还是模型自动调用边界？
3. 这是稳定产品心智，还是 provenance / telemetry / truncation 细节？
4. 这条技能是常驻的，还是只有文件触达后才出现？

只要这四问没先答清，正文就会继续把“技能系统”误写成一张总表。
