## 为什么这轮落点选 `03-参考索引/06`

### 判断一：前门、路径和矩阵都已具备，最后缺的是“高价值入口运行时合同速查”

这条线当前已经具备：

- 任务矩阵

但在 `03-参考索引/06`

而：

- 普通提示词 / 附件 / IDE 选区

还没有形成同主题的显式合同块。

所以这轮最小且最高价值的一刀就是：

- `context intake contract-table`

## 为什么这轮要把 `/add-dir` 从 `/status`、`/doctor`、`/usage` 那组里抽出来

### 判断二：工作面扩张不是运行时自检

把 `/add-dir`

放在：

- `/status`
- `/doctor`
- `/usage`

那组下面，

会不断暗示：

- 它更像设置 / 诊断 / 自检按钮

但它真正回答的是：

- 工作面制度承认
- permission context 变化
- sandbox 可访问目录集合变化

这和：

- 宿主状态
- 系统诊断
- 额度约束

不是同一种主语。

所以这轮必须把它抽回：

- context intake 组

## 为什么这轮要把普通提示词、附件与 IDE 选区写成显式合同

### 判断三：不把“任务说明”和“事实载体”分开，输入编译主语会再次丢失

`08`

正文已经写清：

- 普通提示词更适合声明任务、约束、成功标准
- 附件 / 图片 / 粘贴块 / IDE 选区更适合承载事实本体

但如果 `06`

没有把这层显式收成合同块，

读者在压缩层仍然只会看到：

- `/add-dir`

从而误以为：

- context intake 的高价值合同只有工作面扩张

所以这轮必须显式收：

- 普通提示词、附件与 IDE 选区

## 为什么这轮要把 `/files` 与 `/context` 单列

### 判断四：共同事实面核对与材料接入不是同一种动作

在矩阵层，

我们已经把：

- 事实送入
- 事实面核对
- 工作面扩张

拆成三步。

那合同层也必须对应保护：

- `/files`
- `/context`

不是材料接入入口，

而是：

- 共享事实面与上下文压力账本

如果不单列，

读者就会继续把：

- 送材料
- 看账

混成同一种“补上下文动作”。

## 为什么这轮不把终端效率混进这组

### 判断五：送材料进系统与把终端用顺手不是同一种对象

terminal efficiency 回答的是：

- 主题
- 键位
- Vim
- statusline
- prompt 历史

这些本地交互摩擦。

而 context intake 回答的是：

- 事实怎么进入系统
- 共同事实面怎么成立
- 工作面怎么正式扩张

所以这轮合同块必须只收：

- 输入编译与工作面边界

不把：

- 终端交互效率

混回同层。
