# root quickstart route narrowing 拆分记忆

## 本轮继续深入的核心判断

这一轮最值得收的，

不是继续手写根 `README` 里的“快速上手”局部起步路线，

而是把：

- `快速上手`

这一条

收回到已经存在的：

- `路径 1`

上。

## 为什么这轮值得单独提交

### 判断一：当前问题不是缺入口，而是根前门又在重复维护一条比正式路径更窄的起步路线

根 `README` 当前把“快速上手”写成：

- `00-导读.md`
- `01-主线使用/README.md`

但 `00-阅读路径.md`

已经存在：

- `路径 1：我只想尽快上手`

而且它不止这两步，

还继续覆盖：

- 开工前自检
- 代码工作流
- 上下文接入
- 连续性与记忆

所以根前门这里真正的问题是：

- 又手写了一条更窄的局部路线

### 判断二：快速上手这类目标入口应直接交给阅读路径页，而不是在根前门里局部复刻前两步

“快速上手”本来就是：

- 标准阅读路径目标

不是：

- 只要看导读 + 主线 README 就结束

如果根前门继续保留这两步，

读者就容易误读成：

- 这两步就是权威 quickstart 全部

而不是：

- 这是完整路径里的开头

## 这轮具体怎么拆

### 判断三：把手写两步收成对 `路径 1` 的单点 handoff

这轮只做一件事：

1. 把根 `README` 里的“快速上手”从：
   - `00-导读.md -> 01-主线使用/README.md`
   收成：
   - `00-阅读路径.md` 里的“路径 1：我只想尽快上手”

这样：

- 根前门不再重复维护 quickstart 局部路线
- quickstart 的权威顺序重新回到阅读路径页
- 以后如果路径 1 扩缩，根前门不需要手动同步前两步

## 为什么这轮不顺手改“把结果交付给团队或外部流程”或“提升终端交互效率”

### 判断四：那是其他目标入口，不该和 quickstart 去重混成同一刀

当前最小有效改动只是：

- 收回一个已经有正式阅读路径的 quickstart 入口

如果再同时处理其他目标入口，

就会把：

- quickstart 路线去重

扩大成：

- 根前门多目标入口联动重写

超出当前最小提交边界。
