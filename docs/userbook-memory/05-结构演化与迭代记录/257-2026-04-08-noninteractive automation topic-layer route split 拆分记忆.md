## 为什么这轮该补专题层 route

### 判断一：顶层和路径层已经存在，但专题层还没有把自动化对象分诊抬成 first-hop

当前用户已经可以在：

- 顶层 README
- 路径 9

看见：

- `-p/--print`
- `--bg`
- `--sdk-url`

这条线。

但进入 `04-专题深潜`

之后，

前面的 first-hop 还没有直接逼问：

- 我现在要的是一次性结果
- 持续活着的后台会话
- 还是协议宿主

所以这轮缺的是：

- topic-layer object-boundary route

不是：

- 叶子页再扩写

## 为什么 route 文案必须把对象问题先问出来

### 判断二：这条线最大的误判不是“找不到入口”，而是“把不同自动化对象压成同一条走廊”

路径 9 目前会很快把读者带进：

- `13`
- `20`
- direct connect 深页

如果专题层 route 只说：

- “去看自动化专题”

读者仍然会继续把：

- 一次性输出
- 后台会话
- 协议流接入
- 远端附着

重写成同一种“自动化模式”。

所以这轮 route 的第一句必须先把问题压成：

- “我是想一次性拿结果、持续养后台会话，还是把 Claude Code 当成协议宿主”

## 为什么这轮先给路径 9，再给专题 13

### 判断三：专题层 first-hop 回答的是“先去哪”，路径层仍然承担“按什么顺序继续”的责任

`13`

正文已经能回答：

- `-p/--print`
- `--bare`
- `--continue` / `--resume` / `--fork-session`
- `ps` / `logs` / `attach` / `kill` / `--bg`
- `--sdk-url`

各自的合同。

但如果读者此时还没分清：

- 先固定入口对象
- 再继续读 headless host
- 最后再读结果收束与协议投影

直接只送专题 13，

仍会跳过路径 9 已经给出的顺序价值。

所以这轮更稳的顺序应当是：

- 先给路径 9
- 再给专题 13
- 最后把 deeper branch 接到 `05/非交互结果、summary 与协议流/README`

## 为什么这轮不先动任务矩阵或稳定性标签

### 判断四：矩阵与稳定性已经给了一部分答案，但还没回答“我在驱动什么对象”

`03-参考索引/05`

- `claude -p/--print` 是稳定公开

- 结果对象
- 会话对象

这说明当前更急的缺口是：

- 对象边界前门

而不是：

- 先给矩阵加更多稳定/灰度描述
