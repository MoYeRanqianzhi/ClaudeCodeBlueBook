# session-viewer vs remote-creation split 拆分记忆

## 本轮继续深入的核心判断

这一轮不再扩大到整页多前端分层，

而是只补：

- `/session`
- `--remote`

之间最容易混写的一条关系。

## 为什么这轮值得单独提交

### 判断一：当前剩下的最高价值稳定/灰度边界，不是新的入口，而是“稳定 viewer 接续面如何依赖灰度会话物化”

并行只读分析指出，

前端/远端这组里最持续拉扯的，

不是：

- `/desktop`
- `/mobile`

而是：

- `/session` 已经被写成稳定但带前提的 viewer 接续面
- `--remote` 仍是隐藏/灰度创建面

如果不把两者关系点明，

读者就会继续把：

- `remote mode 为真`
- `/session` 可交付 URL`
- `--remote` 是稳定主线`

误听成同一条链。

### 判断二：这刀比继续扩写 `06` 或 `16` 更稳

`06` 和 `16`

前面几轮已经分别收过：

- 稳定/条件前端语气
- `/session` 的 URL-ready 合同

当前最小剩余缺口，

其实就只剩这一句：

- `稳定 viewer 接续面 != 灰度会话创建面`

## 这轮具体怎么拆

这轮只做一件事：

1. 在 `16`
   现有“`/session` 的 alias 里有 `/remote`，但创建远程会话真正对应的是 CLI `--remote`”后面，
   再补一句：
   - `/session` 是稳定 viewer 接续面；`--remote` 则是把这条会话物化出来的灰度创建面

这样：

- slash command 与 CLI flag 的关系更明确
- 稳定接续面与灰度创建面不再被写成同一层

## 苏格拉底式自审

### 问：为什么要单独补这一句，前面不是已经写过两者不同了吗？

答：前面写的是“不是同一条链”，这句则把“谁消费、谁创建、谁稳定、谁灰度”一次点明。它补的是对象关系，不是重复结论。
