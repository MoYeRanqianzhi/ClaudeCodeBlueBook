# noninteractive hidden-flag visibility alignment 拆分记忆

## 本轮继续深入的核心判断

这一轮最小但最硬的一刀，

不是继续改整块前端/远端分层，

而是先把：

- `04-专题深潜/13-非交互、后台会话与自动化专题.md`

里两个最明确的可见性错位收回来：

1. `--sdk-url`
2. `--max-turns`

## 为什么这轮值得单独提交

### 判断一：这一页当前把内部/托管入口和隐藏/灰度约束写得过稳，已经和命令索引冲突

并行只读分析指出：

- `--sdk-url`
  不该写成“隐藏但完整接线”的稳定协议面，
  更稳的说法是：
  - `内部/托管宿主协议入口`
- `--max-turns`
  在命令索引里明写为：
  - `隐藏/灰度`
  但这页却把它列进了“稳定主线”

对应索引证据在：

- `03-参考索引/01-命令工具/03-CLI 旗标与根命令索引.md`

所以这不是文风差异，

而是：

- 同一本 userbook 内部的能力可见性冲突

### 判断二：这刀比先改 `06-前端与远程专题` 更小，也更确定

`06`

的稳定/条件边界也值得继续修，

但它牵涉：

- `--remote`
- `/desktop`
- `/session`
- `/voice`
- `/chrome`

要动的对象更多。

相比之下，

`13`

这一页当前有两处最直接、最无争议的错位：

- `--sdk-url` 过满
- `--max-turns` 过稳

更适合作为当前最小提交。

## 这轮具体怎么拆

这轮只做两件事：

1. 把 `--sdk-url`
   从：
   - `隐藏但完整接线的协议流入口`
   收成：
   - `内部/托管宿主使用的协议流入口`
   并补一句：
   - 自动切到 `print + stream-json + verbose`
     不等于所有消息和元数据都同宽暴露
2. 把 `--max-turns`
   从：
   - `稳定主线`
   移到：
   - `条件公开或隐藏集成面`

这样：

- 自动化专题与命令索引重新对齐
- 稳定公开能力、隐藏约束、托管宿主入口重新分层

## 苏格拉底式自审

### 问：为什么 `--sdk-url` 不能继续写成“完整接线”？

答：因为“接上协议流”不等于“对外暴露完整统一合同”。这句话太容易把托管宿主入口误听成普通用户可稳定依赖的公开协议面。

### 问：为什么要把 `--max-turns` 挪出稳定主线？

答：因为索引已经把它定成 `隐藏/灰度`。如果正文继续把它写成稳定主线，读者就会把当前隐藏约束误当成长期承诺的公开 ABI。
