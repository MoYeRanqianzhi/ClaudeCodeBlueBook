## 本轮继续深入的核心判断

这一轮最值得收的，

不是继续扩写“结构化输出”专题，

而是把：

- `StructuredOutput`

从：

- `稳定基础工具`
- `稳定核心工具`

这两个桶里移出去。

## 为什么这轮值得单独提交

### 判断一：结构化输出能力是稳定入口，不等于 `StructuredOutput` 这个工具本体就是常驻稳定工具

当前文档里已经有两个稳定事实同时存在：

1. `--json-schema` 是稳定公开旗标，而且明确服务于：
   - `print/JSON` 路径
2. `SyntheticOutputTool` 在工具 atlas 里被放进：
   - `内部或辅助子集`

问题在于，

之前的两页文档把：

- 结构化输出这条稳定能力线

直接写成了：

- `StructuredOutput` 是常驻稳定基础/核心工具

这会把：

- 稳定公开的结构化输出入口

误写成：

- 默认常驻、对所有会话都同样存在的基础工具

### 判断二：当前最小一致性切片不是只改一页，而是同步修掉两处同类错桶

目前同类误导同时出现在：

- `03-参考索引/01-命令工具/02-工具索引.md`
- `02-能力地图/02-执行与工具/01-工具系统：文件、Shell、Web、Notebook、LSP 与结构化输出.md`

如果只改其中一页，

另一页仍会把：

- `StructuredOutput`

吸回稳定桶。

所以这轮最小一致性切片是：

- 两页一起改
- 但逻辑只修一件事

## 这轮具体怎么拆

### 判断三：保留“结构化输出能力”可见性，撤掉“常驻稳定工具”误写

这轮只做两类动作：

1. 从稳定工具清单里移出：
   - `StructuredOutput`
2. 在条件工具/条件可见工具位置补回：
   - 主要见于非交互 + `--json-schema` 路径
   - 作为结构化输出 synthetic/辅助工具动态注入

这样：

- `StructuredOutput` 仍然没有从图谱里消失
- 但不再和 `Read`、`Edit`、`Bash`、`WebFetch` 一起被读成常驻稳定核心工具

## 为什么这轮不顺手重写 `03-CLI 旗标与根命令索引.md`

### 判断四：`--json-schema` 的稳定公开性并不是当前问题

`03-CLI 旗标与根命令索引.md`

目前已经把：

- `--json-schema`

写成：

- 稳定公开
- 服务于 print/JSON 路径

这部分没有问题。

当前要修的是：

- 工具本体分桶

而不是：

- CLI flag 的可见性说明

如果这轮顺手去改 CLI 页，

就会把：

- 工具分桶纠偏

扩大成：

- 结构化 I/O 路径整体重写
