## 为什么这轮该先抬升顶层入口

### 判断一：顶层用户问题先问“这些入口是不是一层”，再问“长任务怎么运营节奏”

`04-专题深潜/10`

已经很适合回答：

- 长任务如何看状态
- 预算怎么影响推进
- 模型和 effort 怎么配合任务阶段

但顶层入口用户的第一问题往往更早：

- `/status`
- `/doctor`
- `/usage`
- `/config`
- `/model`
- `/effort`

这些到底是不是一类命令。

这不是专题 10 的首要对象，

而是：

- `路径 16`

和对应的索引 / 控制面页在回答的问题。

所以顶层入口应先交给 `路径 16`。

## 为什么这轮不直接改索引页

### 判断二：中层骨架已经齐，当前缺的是最外层发现性

这条线现在的中层已经不缺：

- 阅读路径 16
- 命令工具索引 05
- 控制面 06 / 10

甚至正文也已经分成：

- 节奏运营视角
- 控制面对象视角

当前缺口出现在：

- 最外层 README 仍把用户直接送进 `04-专题深潜/10`

所以更稳的顺序是：

1. 先补顶层第一跳
2. 再看中层索引是否还要继续抬升

## 为什么这里保留专题 10 作为第二跳

### 判断三：这条线既有对象分层问题，也有运营节奏问题

如果顶层 README 只改成：

- 直接去路径 16

用户会更容易看到对象分层，

但可能看不见：

- 这条线其实最终也能回到长任务节奏运营

所以这轮更稳的写法是：

- 先去路径 16
- 再说明如果你已经分清对象，再回专题 10

这样既不把顶层首跳继续做错，

也不丢掉原来专题 10 的价值。

## 为什么这轮仍然是低 churn

### 判断四：只改顶层 README 的一条入口，不动路径与正文

这轮没有改：

- `00-阅读路径.md`
- `03-参考索引/01-命令工具/05`
- `05-控制面深挖/06`
- `05-控制面深挖/10`
- `04-专题深潜/10`

只改：

- 顶层 README 的一条目标入口

所以这轮属于：

- top-level route lift

而不是：

- settings/status/budget 结构工程
