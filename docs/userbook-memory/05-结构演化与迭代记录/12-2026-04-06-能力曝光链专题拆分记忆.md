# 能力曝光链专题拆分记忆

## 本轮继续深入的核心判断

上一批已经把“技能从哪来”和“为什么 `/skills` 不是能力全集”拆开了。

但仍缺一层专门回答：

- Claude 自己到底通过哪些链看到可用能力
- remote client 看到的 `system/init` 与模型侧看到的 broad inventory / relevant reminder 为什么不同
- UI 短提示与模型真正收到的 meta reminder 为什么不能混写

如果这一层不拆，正文会继续把四类不同对象混成“Claude 看见的技能列表”：

- `/skills`
- `system/init`
- `skill_listing`
- `skill_discovery`
- SkillTool runtime 校验

## 苏格拉底式自审

### 问：为什么不把这些内容继续塞进上一批“技能来源与暴露面”专题？

答：因为上一批回答的是“技能从哪里来、为什么会出现在不同投影里”，这一批回答的是“Claude 与 remote client 实际通过什么媒介看见它们”。继续混写会让“来源”与“曝光机制”再次缠在一起。

### 问：为什么不只在索引里加一张矩阵？

答：因为读者最容易误判的不是表格分类，而是把 UI 提示、system reminder、SDK init 元数据都当成同一类“列表”。这需要一篇长文先把受众和对象讲透，再让索引负责速查。

### 问：为什么这一批仍放在 `05-控制面深挖`？

答：因为它讨论的是能力曝光控制面，而不是某个具体任务工作流。它回答的是“为什么 Claude 并不是平等看到全部能力”，本质上是控制面问题。

### 问：正文里最该避免的假并列是什么？

答：

- `/skills` vs `system/init.skills`
- `skill_listing` vs `skill_discovery`
- UI 短提示 vs 模型 meta reminder
- 菜单可见 vs SkillTool 真可调用
- 默认主线 vs `EXPERIMENTAL_SKILL_SEARCH`

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- 为什么作者要把“来源”与“曝光”再拆成两批
- search 开启时 bundled + MCP / bundled-only 的裁剪细节该如何降级措辞
- subagent 与主线程在技能提醒 framing 上的实现差异是否适合写成长期产品承诺

这些都是维护判断，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- 能力曝光不是一条链，而是多条不同受众的链
- `system/init` 是给 remote / SDK 客户端的，不是给模型的
- `skill_listing` 是 broad inventory，`skill_discovery` 是相关子集提醒
- UI 短提示与模型 meta reminder 的差异
- SkillTool runtime 校验与菜单可见面的差异

### 不应写进正文的

- 具体 token budget 数字与 truncation 统计逻辑
- search 开启时 exact fallback 阈值为何这样取值的维护层理由
- 哪些内部短 id / 反馈路径只在 ant 环境出现

这些继续只放 `docs/userbook-memory/`。

## 后续继续深入时的检查问题

1. 我现在写的是“谁看到什么”，还是“这条技能从哪里来”？
2. 这是 UI 可见面，还是模型可见面，还是 remote 元数据面？
3. 这条行为是默认主线，还是实验搜索链下才成立？
4. 我是不是又把 UI 上看到的短提示误写成模型真正收到的完整内容？

只要这四问没先答清，正文就会继续制造“Claude 看见一张总表”的假象。
