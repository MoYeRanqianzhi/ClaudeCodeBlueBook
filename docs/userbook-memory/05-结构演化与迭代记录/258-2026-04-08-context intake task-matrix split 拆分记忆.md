# context intake task-matrix split 拆分记忆

## 本轮继续深入的核心判断

在完成：

- 顶层 README 的 context intake 前门
- 路径 5
- `04-专题深潜/08`
- `04-专题深潜/README` 的 topic route

之后，

这条线当前缺的不是：

- 再开阅读路径
- 再扩专题正文
- 再补合同速查

而是：

- 把 context intake 压进任务视角的一跳矩阵

## 为什么这轮落点选 `03-参考索引/05`

### 判断一：`03-参考索引/05` 回答的是“我现在该先按哪个入口”

前几轮已经把这条线补成：

- 可发现
- 可分流
- 可专题阅读

但矩阵层仍然没有直接回答：

- 我现在要把材料送进会话，该先走什么入口
- 我现在要核对共享事实面，该先走什么入口

所以这轮该补的是：

- task-to-entry matrix

不是：

- 目录新节点

## 为什么这轮不用一行，而是两行

### 判断二：context intake 在任务层最稳的拆法，本来就至少是“送入”与“核对”两种动作

`08-上下文接入、附件与提示编译专题`

已经明确拆开：

- 附件 / 图片 / IDE 选区 / path refs 这类事实载体
- `/files` / `/context` 这类共享事实面与压力核对
- `/add-dir` 这类正式扩工作面

其中：

- `/add-dir`

在矩阵层已经有独立一行。

如果这轮只补一行：

- “补上下文”

就会把：

- 事实送入
- 事实面核对
- 工作面扩张

重新压平。

所以这轮应拆成两行：

- 把事实材料稳定送进当前会话
- 核对当前共同事实面与上下文压力

并保留既有：

- 扩张当前工作面

那一行。

## 为什么这轮不把 prompt compilation 抬成用户入口

### 判断三：prompt compilation 是内部机制，不是任务视角的一跳入口

这条线虽然在正文里强调：

- Claude Code 首先是输入编译器

但矩阵层回答的是：

- 用户现在该先做什么

不是：

- 运行时内部怎样再解释输入

所以这轮矩阵只收：

- 普通提示词
- 附件 / 图片 / IDE 选区 / path refs / 大块粘贴
- `/files`
- `/context`

不把：

- prompt compilation

写成一个平行入口。

## 为什么这轮不把 `/add-dir` 改写回这两行

### 判断四：`/add-dir` 回答的是制度承认，不是一般材料接入

如果把 `/add-dir`

重新揉回“补上下文”行，

会再次模糊：

- 把事实送进会话

与：

- 把目录正式纳入当前工作面

这两个主语。

所以这轮矩阵只新增：

- 事实送入
- 事实面核对

并保留已有：

- 扩张当前工作面
