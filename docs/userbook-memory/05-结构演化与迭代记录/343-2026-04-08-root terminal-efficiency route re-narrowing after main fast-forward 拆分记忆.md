# root terminal-efficiency route re-narrowing after main fast-forward 拆分记忆

## 本轮继续深入的核心判断

这一轮最小但必要的修正，

不是再补一条新的终端交互说明，

而是把快进 `main` 后重新出现的根前门直连：

- `提升终端交互效率 -> 04-专题深潜/11-终端交互、状态栏与输入效率专题.md`

重新收回到正式的：

- `路径 7：我想让终端更顺手`

上。

## 为什么这轮值得单独提交

### 判断一：这不是新增信息，而是恢复根前门与正式阅读路径之间的职责边界

`00-阅读路径.md`

已经给出：

- `路径 7`

并且它不是单页入口，

而是：

- `04-专题深潜/11...`
- `03-参考索引/01-命令工具/01-命令索引.md`

的两跳顺序。

根 `README`

如果再次只把读者直送 `04/11`，

等于重新在前门维护一条更窄的局部路线，

把正式路径压扁成“只剩专题页”。

### 判断二：快进 main 后重新出现的直连，说明这类入口收窄需要被持续记忆化，而不是假设一次修完就不会回流

本轮开始前，

当前 worktree 先执行了：

- `git fetch origin main`
- `git pull --ff-only origin main`

发现 `origin/main` 无法直接快进当前 `userbook`，

继续核对后确认：

- 当前 `userbook` 只是落后于本地 `main`
- 最终通过 `git merge --ff-only main` 把当前 worktree 快进到本地 `main`

快进后，

这条根前门直连又回来了。

所以这轮修正顺带证明一件事：

- 入口收窄不只是“写一次正文”
- 还要在记忆里明确哪些旧直连在主线合流后最容易回流

### 判断三：终端交互效率的前门不该跳过“路径页先定义问题对象”

`路径 7`

先定义的是：

- 你是在想把主题、按键、Vim、状态栏、输入回调真正用顺
- 还是只是想在命令面里速查具体入口

而根前门直接落到 `04/11`

会默认把“终端更顺手”偷换成：

- “我已经确认自己要读终端交互专题正文”

这会让读者错过：

- 后续命令索引二跳

也削弱根前门“先分型，再二跳”的职责。

## 这轮具体怎么拆

这轮只做一件事：

1. 把根 `README` 里的：
   - `提升终端交互效率 -> 04/11`
   收成：
   - `提升终端交互效率 -> 00-阅读路径.md -> 路径 7`
2. 保留一条条件化下钻：
   - 只有在已经确认卡点确实是主题、按键、Vim、状态栏或输入回调时，才继续直读 `04/11`

这样：

- 根前门重新回到正式阅读路径层
- 深页仍保留为条件化二跳
- 正式路径重新承担“专题页 + 命令索引”的组合编排

## 为什么这轮不顺手同时修“交付”“自动化”“多前端接续”等其它回流项

因为当前最小有效边界仍然是：

- 一条重新回流的根前门目标入口

如果把其它回流项一起处理，

这次提交就会从：

- 单条入口重收敛

膨胀成：

- 快进 main 后的一揽子回流清扫

不利于后续追踪到底是哪一类回流最常见。

## 苏格拉底式自审

### 问：为什么明明之前已经收过这条入口，现在还要再收一次？

答：因为这次不是在证明“之前判断错了”，而是在证明这类根前门职责漂移会随着主线快进回流，值得被单独记忆化。

### 问：根前门能不能直接给深页，只要深页确实相关就行？

答：不能。相关不等于适合作为前门。只要正式路径已经承担“问题分型 + 顺序编排”，根前门就不该再维护更窄的局部版本。

### 问：为什么还保留一个“如果你已经确认卡点”的下钻句，而不是完全只留路径 7？

答：因为这里要收的是默认入口职责，不是抹掉深页的存在。条件化下钻仍有价值，但它必须从属于正式路径，而不是取代正式路径。
