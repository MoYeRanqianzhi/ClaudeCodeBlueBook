# bridge 生命周期模式专题拆分记忆

## 本轮继续深入的核心判断

上一批已经拆清了：

- workspace trust
- bridge eligibility
- trusted-device

但 bridge 自身的生命周期模式仍没有被正文稳定拆开：

- 显式 remote-control
- startup default
- outbound-only mirror
- assistant/perpetual continuity
- bridge pointer continue

如果不补这一批，正文会继续犯四种错：

- 把 `remoteControlAtStartup` 当成“自动等于手动开启”
- 把 mirror 写成“弱化版 remote-control”
- 把 perpetual 写成“重连成功”
- 把 `remote-control --continue` 写成普通 `--continue`

## 苏格拉底式自审

### 问：为什么这批不能塞回第 23 页？

答：第 23 页回答的是：

- bridge 要过哪几层前提

而本轮问题变成：

- 当前 bridge 已经在跑时，它到底是以哪种模式在跑

这已经从“准入条件”切换成：

- 生命周期模式

因此需要再单独拆一页。

### 问：为什么这批也不能塞回第 21 页？

答：第 21 页回答的是：

- host / viewer / health-check 是什么对象

而本轮关注的是：

- host 已经成立后，bridge 还有哪些不同状态机模式

所以不再是对象类型，而是：

- host 子系统内部的模式错位

### 问：为什么这批值得进入正文，而不只留在 memory？

答：因为这里有明显的用户误判成本：

- “为什么这轮没显式开 remote-control，footer 却有桥状态？”
- “为什么 web 能看到会话但控制不了本地？”
- “为什么 assistant 跨重启还像同一条会话？”
- “为什么 `remote-control --continue` 找到的是另一条 session？”

这些已经是用户可感知行为，不只是作者内部分类。

### 问：这批最该防的假并列是什么？

答：

- enabled = explicit
- mirror = remote-control
- connected = sessionActive
- perpetual = reconnect
- bridge continue = generic continue

只要这五组没先拆开，bridge 的行为就会被写成一条糊掉的“自动重连”。

## 正文落笔纪律

### 应写进正文的

- `remoteControlAtStartup` 的 precedence
- `replBridgeExplicit` 与 startup default 的区别
- `CCR Mirror` 只在没进 full remote-control 时带起
- outbound-only 会拒绝 mutable control requests
- `connected` 与 `sessionActive` 是两段不同状态
- assistant/perpetual 的 pointer reuse 与 clean-exit survive 语义
- `remote-control --continue` 读取 bridge pointer across worktrees

### 不应写进正文的

- pointer 文件 JSON 结构本身
- v1/v2 env-less bridge 的内部协议细节
- ant-only rollout 与 test hooks
- 后端 lease/TTL 的实现细枝末节

这些内容只保留为作者判断依据，不回流正文。

## 本轮特殊注意

### “bridge 已启用”永远不够

正文里不能只写：

- bridge is enabled

更稳的句式必须补对象：

- 是显式开的还是默认带起的
- 是双向还是 outbound-only
- 是普通还是 perpetual

### mirror 不应继续被降格成 UX 变体

mirror 与 remote-control 的差异不是：

- 提示词多少
- UI 明不明显

而是：

- inbound control 是否允许

这是正文必须保留的第一性原理。
