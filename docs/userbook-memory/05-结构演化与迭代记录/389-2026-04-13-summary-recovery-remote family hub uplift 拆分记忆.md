## 为什么这轮值得单独提交

`100-149` 现在已经同时拥有：

- `208/209/210` 对 `100-115` 的 summary / wire / callback / UI 可见性骨架

这说明它缺的已经不是：

- 新的结构页
- 新的范围图

而是：

- 在主 `README` 的扁平时间流之外，
  先给这一整簇文件一个可浏览的家族入口

### 判断二：这轮比 `191-206` 更优先，因为 `100-149` 的三段主骨架已经稳定成型

`191-206` 目前更像：

- `197` ingress 阅读链
- `203` permission tail 分叉图
- `206` blocked-state ceiling

再加上一层 stable-gray hardening

还不如 `100-149` 这种已经天然长成三段稳定主骨架的范围，

更适合先做 family hub。

所以当前更稳的优先级不是：

- 先把 `191-206` 再包一层

而是：

- 先把 `100-149` 从“散落在时间流里的三段结构链”提升成一个可浏览入口

## 苏格拉底式自审

### 问：为什么这轮不是继续补 `213` 或 `206`？

答：因为 `100-149` 现在已经有足够多的结构页和 scope guard，真正的瓶颈变成“读者怎么在记忆层找到这一簇”，不是“还缺哪一张单页图”。

### 问：为什么 `100-149` 比 `191-206` 更适合先做 family hub？

答：因为 `100-149` 已经自然收成三段主骨架，而 `191-206` 目前更依赖局部 anti-overlap map 与 stable-gray hardening，家族边界还不如前者完整。

### 问：为什么仍然不搬原始记忆文件？

答：因为当前最需要优化的是导航，不是物理路径。先加 hub，后谈搬迁，风险更低。
