## 为什么这轮值得单独提交

### 判断一：`128-138` 已经天然收成两段稳定骨架

这一段现在已经清楚分成：

- `128-132` transport contract / presence / ledger / topology
- `133-138` schema-store / bridge chain / interaction shell

而且这两段已经分别被：

- `214`
- `215`

固定成两张结构页。

这说明它已经不只是 `122-149` 中的过渡段，

而是可以被单独浏览的一段更窄子家族。

### 判断二：这一刀比继续补 `214/215` 的层级说明更值

`214` 与 `215` 现在都已经有：

- 页首范围声明
- stable / conditional / gray 分层

相比之下，

目录层还缺少一条能把读者直接送入 `128-138` 的短路。

所以更稳的最小改动不是再给旧页加注，

而是加一个 README-only subfamily hub，

把 `214/215` 这一对结构页和它们的护栏记忆直接抽成入口。

## 苏格拉底式自审

### 问：为什么这轮不是继续补 `214/215`？

答：因为这两页现在已经有足够清晰的结构护栏和分层。当前更缺的是目录层怎样让人直接跳到它们，而不是再加一层正文说明。

### 问：为什么 `128-138` 到现在才拆？

答：因为之前 `139-149` 和 `144-149` 的入口收益更高；现在这两层已经落地，`128-138` 的抽出才开始显得必要。

### 问：为什么仍然不搬原始记忆文件？

答：因为当前最需要优化的是导航，不是物理路径。先加 subfamily hub，后谈搬迁，风险更低。
