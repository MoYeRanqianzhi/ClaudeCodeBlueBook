# streamlined-action stable-gray hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经补完：

- `114`
- `115`

这两页的 stable/conditional/gray 分账之后，

当前更值钱的下一刀，

不是继续裂 `112-113` 目录，

而是回到：

- `112-streamlined path 的纳入 gate、替换入口与抑制返回不是同一种消息简化.md`

把它从旧式的“当前源码能稳定证明什么 / 不能稳定证明什么”收尾，

补成和相邻成熟结构页一致的：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

三层分账。

## 为什么这轮值得单独提交

### 判断一：当前更稀缺的不是入口，而是 `112` 自己的合同厚度

现在读者已经能稳定走到：

- `111-115`

这一支线；

继续新增 `112-113` 目录入口的边际收益已经下降。

相反，

`112`

仍停在旧式的：

- 能稳定证明什么
- 不能稳定证明什么

这会让：

- inclusion
- replacement
- passthrough
- suppression

这四层对象仍然容易和 helper / host wiring 写平。

### 判断二：`112` 比继续裂目录更适合承接“保护稳定功能 / 隔离灰度功能”

这页最需要保住的稳定层是：

- inclusion gate != transformation action

最需要降到条件或灰度层的是：

- helper 是否真的参与当前主流程 gating
- host/consumer route 是否会继续让 `result` 扮演同一种保留角色
- exact helper 顺序与 wiring

这比继续加一个 README-only hub，

更直接地服务于：

- 保护稳定功能
- 隔离灰度功能

### 判断三：这轮也在保护“目录层收敛”本身

当前选择不再继续裂目录，

而是回到正文补厚，

本身就是一种结构优化：

- 让入口层负责 findability
- 让正文层负责 contract thickness

避免把“找得到”和“说得稳”继续混成一个动作。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `112-...消息简化.md`
   - 把原来的“稳定证明 / 不能稳定证明”改写成
   - `稳定可见`
   - `条件公开`
   - `内部/灰度层`
2. 新增这条 `408` 记忆
   - 记录为什么这轮继续补正文合同厚度，而不是再裂目录
3. 更新 `memory/05 README`
   - 同步批次索引到新的 `408`

这样：

- `112`
  会和 `114/115`
  共享同一套分层口径
- streamlined 这条线不会再只停在“当前源码能不能证明”的旧式尾部
- 后面如果继续去补 `100-104` 或别的厚度页，也有清晰模板

## 苏格拉底式自审

### 问：为什么这轮不是继续去做 `112-113` 的子家族入口？

答：因为当前更稀缺的不是 findability，而是合同厚度。读者已经能稳定走到这条线，缺的是如何不把 gate、passthrough 与 suppression 再写平。

### 问：为什么这轮不是回去做 `100-104`？

答：因为 `100-104` 的结构边界虽然还值得做，但 `112` 现在保留旧式尾部的缺口更直接、更局部，也更容易独立提交。

### 问：为什么这也算“优化相关目录结构”？

答：因为结构优化不只等于新增目录。当前明确停止继续裂目录，本身就在保护层级职责，把入口层与合同层重新分开。
