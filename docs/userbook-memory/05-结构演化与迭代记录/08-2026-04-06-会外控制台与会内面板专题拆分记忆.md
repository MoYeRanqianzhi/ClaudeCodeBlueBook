# 会外控制台与会内面板专题拆分记忆

## 本轮继续深入的核心判断

上一批已经把：

- CLI 根入口
- root flags
- root commands
- fast-path

从启动层拆出来，但仍缺一层非常容易污染正文的“同名双层入口”：

- `claude auth ...` vs `/login` `/logout`
- `claude mcp ...` vs `/mcp`
- `claude plugin ...` vs `/plugin`
- `claude agents` vs `/agents`
- `claude doctor` / `setup-token` vs `/doctor` `/status`

如果这一层不独立，正文会继续在三个地方犯同一种错：

- 把 root command 写成 slash command 的 CLI 版
- 把 slash panel 写成 root command 的 UI 版
- 给 root-only / slash-only 入口硬找“对偶关系”

## 苏格拉底式自审

### 问：为什么不直接在 `18-CLI 根入口、旗标与启动模式专题` 里继续补？

答：因为 `18` 的对象是“进程启动层”，而这轮的对象是“同名入口跨层时如何区分”。继续往 `18` 里补，会把启动层和会话层重新缠回一起。

### 问：为什么不把这些内容塞回 `15-账户专题`、`17-扩展运维专题`、`14-初始化专题`？

答：因为那些页是在按对象讲：

- 账户面
- 扩展运维面
- setup 面

这轮则是在按“层级错位”讲。如果散落回去，读者还是会在不同页里反复撞上同一个边界问题。

### 问：哪些地方最容易写出假对称？

答：

- `claude auth status` 与 `/status`
- `claude mcp serve` 与 `/mcp`
- `claude plugin install` 与 `/reload-plugins`
- `claude agents` 与 `/agents`
- `setup-token` 与 `/login`

### 问：哪些内容必须沉到 memory，而不能回到正文？

答：

- 为什么“同名双层入口”值得独立成页
- 哪些看似能配对的入口其实不该配对
- 作者在“root-only / slash-only / 成对出现”三类之间如何保守落笔

这些都属于维护知识，不是读者工作知识。

## 正文落笔纪律

### 读者正文里该出现的

- 会外生命周期管理与会内状态面板的区别
- 为什么某些入口同名但根本不是同一个对象
- 哪些入口成对出现，哪些本来就是 root-only 或 slash-only

### 读者正文里不该出现的

- 作者为什么觉得旧目录“仍然有边界泄漏”
- “这批还没拆会继续污染哪几页”的维护层担忧
- 下一轮还想拆哪些 pair 的推进语句

这些继续留在 `docs/userbook-memory/`。

## 本轮边界保护

### 应写成“成对但不同层”的

- `claude auth login/logout` vs `/login` `/logout`
- `claude mcp ...` vs `/mcp`
- `claude plugin ...` vs `/plugin`
- `claude agents` vs `/agents`
- `claude doctor` vs `/doctor`

### 应明确写成 root-only 的

- `setup-token`
- `claude auth status`
- `claude mcp serve`
- `claude install`
- `claude update`

### 应明确写成 slash-only 或 slash-first 的

- `/status`
- `/reload-plugins`
- `/skills`
- `/hooks`

## 后续继续深入时的检查问题

1. 这段内容是在讲“会外生命周期管理”，还是“会内当前状态”？
2. 这个入口真有跨层对偶项，还是只是名字看起来相似？
3. 我现在写的是读者行动知识，还是作者的边界维护知识？

只要这三问没有先答清，正文就会继续制造假对称。
