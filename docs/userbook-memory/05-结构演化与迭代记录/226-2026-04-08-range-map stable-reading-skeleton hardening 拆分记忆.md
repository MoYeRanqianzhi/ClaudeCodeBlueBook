# range-map stable-reading-skeleton hardening 拆分记忆

## 本轮继续深入的核心判断

`205/206/218/219` 这组页真正稳定下来的，是：

- 阅读骨架

不是：

- 它们内部列出来的 branch 节点名本身已经变成稳定公开能力

如果不先把这两层拆开，

最容易发生的误写就是：

- 把 `bridge authority`
- `env truth`
- `session_context`
- `teleport runtime zoom`
- `model ledger`
- `bridge outbound trunk`

这些分析节点，

直接写成：

- 稳定公开对象层

这会把 range-map 的职责从：

- 固定阅读顺序

滑成：

- 替对象成熟度背书

## 为什么 range-map 不能直接借用能力边界里的“稳定公开能力”话语

### 判断一：range-map 解决的是“怎么读”，不是“哪些对象可对普通用户承诺”

`205/206/218/219` 这一组页的首要工作是：

- 把 `168-190` 从线性编号改写成树状阅读骨架

所以这些页里“稳定”的真正含义是：

- 这棵树怎么分枝已经固定

而不是：

- 树上的每个中间节点都可当成功能目录名来承诺

一旦把这两件事混起来，

读者就会自然把：

- `bridge authority`
- `teleport runtime zoom`
- `model ledger`

读成：

- 产品已经稳定公开的一层对象

这违反了能力边界写作规范里对“稳定公开能力”的定义。

## 为什么这轮要把 stable / conditional / internal 改写成阅读骨架话语

### 判断二：条件公开与内部证据仍然有用，但第一列必须先改名

原来的问题不在于：

- 条件公开 / 内部灰度 这两层不该写

而在于：

- 第一列写成 `稳定可见`

会让整张表看起来像在做成熟度分级。

所以这轮更稳的做法是：

1. 明说这里的“稳定”只指阅读骨架稳定
2. 把第一列统一收成 `稳定阅读骨架`
3. 把最后一列收成 `内部证据层`

这样保留下来的三层关系就变成：

- 稳定阅读骨架：固定树怎么长
- 条件公开：哪些运行时差异会改变这棵树在不同宿主里的可见厚度
- 内部证据层：哪些节点名、helper、账本只该留在证明层

## 为什么入口 README 也要补一句免责声明

### 判断三：如果入口层不先提醒，读者会在目录页就发生误读

`接续与桥接分叉/README.md`、`03-参考索引/02-能力边界/README.md`、`05-控制面深挖/README.md` 都会把这组页当成入口列出来。

如果入口层只写：

- 先看 `205/206`
- 先看 `218/219`

却不补一句：

- 这里稳定的是阅读骨架，不等于节点名已经是稳定公开能力

读者会在还没进正文时，

就把这些页误读成：

- 又一组能力目录

所以这轮顺手在入口层补一句总括免责声明，

不是重复正文，

而是为了把误读拦在目录层。

## 本轮最关键的新判断

### 判断四：阅读骨架稳定，和能力成熟度稳定，必须显式分账

之前几轮更多在修：

- handoff
- range split
- physical entry

这轮说明另一个结构纪律：

- tree-structure stable
- object-contract stable

不能共用同一个“稳定”。

前者服务于：

- 阅读与导航

后者服务于：

- 对普通用户的能力承诺

不显式分账，就会把内部分析页写成产品目录。

## 苏格拉底式自审

### 问：我现在是在说“这页该怎么读”，还是在说“这对象已经稳定公开”？

答：如果答案是前者，就不能直接用“稳定公开能力”的口气落字。

### 问：我是不是因为某个节点在 range-map 里反复出现，就把它升级成了用户可依赖对象层？

答：节点频繁出现，通常只能说明它是阅读骨架上的关键分叉，不等于它有公开入口。

### 问：我是不是把能力边界的四分法，生搬硬套到了 range-map 的树结构页？

答：如果第一页不先改成“稳定阅读骨架”，那就是。
