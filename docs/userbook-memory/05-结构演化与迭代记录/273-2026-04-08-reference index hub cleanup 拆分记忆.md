## 为什么这轮落点选 `03-参考索引/README.md`

### 判断一：参考索引首页首先应该回答“这一层有哪些入口”

`03-参考索引`

的真实子结构其实很清楚：

- `01-命令工具`
- `02-能力边界`
- `03-技能与扩展`
- `04-功能面七分法`
- `05-任务到入口速查矩阵`
- `06-高价值入口运行时合同速查`

但在这轮之前，

首页一打开先看到的却是：

- 大体量问题账本

而不是：

- section-level hub

这会让目录页先失去目录职责，再被迫同时承担深清单职责。

## 为什么这轮不做重命名或文件搬移

### 判断二：真正缺的是前门分层，不是物理路径变化

当前问题不在于：

- 子目录命名错误
- 文件需要改名
- 结构需要大迁移

而在于：

- section hub 没有先露出来
- deep inventory 没有被显式下沉到第二层阅读动作

所以这轮最小且最稳的一刀应该是：

- 先在同一 README 内把 hub 和 ledger 分层

而不是：

- 动物理结构

## 为什么要把“先按目录结构进入”显式写出来

### 判断三：如果不先给 section hub，读者就会被迫从问题清单里猜目录结构

目录页的第一职责应该是：

- 帮用户先找到该进哪一类索引

而不是：

- 让用户先在长问题清单里做精细匹配

如果不显式写出：

- 先按目录结构进入

读者会继续把：

- 命令索引
- 能力边界
- 扩展索引
- 功能面分层
- 任务矩阵
- 合同速查

当作一组埋在底部的附录链接。

## 为什么深问题清单也要保留

### 判断四：问题清单不是噪音，而是第二层入口

这轮不是要删掉：

- long ledger

而是要把它改回：

- 第二层入口

因为用户一旦已经知道自己卡住的是某个误判簇，

这份清单仍然是最快的回跳面。

所以这轮的目标不是：

- 简化掉问题清单

而是：

- 先让 hub 像 hub
- 再让 ledger 像 ledger
