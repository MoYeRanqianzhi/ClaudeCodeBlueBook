# terminal efficiency topic-layer route split 拆分记忆

## 本轮继续深入的核心判断

在完成：

- 顶层 README 的终端效率入口
- `00-阅读路径` 的路径 7
- `04-专题深潜/11`

- `04-专题深潜/README`
  里的 topic-layer route

## 为什么这轮该补专题层 route

### 判断一：顶层和路径层已经存在，但专题层还没把终端效率抬成 first-hop

当前用户已经可以在：

- 顶层 README
- 路径 7

看到这条线，

但进入 `04-专题深潜`

之后，

terminal efficiency 还只是平铺列表里的：

- `11-终端交互、状态栏与输入效率专题`

没有像：

- context intake
- noninteractive automation
- delivery

那样被抬成 first-hop。

所以这轮缺的是：

- topic-layer discoverability

不是：

- 再写正文

## 为什么 route 文案必须先保护“交互效率对象”

### 判断二：终端效率线上最容易被混写的，不是入口有没有，而是不同摩擦对象会不会被压成同一种皮肤设置

`11`

正文已经拆开：

- `/theme`
- `/color`
- `/vim`
- `/keybindings`
- `/statusline`
- 输入历史检索

这些对象，

并指出：

- `/statusline`

是配置持续观测，

不是只读显示。

如果专题层 route 只说：

- “去看终端专题”

读者仍然会继续把：

- 主题
- 键位
- 输入模式
- 状态栏
- prompt 历史

写成同一种“界面偏好”。

所以这轮 route 的第一句必须先把问题压成：

- 主题、颜色、Vim、快捷键、statusline 和输入历史到底分别解决什么摩擦

## 为什么这轮要显式排除灰度语音入口

### 判断三：`/voice` 属于条件/灰度输入面，不应被写成终端效率主线

当前矩阵里：

- `/voice`

已经是：

- 灰度 / 条件

而 `11`

正文的主线回答的是：

- 稳定终端交互面

如果 route 不显式写出：

- “而不是灰度语音入口”

读者很容易把：

- 终端输入效率

和：

- 条件语音输入

混成同一类入口。

## 为什么这轮先给路径 7，再给专题 11

### 判断四：专题层 first-hop 回答“先去哪”，路径层仍然负责“按什么顺序继续”

`11`

正文已经能回答对象边界，

但如果读者此时还没分清：

- 先从终端交互主线进入
- 再在专题内部区分外观层、输入层、持续观测层

直接只送专题 11，

仍会跳过路径 7 已经给出的顺序价值。

所以这轮更稳的顺序应当是：

- 先给路径 7
- 再给专题 11
- 最后把“只缺命令名”分流到命令索引

## 为什么这轮顺手接命令索引

### 判断五：terminal 线上一个常见误判是把“缺命令名”与“缺对象边界”混成一件事

终端效率线里，

经常有人并不是没懂对象，

而只是忘了：

- 具体 slash 入口叫什么

所以这轮 route 需要显式接出：

- `03-参考索引/01-命令工具/01-命令索引`

把：

- 命令名检索

和：

- 交互对象理解

分开。

## 苏格拉底式自审

### 问：我现在缺的是再解释 `11` 的正文，还是缺把这条高频终端线在专题层抬出来？

答：如果顶层、路径和专题正文都已存在，先缺的是 topic-layer route。

### 问：我是不是又想把主题、键位、Vim、statusline 和输入历史压成同一种皮肤设置？

答：如果 route 不先保护这些对象的摩擦差异，这个误判会立刻复发。

### 问：我是不是把灰度 `/voice` 混成了终端效率主线？

答：如果 route 不显式排除条件语音入口，这个误写会再次出现。
