# bridge 故障提示、停机与默认回退拆分记忆

## 本轮继续深入的核心判断

第 31 页已经拆开了：

- 状态词
- 恢复厚度
- 动作上限

但状态页之后，bridge host 仍然缺一层非常容易混写的对象：

- 故障提示
- 当前会话停机
- 自动熔断停机
- 默认策略回退

如果不单独补这一批，正文会继续犯六种错：

- 把 `Remote Control failed` 写成手动关闭结果
- 把 `Remote Control disconnected.` 写成失败恢复
- 把 `replBridgeEnabled=false` 写成单一原因
- 把 Bridge Dialog `d` 写成只断当前 session
- 把 `remoteControlAtStartup=false` 写成当前桥为什么失败
- 把当前停机与 future default policy 混成一个动作

## 苏格拉底式自审

### 问：为什么这批不能塞回第 31 页？

答：第 31 页解决的是：

- 当前这个绿色词到底有多厚

而本轮问题已经换成：

- 当前这个“关闭结果”到底是谁触发的、影响哪一层

这已经从：

- status semantics

继续下钻到：

- stop semantics

所以需要新起一页。

### 问：为什么这批值得进入正文，而不是只留在记忆？

答：因为用户遇到的是非常具体的错觉：

- failed 了就等于系统已经帮我正常断开
- dialog 里按 `d` 只会断当前这条 session
- `remoteControlAtStartup=false` 只是某个失败残留

这些都是 user-facing 误判，不是作者内部分类。

### 问：这批最该防的偷换是什么？

答：

- failure signal = disconnect result
- current-session stop = future default rollback
- `enabled=false` = 单一原因
- settings sync = failure explanation

只要这四组没拆开，bridge 关闭语义正文就还会继续糊。

## 正文落笔纪律

### 应写进正文的

- `Remote Control failed` 是 failure signal
- `Remote Control disconnected.` 是手动停机结果
- failed 后的 auto-disable 不等于手动 disconnect
- Bridge Dialog `d` 可能同时影响当前与未来默认
- `remoteControlAtStartup=false` 是 future default policy，不是 failure explanation

### 不应写进正文的

- `BRIDGE_FAILURE_DISMISS_MS` 的具体值
- 连续 init failure 的具体计数上限
- 各个失败路径先写哪个字段后清哪个字段的顺序
- overlay / notification priority 的实现细节

这些只保留为作者判断依据。

## 本轮特别保留到记忆的源码发现

### failed 与 init failed 都会走 failure fuse

`useReplBridge.tsx` 在：

- runtime `failed`
- init throw

两条路径上都会：

- 先写 error / 通知
- 再在短暂窗口后 auto-clear `replBridgeEnabled`

正文只需要写“自动熔断”，不必写具体超时毫秒数。

### `/remote-control` disconnect 与 BridgeDialog `d` 不是同一套停机语义

`commands/bridge/bridge.tsx` 的 disconnect 会直接重写：

- `replBridgeEnabled`
- `replBridgeExplicit`
- `replBridgeOutboundOnly`

并返回：

- `Remote Control disconnected.`

而 `BridgeDialog.tsx` 的 `d` 只先关当前 `replBridgeEnabled`，再在 explicit 时持久化：

- `remoteControlAtStartup = false`

这条区别必须保留在记忆中，防止正文偷写成单一 disconnect。

### settings 政策面会同步当前 AppState，但对象仍然是 future default

`Settings/Config.tsx` 在写 `remoteControlAtStartup` 后，会立刻让当前 `AppState` 跟随。

这很容易诱发误判：

- 以为设置面就是当前桥的失败解释

正文必须明确反过来写：

- 这是政策面写入，当前同步只是副作用
