# 2026-04-06 运行时深线扩容与正文去记忆化记忆

## 本轮新增的结构规则

### 1. 命令总线必须拆成“控制层”和“对象层”

以后写命令系统时，不要只写：

- builtin commands
- skills
- plugins

还必须先交代：

- 输入如何进入 `messageQueueManager`
- slash command 如何被 `processUserInput/processSlashCommand` 接管
- `Command` 协议如何承接不同来源

否则正文会把路由器、注册器、技能系统和命令对象混成一页。

### 2. 会话真相层必须与连续性工具层分开

以后不要再把下面这些东西写成同一层：

- `SessionState`
- `pending_action`
- `task_summary`
- transcript JSONL
- `resume`
- `compact`
- `memory`

更稳的拆法是：

- 一页写“实时状态 + 恢复账本”
- 一页写“compact / resume / memory 如何维持现场”

### 3. 任务运行时要从治理页中独立出来

`多 Agent、任务、团队、计划模式与 Worktree` 那页更适合写治理边界。

真正的运行态任务主链：

- 任务对象
- output file
- `<task-notification>`
- stop
- restore

应单独落在 `02-能力地图/02-执行与工具/`，否则读者会只看到治理，不理解结果如何回流。

### 4. 多前端入口页与多宿主生命周期页不能合并

以后涉及：

- `--remote`
- `/session`
- `/remote-control`
- bridge host
- desktop/mobile/voice/chrome

至少要拆成两页：

- 一页讲前端与接入面
- 一页讲宿主生命周期

如果继续混写，正文会稳定地产生一个误导：

- “所有远程相关入口本质一样”

这是错误的。

## 本轮新增的正文净化规则

### 只允许进入正文的内容

- 读者为什么要用这条路径
- 这条路径背后的运行时合同
- 稳定 / 条件 / 内部边界
- 误用风险
- 源码锚点

### 必须留在记忆层的内容

- “为什么这次要拆成这四条运行时深线”
- “这一页和那一页如何避免重复”
- “苏格拉底式自我追问的具体痕迹”
- “作者下一轮准备如何批次推进”

判断规则：

- 如果一段话的主要受益者是后续维护者，而不是当前读者，就不要回流正文。
