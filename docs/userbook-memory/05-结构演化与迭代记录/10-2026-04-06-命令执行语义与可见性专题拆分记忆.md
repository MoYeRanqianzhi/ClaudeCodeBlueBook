# 命令执行语义与可见性专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经分别写清了：

- 输入会被怎样分流
- root commands 与 slash commands 的层级差异
- Settings tab、独立 screen、调参命令与预算分流的对象差异

但仍缺一层专门回答：

- slash command 这个外观统一的入口，内部其实有几种命令对象
- `immediate`、`supportsNonInteractive`、`context`、`availability`、`isEnabled()`、`isHidden` 分别属于哪一层制度
- 为什么“能看到命令名”不等于“在任何宿主里都能这么执行”

如果这一层不拆，正文会持续把三组完全不同的事实写成一类“命令属性”：

- 命令对象类型
- 执行调度语义
- 可见性与资格边界

## 苏格拉底式自审

### 问：为什么不把这些字段继续塞回“输入分流、命令分发与入口编译”？

答：因为那一页的对象是“输入如何被编译”，而不是“命令对象自身有哪些运行时合同”。继续往里加，会让“入口编译”与“命令制度”重新缠在一起。

### 问：为什么不只在索引页补一张字段表？

答：因为读者真正会误判的不是字段名，而是把不同字段当同类。例如把 `availability`、`immediate`、`supportsNonInteractive` 都看成“功能标签”。这需要一页长文先把三张正交坐标讲清，再让索引页负责速查。

### 问：为什么这一批仍放在 `05-控制面深挖`，而不是继续扩 `04-专题深潜`？

答：因为这一批讨论的是控制面制度，而不是某个用户目标工作流。它回答的是“为什么相邻 slash command 不能按一类按钮理解”，更像控制面而不是任务专题。

### 问：正文里最该避免的假并列是什么？

答：

- `prompt` / `local` / `local-jsx`
- `immediate` / `supportsNonInteractive` / `disableNonInteractive`
- `availability` / `isEnabled()` / `isHidden`
- `REMOTE_SAFE_COMMANDS` / `BRIDGE_SAFE_COMMANDS` / headless 过滤
- `userInvocable: false` / `disableModelInvocation: true`

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- 为什么这批专题应该按“三张正交坐标”继续拆
- 作者如何判断哪些字段属于对象层、哪些属于调度层、哪些属于资格层
- 当前 visible source 里哪些能力更像运行时保留能力而不是稳定主线路径

这些都是维护判断，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- `prompt`、`local`、`local-jsx` 分别是什么对象
- `immediate`、排队、`context: fork`、headless 过滤分别控制什么
- `availability`、`isEnabled()`、`isHidden`、`userInvocable`、`disableModelInvocation` 分别控制什么
- remote、bridge、headless 三套宿主过滤为什么不能混写

### 不应写进正文的

- 作者为何认为旧结构还缺“命令制度层”
- 当前哪些字段在未来最值得继续拆成专门章节
- 对尚未大规模公开使用的运行时能力应怎样降级措辞的写作犹豫

这些继续只放 `docs/userbook-memory/`。

## 本轮边界保护

### 应明确写成对象类型的

- `prompt`
- `local`
- `local-jsx`

### 应明确写成调度语义的

- `immediate`
- `context: fork`
- `disableNonInteractive`
- `supportsNonInteractive`

### 应明确写成可见性/资格边界的

- `availability`
- `isEnabled()`
- `isHidden`
- `userInvocable`
- `disableModelInvocation`

### 应明确写成宿主过滤集合的

- `REMOTE_SAFE_COMMANDS`
- `BRIDGE_SAFE_COMMANDS`
- headless 命令过滤

## 后续继续深入时的检查问题

1. 这个字段是在定义“命令是什么”，还是“命令什么时候跑”？
2. 它控制的是执行宿主，还是用户资格，还是 UI 可见性？
3. 这条边界是普通用户稳定主线，还是条件公开，还是只是运行时保留能力？
4. 这段分析是不是又把 remote、bridge、headless 三条边界写混了？

只要这四问没有先答清，正文就会继续制造假并列。
