# bridge 设置面、控制面与状态面拆分记忆

## 本轮继续深入的核心判断

第 24 页已经把 bridge 生命周期模式拆成：

- startup default
- mirror
- perpetual continuity
- bridge pointer continue

但读者在实际使用里仍会把另一组对象混成同一个“远控开关”：

- Settings 里的 `remoteControlAtStartup`
- `ConfigTool`
- `/remote-control`
- Footer 状态 pill
- Bridge Dialog

如果不再补这一批，正文会继续犯六种错：

- 把“默认策略”写成“当前显式动作”
- 把 `ConfigTool` 写成远控命令替身
- 把“看不见 footer pill”写成“当前没有 bridge”
- 把 `Remote Control active` 写成完整内部状态
- 把 dialog disconnect 写成统一持久化设置
- 把 Settings UI 与 `ConfigTool` 的可见性/读值差异误写成“完全同一个入口”

## 苏格拉底式自审

### 问：为什么这批不能塞回第 24 页？

答：第 24 页回答的是：

- bridge 现在处在哪一种生命周期模式

而本轮问题变成：

- 用户眼前这些入口分别在改什么对象

这已经从“mode boundary”切换成：

- surface boundary

所以需要单独拆一页。

### 问：为什么这批也不能回退成一般的设置说明？

答：因为这里不是普通 Settings 文档，而是 bridge 特有的对象错位：

- 默认配置
- 当前显式控制
- 运行时状态展示
- 当前桥查看/断开

只有把四张面一起放进同一页，读者才不会把它们重新压成一个按钮。

### 问：为什么这批值得进入正文，而不是只留在记忆？

答：因为这里都是用户可直接感知的行为：

- 为什么改了 setting，当前桥会跟着反应？
- 为什么桥似乎在跑，但 footer 不显示？
- 为什么看到 `Remote Control active` 还不等于知道全部状态？
- 为什么 dialog 里断开有时会顺便改默认，有时不会？

这已经不是作者自己的笔记，而是用户会真实踩到的误判。

### 问：这批最该防的假并列是什么？

答：

- setting = command
- visible status = bridge existence
- `active` = `sessionActive`
- disconnect = persist default change

只要这四组没先拆开，正文就会把设置面、控制面、状态面写成同一个“远控开关”。

## 正文落笔纪律

### 应写进正文的

- `remoteControlAtStartup` 是默认策略面
- `ConfigTool` 与 Settings 是同一张默认策略面的两个入口
- `ConfigTool` GET 读的是 effective value，`default` 只是 SET 特殊 token
- `/remote-control` 是当前 session 的显式控制动作
- footer pill 有意隐藏 implicit-and-not-reconnecting bridge
- `Remote Control active` 是 coarse status，不是完整内部状态
- Bridge Dialog 的 disconnect 是否持久化，取决于 `explicit`

### 不应写进正文的

- `AppState` 全字段清单
- footer pill 与 selection navigation 的全部内部协调细节
- GrowthBook gate 的具体实验名与 rollout 历史
- transpiled 文件里的 source-map 噪音

这些只作为作者判据保留，不回流正文。

## 本轮特殊注意

### “当前看见某个 UI”永远不够

正文里不能只写：

- 用户看见了 Remote Control

更稳的句式必须补对象：

- 改的是默认策略还是当前动作
- 显示的是粗粒度状态还是完整状态机
- 当前断开是 session-only 还是顺便改默认

### “立即同步到 AppState”不等于“语义变成命令”

本轮最容易犯的过度简化是：

- Settings/ConfigTool 会马上触发 bridge 响应

于是误写成：

- 它们就等价于 `/remote-control`

这正是正文必须避免的偷换。

### “都是同一个 setting”也不等于“用户在所有入口看到同样语义”

本轮另一个容易滑坡的点是：

- Settings UI 和 `ConfigTool` 都在碰 `remoteControlAtStartup`

于是误写成：

- 它们对用户完全等价

但源码显示：

- Settings UI 还受 `isBridgeEnabled()` 约束
- `ConfigTool` GET 返回的是 effective value

所以正文必须写成：

- 同一张默认策略面
- 但不同入口对可见性与读值的呈现并不完全相同

3. 我写的是默认策略、当前动作、状态展示，还是查看/断开语义？
4. 我是不是又把 setting、command、status、dialog 压回一个“远控开关”？
