## 为什么这轮该补专题层入口

### 判断一：`04-专题深潜/README` 不是纯目录，而是高频复杂工作的首跳分诊页

这页前面已经显式抬了：

- `安全放大工作面`
- `多前端接续与条件远端`
- `启动时决定什么，进会话后决定什么`
- `status/doctor/usage/config/model/effort` 的对象分流

说明它承担的不是：

- 单纯列标题

而是：

- 给高频复杂工作一个 first-hop

既然 `/plugin`、`/mcp`、`/skills`、`/hooks`、`/agents`

以及 `/reload-plugins`

这条线也已经具备：

- 独立阅读路径
- 专题主文
- 控制面页

它就应该在专题层得到同级入口。

## 为什么专题层 first-hop 仍应先落到路径 91

### 判断二：专题层的“先去哪”与路径层的“按什么顺序继续”不是同一件事

`04-专题深潜/17`

已经能回答：

- 包管理、server 管理、暴露面查看、当前 session 刷新

这些对象的区别。

但用户如果此时还没分清：

- 该先看运维对象
- 还是先看扩展层选择

直接只送专题 17 仍然会跳过路径层的排序价值。

所以专题层入口更稳的顺序应当是：

- 先给路径 91
- 再给专题 17
- 最后把抽象层选择导向控制面 02

## 为什么这轮不同时改扩展参考索引

### 判断三：当前缺的是“工作对象入口”，不是“技能索引说明”

`03-参考索引/03-技能与扩展/README`

已经说清：

- 这里是技能索引切入
- “如何扩展”应优先去控制面 02

它的问题不是职责含混，

而是：

- 还不是用户最容易物理踏入的那层

所以这轮不应把：

- 专题层缺口

误修成：

- 参考索引层补丁
