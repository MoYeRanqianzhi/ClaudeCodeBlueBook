# bridge 断开、退出与恢复轨迹拆分记忆

## 本轮继续深入的核心判断

第 24 页已经拆开：

- auto
- mirror
- perpetual
- continue

第 32 页已经拆开：

- failure signal
- current-session stop
- future default rollback

但 bridge host 还缺最后一层非常容易被压扁的 closeout boundary：

- disconnect dialog
- hook cleanup
- perpetual teardown
- pointer cleanup / refresh
- `remote-control --continue`

如果不单独补这一批，正文会继续犯六种错：

- 把 Disconnect Dialog 的 `Continue` 写成重新连接
- 把手动 disconnect 与 clean exit 写成同一种 teardown
- 把 perpetual teardown 写成“普通退出但下次还能连”
- 把 pointer 写成只属于 `--continue` 的临时文件
- 把 fresh start 清 pointer 与 `--continue` 消费 pointer 写成同一策略
- 把 closeout trajectory 与 stop semantics 重新压成一页

## 苏格拉底式自审

### 问：为什么这批不能塞回第 24 页？

答：第 24 页解决的是：

- resume source

而本轮问题已经换成：

- 当这条 continuity / resume 真的收口时，哪些资产被保留、哪些被清掉、下次怎么回来

这已经从：

- lifecycle mode

继续下钻到：

- closeout trajectory

所以需要新起一页。

### 问：为什么这批也不能塞回第 32 页？

答：第 32 页解决的是：

- failed
- disconnect
- rollback

而本轮问题更偏：

- stop 之后实际跑了哪条 teardown 轨迹

也就是：

- stop semantics

之后的：

- closeout semantics

所以不能再混回去。

### 问：这批最该防的偷换是什么？

答：

- `Continue` = reconnect
- disconnect = full teardown
- clean exit = pointer always survives
- `--continue` = generic continue

只要这四组没拆开，bridge 收口正文就还会继续糊。

## 正文落笔纪律

### 应写进正文的

- Disconnect Dialog 的 `Continue` 只是保持当前桥不变
- explicit disconnect 会链到 non-perpetual teardown
- perpetual teardown 是 local-only closeout
- pointer 在不同 closeout 路径下 fate 不同
- `remote-control --continue` 读取的是 bridge pointer 恢复轨迹

### 不应写进正文的

- pointer JSON 结构
- TTL / mtime 的具体数字
- worktree fanout 上限
- transport close / POST drain 顺序

这些只保留为作者判断依据。

## 本轮特别保留到记忆的源码发现

### REPL bridge 在 session 创建后就会立刻写 pointer

`replBridge.ts` 在 session 创建完成后立即写：

- `writeBridgePointer(dir, { sessionId, environmentId, source: 'repl' })`

这意味着 pointer 是 bridge 生命周期资产，不是 `--continue` 临时生成物。

### non-perpetual teardown 与 perpetual teardown 的 pointer fate 完全不同

- non-perpetual：`clearBridgePointer(dir)`
- perpetual：刷新 `writeBridgePointer(...)`

这条必须保留在记忆中，防止正文偷写成“退出后 pointer 有时在有时不在”的现象描述。

### fresh start 与 `--continue` 对 pointer 的政策相反

`bridgeMain.ts` 里：

- 不带 `--continue` 的 fresh start 会先清 leftover pointer
- 带 `--continue` 的恢复流会读取 pointer across worktrees，并在 deterministic failure 时清理正确那份 pointer

正文只需要写政策差异，不必把每个清理分支细抄出来。

### perpetual 与 `--continue` 虽然都依赖 pointer，但不是同一套过滤规则

- perpetual init 只复用 `source:'repl'` 的 prior pointer
- `remote-control --continue` 则先读取统一 bridge pointer，再进入 server 校验链

这意味着正文可以写：

- 二者都依赖 pointer

但不能偷写成：

- 二者读取的是完全同一种 prior state
