# recovery-object taxonomy uplift 拆分记忆

## 本轮继续深入的核心判断

这一轮不去大改整页连续性结构，

而是先在：

- `02-能力地图/01-运行时主链/04-会话真相层：实时状态、Transcript 与恢复账本.md`

补一个最小的：

- 恢复对象谱系

小节。

## 为什么这轮值得单独提交

### 判断一：当前 userbook 已经有足够材料，但“恢复什么对象”还主要靠散点吸收

现有几页已经反复讲清：

- `resume restores worksite, not sovereignty`
- `summary != witness`
- `memory != transient state`
- 本地子代理与远程代理恢复不对称

但这些判断还分散在：

- 会话真相层
- 压缩、恢复与记忆
- 任务对象、输出回流、通知与恢复

读者要自己把它们拼成：

- `resume / compact / memory / task recovery`
  各自到底在恢复什么对象

所以当前最小有效修正，

不是扩写更多 why，

而是把对象谱系先显式列出来。

### 判断二：这刀比继续扩成“恢复大总图”更稳

并行只读分析给出的结论是：

- 恢复对象值得继续做
- 但它仍排在 `控制 vs 治理` 之后

现在 `控制 vs 治理`
已经补了 crosswalk，

所以恢复对象线最合适的下一步，

不是一次性再造大结构，

而是：

- 先把对象谱系收成一张最小表

## 这轮具体怎么拆

这轮只做一件事：

1. 在“`/resume` 恢复的是工作现场，不只是标题”之后，
   增加 `## 恢复对象谱系`
2. 用 4 行把对象拆开：
   - `resume`：恢复工作现场与继续资格装配
   - `compact`：重写恢复账本主账形状
   - `memory`：外置长期稳定约束
   - `task recovery`：恢复后台任务对象与输出回流链

这样：

- `resume / compact / memory / task recovery`
  不再只靠散文段落理解
- 后续如果继续做恢复对象总图，
  也已经有了最小对象骨架

## 苏格拉底式自审

### 问：为什么不把这张谱系加到 `05-压缩、恢复与记忆`？

答：因为这里的中心问题是“恢复账和实时账怎么分”，而恢复对象谱系本身更接近真相层对象分类。`05` 更适合继续讲 continuity law，不适合先做对象总表。

### 问：为什么不把任务恢复单独再展开？

答：因为当前最小提交只需要先把它作为第四类恢复对象点出来，避免继续被读成 `resume` 的附属行为。更细的不对称恢复语义仍留在任务页承担。
