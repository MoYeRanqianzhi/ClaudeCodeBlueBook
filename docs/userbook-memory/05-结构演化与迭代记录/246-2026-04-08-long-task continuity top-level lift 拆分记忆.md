## 为什么这轮先补顶层入口

### 判断一：顶层“按目标进入”已经覆盖高频复杂目标，但连续性主线仍然隐在正文里

顶层入口已经显式抬了：

- 启动分层
- 安全放大工作面
- 上下文接入
- 交付
- 状态、预算与节奏
- 会话检索
- 扩展运维

但长任务里最反复发生的成本之一其实是：

- 上下文变长后怎么继续
- 中断后怎么回来
- 长期约束该沉到哪里

如果这条线只停留在：

- 主线正文 04
- 专题 02

用户就仍然需要先“知道它存在”，再自己去翻目录。

所以这轮该补的是：

- top-level discoverability

不是：

- deeper body rewrite

## 为什么顶层入口应先落到主线 04，再接专题 02

### 判断二：主线页先回答“什么时候用哪条连续性手段”，专题页再回答“这些对象各自延续什么”

`01-主线使用/04`

擅长解决的是：

- `/resume`
- `/compact`
- `/memory`
- `/rename`
- `/export`

什么时候用。

`04-专题深潜/02`

擅长解决的是：

- 记忆层级
- compact 不是清屏
- resume 不是聊天记录搜索
- export 不是连续性替代品

这些更深的对象边界。

所以顶层更稳的入口顺序应当是：

- 先去主线 04 建立动作判断
- 再去专题 02 建立对象边界

## 为什么这轮要把 continuity 与“找回旧会话”拆开写

### 判断三：会话检索只是连续性的一部分，不是整条连续性主线

`04-专题深潜/12`

更接近回答：

- 旧会话怎么发现
- 历史怎么检索
- 恢复选择怎么做

但 continuity 这条线还包括：

- 记忆层沉淀
- 上下文压缩
- 工作态恢复

如果顶层仍只给：

- 找回过去的会话与提问

用户会继续把：

- `/resume`
- `/memory`
- `/compact`

误收成“会话历史功能”。

所以这轮需要在顶层显式拆开：

- 连续性合同
- 会话检索与恢复选择

## 为什么这轮先不做阅读路径

### 判断四：先证明顶层入口缺口，再决定是否需要路径层排序

并行只读分析建议下一条高价值线是：

- long-task continuity

但从现状看，

这条线首先缺的是：

- 在顶层被看见

而不是立刻缺：

- 一条全新的路径号

所以这轮先补：

- 顶层 README

等顶层入口抬起来后，再判断是否需要把：

- `/memory`
- `/compact`
- `/resume`
- `/session`
- `--remote`

压成一条专门的路径或合同表。
