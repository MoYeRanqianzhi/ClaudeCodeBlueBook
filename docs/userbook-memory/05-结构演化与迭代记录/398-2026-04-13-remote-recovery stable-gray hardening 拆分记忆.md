# remote-recovery stable-gray hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 里，

`204/213/214/215/216/217`

这一批结构页已经逐步收敛到同一种后段节奏：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

这时回头看：

- `212-remote recovery、viewer ownership、transport terminality 与 compaction contract：为什么 122、123、124、125、126、127 不是一条 recovery 后续链.md`

当前最值钱的继续深入，

不是再新开一个 remote-recovery 子家族目录，

也不是回去扩写 `122-127` 的 leaf 证明，

而是把 `212` 从旧的

- `稳定主干`
- `条件 / 灰度证据`

两桶尾部，

补成和相邻成熟结构页一致的三层分账。

## 为什么这轮值得单独提交

### 判断一：当前缺口已经不在“212 有没有 branch map”，而在“212 会不会把条件公开和内部实现继续混在一起”

`212` 的前半段已经把这几件事讲清了：

- `122` 是 owner-side recovery 根页
- `123` 是 ownership 侧枝
- `124` 是 signer / proof zoom
- `125` 是 transport authority 的降层根页
- `126/127` 从 `125` 分到 terminality 与 compaction 两侧

但旧写法仍把：

- 宿主条件
- transport 族别差异
- timeout / reconnect 常量
- keep-alive cadence
- compaction bookkeeping

压在同一个：

- `条件 / 灰度证据`

桶里。

这会让读者虽然看懂树，

却还不容易分清：

- 哪些是当前可以被外部观察、但仍受宿主限制的条件公开行为
- 哪些只是内部实现或灰度证据

### 判断二：这轮优化目录结构的方式，不是继续开目录，而是显式停止过度拆分

当前 `05-结构演化与迭代记录`

已经有：

- `122-149-remote recovery、surface truth 与 shell 家族`
- `139-149-remote truth、shell 与 memory 家族`
- `144-149-session pane、command shell 与 remote memory 家族`

目录层已经足够承接 remote recovery 继续下钻。

这一刀如果再抽一个更细的 `122-127` 或 `122-127-transport tail` 子家族，

收益会低于直接把 `212` 本身加固。

所以本轮对目录结构更优的动作其实是：

- 保持现有子家族不再继续裂变
- 把结构页正文自身补硬
- 再把这个“不继续拆”的判断写进持久化记忆

### 判断三：这轮补的不是新结构，而是合同厚度

本轮真正增加的不是：

- 新的 branch map
- 新的 hub
- 新的 leaf proof

而是让 `212` 也承担：

- 保护稳定骨架
- 标出条件公开
- 隔离内部 / 灰度证据

的责任。

这样一来，

`212`

就不再只是 remote recovery 的关系图，

而会成为和 `213/214/215/216/217` 同口径的合同厚度页。

## 这轮具体怎么拆

这轮只做一件事：

1. 在 `212-...remote recovery...md`
   的后段，
   把原来的
   - `稳定主干`
   - `条件 / 灰度证据`
   改写成：
   - `稳定可见`
   - `条件公开`
   - `内部/灰度层`
2. 同时把 “不继续新开 remote-recovery 子家族目录” 记成显式结构判断，而不是隐性搁置

这样：

- `212`
  会和相邻成熟结构页共享同一套收束口径
- remote recovery 这簇页面的稳定层、条件层、灰度层不再继续混桶
- 目录结构优化也不再等于“持续加目录”，而是等于“只在结构收益足够高时才加入口”

## 苏格拉底式自审

### 问：为什么这轮不是继续去拆 `122-127` 的新家族 hub？

答：因为当前真正稀缺的不是入口，而是 `212` 自己的合同厚度。目录再细一层，仍然不能解决正文把条件公开和内部实现混在一起的问题。

### 问：为什么“不给目录继续加层”也算目录优化？

答：因为结构优化的目标不是目录数变多，而是读者的 first-stop 成本变低。当前 remote recovery 入口已经够细，继续加 hub 只会把正文缺口伪装成目录工作。

### 问：为什么 `212` 必须补成三层，而不是保留两桶式尾部？

答：因为两桶式写法会让宿主条件、transport 差异和内部常量重新混桶；三层写法才能明确告诉读者哪些是稳定合同，哪些是条件显影，哪些只是内部举证。
