现在读者已经能顺着：

这三层关系写硬。

### 判断二：`113` 正好卡在“看起来都像保留原样”但最容易把语义写平的地方

这页最容易被误写成：

- `result` 没被改写，所以它天然就是终态语义中心

但当前源码真正拆开的，是两种相邻而不同的保留：

- stream payload preservation

读者就会继续把：

- `return message`
- `lastMessage`
- `json/default/gracefulShutdownSync`

### 判断三：这轮本身也在做目录结构优化

当前明确不再继续裂 `112-113`，

而是把结构优化停在：

- 入口层负责 findability
- 正文层负责 contract thickness

这本身就是在保护目录层收敛，

避免 README/hub 和正文责任再次重叠。

## 苏格拉底式自审

答：因为 `100-104` 还有价值，但 `113` 是当前同家族里最局部、最直接、也最适合独立提交的成熟度缺口，先补它更稳。

答：因为结构优化不只等于新建目录。当前选择停止继续裂目录，本身就在保护层级职责，让入口层与正文层重新分工。
