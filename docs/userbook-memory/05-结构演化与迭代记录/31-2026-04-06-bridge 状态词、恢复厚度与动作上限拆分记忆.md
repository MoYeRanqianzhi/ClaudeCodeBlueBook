### 问：为什么这批值得进正文，而不是只留在安全文档？

答：因为用户真实会撞上的不是安全抽象，而是这些非常具体的误读：

- 看到 `Remote Control active` 就以为远端现在一切正常
- 看到 URL 就以为已完全接好
- 看到 bridge 还亮着就以为远端还能完整反控
- 看到 pill 没了就以为 bridge 完全不存在

这些是 userbook 应该直接纠正的使用问题。

### 问：这批最该防的偷换是什么？

答：

- 状态词 = 状态节点
- 恢复厚度 = 动作上限
- 可见 pill = 全状态证明
- bridge host runtime = remote session runtime

只要这四组没拆开，下一页继续深入还会糊。

## 正文落笔纪律

### 应写进正文的

- `Remote Control active` 是 coarse summary，不是完整恢复承诺
- `connected` 与 `sessionActive` 是不同恢复厚度
- `Connect URL` 与 `Session URL` 是不同 surface
- outbound-only 收窄的是动作上限，不是视觉弱态
- `BRIDGE_SAFE_COMMANDS` 说明即使 active 也不等于动作无限
- footer pill、Bridge Dialog 与 notification 不是同一张展示面

### 不应写进正文的

- failure fuse 的具体超时
- transport poll / backoff / ingress 实现细节
- mirror 分支的内部 tags
- 安全文档里更原始的作者术语，如 `write-ready/read-blind`

这些只保留为作者判断依据。

## 本轮特别保留到记忆的源码发现

### mirror / outbound-only 的状态写回本来就更薄

`useReplBridge.tsx` 的 outbound-only 分支只同步：

- `replBridgeConnected`

不会像 full remote-control 那样完整维护：

- `replBridgeSessionActive`
- `replBridgeReconnecting`
- URL / ID / error 全套 surface

这说明“mirror-only 是更薄的恢复”这条作者心智有源码依据，但正文不必把每个内部写回都细抄出来。

### footer 明确不承担 failed 的主展示面

`PromptInputFooter.tsx` 写得非常直白：

- failed state 走 notification
- 不是 footer pill

这条很重要，但正文只需要写“不同风险状态走不同展示面”，不需要展开所有底栏实现。

### `active` 同时吃掉 `connected` 与 `sessionActive`

`getBridgeStatus()` 直接把：

- `sessionActive || connected`

统一压成：

- `Remote Control active`

正文必须写出“coarse summary”这层意思，不能顺手把它翻译成：

- 完全恢复
