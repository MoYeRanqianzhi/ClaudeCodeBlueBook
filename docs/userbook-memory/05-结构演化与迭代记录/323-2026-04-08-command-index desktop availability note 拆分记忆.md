# command-index desktop availability note 拆分记忆

## 本轮继续深入的核心判断

这一轮最值得收的，

不是重排整组 `设备与前端` 命令，

而是给：

- `/desktop`

补上一句极短的可见性边界说明。

## 为什么这轮值得单独提交

### 判断一：公开入口不等于无条件稳定入口

`01-命令索引.md` 当前把：

- `/desktop`

写成：

- `继续到 Claude Desktop`

这会让它看起来像：

- 所有人都稳定可见
- 只是一个普通前端跳转入口

但 `06-命令类型、执行语义与可见性索引.md`

已经把 `/desktop` 放进：

- `availability`

这一类典型例子里。

这说明它首先回答的是：

- 谁能看见

而不是单纯：

- 进入哪一个前端

### 判断二：当前问题不是它属不属于“设备与前端”，而是缺失边界尾注

`/desktop` 仍然属于：

- 设备与前端

这一组。

所以这轮不需要挪组，

只需要补齐一句：

- 公开，但受可见性资格约束

就足以把：

- 裸稳定入口

和：

- 有 availability 前提的公开入口

区分开。

## 这轮具体怎么拆

### 判断三：只改一个句尾，不扩成整组标签清洗

这轮只做一件事：

1. 把 `/desktop` 从：
   - `继续到 Claude Desktop。`
   改成：
   - `继续到 Claude Desktop，公开入口，但受平台 / 账户可见性约束。`

这样：

- 读者仍知道它是公开前端入口
- 也不会把它误读成默认所有用户、所有环境都稳定可见
- 与 `06` 页的 `availability` 语义重新对齐

## 为什么这轮不顺手改 `/mobile` `/chrome` `/voice`

### 判断四：当前最小失配只集中在 `/desktop` 的“裸稳定”表述

同组其它条目里：

- `/chrome` 已标 `条件公开 / Beta`
- `/voice` 已标 `条件公开`

它们自己的边界信号已经在。

而 `/mobile` 是否也要补更细边界，

不是这一轮必须一起处理的最小问题。

所以这轮只封住一个最明显的信号泄漏口：

- `/desktop` 的 availability 缺失

## 苏格拉底式自审

### 问：我是不是把“公开可见”误写成了“稳定无条件”？

答：是。这类误判最容易出现在前端/入口类命令上，因为它们看起来像产品按钮，但源码里仍受 availability 边界控制。

### 问：我是不是应该因为它有可见性前提就把 `/desktop` 挪出“设备与前端”？

答：不该。对象分类没错，错的是边界说明缺失。

### 问：我是不是在做过细的修辞编辑？

答：不是。这里补的是资格边界信号，不是润色语气；它直接影响读者如何区分稳定面和条件面。
