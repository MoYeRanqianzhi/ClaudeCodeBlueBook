# headless 启动链专题拆分记忆

## 本轮继续深入的核心判断

前两批已经把：

- 非交互 / 后台 / 自动化
- 插件物化与刷新
- root command 与 slash 面板

拆开了。

但仍然缺一篇专门回答：

- 为什么 `print` 不是没有 UI 的 REPL
- 为什么 `--sdk-url` 不是普通附加参数
- 为什么 headless 的首问前等待链与交互 REPL 完全不是一套对象

如果这层不拆，正文会继续犯三种错：

- 把 headless 写成“REPL 的文本版”
- 把 `stream-json` 写成格式参数，而不是宿主协议
- 把首问前 cache-only、await、bounded wait、background 补连写成同一种 ready

## 苏格拉底式自审

### 问：为什么不把这批继续塞进 `04-专题深潜/13-非交互、后台会话与自动化专题.md`？

答：13 号专题按用户目标组织，回答的是：

- 什么情况下用 `-p`
- 什么情况下用 `--bg`
- 什么情况下用 `--sdk-url`

而本轮新增的核心问题是控制面对象：

- 入口何时分流
- 命令树何时被跳过
- I/O 何时从终端文本切成 `StructuredIO`
- 首问前到底 join 哪些 promise

这些已经不是“自动化怎么用”，而是“自动化宿主到底是什么对象”。

### 问：为什么不把这批继续塞进 `04-专题深潜/18-CLI 根入口、旗标与启动模式专题.md`？

答：18 号专题回答的是：

- root flags / root commands / 启动模式如何分层

但本轮更深一层，已经进入：

- 模式判定以后，headless runtime 怎么装配
- `print` 与 `sdk` 怎么改写 I/O 与首问就绪链

继续堆在 18 号页里，会把“模式选择”与“选完模式后的宿主合同”混成一页。

### 问：为什么这批应该放在 `05-控制面深挖` 而不是别处？

答：因为这页真正回答的是：

- 为什么相邻入口不能混写
- 为什么同样是 `print` 相关参数，它们不处于同一层
- 为什么“ready”不是单一状态，而是多条等待链的组合结果

这正是控制面深挖层最该承接的对象。

### 问：正文里最该避免的假并列是什么？

答：

- `print` vs “REPL 去 UI”
- `--sdk-url` vs “额外远端输出”
- `stream-json` vs “JSON 格式”
- cache-only plugin / skill visibility vs fresh source availability
- regular MCP batch await vs `claude.ai` connector bounded wait
- 插件已物化 vs 当前 headless session 已 refresh
- `--bare` vs “更快的默认 print”

## 正文落笔纪律

### 应写进正文的

- print/headless 在启动早期就分流，而不是 REPL 末端换皮
- `--sdk-url` 会自动改写输入输出格式、verbose 与 print 模式
- `StructuredIO` / `RemoteIO` 表示控制协议宿主，不是排版层
- headless 首问前会选择性 await 关键对象，不是全量静态 ready
- `--bare` 是最小宿主模式，不是默认 headless 推荐值
- print 的恢复、中断、持久化合同比交互更硬

### 不应写进正文的

- CCR v2 internal event writer / reader 的实现细节
- keep_alive 的具体轮询来源与 bridge-only 网络路径
- `CLAUDE_CODE_SYNC_PLUGIN_INSTALL_TIMEOUT_MS` 一类精确环境变量
- `permissionPromptTool` lazy lookup 的完整错误路径
- `open <cc-url>`、server、daemon worker 的低层实现细节

这些属于维护判断或宿主内部线索，应继续沉在 memory，而不是再污染用户正文。

## 本轮需要特别防止回流进正文的实现细节

### `StructuredIO` / `RemoteIO` 的内部事件机制

正文只需要让读者知道：

- 这是结构化控制协议

不需要把：

- `pendingRequests`
- internal event queue
- CCRClient wiring
- lifecycle delivery ack

写进正文。

### MCP 的精确等待策略

正文适合写：

- regular MCP 首问前 await
- `claude.ai` connectors bounded wait

但不应继续展开：

- 5 秒超时值
- dedup 细节
- clearServerCache 的清理路径

### plugin sync 的精确环境变量门控

正文适合写：

- sync install 会改变首问前保证

但不该继续堆：

- 哪个 env var 控哪个 timeout
- setup / refresh / MCP diff 的所有 call order

## 下一批优先切口

本页已经把：

- `print`
- `--sdk-url`
- `StructuredIO`
- 首问前 ready 链

拆成了一个完整批次。

下一批更自然的延伸，不应继续塞回本页，而应另起一页处理：

- internal `open <cc-url>`
- `assistant [sessionId]` viewer / client
- root `doctor`
- `mcp list` / `mcp get`
- 它们为什么会 skip trust、做 health check、甚至启动 `.mcp.json` 的 stdio server

这一批的正确写法应是：

- 宿主连接入口与健康检查边界

而不是：

- headless 主线的补充说明

否则正文又会重新把：

- 用户任务入口
- transport / bridge / viewer
- 健康检查 / side-effectful read

写回同一条主线里。
