# session operations task-matrix split 拆分记忆

## 本轮继续深入的核心判断

在完成：

- session operations 顶层前门
- 专题层 route
- 专题页 scope guard
- 参考索引 contract table

之后，

这条线当前缺的不是：

- 再开阅读路径
- 再扩长文正文

而是：

- 把 session operations 压进任务视角的一跳矩阵

## 为什么这轮落点选 `03-参考索引/05`

### 判断一：`03-参考索引/05` 回答的是“真实任务该先走哪个入口”

前几轮已经把这条线补成：

- 可发现
- 可分流
- 可速查

但矩阵层仍然没有回答：

- 当用户真正要运营当前会话对象时
- 在任务视角上应先走哪些入口

所以这轮该补的是：

- task-to-entry matrix

不是：

- 新的目录节点

## 为什么这轮不用一行，而是三行

### 判断二：session operations 在任务层最稳的拆法，本来就是三种动作

`07-会话运营、分叉与回退专题`

已经把这条线拆成：

- 可识别
- 可交付
- 时间线改写

如果矩阵层只写成一行：

- “运营当前会话对象”

就会把：

- `/rename`
- `/copy` / `/export`
- `/branch` / `/rewind` / `/clear`

重新压平。

所以这轮应顺着专题结构，把矩阵拆成三行：

- 给当前会话稳定命名
- 把当前结果带出会话
- 改写当前工作对象时间线

## 为什么这轮不把 `/session`、`/remote-control`、`/memory`、`/resume` 混进这几行

### 判断三：矩阵层仍然必须保住“当前工作对象运营”这条主语

这三行回答的是：

- 当前工作对象怎样被命名
- 怎样被带出会话
- 怎样被分叉、回退或切空

它们不回答：

- 当前工作对象怎样续上
- 过去工作对象怎样被找到
- viewer / host continuation

所以这轮矩阵行只收：

- `/rename`
- `/copy`
- `/export`
- `/branch`
- `/rewind`
- `/clear`

不把：

- `/memory`
- `/compact`
- `/resume`
- `/session`
- `/remote-control`

重新写回同层。

## 为什么这轮不直接立 dedicated reading path

### 判断四：矩阵层比路径层更直接回答“我现在该先按哪个按钮”

并行只读分析指出：

- session operations 目前还不值得立刻起 dedicated reading path

因为这条线虽然已成形，

但在阅读路径层仍更像：

- 单专题 + 专题内转诊

而矩阵层现在却已经明确缺位。

所以这轮应该优先补：

- `03-参考索引/05`

不是：

- `00-阅读路径`

## 苏格拉底式自审

### 问：我现在缺的是再开新的入口层，还是缺把 session operations 写回“真实任务先用哪个入口”的矩阵层？

答：如果入口链和速查表都已存在，矩阵层才是当前缺口。

### 问：我是不是又想用一行把六个命令压平，从而丢掉专题 07 已经建立好的三种动作结构？

答：如果矩阵不拆成三行，这个误判会立刻复发。

### 问：我是不是又要把 continuity、discovery、viewer/host continuation 混回这几行里？

答：如果这些对象被混入矩阵行，session operations 的主语就会再次丢失。
