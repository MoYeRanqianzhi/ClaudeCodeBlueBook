# reading-path numbering collision repair 拆分记忆

## 本轮继续深入的核心判断

继续补 `safe action-radius` 这条线之前，

先暴露出了一个更基础的结构问题：

- `00-阅读路径.md` 里已经有“路径 14：我想分清 root flags、root commands 和 slash commands”
- 但后来新增的 `safe action-radius` 独立路径也被写成了“路径 14”

这不是小编号问题，

而是：

- 所有入口提示都会开始指向一个不再唯一的对象

所以这轮更稳的做法不是继续补目标入口，

而是：

- 先修复路径编号唯一性

## 为什么这轮不只改 `safe action-radius` 一个编号

### 判断一：真正出问题的是末尾追加的整组高频路径，不只是单个路径标题

文件末尾这一组后来追加的高频路径原本是：

- 路径 10
- 路径 11
- 路径 12
- 路径 13
- 路径 14

但文件前半已经存在：

- 路径 14 到路径 25

这说明当前问题不是：

- 只有 safe action-radius 撞号

而是：

- 末尾追加块整体还停留在一组临时低编号

如果只把 `safe action-radius` 改成另一个孤立编号，

就会留下：

- `10-13` 仍然悬在 `87` 之后

所以这轮更稳的修法是：

- 把这整组末尾追加块整体抬到 `88-92`

既保留它们的局部顺序，

又恢复全文件的编号唯一性。

## 为什么这里选择 `88-92`

### 判断二：最小 churn 的原则，不是回头重排整本文件

`00-阅读路径.md` 当前最高已经到：

- 路径 87

所以把末尾追加块抬到：

- 88
- 89
- 90
- 91
- 92

有三个好处：

1. 不碰前面已经稳定引用的 14-25
2. 不需要全文件大重排
3. 让追加块继续保留追加块的位置与顺序

这轮解决的是：

- uniqueness

不是：

- visual order perfection

## 为什么这轮要同步修所有回链提示

### 判断三：编号一旦改动，入口、前门、主线分流句都必须一起改

`safe action-radius` 这条线现在至少被这些层同时引用：

- `bluebook/userbook/README.md`
- `04-专题深潜/安全放大工作面/README.md`
- `01-主线使用/02-提问、补上下文与让模型继续工作.md`
- `00-阅读路径.md`
- 结构演化记忆

如果只改标题不改回链，

读者会得到另一种更糟的错：

- 入口里还写“路径 14”
- 文件里却已经没有对应的 safe action-radius 路径 14

所以这轮必须把所有显式回链一起收口。

## 为什么这轮优先级高于继续补“启动入口分流”的顶层入口

### 判断四：编号冲突会污染后续所有发现性优化

这轮原本更像要做：

- 给启动入口分流补顶层 goal-entry

但如果路径编号本身已经撞号，

任何继续补的入口都会建立在：

- 不再唯一的 path label

之上。

所以更稳的顺序是：

1. 先修编号唯一性
2. 再继续补新的顶层发现性
