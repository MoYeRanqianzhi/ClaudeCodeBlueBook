# slash-command path serialization and boundary hardening 拆分记忆

## 本轮继续深入的核心判断

`16/18/19` 这一组文档真正暴露的问题，不只是：

- 物理路径被 slash command 误切成目录

而是：

- 标题层、入口层、物理路径层混进了同一种命名

于是读者会同时遇到三种错觉：

1. README 里像是平铺 `.md`，磁盘上却变成“前缀目录 + 叶子文件”
2. `/hooks`、`/reload-plugins` 既像标题文案，又像 basename 的一部分
3. `16/18/19` 虽然本来只是三张并列长文，却在树形浏览里像假 range root

所以这轮最稳的做法不是：

- 再开一张新地图
- 把这三页搬进新的物理子目录

而是：

- 先把 slash command 从物理路径层序列化出去
- 再把 `16/18/19` 的语义边界压回入口层和索引层

## 为什么物理文件名必须用可序列化 token

### 判断一：slash command 属于标题层，不属于文件系统分隔层

`/hooks`、`/reload-plugins` 这些字面量对读者有价值，因为它们对应：

- 实际 CLI / UI 可见命令

但文件系统对 `/` 的解释是：

- path separator

如果把同一符号同时交给：

- 标题层
- 物理路径层

就会把“命令字面量”误写成“目录结构”。

因此这轮固定的新契约是：

- 标题、H1、README 显示层可以继续保留 ``/hooks``、``/reload-plugins``
- 物理 basename 统一改用不含 `/` 的 canonical token
- 对 slash command 的序列化统一写成 `slash-*`

于是本轮落地为：

- `16-...为什么 slash-hooks 看到的不是实际会跑的 hooks.md`
- `18-...为什么 slash-reload-plugins 不是安装器.md`
- `19-...为什么插件有时会自己出现、有时只提示 slash-reload-plugins.md`

这解决的不是“好不好看”，而是：

- basename 必须可逆、可扁平化、不可再触发目录分裂

## 为什么这轮要顺手补入口层边界

### 判断二：只修路径，不修入口语义，读者仍会把 `16/18/19` 写回同一种插件/扩展总入口

这组三页本来分别回答：

- `16` = `/hooks` 的 UI 观察面与实际注册/执行面的错位
- `18` = `/reload-plugins` 作为当前 session 手动激活原语
- `19` = 宿主 / trust / headless 路径为何有时自动推进、有时停在待刷新提示

如果目录修好了，但入口层仍只做平铺列举，

读者仍然容易把它们重新压回：

- “hooks / plugin 的同一组专题”

所以这轮同步做了两件低成本但必要的事：

1. 在 `05-控制面深挖/README.md` 与 `03-参考索引/02-能力边界/README.md` 直接写出 `16/18/19` 的分工线
2. 在 `05/07/08` 三张索引顶端加一层 stable / conditional / internal 提示

这样目录层不只恢复可浏览性，

也恢复：

- 主题边界

## 为什么 `05/07/08` 要从代码样式路径改成真实入口

### 判断三：索引页顶部如果仍是硬编码物理路径文本，就会继续漂移

之前 `05/07/08` 顶端写的是：

- backtick 包起来的物理路径

这有两个问题：

1. 路径一旦调整，索引最先过时
2. 入口层明明应该负责导航，却退化成了“抄当前文件名”

因此这轮同步收敛为：

- 顶部改成真实可点击链接
- 代码样式物理路径只保留在需要记录历史结构的 memory 层

这不是样式优化，而是让索引页回到：

- 导航入口

而不是：

- 文件名展示板

## 本轮最关键的新判断

### 判断四：目录结构优化也要显式保护 stable / conditional / gray 边界

之前几轮更多在修：

- range handoff
- bridge branch map
- physical entry

这轮说明另一件事：

- path cleanup 也不能只是机械重命名

如果不同时补边界提示，

读者仍会把：

- `/hooks` 菜单细节
- SDK / background / headless refresh path
- auto-refresh 触发条件

误当成稳定主线。

所以这轮的目录修正必须和边界修正绑在一起完成。

## 苏格拉底式自审

### 问：我修的是文件名，还是在偷造一层新的结构图？

答：如果目标只是恢复扁平可浏览性，就不该新增编号地图。

### 问：我是不是把命令字面量和物理路径混成了同一个主语？

答：只要 basename 里还保留真实 `/`，答案就是“是”。

### 问：我是不是只修通了链接，却没有修通主题边界？

答：如果 `16/18/19` 在入口层仍只是并排文件名，那么读者还会继续把它们混写成一类。
