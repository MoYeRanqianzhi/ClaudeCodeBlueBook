# remote 控制合同拆分记忆

## 本轮继续深入的核心判断

第 28 页已经拆开了：

- remote session client
- viewer client
- bridge host

但还缺一层更细的“控制合同”边界：

- `can_use_tool`
- bridge / channel permission callbacks
- `set_permission_mode`
- `set_model`
- `set_max_thinking_tokens`
- `BRIDGE_SAFE_COMMANDS`
- `REMOTE_SAFE_COMMANDS`

如果不补这一批，正文会继续犯六种错：

- 把单次权限响应写成完整 session 控制权
- 把 `set_permission_mode` 写成“远端点一次权限”
- 把 `BRIDGE_SAFE_COMMANDS` 和 `REMOTE_SAFE_COMMANDS` 混成一张白名单
- 把 viewerOnly 写成“弱化版 bridge controller”
- 把 bridge transport 写成唯一的远端权限来源
- 把权限提示、会话控制和命令白名单重新压成一种“远端控制”

## 苏格拉底式自审

### 问：为什么这批不能塞回第 28 页？

答：第 28 页回答的是：

- 当前远端侧到底是 client、viewer，还是 host

而本轮问题变成：

- 在这些角色既定后，控制权是如何被进一步拆成权限响应、session control request 和命令合同的

这已经从：

- role boundary

继续下钻到：

- control-contract boundary

所以需要新起一页。

### 问：为什么这批值得进入正文，而不是只留在记忆？

答：因为这些都是用户会真实碰到的错位：

- 为什么手机上能点权限，但不是所有命令都能跑？
- 为什么远端能改 permission mode / model，这和单次审批又有什么不同？
- 为什么 assistant viewer 看起来“连上了远端”，却不该拥有一样的控制合同？

这已经是主线使用问题，不是作者内部分类。

### 问：这批最该防的假并列是什么？

答：

- permission response = session control
- bridge-safe = remote-safe
- viewerOnly = bridge controller

只要这三组没拆开，远端控制正文就还会继续糊。

## 正文落笔纪律

### 应写进正文的

- `can_use_tool` 是单次权限响应，不是 session control
- bridge / channel / local UI / hook / classifier 是竞态 rac ers
- `set_permission_mode` / `set_model` / `set_max_thinking_tokens` 是 session control request
- `BRIDGE_SAFE_COMMANDS` 与 `REMOTE_SAFE_COMMANDS` 的对象不同
- viewerOnly 会收窄控制合同，不应被写成 bridge controller

### 不应写进正文的

- control message 的完整 schema
- auth recovery / WS kill timeout 细节
- 所有竞态实现细枝末节
- 所有函数型 callbacks 的实现方式

这些内容只保留为作者判断依据。

## 本轮特殊注意

### “远端能点权限”不等于“远端什么都能控制”

本轮最容易犯的偷换是：

- 既然远端可以批准工具

于是误写成：

- 远端拥有完整 session 控制权

但源码清楚区分：

- 单次 `can_use_tool`
- session-level control request
- command whitelist

正文必须保留这种分层。

### “safe commands”也是两张不同合同

另一个容易滑坡的点是：

- 都叫 safe commands

于是误写成：

- `BRIDGE_SAFE_COMMANDS` 和 `REMOTE_SAFE_COMMANDS` 只是不同地方的同一个名单

这会直接抹掉 transport 和 client role 的差异。
