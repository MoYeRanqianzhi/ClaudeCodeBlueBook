# command-tools legend taxonomy clarification 拆分记忆

## 本轮继续深入的核心判断

这一轮最值得收的，

不是直接重写所有命令条目，

而是先把命令工具区自己的图例说明校正。

## 为什么这轮值得单独提交

### 判断一：问题先出在图例把“层级”和“展示属性”写成同一桶

`03-CLI 旗标与根命令索引.md`

当前把速读标签写成：

- `条件/高级`
- `隐藏/灰度`
- `内部/托管`

这会把：

- 条件公开
- 隐藏 help
- 内部面

写成同一种 taxonomy。

而边界正文已经明确：

- “隐藏”是展示属性
- 正式能力边界至少分稳定公开、条件公开、内部、影子四层

所以如果不先修图例，

后面的条目再精细也会继续在命令工具区被旧桶牵回去。

## 这轮具体怎么拆

### 判断二：只校正图例解释，不先大面积改动所有标签值

这轮只做两件事：

1. 把 `03-CLI 旗标与根命令索引.md` 的“标记方法”改成“速读标签”
   并明确：
   - `条件/高级` 是速读简写，不是正式层级
   - `隐藏/灰度` 只是把 help 可见性受限和条件暴露并排提醒
   - `内部/托管` 不自动覆盖影子接近面
2. 把 `06-命令类型、执行语义与可见性索引.md`
   里 `INTERNAL_ONLY_COMMANDS` 的说明从：
   - “内部或灰度”
   改成：
   - “内部面；若只剩 help/stub 接近面，再单独标影子或隐藏属性”

这样：

- 命令工具区不会再把图例当正式 taxonomy
- 下一轮若要收具体命令条目，也有了更稳的上位说明

## 为什么这轮不直接改 `01-命令索引.md` 的所有分组与条目

### 判断三：具体命令条目是下一层问题，不该和图例修正混成一刀

`01-命令索引.md`

里确实还存在：

- 分组混桶
- 条目边界混写

但那属于：

- 具体命令对象层

而这轮处理的是：

- 命令工具区的公共图例层

两者拆开更稳。
