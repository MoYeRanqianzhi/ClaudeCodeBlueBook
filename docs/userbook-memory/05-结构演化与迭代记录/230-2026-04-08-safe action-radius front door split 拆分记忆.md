## 为什么这轮该补“物理前门”

### 判断一：用户先问的是“当前该把行动半径开到哪一层”，不是“控制面目录是不是还要再编号”

只要读者已经意识到：

- 权限
- 计划模式
- worktree
- `/add-dir`

都跟治理和安全有关，

下一步最常见的误判就不是：

- “这些功能分别叫什么”

而是：

- “它们是不是同一种治理按钮”

如果没有物理前门，

读者就会在：

- `04-专题深潜/03`
- `05-控制面深挖/01`
- `05-控制面深挖/07`
- `03-参考索引/05`

之间自己重建顺序。

这说明当前缺的是：

- 可浏览的前门

不是：

- 更多编号结构

## 为什么 `/add-dir` 必须被收回这条稳定使用线

### 判断二：`/add-dir` 解决的是“正式扩张工作面”，不是“补一点上下文”

如果只把 `/add-dir` 放在“把上下文送准”这条线里，

读者很容易继续把它理解成：

- `cd` 的产品化包装
- 给模型多塞一些目录

但从运行时对象看，它真正修改的是：

- 工作面
- 权限上下文
- sandbox 可达集合

所以它在使用线里回答的不是“上下文够不够”，

而是：

- 当前是不是应该先扩大世界边界，再谈后续动作

这和 `/permissions`、`/plan`、EnterWorktree`

共同回答的是同一个第一性问题：

- 当前行动半径到底该放多大

## 为什么这轮只补最短反向链接，不继续扩正文

### 判断三：当前缺的是入口闭环，不是正文证明

`05-控制面深挖/01` 和 `05-控制面深挖/07` 的正文都已经成立。

这轮如果继续：

- 把三层治理链改写成四层长文
- 在 `01-主线使用` 再同步开更多解释
- 重排更多阅读路径

就会把一次本来只该补前门的切片，

做成：

- 结构和正文同时 churn

更稳的做法是：

1. 前门先把问题收束成“权限 / 计划 / 隔离 / 扩张”四分
2. 深页只补最短回链
3. 后续如需继续深入，再开独立切片

## 为什么这轮要把判断沉到结构记忆

### 判断四：目录优化的优先级，已经高于继续压叶子细节

之前几轮已经补了：

- slash-command path serialization
- stable reading skeleton

- multi-frontend continuity vs conditional-remote

- 叶子分析再深，如果入口层没有把高频误判收成物理前门，读者仍会退回叶子页里自己拼图

- 是不是缺物理前门
