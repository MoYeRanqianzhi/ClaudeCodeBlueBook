# CLI 根入口与启动模式专题拆分记忆

## 本轮继续深入的核心判断

`bluebook/userbook/` 之前已经把：

- 自动化
- 初始化 / 安装
- 多前端接续

拆成了独立专题，但还缺一层稳定“入口语法学”：

- 什么属于 `claude ...` 进程启动层
- 什么属于 root command
- 什么属于 slash command
- 什么只是 fast-path 宿主或隐藏旗标

如果这一层不补，正文会持续在三个地方反复混写：

- 把 `--resume` 和 `/resume` 写在一起
- 把 `doctor`、`install`、`setup-token` 混进 slash command 列表
- 把 `remote-control`、`--remote`、`--teleport`、`/session` 混成同一种“远程入口”

## 苏格拉底式自审

### 问：既然 `04-专题深潜/13` 已经写了 `--print`、`--bare`、`--sdk-url`，为什么还要再开一页？

答：因为 `13` 回答的是“自动化怎么做”，不是“CLI 根入口层如何分类”。如果继续在自动化页补 root commands 和 fast-path，主题会再次漂移。

### 问：为什么不直接在 `01-命令索引.md` 里加一大段 root flags？

答：因为那页已经被读者自然理解成 slash commands 索引。继续往里塞 root flags，只会让“命令树”和“启动层”重新混成一页。

### 问：`ps`、`logs`、`attach`、`kill` 明明长得像 root commands，为什么还要强调 fast-path？

答：因为源码里它们首先在 `cli.tsx` 被提前截走，真正重要的是“会不会加载完整 CLI”而不是“Commander 树长什么样”。

### 问：哪些内容必须继续沉到 memory，而不能进入正文？

答：

- 为什么要把“CLI 根入口层”从自动化/初始化专题中再切出来
- 哪些名字相近的入口最容易污染正文边界
- 作者在“稳定公开 / 条件公开 / 隐藏灰度 / 内部托管”之间如何保守落笔

这些是维护判断，不是读者行动知识。

## 正文落笔纪律

### 读者正文里该出现的

- root flags 如何改变启动形态
- root commands 的对象是什么
- fast-path 启动模式为何不能按普通命令树理解
- 稳定、条件、隐藏、内部这四层边界

### 读者正文里不该出现的

- 作者为何觉得旧目录“还不够干净”
- “如果不拆会继续污染哪里”的维护层担忧
- 苏格拉底式反问本身

这些继续只留在 `docs/userbook-memory/`。

## 后续继续深入时的检查问题

1. 这段内容回答的是“如何启动 Claude Code”，还是“进入会话后如何操作”？
2. 这条入口是在 Commander 命令树里，还是在 `cli.tsx` fast-path 里先被截走？
3. 这条能力对普通用户是稳定默认面，还是只应被写成隐藏边界证据？

只要这三个问题没先答清，正文就很容易再把启动层和会话层混回去。
