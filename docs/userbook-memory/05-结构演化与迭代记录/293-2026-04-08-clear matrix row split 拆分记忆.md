# clear matrix row split 拆分记忆

## 本轮继续深入的核心判断

上一刀已经把：

- 专题页里的 `/clear`
- 阅读路径里的 `/clear`
- 索引页里的 `/clear` 定义

都收回到了：

- 新的空工作面 / 新 session id

但任务矩阵里，

`/branch`、`/rewind`、`/clear`

还挤在同一行。

## 为什么这轮落点选 `03-参考索引/05`

### 判断一：矩阵是高频第一跳，不能继续把“沿现有时间线动作”和“切出新工作面”写成同一任务

任务矩阵原来写成：

- 改写当前时间线或切出新工作面
- `/branch` / `/rewind` / `/clear`

这虽然已经比更早的“改写当前工作对象时间线”好，

但仍把两类不同对象压在同一行：

- `/branch`、`/rewind` 是沿现有时间线动作
- `/clear` 是新的空工作面

如果矩阵不继续拆，

读者第一跳仍会先吸收：

- 它们是同一任务下的并列入口

这会再次冲掉专题层和合同页刚收好的主语。

## 为什么这轮不把 `/branch` 与 `/rewind` 再继续拆开

### 判断二：这两者仍属于同一对象家族，可以接受同排

`/branch`

和：

`/rewind`

虽然动作不同，

但都还在处理：

- 现有时间线

一个是分叉，

一个是回到较早恢复点。

它们和：

- `/clear`

最大的差异不在动作细节，

而在：

- 是否还沿同一时间线 / 同一前台工作面继续

所以这轮最小有效切片不是拆三行，

而是先拆成：

- 时间线动作一行
- 新工作面一行

## 这轮具体怎么拆

### 判断三：保留矩阵压缩度，只在用户目标层把对象拆开

矩阵改成：

- 沿当前时间线分叉或回到较早恢复点 -> `/branch` / `/rewind`
- 切出新的空工作面 -> `/clear`

这样：

- `/branch` / `/rewind`

仍保留速查压缩度，

而：

- `/clear`

被正式从旧行里拆出来。

## 为什么这轮不顺手再改命令索引或合同页

### 判断四：那三处上一刀已经同步完成，这轮只补矩阵残留

命令索引、合同速查、专题页、阅读路径

在上一刀后已经统一成：

- `/clear` 是新的空工作面

这轮剩下的唯一高频残留，

就是矩阵。

所以这轮只补：

- `03-参考索引/05-任务到入口速查矩阵.md`

避免再扩大范围。
