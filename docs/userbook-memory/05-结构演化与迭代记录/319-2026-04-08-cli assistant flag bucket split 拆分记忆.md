这一轮最值得收的，

而是把：

- `--assistant`

## 为什么这轮值得单独提交

### 判断一：同一分组里不该同时把条件公开项和内部托管项并列当成同类

`03-CLI 旗标与根命令索引.md`

当前在：

- `条件公开但默认不应承诺的一组`

下面放了：

- `--advisor`
- `--enable-auto-mode`
- `--channels`
- `--dangerously-load-development-channels`

这些都还能解释成：

- 条件公开 / 高级控制面

但同组又塞了：

- `--assistant`

而它自己的可见性已经明确写成：

- `内部/托管`

这会让分组标题重新把内部托管项吸回条件公开桶。

## 这轮具体怎么拆

### 判断二：只把 `--assistant` 独立成一个很小的内部托管子组

这轮只做两件事：

1. 从：
   - `条件公开但默认不应承诺的一组`
   里移出 `--assistant`
2. 紧接着补一个小子组：
   - `更偏内部托管的一项`
   用来单独承载 `--assistant`

这样：

- 原条件公开组重新纯化
- `--assistant` 也不会因为仍留在同一页就被误吸回去

## 为什么这轮不继续改 `隐藏但仍能说明运行时合同的一组`

### 判断三：那一组仍有更大范围的标签问题，但不适合和这刀混做

当前最小有效切片只是：

- 把一个明显放错桶的内部 flag 拆出来

如果继续同时改隐藏组，

会把这刀扩大成整页重排。
