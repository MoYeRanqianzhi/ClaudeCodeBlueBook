# 插件刷新三层专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经写过：

- `/plugin` 是插件包管理面
- `/reload-plugins` 让当前 session 吃到变更
- `/plugin`、`/mcp`、`/skills`、`/hooks`、`/agents` 是不同暴露面

但仍缺一层专门回答：

- 安装、启用、待刷新、当前会话激活为什么不是同一状态
- 为什么交互主线故意不自动 refresh
- 为什么 `/reload-plugins` 刷新的其实是插件带来的暴露结果，而不是“重新安装插件”

如果这层不拆，正文会继续犯两类错：

- 把 `/reload-plugins` 写成安装器
- 把 pending / needsRefresh 写成“插件坏了”或“插件已生效”

## 苏格拉底式自审

### 问：为什么不把这批继续塞进“插件、MCP、技能、Hooks 与 Agents 运维”专题？

答：“插件、MCP、技能、Hooks 与 Agents 运维”专题已经交代了运维顺序，但重点仍是“哪个入口管什么对象”。本轮更偏对象状态机：

- 意图层
- 物化层
- 激活层

如果继续往这页堆，会把入口分工和状态分层混成一页。

### 问：为什么不把这批继续塞进命令执行语义页？

答：11 号页回答的是 slash command 对象、执行语义与可见性，不适合承载：

- `needsRefresh`
- background install auto-refresh
- SDK reload_plugins control request
- 当前 session Layer-3 swap

这批更像“扩展控制面中的状态层”。

### 问：正文里最该避免的假并列是什么？

答：

- 安装完成 vs 当前 session 已激活
- `/plugin` vs `/reload-plugins`
- pending toggle / pending update vs active components
- 交互 REPL 手动刷新 vs headless / background 自动刷新
- slash command `/reload-plugins` vs SDK `reload_plugins`

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- `clearAllCaches()`、`clearPluginCacheExclusions()` 等具体缓存清理细节
- `pluginReconnectKey` bump 后谁重新响应
- seed marketplace、zip cache、background install 的内部时序
- remote 模式下 settingsSync 的精确触发条件

这些都是维护判断，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- 插件至少有意图、物化、当前会话激活三层状态
- `/reload-plugins` 是当前 session refresh primitive，不是安装器
- 交互主线故意不自动刷新，只提示用户显式运行 `/reload-plugins`
- `/reload-plugins` 刷新的其实是 commands / agents / hooks / plugin MCP / LSP 的暴露结果
- SDK 与 REPL 在语义上共享同一刷新能力，但入口不同

### 不应写进正文的

- 具体清理哪些缓存
- `pluginReconnectKey` 的内部传播细节
- background install 与 headless install 的内部时序
- remote 模式下 settingsSync 的具体条件判断

这些继续只放 `docs/userbook-memory/`。

## 本轮从正文下沉回 memory 的实现细节

### Layer-3 refresh 的直接源码证据

- `refresh.ts` 明确写了三层模型：
  - Layer 1: intent
  - Layer 2: materialization
  - Layer 3: active components
- `/reload-plugins` 与 background/headless refresh 共用这一 primitive。

正文可以稳写三层模型；但缓存清理与 reinit 细节继续放 memory。

### 交互主线为什么不自动 refresh

- `useManagePlugins` 的注释明确写了：交互模式只发通知，不自动 refresh。
- 旧 auto-refresh 路径曾有 stale-cache 和覆盖不全问题；现在统一要求 `/reload-plugins` 走完整 Layer-3 swap。

正文应写：

- 交互主线故意不自动刷新

但不应复述历史 bug 细节。

### 自动 refresh 的特例

- 新 marketplace install 的 background path 会尝试 auto-refresh。
- headless / CCR 也有自己的安装与 refresh 流。
- SDK callers 走 control request，不走文本 prompt。

正文可把它们降级为“特定宿主的自动或结构化刷新路径”；具体 callsite 留在 memory。

## 后续继续深入时的检查问题

1. 当前变化属于意图层、物化层，还是当前 session 激活层？
2. 这条提示是在说“已安装”，还是“还没应用到当前 session”？
3. 当前宿主是交互 REPL、SDK，还是 headless / remote？
4. 我是不是把 `/reload-plugins` 写成了安装器？
5. 我是不是把 pending / needsRefresh 写成了已生效结果？

只要这五问没先答清，正文就会继续把插件刷新写成一张假的“安装完成状态表”。
