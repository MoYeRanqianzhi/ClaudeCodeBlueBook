# surface-shell handoff hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经补完：

- `memory/05` 根 `README` 的暴露层级收窄
- `122-149` hub 的第二次分流 handoff
- `139-149 -> 144-149` 的 truth-shell handoff

之后，

当前最值钱的下一刀，

不是继续回根索引做曝光，

也不是继续新增更细目录，

而是把：

- `128-138-surface presence、bridge chain 与 interaction shell 家族/README.md`

也补成一张真正承担：

- `128-138 -> 133-138`

前向 handoff 的父 hub。

## 为什么这轮值得单独提交

### 判断一：现在缺的不是 `133-138` 这个入口，而是父 hub 还没把人送进去

`133-138`

这个更窄的子家族已经存在。

而且 `122-149` 在上轮之后也已经明确承担了：

- `122-127`
- `128-138`
- `139-149`

三段分流。

但 `128-138` 目前仍停在：

- 给出 `215`
- 给出若干 leaf 记忆
- 再给远处的后继回跳

这会让读者虽然已经走对了层级，

却还得自己判断：

- 现在是不是该直接进 `133-138`

所以当前缺口已经不在“有没有子家族”，

而在：

- 父 hub 还没承担 next-hop 责任

### 判断二：目录优化在这一轮表现为“补树内连通”，不是“继续加层”

如果这时回根索引再挂：

- `133-138`

会把刚收窄好的根入口重新写宽。

如果这时继续再抽一个更细的新 hub，

又会把真正的问题掩盖掉：

- 已有的父 hub 没把子 hub 送出去

更稳的最小动作其实是：

1. 在 `128-138` 里明说：
   - 这里就是从 `122-149` 进来后完成 `128-132 / 133-138` 分流的地方
2. 在 `133-138` 小节里
   明说：
   - 如果你已经确认自己要在 schema-store / bridge-shell 这半段继续浏览，就直接跳进子家族

这样目录结构没有新增层级，

但父子层之间的职责终于闭合。

### 判断三：这轮同样在保护稳定/条件/灰度的阅读边界

`128-138`

这层稳定保护的是：

- `214 -> 215 -> 204`
  的阅读骨架
- `128-132`
  与 `133-138`
  不是两份无关旧批次

它不该把：

- presence 状态词
- schema/store helper
- bridge/runtime 名

升格成公开能力入口。

所以当前最稳的 handoff 写法也必须明确：

- 父 hub 只签骨架与 next-hop
- 子 hub 才继续展开三条两步后继线

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `128-138` hub
   - 增加“从 `122-149` 进来后，这里负责完成 `128-132 / 133-138` 分流”的说明
2. 在 `128-138` hub 的 `133-138` 小节里
   - 增加直达 `133-138` 子家族的显式入口
3. 新增这条 `401` 记忆
   - 记录为什么这轮优先补父 hub 的 handoff，而不是继续回根索引做曝光

这样：

- `122-149 -> 128-138 -> 133-138`
  会形成连续的 hub handoff 链
- 根入口的收窄不会损失 findability
- 目录结构优化也和第一性原理一致：上一层只把你送到下一层，不替下一层继续说话

## 苏格拉底式自审

### 问：为什么这轮不是继续让根索引重新挂回 `133-138`？

答：因为那会把已经收窄好的层级重新写平。当前问题不是找不到 `133-138`，而是 `128-138` 还没明确承担把人送过去的责任。

### 问：为什么这轮不是继续抽一个更细的新 hub？

答：因为下一跳入口已经存在。继续加目录只是在制造新名词，不能解决父 hub 缺 handoff 的问题。

### 问：为什么这也算“加强对使用功能的剖析”？

答：因为 userbook 的使用剖析依赖逐层收窄。`128-138` 如果不能明确告诉读者何时该继续进 `133-138`，读者就会重新退回按页碰运气，而不是沿着 surface -> schema-store / bridge-shell 的功能边界下钻。
