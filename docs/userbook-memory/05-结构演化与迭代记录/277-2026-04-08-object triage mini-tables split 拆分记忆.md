# object triage mini-tables split 拆分记忆

## 本轮继续深入的核心判断

在已经完成：

- task-matrix object-triage cleanup

之后，

当前矩阵层缺的，

已经不是：

- 再补更多一句话提醒

而是：

- 把三组高频对象错位压成可扫描的迷你对照表

## 为什么这轮仍落点选 `03-参考索引/05`

### 判断一：first-hop 分诊如果只停留在提示句，扫描效率仍然不够高

前一轮已经补了：

- 规则句
- 修了几条误导性的 `次选入口`

但对真正站在矩阵页上的读者来说，

最需要的是：

- 一眼看清这几个入口分别在改什么对象

而不是：

- 先读完长句再自己做重组

所以这轮仍然应该继续压：

- `03-参考索引/05`

## 为什么这轮改成 mini-tables，而不是继续加 bullets

### 判断二：这几组对象本来就共享同一个比较维度

这三组高频易混入口本来就都在回答同一类问题：

- 什么时候用
- 改什么对象
- 边界提醒是什么

如果继续只用 bullets，

读者就还得自己从长句里抽：

- 主语
- 动词
- 不要混写成什么

而迷你对照表可以直接把：

- 用途
- 对象
- 边界

放在同一横轴上。

## 为什么这轮只压三组，不继续扩成更多对象簇

### 判断三：这轮的目标是高频误判止血，不是把矩阵页改成第二张合同速查表

当前最容易复发的三组对象错位是：

- `setup-token` vs `/login`
- `/desktop` vs `/session` vs `/remote-control`
- `/plugin` vs `/reload-plugins` vs `/skills|/hooks|/agents|/mcp`

如果这一轮继续把更多对象簇都塞进来，

矩阵页就会开始向：

- 第二张 `03-参考索引/06`

膨胀。

所以这轮只压：

- 高频且已在前文反复出现的三组

## 为什么 `/mcp` 也要显式进入第三张对照表

### 判断四：只写插件刷新链还不够，外部 server 管理面也常被误并到同一层

`/plugin`

和：

- `/reload-plugins`

主要回答：

- 包管理
- 当前 session 刷新

而：

- `/mcp`

回答的是：

- 外部 MCP server 管理

如果不把 `/mcp` 一起纳入同表，

读者会继续把：

- 插件包本体
- 插件刷新
- 外部 server 管理

写成同一种扩展操作。

## 苏格拉底式自审

### 问：我现在缺的是再加一句边界提醒，还是把比较维度直接结构化出来？

答：如果用户已经站在矩阵页，当前更缺的是结构化比较，而不是更多长句。

### 问：我是不是又想把矩阵层写成第二张合同速查表？

答：如果继续扩太多对象簇，这页就会膨胀成第二张 `06`，所以这轮只收三组最高频误判。

### 问：我是不是把 `/mcp` 又留在表外，默认它不会被和插件刷新链混写？

答：如果不把 `/mcp` 一起纳入第三张对照表，扩展运维对象还是会被重新压平。
