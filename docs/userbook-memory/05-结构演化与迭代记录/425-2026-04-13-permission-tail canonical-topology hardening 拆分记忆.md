# permission-tail canonical-topology hardening 拆分记忆

## 本轮继续深入的核心判断

上一刀已经把 `203`

定型成：

- successor skeleton / anti-overlap map

但并行复核又暴露出一个更细的结构风险：

- 读者仍可能把 `202` 听成和 `198/199/201` 同层并列的第五个尾页

原因不是 `203` 没写出 `202` 与 `201` 的关系，

而是：

- 这层关系出现得太晚
- 页首与骨架表还没有把 canonical 拓扑公式一眼写死

所以这轮继续深入的最小高价值动作，

不是再扩 leaf 证明，

而是把 `203`

页首可见的 branch formula 改成：

- `196 -> 198 / 199 / (201 -> 202)`

## 为什么这轮值得单独提交

### 判断一：上一刀虽然把 `203` 从 leaf-proof 摘要拉成结构页，但还没有完全压住“编号即线性主链”的误听

`203`

现在已经明确说：

- `196` 是根页
- `198` 是 closeout
- `199` 是 re-evaluation
- `201` 是 host sweep
- `202` 是 persist write surface

可如果页首只写成：

- `196 -> 198 / 199 / 201 / 202`

读者还是可能顺手把：

- `202`

听成第五个同层 sibling。

这轮补的，

就是把：

- `202` 是 `201` 之后才显形的 persist zoom

写成页首可见的 canonical 公式。

### 判断二：这一刀继续保护“结构页只负责骨架，不负责把 leaf 页重新讲平”

`202`

并不是要被降回不重要，

而是要被放回更准确的位置：

- 它不是与 `198/199/201` 完全并列的尾页族成员
- 它是在 sandbox-network 这条支线里，
  由 `201` 的 same-host settle 再往后长出的 persist write-surface 后继

这能继续避免把：

- closeout
- local re-evaluation
- host sweep
- persist zoom

重新压成同一层“permission tail 细节”。

### 判断三：这也是第一性原理式收束，而不是实现细节微调

从第一性原理看，

读者先该问的不是：

- helper 到底先后怎么调

而是：

- 这几个编号到底是不是同层对象

只要这一层还没钉死，

后面的 helper / queue / cleanup / settings 细节就会再次串层。

所以把：

- `196 -> 198 / 199 / (201 -> 202)`

放到页首和骨架表里，

本质上是在继续保护结构位阶。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `203-...四种后继问题.md`
   - 把页首公式改成 `196 -> 198 / 199 / (201 -> 202)`
   - 把骨架表同步成 `201 -> 202` 的嵌套关系
   - 把结论同步成 `202` 不是第五个同层尾页，而是 `201` 之后的 persist zoom
2. 新增这条 `425` 记忆
   - 记录为什么这轮优先补 canonical 拓扑公式
3. 更新 `memory/05 README`
   - 把最新批次索引推进到 `425`

## 苏格拉底式自审

### 问：为什么这轮不去改 `201/202` 各自页首？

答：因为当前最高 ROI 还是先把 `203` 这张总图钉死。结构总图不稳时，直接去修 leaf 页页首，会先撞进局部承接句与全局 canonical 关系混写的问题。

### 问：为什么 `202` 不能继续被写成第四条或第五条并列尾线？

答：因为 `202` 讨论的是 persist write-surface propagation，它要以 sandbox-network settle 这条支线已经出现为前提；把它写成同层 sibling，会把 `201` 的 host sweep 与 `202` 的 persist zoom 写平。

### 问：为什么这轮也算“加强对使用功能的剖析”？

答：因为用户最终读到的不是 helper 名，而是后继问题的分层关系。把 canonical branch formula 提前写死，正是在增强 permission tail 这组使用功能的可解释性。
