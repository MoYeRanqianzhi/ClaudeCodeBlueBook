## 为什么这轮落点选 `03-参考索引/05-任务到入口速查矩阵.md`

### 判断一：矩阵是高频前门，不能把 hooks 写成“看见就能稳定插”

当前边界索引和控制面长文已经明确：

- `/hooks` 只是当前 session 的 UI 观察面
- hooks 还分加载、注册、执行、UI 四层
- 实际执行会继续受来源信任、plugin-only、managed-only、`disableAllHooks` 等策略改写

但任务矩阵这一行仍写成：

- `在关键事件插钩子 | hooks | 无 | 稳定`

这会把：

- 能看到 hooks 对象

误写成：

- 能稳定插钩并执行

## 这轮具体怎么拆

### 判断二：只改这一行的入口说明和稳定性，不扩写成长文

这轮只做一条矩阵修补：

1. 把首选入口从笼统的：
   - `hooks`
   收窄成：
   - `hooks（配置面）`
2. 把稳定性从：
   - `稳定`
   改成：
   - `条件公开，/hooks 仅是 UI 观察面，执行受来源信任与策略门控`

这样：

- 前门不会再把 hooks 写成稳定执行面
- 但仍保留“要插 hooks 时从哪类对象进入”的快速提示

## 为什么这轮不顺手改 `/skills` `/hooks` `/agents` 观察面那一行

### 判断三：那一行回答的是“看”，这一行回答的是“插”

`查看当前 skills / hooks / agents`

这行关注的是：

- 看当前可见面

而：

- `在关键事件插钩子`

这行关注的是：

- 注册与执行

真正被写平的是后者，不是前者。

## 苏格拉底式自审

### 问：我是不是因为 `/hooks` 对普通用户可见，就把 hooks 执行面也一起写成稳定公开？

答：不能。可见 UI、可注册对象、可实际执行是三层不同问题。

### 问：我是不是该把整张矩阵里的扩展层都一起重写？

答：不需要。这轮唯一高信号残留就是 hooks 这一行把边界写高了。

### 问：我是不是把 hooks 说得太灰，反而让读者以为它完全不能用？

答：也不对。正确写法不是“不可用”，而是“条件公开，且执行受门控”，这样才能同时保护稳定面和灰度边界。
