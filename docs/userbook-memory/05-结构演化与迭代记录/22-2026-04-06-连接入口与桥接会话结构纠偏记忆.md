# 连接入口与桥接会话结构纠偏记忆

## 本轮继续深入的核心判断

我最初把一个结构缺口判断成：

- `04-专题深潜` 应再补一篇“连接入口、桥接会话与宿主健康检查专题”

继续复核后发现，这是一个假缺口。

真正的问题不是：

- `04` 缺了一页

而是：

- 顶层 `README` 的 `04-专题深潜` 职责列表发生了漂移
- 把已经落到 `05` 的控制面主题继续挂在 `04` 名下

因此本轮应做的是：

- 纠偏目录职责与导航

而不是：

- 再造一篇会重新混层的 `04/20`

## 苏格拉底式自审

### 问：为什么最初会误判成“应该补 `04/20`”？

答：因为顶层 `README` 的“`04-专题深潜` 按真实工作目标组织”下面，仍然列着：

- 连接入口、桥接会话与宿主健康检查
- 设置面板、诊断屏与运营命令
- 命令对象、执行语义与可见性
- 技能来源、可见性与触发
- system-init、SkillTool
- Trust Dialog / plugin-only policy
- hooks / MCP / plugin refresh / startup trust / headless 等

这会制造一个假象：

- 好像 `04` 还缺一串应写未写的主题页

但这些主题实际上已经稳定落在 `05`。

### 问：是什么证据说明这不是缺页，而是漂移？

答：

- `05/21` 已经覆盖 host / viewer / health-check 的对象边界
- `05/22` 已经覆盖 trust / project approval / health-check 的边界
- `04/13`、`04/16`、`04/18`、`04/19` 已经覆盖自动化、会话接续、CLI 启动、root/slash 分层
- `03/02/10`、`03/02/11` 也已形成速查入口

因此读者缺的不是：

- 再来一篇把这些主题打平重讲的 `04` 长文

而是：

- 一份不误导的目录职责
- 一条更清楚的阅读路径

并行源码收敛还进一步确认了两条事实：

- `claude remote-control` / `/remote-control` 的核心仍是 bridge host，而 standalone bridge fast-path 依赖此前已建立 trust
- `assistant [sessionId]` 的核心仍是 viewer attach，而不是本地再开一条普通工作线程

这两条都支持：

- 继续留在 `05` 作为对象边界文

而不是：

- 回填成 `04` 的平行正文

### 问：为什么这批不能继续保留 `04/20` 草稿？

答：因为那会把已经明确沉到 `05` 的对象边界，又重新拉回 `04`。

这会重新制造三种混层：

- 把控制面边界文伪装成任务文
- 让 `04` 与 `05` 的目录职责再次重叠
- 让读者误以为这一组主题在两层都应完整展开

### 问：这批最该防的假并列是什么？

答：

- “`04` 缺失连接专题” = “必须新增 `04/20`”
- “README 有这行职责” = “目录里就该再有一篇正文”
- “读者需要更好路由” = “必须再造一篇长文”

如果不先拆开这三层，目录优化就会重新变成正文扩张。

## 本轮结构决策

本轮实际应做的是：

- `bluebook/userbook/README.md`
- `bluebook/userbook/00-阅读路径.md`
- `bluebook/userbook/04-专题深潜/README.md`
- `docs/userbook-memory/05-结构演化与迭代记录/README.md`

核心动作：

- 清理顶层 `README` 中 `04`/`05` 职责漂移
- 保持 `04` 只描述真实工作目标
- 保持 `05` 只描述控制面与对象边界
- 用阅读路径把读者导向现有稳定文档，而不是补一篇重复长文

### 本轮不应提交什么

- 不应提交新的 `04/20` 正文
- 不应把已写在 `05` 的对象边界再复制回 `04`
- 不应因为“结构看上去缺一项”就新增平行专题

## 正文与记忆边界

### 应写回正文/导航的

- `04` 与 `05` 的真实职责
- 现有稳定专题应该如何被路由
- 哪条阅读路径最适合宿主/viewer/health-check 一组误判

### 只应保留在记忆的

- 为什么我最初误判成要补 `04/20`
- 哪个 README 条目制造了假缺口
- 哪些现有页面已经足够覆盖，不应再复制
- “下一篇真正该写什么”这一类后续候选

## 本轮特殊注意

### 结构纠偏比补新正文更重要

当目录职责已经漂移时，先修职责，再决定要不要补正文。

否则新正文很容易只是：

- 对错误目录结构的迎合
- 而不是对用户问题的真正增量回答

### 阅读路径可以新增，但正文不能为路径而造页

如果读者确实需要一条专门的路径：

- 可以在 `00-阅读路径.md` 里新增一条

但这不自动推出：

- 还要新增一篇长文来配路径

## 后续继续深入时的检查问题

1. 我看到的“缺页”是真缺，还是导航漂移造成的假缺？
2. 这个主题的核心是用户目标，还是控制面边界？
3. 现有正文是否已经足够回答，只是路由不够清楚？
4. 我现在是在优化结构，还是在为了编号继续扩正文？
5. 哪些判断应该回到 `docs/userbook-memory/`，而不是再写进 `bluebook/userbook/`？

只要这五问没先答清，结构优化就会重新退化成：

- 重复造页
- 目录继续混层
- 正文再次混入作者记忆
