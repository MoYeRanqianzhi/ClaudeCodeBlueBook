# streamlined-root hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经把：

- `112`
- `113`

这条 streamlined 后继线的 stable / conditional / gray 分账统一起来之后，

`105-110`

这一支里最值得优先补的，

不是再去新建局部 hub，

而是回到：

- `109-headless print 的 streamlined output 是 pre-wire rewrite.md`

把这张 root page 自己从旧式的：

- 当前源码能稳定证明什么
- 当前源码不能稳定证明什么

抬升成和 `112/113`

一致的：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

三层口径。

## 为什么这轮值得单独提交

### 判断一：`109` 是 streamlined 支线的根页，根页口径落后会反过来拖慢叶页

`109`

回答的是：

- streamlined output 为什么属于 pre-wire rewrite

而不是：

- terminal semantics 后处理

既然它是这条支线的 timing root，

那它如果还停在“能证明 / 不能证明”的旧尾部，

就会让：

- `112`
- `113`

这些已经成熟的叶页，

反而比根页更像完整合同。

### 判断二：当前更值钱的是把 “conditional pre-wire rewrite” 写成稳定对象

这页最容易被误写成：

- 所有 `stream-json` 都天然等于 streamlined output

这轮真正要保住的稳定层是：

- conditional pre-wire rewrite

最需要降到条件或灰度层的是：

- gate 是否被打开
- `null` suppression 是否发生
- 别的宿主会不会继续让 streamlined 扮演同一种替换角色
- exact helper 顺序与 feature wiring

这比继续补新导航，

更直接服务于：

- 保护稳定功能
- 隔离灰度实现

### 判断三：这轮也在继续优化根页与叶页的术语一致性

当前明确不再给 `105-110` 再裂目录，

而是让：

- `106`
  维持相近的三层表达
- `109`
  跟上最近统一过的 `112/113`

这本身就是在减少支线里的命名漂移。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `109-...pre-wire rewrite.md`
   - 把原来的“稳定证明 / 不能稳定证明”改写成
   - `稳定可见`
   - `条件公开`
   - `内部/灰度层`
2. 新增这条 `413` 记忆
   - 记录为什么这轮优先补 root page 合同厚度，而不是去做 `108/110` 或新增导航
3. 更新 `memory/05 README`
   - 把最新批次索引推进到 `413`

## 苏格拉底式自审

### 问：为什么这轮不是先去做 `108`？

答：因为 `108` 当然也仍有价值，但 `109` 是 streamlined 支线的 timing root。先把根页口径统一，能给 `112/113` 提供更稳的上游参照。

### 问：为什么这轮不是直接去做 `110`？

答：因为 `110` 更像 exclusion-reason split 的叶页；若 `109` 这张根页还停在旧尾部，叶页再成熟也仍缺上游锚点。

### 问：为什么这也算“优化相关目录结构”？

答：因为结构优化不只等于新增 README。当前让根页与叶页共享同一套分层术语，本身就在降低目录系统里的口径漂移。
