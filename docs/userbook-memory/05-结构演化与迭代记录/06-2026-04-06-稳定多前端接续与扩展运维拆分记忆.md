# 稳定多前端接续与扩展运维拆分记忆

## 本轮继续深入的核心判断

`bluebook/userbook/` 已经把自动化、初始化与账户面拆开，但还有两类稳定用户入口仍然被旧专题混写：

- 多前端接续
- 扩展运维

如果不拆，读者会在两个地方持续犯错：

- 把 app 下载、当前会话 handoff、remote viewer、remote host 混成一个“远程入口”
- 把 plugin、mcp、skills、hooks、agents、reload-plugins 混成一个“扩展入口”

## 苏格拉底式自审

### 问：`/mobile` 和 `/desktop` 都算前端入口，它们为什么不能放在同一层描述里就完事？

答：因为对象不同。`/desktop` 是当前会话 handoff；`/mobile` 只是 app 获取面。继续把它们并写，会让读者误把下载动作当成 session continuation。

### 问：`/session` 与 `/remote-control` 都在处理跨前端继续工作，为什么不能写成一组稳定主线？

答：因为一个是 viewer 接续，一个是 host 宿主；前者依赖 remote mode 已建立，后者则在建立 bridge 宿主。两者工作对象和风险前提都不同。

### 问：`/plugin`、`/skills`、`/hooks`、`/agents` 为什么不能继续放在“扩展与集成专题”里粗讲？

答：因为它们分别在管理包、管理 server、看可见能力、看事件插针、看代理配置；如果只写“扩展层怎么选”，读者仍然不知道日常运维到底该进哪个面。

### 问：`/reload-plugins` 为什么值得被单独写进 userbook，而不是留在命令索引？

答：因为它回答的是“当前 session 什么时候真正吃到插件变更”这个高频问题。它不是安装，也不是管理包，而是当前会话刷新面。

### 问：哪些内容必须写进记忆，而不能混回正文？

答：

- 为什么旧专题会误导的作者侧诊断
- “如果不拆会继续导致什么误判”的维护性判断
- 目录继续演化的拆分策略

这些是作者维护知识，不是读者工作知识。

## 结构决策

本轮新增两篇专题深潜：

- `16-IDE、Desktop、Mobile 与会话接续专题.md`
- `17-插件、MCP、技能、Hooks 与 Agents 运维专题.md`

并保留旧页，但让它们的职责更明确：

- `06-前端与远程专题` 继续作为宽口径总页，但不再独占稳定多前端接续叙事
- `04-扩展与集成专题` 继续作为选择层总页，但不再独占日常运维叙事

## 同步纠偏规则

### 前端接续线

- `/ide`、`/desktop`、`/mobile` 应从“条件前端”里分离出来
- `/session` 写成 remote mode 下的条件接续面
- `/remote-control`、`/remote-env`、`/chrome`、`/voice` 保持条件/隐藏标注

### 扩展运维线

- `/plugin` 写成插件包管理面
- `/mcp` 写成 server 管理面
- `/skills`、`/hooks`、`/agents` 写成当前可见配置面
- `/reload-plugins` 写成当前 session refresh 面

### 参考索引纠偏

本轮应同步修正至少这些误写：

- `/init-verifiers` 不再写进普通稳定代码工作流
- `/rate-limit-options` 不再写成普通公开命令
- `/session` 标明 remote mode only
- `/remote-control` 标明条件/隐藏
- `/feedback` 标明受 provider / policy / privacy level 约束

## 后续纪律

继续深化时，优先问：

1. 这页是在按“用户目标”拆对象，还是在按“源码子系统”堆功能？
2. 这条命令是在管理包、管理 server、看当前暴露面，还是刷新当前 session？
3. 这条前端入口是在 viewer 侧、host 侧、handoff 侧，还是 device-prep 侧？

只要这三个问题没有先答清，正文就还会继续混写。
