## 为什么这轮值得单独提交

### 判断一：当前问题不是缺少路线，而是根前门又在重复维护一条局部路线

根 `README` 当前这条写法已经手写了：

- `01-主线使用/04-会话、恢复、压缩与记忆.md`
- `04-专题深潜/02-连续性与记忆专题.md`

然后再补一句：

- 如果你主要卡在旧会话怎么找回来，再看 `04-专题深潜/12-会话发现、历史检索与恢复选择专题.md`

但 `00-阅读路径.md` 已经有完整对应的：

- `路径 93：我想让长任务稳定续上，而不是每次回来都重讲`

并且它正好把这三跳整理成一条顺序明确的阅读路径。

所以根前门这里真正的失配，

不是“缺内容”，

而是：

- 重复维护半条路线
- 让目标入口和阅读路径同时承担同一导航职责

### 判断二：这条目标入口最适合交给阅读路径，而不是继续在根页局部展开

这一条的难点在于它天然横跨两个问题：

- 当前工作对象如何续上
- 以前的工作到底在哪里、该恢复哪个会话

如果根前门继续自己手写，就容易不断长出：

- 第一跳
- 第二跳
- 例外说明
- 补充第三跳

而 `路径 93` 已经把这两个子问题压在同一条宽路径里。

所以这轮更稳的做法不是继续润色根页，

而是把根页明确降回：

- 目标入口提示

把真正的顺序导航交给阅读路径页。

## 这轮具体怎么拆

### 判断三：把整条手写局部路线收成对 `路径 93` 的单点 handoff

这轮只做一件事：

1. 把：
   - `01-主线使用/04 ... -> 04-专题深潜/02 ...`
   - 以及后面的 `04-专题深潜/12 ...` 补充说明
   收成：
   - `00-阅读路径.md` 里的“路径 93：我想让长任务稳定续上，而不是每次回来都重讲”

这样：

- 根前门不再重复维护局部路线
- 路由责任重新回到阅读路径页
- “续上当前工作对象”和“找回旧会话”这两个子问题也重新落回同一条精确路径

## 为什么这轮不顺手改“找回过去的会话与提问”
