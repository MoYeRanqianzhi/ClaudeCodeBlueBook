# 2026-04-06 交互效率与会话发现专题拆分记忆

## 本轮结构判断

继续深入后，出现了另一类拥挤点：

- 交互效率相关能力散落在命令索引、前端专题和主线页里
- 会话发现、历史检索、恢复选择又与“当前会话运营”混在一起

这会让正文持续出现两个误导：

1. 终端交互只是皮肤和偏好，不值得单独分析
2. 找回旧 prompt、找回旧 session、运营当前 session 是同一类动作

这两条都不对。

## 新的拆页纪律

### 1. 会话运营与会话发现必须分开

以后以下动作默认仍属于“当前会话运营”：

- `/rename`
- `/copy`
- `/export`
- `/branch`
- `/rewind`
- `/clear`

而以下动作应单独写成“会话发现与恢复选择”：

- `/resume`
- resume picker
- custom title 搜索
- prompt history search
- session preview

### 2. 终端交互效率必须从“设备与前端”里剥离出来

以下内容不应再只当作“界面杂项”：

- `/theme`
- `/color`
- `/vim`
- `/keybindings`
- `/statusline`
- prompt history recall

它们影响的是长期使用效率，应有自己的用户目标长文。

### 3. `/statusline` 的正文写法必须纠偏

以后不要把 `/statusline` 写成简单状态查看命令。

更稳的写法是：

- 它是 prompt command
- 会调用专门 agent 去生成或改写状态栏配置
- 目标是“持续显示”，不是“临时查询”

## 本轮去记忆化要求

本轮如果要解释“为什么把交互效率和会话发现拆出来”，这段理由只应留在记忆层。

正文里只写：

- 用户目标
- 第一性原理
- 正确入口
- 运行时合同
- 误用边界

## 下一轮继续深入时优先怀疑点

1. 是否又把 prompt history 和 session recovery 写成一类。
2. 是否又把终端交互层写成“UI 杂项”。
3. 是否又把当前会话对象操作和过去会话对象发现混成一页。
