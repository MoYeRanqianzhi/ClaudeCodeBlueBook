# 来源信任与插件锁定专题拆分记忆

## 本轮继续深入的核心判断

之前 userbook 已经分别写过：

- 扩展层如何选择
- 技能来源与暴露面
- Claude 与 remote client 如何看见可用能力

但仍缺一层专门回答：

- 为什么同样是 skills/hooks/mcp/agents，不同来源会被分级信任
- Trust Dialog 与 plugin-only policy 为什么不是同一类门
- frontmatter hooks / MCP servers 为什么会在“对象还在”的情况下继续被裁掉

如果这一层不拆，正文会持续把三种不同性质的限制混成一个“安全限制包”：

- 工作区信任
- 来源信任
- 执行期门控

## 苏格拉底式自审

### 问：为什么不把这些内容继续塞进“权限、沙箱、审批与安全边界”？

答：因为那一页的对象是工具权限与沙箱治理，而这一批的对象是扩展来源信任。继续往里加，会把 tool approval、sandbox、workspace trust、source trust 混成一个安全大杂烩。

### 问：为什么不把这些内容继续塞进“扩展与集成专题”？

答：因为扩展专题回答的是“为什么选 skills/plugin/MCP/hooks 这一层”，不是“同一层为什么还按来源分级信任”。这需要独立控制面页。

### 问：为什么这一批适合补到 `03-参考索引/02-能力边界`？

答：因为这批本质上是在回答“什么来源是稳定主线、什么来源受组织锁定、什么来源只是管理员例外”，更像边界索引，不像扩展对象索引。

### 问：正文里最该避免的假并列是什么？

答：

- Trust Dialog vs plugin-only policy
- skill / agent 对象存在 vs frontmatter hooks / MCP 仍可注册
- `allowManagedHooksOnly` vs `disableAllHooks`
- plugin-only surface lock vs 功能全局禁用

### 问：哪些内容必须继续沉到 memory，而不能进正文？

答：

- TrustDialog 当前精确扫描哪些 `loadedFrom/source` 组合来检测 Bash 风险
- plugin hooks 在 session start 里被跳过的具体时机
- 哪些 source 被列为 admin-trusted 的实现层理由，而不是用户心智层理由

这些都是维护判断，不是读者行动知识。

## 正文落笔纪律

### 应写进正文的

- Trust Dialog 是工作区信任边界
- plugin-only policy 锁的是来源 surface
- hooks 还有 managed-only / all-disabled 的专属总闸
- skills / agents 的 frontmatter hooks / MCP 仍会继续受来源信任判断影响

### 不应写进正文的

- 哪些条件组合恰好触发 TrustDialog 的内部实现细节
- admin-trusted source 名单未来是否会扩展
- 某些 root command 跳过 Trust Dialog 的精确帮助文案

这些继续只放 `docs/userbook-memory/`。

## 本轮从正文下沉回 memory 的实现细节

### Trust Dialog 的当前实现证据

- `TrustDialog.tsx` 当前会额外检查带 Bash 能力的 prompt 扩展，而不是泛化到“任意 skill 都危险”。
- 具体扫描包含两组实现分支：
  - `loadedFrom` 落在 skills / plugin 一侧的 prompt 扩展
  - `loadedFrom === 'commands_DEPRECATED'` 的旧式命令包装
- 上述分支还会结合 source 是否来自 `projectSettings` / `localSettings` / plugin，以及 `allowedTools` 是否含 `Bash(...)`。

这些细节保留为作者证据，正文只保留“Trust Dialog 重点防仓内可执行扩展”这一层判断。

### Trust Dialog 的非统一入口证据

- 当前实现里，部分非交互路径与若干 root command 会绕开 Trust Dialog。
- 这类路径改用警示文案或入口约束，而不是复用同一个交互确认对话。

正文因此只应写：

- Trust Dialog 是交互工作区信任边界
- 它不是所有入口统一必经的总闸

## 本轮边界保护

### 应明确写成工作区信任层的

- Trust Dialog
- 非交互/某些 root command 对 Trust Dialog 的绕过文案边界

### 应明确写成来源信任层的

- `strictPluginOnlyCustomization`
- `isSourceAdminTrusted()`
- admin-trusted source

### 应明确写成执行门控层的

- `allowManagedHooksOnly`
- `disableAllHooks`
- frontmatter hooks / MCP server 注册 gating

## 后续继续深入时的检查问题

1. 这条限制是在问“我信不信这个 workspace”，还是“我信不信这个 source”？
2. 这是对象加载边界，还是附带 hooks / MCP 注册边界？
3. 我是不是把 plugin-only policy 误写成了能力总禁用？
4. 这条 hooks 限制来自 policy source 还是 non-managed source？

只要这四问没先答清，正文就会继续把来源信任写成一锅模糊的“安全设置”。
