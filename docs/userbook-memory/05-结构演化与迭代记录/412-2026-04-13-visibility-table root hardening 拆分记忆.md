# visibility-table root hardening 拆分记忆

## 本轮继续深入的核心判断

在当前 worktree 已经把：

- `112`
- `113`
- `114`
- `115`

这几页的 stable / conditional / gray 分账统一起来之后，

`111-115`

这一支里剩下最明显的旧尾部，

就是：

- `111-builder transport、callback surface 与 UI consumer 不是同一张可见性表.md`

所以这轮更值钱的下一刀，

不是再给 `111-115` 新建 hub，

而是把 `111` 这张根页自身，

从旧式的：

- 当前源码能稳定证明什么
- 当前源码不能稳定证明什么

抬升成和 `112-115` 一致的：

- `稳定可见`
- `条件公开`
- `内部/灰度层`

三层口径。

## 为什么这轮值得单独提交

### 判断一：`111` 是这条线的根页，根页尾部如果停在旧口径，会反过来拖慢整条分支

`111`

回答的是：

- builder/control transport
- public/core SDK surface
- callback surface
- UI consumer

为什么根本不是同一张表。

既然它是这条线的 ladder root，

那它自己如果还停在“能证明 / 不能证明”的旧尾部，

读者就会继续把：

- `112`
- `113`
- `114`
- `115`

这些已经成熟的叶页，

误看成“局部页比根页还更成熟”的反向关系。

### 判断二：当前更值钱的是把“多张收窄表”写成稳定对象，而不是继续补导航

这页最容易被误写成：

- Claude Code 对外只有一张消息可见性表

这轮真正要保住的稳定层是：

- multiple narrowing visibility tables

最需要降到条件或灰度层的是：

- 别的宿主是否严格沿同一梯子工作
- family 在各层是否一一对应
- manager / adapter 的 exact helper 顺序与 host wiring

这比继续给 `111-115` 再加入口，

更直接服务于：

- 保护稳定功能
- 隔离灰度实现

### 判断三：这轮本身也在继续优化目录结构的职责分工

当前明确不再给 `111-115` 加新 hub，

而是让：

- 结构页负责 findability
- 根正文负责稳定合同
- 叶页负责局部细化

这本身就是在保护目录层收敛。

## 这轮具体怎么拆

这轮只做三件事：

1. 修改 `111-...可见性表.md`
   - 把原来的“稳定证明 / 不能稳定证明”改写成
   - `稳定可见`
   - `条件公开`
   - `内部/灰度层`
2. 新增这条 `412` 记忆
   - 记录为什么这轮继续补根页合同厚度，而不是再裂目录
3. 更新 `memory/05 README`
   - 把最新批次索引推进到 `412`

## 苏格拉底式自审

### 问：为什么这轮不是继续去做 `108` 或 `110`？

答：因为它们当然也还有价值，但 `111` 是这条线的根页。先把根页尾部口径统一，能给后续所有同支页面提供更稳的参照面。

### 问：为什么这轮不是跳去 `security/112`？

答：因为 `security/112` 是另一条线；而 `111-115` 这条控制面支线当前已经被连续补到只剩根页尾部口径没统一，局部完成度更高，也更适合独立提交。

### 问：为什么这也算“优化相关目录结构”？

答：因为结构优化不只等于新增 README。当前让根页与叶页共享同一套术语，本身就在减少目录系统里的命名漂移。
