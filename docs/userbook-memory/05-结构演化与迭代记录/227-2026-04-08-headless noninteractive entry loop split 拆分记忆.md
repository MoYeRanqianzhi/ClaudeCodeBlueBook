# headless noninteractive entry loop split 拆分记忆

## 本轮继续深入的核心判断

headless / 非交互这条线当前真正缺的不是：

- 新的编号总图
- 新的叶子页

而是：

- 一个能把 `03 -> 04 -> 05` 接顺的前门

现状里，正文已经把很多关系讲清了：

- `04-专题深潜/13` 解释 `-p/--print`、`--bg`、`--sdk-url`
- `05-控制面深挖/20` 固定 headless 启动宿主
- `207/208/209` 已经把结果收束、summary / tail / restore、wire / projection 各自收成结构页

但入口层还缺一件事：

- 把这几层从“散落的长文”收成一条能直接浏览的高频使用路径

所以这轮最稳的做法不是：

- 再开 `207-209` 的编号范围图

而是：

- 新增一个物理入口 `05-控制面深挖/非交互结果、summary 与协议流/README.md`
- 再把 `README`、`03-参考索引/06`、`03-参考索引/02-能力边界/README`、`04-专题深潜/13`、`04-专题深潜/18`、`05-控制面深挖/README` 接到这个前门

## 为什么这轮应该补“物理前门”

### 判断一：用户先问的是“从哪里继续读”，不是“92-110 的树还该不该再压一层”

只要用户已经分清：

- 这是非交互入口
- 这是后台会话
- 这是协议流

下一步最常见的问题就不再是：

- “这些功能分别叫什么”

而是：

- “接下来该往哪条 deeper line 继续走”

如果没有物理前门，

读者就会被迫在：

- `207`
- `208`
- `209`
- `20`

之间自己重建入口顺序。

这说明缺的是：

- browseable entry

不是：

- more numbered structure

## 为什么 `207/208/209` 要被收成一组

### 判断二：这三页解决的是同一条非交互深线的三个后继问题

这三页并不并列回答三个随机问题。

它们更像：

1. `207`：结果如何从多账本前提收束到真实交付
2. `208`：summary / terminal tail / observer restore 怎样继续分叉
3. `209`：wide wire、raw wire、callback 与 streamlined projection 怎样再分叉

如果入口层不先把这三层拆开，

读者最容易继续把：

- `-p/--print`
- `--bg`
- `--sdk-url`
- `stream-json`
- `post_turn_summary`
- `streamlined_*`

重新混写成同一种“自动化模式”。

所以这轮新入口保护的不是某个单页，

而是：

- 这条高频使用线的继续深入顺序

## 为什么这轮顺手改 `03/06`

### 判断三：高价值入口速查页必须承认 `-p/--print`、`--bg` 与 `--sdk-url` 是不同入口

`03-参考索引/06-高价值入口运行时合同速查.md` 原来已经写：

- `/commit`
- `/commit-push-pr`
- `/review`
- `/status` / `/doctor` / `/usage`
- `/session` / `/remote-control`

但没有把：

- `-p/--print`
- `--bg`
- `--sdk-url`

收进高价值入口合同。

这会让非交互入口在速查层里失去位置，

仿佛它们只是：

- 进阶 flags

而不是：

- 独立运行时合同

所以这轮顺手把这组三个入口也提到速查层。

## 为什么这轮不去大改 `00-阅读路径`

### 判断四：先修闭环主链，再谈路径层大排版

`00-阅读路径` 里已经有 headless 深线，

只是入口偏叶子化、顺序还不够前置。

如果这轮同时去大改：

- 路径编号
- 大段排序
- 更多 topic 之间的相互前导

就会把一次本来只该补前门的切片，

做成：

- 大规模导航 churn

这轮不需要承担那层风险。

更稳的顺序是：

1. 先让 `03 -> 04 -> 05` 主闭环成立
2. 等这条入口跑顺后，再决定是否重排 `00-阅读路径`

## 本轮最关键的新判断

### 判断五：高频使用功能的目录结构优化，优先级高于继续压叶子页

之前几轮已经解决了不少：

- range handoff
- bridge branch split
- stable-gray wording

这轮说明另一件事：

- 即使叶子分析已经够深，入口层如果没有把高频使用路径收成闭环，读者仍然会退回叶子页里自己拼结构

所以这轮优先补：

- 入口闭环

而不是：

- 再加更多 leaf-level 证明

## 苏格拉底式自审

### 问：我现在缺的是更多分析，还是缺一个让分析被读到的前门？

答：如果正文已经存在，但读者还要自己在叶子页里重建顺序，那缺的是前门。

### 问：我是不是因为 `207/208/209` 已经是结构页，就又想再造一张更大的结构图？

答：如果新页只是为了把三页重新挂一遍，那它应该是物理入口，不该是新的编号页。

### 问：我是不是把 `-p/--print`、`--bg`、`--sdk-url` 又写成了同一种自动化模式？

答：只要不先分入口、后台生命周期、协议流可见性，这个误写就会复发。
