## 为什么这轮不该再造编号页

### 判断一：结构图已经够了，缺的是物理浏览入口

当前文档已经把逻辑结构讲清：

- `218` 固定 `168-179`
- `219` 承接 `180-190`
- `197 / 203 / 204 / 206` 继续承接 bridge / remote 的后继问题

所以读者现在更容易卡住的不是：

- 不知道逻辑上怎么接

而是：

- 在 `05-控制面深挖/` 的平铺文件里，不知道哪个物理入口最适合先点进去

因此这轮最稳的做法不是再发明一个新编号，

而是补一个：

- browseable physical entry

## 为什么不在这一轮移动旧页

### 判断二：先给物理入口，再谈历史页搬迁

当前 `05-控制面深挖/` 里已经存在一些 accidental 目录，

但把旧页批量搬进新目录会立即引入：

- 链接大面积改写
- 历史编号页物理路径变化
- 与现有阅读路径、README、记忆的同步 churn

这轮不需要承担那一层风险。

所以更稳的顺序是：

1. 先补一个稳定目录入口
2. 让它只做导航，不改动现有编号页路径
3. 之后若真有必要，再单独评估物理搬迁

## 本轮最关键的新判断

### 判断三：目录层也该区分“逻辑结构已闭合”和“物理入口仍缺失”

之前的几轮已经解决了：

- 结构图

这会导致不必要的编号膨胀。
