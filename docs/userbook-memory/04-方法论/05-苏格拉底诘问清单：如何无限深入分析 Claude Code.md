# 苏格拉底诘问清单：如何无限深入分析 Claude Code

这一页不提供答案，而提供下一轮继续深挖时必须反复追问自己的问题。

## 一、关于入口

1. 这个功能的真正入口是什么：普通提示词、命令、工具、技能、插件还是远程 UI？
2. 用户看到的入口和真正执行入口是不是同一个？
3. 这个命令是 `local`、`local-jsx` 还是 `prompt`，为什么？

## 二、关于能力

1. 这是稳定公开能力，还是条件公开能力？
2. 如果当前会话里看不见它，是被权限过滤、被 `isEnabled()` 关闭，还是被 feature gate 消除了？
3. 这个能力有没有更低协调成本的正式入口？

## 三、关于治理

1. 这个能力在哪一层被约束：工具过滤、调用审批、规则文件、计划模式还是远程策略？
2. 如果把它从治理面拿掉，会出现什么错误顺序或越权行为？
3. 用户是否被鼓励走一条更安全的默认路径？

## 四、关于连续性

1. 这个功能如何在长任务中继续发挥作用？
2. 它是否与压缩、恢复、记忆、后台任务有关？
3. 如果会话中断，这个功能的状态能否恢复？

## 五、关于扩展

1. 这是 Claude Code 内核能力，还是通过 MCP/插件/skills/hooks 注入的能力？
2. 如果把外部扩展拔掉，最小可退化行为是什么？
3. 这个扩展层是否已经被产品化，还是只是一层实现影子？

## 六、关于多执行体

1. 这个能力是单线程语义，还是多代理语义？
2. 如果多个执行体同时使用它，会不会互相踩上下文、文件或责任边界？
3. 是否需要 worktree、task list 或 team context 才能安全使用？

## 七、关于“源码可见”

1. 我看到的是实现、注册点、门控影子，还是注释暗示？
2. 我的结论是事实，还是从门控与注释推出来的推论？
3. 这个推论在发布构建里有多大把握成立？

## 八、关于 userbook 自身

1. 这章是在帮助用户做决定，还是只在复述源码结构？
2. 这章是否把稳定面和灰度面混写了？
3. 这章有没有告诉用户不该误用什么？
4. 这章如果只留一句话，最有价值的判断是什么？

## 结论

“无限深入”不是无限堆字，而是不断把下面三件事做得更锋利：

1. 入口判断更准。
2. 能力边界更清。
3. 用户决策成本更低。
