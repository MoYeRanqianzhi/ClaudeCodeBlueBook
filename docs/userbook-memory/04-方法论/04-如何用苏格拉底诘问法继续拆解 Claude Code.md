# 如何用苏格拉底诘问法继续拆解 Claude Code

这章不是在解释 Claude Code 本身，而是在定义一套继续深挖它的方法。

## 一、先问对象，不问实现

错误问法：

- 这个文件是干嘛的？
- 这个模块为什么这么大？

更好的问法：

- 这个系统真正要保护的对象是什么？
- 它到底在维护什么不变量？
- 它把什么当成权威面？

例如：

- 对 `processUserInput`，真正的问题不是“怎么分流”，而是“为什么输入必须先被制度化”。
- 对 `tools.ts`，真正的问题不是“有哪些工具”，而是“为什么工具池要先被治理再给模型看”。
- 对 `compact`，真正的问题不是“怎么压缩”，而是“系统到底想保住什么继续资格”。

## 二、先问失败，再问成功

很多源码设计只有在失败路径里才暴露真实意图。

可持续使用的诘问模板：

1. 如果这里失败，会破坏什么？
2. 系统为什么在这一层而不是下一层阻止它？
3. 哪条错误路径被显式封死了？
4. 还有哪些近邻错路没有被封死？

## 三、先问边界，再问能力

Claude Code 的功能很多，但真正决定其产品气质的是边界：

- 什么情况下模型看不到某工具。
- 什么情况下命令不进入模型。
- 什么情况下计划模式不能开启。
- 什么情况下恢复不允许直接继续。
- 什么情况下灰度能力只留下影子。

所以继续深挖时，优先追问：

- 这条能力的边界在哪里？
- 边界由谁决定？
- 边界变化会不会改变用户主线？

## 四、先问组合，再问单点

一个单点功能往往解释不了 Claude Code 的真实设计。真正重要的是组合关系：

- 命令如何和技能、工具、hooks 组合。
- 计划模式如何和 permissions、worktree、teammate 审批组合。
- compact 如何和 session memory、resume、prompt cache 组合。
- agent 如何和 task、team、cron、remote trigger 组合。

当一个功能看起来“只是个按钮”时，追问它和哪几条链耦合，通常会看到更深的结构。

## 五、把每个结论都分成三类

继续写 userbook 时，应把结论显式标成三类：

### 直接事实

源码能直接证明。

### 工程推论

源码没有直说，但结构和注释强烈支持。

### 能力边界判断

不是在说“怎么实现”，而是在说“普通用户是否应稳定依赖”。

这样能显著降低把灰度影子误写成稳定功能的风险。

## 六、专门追问“为什么不用更简单方案”

这是最有效的苏格拉底问题之一。

例子：

### 为什么不用纯聊天输入，而要有 `local-jsx`

因为 Claude Code 需要本地控制面。

### 为什么不用纯弹窗审批，而还要计划模式

因为审批只能拦动作，不能强制先做设计。

### 为什么不用更大上下文窗口，而还要 compact 和 session memory

因为连续性问题不只来自 token 上限。

### 为什么不用 Git branch，而还要 worktree

因为 branch 解决的是历史指针，不是物理执行面隔离。

## 七、下一轮最值得继续追问的八个问题

1. Claude Code 的查询循环到底把什么视为“一个完整行动回合”？
2. 什么时候系统会让模型少看工具，而不是少用工具？
3. compact 到底保留了哪些对象，丢弃了哪些对象？
4. session memory 与普通 memory 文件的角色分工是什么？
5. plan 文件在审批、恢复、远程和多代理之间扮演什么共同引用面？
6. agent 的 isolation、cwd、background 三个字段之间的真实边界是什么？
7. MCP、技能、插件、hooks 四层扩展最小可行组合分别是什么？
8. 哪些灰度能力已经对当前架构形成反向约束？

## 结论

真正有效的源码剖析不是“知道更多文件名”，而是不断追问：

- 系统在保护什么对象。
- 系统为何把边界设在这里。
- 系统在什么地方拒绝简单方案。

只要沿这三条线追问，Claude Code userbook 就可以持续无限深入，而不会沦为一次性的功能清单。
