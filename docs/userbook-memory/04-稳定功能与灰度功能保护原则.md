# 稳定功能与灰度功能保护原则

这页的目标只有一个：避免 userbook 把“普通用户可稳定依赖的能力”和“源码里可见但条件化/内部化的能力”混成一张清单。

## 原则 1：先看公开主路径，再看实现细节

一个能力先要满足：

- 出现在主注册路径。
- 没有明显的 ant-only 条件。
- 没有强依赖 feature gate 才出现。

之后才有资格先被写作稳定面。

## 原则 2：出现不等于承诺

下面几种情况都不该被写成稳定功能：

- 在源码中有名字，但被 gate 包裹。
- 在源码中有命令，但依赖账户、平台或策略。
- 在注释或分析文档里被提到，但实现缺失。

## 原则 3：影子能力必须明确标“推论”

对于只剩：

- `require()` 占位
- 注释
- TODO
- 缺失实现文件

的能力，只能写“存在证据表明系统朝这个方向设计”。

## 原则 4：内部能力要保护普通用户阅读体验

内部能力不是不能写，而是不应挤占公开主线。

因此文档中应：

- 先给出稳定主线。
- 再给出条件公开。
- 最后单列内部/灰度/影子能力。

## 原则 5：章节内部也要分层

即便在单个专题里，也应明确区分：

- 当前用户可以直接依赖的部分。
- 需要验证环境前提的部分。
- 只能当作源码趋势观察的部分。

## 原则 6：默认路径与条件路径不能并列写成主线

如果一条稳定路径和一条条件路径同时存在，不要用“或”把它们写成一个默认工作流。

例如：

- `/review` 是默认路径，`/ultrareview` 是条件补充路径。
- `/session` 是默认接续路径，`/remote-control` 是条件补充路径。

## 一句话判据

如果一个读者照着这页 userbook 去建立工作流，而当前能力其实大概率拿不到，那这章就把灰度能力写过头了。
