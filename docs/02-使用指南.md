# 使用指南

## 先说结论

如果你想把 Claude Code 用到接近源码设计预期，最重要的不是会写更长的 prompt，而是顺着它的 runtime 设计来工作：

1. 把任务状态外化
2. 把上下文当稀缺资源
3. 把 subagent 当受限 worker，而不是第二个主脑
4. 把权限做细粒度控制
5. 把扩展能力装进 Skills、Plugins、MCP，而不是堆进主提示词

## 一、把任务状态外化

源码里规划和执行是分离的。`TodoWrite`/`TaskCreate` 不是装饰物，而是显式状态管理手段。

关键证据：

- `src/tools/TodoWriteTool/TodoWriteTool.ts:31`
- `src/tools/TodoWriteTool/TodoWriteTool.ts:65`
- `src/tools/TaskCreateTool/TaskCreateTool.ts:48`
- `src/tools/TaskCreateTool/TaskCreateTool.ts:80`

实践建议：

- 长任务先拆 checklist，再开始执行
- 多阶段任务持续更新状态，不要只在脑中维护
- 并发任务使用任务系统或多 agent 明确边界

原因很简单：Claude Code 本身就是靠这套机制，把“思考”“执行”“回收结果”分开管理的。

## 二、fork 与 fresh subagent 要分开用

`AgentTool` 的 prompt 文案写得很直白：

- fork 适合不想把中间噪音留在当前上下文的任务
- fresh subagent 适合需要一个全新、隔离视角的 worker

关键证据：

- `src/tools/AgentTool/prompt.ts:76`
- `src/tools/AgentTool/prompt.ts:85`
- `src/tools/AgentTool/prompt.ts:99`
- `src/tools/AgentTool/prompt.ts:112`

源码导出的最佳实践可以直接翻译成使用规则：

### 适合 fork 的任务

- 开放式研究
- 并行独立调查
- 会产生大量中间输出的实现任务

### 适合 fresh subagent 的任务

- 需要独立视角的 review
- 需要明确职责边界的专职 worker
- 需要特定 agent type 的任务

### 写 subagent prompt 的规则

- 不要只下命令，要交代目标和背景
- 不要把理解工作甩给 subagent
- 要给文件路径、行号、边界条件、预期产物

一句话：subagent 要像“交给一位新进房间的聪明同事”，不是“把任务丢给另一个你”。

## 三、并行要做，但不要乱并行

源码在 Agent prompt 里明确鼓励并发启动多个 agent，以提高整体吞吐。

关键证据：

- `src/tools/AgentTool/prompt.ts:242`
- `src/tools/AgentTool/prompt.ts:248`

但并行的正确方式是：

- 并行独立问题，不并行互相依赖的问题
- 并行研究、审查、扫描、验证
- 对写入同一块代码的实现任务要先切职责边界

如果任务会互相覆盖上下文或文件，就不是真正的并行收益，而是并发噪音。

## 四、优先使用 background 和 worktree 做隔离

`AgentTool` 的 schema 直接暴露了：

- `run_in_background`
- `isolation: "worktree"`
- 在部分能力下还存在 `isolation: "remote"`

关键证据：

- `src/tools/AgentTool/AgentTool.tsx:81`
- `src/tools/AgentTool/AgentTool.tsx:90`
- `src/tools/AgentTool/AgentTool.tsx:99`
- `src/tools/AgentTool/prompt.ts:263`
- `src/tools/AgentTool/prompt.ts:272`

使用建议：

- 可能跑很久的任务放后台
- 有文件改动风险的并发实现放 worktree
- 需要全新沙箱的长任务再考虑 remote

这与传统工程实践一致：并行开发先隔离工作目录，再回收结果。

## 五、不要把权限系统当摆设

源码里的权限不是 UI 装饰。真正的授权链条包括：

- mode 组装
- allow/deny 规则
- auto mode classifier
- hooks
- 无 UI 场景降级

关键证据：

- `src/utils/permissions/permissionSetup.ts:510`
- `src/utils/permissions/permissionSetup.ts:597`
- `src/utils/permissions/permissions.ts:473`
- `src/utils/permissions/permissions.ts:688`

实战建议：

- 少用粗放全局 allow
- auto mode 下尤其不要试图 YOLO 全开
- 对危险工具给精细粒度规则
- 需要自动化时，把规则写清楚，而不是指望模型自行克制

从源码看，官方设计预期就是“细粒度授权 + 自动分类 + 运行时判定”。

## 六、背景任务结束后，优先看输出文件

`TaskOutputTool` 在源码里已经明显处于兼容层位置，不适合再作为新的主工作流依赖。

关键证据：

- `src/tools/TaskOutputTool/TaskOutputTool.tsx:144`
- `src/tools/TaskOutputTool/TaskOutputTool.tsx:158`

建议：

- 把输出文件当主回收面
- 把任务通知当完成信号
- 不要反复轮询或 tail 中间产物污染上下文

## 七、把扩展能力放到 Skills、Plugins、MCP

Claude Code 的可扩展面不是单一插件机制，而是三层：

1. Skills：为某类任务封装 prompt、工具权限、frontmatter
2. Plugins：可启停、可携带 skills/hooks/MCP
3. MCP：把外部工具能力挂入 Claude Code，或反向把 Claude Code 暴露出去

关键证据：

- `src/skills/loadSkillsDir.ts:78`
- `src/skills/loadSkillsDir.ts:185`
- `src/plugins/builtinPlugins.ts:7`
- `src/plugins/builtinPlugins.ts:52`
- `src/entrypoints/mcp.ts:59`

使用建议：

- 团队规范优先沉淀成 project-level skills
- 复用能力优先做 plugin
- 工具互联优先走 MCP，而不是手写碎片胶水

## 最后一句

源码里最重要的用户指导其实不是一句 prompt 技巧，而是一条工作原则：

把 Claude Code 当成一个带状态、带权限、带扩展面、带并发控制的开发 runtime 来使用，而不是一个会写代码的聊天框。
