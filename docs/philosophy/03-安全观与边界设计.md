# 安全观与边界设计

## 1. 先说结论

Claude Code 的安全观不是“尽量少给能力”，而是“能力必须在边界内流动”。

它的边界至少有四层：

1. trust / init 边界
2. permission mode 边界
3. classifier / hook / policy 边界
4. transport / remote / MCP 状态边界

## 2. trust 先于完整能力

`init.ts` 明确区分：

- trust 前只能应用 safe env vars
- 完整环境变量和 telemetry 在 trust 后才继续

证据：

- `claude-code-source-code/src/entrypoints/init.ts:62-79`
- `claude-code-source-code/src/entrypoints/init.ts:240-303`

这说明 Claude Code 的安全边界不是只在工具执行时才出现，而是从启动阶段就开始了。

## 3. mode 不是体验开关，而是安全边界

`transitionPermissionMode()`、`prepareContextForPlanMode()`、`transitionPlanAutoMode()` 说明：

- mode 会改写 context
- 进入 auto 时会 strip dangerous rules
- 离开 auto 时会 restore dangerous rules
- plan mode 还可能叠加 auto 语义

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:505-553`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:597-645`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:1457-1532`

所以 permission mode 不是 UX toggle，而是 policy transform。

## 4. 为什么它特别警惕“危险 allow rule”

从源码看，它真正警惕的不是“工具强”，而是“工具在 classifier 之前被直接放行”。

例如：

- `Bash(*)`
- `Bash(python:*)`
- `PowerShell(iex:*)`
- `Agent(*)`

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:84-147`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:149-233`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:235-245`

这说明它的安全观非常清楚：  
真正危险的是“绕开仲裁机制”，而不是“能力本身存在”。

## 5. classifier 在这套系统里扮演什么

classifier 不是万能裁判，而是 auto mode 下的一段 runtime stage。

它前面还有：

- safetyCheck
- PowerShell 特判
- acceptEdits fast-path
- safe-tool allowlist

它后面还有：

- denial tracking
- denial-limit
- unavailable 时 fail-open / fail-closed 分支
- headless fallback

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:518-686`
- `claude-code-source-code/src/utils/permissions/permissions.ts:688-926`
- `claude-code-source-code/src/utils/permissions/permissions.ts:929-951`

所以 Claude Code 的安全机制不是“把大模型再套一个小模型”，而是把 classifier 嵌进整体状态机。

## 6. 远程与 MCP 也有显式边界

MCP server 连接状态不是简单 connected / disconnected，而是：

- connected
- failed
- needs-auth
- pending
- disabled

证据：

- `claude-code-source-code/src/services/mcp/types.ts:180-222`

RemoteIO 也不是裸传输，它会：

- 注入 session ingress token
- 在 CCR v2 下接管 internal event 持久化
- 在 bridge 模式下发送 keepalive

证据：

- `claude-code-source-code/src/cli/remoteIO.ts:54-93`
- `claude-code-source-code/src/cli/remoteIO.ts:111-168`
- `claude-code-source-code/src/cli/remoteIO.ts:174-199`

这说明 Claude Code 不是把 remote 当“能力外延”，而是把它纳入安全和状态管理。

## 7. 从第一性原理看，这种安全观在回答什么

如果 agent 要在真实开发环境里长期工作，安全问题的本质不是“禁止行动”，而是：

1. 谁在什么边界内可以行动。
2. 哪些行动必须经过仲裁。
3. 没有交互条件时系统如何收口。
4. 远程/多进程/插件/外部工具如何不破坏这些边界。

Claude Code 的答案是：  
把这些问题都前置成 runtime 结构，而不是把希望寄托在 system prompt 里。

## 8. 一句话总结

Claude Code 的安全观不是减少能力，而是边界优先。它允许强能力存在，但要求强能力必须经过 mode、rule、classifier、hook、transport state 这些边界层。
