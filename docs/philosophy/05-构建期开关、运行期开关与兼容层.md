# 构建期开关、运行期开关与兼容层

这一章把前面分散出现的 `feature()`、GrowthBook、entitlement、shim 再往下拆一层。  
目标不是重复说“Claude Code 有很多 feature flag”，而是回答：为什么这些 gate 其实构成了一套产品演化方法。

## 1. 先说结论

Claude Code 的能力边界至少由三层机制共同决定：

1. 构建期开关：决定某些能力是否进入产物。
2. 运行期开关：决定进入产物后的能力是否对当前用户、组织、版本、生效环境开放。
3. 兼容层：在能力切换或协议迁移时，保证旧前端、旧会话语义、旧 CLI 版本不被瞬间打断。

关键证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-202`
- `claude-code-source-code/src/skills/bundled/index.ts:15-79`
- `claude-code-source-code/src/plugins/bundled/index.ts:1-23`

因此它不是简单的“开关多”，而是把产品变化本身做成了架构的一部分。

## 2. 第一层：构建期开关决定能力是否进入当前 build

### 2.1 `feature('BRIDGE_MODE')` 不是 UI 条件，而是产物边界

`bridgeEnabled.ts` 多次强调：

- 只有在 `feature('BRIDGE_MODE')` 为真时，相关字符串字面量与逻辑才应进入当前 build。
- 代码里甚至明确写了 positive ternary pattern，目的是让 dead code elimination 真正把外部 build 不需要的字面量裁掉。

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-36`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:160-172`

这说明 build-time gate 的职责不是“少显示一个按钮”，而是“改变产物形态和可见痕迹”。

### 2.2 bundled skills 也按 build feature 选择性装入

`initBundledSkills()` 先注册一批稳定技能，再按 feature 条件追加：

- `dream`
- `hunter`
- `loop`
- `scheduleRemoteAgents`
- `claudeApi`
- `runSkillGenerator`

证据：

- `claude-code-source-code/src/skills/bundled/index.ts:24-79`

如果从第一性原理看，这表示“生态能力也不是固定清单，而是可裁剪的产品层”。

### 2.3 bundled plugin 当前还是脚手架，本身也是一种 build-stage 信号

`plugins/bundled/index.ts` 明说：

- built-in plugin 是未来用于迁移一部分 bundled features 的脚手架
- 不是所有 bundled feature 都应成为 built-in plugin
- 复杂 setup 或自动启用逻辑仍应放在 `skills/bundled/`

证据：

- `claude-code-source-code/src/plugins/bundled/index.ts:1-23`

这反映的不是“插件系统还没写完”，而是团队在刻意区分“适合用户开关的能力”和“仍处于更强流程绑定中的能力”。

## 3. 第二层：运行期开关决定能力对谁开放、何时开放

### 3.1 entitlement gate：先判断用户是不是对的人

`isBridgeEnabled()` 和 `isBridgeEnabledBlocking()` 都先要求 `isClaudeAISubscriber()`，再去看 `tengu_ccr_bridge`。

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:16-55`

这意味着 bridge 不是“谁看到了谁就能点”，而是先要满足身份与登录前提。

### 3.2 diagnostic path：不是只给 false，而是给原因

`getBridgeDisabledReason()` 细分了至少四类失败：

- 没有 claude.ai 订阅
- token 没有完整 profile scope
- 缺少 organization 信息
- gate 尚未对账号打开

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:57-87`

这很重要。成熟产品不是只有 gate，还要有可行动的失败解释。

### 3.3 同一个能力内部还能继续切版本

bridge 本身可用，不等于所有 bridge 实现都打开。  
`isEnvLessBridgeEnabled()` 单独控制 env-less v2 REPL bridge；v1/v2 有独立版本地板与独立实现路径。

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:118-129`
- `claude-code-source-code/src/bridge/bridgeEnabled.ts:150-172`

这说明 runtime gate 不是只有“有或没有”，也包括“走哪一代实现”。

### 3.4 rollout 默认值也是运行时策略的一部分

bridge 相关默认行为还受：

- `getCcrAutoConnectDefault()`
- `isCcrMirrorEnabled()`

控制。

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:175-202`

因此 rollout 不只是放行，还包括默认策略、连接方式和镜像模式。

## 4. 第三层：兼容层负责迁移，而不是让旧世界立刻失效

### 4.1 `isCseShimEnabled()` 体现的是协议迁移思维

这段注释写得非常直白：

- v2 worker 会返回 `cse_*`
- 兼容层把它 retag 到 `session_*`
- 当前 frontend / validator 仍依赖旧语义
- 等 server 与 frontend 都升级后，shim 才能关掉

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:132-148`

这不是普通 feature flag，而是 migration shim。

### 4.2 版本地板本身也是兼容层的一部分

`checkBridgeMinVersion()` 允许 bridge 按最小版本滚动升级；如果 CLI 太旧，就明确告诉用户需要更新。

证据：

- `claude-code-source-code/src/bridge/bridgeEnabled.ts:150-172`

换句话说，兼容层不只做 ID 转换，也做版本边界管理。

## 5. 为什么这不是“很多 feature flag”，而是一种演化架构

如果只是普通 feature flag，通常只回答一个问题：

- 这个按钮今天开还是关。

但这里的三层机制分别回答的是：

1. 这个能力是否应该进入当前产物。
2. 这个产物中的能力是否该向当前用户/组织/环境开放。
3. 新旧实现并存时，如何让旧语义和旧客户端继续工作。

所以更准确的说法不是 feature flag，而是：

- artifact gating
- entitlement / rollout gating
- compatibility / migration layer

## 6. 从第一性原理看，为什么必须这样设计

如果 Claude Code 要同时面对：

- 外部公开构建
- 内部 ant build
- claude.ai 订阅用户
- console API 用户
- 第三方 provider 环境
- 本地 REPL、远程 bridge、镜像 session

那么它不可能靠“一张平面能力表”运营。

从第一性原理看，它必须同时解决四个不可约问题：

1. 产物安全：不该进入公开 build 的能力要在构建期就消失。
2. 运营灰度：该进入产物但不该对所有账号开放的能力，要能分批 rollout。
3. 迁移连续性：协议和 ID 语义切换时，旧系统不能瞬间全断。
4. 默认策略：一项能力开放以后，默认是否自动连接、是否镜像，仍需独立调控。

这四个问题合起来，才构成 Claude Code 的“演化方法”。

## 7. 苏格拉底式追问

### 7.1 如果只保留 runtime gate，不做 build-time gate，会怎样

那么 external build 里仍会包含不该公开的字面量、入口和死代码痕迹。  
对一个有 internal-only 能力和多 build 形态的系统，这是不够的。

### 7.2 如果只保留 build-time gate，不做 runtime rollout，会怎样

那每次试验都只能靠重新出 build。  
对于 entitlement、组织灰度、版本切换和默认连接策略，这显然太粗糙。

### 7.3 如果只做 rollout，不做 compat shim，会怎样

那协议一升级、ID 一换代、frontend 一滞后，旧链路就会直接断。

### 7.4 如果所有能力都强行变成 plugin，会更好吗

`plugins/bundled/index.ts` 的注释已经给出否定答案。  
有些能力更像自动注入的技能或深度绑定主流程的运行时功能，不适合立刻做成用户手动开关。

## 8. 对蓝皮书的直接启发

这对蓝皮书目录结构有三个强约束：

1. `bluebook/` 只能写“能力空间的主线判断”，不能把所有 gate 细节塞进去。
2. `api/` 负责写字段、可用性、入口和控制面。
3. `philosophy/` 负责解释为什么要把变化本身工程化。

否则读者会把“代码里出现过”误读成“对外稳定公开”。

## 9. 事实与推论边界

直接事实：

- build-time `feature()` 广泛参与 bridge 与 bundled skill 装配。
- bridge 运行时至少受 entitlement、profile scope、organization、GrowthBook、版本地板、默认连接策略、mirror 模式共同约束。
- `cse_*` 到 `session_*` 的 shim 明确存在，并带有后续退场条件。
- built-in plugin 目前仍处于 scaffolding 阶段。

较强推论：

- Claude Code 团队把产品演化视作架构问题，而不是运营补丁。
- bridge 是一个“多代实现并存”的长期演化面，而不是单次实验。
- bundled skill 与 built-in plugin 的区分，反映的是能力成熟度和用户控制面的不同设计阶段。

## 10. 一句话总结

Claude Code 的一个深层设计内涵是：  
它不仅工程化了 agent runtime，也工程化了产品变化本身。构建期开关负责裁剪形态，运行期开关负责灰度开放，兼容层负责迁移连续性。
