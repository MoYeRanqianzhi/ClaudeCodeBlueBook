# 权限系统与安全状态机

## 1. 先说结论

Claude Code 的权限系统不是“弹一个确认框”这么简单，而是一个多阶段状态机：

1. 启动前/配置期决定默认 mode。
2. mode 切换时重写 permission context。
3. 工具执行前做规则匹配与安全检查。
4. 在 auto mode 下决定是否走 classifier、fast-path 或人工确认。
5. 在 headless / async 场景下退化为 hooks 或 auto-deny。

关键证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:510-553`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:597-645`
- `claude-code-source-code/src/utils/permissions/permissions.ts:473-955`

## 2. 为什么它先讨论“危险规则”，而不是先讨论“允许什么”

`permissionSetup.ts` 最早的一批函数不是 UI 逻辑，而是在定义哪些 allow rule 会破坏 auto mode 的安全边界。

### 2.1 Bash 危险规则

以下规则会被视为危险：

- `Bash(*)`
- `Bash` 的空内容全放行
- `python:*`、`node:*` 等解释器前缀
- 其他能执行任意代码的模式

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:84-147`

### 2.2 PowerShell 危险规则

PowerShell 同样会拦：

- 全放行
- `iex`
- `invoke-command`
- `start-process`
- 嵌套 shell
- 其他明显的代码执行逃逸口

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:149-233`

### 2.3 Agent 危险规则

`Agent(*)` 级别的允许规则同样危险，因为它会让 subagent 在 classifier 之前就被自动放行。

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:235-245`

这背后的安全观很清晰：  
系统真正害怕的不是“工具会执行”，而是“执行在 classifier 之前就被绕开”。

## 3. 进入 auto mode 时到底发生了什么

`stripDangerousPermissionsForAutoMode()` 会：

1. 收集当前 `alwaysAllowRules`
2. 找出危险规则
3. 从上下文中移除这些规则
4. 把被移除的规则 stash 到 `strippedDangerousRules`

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:505-553`

而 `restoreDangerousPermissions()` 则在离开 auto mode 时把它们加回来。

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:555-579`

这不是“删配置”，而是“根据 mode 动态改写内存里的安全边界”。

## 4. `transitionPermissionMode()` 是真正的 mode 状态机

切 mode 时，Claude Code 不只改一个字符串。

它会统一处理：

- `plan` enter/exit
- `auto` activate/deactivate
- `prePlanMode` stash/restore
- dangerous rule strip/restore

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:581-645`

这说明 permission mode 是带 side effects 的 runtime transition，不是纯 UI 状态。

## 5. 默认 mode 是怎么选出来的

`initialPermissionModeFromCLI()` 会综合：

- CLI flag
- `dangerouslySkipPermissions`
- settings defaultMode
- GrowthBook circuit breaker
- remote 环境限制

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:686-790`

特别值得注意的是：

- auto mode 会被 circuit breaker 拦掉
- remote 环境不会接受所有本地 mode
- bypassPermissions 也会被策略层禁用

所以默认 mode 的本质是“策略求交集”，不是“按用户偏好直接生效”。

## 6. plan mode 和 auto mode 不是互斥，而是可叠加

`prepareContextForPlanMode()` 和 `transitionPlanAutoMode()` 表明：

- plan mode 可以带 auto 语义
- 这取决于用户 opt-in、gate 和 settings
- plan 中仍然可能重新 strip dangerous permissions

证据：

- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:1441-1455`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:1457-1493`
- `claude-code-source-code/src/utils/permissions/permissionSetup.ts:1495-1532`

这解释了为什么 Claude Code 的 mode 不是平面切换，而更像叠层状态。

## 7. 运行时真正的权限决策链

`hasPermissionsToUseTool()` 才是工具执行前的最终决策入口。

它的分支大致是：

1. 先拿基础决策 `hasPermissionsToUseToolInner(...)`
2. 如果已经 `allow`，直接通过并刷新 denial tracking
3. 如果 mode 是 `dontAsk`，把 `ask` 强制改成 `deny`
4. 如果是 auto / plan+auto，尝试 classifier 路径
5. 如果是 headless/async 且不能弹窗，先跑 hooks，再 auto-deny

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:473-518`
- `claude-code-source-code/src/utils/permissions/permissions.ts:929-951`

## 8. auto mode 下不是“一律跑 classifier”

Claude Code 在 auto mode 里仍然会做多层 fast-path：

### 8.1 某些 safetyCheck 直接要求人工

如果一个 safety check 被标记为不可 classifier-approvable，它不会被 auto 直接吞掉。

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:526-548`

### 8.2 PowerShell 可能被强制挡在 classifier 之外

如果没有对应 feature 支持，PowerShell 会要求显式人工批准。

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:560-590`

### 8.3 acceptEdits fast-path

如果某个动作在 `acceptEdits` 语义下本来就会被允许，就跳过 classifier，避免多余开销。

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:593-649`

### 8.4 safe-tool allowlist

如果工具本身在 classifier safe allowlist 中，也直接放过。

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:658-686`

所以 auto mode 的真实含义不是“全部走大模型裁决”，而是“在 fast-path 用尽之后再用 classifier”。

## 9. classifier 真正负责什么

在 auto mode 下，classifier 负责：

- 读取当前消息上下文
- 看本次工具动作
- 决定 `allow / block / unavailable`
- 记录成本、token、阶段、时延、denial streak 等遥测

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:688-817`

如果 classifier block：

- 可能更新 denial tracking
- 可能触发 denial-limit 逻辑
- 可能回退到人工批准

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:818-926`

## 10. headless / async 场景如何收口

当前环境无法弹出 permission prompt 时，Claude Code 会：

1. 先尝试 `PermissionRequest` hooks
2. 如果 hooks 没给结论，再 auto-deny

证据：

- `claude-code-source-code/src/utils/permissions/permissions.ts:929-951`

这也是为什么 background/async subagent 默认权限更严格：  
它们没有前台交互条件，就必须更保守。

## 11. 从第一性原理看，这套权限系统在解决什么

它解决的不是“怎么少弹几个框”，而是两个更本质的问题：

1. 如何让 agent 具备行动能力，但不把危险能力在策略层之外偷跑。
2. 如何让同一套运行时同时支持前台、后台、headless、subagent、remote 场景。

从这个角度看，Claude Code 的权限系统更像 runtime policy engine，而不是 UI permission dialog。

## 12. 一句话总结

Claude Code 的权限系统强在它把 mode、rules、classifier、hooks、headless fallback 全部做成了同一条安全状态机；真正的安全边界不在提示词里，而在运行时。
