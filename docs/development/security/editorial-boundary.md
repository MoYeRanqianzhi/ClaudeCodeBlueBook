# 安全专题正文与记忆边界

## 目标

把 `bluebook/security/` 重新收敛成真正的正文层，避免把章节推进计划、后续写作候选和维护者记忆继续混进主线论证。

## 正文允许保留什么

安全正文允许保留五类内容：

1. 当前要回答的安全问题
2. 对应源码证据、字段、状态机与对象边界
3. 第一性原理与苏格拉底式追问
4. 技术先进性、设计内涵与工程启示
5. 对下一代系统设计的直接机制性推论

## 正文不得继续混入什么

以下内容一律不再写入 `bluebook/security/` 正文：

1. “下一步最自然的延伸就是……”
2. “这一章之后还缺什么……”
3. “对目录结构与后续研究的启发……”
4. “`74-99` 应被视为一条连续小循环……”
5. “下一步最值钱的候选应切到……”
6. 任何直接指定未来章节编号、附录编号、目录编排动作的写作提示

这些内容属于研究推进记忆，不属于 Claude Code 安全机制本体。

## 应该分别写到哪里

- 章节推进、未来候选、目录编排：写入 `docs/development/security/long-term-memory.md`
- 每轮实际推进、证据回写、提交记录：写入 `docs/development/research-log.md`
- 全仓级目录职责或统一写作规则：写入 `docs/README.md`、`docs/development/03-反思与迭代准则.md`

## 判断准则

如果一句话回答的是：

- “Claude Code 安全机制是什么、为什么这样设计、源码如何实现”

它属于正文。

如果一句话回答的是：

- “蓝皮书下一篇写什么、哪张表该新增、哪条小循环该怎样继续推进”

它属于记忆层。

## 当前执行结果

本轮已经把 `bluebook/security/` 中统一模式的：

- `这一章之后还缺什么`
- `对目录结构与后续研究的启发`

这两类尾段从正文中移除，并把安全专题的推进记忆开始集中到本目录。
