# 研究方法

## 目标

本项目的目标不是生成“看起来很懂”的分析，而是持续产出可追溯、可验证、可解释的蓝皮书内容。

## 方法原则

### 1. 不确定即查

- 优先读本地源码
- 优先读源码注释、schema、入口、注册表
- 优先找调用链而不是猜接口意图

### 2. 以锚点为单位沉淀结论

每个重要结论尽量附：

- 文件路径
- 关键行号
- 为什么该锚点支持该结论

### 3. 结论分三级

- 事实：源码直接写了什么
- 解释：这些事实合起来说明什么设计
- 推论：对使用指南和产品哲学意味着什么

### 4. 先搭目录，再逐章深挖

蓝皮书不是一次性写完，而是：

1. 先建立稳定目录
2. 再用章节不断补全
3. 把开发文档当共享长期记忆

## 推荐分析路径

每个主题尽量按下面顺序走：

1. 入口文件
2. 注册表
3. 运行时协议
4. feature gate
5. UI/任务/持久化回收面

## 文档更新规范

- 新章节优先放到 `docs/`
- 研究过程、规划、证据索引放到 `docs/development/`
- 没有证据的判断，不写成定论
- 可疑能力标记为“可能 gated/internal”，不要误写成公开功能
