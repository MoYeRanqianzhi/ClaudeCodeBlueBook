# 研究方法

## 目标

本项目的目标不是生成“看起来很懂”的分析，而是持续产出可追溯、可验证、可解释的蓝皮书内容。

## 方法原则

### 1. 不确定即查，但先判证据模式

- `mirror present`
  优先读本地源码、源码注释、schema、入口、注册表与调用链。
- `mirror absent`
  优先读公开 artifact、owner page、证据索引与已归档锚点，不把通常存在于源码镜像里的对象写成当前已核实事实。
- `public-evidence only`
  只签 `contract / registry / current-truth claim state / provisional claim / consumer subset / gap note`，不替 runtime-core 缺口脑补 certainty。

更稳一点说，`不确定即查` 的第一问不是“先读哪个文件”，而是：

1. 当前 worktree 是否真的含有镜像。
2. 这条结论属于 mirror evidence，还是 public-evidence ceiling 下的保守判断。
3. 若镜像缺席，我现在写的是 `candidate`、`ceiling`，还是越界写成了 certainty。
4. 若新词与当前 canonical ladder 冲突，是否已经先回 `guides/102` 对齐 `claim-state / provisional claim / consumer subset / gap note`。

### 2. 以锚点为单位沉淀结论

每个重要结论尽量附：

- 文件路径
- 关键行号
- 为什么该锚点支持该结论

### 3. 结论分三级

- 事实：源码直接写了什么
- 解释：这些事实合起来说明什么设计
- 推论：对使用指南和产品哲学意味着什么

### 4. 先搭目录，再逐章深挖

蓝皮书不是一次性写完，而是：

1. 先建立稳定目录
2. 再用章节不断补全
3. 把开发文档当共享长期记忆

### 5. 先判发言权，再判写到哪一层

同一个结论在不同目录里拥有不同的合法位置：

- `philosophy`
  只判必要性与第一性原理。
- `architecture`
  只证对象链、chokepoint 与 current-truth writeback。
- `api`
  只承认 contract、schema 与 host-facing truth。
- `guides`
  只把判断压成问题梯子与模板。
- `playbooks`
  只对已承认对象链发 verdict。
- `casebooks`
  只保存 distortion 与伪证。
- `navigation`
  只路由与校正，不代言深页结论。
- `docs`
  只存记忆、过程与变更，不重判正文。

如果一个新增结论说不清自己是在“判必要性 / 证对象 / 定承认 / 发 verdict / 存伪证 / 存记忆”哪一层，它就还不配回写正文。

更稳的发言权判断还应至少回答三件事：

1. 这一层能合法说什么
2. 这一层不能改判什么
3. 如果读者或 later maintainer 不服，异议应往哪一层申诉

凡入口页说不清自己的上游 canonical owner 与异议去向，都应先按“第二叙述者风险”拒收。

### 6. 目录优化先判缺口，再决定动作

更稳的目录优化，不是先问“要不要新开一页”，而是先问：

1. 这次缺的是 canonical formula、first reject order、evidence ceiling，还是 execution verdict。
2. 现有 owner page 是否已经能承接，只是入口标签、route 顺序或申诉链没写清。
3. 这次改动能否减少“第一次回答同一问题”的入口数，而不是再造一个摘要页。
4. 如果删掉这页，哪条 first-answer noun、最小顺序或申诉链会真正消失。
5. 如果不用这层 README / 卡片 / bridge，而改走更简单方案，哪项不可约约束会失败。
6. 当前 worktree 缺镜像时，新入口会不会放大 second narrator risk，把 public evidence 写成 runtime certainty。
7. 新页是否真的承接了新的第一问或新的申诉位阶。

默认动作顺序应先按便宜到昂贵执行：

1. 改可见标签、H1、导语或 owner 声明。
2. 做 route trim、next-hop 修正或阅读顺序收口。
3. 补 speaking rights、`cannot overrule` 与 appeal chain。
4. 同步专题 README 与高流量索引页。
5. 只有前四步都不够，才准新开入口页。

如果答不上这七问，先把结论记进 `docs/`，不要急着升正文。

### 7. 先判是否该进正文，再决定是否只进记忆

更稳的判断是：

- 会改变 frontdoor formula、first reject path、对象链、verdict 顺序或目录发言权的，才配进正文。
- 只改变批次优先级、候选页、写作提醒、残留 debt 或尚未形成稳定 doctrine 的，先只进 `docs/`。
- 批次收口、推荐顺序、入口切换协议、兼容标签/路径债这类“编排真相”，即使稳定，也默认先只进 `docs/`；只有它真的改动 canonical chain / frontdoor formula / first reject path，才升正文。

如果一个结论离开当前批次就失去价值，它更可能是记忆，不是正文。

## 推荐分析路径

每个主题尽量按下面顺序走：

1. 入口文件
2. 注册表
3. 运行时协议
4. feature gate
5. UI/任务/持久化回收面

## 文档更新规范

- 新章节优先放到 `docs/`
- 研究过程、规划、证据索引放到 `docs/development/`
- 没有证据的判断，不写成定论
- 可疑能力标记为“可能 gated/internal”，不要误写成公开功能
- 新入口页若只是在重复已有前门，而没有新增稳定的对象、顺序、发言权或申诉链，优先拒收
- 新增入口前，先做更便宜的动作：先改可见标签，再补 `navigation`，再同步专题 README；只有现有入口确实承载不了新的第一问或新的申诉位阶时，才准新开前门
- 若当前 worktree 缺席镜像，入口页应先暴露 `evidence mode`，再暴露阅读顺序；不要把历史锚点、归档锚点或公开材料写成“当前本地已核验”
