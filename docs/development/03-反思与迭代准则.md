# 反思与迭代准则

## 目的

这份文档不是源码结论，而是给蓝皮书写作者自己的约束。

目标只有一个:

让后续迭代越来越像“可验证研究”，而不是“会写长文的总结”。

## 一、苏格拉底式自问清单

每写完一章，都先问四件事：

1. 这段最先会错在哪。
2. 现在哪条证据还不够。
3. 哪层边界被混写了。
4. 如果源码升级，哪一处会先失效。

## 二、本轮发现的改进方向

### 1. 不能只偏爱架构

第一阶段已经能回答“为什么 Claude Code 强”，但对“接口怎么暴露、宿主怎么接、消息怎么流动”的支持还不够。

### 2. 不能把能力边界写模糊

凡是 `feature()`、GrowthBook、policy、account entitlement 参与的能力，都必须明确写边界。

### 3. 不能只写概念，不写状态机

Claude Code 很多真正的价值在状态流转里:

- permission request
- task lifecycle
- MCP connection states
- session restore
- compact / reactive compact

后续必须多写状态机、多写时序，而不只是多写形容词。

### 4. 不能把主线蓝皮书写成字段百科

`bluebook/` 应该越来越像“高密度主线判断”，字段级索引、算法级机制和 gate 细拆应继续下沉到 `api/`、`architecture/`、`philosophy/`。

### 5. 不能低估“状态面”

Claude Code 不只是一套控制面和执行面，它还有非常重的状态面：

- transcript
- session metadata
- layered memory
- session memory
- file history / rewind

如果后续写作继续偏爱“能看见动作的部分”，蓝皮书会系统性低估 Claude Code 的可恢复性设计。

### 6. 不能把 SDK 写成文本输出面

如果后续只写 assistant reply、最终 result，而不写：

- `stream_event`
- `tool_progress`
- `task_*`
- `auth_status`
- `session_state_changed`
- `files_persisted`

就会把 Claude Code SDK 误写成“把模型回答嵌进 UI”的薄封装，而不是把 runtime 事件流暴露给宿主的集成面。

### 7. 不能把 MCP 写成工具列表

如果继续只按“有哪些 server / tool”来写 MCP，就会漏掉它真正更重要的四层结构：

- config scope
- transport
- connection state
- control actions

后续必须持续把 MCP 写成连接平面，而不是插件附属清单。

### 8. 不能把 Claude Code 写成 terminal shell

如果继续只从：

- REPL
- slash command
- assistant 输出

理解 Claude Code，就会低估：

- `control_request` / `control_response`
- `control_cancel_request`
- `interrupt`
- `get_context_usage`
- `mcp_set_servers`
- remote session / direct connect / bridge host

后续必须持续区分聊天外壳、transport plane 与 runtime control plane。

### 9. 不能把协议全集直接等同于宿主实现

如果继续只看：

- `controlSchemas.ts`
- subtype 列表
- `StructuredIO`

而不看：

- bridge 实际支持面
- direct-connect 实际支持面
- `RemoteSessionManager` 实际支持面

就会把“协议全集”和“适配器子集”写成一回事。

后续必须在每一处宿主分析里显式标出：

1. schema 全集
2. 控制平面主路径
3. 当前适配器子集

### 10. 不能只写 happy path，不写失败路径

如果继续只写：

- 正常 `control_request`
- 正常 `control_response`
- 正常权限批准

而不写：

- `control_cancel_request`
- unknown subtype error
- outbound-only error
- abort reject
- auth recovery dropping

蓝皮书就会把时序真相抹平。

后续必须持续把成功路径与失败路径并排写清。

### 11. 不能把闭环写成单向 RPC

如果后续只写：

- request subtype
- response payload

而不写：

- 后续 `SDKMessage`
- `session_state_changed`
- `status`
- `task_*`
- `elicitation_complete`

就会把 Claude Code 的宿主协议误写成普通 RPC，而不是 control loop。

后续必须持续把 request / response / follow-on message 一起写成闭环。

### 12. 不能把运行时真相写成单通道

如果后续只写：

- `SDKMessage`
- `stream_event`
- `task_*`
- `status`

而不写：

- `worker_status`
- `requires_action_details`
- `external_metadata`
- adapter 的 consumer subset

蓝皮书就会把 Claude Code 误写成“让宿主自己从消息里猜状态”的系统。

后续必须持续区分：

1. 事件时间线
2. 最新快照
3. 恢复后如何重建
4. 哪些 consumer 实际消费哪些信号

### 13. 不能把 prompt 魔力写成神秘文案

如果后续只摘：

- system prompt 句子
- coordinator prompt 片段
- AgentTool prompt 话术

而不写：

- prompt 装配链
- 角色分层
- attachment 注入
- prompt cache 稳定性
- fork / worker / coordinator 合同差异

蓝皮书就会退回成“prompt 评论”，而不是 runtime 分析。

后续必须持续区分：

1. 文案内容
2. 装配顺序
3. 角色分层
4. cache / token 约束

### 14. 不能把源码质量写成抽象赞美

如果后续只写：

- 代码很强
- 结构很先进
- 工程味很重

而不写：

- invariants 在哪
- cache 语义在哪
- 恢复语义在哪
- 依赖图是如何被维护的

这些判断就不可检验。

后续必须持续把“质量”拆成具体结构对象来写。

### 15. 不能把 continue 语义写成普通 retry

如果后续只写：

- 模型失败了会重试
- compact 后会继续

而不写：

- `State`
- `transition`
- tool follow-up continuation
- recovery self-loop
- policy continuation
- abort consistency

蓝皮书就会继续低估 `query.ts` 的 turn runtime 本体。

后续必须持续区分：

1. 正常下一 turn
2. recovery continuation
3. policy continuation
4. resume / continue 的外层恢复

### 16. 不能把 permission modal 写成权限系统本体

如果后续只写：
- ask UI
- approve / reject 按钮
- 交互弹窗措辞

而不写：

- typed decision engine
- `can_use_tool` control request
- deny rules / allowlist / mode 覆写
- hooks / classifier / sandbox / remote managed settings

蓝皮书就会把安全系统误写成 UI 行为。

后续必须持续区分：

1. 权限 renderer
2. 权限决策
3. 权限边界
4. 权限治理

### 17. 不能把目录继续写成“放什么”的列表，而不写“谁有权说什么”

如果后续还只会写：

- `philosophy` 放哲学
- `api` 放接口
- `playbooks` 放手册

却不写：

- 谁拥有必要性判断权
- 谁拥有对象承认权
- 谁拥有 verdict 签发权
- 谁只有翻译权、路由权或记忆权

目录就会继续看起来分层正确，却仍会在改写时长出第二叙述者。

后续必须持续把目录职责写成发言权限与申诉链，而不只是文件归类表。

更硬一点说，目录发言权还必须至少回答：

1. 我这一层能合法说什么
2. 我这一层不能改判什么
3. later maintainer 若不服，应往哪一层正式申诉

### 18. 不能把 future maintainer 写成价值口号

如果后续只写：

- 未来维护者也是正式消费者
- 代码要方便后来的人

却不写：

- later maintainer 是否拥有局部可反对性
- 不追全仓是否也能指出 authority / truth plane / danger surface / first retreat layer

那么这条判断就还不是制度能力，只是善意表态。

后续必须持续把 future maintainer 写成可执行的 reject 条件。

更稳的写法也应继续压成最小见证集：

- `authority surface + writeback seam + first retreat layer`

如果 later maintainer 连这三件都不能局部点名，“可维护”就仍只是作者视角。

### 19. 不能把什么都回写正文

如果后续把下列内容直接回写到蓝皮书正文：

- 本轮候选页
- 写作优先级
- 暂时还不稳定的桥接句
- 只对当前批次有价值的 debt 提示

正文就会重新退回施工日志。

后续必须持续先问：

1. 这会改变稳定 formula、对象链、first reject path 或发言权限吗
2. 如果不会，它为什么不该只进 `docs/`

这里还应再多拒一类内容：

- 批次收口
- 推荐顺序
- 入口切换协议
- 兼容标签 / 路径债

这些“编排真相”默认都先只进记忆层，不直接进正文。

### 20. 不能低估“新增入口”的治理成本

每加一个新入口，不只是在加一个目录节点，也是在给 later maintainer 增加一个新的第一回答点。

如果新增入口：

- 不提供新的稳定主语
- 不提供新的固定顺序
- 不提供新的发言权限校正
- 只是重复已有前门的话术

那它就不是优化，而是在制造第二入口债。

后续必须持续把“是否新增入口”当成高成本决策，而不是整理动作。

更稳的动作顺序也应先记成：

1. 先改可见标签
2. 再补 `navigation`
3. 再同步专题 README
4. 最后才考虑新开前门

只有当现有入口真的承载不了新的第一问或新的申诉位阶时，新增入口才算必要。

### 21. 不能让章节增长速度快过检索能力

如果继续只补新章节，而不补：

- 导航层
- 问题索引
- 读者任务导向入口
- 章节所属平面

那么蓝皮书会越来越完整，也越来越难用。

后续必须持续把“写更多”改成“更容易找到”。

### 22. 不能把多 Agent prompt 写成提示词技巧

如果后续只写：

- coordinator prompt 句子
- worker prompt 模板

而不写：

- fresh subagent vs fork
- team context / teammate mailbox
- task-notification
- ownership / write set
- 隔离优先于并行

那就会把协作 runtime 再次退化成“prompt 工程”。

### 23. 不能把 `services` / `utils-heavy` 草率写成结构问题

后续看到大 `utils/` 时，必须先问：

1. 这是 helper，还是 invariant。
2. 这是单域逻辑，还是跨域基础设施。
3. 它造成的风险是边界失效，还是热点文件过载。

否则会把一个工程上合理的 shared-kernel 写成坏味道清单。

### 24. 不能把恢复写成“重新读消息”

如果后续只写：

- transcript
- resume
- 历史会话

而不写：

- sidechain transcript
- `TaskOutput`
- file history / attribution
- context collapse restore
- internal-events / ingress 路径

就会继续把 Claude Code 的恢复能力写浅。

### 25. 不能把前台交互写成聊天 UI 细节

如果后续只写：

- 有个 REPL
- 有个 transcript mode
- 有个搜索框

而不写：

- sticky prompt
- message actions
- 主语切换
- queued commands
- transcript pager 语义

就会低估“人类如何保持对 agent runtime 的控制”。

### 26. 不能把哲学母题拆成孤立结论

如果后续继续把：

- prompt 魔力
- 安全设计
- token 经济
- 工程质量

分开夸，却不问它们是不是同一设计原则的不同投影，蓝皮书就会越来越细，但越来越散。

后续必须持续追问：

1. 这些现象背后是不是同一个 runtime plane。
2. 它们之间共享了什么 invariant。
3. 哪些章节应该先写母题，再写专题。

### 27. 不能让索引入口落后于正文生长

如果 `bluebook/README.md`、`navigation/`、专题 README 仍停留在上一轮，而正文已经长出新判断标准，读者会先迷路，再误解。

后续必须持续同步：

1. 主入口阅读链
2. 问题导向入口
3. 目录级能力地图
4. 新章节属于哪条母线

### 28. 不能把“第一性原理使用法”退回命令清单

如果 `guides/` 只继续追加命令和技巧，而不把：

- 目标
- 预算
- 对象
- 边界
- 回写

写成稳定方法，使用专题就会重新退回“功能教程”，而不是 runtime 用法。

后续必须持续追问：

1. 这条使用建议对应哪种对象模型。
2. 它在收紧哪一类预算。
3. 它要求用户回写什么稳定状态。

### 29. 不能让 Explainability 壳越位成改判层

如果后续只写：

- release card
- steady verdict
- handoff prose
- summary 命名

而不写：

- `Authority / Boundary / Transcript / Lineage / Continuation`
- 这些前置对象为什么仍然成立
- Explainability 只是在解释哪一环断了

那么蓝皮书就会把诊断层重新写成第二宪法。

后续必须持续把 `Explainability` 限定为 diagnostic naming，而不是 adjudicative rewrite。

### 30. 不能只统一词汇，不统一顺序

如果后续入口层只会复用：

- 同一组公式
- 同一组 noun
- 同一组 slogan

却不给 later maintainer 同一条最小排查顺序，目录就仍然只完成了“同词化”，没有完成“同序化”。

后续必须持续同步三组顺序：

1. Prompt witness order
2. governance failure order
3. current-truth reject order

### 31. 不能把目录优化做成入口增殖

如果后续所谓“优化目录”主要表现为：

- 再补一个总览页
- 再补一条 route map
- 再补一层兼容入口

而不先问：

- 这是否减少了第一次回答同题的入口数
- 这是否更清了 `speaking rights / appeal chain`
- 这是否增强了 later maintainer 的局部可反对性

那就不是优化，而是在制造新的入口债。

后续必须持续把目录优化写成“减少第一回答点 + 固定申诉链”的治理动作。

### 32. 不能让 canonical object 只活在正文中段

如果后续某个对象已经在正文中段被稳定写成：

- canonical object
- shared ABI
- stable verdict reason

却没有同步回写：

- H1
- 顶层对象清单
- 消费顺序
- 问题级首答路由

later consumer 读到的就仍是旧世界，真正成熟的对象反而只活在正文内部。

后续必须持续把“mid-body 已升格为 canonical”的对象，在同一批次内同步回标题、对象表、顺序表与入口层；否则这不叫深化，而叫局部真相外溢。

## 三、后续写作优先级

1. 先补 workflow engine 当前可见边界与实现缺口。
2. 再补 MCP 实战配置、plugin manifest、marketplace、MCPB、LSP、channels 的更细产品/接入手册。
3. 再补 REPL 更细的 scroll/search/selection 时序图。
4. 再补 memory / CLAUDE.md / scratchpad / durable knowledge 的统一知识层专题。
5. 继续把事件时间线、状态回写、consumer subset 做成双通道对照。
6. 继续把 request / response / follow-on message 做成闭环对照。
7. 继续把 prompt 装配链、角色合同、前台控制面与协作 runtime 做成独立阅读链。
8. 继续把源码质量拆成依赖图、schema、cache、恢复、工程债务几个可检验面。
9. 最后再升维到哲学总结。
10. 主线章节只保留一跳式结论，深挖专题进入对应目录。
11. 每一轮都明确自己是在补控制面、执行面、宿主面、适配器面、时序面、闭环面、事件面、连接面、状态面、真相面、提示词面、协作面、前台面还是演化面。
12. 每一轮正文增长后，优先补主入口、导航与问题索引，再继续扩正文。

## 四、蓝皮书目录优化原则

后续尽量遵守:

- `bluebook/`: 给读者的正式主线
- `architecture/`: 结构深挖
- `api/`: 能力与接口支持
- `guides/`: 用法与工作流
- `philosophy/`: 设计思想与第一性原理
- `development/`: 研究过程与长期记忆

## 五、完成标准

当一个章节同时满足下面五条，才算真正完成:

1. 有结论
2. 有证据
3. 有边界
4. 有可操作启发
5. 有正确层级与明确申诉链
