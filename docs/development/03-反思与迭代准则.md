# 反思与迭代准则

## 目的

这份文档不是源码结论，而是给蓝皮书写作者自己的约束。

目标只有一个:

让后续迭代越来越像“可验证研究”，而不是“会写长文的总结”。

## 一、苏格拉底式自问清单

每写完一章，都问自己:

1. 我写的是事实、解释，还是推论。
2. 这个结论能被哪几个源码锚点直接支持。
3. 这个判断是否把 public / gated / internal 混在了一起。
4. 如果读者只想接入功能而不是理解架构，这一章对他是否仍然有用。
5. 如果源码升级，哪一段最可能先过时。

## 二、本轮发现的改进方向

### 1. 不能只偏爱架构

第一阶段已经能回答“为什么 Claude Code 强”，但对“接口怎么暴露、宿主怎么接、消息怎么流动”的支持还不够。

### 2. 不能把能力边界写模糊

凡是 `feature()`、GrowthBook、policy、account entitlement 参与的能力，都必须明确写边界。

### 3. 不能只写概念，不写状态机

Claude Code 很多真正的价值在状态流转里:

- permission request
- task lifecycle
- MCP connection states
- session restore
- compact / reactive compact

后续必须多写状态机、多写时序，而不只是多写形容词。

### 4. 不能把主线蓝皮书写成字段百科

`bluebook/` 应该越来越像“高密度主线判断”，字段级索引、算法级机制和 gate 细拆应继续下沉到 `api/`、`architecture/`、`philosophy/`。

### 5. 不能低估“状态面”

Claude Code 不只是一套控制面和执行面，它还有非常重的状态面：

- transcript
- session metadata
- layered memory
- session memory
- file history / rewind

如果后续写作继续偏爱“能看见动作的部分”，蓝皮书会系统性低估 Claude Code 的可恢复性设计。

### 6. 不能把 SDK 写成文本输出面

如果后续只写 assistant reply、最终 result，而不写：

- `stream_event`
- `tool_progress`
- `task_*`
- `auth_status`
- `session_state_changed`
- `files_persisted`

就会把 Claude Code SDK 误写成“把模型回答嵌进 UI”的薄封装，而不是把 runtime 事件流暴露给宿主的集成面。

### 7. 不能把 MCP 写成工具列表

如果继续只按“有哪些 server / tool”来写 MCP，就会漏掉它真正更重要的四层结构：

- config scope
- transport
- connection state
- control actions

后续必须持续把 MCP 写成连接平面，而不是插件附属清单。

### 8. 不能把 Claude Code 写成 terminal shell

如果继续只从：

- REPL
- slash command
- assistant 输出

理解 Claude Code，就会低估：

- `control_request` / `control_response`
- `control_cancel_request`
- `interrupt`
- `get_context_usage`
- `mcp_set_servers`
- remote session / direct connect / bridge host

后续必须持续区分聊天外壳、transport plane 与 runtime control plane。

### 9. 不能把协议全集直接等同于宿主实现

如果继续只看：

- `controlSchemas.ts`
- subtype 列表
- `StructuredIO`

而不看：

- bridge 实际支持面
- direct-connect 实际支持面
- `RemoteSessionManager` 实际支持面

就会把“协议全集”和“适配器子集”写成一回事。

后续必须在每一处宿主分析里显式标出：

1. schema 全集
2. 控制平面主路径
3. 当前适配器子集

### 10. 不能只写 happy path，不写失败路径

如果继续只写：

- 正常 `control_request`
- 正常 `control_response`
- 正常权限批准

而不写：

- `control_cancel_request`
- unknown subtype error
- outbound-only error
- abort reject
- auth recovery dropping

蓝皮书就会把时序真相抹平。

后续必须持续把成功路径与失败路径并排写清。

### 11. 不能把闭环写成单向 RPC

如果后续只写：

- request subtype
- response payload

而不写：

- 后续 `SDKMessage`
- `session_state_changed`
- `status`
- `task_*`
- `elicitation_complete`

就会把 Claude Code 的宿主协议误写成普通 RPC，而不是 control loop。

后续必须持续把 request / response / follow-on message 一起写成闭环。

### 12. 不能把运行时真相写成单通道

如果后续只写：

- `SDKMessage`
- `stream_event`
- `task_*`
- `status`

而不写：

- `worker_status`
- `requires_action_details`
- `external_metadata`
- adapter 的 consumer subset

蓝皮书就会把 Claude Code 误写成“让宿主自己从消息里猜状态”的系统。

后续必须持续区分：

1. 事件时间线
2. 最新快照
3. 恢复后如何重建
4. 哪些 consumer 实际消费哪些信号

## 三、后续写作优先级

1. 先补接口面。
2. 再补事件流、状态机和时序图。
3. 再补宿主控制平面与远程 host 适配。
4. 再把协议全集与适配器子集的边界写成矩阵和对照图。
5. 再把字段对照与失败路径做成样例和时序图。
6. 再把 request / response / follow-on message 做成闭环对照。
7. 再把事件时间线、状态回写、consumer subset 做成双通道对照。
8. 最后再升维到哲学总结。
9. 主线章节只保留一跳式结论，深挖专题进入对应目录。
10. 每一轮都明确自己是在补控制面、执行面、宿主面、适配器面、时序面、闭环面、事件面、连接面、状态面、真相面还是演化面。

## 四、蓝皮书目录优化原则

后续尽量遵守:

- `bluebook/`: 给读者的正式主线
- `architecture/`: 结构深挖
- `api/`: 能力与接口支持
- `guides/`: 用法与工作流
- `philosophy/`: 设计思想与第一性原理
- `development/`: 研究过程与长期记忆

## 五、完成标准

当一个章节同时满足下面四条，才算真正完成:

1. 有结论
2. 有证据
3. 有边界
4. 有可操作启发
