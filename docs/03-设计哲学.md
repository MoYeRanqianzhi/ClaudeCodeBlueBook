# 设计哲学

## Claude Code 为什么强

从源码出发，我更愿意把 Claude Code 的能力公式写成：

`开发表现 = 模型能力 × harness 工程 × 运行策略 × 扩展生态`

如果只看模型，Claude Code 很难解释自己为什么比“同模型 + 简单 shell wrapper”强那么多。真正拉开差距的是后面的三项。

## 一、Harness First

`QueryEngine` 和 `query()` 说明 Claude Code 的中心不是 prompt，而是完整 runtime。

关键证据：

- `src/QueryEngine.ts:175`
- `src/QueryEngine.ts:209`
- `src/query.ts:219`

它把这些能力都编排在 loop 周围：

- 输入预处理
- 系统 prompt 装配
- 工具调度
- 权限判定
- 上下文压缩
- 异常恢复
- 任务通知
- 结果持久化

这意味着 Claude Code 的设计哲学不是“让模型一次答对”，而是“让系统在多轮、不确定、可中断的开发现场持续运作”。

## 二、上下文是预算，不是垃圾桶

Agent prompt、fork 机制、attachment 机制都在传达同一个哲学：上下文是昂贵资产。

关键证据：

- `src/tools/AgentTool/prompt.ts:48`
- `src/tools/AgentTool/prompt.ts:53`
- `src/tools/AgentTool/prompt.ts:85`
- `src/tools/AgentTool/prompt.ts:91`

几个典型取舍：

- agent list 能放附件，就不放进动态 tool 描述里，避免 prompt cache bust
- 能 fork 就 fork，避免把工具噪音带回主线程
- 不能直接失败时，就 compact、collapse、续写

这是一种非常“系统工程”的思维：不是让每次请求尽量大，而是让整个会话尽量稳。

## 三、默认隔离，按需共享

Claude Code 的多 Agent 设计不是“复制几个会话一起跑”，而是明确区分共享与隔离。

关键证据：

- `src/tools/AgentTool/AgentTool.tsx:217`
- `src/tools/AgentTool/AgentTool.tsx:218`
- `src/tools/AgentTool/AgentTool.tsx:256`
- `src/tools/AgentTool/runAgent.ts` 中关于 sidechain transcript、agent MCP、subagent context 的逻辑

它背后的原则是：

- 默认不要让 worker 污染主上下文
- 默认不要让 worker 继承过宽权限
- 默认不要让并发执行写进同一工作目录

这也是为什么 background、worktree、remote 这些能力会自然出现在 AgentTool 的输入 schema 中，它们不是附加参数，而是多 Agent 架构的安全边界。

## 四、能力增长必须经过统一协议

Claude Code 没有把扩展能力散落在各处，而是尽量通过统一协议接入：

- Tool protocol
- Skill frontmatter
- Plugin registry
- MCP transport

关键证据：

- `src/Tool.ts:123`
- `src/skills/loadSkillsDir.ts:180`
- `src/plugins/builtinPlugins.ts:1`
- `src/entrypoints/mcp.ts:59`

这个设计有两个直接好处：

1. 新能力更容易接入现有权限、日志、状态系统
2. 新能力更容易被 product gate 和 policy gate 管理

这解释了 Claude Code 为什么能同时演化出 Skills、Plugins、MCP，而没有变成三套互不兼容的孤岛。

## 五、性能优化不是附加项，而是产品原则

只看入口层就能看出它对启动速度和成本的敏感：

- `cli.tsx` 先做快路径，避免不必要模块加载
- `init.ts` 提前做预连接、异步预取与延迟导入
- `main.tsx` 在真正装配工具前完成 runtime 底座

关键证据：

- `src/entrypoints/cli.tsx:28`
- `src/entrypoints/cli.tsx:36`
- `src/entrypoints/init.ts:90`
- `src/entrypoints/init.ts:153`

这不是简单的“写得更快”，而是把 latency 视为 agent 体验的一部分。开发助手如果总是慢半拍，用户就不会把它当同伴，而只会把它当接口。

## 六、安全与产品实验内生于运行时

Claude Code 的源码里同时存在：

- feature gate
- runtime gate
- managed settings
- policy limits
- classifier
- deny/allow 规则

这说明它不是一个只依赖 system prompt 的 agent，而是一个可以被运营、被限制、被实验、被托管的产品 runtime。

关键证据：

- `src/entrypoints/cli.tsx:108`
- `src/entrypoints/init.ts:120`
- `src/tools.ts:193`
- `src/utils/permissions/permissions.ts:473`

从产品设计角度看，这是一种成熟信号。因为真正的企业级 agent 不可能只考虑“能不能做”，还必须考虑“谁可以做、何时可以做、出了问题如何收口”。

## 七、苏格拉底式追问：它的本质是什么

如果继续追问：

1. 它为什么不像普通 CLI 那样薄。
2. 它为什么不像普通 chat app 那样松。
3. 它为什么把 task、permission、worktree、plugin、MCP 都拉进同一系统。

那么一个更接近第一性原理的答案会是：

Claude Code 试图解决的不是“让模型会写代码”，而是“让一个模型驱动的行动体，能在真实软件工程环境中可控地工作”。

它的本质不是一个更大的 prompt，而是一套围绕软件工程现实约束构建出来的 agent operating environment。

## 可迁移原则

如果要把 Claude Code 的强点迁移到自己的 Agent 产品里，最值得抄的不是文案，而是这些原则：

- 把 runtime 当产品，而不是把 prompt 当产品
- 把上下文当预算运营
- 让扩展能力都经过统一协议
- 让并发默认隔离
- 让权限成为运行时系统，而不是提示词声明
- 让恢复链和压缩链成为默认路径，而不是报错后的补丁
