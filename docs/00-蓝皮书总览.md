# 蓝皮书总览

## 一句话结论

Claude Code 的“魔力”不在单次模型输出，而在它把模型嵌进了一套可持续运作的开发操作系统里：有启动链路，有权限系统，有工具调度，有上下文压缩，有异步任务，有多 Agent 隔离，也有插件、Skills、MCP 与远程能力。

## 研究边界

本书当前聚焦公开 npm 包可恢复出的本地源码基线：

- 包名：`@anthropic-ai/claude-code-source`
- 版本：`2.1.88`
- 文件规模：`1902` 个 `ts/tsx/js` 文件
- 核心源码目录：`src/`
- 关键入口：`src/entrypoints/cli.tsx`、`src/main.tsx`、`src/QueryEngine.ts`、`src/query.ts`

补充说明：

- 源码大量使用 `bun:bundle` 的 `feature()` 和 `MACRO`，说明构建阶段会做强烈的编译期开关裁剪。
- 部分能力明显是 feature-gated/internal，不能把“代码里提到”直接等同于“对外可用”。

## 当前主张

### 主张一：Claude Code 是 harness-first，而不是 prompt-first

单看 `query()`，它当然还是典型的 message -> model -> tool_use -> tool_result 循环；但源码真正复杂的部分并不在“调模型”本身，而在模型前后那一圈工程化护栏：

- 启动优化与快路径
- 配置与安全环境变量装配
- 权限判定与自动分类
- 上下文压缩与恢复链
- 多 Agent、后台任务、远程任务
- Skills、Plugins、MCP 等扩展面

### 主张二：它把“上下文”当作稀缺资源来运营

源码里反复出现的设计动机不是多加能力，而是避免上下文污染：

- 能快路径就不加载全量模块
- 能 fork 就不把子任务噪音留在主上下文
- 能 attachment 就不把动态信息塞进工具 schema
- 能 summary/compact 就不让历史无限膨胀
- 能后台跑就不阻塞当前交互

### 主张三：它把“安全”和“产品实验”都写进运行时

Claude Code 不是一个只靠 prompt 约束的 agent。权限模式、deny/allow 规则、classifier、hooks、managed settings、GrowthBook gate 都是运行时真实存在的行为边界。

## 读者该带着什么问题继续读

1. 它的启动链路如何把一个 CLI 启成一个长期运行的 agent runtime。
2. 它的 Agent loop 如何处理工具、压缩、恢复和异步任务。
3. 它为什么强调 subagent、fork、background、worktree。
4. 它怎样把 Skills、Plugins、MCP 融合成统一扩展面。
5. 它的设计原则里，哪些适合迁移到自己的 Agent 产品里。
