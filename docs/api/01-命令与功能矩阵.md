# 命令与功能矩阵

本章回答两个问题:

1. Claude Code 的功能面如何在命令层组织。
2. 命令、skill、plugin command 在运行时如何汇流。

## 1. 命令不是静态表，而是多来源装配

`COMMANDS()` 只是内置命令主表。真正进入用户会话的命令列表，还会继续叠加:

- bundled skills
- builtin plugin skills
- skill dir commands
- workflow commands
- plugin commands
- plugin skills
- dynamic skills

证据:

- `claude-code-source-code/src/commands.ts:258-346`
- `claude-code-source-code/src/commands.ts:353-398`
- `claude-code-source-code/src/commands.ts:449-469`
- `claude-code-source-code/src/commands.ts:476-517`

结论:

Claude Code 的命令系统本质上是“能力索引层”，不是一个固定 enum。

## 2. Command 类型模型

源码把命令分成三类:

### 2.1 `prompt`

适合 skill 和 prompt 型 slash command。它可以携带:

- `allowedTools`
- `model`
- `hooks`
- `skillRoot`
- `context`
- `agent`
- `effort`
- `paths`

证据:

- `claude-code-source-code/src/types/command.ts:25-57`

### 2.2 `local`

适合纯本地执行、返回文本或 compact 结果的命令。

证据:

- `claude-code-source-code/src/types/command.ts:59-78`

### 2.3 `local-jsx`

适合需要本地 UI、异步回调、命令完成后决定是否继续 query 的命令。

证据:

- `claude-code-source-code/src/types/command.ts:80-152`

## 3. 命令可用性不是单一开关

命令有两层边界:

1. `availability`: 用户是谁能不能看见
2. `isEnabled()`: 当前环境下有没有启用

`availability` 目前至少区分:

- `claude-ai`
- `console`

证据:

- `claude-code-source-code/src/types/command.ts:154-173`
- `claude-code-source-code/src/commands.ts:417-442`

这意味着命令系统同时受 auth/provider、feature gate、env、settings 等多层控制。

## 5. 为什么命令系统很重要

如果从产品表面看，很多人会把 Claude Code 误解为“对话 + 工具”。

但命令系统说明，它实际上还有一个明确的 workflow control plane。  
用户不仅可以提问，还可以:

- 改运行模式
- 看状态
- 管理插件和 MCP
- 恢复会话
- 调整权限
- 获取上下文使用信息

所以 Claude Code 的用户模型并不是纯聊天用户，而是 orchestrator。

## 6. 命令系统的三个设计含义

1. 命令是 workflow control，不只是快捷方式。
2. skill 与 plugin command 会进入同一命令汇流层，说明扩展生态不是旁路。
3. availability 与 isEnabled 分离，说明 Claude Code 把“谁能看见”和“现在是否启用”视为不同问题。

## 7. 待补全项

3. 交叉核对 `commands/` 子目录与 `COMMANDS()` 主表，区分内置、internal-only、gated。
